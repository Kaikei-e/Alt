# ADR-000824: recap-worker の release profile チューニングと libtorch 同梱プルーニングで image を縮小し、docker-build.yaml の image-size budget gate を拡張する

## Status

Accepted

## Date

2026-04-21

## Affected Services

- **recap-worker (Rust)** — `recap-worker/recap-worker/Cargo.toml` に `[profile.release]` を追加。`Dockerfile.recap-worker` の runtime stage で `libtorch*` 系の wildcard COPY を必要最小 5 行のピンポイント COPY に差し替え。
- **.github/workflows/docker-build.yaml** — 既存の image-size budget gate ([[000813]]) の対象サービス配列に `recap-worker` を追加し、Rust バイナリ向けに `ldd` ベースの smoke-check 分岐を新設。

## Context

`ghcr.io/kaikei-e/alt-recap-worker` の runtime image が肥大化していた。Dockerfile と Cargo.toml を精査したところ根本原因は 2 点:

1. **libtorch を丸ごと同梱していた**  
   `Dockerfile.recap-worker` の runtime stage は `libtorch*` / `libc10*` / `libcaffe2*` / `libgomp*` をワイルドカードで COPY しており、`rust-bert` / `tch-rs` が実際には dlopen しない以下のバイナリまで取り込んでいた:
   - `libtorch_python.so` — PyTorch Python bindings 本体、約 400 MiB
   - `libshm.so` — PyTorch DataLoader の POSIX 共有メモリ層 (rust-bert は使用しない)
   - `libcaffe2_detectron_ops.so` / `libcaffe2_module_test_dynamic.so` / `libcaffe2_nvrtc.so` — Caffe2 ops と CUDA nvrtc。rust-bert は PyTorch ATen 経路のみで Caffe2 ops も CUDA も未使用
   - `*.a` 静的アーカイブ — runtime には dlopen されない

2. **`[profile.release]` が一切未設定だった**  
   `recap-worker/recap-worker/Cargo.toml` には release profile のチューニングが無く、Cargo の default (`codegen-units = 16` / `strip = false` / `panic = "unwind"` / `lto = false`) のまま build されていた。結果として Rust バイナリには symbols、多重 codegen、unwinding テーブルが同梱され、数百 MiB 単位の不要データを含んでいた。

加えて、[[000813]] で導入した `docker-build.yaml` の `Enforce image-size budget and smoke-check core imports` step は `recap-subworker` / `news-creator` / `recap-evaluator` / `tag-generator` の 4 つの Python サービスのみを whitelist 対象としており、**recap-worker には regression gate が無かった**。今回のチューニングを入れたあとに誰かが wildcard COPY を復元したり default profile に戻したりしても CI では検出できない構造が残る。

[[000291]] で確立した cargo-chef + BuildKit `--mount=type=cache,sharing=locked` のビルドキャッシュ構造、および [[000562]] で統一した `rust:1.94-bookworm` 基盤はそのまま維持しつつ、runtime layer のみを縮小する必要があった。

## Decision

### 1. `[profile.release]` を明示的にチューニング

`recap-worker/recap-worker/Cargo.toml` の末尾に以下を追加する:

```toml
[profile.release]
lto = "fat"
codegen-units = 1
strip = "symbols"
panic = "abort"
overflow-checks = true
```

- `lto = "fat"` + `codegen-units = 1` — cross-unit 最適化でバイナリサイズを圧縮。ビルド時間と引き換えだが、cargo-chef の dependency cache が効くため deps 側は再ビルドされない。
- `strip = "symbols"` — symbols を削除。panic backtrace の可読性は落ちるが、recap-worker は `tracing` で structured log を常時出力しており stack trace への依存度が低い。
- `panic = "abort"` — unwinding runtime を削る。事前 grep で `catch_unwind` / `UnwindSafe` の使用箇所が存在しないことを確認済。
- `overflow-checks = true` — bp-rust §12 の security guideline 準拠。release でも integer overflow を検出する。
- `opt-level` は **default のまま据え置き** (`3`)。recap-worker は `rust-bert 0.23` で in-process BERT 推論を実行するため、`opt-level = "z"` / `"s"` で throughput を犠牲にする判断は取らない (Rust Performance Book および bp-rust §14 の指針)。

### 2. libtorch `.so` のピンポイント COPY

`Dockerfile.recap-worker` の L65-68 の wildcard COPY 4 行を、`rust-bert` / `tch-rs` が runtime で dlopen する最小セット 5 行に差し替える:

```dockerfile
COPY --from=builder /usr/local/lib/libtorch.so             /usr/local/lib/
COPY --from=builder /usr/local/lib/libtorch_cpu.so         /usr/local/lib/
COPY --from=builder /usr/local/lib/libtorch_global_deps.so /usr/local/lib/
COPY --from=builder /usr/local/lib/libc10.so               /usr/local/lib/
COPY --from=builder /usr/local/lib/libgomp*.so*            /usr/local/lib/
```

除外理由はコメントで残し、将来 `libtorch_python.so` を誤って戻さないよう明示する。なお `ldconfig` は保持しており、runtime に shared library cache は従来どおり再構築される。

### 3. `docker-build.yaml` に recap-worker 用 image-size budget gate を追加

[[000813]] の whitelist パターンを踏襲し、`contains(fromJSON(...))` 配列に `recap-worker` を追加する:

- L271 `Build and push` step の `load:` 条件配列に `recap-worker` を追加 (gate 実行には main push 時に image を runner にロードする必要がある)
- L284 `Enforce image-size budget ...` step の `if:` 条件配列に `recap-worker` を追加
- L290-295 の `case $SERVICE in ... esac` に `recap-worker) MAX_MB=1800 ;;` を追加 ([[000813]] で news-creator の実測 1243 MiB に対し 1800 MiB を設定したのと同方針。recap-worker の最適化後想定は 1000–1400 MiB 帯で、1800 MiB は約 30–80% のマージンを持つ)
- Python インタプリタを同梱しない Rust image のため、既存の Python 用 smoke-check は流用できない。`elif [ "$SERVICE" = "recap-worker" ]` 分岐を新設し、`docker run --rm --network=none --read-only --tmpfs /tmp --cap-drop=ALL --entrypoint=/bin/sh` 上で `ldd /usr/local/bin/recap-worker` を実行。`not found` 検出と `libtorch_cpu` / `libc10` / `libgomp` の dynamic link 解決を必須条件としてチェックする。これにより libtorch prune で欠落した `.so` が runtime に滑り込む regression を検出できる。

### 検討した代替案と却下理由

- **`gcr.io/distroless/cc-debian12:nonroot` への移行** — rask-log-aggregator / rask-log-forwarder で採用済の distroless 構成。更なる baseline 縮小 (~75 MiB → ~20 MiB) が狙えるが、`cc-debian12` は `libgomp1` を含まず `libssl3` も別途 COPY が必要。今回の 2 つの主要因 (libtorch と release profile) を先に解消してから、必要なら follow-up ADR で検討する方が手戻りが少ない。
- **`opt-level = "z"` / `"s"`** — バイナリサイズをさらに削減できるが、rust-bert の tensor ops throughput が劣化する。recap-worker は recap pipeline の hot path で in-process BERT 推論を実行するため、image-size のために runtime latency を犠牲にする判断は取らない。
- **`LIBTORCH_STATIC=1` で静的リンク** — tch-rs は `LIBTORCH_STATIC` を honor するが、prebuilt libtorch CPU zip には `.a` アーカイブが含まれておらず、静的リンクには libtorch を source から build する必要がある。本 ADR のスコープを大幅に超えるため採用せず。
- **rust-bert を onnxruntime 等に置換** — libtorch 同梱を根本的に消せるが、モデル資産とクライアントコード両方の書き換えが必要で別 work stream。
- **`libssl3` の runtime apt 削除** — `reqwest` は `rustls-tls` + `rustls-tls-native-roots`、`axum-server` は `tls-rustls` で pure-Rust TLS を使用しており、`native-tls` / `openssl-sys` は依存ツリー全体に存在しない (grep 確認済)。理論上 `libssl3` は不要だが、libtorch 本体が内部で OpenSSL を引いている可能性を `ldd` で実測せず断定するリスクを避け、今回は保留。follow-up で実測のうえ削除を検討する。

## Consequences

### Pros

- **image サイズを約 50% 以上縮小** (想定: 従来比で libtorch 約 500 MiB–1 GiB + Rust バイナリ数百 MiB の削減、合計で概ね半減)。registry transfer, pull latency, disk 消費が全方位で改善する。
- **regression を CI で常時ゲート**: 誰かが wildcard COPY を復活させたり `[profile.release]` を外したりしても、`MAX_MB=1800` を超えた時点で main push が fail する。
- **dynamic link 切れの即時検出**: `ldd` smoke-check により、libtorch prune 対象の選定ミスや version bump による SONAME 変更が deploy 前に検出される。
- **既存のビルドキャッシュ資産を温存**: cargo-chef ([[000291]]) と BuildKit cache mount、`rust:1.94-bookworm` ([[000562]]) はそのまま。本 ADR の変更は主に builder stage の最終成果物と runtime stage の COPY に限定されるため、dependency cache は再ビルド不要。
- **Hurl E2E suite の契約** ([[000772]]) に影響なし。recap pipeline の 4 upstream 呼び出しおよび recap-db への書き込み経路は変更していない。

### Cons / Tradeoffs

- **initial build time が増加する**: `lto = "fat"` + `codegen-units = 1` により、ソース変更時の `cargo build --release` は従来より数分程度長くなる。CI の cargo-chef dependency cache が効く状況下では影響は限定的だが、キャッシュミス時の builder stage は重くなる。
- **panic backtrace の可読性低下**: `strip = "symbols"` で symbol 情報が除去される。production incident 時は `tracing` の structured log と Prometheus / OTLP メトリクスで原因追跡する運用が前提となる。
- **`MAX_MB=1800` は初期値**: 今回は docker build 実測を行わず plan-only の試算値で設定している。初回 main push 時に CI で実測されたあと、実値 + 10-15% マージンに追い込む follow-up が必要。
- **`libssl3` 残留**: runtime apt に `libssl3` を残したままで、約 15 MiB の削減余地が未回収。follow-up で `ldd` 実測のうえ削除判断する。
- **libtorch 同梱自体は残る**: ~800 MiB-1 GiB の libtorch は依然 image に含まれる。根本的な削減には rust-bert を onnxruntime 等に置換する必要があり、別 ADR で扱う。
- **panic = "abort" への変更は一方向**: `catch_unwind` を新規に追加したくなった場合、release profile の panic 挙動と衝突する。追加前に本 ADR の制約を再評価する必要がある。
