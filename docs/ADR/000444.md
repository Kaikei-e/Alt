# ADR-000444: Ask Augur の回答品質を向上させるために最低文字数を800に引き上げRAGベストプラクティスを適用する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- rag-orchestrator (Go)

## Context

Ask Augur の回答品質に以下の課題があった。

1. **回答の情報密度不足** — プロンプトで「500文字以上」と指示していたが、コードレベルでの検証がなく、短い回答がそのまま返却されることがあった。特に8Bモデル（gemma3-4b-rag）では、トークン生成が途中で切れやすく、ヘッダのみの空回答が発生していた
2. **ハルシネーション抑制の不足** — プロンプトにグラウンディング指示（コンテキストのみに基づく回答）が明示されておらず、モデルが外部知識で補完してしまうリスクがあった。2025-2026年のRAG研究（Anthropic、ZeroEntropy等）では、明示的なグラウンディング指示で60-80%のハルシネーション削減が報告されている
3. **マルチターン会話の意図把握** — 会話履歴のトランケーションが300文字と短く、指示代名詞（「それ」「この件」等）の解決指示がプロンプトに含まれていなかった。ACL 2025 MTRAGの研究では、明示的なコリファレンス解決指示により87%のクエリで意図把握が改善されている

## Decision

以下の3施策を同時に適用する。

### 1. 最低文字数を500→800に引き上げ、ソフトバリデーションを追加する

- プロンプトの品質基準を「800文字以上」に更新
- `OutputValidator` にルーン長ベースのソフトバリデーションを追加（`utf8.RuneCountInString`）
- **ハードリジェクトではなくソフトフラグ**（`ShortAnswer bool`）とする。8Bモデルはnarrow queryに対して800文字に達しない場合があり、正当な短い回答まで拒否するのは不適切
- `MinAnswerLength` は環境変数 `RAG_MIN_ANSWER_LENGTH` で設定可能（デフォルト800）
- 短い回答は `slog.Warn` でログ出力し、モニタリングで傾向を把握する

### 2. グラウンディング・反ハルシネーション指示を追加する

プロンプトの品質基準セクションに以下を追加:
- 提供されたコンテキスト情報のみに基づく回答の義務付け
- コンテキスト外の事実・数値の推測・捏造の禁止
- 情報不足時の明示的な不足点の報告

### 3. マルチターン会話の改善

- 会話履歴のトランケーションを300→500文字に拡大
- 指示代名詞の解決指示をプロンプトに明示追加
- 回答構造の各サブセクションに引用・出典の具体的指針を追加

### 代替案の検討

| 案 | 評価 |
|---|---|
| ハードリジェクト（800文字未満は再生成） | 8Bモデルの再生成は品質改善が保証されない。レイテンシが倍増するため不採用 |
| クライアント側での文字数チェック | バリデーションロジックが分散し、モデル変更時の追従が困難。サーバー側で一元管理する |
| RAG-Fusion（複数クエリ生成→統合） | 効果は高いが、LLM呼び出し回数が3-5倍に増加。現行の8Bモデルでは実用的でない |

## Implementation

### Config層

`RAGConfig` に `MinAnswerLength int` フィールドを追加。環境変数 `RAG_MIN_ANSWER_LENGTH` で上書き可能、デフォルト800。

### OutputValidator

```
NewOutputValidator(minAnswerLength int) → OutputValidator
```

- ステートフル化: `minAnswerLength` フィールドを保持
- `LLMAnswer` に `ShortAnswer bool` フィールド追加（JSONタグなし、内部フラグ）
- `Validate()` のサニタイズ後に `utf8.RuneCountInString` でルーン長チェック
- `minAnswerLength=0` で長さチェック無効（テスト用）

### プロンプト改善（prompt_builder.go）

| セクション | 変更内容 |
|---|---|
| 回答の品質基準 | 500→800文字、グラウンディング3指示追加 |
| 回答構造 | 各サブセクションに引用・正確性の具体的指針 |
| 会話履歴 | コリファレンス解決指示追加、トランケーション300→500文字 |

### ログ出力

`answer_with_rag_usecase.go` と `rag_answer_stream.go` の両パスで、バリデーション後に `ShortAnswer` フラグを検出して `slog.Warn` を出力。`answer_rune_length` と `query` をログに含め、モニタリングで短い回答の傾向を分析可能にする。

### DI

`container.go` で `NewOutputValidator(cfg.RAG.MinAnswerLength)` を注入。

## Consequences

### Pros

- **回答品質の向上** — グラウンディング指示により、コンテキスト外の情報混入リスクを大幅に低減
- **可観測性の向上** — 短い回答のログ出力により、モデル性能の傾向を定量的に把握可能
- **設定の柔軟性** — `RAG_MIN_ANSWER_LENGTH` 環境変数で、モデル変更時やA/Bテスト時に閾値を調整可能
- **マルチターン精度の改善** — 指示代名詞の解決指示とトランケーション拡大により、文脈を踏まえた回答が生成されやすくなる
- **ルーン長ベース** — 日本語テキストの正確な文字数カウント（`len()` はバイト数で3倍の差が出る）

### Cons / Tradeoffs

- **ソフトバリデーションの限界** — フラグを立てるのみで回答を拒否しないため、短い回答がそのままユーザーに表示される。将来的にモデルの品質が安定したらハードリジェクト+再生成への移行を検討
- **プロンプト長の増加** — グラウンディング指示等の追加により、プロンプトが約200文字増加。`maxPromptTokens=6000` の制約内に収まっているが、コンテキストチャンク数がわずかに減る可能性がある
- **8Bモデル依存** — 現行のgemma3-4b-ragでは800文字以上の高品質な回答が常に得られるとは限らない。モデルのアップグレード（14B等）と併せて効果を検証する必要がある
