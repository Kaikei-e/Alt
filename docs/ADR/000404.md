# ADR-000404: Knowledge Home Admin ダッシュボードを導入する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — KnowledgeHomeAdminService 新設、OTel メトリクス定義
- alt-butterfly-facade (Go) — Admin API のプロキシ
- alt-frontend-sv (SvelteKit) — Admin UI ページ・コンポーネント

## Context

Knowledge Home Phase 1-2 では projector・イベントストア・UI が構築されたが、運用者がシステムの健全性を確認する手段がなかった。具体的に以下の情報が不足していた:

- **Projector の投影状態** — checkpoint がどこまで進んでいるか、lag はどの程度か
- **Backfill ジョブの進捗** — 実行中・完了・失敗の状態と進捗率
- **Feature Flag の状態** — 現在どのフラグが有効で、rollout は何%か
- **Projection バージョン** — どのバージョンが active か

既存の Dashboard ページは RSS フィードの統計（記事数、サマリ数）に特化しており、Knowledge Home の投影パイプラインの「裏側」を可視化する場所がない。Stats ページもフィード統計のグラフであり、用途が異なる。

## Decision

**Knowledge Home 専用の Admin ダッシュボードを Settings カテゴリ配下に新設し、service-token 認証で保護された Admin API で投影運用情報を提供する。**

### Admin API 設計

`KnowledgeHomeAdminService` を新しい proto ファイル `knowledge_home_admin.proto` に定義し、既存の `KnowledgeHomeService`（ユーザー向け）と分離する。

| RPC | 用途 |
|-----|------|
| `TriggerBackfill` | Backfill ジョブ作成 |
| `PauseBackfill` / `ResumeBackfill` | ジョブの一時停止・再開 |
| `GetBackfillStatus` | 個別ジョブの状態照会 |
| `GetProjectionHealth` | Active version、checkpoint、lag、backfill jobs 一覧 |
| `GetFeatureFlags` | 現在のフラグ構成（read-only） |

**認証**: `ServiceAuthInterceptor` を使用（`X-Service-Token` ヘッダー）。既存の `BackendInternalService` と同じパターンで、ユーザー JWT 認証とは独立した service-to-service 認証。

### UI 設計

**配置**: `/settings/knowledge-home-admin` — Settings カテゴリ配下に配置し、サイドバーの Settings > KH Admin からアクセス。

**既存ページとの棲み分け:**
- **Dashboard** — RSS フィード統計（記事数、サマリ数、未読数）→ ユーザー向け
- **Stats** — フィード統計のトレンドグラフ → ユーザー向け
- **Recap Job Status** — recap-worker のジョブ監視 → ジョブ運用
- **KH Admin** — Knowledge Home の projection 運用に特化 → projector 運用

**コンポーネント構成:**

| コンポーネント | 役割 |
|--------------|------|
| `ProjectionStatusPanel` | Active version、checkpoint seq、last updated |
| `BackfillJobsTable` | Backfill ジョブ一覧（status badge、progress bar、error） |
| `FeatureFlagPanel` | 3 フラグの on/off 状態 + rollout% を read-only 表示 |
| `ReasonDistributionChart` | why_reason コード分布の横棒グラフ |
| `InteractionFunnelPanel` | generated → seen → opened → dismissed の漏斗 |
| `AdminMetricCard` | 汎用的な数値 + ラベル + ステータス色カード |
| `DegradedModeBanner` | Home ページの degraded mode 警告バナー |

**データ取得**: `useKnowledgeHomeAdmin.svelte.ts` composable が 10 秒間隔でポーリング。Svelte 5 Runes パターン（`$state` / `$derived`）を使用。

### OTel メトリクス

`alt.home.*` namespace で 10 メトリクスを定義:

| メトリクス | 種類 | 用途 |
|-----------|------|------|
| `alt.home.projector.events_processed` | Counter | 処理イベント数 |
| `alt.home.projector.lag_seconds` | Gauge | 投影遅延 |
| `alt.home.projector.batch_duration_ms` | Histogram | バッチ処理時間 |
| `alt.home.projector.errors` | Counter | エラー数 |
| `alt.home.page.served` | Counter | ページ提供数 |
| `alt.home.page.degraded` | Counter | degraded レスポンス数 |
| `alt.home.items.exposed` / `opened` / `dismissed` | Counter | ユーザー行動 |
| `alt.home.backfill.events_generated` | Counter | backfill 生成イベント数 |

これらのメトリクスは Prometheus exporter 経由で `/metrics` エンドポイントから取得可能。将来的に Grafana ダッシュボードから参照する際の命名規約を `alt.home.*` で統一した。

## Consequences

### Pros

- **運用可視性** — Projector の健全性を即座に確認でき、lag 増大や backfill 失敗に気づける
- **既存 UI との明確な棲み分け** — Dashboard/Stats はユーザー向け統計、KH Admin は projection 運用に特化
- **Proto 分離** — Admin API を別 proto ファイルに定義したため、ユーザー向け API の proto 変更が Admin に影響しない
- **セキュリティ** — service-token 認証により、一般ユーザーからは Admin API にアクセスできない
- **OTel 標準準拠** — メトリクスが OpenTelemetry 標準に従っており、Prometheus/Grafana エコシステムとの統合が容易

### Cons / Tradeoffs

- **ポーリングベース** — 10 秒間隔のポーリングであり、WebSocket/SSE によるリアルタイム更新ではない。運用監視としては十分だが、急激な状態変化の検知には最大 10 秒の遅延がある
- **Reason Distribution / Funnel はプレースホルダー** — 現時点ではメトリクス集計の専用エンドポイントがなく、静的なサンプルデータで UI を構築している。実データ表示には別途集計クエリの実装が必要
- **Admin UI は認証なし** — ページ自体は Settings 配下で誰でもアクセス可能。API は service-token で保護されているが、UI 側のアクセス制御（admin ロール判定等）は未実装
