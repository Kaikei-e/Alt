# ADR-000749: Knowledge Home stream の user/tenant/lens 境界欠落を JWT claim 分離と配信時 lens 可視性チェックで閉じる

## Status

Accepted

## Date

2026-04-16

## Affected Services

- alt-backend (Go) — Connect-RPC interceptor、stream handler、port、driver
- knowledge-sovereign (Go) — proto、handler、SQL

## Context

`StreamKnowledgeHomeUpdates` の Phase 4 監査（`docs/review/knowledge-home-phase0-4-audit-2026-03-18.md`）で 3 つの認可境界欠落が報告されていた。

1. **lens 境界欠落**: frontend は stream request に `lensId` を載せているが、backend handler はパースとログ出力のみで配信時の filter には一切使っていなかった。lens 範囲外の article 更新が配信され、Phase 2 の unary `GetKnowledgeHome` lens read path との整合が崩れていた。
2. **tenant 境界欠落**: Connect-RPC 用 interceptor (`auth_interceptor.go`) の `BackendClaims` 構造体は `tenant_id` claim をフィールドに持たず、`validateToken` は `UserContext.TenantID = userID` という単一テナント縮約に依存していた。auth-hub 側 JWT は既に `tenant_id` claim を発行しており、Echo 旧 middleware も読んでいたが、Connect 側だけが未連携だった。sovereign の SQL にも `tenant_id` 述語がなく、`user_id IS NULL` の system event は全テナントに漏れる構造だった。
3. **user 境界**: 既に SQL レベルで `user_id = $n OR user_id IS NULL` を適用済みだったが、回帰テストで境界を契約レベルで固定する必要があった。

OWASP A01 Broken Access Control（CWE-284、CWE-639 IDOR）の典型例。情報漏洩を契約テストで二度と起こさない形に閉じることがゴール。

なお現在は単一テナント運用で実質 `tenant_id == user_id` だが、claim を独立に配線しておくことで、将来のマルチテナント化で middleware と SQL を連鎖的に書き換えるリスクをここで吸収する。

## Decision

### lens scoping

**配信時フィルタ方式を採用する**。stream subscriber 側で lens を解決し、各 tick で取得した event のうち article aggregate に該当するものを「lens 範囲内か」を sovereign の projection 読み取りで判定し、合致しないものは drop する。

- 接続開始時に既存の `ResolveKnowledgeHomeLens` usecase で lens を 1 回解決し、接続中固定で使う。lens 切替は新規接続で表現する。
- 不正 / 他者所有の `lens_id` には `connect.CodeNotFound` を返す。`PermissionDenied` だと存在 oracle になるため不採用。
- 配信時 filter は `home_session` / `recap` aggregate には適用しない（既に user 境界で閉じている user action 通知だから）。
- 可視性 RPC が失敗した tick は article event を全 drop（fail-closed）。漏洩より遅延を選ぶ。

### tenant boundary

Connect-RPC interceptor の `BackendClaims` に `TenantID` フィールドを追加し、`validateToken` で UUID パースして `UserContext.TenantID` に格納する。claim 欠落は `errInvalidClaims` として **fail-closed** で拒否（縮約フォールバック禁止）。

sovereign 側 SQL にも `tenant_id = $n` 述語を追加し、proto request に `tenant_id` フィールドを必須として通す。

### sovereign API 拡張

新規 RPC `AreArticlesVisibleInLens` を追加。bulk で article_ids を受け、`knowledge_home_items` projection に対して `GetKnowledgeHomeItems` と同じ lens predicate（tag、source、time window、query text）で visibility map を返す。

`ListKnowledgeHomeItems` の拡張ではなく独立 RPC にしたのは、ページング/カーソル仕様を visibility 判定で汚染しないためと、lens 評価責務の所在を明確にするため。

### 検討した代替案

- **sovereign SQL に lens filter を組み込む**: 1 round trip で済むが、event ストアと lens 表現の責務が混ざるため不採用。lens は alt-backend 側 view 責務とし、sovereign は event/projection の真実所有者に留める。
- **frontend 側で lens フィルタする**: stream に他人の event を載せた時点で漏洩なので不可。
- **`PermissionDenied` を返す**: 存在 oracle 化を避けるため `NotFound` を採用。

## 実装の概要

### Clean Architecture 層別

| 層 | 変更 |
|---|---|
| Handler (`connect/v2/knowledge_home/home_query.go`) | lens 解決・配信時 filter・tenant 伝播 |
| Handler (`connect/v2/middleware/auth_interceptor.go`) | `tenant_id` claim 配線、fail-closed |
| Port (`port/knowledge_event_port/port.go`) | 既存 port に `tenantID` 引数追加、新 port `IsArticleVisibleInLensPort` |
| Driver (`driver/sovereign_client/read_client.go`) | proto に tenant_id を載せて転送、`AreArticlesVisibleInLens` の thin wrapper |
| Sovereign Handler (`handler/rpc_infra.go`) | tenant_id バリデーション、新 RPC server 実装 |
| Sovereign Driver (`driver/sovereign_db/`) | SQL に tenant 述語追加、`lens_visibility.go` 新設 |

### イミュータブルデータモデル原則の遵守

[[plan/IMPL_BASE]] で定めた append-first / reproject-safe / disposable projection の原則を破らないことを確認した。

- **append-first**: `knowledge_events` は INSERT only のまま、書き込み・スキーマ変更なし。
- **reproject-safe**: `projector` には lens / tenant 知識を一切持ち込まず、本変更は読み取り側にのみ閉じる。`knowledge_projector.go` 系は無修正。
- **disposable projection**: lens 可視性判定は `knowledge_home_items` projection を read として参照するだけで mutate しない。
- **versioned artifacts / why as first-class**: 不変。

### イベント種別ごとの配信ルール

| EventType / aggregate_type | 配信判定 |
|---|---|
| `ArticleCreated` / `SummaryVersionCreated` / `TagSetVersionCreated` / `ReasonMerged` (`article`) | tenant filter（DB） + lens visibility（handler） |
| `HomeItemOpened` / `HomeItemDismissed` (`home_session`) | tenant + user 境界済み。lens 判定 skip |
| `SummarySuperseded` / `TagSetSuperseded` | aggregate に応じて article 経路と同様 |
| その他 system 系 (`user_id IS NULL`) | tenant 一致が前提。article 紐付き event なら lens 判定も適用 |

### Migration

`knowledge-sovereign/migrations/00010_add_tenant_event_seq_index_for_stream_filter.sql` で `idx_knowledge_events_tenant_seq (tenant_id, event_seq)` 複合 index を追加。stream poller の `WHERE event_seq > $1 AND tenant_id = $2 AND (user_id = $3 OR user_id IS NULL)` クエリと latest-seq クエリの両方を 1 つの index で覆う。

### TDD で固めた回帰ネット

- `auth_interceptor_test.go`（4 ケース）: `tenant_id` claim 欠落 → 拒否、claim 値が UserID と異なるケースで縮約されないこと。
- `stream_scoping_test.go`（7 ケース）: 配信時 filter の通過 / 削除、`home_session` 経路の bypass、不正 lens で `NotFound`、可視性失敗時の fail-closed、article ID の dedupe、tenant の伝播。
- `knowledge-home-contract.test.ts`（2 ケース追加）: `StreamKnowledgeHomeUpdatesRequest` の `lens_id` フィールド存在を proto レベルで固定（lens 境界が将来こっそり消えないようにする regression net）。

## Consequences

### Pros

- マルチテナント情報漏洩のリスクが消滅し、stream と unary の lens 意味論が初めて整合した。
- tenant 境界が JWT claim 起点で first-class になり、将来のマルチテナント化で middleware/SQL を連鎖修正するコストを今ここで吸収した。
- fail-closed 設計で「漏洩より遅延」を保証。観測ログで filter 適用回数と lens 解決失敗回数が見えるようになり、不正 `lens_id` 試行の検出基盤も整った。
- イミュータブルデータモデル原則（append-first / reproject-safe / disposable projection）を一切破らずに防御層を追加した。projector は無修正。
- 契約テストで lens 境界が proto schema 変更に対して回帰検出されるようになった。

### Cons / Tradeoffs

- 1 tick あたり最大 1 回の追加 RPC（`AreArticlesVisibleInLens`）が発生する。bulk 化（最大 50 article_ids/RPC）で N+1 は回避済みだが、stream 接続数が増えると sovereign 側の read 負荷は増える。
- lens 切替を「新規接続」で表現するため、frontend が lens を頻繁に変える UX では reconnect コストが顕在化する可能性がある。`useStreamUpdates.svelte.ts` 側に再接続デバウンスが入っている前提。
- `tenant_id` claim を必須化したため、claim 未付与の旧 token は Connect-RPC で拒否される。auth-hub は常に claim を発行しているが、長寿命セッションの再ログインが必要なケースが理論上ありうる。
- `TenantID = UserID` 縮約コードを安全網としては残さず削除したため、claim 改ざん耐性は upstream JWT 署名検証に完全依存する。これは設計通りだが、JWT secret 漏洩時の影響範囲が tenant 軸まで広がる点は明示しておく。

### スコープ外（別 issue で扱う）

- `HomeItemOpened` event payload への `tenant_id` / `opened_at` 追加（[[review/knowledge-home-phase1-5-remediation-directives-2026-03-18]] P0-1）。
- `ReasonMerged` プロデューサ実装（同 P1-1）。
- `TodayDigest` availability fields の projector backing（同 P0-2）。
- 真のマルチテナント運用（複数 tenant への user 所属モデル）。本 ADR は claim 分離と filter 反映までを範囲とする。
