# ADR-000384: Push-Down Anti-Joinパターンによるバックフィルアルゴリズムの修正

## Status

Accepted

## Date

2026-03-14

## Affected Services

- alt-backend (Go) — `GetEmptyFeedID` RPC追加、`CheckFeedHasArticles` RPC削除
- pre-processor (Go) — バックフィルロジックの簡素化
- proto定義 — `internal.proto` のメッセージ・RPC変更
- search-indexer (Go), tag-generator (Python) — 生成コードの再生成（実装変更なし）

## Context

前回のADR（`CheckFeedHasArticles` RPC実装）後、バックフィルが**0件挿入**で終了する問題が発生した。

### データモデルの特性

```
feed_links (id, url)  ← 1つのRSS URL
    ↓ 1:N
feeds (id, feed_link_id)  ← ユーザーごとに1 feed
    ↓ 1:N
articles (id, url UNIQUE, feed_id, user_id)
```

1つの `feed_url` に対して多数の `feed_id` が存在する（ユーザーごとに1 feed）。例えば `theguardian.com/world/rss` に対して16,104のfeedsが存在し、そのうち4,915が記事あり、11,189が空だった。

### 問題の原因

1. `GetFeedIDByURL` が `QueryRow` で**最初の1件だけ**を返す
2. その1件がたまたま記事を持つfeedだった場合、`CheckFeedHasArticles` が `true` を返す
3. 結果、全候補がスキップされてバックフィルが0件になる

つまり「このfeed URLに記事がないfeedはあるか？」という問いに対して、「たまたま見つけた1つのfeedに記事があるか？」という間違った問いに答えていた。

### レガシーモード（single-DB）との乖離

レガシーモードでは4テーブルJOINの `NOT EXISTS` 句で「空フィードのfeed_idを直接取得」していた。これはpush-down anti-joinパターンそのもので、正しく動作していた。

## Decision

**`GetFeedID` + `CheckFeedHasArticles` の2段階RPCを、`GetEmptyFeedID` 1段階RPCに統合する。**

### SQL（push-down anti-join）

```sql
SELECT f.id
FROM feeds f
JOIN feed_links fl ON f.feed_link_id = fl.id
WHERE fl.url = $1
AND NOT EXISTS (SELECT 1 FROM articles a WHERE a.feed_id = f.id AND a.deleted_at IS NULL)
LIMIT 1
```

feed URLを受け取り、記事を持たないfeed IDを1つ返す。全feedに記事がある場合は空文字を返す（= バックフィル不要）。

### 検討した代替案

| パターン | 概要 | 判断 |
|---------|------|------|
| **Push-down anti-join** | フィルタをalt-dbにpush downし、NOT EXISTS句でanti-join | **採用** — レガシーと同じロジック、1 RPC |
| **Batch set difference** | IDセットを一括送信し、不在分だけ返す | 適用可能だが、1 feed URLにつき1 RPCで十分 |
| **Bloom filter** | 既存記事のfeed_idセットをBloom filterで保持 | 過剰設計。feed数が数万なのでin-memoryセットで十分 |
| **GetFeedID修正（全feed_id返却）** | 全feed_idを返してクライアント側でフィルタ | N+1問題が残る |

### 実装の概要

#### alt-backend

- **Proto**: `CheckFeedHasArticles{Request,Response}` を `GetEmptyFeedID{Request,Response}` に置換
- **Port**: `CheckFeedHasArticlesPort` → `GetEmptyFeedIDPort`（引数が `feedID string` → `feedURL string`）
- **Driver**: push-down anti-join SQLを持つ `GetEmptyFeedID` メソッドを新規作成
- **Gateway/Handler**: メソッド置換、`WithBackfillPorts` の型変更

#### pre-processor

- `FetchInoreaderArticlesForEmptyFeeds` 内の2段階ロジック（`getFeedID` → `checkFeedHasArticles`）を `getEmptyFeedID` 1段階に統合
- キャッシュも `feedURL → feedID` と `feedID → hasArticles` の2マップから `feedURL → feedID` の1マップに簡素化

## Consequences

### Pros

- **正しい動作**: 「空フィードのfeed_id」を直接返すため、1つのfeed URLに複数feedがある場合でも正しく空フィードを選択する
- **N+1解消**: 2 RPC（`GetFeedID` → `CheckFeedHasArticles`）が1 RPC（`GetEmptyFeedID`）に
- **レガシーとの一貫性**: single-DBモードと同じpush-down anti-joinパターン
- **コード簡素化**: pre-processor側のキャッシュが2マップ→1マップ、ロジックがシンプルに

### Cons / Tradeoffs

- **`CheckFeedHasArticles` の汎用性喪失**: 単独で「feedに記事があるか」を問い合わせるRPCがなくなった。現時点ではバックフィル以外の用途がないため問題ないが、将来必要になれば再追加する
- **SQLの複雑性**: 3テーブルJOIN + NOT EXISTSサブクエリのため、単純なSELECT EXISTSより負荷が高い。ただしバックフィルはバッチ処理でありfeed URLごとにキャッシュされるため実運用上の影響は軽微
