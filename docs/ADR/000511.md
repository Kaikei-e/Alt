# ADR-000511: リモート予約状態を明示化して解放保証と観測性を追加する

## Status

Accepted

## Date

2026-03-21

## Affected Services

- news-creator (Python)

## Context

[[000510]] により、BE 要約の配分戦略は「空いた実行先に次を渡す」方式へ変わった。
この方式では、リモート実行先が idle か busy かを service 側が正しく把握できないと、以下の障害が起きやすい。

1. 失敗や例外の経路で予約が解放されず、busy のまま固着する
2. retry cascade の途中で状態が崩れ、同一実行先へ過剰に再送する
3. 運用側から見ると「なぜ local fallback したのか」を判別できない

従来の `RemoteHealthChecker` は生死監視と簡易的なラウンドロビン選択が中心で、予約中かどうか、いつ割り当てられたか、いつ完了したかを表現していなかった。
idle-aware scheduling を安全に運用するには、health だけでは不十分である。

## Decision

**リモート実行先ごとに予約状態を明示的に管理し、成功・失敗・未使用解放のすべてで状態遷移を保証する。**

追加する管理項目は以下とする。

- `busy`
- `in_flight_count`
- `last_assigned`
- `last_completed`

追加する動作は以下とする。

1. dispatch 前に idle な healthy 実行先を `acquire` し、その時点で予約状態へ遷移させる
2. remote 実行成功時は `mark_success` で完了時刻を更新しつつ解放する
3. remote 実行失敗時は `mark_failure` で failure 状態更新と解放を同時に行う
4. context を抜けるまでに実際の dispatch が発生しなかった場合でも、予約を必ず解放する
5. status 出力に予約状態を含め、busy による fallback と unhealthy による fallback を区別可能にする

## Consequences

### Pros

- 例外や retry の経路でも busy 固着を防ぎやすくなる
- idle-aware scheduling の正しさを state machine として表現できる
- fallback 理由や各実行先の利用状況を運用で観測しやすくなる

### Cons / Tradeoffs

- `RemoteHealthChecker` の責務が増え、実装の状態管理は round-robin 時代より複雑になる
- status 出力とログ項目が増えるため、監視やテストの追随が必要になる
- `in_flight_count` は将来の拡張余地を含む設計であり、現時点では同時実行数 1 を前提に運用する
