# ADR-000479: Recall candidate read path を articles fallback で補完する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-backend (Go)

## Context

[[000475]] により Recall Rail の read path は `recall_candidate_view` と `knowledge_home_items` を結合し、Recall candidate に `KnowledgeHomeItem` を read-time enrichment する構成になっていた。しかし実運用では、Recall candidate 自体は存在しても `knowledge_home_items` 側の投影が未生成または欠落しているケースが残っていた。

この状態では backend は `RecallCandidate.item = nil` のまま返し、frontend は [[000476]] で表示契約を整えていても `article:<uuid>` 形式の生 key しか表示できない。特に次の条件で再発しやすい。

- projection lag により recall projection と home projection の到着順がずれる
- 再投影・部分復旧の途中で `recall_candidate_view` の方が先に可視化される
- 古い article が Recall candidate に残っているが、Home item としては投影されていない

Recall Rail は「何を思い出させたいか」を出す UI であり、summary や why が未完成でも最低限 title と link を返せないと UX が破綻する。一方で、read path のために Home projection の存在を前提条件にすると、append-first / rebuildable projection の設計意図にも反する。

## Decision

**Recall candidate の read path は `knowledge_home_items` を第一候補としつつ、欠落時は `articles` を fallback source として最小限の `KnowledgeHomeItem` を組み立てる方針を採用する。**

具体的には以下を採用する。

1. `GetRecallCandidates` は従来どおり `knowledge_home_items` を優先する。
2. `knowledge_home_items` が見つからない場合だけ `articles` を参照し、title / url / published_at を用いて最小限の item を生成する。
3. fallback は article candidate のみに限定し、`item_key` から article ID を抽出できる場合だけ `articles` に JOIN する。
4. UUID 変換は無条件 cast ではなく、article key の形式を確認した上で行う。形式不正の key は query error にせず、その candidate は `item=nil` のまま返す。
5. fallback item では summary / tags / why を再構築しない。`SummaryState` は `missing` とし、read model にない情報を推測で埋めない。

この決定により、Recall Rail は Home projection の完全性に依存せず、article の基礎情報だけで最低限の表示を維持する。

### 検討した代替案

- `knowledge_home_items` がない candidate を丸ごと非表示にする  
  Recall candidate 自体は正当でも、projection gap だけで UI から消える。Recall の優先順位が失われるため不採用。

- `articles` から summary / why まで再構築する  
  Home projection の責務を read path に持ち込み、再投影可能な read model の境界を崩すため不採用。

- `item_key` を常に UUID cast する  
  不正 key や旧データで query failure を引き起こすため不採用。

## Consequences

### Pros

- `knowledge_home_items` が欠けていても Recall Rail が title と link を表示できる
- projection lag や部分復旧時でも Recall Rail の UX が劣化しにくくなる
- article key の形式不正を query error にしないため、read path 全体の耐障害性が上がる
- Home projection の責務を侵食せず、fallback を article の最小表示に限定できる

### Cons / Tradeoffs

- fallback item には summary / tags / why がないため、表示情報は KHI 由来 item より貧弱になる
- recall candidate の item shape が「完全 item」と「最小 fallback item」の 2 段階になる
- article item 以外の recall candidate にはこの fallback を使えない
- `articles` 参照を追加するため、read query の複雑さは増える
