# ADR-000930: Knowledge Loop の UI 表層を spotlight+bench に戻し EvidenceRef.Label を Kind から分離する

## Status

Accepted

## Date

2026-05-28

## Affected Services

- alt-frontend-sv (TypeScript / Svelte 5) — `/loop` NOW プレーンが workspace spotlight の entry を二重描画していた点を foreground filter で解消。`WhyTypography` が `EvidenceRef` を kind ラベル付きで表示し raw refId を可視テキストから外す
- knowledge-sovereign (Go) — `appendRefIfPresent` の interleaved 文字列 API を `evidenceCandidate` 構造体ベースの `appendRef` に置換、`labelToEvidenceKind` を撤去、`sanitizeLabel` を新設、6 enricher call sites を rewrite

## Context

2026-05-27 にユーザが `/loop` のスクリーンショットを提示し、ADR-000929 (SSE 配信路 4 層修復) を反映した後でも UI 表層に 2 つの欠陥が残っていることが分かった。

### 症状 1: 同一 entry の操作 UI が 2 段に並ぶ

`/loop/+page.svelte` は ADR-000831 の OODA spotlight 設計に従い、上部に `.ooda-workspace` セクションで `activeEntry` (= `activePlaneEntries[0]`) を独立に表示する一方、下の `LoopPlaneStack` が NOW プレーンを丸ごと描画する。NOW プレーンの snippet (旧 `+page.svelte:933-954`) は `{#each foreground as entry}` で foreground 配列を filter せずに iterate していたため、`foreground[0]` = `activeEntry` が:

- 上の workspace ペインで `<h2>{activeEntry.whyPrimary.text}</h2>` と Decide 用の `<ol class="decision-options">` (I/II/III intent) + `OPEN`/`ASK` の `.workspace-command` で表示
- 下の `.foreground-tiles` で `LoopEntryTile` の `.cta-row` (REVISIT/ASK/SNOOZE/DISMISS/I GOT THIS) として再表示

の二重露出になっていた。設計意図 (spotlight + bench) との差分。

### 症状 2: EVIDENCE 欄に raw UUID が露出

同じスクリーンショットの EVIDENCE 欄は `8ddee42d-68d1-4dcd-beb6-230f0fad4674 — summary` のような表示。`WhyTypography.svelte` の `<span class="why-refs-id">{ref.refId}</span>` がそのまま可視テキストとして UUID を吐き、その横に `{ref.label}` が `summary` / `article` 等を出していた。

根本原因は projector の `enricher.go` で、`appendRefIfPresent(acc, ...pairs)` という interleaved `(refID, label-string)` 可変長引数 API が:

- `Label` フィールドに **kind を表す文字列** (`"summary"`, `"article"`, `"tags"`, `"previous_summary"`, `"new_summary"`, `"what_changed"`, `"conversation"`, `"open_event"`, `"dismiss_event"`, `"entry"`, `"previous_entry"`, `"new_entry"`) を詰めていた
- `Kind` enum も `labelToEvidenceKind(label)` を通して同じ文字列から derive していた

つまり 1 つの文字列が「人間可読 Label」と「routing 用 Kind」の二役を兼ねていた。proto schema には `label` (human-readable) と `kind` (enum) が別フィールドで定義されているのに、projector が両者を kind 文字列で埋めていたため、FE は kind を二重に rendering (UUID — kind) してしまっていた。

## Decision

### Fix A: `/loop` NOW プレーンで workspace spotlight を除外

`alt-frontend-sv/src/routes/(app)/loop/+page.svelte` の NOW プレーン snippet を最小編集する。

```svelte
{#each foreground.filter((e) => e.entryKey !== activeEntry?.entryKey) as entry, i (entry.entryKey)}
```

ADR-000831 の "spotlight + bench" を素直に表現する。`foreground.length === 0` の `EmptyNow` 分岐は据え置く (foreground 全空 = workspace も空 = EmptyNow が正)。CONTINUE / CHANGED / REVIEW プレーンは現状維持。`activePlaneKey` が他プレーンの時は `activeEntry` も他プレーン先頭になり、NOW プレーンの filter は entryKey 不一致で除外なしに自然落ちする。

ソースガード (`page.source.spec.ts`) に「foreground を `.filter(...activeEntry?.entryKey...)` で渡している」regex を追加し、再導入を機械的に防ぐ。

### Fix B: `EvidenceRef.Label` を `Kind` から構造的に分離 (canonical, reproject-safe)

`knowledge-sovereign/app/usecase/knowledge_loop_projector/enricher.go` を以下に置き換える。

1. **Helper の型化**: `appendRefIfPresent(acc, ...pairs)` の interleaved string API を撤去し、`evidenceCandidate { RefID, Label, Kind }` 構造体ベースの `appendRef(acc, refs ...evidenceCandidate)` に置換。`Label` は人間可読、`Kind` は `sovereignv1.EvidenceKind` の enum 値、`RefID` 空はスキップ。
2. **`labelToEvidenceKind` を削除**: kind は呼び出し側が `EvidenceKind_EVIDENCE_KIND_SUMMARY` / `_ARTICLE` / `_UNSPECIFIED` を明示する。文字列→enum の暗黙変換が不要になる。
3. **`sanitizeLabel`**: 既存 `sanitizePlainText` と同じ `stripAngleSpans` を再利用しつつ上限を `maxLabelBytes = 80` に詰めた専用 sanitizer を新設。改行除去 + UTF-8 セーフ truncate。Label は FE で 1 行に収まる長さに統一する。
4. **6 enricher call sites を rewrite**: `enrichSummaryVersion` / `enrichHomeItemsSeen` / `enrichHomeItemAsked` / `enrichHomeItemOpened` / `enrichSuperseded` / `enrichHomeItemDismissed` の各 call site を構造体引数に書き換える。`p.ArticleTitle` が event payload にあれば ARTICLE-kind ref の `Label` に `p.ArticleTitle` を入れる (sanitize 経由)。それ以外の ref (Summary / TagSet / Conversation / EventID / EntryKey 系) は当面 `Label=""` で出す — FE 側 (Fix C) が kind ラベルにフォールバックする。
5. **legacy assertion の移行**: 旧 `evidence_kind_test.go` (`TestLabelToEvidenceKind` / `TestAppendRefIfPresent_TagsKindFromLabel`) と `enricher_test.go:309` の `Label == "what_changed"` 期待値はすべて新 `Kind` 起点に書き直し。`TestEvidenceRef_LabelDoesNotLeakKindStrings` を新規追加し、6 event 経路で `Label` がいかなる kind 文字列にも一致しないことを table-driven で固定する。

### Fix C: `WhyTypography` で raw refId を可視テキストから外す (defense-in-depth)

`alt-frontend-sv/src/lib/components/why/WhyTypography.svelte` を以下に変更する。

- `WhyEvidenceRef` interface に `kind?: EvidenceKindKey` を追加し、`EVIDENCE_KIND_LABELS` で `Article` / `Summary` / `Web` / `Reference` の表示名を持つ。
- evidence と counter-evidence の `<li>` を `<span class="why-refs-kind">{displayKindLabel(ref.kind)}</span>` で始め、`ref.label` があれば `· {ref.label}` を続ける。**raw refId は可視テキストから完全に外す** (`data-ref-id` 属性に格納して routing 経路だけが触れる)。
- ソースガード (`WhyTypography.spec.ts`) に「`<span class="why-refs-id">{ref.refId}</span>` というパターンを許さない」「`ref.kind` を参照している」regex を追加。

これで backend Fix B のロールアウトが段階的でも、既存 row が reproject される前でも、raw UUID は画面から消える。

### 検討した代替案

- **Fix A を CONTINUE / CHANGED / REVIEW にも適用**: ユーザが他タブに切り替えた時の二重描画も同じパターンで防げるが、現状の screenshot は NOW のみが対象。各プレーンのレンダラ (ContinueStream / ChangedDiffCard / ReviewDock) は独自の semantics を持つので、影響が出るかは個別検証が必要。Out of scope。
- **`EvidenceRef.Label` をそのまま FE で hide する display-layer only fix**: WhyTypography だけ書き換える案。proto/projector を触らずに済むが、`Label` の意味が「kind の文字列複製」のまま残り、Augur citation rail や他の consumer (将来) を巻き込む技術的負債になる。projector を canonical state に直すほうが reproject-safe。
- **`Label` を event payload からの hydration ではなく external lookup で埋める**: `knowledge-sovereign` から alt-db の articles テーブルを直引きする案。`docs/services/knowledge-sovereign` の "No shared DB access" 不変条件に違反するため却下。
- **proto schema に title / excerpt を別フィールドで追加**: `Label` を捨てて `ArticleTitle` / `SummaryExcerpt` を proto に増設する案。proto 変更は consumer (alt-frontend-sv / alt-backend / rag-orchestrator) すべての rebuild が必要で過剰。既存 `Label` フィールドを本来の意味 (human-readable) で使えば足りる。

## Consequences

### Pros

- `/loop` 画面で同一 entry が二重に CTA を持つ「重複ボタン」UX が解消。spotlight + bench に整理され、ユーザは「上で操作する」「下で他候補を見る」を直感的に区別できる。
- EVIDENCE 欄から raw UUID が消え、`Article · タイトル` / `Summary` 等の人間可読 row になる。reproject 前の旧 row でも FE フォールバックで kind ラベルが出るため、画面破綻なく段階移行できる。
- `Label` と `Kind` が proto schema 通りに分離された。今後 Augur citation rail や Loop tile UI が `Kind` で純粋に routing でき、`Label` を表示にのみ使える契約が型レベルで担保された。
- `sanitizeLabel` 80 byte 上限で長すぎる article title が 1 行に収まる。`stripAngleSpans` 経由なので XSS regression リスクは既存ポリシーと同等。
- TDD で RED → GREEN を 3 fix それぞれ commit 分割 (6 commits)。`TestEvidenceRef_LabelDoesNotLeakKindStrings` が将来の再導入を機械的に防ぐ。

### Cons / Tradeoffs

- 既存 `knowledge_loop_entries` の `evidence_refs` JSONB は旧 enricher で書かれた "kind 文字列を Label に入れた" 形のまま残る。Fix C のフォールバックで画面は壊れないが、Label として excerpt が出るのはデプロイ後の reproject 完了行のみ。`docs/runbooks/knowledge-loop-reproject.md` に従って projection version bump → reproject を本番反映の同 PR サイクル内で実施する必要がある。
- ARTICLE 以外の kind (Summary / Tags / Conversation / Event) では現状 `Label=""` を出す。将来 SummaryVersionCreated payload に excerpt フィールドを追加すれば Summary ref の Label を埋められるが、proto / 上流 emitter 変更が必要なため本 ADR の範囲外。
- Fix A の filter は NOW のみ。CONTINUE / CHANGED / REVIEW タブで同じ二重露出が出ても今回は手当てしない (該当 screenshot がないため)。発見次第別 ADR で同パターンを波及させる。
- `labelToEvidenceKind` を撤去したので、第三者が「string → enum」変換を期待して `appendRef` を呼び出した場合に kind が `UNSPECIFIED` のまま流れる可能性がある。新 helper のシグネチャを `evidenceCandidate{...}` に強制したのでコンパイラが落とすが、レビュー時に kind の明示を必ず確認する。
- ブランチ `test/hurl-coverage-expand` 上にコミットを積んだ。本来 main 直系の bugfix だが、現作業ブランチに沿わせた。デプロイ前に main へマージするか cherry-pick する。
