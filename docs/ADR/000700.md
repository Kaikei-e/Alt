# ADR-000700: Acolyte パイプラインの構造安定化と品質改善を実施する

## Status

Accepted

## Date

2026-04-11

## Affected Services

- acolyte-orchestrator (Python)

## Context

[[000698]] および [[000699]] で XML DSL 移行が完了したが、v6 パイプライン実行で深刻な退行が発生した。72 分かけて 310 文字しか生成できず、**LLM トークンの 88% が thinking に浪費**されていた（eval_count=67,517 のうち有効トークンは 7,973）。

根本原因は Ollama の thinking mode が `num_predict` 予算を全消費する問題。Writer は `/api/generate` に LONGFORM リクエストを送信していたが、Ollama の `/api/generate` は Qwen3.5 で `think=false` を無視する（ollama/ollama#14793）。`/api/chat` は top-level `think=false` を正しく処理するため、エンドポイント切り替えが必要だった。

加えて、以下の品質課題が未解決だった:

- FactNormalizer の confidence が float（0.0-1.0）で、ローカル reasoning model では精密な小数出力が不安定
- Executive Summary が LLM 自由作文に依存し、thinking exhaustion で全文切断される
- Writer/SectionPlanner の prompt に `[クレーム]` 等のメタ語彙が混入し、最終出力に漏れる
- Citation が UUID 形式のまま残存し、記事タイトルへの解決が未実装
- Critic FM8（Conclusion-Analysis 重複検出）が bigram Jaccard のみで、cross-claim synthesis を考慮しない

## Decision

report.md の改善方針に従い、6 段階の構造安定化と品質改善を実施する。

### Phase 0: Writer thinking exhaustion fix（P0 前提条件）

OllamaGateway に `_generate_chat_freetext()` メソッドを追加し、LONGFORM および STRUCTURED（format なし）の呼び出しを `/api/chat`（format なし）+ top-level `think=false` 経由に切り替える。Settings に `longform_think: bool = False` を追加。

これにより `/api/generate` の `think=false` 無視問題（ollama/ollama#14793）を回避し、Gemma4・Qwen3.5 両方で thinking token の消費を確実に抑制する。

### Phase 1: FactNormalizer confidence banding

`FactNormalizerOutput.confidence` を `float` から `Literal["low", "medium", "high"]` に変更。XML DSL の `<confidence>` タグも `low|medium|high` の離散値に変更。

downstream の ranking（`_rank_facts_for_synthesis`）では `confidence_to_score()` ヘルパーで `low=0.3, medium=0.6, high=0.9` に写像する。生成責務と評価責務の分離により、ローカル model の出力安定性が向上する。

### Phase 2: Executive Summary を deterministic renderer に移行

`ExecutiveSummaryRenderer` クラスを新設し、Writer の ES パスを LLM 呼び出しから deterministic rendering に差し替える。accepted claims の claim テキストと numeric_facts を連結して日本語サマリを生成する。

LLM 呼び出しゼロのため truncation リスクがなく、完走を保証する。

### Phase 3: SectionPlanner / Writer prompt の全面清掃

- `クレーム` → `分析ポイント`（PARAGRAPH_WRITER_PROMPT）、`統合的な知見`（CONCLUSION_PARAGRAPH_PROMPT）に置換
- `計画済みクレーム` → `計画済み分析ポイント` に統一
- CONCLUSION_PLANNER_PROMPT を `<data>` タグで入力データを先頭配置し、指示を numbered steps に変更
- 内部メタ語彙（claim, section, evidence）は XML タグ名に閉じ込め、出力テキストに漏れない構造に

### Phase 4: Citation resolver（SourceMap）の導入

`SourceMap` クラスを新設（`acolyte/domain/source_map.py`）。UUID source_id を `S1, S2, ...` の短い安定 ID に写像する。

- Curator ノードが curated evidence から source_map を構築し state に格納
- Writer は短縮 ID のみ参照
- Finalizer ノードで `[S1]` → `[記事タイトル]` に解決する `resolve_citations()` 関数を追加

### Phase 5: Critic FM8 の複合判定化

`detect_conclusion_analysis_duplication()` を拡張し、bigram Jaccard overlap に加えて cross-claim synthesis ratio を評価する。

- `_claim_synthesis_ratio()`: Conclusion paragraph が参照する analysis claim_id の集合を検査し、複数 claim を横断する paragraph の割合を計算
- 複合判定: `overlap > 0.20 AND synthesis_ratio < 0.5` → blocking、`overlap > 0.20 AND synthesis_ratio >= 0.5` → warning

## Consequences

### Pros

- **thinking exhaustion 解消**: Writer が `/api/chat` + `think=false` を使うことで、全トークンが出力に使われる。v6 の 310 文字 → 3,000+ 文字への改善が期待できる
- **ES 完走保証**: deterministic renderer により truncation リスクゼロ
- **confidence 安定化**: 離散 band により LLM の出力ブレが減少。downstream の ranking 品質が向上
- **メタ語彙漏れ根絶**: `[クレーム]` 等が最終出力に混入しなくなる
- **Citation 解決**: UUID 漏れと `[1]` 未解決が SourceMap + resolve_citations で根本解決
- **FM8 精度向上**: cross-claim synthesis を考慮することで、synthesis としての結論が不当にブロックされなくなる

### Cons / Tradeoffs

- **ES の表現力低下**: deterministic renderer は LLM ほど自然な文章を生成しない。要点の網羅と完走を優先した設計判断
- **`/api/generate` の非使用化**: LONGFORM と STRUCTURED（XML DSL）が `/api/chat` に統一されたため、`/api/generate` は backward compat fallback（mode=None）でのみ使用。将来的に `/api/generate` の `think=false` 修正が入った場合の再評価が必要
- **SourceMap の状態管理**: state に `source_map` が追加されたため、LangGraph checkpoint のサイズが微増する
