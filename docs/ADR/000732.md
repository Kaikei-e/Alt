# ADR-000732: recap-subworker の Phase 4 VO 導入と Phase 5 HTTP タイムアウト堅牢化

## Status

Accepted

## Date

2026-04-15

## Affected Services

- recap-subworker (Python 3.14 / FastAPI)

## Context

[[000726]] の 6 フェーズ計画のうち、Phase 4（生 `dict[str, Any]` を Pydantic VO に置換）と Phase 5（async 堅牢化）を同一セッションで続けて実施する。Phase 3A ([[000731]]) で Clean Architecture 境界 guard が CI gate 化された状態から、「内部 dict の型安全化」と「HTTP 呼び出しのタイムアウト堅牢化」という 2 つの独立したテーマを小さなスコープで片付け、Phase 6 (test hardening) の前に技術的負債を圧縮する。

### Phase 4 の動機

`recap_subworker/services/run_manager.py::_serialize_diagnostics` は `EvidenceResponse.diagnostics` から手動で `dict[str, Any]` を組み立てていた。このままでは:

- 新規フィールド追加時に受け手側（Pact consumer / DB column）と serializer の片方だけ変更するリスクがある
- `None` フィールドの有無を都度 `if is not None` で書き分けるため、条件漏れが潜む
- ユニットテストで serializer の挙動を検証しにくい

### Phase 5 の動機

`recap_subworker/services/learning_client.py` は `httpx.Timeout(timeout_seconds)` のようにスカラー 1 値を渡していた。httpx はこれを 4 ステージ（connect/read/write/pool）すべてに同じ値で展開するため、DNS / TCP が詰まった場合にも read 予算まで使い果たしてから失敗する。さらに `asyncio.timeout` 経由の外郭キャンセルが無く、サーバが黙ったまま応答しないときに request が永遠に宙吊りになる可能性があった。

## Decision

以下を採用する:

### Phase 4

1. `recap_subworker/domain/value_objects.py` に **`ClusterDiagnostics`** を追加する:
   - Pydantic v2 `BaseModel`, `model_config = ConfigDict(frozen=True, extra="forbid")`
   - `dedup_pairs` / `umap_used` / `partial` / `total_sentences` は必須、その他 (`embedding_ms` / `hdbscan_ms` / `noise_ratio` / `dbcv_score` / `silhouette_score` / `hdbscan`) は `None` 許容
   - フィールドに `Field(ge=0)` / `Field(ge=0.0, le=1.0)` の範囲制約を付与
2. `RunManager._serialize_diagnostics` を VO 経由の実装に置換:
   - `ClusterDiagnostics(...)` を構築し、`.model_dump(mode="json", exclude_none=True)` で wire JSON を得る
   - 旧 dict と byte-equal を保証する回帰テストを用意（[[000591]] Pact 契約保持のため）

### Phase 5

1. `services/learning_client.py` の timeout を **4 ステージ個別化**:
   - `connect=2.0`（DNS / TCP 失敗の早期検出）
   - `read=max(caller, 2.5)`（呼び出し側指定、ただし connect より必ず大きい floor）
   - `write=10.0` / `pool=5.0`（定数）
2. POST 呼び出し全体を `async with asyncio.timeout(read * 2)` で包む:
   - 個別 stage timeout の隙間で stall しても deterministic にキャンセル
   - 既存の `httpx.HTTPStatusError` / `httpx.RequestError` ハンドラはそのまま保持
3. 回帰テストで invariants を固定:
   - `test_create_uses_per_stage_httpx_timeout`: `connect < read` を assert、scalar timeout を拒否
   - `test_send_learning_payload_applies_asyncio_timeout`: 応答しない `httpx.AsyncBaseTransport` に対して 2 秒以内にキャンセルされることを検証

## Consequences

### Pros

- `RunManager._serialize_diagnostics` の「新規フィールド追加 → None 条件分岐追加」という忘れやすい 2 段階作業が Pydantic モデル 1 箇所にまとまり、フィールド追加は VO にフィールドを生やすだけで済むようになった。
- `ClusterDiagnostics` は `extra="forbid"` + `frozen=True` なので、未定義キーを紛れ込ませたり後段で破壊的に書き換えたりする経路が閉じられた。テスト側の snapshot 保証が緩まない。
- Phase 5 の connect < read invariant が test で固定されたため、将来「設定忘れで scalar timeout に戻る」退行が自動検出される。
- `asyncio.timeout` 外郭キャンセルにより、`GET /admin/learning` 経路が応答しない recap-worker に対して 2 × read 秒以内に必ず 5xx を返すようになった。TestClient / lifespan の stall 源を 1 つ潰した。
- Phase 1 unblock で導入した conftest autouse override、Phase 2 の per-operation Usecase、Phase 3A の boundary guard、いずれの invariants も一切壊していない。

### Cons / Tradeoffs

- Phase 4 で導入した VO は **`ClusterDiagnostics` のみ**。Plan で言及された `PipelineArtifact` / `RunPayload` / `ClassifyPayload` / `EvidenceRecord` は未着手で、まだ `dict[str, Any]` 経路が残っている (`services/run_manager.py:747-768` 等)。これは Phase 4B 相当として次 Phase に引き継ぐ。
- Phase 5 の対象は `learning_client.py` のみ。Plan で挙がっていた `services/async_jobs.py` / `services/classification_runner.py` の `asyncio.Semaphore` → `asyncio.TaskGroup` 移行、broad `except Exception` の narrow 化、`multiprocessing.Pool` → `ProcessPoolExecutor` 置換の 3 点は別途 Phase 5B 以降で扱う。
- `read = max(caller, connect + 0.5)` という floor は、Phase 1 unblock で autouse override が learning_client を叩かないよう Stub している前提で安全。統合テストが実 recap-worker を叩く場合、短すぎる timeout 指定が自動的に 2.5 秒に持ち上がることを運用メモに残す必要がある。
- Pydantic VO 導入で `model_dump(mode="json", exclude_none=True)` の cost が発生（Python dict 直接構築より僅かに遅い）。ClusterDiagnostics は run 完了時 1 回しか作られないため実質無視できるが、hot path への同様適用時は profile を要確認。
