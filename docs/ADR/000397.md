# ADR-000397: Shared Database の Legacy DB モードを完全廃止し API モードを唯一の動作モードにする

## Status

Accepted

## Date

2026-03-17

## Affected Services

- search-indexer (Go)
- tag-generator (Python)
- pre-processor (Go)
- alt-db (PostgreSQL) — テーブル DROP・ユーザー DROP
- compose/infra — シークレット・環境変数の整理

## Context

[[000241]] で「alt-backend を唯一のデータオーナーにし、他サービスは Connect-RPC 経由でアクセスする」方針を決定し、Phase 0–3 を段階的に実装した。全サービス（search-indexer, pre-processor, tag-generator）は `BACKEND_API_URL=http://alt-backend:9101` で本番稼働しており、API モードへの移行は完了していた。

しかし、**Legacy DB モードのコードが全サービスに残存**しており、以下の問題を引き起こしていた：

1. **デッドコードの保守コスト** — 環境変数 `BACKEND_API_URL` を外すと psycopg2/pgx 経由で alt-db にフォールバックする分岐が各サービスに残り、テストや CI で維持が必要だった
2. **認知負荷** — 開発者は dual-mode の分岐（`if conn is not None`、`UseBackendAPI()` など）を理解する必要があった
3. **セキュリティリスク** — 不要な DB ユーザー（`tag_generator`, `search_indexer_user`, `pre_processor_user`）が alt-db に残存し、最小権限の原則に反していた
4. **不要テーブルの残存** — [[000246]] Phase 5 で予定されていた `inoreader_articles`, `inoreader_subscriptions`, `sync_state`, `api_usage_tracking` の DROP が未実施だった

推定デッドコード量：約 3,500 行（Go + Python）。

## Decision

**Legacy DB モードを完全に削除し、API モードを唯一の動作モードにする。** 併せて alt-db から不要テーブルとユーザーを DROP する。

実行は 6 Phase で一括実施する：

| Phase | 対象 | 内容 |
|-------|------|------|
| 1 | search-indexer | Legacy DB ドライバ・SSL 設定・DB 接続コード削除、pgx 依存除去 |
| 2 | tag-generator | `ArticleFetcher`/`TagInserter`/`DatabaseManager` 削除、`conn is not None` ガード除去 |
| 3 | pre-processor | alt-db 用リポジトリ・ドライバ削除（pre-processor-db は保持） |
| 4 | インフラ | Compose シークレット・環境変数・DB init スクリプト整理 |
| 5 | alt-db テーブル | `inoreader_articles` 等 4 テーブル DROP（Atlas マイグレーション） |
| 6 | alt-db ユーザー | `tag_generator`, `search_indexer_user`, `pre_processor_user` DROP |

### 代替案の検討

- **段階的廃止（サービスごとに別ブランチ）**: 安全だがリリース回数が増え、中間状態でのテスト負荷が高い。全サービスが既に API モードで本番稼働しているため一括実施のリスクは低いと判断した。
- **Feature flag で残す**: `BACKEND_API_URL` を外せば Legacy モードに戻れる安全弁だが、テスト・認知負荷を永続的に負うことになる。API モードの稼働実績が十分あるため不要と判断した。

### 実装の詳細

#### Phase 1: search-indexer

- `driver/database_driver.go`（465 行）を削除
- `config/ssl_config.go` と全テストを削除
- `config/config.go` から `DatabaseConfig` 構造体、`UseBackendAPI()` メソッド、DB 必須バリデーションを削除し、`BACKEND_API_URL` を必須化
- `bootstrap/drivers.go` の `initArticleDriver` を API クライアント生成のみに簡素化（`ctx` 引数・cleanup 関数を除去）
- `go mod tidy` で pgx 依存を除去

#### Phase 2: tag-generator

- `article_fetcher/fetch.py`（438 行）を削除
- `tag_inserter/upsert_tags.py` を `BatchResult` TypedDict のみに縮小（`TagInserter` クラスは API モードの `ConnectTagInserter` が代替）
- `tag_generator/infra/database.py`（`DatabaseManager`）を削除
- `service.py` から `_NullDatabaseManager`・dual-mode 分岐・`_get_database_dsn` を削除し、`create_backend_client()` 失敗時は `RuntimeError` を送出
- `batch_processor.py` から 17 箇所の `conn is not None` ガードとトランザクション管理コードを除去
- `cursor_manager.py` から直接 SQL クエリを除去し、API モードフォールバック（現在時刻）のみに
- `ports.py` の `conn: Connection` を `conn: Any` に変更（ポートインターフェースのシグネチャは互換性のため維持、値は常に `None`）

#### Phase 3: pre-processor

pre-processor には 2 種類の DB アクセスがあり、区別が最重要：

- **alt-db 操作（削除対象）**: `articles`, `feeds`, `article_summaries`, `feed_links` テーブルへのクエリ
- **pre-processor-db 操作（保持）**: `inoreader_*`, `sync_state`, `api_usage_tracking`, `summarize_job_queue` テーブル

実施内容：
- `repository/article_repository.go`, `summary_repository.go`, `feed_repository.go`（Legacy リポジトリ）を削除
- `driver/db_feeds.go`, `db_summaries.go`, `db_batch_operations.go` を削除
- `driver/db_articles.go` から alt-db 関数を削除し、`GetInoreaderArticles`/`GetInoreaderArticlesForBackfill`（pre-processor-db クエリ）のみ保持
- `driver/db_connection.go` から `Init()`（alt-db 接続）を削除し、`InitPreProcessorDB()` のみ保持
- `driver/ssl_config.go` から `NewDatabaseConfig()`（alt-db 用）を削除し、`NewDatabaseConfigWithPrefix()` のみ保持
- `bootstrap/wire.go` で `BACKEND_API_URL` を必須化し、Legacy mode 分岐を削除
- `driver/models.go` に `ArticleWithSummary` 構造体を移設（quality-checker が依存）

#### Phase 4: インフラ

- `compose/base.yaml`: `tag_generator_db_password`, `search_indexer_db_password`, `pre_processor_db_password` シークレット定義削除
- `compose/workers.yaml`: tag-generator の `DB_TAG_GENERATOR_PASSWORD_FILE`, `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_TAG_GENERATOR_USER` 環境変数削除、`postgres_password` シークレット参照削除
- `compose/ai.yaml`: pre-processor の `DB_HOST`, `DB_PORT`, `DB_NAME`, `PRE_PROCESSOR_DB_USER`, `PRE_PROCESSOR_DB_PASSWORD_FILE`, `DB_SSL_MODE` 環境変数削除
- `compose/db.yaml`: `PRE_PROCESSOR_DB_USER`, `PRE_PROCESSOR_DB_PASSWORD_FILE` 環境変数と `pre_processor_db_password` シークレット参照削除
- `db/init/01-create-users.sh`: `tag_generator`, `search_indexer_user`, `pre_processor_user` のロール作成を削除
- `secrets/` ディレクトリから不要なパスワードファイル 3 件を削除

#### Phase 5–6: alt-db クリーンアップ

Atlas マイグレーション 2 本を追加：

1. `20260317000000_drop_pre_processor_legacy_tables.sql` — `inoreader_articles`, `inoreader_subscriptions`, `sync_state`, `api_usage_tracking` を DROP
2. `20260317000001_drop_legacy_service_users.sql` — `REASSIGN OWNED` + `DROP OWNED` + `DROP ROLE` で `tag_generator`, `search_indexer_user`, `pre_processor_user` を安全に削除

`schema.hcl` から該当テーブル定義を除去し、`atlas migrate hash` でチェックサム再計算。

### `conn` パラメータの扱い（tag-generator 固有）

tag-generator の `ArticleFetcherPort`/`TagInserterPort` のメソッドシグネチャには `conn` パラメータが残る。Connect 版ドライバ（`ConnectArticleFetcher`, `ConnectTagInserter`）は `conn` を無視するが、ポートインターフェースの互換性のためシグネチャは維持し、値は常に `None` を渡す。型は `Connection` から `Any` に変更した。

## Consequences

### Pros

- **約 3,500 行のデッドコード削除** — 保守コスト・認知負荷の大幅な削減
- **セキュリティ強化** — 不要な DB ユーザー 3 名と不要テーブル 4 件を alt-db から除去。alt-db への接続は `alt_db_user`（alt-backend）のみに
- **設定の簡素化** — 各サービスの環境変数が減少。dual-mode 分岐が消えたことで設定ミスのリスクが低下
- **CI/テストの高速化** — DB モック・DB 接続テストが不要に。tag-generator は psycopg2 への依存が実質ゼロに
- **アーキテクチャの明確化** — alt-backend が唯一のデータオーナーであることがコードレベルで強制される

### Cons / Tradeoffs

- **ロールバック不可** — Legacy DB モードのコードが削除されたため、`BACKEND_API_URL` を外して DB フォールバックする手段がなくなる。ただし API モードの本番稼働実績が十分あり、alt-backend 障害時のロールバック手段としては Connect-RPC のリトライ・サーキットブレーカーが機能する
- **テーブル DROP は不可逆** — `inoreader_*` 等のテーブルは alt-db から完全に消える。データ自体は pre-processor-db に存在するため復元は可能
- **既存マイグレーションの GRANT 文が空振り** — 過去のマイグレーション（`grant_tag_generator_permissions.sql` 等）は存在しないユーザー/テーブルへの GRANT を含むが、Atlas の `--baseline` で通過済みのため影響なし
