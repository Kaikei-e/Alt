# ADR-000850: Knowledge Loop に paper-stack depth UI と Surface Planner v2 contract foundation を additive に追加する

## Status

Accepted

## Date

2026-04-26

## Affected Services

- proto: `proto/alt/knowledge/loop/v1/knowledge_loop.proto` の additive 拡張
- alt-frontend-sv: 新コンポーネント `LoopPlaneStack.svelte` / `loop-plane-keys.ts`、`ChangedDiffCard.svelte` の redline-proof 化、`/loop/+page.svelte` の rewire、`ChangeSummaryData` TS 型と proto-to-data mapper の拡張
- alt-backend / rag-orchestrator / alt-butterfly-facade: proto stub の再生成（buf gen）。挙動変化なし
- canonical contract `docs/plan/knowledge-loop-canonical-contract.md` と runbook `docs/runbooks/knowledge-loop-reproject.md`

## Context

外部レビュー (`fb.md`) で Knowledge Loop は構想実現度 v0.80 と評価された。fb.md が指摘した DEFER UI 接続バグは [[000848]] 後の最新 main で既に解消済みであり、残課題は次の 3 軸に集約された:

1. **Surface Planner**: 現状のプロジェクタ ([[000844]]) は `event-type → bucket` 固定マッピングで bucket を決めている。構想では「ユーザの recent activity / topic affinity / unfinished thread / version drift から bucket を計画」する knowledge-state ベースに格上げする必要がある
2. **Semantic Orient**: enricher ([[000840]]) は event payload only / pure を維持しているが、recap clusters / tag frequency / Augur thread continuity を組み合わせて Why を深化する余地が大きい
3. **Plane-level depth UX**: tile 単位の Z-axis depth は OODA stage 表現として機能している ([[000848]]) ものの、Now / Continue / Changed / Review という plane 単位での Time Machine 体験がまだ無い。fb.md は「現状は奥行きの記号が入り始めた段階」と評価した

これら 3 軸を一気に landed させることはイミュータブル不変条件 ([[000831]] §3, [[knowledge-home-immutable-model]]) と service boundary を踏まえると現実的でない。具体的には:

- Surface Planner v2 は recap-worker / augur から versioned snapshot event を受け取る前提であり、上流サービスの event 発火が landed していない段階で projector を切替えると reproject-safe 不変が崩れる (latest state lookup を踏みやすい)
- ChangedDiff の redline 充実は projector 側の `summary_versions` lookup が landed していない段階では UI 単独でしか実現できない
- WhyMapping の bump は full reproject runbook 経由の運用が前提

そこで Wave 1 として、後続作業を additive にスムーズに連結できる「contract foundation」と、UI 単独で完結する depth UX 改善を先行させる。

## Decision

Knowledge Loop に対して以下を Wave 1 として additive に追加する。proto / read model / 永続化スキーマの変更は **既存 consumer を一切壊さない** という制約で設計し、Wave 2 以降が踏み込む実装ロジックの足場のみを提供する。

### 1. proto canonical contract の additive 拡張

`proto/alt/knowledge/loop/v1/knowledge_loop.proto` に以下を追加。`buf lint` / `buf breaking` がいずれもパスし、4 consumer (alt-backend / rag-orchestrator / alt-butterfly-facade / alt-frontend-sv) は再生成後に build clean を維持する:

- `WhyKind` の v3 拡張: `WHY_KIND_TOPIC_AFFINITY` (=5) / `WHY_KIND_TAG_TRENDING` (=6) / `WHY_KIND_UNFINISHED_CONTINUE` (=7)。projector の WhyMappingVersion を v2 → v3 に bump させる足場
- 新 enum `SurfacePlannerVersion { UNSPECIFIED, V1, V2 }`。bucket 配置を行ったプランナの世代を entry に固定し、bump を full reproject runbook 経由のみに縛る
- `KnowledgeLoopEntry.surface_planner_version` (field 31)。既存の `reserved 21 to 30` を温存するため意図的に reserved range の外に切る。`buf breaking` の reserved-range narrowing ルールに引っかかるのを避ける目的
- `ChangeSummary` を additive 拡張: `added_phrases[]` / `removed_phrases[]` / `unchanged_phrases_count` / `added_tags[]` / `removed_tags[]`。古い consumer は新 field を読み飛ばし、既存 `summary` + `changed_fields[]` のままで描画できる
- 新 message `KnowledgeLoopSurfacePlanRecomputedPayload`。system-only event の足場であり、user-emittable trigger allowlist には含めない

canonical contract `docs/plan/knowledge-loop-canonical-contract.md` には §6.5「Surface Planner versioning」、§11 の WhyMappingVersion v3 行、§16 の受入基準 #13–14 を追記。runbook `docs/runbooks/knowledge-loop-reproject.md` には WhyMappingVersion v3 履歴行と、新設の SurfacePlannerVersion history table、v2 cutover checklist を追記。

### 2. ChangedDiffCard を redline-proof 描画に格上げ

`alt-frontend-sv/src/lib/components/knowledge-loop/ChangedDiffCard.svelte` を以下のように刷新する:

- `change_summary.added_phrases[]` / `removed_phrases[]` / `added_tags[]` / `removed_tags[]` のいずれかが populated されている場合、editorial proof 風の redline を描画する。removed phrase は取り消し線 (`#6e1f1f`)、added phrase は bold underline、tag chips は added (黒) / removed (灰 + 取り消し線) を区別。unchanged tag は表示しない（差分のみ強調）
- 新 fields が全て空の場合は既存 Then / Now diptych に fallback する。これにより projector が Wave 2 で `ChangeSummary` の新 fields を埋めるようになる前から component は安全に出荷できる
- `aria-label` は `ariaSummary(entry)` ヘルパが組み立てる:「Critical priority. Summary changed: 3 phrases revised, 2 tags added」のように、`loopPriorityAriaLabel[entry.loopPriority]` を base にして diff カウントを後段で足す
- `{@html}` は使わない。redline は CSS class でのみ表現し、XSS 経路を増やさない

`ChangeSummaryData` TS 型を proto に揃えて拡張し、`mapProtoEntry` は新 array fields が undefined / 空のときも安全に動くよう defensive にハンドルする。

source-spec 監査 `SurfacePlaneComponents.source.spec.ts` は「`aria-label` で `loopPriorityAriaLabel[entry.loopPriority]` に到達すること」を 2 行の `expect` に分割し、ヘルパ経由のアクセスも許容する。

### 3. LoopPlaneStack — paper-stack depth UI

`alt-frontend-sv/src/lib/components/knowledge-loop/LoopPlaneStack.svelte` と `loop-plane-keys.ts` を新設する。

- Now / Continue / Changed / Review の 4 plane を、`transform-style: preserve-3d` の単一 perspective context (`perspective: 1100px`) の中に重ねた紙束として描画する
- depth state は active / near (1 個隣) / mid (2 個隣) / far (3 個隣) の 4 段で、`translateZ(0/-8/-16/-26px)` + `scale(1/0.965/0.93/0.9)` + saturation + opacity + blur 0.4–0.6px の段差を割り当てる
- inactive plane は body を `max-height:0` で潰し、ヘッダだけが「紙の縁」のように覗く形にする。これで active plane の縦方向スペースを奪わず、stack の存在感も残る
- ink density ramp は `--ink-color` `#0f0f0f → #2a2a2a`、`--paper-bg` `#fafaf7 → #e9e6e0` の 4 段。Alt-Paper モノクロ言語に揃えており、Vaporwave 化はしない ([[feedback_alt_paper_design]])
- 切替トリガは keyboard `[`/`]` と `j`/`k`、および各 plane のタブ。`Cmd`/`Ctrl`/`Alt` 修飾時は OS のショートカットを潰さないよう no-op。`<input>` / `<textarea>` / `[contenteditable]` 中は文字入力を奪わない
- `aria-live="polite"` で plane 切替を「Continue plane, 4 items」のように polite アナウンスする。inactive plane には `inert` を付けて focus を漏らさない
- `prefers-reduced-motion: reduce` で `translateZ` / `scale` / blur をすべて停止し、saturation + opacity のみで active / inactive を区別する 80–120ms cross-dissolve に倒す ([[000831]] §3 reduced-motion invariant 準拠)
- `@media (max-width: 768px)` で stack は flat 化し、active plane のみ vertical 表示、inactive plane は `display:none`。タッチデバイスで gyroscope-driven parallax は実装しない (バッテリ + a11y の理由)
- `+page.svelte` は `bucketEntries` が populated のとき LoopPlaneStack に流し込み、空のときは既存 LoopSurfacePlane + bucket-index を fallback として温存する

stack 自体の depth と既存 OODA stage 由来の per-tile depth ([[000848]]) は同一 perspective コンテキスト下で合成される。`will-change: transform` は active plane にのみ付け、メモリ消費を抑える。

### 検討した代替案と却下理由

- **buf breaking を一旦無視して `KnowledgeLoopEntry` 内 reserved 21 を再利用**: 既存 reservation が将来の隠れた拘束に育つ可能性を過小評価できない。reserved range はそのままに field 31 で切る方が安全
- **`SurfacePlanRecomputedPayload` を user-emittable な TransitionTrigger にも露出**: F-003 (system event の権限昇格) を踏む。BFF で system event を default-deny にしておく方が seguro
- **ChangedDiff の redline を Wave 2 まで延期し、Wave 1 は既存 Then/Now を維持**: UI と projector の段階的 rollout を考えると、UI 側を先行させて projector が新 fields を埋めても OK な状態にしておく方が運用上スムーズ。fallback 経路を残すのでリスクは低い
- **LoopPlaneStack を新規作らず、既存 LoopSurfacePlane を depth 対応に拡張**: LoopSurfacePlane は section band 単位の責務 (一段ずつ縦に並ぶ) で、stack の active 切替や aria-live announcement とは責務が乖離する。component 分離を選ぶ
- **plane 切替に scrolljack を採用**: モバイル / タッチ / a11y のいずれでも問題が大きい。明示 CTA + keyboard に縛る
- **Wave 1 の段階で WhyMappingVersion を v3 へ bump**: projector 側の実装が landed していない状態で bump すると過去 entry の why が中途半端な mapping を踏む可能性がある。runbook 履歴行は予約 (`tbd`) としつつ、bump 自体は Wave 2 の projector 実装と同時に行う

## Consequences

### Pros

- proto / canonical contract / runbook の v3 + planner v2 用の足場が landed したことで、Wave 2 (上流 event 発火 + projector v2 + Atlas migration + diff_calculator) は contract 衝突を起こさずに進められる
- 全 proto 変更が additive であるため、既存の alt-backend / rag-orchestrator / alt-butterfly-facade / alt-frontend-sv は再生成後そのまま稼働する。`buf breaking` と 4 サービスの build clean で確認済み
- ChangedDiffCard は projector が新 fields を埋めるかどうかに関わらず安全に描画でき、UI と projector の段階的 rollout を分離できる
- LoopPlaneStack によって Now / Continue / Changed / Review の Time Machine 体験が tile 単位 → plane 単位に格上げされ、構想実現度が即時に押し上がる
- reduced-motion / mobile fallback / `inert` / `aria-live` / `{@html}` 排除 / keyboard navigation / 修飾キー保護 が全部一発で揃い、a11y baseline を高い位置で固定できる

### Cons / Tradeoffs

- WhyMappingVersion v3 と SurfacePlannerVersion v2 はまだ projector 側が消費していないため、runbook 履歴行は `tbd` プレースホルダになっている。Wave 2 で必ず日付と ADR 番号を埋め直す運用負債を残す
- `ChangeSummary` の新 fields は projector がまだ populate しないため、Wave 1 単独では ChangedDiffCard は常に Then/Now fallback 経路で描画される。redline-proof 化は Wave 2 で `summary_versions` lookup を備えるまで実視できない
- Wave 1 だけでは Surface Planner は依然 `v1` (event-type 固定マッピング) のままなので、fb.md が指摘した「knowledge state ベース」課題は未解消。後続 Wave 2 を打たないとレビューワに対する応答にならない
- `SurfacePlanRecomputedPayload` は system-only event だが、payload 自体が proto に landed したため、悪意ある呼び出しから守る default-deny を BFF / classifier 側で必ず維持する責務が増える ([[000843]] と同方針)
- `KnowledgeLoopEntry.surface_planner_version` (field 31) はリザーブ範囲の外にあるため、将来的に reserved 範囲を 11–30 → 11–31 へ拡張するなどのスキーマ整理を行う場合は注意が要る
- 新 component LoopPlaneStack は既存の LoopSurfacePlane と責務が部分的に重なる。Wave 2 以降で legacy fallback 経路をどこまで残すかの判断が必要
