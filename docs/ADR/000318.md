# ADR-000318: Tag Trail オンザフライタグ生成の包括的修正

## ADR's STATUS

完了

## CONTEXT

Tag Trail はランダムフィードの記事に対して、Request-Reply パターン（Backend → mq-hub → Redis Streams → tag-generator）でタグを動的に生成する機能である。以下6つの原因により、オンザフライ生成が高頻度で失敗していた。

### 1. 生成タグが DB に永続化されない（Critical）

Gateway がタグ生成後に `UpsertArticleTags` を呼ばず、生成結果がメモリ上のレスポンスとしてのみ返却されていた。同じ記事を再表示するたびに再生成が必要となり、レイテンシの増大とリソースの無駄遣いが発生していた。

### 2. FeedID が常に空文字列（Critical）

`ArticleContent` ドメインモデルに `FeedID` フィールドがなく、`FetchArticleByID` の SQL でも `feed_id` カラムを取得していなかった。結果、mq-hub へのリクエストで `FeedID: ""` が送信され、tag-generator 側での DB 永続化（`feed_tags` テーブルは `feed_id` が必須）が常にスキップされていた。

### 3. 30秒タイムアウトが不十分（High）

ML モデル推論（KeyBERT + sentence-transformers）のコールドスタートを含むと、初回リクエストで 30秒を超過することがあった。

### 4. Consumer がビジーで遅延（High）

tag-generator の Redis Streams consumer は `alt:events:articles` ストリーム1本を購読しており、`ArticleCreated` イベントのバッチ処理と `TagGenerationRequested` の即座応答が競合していた。`batch_size=10`, `block_timeout_ms=5000` の設定ではオンザフライ生成の低レイテンシ要件を満たせなかった。

### 5. リトライ機構がない（Medium）

Gateway でのタグ生成呼び出しは1回限りで、一時的なネットワークエラーや Redis の一過性障害で即座に空配列を返していた。

### 6. フロントエンドの状態表示が不十分（Medium）

タグ生成中のローディング表示はあるが、タイムアウトやエラー時のフィードバックがなく、ユーザーには無限ローディングに見えていた。

## DECISION MAKING

3フェーズに分けて段階的に修正を実施した。

### Phase 1: Backend 永続化 + FeedID 修正（Issues 1, 2, 5）

**1-1. `ArticleContent` に `FeedID` 追加**

`domain/article.go` に `FeedID string` フィールドを追加し、`FetchArticleByID` / `FetchArticleByURL` の SQL を `COALESCE(feed_id::text, '') AS feed_id` で拡張した。

**1-2. Gateway リファクタリング**

`fetch_article_tags_gateway` に以下の改修を実施：

- テスト容易性のための `tagGenerator` / `articleDB` ローカルインターフェースを導入
- `generateTagsWithRetry` メソッド追加（最大2回試行、500ms バックオフ）
- 生成成功後に `UpsertArticleTags` を呼び出し DB に永続化
- `article.FeedID` を mq-hub リクエストに伝播
- Upsert 失敗時も生成タグは返却（fail-open 方針の維持）
- `FeedID` が空の場合は Upsert をスキップ（`feed_tags` テーブルの NOT NULL 制約に対応）

**1-3. テスト**

6つのテストケースを新規作成：
- `ExistingTags_ReturnsCached` — DB にタグがある場合は生成をスキップ
- `NoTags_GeneratesAndPersists` — 生成 + 永続化 + FeedID 伝播を検証
- `GenerateFails_RetriesOnce` — 1回目失敗 → リトライ成功
- `UpsertFails_StillReturnsTags` — DB 書込失敗でもタグを返却
- `EmptyFeedID_SkipsUpsert` — FeedID 空で Upsert スキップ
- `AllRetriesFail_ReturnsEmpty` — 全リトライ失敗時に空配列（fail-open）

### Phase 2: 専用ストリーム分離 + タイムアウト改善（Issues 3, 4）

**2-1. mq-hub: Publish 先を `StreamKeyTags` に変更**

`GenerateTagsUsecase` の Publish 先を `StreamKeyArticles`（`alt:events:articles`）から `StreamKeyTags`（`alt:events:tags`）に変更した。これにより、オンザフライタグ生成リクエストが `ArticleCreated` イベントと同一ストリームで競合しなくなる。

**2-2. tag-generator: 専用 tags consumer 追加**

`ConsumerConfig.tags_stream_from_env()` ファクトリメソッドを追加：
- `stream_key`: `alt:events:tags`
- `group_name`: `tag-generator-tags-group`
- `batch_size`: 1（低レイテンシ優先）
- `block_timeout_ms`: 1000（1秒ポーリング）

`main.py` で既存の articles consumer に加え、tags consumer を別スレッドで起動するよう構成した。移行中は旧パス（articles ストリーム経由）も動作するため、ローリングデプロイが可能。

**2-3. ML モデルプリウォーム**

`TagExtractor` に `warmup()` メソッドを追加し、`TagGeneratorService.__init__` から呼び出すようにした。コンテナ起動時にモデルとストップワードを事前ロードすることで、初回リクエストのコールドスタートレイテンシを排除する。

**2-4. タイムアウト延長**

Backend Gateway と mq-hub のデフォルトタイムアウトを 30秒 → 60秒に延長した。

### Phase 3: フロントエンド改善（Issue 6）

- `loadFeedTagsAsync` に 45秒のクライアントサイドタイムアウトを追加
- タイムアウト時：「タグ生成がタイムアウトしました。更新してください。」を表示
- ストリーミングエラー時：「タグの読み込みに失敗しました。更新してください。」を表示
- Refresh ボタンでリトライ可能
- コンポーネント破棄時のタイマークリーンアップ

## RESULTS, EFFECTS

### PROS

- **キャッシュ効果**: 一度生成されたタグは DB に永続化され、同じ記事への再アクセス時は DB ルックアップのみで済む
- **レイテンシ改善**: 専用ストリーム（batch_size=1, block_timeout=1s）により、オンザフライリクエストが articles バッチ処理に阻害されなくなった
- **コールドスタート排除**: モデルプリウォームにより初回推論レイテンシが数秒短縮
- **耐障害性向上**: リトライ機構とタイムアウト延長により、一時的な障害での失敗率が低下
- **UX 改善**: ユーザーはタイムアウトやエラー時に明確なフィードバックとリトライ手段を得られる
- **テスト容易性**: Gateway のインターフェース導入により、外部依存なしでの単体テストが可能に

### CONS, TRADEOFF

- **Gateway の複雑度増加**: リトライ + 永続化ロジックの追加により、Gateway のコード量が約2倍になった
- **Consumer スレッド増加**: tag-generator が2つの consumer スレッドを持つようになり、リソース使用量がわずかに増加
- **タイムアウトの二重管理**: Backend（60s）、mq-hub（60s）、Frontend（45s）の3段階タイムアウトがあり、整合性の維持が必要

```
Phase 1 (alt-backend) → Phase 2 (mq-hub → tag-generator) → Phase 3 (alt-frontend-sv)
```
