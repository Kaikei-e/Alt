# ADR-000506: feed availability reset を full healthy state に揃える

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-backend (Go)

## Context

既存の `ResetFeedLinkFailures()` は `consecutive_failures = 0` だけを更新しており、`is_active` や failure reason のクリアまでは担っていなかった。そのため、以前に失敗や disable を経験した feed を成功で再評価しても、health 状態が fully healthy に戻らない余地があった。

今回、登録成功時にもこの reset 経路を再利用することにしたため、reset の意味論を明確にしておく必要が生じた。単に failure count を 0 にするだけでは「初期 healthy 状態を作る」という契約として弱い。

## Decision

**`ResetFeedLinkFailures()` は failure count のみでなく、feed を full healthy state に戻す操作として扱う。**

今回の決定では以下を採用する。

- UPSERT 時に `is_active = true` を明示する
- `consecutive_failures = 0` を維持する
- `last_failure_at = NULL` と `last_failure_reason = NULL` をクリアする
- 既存行が無い場合は新規 INSERT、既存行がある場合は healthy 状態へ正規化する

### 代替案

| 案 | 評価 |
|---|---|
| failure count だけ 0 に戻す | healthy への復帰契約として不完全なため不採用 |
| reset と revive を別メソッドに分ける | API は明確になるが現時点では過剰で、既存 call site が少ないため不採用 |
| disable 状態を人手でしか戻せないままにする | 一時的失敗からの自動回復を阻害するため不採用 |

## Consequences

### Pros

- reset の意味が「healthy へ戻す」に統一される
- 登録成功時と collector 成功時で同じ health 正規化ロジックを使える
- failure metadata が stale なまま残る不整合を防げる

### Cons / Tradeoffs

- 直近失敗の履歴を `feed_link_availability` 上には残さなくなる
- disabled 状態の解除が成功 fetch と同値になる
- 将来、監査用に failure 履歴を残したい場合は別テーブルやイベント化が必要になる
