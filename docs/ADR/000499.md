# ADR-000499: RAG ローカル優先ルーティングと batch safety-net の競合緩和

## Status

Accepted

## Date

2026-03-20

## Affected Services

- pre-processor (Go) — batch safety-net の競合緩和
- rag-orchestrator (Go) — RAG 回答生成のローカル優先ルーティング
- news-creator (Python/FastAPI) — compose 設定変更による RAG エンドポイント切替

## Context

[[000497]] により BE 要約のリモート分散ディスパッチが安定稼働し、ローカルの負荷は大幅に低下した。しかし運用監視で以下のミスマッチが判明した:

1. **ローカル GPU が遊休状態**: BE はリモートへ 100% ディスパッチされており、ローカル GPU は `0/2 slots` で完全に空いている
2. **RAG がリモートに寄りすぎ**: RAG 回答生成が `AUGUR_EXTERNAL` 経由でリモートの Ollama を使っており、応答レイテンシが `p50 30.5s` と遅い
3. **batch safety-net が無条件で GPU を消費**: `processSummarizationBatch`（5 分間隔）は queue worker の backlog 状態を考慮せず、ローカル GPU を食い続ける可能性がある

[[000497]] の設計思想は「RT / 対話系はローカル優先、BE はリモート優先」だが、RAG のルーティングがこの方針と逆行していた。

## Decision

以下の 3 つの施策を同時に採用する。

### 方針 A: RAG 回答生成をローカル GPU 優先に変更

`AUGUR_EXTERNAL` のデフォルトを `http://news-creator-backend:11435`（ローカル Ollama）に変更し、RAG 回答生成をローカル GPU 優先にする。

- `AUGUR_KNOWLEDGE_MODEL` を `gemma3-4b-8k` に変更し、RT / query expansion と同一モデルを共有
- `OLLAMA_MAX_LOADED_MODELS=1` のままモデル競合なし（同一モデルのため）
- リモートを使いたい場合は `.env` で `AUGUR_EXTERNAL` を明示的にオーバーライド可能

**代替案と評価:**

| 案 | 評価 |
|---|---|
| rag-orchestrator に Decorator パターン導入 | Go 側に新規コンポーネント追加が必要。RAG はリクエスト頻度が低く、分散の必要性が薄い |
| news-creator の DistributingGateway に RAG 除外ルール追加 | RAG 回答生成は rag-orchestrator → Ollama 直通であり news-creator を経由しない。的外れ |
| **compose/config のデフォルト変更（採用）** | コード変更最小。rag-orchestrator は既に `AUGUR_EXTERNAL` を env で受け取る設計のため、デフォルト値の変更のみで完結 |

### 方針 B: BE 要約は ADR-497 のまま remote 優先

変更なし。`DistributingGateway` による BE のリモートディスパッチはそのまま維持する。

### 方針 C: batch safety-net の条件付き deferral

`processSummarizationBatch` に queue worker の backlog チェックを追加し、queue worker が処理中のときは batch を defer する。ただし無期限 skip を防ぐため、30 分ごとの強制 sweep を設ける。

**ロジック:**

```
if queueWorker != nil && batchSweepForceInterval > 0:
    if lastBatchSweep が forceInterval 超過 → 強制実行
    else if HasPendingJobs() == true → defer (return nil)
    else if HasPendingJobs() error → fail-open で実行
SummarizeArticles() 実行
lastBatchSweep 更新
```

- `batchSweepForceInterval = 0` で deferral 完全無効化（feature flag 相当）
- `HasPendingJobs()` エラー時はフェイルオープンで処理続行

## Implementation

### pre-processor: batch safety-net 競合緩和

**変更ファイル:** `handler/job_handler.go`

`jobHandler` 構造体に 2 フィールド追加:

- `lastBatchSweep time.Time` — 最後に batch が実際に実行された時刻
- `batchSweepForceInterval time.Duration` — 強制 sweep 間隔（デフォルト 30 分）

`processSummarizationBatch` に queue backlog チェックを追加。既存の `processQualityCheckBatch` と同じフェイルオープンパターンを踏襲しつつ、強制 sweep タイマーで無期限 skip を防止。

**テスト:** 5 ケース追加（defer / run-when-empty / fail-open / force-sweep / nil-worker）

### rag-orchestrator: ローカル優先ルーティング

**変更ファイル:**

| ファイル | 変更内容 |
|---------|---------|
| `compose/rag.yaml` | `AUGUR_EXTERNAL` デフォルトを `http://news-creator-backend:11435` に変更。`AUGUR_KNOWLEDGE_MODEL` デフォルトを `gemma3-4b-8k` に変更 |
| `internal/infra/config/config.go` | コードレベルのデフォルト値を compose と一致させる |
| `internal/infra/config/config_test.go` | デフォルト値のアサーションを更新 |
| `.env.template` | ローカル優先の設定例とリモートへの切替方法をドキュメント化 |

### モデル共存設計

| パス | モデル | 配置先 |
|------|--------|--------|
| RT（ストリーミング要約） | `gemma3-4b-8k` | ローカル GPU |
| Query expansion | `gemma3-4b-8k` | ローカル GPU |
| RAG 回答生成 | `gemma3-4b-8k` | ローカル GPU（**本 ADR で変更**） |
| BE（バッチ要約） | `gemma3:4b-it-qat` | リモート群 |

ローカル GPU は単一モデル（`gemma3-4b-8k`）のみロードし、`OLLAMA_MAX_LOADED_MODELS=1` の制約内で RT / query expansion / RAG が共存する。BE は [[000497]] のとおりリモートへ逃がすため、ローカル GPU の headroom を圧迫しない。

## Consequences

### Pros

- **RAG 応答レイテンシ改善**: リモート（p50 30.5s）からローカル GPU へ切替えることで、ネットワーク RTT + リモート負荷を排除
- **遊休 GPU の有効活用**: ローカル GPU が `0/2 slots` で放置される状態を解消
- **モデル競合なし**: RT / query expansion / RAG がすべて同一モデル（`gemma3-4b-8k`）を使うため、モデルスワップによる cold start が発生しない
- **batch 競合緩和**: queue worker が忙しいときに batch safety-net が追加で GPU を消費する事態を回避
- **既存動作への影響ゼロ**: BE の `DistributingGateway` ルーティングは変更なし。RAG は `.env` オーバーライドで従来動作に戻せる
- **feature flag 相当**: `batchSweepForceInterval = 0` で deferral を無効化可能

### Cons / Tradeoffs

- **RAG がローカル GPU に集中**: RT / query expansion / RAG が同一 GPU を共有するため、同時リクエストが増えるとスロットが枯渇する可能性がある。ただし RAG のリクエスト頻度は低く、実運用上の影響は限定的
- **RAG 回答品質の変化**: `gemma3-4b-rag`（RAG 特化チューニング）から `gemma3-4b-8k`（汎用）への切替えにより、回答品質が変化する可能性がある。監視して問題があれば `AUGUR_KNOWLEDGE_MODEL` でロールバック可能
- **batch 強制 sweep の 30 分間隔は経験的値**: 最適値は負荷パターンに依存する。運用データを見て調整が必要
- **compose 間の暗黙的依存**: `rag.yaml` が `ai.yaml` で定義された `news-creator-backend` サービスを前提とするが、`depends_on` で明示できない（別 compose ファイルのため）。RAG プロファイルのみ起動した場合は `AUGUR_EXTERNAL` の明示的設定が必要
