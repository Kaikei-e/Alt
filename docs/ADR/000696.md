# ADR-000696: query planner の SearchQueries を retrieval パイプラインに直結し expand-query のフィルタ過剰問題を解消する

## Status

Accepted

## Date

2026-04-11

## Affected Services

- rag-orchestrator (Go)
- news-creator (Python)

## Context

[[000695]] で expand-query プロンプトに `English({count}):` ラベルと `1.` プレフィルを追加し、BM25 マルチクエリ検索を導入した。しかし Gemma4 4B (gemma4-e4b-12k) が `1.` プレフィルに対してプロンプト構造をエコーし、25 行の不要出力を生成する問題が発生した。

この出力は 2 段フィルタチェーンを通過する:

1. **news-creator (Python)**: `_parse_expansion_lines` → `_deduplicate_preserving_order` → `_filter_instruction_leaks` で 25 行 → 1 件
2. **rag-orchestrator (Go)**: `filterExpandedQueries` （長さ・ゴミ・ローマ字・日付・instruction leak・XML タグ・会話メッセージリーク）で 1 件 → 0 件

結果として `expanded_queries: 0`、BM25 は元クエリのみで検索され、ADR-695 で導入した言語横断検索が機能していなかった。

一方、query planner（Gemma4 26B）は高品質なバイリンガル `SearchQueries` を 4 件生成していたが、`result.expandedQueries` にメタデータとして格納されるだけで、retrieval パイプラインの `RetrieveContextInput` には渡されていなかった。

```
query planner → SearchQueries: 4 (JP+EN)
  → FilterSearchQueries() → result.expandedQueries（メタデータのみ）
  → RetrieveContextInput には渡されない ← 設計断絶
  → retrieval pipeline は独自に expand-query 呼出 → 0 件に
```

## Decision

**query planner の SearchQueries を retrieval パイプラインに直接渡し、planner クエリが利用可能な場合は expand-query LLM 呼出をスキップする。** 副次的に news-creator のパーサーバグも修正する。

### 検討した代替案

| 案 | 評価 |
|---|---|
| プロンプト構造の変更（ハイフンプレフィル等） | Gemma4 4B の出力が予測困難。根本解決にならない |
| フィルタチェーンの閾値緩和 | ゴミ出力を通す可能性があり、品質低下リスク |
| JSON 形式での構造化出力 | Gemma4 4B の JSON 能力に依存し、信頼性不足 |
| **planner SearchQueries の直結（採用）** | 高品質クエリ活用、LLM 呼出削減、フィルタ問題回避 |

### 実装概要

**A. Root Fix — planner SearchQueries のパイプライン直結（rag-orchestrator）**

データフロー:

```
buildPromptWithQueryPlanner()
  → FilterSearchQueries(qPlan.SearchQueries) → result.expandedQueries
  → RetrieveContextInput.SearchQueries に設定（新フィールド）
    → StageContext.PlannerQueries に伝播（新フィールド）
      → ExpandQueries(): goroutine A スキップ（B: tag search, C: embedding は継続）
        → sc.ExpandedQueries = sc.PlannerQueries
          → EmbedAndSearch(): bm25Queries() に自動的に流れる
```

変更箇所:

- `StageContext` に `PlannerQueries []string` フィールド追加
- `RetrieveContextInput` と `GraphInput` に `SearchQueries []string` フィールド追加
- `ExpandQueries()` の goroutine A 先頭で PlannerQueries チェック（スキップ時はログ出力）
- `buildPromptWithQueryPlanner()` で `retrieveInput.SearchQueries` に既存の `result.expandedQueries` を設定

Tag search（goroutine B）と Original embedding（goroutine C）はスキップしない。expand-query の LLM 呼出（goroutine A）のみをバイパスする。

**B. Fallback Fix — news-creator パーサー修正**

`_parse_expansion_lines` の番号除去ロジックが単一桁（`trimmed[0].isdigit() and trimmed[1] in ".):"` ）にしか対応していなかった。`10.` 以上の番号付きリストが除去されず、後段フィルタで誤判定される原因となっていた。

正規表現 `^\d{1,3}[.):][ \t]` に置換し、1〜999 の番号に対応した。

加えて、`_parse_expansion_lines` 呼出前に LLM 生出力のデバッグログを追加し、Gemma4 の実出力パターンの可視化を可能にした。

## Consequences

### Pros

- **BM25 展開クエリ復活**: `expanded_queries: 0 → 4`、`bm25_hits: 0 → 115-205` に改善
- **言語横断検索の実質稼働**: `expanded_hits_unique: 0 → 87-124` に改善。ADR-695 の設計意図が初めて正常動作
- **Stage 1 レイテンシ削減**: expand-query LLM 呼出（~3.4s）をスキップ。Stage 1 は 315ms-1.6s に短縮
- **後方互換性維持**: `SearchQueries` 未設定時は従来の expand-query パスにフォールバック。CDC テスト変更不要
- **expand-query の契約変更なし**: news-creator の API は変更なし。呼出をスキップするだけ

### E2E ベンチマーク結果

**retrieval パイプラインの改善（ログ実測値）:**

| クエリ | planner_queries | bm25_queries | bm25_hits | expanded_hits_unique | retrieval_ms |
|--------|----------------|-------------|-----------|---------------------|-------------|
| イラン石油危機 | 4 | 5 | 190 | 87 | 4,268 |
| イラン石油危機 (follow-up) | 4 | 5 | 205 | 124 | 2,140 |
| NVIDIA Blackwell | 4 | 5 | 139 | 109 | 12,470 |
| 全固体電池 | 4 | 5 | 115 | 97 | 2,734 |

全 12 ゴールデンケースで `expand_query_skipped_planner_queries_available` が発動し、planner クエリ 4-5 件が BM25 検索に直接使用された。

**回答品質 (Augur eval 12 ケース):**

| ケース | 結果 | answer_len | citations |
|--------|------|-----------|-----------|
| iran-oil-crisis-causal | OK | 444 | 6 |
| oil-crisis-root-cause | OK | 400 | 4 |
| simple-temporal-query | OK | 735 | 7 |
| comparison-query | OK | 1,109 | 7 |
| ambiguous-is-that-true | OK | 171 | 1 |

回答生成に成功したケースでは平均 5 件の引用を伴う回答を生成。retrieval の改善により、従来は `expanded_hits_unique: 0` だったクエリで 87-124 件のユニークヒットを取得できるようになった。

**注意: Fallback Rate の悪化について**

前回ベースライン (2026-04-06) の Fallback Rate は 16.7% (2/12)、今回は 58.3% (7/12)。ただし今回のビルドには ADR-693（BM25 degraded mode）、ADR-695（BM25 マルチクエリ）、本 ADR-696 の 3 ADR 分の未コミット変更が含まれており、悪化の原因を単一 ADR に帰属させることはできない。

特に `cjk-short-query-bm25`（量子コンピュータ 実用化）は PASS → FALLBACK に回帰しており、`synthesis-broad-topic` 等 4 ケースが answer 生成 → 空応答に悪化。RelevanceGate が retrieved contexts を低品質と判定しているため、retrieval パイプライン全体（マルチクエリ化 + RRF fusion + degraded mode）の複合的な影響を調査する必要がある。

### Cons / Tradeoffs

- **planner 未使用パスは引き続き問題**: legacy パス（`queryPlanner == nil`）では expand-query がフォールバックとして使われ、フィルタ過剰問題が残る。ただしパーサーの多桁番号修正で部分的に改善
- **planner クエリの品質に依存**: planner (Gemma4 26B) が低品質なクエリを返した場合、`FilterSearchQueries` のフォールバック（`resolvedQuery` を返す）に頼ることになる
- **expand-query の段階的廃止が視野に入る**: planner 統合が完了した現在、expand-query エンドポイントの存在意義は fallback パスのみ。将来的に planner の信頼性が確立されれば廃止を検討すべき
- **Fallback Rate 悪化の切り分けが未完了**: ADR-693/695/696 の 3 変更が同時にデプロイされたため、eval の Fallback Rate 悪化 (16.7% → 58.3%) の原因を ADR-696 単体に帰属させるか、他の変更に帰属させるかの切り分けができていない。`cjk-short-query-bm25` の回帰は要調査
