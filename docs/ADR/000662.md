# ADR-000662: Conclusion を専用 contract に分離し Analysis 重複を構造的に防止する

## Status

Accepted

## Date

2026-04-09

## Affected Services

- acolyte-orchestrator (Python 3.14+)

## Context

[[000659]] で `SectionPlannerNode` と `ClaimPlan` を導入し、Writer は構造化された `PlannedClaim` からのみ本文を生成するようになった。[[000661]] で `section_citations` を claim 単位で永続化し、attribution が構造化された。しかし、**Conclusion セクションが Analysis と同一の prompt・同一のロジックで生成されている**問題が残っていた。

具体的な品質問題:

1. **Analysis↔Conclusion 重複**: 品質評価で bigram 重複率 0.32。Conclusion が Analysis の事実をそのまま再叙述している
2. **Conclusion の新規引用率 0%**: Conclusion に独自の洞察・統合判断がなく、Analysis のコピーになっている
3. **section_role 概念の不在**: PlannerNode の出力 outline に `key/title/search_queries` しかなく、セクションの「役割」が型として表現されていない
4. **SectionPlannerNode が全セクション共通 prompt**: Analysis も Conclusion も同じ `SECTION_PLANNER_PROMPT` を使い、extracted_facts から同じ方法で claims を生成
5. **WriterNode が全セクション共通 prompt**: `CLAIM_WRITER_PROMPT` が全セクションに適用され、Conclusion 固有の制約（新事実追加禁止、統合のみ）が構造的に表現されていない

学術的背景:

- **Attribute First, then Generate** (ACL 2024): section ごとの role を明示しないと重複しやすく、sentence planning 分離の方が control しやすい
- **FActScore** (EMNLP 2023): 長文の factuality は atomic fact 単位で扱うべきであり、Conclusion は新事実ではなく既存 facts の統合であるべき

## Decision

**`section_role` を planner に導入し、Conclusion に専用の planner prompt・writer prompt・critic failure mode を設ける。Conclusion は Analysis の accepted claim 群のみを入力とし、raw evidence を直接参照しない。**

### 主要な設計判断

#### 1. Planner に `section_role` を追加（型として役割を固定）

`_PLANNER_FORMAT` の JSON schema に `section_role` (enum: `"analysis"`, `"conclusion"`, `"executive_summary"`, `"general"`) を追加。LLM が返さなかった場合に備え、`_infer_section_role(key, title)` ヒューリスティックで key/title から推論する。Gemma4 26B の structured output は不安定（ollama/ollama#15260）なため、ヒューリスティックフォールバックは必須。

`_default_fallback_sections()` にも `section_role` を明示し、fallback パスでも role が常に存在する状態を保証する。

#### 2. SectionPlannerNode: Conclusion は Analysis claims から plan を生成

`section_role == "conclusion"` の場合:
- `extracted_facts` からではなく、既に構築済みの analysis セクションの `claim_plans` を入力として受け取る
- `CONCLUSION_PLANNER_PROMPT` で synthesis-only claims を生成（新事実禁止、各 claim は 2+ analysis claims を統合）
- ループが sequential であるため、analysis は conclusion より前に処理され、claims は自然に利用可能

`section_role != "conclusion"` の場合は既存ロジック（extracted_facts ベース）を維持。

#### 3. WriterNode: Conclusion に専用 prompt

`section_role == "conclusion"` の場合に `CONCLUSION_WRITER_PROMPT` を使用。このプロンプトは:
- 新事実追加を明示的に禁止
- Analysis の文の再掲を禁止
- 出力を「意味づけ」「リスク」「優先順位」「推奨行動」に限定
- claims ブロックのヘッダーを「統合クレーム:」に変更（通常の「計画済みクレーム:」と区別）

#### 4. CriticNode: FM8 重複検出ヒューリスティック（LLM 不要）

`detect_conclusion_analysis_duplication()` 関数で analysis body と conclusion body の文字 bigram Jaccard 係数を計算。閾値 0.20 超で `FM8_CONCLUSION_ANALYSIS_DUPLICATION` (blocking) を発火。

0.20 を採用した理由: [[000656]] の品質評価で bigram 重複率 0.32 が問題視された。0.20 は重複を早期検出する保守的閾値。文字 bigram は日本語でも単語分割不要で言語非依存。

LLM 呼び出しは追加しない（FM2/FM3 と同様のヒューリスティック）。

#### 5. Pydantic モデルは追加しない

`ClaimPlannerOutput` を再利用する。conclusion 固有の制約（`claim_type == "synthesis"` のみ、等）はプロンプト側で表現する。PlannedClaim の `claim_type` フィールドが既に "synthesis" をサポートしており、新モデルは不要。

### 不採用とした代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| Conclusion 専用ノードを LangGraph に追加 | グラフ配線が複雑化。revision ループとの整合性が煩雑。SectionPlannerNode 内の分岐で十分 |
| LLM で conclusion-analysis 重複を検出 | コスト対効果が低い。bigram Jaccard で十分に検出可能。LLM 呼び出しバジェットを増やさない方針 |
| `section_role` を state の独立フィールドにする | outline が `list[dict]` なので追加フィールドは自然に格納可能。独立フィールドは outline との同期リスクが生じる |
| Conclusion planner 用の専用 Pydantic モデル | `ClaimPlannerOutput` で十分。制約はプロンプトで表現。モデル分離のメリットが薄い |

### Conclusion 分離の動作確認

1. **PlannerNode**: fallback sections に `section_role` 付与。LLM 出力に role がなければ `_infer_section_role()` で推論
2. **SectionPlannerNode**: `section_role == "conclusion"` → analysis claims を入力として `CONCLUSION_PLANNER_PROMPT` で synthesis claims 生成
3. **WriterNode**: `section_role == "conclusion"` → `CONCLUSION_WRITER_PROMPT` で統合判断のみ出力。「統合クレーム:」ヘッダー
4. **CriticNode**: FM8 bigram Jaccard チェックが FM2/FM3 と並列で実行

## Consequences

### Pros

- **Analysis↔Conclusion 重複の構造的防止**: Conclusion は raw evidence を直接参照せず、Analysis の accepted claims のみから synthesis claims を生成。再叙述が構造的に不可能
- **section_role の型化**: セクションの役割が outline dict に明示され、下流ノード（planner/writer/critic）が role に基づいて振る舞いを変えられる基盤が整った
- **LLM コスト追加 0**: FM8 は bigram ヒューリスティック、conclusion planner は既存の N calls に含まれる
- **既存パイプライン構造の維持**: LangGraph のノード追加やエッジ変更なし。SectionPlannerNode/WriterNode 内の分岐のみ
- **テスト充実**: 12 テスト追加で section_role 推論・conclusion 分岐・重複検出をカバー
- **Issue 5 (section contract) の基盤**: `section_role` の導入は Issue 5 の `section contract` 拡張の前提条件を満たす

### Cons / Tradeoffs

- **_infer_section_role ヒューリスティックの限界**: key/title に "conclusion" 等が含まれない場合は "general" にフォールバック。LLM が完全に異なるセクション名を生成した場合は role が正しく推論されない
- **Sequential processing 前提**: SectionPlannerNode のループは analysis → conclusion の順序を前提とする。LLM が outline で conclusion を analysis より前に配置した場合、analysis claims が空になり conclusion も空になる。ただし `_default_fallback_sections` は常に analysis → conclusion の順序を保証
- **FM8 の閾値チューニング**: 0.20 は保守的な初期値。実運用データでチューニングが必要な可能性がある。false positive（同一ドメインの語彙重複）と false negative（パラフレーズ重複）のバランス
- **WriterNode の分岐拡大**: claim-based / legacy / conclusion の 3 パスが WriterNode に存在。今後 executive_summary も専用化すると分岐が増える。将来的にはストラテジーパターンへの分離を検討する余地がある
- **RerunSectionUsecase との非統合**: `RerunSectionUsecase` は legacy の `WRITER_PROMPT` を使用し、conclusion 専用 prompt を使わない。セクション単位再生成では conclusion の品質制約が適用されない
