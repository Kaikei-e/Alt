# ADR-000523: stale pending summarize job の監査と除去を運用導線として固定する

## Status

Accepted

## Date

2026-03-21

## Affected Services

- pre-processor (Go) — `summarize_job_queue` に対する stale pending の count / delete helper を追加する
- scripts — one-shot で実行できる監査・削除 SQL を追加する

## Context

[[000515]] では「summary 保存済みの stale pending job だけを除去する」という運用方針を定めていたが、実装側にはその判断を再利用できる query や運用導線が存在しなかった。

その結果、stale pending の存在を確認できても、毎回 ad-hoc な SQL を手で書かなければならず、年齢だけで削除する誤操作や、summary 未保存 row の巻き込み削除を招きやすい状態だった。

今回の修正では queue admission 側の防御線だけでなく、すでに残っている stale row を安全に減らせることも必要だった。特に `pending` 件数を live backlog に近づけるには、「何件あるかを数える」「対象だけを削除する」「削除条件をコードと運用 SQL の両方で固定する」の 3 点が必要になる。

## Decision

**summary がすでに存在する `pending` row だけを stale pending とみなし、その count と delete を再利用可能な helper / SQL として固定する。**

今回の決定では以下を採用する。

1. `pre-processor/repository` に stale pending の count query を追加する
2. 同じ判定条件で stale pending を削除する delete query を追加する
3. 削除条件は `status = 'pending'` かつ `article_summaries` に対応 row が存在するものに限定する
4. 運用向けに one-shot の SQL スクリプトを追加し、削除前監査・サンプル確認・削除件数確認を一連で実行できる形にする

### 採用しなかった代替案

| 案 | 評価 |
|---|---|
| 古い `pending` を created_at だけで消す | 未要約記事を消す危険があり不採用 |
| 運用 SQL だけ置いて repository helper は持たない | アプリケーション側で同じ条件を再利用できず、判定条件が散逸する |
| `completed` 履歴ごと広く purge する | stale pending の安全除去という目的を超え、別問題の terminal purge と混ざる |

## Consequences

### Pros

- stale pending の判定条件がコードと運用 SQL の両方で固定され、ad-hoc な削除を避けやすくなる
- `pending` 件数を実 workload に近づける cleanup を、機械的に再実行しやすくなる
- `ADR-515` の運用方針を「実行可能な導線」に変換できる

### Cons / Tradeoffs

- delete helper は one-shot cleanup 前提であり、恒常的な自動 purge job にはまだしていない
- `article_summaries` の存在を stale 判定に使うため、summary 保存先との整合性が前提になる
- cleanup 自体は queue の根本原因を止めるものではなく、再発防止は別途 upstream/event semantics に依存する
