# ADR-000913: Knowledge Loop の OODA closure を frontend dwell timer + `EmitActOutcome` RPC で即時化し Surface Planner v2 が 7 日待たず engagement 信号を学習できるようにする

## Status

Accepted

## Date

2026-05-24

## Affected Services

- proto (alt.knowledge.loop.v1)
- alt-backend (Go)
- knowledge-sovereign (Go)
- alt-frontend-sv (TypeScript / Svelte 5)

## Context

[[000908]] §Δ1 は `knowledge_loop.act_outcome.v1` event を導入し、`KnowledgeLoopActed` の **結果** (engaged / deep_engagement / accepted_change / stale_save / no_engagement) を Surface Planner v2 が `ActOutcomeSignal` として bucket placement に反映できるようにした。

ただし `act_outcome.v1` の **producer** は当時 `act_outcome_cron` のみで、これは「7 日経って何の outcome も書かれなければ no_engagement を fallback emit する」run-batch。つまり Surface Planner は **engagement の不在** からしか学習できず:

1. **学習が 7 日遅延する** — engaged な記事も deep_engagement な記事も、7 日経過するまで surface 側に positive feedback が届かない
2. **negative bias** — cron が emit するのは no_engagement のみ。positive な engagement signal を誰も emit しない状態で `ActOutcomeSignal` を引数に取る `decideBucketV2` の Review demotion (≤ -2 で Now/Continue → Review) は片肺の signal で動く
3. **「I got this」相当の積極的 closure が無い** — 構想層 §6 (loop closure) で planned されている epistemic graduation を画面から記録する手段が proto に存在しない

frontend 側で dwell timer / ask turn counter / explicit CTA を計測し、proto 経由で append すれば 7 日遅延を解消できる。だが愚直に新規 RPC を追加すると次の risk が立つ:

- **二重発火** — FE が同じ閉路で複数 event を出すと、event log が太り projector が double-count する。idempotency key が必要
- **cron との namespace 衝突** — FE が engaged を emit し、cron が no_engagement を別途 emit すると、`ActOutcomeSignal` aggregation (engaged=+1, no_engagement=-2) の合計が -1 になり「engagement したのに demote される」逆転バグになる
- **`ACT_OUTCOME_KIND_INTERNALIZED` の扱い** — proto enum に存在せず、dismiss_state の transition で表現すべき。FE-emit 経路に混ぜると意味論が壊れる

## Decision

3 層の wiring を **1 commit** で landed し、上の 3 つの risk を設計で吸収する。

### Layer 1: Proto additive bump

`proto/alt/knowledge/loop/v1/knowledge_loop.proto`:

```proto
service KnowledgeLoopService {
  ...
  rpc EmitActOutcome(EmitActOutcomeRequest) returns (EmitActOutcomeResponse);
}

message EmitActOutcomeRequest {
  string entry_key = 1;
  ActOutcomeKind outcome = 2;
  string client_outcome_id = 3;  // UUIDv7, idempotency key
  google.protobuf.Timestamp occurred_at = 4;  // event-time, not wall-clock
  optional uint32 dwell_seconds = 5;
  optional uint32 ask_turns = 6;
  optional string lens_mode_id = 7;
  reserved 8 to 15;
  reserved "user_id", "tenant_id";  // JWT sourced
}

message EmitActOutcomeResponse {
  bool accepted = 1;
  bool deduplicated = 2;
  int64 event_seq = 3;
}
```

`buf breaking` clean (additive only)。`ActOutcomeKind` enum は既存 ([[000908]]) のまま再利用。

### Layer 2: alt-backend usecase + handler

`EmitActOutcomeUsecase` は:

- **Allowed outcomes**: `ENGAGED` / `DEEP_ENGAGEMENT` / `STALE_SAVE` / `ACCEPTED_CHANGE` のみ
- **Rejected**:
  - `NO_ENGAGEMENT` — cron 専用 namespace
  - `INTERNALIZED` — proto enum に存在しない (dismiss_state transition 経路へ。後続 ADR)
- **dedupe_key**: `knowledge_loop.act_outcome.v1:fe:<user_id>:<entry_key>:<client_outcome_id>` で構築
  - cron は `knowledge_loop.act_outcome.v1:<acted_event_id>:no_engagement` を使うため **prefix `:fe:` で namespace が衝突しない**
  - 同じ `client_outcome_id` での retry は `knowledge_event_dedupes` UNIQUE で `AppendKnowledgeEvent` が `event_seq=0` を返し、handler は `deduplicated=true` で idempotent-OK
- **event payload**: `{ entry_key, lens_mode_id, outcome, observed_at, emitter: "frontend", dwell_seconds?, ask_turns? }`
  - `acted_event_id` は載せない (FE は知らない)。代わりに Layer 3 (cron 互換性) で entry_key 突き合わせを足す

### Layer 3: cron NOT EXISTS の widening

`knowledge-sovereign/app/driver/sovereign_db/read_events.go` の `ListActedEventsWithoutOutcome` を:

```sql
WHERE acted.event_type = 'knowledge_loop.acted.v1'
  AND acted.occurred_at <= $1
  AND NOT EXISTS (
    SELECT 1 FROM knowledge_events AS outcome
    WHERE outcome.event_type = 'knowledge_loop.act_outcome.v1'
      AND (
        outcome.payload->>'acted_event_id' = acted.event_id::text
        OR (
          outcome.user_id = acted.user_id
          AND outcome.aggregate_id = acted.aggregate_id
          AND outcome.occurred_at >= acted.occurred_at
        )
      )
  )
```

FE は `acted_event_id` を持たないため (上記)、subquery を **`(user_id, aggregate_id=entry_key, occurred_at >= acted.occurred_at)` の OR 分岐** で widening する。これで FE が positive な outcome (engaged 等) を emit すれば、cron は同じ acted event に対して no_engagement を fallback しない。

### Layer 4: FE dwell emitter + BFF

`useActOutcomeEmitter.svelte.ts`:

- `createActOutcomeEmitter({ post, now, tickIntervalMs })` factory
- per-entry state: `{ startedAt, askTurns, emitted: Set<Outcome> }`
- 閾値:
  - `engaged` — dwell ≥ 30s で once
  - `deep_engagement` — dwell ≥ 120s **OR** ask turns ≥ 3 で once (満たした方が先に fire)
- `start(entryKey)` で arming、`recordAskTurn(entryKey)` で turn 増分、`stop(entryKey)` で deregister
- `visibilitychange=hidden` で `flush()` (= `tick()`) を 1 回回す → tab 切替時の積み残しを救出。`fetch(..., { keepalive: true })` で送出
- network 失敗時は `emitted.delete(outcome)` で次の tick の retry を許可。server-side dedupe が二重書き込みを保護
- `+page.svelte` の `onEntryOpen` で `start()`、`onAsk` で `recordAskTurn()`、`onDestroy` で `teardown()`

BFF (`/loop/act-outcome/+server.ts`):

- POST body を validate (entry_key / outcome enum / client_outcome_id / ISO occurred_at)
- `locals.backendToken` で `emitActOutcomeForUser` を call
- Connect error → HTTP (`invalid_argument` → 400 / `unavailable` → 502 / `deadline_exceeded` → 504 / `resource_exhausted` → 429)

### 検討した代替案

- **deterministic UUIDv7 を `derive(user, entry, 7d_window_id)` で FE/cron 共有** — 元々の plan agent 提案。cron と FE が同じ UUID を生成して衝突回避するアイデアだが、(a) cron の wall-clock 依存が増える (b) windowing で edge case が増える (c) NOT EXISTS の widening で十分 — の 3 点で却下。
- **dwell timer を IntersectionObserver で view-based に** — 画面に映っていれば dwell とみなす案。ユーザがタブを開いたまま離席するケースで false positive を生む。**Open intent 後の経過時間** に限定する明示的な start() ベースに決定。
- **`INTERNALIZED` も EmitActOutcome に流す** — proto enum を追加すれば可能だが、INTERNALIZED は entry の dismiss_state を flip する transition セマンティクスで、outcome の signal とは独立。次の wave で transition trigger 経由 (TRANSITION_TRIGGER_INTERNALIZE 等) で扱う。本 ADR の scope 外。

## Consequences

### Pros

- **Surface Planner v2 が session 内で学習する** — `decideBucketV2` の `ActOutcomeSignal` が engaged=+1 / deep_engagement=+2 を即時受け取れるようになり、bucket placement が同セッション内で適応する
- **idempotency が透明** — `client_outcome_id` (UUIDv7) を retry でそのまま再送できる。`knowledge_event_dedupes` UNIQUE で server-side dedupe、failure-retry でも double-count しない
- **cron との namespace 分離** — dedupe_key prefix `:fe:` で衝突を構造的に排除。`ListActedEventsWithoutOutcome` の widening で cron が FE-supplied positive signal を上書きしない
- **event-time purity** — `occurred_at` は client が threshold を超えた瞬間の wall-clock を 1 度だけ取得して server に送る。server は受信時刻でなく client 時刻を payload に書く。reproject で同じ event 列を流せば同じ outcome が再現
- **observability** — [[000910]] の `act_outcome_emitted_total{outcome}` counter が FE 経由でも observe される。FE / cron どちらから emit されたかは `payload.emitter = "frontend"` で識別可能 (Grafana で stack)

### Cons / Tradeoffs

- **dwell timer の precision は client-side** — システム時計の急ジャンプ (NTP sync) で false threshold cross が起きる可能性。production traffic で監視必要。`fetch keepalive` のサイズ上限 (64 KiB) は payload 200 byte 程度なので余裕
- **"I got this" CTA は本 ADR で landed していない** — 構想層 §6 (loop closure) の epistemic graduation 経路 (INTERNALIZED dismiss_state flip) は別 trigger 経路で後続 ADR (詳細未確定)。本 ADR は engagement signal の 4 種 (engaged / deep_engagement / accepted_change / stale_save) に限定
- **Recall ↔ Loop ↔ Digest の state 同期** は scope 外。`LoopCrossLink.svelte` の deep link のみ現状維持。event 経由の同期は別 ADR
- **Pact CDC test (alt-frontend-sv → alt-backend for EmitActOutcome)** は follow-up で起票。proto は additive のため immediate breaking はないが、provider verification を組み込まないと将来の handler 変更で contract drift を検知できない
- **Playwright e2e** (dwell-based emit / visibilitychange flush / dedupe retry の三重シナリオ) も follow-up。`useActOutcomeEmitter.test.ts` の 5-case vitest unit でロジックは pin 済みだが、stack 越しの fail mode は手動 smoke のみで本番投入する
- `_outcome_missing_total` counter ([[000910]]) の値が落ちることが期待される (cron の no_engagement 補完が減るため)。逆に落ちなければ FE emit 経路に bug がある signal となる。dashboard 観察を運用フォローとして記録
