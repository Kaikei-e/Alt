# ADR-000794: search-indexer に Meilisearch hybrid search を配線し Ollama qwen3-embedding でクロスリンガル retrieval を有効化する

## Status

Accepted

## Date

2026-04-19

## Affected Services

- **search-indexer (driver)** — `HybridConfig` 新設 + 既存 SearchRequest 構築を `newBaseSearchRequest` に集約
- **search-indexer (config / bootstrap)** — `MEILI_HYBRID_EMBEDDER` / `MEILI_HYBRID_SEMANTIC_RATIO` の env 駆動
- **Meilisearch (運用側)** — `articles` インデックスに `qwen3` embedder を登録、112k ドキュメントの vector backfill

## Context

[[000669]] で Gatherer に multi-query RRF fusion を入れた際、Phase B として embedding model + Meilisearch ネイティブ hybrid search の導入は「VRAM 予算 (Gemma4 Q4 で RTX 4060 8GB の ~6GB 使用)」を理由に **延期**された。加えて [[000777]] の Cons C でも bge-reranker-v2-m3 が同じ理由で却下されていた。

2026-04-19 のユーザ報告 (イラン情勢分析レポート 2026年4月19日に英語ソースなし) に対する原因調査 ([[000792]] / [[000793]]) で、「BM25 + 多言語 locales の構成では**コンセプトレベルのクロスリンガルマッチが弱い**」ことが改めて露呈した。Curator quota が backfill 完了で発火する状態になっても、pool に十分な「英語クエリにヒットする日本語的トピック」の候補が上がってこない限り、quota は単に低スコア記事で埋めるだけになる。

追加調査で以下が判明:

- **Meilisearch は v1.27.0 で稼働中** ([[000669]] 時点想定の v1.8 から大きく先行)。ネイティブ hybrid search と `source: ollama` embedder が公式サポート
- **Ollama の `qwen3-embedding:0.6b`** は量子化 Q8_0 で 639MB、100+ 言語対応、MIRACL top-tier。news-creator-backend の Ollama (既に Gemma4 用に稼働中) に `pull` するだけで利用可能
- GPU 予算は **Gemma4 Q4 (~6GB) + qwen3 0.6B Q4 (~0.4GB) = 6.4GB / 8GB** で 1.6GB の headroom あり
- bge-reranker-v2-m3 は ONNX CPU でも回せるため別フェーズに送れる

ユーザと協議の結果、**単一マシン制約を守りながら GPU 同居を許容**する方針が合意された (セッション内選択: 「GPU も使っていいよ」→「最強版」)。これにより ADR-777 / ADR-669 で却下されていた「embedding による semantic retrieval」の再評価が成立した。

## Decision

search-indexer を Meilisearch ネイティブ hybrid search 経由で cross-lingual retrieval を提供できるようにする。クライアント (Acolyte Gatherer 等) は何も変えず、search-indexer 内で完結させる。

- **HybridConfig の導入**: `driver/meilisearch_hybrid.go` に `HybridConfig{Embedder string, SemanticRatio float64}` と `Enabled()` / `toSDK()`。`SemanticRatio` は `[0.0, 1.0]` にクランプし、security-auditor 指摘 SA-1 (ASI-04 Tool Misuse 範囲外の値で Meilisearch に不正パラメータを渡さない) を満たす
- **SearchRequest 構築の一本化**: `MeilisearchDriver.newBaseSearchRequest(query, limit)` を新設し、4 つの既存 Search* メソッドが同じベースから派生する。Hybrid の `{embedder, semanticRatio}` は base の時点で差し込む。localised なロジック (Filter / Locales / Offset) は派生先で上書き
- **Env 駆動のオプトイン**: `MEILI_HYBRID_EMBEDDER`（空 = BM25 only にフォールバック）と `MEILI_HYBRID_SEMANTIC_RATIO` (既定 0.5) で runtime 制御。`bootstrap/app.go` は `driver.WithHybrid(&HybridConfig{...})` を呼ぶだけ
- **Meilisearch embedder 登録**: 運用コマンドで `PATCH /indexes/articles/settings/embedders` にて `qwen3` を登録。URL は `http://news-creator-backend:11435/api/embed` (11434 ではない — Ollama は 11435 でリッスン、初回セットアップで**接続リフューズエラー**に嵌った)。documentTemplate は Liquid で `{{doc.title}}\n{{doc.content | truncate: 2000}}`
- **embedder 名ベースの切替**: search-indexer が知っているのは embedder 名だけ。Meilisearch index settings 側で embedder 定義を差し替えれば、search-indexer 再デプロイなしで検索モードを切り替えられる

### 検討した代替案

| 案 | 採用 | 却下理由 |
|---|---|---|
| **Meilisearch native hybrid + Ollama qwen3 (GPU)** (採用) | ○ | Meilisearch v1.27 で公式サポート。Ollama が既に news-creator で稼働、追加 runtime なし。GPU 余剰 1.6GB で安全 |
| BGE-M3 ONNX CPU を別サイドカーとして建てる | × | Acolyte だけが使う semantic の重複 embedder を管理する必要。Meilisearch ネイティブに embedder を持たせる方が運用簡潔 |
| bge-reranker-v2-m3 を今回ここで同時導入 | × | hybrid search の初期効果を先に baseline ([[000793]]) と比較したい。reranker は次フェーズ ([[000795]] 以降) で検討 |
| HyDE を search-indexer 内で実行 | × | HyDE は LLM 呼び出しで latency が重く、search-indexer のホットパスに置かない。Acolyte Gatherer 側に置く ([[000795]]) |
| 言語別 index (`articles_ja` / `articles_en`) | × | Meilisearch の LocalizedAttributes が既に jpn + eng で稼働、分割すると保守コスト増 |
| semanticRatio を Acolyte から API で可変に渡す | × | 今回は search-indexer レベルでデフォルト 0.5 固定で開始、クライアント毎のカスタマイズは proto 拡張が必要になるため deferred |

## Consequences

### Pros

- **クロスリンガルの concept match が成立**: 検証クエリ `「イラン情勢 中東」` (semanticRatio=0.7, limit=5) に対して英語記事 3 件 / 日本語記事 2 件が返るようになった。Iran's new supreme leader / Iran escalates attacks / Iran protests の 3 件は BM25 では単語マッチで上位に来ない候補
- **Acolyte 側のコード変更ゼロ**: Gatherer は従来通り `/v1/search` を叩くだけ、embedder 切替は search-indexer / Meilisearch の env + settings で閉じる。デプロイ順序の縛りが少ない
- **Rollback は env off**: `MEILI_HYBRID_EMBEDDER=""` で即 BM25 only に戻る。Meilisearch 側の vector 列は残るが query 側が触らない
- 新規 5 テスト (HybridConfig enable / toSDK clamp / search request 統合) + 既存 15 パッケージ全 green。`go vet` / `go test -race` / `go test -cover` クリーン

### Cons / Tradeoffs

- **GPU 予算の余裕は 1.6GB しか残らない**。bge-reranker-v2-m3 を GPU 版で追加すると衝突する。reranker は ONNX CPU 版を前提にする ([[000795]] か今後)
- **Ollama の listen port が 11435 なのに気づかず 11434 で embedder 登録して 20 分詰まった**。docker inspect で Ollama コンテナの expose を事前確認する runbook 追記が必要。初回 task は `POST /tasks/cancel?uids=...` が retry backoff に入ると即効しないため Meilisearch コンテナ restart が必要だった
- **112k docs の embedding backfill は Meilisearch 側で ~15 分**。本番適用時には backfill 完了までは `semanticRatio` 実効 0 に近い挙動 (vector がない) になる。運用手順上、backfill 完了を `/indexes/articles/stats` の `numberOfEmbeddedDocuments` で確認してからハイブリッド有効化を推奨
- **qwen3 次元 1024 の vector を 112k 件保持**。Meilisearch storage が数百 MB 増える。ディスク ops に変化はない
- **search-indexer の Pact 契約は変更なし** (hybrid は内部切替のため)。consumer (Acolyte 等) 側の契約反映は不要だが、hybrid 有効時の response score 性質が変わるため [[000669]] RRFFusion の重み付けは後日再調整の余地
