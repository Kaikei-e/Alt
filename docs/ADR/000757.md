# ADR-000757: pki-agent のサイレント listener 死亡を errgroup ライク終了・listener プローブ・deploy cascading で根絶する

## Status

Accepted

## Date

2026-04-17

## Affected Services

- pki-agent (Go 1.26) — sidecar として全 east-west サービスに 1 基ずつ同居
- acolyte-orchestrator / alt-butterfly-facade — 今回の障害で可視化されたピア
- scripts/deploy.sh (および `scripts/_deploy_lib.sh:deploy_single_service`)

## Context

### 直接の症状

ブラウザから `/api/v2/alt.acolyte.v1.AcolyteService/ListReports` を呼ぶと 502 を返す。
`alt-butterfly-facade` (BFF) ログに次のエラーが連続していた:

```
Post "https://acolyte-orchestrator:9443/alt.acolyte.v1.AcolyteService/ListReports":
dial tcp 172.18.0.35:9443: connect: connection refused
```

`acolyte-orchestrator` (FastAPI :8090) は healthy。BFF は mTLS enforce モードで
`https://acolyte-orchestrator:9443` (= `pki-agent-acolyte-orchestrator` が終端する
mTLS reverse proxy) 経由でしか話さない。その :9443 が死んでいた。

### 診断経路

- `pki-agent-acolyte-orchestrator` は `Up 7 hours (healthy)`。Docker 的には「生きてる」。
- `docker logs` には起動時の `TLS reverse proxy listening addr=:9443` と以降 `tick ok`
  しかない。proxy goroutine から死亡ログは一切出ていない。
- `docker exec acolyte-orchestrator python3 ... connect_ex(('127.0.0.1', 9443))` → `111` (ECONNREFUSED)。
- `docker inspect alt-pki-agent-acolyte-orchestrator-1 --format '{{.HostConfig.NetworkMode}}'`
  → `container:debe114cb69b...`。ところが現在の acolyte-orchestrator コンテナ ID は
  `20b4306fd0d0`。**pki-agent sidecar は既に存在しない古いコンテナの netns を指していた**。
- `docker compose restart pki-agent-acolyte-orchestrator` は
  `joining network namespace of container: No such container: debe114...` で失敗。
  `--force-recreate` でのみ復旧。

### 真因 (3 層)

1. **compose の `network_mode: service:<svc>` semantic**:
   pki-agent sidecar は親の netns に join する形で動く。親が `--force-recreate`
   で入れ替わると、join 先が消滅する。kernel 側では元の netns が fd の参照で
   生存し続けるため、pki-agent プロセスの内部 listener は `LISTEN` のまま見え続け、
   外部 IP (`172.18.0.35:9443`) からは「何も listen していない」状態になる。
2. **pki-agent の healthcheck**: `/healthz` は cert ローテーションの鮮度だけを見ており、
   TLS reverse proxy listener は一切プローブしない。だから上記の netns 幽霊化を
   検知できない。Docker は `healthy` を維持、`restart-on-unhealthy` が発火しない。
3. **deploy.sh の recreate 方針**: `scripts/_deploy_lib.sh:deploy_single_service` は
   `docker compose up -d --no-deps --force-recreate <svc>` を実行するが、その svc に
   対応する `pki-agent-<svc>` sidecar を一緒に recreate しない。ADR-000752 の Pact
   ゲート付き deploy で `acolyte-orchestrator` が再作成された瞬間に sidecar が
   幽霊化し、次のユーザー操作まで気づかない、という時限発火の構造になっていた。

### Web best-practice 調査

- [golang.org/x/sync/errgroup](https://pkg.go.dev/golang.org/x/sync/errgroup) —
  子 goroutine のエラーは最初の non-nil を親に返し、context を cancel する設計が
  Go 公式の推奨パターン。`http.Server.ListenAndServe*` は終了時に返るが、その戻り値を
  捨てると終了がサイレントになる。
- [WCAG/Docker healthcheck guides](https://cloud.google.com/blog/products/containers-kubernetes/kubernetes-best-practices-setting-up-health-checks-with-readiness-and-liveness-probes) —
  liveness probe はプロセスの「生存」ではなくサービスが受理する機能的状態を
  プローブすべき。sidecar の場合は専用のポート (ここでは :9443) をプローブする。
- [Docker `network_mode: service:X`](https://docs.docker.com/reference/compose-file/services/#network_mode) —
  公式的に共用 netns の再作成時の再結合は自動化されていない。
  cascading な再作成がオペレータの責務。

## Decision

3 層すべてに対策を入れ、「silent listener death → 502」を **構造的に発生させない**。

### Decision 1: pki-agent の goroutine 終了を致命化する (`cmd/pki-agent/main.go`)

TLS reverse proxy と metrics server を別 goroutine で走らせる構造は維持しつつ、
片方でも `http.ErrServerClosed` 以外で戻ったら `serverErr` チャネルで main loop に
通知し、そのまま `os.Exit(4)` する。

```go
serverErr := make(chan error, 2)
go func() {
    if err := srv.ListenAndServe(); err != nil && !errors.Is(err, http.ErrServerClosed) {
        serverErr <- fmt.Errorf("metrics server: %w", err)
    }
}()
// ... proxy goroutine 同様 ...
select {
case <-ctx.Done():              shutdown(0)
case err := <-serverErr:        slog.Error(...); shutdown(4)  // ← non-zero exit
case <-ticker.C:                rotator.Tick(...)
}
```

`errgroup` を外部依存に足す案も検討したが、Alt の pki-agent は `prometheus/client_golang`
以外の transitive dep を持たない 60KB バイナリで、`chan error` で十分だった。
外部 dep を避けつつ同じ意味論を得る最小形を採用した。

Docker は `restart: unless-stopped` で exit 4 を受けて再起動を試みる。親 netns が
生きていれば自動復旧、死んでいれば restart が繰り返し失敗するため「立て続けに
fail する sidecar」として `docker ps` / metrics / アラート側で可視化される。
silent healthy の状態には絶対に戻らない。

### Decision 2: healthcheck に proxy listener プローブを足す (`cmd/pki-agent/healthcheck.go` 新設)

`pki-agent healthcheck` サブコマンドは従来 `GET http://127.0.0.1:9510/healthz` だけ
叩いていた。これに **`PROXY_LISTEN` が環境変数で設定されているなら** `127.0.0.1:<port>`
への TCP dial を追加する。loopback 側で listener が受け答えしない状態
(= goroutine 死亡 or file descriptor exhaust 等) を sub-seconds で検知できる。

```go
func runHealthcheck(metricsAddr, proxyListen string) error {
    if err := probeMetrics(metricsAddr); err != nil { return err }
    if proxyListen != "" { if err := probeProxy(proxyListen); err != nil { return err } }
    return nil
}
```

compose 側の `healthcheck.test: ["CMD", "/usr/local/bin/pki-agent", "healthcheck"]`
は変更不要 (環境変数 `PROXY_LISTEN` は既に sidecar 個別の service 定義で設定済)。

**既知の限界**: netns 幽霊化シナリオに限って言えば、pki-agent プロセスの内部
loopback (旧 netns 側) は listener が生きているように見えるため、この dial は
false-green になる。本ケースは Decision 3 の cascading recreate で根絶する
必要がある。listener プローブは errgroup 致命化と合わせて他のリスナー死亡
パターン (ファイル記述子不足、パニック、tls config reload エラー等) を拾う
防御多層化の 1 枚目として機能する。

### Decision 3: deploy.sh が pki-agent sidecar を cascading で recreate する (`scripts/_deploy_lib.sh`)

`deploy_single_service <svc>` が svc を recreate したあと、`service_in_compose
"pki-agent-<svc>"` が真なら続けて sidecar を `docker compose up -d --no-deps
--force-recreate pki-agent-<svc>` で必ず recreate し、`wait_until_healthy` まで
ガードする。

```bash
"${DOCKER_BIN:-docker}" compose -f "${COMPOSE_FILE}" up -d --no-deps --force-recreate "$svc" || return 1
if ! wait_until_healthy "$svc"; then ... ; fi

local sidecar="pki-agent-${svc}"
if service_in_compose "$sidecar"; then
  echo "  [recreate] ${sidecar} (cascaded — shared netns)"
  "${DOCKER_BIN:-docker}" compose -f "${COMPOSE_FILE}" up -d --no-deps --force-recreate "$sidecar" || return 1
  if ! wait_until_healthy "$sidecar"; then ... ; fi
fi
```

これで「親サービスが recreate されるたびに sidecar も必ず再 join する」動作が
`scripts/deploy.sh` の中で保証される。ユーザーが手動で `docker compose up
--force-recreate` を叩く経路もあるが、そちらは runbook に pki-agent を含めて
記載する (Cons 参照)。

### TDD / テスト

- `cmd/pki-agent/healthcheck_test.go` 新設。死んだ port への dial で
  `runHealthcheck` が error を返すこと、`PROXY_LISTEN=""` のときは metrics
  のみプローブすること、bare `:PORT` の dial が loopback に解決されることを
  `httptest.NewServer` + `net.Listen("tcp", "127.0.0.1:0")` で検証。
- `go test ./... -race` 緑を確認。既存 proxy_test.go の mTLS 契約テストも不変。

### 却下した代替案

- **毎 tick 毎に pki-agent が自分自身の external IP (172.18.0.35:9443) に dial する**:
  IP の発見が不安定 (compose の IPAM allocation に依存)。`hostname -I` parsing や
  `/proc/net/route` から guess する方向は壊れやすい。
- **親コンテナのライフサイクルを inotify/PID 監視で追う**: cgroup v2 の PID 失踪検知は
  権限が要る上に OS 差分が大きい。pki-agent の責務範囲を超える。
- **`network_mode: service:X` を廃止して pki-agent を別 netns に出す**: 親の FastAPI
  が `127.0.0.1:8090` loopback バインドのままでは proxy upstream に届かなくなる。
  親側を `0.0.0.0:8090` バインドに変える案は **平文 HTTP が Pod 外に漏れる** ため
  セキュリティ上却下。Decision 3 で足す手間より、リスクが桁違いに大きい。
- **`autoheal` コンテナで unhealthy 再起動を自動化する**: Decision 2 の
  healthcheck 拡張と組み合わせれば netns 幽霊化にも部分対応できるが、単一プロセス
  autoheal に全 sidecar の運命を預けると SPOF が増える。同じ目的は
  `restart: unless-stopped` + exit(4) + deploy cascading の 3 点で既に担保できる。

## Consequences

### Pros

- **"healthy だが 502"** というクラスの silent failure が構造的に消える。
  listener が落ちれば exit(4) → Docker restart、または deploy.sh cascading で
  必ず recreate がかかる。
- pki-agent のバイナリ単体で test ができる形に healthcheck を切り出した
  (`runHealthcheck` 関数)。`os.Exit` を直接触らないため future refactor が安全。
- deploy.sh の cascading は既存の `wait_until_healthy` を再利用しており、
  sidecar が起動しない場合 deploy を失敗させる。部分デプロイで 502 が残る経路が
  なくなる。
- go.mod に新規依存を足していない。`chan error` + `select` は Go 公式
  concurrency patterns に準拠した最小構成。
- ADR-000715 (pki-agent bootstrap) / ADR-000747 (mTLS cert ライフサイクル移行) /
  ADR-000752 (Pact ゲート付き manual deploy) と矛盾せず、それぞれに missing layer
  だった fault detection を埋める補完関係。

### Cons / Tradeoffs

- **deploy 時間がサービス 1 つあたり \~15〜30 秒伸びる**: sidecar recreate + healthy
  wait が直列に乗るため。tolerable。並列化は sidecar 間で干渉なしのため後日可能。
- **手動 `docker compose up --force-recreate <svc>`** (deploy.sh を経由しない
  経路) では cascading が効かない。runbook に `compose up --force-recreate <svc>
  pki-agent-<svc>` を併記する必要がある。将来的には compose の `depends_on` に
  `restart: true` (Compose v2.30+) を入れる選択肢がある — 別途検証。
- healthcheck の listener プローブは netns 幽霊化の検知では false-green になる
  既知の限界あり (上で論じた理由による)。Decision 3 が第一線の防御、Decision 2
  は他のリスナー死亡パターン用の防御多層化。
