# ADR-000790: tag-generator Hurl E2E スイートに /api/v1/extract-tags の negative path 3 本を追加する

## Status

Accepted

## Date

2026-04-19

## Affected Services

- tag-generator (Python) — プロダクションコード / FastAPI アプリは無変更。`e2e/hurl/tag-generator/` に `05-extract-tags-missing-title.hurl` / `06-extract-tags-missing-content.hurl` / `07-extract-tags-empty-strings.hurl` を追加し、README を更新

## Context

[[000767]] で tag-generator Hurl E2E スイートの第 1 増分（scenarios 00-04）を導入した時点で、`/api/v1/extract-tags` の **negative path は Out of scope** として残された。README の当該記述:

> **`POST /api/v1/extract-tags` negative paths** — missing `title`, oversized `content`, 503 before lifespan ready.

happy path 4 本 (`/health` / `/extract-tags` smoke / `/extract-tags` keyword / mq-hub 往復) は揃っていたが、Pydantic スキーマ違反で弾かれる経路 (422 Unprocessable Entity) と、ML パイプラインが「入力シグナルなし」時に返す graceful なゼロ応答が Hurl では未検証だった。[[000789]] で `/api/v1/tags/batch` と `tag_fetcher.py` を削除した結果、本 E2E スイートがカバーすべき runtime サーフェスは `/health` + `/extract-tags` (および mq-hub 経由の Redis Streams 往復) のみに絞られ、`/extract-tags` の contract を厚めに守る価値が相対的に高まった。

具体的に未検証だった分岐:

- Pydantic の required field 欠落検証 (`title` / `content` のどちらか欠けると 422)。ハンドラが `verify_service_token` → 本体処理に到達する前に body validation で弾かれることを E2E で保証していなかった
- `title: ""` + `content: ""` 入力で ML パイプラインが返す「success=true / tags=[] / confidence=0」契約。recap-worker が extract-tags 経路で genre 名を使うため、稀にカラ本文が渡った場合の挙動が契約として暗黙だった

同じ increment で DB 残置の除去 ([[000789]]) と Hurl 拡張を扱うと commit が混ざりレビュー難易度が上がるので、本 ADR で Hurl 拡張だけ別契機として切り出した。

## Decision

**`e2e/hurl/tag-generator/` に 3 本の negative path シナリオ (05-07) を追加し、既存の 00-04 を無変更のまま残す。** 増分単位の小さな変更で、DB 拡張や stub 追加は伴わない。具体構成:

1. **05-extract-tags-missing-title.hurl** — `{"content": "..."}` のみの body を `/api/v1/extract-tags` に送ると FastAPI + Pydantic が 422 を返すことを固定。`detail` は collection であり、`detail[0].loc` / `detail[0].type` が存在することまで assert する。retry は入れない（scenario 02/03 の retry が既にサービスの readiness を保証しているため）。
2. **06-extract-tags-missing-content.hurl** — 同じく `{"title": "..."}` のみで 422。`ExtractTagsRequest` の required field 2 本それぞれが独立して validate されることを pin する。
3. **07-extract-tags-empty-strings.hurl** — `{"title": "", "content": ""}` で 200 応答を返し、`success=true` / `tags=[]` / `tags count == 0` / `confidence == 0` / `inference_ms >= 0` / `language exists` であることを固定。ML パイプラインがゼロ入力に対して例外を出さず graceful に空応答を返す契約を担保する。
4. **`run.sh` は変更不要**。既存の glob `0[1-9]-*.hurl` が自動的に 05-07 を拾うため、`--jobs 1` の serial 実行規律と scenario 03 による SBERT warm-up の依存関係もそのまま維持される。
5. **README の Scenarios 節に 05-07 を追記**し、「Out of scope」節の "missing title" / "empty content" に相当する記述を削除。残り out-of-scope (`oversized content` / `readiness race` / `TagGenerationRequested validation failure`) は後続の increment に委ねる旨だけを残す。
6. **fixture 追加はしない**。全シナリオの body は 1-2 行の JSON なので Hurl scenario 内に直書きする方が可読性が高い (04 と同じ方針)。

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| negative path を Python unit test (pytest + httpx TestClient) で担保し Hurl には追加しない | unit test では FastAPI middleware chain / CORS / mTLS 無効化 / uvicorn の JSON エラーレンダリングまで含めた実際の wire format を検証できない。Hurl suite は `alt-staging` compose slice 越しに実 FastAPI を叩くため、422 レスポンスの `detail` 契約そのものを保証できる |
| oversized content (10MB など) の 422 / 413 シナリオも同時に足す | 現行 `ExtractTagsRequest` は `max_length` を持たないので oversized でも 200 を返す。サーバ側に長さ制約を入れる決定を先に行わないと意味のない scenario になるため、READMEの Out of scope に明示して後続に委ねる |
| readiness race (503 before lifespan ready) の scenario を足す | FastAPI の lifespan は既に scenario 02 の retry budget (20 × 500ms) で吸収されている。cold-start probe を別 scenario にするには `--jobs 1 -serial` の先頭で個別に発火させる必要があり、run.sh の構造を壊す割に情報量が低い |
| 全 negative path を単一の `.hurl` に [Group] でまとめる | Hurl の report 単位が scenario ファイルなので、個別 JUnit entry として分けた方が CI 上で何が失敗したかが明確。3 本に分割するコストは低い |

### 範囲外 (今回やらないこと)

- `/api/v1/extract-tags` の oversized content — サーバ側の `max_length` 方針決定後
- FastAPI lifespan ready 前の 503 応答 — 既存 scenario 02 retry が吸収済み
- `TagGenerationRequested` validation failure arm — mq-hub 経由の error レスポンス契約は mq-hub 側の Hurl suite で扱う領域
- 認証付きエンドポイント (`/api/v1/generate-tags` / `/api/v1/user-preferences`) の negative path — [[000767]] 準拠で alt-auth が staging に入るまで deferred

## Consequences

### Pros

- **`/api/v1/extract-tags` の contract が厚くなる**。happy path 2 本 (02/03) + negative path 3 本 (05/06/07) で required field validation と graceful-zero の分岐がすべて黒箱で固定される
- **リグレッション検出が増える**。Pydantic v2 への upgrade や middleware 追加で 422 の body shape が変わる変更が入れば、次回 PR の Hurl ゲートで拾える
- **実装コストが極小**。glob `0[1-9]-*.hurl` が自動で拾うため run.sh / compose / fixture 変更が不要。3 本のシナリオファイル追加と README 更新のみ
- **OSS ドキュメントとしての整合性**。README の Scenarios 節が増え、tag-generator の公開 contract を誤解なく読み取れる

### Cons / Tradeoffs

- **scenario 数の増加で run.sh の実行時間がわずかに伸びる**。ただし各 scenario は 1 リクエスト + 最小 assert で、`--jobs 1` でも 1 秒未満。CI 上の体感は無視可能
- **422 レスポンスの `detail` 構造に scenario が依存する**。FastAPI / Pydantic v2 のマイナーバージョンアップで `detail[0].type` のキーが変わると 05/06 が赤くなる。発生時は同時に scenario も更新する運用が必要
- **`/extract-tags` の oversized 分岐は依然未検証**。README の out-of-scope に残置したため、運用で max_length を入れる判断が出た時に scenario 追加も忘れないようにする
