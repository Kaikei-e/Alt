# ADR-000583: FetchArticleContentにフルテキスト閾値を導入しInoreader短文キャッシュを自動バイパスする

## Status

Accepted

## Date

2026-03-25

## Affected Services

- alt-backend (Go) — `FetchCompliantArticleWithRefresh` usecase

## Context

Inoreader からインポートされた記事が Knowledge Home の Article 画面で低品質（短い）RSS サマリーのまま表示される問題が発生していた。

[[000471]] で導入した「longest-content-wins」ガードは、Inoreader API の再取得で既存の長いコンテンツが短いコンテンツに上書きされることを防ぐ仕組みである。また [[000391]] で導入した `force_refresh` はユーザーが手動でキャッシュをバイパスする機能である。しかし、以下のシナリオではどちらも機能しない:

1. Inoreader sync が RSS サマリー（短いコンテンツ）を `articles` テーブルに初回保存する
2. ユーザーが Knowledge Home から記事を開く → `FetchArticleContent` RPC が呼ばれる
3. `FetchCompliantArticleWithRefresh(forceRefresh=false)` が DB にキャッシュされた記事を発見し、**コンテンツの長さに関係なく即座に返す**
4. ユーザーは短い RSS サマリーを見る。`force_refresh` の存在を知らない限り、フルコンテンツを取得する手段がない

問題の本質は、DB キャッシュのヒット判定が「記事が存在するか」のみで、「キャッシュされたコンテンツがフルテキスト品質か」を考慮していなかったことにある。

## Decision

**`FetchCompliantArticleWithRefresh` usecase にフルテキスト閾値（`minFulltextContentLength = 500` bytes）を導入し、閾値未満のキャッシュを自動的にバイパスして Web からフルコンテンツを取得する。**

### 設計方針

- **Usecase 層にロジックを閉じる**: 閾値判定はビジネスロジックであり、Driver/SQL 層ではなく usecase 層で行う。[[000452]] の「SQL にビジネスロジックを持たせない」原則に準拠
- **`SaveArticle` は変更しない**: [[000471]] で明記された通り、Web 抽出パスは意図的なコンテンツ置換であり、ガード不要。本修正も `SaveArticle` には手を加えない
- **既存の longest-content-wins ガードとの協調**: Web fetch 後に `SaveArticle` がフルコンテンツを保存 → 以降の Inoreader sync は [[000471]] のガードで保護される

### 閾値 500 bytes の根拠

| コンテンツ種別 | 典型的なサイズ |
|---|---|
| Inoreader RSS サマリー（日本語） | 50〜150 文字 → 150〜450 bytes |
| Inoreader RSS サマリー（英語） | 100〜300 文字 → 100〜300 bytes |
| フルテキスト記事（日本語） | 1000+ 文字 → 3000+ bytes |
| フルテキスト記事（英語） | 2000+ 文字 → 2000+ bytes |

500 bytes は RSS サマリーとフルテキストの境界として妥当な値である。`SaveArticle` の既存 `minContentLength = 100` は抽出失敗の警告閾値であり、目的が異なる。

### 検討した代替案

| 案 | 評価 |
|---|---|
| フルテキスト閾値チェック（採用） | usecase 層で完結。既存パスへの影響なし。シンプル |
| `articles` テーブルに `is_fulltext` フラグ追加 | スキーマ変更が必要。Inoreader 取り込みパスの修正も必要で侵襲性が高い |
| 初回表示時に常に Web fetch | レート制限とパフォーマンスの観点で不適切。フルテキストが既にキャッシュされている場合も無駄な fetch が発生する |
| `force_refresh` の UI 露出を改善 | 根本解決にならない。ユーザーに品質判断を委ねるべきではない |

### 実装概要

`FetchCompliantArticleWithRefresh()` の DB キャッシュヒット判定を変更:

```go
// 変更前
if existingArticle != nil {
    return existingArticle.Content, existingArticle.ID, cachedOgImage, nil
}

// 変更後
const minFulltextContentLength = 500
if existingArticle != nil && len(existingArticle.Content) >= minFulltextContentLength {
    return existingArticle.Content, existingArticle.ID, cachedOgImage, nil
}
if existingArticle != nil {
    // 短いキャッシュ → Web fetch に進む
    logger.Logger.InfoContext(ctx, "Cached article content below fulltext threshold, fetching from web", ...)
}
```

### エッジケース

| シナリオ | 動作 |
|---|---|
| 初回 Inoreader 取り込み（短い）→ ユーザーが開く | Web fetch → フルコンテンツ保存 |
| フルテキストキャッシュあり → ユーザーが開く | キャッシュ返却（変更なし） |
| 短いキャッシュ → Web fetch 失敗 | エラー返却（既存の cache miss と同じ動作） |
| `forceRefresh=true` | DB キャッシュ完全スキップ（変更なし） |
| 正当に短い記事（< 500 bytes） | 毎回 Web fetch → `SaveArticle` で上書き（同じ短いコンテンツ） |

## Consequences

### Pros

- **ユーザー体験の改善**: Inoreader 由来の短い RSS サマリーがそのまま表示される問題が解消される。ユーザーが `force_refresh` を知らなくても自動的にフルコンテンツを取得する
- **既存ガードとの協調**: Web fetch でフルコンテンツが保存された後は、[[000471]] の longest-content-wins ガードが Inoreader からの再上書きを防止する
- **最小侵襲**: usecase 内の条件式1箇所の変更のみ。インターフェース変更なし、スキーマ変更なし、他サービスへの影響なし
- **透明性**: 閾値バイパス時にログを出力するため、動作の追跡が可能

### Cons / Tradeoffs

- **正当に短い記事への影響**: 500 bytes 未満のフルテキスト記事は毎回 Web fetch が走る。ただし rate limiter（5秒間隔）と robots.txt チェックが適用されるため安全であり、取得結果は `SaveArticle` で保存されるため外部への影響は初回のみ
- **閾値のハードコード**: 500 bytes は定数として usecase 内に定義。将来的に調整が必要になった場合はこの定数を変更する。設定ファイル化は現時点ではオーバーエンジニアリングと判断
- **Web fetch 失敗時のフォールバックなし**: 短いキャッシュが存在していても、Web fetch が失敗した場合はエラーを返す（短いキャッシュをフォールバックとして返さない）。`force_refresh` と同じ挙動であり、ユーザーはリトライ可能
