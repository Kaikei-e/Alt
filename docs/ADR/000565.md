# ADR-000565: スワイプフィードカードの XSS 脆弱性を修正する

## Status

Accepted

## Date

2026-03-25

## Affected Services

- alt-frontend-sv (TypeScript / SvelteKit)

## Context

包括的セキュリティレビュー（OWASP Top 10:2025 準拠）の過程で、モバイル向けスワイプフィードカードに 2 種類の XSS 脆弱性が発見された。

### 脆弱性 C-3: `feed.description` のエンティティデコード後 XSS

`sanitizeFeed()` は RSS フィードの `description` から HTML タグを除去し、HTML エンティティをデコードする。デコード結果は**プレーンテキスト**だが、`&lt;script&gt;` → `<script>` のようにアングルブラケットが復元される。この結果を Svelte の `{@html}` で描画していたため、エンティティエンコードされた悪意あるペイロードがそのまま実行される状態だった。

**攻撃ベクター**: RSS フィードの description に `&lt;img src=x onerror=&quot;alert(1)&quot;&gt;` を含めると、`decodeHtmlEntities()` 通過後に `<img src=x onerror="alert(1)">` となり、`{@html}` 経由で XSS が成立する。

### 脆弱性 C-4: `fullContent` の未サニタイズ HTML 描画

外部 Web サイトからスクレイプした記事全文（`fullContent`）を `sanitizedFullContent` という変数に格納していたが、実態は `$derived(fullContent)` — サニタイズ処理が一切行われていなかった。変数名が「sanitized」であるため、レビュー時に見落とされやすい偽の安全性を生んでいた。

**攻撃ベクター**: 攻撃者が管理する Web サイト（または侵害された RSS ソース）の記事 HTML に `<script>document.cookie</script>` 等を含めれば、そのまま実行される。

両方とも `VisualPreviewCard.svelte` と `SwipeFeedCard.svelte` の 2 コンポーネントに存在した。一方、同プロジェクトの `RenderFeedDetails.svelte` では `sanitizeHtml()` (DOMPurify) を正しく適用しており、修正の手本となるパターンが既にコードベース内に存在していた。

## Decision

以下の方針で修正を行う。

### C-3: `{@html}` を除去し、Svelte のテキスト補間に変更する

`feed.description` は `sanitizeFeed()` で全タグ除去済みのプレーンテキストである。HTML レンダリングが不要なコンテンツに `{@html}` を使う理由がないため、Svelte の安全なテキスト補間 `{feed.description}` に変更する。

これは OWASP XSS Prevention Cheat Sheet の原則「危険な API を不要な場面で使わない」に従う判断である。DOMPurify を適用する代替案も検討したが、そもそも HTML レンダリングが不要なコンテンツにサニタイザーを挟むのは過剰であり、テキスト補間のほうがシンプルかつ確実に安全である。

### C-4: `sanitizeHtml()` (DOMPurify) を `$derived` で適用する

`fullContent` は外部 Web サイトの HTML であり、リッチテキスト構造（見出し、リスト、リンク、強調等）を維持して描画する必要がある。そのため `{@html}` は必要だが、描画前に DOMPurify でサニタイズする。

```typescript
// Before (vulnerable)
const sanitizedFullContent = $derived(fullContent);

// After (safe)
const sanitizedFullContent = $derived(fullContent ? sanitizeHtml(fullContent) : null);
```

既存の `sanitizeHtml()` ユーティリティは以下の防御を提供する:
- `<script>`, `<iframe>`, `<form>`, `<img>`, `<svg>` タグの除去
- イベントハンドラ属性（`onerror`, `onload` 等）の除去
- `javascript:`, `data:`, `vbscript:` URL スキームのブロック
- 許可タグのホワイトリスト方式（構造要素、テキスト書式、リンク、テーブルのみ）
- 全リンクに `target="_blank"` と `rel="noopener noreferrer"` を自動付与

### テストの追加

- `sanitize.test.ts`: エンティティデコード後の description が XSS ペイロードを含みうることを文書化するテスト（2 件）
- `sanitizeHtml.test.ts`: 外部スクレイプ記事コンテンツに対する DOMPurify の XSS 無害化テスト（2 件）

## Consequences

### Pros

- OWASP Top 10:2025 A03 (Injection) に該当する 2 つの Critical XSS 脆弱性を解消
- `RenderFeedDetails.svelte` と同一パターンに統一され、コードベース全体で一貫したサニタイズ方針になった
- 変数名 `sanitizedFullContent` が実態と一致するようになり、将来のレビューで誤解が生じにくい
- テストが攻撃ベクターを文書化しており、回帰防止として機能する

### Cons / Tradeoffs

- C-3 の修正により、`feed.description` に意図的に含まれていた HTML 書式（太字等）があった場合は表示されなくなる。ただし `sanitizeFeed()` が既にタグを除去しているため、実質的な表示変更は発生しない
- C-4 の修正で DOMPurify のサニタイズ処理が追加されるため、大量の記事コンテンツ描画時にわずかなパフォーマンスコストが生じる。ただし DOMPurify は DOM パーサーベースで高速であり、ユーザー体験への影響は無視できる
