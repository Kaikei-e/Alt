# ADR-000762: pact-publish-and-verify ジョブを matrix 並列化し tts-speaker を CI 検証から除外する

## Status

Accepted

## Date

2026-04-17

## Affected Services

- **.github/workflows/proto-contract.yaml** — `pact-publish-and-verify` を 3 ジョブに分解 (`pact-verify-go` / `pact-verify-python` matrix / aggregator)。`jlumbroso/free-disk-space@main` と `setup-uv@v6` の `enable-cache: true` を導入。tts-speaker の provider verification step 2 つを削除
- **tts-speaker** — `torch` を `[project.dependencies]` に明示追加し `[tool.uv.sources]` で `pytorch-cpu` index に固定。`uv.lock` 再生成で CUDA 系 20 パッケージを除去、torch を 2.10.0 → 2.11.0+cpu に切替

## Context

2026-04-17 の `main` への push (`f58488d49`) に対する GitHub Actions 実行 (run `24567994145`) で、`Proto & Contract Validation / Pact Publish & Provider Verification` ジョブの `Install tts-speaker dependencies` step が `No space left on device (os error 28)` で失敗した。失敗は contract の regression ではなく、`uv sync --dev` が `torch==2.10.0` の manylinux CUDA wheel (873 MiB) を展開している途中で ubuntu-latest ランナーの空き容量を使い切ったことに起因する。

ubuntu-latest の実質利用可能容量は公称 14 GB 保証 / 実際 25-29 GB で、2025 年 2 月以降さらに削減された ([GitHub Docs](https://docs.github.com/en/actions/reference/runners/github-hosted-runners) / [actions/runner-images#9344](https://github.com/actions/runner-images/issues/9344))。この予算の上で Pact ジョブは以下の 3 つの非対称を合成していた:

1. **tts-speaker の torch は main 依存の推移解決で入る** — `tts-speaker/pyproject.toml` 自体は `kokoro>=0.9.4` / `misaki[ja]>=0.9.3` を宣言するだけで `torch` は持たないが、kokoro の推移依存で CUDA 版 torch (900 MiB クラス) が引かれる。production Docker ビルドは base image の torch を流用して PyPI 版を一切入れないため、この問題は CI でのみ顕在化する
2. **Python provider 4 サービス (news-creator / recap-subworker / tag-generator / tts-speaker) を単一ジョブで逐次 `uv sync --dev`** していた。各 step が `.venv` と uv の wheel cache を積み上げ、4 サービス目 (tts-speaker) に到達した時点で累積が上限を超えた
3. **`setup-uv@v6` が `enable-cache: false`** で呼ばれており、CI 実行ごとに全 wheel を再 DL。間欠失敗時の再走は同じディスク圧を毎回作る

加えて `c2quay.yml:43-50` の注記 *"tts-speaker runs on a separate GPU host and is intentionally NOT declared in compose/compose.yaml"* が示すとおり、tts-speaker はこの compose スタックのデプロイ対象ではなく、別ホストでホストされる。ローカルの `scripts/pact-check.sh --broker:276-292` は tts-speaker の provider verification を (recap-worker の `rust-stub-replay` と同じ) manual verification publish で処理しており、これが deploy ホストの永続 broker に書き込まれ c2quay の `can-i-deploy` が参照する唯一の実体になっている。つまり **CI だけが独自に重いロードを強制する非対称** が存在した。

本 ADR はこの disk 障害を一時対症 (free-disk-space 追加) で済ませず、構造的な不整合を一括で正す。

## Decision

### 1. jlumbroso/free-disk-space + setup-uv キャッシュ + torch CPU wheel 固定

各 provider verify ジョブの冒頭に `jlumbroso/free-disk-space@main` を `if: runner.environment != 'self-hosted'` ガード付きで追加 (`recap-worker.yml:31-41` と同じパターン)。`tool-cache: false`, `android: true`, `dotnet: true`, `haskell: true`, `large-packages: true`, `docker-images: true`, `swap-storage: true` で 25-31 GB 回復する ([free-disk-space-ubuntu](https://github.com/marketplace/actions/free-disk-space-ubuntu))。

`setup-uv@v6` を `enable-cache: true` + `cache-dependency-glob: ${{ matrix.lock }}` に切替。matrix の各 service が個別 lock で cache key を分離するため cross-service 汚染は起きない。

`tts-speaker/pyproject.toml` に uv 公式の PyTorch 統合パターンを適用:

```toml
[project]
dependencies = [..., "torch>=2.6.0"]

[[tool.uv.index]]
name = "pytorch-cpu"
url = "https://download.pytorch.org/whl/cpu"
explicit = true

[tool.uv.sources]
torch = { index = "pytorch-cpu" }
```

`[tool.uv.sources]` の source override は `[project.dependencies]` / `[dependency-groups]` / `[tool.uv]` に直接宣言されたパッケージに対してのみ適用されるため ([uv docs](https://docs.astral.sh/uv/concepts/projects/dependencies/))、transitive の torch を上書きするには torch を直接依存として宣言する必要がある。`uv lock --upgrade-package torch` で re-resolve すると cuda-bindings / cuda-toolkit / nvidia-* / triton を含む 20 パッケージが消え、torch は `2.11.0+cpu` に切り替わる。ローカルで `uv run pytest tests/contract/` を実行し 4/4 pass を確認済み。

### 2. pact-publish-and-verify を 3 ジョブに分割

新構造:

| Job (internal id) | name (display / required check) | Depends on | 役割 |
|---|---|---|---|
| `pact-verify-go` | `Pact Provider Verify (Go)` | pact-cdc-{go,rust,python-consumer} | broker publish + alt-backend / search-indexer / mq-hub の provider verify |
| `pact-verify-python` | `Pact Provider Verify (Python)` | pact-cdc-{go,rust,python-consumer} | matrix 3 要素 (news-creator/app, recap-subworker, tag-generator/app) を並列で broker publish + pytest 実行 |
| `pact-publish-and-verify` | `Pact Publish & Provider Verification` | pact-verify-{go,python} | aggregator。実 work 無し、`needs` 連鎖で上記 2 ジョブが success のときだけ pass |

aggregator の display name を従来どおり `Pact Publish & Provider Verification` に固定することで、main ブランチ保護の required status check 名を変更せずに済む。Python matrix は fail-fast: false で 3 並列、各要素が `services: pact-db + pact-broker` を自前で立ち上げ、pact artifacts を download → broker publish → `uv sync --dev` → `uv run pytest tests/contract/` を実施する。

matrix に `extra-env` キーを用意し、`recap-subworker` の `PYO3_USE_ABI3_FORWARD_COMPATIBILITY=1` を表現した。現在は 1 要素のみが使用するが、将来の env 注入に拡張しやすい。

### 3. tts-speaker を CI の provider verify 対象から除外

`proto-contract.yaml` から tts-speaker の 2 step (`Install tts-speaker dependencies` / `Run tts-speaker provider verification`) を削除し、matrix にも含めない。CI broker は per-job の `services:` で立ち上がる ephemeral な存在であり、CI で verification record を publish しても deploy ホストの永続 broker には届かない。deploy 時の `can-i-deploy` が参照する唯一の実体は `scripts/pact-check.sh --broker:276-292` が deploy ホスト上で publish する manual verification 側であり、CI と local の duplicate 解消策としては **CI 側を降ろす** のが spec に沿う。

header コメントに以下を明記して将来の読み手の文脈を固定した:

> tts-speaker provider verification is not run in CI by design — it is hosted separately and its can-i-deploy record is published via scripts/pact-check.sh against the persistent deploy-host broker.

### 却下した代替案

1. **free-disk-space だけ追加して構造は維持** — 対症療法。matrix 不在で venv 累積は残り、setup-uv `enable-cache: false` も残る。次の依存更新で torch サイズが増えたらすぐ再発する
2. **torch を `sed` で `uv pip compile` 後に strip する** — Dockerfile 側と同じ手法を CI に移植する案。動作はするが `uv sync --dev` の lock-file 運用と噛み合わず、保守が CI 固有スクリプトに寄る。`[tool.uv.sources]` + index は uv 公式推奨の単純な構造で、pyproject.toml だけで完結する
3. **`dorny/paths-filter@v4` で Python 変更時のみ走らせる** — workflow top-level の `on.push.paths` / `on.pull_request.paths` で既に contract ディレクトリに限定されている。job 粒度の二重 filter を追加しても実益が薄く、required status check の `skipped = pass` 問題を管理する手間が増えるだけ。将来モノレポが大きく膨らんだ時点で再検討する
4. **`[dependency-groups].contract` を各 Python service に追加し `uv sync --no-default-groups --group contract`** — dev group から ruff/bandit/pyrefly を落として contract test に必要な最小 (pytest/pytest-asyncio/httpx/pact-python) だけ入れる案。各 service の dev group は既に軽量で、torch CPU wheel 化で disk 圧の主因が消えた現状では gain が小さい。将来別の service で同種の問題が出たら導入する
5. **CI 側でも manual verification を publish** — CI broker は ephemeral なので、publish しても consumer/provider の record は job 終了とともに消える。deploy ホストの永続 broker には触らない原則 (header コメントの *"no prod credentials or state in CI"*) と矛盾しないよう、CI 側では publish しない
6. **tts-speaker を compose に含めて CI でも docker compose 起動から provider verify** — 別ホスト前提を崩すことになる。`c2quay.yml:43-50` の設計判断を覆す重さに見合う利得がない

## Consequences

### Pros

- **disk full の恒久対策**。torch CPU wheel (873 MiB → 150 MiB クラス) + free-disk-space (+25-31 GB) + setup-uv cache (2 回目以降の再 DL 削減) の 3 重。次の torch minor bump や他 Python service の追加でも耐える
- **Python provider のフィードバック時間短縮**。matrix 並列化で 4 service 逐次 (~15-20 分) から 3 並列 (~5-7 分) へ。各要素が独立ランナーで .venv を持つため累積汚染も消える
- **`Pact Publish & Provider Verification` required check 名を維持**。branch protection 設定を一切触らずに済む。aggregator が `needs: [pact-verify-go, pact-verify-python]` で連鎖するため gate セマンティクスは保たれる
- **CI と local で tts-speaker の扱いが一致**。どちらも「CI では触らない、deploy ホスト上の manual publish だけが正」。header コメントに明記したため、将来 tts-speaker を CI に入れ直す判断をする人は前提を意識してから動ける
- **uv 公式パターンへの収束**。`[tool.uv.sources]` + explicit index は公式 PyTorch 統合ガイドそのまま。uv バージョンアップで挙動が変わるリスクが低い

### Cons / Tradeoffs

#### C1: 新ジョブ `pact-verify-go` / `pact-verify-python` は branch protection に個別登録していない
aggregator の `needs` 連鎖で事実上 gate されるが、branch protection の required status check として明示登録しておくと、aggregator が何らかの理由でスキップされても新ジョブ単独で merge を blocking できる。現状は `Pact Publish & Provider Verification` のみ required という従来どおりの設定で、aggregator に依存する。GitHub のデフォルト挙動で skipped required check が pass 扱いになる件と組み合わせた failure mode を回避するため、将来 matrix 要素数を増やすときに branch protection を見直す。

#### C2: matrix 3 要素で broker 起動コストが 3 倍
各 matrix 要素が `services: pact-broker` を立ち上げるため、broker 起動 (~60 秒) が 3 回発生する。並列実行なので wall-clock は増えないが、CPU 時間ベースでは 180 秒分の余剰コスト。共通 broker を 1 ジョブで立てて downstream matrix が共有する設計 (例: `workflow_run` 連鎖 + `services:` 外出し) は GitHub Actions の基本セマンティクスから外れるため採らない。

#### C3: tts-speaker の provider verify は 100% local 運用に依存
CI で verify しないため、`scripts/pact-check.sh --broker` を手で流し忘れると deploy ホストの broker に新 SHA の verification record が publish されず c2quay の can-i-deploy で詰まる。これは従来も同じ運用だったが (CI でも manual publish はしていなかった)、明示化された分だけ責任が local ワークフローに集中する。`scripts/deploy.sh` が先頭で pact-check.sh を呼ぶため通常運用では問題にならないが、`deploy.sh` をバイパスした手動 compose up を禁じるプロジェクトルールが前提条件になる。

#### C4: uv `[tool.uv.sources]` の制約を吸い込んだ
transitive な torch を CPU wheel に向けるために、tts-speaker/pyproject.toml に torch の直接宣言 (`torch>=2.6.0`) を増やした。これは uv の source override が直接依存にしか効かないという現仕様 ([uv#6729](https://github.com/astral-sh/uv/issues/6729)) への適応策で、kokoro の実際の torch 要件とは独立した宣言になる。kokoro が将来 torch の major version を上げたら `torch>=2.6.0` の下限を合わせて更新する必要がある。

#### その他
- 新構造の YAML はファイル長が約 330 行増える (broker services + free-disk-space step が verify job 2 つで重複)。GitHub Actions では `services:` 共有や YAML anchor が使えないため、重複は構造上不可避
- `setup-uv` の `cache-dependency-glob` を matrix ごとに分けたが、matrix `${{ matrix.lock }}` を expression として展開するため lint で警告が出る可能性がある。今回は動作実測で問題なし
