# ADR-000789: tag-generator の Shared Database 残置を除去し recap-worker を alt-backend の新 RPC 経由に切り替える

## Status

Accepted

## Date

2026-04-19

## Affected Services

- alt-backend (Go) — 新 RPC `BatchGetTagsByArticleIDs` を追加
- recap-worker (Rust) — タグ取得先を tag-generator から alt-backend に付替、Pact 契約も移設
- tag-generator (Python) — 旧 `/api/v1/tags/batch` と `psycopg2` 直読経路 (`tag_fetcher.py` / `db_pool.py`) を完全削除
- e2e/stubs/recap-pipeline-stub (Python) — Connect-RPC stub の入れ替え

## Context

[[000241]] で「alt-backend を唯一のデータオーナーにし、他サービスは Connect-RPC 経由でアクセスする」方針を決定し、[[000397]] で Legacy DB モード (psycopg2 / pgx 経由の `articles` / `article_tags` / `feed_tags` 直読) を完全廃止したはずだった。tag-generator 側では `article_fetcher/fetch.py` など書き込み系は除去済みで、alt-db の `tag_generator` ロールも [[000397]] Phase 6 のマイグレーション (`20260317000001_drop_legacy_service_users.sql`) で DROP されていた。

しかし読み出し系の `tag_fetcher.py` と、そこから呼ばれる `/api/v1/tags/batch` エンドポイント、同時期のデッドコードだった `db_pool.py` はいずれも [[000397]] Phase 2 の取りこぼしとして残っており、下流の recap-worker が本番で `POST /api/v1/tags/batch` を叩く運用になっていた。直読クエリは `articles INNER JOIN article_tags INNER JOIN feed_tags` と、全て alt-backend が所有するテーブルを跨いでいる。状態:

- **Shared Database アンチパターンの残置**: alt-db の DROP 済みユーザーに向かって DSN だけが生き残る矛盾状態。`DB_TAG_GENERATOR_*` env を staging / .env.example / .env.template / `for-development/start-dev.sh` に引きずり、新規参画者に「tag-generator は DB を持つのか？」という誤解を与え続けていた
- **責務の混線**: tag-generator は本来「タグを生成する compute サービス」であり、「タグを配信する data サービス」ではない。recap-worker が tag-generator を経由してタグを読む構造は、compute と data の責務を中継層に背負わせる設計債だった
- **E2E 拡張の制約**: [[000767]] で tag-generator Hurl suite を導入した際、`/api/v1/tags/batch` の happy path は「`articles + article_tags + feed_tags` を持つ Postgres を staging に立てる必要がある」ため out-of-scope 宣言されており、そのまま扱うと staging に Shared DB を復活させる形で [[000241]] に反する展開になっていた

ユーザが tag-generator E2E 拡張を依頼した際、「実 DB につながっていれば立ち上げて E2E するように」という指示と、[[000241]] / [[000397]] の北極星の間で矛盾が顕在化した。単純に staging へ DB を追加すれば tag-generator の Shared DB 違反を復権させてしまうため、E2E 拡張と同じ increment で設計債の返済を同時に行う判断を取った。

## Decision

**tag-generator の `/api/v1/tags/batch` と `tag_fetcher.py` / `db_pool.py` を完全削除し、recap-worker を alt-backend の新 RPC `BatchGetTagsByArticleIDs` に向き替える。** alt-backend を唯一のデータオーナーとする [[000241]] の北極星をコード上で強制し、[[000397]] Phase 2 の取りこぼしを解消する。具体構成:

1. **alt-backend に `BatchGetTagsByArticleIDs` Connect-RPC を新設**する。`BackendInternalService` の `GetArticleByID` 直後に配置し、`BatchGetTagsByArticleIDsRequest { repeated string article_ids = 1 }` / `BatchGetTagsByArticleIDsResponse { repeated ArticleTagsEntry items = 1 }` を追加。`ArticleTagEntry` に `tag_name` / `confidence` / `source` / `updated_at` を持たせ、旧 `/api/v1/tags/batch` の JSON 契約 (`{tag, confidence, source, updated_at}`) を proto 型として正式化する。
2. **driver 層は単一 SQL** (`FROM articles a INNER JOIN article_tags at ON a.id = at.article_id INNER JOIN feed_tags ft ON at.feed_tag_id = ft.id WHERE a.id = ANY($1::uuid[])`) で実装する。既存 `FetchArticleTags` は 1 article-id 固定なので流用せず、複数 ids を 1 ラウンドトリップで返す別メソッド `BatchGetTagsByArticleIDs` を `TagRepository` に追加。`ORDER BY a.id, ft.confidence DESC` で決定的な順序を担保。
3. **handler 側に 1000 件の上限**を置く。旧 Python 実装 (`auth_service.py`) が `if len(article_ids) > 1000` で弾いていた invariant を `maxBatchGetTagsArticleIDs` 定数として handler に移設し、driver が無制限 `ANY($1::uuid[])` を発行しないようサーバ側で保証する。空配列は DB を叩かず空レスポンスで返す。
4. **recap-worker を alt-backend に向き替える**。`AltBackendClient::batch_get_tags_by_article_ids` を追加し、`pipeline/fetch.rs` の `AltBackendFetchStage` が旧 `TagGeneratorClient::fetch_tags_batch` ではなく新メソッドを呼ぶ形に変更。`TagGeneratorClient` は `/api/v1/extract-tags` 専用に縮退させる (persist stage が使う)。`pipeline/morning.rs` / `pipeline/orchestrator.rs` のコンストラクタ引数からも tag-generator client を外す。
5. **Pact 契約を移設する**。`recap-worker-tag-generator.json` から `a batch tags request` interaction を削除し、`recap-worker-alt-backend.json` に `a batch tags request by article ids` interaction (`services.backend.v1.BackendInternalService/BatchGetTagsByArticleIDs`) を追加。`clients/alt_backend_contract.rs` に対応する consumer test を追加し、`clients/tag_generator_contract.rs` の batch interaction を削除。
6. **tag-generator 側の Legacy コードを削除する**。`auth_service.py` から `/api/v1/tags/batch` ハンドラ・`BatchTagsRequest`・`from tag_fetcher import ...` を除去。`tag_fetcher.py` / `db_pool.py` / `pyproject.toml.ultra-minimal` をファイルごと削除。`tag_generator/infra/config.py` の `DatabaseConfig` と `tag_generator/config.py` の再 export を除去。provider verification test (`test_provider_verification.py`) から `batch_tags` スタブと "tags exist for the requested articles" state を削除。`docs/services/tag-generator.md` / compose staging コメント / `.env.example` / `.env.template` / `for-development/start-dev.sh` の `DB_TAG_GENERATOR_*` 参照を削除。
7. **e2e/stubs/recap-pipeline-stub を新経路に合わせる**。`recap-worker` Hurl suite 用の stub に `POST /services.backend.v1.BackendInternalService/BatchGetTagsByArticleIDs` ハンドラを追加し、旧 `POST /api/v1/tags/batch` ハンドラを削除する。既存の `ALT_BACKEND_BASE_URL=http://rw-stub-alt-backend/` 構成のまま stub が新 RPC を受けるので compose profile の変更は不要。

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| staging に独立な `tag-generator-db` を立て、alt-db から `articles` / `article_tags` / `feed_tags` の subset を複製して `/api/v1/tags/batch` の happy path を Hurl suite に載せる | Shared Database を別名で復権させる形で [[000241]] の北極星に逆行。[[000397]] 完了時点で DROP したユーザーを staging だけ生かし続けるのも運用債になる |
| `/api/v1/tags/batch` を残し、内部だけ psycopg2 から Connect-RPC proxy (alt-backend → tag-generator → recap-worker) に書き換える | tag-generator を「データ配信の薄いファサード」に固定化する。compute / data 責務の混線が恒久化し、URL 互換を残すコストが本質的な設計改善を阻む |
| `/api/v1/tags/batch` は現状維持で Hurl E2E は negative path のみ追加 | Legacy コードと Shared DB 残債が無期限に残り、[[000397]] Phase 2 の取りこぼしを毎回持ち越す。OSS として公開される `docs/services/tag-generator.md` に偽の API 責務が残り続けるのも害 |
| alt-backend 側 RPC を `BatchGetArticlesByIDs` (既存 `GetArticleByID` の複数形、`ArticleWithTags[]` を返す) にする | recap-worker はタグだけが必要で article 本文を引く必要がない。payload が膨らみ、SQL も `articles` JOIN を必ず含める必要が出るため、narrow な `BatchGetTagsByArticleIDs` を採用 |

### 範囲外 (今回やらないこと)

- `scripts/build_label_graph.py` が `recap_genre_learning_results` (recap-db) を psycopg2 で読む経路 — 実行時サービスではなくオフラインのメンテスクリプトのため、runtime Shared DB の論点からは分離。必要なら別 increment で recap-worker RPC 化
- alt-backend に既存する `GetArticleByID` / `ListArticlesWithTags` と本 RPC の統合 — 用途が異なる (タグだけ / 記事と一緒 / paginated) ため併存させる
- Atlas マイグレーション `20240101002600_create_tag_generator_user.sql` の削除 — Atlas の append-only 規則に反するため残置。[[000397]] Phase 6 の DROP マイグレーションで既に無効化済みで実害なし

## Consequences

### Pros

- **Shared Database アンチパターンの残置が完全消滅**する。[[000241]] の「alt-backend を唯一のデータオーナーに」がコード上で強制され、[[000397]] の Phase 2 取りこぼしが閉じた
- **tag-generator の責務が compute 側に一本化**される。`tag-generator/app/` から psycopg2 を参照する runtime パス・`DatabaseConfig`・`DB_TAG_GENERATOR_*` env が消え、新規参画者が「tag-generator は DB を持たない」ことをコードと docs の両方から読み取れる
- **タグ取得契約が proto に正式化**された。旧 `/api/v1/tags/batch` は hand-rolled JSON で `source` / `updated_at` が暗黙の仕様だったが、`ArticleTagEntry` proto message が正となり、Pact Broker での can-i-deploy 判定も `BackendInternalService` 側で一元化される
- **staging 起動コストが増えない**。新 RPC は既存の `recap-worker` profile が使う `rw-stub-alt-backend` stub に新ハンドラを足すだけで済むため、tag-generator Hurl suite を DB 付きで立てる案 (却下案) に比べて compose 起動時間の悪化がゼロ

### Cons / Tradeoffs

- **proto の公開サーフェスが増える**。`BackendInternalService` の RPC メソッドが 1 本増え、`internal.pb.go` と `internal.connect.go` の自動生成量も増えるが、alt-backend は内部 RPC なので外部互換性リスクは限定的
- **provider verification が alt-backend 側に重荷になる**。tag-generator の provider verification (`test_provider_verification.py`) から batch interaction が消えた一方、alt-backend の Pact provider test suite は `BackendInternalService/BatchGetTagsByArticleIDs` の検証を新たに負う。Broker 側で consumer version の主/deployed pact が両方乗るため、リリース時の mutual verification は自動で回る
- **一時的に consumer / provider を同じ PR に同居させる**必要がある。Pact Broker での deploy 順は通常 consumer 先・provider 後だが、本 increment は tag-generator から tags/batch を削除する時点で recap-worker がそれを呼ばなくなっている必要があるため、3 サービスを単一リリースで揃える (deploy.sh で Pact ゲートが pre-check する)
- **`scripts/build_label_graph.py` の psycopg2 依存が残る**。`tag-generator/app/pyproject.toml` の `psycopg2-binary` / `types-psycopg2` は本スクリプトのために残置。今回の runtime cleanup の目的からは外れるが、将来的に recap-worker の `genre_learning` RPC 経由に移行するまでの技術債として README に記録する必要がある
