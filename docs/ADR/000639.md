# ADR-000639: モバイル BottomNav の Library タブを Menu タブに置換し全機能アクセスハブを新設

## Status

Accepted

## Date

2026-04-07

## Affected Services

- alt-frontend-sv (TypeScript / SvelteKit)

## Context

[[000627]] でモバイルアプリシェルを 5-tab BottomNav に再構築し、[[000628]] で最適化を行った。しかし運用の結果、以下の課題が浮上した。

1. **機能へのアクセスが分散している** — Ask Augur、Settings、Statistics、Job Status、Explore 等のモバイル実装済み機能は Home タブの 3-dot メニュー (MobileMoreSheet) 経由でしかアクセスできず、ユーザーは Home に戻る必要があった
2. **Library タブの情報密度が低い** — `/feeds` はフィードリストのみで、Favorites (`/feeds/favorites`)、Viewed (`/feeds/viewed`)、Manage Feeds (`/settings/feeds`) 等の関連機能への導線がなかった
3. **3-dot メニューの発見性が低い** — bottom sheet はプログレッシブ・ディスクロージャーとして優れるが、「あの機能はどこだっけ？」という認知負荷を生んでいた

2026年のモバイル UX ベストプラクティス調査（NN/g、Material Design、各種 UX リサーチ）を踏まえ、以下の知見を設計に反映した。

- **"More" tab パターン**: 5 タブ以上の機能がある場合、最後のタブをメニューハブにするのは iOS / Android 両プラットフォームで確立されたパターン
- **Grid tile メニュー**: セクション分けされたタイルレイアウトは一覧性が高く、認知負荷が低い
- **Icon + label 必須**: ラベルなしアイコンは "mystery meat navigation" になるため必ず併用する
- **48dp タップターゲット**: WCAG AA / Material Design 準拠

## Decision

**Library タブを Menu タブに置換し、全モバイル実装済み機能へのアクセスハブとする。Home タブの 3-dot メニューは削除し、Menu タブに一本化する。**

### BottomNav 変更

| Before | After |
|--------|-------|
| Home / Swipe / Search / Recap / **Library** | Home / Swipe / Search / Recap / **Menu** |

- アイコン: `BookOpen` → `Menu` (Lucide)
- ルート: `/feeds` → `/menu`

### Menu ページレイアウト

3列グリッドタイルをセクション分けして配置する。

| セクション | 項目 |
|-----------|------|
| **Browse** | Library (`/feeds`), Favorites (`/feeds/favorites`), Viewed (`/feeds/viewed`), Explore (`/feeds/tag-trail`), Visual Preview (`/feeds/swipe/visual-preview`), Tag Verse (`/feeds/tag-verse`, Desktop badge) |
| **AI & Insights** | Ask Augur (`/augur`), Statistics (`/stats`), Job Status (`/recap/job-status`) |
| **Settings** | Manage Feeds (`/settings/feeds`) |
| **Admin** | Admin (`/admin/knowledge-home`, admin ロールのみ表示) |

### 設計判断

1. **Menu 項目を全ページ共通にする** — コンテキスト別に分けると「どのタブにあったか」で認知負荷が増す。共通メニューの方が学習コストが低い
2. **Home の 3-dot メニューを削除する** — Menu タブが唯一のアクセスハブとなり、ナビゲーション導線が一本化される
3. **共通ヘッダーコンポーネントは作らない** — Home と Library/Menu はヘッダー内容が大きく異なり（Home: digest/lens/recall, Menu: grid tiles）、無理に共通化すると over-abstraction になる
4. **Desktop では `/feeds` にリダイレクト** — デスクトップはサイドバーで全ナビゲーションが見えるため Menu ページは不要
5. **Library (`/feeds`) ページは残す** — Menu のタイルから遷移先として引き続き使用

### Alt-Paper デザイン準拠

- セクションラベル: `font-body`, `11px`, `uppercase`, `tracking-widest`
- タイルアイコン: `24px`, `strokeWidth 1.8`, `text-secondary`
- タイルラベル: `font-body`, `text-xs`, `text-primary`
- ヘッダー: `font-display`, `22px`, `font-bold`
- タイル: `rounded-xl`, `border`, `bg-surface`, `min-h-[72px]`
- タップフィードバック: `active:bg-[var(--surface-hover)]`
- Safe area: `pb-[calc(2.75rem+env(safe-area-inset-bottom))]`

## Consequences

### Pros

- **全機能がワンタップ圏内** — どのタブからでも Menu → 目的ページの 2 タップで到達
- **ナビゲーション導線の一本化** — Home の 3-dot メニューと Menu タブの二重構造を解消
- **発見性の向上** — グリッドタイル + セクションラベルで全機能が一覧でき、"mystery meat" を排除
- **拡張容易性** — `MENU_SECTIONS` にアイテムを追加するだけで新機能の導線を追加可能
- **テスト充実** — menu-page 12 テスト + bottom-nav 17 テスト = 29 テスト全パス

### Cons / Tradeoffs

- **Library への直接アクセスが 1 タップ増加** — 以前は BottomNav から直接 `/feeds` に行けたが、Menu → Library の 2 タップになった。ただし Library は Swipe タブほど頻用されないため影響は軽微
- **Home ヘッダーの空白化** — 3-dot ボタン削除により、Home ヘッダー右側は service quality indicator のみになった。将来の Home 固有アクション（例: TTS コントロール）の配置場所として活用可能
- **MobileMoreSheet / FloatingMenu の二重構造残存** — Swipe ページ (`/feeds/swipe`) には独自の `FloatingMenu` が残っている。将来的に Menu タブとの整合性を検討する余地がある
