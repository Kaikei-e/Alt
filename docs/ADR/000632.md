# ADR-000632: Gemma4+Ollama structured output の信頼性を reasoning-first フィールド順序と few-shot で改善する

## Status

Accepted

## Date

2026-04-06

## Affected Services

- news-creator (Python)
- rag-orchestrator (Go)

## Context

[[000630]] で導入した LLM 駆動 QueryPlanner を DI 配線し実稼働させたところ、eval ベンチマーク（12 golden cases）の pass rate が 6/12 → 1/12 に悪化した。rag-orchestrator のログ分析で以下の障害パターンが判明した:

1. **`should_clarify: true` の乱発**: 「量子コンピュータ 実用化」のような明確な単独クエリでも clarification を要求
2. **`resolved_query` の崩壊**: Transformer 関連クエリで `":"` のみが返される
3. **`retrieval_policy` の不正値**: `"general"`, `"search_queries"` 等、定義外の値が返される

`should_clarify: true` → `buildPromptWithQueryPlanner` が retrieval をスキップ → `answer_len=0` の連鎖で、ほぼ全クエリが空回答になった。

### 調査結果: Ollama structured output の制約

Web 調査（Ollama 公式ドキュメント、DSdev 2025、RANLP 2025、ACL 2025 Industry Track、NeurIPS 2024 等）で以下が判明:

- **Ollama の `format` パラメータは GBNF 文法変換で構文のみ保証**。意味的正確性（正しい enum 値の選択、boolean の適切な判定）は保証しない
- **モデルは `format` パラメータを見ない**。GBNF 文法はサンプリング時にトークンマスクとして適用されるが、プロンプトにスキーマ説明がなければモデルは出力構造を理解できない
- **フィールド順序が精度に直結**: `reasoning` フィールドを decision フィールドの前に配置すると +8pp の精度向上（DSdev 2025 実測）。モデルが推論を書き出してから判断することで、boolean や enum の品質が向上する
- **few-shot なしの小型モデルはスキーマ精度がほぼゼロ**（RANLP 2025）。文法制約は構文を保証するが、フィールドの意味的正確性は few-shot 例が必須
- **14B 未満のモデルでは constrained decoding が逆に精度を向上**させる（ACL 2025 Industry Track）。文法マスクが小型モデルの確率分布を安定化する効果がある
- **boolean フィールドの偏り**: Grammar-Aligned Decoding（NeurIPS 2024）により、トークンマスク後の確率再正規化で `true`/`false` の偏りが増幅される

## Decision

**QueryPlan スキーマに reasoning-first フィールド順序を導入し、few-shot 例とスキーマ記述をプロンプトに含め、LLM パラメータを最適化する。**

### 1. reasoning-first フィールド順序

`QueryPlan` Pydantic モデルのフィールド順序を変更し、`reasoning` を先頭に配置:

```python
class QueryPlan(BaseModel):
    reasoning: str       # 先に推論を書き出す
    resolved_query: str  # 推論結果に基づいてクエリを構成
    search_queries: list # 推論から派生する検索クエリ
    intent: str          # 推論で判断した意図
    ...
    should_clarify: bool # 最後に判断（推論が十分なら false）
```

Ollama の GBNF 文法変換はフィールド順序を保持するため、JSON 出力もこの順序でトークンが生成される。モデルが `reasoning` を書き終えてから `should_clarify` を判断するため、boolean の品質が構造的に向上する。

### 2. few-shot 例をプロンプトに含める

2 件の few-shot 例を追加:
- **単独クエリ例**: 明確なトピックに対して `should_clarify: false`、適切な intent と search_queries を示す
- **follow-up 例**: 会話履歴からの照応解決を示し、「それ」を具体的なトピックに展開する過程を reasoning で記述

### 3. スキーマ記述をプロンプトに含める

`format` パラメータはモデルに見えないため、各フィールドの説明・許可値・デフォルト値をプロンプト本文に記述。`retrieval_policy` は `global_only` / `article_only` の 2 択に簡略化（`tool_only` / `no_retrieval` は使用実績なし）。

### 4. LLM パラメータ最適化（モデルリロード回避保証）

| パラメータ | Before | After | 根拠 |
|-----------|--------|-------|------|
| temperature | 0.2 | 0 | スキーマ遵守最大化 |
| num_predict | 256 | 512 | reasoning フィールド追加分 |
| repeat_penalty | 1.1 | 削除（ベース値に委譲） | temperature=0 で不要 |

`options` dict には `temperature` と `num_predict` のみを渡す。gateway の `_merge_options()` が `config.get_llm_options()` をベースにマージするため、`num_ctx=8192`, `num_batch=1024`, `num_keep=-1` は自動で一致し、**モデルリロードは発生しない**。モデル名は `gemma4-e4b-12k`（expand-query と同一）。

### 5. rag-orchestrator 側の sanity check

LLM 出力の意味的正確性は保証されないため、クライアント側で防御的に正規化:
- `resolved_query` が 2 rune 未満 → 元のクエリにフォールバック
- 会話履歴なし + `should_clarify: true` → false に矯正
- `retrieval_policy` の不正値 → `global_only` にマッピング（`"general"`, `"search_queries"` 等を正規化）
- `search_queries` 空 → `[resolved_query]` で補完

## Consequences

### Pros

- **回答生成復帰**: 1/12 → 10/12 のケースで回答が生成される（`should_clarify` 乱発が解消）
- **Citation 回復**: 1/12 → 9/12 で citation 付き回答
- **Fallback Rate 改善**: 0%（空回答）→ 16.7%（2 件のみ、正当な retrieval 不足理由）
- **モデルリロード回避**: パラメータ設計で GPU 上のモデル入れ替えが発生しない
- **構造的な品質保証**: reasoning-first はフィールド順序という機械的な仕組みで品質を担保。プロンプトの文言調整に依存しない

### Cons / Tradeoffs

- **Intent 分類精度はまだ低い**: causal → `general`、comparison → `general` への誤分類が多い（7/12 件）。few-shot 例で intent の種類をさらに網羅するか、intent 分類をルールベースで補完する必要がある
- **reasoning フィールドの追加トークン**: num_predict を 256 → 512 に増やしたが、レイテンシは約 1-2 秒増加
- **few-shot 例の網羅性**: 2 件では causal/comparison/synthesis 等の intent を十分にカバーできない。例の追加が今後の改善レバー
- **Follow-up 回答が短い**: 照応解決は改善されたが、follow-up クエリの retrieval で関連コンテンツが不足するケースが残存

### ベンチマーク結果推移

| 指標 | Baseline | Round 2 (配線修正) | Round 3 (DI配線) | Round 4 (本ADR) |
|------|----------|-------------------|-----------------|----------------|
| Pass Rate | 0/12 | 6/12 | 1/12 | 2/12 |
| 回答生成 | 12/12 | 12/12 | 3/12 | **10/12** |
| Citation 付き | 0/12 | 8/12 | 1/12 | **9/12** |
| Fallback Rate | 33% | 25% | 0% (空回答) | **17%** |
