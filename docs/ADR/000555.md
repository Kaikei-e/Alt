# ADR-000555: StreamSummarize を nginx streaming location に収容し HTTP/3 回帰を防止する

## Status

Accepted

## Date

2026-03-24

## Affected Services

- nginx（リバースプロキシ設定）
- alt-frontend-sv (TypeScript/SvelteKit)

## Context

[[000553]] で StreamSummarize を Connect-RPC に移行しハートビートを追加したが、ブラウザで `ERR_QUIC_PROTOCOL_ERROR 200 (OK)` が発生し、ストリーミングテキストが UI に表示されない問題が残った。

### 直接原因

`POST /api/v2/alt.feeds.v2.FeedService/StreamSummarize` が nginx の streaming 専用 location にマッチしていなかった。

nginx の streaming location は `alt.(augur|morning_letter).v2` のみを対象にしていた:

```nginx
# Before: feeds が含まれていない
location ~ ^/api/v2/alt\.(augur|morning_letter)\.v2\..+/Stream {
    proxy_buffering off;
    ...
}
```

結果として StreamSummarize は汎用の `/api/` location に fallback し、以下の不適切な設定で処理されていた:

- `proxy_buffering on` — レスポンスがバッファされ、チャンクがリアルタイムに転送されない
- `Connection: upgrade` ヘッダー — Connect-RPC のストリーミングプロトコルと不整合
- `Alt-Svc` ヘッダーなし — Cloudflare が HTTP/3 (QUIC) を使用し、Connect-RPC のバイナリストリーミングフレームで `ERR_QUIC_PROTOCOL_ERROR` が発生

### 副次的原因

SvelteKit の Connect-RPC プロキシ（`+server.ts`）が streaming レスポンスに `Alt-Svc: clear` を付与していなかったため、Cloudflare ↔ ブラウザ間が HTTP/3 で通信し、Connect-RPC の長時間ストリーミングで QUIC プロトコルエラーが発生した。

## Decision

### 1. nginx streaming location に `feeds` を追加する

```nginx
# After: feeds を追加
location ~ ^/api/v2/alt\.(augur|morning_letter|feeds)\.v2\..+/Stream {
    proxy_buffering off;
    proxy_request_buffering off;
    ...
    add_header Alt-Svc "clear" always;
    add_header Cache-Control "no-cache, no-transform" always;
    add_header X-Content-Type-Options "nosniff" always;
}
```

これにより StreamSummarize と StreamFeedStats が streaming 専用の設定で処理される。`proxy_request_buffering off` も追加し、リクエストボディのバッファリングも無効化した。

Knowledge Home の streaming location にも同じヘッダーを追加し、一貫性を確保した。

### 2. SvelteKit プロキシに `Alt-Svc: clear` を追加する

Connect-RPC streaming レスポンスに `Alt-Svc: clear` を付与し、ブラウザに HTTP/3 の使用を停止させる。これにより Cloudflare ↔ ブラウザ間が HTTP/2 にフォールバックし、`ERR_QUIC_PROTOCOL_ERROR` を回避する。

### 3. 回帰防止テストを追加する

- `connect-rpc-proxy-route.test.ts` — SvelteKit プロキシが streaming レスポンスに `X-Accel-Buffering: no`, `Cache-Control: no-cache, no-transform`, `Alt-Svc: clear` を付与することを検証
- `nginx-connect-streaming-config.test.ts` — nginx 設定で feeds streaming が専用 location に入ることを検証

## Consequences

### Pros

- StreamSummarize が適切な nginx streaming 設定で処理され、チャンクがリアルタイムにブラウザに到着する
- `Alt-Svc: clear` により HTTP/3 の QUIC プロトコルエラーが回避される
- 回帰防止テストにより、将来の nginx 設定変更やプロキシ変更で streaming が壊れにくくなる
- Knowledge Home streaming location にも同じヘッダーが追加され、一貫性が確保された

### Cons / Tradeoffs

- nginx 設定にサービス名（`feeds`）がハードコードされるため、新しいサービスの streaming RPC を追加する際に nginx 設定の更新が必要

### 補足: `Alt-Svc: clear` は Cloudflare に上書きされるため撤去

nginx と SvelteKit プロキシに `Alt-Svc: clear` を設定したが、Cloudflare は `alt-svc` ヘッダーを保護しており、オリジンの値を常に `h3=":443"; ma=86400` で上書きする（Transform Rules でも Worker でも除去不可）。実環境のレスポンスヘッダーで確認済み。

HTTP/3 QUIC エラーの根本的な解決は **Cloudflare ダッシュボードでゾーン全体の HTTP/3 を OFF** にすることで対処した。将来 HTTP/3 が Web ページパフォーマンスに必要になった場合は、RPC 用サブドメイン（例: `api.curionoah.com`）を分離して HTTP/3 OFF を個別設定する。

`Alt-Svc: clear` のコードは無効のため撤去した。
