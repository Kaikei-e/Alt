# ADR-000853: Knowledge Loop Wave 4 を 4 つの sub-wave (cross-service emit × 2 / 実 resolver / RLS rollout) に分割し、各 sub-wave を独立 PR + Pact CDC + alt-deploy 同期で landed させる

## Status

Accepted

## Date

2026-04-26

## Affected Services

- rag-orchestrator: Wave 4-A で `AugurConversationLinked` を knowledge-sovereign の `AppendKnowledgeEvent` 経由で emit。新規 sovereign client driver + Pact consumer test を追加
- recap-worker: Wave 4-B で `RecapTopicSnapshotted` を knowledge-sovereign の `AppendKnowledgeEvent` 経由で emit。Rust 側 sovereign client + Pact consumer test を追加
- knowledge-sovereign: Wave 4-C で `EventLogSurfaceScoreResolver` を実装し `Projector.WithScoreResolver` で plug、WhyMappingVersion を v5 → v6 に bump。Wave 4-D で `00014_add_rls_to_knowledge_loop_entries.sql` 投入と read caller の session-var 配線
- alt-deploy: 各 sub-wave で services.yaml / release-deploy.yaml / run-e2e-suite.yml の pacticipant role + e2e allowlist を手動同期 ([[alt_deploy_pacticipant_sync]])

## Context

Wave 1 ([[000850]]) / Wave 2 ([[000851]]) / Wave 3 ([[000852]]) では Knowledge Loop の構造的な足場 (proto contract additive 拡張、Atlas migration、pure function 化された `decideBucketV2` / `computeChangeDiff` / `buildChangeSummaryJSON`、Prometheus 観測基盤、`SurfaceScoreResolver` port + `NullSurfaceScoreResolver` default + `Projector.resolveBucket` 経由の null-equivalent wiring) をすべて additive に landed させた。

残課題は **live behavior を実際に変える 4 つの作業**で、性質上いずれも cross-service の Pact CDC contract と alt-deploy の pacticipant 同期を伴う:

1. `AugurConversationLinked` を rag-orchestrator が emit (augur が会話を loop entry にリンクするタイミング、`CreateSessionFromLoopEntry` 成功後)
2. `RecapTopicSnapshotted` を recap-worker (Rust) が emit (recap pipeline が `recap_subworker_clusters` を確定したタイミング)
3. knowledge-sovereign 内の `Projector` が、上の 2 events + `SummaryVersionCreated` 等を `knowledge_events` から payload-resident に拾って `SurfaceScoreInputs` を組み立てる real `EventLogSurfaceScoreResolver` を実装し、`WithScoreResolver` 経由で plug
4. `knowledge_loop_entries` に RLS policy を当てて F-001 (cross-user evidence_refs leak) の defense-in-depth を完成

これら 4 件をどう分けるかが本 ADR の主題。次の制約から「1 つの大 PR」は採らない:

- 各 cross-service emit は **新しい consumer-provider Pact contract** を要求する。consumer = rag-orchestrator / recap-worker、provider = knowledge-sovereign。Phase 1b playbook 通りに consumer pact が無いまま provider 側に新要件 (今回は `AppendKnowledgeEvent` への新 event_type) を入れると、過去 X-Service-Token 401 cascade ([[000722]] 系) と同型の事故になる
- alt-deploy の `services.yaml` / `release-deploy.yaml` の `PACTICIPANT_ROLES` / `run-e2e-suite.yml` allowlist は手動同期 ([[alt_deploy_pacticipant_sync]])。consumer / provider 配線が増えるたびに 3 ファイル更新が漏れがち
- RLS は順序を間違えると本番で `knowledge_loop_entries` が全行不可視になり UI が空白になる。caller 全員が `SET LOCAL alt.user_id = ...` を毎クエリ発行する側を先に出荷し、その後で `ENABLE ROW LEVEL SECURITY` migration を投入する逆順制約がある
- WhyMappingVersion bump は full reproject runbook ([[knowledge-loop-reproject]]) を伴うので、bump の意味が立ったタイミング (新 WhyKind を実際に emit する resolver が動き出すタイミング) に同期させたい

これらをまとめてひとつの PR にすると、Pact contract 抜けや services.yaml の漏れ、RLS の順序逆転が混じったときに切り分けが効かなくなる。

## Decision

Wave 4 を以下の 4 つの sub-wave に分割し、それぞれ独立した PR + Pact CDC + alt-deploy 同期を伴う形で landed させる。詳細実装は同セッションで作成した `docs/plan/knowledge-loop-wave4-implementation-plan.md` に文書化されており、本 ADR はその採用を確定する。

### Wave 4-A — rag-orchestrator → knowledge-sovereign / `AugurConversationLinked` emit

- `internal/adapter/sovereign_client/` を新設 (alt-backend の `driver/sovereign_client/` を参考に Connect-RPC client を作る)
- consumer Pact を `internal/adapter/sovereign_client/contract/append_event_contract_test.go` に追加し、`event_type = augur.conversation_linked.v1` の `AppendKnowledgeEvent` を pin。service token / mTLS の現行要件があれば必ず matcher で固定する (Phase 1b playbook)
- usecase + port を介して augur handler `CreateSessionFromLoopEntry` の成功後に warn-and-continue で emit
- knowledge-sovereign の provider verification (`app/driver/contract/provider_test.go`) に rag-orchestrator pact を追加
- alt-deploy の `services.yaml` / `release-deploy.yaml` / `run-e2e-suite.yml` を更新

### Wave 4-B — recap-worker → knowledge-sovereign / `RecapTopicSnapshotted` emit

- `recap-worker/recap-worker/src/clients/sovereign.rs` (Rust) を新設、Connect-RPC client
- `pact_consumer` (Rust) で consumer pact を作成、`recap-worker/recap-worker/src/clients/sovereign_contract.rs` (gen-ignored / cargo test --ignored)
- recap pipeline の `recap_subworker_clusters` 確定箇所に emit を組込み、failure は warn-and-continue
- provider 側 verification + alt-deploy 同期は 4-A と同型

### Wave 4-C — `EventLogSurfaceScoreResolver` 実装

- `knowledge-sovereign/app/usecase/knowledge_loop_projector/score_resolver_event_log.go` を新設
- 入口で必ず `event.user_id` を SQL の bind パラメータとして渡し、cross-DB / cross-row の latest 参照を行わない (F-001 物理境界)
- 時刻窓は `event.occurred_at - 7d ≤ ev.occurred_at < event.occurred_at` で固定。`time.Now()` / `NOW()` を business fact に登場させない
- `Projector.WithScoreResolver(NewEventLogSurfaceScoreResolver(...))` を `main.go` で wire-up
- `WhyMappingVersion` を v5 → v6 に bump し、`enricher.go` に `topic_affinity_why` / `tag_trending_why` / `unfinished_continue_why` の case を追加
- `surface_planner_version` 列を SQL UPSERT で 2 として書く wiring を同時に投入 (Wave 3 では未投入)
- `runbooks/knowledge-loop-reproject.md` の v6 行に正式な日付と本 ADR 番号を埋め直す

### Wave 4-D — RLS policy on `knowledge_loop_entries`

- 順序: caller 配線 → metric 確認 → migration 投入
  1. read caller (sovereign 自身の handler / read RPC、alt-backend の sovereign_client 経路) に `SET LOCAL alt.user_id = $1` を毎クエリ最初に発行する patch を出荷
  2. log-based metric で「session-var 未設定」が 0 件であることを確認
  3. `00014_add_rls_to_knowledge_loop_entries.sql` で `ENABLE ROW LEVEL SECURITY` + policy `USING (user_id::text = current_setting('alt.user_id', true))` を投入
- pgbouncer は transaction pooling 前提なので `SET LOCAL` を毎トランザクションで発行する。`SET` (session) は pooling 越えで漏れるため不可
- 切り戻し: `git revert` で caller 配線を戻し、policy DROP migration を逆向き投入

### 検討した代替案と却下理由

- **Wave 4 を 1 PR にまとめる**: Pact contract 抜けや services.yaml 抜けが review で気づきにくい。失敗時に git revert したい範囲が広がる
- **augur emit と recap-worker emit を 1 セッションでまとめる**: codebase が Go と Rust で別、Pact testing も別ツール (`pact-go` vs `pact_consumer` Rust)。mental context switching コストの方が高い
- **RLS migration を Wave 3 で先行**: caller 配線が landed していない状態で migration が当たると本番 UI 空白。順序逆転 fail-closed リスク
- **WhyMappingVersion を Wave 3 で先行 bump**: 新 WhyKind を実際に emit する resolver が動いていない状態で bump しても empty-mapping 領域が広がるだけ。bump は 4-C 内の resolver 実装と同時が正しい
- **rag-orchestrator が直接 knowledge-sovereign DB に書く**: append-first invariant ([[knowledge-loop-canonical-contract]] §3.1) を破壊する。必ず `AppendKnowledgeEvent` RPC 経由
- **alt-backend の `BackendInternalService` に新 RPC を追加して経由**: hop が増えるだけで benefit がない。sovereign が source of truth なので直接叩く方が自然

## Consequences

### Pros

- 各 sub-wave が独立 commit / deploy 可能。失敗時は対象 sub-wave のみ git revert + 再 deploy で 1 段階だけ戻せる
- Pact CDC consumer-provider 関係が sub-wave 単位で 1 対 1。`scripts/pact-check.sh --broker` の出力が読みやすく、broker matrix の状態が把握しやすい
- alt-deploy の services.yaml 同期も sub-wave 単位で済む。Pacticipant 抜けに気付くチェックが各 PR で効く
- RLS rollout の順序制約 (caller 先、migration 後) が独立 sub-wave (4-D) で明示的に守られる。1 つの PR に混ぜると順序逆転を見落としやすい
- Wave 1/2/3 で landed させた null-resolver wiring が 4-C で `WithScoreResolver` 一行で real implementation に切り替わる。回帰検出が `bun run test:e2e` + `surface_bucket_assigned_total{version="v2"}` の立ち上がりで取れる

### Cons / Tradeoffs

- 4 sessions に分けるので、Wave 4 完成までのカレンダ上のリードタイムは 1 PR 案より長い (それぞれ別日)
- WhyMappingVersion v6 の reproject 実施は 4-C 投入後の運用作業。runbook 履歴行を埋める作業が後追いになる
- alt-deploy の services.yaml / release-deploy.yaml / run-e2e-suite.yml の手動同期は機械化されておらず、抜けると pact-publish/gate が silent skip ([[alt_deploy_pacticipant_sync]])。各 sub-wave で `scripts/compose-image-audit.py` 相当のゲートが pacticipant 側にも欲しいが現状は手動 review に依存
- RLS rollout の caller 配線先・migration 後を破ると本番 UI 空白。runbook で fail-closed 仕様を強調しても、慎重さに依存する運用負債が一定残る
- recap-worker の Rust Pact 経路は `pact_consumer` をテスト時のみロードする `--ignored` パターンで、CI 上の test 工程設計が他 Go consumer と異なる ([[000749]] 等の知見)。新 sub-wave 4-B はこの違いを踏襲する
