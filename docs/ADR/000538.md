# ADR-000538: Projection version activate の silent failure を修正し toVersion 自動作成を導入する

## Status

Accepted

## Date

2026-03-23

## Affected Services

- alt-backend (Go)

## Context

v2→v3 の reproject run（`45afb5d6`）が `swapped` まで正常に進行したにもかかわらず、`knowledge_projection_versions` テーブルに version 3 が `active` として存在しない状態が発生した。この結果、以下の障害が継続していた:

1. **knowledge projector が毎回 fallback** — `GetActiveProjectionVersion` が `no rows in result set` を返し、projector はデフォルト version で動作。v3 で replay された結果（dismiss replay 含む）が反映されていなかった
2. **sovereign-reconciliation job が常時失敗** — active version がないため `CompareProjections` が実行できなかった
3. **全バージョンが `inactive`** — v1 も v2 も inactive にされたが、v3 は行が存在しないため active にできなかった

### 根本原因の調査

DB 状態の調査により、2つのバグが連鎖して障害を引き起こしていたことが判明した。

**バグ 1: `ActivateProjectionVersion` の silent failure**

```go
// 修正前: rows_affected を検証していない
_, err = r.pool.Exec(ctx,
    `UPDATE knowledge_projection_versions SET status = 'active', activated_at = $2 WHERE version = $1`,
    version, now)
```

この UPDATE は `WHERE version = 3` で実行されるが、v3 の行が `knowledge_projection_versions` に存在しなかったため `rows_affected = 0` で空振り。しかしエラーを返さないため、`SwapReproject` は成功として処理を続行した。

同じトランザクション内で先に実行される deactivate SQL（`SET status = 'inactive' WHERE version != 3 AND status = 'active'`）により v2 が `inactive` に遷移。結果として全バージョンが `inactive` になった。

**バグ 2: `StartReproject` が toVersion の行を作成しない**

`StartReproject("full", "v2", "v3", ...)` は `knowledge_reproject_runs` レコードを作成するが、`knowledge_projection_versions` テーブルに v3 の行を INSERT しない。v1→v2 の reproject では v2 の行が事前に（別の操作で）作成されていたため問題が顕在化しなかったが、v2→v3 では v3 の行が作成されないまま reproject が完了し、swap 時に silent failure が発生した。

### 時系列

| 時刻 | 操作 | 結果 |
|---|---|---|
| 3/22 02:56 | `StartReproject("full", "v2", "v3")` | reproject_run 作成。**v3 行は未作成** |
| 3/22 03:28 | reproject 完了 → swappable | 763,398 events 処理、v3 の home_items 40,691 件作成 |
| 3/22 (swap) | `SwapReproject` 実行 | `ActivateVersion(3)` → v2 を inactive に → v3 activate は**空振り** → run は `swapped` に |
| 3/22〜3/23 | projector 毎回エラー | fallback でデフォルト version を使用 |

## Decision

2つのバグを修正し、データを復旧する。

### 修正 1: `ActivateProjectionVersion` に rows_affected 検証を追加

`knowledge_projection_version_repository.go` の `ActivateProjectionVersion` で、activate の UPDATE 後に `commandTag.RowsAffected() == 0` を検証し、version が存在しない場合は明示的にエラーを返す。

```go
commandTag, err := r.pool.Exec(ctx,
    `UPDATE knowledge_projection_versions SET status = 'active', activated_at = $2 WHERE version = $1`,
    version, now)
if err != nil {
    return fmt.Errorf("ActivateProjectionVersion activate: %w", err)
}
if commandTag.RowsAffected() == 0 {
    return fmt.Errorf("ActivateProjectionVersion: version %d not found in knowledge_projection_versions", version)
}
```

これにより `SwapReproject` が `ActivateVersion` のエラーを受け取り、swap を中止して呼び出し元にエラーを伝播する。

### 修正 2: `StartReproject` で toVersion の行を自動作成

`knowledge_reproject_usecase/usecase.go` の `StartReproject` に、toVersion の `knowledge_projection_versions` 行を自動作成するロジックを追加する。

```go
if u.createVersionPort != nil {
    targetVersionNum, parseErr := strconv.Atoi(strings.TrimPrefix(strings.ToLower(toVersion), "v"))
    if parseErr == nil {
        _ = u.createVersionPort.CreateVersion(ctx, domain.KnowledgeProjectionVersion{
            Version:     targetVersionNum,
            Description: fmt.Sprintf("V%d reproject from %s", targetVersionNum, fromVersion),
            Status:      "inactive",
            CreatedAt:   now,
            ActivatedAt: &now,
        })
        // duplicate key error は無視（既存 version がある場合は何もしない）
    }
}
```

`createVersionPort` は variadic パラメータとして `NewUsecase` に追加し、既存の呼び出し元との後方互換を維持する。

### データ復旧

`knowledge_projection_versions` に v3 行を INSERT し `status = 'active'` に設定:

```sql
INSERT INTO knowledge_projection_versions (version, description, status, created_at, activated_at)
VALUES (3, 'V3 reproject via Sovereign write path', 'active', NOW(), NOW());
```

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|---|---|---|
| `ActivateProjectionVersion` を UPSERT に変更 | version 行が存在しない状態は本来異常 | UPSERT で暗黙的に行を作ると、StartReproject のバグが隠蔽される |
| `SwapReproject` 側で version 存在チェックを追加 | defense in depth として有効 | driver 層（ActivateProjectionVersion）で検証する方が Clean Architecture に準拠 |
| `createVersionPort` を必須パラメータに変更 | 設計的には正しい | 既存テスト 10+ 件の constructor 呼び出しを全て変更する必要があり影響が大きい |

## Consequences

### Pros

- **Silent failure の解消** — 存在しない version の activate が明示的にエラーになり、swap が中止される
- **Version 行の自動作成** — `StartReproject` で toVersion の行が確実に存在するようになり、同じ障害が再発しない
- **既存テスト完全保持** — `createVersionPort` は variadic パラメータのため、既存の `NewUsecase` 呼び出しを変更する必要がない
- **データ復旧完了** — v3 が active に設定され、projector の fallback エラーと sovereign-reconciliation の失敗が解消

### Cons / Tradeoffs

- **duplicate key error の無視** — `CreateVersion` の失敗を `_ =` で無視しているため、DB 接続エラー等の本質的な失敗も握りつぶす可能性がある。ただし、version 行の作成は「便宜的な補助」であり、失敗しても reproject 自体は（v3 行なしでも）動作する。真の修正は `ActivateProjectionVersion` の rows_affected 検証である
- **variadic パラメータ** — `createVersionPort` が optional のため、`nil` のまま `NewUsecase` を呼ぶ古いコードでは toVersion 自動作成が機能しない。DI container では正しく配線済み
