# ADR-000927: Ask Augur の RAG 回答経路に CitationKind discriminator を貫通配線し近傍記事を inline projection する

## Status

Accepted

## Date

2026-05-27

## Affected Services

- proto (`alt.augur.v2`): `MetaPayload` / `DonePayload` / `ChatMessage` に `repeated Citation related_citations` を additive 追加
- rag-orchestrator (Go): retrieval / usecase / handler / repository に ArticleID 配線と近傍検索を追加
- rag-migration-atlas: `augur_messages.related_citations JSONB` カラム追加
- alt-frontend-sv (TypeScript / Svelte): CitationRail に Related 区画と Title フォールバック多重化
- alt-backend (Go), alt-butterfly-facade (Go): 再生成 stub の受け入れ (passive consumer)

## Context

ユーザ報告:「Ask Augur に質問しても元記事が参照されずに回答が返る」。直近の fix 群 ([[000836]] / [[000924]] / [[000925]] / [[000926]]) が augur citation 周辺に複数回入っているのに、症状は解消していなかった。

### 過去 ADR で解決しなかった理由 (evidence-backed)

`/log-seeker` 調査と code 探査で判明したのは、過去 ADR が直したのは **Knowledge Loop → Augur handshake 経路** (`CreateAugurSessionFromLoopEntry`) のみで、**「ユーザが Ask Augur に普通に質問する RAG retrieve 経路」** には discriminator が一度も配線されていなかった、というレイヤずれだった。

- [[000836]] BFF が sovereign で resolve してから渡す前提を確立。`CreateAugurSessionFromLoopEntry` のみカバー
- [[000924]] `CodeNotFound` を alt-backend で透過。citation 内容には触れていない
- [[000925]] `act_targets[0].source_url` 欠落の knowledge-sovereign 側 fix。/loop の Open ボタン回復で Augur citation とは別レイヤ
- [[000926]] `CitationKind` discriminator を proto / FE / Loop handshake / projector に配線。**ただし `convertCitationsToProtoCitations` / `convertContextsToCitations` (RAG retrieve 経路) は kind/ref_id を埋めず取り残された**。`UNSPECIFIED` fallback が暗黙の安全弁になり、server 側で「kind 忘れ」が検知不能になっていたことは [[000926]] Cons §3 で予見されていたが、当時の影響範囲が Loop seed のみと判断されていたため未着手

加えて、構造的な drop point があった:

1. **`domain.SearchResult.ArticleID` (`internal/domain/repository.go:103`) は HybridSearch / SearchClient で取得できていた**が、`SearchResult → ContextItem` 変換 (`retrieval/fuse_results.go:122-131`、`retrieval/allocate.go:61-69` / `106-114`) で **`ArticleID` を写していなかった**
2. `usecase.ContextItem` (`retrieve_context_usecase.go:33` と `retrieval/types.go:44` の 2 定義) にも `ArticleID` フィールドが**存在しなかった**
3. `usecase.Citation` (`rag_answer_types.go:33`) も `ChunkID` / `URL` / `Title` / `Score` / `DocumentVersion` のみで `ArticleID` を持たなかった
4. 結果として `convertCitationsToProtoCitations` (`handler.go:458-469`) と `convertContextsToCitations` (`handler.go:445-456`) は `Url` / `Title` だけ書き、`Kind = UNSPECIFIED` / `RefId = ""` のまま wire に流していた
5. FE 側 `citation-href.ts` の `default` 分岐で `href = undefined` となり、`CitationRail.svelte:144-148` で `<span>` (disabled) 表示 → ユーザには「citation が無い / リンクできない」ように見える

`/log-seeker` 観測 (2026-05-27 09:52Z, 過去 1h):

```
alt-backend: 1001 ERROR ; pre-processor: 5 ; news-creator: 1
```

エラー内訳は robots.txt / GetFeedID / projection checkpoint が主で、citation 直接エラーは出ていない。これは **silent failure mode** (UNSPECIFIED で書かれ続け wire は通る) に陥っていた裏付けで、サーバ側ログでは検知できない構造的負債だった。`augur` / `acolyte` / `alt-backend` のコンテナがローカルで停止していたことから、ADR-926 の binary は production にも未反映と推定。

### 追加要件: 近傍記事の引用

ユーザは「もし記事が参照されたら、その記事プラス α で近傍の記事も引用」も要件として提示。Perplexity の "Sources" / "Related" 風に **direct (LLM が引いた) と related (システムが近傍として提示する)** を **分けて見せる** UX を求めている。Alt の Newspaper Style と整合させつつ、append-only / reproject-safe 不変条件 ([[IMPL_BASE]]) を壊さない設計が必要だった。

## Decision

`CitationKind` discriminator (kind=ARTICLE/WEB/SUMMARY/UNSPECIFIED) を **RAG retrieve 経路にも貫通配線**し、direct citation の確定と同時に近傍記事 (k=3) を **同一 INSERT に inline projection** する形で persist する。proto は flat sibling の `repeated Citation related_citations` を `MetaPayload` / `DonePayload` / `ChatMessage` に additive 追加し、UI は Sources と Related を 2 セクションで描画する。

### 採用設計

1. **ArticleID 貫通配線**
   - `retrieval.ContextItem` / `usecase.ContextItem` 両方に `ArticleID string` を additive 追加
   - `retrieval/fuse_results.go`, `retrieval/allocate.go` の `SearchResult → ContextItem` 変換ループで `ArticleID: res.ArticleID` を書く
   - `usecase.Citation` に `ArticleID string` と `PublishedAt string` を additive 追加
   - `buildCitations` (`answer_with_rag_usecase.go:754`) で `meta.ArticleID` / `meta.PublishedAt` を `Citation` に写す。同メソッドは 3 callsite (`answer_with_rag_usecase.go:304` / `rag_answer_stream.go:527` / `stream_hybrid_longform.go:380`) から共有されており単一定義の修正で全経路をカバー

2. **handler 変換層の kind 判定**
   - 新ヘルパ `classifyCitation(c usecase.Citation) (kind, refID)`: `ArticleID` が UUID parse 可能なら `kind=ARTICLE` + `refId=ArticleID`。そうでなく URL が http(s) なら `kind=WEB`。いずれでもなければ `UNSPECIFIED` のまま (= FE 側で disabled span)
   - 新ヘルパ `sanitizeCitationTitle(title)`: 8-4-4-4-12 hex (canonical UUID) パターンに matche する Title は空文字に正規化。後段 FE の `formatDomain(URL)` → "Untitled source" フォールバックが取り回す。**UUID 文字列を Title として絶対に wire に乗せない**多重防御
   - `convertCitationsToProtoCitations` (done event) と `convertContextsToCitations` (meta event) 両方に適用

3. **近傍記事の inline projection**
   - `domain.HybridSearcher` interface に additive method `SearchNeighbors(ctx, queryVector, queryText, seedArticleIDs, limit) ([]SearchResult, error)` を追加
   - 実装 (`adapter/repository/hybrid_search_repo.go`) は既存 HybridSearch の RRF (pgvector + tsvector) パイプラインを派生。`text_matches` arm は常に走り、`queryVector` が空でない時のみ `vector_matches` arm を CTE に含める (= encoder fallback でも lexical neighbors は取れる)
   - `WHERE d.article_id NOT IN ($seeds)` ではなく **post-filter 方式** を採用: candidate pool を `limit + len(seeds)` に inflate、metadata enrichment 後に seed を drop。理由は `rag_chunks` が `article_id` を直接持たず JOIN 後にしか確定しないため、SQL のフィルタ条件が複雑化するのを避けた
   - `answer_with_rag_usecase.buildRelatedCitations(ctx, direct, originalQuery)`:
     - `neighborSearcher == nil` または `len(direct) == 0` または UUID-parseable な seed が 0 件のとき **何も呼ばず nil 返却** (= ユーザ要件の「もし記事が参照されたら」の "if" を厳密に守る)
     - synthetic query は cited titles を空白結合した文字列。タイトル不在なら user query にフォールバック
     - searcher が err を返した場合は `Warn` ログ 1 行と nil 返却 (UX 退化は受容、回答全体は壊さない)
   - 3 callsite (Execute / Stream / Hybrid Longform) すべてで `output.RelatedCitations` を埋める。fallback パス (`prepareFallback`) は意図的に呼ばない (direct citation 不在のため)

4. **persistence (append-only 維持)**
   - `domain.AugurMessage` に `RelatedCitations []AugurCitation` を additive 追加。`AugurMessage` は append-only ([[000836]] 由来) を継承
   - `AppendAssistantTurn(ctx, convID, content, citations, relatedCitations)` に第 5 引数を追加。done event の **1 つの INSERT** で `citations` と `related_citations` を同時 commit。**partial-flush パス** (client abort) は `relatedCitations = nil` で書き、後追い backfill は行わない
   - Atlas migration `20260527100000_add_related_citations.sql`: `ALTER TABLE augur_messages ADD COLUMN related_citations JSONB NOT NULL DEFAULT '[]'::jsonb`。既存行は default で穏便に読める

5. **proto wire layer**
   - `MetaPayload` (preview event) は `related_citations` を**送らない**: neighbor は done event 確定後に 1 度だけ計算される設計のため、meta event は常に空配列
   - `DonePayload` (authoritative event) と `ChatMessage` (履歴読み出し) が authoritative source。`GetConversation` の読み出しパスは persisted 両カラムを wire に乗せる
   - 3 proto message 全てに `related_citations` を additive で配置 (`buf breaking` clean)

6. **frontend Related セクション**
   - `connect/augur.ts` に `safeTitle(title)` を追加し UUID-only Title を空に正規化。`mapProtoCitations` 経由で direct / related の両方が同じ衛生処理を経由
   - `AugurStreamResult.relatedCitations` を additive 追加。`getAugurConversation` の `AugurStoredMessage.relatedCitations` も同様
   - `CitationRail.svelte` の props に `relatedCitations?: Citation[]` を追加。`relatedCitations.length > 0` のときのみ "Related" `<header>` + `<ol>` を描画 (legacy 会話は空配列で section ごと消える)
   - 視覚的差別化: Sources は番号 (`01`...) + accent-primary border、Related は `★` プレフィックス + accent-secondary + `font-size: 0.82rem` + `opacity: 0.92`。Newspaper Style の見出し階層 (Playfair Display + IBM Plex Mono) を維持
   - `displayTitle(c)` で **Title → formatDomain(URL) → "Untitled source"** の 3 段フォールバック。`refId` を visible text には絶対使わない
   - `AugurChat.svelte` の `railRelatedCitations` (`$derived.by`) が最新 assistant turn の `relatedCitations` を引いてレールに渡す。`useAugurPane.svelte.ts` (inline pane) も同型 plumbing
   - `convertCitations` を `(citations: AugurCitation[] | undefined) => Citation[]` に defensive 化。既存 1356 テストの互換を維持しつつ新コードパスを安全化

### 検討した代替案と却下理由

#### (A) `Citation` に `enum CitationRole { DIRECT, RELATED }` を追加して単一配列にする

却下。Plan agent はこちらを推奨したが、ユーザは flat sibling 配列を明示的に選択した。理由として、Sources と Related は **意味が異なる** (LLM が引いた根拠 vs システムが提示した次に読むべきもの)、Perplexity / ChatGPT などの先行例も別セクション、FE の `groupBy(role)` ロジックを避けたい、の 3 点。enum 拡張時の同期コストは小さくない。

#### (B) `Citation` を nested `repeated Citation related = N` で持たせる (per-citation neighbors)

却下。1 message に direct 3 × neighbors 3 = 12 行が rail に並ぶと Newspaper Style の rail (compact 鉛筆書き) が崩れる。direct citation の seed 集合は通常 ≤ 4 で重複が大きいため、per-message k=3 にまとめた方が duplicate retrieve が起きず精度も安定する。

#### (C) `RelatedArticlesTool` (既存 `adapter/tools/related_articles.go`) を structured 返却に拡張

却下。`RelatedArticlesTool` は Markdown 文字列を返す agent tool-use 経路で、`SearchClient` (Meilisearch ベース) を使う。HybridSearch (pgvector + tsvector) とは別系統で、neighbor citation を tool 経路に乗せると tool_only / no_retrieval ポリシーとの相性が悪い。HybridSearcher を direct call する方が検索一貫性も latency 予算も予測可能。

#### (D) 後追いで neighbor を別テーブルに書く / 読み出し時に毎回 retrieve する (disposable projection)

却下。後追い書き込みは `augur_messages` のライフサイクルと分裂してジョイン整合性問題を生む。読み出し時 retrieve は ADR-926 の「Citations は INSERT 時の business fact」原則と矛盾し、回答時点のスナップショットを保てない。**inline projection (done event の 1 INSERT で direct + related を同居書き込み)** が append-only / reproject-safe / business-fact-purity の 3 不変条件を同時に満たす唯一の選択肢だった。

#### (E) tag overlap (knowledge-sovereign の `summary_versions` / `tag_set_versions`) も neighbor source に加える

第一弾では却下。tag overlap は価値があるが knowledge-sovereign への RPC が追加され reproject-safe 設計と整合させる工数が大きい。Phase 1 は HybridSearch のみで MVP として十分。第二弾以降で評価 (Recall / Precision メトリクス) を経てから追加検討。

## Consequences

### Pros

- **「Ask Augur で元記事が参照されない」現象が構造的に消える**: RAG retrieve 経路で `kind=ARTICLE` + `ref_id=<UUID>` が常に乗るので、`citation-href.ts` の switch が `/articles/<refId>` を返し、FE rail はクリック可能なリンクを描画する。disabled span fallback は legacy 行 (kind 空) のみに限定される
- **UUID の UI 露出が二重防御で消える**: backend `sanitizeCitationTitle` で wire 前に UUID-only Title を空文字化、FE `safeTitle` で decode 後にも同じパターンを再検査、FE rail の `displayTitle` で Title → formatDomain(URL) → "Untitled source" の 3 段フォールバック。3 層のどれかが破られても次が catch する
- **近傍記事が同 INSERT で append-only 不変を守る**: done event 受信時に direct + related を 1 度だけ確定し JSONB 2 列に同居書き込み。後追い backfill / UPDATE が一切無いので [[000836]] / [[IMPL_BASE]] の append-first 原則を完全に保つ
- **proto は additive のみ**: `buf breaking` clean、rolling deploy 中も古い consumer は新 field を無視するだけで壊れない。alt-backend / alt-butterfly-facade は passive consumer (再生成 stub の受け入れのみ)
- **observability が改善する**: 今まで silent failure だった「kind 忘れ」が `convertCitationsToProtoCitations` を経由する citation 全件に対して観測可能になる (`otel_logs` で `kind=UNSPECIFIED` の発生数が citation 件数の補集合として可視化される)。退行を post-deploy で数値で追える
- **HybridSearcher 既存資産の再利用**: 新規 retrieval パスを作らず既存 RRF パイプラインの薄い派生で済んだので、検索品質とレイテンシは実証済み水準を継承する

### Cons / Tradeoffs

- **RAG 経路に neighbor lookup 1 回ぶんのレイテンシが追加**: HybridSearch + post-filter で +50-200ms 程度を見込む。`total_duration_ms` の p95 を post-deploy で観測する必要があり、閾値超過時は (a) async (post-stream / 別 endpoint) 化、または (b) neighborLimit を下げる、で緩和する。`buildRelatedCitations` 内で goroutine 化していないため Stream のレイテンシ予算に直接乗る点は次フェーズの対象
- **`ContextItem` が 2 定義に分裂したまま残る**: `retrieve_context_usecase.go:33` と `retrieval/types.go:44` の重複は本 ADR では両方に `ArticleID` を追加して同期させただけで、定義そのものの統合は別 ADR (技術負債) として保留した。将来 ContextItem に field が増えるたびに同期忘れリスクが残る
- **`UNSPECIFIED` fallback が温存されているため新規 emitter の kind 忘れを検知できない**: 既存テストで「ArticleID が UUID なら Kind=ARTICLE が必ず立つ」contract test を追加し regression を 1 本止めたが、別の新規経路が UNSPECIFIED を default にして emit する経路を作っても server 側では検知できない構造的負債は残る (ADR-926 Cons §3 と同じ指摘)。citation 種別の追加が起きるたびに lint / contract test を追加する規律が前提
- **過去 `augur_messages` 行は backfill しない**: 既存の壊れた citation 行 (`kind=UNSPECIFIED` で書かれたもの) は FE の disabled span fallback で穏便に表示される。歴史的 chat に戻ったユーザが clickable な citation を期待した場合の UX 後退は受容する (ADR-926 と同じ判断を継承)
- **knowledge-sovereign の tag overlap は未統合**: 第一弾は HybridSearch のみで MVP として確定。tag-aware neighbor は次フェーズで eval を経てから増分追加する想定で、本 ADR の範囲外
- **デプロイ未実施**: 本 ADR 起票時点で local commit のみ。push / build / production reproject はユーザ明示指示まで待機する (CLAUDE.md 「Plan 承認 ≠ build / push / deploy 許可」)。`augur` / `acolyte` / `alt-backend` コンテナはローカル停止中で、production の citation がいつから UNSPECIFIED で書かれているかは binary deploy 時刻に依存する

## Deploy-model 整合性セルフチェック

本 ADR は compose / CI / deploy パターンを触らない (proto / Go コード / Atlas migration / FE のみ)。`docker-compose.yaml`, `depends_on`, named volume, one-shot init container いずれにも影響しない。

- [x] 新設 compose service が `depends_on` で他 service を gate するか？ → **N/A** (compose 変更なし)
- [x] 新設 compose service に `restart: "no"` の one-shot init container を含むか？ → **N/A** (init container を追加しない)
- [x] 新設名前付き volume を populate する責務はどこにあるか？ → **N/A** (新規 volume なし)
- [x] いずれも該当しない → **rolling 互換**。`rag-orchestrator` + `alt-frontend-sv` の rebuild + restart のみで新 binary に切り替わる。Atlas migration (`20260527100000_add_related_citations.sql`) は `ALTER TABLE ADD COLUMN ... DEFAULT '[]'::jsonb` で旧バイナリが SELECT しても新カラムを無視できるため、`migrate → deploy` の順を守れば rolling 非互換は発生しない。compose 構造には変更が入らないので [[000826]] / [[PM-2026-037]] の rolling 非互換問題には該当しない
