# ADR-000483: Stream Processing パスで ArticleCreated イベントの feed_id をフォールバック補完する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- tag-generator (Python)

## Context

tag-generator の Stream Processing パス（`ArticleCreated` Redis Streams イベント経由）で `Article missing feed_id` 警告が毎サイクル約200件発生していた。

原因は以下のデータフロー上のギャップにある:

1. alt-backend の `PublishArticleCreated` は `ArticleCreatedPayload` に `feed_id` を含めてイベントを発行する
2. `stream_event_handler.py` の `_handle_article_created` はイベントペイロードから `article_id` のみを取り出す
3. `fetch_article_by_id()` は `GetArticleContent` RPC を使って記事を再取得するが、この RPC レスポンスには `feed_id` が含まれない
4. 結果として `connect_article_fetcher.py` は `feed_id: None` をハードコードで返す

イベントペイロードには `feed_id` が既に存在するにもかかわらず、使われていなかった。

## Decision

`_handle_article_created` → `_process_article` → `_process_article_sync` のコールチェーンでイベントペイロードの `feed_id` を伝搬し、`fetch_article_by_id()` が返す article dict に `feed_id` が欠落している場合のみフォールバック補完する方針を採用する。

### 代替案: `GetArticleContent` RPC にfeed_id を追加する

alt-backend 側の RPC レスポンスに `feed_id` フィールドを追加する方法も検討したが、以下の理由で見送った:

- alt-backend と tag-generator 双方の proto 定義変更が必要で影響範囲が広い
- イベントペイロードに既に含まれている情報を二重に取得する非効率性
- Stream Processing パスのみの問題であり、局所的な修正で十分対処可能

### 実装の概要

`stream_event_handler.py` に対する3箇所の変更:

1. **`_handle_article_created`**: ペイロードから `feed_id` を抽出し `_process_article` に渡す
2. **`_process_article`**: `feed_id` キーワード引数を受け取り executor 経由で `_process_article_sync` に伝搬
3. **`_process_article_sync`**: `fetch_article_by_id` の結果に `feed_id` が欠落している場合のみ補完（既存値は上書きしない）

## Consequences

### Pros

- 毎サイクル約200件の `Article missing feed_id` 警告が解消され、ログノイズが大幅に削減される
- feed_id に依存する下流処理（タグ生成のフィード固有ロジック等）が Stream Processing パスでも正しく動作する
- alt-backend 側の変更が不要で、tag-generator 内で完結する局所的な修正

### Cons / Tradeoffs

- イベントペイロードの `feed_id` に依存するため、発行元が `feed_id` を含めなくなった場合は従来の警告が再発する（ただし alt-backend 側のペイロード仕様は安定しており、リスクは低い）
- `GetArticleContent` RPC 自体の `feed_id` 欠落は根本解決されていない（バッチ処理パス等で同じ問題が顕在化する可能性はある）
