# ADR-000936: visual-preview の OG 画像 backfill を件数ベースから内容ベース冪等方式へ是正し read 経路の OG 解決欠落を塞ぐ

## Status

Accepted

## Date

2026-06-02

## Affected Services

- alt-frontend-sv (TypeScript/Svelte) — `/feeds/visual-preview` desktop の OG 画像 prefetch を内容ベース冪等 backfill に置換
- alt-backend (Go) — `FetchReadFeedsListCursor` の driver query と read gateway に OG 画像 / article_id 解決を追加

## Context

`/feeds/visual-preview` の desktop UI で Unread フィードを閲覧中、フィードを既読にすると OG 画像（サムネイル）が
placeholder（fallback グラデーション）に置き換わる現象が頻発していた。再現は確定的でなく「がち（intermittent）」だった。

フロント / バックエンド / 実ログ・DB の 3 軸で並列調査した結果、原因は backend の SQL ではなくフロントの backfill 制御にあると判明した。

- **DB / backend SQL は正しい**: `ogImageSelectExpr` は 7 日ウィンドウ（`f.created_at >= NOW() - INTERVAL '7 days'`）の
  下で `COALESCE(article_heads.og_image_url, feeds.og_image_url)` を解決しており、read/unread に依存しない（[[000935]] の不変条件どおり）。
  実 DB でも read/unread を問わず 7 日内なら OG が解決されることを確認した。
- **新規 unread フィードは backend 時点で `og_image_url` が空**になりがち（article_heads 未スクレイプ、`feeds.og_image_url` も NULL）。
  この場合 backend は `og_image_proxy_url=""` を返し、フロントが `batchPrefetchImagesClient(articleIds)` で
  **クライアント側 backfill** して初めて画像が表示される設計になっている。
- **直接原因**: backfill を駆動する `$effect`（visual-preview `+page.svelte`）のガードが **件数差分**
  `if (visibleFeeds.length === prefetchedCount) return;` だった。既読操作は
  「`removeFeedByUrl`（−1）→ `fetchReplacementFeed`（+1, 非同期 fire-and-forget）」で visible 件数が往復するため、
  削除と replacement append が同一 effect flush に重なる / 件数が偶然一致するタイミングで backfill がスキップされ、
  replacement カードが恒久的に placeholder で固定される。これが intermittent な再現性の正体である。
- **関連欠落（read 経路）**: `FetchReadFeedsListCursor` だけが driver query で `ogImageSelectExpr` / `article_id` を
  selectしておらず、read gateway も `OgImageURL` / `ArticleID` を domain に map していなかった。Read タブは常に placeholder になる。
  本症状（Unread 表示中の既読化）の直接原因ではないが、同じ「read → placeholder」クラスの欠落である。

## Decision

**フロント backfill を「visible 件数の差分」ではなく「未取得 articleId の内容」で駆動する冪等方式に置換する。**
併せて **backend の read 経路（driver query + gateway）に OG 画像 / article_id 解決を追加**し、unread / all / favorite と挙動を揃える。

- フロント: 純粋関数 `selectOgImagePrefetchIds(visibleFeeds, requested)`（`src/lib/utils/ogImagePrefetch.ts`）を新設し、
  「`articleId` を持ち `ogImageProxyUrl` が空で、まだ要求していない」フィードの articleId のみを抽出する。
  `+page.svelte` の `$effect` は件数スカラー `prefetchedCount` を廃し、非リアクティブな `Set<string>`
  （`requestedOgImageArticleIds`）で in-flight を記録する。要求前に Set へ登録（二重要求と件数往復スキップを防止）、
  失敗時は Set から削除して再試行を許可する。
- backend: `FetchReadFeedsListCursor` の driver query に `article_id` サブクエリと `ogImageSelectExpr` を注入し 9 列 scan に揃え、
  read gateway で `OgImageURL` / `ArticleID` を `domain.FeedItem` に map する。`enrichWithProxyURLs` は read 経路でも
  既に呼ばれているため、`OgImageURL` が入れば proxy URL も生成される。

### 検討した代替案

- **件数ガードの比較対象を増やす（前回 visibleFeeds の id 集合と差分を取る）**: 状態が増え reproject 不要なフロントでも
  複雑化する。冪等 Set 方式の方が単純で取りこぼしが原理的に起きない。却下。
- **backend が新規 unread フィードに対し OG 取得を同期で待ってから返す**: レイテンシ悪化と外部スクレイプの 5 秒レート制限に抵触。
  既存の「空で返してクライアント backfill」設計を維持する方が整合的。却下。
- **read 経路の OG 欠落をフロントの backfill だけで吸収する**: backend が canonical に OG を返せる以上、source を直す方が正しい。
  フロント backfill は「backend が空のときの補完」に留めるべき。却下。

## Consequences

### Pros

- 既読操作後も replacement フィードの OG 画像が確実に backfill され、placeholder への化けが解消する。
- backfill が visible 件数の増減から完全に切り離され、remove + append レースでも取りこぼさない。infinite-scroll の append にも同じく作用する。
- `Set` による in-flight 記録で `batchPrefetchImages` の二重呼び出しが消える。
- Read タブ / read 経路でも OG 画像が backend から canonical に解決され、unread / all / favorite と挙動が一致する。

### Cons / Tradeoffs

- `requestedOgImageArticleIds` はページ生存期間中単調増加する（フィルタ変更でクリアしない）。1 セッションの articleId 数では
  メモリ上問題にならないが、極端な長時間閲覧では緩やかに増える。失敗時のみ delete で巻き戻す設計。
- read query が article_id サブクエリ + ogImageSelectExpr 分だけ重くなる（unread / all と同等のコスト）。7 日ゲートと
  `LIMIT` があるため実害は小さい。
- 画像を本当に持たないフィードが placeholder になるのは仕様どおり。本修正は「持っているのに出ない」ケースのみを直す。
