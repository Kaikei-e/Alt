# ADR-000670: hydrator/extractor 間に extractive CompressorNode を追加し context packing を最適化する

## Status

Accepted

## Date

2026-04-10

## Affected Services

- acolyte-orchestrator (Python 3.14+)

## Context

exec1.md Issue 8 の実装。Issues 1-7（[[000659]] SectionPlannerNode、[[000661]] section_citations、[[000662]] Conclusion 分離、[[000664]] ES 再合成、[[000666]] section contract Planner、[[000667]] query decomposition、[[000669]] multi-query RRF fusion）により composition 層と retrieval 構造化の基盤は整ったが、**hydrator → extractor 間のコンテキスト品質**に以下の課題が残っていた。

1. **硬直的な切り落とし** — `ExtractorNode` が `body[:2000]` で記事本文を一律切断。長い記事の後半に重要情報がある場合に完全に失われる
2. **ノイズの混入** — 短い切り出しでも無関係な文が含まれ、Gemma4 の structured output 生成が不安定になる（`num_predict` と timeout を盛って押し切る方向は resolve で明確に否定）
3. **section-aware でない** — 記事がどのセクションに curated されたかを考慮せず、一律に同じ 2000 文字を渡していた

### 外部知見

- **RECOMP (ICLR 2024)**: retrievd documents の圧縮は extractive（文選択）と abstractive（要約生成）に分かれ、extractive で最大 10 倍圧縮・精度劣化最小。ただし論文の実装は訓練済み compressor であり、単純 keyword overlap は first-cut ヒューリスティックと位置づけるべき
- **Lost in the Middle (Liu et al. 2023) / LongLLMLingua (Jiang et al. 2023)**: 重要情報はプロンプトの先頭・末尾が有利。圧縮後の packing 順は「文書順に戻す」だけでは不十分で、score-descending（強い evidence を先頭）にする必要がある
- **Ollama #15260 (OPEN, 2026-04-10 時点)**: Gemma4 で `think=false` + `format` を併用すると structured output 制約が無視される。新しい LLM 呼び出しの追加は避けるべき
- **resolve doc**: 「Extractor を quote selection → fact normalization の 2 段階に分解」「長い 1 発 JSON をやめること」

### 設計判断の分岐: LLM-based compression vs heuristic extractive

| 案 | メリット | デメリット |
|---|---|---|
| A. LLM-based compression（per-article） | 意味的理解で精度が高い | Ollama #15260 のリスク、per-article LLM 呼び出しでレイテンシ倍増、カスケード障害リスク |
| B. Heuristic extractive compression | LLM 不使用で信頼性・速度が安定、依存ゼロ（Python 標準ライブラリのみ） | 語彙的類似のみ。意味的類似は Stage 2（ExtractorNode）の LLM が補完 |

**案 B を採用**。resolve の「Gemma4 を直そうとしすぎないこと」に準じ、infra リスクを最小化。2 段階抽出の Stage 1（span 選択）を heuristic、Stage 2（fact 正規化）を LLM とする分離が最も安定。

## Decision

**hydrator と extractor の間に `CompressorNode` を追加し、section-aware の heuristic extractive compression を行う。**

### 1. Domain Model: `acolyte/domain/compressed_evidence.py`（新規）

純粋関数のみ。LLM 不使用。

- **`CompressedSpan`**: frozen dataclass（`text`, `char_offset`, `relevance_score`）
- **`split_sentences(text)`**: JP/EN 混在対応の conservative な文分割。`。！？` で分割（日本語）、`.!?` + 空白 + 大文字で分割（英語）。略語（`U.S.`）や小数（`3.14%`）を誤分割しない
- **`score_sentence(sentence, query_terms)`**: keyword overlap scoring。case-insensitive、CJK は substring match。hits / query_terms_count で正規化（0.0-1.0）
- **`compress_article(body, queries, *, char_budget=1000)`**:
  1. 文分割 → 各文をスコアリング
  2. Selective augmentation: 全スコアが閾値未満なら `[]` を返す（不要なコンテキストを渡さない）
  3. Score descending で上位文を `char_budget` まで選択
  4. **Packing 順は score-descending を維持**（Lost-in-the-Middle 対策: 最強 evidence を先頭に配置）

`char_budget=1000` は resolve の「800〜1200 文字程度のチャンクに切る」の中央値。

### 2. CompressorNode: `acolyte/usecase/graph/nodes/compressor_node.py`（新規）

- **Query context 構築**: `_collect_queries_for_articles()` は `outline[*].query_facets` を **優先的に参照**（ADR-667/669 の faceted retrieval と整合）。`query_facets` がない場合のみ `search_queries` にフォールバック。topic は常に追加
- **Per-article 圧縮**: hydrated body → `compress_article()` → CompressedSpan dicts
- **ログ**: 圧縮率（original_chars / compressed_chars）を structlog で出力

### 3. State 拡張: `acolyte/usecase/graph/state.py`

`compressed_evidence: dict[str, list[dict]]` を追加。article_id → CompressedSpan dicts のマップ。

### 4. ExtractorNode 修正: `acolyte/usecase/graph/nodes/extractor_node.py`

Body 解決ロジックを tiered fallback で変更:
1. **Tier 1**: `compressed_evidence[article_id]` が存在し **非空** → compressed spans を join して使用
2. **Tier 2**: `compressed_evidence[article_id]` が **空リスト** → `hydrated_evidence[:2000]` にフォールバック + warning ログ。EXIT framework (ICLR 2025) の「閾値未満時は原文 fallback」パターンに準拠
3. **Tier 3**: `compressed_evidence` に article_id がない → compressor 未実行として `hydrated_evidence[:2000]` にフォールバック（backward compat）

### 5. Graph 配線: `acolyte/usecase/graph/report_graph.py`

```
planner → gatherer → curator → hydrator → compressor → extractor → section_planner → writer → critic → finalizer
                                             ^^^^^^^^
                                             NEW NODE
```

`content_store` なしのパイプラインには compressor を追加しない（backward compat）。

### テスト

TDD（RED → GREEN → REFACTOR）で実装。31 件のテストを新規作成:

- `test_compressor_node.py`: CompressedSpan frozen、文分割（EN/JP/mixed/略語/小数/空/単一文/改行/箇条書き）、スコアリング（overlap/no-match/case-insensitive/多語/日本語コンテンツ/CJK bi-gram）、圧縮（選択/score-descending/passthrough/selective-augmentation-empty/empty-body/budget/日本語記事統合）、クエリ抽出（日本語句読点分割/混合言語）、CompressorNode（基本/query_facets優先/multi-section/empty）、ExtractorNode（compressed優先/fallback/empty→hydrated fallback）
- `test_pipeline.py`: 統合テスト 1 件追加（長い記事が compression されることを検証）

全 243 unit tests + 4 contract tests が pass。

### 品質改善 Hotfix（同日）

初版実装で日本語レポート（`743372a2` 「AIチップの2026年動向」）が全バージョン body_len=0 になる問題を発見・修正。根本原因 5 件を特定し修正:

1. **`_extract_query_terms()` CJK regex**: `\u3000-\u9fff` が句読点を含み日本語文全体が1トークンに → CJK 文字のみ（ひらがな/カタカナ/漢字）を抽出、句読点を区切りとして機能させるよう修正
2. **`score_sentence()` CJK 対応**: ASCII word match のみ → dual scoring（ASCII word + CJK character bi-gram Jaccard）に拡張。pg_trgm と同様の character n-gram approach
3. **`split_sentences()` 改行対応**: `\n` 単独の改行（RSS 箇条書き・見出し）を区切りとして追加
4. **ExtractorNode tiered fallback**: `compressed==[]` で記事スキップ → hydrated body にフォールバック（EXIT framework パターン）
5. **CompressorNode 監視ログ**: 全記事で compressed=[] の場合に warning を出力

## Consequences

### Pros

- **情報保全**: `body[:2000]` の一律切り落としがなくなり、記事後半の重要情報もスコアに基づいて選択される
- **ノイズ削減**: 無関係な文が除外され、ExtractorNode の structured output 生成が安定化
- **Lost-in-the-Middle 対策**: score-descending packing により最重要 evidence がプロンプト先頭に配置
- **Selective augmentation**: 無関係な記事をスキップし、LLM トークンの無駄遣いを回避
- **LLM 呼び出し増なし**: heuristic のみで動作。Gemma4 + structured output の不安定さを踏まない
- **ADR-667/669 との整合**: `query_facets` を優先参照し、faceted retrieval の意図を compression に反映

### Cons / Tradeoffs

- **語彙的類似のみ**: keyword overlap は意味的類似をカバーしない。「semiconductor」と「chip」は query 側に両方含まれていないとマッチしない。Stage 2（ExtractorNode の LLM）が補完する前提
- **文分割の限界**: `re` ベースの conservative 分割は、複雑な引用構造や箇条書きで不正確になりうる。UAX #29 準拠の segmenter 導入は将来課題
- **圧縮パラメータの固定**: `char_budget=1000`, `_RELEVANCE_THRESHOLD=0.01` はハードコード。運用データに基づくチューニングは未実施
