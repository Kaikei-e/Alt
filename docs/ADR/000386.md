# ADR-000386: Ask Augurパイプラインの3層ボトルネック解消とrerank-serverデプロイ自動化

## Status

Accepted

## Date

2026-03-14

## Affected Services

- rag-orchestrator (Go) — Query Expansionタイムアウト延長、Rerankerタイムアウト調整
- news-creator (Python/FastAPI) — Query Expansion APIにpriority伝搬を追加
- rerank-server (Python/FastAPI) — M-series MacへのlaunchdデプロイとMakefile統合

## Context

Ask Augur（RAGチャット機能）のレスポンスが85秒〜3.5分と極端に遅く、72時間で観測された3リクエスト中2件がユーザーの途中放棄に至っていた。ADR-036で最適化された検索パイプライン（embedding 160ms、ベクトル検索 31ms、BM25 3ms）自体は正常に動作しており、**検索前後の3つの設定レベルのボトルネック**が複合的に遅延を生んでいた。

### ボトルネック1: Query Expansion リソース競合（全体の60-70%）

`expand_queries.go`のレースパターンでは、news-creator（gemma3-4b-8k）とollama-legacy（gemma3-12b-rag）を並行して起動し、先に完了した方を採用する。しかし:

- news-creatorのOllamaバックエンド（`OLLAMA_NUM_PARALLEL=2`）が要約処理で常時占有されている
- Query expansionは`priority="low"`（デフォルト）でキューに入り、要約リクエストの後ろに並ぶ
- 実測でキュー待ち8.7秒が発生し、3秒のタイムアウト（`QUERY_EXPANSION_TIMEOUT=3`）では到底間に合わない
- news-creatorが常にタイムアウトし、12Bモデルのollama-legacyにフォールバック
- ollama-legacyはAsk Augurの低利用頻度（72時間で3リクエスト）によりモデルがほぼ常にコールド → 60-70秒

### ボトルネック2: 死んだRerankerサービス（毎回+15秒）

`RERANK_ENABLED=true`（デフォルト）の状態で、rerank-serverが稼働していないホストに対して毎回15秒のHTTPタイムアウトが発生していた。3リクエスト全てで100%の失敗率。Rerankingの品質改善効果はゼロで、15秒は純粋な浪費だった。

### ボトルネック3: LLMコールドスタート（12-22秒 TTFT）

gemma3-12b-ragのモデルロード時間に起因。低利用頻度のためOllamaがVRAM圧により自動アンロード。本ADRの直接の対象外だが、ボトルネック1の解消で全体のレイテンシが改善されるため、相対的に影響が軽減される。

### 観測データ（修正前）

| ステージ | Request 1 (cold) | Request 2 (warm) | Request 3 (cold) |
|---|---|---|---|
| Query Expansion | 71,224ms | 17,484ms | 69,600ms |
| 検索 | 158ms | 1,565ms | 161ms |
| Reranking (常に失敗) | 15,000ms | 15,000ms | 15,000ms |
| **Retrieval合計** | **86,402ms** | **34,056ms** | **84,795ms** |

## Decision

以下の3つの修正を採用する。

### 1. Query Expansionタイムアウト延長 + 高優先度化（Option A+B併用）

**Option A**: `QUERY_EXPANSION_TIMEOUT`を3秒→15秒に延長する。news-creatorのキュー待ち（実測~9秒）+ 処理時間（~1秒）に十分対応可能。

**Option B**: Query Expansion APIリクエストに`priority="high"`を追加し、`HybridPrioritySemaphore`のRT予約スロットを使用する。要約処理のキューをスキップし、news-creatorが確実にレースに勝てるようにする。

検討した代替案:
- **Query Expansion専用Ollamaインスタンスの追加**: リソース競合を完全に排除できるが、追加のGPUメモリとプロセス管理が必要。現時点ではpriority昇格で十分なため見送り
- **タイムアウトのみの延長（Option Aのみ）**: キュー混雑時にはタイムアウト延長だけでは不十分な場合がある。A+Bの併用で確実性を高める

### 2. rerank-serverの外部デプロイと安定運用

rerank-serverを無効化するのではなく、M-series Mac上で安定稼働させる。Cross-Encoder Rerankingの品質改善効果（NDCG@10 +15-30%、LLMハルシネーション -35%）を維持するため。

検討した代替案:
- **`RERANK_ENABLED=false`で恒久無効化**: Hybrid Search + Rerankerの品質上の利点が失われる。暫定措置としてのみ使用する
- **news-creator内でRerankを実行**: query expansion / summarizationとのリソース競合が発生する。専用サービスとして分離した方が安定する

### 3. socat不要の直接接続アーキテクチャ

当初はrerank-serverの`0.0.0.0:8080`リッスンに加えてsocatでTailscale IP:8080→localhost:8080のフォワーディングを設定していたが、**ポート競合**が発生した。`0.0.0.0`は全インターフェース（Tailscale含む）でリッスンするため、socatとの二重バインドによりDockerコンテナからの接続が不安定になった。

socatを廃止し、rerank-serverの`0.0.0.0:8080`バインドへの直接接続を採用する。

## Implementation

### rag-orchestrator

**`compose/rag.yaml`**:
- `QUERY_EXPANSION_TIMEOUT`: 3 → 15
- `RERANK_ENABLED`: true（rerank-server稼働確認後）
- `RERANK_TIMEOUT`: 15 → 12（MPS上のCross-Encoder推論は15 candidates で約7秒）

**`internal/adapter/rag_augur/query_expander_client.go`**:
- `ExpandQueryRequest`構造体に`Priority`フィールド（`json:"priority"`）を追加
- `ExpandQuery()`と`ExpandQueryWithHistory()`の両方で`Priority: "high"`を設定

### news-creator

**`domain/models.py`**:
- `ExpandQueryRequest`に`priority`フィールドを追加（デフォルト`"high"`、`^(high|low)$`バリデーション）
- デフォルトを`"high"`にした理由: Query Expansionはリアルタイムのユーザーリクエストに紐づくため、バッチ的な要約処理より優先されるべき

**`handler/expand_query_handler.py`**:
- `request.priority`をusecaseに伝搬

**`usecase/expand_query_usecase.py`**:
- `expand_query()`メソッドに`priority`パラメータを追加
- `llm_provider.generate(priority=priority)`でセマフォ取得時の優先度を制御

### rerank-server デプロイ自動化

**`rerank-server/deploy.sh`** (新規):
- `.env`から`RERANK_EXTERNAL_HOST`と`RERANK_SSH_USER`を読み取る（ハードコードなし、OSS対応）
- 3ステップ: rsync → venv + deps → launchd登録
- staleプロセスとsocat競合の自動クリーンアップ
- 起動後にヘルスチェックで疎通確認（初回デプロイ時はモデルダウンロード待ち最大120秒）

**`Makefile`**:
- `make rerank-deploy`: deploy.sh実行
- `make rerank-status`: ヘルスチェック + compose設定確認
- `make rerank-enable` / `make rerank-disable`: RERANK_ENABLED切り替え + rag-orchestrator再ビルド

**`.env.template`**:
- `RERANK_SSH_USER`を追加

### launchd常駐化

- `com.alt.rerank-server`: KeepAlive（クラッシュ時自動再起動）、RunAtLoad（Mac起動時自動開始）
- ログ: `~/.rerank-server/logs/server.{stdout,stderr}.log`
- `PYTORCH_MPS_HIGH_WATERMARK_RATIO=0.0`でMPSメモリキャッシュを制限

### テスト

- news-creator usecase: `test_expand_query_passes_priority_to_llm`、`test_expand_query_default_priority_is_low`（2テスト追加）
- news-creator handler: `test_expand_query_handler_with_priority`、`test_expand_query_handler_default_priority`（2テスト追加）
- 既存テスト2件を`priority`パラメータ追加に対応して更新
- rag-orchestrator: 全テストパス、ビルド成功

## Consequences

### Pros

- **Retrieval時間 88%短縮**: 85-107秒 → 11.7秒（実測）
- **最初のトークンまでの時間 78%短縮**: 97-129秒 → 約27秒（実測）
- **Reranking品質の回復**: Cross-Encoder（BAAI/bge-reranker-v2-m3）による再ランキングが機能し、NDCG@10の改善とハルシネーション削減が期待できる
- **news-creatorがレースに勝利**: 4Bモデル（4.5秒）が12Bモデル（6-70秒）に確実に勝ち、ユーザー体感レイテンシが安定
- **ワンコマンドデプロイ**: `make rerank-deploy`でM-series Macへのデプロイが完了。launchdでMac再起動時も自動復旧
- **OSS互換**: `.env`からIP/ユーザー名を読み取り、ハードコードなし

### Cons / Tradeoffs

- **外部ホスト依存**: rerank-serverはTailscale経由のM-series Macに依存。Mac停止時はRerankingが12秒タイムアウト後にフォールバック（ベクトルスコアのみ）
- **Tailscale IP変更時の追従**: `RERANK_EXTERNAL_HOST`の手動更新が必要。MagicDNSはDocker extra_hostsでは直接使えない
- **LLM TTFT未改善**: ボトルネック3（12-22秒のコールドスタート）は本ADRでは未対処。利用頻度が増えればモデルのウォーム状態が維持されるため、自然に改善される見込み
- **Reranking処理時間の追加**: 成功時に約7秒のオーバーヘッド。ただしこれは品質改善のための意図的なトレードオフであり、15秒の無駄なタイムアウトと比べて正味+7秒で品質が向上
