# ADR-000769: news-creator Hurl E2E Phase 2 で summarize / generate / recap happy path をカバーする

## Status

Accepted

## Date

2026-04-18

## Affected Services

- news-creator (Python 3.14 / FastAPI) — ランタイムは無変更。Hurl E2E スイートに 4 シナリオ (`/api/v1/summarize` 非ストリーミング、`/api/generate`、`/v1/summary/generate`、`/v1/summary/generate/batch`) を追加
- news-creator-ollama-stub (Python+FastAPI) — `/api/generate` と `/api/chat` (非ストリーミング) を実装。`format` フィールドの有無で構造化 JSON / プレーンテキストを切り替える

## Context

[[000768]] で news-creator Hurl E2E の Phase 1 (基盤 + `/health` + `/queue/status`) を投入した。Phase 2 は本サービスの主要 happy path を取り込む段で、対象は pre-processor / acolyte-orchestrator が叩く `/api/v1/summarize` 非ストリーミング、Ollama pass-through の `/api/generate`、recap-worker が叩く `/v1/summary/generate` 単発と `/v1/summary/generate/batch` の 4 系統である。Pact CDC で形状は pin されているが、handler → usecase → gateway → driver → Ollama スタブ までを HTTP 一発で通せるかは contract 検証の外側にある。

stub Ollama を Phase 1 では `/api/tags` だけにとどめていた。Phase 2 を回すには:

- `/api/v1/summarize` と `/api/generate` のため、Ollama 互換の `/api/generate` レスポンス (`{model, response, done, done_reason, prompt_eval_count, eval_count, total_duration, ...}`) を返す必要がある。
- recap-worker の `_call_llm_for_recap` は `chat_generate` を優先するため、`/api/chat` (`stream=false`) も必要。`format` JSON schema (Ollama Structured Outputs) を渡されたときは `message.content` が `RecapSummary` (`{title, bullets[], language: "ja"}`) としてパース可能な JSON 文字列でなければならない。
- summarize_usecase の `len(stripped) < 10` ガードと repetition 検出を踏まないよう、レスポンス本文は 10 文字以上で語彙が分散している必要がある。

これら制約を一つの stub にまとめる設計判断を本 ADR で固定する。

## Decision

### stub Ollama を `/api/generate` + `/api/chat` (非ストリーミング) で拡張する

`compose/news-creator-ollama-stub/app.py` を v0.2.0 に更新し、リクエスト本文に `format` フィールドが含まれているかどうかを唯一の分岐条件として、応答コンテンツを以下のように切り替える:

- `format` あり (recap chat path / `generate()` フォールバック) → `STRUCTURED_PAYLOAD` を `json.dumps(..., ensure_ascii=False)` した文字列を返す。`{"title", "bullets" (3 件), "language": "ja"}` で `RecapSummary` 検証 + `_parse_summary_json` を通る。
- `format` なし (一般の summarize / generate) → `PLAIN_TEXT_RESPONSE` (3 文の英語固定文) を返す。≥ 10 文字 + 異なる語彙で repetition 検出を回避。

両エンドポイントとも `model` リクエスト値があればそれをそのまま返し、無ければ `gemma3:4b-it-qat` をデフォルトとする。`prompt_eval_count` / `eval_count` / `total_duration` 等の補助フィールドはハードコード値で埋める (本 Phase の検査対象外)。

### Hurl シナリオ 4 本を追加する

`--jobs 1` の直列実行を継続。各シナリオはフィクスチャファイル (`file,e2e/fixtures/news-creator/<name>.json;`) を `--file-root $ROOT` 経由で読み込む。

| ファイル | リクエスト | 主要アサーション |
|---|---|---|
| `03-summarize-non-stream.hurl` | POST `/api/v1/summarize` | `{success, article_id, summary, model}` の形状 + `summary matches ".+"` |
| `04-generate.hurl` | POST `/api/generate` | Ollama pass-through (`{model, response, done, done_reason, prompt_eval_count, eval_count, total_duration}`) |
| `05-recap-summary.hurl` | POST `/v1/summary/generate` | `{job_id, genre, summary: {title, bullets[], language}, metadata: {model, summary_length_bullets, is_degraded}}` |
| `06-recap-batch.hurl` | POST `/v1/summary/generate/batch` | `responses[]` count == 2、`errors[]` count == 0、各レスポンスの `job_id` / `genre` 整合 |

フィクスチャ:

- `article-normal.json` — 100 字超の日本語サンプル (`min_content_length=100` ガード回避)
- `generate-prompt.json` — `/api/generate` 用の短いプロンプト
- `recap-clusters.json` — UUID `00000000-0000-4000-8000-000000000101`、`representative_sentences` 2 件
- `recap-batch.json` — 2 リクエスト bundle (genre = `ai` + `security`)

### 検討した代替案

- **stub レスポンスを常に JSON 文字列にする** — 却下。summarize 応答の `summary` フィールドが JSON 文字列になり、本番との形状乖離が大きい。`format` 分岐の方が「リクエスト意図に応じたコンテンツ」を返せる。
- **stub に handler 単位の分岐ロジックを持たせる (URL ベース)** — 却下。recap-worker は `chat_generate` フォールバック時に `/api/generate` を叩くため、URL では正しい挙動を選べない。`format` フィールドが Structured Outputs 採用の唯一の信号。
- **stub をやめて real Ollama に切り替える** — 却下。Phase 1 の判断 ([[000768]]) と整合する。
- **Phase 2 で streaming も含める** — 却下。Hurl 7.1 の body 完全受信制約と SSE 改行整形の再現コスト分けて Phase 4 で扱う。

## Consequences

### Pros

- pre-processor (`/api/v1/summarize`)、acolyte-orchestrator (`/api/v1/summarize`)、recap-worker (`/v1/summary/generate(/batch)`)、Ollama pass-through (`/api/generate`) の 4 主要経路の HTTP shape を黒箱で gate できるようになった。Pact が pin する個別 contract に加えて handler → usecase → gateway → driver の合成挙動を CI で守れる。
- stub の分岐ロジックは `format` フィールドの存在チェック 1 行のみ。recap が将来 `format` を外す変更をしても test がそのまま通るほどの素朴さで、メンテナンス負債を残さない。
- Hurl の実行時間は 6 シナリオ合計 55 ms (ローカル計測) と Phase 1 比 +51 ms に抑えられた。コンテナ再ビルドは含まない (compose 起動時 `--build` で stub 差分のみ rebuild)。

### Cons / Tradeoffs

- stub のレスポンスは固定文字列 / 固定 JSON。`prompt_eval_count` / `eval_count` 等の数値も本物の token 計測ではない。Pact contract が要求する型 (integer) は満たすが、計算ロジックの回帰は本スイートでは検出できない。`metadata.processing_time_ms` 等の usecase 側計算経路の単体テストに引き続き依存する。
- recap usecase の hierarchical reduce 経路 (8K/60K context 越えで起動) は本 Phase の対象外。stub はあくまで `/api/generate` と `/api/chat` (single-shot) しか動かさず、`_recursive_reduce_phase` の経路はカバーされない。
- summarize_usecase の `_clean_summary_text` が stub の英語固定文を加工した結果が `summary` 文字列に入るため、表示上「日本語要約に英語混入」のように見える。E2E は形状検査のみなので機能影響無し、ただし手動でレポートを開いた人が混乱する余地はある。README に明記済み。
