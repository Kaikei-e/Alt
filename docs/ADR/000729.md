# ADR-000729: user-facing REST を Connect-RPC へ移行し BFF outbound を全 mTLS 化する

## Status

Accepted

## Date

2026-04-15

## Affected Services

- alt-backend (Go) — 新規 2 RPC (`FeedService.GetFeedTags`, `RSSService.RandomSubscription`) 実装、既存 usecase/repository 再利用
- alt-butterfly-facade (Go) — Connect-RPC と REST の transport を分離、Acolyte も mTLS 経由に移行、ServerName 自動派生へ修正
- alt-frontend-sv (SvelteKit / Svelte 5) — `tagTrail.ts` の 4 REST caller を Connect-RPC へ全置換、対応する Connect ラッパ (`connect/rss.randomSubscription`, `connect/feeds/tags.getFeedTags`) を新設
- acolyte-orchestrator (Python) — 既存 nginx TLS sidecar `:9443` を BFF からの上流として採用（実装変更なし）

## Context

[[000725]] → [[000727]] で確立した東西 mTLS は、alt-butterfly-facade (BFF) だけが `MTLS_ENFORCE=false` のまま残っていた。原因は BFF が単一 `http.RoundTripper` で Connect-RPC (`http://alt-backend:9101`) と REST proxy (`http://alt-backend:9000`) の両方を扱っており、mTLS transport (`http2.Transport{TLSClientConfig, https_only}`) を共有すると plaintext `http://` に対しスキーム不一致で 502 を発生させたためである。

さらに BFF は alt-backend 以外の upstream (acolyte-orchestrator、TTS) にも同一 transport を流用しており、単純に mTLS を強制すると Acolyte 経路が無条件で切れる構造になっていた。実際、mTLS 永続化を試した際に「Acolyte が見れない」状態が発生している。

user-facing REST の残骸も問題だった。`src/lib/api/client/tagTrail.ts` にある 4 件 (`/feeds/random`, `/articles/by-tag`, `/articles/:id/tags`, `/feeds/:id/tags`) だけが SvelteKit の REST proxy 経路 `callClientAPI` に依存し、BFF を通して REST 経路 (`/v1/*`) を保持し続ける根拠になっていた。他のクライアント (`feeds.ts` / `articles.ts` / `feedLinks.ts`) は [[000718]] で既に Connect-RPC 化済。

本ADR のゴールは以下の 2 本立て。

1. **user-facing REST の Connect-RPC 化**: tagTrail を Connect-RPC へ移行し、BFF の plaintext REST 依存を user-facing 経路から取り除く。足りない 2 RPC のみ新設し、残り 2 件は既存 RPC を流用。
2. **BFF outbound の全 mTLS 化**: Connect-RPC / REST / Acolyte / TTS を用途別の transport に分離し、MTLS_ENFORCE=true のときは Acolyte も `https://acolyte-orchestrator:9443` (既存 nginx TLS sidecar) 経由にする。REST は plaintext のまま温存（OPML/dashboard/image-proxy の構造要件）。

ADR 着手前に Explore エージェントで 49 REST endpoint をマップし、既に **36 件 (73%)** は Connect-RPC 等価メソッドを持つと確認していた。残る 13 件のうち構造上 REST を維持すべきは 9 件 (OPML/dashboard/image-proxy/admin)、新規 RPC が必要なのは user-facing 4 件。さらに survey 後のコード再確認で `RSSService.RegisterFavoriteFeed` / `RemoveFavoriteFeed`、`ArticleService.FetchArticlesByTag` / `FetchArticleTags` は既に実装・frontend からも呼ばれていることが判明し、**真に未整備だったのは `GetFeedTags` と `RandomSubscription` の 2 件だけ**に圧縮できた。

## Decision

### alt-backend: 新規 2 RPC を既存 usecase/repository に薄くかぶせる

- `proto/alt/feeds/v2/feeds.proto` に `GetFeedTags(feed_id, cursor, limit) → tags[]` を追加
- `proto/alt/rss/v2/rss.proto` に `RandomSubscription() → {id, title, description, link, published_at}` を追加
- `alt-backend/app/connect/v2/feeds/feed_tags.go` は REST 側 `RestHandleFetchFeedTagsByID` と同じ `AltDBRepository.FetchFeedTags` を直接呼び、cursor (RFC3339) / limit (default 20, cap 100) の境界値ロジックも同一に保つ
- `alt-backend/app/connect/v2/rss/handler.go` の `RandomSubscription` は既存 `FetchRandomSubscriptionUsecase` を再利用し、`domain.ErrNoSubscriptions` を `connect.CodeNotFound` にマップ
- 両 handler とも `middleware.GetUserContext` で unauthenticated を弾いたあと、usecase 未配線時は `connect.CodeUnimplemented` を返す fail-safe を持つ
- 全 Connect-RPC テスト pass (`go test ./connect/v2/...`)。新 RPC に unauth + unimplemented の smoke test を追加

### alt-frontend-sv: tagTrail を Connect-RPC へ完全置換

- `src/lib/api/client/tagTrail.ts` を書き換え、REST util `callClientAPI` への依存を削除
- `createClientTransport` は SvelteKit `/api/v2/[...path]/+server.ts` proxy 経由で **BFF に向く**（既存）。backend 直結はしない (`feedback_no_direct_frontend_backend` 準拠)
- 既存 `fetchArticlesByTag` / `fetchArticleTags` を流用し、新規 `randomSubscription` (`connect/rss.ts`) と `getFeedTags` (`connect/feeds/tags.ts`) を追加
- 型変換レイヤはドメイン層 (`RandomFeedResponse` の `{ feed: {...} | null }` 形) に合わせる。空フィード (`id === ""`) を `null` にマップしてコンポーネントの既存契約を壊さない
- `bun run check` 0 errors

### alt-butterfly-facade: Connect-RPC と REST で transport を分離、Acolyte も mTLS へ

- `server.NewServerWithTransports(cfg, logger, connectTransport, restTransport)` を追加し、既存 `NewServerWithTransport` は両方に同じ transport を渡す互換シムにした
- `main.go` は MTLS_ENFORCE=true のとき `newMTLSBackendTransport()` で作る http2 mTLS transport を Connect-RPC + Acolyte + TTS に渡し、REST proxy には `nil` (= `http.DefaultTransport`) を渡す
- Acolyte URL は `ACOLYTE_CONNECT_MTLS_URL=https://acolyte-orchestrator:9443`、TTS は `TTS_CONNECT_MTLS_URL` で切替可能（env 空なら既存 plaintext URL）
- mTLS transport の `tls.Config.ServerName` の env 経由 override を削除。複数 upstream (alt-backend / acolyte-orchestrator) を同じ transport で流すため、SNI は URL から自動派生させる必要があった（旧実装は `ServerName=alt-backend` 固定で Acolyte 宛に SAN mismatch が発生していた）
- `compose/bff.yaml` のデフォルトを `ALT_BUTTERFLY_FACADE_MTLS_ENFORCE:-true` に昇格し、再起動しても mTLS が維持される
- `server_test.go` の既存テスト (`TestServer_TTSRoute_Success` / `TestServer_AcolyteRoute_Success`) は `httptest` で `http.DefaultTransport` を渡す形なので、上記分離によって regression なし

### 旧 nginx TLS sidecar の netns 復旧

Acolyte / tag-generator の TLS sidecar (`network_mode: service:<svc>`) は `docker compose up --force-recreate <service>` で対象サービスを作り直すと netns が入れ替わり、sidecar は `Cannot restart container ... joining network namespace ... No such container` で再起動不能になる。今回の作業中に実際にヒットしたので、**sidecar も同時に recreate する運用**を手順として確定した。

## Consequences

### Pros

- **Acolyte が mTLS 経由で BFF から復旧**。`peer=alt-butterfly-facade` として監査ログに乗り、トラスト境界が内部まで一貫した
- **user-facing REST が alt-frontend-sv の call surface から実質ゼロ**に。残る REST は OPML import/export、dashboard、image proxy、admin scraping、csrf-token、health の 9 件のみ
- **BFF の MTLS_ENFORCE=true を永続化**。`docker restart` / ホスト再起動でも mTLS が維持され、運用事故 (env 忘れ) が原理的に起きなくなる
- **Connect-RPC スタックを 73% → ほぼ 100% の user-facing カバレッジへ**。新 RPC は 2 件だけで済み、差分は最小
- **transport 分離設計が将来拡張しやすい**。REST を残したい endpoint (OPML 等) がある限り同じ seam が使い続けられ、Phase 3 の BFF allowlist 化にもスムーズに継承可能
- **SNI 自動派生化で mTLS の multi-upstream 再利用が正しく動作**。同じ leaf cert で任意の内部サービスに対してハンドシェイクできる

### Cons / Tradeoffs

- **BFF outbound に 2 種の transport が共存**。Connect-RPC ルート (mTLS) と REST ルート (plain) の混在はしばらく続く。完全一本化は「残 9 REST endpoint の RPC 化 or 廃止」が前提で、本ADR スコープ外
- **SNI 固定 override を失ったことによる運用柔軟性の低下**。将来 internal LB 越しで SNI を明示したいケースがあれば再導入が要るが、現状の Compose 直接通信では URL の hostname が SAN に含まれているため実害なし
- **nginx TLS sidecar の recreate 運用**。`docker compose up --force-recreate <svc>` 後に必ず sidecar も再作成する必要がある。docker compose に共有 netns の追従機能がないため、運用手順で吸収する
- **Acolyte/TTS の mTLS client cert 検証は sidecar 側で未強制** (`VERIFY_CLIENT=off`)。BFF が leaf cert を提示はしているが、sidecar は受理するだけで peer allowlist 照合をしていない。これも follow-up で `VERIFY_CLIENT=on` に上げる余地がある
- **Phase 3 (BFF REST allowlist) は未着手**。`/v1/*` catch-all は引き続き restTransport で alt-backend:9000 に流しているため、将来追加される unsanctioned REST endpoint を素通しで proxy する恐れが残る

### 検証状況

- alt-backend / alt-butterfly-facade / alt-frontend-sv / acolyte-orchestrator 全サービス healthy
- alt-backend の peer identity ログに `alt-butterfly-facade` / `search-indexer` / `pre-processor` が継続的に出現
- BFF 内部から `https://acolyte-orchestrator:9443/health` に mTLS 経由で curl 成功 (`{"status":"ok","service":"acolyte-orchestrator"} http=200`)
- BFF 起動ログ `BFF outbound clients: mTLS enforce enabled backend=https://alt-backend:9443 acolyte=https://acolyte-orchestrator:9443` を確認
- Cloudflare 越しの user path (`https://curionoah.com/api/v2/...`) は anonymous 302 (auth redirect) で想定通り
- alt-backend Connect-RPC テスト (12 services 全 pass)、BFF server_test (既存テスト破壊なし)、frontend `bun run check` (0 errors)
