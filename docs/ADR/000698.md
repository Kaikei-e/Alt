# ADR-000698: Acolyte structured output を Ollama format 依存から XML DSL + 自前パーサに移行する

## Status

Accepted

## Date

2026-04-11

## Affected Services

- acolyte-orchestrator (Python)

## Context

Acolyte のレポート生成パイプラインでは、Planner・SectionPlanner・Critic の 3 ノードが LLM に structured output を要求している。従来は Ollama の `format` パラメータに JSON Schema を渡し、GBNF 文法で出力を制約していた。

しかしこの方式には以下の問題があった。

1. **thinking token 予算枯渇**: Gemma4 の thinking tokens が `num_predict` 予算を消費し、JSON 出力が途中で切断される (ADR-671)。`think=false` と `format` の併用は Ollama #15260 バグを踏む
2. **format 制約の脆弱性**: Ollama の structured output は thinking/reasoning 系と未整理で、空の thought block が混入したり出力が不安定になる
3. **fallback 制御の困難**: JSON Schema 制約下では部分的に壊れた出力の salvage が難しく、全体が parse 失敗になる

refine.md の設計方針に基づき、Gemma4 を「structured-output model」ではなく「tagged text generator」として扱い、XML 風タグ付き DSL + 自前パーサで構造を回収する方式に移行する。

## Decision

**Planner・SectionPlanner・Critic の structured output を XML DSL + 3 層自前パーサに移行し、Ollama `format` パラメータへの依存を廃止する。**

### 設計原則

- LLM は自由文で XML タグ付きテキストを生成する（`format` 制約なし）
- 構造の回収はフレームワーク側（自前パーサ）が責任を持つ
- Writer は LONGFORM モードのまま変更しない（制御層のみ structured、本文生成は longform）
- 既存の Pydantic モデル (`QueryExpansionOutput`, `ClaimPlannerOutput`, `PlannedClaim`) は一切変更しない

### XML DSL 仕様

3 つのルートタグを定義:

| ノード | ルートタグ | 主要子要素 |
|---|---|---|
| Planner | `<plan>` | `<reasoning>`, `<section><key>...<query>...` |
| SectionPlanner | `<section_plan>` | `<reasoning>`, `<claim><text>...<claim_type>...<evidence_id>...` |
| Critic | `<critic>` | `<reasoning>`, `<verdict>`, `<revise_section>`, `<feedback>` |

## Implementation

### 3 層パーサ (`acolyte/usecase/graph/xml_parse.py`)

| 層 | 責務 | 関数 |
|---|---|---|
| Pre-cleaner | Gemma4 thought block 除去、XML ブロック抽出 | `strip_gemma_thought_block()`, `extract_tag_block()` |
| Tolerant parser | `xml.etree.ElementTree` + bare `&` 修復 | `parse_xmlish_block()` |
| Normalizer | parsed XML → Pydantic モデル形状の dict 変換 | `normalize_plan_output()`, `normalize_critic_output()`, `normalize_section_plan_output()` |

オーケストレータ `generate_xml_validated()` は既存の `generate_validated()` と並行配置:
- `format=` を **注入しない** (JSON Schema 制約なし)
- LLM raw text → parse → normalize → Pydantic validate
- truncation 検出: closing root tag 欠損 + `completion_tokens >= 95% * num_predict` で budget 25% 増

### Gateway 変更 (`acolyte/gateway/ollama_gw.py`)

`mode=LLMMode.STRUCTURED` + `format=None` の場合、`/api/generate` + `think=False` にルーティング:

```python
if mode == LLMMode.STRUCTURED:
    if format:
        return await self._generate_structured(...)  # 従来パス (JSON Schema)
    return await self._generate_freetext(..., think=False)  # XML DSL パス
```

これにより XML ノードは `temperature=0` + `think=False` で自由文生成しつつ、structured モードのデフォルト設定を継承する。

### ノード変更

3 ノードの変更内容は共通パターン:
1. プロンプトの "Return JSON with..." → XML DSL 指定 + 1 例に変更
2. `generate_validated()` → `generate_xml_validated()` に切替
3. `_CRITIC_FORMAT` 等の JSON Schema 定数を削除
4. parse failure の fallback ロジックは既存と同一を維持

### テスト

34 件の XML パーサ専用テスト (`test_xml_parse.py`) を新設。既存テストの FakeLLM レスポンスを JSON → XML に更新。全 454 テストパス。

## Consequences

### Pros

- thinking token と format 制約の干渉問題を根本回避。`think=false` + `format` の Ollama バグ (#15260) に依存しない
- 部分的に壊れた XML 出力の salvage が可能 (bare `&` 修復、thought block 除去、prose 除去)
- パーサの各層が独立テスト可能。LLM 出力の新しい壊れ方に対して修復ルールを追加しやすい
- 既存の Pydantic モデルが不変。Gateway・Usecase 間の契約が維持される
- `generate_validated()` (JSON Schema 方式) は残存しており、必要に応じて切り戻し可能

### Cons / Tradeoffs

- XML 出力の品質は LLM のプロンプト追従性に依存。JSON Schema の GBNF 文法制約ほど厳密ではない
- パーサの修復ルールは最小限に保つ必要がある。過剰な auto-repair は新しい壊れ方を作る
- `xml.etree.ElementTree` を使用 (untrusted data に対する XXE リスク)。LLM 出力のみを対象とし外部入力は受け付けないため許容
- Gemma4-E4B (4.5B effective) の XML 追従性は実運用で検証が必要。malformed rate のモニタリングが今後の課題
