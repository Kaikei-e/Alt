# ADR-000725: 東西通信に step-ca ベースの mTLS 基盤を段階導入し alt-backend に HTTPS リスナーを追加する

## Status

Accepted

## Date

2026-04-14

## Affected Services

- alt-backend (Go) — `:9443` HTTPS リスナー追加、`alt-backend/app/tlsutil/` 共有パッケージ新設
- step-ca (smallstep/step-ca) — 新規 PKI サービス
- step-ca-bootstrap (oneshot) — ルート CA バンドル配布
- alt-backend-cert-init / alt-backend-cert-renewer — 証明書発行・自動更新サイドカー

## Context

Alt の東西通信は `X-Service-Token` 共有ベアラ 1 本に依存している。単一の Docker secret (`secrets/service_secret.txt`) が 10 以上のサービスにマウントされ、以下の High 重大度課題が残存していた。

- **T1 横展開**: 任意 1 コンテナ侵害で全東西通信を完全に偽装可能
- **T3 呼び出し元不明**: 受信ミドルウェアが `service.authenticated=true` しか記録せず、監査ログで caller を追跡不能
- **T5 ローテ**: 共有秘密のため全サービス同時再起動以外の方法でローテ不可
- **T7 平文伝送**: コンテナ間通信は全て `http://` で、トークン自体が `alt-network` 上を平文で流れる

共有トークン層のコード側 Critical ハードニング（定数時間比較など）は別トラックで進行中であり、本ADR ではそれと競合しない「新規追加のみ・可逆」でトランスポート層 mTLS をオーバーレイとして重ねる設計を採用する。

先行して Web 調査および OWASP / NIST ガイダンスに基づき PKI 選定を実施した結果、単一 Compose ホスト・~20 サービス規模では smallstep `step-ca` が運用負荷とセキュリティ要件のバランスで最適と判断した（SPIFFE/SPIRE・Istio/Linkerd・Consul Connect は過剰または K8s 前提で不採用）。

本ADR は段階展開計画（Phase 0〜4）のうち、Phase 0（PKI 基盤）および Phase 1 の最初のマイルストーン（alt-backend の受信側 TLS 有効化）の決定を記録する。

## Decision

### PKI 基盤

**smallstep step-ca** を Compose 内 PKI として採用する。

- 新規 `compose/pki.yaml` に `step-ca` サービスと `step-ca-bootstrap` oneshot を追加
- ルート CA (ECDSA P-256, 10 年有効) と中間 CA を自動初期化 (`DOCKER_STEPCA_INIT_*` 環境変数)
- JWK bootstrap provisioner をパスワードファイル保護 (`secrets/step_ca_root_password.txt`) で設定
- Remote Management / ACME は無効化 (Phase 0 では用途なし)
- `step-ca-bootstrap` oneshot がルート CA バンドルを共有読み取り専用 volume `pki_trust_bundle` の `/trust/ca-bundle.pem` に公開

### 証明書発行・更新パターン

サービスごとに 2 つの補助コンテナを配置する。

- **`<svc>-cert-init`** (restart: no, one-shot): 起動時に cert volume が空であれば step-ca から leaf cert を取得して保存。冪等。
- **`<svc>-cert-renewer`** (restart: unless-stopped, sidecar): `step ca renew --daemon --expires-in=8h` で 24h 有効の cert を 8h 残で自動更新。

SAN 戦略: `CN=<compose-service-name>`, `SAN=<service>, localhost`。Compose DNS での handshake を成立させる。

Compose 共通設定は `core.yaml` の `x-tls-cert-sidecar` YAML アンカーで共通化する（anchor は YAML ファイル境界を越えないため、他 yaml に展開する際は同等の anchor を定義する）。

### alt-backend の HTTPS リスナー追加

- 新規共有 Go パッケージ `alt-backend/app/tlsutil/` を TDD で実装
  - `LoadServerConfig` / `LoadClientConfig`: `GetCertificate` / `GetClientCertificate` コールバック経由で mtime ベースのホットリロードを提供（fsnotify 不要の簡潔な実装）
  - `NewMTLSHTTPServer`: `IdleTimeout: 60s` を強制し、HTTP/2 コネクション再利用が leaf cert の有効期限を越えることを防ぐ
  - `MinVersion: tls.VersionTLS13` 固定
- `alt-backend/app/main.go` に Phase 1 用の追加 HTTPS リスナーを新設
  - `MTLS_LISTEN=true` env で `:9443` を追加起動（既定オン）
  - Phase 1 は `ClientAuth=NoClientCert` のサーバ証明書のみ。クライアント認証は引き続き `X-Service-Token` で継続
  - 既存の REST `:9000` と Connect-RPC `:9101` 平文リスナーは従来どおり稼働（可逆性担保）
  - Connect-RPC ハンドラを mTLS リスナーで共用し、将来 Phase 2 で `RequireAndVerifyClientCert` に昇格しやすい構造にした

### 既存コードへの非干渉原則

本変更は `service_auth_middleware.go` / `service_auth_interceptor.go` 等の共有トークン認証実装に**一切手を入れていない**。別セッションで進行中の共有トークン・ハードニングと競合しないよう、新規ファイル追加と新規 env flag 分岐のみで構成した。

## Consequences

### Pros

- **監査上の脅威閉鎖の足場を確立**: PKI が稼働し、alt-backend が `:9443` で TLSv1.3 を提供する状態に到達。これにより T7（平文伝送）の解消準備と T3（呼び出し元不明）解消の土台（後段 Phase 2 で peer identity interceptor 追加）が整った。
- **完全可逆**: `MTLS_LISTEN=false` への切替と再起動だけで従来状態に復帰。共有トークン経路は完全に温存。
- **自動ローテ**: 24h 有効の leaf を 8h 残で renewer sidecar が更新。無停止でのローテを常態化（T5 の構造的解決）。
- **観測容易性**: `:9101` と `:9443` の両ポートを公開したため、段階移行中は caller が TLS を使ったか否かをネットワークレベルで区別できる。
- **TDD による品質担保**: `tlsutil` パッケージは 6 テスト全 pass（`-race` クリーン）。mtime ベースのリロード、ファイル欠損時のエラー、IdleTimeout バウンド等を網羅。

### Cons / Tradeoffs

- **JWK provisioner のパスワードが root CA パスワードと同一**: 運用簡略化のため Phase 0 で `DOCKER_STEPCA_INIT_PASSWORD_FILE` を使った初期化時に両者が同じパスワードになっている。Phase 2 以降で独立した provisioner password に分離する技術的負債。
- **step-ca 初期化ログにプロビジョナ admin password が 1 回だけ平文表示される**: `DOCKER_STEPCA_INIT_REMOTE_MANAGEMENT=false` により remote API 経由では無効だが、ログローテーションの運用が必要。
- **cert volume を alt-backend 側で `ro` マウント**: cert-init/renewer と alt-backend が同じ named volume (`alt_backend_certs`) を共有する形。SecurityContext を細かく制御する余地は小さい（Compose の制約）。
- **Python/他 Go サービスは未対応**: 本ADR では alt-backend 単体のみ。pre-processor / search-indexer / alt-butterfly-facade / tag-generator / acolyte-orchestrator / tts-speaker / recap-worker への展開は Phase 1 残タスクと Phase 3 でフォローアップ。
- **Compose YAML anchor のファイル間非伝播**: `x-tls-cert-sidecar` アンカーは `core.yaml` 内でのみ有効。他サービスへ展開する際は各 yaml に同等 anchor を定義する冗長性がある。

### 検証状況

- `step-ca` コンテナ healthy、Root CA (ECDSA P-256) を発行、trust bundle を `pki_trust_bundle` volume に配布
- `alt-backend-cert-init` が leaf cert を発行（CN=alt-backend、Issuer=alt-ca Intermediate CA、24h 有効）
- `alt-backend` healthy、`:9000 :9101 :9443` の 3 リスナー稼働を確認
- 同一ネットワーク上のクライアントから `curl --cacert /trust/ca-bundle.pem https://alt-backend:9443/` を実行し、TLSv1.3 ハンドシェイク成功・SAN 検証通過を確認
- `:9101` 平文経路は従来どおり稼働（`X-Service-Token` 経路に影響なし）
- `go test ./tlsutil/... -race` 全件 pass
