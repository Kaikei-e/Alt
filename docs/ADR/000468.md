# ADR-000468: Recall Rail のユーザー探索を DB 動的取得に変更し projector を reproject-safe にする

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-backend (Go)

## Context

Knowledge Home の Recall Rail が常に「Nothing to recall right now.」を表示し、24時間以上前に記事を開いていても候補が出ない問題が発生していた。

調査の結果、3層の原因が特定された。

### 原因1: RecallProjectorJob が空の AllowedUserIDs に依存

`main.go` で `.env` の `KNOWLEDGE_HOME_ALLOWED_USER_IDS` をパースして `recallUserIDs` スライスを構築し、`RecallProjectorJob` と `DigestAvailabilityReconcileJob` に渡していた。この値が空文字列のため、スライスは常に空になり、両 job とも `user_count=0` で即 return していた。

`KNOWLEDGE_HOME_ROLLOUT_PERCENTAGE=100` とは別系統のため、UI 上は Recall Rail のセクションが表示されるが、projector が動かないため候補が一切生成されない状態だった。

### 原因2: `projectHomeItemOpened()` が reproject-unsafe

`knowledge_projector.go` 内の `projectHomeItemOpened()` で `now := time.Now()` を使用し、`eligibleAt := now.Add(24 * time.Hour)` として recall candidate の eligible 時刻を計算していた。reproject 時にこの値が再計算されるため、25時間前の open イベントでも `eligibleAt` が未来になり、eligible candidate が 0 件のままとなっていた。

これは Alt のイミュータブルデータモデル不変条件「Reproject-safe: Projectors use event payload only, never latest state queries」に違反していた。

### DB 実データ（調査時点）

| テーブル | 行数 | 状態 |
|---------|------|------|
| `knowledge_events` | 335,824 | HomeItemOpened: 15件 |
| `recall_signals` | 0 | シグナル未生成 |
| `recall_candidate_view` | 11 | 全て `next_suggest_at > NOW()` |

## Decision

### 1. `ListDistinctUserIDsPort` による動的ユーザー探索を採用する

静的な `AllowedUserIDs` 設定に依存せず、DB から動的にユーザーを取得する方式に変更する。

- `knowledge_home_port.ListDistinctUserIDsPort` インターフェースを新設
- `knowledge_home_items` テーブルから `DISTINCT user_id` を取得（nil UUID を除外）
- `RecallProjectorJob` と `DigestAvailabilityReconcileJob` の両方でこの port を使用

取得元として `knowledge_home_items`（disposable projection）を選択した理由:

- Knowledge Home のアクティブユーザーのみが対象となり、system events の UUID 0 が自然に除外される
- ユーザー別に投影済みのため行数が少なく、`DISTINCT` クエリが高速
- disposable projection からの取得であるため、projection rebuild 中は一時的にユーザーが見つからない可能性があるが、job は次回実行時にリトライされるため実害はない

### 2. `projectHomeItemOpened()` を reproject-safe にする

`time.Now()` の代わりに `event.OccurredAt` を使用して以下の値を計算する:

- `LastInteractedAt`: `event.OccurredAt`（いつ操作したかはイベント時刻が正）
- `eligibleAt` / `NextSuggestAt` / `FirstEligibleAt`: `event.OccurredAt + 24h`（reproject しても同じ値になる）

`GeneratedAt` と `UpdatedAt` は「projection が生成/更新された時刻」の意味を持つため、`time.Now()` のまま維持する。

### 3. `main.go` から静的 AllowedUserIDs パースを削除する

`recallUserIDs` スライスの構築コード（`strings.Split` + `uuid.Parse`）を削除し、`container.KnowledgeHomeGateway` を `ListDistinctUserIDsPort` として直接渡す。

## Consequences

### Pros

- Recall Rail が正常に機能する（`user_count=1` をログで確認済み）
- `.env` の `KNOWLEDGE_HOME_ALLOWED_USER_IDS` 設定が不要になり、運用負荷が軽減
- 新規ユーザーが Knowledge Home を使い始めた時点で自動的に recall/digest 処理の対象になる
- `projectHomeItemOpened` が reproject-safe になり、reproject 実行後も recall candidate の `next_suggest_at` が正しい値を保つ
- イミュータブルデータモデルの不変条件との整合性が回復

### Cons / Tradeoffs

- `knowledge_home_items` は disposable projection であるため、projection rebuild 中はユーザーが一時的に発見されない可能性がある。ただし job は定期実行のため、rebuild 完了後の次回実行で回復する
- 毎回 `DISTINCT user_id` クエリを実行するが、現時点でユーザー数が少なく（1名）、将来的にも Knowledge Home 利用者数は限定的なため、パフォーマンス影響は無視できる
- `recall_signals` が 0 件のままのため、signal ベースの候補生成は依然として新規アクション待ち。今回の修正で解消されるのは「job がユーザーを走査できるようになる」点と「既存 open candidate の時刻補正」の2点
