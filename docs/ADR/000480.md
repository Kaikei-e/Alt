# ADR-000480: Recall Rail 更新を canonical recall_changed と Home refresh 同期で扱う

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-backend (Go)
- alt-frontend-sv (SvelteKit / TypeScript)

## Context

Knowledge Home の stream contract では `recall_changed` が canonical event type として予約されていたが、backend 実装では recall 系イベントが `digest_changed` に寄せられていた。このため Recall Rail に特化した変更があっても、stream 上では digest refresh の一種としてしか扱えなかった。

同時に frontend の Home ページは stream apply 後に `home.fetchData()` だけを呼んでおり、single-fetch contract で返ってくる `recallCandidates` を Recall Rail store に再注入していなかった。その結果、Home 本体は更新されても Recall Rail が古い candidate 配列を保持し続けることがあった。

この状態には 2 つの問題がある。

- stream contract と実装がずれ、Recall update を独立した関心事として観測できない
- apply 後に Home response を取得しても、Recall Rail store に反映されず UI が stale になる

Recall Rail は Home 画面に埋め込まれているため、canonical stream type と page-level refresh の両方で整合性を取る必要がある。

## Decision

**Recall Rail の更新は canonical `recall_changed` として配信しつつ、Home 画面の apply refresh 完了後に `recallCandidates` を必ず再同期する方針を採用する。**

具体的には以下を採用する。

1. backend は `RecallSnoozed` と `RecallDismissed` を `recall_changed` に map する。
2. `StreamKnowledgeHomeUpdatesResponse` では `recall_changed` に対して最小 `recall_change` payload を生成し、可能な場合は現在の recall candidate で enrich する。
3. enrich で該当 candidate が見つからない場合でも event 自体は送る。dismiss や snooze の結果 candidate が上位候補から消えるケースを失敗扱いにしない。
4. frontend の stream hook は `recall_changed` payload を lossless に保持する。
5. Home ページの `onRefresh` は `home.fetchData()` の完了後に `recall.setCandidates(home.recallCandidates)` を呼び、single-fetch contract を Recall Rail の再同期 source とする。
6. page-level refresh の責務は維持し、今回の段階では `recall_changed` payload だけで Recall Rail DOM を直接差し替えない。

この構成により、stream contract は canonical になり、UI は既存の safe refresh モデルを保ったまま Recall Rail を stale にしない。

### 検討した代替案

- recall 系イベントも引き続き `digest_changed` に寄せる  
  contract と実装のズレが残り、Recall update を独立に扱えないため不採用。

- `recall_changed` を導入し、frontend は payload だけで即時 DOM 更新する  
  より反応は速いが、Home response を source of truth とする現行設計と二重管理になる。今回の修正範囲では不採用。

- page refresh 時に Recall Rail を個別 unary fetch する  
  single-fetch contract を壊し、API 呼び出し数も増えるため不採用。

## Consequences

### Pros

- Recall update が canonical stream contract 上で明示的になる
- `StreamKnowledgeHomeUpdates` の利用者が digest と recall を区別して扱える
- Home 画面の apply 後に Recall Rail が stale state を保持しなくなる
- single-fetch contract を保ったまま、Recall Rail を Home response と一致させられる

### Cons / Tradeoffs

- Recall Rail の更新経路が「stream payload」と「apply 後の full refresh」の二層になる
- `recall_changed` payload は enrich できない場合に最小 shape へ退化する
- Recall Rail を payload 差分だけで直接反映する最適化は次段階へ持ち越しになる
- backend は stream event ごとに recall candidate 解決を試みるため、stream 処理の責務が少し増える
