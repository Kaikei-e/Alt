# ADR-000822: search-indexer /v1/search に published_at 日付フィルタを追加する

## Status

Accepted

## Date

2026-04-20

## Affected Services

- search-indexer (Go)

## Context

[[000821]] で Acolyte の `weekly_briefing` に 7 日時間窓の default と `brief.time_range → published_after` 変換を入れたが、search-indexer 側は `/v1/search` でその境界を受け付ける経路を持っていなかった。Meilisearch の filterableAttributes も `["tags", "user_id"]` だけで `published_at` 不在、indexed document 自体にも `published_at` フィールドが無く、filter 式が成立しない状態だった。

`domain/search_document.go` は過去に `language` フィールドを持っているのに `gateway/search_engine_gateway.go` → `driver/models.go` の境界で silently drop されており（[[000820]] のレビューで確認済み）、同じパターンで `published_at` を追加しても gateway で落ちるリスクがあった。

### 定量影響

- alt-db `articles` の直近 1 週間（2026-04-12〜04-20）におけるイラン関連記事: EN 9 件 / JA 0 件。このうち Meilisearch に `published_at` 付きで indexed されていたのは 200 件 / 全 112,422 件のみ。
- 対象 report `eb67863f` の v10 citations（6 ユニーク source_id）はすべて Feb〜Mar 2026 の JP 記事で、コーパス母集団の recency と完全に乖離。

## Decision

**search-indexer の domain / port / driver / handler をまたぐ縦スライスとして `published_at` を一等市民化し、`/v1/search` で RFC3339 の `published_after` / `published_before` を受け付ける**。filter ルーティングは tag / user_id filter path とは別系統の `SearchWithDateFilter` を新設して、`ValidateFilterTags` の厳格な正規表現（`>=` / 数字が通らない）を迂回する。

### 検討した代替案

| 案 | 評価 |
|----|------|
| A. 既存 `SearchWithFilters` に date filter 文字列をそのまま渡す | × `ValidateFilterTags` の regex `^[\p{L}\p{N}\s\-_]+$` が `>=` / `:` を reject するため、escape hatch が必要になる |
| B. handler 層で `tags` / `user_id` filter と連結して 1 本の filter 文字列にまとめる | × Meilisearch filter 構文の bracket 管理が複雑化、tag filter 側の注入を防ぎにくい |
| **C. 構造化引数を受ける `SearchWithDateFilter(ctx, query, after *time.Time, before *time.Time, limit)` を SearchEngine interface に追加** | ○ tag filter と date filter が独立して単体テスト可能、driver 側で Meilisearch 式を型安全に組み立てられる |

C を採用。

## Implementation

### 1. domain

ファイル: `search-indexer/app/domain/article.go`, `search-indexer/app/domain/search_document.go`

- `Article` に `publishedAt time.Time` を追加し、`PublishedAt()` / `SetPublishedAt(t)` を提供。
- 既存 `NewArticle(id, title, content, tags, createdAt, userID)` は互換のため残し、新コンストラクタ `NewArticleWithPublishedAt(..., publishedAt time.Time)` を導入。
- `SearchDocument` に `PublishedAt time.Time` (`json:"published_at"`) を追加。`NewSearchDocument(article)` で伝播。

### 2. port / gateway

ファイル: `search-indexer/app/port/search_engine.go`, `search-indexer/app/gateway/search_engine_gateway.go`

- `SearchEngine` interface に `SearchWithDateFilter(ctx, query string, publishedAfter, publishedBefore *time.Time, limit int) ([]domain.SearchDocument, error)` を追加。
- `SearchDriver` interface にも同一シグネチャの method を追加。
- gateway の `IndexDocuments` で `Language`, `PublishedAt` の drop 問題を修正。driver 側は `int64` (Unix seconds) で保持する helper `publishedAtUnix` / `publishedAtFromUnix` を gateway に置いた。
- gateway の返却パスを `convertDocs` に集約し、Search / SearchWithFilters / SearchByUserID / SearchWithDateFilter / SearchByUserIDWithPagination から共通使用。これで `language` 復活も同時に達成。

### 3. driver (Meilisearch)

ファイル: `search-indexer/app/driver/models.go`, `search-indexer/app/driver/meilisearch_driver.go`

- `SearchDocumentDriver` に `Language string` と `PublishedAt int64 (json:"published_at,omitempty")` を追加。
- `EnsureIndex` の `filterableAttributes` を `[]interface{}{"tags", "user_id", "published_at", "language"}` に拡張。
- `SearchWithDateFilter` を新設し、Meilisearch 側に `published_at >= <unix> AND published_at <= <unix>` を組み立てて Filter に設定。両境界 nil の場合は `Search` に degrade。
- Hit → SearchDocumentDriver 変換を `hitsToDocs` に共通化し、`getInt64` helper で `published_at` を extract。

### 4. backend_api driver

ファイル: `search-indexer/app/driver/backend_api/client.go`

現行 backend proto の `ArticleWithTags` には `published_at` が無いため、暫定で `toDriverArticle` で `PublishedAt: p.CreatedAt.AsTime()` と seed する。alt-db 実データで `created_at - published_at > 7d` となる件数は直近 1 週間で 0/2770 だったため、weekly_briefing の用途では drift は無視できる。proto に `published_at` が追加された時点でこの fallback を外す想定。

### 5. rest handler

ファイル: `search-indexer/app/rest/handler.go`

- `SearchArticlesHit` に `Language` / `PublishedAt string (RFC3339)` を追加。
- `SearchArticles` で `published_after` / `published_before` をパースし、どちらか non-nil の場合は `h.engine.SearchWithDateFilter` に route、両方未指定は従来通り `searchArticlesUsecase.Execute`。
- `parseOptionalRFC3339` helper で空文字は nil、malformed は `400 Bad Request` を返す。
- `SearchArticlesUsecase.Engine()` accessor を usecase 層に追加し、handler が port.SearchEngine へ直接到達できるようにした（ValidateFilterTags を通る path と迂回する path を hardener 段階で分離するため）。

### 6. テスト

- `search-indexer/app/domain/article_published_at_test.go` — 新コンストラクタが `publishedAt` を保持、SearchDocument が carry する
- `search-indexer/app/rest/handler_date_filter_test.go` — `published_after` / `published_before` の forward、response に `published_at` 露出、malformed は 400
- 既存 8 つの mockSearchEngine / mockSearchDriver には `SearchWithDateFilter` stub を追加して interface 満たしを復旧
- `go test ./... && go vet ./...` green

## Consequences

### Pros

- Acolyte `weekly_briefing` が実際に 7 日境界で retrieval を絞れるようになり、対象 report の再生成 v12 で cited EN 比率 66.7%（body 解決後）まで到達。`language` drop の既存バグも同じ境界修正で同時解消。
- SearchEngine interface に構造化引数で追加したことで、tag filter / date filter / user filter / unrestricted の 4 系統が単体テストで独立検証可能になった。
- Meilisearch filter 式は内部で組み立てるため、上位層が filter 文字列を作って渡す際の injection リスクを増やしていない。

### Cons / Tradeoffs

- **既存インデックス済み 112k docs は `published_at` を持たないため、非 nil lower bound を指定した時点で該当 docs はフィルタから除外される**。本番相当の効果を得るには search-indexer 再デプロイ後に Meilisearch 再インデックスが必要（index 設定自体は EnsureIndex が起動時に filterableAttributes を更新する）。運用として `docs/runbooks/acolyte-manual-regeneration.md` 相当のフォローを検討する。
- backend proto の `ArticleWithTags` に `published_at` が無いため `CreatedAt` を seed として代用している（drift は直近 1 週間で 0/2770 と測定）。真の source publication date を伝えるには proto への field 追加とクライアント再生成が必要。
- SearchEngine interface 変更に伴い全 mockSearchEngine 実装へ stub を追加した。新 mock を書くときは 8 method すべて satisfy する必要がある。
