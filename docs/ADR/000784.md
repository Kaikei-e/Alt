# ADR-000784: pki-agent に TLS listener self-probe と fail-fast を追加してコンテナ状態と機能 liveness を一致させる

## Status

Accepted

## Date

2026-04-19

## Affected Services

- **alt-pki-agent** — `main.go` の tick loop に TLS listener self-probe を追加。`127.0.0.1:${PROXY_LISTEN_PORT}` へ短い TLS handshake を走らせ、N 回連続失敗で `os.Exit(1)` する。compose の `restart: unless-stopped` で再起動した新コンテナが現親の netns に attach することで自己治癒する
- **compose** — `x-pki-agent` anchor の `healthcheck` を cert 鮮度ベースから listener probe ベースに切り替え。既存環境変数 `PROXY_LISTEN` を再利用する

## Context

[[000782]] で実測した通り、2026-04-19 の recurrence において pki-agent サイドカーは `docker inspect ... --format '{{.State.Health.Status}}'` で **`healthy`** を返し続けていた。ログにも:

```
{"time":"...","level":"INFO","msg":"tick ok","state":"fresh"}
```

が刻まれ続け、一方で同じ瞬間 BFF からは `dial tcp 172.18.0.14:9443: connect: connection refused` が出ていた。これは pki-agent の healthcheck が **cert 鮮度だけを確認しており、listener が正しい netns で bind している事実を確認していない**ことによる silent failure である。

### 同パターンの履歴

- [[PM-2026-028]] tag-generator 系での sidecar listener 孤立
- [[PM-2026-030]] pki-agent stale netns で Acolyte が 502（初回）
- [[PM-2026-031]] mTLS cutover 残タスクで Acolyte 502 再発（2 回目）
- [[000782]] 2026-04-19 の 3 回目

どの PM も共通して「コンテナ `Up (healthy)`、cert fresh、でも機能は死」と書いている。現行 healthcheck のセマンティクスが cert 鮮度に留まっている限り、orchestration 層から見ると緑のまま止まる silent failure は構造的に避けられない。

### 問題の本質

pki-agent の責務は **TLS reverse proxy を提供すること**である。cert 鮮度はその前提条件に過ぎない。healthcheck は責務 = listener liveness を直接測るべきで、前提条件の一部だけで緑を返すべきではない。

また、[[000783]] で alt-deploy 側に cascade recreate を入れても、**deploy 経路を通らない parent 再起動**（手動 `docker restart`, OOM kill, node reboot など）では cascade は走らない。pki-agent 自身が孤立を検知して exit できなければ、compose の `restart` policy を活用できない。

## Decision

**pki-agent の tick loop に自 listener への TLS handshake probe を追加し、N 回連続失敗で `os.Exit(1)` する。compose の restart policy で再作成されたコンテナが現親の netns に attach することで自己治癒させる。** 採用構成:

1. **tick loop で listener self-probe を走らせる**。既存の `tick_interval` (`5m`) を踏襲し、cert 更新と同じ頻度で listener の生死を検査する。probe は `tls.Dial("tcp", "127.0.0.1:${PROXY_LISTEN_PORT}", tlsConfig)` で short handshake を試みる。成功で counter をリセット、失敗で counter を増やす
2. **N = 3 連続失敗で `os.Exit(1)` する**。N はビルド時定数で始め、運用知見が溜まったら環境変数 (`LISTENER_PROBE_FAIL_THRESHOLD`) で tunable に拡張する。初期値は「transient errors を 1-2 回吸収、構造的 orphan は 15 分以内に自殺」を満たす
3. **compose の `restart: unless-stopped`** が再作成時に新親の netns に attach する。`network_mode: service:<parent>` は compose が解決するため、pki-agent 側は exit するだけで良い
4. **healthcheck を listener probe ベースに書き換える**。`x-pki-agent.healthcheck` を `openssl s_client -connect 127.0.0.1:${PROXY_LISTEN_PORT:-9443} -quiet </dev/null` 等、実際のハンドシェイクを成立させるコマンドに差し替える。cert 鮮度は tick loop が担うため healthcheck からは外す
5. **metrics 出力を拡張する**。`:9510` に `pki_agent_listener_probe_failures_total` カウンタと `pki_agent_listener_probe_last_success_seconds` ゲージを追加。rask-log-aggregator 側のダッシュボードで可視化する
6. **段階適用 (pki-agent-acolyte-orchestrator 先行)**。最初にリスクの高い acolyte-orchestrator sidecar のみで有効化し、運用 1 週間で exit 頻度と回復時間を計測。問題なければ tag-generator, recap-subworker など他 sidecar に横展開する

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| 現行の "cert fresh" tick log を healthcheck に引き直すだけ | 症状不変。cert 鮮度と listener 生死は別事象なので、記述整理だけでは silent failure モードが消えない |
| Prometheus 側で listener probe を外部から行う | 外部 probe は pki-agent の netns を跨ぐ必要があり実装が煩雑。self-probe の方が同じ netns で完結し、原理的に正しい |
| compose の healthcheck に `nc -z 127.0.0.1 9443` だけ入れる | TCP listen 有無しか見ない。TLS handshake まで試すほうが cert 更新直後の listener 再起動失敗も検出できる |
| exit ではなく graceful reload で listener を再 bind する | reload は同じプロセス内の事象で、親 netns が変わった場合は効かない。exit + compose restart の方が原理的に正しい |
| exit 閾値を 1 にして即殺 | transient TLS error（特に cert rotation 中のリスタート直後）で偽陽性が出やすい。3 回で 15 分猶予を持つ方が安全 |
| tick_interval を 30s に短縮して早期検知 | cert 更新のオーバーヘッドが 10 倍に。listener probe だけなら別 interval を持たせる方向もあるが、運用複雑化を避ける |
| Kubernetes readiness/liveness probe に丸投げ | Alt は Compose-first ([[CLAUDE]]) であり、k8s 前提の設計は対象外 |

### 範囲外 (今回やらないこと)

- alt-deploy 側の cascade recreate step — [[000783]] で別 ADR
- sidecar 以外 (news-creator backend など) への同パターン適用 — 要否は運用知見後に判断
- tick loop 全体のリファクタ — self-probe は既存 tick に最小侵襲で追加
- tls.Dial の暗号スイート固定化 — 本 ADR は liveness 検証が主目的で、crypto policy は別 ADR で扱う

## Consequences

### Pros

- **pki-agent のコンテナ状態と機能 liveness が一致する**。`healthy` = listener が現親の netns で bind している、という運用前提が成立する
- **deploy 経路を通らない親再起動（手動 restart, OOM kill, node reboot）でも自己治癒する**。[[000783]] の alt-deploy cascade は deploy 経路のみカバーするため、本 ADR と組み合わせて defence-in-depth を構成する
- **silent failure が noisy failure になる**。失敗時は exit で container restart カウントが増え、監視からも即座に可視化される
- **metrics (`listener_probe_failures_total`) がダッシュボード化できる**。exit に至る前の transient error 頻度が可観測化され、cert rotation 中の一瞬の失敗か構造的孤立かを区別できる
- **実装は main.go の tick 内ブランチ追加で済む**。既存コードへの侵襲が小さく、review 面積を抑えられる

### Cons / Tradeoffs

- **exit を伴う自己治癒はコンテナ再起動を引き起こす**。頻度次第で rask-log-aggregator のログに restart event が増える。閾値 N と tick_interval のチューニングが要る
- **TLS self-probe は PROXY_VERIFY_CLIENT=on の sidecar では自身向けに client cert を用意する必要がある**。acolyte-orchestrator sidecar は `ACOLYTE_ORCHESTRATOR_VERIFY_CLIENT:-on` のため、localhost probe 用に VERIFY をバイパスするルート（loopback 専用 listener or SNI ベース例外）が必要。実装コストとして認識
- **healthcheck を listener probe ベースに変えると初回起動直後に一瞬 `unhealthy` になる可能性**。compose の `start_period` を適切に設定しておかないと deploy が誤判定する
- **exit で netns 孤立以外のバグが覆い隠される可能性**。listener が一時的に落ちる別原因（step-ca 間欠的失敗など）でも container restart になり、根本原因のログが流れる。metrics + rask-log の保持期間で補う
- **段階適用期間中は sidecar 間で挙動が不揃いになる**。pki-agent-acolyte-orchestrator だけ exit 機構ありで他は cert-only、という状態が 1-2 週間残る
- **本 ADR のロールバックには code revert + image rebuild が必要**。compose の toggle だけでは戻せないため、オンコール判断が重くなる
