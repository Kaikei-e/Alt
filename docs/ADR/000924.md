# ADR-000924: Knowledge Loop の Revisit / Ask 体験を回復しつつ projector を reproject-safe に直す

## Status

Accepted

## Date

2026-05-26

## Affected Services

- alt-frontend-sv (Svelte 5, knowledge_loop): `canTransition` を trigger-aware に拡張、Revisit を `intent_signal` 同一ステージ経路で発火
- alt-backend (Go): `knowledge_projector` ハンドラの `time.Now()` を `event.OccurredAt` に置換、`feed_url_normalize` utility と `GetFeedID` gateway の fallback lookup を追加
- knowledge-sovereign (Go): `SurfaceScoreInputs.ArticleID` を追加、resolver が前段の `SummaryVersionCreated` から pin、`surface_planner_cron` に空 tick heartbeat を追加
- pre-processor (Go): `UpsertArticles` に per-batch `FeedURL → FeedID` キャッシュ (positive / negative 両方) を追加
- docs/runbooks/knowledge-loop-reproject.md: heartbeat 後の SLO 観測手順を追記

## Context

2026-05-26 のトリアージで、Knowledge Loop に関するユーザー報告 3 件 ―
「Revisit がクリックできない」「Ask 遷移後に Summary と Article がクリックできずにエラーが出る」「Knowledge Loop の新着が止まる」―
が観測された。`/log-seeker` の 24 時間バンドル (`/tmp/log-seeker-20260526T112356Z/`) と
`knowledge-sovereign-db` 直読を組み合わせて掘り下げたところ、症状は単一原因ではなく
**独立した 5 つの欠陥**が同じ表面に滲み出ている状態だった。

| 観測 | 根本原因 | 影響 |
|---|---|---|
| `alt-backend` 262 件の `Connect-RPC GetFeedID "feed not found by URL"` バースト (10:38:52, trace_id 全て 0)、ClickHouse `otel_error_logs` で alt-backend=537 / 1h | `pre-processor.UpsertArticles` が article 単位で `GetFeedID` を発射、Reddit r/webdev RSS が `alt-db.feed_links` 未登録 (`SELECT ... WHERE url ILIKE '%reddit%webdev%'` で 0 rows) | 同一 URL に 262 回ヒット、warn ログ埋もれ、ingestion が止まり Knowledge Home の `generated_at` が 3 時間古いまま |
| `surface_planner_v2` projector lag が `now() - updated_at` で 142 秒、checkpoint が `last_event_seq=1304271` で凍結 | `surface_planner_cron.RunBatch` は signal-type 不一致で `events_seen=0` のとき `UpdateProjectionCheckpoint` を呼ばずに早期 `break` する。reproject 直後 (2026-05-25T15:11:20 に entries 58,936 件 truncate + checkpoint reset) で event 流入が薄いと数分単位で `updated_at` が動かない | SLO 監視を空振りで誤発火、runbook 上の lag 数値が projector liveness を反映しない |
| Revisit を押しても何も起きない | `LoopEntryTile.svelte` で `ctaToStage("revisit") => "orient"`、`canTransition("orient","orient")` は `loop_transition_policy.json` の `allowed_edges` に同一ステージを含まないため `false` を返し silent return。`toggleExpanded` がタイル展開と同時に `observe → orient` を optimistic 発火するため、Revisit ボタンが表示された時点で entry は既に `orient` 以降にいる | `decision_presented.v1` / `oriented.v1` が直近 4 時間で各 2 件しか積まれない (DB 実測)、OODA 1 周が完成しない |
| Ask 後に Article / Summary が押せずエラー | `augur.conversation_linked.v1` を投影するとき `seedActTargets(ev, inputs)` は現在 event の payload しか見ない。Augur イベントは `entry_key` と `conversation_id` だけを携える → article target が消え、FE `sourceUrl()` が null → `openInlineError="URL UNAVAILABLE — not_found"` | Ask の度に「もう開けない」状態に落ちる |
| `projectSummaryVersionCreated` 等が `time.Now()` を `GeneratedAt` / `UpdatedAt` / `Score` 算出に使用 | `IMPL_BASE` §"reproject-safe" の不変条件違反。同一 event log を 2 回流すたびに read model 行が差分を持つ | review `knowledge-home-phase1-5-remediation-directives-2026-03-18.md` の P0-1 / P0-4 が未消化として残っていた |

5 つの欠陥は経路が独立で互いに引き金にならないが、ユーザー体験としては
「Loop が止まって見える」ひとつの症状に集約していた。場当たり的に1件ずつ
PR を切ると別欠陥が次の triage を引き寄せるため、まとめて 1 commit
(139a8ab03) に束ねた。

## Decision

5 Pillar を **TDD outside-in (RED → GREEN → REFACTOR)** で束ねて landing する。

### Pillar 1: Frontend `canTransition` を trigger-aware にする

`loop-transitions.ts` の `canTransition(from, to)` は ADR-000831 §7 の
cross-stage allowlist だけを評価していた。ADR-000914 で proto canonical
`loop_transition_policy.json` に追加された
`same_stage_triggers: ["defer","recheck","archive","mark_reviewed","compare","internalize","intent_signal"]`
を FE 側でも評価できるよう、optional な `trigger` 引数を受ける形に拡張する。
同一ステージは `trigger` が同集合に含まれるときのみ通す。

Revisit は decision intent としては「再評価」の意味を持つので、`ctaToStage`
で `"orient"` に固定して cross-stage で発火するのは canonical policy と
食い違っていた。`LoopEntryTile.svelte` / `routes/(app)/loop/+page.svelte`
両方で intent → trigger の対応表 (`SAME_STAGE_INTENT_TRIGGER`,
`WORKSPACE_SAME_STAGE_INTENT_TRIGGER`) を導入し、Revisit は
`onTransition(entryKey, effectiveStage, "intent_signal", metadata)` で
同一ステージ発火に切り替える。

**却下した代替案**: canonical JSON に `decide→orient` / `act→orient`
の backward edge を追加する案 (Revisit を cross-stage のままにする) は、
ADR-914 が positive enumeration で同一ステージに寄せた設計意図と矛盾し、
backend classifier 側にも波及する。1 ファイルで閉じる trigger-aware 案を採用。

### Pillar 2: Projector `act_targets` の article を不変にする

`augur.conversation_linked.v1` event を projector が処理すると
`seedActTargets(ev, inputs)` は `ev.Payload` から `article_id` を取れず
article target を返さなくなる。これを単純に「article: 接頭辞があれば
そのまま」と書き換えると非 article entry を壊す可能性がある。

`SurfaceScoreInputs` に `ArticleID string` を追加し、`EventLogSurfaceScoreResolver`
が同一 `entry_key` の前段 `SummaryVersionCreated` から `article_id` を
pin する。`seedActTargets` は event 由来の `articleActTargetID(ev)` を
優先し、空ならば `inputs.ArticleID` にフォールバック。`marshalSurfaceScoreInputs`
/ `parseSurfaceScoreInputs` にも `article_id` を載せ、`SurfacePlanRecomputed`
event 再生時にも pin が残るようにする。

加えて防御-in-depth として、FE `LoopEntryTile.svelte` の `sourceUrl()`
に `articleFromEntryKey(entry.entryKey)` フォールバック (UUID 形式の
tail のみ受理) を追加。projector が将来再退行しても Open Article が
silent dead にならない。

**却下した代替案**: `articleActTargetID` に `entry:` プレフィックス剥がしを
直接足す案 (1 行で済む) は非 article entry (recap-only / cluster-only)
を破壊するリスクが残るため、resolver-pinned ArticleID を採用。
reproject-safety は維持される (`inputs.ArticleID` は event 集約から導出
されるピュアな値)。

### Pillar 3: `surface_planner_cron` に空 tick heartbeat を入れる

`RunBatch` のループは `if len(events) == 0 { break }` で抜け、
`UpdateProjectionCheckpoint` を呼ばずに帰っていた。空 tick でも
現在の `lastSeq` で `UpdateProjectionCheckpoint` を呼ぶことで `updated_at`
を bump し、`now() - updated_at` の SLO クエリが projector liveness を
正しく反映するようにする (`checkpointBumped` フラグで「ループ内で
すでに bump 済みなら二重 update しない」分岐)。

`docs/runbooks/knowledge-loop-reproject.md` の post-check セクションに
「reproject 後 5 分以内に全 projector lag が tick 間隔の 2 倍以内に収束
する」観測手順 (3-projector × `lag_seconds` の SQL) を追記。

**却下した代替案**: SLO 監視側で「event arrival rate が低いと lag は
増えるが OK」ロジックを足す案は監視を複雑化させる。projector 側の
heartbeat で row の `updated_at` を意味的に「最後に cron が触った時刻」
に揃える方が単純で誤検知を作らない。

### Pillar 4: `GetFeedID` バーストの per-batch coalesce + 正規化 fallback

`pre-processor.driver/backend_api.ArticleRepository.UpsertArticles`
に `feedIDCache map[string]string` (空文字 = NotFound 確定) を追加し、
同一 FeedURL の RPC が batch 内で 1 回に抑え込まれるようにする。
warn ログも feed-URL 単位で 1 行に集約 (`"feed not found, skipping
articles for feed"` + `first_url`)。

並行して alt-backend に `utils/feed_url_normalize` パッケージ
(scheme/host lower-case、trailing slash 剥がし、`.` host 剥がし、
default port 剥がし。idempotent な pure 関数) を新設。`FeedURLToIDGateway.GetFeedIDByURL`
は (a) literal lookup → (b) 失敗時に normalized で再 lookup の二段構えに変更。
literal を先に試すことで、登録時に非正規化形で保存された既存行への
互換性を壊さない。

**却下した代替案**: SQL レベルで `WHERE lower(rtrim(fl.url, '/')) = ...`
を書く案は `feed_links.url` の btree index を効かなくして 50ms+ 化する
ため、アプリ層の二段 lookup を選んだ。register 側の正規化は次の PR に
分離 (本 ADR の対象外、操作は migration が必要)。

### Pillar 5: `time.Now()` を `event.OccurredAt` に置換し replay 不変を assert

`alt-backend.job/knowledge_projector.go` の以下 3 ハンドラの
`now := time.Now()` を `now := event.OccurredAt` に置換:

- `projectArticleCreated`: `FreshnessAt`, `GeneratedAt`, `UpdatedAt`,
  freshness score (`time.Since(*publishedAt)` → `now.Sub(*publishedAt)`)
- `projectSummaryVersionCreated`: `GeneratedAt`, `UpdatedAt`, `TodayDigest.DigestDate`/`UpdatedAt`
- `projectTagSetVersionCreated`: `GeneratedAt`, `UpdatedAt`, `TodayDigest.DigestDate`/`UpdatedAt`

加えて `TestKnowledgeProjectorJob_EventLogReplay_IsBitIdentical`
(`ArticleCreated → SummaryVersionCreated → TagSetVersionCreated` の 3
event chain を 2 回 replay して `homeItemsPort.upserted[i]` の
`GeneratedAt` / `UpdatedAt` / `Score` / `SummaryExcerpt` / `WhyReasons`
が完全一致することを確認) を追加。

P0-2 (today digest availability) と P0-3 (recall single-source) は
本ブランチで該当コードを再読し、それぞれ migration 20260322000001 +
`digest_availability_reconcile_job` + `home_query.go` プロパゲーション、
`hooks/useRecallRail.svelte.ts` が `setCandidates(data)` 経由の単一
fetch (ADR-913 §D-9) 経由で実装済みであることを確認。コード変更なし。

**却下した代替案**: projector 全体の `time.Now()` を一括置換する案
(他にも `projectHomeItemOpened` 等で残っている) は scope を膨らませる。
今回は P0-1 / P0-4 が直接指す 3 ハンドラに絞り、残りは別タスクに分離。

## Consequences

### Pros

- **ユーザーが報告した 3 症状が一括で消える**: Revisit はどの stage
  からでもクリックできる (`decision_presented.v1` event が増えることで
  追って評価可能)、Ask 後の Article / Summary が再びクリック可能、
  Knowledge Home の新着停止は ingestion side のバースト潰しで間接的に
  解消。
- **reproject-safe 不変が機械的に守られる**: 新しい
  `TestKnowledgeProjectorJob_EventLogReplay_IsBitIdentical` が
  CI に組み込まれることで、将来誰かが `time.Now()` を business-fact に
  紛れ込ませると RED になる。`IMPL_BASE` の核心不変条件が test で
  pin される。
- **SLO 監視の S/N が上がる**: `surface_planner_v2 lag` が 142s で
  鳴り続ける誤検知が消える。runbook の post-check が "tick 間隔の 2
  倍以内" という具体的閾値で書かれるため on-call の判断が速くなる。
- **GetFeedID の RPC コストが分かりやすく落ちる**: 同一 batch の
  N 件 → 1 件、warn ログも 1 行。`otel_http_requests` の p50/p95/p99
  に直接効く (測定は post-deploy に runbook 経由で行う)。
- **canonical contract への忠実度が上がる**: FE `loop-transitions.ts`
  が `loop_transition_policy.json` の `same_stage_triggers` を初めて
  honor。BE classifier との二重実装ドリフトが減る。

### Cons / Tradeoffs

- **`event.OccurredAt` 移行は他ハンドラに伝播していない**: `projectHomeItemOpened`
  以降は別 ADR で同パターンを適用する必要がある (`grep -n "time.Now()"
  knowledge_projector.go` で 9 箇所残)。本 ADR は scope を最小化したため、
  残箇所が混ざった replay は依然 bit-identical を保証しない。
- **`feed_url_normalize` の register 側適用が未完**: 既存の非正規化
  行に対しては今回追加した fallback でカバーするが、新規登録時の
  正規化 (および既存行の一括正規化マイグレーション) は別 PR が必要。
  バリアントが増えれば再びミスマッチが滲み出る。
- **FE の defense-in-depth `articleFromEntryKey` は UUID 形式に限定**:
  `entry:<UUID>` 以外の entry_key 形式 (将来 recap-only entry 等で
  `entry:recap-<id>` が出る可能性) は recover できない。projector
  正常時は使われないため運用影響は限定的だが、将来の non-article entry
  サポート時に再評価する。
- **`useRecallRail` の `setCandidates` 結線は外注済み**: 単一 fetch は
  ADR-913 で landing 済み、本 ADR では検証のみ。残った提案は本 ADR で
  記録しない (重複)。
- **`projectSummaryVersionCreated` で `summaryVersionPort.GetSummaryVersionByID`
  経由のリードが残る**: これは event payload の `summary_version_id` を
  キーにした versioned artifact lookup なので reproject-safe (`summary_versions`
  は append-only)。ただし `summary_version_id` が無効な UUID の場合
  `return fmt.Errorf("parse summary_version_id: %w", err)` で event 1 件
  分を失敗扱いにする現状の挙動は本 ADR でも維持される。
- **手動本番反映が前提**: 本 ADR のコード変更は 4 サービス (alt-backend,
  alt-frontend-sv, knowledge-sovereign, pre-processor) に跨るため、
  Pact CDC ゲート → `c2quay deploy --env production` の手順を踏む。
  reproject トリガにはならない (`WhyMappingVersion` 据え置き、schema
  変更なし) が、`event.OccurredAt` への切替で `knowledge_home_items.generated_at`
  が "イベント時刻" の意味に変わるため、UI 上の "fresh" 表示の見え方が
  微妙に変わる可能性がある (実装後、staging で観測)。
