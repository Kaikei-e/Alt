# ADR-000885: Knowledge Loop の受動 dwell trigger を撤去し、Open CTA を BFF lookup recovery で再設計する

## Status

Accepted

## Date

2026-05-04

## Affected Services

- **alt-frontend-sv** — `src/lib/actions/observe-tiles.{ts,test.ts}` と `src/lib/hooks/loop-observe-throttle.{ts,test.ts}` を削除。`useKnowledgeLoop.svelte.ts` から `observe()` メソッドと `observeThrottleStorage` option を撤去。`LoopEntryTile.svelte` の tap-to-expand に明示的 `(observe → orient, user_tap)` 発火を結合（`optimistic: true`）。`src/lib/utils/loop-source-url.ts` に `resolveLoopSourceUrlAsync(entry, fetcher)` を追加。`src/routes/(app)/loop/article-source-url/+server.ts` を新設し JWT scoped で BFF proxy。`+page.svelte` の Open CTA を enabled 維持 + secondary label `Open · resolve url` + `loop-open-resolve-error` inline error UI に置換。
- **alt-backend** — `usecase/knowledge_loop_usecase/validator.go` の `ValidateDwellTriggerTarget` を全 to_stage で `ErrInvalidArgument` に降格。`classify_transition_event.go` も先頭で同 trigger を拒否（直接呼び出し系の defense-in-depth）。`usecase/get_article_source_url_usecase/usecase.go` 新設、既存 `ArticleURLLookupPort` を流用したテナント scoped 読み取りに限定。`connect/v2/articles/handler.go` に `GetArticleSourceURL` を実装し、`di/article_module.go` / `di/container.go` / `connect/v2/server.go` で wiring。`usecase/track_home_action_usecase/usecase.go` の `LookupArticleURL` を 3 回 retry（100ms backoff）に強化、最終失敗でも append-empty で続行。
- **proto/alt/articles/v2/articles.proto** — `service ArticleService` に `rpc GetArticleSourceURL(GetArticleSourceURLRequest) returns (GetArticleSourceURLResponse)` を追加。`articleId` のみを request に持たせ、tenant は JWT 由来とすることを docstring で明記。

## Context

[[000879]] で導入した `actTargets[].source_url` 経路は、producer（`alt-backend/app/usecase/track_home_action_usecase/usecase.go`）が `articles.url` の lookup に成功したときだけ payload に注入される設計だった。lookup 失敗時は warn-log のみで `source_url` 空のまま append されるため、

- ローカルで一度 lookup miss した legacy projection 行
- producer の DB 接続が瞬断したタイミングで生成された行

の双方が、対応する `ArticleUrlBackfilled` corrective event がまだ流れていない区間で `act_targets[0].source_url` を持たないままになる。`/loop` の Active Workspace は `resolveLoopSourceUrl(entry)` の結果で Open ボタンを `disabled` にしていたため、ユーザは「Open を押せるはずの記事が押せない」状態を見続けることになった。直近の障害報告（スクリーンショット添付）はまさにこのパスで、production の Open CTA が複数 entry にまたがり長期間 disabled のまま放置されていた。Nielsen Norman Group の disabled-button 系記事は一貫して「disabled は理由を隠し、行動を奪う」anti-pattern として扱っており、Linear など serious command-center UI も「アクション不可」を recovery 動作付きの primary action に置き換えている。

並行してユーザから「OODA loop が UI 側で勝手に回る」体験劣化の報告があった。`src/lib/actions/observe-tiles.ts` の IntersectionObserver action は、tile が viewport の 50% 以上に入ると `loop.observe(entryKey)` を呼び、これが `(from_stage=observe, to_stage=orient, trigger=dwell)` を `/loop/transition` に post する。`alt-backend/app/usecase/knowledge_loop_usecase/classify_transition_event.go` では `(observe → orient, dwell)` を `EventKnowledgeLoopObserved` として受け入れていたため、`projector` 側で entry の session-state が前進し、UI 上の stripe / kicker / CTA が変化する。60s throttle はあるものの、ユーザがスクロールしただけで proposedStage が前進する動作は [[000831]] §1.3 が定めた **`proposedStage` / `currentSessionStage` / `surfaceBucket` を独立軸として扱う** 不変条件と矛盾する。Boyd の OODA Loop は Orientation を意識的に通過する段階として定義しており（"OODA Loop", The Decision Lab／Wikipedia）、Linear の Triage は read-state と workflow-status を分離して開発しているという changelog がある。受動の dwell が cross-stage transition を発火する設計は、これら command-center UI の規範いずれからも棄却される。

[[000873]] / [[000866]] / [[000831]] で fb.md に列挙された他の論点（dismiss_state の split、stream `inline_entry`、transition payload に intent を含める、ordering、Plane Stack 常時表示、popup-blocker risk）はすでに解決済みである。残った 2 点だけが Knowledge Loop の体験を直接壊しており、本 ADR はこの 2 点に絞った再設計の決定を記録する。

## Decision

以下を採用する。

### 1. 受動 dwell trigger を全層で撤去し、stage 進行は明示的 user gesture に閉じる

`alt-frontend-sv` の `src/lib/actions/observe-tiles.ts` と `src/lib/hooks/loop-observe-throttle.ts` を削除する。`useKnowledgeLoop.svelte.ts` から `observe()` メソッドと `observeThrottleStorage` option を撤去し、`+page.svelte` の `use:observeTiles` directive と `onObserve` callback を消す。passive viewing は projector へ何の event も送らず、event log にも痕跡を残さない。Boyd の Observe ステップは「視覚的接触」ではなく「意識的観測」であり、Linear の `read state` 相当を別軸で扱いたくなったときは `KnowledgeLoopRead` 的な専用 event を将来追加する余地を残す（今回はそこまでの必要は見えていない）。

cross-stage への進行は `LoopEntryTile.svelte` の tap-to-expand 動作に統合する。`toggleExpanded` は collapsed → expanded への遷移時に `effectiveStage === "observe"` かつ `isAllowed("orient")` のときだけ `onTransition(entry.entryKey, "orient", "user_tap", undefined, { optimistic: true })` を fire-and-forget で呼ぶ。`useKnowledgeLoop.svelte.ts` の `transitionTo` は新たに `optimistic` option を取り、true のときは `applyLocalStage` を `await post()` の前に走らせ、サーバが拒否（`stale` / `forbidden` / `error` / `rate_limited`）したら `previousStage` に revert する。default は false なので workspace pipeline の `advanceEntry` / `onPipelineStageSelect` などはこれまで通り「accepted を確認してから flip」する保守的な挙動を保つ。

backend 側は `alt-backend/app/usecase/knowledge_loop_usecase/validator.go` の `ValidateDwellTriggerTarget` を「`TRANSITION_TRIGGER_DWELL` であれば to_stage に関わらず `ErrInvalidArgument` を返す」に書き換える。`classify_transition_event.go` も最先頭で同 trigger を拒否し、（既存の defer / review-lane 短絡の後に）直接呼び出す unit test や将来の consumer がうっかり受け入れないようにする defense-in-depth を維持する。Hurl の `e2e/hurl/alt-backend/29-loop-transition-dwell-rejected.hurl` は `(observe, orient, dwell)` / `(observe, observe, dwell)` / `(orient, observe, dwell)` を 400、`(observe, orient, user_tap)` を 200 で pin する。

#### 検討した代替案

- **Same-stage dwell ack を残す案**（dwell trigger を `(observe, observe)` に降格して `KnowledgeLoopObserved` として残す）: passive な「見た」を後段の Surface Planner v2 が score input として活用できる余地を残す利点があったが、ユーザ要望は完全廃止であり、Linear が read-state を server-side workflow-status と分離している規範にも整合する。Surface Planner v2 は既に `SummaryVersionCreated` / `RecapTopicSnapshotted` / `AugurConversationLinked` 等の能動 event から十分な signal を得ているため、passive ack を保持するメリットは小さい。
- **Client-only dwell カウンタ**（localStorage で「見た」を貯めるが BE には送らない）: 将来必要になった時点で localStorage 起点で `KnowledgeLoopRead` 等の新 event を専用 BFF route で送ればよい。今 client-only の状態を持たせると、デバイス間で同期されない読み履歴という負債を抱える。撤去の方が clean。

### 2. Open CTA は disabled をやめ、BFF lookup recovery で再設計する

`/loop` の workspace Open ボタンは `disabled={!activeEntrySourceUrl}` をやめ、enabled のまま secondary label `Open · resolve url` を表示する。click 時は `resolveLoopSourceUrlAsync(entry, fetcher)` を呼ぶ。`fetcher` は `+page.svelte` 内で定義した `bffArticleSourceUrlFetcher` で、`/loop/article-source-url?article_id=...` に GET する。レスポンスが 200 なら `safeArticleHref` を通したうえで `onEntryOpen(entry, sourceUrl)` で SPA reader へ遷移し、4xx/5xx なら `loop-open-resolve-error` data-testid の Alt-Paper monospace kicker 行 (`URL UNAVAILABLE — <code>`) を workspace command 列の下に出す。失敗で navigation はしない。

新設 BFF route `src/routes/(app)/loop/article-source-url/+server.ts` は `locals.backendToken` 必須、`article_id` を UUIDv4/v7 regex で 400 弾き、`getArticleSourceURLForUser` 経由で alt-backend Connect-RPC を呼ぶ。Connect エラーは `extractConnectCode` で 400 / 404 / 401 / 502 にマッピング。

alt-backend 側は新 usecase `get_article_source_url_usecase` を作り、`port/article_url_lookup_port` の `LookupArticleURL(ctx, articleID, userID)` を再利用する。malformed UUID は `ErrInvalidArgument`、空文字応答は `ErrNotFound`、driver 失敗は wrap した driver error。Connect handler `connect/v2/articles/handler.go` の `GetArticleSourceURL` がこれらを `CodeInvalidArgument` / `CodeNotFound` / `CodeUnavailable` に翻訳する。本 RPC は **read-side query** であり、event を append しない。reproject-safe / append-first 不変条件を破らない。

producer 側 `track_home_action_usecase/usecase.go` の `LookupArticleURL` 呼び出しは 3 回 retry（100ms backoff、`ctx.Done()` で中断可）に強化する。3 回失敗しても event は append し、`url` キーを payload から省略する。append-first 不変条件は保たれ、欠損 URL は corrective `ArticleUrlBackfilled` で順次 self-heal する（[[000879]] の corrective branch の延長）。

#### 検討した代替案

- **Producer の retry 失敗時に event 自体を reject する案**: immutable-design-guard 監査で「ユーザの観測行為を log から落とす」append-first 違反のリスクを指摘された。append always で URL のみ欠損させる方が、Knowledge Loop の event-log が「ユーザは何をしたか」の完全な記録であり続けるという不変条件と整合する。ArticleUrlBackfilled の自己修復が同じ region で長期 SLO を担保している。
- **Open ボタンを silent fallback で whyPrimary.evidenceRefs[0].refId に常に倒す案**: refId が public HTTPS URL である保証がない（fixture 上は `article-id` の plain UUID 文字列でしかない）。`safeArticleHref` を通せば落ちるが、ユーザに「なぜか動かない」状態を見せ続けるだけになる。recovery action として明示的に lookup を打つ方が NN/G の "explain why" 規範と一致する。
- **dismiss_state の追加 split**（fb.md が当初提案）: [[000873]] が `visibility_state` + `completion_state` で既に解決済み。本 ADR の scope 外。

### 3. 既存 E2E spec を新 contract に適合させる

`tests/e2e/desktop/loop/canonical-contract.spec.ts` の Revisit CTA テストは tile.click だけで `(observe, orient, user_tap)` を pin するシナリオに置換する。`dismiss-persistence.spec.ts` と `loop-dismiss-no-overlap.spec.ts` は tile.click 直後に dismiss を打つ際、optimistic flip と animate:flip の同時実行で Playwright のポインタ click が "not stable" でリトライ枯渇する。`button.cta--dismiss` を class locator で取り直し、`evaluate(el => el.click())` で直接 DOM click を打つことで race を解消する。`Dismiss` ボタンの disabled gate は `inFlight || dismissing` から `dismissing` のみに変更する（dismiss は loop.dismiss が独立した defer post を打つため、transitionTo の inFlight に gating される必要がない）。

## Consequences

### Pros

- ユーザが報告した 2 つの本質的体験劣化（Open 無効化 / 勝手に進む OODA）が消える。tile を tap した瞬間に stage は前進し、tap しなければ event log は静かなまま。
- [[000831]] §1.3 の独立軸不変条件と Boyd OODA の Orientation 規範の双方を満たす。Linear が辿った read-state ≠ workflow-status の分離と整合し、将来の plane-state 拡張のための語彙の余地を残す。
- Open CTA は `actTargets[].source_url` が空でも primary action として「ユーザが行動できる状態」を維持する。NN/G の disabled-button 推奨に整合し、失敗時は inline error で原因を伝える。
- `GetArticleSourceURL` は read-side query なので projector / event log に副作用を残さない。tenant scope は既存 `LookupArticleURL` driver の `WHERE id = $1 AND user_id = $2` を再利用しており、cross-tenant URL disclosure は driver 層で防がれている。
- producer の retry 強化は実 lookup miss を ~3 倍ぶん吸収する。3 回失敗してもイベント自体は記録され、`ArticleUrlBackfilled` corrective branch が後段で URL を埋める。append-first / reproject-safe 双方を維持する。
- IntersectionObserver / localStorage throttle / observeTiles action / 関連テストが消え、frontend の表面積が減った。死んだ dwell コードパスが残らない。

### Cons / Tradeoffs

- 「ユーザが特定の記事を見ただけ」の signal が event log に残らなくなった。将来 Surface Planner や Acolyte が "user has glanced this" を能動的 score input にしたい場合は、専用 event を別 ADR で復元する必要がある。
- tap-to-expand の意味が「展開」だけでなく「stage を進める」を含むことになり、UI の習得コストがわずかに上がる。drift 時のみ表示される `⌃ press / tap to orient` 系 affordance hint は本 ADR の scope では未実装で、Phase 4 polish として将来追加する。
- `Open · resolve url` のラベルは `actTargets[].source_url` が欠損した entry でだけ表示される。表示条件はユーザにとって暗黙的で、内部的には projection の状態を漏らしている。lookup が成功した直後に snapshot が更新されればラベルは普通の `Open` に戻るが、polling/stream 経由の更新までラベルが揺れる可能性がある。受け入れるトレードオフ。
- `dispatchEvent` 経由の DOM click を使う既存 dismiss spec の hack は、Svelte 5 の animate:flip + 同時 Stage 遷移の race を回避する Playwright 側の妥協で、根本解決ではない。将来 LoopEntryTile が optimistic 更新時の transition を `prefers-reduced-motion` 風に短縮できれば、戻せる。
- producer retry の合計遅延（worst case 200ms 程度）は HomeItemOpened action のレスポンス時間に乗る。ユーザ体験には fire-and-forget で隠れているが、`track_home_action_usecase` の goroutine 寿命が伸びることに注意（context が cancel されれば retry は早期離脱する）。
