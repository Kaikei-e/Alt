# ADR-000409: Knowledge Lens を導入しストリームのフィルタリングビューを保存可能にする

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — domain, ports, usecases (5 本), gateways, repositories, handler (5 RPC)
- alt-frontend-sv (SvelteKit) — Connect client, hooks, LensSelector コンポーネント

## Context

Knowledge Home のストリームは全アイテムを一律に表示しており、「特定の観点で知識世界を見る」手段がなかった。ユーザーが繰り返し使うフィルタ条件（特定タグ群、時間範囲、recap 含む/除外など）を保存して切り替えられる仕組みが求められていた。

既存の `/feeds` や `/search` にはフィルタ機能があるが、Knowledge Home 固有の多次元フィルタ（タグ + 時間 + ソースタイプ + ソート）を保存・再利用する仕組みは存在しなかった。

## Decision

**Knowledge Lens（保存可能なビューポイント）を導入し、CRUD + 選択 RPC でストリームのフィルタリングを可能にする。**

### データベース設計

**3 テーブル構成（バージョニング対応）:**

```
knowledge_lenses (1)
  ├── knowledge_lens_versions (N) — 設定変更は新バージョン作成
  └── knowledge_current_lens (1:1 per user) — アクティブな lens の追跡
```

**`knowledge_lenses`:**

| カラム | 型 | 説明 |
|-------|---|------|
| lens_id | UUID PK | レンズ一意キー |
| user_id | UUID | 所有ユーザー |
| tenant_id | UUID | テナント |
| name | TEXT | 表示名 |
| description | TEXT | 説明（任意） |
| archived_at | TIMESTAMPTZ | ソフトデリートタイムスタンプ |

部分インデックス: `(user_id) WHERE archived_at IS NULL`

**`knowledge_lens_versions`:**

| カラム | 型 | 説明 |
|-------|---|------|
| lens_version_id | UUID PK | バージョン一意キー |
| lens_id | UUID FK | 親レンズ |
| query_text | TEXT | テキスト検索クエリ |
| tag_ids_json | JSONB | フィルタ対象タグ ID 群 |
| time_window_json | JSONB | 時間範囲（"7d", "30d" 等） |
| include_recap | BOOLEAN | recap アイテムの包含 |
| include_pulse | BOOLEAN | pulse アイテムの包含 |
| sort_mode | TEXT | relevance / chronological / trending |
| superseded_by | UUID FK | 次バージョンへの参照（自己参照） |

**`knowledge_current_lens`:**

| カラム | 型 | 説明 |
|-------|---|------|
| user_id | UUID PK | ユーザー（1 ユーザー 1 アクティブ lens） |
| lens_id | UUID FK | 選択中のレンズ |
| lens_version_id | UUID FK | 選択中のバージョン |
| selected_at | TIMESTAMPTZ | 選択日時 |

### バージョニング設計の理由

Lens の設定変更を「上書き」ではなく「新バージョン作成」とした理由:

1. **変更履歴の保持** — いつどの設定で見ていたかを追跡可能
2. **Supersede パターンとの整合** — alt 全体の versioned artifacts 不変条件に準拠
3. **元に戻す操作** — 旧バージョンを再選択するだけで設定を巻き戻せる

### Clean Architecture レイヤー

```
Handler (CreateLens, UpdateLens, DeleteLens, ListLenses, SelectLens)
  → Usecase (create_lens, update_lens, archive_lens, list_lenses, select_lens)
    → Port (knowledge_lens_port: 7 インターフェース)
      → Gateway (knowledge_lens_gateway)
        → Driver (knowledge_lens_repository)
```

各 usecase はオーナーシップ検証（`lens.UserID != userID` → エラー）を行い、他ユーザーの lens への操作を防ぐ。

### Proto RPC

| RPC | 説明 |
|-----|------|
| `CreateLens` | lens + 初期 version 作成 |
| `UpdateLens` | 新 version 作成（旧 version は supersede） |
| `DeleteLens` | archived_at 設定（ソフトデリート） |
| `ListLenses` | archived_at IS NULL でフィルタ |
| `SelectLens` | knowledge_current_lens を upsert |

`GetKnowledgeHomeRequest` に `optional string lens_id = 4` を追加。将来の実装で、このパラメータに基づきストリームのサーバーサイドフィルタリングを行う。

### フロントエンド

| コンポーネント | 説明 |
|-------------|------|
| `LensSelector.svelte` | pill 形式のフィルタバー（"All" + 各 lens + "Save view"） |
| `useLens.svelte.ts` | CRUD / select / reset の状態管理 |

Feature flag `enable_lens` で全体を制御。OFF 時は LensSelector 非表示。

## Consequences

### Pros

- **パーソナライズされたビュー** — ユーザーごとの関心領域に合わせたフィルタリング
- **ワンクリック切替** — pill UI で瞬時にビューポイントを切り替えられる
- **バージョン管理** — 設定変更の履歴が保持され、巻き戻しが可能
- **段階的展開** — feature flag で制御、OFF 時は従来のフィルタなし表示

### Cons / Tradeoffs

- **サーバーサイドフィルタリング未実装** — Phase 4 時点では `lens_id` パラメータの受け渡しのみ。実際のフィルタリングロジックは次フェーズで実装
- **3 テーブル構成** — シンプルな key-value 保存と比較してスキーマが複雑だが、バージョニングの恩恵がある
- **lens 数の制限なし** — 現時点ではユーザーごとの lens 数に上限を設けていない。大量作成時のパフォーマンスは将来の課題
