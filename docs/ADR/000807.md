# ADR-000807: SectionPlanner の analysis セクションに post-LLM deterministic fallback を追加して空ボディ回帰を封じる

## Status

Accepted

## Date

2026-04-20

## Affected Services

- **acolyte-orchestrator (usecase)** — `SectionPlannerNode` の post-LLM fallback 分岐を analysis にも拡張
- **acolyte-orchestrator (tests)** — fallback 関連 32 tests

## Context

2026-04-19 再生成されたレポート `eb67863f-f6f0-48a4-bd76-a9f1d9a79054`（イラン情勢 分析レポート 2026-04-19）は v3 まで正常な分析本文を持っていたが、**v4 (15:07 UTC) 以降に analysis セクションの本文が 5 回中 4 回で 0 字化**した:

| version | 時刻 (UTC) | analysis 本文 | analysis 引用 |
|---------|-----------|--------------|---------------|
| v3 | 10:23 | 1,496 字（散文） | 4 |
| v4 | 15:07 | **0 字** | **0** |
| v5 | 15:11 | 1,373 字（散文） | 4 |
| v6 | 15:21 | **0 字** | **0** |
| v7 | 16:00 | **0 字** | **0** |
| v8 | 16:57 | **0 字** | **0** |

`acolyte-orchestrator` のログは以下の順で症状を示していた:

```
event="xml_parse_success" root_tag="plan"       # 全 20/20 成功
event="xml_parse_success" root_tag="facts"      # 全 20/20 成功
event="XML parse failed"  attempt=1 max_attempts=2 error="not well-formed (invalid token): line 16, column 10"
event="XML parse failed"  attempt=2 max_attempts=2 error="not well-formed (invalid token): line 39, column 143"
event="xml_parse_exhausted" root_tag="section_plan" attempts=2
event="No claims for section, producing empty body" section_key="analysis"
```

つまり **SectionPlannerNode が analysis セクション用の `<section_plan>` XML を Gemma 4 26B MoE (AIX) に生成させる途中で壊れる**。一方 `<plan>` / `<facts>` の XML は同じ LLM + 同じ温度で全件成功しているので、モデル全体の問題ではなく section_plan 固有の境界ケース (max_claims=7 × 日本語 supporting_quote の長尺出力) で発火する。

コード (改修前) の `section_planner_node.py` には post-LLM fallback が conclusion / executive_summary にのみ存在し、**analysis を含むそれ以外の role は claim_plans[key] = [] のまま Writer に渡る**設計だった:

```python
# 既存の post-LLM fallback (改修前)
if not claim_dicts and section_role in ("conclusion", "executive_summary"):
    if section_role == "conclusion":
        analysis_claims = _collect_analysis_claims(claim_plans, outline)
        claim_dicts = _deterministic_conclusion_claims(analysis_claims, extracted_facts, topic=topic)
    else:
        claim_dicts = _deterministic_es_claims(claim_plans, extracted_facts, topic=topic)
```

Writer の `_generate_paragraph` 経路は `claim_plans[key]` が空の場合、line 735 で次のログを出して即空ボディ化する:

```
event="No claims for section, producing empty body" section_key="analysis"
```

conclusion / executive_summary には Writer 側でも `_deterministic_fallback_body` が用意されているが analysis は対象外。**SectionPlanner 層で claim_plans を担保しないと Writer 層では救えない**。これが v4-v8 の 4/5 空ボディの直接原因だった。

FactNormalizer は 20 facts を全件成功しており、原材料は揃っている。単に分配するだけで analysis の claim_plans を埋められる状態だった。

## Decision

`section_planner_node.py` に以下 2 点を入れる:

### 1. `_deterministic_analysis_claims` 関数を新設

```python
def _deterministic_analysis_claims(
    extracted_facts: list[dict],
    *,
    max_claims: int = 7,
) -> list[dict]:
    if not extracted_facts:
        return []
    ranked = _rank_facts_for_synthesis(extracted_facts)
    result: list[dict] = []
    for i, fact in enumerate(ranked[:max_claims], 1):
        claim = _fact_to_synthesis_claim(fact, claim_id=f"analysis-fallback-{i}")
        # analysis は evidence 提示が役割、synthesis ではないので factual 固定
        claim["claim_type"] = "factual"
        result.append(claim)
    return result
```

既存の `_rank_facts_for_synthesis` (numeric → source diversity → confidence) と `_fact_to_synthesis_claim` を再利用する。conclusion / executive_summary の synthesis と違い、analysis の役割は「evidence を整理して提示する」ことなので `claim_type` は `factual` に矯正する。

### 2. Post-LLM fallback 分岐を analysis / その他の role にも拡張

```python
if not claim_dicts:
    if section_role == "conclusion":
        analysis_claims = _collect_analysis_claims(claim_plans, outline)
        claim_dicts = _deterministic_conclusion_claims(analysis_claims, extracted_facts, topic=topic)
    elif section_role == "executive_summary":
        claim_dicts = _deterministic_es_claims(claim_plans, extracted_facts, topic=topic)
    else:
        # 新規: analysis や未来の新 role 向けの fallback
        section_evidence_for_fallback = curated_by_section.get(key, [])
        fallback_evidence_ids = {item.get("id", "") for item in section_evidence_for_fallback}
        fallback_facts = _filter_facts_for_section(extracted_facts, fallback_evidence_ids)
        max_claims = section.get("max_claims", 7)
        claim_dicts = _deterministic_analysis_claims(fallback_facts, max_claims=max_claims)
    if claim_dicts:
        logger.info("Post-LLM deterministic fallback", section_key=key, claim_count=len(claim_dicts))
```

fallback 起動時に curated_by_section を再フィルタする理由: 上段の標準パスで定義される `section_facts` はその分岐内のローカル変数で、fallback 分岐からは静的に参照できない (pyrefly 警告)。再計算コストは section_key で O(facts) 程度で無視できる。

**却下した代替**:

- **Writer 側で analysis に `_deterministic_fallback_body` を追加する** — 対症療法にはなるが、SectionPlanner が壊れている事実が Writer ログまで埋もれ原因追跡が困難になる。根本は「SectionPlanner で claim_plans を担保」で閉じる。
- **XML パーサを強化して壊れた section_plan を部分救出する** — `_repair_xml` の拡張は別 ADR ([[000808]] / 将来の robust-xml 系) で扱う。本 ADR は SectionPlanner が `claims=[]` を返したときの契約上の振る舞いを変える軸のみに絞る。
- **LLM リトライ回数を上げる** — `generate_xml_validated` は既に max_attempts=2 を消費しており、追加リトライは latency 悪化に直結する。確率的解決より決定論的解決を選ぶ。
- **analysis の claim_type を synthesis に寄せて conclusion と同じ経路を流す** — semantics が崩れる。analysis は「生の evidence を提示する」、conclusion は「統合する」という役割分担 ([[000777]] / [[000795]]) を守るため、fallback でも claim_type は `factual` に固定する。

## Consequences

### Pros

- **v4-v8 型の回帰が構造的に起きなくなる**。XML parse が失敗しても extracted_facts が存在する限り analysis は非空になる。
- **既存 `_rank_facts_for_synthesis` / `_fact_to_synthesis_claim` を再利用**しているので、ランキング基準 (numeric → diversity → confidence) が conclusion / executive_summary と整合する。
- **RED → GREEN の 7 tests が回帰防止として残る**。fallback が claim_type=factual を守る unit test、unbound variable を踏まない node-level test、既存 `test_planner_fallback_on_parse_failure` の挙動更新で将来の挙動変更をすべて tests が検知する。
- **Writer 側の `_deterministic_fallback_body` との二重防御の 1 段目**。Writer 側の analysis fallback は別 ADR 案件として残しているが、SectionPlanner 段で claim_plans を埋めることで Writer 側に到達する頻度そのものが減る。

### Cons / Tradeoffs

- **fallback 時は LLM の reasoning (分類 / 優先度判断 / 統合) が失われる**。ランキングは決定論のため、微妙なトピック適合度は LLM 成功時より低い可能性がある。ただし「空ボディ」よりは桁違いに良く、monitoring で fallback 起動頻度を見て LLM 側のプロンプト改善を判断する運用に移す。
- **curated_by_section から facts を再フィルタする小さな冗長処理**が fallback 起動時だけ入る。fallback は異常系なのでホットパスではなく許容。
- **Gemma 4 26B MoE の section_plan XML 壊れは根治していない**。XML 堅牢化や num_predict 増は並行して別 ADR で進める ([[000808]]）。本 ADR は「壊れても analysis が空にならない」だけをスコープに閉じる。
