# ADR-000808: SectionPlanner の num_predict を 2048 から 4096 に引き上げて長尺 CJK section_plan の truncation を解消する

## Status

Accepted

## Date

2026-04-20
　
## Affected Services

- **acolyte-orchestrator (usecase)** — `SectionPlannerNode` の LLM 呼び出し budget を +2048 トークン
- **acolyte-orchestrator (tests)** — `num_predict >= 4096` を assert する protective test 1 本

## Context

[[000807]] で SectionPlanner の analysis セクションに post-LLM deterministic fallback を追加し、XML 解析失敗時の空ボディ回帰は構造的に封じた。ただし fallback は LLM の reasoning (分類・優先度判断・統合) を失うので、**まず LLM 呼び出しが成功する確率を底上げする方が質が高い**。

2026-04-19 のレポート `eb67863f-f6f0-48a4-bd76-a9f1d9a79054` でログに残った XML 失敗は次の形だった:

```
event="XML parse failed" attempt=1 max_attempts=2 error="not well-formed (invalid token): line 16, column 10"
event="XML parse failed" attempt=2 max_attempts=2 error="not well-formed (invalid token): line 39, column 143"
event="xml_parse_exhausted" root_tag="section_plan" attempts=2
```

`line 39, column 143` という位置は、max_claims=7 × `<supporting_quote>` + `<numeric_fact>` + `<must_cite>` の子要素がぎっしり詰まった XML の**後半で切れたことを強く示唆する**。同じモデル (AIX = Gemma 4 26B MoE) + 同じ温度 (T=0.0) + 同じ think=false で `<plan>` と `<facts>` の XML は 20/20 成功しているため、モデル全体の構造出力能力ではなく、**section_plan 固有の「長尺 CJK + XML」が 2048 トークンの budget を食い潰す**現象と判定した。

改修前の呼び出しは次のとおり:

```python
result = await generate_xml_validated(
    self._llm,
    prompt,
    ClaimPlannerOutput,
    root_tag="section_plan",
    normalizer=normalize_section_plan_output,
    temperature=0,
    num_predict=2048,   # ← ここが 2048 で閉じタグが出ずに truncation
    fallback=fallback,
    mode=LLMMode.STRUCTURED,
)
```

トークン見積もり (Gemma 4 tokenizer 概算):

- `<section_plan>` ラッパ + `<reasoning>` 中の日本語 reasoning 200-300 字 ≈ 350 tokens
- `<claim>` 1 件 ≈ 180-260 tokens (日本語 claim + 日本語 supporting_quote + 属性子要素)
- max_claims=7 × 220 tokens ≈ 1540 tokens
- 予備 (タグの閉じ・空白) ≈ 100 tokens

合計 1990 tokens ± 300。2048 は中央値すれすれで、長めの supporting_quote が 1-2 本入ると閉じタグ `</section_plan>` が出る前に budget が尽きる。

## Decision

`num_predict` を **2048 → 4096** に引き上げる。他の設定 (temperature=0, mode=STRUCTURED, think の pin 状況) は触らない。

```python
result = await generate_xml_validated(
    self._llm,
    prompt,
    ClaimPlannerOutput,
    root_tag="section_plan",
    normalizer=normalize_section_plan_output,
    temperature=0,
    num_predict=4096,   # 2048 → 4096: max_claims=7 × 日本語 supporting_quote で余裕
    fallback=fallback,
    mode=LLMMode.STRUCTURED,
)
```

protective test として `test_planner_uses_larger_num_predict_so_section_plan_does_not_truncate` を追加し、`llm.kwargs_list[0]["num_predict"] >= 4096` を assert する。将来誰かが「latency を下げたい」と 2048 に戻した瞬間にテストが落ちる。

**却下した代替**:

- **Settings (env) で可変にする** — HyDE の `hyde_num_predict` ([[000795]]) と違い SectionPlanner の budget は「max_claims の outline 側決定」と直結している。運用側で env チューニングする理由が無く、コードで固定する方が変更履歴が追える。env を増やす判断は `max_claims` / `hyde_num_predict` / `paragraph_num_predict_*` の再設計 ADR に回す。
- **max_claims を 7 → 5 に下げて budget を節約する** — outline の設計 ([[000795]] / [[000796]]) で 7 を採用しているため、budget 都合で semantics を下げるのは筋が悪い。
- **prompt を圧縮して出力長を削る** — `<reasoning>` を削ると LLM の claim 品質が低下する。prompt 圧縮の検討は別軸。
- **XML DSL を JSON schema に差し替える** — vLLM guided decoding など選択肢はあるが、本 ADR のスコープ (v4-v8 回帰の即時緩和) を超える。将来の別 ADR で検討。
- **num_predict=8192 以上に一気に振る** — AIX 側 latency が +2s-3s/呼び出し 増え、report 生成全体の tail latency に効く。4096 で十分な根拠を持つため過剰投資は避ける。

## Consequences

### Pros

- **通常サイズの分析セクションは挙動変化なし**、長尺 CJK のみ完了到達率が上がる。`line 39, column 143` 型の truncation は 4096 tokens まで発生しない。
- **protective test が回帰防止**として残る。num_predict を下げる変更は unit test レベルで即座に落ちる。
- **[[000807]] の deterministic fallback と組合せて二重防御**。まず本 ADR で LLM 成功率を上げ、どうしても壊れた場合だけ 000807 の fallback に落ちる。監視では「fallback 起動頻度」がそのまま LLM 側の品質指標になる。

### Cons / Tradeoffs

- **tail latency が +3s-4s 程度増える可能性**がある (AIX = Gemma 4 26B MoE で 2048 追加トークン)。実測平均で 8.4s → 11-12s 程度と推定。分析セクションは 1 回しか呼ばないので report 全体では軽微だが、将来 `rerun_section` で頻度が上がるときは改めて計測する。
- **AIX 側 VRAM の瞬間使用量はほぼ変わらない** (context window 256K の MoE)。ただし複数 report 並列走行時は KV cache が積み上がるので、大量並列シナリオでの measurement は [[000799]] 系の per-service GHCR image tag 切替時に見直す。
- **`max_claims` を将来 9 以上に引き上げたい要件が来ると 4096 でも足りなくなる**。そのときは「prompt 圧縮 + schema 化 + guided decoding」の別 ADR で根治する。
