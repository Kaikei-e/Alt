# ADR-000724: alt-backend の Low 深刻度 5 件を Dockerfile digest pin・dashboard クランプ・SSRF ポート整理で解消する

## Status

Accepted

## Date

2026-04-14

## Affected Services

- alt-backend (Go) — `Dockerfile.backend` のベースイメージ digest pin (L-001)、`rest/dashboard_handlers.go` の `clampLimit` / `clampWindowSeconds` 追加と全 dashboard ハンドラの上限適用 (L-002)、`utils/security/ssrf_validator.go` の `allowedPorts` から非標準ポート削除 (L-004)、関連テストの追従修正

## Context

[[000717]] / [[000718]] / [[000721]] でセキュリティレビュー (`docs/review/alt-backend-security-review-2026-04-14.md`) の Critical 3 / High 5 / Medium 9 を解消し、残るは Low 5 件 (L-001〜L-005) のみとなった。

最初に整理しておくと、Low 5 件のうち **L-003 と L-005 はすでに Medium 修正の副作用として解消済**である:

- **L-003** (`SSRFValidator` のデフォルト `allowedDomains: ["example.com"]` プレースホルダ) は ADR-000721 の M-003 で `nil` に変更済。
- **L-005** (`isWhitelistedPath` の `/stream` / `/sse` substring バイパス) は ADR-000721 の M-005 で完全一致 / プレフィクス一致に厳格化済。

そのため本 ADR の実装スコープは **L-001 / L-002 / L-004 の 3 件**となる。それぞれの背景は以下のとおり。

1. **L-001 (Dockerfile タグ不固定)**: `alt-backend/Dockerfile.backend` の `FROM golang:1.26-alpine` は floating tag で、ビルド時刻ごとに異なる patch / digest が引かれる可能性があった。再現性の観点からも、ADR-000721 で導入した `govulncheck` SARIF と SBOM 生成と整合させるためにも、明示的な patch version + sha256 digest を pin する必要があった。distroless ベース (`gcr.io/distroless/static-debian12:nonroot`) も同じ理由で digest pin が望ましい。

2. **L-002 (dashboard logs の limit 上限なし)**: `rest/dashboard_handlers.go` の `handleGetLogs` は `limit` クエリの既定値 2000、上限なしで運用ログを返していた。ADR-000717 (C-002) で `RequireAuth + RequireAdmin` を付与済とはいえ、admin トークンが万一漏洩した場合に `?limit=999999999&window=2147483647` 等で過大な query を投げて backend を内部疲弊させる経路が残っていた。`window` 秒も同様に上限なしで、ログ DB を長時間スキャンさせる攻撃面があった。

3. **L-004 (SSRF allowedPorts に 8080 / 8443 を含む)**: `NewSSRFValidator()` のデフォルト `allowedPorts` に `8080` と `8443` が入っていた。これらは標準 HTTP/HTTPS 以外で「社内 / 開発サーバ / 管理 UI」用に使われがちなポートで、SSRF 攻撃時に内部の管理機能へ届きやすい。ADR-000721 (M-002) で HTTPS-only モードを足し、(M-003) でプレースホルダ allowedDomains を消したのと同じ流れで、デフォルト許容ポートも標準のみに絞り込みたかった。

## Decision

以下の三点を一括適用する。

1. **L-001 (Dockerfile を patch version + digest で pin)**: `alt-backend/Dockerfile.backend` を以下に書き換える。

   ```dockerfile
   FROM golang:1.26.2-alpine@sha256:c2a1f7b2095d046ae14b286b18413a05bb82c9bca9b25fe7ff5efef0f0826166 AS builder
   ...
   FROM gcr.io/distroless/static-debian12:nonroot@sha256:a9329520abc449e3b14d5bc3a6ffae065bdde0f02667fa10880c49b35c109fd1
   ```

   digest 値は `docker buildx imagetools inspect` で取得した時点の最新を採用。再現性と SBOM の安定性を担保する。digest 更新は ADR-000721 で導入した `govulncheck` 結果 + `scripts/generate-sbom.sh` の差分監視と合わせて定期 PR を回すフローを将来運用化する。

2. **L-002 (dashboard handlers の query パラメータをクランプ)**: `rest/dashboard_handlers.go` に汎用ヘルパを追加する。

   ```go
   const (
       dashboardLimitMax       = int64(1000)
       dashboardWindowDefault  = int64(14400)            // 4 hours
       dashboardWindowMax      = int64(7 * 24 * 60 * 60) // 7 days
   )

   func clampLimit(value, ceiling, def int64) int64 { ... }
   func clampWindowSeconds(value, def, ceiling int64) int64 { ... }
   ```

   全 5 ハンドラ (`handleGetMetrics` / `handleGetOverview` / `handleGetLogs` / `handleGetJobs` / `handleGetRecapJobs`) で `clampLimit(rawLimit, dashboardLimitMax, ...Default)` と `clampWindowSeconds(rawWindow, dashboardWindowDefault, dashboardWindowMax)` を経由させる。`handleGetLogs` の既定値は 2000 → 500 に縮小 (admin が普段見るログ量の現実値)。負値 / 巨大値はすべて既定値 / ceiling に丸まる。

3. **L-004 (SSRF allowedPorts を標準ポートのみに)**: `NewSSRFValidator()` の初期化を以下に変更する。

   ```go
   allowedPorts: map[string]bool{
       "80": true, "443": true,
   },
   ```

   8080 / 8443 は default から外し、必要な呼び出し側だけ後付けで `allowedPorts["8080"] = true` する設計に倒す (現状そのような呼び出しはない)。

### 検討した代替案

- **L-001 代替**: `golang:1.26.2-alpine` だけ (digest なし) で十分とする案。digest pin のメンテナンスコストは下がるが、レジストリ側で同タグが上書きされた場合に検知できないため、SBOM のハッシュ整合性を担保できなくなる。digest pin を採用した。
- **L-002 代替**: クランプを usecase 層に置く案。クランプは「外部入力サニタイズ」の責務で usecase の純粋ビジネスロジックではないため、handler 層に置いた。usecase 内で再度 limit 上限をチェックする多重防御は将来検討。
- **L-004 代替**: 8080 / 8443 を config 経由で opt-in にする案。現状 alt-backend 自身が 8080/8443 を必要とする外部 URL は無く、config の表面積を増やす意義が薄いため、削除のみとした。本当に必要になった時点で `allowedPorts["8080"] = true` を呼び出す位置に書けば良い。

## Implementation Overview

### 1. Dockerfile pin (L-001)

```dockerfile
# alt-backend/Dockerfile.backend
FROM golang:1.26.2-alpine@sha256:c2a1f7b2095d046ae14b286b18413a05bb82c9bca9b25fe7ff5efef0f0826166 AS builder

WORKDIR /app
# ...

FROM gcr.io/distroless/static-debian12:nonroot@sha256:a9329520abc449e3b14d5bc3a6ffae065bdde0f02667fa10880c49b35c109fd1

# 静的 wget をコピー (compose.yaml ヘルスチェック用)
COPY --from=busybox /bin/wget /usr/bin/wget
# ...
USER nonroot:nonroot
ENTRYPOINT ["/server"]
```

digest 値はその時点で最新の patch version (golang は 1.26.2、distroless は最新 nonroot タグ)。コメントで「digest 更新は M-001 govulncheck の結果と合わせて定期 PR で回す」と記載した。

### 2. dashboard クランプ (L-002)

```go
// rest/dashboard_handlers.go
const (
    dashboardMetricsLimitDefault  = int64(500)
    dashboardOverviewLimitDefault = int64(200)
    dashboardLogsLimitDefault     = int64(500)  // was 2000 before L-002
    dashboardJobsLimitDefault     = int64(200)
    dashboardRecapLimitDefault    = int64(200)
    dashboardLimitMax             = int64(1000)
    dashboardWindowDefault        = int64(14400)            // 4 hours
    dashboardWindowMax            = int64(7 * 24 * 60 * 60) // 7 days
)

func clampLimit(value, ceiling, def int64) int64 {
    if value <= 0 {
        return def
    }
    if value > ceiling {
        return ceiling
    }
    return value
}

func clampWindowSeconds(value, def, ceiling int64) int64 {
    if value <= 0 {
        return def
    }
    if value > ceiling {
        return ceiling
    }
    return value
}

// 全ハンドラの query パース箇所:
rawWindow, _ := strconv.ParseInt(c.QueryParam("window"), 10, 64)
windowSeconds := clampWindowSeconds(rawWindow, dashboardWindowDefault, dashboardWindowMax)
rawLimit, _ := strconv.ParseInt(c.QueryParam("limit"), 10, 64)
limit := clampLimit(rawLimit, dashboardLimitMax, dashboardLogsLimitDefault)
```

`rest/dashboard_clamp_test.go` に `clampLimit` の上限・既定値・負値フォールバック、`clampWindowSeconds` の上限・既定値の 4 ケースを追加した。

### 3. SSRF allowedPorts (L-004)

```go
// utils/security/ssrf_validator.go
func NewSSRFValidator() *SSRFValidator {
    return &SSRFValidator{
        allowedDomains: nil,
        // ...
        allowedPorts: map[string]bool{
            "80": true, "443": true,
            // 8080 / 8443 removed (L-004): non-standard ports are common
            // targets for internal admin services, so we no longer allow
            // them implicitly. Callers add them explicitly when needed.
        },
        // ...
    }
}
```

既存テストの `TestSSRFValidator_PortValidation/allowed_port_8080` は「8080 が許可される」前提だったので「8080 が拒否される」アサートに反転。`TestSSRFValidator_ComprehensiveAttackScenarios/DNS_Rebinding_to_localhost` と `TestImageFetchGateway_FetchImage_SSRF_PrivateNetworks/localhost_*` は port 8080 を 80/443 (デフォルト許可) に置き換えて、本来テストしたい DNS rebinding / private IP 検出経路を維持した。

### 4. ランタイム検証 (smoke)

`docker compose -f compose/compose.yaml -p alt up --build -d alt-backend` で再ビルド・起動し `healthy` を確認後、以下を実行した。

- `head -5 alt-backend/Dockerfile.backend` で `FROM golang:1.26.2-alpine@sha256:...` が pin されていることを確認。
- `curl -s http://localhost:9000/v1/health` → 200。
- `curl -s http://localhost:9000/v1/dashboard/logs` (JWT なし) → 401 (RequireAuth + RequireAdmin が引き続き有効)。
- `go test ./... -count=1` (alt-backend) で全モジュール GREEN。

## Consequences

### Pros

- **セキュリティレビュー全件クローズ**: Critical 3 + High 5 + Medium 9 + Low 5 = **22/22** がすべて修正済となった。同レビュー由来の残件はゼロ。
- **再現性のあるイメージビルド**: digest pin により、同じソースから同じバイナリが出る (reproducible build) が担保される。SBOM の差分検知 (M-001) と組み合わさり、依存変更の追跡が確実になる。
- **dashboard のリソース消費が予測可能**: 認証された admin であっても上限を超えた `limit` / `window` を投げられないため、ログ DB / メトリクス DB への過大な query を構造的に防げる。`handleGetLogs` の既定値 2000 → 500 縮小も、誤って admin が大量取得する事故を減らす。
- **SSRF 攻撃面の縮小**: 8080 / 8443 が不許可になり、内部の開発サーバ / 管理 UI に SSRF で到達するパターンが一段塞がれた。private IP / DNS rebinding 防御と組み合わせて多層になる。
- **TDD 維持**: 全ての修正で RED テスト先行 → GREEN を守り、回帰テスト (`go test ./...`) も通過。

### Cons / Tradeoffs

- **Dockerfile digest pin の更新コスト**: 新しい patch version (例: golang 1.26.3) が出た時に digest 更新 PR が必要。`govulncheck` がブロックしたタイミングで CI が落ちて気付けるようにする運用ルールが要る。
- **既存テストの数値固定アサーションの見直し**: SSRF テストで 8080 を期待していたケースを書き直した。同じ想定で書かれた将来のテストは 80/443 を選ぶ規律が必要。
- **dashboard limit 1000 の上限が将来狭すぎる可能性**: 1000 件で済まないユースケース (例: 1 日分の job 一覧) が出てきたら、ページネーション (`cursor` / `offset`) を導入する必要がある。当面はクランプで停めて、必要になった時点で別 ADR で拡張。
- **8080 / 8443 が必要な image proxy / RAG 経路**: 現状そのようなニーズは無いが、もし将来必要になったら呼び出し側で `allowedPorts["8080"] = true` を opt-in する規約を周知する必要がある。

### 将来の検討課題

- Dockerfile digest pin の自動更新 (Renovate / Dependabot 連携)。
- dashboard へのページネーション API 化 (`cursor` / `offset` 導入で limit 上限の制約を解消)。
- `URLSecurityValidator` と `SSRFValidator` の完全統合 (ADR-000721 で flag した将来課題を継続)。
- Permissions-Policy ヘッダのエッジ層配信 (ADR-000721 で flag した将来課題を継続)。
- セキュリティレビューの定期再実施 (新しい OWASP Top 10 改定や依存ライブラリ追加に対して)。
