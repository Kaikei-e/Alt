# APIモードバックフィルに空フィードフィルタを追加

## ADR's STATUS

Accepted

## CONTEXT

バックフィルジョブはLegacy（single-DB）モードとAPI（split-DB）モードの2つで動作する。

- **Legacyモード**: `GetInoreaderArticlesForEmptyFeeds`が`NOT EXISTS`フィルタ付き4テーブルJOINで、記事が1件もないフィード（空フィード）のみを対象に記事を取得する。正しく動作している。
- **APIモード**: `GetInoreaderArticlesForBackfill`が**全inoreader記事を取得**し、feedID解決後にそのまま下流へ投入する。SyncArticlesと同じ記事を重複処理してしまう。

実測では1,841件が「処理」されたが、全て既存記事のUPDATE（`CheckExists`による重複スキップ）で**新規挿入は0件**。空フィード29,000件以上は放置されたまま。

原因: APIモードではinoreaderテーブル（pre-processor-db）とarticlesテーブル（alt-db）が別DBにあるため、SQLレベルの`NOT EXISTS`フィルタが使えない。

## DECISION MAKING

### 方針: `CheckFeedHasArticles` RPCの新設

alt-backendに`CheckFeedHasArticles` RPCを新設し、pre-processorのAPIモードバックフィルで空フィード判定に使う。

### 代替案: バッチエンドポイント

`GetFeedIDsWithoutArticles`のようなバッチRPC（feedID一覧を受け取り、記事のないfeedIDだけを返す）も検討した。バッチあたり1 RPCで済む利点がある。しかし、現行のバックフィルフローはfeedURL→feedID解決と記事取得が交互に進むため、全feedIDが確定するタイミングが存在しない。per-feedチェック方式の方がフローに自然に組み込める。

### fail-open戦略

`CheckFeedHasArticles` RPCが失敗した場合はfail-open（バックフィル対象に含める）とした。下流の`UpsertArticlesWithFeedID`が`CheckExists` + `ON CONFLICT DO NOTHING`で既存記事をスキップするため、安全性は担保されている。このフィルタは正確性ではなく効率のための最適化である。

## RESULTS, EFFECTS

### 実装の概要

#### 1. Proto定義

`proto/services/backend/v1/internal.proto`に以下を追加:

- `CheckFeedHasArticlesRequest` / `CheckFeedHasArticlesResponse`メッセージ
- `BackendInternalService`に`CheckFeedHasArticles` RPC

#### 2. alt-backend側（サーバー）

Clean Architectureに従い、以下のレイヤーを追加:

| レイヤー | ファイル | 内容 |
|---------|---------|------|
| Port | `port/internal_feed_port/port.go` | `CheckFeedHasArticlesPort`インターフェース |
| Driver | `driver/alt_db/check_feed_has_articles_driver.go` | `SELECT EXISTS(SELECT 1 FROM articles WHERE feed_id = $1 AND deleted_at IS NULL)` |
| Gateway | `gateway/internal_article_gateway/gateway.go` | Driver委譲 |
| Handler | `connect/v2/internal/handler.go` | `WithBackfillPorts`オプション + ハンドラメソッド |
| DI | `connect/v2/server.go` | `WithBackfillPorts(gw)`配線 |

SQLクエリは`feed_id`インデックスにより最初の1行で停止するため、サブミリ秒で完了する。

#### 3. pre-processor側（クライアント）

`driver/backend_api/article_driver.go`の`FetchInoreaderArticlesForEmptyFeeds`を修正:

- feedID解決後に`checkFeedHasArticles` RPCで空フィード判定
- feedID単位でキャッシュ（`feedHasArticlesCache map[string]bool`）し、同一feedIDへの重複RPCを防止
- バッチあたり約38ユニークfeed → 38 RPC呼び出し（キャッシュ済み）

#### 4. テスト

- alt-backend: 5テストケース（`_True`, `_False`, `_EmptyFeedID`, `_Error`, `_Unimplemented`）
- pre-processor: 3テストケース（`_True`, `_False`, `_Error`）

### PROS

- **バックフィルの効率が劇的に改善**: 全記事処理（1,841件→全UPDATE）から空フィードのみ処理へ
- **既存フローへの影響が最小限**: フィルタの追加のみで、下流の`UpsertArticlesWithFeedID`は変更なし
- **fail-openで安全**: RPC失敗時もバックフィルは継続し、下流の重複防止が安全弁として機能
- **キャッシュにより負荷は軽微**: バッチあたり約38回のサブミリ秒クエリ

### CONS, TRADEOFF

- **per-feed RPCのオーバーヘッド**: バッチRPCと比較して、ネットワークラウンドトリップが多い。ただしfeedID単位キャッシュにより実用上は問題にならない
- **一時的な不整合**: バックフィル中に他プロセスが記事を挿入した場合、チェック結果が古くなる可能性がある。しかし次回バッチで正しくフィルタされるため実害はない
