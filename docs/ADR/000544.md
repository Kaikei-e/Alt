# ADR-000544: Knowledge Home の link カラム書き込みパスを実装し Open ボタンの記事遷移を復旧する

## Status

Accepted

## Date

2026-03-23

## Affected Services

- alt-backend (Go) — projector、backfill job
- knowledge-sovereign (Go) — UpsertKnowledgeHomeItem、read_projections

## Context

Knowledge Sovereign の完全分離（[[000537]]）により、`knowledge_home_items` の読み取りが alt-db から sovereign-db に移行した。

**分離前**: alt-backend の `GetKnowledgeHomeItems` は `LEFT JOIN articles art ON khi.primary_ref_id = art.id` で記事 URL を取得し、`COALESCE(art.url, '') AS link` としてフロントエンドに返していた。

**分離後**: sovereign の `read_projections.go` は `COALESCE(khi.link, '') AS link` を直接読む設計に変更。`knowledge-sovereign-full-separation.md` の Section 2.3 で以下の方針を定めていた:

> sovereign-db の `knowledge_home_items` テーブルに `link` カラムを追加（migration 00002）。projector が upsert 時に article URL を含めて書き込む。sovereign の read は articles JOIN なしで完結。

しかし、migration 00002 で `link TEXT NOT NULL DEFAULT ''` カラムは追加されたものの、**書き込み側が未実装**のまま分離が完了していた:

1. `articleCreatedPayload` 構造体に URL フィールドがない
2. projector の `projectArticleCreated` が `item.Link` を設定しない
3. sovereign の `UpsertKnowledgeHomeItem` SQL に `link` カラムが含まれていない

結果として `link` カラムは常に空文字のままとなり、フロントエンドの Knowledge Home カードの Open ボタンが `item.link` の truthy チェック（`if (item.link) { goto(...) }`）を通過できず、記事ページへの遷移が無反応になった。

## Decision

**link カラムの書き込みパスを projector → sovereign upsert の全経路で実装し、既存データはワンショット SQL で一括バックフィルする。**

### 1. projector の ArticleCreated イベントに link を追加

`articleCreatedPayload` に `Link string` フィールドを追加し、`projectArticleCreated` で `item.Link = payload.Link` を設定する。これにより reproject-safe 不変条件を維持したまま（イベントペイロードのみ使用）、link を projection に反映できる。

### 2. sovereign の UpsertKnowledgeHomeItem に link カラムを追加

INSERT に `link` カラムを追加し、ON CONFLICT のマージロジックに `link = CASE WHEN EXCLUDED.link != '' THEN EXCLUDED.link ELSE knowledge_home_items.link END` を追加。空文字での上書きを防ぐ。

### 3. backfill event に URL を含める

`KnowledgeBackfillArticle` 構造体に `URL` フィールドを追加、`ListBackfillArticles` の SELECT に `url` カラムを追加、`GenerateBackfillEvent` に `link` パラメータを追加。将来のバックフィル実行時にも link が伝搬される。

### 4. 既存データのワンショット SQL バックフィル

alt-db の `articles` テーブルから 37,668 件の URL をエクスポートし、sovereign-db の `knowledge_home_items` に一括 UPDATE を実行。120,376 行を更新、残り 0 件。

## Consequences

### Pros

- Knowledge Home の Open ボタンが正常に `/articles/{id}` に遷移するようになった
- sovereign-db 内で完結する読み取りパス（articles JOIN 不要）が設計どおり機能する
- reproject-safe 不変条件を維持: projector はイベントペイロードの `link` フィールドのみ使用
- backfill job も URL を含むため、将来の再バックフィルでも link が失われない

### Cons / Tradeoffs

- ArticleCreated イベントの producer 側（mq-hub consumer、outbox 経由）が `link` フィールドを含めない場合、新規記事の link は空のまま projector に到達する。現状の ArticleCreated イベントが URL を含んでいない producer がある場合は個別に修正が必要
- ワンショット SQL は冪等（`WHERE khi.link = ''`）だが、記事 URL が変更された場合の追従は別途考慮が必要
