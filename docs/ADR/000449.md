# ADR-000449: Recall candidates を Knowledge Home の single-fetch 契約に統合する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-frontend-sv (TypeScript/SvelteKit)

## Context

Knowledge Home の backend handler (`GetKnowledgeHome`) は、feature flag が有効な場合に `recall_candidates` を response に埋め込んで返している。proto 定義にも `repeated RecallCandidate recall_candidates = 8` が存在し、connect layer の変換処理も実装済みである。

しかし frontend 側では、`+page.svelte` の `onMount` で `recall.fetchCandidates()` を独立に呼び出し、`GetRecallRail` RPC で別途取得していた。これにより:

1. **二重ラウンドトリップ**: Home の初期表示で 2 つの API 呼び出しが発生
2. **データ不整合リスク**: Home の items と recall candidates が異なるタイミングの snapshot を参照する可能性
3. **single-fetch 原理違反**: Knowledge Home は「1 リクエストで Home 面に必要な全データを返す」設計だが、recall だけが別経路

これは Knowledge Home Phase 1-5 是正指示書で指摘された P0 項目である。

## Decision

**frontend を変更し、`GetKnowledgeHome` response に含まれる `recallCandidates` を使うようにする。独立した `GetRecallRail` 呼び出しを初期ロードから削除する。**

### 具体的な変更

1. `useKnowledgeHome` hook に `recallCandidates` state を追加し、`fetchData` で `result.recallCandidates` を保持・公開
2. `useRecallRail` hook に `setCandidates(data)` メソッドを追加（外部からの初期データ注入ポイント）
3. `+page.svelte` の `onMount` から `recall.fetchCandidates()` を削除
4. Home data の fetch 完了後に `recall.setCandidates(home.recallCandidates)` で注入
5. Lens 切り替え時も同様に再注入

### useRecallRail の fetchCandidates を残す理由

`fetchCandidates` は削除しない。Recall Rail 内でのリフレッシュ操作（snooze/dismiss 後の再取得等）に引き続き使える補助メソッドとして残す。初期ロードのみ Home response からの注入に切り替える。

## Consequences

### Pros

- 初期表示の API 呼び出しが 2 → 1 に削減され、レイテンシが改善
- Home items と recall candidates が同一 snapshot から取得されるため、データ整合性が向上
- single-fetch 契約に準拠し、Knowledge Home の設計原理と一致

### Cons / Tradeoffs

- `useRecallRail` に `setCandidates` という注入メソッドが増え、データフローが「pull（自分で取得）+ push（外部から注入）」の 2 系統になる。ただし初期ロードは push、以降の操作は内部 state 管理という明確な分離で複雑さは限定的
- Backend の `GetRecallRail` RPC は削除しない（他の画面や将来の用途で単独取得が必要になる可能性）
