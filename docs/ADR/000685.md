# ADR-000685: Critic heuristic-first を品質チューニング段階へ進める

## Status

Accepted

## Date

2026-04-10

## Affected Services

- acolyte-orchestrator (Python / LangGraph)

## Context

[[000673]] で Critic に heuristic-first アーキテクチャを導入し、FM4（empty body）・FM5（zero claims）・FM11（ES numeric absence）の 3 つの heuristic チェックを追加した。v5 ではこの方針が当たり、Critic は LLM 呼び出し 0 回で body=0 を即検出し、推論コストを削減できている。

しかし以下の品質課題が残っていた:

1. **`_build_claim_feedbacks()` が粗い** — blocking section 内の rejected/empty paragraph に一律「body empty」としか返さず、Writer が具体的にどう修正すべきか不明だった
2. **FM8/FM10 が section レベルのみ** — Conclusion-Analysis 重複や novelty 違反を section 全体で検出するが、**どの paragraph が重複しているか**を特定できなかった
3. **`MIN_SECTION_LENGTH=100` が全 role 共通** — ES は 80 chars でも十分だが、analysis は 100 chars で通ってしまっていた
4. **Paragraph 単位の citation チェックがない** — `must_cite=True` の claim でも paragraph レベルで citation の有無を検証していなかった
5. **Warning の蓄積メカニズムがない** — 個々の warning は blocking にならないが、同一 section に warning が集中した場合にも見逃していた

## Decision

Critic の heuristic ルールを精緻化し、paragraph/claim 単位のフィードバックを返せるようにする。LLM critic は引き続き補助的な最終手段に留める。

### Per-role minimum section length

`MIN_SECTION_LENGTH=100` のフラットな閾値を、section_role 別のマッピングに置換:

| section_role | min chars | 根拠 |
|---|---|---|
| analysis | 200 | 事実分析は一定の情報密度が必要 |
| conclusion | 100 | synthesis は簡潔でもよいが空ではいけない |
| executive_summary | 80 | 要約は本来簡潔。80 chars あれば最低限の情報を含む |
| general (default) | 100 | 後方互換 |

`detect_incomplete_sections()` が `outline` の `section_role` フィールドを参照して閾値を選択する。

### FM12: Paragraph-level duplication detection

新関数 `detect_paragraph_duplication()` を追加。2 つのスコープで bigram Jaccard overlap を検出:

1. **Intra-section**: 同一 section 内の paragraph 同士。threshold 0.30 超で FM12 blocking
2. **Cross-section**: `novelty_against` 指定がある section の paragraph vs 対象 section の paragraph

section レベルの FM8/FM10 とは独立に動作し、**どの paragraph が重複しているか**を `claim_id` 付きで返す。

### FM13: Paragraph-level missing citation

新関数 `detect_paragraph_missing_citation()` を追加。`claim_plans` の `must_cite=True` と `section_paragraphs` の `citations` リストを照合し、citation なしの paragraph に warning を発行。

### `_build_claim_feedbacks()` の拡充

blocking section 内の paragraph に対し、FM 種別に応じた具体的な reason を返す:

| FM | reason 例 |
|---|---|
| FM4 (empty) | `"body empty — regenerate with claim evidence"` |
| FM12 (duplication) | `"duplicate content — Paragraph 'c-1' duplicates 'a-1' (Jaccard 0.85)"` |
| FM13 (missing citation) | `"missing citation — Include citation reference for claim 'a-1'."` |
| FM3 (short) | section レベルの detection から `failure_modes` に記録 |
| rejected | `"paragraph rejected — regenerate with claim evidence"` |

Writer は `<delta_feedback>` タグ内でこの reason を受け取り、具体的な修正指示として活用できる。

### Warning accumulation → blocking promotion

同一 section に 3 件以上の warning が蓄積した場合、blocking に昇格して revision をトリガーする。`WARNING_ACCUMULATION_THRESHOLD = 3` として定数化。

例: ES に FM11 (numeric absence) + FM13 (missing citation) x2 + FM3 (short) = 4 warnings → blocking promotion → verdict=revise。

### CriticNode への統合

`CriticNode.__call__` の heuristic chain に FM12 と FM13 を追加。既存の「blocking heuristic があれば LLM スキップ」のロジックは維持:

```
FM4 → FM5 → FM11 → FM2 → FM3 → FM8 → FM9 → FM10 → FM12 → FM13 → warning accumulation → blocking check
```

### FailureMode enum 拡張

`domain/critic_taxonomy.py` の `FailureMode` enum に 2 値を追加:

- `FM12_PARAGRAPH_DUPLICATION = "paragraph_duplication"`
- `FM13_PARAGRAPH_MISSING_CITATION = "paragraph_missing_citation"`

### テスト

`tests/unit/test_critic_quality_tuning.py` に 10 件のテストを追加:

| テスト | 検証内容 |
|---|---|
| `test_min_section_length_varies_by_role` | analysis 150 chars → FM3 warning; ES 80 chars → pass |
| `test_paragraph_duplication_within_section` | intra-section Jaccard > 0.30 → FM12 |
| `test_paragraph_duplication_cross_section` | cross-section (novelty_against) → FM12 |
| `test_paragraph_missing_citation_must_cite` | must_cite + no citation → FM13 warning |
| `test_paragraph_missing_citation_not_triggered_when_cited` | citation あり → FM13 なし |
| `test_claim_feedback_includes_specific_reason_for_duplication` | FM12 → reason に overlap 情報 |
| `test_claim_feedback_includes_specific_reason_for_empty` | FM4 → reason に regenerate 指示 |
| `test_claim_feedback_includes_specific_reason_for_short` | FM3 → failure_modes に記録 |
| `test_warning_accumulation_promotes_to_blocking` | warning 3+ → verdict=revise |
| `test_llm_critic_still_skipped_on_blocking_heuristic` | 新 heuristic 追加後も blocking 時は LLM 呼び出しなし |

既存テスト（FM2, FM4, FM5, FM8, FM9, FM10, FM11, parse failure）の regression なし。

## Consequences

### Pros

- Writer が paragraph 単位で具体的な修正指示を受け取れるようになり、revision の質が向上
- Per-role min length により、analysis の 100 chars 通過を防止しつつ ES の過剰検出を回避
- FM12 の paragraph-level duplication 検出により、section 全体のリライトではなく**重複した段落だけ**の再生成が可能
- FM13 により citation の欠落を heuristic だけで検出でき、LLM critic に頼る必要がない
- Warning accumulation により、個々は軽微でも集中すると品質問題になるケースをキャッチ
- LLM critic のスキップ率は維持: heuristic で十分検出できるケースでは引き続き LLM 非呼び出し

### Cons / Tradeoffs

- FM12 の bigram Jaccard は短い段落（< 50 chars）で false positive が発生しやすい。今後の run で threshold 0.30 の適切性を検証する必要がある
- `_build_claim_feedbacks()` の reason 文字列が Writer の `<delta_feedback>` に直接渡されるため、reason の表現が Writer の出力品質に影響する。日本語/英語の混在も要検討
- Warning accumulation threshold 3 は empirical な値であり、section_role ごとの調整が将来必要になる可能性がある
- FM12/FM13 の追加により heuristic chain が長くなったが、全て O(n*m) の bigram 計算で実質的な遅延は無視できる
