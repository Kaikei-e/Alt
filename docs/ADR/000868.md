# ADR-000868: Feed の Website URL を WebsiteURL canonical 名で全層統一し URL-vs-Link 命名違反を CI lint で防衛する

## Status

Accepted

## Date

2026-04-28

## Affected Services

(frontmatter 参照)

## Context

ADR-000867 (PR 1) で Article URL カテゴリを全層 canonical (`URL` / `url`) に統一し、PM-2026-041 真因 (DDD ubiquitous language drift) を構造的に塞いだ。同 PR の `docs/glossary/ubiquitous-language.md` で 3 カテゴリの canonical 名を pin したが、執筆中に **Feed Subscription URL カテゴリの実体を誤って `Feed.Link` と記述** していた。

実際の `domain.Feed` 構造体を直接確認したところ、**2 つの URL field が並存**しており命名意味論が異なる:

| Go field | json/db tag | コメント | 実体 |
|---|---|---|---|
| `URL` | `url` | "Feed URL" | RSS XML feed file の URL (subscription URL、`feed_links.url` から JOIN) |
| `Link` | `link` | "Website URL" | サイト本体の URL (RSS `<channel><link>` 要素の値) |

つまり `Feed.Link` は **subscription URL ではなく website URL** であり、命名意味論上 `Feed.URL` (subscription) と並んで意味を区別する役割を持っていた。glossary の「subscription URL の rename ターゲット」という分類は誤り。

しかし `Link` を維持する選択肢にも問題がある:
- HTML の `<link>` element (stylesheet 等の metadata) と意味重複
- HTML `<a>` hyperlink と汎用語として混同
- ADR-000867 で確立した「Knowledge layer では Article URL を `URL` と呼ぶ」規約と読者視点で衝突 (`Link` という field を見ても Article URL なのか Website URL なのか即座に判別できない)

ユーザの判定 (2026-04-28 セッション中) で**「`Link` は overloaded すぎる、明示 rename で `WebsiteURL` を使う」を選択**。RSS spec の `<channel><link>` element 名は RSS parser 内部 (driver/gateway) の翻訳層で受け止め、domain entity 以降は `WebsiteURL` として扱う設計とする。同時に、Knowledge Home / Knowledge Loop layer に `Link` / `"link"` が再導入されないよう CI lint で構造的に防御する。

## Decision

### D-1. `Feed.Link` → `WebsiteURL` を全層 rename

- **Go domain**: `alt-backend/app/domain/feed.go` の `Link string \`json:"link" db:"link"\`` を `WebsiteURL string \`json:"website_url" db:"website_url"\`` に rename。`URL` (RSS subscription URL) field と並べた struct コメントで 2 つの URL の役割差を明示
- **Go driver model**: `alt-backend/app/driver/models/feed.go` の `Link string \`db:"link"\`` を `WebsiteURL string \`db:"website_url"\`` に rename
- **DB**: Atlas migration `20260428000000_rename_feeds_link_to_website_url.sql` で `ALTER TABLE feeds RENAME COLUMN link TO website_url` (PostgreSQL metadata-only)、関連 7 件の index と unique constraint も `link` を含む名前から `website_url` に rename (`idx_feeds_link` → `idx_feeds_website_url` など)
- **Cascade rename**: handler / gateway / driver / usecase / job / rest layer の全 `Feed.Link` 参照 (約 60 箇所) を `Feed.WebsiteURL` に追従、SQL 文字列の `feeds.link` / `WHERE link =` / `INSERT INTO feeds (..., link, ...)` を `website_url` に書き換え
- **Test fixtures**: 関連 test 約 12 ファイルの mock SQL expectation と struct literal を新 column / field 名に更新

### D-2. RSS spec preserve 境界の明文化

driver/gateway 層で gofeed (3rd-party RSS parser) が返す `*gofeed.Feed.Link` を読む際は **そのまま `Link` を使う** (例: `convertGofeedToDomain` 内)。**driver→domain の境界 (gateway 出口) で `Link` → `WebsiteURL` に rename** して以降は `WebsiteURL` で統一。glossary の境界規約セクションを追記。

`domain.RSSFeed.Link`、`domain.RSSItem.Link`、`domain.FeedItem.Link`、`domain.TagTrail.Link` は **RSS Item Link カテゴリ** (item-level) として `Link` 維持。

### D-3. CI 命名違反 lint の追加

`scripts/lint/check-link-vs-url-naming.sh` を新設。Knowledge Home / Knowledge Loop layer の以下 file pattern を scan:
- `alt-backend/app/domain/knowledge_*.go`
- `alt-backend/app/connect/v2/knowledge_*/...`
- `alt-backend/app/usecase/knowledge_*/...`
- `alt-backend/app/job/knowledge_*.go`
- `alt-frontend-sv/src/lib/components/knowledge-home/...`
- `alt-frontend-sv/src/lib/components/knowledge-loop/...`
- `alt-frontend-sv/src/lib/connect/knowledge_*`

5 つの正規表現で `Link string` field 宣言、`json:"link"` tag、`db:"link"` tag、TS `link?: string` interface field、TS `"link":` literal type を検出。RSS-spec preserve が legitimately 必要な箇所は per-line `// allow:link-rss-spec` コメントで opt-out 可能。

### D-4. ADR-000867 の glossary を訂正

PR 1 の glossary に書いた誤分類 (Feed Subscription URL = `Feed.Link`) を撤回:
- 旧分類「Feed Subscription URL」→ 新分類「**Website URL**」 (実体: `Feed.WebsiteURL` / `feeds.website_url`)
- 新分類「**RSS Subscription URL**」 (実体: `domain.Feed.URL` / `feed_links.url` の JOIN populate) を追加

### 検討した代替案

- **`Feed.Link` を維持** (RSS spec の `<channel><link>` element 名を引き継ぐ argument) — 却下。RSS spec 側の`<link>`命名は parser 境界で扱えば十分で、domain layer 全体に影響を与える必要はない。HTML `<link>` / hyperlink との overload と Article URL カテゴリ (`URL`) との読者視点の混同コストが大きい
- **`Feed.SiteURL` などの別命名を採用** — 却下。`WebsiteURL` のほうが 1 単語名詞「Website」+ 役割 suffix「URL」で明確。Site は曖昧 (web site / database site / build site)
- **DB column rename を据え置き、Go field のみ rename** — 却下。1 PR で全層揃えないと「Go では URL、DB では link」の drift が再発する
- **lint script を golangci-lint custom analyzer として実装** — 却下 (本 PR scope)。bash + grep で十分、後で golangci-lint に移植可能。golangci-lint 側だと TS/Svelte file を scan できず Knowledge Home FE 側がカバー対象から漏れる
- **lint scope を全 directory に広げる** — 却下。RSS-spec preserve の `tag_trail.go` / `rss_feed.go` / `feeds/client.ts` / `articles.ts` 等を flag してしまう。Knowledge Home / Knowledge Loop layer に絞って false positive を抑制

## Consequences

### Pros

- **DDD ubiquitous language の二重化を解消**: `Feed.URL` (RSS subscription) と `Feed.WebsiteURL` (website URL) の役割が field 名から明確に読める。`Link` という overloaded 名詞がドメイン層から消える
- **Knowledge Home / Knowledge Loop の Article URL canonical (`URL`) と読者視点で衝突しない**: 同じ Knowledge layer 内で `URL` (article) と `Link` (?) が混在する状況が消える
- **CI lint で構造的防衛**: `scripts/lint/check-link-vs-url-naming.sh` が今後の新規コードに対し Knowledge layer の `Link` 復活を検出。PM-2026-041 / ADR-000865 / ADR-000867 で扱った drift class を recurrence-proof に近づける
- **Atlas migration は metadata-only**: PostgreSQL `RENAME COLUMN` は row rewrite なし、index 自動更新あり。production への影響は数 ms 程度
- **RSS spec との互換性も維持**: gofeed parser から domain への翻訳境界に `Link` → `WebsiteURL` rename を入れることで spec compliance と domain 命名の両立

### Cons / Tradeoffs

- **wide blast radius**: 約 60 references を全層 rename、test fixtures 12 ファイル修正。レビュー負荷が高い (semantic commit 単位での分割で軽減)
- **migration → service start の順序依存**: column rename は metadata-only なので旧 binary が新 column 名を読めなくなるが、新 binary は旧 column 名を参照しないので migration → service start を厳守 (alt-deploy playbook 既存)
- **lint script は bash + grep ベース**: golangci-lint custom analyzer ほど統合度が高くない。CI workflow 側で明示的に invoke する必要あり (本 PR では script のみ追加、CI workflow への組み込みは別 PR で `.github/workflows/` に追加予定)
- **RSS Subscription URL の `Feed.URL` field は本 PR で触らない**: `Feed.URL` は別 table (`feed_links.url`) からの JOIN populate であり、本 PR の `feeds.link` rename とは独立。命名は既に `URL` で canonical
- **glossary の旧記述に依存していた読者がいる場合の混乱**: ADR-000867 の glossary で「Feed Subscription URL = `Feed.Link`」と読んだ人は本 ADR で訂正されたことに気づく必要あり。glossary の "マイグレーション ロードマップ" セクションを更新して新 status を記録
- **lint script は false positive を出す可能性**: 新規追加された Knowledge layer file が `Link` を含む RSS-related 機能を持つ場合は `// allow:link-rss-spec` opt-out が必要。glossary に opt-out 規約を明文化済
