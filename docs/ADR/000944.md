# ADR-000944: Knowledge Home projector を knowledge-sovereign へ re-own する

## Status

Accepted

## Date

2026-07-15

## Affected Services

- alt-backend (Go)
- knowledge-sovereign (Go)

## Context

`docs/review/architecturereview20260713.md` の F-01 finding が、Knowledge Home / recall projector の split-brain を指摘した。knowledge_events・全 read model・checkpoint は既に knowledge-sovereign-db に一元化済みだったが、同一ログを fold するロジック (`alt-backend/app/job/knowledge_projector.go`、9 種のイベント型・913 行) だけが alt-backend に残留し、投影が sovereign への RPC 往復に律速されていた。

[[000845]] で Knowledge Loop projector (現 Knowledge Trail の前身) は既に同型の re-own を完了しており、`main.go` のコメントもその完了を宣言していたが、これは Home 側の未完遂を覆い隠していた。Home と Trail/Loop は別系統 (Home は Loop 以前から独立して存在し、Loop→Trail の置換とは無関係に現役継続) であり、この混同自体が調査を要した。

加えて、旧実装には既知の非対称バグがあった: `projectSummaryVersionCreated` は `GetSummaryVersionByID` 失敗時に hard fail (checkpoint を進めずリトライ) するのに対し、`projectTagSetVersionCreated` は `GetTagSetVersionByID` 失敗時に `ErrorContext` ログのみで空タグのまま処理を続行していた (`knowledge_projector.go:614-616`)。checkpoint が進むためイベントロストになる Rule 8 違反。

knowledge-sovereign は no-shared-DB 原則 (`knowledge-sovereign/app/CLAUDE.md`) により alt-db を直接読めない。旧 fold ロジックは `SummaryVersionGateway`/`TagSetVersionGateway` 経由で alt-db から本文・タグを再取得していたため、単純な re-own では sovereign 側が同じ alt-db 参照を必要としてしまい、no-shared-DB を破る。

## Decision

1. **event payload を拡張する** (CDC RED 先行): `SummaryVersionCreated` イベントの payload に `summary_text` と `quality_score` を追加した。調査の結果、`TagSetVersionCreated` は既に `tags`/`generator` を payload に含んでいたが、旧 alt-backend 側 projector がそれを使わず alt-db に再クエリしていただけと判明したため、拡張不要だった。
2. **knowledge-sovereign に `knowledge_home_projector` usecase を新設**する。[[000845]] の Trail projector パターン (`NewProjector` → `time.NewTicker` → `RunBatch` の in-process ループ、narrow Repository interface) をそのまま踏襲した。fold は event payload のみから導出し (`foldSummaryVersionCreated` は `payload.SummaryText` を直接 excerpt に使う、`foldTagSetVersionCreated` は `payload.Tags` を直接使う)、alt-db への再クエリを一切行わない。これにより旧来の tag lookup silent swallow バグは、そのコードパス自体が存在しないため再発しない。
3. **alt-backend から旧実装を削除**する: `job/knowledge_projector.go` を含む 15 ファイル (実装 7 + テスト 8)、`job/registry.go` の 5 ジョブ登録 (knowledge-backfill / summary-narrative-backfill / knowledge-reproject / recall-projector / digest-availability-reconcile)、`main.go` の起動コードを削除した。`GetKnowledgeHomeUsecase` (5-way 集約の read-composition, 239 行) と event append 経路 (`AppendKnowledgeEventPort` 経由) は現役機能として残置した。
4. **代替案として検討した「payload 拡張前の旧イベントを sovereign 側で alt-db にフォールバック読みする」案は採用しなかった** — no-shared-DB 原則と根本的に矛盾し、実装不可能だった。

## Consequences

### Pros

- Knowledge Home の fold ロジックが単一サービス (knowledge-sovereign) に集約され、投影が RPC 往復に律速されなくなった。
- tag lookup silent swallow バグがコードごと消滅し、summary 側と対称的な reproject-safe fold になった。
- alt-backend から 913 行 + 関連ジョブ・テスト計 5719 行 (net) が削除され、責務が read-composition + event append に絞られた。

### Cons / Tradeoffs

- **payload 拡張前 (本 PR 以前) に発行された `SummaryVersionCreated` イベントは `summary_text` を持たないため、reproject すると当該アイテムの `summary_state` が `pending` に戻る** (`payload.SummaryText == ""` の場合、`foldSummaryVersionCreated` は `whyNewUnread` のみを付与し `ready` にしない)。alt-db フォールバックは no-shared-DB 制約により実装しておらず、対象記事に新しい `SummaryVersionCreated`/再要約イベントが発行されるまで解消しない。恒久対応 (バックフィルイベントの発行、または旧イベントの payload 補完) は本 ADR のスコープ外とし、reproject 頻度が低い運用であれば実害は限定的と判断した。
- `checkpoint` は `knowledge_projection_checkpoints` テーブルに `knowledge-home-projector` という新しい `projector_name` で相乗りしており、テーブル自体の追加マイグレーションは発生していない。
