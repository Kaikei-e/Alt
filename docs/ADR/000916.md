# ADR-000916: tag prefix search の ILIKE seq scan を lower(tag_name) text_pattern_ops の functional B-tree で解消する

## Status

Accepted

## Date

2026-05-24

## Affected Services

- alt-backend (Go) — `app/driver/alt_db/search_tags_driver.go:20` の WHERE 句を `ft.tag_name ILIKE $1 || '%'` から `lower(ft.tag_name) LIKE lower($1) || '%'` に書き換え
- alt-db (PostgreSQL 17) — `feed_tags(lower(tag_name) text_pattern_ops)` の functional B-tree (`idx_feed_tags_tag_name_lower`) を追加。Atlas migration 1 本

## Context

[[000915]] のグローバル検索 cold-start 修正を deploy した直後の 3 件本番計測で、`global_search.section.tags` span が p50/p95 共に **76-84 ms** で残っていた。article section (warm hybrid) が同じ 250-400 ms 帯で並列実行するため user-visible にはまだ隠れているが、将来 article 側がさらに下がると tag が単独で表面化する。

`alt-backend/app/driver/alt_db/search_tags_driver.go` の SQL を `EXPLAIN (ANALYZE, BUFFERS)` で実測:

```
Parallel Seq Scan on feed_tags ft  (rows=240,520, filter discards 224,664)
   Filter: tag_name ~~* 'A%'
Parallel Seq Scan on article_tags  (rows=912,462)
Hash Join + GroupAggregate + top-N heapsort
Execution Time: 180.740 ms
```

既存 `idx_feed_tags_tag_name` (素の B-tree) は ILIKE prefix では使えない。PostgreSQL 17 公式 `indexes-types` の制約に「`LIKE` は left-anchored constant のみ btree 直行可、`ILIKE` は先頭が case 変換の影響を受けない文字のみ」と明記されている。英字 prefix は対象外で seq scan に落ちる。

Web リサーチ (PG 16/17 公式 docs + Crunchy Data fuzzy-name-matching blog + Cybertec の Albe Laurenz による調査) で得た結論:

- **`lower(col) text_pattern_ops` の functional B-tree + クエリ書き換え** が公式推奨パターン。Crunchy Data の事例 (`names` テーブルへの surname 検索) で同形式が掲載
- Paul Ramsey の単純 SELECT prefix scan ベンチで 34ms → 0.117ms (~290x) の効果
- `text_pattern_ops` は collation を無視した char-by-char 比較で LIKE prefix の range scan を可能にする (PG 17 `indexes-opclass`)

## Decision

`feed_tags` に `idx_feed_tags_tag_name_lower btree (lower(tag_name) text_pattern_ops)` を追加し、`search_tags_driver.go` の WHERE 句を `lower(ft.tag_name) LIKE lower($1) || '%'` に書き換える。

### Migration (atlas)

```sql
-- migrations-atlas/migrations/20260524000000_add_feed_tags_lower_tag_name_idx.sql
CREATE INDEX IF NOT EXISTS idx_feed_tags_tag_name_lower
    ON feed_tags (lower(tag_name) text_pattern_ops);
ANALYZE feed_tags;
```

`CREATE INDEX CONCURRENTLY` は採用しない。`migrations-atlas/migrations/atlas.hcl:33-37` が `concurrent_index { create = false }` で統一されており、`-- atlas:txmode none` 前例ゼロ。240k 行 + 1 列 functional B-tree の build は実測 < 5 秒で、Atlas の transactional migration で安全。

### Driver 書き換え

```sql
-- before
WHERE ft.tag_name ILIKE $1 || '%'

-- after (機能等価、index range scan が効く)
WHERE lower(ft.tag_name) LIKE lower($1) || '%'
```

### 検討した代替案

| 代替案 | 却下理由 |
|---|---|
| `citext` 型移行 | PG 公式が nondeterministic collation に置き換え推奨。btree dedup 喪失で index が膨らみ、PG 17 まで LIKE が citext/collation 系に乗らない (Cybertec/Albe Laurenz)。型変更は alembic-style の重い migration |
| `pg_trgm` GIN trigram | substring/fuzzy 用途の opclass。prefix-only にはオーバーキル、index size 3-5x、write very expensive (depesz)。Elephant Tamer の 12M GeoNames ベンチで btree functional より遅い |
| `COLLATE "C"` 列化 | tag_name の自然順 sort を破壊 (大文字優先の C 順)。他クエリ ORDER BY tag_name 系への副作用が広い |
| `CREATE INDEX CONCURRENTLY` (`-- atlas:txmode none`) | Atlas 既存 convention 外。240k 行は non-concurrent 5 秒以内、tag prefix は read-only path で短時間 retry 許容 |
| alt-backend に tag prefix LRU cache | [[000915]] PR-B2 と同パターンだが、index で 1-2 ms に落ちれば cache の付加価値は限定的。無効化運用コストを新規に背負わない |
| クエリ側で `lower(...)` をやらず client 側で lower した値を bind | planner が `lower(const)` を定数畳み込みするので SQL 側のままで OK。client 側分岐は不要 |

## Consequences

### Pros

- **autocomplete 帯 (3 文字以上 prefix) で 2-9 ms** (前 180 ms、**~30-80x ↓**) を実測:
  - `Python` 5x avg: 6.6 ms
  - `JavaScript` 5x avg: 2.2 ms
  - `kubernetes` 5x avg: 2.1 ms
  - `q` (1 文字) 5x avg: 18 ms (10x ↓)
- EXPLAIN で `Bitmap Index Scan on idx_feed_tags_tag_name_lower` + `Index Cond: (lower(tag_name) ~>=~ 'x') AND (lower(tag_name) ~<~ 'y')` が走り、目的の range scan に切り替わったことを確認
- 既存 `gateway/global_search_gateway/tag_search_gateway.go` の呼出契約は変わらず、上位 usecase / handler / Pact / Hurl いずれも変更不要 (機能等価 refactor)
- TDD で `pgxmock` の SQL pattern 一致 / ILIKE 不在の 2 軸 assertion を `search_tags_driver_test.go` に固定、将来の regression を即検知

### Cons / Tradeoffs

- **`A` のような broad single-character prefix では 80-127 ms に留まる** (1.5-2x 改善)。`feed_tags` 240k 中 ~16k がヒットし、後段の `INNER JOIN article_tags (912k)` + `COUNT(DISTINCT)` が支配的になるため。autocomplete UX で 1 文字検索を許す場合は事実上の上限
- **既存 `idx_feed_tags_tag_name` (素の B-tree on tag_name) が冗長になる**。新 index は完全置換ではなく ILIKE / case-sensitive search で使い分けが残る可能性があり、本 PR では削除しない。書き込み index 維持コストは < 1 %
- **書き込みオーバーヘッド +1 index 分** (insert/update on `feed_tags`)。`feed_tags` は indexing pipeline からの append 主体で、tag 更新頻度は低く実用上問題なし
- **`lower(...)` の SQL 書き換えを忘れた呼出元が将来現れると seq scan に戻る**。`search_tags_driver_test.go:TestTagRepository_SearchTagsByPrefix_NoBareILIKE` で guardrail 済
