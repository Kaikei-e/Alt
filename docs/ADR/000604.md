# ADR-000604: rag-orchestrator に会話状態レイヤーとルールベースプランナーを導入し Conversational RAG を再設計する

## Status

Accepted

## Date

2026-03-28

## Affected Services

- rag-orchestrator (Go)

## Context

Augur（Ask Augur）の会話型 RAG は、毎ターンの raw conversation history のみに依存する stateless retrieval + prompt stitching で実装されていた。この設計には以下の構造的な問題がある。

**1. 会話状態の欠如**

フォローアップ質問（「もっと詳しく」「それって本当？」「関連する記事はある？」）が毎回独立した検索クエリとして処理される。直前の回答で何を扱ったか、どの記事スコープにいるか、どのエンティティに注目しているかの情報が失われるため、topic drift や misaligned retrieval が発生する。

**2. プランナーの欠如**

ユーザーの意図解釈と retrieval policy 決定が `buildPrompt()` 内の inline switch 文に混在している。SubIntent 分類の結果がそのまま retrieval 方法を決定するため、曖昧な参照解決や clarification の余地がない。

**3. SubIntentNone の unconditional general re-retrieval**

SubIntent が分類できなかったフォローアップ（`SubIntentNone`）は無条件で general re-retrieval を実行し、記事チャンクとグローバル検索結果をマージしていた。これにより記事スコープの会話に無関係なコンテキストが混入する。

**4. 曖昧な参照が検索として処理される**

「もっと詳しく」のような曖昧なフォローアップは、reference resolution を経ずにそのまま query expansion → vector search に渡される。その結果、ユーザーが意図した対象とは異なるコンテキストが retrieved される。

これらの問題は既存の SubIntent 拡張（[[000600]]、[[000603]]）では解決できない構造的な課題であり、会話システムとして必要な state / planner / policy separation の導入が必要であった。

### 調査に基づく設計判断

Agentic RAG の最新研究（arXiv 2501.09136）および Azure Agentic Retrieval、DH-RAG（arXiv 2502.13847）の設計パターンを参照し、以下の知見を設計に反映した:

- **Planner と Answerer の分離**: LLM に query decomposition と answer generation を同時に行わせるのではなく、明示的なプランニングステップを挿入する
- **Operation-driven retrieval policy**: 質問タイプごとに異なる retrieval 戦略を適用する（detail は article-only、related_articles は tool-first、fact_check は article+global）
- **Clarification-first**: 曖昧性が高い場合は retrieval を実行せず、ユーザーに明確化を求める

ただし Gemma 3 4Bの制約上、LLM ベースのプランニングは採用せず、ルールベースの分類に基づく設計とした。

## Decision

rag-orchestrator に **会話状態レイヤー**（ConversationState）と**ルールベースプランナー**（ConversationPlanner）を導入し、retrieval pipeline の前に明示的なプランニングステップを挿入する。

以下の 6 フェーズで実装する:

### Phase A: ConversationState レイヤー

- **`domain/conversation_state.go`**: `ConversationState`（ThreadID, Mode, CurrentArticleID, LastAnswerScope, FocusEntities, FocusClaims, LastCitations, TopicConfidence, TurnCount）をドメイン層の値オブジェクトとして定義
- **`usecase/conversation_store.go`**: in-memory LRU ストア（`expirable.LRU`、TTL 30分、最大 1024 エントリ）
- **`usecase/conversation_state_updater.go`**: `DeriveStateUpdate()` — 前の状態 + intent + planner output + answer output からイミュータブルに次の状態を導出

### Phase B: ConversationPlanner

- **`domain/planner.go`**: `PlannerOutput`（Operation, RetrievalPolicy, NeedsClarification, Confidence 等）をドメイン層に定義
- **`usecase/planner.go`**: ルールベースの `ConversationPlanner.Plan()` メソッド

プランナーは 4 ステップで動作する:

1. **参照解決**: 曖昧なパターン（「もっと詳しく」「それって本当？」「別の観点では？」）を検出し、ConversationState を用いて具体的な operation に解決
2. **Operation 分類**: 既存の SubIntentType / IntentType から PlannerOperation にマッピング
3. **Retrieval Policy 決定**: Operation + 会話モードから deterministic に policy を決定
4. **Clarification 判定**: 曖昧な参照 + state なし（新規セッション）/ 低 TopicConfidence の場合に NeedsClarification を発行

### Phase C: Policy-Driven Retrieval

`buildPrompt()` 内の約 70 行の inline SubIntent switch を `PlannerOutput.RetrievalPolicy` ベースの switch に置換:

| Policy | 動作 |
|---|---|
| `article_only` | 記事チャンクのみ。general re-retrieval なし |
| `article_plus_global` | 記事チャンク + quality gate で general 追加 |
| `global_only` | グローバル検索のみ |
| `tool_only` | vector retrieval スキップ。ツール出力が primary data |
| `no_retrieval` | retrieval 完全スキップ（clarification パス） |

**後方互換**: Planner 未設定時は既存の SubIntent switch がそのまま動作する legacy パスを保持。

### Phase D: Tool-First パス

`ToolDispatcher` に `DispatchForPlan()` メソッドを追加。`PolicyToolOnly` の場合、`PlannerOutput.Operation` に基づいてツールを選択・実行する（現在は `related_articles` のみ対応）。

### Phase E: プロンプト簡素化

`PromptInput` に `PlannerOutput` フィールドを追加。将来的に Operation 駆動のマイクロテンプレートに移行する基盤を整備。

### Phase F: Clarification-First UX

- `StreamEventKindClarification` イベントタイプを追加
- `StreamClarification` ペイロード（Message + Options）を定義
- `Stream()` 内で `NeedsClarification == true` の場合、retrieval と LLM 生成をスキップし、clarification イベントを即時返却

### Clean Architecture レイヤー配置

| 型/ロジック | 層 | 理由 |
|---|---|---|
| ConversationState, PlannerOutput | Domain | 純粋な値オブジェクト。外部依存なし |
| ConversationStore, DeriveStateUpdate | Usecase | ビジネスロジック |
| ConversationPlanner | Usecase | ルールベースのプランニングロジック |

## Consequences

### Pros

- **Topic drift の抑制**: ConversationState が記事スコープとフォーカスエンティティを明示的に保持するため、フォローアップで不要な global re-retrieval が発生しない
- **曖昧な参照の解決**: 「もっと詳しく」が state.LastAnswerScope に基づいて detail / clarification に適切にルーティングされる
- **SubIntentNone anti-pattern の解消**: Planner が SubIntentNone を明示的に operation 分類するため、unconditional general re-retrieval が不要になる
- **Clarification パス**: 曖昧性が高い場合に hallucinated best guess ではなく明確化質問を返すことで、回答品質の底上げが可能
- **Observability 向上**: `AnswerDebug` に `PlannerOperation`, `PlannerConfidence`, `NeedsClarification` が追加され、失敗モードの分類が machine-actionable になる
- **後方互換**: Planner 未設定時は既存動作を完全に保持。段階的なロールアウトが可能
- **ルールベースで追加レイテンシなし**: Planner は純粋な Go 関数（0ms、LLM 呼び出しなし）

### Cons / Tradeoffs

- **In-memory state はプロセス再起動で消失**: ConversationStore は LRU キャッシュであるため、rag-orchestrator の再起動で全会話状態が失われる。現時点ではクライアントが full history を毎回送信する設計のため実害は限定的だが、long-lived session が重要になった場合は永続化が必要
- **ルールベースプランナーの限界**: キーワードマッチングに依存するため、「それについてもう少し」のような間接的な参照は検出できないケースがある。将来的に小型モデルによる bounded classification への昇格が考えられるが、現在の GPU 制約では Gemma 3 4B をプランニングに使う余裕がない
- **TopicConfidence のヒューリスティック**: 同一記事 ID / 同一会話モードで信頼度を加算する単純な計算であり、実際のトピック一貫性を完全には反映しない
- **FE 対応が未実装**: Phase F の clarification イベントを受信・表示する alt-frontend-sv 側の対応は本 ADR のスコープ外。proto 変更も Phase F 本格化時に実施予定
- **Legacy パスの二重管理**: Planner 導入期間中、`buildPrompt()` に planner パスと legacy パスの両方が共存する。Planner の検証完了後に legacy パスを削除する予定
