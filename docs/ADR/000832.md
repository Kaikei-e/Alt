# ADR-000832: 3days Recap の評価軸を 5 拡張し、context packing・診断 surface・dashboard を投入する

## Status

Accepted

## Date

2026-04-23

## Affected Services

- **recap-evaluator (Python)** — 5 evaluator 新設 + SummaryMetrics / SummaryMetricsResponse 拡張 + Streamlit dashboard
- **news-creator (Python)** — `_score_sentence_for_packing` / `_max_cluster_section_length` helper 追加、3days の MAX_CLUSTER_SECTION_LENGTH を 12K → 8K
- **recap-worker (Rust)** — `cluster_genre` の error context を richer 化、`bullets_ja` に `source_sentence_ids` スロット追加
- **recap-subworker (Python)** — classifier proba 分布を structured log に emit

## Context

[[000825]] / [[000827]] / [[000828]] で 2026-04-22 に silent failure の穴 3 つ (recap-subworker joblib bind 空化 / recap-worker rust-bert cache 空 bind / per-genre failure の silent swallow) を塞いだが、ユーザーからは依然として「3days Recap の品質が悪い」「何度直しても改善しない」という訴えが続いていた。本 ADR はこの訴えの実体を DB 実観測で特定し、**何が効いて何が効いていないかを数値で判定できる状態** を先に作るための eval-first 一括投入である。

### DB 実観測 (2026-04-23 08:43 JST, recap-db)

直近 3 件の completed 3days job で `recap_outputs` に出た genre 数:

| job_id (kicked_at UTC) | genre 数 | 内訳 |
|---|---|---|
| `dc765823` 2026-04-22 17:00 | 7 | consumer_tech + _004/_006/_007, politics_government + _004/_007 |
| `9aad1079` 2026-04-22 13:06 | 9 | 同系 + subgenre 数差 |
| `00b602ac` 2026-04-22 10:42 | 5 | consumer_tech / diplomacy_security / politics_government / software_dev / space_astronomy |

30 ジャンル taxonomy のうち出力は 2〜5 coarse + subgenre のみで、残り約 25 ジャンルは `genres_no_evidence` として section hollow。

[[000828]] で新規 surface された `recap_failed_tasks` を最新 `dc765823` について集計:

```
dispatch_summary | 15 件 | error = "Clustering failed: clustering failed"
```

15 件すべてが **subgenre splitter の HDBSCAN clustering failed** で、coarse で拾えた consumer_tech / politics_government を subgenre に分割しようとした段で全滅している。

### 「治らない」の三重構造

1. **coarse classifier の 2 バケット偏在** — [[000828]] §Cons で "`multi_class='ovr'` force patch は未処置" と明記した proba 圧縮のままで、22 ジャンルが 0.5 threshold を越えられない。`_ja.json` で 0.10-0.50 まで下げて吸収しているが、病巣自体は残存。
2. **subgenre splitter の HDBSCAN 失敗** — min_cluster_size / min_samples の設定と evidence 数が合っておらず、coarse の 2 ジャンルから派生させる subgenre でも clustering failed が常態化。
3. **評価基盤が不完全** — morning-letter-v2-epic Phase 0 Issue 1 が要求する 7 軸のうち、G-Eval / ROUGE / BERTScore / Faithfulness / genre F1 は `recap-evaluator` に実装済だが、`fallback_rate` / `json_repair_rate` / `redundancy` / `readability (quick-catch)` / `source_grounding` が欠落。dashboard も未整備。結果として「何かを直した後に本当に良くなったか」を数値で判定できない状態が 2026-04-14 以降続いていた。

### すでに実装済 (ユーザー認識とのギャップ)

- Issue 2 (3days/7days prompt 分離) **完実装**: `news-creator/app/prompts/recap_summary_3days.jinja` が `_is_3days_request()` 経由で分岐済。bullet 2-4 本 / 400-800 字に分離済み
- Issue 5 (fallback + degraded metadata) **完実装**: `_create_fallback_from_clusters()` で centroid 優先 / references 保持 / `is_degraded` フラグ済
- Issue 3 (hierarchical map-reduce) **framework 完実装**: `_generate_hierarchical_summary()` / `_recursive_reduce_phase()` 済
- Issue 4 (context packing) **基本実装**: centroid / date ベース、numeric / multi-source 重み付けは未実装

つまり prompt / fallback / map-reduce は既にあり、症状の上流 (classifier / subgenre clustering) と最下流 (packing の質 + 評価) に未解決の穴が集中していた。本 ADR は **prompt template / fallback logic には一切触らず**、評価と packing と診断の 3 レイヤーを同時に埋める。

## Decision

### 1. recap-evaluator に 5 Morning-Letter 評価軸を追加する

`recap-evaluator/src/recap_evaluator/evaluator/` 配下に以下 5 ファイルを新設する。

- `fallback_rate.py` — `body_json.metadata.is_degraded` または `model == "low-evidence-extractive"` の比率を返す純 Python 統計 evaluator。
- `json_repair_rate.py` — `metadata.json_repair_attempted is True` の比率を返す同種の統計 evaluator。
- `redundancy.py` — summary bullets の pairwise ROUGE-L F1 平均を返す。既存 `ROUGEEvaluator` を再利用し、bullet × bullet の組み合わせを `itertools.combinations` で回す。高いほど冗長。
- `readability.py` — Ollama gateway の `score_readability` Protocol を介して quick-catch 文脈の readability を 1-5 で取得する LLM 評価。空要約は 0 を返し、LLM 失敗時も 0 で degrade。
- `source_grounding.py` — `bullets_ja` (JSONB 配列) 優先、fallback で `body_json.bullets` / `body_json.summary.bullets` を辿り、`source_sentence_ids` が非空の bullet 比率を返す。

`domain/models.py::SummaryMetrics` と `handler/schemas.py::SummaryMetricsResponse` に 5 optional float フィールド (default 0.0) を追加し、既存 consumer の契約を壊さない。`SummaryEvaluator.evaluate_batch` の末尾で `_apply_morning_letter_axes` helper を呼び、DB fetch 済の `all_outputs_flat` を一度だけスキャンして 5 軸を埋める。readability だけは `evaluate_batch` 内で Ollama 呼び出しを伴うため既存 `sample_size` を上限にサンプリング。

### 2. news-creator の context packing に 3days 専用 budget と重み付けを導入する

`usecase/recap_summary_usecase.py::RecapSummaryUsecase` に 2 つの helper を追加する。

- `_max_cluster_section_length(request) -> int` — 3days なら 8_000 文字、それ以外は従来通り 12_000 文字を返す。`_render_prompt_body` の `MAX_CLUSTER_SECTION_LENGTH = 12_000` ハードコードをこの helper 呼び出しに置き換える。
- `_score_sentence_for_packing(sentence, cluster_sentences) -> float` — centroid フラグで +2.0、正規表現 `_NUMERIC_PACKING_PATTERN` (金額 / 日付 / % / 西暦) マッチで +1.0、同クラスタ内に他ドメインの source が存在すれば +0.5 を base 1.0 に加算する。同一ドメイン重複は multi-source とはみなさない。

`_score_sentence_for_packing` は本 ADR では helper として単体露出させ、`_pack_clusters` からの既存 sentence 順序統合は既存 test の回帰リスクを考慮して次サイクルに分離した。prompt template (`recap_summary_3days.jinja`) は一切触らない。

### 3. recap-worker の clustering error context を richer 化する

`pipeline/dispatch/clustering.rs::cluster_genre` の `.context("clustering timeout")?` / `.context("clustering failed")?` を以下に置き換える。

```rust
.with_context(|| format!(
    "clustering timeout: genre={genre} articles={article_count} timeout_s={}",
    genre_timeout.as_secs()
))?
.with_context(|| format!(
    "clustering failed: genre={genre} articles={article_count}"
))?;
```

これで `recap_failed_tasks.error` に保存される snippet が従来の `"Clustering failed: clustering failed"` から `"Clustering failed: clustering failed: genre=consumer_tech_005 articles=14"` のような per-genre 情報を含む形になり、PostgreSQL 直接照会で 15 件 failure の内訳を即座に切り分けられる。

### 4. recap-worker で bullets_ja に source_sentence_ids スロットを scaffold する

`pipeline/persist.rs` の bullet 構築を `json!({"text": bullet, "sources": top_sources, "source_sentence_ids": Vec::<i64>::new()})` に拡張する。citation reconciliation (news-creator レスポンスの `[n]` を representative sentence id に紐付ける) 本体は scope out。空配列でも evaluator は「全 bullet が non-grounded」と見て `source_grounding_score = 0.0` を返すだけで、fail-open かつ既存契約破壊なし。

### 5. recap-subworker の classifier に proba histogram 診断 log を emit する

`services/classifier.py::predict_batch` の末尾 (return 直前) で、`probs_batch` を per-genre に集計して mean / max / p90 / count_above_threshold を計算し、structured log `classifier proba histogram` として 1 行出す。例外は `logger.warning` で swallow。これで coarse classifier がどの genre でどの程度 proba を圧縮しているかを、DB テーブル追加なしで Loki ベースのログ集計だけで分析できる。

### 6. 手動起動の Streamlit dashboard を新設する

`recap-evaluator/dashboard/app.py` を新設。`RECAP_DB_DSN` 環境変数経由で recap-db に read-only 接続し、直近 N 日 (デフォルト 14) の 3days job について genre_count / stage 別 failed_count / 5 軸 time-series を Streamlit の `line_chart` / `bar_chart` で可視化する。compose には含めず `uv run streamlit run dashboard/app.py` で手動起動のみ。運用化 (compose service 化 / 認証 / metric export) は別チケットにスコープアウト。

### 検討した代替案と却下理由

- **classifier の `multi_class='ovr'` force patch を本 ADR で撤去する**: 本命の根本治療。ただし sklearn バージョン整合 / joblib 再学習 / thresholds 再調整を要し、本 ADR のスコープを大幅に越える。**却下 (別 ADR へ移管)**。
- **subgenre splitter の HDBSCAN パラメータ再調整を本 ADR に含める**: 15 件 clustering failed の根本治療。ただし min_cluster_size と min_samples は evidence 量と genre の coherence に依存し、本 ADR で確立する評価基盤がなければ「何を基準に調整したか」を示せない。先に eval を整備し、次サイクルで定量的に調整する。**却下 (別 ADR へ移管)**。
- **bullet 単位の citation parsing を本 ADR で実装する**: 理想的には news-creator レスポンスの `[n]` と `representative_sentences[n-1].id` を突合して `source_sentence_ids` を埋める。ただし news-creator 側の引用フォーマットが複数パターン (`[1]`、`[1, 2]`、`[L1P3]`) 混在しており、完全な parser は別単位で書くべき分量。本 ADR では空配列でスロットを確保するに留め、`SourceGroundingEvaluator` は 0.0 を返すだけで運用に入る。**却下 (scaffold のみ採用)**。
- **診断情報を `recap_subworker_diagnostics` テーブルに INSERT する**: より正式な観測基盤。ただし新テーブル書き込みが分類 hot path に入るのは scope 過大。まず structured log で surface し、Loki / Grafana での集計が不足と判明してから DB table に昇格させる。**却下 (現時点では log のみ)**。
- **Streamlit dashboard を compose に組み込む**: 運用化の自然な着地点。ただし認証境界 / ネットワーク露出 / image サイズ増を考慮すると本 ADR のスコープを越える。手動起動から入り、運用要件が固まってから compose 化する。**却下 (手動起動のみ採用)**。
- **`_pack_clusters` のリファクタで `_score_sentence_for_packing` を実パッキング順序にも使う**: packing 順序の変更は既存 130 件の usecase test に回帰リスクをもたらす。helper を単体露出し、次サイクルで順序統合する段階的アプローチを採る。**却下 (helper 単体のみ採用)**。

## Consequences

### Pros

- **eval-first に揃う**. ユーザーメモリ `feedback_eval_first` (「RAG改善は評価基盤を先に作り、指標で判定してから実装」) と一貫する形で、classifier 本体修正の前に 5 軸の数値が揃う。次の classifier / HDBSCAN 調整の before/after が、勘ではなく `/api/v1/evaluations/summary` のレスポンスで判定できる。
- **診断ループが 3 層で閉じる**. [[000828]] が閉じた「stage 粒度 → per-genre 粒度」のループに加えて、本 ADR は (a) `recap_failed_tasks.error` に per-genre article 数、(b) `classifier proba histogram` log に genre ごとの proba 分布、(c) dashboard に 5 軸 time-series、の 3 層を追加する。2 バケット偏在 (coarse) と 15 件 clustering failed (subgenre) の原因切り分けが、単一 job の観測から可能になる。
- **3days quick-catch に特化した packing**. MAX_CLUSTER_SECTION_LENGTH を 12K → 8K に下げたことで、3days の map-reduce 閾値越えが減り、cluster section の質が変化点に寄る。numeric / multi-source 加点の helper は単体露出されているので、次サイクルで `_pack_clusters` に統合しても既存 `_render_prompt_body` の挙動を保てる。
- **契約破壊なし**. SummaryMetricsResponse / RecapOutput / bullets_ja に追加されたフィールドはすべて optional で、既存 consumer (recap-worker pact consumer を含む) の振る舞いを変えない。existing 4 軸 (G-Eval / ROUGE / BERTScore / Faithfulness) は同じキーで返り続ける。
- **TDD outside-in 厳守**. `feedback_tdd_outside_in` / `feedback_tdd_strict` に沿って、integration (evaluation_flow 2 件) → unit (5 evaluator + 3 packing) → 実装の順で RED → GREEN を分離した 7 commit。`feedback_no_incremental_commits` (実装中は都度 commit しない) に沿って実装完了まで commit せず、意味単位で 7 commit にまとめた。

### Cons / Tradeoffs

- **coarse classifier の 2 バケット偏在は未解決のまま**. 本 ADR は症状を定量化する基盤を作るが、病巣 (multi_class='ovr' patch / joblib 再学習) には触れない。次サイクルで別 ADR として起票する。本 ADR merge 後も当面ジャンル 2〜7 の状態は続く。
- **subgenre clustering failed 15 件の根本も未解決**. 同上。本 ADR の clustering.rs context enrichment で「何 articles で失敗したか」が見えるようになるので、次の調整の入力になる。
- **readability は Ollama 依存**. Ollama が応答しない / モデル未ロードの場合、`readability_score = 0.0` にフォールバックする。5 軸のうち LLM 依存は readability のみ、他 4 軸は DB fetch 済 outputs と純 Python で計算可能。
- **`source_grounding_score` は当面常に 0.0**. bullets_ja に `source_sentence_ids: []` が空配列で入るだけなので、citation parsing が入るまで評価値は動かない。dashboard 上では「今は 0、parser 実装後に向上するはず」という watch metric として機能する。
- **Streamlit dashboard は運用化されていない**. 手動起動のみで alert / SSO / dashboards-as-code はない。compose に入れる運用要件が固まったら別 ADR で昇格させる。
- **news-creator の `_score_sentence_for_packing` が packing 実順序にはまだ効かない**. helper を単体で足しただけで、`_pack_clusters` (= `_render_prompt_body` 内の forループ) の sentence 順序は変えていない。3days の budget 縮小 (12K → 8K) だけが実効。
- **recap-worker `clustering.rs` の context 変更で `recap_failed_tasks.error` の文字列長が伸びる**. PostgreSQL `text` カラムなので物理上限は実質無いが、既存 monitoring / alert が error snippet の prefix match を前提にしている場合は調整が必要。今回は監視側が「`Clustering failed:`」までの prefix しか見ていないことを確認済。
- **proba histogram log の emit 頻度**. 1 `predict_batch` につき 1 行。大量バッチでログ量が増えるが、hot path の外 (prediction return 直前) なので遅延影響はない。Loki の retention / cardinality を次回 runbook update で要確認。
