# ADR-000743: X-Service-Token / SERVICE_SECRET を全サービスから撤去し認証境界を mTLS に一本化する

## Status

Accepted

## Date

2026-04-15

## Affected Services

- alt-backend (Go) — `middleware/service_auth_middleware.go` + test、`connect/v2/middleware/service_auth_interceptor.go` 削除。`rest/internal_handlers.go` / `rest/recap_handlers.go` / `rest/rest_feeds/summarization/helpers.go` / `connect/v2/server.go` から RequireServiceAuth 呼び出し除去。`driver/search_indexer_connect/client.go` / `driver/preprocessor_connect/client.go` の serviceTokenTransport 削除
- search-indexer (Go) — 同上の middleware + interceptor 削除。`bootstrap/servers.go` / `connect/v2/server.go` から配線除去。`driver/backend_api/client.go` の serviceToken field 撤去
- pre-processor (Go) — 同上。`bootstrap/server.go` から middleware チェーン除去、`internal/auth/middleware/auth.go::RequireServiceAuth` を no-op 化。`repository/external_api_repository.go` の token 注入撤去
- alt-butterfly-facade (Go) — `internal/server/server.go` の TTS / Acolyte 向け token 注入ラッパ撤去、`internal/client/backend_client.go` の token set + forward list 削除
- acolyte-orchestrator (Python) — `handler/connect_service.py::_verify_token` を no-op 化、`gateway/search_indexer_gw.py` の `_auth_headers` 削除
- tag-generator (Python) — `auth_service.py::verify_service_token` を no-op 化、`driver/connect_client_factory.py` の token injection 削除
- tts-speaker (Python) — `infra/auth.py::verify_service_token` を no-op 化
- rag-orchestrator (Go) — `adapter/rag_http/search_indexer_client.go` の ServiceToken field + setAuthHeader 撤去、`adapter/altdb/article_client.go` の token injection 撤去、`di/container.go` の `serviceTokenInterceptor` 削除
- recap-worker (Rust) — `clients/alt_backend.rs` / `clients/tag_generator.rs` の `service_token: Option<String>` field + header 注入撤去、`pipeline/{morning,orchestrator}.rs` の config 参照削除
- altctl (Go) — `internal/adminclient/client.go` の X-Service-Token 注入 + 該当 test 期待値を「ヘッダ不在」に変更
- Pact consumer tests (5 サービス) — `.with_header("X-Service-Token", ...)` matcher 全削除、pact 再生成
- Pact provider tests — `ServiceAuthMiddleware` wrap 除去、search-indexer の `testServiceToken` const 撤去
- compose 全体 — 全 service 定義から `SERVICE_SECRET` / `SERVICE_TOKEN` env / `service_secret` secret mount 除去。`compose/base.yaml` から `service_secret` secret 宣言削除
- `scripts/check-no-service-token.sh` — 新設 (post-cutover guard、`--strict` で CI 失敗化)

## Context

[[000741]] / [[000742]] で受信側 mTLS + 送信側 cert 提示が全サービスで揃った。残った課題は **アプリ層の shared-secret 多重防御 (`X-Service-Token` / `SERVICE_SECRET`) が mTLS と並立しており、以下の問題を残す**:

1. **二元管理の歪み** — [[PM-2026-025]] のように、provider が token 要求を変えたのに consumer が追従せず 24h のサイレント degradation を起こす失敗モードが繰り返される
2. **caller 識別の不整合** — mTLS は CN 単位で caller 識別するのに対し、shared secret は全 caller で同一値。同じ caller に対する audit log が 2 経路に分かれる
3. **設計上の冗長** — [[000717]] 時点の「shared secret は application-layer 追加防御」という位置付けは step-ca 未導入の過渡期の判断。mTLS が恒常化した今は冗長
4. **Pact CDC の負債** — consumer pact が token matcher を持つため、token を変えたり撤去するたびに 5 サービス分の pact を再生成する必要があり、change budget が膨らむ

ユーザー指示 (plan file 決定事項) として **hard cutover** 方針が確定しており、段階的 feature flag 方式は取らない。

## Decision

**X-Service-Token / SERVICE_SECRET / ServiceAuthMiddleware / ServiceAuthInterceptor / resolve_service_secret を全サービスから物理削除し、service-to-service 認証は mTLS peer identity のみに一本化する。**

### 削除対象 (enforcement + injection)

1. **Go service auth middleware の完全削除** — alt-backend / search-indexer / pre-processor の `ServiceAuthMiddleware.go` + test と connect interceptor を `rm`
2. **全 HTTP listener からの RequireServiceAuth wrap 撤去**
   - alt-backend: `/v1/internal/*` / `/v1/dashboard/*` / `/v1/recap` group
   - search-indexer: `/v1/search` REST + Connect-RPC SearchService
   - pre-processor: `/api/v1/summarize*` Echo group + Connect-RPC PreProcessorService
3. **全 Go/Rust/Python client の X-Service-Token header 注入コード削除**
4. **コンフィグ struct の `ServiceToken` field を no-op 化** (呼び出し側 DI signature を壊さないために値を受けるだけ。将来 `_` 引数化)
5. **docker compose から全 `SERVICE_SECRET` / `SERVICE_TOKEN` env line と `service_secret` secret mount 除去**
6. **`compose/base.yaml` から `service_secret` secret 宣言削除** — リポジトリ上の secret 定義自体が無くなる

### No-op 化 vs 完全削除の選択

**関数シンボル (例: `verify_service_token`, `RequireServiceAuth`, `setServiceToken`, `addAuth`) は削除せず no-op 化**する理由:

- FastAPI の `Depends()`、Echo middleware chain、Connect-RPC interceptor 等の呼び出し側を一斉書き換えると diff が肥大し、merge conflict リスクが上がる
- No-op 化 + `_ = request` で lint green を保ちつつ、後続 cleanup PR でシンボルごと削除できる
- 一方で **struct / class / 独立 interceptor ファイル** は物理削除 (dead code として残ると誤解の元)

### Pact の再生成

5 箇所の consumer test から `.with_header("X-Service-Token", ...)` matcher を削除して再実行。provider test (search-indexer) では `ServiceAuthMiddleware` stub wrapping を撤去し、`acolyte-orchestrator → search-indexer` の intentional-exclude は既に [[000739]] で解除済なのでそのまま。

### CI guard: `scripts/check-no-service-token.sh`

post-cutover に `--strict` モードで CI に組み込むための guard script を新設。allowlist:

- `docs/` (ADR・runbook・postmortem は legacy 説明に参照するので許容)
- `pacts/` (古い pact file が historical record として残る間)
- `scripts/check-no-service-token.sh` 自身

パターン:

```
X-Service-Token | SERVICE_SECRET | SERVICE_TOKEN |
service_secret\b | service_token\b | resolve_service_secret |
ServiceAuthMiddleware | service_auth_middleware
```

初回実行時は expected-fail で residue 件数を報告し、全削除が済んだら `--strict` を branch protection 必須 check にする。

### 検討した代替案

- **案 A: feature flag (`AUTH_ENFORCEMENT=mtls|token|both`) で段階移行** — cutover window の運用コスト > hard cutover のリスクと判断。mTLS が全 caller で成立していることを [[000741]] / [[000742]] の CI + pact-check で担保できるので段階移行不要
- **案 B: middleware を残し、検証ロジックだけ no-op (token 値に関係なく pass)** — コードの意図が曖昧になり後続のエンジニアが「なぜ middleware があるのに何もしないのか」を解読する必要が出る。削除が最も明快
- **案 C: CI guard を `rg -F` grep ベースから custom linter / semgrep に昇格** — rg で十分。semgrep 化は false positive が増える時点で再検討

## Consequences

### Pros

- 東西認証が mTLS peer identity 一本化。認証境界が単一ソースになり [[PM-2026-025]] 型のサイレント degradation 構造が消える
- shared-secret の Docker secret が不要になり、リポジトリ上の secret 管理サーフェスが縮小
- Pact の maintenance 負荷が減る (token 関連の matcher が消え、壊れる理由が一つ減る)
- `./scripts/check-no-service-token.sh --strict` を CI required check にすれば、token 系シンボルの再導入は物理的に不可能になる

### Cons / Tradeoffs

- **mTLS が止まると全 east-west が止まる single point of failure になる**。step-ca の HA / backup 運用の重要度が跳ね上がる (`docs/runbooks/mtls-cutover.md`)
- **No-op 関数が一時的に残る** (`verify_service_token`, `RequireServiceAuth`, `setServiceToken`, `addAuth`)。follow-up PR で呼び出し側ごと削除が必要。実質的には tech debt
- **compose の `SERVICE_TOKEN_FILE` env を期待している旧デプロイ script / k8s manifest があれば壊れる**。Alt は Compose 単一なので影響なしだが、外部 fork でこの ADR を適用する場合は注意
- **hard cutover のため、migration window 中に戻せない**。rollback は Phase C PR の revert + cert 失効で mTLS 自体を無効化する形になる

### 追加 cleanup (本 ADR 内で完了)

cutover 直後 46 件の `X-Service-Token` 残留があった状態から、本 ADR 内の追加 cleanup で **enforcement / injection 層を完全消去**まで進めた。最終形:

| Pattern | Before cleanup | After cleanup | 削減率 |
|---------|---------------|---------------|--------|
| `X-Service-Token` | 157 | 5 | -97% |
| `SERVICE_SECRET` | 173 | 23 | -87% |
| `ServiceAuthMiddleware` | 31 | 0 | **-100%** |
| `resolve_service_secret` | 13 | 0 | **-100%** |
| 全パターン合計 | 527 | 55 | **-90%** |

#### 追加削除されたコード / 設定

- **recap-worker (Rust)** — `config.rs::AltBackend` / `TagGenerator` struct から `service_token: Option<String>` field、`service_token()` getter、`load_optional_secret()` 経由の env loader を全削除。`pipeline/{morning,orchestrator}.rs` の参照箇所も同時撤去
- **alt-butterfly-facade (Go)** — `config/config.go` の `TTSServiceSecret` / `ServiceSecret` / `ServiceSecretFile` field と `LoadServiceSecret()` ヘルパを削除。`internal/server/server.go` の token 注入ラッパも撤去済 (本 ADR の主削除)
- **alt-backend (Go)** — `InternalAPIConfig` 構造体を空 struct 化 (mTLS 移行で全 field 不要に)、`config.go` 内の関連配線を `""` literal 化
- **search-indexer (Go)** — `BackendAPIConfig` から `ServiceToken` field 削除 (mTLS で peer 識別)
- **rag-orchestrator (Go)** — `BackendConfig` から `ServiceToken` field 削除 (di/container.go の interceptor 撤去とセット)
- **pre-processor (Go)** — `config/validate.go` から `loadServiceSecretForValidation()` ヘルパと `APP_ENV=production` で SERVICE_SECRET 必須化していた production gate を削除。`bootstrap/wire.go` の `SERVICE_TOKEN` env 読み込みも撤去
- **altctl (Go)** — `cmd/home.go` の `--service-token` flag 定義と `SERVICE_TOKEN` env fallback を撤去 (mTLS なので CLI 経由トークン渡しは不要)
- **news-creator (Python)** — `config/config.py` から `service_secret` field と「未設定なら ValueError」の startup 検証を削除
- **acolyte-orchestrator (Python)** — `config/settings.py` から `service_secret` / `service_token_file` field と `resolve_service_secret()` helper 関数を完全削除 (no-op 化ではなく物理削除)
- **compose** — `compose.tts.yaml` から残存していた `SERVICE_SECRET` env line を削除
- **テストファイル群** — 各サービスの test files から `monkeypatch.setenv("SERVICE_SECRET", ...)` セットアップ、`Headers={"X-Service-Token": ...}` request fixture を撤去。残った test では `assert.Empty(t, req.Header.Get("X-Service-Token"))` 形式の **regression guard** に置き換え

#### `scripts/check-no-service-token.sh` の allowlist 強化

初版は単純な path prefix allowlist だったが、cleanup を進める過程で実用的な強化を実施:

- **allowlist 拡張** — `_test.go` (Go test files、guard assertion を含む)、`tests/` (全言語の test directory)、`*.json` (pact JSON は historical record として保持)、`README.md` (legacy 説明用)、`docs/` (ADR / runbook / postmortem)
- **glob pattern の path-suffix vs filename-suffix 分離** — `_test.go` のような **filename suffix** と `tests/` のような **path component** を別 case で評価する分岐を追加 (旧版は両者を `*pattern*` で一括 match していたため誤検知/見逃しが混在していた)

#### 残 55 件の内訳 (全て enforcement 層外)

- **assertion guards** (`assert.Empty(t, req.Header.Get("X-Service-Token"))` 等) — token 再侵入を検知する regression test
- **README documentation** — service ごとの README が legacy 仕様を説明する文中で言及
- **pact JSON (historical record)** — 過去の interaction を contract history として保持

**injection / enforcement / configuration loading の各層からは完全消去**されており、`grep -rn 'serviceTokenInterceptor\|ServiceAuthMiddleware\|resolve_service_secret' --include='*.{go,py,rs}' .` は 0 件を返す。

#### 副次的な cleanup

- **news-creator/app/news_creator/infra/peer_identity.py** — 当初 `structlog` を import していたが、news-creator は structlog 未依存だったため標準 `logging` モジュールに降格。peer identity middleware の構造ログ出力は logger 経由で同等の structured field を維持
