# ADR-000834: knowledge-sovereign の CI/CD を対称化し E2E gate と Pact CDC を同時に入れる

## Status

Accepted

## Date

2026-04-23

## Affected Services

- **knowledge-sovereign (Go)** — プロダクションコードは一切触らない。追加物は (a) `.github/workflows/knowledge-sovereign.yaml`、(b) `driver/contract/provider_test.go` (Pact provider 検証)、(c) `pact-go/v2 v2.4.3` の go.mod 追加、(d) `gofmt -w` による既存ファイルの空白整列のみ。
- **alt-backend (Go)** — `driver/sovereign_client/contract/consumer_test.go` を新設。`ApplyProjectionMutation` / `ApplyRecallMutation` / `ApplyCurationMutation` の wire contract を 4 interaction で固定。`sovereign_client/client.go` 等のランタイム側は無変更。
- **altctl (Go)** — `internal/sovereignclient/contract/consumer_test.go` を新設。admin REST (`/admin/snapshots/*`, `/admin/retention/*`, `/admin/storage/stats`) の PascalCase response shape を 5 interaction で固定。
- **services.yaml / c2quay.yml / docker-build.yaml / scripts/pact-check.sh** — knowledge-sovereign を `kind: runtime` から `kind: pacticipant` に昇格し、13 既存 pacticipant と並ぶ 14 本目として `can-i-deploy` gate に載せる。`scripts/regen-service-registry.py` の自動再生産物 3 ファイルも同時更新。
- **.github/workflows/e2e-hurl.yml** — 既存の 8 サービスジョブ (alt-backend / mq-hub / recap-worker / news-creator / acolyte-orchestrator / tag-generator / auth-hub / rag-orchestrator) に並ぶ 9 本目として `knowledge-sovereign` job を追加。

## Context

[[000532]] で確立したとおり knowledge-sovereign は Alt の durable knowledge state の single owner であり、`alt-backend` の Knowledge Loop / Home projector が Connect-RPC (`:9500`、49 RPC) で、`altctl home snapshot/retention/storage` が admin REST (`:9501`) で書き込む唯一の宛先である。加えて 11 本の Atlas migration と `knowledge_events` partition 運用を抱えている。ところが CI/CD 面では他の Go pacticipant と比べて非対称が 4 箇所残っていた:

1. **Go quality workflow が存在しない**。`backend-go.yaml` / `pre-processor-sidecar-go.yaml` のように `reusable-test-go.yaml` + `reusable-go-quality-gates.yaml` を呼ぶ `knowledge-sovereign.yaml` が無く、sovereign の Go 変更で `gofmt` / `go vet` / `golangci-lint v2.1` / `gosec` / `govulncheck` が PR で走っていない。49 RPC + 11 migration を抱える service が format / vuln gate 未装備という状態。

2. **Hurl E2E スイートが CI に繋がっていない**。[[000765]] で 18 scenario と `run.sh` と `compose.staging.yaml` profile までは整備したが、`.github/workflows/e2e-hurl.yml` 側に **job も path filter も無い**ため PR CI では一度も実行されていなかった (`e2e/hurl/knowledge-sovereign/` に既に 18 本 `.hurl` が居るのに)。同じフレームワークで後発の [[000791]] rag-orchestrator は Phase 1 段階で既に PR gate 化されている。

3. **Pact CDC がゼロ**。`services.yaml` で `kind: runtime`、`c2quay.yml` の 13 pacticipant に含まれず、`scripts/pact-check.sh` に登録なし。alt-backend の `sovereign_client/client.go` は `ApplyProjectionMutation` / `ApplyRecallMutation` / `ApplyCurationMutation` を Connect-RPC で連投しているが契約固定 0 本、altctl の admin REST (`SnapshotID` / `EventSeqBoundary` / `sha256:*` checksum 等の PascalCase shape) も同様。結果として「event store の single owner が消費者との contract を 1 本も持っていない」状態。[[000591]] の全面 CDC 展開方針とも乖離していた。

4. **alt-deploy の E2E matrix leg にも乗っていない**。`playbooks/run-e2e-suite.yml` lines 71-84 の pacticipants allowlist に knowledge-sovereign が無いため、release-deploy 側の E2E gate は短絡していた。

特に 3 は危険度が高い。append-first / reproject-safe / versioned artifacts という Alt の immutable invariant は provider 側実装では守られているが、**契約として broker に publish されていない**ため、将来 sovereign が `eventSeq` を 0 起点に変えたり `dedupe_key` の UNIQUE 制約を緩めたりしても consumer 側は検知できない。[[000749]] の tenant boundary も同様で、Hurl E2E は positive path のみ ([[000765]] Phase 1 方針) なので cross-tenant reject の回帰を追う経路が無い。

## Decision

**knowledge-sovereign の CI/CD 非対称を 1 チェーン (7 commit) で埋める**。production code は触らず、test / workflow / registry のみを追加する:

### 採用した構成

1. **Go quality workflow 追加** (`.github/workflows/knowledge-sovereign.yaml`)
   - `backend-go.yaml` をテンプレに、`reusable-test-go.yaml` (test job, sparse-checkout = `knowledge-sovereign`, coverage-flags = `knowledge-sovereign`) と `reusable-go-quality-gates.yaml` (quality job, `lint-only-new: true`, SARIF upload) を呼ぶ
   - path filter は `knowledge-sovereign/**` + 自身。他サービスの PR では cold build 費用を払わない
   - 既存 10 ファイル (`driver/sovereign_db/*.go`, `handler/*.go`) の `gofmt` drift を同時に解消。空白整列のみ、semantic 変更なし。これをやらないと新 gate が初回 PR で RED になり suppression へ誘導される。gate 追加時に grandfather する運用は採らない

2. **E2E Hurl PR gate** (`.github/workflows/e2e-hurl.yml`)
   - `rag-orchestrator` job (行 483-533) をテンプレに `knowledge-sovereign` job を追加 (行 539 以降)
   - `docker build -t ghcr.io/kaikei-e/alt-knowledge-sovereign:ci -f knowledge-sovereign/Dockerfile.sovereign knowledge-sovereign` で image を local build し、`bash e2e/hurl/knowledge-sovereign/run.sh` を `KEEP_STACK=1` で実行
   - 失敗時は profile-gated container (`alt-staging-knowledge-sovereign{,-db,-db-migrator}`) を `docker logs` で個別吸出し (profile-gated service は `docker compose logs` に素直に出ない)
   - `pull_request.paths` / `push.paths` の両方に `knowledge-sovereign/**` と `e2e/hurl/knowledge-sovereign/**` を追加

3. **Pact CDC — Connect-RPC consumer (alt-backend → sovereign)**
   - `alt-backend/app/driver/sovereign_client/contract/consumer_test.go` (`//go:build contract`)
   - 4 interaction:
     - `ApplyProjectionMutation` `upsert_home_item` happy (`success: true`)
     - `ApplyProjectionMutation` `dismiss_home_item` の projection version mismatch rejection (`success: false` + `errorMessage` non-empty)。append-first の reject 経路契約
     - `ApplyRecallMutation` `snooze_candidate` happy
     - `ApplyCurationMutation` `create_lens` happy
   - `mutationType` は literal string 一致 (`knowledge_sovereign_port.MutationUpsertHomeItem` 等の enum drift を検出)。他 field は `matchers.Like` で runtime value を自由に
   - pact は `alt-backend/pacts/alt-backend-knowledge-sovereign.json` に出る (既存 alt-backend consumer pact convention と揃える)

4. **Pact CDC — admin REST consumer (altctl → sovereign)**
   - `altctl/internal/sovereignclient/contract/consumer_test.go` (`//go:build contract`)
   - 5 interaction で disaster recovery 経路を固定:
     - `POST /admin/snapshots/create`
     - `GET /admin/snapshots/latest`
     - `GET /admin/snapshots/list`
     - `GET /admin/retention/eligible`
     - `POST /admin/retention/run` (`dry_run: true`)
   - `SnapshotID` は UUIDv4 regex、`EventSeqBoundary` は `>= 1` assertion (append-first invariant 反映)、`ItemsChecksum` / `VersionsChecksum` / `DedupesChecksum` は `^sha256:[0-9a-f]{64}$` regex で pin
   - pact は `altctl/pacts/altctl-knowledge-sovereign.json` に出る (`scripts/pact-check.sh` の publish scan 対象に `altctl/pacts/*.json` を追加)

5. **Pact CDC — provider verification (sovereign side)**
   - `knowledge-sovereign/app/driver/contract/provider_test.go` (`//go:build contract`)
   - `alt-backend/app/driver/contract/provider_test.go` の stub-server pattern を踏襲。ephemeral listener 1 つで Connect-RPC と admin REST 両方をハンドル。Postgres は立てない (契約は wire shape だけの責務分離)
   - 2 検証 test: `TestVerifyAltBackendConsumerContract` + `TestVerifyAltctlConsumerContract`
   - Consumer pact の resolver は 2 候補パスを順番に探し、file mode (local) と broker mode (`PACT_BROKER_BASE_URL` 経由) を両対応
   - Broker mode では `ConsumerVersionSelectors: MainBranch=true + DeployedOrReleased=true`、`EnablePending=true` (WIP pact を落とさない)、`IncludeWIPPactsSince` 対応

6. **pacticipant 昇格とレジストリ同期**
   - `services.yaml` の `knowledge-sovereign` を `kind: runtime` → `kind: pacticipant` に移し、`compose_files: [sovereign.yaml]` を付与
   - `python3 scripts/regen-service-registry.py --write` を実行して 3 生成物を同期:
     - `c2quay.yml` environment `production.services` に `knowledge-sovereign: { pacticipant: knowledge-sovereign }` を追加 (13 → 14 pacticipant)
     - `.github/workflows/docker-build.yaml` SERVICES 配列で位置を pacticipant block に移動 (build 挙動は無変更)
     - `.github/generated/alt-deploy-service-paths.js` snippet を更新 (alt-deploy 側への反映は別 PR)
   - `scripts/pact-check.sh` の `STEPS_CONSUMER` に 2 行 (alt-backend sovereign consumer / altctl sovereign consumer)、`STEPS_PROVIDER` に 1 行 (knowledge-sovereign provider) を追加
   - `FILES_TO_PUBLISH` の publish scan ループに `altctl/pacts/*.json` を追加

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| Hurl E2E は入れるが Pact CDC は別 ADR に繰り延べ | Hurl は positive path のみ ([[000765]] Phase 1)。append-first / single-emission / tenant boundary の negative contract を守る経路が残らず、[[000591]] の CDC 全面展開方針に逆行する。E2E と CDC は責務が違うので同時投入が正 |
| provider 検証で実 Postgres + Atlas migration を立てる | pact-go v2 は wire shape の検証が本務。Postgres を CI で立てると testcontainers-go 依存 + migration apply の時間コストが乗り、`scripts/pact-check.sh` の file mode で完結できなくなる。`alt-backend/app/driver/contract/provider_test.go` 既存パターン (stub server) に揃える方が保守コストが低い |
| 既存 gofmt 違反 10 ファイルは触らず suppress 追加 | `feedback_plan_audit` / `feedback_ci_parity_local` に反する。空白整列のみ (semantic 不変) なので 1 コマンドで解消でき、技術的負債を増やさない |
| altctl を c2quay.yml の deploy matrix 13 pacticipant に追加 | altctl は CLI バイナリで ghcr image 化していない。all-or-nothing の deploy gate に入れる意味がない。pact だけ broker に publish すれば `can-i-deploy` 側は自動で対応 (pacticipant 自動登録) |
| Knowledge Loop 5 RPC ([[000831]]) も同時 CDC 化 | [[000831]] は `proposed` 段階。accepted 後に別 ADR として追加する方が review 粒度が揃う |
| `WatchProjectorEvents` (server streaming) を含める | Hurl も pact-go v2 も unary のみ対応。streaming は別途 buf conformance 等で扱う必要あり。本 ADR スコープ外 |
| `atlas migrate lint --latest 1` を PR gate に追加 | 本 ADR は「ランタイムコード無変更で CI gate を整える」に絞る。migration lint は別 ADR で横展開 |

### 範囲外 (別 ADR / 別 PR)

- **alt-deploy 側の allowlist 更新** — `playbooks/run-e2e-suite.yml` lines 71-84 の pacticipants allowlist に `knowledge-sovereign` を足す PR が private repo 側で必要。本 ADR は public 側 prep までで閉じる
- Knowledge Loop CDC ([[000831]] accepted 後)
- E2E negative path ([[000765]] Phase 2)
- `scripts/smoke.sh` に sovereign `/health` probe を追加 (alt-deploy 側別 PR)
- Atlas migration lint の PR gate 化

## Consequences

### Pros

- **append-first / reproject-safe / versioned artifacts の不変条件が broker 契約として固定された**。sovereign 側が `eventSeq` を 0 起点に変えたり `SnapshotID` UUID 形式を変えたりしても、consumer pact の regex assertion で PR 時に検出される
- **disaster recovery 経路 (admin REST snapshot/retention) が回帰網に入った**。`altctl home snapshot create/list/latest` / `altctl home retention eligible/run --dry-run` の JSON shape drift が contract 違反として即 RED 化される。災害対応中に CLI が暗黙に壊れるリスクが落ちた
- **CI 非対称が 4 箇所すべて解消**。Go unit / lint / vuln (gate 1) / E2E Hurl (gate 2) / Pact CDC (gate 3) / broker can-i-deploy (gate 4) の 4 段で knowledge-sovereign が他 Go pacticipant と同じ扱いになった
- **既存パターンに完全準拠**。alt-backend の consumer test / provider stub / pact directory convention / regen script / `scripts/pact-check.sh` registry / [[000763]] family の Hurl job テンプレ全てにそのまま乗る。新規発明ゼロ
- **ローカル完走が `./scripts/pact-check.sh --services knowledge-sovereign,alt-backend,altctl` で green**。5 pass / 0 fail / 16 skip (他 pacticipant)、consumer + provider を壊さず追加できることを ADR 書き下しより前に確認済み

### Cons / Tradeoffs

- **alt-deploy 側の allowlist 更新が間に合うまでは release-deploy の E2E matrix leg は knowledge-sovereign を skip する**。public repo のみの変更では deploy gate が完全対称にならない。本 ADR のフォロー PR (private) で埋める
- **gofmt cleanup 10 ファイルが混在する**。本 ADR では style commit を 1 本独立させたが、review 時に「なぜ今?」という疑問が湧く。commit message + 本 ADR 本文で「新 gate の RED 回避 + suppression 忌避」と説明
- **pact-go v2 は Connect-RPC の streaming RPC を扱えない**。`WatchProjectorEvents` (server streaming) は本 ADR の CDC では守れず、当面は unit test のみ。別 ADR で buf conformance 等の枠組みを立てる必要
- **provider stub は Postgres を立てないため、DB FK / dedupe constraint 違反は E2E 側でしか捉えられない**。これは責務分離としては正しいが、「Pact が green でも DB 挙動が壊れている」パターンは残る。Hurl E2E との役割分担を運用で維持する必要
- **`only-new-issues: true` で pre-existing な golangci-lint 違反 17 件が猶予される**。新 gate が追加した日以降の新規違反だけを止める運用。grandfathered 違反の返済は別 task として棚上げ
- **pacticipant 昇格により c2quay の all-or-nothing deploy gate の対象サービスが 13 → 14 本に増える**。sovereign の contract が壊れると他 13 本も一緒にブロックされる。これは正の property (contract drift の早期検知) だが、incident response 中に圧が上がる可能性。[[pact-broker-ops]] runbook の復旧経路を運用が把握している前提
