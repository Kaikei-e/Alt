# ADR-000387: 要約ジョブキューの3重障害を修正しバックログ4Mを解消

## Status

Accepted

## Date

2026-03-14

## Affected Services

- pre-processor (Go) — `CreateJob` 冪等化、`RecoverStuckJobs` 追加、キュー処理ループへの定期リカバリ統合
- pre-processor-db (PostgreSQL) — `dead_letter` CHECK 制約マイグレーション追加、バックログ一括クリーンアップ

## Context

要約処理が「詰まっている」状態が継続していた。調査の結果、以下の3つの根本原因が複合的に発生していたことが判明した。

### 観測データ（修正前）

| 指標 | 値 |
|------|-----|
| pending ジョブ | **4,100,411** |
| running ジョブ（stuck） | **14** |
| completed ジョブ | 89,019 |
| ユニーク article_id (pending) | 1,905,169 |
| 重複最大回数 | 895回（同一 article_id） |
| 処理速度 | ~400件/時 |
| バックログ消化予測 | **~400日** |

### 原因1: `dead_letter` マイグレーション未適用（CRITICAL）

ADR-000314 で `dead_letter` ステータスを CHECK 制約に追加するマイグレーション（`20260304000000_add_dead_letter_status_to_summarize_job_queue.sql`）を作成済みだったが、**pre-processor-db には適用されていなかった**。

ADR-000246 で `pre-processor-db` が `alt-db` から分離された際、マイグレーション適用先が変わったことが原因。Atlas マイグレーションは `alt-db` に対して適用されたが、`pre-processor-db` 側の `summarize_job_queue` テーブルには旧制約（`pending`, `running`, `completed`, `failed`のみ）が残っていた。

結果として、`retry_count >= max_retries` に達しても `dead_letter` に遷移できず、ジョブが `failed → pending` のループに永遠にハマり、SQLSTATE 23514 エラーが継続的に発生していた。

### 原因2: `CreateJob` に重複チェックがない（HIGH）

`repository/summarize_job_repository.go` の `CreateJob` が無条件に `INSERT` していた。同一 `article_id` で `pending`/`running` のジョブが既に存在しても新規投入される。

重複投入の経路:
1. `consumer/event_handler.go` — `ArticleCreated` / `SummarizeRequested` イベントごと
2. `handler/summarize_handler.go` — HTTP API 経由

article-sync（1時間ごと）がフィードを収集し、記事ごとにイベントを発行するため、既に要約済みの記事にもジョブが投入され、1記事あたり最大895件の重複が発生していた。

### 原因3: running ジョブのリカバリ不在（HIGH）

ワーカーがクラッシュ/再起動すると `running` のまま放置されるジョブが発生。`GetPendingJobs` は `WHERE status = 'pending'` のみ取得するため、stuck running ジョブは永遠に処理されない。`FOR UPDATE SKIP LOCKED` によるロック管理はあるが、`running` 遷移後のクリーンアップ機構がなかった。

## Decision

以下の3層修正を採用する。

### 1. `CreateJob` の冪等化

`INSERT ... SELECT ... WHERE NOT EXISTS` パターンを採用し、同一 `article_id` で `pending` または `running` のジョブが既に存在する場合は INSERT をスキップする。

```sql
INSERT INTO summarize_job_queue (article_id, status)
SELECT $1, 'pending'
WHERE NOT EXISTS (
    SELECT 1 FROM summarize_job_queue
    WHERE article_id = $1
      AND status IN ('pending', 'running')
)
RETURNING job_id
```

検討した代替案:
- **partial unique index + `ON CONFLICT DO NOTHING`**: より強い一貫性保証だが、DB マイグレーションが必要。アプリケーション層での制御で十分であり、マイグレーション追加を避けるため見送り
- **アプリケーション層での事前チェック（SELECT → INSERT）**: TOCTOU 競合が発生する。`WHERE NOT EXISTS` なら単一文で原子的

### 2. `RecoverStuckJobs` の定期実行

`SummarizeJobRepository` に `RecoverStuckJobs` メソッドを追加。`running` 状態で `started_at` が10分以上前のジョブを `pending` にリセットする。

`SummarizeQueueWorker.ProcessQueue()` の先頭で呼び出すが、5分間のスロットリングを設け、DB負荷を抑える。ProcessQueue 自体は10秒間隔で実行されるため、リカバリは実質5分に1回。

検討した代替案:
- **独立した goroutine / cron ジョブ**: リカバリの実行タイミングが ProcessQueue と独立するため、リカバリ直後に処理が走る保証がない。ProcessQueue の先頭で呼ぶことで、リカバリ→即座に処理という流れが自然に実現
- **PostgreSQL の advisory lock + scheduled function**: DB 側で完結するが、アプリケーションのログ・メトリクスとの統合が困難

### 3. `dead_letter` CHECK 制約の pre-processor-db マイグレーション追加

ADR-000314 のマイグレーションは `migrations-atlas/`（alt-db 用）にのみ存在し、`pre-processor-migration-atlas/`（pre-processor-db 用）には含まれていなかった。`pre-processor-migration-atlas/migrations/20260314000001_add_dead_letter_status.sql` を新規追加し、`schema.hcl` の CHECK 制約定義も更新した。Atlas migrator によるコンテナ起動時の自動適用で、今後の再デプロイ・新規環境構築でも確実に適用される。

### 4. バックログの一括クリーンアップ

以下の順序で実行:
1. completed 記事の重複 pending ジョブを削除 → **1,056,181件**
2. 重複 pending ジョブの削除（各 article_id につき最新1件だけ残す） → **1,138,630件**
3. stuck running → pending リセット → 0件（再ビルド時に処理済み）
4. リトライ上限到達ジョブを dead_letter へ → 0件（該当なし）

## Consequences

### Pros

- **バックログ53%削減**: 4,100,411 → 1,905,546 pending（2,194,811件除去）
- **重複ゼロ達成**: unique_articles = pending count（1:1完全一致）
- **dead_letter 正常動作**: 23514 エラー完全解消。"content too short" 等の非リトライ可能エラーが正しく dead_letter に遷移
- **将来の重複蓄積を防止**: CreateJob の冪等化により、article-sync の再実行やイベント再送で重複が増えない
- **stuck running の自動復旧**: ワーカー再起動時にジョブが失われない。5分ごとに自動検知・リカバリ

### Cons / Tradeoffs

- **CreateJob の冪等化はアプリケーション層のみ**: DB レベルの partial unique index ではないため、理論上は高並行時に race condition で重複する可能性がある。ただし、現在の投入パターン（イベント駆動 + HTTP API）では同一 article_id への同時投入は極めて稀であり、実用上問題ない
- **RecoverStuckJobs の10分閾値**: 正当に10分以上かかるジョブがあった場合、誤ってリセットされる。現在の Ollama バックエンドでの処理時間は平均9秒、最大でも60秒程度のため、10分は十分な安全マージン
- **バックログ1.9M は依然として残存**: 処理速度 ~400件/時では約200日かかる。これはバックエンドの処理能力（Ollama スロット数）の問題であり、本 ADR のスコープ外
- **pre-processor-db のマイグレーション管理**: Atlas マイグレーションが alt-db と pre-processor-db で分離されているにもかかわらず、マイグレーションファイルが共有ディレクトリにある。今後同様の適用漏れが発生するリスクがある

### 観測データ（修正後）

| 指標 | Before | After | 改善 |
|------|--------|-------|------|
| pending ジョブ | 4,100,411 | 1,905,546 | -53% |
| 重複率 | 2.15x (4.1M/1.9M unique) | 1.0x (重複なし) | 完全解消 |
| dead_letter | 0 (制約エラーで遷移不可) | 16+ (正常遷移中) | 正常化 |
| 23514 エラー | 継続的に発生 | 0 | 完全解消 |
