# PgBouncer導入によるPostgreSQLコネクションプール最適化

## ステータス

Accepted

## コンテキスト

3000VU負荷テストにおいて、alt-backendの `DB_MAX_CONNS=500`、PostgreSQLの `max_connections=1000`（runtime override）が必要になっていた。PostgreSQLの推奨同時接続数は `(CPU cores × 2) + spindle_count` ≒ 10〜20本であり、500本超の接続はコンテキストスイッチコストとメモリ消費（接続あたり約10MB）によりスループットを**下げる**。

### コード精査による事実確認

レビューで「外部I/O中にDB接続を保持している」との指摘があったが、コードを精査した結果、以下の事実が確認された:

- `RegisterRSSFeedLink`フロー: HTTP fetchとDB操作は完全に分離。各DB操作は独立してコネクションを取得・返却している
- Usecase全体フロー: 各ステップ（RegisterRSSFeedLink → FetchFeedLinkIDByURL → FetchFeeds → RegisterFeeds → publishFeedEvents → autoSubscribeUser）で、DB操作は短いトランザクション（<50ms）のみ

**真のボトルネック**: pgxpoolの `MinConns=100` で常時100本確保、`MaxConns=500` でバースト時に最大500本の実接続が可能になっていたこと。

### DBトポロジ

| DB | 利用サービス | 問題 |
|---|---|---|
| `db` (main) | alt-backend, pre-processor | MaxConns=500で接続過多 |
| `kratos-db` | kratos x1〜3 | 3レプリカのramp-up時にtimeout |

## 決定事項

PgBouncer（transaction poolingモード）を `db` および `kratos-db` の前段に導入し、アプリケーションからのコネクション数をPgBouncer内で多重化する。これによりPostgreSQLの実接続数を大幅に削減する。

### 接続トポロジ

```
alt-backend (MaxConns=30) → pgbouncer:6432 → db:5432
kratos (max_conns=30) → pgbouncer-kratos:6432 → kratos-db:5432

# PgBouncerバイパス（DDL/session-level lock）
migrate → db:5432 (直接)
kratos-migrate → kratos-db:5432 (直接)
pre-processor → db:5432 (直接, MaxConns=20)
```

## 実装の概要

### 1. PgBouncer Composeサービス (`compose/pgbouncer.yaml`)

- `pgbouncer`: main DB用。`DEFAULT_POOL_SIZE=30`, `MAX_DB_CONNECTIONS=100`
- `pgbouncer-kratos`: kratos-db用。`DEFAULT_POOL_SIZE=20`, `MAX_DB_CONNECTIONS=60`, `MAX_PREPARED_STATEMENTS=256`（kratosのprepared statement対応）
- 共通設定: `POOL_MODE=transaction`, `LISTEN_PORT=6432`, `AUTH_TYPE=plain`, `SERVER_RESET_QUERY=DISCARD ALL`
- イメージ: `edoburu/pgbouncer:1.25.1`

### 2. pgx SimpleProtocol (`alt-backend/app/driver/alt_db/init.go`)

```go
config.ConnConfig.DefaultQueryExecMode = pgx.QueryExecModeSimpleProtocol
```

pgx v5はデフォルトで拡張プロトコル+prepared statementキャッシュを使用する。PgBouncer transaction poolingではトランザクション毎にバックエンド接続が変わるため、prepared statementが不在エラーになる。`SimpleProtocol`はSQL全文送信でこれを回避する。

### 3. Kratos DSN条件分岐 (`kratos/entrypoint.sh`)

PgBouncer経由（port 6432）の場合のみ `prepared_statements=false` をDSNに付与。直接接続（port 5432）の場合は付与しない。PostgreSQL自体は `prepared_statements` パラメータを認識しないため、直接接続時に付与するとFATALエラーになる。

pgbouncer-kratosでは `MAX_PREPARED_STATEMENTS=256` を設定し、PgBouncer 1.21+のprepared statement tracking機能でtransaction pooling下でもprepared statementを利用可能にした。

### 4. PostgreSQL設定最適化 (`docker/postgres/postgresql.conf`)

- `shared_buffers`: 128MB → 256MB
- `work_mem`: 16MB（新規追加）
- `effective_cache_size`: 1536MB（新規追加）
- `max_connections=200` は据え置き

### 5. 負荷テスト設定 (`alt-perf/scripts/run-feed-load-test.sh`)

- DB: `max_connections=250`（1000から削減）
- PgBouncer: `DEFAULT_POOL_SIZE=50`, `MAX_DB_CONNECTIONS=130`
- alt-backend: `DB_MAX_CONNS=100`（500から削減）

### 6. マイグレーションのPgBouncerバイパス

DDLやsession-level lockが必要なマイグレーションはPgBouncerを経由せず直接DBに接続:

- `migrate`（Atlas）: `DB_HOST=db`, `DB_PORT=5432`（変更なし）
- `kratos-migrate`: `KRATOS_DB_HOST=kratos-db`, `KRATOS_DB_PORT=5432`（ハードコード）

## 結果・効果

### メリット

- **PG実接続数の大幅削減**: db 520本→~53本、kratos-db 150本→~20本
- **メモリ使用量削減**: 接続あたり約10MBとして、db側で約4.7GB削減
- **max_connections=200で運用可能**: PostgreSQLの推奨範囲内に収まる
- **負荷テストの安定化**: kratos-db timeoutの解消が期待される
- **PgBouncer単体のメモリフットプリント**: 256MB + 128MB = 384MBと軽量

### デメリット・トレードオフ

- **PgBouncer単一障害点**: 障害時は `DB_HOST` を直接DB名に切り替えてフォールバック可能
- **SimpleProtocol**: prepared statementキャッシュによる性能最適化が無効になる。ただし、短いクエリ（<50ms）が主であり影響は限定的
- **AUTH_TYPE=plain**: PgBouncerのクライアント認証が平文比較。Docker内部ネットワークのみでの使用のため許容。本番ではSCRAM対応を検討
- **pgbouncer-kratosのMAX_PREPARED_STATEMENTS=256**: PgBouncer側でprepared statementをトラッキングするメモリオーバーヘッド。kratos-dbの負荷では問題にならない規模
- **運用複雑性の増加**: PgBouncerの監視・管理が必要（`SHOW POOLS`、`SHOW STATS`）
