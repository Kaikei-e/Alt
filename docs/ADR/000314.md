# ADR-000314: summarize_job_queue CHECK 制約に dead_letter ステータスを追加

## ADR's STATUS

完了

## CONTEXT

本番環境で pre-processor が要約ジョブを `dead_letter` ステータスに更新しようとした際、PostgreSQL の CHECK 制約違反（SQLSTATE 23514）が 230 回発生していた。

### 発生していたエラー

```
ERROR: new row for relation "summarize_job_queue" violates check constraint "summarize_job_queue_status_check" (SQLSTATE 23514)
```

### 根本原因

`20251213000000_create_summarize_job_queue.sql` で定義された CHECK 制約が、`('pending', 'running', 'completed', 'failed')` の 4 ステータスのみを許可していた。一方、Go コードでは `dead_letter` ステータスが以下の箇所で使用されている:

| ファイル | 用途 |
|---------|------|
| `pre-processor/app/domain/job.go` | `SummarizeJobStatusDeadLetter` 定数定義 |
| `pre-processor/app/repository/summarize_job_repository.go` | `retry_count + 1 >= max_retries` 時に SQL で `dead_letter` へ遷移 |
| `pre-processor/app/service/summarize_queue_worker.go` | リトライ不可エラー時に `dead_letter` へ直接遷移 |

つまり、アプリケーション層では `dead_letter` への遷移ロジックが実装済みだったが、DB スキーマ側の制約が追従していなかった。

### ジョブステータス遷移図

```
pending → running → completed
              ↓
           failed → pending (retry_count < max_retries)
              ↓
         dead_letter (retry_count >= max_retries、またはリトライ不可エラー)
```

## DECISION MAKING

### 方針: ALTER TABLE による CHECK 制約の再定義

既存の CHECK 制約を DROP し、`dead_letter` を含む新しい制約を再作成する。Atlas マイグレーションで管理し、既存データに影響を与えない。

### マイグレーション

ファイル: `migrations-atlas/migrations/20260304000000_add_dead_letter_status_to_summarize_job_queue.sql`

```sql
ALTER TABLE summarize_job_queue
    DROP CONSTRAINT IF EXISTS summarize_job_queue_status_check;

ALTER TABLE summarize_job_queue
    ADD CONSTRAINT summarize_job_queue_status_check
    CHECK (status IN ('pending', 'running', 'completed', 'failed', 'dead_letter'));
```

`DROP CONSTRAINT IF EXISTS` により、冪等性を確保している。

### 適用手順

1. Atlas ハッシュ再計算（`atlas migrate hash`）
2. マイグレーション適用（`docker compose up migrate`）
3. スタックしたジョブの救済（`failed` で `retry_count >= max_retries` のジョブを `dead_letter` へ、`running` で放置されたジョブを `pending` へリセット）
4. pre-processor コンテナの再ビルド・再起動
5. ログ監視で制約違反エラーが消失したことを確認

## RESULTS, EFFECTS

### 適用結果

| 項目 | 結果 |
|------|------|
| マイグレーション | 1 migration, 3 SQL statements 正常適用 |
| CHECK 制約 | `('pending', 'running', 'completed', 'failed', 'dead_letter')` に更新済み |
| スタックしたジョブ | 0 件（適用時点で `failed` かつ `retry_count >= max_retries` のジョブなし） |
| ジョブ分布 | completed: 200, dead_letter: 35 |
| 制約違反エラー | 適用後 0 件 |

### PROS

- `dead_letter` ステータスが DB に正常保存され、リトライ上限到達ジョブが適切に終端処理される
- 制約違反による 230 回のエラーログが解消され、ログの SN 比が改善
- リトライ不可エラー（HTTP 422 等）時の即時 `dead_letter` 遷移が正常に機能する
- `DROP CONSTRAINT IF EXISTS` による冪等なマイグレーションで、再実行時にもエラーが発生しない

### CONS, TRADEOFF

- `dead_letter` ジョブに対する運用フロー（通知、手動調査、再投入）は未整備。現時点では DB 上のステータスとして記録されるのみ
- CHECK 制約の変更はテーブルロックを伴うが、`summarize_job_queue` は行数が少ないため実質的な影響はない

### 検証コマンド

```bash
# マイグレーション適用
docker compose -f compose/compose.yaml -p alt up migrate

# 制約確認
docker compose -f compose/compose.yaml -p alt exec -T db psql -U <db_user> -d <db_name> \
  -c "SELECT conname, pg_get_constraintdef(oid) FROM pg_constraint WHERE conname = 'summarize_job_queue_status_check';"

# ジョブ状態確認
docker compose -f compose/compose.yaml -p alt exec -T db psql -U <db_user> -d <db_name> \
  -c "SELECT status, count(*) FROM summarize_job_queue GROUP BY status ORDER BY status;"

# pre-processor 再ビルド・再起動
docker compose -f compose/compose.yaml -p alt up -d --build pre-processor

# エラー確認
docker compose -f compose/compose.yaml -p alt logs pre-processor --since 5m | grep -i "status_check\|23514"
```
