# ADR-000541: Server Streaming RPC による projector 通知の database-per-service 対応

## Status

Accepted

## Date

2026-03-23

## Affected Services

- alt-backend (Go)
- knowledge-sovereign (Go)

## Context

[[000539]] で alt-backend の全 read/write を sovereign RPC 経由に切り替えた後、**knowledge projector が停止する障害**が発生した。

### 障害の発見経緯

- Knowledge Home で新しい要約が反映されなくなった
- ログに `"processing knowledge events"` が一切出力されていなかった
- エラーログも出ていない — **サイレントに停止**

### 根本原因

projector runner は PostgreSQL の `LISTEN/NOTIFY` で events テーブルの INSERT を検知して起動する。`LISTEN` 接続先は **alt-db** にハードコードされていた。

sovereign 分離後の状態:
- `knowledge_events` テーブルは **sovereign-db にのみ存在**
- alt-db への `LISTEN` は成功する（DB 自体は存在する）が、events テーブルへの INSERT が来ないため NOTIFY は永遠に発火しない
- runner の poll fallback は、listener 接続が成功している場合は有効にならない
- **結果: projector は永遠に notification を待ち続けて停止**

この問題は database-per-service パターンの典型的な落とし穴: 物理分離時に **データアクセス** は API 経由に切り替えても、**通知経路** は DB 直接接続のまま残ることがある。

## Decision

**sovereign が自身の DB で `LISTEN` し、Connect-RPC Server Streaming RPC 経由で alt-backend に通知を push するアーキテクチャを採用する。**

### 検討した代替案

| 案 | 評価 | 不採用理由 |
|----|------|-----------|
| Polling only (5s interval) | 簡易 | 98.5% の poll が無駄（[参考](https://medium.com/@stoic.engineer/webhooks-vs-polling-vs-streaming-choosing-the-right-event-communication-pattern-c7eab10c7191)）。根本解決ではなくワークアラウンド |
| alt-backend → sovereign-db 直接 LISTEN | 低遅延 | database-per-service 原則に違反。sovereign の DB 境界を破壊する |
| Webhook（sovereign → alt-backend HTTP POST） | Push 型 | alt-backend に webhook endpoint が必要。既存 runner ループとの統合が複雑 |
| MQ-Hub 経由 | 疎結合 | Connect-RPC で十分な場合に不要なインフラ追加 |
| **Server Streaming RPC**（採用） | Push 型 + DB 境界遵守 | Connect-RPC は既存インフラ。sovereign が自 DB の owner として LISTEN する権利を持つ |

参考: [gRPC Long-lived Streaming](https://dev.bitolog.com/grpc-long-lived-streaming/), [gRPC Keepalive](https://grpc.io/docs/guides/keepalive/)

### 実装

#### sovereign 側

1. **NOTIFY trigger** (`00004_add_projector_notify_trigger.sql`): `knowledge_events` テーブルへの INSERT 時に `pg_notify('knowledge_projector', event_seq)` を発火
2. **`WatchProjectorEvents` streaming RPC** (`rpc_watch.go`): sovereign-db に `LISTEN` → notification を受信 → stream に push。heartbeat interval 3s で接続を維持
3. handler に `WithDatabaseURL` option を追加し、LISTEN 用の dedicated pgx connection を作成

#### alt-backend 側

4. **`ProjectorListener`** (`watch_client.go`): channel-based pump パターンで実装

```
pump goroutine:          for stream.Receive() { notify <- struct{}{} }
WaitForNotification:     select { case <-ctx.Done(): ... case <-notify: ... case err := <-errCh: ... }
```

- pump goroutine は listener 生成時に **1 つだけ** 起動（goroutine リーク防止）
- `notify` channel は buffered(1) で、runner が消費前の通知は drop（最新のみ）
- 既存の `KnowledgeProjectorListener` interface と `KnowledgeProjectorRunner` を **一切変更しない**

5. **streaming 専用 HTTP client**: unary RPC 用の `http.Client` はグローバル Timeout が設定されている。streaming は long-lived なのでタイムアウトなしの別 client を作成

6. **`NewSovereignProjectorListenerFactory`** (`knowledge_projector_runner.go`): runner が listener を生成する際に sovereign streaming に接続する factory

### 実装中に踏んだ落とし穴

| 問題 | 原因 | 解決 |
|------|------|------|
| streaming が 30s で切断 | unary 用 `http.Client.Timeout: 30s` が streaming を kill | streaming 専用の Timeout なし client を分離 |
| runner の context timeout と `Receive()` が不整合 | `Receive()` は context を無視してブロック。goroutine per-call でラップするとリーク | channel-based pump goroutine（1 つだけ）+ `select` |
| heartbeat が runner timeout 内に届かない | server heartbeat 30s > runner poll 5s → 常に DeadlineExceeded | heartbeat を 3s に短縮（poll interval より短く） |

## Consequences

### Pros

- projector が sovereign-db の events を検知可能になり、要約パイプラインが復旧
- database-per-service 原則を完全遵守 — alt-backend は sovereign-db に一切直接接続しない
- ほぼゼロ遅延の通知（pg NOTIFY は即座、streaming push もサブミリ秒）
- runner の既存 poll fallback が streaming 障害時の安全網として機能（listener factory がエラーを返すと自動的に polling に切り替わる）

### Cons / Tradeoffs

- sovereign に long-lived streaming 接続が 1 本追加（リソース消費は軽微）
- NOTIFY trigger が sovereign-db の INSERT パスに 1 関数呼び出しを追加
- streaming 接続の再接続ロジックは runner の listener recreation に依存（専用の reconnect with backoff は未実装）
