# ADR-000515: summary 保存済みの stale pending job を監査して除去する

## Status

Accepted

## Date

2026-03-21

## Affected Services

- pre-processor (Go)

## Context

`pending` は本来 live backlog だが、実運用では「既に summary が保存されているにもかかわらず `pending` に残っている row」が混在していた。
この状態では `pending` 件数が真の未処理量より大きく見え、throughput の評価や ETA の見積もりを誤らせる。

監査の結果、古い `pending` の大半は article 自体は存在しており、しかも summary も既に存在していた。
つまりそれらは「これから処理すべき仕事」ではなく、**過去の不整合や再投入の結果として残った stale row** だった。

一方で、古い `pending` の中には summary がまだ存在しないものも残っていた。
そのため、古い `pending` を年齢だけで一括削除するのではなく、**既要約かどうか** で安全に分類する必要があった。

## Decision

**古い `pending` を監査し、summary が既に保存されている stale row だけを除去する。**

具体的には以下を採用する。

1. 古い `pending` を監査対象として抽出する
2. article の存在有無と summary の存在有無を確認する
3. article が存在し、summary も存在する `pending` は stale row と見なして削除対象にする
4. summary が存在しない `pending` は処理対象として残し、別途原因調査する
5. 年齢だけで `pending` を一括 purge しない

この決定により、`pending` 件数を真の未処理 workload に近づけつつ、未要約記事の取りこぼしを防ぐ。

### 採用しなかった代替案

| 案 | 評価 |
|---|---|
| 古い `pending` を年齢だけで削除する | 未要約記事まで失う危険があり、データ欠落を招く |
| `pending` を一切消さずに放置する | stale row が backlog 指標を汚染し続ける |
| stale 判定を article の存在確認だけで行う | article が存在しても summary 済みか未要約かが分からず、判定として不十分 |

## Consequences

### Pros

- `pending` 件数が実 workload に近づく
- 既に要約済みの記事を backlog として数え続ける状態を解消できる
- 残った古い `pending` を「本当に調べるべき仕事」に絞り込める

### Cons / Tradeoffs

- 手動監査や補助スクリプトなしでは cleanup がしにくい
- stale 判定時点と worker 実行が競合しうるため、削除件数は事前件数と一致しないことがある
- summary が存在することを stale 判定に使うため、summary 保存先との整合確認が前提になる
