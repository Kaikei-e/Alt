# ADR-000629: Admin Dashboard に System Observability タブと実 SLI メトリクスを追加

## Status

Accepted

## Date

2026-04-06

## Affected Services

- alt-backend (Go) — 新 RPC `GetSystemMetrics`、メトリクススナップショット基盤、ヘルスチェッカードライバー
- alt-frontend-sv (TypeScript/Svelte) — 新 System タブ、8 新規コンポーネント、SLO タブ強化
- alt-butterfly-facade (Go) — BFF 透過プロキシ（変更なし、新 RPC は自動通過）

## Context

Knowledge Home Admin ダッシュボードは 5 タブ（Overview / SLO / Reproject / Storage / Audit）を持っていたが、以下の課題があった：

1. **SLI プレースホルダー問題**: 5 SLI 中 4 つ（availability, action_durability, stream_continuity, correctness_proxy）がハードコード 100% のプレースホルダーで、実際のシステム健全性を反映していなかった。freshness のみが sovereign 経由の projection lag から実値を取得していた。

2. **OTel メトリクスの未活用**: alt-backend には 30+ の OTel カウンター/ゲージ/ヒストグラムが計装済み（projector, handler, tracking, stream, correctness, sovereign, recall）だが、Prometheus `/metrics` エンドポイント経由でしかアクセスできず、Admin UI には露出していなかった。

3. **システム深部の不可視性**: サービス間依存関係の健全性、プロジェクターパイプラインのスループット、ストリーム接続状態、Recall パイプラインの処理状況などが Admin UI から確認できなかった。

4. **未配線コンポーネント**: `InteractionFunnelPanel` と `ReasonDistributionChart` が実装済みだったが、どのタブにも配線されていなかった。

## Decision

全レイヤー（proto → Backend → BFF → Frontend）を通して、Admin ダッシュボードにシステム深部の観測機能を追加する。Desktop UI のみを対象とし、Mobile 対応は含めない。

### アーキテクチャ方針

**OTel メトリクスの読み出し**: OTel SDK のカウンター/ゲージは write-only インターフェースであり、プログラム的な読み出し API を持たない。ManualReader や Prometheus スクレイピングの代わりに、**MetricsSnapshot パターン**を採用する。OTel instruments への `.Add()` / `.Record()` 呼び出しと同時に、atomic カウンターベースの `MetricsSnapshot` 構造体にも値を書き込む。Admin API はこのスナップショットから値を読む。

**サービスヘルスチェック**: `health_checker` ドライバーが下流サービスの `/health` エンドポイントを並行呼び出し（3 秒タイムアウト）。環境変数未設定時はコンテナ内デフォルト URL にフォールバック。

### Proto 拡張

`knowledge_home_admin.proto` に `GetSystemMetrics` RPC と 8 メッセージ型を additive に追加（breaking change なし）:

- `ProjectorMetrics` — イベント処理数、ラグ、バッチ処理時間 P50/P95/P99、エラー数
- `HandlerMetrics` — ページ配信数、degraded 率
- `TrackingMetrics` — 露出/開封/却下数、開封率/却下率
- `StreamMetrics` — 接続/切断/再接続/配信数、切断率
- `CorrectnessMetrics` — 空レスポンス/malformed why/orphan/supersede mismatch、正確性スコア
- `SovereignMetrics` — ミューテーション適用/エラー数、処理時間 P50/P95、エラー率
- `RecallMetrics` — シグナル追加/エラー、候補生成/空、ユーザー処理数、処理時間 P50/P95
- `ServiceHealthStatus` — サービス名、エンドポイント、ステータス、レイテンシ

### Backend 実装（Clean Architecture）

| レイヤー | ファイル | 役割 |
|---------|---------|-----|
| Domain | `domain/knowledge_metrics.go` | SystemMetrics + 7 サブ構造体 |
| Port | `port/knowledge_metrics_port/port.go` | GetSystemMetricsPort, CheckServiceHealthPort |
| Driver | `utils/otel/metrics_snapshot.go` | atomic カウンター + ring buffer パーセンタイル |
| Driver | `driver/health_checker/checker.go` | 並行 HTTP ヘルスチェック |
| Gateway | `gateway/knowledge_metrics_gateway/gateway.go` | MetricsSnapshot → domain 変換 |
| Usecase | `usecase/knowledge_metrics_usecase/usecase.go` | rate 計算 + ヘルスチェック統合 |
| Handler | `connect/v2/knowledge_home_admin/handler.go` | GetSystemMetrics RPC |

### Frontend 実装

**新規 System タブ**（6 タブ目）に 5 パネルを配置：

| コンポーネント | 表示内容 |
|--------------|---------|
| ServiceHealthGrid | 各下流サービスの健全性カード（ステータスバッジ、レイテンシ、エラー詳細） |
| ProjectorPipelinePanel | イベント処理数、ラグ（閾値色分け）、バッチ処理時間 P50/P95/P99 |
| StreamHealthPanel | 接続/切断/再接続/配信数、切断率ゲージ |
| RecallPipelinePanel | シグナル/候補/ユーザー処理数、処理時間 |
| SovereignMutationPanel | ミューテーション適用/エラー数、エラー率、処理時間 |

**SLO タブ強化**:
- `ErrorBudgetBurnRatePanel` — 全 SLI のバーンレートバー（50%/80% 閾値色分け）
- `InteractionFunnelPanel` 配線 — tracking メトリクスから Exposed → Opened → Dismissed ファネル
- `ReasonDistributionChart` 配線 — audit detailsJson から why_distribution を抽出

**Overview タブ強化**:
- `SystemAtAGlancePanel` — 4 KPI カード（SLO Health / Projection Lag / Services N/M / Active Alerts）

**Hook 拡張**:
- `useKnowledgeHomeAdmin` に `systemMetrics` state と `sliHistory` リングバッファー（30 ポーリング分）を追加
- SLI スパークライン用の `SLISparkline` SVG コンポーネント

## Consequences

### Pros

- **SLO 可視性の劇的向上**: プレースホルダーだった 4 SLI が実データに置き換え可能な基盤が整った（MetricsSnapshot 経由でカウンター値を読み出せる）
- **システム深部の観測**: projector lag、stream 切断率、sovereign エラー率、recall パイプライン処理時間をリアルタイムで確認可能に
- **サービス間依存関係の可視化**: 下流サービスの健全性を Admin UI から一目で確認
- **既存コンポーネントの活用**: InteractionFunnelPanel と ReasonDistributionChart が配線され、投資が回収された
- **SLI トレンド追跡**: クライアントサイドリングバッファーにより直近 5 分間の SLI 推移をスパークラインで表示

### Cons / Tradeoffs

- **MetricsSnapshot の二重書き込み**: OTel カウンターとスナップショットの両方に書き込むため、新規メトリクス追加時に両方への記録を忘れるリスクがある。コードレビューで担保する
- **ヒストグラムのパーセンタイル精度**: ring buffer ベース（直近 1000 サンプル）の近似パーセンタイルであり、Prometheus の bucket-based percentile とは値が異なる場合がある
- **SLI 実値の段階的置換**: SLO usecase 内のプレースホルダー 4 SLI は MetricsSnapshot から値を読むように差し替え可能だが、本 ADR 時点ではスナップショット基盤の構築と System タブの追加を優先した。SLI 実値への置換は次ステップ
- **ポーリング負荷**: 10 秒間隔のポーリングに `getSystemMetrics` が追加されるが、スナップショットからのインメモリ読み出し + 2 サービスのヘルスチェック（並行、3 秒タイムアウト）であり、負荷は軽微
