# 000360: Tag Verse 計算量最適化 — Barnes-Hut O(n log n) + インメモリキャッシュ

## ADR's STATUS

Accepted

## CONTEXT

ADR-351 で実装した Tag Verse のフォースグラフレイアウトと共起クエリに、以下の2つの O(n^2) 計算量ボトルネックが指摘されていた:

1. **フォースグラフレイアウト**: 全ペア反発力計算が O(n^2)。300反復 x n^2 で、n=500（limit上限）の場合に毎反復 125,000 ペアの計算が必要
2. **共起クエリ**: `article_tags` テーブルの自己結合が O(n^2)。タグ数・タグ付き記事数の増加に伴いレスポンスタイムが悪化するリスク

いずれも毎リクエストで実行されるため、タグ数増加時のスケーラビリティに課題があった。

## DECISION MAKING

### 1. Barnes-Hut アルゴリズムによるレイアウト最適化

天体物理学の N 体問題で確立された Barnes-Hut アルゴリズムを 3D フォースグラフに適用し、反発力計算を O(n^2) から O(n log n) に改善した。

**Octree（八分木）データ構造**:
- 3D 空間を再帰的に8分割し、各ノードに center of mass（重心）と total mass（総質量）を保持
- 反復ごとに全ノードの現在位置から octree を再構築

**力計算の近似**:
- Opening parameter `theta = 1.0` を採用。セルの幅 / 距離の比が theta 未満なら、そのセル内の全ボディを重心の1点として近似
- theta=1.0 は速度と精度のバランスが良く、相対誤差は約 4.3%（50ノードのテストで測定）

**代替案の検討**:

| 案 | 評価 |
|----|------|
| theta=0.5（高精度） | 精度は上がるが再帰走査の深さが増し、O(n log n) の定数係数が大きくなる。可視化用途では過剰 |
| **theta=1.0（採用）** | 誤差 ~5% は可視化で視覚的に識別不可能。d3-force 等の主要ライブラリでも標準的な値 |
| theta=1.5（高速） | 速度は上がるが誤差が 10% を超え、レイアウトの質が低下するリスク |

**ファイル構成**: Octree のデータ構造と Barnes-Hut 力計算を `layout_octree.go` に分離し、`layout.go` は高レベルのフォースグラフアルゴリズム（引力・反発力・中心力・ダンピング）に集中する設計とした。

### 2. インメモリ TTL キャッシュ

タグクラウドのデータはシード固定で決定的であり、かつ更新頻度が低い（タグ付けは記事単位で数時間〜日単位）。このため、レイアウト計算済みの結果をインメモリキャッシュし、同一リクエストの重複計算を回避する。

**キャッシュ設計**:
- **キャッシュ位置**: Usecase 層（DB クエリ + レイアウト計算の両方をスキップするため）
- **TTL**: 30分。タグデータの更新頻度に対して十分短く、一方でリクエストの大部分をキャッシュヒットでカバーできる
- **キャッシュキー**: `limit` パラメータ（デフォルト200が主用途）
- **スレッドセーフティ**: `sync.RWMutex` で concurrent access 対応
- **Deep copy**: キャッシュから返す際に deep copy を行い、caller による mutation を防止

### 3. 共起クエリの最適化

**CTE ベースクエリ**: `feed_tags` で対象タグの `id` を先に絞り込み、その `id` セットで `article_tags` の自己結合をフィルタする。対象タグ数（最大500）が `feed_tags` テーブル全体に比べて小さい場合、結合する行数が大幅に減少する。

**カバリングインデックス**: `(feed_tag_id, article_id)` の複合インデックスを追加。既存の `idx_article_tags_feed_tag_id` は `feed_tag_id` 単独で、`article_id` の取得にテーブルアクセスが必要だった。カバリングインデックスにより index-only scan が可能になる。

## RESULTS, EFFECTS

### PROS

- **反発力計算の計算量改善** — O(n^2) から O(n log n) に改善。n=500 の場合、1反復あたりの計算量が ~125,000 ペアから ~4,500 計算に削減（約28倍）
- **キャッシュによるレスポンス短縮** — 2回目以降のリクエストでは DB クエリとレイアウト計算の両方がスキップされ、deep copy のみで応答
- **共起クエリの効率化** — CTE による事前フィルタとカバリングインデックスの併用で、自己結合のスキャン範囲を限定
- **決定性の維持** — Barnes-Hut は入力と theta が同一なら決定的に同じ結果を返す。シード固定（42）と合わせて再現性を保証
- **既存テスト全パス** — 6件の既存レイアウトテスト（行動テスト）が Barnes-Hut 移行後もすべて通過
- **精度の検証済み** — theta=1.0 での相対誤差は 4.27%（50ノード測定）。theta=0 での完全一致テストも通過

### CONS, TRADEOFF

- **Octree のメモリアロケーション** — 反復ごとに octree を再構築するため、300反復で多数の小さなノードが GC 対象になる。500ノードの場合、反復あたり約 1,000 ノード x 120 バイト = 120KB のアロケーションが発生する。現状のベンチマーク（500ノード: 336ms）では問題ないが、ノード数を 1,000 以上に拡張する場合は `sync.Pool` によるノードプーリングを検討
- **Barnes-Hut の近似誤差** — theta=1.0 では約 4〜5% の相対誤差が生じる。可視化用途では視覚的に識別不可能だが、厳密な力計算が必要なユースケースには不適
- **単一キーキャッシュ** — `limit` 値ごとにキャッシュを1つしか保持しない。異なる `limit` 値が頻繁にリクエストされる場合、キャッシュヒット率が低下する。現状はデフォルト200が主用途のため問題ないが、多様な `limit` 値が使われる場合は LRU キャッシュへの拡張を検討
