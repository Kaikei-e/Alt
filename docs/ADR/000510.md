# ADR-000510: ラウンドロビン配分を idle-aware な完了順割り当てに置き換える

## Status

Accepted

## Date

2026-03-21

## Affected Services

- news-creator (Python)

## Context

BE 要約の分散ディスパッチは [[000497]] で導入済みだったが、healthy なリモート実行先に対して等重みの round-robin を採用していた。

運用観測では、直近 10 分窓の完了件数は 66 件で、停止はしていない一方、処理能力の異なる複数の実行先へ均等に振り分けられていた。
この条件では、速い実行先が先に空いても次のタスクを優先的に受け取れず、全体の throughput が遅い実行先に引きずられる。

一方で `pre-processor` 側の整合性修正は [[000509]] で先行実施済みであり、次のボトルネックは BE 要約の配分戦略になった。
ここで weighted dispatch を先に入れる選択肢もあったが、性能重みの運用や継続的な調整が必要になり、変更の切り分けが難しくなる。

## Decision

**BE 要約の配分戦略として、等重み round-robin を廃止し、idle-aware な完了順割り当てを採用する。**

具体的には以下の方針とする。

1. healthy かつ idle のリモート実行先があれば、その中から 1 台を確保して dispatch する
2. 空いている実行先が複数ある場合は、最後に完了してから最も長く空いている実行先を優先する
3. 全リモート実行先が busy または unhealthy の場合は、待機キューを service 内に増やさずローカル処理へ fallback する
4. RT リクエストの経路は変更しない

この決定により、公平な分配ではなく「空いた実行先に次を渡す」ことを優先し、異種構成の実行先群から実効 throughput を引き出す。

### 採用しなかった代替案

| 案 | 評価 |
|---|---|
| 等重み round-robin 維持 | 実装は単純だが、実行先の性能差を吸収できず throughput 改善余地が小さい |
| weighted dispatch を先に導入 | 理論上は有効だが、重みの初期値設計と再調整の運用負荷が高く、今回の変更点を増やしすぎる |
| `pre-processor` 側で配分制御 | BE 配分責務が `news-creator` から漏れ、設計境界が悪化する |

## Consequences

### Pros

- 先に空いた実行先へ次のタスクを渡せるため、速い実行先の遊休時間を減らせる
- 配分責務を `news-creator` に閉じ込められ、`pre-processor` 側の変更が不要
- 実装と検証の単位を最小限に保ったまま、round-robin のボトルネックを解消できる

### Cons / Tradeoffs

- dispatch 数は均等にならず、特定の実行先に偏ることがある
- 全リモート実行先が busy の場合はローカル fallback に流れるため、ローカル負荷を継続監視する必要がある
- throughput は改善しても、実行先ごとの能力差を最大化するには今後 weighted dispatch の検討余地が残る
