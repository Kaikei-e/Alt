# ADR-000742: 全内部サービスの送信側 mTLS クライアントヘルパを言語横断で整備する

## Status

Accepted

## Date

2026-04-15

## Affected Services

- rag-orchestrator (Go) — `internal/infra/httpclient/mtls.go` 新設、`NewPooledClient` を MTLS_ENFORCE で自動切替、`PreflightMTLS()` を DI 起動時に呼ぶ
- recap-worker (Rust) — 既存 `clients/mtls.rs` を `app.rs` から news-creator / subworker クライアントにも配線、`clients/news_creator::new_with_client` / `clients/subworker::new_with_client` 追加
- tag-generator / news-creator / recap-subworker / recap-evaluator / acolyte-orchestrator (Python) — 各 `infra/mtls_client.py` に `build_ssl_context()` ヘルパを配置、httpx `AsyncClient(verify=ssl_ctx)` に注入

## Context

受信側 ([[000741]]) を整えても、送信側 client が leaf cert を提示しなければ mTLS handshake が成立しない。既存の状況:

- **alt-backend / pre-processor / alt-butterfly-facade / search-indexer** の Go client は [[000727]] / [[000729]] で mTLS 対応済
- **recap-worker (Rust)** は alt-backend / tag-generator 呼び出しのみ `clients/mtls.rs` 経由で mTLS 済。news-creator / recap-subworker 呼び出しは plain `reqwest::Client`
- **rag-orchestrator (Go)** の `httpclient.NewPooledClient` は plaintext transport のみ。mTLS transport への切替ロジックが無い
- **Python 5 サービス (acolyte / tag-generator / news-creator / recap-subworker / recap-evaluator)** の httpx 呼び出しに `ssl.SSLContext` の注入経路が無い (acolyte-orchestrator のみ `main.py::_build_mtls_context()` で部分対応)

受信側 mTLS のみ進めて送信側が追従しないと、`MTLS_ENFORCE=true` にした瞬間に caller が全滅する。本 ADR は送信側の cert 提示経路を言語横断で揃え、`MTLS_ENFORCE` env flag 一本で all-or-nothing の切替ができる状態を目指す。

## Decision

### Go: `rag-orchestrator/internal/infra/httpclient`

`mtls.go` を新設。`NewMTLSClient(timeout)` は `MTLS_ENFORCE=true` 時に `tls.Config{Certificates, RootCAs, MinVersion: TLS13}` を注入した `*http.Client` を返す。`NewPooledClient` は `MTLS_ENFORCE` を自動検知し mTLS transport を `sharedTransport` と同じ接続プール方針で構築。`PreflightMTLS()` を DI container (`internal/di/container.go`) の先頭で呼び、cert / key / CA のロードエラーを起動時に surface。

5 ケース unit test (MTLS_ENFORCE false by default / true when env=true / NonMTLS returns plain / MissingEnv fails closed / BadCertPath fails closed / PreflightMTLS no-op when disabled)。

### Rust: recap-worker の送信クライアント全路統一

`news_creator::NewsCreatorClient` と `subworker::SubworkerClient` に `new_with_client(url, timeout, client)` コンストラクタを追加。`app.rs::ComponentRegistry::build()` で `MtlsPaths::from_env()` を一度だけ解決し、存在すれば `mtls::build_mtls_client` で reqwest::Client を構築して注入。これで alt-backend / tag-generator / news-creator / subworker の 4 outbound 全てが同じ identity を提示する。

318 既存テストは継続 pass。mTLS 有効時のみ identity が刺さる fail-open ではなく、`MTLS_ENFORCE=true` が立っているのに cert が読めない場合は startup エラーで fail-closed。

### Python: 共通 `mtls_client.py` ヘルパの複製配置

`build_ssl_context()` は以下ルール:

- `MTLS_ENFORCE != "true"` なら `None` を返す (httpx は `verify=True` にフォールバック)
- `MTLS_CERT_FILE` / `MTLS_KEY_FILE` / `MTLS_CA_FILE` のいずれか欠落なら `RuntimeError` を raise (fail-closed)
- `ssl.create_default_context(Purpose.SERVER_AUTH, cafile=ca)` + `load_cert_chain(cert, key)` + `minimum_version=TLSv1_3` を組み立て

acolyte-orchestrator の `main.py` は既存の `_build_mtls_context()` をこのヘルパ呼び出しに差し替え。tag-generator / news-creator / recap-subworker / recap-evaluator の lifespan / httpx 構築に同じパターンを注入。

各サービスに 5 ケース unit test (enforcement off / on / fail-closed on missing paths / cert unreadable / header propagation)。

### 検討した代替案

- **案 A: Python 側を `httpx.AsyncClient` の cert 指定プール化せず、各 request に cert 引数を渡す** — Ollama の streaming 経路などで client 再生成が頻発し、connection pool が効かなくなる。reject
- **案 B: Rust 側を一つの Arc<Client> に統合して全 outbound が同一 client を共有** — 各 service で timeout 設定が異なるため client 分離は維持。mtls config だけ共通化する折衷
- **案 C: DNS SAN 経由の SPIFFE URI identity** — step-ca は現状 CN ベース。SPIFFE URI SAN に移行するのは本 ADR のスコープ外 ([[000725]] の将来課題)

## Consequences

### Pros

- 東西通信の全 client が `MTLS_ENFORCE=true` 一本で cert 提示に切り替わる
- Fail-closed 設計: cert パス欠落で plaintext fallback しない (= サイレント失敗なし)
- Python ヘルパは同一コピーで 5 サービスに展開。将来 shared package 化するときの移行が trivial
- rag-orchestrator の `PreflightMTLS()` で cert ロードエラーが「起動失敗」として出るため、「最初の request まで気づかない」シナリオを防ぐ

### Cons / Tradeoffs

- **Python ヘルパの copy-paste 5 箇所**。仕様変更時の追従コスト。5 箇所以上になる時点で shared package 化
- **Rust 側は Option<ServiceToken> の Phase C 撤去と同時でないと、client が token を送るのに provider 側は既に見ない冗長状態**。本 ADR はこの overlap を許容し、[[000743]] で整合させる
- **既存の app 層 retry / backoff 挙動が TLS handshake エラーで変わる可能性**。reqwest/httpx の TLS エラーは `ConnectionError` に包まれるため既存 retry は効くが、timeout が長めに観測されるケースは追跡対象
