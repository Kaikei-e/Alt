# ADR-000890: bullet `[n]` citation を `recap_subworker_sentences.id` に reconcile して `source_sentence_ids` を埋め直す

## Status

Accepted

## Date

2026-05-05

## Affected Services

- **recap-worker (Rust)** — citation reconciliation を実装する 6 ファイル変更。
  - `recap-worker/recap-worker/src/store/dao/traits/output.rs`: `OutputDao` trait に `get_sentence_ids_by_run(run_id) -> Result<HashMap<String, Vec<i64>>>` を追加。
  - `recap-worker/recap-worker/src/store/dao/output.rs`: `recap_subworker_sentences` と `recap_subworker_clusters` を JOIN する 1 query 実装。`(article_id -> Vec<sentence DB id>)` に Rust 側で集約。
  - `recap-worker/recap-worker/src/store/dao/impls/unified.rs`: `UnifiedDao` から実装関数への委譲。
  - `recap-worker/recap-worker/src/store/dao/compat.rs`: legacy `RecapDao` supertrait と blanket impl にメソッドを追加。
  - `recap-worker/recap-worker/src/store/dao/mock.rs`: `MockRecapDao` に `outputs: Mutex<Vec<RecapOutput>>` capture、`set_article_metadata` / `set_sentence_ids` builder、`get_sentence_ids_by_run` stub を追加。clippy `type_complexity` を回避するため `ArticleMetadataMap` / `SentenceIdsByArticle` / `SentenceIdsByRun` の 3 type alias を導入。
  - `recap-worker/recap-worker/src/pipeline/persist.rs`: `REFERENCE_MARKER_RE` 静的 regex (`\[(\d+)\]`)、純粋関数 `reconcile_bullet_citations(bullet, refs, url_to_article, article_to_sentence_ids) -> Vec<i64>`、`FinalSectionPersistStage::persist` の bullet 構築直前に DAO 呼び出しと map 構築、bullet loop で `Vec::<i64>::new()` ハードコードを reconciler 出力に置換。9 本の edge-case unit test (`reconcile_*`) と 1 本の persist 統合 test (`persist_writes_non_empty_source_sentence_ids_when_references_resolve`) を追加。

Atlas migration / Pact contract / compose service / 名前付き volume いずれも変更なし。

## Context

production の `recap_outputs` を直接観測した結果、直近 7 日に書かれた 331 bullets すべての `bullets_ja[].source_sentence_ids` が空配列 `[]` になっていた:

```sql
SELECT count(*) AS total, count(*) FILTER (WHERE jsonb_array_length(b->'source_sentence_ids') > 0) AS grounded
FROM recap_outputs, LATERAL jsonb_array_elements(bullets_ja) AS b
WHERE created_at > NOW() - INTERVAL '7 days';
-- total=331, grounded=0
```

`recap-evaluator/src/recap_evaluator/evaluator/source_grounding.py::SourceGroundingEvaluator.compute` は bullet 内の `source_sentence_ids` の長さ > 0 を grounding 判定の唯一の指標としているため、citation grounding score が構造的に 0.0 で固定される。`SummaryMetrics.source_grounding_score` の dashboard が永久に 0 のまま動かないという報告は、観測データと整合する。

ユーザは「ADR に従って改修したのに直っていない」と報告。ADR 履歴と実装を照合した結果、根本原因は単純だった:

- [[000832]] §Decision §4 が `bullets_ja` の `source_sentence_ids` スロットを **scaffold だけ** 入れ、実際の reconciliation parser は明示的に scope-out していた:
  > 本 ADR では空配列でスロットを確保するに留め、`SourceGroundingEvaluator` は 0.0 を返すだけで運用に入る。
  > `source_grounding_score` は当面常に 0.0 …citation parsing が入るまで評価値は動かない。
- 後続の [[000886]] / [[000887]] / [[000888]] / [[000889]] は morning-letter 系列の修復で、reconciliation には触れていない。
- `recap-worker/recap-worker/src/pipeline/persist.rs` 上の最終コミットは [[000832]] 由来の `bdc2a2555 feat(recap-worker): enrich clustering error context and scaffold bullet source_sentence_ids` のままで、bullet 構築箇所は `"source_sentence_ids": Vec::<i64>::new()` をハードコードしていた。

つまり「ADR で予告された後送り作業がそのまま放置されていた」状態であり、本 ADR がその後続を担う。

### 利用可能な signal の整理

- news-creator は `Summary.references: Option<Vec<Reference{ id, url, domain, article_id }>>` を返し、bullets に `[n]` markers を含める (`news-creator/app/news_creator/usecase/recap_summary_usecase.py:1939-1947` で auto-inject も行う)。production sample で 182/183 row が non-empty references を持っていた。
- `recap_subworker_sentences` テーブルは `(id bigint PK, cluster_row_id, source_article_id, sentence_id, sentence_text, ...)` を保持し、cluster は `recap_subworker_clusters.run_id` で run に紐付く。
- recap-worker は genre dispatch 時点で `clustering_response.run_id: i64` を保持しており、persist stage で参照可能 (`pipeline/dispatch/types.rs:27`、`clients/subworker/types.rs:40`)。
- persist.rs は既に `dao.get_article_metadata(job_id, article_ids)` で `(article_id -> (published_at, source_url))` を取得し `top_sources` を組んでいるので、URL fallback 用の `(url -> article_id)` map は追加クエリ無しで作れる。

### LLM 出力の不確実性

production reference のサンプル `[{"id": 1, "url": "https://dev.to", "domain": "dev.to"}]` には `article_id` が欠落しており、LLM は input の `representative_sentences[].article_id` を必ずしも忠実に echoback していない。完全な article_id only 解決は失敗し、URL fallback が必要。URL も generic root (`https://dev.to`) になりがちで、複数 article にマッチする。`SourceGroundingEvaluator` は `len > 0` の二値判定なので、URL fallback の multi-match は false positive のコストが極めて低い。

## Decision

`recap-worker/recap-worker/src/pipeline/persist.rs` に DAO 1 query + 純粋関数の citation reconciler を導入し、`FinalSectionPersistStage::persist` の bullet 構築箇所で `source_sentence_ids` を埋める。

### 1. `OutputDao` に `get_sentence_ids_by_run` を追加する

genre 毎に 1 回呼ばれる SQL:

```sql
SELECT s.id, s.source_article_id
FROM recap_subworker_sentences s
JOIN recap_subworker_clusters c ON c.id = s.cluster_row_id
WHERE c.run_id = $1
ORDER BY s.id
```

Rust 側で `HashMap<String, Vec<i64>>` に group する。`OutputDao` trait (新形式) と `RecapDao` supertrait (旧 compat) と `UnifiedDao` impl と `MockRecapDao` 全てに配線。

### 2. 純粋関数 `reconcile_bullet_citations` を導入する

```rust
fn reconcile_bullet_citations(
    bullet: &str,
    refs: &[Reference],
    url_to_article: &HashMap<String, String>,
    article_to_sentence_ids: &HashMap<String, Vec<i64>>,
) -> Vec<i64>
```

`REFERENCE_MARKER_RE = LazyLock<Regex>` で `\[(\d+)\]` を抽出し、各 marker に対し:

1. index が `1..=refs.len()` の範囲外なら無視 (out-of-range / `[0]` / 非数値)。
2. `refs[n-1].article_id` が `Some` ならそれを採用 (singleton)。
3. それ以外は `url_to_article` で URL マッチ (multi-match を許容)。
4. 解決した article_id 群について `article_to_sentence_ids.get(...)` で sentence id を集める。
5. `BTreeSet<i64>` で重複除去、最後に `Vec<i64>` 昇順で返す。

引数は全て borrow、戻り値は所有 (`Vec<i64>`)。`unwrap()` / `panic!()` を使わず、解決不能 marker は `debug!` ログだけ出して捨てる。

### 3. `FinalSectionPersistStage::persist` に配線する

`top_sources` 構築直後、bullet loop の直前で:

- `top_sources` (既存) から `(url, article_id)` を抽出して `url_to_article: HashMap` を作る。
- `genre_result.clustering_response.as_ref().map(|c| c.run_id)` を取得し、`Some(run_id) if run_id > 0` のときだけ `dao.get_sentence_ids_by_run(run_id)` を await。失敗時は `warn!` ログ + 空 map で **fail-open** に倒す。
- `summary_response.summary.references.as_deref().unwrap_or(&[])` を `refs` として用意。

bullet 構築は `Vec::<i64>::new()` ハードコードを削除し、各 bullet について `reconcile_bullet_citations(...)` の戻り値を `source_sentence_ids` に書き込む。

### 4. Resume 経路は scope out

`clustering_response: None` (resume / `body_json` 復元経路) では `run_id` が手元に無い。本 ADR ではこの経路を `[]` のまま落とし、`// TODO(ADR-followup): resume path needs run_id from recap_subworker_runs lookup` をコメントで残す。`GenreResultLightweight.clustering_run_id` (`pipeline/dispatch/types.rs:50`) を流用すれば後続 ADR で 1 行修正可能。

### 5. 検討した代替案

- **Strategy 1 (URL fallback only)**: persist 時に `references[n-1].url` を `url_to_article` だけで解決し、`recap_subworker_sentences` には触れない。最終的に sentence id を埋めるためには結局 DAO クエリが必要なので Strategy 2a と等価。**統合採用**。
- **Strategy 2b (representative_sentence に sentence_id を持たせる)**: prompt 構築時に `(article_id, sentence_id)` を memoize して persist まで持ち越し、DAO クエリ自体を省略する。ただし `recap_subworker_sentences.id` は subworker DAO insert (`nextval`) 後にしか確定しないため、client 側で memoize できない。**却下**。
- **Strategy 3 (news-creator schema 拡張)**: `Summary.references` に `sentence_id` を持たせて LLM に明示出力を強制する。Pact 破壊・news-creator プロンプト改修・consumer 側全面再生成が要る重い変更で、本件 surgical fix の射程を超える。**却下**。
- **`SubworkerDao` に置く**: `recap_subworker_*` テーブルだけを参照するクエリなのでセマンティクス的には `SubworkerDao` の方が自然。ただし persist.rs は既に `OutputDao` を使っており、本 ADR で trait 横断を増やすと wire 量が増える。`OutputDao` に置いても trait の意味論上のずれは小さい (output の grounding 用 lookup) と判断。**`OutputDao` 採用**。
- **過去 row の backfill**: 既存 331 bullets を遡って埋め直す。disposable projection の不変条件 ([[IMPL_BASE]]) と整合せず、reproject 経路で自然に置換すべき領域。本 ADR では新規 row のみ対象とし、過去 row は `0` のまま放置する ([[000888]] §Cons line 149 と同方針)。**却下**。

## Consequences

### Pros

- Citation grounding score が構造的に 0.0 で固定される現象が解消する。次の recap pipeline tick から `bullets_ja[].source_sentence_ids` が article 群の sentence id で埋まり、`SourceGroundingEvaluator` の二値判定で grounded 扱いになる。
- [[000832]] で予告した後送り作業が初めて完結する。`feedback_eval_first` (eval を先に整え、指標で判定する) ループが本 ADR で機能し始め、今後 classifier や clustering を tune した際の改善幅を `source_grounding_score` の変動で確認できるようになる。
- 9 本の edge-case unit test と 1 本の persist 統合 test が contract を pin するので、reconciliation の挙動が将来の refactor で意図せず壊れたら CI で気付ける。`MockRecapDao` の `outputs` capture は今後の persist 系新機能でも再利用できる。
- DAO 拡張は単純 SELECT JOIN 1 本で hot path に追加コストが小さい。`recap_subworker_sentences (cluster_row_id)` には既に `idx_recap_subworker_sentences_cluster_row_id` index があり、`recap_subworker_clusters.run_id` も unique 制約配下で抽出が高速。
- 不変条件の整合: append-first / reproject-safe / disposable projection / merge-safe upsert / event-time purity いずれも違反なし。`bullets_ja` は既存の disposable projection で、新規 row のみ richer に書き直され、既存 row は次の reproject まで `[]` のまま (ADR-888 §Cons と同方針)。
- Pact / Atlas / compose 全て不変。news-creator schema、recap-worker → recap-evaluator interface、knowledge-sovereign 経路すべて触らないため deploy blast radius が `recap-worker` 単独に限定される。

### Cons / Tradeoffs

- LLM が `Reference.article_id` を埋めない傾向が観測されており、URL fallback で multi-match が起きうる。`SourceGroundingEvaluator` は二値判定なので実害は小さいが、後続で「sentence-level 精度の citation」を要求する評価軸が増えると false positive が問題になる。本 ADR では割り切るが、`feedback_eval_first` 配下で精度の二段階評価 (article-level / sentence-level) を入れる余地を残す。
- Resume 経路 (`clustering_response: None`) で `source_sentence_ids` が空のまま落ちる。レアケースだが、長時間ジョブの再開でエッジを踏むと不均一に grounded が抜ける row が混じる。`GenreResultLightweight.clustering_run_id` を読む 1 行追加で解決できるので、観測してから後続 ADR で潰す。
- `MockRecapDao` の type 定義が膨らみ、3 つの type alias (`ArticleMetadataMap` / `SentenceIdsByArticle` / `SentenceIdsByRun`) を導入した。clippy `type_complexity` を回避する正規アプローチではあるが、テストコード読解時の間接層が 1 段増える。
- 既存 row は backfill しない設計。dashboard の history を直近 7 日 0 → 段階的に上昇する形で見ることになり、修復タイミングの可視化はできるが、過去のレポート品質を後追いで上げる経路はない。retention cleanup の別 ADR が来た時に整合させる。
- 新 DAO メソッドは `recap_subworker_sentences` の row 数に O(n)。極端に大きな run (例: 数万 sentences) では返却 `HashMap` のメモリが膨らむ可能性がある。現状 production の 1 run あたり数百〜数千 sentences なので問題ないが、`LIMIT` を付けるか article_id の prefilter を入れる選択肢を将来検討する余地。
