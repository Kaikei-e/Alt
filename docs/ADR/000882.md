# ADR-000882: ADR-000880 deploy 後の residual cohort を accept し 4 件の follow-up に分流する

## Status

Accepted

## Date

2026-05-02

## Affected Services

- **knowledge-sovereign (PostgreSQL)** — `knowledge_event_dedupes` / `knowledge_events` 整合性の観測対象。本 ADR ではコード変更を伴わず、follow-up (a) で原因解析と修正を扱う。
- **alt-backend (Go)** — `EmitArticleUrlBackfill` admin endpoint の async 化対象。本 ADR ではコード変更を伴わず、follow-up (c) で扱う。
- 観測そのものは Loop projection の一貫性に対する triage であり、新規実装は含まない。

## Context

[[000880]] を rolling deploy したのち admin emit を 4 回（4 invocation、最後で `more_remaining: false`）走らせ、Loop projector が catch up 完了した時点で `knowledge_loop_entries` の `act_targets[0].source_url` 充填率を計測した。

```
SELECT
  COUNT(*) FILTER (WHERE act_targets::text LIKE '%source_url%') AS with_source_url,
  COUNT(*) FILTER (WHERE act_targets IS NOT NULL AND act_targets::text NOT LIKE '%source_url%') AS without_source_url,
  COUNT(*) AS total
FROM knowledge_loop_entries;
-- with=52,706 / without=47 / total=52,753
```

Recovery rate **99.91%**（52,706 / 52,753）。これは ADR-000880 §6 の Decision Criteria 5「本番 emit 後の without_source_url 数を再計測し効果を検証」に対する直接的な観測結果である。

### 残 47 件の triage

| パターン | 件数 | 観測根拠 | 復旧経路 |
|---|---|---|---|
| **(P1) sovereign dedupe / event log atomicity** | 2 | `knowledge_event_dedupes` v2 entry あり、`knowledge_events` の `ArticleUrlBackfilled` 行が無い | sovereign 側 `AppendKnowledgeEvent` の atomicity 修正 |
| **(P2) emit 完走後の新着 article** | 1 | 04:44Z の最後の emit 後に ingest された article、v2 dedupe entry も無い | 次回 emit で復旧（運用循環） |
| **(P3) Loop entry 後発生成 race** | 44 | v2 dedupe entry あり、v2 event は emit 済、しかし `projected_at` が v2 event の `occurred_at` より後 — Loop projector が ArticleUrlBackfilled を消費した時点で対象 Loop entry が未生成 | Loop entry 生成系 event 経路への URL 注入網羅性検証 |

### 計測上の補強

- v1 dedupe 29,213 + v2 dedupe 52,774 = 81,987 行
- `knowledge_events` の `ArticleUrlBackfilled` 行（全 partition 横断）= 81,985 行
- 差分 2 = (P1) の件数と一致 → atomicity 違反の直接証拠
- Loop projector lag = 0、`last_event_seq` = `MAX(event_seq)` → catch up 完了
- (P3) のサンプル: `loop_projected_at = 2026-05-02 04:59:47` / `v2_emitted = 2026-05-02 04:28:40` で 31 分遅れて Loop entry 化、消費済 corrective に追加で patch される経路は projector に存在しない

### 不変条件への影響

- Append-first / Reproject-safe / Disposable projection / Dedupe ≠ projection の各不変条件は本 ADR で **新たに破られていない**
- (P1) は append-first の前提（dedupe 登録 ⇒ event row が必ず存在）に対する事故的不整合であり、不変条件「再投影で同じ結果になる」を脅かす可能性がある → follow-up (a) で扱う

## Decision

ADR-000880 の deploy 結果を **accept** し、本 ADR を観測記録とする。残 47 件の追跡は次の 4 件の follow-up ADR に分流し、本 ADR では新規コード変更を行わない。

| ID | 対象 | 想定範囲 | 駆動原因 |
|---|---|---|---|
| (a) | sovereign `AppendKnowledgeEvent` の dedupe / event log atomicity | RPC ハンドラ層を 1 つの transaction にまとめる、または compensating cleanup を入れる | (P1) の 2 件、再発リスク防止 |
| (b) | dedupe registry TTL / GC | TTL 列追加 + 定期 cleanup、または generation counter で世代管理 | v1 + v2 で 81,987 行が永続蓄積（[[000880]] §5 Cons #1 / proposal §8 Q2） |
| (c) | `EmitArticleUrlBackfill` の async 化 | background goroutine + status polling endpoint | 502 表示問題（proposal §8 Q3）、運用負荷 |
| (d) | Loop entry 生成系 event の URL 注入網羅性 | `track_home_action_usecase` 以外の Loop entry 生成パスを列挙し、各々の payload に `url` を注入できているか監査 | (P3) の 44 件、新着分でも継続発生する race |

### 検討した代替案

#### (却下 1) 残 47 件を手動 SQL で patch する

- 利点: 数分で 100% 復旧
- 欠点: `Disposable projection` 不変条件違反（projection に対する write path 外の直接 mutate）。reproject で再現できないため監査不能。[[feedback_no_sql_logic]] の禁則対象
- 却下理由: 一度許すと corrective event の意義が崩れ、[[000867]] [[000880]] で確立した「append-first で復旧する」設計が形骸化する

#### (却下 2) すぐ v3 namespace bump で再発火する

- 利点: (P1)(P2)(P3) の混在を均一に再 emit で潰せる
- 欠点: 47 件の小規模 cohort に対して 81,987 行の dedupe registry を更にもう一世代増やす。費用対効果が悪い。(P3) の race は再 emit でも同じ理由で raceしうる
- 却下理由: dedupe registry の累積を加速させるだけで、根因解決にならない

#### (却下 3) Loop projector に「entry 後発生成時の遡及 patch」分岐を追加

- 利点: (P3) を即時解消できる
- 欠点: projector が「過去の event を保持して後続の entry 作成時に再適用する」という stateful 処理を持つ → reproject-safe 不変条件を壊す（projector が event payload と stable resource 以外、すなわち過去 event の cache に依存する）
- 却下理由: イミュータブル設計の北極星から外れる。生成側の event payload に URL を入れるのが正解（follow-up (d)）

#### (却下 4) 本 ADR 内で (a)(d) を実装まで進める

- 利点: 1 ADR で完結する
- 欠点: (a) は sovereign 側の write path 改修、(d) は Loop entry 生成系 event の網羅監査と payload 改修。それぞれ独立した設計判断を含み、blast radius が大きく分けるべき
- 却下理由: ADR は「決定の最小単位」。混ぜると後続レビューで分割再構成が必要になる

## Consequences

### Pros

- ADR-000880 の deploy 結果を定量で accept し、recovery rate 99.91% を文書化
- 残 47 件を放置せず、4 件の follow-up に分流して責務を明確化
- (P1) の atomicity 不整合を「観測された事実」として記録し、再発時の参照点とする
- 本 ADR ではコードを変更しないため deploy 不要、blast radius ゼロ

### Cons / Tradeoffs

- 残 47 件は本 ADR 単独では復旧しない。完全 0 を目指す場合は follow-up (a)(d) の完了を待つ
- (P3) の 44 件は新規 Loop entry が生成されるたびに同種の race を起こす可能性があり、follow-up (d) を後回しにすると残債が増え続ける
- (P1) の atomicity 不整合は再発リスクが残る — 同じパターンで他 event type にも発生しうるが本 ADR では追跡しない

### 未解決

- (a) sovereign `AppendKnowledgeEvent` の atomicity 修正
- (b) dedupe registry TTL / GC
- (c) `EmitArticleUrlBackfill` の async 化
- (d) Loop entry 生成系 event の URL 注入網羅性検証

## Verification

本 ADR は計測結果の triage であり、追加のコード変更・テストを含まない。検証手順は ADR-000880 §Verification の post-projection SQL を再実行し、`without_source_url` の絶対値が極小（本 ADR 時点で 47）であることを確認するのみ。

```sql
SELECT
  COUNT(*) FILTER (WHERE act_targets::text LIKE '%source_url%') AS with_source_url,
  COUNT(*) FILTER (WHERE act_targets IS NOT NULL AND act_targets::text NOT LIKE '%source_url%') AS without_source_url,
  COUNT(*) AS total
FROM knowledge_loop_entries;
```

(P1) の検出には dedupe / event row 数の差分も併用する:

```sql
SELECT
  (SELECT COUNT(*) FROM knowledge_event_dedupes WHERE dedupe_key LIKE 'article-url-backfill%') AS dedupe_total,
  (SELECT COUNT(*) FROM knowledge_events WHERE event_type = 'ArticleUrlBackfilled') AS event_total;
```

差分が拡大する兆候があれば follow-up (a) の優先度を引き上げる。
