# ADR-000845: Knowledge Loop の projector を knowledge-sovereign に移管し OODA 契約を E2E / CDC で固定する

## Status

Accepted

## Date

2026-04-26

## Affected Services

- knowledge-sovereign (Go) — `app/usecase/knowledge_loop_projector/` 新設 (`enricher.go` / `projector.go` / `constants.go` + 対応 test)。`app/main.go` で 5 秒 ticker により in-process projector を起動
- alt-backend (Go) — `app/job/knowledge_loop_projector.go` + test、`app/usecase/knowledge_loop_usecase/why_enricher.go` / `why_mapper.go` + test を削除。`app/main.go` から旧 `KnowledgeLoopProjectorJob` 起動コードを除去。`app/usecase/knowledge_loop_usecase/validator.go` の `WhyMappingVersion` を v3 → v4 に bump
- alt-frontend-sv (SvelteKit) — `src/test/contracts/knowledge-loop-contract.test.ts` (CDC 12 ケース)、`tests/e2e/desktop/loop/canonical-contract.spec.ts` (Playwright 5 ケース)、`tests/e2e/fixtures/factories/knowledgeLoopFactory.ts` を追加
- e2e/hurl — `alt-backend/26-knowledge-loop-canonical-contract.hurl` (§7 allowlist 全 7 ケース)、`knowledge-sovereign/19-knowledge-loop-projector-canonical-contract.hurl` (append→tick→read の end-to-end) を追加。`alt-backend/run.sh` に UUIDv7 変数を 7 本追加

## Context

[[000844]] で `/loop` UI の表面バグ (`whyPrimary.text` が `"New summary"` 固定 / Observe entry に observe → act CTA が seed されて全 disabled / Augur seed が placeholder) を in-place で修正した。同 ADR の Cons に「enricher と projector job が依然 alt-backend に残っており、Knowledge Home/Loop の責務オーナーが knowledge-sovereign であることと不整合」を残した。

[[000831]] で固定した責務分担は次の通り:

- knowledge-sovereign は `knowledge_events` (append-only)、`knowledge_loop_*` projection、curation state、retention の **single owner** ([[000832]] / [[knowledge-home-projection-recovery]])
- alt-backend は read-side の Connect-RPC handler を持ち、書き込みは sovereign に委譲する

ところが [[000840]] で導入された projector は **alt-backend の `app/job/knowledge_loop_projector.go` に残ったまま**で、`sovereign_client` 経由で knowledge-sovereign の `ApplyKnowledgeLoopMutation` を叩いていた。これは次の負債を抱えていた:

1. **責務分散** — projector のロジック (event → enrich → projection upsert) が alt-backend、durable state が knowledge-sovereign。reproject-safe / no `time.Now()` / merge-safe upsert といった不変条件 ([[000398]] / [[IMPL_BASE]]) のテストが両サービスに分散する
2. **不要な RPC ホップ** — projector の毎バッチが (a) 自サービス内で event を読む RPC + (b) sovereign への mutation RPC を経由。同一 DB に対する write を network 越しに変換する
3. **デプロイ単位の重複** — projector の修正のたびに alt-backend を rebuild する必要があり、knowledge-sovereign の変更とリリース粒度がずれる
4. **TDD outside-in の規律違反** — [[000844]] では Phase 3 (Unit) のみ書いて Phase 1 (E2E) と Phase 2 (CDC) を deferred にした。`feedback_tdd_outside_in` (「ユニットだけでは不十分。新機能・サービス境界変更は E2E/CDC 先行」) と canonical contract の §7 / §11 を pin するテストが欠けていた

ユーザは「Knowledge Sovereign が Knowledge Home の責務を担う」前提を改めて明示し、移管をやり切ること、および TDD 規律を E2E (Playwright + Hurl 両層) と CDC まで埋めることを要求した。

### 制約

- 既存の `knowledge_loop_entries` 行を破壊しない (UPDATE/DELETE 禁止)
- `WhyMappingVersion` bump で reproject 経路を発火させ、ownership move を operator に明示する
- knowledge-sovereign は現状 Connect-RPC server only で job runner を持たないため、main.go に periodic loop を新設する必要がある
- proto / Atlas migration / Connect-RPC の wire format は変えない (FE と alt-backend handler の改修不要)

## Decision

projector + enricher + why-mapper の **オーナーシップを knowledge-sovereign に移管**し、alt-backend は read-side proxy のみを残す。`WhyMappingVersion` を 3 → 4 に bump して移管をシグナルとする。同時に [[000844]] で deferred にした TDD outside-in を **Playwright + Hurl × 2 + CDC** で埋め、§7 transition allowlist と §11 Why-as-first-class を契約レベルで pin する。

### 1. knowledge-sovereign への移管

新パッケージ `knowledge-sovereign/app/usecase/knowledge_loop_projector/`:

- `enricher.go` — pure な `EnrichWhyFromEvent` (event payload only / no DB lookup / no `time.Now()`)。WhyKind 別の narrative を proto type `*sovereignv1.KnowledgeLoopWhyPayload` で直接返す
- `projector.go` — `Repository` interface (`ListKnowledgeEventsSince` / `GetProjectionCheckpoint` / `UpdateProjectionCheckpoint` / `UpsertKnowledgeLoopEntry` / `UpsertKnowledgeLoopSessionState` の最小集合) + `Projector.RunBatch` 主ループ + stage 別 `seedDecisionOptions`
- `constants.go` — event type 定数 + `WhyMappingVersion = 4`

`main.go` に in-process ticker を追加:

```go
loopProjector := knowledge_loop_projector.NewProjector(repo, slog.Default(),
    knowledge_loop_projector.Config{BatchSize: 100})
loopTick := time.NewTicker(5 * time.Second)
go func() {
    defer loopTick.Stop()
    for {
        select {
        case <-ctx.Done(): return
        case <-loopTick.C:
            if err := loopProjector.RunBatch(ctx); err != nil {
                slog.Error("knowledge_loop_projector batch failed", "error", err)
            }
        }
    }
}()
```

alt-backend からは projector job、enricher、why_mapper、対応 test を削除 (1491 行削除)。`app/main.go` の `KnowledgeLoopProjectorJob` 配線も除去。read-side の Connect-RPC handler (`app/connect/v2/knowledge_loop/handler.go`) と handler-boundary validator (`validator.go`) は alt-backend に残置。

### 2. TDD outside-in の補完

| 層 | ファイル | 対象 |
|---|---|---|
| **E2E (Playwright)** | `tests/e2e/desktop/loop/canonical-contract.spec.ts` (5 ケース) + factory | DOM 上の why_text / Revisit-Ask-Snooze CTA enabled / Revisit→orient の TransitionKnowledgeLoop / Augur handshake POST 内容 |
| **E2E (Hurl alt-backend)** | `26-knowledge-loop-canonical-contract.hurl` (7 ケース) | §7 transition allowlist 全 5 許可遷移 + 3 禁止遷移 (orient→act / decide→observe / act→act) を handler-boundary で網羅 |
| **E2E (Hurl knowledge-sovereign)** | `19-knowledge-loop-projector-canonical-contract.hurl` | `AppendKnowledgeEvent (SummaryVersionCreated, article_title in payload)` → projector の 5s tick (retry 20×1s) → `GetKnowledgeLoopEntries` で §11 narrative + `LOOP_STAGE_OBSERVE` の存在を end-to-end 検証 |
| **CDC (consumer)** | `src/test/contracts/knowledge-loop-contract.test.ts` (12 ケース) | proto schema round-trip + `STAGE_ALLOWED_INTENTS` table で §7 allowlist を pin + placeholder regex `/^New summary$/` 拒否 |

CDC 側で特筆すべきは、`STAGE_ALLOWED_INTENTS` を proto 列挙値の table として定義し、`it.each` で各 stage の DecisionIntent を検査する形にしたこと。projector が将来 Observe entry に `OPEN` / `SAVE` を seed したくなっても、この table を更新せずに contract を破ると test が落ちる。

### 3. WhyMappingVersion v3 → v4 の意味

[[000844]] で v2 → v3 bump (narrative 書き換え + stage-appropriate seed) を行ったため、本 ADR で v3 → v4 を bump する必要は実装上は無い。しかしオペレータ視点で「**projector の在処が変わった**」を runbook で検出可能にするため、敢えて bump する。`docs/runbooks/knowledge-home-projection-recovery.md` がこのバージョン差分を見て full reproject を発火する。

### 検討した代替案

1. **「shared module を作って alt-backend と knowledge-sovereign で enricher を共有」** — 却下。Alt は各サービスを独立した Go module として運用しており、cross-module import には `replace` directive か go workspace が必要。fragile な依存を増やすコストに見合わず、責務移管の趣旨にも反する。
2. **「migration を後続 PR に分離する」** — 却下。[[000844]] の Cons で deferred と書いたが、ユーザは「やり切って」と明示した。1 PR 内で `Phase 0 (移管) → Phase 1/2/3 (E2E/CDC/Unit)` の outside-in を貫くほうが、契約と実装の対応が辿りやすい。
3. **「knowledge-sovereign に Connect-RPC `RunProjectorBatch` を生やして alt-backend がそれを叩く」** — 却下。RPC ホップを 1 段増やすだけで、責務分担の根本問題 (どっちがオーナー?) を解決しない。in-process job が最短の正解。
4. **「projector の cadence を 1 秒や PgListener (`LISTEN/NOTIFY`) ベースにする」** — 却下。今回はオーナーシップ移管が主題。reactive な発火は `ConnectProjectorWatch` の sovereign 側実装と整合させる別 ADR で扱う。
5. **「Hurl は knowledge-sovereign 側だけにして alt-backend を skip」** — 却下。alt-backend は handler-boundary validator (`ValidateDwellTriggerTarget`、§7 allowlist 自体の拒絶) を依然責任として持つ。境界の両側でテストする ([[000844]] の context にもある「§7 transition allowlist は handler でも projector でも守らねばならない」二重防御)。

## Consequences

### Pros

- Knowledge Loop projection の write path が **knowledge-sovereign に集約**され、append-first / reproject-safe / merge-safe upsert / no `time.Now()` の不変条件テストが一箇所で完結する ([[knowledge-home-projection-recovery]] / [[IMPL_BASE]] の責任ライン明確化)
- alt-backend のコード量が 1491 行減 (`+32 / -1491`)。job runner 周りの依存も軽くなる
- in-process write でネットワークホップ 1 段削減。`COALESCE` merge と `seq-hiwater` guard は driver 側で既に実装済み (`write_knowledge_loop.go`) なので、in-process 化しても merge-safety は維持
- TDD outside-in が **Playwright (DOM) → Hurl × 2 (両 Connect-RPC handler) → CDC (proto schema + §7 table) → Unit (純粋関数)** の 4 層で揃った。`/loop` の OODA 契約 (§7 allowlist / §11 narrative / §1 軸分離) を contract レベルで pin
- 新 Hurl が `append → projector tick (5s) → read` を end-to-end で叩くため、in-process projector の ticker が機能していることを CI で検証できる
- `WhyMappingVersion` v4 が runbook の reproject 発火基準として明示的にシグナルされる

### Cons / Tradeoffs

- knowledge-sovereign に goroutine + ticker が増えた。サービス起動コストが微増。ticker 自体は context cancel で停止するが、`RunBatch` 内で長時間処理が走った場合は graceful shutdown のタイムアウト 15s に間に合わない可能性がある (現状の最大バッチ = 100 events × 数 ms 想定なので実用上問題なし)
- 既存の `knowledge_loop_entries` 行は **deploy 直後は古い projector 由来のまま**。`WhyMappingVersion=4` を見て runbook で full reproject を実行するまで切り替わらない
- alt-backend `validator.go` に `WhyMappingVersion` 定数の duplicate を残した。authoritative copy は knowledge-sovereign 側だが、ハンドラ層のバージョン整合性チェックが alt-backend 側にあるため両方に置いてある (コメントで明示済み)。将来 alt-backend 側を remove するときは追加 ADR を立てる
- Hurl 19 (knowledge-sovereign) は projector tick 5s に依存している。CI 上で polling timeout を 20×1s = 20s 確保したが、極端に遅い CI 環境では false negative の可能性が残る
- Augur ホスト先 (rag-orchestrator) の `CreateAugurSessionFromLoopEntry` 永続化経路は本 ADR では touch していない。[[000836]] で扱った seed message 永続化の責務はそのまま (whyText を contract narrative にしたので結果的に正しいテキストが届く)
