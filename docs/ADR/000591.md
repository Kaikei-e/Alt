# ADR-000591: Pact CDC テストを全サービスペアに全面展開し Pact Broker による契約管理を確立する

## Status

Accepted

## Date

2026-03-26

## Affected Services

- recap-worker (Rust) — Consumer: 4 Provider に対する 10 interactions
- search-indexer (Go) — Consumer: 2 Provider に対する 6 interactions（HTTP/Connect-RPC + Message）
- mq-hub (Go) — Message Producer: 4 interactions（Redis Streams イベント契約）
- recap-evaluator (Python) — Consumer: 3 interactions
- recap-subworker (Python) — Provider 検証
- tag-generator (Python) — Provider 検証
- alt-backend (Go) — Provider 検証（REST API）
- news-creator (Python) — Provider 検証（Broker 対応拡張）

## Context

[[000588]] で Buf + Pact CDC の 2 層 Contract Testing 戦略を採用し、[[000589]] で最重要 3 ペア（alt-backend → pre-processor、pre-processor → news-creator、rag-orchestrator → news-creator）の CDC テストを実装した。しかし、Alt の 23+ マイクロサービス間には 11 以上の通信パスが存在し、未カバーの 8 ペアが残っていた。

特に recap-worker は 4 つの Provider（news-creator、recap-subworker、alt-backend、tag-generator）を呼び出す最大のコンシューマーハブであり、Rust で書かれているため Pact エコシステムの新たな言語対応が必要だった。また、Redis Streams による非同期イベント通信（mq-hub ↔ search-indexer/tag-generator）はこれまで契約テストの対象外だった。

Pact ファイルの共有はファイルシステム経由で行われており、CI でのサービス間依存管理やデプロイ安全性チェック（can-i-deploy）ができない状態だった。

## Decision

**Pact CDC テストを全サービスペアに全面展開し、Pact Broker による契約の一元管理を確立する。**

### 1. Pact Broker 統合

CI ワークフロー（`.github/workflows/proto-contract.yaml`）の `pact-cdc-python` ジョブに Pact Broker（PostgreSQL + pact-broker コンテナ）を GitHub Actions services として追加。Consumer テスト実行後に pact ファイルを Broker に publish し、Provider 検証は Broker から契約を取得して実行する。

Provider 検証テスト（`news-creator/app/tests/contract/test_provider_verification.py`）は `PACT_BROKER_BASE_URL` 環境変数による Broker モードとローカルファイルモードのデュアル対応に改修。

### 2. Rust Pact 対応（recap-worker）

`pact_consumer` v1.4.2 クレートを recap-worker の `[dev-dependencies]` に追加。Java 依存ゼロ（ネイティブ Rust 実装）。

テストは `#[ignore]` 属性で通常の `cargo test` から分離し、`cargo test --lib contract -- --ignored` で実行。既存の `wiremock` ベースの unit テスト（286 テスト）との共存を確認済み。

4 Provider に対する Consumer テストを実装：

| Provider | エンドポイント | テスト数 |
|----------|-------------|---------|
| news-creator | `/v1/summary/generate`, `/v1/summary/generate/batch` | 3 |
| recap-subworker | `/v1/classify-runs`, `/v1/classify/coarse`, `/v1/clustering/{id}` | 4 |
| alt-backend | `GET /v1/recap/articles` | 1 |
| tag-generator | `/api/v1/tags/batch`, `/api/v1/extract-tags` | 2 |

### 3. Go Consumer テスト拡張（search-indexer、mq-hub）

search-indexer に 2 Provider に対する Consumer テストを追加：

| Provider | プロトコル | テスト数 |
|----------|-----------|---------|
| alt-backend | Connect-RPC（BackendInternalService） | 2 |
| recap-worker | HTTP/REST（`/v1/recaps/genres/indexable`） | 1 |

### 4. Python Consumer テスト（recap-evaluator）

`pact-python` v3.2.1 を recap-evaluator の dev dependencies に追加。recap-worker の評価 API（`/v1/evaluation/genres`）に対する 3 interactions の Consumer テストを実装。

### 5. Redis Streams メッセージ契約（Pact V4 Message）

mq-hub で Pact V4 の `AsynchronousMessage` を使用し、Redis Streams のイベントスキーマを契約として定義。search-indexer 側では Message Consumer テストで受信側のデシリアライズ互換性を検証。

| Producer/Consumer | イベントタイプ | テスト数 |
|-------------------|-------------|---------|
| mq-hub → search-indexer | ArticleCreated（thin/fat） | 2 |
| mq-hub → tag-generator | TagGenerationRequested | 1 |
| search-indexer ← mq-hub | ArticleCreated, IndexArticle | 3 |

### 6. Provider 検証テスト

新規 Provider となるサービスに検証テストを追加：

- recap-subworker: FastAPI スタブサーバーで 4 interactions を検証
- tag-generator: FastAPI スタブサーバーで 2 interactions を検証
- alt-backend: Go テストサーバーで REST API の 1 interaction を検証

### 7. BFF（alt-butterfly-facade）の除外

alt-butterfly-facade は alt-backend への透過プロキシであり、リクエスト/レスポンスの変換を行わない。Buf breaking + FE Proto Conformance テストで十分カバーされるため、CDC テストの対象外とした。

### 8. CI パイプライン拡張

| ジョブ | 内容 |
|--------|------|
| `pact-cdc-go` | matrix に search-indexer、mq-hub を追加 |
| `pact-cdc-rust` | 新設。recap-worker の Rust Consumer テスト |
| `pact-cdc-python-consumer` | 新設。recap-evaluator の Python Consumer テスト |
| `pact-cdc-python` | recap-subworker、tag-generator の Provider 検証を追加 |

## Consequences

### Pros

- Alt の全 HTTP/Connect-RPC/Redis Streams 通信パスが Pact 契約で保護され、サービス間の暗黙的な契約破壊を CI で自動検知できるようになった
- Rust（`pact_consumer`）の導入により、Go/Python/Rust の 3 言語で統一的な CDC テストフレームワークが確立された
- Pact Broker 統合により、契約の可視化（Broker UI）、バージョン管理、ブランチタグ付けが可能になった
- 既存テスト（Go 286 テスト、Python テスト）への影響ゼロ。ビルドタグ/ignore 属性で完全分離
- Redis Streams のイベントスキーマが初めて契約として明示化され、非同期通信の破壊も検知可能になった
- 全テスト合計実行時間: Go < 0.1s、Rust < 0.15s、Python < 1s（mock サーバーベースのため高速）

### Cons / Tradeoffs

- `pact_consumer` クレートの追加により recap-worker のコンパイル時間が増加（初回ビルド +52s）
- CI のジョブ数が 4 → 7 に増加（ただし matrix 実行で並列化）
- Pact Broker の CI サービスコンテナにより `pact-cdc-python` ジョブの起動時間が +15s
- recap-worker の Provider 検証（search-indexer、recap-evaluator からの契約検証）は Axum サーバーの起動が必要で、Rust テストインフラの複雑性が増す
- `can-i-deploy` ゲートは Broker 統合の基盤としてのみ準備済み。永続的な Broker インスタンスが稼働してからデプロイゲートとして有効化する必要がある
- Streaming RPC の Pact 検証は引き続き Content-Type レベルに限定（NDJSON チャンク単位の検証は非対応）
