# ADR-000830: news-creator-backend の OLLAMA_MAX_LOADED_MODELS 既定値を 1 から 2 に引き上げて RAG 生成モデルと Meilisearch hybrid-search embedder を共存させる

## Status

Accepted

## Date

2026-04-23

## Affected Services

- news-creator-backend (Ollama) — `compose/ai.yaml` の環境変数既定値を 1 行変更。コンテナ本体コード・イメージは変わらない。
- meilisearch — 挙動面で恩恵を受ける側（hybrid search の wall-clock レイテンシが安定）。設定は触らない。
- search-indexer / alt-backend — 変更なし。[[000829]] が残した embed cold reload への追加対応。

## Context

[[000829]] で `/search?q=n8n` の articles 欠落（`degraded=[articles]` バナー）は消え、alt-backend 側は
`global search ok query=n8n degraded=null` を返すようになった。一方でユーザ体感では依然検索が遅く、
Meilisearch ログでは `POST /indexes/articles/search` のスパン idle time が数秒級になる事例が観測されていた。

内訳の分解:

- Meilisearch articles index は 10 万件オーダの文書に embedding を付けた hybrid search 構成で、
  検索のたびに embedder を呼び出してクエリベクトルを得る。
- embedder は news-creator-backend の Ollama に登録されており、RAG 生成モデルと **同一の Ollama runtime**
  を共有している。ここで `OLLAMA_MAX_LOADED_MODELS=1` の制約がかかっており、生成モデルと埋め込みモデルは
  **同時にはロードできない**。
- その結果、検索が embed を要求するたびに Ollama は生成モデルを evict → 埋め込みモデルをロードして embed。
  次に RAG パスが呼ばれると逆方向の evict/reload が走り、スラッシング状態になる。
- 直接計測で内訳を切り分けた結果:
  - keyword-only search: 20〜30 ms
  - warm hybrid search (埋め込みモデルがロード済): 200〜300 ms
  - cold hybrid search (evict → load → embed): 先頭の埋め込みモデルロードで秒オーダ、初回 embed で追加 1〜2 秒
    が重なり、span 全体で 3〜4 秒

補助的な事実:

- 埋め込みモデルは既に news-creator-backend 側に pull 済で、ディスク上にも存在する。
- 公式 [Ollama FAQ](https://github.com/ollama/ollama/blob/main/docs/faq.md#how-does-ollama-handle-concurrent-requests) は、
  同一インスタンスで複数モデルを使う構成では `OLLAMA_MAX_LOADED_MODELS` を 2 以上に設定してロード保持数を広げることを推奨している。
- Meilisearch の search queue と write queue は別系統（[issue #4489](https://github.com/meilisearch/meilisearch/issues/4489) / PR #4536）。
  今回の秒オーダ遅延は [[000829]] で扱った synonyms 書き込みキュー詰まりでは説明できず、**embedder RPC の await** で消費される。

## Decision

`compose/ai.yaml` の news-creator-backend コンテナの `OLLAMA_MAX_LOADED_MODELS` 既定値を
`${OLLAMA_MAX_LOADED_MODELS:-1}` から `${OLLAMA_MAX_LOADED_MODELS:-2}` に引き上げる。

`.env` override のインタフェースはそのまま残すので、運用者の GPU が組合せに耐えられないと判明した場合は
`.env` に `OLLAMA_MAX_LOADED_MODELS=1` を書くだけで旧挙動に戻せる。

あわせて直前のコメントを、「1 モデルに絞ってロードを強制」する旧意図から「生成と埋め込みを共存させる意図と、
`.env` で旧値に戻す運用的 escape hatch」が分かる文面に書き換える。コード・イメージビルド・他コンテナは触らない。

### 検討した代替案と却下理由

- **(A) 埋め込みモデルを別ホストの Ollama に移設**  
  運用者が管理する別の自ホスト Ollama インスタンスに埋め込みモデルを pull し、Meilisearch の embedder URL を
  そちらに向ける案。GPU を構造的に分離できる利点はあるが、altctl-mac 側の CLI 拡張 (`model pull` / `tune-env`)
  を TDD で新設、Meilisearch embedder 設定の PATCH、別ホスト側の運用手順追加、と工数が本 ADR の 1 行変更に
  比べて 1〜2 桁大きい。GPU に組合せが収まらないと判明したときの第二段階として保留する。
- **(B) 別ホストの Ollama に移設しつつ軽量な埋め込みモデルに変える**  
  別ホストに既に存在する `mxbai-embed-large` / `embeddinggemma` 等を使う案。次元数差 (1024 → 768 等) を
  吸収するため **articles index の全 embedding を再計算する必要** があり、長い非同期 task が必要。
  さらに検索品質も変動する。長期の選択肢として保留。
- **(C) search-indexer driver で動的 semanticRatio フォールバック**  
  短い英数字クエリでは `semanticRatio=0`（keyword-only）にフォールバックし、長文クエリのみ hybrid を使う案。
  Go 側のコード変更で済むが、「キーワードに見えて意味検索の恩恵もほしい」ケース (例: `n8n`) を失う副作用があり、
  また RAG 側の cold reload は残るため、根本対処にならない。
- **(D) `OLLAMA_MAX_LOADED_MODELS=1` のまま `OLLAMA_KEEP_ALIVE` を長くする**  
  既に 24h。keep_alive は同一モデルの保持時間の話で、**別モデル要求による evict** は防げない。効かない。
- **(E) hybrid search を無効化**  
  意味検索の UX が消える。却下。

本 ADR の選択は「最短で cold reload を消す。GPU が耐えられなければ `.env` override or revert で戻し、
別ホスト移設 (代替案 A) に進む」段階的な 1 step。

## Consequences

### Pros

- Meilisearch hybrid search の wall-clock レイテンシが cold reload 分の秒オーダぶん短縮され、warm 相当
  (数百 ms 帯) に寄る想定。`/search?q=n8n` 含むユーザ体感の遅さが解消される見込み。
- RAG 側にも恩恵。以前は embedding 呼び出し直後の RAG クエリが逆方向の evict/reload を踏んでいたのが消える。
- compose の 1 行変更 + `.env` override 互換 + revert 1 行の 3 レイヤで rollback 可能。運用影響は最小。
- [[000829]] の sectionTimeout 10s に加えて、articles section の実所要時間が余裕を持って timeout 内に収まる
  ので、Meilisearch 詰まり由来の `degraded=[articles]` 再発を二重に塞ぐ。

### Cons / Tradeoffs

- 2 モデル同時ロードは GPU 余裕が前提。運用者のハードウェアによっては組合せが収まらず、Ollama が partial
  CPU offload や OOM を起こす可能性がある。検知手段は Ollama `/api/ps` で models の入れ替わり頻度、
  GPU telemetry、news-creator-backend healthcheck の連続失敗、`dmesg` の OOM kill。
- rollback は `.env` override または revert コミットで可能。[[000829]] の timeout 10s の余裕があるので、
  仮に評価中に詰まったとしても degraded に一時退行するだけで UI は生き続ける。
- 生成モデル側の KV cache が増える状況 (長文 prompt 連続) では GPU 使用がさらに伸びる余地があり、埋め込み
  モデルが押し出される可能性は残る。`OLLAMA_NUM_PARALLEL=1` を維持しているのは併用条件のひとつ。
- 長期的には GPU を分離したい (代替案 A / B)。本 ADR はその前段の cheap win。

## Related ADRs

- [[000829]] `/search?q=n8n` の articles 欠落を解消するため search-indexer の synonyms PUT をバッチ coalesce し alt-backend の global-search section timeout を 10 秒に引き上げる（本 ADR の直接の前提）

## Deploy-model 整合性セルフチェック

本 ADR は `compose/ai.yaml` の既存 service `news-creator-backend` の environment を 1 行変えるだけで、
service を新設も削除もしない。`depends_on` / `restart` / volume / image には触らない。per-service rolling
deploy model に対しての追加リスクは無い。

- [ ] 新設 compose service が `depends_on` で他 service を gate するか？
  - N/A (service を追加しない)
- [ ] 新設 compose service に `restart: "no"` の one-shot init container を含むか？
  - N/A (追加しない)
- [ ] 新設名前付き volume を populate する責務はどこにあるか？
  - N/A (volume を触らない)
- [ ] いずれも該当しない: 「rolling 互換」と明記して pass
  - **YES. rolling 互換.** news-creator-backend を単独 rolling replace しても、新コンテナは起動時に Ollama を
    起動し、生成モデルと埋め込みモデルを demand-load するだけ。旧コンテナと新コンテナが短時間同時に走るケースでも
    listener ポートは最後に起動したものが受けるだけで、互換性の非互換は発生しない。
