# ADR-000717: alt-backend の Critical 認証欠陥を修正し /v1/internal/* と /v1/dashboard/* を閉じる

## Status

Superseded by [[000743]] — X-Service-Token / SERVICE_SECRET を全サービスから撤去し認証境界を mTLS peer identity に一本化したため、本 ADR が確立した shared-secret 多層防御は廃止された。

## Date

2026-04-14

## Affected Services

- alt-backend (Go) — `middleware/auth_middleware.go` の `RequireAuth` / `OptionalAuth` / 新規 `RequireAdmin`、`middleware/service_auth_middleware.go` の定数時間比較化、`rest/internal_handlers.go` への `RequireServiceAuth` 付与、`rest/dashboard_handlers.go` のグループ化
- pre-processor (Go) — `config/types.go` に `AltServiceConfig.ServiceToken` 追加、`config/load.go` で `SERVICE_TOKEN` / `SERVICE_TOKEN_FILE` を読み込み、`repository/external_api_repository.go` の `/v1/internal/system-user` 呼び出しに `X-Service-Token` ヘッダを付与
- rag-orchestrator (Go) — `adapter/altdb/article_client.go` の `HTTPArticleClient` に `serviceToken` 引数を追加、`/v1/internal/articles/recent` 呼び出しに `X-Service-Token` ヘッダを付与、`di/container.go` で `cfg.Backend.ServiceToken` を渡す

## Context

2026-04-14 に実施した alt-backend の OWASP Top 10:2025 ベースのセキュリティレビュー (`docs/review/alt-backend-security-review-2026-04-14.md`) で、認証境界に関する Critical 3 件と High 1 件が特定された。

1. **C-001 — `/v1/internal/*` に認証ミドルウェアが付いていない**: `rest/internal_handlers.go` の `e.Group("/v1/internal")` は「network policy で制限を推奨」というコメントのみで、`ServiceAuthMiddleware.RequireServiceAuth()` が付与されていなかった。結果として、同一 Docker network に到達できる任意のコンテナから `/v1/internal/system-user` (Kratos の system user ID 漏洩) と `/v1/internal/articles/recent?limit=500` (全ユーザー記事メタデータ最大 500 件) が無認証で取得できる状態だった。[[000618]] が確立した「Public API = X-Alt-Backend-Token (JWT) / Internal API = X-Service-Token」という認証境界の原則から、internal routes はその境界の逆側にあるべきだった。

2. **C-002 — `/v1/dashboard/*` に認証がかかっていない**: `rest/dashboard_handlers.go` は `registerDashboardRoutes(v1, container)` で登録され、`v1` グループには `PathProbe` しか付与されていない。SvelteKit の `api/v1/dashboard/*` ルートは既に `X-Alt-Backend-Token` を付けて BFF 経由で呼び出していた ([[000395]] の統合方針どおり) が、alt-backend 側が検証していなかったため JWT なしでも運用メトリクス・ジョブ状態・アプリケーションログ最大 2000 件が取得可能だった。

3. **C-003 — JWT subject クレームの `uuid.Parse` エラーを握りつぶしていた**: `middleware/auth_middleware.go` の `RequireAuth` / `OptionalAuth` は `userID, _ := uuid.Parse(jwtUserCtx.ID)` でエラーを破棄していた。攻撃者が有効な署名を持つ JWT の `sub` を `"admin"` や任意文字列に細工すると、`uuid.Nil` を UserID として後段へ渡せる可能性があった。`domain.UserContext.IsValid()` は Nil UUID を弾くため最終的な被害は限定されるものの、`c.Set("auth.valid", true)` は既に立っており、防御線が一段崩れていた。

4. **H-002 — Service Token の比較が非定数時間**: `service_auth_middleware.go:78` は `token != m.serviceSecret` の素朴な文字列比較で、タイミング攻撃に対し脆弱だった。隣接コンテナからであれば数十ナノ秒差でプレフィクス長が推定できる。

これらは「すでに攻撃面として露出している」という意味で他の Medium/Low 案件よりも優先度が高く、同日中に修正することにした。

認証方式を決めるにあたり、**「`X-Service-Token` (shared secret header) は microservices ベストプラクティスか」** をあらためて Web で調査した (OWASP Microservices Security Cheat Sheet, SPIFFE/SPIRE, Istio Ambient)。OWASP が明示的に推奨するのは **(a) mTLS** と **(b) 署名付きトークン (caller ID と scopes を含む JWT を STS が発行)** の二つで、シンプルな shared-secret custom header は「acceptable だが best practice ではない」位置付けだった。Alt プロジェクトの実情として、Linkerd mTLS が `sidecar-proxy` で提供済み、単一 Docker Compose ネットワーク、`SERVICE_SECRET` は既に Docker Secrets 経由で注入、という条件を踏まえれば、`X-Service-Token` を application-layer の追加防御として維持するのは合理的と判断した。ただし caller 識別ができない点は将来課題として残る。

## Decision

以下の四点をまとめて適用する。

1. **`/v1/internal/*` に `ServiceAuthMiddleware.RequireServiceAuth()` を付与する** — [[000618]] の認証境界を忠実に再実装する。「network policy に頼る」アプローチは defense-in-depth の観点で不十分であり、アプリケーション層でも `X-Service-Token` を必須化する。同時に `service_auth_middleware.go` の比較ロジックを `crypto/subtle.ConstantTimeCompare` に切替え、長さ違いのトークンは事前に弾きつつ等長比較時は定数時間で評価する。

2. **`AuthMiddleware.RequireAdmin()` を新設し `/v1/dashboard/*` に二段階適用する** — 運用ダッシュボードは「認証済かつ管理者ロール」を要求する。既存の `domain.UserContext.IsAdmin()` (role == `admin`) を再利用し、未認証は 401、ロール不一致は 403 を返す。`rest/dashboard_handlers.go` を `v1.Group("/dashboard", authMiddleware.RequireAuth(), authMiddleware.RequireAdmin())` でグループ化し、下位パスを明示的にネストする。

3. **JWT subject クレームの UUID 検証を分離した `parseSubjectUUID()` で行う** — 非 UUID 文字列と Nil UUID の両方を拒否する小さなヘルパーを `middleware/auth_middleware.go` に置き、`RequireAuth` ではエラー時に 401、`OptionalAuth` では匿名継続 (V-005 の既存哲学に合わせる) とする。エラーは握り潰さず `slog` に warn として記録する。

4. **内部 API 呼び出し元 (pre-processor, rag-orchestrator) に `X-Service-Token` 送信を追加する** — alt-backend 側だけ閉めるとバッチ系・RAG 系が 401 で止まるため、同一コミットで呼び出し元も修正する。どちらのサービスも `SERVICE_SECRET` / `SERVICE_TOKEN` を既に扱う基盤があり (pre-processor は Connect-RPC 用 `backend_api.Client`、rag-orchestrator は Connect-RPC 用 `serviceTokenInterceptor`)、ただし REST 呼び出し経路がその基盤から外れていた。REST クライアント側にも明示的に token を渡すよう配線する。

### 検討した代替案

- **代替案 A: /v1/internal/* を別 Echo インスタンス・別ポートに分離** — ネットワーク分離としては理想だが、現状の compose 構成を大きく変更する必要があり、今回の Critical 修正のスコープを超える。将来の改善候補として残す。
- **代替案 B: mTLS のみで認証を済ませ X-Service-Token を廃止** — Linkerd で mTLS は既に有効だが、mTLS は「接続元ワークロード」を識別するものであり、alt-backend のアプリケーション層が caller を確認するには SPIFFE ID をアプリに伝播させる追加設計が必要。今回のスコープでは application-layer の追加防御を維持する。
- **代替案 C: /v1/dashboard/* を完全に閉じて BFF 内ダッシュボードに移す** — ダッシュボード自体が運用向け UI の中核なので廃止は非現実的。RequireAuth + Admin の二段防御で十分とした。
- **代替案 D: C-003 で OptionalAuth も 401 にする** — V-005 の既存設計 (Invalid JWT は anonymous として続行) との整合を優先し、OptionalAuth は非 UUID subject も anonymous として扱う。`domain.UserContext` が attach されないので後段 usecase も正しく anonymous 扱いする。

## Implementation Overview

### 1. alt-backend: JWT subject の UUID 検証 (C-003)

```go
// middleware/auth_middleware.go
func parseSubjectUUID(sub string) (uuid.UUID, error) {
    id, err := uuid.Parse(sub)
    if err != nil {
        return uuid.Nil, fmt.Errorf("subject is not a valid UUID: %w", err)
    }
    if id == uuid.Nil {
        return uuid.Nil, errors.New("subject is the nil UUID")
    }
    return id, nil
}
```

`RequireAuth` ではエラー時 401 + warn ログ、`OptionalAuth` では匿名継続 + warn ログ。テスト (`auth_middleware_headers_test.go`) に非 UUID subject・Nil UUID subject・OptionalAuth 匿名扱いの三ケースを追加した。

### 2. alt-backend: RequireAdmin と dashboard 保護 (C-002)

```go
// middleware/auth_middleware.go
func (m *AuthMiddleware) RequireAdmin() echo.MiddlewareFunc {
    return func(next echo.HandlerFunc) echo.HandlerFunc {
        return func(c echo.Context) error {
            user, err := domain.GetUserFromContext(c.Request().Context())
            if err != nil || user == nil {
                return echo.NewHTTPError(http.StatusUnauthorized, "authentication required")
            }
            if !user.IsAdmin() {
                return echo.NewHTTPError(http.StatusForbidden, "admin role required")
            }
            return next(c)
        }
    }
}

// rest/dashboard_handlers.go
dashboard := v1.Group("/dashboard", authMiddleware.RequireAuth(), authMiddleware.RequireAdmin())
dashboard.GET("/metrics", handleGetMetrics(container.DashboardMetricsUsecase))
// ... 他の 4 ハンドラも同じグループ下に移動
```

テスト (`auth_middleware_headers_test.go`) に admin 通過・user 拒否 (403)・user context なし拒否 (401) の三ケースを追加。

### 3. alt-backend: /v1/internal/* の ServiceAuth 付与と定数時間比較 (C-001 + H-002)

```go
// rest/internal_handlers.go
serviceAuth := middleware_custom.NewServiceAuthMiddleware(logger.Logger)
v1 := e.Group("/v1/internal", serviceAuth.RequireServiceAuth())
v1.GET("/system-user", ...)
v1.GET("/articles/recent", handleFetchRecentArticles(container))

// middleware/service_auth_middleware.go
expected := []byte(m.serviceSecret)
provided := []byte(token)
valid := len(expected) == len(provided) &&
    subtle.ConstantTimeCompare(expected, provided) == 1
```

テスト (`internal_handlers_auth_test.go`) に「X-Service-Token なし → 401」「誤った token → 401」「SERVICE_SECRET_FILE 経由読み込み」の三ケースを追加。既存の `service_auth_middleware_test.go` は同等仕様を維持しつつ差分長テストを明示的に含める。

### 4. pre-processor: REST クライアントの X-Service-Token 付与 (C-001 coordinated fix)

`AltServiceConfig` に `ServiceToken` フィールドを追加し、`loadAltServiceConfig` から `SERVICE_TOKEN` / `SERVICE_TOKEN_FILE` を読み込む。Docker Secrets 優先の挙動は既存パターンに合わせた。

```go
// config/load.go
func loadServiceTokenEnv() string {
    if filePath := os.Getenv("SERVICE_TOKEN_FILE"); filePath != "" {
        if content, err := os.ReadFile(filePath); err == nil {
            return strings.TrimSpace(string(content))
        }
    }
    return os.Getenv("SERVICE_TOKEN")
}

// repository/external_api_repository.go
if token := r.config.AltService.ServiceToken; token != "" {
    req.Header.Set("X-Service-Token", token)
}
```

テスト (`external_api_repository_test.go`) に「token 設定時はヘッダ送信」「token 未設定時はヘッダ省略」の二ケースを追加した。

### 5. rag-orchestrator: HTTPArticleClient に serviceToken を注入

既存の Connect-RPC クライアントは `serviceTokenInterceptor` 経由で `X-Service-Token` を付けていたが、REST 呼び出しの `HTTPArticleClient` はその経路を通っていなかった。コンストラクタシグネチャを変更して明示的に token を受け取る。

```go
// adapter/altdb/article_client.go
func NewHTTPArticleClient(baseURL string, timeout time.Duration, serviceToken string, logger *slog.Logger) *HTTPArticleClient {
    return &HTTPArticleClient{
        baseURL:      baseURL,
        serviceToken: serviceToken,
        httpClient:   &http.Client{Timeout: timeout},
        logger:       logger,
    }
}

// di/container.go
articleClient := altdb.NewHTTPArticleClient(
    cfg.Backend.URL,
    time.Duration(cfg.Backend.Timeout)*time.Second,
    cfg.Backend.ServiceToken,
    log,
)
```

`cfg.Backend.ServiceToken` は既に `infra/config/config.go` で `SERVICE_TOKEN` / `SERVICE_TOKEN_FILE` から読み込まれていたため追加の config 変更は不要だった。テスト (`article_client_test.go`) を新設し「token 設定時はヘッダ送信」「空 token 時はヘッダ省略」を検証している。

### 6. ランタイム確認

`docker compose -f compose/compose.yaml -p alt up --build -d alt-backend pre-processor rag-orchestrator` で 3 サービスを再ビルド・起動後、以下を確認した。

- `GET /v1/health` → 200 (変更なし)
- `GET /v1/internal/system-user` (X-Service-Token なし) → 401 ← 修正前は 200 が返っていた
- `GET /v1/dashboard/metrics` (X-Alt-Backend-Token なし) → 401 ← 修正前は 200 が返っていた

全 3 サービスのコンテナが `healthy` / `Up` で安定稼働し、pre-processor の article sync が起動後も継続している (X-Service-Token が正しく伝播している)。

## Consequences

### Pros

- **Critical 3 件が同日中にクローズ**: レビュー報告に記録された C-001 / C-002 / C-003 が解消され、無認証で取得可能だったユーザー識別子・記事メタデータ・運用ログが閉じた。加えて H-002 (タイミング攻撃) も副次的に修正済。
- **ADR-000618 の境界原則が実装レベルで一貫**: Public API = JWT / Internal API = Service Token の二極化が、設計文書上だけでなくコード上でも例外なく適用された。将来 internal route を追加する際の設計ミスが起きにくい。
- **Dashboard が admin ロール必須に**: UI 経由で誤って一般ユーザーが運用ログを閲覧できる経路がなくなった。認可ロジックが handler 外の middleware に集約されたため、個別ハンドラが認可を忘れる事故も防げる。
- **JWT subject 検証が明示化**: `parseSubjectUUID` を共通ヘルパーにしたことで、今後ほかの場所で `sub` を UUID として扱う時も同じ検証ロジックを再利用できる。
- **定数時間比較によるサイドチャネル緩和**: shared secret の長さ推定攻撃が現実的でなくなった。Linkerd mTLS との組合せで「どちらか一方が破られても他方が残る」多層防御になる。
- **TDD 原則を維持**: すべての修正で RED テストを先に書き、GREEN に倒してから次の修正に進む順序を厳守した。回帰テスト (alt-backend 全体) も通過している。

### Cons / Tradeoffs

- **内部 API 呼び出し元は SERVICE_TOKEN を正しく設定する必要がある**: pre-processor と rag-orchestrator の `SERVICE_TOKEN` (あるいは `SERVICE_TOKEN_FILE`) が alt-backend の `SERVICE_SECRET` と一致していなければ全 internal RPC が 401 で止まる。secret 名が呼び出し側 (`SERVICE_TOKEN`) と受け側 (`SERVICE_SECRET`) で異なるため、compose / env 運用で同期させる手順を必ず徹底する必要がある。
- **caller 識別ができない状態は残る**: 今回は shared secret を維持したため、alt-backend のログから「どのサービスが internal API を呼んだか」を区別できない。将来、per-caller の JWT (iss=pre-processor / rag-orchestrator、sub=service UUID) に移行する余地を残す。
- **dashboard の admin ロール運用が必須に**: JWT 発行側 (auth-hub) が role=admin を正しく発行する設定が前提。ロール付与が未整備の環境では dashboard が使えなくなるため、運用ロール付与の手順整備が必要。
- **Medium / Low の 17 件は別タスクに持ち越し**: レビュー報告の残 17 件 (BodyLimit 未設定・CSP のハード化・SSRF の設計整理・rate limiter の未バウンド・tenantID = userID など) は今回の PR に含めていない。順次対応する。
- **ADR-000618 の「Connect-RPC Internal API に統一」方針との両立**: Connect-RPC 側は既に `ServiceAuthMiddleware` を通していたため、REST 側が遅れて追随した形。Connect-RPC への統一が進めば REST internal エンドポイントは最終的には廃止対象となる。

### 将来の検討課題

- caller 識別のため、per-caller 署名付きトークン (JWT iss/sub) または SPIFFE ID 伝播への移行。
- `parseSubjectUUID` を使って JWT に `tenant_id` クレームを明示追加し、`tenantID = userID` の single-tenant 暫定実装 (レビュー報告 H-003) を解消する。
- dashboard 認可を `admin` 単一ロールから RBAC (scope-based) に発展させる。
- BodyLimit 未設定 (レビュー報告 H-005)、rate limiter クリーンアップ (M-006)、DoS 保護の RWMutex 修正 (H-004) を順次 ADR 化して対応。
