# ADR-000852: Knowledge Loop projector に Prometheus 観測基盤を導入し Surface Planner v2 を null-resolver で安全に live wiring する

## Status

Accepted

## Date

2026-04-26

## Affected Services

- knowledge-sovereign (Go):
  - `app/go.mod` に `github.com/prometheus/client_golang v1.23.2` を direct dep として追加
  - `app/main.go` に `promhttp.Handler()` を追加し既存 metrics server (`cfg.MetricsAddr`) で `/metrics` を公開
  - `app/usecase/knowledge_loop_projector/metrics.go` 新設、4 つの Knowledge Loop projector counter を vocab として確定
  - `app/usecase/knowledge_loop_projector/score_resolver.go` 新設、`SurfaceScoreResolver` port + `NullSurfaceScoreResolver` default + `Projector.resolveBucket` を実装
  - `app/usecase/knowledge_loop_projector/projector.go` の `Projector` struct に `scoreResolver` field、`NewProjector` で Null 初期化、`WithScoreResolver` option を追加。`projectEvent` の 4 つの hardcoded-bucket 分岐をすべて `resolveBucket` 経由に置換、SummarySuperseded path に `observeChangeSummaryWritten` を wired
  - `app/usecase/knowledge_loop_projector/score_resolver_test.go` 新設、Null/v2/nil-ignore の 3 観点を 4 テスト + 7 サブテストで pin

## Context

[[000851]] (Wave 2) で `decideBucketV2` / `computeChangeDiff` / `buildChangeSummaryJSON` を pure function として landed させ、unit test で挙動を凍結した。残課題は次の 3 点だった:

1. **pure function を実 write path に組み込む**: pure 関数として正しいことは固定したが、`projectEvent` から呼ばれない限り live で何も起きない。組み込み方によっては bucket 配置の挙動が変わってしまう (live 影響) リスクがあるため、慎重に additive な seam を作る必要がある
2. **rollout を観測する手段がない**: Wave 1/2 で proto 拡張・runbook 更新まで進めたが、知識-sovereign に Prometheus 計測の枠組みが無く、Surface Planner v2 や ChangedDiffCard redline がどれだけ live で起きているか scrape できなかった。F-001 (cross-user evidence_refs leak)・F-002 (summary user_id mismatch)・F-003 (system event の権限昇格) に対する alert-on-positive な counter も未整備
4. **Wave 4 の seam が未確定**: cross-service resolver (alt-backend の `tag_set_versions` / `recap_topic_snapshots` / `augur_conversations` を user_id bind で query するもの) を後続で plug する場所が定義されていなかった。port を後出しすると blast radius が広がりやすい

これらを Wave 3 の同セッションで一気に解くと「行動を変える wiring + 観測 + 拡張点」が混じって PR が大きくなりすぎる。そこで本 ADR では **「現状の挙動は bit-for-bit 同じだが、Wave 4 で resolver を plug した瞬間に v2 placement と redline rollout が観測可能になる」** という、live 影響ゼロかつ前進ありのスナップショットを landed させる。

## Decision

Knowledge Loop projector に Prometheus client を導入し、`SurfaceScoreResolver` port を介して `decideBucketV2` を `projectEvent` に組込む。default 実装 `NullSurfaceScoreResolver` は何も query せず event の `EventType` + `OccurredAt` だけを返すので、`decideBucketV2` は v1 mapping にフォールバックし live behavior は変わらない。

### 1. Prometheus 観測基盤の導入

`github.com/prometheus/client_golang v1.23.2` を knowledge-sovereign の direct dep に追加し、既存 metrics server (`cfg.MetricsAddr`) に `/metrics` ハンドラを追加した。projector 直近で必要な counter を 4 つだけ確定し、それぞれの label vocab を低 cardinality に絞った:

- `knowledge_loop_projector_surface_bucket_assigned_total{version,bucket}`: v1 / v2 placement の比率を時系列で見える化
- `knowledge_loop_projector_change_summary_written_total{redline_capable}`: ChangedDiffCard rollout の進捗 (Then/Now → redline)
- `knowledge_loop_projector_event_dropped_total{reason}`: F-002 / F-003 enforcement。reason ラベルは `DropReason*` 定数で固定済み
- `knowledge_loop_projector_cross_user_isolation_violation_total`: F-001 alert-on-positive。常に 0 を期待し非ゼロが alert を発火させる

`bucketMetricLabel` で proto enum → "now"/"continue"/"changed"/"review"/"unspecified" に bounded mapping、reason ラベルは `DropReasonNoUserID` / `DropReasonInvalidEntryKey` / `DropReasonUserMismatch` / `DropReasonArticleMismatch` / `DropReasonUnknownEventType` の 5 値で固定する。これにより dashboard の panel filter が膨張しない。

`observeChangeSummaryWritten(redlineCapable)` は既に Wave 2 で landed していた `slog.DebugContext` の隣に追加した。debug log 一本だけだと本番の info-only 設定で観測点が消える脆さがあったため、metric counter で常時可視化する形に補強する。

### 2. SurfaceScoreResolver port + NullSurfaceScoreResolver default

```go
type SurfaceScoreResolver interface {
  Resolve(ctx context.Context, ev *sovereign_db.KnowledgeEvent) SurfaceScoreInputs
}
```

実装の reproject-safety 契約をコメントで明文化:
- `*_versions` テーブルと event payload のみを参照する
- 全 cross-source query は `user_id` (および article_id) で bind する。F-001 mitigation はこの seam で受ける
- 時間窓は `event.occurred_at` に bind し、`time.Now()` を business fact に登場させない

`NullSurfaceScoreResolver` は `EventType` と `FreshnessAt = ev.OccurredAt` だけを設定して空の SurfaceScoreInputs を返す。これにより `decideBucketV2` が必ず v1 fallback パスに入り、Wave 1/2 までの挙動と完全に一致する。

`Projector` struct に `scoreResolver` を持たせ、`NewProjector` で Null 初期化、`WithScoreResolver(r SurfaceScoreResolver)` で後から差し替えできる option パターンに揃えた (`r == nil` のときは既存 resolver を温存)。`Projector.resolveBucket(ctx, ev)` は resolver を呼んで `decideBucketV2` を回し、`observeSurfaceBucketAssigned(version, bucket)` で counter を bump する。`hasV2Signal` で「v2 evidence のいずれかがゼロ非」のときだけ version label を `"v2"` に切替えるので、Null default では常に `"v1"` が記録される。

### 3. projectEvent の hardcoded bucket を resolveBucket 経由へ置換

`projectEvent` で従来 hardcoded していた 4 case (`SummaryVersionCreated` / `HomeItemsSeen` / `HomeItemAsked` / `HomeItemOpened` / `HomeItemDismissed` / `HomeItemSuperseded` / `SummarySuperseded`) を、すべて `bucket := p.resolveBucket(ctx, ev)` → `p.buildEntryFromEvent(..., bucket)` の形に書き換えた。`SummarySuperseded` 系は `superseded_by_entry_key` 抽出と `change_summary` JSONB 構築 ([[000851]]) も合流させた状態で resolveBucket 経由になる。

### 4. テスト

`score_resolver_test.go` を新設し以下を pin した:

- `TestNullSurfaceScoreResolver_Resolve`: Null resolver が cross-source 値をすべて 0 / false で返すこと、`EventType` と `FreshnessAt` だけが echo されること
- `TestProjector_ResolveBucket_NullDefault` (7 event-type サブテスト): `SummaryVersionCreated` / `HomeItemsSeen` / `HomeItemAsked` → NOW、`HomeItemOpened` → CONTINUE、`HomeItemDismissed` → REVIEW、`HomeItemSuperseded` / `SummarySuperseded` → CHANGED が Null resolver 経由で得られること (= v1 mapping と bit-equivalent)
- `TestProjector_ResolveBucket_V2EvidencePromotesToChanged`: `fakeResolver{out: SurfaceScoreInputs{VersionDriftCount: 1}}` を plug すると `EventHomeItemsSeen` (v1 NOW) でも CHANGED に promote すること
- `TestProjector_WithScoreResolver_NilIgnored`: `WithScoreResolver(nil)` が既存 resolver を尊重し上書きしないこと

### 検討した代替案と却下理由

- **resolver を `NewProjector(repo, logger, cfg, resolver)` の必須引数にする**: 既存 caller (handler / main) を全部修正する必要があり、Wave 3 PR の表面積が大きくなる。`NewProjector` の signature を保ち、option pattern の `WithScoreResolver` で plug する方が回帰検出が小さい
- **`scoreResolver = nil` を許容**: `resolveBucket` で nil チェック + フォールバックすると、忘れたら panic する経路ができてしまう。`NewProjector` で `NullSurfaceScoreResolver{}` を必ず set する pattern が一貫していて安全
- **Prometheus client ではなく自作の counter を slog で出すだけ**: log-based metrics は scrape のレイテンシ / cardinality / cost が悪化する。`/metrics` を出した方が Grafana / alert rule が一発で書ける
- **counter の名前を `loop_*` のような短い形にする**: scrape されてしまうと他サービスと衝突しやすい。`knowledge_loop` namespace + `projector` subsystem で衝突回避
- **`surface_planner_version` 列を SQL UPSERT で書き始める**: Wave 3 partial は live 挙動を bit-for-bit 同じにしたい。Null resolver では実質 v1 のままなので、列に書く値は default 1 のままで意味を保てる。Wave 4 で real resolver が plug されるタイミングで列の write も合流させる

## Consequences

### Pros

- live behavior が bit-for-bit 同じまま、Wave 4 で real resolver を plug した瞬間に v2 placement と redline rollout が動き始める。`WithScoreResolver` 一行で切替えられる
- Prometheus counter で F-001 (`cross_user_isolation_violation_total`) と F-002 / F-003 (`event_dropped_total{reason}`) が `> 0` 即 alert に紐付けられる。Wave 4 で resolver が cross-table read を始めたとき、user_id mismatch を本番側で検知できる
- v1 / v2 placement の比率が `surface_bucket_assigned_total{version,bucket}` で可視化できるので、Wave 4 の rollout 時に「想定通り v2 が増えているか」「v2 で特定 bucket だけ減ってないか」が dashboard で読める
- ChangedDiffCard の redline-capable 比率も常時可視化できる。upstream emitter が excerpts を含めるようになったら counter が立ち上がる
- pure function ([[000851]]) と wiring (本 ADR) を分けたことで、回帰が起きたら原因が「テスト凍結された pure 関数」か「wiring」かが切り分けやすい

### Cons / Tradeoffs

- counter 名 / label 値の vocab を後から狭めることはできるが、広げる方は dashboard / alert rule との同期が要る。`bucketMetricLabel` と `DropReason*` 定数の値追加は **dashboard panel の filter 同期を伴う運用負債** として残る
- Null resolver default は本番では常に "v1" version label の counter しか上がらない。Wave 4 で cross-service resolver を plug するまで「v2 placement が 0 件」が dashboard に出続けるので、空グラフが正常な期間が一定期間続く点を SRE 側で握る必要がある
- `client_golang v1.23.2` は go.sum を膨らませる (transitive deps を含めて 30+ 行追加)。dependabot による update PR を Wave 4 以降で日常的に処理する負担が出る
- `/metrics` endpoint は既存の認証なし metrics server (cfg.MetricsAddr) で公開される。本番ネットワークで scraper 以外からの到達を遮断する責任は infra 側 (firewall / NetworkPolicy) に残る
- `surface_planner_version` 列を SQL UPSERT で書く wiring は本 ADR では入れていない。Wave 4 で real resolver を plug するときに同時に landed させる必要があり、忘れると v2 placement が起きても DB 側に履歴が残らない不整合 window が生まれうる
