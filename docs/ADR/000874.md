# ADR-000874: SurfacePlanRecomputed の producer cron を knowledge-sovereign に追加し projector / planner / stream cadence を env で運用可能にする

## Status

Accepted

## Date

2026-05-02

## Affected Services

- **knowledge-sovereign (Go)** — surface_planner_cron 新設 + projector / planner tick env 化
- **alt-backend (Go)** — knowledge_loop stream の poll / heartbeat を env 化

## Context

ADR-000873 で `KnowledgeLoopSurfacePlanRecomputed` の projector branch (`PatchKnowledgeLoopEntrySurfacePlan`) を実装した。これにより event payload の `entry_inputs[]` に基づき projector が planner-only 列だけを patch する経路が完成した。しかしこの projector branch には対応する **producer** が monorepo 内に一切存在せず、idle 状態で静止していた。

検証結果 (Phase 1 探索):

```
$ grep -r "KNOWLEDGE_LOOP_SURFACE_PLAN_RECOMPUTED" --include="*.go" --include="*.py" --include="*.rs" --include="*.ts"
(zero matches outside the projector consumer)
```

fb.md issue 9 はこの状況を「Surface Planner v2 が後方参照に留まっている」と表現していた:

> 必要なのは、単なる score resolver ではなく、Knowledge context changed → affected entries を再評価 → SurfacePlanRecomputed → UI plane が動く、という明示的な再計画イベントです。

加えて、knowledge-sovereign の projector tick / planner tick / alt-backend の stream poll / heartbeat はすべて hardcoded で:

| 箇所 | 値 |
|---|---|
| `knowledge-sovereign/app/main.go:116` `loopTick` | 5s |
| `alt-backend/app/connect/v2/knowledge_loop/handler.go:27` `streamPollInterval` | 5s |
| `alt-backend/app/connect/v2/knowledge_loop/handler.go:28` `streamHeartbeatEvery` | 10s |

CI / staging / 本番で同じ cadence を強要するため、producer cron をテストする際の cadence チューニングや障害時の一時短縮ができない状況だった。

## Decision

### 1. surface_planner_cron 新パッケージで producer cron を実装

`knowledge-sovereign/app/usecase/surface_planner_cron/` 新設:

```
surface_planner_cron/
  cron.go        # Cron struct + RunBatch
  cron_test.go   # 7 unit cases (TDD outside-in を unit 段で完結)
```

`Cron.RunBatch(ctx)` は projector と同じ shape:

1. `GetProjectionCheckpoint("surface_planner_v2")` で前回処理済 seq を取得
2. `ListKnowledgeEventsSince(checkpoint, batchSize)` で signal 候補を pull
3. `signalEventTypes` (現状 `augur.conversation_linked.v1` のみ) で filter
4. `(tenant_id, user_id)` でグループ化、entry_key 単位で重複排除
5. グループごとに `entry_inputs[]` を組み立て、一つの `KnowledgeLoopSurfacePlanRecomputed` を emit
6. `UpdateProjectionCheckpoint(maxSeq)` で進める

dedupe_key は `surface_plan_recomputed.v1:<user>:<lens_mode>:<batch_max_seq>` を採用。同 batch を複数回処理しても `AppendKnowledgeEvent` の dedupe registry が `RowsAffected = 0` で no-op を返すため idempotent。

`occurred_at` は **triggering signal の `OccurredAt`** をそのまま使用。`time.Now()` を業務時刻に持ち込まないため reproject-safety が保たれる (`immutable-design-guard` 適用)。

projector の既存 `surface_plan_recomputed.go` ([[000873]]) は producer から流れてきた event を受けて `PatchKnowledgeLoopEntrySurfacePlan` で planner-only 列のみ更新する。lifecycle / why / freshness / artifacts は触らない契約は ADR-000873 で SQL レベルに固定済み。

### 2. v1 scope は AugurConversationLinked のみ

producer の signal allowlist は意図的に narrow:

- ✅ `augur.conversation_linked.v1` — payload に entry_key を持つので、対象 entry が一意に決まる
- ⏭️ `recap.topic_snapshotted.v1` — payload には `top_terms` だけで entry_key が無い。tag fan-out は別 ADR で扱う
- ⏭️ `SummaryVersionCreated` / `SummarySuperseded` / `HomeItemOpened` — projector が直接 entry を生成 / 更新するため、producer が二重発火するとプロジェクション競合のリスク

この narrow な v1 でも producer-consumer pair が end-to-end で接続され、Augur 経由のコンテキスト変化が UI plane に反映される動作は完成する。

### 3. Cadence を env 化

`knowledge-sovereign/app/main.go` に `parseDurationEnv(name, fallback)` ヘルパを追加:

| env | default | 用途 |
|---|---|---|
| `KNOWLEDGE_SOVEREIGN_PROJECTOR_TICK_INTERVAL` | 5s | knowledge_loop_projector |
| `KNOWLEDGE_SOVEREIGN_PLANNER_TICK_INTERVAL` | 60s | surface_planner_cron |
| `ALT_BACKEND_LOOP_STREAM_POLL_INTERVAL` | 5s | StreamKnowledgeLoopUpdates poll |
| `ALT_BACKEND_LOOP_STREAM_HEARTBEAT_EVERY` | 10s | StreamKnowledgeLoopUpdates heartbeat |

不正値 (parse error / 0 以下) は warn ログを出して default にフォールバック。CrashLoop ではなく silent fallback を採るのは「cadence は安全側のデグレード対象である」という運用観点。

### 4. E2E シナリオ追加

`e2e/hurl/knowledge-sovereign/22-surface-planner-cron.hurl`:

1. SummaryVersionCreated を append → projector が entry を作る
2. augur.conversation_linked.v1 を append → producer cron が picks up
3. ListKnowledgeEventsSince で `knowledge_loop.surface_plan_recomputed.v1` の出現を `retry: 30` で検知
4. GetKnowledgeLoopEntries で `surfacePlannerVersion = SURFACE_PLANNER_VERSION_V2` を assert

CI 側で `KNOWLEDGE_SOVEREIGN_PLANNER_TICK_INTERVAL=2s` を渡せば、retry 30 回 × 1.5s = 45s 以内に producer-consumer pair が観測される。

### 検討した代替案と却下理由

- **(却下) producer を alt-backend / pre-processor / recap-worker に置く**: 既存の event emitter (Recap=Rust, Augur=Go) はそれぞれ自分の領域 event しか出さない原則。SurfacePlanRecomputed は **system event** であり owner は projection を持つ knowledge-sovereign が自然。
- **(却下) producer を proto の `service` に追加して同期 RPC で発火**: replay 不能になる。append-only event log を使うことで「context が変わった」事実が永続化され、再 projection で同じ結果が得られる。
- **(却下) v1 で RecapTopicSnapshotted も処理する**: tag fan-out のためには entry の source article のタグ情報が必要で、現 schema では entries に保持されていない。schema 拡張 + 新 driver query が必要となり scope を膨らませる。AugurConversationLinked のみで minimum-viable な producer-consumer pair を立ち上げ、Recap fan-out は別 ADR で段階的に追加する。
- **(却下) checkpoint を新 table `knowledge_loop_planner_checkpoints` で持つ**: Phase 1 探索で `knowledge_projection_checkpoints` table が既存で `(projector_name, last_event_seq)` schema を提供することが判明。新 migration を作らず `projector_name = "surface_planner_v2"` の row として共存可能。migration 数を最小化。
- **(却下) cadence を `os.Getenv` 直書き**: parse error / 0 / negative の防御がないと crash の温床。`parseDurationEnv` で silent fallback + warn ログにすることで運用視認性とロバスト性を両立。

## Consequences

### Pros

- **ADR-000873 で実装した projector branch が初めて生きる**。fb.md issue 9 が指摘する「Surface Planner v2 の後方参照」状態が解消し、Augur conversation の発生時に UI plane が再配置される経路が end-to-end で接続される。
- **Reproject-safety を構造的に維持**。producer は event payload の `OccurredAt` のみを使う。replay すれば同じ SurfacePlanRecomputed が同じ順序で再生成される。
- **dedupe_key の `batch_max_seq` 戦略により producer の reentrant 起動が安全**。CrashLoop 中に複数 tick が走っても、AppendKnowledgeEvent の dedupe registry が同一 batch の重複を吸収。
- **既存 `knowledge_projection_checkpoints` table を再利用**することで migration 数を 0 件に抑えた (ADR-000873 で 00018 を 1 件追加した直後なので、続けざまの migration を避ける運用配慮)。
- **Cadence env 化で staging / CI のテスト回転が早まる**。`KNOWLEDGE_SOVEREIGN_PLANNER_TICK_INTERVAL=2s` を CI に渡すだけで Hurl の retry 待ちが大幅に短縮される。本番は default 60s で安全側を維持。
- **TDD outside-in 遵守**。cron_test.go で 7 ケース (no signals / 1 augur / coalesce / multi-user / no entry_key / non-signal-event / dedupe / no-wallclock) を RED 先行 → GREEN 実装。`alt/usecase/knowledge_loop_usecase` / `knowledge-sovereign/usecase/surface_planner_cron` 共に `-race` 付き go test green、golangci-lint 0 issues。

### Cons / Tradeoffs

- **v1 scope は AugurConversationLinked のみ**。RecapTopicSnapshotted の tag fan-out / SummaryNarrativeBackfilled 後の planner reconsideration は未対応。fb.md issue 9 を半分しか塞げていない。Recap 経路は別 ADR (Wave 1.5) で追跡する必要がある。
- **producer cron は `auger.conversation_linked.v1` の `entry_key` フィールドを payload からのみ読み取る**。Augur 側がペイロード schema を変えると producer が silent に skip し始める。Pact CDC が無いため (Augur は consumer 不在の system event)、payload schema drift は wire-protocol レベルで観測できない。runbook で "Augur conversation event の payload contract は append-only" を明文化する follow-up が必要。
- **knowledge_projection_checkpoints の `surface_planner_v2` row の lifecycle が暗黙**。migration 0 件で済んだ代償として、新規 deploy 時には `INSERT INTO knowledge_projection_checkpoints (projector_name, last_event_seq) VALUES ('surface_planner_v2', 0)` を migration で打つか、`GetProjectionCheckpoint` の `pgx.ErrNoRows → 0` フォールバックに依存する。現コードは後者を選んでいるが、operator 視認性のためには row の存在を runbook に明記する follow-up が望ましい。
- **stream cadence env を本番で短縮するとサーバ負荷上昇**。`ALT_BACKEND_LOOP_STREAM_POLL_INTERVAL=1s` のような設定は connection あたり 5x 負荷になる。env field description に "本番では 5s 以上を推奨" を残すか、`MetricsHandler` で実 cadence を Prometheus に publish する運用補強が将来必要。
- **producer の Hurl scenario はまだ run.sh からは外れている**。本 ADR ではファイル追加と unit 緑のみ確認、E2E 実走は次回の Hurl 一括実行に委ねる。
- **Wave 1 (planner cron) と Wave 4 (cadence env 化) を 1 ADR にまとめた**。後者は Wave 1 のテスト容易化のために前倒しした副次変更だが、commit 履歴上は別 commit に分けてある。

- [x] 新設 compose service に `restart: "no"` の one-shot init container を含むか？ — **N/A**
