# ADR-000710: Morning Letter ArticleDeck の遷移先を修正しチャット先を Augur に再ルーティング

## Status

Accepted

## Date

2026-04-13

## Affected Services

- alt-backend (Go) — `morning_letter_enrichment_usecase` の href 構築・関連記事ハイドレート
- alt-frontend-sv (TypeScript / Svelte 5) — `ArticleDeckCard` ラベル・href バインディング
- alt-butterfly-facade (Go) — proto 再生成のみ
- rag-orchestrator (Python) — proto 再生成のみ
- proto/ — `acolyte_href` → `chat_href` のフィールドリネーム

## Context

[[000707]] で Morning Letter に「Develop from here」セクション (ArticleDeck) を導入した際、二つのバグが残っていた。

1. **記事リンクが無効** — ArticleDeckCard のタイトルリンクは `enrichment.article_alt_href` を `/articles/<uuid>` の bare path で出力していた。一方で SvelteKit ルート `src/routes/(app)/articles/[id]/+page.svelte` は `?url=<encoded>&title=<encoded>` のクエリ文字列に依存し、`getFeedContentOnTheFlyClient(url)` で本文を取得する設計になっている。クエリが欠落すると `articleUrl === null` の分岐に入り、画面には常に "No article URL provided. Unable to load content." が表示されていた。本来クリックして記事を読むためのカードが、毎回エラー画面に着地していた。
2. **チャット先が誤り** — ArticleDeckCard のプライマリ CTA は "Chat with Acolyte →" というラベルで `/acolyte/new?article_id=…&topic=…` を指していた。しかし Alt のドメイン上、Acolyte はレポーティング (knowledge briefing) 用、チャットは Augur が担当するという責務分離が確立している (`/augur` ルートは `resolveAugurEntry({ q, context, articleId })` でクエリを受ける)。プロトコル仕様 (`acolyte_href`)・バックエンドビルダ (`buildAcolyteHref`)・UI ラベルすべてが製品意図と乖離しており、ユーザーがチャットを開きたい操作で報告作成フォームに飛ばされていた。

ホームフィードや検索結果はすでに `/articles/<id>?url=&title=` 規約で記事ページに遷移している (`ArticleSearchSection.svelte`、`/home` ページ等)。ArticleDeck だけが ADR-000707 で導入された規約から逸脱していた形となる。

## Decision

以下の三点をまとめて適用する。

1. **`article_alt_href` に `?url&title` を埋め込む** — alt-backend の `morning_letter_enrichment_usecase` で `buildArticleAltHref(articleID, articleURL, title string) string` ヘルパーを抽出し、article メタデータが揃っている場合は `/articles/<id>?url=…&title=…` を生成する。`net/url.Values.Encode()` でエスケープし、`?` や `&` を含むタイトル/URL を安全に伝搬させる。article が nil のとき (削除済み・未到達) は bare `/articles/<id>` のままとし、[[000707]] のグレースフルディグラデーション不変条件 (リンクは常に生成する) を保つ。

2. **`acolyte_href` を `chat_href` にリネームし `/augur` に向ける** — proto field を `acolyte_href` → `chat_href` (Tag 10、互換性は保たない) にリネーム。バックエンドヘルパーは `buildChatHref(articleID, title) → /augur?articleId=<uuid>&context=<title>` に置換。クエリキー (`articleId`, `context`) はフロント側 `resolveAugurEntry` の入力に正確に合わせる。命名は意図を明示する `chat_href` を採用し、将来チャット実装が Augur 以外に変わってもフィールド名のリネームを必要としないようにした。UI ラベルは "Chat with Augur" として遷移先を明示する。

3. **関連記事 (related teasers) も `?url&title` を持たせる** — search-indexer のレスポンス (`SearchHit`) には URL が含まれない (proto `services/search/v2/search.proto` を参照)。search-indexer 本体・Meilisearch index schema を改修するのは scope が広すぎるため、alt-backend 内に閉じる方針を採用。検索結果から得た hit ID 群を `articleBatch.FetchArticlesByIDs` で再ハイドレートし、URL/タイトルを Alt DB から引き直して teaser の href を組み立てる。DB が source of truth であり、検索インデックスが古くてもリンクは正しく出る。

## Implementation Overview

### 1. proto 変更とコード生成

`proto/alt/morning_letter/v2/morning_letter.proto` の `MorningLetterBulletEnrichment.acolyte_href` を `chat_href` に変更。`cd proto && buf generate` で 4 サービス (alt-backend / rag-orchestrator / alt-butterfly-facade / alt-frontend-sv) の生成コードを一括更新。protobuf wire 上のフィールド番号は Tag 10 のまま、文字列名のみリネームしているので互換は無いが、外部 API 消費者は存在せずプロジェクト内完結で問題なし。

### 2. alt-backend の usecase 改修

`alt-backend/app/usecase/morning_usecase/morning_letter_enrichment_usecase.go` の主要変更:

- `buildArticleAltHref(articleID, articleURL, title)` ヘルパーを抽出
- `buildAcolyteHref` を削除し `buildChatHref` を新設 (`/augur?articleId=&context=`)
- `hydrateRelatedArticleMeta(ctx, articleBatch, relatedByID)` を追加。検索ヒットの hit.ID を `uuid.Parse` し、ユニーク化したうえで `FetchArticlesByIDs` を 1 回呼び、`map[string]*domain.Article` を返す
- `enrichRelatedTeasers(teasers, metaByID)` で各 teaser の `ArticleAltHref` を上書き。DB に該当が無ければ bare href のまま残す
- 組み立てループでは article の有無で分岐し、欠損時は bare href、存在時は完全形 (`?url&title`) と `ChatHref` を設定

`alt-backend/app/domain/morning_enrichment.go` で `AcolyteHref` → `ChatHref` にリネーム。`alt-backend/app/connect/v2/morning_letter/handler.go` の proto マッピングも `e.ChatHref` に追従。

### 3. テスト (TDD: RED → GREEN)

`alt-backend/app/usecase/morning_usecase/morning_letter_usecase_test.go` を更新。

- `TestGetLetterEnrichment_BuildsCardsWithAltHrefAndChatLink` — 旧 "AcolyteLink" テストを刷新。`url.QueryEscape` でエンコード済みのクエリ文字列を含むことをアサート。related teaser についても 2 度目の `articleBatch.FetchArticlesByIDs([relatedID])` がモックされ、ハイドレート後の href に `url=` と `title=` が含まれていることを検証
- `TestGetLetterEnrichment_GracefullyHandlesMissingArticle` — article 取得が空配列の場合に `ArticleAltHref` が bare path、`ChatHref` が空文字であることを確認 (グレースフルディグラデーション不変条件の維持)

`go test ./...` → 全パッケージ GREEN、レグレッションなし。

### 4. alt-frontend-sv の更新

`alt-frontend-sv/src/lib/components/morning-letter/ArticleDeckCard.svelte`:

- `enrichment.acolyteHref` → `enrichment.chatHref` のバインディング差し替え
- ラベルを "Chat with Acolyte →" → "Chat with Augur →" に変更
- `data-role="deck-acolyte-cta"` → `data-role="deck-chat-cta"` (E2E セレクタ)

`ArticleDeck.svelte` のセクション説明文と `lib/connect/morning_letter.ts` のドキュメンテーションコメントも "Acolyte" 表記から "Augur chat" 表記へ更新。`bun run check` (tsgo) で 0 errors。

## Consequences

### Pros

- **Develop from here のクリック動線が機能するようになった** — タイトルリンクから記事本文が on-the-fly で取得されるホームフィードと同じ規約に統一。ArticleDeck だけ別動作だった違和感が解消
- **チャット動線が責務分離と整合** — Acolyte = reporting、Augur = chat の境界が API 層・UI 層の両方で明示。今後の機能追加でも責務を取り違えにくい
- **関連記事リンクも DB 由来で堅牢** — 検索インデックスが古くても、Alt DB に存在する記事であれば URL/タイトルが正しく注入される
- **グレースフルディグラデーション不変条件を維持** — article メタデータ欠損時も bare `/articles/<id>` が出力され、カードは表示され続ける ([[000707]] L159-161 と整合)

### Cons / Tradeoffs

- **proto フィールドの破壊的リネーム** — `acolyte_href` を消して `chat_href` に置き換えたため、もし古いクライアント (alt-butterfly-facade / rag-orchestrator / alt-frontend-sv 以外の独立した consumer) が存在した場合は壊れる。プロジェクト内完結を前提にしたトレードオフ。後方互換シムは入れない方針 (no backwards-compat shims) に従う
- **related-article ハイドレートで articleBatch を 2 回呼ぶ** — 1 回目はソース記事、2 回目は検索ヒットのハイドレート用。N+1 ではなくバッチ 2 回で抑えているが、search-indexer 自体に URL を持たせる根本対応 (proto SearchHit 拡張 + Meilisearch index schema 変更) は将来課題として残る
- **チャット先 URL の意味的紐付け責任は frontend `resolveAugurEntry` 側にある** — `articleId` / `context` のキー名を変える場合は両方を同期して修正する必要がある (現状はテストで担保)
