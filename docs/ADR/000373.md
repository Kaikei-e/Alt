# 000373: Tag Trail の「No Tag Available」問題とFetchArticleContent不要呼び出しの修正

## ADR's STATUS

Accepted

## CONTEXT

Tag Trail 機能（`/sv/feeds/tag-trail`）で、以下の2つの問題が発生していた。

### 問題1: タグが「No Tag Available」と表示される

`FetchRandomFeed` のバックエンドSQLクエリが `feeds` テーブルから無差別にランダム選択していた。

```sql
SELECT id, title, description, link FROM feeds ORDER BY RANDOM() LIMIT 1
```

この結果、`articles` テーブルに記事が存在しない、または `article_tags` にタグが紐付いていないフィードが高確率で返される。ハンドラ側で `FetchLatestArticleByFeedID` が nil を返すと、バックエンドはタグなし（空配列）でレスポンスを返す。

フロントエンドはタグが空の場合のフォールバックとして `fetchArticleContent(feedUrl)` を呼び出すが、渡される URL はフィードのサイト URL（`feeds.link`、例: `https://example.com`）であり記事URLではないため、以下の問題が発生する:

- バックエンド側で外部サイトへHTTPリクエスト → 404応答 → `ConnectError: [not_found] article not found`
- 15秒のタイムアウト後にエラー表示

### 問題2: 不要なHTMLコンテンツの返却

フォールバックパスの `FetchArticleContent` は、記事の `articleId` を取得するためだけに呼び出されていたが、レスポンスにはフルHTMLコンテンツ（最大164KB）が含まれており、帯域の浪費が発生していた。nginx ログから `FetchArticleContent` が164,503バイトのレスポンスを返すケースが確認された。

### 根本原因の特定方法

nginx アクセスログとバックエンドコンテナログの時系列分析により、以下のリクエストフローを特定した:

```
FetchRandomFeed → (タグ空) → loadFeedTagsAsync → FetchArticleContent(サイトURL) → 404 or 巨大HTML
```

## DECISION MAKING

バックエンドとフロントエンドの両方を修正し、根本原因を排除する方針とした。

### 1. `FetchRandomFeed` SQLクエリの改善（バックエンド）

`feeds` テーブルを `articles` + `article_tags` と INNER JOIN し、タグ付き記事が存在するフィードのみを返すようにクエリを変更した。

```sql
SELECT f.id, f.title, f.description, f.link
FROM feeds f
INNER JOIN articles a ON a.feed_id = f.id
INNER JOIN article_tags at ON at.article_id = a.id
GROUP BY f.id, f.title, f.description, f.link
ORDER BY RANDOM()
LIMIT 1
```

これにより、返されるフィードには必ずタグ付き記事が存在することが保証される。

### 2. `latest_article_id` フィールドの追加（Proto + バックエンド）

`FetchRandomFeedResponse` に `latest_article_id` フィールド（field number 6）を追加した。バックエンドハンドラが `FetchLatestArticleByFeedID` で取得した記事IDをレスポンスに含めることで、フロントエンドが `StreamArticleTags` を直接呼び出せるようにした。

### 3. `FetchArticleContent` フォールバックの排除（フロントエンド）

`DesktopTagTrailScreen.svelte` と `TagTrailScreen.svelte`（モバイル版）から `fetchArticleContent` の呼び出しを完全に排除した。タグが空でも `latestArticleId` があれば `streamArticleTags` で直接ストリーミング取得する。`latestArticleId` も存在しない場合は即座に「No tags available」を表示する。

### 4. 不要なasync goroutineの削除（バックエンド）

タグなしフィードに対して非同期でコンテンツ取得・タグ生成を行っていた goroutine を削除した。改善されたSQLクエリにより、タグなしフィードが選択されるケース自体がほぼ発生しなくなったため不要となった。

## RESULTS, EFFECTS

### PROS

- **タグの確実な表示**: JOIN クエリにより、返されるフィードには必ずタグ付き記事が存在する。「No Tag Available」の表示がほぼ解消される
- **不要なHTTP通信の排除**: `FetchArticleContent` の呼び出しがTag Trailから完全に排除され、最大164KBのHTML転送と15秒のタイムアウト待ちがなくなる
- **レスポンス時間の改善**: 外部サイトへのHTTPリクエストが不要になるため、ページ表示までの時間が大幅に短縮される
- **Proto後方互換性**: `latest_article_id` はオプショナルフィールドとして追加されており、古いクライアントへの影響はない

### CONS, TRADEOFF

- **SQLクエリの複雑化**: `feeds` テーブルへの単純な `ORDER BY RANDOM()` から、3テーブルJOIN + GROUP BY を含むクエリに変更された。`idx_articles_feed_id` インデックスと `article_tags` の主キーにより実用上問題ないが、フィード数が極端に増加した場合はパフォーマンスモニタリングが必要
- **タグなしフィードの表示機会**: タグ付き記事がないフィードはTag Trailで表示されなくなる。これは意図された動作であり、タグのないフィードを表示しても探索が成立しないため問題ない
