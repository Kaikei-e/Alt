# ADR-000713: alt-frontend-sv の Docker ビルドを Bun 公式パターンと BuildKit cache mount で高速化する

## Status

Accepted

## Date

2026-04-14

## Affected Services

- alt-frontend-sv (SvelteKit / Bun / TypeScript)

## Context

`alt-frontend-sv` のコンテナビルド時間が開発体験を悪化させていた。改修前の Dockerfile は multi-stage で `deps` と `prod-deps` を独立ステージに分けており、それぞれが `COPY package.json bun.lock` + `bun install` を走らせる構造だった。以下の要因が重なっていた。

- **Bun install の DNS/IPv6 タイムアウト**: 依存が 1.7GB 規模（three.js, @threlte, chart.js, playwright, msw 等を含む）で、ステージ分割により `--production` 側の 2 回目 install で接続が張り直しになり、[Bun issue #28278](https://github.com/oven-sh/bun/issues/28278) の通り 60 秒級のタイムアウトを踏みやすい状態だった。
- **Vite / SvelteKit のビルドキャッシュが毎回ゼロから**: `builder` ステージでは `.svelte-kit/` や `node_modules/.vite/` の事前バンドルキャッシュを BuildKit の `--mount=type=cache` で保持しておらず、ソース変更のたびに Vite の依存 pre-bundle（oxc + rolldown）を完全に再実行していた。
- **`.dockerignore` の漏れ**: `docs/`, `stats.html` (2MB), `coverage/`, `*.tsbuildinfo` 等がビルドコンテキストに含まれており、転送量と `COPY . .` のレイヤーハッシュ安定性に影響していた。
- **Bun バージョンが古い**: `oven/bun:1.3.10-alpine` 固定のまま。1.3.11–1.3.12 で IPv6 タイムアウトとライフサイクルスクリプトのハング修正が入っている。

改修後のクリーンビルドは約 **1m12s**（Vite build 本体は 15.05s）、`up --build` で起動確認まで **1m31s** に短縮できた。

## Decision

以下の 2 ファイルを改修し、Bun 公式 Docker ガイドと BuildKit のキャッシュ最適化ベストプラクティスに準拠する。

### 1. `alt-frontend-sv/Dockerfile` の書き直し

- **install を 1 ステージに統合**: Bun 公式推奨の `/temp/dev` + `/temp/prod` パターンに切り替え、1 つの `install` ステージ内で dev と prod を順次 install する。DNS/ネットワーク接続を温存し、IPv6 タイムアウトの発生確率を下げる。`--mount=type=cache,id=bun-alt-frontend-sv` は両 install で共有。
- **Vite / SvelteKit キャッシュ mount を追加**: `builder` ステージの `bun run build` に以下を付与。
  - `--mount=type=cache,id=vite-alt-frontend-sv,target=/app/node_modules/.vite`
  - `--mount=type=cache,id=sveltekit-alt-frontend-sv,target=/app/.svelte-kit`
- **`COPY --link` 採用**: 全ステージの `COPY` を `COPY --link` 化し、上流レイヤが変わっても物理コピーを再利用できるようにする。
- **Bun イメージを `oven/bun:1.3.12-alpine`** に更新。`syntax=docker/dockerfile:1.7` も明示。
- **runtime は既存構成を維持**: `build/`, 本番 `node_modules/`, `package.json` のみをコピーする最小構成と非 root ユーザー (`sveltekit:1001`) 起動を踏襲。

### 2. `alt-frontend-sv/.dockerignore` の拡張

ビルドコンテキストに入る必要のないものを除外追加: `docs/`, `coverage/`, `.turbo/`, `.bun/`, `.vite/`, `.vitest-cache/`, `tsconfig.tsbuildinfo`, `*.tsbuildinfo`。

### 3. docker-compose 側は変更しない

`compose/core.yaml` の `alt-frontend-sv.build` への `cache_from` ヒント付与は単一マシン開発では効果が薄いと判断し対象外とした。BuildKit は既定で有効。

## Consequences

### Pros

- **初回クリーンビルドの短縮**: install ステージ統合により DNS タイムアウトの発生確率が下がり、クリーンビルド（cache prune 後）で約 1m12s を達成。
- **インクリメンタルビルドの高速化**: Vite / SvelteKit の cache mount により、ソース変更時でも Vite の依存事前バンドルフェーズが skip され、`bun run build` の主要コストを削減できる。
- **`COPY --link` + `.dockerignore` 拡張**: 依存変更がないリビルドでは多くのレイヤーが再利用でき、ビルドコンテキスト転送量も削減。
- **Bun 1.3.12 への更新**: IPv6 タイムアウトとライフサイクルスクリプトハングの既知修正を取り込める。
- **Bun 公式パターン準拠**: 将来 Bun が Docker ガイドを更新した際の差分追従が容易。

### Cons / Tradeoffs

- **`--mount=type=cache` のキャッシュ汚染リスク**: 稀に Vite の `node_modules/.vite` キャッシュが古い状態で残り続けて奇妙なエラーを引き起こすことがある。問題時は `docker builder prune` で該当 id のキャッシュを明示的に削除する運用を周知する必要がある。
- **BuildKit 前提**: 現代の Docker では既定で有効だが、旧環境や特殊な CI で BuildKit を無効化している場合は `COPY --link` と `--mount=type=cache` が機能しない。
- **Bun バージョンを `1.3.12-alpine` に固定**: floating tag `oven/bun:1-alpine` を選ばなかったため、次のセキュリティパッチ取り込みは手動更新になる。再現性を優先した判断。
- **Vite プラグイン側のボトルネックは未解決**: ビルドログに `vite-plugin-sveltekit-guard (45%)` の PLUGIN_TIMINGS 警告が残っている。これは上流プラグインの課題で、今回のスコープ外。

### 検証結果

- `docker compose -f compose/compose.yaml -p alt up --build -d alt-frontend-sv` が約 1m31s で完了し、`alt-alt-frontend-sv-1` が `Up (healthy)` に到達。
- `curl -sf http://localhost:4173/` が HTTP 200 を返却。
- Vite build 本体は 15.05s、`bun run build` ステージ全体で 25.8s。
