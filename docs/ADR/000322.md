# ADR-000322: search-indexer WaitForTask ポーリング間隔バグの修正

## ADR's STATUS

完了

## CONTEXT

search-indexer のコードレビューにより、`meilisearch_driver.go` の `WaitForTask` 呼び出しでポーリング間隔の指定が誤っていることが判明した。

```go
// 修正前: 15,000 ナノ秒 = 0.015 ミリ秒でポーリング
_, err = d.index.WaitForTask(task.TaskUID, 15*1000)
```

meilisearch-go v0.36.1 の `WaitForTask` シグネチャ:

```go
func (i *index) WaitForTask(taskUID int64, interval time.Duration) (*Task, error)
```

`time.Duration` はナノ秒ベースの型であり、`15*1000` は untyped integer constant として `time.Duration(15000)` = **15 マイクロ秒**に解釈される。意図は 15 秒間隔のポーリングだが、実際には約 100 万倍の頻度でポーリングしていた。タスク完了まで高速ポーリングするため機能的には動作するが、CPU リソースの無駄遣いとなっていた。

同一パターンがファイル内 5 箇所に存在:

| 行 | メソッド |
|----|---------|
| 36 | `IndexDocuments` |
| 61 | `DeleteDocuments` |
| 158 | `EnsureIndex`（インデックス作成） |
| 169 | `EnsureIndex`（ダミードキュメント削除） |
| 290 | `RegisterSynonyms` |

## DECISION MAKING

5 箇所すべてで `15*1000` → `15 * time.Second` に修正。`"time"` パッケージのインポートを追加。

```go
// 修正後: 15 秒間隔でポーリング
_, err = d.index.WaitForTask(task.TaskUID, 15 * time.Second)
```

## RESULTS, EFFECTS

### PROS

- **CPU 負荷削減**: WaitForTask のポーリングが約 100 万分の 1 の頻度に正常化。インデクシング・シノニム登録・インデックス初期化すべてに適用
- **コードの意図との一致**: コメントに `// 15 seconds timeout` と書かれていた意図が、実際の動作と一致するようになった

### CONS, TRADEOFF

- **ポーリング間隔変更による影響**: 修正前はタスク完了を即座に検出していたが、修正後は最大 15 秒の遅延が生じうる。ただし Meilisearch のインデクシングタスクは通常数百ミリ秒で完了するため、実質的な影響は限定的
