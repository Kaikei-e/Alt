# ADR-000705: Morning Letter を event-sourced editorial に再定義しフルスタックで再稼働する

## Status

Accepted

## Date

2026-04-13

## Affected Services

- alt-backend (Go)
- recap-worker (Rust)
- news-creator (Python)
- alt-frontend-sv (SvelteKit / Svelte 5)
- proto (`proto/alt/morning_letter/v2`)

## Context

Morning Letter 画面 (`/recap/morning-letter`) が長期にわたって
**"Morning Letter / No Edition Available / The morning briefing hasn't been
generated yet. Check back soon."** を表示し続けていた。
コードを棚卸しすると、UI・BFF・recap-worker パイプライン・news-creator LLM・
`morning_letters` / `morning_letter_sources` テーブルまで一式は出揃っていて、
詰まっていたのは**生成トリガ**だけだった。
`morning-letter-v2-epic.md` Phase 5 (daemon の恒久有効化 + フィーチャーフラグ +
監視) が未デプロイで、`morning_letters` に行が一度も入っていなかった結果、
`GetLatestLetter` が `CodeNotFound` を返し、フロント側がそれを `null` に
変換して Empty コンポーネントを描画していた。

加えて、既存の v2 スペック (ADR-000633/634/635/680) のままデーモンを
単に有効化しても、次の 3 つの構造的な弱さが残っていた。

1. **意味の被り**: 「昨夜〜今朝にかけて何が起きたか」を示す読み口は
   `today_digest_view` と Knowledge Home ストリームが既に担っている。
   recap + overnight を文書化しただけの Morning Letter は同じ情報を
   弱い provenance で二度配っているだけになりがち。
2. **LLM 依存の脆さ**: v2 は bullet まで LLM の structured output に
   依存していた。Gemma4 の JSON パース不安定 (ADR-000632) が残っているため
   degraded mode が常態化する。
3. **縦の糸がない**: 各日の Letter が独立しており、ユーザーの知識イベントの
   長期的な「流れ」が見えない。append-first な `knowledge_events` という
   Knowledge Home の北極星 (`knowledge_home_immutable_model`) を活かせていない。

そこでユーザー確認 (AskUserQuestion) の上、**(A) "event-sourced editorial"
への再定義** / **recap-worker にプレーンを残す** / **RegenerateLatest RPC を
追加** の 3 択を選び、実装を進めた。

## Decision

**Morning Letter を「直近 24h の `knowledge_events` を 1 つの through-line に
編み、why-reason と前日 Letter へのリンクを一次情報として持つ個人向け
editorial」として再定義する。** 生成プレーンは recap-worker (Rust) に
残し、alt-backend から明示的に叩ける `RegenerateLatest` RPC を追加する。

具体的には、現行 v2 の ADR-000634 (Read API) / ADR-000635 (UI) /
ADR-000680 (Alt-Paper 化) の提供面はすべて維持したまま、
ADR-000633 の**生成戦略部分のみ**を以下で差し替える。

- **deterministic projector**: through-line (一行エディトリアル) と
  per-bullet `why_reasons` は projector が recap + overnight の集計から
  決定論的に組み立てる。LLM 失敗時もこの情報は消えない。
- **LLM は散文のみ**: news-creator は `lead` と各 section の `narrative`
  (1〜3 文のブリッジ段落) だけを埋める。bullet は触らない。
- **longitudinal 連結**: recap-worker の `MorningPipeline` が
  `get_previous_morning_letter(edition_timezone, before)` で直前 Letter を
  引き、`previous_letter_ref` として persist する。UI は
  "Since yesterday" band で前日の through-line にリンクする。
- **reproject-safe**: through-line ビルダーは純関数で、同一入力から
  同一出力 (determinism テストで担保)。`morning_letters` の
  `generation_revision` を再計算しても意味は変わらない。
- **ユーザー起点の復旧経路**: `MorningLetterReadService.RegenerateLatest`
  を追加し、1h/user の cooldown を usecase 側で課す。クールダウン中は
  直近キャッシュを返し、UI に "Using the cached edition — try again in N
  minutes" を出す。
- **デーモン再有効化**: `MORNING_DAEMON_ENABLED=true` のとき
  `spawn_morning_update_daemon` を起動。デフォルト false でロールアウトを
  制御する。

## 実装の概要

### Proto (`proto/alt/morning_letter/v2/morning_letter.proto`)

後方互換な追加のみ。

- `MorningLetterReadService.RegenerateLatest` RPC 追加 +
  `RegenerateLatestRequest/Response` (`letter`, `regenerated`,
  `retry_after_seconds`)。
- `MorningLetterBody` に `through_line`, `previous_letter_ref`。
- `MorningLetterSection` に `narrative`, `repeated WhyReason why_reasons`。
- `WhyReason` の shape は `{code, ref_id, tag}` で、
  `alt-backend/app/domain/knowledge_home_item.go` の `WhyReason` と一致
  させ、フロントの `WhySurfacedBadge` をそのまま再利用できるようにした。
- `buf generate` で alt-backend / rag-orchestrator / alt-butterfly-facade /
  alt-frontend-sv の生成物を更新。

### alt-backend (Go)

Clean Architecture (Handler → Usecase → Port → Gateway → Driver) を
崩さず延長。

- `domain/morning.go`: `PreviousLetterRef` 追加、`MorningLetterBody` に
  `ThroughLine`/`PreviousLetterRef`、`MorningLetterSection` に
  `Narrative`/`WhyReasons`。既存の `domain.WhyReason` を再利用。
- `port/morning_letter_port/morning.go`: `MorningLetterRepository` と
  `MorningLetterUsecase` に `RegenerateLatest` を追加。
- `gateway/morning_gateway/morning_letter_gateway.go`: REST DTO を新
  フィールド対応に拡張、`RegenerateLatest` を `POST
  /v1/morning/letters/regenerate` へブリッジ。
- `usecase/morning_usecase/morning_letter_usecase.go`: 1 時間の
  per-user cooldown を `sync.Mutex + map[userID]time.Time` で実装。
  エラー時は cooldown をロールバックし、ユーザーが失敗で冷やされないよう
  にした。
- `connect/v2/morning_letter/handler.go`: `RegenerateLatest` handler と
  `domainToProto` を新フィールド対応に拡張。`UserContext.UserID` を
  cooldown key に使う。
- `go.uber.org/mock` でモック再生成、ユースケース向けに 3 ケース
  (フレッシュ生成 / rate-limited / 失敗時の cooldown 解除) の
  テーブルテストを追加し全体 `go test ./...` グリーン。

### recap-worker (Rust)

生成プレーンを保持しつつ editorial 化する最小差分。

- `store/models.rs`: `MorningLetterContent` に `through_line`,
  `previous_letter_ref`、`MorningLetterSection` に `narrative`,
  `why_reasons`、`PreviousLetterRef` / `WhyReason` 追加。
- `store/dao/morning.rs` + trait / compat / mock / unified impl に
  `get_previous_morning_letter(edition_timezone, before)` を追加。
  SQL は `edition_timezone = $1 AND target_date < $2 ORDER BY
  target_date DESC LIMIT 1`。
- `pipeline/morning.rs`: `build_through_line(overnight, recap,
  is_degraded, window_days)` を追加。純関数・deterministic。
  `persist_morning_letter` が `result_jsonb` に
  `through_line`/`previous_letter_ref`/per-section `narrative` +
  `why_reasons` を書き込むよう更新。why code は既存の
  Knowledge Home 定義 (`in_weekly_recap`, `pulse_need_to_know`,
  `new_unread`) を流用。
- `api.rs` + `api/fetch.rs`: `POST /v1/morning/letters/regenerate` を
  追加し、`Scheduler.run_morning_update(JobContext)` を同期実行して
  persisted な `morning_letters` 行を返す。
- `main.rs`: `MORNING_DAEMON_ENABLED` env フラグで
  `spawn_morning_update_daemon` を条件付き起動。
- `cargo test --lib` 全 306 件グリーン (5 件の through-line
  determinism テストを追加)。

### news-creator (Python)

- `domain/models.py`: `MorningLetterSection` に `narrative: Optional[str]`。
- `usecase/morning_letter_usecase.py`: LLM options を
  `config.get_llm_options()` ベースに統一
  (`feedback_unify_ollama_options.md` 遵守)、プロンプトに
  `narrative` 欄を追加。LLM 失敗時の extractive fallback はそのまま
  (`narrative` 空 + bullet は残る)。
- `uv run pytest` 全 496 件グリーン。

### alt-frontend-sv (Svelte 5 / SvelteKit)

Alt-Paper デザイン言語 (ADR-000680) のトークンのみ使用。
マストヘッド等のパターン丸パクリは避け、Morning Letter 固有の
through-line hero と since-yesterday band を新規に表現
(`feedback_design_philosophy_not_copy.md`, `feedback_alt_paper_shared_language.md`)。

- `lib/connect/morning_letter.ts` + `lib/connect/index.ts`:
  `regenerateLatestLetter(transport, editionTimezone?)` を export。
- `lib/hooks/useMorningLetter.svelte.ts`: `regenerate()`,
  `regenerating`, `regenerateCooldownMsg` を追加。
- `lib/components/morning-letter/ThroughLineHero.svelte` (新規):
  3px double ルール + Playfair Display で through-line を主役化。
  Degraded バッジもここに集約。
- `lib/components/morning-letter/SinceYesterdayBand.svelte` (新規):
  Alt-Paper の dark paper ink を背景に、前日 Letter の through-line に
  ボタンで遷移。
- `MorningLetterDocumentCore.svelte`: hero と band をレイアウト、
  `onPreviousLetterSelected` を `+page.svelte` 経由で `goto` に接続。
- `MorningLetterSectionCore.svelte`: `narrative` 段落、bullet と
  `why_reasons` の 1:1 対応を `WhySurfacedBadge` (Knowledge Home と
  共用) で描画。
- `MorningLetterEmpty.svelte`: **"check back soon" を廃止**。
  "Read the {lastPublished} edition →" リンクと "Generate today's
  letter" CTA、クールダウンメッセージを表示。
- `+page.svelte`: `handleRegenerate` + `handlePreviousLetterSelected`
  を追加し、`?date=...` で replaceState 遷移。
- `bun test`: 893 件グリーン、`bun run check` 0 エラー、
  `bun run build` 成功。

### 不変条件との整合

- **append-first**: `morning_letters` は derivable な結果テーブル。
  source of truth は `knowledge_events` + 既存の recap / overnight
  パイプラインで、projector 側は副作用更新を行わない。
- **reproject-safe**: through-line / why_reasons は純関数。LLM 部分
  (`lead`, `narrative`) は失敗耐性があり、失敗しても bullet は残る。
- **versioned artifacts**: `schema_version` / `generation_revision` は
  既存の UPSERT ロジックをそのまま流用。regenerate しても
  `generation_revision` が増えるだけで過去 Letter は消えない。

## Consequences

### Pros

- 空状態が**仕様**で終わらず、ユーザーが "Generate today's letter" を
  押せば即座に復旧する運用ループができた。
- Knowledge Home の `WhyReason` 型と `WhySurfacedBadge` を共通言語として
  Morning Letter 側でも再利用し、why-attribution の UX が全サーフェスで
  一貫した。
- through-line / bullet が LLM 非依存に決定論化したことで
  `ADR-000632` の JSON パース不安定性の影響が限定され、degraded mode
  でも Letter に「編集的な意味」が残る。
- `previous_letter_ref` により「昨日のあなた → 今日のあなた」の
  継続性が一次情報として Letter に格納され、再処理でも復元可能になった。
- デーモンを env フラグで段階投入できるようにしたため、
  ロールアウトが可逆。

### Cons / Tradeoffs

- `RegenerateLatest` の 1h/user クールダウンはプロセスローカル
  (`map[userID]time.Time`) で、alt-backend を複数レプリカにしたとき
  per-user の厳密保証は失われる。本気の共有 rate limit は recap-worker
  側 or Redis ベースで追って導入する余地を残した。
- `POST /v1/morning/letters/regenerate` は同期実行で MorningPipeline を
  走らせるため、news-creator の LLM が遅い環境では request-path が
  数分ブロックされる。将来的には 202 + polling 化を検討する。
- 今回の「event-sourced」化は pragmatic 版であり、厳密には
  `knowledge_events` を直接読む projector にはまだ届いていない
  (recap + overnight group を event 代替として扱っている)。
  完全な event-sourced 化は `KnowledgeSovereignService.ListKnowledgeEvents`
  を使う次フェーズの課題として残った。
- supersede / `ReasonMerged` producer (P1-5 remediation) はスコープ外で
  未着手。
