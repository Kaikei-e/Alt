# ADR-000786: Kratos staging config を prod から fork する

## Status

Accepted

## Date

2026-04-19

## Affected Services

- auth-hub (Go) — Hurl E2E suite の Kratos dependency に影響。プロダクションコード無変更
- `e2e/fixtures/auth-hub/kratos/kratos.yml` — 新規
- `e2e/fixtures/auth-hub/kratos/identity.schema.json` — prod のコピー

## Context

[[000785]] で追加した auth-hub Hurl E2E suite は `oryd/kratos:v1.3.0` を staging 環境で起動し、そこに向けて auth-hub の `/validate` / `/session` / `/csrf` を叩く。Kratos 本体には config ファイル (`kratos.yml`) が必要で、本番は `/home/koko/Documents/dev/Alt/kratos/kratos.yml` を `/etc/config/kratos/` に bind mount して使っている。

この prod config をそのまま staging に流用しようとすると以下のハードコードが壊れる:

```yaml
# /home/koko/Documents/dev/Alt/kratos/kratos.yml
serve:
  public:
    base_url: https://curionoah.com/ory         # 行 7
  admin:
    base_url: http://kratos-admin:4434/          # 行 34

selfservice:
  default_browser_return_url: https://curionoah.com/feeds  # 行 37
  allowed_return_urls:
    - https://curionoah.com/                                # 行 39
    - https://curionoah.com/home
    # ... (計 14 件の curionoah.com + localhost URL)

session:
  cookie:
    domain: .curionoah.com                       # 行 157
```

staging network (`alt-staging`, `internal: true`) では以下の問題が起きる:

1. `base_url: https://curionoah.com/ory` と `cookie.domain: .curionoah.com` は container 内からは到達不可能・cookie 属性ミスマッチで login flow の redirect と cookie set が両方壊れる
2. `allowed_return_urls` に staging の `http://auth-hub:8888/` が含まれないため self-service login の after-success redirect が 400 になる
3. `verification`, `recovery` が enabled のままだと Kratos が SMTP 要求を開始し、`courier.smtp.connection_uri` のプレースホルダで起動時警告 (+ 一部 flow が失敗)
4. `totp`, `lookup_secret`, `link`, `code` が enabled のまま残ると self-service の UI JSON が肥大し、browser flow で `csrf_token` node を走査する [[000787]] の [Captures] 表現が複雑になる
5. `bcrypt.cost: 12` は prod の正解だが、CI で password seed → login を毎回回すと ~1s × 回数だけ余計な時間がかかる
6. `session.lifespan: 720h` (30 日) は prod の運用的には妥当だが、Hurl E2E は 1 回の CI 実行が ~90 秒なので過剰

## Decision

**`e2e/fixtures/auth-hub/kratos/kratos.yml` として新規に minimal な staging config を書き起こす。** prod `kratos/kratos.yml` は一切修正しない。`identity.schema.json` のみコピーし、将来 prod schema と drift しないよう `e2e/fixtures/auth-hub/kratos/identity.schema.json` に同内容を置く。

### Staging config の差分方針

| 項目 | prod | staging |
|---|---|---|
| `serve.public.base_url` | `https://curionoah.com/ory` | `http://kratos:4433/` |
| `serve.admin.base_url` | `http://kratos-admin:4434/` | `http://kratos:4434/` (同一 container) |
| `selfservice.default_browser_return_url` | `https://curionoah.com/feeds` | `http://auth-hub:8888/health` |
| `allowed_return_urls` | curionoah.com + localhost:4173 系 14 件 | `http://auth-hub:8888/`, `http://kratos:4433/` の 2 件のみ |
| `selfservice.methods.password.config.min_password_length` | 6 | 8 (seed password 長と揃える) |
| `selfservice.methods.totp` / `lookup_secret` / `link` / `code` | enabled | **disabled** (E2E で使わない) |
| `selfservice.flows.recovery.enabled` | true | **false** (SMTP 不要化) |
| `selfservice.flows.verification.enabled` | false | false (変更なし) |
| `log.leak_sensitive_values` | false | false (同じ。本番方針維持) |
| `hashers.bcrypt.cost` | 12 | **4** (CI 速度) |
| `session.lifespan` | 720h | **1h** (1 run で十分) |
| `session.cookie.domain` | `.curionoah.com` | **省略** (container-local, host-relative binding) |
| `session.cookie.same_site` | Lax | Lax (同じ) |
| `session.cookie.path` | / | / (同じ) |
| `secrets.cookie` / `secrets.cipher` | `${KRATOS_COOKIE_SECRET}` / `${KRATOS_CIPHER_SECRET}` | 同じ `${VAR}` 表記 (後述) |
| `courier.smtp` | 本番用プレースホルダ | **セクション自体を削除** (verification/recovery disabled で不要) |
| identity schema の URL | `file:///etc/config/kratos/identity.schema.json` | 同じ (mount 先が同一) |

### Secret 置換は prod と同じ経路で

Kratos は YAML 内の `${VAR}` を native には expansion しない。prod は `kratos/entrypoint.sh` (shell script) が `sed` で置換する:

```sh
sed "s|\${KRATOS_COOKIE_SECRET}|${KRATOS_COOKIE_SECRET}|g; s|\${KRATOS_CIPHER_SECRET}|${KRATOS_CIPHER_SECRET}|g; s|\${DSN}|${DSN}|g" \
    /etc/config/kratos/kratos.yml > /tmp/kratos.yml
```

staging でもこの entrypoint.sh を **そのまま bind mount** し、container 側で同じ置換経路を通す。staging 固有の shell script は書かない。secrets は Docker compose の `secrets:` ブロックで `/run/secrets/kratos_cookie_secret` / `/run/secrets/kratos_cipher_secret` (prod と同じ path) にマウントし、entrypoint が `cat` して env にセットする構造。

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| prod `kratos.yml` の curionoah.com URL を `${PUBLIC_BASE_URL}` 等に env 化して staging と共用 | prod config を E2E のために汚す。`allowed_return_urls` は list なので env で切り替えるのは特に厄介 (Kratos config loader は list の `${VAR}` expansion をサポートしていない) |
| prod `kratos.yml` を無変更でマウントし、Kratos の環境変数 override (`SELFSERVICE_*`, `SERVE_*`) で URL を上書き | Kratos の env override はドット記法で深いキーを狙えるが、list の上書きや部分更新ができない。`allowed_return_urls` や flow ui_url を全て env で書き直すと compose の environment ブロックが 20 行増える |
| identity schema を symlink で prod と共有 | symlink は bind mount 越しに保たれないケースがある (docker の bind mount は symlink の target を解決する挙動が環境依存)。コピーの方が確実 |
| `--dev` を使わず HTTPS 証明書を staging 内部で発行する | self-signed cert の配布 + `ROOT_CAS` env 設定で CI が重くなる。staging は `internal: true` で外部到達なし、`--dev` で HTTPS 要求を緩めるのが最小コスト |
| `session.lifespan` を `0s` にして永続 session を避ける | Kratos は 0 を invalid と reject する。短い値 (1h) で十分 |
| `hashers.bcrypt.cost: 4` ではなく `argon2` に切り替え | Kratos の `hashers.algorithm: argon2` は別途チューニングが要る。bcrypt.cost を下げるほうが差分最小 |

### 範囲外

- prod `kratos/kratos.yml` の改修 — 本 ADR は staging fork の追加のみ
- identity schema の sync 自動化 — 将来 schema を差し替える際は `diff kratos/identity.schema.json e2e/fixtures/auth-hub/kratos/identity.schema.json` を README で案内する程度
- Kratos v1.4+ 移行 — 現状 v1.3.0 固定

## Consequences

### Pros

- **prod config を E2E の都合で汚さない**。prod の `kratos.yml` は運用実績のある設定で、E2E の要請で env 化や list 分岐を入れると本番が壊れやすい
- **staging の設定意図が 1 ファイルで把握できる**。`diff` や `meld` で本番との差分を追えるので、レビュー時に「staging は何を無効化しているのか」が明確
- **`kratos/entrypoint.sh` を本番と staging で共有できる**。sed 置換経路が同一なので、prod で動いている挙動がそのまま staging でも動く (drift 原因が減る)
- **staging の bcrypt cost と session lifespan を E2E 専用にチューニングできる**。CI 1 回の実行時間が ~1–2 秒短くなる (bcrypt hash 計算を 12 → 4 コスト)

### Cons / Tradeoffs

- **prod と staging の 2 ファイルを独立に保守する**。prod の identity schema や enabled method を変更したとき、staging 側への追従漏れが起こりうる。ADR 内の表で差分を列挙し、README でも差分の意図を説明するが、自動 diff は未整備
- **Kratos v1.3.0 の config schema 変更に 2 箇所追従が必要**。v1.4+ に上げる際に 2 ファイルとも更新
- **staging config が `--dev` 前提**になるため、config を `--dev` なしで回すと self-service flow が HTTPS 必須になって再度壊れる。`compose.staging.yaml` の `command: kratos serve all --dev` と強く結合する
- **identity schema の drift を検知する仕組みがない**。将来 schema にフィールドが追加されて staging 側に反映漏れが起きると、Hurl 側の seed payload が "additionalProperties=false" で弾かれる可能性がある。検知は CI failure 時に出る schema validation エラーに依存
