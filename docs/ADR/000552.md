# ADR-000552: Visual Preview の要約不一致 race condition を修正し user_id スコープを追加

## Status

Accepted

## Date

2026-03-23

## Affected Services

- alt-frontend-sv (TypeScript/SvelteKit)
- alt-backend (Go)

## Context

Desktop UI の Visual Preview モーダルで、記事タイトルとまったく無関係な AI Summary が表示される不具合が報告された。具体的には "Astro 5.0 in 2026" の記事を開いた際に、ソニー "Project Chimera" に関する要約が表示されていた。Desktop UI `/feeds` ページでも再現が確認された。

調査の結果、以下の 2 つの原因が特定された。

### 原因 1: フロントエンドの stale-response race condition

`FeedDetailModal.svelte` の `handleSummarize` 関数のストリーミングコールバック（`onChunk`, `onComplete`, `onError`）に stale-response guard が欠如していた。対照的に、同コンポーネントの `handleFetchFullArticle` には `if (feed.normalizedUrl !== targetFeedUrl) return;` による guard が存在した。

この不整合により、ユーザーが矢印キーで記事間をナビゲートした際に以下の race condition が発生した:

1. 記事 A の StreamSummarize を開始
2. バックエンドがキャッシュ済みサマリーを即座に返す（`isCached: true`）
3. ユーザーが記事 B にナビゲーション → `$effect` が `summary = null` にリセット
4. しかしバッファ済みレスポンスの microtask が残存
5. 旧ストリームの `onChunk` が発火 → 記事 B の画面に記事 A のサマリーが上書きされる

同じ脆弱性が `SwipeFeedCard.svelte` と `VisualPreviewCard.svelte` にも存在した。

### 原因 2: バックエンドの user_id フィルター欠如

`FetchArticleSummaryByArticleID` と `FetchArticleByURL` のクエリに `user_id` フィルターがなかった。`articles` テーブルの UNIQUE 制約は `(url, user_id)` であり、`article_summaries` テーブルの UNIQUE 制約は `(article_id, user_id)` であるにもかかわらず、読み取りクエリは `user_id` でスコープされていなかった。

```sql
-- 修正前: 非決定的な結果を返す
SELECT summary_japanese FROM article_summaries WHERE article_id = $1 LIMIT 1

-- 修正前: 他ユーザーの記事を返す可能性
SELECT id, title, content, url FROM articles WHERE url = $1 AND deleted_at IS NULL
```

## Decision

### フロントエンド: 全要約ストリーミングコールバックに stale-response guard を追加する

`handleFetchFullArticle` と同じパターンで、`handleSummarize` 呼び出し時に `const targetFeedUrl = feed.normalizedUrl` をキャプチャし、全コールバックで `if (feed.normalizedUrl !== targetFeedUrl) return;` を検査する。

この guard パターンは Svelte 5 ベストプラクティスの `$effect` + async における `cancelled` フラグパターンに相当し、非同期処理と reactive state 更新の race condition を防ぐ。

### バックエンド: 読み取りクエリに user_id スコープを追加する

`GetUserFromContext` でユーザーコンテキストを取得し、利用可能な場合は `user_id` でフィルターする。内部 API コール等でユーザーコンテキストが不在の場合はフォールバックとして従来のクエリを使用する。

## Consequences

### Pros

- 記事ナビゲーション時の要約不一致が解消される
- マルチユーザー環境での要約・記事の意図しないクロスリークが防止される
- `handleFetchFullArticle` と `handleSummarize` の guard パターンが統一される
- 読み取りクエリが `(url, user_id)` / `(article_id, user_id)` の UNIQUE 制約と一貫する

### Cons / Tradeoffs

- `FetchArticleSummaryByArticleID` と `FetchArticleByURL` で `GetUserFromContext` を呼ぶため、ユーザーコンテキストの有無による分岐が driver 層に入る
- フォールバック（user_id なし）パスが残るため、内部 API 経由の呼び出しでは従来と同じ非決定的な挙動が残る
