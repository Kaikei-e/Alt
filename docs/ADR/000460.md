# ADR-000460: Knowledge Home read model を active projection version 前提の複合主キーで管理する

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-backend (Go)

## Context

Knowledge Home には `projection_version` 列と `knowledge_projection_versions` テーブルが存在していたが、`knowledge_home_items` の主キーは `(user_id, item_key)` のままで、read path も active version を条件にしていなかった。このため、version ごとの projection を作る設計意図と、実際の read model の一意性・取得方法が一致していなかった。

この不整合があると、reproject で `to_version` を指定しても次の問題が起きる。

- target version 用の row を既存 row と分離して保持できない
- `CompareProjections` や version 切替が実データに基づく比較にならない
- dismiss や supersede state が後続 upsert と競合し、どの version の状態かを明示できない

特に Knowledge Home の dismiss 回復では、version を意識した replay と active version に基づく read path の両方が必要だった。

## Decision

Knowledge Home read model は `projection_version` を一級の識別子として扱い、`(user_id, item_key, projection_version)` を主キーとする。

あわせて以下を採用する。

- `knowledge_home_items` の主キーを `(user_id, item_key, projection_version)` に変更する migration を導入する
- `UpsertKnowledgeHomeItem`、`DismissKnowledgeHomeItem`、`ClearSupersedeState` は projection version を明示的に受け取り、その version の row だけを更新する
- `GetKnowledgeHomeItems`、件数集計、Today Digest 関連 query は active projection version を条件に含め、UI には現在有効な version だけを返す
- visible item 向け index も `projection_version` を含む形に再構成し、read path の性能を維持する

この決定により、read model は versioned projection として整合し、reproject で新しい version を作って比較し、切り替える流れを正しく支えられる。

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|--------|------|-----------|
| `(user_id, item_key)` 主キーを維持し、version は列だけ持つ | 移行は軽い | 複数 version を共存できず、reproject と compare の意味が崩れる |
| version ごとに別テーブルを作る | 分離は明確 | query と運用が複雑になり、共通 repository を保ちにくい |
| active version 切替をせず常に上書きする | 実装は単純 | safe rollout、compare、rollback の余地を失う |

## Consequences

### Pros

- projection version ごとの row を同時に保持でき、reproject と compare が実データに基づくようになる
- read path が active version を明示的に参照するため、UI に返す状態が安定する
- dismiss や supersede state を version 単位で管理でき、後続 upsert との競合を整理しやすい
- migration と index の見直しで versioned read model の性能劣化を抑えられる

### Cons / Tradeoffs

- 主キー変更を伴うため migration の影響範囲が広く、read/write の両方を揃えて更新する必要がある
- version を引数で受ける repository contract が増え、呼び出し側の責務が明示的になる
- active version 管理を誤ると、正しい row が存在しても UI から見えない状態を作り得る
