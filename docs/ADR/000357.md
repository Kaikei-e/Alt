# buf.yaml `excludes: [clients]` ハックの廃止 — buf v2 managed mode による proto 重複排除

## ADR's STATUS

Accepted

## CONTEXT

### `clients/` ディレクトリの経緯

`proto/` 配下には `services/` と `clients/` という2つのディレクトリが存在していた。Go の `protoc-gen-go` は `go_package` オプションをプロトファイル内にハードコードする仕様のため、同じ `.proto` ファイルでも消費する Go モジュールごとに異なる `go_package` を設定する必要があった。

この制約を回避するため、`clients/` に `services/` の proto ファイルをコピーし、`go_package` だけを変更した**重複ファイル**を5つ維持していた:

| services/ (本体) | clients/ (コピー) | 消費サービス |
|---|---|---|
| `services/search/v2/search.proto` | `clients/search/v2/search.proto` | alt-backend |
| `services/mqhub/v1/mqhub.proto` | `clients/mqhub/v1/mqhub.proto` | alt-backend, pre-processor |
| `services/preprocessor/v2/preprocessor.proto` | `clients/preprocessor/v2/preprocessor.proto` | alt-backend |
| `services/backend/v1/internal.proto` | `clients/backend/v1/internal.proto` | search-indexer |
| `services/backend/v1/internal.proto` | `clients/preprocessor-backend/v1/internal.proto` | pre-processor |

### 問題点

1. **protobuf パッケージ名の衝突**: `clients/` と `services/` が同一の protobuf パッケージ名（例: `services.backend.v1`）を持つファイルを含んでいたため、`buf lint` でシンボル重複エラーが発生。回避策として `buf.yaml` に `excludes: [clients]` を設定していた

2. **コード生成ワークフローの煩雑さ**: `clients/` 用のコード生成時に excludes を一時的に外す → 生成 → 戻す、という手動操作が必要だった

3. **ワイヤー非互換バグ**: `clients/preprocessor/v2/preprocessor.proto` のパッケージ名が `clients.preprocessor.v2` であったため、Connect-RPC の呼び出しパスが `/clients.preprocessor.v2.PreProcessorService/Summarize` となり、サーバー側の `/services.preprocessor.v2.PreProcessorService/Summarize` と一致しない潜在的バグが存在していた

4. **同期漏れリスク**: `services/` 側のメッセージ定義を変更した際に `clients/` 側の更新を忘れると、サイレントな型不整合が発生する

## DECISION MAKING

### buf v2 managed mode による `go_package` の生成時オーバーライド

buf v2 の managed mode を使い、コード生成時に `go_package` を proto ファイルの外側からオーバーライドする方式に移行した。これにより、1つの正規 proto ファイル（`services/`）から複数の Go モジュール向けに異なる `go_package` でコード生成が可能になる。

```yaml
# buf.gen.alt-backend-services.yaml（例）
version: v2
managed:
  enabled: true
  override:
    - file_option: go_package
      path: services/search/v2/search.proto
      value: alt/gen/proto/services/search/v2;searchv2
```

### 変更概要

#### 1. `proto/buf.yaml` — excludes の削除

`clients/` ディレクトリが消えたことでパッケージ名の衝突がなくなり、`excludes` 設定が不要になった。

#### 2. `proto/clients/` — ディレクトリの完全削除

5つの重複 proto ファイルをすべて削除。正規の `services/` ファイルのみを残す。

#### 3. 3つの gen config を managed mode に移行

| 設定ファイル（リネーム後） | 対象サービス | オーバーライド内容 |
|---|---|---|
| `buf.gen.alt-backend-services.yaml` | alt-backend | search, mqhub, preprocessor の `go_package` |
| `buf.gen.search-indexer.yaml` | search-indexer | backend の `go_package` |
| `buf.gen.pre-processor-services.yaml` | pre-processor | mqhub, backend の `go_package` |

#### 4. 生成済みコードのパス変更

各サービスの `gen/proto/clients/` を削除し、`gen/proto/services/` に統一:

- `alt-backend`: `gen/proto/clients/{search,mqhub,preprocessor}` → `gen/proto/services/{search,mqhub,preprocessor}`
- `search-indexer`: `gen/proto/clients/backend` → `gen/proto/services/backend`
- `pre-processor`: `gen/proto/clients/{mqhub,preprocessor-backend}` → `gen/proto/services/{mqhub,backend}`

#### 5. Go import パスの更新

影響を受ける Go ソースファイル（9ファイル）の import を `clients/` → `services/` に変更:

- **alt-backend** (3ファイル): `driver/search_indexer_connect`, `driver/mqhub_connect`, `driver/preprocessor_connect`
- **search-indexer** (1ファイル): `driver/backend_api/client.go`
- **pre-processor** (5ファイル): `driver/backend_api/*.go`, `driver/mqhub_connect/client.go`

#### 6. gen config ファイルのリネーム

`clients` という概念が消えたため、ファイル名を実態に合わせて変更:

- `buf.gen.alt-backend-clients.yaml` → `buf.gen.alt-backend-services.yaml`
- `buf.gen.pre-processor-clients.yaml` → `buf.gen.pre-processor-services.yaml`

Makefile の参照も合わせて更新。

## RESULTS, EFFECTS

### PROS

- **proto の単一ソース化**: 各 RPC 定義が `services/` に1つだけ存在し、重複管理が不要になった
- **ビルドワークフローの簡素化**: `make buf-generate` を1回実行するだけで全サービスのコード生成が完了。excludes の着脱操作が不要
- **ワイヤー互換性の修正**: alt-backend の preprocessor クライアントが `services.preprocessor.v2` パッケージのスタブを使うようになり、サーバーと一致するようになった
- **`buf lint` の完全適用**: excludes がなくなり、すべての proto ファイルが lint 対象になった
- **同期漏れリスクの排除**: ソースが1つしかないため、定義変更が自動的に全消費サービスに反映される

### CONS, TRADEOFF

- **buf v2 managed mode への依存**: buf v1 では利用できない機能。buf のバージョンアップが前提となる（既に v2 を使用中のため実質的な影響なし）
- **gen config ファイルの複雑化**: 各 config に `managed.override` セクションが追加される。ただし `clients/` の重複管理と比較すれば運用コストは低い
