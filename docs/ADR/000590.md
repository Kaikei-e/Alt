# ADR-000590: tdd-workflow スキルに CDC Contract Testing フローを統合する

## Status

Accepted

## Date

2026-03-26

## Affected Services

- 全サービス（開発ワークフロー変更）

## Context

[[000589]] で Pact CDC テスト基盤を導入したが、開発者が日常のコーディングフローで CDC テストを忘れずに実行する仕組みがなかった。

Alt プロジェクトでは Claude Code の `/tdd-workflow` スキルが TDD の RED→GREEN→REFACTOR を強制する開発フローの中核を担っている。サービス境界を跨ぐ変更時に CDC テストの実行を自然に促すには、このスキルに組み込むのが最も効果的。

### PM から学んだ教訓

| PM | 教訓 | tdd-workflow への反映 |
|----|------|---------------------|
| PM-004 | REST/Connect-RPC 混在 | Phase 0 で proto 互換性チェック |
| PM-006 | セマフォバイパス | Service Boundary Checklist |
| PM-008 | options 不一致 | LLM パラメータ一致チェック |

## Decision

`.claude/skills/tdd-workflow/SKILL.md` に以下の変更を加える。

### Phase 0: CONTRACT CHECK の追加

既存の Phase 1（RED）の前に Phase 0 を新設。サービス境界を横断する変更かどうかを判定し、該当する場合は CDC テストを先行して書く。

**判定基準:**
1. proto ファイルの変更 → `buf lint` + `buf breaking`
2. サービス間 HTTP API の変更 → Pact consumer テスト先行
3. Ollama options / LLM パラメータの変更 → options 一致検証

### Phase 3 に Contract Regression Check を追加

リファクタリング後に CDC テストを再実行し、契約が壊れていないことを確認する。

### Service Boundary Checklist の追加

PM から学んだ 5 項目のチェックリストをスキルに埋め込み:

- Proto 互換性（`buf breaking`）
- Options 一致（全リクエストパスで同一 LLM パラメータ）
- セマフォルーティング（GPU リクエストが HybridPrioritySemaphore を経由）
- Content-type ハンドリング（Connect-RPC の全シリアライゼーション形式対応）
- CDC テスト更新（Consumer の期待と Provider の実装が一致）

### Anti-Patterns に 3 項目を追加

- CDC テスト更新なしの API 変更
- 異なるリクエストパスからの LLM options 不一致
- GPU セマフォバイパス

### テストコマンドテーブルの拡張

Unit Test コマンドに加え、CDC Consumer Test コマンドを言語別に追記。

## Consequences

### Pros

- サービス境界の変更時に CDC テストの実行が TDD フローに自然に組み込まれる
- PM-004/006/008 で学んだ教訓がスキルのチェックリストとして定着し、再発防止が自動化される
- 新規開発者が `/tdd-workflow` を使うだけで CDC の存在と実行方法を知ることができる

### Cons / Tradeoffs

- Phase 0 の追加でスキルの実行ステップが増える（ただし、サービス境界に関係ない変更ではスキップ）
- SKILL.md が 120 行 → 182 行に増加（参照テーブルの追加分）
