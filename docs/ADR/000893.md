# ADR-000893: /loop の Decide / Act を semantic feedback loop に揃え、knowledge_loop.acted.v1 を Surface Planner v2 の Continue 信号として直接消費する

## Status

Accepted

## Date

2026-05-09

## Affected Services

- proto
- alt-frontend-sv (SvelteKit)
- alt-backend (Go)
- knowledge-sovereign (Go)
- docs (runbook / plan)

## Context

`fb.md` (2026-05-08 audit) と [[knowledge-loop-canonical-contract]] が共通して指摘していた未達は **「ユーザーが何を選んで何に対して行動したか」が `/loop` 上の操作から知識状態に戻っていない**ことだった。具体的には次の 3 点に集約される。

1. **Tile / workspace CTA の semantic 不揃い**: `LoopEntryTile.svelte:183-205` の `transitionMetadata` は `open` / `save` のときしか `acted_intent` / `target_type` を組み立てず、`compare` / `revisit` は metadata を欠いた状態で transition を発行していた。`snooze` / `ask` / `Open Recap` に至っては transition そのものが飛ばされており、`+page.svelte:561-586` の workspace decision option button は `onclick={() => advanceEntry(activeEntry)}` でステージを 1 段進めるだけだった。
2. **Backend payload の挙動非対称**: `transition_knowledge_loop_usecase.go::buildTransitionEvent` は `continue_flag=true` のときだけ payload に書き、`false` は省略していた。これでは projector / resolver が「false」と「未指定」を区別できず、Save (continue=false) と stage-only Acted が同じ payload 形になっていた。
3. **Surface Planner v2 が user behavior を見ていない**: `EventLogSurfaceScoreResolver.resolverEventTypes` は `knowledge_loop.*` を allowlist から除外しており、Wave 4-A で配線済みの v2 resolver が Recap / Augur / version drift しか参照できていなかった。`/loop` の bucket 配置は実質 v1 fallback (event-type → bucket fixed mapping) のままで、ユーザー行動が Continue 配置に効かない状態だった。

これらの結果、Surface Planner v2 の足場 ([[000852]] / [[000854]] / [[000855]] / [[000892]]) は本番に組み込まれているのに、Knowledge Loop の "賢さ" は依然として event-type router 寄りで、構想 ([[knowledge-loop-canonical-contract]] §6) の "知識状態に応じて UI が再配置される" 段階に届いていなかった。

`docs/plan/knowledge-loop-completion-02-semantic-decide-act.md` (Phase 2) は本作業の北極星で、§1 の操作 → intent / target マッピング表をエンドツーエンドで実装することが求められた。

## Decision

Phase 2 を以下の 6 つの判断に分けて全層で land する。Acolyte の deep Orient / Decide engine 化 ([[acolyte/data-flow]]) は Phase 4 で別判断するため、本 ADR の範囲には含めない。

### 1. ActTargetType に additive な proto bump を入れる

plan §1 の操作表は Ask に `target_type=conversation`、Snooze / Revisit に `target_type=entry` を求めた。既存 proto allowlist (`article` / `ask` / `recap` / `diff` / `cluster`) に流すと前者は intent と同名の `ask` に潰れ、後者は表現できず target を省略するしかなくなる。意味論を犠牲にせず Phase 2 を閉じるため、`proto/alt/knowledge/loop/v1/knowledge_loop.proto` の `ActTargetType` enum に `CONVERSATION = 6` / `ENTRY = 7` を **末尾追加**する additive bump を採用した。

却下した代替案:

- **Phase 4 まで proto を据え置く**: 既存 enum で Ask=`ask`、Snooze/Revisit=`article` 流用 / target 省略。`ask` は intent と target で意味が衝突し、Snooze/Revisit が article 以外の対象 (entry そのもの) を持つときに target を表現できず、projector が user behavior を `target_type=article` で誤って分類する。Phase 2 完了後に Phase 3 の Review 設計で同じ衝突を踏み直す可能性も高いと判断し却下。
- **新 enum `EventLogActor` のような中間概念を別途作る**: 設計がオーバーエンジニアリングで、proto best practices ([Protobuf Style Guide](https://protobuf.dev/programming-guides/style/)) の「enum 値は append-only」原則とずれる。

### 2. Tile / workspace CTA の全てを共通 helper 経由の semantic transition に揃える

`alt-frontend-sv/src/lib/components/knowledge-loop/transition-metadata.ts` を新設し、`buildTransitionMetadata(entry, option)` / `buildAskTransitionMetadata(entry, conversationId)` / `buildRecapTransitionMetadata(entry, recapTarget)` の 3 関数を export する。tile (`LoopEntryTile.svelte`) と workspace (`+page.svelte`) は同じ helper を呼ぶので、押下場所による request body 差異が出ない。

CTA 別の意味論決定:

| CTA | acted_intent | target_type | continue_flag | trigger | 補足 |
|---|---|---|---|---|---|
| Open (article) | open | article | true | user_tap | tile / workspace 共通 |
| Save | save | article | false | user_tap | tile only |
| Ask | ask | conversation | true | user_tap | Augur 成功後に conversation_id を target_ref に詰める |
| Compare | compare | diff | false | user_tap | option が diff target を持つとき |
| Revisit | revisit | entry | true | user_tap | target_ref = entry.entryKey |
| Snooze | snooze | entry | false | defer | target_ref = entry.entryKey、dismiss path 前に transition emit |
| Open Recap | open | recap | true | user_tap | `<button>` に昇格、`safeRecapRoute` の URL ガードは流用、navigation 前に transition emit |

`useKnowledgeLoop.svelte.ts::onAsk` を `Promise<{conversationId?: string} | void>` に拡張し、Augur session 作成失敗時は Loop transition を一切発火しない (plan §3 「session 作成失敗時は Loop transition を送らない」を厳密に守る)。Open Recap は warn-and-continue: transition が失敗しても navigation を取り消さない (UI は fail loud にしない)。

### 3. Backend payload に `continue_flag` を常時書き出し、`presented_intents` を defense-in-depth で正規化する

`alt-backend/app/usecase/knowledge_loop_usecase/transition_knowledge_loop_usecase.go::buildTransitionEvent` を以下の挙動に変更:

- `actedIntent` が presented されているなら `continue_flag` を **常時** map に書く (true でも false でも)。stage-only Acted (Phase 1 以前のレガシー shape) のみ従来通り省略。
- `presented_intents` を `dedupeBounded(in, 8)` で正規化。BFF が validation で len <= 8 とユニークを既に enforce しているが、misbehaving caller への防衛線として backend にも同じ cap を持たせる。

### 4. Projector が `knowledge_loop.acted.v1` の semantic payload を `continue_context.recent_action_labels` に投影する

`knowledge-sovereign/app/usecase/knowledge_loop_projector` 配下に以下を land:

- `parseLoopTransitionPayload` を `acted_intent` / `action_id` / `target_type` / `target_ref` / `continue_flag` / `presented_intents` まで読むよう拡張。
- `continue_context_builder.go::buildContinueContextFromActed(recent)` を新設。引数は newest-first の Acted event slice、出力は JSONB body。最大 5 件 (`recentActionLabelsBound`) で bounded、同一 intent は dedupe、`last_interacted_at` は最新 event の `occurred_at`。pure function — projection row を一切読まない (reproject-safe)。
- `projector.go::projectTransition` の Acted 分岐に「`acted_intent != ""` なら `ListKnowledgeLoopActedEventsForEntry(ctx, user_id, tenant_id, lens_mode_id, entry_key, ev.EventSeq, 5)` で event log だけから直近 5 件を引き、上の builder で JSONB を作り `PatchKnowledgeLoopEntryContinueContext` で patch する」経路を追加。
- driver 層に `PatchKnowledgeLoopEntryContinueContext` (write) と `ListKnowledgeLoopActedEventsForEntry` (read) を新設。前者は `continue_context` 列だけを patch する `UPDATE ... WHERE projection_seq_hiwater <= $5` クエリで、why_text / freshness / surface_bucket は unchanged。後者は `event_seq <= untilSeq` で event 発生時点の seq で windowing するため、reproject 時も同じ事象集合を返す。
- F-001 mitigation: `ListKnowledgeLoopActedEventsForEntry` は `WHERE user_id = $1 AND tenant_id = $2` で物理 bind し、cross-user / cross-tenant の Acted を絶対に混入させない。

### 5. Surface Planner v2 resolver に Acted の Continue 信号を流し、feature flag なしで直 land する

`score_resolver_event_log.go::resolverEventTypes` allowlist に `EventKnowledgeLoopActed` を追加。resolver loop 内で:

- `readPayloadBool(payload, "continue_flag") == true` で gate (Snooze の continue=false は Continue 配置に効かない)。
- `readEntryKey(payload) == thisEntryKey` で entry-keyed (cross-entry の behaviour bleed を防ぐ)。
- 該当時に `RecentContinueActionCount` を increment。

`SurfaceScoreInputs` に `RecentContinueActionCount uint32` を追加し、`decideBucketV2` の Continue 判定式を `HasAugurLink || HasOpenInteraction || QuestionContinuationScore > 0 || RecentContinueActionCount > 0` に拡張する。

却下した代替案:

- **env flag (例: `KNOWLEDGE_SOVEREIGN_USE_LOOP_ACTED_CONTINUE_SIGNAL`) で gate**: rollout を遅らせる代わりに blast radius を絞れるが、resolver は本番に組み込み済みで invariants test と F-001 isolation test がカバーしているため、新 signal の追加は既存の安全網で十分。flag が長期化すると config 負債になる ([[deploy]] の "feature flag は cleanup 計画とセット" 原則)。

### 6. Pact CDC contract を semantic Acted で拡張する

`alt-backend/app/driver/sovereign_client/contract/knowledge_loop_event_consumer_test.go` に `TestAppendKnowledgeEvent_KnowledgeLoopActed_SemanticPayload` を sibling test として追加。base64 encode した payload に semantic field 6 種 (`presented_intents` / `acted_intent` / `action_id` / `target_type=ACT_TARGET_TYPE_CONVERSATION` / `target_ref=<conversation_id>` / `continue_flag=true`) を含める。既存の `TestAppendKnowledgeEvent_KnowledgeLoopActed` (stage-only) は legacy shape の retention guard として残す。

## Consequences

### Pros

- **意味のある user signal が初めて Surface Planner v2 を駆動する**: Open / Ask / Revisit / Open Recap で `RecentContinueActionCount > 0` が立った entry が Continue に再配置されるようになり、`/loop` が "event-type router" を脱して "user-state router" へ進む第一段が land した。`fb.md` §15 Priority 3 (Decide / Act を semantic event にする) を直接閉じる。
- **Reproject-safe を死守したまま実現**: `continue_context` 再構築は event log + versioned artifact だけで完結し、`PatchKnowledgeLoopEntryContinueContext` は `WHERE projection_seq_hiwater <= eventSeq` で seq guard 付き、`ListKnowledgeLoopActedEventsForEntry` は `event_seq <= untilSeq` で windowing する。truncate → projector run の reproject で `continue_context` と `act_targets` が同じ JSONB に再現される。canonical contract §3 / §6.5 の不変条件 ([[knowledge-loop-canonical-contract]]) と整合。
- **Single-emission rule を破らない**: `/loop` の user action は `knowledge_loop.acted.v1` 一系統のままで、`HomeItem*` を共発火しない。`augur.conversation_linked.v1` ([[000855]]) は引き続き rag-orchestrator から system signal として発火し、user transition と分離される。
- **CTA 実装が共通化されて divergence しなくなった**: tile と workspace で同じ helper を経由するため、push 場所による `/loop/transition` body の不一致が発生しない。E2E (`loop-decide-option-semantic.spec.ts` / `loop-open-recap-emits-transition.spec.ts`) と vitest helper test (`transition-metadata.spec.ts` 11/11 green) が回帰を捕捉する。
- **Pact contract が semantic shape を pin**: 将来 Acted から semantic field を落とすような regression は consumer test と provider verification で必ず捕まる。

### Cons / Tradeoffs

- **proto consumer の再生成と broker 同期が必要**: ActTargetType の 2 値追加で Go (`alt-backend`, `alt-butterfly-facade`, `rag-orchestrator`, `knowledge-sovereign`) と TS (`alt-frontend-sv`) の binding を `buf generate` で再生成した。すべて consumer 側で additive 互換だが、enum を新値で受け取る consumer がまだ無い場合の `default:` fallback 挙動は実装依存なので、新 consumer を追加する際は default 振る舞いを ADR にメモする運用が必要。
- **`recent_action_labels` の bound (5) は経験則**: 短いほど UI の "Continue context" 行が読みやすいが、user の連続行動が多い entry では履歴が頻繁に押し出される。本番観測 (Wave 4 dashboard / [[000892]] alert) で trim 過多の兆候が出たら 7-10 まで上げ得る。bound を上げても reproject-safe は壊れない。
- **Snooze の continue_flag=false は Continue 配置に効かない**: 仕様としては正しい (Snooze は "後で再考" であり継続意思の表明ではない)。一方で Snooze 直後の entry が Now / Review に移ってしまう挙動はユーザー体験上の議論余地があり、Phase 3 (Review 再評価エンジン) で改めて扱う。
- **Workspace の Decide button refactor で `advanceEntry` パスが減った**: option を持たない stage advance は依然 `advanceEntry` だが、option-driven Decide は `onWorkspaceDecide(entry, option)` 経由になった。新 prop の伝搬を別の workspace コンポーネント (Acolyte report panel など Phase 4 で導入予定) でも統一する必要がある。
- **E2E 2 本 (`loop-decide-option-semantic` / `loop-open-recap-emits-transition`) は fixture wiring 待ち**: `lens=e2e-decide` / `lens=e2e-recap` を返す mock handler を `tests/e2e/infra/handlers/backend.ts` に追加するまで dev server 上で実行できない。helper 単体 (`transition-metadata.spec.ts`) と Go projector / resolver 側のテストが意味論を pin しているため信号は失われていないが、Playwright 側の自走を完全に立てる作業が follow-up として残る。
