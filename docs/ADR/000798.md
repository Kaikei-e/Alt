# ADR-000798: Pact Broker publish の並列レースを consumer 所有制 + pact-broker-cli --merge で解消する

## Status

Accepted

## Date

2026-04-19

## Affected Services

- **scripts/pact-check.sh** — `--publish-only` モードの publish ブロックを全面書き換え
- **scripts/tests/test-pact-check-filter.sh** — publish フィルタの TDD 用 dry-run assertion を追加
- **alt-deploy (release-deploy.yaml)** — 呼び出し側は無改修、挙動のみ leg 所有制に追従

## Context

`release-deploy` ワークフローの `pact-publish` matrix leg が HTTP 409 Conflict で失敗した（run 24628291603, job 72012459447）。最終ログは以下:

```
Publishing recap-evaluator -> recap-worker
curl: (22) The requested URL returned error: 409
Process completed with exit code 22
```

### 根本原因

`scripts/pact-check.sh` の publish ループ（旧 L387-406）は `--services` フィルタを**一切参照せず**、ワークスペース配下の全 pact ファイル (`alt-backend/pacts/*.json pacts/*.json rag-orchestrator/pacts/*.json`) を一括 publish していた。このため:

- alt-deploy の matrix は `(alt-backend, rag-orchestrator, recap-worker, tag-generator)` × `(consumer, provider)` の最大 8 並列で同じ 18 本の pact ファイルを同時 `PUT /pacts/provider/X/consumer/Y/version/$SHA` に投げる
- consumer テストが leg ごとに一部ファイルを再生成するため、**同一 `consumer_version`（= git SHA）に微妙に異なる body** が並列到着する
- 公式の legacy PUT エンドポイントは、`allow_dangerous_contract_modification=false`（既定かつ推奨）のもとで異内容再 publish を 409 で拒否する。curl は `--fail` により 22 で抜け、notices（diff ボディ）は隠される

### 公式仕様の確定

Web 調査（2026-04-19 時点）で一次ソースを確認:

- **409 の厳密な条件**（[publish-contracts.markdown](https://github.com/pact-foundation/pact_broker/blob/master/lib/pact_broker/doc/views/index/publish-contracts.markdown)）: 「If there is a conflict with an existing published pact and `allow_dangerous_contract_modification` is set to false.」。同一 content の再 publish は no-op で安全
- **`pact-broker-cli publish --merge`**（[pact-broker-cli README](https://github.com/pact-foundation/pact-broker-cli)、Rust 版公式 CLI）: 「If a pact already exists for this consumer version and provider, merge the contents. Useful when running Pact tests concurrently on different build nodes.」
- **consumer_version は git SHA 推奨**（[docs.pact.io versioning](https://docs.pact.io/getting_started/versioning_in_the_pact_broker)）: Alt は既に PR/SHA を `PACT_PROVIDER_VERSION` に流しているので追加変更不要
- **`allow_dangerous_contract_modification=true` は非推奨**: 「dangerous」の名の通り contract 破壊を隠蔽するので broker 側設定は既定のまま

### 影響

- 失敗した run は CI のみで止まり、本番反映は走っていない（`deploy` ステップは gate-pass した他サービスのみ反映、tag-generator 以外は pass していたため小規模反映済み）
- 同一 SHA で再 dispatch を試みる限り必ず 409 が再発する構造的欠陥だった（SHA を進めるまで詰む）

## Decision

以下 3 点を恒久対応として採用する。

### (1) Publish の consumer 所有制（案 A）

`pact-check.sh` の publish ループを「`.consumer.name` が `--services` CSV に一致する pact ファイルだけ」に絞る。`should_publish_pact_file` を新設し、filter に該当しないファイルは `continue` で捨てる。各 matrix leg は「自分が consumer である pact」だけを publish し、重複 publish がそもそも起こらない。

```bash
should_publish_pact_file() {
  local consumer_name="$1"
  [[ -z "$SERVICE_FILTER" ]] && return 0
  local IFS=','
  for svc in $SERVICE_FILTER; do
    [[ -z "$svc" ]] && continue
    [[ "$consumer_name" == "$svc" ]] && return 0
  done
  return 1
}
```

### (2) Provider role leg は publish をスキップ

`ROLE_FILTER == "provider"` の leg は pact ファイルを生成しない（provider 検証しかしない）ので、publish ブロック自体に入らない。重複 publish の発生源を物理的に半減させる。

### (3) `curl` 直 PUT を `pact-broker-cli publish --merge` に統一（案 B）

legacy の `PUT /pacts/provider/X/consumer/Y/version/Z` を廃止し、`pact-broker-cli` の `/contracts/publish` エンドポイント経由に揃える。`--merge` を恒久付与し、残存する並列 build-node race（タイムスタンプ・interaction 順序の揺れ）に対する防御深度を確保する。副次効果として 409 時の notices（diff ボディ）が CLI ログに展開され、観察性が上がる。

```bash
pact-broker-cli publish "${FILES_TO_PUBLISH[@]}" \
  --broker-base-url "$PACT_BROKER_BASE_URL" \
  --broker-username "$PACT_BROKER_USERNAME" \
  --broker-password "$PACT_BROKER_PASSWORD" \
  --consumer-app-version "$VERSION" \
  --branch "$BRANCH" \
  --build-url "${GITHUB_SERVER_URL}/${GITHUB_REPOSITORY}/actions/runs/${GITHUB_RUN_ID}" \
  --merge
```

`--branch` 指定で `PUT /pacticipants/X/branches/Y/versions/Z` の副次呼び出しを CLI が内蔵するため、旧コードの 2 本目の curl PUT は削除。

### 代替案と却下理由

| 案 | 却下理由 |
|---|---|
| `allow_dangerous_contract_modification=true` に倒す | 「dangerous」の名の通り contract 破壊を隠蔽し、can-i-deploy の信頼性が崩壊する。公式 anti-pattern |
| matrix 間に lock / mutex artifact を挟む | workflow 複雑度が増す割に、content 差分が本質の問題（非決定性）には効かない |
| consumer_version に leg 識別子を付与（例: `$SHA-$SERVICE`） | broker 上に同一 SHA で複数 version が乱立し、can-i-deploy の照合先が曖昧化。公式推奨の「git SHA ベース」に反する |
| `--merge` 単独（案 A 無し） | 並列 8 publish 全部に merge を通すとワークフローが泥沼化する（Ruby 版コミュニティの「`write_method = :merge` はワークフローを泥沼化させる」警告と整合）。構造的解決（案 A）が先 |

### TDD 手順

1. **RED**: `scripts/tests/test-pact-check-filter.sh` に `WOULD PUBLISH: consumer=X provider=Y file=Z` を期待する 12 assertion を先に追加 → 6 FAIL を確認
2. **GREEN**: `pact-check.sh` を書き換え → 30/30 pass
3. **Phase 5 CI parity**: `docker run --rm -v $PWD:/mnt koalaman/shellcheck:stable ...` で shellcheck を通し、新規警告ゼロを確認（既存 SC2034 は nameref 起因の false positive）

## Consequences

### Pros

- **409 が構造的に発生しなくなる**: consumer 所有制で同一 pact を複数 leg が publish する経路を断つ
- **並列 build node race への防御**: 万一 race が残っても `--merge` が救う（公式想定のユースケース）
- **観察性向上**: 409 時の notices（diff ボディ）が `pact-broker-cli` ログに展開される。従来は `curl --fail` で隠されていた
- **dry-run 網羅性**: `WOULD PUBLISH:` 行でフィルタ挙動をローカル検証可能。filter のデグレを CI で即検出できる
- **workflow 不変**: `alt-deploy` の `release-deploy.yaml` は無改修。`pact-check.sh` 内部に閉じた変更

### Cons / Tradeoffs

- **`pact-broker-cli` が runner に必須**: 既に `ensure_kratos_external_version` で利用中（L480）なので実質追加制約なし。`PACT_BROKER_BIN` 環境変数で差し替えは可能
- **provider leg が publish しなくなる**: 既存運用では consumer leg が先行して publish していたので実害なし。ただし「何らかの事情で consumer leg が走らず provider leg だけ走る」運用モードが将来登場した場合、pact が publish されないので `can-i-deploy` が古い version で判定する。現在の matrix 構成では発生しない前提
- **`netrc` / `validate_pacticipant_name` / `curl` 直 PUT ヘルパの残骸**: `pact-broker-cli` が内蔵するため冗長化したが、本 PR では最小差分を優先して残す。次 PR で整理予定
- **pact 非決定性の根治ではない**: 同一 SHA で content が揺れる真因（タイムスタンプ・順序非決定性）は本策で蓋をするだけ。頻発するなら consumer 側 pact 生成器の決定性を別途上げる必要がある
