# ADR-000508: Feed management UI で購読状態を mutation 後に再同期する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-frontend-sv (SvelteKit)

## Context

Feed management 画面では、feed 一覧と購読一覧を別 API から取得している。ところが登録成功後は `feedLinks` だけを再取得し、`subscriptions` は初期読込時の内容を保持したままだった。また、購読トグル更新時のローカル state 更新は `subscriptions.map(...)` に依存しており、新しく登録された feed が配列に未存在の場合は UI が変化しなかった。

その結果、backend では auto-subscribe や subscribe API が成功していても、管理画面上ではトグルが操作不能に見える stale state が発生していた。

## Decision

**Feed management UI は、登録・削除・購読変更の mutation 後に購読状態を再同期し、未存在要素にも対応できるローカル state 更新を採用する。**

今回の決定では以下を採用する。

- 登録成功後は `feedLinks` と `subscriptions` を両方再取得する
- 削除成功後も `feedLinks` と `subscriptions` を両方再取得する
- トグル操作時は既存要素の更新だけでなく、未存在なら新規 state を補う
- トグル失敗時はローカル state を信用せず、`subscriptions` を再取得して正に戻す

### 代替案

| 案 | 評価 |
|---|---|
| mutation 後は常にページ全体を再読込する | 単純だが UX と実装の局所性が悪いため不採用 |
| ローカル state を完全 optimistic にして再取得しない | API 非同期性や auto-subscribe の遅延に弱いため不採用 |
| `feedLinks` API に購読状態を混ぜる | 責務分離を崩すため不採用 |

## Consequences

### Pros

- 新規登録直後でもトグル表示が stale になりにくい
- subscribe / unsubscribe 成功時の UI 反映が安定する
- API が分離されたままでも管理画面の一貫性を維持できる

### Cons / Tradeoffs

- mutation 後の API 呼び出し回数は増える
- UI state 管理がやや複雑になる
- 一時的な遅延時には複数データソースの同期を意識する必要がある
