# ADR-000574: alt-backend の Handler DIP 改善とテストのドメイン Repository 移行を完了する

## Status

Accepted

## Date

2026-03-25

## Affected Services

- alt-backend (Go)

## Context

[[000572]]（Phase 1-3, 7）と [[000573]]（Phase 4, 6, 8）に続く SOLID リファクタリングの最終フェーズ。以下の2つの問題が残存していた。

1. **Handler の DIP 違反**: Feeds Handler と Articles Handler が `*di.ApplicationComponents`（70+ フィールドの DI コンテナ全体）に依存しており、実際に使用する 13〜17 個の Usecase 以外にもアクセス可能だった。テスト時にもコンテナ全体の構築が必要で、依存関係が型から不可視だった。
2. **テストの AltDBRepository 依存**: Phase 1 で AltDBRepository を 13 ドメイン Repository に分割したが、テストファイルは移行措置として `NewAltDBRepository(mock)` を使い続けていた。これではドメイン分割の恩恵（最小依存・テスト効率）が活かせていなかった。

## Decision

### Phase 5: Handler の DIP 改善 — Deps struct パターンの導入

Handler が必要とする Usecase のみを明示的に注入する `Deps struct` パターンを採用する。

**Feeds Handler（17 依存）:**

```go
type FeedHandlerDeps struct {
    // Feed read
    CachedFeedList           *cached_feed_list_usecase.CachedFeedListUsecase
    FetchReadFeedsCursor     *fetch_feed_usecase.FetchReadFeedsListCursorUsecase
    FetchFavoriteFeedsCursor *fetch_feed_usecase.FetchFavoriteFeedsListCursorUsecase
    FeedSearch               *search_feed_usecase.SearchFeedMeilisearchUsecase
    ListSubscriptions        *subscription_usecase.ListSubscriptionsUsecase
    // Feed write
    ArticlesReadingStatus    *reading_status.ArticlesReadingStatusUsecase
    Subscribe                *subscription_usecase.SubscribeUsecase
    Unsubscribe              *subscription_usecase.UnsubscribeUsecase
    // Feed stats
    FeedAmount               *fetch_feed_stats_usecase.FeedsCountUsecase
    UnsummarizedCount        *fetch_feed_stats_usecase.UnsummarizedArticlesCountUsecase
    SummarizedCount          *fetch_feed_stats_usecase.SummarizedArticlesCountUsecase
    TotalCount               *fetch_feed_stats_usecase.TotalArticlesCountUsecase
    TodayUnreadCount         *fetch_feed_stats_usecase.TodayUnreadArticlesCountUsecase
    // Feed summary
    AltDBRepository          *alt_db.AltDBRepository
    PreProcessorClient       *preprocessor_connect.ConnectPreProcessorClient
    CreateSummaryVersion     *create_summary_version_usecase.CreateSummaryVersionUsecase
    // Shared
    ImageProxy               *image_proxy_usecase.ImageProxyUsecase
}

type Handler struct {
    deps   FeedHandlerDeps
    logger *slog.Logger
    cfg    *config.Config
}
```

**Articles Handler（13 依存）:**

```go
type ArticleHandlerDeps struct {
    AltDBRepository         *alt_db.AltDBRepository
    ArchiveArticle          *archive_article_usecase.ArchiveArticleUsecase
    Article                 fetch_article_usecase.ArticleUsecase
    FetchArticlesByTag      *fetch_articles_by_tag_usecase.FetchArticlesByTagUsecase
    FetchArticlesCursor     *fetch_articles_usecase.FetchArticlesCursorUsecase
    FetchArticleSummary     *fetch_article_summary_usecase.FetchArticleSummaryUsecase
    FetchArticleTags        *fetch_article_tags_usecase.FetchArticleTagsUsecase
    FetchInoreaderSummary   fetch_inoreader_summary_usecase.FetchInoreaderSummaryUsecase
    FetchLatestArticle      *fetch_latest_article_usecase.FetchLatestArticleUsecase
    FetchRandomSubscription *fetch_random_subscription_usecase.FetchRandomSubscriptionUsecase
    FetchTagCloud           *fetch_tag_cloud_usecase.FetchTagCloudUsecase
    ImageProxy              *image_proxy_usecase.ImageProxyUsecase
    StreamArticleTags       *stream_article_tags_usecase.StreamArticleTagsUsecase
}
```

Deps struct を選択した理由:
- 17 個の位置引数は可読性が低く、引数の順序ミスを招く
- Deps struct はフィールド名付きで初期化するため、順序に依存しない
- テスト時に必要なフィールドだけ設定し、残りはゼロ値（nil）で済む
- knowledge_home / internal handler は既にコンストラクタ引数パターンで container 非依存だったため変更不要

`server.go` で Deps struct を DI container フィールドから構築し、Handler に渡す。これにより `connect/v2/feeds/` と `connect/v2/articles/` パッケージから `alt/di` import が完全に除去された。

### Phase 9: テストのドメイン Repository 移行

12 テストファイル・34 箇所の `NewAltDBRepository(mock)` を、テスト対象メソッドのレシーバー型に対応するドメイン Repository に置換した。

| ドメイン Repository | 対象テストファイル数 | 置換数 |
|---|---|---|
| `FeedRepository` | 7 | 14 |
| `DashboardRepository` | 2 | 7 |
| `InternalRepository` | 1 | 5 |
| `TagRepository` | 1 | 4 |
| `ScrapingRepository` | 1 | 4 |

例:
```go
// Before: 全13ドメインのRepositoryを内部生成
repo := NewAltDBRepository(mock)

// After: テスト対象のドメインRepositoryのみ構築
repo := &FeedRepository{pool: mock}
```

これにより:
- テストが実際にどのドメインのメソッドをテストしているか型から明確になった
- 不要な12ドメイン分の Repository 構築を省略し、テスト実行が効率化された
- `NewAltDBRepository(mock)` の使用箇所が driver/alt_db テストから 0 件になった

## Consequences

### Pros

- **DIP 完全遵守（Connect-RPC Handler）**: Feeds/Articles Handler が `alt/di` パッケージに一切依存しない。依存は Deps struct の型シグネチャで完全に明示
- **テストのセットアップ簡素化**: Handler テストで `di.ApplicationComponents` 全体を構築する必要がなくなり、必要なフィールドだけ設定すれば済む
- **コンパイル時の依存制御**: Handler が使用していない Usecase にアクセスする手段がない（`h.deps` に存在しないフィールドは参照不可）
- **ドメイン分割の恩恵がテストに浸透**: 各テストが正しいドメイン Repository を直接使用し、メソッドの所属が型レベルで文書化された
- **全 122 packages テスト GREEN**: リグレッションなし

### Cons / Tradeoffs

- **REST Handler は未改善**: `rest/` パッケージの 24 ファイルは依然として `*di.ApplicationComponents` を受け取るクロージャパターンを使用。Connect-RPC への移行が進んでいるため優先度を下げた
- **Deps struct のフィールド数**: FeedHandlerDeps が 17 フィールドは多いが、これは Feeds Handler が実際に 17 の異なる Usecase/Driver を必要としている事実の反映であり、隠蔽すべきではないと判断した。将来的にサブハンドラ分割（read/write/stats 別の struct）に進化させることも可能
