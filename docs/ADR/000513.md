# ADR-000513: 要約 queue 作成前に summary 既存と直近成功を確認する

## Status

Accepted

## Date

2026-03-21

## Affected Services

- pre-processor (Go)

## Context

`summarize_job_queue` の `CreateJob` は、同一 `article_id` に対して `pending` / `running` が存在する場合だけ insert を抑止していた。
このため、いったん `completed` になった記事でも、上流の再通知や再保存が発生すると再び queue に積むことができた。

実データでは queue 履歴総数が記事総数を大きく上回り、一部の記事には多数の completed job が蓄積していた。
これは worker の throughput 不足だけでは説明できず、**queue 作成前の admission control が弱すぎる** ことを示していた。

上流の event 意味論修正 ([[000512]]) だけでも発生量は減るが、event の揺れや運用上の再送が完全にゼロになる保証はない。
そのため `pre-processor` 自身でも、要約済み記事に対する queue 作成を抑止する防御線が必要だった。

## Decision

**要約 job を新規作成する前に、既存 summary と直近成功 job を確認する guard を導入する。**

具体的には以下を採用する。

1. `article_summaries` に summary が既に存在する場合は queue を作成しない
2. summary がなくても、同一記事に対して直近 24 時間以内の successful job がある場合は queue を作成しない
3. この guard を event consumer、HTTP handler、Connect handler の queue 作成経路に共通適用する
4. skip 時はエラーではなく明示的な `skipped` 応答またはログで扱う

この決定により、queue 側は「今あらためて要約が必要な記事だけを受け入れる」役割へ近づく。

### 採用しなかった代替案

| 案 | 評価 |
|---|---|
| `CreateJob` の `pending/running` チェックだけ維持する | completed 後の再 enqueue を防げず、今回の膨張を止められない |
| summary 存在だけを見る | 保存直前の race や一時不整合時に再投入が起こりうるため、防御線として弱い |
| 永久に 1 記事 1 job にする | 再要約や将来の再処理要求を完全に塞いでしまい、運用の自由度を失う |

## Consequences

### Pros

- 既に要約済みの記事へ queue を積み増す事態を大幅に減らせる
- event の揺れや再送があっても `pre-processor` 側で自己防衛できる
- queue の `pending` 件数が「実際に処理すべき仕事量」に近づく

### Cons / Tradeoffs

- queue 作成前に追加の read が入るため、job 作成 path の DB/API 往復が増える
- 24 時間の成功 window は運用上の値であり、将来調整が必要になる可能性がある
- 意図的な再要約要求を行う場合は、この guard をバイパスまたは条件変更する設計が別途必要になる
