# ADR-000746: 本番 Pact gate と record-deployment を単一ホスト側の手動デプロイスクリプトに閉じる

## Status

Accepted

## Date

2026-04-15

## Affected Services

`scripts/pre-deploy-verify.sh` と `scripts/deploy.sh` の gate を通過する
pacticipant 14 サービス全て。

- alt-backend (Go)
- pre-processor (Go)
- search-indexer (Go)
- mq-hub (Go)
- rag-orchestrator (Go)
- recap-worker (Rust)
- recap-subworker (Python)
- recap-evaluator (Python)
- news-creator (Python)
- tag-generator (Python)
- tts-speaker (Python)
- acolyte-orchestrator (Python)
- alt-butterfly-facade (Go)
- auth-hub (Go)

## Context

[[000740]] で Pact Broker を default profile に昇格させ、GitHub Actions の
`release-gate.yaml` で `can-i-deploy` を回して Docker publish を止める設計
にした。CI 側から Broker (`localhost:9292`) に到達させるため self-hosted
runner を要し、Broker の basic-auth パスワードを `secrets.PACT_BROKER_PASSWORD`
として GitHub 側に保存していた。`deploy.yaml` は `appleboy/ssh-action` で
本番ホストへ SSH し、`DEPLOY_HOST` / `DEPLOY_USER` / `DEPLOY_SSH_KEY` も
GitHub secret として抱えていた。

この構成には 3 つの噛み合わせ問題があった。

1. **OSS リポに本番資格情報を持たせている** — Alt は公開リポで、コミッター
   以外にも fork/PR が来る。GitHub secret はワークフロー実行時に渡されるが、
   悪意のある PR の secret 流出は過去に何度も報告されている。本番 SSH 鍵と
   Pact Broker パスワードを GitHub 側に置く必要性自体が弱い。
2. **本番は単一ホスト構成で外部公開されていない** — `localhost:9292` で
   Broker が常時起動しているのは本番ホストそのもの。CI から Broker を
   叩くには self-hosted runner を常に本番ホスト上で動かす必要があり、結局
   "本番ホストが走らせている" のと変わらない。CI を経由させる経路が余分。
3. **デプロイ発火が push/PR 契機で不本意** — Alt の運用実態は "ADR を
   書いた直後に人間が手で叩く" である。push 契機の自動デプロイは、ADR の
   commit とデプロイのタイミングがズレる・想定外のコミットでデプロイが走る
   などの事故源でしかない。

加えて、単一ホスト Docker Compose 構成では Blue-Green も Argo Rollouts も
使えない。[[000743]] で mTLS hard-cutover を完了し、X-Service-Token と
SERVICE_SECRET を全廃したので、境界の保護は consumer-driven contract
(Pact) + mTLS 相互認証の 2 本立てに一本化されている。**Pact gate が
回らない / 失敗検出ができないまま `docker compose up -d` が走ると、
mTLS は繋がるが契約不整合で 4xx/5xx が出て気づく** という事故の芽は残る。

## Decision

本番 gate と `record-deployment` を **GitHub Actions から単一ホスト側の
手動デプロイスクリプトへ全面移管** する。CI には本番資格情報を一切置かない。

具体策:

1. **`scripts/pre-deploy-verify.sh` を新設** する。以下を直列で実行し、
   どこかで失敗すれば exit 非ゼロで止まる:
   - Pact Broker heartbeat 到達確認
   - `pact-broker create-environment --production` を冪等呼出し (既存なら tolerate)
   - `scripts/pact-check.sh --broker` (既存の consumer+provider 全走)
   - `pact-broker can-i-deploy` を 14 pacticipants 分ループし、対象環境と
     互換 (compatible) であることを確認
2. **`scripts/deploy.sh` を新設** する。ADR 直後に人間が 1 コマンドで叩く
   エントリーポイント。パイプライン:
   - `pre-deploy-verify.sh` (`--skip-verify` で緊急回避可能・記録必須)
   - 直前 SHA を `.deploy-prev` に保存
   - `docker compose pull`
   - **レイヤ順 rolling recreate** — `_deploy_lib.sh:DEFAULT_LAYERS` の
     順に 1 サービスずつ `up -d --no-deps --force-recreate` → `docker
     compose ps` で healthy を最大 120s 待機 → 次レイヤへ
   - global smoke (`nginx /health`, `alt-backend /v1/health`,
     `alt-butterfly-facade /health`, `meilisearch /health`)
   - 成功時のみ `pact-broker record-deployment` を 14 件打刻
   - 途中失敗時は自動で前 SHA の `compose/` を `git checkout` して
     `docker compose up -d --remove-orphans` を叩き、`record-deployment`
     は打たずに非ゼロで終了
3. **`.github/workflows/release-gate.yaml` と `.github/workflows/deploy.yaml`
   を削除** する。CI は PR 時の `proto-contract.yaml` (file-based pact-check
   + buf breaking) のみ担い、本番状態を触らない。
4. **運用 runbook を 2 本追加/更新**:
   - `docs/runbooks/deploy.md` (新規) — ADR → `deploy.sh production`
     の手順、フラグ、レイヤ順、ロールバック、DB migration との同期
   - `docs/runbooks/pact-broker-ops.md` (更新) — ローカル gate を正規手順
     として昇格し、CI 側手順を履歴的注記に格下げ
5. **TDD テスト** — `tests/scripts/{lib.sh, test_pre_deploy_verify.sh,
   test_deploy.sh, run.sh}` に 11 ケースを bash で書き、bats-core などの
   追加依存なしで `bash tests/scripts/run.sh` 一発実行可能にする。
   stub は `$STUB_BIN` の PATH 前置と conditional shell stub で行う。

### 検討した代替案

| 代替案 | 採否 | 理由 |
|---|---|---|
| CI + self-hosted runner を維持 | 不採用 | OSS リポに本番 secret を置く根本問題が残る |
| `wowu/docker-rollout` で zero-downtime | 不採用 (将来候補) | `container_name`/`ports` 除去など compose 全面改修が必要。今回の関心事 (Pact gate) と切り離したい |
| Docker Swarm 単一ノード | 不採用 | [single-machine architecture 方針](../../memory) と矛盾。追加オーケストレータを入れたくない |
| record-deployment をローカルと CI で両打ち | 不採用 | Broker matrix の二重記録で整合が混乱する。真実ソースは 1 つに絞る |

## Implementation Overview

### ファイル構成

```
scripts/
  pact-check.sh               # 既存・無変更 (内部呼び出しのみ)
  pre-deploy-verify.sh        # 新規 — Broker gate
  _deploy_lib.sh              # 新規 — レイヤ順/待機/ロールバック/記録
  deploy.sh                   # 新規 — 手動デプロイ entrypoint

tests/scripts/
  lib.sh                      # 新規 — 最小 bash テストハーネス
  test_pre_deploy_verify.sh   # 新規 — 5 ケース
  test_deploy.sh              # 新規 — 6 ケース
  run.sh                      # 新規 — ディスパッチャ

docs/runbooks/
  deploy.md                   # 新規
  pact-broker-ops.md          # 更新 (ローカル gate を TL;DR に追加)

.github/workflows/
  release-gate.yaml           # 削除
  deploy.yaml                 # 削除
  proto-contract.yaml         # コメント差し替え (参照先 runbook を更新)
```

### `pre-deploy-verify.sh` の stub point

テスト容易性のため、以下を環境変数で差し替え可能にした:

- `PACT_CHECK_SCRIPT` — デフォルト `$REPO_ROOT/scripts/pact-check.sh`
- `PACT_BROKER_BIN` — デフォルト `pact-broker` (PATH 解決)
- `PACT_BROKER_PASSWORD` — 未設定なら `secrets/pact_broker_basic_auth_password.txt` から読む

14 pacticipants のリストは [[000740]] `release-gate.yaml:72-87` のものを
そのまま移植した。ずれた場合はローカル gate が真実ソースとなる。

### `deploy.sh` のレイヤ順

`_deploy_lib.sh:DEFAULT_LAYERS` に定義:

1. `step-ca`
2. `kratos`, `auth-hub`
3. `alt-backend`, `search-indexer`, `mq-hub`, `pre-processor`
4. `news-creator`, `tag-generator`, `recap-worker`, `recap-subworker`,
   `recap-evaluator`, `acolyte-orchestrator`, `rag-orchestrator`,
   `tts-speaker`
5. `alt-butterfly-facade`, `alt-frontend-sv`, `nginx`

DB 層 (`db`, `kratos-db`, `pre-processor-db`, `rag-db`, `meilisearch`,
`clickhouse`, `pact-db`) は **recreate 対象外**。schema 変更は Atlas
(`cd migrations-atlas && atlas migrate apply`) で単独実行し、その後
`deploy.sh` を回す。この順を逆にすると新アプリが旧スキーマで起動して
healthcheck が通らず自動ロールバックへ流れる。

### ロールバック

- `deploy.sh` 開始時に `git rev-parse HEAD` を `.deploy-prev` に記録
- layered recreate / global smoke のどこかで失敗 → `_deploy_lib.sh:rollback_to_previous`
- 処理: 直前 SHA の `compose/` を `git checkout` → `docker compose up -d
  --remove-orphans` → Broker への `record-deployment` は **打たない**
- matrix 整合は壊れない (前の本番バージョンがそのまま production に
  記録されたまま)

### テスト結果

```
bash tests/scripts/run.sh
  pre-deploy-verify tests  …  Passed: 5 / Failed: 0
  deploy tests             …  Passed: 6 / Failed: 0
  ===> All test files passed
```

11 ケースで以下を検証済:

- Broker 到達不能での fail-fast
- `pact-check.sh` 失敗伝搬
- `can-i-deploy` が 1 つでも NG で止まる
- `create-environment` 冪等性
- gate 失敗時のデプロイ拒否
- レイヤ順遵守
- healthcheck timeout 時の自動ロールバック + record 抑止
- 成功時の `record-deployment` 14 打刻
- `--dry-run` で `docker compose up -d` も `record-deployment` も呼ばれない
- `--skip-verify` で gate バイパス + 警告出力

## Consequences

### Pros

- **OSS リポに本番資格情報を持たない**。`.github/workflows/` 配下に
  PACT_BROKER_PASSWORD / DEPLOY_HOST / DEPLOY_SSH_KEY などを置く必要が
  消えた。fork 経由の secret 流出リスクが構造的にゼロ。
- **デプロイ発火源が一元化**。`scripts/deploy.sh production` の手動実行
  だけが production を変更できる。ADR とデプロイが必ずペアになる運用
  慣習を仕組みで強制できる。
- **Broker gate をローカルで高速に回せる**。PR 送信前に手元で
  `scripts/pre-deploy-verify.sh` を叩けば、Broker が見ている本番状態に
  対して現在の SHA が互換かを即時確認できる。
- **テストでデプロイパイプライン自体を回帰確認できる**。11 ケースの
  bash テストは bats-core 追加なしで動き、CI 不要でローカルで通る。
  ロールバック・レイヤ順などの非機能要件も回帰対象にできる。
- **Blue-Green / Argo Rollouts 不在の環境で、最低限の「落とさず入れる」
  を実現**。レイヤごとに healthcheck 待ちを挟むことで、境界を跨いだ
  契約不整合 (consumer が先に新バージョン、provider が旧バージョン) が
  数秒以上続かない。

### Cons / Tradeoffs

- **rolling recreate は真の zero-downtime ではない**。各サービスの
  recreate 中の数秒〜数十秒、当該サービスは 1 インスタンス不在になる。
  `alt-frontend-sv` や `nginx` が入れ替わる瞬間はリクエストが失敗しうる。
  許容できない層ができた時点で [[wowu/docker-rollout]] 相当の仕組みを
  別 ADR で検討する。
- **デプロイがホスト上の手動作業に依存**。リモートから叩けず、障害発生
  時に離席中だと対応が遅れる。ただし Alt はもともと single-machine
  architecture なので、この構造は変わらない。
- **ロールバックは compose/ 配下のみ**。アプリコンテナのイメージ tag が
  `:latest` のままだと、前 SHA の compose 定義で起動しても実体は新
  イメージになる。将来的に `image: ...@sha256:...` 形式の digest pin
  を compose に入れて完全な巻き戻し性を確保する余地がある (今回は
  スコープ外)。
- **14 pacticipants の can-i-deploy は直列で数十秒かかる**。Broker が
  `localhost` なので現実的だが、将来 Broker を外部公開する方針に
  変わった場合は並列化する必要がある。

## Verification

```bash
# Broker 稼働確認
docker compose -f compose/compose.yaml ps pact-broker   # → Up (healthy)

# テスト群 (zero external deps)
bash tests/scripts/run.sh                               # → All test files passed

# dry-run でパイプライン検証
DRY_RUN=1 scripts/deploy.sh production                  # docker compose up が呼ばれない
```

本番での初回実行手順は `docs/runbooks/deploy.md` 参照。
