# ADR-000888: Morning Letter の grounding を recap_outputs の structured columns に切り替え daemon の幻覚 fallback を解消する

## Status

Accepted

## Date

2026-05-04

## Affected Services

- **recap-worker (Rust)** — `recap-worker/recap-worker/src/store/dao/output.rs:212-238` の `get_genres_by_job` で SELECT 列に `title_ja`, `bullets_ja` を追加。`recap-worker/recap-worker/src/store/models.rs:18-23` の `GenreWithSummary` に `title_ja: Option<String>` と `bullets_ja: Option<serde_json::Value>` を追加。`recap-worker/recap-worker/src/pipeline/morning.rs:226-245` の filter を新ヘルパ `extract_title_and_bullets_from_genre` 経由に書き換え、`recap-worker/recap-worker/src/pipeline/morning.rs:554-597` に structured-first / legacy-fallback の解決順を実装。同ファイルの `#[cfg(test)] mod tests` に 5 本のユニットテストを追加し、structured 優先・object/string bullets 両対応・legacy JSON フォールバック・empty-array フォールバック・prose-only drop の各経路を pin。

## Context

[[000886]] で Morning Letter daemon を `MORNING_DAEMON_ENABLED=true` で恒久 ON 化し、[[000887]] で `MorningLetterSection.key` の `\-` escape を撤去して GBNF parser エラーを潰した。両 ADR の修正はいずれも commit + push 済みで、container も鮮度のあるイメージで動いていた:

- `recap-worker` は 30 分間隔で `running morning update job` → `loaded recap summaries for morning letter genre_count=24` → `morning letter generated and saved` を出していた。
- `news-creator-backend` (Ollama) のログから grammar parse error は完全に消失し、`/api/generate` は HTTP 200 を返していた。
- `recap-db.morning_letters` には当日 `target_date=2026-05-04` の row が daemon tick ごとに UPSERT されていた。

ところが production の row を直接覗くと依然劣化していた:

```
target_date | model           | is_degraded | degradation_reason
2026-05-04  | gemma4-e4b-12k  | t           | No recap summaries available; using overnight data only
```

`model` が `gemma4-e4b-12k` であることから LLM call 自体は成功している（[[000706]] の extractive fallback path には落ちていない）。しかし `degradation_reason` は `news-creator/app/news_creator/usecase/morning_letter_usecase.py:63-68` で `request.recap_summaries is None` のときだけ立つフラグで、**recap-worker から news-creator に渡る request payload で `recap_summaries` が `None` になっている** ことを意味する。実際に出力された letter の lead は `"TechFusion 社が AI スタートアップ Nova Labs を 12 億ドルで買収すると発表した"` という、実在しない企業による幻覚だった。LLM は overnight 5 件分の article stub だけを与えられ、recap context なしでそれらを無理に編集した結果である。

### Root cause

`recap-worker/recap-worker/src/pipeline/morning.rs:226-245` の filter は、各 genre に対して `extract_title_and_bullets(g.summary_ja.as_deref())` を呼び、`bullets.is_empty()` なら drop していた。`extract_title_and_bullets` は `summary_ja` を JSON として `serde_json::from_str` で parse し、失敗したら `(raw, vec![])` を返す（`recap-worker/recap-worker/src/pipeline/morning.rs:555-609`）。

`recap_outputs.summary_ja` の production data は **編集済みの日本語プロース** で、JSON 形式ではなかった:

```
著名研究者ヤン・ルカン氏らが主導する次世代 AI 研究では、現在の LLM や画像・動画モデルが
単に次の単語やピクセルを予測することに特化しているという限界が指摘されています [zenn.dev]。
ルカン氏は、AI が膨大なデータを用いて学習しても、人間が短時間で習得するような…
```

`serde_json::from_str` は当然 fail し、parser は **24 genre 全件で `bullets = []` を返した**。filter で全件 drop された結果 `inputs` が空になり、`RecapContext.summaries` は `None`、news-creator には `recap_summaries: null` が送られ、上記の幻覚 lead に至った。

一方で `recap_outputs` テーブル本体には structured 出力が並んでいた:

```sql
              Table "public.recap_outputs"
   Column    |  Type    | Nullable | ...
-------------+----------+----------+----
 title_ja    | text     | not null
 summary_ja  | text     | not null   -- 編集 prose
 bullets_ja  | jsonb    | not null   -- [{text, sources, source_sentence_ids}, ...]
 body_json   | jsonb    | not null
```

production の `bullets_ja` を 5 件サンプリングしたところ、すべて非空 (`length 2-3`)、`title_ja` も非空だった。要約の reduce 段が **structured と prose を別カラムに書き分けている** にもかかわらず、Morning Letter pipeline は **prose 側だけを取りに行って** いた。

`recap-worker/recap-worker/src/store/dao/output.rs:218` の SELECT は `genre AS genre_name, summary_ja` の 2 列のみで、`title_ja` / `bullets_ja` は取得すらされていなかった。生産者 (recap reduce 段) と消費者 (Morning Letter pipeline) の間で「summary_ja は JSON である」という silent contract が暗黙に書かれており、production data は最初からその契約を満たしていなかった。

### なぜ [[000886]] / [[000887]] でも検知されなかったか

[[000886]] / [[000887]] のテストは mocked / synthetic JSON-shaped data を入力していた。test-time の `GenreWithSummary { summary_ja: Some(r#"{"summary":{"title":"…","bullets":[…]}}"#.into()) }` は parser が通ってしまうので、本番の prose 入力との乖離はテストで観測できなかった。Pact / CDC レベルの contract test も recap-worker ↔ news-creator 間には張られていなかったため、「JSON である」という暗黙契約が production data 形式とズレていることが CI 経路で気付けなかった。

## Decision

DAO 層と pipeline 層の 2 commit でこの consumer 側 grounding バグを直す。producer 側の `summary_ja` 形式は変更しない（prose は表示用に保持し、structured 側は機械可読出力として既に書かれている責務分離を尊重する）。

### 1. `get_genres_by_job` に structured columns を含める

`recap-worker/recap-worker/src/store/dao/output.rs` の SELECT を

```sql
SELECT genre AS genre_name, summary_ja, title_ja, bullets_ja
FROM recap_outputs
WHERE job_id = $1
ORDER BY genre
```

に変更し、対応する `recap-worker/recap-worker/src/store/models.rs` の `GenreWithSummary` に `title_ja: Option<String>` と `bullets_ja: Option<serde_json::Value>` を追加する。Option で型付けすることで、列が NULL 化されるマイグレーション過渡期にも DAO 層の `try_get(...).ok()` だけで吸収できる。

#### 検討した代替案

- **`recap_outputs.summary_ja` を生産者側で JSON 化する**: 影響範囲が広く（recap-worker の reduce 段、news-creator のプロンプト contract、UI 側の prose 表示）、本件 root cause は consumer 側が用意済みの structured 列を使っていないことなので非対称な修正になる。やらない。
- **`body_json` を読む**: 形が genre ごとに揺れていて読み出し側の責務が増える。`bullets_ja` は配列形が固定で扱いやすい。
- **`summary_ja` の prose を heuristic で文 split して bullets に再構築する**: `[1]`/`[zenn.dev]` 等のシティテーション marker で文を切るアプローチ。再構築コストが高く LLM-grounding に値する精度が出ない。既に machine-readable な列があるのでそれを使うのが正解。

### 2. `extract_title_and_bullets_from_genre` を導入し structured-first にする

`recap-worker/recap-worker/src/pipeline/morning.rs` に新ヘルパを追加する。解決順は次の通り:

1. `g.bullets_ja` が `serde_json::Value::Array` で非空 → 各要素を `String` または `Object{text}` から trim して非空のみ採用。`title` は `g.title_ja.unwrap_or_default()`。
2. 上で空になる、または `bullets_ja` が `None` → 既存の `extract_title_and_bullets(g.summary_ja.as_deref())` を呼ぶ legacy path。`title` と `bullets` は同一ソースから取り、片方を別ソースから合成しない。
3. legacy path も空 (= prose-only) → `(raw_or_empty, vec![])` を返し、呼び出し側の `bullets.is_empty()` filter で genre が drop される。

filter 入口 (`pipeline/morning.rs:226-245`) は `extract_title_and_bullets(g.summary_ja.as_deref())` から `extract_title_and_bullets_from_genre(g)` に差し替える。既存の `extract_title_and_bullets` は legacy path として残す（フォールバック専用に doc を更新）。

#### 検討した代替案

- **structured columns に存在を assert して fail-loud にする**: `bullets_ja` がない genre を見たらエラーで止める案。production は既に structured を書いているが、過去 row の互換性が壊れる。filter で drop するソフトな運用 ([[000706]] の degraded mode 思想と整合) を維持する。
- **title だけ structured、bullets だけ legacy のような mix**: 「title が advertise している内容」と「bullets の中身」が違う出典になる事故が起きる。title と bullets は必ず同一ソースから取る方針を pin。

### 3. 5 本のユニットテストで contract を pin する

`recap-worker/recap-worker/src/pipeline/morning.rs` の `#[cfg(test)] mod tests` に追加:

- `from_genre_prefers_structured_bullets_ja_over_prose_summary` — 本件本番症状の再現。structured が非空なら prose は無視。
- `from_genre_handles_string_bullets_in_jsonb_array` — `["A", "B"]` 形と `[{text: "A"}]` 形の両対応。
- `from_genre_falls_back_to_legacy_summary_ja_when_structured_missing` — 旧 row 互換性。
- `from_genre_falls_back_when_bullets_ja_is_empty_array` — 空 structured + 旧 JSON が両方ある場合の優先順。
- `from_genre_returns_empty_when_only_prose_summary_ja_is_present` — production failure mode の inverse pin。prose-only は drop されること。

合計 19 本の `pipeline::morning::tests` が green、lib 全体で 344 tests pass、`cargo clippy --all-targets -- -D warnings` も clean。

#### 検討した代替案

- **本物の Postgres を立てた integration test を必須にする**: schema drift には強いが、CI で常時起動するコストが既存方針と合わない。本件は「DAO の SELECT 列追加 + parser refactor」で構造的に表面化しやすい変更なので、unit test で contract を固めれば再発時に CI が落ちる。

## Consequences

### Pros

- Morning Letter daemon が次の 30 分 tick で `is_degraded=false` の正版を `(target_date, edition_timezone)` UPSERT で書き直す。`morning_letters.5/4` row は手動マイグレーションなしに自然回復する（[[000886]] と同じ projection 自然修復モデル）。
- LLM 出力の lead が overnight stub だけで生成された幻覚から、当日の recap genre context (24 genres × 2-3 bullets) に grounding した内容に変わる。架空の企業名や架空の数値が消える。
- `recap-worker ↔ news-creator` の prompt 入力 contract が、production data の真の structured 出力を反映するようになる。silent contract（`summary_ja` が JSON）から explicit contract（`bullets_ja` が array）への置き換え。
- 19 本の morning pipeline test のうち 5 本が新規 contract を直接 pin するので、将来 `recap_outputs` の structured 列を取り壊す変更や、`extract_title_and_bullets_from_genre` を refactor で潰してしまう変更が CI で落ちる。
- DAO の SELECT 拡張は単純加算で、`recap_outputs` テーブル schema や API の wire format には変更がないため Atlas migration / Pact 影響なし。

### Cons / Tradeoffs

- 過去の `morning_letters` row（4/14〜5/3 の欠損期間 + 5/4 朝の幻覚 row）は backfill しない。daemon は当日 `target_date` のみ書く設計のため。UI は `ORDER BY target_date DESC` で常に最新を取り、4/13 / 5/4 の旧 row は将来の retention cleanup ADR で消す。
- `bullets_ja` の Object 要素から `text` 以外の field（`sources`, `source_sentence_ids` 等）は今回 drop している。Morning Letter のセクション粒度では出典別の URL 列挙は不要だが、将来 [[000707]] が要求する ArticleDeck 導線のために sources を読み出したくなったら、本ヘルパの戻り値型を拡張する必要がある。本件は scope 外。
- 生産者側 `recap_outputs.summary_ja` が prose のままなので、別の consumer が同じ silent contract に引っかかる余地は残る。`api/fetch.rs` の RecapGenreResponse は既に `normalize_summary_text(genre.summary_ja)` で prose をそのまま返しており健全だが、新規 consumer が出てきたら同じ罠を踏みうる。後続 ADR で「`recap_outputs` の consumer は structured 列を読む」を runbook 化する余地あり。
- `extract_title_and_bullets` (legacy path) は今回も削除しなかった。production にもう旧 JSON 形式の row は残っていないと推定されるが、観測コストの代わりに kill 判断を保留した。1 リリース cycle 様子見の上で次 ADR で削除候補。
