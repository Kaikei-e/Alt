# ADR-000433: Knowledge Home backfill の ArticleCreated dedupe を常設 write path と統一する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — `knowledge_backfill_job`, `knowledge_events`, `CreateArticleInternal`, `SaveArticle`

## Context

Knowledge Home の Phase 1 では、新着記事を backfill に依存せず `ArticleCreated` の常設 append 経路で Home へ流し込むことを優先した。実装上は `CreateArticleInternal` と `SaveArticle` がどちらも `article-created:{article_id}` を dedupe key に使って `knowledge_events` へ append している。

一方、backfill job が生成する synthetic `ArticleCreated` は `backfill:{article_id}` を使っていた。この状態では、同じ article に対して以下の二重経路が両立してしまう。

1. 通常 write path で `article-created:{article_id}` が append される
2. 後から backfill が `backfill:{article_id}` を append する

event store から見れば別イベントとして扱われるため、`knowledge_home_items` の upsert が最終的に 1 行へ収束しても、`today_digest_view` のような加算系 projection は二重加算される。Phase 1 で backfill を「補助用途」に格下げするなら、重複記事に対して event 自体が増えないことを先に保証する必要があった。

## Decision

Knowledge Home の backfill が生成する `ArticleCreated` は、**常設 write path と同じ dedupe key を使う**。

具体的には以下を採用する。

- backfill の synthetic event も `article-created:{article_id}` を dedupe key とする
- event type は引き続き `ArticleCreated`、aggregate は `article` のまま維持する
- actor は `backfill` のまま残し、「誰が append を試みたか」は追えるようにする
- backfill の責務は「新着流入の代替」ではなく「初期投入・欠損復旧」に限定する

この決定により、通常 write path で既に append 済みの記事を backfill しても、`knowledge_events` の `ON CONFLICT (dedupe_key) DO NOTHING` が効き、projection 側に二重 event が流れなくなる。

代替案として、backfill 専用の dedupe key を維持したまま projector や digest 更新側で重複判定する案もあった。しかし projection ごとに重複吸収ロジックを持つと責務が分散し、加算系 read model の追加時に同じ事故を繰り返すため採用しない。

## Consequences

### Pros

- backfill を実行しても、常設経路で処理済みの article に対する二重 projection を防げる
- `knowledge_home_items` だけでなく `today_digest_view` のような加算型 read model も守れる
- dedupe policy が write path と recovery path で一致し、運用判断が単純になる
- event store レベルで重複を止めるため、新しい projection を増やしても同じ方針を再利用できる

### Cons / Tradeoffs

- backfill 実行数と実際に append された event 数が一致しないため、運用者は「処理した件数」と「新規 event 件数」を分けて見る必要がある
- backfill による再送を別イベントとして監査したい場合は、dedupe key 以外の監査手段が必要になる
- 将来「同じ article に対して別の再構築イベントを明示的に流したい」要件が出た場合は、`ArticleCreated` とは別の event type を設計する必要がある
