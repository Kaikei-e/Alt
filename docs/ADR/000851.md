# ADR-000851: Knowledge Loop の Surface Planner v2 と change_summary redline diff の核ロジックを reproject-safe な pure function として導入する

## Status

Accepted

## Date

2026-04-26

## Affected Services

- knowledge-sovereign (Go): `decide_bucket_v2.go` / `diff_calculator.go` / `change_summary_builder.go` 新設、projector の `EventHomeItemSuperseded` / `EventSummarySuperseded` 経路で `buildChangeSummaryJSON` を呼出し、`slog.DebugContext` で書込みを観測可能化
- knowledge-sovereign migrations: `00013_add_surface_planner_v2_columns.sql` で `surface_planner_version` SMALLINT (DEFAULT 1, CHECK 1|2) と `surface_score_inputs` JSONB を additive 追加、`why_kind` enum に v3 値 (`topic_affinity_why` / `tag_trending_why` / `unfinished_continue_why`) を追加。`atlas.sum` 再ハッシュ
- alt-backend (Go): `decodeChangeSummary` を additive 拡張し、JSONB から redline fields (`added_phrases` / `removed_phrases` / `unchanged_phrases_count` / `added_tags` / `removed_tags`) を proto に詰めて返却
- canonical contract `docs/plan/knowledge-loop-canonical-contract.md` の §6.4.1 として upstream snapshot events (`RecapTopicSnapshotted` / `AugurConversationLinked` / `KnowledgeLoopSurfacePlanRecomputed`) の system-only 仕様を documented

## Context

[[000850]] (Wave 1) で proto contract / canonical doc / Atlas runbook の足場と、UI 側の paper-stack depth + ChangedDiffCard redline 描画を additive に landed させた。Wave 1 の時点では bucket 配置を実際に動かす projector ロジックと、change_summary を埋める projector パスが未着手のままだったため、外部レビュー (`fb.md`) が指摘した次の 2 点は **構造としては解けたが実体が伴わない** 状態だった:

1. **Surface Planner**: 現状 projector は `event-type → bucket` の固定マッピングで、ユーザの recent activity / topic affinity / unfinished thread / version drift を一切参照していない
2. **ChangedDiff**: UI は redline-proof を描画できるが、projector が `change_summary.added_phrases[]` 等を埋めていないため、常に Then/Now fallback で描画される

この 2 点を解くにあたり、いきなり projector の write path を全面差し替えると次の不変条件 ([[000831]] §3 / [[knowledge-home-immutable-model]]) を踏みかねない:

- reproject-safe: projector が cross-table の latest state を覗くと、replay 時に挙動がブレる
- event-time purity: `time.Now()` / `NOW()` を business fact に登場させない
- versioned artifact 経由: tag / summary は `*_version_id` でのみ参照
- WhyMappingVersion bump は full reproject runbook 経由

そこで Wave 2 では「核ロジックを pure function として独立させ、unit test で固定する」段階を切り出した。pure function 化により:

- upstream event (recap-worker / augur) が発火していなくても、bucket 配置と change_summary diff の正しさをテーブル駆動 + determinism guard で固定できる
- `projectEvent` に組込む際は依存注入だけで済み、副作用面は最小化される
- フィードバック ([[000843]] / [[000846]]) で踏んだ「projector の write path 改修で副作用が広がる」リスクを回避できる

加えて、Wave 1 で proto に landed した redline fields を実際に書込めるよう、`SummarySuperseded` 系イベントを projector が拾った瞬間に payload-resident な excerpts から `change_summary` JSONB を構築する経路も同セッションで wired した。これにより FE は projector の実装と同期して、redline 描画と Then/Now fallback を payload の有無で自然に切替えられる。

## Decision

Knowledge Loop の Surface Planner v2 と change_summary redline diff の核ロジックを knowledge-sovereign 内に pure function として導入し、Atlas migration で v2 schema の足場を additive に landed させる。projector の write path への組込みは「呼び出し可能・テスト可能・観測可能」までを Wave 2 のスコープとし、cross-service の port + gateway + driver による versioned-table fetch は [[000852]] (Wave 3) に切り分ける。

### 1. Atlas migration `00013_add_surface_planner_v2_columns.sql`

`knowledge_loop_entries` を additive に拡張する:

- `surface_planner_version SMALLINT NOT NULL DEFAULT 1` + `CHECK (surface_planner_version IN (1, 2))`
- `surface_score_inputs JSONB`: reproject-safe な evidence のみ格納する。`COMMENT ON COLUMN` で「decay/score 値は格納しない」を明文化
- `why_kind` enum に `topic_affinity_why` / `tag_trending_why` / `unfinished_continue_why` を `ALTER TYPE ... ADD VALUE IF NOT EXISTS` で追加 (各 ALTER は単独 statement)

`atlas migrate hash` で `atlas.sum` を再生成し、`atlas migrate lint --latest 1` で 0 diagnostics を確認した。RLS policy は本 ADR では追加しない (read-side caller の coordinated rollout が必要なため Wave 3 へ送る)。

### 2. `decideBucketV2` pure function (`decide_bucket_v2.go`)

knowledge-state-based bucket scoring を、入力 struct → bucket enum の純関数として定義する:

```go
type SurfaceScoreInputs struct {
  TopicOverlapCount, TagOverlapCount, VersionDriftCount uint32
  HasAugurLink, HasOpenInteraction bool
  FreshnessAt time.Time
  EventType   string
}
func decideBucketV2(in SurfaceScoreInputs) sovereignv1.SurfaceBucket
```

優先順位 (canonical contract §6 を踏襲): **Changed > Continue > Now > Review > v1 fallback**

- 「version drift が 1 件でもあれば Changed」: 既知の article に対する mental model 更新が最強の signal だと判定
- 「Augur link / open interaction があれば Continue」: 進行中スレッドが fresh material を上回る
- 「topic / tag overlap があれば Now」: 1 つの overlap でも、その article は今ユーザが考えている文脈と接続する
- v2 evidence が一切無いときは v1 mapping (event-type → bucket) に静かに fallback。これにより `surface_planner_version` を per-entry に切替える segmental rollout が可能

table-driven test 17 例 + 1000-iter determinism guard で固定。`time.Now()` / 副作用は登場しない。

### 3. `computeChangeDiff` pure function (`diff_calculator.go`)

sentence-level の集合差分 + tag の case-insensitive 集合差分を、純関数として定義する:

- sentence 区切りは正規表現 `(?m)([.!?])\s+|\n+` (sentence-terminator + whitespace, または改行)。空白の正規化と空文字スキップを伴う
- tag は trim + lowercase で normalize したキー比較。元の casing は新側を保持
- `dedupePreserveOrder` で出力順を入力順に揃え、redline の上から下への自然な読みを担保
- 11 unit tests + determinism guard

[[000846]] (SummaryNarrativeBackfilled) で踏んだ「あとから why を改善するパスは projection の他フィールドを破壊しない」設計と同じ筋に揃え、副作用ゼロ・入力専用の関数として切り出した。

### 4. `buildChangeSummaryJSON` builder + projector wiring (`change_summary_builder.go`)

event payload からシノニムキー (`previous_summary_excerpt` / `old_summary_excerpt` / `previous_summary_text` などをまとめて読む) を吸収する parser と、redline 計算の有無を判定する builder を切り出した:

- `payload.canRedline()` が true (両方の excerpt が揃う、または tag 配列の片側でも非空) のときだけ `computeChangeDiff` を呼ぶ
- それ以外は legacy `summary` + `changed_fields[]` のみの JSONB を返す
- payload が完全に空のときは `nil` を返し、projector は `entry.ChangeSummary` を設定しない

projector の `EventHomeItemSuperseded` / `EventSummarySuperseded` 共通分岐に組込み、`slog.DebugContext` で `event_type` / `event_seq` / `entry_key` / `redline_capable` / `change_summary_bytes` を構造化ログに残す (Phase D の partial delivery: prometheus/client_golang への依存を増やさず、観測点を増やすコスト最小の方法を採った)。5 unit tests で empty / legacy-only / redline / alternate-keys / malformed payload を固定。

### 5. alt-backend `decodeChangeSummary` の additive 拡張

JSONB → proto `loopv1.ChangeSummary` のデコーダに、新 fields を読む生 struct と marshalling を追加した。既存 fields のマッピングは無変更で、handler test は緑のまま。FE 側は Wave 1 で既に redline 描画 ready なので、projector が JSONB を埋めた瞬間から E2E で redline が出る。

### 6. canonical contract §6.4.1 — Upstream snapshot events

projector が `tag_set_versions` / `recap_topic_snapshots` / `augur_conversations` を直接 query して latest state を覗くと reproject-safe を破る。代わりに上流サービス (recap-worker / augur / knowledge-sovereign 内部) が過去分詞 event を発火し、payload に versioned id と immutable な評価値を凍結する契約を documented した:

- `RecapTopicSnapshotted` (recap-worker): `top_terms[]` + `recap_topic_snapshot_id`
- `AugurConversationLinked` (augur): `(conversation_id, article_id, linked_at)`
- `KnowledgeLoopSurfacePlanRecomputed` (knowledge-sovereign internal): プランナ世代 bump 時の `EntryScoreInput[]`

すべて system-only。BFF は user transition path で `KnowledgeLoopSurfacePlanRecomputedPayload` を reject する (default-deny)。

### 検討した代替案と却下理由

- **`decideBucketV2` を最初から `projectEvent` に組込む**: upstream event がまだ無いので組込んでも常に v1 fallback を返すだけになり、Wave 2 で何が動いたかが見えなくなる。pure function に切ってテストで動作を固定する方が docs としても強い
- **`computeChangeDiff` を word-level diff にする**: stylistic edit (語順入替, 同義語) が大量の noise を発生させ、redline の信号 / ノイズ比が悪化する。sentence-level は実用上ちょうど良い粒度で、ChangedDiffCard 側の表現 (取り消し線 + 下線) ともマッチする
- **Atlas migration で RLS policy を Wave 2 に同梱**: pgbouncer 側の session-var 配線と全 read caller の coordinated rollout が必要。同セッションで全部やると blast radius が広がる。port-level user_id binding (Wave 3) を primary defense にし、RLS は次の deepening layer として切り出す
- **prometheus/client_golang を Wave 2 で導入**: `/metrics` endpoint の追加 / Grafana dashboard / alert rule のセットがすべて新規。Wave 2 のスコープ (ロジックを pure に切ってテストする) から外れる。`slog.DebugContext` を観測点として打っておけば log-based metrics で十分代替可能で、prometheus 化は Wave 3 で独立 commit に切り出す
- **WhyMappingVersion を Wave 2 のうちに v2 → v3 に bump**: projector が新 WhyKind 値を実際に emit するのは Wave 3 (recap_topic_snapshots を fetch して `topic_affinity_why` を組み立てる) の段階。先に bump すると過去 entry の why が中途半端な mapping を踏む。runbook 履歴の v3 行は `tbd` で予約 ([[000850]]) し、bump 自体は Wave 3 で実施

## Consequences

### Pros

- bucket 配置と change_summary diff の正しさがテーブル駆動 + determinism guard で凍結された。Wave 3 で projector に組込むときの回帰検出は unit test の単純な再実行で済む
- Atlas migration が landed したことで、Wave 3 で projector が `surface_planner_version=2` を書き始めるときの schema 衝突がない。`CHECK (... IN (1, 2))` により未知の値が混入したらすぐに検出される
- `change_summary` JSONB が payload-resident な excerpts でだけ redline を埋めるため、projector の振る舞いは upstream emitter の進捗に応じて自然にグラデーションする (FE は redline と Then/Now を両方扱える)
- `slog.DebugContext` を打ち込んだことで、log-based metrics 経由で「change_summary が何件 redline-capable で書かれたか」を観測できるようになった。prometheus への切替えは破壊なしで後追い可能
- canonical contract §6.4.1 で upstream event の責務が明文化され、recap-worker / augur 側の実装が「契約に沿うかどうか」で評価できる

### Cons / Tradeoffs

- `decideBucketV2` はまだ projector から呼ばれていない。Wave 3 で `projectEvent` に組込むまで、`surface_planner_version` は実運用では常に 1 のまま。Atlas migration の DEFAULT 1 とは整合するが、v2 列が「実体のない準備済み枠」である状態が一定期間続く
- `computeChangeDiff` は payload に excerpts が含まれている前提で動く。現状の SummarySuperseded emitter は excerpt を含めていない可能性があり、その場合は redline は出ない (legacy `summary` + `changed_fields` 経路で描画される)。emitter 側を Wave 3 で確認 / 拡張する必要がある
- WhyMappingVersion / SurfacePlannerVersion の runbook 履歴行はまだ `tbd` プレースホルダ。Wave 3 で bump 実施時に日付と ADR 番号を埋め直す運用負債が残る
- `prometheus/client_golang` 依存追加と RLS policy 適用は Wave 3 へ持ち越し。それまで F-001 (cross-user evidence_refs leak) の defense は port-level user_id binding が一義的に頼り (Wave 3 の必達タスク)
- `slog.DebugContext` は `LogLevel >= debug` 設定時にしか出ない。本番が info レベル運用の場合、観測点としては効かない可能性があり、Wave 3 で metric counter に置換える前提
