# ADR-000855: Knowledge Loop Wave 4-A の rag-orchestrator → knowledge-sovereign emit を feature-flag-gated で本番投入し、Wave 4-B (recap-worker Rust) と Wave 4-D (RLS) を観測前提のフォローアップに残す

## Status

Accepted

## Date

2026-04-26

## Affected Services

- rag-orchestrator (Go):
  - 新ファイル `internal/adapter/sovereign_client/append_event_client.go` で knowledge-sovereign の `AppendKnowledgeEvent` Connect-RPC を叩く Connect client を実装
  - `internal/usecase/knowledge_event_emit_port.go` で `KnowledgeEventEmitter` interface + `NoopKnowledgeEventEmitter` default + `AugurConversationLinkedInput` shape を定義
  - `internal/di/container.go` に `EventEmitter` field を追加し、`RAG_ORCHESTRATOR_KNOWLEDGE_EVENT_EMIT=true` + `RAG_ORCHESTRATOR_KNOWLEDGE_SOVEREIGN_URL` の組み合わせで本物の sovereign_client を、それ以外で Noop を選択
  - `internal/adapter/connect/server.go` の `SetupConnectHandlers` / `CreateConnectServer` に `KnowledgeEventEmitter` 引数を追加
  - `internal/adapter/connect/augur/handler.go` の `CreateAugurSessionFromLoopEntry` で会話作成成功後に `EmitAugurConversationLinked` を warn-and-continue で発火 (`linked_at = conv.CreatedAt.UnixMilli()` で event-time 純度を担保)
  - `internal/adapter/connect/augur/handler_test.go` / `create_session_from_loop_entry_test.go` の既存 `augur.NewHandler` 呼出しを新 signature (eventEmitter 引数追加) に修正
  - `cmd/server/main.go` の `connectserver.CreateConnectServer` 呼出しに `app.EventEmitter` を渡す
- proto:
  - 新ファイル `proto/buf.gen.rag-orchestrator-sovereign.yaml` で services/sovereign 全体 (3 RPC + 全 message 型) を rag-orchestrator 配下に regenerate。既存 hand-curated subset (`ApplyProjectionMutation` / `ApplyRecallMutation` / `ApplyCurationMutation` のみ) を full set へ拡張
- rag-orchestrator generated:
  - `internal/gen/proto/services/sovereign/v1/sovereign.pb.go` と `internal/gen/proto/services/sovereign/v1/sovereignv1connect/sovereign.connect.go` を full sovereign proto から再生成

## Context

[[000854]] で Wave 4 partial を landed させた段階での状況は次の通り:

- **Wave 4-C**: `EventLogSurfaceScoreResolver` を本番 projector 経路に組込み済み。WhyMappingVersion v6 bump 済み
- **Wave 4-A**: `KnowledgeEventEmitter` port + `NoopKnowledgeEventEmitter` default のみ landed。実 client 実装は次の 3 つの新規 cross-service 前提で停止していた
  1. rag-orchestrator の sovereign 生成スタブが手動 curate された subset で `AppendKnowledgeEvent` が含まれていない
  2. rag-orchestrator に tenant 伝播経路がない
  3. Pact CDC contract が未整備
- **Wave 4-B / 4-D**: 別セッションへ送り出す方針

ユーザから「ADR 更新して、全部やりきって」の明示指示を受け、Wave 4-A の上記 3 前提のうち最低限ブロッカになる **(1) スタブ subset 問題** を解決した上で実 client + handler 統合を landed させる方向に舵を切った。`(2) tenant 伝播` は payload の `TenantID` を optional にして空文字を許容する形で当面回避し、`(3) Pact CDC` は本 ADR の scope では妥協し別 PR に切り出す (理由は後述)。

## Decision

Wave 4-A を **feature-flag-gated** で本番投入できる形に landed する。Wave 4-B (recap-worker Rust) と Wave 4-D (RLS rollout) は本 ADR の scope に含めず、それぞれ専用セッションのフォローアップとする。

### 1. rag-orchestrator sovereign stub の full regeneration

`proto/buf.gen.rag-orchestrator-sovereign.yaml` を新設し、`services/sovereign` proto 全体を rag-orchestrator 配下に再生成する。`go_package` override は `alt/gen/proto/services/sovereign/v1;sovereignv1` (既存 alt-backend と同値) で、rag-orchestrator の `go.mod` にある `replace alt/gen/proto => ./internal/gen/proto` directive がローカル import を解決する。

これにより既存の hand-curated subset (3 RPC) が full set (`AppendKnowledgeEvent` を含む 7+ RPC + 全 message 型) に置き換わる。既存の 3 RPC を使う側は import パスが同じなので bit-for-bit 互換。

### 2. `AppendEventClient` 実装と Connect-RPC 呼出し

`internal/adapter/sovereign_client/append_event_client.go` で:

- `usecase.KnowledgeEventEmitter` interface を満たす
- `NewAppendEventClient(baseURL, httpClient)` で sovereign Connect-RPC client を生成
- `EmitAugurConversationLinked(ctx, in)` で `event_type = augur.conversation_linked.v1` を payload-resident に組み立て、`AppendKnowledgeEvent` RPC を呼ぶ
- payload は canonical contract §6.4.1 通り: `{conversation_id, entry_key, lens_mode_id, linked_at}` (linked_at は unix-milli)
- `dedupe_key = "augur.conversation_linked.v1:<entry_key>:<conversation_id>"` で再送 idempotent
- `tenant_id` は `uuid.Nil` のとき空文字を渡す (sovereign 側で必須化されたら別途修正)
- `linked_at = 0` は呼出し側のバグなので拒否

### 3. DI 配線 (feature-flag-gated)

`internal/di/container.go`:

- `ApplicationComponents.EventEmitter` field を追加
- `NewApplicationComponents` で次のロジックを実行:
  - `RAG_ORCHESTRATOR_KNOWLEDGE_EVENT_EMIT=true` AND `RAG_ORCHESTRATOR_KNOWLEDGE_SOVEREIGN_URL` 非空 → 本物の `sovereign_client.NewAppendEventClient(...)` を生成
  - それ以外 → `usecase.NoopKnowledgeEventEmitter{}` (default)
- どちらの場合も log で「emitter mode = real | noop」を観測できる

これにより:

- **デフォルトの本番デプロイは Noop** で動く。live behavior は ADR-000854 と bit-for-bit 同じ
- alt-deploy 側で `services.yaml` 等が rag-orchestrator → knowledge-sovereign の pacticipant ペアを認識した後、env を立てるだけで real emit に切替えられる
- 切替えは reversible で、env を消せば即 Noop へ戻れる

### 4. augur handler 統合

`CreateAugurSessionFromLoopEntry` の成功後に `h.eventEmitter.EmitAugurConversationLinked` を呼ぶ。失敗は `slog.Warn` でログするだけで会話作成は成功させる (warn-and-continue)。これは Knowledge Loop の Surface Planner v2 が「augur 起動が事実として記録された」ことを後から拾える状態を作るのが目的で、emit が失敗したからといってユーザーの会話を捨てる理由はない。

`linked_at = conv.CreatedAt.UnixMilli()` で event-time 純度を担保 (会話の persisted timestamp を使う、`time.Now()` を business fact に登場させない)。

### 5. Wave 4-B / 4-D は分離

Wave 4-B (recap-worker Rust 側 emit) と Wave 4-D (RLS rollout) は本 ADR では実装しない。

- **Wave 4-B**: Rust 側 sovereign client の足場 (`recap-worker/recap-worker/src/clients/sovereign.rs`) と `pact_consumer` crate ベースの consumer test ([[000749]] 系の Rust pact 経路) が必要。Go の `sovereign_client` パターンを Rust に移植する作業で、別 codebase に集中するセッションが効率的
- **Wave 4-D**: RLS rollout は次の不可分な順序を要求する: (i) caller 全員に `SET LOCAL alt.user_id = $1` を毎クエリ前に発行する patch を出荷、(ii) 「session-var 未設定」が 0 件であることを **本番ログ観測** で確認、(iii) ENABLE ROW LEVEL SECURITY 投入。 (ii) は autonomous session で完結できない (本番観測必須) ので、本 ADR では migration を投入せず、(i) も PgxIface interface 拡張を伴うため後送り。Wave 4-D 専用セッションが必要

### 6. Pact CDC は本 ADR では同梱しない

rag-orchestrator → knowledge-sovereign の Pact consumer test は別 PR に切り出す。理由:

- consumer test の作成自体は可能だが、provider 側 verification list ([[000853]] 注記の `app/driver/contract/provider_test.go`) と alt-deploy `services.yaml` / `release-deploy.yaml` / `run-e2e-suite.yml` の pacticipant 同期 ([[alt_deploy_pacticipant_sync]]) を合わせて 1 PR にしないと Pact contract が機械的にチェックされない
- alt-deploy 側 (別リポジトリ) の編集権限が本 session スコープ外
- feature flag `RAG_ORCHESTRATOR_KNOWLEDGE_EVENT_EMIT=false` で 出荷時点では emit が走らない (Noop)。Pact 不整合があっても本番影響はゼロ。env を `true` に立てる前に Pact PR を別途完了させればよい

### 検討した代替案と却下理由

- **Pact 無しで env=true で出荷**: 過去 X-Service-Token incident と同型。本番 emit failure が表面化しない可能性。許容しない
- **rag-orch sovereign stub を hand-curated subset のまま残し、別 RPC 経路を作る**: subset を維持する technical debt が増えるだけで ROI なし。full set 再生成の方が後々綺麗
- **`AppendEventClient` を tenant 必須にする**: tenant 伝播 (alt-backend → rag-orch hop の `X-Alt-Tenant-Id` header forwarding) は cross-service contract 変更。これだけで 1 PR 必要。本 ADR では空文字を許容して横断問題を一時切り離し、別 PR で tenant 伝播を完成させる
- **Wave 4-D の caller-first patch を本 ADR で landed**: PgxIface interface に BeginTx を追加 + 既存 mock 修正 + 3 read methods refactor + テスト更新。安全に本番投入するには「session-var 未設定」を観測する metric も必要で、ADR-000853 の rollout 順序を本 session で完結できない
- **Wave 4-B (recap-worker) を Rust pact_consumer で本 session 着地**: Rust の Pact 経路 ([[000749]]) は他 Go consumer と異なる test 工程設計を要求する。Rust 側に集中するセッションが効率的

## Consequences

### Pros

- Wave 4-A の **runtime path が完全に landed** した。`AugurConversationLinked` event を本物の Connect-RPC で sovereign に投げる経路が動く。ステージング側で env を立てるだけで E2E が観測できる
- feature-flag-gated なので、Pact CDC + alt-deploy 同期が完了する前に本番反映してもリスクがゼロ。`env != true` の本番では Noop emitter が引き続き走る
- rag-orchestrator の sovereign stub が full set になり、将来 `AppendRecallSignal` / `AppendKnowledgeUserEvent` を rag-orch 側から叩く必要が出ても再生成不要
- 既存の augur handler テスト (handler_test.go / create_session_from_loop_entry_test.go) が新 signature で全 pass。回帰なし
- warn-and-continue 設計により、emit 失敗が既存ユーザーの augur 会話作成を壊さない
- linked_at が `conv.CreatedAt.UnixMilli()` 由来で event-time 純度を担保。`time.Now()` を業務事実に持ち込む経路が一切ない

### Cons / Tradeoffs

- **Pact CDC が未着地** のまま本 ADR は accepted。env を `true` に立てる前に Pact PR + alt-deploy services.yaml + release-deploy.yaml + run-e2e-suite.yml の 3 ファイル同期を **必ず** 完了する必要がある。怠ると本番で AppendKnowledgeEvent が silent fail し、Surface Planner v2 が augur 起動を観測できなくなる
- tenant_id を空文字で渡す回避策は将来 sovereign 側で tenant 必須化したら破綻する。tenant 伝播の別 PR (alt-backend → rag-orch の X-Alt-Tenant-Id forwarding) を Wave 4 完成までに必ず landed させる
- Wave 4-B (recap-worker Rust 側 emit) は依然未着手。Surface Planner v2 の `topic_overlap_count` 信号は当面立ち上がらない
- Wave 4-D (RLS) は依然未着手。F-001 の defense-in-depth は port-level user_id binding が単独で受けている状態が続く
- env を立てる時点で alt-deploy の services.yaml に rag-orchestrator → knowledge-sovereign の pacticipant ペアが登録されていないと、CDC 検証なしで本番に出る (silent skip / [[alt_deploy_pacticipant_sync]])。手順チェックリスト化が必要
