# ADR-382: Inoreader記事のコアArticleバックフィル

## STATUS

Accepted

## CONTEXT

alt-dbに`feeds`レコードが存在するにもかかわらず、対応する`articles`が0件のフィードが多数存在していた。一方、Inoreaderから取り込んだ`inoreader_articles`テーブルにはそれらのフィードに紐づく記事データが蓄積されていた。

既存の`ArticleSyncService.SyncArticles()`は直近24時間の`inoreader_articles`のみを対象としており、以下のケースをカバーできていなかった:

- フィードが後からalt-dbに追加されたケース（Inoreader記事が先に取り込まれていた）
- 24時間より前に取り込まれた過去記事

さらに、2つの技術的課題が存在した:

1. **URL解決の不整合**: 既存の`GetFeedID`は`feeds.link`（ウェブサイトURL）でマッチするが、Inoreader側の`feed_url`はRSS URL。直接マッチでは解決できない
2. **品質劣化リスク**: 既存の`UpsertArticlesBatch`は`ON CONFLICT (url) DO UPDATE`で既存記事を無条件上書きするため、フルテキスト記事がInoreaderのRSS要約（短い）で上書きされる可能性があった

## DECISION MAKING

### バックフィルパイプラインの追加

pre-processorの`ArticleSyncService`に`BackfillEmptyFeeds`メソッドを追加し、起動時即実行+毎時実行のジョブとして組み込んだ。

### URL解決: `feed_links`経由のJOIN

`inoreader_subscriptions.feed_url` → `feed_links.url` → `feeds.feed_link_id` の経路でJOINし、`feed_id`をSQL内で直接解決する設計とした。アプリケーション側での再解決が不要になり、`GetFeedID`の`feeds.link`マッチ問題を完全に回避できる。

### ON CONFLICT DO NOTHING による品質保護

バックフィル専用の`InsertArticlesBatchNoConflict`関数を新設し、`ON CONFLICT (url) DO NOTHING`で既存記事をスキップする。Inoreader RSS要約によるフルテキスト記事の上書きを防止した。

### Legacy/API両モード対応

pre-processorはsingle-DB（Legacy）とsplit-DB（API経由）の2モードで動作する:

- **Legacyモード**: `GetInoreaderArticlesForEmptyFeeds`で`NOT EXISTS`フィルタ付きの4テーブルJOINクエリを実行。`InsertArticlesBatchNoConflict`でバッチ挿入
- **APIモード**: `GetInoreaderArticlesForBackfill`でInoreaderテーブルのみクエリし、`feedURL→feedID`をbackend API経由で解決。`CheckExists`で存在確認後に`Create`を呼ぶ（DO NOTHING相当）

## RESULTS, EFFECTS

### 実装の概要

**変更ファイル一覧:**

| レイヤー | ファイル | 変更内容 |
|---------|---------|---------|
| Driver | `driver/db_articles.go` | `GetInoreaderArticlesForEmptyFeeds`, `GetInoreaderArticlesForBackfill`, `InsertArticlesBatchNoConflict` 追加 |
| Driver (API) | `driver/backend_api/article_driver.go` | `FetchInoreaderArticlesForEmptyFeeds`, `UpsertArticlesWithFeedID`（存在チェック付き）追加 |
| Repository | `repository/interfaces.go` | `FetchInoreaderArticlesForEmptyFeeds`, `UpsertArticlesWithFeedID` 追加 |
| Repository | `repository/article_repository.go` | Legacy実装（`InsertArticlesBatchNoConflict`使用） |
| Service | `service/interfaces.go` | `BackfillEmptyFeeds` 追加 |
| Service | `service/article_sync_service.go` | バックフィルロジック実装（サニタイズ・バリデーション付き） |
| Handler | `handler/interfaces.go`, `handler/job_handler.go` | `StartBackfillJob`（毎時、起動時即実行） |
| Bootstrap | `bootstrap/lifecycle.go` | ジョブ起動登録 |

**処理フロー:**

1. システムユーザーID取得（キャッシュ済みなら再利用）
2. `FetchInoreaderArticlesForEmptyFeeds(ctx, cursor, 1000)` で対象記事を最大1000件取得
3. 各記事に `SanitizeHTMLAndTrim()` を適用、空コンテンツ・空タイトルをスキップ
4. `UpsertArticlesWithFeedID` でバッチ挿入（既存記事はスキップ）
5. カーソルを進めて次回バッチに備える

**テスト（7件）:**

- 正常系: 空フィードの記事がバックフィルされること
- 0件: 対象記事がない場合のハンドリング
- サニタイズ: XSSスクリプトが除去され空コンテンツがスキップされること
- 空タイトル: タイトルなし記事がスキップされること
- エラー系: DB接続エラー、認証エラーの伝播
- FeedID保持: SQL JOINで解決済みのFeedIDが再解決されずそのまま使われること

### PROS

- **べき等性**: `ON CONFLICT DO NOTHING` + カーソルにより何度実行しても安全
- **品質保護**: 既存のフルテキスト記事はInoreader要約で上書きされない
- **段階的処理**: LIMIT 1000のバッチ処理で毎時実行。DB負荷を分散しつつ徐々にフィードを埋める
- **既存コードの再利用**: `Sanitizer`, `GetSystemUserID`, `JobRunner`パターンなど既存インフラを活用
- **両モード対応**: Legacy（single-DB）とAPI（split-DB）の両環境で動作

### CONS, TRADEOFF

- **Inoreaderコンテンツはフルテキストではない**: `inoreader_articles.content`はRSS要約であり、本文全体ではない。バックフィルされた記事は要約品質が低い可能性がある。ただし、記事が0件のフィードよりは有用
- **`feed_links`テーブルへの依存**: LegacyモードのJOINは`feed_links.url`と`inoreader_subscriptions.feed_url`の完全一致を前提としている。URL正規化の差異があるとマッチしない
- **毎時実行のオーバーヘッド**: 全フィードが埋まった後もクエリは実行される（`NOT EXISTS`で0件を返すだけだが）

### SQLクエリ（Legacyモード）

```sql
SELECT ia.id, ia.article_url, ia.title, ia.content, ia.published_at, ia.fetched_at, f.id AS feed_id
FROM inoreader_articles ia
INNER JOIN inoreader_subscriptions isub ON ia.subscription_id = isub.id
INNER JOIN feed_links fl ON isub.feed_url = fl.url
INNER JOIN feeds f ON f.feed_link_id = fl.id
WHERE NOT EXISTS (
    SELECT 1 FROM articles a WHERE a.feed_id = f.id AND a.deleted_at IS NULL
)
AND ia.content IS NOT NULL AND ia.content_length > 0
ORDER BY ia.fetched_at ASC
LIMIT $1
```
