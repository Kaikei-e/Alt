# ADR-000767: tag-generator に Hurl E2E スイートを追加し段階的全面展開の第 4 サービスとする

## Status

Accepted

## Date

2026-04-18

## Affected Services

- tag-generator (Python 3.14 / FastAPI) — プロダクションコードは一切触らず、E2E スイート 5 シナリオ (readiness + /health schema + extract-tags smoke/keyword + mq-hub 経由 Redis round-trip) と staging compose 片肺 (tag-generator service + 既存 redis-streams/mq-hub/stub-backend の profile 拡張) を追加する
- mq-hub (Go) — ランタイム無変更。scenario 04 の HTTP エントリポイント `services.mqhub.v1.MQHubService/GenerateTagsForArticle` を staging で公開するため `profiles` に `tag-generator` を足し、image を同スライスに同梱する

## Context

[[000763]] で Hurl E2E フレームワークを Increment 1 として `search-indexer` に導入、[[000764]] で Increment 2 として `mq-hub`、[[000765]] で Increment 3 として `knowledge-sovereign` を加えた。これらは **HTTP handler のみ** / **Connect-RPC + Redis Streams** / **DB-backed state + admin HTTP** の 3 パターンだったが、Alt の AI ハンドラ層の起点である `tag-generator` ([[000358]]) は次の点でこれらと別個のサーフェスを持つ。

1. **応答コーデックが Connect-RPC でも encoding/json でもなく「FastAPI デフォルト」** である。`auth_service.py` の `/health` `/api/v1/extract-tags` は Pydantic モデル + FastAPI の標準 JSON エンコーダで出力されるため、フィールドは **snake_case** (`inference_ms`, `language`, `confidence`)。[[000765]] で確立した「proto3-JSON camelCase vs encoding/json PascalCase」の 2 コーデックに加えて 3 つ目のパターンになる。同一スイートで scenario 02/03 (snake_case) と scenario 04 (mq-hub proto3-JSON camelCase) を併存させる形でこれを明示する。
2. **Redis Streams の reply stream を Hurl で直接観測できない**。tag-generator の実スループットは `TagGenerationRequested` → ML inference → `TagGenerationCompleted` のリプライ経路だが、Hurl 7.1 は HTTP クライアントであり `XREAD` できない。proxy 層を足すか、HTTP ラッパーを経由するかの 2 択。
3. **mq-hub scenario 14 が happy path 側を空けている**。[[000764]] では `GenerateTagsForArticle` を tag-generator 不在で叩く **timeout variant** を pin したが、tag-generator 実装済みの happy path は「後続スイートで埋める」と明記して deferred にしていた。Increment 4 で最短経路を埋められる。
4. **ML inference の cold-start を踏む**。`paraphrase-multilingual-MiniLM-L12-v2` は Dockerfile のビルドステージで `/home/appuser/.cache/huggingface` に焼き込み済みなので download はゼロ秒だが、first-invocation は mmap + torch graph 初期化で 2-core の CI runner で 5–12 秒かかる。scenario 04 が直接 15 秒の round-trip を期待すると 1 回目から揺れる。
5. **`_background_tag_service` は FastAPI lifespan で初期化される** が、`/health` (`:9400`) は uvicorn が port bind した瞬間に 200 を返す。すなわち **healthcheck が緑でも `/api/v1/extract-tags` は 503 を返し得る** 空白時間帯が存在する (`auth_service.py:519` — "Tag extraction service not ready")。
6. **`/api/v1/tags/batch` は `psycopg2` 直結**。`tag_fetcher.py` は `DB_TAG_GENERATOR_USER/PASSWORD/HOST/PORT/NAME` を env から DSN 構築する旧パス (API mode 移行前の残置) で、happy path を動かすには `articles + article_tags + feed_tags` schema を持つ Postgres と seeded data が必要。[[000765]] の knowledge-sovereign 規模の追加になるため Phase 1 には重すぎる。
7. **認証系は staging で no-op**。`/api/v1/generate-tags` / `/api/v1/user-preferences` は `alt_auth.client` を使うが、staging image には `alt_auth` が含まれないため `_ALT_AUTH_AVAILABLE=False` 経路で `require_auth` が常に通過する。この状態の endpoint を E2E でテストしても no-op の成功しか観測できず、covered にならない。

## Decision

**tag-generator に対する Hurl E2E スイートを readiness + 4 シナリオの計 5 ファイルで追加する。scenario 04 は mq-hub の `GenerateTagsForArticle` Connect-RPC を HTTP ラッパーとして利用し Redis Streams round-trip を Hurl-observable にする。`--jobs 1` で scenario 03 (SBERT warm-up) を scenario 04 (15s round-trip) より先に走らせる。negative path / Postgres 依存エンドポイント / 認証系 / mTLS peer-identity は Phase 2 (別 PR) に繰り延べる。**

### 採用する構成

1. **Hurl スイート**: `e2e/hurl/tag-generator/` に以下 5 本 + run.sh + README.md を置く。
   - `00-setup.hurl` — `:9400/health` を retry 30 × 500 ms で探る readiness probe。`status=="healthy"`, `service=="tag-generator"`。
   - `01-health-schema.hurl` — retry なしで `/health` を pin。`Content-Type: application/json` 含むことも assert してエンコーダ契約を固定。
   - `02-extract-tags-smoke.hurl` — `POST /api/v1/extract-tags` に最小 body。`[Options] retry: 20 retry-interval: 500` で `_background_tag_service` 初期化遅延を吸収。assert: `$.success==true`, `$.tags isCollection`, `$.confidence`/`$.inference_ms isNumber`, `$.language exists`。
   - `03-extract-tags-keyword.hurl` — keyword-dense な英語段落で `$.tags count > 0` と `$.confidence > 0` を assert。同時に SBERT モデルを warm。
   - `04-generate-tags-via-mqhub.hurl` — `POST {{mqhub_base_url}}/services.mqhub.v1.MQHubService/GenerateTagsForArticle` (proto3-JSON camelCase, `timeoutMs=15000`)。body は `{{run_id}}` 展開のためインライン (`file,…;` の展開未対応は [[000765]] で確認済)。assert: `$.success==true`, `$.articleId` 一致, `$.tags isCollection`, `$.inferenceMs > 0`。
2. **staging compose (`compose/compose.staging.yaml`)**:
   - 新規サービス `tag-generator` を `profiles: [tag-generator]` で追加。image は `ghcr.io/${GHCR_OWNER:-kaikei-e}/alt-tag-generator:${IMAGE_TAG:-main}`、`MTLS_ENFORCE=false`, `PEER_IDENTITY_TRUSTED=off`, `CONSUMER_ENABLED=true`, `BACKEND_API_URL=http://stub-backend`。`DB_TAG_GENERATOR_*` は **意図的に未設定** とし `/api/v1/tags/batch` を実質無効化 (import-time ではなく call-time 失敗なので Phase 1 スコープで叩かない限り安全)。`start_period: 20s` で FastAPI lifespan の SBERT 初 mmap に余裕を持たせる。
   - 既存サービスの `profiles` をそれぞれ additive に拡張: `redis-streams: [mq-hub, tag-generator]`, `mq-hub: [mq-hub, tag-generator]`, `stub-backend: [search-indexer, tag-generator]`。mq-hub suite / search-indexer suite の実行には影響しない。
3. **run.sh (`e2e/hurl/tag-generator/run.sh`)**: knowledge-sovereign の run.sh を簡略化 (DB FK がないので UUID 生成は `RUN_ID` のみ)。`docker compose --profile tag-generator up -d --wait redis-streams stub-backend mq-hub tag-generator` → Hurl を `--network alt-staging` 内で起動 → `00-setup.hurl` で readiness → `--jobs 1 --report-junit --report-html` で 01–04 を直列実行 → `trap cleanup` で `down -v --remove-orphans`。`KEEP_STACK=1` でスタック維持 (デバッグ用)。
4. **CI (`.github/workflows/e2e-hurl.yml`)**: path filter に `tag-generator/**`, `e2e/hurl/tag-generator/**`, `e2e/fixtures/tag-generator/**` を追加。`mq-hub` job と同列に `tag-generator` job を追加し、**tag-generator + mq-hub 両方の image** を `:ci` で local build してから run.sh を呼ぶ。scenario 04 が mq-hub 経由なので image を欠くと 04 だけ常失敗になる。`KEEP_STACK=1` + `failure: logs --tail=300` + `artifacts: e2e/reports/tag-generator-*` で runbook 互換のデバッグパスを揃える。

### 検討した代替案

**代替案 A: scenario 04 の代わりに tag-generator 自身のカスタム `/debug/last-reply` エンドポイントを追加**。ランタイムを触らなくて済むメリットが消えるので却下。Hurl E2E は「プロダクションコードを一切変えずに観測可能性を追加する」という [[000763]] [[000764]] [[000765]] の不変条件に違反する。

**代替案 B: Phase 1 を 00–03 の HTTP smoke のみにして 04 を Phase 2 へ繰り延べ**。mq-hub scenario 14 の timeout pin の happy path が埋まらず、round-trip が実機で壊れても catch できない期間が発生する。mq-hub image build と staging 配線は既存資産で賄えるため、コスト対効果は 04 込みの方が高い。採用しない。

**代替案 C: Postgres を staging に足して `/api/v1/tags/batch` も Phase 1 に入れる**。knowledge-sovereign 相当の追加 (db + migrator) が必要で、今回は `tag_fetcher.py` のスキーマが alt-backend 本番と結合している (`articles + article_tags + feed_tags`) ため migration を切り出すだけでは済まない。別 increment で扱う。

**代替案 D: SBERT の warm-up を run.sh 内で `curl` によって先行実行**。scenario 03 が本来の assertion を持つテストなので、warm-up を副作用として使う方が「テストが 1 つのことだけをする」原則に沿う。run.sh に side-channel inference を入れるのは却下。

### 不変条件との整合

- TDD first: スイート → 失敗確認 → compose/CI 配線 → 緑化の順で進められる構成になっている。
- reproject-safe / append-first 等の Knowledge Home 不変条件: tag-generator は projector ではなく touch しない。
- [[000591]] Pact CDC との棲み分け: `mq-hub → tag-generator` の message pact は既存 ([[000764]] で言及) で Phase 1 の scenario 04 は契約を再検証するのではなく「実機で round-trip が成立すること」を pin する。Pact は shape、Hurl は actuality。

## Consequences

### Pros

- `GenerateTagsForArticle` の happy path が mq-hub scenario 14 の timeout 側と対になり、round-trip 破壊を PR 単位で拾える。
- FastAPI snake_case / Connect-RPC camelCase の 2 エンコーダを 1 スイートでカバーしたので、後続の AI ハンドラ層サービス (recap-evaluator 等) の追加時に template として流用できる。
- staging stack の service 追加は additive で既存 3 スイートを壊さない (compose profiles の性質)。`docker compose config --quiet` で validate 済み。
- CI の gating は path filter で per-suite に閉じているので無関係 PR は image build コストを払わない (他 3 スイートと同じ慣行)。
- retry + `--jobs 1` で lifespan race / cold-start の 2 つの非決定性を同じ仕組みで吸収している。別々の workaround を足すより運用負債が少ない。

### Cons / Tradeoffs

- scenario 04 は mq-hub image に hard dependency を持つ。tag-generator だけのバグ改修 PR でも mq-hub build が走るので CI 時間が 2 つ目の Docker layer 分伸びる。対策: path filter で mq-hub 側変更 PR ではこちら job を走らせない運用を維持する (workflow 既存ポリシー通り)。
- `/api/v1/tags/batch` が Phase 1 に入らないため、`fetch_tags_by_article_ids` の DB 契約は引き続き contract test でのみ保護される。Phase 2 で Postgres profile を足すまで Broker の当該 pacticipant がズレる変更に気付けない。
- `DB_TAG_GENERATOR_*` 未設定は意図的だが、将来誰かが staging に `/api/v1/tags/batch` を叩くテストを足したときに `psycopg2` エラーで 500 を返すだけの診断性の悪いモードに落ちる。README `Out of scope` に明記した。
- SBERT 初 mmap を吸収するために `start_period: 20s` としたが、CI runner の IO が遅い場合は 20 秒でも足りない可能性がある。`retry: 20 × 500 ms` の scenario 02 が補償するので致命的ではないが、将来 retry 回数の調整が入る余地はある。
- `inferenceMs > 0` アサートは float 精度に依存する。KeyBERT が 1ms 未満で返すほど短いテキストを投入すると fail する可能性があるが、現在の fixture は 400 字程度あるので余裕がある。
