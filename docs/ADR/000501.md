# ADR-000501: 要約キューワーカーの並列度を設定可能にして backlog 消化速度を引き上げる

## Status

Accepted

## Date

2026-03-20

## Affected Services

- pre-processor (Go)

## Context

BE 要約は下流側で分散実行できるようになっていたが、pre-processor 側の queue worker は 1 件ずつ逐次処理していた。そのため queue に大規模 backlog が存在する場合、下流の処理能力が複数実行先に広がっていても、上流が単一 worker のままでは throughput が頭打ちになる。

今回のボトルネックは「単一ジョブの処理失敗」ではなく、**worker 側の供給量不足** だった。特に pending backlog が大きいとき、逐次処理は backlog age の改善速度を著しく下げる。

一方で、無制限な並列化は下流サービス overload 時の挙動を不安定にする。したがって、固定の多重化ではなく、設定可能で保守的な並列度が必要になった。

## Decision

**SummarizeQueueWorker に設定可能な concurrency を導入し、複数ジョブを並行処理できるようにする。**

今回の決定では以下を採用する。

- `SUMMARIZE_QUEUE_CONCURRENCY` を新設する
- デフォルト値は `3` とする
- worker は取得済み pending ジョブを fan-out で処理する
- downstream overload 検知時は既存方針に合わせて backoff し、残り job の投入を止める
- concurrency 未設定または不正値時は `1` にフォールバックせず、設定バリデーションで弾く

### 代替案

| 案 | 評価 |
|---|---|
| 逐次処理のまま維持 | backlog 消化速度が不足するため不採用 |
| 無制限 goroutine | 下流 overload 時の制御が難しく、不安定化リスクが高いため不採用 |
| 複数プロセスに分割 | デプロイ複雑性が上がる割に、まずは単一プロセス内の concurrency で十分 |

## Consequences

### Pros

- backlog の消化速度を引き上げられる
- 下流の分散実行能力を pre-processor 側でも活かせる
- concurrency を環境変数で調整でき、段階的に運用できる

### Cons / Tradeoffs

- 同時実行が増えるため、下流 overload 時のログや挙動がやや複雑になる
- queue 内ジョブの完了順は投入順と一致しなくなる
- 過大な concurrency 設定をすると下流実行基盤を過負荷にするリスクがある
