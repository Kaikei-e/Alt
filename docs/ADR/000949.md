# ADR-000949: Knowledge Trail を調査の再開点として再設計し episode spine と patch-exit branch を導入する

## Status

Accepted

## Date

2026-07-19

## Affected Services

- knowledge-sovereign (Go)
- alt-backend (Go)
- alt-frontend-sv (TypeScript / SvelteKit)
- proto (alt/knowledge_trail/v1, services/sovereign/v1)

## Context

[[000940]] で Knowledge Loop を retire して立ち上げた Knowledge Trail は、Wave 2–6 で
spine / path wear / branch / trail closure の配線を完了していたが、実画面は
「何をしたいのか・どんな便益があるのか」が読み取れない状態に退化していた。
2026-07-18 の grill セッション (決定 D23–D30) で、コード探索と HCI 文献調査
(Web 再訪研究 CHI 2007/2008、Keeping Found Things Found、information foraging、
explainable recommendation) を突き合わせて次の構造要因を特定した:

- **仕事の未定義**: 「primary surface」を掲げていたが、履歴を背骨にしたページは実証的に
  低頻度・高価値 (rediscovery は再訪全体の約 8%、ブラウザ履歴の利用率は全リクエストの
  0.2%)。毎日開く前提の設計と素材の性質が最初から矛盾していた。
- **flat 時系列 + 日別区切り**は「使われないブラウザ履歴」と同じ実証済みの失敗形。
  同一記事の再読が日ごとに行を増やし、重複が画面を占拠していた。
- **lens バー**が読み込んだ footprint の生タグ (ML 生成、[[000182]] 系の既知ノイズ) を
  無加工・無上限で union 表示しており、"also" "could" 等のストップワードまで
  チップ化していた (死んだタグクラウドの再演)。
- **branch** は 4 点セット (型/why/evidence/confidence) を schema 強制するなど配線は
  健全だったが、生成されるのは最も凡庸な cluster 型のみで、提示場所も低頻度なはずの
  Trail ページ内に限られていた。低頻度サーフェスの内側だけに提案を置くのは自己矛盾。

## Decision

Trail の仕事を**調査の再開点 (resume surface)** と定義し直し (D23、"primary surface" は
撤回)、以下を 5 つの垂直スライス (Wave 7–11 前半) として TDD (E2E → CDC → Unit、
RED/GREEN 分離コミット) で実装することを採用する。

1. **接触の折り畳みとタグ浄化 (Wave 7)** — spine の読み取りで (item_key, verb) ごとに
   接触を折り畳み、`contact_count` / `first_occurred_at` を付与する (D24)。ML タグは
   `tagclean` パッケージ (stopword / 桁のみ / URL 断片 / 大文字・単複統合) を通してから
   サービス外へ出す (D25)。lens チップバーは撤去。
2. **episode spine (Wave 8)** — spine のデフォルト表示単位を導出 episode にする (D24/D30):
   同一記事への全接触は無条件連結、記事間は浄化タグ 2 個以上の重なり × 14 日窓で連鎖。
   純関数 `trail_episodes.Derive` として実装し、実体テーブル・新イベントは作らない
   (second spine の構造的禁止を維持)。BFF が代表記事の OG 画像を既存 image-proxy signer で
   署名して添付し (D29)、未解決はテキストへ退化。日付は grouping 軸から目盛りに降格。
3. **trail search (Wave 9)** — lens の後継として検索一本 (D25)。既存 Meilisearch 記事
   index (SearchArticles port) のヒットとユーザの spine を BFF で交差し、sovereign は
   `filter_item_keys` で導出 episode を絞る (member 文脈は完全保持)。UI は pull-only
   (submit のみ発火・$effect ゼロ)。新 index は増設しない。
4. **patch-exit branch (Wave 10)** — branch の主戦場を記事詳細の読了点に移す (D26、
   `GetItemBranches`、最大 2 件)。Trail ページは episode ヘッダ従属の 1 件のみとし、
   無文脈な branch inbox を撤去。planner は **anchored why** (anchor タイトルの引用) を
   emit 時に強制し、解決不能なら loud log の上で emit しない (D28)。dismiss には
   1 タップ理由 (allowlist: not_following_topic / already_known / wrong_relation) を
   添え、resolution イベント payload に記録する (migration なし・event log のみ)。
   効果測定は CTR ではなく taken→engaged dwell 率 (構造化ログ → ClickHouse)。
5. **Continuation 型 (Wave 11 前半)** — 「静かになったスレッドの再開」提案を planner に
   追加する (D27): 接触 1–3 回・ask なし・engaged dwell なし・最終接触 3–21 日前・
   未提案のアイテムを 1 run 1 件だけ、anchor = target 自身で emit する。

**検討した代替案と却下理由**:

- *lens バーの浄化 top-8 縮小* — facet 設計のベストプラクティスには沿うが、episode が
  テーマの塊を既に見せるため役割が重複し、rediscovery に必要な全文検索の代替にならない。
- *episode の実体テーブル化* — 再読・連鎖判定の更新コストと second spine 禁止
  (核心原則 2) に抵触。純導出なら閾値変更に reproject が不要。
- *embedding による episode 連鎖* — タグノイズは回避できるが GPU/計算コストと
  reproject 決定性 (モデル更新で導出が変わる) の問題。measure-gate で将来判断。
- *Contradiction 型の同時実装* — sensemaking 研究上は看板機能だが、主張抽出・スタンス
  判定に LLM が必須で、誤誘導 scent の信頼毀損コストが全型中最大。eval-first
  (評価データセットを先に設計・計測) の明示的な空白として plan 文書に記録し、本 ADR の
  スコープから外した。

## Consequences

### Pros

- 画面から重複行と生タグチップが消え、「調査の流れ」単位で読める spine になった。
  価値提案 (中断した調査に戻る・読んだはずのものを再発見する) が UI 構造と一致する。
- episode / 検索 / 折り畳みはすべて読み取り側の純導出で、新イベント語彙はゼロ。
  append-first / reproject-safe / 語彙最小の不変条件は無傷。
- branch が注意の解放点 (読了点) に出るようになり、anchored why の契約強制で
  generic why (placebo 説明) が構造的に emit 不能になった。
- 契約は全段 additive で、pact (12 interaction) と provider 検証が新 field / 新 RPC を
  pin 済み。E2E は 19/20 green (残 1 は実セッション前提の既存環境問題)。

### Cons / Tradeoffs

- episode 導出の閾値 (共有タグ 2・14 日窓・500 行導出窓) と Continuation の窓
  (3–21 日) は理論駆動の初期値であり、実データでの較正が未着手 (D21 の planner 較正
  同様、measurement-first の明示的な空白)。
- episode ページングは固定導出窓 + offset cursor で、spine が 500 行を超える古い区間は
  導出対象外。実利用で窓を超えたら再設計が必要。
- legacy の flat `footprints` field は wire 互換のため残置 (常に空)。次の breaking
  window で削除する負債。
- 記事→記事の連鎖 branch-take では同一ルート再遷移のため dwell 計測の再アームが
  行われない既知の限界がある (現行の全テストフローは非影響)。
- E2E integration の実セッション認証はモック cookie が実 Kratos を通らない構造問題が
  残っており、1 テストが常時 fail する (本変更とは独立の既存問題)。
