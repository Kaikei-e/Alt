# ADR-000679: LangGraph Postgres checkpointer を有効化し durable execution を確立する

## Status

Accepted

## Date

2026-04-10

## Affected Services

- acolyte-orchestrator (Python 3.14+ / LangGraph / Starlette)

## Context

Acolyte v5 のパイプラインは 1 run あたり約 70 分かかり、うち 60 分以上が LLM 推論待ちである。LLM タイムアウト、プロセス crash、DB 接続断など、長時間 run 中の中断リスクは高い。

checkpoint インフラは ADR-673 で設計され、コード上は `AsyncPostgresSaver` / `aget_state` / `ainvoke(None)` の resume ロジックが実装済みだったが、以下の問題が残っていた:

1. **`CHECKPOINT_ENABLED` が false のまま** — compose で有効化されておらず、実運用で checkpoint が保存されていなかった
2. **durability mode が未指定** — LangGraph のデフォルト durability では、長時間バッチで checkpoint 書き込みが遅延するリスクがあった
3. **resume の外部入口が不在** — `StartReportRun` は毎回新しい `run_id` を生成するため、既存 run を resume する手段がなかった
4. **completed 判定が曖昧** — `snapshot.values` の truthiness だけで判定しており、Finalizer 未到達の terminal error state を completed と誤判定するリスクがあった
5. **checkpoint factory の unit test がなかった** — `setup()` 呼び出しが検証されていなかった
6. **運用手順書が存在しなかった** — resume の実行方法が文書化されていなかった

## Decision

### 1. Compose で `CHECKPOINT_ENABLED=true` を設定する

`compose/acolyte.yaml` の `acolyte-orchestrator` environment に `CHECKPOINT_ENABLED=true` を追加する。`Settings` のデフォルトは `False` のまま維持し、deploy 側で opt-in する設計とする。

### 2. `durability="sync"` を明示する

`_run_pipeline()` の `ainvoke()` 呼び出しに `durability="sync"` を渡す。`sync` モードでは各 super-step 完了時に checkpoint が確実に Postgres に書き込まれてから次の step に進む。

```python
invoke_kwargs: dict[str, object] = {"config": config}
if self._settings.checkpoint_enabled:
    invoke_kwargs["durability"] = "sync"
result = await self._graph.ainvoke(invoke_input, **invoke_kwargs)
```

性能コストはあるが、Issue 3 の目的は途中成果保全であり、71 分の run では checkpoint 書き込みの数秒は無視できる。

### 3. completed 判定を `final_version_no` ベースに強化する

従来の `snapshot.values` truthiness 判定を、`final_version_no is not None` による明示判定に変更する。

- `snapshot.next` が truthy → pending state、resume する
- `snapshot.next == ()` かつ `final_version_no is not None` → 正常完了、skip する
- `snapshot.next == ()` かつ `final_version_no` なし → terminal error state、warning ログを出して再実行を試みる

### 4. operator 用 resume script を追加する

`scripts/resume_run.py` を新設し、既存 `run_id` を指定して checkpoint から resume できるようにする。

- `--run-id <uuid>` を受け取り、DB から `report_id` / brief を解決
- 既存の `_run_pipeline()` を再利用し、同一 `thread_id` (`acolyte-run:{run_id}`) で `ainvoke(None)` を実行
- proto / public API の変更は不要

### 5. checkpoint factory の unit test を追加する

`test_checkpoint_factory.py` で `AsyncPostgresSaver.from_conn_string()` → `setup()` → yield の contract を検証する。実 DB integration は不要だが、factory の振る舞いを mock で押さえる。

### 6. 運用手順書を作成する

`docs/runbooks/acolyte-checkpoint-resume.md` に以下を記載する:

- checkpoint の仕組み (super-step 境界保存、thread_id、durability)
- resume 手順 (障害確認 → DB 健全性確認 → script 実行 → 結果確認)
- 障害パターン別の対処 (LLM timeout, OOM, DB 接続断, completed run)
- 制約事項 (mid-node 再開不可、replay の性質、idempotency)

## Implementation

### 変更ファイル

| ファイル | 変更 |
|---------|------|
| `compose/acolyte.yaml` | `CHECKPOINT_ENABLED=true` 追加 |
| `acolyte/handler/connect_service.py` | `durability="sync"` 付与、completed 判定の `final_version_no` ベース化 |
| `tests/unit/test_connect_service_checkpoint.py` | `test_run_pipeline_uses_sync_durability_when_checkpoint_enabled`, `test_run_pipeline_completed_checkpoint_requires_final_version` 追加 |
| `tests/unit/test_checkpoint_factory.py` | 新規: `test_create_checkpointer_calls_setup`, `test_create_checkpointer_yields_saver` |
| `scripts/resume_run.py` | 新規: operator 用 resume entrypoint |
| `docs/runbooks/acolyte-checkpoint-resume.md` | 新規: resume 運用手順書 |

### checkpoint が保存される node 境界

```
planner → [CP] → gatherer → [CP] → curator → [CP] → hydrator → [CP] →
compressor → [CP] → quote_selector → [CP] → fact_normalizer → [CP] →
section_planner → [CP] → writer → [CP] → critic → [CP] → finalizer → END
```

critic → writer の revision ループでも各 node 出入りで checkpoint が保存される。

### mid-node 制約

QuoteSelector (per-article ループ)、FactNormalizer (per-quote ループ)、Writer (per-section ループ) は node 内に多数の LLM 呼び出しを持つ。これらの node 内途中で失敗した場合、node 頭からやり直しになる。node 分割は将来課題。

## Consequences

### Pros

- **途中成果保全** — 70 分 run の途中で crash しても、完了済み node の出力は checkpoint に保存されており、resume で skip できる
- **確実な永続化** — `durability="sync"` により、各 super-step 完了時に checkpoint が Postgres に書き込まれる
- **運用可能な resume** — operator が `scripts/resume_run.py` で既存 run を resume でき、runbook に手順が文書化されている
- **安全な completed 判定** — `final_version_no` ベースの判定により、terminal error state を completed と誤判定しない
- **factory contract 検証** — `setup()` の呼び出しが unit test で保証される

### Cons / Tradeoffs

- **mid-node 保全不可** — FactNormalizer の 20 quotes 処理中に 15 で失敗しても、quote 1-14 の結果は消失する。LangGraph の super-step 設計制約
- **`durability="sync"` の性能コスト** — 各 super-step 後に同期的に Postgres 書き込みが発生する。ただし 70 分 run に対しては無視できる
- **checkpoint テーブルは LangGraph 管理** — `setup()` が DB 内に自動作成する。Atlas migration の対象外であり、schema は langgraph-checkpoint-postgres に依存する
- **resume は replay** — 同じ行から継続するのではなく、失敗した node を最初から再実行する。side effects の idempotency が前提
