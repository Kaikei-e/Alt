# ADR-000940: Knowledge Loop を Knowledge Trail へ retire し loop projection を DROP して event log から再射影する

## Status

Accepted

本 ADR は Knowledge Loop の read 表層系統（[[000929]] / [[000930]] / [[000933]] / [[000937]] / [[000938]] / [[000939]]）を実体として retire する。これらの ADR の内容は設計史として残すが、その read model・RPC・UI は本 ADR をもって削除済みとなる。

## Date

2026-06-11

## Affected Services

- knowledge-sovereign (Go) — trail read model を自 DB へ移設、loop projection を DROP、loop read-path の dead code を除去
- alt-backend (Go) — trail は sovereign RPC client 経由で参照（直 SQL なし）
- alt-frontend-sv (SvelteKit) — `/loop` を retire し `/knowledge/trail` へ置換
- migrations-atlas — alt_db に誤って作られた trail table を idempotent DROP
- proto — `alt/knowledge_trail/v1` + `services/sovereign/v1` の `GetTrailFootprints` が loop RPC を置換

## Context

Knowledge Loop は「Knowledge パイプラインに OODA を組み込む」という構想としては妥当だったが、read 表層の実装が繰り返し失敗した。

- **3軸直交モデル**（proposed_stage × session_stage × surface_bucket）は、ユーザも実装者も LLM も同時に保持できず、context rot 下で single-axis collapse（一軸への退化）が混入し続けた。
- **4-bucket UI**（now / continue / changed / review）はシステム分類学のユーザへの露出であり、decorated feed に退化した。
- **SSE 配信路**は死んだ配線が silent failure の温床となった（[[PM-2026-045]]）。

read 表層を立て直す試みを ADR を重ねて行った（OODA を記述レンズへ再接地 [[000933]]、UI を spotlight+bench へ [[000930]]、SSE を4層で固める [[000929]]、関係ループへ再設計 [[000937]]、relation-set を loop.v1 正準 read へ統合 [[000938]]、evidence を co-projected accumulator へ [[000939]]）が、いずれも体験へ降りきらなかった。構想は維持しつつ、read 表層そのものを別の語彙（footprints + typed branches）で作り直す判断に至った。

不変条件の観点では、`knowledge_loop_*` projection は disposable read model（append-only な `knowledge_events` から再構築可能）であるべきものだった。Trail への置き換えは、この projection を「捨てて event log から再生できる」ことを実地で証明する機会でもある。

加えて、retire 実装中に **runtime で Trail を機能不全にする重大バグ** を発見した。Trail の read model migration（`knowledge_trail_footprints` / `knowledge_trail_branches`）が `migrations-atlas/` 配下に置かれていた。`migrations-atlas` は `alt_db` をターゲットにするが、Trail の projector / branch planner / read handler を動かす sovereign service は `knowledge_sovereign` DB（`knowledge-sovereign/migrations` から migrate）を使う。結果として trail table は sovereign 側の DB に存在せず、projector も read handler も runtime で落ちる状態だった。このバグは PR #171（trail spine / branch producer）で混入し、すでに main にマージ済みだった。

## Decision

**Knowledge Loop の read 側（`KnowledgeLoopService` RPC・`/loop` UI・3軸直交モデル・4-bucket UI・`knowledge_loop_*` projection tables）を削除・DROP し、Knowledge Trail（footprints の背骨 + system 提案の typed branch）へ置き換える。** あわせて以下を確定する。

- **event log は永久保存する。** `knowledge_events` は INSERT-only のまま touch しない。過去の `knowledge_loop.*` イベントも残し、Trail projector は `knowledge_loop.acted.v1` を read footprint として再射影する。これにより Loop 時代の足跡が Trail の背骨へ連続する。
- **branch は four-tuple を schema で強制する。** `knowledge_trail_branches` は `relation_kind` / `why` / `evidence_refs_json` / `confidence` を NOT NULL とし、四つ揃わない branch 行が存在し得ない。「untyped branch（= decorated feed への退化）」を構造的に不可能にする。
- **read model は正しい DB に置く。** trail の CREATE migration を `knowledge-sovereign/migrations/00026・00027` に移設し、loop projection の DROP を `00028` とする（`knowledge_events` は対象外）。`alt_db` に誤って作られた orphan trail table は `migrations-atlas` の `DROP TABLE IF EXISTS`（idempotent）で除去し `schema.hcl` と整合させる。
- **DROP は destructive migration として明示する。** `00028` に `-- atlas:nolint destructive` を付け、`atlas migrate lint` で「no diagnostics」を確認した（loop table/view/enum がすべて実在し、参照ミスがないことの裏取りでもある）。

### 検討した代替案と却下理由

1. **Loop を freeze（凍結）して残す** — 却下。死んだ read 配線は silent failure の温床（[[PM-2026-045]]）。read RPC を残すと consumer が幻の契約に依存し続ける。削除が正。
2. **loop projection を残したまま Trail を別 table で並走させる** — 却下。disposable projection の「捨てて再生できる」証明にならず、二重 read model が drift する。retire の目的（不変条件の実証）を満たさない。
3. **trail table を `alt_db` に置いたまま sovereign から cross-DB 参照する** — 却下。アーキ規則「No shared DB access / producer は API・event 経由」に反する。sovereign は自分の durable read model の単一所有者であるべき。
4. **混入済みの `migrations-atlas` trail CREATE を履歴から削除する** — 却下。すでに `alt_db` へ適用済みの可能性があり、適用済み migration の削除は Atlas の drift 検出を壊す。append-only で `DROP` を足すのが安全かつ自己整合的。

## Consequences

### Pros

- read model が event log から再構築可能であることを実証し、イミュータブルデータモデルの正しさ（disposable projection / reproject-safe）を担保した。
- 死んだ Loop 配線（RPC・SSE・UI）を消し、silent-failure の面を縮小した。
- four-tuple を schema NOT NULL で強制したことで、Loop を退化させた「untyped branch / decorated feed」が構造的に再発しない。
- sovereign が自分の read model を自分の DB に持つ形になり、DB 所有権とアーキ層境界が整合した。
- Trail を機能不全にしていた runtime バグ（trail table 不在）を本番反映前に根治した。

### Cons / Tradeoffs

- 過去の `knowledge_loop.*` イベントは永久に残るが、現行 projector はそれを read footprint としてしか解釈しない。将来 Loop の forensic reproject が必要なら projector を書き直す必要がある（event は残るので再現は可能）。
- `migrations-atlas` 側に CREATE→DROP の churn が残る（`alt_db` に一瞬 trail table が作られて消える）。append-only を守る代償。
- `00028` は destructive。deploy 時にのみ適用され、必ず `migrate → deploy` の順を要する。ロールバックは event log からの reproject に依存する。
- branch resolution に伴う `act_outcome`（dwell timer → `act_outcome.v1`）は本 retire の範囲外で未実装。Trail の「行動の帰結」観測は別途。

## Deploy-model 整合性セルフチェック

新設 compose service はない（compose / deploy pattern を触らない）。ただし destructive な DB migration（`00028` の loop projection DROP、`migrations-atlas` の orphan trail DROP）を含むため、必ず `migrate → deploy` の順で適用する。逆順だとアプリが新スキーマを期待したまま旧スキーマで起動し healthcheck が通らない。
