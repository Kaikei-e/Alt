# ADR-000937: Knowledge Loop を OODA 忠実な関係ループへ再設計し体験へ降ろす

## Status

Accepted

本 ADR は実装前の decision record である。採用した設計方針を固定し、後続の実装 PR の指針とする。projection schema / proto / UI に触れる具体実装は別 PR で行い、その際に本 ADR の Decision を contract とする。

## Date

2026-06-03

## Affected Services

- knowledge-sovereign (Go)
- alt-frontend-sv (TS/Svelte)
- recap-worker (Rust)
- rag-orchestrator (Go)

## Context

Knowledge Loop は 15 回以上再実装され、accepted ADR は 84 件を超えるが、**体験は「装飾された feed」のまま**で OODA の理念が体験に降りていない。`docs/review/knowledge-loop-retrospective-2026-05-28.md` は、病は「incomplete implementation」ではなく「**どの再実装も durable にならない構造的プロセスギャップ**（沈黙する evidence 劣化 + 事後対応の観測）」だと結論づけた。

今回 grill-with-docs セッションで、実機の `/loop` を触った上で 4 つの体験的欠落が確認された:

1. **Orient/Decide が空洞** — タップで `observe→orient` に自動前進するだけで、OODA の schwerpunkt である「方向づけ」が stage ラベルの付け替えに縮退している。
2. **フィードバックの弧が不可視** — act してもイベントは emit されるが、ユーザーは「自分の行動で次の observe がどう変わったか」という戻りを一度も見ない。`OodaPipeline` の `↻` は cosmetic。
3. **依然 stepper に感じる** — OodaPipeline は受動リボンとして実装済みだが、体験は依然 O→O→D→A の段階移動に感じられる。
4. **ループでなく装飾された feed** — 認知ループが theater になっている。

コード裏取りで、この体験的欠落の構造的原因が特定された:

- `EventLogSurfaceScoreResolver` は本番 DI で配線済み（`knowledge-sovereign/app/main.go:127`）で、継続/矛盾/クラスタ/問いに対応する豊かな `SurfaceScoreInputs`（TopicOverlapCount / VersionDriftCount / HasAugurLink / RecentContinueActionCount …）を event log の純関数として算出している。**燃料はすでに作られている。**
- しかし `decideBucketV2`（`knowledge_loop_projector/decide_bucket_v2.go:163`）が、その関係セットを優先順位で **単一の `SurfaceBucket` enum に畳んで関係を捨てている**。これがデータ層の single-axis collapse であり、UI には bucket + why 一文しか渡らない。「関係を持っているのに捨てている」。
- `plannerVersionForResolver`（`score_resolver.go:94`）は resolver の **型** が non-null なら無条件で `"v2"` とタグ付けし、実 evidence の有無を見る `hasV2Signal()` は定義されているのに使われていない。**計器が「v2 planner 100%」と表示しながら実配置は全件 v1 fallback、という嘘が成立しうる。**
- projector / resolver path に `if port == nil { ... }` 型の silent fallback が残存し、producer 未配線と「意図的に無効」を区別できない（[[000928]] と同じ failure mode）。

producer 自体は実在する（recap-worker が `recap.topic_snapshotted.v1`、rag-orchestrator が `augur.conversation_linked.v1` を emit）。したがって不足は「アーキの不在」ではなく、「**精巧な evidence が単軸に潰され、UI 上で方向づけの面にならず、計器がその劣化を隠している**」ことである。

## Decision

Knowledge Loop を、システムが既に算出している evidence を OODA 忠実な体験へ変換する**関係ループ**として再設計する。backend の建て直しではなく、「潰すのをやめて関係を表層まで運ぶ」ことを構造的核とする。具体的に以下を採用する。

1. **Orient = 関係的 synthesis 面を採用する。** Orient は「進む stage」ではなく always-present な面とし、entry をユーザーの既存 orientation に situate する関係（継続 / 矛盾・差分 / クラスタ / 問い）を提示する。「なぜこれが *あなたに* 今関係するか」を示す。
   - 却下した代替: 判断フレーミング（for/against のみ）/ 問い中心（inquiry のみ）。いずれも Boyd の orientation = 既存 orientation への situate という核を部分的にしか満たさない。関係的 synthesis を主とし、for/against と問いは関係の一属性として内包する。

2. **関係セットを projection の主軸とし、bucket を 1 レンズに降格する。** projection（`knowledge_loop_entries`）を `surface_bucket:enum + why:1` から **`relations:[]Relation`** へ拡張する。`decideBucketV2` は「配置の権威」を失い、関係セットから 1 つの bucket ラベル（now/continue/changed/review）を導くレンズ要約役に降格する（削除はしない）。
   - 却下した代替: bucket を spine のまま中身を濃くする案（「単軸に見える」表層が残りリスク）/ 単一優先スタックへ統合する案（changed/review の独立性が薄れる）。

3. **状態を observe / act の 2 状態に簡素化する。** Orient/Decide を「stage」から退場させ、Orient は常在の synthesis 面に、Decide は暗黙（どの関係を追うか選ぶこと）に解消する。`knowledge_loop.oriented` / `knowledge_loop.decision_presented` 遷移と session_stage projection は retire する。イベントは append-only で log に残るが projection が stage を導出しなくなる（reproject-safe）。OodaPipeline の 4 グリフは作り直す。
   - 却下した代替: stage を受動 byline に退かせる案（既に passive なのに stepper と感じられているため、保持だけでは治らない）/ UI のみ手入れ（「OODA の理念に則っていない」根が残る）。

4. **ループの閉じ方 = 関係の状態遷移を戻りで見せる。** ユーザーが関係を選んで act した結果を、次の observe で当該関係の状態 diff として見せる（矛盾→解消、スレッド→進展、クラスタ→拡張、問い→回答）。append-only event → reproject と整合し、`[[PM-2026-039]]` の invalidate storm を避ける per-entry diff 方式を採る。
   - 却下した代替: macro digest（行動と帰結の紐付けが弱い）/ ライブ再 orientation アニメーション（PM-2026-039 の fetch 暴走再現リスクが高い）。

5. **最初の垂直スライス = Continuation 1 関係種を貫通 + fail-loud。** 過去 15 回が「片脚だけ建てた」「silent fallback が未配線を隠した」ために durable にならなかった反省から、最初のスライスは両脚が小さな面で同時に立つことを証明する形にする。Continuation（継続スレッド）1 関係種だけを evidence → relation-set projection → Orient render → act → 戻り diff まで貫通させ、同時に (a) silent fallback を loud log + panic（[[000928]] の root cause 対策、CLAUDE.md #8）に、(b) 嘘の v2-ratio を「≥1 実関係を持つ entry の割合 = relation coverage %」という正直な SLI に置換、(c) 新 projection と producer に CDC RED test を先行（CLAUDE.md #7）させる。証明後に他の関係種へ scale する。
   - 却下した代替: 全関係種の projection を先に建てる backend-first（「proto+projection 完了、体験が降りない」歴史的失敗パターン）/ stub データの UI-first prototype（「燃料の無い美しい Orient」リスク）。

immutable invariants（append-first / reproject-safe / disposable projection / why first-class / event-time purity）は全て保持する。relations は event log の純関数であり、reproject で bit-for-bit 再生される。

## Consequences

### Pros

- **OODA の理念が体験に降りる。** Orient が schwerpunkt として常在し、Decide が関係選択として暗黙化し、ループが「閉じる」戻りを持つことで、4 つの体験的欠落が同時に解消される。
- **既存の燃料を活かす。** resolver が既に算出している `SurfaceScoreInputs` をそのまま relation-set の素材にできるため、evidence 機構の建て直しが不要。
- **single-axis collapse をデータ層で根絶する。** 関係を畳まず保持することで、UI が「装飾された feed」から「関係グラフの面」へ変わる。
- **attempt #16 を構造的に回避する。** 1 関係種の垂直貫通 + fail-loud + CDC RED により、両脚が同時に立つことを小さな面で先に証明する。silent fallback の loud 化と正直な relation coverage SLI で、劣化が計器で見えるようになる。
- **reproject-safe を崩さない。** relations は純関数、retire した stage イベントは log に残るため、過去状態は再生可能。

### Cons / Tradeoffs

- **projection schema が肥大する。** `surface_bucket:enum` から `relations:[]Relation` への拡張は migration を要し、read path / proto / UI を横断して変わる。hard-to-reverse。
- **stage マシン retire の移行コスト。** oriented/decision_presented 遷移と session_stage を projection から外す際、これらに依存する既存 UI / metrics / E2E を洗い替える必要がある。
- **未決の実装サブ決定が残る。** (a) `Relation` 型の正確な shape（kind / target_ref は versioned ID / magnitude / evidence_refs / why / 戻り用 state）、(b) honest SLI の正確な定義と閾値、(c) reproject migration の手順、は本 ADR では確定せず実装 PR で詰める。
- **戻り diff の鮮度依存。** per-entry の状態 diff は projector tick と producer emit の遅延に依存するため、「戻りが見えるまでの体感遅延」を SLI で監視する必要がある。
- **decideBucketV2 を残すことの認知負債。** 配置権威を剥がしてもコードは残るため、「bucket がまだ placement を決めている」という誤読を招かないよう、降格をコメント / metric label で明示する必要がある。
