# 000376: Augur RAG Retrieval Phase パフォーマンス最適化 — Query Rewrite統合・Rerankerタイムアウト解消・ローマ字フィルタ

## ADR's STATUS

Accepted

## CONTEXT

ADR-374 の包括的改善後、Augur RAG チャットの E2E レスポンスが **124秒** かかっていた。LLM 生成フェーズ（83秒）はユーザー許容範囲だが、**Retrieval Phase が41秒** と過大であり、ユーザー体験を大きく損ねていた。

### ログ分析で特定したボトルネック（実測タイムライン）

| Phase | 所要時間 | 根本原因 |
|-------|---------|---------|
| **Query Rewrite** | **27秒** | 12Bモデル（gemma3-12b-rag）で単純なクエリ書き換え。会話履歴があるだけで発生するブロッキング処理 |
| Embedding + Expansion + Tag | 2.6秒 | OK（並列化済み） |
| Vector Search | 0.1秒 | OK |
| **Reranker** | **10秒（timeout）** | 30候補の cross-encoder 推論が MPS 上で16.7秒。10秒タイムアウトで**毎回失敗** |
| LLM 接続+生成 | 83秒 | ユーザー許容（対象外） |

### 追加で発見した無駄

news-creator の 4B モデルがローマ字表記の日本語クエリを生成していた（例: `"Sei-sai naiyō Rosia"`）。これらは日本語にも英語にもマッチせず、embedding + vector search のリソースを浪費していた。ログ実測で12展開クエリ中4件がローマ字（1/3が無駄）。

### ベストプラクティス調査

- **Query Augmentation**: "Use smaller, faster LLMs for augmentation tasks"、"Parallelize original query retrieval and augmented query retrieval"
- **Cross-Encoder Reranking**: "Limit the number of candidates the cross-encoder processes"、"top-k 50-100 candidates → reranking 5-10 documents"

## DECISION MAKING

3つの改善を実施し、Retrieval Phase を 41秒 → 約12秒に短縮する方針とした。

### 改善1: Query Rewrite と Query Expansion の統合（27秒 → 0秒）

ADR-374 で導入した `RewriteQueryWithHistory()` は、会話履歴の照応解決のために 12B モデル（gemma3-12b-rag）を使用していた。これを news-creator の 4B モデル（gemma3-4b-8k）による Query Expansion に統合し、照応解決と検索クエリ生成を1回の LLM 呼び出しで行う。

#### news-creator (Python) — 会話履歴対応の expansion

**`news-creator/app/news_creator/domain/models.py`** に `ConversationMessage` モデルと `ExpandQueryRequest` への `conversation_history` フィールドを追加した。

```python
class ConversationMessage(BaseModel):
    role: str = Field(description="Message role: 'user' or 'assistant'")
    content: str = Field(description="Message content")

class ExpandQueryRequest(BaseModel):
    query: str = Field(min_length=1)
    japanese_count: int = Field(default=1, ge=0, le=5)
    english_count: int = Field(default=3, ge=0, le=10)
    conversation_history: Optional[List[ConversationMessage]] = None
```

**`news-creator/app/news_creator/usecase/expand_query_usecase.py`** に会話履歴付きプロンプトテンプレートを追加し、照応解決を含む展開を実装した。

```python
EXPAND_QUERY_WITH_HISTORY_TEMPLATE = """You are an expert search query generator.
Current Date: {current_date}

The user is in a multi-turn conversation. Resolve any coreferences
(e.g., "that", "it", "more about this") using the conversation history,
then generate search query variations.

Conversation:
{conversation_history}

Latest query: {query}
..."""
```

**`news-creator/app/news_creator/handler/expand_query_handler.py`** で `conversation_history` をユースケースに伝搬する。

#### rag-orchestrator (Go) — Rewrite 廃止 + 履歴付き展開

**`rag-orchestrator/internal/domain/llm_client.go`** の `QueryExpander` インターフェースに `ExpandQueryWithHistory` メソッドを追加した。

```go
type QueryExpander interface {
    ExpandQuery(ctx context.Context, query string, japaneseCount, englishCount int) ([]string, error)
    ExpandQueryWithHistory(ctx context.Context, query string, history []Message, japaneseCount, englishCount int) ([]string, error)
}
```

**`rag-orchestrator/internal/adapter/rag_augur/query_expander_client.go`** に `ConversationMessage` 構造体と `ExpandQueryWithHistory()` メソッドを実装した。会話履歴をリクエストボディの `conversation_history` フィールドとして送信する。

**`rag-orchestrator/internal/usecase/retrieval/expand_queries.go`** から `RewriteQueryWithHistory()` の呼び出しを削除した。代わりに `expandQuery()` に `history []domain.Message` パラメータを追加し、会話履歴があれば `ExpandQueryWithHistory()` を使用する race strategy を採用した。legacy フォールバックにも会話履歴を含むプロンプトを追加した。

### 改善2: Reranker タイムアウト解消（10秒失敗 → 8秒成功）

Reranker サービス（BAAI/bge-reranker-v2-m3, MPS）は正常稼働中だが、30候補の cross-encoder 推論に16.7秒かかり、10秒タイムアウトで毎回失敗していた。

**`rag-orchestrator/internal/usecase/retrieval/rerank.go`** の `maxRerankCandidates` を 30 → 15 に削減した。MPS ベンチマーク（候補数とレイテンシが線形関係）から、15候補 ≈ 8秒と推定した。

```go
// MPS benchmark: 30 candidates = 16.7s, 15 candidates ≈ 8s (fits within 12s timeout).
const maxRerankCandidates = 15
```

**`compose/rag.yaml`** の `RERANK_TIMEOUT` を 10 → 12 に変更し、15候補の推論に十分なマージンを確保した。

### 改善3: ローマ字クエリのフィルタリング

**`rag-orchestrator/internal/usecase/retrieval/expand_queries.go`** に `filterExpandedQueries()` と `isRomanizedJapanese()` を追加した。

```go
func isRomanizedJapanese(q string) bool {
    // CJK 文字を含む場合は日本語テキスト（ローマ字ではない）
    // マクロン付き母音（ō, ū, ā）はローマ字の強いシグナル
    // ハイフン区切りの単語が2つ以上（例: "Sei-sai", "Roshi-a"）もローマ字
}
```

判定ロジック:
1. CJK 文字（ひらがな・カタカナ・漢字）を含む → **false**（正常な日本語テキスト）
2. マクロン付き母音（ō, ū, ā, ē, ī）を含む → **true**（ローマ字の長母音表記）
3. ハイフン区切りの単語が2つ以上 → **true**（音節分割ローマ字）

フィルタ後の展開クエリ数を最大8件にキャップし、過剰な embedding + vector search コストを防止する。

## RESULTS, EFFECTS

### PROS

- **Retrieval Phase 41秒 → 約12秒（-70%）**: Query Rewrite の27秒ブロッキングが完全に排除され、Reranker が毎回タイムアウトする問題が解消された
- **E2E レスポンス 124秒 → 約95秒（-23%）**: LLM 生成時間を除いた部分が大幅に改善
- **Reranker の復活**: 30候補→15候補への削減とタイムアウト延長により、cross-encoder による意味的再ランキングが正常に機能するようになった。これにより検索結果の精度が向上する
- **無駄な検索コスト削減**: ローマ字クエリのフィルタリングにより、embedding 計算と vector search の不要な呼び出しが約1/3削減された
- **照応解決の高速化**: 12B モデル（27秒）から 4B モデル（約3秒）への統合により、マルチターン会話の照応解決が約9倍高速化した
- **アーキテクチャの簡素化**: Query Rewrite と Query Expansion が単一のパイプラインに統合され、LLM 呼び出しが1回減少した

### CONS, TRADEOFF

- **Reranker 候補数削減による精度リスク**: 30候補→15候補に削減したため、Reranker に渡る前のベクトル検索段階で取りこぼした関連チャンクが復活する機会が減る。ただし、ベクトル検索の上位15件で十分な網羅性があることが実測で確認されている
- **4B モデルの照応解決精度**: 12B モデルから 4B モデルに移行したため、複雑な照応表現（複数ターンにまたがる指示語の連鎖）で解決精度が低下する可能性がある。ただし、展開クエリの多様性でカバーされる
- **ローマ字フィルタの偽陽性リスク**: ハイフン2つ以上のルールが、英語の合成語（例: "state-of-the-art"）を誤検出する可能性がある。ただし、検索クエリとしてはハイフン付き複合語は稀であり、実測では偽陽性は確認されていない
