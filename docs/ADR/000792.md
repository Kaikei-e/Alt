# ADR-000792: pre-processor に articles.language 一括 backfill CLI を追加し ADR-776 配管の空振りを解消する

## Status

Accepted

## Date

2026-04-19

## Affected Services

- **pre-processor (CLI)** — `cmd/backfill-language/main.go` 新規。一発実行のバイナリとして分離し、通常の常駐プロセスには触らない
- **pre-processor (service)** — `service/language_backfill.go` に `LanguageBackfiller` を追加。既存 `service.DetectLanguage` を流用
- **pre-processor (repository)** — `repository/articles_language_repo.go` 新規。pgxpool 経由で articles に直接 UPDATE する

## Context

[[000776]] で `articles.language` カラムと ingestion 時の language_detector が導入され、[[000777]] で Curator の soft quota と Writer の `_CROSS_LINGUAL_RULE` が入った。ところが 2026-04-19 朝のレポート (`eb67863f-f6f0-48a4-bd76-a9f1d9a79054` イラン情勢分析レポート 2026年4月19日) は citations が 4 件とも日本語ソースで、英語引用はゼロだった。

DB 実査で 3 段のギャップが見えた:

1. migration `20260419000000_add_language_to_articles.sql` が `atlas.sum` に載っているのに本番 DB に未適用。`articles.language` カラムが存在しない
2. 既存記事 48,795 件すべてが migration 適用後も `'und'` のまま (新規 ingestion しか language を書かない)
3. pre-processor コンテナが 2026-04-17 ビルドで `service/language_detector.go` コード不在

結果として `rebalance_by_language()` の pool が常に空 en で no-op、Writer が `[Sn][en]` quote を受け取れない、ADR-777 の decision がランタイムで空振りしていた。

Migration は `migrate` compose サービス経由で適用すれば済む。残るのは既存 48,795 行の一括言語付与で、これを専用ジョブとして起こす必要があった。Acolyte orchestrator 側で個別問い合わせに差し込む形は「select-then-update」の race と API ホットパスの肥大化を招く。アプリが持つ言語判定ロジックを OSS 側と共有したいので SQL の CASE 式で済ませる選択も採らない。

## Decision

pre-processor に単発の管理 CLI `backfill-language` を新設する。既存の常駐プロセスとは分離し、本番適用は「migration apply → CLI 1 回実行」の 2 ステップで済むようにする。

- **Binary 分離**: `cmd/backfill-language/main.go` の独立 main。pre-processor main.go の healthcheck 分岐とは別。`CGO_ENABLED=0` で静的リンクし、docker cp で本番 DB 疎通経路 (alt-db の Unix socket か pgbouncer) に運んで一発で走らせる
- **Fetch は cursor paginate**: `SELECT id::text, title, content FROM articles WHERE language='und' AND deleted_at IS NULL AND id > $1::uuid ORDER BY id LIMIT $2`。空 afterID の時だけ WHERE 節を丸ごと外す別クエリにしてパラメータ型エラーを回避
- **UPDATE は CASE 式 + 防御 WHERE**: `UPDATE articles SET language = CASE id WHEN $1::uuid THEN $2 WHEN $3::uuid THEN $4 ... END WHERE id IN (...) AND language = 'und'`。`language = 'und'` 条件を重ね掛けすることで、新規 ingestion が並走して確定値を書き込んだ行を上書きしない idempotent な UPDATE になる。再実行は常に安全
- **Title-only detection**: 当初は `title + "\n" + content` で判定していたが、articles.content は HTML 断片 (`<div>`, `<p>`, `href="..."` などの Latin 文字) を保持しているため英語文字数が水増しされ、一部の日本語ニュース記事が en に誤分類された (2026-04-19 時点で title+content: 35,585 en / 13,210 ja → title-only: 35,155 en / 13,730 ja)。title は HTML サニタイズ済みの短文なので、detection は title のみに変更した
- **Throttle は注入可能**: `LanguageBackfillConfig.Sleep func(time.Duration)` を持ち、本番は `time.Sleep`、テストはフェイク。既定 100ms × 97 batch でおよそ 30 秒で完了する
- **Graceful shutdown**: `signal.NotifyContext` で SIGINT/SIGTERM を受け、in-flight batch を完了してから抜ける。UPDATE は transaction 単位で閉じているため、途中キャンセルしても half-written 行は残らない
- **CLI は stdlib flag のみ**: 余計な依存を増やさず、`--dsn`、`--batch-size`、`--throttle-ms`、`--dry-run`、`--resume-from-id`、`--connect-timeout`、環境変数 `ARTICLES_DB_DSN` / `DATABASE_URL`。`--dry-run` は SELECT + detect だけで UPDATE を発行しない (summary の `would_update` カウントで検算)

### 検討した代替案

| 案 | 採用 | 却下理由 |
|---|---|---|
| **pre-processor に独立 CLI**（採用） | ○ | 既存の DetectLanguage を再利用でき、本番ホットパスに影響なし。再実行の idempotency が WHERE 句だけで成立する |
| alt-backend に admin API を追加してオンライン適用 | × | ホットパスの lock と retry 設計が増える。48k 行の一回作業のために永続エンドポイントを増やさない |
| 純 SQL (`UPDATE ... SET language = CASE WHEN title ~ '[あ-んア-ンー一-龠]' THEN 'ja' ELSE 'en' END`) | × | Go 側の判定ロジックとドリフトする。`feedback_no_sql_logic` にも反する |
| 既存記事を `und` のまま放置し、新規 ingestion のみ語彙化 | × | 全記事 48,795 件のうち 48,795 が und では Curator の pool に en が入らず quota が永久 no-op。ADR-777 の effective scope が空になる |
| Title + content 判定を維持 | × | HTML 混入で en に誤分類される。title-only に切替後も Rails / React 技術タイトルが en に入るが、これはクロスリンガル RAG のユースケースでは許容範囲 (英語クエリから日本語の tech 記事を拾う動作を損なわない) |

## Consequences

### Pros

- migration を apply するだけで ADR-777 の Curator quota と Writer cross-lingual rule が実効的に動き出す。本番 eval で `lang_en_share_mean` が 0.167 (Phase 0 完了時点の db-replay) から、embedding 稼働後の hybrid search + quota によって向上する見込み
- CLI は一度走らせれば DB が十分な状態になるため、後続のデプロイで繰り返す必要はない (migration ↔ CLI の 2 ステップ runbook)
- TDD で書いた 19 テスト (dry-run、resume、idempotent、throttle、context cancel、repo fetch/update 防御、CLI flag parse) が全 green。ruff / go vet / race 検出もクリーン
- `--dry-run` が事前検算を支える。48,795 件を 27 秒で scan し、language 分布 (en 35,585 / ja 13,210 / und 5) を JSON 出力する

### Cons / Tradeoffs

- **CLI の実行は本番 DB への直接接続**を前提にしているため、接続パスワードの取り回しが必要。開発環境の `secrets/db_password.txt` と本番で deploy runner が配布する secret とが差分を持つことが判明し、今回の運用では alt-db コンテナの Unix socket (`postgres:///alt?host=/var/run/postgresql&user=alt_db_user`) 経由で `docker exec` から一発走らせた。runbook に手順を残す
- **title-only detection は技術ブログ系の code-switch タイトル (「Rails.logger をJSONで出力する」等) が en に流れる**。クロスリンガル RAG の観点では「日本語技術ブログを英語クエリから拾う」挙動は逆に有用だが、集計値としての ja/en 比率は Japanese-dominant ではなくなる (en 72% / ja 28%)。受容し、運用側で把握する
- **CLI は pre-processor バイナリとは別の main** なので、Docker image には後続で `cmd/backfill-language` を組み込む必要がある。今回は `go build` した static binary を docker cp で持ち込んで 1 回走らせた。恒常的な使い勝手にするなら Dockerfile での成果物追加が次手
- **cursor は memory 非永続**。途中で落ちた場合は `--resume-from-id` の値を手で渡して再開する。`WHERE language = 'und'` が残っていれば再実行は idempotent なので、resume 忘れでも最悪やり直せる
