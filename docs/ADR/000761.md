# ADR-000761: ListRecapArticles を BackendInternalService に移設し c2quay 運用課題を明記する

## Status

Accepted

## Date

2026-04-17

## Affected Services

- **alt-backend** — `proto/alt/recap/v2/recap.proto` から `ListRecapArticles` と関連 message を削除し、`proto/services/backend/v1/internal.proto` の `BackendInternalService` に移設。`app/connect/v2/recap/handler.go` から RPC メソッドを除去、`app/connect/v2/internal/handler.go` に `WithRecapArticlesUsecase` HandlerOption と `ListRecapArticles` メソッドを新設。`app/connect/v2/server.go` の internal handler 登録で `WithRecapArticlesUsecase(container.RecapArticlesUsecase)` を追加。`app/driver/contract/provider_test.go` の mock を新 path に差し替え
- **recap-worker** — `recap-worker/recap-worker/src/clients/alt_backend.rs` の `LIST_RECAP_ARTICLES_PATH` を `services.backend.v1.BackendInternalService/ListRecapArticles` に更新、`RecapArticlesResponse` の数値・bool・Vec に `#[serde(default)]` 付与、`alt_backend_contract.rs` の Pact を新 path で再契約
- **alt-butterfly-facade / rag-orchestrator / alt-frontend-sv** — `buf generate` による proto stub 再生成差分のみ (挙動変更なし)
- **scripts / compose** — 過去 commit で混入した ADR/PM 番号コメントを全削除

## Context

[[000760]] で `/v1/recap/articles` REST を Connect-RPC `alt.recap.v2.RecapService/ListRecapArticles` に移行した直後の実行時検証で、recap-worker が **`HTTP 401 unauthenticated: missing backend token`** で新 RPC を通過できない事実が判明した。原因は `RecapService` が `alt-backend/app/connect/v2/server.go:52-59` で構成される既定の interceptor chain (`cancelInterceptor` + `authInterceptor` + `otelInterceptor`) で登録されており、`authInterceptor` が `X-Alt-Backend-Token` ヘッダ (JWT) を必須化している点にあった。一方 `BackendInternalService` は同ファイル `:200-204` で **`cancelInterceptor` のみ** を含む `internalOpts` で登録され、*"Auth is established at the TLS transport layer (mTLS peer-identity)"* と明記してある。service-to-service の RPC は後者に属する場所になければならず、[[000760]] は namespace を誤っていた。

また同じ deploy サイクルで **c2quay の運用上の落とし穴** を 3 件踏んだ:

1. `docker compose up --wait --remove-orphans` は Dockerfile / ソースが変わっても image を再ビルドしない。deploy を 3 回流しても alt-backend と recap-worker は OLD バイナリのまま稼働し続けた
2. image を明示的に `docker compose build` しても `up --wait` は config 差分のみを見て container を touch しない。`up -d --no-deps --force-recreate` を追加で叩かないと新バイナリが起動しない
3. Pact Broker の `DeployedOrReleased` セレクタが旧 path (`/alt.recap.v2.RecapService/...`) を記録し続け、新 provider が 404 を返して deploy gate が fail する。[[000758]] で `--skip-verify` は廃止したため迂回路が無い

これらは contract の互換性ではなくインフラ挙動の話だが、**今回の 2 度の migration で同じ手順を 3 回繰り返す** 結果になったため、次回の breaking migration を控えて決定として残す必要があった。

副次タスクとして、過去 4 commit (`e77039ec5` / `254144071` / `e70e15476` / `55507ac47`) で私が混入させた ADR / PM 番号コメントが `scripts/deploy.sh` / `scripts/cascade-pki-sidecars.sh` / `compose/pki.yaml` / `alt-backend/app/rest/routes.go` / `proto/services/backend/v1/internal.proto` に散っていた。`No issue numbers in code` のプロジェクトルール違反であり、本 ADR の一部として一括除去した。

## Decision

### 1. proto を `services/backend/v1/internal.proto` に移設

`alt/recap/v2/recap.proto` から `ListRecapArticles` と 5 message を削除し、`services/backend/v1/internal.proto` の `BackendInternalService` に `rpc ListRecapArticles(ListRecapArticlesRequest) returns (ListRecapArticlesResponse);` を追加。ファイル末尾に `ListRecapArticlesRequest` / `RecapArticleRange` / `RecapArticleItem` / `ListRecapArticlesResponse` を配置。`RecapArticleTag` は [[000760]] で consumer pact から外しており実装でも使っていなかったため本 ADR で廃止。`cd proto && buf generate --template buf.gen.backend-internal.yaml` (加えて default `buf generate`) で Go / alt-frontend-sv / rag-orchestrator / alt-butterfly-facade の stub を再生成。

### 2. Go handler を internal package に実装

`alt-backend/app/connect/v2/internal/handler.go` の `Handler` 構造体に `recapArticlesUsecase` フィールドを追加、同ファイル内に以下を配置:

- `recapArticlesUsecase` interface (最小面: `Execute(ctx, recap_articles_usecase.Input) (*domain.RecapArticlesPage, error)`)
- `WithRecapArticlesUsecase(uc)` HandlerOption
- `ListRecapArticles(ctx, req)` メソッド (旧 REST のバリデーション + 既存 `RecapArticlesUsecase` への委譲をそのまま Connect-RPC 形にした)

`alt-backend/app/connect/v2/server.go:217` の `internalhandler.NewHandler(...)` 呼び出しに `internalhandler.WithRecapArticlesUsecase(container.RecapArticlesUsecase)` を 1 行追加。`BackendInternalService` の登録側 interceptor chain は無変更 (`cancelInterceptor` のみ) なので `ListRecapArticles` は自動的に auth-free になる。

`alt-backend/app/connect/v2/recap/handler.go` からは `ListRecapArticles` / `recapArticlesToProto` / `articlesUsecase` フィールドを削除し、`recap` package は user-auth 域の RPC だけを保持する。

### 3. Rust client の path 更新と protojson 耐性強化

`recap-worker/recap-worker/src/clients/alt_backend.rs`:

- `LIST_RECAP_ARTICLES_PATH` を `services.backend.v1.BackendInternalService/ListRecapArticles` に差し替え
- `RecapArticlesResponse` の `total` / `page` / `page_size` / `has_more` / `articles` に `#[serde(default)]` を付与。protojson は zero-valued なフィールド (`has_more: false`, `total: 0`, 空 `articles: []`) を JSON から省略するため、厳格な struct 定義だと `missing field hasMore` で deserialize 失敗する。実測で alt-backend が `hasMore` を省略した 5.9 MB 応答を返したときに踏んだ
- unit test の `RPC_PATH` 定数も新 path に更新

### 4. Pact consumer / provider を再契約

consumer `alt_backend_contract.rs` の request path / mock server path を `/services.backend.v1.BackendInternalService/ListRecapArticles` に変更し、`cargo test --lib "clients::alt_backend_contract::contract_alt_backend_recap_articles" -- --ignored` で `pacts/recap-worker-alt-backend.json` を再生成。provider `alt-backend/app/driver/contract/provider_test.go:startStubServer` の `mux.HandleFunc` パスを同じく新 path に差し替え、`go test -tags contract -run 'TestVerifyRecapWorkerContract'` で pass を確認。

### 5. コード内から ADR/PM 番号を除去

`No issue numbers in code` のプロジェクトルールに従い、過去の私の commit に含まれていた参照を以下で削除した。理由説明コメントは残すが識別子は git blame と `docs/ADR/` 側に委ねる:

| ファイル | 変更 |
|---|---|
| `scripts/deploy.sh` | `"netns-sharing sidecar cascade; closes PM-2026-030 gap"` → `"netns-sharing sidecar cascade"` |
| `scripts/cascade-pki-sidecars.sh` | file header の `"Closes the PM-2026-030 gap"` を同等の動作説明に書き換え |
| `compose/pki.yaml` (`pki-agent-tag-generator` / `pki-agent-acolyte-orchestrator` の `depends_on`) | `"the PM-2026-030 failure mode"` / `"the c2quay cutover (ADR-000758) removed"` / `"hit PM-2026-030 twice"` / 時刻参照などを抽象的な挙動説明に書き換え |
| `alt-backend/app/rest/routes.go` | `"ADR-000760 で Connect-RPC"` → `"Connect-RPC"` と path 名のみ |
| `proto/services/backend/v1/internal.proto` | 新設した `"── Recap Article Window (ADR-000760: recap-worker paginated fetch) ──"` → `"── Recap Article Window ──"` |

### 却下した代替案

1. **`alt.recap.v2.RecapService` に留めたまま `AuthInterceptor` に path-prefix skip list を追加** — interceptor に例外リストを抱えるとメソッドが増えるたびに編集が必要で、別レイヤ (proto namespace) との二重管理になる。保守性が低い
2. **`RecapService` を 2 回登録して user 向けと service 向けを同じ proto で分ける** — Connect-RPC の path は proto で一意に決まるため実装不可能。無理に外側でルーティングするのは hack
3. **[[000760]] を `git revert` して REST ハイブリッドに戻す** — 時間的には最短だが、[[000759]] Cons で予告した fast-follow を逆戻りさせるだけで技術的負債が再発する
4. **tonic / prost を Rust に導入して proto stub 経由にする** — zero-valued omission は protojson 自体の仕様で、生成 stub を使っても解決しない。既存 reqwest + serde 路線で `#[serde(default)]` が最小コスト

## Consequences

### Pros

- **recap-worker が mTLS peer-identity だけで auth を通過**。JWT を発行する必要がなく、`BackendInternalService` の既存 convention と完全一致
- **namespace 分離が完結**。`alt.*` = user-facing (BFF 経由 + JWT 必須)、`services.*` = service-to-service (mTLS のみ)。新しい RPC をどちらに置くかの判断基準が一本化された
- **protojson zero-valued omission への耐性が Rust 側で永続化**。今後追加する Connect-RPC でも同じ落とし穴を回避可能
- **コード内 ADR/PM 番号がゼロ**。ルール準拠、ADR 再番号付けや PM 付番変更時に grep miss しない
- **3days Recap が end-to-end で再び動く**。job 448 で fetch (951 記事取得) → preprocess → dedup まで進行を実測。以降の classification 失敗は recap-subworker 側の別下流障害で本 ADR の対象外

### Cons / Tradeoffs

#### C1: c2quay は image を再ビルドしない
`docker compose up --wait --remove-orphans` は既存 image タグだけを見るため、Dockerfile / ソース変更を自動検知しない。今回は deploy を 3 回流した後で初めて気づき、手動で `docker compose build <affected>` を前段に挟んで解決。恒久策は未実装 — `scripts/deploy.sh` 前段に pre-build step (例: `changed_services.sh` が git diff から変更 service を拾って `docker compose build` に渡す) を組み込む別 ADR で対応する前提。

#### C2: image tag 同一時に container を recreate しない
image を新しくビルドしても `docker compose up --wait` は config 差分ベースで recreate 判定するため、tag (`alt-alt-backend:latest`) が変わらない限り container を touch しない。`up -d --no-deps --force-recreate <svc>` を追加で叩く必要がある。cascade-pki-sidecars と同列の hook として `force-recreate-changed-images` を `scripts/deploy.sh [5/5]` 相当に追加する案を別 ADR で起票。

#### C3: Pact `DeployedOrReleased` セレクタが breaking migration を詰まらせる
Consumer の Connect-RPC path を変更すると、Broker には旧 path の "Deployed" 契約が残り続ける。provider は新 path のみ提供しているので verify 落ち → deploy gate が fail。`--skip-verify` は [[000758]] で廃止済みのため迂回路が無い。`EnablePending: true` は新 pact 側だけに効き `DeployedOrReleased` 側は相変わらず blocking。運用回避として:

- (a) Broker 管理 UI で旧 deployment record を手動 expire
- (b) provider_test.go の `ConsumerVersionSelector{DeployedOrReleased: true}` を migration 中だけコメントアウトし、deploy 成功後に復元
- (c) Pact "branch" 機能で旧 branch を deprecate する公式フロー

現時点では (b) を一時運用、恒久策は runbook (`docs/runbooks/pact-broker-ops.md`) に手順として起票する前提。

#### その他
- **alt-backend `:9000` REST 面の削減は部分的**。`/v1/recap/articles` は消えたが `/v1/dashboard/*` `/v1/morning/*` `/v1/articles/*` などは残存。internal-only 化は別 ADR
- **`RecapArticleTag` を proto から削除した**。consumer pact からも外したが、将来 alt-backend が tags を返す仕様に戻す場合は proto を再度拡張する必要がある。現状の Rust 側 `AltBackendArticle.tags` は `#[serde(default)]` で空 Vec を受け取れるので upstream 互換は保たれる
