# ADR-000402: Knowledge Home の段階的公開を Feature Flag で制御する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — Feature Flag 判定、Handler ガード、Config 拡張
- alt-frontend-sv (SvelteKit) — Flag 状態の取得と条件分岐

## Context

Knowledge Home は Phase 1-2 でイベントストア・projector・UI の基盤が完成したが、全ユーザーへの公開にはデータ投影の安定性や UI の完成度を段階的に検証する必要がある。

既存の Alt にはフィーチャーフラグの仕組みがなく、新機能の公開は「デプロイ＝全ユーザー公開」になっていた。Knowledge Home のように projector lag やデータ量に依存する機能は、一部ユーザーで先行検証してからロールアウトする手段が不可欠である。

要件:
- **環境変数だけで制御可能** — コード変更なしにロールアウト比率を変更できること
- **ユーザー単位の判定** — 特定ユーザーのみに先行公開できること（allowlist）
- **パーセンテージロールアウト** — 段階的に公開比率を上げられること
- **3 つの独立したフラグ** — ページ表示・トラッキング・projection v2 をそれぞれ独立に制御できること

## Decision

**環境変数ベースの Feature Flag gateway を alt-backend に導入し、Connect-RPC レスポンスでフラグ状態をフロントエンドに伝搬する方式を採用する。**

### 設計判断

1. **外部サービス（LaunchDarkly 等）は不採用** — Alt は単一テナント・セルフホスト前提であり、外部依存を増やす正当性がない。環境変数 + コンテナ再起動で十分な制御粒度を得られる。

2. **crc32 ハッシュによるパーセンテージ判定** — UserID のバイト列を crc32 でハッシュし、`hash % 100 < rolloutPercentage` で判定する。同一ユーザーは常に同じバケットに入るため、ロールアウト比率を上げた際に既存ユーザーの体験が変わらない（単調増加）。

3. **Allowlist 優先** — `KNOWLEDGE_HOME_ALLOWED_USER_IDS` に含まれるユーザーは rollout percentage に関係なく有効になる。開発者の先行検証に使う。

4. **フラグの粒度** — 3 本のフラグを独立に制御できるようにした:
   - `enable_knowledge_home_page` — GetKnowledgeHome API のアクセス制御
   - `enable_knowledge_home_tracking` — TrackHomeItemsSeen / TrackHomeAction の no-op 化
   - `enable_knowledge_home_projection_v2` — 将来の projection ロジック切り替え用

### 実装概要

**バックエンド:**

- `config.KnowledgeHomeConfig` に 5 つの環境変数を追加（`KNOWLEDGE_HOME_ENABLE_PAGE`, `_TRACKING`, `_PROJECTION_V2`, `_ROLLOUT_PERCENTAGE`, `_ALLOWED_USER_IDS`）
- `port/feature_flag_port.FeatureFlagPort` インターフェース — `IsEnabled(flagName string, userID uuid.UUID) bool`
- `gateway/feature_flag_gateway.Gateway` — Config + crc32 + allowlist で判定
- `GetKnowledgeHome` handler — フラグ無効時は `connect.CodePermissionDenied` を返す
- `TrackHomeSeenUsecase` / `TrackHomeActionUsecase` — フラグ無効時は no-op で即座に return
- レスポンスに `repeated FeatureFlagStatus feature_flags` フィールドを追加 — フロントエンドがフラグ状態を知るため

**フロントエンド:**

- `useFeatureFlags.svelte.ts` — レスポンスの `feature_flags` から状態を管理する Runes composable
- Home ページ — tracking flag off 時は `trackSeen` / `trackAction` 呼び出しをスキップ
- degraded mode 時は `DegradedModeBanner` コンポーネントを表示

**ロールアウト手順（コード変更不要）:**

| Step | 環境変数 | 対象 |
|------|---------|------|
| 1 | `ALLOWED_USER_IDS=<自分>`, `ENABLE_PAGE=true` | 開発者のみ |
| 2 | `ENABLE_TRACKING=true`, `ROLLOUT=100` | tracking 全員 |
| 3 | `ENABLE_PAGE=true`, `ROLLOUT=10` | 10% 公開 |
| 4 | `ROLLOUT=100` | 全公開 |

**Rollback**: フラグを `false` に戻してコンテナ再起動するだけ。

## Consequences

### Pros

- **ゼロダウンタイムロールアウト** — 環境変数変更 + 再起動のみで段階的公開が可能
- **ユーザー体験の一貫性** — crc32 の単調増加性により、ロールアウト拡大時に既存ユーザーの状態が変わらない
- **独立したフラグ** — ページ表示と tracking を別々に制御でき、tracking だけ先行して全ユーザーに有効化するなどの戦略が取れる
- **外部依存ゼロ** — LaunchDarkly 等の SaaS 不要、ネットワーク遅延もなし

### Cons / Tradeoffs

- **リアルタイム変更不可** — フラグ変更にはコンテナ再起動が必要（ホットリロードはない）
- **Knowledge Home 専用** — 汎用的なフィーチャーフラグ基盤ではなく、他機能への転用には拡張が必要
- **パーセンテージの分解能** — crc32 mod 100 のため 1% 刻み。十分だが 0.1% 刻みはできない
