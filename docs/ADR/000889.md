# ADR-000889: Morning Letter プロンプトを producer 側で cap して LLM 出力経路を文脈窓に収める

## Status

Accepted

## Date

2026-05-04

## Affected Services

- **recap-worker (Rust)** — `recap-worker/recap-worker/src/pipeline/morning.rs` に producer-side budget cap を追加。
  - `MAX_RECAP_GENRES_FOR_PROMPT = 8`、`MAX_BULLETS_PER_GENRE_FOR_PROMPT = 2`、`MAX_BULLET_CHARS_FOR_PROMPT = 200`、`MAX_OVERNIGHT_GROUPS_FOR_PROMPT = 40` を module 定数として固定。
  - 新ヘルパ `cap_recap_inputs_for_prompt(Vec<MorningLetterRecapInput>) -> Vec<MorningLetterRecapInput>` と `cap_overnight_groups_for_prompt(Vec<MorningLetterGroupInput>) -> Vec<MorningLetterGroupInput>` を `execute_update` から呼び出し、`MorningLetterGenerateRequest` に詰める前にキャップする。
  - bullet trim 後に空になった genre は自動 drop。短い payload は no-op で素通り。`#[cfg(test)] mod tests` に 5 本のユニットテスト (`cap_recap_inputs_for_prompt_keeps_top_genres_and_truncates_bullets` / `cap_recap_inputs_preserves_short_payloads_unchanged` / `cap_recap_inputs_drops_genre_entirely_when_no_bullets_survive` / `cap_overnight_groups_for_prompt_limits_count` / `cap_overnight_groups_preserves_short_inputs`) を追加し、合計 24 本の morning pipeline test、349 本の lib test 全体が green、`cargo clippy --lib --all-targets -- -D warnings` も clean。

## Context

[[000888]] で Morning Letter pipeline を `recap_outputs.title_ja` / `bullets_ja` に grounding しなおし、production の 5/4 row が「TechFusion / Nova Labs」のような幻覚 lead を返していた問題を直した。grounding 自体は意図通りに動き、recap-worker のログには `loaded recap summaries for morning letter genre_count=24` が出るようになり、news-creator が `recap_summary_count=24` を受け取るようになった。

ところが deploy 直後の morning daemon tick (2026-05-04 09:15 UTC) で `morning_letters` row は `model=extractive-fallback`, `is_degraded=true`, `degradation_reason="pydantic_validation: 1 validation errors: [{'type': 'missing', 'loc': ('generated_at',), 'msg': 'Field required', 'input': {'lead': \")', \", 'sections': [{'key': 'lead', 'title': \")', \", 'bullets': ['... (andoneoneoneoneoneone...']}]}` となっていた。lead は無事 `recap_outputs.ai_data` の実プロース（「著名研究者ヤン・ルカン氏らが主導する次世代AI研究…」）に置き換わっていたが、これは `_build_extractive_fallback` ([[000706]]) が recap context をそのまま流用して生成したもので、LLM polished narrative ではない。

news-creator のログを照合すると原因は明確だった:

```
Generating with Ollama: model=gemma4-e4b-12k,
  prompt_length=32530 chars, estimated_tokens=8132,
  num_predict=1200, context_window=8192,
  usage_percent=99.3%
```

`gemma4-e4b-12k` の context_window は 8192 token。プロンプト単体で 8132 token を消費し、出力に許される token は 60 弱。`num_predict=1200` を要求しているのに物理的に書ききれず、モデルは `'lead': ")'`、`'bullets': ['...andoneoneoneoneoneone...']` のような degenerate な反復を返した。Pydantic validator が `generated_at` field 欠落で reject → `_fallback_with_reason("pydantic_validation", ...)` 経路で extractive fallback ([[000706]]) に逃避。

プロンプトが 32K chars に膨れた直接の原因は producer 側の素通し設計だった:

- recap_summaries: [[000888]] 後は 24 genres を全件渡している。`recap_outputs.bullets_ja` は recap reduce 段が編集した完全プロース (各要素 600〜900 chars) で、24 × 2-3 bullets で容易に 50K chars 級になる。
- overnight_groups: dedup pipeline が 200〜250 グループを emit するのを丸ごと渡していた (`groups_count=257` を観測)。

LLM 観点の容量計算:

| 区分 | キャップ前 (5/4 09:15 観測) | 出力余地 |
|---|---|---|
| prompt | 32530 chars / 8132 token | 60 token |
| 必要な出力 | `num_predict=1200` | 不足 1140 token |

[[000888]] で grounding を直す前は `recap_summaries=None` なので prompt は overnight 主体で〜16K chars 程度に収まっており、LLM は出力できていた (ただし context が無いため幻覚していた)。grounding を直すと recap 側が肥大して overflow するという、対称な引き当て関係になっていた。

producer 側 (recap-worker) でキャップを掛けるのが筋だと判断した。consumer (news-creator) は `MorningLetterGenerateRequest` の正当性に対して責務を持つが、payload サイズ制御は文脈窓を所有する LLM 側に近い recap-worker 側のほうが見通しが良い。news-creator は同じ Ollama model を他の経路でも使うため、Morning Letter 固有のキャップを news-creator に積むと cross-cutting concern が増える。

## Decision

`recap-worker/recap-worker/src/pipeline/morning.rs` に producer-side prompt budget cap を導入し、`execute_update` で `MorningLetterGenerateRequest` を組む直前に通す。

### 1. 定数で上限を固定する

```rust
const MAX_RECAP_GENRES_FOR_PROMPT: usize = 8;
const MAX_BULLETS_PER_GENRE_FOR_PROMPT: usize = 2;
const MAX_BULLET_CHARS_FOR_PROMPT: usize = 200;
const MAX_OVERNIGHT_GROUPS_FOR_PROMPT: usize = 40;
```

容量計算:

| 区分 | キャップ後上限 |
|---|---|
| recap section | 8 × 2 × 200 chars = 3,200 chars (≈ 800 token) |
| overnight section | 40 × ~150 chars = 6,000 chars (≈ 1,500 token) |
| 静的テンプレ | ~3,000 chars (≈ 750 token) |
| 合計プロンプト | ≈ 12K chars / 〜3,050 token |
| 出力余地 (8192 ctx) | 〜5,140 token (`num_predict=1200` を内包) |

8k context の半分以下に収まり、`num_predict=1200` を充足したうえで余白も残る。

### 2. キャップ関数を 2 本に分ける

`cap_recap_inputs_for_prompt` と `cap_overnight_groups_for_prompt` の責務を分離する。前者は genre / bullet 二重キャップに加え、bullet 全件が trim で空になった genre を `filter_map` で drop して phantom header を防ぐ。後者は単純な `take(N)`。`Vec` を所有権ごと受け取り所有権ごと返すことで、呼び出し側 (`execute_update`) でテンポラリ束縛を増やさず、テストも単純な vec を渡すだけで済む。

bullet 截断は `truncate_to_chars(s: &str, limit: usize)` で `chars().take(limit).collect()` 経由で行う。マルチバイト文字列で byte slice 截断したときに発生する panic と文字化けを避ける狙い。

### 3. degraded か否かに依存させない

キャップは `is_degraded` 判定の前段で常に通す。`recap_summaries` が `None` のときも、`overnight_groups` 単体で 200+ 件流れて news-creator のプロンプトを膨らませる事象が再発しうるので、片側だけキャップしても再発予防にならない。

### 検討した代替案

- **news-creator の prompt template (Jinja) でキャップする**: Morning Letter プロンプトの肥大は Morning Letter usecase 固有の問題なので news-creator を一見薄く保てるが、news-creator は同じテンプレを他経路 (recap_summary 等) でも使う傾向にあり、cross-cutting なキャップ制御を news-creator 側に置くと「どの経路がどのキャップ前提か」を分散管理することになる。recap-worker が「どのモデルに何を投げるか」を握っている本来の責務に揃えた。
- **Ollama model の context_window を 12k に拡張する**: `gemma4-e4b-12k` という名前が示唆するように、Modelfile を `num_ctx 12288` で更新すれば 8192 → 12288 に増やせる。しかし VRAM 消費が増え、`HybridPrioritySemaphore` ([[queue-saturation]]) 配下の他経路にも影響する。本件は producer 側で 3K token に収まるのでモデル側を触る理由が無い。別 ADR で全体方針として評価する余地は残る。
- **環境変数で外部設定可能にする**: 「次にチューニングしたくなったとき」を見越した拡張だが、実運用で 8 / 2 / 200 / 40 を細かく動かす場面を即座に思いつかない。設定軸を増やす前にこの定数値で十分かを観察する。再調整の必要が出たら module 定数 → env / config に昇格する。
- **bullet 截断ではなく要約 LLM call を挟む**: 「太いプロースを 200 chars に縮める」を別 LLM call で正しい文章として書き直す案。明らかに過剰。recap reduce 段で既に編集済の bullet を機械的に切るだけで grounding 目的には十分。
- **MAX_BULLET_CHARS = 150 / 250 / 300 等の別バリアント**: 200 を選んだのは「2 文程度が収まる粒度」と「容量計算で 8k context の半分以下に収まる安全マージン」の交点として。観測しながら必要に応じて調整する。

## Consequences

### Pros

- LLM polished な Morning Letter narrative が復旧する。`is_degraded=false`, `model=gemma4-e4b-12k` で当日 row が UPSERT されるようになる。
- recap reduce 段が将来 bullet を更にリッチに書くようになっても、producer 側のキャップが文脈窓と LLM 出力を構造的に守る。回帰テストとして 5 本の単体テストが「上限を超える入力でも安全に切られる」こと、「短い入力は no-op で素通り」「全 bullet が空になる genre は drop」「短い overnight は素通り」を pin。
- `feedback_eval_first` (RAG eval-first) と同方向: 「LLM に渡す前に prompt 形を観測可能な定数で決める」ことで、症状が再発したときに「どの定数を動かせば良いか」が即座に分かる。
- 単体テスト 24 本 / lib 全体 349 本 / `cargo clippy -D warnings` 全て green。CI 経路の追加負担なし。

### Cons / Tradeoffs

- 24 genres のうち 8 genres しか LLM に届かない。Morning Letter は「日次の概観」であり 8 genres あれば十分という仮説に立つが、ジャンル数が増えてコンテンツの多様性が落ちる可能性は残る。`get_genres_by_job` が `ORDER BY genre` で alphabetical 順なので、頭文字が偏る曜日に同じ genre 群ばかり LLM に届くバイアスもある。後続改修で「日次 top-N genre 選定」を recap reduce 段の score を使って組み直す余地。
- bullet 200 chars 截断は文末を物理的に切るため、文末の括弧 / 丸付き番号 / `[zenn.dev]` 形式の citation が中途半端に欠ける場合がある。LLM が grounding 用に使うインプットなので可読性より「一意に識別できる事実の核」を残せていれば実用上問題ない、と判断したが、UI に直接見える経路ではないことを前提にしている。
- overnight cap の `40` は dedup pipeline の出力順 (Vec の先頭から `take`) に依存している。dedup の並び順が「重要度順」でない場合、ランダムに先頭 40 件を選んでいるのと等価になる。本 ADR の scope 外だが、後続で「top-N 選定」を入れる動機の一つ。
- 4 つの定数は将来 env / config に昇格する蓋然性が高いが、本件では module-internal に留めた。動機薄なまま設定軸を増やすと、他開発者から見て「触れる箇所」「触らない箇所」の判断コストが上がる。
- prompt 形が短くなる分、`num_predict=1200` を埋めようとして LLM が出力を冗長に膨らませる可能性が残る。出力側の挙動は morning_letter usecase ([[000706]]) の責務で、本 ADR では触らない。
