# ADR-000645: Augur RAG プロンプトビルダーを system role 分離に移行し instruction following を根本改善する

## Status

Accepted

## Date

2026-04-08

## Affected Services

- rag-orchestrator (Go)

## Context

Augur の応答が system prompt の指示に従わない問題が、12 以上の ADR（[[000607]]〜[[000644]]）による段階的改善にも関わらず根本解決しなかった。因果分析クエリに対して短文しか返らない、800 文字以上の要件が無視される、回答構造（直接要因/構造的背景/不確実性）が守られないなどの症状が継続していた。

### 根本原因の特定

Web リサーチと一次情報の調査により、以下の 4 つの根本原因を特定した:

1. **System role 不使用（最大原因）**: `XMLPromptBuilder` が全指示（役割定義・品質基準・出力形式・intent 固有ガイダンス）を単一の `user` メッセージに詰め込んでいた。Gemma 4 は `<|turn>system` トークンを特権的指示として訓練されているが、これを完全にバイパスしていた。同一コードベースの `morning_letter_prompt_builder.go` と `agentic_synthesis_strategy.go` は正しく `{Role: "system"}` を使用しており、Augur の RAG プロンプトビルダーだけが取り残されていた
2. **Multi-turn での system 再注入なし**: 2 ターン目以降、役割・品質基準・出力形式の指示が完全に消失。ACL 2024 の研究（arxiv 2402.10962）で 70B モデルでも 8 ラウンドで有意なペルソナドリフトが確認されている
3. **Lost in the Middle 効果**: ACL 2024（aclanthology.org/2024.tacl-1.9）で中間位置の情報に 30%+ の精度低下が確認されている。現在の構成では指示がロール定義とクエリの間に埋没していた
4. **Instruction sandwich なし**: 重要ルールの末尾再提示がなく、LLM の recency bias を活用できていなかった

### 先行調査

- Google AI 公式ドキュメント: Gemma 4 は native system role をサポートし、Ollama が `role: "system"` を `<|turn>system` トークンに自動マッピング
- arxiv 2411.10541: プロンプトフォーマットの違いだけで最大 40% のパフォーマンス差
- arxiv 2509.08182: XML タグ付きプロンプトがスキーマ遵守を改善する形式保証
- arxiv 2603.16152: 明示的な優先順位付け（Instruction Hierarchy）が遵守率を向上

## Decision

`XMLPromptBuilder` を `morning_letter_prompt_builder.go` パターンに合わせてリファクタリングし、LLM への指示を **system role** で送信する構成に移行する。

### 1. System + User メッセージ分離

**Single-turn**:
- `system` メッセージ: 役割定義、品質基準、回答構造、intent 固有指示、出力形式、few-shot example、instruction sandwich
- `user` メッセージ: コンテキストチャンク、クエリのみ

**Multi-turn**:
- `system` メッセージ: core instructions + follow-up ルール + instruction sandwich（**毎ターン再注入**）
- 中間: 会話履歴（user/assistant ターン、最大 6 メッセージ）
- `user` メッセージ: コンテキスト + クエリのみ

### 2. Instruction Sandwich

system メッセージ末尾に最重要 4 ルールを再掲:
- 日本語回答の強制
- コンテキスト限定（外部知識禁止）
- 具体的事実・データの包含
- ソース引用 `[番号]` の付与

### 3. Few-shot Example

system プロンプトに intent 別の 1 例を追加:
- 一般クエリ: 概要/詳細/まとめ構造の完全な JSON 例
- 分析的 SubIntent: 弱点分析の構造化 JSON 例
- 関連記事: ランク付きリスト形式の JSON 例

### 4. 新メソッドの追加

- `writeFollowUpRules(sb, input)`: 従来 `buildMultiTurn` 内にインラインだった follow-up 指示を抽出
- `writeInstructionSandwich(sb)`: 最重要 4 ルール再掲
- `writeFewShotExample(sb, input)`: SubIntent 条件分岐付き 1 例出力

### 変更ファイル

| ファイル | 変更内容 |
|----------|----------|
| `rag-orchestrator/internal/usecase/prompt_builder.go` | `buildSingleTurn`/`buildMultiTurn` を system+user 分離にリファクタリング。3 つの新メソッド追加 |
| `rag-orchestrator/internal/usecase/prompt_builder_test.go` | 既存テスト 6 件の assertion 更新 + 新テスト 7 件追加（system role 検証、instruction sandwich、few-shot、multi-turn 再注入） |
| `rag-orchestrator/internal/usecase/answer_with_rag_usecase_test.go` | mock matcher 8 箇所の `len(msgs)` / `Role` assertion 更新 |

### 変更不要の確認

| ファイル | 理由 |
|----------|------|
| `ollama_generator.go` | `toChatMessages()` が任意の role を透過する設計。system role は自動的に Ollama の `<\|turn>system` にマッピングされる |
| `answer_with_rag_usecase.go` | `promptBuilder.Build()` の結果をそのまま LLM に渡すだけ |
| `rag_answer_stream.go` / `stream_hybrid_longform.go` | `buildPrompt` 経由で自動的に新形式を使用 |
| PromptBuilder interface | `[]domain.Message` を返す契約は不変 |

## Consequences

### Pros

- **Instruction following の根本改善**: Gemma 4 の native system role を活用し、指示を特権的コンテキストとして処理
- **Multi-turn ペルソナドリフト防止**: system メッセージの毎ターン再注入により、長い会話でも品質基準を維持
- **Lost in the Middle 効果の回避**: 指示（system）とデータ（user）の分離により、context window 内の注意力分布を最適化
- **Recency bias の活用**: Instruction sandwich により、LLM が最も注意を払う末尾位置に重要ルールを配置
- **コードベースの一貫性**: `morning_letter_prompt_builder.go`、`agentic_synthesis_strategy.go` と同じパターンに統一
- **実測結果**: 因果分析クエリの応答が 2 行短文 → 5,250 bytes の構造化回答（直接要因/構造的背景セクション付き）に改善

### Cons / Tradeoffs

- **トークン予算の微増**: few-shot example（約 150 トークン）+ instruction sandwich（約 80 トークン）= 約 230 トークン増。典型的なプロンプトの 3-8K トークンに対して 3-5% の増加
- **KV cache 効率**: system メッセージが静的であるため Ollama の prefix cache が有効に機能するが、intent/SubIntent により system 内容が変化するため完全な cache hit は保証されない
- **テスト影響範囲**: prompt_builder_test.go と answer_with_rag_usecase_test.go の両方で assertion 更新が必要だった（合計 20+ 箇所）
