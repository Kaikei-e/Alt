# ADR-000682: Dashboard UIをAlt-Paperデザイン言語に刷新し「Editorial Briefing Sheet」形式にする

## Status

Accepted

## Date

2026-04-10

## Affected Services

- alt-frontend-sv (TypeScript / SvelteKit)

## Context

Dashboard はデスクトップ専用の運用概観ページであり、フィード統計・最新記事・3日間レキャップを一覧表示する。[[000674]] で Alt-Paper デザイン言語と Responsibility-Driven Design が確立され、[[000678]]（Tag Trail）、[[000680]]（Morning Letter）と段階的に UI 刷新が進行しているが、Dashboard はまだ旧スタイルのままであった。

### 主な違反

- 汎用 `PageHeader` コンポーネント（"Dashboard — Overview of Alt RSS Reader"）の使用
- `Loader2` スピナー（Alt-Paper: パルスドット + italic テキスト）
- Lucide アイコン 7 種（`Rss`, `FileText`, `CheckCircle`, `Wifi`, `WifiOff`, `ArrowRight`, `Tag`）による視覚表現
- カラーアイコン背景（`bg-[var(--alt-blue)]/10` 等）— Alt-Paper のモノクロ方針に違反
- `bg-white` — Alt-Paper の `--surface-bg` トークン未使用
- 全ウィジェットで視覚スタイリングが Tailwind ユーティリティクラスに依存（scoped style 方針と不整合）
- ページリビールアニメーションなし

### Dashboard の責務

Dashboard の責務は**運用概観（Operational Awareness）**である。新聞に例えると「編集者の机（Editor's Desk）」— 朝のブリーフィングシートで発行部数・最新入電・要約を一覧するメタファーに該当する。

Acolyte のマストヘッド（レポート出版の責務）や Ask Augur のアバターバイライン（コンサルテーションの責務）とは異なる、Dashboard 固有のビジュアル表現が必要である。

## Decision

Dashboard 全コンポーネント（ルートページ + 3 ウィジェット）を Alt-Paper デザイン言語に準拠する形で「**Editorial Briefing Sheet**」形式に刷新する。データ取得ロジック・API 接続・型定義は一切変更しない。

### Editorial Briefing Sheet 形式

```
┌────────────────────────────────────────────────────┐
│  Thursday, April 10, 2026        (monospace date)  │
│  Editorial Brief                 (serif heading)   │
│  ──────────────────────────────── (1px rule)       │
│                                                    │
│  FEEDS 127 │ ARTICLES 4,231 │ SUMMARIZED 3,890    │
│  ● Live                      (monospace figures)   │
│  ──────────────────────────────── (1px rule)       │
│                                                    │
│  LATEST DISPATCHES     │  THREE-DAY BRIEF          │
│  ────────────────      │  ────────────────         │
│  APR 10 · Reuters      │  Technology               │
│  Article title...      │  Summary text...          │
│  View All →            │  Updated: 04/10 08:12     │
└────────────────────────────────────────────────────┘
```

### ヘッダー

汎用 `PageHeader` を除去し、Dashboard 固有の editorial header をルートページに直接配置:

- モノスペース日付（`--font-mono`, 0.7rem, `--alt-ash`）
- セリフ見出し "Editorial Brief"（`--font-display`, 1.6rem, weight 800, `--alt-charcoal`）
- 1px `--surface-border` thin rule

これは Acolyte の 2px 二重線マストヘッドとは明確に異なる表現。左寄せ・コンパクトなブリーフィングシートの体裁。

### 数値バー（StatsBarWidget → Circulation Figures）

Lucide アイコン 5 種 + カラー背景の 4 カラムグリッドを、モノスペースの横並び数値バーに置換:

- `FEEDS` / `ARTICLES` / `SUMMARIZED` ラベル（0.65rem, uppercase, letter-spacing 0.08em, `--alt-ash`）
- 数値（`--font-mono`, 1.1rem, weight 600, `tabular-nums`, `--alt-charcoal`）
- 1px `--surface-border` 縦セパレーター
- 接続状態: 6px パルスドット + "Live" / "Offline" モノスペーステキスト

### 最新入電（UnreadFeedsWidget → Latest Dispatches）

- セクションラベル "LATEST DISPATCHES"（`--font-mono`, 0.65rem, uppercase, letter-spacing 0.12em）
- 各記事: モノスペースデートライン + セリフタイトル + プロース抜粋
- ローディング: パルスドット + italic "Retrieving dispatches…"
- "View All →" テキストリンク（アイコンなし）

### 3日間ブリーフ（RecapSummaryWidget → Three-Day Brief）

- セクションラベル "THREE-DAY BRIEF"（同上のモノスペース体裁）
- ジャンルエントリ: セリフ見出し + プロースサマリ + モノスペースインライン用語（`·` 区切り、Tag アイコンなし）
- 更新タイムスタンプ: モノスペース表示
- ローディング: パルスドット + italic "Retrieving brief…"

### ウィジェットの Props ベース化

全 3 ウィジェットを内部データ取得から props 受け取りの純粋表示コンポーネントに変更。データ取得はルートページ `+page.svelte` に集約:

- `StatsBarWidget`: `feedAmount`, `totalArticlesAmount`, `unsummarizedArticlesAmount`, `isConnected`
- `UnreadFeedsWidget`: `feeds`, `isLoading`, `error`
- `RecapSummaryWidget`: `recapData`, `isLoading`, `error`

`useFeedStats()` フックの呼び出しもルートページに引き上げ。フック自体のロジックは変更なし。

### スタッガードリビール

- ページ全体: `opacity: 0 → 1` + `translateY(6px → 0)` の 0.4s トランジション
- 各セクション: `--delay` CSS 変数 × 100ms のスタッガード
- `prefers-reduced-motion: reduce` でアニメーション完全スキップ

### テスト

vitest-browser-svelte + Playwright による 24 件のブラウザテストを新規作成:

| ファイル | テスト数 | 検証内容 |
|---------|---------|---------|
| `StatsBarWidget.svelte.test.ts` | 8 | 数値表示、ラベル、Live/Offline 状態 |
| `UnreadFeedsWidget.svelte.test.ts` | 7 | セクション見出し、記事表示、ローディング/空/エラー状態、リンク |
| `RecapSummaryWidget.svelte.test.ts` | 9 | セクション見出し、ジャンル/サマリ/用語表示、ローディング/空/エラー状態、タイムスタンプ |

共有フィクスチャ `dashboard-fixtures.ts` に `MOCK_FEEDS` と `MOCK_RECAP` を定義。

### vitest ブラウザテスト環境修正

ブラウザテスト実行時に esbuild が `node_modules` 内の `.svelte` ファイルを処理できない既存の不具合を発見・修正。`vitest.config.ts` の `optimizeDeps.exclude` に Svelte ベースのパッケージ（`bits-ui`, `@lucide/svelte`, `@threlte/core`, `@threlte/extras`, `@tanstack/svelte-query`）を追加し、プロジェクト全体のブラウザテストを実行可能にした。

## Consequences

### Pros

- Alt-Paper 段階的刷新の 4 件目として、Dashboard がデザイン言語に完全準拠
- Responsibility-Driven Design 原則の新しい適用例:「Editorial Briefing Sheet」は Dashboard 固有の表現であり、Acolyte マストヘッドや Augur バイラインのコピーではない
- ウィジェットの props ベース化により、テスタビリティが大幅向上（モック不要で 24 件のブラウザテストが高速実行）
- scoped `<style>` + CSS 変数への移行で、デザイントークン変更が CSS 値の変更のみで伝播
- Lucide アイコン 7 種の依存を除去し、タイポグラフィのみで情報階層を表現
- vitest ブラウザテスト環境の修正により、プロジェクト全体の `.svelte.test.ts` テストが実行可能に
- `prefers-reduced-motion` 対応をアニメーション追加と同時に実装

### Cons / Tradeoffs

- `PageHeader` からの離脱により、将来 `PageHeader` に一括変更を加えた場合に Dashboard は個別対応が必要（ただし他の刷新済み UI も同様の方針）
- ウィジェットの props ベース化により、データ取得の責務がルートページに集約。ウィジェット単体での再利用時にはデータ取得ロジックを別途用意する必要がある
- `optimizeDeps.exclude` の追加はワークアラウンドであり、Vite / vitest のアップデートにより将来不要になる可能性がある
