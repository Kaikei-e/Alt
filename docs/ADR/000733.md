# ADR-000733: recap-subworker の Phase 5B domain error narrowing と Phase 6 E2E 基盤導入

## Status

Accepted

## Date

2026-04-15

## Affected Services

- recap-subworker (Python 3.14 / FastAPI)

## Context

[[000726]] の 6 フェーズ計画で、Phase 5 ([[000732]]) にて `LearningClient` に per-stage timeout + `asyncio.timeout` 外郭キャンセルを導入したが、タイムアウト時に呼び出し側へ漏れる例外が依然として `asyncio.TimeoutError` / `httpx.TimeoutException` の素の生ペアだった。handler / caller は 5 層以上のライブラリ内部例外を `except` で拾う必要があり、ドメイン的意味（「学習クライアントが時間内に応答しなかった」）が消えていた。

また Phase 6 として E2E 系の整備が計画されていた。既存のユニットテストは Phase 1 unblock の autouse override ([[000728]]) により handler 層を安全に叩けるが、FastAPI の lifespan + DI + ASGI パイプラインを「本番構成に近い経路で」通すテストは `tests/unit/test_api_runs.py` の `TestClient` パターンに限られていた。`TestClient.__enter__` への monkey-patch で override を注入するため、非同期ネイティブな `httpx.AsyncClient + ASGITransport` 経路がカバーされていなかった。

本 ADR は上記 2 点を合わせて扱う:

- **Phase 5B**: `LearningClient` が timeout を domain error `LearningClientTimeoutError` に narrow する
- **Phase 6（導入編）**: `httpx.ASGITransport` ベースの E2E テスト基盤を `tests/integration/` に据える。Phase 6 で計画された他のテスト（`test_failure_modes.py`, `test_logging_context.py`, `test_router_reload.py`, `test_classify_e2e.py`）は後続 Phase 6B 以降に繰越

## Decision

### Phase 5B

1. `domain/errors.py` に以下を追加:
   - `LearningClientError(SubworkerError)`: base
   - `LearningClientTimeoutError(LearningClientError)`: detail を受け取り message に含める

2. `services/learning_client.py::send_learning_payload` を以下の構造に:

   ```python
   try:
       async with asyncio.timeout(outer_budget):
           response = await self._client.post(endpoint, json=payload)
       response.raise_for_status()
       return response
   except (TimeoutError, httpx.TimeoutException) as exc:
       raise LearningClientTimeoutError(
           f"{endpoint} exceeded {outer_budget:.1f}s budget"
       ) from exc
   except httpx.HTTPStatusError as exc:
       ...  # 既存ハンドラ
   except httpx.RequestError as exc:
       ...  # 既存ハンドラ
   ```

   - `TimeoutError` (Python 3.11+ で `asyncio.TimeoutError` と同一) と `httpx.TimeoutException` を同じ `except` で捕捉
   - `raise ... from exc` で原因 traceback を保持（[[000651]] 系の診断価値を維持）

3. 回帰テスト `tests/unit/gateway/test_learning_client_errors.py`:
   - `_NeverRespondTransport` で永遠に `await asyncio.sleep(60)` する偽トランスポート
   - `pytest.raises(LearningClientTimeoutError)` で domain error への変換を assert

### Phase 6（導入編）

1. `tests/integration/` を有効化（`__init__.py` を新規作成）
2. `tests/integration/test_runs_e2e.py` に 2 ケース追加:
   - `test_post_then_get_runs_round_trip_via_asgi_transport`: POST `/v1/runs` → 202 → GET `/v1/runs/{id}` → 200 (status=running) を `httpx.ASGITransport` + `httpx.AsyncClient` 経由で往復
   - `test_missing_run_returns_404_via_asgi_transport`: 空ストアに対する GET → 404
3. 重量級 `ServiceContainer` 依存を経由しないよう、`SubmitRunUsecase` / `GetRunUsecase` を `_InMemoryRunStore` で override（Phase 1 unblock の精神を維持）
4. Phase 6 で計画されていた他のテスト群は Phase 6B に繰越:
   - `test_failure_modes.py` (DB down → 503, embedder down → 503, timeout → 504)
   - `test_logging_context.py` (structlog `service` / `run_id` / `trace_id` bind)
   - `test_router_reload.py` (importlib.reload パターン)
   - `test_classify_e2e.py`
   - `scripts/pact-check.sh` の CI gate 化（現状 opt-in）

## Consequences

### Pros

- handler / caller が `except (asyncio.TimeoutError, httpx.TimeoutException)` と書く必要がなくなり、`except LearningClientTimeoutError` だけで意味を捉えられる。将来 `/admin/learning` 経路に retry ロジックを追加するとき、ドメイン名前空間で意図が通る。
- `raise ... from exc` により原因スタックが保持されるため、Pyrefly の `TRY` 系ルール（後続 Phase で有効化予定）と親和性がある。
- Phase 6 の ASGITransport ベーステストは、Phase 1 unblock の `TestClient.__enter__` monkey-patch を経由せず、本番同様の async 経路で handler / usecase / dependency の配線を確認できる。Phase 2 で追加した 4 Usecase 境界が ASGI ストリームを介しても期待どおり挙動することを保証。
- `httpx.AsyncClient(transport=ASGITransport(app=app))` パターンは FastAPI 公式推奨の async テスト pattern（[FastAPI async tests](https://fastapi.tiangolo.com/advanced/async-tests/)）に沿っている。将来 `TestClient` の非同期的制約に当たった際の移行コストが低い。
- `_InMemoryRunStore` は Port 契約だけを満たす 15 行の最小 class。Phase 2 の Port 設計が実用的であることを E2E で裏打ちした。

### Cons / Tradeoffs

- Phase 5B は `LearningClient` のみ。`services/async_jobs.py` / `services/classification_runner.py` の broad `except Exception` はそのまま残っており、Phase 5C 相当での narrow 化が必要。
- Phase 6 は E2E 導入編に留まる。障害系（DB 503 / embedder 503 / upstream timeout 504 のルーティング）は test fixture が未整備で、Phase 6B での test infra 整備（docker-compose の testcontainers 代替等）とセットで扱う必要がある。
- `_InMemoryRunStore` は `create_run` / `get_run` のみ実装。Idempotency mismatch / concurrent run / pipeline 実行の E2E は未カバー。これらは Pact Provider 検証 (`tests/contract/test_provider_verification.py`) で担保するが、Pact ペイロードと本 E2E の両輪で拾う体制は Phase 6B で整備する。
- Phase 6 で計画されていた `scripts/pact-check.sh` の PR 毎必須 gate 化は未実施。現状 `./scripts/pact-check.sh` は opt-in 実行。
