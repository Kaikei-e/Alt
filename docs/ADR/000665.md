# ADR-000665: Acolyte パイプラインの LLM 呼び出し安定化と search-indexer 誤検知を修正する

## Status

Accepted

## Date

2026-04-10

## Affected Services

- acolyte-orchestrator (Python 3.14+)
- search-indexer (Go 1.26+)

## Context

2026-04-09 のパイプライン実行ログ調査で、Acolyte レポート生成パイプラインに 3 種の障害が確認された。いずれもパイプラインの完走を阻害し、レポート生成に失敗する原因となっていた。

### 障害 1: Extractor ノードの JSON 切断と ReadTimeout

ExtractorNode は 5 記事を順次処理し、各記事から atomic facts を LLM (Gemma4 26B) で抽出する。`generate_validated()` 経由で Ollama `/api/generate` を呼ぶが、以下の複合的な問題が発生していた:

1. **`num_predict=4096` でトークン制限ヒット** — GBNF grammar が JSON を閉じきれず `Unterminated string` エラー。verbatim_quote フィールドが長大になりバジェットを超過
2. **Gemma4 thinking tokens の JSON 外流出** — `ExtractorOutput` に `reasoning` フィールドがなかったため、thinking tokens が Ollama の `thinking` response field に流出し、`num_predict` バジェットを浪費（response_len=28 chars に対し eval_count=836 tokens）
3. **`httpx.ReadTimeout`** — `num_predict` 増加後、1 記事あたりの生成時間が 10 分を超え ReadTimeout 発生。さらに `generate_validated()` 内の `llm.generate()` が try/except の外にあり、ReadTimeout 時にリトライが効かなかった
4. **1 記事の失敗で全結果消失** — Extractor のループに try/except がなく、1 記事の失敗でそれまでの成功記事の facts も LangGraph state に返されなかった

### 障害 2: Planner ノードの JSON parse 失敗

`num_predict=512` では日本語 JSON（reasoning + 3-5 セクションの title/search_queries）を完結できず、毎回フォールバック（固定 3 セクション）が使われていた。

### 障害 3: search-indexer の SQL インジェクション誤検知

`search_articles.go` の SQL インジェクション防止正規表現 `(?i)(union|select|...|exec|execute|...)` が、Acolyte Gatherer のクエリ `"executive summary"` の `exec` を部分一致で検出し 500 エラーを返していた。本日 5 回発生。

### 技術的背景

| 事実 | 出典 |
|------|------|
| thinking tokens は num_predict に含まれる（別枠ではない） | ollama/ollama#14793 |
| think_budget パラメータは Ollama 未実装 | ollama/ollama#10925 |
| think=false を設定すると format (structured output) が壊れる (Gemma4 固有) | ollama/ollama#15260 |
| JSON schema に `reasoning` フィールドがあると thinking が JSON 内に吸収される | ADR-632 の reasoning-first パターン |
| thinking 有効モデルの思考トークンは 10x-20x のオーバーヘッド | arxiv:2604.07035 |

## Decision

**4 つの修正を実施する。**

### 1. Extractor `num_predict` を 4096 → 6000 に調整

8192 は生成時間が長すぎ（~7.4 分/記事）ReadTimeout のリスクが高い。6000 は 4096 より 47% 多く、10 分 timeout 内に余裕で収まる（~5.4 分/記事）。

num_ctx バジェット: prompt(~2,700) + 6,000 = ~8,700 / num_ctx 12,288 → 余裕 29%。

### 2. ExtractorOutput に `reasoning` フィールドを追加 + Extractor ループに try/except

**reasoning フィールド**: Gemma4 の thinking tokens を JSON 内に吸収するため、`ExtractorOutput` Pydantic モデルに `reasoning: str` を先頭フィールドとして追加。これは Planner・Critic・SectionPlanner で既に確立された ADR-632 パターン（reasoning-first field order）の Extractor への適用。

**try/except**: Extractor の記事ループを try/except で囲み、1 記事の失敗が他の記事の結果を消失させないようにする。

### 3. `generate_validated()` のリトライ範囲を拡大

`llm.generate()` 呼び出しを try/except 内に移動し、ReadTimeout 等のネットワーク例外もリトライ対象にする。これにより Ollama のキュー待ちによる一時的な timeout からの自動復旧が可能になる。

### 4. search-indexer SQL インジェクション正規表現にワードバウンダリ追加

SQL キーワード（`union`, `select`, `exec` 等）に `\b` を追加し、`executive` → `exec` のような部分一致を防止。SQL 記号系（`--`, `;`, `'`, `"` 等）はワードバウンダリ不要。

### 5. Planner `num_predict` を 512 → 2048 に増加

日本語 JSON の推定トークン消費: 500-1200 tokens。2048 で十分な余裕を確保。

### 6. httpx read timeout を 300s → 600s に延長

`num_predict=6000` + Gemma4 26B の生成速度を考慮し、1 リクエストあたり 10 分の timeout を確保。

## Consequences

### Pros

- **Extractor の部分結果保全**: 5 記事中 1 記事が失敗しても、他 4 記事の facts はパイプラインに渡される
- **ReadTimeout 自動復旧**: `generate_validated` が LLM 呼び出し例外をリトライするため、一時的な timeout から自動復旧
- **search-indexer 誤検知解消**: `executive summary` 等の正当なクエリが通るようになり、Gatherer の evidence 収集が改善
- **Planner のトークン余裕**: 2048 tokens で日本語 JSON 出力に十分な余裕
- **reasoning-first パターンの統一**: 全 structured output ノード（Planner, Extractor, SectionPlanner, Critic）が reasoning フィールドを持ち、Gemma4 thinking tokens の JSON 内吸収を統一

### Cons / Tradeoffs

- **Extractor の生成時間増加**: num_predict=6000 は 4096 比で ~45% 長い生成時間。5 記事で ~27 分
- **Planner は依然フォールバック使用**: 2048 tokens でもフォールバックが発生する場合がある（Gemma4 の thinking + reasoning が長い場合）。根本解決には Ollama の think_budget 実装待ち
- **search-indexer の `\b` が Go regexp で特殊な挙動をする可能性**: Unicode 文字境界の扱いが ASCII と異なる可能性。テストで検証済み
- **LangGraph checkpointer 未導入**: パイプライン全体の中間状態はまだ揮発性。将来 PostgresSaver 等で対応予定
