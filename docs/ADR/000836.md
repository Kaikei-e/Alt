# ADR-000836: Knowledge Loop の Ask CTA を Augur へ handshake する RPC を追加する

## Status

Accepted

## Date

2026-04-23

## Affected Services

- **proto** — `proto/alt/augur/v2/augur.proto` に `CreateAugurSessionFromLoopEntry(CreateAugurSessionFromLoopEntryRequest) returns (CreateAugurSessionFromLoopEntryResponse)` を追加。既存 RPC は無変更。新規 `LoopEvidenceRef` メッセージは sovereign の `KnowledgeLoopEvidenceRef` と形が揃う (`ref_id` / `label`)。
- **rag-orchestrator (Go)** — `internal/adapter/connect/augur/handler.go` に `CreateAugurSessionFromLoopEntry` ハンドラを追加。`internal/usecase/augur_conversation_usecase.go` の `AugurConversationUsecase` に `CreateSessionFromLoopEntry(ctx, CreateSessionFromLoopEntryInput)` を追加し、append-only で conversation + seeded assistant message を書き込む。既存の `EnsureConversation` / `AppendUserTurn` / `AppendAssistantTurn` は無変更。
- **alt-backend (Go)** — `app/connect/v2/augur/handler.go` に同名のハンドラを追加。認証済みの user UUID を `X-Alt-User-Id` に詰め直して `RagStreamPort.CreateAugurSessionFromLoopEntry` へ転送。`port/rag_stream_port/rag_stream_port.go` と `gateway/rag_connect_gateway/client.go` に同じメソッドを追加し `mockgen` で mock を再生成。
- **alt-frontend-sv** — 新規 `src/routes/(app)/loop/ask/+server.ts`(POST) が sovereign 経由で entry を lookup し why_text + evidence_refs を enrich して alt-backend へ転送。`src/lib/connect/augur.ts` に `createAugurSessionFromLoopEntry` helper、`src/lib/server/knowledge-loop-api.ts` に `createAugurSessionFromLoopEntryForUser`。`LoopEntryTile.svelte` は `onAsk` prop を受け取り `ask` intent の CTA を有効化 (前 PR-L2 では silent-skip していた)。`/loop/+page.svelte` は POST → `goto('/augur/<conversationId>')` を駆動する。

## Context

- PR-L2 で `/loop` tile の OODA 遷移 (Open / Save / Snooze / Dismiss) と dwell-based Observe は配線済みだが、`ask` intent の CTA は silent-skip されていた。projector が `source_why` で `[open, ask, save, snooze]` を seed するため、`ask` だけ未配線なのは UX 的にも実装的にも放置できない状態だった。
- 親プラン (`plan-core-md-plan-schema-md-knowledge-l-warm-candy.md` の PR-L3 セクション) は「Augur service 側で sovereign 経由で entry lookup → why_text + evidence_refs を pre-load → session row を作る」設計を前提にしていた。投入調査で `rag-orchestrator` は augur の実装サービスだが、sovereign Connect-RPC client を持たない (proto stub のみ生成済) ことが分かった。
- 既存 Augur コンテキスト: 会話は `augur_conversations` + `augur_messages` (append-only) に永続化される。`conversation_id` 単位で `/augur/<id>` が存在し、既存の `getAugurConversation` が最初から描画できる。
- 既存の trust boundary: **alt-backend が JWT 境界**、`X-Alt-User-Id` で rag-orchestrator に伝播。sovereign への読み取りは alt-backend の sovereign client + SvelteKit BFF の `getKnowledgeLoopForUser` が保有。
- 新規 RPC は Pact CDC 未済みの領域 (Augur は既存 pact contract 無し)。本 PR では Pact 契約は追加しない。

## Decision

Knowledge Loop → Augur の handshake を以下の 1 つの新 Connect-RPC として導入する。

```proto
rpc CreateAugurSessionFromLoopEntry(CreateAugurSessionFromLoopEntryRequest)
    returns (CreateAugurSessionFromLoopEntryResponse);

message CreateAugurSessionFromLoopEntryRequest {
  string client_handshake_id = 1; // UUIDv7 idempotency key (server enforces format)
  string entry_key           = 2; // audit breadcrumb; NOT re-verified server-side
  string lens_mode_id        = 3; // audit breadcrumb
  string why_text            = 4; // 1..512 runes, enriched by BFF
  repeated LoopEvidenceRef evidence_refs = 5; // length <= 8, enriched by BFF
}

message CreateAugurSessionFromLoopEntryResponse {
  string conversation_id = 1;
}
```

**信頼境界**: `why_text` と `evidence_refs` は **alt-frontend-sv BFF が sovereign から取得したもの** で、alt-backend と rag-orchestrator はそれを正本として受け取る。Augur は sovereign に問い合わせない。これは以下の理由で意図的な設計選択である。

1. **sovereign client を rag-orchestrator に足さないで済む** — proto stub のみ生成済みで gateway / usecase / driver が未整備。今回追加すると PR 粒度が膨らみ、sovereign 側の auth / tenant context 配線も必要になる。
2. **BFF は既存の読み取りパス (`getKnowledgeLoopForUser`) を再利用できる** — /loop 画面が持つ最新の foreground 情報と同じソースから `entryKey` を lookup するので、UI と API が見ている世界が一致する。
3. **権限境界は変わらない** — SvelteKit server routes が唯一のアウトバウンド actor で、browser は alt-backend に直接触らない (invariant「alt-frontend-sv は alt-backend に直結しない」)。BFF を trusted enrichment 層として扱うのは既存の KnowledgeLoop transition BFF と同じパターン。

**会話の seed 仕様**: Augur の `CreateSessionFromLoopEntry` usecase は:

- `augur_conversations` に 1 行 INSERT (append-only)。title は `titleFromFirstMessage(why_text)` で 80 runes に truncate。
- `augur_messages` に 1 行 INSERT。role は `"assistant"`、content は `"From your Knowledge Loop:\n\n{why_text}..."`、citations は evidence_refs 由来の `AugurCitation[]`。
- ユーザが `/augur/<id>` に着地すると 1 個の assistant turn が既に存在する状態で描画される (既存 `getAugurConversation` でそのまま読める)。

**UUIDv7 idempotency**: 現 PR では server 側で (user_id, client_handshake_id) の重複検知は実装しない。二重 click は BFF + UI の in-flight guard で防ぐ (`useKnowledgeLoop` の inFlight + page の `askInFlight` Set)。将来 retry on network failure が実運用で出てきたら `augur_conversations.client_handshake_id` に unique index を足し AlreadyExists 系エラーで replay する形に拡張する。

**CTA 表示ポリシー**: `LoopEntryTile.svelte` は `onAsk` callback が注入されない環境 (例: テスト / 一部のページ埋め込み) では `ask` intent を silent-skip し続ける。プロップ駆動の漸進的 rollout で、PR-L3 ロールアウト後に他ページから tile を使い回すとき UI が勝手に Ask CTA を出さない。

## Consequences

### 得られるもの

- /loop tile の 4 CTA (`Open` / `Ask` / `Save` / `Snooze`) + `Dismiss` が完全配線される。`source_why` タイプ entry で 1 tap で Ask Augur に文脈付きで入れる。
- Augur の conversation 履歴に Knowledge Loop 由来の prelude が残り、ユーザが後から `/augur/history` で掘り返しても文脈が保持される (`augur_conversation_index` view 経由で list-visible)。
- proto 側に公開 RPC が 1 本増え、`buf breaking` は additive なので OK。既存 consumer (alt-butterfly-facade / alt-frontend-sv 以外) には影響しない。

### 失うもの / 今後の技術的負債

- **rag-orchestrator が sovereign に問い合わせない** ため、BFF を迂回した直接呼び出しに対しては `why_text` / `evidence_refs` の正当性が保証されない。将来 mobile native client などが直に alt-backend を叩くなら、alt-backend 側で `entry_key` の owner 検証を足すか、sovereign client を rag-orchestrator に導入する必要がある。
- Idempotency が UI/BFF 層のガードに頼っているため、ネットワーク断 + 自動再試行のような環境では 1 つの entry に対して複数 conversation が生まれる可能性がある (許容する方針)。
- Pact CDC が未整備の Augur service 境界に 1 本新規 RPC を足した形。alt-frontend-sv → alt-backend、および alt-backend → rag-orchestrator の 2 本の pact を別 PR で入れる余地がある。

### 運用・監視

- alt-frontend-sv BFF のログ: entry lookup ミス (404) は UX の問題なので info レベル、upstream 5xx は warn。
- rag-orchestrator: `create augur loop session failed` を error で残す。既存の structured logger 規約に乗っている。
- DB サイズ: 1 entry 1 conversation + 1 message。既存のクリーンアップ機構 (augur_conversation_index + cascade delete) に乗る。

## Alternatives considered

1. **rag-orchestrator に sovereign gateway を追加する案** — 親プランの素直な実装。却下理由は「stub しか無い状態からの gateway 新設 + auth 経路配線が本 PR のスコープを大幅に超えるため」。必要になった時点で段階的に導入できる (RPC shape を変えなくてよい)。
2. **`CreateAugurSessionFromLoopEntry` を alt.knowledge.loop.v1 proto 側に置く案** — conversation 作成を Loop サービスが行うことになり、augur の永続化責務を越境する。却下。
3. **Loop tile の Ask を既存 `/augur?q=...` prefill で済ませる案** — ADR-000831 では当初 `/augur?q=<why>` 案も出ていた。却下理由は (a) evidence_refs が URL でない場合に prefill できない、(b) 会話履歴に Loop 起点の事実が残らない、(c) Augur 側 prompt builder が Loop prelude を区別できない。
4. **idempotency を server 側に倒す案** — `augur_conversations.client_handshake_id UNIQUE` を migration で追加し AlreadyExists を replay するパターン。却下理由は「PR-L3 MVP に含める必然性が弱く、マイグレーションが別トラックの alt-deploy 変更を呼ぶ」。実運用で再試行が問題化したら別 PR で導入する。
