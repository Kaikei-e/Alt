# ADR-000527: loadMergedFeeds 並列化と Connect-RPC OTel 計装で /feeds 初期表示レイテンシを改善

## Status

Accepted

## Date

2026-03-22

## Affected Services

- alt-backend (Go) — `CachedFeedListUsecase` 並列化、Connect-RPC OTel interceptor 追加
- alt-frontend-sv (TypeScript/SvelteKit) — `/feeds` サーバーロード最適化

## Context

Desktop UI の `/feeds` および `/feeds/visual-preview` ページの初期表示が遅い問題が報告された。

24 時間分のコンテナログ（nginx upstream timing、alt-backend `feed_read_perf` 構造化ログ）、ClickHouse `otel_traces` テーブルを横断調査した結果、以下の状況が判明した。

### 計測データ（改善前、サンプル数は少ないため暫定値）

| 指標 | 値 |
|---|---|
| `/feeds/__data.json` cold p95 | 1,758ms (n=3) |
| `/feeds/visual-preview/__data.json` cold p95 | 1,979ms (n=5) |
| `GetUnreadFeeds` usecase cold | 675–1,878ms (n=7) |
| `GetUnreadFeeds` usecase warm | 49–54ms |
| `GetAllFeeds` usecase | 25–48ms（常にキャッシュ warm） |

### ボトルネック

`CachedFeedListUsecase.loadMergedFeeds()` がユーザの全サブスクリプション（50–100 件）に対して `FeedPageCache.GetFeedPage()` を **直列で** 呼び出していた。`SharedCache` の TTL（2 分）+ staleTTL（1 分）が失効すると、最初のリクエストで全サブスクリプション分の DB クエリが同期的に走り、50 件 × 各 20–30ms ≈ 1,000–1,500ms の遅延が発生していた。

### OTel の欠損

Connect-RPC ハンドラ（port 9101、`h2c.NewHandler`）は Echo の `otelecho.Middleware` の管轄外であり、`GetUnreadFeeds` 等の unary RPC スパンが ClickHouse `otel_traces` テーブルに一切記録されていなかった。ボトルネックの定量分析が困難な状態だった。

### auth-hub の冗長呼び出し

`/feeds/+page.server.ts` は `$lib/api` 経由で `createServerTransport(cookie)` を使っており、SSR のたびに auth-hub `/session` を呼んでいた。一方 `/feeds/visual-preview/+page.server.ts` は `locals.backendToken` を使ってこの呼び出しを回避済みだった。

## Decision

以下の 3 施策を採用する。

### 1. `loadMergedFeeds` の bounded concurrency 並列化

`errgroup.Group` + `SetLimit(8)` を使い、サブスクリプションごとの `GetFeedPage` 呼び出しを並列実行する。goroutine 間のデータ競合を避けるため、`[][]*domain.FeedItem` の per-subscription result buffer に結果を格納し、全 goroutine 完了後に merge + sort を一度だけ実行する設計とした。

並列数 8 は暫定値であり、DB connection pool 上限や DB 並列クエリ性能による頭打ちを考慮して保守的に設定した。

### 2. Connect-RPC OTel interceptor 追加

`connectrpc.com/otelconnect` パッケージの `otelconnect.NewInterceptor()` を Connect-RPC の interceptor チェーンに追加し、全 unary RPC のスパンを自動生成する。`traceparent` ヘッダの伝播も自動で行われる。

### 3. `/feeds/+page.server.ts` の backendToken 最適化

import 元を `$lib/api` から `$lib/server/feed-api` に変更し、`locals.backendToken` を `getFeedsWithCursor` に渡すことで、SSR 時の auth-hub `/session` 呼び出しを削減する。`/feeds/visual-preview` と同じパターン。

### 検討したが見送った施策

- **FeedPageCache TTL 延長（2 分→10 分）**: 並列化の効果が十分であったため保留。メモリ影響（subscription 数 × user 数 × feed page size）の見積もりと invalidate 経路の網羅確認が未済。
- **payload 最適化（description 切り詰め）**: proto 契約上 `description` は "sanitized, truncated" と記述されているが実装は truncate していない。frontend が `{@html feed.description}` で描画している箇所があり、単純な文字数切り捨ては HTML 破損を招く。契約整理が先のため別タスクに分離。

## Implementation

### alt-backend

**`usecase/cached_feed_list_usecase/usecase.go`**

```go
func (u *CachedFeedListUsecase) loadMergedFeeds(ctx, userID, excludeFeedLinkID) {
    // 1. excludeFeedLinkID を先にフィルタ
    // 2. errgroup.SetLimit(8) で bounded concurrency
    // 3. results[i] に per-subscription buffer として格納（共有 slice への append を回避）
    // 4. g.Wait() 後に merge + sort
}
```

**`connect/v2/server.go`**

```go
otelInterceptor, _ := otelconnect.NewInterceptor()
opts := connect.WithInterceptors(
    cancelInterceptor.Interceptor(),
    authInterceptor.Interceptor(),
    otelInterceptor,  // 追加
)
```

### alt-frontend-sv

**`src/routes/(app)/feeds/+page.server.ts`**

- import: `$lib/api` → `$lib/server/feed-api`
- load signature: `({ request })` → `({ request, locals })`
- `getFeedsWithCursor(cookie, undefined, LIMIT, locals.backendToken)`

### テスト

`usecase_test.go` に 4 件の挙動回帰テストを追加:

- `excludeFeedLinkID` による subscription 除外
- 同一 `PublishedParsed` 時のソート安定性（Link 降順）
- `GetFeedPage` 一部失敗時のエラー伝播
- `hasMore` フラグの limit 境界動作

全 6 テスト PASS（`-race` フラグ付き）。

## Consequences

### Pros

- **cold cache レイテンシ 70% 削減**: `GetUnreadFeeds` が 1,878ms → 516ms に改善
- **warm cache レイテンシ 50% 削減**: 54ms → 26ms
- **OTel 可観測性の回復**: `GetUnreadFeeds`, `GetAllFeeds`, `ListSubscriptions` 等の Connect-RPC スパンが `otel_traces` テーブルに記録され、Grafana Trace Explorer で確認可能に
- **auth-hub 呼び出し削減**: `/feeds` SSR で 1 回 → 0 回

### Cons / Tradeoffs

- **bounded concurrency の並列数 8 は暫定値**: DB connection pool や DB 並列性能によっては最適値が異なる。実測比較（4/8/16）は未実施。
- **TTL 2 分は未変更**: cold cache は依然として発生する（頻度は並列化で影響が軽減されたが根本的には残る）。必要に応じて Step 4 として TTL 調整を検討。
- **payload サイズ未改善**: protobuf サイズで 20 件 ≈ 250KB は変わっていない。description truncate は契約整理後の後続タスク。

### 計測結果（改善後）

| 指標 | 改善前 | 改善後 | 変化 |
|---|---|---|---|
| `GetUnreadFeeds` cold | 675–1,878ms | 462–516ms | ~70% 削減 |
| `GetUnreadFeeds` warm | 49–54ms | 26ms | ~50% 削減 |
| OTel Connect-RPC coverage | 欠損 | 全 unary handler 可視化 | 解消 |
| auth-hub `/feeds` SSR | 1 回 | 0 回 | 削減 |
