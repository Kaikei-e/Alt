# ADR-000745: Pact Python provider state を Registry パターンで構造化する

## Status

Accepted

## Date

2026-04-15

## Affected Services

- news-creator (Python / FastAPI) — provider verification 4 interaction
- tag-generator (Python / FastAPI) — provider verification 2 interaction
- recap-subworker (Python / FastAPI) — provider verification 4 interaction
- tts-speaker (Python / Starlette) — provider verification 2 interaction

## Context

Alt の Python 製プロバイダ 4 サービスは既に `pact-python>=3.2.1` を採用し、
生成 pact JSON も `pactSpecification: 4.0` で v3 ライブラリ移行は完了して
いた。しかし provider verification テスト側のコードは v2 期に書かれた
パターンが残っており、v3 ベストプラクティス上 2 点の未達があった:

1. **provider state ハンドラが非構造化**
   `/_pact/provider-states` エンドポイントが state 名を `if "queue is full"
   in state.lower()` のような文字列部分一致で分岐しており（news-creator）、
   tag-generator / recap-subworker / tts-speaker に至っては受け取った state
   名を `provider_state["current"] = req.state` で握り潰すだけで何の副作用も
   無かった。consumer 側 pact JSON が新しい given(...) を増やしても provider
   は無音で「成功扱い」するため、CDC で本来検出すべき「state を増やしたが
   セットアップ忘れ」が見逃される構造になっていた。

2. **pact-python のバージョンが暗黙のまま**
   全 6 サービス（provider 4 + consumer 2 = recap-evaluator,
   acolyte-orchestrator）のピンが `>=3.2.1` で、PyPI 上の最新版（3.2.1,
   2025-12-10 リリース）と一致するものの、明示的に最新と確認されたタイミング
   が記録されていなかった。

## Decision

### 1. State Registry パターンの採用

各 provider テスト配下に共通ヘルパ `tests/contract/_pact_state.py` を新設し、
以下を提供する:

- `StateRegistry = dict[str, StateAction]`（state 名 → async コールバック）
- `ProviderStateRequest`（pydantic モデル、`state` / `params` / `action` を
  保持し未知フィールドを許容）
- `dispatch(registry, payload)`
  - `action == "teardown"` は no-op
  - 未登録 state は `logger.warning("Unknown provider state: %s", ...)` を
    出して握り潰さない
  - 登録済み state は対応する async 関数を `params` 付きで呼ぶ

各 provider テストは `_pact/provider-states` ハンドラ内で
`await dispatch(registry, payload)` を呼ぶだけにし、state 別の副作用は全て
レジストリエントリへ集約する。news-creator の `queue_full` 切替も文字列
部分一致を排し、

```python
"the LLM queue is full": _set_queue_full,
"the LLM model is loaded and ready": _set_default,
...
```

のように pact JSON 上の正式な state 名を辞書キーで明示マッピングする。

### 2. pact-python バージョン明示確認

PyPI に対し authoritative な version 確認を行い（最新: `3.2.1` /
2025-12-10）、全 6 `pyproject.toml` の `pact-python>=3.2.1` および
`uv.lock` 内 `version = "3.2.1"` がすべて最新と一致することを確認した。
pin 文字列の変更は不要、本 ADR で「最新確認済み」を記録する。

### 3. スコープ外（明示的に今回はやらない）

- consumer 側 matcher DSL（`pact.match.like` 等）採用 — bodies が
  `json.dumps(...)` でハードコードされている件は別タスク
- `docs/features/acolyte/extending.md:279` の v2 API サンプル（`from pact
  import Consumer, Provider`）の更新
- 他言語（Go / Rust）の pact テスト

## Consequences

### Pros

- **state 漏れの可視化**: consumer が新しい `given(...)` を追加した時点で
  provider 側に「Unknown provider state」warn が出るため、
  state セットアップ追加を強制できる。
- **責務分離**: 「副作用は何か」がレジストリ 1 箇所に集まり、HTTP
  ハンドラは dispatch するだけになる。news-creator の `queue_full` 切替が
  文字列 `in` 判定から消え、リファクタリング耐性が上がった。
- **pact spec v4 の `params` 対応**: `params` を `dict[str, Any]` で受け、
  state action へそのまま渡す形にしたため、interaction 単位で動的データを
  渡す将来の拡張（例: `given("article {id} exists", {"id": 42})`）に
  追加実装なしで対応できる。
- **TDD 工程の保全**: `_pact_state.py` ごとに 3 件のユニットテスト
  （`test_pact_state.py`）を先行して書き、レジストリ動作・unknown 警告・
  teardown スキップを検証してから provider verification を置き換えたため、
  refactor 中の挙動変化を即座に捕捉できる。
- **`pacts/*.json` 互換**: consumer 側出力には触れていないため、CDC の
  pact ファイル差分は 0。Broker への影響なし。

### Cons / Tradeoffs

- **ヘルパが 4 サービスにコピペ**: `_pact_state.py` を共有パッケージに
  切り出していないため、ロジック修正時は 4 箇所の同期が必要。Python の
  monorepo において workspace 共通テストパッケージを持たない現構成に
  合わせた割り切り。将来的に同等ヘルパが 5 サービス以上に増えるなら
  `tools/pact-state` 等への抽出を検討する。
- **未知 state を warn のみで通す**: warn ではなく test failure に
  したい意見はあり得るが、Pact verifier の挙動として state ハンドラが
  HTTP 200 を返さないと verification 自体が止まるため、現状は
  「warn を出し検証は継続」のソフト fail を採用。CI 上で
  `caplog`/`pytest -W error` を組み合わせれば後から hard fail へ昇格できる。
- **コンテナ再ビルド省略**: 変更は `tests/contract/` 配下のみで
  Dockerfile ステージにテストコードは含まれない（production イメージは
  `app/` のみ）。よってサービス再起動による smoke 検証は意味を持たず、
  ローカル CI 相当（ruff / pyrefly / pytest）でグリーンを確認する方針を
  採った。

## ローカル CI 相当の検証結果

各 provider サービスについて以下を実行し、すべて 0 error / pass:

| サービス | ruff | pyrefly | pytest (contract) |
|---|---|---|---|
| news-creator | ✓ | 0 errors | 5/5 passed |
| tag-generator | ✓ | 0 errors | 4/4 passed |
| recap-subworker | ✓ | 0 errors | 4/4 passed |
| tts-speaker | ✓ | 0 errors | 4/4 passed |

加えて consumer 側 `recap-evaluator` (3/3) と `acolyte-orchestrator` (4/4)
の contract 出力に差分が無いこと（`git diff pacts/` が空）を確認済み。
