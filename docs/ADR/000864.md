# ADR-000864: alt-frontend-sv の TypeScript 7 完全移行 (tsgo 一本化)

## Status

Accepted

## Date

2026-04-27

## Affected Services

- **alt-frontend-sv** — `package.json` / `bun.lock` / `CLAUDE.md` を更新。実装コードは無変更
- **CI (`.github/workflows/alt-frontend-sv-unit-test.yaml`)** — `Type check` step の前に tsgo version log step を追加

## Context

Microsoft が **2026-04-21** に TypeScript 7.0 Beta を正式公開した ([Announcing TypeScript 7.0 Beta](https://devblogs.microsoft.com/typescript/announcing-typescript-7-0-beta/))。Go-port された `tsgo` (`@typescript/native-preview`) は型検査セマンティクスを 6.0 と構造的に同一に保ちつつ、共有メモリ並列化により **約 10 倍高速** (1090 → 220 ms クラス) になった。npm registry の `@typescript/native-preview` は dist-tag のみで運用されており、Beta 公開時点で `@beta` は `7.0.0-dev.20260421.2` を、`@latest` は日次更新の dev build を指す。Stable 7.0.0 は 2026-06 頃に予定。

`alt-frontend-sv` は既に大半が tsgo に乗っていた:

- `bun run check` は `svelte-check --tsgo --tsconfig ./tsconfig.json` を呼ぶ
- devDeps に `@typescript/native-preview@7.0.0-dev.20260416.2` (Beta 公開直前の dev build) を保持
- `alt-frontend-sv-unit-test.yaml` の typecheck job が `bun run check` 経由で tsgo を実行
- `alt-frontend-sv/CLAUDE.md` も "TypeScript 7 (tsgo)" を primary と宣言

ただし **3 つの不整合** が残っていた:

1. **dev build 固定がリリース前の任意日付 (`20260416.2`)**: Beta 公開イベントと無関係な版で、再現性の根拠が「ある日たまたま入った」レベル。Beta タグが指す版に揃え直す動機がある
2. **`check:tsc` / `check:watch` が `tsc` (TS6) のままの fallback**: tsgo と tsc で型エラーが分岐した時の "どちらを真とするか" が曖昧。ローカル開発者が `check:watch` を使うと TS7 ではなく TS6 で検査される silent 不整合
3. **`alt-frontend-sv.yml` (Playwright) と `alt-frontend-sv-unit-test.yaml` (typecheck + vitest) が分離されている**: typecheck gate は後者にあるが、CI ログに tsgo の実版数が出ない (debug 不能)

加えて、SvelteKit 周りで懸念されていた tsgo 互換性 (`rootDirs` 不対応、`pnpm` + Prisma 型推論の false-positive) はそれぞれ [microsoft/typescript-go#1367](https://github.com/microsoft/typescript-go/issues/1367) と [#1034 (= #2185 の duplicate)](https://github.com/microsoft/typescript-go/issues/2185) で 2025-10〜2025-12 に解消済み。`sveltejs/kit#13970` も同期間に Closed。SvelteKit 2.57.1 の auto-generated `.svelte-kit/tsconfig.json` は既に `moduleResolution: bundler` / `target: esnext` / `module: esnext` を採用しており、TS7 の breaking changes (`target: es5` / `node10` / `classic` / `baseUrl`) はいずれも踏まない。

## Decision

**`alt-frontend-sv` の型検査経路を tsgo 1 本に統一する。** 具体的には次の 5 点を採用する:

1. **`@typescript/native-preview` を `7.0.0-dev.20260421.2` (= `@beta` 指す版) に exact pin** — npm 上に `7.0.0-beta.X` 形式の published version が存在せず `^7.0.0-beta` semver range は dev build を満たさないため、dist-tag が指す具体バージョンを直接固定する
2. **`check:tsc` script を削除** — `package.json` の `"check:tsc": "svelte-kit sync && svelte-check --tsconfig ./tsconfig.json"` を消す。`bun run check:tsc` は "Script not found" で fail-close する
3. **`check:watch` を `--tsgo` に切替** — TS6 fallback を残さない
4. **`typescript@^6.0.2` devDep は維持** — `svelte-check@4.4.6` の `peerDependencies.typescript: ">=5.0.0"` 制約のため。tsgo は型検査の実行系で svelte-check 自体の peer は別物
5. **CI workflow に tsgo version log step を追加** — `.github/workflows/alt-frontend-sv-unit-test.yaml` の `Type check` step の前に `bunx tsgo --version` を入れて、CI ログから tsgo 実版数を即時に追えるようにする

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| `^7.0.0-beta` semver range を使う | npm に `7.0.0-beta.X` 形式の published version が無く、`7.0.0-dev.YYYYMMDD.N` のみ。bun の semver matcher は pre-release tag 名 (`dev` vs `beta`) を別系統として扱うため `^7.0.0-beta` は実質 0 件マッチ |
| `"@typescript/native-preview": "beta"` (dist-tag 文字列をそのまま指定) | dist-tag は mutable で Microsoft が日次に動かす可能性がある。reproducibility が壊れる |
| `check:tsc` を残し定期的に並走させる | tsc と tsgo で「どちらが真か」が曖昧化する。Microsoft が tsgo の compiler test 99.6% (19,926 / 20,000) 通過 + semantics 構造同一を保証している前提に依拠してリスク受容する方が単純 |
| `typescript@^6.0.2` を削除 | `svelte-check@4.4.6` の peer 制約で install fail。svelte-check 自体が tsgo native ラッパに置き換わる将来 (`harshmandan/svelte-check-native` Rust 実装等) で再検討 |
| TS7 stable (2026-06 頃) を待つ | 既に dev build が production で動作中。Beta タグへの昇格は単に reproducibility を上げるだけで運用リスクは増えない |
| Stable 出荷後に `@latest` に切替 | 別タスクで実施。本 ADR は Beta タグ固定までを scope に含める |

## Consequences

### Pros

- **型検査セマンティクスが 1 系統に統一される** — tsgo / tsc で違うエラー集合を見る silent ドリフトが構造的に起きない。`bun run check` と `bun run check:watch` が同一のチェッカーを叩く
- **再現性が Beta リリース日に固定される** — 任意の dev build (`20260416.2`) から、Microsoft が公式 Beta タグで指した版 (`20260421.2`) に揃う。後発の dev build で挙動が変わっても本リポジトリは固定。CI / Docker / 手元で同一 tsgo binary を使う
- **CI ログから tsgo 版数が即座に分かる** — `bunx tsgo --version` step が `Type check` の直前で `Version 7.0.0-dev.20260421.2` を出力。tsgo 起因の typecheck 失敗時に「どの版で落ちたか」を最初の 5 秒で確定できる
- **Stable への昇格が小さなパッチで済む** — TS7 stable が出た時、`package.json` の 1 行 (`7.0.0-dev.20260421.2` → `^7.0.0`) と CLAUDE.md の "Beta" ラベル更新だけで完了する
- **scripts が読みやすくなる** — `check` / `check:watch` の 2 つに収束し、`check:tsc` の存在意義 ("万一の時の緊急 escape hatch") を考えなくてよくなる

### Cons / Tradeoffs

- **tsc fallback を失う** — Microsoft が tsgo の compiler test 99.6% 通過 + semantics 構造同一を保証していると言っても、残り 0.4% のうち `alt-frontend-sv` で踏みうるエッジケース (例: 未知の Svelte 5 runes 型推論パターン、Threlte の WebGPU intersection 型) で tsc-only の挙動が必要になった時、`bun add typescript@^6 && bunx svelte-check --tsconfig ./tsconfig.json` を都度叩くしかない。発生頻度は低いと判断するが zero ではない
- **`@beta` dist-tag 更新で他開発者が混乱しうる** — Microsoft が `@beta` を別 dev build に動かしても、本リポジトリは exact pin で動かない。`bun update @typescript/native-preview` を「最新を取りに行く」感覚で叩くと予期せず別版に飛ぶ。`bun update --to <version>` を使う運用ルールを README/CLAUDE.md に書くか検討の余地
- **`typescript@^6.0.2` は dead-weight として残る** — svelte-check が peer で要求するが実行経路としては使われない。devDeps install size に約 70MB / lockfile 上に未使用エントリが残る。svelte-check が tsgo native 対応する将来まで解消できない
- **scripts/deploy.sh 互換性** — 本 ADR は package.json / CI workflow / docs のみの変更で、本番ランタイム binary には影響しない。`alt-deploy` 経路 ([[000763]]) は alt-frontend-sv image を rebuild するが、Dockerfile は `bun run build` のみで tsgo を呼ばないため影響無し

## Verification

実施した検証 (2026-04-27 ローカル):

| Phase | コマンド | 結果 |
|---|---|---|
| tsgo binary | `bunx tsgo --version` | `Version 7.0.0-dev.20260421.2` |
| Script removal | `bun run check:tsc` | `error: Script not found "check:tsc"` (期待通り fail-close) |
| Lockfile | `bun.lock` の `@typescript/native-preview` 行 | `7.0.0-dev.20260421.2` (sha512 integrity 付き) に更新 |
| Type check | `bun run check` | 348 files / **0 errors** / 11 svelte 既存 warnings (CSS unused / a11y redundant role / state_referenced_locally) |
| Unit/Component | `bun run test` (vitest) | **107 files / 1109 tests, 全 pass** (22.05s) |
| Lint | `bun run lint` (biome) | 16 warnings + 3 infos (既存)、エラー無し、no fixes applied |
| Format | `bun run format` (biome) | 841 files clean |
| E2E (Playwright) | `bun run test:e2e` (auth + desktop-chromium + mobile-chrome 43 spec) | 別途実行・成果は pull request 添付 |

`vitest` 実行時の注意: `bun test` は **Bun 内蔵の test runner** を起動する別ルートで、現状 `vitest-browser-svelte@2.1.1` の internal module 初期化と非互換 (`coreRender` ReferenceError + segfault)。本リポジトリのユニット/コンポーネントテストは `bun run test` (= `vitest run`) を使う。CLAUDE.md にも明記した。
