# ADR-000477: useStreamUpdates ブラウザテストの非同期パターンを修正する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-frontend-sv (TypeScript / SvelteKit)

## Context

`useStreamUpdates` hook を getter + `$effect` パターンに変更し reactivity gap を修正した後、Vitest browser mode（Playwright / Chromium）で実行されるテスト 12 件のうち 5 件が FAIL していた。

### 失敗パターン

| テスト | 読む値 | 結果 |
|--------|--------|------|
| filter heartbeat | `pendingCount=1` | FAIL |
| coalesce updates | `pendingCount=2` | FAIL |
| fallback_to_unary | `isFallback=true` | FAIL |
| include item data | `pendingCount=1` | FAIL |
| applyUpdates | `pendingCount=1` | FAIL |

共通点: **`$state` が初期値から変更された後の値を読む必要があるテスト**が全て失敗。初期値のままのアサーション（`stream_expired`, `disconnect`）は成功していた。

### 試行した手法と結果

1. **`vi.waitFor()` + real timers** — `$state` の値が 5 秒間ポーリングしても変化しない。失敗。
2. **`vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync()`** — Vitest browser mode（Playwright）では `advanceTimersByTimeAsync` がブラウザの microtask queue を適切にフラッシュしない。失敗。
3. **`$effect` 内アサーション + `createHangingMockStream`** — 成功。本 ADR で採用。

## Decision

**2 つの修正を組み合わせて採用する。**

### 1. `createHangingMockStream` — mock stream のライフサイクル修正

従来の `createMockStream` は全イベント配信後に `done: true` を返していた。これにより `connect()` が正常終了 → `scheduleReconnect()` → 1 秒後に再接続 → exhausted stream で即終了 → 再度 reconnect、というサイクルが 3 秒の COALESCE_DELAY より先に `disconnect()` 経由で `coalesceTimer` をクリアしていた。

`createHangingMockStream` はイベント配信後に永続的に pending な Promise を返し、実際の SSE ストリームと同じ「開きっぱなし」の振る舞いを再現する。

```typescript
function createHangingMockStream(events) {
  let index = 0;
  return {
    [Symbol.asyncIterator]() {
      return {
        next() {
          if (index < events.length) {
            return Promise.resolve({ value: events[index++], done: false });
          }
          return new Promise(() => {}); // hang forever
        },
      };
    },
  };
}
```

### 2. `waitForEffect` — `$effect` 内リアクティブアサーション

Svelte 5 の `$state` signal は `$.get(signal)` で常に最新の `signal.v` を返すが、**リアクティブな通知**は `$effect` 内で読み取った場合にのみ機能する。`vi.waitFor()` のようなポーリングはリアクティブ context 外で `$.get()` を呼ぶため、Batch flush による effect 再実行トリガーを受け取れない。

`waitForEffect` は `$effect.root` + `$effect` で watcher を構成し、assertion 内で `$state` getter を読むことで signal 依存を登録する。signal が変更されると Svelte の batch 機構が effect を dirty マークし再実行、assertion が通れば Promise を resolve する。

```typescript
function waitForEffect(assertion: () => void, timeoutMs = 8000): Promise<void> {
  let cleanupRoot: (() => void) | null = null;
  const promise = new Promise<void>((resolve, reject) => {
    const timer = setTimeout(() => {
      try { assertion(); resolve(); }
      catch (e) { reject(e); }
    }, timeoutMs);

    cleanupRoot = $effect.root(() => {
      $effect(() => {
        try {
          assertion();
          clearTimeout(timer);
          resolve();
        } catch { /* wait for next reactive update */ }
      });
    });
  });
  return promise.finally(() => cleanupRoot?.());
}
```

### 検討した代替案

| 案 | 評価 |
|----|------|
| `vi.useFakeTimers()` + `advanceTimersByTimeAsync()` | Vitest browser mode で microtask flush が不完全。却下。 |
| `vi.waitFor()` + real timers (polling) | `$state` 読み取りがリアクティブ context 外のため signal 変更を検知できない。却下。 |
| プロダクションコードに `_onStateChange` callback を追加 | テストのためだけにインターフェースを汚染。最終手段として保留。 |

## Consequences

### Pros

- **12/12 テスト PASS** — browser (7 既存 + 5 修正) と Node (4) の全テストが通る
- **mock stream が実際の SSE 動作に近い** — `createHangingMockStream` は reconnection noise を排除し、テストの意図が明確になる
- **Svelte 5 runes のテストパターンが確立** — `waitForEffect` は他の `$state` 依存テストにも再利用可能
- **プロダクションコード変更なし** — テストファイルのみの修正

### Cons / Tradeoffs

- **COALESCE_DELAY (3 秒) が実時間で経過する** — `pendingCount` を検証する 4 テストはそれぞれ約 3 秒かかる（合計約 12 秒）。fake timers が使えないため受容する
- **`createHangingMockStream` の Promise リーク** — `new Promise(() => {})` は GC されるまで pending のまま。テスト単位のライフサイクルでは問題にならないが、大量テストでは注意
- **Vitest browser mode 固有の制約への依存** — 将来 Vitest が microtask flush を改善した場合、fake timers パターンに戻せる余地がある
