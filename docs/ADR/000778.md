# ADR-000778: alt-backend に Hurl E2E スイートを追加し 3 サービス共通 dispatch 規約の第 5 増分に載せる

## Status

Accepted

## Date

2026-04-19

## Affected Services

- alt-backend (Go) — プロダクションコードは触らず、テスト資産のみ追加:
  - `e2e/hurl/alt-backend/` 配下に 22 本の `.hurl` + README + `run.sh`
  - `compose/compose.staging.yaml` に新規 profile `alt-backend`
  - `.github/workflows/e2e-hurl.yml` に path-gated job 追加

## Context

[[000763]] で Hurl を E2E 駆動の基盤として導入し、[[000764]] mq-hub、[[000765]] knowledge-sovereign、[[000766]] search-indexer と段階的に suite が揃い、`bash e2e/hurl/<svc>/run.sh` が 3 サービス共通の dispatch 接合点として確立された。その後 tag-generator / news-creator / recap-worker / acolyte-orchestrator まで拡張し、計 7 サービスが同じ規約で動いている。

一方、alt-backend は Alt の中核サービスで REST 50+ エンドポイント + Connect-RPC (port 9101) + admin dashboard を持つが、E2E では未カバーだった。alt-backend 側に存在するのは Pact CDC consumer 契約（outbound で呼ぶ search-indexer / pre-processor 等への呼び出し契約）のみで、REST 全域をブラックボックスで叩く suite がない。

- 新機能追加時に REST 層の regression を検出できない。CORS / middleware / CSRF / rate-limit の wiring が壊れても単体テストでは気づきにくい
- 他サービスが alt-backend を呼ぶ Pact provider 側検証は流れているが、**alt-backend が自分の REST surface を守る** 責務は抜け落ちていた
- 段階展開の方針 ([[000766]] follow-up) が「次は alt-backend」であり、public repo 側に dispatch 接合点を揃えるのが Increment 5 の範囲

既存 suite と別規約で始めてしまうと、将来 `release-deploy.yaml` の matrix 化 ([[000766]] Scope out) で alt-backend だけ例外処理が必要になり、dispatch 経路が再び割れる。

## Decision

**`e2e/hurl/alt-backend/` を 1 セット追加し、[[000766]] の規約を一切変更せずに第 5 増分として載せる。** 具体構成:

1. **run.sh は knowledge-sovereign/run.sh を雛形にする**。`set -euo pipefail` + `trap cleanup EXIT` + `IMAGE_TAG` / `GHCR_OWNER` / `HURL_IMAGE` / `RUN_ID` / `KEEP_STACK` の env 規約をそのまま踏襲。Hurl は Docker で `--network alt-staging -v "$ROOT:$ROOT"` として起動し、`--file-root "$ROOT"` で `file,e2e/fixtures/...;` を解決する。
2. **`--profile alt-backend` で staging stack を起動**する。`compose/compose.staging.yaml` に新規 profile (`alt-backend-db` / `alt-backend-db-migrator` / `alt-backend-deps-stub` / `alt-backend`) を追加。既存 profile との干渉はない（profile 未指定サービスは compose デフォルトで起動しない）。
3. **setup と本体を分離する**。`00-setup.hurl` は `--jobs 1` 相当で先行し REST :9000 と Connect-RPC :9101 の両方を readiness probe する（`retry: 60` でコンテナ healthcheck の settling を吸収）。本体 `0[1-9]-*.hurl` / `[1-9][0-9]-*.hurl` は `--jobs 4 --retry 5 --retry-interval 500` で並列実行。
4. **scenario は domain 単位に番号帯を切る**（01 系: health/csrf/auth 負、10 系: rss-feed-link CRUD、20 系: feeds、30 系: articles、40 系: morning-letter、50 系: images、60 系: augur/rag、70 系: admin dashboard + scraping-domain、80 系: csp-report）。順序依存のある書き込み→読み取り→削除は単一ファイル内で `[Captures]` を使って連鎖させ、ファイル間の隠れ依存は作らない。
5. **CI は `.github/workflows/e2e-hurl.yml` に新規 `alt-backend` job を追加する**。既存 mq-hub job をそのまま雛形とし、path filter に `alt-backend/**` / `e2e/hurl/alt-backend/**` / `e2e/stubs/alt-backend-deps-stub/**` / `e2e/fixtures/alt-backend/**` / `migrations-atlas/**` / `compose/compose.staging.yaml` / workflow 自身を列挙して無関係 PR で走らないようにする。
6. **alt-deploy private repo の matrix 化は本 ADR の対象外**。[[000766]] §Scope out と同じ方針で、public 側の dispatch 接合点追加に留める。private 側 PR は別タイミングで起こす。

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| Pact provider 側検証だけで十分とする | Pact は「alt-backend を呼ぶ側の期待」を守るだけで、alt-backend の REST surface (CORS / CSRF / rate limit / middleware chain / error envelope) は検証しない。Hurl suite は alt-backend 自身の HTTP contract を守る責務を持つ |
| compose.staging.yaml の既存 profile を広げて alt-backend + 全実サービスを起動する | search-indexer / pre-processor / recap-worker / rag-orchestrator / knowledge-sovereign / mq-hub を全部立てると build 時間と依存 DB が 7 個増える。E2E suite のフィードバック時間が分単位で悪化し、`--jobs 4` 並列の意味が消える |
| Hurl ではなく Playwright / k6 で REST E2E を書く | 既存 7 サービスの dispatch 規約と揃わず、`release-deploy.yaml` の matrix 化時に alt-backend だけ例外処理が必要になる。alt-frontend-sv の Playwright は UI E2E 専任で責務が異なる |
| 初回 PR で Connect-RPC 全メソッドも scenario 化する | Connect-RPC の method 契約は Pact CDC が担う（[[000591]]）。Hurl は port readiness と REST surface の健全性に集中し、RPC method は 2 段階目で別 ADR を起こす |

### 範囲外 (今回やらないこと)

- alt-deploy private repo の `release-deploy.yaml` matrix 化 — [[000766]] と同じ方針で別 PR
- `POST /v1/feeds/summarize/stream` の NDJSON chunked 検証 — 後続 scenario で対応
- Connect-RPC method contract の深い検証 — Pact consumer/provider で担保
- 実 search-indexer / pre-processor との相互作用検証 — 各サービス自身の Hurl suite と Pact が責務

## Consequences

### Pros

- **public repo 側の dispatch 接合点が alt-backend まで揃った**。`bash e2e/hurl/<svc>/run.sh` が 8 サービス共通になり、`release-deploy.yaml` の matrix 化時に alt-backend が追加対象としてそのまま乗る
- **REST surface の regression ゲートが PR ごとに走る**。21 scenario で CORS / CSRF / middleware chain / admin RBAC / SSRF / DOS whitelist を横串検査。21/21 green + 2 連続安定実行を local で確認済み
- **オンボーディング摩擦が他サービスと同形**。新規メンバーが alt-backend の E2E をローカル再現する時、他 7 サービスと同じ 1 行コマンド (`bash e2e/hurl/alt-backend/run.sh`) で動く。README の narrative も同じフォーマット
- **path-gated CI で無関係 PR は alt-backend job をスキップ**する。docker build + stack bring-up コスト (~60 秒) は実際に alt-backend / suite / compose staging を触った PR だけに限定

### Cons / Tradeoffs

- **8 サービス目の run.sh が増えた**。[[000766]] Cons で触れた「共通シェル関数抽出」の保留が続く。4 サービスを超えて boilerplate の重複が顕在化しているため、次の増分（Increment 6）までに `hurl_in_network` 抽出の ADR を別途起こすべき
- **初回 PR が 7 commit に分割された**。fixture / scenarios / stub / compose profile / CI / format / green-up fix の 5 フェーズ + 軽微な 2 commit 構成で、1 PR として読む時の history は長め。ただし TDD 的な RED (scenarios) → GREEN (stub + compose) → CI の線が明確なのでレビュー負荷は実質低い
- **staging 専用 fixture を `e2e/fixtures/staging-secrets/` にもう 1 個増やした**。[[000763]] で合意済みのパターンではあるが、fixture rotation 時に JWT の再発行が必要になる（exp=2099 で 70 年の猶予）
- **run.sh 内の `--jobs 4` 並列は alt-backend 本体の並行処理耐性を前提にしている**。書き込み系の scenario は単一ファイル内で順序付け、ファイル間では URL に `run_id` 相当の分離を入れて衝突を避けているが、将来 scenario が増えて race が顕在化したら `--jobs` を下げる運用見直しが必要
