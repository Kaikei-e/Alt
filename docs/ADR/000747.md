# ADR-000747: mTLS cert ライフサイクルを compose 埋め込み shell から専用 Go サイドカー pki-agent に移行する

## Status

Accepted

## Date

2026-04-16

## Affected Services

- **pki-agent** (新規 Go サービス) — `pki-agent/` 一式 (cmd/pki-agent/main.go,
  internal/{domain,usecase,adapter/handler,infrastructure}, config/config.go,
  Dockerfile, go.mod, go.sum, CLAUDE.md)
- **compose 基盤** — `compose/pki.yaml` に `x-pki-agent` anchor + 8 本の
  `pki-agent-<svc>` サービス追加、`compose/{core,bff,auth,workers,recap,ai,acolyte}.yaml`
  から `*-cert-init` / `*-cert-renewer` を全て削除、consumer の `depends_on` を
  `pki-agent-<svc>: service_healthy` に付け替え
- **observability** — `observability/prometheus/prometheus.yml` に pki-agent 8
  target の scrape job、`observability/prometheus/rules/pki-agent-alerts.yml`
  (CertExpirySoon / CertExpired / RenewalFailing / Down の 4 アラート)
- **secrets** — `secrets/.gitignore` 新設 (誤コミット対策)
- **runbook** — `docs/runbooks/pki-agent-recovery.md`
- **security audit** — `docs/review/pki-agent-security-audit-2026-04-16.md`

## Context

### きっかけの障害

2026-04-15 14:28:20Z、alt-backend の mTLS サーバー証明書が期限切れ。BFF (alt-butterfly-facade)
から alt-backend へ行く全 Connect-RPC (`GetKnowledgeHome`, `ListLenses`,
`StreamKnowledgeHomeUpdates`, `ListSubscriptions`, `StreamFeedStats` 等) が
`tls: failed to verify certificate: x509: certificate has expired` で失敗し、
**Knowledge Home が死んだ** (UI は空、feeds も出ない)。

step-ca は healthy だったが、調査の結果以下が判明:

1. 8 サービス分の `*-cert-init` / `*-cert-renewer` サイドカーが **ゼロ稼働**
   (docker ps -a にも残っていない)。過去の部分的な `docker rm` などで消去された後、
   再起動されずに放置されていた
2. step-ca の leaf 有効期限は 24h デフォルトのため、renewer 不在でそのまま期限切れ到達
3. 既存の compose 埋め込み shell `cert-init` スクリプトには

   ```sh
   if [ -f /certs/svc-cert.pem ]; then exit 0; fi
   ```

   という早期 exit があり、**期限切れの svc-cert.pem がファイルとして残っていると再発行されない**
   重大な欠陥

### 構造的な弱点

- compose YAML 内にインライン shell (set -e, `$$` escape, pipefail の脆さ)
- smallstep CLI のバージョン変更で挙動が変わっても気付けない
- renewer の死活が compose の label だけでは見えず、Prometheus metric が無い
- 期限切れ cert 残存時の再発行フォールバックが無い
- `step ca renew` は期限切れ cert では動かない (client auth 失敗) ため、
  期限切れ fallback 経路が必要だが shell 版には無い

`feedback_tdd_strict` / `alt-paper` 系の設計哲学に反し、**テストが書けないグルー**が
本番 east-west 境界の生命線を預かっている状態だった。

### ユーザー指示

> 「そもそも、そんな軟弱なシェルスクリプトに頼っているからいけないんじゃない？
> 専用の超小型責務のマイクロサービスを Go で作って」

## Decision

1. **新規サービス `pki-agent`** を Go 1.26.2 で作り、8 consumer 分 1:1 のサイドカーとして
   配置する。責務は 1 つだけ: 「対象 volume の `svc-cert.pem` / `svc-key.pem` を
   常に有効期限内に保つ」。既存 `*-cert-init` + `*-cert-renewer` の 16 本を 8 本に半減する
2. ライフサイクルの state machine は Go で書いて `go test -race` で TDD ゲートに乗せる。
   OTT 発行と CSR 署名だけ `step-cli` を `os/exec` で呼ぶ (CA プロトコル知識を再発明しない)
3. 期限切れ時は `step ca renew` ではなく **新 OTT を使った re-enrollment** にフォールバック
   (Smallstep 公式推奨。`--allow-renewal-after-expiry` は security risk のため非採用)
4. Cert は atomic write (tmpfile + rename)、chown/chmod は rename 前の tmpfile に適用して
   TOCTOU を断つ
5. `tls.Config.GetCertificate` ホットリロード (`auth-hub/tlsutil/tlsutil.go:44-97`) と
   組み合わせ、**consumer 再起動なしで cert rotation** が成立する
6. Prometheus metrics (`pki_agent_cert_remaining_seconds` 等) + 4 本のアラートルールで
   **期限切れ 4h 前・0h 時点・renewal loop 失敗・プロセス down を自動検知**
7. セキュリティ監査 (OWASP Top 10:2025 / ASVS 5.0) で Critical 1 / High 3 を検出。
   `secrets/.gitignore` 新設で F-003 を即時閉止、`x509.allowedNames` 付きの専用
   provisioner は Phase 2.5 として次 ADR に繰り延べ

## Architecture

```
pki-agent/                         # single-responsibility sidecar
├── cmd/pki-agent/main.go          # signal handling, ticker loop
├── internal/
│   ├── domain/                    # entities + sentinel errors + port interfaces
│   │   ├── cert.go                # CertState, ClassifyRemaining() (pure)
│   │   ├── errors.go              # ErrCertNotFound, ErrCAUnreachable, ...
│   │   └── port.go                # CertLoader, CAIssuer, CertWriter, Observer
│   ├── usecase/
│   │   └── rotate.go              # Tick() state machine, depends on domain only
│   ├── adapter/handler/server.go  # /healthz + /metrics
│   └── infrastructure/
│       ├── certfile.go            # atomic write + Load (domain.CertLoader + CertWriter)
│       ├── stepca.go              # step-cli subprocess wrapper (domain.CAIssuer)
│       └── metrics.go             # Prometheus promauto observer (domain.Observer)
├── config/config.go               # env + _FILE secret parsing
├── Dockerfile                     # FROM smallstep/step-ca:0.27.5 (step-cli)
│                                  # + copied Go binary from golang:1.26.2-alpine
├── go.mod / go.sum
└── CLAUDE.md
```

層の依存方向は **Handler → Usecase → Port (domain) → Gateway/Infrastructure** で
auth-hub / mq-hub と同一 (clean-architecture skill 準拠)。

## Consequences

### 良い結果

- **Knowledge Home 復旧**: Phase 0 で既存 shell cert-init/renewer を再起動して即時復旧。
  BFF ログから cert expired エラー消滅、Knowledge Home / feeds / subscriptions が再生
- **再発防止**: 4 本のアラートが Grafana → Prometheus で cert 期限切れ発生前に発火する。
  `pki_agent_cert_remaining_seconds < 4*3600` で page、`pki_agent_healthy == 0` で page
- **TDD ゲートの復活**: ライフサイクルロジックが `go test ./... -race` で回る。現状 4 ファイルの
  単体テスト (domain/cert_test.go, usecase/rotate_test.go, infrastructure/certfile_test.go,
  infrastructure/stepca_test.go) すべて GREEN
- **secrets 誤コミット経路を閉止**: `secrets/.gitignore` 追加
- **16 → 8 に sidecar 数を半減**: 運用ノイズ・リソース・起動時間削減

### 残リスク (Phase 2.5 以降)

- **F-001 (Critical)**: bootstrap JWK provisioner を 8 pki-agent が共有する状態は
  初版そのまま。侵害時の blast radius 縮退のため、次 ADR で以下を実装する
  - 専用 `pki-agent` JWK provisioner を `step ca provisioner add` で作成
  - `x509.allowedNames.commonNames` に 8 CN を allowlist
  - pki-agent の `STEP_CA_PROVISIONER=pki-agent` に切替
- **F-007 (Medium)**: tag-generator / acolyte-orchestrator は nginx TLS sidecar 経由の
  ため、cert rotation 時に SIGHUP が必要。現状 nginx sidecar は再起動に任せているが、
  `inotifywait -e create,moved_to /certs` からの `nginx -s reload` を入れる計画を
  Phase 2.5 に含める
- **observability**: Grafana ダッシュボード JSON はまだ書いていない。`pki_agent_*` を
  カード化する panel を別 PR で追加

## Alternatives considered

1. **shell scripts を TDD 化して延命** — bash で TDD は本質的に弱く、step CLI の
   subcommand 挙動変更への追従コストが残る。却下
2. **smallstep/certificates/ca Go ライブラリで full native 化** — OTT 生成が JWK
   の JWE decrypt を含み、実装量と差分テストコストが増える。step-cli subprocess で
   「CA プロトコル層は smallstep 公式に委任、オーケストレーションは Go」が最小コスト・最大
   robustness のバランス。Phase 3 以降で native 化を再評価
3. **中央集権 `pki-sentinel` 1 プロセスで 8 cert 全面倒** — blast radius が拡大し、
   単一プロセスが死ねば全 mTLS が停止。既存 1:1 sidecar パターンと相性が悪い。却下
4. **cert-manager (k8s) 相当を Alt 独自に** — k8s 依存 or 巨大プロジェクトを取り込む
   ことになり、docker-compose-first の原則に反する。却下

## Implementation record

Phase 0: 2026-04-16 00:25 JST に期限切れ cert 8 volume を削除、既存 shell
cert-init/renewer を `docker compose up -d --force-recreate` で再起動、consumer を
restart して Knowledge Home 復旧 (15:26Z BFF restart、cert expired error 0/min)。

Phase 1: pki-agent Go モジュール新設、`go test ./... -race` で全テスト GREEN。
`docker build -t alt/pki-agent:dev .` 成功。実 step-ca 相手の smoke test で
initial tick ok / state=fresh / `pki_agent_cert_remaining_seconds` が 86390s を
返すことを確認 (15:35Z)。

Phase 2: compose cutover。8 pki-agent サイドカーを `docker compose up -d --build`
で起動、全部 healthy、既存 cert は fresh 判定で no-op、consumer の mTLS は継続。
`docker ps` で 16 旧サイドカー消滅、8 新サイドカー Up 確認 (15:42Z)。

Phase 3: Prometheus `prometheus.yml` に pki-agent scrape job 追加、
`pki-agent-alerts.yml` 4 ルール、prometheus restart で反映、
`pki_agent_cert_remaining_seconds{subject="recap-worker"} = 23.73h` を PromQL 経由で
取得できることを確認 (15:45Z)。

## Pact / contract-testing scope

pki-agent は **`DEPLOY_PACTICIPANTS` に追加しない**。理由を明示しておく:

- pki-agent は east-west の Connect-RPC 参加者ではない。consumer-facing の
  provider API を持たず、外部との境界は次の 3 つ:
  1. **step-ca への HTTPS 呼び出し** — smallstep の proprietary protocol。相手は
     ベンダ製でバージョン互換性は smallstep 側がハンドル。Pact で再契約しない
  2. **consumer との filesystem 境界** — `/certs/svc-cert.pem` の PEM 出力。形式は
     RFC 7468 で決まっており Pact の適用範囲外 (contract = OS + x509 標準)
  3. **`/metrics` / `/healthz` の HTTP** — Prometheus exposition 形式と 200/503 の 1 bit。
     可観測性プロトコルであり consumer は Prometheus のみ。consumer 独自実装ではない
- 故に **Pact CDCT の対象境界が無い**。ADR-000746 の `scripts/deploy.sh:33`
  `DEPLOY_PACTICIPANTS` に pki-agent を追加すると、毎 deploy で空の pact verification が
  false-negative を吐き続ける結果になる
- ただし pki-agent 自体の挙動テストは Alt の TDD ガードに含める:
  - Unit: `go test ./... -race` (現状 4 テストファイル, all GREEN)
  - 擬似結合: `StepCACLI_Issue` を fake step binary に対して検証済
  - 実環境 smoke: `docker run` で step-ca に対して cert 発行成功を確認 (Phase 1 記録)
- 将来 pki-agent に Connect-RPC エンドポイント (例: 管理者向け `/admin.Rotate`) を
  足す場合は、その RPC は alt-backend / auth-hub と同じ Pact フロー (`pacts/` consumer
  pact + `can-i-deploy`) に乗せる。**現バージョンではエンドポイントが無いので非対象**
