# ADR-000917: Mobile TTS をオーバーフローメニュー起点の bottom-sheet + layout-mounted mini-player で導入する

## Status

Accepted

## Date

2026-05-25

## Affected Services

- alt-frontend-sv (Svelte 5 / SvelteKit / TypeScript 7) — mobile reader (`/articles/[id]`) に TTS UI を追加。既存の `useTtsPlayback` (per-component hook) と並行して、`(app)/+layout.svelte` に singleton `TtsPlaybackStore` を context 注入し、`ResponsiveLayout.svelte` の mobile 分岐に `TtsMiniPlayer` をマウントする

## Context

mobile reader にはこれまで TTS の入口がなく、サーバーサイド TTS (`alt.tts.v1.TTSService/SynthesizeStream` + Web Audio gapless scheduler) は desktop の `FeedDetailModal` / `ArticleDetailPanel` / `RecapDetail` / Home からしか到達できなかった。プロダクト判断は「optional, low-frequency」だが、mobile reader の主体は片手スクロール読書であり TTS の需要は副次的に存在する。primary action bar を埋めずに、必要な時だけ確実に到達できる入口が必要だった。

Web 調査 (2026-05-25; MDN / WebKit blog / NN/G / Sarah Higley / Mobbin 等の S/A tier ソース) の要点:

- iOS Safari Reader View・Edge mobile はいずれも TTS を overflow menu に格納し、再生中は top mini-bar を出す。FAB は HIG / NN/G ともに「最頻 primary action」用途で、副次機能には不適。
- Bottom sheet は NN/G が「progressive disclosure・常時表示しない情報向け」と明文化。`role="dialog" aria-modal="true"` + Close button が必須。
- 再生 UI の永続化は Spotify/YouTube Music 型の「layout 直下に singleton mini-player」が現状のコンセンサス (Mobbin 2024-25 corpus)。SvelteKit では `+layout.svelte` 直下に component を置くと route 遷移で unmount されない。
- Media Session API は `HTMLMediaElement` 駆動が前提で、`speechSynthesis`/独自 AudioContext 再生では呼び出せない。lock-screen / 背景再生は platform 制約として **v1 非対象**。
- Web Speech API の iOS Safari 既知問題 (15s cut-off / `voiceschanged` async / background pause) は本実装には**該当しない**。Alt の TTS はサーバーで WAV をストリーミング合成し、クライアントは `SeamlessTtsPlayer` (`AudioContext` + sample-accurate scheduling) で再生するためである。

実装上の既存資産:

- `useTtsPlayback.svelte.ts` — per-component hook。`createClientTransport` → `synthesizeSpeechStream` → `splitTextForTts(30_000)` → `SeamlessTtsPlayer.append/done/cleanup` の play loop を保持
- `src/lib/components/ui/sheet/` — bits-ui Dialog ベースの bottom-sheet wrapper (`side="bottom"` 対応)
- `src/lib/stores/auth.svelte.ts` — class + `$state` + factory + Symbol context key + `getContext` retriever の canonical な store パターン

per-component hook は layout-mounted mini-player の backing には使えない (component unmount で state が消える)。「一度に再生できる TTS は app 全体で 1 本」というドメイン制約とも合わない。

## Decision

mobile TTS は次の 4 要素で構成する。

1. **singleton store**: `TtsPlaybackStore` (`src/lib/stores/ttsPlayback.svelte.ts`) を `(app)/+layout.svelte` で `createTtsPlaybackStore()` → `setContext(TTS_PLAYBACK_KEY, ...)`。`AuthStoreImpl` パターンを踏襲した class + `$state` 実装。`stop()` / `play()` 入場時に **per-invocation token** を bump し、`isActive()` で stale loop が新しい state を踏まないよう保証する。設定は `TtsPreferencesStore` (`src/lib/stores/ttsPreferences.svelte.ts`) に速度 (0.75 / 1.0 / 1.25 / 1.5 / 2.0) を localStorage 永続化する。
2. **discoverable but secondary trigger**: `ArticleOverflowMenu.svelte` を action bar 末尾に置く。bits-ui の `DropdownMenu` ではなく素の `<button> + role="menu"` 構成にした (browser-test 環境での hydration 不具合を踏まえた現実解)。Listen が唯一の項目だが、将来 Share / Bookmark を追加する受け皿として残す。
3. **setup sheet**: `TtsSetupSheet.svelte` を `Sheet.Root side="bottom"` で実装。Source (Summary / Article) を radio で選び、Body fetch 前なら Article、要約取得前なら Summary を disabled にする。Speed chips、VoiceOver 二重発話の注意書きを併設。Start ボタンの click が AudioContext を unlock する user gesture を兼ねる。
4. **layout-mounted mini-player**: `TtsMiniPlayer.svelte` を `ResponsiveLayout.svelte` の mobile 分岐に 1 個だけマウントする。`MobileBottomNav` の上 (`bottom: calc(2.75rem + safe-area)`) に sticky 配置し、`store.isActive && store.track` の時だけ region landmark として render。Play/Pause は ARIA name change (`Sarah Higley pattern`)、Close は `store.stop()` で playback を終了する。

useTtsPlayback hook と TtsPlaybackStore で重複していた play loop は `src/lib/utils/ttsStream.ts` (`runTtsStreamLoop`) に切り出し、両者から `isCancelled` / `onChunkAppended` callback を渡して共通化した。store / hook はそれぞれ自分の state 機械を維持し、helper はネットワーク → player の handoff だけ受け持つ。

### 検討した代替案と却下理由

| 代替案 | 却下理由 |
|---|---|
| primary action bar に Headphones アイコンを追加 | primary action と誤読される。「optional, low-frequency」前提に反し、Fetch/Summarize/Open と視覚的に等価に並ぶ |
| floating action button (FAB) | NN/G・iOS HIG の双方で primary action 用途とされる。本文を覆い、副次機能には過剰 |
| 記事末尾の inline CTA「この記事を聴く」 | 初見時の discoverability が低い。記事が短い時 / fetch 前の状態と相性が悪い |
| Web Speech API へのフォールバック | iOS Safari の既知不具合 (15s cut-off, voiceschanged race, background pause) を抱える。サーバー側で品質コントロールできなくなる |
| Media Session API による lock-screen 制御 | `HTMLMediaElement` に紐付かない再生方式では発火しない。silent `<audio>` を裏で鳴らす hack は WebKit で確実性が乏しいため v1 では非対象 |
| 句単位 (±1 sentence) skip ボタン | server WAV chunk 境界に client が seek する仕組みが要る。v1 の scope を逸脱 |
| per-component hook を layout 共通化せず page 直下にだけ mini-player を置く | route 遷移で unmount。Spotify 型の「再生は続けたまま読み物を移れる」UX が成立しない |
| desktop の hook 呼び出しも一括で store に寄せる | 既存 3 component の振る舞いを変えるリスクがある。本 ADR の scope は mobile 導入に絞り、helper 切り出しによる重複除去だけ既存に波及させた |

## Consequences

### Pros

- mobile reader でも TTS にアクセスできる。Fetch/Summarize/Open は今まで通りで、low-frequency という前提を UI レイアウトに正しく反映できた。
- route 遷移 (`/articles/[id]` ↔ `/home` ↔ `/feeds/swipe` 等) を跨いで再生が継続する。E2E (`tests/e2e/mobile/articles/tts.spec.ts`) で SPA navigation 越しの mini-player 残存と Close 押下による解除を担保している。
- store の per-invocation token により、2 連続 `play()` で stale loop が新しい state を上書きする race を構造的に防止できた (`useTtsPlayback` の closure-flag 方式では潜在的に存在)。token bump + `isActive()` ガードはユニットテスト (`ttsPlayback.test.ts` の "cancels a previous playback") で固定。
- play loop を `runTtsStreamLoop` に切り出したことで、hook と store のロジックが 1 箇所に集約された。今後 TTS の chunking / cancellation 戦略を変える時は helper 1 ファイルだけ触ればよい。
- a11y baseline (role=menu / role=dialog + aria-modal / Sarah Higley の name-change play-pause / region landmark) を最初から備えており、後付け改修が不要。

### Cons / Tradeoffs

- mini-player は browser tab が background になると pause する。これは platform 制約だが、Setup sheet 内の注意書きで明示しユーザ期待値を合わせている。lock-screen 操作も提供しない。
- v1 の play/pause トグルは pause = stop。SeamlessTtsPlayer に「途中再開」の primitive が無いため、本当の意味の pause は別 ADR で扱う。
- bits-ui `DropdownMenu` を採用せず `<button> + role="menu"` を手書きしたため、複数項目化した時にキーボードナビ (Arrow Up/Down) を自前で書き足す必要がある。1 項目しかない現状では負担が無いが、将来 Share/Bookmark を入れる際に再考の余地がある。
- desktop の `useTtsPlayback` 利用箇所 (`FeedDetailModal` / `ArticleDetailPanel` / `RecapDetail` / Home) は今回手を入れていない。実行時に「desktop で再生中・mobile に viewport を縮めて TTS UI を触る」のような edge case は想定外として残している。
- E2E の WebKit (mobile-safari) 実行はローカルに Playwright WebKit binary が無いため未実施。`mobile-chrome` のみ通っている。CI 環境で `mobile-safari` の binary を引いた時に同じ flow が通る前提。
