# ADR-000407: auth-hub Clean Architecture レイヤーで JWT ロール伝搬を実装する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- auth-hub (Go) — `internal/domain`, `internal/adapter/gateway`, `internal/infrastructure/token`, `internal/usecase`

## Context

[[000405]] で設計された admin ロール伝搬パイプラインは、auth-hub のレガシー `client/` パスについて記述していた。しかし auth-hub は Clean Architecture への移行途上にあり、新しい `internal/` レイヤー（domain → usecase → adapter/gateway → infrastructure）でも同等の修正が必要だった。

具体的には、新 CA レイヤーに以下の欠陥があった:

1. **`domain.Identity`** に `Role` フィールドがない
2. **`KratosGateway.ValidateSession`** が Kratos traits から `role` を抽出しない（`email` のみ抽出）
3. **`JWTIssuer.IssueBackendToken`** が `Role: "user"` をハードコード
4. **`domain.CachedSession`** に `Role` フィールドがない
5. **`get_session.go`** の `SessionResult.Role` が `"user"` ハードコード

これにより、Knowledge Home Admin ページ（`/admin/knowledge-home`）が BFF 経由で alt-backend に接続する際、JWT の `role` claim が常に `"user"` となり HTTP 403 が発生していた。SvelteKit 側は Kratos traits から直接 `role: "admin"` を読み取れるためページアクセスは許可するが、BFF 経由の API 呼び出しがすべて失敗する不整合が生じていた。

## Decision

**auth-hub の新 Clean Architecture レイヤーで、レガシー `client/kratos_client.go` と同等のロール抽出ロジックを実装する。**

### 変更箇所（5ファイル）

| ファイル | 変更 |
|---------|------|
| `internal/domain/session.go` | `Identity` に `Role string` 追加、`CachedSession` に `Role string` 追加 |
| `internal/adapter/gateway/kratos.go` | `ValidateSession` で `traits["role"]` を抽出（`"admin"` のみ許可、他は `"user"`） |
| `internal/infrastructure/token/jwt.go` | `identity.Role` を使用し、空の場合 `"user"` にフォールバック |
| `internal/usecase/get_session.go` | cache hit/miss 両パスで `role` を伝搬、`SessionResult.Role` に反映 |
| `internal/domain/port.go` | 構造体変更のみ（`CachedSession` に `Role` 追加で自動的に対応） |

### TDD テスト

| テスト | 検証内容 |
|-------|---------|
| `TestGetSession_AdminRole_CacheMiss` | Kratos から admin role → SessionResult.Role = "admin"、cache に Role 保存 |
| `TestGetSession_AdminRole_CacheHit` | cache に Role="admin" → SessionResult.Role = "admin" |
| `TestGetSession_EmptyRole_DefaultsToUser` | Role="" → "user" にフォールバック |
| `TestJWTIssuer_AdminRole` | Role="admin" → JWT claim の role = "admin" |
| `TestJWTIssuer_EmptyRole_DefaultsToUser` | Role="" → JWT claim の role = "user" |

## Consequences

### Pros

- **403 エラー解消** — admin ユーザーの Knowledge Home Admin ページが正常に機能する
- **レガシー互換** — `client/` パスと `internal/` パスで同じロール抽出ロジックを使用
- **後方互換** — `Role` フィールドが空の既存ユーザーは `"user"` にフォールバック
- **テスト網羅** — cache hit/miss 両パスおよび JWT 生成で admin/user/empty のケースをカバー

### Cons / Tradeoffs

- **二重実装** — `client/kratos_client.go` と `internal/adapter/gateway/kratos.go` に同じロール抽出ロジックが存在する。レガシーパスの廃止時に統合が必要
- **allowlist の分散** — `"admin"` のみ許可するロジックが複数箇所にある。将来のロール追加時は全箇所の更新が必要
