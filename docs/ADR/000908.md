# ADR-000908: WhyPayload に counter-evidence と confidence ladder を、Knowledge Loop に Act outcome event と internalized dismiss state を additive で導入し loop closure / 認知卒業概念を契約に下ろす

## Status

Accepted

## Date

2026-05-20

## Affected Services

- proto (`proto/alt/knowledge/loop/v1/knowledge_loop.proto`)
- knowledge-sovereign (Go)
- alt-backend (Go)
- alt-frontend-sv (TypeScript / Svelte 5)

## Context

[[knowledge-loop-canonical-contract]] と完成計画 6 本に基づく Knowledge Loop は、現状以下 3 つの **構想層 (思想) の欠陥** を残している (2026-05-20 plan §I-§II 立証):

1. **Loop が閉じていない** — Boyd の OODA は閉ループ (Act → 環境変化 → 新規 Observation) だが、現行 contract §5 は "Act → Observe / Continue" を許可するだけで「Act の結果がどう次の Observe の入力になるか」が定義されていない。state machine ではなく state sequence になっている。
2. **Why は trust contract として不十分** — `WhyPayload.kind/text/evidence_refs` は良いが、「**この why が間違っていたら何を見ればわかるか** (counter_evidence)」「**どれくらい自信があるか** (confidence ladder)」が optional 止まり。epistemic 透明性 (verifiable claim) として未完。
3. **卒業概念が無い** — Act 後にユーザが内部化した (=「もう surface しなくていい」) 知識を Loop から外す仕組みがない。結果として Loop は単調増加 inbox に近づく。

これら 3 つを **additive proto bump** で contract に下ろす。proto は破壊変更 (breaking) ではなく additive のみで、既存 consumer は v8 narrative + outcome event を ignore したまま動作継続可能。

## Decision

3 つの additive bump を 1 PR (proto regen + projector + UI) でまとめて landed する。proto は additive なので consumer の段階的 catch-up が可能。

### Δ4: WhyPayload に counter-evidence と confidence ladder

`WhyPayload` proto message に additive fields:

```proto
message WhyPayload {
  WhyKind kind = 1;
  string text = 2;                                // existing
  optional double confidence = 3;                 // existing
  repeated EvidenceRef evidence_refs = 4;         // existing
  // additive:
  repeated EvidenceRef counter_evidence_refs = 5; // max 4
  optional ConfidenceLadder confidence_ladder = 6;
  optional string what_would_change_my_mind = 7;  // 1..256 chars
}

enum ConfidenceLadder {
  CONFIDENCE_LADDER_UNSPECIFIED = 0;
  CONFIDENCE_LADDER_SPECULATION = 1;
  CONFIDENCE_LADDER_PATTERN = 2;
  CONFIDENCE_LADDER_EVIDENCE = 3;
  CONFIDENCE_LADDER_VERIFIED = 4;
}
```

`WhyMappingVersion` を v8 へ bump し、full reproject 経路で過去 entry を新 narrative に置換。enricher.go は `SurfaceScoreInputs` のうち negative signals (`contradiction_count`, `staleness_score`) を `counter_evidence_refs` に転記する deterministic mapping を持つ。

UI 側 `LoopEntryTile.svelte` は why text の下に "?" disclosure を追加。default 非表示、tap で `counter_evidence_refs` と `what_would_change_my_mind` を Progressive Disclosure で展開。情報過多を回避。

### Δ1: KnowledgeLoopActOutcome event

新 system-only event を additive 追加:

```proto
message KnowledgeLoopActOutcomePayload {
  string acted_event_id = 1;     // 紐づく KnowledgeLoopActed の event_id
  ActOutcomeKind outcome = 2;
  string observed_at = 3;        // RFC3339, event-time
  repeated EvidenceRef signal_evidence_refs = 4;
}

enum ActOutcomeKind {
  ACT_OUTCOME_UNSPECIFIED = 0;
  ACT_OUTCOME_ENGAGED = 1;          // open → dwell ≥ 30s
  ACT_OUTCOME_DEEP_ENGAGEMENT = 2;  // ask → 3 ターン以上
  ACT_OUTCOME_STALE_SAVE = 3;       // save → 7d 未読
  ACT_OUTCOME_ACCEPTED_CHANGE = 4;  // compare → diff 確認 + dismiss
  ACT_OUTCOME_NO_ENGAGEMENT = 5;    // 7d 経過 fallback
}
```

emitter:

- **即時 outcome** — alt-backend `view tracker` (article dwell ≥ 30s / ask conversation turn ≥ 3) が `ACT_OUTCOME_ENGAGED` / `_DEEP_ENGAGEMENT` を emit
- **遅延 fallback** — knowledge-sovereign `act_outcome_cron.go` が 7d 経過後 outcome 未到来の `KnowledgeLoopActed` に対して `ACT_OUTCOME_NO_ENGAGEMENT` を system-emit。cron は `event.occurred_at + 7d` ベースで `time.Now()` を business fact に使わない (event-time purity)

projector enricher が outcome を `SurfaceScoreInputs.act_outcome_signal` に取り込み、`decide_bucket_v2` で：

- `engaged` / `deep_engagement` → 次回類似 entry の Continue / Now boost
- `stale_save` / `no_engagement` → 次回類似 entry の Review 寄り / down-weight

これにより Loop が **学習する** (構想層 §2 loop closure)。

### Δ3: DismissState `internalized`

`DismissState` enum を additive bump:

```proto
enum DismissState {
  DISMISS_STATE_UNSPECIFIED = 0;
  DISMISS_STATE_ACTIVE = 1;       // existing
  DISMISS_STATE_DEFERRED = 2;     // existing
  DISMISS_STATE_DISMISSED = 3;    // existing
  DISMISS_STATE_COMPLETED = 4;    // existing
  DISMISS_STATE_INTERNALIZED = 5; // new
}
```

migration `00019_add_internalized_dismiss_state.sql` で `knowledge_loop_entries.dismiss_state` の CHECK 制約に `internalized` を additive 追加 (column 増減なし)。

trigger:

- **明示 CTA** — `LoopEntryTile.svelte` の Act-stage Save target 横に "I got this" button を追加。click で `dismissState=internalized` の transition を emit
- **MVP では自動推論を入れない** — 「Act = save 後、30d 以内に関連 article を再オープン」「同テーマで `compare` 経由で diff 確認 + dismiss」等は構想として残すが、誤推論で必要 entry が消えると trust を毀損するため Phase F MVP では明示 CTA 経由のみ

read path filter:

- `alt-backend` の `GetKnowledgeLoop` で `dismissState='internalized'` を foreground / bucket entries から除外
- ただし event log には残るので reproject で再現可能、`/loop` byline で "5 internalized this week" として macroState に集計 (ADR-000909 Phase G)

### 検討した代替案

- **counter-evidence を free JSON にする** — canonical contract §11 が自由 JSON 禁止 (`WhyPayload` は構造化必須)。`EvidenceRef` 再利用で型固定する。
- **`internalized` を自動推論で入れる** — MVP では誤検知の cost (信頼喪失) > 機能の cost (面倒さ) と判断。明示 CTA のみ。後続 ADR で `(save + reopen ≤ 30d) → internalized` を opt-in feature flag で導入する余地を残す。
- **Outcome event を user-emittable にする** — system-only に固定。ユーザ意図 (`acted_intent`) と system observation (`outcome`) を mix すると event log の意味論が壊れる。
- **`no_engagement` を cron ではなく request-time に判定** — `request-time` で 7d window を引くと N+1。cron で event-time bound に決定する方が reproject-safe。

## Consequences

### Pros

- Loop が閉じる (構想層 §2)。Act 後の outcome が次回 placement に反映され、system が **学習する**
- Why が verifiable claim になる (構想層 §4)。ユーザは Loop を盲信せず epistemic に検証可能
- 卒業概念が contract に組み込まれ、Loop の単調増加 inbox 化を防ぐ (構想層 §3 macro 層)
- すべて additive proto bump で既存 consumer は段階的 catch-up 可能 (breaking なし)
- `act_outcome_signal` が v2 Surface Planner の入力になり、Continue / Review placement の学習が回り始める
- Phase H (observability) で `act_outcome_emitted_total{outcome}` / `internalized_count` を dashboard 化できる

### Cons / Tradeoffs

- proto 3 件 + migration 1 件 + projector 拡張 + UI 拡張で **影響範囲が広い**。1 PR で landed する複雑性 vs 分割で生む混乱の trade-off で、proto は破壊しない前提で 1 PR を選択 (consumer は ignore で動く)。
- WhyMappingVersion v8 bump は **full reproject** 必須 (ADR-000907 と整合)。
- `act_outcome_cron` の 7d window batch が大量 event を一気に追加し event log の sequence 進行が一時的に boost する。`event_dropped_total` 監視で異常検知。
- "I got this" CTA は Newspaper Style の文言ガイドに沿うこと (`I got this →` で en-dash arrow)。
- counter-evidence の `what_would_change_my_mind` text 生成は deterministic template が必要だが、現状 evidence の意味を 1 文に圧縮する template が未定。Phase F 内で template ladder を定義 (`stale → "If you re-read and the take has changed."` 等)。
