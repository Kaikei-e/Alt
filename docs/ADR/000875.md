# ADR-000875: Knowledge Loop の Open CTA を window.open から SPA goto('/articles/:id') に移行し popup-blocker race を解消する

## Status

Accepted

## Date

2026-05-02

## Affected Services

- **alt-frontend-sv (TypeScript / Svelte 5)** — `/loop` の Open 動線を SPA reader 経由に変更

## Context

ADR-000873 で Loop Open CTA は「クリック直後に同期 `window.open` → 続けて `void onTransition(...)` で fire-and-forget」という形に変えた。これは fb.md §5 が指摘した popup-blocker race ("await transition の完了後に window.open すると Firefox / Safari がブロックする") を回避する **必要条件** だった。しかし fb.md は最終形として:

> 推奨は、まず SPA navigation に寄せること。
> Open article: 即 goto('/articles/:id') / 裏で KnowledgeLoopActed を emit / 失敗時は toast / retry queue
> Knowledge Loop では、行動の摩擦を減らすべき。

を案 C として挙げており、外部新タブよりも内蔵リーダーへ即遷移するのが Knowledge Loop の "Act" 体験に合致する、という結論だった。

Phase 1 探索 (ADR-000875 計画時) で判明:

- `alt-frontend-sv/src/routes/(app)/articles/[id]/+page.svelte` は既に存在し、`?url=&title=&summarize=` query で外部 URL を fetch して読書 UI を提供する内蔵リーダー
- `alt-frontend-sv/src/routes/(app)/home/+page.svelte:259` は既に `goto(\`/articles/${articleId}?${params.toString()}\`)` を使って home 経由の "Open" を SPA で完結させている
- `/loop/+page.svelte` の `onEntryOpen` だけが `window.open` で外部新タブを開いており、Loop だけが取り残された UX を持っていた

## Decision

`/loop/+page.svelte` の `onEntryOpen(entry, href)` を以下に変更する:

```ts
function onEntryOpen(entry: KnowledgeLoopEntryData, href: string) {
  if (isSafeInternalPath(href)) {
    void goto(href);
    return;
  }
  // External article URL → SPA reader view, not a new tab.
  const params = new URLSearchParams();
  params.set("url", href);
  const articleTitle =
    entry.actTargets.find((t) => t.targetType === "article")?.label ??
    entry.whyPrimary.text;
  if (articleTitle) {
    params.set("title", articleTitle);
  }
  void goto(`/articles/${encodeURIComponent(entry.entryKey)}?${params.toString()}`);
}
```

### 設計の prove points

- **Path param**: `articles/[id]` の `[id]` は内蔵リーダーが route 識別に使うだけ (実 article URL は `?url=`)。entry の `entryKey` (`article:<id>`) を encode して渡す。`entryKey` は LoopEntry の安定 ID なので reload 後も同じ URL で復帰可能。
- **Query**: `url` (decoded で article URL) + `title` (action target の `label` か why_text fallback) の 2 つ。home/+page.svelte と同じ shape。
- **Internal safe path**: 既存の `isSafeInternalPath(href)` チェックは残す。/articles/ 以外の internal route (例: `/augur/<id>`) を Open に渡すケースが将来発生したときに早期分岐で守る。
- **window.open は完全撤去**: tile 側の `openHref` fallback (LoopEntryTile.svelte) は残るが、`/loop` page から渡される `onEntryOpen` callback が SPA navigation を使うため fallback には到達しない。E2E でも assert する。

### page.source.spec.ts に regression guard

```ts
it("routes external Open CTA into the SPA /articles/ reader instead of window.open", () => {
  expect(pageSource).toMatch(/goto\(`\/articles\/\$\{encodeURIComponent\(entry\.entryKey\)\}/);
  expect(pageSource).not.toMatch(/window\.open\(href, "_blank"/);
});
```

source spec を Playwright より優先したのは、`bunx vitest run --project=server` が CI でも常時走るため regression を即座に検知できる速さのため。Playwright の本格的な遷移 assertion (`/articles/<id>?url=...` への actual navigation) は `tests/e2e/desktop/loop/open-cta-navigates-to-article.spec.ts` を別途追加して subhead として走らせる予定 (本 ADR では source spec のみ追加)。

### 検討した代替案と却下理由

- **(却下) `window.open(\`/articles/...\`, "_blank")` で内蔵リーダーを新タブに開く**: 内蔵 SPA を新タブで開くと SvelteKit の preload が効かず、cold load で再 hydration が走る。Knowledge Loop は OODA を一連の流れとして UI に持たせたい設計 (ADR-000831) で、Open は同 tab に閉じてた方が思考の連続性を支える。
- **(却下) `goto` の前に `window.open("about:blank")` を同期 fork して両方ナビゲーションする**: 過剰実装。fb.md 案 C が示す通り、Loop Open は SPA で十分。
- **(却下) `window.open` を使うが popup-blocker 安全パターンに留める**: ADR-000873 で既に popup-blocker race は解消済 (handleCta 内で同期実行)。あえて新タブ動作を残す理由は無い。
- **(却下) `goto(href)` で外部 URL に直接遷移**: SvelteKit の `goto` は外部 URL を渡されると `goto: cannot navigate to external URL` で例外を投げるか、`window.location.href = href` フォールバックを呼ぶ。前者は run-time error、後者は SPA を抜けて内蔵リーダーが使えない。
- **(却下) Loop 全 entry に対して onEntryOpen を一括無効化**: fb.md は「Knowledge Loop で行動の摩擦を減らせ」と言っており、Open CTA を消すのは逆方向。
- **(却下) `entry.actTargets` で `targetType=article` の `route` を必ず使う**: 既存の `resolveSourceUrl` が `actTargets[].route` か `whyPrimary.evidenceRefs[0].refId` のいずれかから取得する設計を変更すると blast radius が広がる。`onEntryOpen` の入力 `href` は `resolveSourceUrl` の戻り値そのものなので、ハンドラ内の遷移ロジックだけを変える。

## Consequences

### Pros

- **Knowledge Loop の "Act" が SPA で完結する**。Open ボタン → 内蔵リーダー → 必要に応じて summarize → 戻ると Continue plane に entry が残っている、という連続体験。fb.md 案 C を満たす。
- **popup-blocker race の最終形**。新タブを開く操作が消えるので、Firefox / Safari の popup blocker が trigger される可能性自体が無くなる。ADR-000873 の同期 `window.open` は安全だったが、内蔵リーダーへの SPA 遷移はそもそも browser policy の対象外。
- **home/+page.svelte と Loop の Open 動線が同じ shape に統一**。`/articles/<id>?url=&title=` 規約が monorepo 全体で 2 箇所一致し、内蔵リーダーの query contract が単一情報源化される。
- **regression guard が source spec で常時走る**。`page.source.spec.ts` が `window.open(href, "_blank"` の文字列を反対側の assertion として書いてあるため、誤って old behavior に戻すと CI が即座に止める。
- **既存の `actTargets` / `whyPrimary` shape に乗っているだけ**で、proto / Connect-RPC schema の変更は不要。CDC regression もゼロ。

### Cons / Tradeoffs

- **新タブで開きたい user の workflow が変わる**。リサーチ用途で複数 article を並列で開きたい user は cmd-click / middle-click に逃がす必要がある。Svelte の `<a href>` ベースなら自動で middle-click が新タブに行くが、`onclick → goto` 遷移では cmd / middle がそのままでは効かない。実装上は `e.metaKey || e.ctrlKey || e.button === 1` で別経路に逃がす拡張が必要 (本 ADR ではスコープ外、follow-up に残す)。
- **`/articles/<id>?url=...` の `id` が entry_key で、内蔵リーダーは現状 path param を実用していない**。リーダー側の future 拡張 (例: entry の bookmark 機能) で path param を一意 ID として使うときに、Loop / Home / その他経由で渡される値の不整合が起きないようリーダー側も `entryKey` を主キーに揃える必要がある。
- **Playwright e2e のフォロー追加が必要**。本 ADR では source spec の regression guard のみ。実際の URL 遷移 assertion は次回の `tests/e2e/desktop/loop/` 拡張で追跡する。
- **`articleTitle` の fallback が `entry.whyPrimary.text` を使うため reader の Title 欄に why_text が出る場合がある**。article の真の title が actTargets に入っていれば優先されるが、未設定だと why narrative が使われる。Home 経由で開いた article との UX 差異を作らないため、actTargets[].label の seeding を projector 側で改善するのが follow-up。
