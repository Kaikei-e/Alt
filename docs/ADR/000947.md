# ADR-000947: Gemma4 GGUF を stealth update 後の Q4_0 に強制再取得し Ollama を 0.32.1 に上げる

## Status

Accepted

## Date

2026-07-18

## Affected Services

- **news-creator-backend (Ollama)** — ローカル GPU 上の Gemma4 推論 runtime。image tag・entrypoint・Modelfile の GGUF パスを更新。
- **compose/ai.yaml** — force refresh / digest pin の env 配線。消費者サービス（news-creator / pre-processor / rag-orchestrator / recap / acolyte）のモデル名は据え置き。

## Context

2026-07-15 に Google が Gemma 4 ファミリーへ **バージョン番号なしの stealth weight update** を出した。公開された改善は Flash Attention 4（特定データセンター向け GPU 世代での prefill +25–70% / TTFT 最大 -31%）、tool-calling 修正、途中切れ削減など。同名のまま HuggingFace 上の重みが差し替わるため、ローカルにキャッシュした GGUF を取り直さない限り品質修正は入らない。

Alt の primary fleet は `news_creator_models` named volume に `gemma4-e4b-q4km` / `8k` / `12k` を永続化しており、既存の `entrypoint-backend.sh` は `ollama list` にタグがあると **HuggingFace 再 DL を skip** する。再ビルドや再起動だけでは最新化できない。

実装前の HEAD 確認で、従来 URL の `gemma-4-e4b-it-Q4_K_M.gguf` は **404** になった。`ggml-org/gemma-4-E4B-it-GGUF` は 2026-07-16 に再変換され、4-bit は `gemma-4-E4B-it-Q4_0.gguf`（約 4.59GB）のみ。bartowski の Q4_K_M は lastModified 2026-05-03 のままで stealth update を含まない。

ハードウェア制約: ローカル推論ホストの VRAM 上限（約 8GB クラス）では、上流が謳う FA4 高速化の前提ハードウェアを満たさない。取り込む価値は重み修正と Ollama 0.32.x の Gemma4 tool-call 改善側。

廃止済みの remote inference fleet は本 ADR の対象外。knowledge-augur / knowledge-embedder の Ollama `0.22.1` pin も今回は触らない。

## Decision

1. **`news-creator/Dockerfile.backend` を `FROM ollama/ollama:0.32.1` に上げる**（`:latest` は使わない。[[000887]] の pin 方針を維持）。
2. **GGUF ソースを ggml-org の post-refresh `gemma-4-E4B-it-Q4_0.gguf` に切り替える**。Ollama タグ名 `gemma4-e4b-q4km` / `gemma4-e4b-8k` / `gemma4-e4b-12k` は消費者互換のため据え置き（[[PM-2026-016]] のモデル名不一致 EOF を避ける）。
3. **`FORCE_MODEL_REFRESH=1` の one-shot で既存タグを `ollama rm` → 再 DL → 再 create**する。default は `0`。破壊的操作のため loud WARNING を出し、`ollama rm` 失敗は fail-fast（`|| true` 禁止）。
4. **`GGUF_SHA256_EXPECTED` で digest を pin**する。実測 digest `a555b900214b477d8880e7832e0b8925e139b0159640036b09fe472b6f2097f2` を compose default に置き、不一致なら GGUF を破棄して exit 1。sha256sum 欠如も fail-fast。
5. **refresh 成功後は必ず `FORCE_MODEL_REFRESH=0` に戻す**。`1` のまま放置すると再起動のたびにモデル削除 + 数 GB 再 DL が走る。

### 検討した代替案

- **bartowski の Q4_K_M を継続**: 量子化名は維持できるが、2026-05-03 以降未更新で stealth update を取り込めない。却下。
- **`ollama pull gemma4:e4b`**: 約 9.6GB（マルチモーダル projector 込み）でローカル VRAM 上限に収まらない。[[000613]] と同じ理由で却下。
- **FORCE を常時 ON / 起動ごとに digest 比較**: 毎回 4.6GB DL は運用不能。volume 上の Ollama blob と HF digest の差分検知は将来課題とし、今回は明示 one-shot + digest pin に留める。

## Consequences

### Pros

- 2026-07-15 stealth update 後の公式 ggml-org 重みをローカルに反映できた（SHA256 で監査可能）。
- Ollama 0.32.1 の Gemma4 関連改善を取り込みつつ、floating `:latest` を避けた。
- Q4_0（4.59GB）は旧 Q4_K_M（5.34GB）より軽く、ローカル VRAM に余裕が増える。
- FORCE の wipe-on-restart と不正 GGUF を fail-fast で防げる。

### Cons / Tradeoffs

- タグ名 `gemma4-e4b-q4km` と実体 quant（Q4_0）が一致しない。運用ドキュメントと ADR で明示する必要がある。
- 次回 ggml-org が Q4_0 を再アップロードすると、`GGUF_SHA256_EXPECTED` を更新するまで FORCE refresh は起動失敗する（意図的な安全側）。
- FA4 の tok/s 改善は現行ローカル GPU では得られない。期待値管理が必要。
- FORCE=1 中は healthcheck `start_period`（120s）を超え一時 unhealthy になり得る。
