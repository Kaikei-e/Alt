# ADR-000411: OPMLエクスポートのURL正規化・重複排除・HTMLエンティティ修正とフィード登録時のトラッキングパラメータ除去を行う

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — utils, gateway, usecase レイヤー

## Context

[[000393]] で OPML 2.0 インポート/エクスポート機能を導入した後、ユーザーからの実データに基づくフィードバックが 3 点あった:

1. **重複 URL** — 同一フィードが UTM パラメータ違い（`?utm_source=rss` vs `?utm_source=chatgpt.com`）で複数登録されており、OPML エクスポートに重複が出る
2. **トラッキングパラメータ付き URL** — `?utm_source=chatgpt.com` 等のパラメータがエクスポートされた OPML に残り、クリーンでない
3. **HTML エンティティの混在** — フィードタイトルに含まれる `&#39;`（アポストロフィ）や `&amp;`（アンパサンド）が XML 内で二重エスケープされ `&amp;#39;` として出力される

さらに、根本原因として**フィード登録時点でトラッキングパラメータを除去していない**ことが判明した。既存の `NormalizeURL` はすべてのクエリパラメータを除去する記事 URL 用の関数であり、フィードソース URL には正当なパラメータ（`?format=atom`, `?page=2` 等）を保持しつつトラッキングパラメータのみを除去する選択的サニタイズが必要だった。

## Decision

**フィードソース URL 用の選択的トラッキングパラメータ除去関数 `StripTrackingParams` を新設し、フィード登録（個別・一括）とエクスポートの 3 箇所に適用する。** エクスポート時にはさらに重複排除と HTML エンティティデコードを行う。

### 検討した代替案

1. **既存の `NormalizeURL` を拡張する案**: 全パラメータ除去の挙動を選択的除去に変更すると、記事 URL の正規化に影響する。記事 URL とフィードソース URL では要件が異なるため却下。
2. **DB マイグレーションで一括修正する案**: 既存の `scripts/normalize_feed_urls.go` と同等の処理をマイグレーションとして実行する案。一時的な修正にはなるが、登録時に除去しなければ問題は再発するため不十分。
3. **登録時・インポート時・エクスポート時の 3 層で適用する案（採用）**: 入口（登録）でサニタイズし、インポートでバッチ内重複を排除し、エクスポートで既存データの重複も吸収する。既存データの一括修正は `scripts/normalize_feed_urls.go` で別途実施可能。

### 実装の構成

#### 1. `StripTrackingParams` ユーティリティ（utils 層）

`scripts/normalize_feed_urls.go` のトラッキングパラメータリストを再利用し、`utils/url_sanitizer.go` に配置:

- 除去対象: `utm_source`, `utm_medium`, `utm_campaign`, `utm_term`, `utm_content`, `utm_id`, `fbclid`, `gclid`, `mc_eid`, `msclkid`
- 大文字小文字不問（`UTM_SOURCE` も除去）
- 正当なパラメータは保持
- フラグメント除去、末尾スラッシュ除去
- 残ったパラメータはキーでソート（決定的な出力）
- 冪等性保証

既存の `NormalizeURL`（全パラメータ除去、記事 URL 用）とは明確に用途が分かれている。

#### 2. フィード個別登録（Gateway 層）

`RegisterFeedGateway.RegisterFeedLink()` で DB 挿入前に `StripTrackingParams` を適用。エラー時は原本 URL にフォールバック。

#### 3. OPML インポート（Gateway 層）

`ImportGateway.RegisterFeedLinkBulk()` で SSRF 検証後に `StripTrackingParams` を適用し、`seen` マップによるバッチ内重複排除を追加。UTM 違いの同一 URL が同じ OPML ファイルに含まれていてもスキップされる。

#### 4. OPML エクスポート（Usecase 層）

`ExportOPMLUsecase.Execute()` で `deduplicateLinks()` を呼び出し:
- 各 URL に `StripTrackingParams` を適用
- `seen` マップで重複排除（最初のタイトルが優先）
- `html.UnescapeString()` でタイトルの HTML エンティティをデコード

`html.UnescapeString` は `&#39;` → `'`、`&amp;` → `&` に変換し、Go の `encoding/xml` が XML 出力時に適切に再エスケープする。既存の `sanitize.SanitizeDescription` は HTML タグ除去も行うためタイトルには過剰であり、`html.UnescapeString` 単体で十分。

### テスト

| パッケージ | テスト数 | 内容 |
|-----------|---------|------|
| `utils` | 18 + 1 | `StripTrackingParams` の全パターン + 冪等性 |
| `gateway/register_feed_gateway` | 3 | UTM 付き URL の個別登録（3 パターン） |
| `gateway/opml_gateway` | 3 | nil DB、空 URL、バッチ重複 |
| `usecase/opml_usecase` | 2 | UTM 重複排除、HTML エンティティデコード |

## Consequences

### Pros

- **入口でのサニタイズ**: 新規登録されるフィードにはトラッキングパラメータが含まれなくなり、DB の一貫性が向上
- **エクスポート品質の改善**: UTM 重複のない、HTML エンティティが正しくデコードされた OPML が出力される
- **既存の `NormalizeURL` への影響ゼロ**: 記事 URL 用の全パラメータ除去ロジックは変更していない
- **冪等かつ安全**: `StripTrackingParams` はエラー時にフォールバックし、2 回適用しても結果が変わらない

### Cons / Tradeoffs

- **既存データの修正は別途必要**: 今回の変更は新規登録・エクスポート時の処理であり、既に DB に入っている UTM 付き URL は `scripts/normalize_feed_urls.go` の実行が必要
- **トラッキングパラメータリストのメンテナンス**: 新しいトラッキングパラメータ（例: 将来のプラットフォーム固有パラメータ）が登場した場合、`trackingParams` マップの更新が必要
