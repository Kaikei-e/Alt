# ADR-000718: alt-backend の High 深刻度 4 件を SSE 廃止・tenant_id クレーム化・ロック単純化・BodyLimit で解消する

## Status

Accepted

## Date

2026-04-14

## Affected Services

- alt-backend (Go) — `middleware/dos_protection_middleware.go` の Circuit Breaker ロック方式変更、`middleware/jwt_middleware.go` への `BackendClaims.TenantID` / `UserContext.TenantID` 追加、`middleware/auth_middleware.go` の `parseTenantUUID` 抽出と `tenantID = userID` ハードコード除去、`rest/routes.go` への `BodyLimitWithConfig("2M")` 追加と `WhitelistedPaths` 整理、`rest/sse_handlers.go` 削除、`rest/schema.go` から `UnsummarizedFeedStatsSummary` 削除
- auth-hub (Go) — `internal/domain/session.go` の `Identity` に `TenantID` 追加、`internal/infrastructure/token/jwt.go` の `backendClaims` に `tenant_id` claim を追加して `IssueBackendToken` で発行、`internal/usecase/session/get_session.go` および `validate_session.go` で `Identity.TenantID` を明示設定
- alt-frontend-sv (TypeScript / Svelte 5) — `src/lib/hooks/useFeedStats.svelte.ts` を `useStreamingFeedStats` に一本化、`src/lib/hooks/useSSEFeedsStats.svelte.ts` と `src/lib/api/sse.ts` を削除、`useFeedStats.test.ts` を Connect-RPC pass-through に書き換え

## Context

[[000717]] では同日のセキュリティレビュー (`docs/review/alt-backend-security-review-2026-04-14.md`) の Critical 3 件と H-002 を解消した。続く High 4 件 — H-001 / H-003 / H-004 / H-005 — は次の事情でいずれも長期放置されていた。

1. **H-001 SSE 認証なし**: `/v1/sse/feeds/stats` は `EventSource` で叩かれており、ブラウザ側でカスタムヘッダを付けられないため JWT 認証を素直に追加する経路がなかった。一方、alt-backend には Connect-RPC server-streaming RPC `StreamFeedStats` (`alt-backend/app/connect/v2/feeds/feed_stats.go:98-155`) と proto (`proto/alt/feeds/v2/feeds.proto:334`) が既に整備されており、フロントエンドにも `streamFeedStats()` (`alt-frontend-sv/src/lib/connect/feeds/streaming.ts`) と `useStreamingFeedStats` フックが実装済。実際に UI が購読しているのは旧 EventSource 経路のままで、Connect-RPC への切替が `PUBLIC_USE_CONNECT_STREAMING` フィーチャーフラグの「false」固定で塞がれていた。

2. **H-003 `tenantID = userID` のハードコード**: `middleware/auth_middleware.go:62, 95` で `tenantID, _ := uuid.Parse(jwtUserCtx.ID)` と書かれていた。auth-hub 側は `domain.Identity` に `TenantID` を持たず、JWT にも `tenant_id` claim を載せていなかった。結果として alt-backend 内の `domain.UserContext.TenantID` は実質常に UserID と同一であり、`TenantAwareRepository` の `set_current_tenant($1)` (`tenant_repository.go:36-45, 57`) の RLS 風分離が「user 単位」に縮退していた。マルチテナント化のたびにアプリ側に潜在的な user 越境バグを埋め込み続けるリスクがあり、ADR-000717 でも将来課題として明記済だった。

3. **H-004 Circuit Breaker の lock-upgrade race**: `dos_protection_middleware.go:261-282` の `shouldBlock()` は RWMutex を `RLock → RUnlock → Lock → Unlock → RLock` と昇格する典型的なアンチパターンで、TOCTOU により別 goroutine の状態変更を上書きする可能性があった。Go の race detector は data race のみを検出するためテストでは見つからないが、状態機械の線形化が破れると DoS 保護自体が機能不全に陥る潜在バグだった。

4. **H-005 BodyLimit 未設定**: `rest/routes.go` に `middleware.BodyLimit` がなく、`io.ReadAll(c.Request().Body)` で全量を読み込むエンドポイントが多数あった。OPML import (`rest_feeds/opml.go:43`) は handler 側で 1MB 制限を実装していたが、その他の POST / PUT 経路はノーガードで、攻撃者が大きなボディを送りつけるとメモリを食い潰せる状態だった。pgx CVE-2024-27304 (4GB 超の SQL 経路) の前提条件「巨大なリクエストを通す」を成立させる素地でもあった。

ユーザー判断により、H-001 は「すでに Connect-RPC 移行済なら EventSource 経路を削除する」、H-003 は「auth-hub 本体まで改修して完全移行する」と決めた。あわせて Web 調査 (OWASP Microservices Security Cheat Sheet 2025、SPIFFE/SPIRE、Istio Ambient) で、shared-secret + 一意 service identity への将来移行余地を残しつつ、現状の application-layer での JWT claim 拡張が現実解であることを再確認した。

## Decision

以下の四点をまとめて適用する。

1. **H-001: `/v1/sse/feeds/stats` を削除し Connect-RPC `StreamFeedStats` へ一本化する** — alt-backend 側でハンドラ・ルート登録・関連 DTO (`UnsummarizedFeedStatsSummary`) を削除し、`middleware/dos_protection_middleware.go` 経由の SSE バイパスを縮める (`/v1/sse/` を `WhitelistedPaths` から外す)。alt-frontend-sv 側は `useFeedStats` から SSE 経路を除去し `useStreamingFeedStats` (Connect-RPC) に固定。Connect-RPC 経路はすでに `AuthInterceptor` を通っているため、`X-Alt-Backend-Token` JWT 認証が透明に効く。

2. **H-003: auth-hub が `tenant_id` claim を発行し、alt-backend が `BackendClaims.TenantID` から取得する** — `domain.Identity` に `TenantID string` を追加、`backendClaims` に `tenant_id` を追加して `IssueBackendToken` が `identity.TenantID` を載せる (空なら UserID にフォールバック)。alt-backend は `BackendClaims` / `UserContext` に `TenantID` フィールドを増やし、`parseTenantUUID(claim string) (uuid.UUID, error)` で UUID パースして `domain.UserContext.TenantID` に設定する。`RequireAuth` は不正な claim を 401 に落とし、`OptionalAuth` は V-005 哲学を守って anonymous 継続。Kratos identity schema は変更せず、現状の single-tenant 運用 (TenantID == UserID) の意味論を維持する。

3. **H-004: Circuit Breaker のロックを `sync.Mutex` に単純化する** — RWMutex の lock upgrade パターンを廃止し、`shouldBlock()` の判定と Open → Half-Open 遷移を単一クリティカルセクションで行う。read パスも極短いので RWMutex の利点よりも race 回避を優先した。race-free を確認する並列ストレステスト (`TestCircuitBreaker_ConcurrentStateIsRaceFree`) と単一ステップ遷移テストを追加。

4. **H-005: `middleware.BodyLimitWithConfig("2M")` をグローバルに付与する** — Recover 直後に挿入し、`/sse/` と `/stream` を含むパスは Skipper で除外。2MB は OPML import の per-file 1MB + multipart オーバーヘッドを許容する値。OPML handler 側の 1MB 制限は defense-in-depth として残す。

### 検討した代替案

- **H-001 代替**: SvelteKit に server-side SSE proxy を立てて `X-Alt-Backend-Token` を注入する案も検討した。しかし alt-backend 側 / alt-frontend-sv 側ともに Connect-RPC streaming が完成していたため、proxy 層を増やすより既存実装を採用する方が薄く、認証チェーンも短い。SSE への JWT 注入は将来 Connect-RPC が使えない経路 (e.g. 標準ブラウザの低レベル監視) で再評価する。
- **H-003 代替**: 「alt-backend だけ前方互換対応 (claim あれば使う / なければ userID)」案も検討した。実装は最小だが、auth-hub 側に同じ意味論を分散させたままになり、複数年残り続けるリスクがあった。auth-hub 側まで踏み込み、claim を発行することで「将来 multi-tenant 化は Identity の TenantID を populate する一点だけ」になる構造を選んだ。
- **H-004 代替**: atomic.CompareAndSwap で state 遷移する案。実装は早くなるが、`failures` / `lastFailureTime` を別ロックで守る必要がありコードが分散する。`sync.Mutex` 1 本でロックする方が読みやすく保守しやすい。
- **H-005 代替**: per-route の BodyLimit 設定。柔軟だが、忘れる route が出た時点でリスクが復活する。global で deny by default にして例外を Skipper で明示する方が安全側に倒れる。

## Implementation Overview

### 1. Circuit Breaker race の解消 (H-004)

```go
// middleware/dos_protection_middleware.go
type circuitBreaker struct {
    config          CircuitBreakerConfig
    failures        int
    lastFailureTime time.Time
    state           circuitState
    mu              sync.Mutex // was sync.RWMutex
}

func (cb *circuitBreaker) shouldBlock() bool {
    cb.mu.Lock()
    defer cb.mu.Unlock()
    switch cb.state {
    case circuitClosed:
        return false
    case circuitOpen:
        if time.Since(cb.lastFailureTime) > cb.config.RecoveryTimeout {
            cb.state = circuitHalfOpen
            return false
        }
        return true
    case circuitHalfOpen:
        return false
    default:
        return false
    }
}
```

`-race` 付きの並列 stress test (`TestCircuitBreaker_ConcurrentStateIsRaceFree`) と Open → Half-Open 単一ステップ遷移テストを追加。

### 2. グローバル BodyLimit (H-005)

```go
// rest/routes.go (Recover の直後)
e.Use(middleware.BodyLimitWithConfig(middleware.BodyLimitConfig{
    Skipper: func(c echo.Context) bool {
        return strings.Contains(c.Path(), "/sse/") || strings.Contains(c.Path(), "/stream")
    },
    Limit: "2M",
}))
```

`rest/body_limit_test.go` で 3MB POST が 413、512KB POST が 200、`/v1/feeds/summarize/stream` が Skipper で除外されることを検証。OPML handler の per-file 1MB 制限は維持。

### 3. auth-hub: tenant_id claim 発行 (H-003 前半)

```go
// internal/domain/session.go
type Identity struct {
    UserID    string
    TenantID  string  // 追加
    Email     string
    Role      string
    SessionID string
    CreatedAt time.Time
}

// internal/infrastructure/token/jwt.go
type backendClaims struct {
    Email    string `json:"email"`
    Role     string `json:"role"`
    Sid      string `json:"sid"`
    TenantID string `json:"tenant_id"`  // 追加
    jwt.RegisteredClaims
}

func (j *JWTIssuer) IssueBackendToken(identity *domain.Identity, sessionID string) (string, error) {
    // ...
    tenantID := identity.TenantID
    if tenantID == "" {
        tenantID = identity.UserID  // single-tenant fallback
    }
    claims := backendClaims{
        // ...
        TenantID: tenantID,
        // ...
    }
    // ...
}
```

`get_session.go` と `validate_session.go` の usecase は cache miss / hit いずれの経路でも `Identity.TenantID = identity.UserID` を明示設定するように整理した。テストには `TestJWTIssuer_IncludesTenantIDClaim` と `TestJWTIssuer_TenantIDDefaultsToUserID` を追加。

### 4. alt-backend: tenant_id claim 利用 (H-003 後半)

```go
// middleware/jwt_middleware.go
type BackendClaims struct {
    Email    string `json:"email"`
    Role     string `json:"role"`
    Sid      string `json:"sid"`
    TenantID string `json:"tenant_id"`  // 追加
    jwt.RegisteredClaims
}

// middleware/auth_middleware.go
func parseTenantUUID(tenant string) (uuid.UUID, error) {
    if tenant == "" {
        return uuid.Nil, errors.New("tenant_id claim is missing")
    }
    id, err := uuid.Parse(tenant)
    if err != nil {
        return uuid.Nil, fmt.Errorf("tenant_id is not a valid UUID: %w", err)
    }
    if id == uuid.Nil {
        return uuid.Nil, errors.New("tenant_id is the nil UUID")
    }
    return id, nil
}

// RequireAuth 内
tenantID, err := parseTenantUUID(jwtUserCtx.TenantID)
if err != nil {
    return echo.NewHTTPError(http.StatusUnauthorized, "invalid tenant_id claim")
}
```

`OptionalAuth` は不正 tenant_id を anonymous として扱う (V-005 整合)。テストには tenant_id 採用・欠落時 401・非 UUID 時 401・Nil UUID 時 401・OptionalAuth 匿名扱いの 5 ケースを追加。`auth_middleware_repro_test.go` の JWT MapClaims にも `tenant_id` を加えて既存テストを GREEN に保った。

### 5. SSE 経路の削除と Connect-RPC 一本化 (H-001)

**alt-backend 側**:
- `rest/sse_handlers.go` を完全削除
- `rest/routes.go` の `registerSSERoutes(...)` 呼び出しと `WhitelistedPaths` の `/v1/sse/` を削除
- `rest/schema.go` から `UnsummarizedFeedStatsSummary` を削除
- `rest/sse_routes_removed_test.go` で `/v1/sse/feeds/stats` が 404 になることを検証

**alt-frontend-sv 側**:
- `src/lib/hooks/useFeedStats.svelte.ts` を `useStreamingFeedStats` への薄いラッパに置換 (フィーチャーフラグ廃止)
- `src/lib/hooks/useSSEFeedsStats.svelte.ts` と `src/lib/api/sse.ts` を物理削除
- `src/lib/hooks/useFeedStats.test.ts` を pass-through テストに書き換え

### 6. ランタイム検証 (smoke)

`docker compose -f compose/compose.yaml -p alt up --build -d auth-hub alt-backend alt-frontend-sv` で 3 サービスを再ビルド・起動し、いずれも `healthy` で安定稼働を確認。以下の HTTP smoke test を実行:

- `GET /v1/health` → 200
- `GET /v1/sse/feeds/stats` → **404** (修正前は 200)
- `POST /v1/feeds/search` (3MB body) → **413** (修正前は無制限)
- `GET /v1/internal/system-user` (X-Service-Token なし) → 401 (ADR-000717 で適用済)
- `GET /v1/dashboard/metrics` (X-Alt-Backend-Token なし) → 401 (ADR-000717 で適用済)

`go test ./... -count=1` (alt-backend / auth-hub) すべて GREEN。`-race` 付きでは pre-existing なテストフレーク (テストハーネスの並列実行依存) が一部出るが、対象モジュール (`alt/middleware`, `alt/rest`) は単独 `-race` 実行で全 GREEN を確認。

## Consequences

### Pros

- **High 4 件が同日中にクローズ**: レビュー報告の H-001 / H-003 / H-004 / H-005 が解消され、SSE 経路の認証欠落・テナント分離崩壊・Circuit Breaker race・無制限 body の 4 つの攻撃面が同時に閉じた。
- **multi-tenant 化への前方互換性確立**: JWT に `tenant_id` claim が乗ったことで、将来テナント本実装時は auth-hub の `Identity.TenantID` populate 一点だけ変えれば alt-backend 全体の挙動が透明に切り替わる。`tenantID = userID` のハードコードコメントが消え、設計意図がコードレベルで明示された。
- **Streaming 経路の認証一元化**: SSE と Connect-RPC streaming が並走していた状態が解消され、リアルタイム購読は AuthInterceptor 配下の Connect-RPC のみとなった。フィーチャーフラグ (`PUBLIC_USE_CONNECT_STREAMING`) も役目を終えて削除し、コードベースの分岐が減った。
- **Circuit Breaker の正しさを担保**: lock upgrade race のアンチパターンが消え、状態機械が線形化された。`-race` ストレステストが回帰防止の役目を果たす。
- **DoS 攻撃面の縮小**: グローバル BodyLimit により、攻撃者が 100MB 超の JSON を送りつけて worker を占有するシナリオが封じられた。pgx CVE-2024-27304 の前提条件 (巨大リクエスト) が成立しなくなり、二重防御になる。
- **TDD 厳守**: 各修正で RED → GREEN → REFACTOR のサイクルを守り、新規追加した unit test とランタイム smoke の両面で挙動を担保した。

### Cons / Tradeoffs

- **3 サービス間で `tenant_id` 運用ルールの同期が必要**: auth-hub が `tenant_id` を発行しなくなる (バグ等で) と alt-backend は 401 を返す。デプロイ順序として「auth-hub → alt-backend」を守る必要がある。今回は同コミットでロールアウトしたため問題なし。
- **古い JWT (`tenant_id` 未含)** **は `RequireAuth` で拒否される**: auth-hub 再デプロイ前に発行された有効期間内 JWT を持つクライアントは、再ログインまで 401 を受け取る。Backend token の TTL が短いため影響は数分程度の見込み。
- **SSE エンドポイント削除に伴う一時的な互換性破壊**: SvelteKit 側がまだ古いバンドルをキャッシュしているブラウザは、初回ロード時に EventSource が 404 を受けるが、`useFeedStats` の Connect-RPC 経路に切り替わるため自動回復する。長期影響なし。
- **EventSource → Connect-RPC で再接続戦略が変わる**: `useStreamingFeedStats` の retry は exponential backoff (1s 〜 10s, 最大 10 回) で、旧 SSE のヘルスチェック (25s timeout) と挙動が違う。UX 上の差は軽微だが、今後監視で見ていく。
- **Medium / Low 17 件は依然未対応**: SSRF Validator の二重実装整理 (M-002〜M-004)、CSP ハード化 (M-008)、CORS整合 (M-009)、X-Forwarded-For 信頼境界 (M-007)、rate limiter cleanup (M-006)、CSRF/DoS の path 判定 (M-005) などは別 ADR で順次対応する。

### 将来の検討課題

- マルチテナント化への第二段: Kratos identity schema に `tenant_id` を追加し、auth-hub の Kratos アダプタが identity から取得した tenant をそのまま `Identity.TenantID` に流す。本 ADR の「single-tenant fallback」分岐が dead code になった時点で削除する。
- caller 識別 (ADR-000717 の将来課題) と並行して、Service Token を per-caller の signed JWT (iss / sub) に発展させる。Linkerd mTLS の SPIFFE ID 連携も視野に入れる。
- `/v1/feeds/summarize/stream` ほか残る REST streaming 経路の Connect-RPC 化 (本 ADR は feed stats のみが対象)。
- BodyLimit 値の config 化と Per-route override の設計 (`MaxRequestBodySize` 環境変数化)。
- `middleware.TimeoutWithConfig` deprecation 対応 (Echo が `ContextTimeoutWithConfig` への移行を推奨)。
- Medium / Low 17 件の段階的対応 (レビュー報告 `docs/review/alt-backend-security-review-2026-04-14.md` の優先修正リスト参照)。
