# ADR-000920: ReviewReason を projector に持たせ epistemic-change driver を first-class 化する (WhyMappingVersion v12)

## Status

Accepted

## Date

2026-05-25

## Affected Services

- knowledge-sovereign — `proto/alt/knowledge/loop/v1/knowledge_loop.proto` の `KnowledgeLoopEntry` の `reserved 21 to 30` を `21 to 29` に縮め field 30 を `ReviewReason review_reason` として開放。`ReviewReason` enum (UNSPECIFIED / STALENESS / CONTRADICTION / VERSION_DRIFT / UNFINISHED_THREAD / NONE) を追加。sovereign proto も同等に field 29 で追加。`decide_bucket_v2.go` に `decideReviewReason` pure fn 追加。`projector.go` の Entry assembly で `entry.ReviewReason = decideReviewReason(inputs)`。Migration `00022_add_review_reason.sql` で `knowledge_loop_entries.review_reason TEXT NOT NULL DEFAULT 'none'` + CHECK + view refresh。`WhyMappingVersion = 12`。
- alt-backend — `app/domain/knowledge_loop_entry.go` に `ReviewReason` string type + 5 constants。`sovereign_client/knowledge_loop_client.go` に `reviewReasonToProto` / `reviewReasonFromProto`。`connect/v2/knowledge_loop/handler.go` で sovereign domain → `alt.knowledge.loop.v1.ReviewReason` を `mapReviewReason` で変換。
- alt-frontend-sv — `alt.knowledge.loop.v1.ReviewReason` の proto regen により wire 上の review_reason を読める状態に (UI 描画は ADR-000912 で landed の `review-reason-map` が拡張対象)。

## Context

ADR-000907 (Knowledge Loop Review bucket reframing) は Review bucket を「epistemic change-driven re-evaluation queue」と再定義した。Review への配置理由には次の 4 種がある:

1. **version_drift** — SummaryVersionCreated supersede chain で article の認識が古くなった。
2. **contradiction** — SummarySuperseded で article の主張が更新された (drift と異なる SQL 系統)。
3. **unfinished_thread** — Augur conversation がまだ open で、user に continue を促す必要がある。
4. **staleness** — event-time decay (`StalenessScore ≥ 2`) で source_observed_at が 7 日以上前。

現状の `decideBucketV2` は上記 4 種を bucket placement (Changed / Continue / Review) には反映していたが、entry projection には `surface_bucket = 'review'` という結果しか持たず「なぜ Review に入ったか」が失われていた。FE 側 (ADR-000912 で landed の `review-reason-map`) は wire 上に review_reason を期待していたが、projector が emit しないため UI guidance が出せない状態だった。

projector level でこの理由を first-class 化しないと:

- UI は why_kind から「Review に入った理由」を逆推定する必要があり、`why_override` の Kind 選択ロジックと結合する。
- reproject で同じ inputs から同じ reason を再導出できないと「なぜ昔は staleness で今は contradiction なのか」が説明できない。

field 30 はもともと `reserved 21 to 30` で予約されていた (proto 設計時に将来の review-reason のために確保)。今回その予約を消費する。

## Decision

WhyMappingVersion を **11 → 12** に bump し、次を一括で配線する:

1. **proto change**:
   - `alt.knowledge.loop.v1.knowledge_loop.proto`: `KnowledgeLoopEntry` の `reserved 21 to 30` を `21 to 29` に narrow し、`ReviewReason review_reason = 30` を追加。`ReviewReason` enum を追加 (UNSPECIFIED=0 / STALENESS=1 / CONTRADICTION=2 / VERSION_DRIFT=3 / UNFINISHED_THREAD=4 / NONE=5)。
   - `services/sovereign/v1/sovereign.proto`: 同等の `ReviewReason` enum + `KnowledgeLoopEntry.review_reason = 29` を追加。
2. **`decideReviewReason(inputs SurfaceScoreInputs) sovereignv1.ReviewReason`** pure fn を `decide_bucket_v2.go` に追加。priority 順: `VersionDriftCount > 0` → VERSION_DRIFT / `ContradictionCount > 0` → CONTRADICTION / `HasAugurLink || QuestionContinuationScore > 0` → UNFINISHED_THREAD / `StalenessScore >= 2` → STALENESS / それ以外 → NONE。
3. **Migration `00022_add_review_reason.sql`**: `review_reason TEXT NOT NULL DEFAULT 'none'` + `CHECK IN ('staleness', 'contradiction', 'version_drift', 'unfinished_thread', 'none')` + `knowledge_loop_entries_public` view を `CREATE OR REPLACE` で末尾に append。
4. **projector wiring**: `projector.go` の Entry assembly で `entry.ReviewReason = decideReviewReason(inputs)` を全 event-type path で呼ぶ。Review bucket でない場合も NONE で populate して NOT NULL を満たす。
5. **DB driver**: `write_knowledge_loop.go` の UPSERT に `review_reason` を追加 (placeholder $35)。`reviewReasonToDB` / `reviewReasonFromDB` 関数で proto enum ↔ DB string。`read_knowledge_loop.go` の SELECT 句と struct scan を拡張。
6. **alt-backend BFF**: domain `ReviewReason` string type 追加、`reviewReasonToProto` / `reviewReasonFromProto` で sovereign 経路、`mapReviewReason` で public proto への変換。
7. **constants.go**: `WhyMappingVersion = 12` + v12 行を doc string 追加。
8. **runbook**: `knowledge-loop-reproject.md` の cutover ladder に v11 → v12 行を追記。

### priority order の根拠

priority `version_drift > contradiction > unfinished_thread > staleness > none` の根拠:

- version_drift は「versioned artifact の supersede chain」由来で最も deterministic な signal。
- contradiction は同じ supersede chain 由来だが「内容の食い違い」が出ているため、drift と並存しているなら drift を選ぶ (より具体的な evidence)。
- unfinished_thread は user の explicit intent (Augur conversation) なので staleness より優先。
- staleness は event-time decay のみで判断する間接的 signal なので最下層。

### 検討した代替案

- **review_reason を WhyPayload に含める**: 採用しなかった。Why は per-entry の narrative axis、ReviewReason は per-bucket-placement の axis。両者を 1 つの payload に混ぜると proto reproject 時に「why_kind は変わらないが review_reason は変わる」というケース (例: ContradictionCount が 0→1 へ増えるが Kind は CHANGE のまま) の semantics が曖昧になる。
- **WhyKind に NEW_KIND を増やす**: 採用しなかった。Why は user-facing narrative の category、ReviewReason は projector の placement reasoning。WhyKind の数を増やすと UI 側 mapping が肥大化する。

## Consequences

### Pros

- Review bucket への配置理由が wire 上に明示され、UI は guidance line を deterministic に出せる。
- 同じ event log を 2 回 reproject すれば同じ review_reason が出る (pure fn 保証)。
- field 30 は予約済みだったので proto additive 性は維持 (新 field number 消費)。
- DB の NOT NULL DEFAULT 'none' で既存 row も schema migration 1 回で valid 状態に揃う。

### Cons / Tradeoffs

- proto field 30 を反対側 (sovereign 側 field 29) と enum 値で同期する必要があり、片方の改修が忘れられたケースを CI で完全には捕捉できない。`reviewReasonToProto` の switch がフォールバックで UNSPECIFIED を返すため silent な不一致は防げる。
- decideReviewReason は priority order を hard-coded しているため future ADR で順位を変える場合は WhyMappingVersion bump が必要。
- UPSERT 句 placeholder が 34 → 35 に増える。test fixture の更新は今回 `write_knowledge_loop_test.go` で対応済。
- alt-frontend-sv の UI 側は wire 上 review_reason を読めるだけで、実際の guidance line 表示は ADR-000912 の `review-reason-map` 拡張 (別 PR 候補) を必要とする。
