# ADR-000492: Knowledge Home frontend を backend 契約に合わせて縮退する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-frontend-sv (SvelteKit)

## Context

Knowledge Home frontend には、backend がまだ提供していない機能や field を前提とした UI と DTO が残っていた。代表例は `save / unsave` のように UI 上は操作できるが backend では受理しない action と、`knowledge_home.ts` 内で `undefined` や固定値を埋めて維持している field 群である。

また Recall UI には unavailable 表現が存在する一方で、page 側からその状態を渡しておらず、取得失敗と empty state が実際には区別されていなかった。このままでは、frontend が backend 契約より先行した期待を持ち続け、ユーザーには「動くように見えるが正しく接続されていない」体験を与える。

今回の作業では、以下の整理が必要だった。

- backend 未対応の `save / unsave` affordance を止める
- Recall の empty と unavailable を UI で分離する
- backend proto に存在しない値を frontend DTO に仮置きし続ける状態を解消する

## Decision

Knowledge Home frontend は、backend が現在提供している契約に合わせて縮退する。

- backend 未対応の `save / unsave` action は UI から削除する
- dismiss は既存の backend write-through に合わせて維持する
- Recall では empty state と unavailable state を分離し、page が error 状態を明示的に渡す
- `knowledge_home.ts` の DTO から、backend 未提供で固定値投入していた field を削除する
- frontend 派生値として意味があるものだけを残し、契約外の placeholder field を増やさない

今回の実装では、`sourceName`、version 参照、saved/read/dismissed 状態、audio 関連の仮 field など、backend proto と一致しない部分を整理した。Recall については、Home response に候補が含まれない場合のみ既存 endpoint で補完取得を試み、失敗時に unavailable 表示へ倒す。

代替案として「unsupported な field や action を残し、将来 backend が追いつくまで frontend で擬似状態を持つ」も考えられたが、契約 drift を拡大し、回帰時の判断を難しくするため採用しない。

## Consequences

### Pros

- frontend が backend 契約に近づき、実際に動作する機能だけを表示できる
- unsupported action による誤解を防げる
- Recall の取得失敗を empty と分けて扱えるため、UI 上の意味が明確になる

### Cons / Tradeoffs

- `save / unsave` のような将来機能は、一時的に UI 上から見えなくなる
- DTO を縮退したことで、今後 backend が field を追加した際には再度型更新が必要になる
- Recall の unavailable は明示化されたが、失敗理由までは UI に出さない
