# ADR-000777: Acolyte レポート生成に言語クォータ・クロスリンガル執筆・評価ハーネスを導入し、日英ソースの balanced selection を可能にする

## Status

Accepted

## Date

2026-04-19

## Affected Services

- **acolyte-orchestrator (評価ハーネス)** — `evaluation/` パッケージ新規 (`metrics.py` / `dataset.py` / `datasets/baseline.jsonl` / `run_eval.py` / `tests/`)。`pyproject.toml` の `testpaths` に `evaluation/tests` を追加
- **acolyte-orchestrator (Curator)** — `domain/language_quota.py` 新規 + `CuratorNode` 拡張 + `Settings.language_quota_en` / `get_language_quota()`
- **acolyte-orchestrator (Writer)** — `SourceMap.language` 追加、supporting_quotes を `[Sn][lang]` 形式化、3 段落プロンプトに `_CROSS_LINGUAL_RULE` を追加
- **acolyte-orchestrator (CDC)** — `tests/contract/test_search_indexer_consumer.py` で `/v1/search` レスポンスの language フィールドを contract 化

## Context

[[000776]] で `articles.language` → proto → domain → Acolyte `ArticleHit.language` の配管が通り、Acolyte 側から記事言語を観測できるようになった。しかし観測できるだけでは selection は変わらない。ユーザ要件 (2026-04-19 報告) は「日英両方から質の良いものを LLM やシステムが取捨選択できるべき」で、実効的な selection バイアスの是正が必要だった。

現状の Acolyte 選定フロー:

1. `Gatherer` が multi-query + RRF fusion ([[000669]]) で evidence を収集 → BM25 スコア主体
2. `Curator` がセクションごとに `max_evidence=10` を超える場合のみ LLM で上位選択、それ以下はそのまま通す
3. `Writer` が supporting_quotes + evidence_ids を受け取り段落を生成

このフローに言語バイアスが 2 回入る:

- **BM25 レベル**: 日本語クエリ → 日本語記事に高スコア。ACL 2025 Findings `2025.findings-acl.295` で documented な多言語 RAG の同言語偏重。
- **Curator プロンプト**: 現状の `CURATOR_PROMPT` は `id / title / content / tags / score` だけを渡し language は出さない。LLM が明示的に言語バランスを取る根拠を持たない。

Writer 側の課題は別種: 仮に英語記事が evidence_ids に入っても、Writer プロンプトは「日本語で執筆」「supporting_quotes を根拠に」としか指示されず、英語原文をどう本文に溶け込ませるかのルールが無いため、LLM が無視したり、原文を生に貼り付けたりする挙動が混ざる。

さらに前提として、[[feedback_eval_first]] (Alt の Agentic RAG の原則) により「評価基盤を先に作り、指標で判定してから実装」を守る必要がある。改善して悪化したかを言える仕組みが無いと、プロンプト 1 文変えるたびに主観評価のブレで判断が揺れる。

本 ADR はこの 3 つ (評価ハーネス + Curator soft quota + Writer クロスリンガル指示) をまとめて入れる。`Gatherer` は触らない (multi-query RRF + language 無関係な BM25 はそのまま) — Curator 段階で post-rerank quota を掛ける方が、cross-lingual recall を失わずに balance を取れるという設計判断。

## Decision

### 1. 評価ハーネス (B1)

新規パッケージ `acolyte-orchestrator/evaluation/` に CLI と指標を置く。別サービス化せず、単一マシン構成 ([[feedback_single_machine_architecture]]) を守る。

- **`metrics.py`**:
  - `citation_precision(body, source_map, gold_ids) -> float | None`: 本文中の `[Sn]` を解決して得た source_id が gold set に含まれる比率。0 件なら `None` で denominator を守る。
  - `lang_mix_ratio(body, source_map, articles_by_id) -> dict[str, float]`: `[Sn]` で参照された各記事の言語分布。未知言語は `"und"` で集計。
  - `faithfulness(body, evidence_by_short_id, judge) -> float | None`: LLM-as-judge 用のファサード。judge を caller が差し込むため、CI では決定論的 mock、本番では Gemma4 を割り当て可能。
  - `extract_short_ids(body)`: 正規表現 `\[S(\d+)\]` を初出順に抽出 (finalizer と同 regex を独立宣言 — テストで両者 drift を防ぐ)。

- **`dataset.py`**: `EvalCase` dataclass (`topic / query_lang / gold_source_ids: frozenset[str] / expected_lang_mix: dict`) と `load_cases(path)` で JSONL パース。`#` コメント行と空行スキップ。

- **`datasets/baseline.jsonl`**: 5 ケースの初期データセット (JP 2 / EN 1 / 混在 2)。ベースラインは Part A 後に `run_eval` で記録する想定。

- **`run_eval.py`**: `uv run python -m evaluation.run_eval --dataset evaluation/datasets/baseline.jsonl` で JSON 結果を stdout / file に出す。生成関数は caller 側 (本番 Acolyte pipeline / 記録フィクスチャ) が差し込む設計で、CI 側はスケルトン呼び出しのみ。

- **テスト**: 15 ケース (metrics 12 / dataset 3)。`pyproject.toml` の `testpaths` を `["tests", "evaluation/tests"]` に拡張。

### 2. Curator 言語ソフトクォータ (B4)

`domain/language_quota.py::rebalance_by_language(curated, pool, quota)` を新設する。Gatherer の前段で絞ったり Curator プロンプトに言語表示を混ぜる案は下記理由で却下し、**LLM ランキング後のポスト処理**を選ぶ。

```python
def rebalance_by_language(curated, pool, quota):
    # quota: {"en": 0.2} → 全体の最低 20% を en にする
    # curated の順序は維持、足りない言語のみ pool から最高スコアを昇格、
    # 既存 curated の最低スコア (und 優先 → 他言語) を降格
    ...
```

- `quota=None` のとき `{"en": 0.2}` をデフォルトとして**新規 dict を毎回生成**し、mutable default を共有しない。
- `share=0.0` なら enforcement 無効。
- pool が target language を含まなければ no-op。
- `curated` の長さは保つ (swap のみ、拡張しない)。
- 降格候補の優先順は「language=und が最弱」「同言語なら最低スコア」。これにより、正しく ja/en 判定された記事は互いに競合しなければ温存される。

`CuratorNode.__init__` は `settings: Settings | None = None` を取るよう拡張し、`_language_quota()` で `Settings.get_language_quota()` から dict を都度取得。quota が空でない場合だけ `curated_by_section` 各セクションに対して `rebalance_by_language(section_curated, section_pool, quota)` を適用する。

8 ケースの単体テストで「pool に en が居れば promotes」「quota 達成済みなら no-op」「pool に en が無ければ no-op」「curated 長を保つ」「mutable default 非共有」「quota=0 は enforcement 無効」「und 降格優先」などを固定。

### 3. Writer クロスリンガル執筆 (B5)

- `SourceEntry.language: str = "und"` を追加。`SourceMap.register(...)` / `to_dict` / `from_dict` を拡張し、チェックポイント state 互換を保つ (language 未包含の古い state からの deserialize でも `"und"` で初期化)。
- `CuratorNode` が `SourceMap.register(source_id=..., language=item.get("language") or "und")` で language を登録。
- `WriterNode._format_supporting_quotes(quotes, evidence_ids, languages)` を拡張し:
  - language が `"und"` でも空でもなければ `- [S1][ja] "…"` / `- [S2][en] "…"` 形式で出力
  - language 情報が無いかすべて `"und"` のケースでは既存形式 `- [S1] "…"` のまま
- 3 段落プロンプト (analysis / conclusion / executive_summary) に `_CROSS_LINGUAL_RULE` 定数を組み込む:
  ```
  - supporting_quotes の行末に [en] タグが付いている原文を使う場合は、
    本文中で「原文の主旨を要約した日本語」を書き、続けて（原文: "…冒頭40字…"）[Sn] を付す
  ```
- 英語 quote が 1 件も無い場合でも指示自体は残す (「[en] タグが付いている場合」の条件節で無害)。

### 4. CDC の追随

`tests/contract/test_search_indexer_consumer.py::test_search_articles` の contract body を日英 2 ヒットに拡張し、レスポンスに `language: "en"` と `language: "ja"` を含ませる。`tests/contract/` は Pact broker に publish されるため、この時点で consumer 側の期待が固定され、provider (search-indexer) が language を返さなければ pact verification が赤くなる。provider 側実装 ([[000776]]) が既に language を返すよう配線されているため、実運用で pact は green のまま通る。

### 検討した代替案

| 案 | 採用 | 却下理由 |
|---|---|---|
| **Curator post-rerank + Writer gloss + 評価ハーネス**（採用） | ○ | cross-lingual recall を保ったまま決定的に balance を取れる。プロンプト加筆より regression が起きにくく、指標で効果を数値化できる |
| A: Gatherer 前段で言語別クエリ発行 | × | Meilisearch 往復が倍に。RRF fusion で混在がそのまま取れるのを捨てる |
| B: Curator LLM プロンプトに「言語バランスを取れ」を追記 | × | LLM は soft instruction を無視する確率が有意。決定論性に欠け、評価ハーネスでの regression 検知が難しい |
| C: 多言語 reranker (Jina v2 / BGE-reranker-v2-m3) を挟む | × | 単一マシン + ホスト GPU の VRAM を消費。現状の単純 rebalance で初期目標 (`lang_mix_ratio.en >= 0.15`) は達成可能と判断 |
| D: Writer プロンプトだけ入れて Curator は触らない | × | evidence に en が無ければ writer が書きようがない。Curator 段階での確定投入が先 |
| E: 評価ハーネスを後回しにして先に Curator / Writer を入れる | × | [[feedback_eval_first]] と衝突。効果の有無を指標で言えないまま実装すると、調整ループに入れない |
| F: `lang_mix_ratio` を厳格 (minimum) 制約として扱う | × | pool に en が無い時に curator が空になる。soft quota (pool にあれば達成、無ければ no-op) を選ぶ |

## Consequences

### Pros

- `lang_mix_ratio` と `citation_precision` がランタイムで測定可能になり、以降の Acolyte 改善はすべて指標で argue できる。[[feedback_eval_first]] を満たす。
- Curator の post-rerank rebalance が決定論的に作用するため、日英混在トピックで英語記事が pool にあれば最低 `language_quota_en=0.2` (初期値) を達成する。pool に英語が無ければ no-op なので、日本語単独トピックの挙動は壊れない。
- Writer が supporting_quotes に `[en]` ラベルを受け取ると、日本語本文中に英語原文を溶かす定型 (`日本語要約（原文: "…"）[S3]`) が出るようになり、クロスリンガル引用が可読になる。
- SourceMap のシリアライズ拡張は後方互換で、既存チェックポイント state (`language` なし) からの resume も壊れない。
- CDC に language フィールドを組み込んだことで、provider 側 (search-indexer) が将来 language を落としたら pact で検出できる。
- acolyte-orchestrator 591 テスト + evaluation/tests 15 テストが緑で着地。search-indexer の既存 CDC / pre-processor / alt-backend も全緑。

### Cons / Tradeoffs

- **quota の初期値 `en=0.2` はチューニング対象**。baseline eval の `lang_mix_ratio` を見てから、必要に応じ `language_quota_en` を env var で上げる。現状は Settings 固定で per-request 変更はできない。
- **und 記事の優先降格**が rebalance で機械的に起きる。現状 ([[000776]] 導入直後) は既存記事がほぼすべて `und` なので、「言語が確定していない記事はバランス調整で真っ先に外される」副作用が強く出る可能性がある。pre-processor が新規記事を正しく付与し始めると自然に解消するが、backfill ジョブが回るまでは short-term で「英語記事が prioritize されすぎる」ケースが起きうる。
- **Writer `_CROSS_LINGUAL_RULE` はプロンプト追加 1 行**。LLM が guidance を無視して英語原文を生で貼り付けるケースが残りうる。その場合 `citation_format.py` ([[000775]]) のバリデータが `https?://` や長ブラケットだけは弾くが、英語プロパーが入っただけではバリデータは通す。
- **評価ハーネスの faithfulness は LLM-as-judge で、scoring LLM の癖に依存**する。CI では mock / 本番で Gemma4 を差し込む運用想定だが、judge プロンプトは暫定で厳密テンプレート化されていない。精度調査は `run_eval.py` の成熟後に別 ADR で扱う。
- **baseline.jsonl が 5 ケース**しか無く、統計的有意性は弱い。規模拡張は運用で継続的に進める。
- **チェックポイント互換**は language 追加のみで確保したが、`language_quota_en` を Settings 経由で読むため、旧 Settings を持つ環境では `get_language_quota()` 呼び出しが AttributeError にならないよう必ず `Settings` bump と同時にデプロイする必要がある。今回は同一デプロイで落ちるため問題ないが、rollback の順序には注意。
- **quota enforcement の aggressive 挙動**: pool に en が 1 件だけあり、curated が 10 件すべて score の高い ja であっても、最下位の ja が en で置換される。スコア差が大きい場合「relevance を少し犠牲にして balance を取る」ことになる。これは要件通りだが、**relevance 最優先のレポート** (例: `market_analysis` で日本市場 deep-dive) では誤発火する可能性がある。今後 section タグごとに quota を上書きする設計に拡張する余地を残す (現状は全セクション一律)。
