# 000379: PgBouncer UUID 配列エンコードエラーと k6 RegisterFavoriteFeed URL 不一致の修正

## ADR's STATUS

Accepted

## CONTEXT

2026-03-13 実施の sv-connect-composite-10m 負荷試験（ADR 000377 の SSRF バリデータ修正後の検証）で、2 つの残存バグが特定された。

1. **FetchArticlesCursor RPC の 96% 失敗** — `FetchArticlesByIDs` が PgBouncer 経由で `[]uuid.UUID` を渡した際に「OID 0: cannot find encode plan」エラー
2. **RegisterFavoriteFeed RPC の 100% 失敗** — k6 テストシナリオが RSS フィード URL を送信していたが、バックエンドは記事 URL（`feeds.link`）で検索

いずれも SSRF 修正により `FetchArticleContent` が正常動作するようになった結果、初めて顕在化した潜在バグである。browse_flow / discovery_flow は 100% 安定しており、read_flow（4.18%）と manage_flow（0%）が影響を受けていた。

## DECISION MAKING

### バグ 1: PgBouncer + pgx SimpleProtocol での UUID 配列エンコード

**根本原因**: `fetch_articles_by_ids_driver.go` が `[]uuid.UUID`（google/uuid）を pgx Query に直接渡していた。`init.go` で `QueryExecModeSimpleProtocol` を設定済みだが、SimpleProtocol モードでは pgx がクライアントサイドでパラメータをテキスト補間する際に google/uuid の配列型の OID を解決できない。

**既存の解決パターン**: 同一リポジトリ内の `GetReadFeedIDs`（`fetch_feed_driver.go:484-495`）が既に同じ問題を解決していた:

1. `[]uuid.UUID` → `[]string` へ変換
2. SQL で `ANY($1::uuid[])` と明示キャスト
3. `[]string` を Query に渡す

**決定**: `FetchArticlesByIDs` にも同一パターンを適用する。他の `ANY($1)` を使用するクエリ（`article_head_driver.go`, `fetch_tag_cooccurrence_driver.go`, `fetch_inoreader_summaries_driver.go`）は全て `[]string` を渡しており、今回のバグは `FetchArticlesByIDs` のみに限定されることを確認した。

pgx の `AfterConnect` で `RegisterDefaultPgType` を使う方法も検討したが、既存パターンとの一貫性と変更の最小化を優先し、ドライバーレイヤーでの `[]string` 変換を採用した。

### バグ 2: k6 シナリオの RegisterFavoriteFeed URL 不一致

**根本原因**: k6 の `flowManageFeeds` が `RegisterFavoriteFeed` に RSS フィード URL（`feedUrl`）を送信していた。しかしバックエンドの `register_favorite_feed_driver.go` は `SELECT id FROM feeds WHERE link = $1` で検索し、`feeds.link` カラムには RSS `<item><link>` で取得した記事ページ URL が格納されている。

**フロントエンドの実際の挙動**: `VisualPreviewCard.svelte` と `FeedDetails.svelte` は `registerFavoriteFeedClient(feed.link)` を呼び出しており、`feed.link` は記事 URL である。つまりバックエンドの実装は正しく、k6 シナリオが実際のユーザーフローと異なる URL を送信していた。

**決定**: k6 シナリオの `{ url: feedUrl }` を `{ url: articleUrl }` に変更する。`articleUrl` は既に `buildMockArticleUrl(feedKey, 1)` で計算済みであり、mock-rss-server が RSS XML の `<item><link>` として返す値と一致する。

## RESULTS, EFFECTS

### PROS

- **read_flow**: FetchArticlesCursor の UUID エンコードエラーが解消され、PgBouncer トランザクションプーリングとの互換性が確保された
- **manage_flow**: RegisterFavoriteFeed が実際のフロントエンドと同じ URL 形式を使用し、正しくフィードを検索できるようになった
- **既存パターンとの一貫性**: `GetReadFeedIDs` と同じ `[]string` 変換 + `::uuid[]` キャストパターンに統一
- **全 68 パッケージのテストがパス**: リグレッションなし
- **変更の最小化**: ドライバー1ファイル + k6 シナリオ1ファイル + テスト2ファイルのみ

### CONS, TRADEOFF

- `[]uuid.UUID` → `[]string` 変換は各クエリサイトで個別に行う必要がある。pgx の `AfterConnect` で google/uuid の型を一括登録すれば変換コードを省略できるが、既存パターンとの一貫性を優先した
- k6 シナリオの修正は「テストの修正」であり、プロダクションコードのバグ修正ではない。実際のフロントエンドでは問題なく動作していた

### UUID 配列エンコードの修正パターン

```go
// Before (OID 0 エラー)
rows, err := r.pool.Query(ctx, query, articleIDs) // []uuid.UUID

// After (PgBouncer SimpleProtocol 互換)
stringIDs := make([]string, len(articleIDs))
for i, id := range articleIDs {
    stringIDs[i] = id.String()
}
// SQL: WHERE a.id = ANY($1::uuid[])
rows, err := r.pool.Query(ctx, query, stringIDs) // []string
```
