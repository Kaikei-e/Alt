# ADR-000871: Atlas migrator image を build-time network-dependency から解放し production deploy hang 復旧経路を二重化する

## Status

Accepted

## Date

2026-04-30

## Affected Services

(frontmatter 参照)

## Context

### 直接の動機 — alt-deploy run 25143212120 の hang

2026-04-30 の knowledge-sovereign deploy で `Apply Atlas migrations` task が **1 時間 13 分** docker compose run --rm のレベルで停止。診断結果:

| 当初仮説 | 実際の根因 |
|---|---|
| `ALTER TABLE knowledge_home_items RENAME COLUMN link TO url` の ACCESS EXCLUSIVE lock 待ち | DB 上に lock 競合は無く、migration `00017` は **既に適用済** だった (`atlas_schema_revisions.applied = 2`) |
| Atlas advisory lock 残骸 | 該当無し |
| Migrator container の起動失敗 | 起動自体できておらず、build phase で停止 |
| **真因**: `arigaio/atlas:latest-alpine` の上に重ねた `RUN apk add --no-cache postgresql-client` が dl-cdn.alpinelinux.org への HTTP fetch で hang。BuildKit executor 内の `apk` プロセスが 73 分 sleep。fresh container での同 `apk` は 1 秒で完了 → 一過性の TCP/IPv6 問題と推定 |

破綻メカニズム:

1. compose 定義に `image: ghcr.io/...` が無く `build:` だけ → `docker compose run --rm` は毎回 build を試行
2. `Dockerfile.migrator` の `apk add postgresql-client` が build cache invalidation で再実行
3. apk が dl-cdn から package metadata fetch を試みる際に slow path (IPv6 only resolution / TCP half-open / Fastly route degradation) に落ちる
4. atlas migrate apply に到達する前に build phase が無限待機
5. 上位 ansible task (`failed_when: false` でも) には pid alive のため成功も失敗も報告されず、deploy job が hang したまま

### `pg_isready` を `apk add postgresql-client` で持ち込んでいた理由

各 migrator の `migrate.sh test_connection()` で DB readiness を pg_isready 経由で確認していた。これは「Atlas が migrate apply で接続失敗した時のエラーが分かりにくいので、pre-flight で先に DB が起き上がっているか確かめる」防衛コード。

しかし **pre-flight としては過剰**:
- compose の `depends_on: { condition: service_healthy }` で sovereign-db の healthcheck が事前に通っていることは保証済
- Atlas 自身が migrate apply で接続失敗時に十分明確な `pq: connection refused` 等を吐く
- TCP open-port probe (busybox `nc -z`) で「listener が起きてる」確認は十分

### lock_timeout なしの `atlas migrate apply` の二次リスク

今回の hang は build phase だったが、**もし migration が DB 側 lock 競合に当たったら同じく無限待機になる構造**。Postgres `lock_timeout` のデフォルトは 0 (無制限) で、Atlas は `--lock-timeout` を持つが advisory lock 限定 (per-statement の lock 待ちは制御不能)。本番 sovereign 上で稼働中の service が `knowledge_home_items` に shared lock を持っていたら、`ALTER TABLE` は ACCESS EXCLUSIVE 取得で永遠に待つ。今回のシナリオで lock 競合が真因だった場合、同じ "deploy hang 1 時間" になっていた。

## Decision

**A. apk-install 自体を排除し、build-time network 依存をゼロにする** + **B. PostgreSQL `lock_timeout=30s` を DSN 経由で注入し per-statement lock 待ちを 30 秒で fail させる** の 2 階建て対策を 7 migrator (sovereign / alt-backend × 2 / pre-processor / rag / recap / acolyte) に一律適用する。

### 補足: 出版前監査で取り込んだ修正

セキュリティ監査 (OWASP Top 10:2025 / ASVS 5.0) と Web 調査 (postgresql.org / atlasgo.io / docker hub 公式) を経て、以下を本 ADR の決定範囲に追加:

- **Image tag を `latest-alpine` から `1.2.0-alpine` に pin** (security-auditor F-3 / A08 Software Integrity / CWE-1357)。`latest` は moving tag で supply-chain 観点で本番非推奨 ([PostgreSQL libpq URI 仕様](https://www.postgresql.org/docs/current/libpq-connect.html#LIBPQ-CONNSTRING-URIS) と同根の "依存固定" 原則)。`0.31.0-alpine` で先行 pin 済の 3 migrator は不変、残り 4 migrator (sovereign / alt-backend × 2 / rag) を `1.2.0-alpine` に揃える。Atlas は 0.x → 1.x で major bump 済なので 0.31.0 → 1.2.0 一斉昇格は別 PR に切り出す
- **DSN 構築 caller-side 制約を migrate.sh 内コメントで明文化** (security-auditor F-1 / A05 Injection / CWE-74)。本番 Docker secret は alphanumeric+`-_` 構成で現状 exploitable でないが、libpq は password の percent-encoding を行わない仕様 ([PostgreSQL §34.1.1.2](https://www.postgresql.org/docs/current/libpq-connect.html#LIBPQ-CONNSTRING-URIS)) のため、secret rotation policy に `?` `&` `%` `@` `/` `:` `#` 許容が含まれた場合に compose entrypoint 側で encode 必須となる旨を sovereign migrate.sh のコメントに追記

### A. apk-install 削除

7 つの `Dockerfile.migrator` (および `Dockerfile.migrate` / `Dockerfile`) から:

```dockerfile
RUN apk add --no-cache postgresql-client
```

を削除。各 `migrate.sh` の `test_connection()` 内 `pg_isready` を以下に置換:

```bash
if nc -z -w 5 "$DB_HOST" "$DB_PORT"; then
    log_success "Database listener reachable at $DB_HOST:$DB_PORT"
else
    log_error "Cannot reach database listener at $DB_HOST:$DB_PORT"
    exit 1
fi
```

`nc` は `arigaio/atlas:*-alpine` 標準の busybox に含まれるので追加 install 不要。build は base image pull のみで完結 (network 依存は base layer pull 時の 1 回のみ、既存の cache hit で skip 可能)。

### B. lock_timeout DSN 注入

`apply_migrations()` 内で:

```bash
local apply_url="$DATABASE_URL"
case "$apply_url" in
    *\?*) apply_url="${apply_url}&options=-c%20lock_timeout%3D30000" ;;
    *)    apply_url="${apply_url}?options=-c%20lock_timeout%3D30000" ;;
esac

atlas migrate apply --url "$apply_url" ...
```

PostgreSQL は `options=-c lock_timeout=30000` を受け取るとセッション開始時に `SET lock_timeout = '30000ms'` 相当を実行。Atlas が発行する任意の DDL/DML 文は 30 秒以内に必要なロックを取得できなければ `ERROR: canceling statement due to lock timeout` で失敗する → 1 時間 hang ではなく 30 秒で fail 表面化 → operator が即対応可能。

### 検討した代替案

#### 1. Migrator image を CI で build & GHCR push する (`compose-image-audit.py` 的な経路)

却下。

- migrator は services.yaml の pacticipant/runtime に登録されておらず、追加には `docker-build.yaml` の SERVICES 配列拡張 + `compose-image-audit.py` の対象拡張 + 各 migrator service への `image:` 追加が必要 (3 layer の差分)
- migrator image は実質 atlas binary + migrations/ + scripts/ で、本来 build cost が極小。CI で push する経路を太らせる ROI が低い
- apk install を排除すれば build 時間自体が 1 秒台になり、prod build の意味自体が消える

#### 2. Dockerfile を Debian-base atlas image に切り替えて apk を不要にする

却下。

- arigaio が出している official tag は alpine が中心で、debian variant は size と維持コストが大きい
- apk 排除 + busybox nc で 1 行レベルで解決できる問題に対し base image 変更は overkill

#### 3. `atlas migrate apply --lock-timeout 30s` だけで対応

却下。

- Atlas の `--lock-timeout` は advisory lock acquisition 限定で、per-statement lock wait には効かない
- ALTER TABLE が ACCESS EXCLUSIVE 待ちで hang するシナリオに対しては無効
- DSN options で PostgreSQL session-level lock_timeout を注入する方が広範囲に効く

#### 4. apk install を `RUN apk add --network-timeout 30 ...` で短縮

却下。

- `apk` (Alpine BusyBox 由来) には `--network-timeout` flag が無い
- `--no-cache` の挙動を変えるオプションも存在しない
- 排除する方が確実

## Consequences

### Pros

- Migrator image build から network 依存を排除 → CI / 本番ホストの一過性の Alpine CDN 障害が deploy を hang させない
- Build 時間が 1 秒台に短縮 → `docker compose run --rm migrator apply` の overhead が事実上ゼロ
- `lock_timeout 30000` で DB 側 lock 競合シナリオが 30 秒で fail-fast → operator が `pg_terminate_backend` 経路に即移行できる
- 7 migrator 全部に同一パターン適用済 → 将来の migrator 追加時にコピペ元として確立
- compose-image-audit / pact CDC 等の既存 CI gate に影響無し (audit 19 fragment / 21 GHCR-registered service all green)

### Cons / Tradeoffs

- `pg_isready` が DB 側 PostgreSQL バージョン互換性の早期検出器として動いていた可能性 (例: 古いクライアントが新しい DB に接続できないケース) を失う。実害はほぼ無いが、防御は浅くなる
- `nc -z` は TCP listener 開いてるかしか確認しない → DB process が listening port に bind してるが authentication が壊れてるケースは検出できない。Atlas migrate apply 自体が即 fail してくれるので致命的ではない
- `lock_timeout` 注入で DSN 文字列が長くなる。compose の `entrypoint` 内 export には影響無し (script 内で ad-hoc に追加)
- Cosmetic: 同じパターンを 7 箇所コピペした (DRY 違反)。共通 helper script に抽出する余地はあるが、各 migrator の独立性 (rolling deploy で別々に build されうる) を優先した

### 現在の DB 状態への影響

無し。本変更は migrator image の build 段階と migrate.sh の挙動のみで、DB スキーマ・データに変更を加えない。production sovereign-db は既に migration `00017` 適用済 (`column = url`) で稼働継続。
