# HostRateLimiter の burst 設定可能化と ConnectError abort 表示抑制

## ADR's STATUS

Accepted

## CONTEXT

デスクトップ UI の FeedDetailModal で「Mark As Read」ボタンを押すと、次の記事に自動遷移し、同時に2つのエラーが頻発していた。

### エラー1: `[resource_exhausted] rate limited`

Mark As Read → 次の記事へ遷移 → `handleFetchFullArticle()` が即座に API 呼び出し → 500ms 後に `triggerPrefetch()` が同一ホスト (例: zenn.dev) の次々の記事をプリフェッチ → バックエンドの `HostRateLimiter`（**burst=1, interval=10s/host**）が2番目のリクエストを10秒ブロック → `fetchCtx` の 25s タイムアウト超過。

根本原因は `rate.NewLimiter(rate.Every(interval), 1)` の **burst=1 がハードコード** されていたこと。同一ホストへの2リクエスト目は必ず interval（10s）待ちとなり、ユーザーが記事を連続で既読にする操作パターンに対応できなかった。

さらに、feed collector・article fetcher・batch fetcher が**同一の `HostRateLimiter` インスタンスを共有**しており、バックグラウンドのフィード収集ジョブがトークンを消費することで、ユーザーインタラクティブなリクエストがさらに遅延していた。

### エラー2: `[canceled] signal is aborted without reason`

記事遷移時に `contentAbortController.abort()` で前のリクエストをキャンセルすると、Connect-RPC が native `AbortError` を `ConnectError` にラップする。既存のエラーハンドリング `err.name === "AbortError"` ではラップ後の `ConnectError`（`name === "ConnectError"`）を検出できず、エラーメッセージが赤枠で UI に表示されていた。

モバイル側の SwipeFeedCard / VisualPreviewCard では `err.message.includes("abort") || err.message.includes("cancel")` パターンで正しく抑制できていたが、デスクトップの FeedDetailModal には適用されていなかった。

### 前提: 対策A・B（前回実装済み）

- **対策A**: `FetchCompliantArticle` に `singleflight.Group` を導入し、同一 URL への並行リクエストを重複排除 → 同一 URL の問題は解決
- **対策B**: `ArticlePrefetcher` に `activeHostFetches: Set<string>` を導入し、同一ホストの同時プリフェッチを1に制限 → プリフェッチ間の重複は解決

今回は**異なる URL × 同一ホスト**の連続リクエスト問題と、**ConnectError ラッピングによるエラー表示**問題への対策。

## DECISION MAKING

### 対策C: HostRateLimiter の burst パラメータを設定可能に（default: 3）

`NewHostRateLimiter` に可変長引数 `burst ...int` を追加し、後方互換性を保ちつつ burst を設定可能にした。

```go
func NewHostRateLimiter(interval time.Duration, burst ...int) *HostRateLimiter
```

- **burst=3（デフォルト）**: 最初の3リクエストは即座に実行（直接 fetch + prefetch 2件分）、4番目以降は interval 待ち
- **平均レート**: 1/10s（プロジェクトの5秒ルール遵守）
- **可変長引数**: 既存の全呼び出し元（35+箇所）を変更不要

`RateLimitConfig` に `ExternalAPIBurst` フィールドを追加し、環境変数 `RATE_LIMIT_EXTERNAL_API_BURST` でオーバーライド可能とした。DI Container のメインの rate limiter のみ burst=3 を適用し、image proxy 用など他の rate limiter はデフォルト burst=1 のまま。

### 対策D: FeedDetailModal の abort/cancel エラー表示抑制

モバイルコンポーネントで実績のあるメッセージ内容チェックパターンを FeedDetailModal に適用:

```typescript
if (err instanceof Error) {
    if (err.name === "AbortError") return;
    if (err.message.includes("abort") || err.message.includes("cancel")) return;
}
```

`handleFetchFullArticle` の catch、`handleSummarize` の catch、および onError コールバックの3箇所に適用。

## RESULTS, EFFECTS

### PROS

- **Mark As Read 連打が可能に**: 同一ホストの記事を3件連続で既読にしても `rate limited` エラーが発生しない
- **`[canceled]` エラーの非表示化**: ユーザーが記事遷移時に技術的なエラーメッセージを見ることがなくなった
- **後方互換性**: 可変長引数により既存コードの変更が不要。テスト含む35+箇所の呼び出し元はそのまま動作（burst=1）
- **設定可能**: 環境変数でバースト値を調整可能。外部サイトから 429 を受けた場合は `RecordRateLimitHit` で自動バックオフ
- **前回対策との相乗効果**: singleflight（同一URL重複排除）+ 同一ホストプリフェッチ制限 + burst=3 の3層防御

### CONS, TRADEOFF

- **burst=3 で瞬間的に3リクエストが同一ホストに到達**: 外部サイトの rate limit に抵触するリスクがわずかにある。ただし singleflight による重複排除と DB キャッシュにより、実際に外部サイトに到達するのは DB 未ヒットの新規記事のみ
- **メッセージベースのエラー判定**: `err.message.includes("cancel")` は正当な "cancel" を含むエラーメッセージも抑制する可能性がある。Connect-RPC の `ConnectError` に対して code ベースの判定（`Code.Canceled`）が理想的だが、現時点ではメッセージ判定で十分と判断（モバイルで実績あり）
