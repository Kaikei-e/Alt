# ADR-000610: HybridPrioritySemaphore の call_soon_threadsafe を排除し be_slots=0 構成でのスロットリークを修正する

## Status

Accepted

## Date

2026-03-29

## Affected Services

- news-creator (Python)

## Context

[[000609]] の設定統一（`OLLAMA_NUM_PARALLEL=1`）により `total_slots=1, rt_reserved=1` 構成（= `be_slots=0`）で運用を開始したところ、`SLOT INVARIANT VIOLATION` が **109回** 発生し、セマフォスロットが完全に消失する事象が確認された（`rt_available=0, be_available=0, acquired_count=0, total_slots=1`）。

### 影響

- BE リクエストのキュー待ちが最大 **281秒** に到達
- プリエンプション発動時に **500 Internal Server Error** が返却
- ローカル GPU がアイドル状態であるにもかかわらず、セマフォスロット消失により新規リクエストが処理不能

### 根本原因

`release()` メソッドがキュー待機中の waiter を起床する際に `loop.call_soon_threadsafe(future.set_result, home_pool)` を使用していた。`call_soon_threadsafe` は callback をイベントループの次のイテレーションにスケジュールするため、以下の race condition が発生する:

1. `release()` が `future.set_result(home_pool)` をスケジュール（`woke_up=True` でプールカウンタ非加算）
2. `release()` がリターン
3. HTTP クライアントの切断等により `task.cancel()` → `future.cancel()` が先に実行
4. スケジュール済みの `set_result()` がキャンセル済み future に対して実行 → `InvalidStateError`
5. `InvalidStateError` はイベントループの例外ハンドラに吸収され、スロットが永久消失

この問題は [[000601]] で導入した `home_pool` 所有権追跡と [[000606]] で導入した `slot_id` propagation の上位レイヤーで発生しており、所有権追跡自体は正しく動作していた。

## Decision

### 1. `call_soon_threadsafe` を直接 `set_result()` に置換

`release()` は同期メソッドであり、asyncio イベントループと同一スレッドから呼び出される。`call_soon_threadsafe` による遅延スケジューリングは不要であり、直接 `future.set_result()` を呼ぶことで race condition を排除する。

### 2. `_try_wake_waiter()` ヘルパーの導入

waiter 起床ロジックを `_try_wake_waiter(request, home_pool)` メソッドに抽出し、以下を保証する:

- `future.done()` / `future.cancelled()` の事前チェック
- `set_result()` を直接呼び出し（遅延なし）
- `InvalidStateError` をキャッチし、スロットを失わずに次の waiter を試行

### 3. `be_slots=0` 構成のテスト追加

`total_slots=1, rt_reserved=1` での以下のシナリオをカバーする 5 件のテストを追加:

1. BE が RT スロットにフォールバック → release → RT プールに復帰
2. BE release → RT waiter への正常転送
3. BE release → BE waiter への正常転送（home_pool=rt が維持される）
4. **キャンセル済み waiter へのスロット転送失敗時にスロットが消失しない**
5. 20 回の rapid acquire/release で invariant が維持される

## Consequences

### Pros

- **スロットリーク解消**: `call_soon_threadsafe` の race condition が排除され、`SLOT INVARIANT VIOLATION` が発生しない
- **`be_slots=0` 構成の安全な運用**: `total_slots=1` の RAG 専用構成が正式にサポートされる
- **PM-2026-012/013 の未着手アクション完了**: P-6（`be_slots=0` パス監査）が完了

### Cons / Tradeoffs

- **`release(slot_id=...)` の全 caller 必須化は未完了**: legacy fallback（`slot_id=None` → oldest matching slot 推定）はまだ残存。段階的に排除する必要がある
- **property-based testing 未導入**: PM-2026-012 で提案された Hypothesis による形式的テストは今回のスコープ外
