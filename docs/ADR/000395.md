# ADR-000395: alt-frontend完全廃止とalt-frontend-svへの完全移行およびBFF REST統合

## Status

Accepted

## Date

2026-03-17

## Affected Services

- alt-frontend-sv (SvelteKit) — 唯一のフロントエンドに昇格
- alt-butterfly-facade (Go) — REST APIプロキシ機能を追加
- nginx — ルーティング全面再構成
- auth-hub — Kratos URLの`/sv`除去（設定のみ）

## Context

alt-frontend（Next.js）とalt-frontend-sv（SvelteKit）が並行稼働していた。SvelteKitは`/sv/`パス配下で提供されており、Next.jsがルートパスを占有していた。

この二重構成には以下の問題があった:

1. **運用コストの二重化**: 2つのフロントエンドのビルド・デプロイ・監視が必要
2. **nginx設定の複雑化**: Next.js用とSvelteKit用のlocationブロックが混在し、auth_requestフローの違い（Next.jsはnginxレベル、SvelteKitはhooks.server.tsレベル）がルーティングを複雑にしていた
3. **REST通信の直接接続**: SvelteKitの一部API routes（dashboard、OPML等）がBFFを経由せずalt-backendに直接接続しており、JWT検証ポイントが分散していた
4. **`/sv/`ベースパス**: すべてのフロントエンドURLに`/sv/`プレフィックスが付き、URLの可読性とKratos設定の複雑さが問題だった

SvelteKitへの機能移行はすでに完了しており、Next.jsを完全に削除できる段階にあった。

## Decision

**alt-frontend（Next.js）を完全削除し、alt-frontend-sv（SvelteKit）をルートパス（`/`）から提供する。同時に、SvelteKitの全REST通信をBFF（alt-butterfly-facade）経由に統一する。**

具体的に以下の5つの変更を行う:

### 1. BFF REST プロキシの追加

alt-butterfly-facadeに`/v1/`パスのREST APIプロキシ機能を追加する。既存のConnect-RPCプロキシ（`/`）と共存させ、JWT検証付きの透過的転送を行う。

**検討した代替案:**
- SvelteKitから引き続きalt-backendに直接接続 → JWT検証ポイントの一元化ができない
- REST呼び出しをすべてConnect-RPCに移行してからBFF統合 → 移行スコープが大きすぎる（dashboard等にプロトコル定義がない）

BFFのREST プロキシは最小限（JWT検証+転送のみ）とし、BFF機能（キャッシュ、サーキットブレーカー、dedup）は適用しない。REST APIは将来的にConnect-RPCへ移行予定であり、過渡的なプロキシに機能を追加する意味がないため。

HTTP/1.1トランスポートを使用する（alt-backendのREST APIはh2cを使わないため）。これはTTSプロキシの既存パターンと同じ。

### 2. SvelteKit通信のBFF経由統一

`BACKEND_BASE_URL`（alt-backend直接）を廃止し、`BACKEND_CONNECT_URL`（BFF経由）に統一する。8つのREST API routeファイルと`backend-rest-client.ts`が対象。

### 3. `/sv/`ベースパス完全廃止

`svelte.config.js`の`base`を空文字に変更し、25以上のファイルからハードコードされた`/sv/`参照を除去する。テストの期待値もすべて更新。リダイレクトは設けず、完全廃止とする。

### 4. alt-frontendの削除とnginx再構成

alt-frontendディレクトリを物理削除し、nginx設定を全面的に書き換える。Next.js固有のlocationブロック（`/api/frontend/`、`/api/backend/`、`/api/auth/csrf`→auth-hub直接、`/`→Next.js）をすべて削除し、SvelteKit向けのルーティングに置き換える。

`^~ /api/v1/sse/`修飾子でブラウザ直接SSEをregex `~ ^/api/.*(stream|sse)` より優先させ、競合を解決する。

### 5. Kratos URL・CI更新

`.env`のKratos関連URL（`KRATOS_LOGIN_UI_URL`等）から`/sv`を除去。Next.js用のGitHub Actionsワークフロー2つを削除し、残る4つのワークフローからalt-frontend参照を除去する。

## Consequences

### Pros

- **運用コスト半減**: フロントエンドが1つになり、ビルド・デプロイ・リソース消費が削減（Next.jsコンテナ: 1GBメモリ制限が不要に）
- **JWT検証の一元化**: すべてのフロントエンド→バックエンド通信がBFFを経由し、認証ポイントが統一される
- **URLの簡潔化**: `/sv/feeds` → `/feeds` でユーザー体験が向上
- **nginx設定の簡素化**: auth_requestフローが不要になり、全リクエストがSvelteKit→BFF→alt-backendの単一パスを通る
- **Connect-RPC移行の土台**: REST→BFF経由が完了しており、将来的にプロトコル定義を追加するだけでConnect-RPCに移行できる

### Cons / Tradeoffs

- **BFF単一障害点の拡大**: REST通信もBFF経由になるため、BFFダウン時の影響範囲が拡大する。ただしBFFにはすでにサーキットブレーカーとヘルスチェックがあり、リスクは限定的
- **REST→BFF→alt-backend のホップ追加**: 直接接続と比較して1ホップ増えるが、同一Docker network内のため影響は軽微（<1ms）
- **ブラウザキャッシュに旧パスが残る可能性**: SvelteKitのバージョンスタンプによるリフレッシュ強制で対応
- **Kratos return_to URLに旧`/sv/`が残るセッション**: 既存セッションの有効期限切れまで最大24時間、旧URLでのログインリダイレクトが発生しうる。SvelteKitのcatch-allルートが`/home`にリダイレクトするため、ユーザー影響は最小限
