# 000361: Tag Verse 惑星数のデフォルトを 200 → 300 に増加

## ADR's STATUS

Accepted

## CONTEXT

ADR-360 で Barnes-Hut アルゴリズム（O(n log n)）とインメモリキャッシュを導入した結果、Tag Verse のレイアウト計算性能が大幅に向上した。ベンチマーク結果は以下のとおり:

| ノード数 | 計算時間 |
|---------|---------|
| 200 | 36ms |
| 500 | 336ms |

デフォルト limit=200 の場合、レイアウト計算は 36ms で完了しており、300 ノードに増やしても十分なパフォーマンスが期待できる（O(n log n) の性質から推定 60ms 程度）。2回目以降のリクエストはキャッシュヒットするため、レイアウト計算自体が発生しない。

一方、ユーザーのタグ総数が 200 を超えるケースが増えており、より多くのタグを可視化することで Tag Verse の探索体験を向上させたいという要望があった。

## DECISION MAKING

FetchTagCloud API のデフォルト limit を 200 から 300 に変更した。変更はフロントエンドからバックエンドまでの全レイヤーに一貫して適用した。

### 変更箇所

| # | ファイル | 変更内容 |
|---|---------|---------|
| 1 | `proto/alt/articles/v2/articles.proto` | コメント `default: 200` → `default: 300` |
| 2 | `alt-backend/app/connect/v2/articles/handler.go` | デフォルト `limit = 200` → `limit = 300` |
| 3 | `alt-backend/app/usecase/fetch_tag_cloud_usecase/usecase.go` | デフォルト `limit = 200` → `limit = 300` |
| 4 | `alt-frontend-sv/src/lib/connect/articles.ts` | デフォルト引数 `limit = 200` → `limit = 300`、JSDoc コメント更新 |
| 5 | `alt-frontend-sv/src/lib/components/desktop/tag-verse/TagVerseScreen.svelte` | `fetchTagCloud(transport, 200)` → `fetchTagCloud(transport, 300)` |
| 6 | `alt-backend/app/usecase/fetch_tag_cloud_usecase/usecase_test.go` | キャッシュテスト内の limit を 200 → 300 に統一 |

### proto 再生成が不要な理由

`articles.proto` の変更はコメントのみ（`default: 200` → `default: 300`）であり、メッセージフィールドの型・番号・名前に変更はない。proto のデフォルト値はコメントで慣習的に記載しているだけで、実際のデフォルト値のバリデーションはバックエンドの handler/usecase 層で行っている。したがって `buf generate` による再生成は不要。

### max limit を変更しない理由

max limit は handler 層で 500、usecase 層でも 500 のまま据え置いた。ADR-360 のベンチマークで 500 ノード = 336ms であり、デフォルト引き上げとは独立した判断として現状維持とした。

## RESULTS, EFFECTS

### PROS

- **探索体験の向上** — 表示タグ数が 1.5 倍になり、ユーザーがより多様なタグの関連性を視覚的に把握できる
- **パフォーマンスへの影響が軽微** — Barnes-Hut O(n log n) + キャッシュにより、300 ノードでも初回リクエストで推定 60ms 程度。2回目以降はキャッシュヒットで即応答
- **全レイヤー一貫** — proto コメント、handler、usecase、フロントエンドの全箇所を同時に更新し、デフォルト値の不整合を防止
- **テスト全パス** — Usecase テスト 6件 + キャッシュテスト 4件 + Octree テスト 5件 + レイアウトテスト 6件 = 全 21 件 PASS

### CONS, TRADEOFF

- **初回レスポンスの微増** — 200→300 ノードにより、キャッシュミス時のレイアウト計算時間が約 1.7 倍に増加する見込み。ただし絶対値としては 60ms 程度であり、ユーザー体感に影響しないレベル
- **キャッシュキーの変更** — デフォルト limit が 200→300 に変わるため、デプロイ後の最初のリクエストは必ずキャッシュミスとなる。TTL 30分のため影響は一時的
