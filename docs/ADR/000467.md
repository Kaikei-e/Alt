# ADR-000467: summary_state 空文字の既存 74,527 件をバックフィルで修復する

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-backend (Go)

## Context

[[000465]] と [[000466]] により、`summary_state` が非要約イベントで空文字に上書きされる問題と、`summary_completed` が無条件付与される問題のコード修正が完了し、deploy 済みとなった。deploy 後は新規 item に対して正しい `summary_state` が設定されるようになり、直近で 711 件の `ready` items が正常に生成されていることを確認した。

しかし、コード修正前に壊れた既存データは自動修復されない。[[000465]] の SQL CASE 文は `ELSE knowledge_home_items.summary_state END` で既存値を保持する設計であり、既に空文字になっている行はそのまま空文字が維持される。

2026-03-19 時点の調査結果:

| summary_state | 件数 | excerpt あり |
|---|---|---|
| `''` (空文字) | 74,497 | 46,674 |
| `pending` | 41 | 0 |
| `ready` | 711 | 711 |

全 75,249 件中 99% が空文字状態。UI 上では `summary_state` が `ready` でないため正常パスに入らず、`WhySurfacedBadge` が「Summarized」を表示しつつ `SummaryStateChip` や状態依存の表示が不整合になっていた。

`knowledge_home_items` はイミュータブルデータモデルにおける disposable projection（再構築可能な read model）であり、event log (`knowledge_events`) を変更せずに read model を直接修復することは設計上許容される。

## Decision

**`summary_state` が空文字の既存 items を、`summary_excerpt` の有無に基づいて `ready` または `pending` にバックフィルする。**

以下の SQL を直接実行する。

```sql
-- excerpt がある items は 'ready' に修復
UPDATE knowledge_home_items
SET summary_state = 'ready'
WHERE summary_state = ''
  AND summary_excerpt <> ''
  AND summary_excerpt IS NOT NULL;

-- excerpt がない items は 'pending' に修復
UPDATE knowledge_home_items
SET summary_state = 'pending'
WHERE summary_state = ''
  AND (summary_excerpt = '' OR summary_excerpt IS NULL);
```

### 検討した代替案

1. **full reproject で全 item を event log から再構築する**
   [[000465]] の SQL 契約により reproject でも正しい state が再現される。しかし 75,000 件規模の reproject は projection checkpoint のリセットと再処理時間を伴い、その間 Knowledge Home の表示が一時的に劣化する。空文字→正しい state への変換は `summary_excerpt` の有無だけで一意に決定できるため、2 本の UPDATE で十分であり、reproject のオーバーヘッドは不要と判断した。
2. **マイグレーションファイルとして管理する**
   Atlas マイグレーションで管理すると、ステージング・本番の全環境で自動適用される。しかし今回は一度きりのデータ修復であり、スキーマ変更を伴わないため、マイグレーションファイルに含めるとマイグレーション履歴にノイズが増える。ADR に SQL と実行結果を記録し、手動実行とした。

## Consequences

### Pros

- 46,674 件の excerpt ありの items が `ready` に修復され、`KnowledgeCard.svelte` の正常パス（`summaryState === "ready" && summaryExcerpt`）で表示される
- 27,823 件の excerpt なしの items が `pending` に修復され、skeleton アニメーションが正しく表示される
- `SummaryStateChip` と `WhySurfacedBadge` の表示が整合する
- event log (`knowledge_events`) を一切変更しないため、append-first invariant を維持
- 今後 reproject を実行しても [[000465]] の SQL 契約により正しい state が再現されるため、バックフィルの効果が reproject で失われることはない

### Cons / Tradeoffs

- 手動 SQL 実行のため、環境ごとに個別実行が必要（ステージング・本番）
- `summary_excerpt` の有無だけで state を判定しているため、excerpt があるが実際には不完全な要約の item も `ready` になる（ただし、excerpt 非空で要約が不完全なケースは [[000466]] の修正により今後は発生しない）
- 既に誤付与されている `summary_completed`（`why_json` 内）は今回のバックフィルでは修正しない。これは reproject で再構築するか、影響が軽微であれば許容する

### 実行結果

```
UPDATE 46674  -- excerpt あり → ready
UPDATE 27823  -- excerpt なし → pending
```

修復後の状態:

| summary_state | 件数 | excerpt あり |
|---|---|---|
| `pending` | 27,864 | 0 |
| `ready` | 47,385 | 47,385 |
| `''` (空文字) | **0** | — |
