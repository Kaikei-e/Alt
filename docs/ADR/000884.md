# ADR-000884: articlePrefetcher に per-host inflight gate と 429 cooldown を追加して visual-preview の自動 prefetch 経由 429 を抑止する

## Status

Accepted (implementation deferred to follow-up PR)

## Date

2026-05-02

## Affected Services

- alt-frontend-sv (TypeScript / Svelte 5) — `src/lib/utils/articlePrefetcher.ts`, `src/lib/utils/articlePrefetcher.test.ts`, `src/lib/components/mobile/feeds/swipe/SwipeFeedScreen.svelte`, `src/lib/components/mobile/feeds/swipe/VisualPreviewCard.svelte`

## Context

[[000883]] (Connect-Timeout-Ms 5s + VisualPreviewCard summary フォールバック) を 2026-05-02 06:49 UTC に push、06:52 UTC に alt-frontend-sv コンテナへ反映した。デプロイ後にユーザーから *「モバイルから feeds が今 500 になった」* との再報告があり、コンテナログを精査した結果、ADR-883 の対象であった 500 / 502 は解消されているが、**新たに HTTP 429 が顕在化**していることを確定した。

### 計測 (2026-05-02 デプロイ後 90 分)

| 指標 | 値 | 出典 |
|---|---|---|
| nginx で `/api/v2/.../FetchArticleContent` の 5xx | **0 件** | nginx access |
| BFF response の 5xx | **0 件** | BFF proxy_response |
| BFF feeds unary `elapsed_ms` p99 | **85ms** | BFF proxy_response |
| backend `feed_read_perf.total_ms` p99 | **59ms** | alt-backend |
| **nginx で `/api/v2/.../FetchArticleContent` の 429** | **4 件** (08:14:42-47Z, 7 秒間) | nginx access |
| BFF `bff.proxy_response status:429` | 4 件、`elapsed_ms:44 / 75 / 59 / 58` | BFF |
| 対応する backend エラー | `rate limit wait failed for "https://dev.to/...": rate: Wait(n=1) would exceed context deadline` + `RPC cancelled  procedure="/alt.articles.v2.ArticleService/FetchArticleContent"  reason="context canceled"` | alt-backend |
| 全 429 の発生元 UA | `Mozilla/5.0 (iPhone; CPU iPhone OS 18_7 ...) Mobile/15E148 Safari/604.1` (single client) | nginx access |
| 全 429 の Referer | `/feeds/swipe/visual-preview` | nginx access |

### 因果連鎖

1. ユーザーが iPhone から `/feeds/swipe/visual-preview` を表示。
2. `SwipeFeedScreen.svelte:39` で `const prefetchAhead = $derived(isVisualPreview ? 10 : 2);`。visual-preview モードでは **10 件先まで自動プリフェッチ**する設計。
3. `articlePrefetcher.triggerPrefetch(feeds, activeIndex, prefetchAhead)` が `feeds[activeIndex+1..activeIndex+10]` の各 URL に対し `getFeedContentOnTheFlyClient(url)` を呼び、結果を card 表示前に温める。
4. dev.to のような単一ホストに連続記事が並ぶと、同一ホストへ 5-10 並列の HTTP fetch が backend で発生する。
5. backend の **host rate limiter (5 rps / host 想定)** が `Wait(n=1) would exceed context deadline` で待ち行列を deadline 切れさせ、429 を返す。
6. BFF はそのまま透過 (`status:429 elapsed_ms:44ms`)。
7. iOS Safari は HTTP 429 を画面上で「サーバエラー」相当の generic message として表示することがあり、ユーザーは **「500」と報告**するが実際は 429。

`articlePrefetcher.test.ts:103-148` には既に *"skips prefetch when another fetch for the same host is in-flight"* という同ホスト抑止のユニットテストがある。にもかかわらずプロダクションで 429 が観測されたということは、**(a) 抑止ロジックの粒度が記事 URL 単位で同一 URL のみを抑止しているため、同ホスト別 URL が並列に通る**、または **(b) `triggerPrefetch` が `Promise.all` 的に並列起動した直後の競合時間帯で抑止が機能しない**、いずれかの可能性が高い。

### Web リサーチによる裏付け (2026-05-02)

- **Connect-RPC `timeoutMs`**: ADR-883 で導入した `{ timeoutMs: 5000 }` は公式仕様どおり (https://connectrpc.com/docs/web/cancellation-and-timeouts/)。CORS preflight の許可ヘッダに `Connect-Timeout-Ms` が必要 (https://connectrpc.com/docs/cors/) だが、デプロイ後に ConnectError 0 件のため preflight は通過している。
- **SvelteKit deferred Promise streaming**: `__data.json?x-sveltekit-invalidated=N` は load 関数の依存再評価仕様であり、ポーリングではない (sveltejs/kit#13628)。iPhone 観測の 17 連発は invalidate 経路の活発さが原因で、`/feeds/swipe/visual-preview` 自体は `invalidate(...)` を呼ばないため scope 外。iOS Safari portrait 既知問題 (sveltejs/kit#13480, #10315) は scroll-style ページに影響するもので、本ルートの card-stack には影響軽微。
- **EventSource 自動再接続**: BFCache `pageshow.persisted` 検出は MDN 公式パターンで、[[000881]] の `useStreamingFeedStats` で実装済。WebKit Bug 199854 (iOS で `pagehide` が発火しない場合がある) はサーバ側 5 分 stream cap で軽減済。追加対応不要。
- **2026 PWA graceful degradation**: dependency を critical / non-critical に分類し circuit breaker + fallback で degrade するのが標準 (https://sreschool.com/blog/graceful-degradation/)。RSS 閲覧における *full body fetch* は non-critical で、summary 表示で代替可能。ADR-883 のフォールバック markup はこの方針と一致しているが、**auto-fetch 経路には適用されていない**点が gap。

## Decision

以下を採用する。本 ADR は設計判断として accepted、実装は別 PR で行う。

### 1. `articlePrefetcher` に **per-host inflight gate** を追加する (P0)

URL 単位ではなく **host 単位**で同時 fetch を 1 件に直列化する。`articlePrefetcher.ts` 内部に `Map<host, Promise<void>>` を追加し、新規 `triggerPrefetch` が同じ host の既存 in-flight を await してから走る。fan-out 順序は preserve するが、対同一ホスト並列度は常に 1。

これにより visual-preview の `prefetchAhead=10` が dev.to に 10 連発しても、実際の HTTP は 1 件ずつ直列、backend の host rate limiter (5 rps) を尊重する。

### 2. **429 受信時のホスト単位 cooldown** を追加する (P0)

`getFeedContentOnTheFlyClient(url)` が ConnectError code `ResourceExhausted` (HTTP 429) を返した場合、その URL のホストを `Map<host, Date>` の cooldown レジストリに記録し、**30 秒間そのホストへの prefetch をスキップ**する。Phase 2 ではサーバが返す `Retry-After` ヘッダ (RFC 6585) を尊重するように改める。固定値の 30 秒は host rate limiter の typical reset time に合わせた保守的な初期値。

### 3. `VisualPreviewCard` の `onMount` auto-fetch 失敗パスに `contentError` セットを追加する (P1)

現状 `onMount` 内の `getFeedContentOnTheFlyClient(...).catch((err) => console.warn(...))` は silent。429 / 500 / timeout のいずれを踏んでも `fullContent = null` のままで、tap 展開せずに即「中身を出したい」体験では空白に見える。auto-fetch 失敗時にも `contentError = "Source content unavailable."` をセットし、ADR-883 の `data-testid="source-unavailable-notice"` + `feed.description` のフォールバック markup を auto / manual の両経路で共通化する。

### 4. `prefetchAhead` を visual-preview で 10 → **4** に縮小する (P1)

per-host inflight gate と 429 cooldown を入れた上で、なお同一ホストで連続記事が並ぶケースを考慮し prefetch 深度を **10 → 4** に下げる。10 は当初 LCP 体感最適化のための値だったが、429 risk と帯域コストに対しオーバープロビジョニング気味だった。4 は次の 1-2 swipe を gracefully cover する深さで、cellular 帯域への配慮も両立する。Phase 1 / Phase 2 の効果観測後に再評価する。

### 検討した代替案と却下理由

- **backend の host rate limiter 閾値を緩める** — 5 rps は外部サイトに対する礼儀ある budget であり、しかも今回は **同じ user 1 人のセッション内**で踏んでいるため、緩めても根本の暴力的 prefetch は変わらない。symptom suppression にしかならず却下。
- **prefetch を完全に廃止** — visual-preview の card swipe は次の card の本文と OGP image が即出ることを前提に UX が組まれており、廃止は LCP / TTI の体感劣化を招く。並列度の調整で十分。
- **429 を catch せず抑え込まない (= ユーザーに伝播)** — VisualPreviewCard が空白で残るユーザー体験は ADR-883 の方針と矛盾。critical / non-critical 分類で full content は non-critical のため、summary フォールバックが正解。
- **`prefetchAhead=10` を維持し backend の rate limit を per-user / per-IP に変更** — NAT 配下で複数ユーザーが同一 IP からアクセスする一般的な構成 (家庭 / オフィス) を考えると不公平。host 単位の budget が合理的。
- **CSR 側で同一 host 抑止ではなく BFF 側でバッファリング** — BFF はステートレス透過 proxy として保つ方針 ([[000881]] の applyConnectTimeout も同様)。FE 側で発生源を抑えるのが筋。

## Consequences

### Pros

- 自動 prefetch 経由の 429 が事実上 0 になる。backend ログから `rate limit wait failed for ...` のノイズも消える。
- ユーザーが「500」と認識する事象がなくなる (root cause: 429)。
- 外部ホスト (dev.to / zenn / theguardian / hackernoon) への prefetch budget を意図された 5 rps 以内に収め、ban / temporary block 圏に入るリスクを下げる。
- VisualPreviewCard の auto / manual フォールバック経路が統一され、card が空白で残る回帰がなくなる。
- per-host inflight gate のロジックは既存 `articlePrefetcher.test.ts:103-148` の "same-host prefetch limiting" テストの自然な拡張で、TDD で安全に進められる。

### Cons / Tradeoffs

- visual-preview の体感プリフェッチ深度が 10 → 4 に下がる。連続スワイプの 5 件目以降は cache miss する可能性があり、その瞬間に短時間の loading が見える。Phase 1 / Phase 2 の効果観測で許容できるか判定する。
- 30 秒 cooldown は固定値で、本来は backend の `Retry-After` ヘッダ (RFC 6585) を尊重すべき。Phase 2 の TODO として残る。
- per-host inflight gate の状態管理 (`Map<host, Promise<void>>`, `Map<host, Date>`) が `articlePrefetcher` に追加されるため、複雑度が上がる。テスト網羅で対処する。
- `prefetchAhead=4` は heuristic で、ユーザーのスワイプ速度や RSS フィードのホスト分布によっては再調整の余地がある。LCP / TTI の真の最適点は実測で詰める。
- 本 ADR は実装まで含めて確定しているが、デプロイは別 PR に分離するため、429 の解消までユーザーが体感するまでにはタイムラグがある。その間は ADR-883 の summary フォールバックが部分的に救済する。
