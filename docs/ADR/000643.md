# ADR-000643: Augur RAG パイプラインの品質基盤修復と hybrid streaming 導入で回答品質と perceived responsiveness を同時改善する

## Status

Accepted

## Date

2026-04-08

## Affected Services

- rag-orchestrator (Go) — hybrid streaming, CausalStrategy, BM25 修正, progress vocabulary, query expansion guard
- news-creator (Python) — plan-query を `/api/chat` + thinking mode に移行, contrastive few-shot
- search-indexer (Go) — BM25 検索の user_id フィルタ修正
- alt-frontend-sv (TypeScript / SvelteKit) — live draft streaming UI, 6-stage progress 表示

## Context

Augur（Ask Augur）は [[000630]] で LLM 駆動 query planner・cross-encoder RelevanceGate・日本語 BM25 修正を設計し、[[000631]][[000632]] で eval baseline と structured output 信頼性を改善した。しかし、実運用で「イランの石油危機はなにが原因でおきた？詳しく教えて。」と質問しても **184 文字の浅い回答**しか返らず、ADR で設計した改善が発動していなかった。

コンテナログの徹底調査と Web 調査（Gemma 4 thinking mode、CRAG、Agentic RAG 最新知見）の結果、**6 つの独立した根本原因**を特定した:

1. **Plan-query が `/api/generate` + `raw=True`（think 無効）で動作** — Gemma 4 の推論能力が発揮されず intent 分類精度が低い
2. **Plan-query プロンプトに causal_explanation の few-shot 例がない** — 「原因」クエリが temporal/general に誤分類される
3. **`IntentCausalExplanation` 戦略が DI container に未登録** — 正しく分類されても generalStrategy にフォールバック
4. **BM25 検索が `user_id = "rag-orchestrator-system"` でフィルタされている** — 該当 user_id の記事が 0 件のため全クエリで `bm25_hits: 0`
5. **Follow-up クエリ展開が日付文字列のみを返す** — 「もっと詳しく」で無関係記事がヒット
6. **detail/synthesis 経路の UI が blank spinner のまま 30-40 秒待機** — strictLongForm が ChatStream を使わず完了まで delta を送らない

加えて、UX 観点では detail/synthesis 経路で searching/generating の blank spinner が長時間続き、本文 delta が最後まで空になる問題があった（実ログで generation 開始まで約 36 秒）。

## Decision

**品質基盤の 5 件の修正と、strictLongForm を hybrid streaming に置換する 1 件の UX 改善を同時実施する。**

### 1. Plan-query を `/api/chat` + thinking mode に移行

news-creator の `plan_query_usecase.py` を `generate()` → `chat_generate()` に変更。Ollama `/api/chat` で thinking mode を有効化し、intent 分類の推論品質を向上。

Ollama の既知バグに対応:
- `think=false` + `format` = format 制約無視 (Ollama #15260)
- thinking + `format` = content 空 (Ollama #10929)

対策として `format` パラメータを省略し、フリーテキスト JSON + 手動パース (`_extract_json()`) を採用。`num_predict=2048` で thinking + content の両方に十分なトークンを確保。

`DistributingGateway` が `chat_generate` をオーバーライドしていなかったため、local gateway へのデリゲーションを追加。

rag-orchestrator 側の `QueryPlannerClient` タイムアウトを 3 秒 → 60 秒に延長（`QUERY_PLANNER_TIMEOUT` 環境変数で制御可能）。

### 2. Contrastive few-shot 例の追加

`PLAN_QUERY_PROMPT` に causal_explanation の contrastive few-shot 例を追加。特に temporal cue と causal cue が同居するケース（「最近の石油危機の原因は？」）で、causal keywords が temporal を override する推論過程を明示。

intent 選択ガイダンスも追加:
- 「原因」「なぜ」「要因」→ `causal_explanation`
- temporal + causal 同居 → `causal_explanation` が優先

### 3. CausalStrategy の DI 登録

ADR-607 で設計された focused subquery decomposition を `CausalStrategy` として実装。supply-side / geopolitical / economic の 3 軸で並列検索し、`AssessWithIntent` で最高品質の結果を採用、Good で early exit。

`container.go` に `WithStrategy(IntentCausalExplanation, causalStrategy)` を追加。

### 4. BM25 user_id フィルタ修正

**真因**: rag-orchestrator の `SearchBM25` が search-indexer に `user_id=rag-orchestrator-system` を送信 → search-indexer が Meilisearch に `filter: user_id = "rag-orchestrator-system"` を適用 → 該当 user_id の記事が 0 件 → 常に `bm25_hits: 0`。

`localizedAttributes` は既に実装・適用済みであり、Meilisearch 直接検索では 674 件ヒット。問題は user_id フィルタのみ。

修正: RAG 用 BM25 検索で `user_id` パラメータを省略。search-indexer 側は `user_id` なしのリクエストに対して `SearchArticlesUsecase`（フィルタなし検索）に分岐。

### 5. Follow-up クエリ展開の品質ガード

`filterExpandedQueries` に `isDateOnly` チェックを追加。`2026-04-07` のような日付のみの展開クエリを除外。

`buildPromptWithQueryPlanner` で plan-query の `search_queries` に `FilterSearchQueries()` を適用し、全展開クエリがフィルタされた場合は `resolved_query` にフォールバック。

### 6. Hybrid Streaming（strictLongForm → ChatStream + ParagraphFlusher）

`strictLongForm` パスを `streamHybridLongForm` に置換:

- `ParagraphFlusher` 純関数: 段落境界（`\n\n`）、日本語文末（`。！？`）、英語文末（`.!?`）で flush。最小 120 runes。1.5 秒タイムアウト + 80 runes で time-based flush
- ChatStream でリアルタイム token を内部蓄積 → インクリメンタル answer パーサーで answer フィールドを抽出 → ParagraphFlusher で段落単位の provisional delta を送出
- Corrective retry 維持: initial attempt の preview はそのまま送出、retry 時は `progress=refining` を送り新規 delta を停止、retry 完了後 `done.answer` で全文置換

Progress vocabulary を 6 stage に標準化:
- `planning` → `searching` → `reranking` → `drafting` → `validating` → `refining`

### 7. Frontend live draft 対応

`useAugurPane.svelte.ts` に `statusText`、`isProvisional` state を追加。`onThinking` コールバックを有効化して server-generated status text を表示。

`AskSheet.svelte` の spinner を 6 stage に対応した文言に拡張。`refining` stage では本文下に "Refining answer..." indicator を表示。

## Consequences

### Pros

- **BM25 復活**: `bm25_hits: 50` (修正前: 0)。RRF fusion で vector + BM25 の hybrid 検索が正常動作
- **Intent 分類改善**: thinking mode + contrastive few-shot で `causal_explanation` を正しく分類（ログで確認済み: `"reasoning": "これは明確に因果関係を問う質問であり、意図は causal_explanation"`)
- **CausalStrategy 発動**: `causal_strategy_execute` がログに出現。focused subquery decomposition で因果分析最適化が走る
- **Perceived responsiveness 向上**: blank spinner 30-40 秒 → 数秒で段落単位の live draft 表示開始
- **Progress visibility**: 6 stage の進捗表示でユーザーが待機中の状態を把握可能
- **Public proto contract 不変**: StreamChatResponse の shape は維持。progress 文字列のみ標準化
- **done.answer が source of truth**: provisional preview は done で authoritative に置換

### Cons / Tradeoffs

- **Plan-query レイテンシ増加**: thinking mode で 25-30 秒。`QUERY_PLANNER_TIMEOUT=60` で対応。TTFT は増加するが品質を優先
- **Ollama version 依存**: thinking + structured output の挙動は Ollama version に依存。`format` パラメータ省略 + 手動 JSON パースで両方のバグを回避したが、Ollama の修正後は `format` 再有効化を検討
- **Provisional delta の visual consistency**: done.answer で全文置換するため、provisional → final で text が微妙に変わる可能性。UX としては許容範囲（preview は draft として扱う）
- **CausalStrategy の subquery が固定文字列**: 供給/地政学/経済は domain-specific。将来的に LLM-based decomposition への移行が必要
- **search-indexer の user_id フィルタ省略**: RAG パスで全記事検索になったため、マルチテナント環境では別途対応が必要（現在は単一ユーザー前提）
