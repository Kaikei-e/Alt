# ADR-000825: recap-subworker の joblib artefact 欠落を Pydantic validator と named-volume-with-init-container で 2 層 fail-closed にする

## Status

Accepted

## Date

2026-04-22

## Affected Services

- **recap-subworker (Python)** — `Settings` に joblib backend 用 `@model_validator(mode="after")` を追加し、`classifier.py` の model_path guard を `is_file()` に tighten。
- **compose/base.yaml** — named volume `recap_subworker_artifacts` を追加。
- **compose/recap.yaml** — `recap-subworker-artifacts-init` busybox init service を追加し、recap-subworker を 10 本 file-scoped bind から named-volume 1 本に組み替え、init の `service_completed_successfully` を boot gate にする。

## Context

3days / 1day Recap の自動バッチが **2026-04-14 02:00 JST から 2026-04-22 (検知日) まで 8 日連続で失敗していた**。DB 集計は `window_days=3 AND status='failed'` が 23 件、`window_days=1 AND status='failed'` が 11 件。最後の成功は `recap_jobs.job_id=137a4a66-...` (2026-04-13 17:56:36 UTC)。症状は recap-worker 側で `classification returned 0 results for N articles (service may be unavailable)` で、[[000811]] / [[PM-2026-035]] および [[PM-2026-033]] と完全同文のため外見上は「PM-035 の後始末がまだ残っている」ようにも見えたが、`alt-recap-subworker-1` は `Up 39 hours (healthy)` を維持し `/health` は 200 を返し続け、compose 外形監視・Prometheus・smoke のどれも異常を検出していなかった。詳細は [[PM-2026-036]] を参照。

真因は **docker-compose の file-scoped bind mount の missing-source silent footgun**。`compose/recap.yaml` の従来実装は以下のように 10 本の個別ファイル bind を列挙していた:

```yaml
volumes:
  - ../recap-subworker/data/golden_classification.json:/app/data/golden_classification.json:ro
  - ../recap-subworker/data/genre_classifier.joblib:/app/data/genre_classifier.joblib:ro
  - ../recap-subworker/data/genre_classifier_ja.joblib:/app/data/genre_classifier_ja.joblib:ro
  - ../recap-subworker/data/genre_classifier_en.joblib:/app/data/genre_classifier_en.joblib:ro
  - ../recap-subworker/data/tfidf_vectorizer.joblib:/app/data/tfidf_vectorizer.joblib:ro
  - ../recap-subworker/data/tfidf_vectorizer_ja.joblib:/app/data/tfidf_vectorizer_ja.joblib:ro
  - ../recap-subworker/data/tfidf_vectorizer_en.joblib:/app/data/tfidf_vectorizer_en.joblib:ro
  - ../recap-subworker/data/genre_thresholds.json:/app/data/genre_thresholds.json:ro
  - ../recap-subworker/data/genre_thresholds_ja.json:/app/data/genre_thresholds_ja.json:ro
  - ../recap-subworker/data/genre_thresholds_en.json:/app/data/genre_thresholds_en.json:ro
```

docker の file-scoped bind (`host-path:container-path`) は host-path が **存在しないファイルである場合、warning を出さずに空ディレクトリを自動生成して mount target に置く**。`.gitignore:45` で `recap-subworker/data/` が OSS 側で除外されているため、公開リポジトリの `git clone` ベースのチェックアウトには artefact が届かない。artefact は別機構 (alt-deploy の Private sync) で deploy runner に配布される設計だったが、2026-04-14 頃に host 側 `<alt-deploy>/alt/recap-subworker/data/*.joblib` が消失した。結果 10 本全ての bind が空ディレクトリで mount され、コンテナ内 `/app/data/genre_classifier.joblib` は「サイズ 0 のディレクトリ」として実体化した。

`recap_subworker/services/classifier.py:75` の既存ガード:

```python
if not self.model_path.exists():
    raise FileNotFoundError(f"Model not found at {self.model_path}")
self.model = joblib.load(self.model_path)
```

は `Path.exists()` が「空ディレクトリでも True を返す」ので通過し、`joblib.load()` 内部の `open(path, "rb")` が `IsADirectoryError: [Errno 21] Is a directory: 'data/genre_classifier.joblib'` を送出。worker pool 初期化が 300s timeout、5 チャンク全てが空配列、recap-worker は `classification returned 0 results` で fail-fast。

[[000811]] で 2026-04-20 に `classification_backend` デフォルトを `learning_machine` → `joblib` に revert して `_validate_learning_machine_artifacts` fail-closed validator を追加したが、**joblib 経路には対称な validator が無く**、joblib default に revert した瞬間に validator 全経路が短絡する「片肺 fail-closed」状態だった。PM-035 の修正者は「joblib artefacts は別配布機構で deploy runner に届いているので validator は learning_machine 側だけで十分」と暗黙に前提していたが、2026-04-14 以降はその前提そのものが成立していなかった。

**エラー文言が PM-033 / PM-035 と 3 PM 連続で完全同文**である点、**file-scoped bind の missing-source silent 挙動** がチーム内で「silent failure の footgun」として共有されていなかった点、PM-035 の AI #6 (配布経路対応表) / PM-033 の AI #6 (実経路 E2E smoke) / AI #11 (DB 失敗集計の Prometheus expose) / AI #12 (error 文言区分) が 3 PM を跨いで未着手のまま継承されてきた点が重なり、8 日間 silent に潜伏した。

## Decision

以下を採用する:

### 1. `Settings._validate_joblib_artifacts` による起動時 fail-closed

`recap-subworker/recap_subworker/infra/config.py` の `Settings` に `_validate_learning_machine_artifacts` と対称な `@model_validator(mode="after")` を新設する:

```python
@model_validator(mode="after")
def _validate_joblib_artifacts(self) -> Settings:
    if self.classification_backend != "joblib":
        return self

    from pathlib import Path

    joblib_paths = (
        "genre_classifier_model_path",
        "genre_classifier_model_path_ja",
        "genre_classifier_model_path_en",
        "tfidf_vectorizer_path_ja",
        "tfidf_vectorizer_path_en",
        "genre_thresholds_path_ja",
        "genre_thresholds_path_en",
    )
    misconfigured: list[str] = []
    for field in joblib_paths:
        value: str | None = getattr(self, field, None)
        if not value:
            continue
        path = Path(value)
        if path.is_dir():
            misconfigured.append(f"{field}={value} is a directory")
    if misconfigured:
        raise ValueError(
            "joblib artefact path resolves to a directory instead of a "
            "file. This is almost always a docker-compose file-scoped "
            "bind mount whose host source file is missing, which docker "
            "silently replaces with an empty directory. Offenders: "
            + "; ".join(misconfigured)
        )
    return self
```

- **`is_dir()` で明示検出**。存在しない path は fall-through (`Path.is_dir()` = False) させる。unit 環境でも `Settings()` が組めるよう、「欠落」は fail せず「空ディレクトリ型」だけを fail-close する。missing file の runtime は classifier.py のガードが拾う (§2)。
- **違反箇所を列挙**。`misconfigured.append(f"{field}={value} is a directory")` で全 offenders を 1 メッセージに含める。deploy runner の log 行だけで原因ファイルが確定できる。
- **[[000811]] と対称な 2 validator 構成**。`classification_backend == "joblib"` と `"learning_machine"` の各選択肢で fail-closed が担保される。backend の全選択肢で fail-closed 対称性を持たせることを本 ADR の invariant とする。

### 2. `classifier.py` の guard を `is_file()` に tighten

`recap-subworker/recap_subworker/services/classifier.py:75` の既存ガードを:

```python
if not self.model_path.is_file():
    raise FileNotFoundError(f"Model not found at {self.model_path}")
```

に変更する。`Path.exists()` は空ディレクトリで True を返すため model ロード用 guard には不適切。`is_file()` なら missing / directory / symlink-broken の全てが False になり、`joblib.load()` が `IsADirectoryError` を出す前に `FileNotFoundError` で短絡できる。`_validate_joblib_artifacts` が bypass された場合 (例: 直接 `GenreClassifierService(...)` を instantiate する場合) の多層防御。

### 3. `compose` の 10 本 file-scoped bind を named volume + init container に置換

docker file-scoped bind の missing-source silent 挙動を構造的に迂回する。`compose/base.yaml` top-level volumes に:

```yaml
recap_subworker_artifacts:
```

を追加し、`compose/recap.yaml` を以下の形に組み替える:

```yaml
  recap-subworker-artifacts-init:
    image: busybox:1.37
    container_name: recap-subworker-artifacts-init
    volumes:
      - ../recap-subworker/data:/src:ro          # directory-scoped: missing host dir = container fails to create
      - recap_subworker_artifacts:/dst           # named volume
    command:
      - sh
      - -c
      - |
        set -eu
        test -d /src || { echo "FATAL: /src is not mounted"; exit 1; }
        if ! (test -s /src/genre_classifier_ja.joblib || test -s /src/genre_classifier_en.joblib); then
          echo "FATAL: no usable joblib model on the host"
          echo "  checked: /src/genre_classifier_ja.joblib and /src/genre_classifier_en.joblib"
          echo "  both are missing or zero-sized. This is almost always"
          echo "  a lost alt-deploy artefact sync — verify the host path"
          echo "  that docker-compose binds to /src."
          exit 1
        fi
        cp -r /src/. /dst/
        echo "recap-subworker artefacts populated:"
        ls -la /dst/
    networks:
      - alt-network
    restart: "no"

  recap-subworker:
    # ...
    volumes:
      - recap_subworker_artifacts:/app/data:ro
      - ../recap-subworker/recap_subworker/learning_machine/artifacts:/app/recap_subworker/learning_machine/artifacts:ro
    depends_on:
      recap-worker:
        condition: service_healthy
      recap-subworker-artifacts-init:
        condition: service_completed_successfully
```

- **`/src:ro` が directory-scoped bind**。host 側 `recap-subworker/data/` ディレクトリ自体が存在しない場合、docker はコンテナ作成時点でエラーを出す (file-scoped の silent 空ディレクトリ化は起こらない)。
- **`test -s` で事前 assert**。ja / en どちらかの genre_classifier が 1 byte 以上の通常ファイルでなければ `exit 1`。missing file / empty file / directory のどれでも catch する。
- **`cp -r /src/. /dst/` で named volume に複製**。recap-subworker は `recap_subworker_artifacts:/app/data:ro` で named volume を read-only mount。named volume は一度 populate されれば docker daemon が管理するため、再起動時は init container が再実行されて常に最新 host state を反映する。
- **`service_completed_successfully` を boot gate に**。init container が exit 1 で落ちれば recap-subworker は起動自体しない。healthy を偽装する silent failure 経路が塞がる。
- **`learning_machine/artifacts` は directory-scoped bind のまま温存**。このパスは `recap-subworker/recap_subworker/learning_machine/artifacts/` (OSS 側に commit されている dir) を bind しており、missing-source 問題は元々起きない。

### 検討した代替案と却下理由

- **「validator だけ追加して compose は触らない」** — 修正差分が最小で済むが、docker file-scoped bind の silent 挙動自体は残る。将来 `data/` 配下に新 artefact を追加した場合、file-scoped bind を拡張する度に同じ footgun に触れる。「ツール挙動を runbook 知識に頼らず構造で抑える」方針と相反。却下。
- **baked Docker image で artefact を `COPY`** — `Dockerfile.recap-subworker` の build context に `data/` 配下を取り込む案。image が単一 unit として配布でき最も確実だが、(1) `.gitignore` で除外されている artefact を build context に入れるには build-arg 経由の別配布を CI で整備する必要があり、alt-deploy (Private) 側の変更が必須、(2) artefact を更新する度に image 再 build + tag push + pull が必要となり、配布負荷が大きい。今回の投入コストが過大なので採用せず。follow-up ADR で検討余地はある。
- **`LearningMachineStudentClassifier` を eager init に変更** — lazy init 自体が silent failure の温床だが、本件は joblib backend の話で learning_machine と直交。joblib 側は既に first-touch で lazy load しており、validator による起動時 fail-closed で lazy の弱点を補う形にした。本 ADR では classification_worker の init タイミングには触れない。
- **`Path.is_file()` ガードのみで compose は触らない** — classifier.py の guard を `is_file()` にすれば `IsADirectoryError` は出なくなるが、`FileNotFoundError` に変わるだけで worker pool init timeout の症状は変わらない (5 チャンク空配列 → recap-worker fail)。ユーザー視点の症状同文性が解消しない。やはり boot 時点で止める init container が必要。
- **Dockerfile 側で `ENTRYPOINT` に pre-flight script を噛ませる** — busybox init container と同等の効果だが、compose 層に分離した方が (1) 再利用性が高く、(2) recap-subworker image の責務が膨張せず、(3) init の log が別 container 名で分離される。compose 層採用。

## Consequences

### Pros

- **backend 両選択肢で fail-closed 対称性が成立**。[[000811]] の `_validate_learning_machine_artifacts` と本 ADR の `_validate_joblib_artifacts` が pair で存在し、backend の選択肢追加時に「新 backend にも fail-closed validator を付ける」レビュー項目の検出可能性が上がる。`Settings()` が起動時に空ディレクトリ型 artefact を 1 秒以内に検出する。
- **docker file-scoped bind の silent footgun が構造的に塞がれる**。host `recap-subworker/data/` がディレクトリごと消失すれば init container のコンテナ作成で起動失敗、個別ファイルが欠落・ゼロサイズなら init 内部の `test -s` で exit 1、artefact が正常なら named volume に populate されて recap-subworker が起動する。どのシナリオでも silent 成功にならない。
- **recap-subworker コンテナの責務がクリーンに**。artefact populate を init container に分離したため recap-subworker image 自体は artefact 配布機構を知らなくてよい。Clean Architecture の境界線と整合。
- **問題の局所化**。8 日間 silent 潜伏 → 起動 1 秒以内で fail。TTD が 3 桁改善する。`docker compose logs recap-subworker-artifacts-init` 1 行で原因 host path が確定する。
- **既存テスト 302 + 新規 9 = 311 件が GREEN**。zero regression。
- **`.gitignore` 方針を崩さない**。artefact は従来どおり OSS 側で除外し、deploy runner にのみ配布する境界を維持。配布経路そのものを ADR で扱う必要がない。

### Cons / Tradeoffs

- **compose の依存グラフが深くなる**。`recap-subworker` が `recap-subworker-artifacts-init` に depends_on するため compose up の起動順制約が 1 段増える。`service_completed_successfully` の挙動は docker-compose v2.17+ で安定しており問題は想定していないが、compose engine を downgrade する場合は要確認。
- **init container の busybox image pull コストが発生**。`busybox:1.37` は 4 MiB 程度で影響は微小だが、offline 環境では事前に pull されていないと起動しない。
- **named volume の lifecycle を把握する必要が増える**。`docker compose down -v` で volume が消える挙動、`docker volume rm alt_recap_subworker_artifacts` の手動復旧手段、host 側 artefact 更新時は `--force-recreate recap-subworker-artifacts-init recap-subworker` で再 populate する必要などが運用手順に追加される。`docs/runbooks/recap-subworker-artifacts.md` (未作成) への転写 AI を PM-036 で立てる。
- **host 側 artefact の source of truth が deploy runner に依然として依存**。本 ADR はあくまで **「artefact が欠落していることを起動時に検知する」** 装置であり、artefact そのものを自動復旧する機能ではない。`<alt-deploy>/alt/recap-subworker/data/` の復旧は人手作業 ([[PM-2026-036]] AI #4)。
- **PM-2026-033 / PM-2026-035 から継承している AI #6 / AI #11 / AI #12 は依然未着手**。本 ADR は root cause に対する直接対策のみ。横断的検知 (classify-runs smoke / recap_failed_tasks の Prometheus expose / error 文言区分) は別の作業 stream として残る。
- **`Path.exists()` を使う別箇所が残存**。classifier.py 内に `self.tfidf_path.exists()` / `self.svd_path.exists()` / `self.scaler_path.exists()` / `self.thresholds_path.exists()` が残る (本 ADR ではスコープ外)。これらは optional artefact で missing 時に warn fallback するため現状は即時の問題を起こさないが、`is_file()` 一括 tighten は PM-036 AI #11 で追跡する。
- **file-scoped bind を他 service が依然使用している可能性**。本 ADR は recap-subworker だけが対象。`compose/` 配下の他 service に同種の footgun が潜んでいないかの audit は PM-036 AI #10 として別途実施する。

## Addendum: 2026-04-22 — Named volume + init container を撤回、direct bind に移行

本 ADR の初期版 (commits `96a2edc81` + `6931f7c8a`) では named volume + busybox one-shot init container で `/app/data` を populate する設計を採った。実運用の rolling deploy で **init container が起動されない挙動** が確認され、named volume が空のまま recap-subworker が起動し、PM-2026-036 と同じ症状を再発させた (2026-04-22 18:21 JST 手動キック)。

### 非互換の構造

Alt の運用は「1 サービスの失敗が全体を止めない」ため **サービス単位で targeted に compose 操作する rolling model** を採る。この操作は `docker compose up <service>` 相当で行われ、デフォルトで対象サービスの `depends_on` も起動するが、**一括失敗を避ける設計のために `--no-deps` 相当で dependency 起動を抑制している**。そのため `depends_on: service_completed_successfully` で init container を gate する compose パターンは、rolling 時には **init container がそもそも create されない**。

この制約は docker-compose + `--no-deps` の普遍的な挙動 ([docker/compose#9591](https://github.com/docker/compose/issues/9591) で同系の edge case が既知) であり、alt-deploy 固有ではない。rolling deploy で init container を確実に動かすには、init サービスを明示的に targeted set に加える必要があるが、それを運用層で毎回同期する負担を避けたい。

### 採用: direct bind mount

Docker Compose v2.24 以降は directory-scoped bind の source 不在時に container create を refuse する ([docker/compose#12735](https://github.com/docker/compose/issues/12735) で v2.35 / 2025-04 も継続確認)。**compose engine 自体が fail-closed を担う**ため、独立した init container を介さずに、直接 host-persistent path から `/app/data:ro` に bind mount するのが最小構成。

```yaml
recap-subworker:
  volumes:
    - ${RECAP_SUBWORKER_DATA_HOST_PATH:-/var/lib/alt-recap-subworker-data}:/app/data:ro
    - ../recap-subworker/recap_subworker/learning_machine/artifacts:/app/recap_subworker/learning_machine/artifacts:ro
  depends_on:
    recap-worker:
      condition: service_healthy
```

- `recap-subworker-artifacts-init`、`recap_subworker_artifacts` named volume、対応する `depends_on: service_completed_successfully` を全削除
- `/var/lib/alt-recap-subworker-data` は runner bootstrap で事前作成 ([[runner-setup]] §2.6)
- `.env` の `RECAP_SUBWORKER_DATA_HOST_PATH` で dev ワークステーションでは repo 相対パス (`../recap-subworker/data`) にフォールバック可能

### Fail-closed 多層防御は維持

- **Docker Compose v2.24+**: host source 不在 → container create refuse (compose layer)
- **Settings validator** (`_validate_joblib_artifacts`): path が directory → 起動時 `ValidationError` (app layer、Compose 挙動が将来戻った場合の defense-in-depth)
- **`classifier.py:75` の `is_file()` guard**: `joblib.load()` 前の最終 sanity (runtime layer)

3 層のうちどれか 1 層が抜けても他が catch する構造。init container 層が抜けただけで fail-closed 性は変わらない。

### 更新コミット
