# ADR-000712: Acolyte レポートにハードデリートを導入し active run 中は拒否する

## Status

Accepted

## Date

2026-04-13

## Affected Services

- acolyte-orchestrator (Python / Connect-RPC) — `DeleteReport` ハンドラ、`delete_report` / `has_active_run` を Port / Postgres / Memory gateway に実装
- acolyte-migration-atlas — `reports` 連鎖の全 FK に `ON DELETE CASCADE` を付与するマイグレーション
- alt-frontend-sv (TypeScript / Svelte 5) — 詳細画面 (desktop + mobile) の inline confirm 削除 UI、一覧画面の `DELETED` 通知バナー
- proto/ — `DeleteReport` RPC と request/response メッセージ

## Context

Acolyte のレポートは一度作ったら消す手段がなかった。試行錯誤で作った下書きや生成に失敗したレポートが一覧に残り続け、肥大化する。ユーザーからは整理したいという要望が出ていたが、消す RPC もボタンも存在しないため対処できなかった。

削除を入れるにあたり、最初に決めなければならないのは **immutable data model との整合性** だった。Alt の Knowledge Home は append-first / event-sourced を採用しており、削除ではなく tombstone イベントで表現する。一方 Acolyte は `reports` が mutable entity (`current_version` が bump される / `latest_successful_run_id` が上書きされる) であり、immutability は `report_versions` / `report_section_versions` / `report_change_items` の snapshot 群にのみ適用される。つまり Acolyte は event-sourced ではなく "version-first state management" の設計で、report 全体のライフサイクルはオペレーショナルに削除できる。`docs/ADR/` を grep しても Acolyte の soft delete を要求するものは存在せず、tombstone で対応する必要はない。

次に **FK とカスケード** の確認。`20260409000000_create_acolyte_tables.sql` と `20260410000000_create_report_briefs.sql` を読むと、`report_versions` / `report_change_items` / `report_sections` / `report_section_versions` / `report_runs` / `report_jobs` / `report_briefs` のすべての FK が `ON DELETE` 指定を欠いている。素の `DELETE FROM reports WHERE report_id = $1` は FK 違反で必ず失敗する。アプリケーション層で順序付き多段 DELETE をトランザクションに書くか、マイグレーションで `ON DELETE CASCADE` を付与するかの選択になる。

最後に **active run の扱い**。`report_runs` は `run_status IN ('pending', 'running', 'succeeded', 'failed', 'cancelled')` で、LangGraph の pipeline が走っている最中にレコードが削除されると checkpoint が参照先を失い、クラッシュ原因になる。`thread_id = f"acolyte-run:{run_id}"` の checkpoint が DB 上に orphan として残るのも健全ではない。冪等に扱うにはポリシーをどちらかに倒す必要がある。

## Decision

以下を採用する。

1. **ハードデリートを採用する** — Acolyte は event-sourced ではないので soft delete / tombstone は過剰。`DELETE FROM reports WHERE report_id = $1` を使い、関連する child row も同時に消す。version history が恒久的に失われることはトレードオフとして受容する (将来のバックアップ・監査要件が出てきたら再検討)。
2. **Atlas migration で `ON DELETE CASCADE` を付与する** — アプリケーション層での順序付き多段 DELETE は 7 テーブル分のコードで冗長かつ将来テーブル追加時に忘れるリスクが高い。FK 自体に CASCADE を付けることで単一文で済み、テーブル追加時の規約 (新テーブルは `ON DELETE CASCADE` で宣言する) も自然に確立できる。
3. **active run (`pending` / `running`) 中の削除は拒否する** — cancel-then-delete は checkpoint のクリーンアップ責任を招き複雑化する。`FAILED_PRECONDITION` を返し、UI 側で "Generation in progress — wait or cancel before deleting." と案内する。これは既存の "A report generation is already in progress" (`StartReportRun` の同時実行防止) と同じパターンで一貫する。
4. **削除 UI は詳細画面のみ・inline confirm とする** — 一覧画面には破壊的アクションを置かず、Alt-Paper のミニマル志向を保つ。`Delete` クリック → ヘッダーボタン列を `Confirm delete` (terracotta) + `Cancel` (ghost underline) に差し替える inline 変形。モーダル / dialog を使わず、typography だけで確認フローを表現する。

## Implementation Overview

### 1. Atlas migration と hash 再計算

`acolyte-migration-atlas/migrations/20260413000000_cascade_delete_reports.sql` を追加。

```sql
ALTER TABLE report_versions
    DROP CONSTRAINT report_versions_report_id_fkey,
    ADD CONSTRAINT report_versions_report_id_fkey
        FOREIGN KEY (report_id) REFERENCES reports(report_id) ON DELETE CASCADE;

-- report_change_items / report_sections / report_section_versions /
-- report_runs / report_jobs / report_briefs も同形で書き換え
```

7 つの FK を `DROP CONSTRAINT ... ADD CONSTRAINT ... ON DELETE CASCADE` で置換。`cd acolyte-migration-atlas && atlas migrate hash` で `atlas.sum` を更新。`docker compose up --build --force-recreate acolyte-db-migrator` で適用すると 7 ステートメントが ~32ms で完了し、既存データはそのまま保持された。

### 2. proto 拡張

`proto/alt/acolyte/v1/acolyte.proto` に RPC とメッセージを追加。

```proto
rpc DeleteReport(DeleteReportRequest) returns (DeleteReportResponse);

message DeleteReportRequest { string report_id = 1; }
message DeleteReportResponse {}
```

`buf generate --template buf.gen.acolyte.yaml` と `buf generate --template buf.gen.yaml --path alt/acolyte` で Python stub と TypeScript stub を同時再生成。

### 3. Port / Gateway / Handler

`acolyte/port/report_repository.py` に 2 メソッドを追加。

```python
async def has_active_run(self, report_id: UUID) -> bool: ...
async def delete_report(self, report_id: UUID) -> None: ...
```

`PostgresReportGateway.has_active_run` は `SELECT EXISTS(SELECT 1 FROM report_runs WHERE report_id = %s AND run_status IN ('pending', 'running'))`。Postgres 側の index `idx_report_runs_active ON report_runs(run_status) WHERE run_status IN ('pending', 'running')` がそのまま効く。`delete_report` は `DELETE FROM reports WHERE report_id = %s` の 1 行のみ (CASCADE が残りを引き受ける)。

`MemoryReportGateway` は in-memory dict を掃除する軽量実装にした (`has_active_run` は常に False)。テスト用途。

`acolyte/handler/connect_service.py` の `delete_report`:

```python
async def delete_report(self, request, ctx):
    try:
        rid = UUID(request.report_id)
    except ValueError as e:
        raise ConnectError(Code.INVALID_ARGUMENT, f"Invalid report_id: {e}") from e
    report = await self._repo.get_report(rid)
    if report is None:
        raise ConnectError(Code.NOT_FOUND, f"Report {rid} not found")
    if await self._repo.has_active_run(rid):
        raise ConnectError(
            Code.FAILED_PRECONDITION,
            "A report generation is in progress; wait or cancel before deleting.",
        )
    await self._repo.delete_report(rid)
    logger.info("Report deleted", report_id=str(rid))
    return acolyte_pb2.DeleteReportResponse()
```

service token 検証は付けていない。他の RPC (CreateReport / GetReport 等) も `_verify_token` を呼んでいないため一貫させた (BFF / nginx 層で認証を受け持つ)。

### 4. TDD

`acolyte-orchestrator/tests/unit/test_delete_report.py` を新設。`FakeReportRepo` に `has_active_run` / `delete_report` を実装し、3 ケースを RED → GREEN で通した。

- `test_delete_report_happy_path` — 正常系、`DeleteReportResponse` を返し `repo.reports` から消えること
- `test_delete_report_refuses_when_active_run_exists` — `FAILED_PRECONDITION` を raise し、DB 上のレコードは残ること
- `test_delete_report_returns_not_found` — 存在しない UUID で `NOT_FOUND`

`SERVICE_SECRET=test-secret uv run pytest tests/unit/` 全 504 件 passed。

### 5. frontend — inline confirm と一覧バナー

`alt-frontend-sv/src/lib/connect/acolyte.ts` に型ガードと RPC client を追加。

```ts
export function isFailedPrecondition(err: unknown): err is AcolyteRpcError {
    return err instanceof AcolyteRpcError && err.code === "failed_precondition";
}

export async function deleteReport(reportId: string): Promise<Record<string, never>> {
    return rpc("DeleteReport", { reportId });
}
```

詳細画面 (`/acolyte/reports/[id]/+page.svelte`) のヘッダー右端に `Delete` ボタンを追加。hairline outline + muted gray の抑えたスタイルで、hover で terracotta (既存のエラー色) に遷移する。`isGenerating` true のときは `disabled` + tooltip で理由を明示。

クリックで `confirmingDelete = true` にセットすると、ヘッダーのボタン列が `Delete this report permanently?` の prompt + `Confirm delete` (terracotta 塗り) + `Cancel` (underline ghost) に入れ替わる inline 確認になる。モーダルではなく同じ行の置換で表現しているため、ユーザーのフォーカスが画面外に飛ばない。`Confirm delete` で `deleteReport()` を呼び、成功時は `goto(/acolyte?deleted=<encoded title>)` で一覧へ戻る。`FAILED_PRECONDITION` は `isFailedPrecondition(e)` で narrowing し、日本語化せず `Generation in progress — wait for it to finish or cancel before deleting.` のままエラーバナーに表示。

mobile (`MobileAcolyteDetail.svelte`) は `onRequestDelete` / `onConfirmDelete` / `onCancelDelete` を props で受け取り、ヘッダーアクション下に縦積みの inline confirm を表示。min-height 44px のタッチターゲットは維持。

一覧画面 (`/acolyte/+page.svelte`) は URL の `?deleted=<title>` を onMount で読み、`DELETED` kicker + hairline rule + タイトルの serif display body + `Dismiss` の構成でバナー表示。`history.replaceState` で query param を即座に消し、リロードで再表示されないようにする。

### 6. コンテナ再ビルドと smoke

```bash
docker compose -f compose/compose.yaml -p alt build acolyte-orchestrator
docker compose -f compose/compose.yaml -p alt up -d --force-recreate --no-deps acolyte-orchestrator
```

smoke 結果:

| ケース | 期待 | 実測 |
|---|---|---|
| Happy path (create → delete → GetReport) | 200 OK → 404 | 一致 |
| Not found (未登録 UUID) | 404 `not_found` | 一致 |
| Invalid UUID (文字列) | 400 `invalid_argument` | 一致 |
| Cascade (brief 付き report を delete) | reports = 0 / briefs = 0 | 一致 |
| active run 拒否 | 412 `failed_precondition` | unit test で確認 |

## Consequences

### Pros

- **レポート一覧が整理可能になる** — 試行錯誤の下書きを消せるため、Acolyte の使い始めの摩擦が減る
- **FK CASCADE で child row が置き去りにならない** — versions / change_items / sections / section_versions / runs / jobs / briefs が同一トランザクションで一掃される
- **active run は保護される** — pipeline 実行中の checkpoint / DB レコード整合性が壊れない。cancel-first のワークフローが明示的に要求される
- **UI が Alt-Paper 方針と整合** — 破壊的アクション用の別枠モーダルを作らず、typography と色だけで確認フローを成立させている。一覧からは見えない / hover で terracotta / inline confirm、の段階的抑制
- **将来テーブル追加の規約が定着** — `reports` や `report_runs` を参照する新テーブルは `ON DELETE CASCADE` を付与する運用ルールが自然に確立

### Cons / Tradeoffs

- **LangGraph checkpoint の orphan** — 削除された run の `thread_id = acolyte-run:{run_id}` 系の checkpoint rows は CASCADE の範囲外で残る。現状は inert (再入されない) なので実害はないが、長期運用で肥大化する可能性があり、定期クリーンアップジョブが別途必要になる
- **復元不可能** — undo / trash 機能は入れていない。ユーザーが誤って消したら戻せない。inline confirm を 1 段挟んでいるが、複数削除や破壊的操作の繰り返しには弱い
- **一括削除は未対応** — 大量整理したい場合は 1 件ずつの操作が必要 (out of scope)
- **認証は上位レイヤ依存** — ハンドラは service token を検証しないため、BFF / nginx 側が適切にゲートしていることが前提。スタック構成を変える際に注意が必要
