# ADR-000738: pre-processorからnews-creatorへの要約リクエスト重複を抑止する三層防御を導入

## Status

Accepted

## Date

2026-04-15

## Affected Services

- pre-processor (Go) — HTTPクライアント・キューワーカー・ガード層
- news-creator (Python) — 影響を受けるが本ADRでは変更なし（下流の観察対象）

## Context

news-creator のログ上で、同一 `article_id` に対する12チャンクの Map-Reduce 要約が並行実行されるパターンが観測された。詳細調査の結果、次の連鎖が判明している。

1. pre-processor の HTTP 共有クライアントで `ResponseHeaderTimeout = 20s` が全クライアント共通に設定されていた。news-creator の Map-Reduce 要約は12チャンク × 数秒〜十数秒かかるため、ヘッダ応答が到着する前にクライアント側で強制切断される。
2. pre-processor は切断を通常のリトライ対象として扱い、すぐに同一 `article_id` で再リクエストを送る（バックオフなし）。
3. news-creator 側はリクエストをキューに投入するが、前のリクエストがまだ実行中のため新しいリクエストの全チャンクが `QueueFullError` で失敗する。`summarize_usecase` は best-effort でチャンク失敗を吸収するため、12/12 全失敗ではじめて RuntimeError を返し 502 Bad Gateway となる。
4. pre-processor は 3回リトライ後に `dead_letter` 行きにするが、**元のリクエスト自体は news-creator 側で最終的に成功し summary が永続化される**ため、「DB に summary は存在するのに `summarize_job_queue` 上は dead_letter」という不整合が発生していた。

この結果、ジョブテーブルが false-negative な失敗で汚染され、再処理のトリガを失うケースがあった。現行の `queue_guard` は `summary_exists` と `recent_success` のみチェックし、**in-flight 状態のジョブは見ていなかった**ため、重複リクエスト自体を未然に防ぐ経路がなかった。

## Decision

pre-processor 側に三層の防御を導入する。news-creator 側の semaphore/dedup は触らない（責任境界を pre-processor に寄せる）。

### 層1: 共有HTTPクライアントのタイムアウト分離

`utils/http_client_manager.go` の `createOptimizedClient` をクライアント別にパラメータ化し、summary 用クライアントのみ `ResponseHeaderTimeout=0`（= 無効化）とする。`summarizer_api.go` が `context.WithTimeout(ctx, NEWS_CREATOR_TIMEOUT)`（default 600秒）で全体上限を担保しているため、ヘッダ応答の制限は context 側に委譲する。`defaultClient` / `feedClient` は従来通り 20 秒を維持する。

### 層2: in-flight 重複ガード

`queue_guard.go` の `ShouldQueueSummarizeJob` に **in-flight チェック** を追加する。`SummarizeJobRepository` に新設した `HasInFlightJob(ctx, articleID, since)` を呼び、`status IN ('pending','running')` かつ `COALESCE(started_at, created_at) >= since` を満たすジョブが存在する場合は `(false, "in_flight", nil)` を返す。時間窓は10分（`RecoverStuckJobs` と同じ窓）とし、クラッシュしたワーカーが永久にガードしないようにする。

### 層3: dead_letter 直前の upstream recheck

`summarize_queue_worker.go` の `processJob` で、`nextRetryCount >= MaxRetries` に到達した時点で `summaryRepo.Exists(ctx, articleID)` を再チェックする。true であれば「upstream が並行リクエストで既に完了している」とみなし、`SummarizeJobStatusCompleted` に振り替えて dead_letter を回避する。

### 補助: エラー分類とバックオフの拡張

- 新規 sentinel error `domain.ErrUpstreamBusy` を追加し、`summarizer_api.go` で HTTP 502/503/504、`context.DeadlineExceeded`、`net/http: timeout awaiting response headers` をこの error に畳み込む。
- `summarize_queue_worker.go` は `ErrServiceOverloaded` だけでなく `ErrUpstreamBusy` でも overload backoff をトリガする。
- `handler/job_handler.go` の `BackoffOnErrors` に `ErrUpstreamBusy` を追加し、ticker 側の指数バックオフ（30s→5min）を共有する。

## Consequences

### Pros

- Map-Reduce 要約がヘッダタイムアウトで早期切断される事象が解消され、retry storm の根本原因が消える。
- `in_flight` ガードにより、仮にクライアントが重複リクエストを送っても enqueue 段階で `skipped: in_flight` として DB 側で拒否できる。
- dead_letter 直前の recheck により、upstream 成功かつ client タイムアウトのレースで生じる false-negative な dead_letter が減る。
- エラー分類が明示的になり、ログ検索で「upstream が busy」「サービス自体が overload」を区別できる。
- すべて pre-processor 側の変更に閉じており、news-creator のインターフェース契約（HTTP body schema）は変わらないため、Pact contract test の再生成は不要。

### Cons / Tradeoffs

- `summaryClient.Transport.ResponseHeaderTimeout=0` により、slowloris 的な slow-header 攻撃面がわずかに拡大する（受容可能：caller が必ず `context.WithTimeout` を設定しているため実質的な上限は担保される）。この設計前提を失うと再発するため、`hardening_test.go` に「summary client は ResponseHeaderTimeout=0 を許容する」例外を明記した。
- in-flight 窓の10分は `RecoverStuckJobs` のスタック判定と一致させているが、両者が将来ずれると stale なガードが発生しうる。定数を共有するヘルパへの集約は未実施。
- `HasInFlightJob` は pending/running の EXISTS クエリ1本追加となり、enqueue のホットパスにレイテンシが乗る（インデックス `(article_id, status)` があれば実行時間はマイクロ秒オーダー）。
- dead_letter recheck は TOCTOU レースを完全には除去しない。upstream がジョブの `UpdateJobStatus` と summary 永続化の間で落ちると、再度 recheck で取りこぼす可能性は残る。ただしその場合は stale recovery の対象になる。

## 実装詳細

### 変更ファイル

| ファイル | 変更 |
|---|---|
| `utils/http_client_manager.go` | `createOptimizedClient(timeout, responseHeaderTimeout)` にシグネチャ変更。summary クライアントのみ header timeout=0 |
| `utils/http_client_manager_hardening_test.go` | summary クライアントの `ResponseHeaderTimeout=0` を許容する例外を明示 |
| `domain/errors.go` | `ErrUpstreamBusy` sentinel を追加 |
| `driver/summarizer_api.go` | 502/503/504 + header/context timeout を `ErrUpstreamBusy` に分類（blocking/streaming 両パス） |
| `repository/interfaces.go` + `summarize_job_repository.go` | `HasInFlightJob(ctx, articleID, since)` 追加 |
| `service/queue_guard.go` | `ShouldQueueSummarizeJob` に in-flight チェック追加。窓は10分 |
| `service/summarize_queue_worker.go` | dead_letter 直前の summary existence recheck、overload backoff 条件に `ErrUpstreamBusy` を追加 |
| `handler/job_handler.go` | `BackoffOnErrors` に `ErrUpstreamBusy` 追加 |
| `test/mocks/repository_mocks.go` | `go generate` で `HasInFlightJob` モックを再生成 |

### テスト（TDD: RED→GREEN→REFACTOR）

全てのロジック変更はテスト先行で実装した。

- `repository/summarize_job_repository_test.go`: `HasInFlightJob` の nil DB / 空 articleID ガード
- `service/queue_guard_test.go`: in-flight ジョブ存在時に `(false, "in_flight", nil)` を返す case
- `driver/summarizer_api_test.go`: 502/503/504 が `ErrUpstreamBusy` にマップされる table-driven test
- `service/summarize_queue_worker_test.go`: ErrUpstreamBusy が overload backoff をトリガする case、dead_letter 直前に Exists=true なら Completed に振り替える case
- `utils/http_client_manager_test.go`: クライアント別の ResponseHeaderTimeout 値

- `go test ./...` 全パス
- `go vet ./...` クリーン
