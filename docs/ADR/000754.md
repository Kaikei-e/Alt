# ADR-000754: Desktop 記事詳細画面に editorial rail を追加して wide-viewport の composition を回復する

## Status

Accepted

## Date

2026-04-17

## Affected Services

- alt-frontend-sv (SvelteKit 2.x / Svelte 5 Runes / TypeScript / TailwindCSS v4) — `/articles/[id]/+page.svelte` の desktop レイアウトを二段組に刷新

## Context

[[000715]] で `/articles/[id]` をモバイルファースト Alt-Paper editorial 構成に揃えた。その時点の決定は「sticky action bar / editorial masthead / 65ch 本文」の 3 ブロック単一カラムであり、desktop viewport を明示的には設計していなかった。結果として 1440–1920px クラスのモニタで以下の視覚破綻が確認された。

- `.article-page` は `max-width: 820px; margin: 0 auto` で main area に対して中央寄せされる。
- 内側の `:global(.article-content)` は `max-width: 65ch`（Alt-Paper の editorial な可読性担保、[[000715]] Decision 点 8）。ただし `margin-inline: auto` を持たないため、820px 親の **左端に貼り付いた** 状態で描画される。
- さらに親の `<main>` は 240px 幅 sidebar の右側で `flex-1` を占有する。1440px の viewport では実効コンテンツ幅 ≈ 1152px あり、そのうち 65ch (~520px) しか本文に使われず、**右半分が空白のまま**残る。

ユーザからの体感は「記事本文が左寄せになって読みづらい。幅をフルで使っていない」。一方 web の一次資料を踏まえると、`max-width: 100%` で本文を viewport 幅いっぱいに引き伸ばすのは **可読性が明確に劣化** するため採用できない:

- WCAG 2.2 AAA SC 1.4.8 (Visual Presentation): blocks of text は 80 characters (40 CJK) 以下。
- Baymard Institute, *Readability: The Optimal Line Length*: 50–75 CPL、推奨は 66 CPL。CSS 実装は `max-width: 70ch` 程度。
- Tailwind CSS Typography プラグインのデフォルト `max-w-prose` も 65ch。

つまり **左寄せの原因は 65ch が悪いのではなく、65ch の本文が 820px コンテナの中で中央寄せされず、かつ desktop の残余幅を構成的に使っていない** ことにある。

## Decision

[[000715]] の mobile 3 ブロック構成と 65ch editorial measure はそのまま保存し、`@media (min-width: 1024px)` で **editorial rail** を progressive enhancement として重ねる。`RenderFeedDetails.svelte` には一切手を入れず、変更は `src/routes/(app)/articles/[id]/+page.svelte` 1 ファイルに閉じる。

具体方針:

1. **`MediaQuery('(min-width: 1024px)')` を reactive に読み取る**（`svelte/reactivity` の `MediaQuery`、SSR フォールバック `false`）。この `hasRail` derived 値でテンプレートを分岐する。
2. **desktop (≥1024px) は CSS Grid** に切り替え、`grid-template-columns: minmax(0, 72ch) 20rem` + `column-gap: 2rem` の 2 カラム。
   - 左: `.article-body`（`margin-inline: auto; max-width: 72ch`）。内側の 65ch 本文は body cell の中で中央寄せされる。
   - 右: `<aside class="article-rail" data-testid="article-rail" aria-label="Article details">`。`position: sticky; top: calc(env(safe-area-inset-top, 0px) + 1rem); max-height: calc(100vh - 2rem); overflow-y: auto`。
   - `.article-page` の外側 cap を `min(100%, 96rem)` に緩め、ultrawide でも破綻しないよう二重に押さえる。
3. **rail の中身**は 4 セクション:
   - Source（`sourceHost`）
   - Reading time（`articleContent` から HTML を落として 240 wpm で `$derived`、min 1）
   - Actions（Fetch/Re-fetch, Summarize, Open original — mobile の top action-bar の右クラスタ相当）
   - AI Summary（`summarizer.summary` / `summarizer.summaryError`）
4. **Actions と AI Summary は `{#snippet}` + `{@render}` で共有**する。mobile (`!hasRail`) では top action-bar と masthead 下にそれぞれ配置、desktop (`hasRail`) では rail 内にのみレンダされる。`data-testid="fetch-button"` / `"summarize-button"` / `"ai-summary"` は一度しか DOM に現れないため Playwright strict mode と整合する。
5. **mobile (<1024px) は [[000715]] と byte-identical**。tablet (768–1023px) も rail を出さず単一カラム維持（20rem rail + 72ch body が入り切らない幅で無理に二段組にしない）。
6. **scoped `<style>` のみ**で実装し、`rounded-lg` / `rounded-xl` / `rounded-2xl` / `bg-white` / `text-[...]` 等の arbitrary Tailwind 値は一切追加しない（[[000715]] Decision 点 4 / `docs/DESIGN_LANGUAGE.md` の Sharp edges 原則）。rail のカラーは `var(--surface-2)` + `var(--surface-border)` + 3px top rule in `var(--alt-primary)` と既存 `.ai-summary` と同じ treatment を踏襲する。
7. **UI 文言は機能語のみ**（`Source` / `Reading time` / `Actions`）。`Byline` / `Dateline` / `PRESS RUN` 等の newspaper metaphor 語彙は `feedback_no_newspaper_metaphor_in_ui_text` に従い使用しない。

### 却下した代替案

- **本文を `max-width: 100%` に開放する**: WCAG 1.4.8 AAA / Baymard いずれも 80 CPL 超の long-form を明確に非推奨。**可読性と要件の両方に反する** ため不採用。
- **newspaper multi-column (`columns: 2 30ch`)**: Alt-Paper メタファーとの整合性は高いが、RSS 記事は画像 / コードブロック / 埋め込み iframe が混在し CSS multicol は span 制御・flow 制御が破綻しやすい。メタファーのための可読性劣化は `feedback_design_philosophy_not_copy` に反する。
- **measure だけ 80ch に広げて中央寄せ**: viewport の右半分が空白のままでユーザ体感を改善できず、desktop 向け composition も未解決。
- **`ArticleDetailPanel.svelte` (tag-trail slide-over) を standalone 詳細にも流用**: panel は 60% 幅 overlay であり URL ディープリンクされる standalone 詳細と責務が異なる。共通化は過剰抽象化。
- **`ResponsiveLayout.svelte` の `FULL_BLEED_PATHS` に `/articles/[id]` を追加**: 左右 `p-6` を外せるだけで、内側の 820px cap と 65ch 左寄せ問題は解決しない。別の問題に別の既存メカニズムを被せるだけで根治にならない。

### Tags / Related work は scope 外

rail に「tag chips」「same-source からの related articles」を並べる案も検討したが、`src/lib/api/client/articles.ts` に対応する client エンドポイントが存在せず BFF/backend 拡張が必要になる。今回の純粋な layout fix から切り離し、follow-up として別タスクに分離する。rail に空セクションやプレースホルダも置かない。

## Consequences

### Pros

- 1440–1920px desktop で **composition が viewport を埋める**。本文は 65ch のまま中央寄せされ、rail が右余白を意図的に使い切る。Alt-Paper の editorial 構成が desktop でも成立した。
- [[000715]] の mobile decision を **一切改変しない** progressive enhancement。既存 e2e (`detail.spec.ts` の 2 本) と compliance テストを回帰させない。
- 本文 measure の権威は `RenderFeedDetails.svelte:max-width: 65ch` 1 箇所に保ったまま、desktop では body cell の 72ch でさらに保険をかける二重防御。WCAG 1.4.8 AAA を満たし続ける。
- **`MediaQuery` (SSR fallback `false`) + `hasRail` 分岐**により、testid の重複レンダを避けつつ mobile-first SSR を維持。Playwright strict mode と噛み合う。
- Actions と AI Summary を `{#snippet}` で単一定義にしたため、将来 rail / action-bar どちらかの並びが変わっても **ロジックの重複更新が発生しない**。
- rail の `position: sticky` + `max-height: calc(100vh - 2rem)` + `overflow-y: auto` により、長い AI summary でも rail が viewport を飛び越えない。

### Cons / Tradeoffs

- **tablet (768–1023px) は今回のフェーズでは mobile と同じ単一カラム** のまま。body は 820px cap で中央寄せされるが、rail は出ない。1024px 境界でレイアウトが切り替わるため、MacBook Air 13" (1280px 論理) 付近で使用するユーザには「desktop 扱い」になる。境界値の選択は 20rem rail + 72ch body が同時に成立する最低幅に合わせた。
- **`MediaQuery` はクライアント専用**。SSR は `hasRail=false` で mobile 版を描画し、hydrate 後にクライアント側で rail が現れる。初回描画から 1 フレーム程度のレイアウトシフトが desktop で発生する可能性がある。現状は許容する（critical content は body 側にあり、rail は付随情報のため）。
- `data-testid="article-content-surface"` / `data-testid="article-rail"` / `data-testid="ai-summary"` など testid を自動テストの hook として公開し続けるため、将来の class 名変更では testid の互換性を維持する必要がある。
- rail を導入したことで desktop の垂直スクロール体験は **body カラムだけが縦に伸びる** 非対称スクロールになる。WCAG 上は問題ないが、rail の sticky 挙動が短い記事では「不動の右カラム」として目立つ可能性がある。
- 記事メタデータとして rail に表示可能なのは現時点で `Source` と `Reading time` のみ。`url` と `title` しか query parameter に乗っていない制約を踏襲したため、公開日・著者・タグは今回の rail には含まれない。follow-up で BFF に tag-by-article / related endpoints が増えた時点で rail を拡張する。
