# ADR-000856: 記録する Surface Planner v2 の投影メタデータと Semantic WhyKind の end-to-end 伝播

## Status

Accepted

## Date

2026-04-27

## Affected Services

- proto: `services/sovereign/v1/sovereign.proto` に `SurfacePlannerVersion` enum と `KnowledgeLoopEntry.surface_planner_version` / `surface_score_inputs` を追加し、knowledge-sovereign / alt-backend の internal stub を再生成
- knowledge-sovereign: `Projector.resolveBucketAndInputs` が resolver 種別に応じて v1 / v2 を返し、v2 では immutable `SurfaceScoreInputs` JSON を `knowledge_loop_entries` に保存。`sovereign_db` read/write と `knowledge_loop_entries_public` view を更新
- alt-backend: domain model / sovereign client / Connect v2 handler が `surface_planner_version` と `topic_affinity_why` / `tag_trending_why` / `unfinished_continue_why` を lossless に伝播
- alt-frontend-sv: `mapProtoEntry` が planner version と Semantic WhyKind を `KnowledgeLoopEntryData` へ保持し、UI 側で v2 placement を識別可能にする
- docs: `knowledge-loop-reproject` の SurfacePlannerVersion v2 履歴を本 ADR に同期

## Context

[[000853]] で Wave 4-C の完了条件として「`surface_planner_version` 列を SQL UPSERT で 2 として書く wiring」を明記し、[[000854]] で `EventLogSurfaceScoreResolver` を本番 projector 経路へ組み込んだ。しかし、その時点では resolver が v2 evidence を返しても、read model 境界では次の欠落が残っていた。

- `KnowledgeLoopEntry` proto に planner version / score inputs がなく、projection row に記録しても cross-service contract に載らない
- `knowledge_loop_entries_public` view が `surface_planner_version` / `surface_score_inputs` を公開しておらず、read path が `projected_at` を隠す設計を維持したまま新列を読めない
- `sovereign_db` read/write が planner metadata を scan / UPSERT せず、reproject 後に v2 evidence の監査材料が失われる
- 新 WhyKind (`topic_affinity_why` / `tag_trending_why` / `unfinished_continue_why`) が DB → sovereign proto → alt-backend domain → public proto → frontend mapper の途中で `UNSPECIFIED` または legacy kind に潰れる
- `fb.md` の残課題レビューでは、Surface Planner / Semantic Orient / Changed diff / `DEFER` UI 接続が未完成候補として挙がっていた。`DEFER` と Changed diff は既存実装とテストで契約通り動いていたため、本 ADR は実際に欠けていた v2 planner metadata と Semantic WhyKind の end-to-end 伝播を決定として固定する

この欠落を放置すると、UI は v2 placement された entry と v1 fixed mapping の entry を区別できない。さらに reproject 後に「なぜその bucket に入ったか」を planner input から監査できず、Surface Planner v2 が knowledge-state-based planning なのか legacy fallback なのかを運用上判定できない。

## Decision

Surface Planner v2 の投影メタデータを `knowledge_loop_entries` の read model に永続化し、Semantic WhyKind を DB から frontend まで lossless に伝播する。

具体的には次を採用する。

- `services/sovereign/v1.KnowledgeLoopEntry` に `optional SurfacePlannerVersion surface_planner_version = 23` と `bytes surface_score_inputs = 24` を追加する。これは sovereign-internal contract であり、`surface_score_inputs` は JSONB opaque diagnostics として扱う
- `NullSurfaceScoreResolver` 経路は `SURFACE_PLANNER_VERSION_V1`、それ以外の real resolver 経路は `SURFACE_PLANNER_VERSION_V2` として記録する。version 判定は projector 内の resolver seam で行い、frontend や BFF で推測しない
- v2 entry では `SurfaceScoreInputs` の immutable evidence を JSON 化して `surface_score_inputs` に保存する。JSON は `event_type` / `freshness_at` / topic overlap / tag overlap / augur link / version drift / open interaction を持ち、`time.Now()` ではなく event-time 由来の値だけを保存する
- `sovereign_db.UpsertKnowledgeLoopEntry` と read path は `surface_planner_version` / `surface_score_inputs` を明示的に write / scan する。`knowledge_loop_entries_public` view は `projected_at` を引き続き除外しながら新 2 列だけを追加公開する
- alt-backend domain は `SurfacePlannerVersion` と `SurfaceScoreInputs` を保持する。public Connect v2 response には planner version を出し、score inputs は現時点では sovereign-internal diagnostics として UI に直接公開しない
- `topic_affinity_why` / `tag_trending_why` / `unfinished_continue_why` を sovereign DB mapper、sovereign proto mapper、alt-backend domain mapper、public proto mapper、frontend mapper の全層に追加する
- `DEFER` dismiss と Changed diff redline は本 ADR で設計を変えない。既存 contract のまま、same-stage `trigger = defer` と immutable payload 由来 `change_summary` を regression tests で再確認する

検討した代替案と却下理由:

- **planner version を frontend で `why_kind` や `surface_bucket` から推測する**: 同じ bucket でも v1 fixed mapping と v2 evidence placement があり得る。推測は audit trail にならないため却下
- **`surface_planner_version` だけを保存し、`surface_score_inputs` を保存しない**: version は分かるが「どの immutable evidence が placement を動かしたか」を reproject 後に検証できない。運用監査と regression 調査に弱いため却下
- **public API に `surface_score_inputs` をそのまま出す**: UI が当面必要とするのは v1/v2 の識別だけで、score inputs は sovereign-internal diagnostics。payload shape を早期に public contract 化すると後続の planner改善を縛るため却下
- **DB read path で `knowledge_loop_entries` を直接読む**: `projected_at` を API に漏らさない既存方針 ([[000831]]) を破る。public view を更新する方針を維持する
- **Semantic WhyKind を legacy `source_why` に畳み込む**: Surface Planner v2 の semantic orientation が UI / telemetry で識別不能になる。WhyKind は意味論の contract なので lossless に伝播する

## Consequences

### Pros

- reproject 後の各 `KnowledgeLoopEntry` が v1 / v2 のどちらで placement されたかを明示的に持つ。SurfacePlannerVersion bump の運用確認が `SELECT DISTINCT surface_planner_version` でできる
- v2 placement の score inputs が immutable JSON として残るため、bucket placement の再現性と監査性が上がる
- Semantic WhyKind が UI data model まで落ちずに届くため、`topic_affinity_why` / `tag_trending_why` / `unfinished_continue_why` に応じた表示・計測・将来の UX 分岐が可能になる
- `knowledge_loop_entries_public` view 経由の read path を維持し、`projected_at` を外部 contract に漏らさない方針を保てる
- `DEFER` dismiss と Changed diff は既存 contract を変更せず regression tests の対象に留めたため、今回の差分は v2 metadata / semantic propagation に集中できた

### Cons / Tradeoffs

- internal sovereign proto の generated code 差分が大きい。実体は enum + 2 fields の additive 変更だが、review 時は proto 本体を優先して確認する必要がある
- `surface_score_inputs` JSON は diagnostics であり public contract ではない。UI が将来 score detail を直接描画したくなった場合は、別 ADR で public schema を定義する必要がある
- read path が public view の新列に依存するため、deploy 順序は migration first が必要。`00015_refresh_knowledge_loop_public_view_surface_planner.sql` を適用せずに新 binary を出すと SELECT が失敗する
- legacy rows は reproject されるまで v1 または default 扱いで残る。v2 cutover 完了には `knowledge-loop-reproject` runbook に沿った full reproject が必要
- upstream `RecapTopicSnapshotted` / `AugurConversationLinked` の流量が不足している環境では、v2 version が記録されても placement signal は fallback 寄りになる。metric と score inputs で starved state を区別する必要がある
