# ADR-000647: Augur のプロンプトアーキテクチャを Intent 別テンプレートに再設計し pgvector hybrid search と Eino フレームワーク基盤を導入する

## Status

Accepted

## Date

2026-04-08

## Affected Services

- rag-orchestrator (Go)

## Context

[[000607]] から [[000644]] まで、Augur の回答品質改善を継続的に実施してきた。BM25 修正、CausalStrategy 登録、hybrid streaming、plan-query thinking mode 化、nil retrieval 防御など個別の修正は効果を出しているが、以下の**構造的な問題**が残存し、指示追従性の根本改善に至っていなかった。

### 1. プロンプト層: モノリシックなシステムプロンプト

`XMLPromptBuilder` は 438 行のモノリシックなビルダーで、全 Intent のシステムプロンプトを 1 つの `strings.Builder` で組み立てていた。

- **10+ セクション**が 1 つのシステムメッセージに詰め込まれ、"Lost in the Middle" 効果（ICLR 2025）でモデルの注意が分散
- 「必ず日本語で回答すること」が **7 箇所**に重複（Role, Quality, Causal, Comparison, Temporal, DeepDive, Sandwich）
- Few-shot 例が汎用的（3 パターン）で、Intent ごとの期待構造を示せていない
- トークン予算を `estimatedTokens := 500` で固定していたが、実際のシステムプロンプトは 1,500+ トークン
- Gemma 4 の thinking mode が無効化されたまま（`getThinkParam` が `nil` を返す）

### 2. Retrieval 層: チャンクレベル hybrid search の不在

BM25 は Meilisearch 経由で**記事レベル**の検索結果を返し、dense search は pgvector HNSW で**チャンクレベル**の結果を返す。この粒度不一致を Go 側のアプリケーションロジック（`fuse_results.go`）で RRF fusion していたが、チャンクレベルの BM25 マッチングができていなかった。

### 3. Orchestration 層: 手作りのオーケストレーション

Strategy pattern、tool-calling loop、retry logic がすべて手作りで、観測性が低く拡張性に乏しい。特に `AgenticSynthesisStrategy` の tool-calling loop は Ollama の native tool_calls を使っているが、フレームワークレベルの ReAct 制御（middleware, callback, interrupt/resume）がない。

### Web 調査による知見

最新の一次情報を調査し、以下の知見を設計に反映した。

- **SEAL-RAG**（arXiv 2512.10787）: "Replace, don't expand" — 固定 retrieval budget で precision 重視が context dilution を防ぐ
- **Position-aware prompting**: 重要な指示はプロンプトの冒頭と末尾に配置（recency bias の活用）
- **cloudwego/eino**: Go 向け LLM フレームワーク。ChatModelAgent（ReAct loop）、compose.Graph、Graph-as-Tool、callbacks が production-ready
- **pgvector hybrid search**: CTE-based RRF で dense + sparse を DB 内で統合（Jonathan Katz, 2025）
- **RAG prompt engineering** (StackAI 2026): system/user 分離、N-shot per intent、CoT for grounding

## Decision

**Augur のプロンプトアーキテクチャを Intent 別テンプレートレジストリに再設計し、pgvector tsvector による DB 内 hybrid search と cloudwego/eino フレームワーク基盤を同時導入する。全変更を Feature Flag で制御し、即時ロールバック可能にする。**

具体的には以下を採用する。

### 1. Intent 別 PromptTemplate レジストリ（Phase 1）

モノリシックな `XMLPromptBuilder` を `TemplateRegistry` に分割。各 Intent が独自のテンプレートを持つ。

- **8 Intent テンプレート**: General, Causal, Synthesis, Comparison, Temporal, FactCheck, DeepDive, ArticleScoped
- **7 Sub-Intent テンプレート**: Critique, Opinion, Implication, Detail, RelatedArticles, Evidence, SummaryRefresh
- **Position-aware 構造**: preamble（冒頭: 役割 + 制約）→ タスク固有指示（中央）→ few-shot（末尾近く）→ sandwich（末尾: 3 ルール）
- **重複排除**: 「日本語で回答」は preamble と sandwich の 2 箇所のみ（7 → 2）
- **Feature Flag**: `RAG_PROMPT_VERSION=alpha-v2` で有効化、`alpha-v1` で旧ビルダーに即時ロールバック

### 2. トークン予算の正確化（Phase 1）

- `EstimateSystemPromptTokens()` がテンプレートから実測値を返す
- `ComputeContextBudget()` で SEAL-RAG パターンの固定予算方式を適用
- `answer_with_rag_usecase.go` の `estimatedTokens := 500` を置換

### 3. Gemma 4 thinking mode 有効化（Phase 1）

- `getThinkParam` が Gemma 4 に `true` を返すように変更
- Chain-of-thought でコンテキストの grounding 検証を改善
- streaming path は既に thinking token を処理可能

### 4. pgvector tsvector hybrid search（Phase 2）

- `rag_chunks` テーブルに `tsv tsvector GENERATED ALWAYS AS` カラムを追加
- GIN インデックスで全文検索を効率化
- `HybridSearchRepository` が CTE-based RRF を SQL 内で実行
- Feature Flag: `HYBRID_BM25_SOURCE=postgres` で有効化、`meilisearch` で従来通り

### 5. cloudwego/eino フレームワーク基盤（Phase 3-4）

- `ChatModelAdapter`: `domain.LLMClient` を実装する Eino Ollama ラッパー
- `ToolAdapter`: `domain.Tool` → Eino `tool.InvokableTool` ブリッジ
- `RetrievalGraph`: 5-stage retrieval pipeline の単一呼び出しラッパー
- Feature Flag: `LLM_BACKEND=eino` で有効化（段階的移行用）

### 6. 評価基盤の拡充（Phase 0）

- `ExpectedStructure []string`: 回答に期待する見出し/キーワード
- `InstructionAdherenceRate`: 構造指示に従っている割合
- `MeanPromptTokens`: プロンプトサイズの計測
- `PromptTokenCount`: 個別ケースのプロンプトトークン数

### 採用しなかった案

| 案 | 内容 | 判断 |
|---|---|---|
| LangGraph (Python) への全面移行 | rag-orchestrator を Python に書き直す | Go の型安全性・パフォーマンスを失い、既存インフラとの乖離が大きい |
| プロンプト微調整のみ | 既存ビルダー内で文言を調整する | 構造的問題（重複、トークン予算、position bias）は文言では解決しない |
| 専用ベクトル DB（Pinecone, Weaviate）導入 | pgvector から移行 | 単一マシン前提で外部依存を増やすメリットが薄い。pgvector は既に稼働中 |
| Eino の DeepAgents 即時採用 | 高度な自律エージェントを中心に据える | SummarizationMiddleware が開発中。まず ChatModelAgent + compose で始める方が堅い |

## Consequences

### Pros

- **プロンプトサイズ 60% 削減**: Causal テンプレートで ~37 行 → ~15 行。トークン浪費が減り、context chunk に割ける予算が増える
- **Position-aware 配置**: 重要指示が冒頭と末尾に集中し、"Lost in the Middle" 効果を緩和
- **Intent 別 few-shot**: 各テンプレートが期待する回答構造を具体例で示すため、モデルの構造追従性が向上
- **正確なトークン予算**: `500` 固定から実測値に移行し、context overflow を防止
- **Chain-of-thought**: Gemma 4 の thinking mode で grounding 検証が改善
- **チャンクレベル hybrid search**: BM25 が記事レベルからチャンクレベルに精密化、RRF が SQL 内完結
- **Eino 基盤**: 将来の ChatModelAgent、Graph-as-Tool、callbacks への段階的移行が可能
- **全変更が Feature Flag 制御**: `RAG_PROMPT_VERSION`, `HYBRID_BM25_SOURCE`, `LLM_BACKEND` で即時ロールバック

### Cons / Tradeoffs

- **テンプレート管理コスト**: 8 Intent + 7 Sub-Intent = 15 テンプレートのメンテナンスが必要。ただし各テンプレートは小さく独立しているため、モノリシックビルダーより保守しやすい
- **Eino 依存追加**: `go.mod` に cloudwego/eino 関連の依存が増加（+26 パッケージ）。Feature Flag で無効化可能
- **tsvector 生成コスト**: `GENERATED ALWAYS AS STORED` で INSERT/UPDATE 時に tsvector 再計算が走る。現在のチャンク数では問題ないが、大規模化時に要モニタリング
- **Thinking mode のレイテンシ増**: Gemma 4 thinking で推論時間が増加する可能性。streaming path で体感は緩和されるが、TTFT は延びる
- **pgvector hybrid search の recall 検証が未了**: マイグレーション適用後に eval suite での A/B 比較が必要

### 有効化手順

```bash
# Phase 1: プロンプトテンプレート（即時、ロールバック可能）

# Phase 2: pgvector hybrid search（マイグレーション適用後）

# ロールバック
RAG_PROMPT_VERSION=alpha-v1
HYBRID_BM25_SOURCE=meilisearch
```
