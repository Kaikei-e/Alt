# ADR-000567: Ask Augur チャットストリームを優先セマフォ経由にルーティングする

## Status

Accepted

## Date

2026-03-25

## Affected Services

- news-creator (Python / FastAPI)
- rag-orchestrator (Go)

## Context

Ask Augur（RAG ベースの対話型回答機能）の応答が全てタイムアウトする障害が発生した。

### 根本原因: GPU スロット飢餓

LLM 推論サーバー（Ollama）は並列処理スロット数が限られている。rag-orchestrator の Ask Augur チャットストリームは、この Ollama に**直接 HTTP 接続**していた。一方、news-creator はバッチ要約ジョブを HybridPrioritySemaphore 経由で Ollama に送信しており、HIGH / LOW の優先制御が効いていた。

問題は、Ask Augur がセマフォを**バイパス**していたことにある：

- バッチ要約（LOW PRIORITY）が Ollama の並列スロットを長時間占有（1 リクエストあたり最大数分）
- Ask Augur のチャットストリームは Ollama 内部の FIFO キューに入るが、スロットが空くまで永続的に待機
- タイムアウト（デフォルト 300 秒）まで応答が一切返らない
- ユーザーのリトライが複数のチャットリクエストを同時にキュー待ちさせ、状況をさらに悪化

ログ分析により、Ollama の `/api/chat` 完了件数が **0 件**であることが確認された。同期間に `/api/generate`（要約）は 26 件完了しており、GPU リソースがバッチ処理に完全に独占されていた。

### ADR-386 の修正が効かなかった理由

[[000386]] は Query Expansion に `priority="high"` を導入し、news-creator の HybridPrioritySemaphore 経由で優先処理を実現した。しかし Ask Augur のチャットストリームは news-creator を経由せず Ollama に直接接続する設計（ADR-497 の "local-first" アプローチ）だったため、セマフォによる優先制御の対象外だった。

## Decision

**Ask Augur のチャットストリームを news-creator の `/api/chat` プロキシエンドポイント経由にルーティングし、HybridPrioritySemaphore の HIGH PRIORITY で GPU スロットを確保する。**

### 実装

#### 1. news-creator に `/api/chat` ストリーミングプロキシを追加

Clean Architecture に従い、3 層で実装する：

- **Handler** (`chat_handler.py`): Ollama の `/api/chat` リクエスト形式をそのまま受け付け、`StreamingResponse` で NDJSON を返却
- **Gateway** (`ollama_gateway.py`): `chat_stream()` メソッドを追加。`HIGH PRIORITY` でセマフォを取得し、ドライバーを呼び出す。`finally` ブロックでセマフォを解放
- **Driver** (`ollama_stream_driver.py`): `chat_stream()` メソッドを追加。Ollama の `/api/chat` エンドポイントに対して NDJSON ストリーミングを行う

プロキシは**透過的**に動作する。リクエスト・レスポンスの形式は Ollama ネイティブのまま変更しない。追加するのはセマフォによる優先制御のみ。

#### 2. rag-orchestrator の接続先を変更

`compose/rag.yaml` の `AUGUR_EXTERNAL` 環境変数を Ollama 直接接続から news-creator 経由に変更。rag-orchestrator のコード変更は不要。

### 検討した代替案

| 案 | 評価 | 却下理由 |
|---|---|---|
| `OLLAMA_NUM_PARALLEL` 引き上げ | リソース不足で context_length 制約 | VRAM 上限に依存、根本解決にならない |
| バッチ要約に `max_tokens` 制限 | 品質低下リスク | 要約精度への影響が不明 |
| タイムアウト短縮 + リトライ | 対症療法 | GPU 飢餓自体は解消しない |
| Ollama インスタンス分離 | インフラコスト増 | GPU リソースの物理的分離が必要 |

## Consequences

### Pros

- Ask Augur が HIGH PRIORITY でバッチ要約をプリエンプトするため、GPU 飢餓によるタイムアウトが解消
- rag-orchestrator のコード変更なし。環境変数の変更のみで切り替え可能
- news-creator の既存セマフォインフラ（エイジング、プリエンプション、保証帯域幅）がそのまま活用される
- 全 LLM リクエストが単一のセマフォを通過するため、GPU 利用状況の可観測性が統一される

### Cons / Tradeoffs

- Ask Augur のリクエストが news-creator を経由するため、ネットワークホップが 1 つ増える。ただしプロキシは透過的であり、追加レイテンシは無視できる水準
- news-creator が単一障害点になるリスクが微増。ただし既に Query Expansion と Reranking が news-creator を経由しており、実質的なリスク増は小さい
- `AUGUR_EXTERNAL` を `.env` でオーバーライドして外部 GPU サーバーにルーティングしている場合、プロキシではなく直接接続になる。この場合セマフォは適用されないが、外部サーバーは独立した GPU を持つため問題にならない
