# ADR-000915: グローバル検索の cold-start 1.2 秒を観測性 → warmup → LRU/singleflight → cutoff → crop の app-layer 多段で消す

## Status

Accepted

## Date

2026-05-24

## Affected Services

- search-indexer (Go) — otelconnect interceptor、driver layer の LRU cache (1024 entry / 5min TTL) + singleflight、起動時 warmup goroutine、`searchCutoffMs=1500`、`attributesToRetrieve` 最小化 + `attributesToCrop`、Meilisearch `processingTimeMs` の span attribute 化
- alt-backend (Go) — `global_search_usecase` の 3 セクションを個別 span 化し、`sectionTimeout = 10s` 一括値を `Articles 5s / Recaps 5s / Tags 1s` に分割
- news-creator-backend (Ollama) — compose env `OLLAMA_KEEP_ALIVE=24h` を追加 (qwen3-embedding を resident 維持)

## Context

`/alt.search.v2.GlobalSearchService/SearchEverything` (Reference Desk) と内部 caller (rag-orchestrator / butterfly-facade) の `services.search.v2.SearchService/SearchArticles` が遅い、という主訴。ClickHouse `rask_logs.otel_traces` から 2026-05-24 時点の実測:

- `search-indexer` `GET /v1/search` の p50 = 127 ms、max = 1254 ms (~10x の cold/warm 差)
- `alt-backend` `SearchEverything` span の 1 件サンプル = 1242 ms (query=`UAE`)
- Connect-RPC 側の span は **search-indexer から 1 つも otel_traces に到達していない** (HTTP REST `/v1/search` だけ trace 取得)
- Meilisearch `articles` index: 121,686 docs / 12 GB DB / **全件 qwen3-embedding 済** (`numberOfEmbeddings == numberOfDocuments`)
- Meilisearch settings: `MEILI_HYBRID_EMBEDDER=qwen3`、`MEILI_HYBRID_SEMANTIC_RATIO=0.7` がデフォルト ON
- Meilisearch task 履歴: 過去 1,220 件の embedding task が `news-creator-backend:11435` DNS resolve fail で各 400-527 秒消費 (2026-04-23 で復旧、再発リスクは残る)

主犯は **hybrid search が全クエリで無条件 ON** であり、`ollama` の qwen3-embedding-0.6b が他モデルと取り合って evict されるたびに ~1100 ms の cold-start を払うこと。observability も Connect-RPC レイヤで切れているため、何が遅いかの分解ができていない。

ユーザ方針 (確認済):

1. **Hybrid 品質は触らない** — semantic_ratio=0.7 のまま、pre-warm + cache で cold-start を消す
2. **観測性 → tuning の順** — 計測根拠を取り戻してから手を入れる
3. **アプリケーションレイヤー優先**、index 追加 (`articles(id) INCLUDE (feed_id, title, url)` 等) は最終手段
4. スコープは `/SearchEverything` と search-indexer `SearchArticles`。Archive Desk `/feeds/search` は対象外

## Decision

「単一の銀の弾」ではなく **観測性 (Phase A 3 PR) → tuning (Phase B 6 PR) を全部入れる多段 app-layer 改修** を採用する。各層の責務を変えず、driver 内の sideband (span attribute) と LRU cache、bootstrap goroutine、Meilisearch index settings 1 個、usecase の timeout 分割で構成する。

### Phase A — Observability (前提条件)

- **A1**: `search-indexer/connect/v2/server.go` の interceptor stack に `connectrpc.com/otelconnect` を `otel → rateLimit` 順で挿入。失敗時は Warn log + rate-limit のみで継続。これで `SearchArticles` / `SearchRecaps` の Connect span が `otel_traces` に流れる。
- **A2**: `meilisearch_driver.go` の全 5 `searchIndex.Search()` 直後で `result.ProcessingTimeMs` を `appotel.RecordMeilisearchProcessing(ctx, op, ms)` 経由で span attribute (`meilisearch.processing_ms`) と debug log に出力。driver 戻り値の signature は触らず、port / gateway / mock 5 層への波及を避ける。
- **A3**: `alt-backend/usecase/global_search_usecase` の 3 goroutine を `tracer.Start(ctx, "global_search.section.<name>")` で囲み `duration_ms` / `limit` / `degraded` を attribute に乗せる。tracer は `otel.Tracer("alt-backend/global_search")` のデフォルト global (no-op safe)。

### Phase B — Tuning (Phase A 後)

- **B1 (Ollama keep_alive + warmup)**: `compose/ai.yaml` に `OLLAMA_KEEP_ALIVE=${OLLAMA_KEEP_ALIVE:-24h}` を追加し、`search-indexer/bootstrap/warmup.go` で起動時に goroutine から `searchEngine.Search(ctx, "warmup-probe-aaa", 1)` を 1 回叩く。30 s timeout + Warn log で startup は絶対に block しない。`OLLAMA_MAX_LOADED_MODELS=2` (既存、[[000830]]) と組み合わせて qwen3-embedding を resident に保つ。
- **B2 (in-driver LRU cache)**: `search-indexer/driver/meilisearch_cache.go` (新規) で `hashicorp/golang-lru/v2` を使い、1024 entry / 5 min TTL の LRU を導入。cache key は `{query (NFC + ToLower + whitespace fold), user_id, filter, offset, limit, embedder, semanticRatio}` で **tenant 越境を構造的に不可** にし、hybrid config 変更時に旧 key が hit しないようにする。invalidation は TTL のみ (write 時 flush しない)。
- **B3 (`searchCutoffMs=1500`)**: `EnsureIndex` の末尾で `index.UpdateSearchCutoffMs(config.MeiliSearchCutoffMs)` を呼ぶ。Meilisearch が処理時間 budget を超えると partial result + `estimatedTotalHits` が lower bound 化して返ってくるため、長時間ロックを engine 側で打ち切れる。
- **B4 (`attributesToRetrieve` 最小化 + `attributesToCrop`)**: `newBaseSearchRequest` の SearchRequest に `AttributesToRetrieve: [id, title, tags, user_id, language, published_at]` (content を外す) と `AttributesToCrop: [content]` + `CropLength: 120` を追加。`hitsToDocs` / `extractDocs` は `getCropped(hit, "content")` で `_formatted.content` を読む。**Pact contract (`matchers.Like("body")`) は型のみで長さ非依存なので consumer/provider verification とも無変更で pass**。
- **B5 (per-section timeout)**: `sectionTimeout = 10 * time.Second` を `sectionTimeouts{Articles: 5s, Recaps: 5s, Tags: 1s}` に分割。tag は alt-db prefix query のみで 1 s 十分、articles/recaps は warmup + cache + cutoff 後の現実的最悪値が 5 s 以下に収まる。
- **B6 (singleflight)**: `golang.org/x/sync/singleflight` を direct require に昇格し、cache miss 時の Meilisearch 呼出を `sf.DoChan` で coalesce する。production trace で観測された「RAG が同 query を 300 ms 以内に 7 連発」のような cache-warming 期間中の thundering herd を 1 回に集約する。caller 側の ctx cancel が他 waiter を巻き込まないよう `select { <-ctx.Done(): / <-ch: }` で受ける。

### 検討した代替案

| 代替案 | 却下理由 |
|---|---|
| `MEILI_HYBRID_SEMANTIC_RATIO=0` で hybrid を切る | semantic search 品質が下がる。ユーザ方針 1 で除外 (cold-start は warmup + keep_alive で消せるため品質を犠牲にする必要がない) |
| 短いクエリだけ hybrid skip (1-3 文字や ASCII のみ) | 分岐ロジックが increase、relevance regression のリスクあり。後続で必要なら別 ADR |
| Frontend に debounce を入れる | `useGlobalSearch` の caller は Enter / URL navigation 起動で毎 keypress 発火していない。実効性なし |
| PostgreSQL の covering index `articles(id) INCLUDE (feed_id, title, url)` を追加 | ユーザ方針 3 で index 追加は後回し。`feed_url_link_driver.go` は既に `ANY($1)` batch query で N+1 なし。実測で hot path と確認できたら別 ADR で扱う |
| Redis を共有 cache として導入 | 単一マシン前提 ([[000035]] 等の Alt 全体方針) と整合しない。in-memory LRU で十分 |
| driver の戻り値に `(docs, processingMs, err)` を追加 | port / gateway / mock の 5 層に波及する。span attribute の sideband で同じ情報が取れるため signature 変更を避ける |

## Consequences

### Pros

- **cold-start 1254 ms → ~200 ms 以下** を見込める (warmup 後の Meilisearch processingTimeMs は 100-200 ms オーダ)。`otel_traces` で `SpanName LIKE '%SearchArticles%'` の max を継続 watch
- **同一クエリ repeat は ~1 ms** (LRU cache hit、Meilisearch 呼出スキップ)。RAG / scroll / refinement の連投が劇的に軽くなる
- **観測性が完全に戻る**: Connect-RPC span + per-section span + Meilisearch processingTimeMs の 3 段で「どこで何 ms 使ったか」が分解可能。次のチューニング判断がデータドリブンになる
- **tail latency cap が 2 段で効く**: `searchCutoffMs=1500` (engine 側) + section timeout (caller 側)。両者が overlap して安全側に倒れる
- **payload 削減**: 50 hit/req で 375 KB → 約 20 KB。LRU cache の memory footprint も同比例で抑えられる (1024 entry で ~30 MB)
- **structural な tenant isolation**: cache key に `UserID` 必須、`UserID=""` (RAG 経路) は別 namespace。コード migration なしで multi-tenant safety を確保
- **CDC 影響ゼロ**: Pact contract の `Like("body")` matcher は型のみ、長さ非依存。crop で content 短縮しても consumer/provider 両方の verification が無変更で pass

### Cons / Tradeoffs

- **cache 古い結果**: TTL 5 min を許容 → 直近 5 分以内に index した articles は repeat query で 1 回遅延する。Meilisearch 自身の hybrid indexing 遅延が分〜時間オーダなので相対的に無視できると判断
- **`estimatedTotalHits` の精度低下**: `searchCutoffMs` 到達時に Meilisearch は lower bound 値を返す。frontend `HasMore` 表示が cutoff hit で偽 (実際は more available) になる可能性。文言調整 (「約 N 件」化) は本 ADR の scope 外、`processing_ms >= 1500` の hit 率を `otel_logs` で watch しながら対応判断
- **`detectMatchedFields` の content 一致検出が精度劣化**: cropped (120 token) content しか見ないので、それより後ろの位置で content match した場合に検出されない。frontend 表示は title / tags 主体で十分とユーザ確認済、許容
- **OLLAMA_KEEP_ALIVE=24h で GPU VRAM 占有**: qwen3-embedding (640 MB) が常時 resident。`OLLAMA_MAX_LOADED_MODELS=2` ([[000830]]) と組み合わせて gemma4 (5.3 GB) と共存できる前提。VRAM 圧迫が再発したら `OLLAMA_KEEP_ALIVE` を短く override
- **lint pre-existing 残置**: `search-indexer/driver/meilisearch_driver.go` の `15 * time.Second` / `interface{}` / `h2c.NewHandler` deprecation 等は触っていないので残る。CI の `only-new-issues: true` 設定 ([[reusable-go-quality-gates]]) で本 commit は緑、別 hygiene PR で対応
- **PR-B6 の効果は production traffic 形状に依存**: RAG 同 query 連投は計測済だが、定量効果は deploy 後の cache hit rate metric (`SearchCacheHits/Misses/Evicts`) で確認する必要がある

### 効果検証クエリ (deploy 後)

```sql
-- cold-start 消失確認 (ClickHouse rask_logs.otel_traces)
SELECT toStartOfMinute(Timestamp) AS m,
       quantile(0.95)(Duration/1e6) AS p95_ms, max(Duration/1e6) AS max_ms
FROM otel_traces
WHERE ServiceName = 'search-indexer' AND SpanName LIKE '%SearchArticles%'
  AND Timestamp > now() - INTERVAL 1 HOUR
GROUP BY m ORDER BY m DESC;

-- cutoff 到達率 (Meilisearch budget 超え)
SELECT countIf(toInt64(LogAttributes['processing_ms']) >= 1500) * 100.0 / count() AS cutoff_pct
FROM otel_logs
WHERE ServiceName = 'search-indexer' AND Body LIKE '%meilisearch search%'
  AND Timestamp > now() - INTERVAL 1 HOUR;
```

目標値: cold-start max_ms ~300 ms 圏、cutoff_pct < 5 %、cache hit rate ≥ 30 %。
