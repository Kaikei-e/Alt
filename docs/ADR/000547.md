# ADR-000547: recap-worker OOM対策と全分類失敗時の正常完了バグ修正

## Status

Accepted

## Date

2026-03-23

## Affected Services

- recap-worker (Rust)
- compose/recap.yaml (Docker Compose)

## Context

recap-worker が Recap 生成中に OOM (Out of Memory) で kill されていた。加えて、全分類チャンクが失敗しても空の結果で正常完了扱いになるバグが潜在していた。

### OOM の原因

recap-worker は libtorch 2.4.0 (CPU版) + AllMiniLmL12V2 (33.4M params) を常駐ロードしており、固定メモリだけで約 500-710MB を消費する。パイプライン実行時のピーク使用量は 700-850MB だが、glibc malloc のメモリフラグメンテーションにより RSS が実使用量の 1.5-2 倍に膨張し、`mem_limit: 1024m` の制限を超過していた。

### 正常完了バグの原因

recap-subworker が不在の場合、`classify-runs` への全リクエストが接続エラーで失敗する。しかし以下のバグチェーンにより、パイプラインは空の Recap を「成功」として記録していた。

1. `ClassificationJobQueue::wait_for_completion()` が `failed` ステータスのジョブも「完了」として扱い、空の結果を返す
2. `RemoteGenreStage::assign()` が空の分類結果を検証せず `Ok(GenreBundle { assignments: [] })` を返す
3. `evaluate_job_outcome()` の fallback が全カウンタ 0 のケースを `JobOutcome::Success` と判定する

## Decision

### 1. メモリ制限の引き上げと jemalloc 導入

`mem_limit` を 1024m → 2048m に引き上げ、`deploy.resources.reservations.memory: 768M` を追加する。

構造的対策として `tikv-jemallocator` をグローバルアロケータに設定し、libtorch の C++ メモリ管理による glibc フラグメンテーションを根本解決する。`unprefixed_malloc_on_supported_platforms` feature により libtorch の C++ 側の割り当ても jemalloc 経由にする。PyTorch Blog の報告ではピーク RSS が平均 34% 減少する。

### 2. 全分類失敗時のエラーハンドリング修正

2 箇所で防御する:

- **Fail-fast**: `RemoteGenreStage::assign()` に空結果バリデーションを追加。分類結果が 0 件かつ入力記事が存在する場合に即座に `Err` を返す
- **Safety-net**: `evaluate_job_outcome()` の fallback を `JobOutcome::Failed` に変更。`total_genres > 0` だが全カウンタが 0 の場合を「分類が完全に失敗した」と判定する

## Consequences

### Pros

- OOM kill が解消され、2908 記事規模のバッチ処理でも安定動作する
- jemalloc により libtorch 推論のメモリフラグメンテーションが 30-40% 削減される
- 分類サービス障害時にパイプラインが即座に失敗し、空の Recap が DB に永続化されなくなる
- resumable job メカニズムにより、失敗したジョブは次回起動時に自動再実行される

### Cons / Tradeoffs

- `mem_limit: 2048m` は 20+ コンテナが同一ホストで稼働する環境では大きめだが、jemalloc 導入後の実効ピークは 1GB 前後に収まるため余裕として許容する
- `tikv-jemallocator` はビルド時に jemalloc をソースからコンパイルするため、初回ビルド時間が若干増加する（~1-2分）
- Fail-fast バリデーションにより、分類サービスの一時的なダウンでパイプライン全体が失敗するようになる。ただし resumable job 機構により次回起動時に自動再試行されるため、運用上の影響は限定的
