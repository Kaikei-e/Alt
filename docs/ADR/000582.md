# ADR-000582: GetUnreadFeeds パフォーマンス改善 — BFFキャッシュ・covering partial index・ReadStateキャッシュの3層対策

## Status

Accepted

## Date

2026-03-25

## Affected Services

- alt-butterfly-facade (Go) — BFF レスポンスキャッシュ設定追加
- alt-backend (Go) — ReadState キャッシュ追加
- migrations-atlas — articles テーブルに covering partial index 追加

## Context

ダッシュボードの未読フィード一覧取得（`GetUnreadFeeds` RPC）がキャッシュ未ヒット時に体感で遅い。

### リクエストフロー

```
Browser → SvelteKit /api/v2 → alt-butterfly-facade (BFF) → alt-backend (Connect-RPC)
```

### EXPLAIN ANALYZE による実測

| 計測対象 | 実行時間 | 備考 |
|----------|----------|------|
| 直接SQL `FetchUnreadFeedsListCursor` LIMIT 20 | 3.3ms | DB 最適化済み |
| 84購読分 `FetchFeedsByFeedLinkID` 逐次合計 | 120ms | 各1.4ms avg |
| `GetAllReadFeedIDs` (30,477行) | 11ms | Seq Scan（単一ユーザーで全行ヒット） |
| 全購読一括クエリ (65,466行) | 485ms | 相関サブクエリ 65K loops |

### `feed_read_perf` ログによる E2E 実測

| 状態 | usecase_ms | 説明 |
|------|-----------|------|
| SharedCache コールド | **339ms** | 84購読×200件=最大16,800件を全量ロード→ソート→フィルタ |
| SharedCache ウォーム | **16ms** | キャッシュヒットのみ |
| SharedCache 期限切れ直後 | **263ms** | 再ロード |

### 根本原因（3つ）

1. **BFF が `GetUnreadFeeds` をキャッシュしていなかった** — `defaultTTLs` に未登録で、毎リクエストが alt-backend に到達していた
2. **`CachedFeedListUsecase` のコールドパスが非効率** — LIMIT 20 のみ必要なのに 84購読×200件=16,800件を全量ロード→ソート→30K read IDs 取得→メモリ内フィルタ
3. **`GetAllReadFeedIDs` がキャッシュなし** — 毎回 Seq Scan で 30,477行を取得（`subscriptionCache` のような SharedCache が未適用）

### ベストプラクティス調査

Web 検索で以下のベストプラクティスを確認し、設計に反映した:

- **BFF キャッシュ**: per-user feed lists は max-age 10-30s が推奨（Connect-RPC 公式ドキュメント、AWS API Gateway 実践例）
- **PostgreSQL**: boolean カラムの部分インデックスと `INCLUDE` による covering index で index-only scan を実現
- **In-process キャッシュ**: 単一 BFF インスタンスなら Redis 不要、30-60s TTL で十分

## Decision

3つの独立した層でキャッシュ・インデックス改善を行う。

### Phase 1: BFF レスポンスキャッシュに `GetUnreadFeeds` / `GetAllFeeds` を追加

`alt-butterfly-facade/internal/cache/response_cache.go` の `defaultTTLs` に2エントリを追加し、TTL 10秒でキャッシュする。

```go
"/alt.feeds.v2.FeedService/GetUnreadFeeds": 10 * time.Second,
"/alt.feeds.v2.FeedService/GetAllFeeds":    10 * time.Second,
```

TTL 10秒の根拠:
- ベストプラクティス推奨の 10-30s 範囲の下限を採用
- `GetUnreadCount` (15s) より短く設定し、`MarkAsRead` 後の体感反映を速くする
- SvelteKit ハイドレーション時の二重リクエストとスクロール操作を吸収

### Phase 2: articles テーブルに covering partial index を追加

全フィードクエリで使われる相関サブクエリを最適化する:

```sql
CREATE INDEX IF NOT EXISTS idx_articles_feed_active_created_desc
  ON articles (feed_id, created_at DESC)
  INCLUDE (id)
  WHERE deleted_at IS NULL;
```

- `INCLUDE (id)` で index-only scan を実現（サブクエリは `a.id` のみ SELECT）
- `WHERE deleted_at IS NULL` でソフトデリート行を除外する部分インデックス
- Sort ステップの除去（インデックスが `created_at DESC` 順を提供）

### Phase 3: `GetAllReadFeedIDs` に SharedCache を追加

`alt-backend/app/gateway/user_read_state_gateway/gateway.go` に `readStateCache` を追加。既存の `subscriptionCache` と同じ `SharedCache` パターンを踏襲:

- TTL 5秒（stale なし）
- `singleflight` による重複リクエスト排除
- `subscriptionCache` (30s) より短く設定（read state は頻繁に変化するため）

## Consequences

### Pros

- **BFF キャッシュヒット時**: 339ms → 0ms（10秒 TTL 内の繰り返しリクエスト）
- **相関サブクエリ**: Index Scan → Index Only Scan（Heap Fetches: 0）。単一購読クエリ 4.0ms → 0.55ms（7.3x 高速化）。84購読合計 120ms → 40-58ms（2-3x 高速化）
- **ReadState 取得**: 毎回 11ms の Seq Scan → 5秒 TTL 内はキャッシュヒットで 0ms
- 全層で既存パターンを踏襲しており、新しい依存関係やインフラ変更なし

### Cons / Tradeoffs

- **BFF キャッシュの staleness**: `MarkAsRead` 後、最大 10秒間は古い未読リストが返る可能性がある。ただし既存の `GetUnreadCount` (15s TTL) と同等の staleness window であり、ユーザー体験への影響は限定的
- **ReadState キャッシュの staleness**: 最大 5秒間の遅延。BFF TTL と合わせると理論上最大 15秒の staleness（BFF 10s + ReadState 5s）
- **BFF に stale-while-revalidate がない**: 現在の `ResponseCache` は単純 TTL ベースのみ。TTL 切れ時はコールドパスに戻る。将来的に SWR パターンの実装が望ましい
- **`CachedFeedListUsecase` の根本的な非効率は未解決**: LIMIT 20 に対して 16,800件を全量ロードするアーキテクチャ。直接 SQL (3.3ms) を fallback に使うなどの改善は将来の課題
