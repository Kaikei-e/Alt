# ADR-000861: Surface Planner v2 の signal を 6 軸拡張し WhyMappingVersion を 6 から 7 に bump して Review レーンを再評価キューに昇格させる

## Status

Accepted

## Date

2026-04-27

## Affected Services

- knowledge-sovereign: `SurfaceScoreInputs` に `EvidenceDensity` / `RecapClusterMomentum` / `QuestionContinuationScore` / `ReportWorthinessScore` / `StalenessScore` / `ContradictionCount` を additive に追加。`pureStalenessBucket(eventOccurredAt, sourceObservedAt time.Time) uint32` を新ファイル `staleness_bucket.go` で純関数として実装
- knowledge-sovereign: `EventLogSurfaceScoreResolver` が `event.OccurredAt - 7d` window で `SummarySuperseded` / `AugurConversationLinked` / `RecapTopicSnapshotted` をカウントし、新フィールドに反映。`readSourceObservedAt` が payload `published_at` / `source_observed_at` / `observed_at` を許容して staleness を pin
- knowledge-sovereign: `decideBucketV2` の優先順位を canonical contract §6 に整合させて刷新 (CHANGED → CONTINUE → NOW → REVIEW、Staleness が REVIEW を elevate)。`marshalSurfaceScoreInputs` JSON が新 6 フィールドを `omitempty` で出力
- knowledge-sovereign: `WhyMappingVersion` 定数を 6 → 7 に bump
- docs: `docs/runbooks/knowledge-loop-reproject.md` の WhyMappingVersion 履歴に v7 row を追加し、本 ADR への参照を埋め込む

## Context

[[000853]] / [[000854]] / [[000856]] で Surface Planner v2 の resolver / planner version / score inputs 永続化までは完成したが、`fb.md` の §B-2 レビューでは「Review が leftover bucket になっており、staleness や contradiction を反映できていない」「Recap 由来の momentum が NOW promotion に乗っていない」「QuestionContinuation を bool でしか表現していないので、複数 thread の continue が等価扱いされる」など、placement signal の解像度不足が指摘された。

具体的には、本 ADR を書く時点の `decideBucketV2` は次の 5 入力で決定していた。

- `TopicOverlapCount` (uint32)
- `TagOverlapCount` (uint32)
- `HasAugurLink` (bool)
- `VersionDriftCount` (uint32)
- `HasOpenInteraction` (bool)

これは v1 fixed mapping から v2 evidence-based scoring への第一段階としては妥当だったが、以下が表現できていなかった。

- **Contradiction** — `SummarySuperseded` が立っているのに entry が CHANGED に行かないケース (overlap signal のみで上書きされる)
- **Question continuation** — Augur thread が複数走っている entry と 1 件だけの entry を区別できない
- **Recap cluster momentum** — Recap が短期に何度も同じ topic を含む signal を NOW promotion に使えない
- **Staleness** — source の `published_at` から十分時間が経った entry を Review に積極的に押し出せない
- **Evidence density** — `WhyPayload.evidence_refs` に紐がない entry が他と同じ扱い

`fb.md` §B-2 が要求した 6 軸を SurfaceScoreInputs に additive に追加し、`decideBucketV2` の priority order を canonical contract §6 に揃え直すことで、上記 placement gap を埋める必要があった。

加えて canonical contract §3 invariant 1 (reproject-safe) と invariant 6 (event-time purity) を破らないよう、staleness 計算は wall-clock を使えない。`time.Now()` を使った決定は replay の度に違う bucket を返すため、reproject-safety を壊す。`immutable-design-guard` skill での監査で「pure 関数を inputs に適用する」「window は `event.OccurredAt - 7d` で固定する」を要件として確定した。

## Decision

`SurfaceScoreInputs` を 6 軸 additive に拡張し、`decideBucketV2` priority を canonical contract §6 と `fb.md` §B-2 に整合させて刷新する。`WhyMappingVersion` を 6 → 7 に bump して reproject runbook で全 entry の placement を v7 ロジックで再生成できるようにする。

採用した signal:

- **`StalenessScore uint32`** — `pureStalenessBucket(event.OccurredAt, source_observed_at)` で固定境界 (`<1h=0` / `<1d=1` / `<7d=2` / `<30d=3` / `≥30d=4`) を返す純関数。`source_observed_at` が無いイベントでは 0 (fresh) を返し、negative gap (clock skew) も 0 に正規化。`time.Now()` を使わないので reproject-safe
- **`ContradictionCount uint32`** — score window 内の `SummarySuperseded` で `article_id` 一致するもののカウント。`VersionDriftCount` と並走するが、将来 `SummaryContradicted` event を別 emit できるようになったときに分離するための土台
- **`QuestionContinuationScore uint32`** — `AugurConversationLinked` で `entry_key` 一致する events のカウント。`HasAugurLink` (bool) はそのまま残し、count 軸は別フィールドにすることで「1 thread / N threads」の差を Continue 判定で使えるようにした
- **`RecapClusterMomentum uint32`** — `RecapTopicSnapshotted` で term overlap が立った events のカウント。`TopicOverlapCount` と現状は同じ計算結果になるが、将来 `cluster_id` 単位での dedupe 計算へ移行できるよう named-different にしておく
- **`EvidenceDensity uint32`** — `WhyPayload.evidence_refs` の長さ。NullResolver が常に 0 を返すため、`decideBucketV2` の REVIEW elevation 条件には使わない (理由は下の trade-off 節)
- **`ReportWorthinessScore uint32`** — Acolyte レポート連携の予約フィールド。現時点は常に 0、`decideBucketV2` も無視する。事前に proto / JSON shape を確保する目的

`decideBucketV2` priority は次のとおり刷新した。

```
1. VersionDriftCount > 0  || ContradictionCount > 0           → CHANGED
2. HasAugurLink || HasOpenInteraction || QuestionContinuationScore > 0 → CONTINUE
3. TopicOverlapCount > 0 || TagOverlapCount > 0 || RecapClusterMomentum > 0 → NOW
4. StalenessScore >= 2 || isV1ReviewEvent(EventType)          → REVIEW
5. v1FallbackBucket(EventType)                                 → 既存 v1 mapping
```

`marshalSurfaceScoreInputs` JSON は 6 フィールドを全て `omitempty` で出力するため、v6 row との後方互換も維持される。

`WhyMappingVersion = 7` の bump は、operators 向けに「placement priority が変わった。reproject runbook を回せば全 entry に v7 ロジックが反映される」という signal。`docs/runbooks/knowledge-loop-reproject.md` の WhyMappingVersion 履歴 table に v7 row を追加し、何が変わったかを記述した。

検討した代替案と却下理由:

- **`EvidenceDensity == 0` を REVIEW promotion に使う**: NullSurfaceScoreResolver が常に `EvidenceDensity = 0` を返すため、これを REVIEW elevator にすると v1 fallback path に乗るすべての entry が Review に飛ぶ。`immutable-design-guard` skill が指摘した violation で、却下
- **`StalenessScore` を decay 値 (float) で表現**: 連続値だと境界変更で reproject 結果が変わる。固定 bucket (uint32) なら境界を変えるたびに `WhyMappingVersion` を bump するルールが守れる
- **proto に `SurfaceScoreInputs` の 6 フィールドを追加する**: `surface_score_inputs` は [[000856]] で `bytes` opaque JSONB として定義済み。proto 変更ぜずに JSON shape の additive 拡張で済む。proto を触ると alt-backend / alt-frontend-sv にも影響が波及するので、変更コストの観点で却下
- **`pureStalenessBucket` を context.Context 引数に揃えて clock を inject 可能にする**: 純関数として `time.Now()` を一切使わない設計が canonical contract §3 invariant 6 に最も忠実。引数で clock 注入する余地は残さず、replay 時の決定論を最優先

## Consequences

### Pros

- placement signal の解像度が 5 軸 → 11 軸 (5 既存 + 6 新規) に上がり、Review が「leftover」から「再評価キュー」に昇格する。Staleness ≥ 2 の entry が積極的に Review に流れ込む
- `pureStalenessBucket` は 100 反復しても同じ入力で同じ出力を返す determinism test で reproject-safety を担保。`time.Now()` を持ち込まない原則が unit test で固定されている
- `WhyMappingVersion 7` bump が runbook table に明記されているので、operator は本 ADR を読まずとも v7 cutover 手順が分かる
- `proto` を触らない additive な signal 拡張なので、alt-backend / alt-frontend-sv への波及がゼロ。サービス境界の breaking change リスクなし
- `ReportWorthinessScore` を予約フィールドとして wire したことで、Acolyte 連携が来たときに proto 改修なしで signal を流し込める

### Cons / Tradeoffs

- 6 軸の signal は all event-log 由来の derived count であり、source event log に十分な volume が無いテナント / 環境では低解像度に縮退する。これは canonical contract §3 invariant 1 (event-source) を守る代償
- `decideBucketV2` の優先順位刷新は existing v6 entry の placement を一部変える可能性がある。WhyMappingVersion 7 bump で reproject 必須と運用上明記し、reproject 前は混在状態を許容する
- `RecapClusterMomentum` と `TopicOverlapCount` が現状同じ値を返すのは将来 cluster_id 集約への移行に向けた preparation で、現時点は重複情報。観測ダッシュボードを作る際は混乱しないよう field 説明を読む必要がある
- `StalenessScore` の境界 (1h / 1d / 7d / 30d) は経験的に置いた値で、user prefs / lens-mode 別の chunk size を持たない。境界を将来変えるときは WhyMappingVersion を bump して reproject する運用ルールに従う
