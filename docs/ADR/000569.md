# ADR-000569: Agentic RAG の DI 配線・ツール実装・API 契約を完成させる

## Status

Accepted

## Date

2026-03-25

## Affected Services

- rag-orchestrator (Go)

## Context

[[000568]] で Agentic RAG の 5 層アーキテクチャ（Retrieval Quality Gate、Smart Query Classification、Tool Augmentation、Answer Quality Validation、Observability）の骨格を実装した。しかし以下の統合作業が未完了だった。

1. **DI 配線未接続** — `RetrievalQualityAssessor`、`QueryClassifier`、4 つの新 Strategy（temporal, comparison, deep_dive, fact_check）が `di/container.go` に配線されておらず、実行時に機能しない
2. **PromptBuilder が intent を認識しない** — intent 別のプロンプトインストラクション（比較・時系列・深掘り・ファクトチェック）が未実装
3. **ToolDispatcher が buildPrompt() に未統合** — ツール結果がプロンプトに注入されない
4. **回答品質チェックが Execute() に未統合** — `AssessAnswerQuality()` の結果が `AnswerDebug` に反映されない
5. **具体ツール未実装** — `domain.Tool` インターフェースのみで `adapter/tools/` が存在しない
6. **OpenAPI spec 未更新** — `AnswerDebug` の新フィールドが REST API レスポンスに露出しない
7. **Stream() パスが旧 AnswerDebug** — ストリーミング経路で新フィールドが欠落

## Decision

**骨格と統合を分離した 2-ADR 戦略を維持し、残りの 7 タスクを一括で完成させる。**

### 実装の概要

#### 1. DI 配線（`di/container.go`）

`answerUsecase` の生成で `WithStrategy` を 4 intent 分追加し、`WithQualityAssessor`（`QualityGateConfig.Enabled` で制御）と `WithQueryClassifier` を配線した。

```go
answerOpts := []usecase.AnswerUsecaseOption{
    usecase.WithCacheConfig(...),
    usecase.WithStrategy(usecase.IntentArticleScoped, articleScopedStrategy),
    usecase.WithStrategy(usecase.IntentTemporal, usecase.NewTemporalStrategy(retrieveUsecase, log)),
    usecase.WithStrategy(usecase.IntentComparison, usecase.NewComparisonStrategy(retrieveUsecase, log)),
    usecase.WithStrategy(usecase.IntentTopicDeepDive, usecase.NewTopicDeepDiveStrategy(retrieveUsecase, log)),
    usecase.WithStrategy(usecase.IntentFactCheck, usecase.NewFactCheckStrategy(retrieveUsecase, log)),
    usecase.WithQueryClassifier(usecase.NewQueryClassifier(nil, 0)),
}
```

`QualityGateConfig` は `infra/config/config.go` に追加済み（`RAG_QUALITY_GATE_ENABLED` デフォルト有効）。

#### 2. PromptBuilder intent 別インストラクション（`prompt_builder.go`）

`PromptInput` に `IntentType` フィールドを追加し、`Build()` 内の `switch` 文で intent ごとの日本語インストラクションを挿入する。

| Intent | 追加インストラクション |
|--------|---------------------|
| Comparison | 「両者を公平に比較し、共通点・相違点を構造化してください」 |
| Temporal | 「最新の情報を優先して回答してください。日付・時期を明記」 |
| TopicDeepDive | 「背景・詳細・影響を包括的に解説してください」 |
| FactCheck | 「出典を明示し、根拠と判定を構造化して回答してください」 |
| General | 追加なし（既存プロンプトのまま） |

`buildPrompt()` から `PromptInput.IntentType = intent.IntentType` で渡す。

#### 3. ToolDispatcher 統合（`answer_with_rag_usecase.go`）

`WithToolDispatcher` option と `toolDispatcher` フィールドを追加。`buildPrompt()` 内で retrieval 後に `toolDispatcher.Dispatch()` を呼び、結果を `PromptInput.SupplementaryInfo` として prompt に注入する。

`PromptInput.SupplementaryInfo []string` を `prompt_builder.go` に追加し、`Build()` で `### 補足情報（ツール結果）` セクションとして出力。

#### 4. QualityFlags 統合（`answer_with_rag_usecase.go`）

`Execute()` 内で LLM 生成・バリデーション後に `AssessAnswerQuality()` を呼び、結果を `AnswerDebug.QualityFlags` に記録。ログにも出力。

#### 5. 具体ツール実装（`adapter/tools/`）

4 ツールを実装。いずれも既存の domain インターフェースを再利用。

| ツール | 依存 | 概要 |
|--------|------|------|
| `TagSearchTool` | `domain.SearchClient` | Meilisearch でタグ/キーワード検索。上位 5 件を返す |
| `DateRangeFilterTool` | `domain.SearchClient` | 時系列クエリに対して検索を実行。結果を返す |
| `ArticleLookupTool` | `domain.ArticleClient` | alt-backend から直近 7 日の記事を取得 |
| `RelatedArticlesTool` | `domain.SearchClient` | 関連記事を検索して返す |

#### 6. OpenAPI spec + 生成コード更新

`spec/openapi.yaml` の `AnswerDebug` スキーマに 5 フィールド追加。`make generate` で `oapi-codegen` により `server.gen.go`（rag-orchestrator）と `client.gen.go`（alt-backend）を再生成。

REST handler で `ptrIfNotEmpty()` / `ptrSliceIfNotEmpty()` ヘルパーを使い、新フィールドを OpenAPI 生成コードにマッピング。

#### 7. Stream() パスの AnswerDebug 拡張

`rag_answer_stream.go` の `StreamMeta.Debug` 構築で `IntentType`、`RetrievalQuality`、`RetryCount`、`ToolsUsed` を追加。REST と Connect-RPC の両経路で同一の debug 情報が取得可能。

### テストカバレッジ

- PromptBuilder intent 対応: 4 テスト（Comparison / Temporal / FactCheck のインストラクション検証 + General で非混入確認）
- ツール実装: 5 テスト（TagSearch 成功/空結果、DateRangeFilter 成功、名前確認 ×2）
- 既存テスト含め全 211 テストパス、リグレッションなし

## Consequences

### Pros

- **全 Phase が実行可能** — DI 配線により、コンテナ再ビルド後に QualityGate・QueryClassifier・4 Strategy・ToolDispatcher が即座に動作する
- **REST/Connect-RPC 両方で debug 情報が揃う** — intent_type、retrieval_quality、retry_count、tools_used、quality_flags がレスポンスに含まれ、運用時の問題切り分けが容易になる
- **PromptBuilder が intent-aware** — 同じ LLM 呼び出しでも、intent に応じた指示文により回答品質が向上する（追加 LLM コストなし）
- **ツールが再利用ベース** — 4 ツールすべてが既存の `domain.SearchClient` / `domain.ArticleClient` を再利用しており、新規外部依存なし

### Cons / Tradeoffs

- **コンテナ再ビルド未実施** — 実行時動作は未確認。デプロイ前に `docker compose up --build -d rag-orchestrator` と統合テストが必要
- **alt-backend の OpenAPI クライアントも再生成済み** — `make generate` が `alt-backend/app/gateway/rag_gateway/client.gen.go` も更新するため、alt-backend 側のビルド確認も必要
- **ToolDispatcher のルールが限定的** — 現在は Temporal → DateRangeFilter、Comparison → TagSearch のみ。intent とツールのマッピングは今後拡張が必要
