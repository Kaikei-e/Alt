# ADR-000896: /loop の Review / Why / Changed を再評価体験に揃え、WhyMappingVersion を v9 にする

## Status

Accepted

## Date

2026-05-09

## Affected Services

- knowledge-sovereign (Go) — review-lane lifecycle split + Why priority reorder + ChangeSummary 更新ヒント連結 + WhyMappingVersion v9
- alt-frontend-sv (SvelteKit) — Review reason chip / Compare CTA / "What to update" band / Continue 行動ラベル
- docs (runbook) — `knowledge-loop-reproject` に v8 / v9 history rows

## Context

`docs/plan/knowledge-loop-completion-00-index.md` に基づく Knowledge Loop 完成計画の Phase 3。`fb.md` 静的監査と canonical contract `[[knowledge-loop-canonical-contract]]` のギャップを段階的に解消する一連のフェーズの 3 本目で、Review を「dismiss されたものの置き場」ではなく「再評価キュー」へ、Why / Changed / Continue を「何が起きたか」だけでなく「次に何をすべきか」を伝える説明品質に引き上げる作業。

実装前に存在していた具体ギャップは 3 つ。

1. **Review-lane lifecycle の崩壊**: `knowledge_loop.reviewed.v1` の 3 つの trigger (`recheck` / `archive` / `mark_reviewed`) は `PatchKnowledgeLoopEntryDismissState` 一本を経由し、visibility_state を `lifecycleVisibilityForDismissPatch(dismiss_state)` という flat lookup で派生していた。この lookup は COMPLETED → HIDDEN 固定なので、`mark_reviewed` (= 「再評価したと記録するが Review に残す」) が `archive` (= 「読み終えたので隠す」) と区別できず、ユーザーが acknowledge した瞬間に entry が `/loop` から消えていた。read path の `WHERE visibility_state='visible'` フィルタが二者を巻き込んで除外していたため、UI からは実害として「Reviewed が機能していない」状態。

2. **Why narrative の優先順位が canonical contract と逆転**: `why_override.go` は v8 の時点で `version_drift > augur_link > open_interaction > topic_overlap > tag_overlap` の順だった。canonical contract §11 が定義する優先順は `change > unfinished_continue > topic_affinity > tag_trending > recall > source` で、RECALL は「他に signal がないときの residual 候補」。v8 順では「過去に 1 度開いた」記録が「いま active な recap topic と接続している」を踏みつぶし、Topic Affinity / Tag Trending narrative が事実上死んでいた。

3. **Changed カードが「何が変わったか」だけで「どう更新すべきか」を出していない**: `ChangedDiffCard.svelte` は redline + producer summary を表示するが、CTA は `Confirm` のみで semantic intent を持たず、ユーザーは「読んだ → 次へ進む」の意味で押すしかなかった。`acted_intent=compare` を canonical contract / Phase 2 transition-metadata helpers が既にサポートしているのに UI 側が wire していなかった。`Continue` カードでは `recent_action_labels` が backend の past-tense (`opened` / `compared`) のまま表示され、CTA (`Open` / `Compare`) と語彙が一致しないせいで「どのボタンが効いたのか」がユーザーに不可視。

加えて、Phase 3 を実装するにあたり可能な限り contract 変更を避けることが要求されていた (canonical contract §3 の append-first 不変条件 + 新 proto field を入れると pact gate が走り全 consumer が再生成される)。**新 proto field / 新 JSONB field を増やさず**、既存フィールドの再解釈と FE 派生で完了させる方針を取った。

## Decision

### 1. Review-lane lifecycle を trigger ごとに分割する

`reviewLifecycleForReviewedEvent(payload) ReviewLifecycle` を pure 関数として `review_lifecycle.go` に新設。trigger を payload から抽出し、`(dismiss_state, visibility_state, completion_state)` の 3 つ組を独立に返す:

| trigger | dismiss | visibility | completion |
|---|---|---|---|
| `RECHECK` | ACTIVE | VISIBLE | OPEN |
| `ARCHIVE` | COMPLETED | HIDDEN | COMPLETED |
| `MARK_REVIEWED` | COMPLETED | **VISIBLE** | COMPLETED |
| 不明 / 欠落 | COMPLETED | VISIBLE | COMPLETED (fail-closed but non-destructive) |

driver 側に `PatchKnowledgeLoopEntryReviewLifecycle(dismiss, visibility, completion)` を追加し、3 状態を SQL 引数として明示的に受け取る (既存 `patchKnowledgeLoopEntryDismissStateQuery` の SQL を再利用、seq-hiwater guard も継承)。`PatchKnowledgeLoopEntryDismissState` は `knowledge_loop.deferred.v1` 専用のままにして責務を絞る。

**検討した代替案**:

- **`lifecycleVisibilityForDismissPatch` を bucket-aware にする**: defaultVisibilityState 並みに `(dismiss, bucket)` 双方を取れば DISMISSED+REVIEW=VISIBLE が表現できるが、mark_reviewed と archive は同じ COMPLETED+REVIEW なので bucket だけでは区別できない。trigger を patch 関数のシグネチャに渡す必要があり、結局 lifecycle 全体を引数として受け取る方が clean。
- **新しい `DISMISS_STATE_REVIEWED` proto enum を追加**: Pact gate 起動 + 全 consumer 再生成のコストに見合わない。
- **mark_reviewed は ACTIVE のままにして surface_score_inputs に `review_reason=reviewed` を書く**: Surface Planner v2 が dismiss_state を直接 bucket 決定に使っているわけではないので機能はするが、「読み終えた」が ACTIVE になるのは completion_state の意味論と合わず将来の混乱源。

### 2. Why 優先順位を canonical contract §11 に揃え、WhyMappingVersion を v9 にする

`OverrideWhyFromSurfaceInputs` のブランチ順を `version_drift → has_augur_link → topic_overlap > 0 → tag_overlap > 0 → has_open_interaction → enricher fallback` に並び替える。RECALL は「他に signal がないときの residual」で、active な recap-cluster / tag-stream connection が存在するときは負ける。

`WhyMappingVersion` を v8 → v9 にバンプし、`docs/runbooks/knowledge-loop-reproject.md` の history table に v8 (SurfacePlanRecomputed projector branch を追加した過去のバンプ; これまで未記載だった抜け穴を補完) と v9 (本 ADR) を追記。runbook の標準 Procedure に従って full reproject を回せば既存 entries が新優先順位で converge する。

**検討した代替案**:

- **追加なしの additive 変更扱いにしてバンプを skip する**: 既存 WhyKind の意味論は変わらないが優先順位は変わるため、同じ event 列から異なる `why_kind` / `why_text` が出る = 既存 entries の Why が突然書き換わる。Web 調査 (Greg Young / Marten projector versioning) でも「mapping semantics の変更には version bump」が consensus。runbook のバンプ規約にも合致するため、bump を選択。

### 3. ChangeSummary に決定論的 update hint を連結する

`composeChangeSummaryWithHint(producerSummary, changedFields) string` を `change_summary_builder.go` に追加し、`buildChangeSummaryJSON` が producer summary を加工するパスを通す。テンプレートは pinned constants:

| changed_fields に含まれる | 連結句 |
|---|---|
| `summary` | `re-read the lede before quoting` |
| `tags` | `rethink which thread this belongs to` |
| `source` | `verify the canonical link` |

producer summary が空なら hint 単体を capitalized + period 付きで返す。複合句は `; ` で連結。240 char の上限を設ける (FE band 用 + 512-char why_text ceiling との整合性)。LLM は使わない (prompt injection と replay 非決定性が canonical contract §3 の reproject-safety 不変条件を破る)。

**検討した代替案**:

- **`change_summary` JSONB に `update_hint` 新 field を additive 追加**: Pact gate 起動。FE 側も新しい mapping 経路が必要。既存 `summary` field に hint discipline を持たせる方が小手先の変更で済む。
- **producer (alt-backend) に hint 生成責務を移す**: producer は upstream イベントの整形側で、Loop 側の UX semantics を持ち込まない方針 ([[000844]] で projector ownership を knowledge-sovereign に移したのと同じ理由)。

### 4. Review reason / Compare CTA / Continue ラベルを FE 派生で実装する

新 proto / 新 wire field を一切増やさず、既存 fields の再解釈で 3 つのギャップを埋める:

- **`review-reason-map.ts`**: `dismissState` + `surfaceBucket` のみから `Stale evidence` / `Previously dismissed` / `Reviewed` / `Deferred` を派生。read path が `visibility_state='visible'` で hidden を弾いているため、FE が見える状態空間は小さい。Same input → same label の deterministic mapping。
- **`recent-action-label-map.ts`**: backend の past-tense (`compared` / `opened` / `opened_recap` …) を CTA register (`Compare` / `Open` / `Open Recap` …) に humanize。未知ラベルは title-case fallback。
- **Compare CTA**: `ChangedDiffCard.svelte` の `Confirm` を `Compare` に置換、`onCompare` callback prop で受け取り、`/loop/+page.svelte` の `onChangedCompare` が `buildTransitionMetadata(entry, {intent: "compare"})` を組み立てて same-stage transition を発火。`pickTargetForIntent` が既に `compare → diff` を返すため Phase 2 helper を再利用するだけ。stage advance はしない (canonical contract §11 が Changed を comparison bucket と定義しており、Confirm の advance は v8 の暫定挙動だった)。
- **"What to update" band**: `ChangedDiffCard.svelte` 内に `cs.summary` (= 上記 §3 の hint 込み summary) を別 band として表示。空なら band は出さない。
- **UI text は英語**: Alt-Paper の editorial register に揃える ([[000831]] §6 + project memory の `ui-design.md`)。日本語 fallback は今フェーズで導入しない (i18n framework がない上、user 直接指示)。

**検討した代替案**:

- **`review_reason` を proto field として正準化**: 監査ログが綺麗になるが、現状 staleness / dismissed / reviewed の判定は `dismiss_state` + `surfaceBucket` から完全に決まるため、派生で十分。後日 `evidence_density` low (= "Thin evidence" reason) を入れるときに proto field 化を再検討する。
- **Compare を modal / 別 route として実装**: Alt-Paper の density discipline と Reduced Motion 制約 ([[000847]]) に反する。既存 redline = drawer の前提を維持し、CTA は metadata emit のみに絞る。

## Consequences

### Pros

- **mark_reviewed が UI 上で機能する**: Reviewed と archive が visibility 上で完全に分離。`docs/plan/knowledge-loop-completion-03-review-why-quality.md` の §1 / §5 受入基準を満たす。read path の `visibility_state='visible'` フィルタも矛盾なく動作する。
- **Why narrative の優先順位が canonical contract と一致**: TOPIC_AFFINITY / TAG_TRENDING が「過去に 1 回開いた」に潰されなくなり、active な recap-cluster connection が前面に出る。table-driven `TestOverrideWhy_PriorityV9Ladder` で priority ladder 全体を pin。
- **ChangeSummary の "What to update" が deterministic**: producer summary の有無に関わらず hint が決まり、reproject で byte-identical な JSONB が出る。LLM 経由の非決定性を回避。`composeChangeSummaryWithHint` は table-driven test で 7 ケース + bounded length + 5 回回す determinism check で覆っている。
- **proto / wire / pact 変更ゼロ**: contract gate を一切起動せず、consumer 側の再生成も不要。Phase 3 の作業量を BE 純粋関数 + FE 派生 helper のみに圧縮できた。
- **TDD outside-in 維持**: 6 commit すべてに先行する unit test がある。BE projector test は green、FE 1242 unit tests green、`go vet` / `go build` clean、biome lint / format clean (Phase 3 touch のみ)、E2E `loop-surface-planes.spec.ts` は Compare semantics に更新済み。
- **runbook の WhyMappingVersion history が連続**: v8 (本 ADR で過去抜け穴を補完) + v9 (本 ADR) を append 形式で記録、reproject 手順は標準 Procedure をそのまま流用できる。

### Cons / Tradeoffs

- **WhyMappingVersion v9 バンプ = full reproject が必要**: runbook §5 の Procedure を maintenance window で回す必要がある。本 ADR ではコード変更のみ commit し、reproject 実行は本番 deploy 後に手動で行う前提。post-roll の spot-check (`SELECT COUNT(*) ... WHERE why_mapping_version=9` ≈ 表示行数) を runbook に追記済み。
- **`review_reason` を FE 派生にしたため、analytics / observability で reason 別の分布を取りたくなった瞬間に proto field 化が必要**: 後続の Phase 5 (observability rollout) で必要性が出たら additive バンプで対応する想定。今フェーズでは over-engineering を避けた。
- **`mark_reviewed` 後に Surface Planner v2 が同じ entry を別 bucket に出す可能性**: dismiss_state=COMPLETED のままだが SurfaceScoreInputs は変わらないので bucket は維持される、と理解しているが、将来 v2 が `dismiss_state` を bucket decision に組み込むときに振る舞いを再確認する必要がある。`projector_test.go` の reproject determinism test がガードレールとして機能する。
- **「Thin evidence」reason は本フェーズでは出さない**: `evidence_density` は SurfaceScoreInputs に既に存在するが、`NullSurfaceScoreResolver` が常に 0 を返す状態のままなので「全行 thin」になり破綻する。Acolyte 統合 (Phase 4) で resolver が evidence_density を populate するようになってから FE 派生に追加する。
- **i18n framework は未導入**: Alt-Paper UI text は引き続き英語のみ。Phase 4 以降に必要が出れば別 ADR で扱う。
