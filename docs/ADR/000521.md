# ADR-000521: Tag Trail FetchRandomFeed の共有メモリ障害を解消し EXISTS クエリに最適化

## Status

Accepted

## Date

2026-03-21

## Affected Services

- alt-backend (Go) — `FetchRandomFeed` ドライバクエリの書き換え
- compose/db.yaml — PostgreSQL コンテナの `shm_size` 設定追加

## Context

Tag Trail の「New Random Feed」機能が **100% 失敗** していた。`FetchRandomFeed` RPC を呼ぶたびに以下のエラーが発生：

```
ERROR: could not resize shared memory segment "/PostgreSQL.XXXXXXXX"
       to 16777216 bytes: No space left on device (SQLSTATE 53100)
```

原因は2つの問題の複合である。

### 問題 1: Docker shm_size 未設定

PostgreSQL コンテナ（`compose/db.yaml`）に `shm_size` が設定されておらず、Docker のデフォルト 64MB が適用されていた。`postgresql.conf` では `shared_buffers = 256MB`、`work_mem = 16MB` を設定済みだが、動的共有メモリの割当上限がこれを下回っていた。

### 問題 2: FetchRandomFeed クエリの過剰な中間結果セット

従来のクエリは `feeds` × `articles` × `article_tags` の三重 INNER JOIN に `GROUP BY` + `ORDER BY RANDOM()` を適用していた：

```sql
SELECT f.id, f.title, f.description, f.link
FROM feeds f
INNER JOIN articles a ON a.feed_id = f.id
INNER JOIN article_tags at ON at.article_id = a.id
GROUP BY f.id, f.title, f.description, f.link
ORDER BY RANDOM()
LIMIT 1
```

テーブルサイズ：
- `feeds`: 73,533 行
- `articles`: 36,777 行
- `article_tags`: 673,793 行

JOIN により約 674K 行の中間結果が生成され、GROUP BY + RANDOM ソートで `work_mem`（16MB）の共有メモリセグメント割当を要求。`/dev/shm` の 64MB 上限に達して失敗していた。

## Decision

以下の2点を同時に修正する。

### 1. Docker `shm_size: "256m"` の追加

`compose/db.yaml` の `db` サービスに `shm_size: "256m"` を追加する。`shared_buffers = 256MB` に合わせた値であり、`mem_limit: 768m` の範囲内に収まる。

### 2. FetchRandomFeed クエリを EXISTS サブクエリに最適化

三重 JOIN + GROUP BY を廃止し、`feed_tags` テーブルに対する `EXISTS` サブクエリで置換する：

```sql
SELECT f.id, f.title, f.description, f.link
FROM feeds f
WHERE EXISTS (SELECT 1 FROM feed_tags ft WHERE ft.feed_id = f.id)
ORDER BY RANDOM()
LIMIT 1
```

**選定理由：**

- `feed_tags` テーブルには `idx_feed_tags_feed_id` インデックスが存在し、EXISTS はインデックスルックアップで短絡評価される
- `feed_tags` の行は `article_tags` と同一パイプライン（`UpsertArticleTags`）で原子的に作成されるため、「タグ付き記事を持つフィード」という意味論は等価
- 674K 行の JOIN 展開 → 73K 行の feeds スキャン + 27K 行の feed_id HashAggregate に削減
- GROUP BY が不要になり、ソート対象が大幅に縮小

**検討した代替案：**

| 案 | 評価 |
|---|---|
| CTE で `SELECT DISTINCT feed_id FROM feed_tags` | 572K 行のフルスキャンが必要。EXISTS のインデックス短絡評価より劣る |
| `TABLESAMPLE` | フィルタ条件（タグ有無）を適用できないため不適 |
| `OFFSET floor(random() * count)` | COUNT クエリとの2往復が必要。複雑さに見合う性能向上なし |

### 実測結果

修正後の EXPLAIN ANALYZE：

- **実行時間**: 93ms（修正前はエラーで完了不可）
- **ソートメモリ**: 26kB（top-N heapsort）（修正前は 16MB セグメント割当を要求して失敗）
- **Hash メモリ**: 1,530kB（feed_tags の feed_id HashAggregate）

## Consequences

### Pros

- Tag Trail の FetchRandomFeed が正常に動作するようになった（100% 失敗 → 正常）
- クエリのメモリ使用量が 16MB → 26kB に大幅削減
- `shm_size` 設定により、今後の他クエリでも共有メモリ不足が発生しにくくなった
- 返却カラム・型が完全に同一のため、Port / Gateway / Usecase / Handler レイヤーの変更が不要

### Cons / Tradeoffs

- `feed_tags` に行が存在するが対応する `article_tags` がすべて削除されたケース（孤立した feed_tag）では、タグなしフィードが選ばれる可能性がある。ただし下流の Handler は fail-open 設計（タグなしでも latestArticleId でフォールバック）のため実害なし
- `shm_size: 256m` は `mem_limit: 768m` の約 1/3 を占めるが、PostgreSQL のワーキングメモリとしては妥当な配分
