# FetchArticleContent スノーボールレイテンシの修正 + PgBouncerヘルスチェック修正

## ステータス

Accepted

## コンテキスト

Visual Previewモード（`/sv/feeds/swipe/visual-preview`）で、未キャッシュ記事のページ応答が異常に遅くなる事象が報告された。Nginxログでは `FetchArticleContent` のレスポンスタイムが2分以内に1.7秒→57秒へエスカレーションしていた。

### 根本原因

Visual Previewモードでは最大10記事を500ms間隔でプリフェッチする。記事がDBにキャッシュされていない場合、alt-backendが外部サイトから同期的にHTMLを取得する。10件が10の異なるホストに対して同時に実行されると:

1. **グローバル同時実行数の制限がない**: `HostRateLimiter` はホスト単位の流量制御のみ。10ホスト同時アクセスは制限されない
2. **個別フェッチのタイムアウトがない**: HTTPクライアントのデフォルト30秒 + rate limiter待ち時間で60秒超になりうる
3. **RAG upsertが同期的**: ステップ8のRAG連携は既にfail-safe（エラーはログのみ）だが、100-500msの同期レイテンシが毎リクエストに加算される

結果、後続リクエストが先行リクエストの完了を待つ形でキューイングが発生し、レイテンシがスノーボール的に増大する。

### 副次的な問題

ADR-328で `pg_isready` に `-U ${DB_USER}` を追加したが、`-d` フラグが未指定のまま残っていた。`pg_isready` はデフォルトでユーザー名と同名のデータベース（`alt_appuser`）に接続を試みるため、10分あたり120件のスプリアスなログインログが発生していた。

## 決定事項

### 修正1a: ゲートウェイ層にグローバル同時実行セマフォを追加

`FetchArticleGateway` にバッファ付きチャネル（容量3）をカウンティングセマフォとして追加。`FetchArticleContents` の先頭で `select` によりスロットを獲得し、コンテキストキャンセル時は即座に `ctx.Err()` を返す。

```go
type FetchArticleGateway struct {
    rateLimiter   *rate_limiter.HostRateLimiter
    httpClient    *http.Client
    ssrfValidator *security.SSRFValidator
    fetchSem      chan struct{} // max 3 concurrent external fetches
}
```

**セマフォ容量を3にした理由:**
- 10同時→3同時にすることで、リソース競合とTCPソケット枯渇を防止
- 3並列であれば外部フェッチのスループットは十分維持できる
- `golang.org/x/sync/semaphore.Weighted` の代わりにチャネルを採用した理由: 依存追加なし、コードが単純、`select` によるコンテキスト統合が自然

**セマフォの配置をゲートウェイ層にした理由:**
- 外部HTTPフェッチという「インフラ境界の制御」はClean Architectureにおいてゲートウェイ層の責務
- ユースケース層に配置するとポートインターフェースの変更が必要になり、影響範囲が広がる

### 修正1b: ユースケース層にWebフェッチタイムアウトを追加

`FetchCompliantArticle` のステップ4（外部フェッチ）を `context.WithTimeout(ctx, 15*time.Second)` でラップ。サーバーレベルのタイムアウト（300秒）に依存せず、個別フェッチに15秒のハードキャップを設ける。

**15秒にした理由:**
- 大半の外部サイトは5秒以内に応答する
- 15秒あれば遅いサイトにも十分な猶予がある
- セマフォ待ち（最大3スロット×15秒）を考慮しても、リクエスト全体が45秒を超えることはない

### 修正1c: RAG upsertの非同期化

ステップ8の `ragIntegration.UpsertArticle()` をgoroutineで実行。`context.Background()` + 30秒タイムアウトを使用し、呼び出し元のレスポンスに影響を与えない。

**安全性の根拠:**
- 既存コードでRAGエラーはログのみ（リクエスト失敗にしない）であり、非同期化は振る舞いの変更ではない
- DB保存（ステップ7）完了後に実行されるため、データ一貫性に影響なし
- `context.Background()` を使用するため、クライアント切断後もRAG upsertは完了する

### 修正2: PgBouncerヘルスチェックのデータベース名指定

`pg_isready` コマンドに `-d ${POSTGRES_DB}` を追加。デフォルトのユーザー名ベースのDB名推定を回避する。

## 結果・効果

### メリット

- **スノーボールレイテンシの解消**: 最大3並列に制限されることで、10記事プリフェッチ時の最悪ケースが57秒→約20秒（理論値）に改善。実際にはDBキャッシュのヒットにより大幅に短縮される
- **個別フェッチの上限保証**: 15秒タイムアウトにより、遅いサイトが他のリクエストをブロックし続けることを防止
- **レスポンスレイテンシの削減**: RAG非同期化により、毎リクエスト100-500msの同期待ちが解消
- **PgBouncerログノイズの削減**: `db=alt_appuser` のスプリアスログインが停止

### デメリット・トレードオフ

- **セマフォによるスループット制限**: 同時3フェッチの制限により、10記事全てが未キャッシュの場合は直列化が発生する。ただしVisual Previewの500msスタガリングと組み合わせると、実際に3スロットが同時に埋まるケースは稀
- **RAG upsert失敗の検知遅延**: 非同期化によりgoroutineリークのリスクが理論上存在するが、30秒タイムアウトで保護されている
- **15秒タイムアウトのハードコード**: 現在は定数値。将来的に環境変数で設定可能にする余地がある

### レイテンシ改善の見積もり

| シナリオ | Before | After |
|---------|--------|-------|
| 10記事全て未キャッシュ（最悪ケース） | 57秒 | ~20秒 |
| 5記事未キャッシュ | ~30秒 | ~10秒 |
| 全記事キャッシュ済み | <100ms | <100ms（変更なし） |
| 個別フェッチ（遅いサイト） | 最大30秒+α | 最大15秒 |

### セマフォの動作フロー

```
リクエスト1 → [セマフォ獲得] → 外部フェッチ → [セマフォ解放]
リクエスト2 → [セマフォ獲得] → 外部フェッチ → [セマフォ解放]
リクエスト3 → [セマフォ獲得] → 外部フェッチ → [セマフォ解放]
リクエスト4 → [セマフォ待ち...] → (1-3のいずれかが完了) → [獲得] → フェッチ
```
