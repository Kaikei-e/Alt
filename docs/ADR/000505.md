# ADR-000505: RSS 登録成功時に feed health を即時初期化する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-backend (Go)

## Context

RSS feed 登録では、ハンドラ内で URL 検証と RSS fetch が成功したあとに `feed_links` と `feeds` を永続化していた。一方で feed の health 表示は `feed_link_availability` の有無に依存しており、この行が存在しない feed は `unknown` と判定される設計だった。

問題は、登録成功時点では RSS fetch に成功しているにもかかわらず、health 初期化が background collector にしか存在しなかったことにある。collector は起動直後に 1 回、その後は定周期で動くため、登録直後の一覧では新規 feed が `unknown` になる時間窓が発生していた。

実装上は「登録成功」と「health 行作成」が別経路になっており、ユーザーからは正常登録後にも未評価に見える状態だった。

## Decision

**RSS 登録成功時に `feed_link_availability` を即時初期化し、新規 feed を `unknown` にしない。**

今回の決定では以下を採用する。

- `RegisterFeedsUsecase` に availability 初期化依存を追加する
- feed item 永続化成功後に `ResetFeedLinkFailures()` を呼び、初期 healthy 状態を確定する
- availability 初期化に失敗した場合は登録処理全体を失敗として扱う
- background collector の reset / increment / disable 経路はそのまま維持する

### 代替案

| 案 | 評価 |
|---|---|
| `unknown` を維持し UI 文言だけ変える | 登録成功と health 表示の乖離が残るため不採用 |
| collector 実行後にのみ `healthy` へ遷移させる | 登録直後の一貫性が取れず、待ち時間も長いため不採用 |
| 新しい `pending` 状態を追加する | 将来的には可能だが、今回の主問題は初期化欠如なので過剰 |

## Consequences

### Pros

- 登録成功直後に feed health が `healthy` で安定する
- ハンドラの成功レスポンスと一覧表示の整合が取れる
- background collector を待たずに初回状態を確定できる

### Cons / Tradeoffs

- availability 初期化失敗が登録全体の失敗条件に加わる
- 登録 path が availability テーブルにも依存するため、責務はやや増える
- 初回 fetch 成功を即 `healthy` と見なす前提を ADR とテストで維持する必要がある
