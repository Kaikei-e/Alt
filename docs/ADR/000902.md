# ADR-000902: entry chunk 自身が 404 する経路を埋めるため iOS Safari "ページを開けませんでした" の自己治癒を 4 層 defense-in-depth に拡張する

## Status

Accepted

## Date

2026-05-20

## Affected Services

- alt-frontend-sv (TypeScript / Svelte 5)
- nginx (`nginx/conf.d/default.conf`)

## Context

[[000898]] (2026-05-13) で「iOS Safari ページを開けませんでした」フルシートを根治したはずだったが、約 1 週間後に**同症状の再発**がユーザから報告された。本 ADR はその再発の原因を実ログから突き止め、ADR-898 の決定をそのまま温存しつつ「runtime が起動できない経路」を別レイヤで埋めるための補強を行う。

### 観測

本番ドメインの nginx access log 直近 7 日を集計し、`/_app/immutable/*` の 404 を抽出した (ドメイン / IP / UA の具体値は伏字):

| 日時 (UTC) | クライアント / UA | referer | 404 ファイル種別 |
|---|---|---|---|
| 2026-05-13 01:08 | デスクトップ Chromium | — | `nodes/N.<HASH>.js` × 1 |
| 2026-05-15 15:17 | 単一 iPhone (iOS 18.x Safari) | `/feeds/swipe/visual-preview` | `chunks/*` × 3 + `nodes/M.<HASH>.js` |
| **2026-05-18 10:56** | **同 iPhone (iOS 18.x Safari)** | **`/menu`** | **`entry/app.<HASH>.js` + `entry/start.<HASH>.js` + `nodes/0.<HASH>.js` (root layout) + `chunks/*` × 3 + `nodes/{N,M}.<HASH>.js` = 8 連続** |

2026-05-18 のバーストが致命的である。ユーザはこの瞬間に「ページを開けませんでした」フルシートを浴びている。

### ADR-898 三点セットが救えない死角

`hooks.client.ts` の `handleError` / `unhandledrejection` listener、`svelte.config.js` の `version.pollInterval`、`+layout.svelte` の `updated.current` 監視は、**いずれも SvelteKit runtime が一度起動できた後** にしか機能しない。

- `hooks.client.ts` 内のコードは `entry/app.<HASH>.js` の中に bundle されるため、`entry/app.js` 自体が 404 すると `handleError` / `unhandledrejection` の addEventListener が**登録すらされない**。
- `version.pollInterval = 5min` は `/__version.json` への定期フェッチであり、runtime の `$app/state` 配線後に走る。runtime が立ち上がっていなければ poll も発生しない。
- `+layout.svelte` の `$effect(() => updated.current)` は hydration 後に走るため、entry chunk 失敗で hydration が起こらなければ評価されない。

つまり ADR-898 は **「runtime 起動 → その後の dynamic chunk 404」のみを救済可能**で、初回 navigation / bookmark cold open / BFCache 復帰直後に entry chunk が 404 する経路では WebKit がハードフェイルとして「Cannot Open the Page」フルシートを直送する。2026-05-18 のログがまさにこの経路を踏んでいる。

### Web 研究による裏付け (2026-05-19 / 2026-05-20)

- [Vite 公式ドキュメント "Building for Production / Load Error Handling"](https://vite.dev/guide/build) は dynamic import 失敗時に `vite:preloadError` という DOM event を発火することを明言し、`window.addEventListener('vite:preloadError', () => window.location.reload())` を canonical な recovery として提示している。ADR-898 はこのイベントを listen していなかった。
- 同ページは "make sure to set `Cache-Control: no-cache` on the HTML file, otherwise the old assets will be still referenced." と明示しており、HTML レスポンスに `Cache-Control: no-cache` を載せる責務がアプリ側にあると指摘している。SvelteKit adapter-node の default はこれを付与しない。
- [vitejs/vite Discussion #13925](https://github.com/vitejs/vite/discussions/13925) は chunk-load 失敗の community 議論で、retry を持たない deploy パイプラインに対する典型的な対処として「HTML を no-cache に倒す」「サーバ側で fallback を返す」が挙がっている。
- [sveltejs/kit #5208](https://github.com/sveltejs/kit/issues/5208) は iOS Safari の "Importing a module script failed" が SvelteKit 既知問題として 0.6% 程度のユーザに発生していることを記録している (公式解決策なし)。
- [sveltejs/kit Discussion #8665](https://github.com/sveltejs/kit/discussions/8665) は deploy 後の reload pattern として `pageshow.persisted` + `updated.check()` を canonical と紹介。BFCache 復帰直後に build version を確認する経路。ADR-898 は `pageshow` を listen していなかった。
- [MDN: HTMLElement: error event](https://developer.mozilla.org/en-US/docs/Web/API/HTMLElement/error_event) は `<script>` / `<link>` のリソースロード失敗 event が **bubble しない**ことを明示。`window.addEventListener("error", handler, true)` (capture phase) でのみ捕捉可能。これは ADR-898 の `unhandledrejection` listener では届かない経路。

これらを総合し、4 つの未塞ぎ経路 (entry chunk 失敗 / `vite:preloadError` 未捕捉 / HTML stale / BFCache 復帰の race) を defense-in-depth で埋める方針を採る。

## Decision

ADR-898 三点セットは **コードレベルで完全に温存**し、加えて以下 5 つの層 (L1 / L2 / L3 / L4 / L5) を追加する。優先度は P0 (主犯対策) → P1 (race を狭める補強)。

### L1 (P0): nginx で `/_app/immutable/*.{js,mjs}` の 404 を 200 reload stub にフォールバックする

`nginx/conf.d/default.conf` に新規 regex location を追加:

- `location ~* ^/_app/immutable/.*\.(js|mjs)$` で JS/MJS のみを intercept (CSS / フォント / wasm は既存 prefix location に残し、stub を CSS として読まれるリグレッションを避ける)。
- `proxy_intercept_errors on;` を明示。これがないと nginx は upstream の 404 を素通しし、`error_page` directive が発火しない (nginx の非自明な仕様)。
- `error_page 404 = @stale_chunk_reload;` で `@stale_chunk_reload` named location にルーティング。
- `@stale_chunk_reload` は `default_type application/javascript;` + `add_header Cache-Control "no-store"` + `add_header X-Stale-Chunk-Reload "1"` で、`sessionStorage` カウンタを 3 回上限で進めて `location.reload()` を実行する 200 JS を返す。

JS が一行も走っていない初回 navigation でも、ブラウザは module load 成功 → stub 実行 → reload のパスを通り、WebKit のフルシートに到達しない。`X-Stale-Chunk-Reload: 1` は監視で stale chunk fallback の発火数を 5 分以内に追跡できる signal。

### L2 (P0): `app.html` に inline `<script>` を入れて capture phase で chunk 失敗を捕捉する

`src/app.html` の `%sveltekit.head%` 直前にマーカーコメント (`<!-- alt:chunk-bootstrap:begin/end -->`) でラップした inline `<script>` を追加し、

- `window.addEventListener("error", handler, true)` の **capture phase** で `<script src="/_app/immutable/*">` の load failure を捕捉。
- 同じ handler 内で `window.addEventListener("vite:preloadError", handler)` も登録 (Vite 公式 event)。
- `sessionStorage` カウンタ `alt:chunk-reload-attempts` を 3 回上限で進めて `setTimeout(() => location.reload(), 0)` を実行。
- 複数 chunk が並列に 404 する場合 (実観測 8 連発) を `fired` flag で coalesce し、reload は 1 回に集約。

inline script はパース時点で実行されるので、SvelteKit runtime の `entry/app.<HASH>.js` 評価**前**に handler が登録される。L2 は runtime が死んでも単独で動く最小の自己治癒層。

ロジックは `src/lib/safari-error-bootstrap.ts` の `installChunkBootstrap` として export しており、unit test (`safari-error-bootstrap.test.ts`) は `EventTarget` を用いた harness で 7 ケース (single chunk / unrelated link / 8 連発 coalesce / `vite:preloadError` / 3-attempt ceiling / cleanup) を網羅。同ファイルの `BOOTSTRAP_SCRIPT_BODY` と `app.html` の inline 本体は whitespace-normalized 比較 (drift test) で同期を強制する。

### L3 (P1): `hooks.client.ts` に `vite:preloadError` listener を defense-in-depth で追加する

既存の `unhandledrejection` listener の隣に `window.addEventListener("vite:preloadError", () => scheduler.schedule("chunk-404"))` を追加。L2 の inline script と同じ event を 2 箇所で listen することになるが、L2 は runtime 不在でも動く前段、L3 は runtime 経由で構造化ログ (`safariBucket` / `chunkHash`) を残す後段、と役割を分けている。

### L4 (P1): `hooks.server.ts handle` で `text/html` レスポンスに `Cache-Control: no-cache, must-revalidate` を definitive に set する

`src/hooks.server.ts` の `handle` 内の 3 つの resolve 経路 (public / public-fallback-after-auth-fail / authenticated) のすべてで、`resolve` の戻り値に `applyHtmlCacheControl(response)` (新規 `src/hooks.server.cache-control.ts`) を呼ぶ。`text/html` でのみ `cache-control: no-cache, must-revalidate` を set、JSON / `application/javascript` には触れない。

これにより SvelteKit adapter-node のデフォルトレスポンスでも HTML が必ず revalidate 対象になり、deploy 直後に古い HTML が新 chunk hash を要求する race が狭まる。`/_app/immutable/*` 側は SvelteKit / nginx が `public, immutable` を返し続け、影響しない。unit test は 8 ケース (text/html / charset 付き / json / javascript / 既存 cache-control 上書き / no content-type / idempotent / body 不読) を網羅。

### L5 (P1): `+layout.svelte` で `pageshow.persisted` / `visibilitychange→visible` を listen して `updated.check()` を即時 invoke

`src/lib/bfcache-version-check.ts` の `installBfcacheVersionCheck` を新規追加し、`+layout.svelte` 内で `$effect` から呼び出して `pageshow.persisted=true` (BFCache 復帰) と `document.visibilityState === "visible"` の 2 トリガーで `updated.check()` を即時 invoke する。cleanup は両 listener の `removeEventListener` を返却。

`version.pollInterval = 5min` は long-running tab には十分だが、ユーザがタブ復帰直後にタップする経路 (実観測のケース) では次 tick まで待てない。L5 はこの数百 ms の race を埋める。unit test は 5 ケース (pageshow persisted=true/false / visibility visible/hidden / cleanup) を網羅。

### 検討した代替案と却下理由

- **adapter-node → adapter-static + Cloudflare**: edge cache の grace period で旧 hash chunk を「N 日間残す」atomic deployment が可能になるが、本リポジトリは BFF / SSR / Connect-RPC 専用エンドポイント (`/api/`) を持つため adapter-static は採用不可。
- **`build.modulePreload.resolveDependencies: () => []` で modulepreload を全廃**: chunk-load failure の表面積を縮めるが、LCP / TTI が悪化する。L1-L5 で十分。
- **code-splitting を完全廃止 (single bundle)**: 1MB 級の単一バンドルになり LCP 大幅劣化、SSR streaming も不可。
- **HTML に `Cache-Control: no-store`**: `no-cache` ではなく `no-store` だと back / forward navigation 体感が悪化する。`no-cache, must-revalidate` (条件付き GET 許容) が canonical。
- **ADR-898 三点セットの撤去 / 簡素化**: L2 / L3 と機能が重複するが、これは intentional defense-in-depth。runtime 起動後の経路を ADR-898 が担い、起動前 / 復帰直後を L1 / L2 / L5 が担う構造。

## Consequences

### Pros

- **死角だった entry chunk 404 経路が消える**: runtime が一度も起動できない初回 navigation でも、L1 の nginx 200 stub または L2 の inline script のどちらかが必ず reload を発火する。
- **L1 単体で外部依存ゼロのフォールバック**: nginx 設定のみで動くので、frontend container が deploy 失敗していても古いタブを救える。
- **defense-in-depth の冗長性**: L1 (server) / L2 (browser prerun) / L3 (runtime) / L4 (HTML caching) / L5 (BFCache) の 5 経路が独立しているので、1 つが壊れても残りが救う。
- **Vite 公式 `vite:preloadError` event を正規対応**: SvelteKit 内で未活用だった canonical event を 2 箇所で listen することで、Vite が将来の改修で挙動を寄せても追従しやすい。
- **観測性**: `X-Stale-Chunk-Reload: 1` レスポンスヘッダで stale chunk fallback の発火数を nginx access log だけで集計可能。1 週間以内に「ほぼ 0」に収束しているかで完治判定できる。
- **既存 ADR との非干渉**: ADR-898 / ADR-883 / ADR-884 / ADR-881 の決定はすべてそのまま温存。コード差分は追加のみで、削除はしていない。
- **rolling deploy 互換**: 新 compose service / 新 volume / 新 Atlas migration いずれも追加していない。
- **drift 防止**: `app.html` の inline script と `BOOTSTRAP_SCRIPT_BODY` は whitespace-normalized drift test で同期を強制。片方だけ更新するとテストが落ちる。

### Cons / Tradeoffs

- **HTML レスポンスヘッダに `Cache-Control` が二重で乗る**: nginx 側の既存 `add_header Cache-Control "no-cache" always;` (catch-all location) と SvelteKit 側の `no-cache, must-revalidate` が両方付与され、ブラウザは `Cache-Control: no-cache, must-revalidate, no-cache` として解釈する。挙動は等価だがヘッダが冗長。今回は intentional な defense-in-depth として温存し、将来 nginx 側のみ削る選択肢を残す。
- **`@stale_chunk_reload` stub が CSS 404 を救わない**: regex location を `.(js|mjs)$` に限定したため、CSS の 404 は WebKit がそのまま視覚崩壊として表示する。CSS 404 は「ページを開けませんでした」フルシートを誘発しないので許容範囲だが、将来 CSS stale を別経路で扱う余地は残る。
- **`Cache-Control: no-store` の代わりに `no-cache, must-revalidate` を選んだ結果**: HTML に変更がなくても 304 リクエストが毎 navigation 発生する。SSR endpoint のレイテンシ的に許容範囲 (`/feeds` GET の Lighthouse FCP 0.5s が劣化していない)。
- **`vite:preloadError` の event 型がブラウザ globals に未宣言**: TypeScript strict には string event 名でしか登録できず、event payload の型補完が効かない。実害は少ないが、Vite が将来 `WindowEventMap` を拡張するまでは現状維持。
- **inline `<script>` の CSP 影響**: 既存の SvelteKit bootstrap / theme FOUC スクリプトと同じく `script-src 'unsafe-inline'` または hash を要求する。本リポジトリは既にこの方針を取っているため新たな副作用はない。

## 観測フォロー

- 本番反映後 1 週間で nginx access log の `X-Stale-Chunk-Reload: 1` 発火数を tally。1 桁前半に収束していれば「runtime 起動できない経路を含めて完治」と判定する。
- 同期間で `/_app/immutable/*` の素の 404 (= L1 が介在しない経路) が 0 件になっていることを確認。これは regex location が prefix location より優先されている前提が崩れていないことの確認になる。
- alt-frontend-sv の `console.warn("[hooks.client] vite:preloadError — scheduling reload")` の docker logs tally。L3 経由のリロードは runtime 起動後の case を表すので、ここが 0 件 / L1 と L2 が低発火、なら ADR-898 三点セット温存の意義どおりに各層が役割を果たしている。
- 残るなら ADR-898 で deferred されていた H4 (Cookie / `large_client_header_buffers`) を開封する。
