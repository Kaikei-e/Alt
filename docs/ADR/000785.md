# ADR-000785: auth-hub に Hurl E2E スイートを追加し段階的全面展開の第 6 増分に載せる

## Status

Accepted

## Date

2026-04-19

## Affected Services

- auth-hub (Go) — 黒箱 E2E が CI に入るのみ。プロダクションコードは無変更
- `compose/compose.staging.yaml` — `auth-hub` profile を新設
- `.github/workflows/e2e-hurl.yml` — `auth-hub` job と path フィルタを追加
- `e2e/hurl/auth-hub/` — 新規ディレクトリ (00-setup + 12 scenarios + run.sh + README)
- `e2e/fixtures/auth-hub/` — 新規ディレクトリ (Kratos staging config + identity + db-init)
- `e2e/fixtures/staging-secrets/` — 3 ファイル追加 (CSRF / Kratos cookie / Kratos cipher)

## Context

[[000763]] で Hurl を E2E 統一フレームワークとして導入して以降、段階的に各サービスへスイートを展開してきた:

- Increment 1: [[000763]] フレームワーク整備 (self-hosted runner + Pact ゲート)
- Increment 2: [[000764]] mq-hub (Connect-RPC over HTTP/1.1+JSON)
- Increment 3: [[000765]] knowledge-sovereign (DB-backed + `--jobs 1` serial)
- Increment 4: [[000766]] `run.sh` dispatch 規約の共通化
- Increment 5: [[000778]] alt-backend (REST + CSRF + Connect-RPC 混在)

この流れで次に残っていたのが `auth-hub` — Ory Kratos v1.3.0 の前段に立つ Identity-Aware Proxy (Go 1.26, Echo v4, port 8888) である。ルートは以下の 5 本:

| method | path | auth | 目的 |
|---|---|---|---|
| GET | /health | none | liveness |
| GET | /validate | cookie `ory_kratos_session` | nginx `auth_request` が叩く。identity を 4 つのヘッダ + HS256 JWT で返す |
| GET | /session | cookie `ory_kratos_session` | frontend 向け JSON `{ok, user, session}` |
| POST | /csrf | cookie `ory_kratos_session` | CSRF double-submit token の発行 |
| GET | /internal/system-user | `X-Internal-Auth` shared secret | サービス間で system user id を取得 |

[[000780]] では alt-backend 側が auth-hub を「pre-minted HS256 JWT + fixture」で staging からバイパスした。つまり alt-backend 視点では auth-hub を stub する設計が正解だった。ところが **今回はその逆、auth-hub 自身をテスト対象にする suite** を書く必要がある。

ここで検討すべき第一の問いは「Kratos 自体を stub できるか」。答えは NO だった。auth-hub は `whoami` / `list-identities` など Kratos Admin/Frontend API への薄いラッパであり、Kratos を stub すると **auth-hub のほぼ全ロジックを自分で再実装した偽 Kratos** を E2E で叩くだけになる。Pact CDC が contract shape を別レーンで既に検証している今、Hurl が保証すべきは「auth-hub が real Kratos と連携して HTTP surface を正しく返す」ことであり、real Kratos なしでは意味がない。

第二の問い: auth-hub は自身の DB を持たないが、Kratos は kratos-db (Postgres 16) を必要とする。[[000781]] の NOLOGIN role pre-create パターン、[[000765]] の DB + migrator 起動順序、[[000778]] のスライス構成が使えるか。答えは YES で、これらを組み合わせれば Kratos v1.3.0 の schema migration を Atlas ではなく `kratos migrate sql -e --yes` で流し込める。

## Decision

**`e2e/hurl/auth-hub/` を新設し、real Kratos + real kratos-db を立ち上げた上で auth-hub の 5 ルート × 12 シナリオを回す E2E スイートを追加する。** CI も [[000778]] 同様の path-gated ジョブで回す。

### 構成

1. **compose profile `auth-hub` を `compose/compose.staging.yaml` に追加**。4 サービス構成:
   - `auth-hub-db` — `postgres:16-alpine`, `POSTGRES_DB=kratos`, `/docker-entrypoint-initdb.d` に `01-no-op.sql` のみ (Kratos が schema を自力で所有するため NOLOGIN role 追加は不要。フックは将来の seed SQL 用に残す)
   - `auth-hub-db-migrator` — `oryd/kratos:v1.3.0` で `kratos migrate sql -e --yes` を走らせて exit。[[000781]] 同様 `restart: "no"` + `service_completed_successfully` でゲート
   - `kratos` — `oryd/kratos:v1.3.0` + `--dev` (staging network は `internal: true` で HTTPS 外乱なし)。`kratos/entrypoint.sh` を read-only mount で流用し `${KRATOS_COOKIE_SECRET}` / `${KRATOS_CIPHER_SECRET}` の sed 置換を継承
   - `auth-hub` — `auth-hub/Dockerfile` を local build。`BACKEND_TOKEN_SECRET_FILE=/run/secrets/alt_backend_token_secret` で [[000780]] と **同じ** HS256 secret を共有

2. **Kratos staging config は `e2e/fixtures/auth-hub/kratos/kratos.yml` に fork する**。本番 `/home/koko/Documents/dev/Alt/kratos/kratos.yml` は `curionoah.com` 系 URL と `.curionoah.com` cookie domain をハードコードしており container-local な staging network では機能しない。別 ADR [[000786]] で fork の決定を記録する。

3. **セッション取得は Kratos self-service browser flow**。Kratos v1.3 は admin session mint API を持たず、api-flow が返す `session_token` は `ory_kratos_session` cookie value として使えない (cookie auth は HMAC 署名・token auth は別経路)。詳細と経緯は別 ADR [[000787]] で記録する。

4. **Hurl 実行は 2-phase**: まず `00-setup.hurl` を単独で `--report-json` 付きで回し、cookie / user_id / session_id を extract。次に extract した値を `--variable` / `--secret` に乗せて本 suite (12 ファイル) を `--jobs 1` で走らせる。Hurl 7.1 の `--test` では captures が file-scoped であり、単一呼び出し内でも cross-file には流れない点を回避する。この運用は別 ADR [[000788]] で固定化する。

5. **シナリオ 12 本**: `01-health`, `02-validate-happy`, `03-validate-missing-cookie`, `04-validate-bad-cookie`, `05-session-happy`, `06-session-unauth`, `07-csrf-happy`, `08-csrf-unauth`, `09-internal-system-user-happy` (+ retry buffer で cold start 吸収), `10-internal-system-user-missing-auth` (401), `11-internal-system-user-bad-token` (**403** — `InternalAuth` middleware は missing と invalid を区別する), `12-validate-jwt-shape` (`X-Alt-Backend-Token` を base64UrlSafeDecode して `iss`/`aud`/`sub`/`exp` を文字列 contains で検証)。

6. **Secrets 共有**: `alt_backend_token_secret.txt` を auth-hub でも `BACKEND_TOKEN_SECRET` として使う。auth-hub が発行する JWT を alt-backend 側が同じ secret で検証するプロダクションの HS256 共有と一致する。新設したのは CSRF 用 / Kratos cookie 用 / Kratos cipher 用 (**32 バイト固定** — xchacha20-poly1305 要件) の 3 ファイルのみ。

7. **CI job** `auth-hub` を `.github/workflows/e2e-hurl.yml` に追加。`acolyte-orchestrator` job をテンプレートとして `auth-hub/Dockerfile` を ci タグで build → Hurl image pull → `bash e2e/hurl/auth-hub/run.sh` → 失敗時ログ dump (auth-hub / auth-hub-db / auth-hub-db-migrator / kratos) → レポートアーティファクト → teardown、の流れ。path フィルタに `auth-hub/**`, `e2e/hurl/auth-hub/**`, `e2e/fixtures/auth-hub/**`, `kratos/entrypoint.sh` を追加。

8. **範囲外**: rate-limit scenario (429 / `Retry-After`) は `auth-hub/middleware/rate_limit_test.go` 側のユニットテストが責務。cookie flag assertion (`HttpOnly` / `Secure` / `SameSite`) は production Playwright suite に委ねる — staging network は `internal: true` で plain HTTP、`--dev` が `Secure` を緩めるため staging での assertion は本番と乖離する。

### 命名衝突の扱い

`compose.staging.yaml:706` 付近で `alt-backend-deps-stub` が Docker network alias として `auth-hub` を既に使っている。しかしこれは `profiles: [alt-backend]` でのみ materialize し、`render-slice.sh` は run ごとに **単一 profile のみ** activate するため、`--profile auth-hub` 実行時は alt-backend-deps-stub 自体が起動しない → alias も存在しない。衝突はない。

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| Kratos を FastAPI stub で模倣する ([[000779]] パターン) | auth-hub のほぼ全責務が Kratos 連携なので、stub しても「auth-hub が request-forwarding proxy として動くか」しか分からない。Pact CDC が contract shape を取っている今、Hurl の価値は real Kratos 連携の振る舞い確認に絞るべき |
| PgBouncer (`pgbouncer-kratos`) を挟む | 本番 auth profile には存在するが、staging の cold-start surface を 1 つ増やす割に検証できる差分がない。PgBouncer 固有の `prepared_statements=false` 挙動は本番 integration で追う |
| Kratos cookie secret を env 変数 `SECRETS_COOKIE` で直接渡す | Kratos 本体は `${VAR}` expansion を native にはやらない。`kratos/entrypoint.sh` が sed で YAML を書き換える既存運用を使えば、本番と同じ code path を再利用できる |
| prod の `kratos/kratos.yml` を staging でも使う | `curionoah.com` base URL と `.curionoah.com` cookie domain が container-local flow を壊す。詳細は [[000786]] |
| api-flow で得た `session_token` を `Cookie: ory_kratos_session=<token>` として送る | self-hosted Kratos は cookie value を HMAC 署名 + DB lookup で検証する。api-flow token は別経路 (`X-Session-Token` / `Authorization: Bearer`) 用で、cookie 値として送っても whoami が 401 を返す |
| auth-hub に `X-Session-Token` 受け入れを追加 | プロダクションコードの挙動変更を E2E のために入れるのは筋が悪い。browser flow で cookie を取れば既存コードのまま通る |
| `auth-hub` alias と service 名の衝突を避けるため service 名を `auth-hub-svc` 等にリネーム | 他の staging service は実サービス名をそのまま使う慣例がある (`alt-backend`, `acolyte-orchestrator`, `mq-hub`)。profile 分離で衝突しない以上、命名一貫性を優先 |
| rate-limit scenario (429) を suite に含める | 壁時計依存でフレークしやすく、コンテナ cold-start の遅延と混ざる。ユニットテストで決定論的に検証する方が適切 |
| cookie flag (HttpOnly/Secure/SameSite) assertion を Hurl で入れる | staging は plain HTTP + `--dev` で本番とフラグが一致しない。Playwright で HTTPS stack を叩く方が意味のある assertion になる |
| `--jobs 1` ではなく parallel で回す | 内部レート制限 (10 req/min, burst 3, 非 env-tunable) で 09/10/11 が競合する。順序依存を明示するほうが CI ログが読める |

### 範囲外 (今回やらないこと)

- `auth-hub/config/config.go` に `INTERNAL_RATE_LIMIT` env を追加 — 本 ADR はプロダクション未変更を維持
- Kratos の verification / recovery / TOTP flow — staging config で無効化している
- auth-hub の mTLS 経路 (`MTLS_LISTEN=true`, port 9443) — 本 suite は plain HTTP 専用

## Consequences

### Pros

- **auth-hub の 5 HTTP route すべてが real Kratos と一緒に緑になることが CI で保証される**。Hurl 12 シナリオは 1 秒未満で走る (2-phase seed 込みで ~20 秒、コンテナ bring-up 込みで ~60–90 秒)
- **本番 HS256 共有構造が staging にそのまま再現される**。auth-hub が issue した JWT を alt-backend 側が `alt_backend_token_secret.txt` で検証できる — 片方だけ検証しても気付けない「secret drift」が staging で先に落ちる
- **再利用パターンが 6 サービス目で確立**。[[000765]] `--jobs 1` + [[000766]] run.sh + [[000781]] DB init + [[000778]] profile-gated suite + [[000780]] 共有 HS256 secret の 5 パターンを 1 つの ADR が組み合わせて使う形が見える
- **Kratos v1.3.0 のバージョン上げ時の blast radius が可視化される**。browser flow の response shape や session cookie 仕様が変わると、本スイートが落ちて気付ける
- **`kratos/entrypoint.sh` の二重運用が回避される**。prod と staging で同じ entrypoint を bind mount するので、sed substitution の挙動差分が発生しない

### Cons / Tradeoffs

- **Kratos staging config を fork したため prod の kratos.yml との drift が発生する**。verification 無効化など意図的な差は問題ないが、identity schema 変更など共通すべき要素は手で sync する必要がある。sync 漏れを防ぐ仕組みは未整備 ([[000786]] の Cons)
- **内部 rate limit が非 env-tunable なので 09/10/11 のシナリオ順序に暗黙の契約が生じる**。`--jobs 1` で serial 実行する限り問題ないが、将来並列化を検討する場合 config 変更が先に必要
- **Kratos v1.3.0 container image を CI で pull する**。初回は数秒の network 待ち。pre-pull step を入れれば緩和できるが、現時点では必要性が低いので省略
- **CI 所要時間がひとつ増える**。`acolyte-orchestrator` job と合わせて matrix が 6 本になり、PR ごとの CI 時間が path フィルタ未マッチでも background で進む。self-hosted deploy runner の並列数で吸収できる範囲
- **E2E 失敗時の debug が Kratos + auth-hub の 2 container を跨ぐ**。failure hook の log dump は両方を拾うが、どちらの責任か切り分けるには Kratos 固有のエラーメッセージの読み方に慣れる必要がある。README にポインタを置いてカバー
