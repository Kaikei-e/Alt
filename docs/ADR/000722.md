# ADR-000722: search-indexer に ADR-000717 の認証境界を拡張し Critical/High/Medium/Low 全 10 件の脆弱性を閉じる

## Status

Partially superseded by [[000743]] — search-indexer の `X-Service-Token` enforcement (ServiceAuthMiddleware / interceptor) は撤去され mTLS peer identity に置換された。Meilisearch Admin/Search key 分離・Redis Streams DLQ・PII ログ除去・rate limit などの security hardening は引き続き有効。

## Date

2026-04-14

## Affected Services

- search-indexer (Go) — `middleware/service_auth_middleware.go` / `middleware/rate_limit.go` / `connect/v2/interceptor/service_auth.go` / `connect/v2/interceptor/rate_limit.go` の新設、`bootstrap/servers.go` と `connect/v2/server.go` への配線、`driver/backend_api/client.go` の `http.Client` 強化、`usecase/search_articles.go` の入力検証再設計、`bootstrap/panic_log.go` の新設、`consumer/dlq.go` による Redis Streams DLQ 追加、`driver/meilisearch_driver.go` の Admin/Search クライアント分離、`rest/handler.go` / `connect/v2/search/handler.go` の PII ログ除去
- compose 基盤 — `compose/base.yaml` に `meili_search_key` secret を追加、`compose/workers.yaml` の search-indexer に `MEILISEARCH_SEARCH_API_KEY_FILE` 環境変数と secret mount を追加

## Context

[[000717]] が alt-backend の `/v1/internal/*` と `/v1/dashboard/*` の認証欠陥を閉じたが、同じ原則 (Public API = JWT / Internal API = X-Service-Token) が **search-indexer には適用されていなかった**。2026-04-14 に search-indexer 単体で OWASP Top 10:2025 + OWASP ASVS 5.0 + Meilisearch 公式セキュリティガイドを基準に静的レビューを実施し、`docs/review/search-indexer-security-review-2026-04-14.md` に Critical 3 + High 2 + Medium 3 + Low 2 の計 10 件を記録した。

特に深刻だったのは以下の 3 点:

1. **REST `/v1/search` と Connect-RPC `SearchService` に認証ミドルウェアが無かった**。`connect/v2/server.go` のコメントには `"no auth interceptor for internal service communication"` と明記されており、同一 Docker network 上の任意のコンテナから無認証で全ユーザーの記事を検索できる状態だった。[[000717]] が塞いだ alt-backend 内部 API と同じ Critical 欠陥が search-indexer にそのまま残っていた。

2. **クライアント提供の `user_id` クエリパラメータを検証なしで Meilisearch の tenant filter に渡していた**。認証ゲートが無い状態では任意の他ユーザー ID を指定して BOLA (Broken Object-Level Authorization) が成立する。OWASP Multi-Tenant Security Cheat Sheet が明示する「tenant はヘッダではなく検証済み JWT クレームから取得する」原則に反していた。

3. **`driver/backend_api/client.go` が `http.DefaultClient` を timeout 未設定で使用していた**。Go net/http 公式が明示的に警告するアンチパターンで、alt-backend がハングするとインデックスループが停止し goroutine と FD が枯渇するリスクがあった。

加えて、`bootstrap/app.go` の panic recover が panic 値をそのまま `slog.Error` で出力していた (H-001)、`usecase/search_articles.go` が Meilisearch に存在しない SQL / shell / HTML 実行経路に対する regex denylist で false positive を量産していた (H-002)、Redis Streams コンシューマに poison message DLQ が無かった (M-001)、REST/Connect-RPC に rate limiter が無かった (M-002)、`user_id` と raw query が同じログレコードに載っていた (M-003)、Meilisearch の master key 1 本で全操作を行っていた (L-001)、REST response に `total` が無く Connect-RPC の `EstimatedTotalHits` と非対称だった (L-002) など、Medium/Low も多数残っていた。

レビュー報告書の最初の草稿では Critical/High のみを同日修正し、Medium/Low は独立 ADR に分離する方針だったが、ユーザー指示で Medium/Low も同一スコープで対応することになり、10 件を一括で閉じることにした。

## Decision

以下を同一コミットでまとめて適用する。全ての修正は TDD (RED → GREEN → REFACTOR) で進め、Clean Architecture の層境界 (Handler → Usecase → Port → Gateway → Driver) を維持する。

### 1. 認証境界の拡張 (C-001 + C-002)

[[000717]] と同じ `X-Service-Token` + `crypto/subtle.ConstantTimeCompare` による定数時間比較を search-indexer の REST と Connect-RPC の両方に適用する。alt-backend は Echo ベースなので `ServiceAuthMiddleware` を移植するのではなく、search-indexer の `net/http` スタック向けに `middleware.ServiceAuthMiddleware.RequireServiceAuth(http.Handler)` を新設し、Connect-RPC 向けには `connect.UnaryInterceptorFunc` 実装の `interceptor.ServiceAuthInterceptor` を新設する。設定が空文字列 (SERVICE_TOKEN 未設定) の場合は fail-closed で全リクエストを拒否する。

認証されれば信頼された caller (alt-backend / rag-orchestrator / acolyte) のみが `/v1/search` に到達するため、クライアント提供の `user_id` を受け取ること自体は許容する。将来、Meilisearch Tenant Token (server が search key + `searchRules` で署名し Meilisearch 側が JWT 検証するモデル) に移行することで BOLA を根本的に遮断する道は残すが、本 ADR のスコープ外とする。

### 2. backend_api クライアントの timeout 設定 (C-003)

`driver/backend_api/client.go` で `http.DefaultClient` を廃止し、`buildHTTPClient()` helper で `Client.Timeout=30s` / `Dialer.Timeout=5s` / `TLSHandshakeTimeout=5s` / `ResponseHeaderTimeout=10s` / `IdleConnTimeout=90s` を設定した専用 `*http.Client` を構築する。同リポジトリの `driver/recap_api/client.go` が既に採用しているパターンを踏襲し、依存の強い `DefaultClient` 共有を排除する。

### 3. panic ログの情報量削減 (H-001)

`bootstrap/panic_log.go` に `logPanic(ctx, msg, recovered any)` helper を新設し、Error レベルでは `err_type` (型情報) のみ、Debug レベルで `runtime/debug.Stack()` の完全なスタックトレースを記録する運用に切り替える。本番は `LOG_LEVEL=info` でスタックを抑制し、OTel Collector や rask-log-aggregator に panic value が漏洩しないようにする。

### 4. 入力検証の allowlist + 正規化モデル化 (H-002)

Meilisearch は SQL エンジンでもシェル実行環境でもなく、HTML レンダリングも行わない。したがって `usecase/search_articles.go` で実装されていた SQLi / XSS / cmd-injection の regex denylist は false positive の温床でしかなかった。denylist を削除し、以下の allowlist + 正規化モデルに置換する:

- `golang.org/x/text/unicode/norm` による NFC 正規化
- 長さ上限 1000 文字
- 制御文字 (`\x00-\x1F\x7F`) と zero-width 文字の拒否
- Meilisearch の filter 値エスケープは `driver/filter.go` の既存 `escapeMeilisearchValue` に委譲

これにより `"SELECT a book"` や `"drop in temperature"` のような正当な検索が復活し、攻撃面は DoS (過大クエリ) と regex catastrophic backtracking のみに限定される。どちらも長さ上限で十分に防御できる。

### 5. Redis Streams DLQ (M-001)

`consumer/dlq.go` に reaper goroutine を追加し、`XPENDING` で delivery_count が `MaxDeliveries (default=5)` を超えた message を `XCLAIM` で所有権奪取、`XADD` で `alt:events:articles:dlq` 専用 stream に失敗理由・delivery_count・元 ID つきで退避し、`XACK` で元 stream から除外する。Redis Streams には native DLQ が無いため、Redis 公式 docs と GitHub issue 3126 を参照して標準的な外部 DLQ pattern を実装した。`ReaperInterval (default=60s)` で定期実行する。

### 6. Rate limiter (M-002)

`middleware/rate_limit.go` と `connect/v2/interceptor/rate_limit.go` を新設し、`golang.org/x/time/rate` の `rate.Limiter` で **グローバル token bucket** を装着する。内部サービスであり全 caller が X-Service-Token で認証済みの前提 (per-caller 識別は [[000717]] 同様に将来課題) のため、caller ごとの隔離ではなく単一バケットで十分とした。`SEARCH_RATE_LIMIT_RPS` (default=100) / `SEARCH_RATE_LIMIT_BURST` (default=200) を env で調整でき、超過時は REST は 429 + `Retry-After` ヘッダ、Connect-RPC は `CodeResourceExhausted` を返す。

### 7. PII ログ除去 (M-003)

`logger/pii.go` に `HashQuery(query string) string` (SHA-256 先頭 8 byte / 16 hex) を新設し、`rest/handler.go` と `connect/v2/search/handler.go` の全ログ出力で raw `query` を `query_hash` に置換する。user_id は UUID の時点でそれ単体では PII と見なさず残すが、query との結合が解かれることで「user A が何を検索したかの履歴」が rask-log-aggregator に蓄積されなくなる。hash は debug 時のクエリ同一性判定には使えるため、観測性は維持される。

### 8. Meilisearch key のロール分離 (L-001)

Meilisearch 公式セキュリティガイドが推奨する Search / Admin の key 分離を適用する。`driver.NewMeilisearchDriverWithClients(adminClient, searchClient, indexName)` を新設し、読み取り系メソッド (Search, SearchWithFilters, SearchByUserID, SearchByUserIDWithPagination) は `searchIndex` 経由、書き込み系 (IndexDocuments, DeleteDocuments, EnsureIndex, RegisterSynonyms) は `index` 経由で別クライアントにルーティングする。`MEILISEARCH_SEARCH_API_KEY` / `MEILISEARCH_SEARCH_API_KEY_FILE` が未設定の場合は admin key に fallback して既存動作を保つ。起動ログに `search_key_role_split: bool` を出力して配線状態を可視化する。compose 側は `meili_search_key` secret を追加し、Meilisearch が自動生成する `Default Search API Key` (actions=`['search']`) を secrets/meili_search_key.txt に配置する。

### 9. REST response の `total` 追加 (L-002)

`rest.SearchArticlesResponse` に `Total int json:"total"` を追加し、Connect-RPC の `EstimatedTotalHits` と対称な API 契約にする。クライアント側のページング実装が可能になる。

### 検討した代替案

- **代替案 A: REST `/v1/search` を廃止して Connect-RPC に統一** — 既存 caller (rag-orchestrator の BM25 / acolyte) が REST に依存しており、スコープが大きい。将来 [[000618]] の方針に沿って移行する余地を残すが、本 ADR では両方に認証を適用する。
- **代替案 B: Meilisearch Tenant Token 導入で BOLA を根本対策** — Meilisearch 公式推奨だが、alt-backend が token を mint し search-indexer がそのまま forward する設計変更が必要で、[[000717]] のスコープを再度超える。本 ADR では認証ゲートによる第一段階対策に留め、独立 ADR で扱う。
- **代替案 C: rate limit を Meilisearch 手前の nginx / envoy に実装** — インフラ層が理想だが単一 Docker Compose 構成で envoy を導入するコストが高く、app 層に実装する方が deploy 単位が閉じる。
- **代替案 D: per-caller rate limit** — 将来、caller 識別 (SPIFFE ID / per-caller JWT) が導入された時点で差し替え可能な構造にしておき、当面は global bucket で十分とした。

## Implementation Overview

### 認証境界 (C-001 + C-002)

```go
// middleware/service_auth_middleware.go
func (m *ServiceAuthMiddleware) RequireServiceAuth(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        if m.serviceSecret == "" {
            http.Error(w, http.StatusText(http.StatusInternalServerError), 500)
            return
        }
        token := strings.TrimSpace(r.Header.Get("X-Service-Token"))
        if token == "" {
            http.Error(w, http.StatusText(http.StatusUnauthorized), 401)
            return
        }
        expected := []byte(m.serviceSecret)
        provided := []byte(token)
        valid := len(expected) == len(provided) &&
            subtle.ConstantTimeCompare(expected, provided) == 1
        if !valid {
            http.Error(w, http.StatusText(http.StatusUnauthorized), 401)
            return
        }
        next.ServeHTTP(w, r)
    })
}

// connect/v2/interceptor/service_auth.go
func (i *ServiceAuthInterceptor) WrapUnary(next connect.UnaryFunc) connect.UnaryFunc {
    return func(ctx context.Context, req connect.AnyRequest) (connect.AnyResponse, error) {
        if err := i.verify(ctx, req.Header().Get("X-Service-Token"), req.Spec().Procedure); err != nil {
            return nil, err
        }
        return next(ctx, req)
    }
}
```

テスト (`middleware/service_auth_middleware_test.go`, `connect/v2/interceptor/service_auth_test.go`) に missing / invalid / wrong-length / valid / unconfigured-secret の 5 ケースを追加し、fail-closed を契約として担保した。

### Meilisearch key 分離 (L-001)

```go
func NewMeilisearchDriverWithClients(adminClient, searchClient meilisearch.ServiceManager, indexName string) *MeilisearchDriver {
    d := &MeilisearchDriver{
        client:      adminClient,
        index:       adminClient.Index(indexName),
        searchIndex: adminClient.Index(indexName),
    }
    if searchClient != nil {
        d.searchIndex = searchClient.Index(indexName)
    }
    return d
}
```

`bootstrap/drivers.go:initMeilisearchClients()` が admin と search の二つの ServiceManager を返し、search が nil なら admin に fallback する。compose 側で Meilisearch 自動生成の `Default Search API Key` を `secrets/meili_search_key.txt` に配置し、`MEILISEARCH_SEARCH_API_KEY_FILE=/run/secrets/meili_search_key` で注入する。

### Redis Streams DLQ (M-001)

```go
func (c *Consumer) reapPending(ctx context.Context) error {
    pending, _ := c.client.XPendingExt(ctx, &redis.XPendingExtArgs{
        Stream: c.config.StreamKey, Group: c.config.GroupName,
        Start: "-", End: "+", Count: int64(c.config.BatchSize * 2),
    }).Result()
    for _, p := range pending {
        if !shouldSendToDLQ(p.RetryCount, c.config.MaxDeliveries) { continue }
        claimed, _ := c.client.XClaim(ctx, &redis.XClaimArgs{...}).Result()
        values := map[string]any{
            "dlq_reason": "max_deliveries_exceeded",
            "dlq_delivery_count": p.RetryCount,
            "dlq_original_id": p.ID,
            "dlq_reaped_at": time.Now().UTC().Format(time.RFC3339),
        }
        for k, v := range claimed[0].Values { values[k] = v }
        c.client.XAdd(ctx, &redis.XAddArgs{Stream: c.config.DLQStreamKey, Values: values})
        c.client.XAck(ctx, c.config.StreamKey, c.config.GroupName, p.ID)
    }
    return nil
}
```

## Consequences

### Pros

- **Critical 3 件 + High 2 件 + Medium 3 件 + Low 2 件の計 10 件を同日クローズ**。無認証で閲覧可能だった全ユーザー記事検索経路が閉じ、user_id 改変による BOLA リスクが認証ゲートで間接的に遮断された。backend_api client のハング耐性、panic ログの機微情報漏洩、PII 結合、DoS 耐性、poison message 処理、Meilisearch key の最小権限化まで一括で解消した。
- **[[000717]] と search-indexer の認証原則が一貫**。Public API = JWT / Internal API = X-Service-Token / 定数時間比較という原則が設計文書だけでなくコード上でも REST/Connect-RPC 両 transport で適用された。将来 search-indexer に internal route を追加する際の設計ミスが起きにくい。
- **検索 UX の改善**。regex denylist 廃止により `"SELECT a book"` / `"drop in temperature"` / `"executive summary"` といった正当な検索が復活した。Unicode NFC 正規化で合成済み文字と分解形の同一化も実現。
- **Meilisearch key の最小権限化**。search driver が仮に侵害されても write 権限を持たないため、悪意あるインデックス改変・削除ができない。admin key はバッチインデクサ経路のみに限定された。
- **観測性を維持した PII 保護**。query を hash 化しても debug 時のクエリ同一性判定は可能で、トラブルシューティング能力は失われない。一方で user 個人の検索履歴プロファイルはログ基盤に蓄積されない。
- **DLQ による poison message 隔離**。Redis 公式推奨の XPENDING + XCLAIM + 別 stream 退避パターンで、不正 payload が無限リトライされてインデクサが詰まるリスクが解消された。
- **TDD と Clean Architecture 原則の遵守**。全ての修正で RED テストを先に書き、GREEN に倒してから次の修正に進む順序を厳守した。新規追加コンポーネント (middleware, interceptor, DLQ, hash helper, panic helper) は全て既存層に正しく配置され、layer-checker agent で依存違反ゼロを確認している。
- **レビュー報告書が後続エンジニアの参照資料になる**。`docs/review/search-indexer-security-review-2026-04-14.md` に OWASP/CWE/ASVS マッピングと具体的な file:line まで記録されており、次回のレビュー時の diff ベースラインとして機能する。

### Cons / Tradeoffs

- **内部 API 呼び出し元は SERVICE_TOKEN を正しく設定する必要がある**。alt-backend / rag-orchestrator / acolyte が search-indexer を呼ぶ際に `X-Service-Token` を付けていない経路があれば 401 で止まる。compose secret は既に全サービスで統一 (`service_secret`) されているため実質的な影響は無い見込みだが、将来別 compose / 別環境で運用する際は caller 側の env 同期が必須となる。
- **caller 識別ができない状態は残る**。shared secret 方式のため、search-indexer のログから「どのサービスが検索を呼んだか」を区別できない。将来 [[000717]] と同様に per-caller JWT (iss=pre-processor / rag-orchestrator, sub=service UUID) か SPIFFE ID 伝播に移行する余地を残す。
- **Meilisearch Tenant Token 導入は future work**。本 ADR の第一段階では「認証された caller が提供する user_id を信頼する」モデルに留めるため、caller が侵害された場合の横展開リスクは残る。Meilisearch が `searchRules` 署名を強制する第二段階に進めば、caller 侵害でも Meilisearch 側で BOLA が遮断される。
- **Rate limit が global bucket**。内部サービス前提で per-caller 隔離を犠牲にしている。同一 caller の暴走が全 caller に影響するため、caller 識別導入後に per-caller 化する必要がある。
- **search key のローテーション運用**。Meilisearch の `Default Search API Key` はデフォルトで rotation しない。master key を rotate した際に default keys も自動更新されるため、compose 側の `meili_search_key.txt` も同期更新するルーティーンが必要になる。将来 runbook 化する。
- **DLQ 消費者は未実装**。DLQ stream にメッセージが溜まる仕組みはできたが、DLQ の可視化・alert・手動再処理ツールは本 ADR のスコープ外。運用中に poison message が発生するまでは問題にならないが、オプザーバビリティ改善として別タスクで対応する。
- **panic の debug トレースはログレベル依存**。本番で panic が起きた時に詳細が残らない可能性がある。必要に応じて Sentry などの error tracking に panic を直接送る経路を追加する検討余地がある。

### 将来の検討課題
