# フィード登録レイテンシ改善: 5施策による非同期化・重複排除・接続プーリング

## ADR's STATUS

Accepted

## CONTEXT

3000VU負荷試験（Run3）において、Circuit Breaker / Semaphore の上限を500に引き上げてエラー率0%を達成した。しかし、以前 fast-fail していた65%のリクエストが全て処理されるようになった結果、median latency が17.27s（p95=37.6s）に増加した。

ボトルネック分析により、以下が主要因と判明:

| ボトルネック | 推定影響 | 根拠 |
|---|---|---|
| event publishing が同期実行 | 2-10s | コメントは "fire-and-forget" だが実装は同期。10件直列POST + `http.DefaultClient`（MaxIdleConnsPerHost=2） |
| 同一URLの重複fetch | 1-5s | 3000VU / 1000 aliases = 同一URL×3回fetch。全てsemaphore + HTTP fetch通過 |
| HTTPクライアント毎回生成 | 0.5-2s | `createHTTPClient()` が毎回 `http.Transport` を新規作成。TCP接続再利用なし |
| mq-hub クライアントの接続不足 | 0.5-2s | `http.DefaultClient` 使用（MaxIdleConnsPerHost=2）による接続プール枯渇 |
| DB接続プール競合 | 0.5-2s | MaxConns=100 に3000concurrent。3-4 DB ops/request |

**目標**: median 17.27s → 5s以下に削減（エラー率0%を維持）

## DECISION MAKING

5つの独立した施策を優先度順に実施した。

### 施策1: event publishing + auto-subscribe の真の非同期化

**ファイル**: `alt-backend/app/usecase/register_feed_usecase/feeds_usecase.go`

`publishFeedEvents` と `autoSubscribeUser` は設計意図としては fire-and-forget だが、実装は同期的に実行されていた。10件の記事に対して直列で Connect-RPC を呼び出すため、3000VU下では2-10sのレイテンシを追加していた。

`go` キーワードによるgoroutine化 + `context.WithoutCancel(ctx)` でHTTPレスポンス送信後もgoroutineが生存するようにした。両操作とも既にエラーはログ出力のみで呼び出し元に返していないため、動作上の変更はない。

テストは `sync.WaitGroup` + `DoAndReturn` パターンで非同期goroutineの完了を待機し、mock expectation の検証を確実に行えるよう更新した。

### 施策2: singleflight によるRSS fetch重複排除

**ファイル**: `alt-backend/app/gateway/validate_fetch_rss_gateway/gateway.go`

`ValidateAndFetchRSSGateway` に `golang.org/x/sync/singleflight.Group` を追加し、同一URLの並行fetchを1回に集約した。

既存パターン参照: `fetch_article_tags_gateway/gateway.go` で同じく `singleflight.Group` を使用済み。singleflight のスコープはURL validation 後の semaphore + circuit breaker + fetch 全体を包む形とした。これにより、3000VU / 1000 aliases の負荷パターンでは同一URLへの3回のfetchが1回に削減され、semaphore通過リクエスト数が約1/3になる。

### 施策3: HTTPクライアント共有化

**ファイル**: `alt-backend/app/gateway/register_feed_gateway/single_feed_link_gateway.go`

`DefaultRSSFeedFetcher` が `createHTTPClient()` で毎回 `http.Transport` を新規生成していた。3000concurrent下ではTCP接続の再利用が全くなく、接続確立のオーバーヘッドが蓄積していた。

`http.Client` をコンストラクタで1回だけ作成しフィールドに保持する方式に変更。`http.Transport` は goroutine-safe であり、並行アクセスに対応している。接続プールサイズも `MaxIdleConnsPerHost=50`、`MaxIdleConns=200` に拡大した。`createHTTPClient()` メソッドは削除した。

### 施策4: mq-hub クライアントの接続プーリング

**ファイル**: `alt-backend/app/driver/mqhub_connect/client.go`

`http.DefaultClient`（MaxIdleConnsPerHost=2）を、専用の `http.Client`（MaxIdleConnsPerHost=50, MaxIdleConns=100, Timeout=30s）に置換した。施策1でevent publishingは非同期化されるが、mq-hubへのスループット自体も改善され、ephemeral port枯渇問題の根本対策となる。

### 施策5: 負荷試験インフラチューニング

**ファイル**: `alt-perf/scripts/run-feed-load-test.sh`

| パラメータ | 変更前 | 変更後 | 根拠 |
|---|---|---|---|
| `DB_MAX_CONNS` | 100 | 200 | 3000concurrent に対して100接続では競合が激しい |
| `DEFAULT_POOL_SIZE` (PgBouncer) | 50 | 80 | バックエンド接続数増加に合わせてプールサイズ拡大 |
| `MAX_DB_CONNECTIONS` (PgBouncer) | 130 | 200 | DB側max_connections=250に対する余裕確保 |

## RESULTS, EFFECTS

### PROS

- **クリティカルパスからevent publishingを完全除去**: Execute() は feeds の DB 登録完了直後に return するため、event publishing のレイテンシが一切反映されない
- **semaphore 競合の大幅削減**: singleflight により同一URLの重複fetchが排除され、semaphore スロット消費が最大1/3に削減される
- **TCP接続再利用**: RSS fetch と mq-hub 通信の両方で接続プーリングが有効になり、接続確立のオーバーヘッドが解消される
- **既存テスト全パス**: 全4パッケージのテストがパスすることを確認済み
- **既存動作への影響なし**: event publishing / auto-subscribe は元々エラーを握りつぶす設計のため、非同期化による動作変更はない

### CONS, TRADEOFF

- **goroutineリーク検出の複雑化**: fire-and-forget goroutine は呼び出し元で完了を待機しないため、mq-hub が長時間応答しない場合にgoroutineが蓄積する可能性がある。現時点ではHTTPクライアントの30sタイムアウトで上限は制御されるが、将来的にはgoroutine数の監視やworker pool化を検討する余地がある
- **singleflight による一時的な結果共有**: 同一URLへの並行リクエストが同じ `*domain.ParsedFeed` ポインタを共有する。現在のコードでは共有後に各リクエストがfeed itemsのフィールドを書き換える（`item.FeedLinkID = feedLinkID`）が、これは各リクエストで同一の feedLinkID が設定されるため問題にならない。異なるユーザーが異なるfeedLinkIDを設定するシナリオが生じた場合は deep copy が必要になる
- **DB接続プール拡大のメモリ影響**: MaxConns=200 は PostgreSQL 側のメモリ消費を増加させる（接続あたり約10MB）。負荷試験環境ではDB に4GBのメモリ制限を設定しているため許容範囲内
