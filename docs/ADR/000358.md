# Recap タグ品質改善 — tag-generator によるセマンティックタグ付け

## ADR's STATUS

Accepted

## CONTEXT

### top_terms の限界

Recap の各ジャンル出力には `top_terms` フィールドが付与されているが、これは recap-subworker の c-TF-IDF（Class-based Term Frequency-Inverse Document Frequency）で生成された統計ベースのキーワードである。c-TF-IDF はクラスタ内の単語頻度に基づくため、以下の課題があった:

1. **意味的精度が低い**: 頻出するが本質的でない単語がキーワードとして選ばれることがある（例: "designed", "attacks" など一般的な英単語）
2. **言語横断の一貫性が不十分**: 日英混在のコンテンツに対して統計的手法だけでは適切なタグを抽出しにくい
3. **Meilisearch 検索との相性**: ADR-356 で導入した Meilisearch ベースの Recap 検索において、`top_terms` の品質が検索精度のボトルネックになっていた

### tag-generator の活用可能性

一方、Alt には既に tag-generator サービスが存在し、KeyBERT + sentence-transformers を使ったセマンティックなタグ抽出を行っている。このサービスは:

- 記事単位のタグ付けに利用されており、実績がある
- 日英バイリンガル対応（langdetect による自動言語判定）
- sentence-transformers による文脈理解に基づく高精度なキーワード抽出が可能

Recap のサマリーテキスト（summary_ja + bullets_ja）に対しても同じ手法を適用すれば、検索精度を大幅に向上できると判断した。

## DECISION MAKING

### 既存の `top_terms` を維持し、新たに `tags` フィールドを追加する

`top_terms` は c-TF-IDF によるクラスタレベルのキーワードとして引き続き有用（特にクラスタの統計的特徴を表現する用途）であるため、削除せずに維持する。セマンティックタグは別フィールド `tags` として追加し、Meilisearch では `tags` を最優先の searchable attribute に設定する。

### アーキテクチャ: persist ステージ後のタグ付け

recap-worker の既存パイプライン（fetch → cluster → summarize → persist）の persist ステージ内で、DB 保存後に tag-generator を呼び出す設計を採用した。

```
recap-subworker → clustering + c-TF-IDF (top_terms)
       ↓
news-creator → summarization (summary_ja, bullets_ja)
       ↓
recap-worker persist → DB保存
       ↓  ★ NEW
tag-generator → extract_tags(genre, summary+bullets) → tags
       ↓
recap_outputs.tags に保存
       ↓
search-indexer → Meilisearch recaps インデックスに tags 追加
```

### 各サービスの変更

#### 1. tag-generator (Python/FastAPI) — テキストベースのタグ抽出エンドポイント

既存の `TagExtractor.extract_tags_with_metrics()` を再利用する新エンドポイント `POST /api/v1/extract-tags` を追加。サービス間認証（X-Service-Token）で保護し、任意の title + content テキストに対してセマンティックタグを返す。

```python
class ExtractTagsRequest(BaseModel):
    title: str      # ジャンル名
    content: str    # summary + bullets のテキスト
```

レスポンス: `{ tags: [...], confidence: float, inference_ms: int, language: str }`

#### 2. recap-worker (Rust) — persist ステージでのタグ生成

`TagGeneratorClient` に `extract_tags()` メソッドを追加。`FinalSectionPersistStage` に `tag_generator: Option<Arc<TagGeneratorClient>>` を注入し、persist 成功後に各ジャンルの `summary_ja + bullets_ja` を入力としてタグを生成する。

tag-generator が利用不可の場合は空配列にフォールバックし、パイプライン全体の失敗を防ぐ（既存の optional パターンに従う）。

#### 3. DB マイグレーション

`recap_outputs` テーブルに `tags JSONB NOT NULL DEFAULT '[]'::jsonb` カラムを追加。`jsonb_path_ops` による GIN インデックスを作成し、JSONB 含有クエリの高速化を確保した。

#### 4. search-indexer (Go) — Meilisearch recaps インデックスの拡張

- `RecapDocument` / `RecapDocumentDriver` に `Tags []string` フィールドを追加
- Meilisearch の searchable attributes を `["tags", "top_terms", "summary", "genre"]` に設定（`tags` を最優先）
- `AddDocuments` 呼び出しに `DocumentOptions{PrimaryKey: "id"}` を明示的に指定（`id` と `job_id` の2つの `id` サフィックス付きフィールドが存在するため、Meilisearch の primary key 自動推論が失敗する問題を解決）

#### 5. Proto + alt-backend — tags フィールドの伝搬

`proto/services/search/v2/search.proto` の `RecapSearchHit` メッセージに `repeated string tags = 9` を追加。コード生成を実行し、search-indexer と alt-backend の双方に反映。alt-backend の `domain.RecapSearchResult` と Connect-RPC クライアントにも `Tags` フィールドを追加。

## RESULTS, EFFECTS

### PROS

- **検索精度の向上**: KeyBERT によるセマンティックタグが Meilisearch 検索の最優先フィールドとなり、意味的に関連性の高い検索結果が返るようになった
- **既存機能の非破壊**: `top_terms` を維持したため、クラスタの統計的特徴を表現する用途には影響なし
- **フォールバック設計**: tag-generator 障害時にも recap パイプラインが停止しない（空配列でフォールバック）
- **既存インフラの再利用**: tag-generator の `TagExtractor` をそのまま活用し、新たな ML モデルの導入は不要
- **バックフィル対応**: search-indexer のデュアルフェーズ（backfill + incremental）パターンにより、既存データも順次 Meilisearch に反映される（既存データは `tags: []` で格納され、新しい recap job から順次セマンティックタグが付与される）

### CONS, TRADEOFF

- **レイテンシ増加**: persist ステージで tag-generator への HTTP 呼び出しが追加されるため、各ジャンルの persist に数百ミリ秒のオーバーヘッドが発生する。ただし recap job は数時間間隔で実行されるバッチ処理のため、実質的な影響は軽微
- **既存データのタグ不在**: 過去の recap_outputs は `tags: []` のまま。セマンティックタグを付与するには再実行が必要。将来的にバックフィルスクリプトで対応可能
- **tag-generator への負荷**: recap job 実行時に全ジャンル（最大30）分のタグ抽出リクエストが集中する。現状の負荷では問題ないが、大規模化する場合はバッチ API の検討が必要

### Meilisearch recaps インデックス設定

- **Searchable**: `tags` > `top_terms` > `summary` > `genre`（優先度順）
- **Filterable**: `genre`, `window_days`
- **Sortable**: `executed_at`

### Backfill ページネーション修正

ADR-356 で導入した backfill ループにページネーションバグがあり、同時に修正した。

**問題**: `ExecuteBackfill` が `since` パラメータを受け取らず、毎回空文字列で recap-worker API を呼び出していた。結果、同じ先頭200件を無限ループで再インデックスし続け、Meilisearch には200件しか格納されていなかった（DB には 2,747 件の indexable genres が存在）。

**修正内容**:
1. `search-indexer/usecase/index_recaps.go`: `ExecuteBackfill(ctx, since, batchSize)` に `since` パラメータを追加
2. `search-indexer/bootstrap/app.go`: backfill ループで `lastSince` カーソルを `ExecuteBackfill` に渡すように変更
3. `recap-worker/store/dao/output.rs`: `DISTINCT ON` をサブクエリに押し込み、外側で `kicked_at ASC` ソート + `>=` で境界行を含める

**結果**: 200件 → 2,747件（DB の全 indexable genres と完全一致）
