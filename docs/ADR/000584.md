# ADR-000584: knowledge_events を月次パーティション化し dedupe registry を分離する

## Status

Accepted

## Date

2026-03-25

## Affected Services

- knowledge-sovereign (Go) — AppendKnowledgeEvent の INSERT パスを dedupe registry 経由に変更
- knowledge-sovereign-db (PostgreSQL 16) — テーブルパーティション化 + BRIN インデックス追加

## Context

Knowledge Home のイベントストア `knowledge_events` は append-only 設計であり、データが無限に成長する構造的な宿命を持つ。2026-03-25 時点の実測で **786,652 行 / 760 MB**（インデックス含め 1.1 GB 超）に達しており、単一ホスト Docker Compose 環境ではディスク圧迫・VACUUM 効率低下・reproject 時間の線形増加が実害として現れつつある。

[[000398]] は retention policy を「Phase 2 課題」として先送りしており、[[000532]] は Knowledge Sovereign が retention policy matrix を所有すると決定したが、テーブルレベルの構造的対策は未着手だった。

さらに、将来的なアーカイブ（パーティション単位の DETACH → COPY TO → DROP）を実現するには、テーブルがパーティション化されていることが前提条件となる。

### データ膨張の発見

実測で以下が判明した:
- 15,117 ユニーク記事に対して 352,418 件の `TagSetVersionCreated` イベント（記事あたり平均 23.3 回）
- 1 件の記事に最大 7,282 件のイベント
- backfill 期間 (3/19-20) に ~230K-245K events/day のスパイク
- 定常状態では ~1,200-2,600 events/day

## Decision

### 1. knowledge_events / knowledge_user_events を月次パーティション化する

`PARTITION BY RANGE (occurred_at)` で月次パーティションを作成する。

```sql
CREATE TABLE knowledge_events (
  ...
  PRIMARY KEY (event_id, occurred_at)  -- partition key を含む
) PARTITION BY RANGE (occurred_at);
```

初期パーティション: `y2025m11` 〜 `y2026m04` + `default`。

### 2. dedupe registry を非パーティションテーブルとして分離する

PostgreSQL の制約上、partitioned table の UNIQUE/PK には partition key を含める必要がある。現行の `dedupe_key UNIQUE` と `event_seq UNIQUE` はそのまま partition parent に置けないため、グローバル一意性を保証する **非パーティションの dedupe registry** を導入する。

```sql
CREATE TABLE knowledge_event_dedupes (
  dedupe_key    TEXT PRIMARY KEY,
  event_id      UUID NOT NULL,
  occurred_at   TIMESTAMPTZ NOT NULL,
  first_seen_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

`AppendKnowledgeEvent` は 2 ステップに変更:
1. `INSERT INTO knowledge_event_dedupes ... ON CONFLICT (dedupe_key) DO NOTHING`
2. 影響行 > 0 なら `INSERT INTO knowledge_events ... RETURNING event_seq`

### 3. BRIN インデックスを occurred_at に追加する

append-only テーブルでは物理行順序と `occurred_at` が強く相関するため、BRIN が有効。既存の B-tree (`idx_knowledge_events_occurred`) は EXPLAIN 確認後に判断する。

```sql
CREATE INDEX idx_knowledge_events_occurred_brin
  ON knowledge_events USING brin (occurred_at) WITH (pages_per_range = 32);
```

### 4. pg_partman は現時点では不採用

| 軸 | pg_partman | altctl CLI |
|----|-----------|-----------|
| Extension lifecycle | `shared_preload_libraries` + DB 再起動 | 不要 |
| CI/local image 管理 | カスタム PostgreSQL image 必要 | 既存 image |
| 運用の一貫性 | DB 内で完結 | altctl に集約 |

パーティション数が 24 以上（2年分）に達した時点で再評価する。

## Consequences

### Pros

- パーティション単位の VACUUM/ANALYZE が可能になり、大テーブルのメンテナンスコストが分散する
- 将来のアーカイブ（DETACH PARTITION → COPY TO → DROP）の前提条件が整う
- BRIN インデックスにより occurred_at のレンジクエリが効率化する（インデックスサイズ 50x 削減の可能性）
- dedupe registry により冪等性が維持されつつ、パーティション制約と両立する

### Cons / Tradeoffs

- Projector の read path が `Merge Append` になり、全パーティションを走査する（実測 0.9ms で実用上問題なし）
- dedupe registry が非パーティションなのでこちらは単独で成長し続ける（176 MB at 786K rows）
- マイグレーション時にテーブルリネーム + データコピーが必要（786K 行で ~33 秒）
- `AppendKnowledgeUserEvent` の `ON CONFLICT` 句を partial index に合わせる変更が必要だった（`ON CONFLICT (dedupe_key, occurred_at) WHERE dedupe_key != ''`）
