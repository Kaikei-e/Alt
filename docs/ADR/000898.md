# ADR-000898: hooks.client にチャンク 404 自動 reload を入れて iOS Safari の "ページを開けませんでした" を根治する

## Status

Accepted

## Date

2026-05-13

## Affected Services

- alt-frontend-sv (TypeScript / Svelte 5) — `src/hooks.client.ts`, `src/hooks.server.ts`, `src/lib/safari-error-utils.ts`, `src/app.html`, `svelte.config.js`, `src/routes/+layout.svelte`, `src/routes/(app)/feeds/swipe/visual-preview/+page.server.ts`, `src/routes/(app)/feeds/swipe/visual-preview/+page.svelte`, `src/lib/server/nginx-connect-streaming-config.test.ts`, `src/hooks.client.test.ts` (新規)
- nginx (`nginx/conf.d/default.conf`) — `map $http_upgrade $connection_upgrade` 追加と `/api/` / `/` の Connection ヘッダ正規化

## Context

ユーザ報告は「iOS Safari (通常タブ・最新 iOS) で『ページを開けませんでした』フルシートが間欠的に出る。再現は `/feeds` や `/menu` などタブ復帰直後の page-by-page。30 日以内の変更以降にしか起きていない」。[[000881]] / [[000883]] / [[000884]] (いずれも 2026-05-02 accepted) と Safari connection recovery hook (2026-05-03 commit) で **再三対処済**だが残存していた。

### 観測

| 指標 | 値 | 出典 |
|---|---|---|
| 直近 7d iOS Safari の 5xx 率 | **5.38%** | nginx access |
| 直近 7d その他 UA の 5xx 率 | **0.45%** | nginx access |
| iOS Safari 5xx の正体 | `/api/v2/.../FetchArticleContent` の `elapsed_ms ≈ 8000ms` ([[000881]] 由来の 8s deadline) | alt-backend / BFF |
| `_app/immutable/{chunks,nodes}/<hash>.js` の 404 (直近 7d) | **iOS 4 件 / Chrome 6 件** (referer は `/feeds/swipe/visual-preview` / `/menu`) | nginx access |
| alt-frontend-sv stdout の `ECONNRESET / EPIPE / fetch failed / SvelteKitError` | **0 件** | docker logs |
| ClickHouse `otel_logs` の `safari|webkit|aborted|reset` UA 行 | **0 行** | rask_logs |

5xx の絶対数は iOS 側が多いが、それは [[000881]] の 8s deadline を踏んだ際の **in-page フォールバック**経路 (`VisualPreviewCard` の `Source content unavailable.` 表示, [[000883]]) で受け止め済であり、ユーザに **フルシートは出ない**。

### 因果連鎖

調査を log-seeker / web-researcher / Plan サブエージェントの 3 系統で並走させ、以下に絞り込んだ。

1. `vite build` で `/_app/immutable/{chunks,nodes,entry}/<hash>.js` のハッシュ一式が毎回入れ替わる。
2. ユーザのタブが suspend / bfcache から復帰する瞬間に、古い HTML / runtime が新しい chunk URL を要求するか、新しい HTML が origin から消えた古い chunk を要求して **404**。
3. SvelteKit の dynamic import がそのまま reject → 何もハンドリングしていない (本リポジトリには `hooks.client.ts` 未設定) → top-level promise rejection。
4. **iOS Safari は dynamic import の 404 を soft-fail せず "Cannot Open the Page" フルシートに昇格させる**。Chromium / Firefox は soft-fail (再 nav するか blank) なので報告がほぼ iOS Safari に偏る。

Web 調査の裏付け:

- sveltejs/kit#10315, sveltejs/kit#13480 — iOS Safari は streamed response の完了を待ってから初描画する仕様で、stream tail が乱れると同じシートに出る。
- oven-sh/bun#11621, oven-sh/bun#6758 — Bun の HTTP/1.1 chunked terminator / `ReadableStream.cancel` 周りに既知の取りこぼし。SvelteKit 2 の deferred-promise streaming を Bun 1.3.12 で踏むと顕在化する。
- nginx の `proxy_set_header Connection "upgrade"` リテラルは WebSocket でない通常 GET/POST にも常時 Connection: upgrade を流すアンチパターン。canonical pattern は `map $http_upgrade $connection_upgrade { default upgrade; '' close; }`。

Plan サブエージェントによる独立検証で仮説優先順位を **H_top (chunk 404) ≫ H1 (streamed Promise) > H2 (Connection upgrade) > H3 (format-detection)** と確定。さらにユーザ証言 (最新 iOS / 通常 Safari タブ / タブ復帰直後 page-by-page) で **H_top が主犯 1 本**と確定した。`visual-preview` ピンポイントではなく page-by-page で多発しているため、H1 単独では症状を説明できない。

### 取り下げた仮説

- vite `target: "esnext"` × iOS 16.x white screen: ユーザの iOS が最新で無関係。
- Service Worker / Cache Storage 既知 bug: 本リポジトリは Service Worker を登録していない (grep で 0 件)。
- HPACK 境界 / 大 Cookie (h2 GOAWAY): log-seeker で GOAWAY ゼロ。観測強化の保険のみ残置。

## Decision

主犯 1 本 (H_top) の根治と、副次仮説の予防注射を同じ PR にまとめて入れる。

### 1. **クライアント側 chunk-load failure ハンドラ** を追加する (P0, 主犯対策)

`alt-frontend-sv/src/hooks.client.ts` を新規作成し、

- SvelteKit の `HandleClientError` で次の正規表現に該当するメッセージを捕捉:
  `Failed to fetch dynamically imported module|ChunkLoadError|Loading chunk\s+\d+\s+failed|Importing a module script failed|Failed to load module script`
- 同じパターンを `window.addEventListener("unhandledrejection", ...)` でも拾う (ハンドラ未経由の reject 経路)
- 検知時は `sessionStorage` キー `alt:chunk-reload-attempts` を 0 → 1 → 2 → 3 でインクリメントし、**3 回上限**で `window.location.reload()` を発火 (ループバック保険)。
- 構造化 JSON ログに `safariBucket` (`ios-safari | macos-safari | other`) と `chunkHash` (URL から `/_app/immutable/(chunks|nodes|entry)/([^./?\s]+)\.js` を切り出し) を埋める。

純粋関数 (`classifySafari`, `extractChunkHash`, `isChunkLoadError`) は副作用ゼロの `src/lib/safari-error-utils.ts` に切り出し、`hooks.server.ts` からも安全に import できるようにした (hooks.client.ts は最後の `addEventListener` が server で走らない設計だが、共有 helper を独立モジュールに置くほうが意図がはっきりする)。

### 2. **`updated` Rune を 5 分ポーリングで発火させ、レイアウトで監視**する (P0, 主犯対策)

`svelte.config.js` の `kit.version` に `pollInterval: 5 * 60 * 1000` を加え、SvelteKit が `/__version.json` を定期チェックして `$app/state` の `updated.current` を立てるようにする。

`src/routes/+layout.svelte` に次の `$effect` を置く:

```ts
let reloadOnUpdate = false;
$effect(() => {
    if (!updated.current || reloadOnUpdate) return;
    reloadOnUpdate = true;
    if (typeof window !== "undefined") window.location.reload();
});
```

ユーザ操作なしで 5 分以内に新 build へ追従し、`hooks.client` のリアクティブ捕捉 (D1) と二段構えにする。

### 3. `/feeds/swipe/visual-preview` の **deferred Promise を撤去**する (P0, 副次予防)

`+page.server.ts` の `articleData: articleDataPromise` 返却を `await loadFirstArticle(...)` に置換し、SSR レスポンスは常に完全形の HTML 一発で返す。`+page.svelte` 側も `data.articleData.then(...)` を `$derived(data.articleData)` に書き換えた。6s AbortController と feed-level proxy URL 経由の LCP 最適化は [[000881]] の決定を温存している。Bun の chunked terminator race を踏まない経路に倒し、`(app)/+layout.ts` (`ssr=false`) の中で `visual-preview/+page.ts` (`ssr=true`) だけが streamed SSR を踏むグレーゾーンを消す。

### 4. nginx `Connection` を `$connection_upgrade` map で正規化する (P1)

`nginx/conf.d/default.conf` の冒頭 (`server {}` の外) に

```nginx
map $http_upgrade $connection_upgrade {
    default upgrade;
    ''      close;
}
```

を追加し、`/api/` (元 L397-398) と `/` (元 L431-432) の `proxy_set_header Connection "upgrade";` を `proxy_set_header Connection $connection_upgrade;` に置換した。`/ory/` (Kratos の long-poll に upgrade を期待する可能性が薄いが保守的に温存) はリテラル指定を維持。

### 5. `app.html` に `format-detection` を 1 行追加する (P1)

`<meta name="viewport">` の直下に
```html
<meta name="format-detection" content="telephone=no, date=no, address=no, email=no" />
```
を追加。iOS Safari が記事 ID / 時刻 / view count などの数字を `<a href="tel:...">` で勝手にラップし、Svelte 5 の hydrate が DOM mutation で `hydration mismatch` を投げる経路 (sveltejs/svelte#17357) を予防する。本リポジトリは `tel:` / `mailto:` の自前リンクを持っていない (grep で 0 件) ため副作用ゼロ。

### 6. `handleError` 観測強化 + 301 → 303 redirect (P2)

`src/hooks.server.ts` の `handleError` JSON ログに `safariBucket` と `chunkHash` を追加。次回再発時にログから 5 分以内に iOS 偏在と chunk 経路を確認できるようにする。同ファイルの legacy path redirect (`resolveResponsiveRedirect` 経由) を `throw redirect(301, ...)` から `throw redirect(303, ...)` に変更し、iOS Safari の 301 semi-permanent cache から旧 path bookmark を解放した。

### 検討した代替案と却下理由

- **`visual-preview/+page.ts` を `ssr=false` に倒す**: Plan サブエージェントの第一推奨。親 `ssr=false` × 子 `ssr=true` のグレーゾーンを根絶できるが、LCP が 1.0s から 1.3s+ に劣化する想定があり、症状が page-by-page で `visual-preview` ピンポイントでない以上、優先度を下げた。`await` 化 (D3) でも streaming abort 経路は閉じるので同等の予防効果がある。
- **SvelteKit の SSR streaming を `filterSerializedResponseHeaders` で完全に止める**: hooks.server.ts の `resolveOptions` をいじる方針。今回必要なのは「streaming を踏まない 1 ルートだけ」なので、ルート個別の load 書換のほうがブラスト半径が小さい。
- **`/_app/immutable/*` を `Cache-Control: no-store` に倒す**: チャンク 404 を生まない方向の対策。ただし immutable resource は `Cache-Control: public, immutable` が SvelteKit の前提で、no-store にすると LCP / TTI が悪化、Cloudflare edge cache も無効化される。`hooks.client` reload 発火のほうがコストが圧倒的に小さい。
- **Bun 1.3.12 → 1.3.10 ロールバック**: 検証手段としては有効だが、根治ではない。本 ADR の対策で sufficient なので保留。
- **`large_client_header_buffers 8 32k;` を nginx に入れる (H4 保険)**: log-seeker で GOAWAY 観測ゼロなので即適用しない。再発時に開封する。

## Consequences

### Pros

- 主犯 (`_app/immutable/*` 404 後の WebKit ハードフェイル) をユーザ操作ゼロで自動復旧する。タブ復帰 → reload は 1 回で済み、3 回上限のループバック保険で暴走しない。
- `version.pollInterval` 経由でデプロイ後 5 分以内に新 build へ追従するので、ユーザがタブを長時間放置しているケースでも事前に解消する。
- visual-preview の SSR レスポンスが「完全形の HTML 1 発」に正規化され、SvelteKit 2 streamed SSR × Bun chunked terminator × iOS Safari "wait-for-full-response" の三角関係を踏まなくなる。
- nginx の `Connection: upgrade` 常時送出が canonical map に直り、Bun keep-alive プールの予期せぬ close 経路を 1 つ閉じた。WebSocket 経路 (今回該当なし、`/ory/` のみ温存) には影響しない。
- `handleError` JSON ログに `safariBucket` / `chunkHash` が乗ったので、ClickHouse 側で「iOS Safari の chunk-load 失敗だけ」のアラートルールを次の PR で書ける。
- legacy path redirect が 303 になり、iOS Safari の 301 永続キャッシュから旧 bookmark を解放した。
- 既存 ADR ([[000881]] / [[000883]] / [[000884]]) と Safari connection recovery hook の対策はそのまま温存しており、二重対応にはなっていない。
- 純粋 helper を `safari-error-utils.ts` に切り出したので、サーバ / クライアント / 将来の observability gateway から同じ判定を共有できる。

### Cons / Tradeoffs

- `visual-preview` SSR が deferred Promise を捨てたぶん、TTFB が `loadFirstArticle` の完走 (上限 6s) を待つ。LCP は feed-level OGP proxy URL 経由で別途確保されているため最悪 +200-400ms の TTI 劣化想定だが、実機検証は今後 1 週間で計測する。
- `hooks.client.ts` の自動 reload 発火は **ユーザの編集中フォーム** を巻き戻すリスクがある。`sessionStorage` カウンタで 3 回上限を入れたが、reload 直前に未保存状態を保護する仕組みは入れていない (本リポジトリの主要画面は読み中心で副作用が低いため許容)。
- `version.pollInterval = 5min` で `/__version.json` への定期リクエストが増える (HTML keep-alive と相乗りなのでコストはごく小さい)。
- nginx の `/ory/` だけリテラル `"upgrade"` を残したため、将来 Kratos に WebSocket / SSE を入れた場合に再考の余地が残る。テストで「`/api/` と `/` は map 経由」「`/ory/` は対象外」を assert している。
- `301 → 303` 変更で、特殊な SEO / 外部リンク経路で 301 cache を期待しているケースは外す。本リポジトリは認証付きアプリで該当する SEO 経路がなく、影響なし。

## 観測フォロー

- ClickHouse `otel_error_logs` に `safariBucket='ios-safari' AND chunkHash IS NOT NULL AND count() > 5 / 5min` のアラートルールを次 PR で追加する。
