# ADR-000557: Recall Rail の articles fallback を usecase 層で実装する

## Status

Accepted

## Date

2026-03-24

## Affected Services

- alt-backend (Go)

## Context

Knowledge Home の Recall Rail に `article:9574e59a-cd9c-4c2e-a594-e5fa598addd1` のような生の `item_key` がタイトルの代わりに表示される問題が再発した。

[[000479]] で「`knowledge_home_items` が欠落する場合に `articles` テーブルへ fallback する」方針が accepted されたが、その後の Knowledge Sovereign 移行により `articles` テーブル（alt-backend DB）と `knowledge_home_items`（sovereign DB）が異なるデータベースに分離された。Sovereign の `GetRecallCandidates` クエリは `LEFT JOIN knowledge_home_items` のみを行い、`articles` への SQL JOIN は DB 境界を越えるため不可能になった。

この結果、以下の条件で `RecallCandidate.Item = nil` が返り、フロントエンドが `candidate.item?.title ?? candidate.itemKey` の fallback で生キーを表示していた。

- projection lag により recall projection と home projection の到着順がずれる
- 再投影・部分復旧の途中で `recall_candidate_view` が先に可視化される
- 古い article が Recall candidate に残るが Home item として投影されていない

CQRS / Event Sourcing における projection lag は既知の構造的課題であり、Application/Usecase 層での "compensating read"（二次補完パターン）が業界標準のアプローチとして推奨されている。

## Decision

**Recall Rail の articles fallback を alt-backend の `RecallRailUsecase` 層で実装し、Sovereign のサービス境界を侵食しない方針を採用する。**

具体的には以下を採用した。

1. **新規ポート `ArticleFallbackPort`** を `recall_candidate_port` パッケージに定義。`GetArticleTitleAndLink(ctx, articleID) (title, link string, publishedAt *time.Time, err error)` のシグネチャで、title / url / published_at のみを返す軽量インターフェースとした。
2. **新規ドライバメソッド `GetArticleTitleAndURL`** を `alt_db` に追加。`SELECT title, url, published_at FROM articles WHERE id = $1 AND deleted_at IS NULL` の軽量クエリで、content の TOAST decompression コストを回避する。
3. **`RecallRailUsecase.Execute()` に enrichment ループ**を追加。Sovereign から取得した candidate のうち `Item == nil` かつ `article:` prefix のものだけを対象に、articles テーブルから最小限の `KnowledgeHomeItem` を構築する。
4. **Graceful degradation**: fallback 失敗時はエラーを伝播せず `slog.WarnContext` で記録し、`Item = nil` のまま返す。article が見つからない場合も同様。
5. **SummaryState = missing**: fallback item は summary / tags / why を推測で埋めない。[[000479]] の方針に準拠。
6. **nilable fallback port**: `fallbackPort` が nil の場合は従来動作のまま（後方互換）。

### 検討した代替案

- **Sovereign 内で `articles` テーブルを参照する**
  Sovereign は自身の DB のみを管理する single-owner サービスであり、alt-backend の articles テーブルへの直接アクセスはサービス境界を侵食する。不採用。

- **既存の `GetArticleContentPort` を再利用する**
  `SELECT id, title, content, url, user_id` で content（TEXT 型）も取得するため、TOAST decompression のオーバーヘッドが発生する。published_at も取得できない。専用の軽量クエリを新設する方が適切と判断した。

- **recall_candidate_view に article 情報を非正規化する**
  Recall projector の実行時に KHI が存在しないケース（projection lag の本質）を解決できない。非正規化先のデータが同じ理由で欠落するため不採用。

- **バッチクエリ（`WHERE id IN (...)`）に変更する**
  ベストプラクティスではバッチが推奨されるが、対象は最大 20 件中の KHI 欠落分（通常 1-3 件）であり、ローカル DB へのクエリのため N+1 のオーバーヘッドは軽微。現時点では個別ルックアップで十分と判断し、バッチ化は将来のリファクタリング候補として残す。

## Consequences

### Pros

- `knowledge_home_items` が欠落していても Recall Rail がタイトルとリンクを表示できる
- Sovereign のサービス境界を侵食せず、Clean Architecture の依存方向を維持
- 軽量クエリにより content ロードのオーバーヘッドを回避
- fallback port の nil 許容により既存コードとの後方互換を確保
- projection lag / 部分復旧 / 再投影のいずれのシナリオでも UX が劣化しにくい

### Cons / Tradeoffs

- fallback item には summary / tags / why がないため、KHI 由来の item より表示情報が貧弱
- Recall candidate の item shape が「完全 item」と「最小 fallback item」の 2 段階になる
- article 以外の item type（`recap_anchor:`, `pulse_anchor:`）にはこの fallback を使えない
- 個別ルックアップのため、KHI 欠落が多数の場合はクエリ数が増加する（将来バッチ化で対応可能）
- `RecallRailUsecase` が articles DB への依存を持つようになり、usecase の責務がやや広がる
