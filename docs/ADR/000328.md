# PgBouncer接続デススパイラルの修正 - トランザクションリーク・bytea誤エンコード・タイムアウト設定

## ステータス

Accepted

## コンテキスト

ADR-327でPgBouncerを導入後、UIが「Loading Alt」のまま停止する障害が発生した。alt-backendの全DBクエリが `context deadline exceeded` で失敗し、PgBouncerの統計が `0 xacts/s, 0 queries/s` を示していた。

### 障害タイムライン（ログから再構成）

| 時刻 | イベント |
|------|---------|
| 10:41:08 | `BEGIN` + `INSERT INTO outbox_events` → `ERROR: invalid input syntax for type json` |
| 10:41:38 | （30秒後）PostgreSQL: `FATAL: terminating connection due to idle-in-transaction timeout` |
| 10:42:47 | PgBouncer: `server conn crashed?` → 全4サーバー接続がクラッシュ |
| 10:42:47〜 | PgBouncer: `0 xacts/s, 0 queries/s`（完全停止） |
| 以降全て | alt-backend: `context deadline exceeded`（UIが「Loading Alt」で停止） |

### 根本原因の連鎖

3つの独立したバグが連鎖してデススパイラルを形成していた:

**バグ1: `[]byte` → bytea誤エンコード**

`save_outbox_event_driver.go` で `payload []byte` をそのまま `tx.Exec` に渡していた。pgx `QueryExecModeSimpleProtocol`（ADR-327で導入）では `[]byte` をbytea hex形式（`\x7b22...`）にエンコードする。`outbox_events.payload` カラムの型は `json` であるため、PostgreSQLが `invalid input syntax for type json` で拒否。

```go
// Before: []byte はSimpleProtocolでbytea hexにエンコードされる
tx.Exec(ctx, query, eventType, payload)  // payload は []byte

// After: string()変換でJSON文字列として送信
tx.Exec(ctx, query, eventType, string(payload))
```

**バグ2: 変数シャドーイングによるROLLBACK漏れ**

`save_article_driver.go` のトランザクション管理:

```go
// Line 82: 関数スコープの err
tx, err := r.pool.Begin(ctx)

// Line 86-92: defer で関数スコープの err を参照
defer func() {
    if err != nil {  // ← 関数スコープの err を参照
        tx.Rollback(ctx)
    }
}()

// Line 108: ブロックスコープの err を新規作成（:= によるシャドーイング）
if err := r.SaveOutboxEventWithTx(...); err != nil {
    return "", err  // ← ブロックスコープの err で return
}
```

`SaveOutboxEventWithTx` がエラーを返しても、`:=` により**ブロックスコープの `err`** が作成される。deferred関数が参照する**関数スコープの `err`** は `json.Marshal` 成功時点のnil。結果、`ROLLBACK`が実行されない。

**バグ3: `idle_in_transaction_session_timeout = 30s` が攻撃的すぎる**

ROLLBACKされなかったトランザクションが「aborted」状態のまま30秒放置される。PostgreSQLがFATALで接続を強制切断。PgBouncer transaction poolingでは、サーバー接続のFATAL切断がクライアント接続にもカスケードし、プール全体が壊死する。

### カスケード図

```
[]byte → bytea hex エンコード
  → PostgreSQL JSON parse error
    → トランザクション aborted状態
      → 変数シャドーイングで ROLLBACK 漏れ
        → 30秒後 idle_in_transaction_session_timeout FATAL
          → PgBouncer server conn crashed
            → クライアント接続もカスケード切断
              → pgxpool 接続枯渇
                → 全クエリ context deadline exceeded
                  → UI "Loading Alt" 停止
```

## 決定事項

### 修正1: `string(payload)` によるJSON正常送信

`save_outbox_event_driver.go` で `[]byte` を `string()` 変換してからExecに渡す。SimpleProtocolでは `string` 型はテキストとしてそのまま送信されるため、PostgreSQLのJSON parserが正常にパースできる。

### 修正2: 無条件 `defer tx.Rollback(ctx)` パターン

条件付きdefer rollbackを無条件 `defer tx.Rollback(ctx)` に変更。これはGoの標準パターンで、`tx.Commit()` 後の `tx.Rollback()` は `pgx.ErrTxClosed` を返すだけのno-op。変数シャドーイングに依存しない安全なパターン。

同ファイル内の `FetchAndLockPendingOutboxEvents` や `UpsertArticleTags` は既にこのパターンを使用していたが、`SaveArticle` だけが条件付きパターンだった。

### 修正3: `idle_in_transaction_session_timeout` を5分に延長

30秒→5分に変更。PgBouncer transaction poolingではサーバー接続のライフサイクルがPgBouncerに管理されるため、短すぎるタイムアウトがカスケード障害を引き起こすリスクがある。5分あればアプリケーション側のタイムアウト（connect_timeout=90s）で先にエラーハンドリングされる。

### 修正4: PgBouncerヘルスチェックのユーザー指定

`pg_isready -h localhost -p 6432` にユーザー指定がなく、OSユーザー `postgres` でPgBouncerに接続を試みていた。PgBouncerのuserlistに `postgres` がないため、5秒ごとに `no such user: postgres` のログが発生。`-U ${DB_USER}` を追加。

## 結果・効果

### メリット

- **デススパイラルの根本解消**: 3つのバグを全て修正し、カスケード障害の連鎖を断ち切る
- **トランザクション安全性の向上**: 無条件rollbackパターンにより、どのような失敗パターンでもトランザクションがリークしない
- **SimpleProtocol互換性の確保**: `[]byte` → `string` 変換により、pgx SimpleProtocolとJSON型カラムの互換性問題を解消
- **PgBouncer安定性の向上**: タイムアウト延長とヘルスチェック修正により、PgBouncerの運用安定性が向上
- **ログノイズの削減**: `no such user: postgres` が5秒ごとに出力されなくなる

### デメリット・トレードオフ

- **`idle_in_transaction_session_timeout = 5min`**: トランザクションリークの検出が30秒→5分に遅延する。ただし今回の修正でリーク自体が解消されるため、タイムアウトは最終防衛ラインとしての役割に留まる
- **`string(payload)` 変換**: `[]byte` → `string` の変換コスト。JSON payloadは通常数KB以下であり、パフォーマンスへの影響は無視可能

### pgx SimpleProtocolの型エンコード挙動

| Go型 | SimpleProtocolでの送信形式 | PostgreSQL受信 |
|------|---------------------------|----------------|
| `string` | テキストそのまま | JSON型カラムに正常INSERT |
| `[]byte` | bytea hex形式 (`\x7b22...`) | JSON型カラムにparse error |
| `json.RawMessage` (= `[]byte`) | bytea hex形式 | JSON型カラムにparse error |

### Goの変数シャドーイングパターン

```go
// 危険: ブロックスコープの err がシャドーイング
func example() error {
    tx, err := pool.Begin(ctx)
    defer func() {
        if err != nil { tx.Rollback(ctx) }  // 関数スコープの err を参照
    }()

    if err := doSomething(tx); err != nil {  // ブロックスコープの err
        return err  // return時、関数スコープの err は nil のまま → ROLLBACK漏れ
    }
    return tx.Commit(ctx)
}

// 安全: 無条件 defer rollback
func example() error {
    tx, err := pool.Begin(ctx)
    defer tx.Rollback(ctx)  // Commit後はno-op

    if err := doSomething(tx); err != nil {
        return err  // defer により必ず Rollback が実行される
    }
    return tx.Commit(ctx)
}
```
