# ADR-000770: news-creator Hurl E2E Phase 3 で RAG 系 (expand-query / chat / plan-query) をカバーする

## Status

Accepted

## Date

2026-04-18

## Affected Services

- news-creator (Python 3.14 / FastAPI) — ランタイムは無変更。Hurl E2E スイートに 3 シナリオ (`/api/v1/expand-query`、`/api/chat` 非ストリーミング、`/api/v1/plan-query`) を追加
- news-creator-ollama-stub (Python+FastAPI) — `/api/generate` のプレーン応答を複数行 ASCII 化、`/api/chat` の非 format 応答を `QueryPlan` JSON に切り替え

## Context

[[000768]] (Phase 1) と [[000769]] (Phase 2) で news-creator の `/health`、`/queue/status`、summarize/generate/recap 経路までを Hurl で覆った。Phase 3 は rag-orchestrator (Ask Augur) が叩く 3 経路を取り込む段である:

- `/api/v1/expand-query` — `expand_query_usecase` が `/api/generate` を叩き、改行区切りで複数のクエリを取得する。stub の Phase 2 応答 (1 行プレーン文字列) では line split 後の有効行が 1 本しか出ず `expanded_queries` count assertion が安定しない。
- `/api/chat` 非ストリーミング — chat-proxy パスの smoke。Morning Letter / 一般 chat の両方が利用する。
- `/api/v1/plan-query` — `plan_query_usecase` が `/api/chat` (`format` フィールド無し) を叩き、`message.content` を `QueryPlan` Pydantic モデルにパースする。stub の Phase 2 応答 (`format` 無しは plain テキスト) ではパース失敗 → fallback path に落ちる。fallback ではなく LLM 成功経路を CI でガードしたい。

これら 3 経路を一度に取り込むため、stub の応答ロジックを再設計する必要が出た。

`/v1/rerank` も Phase 3 の元計画には含めていたが、`RerankUsecase` は `sentence-transformers/CrossEncoder("BAAI/bge-reranker-v2-m3")` を遅延ロードする。このモデル (約 568 MB) は HuggingFace Hub からの初回ダウンロードを要し、`alt-staging` ネットワークの `internal: true` フラグがそれを遮断する。`HF_HOME` への事前 cache 投入か、news-creator イメージへのモデル同梱、もしくは egress 許可が必要で、いずれも独立した変更となるため Phase 3 では除外し将来 Phase に送る。

## Decision

### stub の応答分岐ロジックを「リクエスト形状ベース」に再設計する

`compose/news-creator-ollama-stub/app.py` を v0.3.0 に更新し、`format` フィールドの有無 + エンドポイントの組み合わせで応答を選ぶ:

| エンドポイント | `format` | 応答 |
|---|---|---|
| `/api/generate` | 有 | `RecapSummary` JSON (recap fallback path) |
| `/api/generate` | 無 | 複数行 ASCII (`stub expansion alpha keyword` 等 3 行) |
| `/api/chat` | 有 | `RecapSummary` JSON (recap chat path) |
| `/api/chat` | 無 | `QueryPlan` JSON (plan-query path、generic chat にも benign) |

複数行 ASCII 応答は同時に 2 つの呼び出し元を満たす:

- summarize_usecase は `raw_summary = llm_response.response` を `_clean_summary_text` 経由で `summary` に格納する。複数行であっても文字列として有効、`len(stripped) >= 10` ガードを通過。
- expand_query_usecase は `_parse_expansion_lines(raw_text)` で改行 split → dedup → instruction-leak filter を実施。stub の 3 行は ASCII で語彙が分散しているため 3 件全部が `expanded_queries` に入る。

`QueryPlan` JSON は `plan_query_usecase._extract_json` + `QueryPlan(**parsed)` を満たし、generic chat smoke (`08-chat-non-stream.hurl`) では `message.content` が文字列であることだけを検査するので副作用なし。

### Phase 3 シナリオ 3 本を追加する

| ファイル | 主要アサーション |
|---|---|
| `07-expand-query.hurl` | `{expanded_queries[], original_query, model}`、`expanded_queries` count >= 1 |
| `08-chat-non-stream.hurl` | `{model, message: {role, content}, done}`、`done == true` |
| `09-plan-query.hurl` | `{plan: {reasoning, resolved_query, search_queries[], intent, retrieval_policy, answer_format, should_clarify, topic_entities[]}, original_query, model}`、`should_clarify == false` |

新規フィクスチャ:

- `expand-query-request.json` — `japanese_count: 2, english_count: 2, priority: high`
- `chat-non-stream-request.json` — 1 ターン + `stream: false`
- `plan-query-request.json` — 短い日本語クエリ + `priority: high`

### `/v1/rerank` を Phase 3 から除外する

`RerankUsecase._get_cross_encoder` が `sentence_transformers.CrossEncoder` を初回呼び出し時にロードし、内部で `~/.cache/huggingface/` に 568 MB のモデルを HF Hub からダウンロードする。`alt-staging` の `internal: true` ネットワーク (DOC: [[000763]] パッケージ宣言) では DNS とインターネット egress が共に塞がれているため、これは確実に失敗する。

代替案:

- **HF_HOME を事前 cache 化** — Dockerfile.app に追加レイヤが必要 (本番イメージ膨張、本 ADR スコープ外)
- **alt-news-creator イメージにモデルを焼き込む** — tag-generator が SBERT をバンドルしているのと同じ手法だが、独立 ADR で管理すべきサイズ・運用変更
- **stub に /v1/rerank を持たせる** — rerank はハンドラ層で `RerankUsecase` を直呼びしており、Ollama proxy ではない。stub では代用不可
- **本 Phase で除外し将来 Phase に送る** — 採用

README と本 ADR の "Out of scope" に明示。

### 検討した代替案

- **stub にプロンプト sniffing を入れる** — 却下。プロンプト本文に依存するロジックは fragile で、prompt template 変更で回帰する。`format` フィールドという明示的シグナルの方が安定。
- **plan-query を fallback path で検査する** — 却下。fallback は LLM 失敗時の防護網であり、CI で常時 fallback を踏むと LLM 成功経路の回帰を見逃す。
- **expand-query 用の separate prompt detection** — 却下。複数行 ASCII を返せば summarize/expand-query の両方が満たせるため、分岐不要。

## Consequences

### Pros

- rag-orchestrator (Ask Augur) が依存する 3 経路すべての HTTP shape + usecase 解析後の正常系シェイプを CI で gate できるようになった。
- stub の分岐は引き続き「`format` の有無」+「エンドポイント」のみ。条件分岐 4 通り全部が小さく、可読・テスト容易。
- Phase 3 を含めた Hurl スイート全体は 10 シナリオ・55 ms (ローカル計測) で完了。コンテナ再ビルドは含まない。
- rerank の HF 依存を Phase 3 で隔離したことで、将来 rerank scenario を追加するときに「モデルバンドル戦略」だけを独立に決めれば良くなった (本 Phase に巻き込まれない)。

### Cons / Tradeoffs

- stub は `/api/chat` 非 format 呼び出しに対して常に `QueryPlan` JSON を返す。chat-proxy 経路 (Phase 4 でストリーミング含めて拡充予定) でも同じ JSON が透過し続ける。生成内容の意味的検証は本スイートの守備範囲外なので問題ないが、stub レポートを目視した人が違和感を覚える可能性は残る。README に明記済み。
- expand-query の stub 応答は ASCII (`stub expansion alpha keyword` 等)。本番では日本語 + 英語の混合クエリが返ることが期待されるが、`expanded_queries` の count と型のみ検査するため shape gate としては成立。多言語性の検証は別途 unit/contract テストに委ねる。
- `/v1/rerank` は Phase 3 で未カバー。本番では rag-orchestrator から呼ばれる経路が CI でガードされない状態が続く。Phase 5 (案: `bge-reranker-v2-m3` を news-creator イメージにバンドル) で解消する想定。
