# ADR-000458: Knowledge Home 投影修正の検証を targeted rebuild とローカル health probe で標準化する

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-backend (Go)
- alt-butterfly-facade (Go)
- alt-frontend-sv (SvelteKit)

## Context

Knowledge Home の dismiss 再表示不具合は、コード修正だけでは十分ではなかった。実際には migration 適用状況、`alt-backend` の再起動、BFF の疎通、SvelteKit フロントエンドの再ビルド結果がそろって初めて、ユーザーがリロード後の挙動を確認できる。

今回の検証では、関連サービスのみを再ビルドして再起動し、次の 3 点を確認した。

- migration コンテナが正常終了し、追加 migration の再適用が不要な状態であること
- `alt-backend`、`alt-butterfly-facade`、`alt-frontend-sv` が `healthy` で起動していること
- ローカル health probe が成功し、UI 経由の確認前に最低限の runtime 疎通が担保されていること

Knowledge Home は projector、BFF、UI をまたぐ機能であり、単一サービスの unit test だけでは runtime の成立性を説明しきれない。

## Decision

Knowledge Home の projection / visibility 修正は、実装完了後に targeted rebuild とローカル health probe を必須の確認手順とする。

標準手順は以下とする。

1. 影響サービスを `docker compose -f compose/compose.yaml -p alt up --build -d ...` でピンポイント再ビルドする
2. `docker compose ... ps` または `ps -a` で常駐系が `healthy`、one-shot migration が `Exited (0)` であることを確認する
3. `migrate`、`alt-backend`、`alt-butterfly-facade`、`alt-frontend-sv` の起動ログを確認し、migration 完了と server 起動を確認する
4. `http://localhost:9000/v1/health`、`http://localhost:9250/health`、`http://localhost/health` のローカル health probe を通す
5. その結果を ADR に残し、修正がコード上だけでなく runtime 上でも成立したことを記録する

この手順は、本番相当の完全 E2E を置き換えるものではないが、projection 系の回帰が「コードは正しいがコンテナ未反映」「migration 未適用」「BFF だけ古い」状態で見逃されることを減らす。

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|--------|------|-----------|
| `go test ./...` のみで検証を完了とする | 速い | migration 適用、BFF 疎通、SvelteKit build 成否を確認できない |
| 常にフルスタック全再起動を行う | 網羅性は高い | 変更範囲に対して重く、検証の反復速度が落ちる |
| 手元確認は行うが ADR には残さない | 実務上は可能 | 後から「どこまで runtime 検証したか」が追えず、再発時の比較材料が残らない |

## Consequences

### Pros

- projection 系の修正について、コード変更と runtime 反映を切り分けずに確認できる
- `migrate` の状態、BFF 疎通、frontend build 成否を同じ手順で残せる
- 後続の障害調査時に「どのサービスを再ビルドし、どの health probe を通したか」を ADR で追跡できる
- フルスタック全再起動より軽く、変更範囲に応じた検証を回しやすい

### Cons / Tradeoffs

- local compose 環境に依存するため、端末差異や既存コンテナ状態の影響を受ける
- health probe 成功だけでは feature-level の正しさまでは保証しない
- targeted rebuild の対象サービス選定を誤ると、古い依存コンテナが残る可能性がある
