# ADR-000640: Gemma 4 チャットテンプレート移行で全プロンプトトークンを一括置換する

## Status

Accepted

## Date

2026-04-07

## Affected Services

- news-creator (Python) — Recap 要約、記事要約、ストップトークン、Modelfile、出力クリーンアップ
- pre-processor (Go) — QualityChecker の JudgeTemplate、ストップトークン
- rag-orchestrator (Go) — コメントのみ（実コードは `/api/chat` 使用済み）

## Context

[[000613]] で Gemma 3 から Gemma 4:E4B Q4_K_M へ全サービスの LLM モデルを移行した。
移行時点では、チャットテンプレートのターン区切りトークンは Gemma 3/4 で共通（`<start_of_turn>` / `<end_of_turn>`）と想定していた。

しかし、移行後に Recap 生成品質が劇的に悪化。調査の結果、Google 公式ドキュメント
（[ai.google.dev/gemma/docs/core/prompt-formatting-gemma4](https://ai.google.dev/gemma/docs/core/prompt-formatting-gemma4)）
で **Gemma 4 はチャットテンプレートのトークンを変更している**ことが判明した:

| | Gemma 3 以前 | Gemma 4 |
|---|---|---|
| ターン開始 | `<start_of_turn>` | `<\|turn>` |
| ターン終了 | `<end_of_turn>` | `<turn\|>` |
| システムロール | user turn に注入 | ネイティブサポート (`<\|turn>system`) |
| 思考モード | なし | `<\|think\|>`, `<\|channel>thought`...`<channel\|>` |

さらに、以下の一次ソースすべてで同一のトークン形式が確認された:

- HuggingFace `chat_template.jinja`（google/gemma-4-12b-it）
- Ollama `model/renderers/gemma4.go` ソースコード
- llama.cpp PR #21309, #21326, #21418

### 品質悪化の根本原因

1. **トークン不一致**: `raw=True` でモデルに直接送信されるプロンプトが旧形式トークンを使用 → Gemma 4 がターン構造を認識不能
2. **ストップトークン不一致**: `<end_of_turn>` で停止設定 → Gemma 4 は `<turn\|>` を生成するため `num_predict` 上限まで走行しゴミ出力を生成
3. **思考トークン漏出リスク**: Gemma 4 が `raw=True` モードでも `<\|channel>thought` ブロックを自発的に生成し、JSON パースを破壊する可能性

## Decision

**`raw=True` を維持したまま、全ハードコードトークンを Gemma 4 形式に一括置換する。**

### 検討した代替案

| 案 | 概要 | 評価 |
|---|---|---|
| A. トークン一括置換（採用） | 全プロンプト・Modelfile・ストップトークンを新形式に更新 | 変更が局所的で即効性あり |
| B. `/api/chat + think=false` 全面移行 | Ollama にテンプレート処理を委譲し、ハードコードトークンを排除 | gateway/driver/usecase の 3 層に跨る大規模変更。後続タスクとする |

案 A を採用した理由:
- 即時の品質回復が最優先
- `_wrap_gemma_prompt()` 1 メソッドで Recap 全パスが修正される集約設計
- 案 B は [[000614]] の `/api/chat` 移行パターンを踏襲できるが、Recap のバッチパスは `generate_raw()` を深く使用しており影響範囲が大きい

### 実装の概要

**3 サービス・14 ファイルを修正:**

#### news-creator (Python) — 11 ファイル

| ファイル | 変更内容 |
|---------|---------|
| `usecase/recap_summary_usecase.py` | `_wrap_gemma_prompt()` トークン置換 + JSON パーサーに思考ブロックフィルタ追加 |
| `domain/prompts.py` | 3 テンプレート（SUMMARY / CHUNK_SUMMARY / RECAP_CLUSTER_SUMMARY）のトークン置換 |
| `config/config.py` | デフォルトストップトークンを `<turn\|>` に変更 |
| `Modelfile.gemma4-e4b-q4km` | テンプレート全体を Gemma 4 形式に更新 + `PARAMETER stop` 変更 |
| `usecase/summarize_usecase.py` | 出力クリーンアップに Gemma 4 トークン + 思考ブロック除去を追加（3 箇所） |
| `gateway/ollama_gateway.py` | コメント更新 |
| `scripts/benchmark_*.py` (2 本) | ストップトークン更新 |

#### pre-processor (Go) — 2 ファイル

| ファイル | 変更内容 |
|---------|---------|
| `quality-checker/quality_judger.go` | `JudgeTemplate` のトークン置換 + ストップトークンに `<turn\|>` 追加 |

#### rag-orchestrator (Go) — 1 ファイル

| ファイル | 変更内容 |
|---------|---------|
| `usecase/prompt_builder.go` | コメントのみ（実コードは `domain.Message` 経由で `/api/chat` 使用済み） |

### 防御策: 思考ブロックフィルタ

Gemma 4 が `raw=True` モードでも思考トークンを自発的に生成する可能性への防御として、以下に思考ブロック除去を追加:

- `_parse_summary_json()` — Recap 最終要約の JSON パース前
- `_parse_intermediate_summary_json()` — 階層要約の中間パース前
- `_clean_summary_text()` — 記事要約の出力クリーンアップ
- ストリームフィルタの `ignored_tokens` セット

正規表現 `<\|channel>thought.*?<channel\|>` で `re.DOTALL` 付きマッチを実行。

### 後方互換

出力クリーンアップコードでは旧トークン（`<start_of_turn>` / `<end_of_turn>`）の `.replace()` を残し、
過去に生成・キャッシュされた出力の処理を継続する。

## Consequences

### Pros

- Recap / Summarize / QualityChecker の品質が即時回復する
- ストップトークン一致により、ゴミ出力の生成が停止する
- Gemma 4 の思考モード漏出に対する防御が確立される
- TDD で全変更が検証済み（news-creator 403 tests passed、pre-processor 全パッケージ OK）

### Cons / Tradeoffs

- `raw=True` 依存を維持しているため、次のモデル移行時にも同様のトークン更新が必要
- `/api/chat` 移行（案 B）を行えば Ollama がテンプレート処理を吸収し、この問題クラスは解消される
- Modelfile テンプレート更新は `raw=True` 使用時には無視されるが、将来の `/api/chat` 移行に備えて更新済み

### 後続タスク

1. `/api/chat + think=false` への移行（[[000614]] パターン）— `raw=True` 依存の解消
2. Gemma 3 レガシートークンの完全削除 — クリーンアップコードから旧 `.replace()` を将来除去
3. knowledge-augur の Gemma 4 移行 — 現在 Gemma 3 のまま、別途移行計画が必要
