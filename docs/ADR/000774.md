# ADR-000774: recap-worker 下流 (recap-subworker / news-creator) を pki-agent reverse-proxy で mTLS サーバ化し tag-generator allowed peers を是正する

## Status

Accepted

## Date

2026-04-19

## Affected Services

- **recap-worker** — `compose/recap.yaml` で `SUBWORKER_BASE_URL` / `NEWS_CREATOR_BASE_URL` を `https://...:9443` に切替。`src/config.rs` に `validate_mtls_url_schemes` を追加し `MTLS_ENFORCE=true` 下で http URL を fail-closed で拒否する起動時アサーションを実装
- **recap-subworker** — `compose/pki.yaml` に `pki-agent-recap-subworker` reverse-proxy サイドカーを追加。plain HTTP 8002 を維持しつつ、外向きは `:9443` の mTLS で client cert を検証
- **news-creator** — `compose/pki.yaml` に `pki-agent-news-creator` reverse-proxy サイドカーを追加。plain HTTP 11434 を維持しつつ、`:9443` で mTLS 終端
- **tag-generator** — `compose/pki.yaml:252` の `PROXY_ALLOWED_PEERS` 既定値に `recap-worker` を追加し、recap-worker 由来の handshake で BadCertificate を返さないようにする

## Context

3days Recap が 2026-04-14 から 2026-04-19 にかけて 18 件連続で失敗していた。`recap_failed_tasks` の内訳:

| エラー区分 | 件数 | 直近 |
|---|---|---|
| `classification_0_results` | 8 | 2026-04-19 03:35 UTC |
| `not_found` (404) | 4 | 2026-04-16 17:00 UTC |
| `certificate_expired` | 3 | 2026-04-18 10:25 UTC |
| `deserialization_error` | 2 | 2026-04-17 12:38 UTC |
| `auth_error 401` | 1 | 2026-04-17 11:58 UTC |

`not_found` と `deserialization_error` は [[000759]] の REST/Connect-RPC ハイブリッド listener 化で、`certificate_expired` は [[000773]] の mTLS クライアント cert hot-reload で解消されていた。残った `classification_0_results` の 8 件 (44%) が **未解明**で、最新ジョブ `d5c15111-de1a-4a87-8e16-03f94856086d` のログには次が連続で出ていた:

```
classify-runs POST request failed for url
  (http://recap-subworker:8002/v1/classify-runs)
  builder error: URL scheme is not allowed
-> classification returned 0 results for 910 articles
-> manual recap job failed: label="3days"
```

切り分け:

- `recap-worker/src/clients/mtls.rs:217-224` の `build_mtls_client(...)` は `reqwest::Client::builder().https_only(true)` を立てている。これは `MTLS_ENFORCE=true` のとき app.rs:101-117 で `subworker_client` と `news_creator_client` の両方に渡される。
- `compose/recap.yaml:97` の `SUBWORKER_BASE_URL=http://recap-subworker:8002` は plain HTTP の URL を指定。
- `recap-subworker/recap_subworker/__main__.py` の uvicorn は plain HTTP の :8002 のみで listen し、mTLS リスナーも pki-agent サイドカーも未配置。

reqwest はリクエスト送信前のスキーム検証段階で `URL scheme is not allowed` を返し、5 チャンクすべてが空配列で fail-fast。同じ症状は news-creator (`http://news-creator:11434`) でも論理的に発火する経路だが、ジョブが先に subworker で止まるため可視化されていなかった。

並行して、recap-worker → tag-generator (`https://tag-generator:9443`) は `BadCertificate` を返していた。`compose/pki.yaml:252` の `PROXY_ALLOWED_PEERS` 既定値が `alt-butterfly-facade,alt-backend,search-indexer` で `recap-worker` を含まないため、tag-generator の pki-agent reverse-proxy が peer verification 段階で recap-worker の leaf cert を弾いていた。

経緯としては、commit `953666412` (2026-04-15, ADR-727 Phase 2 client-side enforcement) で recap-worker の outbound を `MTLS_ENFORCE=true` に切り替えた際、Python サービス (recap-subworker / news-creator) のサーバ側 mTLS 化が抜けたまま enable された。[[000759]] / [[000773]] の修正は別形態 (REST 配線漏れ / クライアント cert hot-reload) を解消したが、本件はそれらと **独立した第三形態の silent failure** として 18 件のジョブ失敗を生んだ。

イミュータブル不変条件 (`knowledge_events` / projector / `summary_versions` / `tag_set_versions`) への影響はない。本変更は transport 層 (compose 設定 + recap-worker config 起動時アサーション) のみ。

## Decision

recap-worker が `MTLS_ENFORCE=true` で叩くすべての下流に、**統一されたパターンとして pki-agent reverse-proxy mode を当てる**:

1. **recap-subworker サーバ化**: `compose/pki.yaml` に `pki-agent-recap-subworker` を追加し、`network_mode: "service:recap-subworker"` で同 netns に join。`PROXY_LISTEN=:9443` で client mTLS を終端し、`PROXY_UPSTREAM=http://127.0.0.1:8002` の uvicorn に loopback で proxy。`PROXY_ALLOWED_PEERS=${RECAP_SUBWORKER_MTLS_ALLOWED_PEERS:-recap-worker}` で peer verification を必須化。

2. **news-creator サーバ化**: 同パターンで `pki-agent-news-creator` を追加 (`PROXY_UPSTREAM=http://127.0.0.1:11434`)。allowed peers は `recap-worker,acolyte-orchestrator,rag-orchestrator,recap-evaluator` を初期値とし、後続の Python orchestrator が `MTLS_ENFORCE=true` に切り替わったときに compose を触らずに済むようにする。

3. **recap-worker URL 切替**: `compose/recap.yaml` で `NEWS_CREATOR_BASE_URL=https://news-creator:9443` / `SUBWORKER_BASE_URL=https://recap-subworker:9443` に変更。

4. **tag-generator allowed peers 是正**: `compose/pki.yaml:252` の `PROXY_ALLOWED_PEERS` 既定値に `recap-worker` を追加。

5. **fail-closed アサーション**: `recap-worker/src/config.rs` に `validate_mtls_url_schemes(...)` を新設し `Config::from_env()` で呼ぶ。`MTLS_ENFORCE=true` のときに `SUBWORKER_BASE_URL` / `NEWS_CREATOR_BASE_URL` が `https://` で始まらない場合 `ConfigError::Invalid` を返して起動拒否。次に同型 silent failure が混入しても、本番反映の起動時に確定的に検知できるようにする。

依存表現には compose v2.30+ の `depends_on.<parent>.{condition: service_healthy, restart: true}` を使う。これは [[000757]] / [[000759]] / [[PM-2026-030]] で確認された「`network_mode: service:X` サイドカーが親の recreate に追随しないと netns orphan になる」問題への既知対策。

### 検討した代替案

| 案 | 採用 | 却下理由 |
|---|---|---|
| **A: pki-agent reverse-proxy サイドカー** (採用) | ○ | tag-generator / acolyte-orchestrator で既に運用実績あり。Python サービス本体に SSL/TLS 設定を入れず、cert hot-reload の言語横断バグ ([[PM-2026-032]]) を再生産しない |
| B: recap-worker に subworker / news-creator 用だけ非 mTLS reqwest クライアントを別途構築 | × | mTLS 全面化の方針 ([[000727]] Phase 2) と矛盾。東西網内通信を保護しないため将来的に同じバグの根を残す |
| C: `clients/mtls.rs` の `https_only(true)` を外す | × | セキュリティ後退。mTLS を要求する経路と任意の http が同居でき、設定ミスを発見しにくくなる |
| D: Python uvicorn 側に `ssl_certfile` / `ssl_keyfile` を直書き | × | uvicorn は SSL context を hot-swap できず、cert 24h ローテのたびにプロセス再起動が必要。pki-agent サイドカーなら自身が cert mtime を監視し、uvicorn を停止せずに新 cert を反映できる |

## Consequences

### Pros

- 3days Recap が `classify-runs` 段階で fail-fast していた blocking バグが消え、`recap_outputs.window_days=3` の生成が再開する。
- 同じ pki-agent reverse-proxy パターンで mTLS サーバ化対象が tag-generator / acolyte-orchestrator / recap-subworker / news-creator の 4 サービスに揃い、新規 Python サービスを mTLS 網に組み込む際のテンプレートが固まる。
- `MTLS_ENFORCE=true` + http URL の組み合わせを起動時に確定的に弾けるようになり、次回同型ミスの平均検知時間 (TTD) が「ジョブ失敗が累積する数日」から「コンテナ起動の数秒」に短縮される。
- news-creator の allowed peers を 4 サービス分先回り宣言しているため、acolyte-orchestrator / rag-orchestrator / recap-evaluator の `MTLS_ENFORCE` 切り替え時に compose を再変更する必要がない。
- tag-generator の BadCertificate も同 PR で解消され、ログのノイズが減って観測性が改善する。

### Cons / Tradeoffs

- **plain HTTP 8002 / 11434 リスナーが残置**される。pki-agent サイドカーは netns 共有で `127.0.0.1` ループバックに proxy するため攻撃面は同 netns に閉じるが、厳密には listener を `127.0.0.1` 限定にバインドする方が望ましい。これは uvicorn の起動引数追加が必要で別 PR とする (Phase 2 として ADR を分離)。
- **証明書ローテ後の sidecar 再起動依存**。`pki-agent-recap-subworker` / `pki-agent-news-creator` はリーフ cert mtime を 5 分間隔で確認し新 cert を再ロードするが、TLS context の差し替えが内部で完全に hot-reload されているかは pki-agent 実装に依存。仕様逸脱があれば [[PM-2026-032]] と同型のバグが再発する余地が残る。
- **smoke テストでの実経路確認は引き続き未整備**。[[PM-2026-031]] Action Item #4 / [[PM-2026-032]] Action Item #5 として TODO のまま、本 ADR の射程外。本 ADR の起動時アサーションは「URL スキーム不一致」だけ早期検知できるが、handshake 段階での失敗 (BadCertificate / CertificateExpired) は依然 smoke 拡張が必要。
- **CERT_OWNER_UID/GID=0** で固定。pki-agent サイドカー本体が root で動いて cert を読むため問題は無いが、後段で uvicorn 側に cert を共有マウントする設計に移行する場合は `recap-subworker` (uid を `useradd -r` 自動割り当て) / `news-creator` (uid 1000) に合わせて変更が必要。
- **本 ADR では recap-evaluator / rag-orchestrator のサーバ化はスコープ外**。これらが将来 `MTLS_ENFORCE=true` の caller から呼ばれた場合、本 ADR と同形の追加サイドカーが必要になる。news-creator allowed peers の先回り宣言で compose 改変は不要だが、サイドカー追加 PR は別途必要。
