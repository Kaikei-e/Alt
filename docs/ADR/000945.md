# ADR-000945: alt-backend を dataplane / orchestrator / shared package へ物理分割する

## Status

Accepted

## Date

2026-07-15

## Affected Services

- alt-backend (Go)

## Context

`docs/review/architecturereview20260713.md` の F-06 finding が、alt-backend が「alt-db 所有の内部データ API (7 mTLS peer が `BackendInternalService` へ fan-in)」と「プロダクト・オーケストレータ (search-indexer / recap-worker / pre-processor / tag-generator への fan-out)」の 2 つの変更理由を同一バイナリに同居させ、4 本の双方向同期循環の中心になっていると指摘した。境界は `:9101`/`:9443` (Connect-RPC + mTLS) と REST (`:cfg.Server.Port`) というリスナー単位では既に物理分離していたが、`usecase`/`port`/`gateway`/`driver` は article/feed/recap/knowledge 等の **domain 軸**で構成されており、data-plane/orchestrator の境界はコード上どこにも可視化されていなかった。

実装前の分類調査 (DI 配線の実トレース、ディレクトリ名からの推測ではなく `di/*.go` → `connect/v2/server.go` → 各 handler.go までを辿った実証ベース) により、次が判明した:

- alt-backend→worker 方向の 4 循環はいずれも明確な orchestrator usecase (search / recap dashboard proxy / summarize / tag generation) に紐づいており、worker→alt-backend 方向は単一の `BackendInternalService` (加えて副次的な `rest/internal_handlers.go` の `/v1/internal/*`) という既存の data-plane 契約点に収束していた。「循環」はコードの絡み合いではなく、両方向のコードが同一バイナリ・同一 DI グラフに同居しているだけだった。
- **5 gateway (`internal_article_gateway`, `event_publisher_gateway`, `summary_version_gateway`, `fetch_tag_cloud_gateway`, `fetch_articles_by_tag_gateway`)・3 usecase (`create_summary_version_usecase`, `fetch_tag_cloud_usecase`, `fetch_articles_by_tag_usecase`)・5 port・3 driver (`alt_db`, `sovereign_client`, `mqhub_connect`) は、data-plane と orchestrator の両方の入口から同一の DI 構築済みインスタンスが再利用されていた**。特に `alt_db` は 59 gateway のほぼ全てが依存する単一 Postgres 接続プールであり、両軸を跨ぐ核そのものだった。

この事実により、当初計画していた「二分木への機械的な振り分け」は棄却した。共有インスタンスを無理に片側へ帰属させれば、もう一方からはパッケージ境界を跨ぐ import が発生するか、インスタンスを複製して DB プール/クライアントの単一性を壊すかのいずれかになる。

## Decision

**usecase・port・gateway・driver を `app/shared/`・`app/dataplane/`・`app/orchestrator/` の 3 系統へ物理移動する**ことを採用した。

- `app/shared/`: 上記の同一インスタンス共有が確認された 5 gateway・3 usecase・5 port・3 driver。
- `app/dataplane/`: `BackendInternalService` (`connect/v2/internal/` → `dataplane/connect/internalapi/`。Go の予約ディレクトリ名 `internal/` との可視性衝突を避けるため改名) からのみ到達する usecase/port/gateway/driver。
- `app/orchestrator/`: facade 向け REST/Connect-RPC、4 つの outbound worker 呼び出し usecase、バックグラウンドジョブ (`job/`) から到達する残り全て。

検討した代替案:
- **物理移動をせず handler 層の境界を docs + lint で宣言するだけの軽量案** — 費用対効果は良いが、「境界を明示化する」という F-06 の最低限の目的しか達成せず、import 単位での強制力がない。今回は見送った。
- **共有分を無理にどちらかへ寄せる案** — 上記の理由で棄却。

`rest/internal_handlers.go` (`/v1/internal/*`、2 エンドポイントの副次的な data-plane 入口) は、`rest` package 内の非公開ヘルパー (`utils.go` の `HandleError` 等) を共有しているため抽出コストが見合わず、`orchestrator/rest/` に意図的な例外として残置した。

`domain/`・`di/`・`config/`・`connect/v2/{codec,middleware,server.go}` 等の横断インフラ層は移動しなかった。これらは data-plane/orchestrator のどちらの business logic でもなく、composition root・技術基盤に属するため。

## Process 分離の判断基準 (今後の決定に向けた明文化)

本 ADR は **package 境界の明示化のみ**を実施し、プロセス分離 (別コンテナ化) はスコープ外とした。`alt_db`/`sovereign_client`/`mqhub_connect` が単一プロセス・単一 DB コネクションプールのまま残るため、orchestrator 側の障害 (goroutine leak、DB プール枯渇等) は data-plane 側 (7 mTLS peer が依存する worker RPC) を依然として道連れにしうる。fault-domain 分離は未達成であり、これは意図的な先送りである。

今後プロセス分離を判断する際は、次を計測してから決める (measurement-first。憶測での先行分離はしない):

1. **障害相関**: postmortem (`PM-2026-NNN`) で、orchestrator 専属コードの障害が同一ウィンドウで data-plane 側の worker RPC 可用性を道連れにした実例が記録されたか。
2. **リソース競合**: OTEL/Prometheus で、orchestrator 側トラフィックのスパイクと data-plane 側 RPC の p99 レイテンシ悪化に相関が見られるか (特に `alt_db` コネクションプールの待ち行列長)。
3. **デプロイ頻度の乖離**: orchestrator 側 (プロダクト機能、頻繁に変更) のデプロイが data-plane 側 (worker 契約、安定志向) の再起動・worker 側の再接続/backoff を頻繁に誘発しているか。
4. **スケーリング特性の非対称性**: orchestrator がユーザトラフィックに応じた水平スケールを必要とする一方、data-plane が `alt_db` プールサイズの都合で単一インスタンスに留めたい、といった相反する要求が顕在化したか。

分離を実施する場合の前提条件: `alt_db` の単一コネクションプール共有をどう解くか (pgbouncer 経由で 2 プロセスに分けるか、プールを分割するか) を先に決めること。単一マシン前提 (architecture-rules) の範囲内であれば、別マシンではなく別コンテナでの分離が妥当。

## Consequences

### Pros

- モジュール境界が import 文に直接現れるようになった (`di/container.go` の import 一覧を見るだけで、どの構築物が data-plane 向けか orchestrator 向けか判別できる)。
- 649 ファイルの移動を `go build`/`go vet`/`go test` (145 パッケージ) 全 green、`golangci-lint` の issue 数が移動前ベースライン (92 件) と完全一致 (新規 issue ゼロ) の状態で完了した。
- 将来の共有インスタンス誤認 (「orchestrator 専属だと思っていたら実は data-plane からも呼ばれていた」等) のリスクが、`app/shared/` という明示的な置き場によって可視化された。

### Cons / Tradeoffs

- **fault-domain 分離は未達成**。本決定単体では、同一バイナリ内の障害波及リスクは変わらない。
- `rest/internal_handlers.go` が `orchestrator/rest/` に例外的に残置されており、data-plane の完全な物理境界にはなっていない (意図的、2 エンドポイントのみの小さな残債)。
- `app/shared/` という第 3 のバケツが増えたことで、新規コード追加時に「data-plane か orchestrator か shared か」の判断コストが発生する。
