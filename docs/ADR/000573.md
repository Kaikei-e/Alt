# ADR-000573: alt-backend の Handler 分割・Setter 除去・エラー型整備を完了する

## Status

Accepted

## Date

2026-03-25

## Affected Services

- alt-backend (Go)

## Context

[[000572]] で実施した SOLID リファクタリングの Phase 1（Repository 分割）, Phase 2（ISP）, Phase 3（DI 分割）, Phase 7（Job Registry）に続き、残りの Phase 4（Handler 分割）, Phase 6（エラー改善）, Phase 8（Setter 除去）を完了させる必要があった。

特に以下の問題が残存していた:

1. **Connect-RPC Handler の肥大化**: `feeds/handler.go`（1,301 LOC）と `knowledge_home/handler.go`（1,056 LOC）がそれぞれ 15〜20 の RPC メソッドを1ファイルに集約しており、可読性・保守性が低下していた。
2. **Setter Injection の残存**: `RegisterFeedsUsecase`（5 Setter）、`TrackHomeActionUsecase`（2 Setter）、`GetKnowledgeHomeUsecase`（1 Setter）が構築後にポートを差し込むパターンを使用しており、依存関係が型シグネチャから不可視だった。
3. **Sentinel Error の不足**: `ErrRateLimited`, `ErrTimeout` 等のインフラ系エラーが `domain/errors.go` に定義されておらず、Handler のエラーマッピングが文字列比較に依存するリスクがあった。

## Decision

### Phase 4: Connect-RPC Handler のサブドメイン別ファイル分割

Handler struct とコンストラクタは `handler.go` に残し、RPC メソッドを機能別に分離する。Connect-RPC のインターフェース実装は struct に紐づくため、struct 自体は分割しない。

**feeds Handler の分割結果:**

| ファイル | LOC | 責務 |
|---|---|---|
| `handler.go` | 42 | Handler struct, NewHandler, 共有ヘルパー |
| `feed_read.go` | 380 | GetUnreadFeeds, GetAllFeeds, GetReadFeeds, GetFavoriteFeeds, SearchFeeds, ListSubscriptions |
| `feed_write.go` | 123 | MarkAsRead, Subscribe, Unsubscribe |
| `feed_stats.go` | 193 | GetFeedStats, GetDetailedFeedStats, GetUnreadCount, StreamFeedStats |
| `feed_summary.go` | 611 | StreamSummarize, resolveArticle, プリプロセッサ連携ヘルパー群 |

**knowledge_home Handler の分割結果:**

| ファイル | LOC | 責務 |
|---|---|---|
| `handler.go` | 252 | Handler struct, NewHandler, Proto 変換ヘルパー |
| `home_query.go` | 417 | GetKnowledgeHome, StreamKnowledgeHomeUpdates, ストリームヘルパー群 |
| `home_tracking.go` | 87 | TrackHomeItemsSeen, TrackHomeAction |
| `home_recall.go` | 156 | GetRecallRail, TrackRecallAction, StreamRecallRailUpdates |
| `home_lens.go` | 204 | CreateLens, UpdateLens, DeleteLens, ListLenses, SelectLens |

分割方針:
- 複数ファイルから呼ばれるヘルパー（Proto 変換、プロキシ URL 付与等）は `handler.go` に残す
- 単一ファイル内でのみ使われるプライベートヘルパーはメソッドと一緒に移動する
- ロジック変更は一切行わず、純粋なファイル分割のみ

### Phase 6: Sentinel Error の拡充

`domain/errors.go` に以下のエラー定数を追加した:

```go
// インフラ関連
ErrRateLimited, ErrServiceUnavailable, ErrTimeout, ErrDatabaseError

// 検索関連
ErrSearchUnavailable

// Knowledge Home 関連
ErrKnowledgeEventNotFound, ErrProjectionStale
```

既存の sentinel error（`ErrFeedNotFound`, `ErrArticleNotFound`, `ErrUnauthorized` 等）は変更なし。今後 Handler のエラーマッピングで `errors.Is()` による判定に統一する際の基盤となる。

### Phase 8: Setter Injection の全廃とコンストラクタ注入への統一

**RegisterFeedsUsecase（5 Setter → Options struct）:**

```go
// Before: 構築後に5つのSetterを呼ぶ
uc := NewRegisterFeedsUsecase(validate, registerLink, registerFeeds)
uc.SetFeedLinkIDResolver(altDB)
uc.SetFeedLinkAvailabilityPort(altDB)
uc.SetFeedPageInvalidator(cacheGw)
uc.SetSubscriptionPort(subscriptionGw)
uc.SetEventPublisher(eventPublisherGw)

// After: RegisterFeedsOpts で一括注入
uc := NewRegisterFeedsUsecase(validate, registerLink, registerFeeds, &RegisterFeedsOpts{
    FeedLinkIDResolver:   altDB,
    FeedLinkAvailability: altDB,
    FeedPageInvalidator:  cacheGw,
    SubscriptionPort:     subscriptionGw,
    EventPublisher:       eventPublisherGw,
})
```

5 つのオプショナル依存を `RegisterFeedsOpts` struct にまとめた。`nil` を渡すと全オプション無効（テスト時に便利）。必須の3引数はコンストラクタの位置引数として維持。

**TrackHomeActionUsecase（2 Setter → コンストラクタ引数拡張）:**

```go
// Before
uc := NewTrackHomeActionUsecase(userEventPort, knowledgePort, flagPort)
uc.SetRecallSignalPort(sovereignCli)
uc.SetDismissPort(sovereignCli, sovereignCli)

// After: 6引数のコンストラクタ
uc := NewTrackHomeActionUsecase(userEventPort, knowledgePort, flagPort,
    recallSignalPort, dismissPort, activeVersionPort)
```

**GetKnowledgeHomeUsecase（1 Setter → コンストラクタ引数追加）:**

```go
// Before
uc := NewGetKnowledgeHomeUsecase(items, digest, lens, freshness, count)
uc.SetTagHotspotPort(trendingTagsGw)

// After: 6引数のコンストラクタ
uc := NewGetKnowledgeHomeUsecase(items, digest, lens, freshness, count, trendingTagsGw)
```

DI モジュール側の変更:
- `feed_module.go`: `newFeedModule(infra, sub)` に `SubscriptionModule` 依存を追加。`container.go` でモジュール作成順序を調整（Subscription → Feed の順に）。
- `knowledge_module.go`: コンストラクタ呼び出しを更新し、Setter 呼び出し行を削除。

## Consequences

### Pros

- **Handler の可読性向上**: 最大 1,301 LOC → 最大 611 LOC。各ファイルが単一の機能群に集中し、コードナビゲーションが改善
- **依存関係の可視化**: 全 Usecase の依存が型シグネチャに明示。Setter 定義は 0 件に（`grep -r "func.*Set[A-Z].*Port" usecase/` で確認済み）
- **テストの明確化**: コンストラクタの引数で依存を制御できるため、テストの setup がシンプルに
- **エラーハンドリングの基盤**: sentinel error が整備され、今後のエラーマッピング一元化の準備完了
- **全 122 packages テスト GREEN**: リグレッションなし

### Cons / Tradeoffs

- **コンストラクタ引数の増加**: `TrackHomeActionUsecase` が 3→6 引数、`GetKnowledgeHomeUsecase` が 5→6 引数に。現時点ではオプションパターン（Functional Options）は不要と判断したが、今後さらに依存が増える場合は `RegisterFeedsOpts` のパターンに揃える必要がある
- **feed_summary.go の 611 LOC**: ストリーミング要約は SSE パース、プリプロセッサ連携、エラーマッピング等の複合ロジックを含むため、これ以上の分割は過分割になると判断。今後の機能追加時に再評価する
- **Handler の DIP 改善は未完了**: Handler は依然として `*di.ApplicationComponents` 全体に依存。Phase 5（次回実施予定）で個別 Usecase 注入に変更する
