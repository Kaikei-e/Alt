# ADR-000865: Knowledge Home projector の event payload wire schema を producer に揃え reproject-safety を回復する

## Status

Accepted

## Date

2026-04-28

## Affected Services

- alt-backend (Go) — projector の `articleCreatedPayload` struct tag を `json:"url"` に揃え、`knowledge_backfill_job.GenerateBackfillEvent` も追従。新規 wire-form contract test (`knowledge_projector_payload_contract_test.go`) を追加
- alt-frontend-sv (Svelte 5) — `KnowledgeCard.svelte` に `linkAvailable` derived state と `Archived · No source URL projected` kicker、`data-testid="kh-card-link-unavailable"`、`console.warn` 診断を追加

## Context

2026-04-28 のセッションで、operator が `/admin/knowledge-home` から Reproject (compare → swap) を実行した直後、Knowledge Home `/home` の article カードを click すると `DESK // ERROR — Article link is not available yet.` トーストが出る不具合が顕在化した。

経路を逆引きしたところ、原因は alt-backend 内の **event producer struct と consumer struct の JSON-tag drift** だった。

| Side | File:line | Field | JSON tag |
|---|---|---|---|
| Producer | `alt-backend/app/driver/mqhub_connect/client.go:77` | `URL string` | `json:"url"` |
| Producer (subset of `ArticleCreated`) | `search-indexer/app/consumer/event_handler.go:26` | `URL string` | `json:"url"` |
| Consumer (broken) | `alt-backend/app/job/knowledge_projector.go:273` | `Link string` | `json:"link"` |

projector の `json.Unmarshal` は wire 上に存在しない `"link"` キーを探して見つけられず、`payload.Link` は常に Go zero value (`""`) のままだった。

しかし live projection ではこの不具合は表面化しなかった。`knowledge-sovereign/app/driver/sovereign_db/repository.go:156` の merge-safe upsert に preserve-on-empty CASE 分岐が含まれているためである。

```sql
link = CASE WHEN EXCLUDED.link != '' THEN EXCLUDED.link ELSE knowledge_home_items.link END
```

live projection の既存行 (何らかの過去の backfill / 旧 dual-write 由来) には `link` が埋まっていたため、後続 event が空文字を持ってきても CASE preserve がそれを温存し続けていた。

Reproject は `(user_id, item_key, projection_version)` をユニークキーに持つため、新しい `projection_version` で初めて INSERT される行に preserve 対象は存在しない。結果として v_new の article 行が **全件** `link=''` となり、swap 後 FE は `getKnowledgeHome` レスポンスで全 article の `link` を空として受け取り、`/home/+page.svelte:181-188` のクリックハンドラが else 枝に落ちて error toast を表示した。

これは canonical immutable invariants §4 (Reproject-safe projector) 違反である。projector は event payload と stable な versioned resource のみから read model を再構築できなければならないが、実際は live projection 上の "歴史的 backfill 値" という disposable projection 内のローカル状態に依存していた。

なお、wire-form contract を検証する unit test は存在しなかった。既存の `TestKnowledgeProjectorJob_ArticleCreated_LinkPropagation` は consumer struct を marshal してから同じ consumer struct で unmarshal していたため、struct tag が自分自身と一致しているかしか確認できず drift を検出できない false-confidence パターンになっていた。

## Decision

projector の `articleCreatedPayload` を **producer の wire schema に合わせて** 修正する。具体的には:

1. **struct tag を揃える**: `Link string \`json:"link"\`` を `URL string \`json:"url"\`` に変更。site-of-use (`projectArticleCreated` 内で `domain.KnowledgeHomeItem.Link` を埋める箇所) は `Link: payload.URL` とする。projection の column 名 `link` は据え置き (DB schema は変更不要)。
2. **wire-form contract test を追加**: `alt-backend/app/job/knowledge_projector_payload_contract_test.go` に `TestArticleCreatedPayloadContract_LinkRoundTrips` を追加。canonical wire payload を **raw `map[string]any`** で組み立て、projector 経由で `KnowledgeHomeItem.Link` が round-trip するか asserts する。consumer struct で marshal しないため tag drift で必ず fail する。
3. **synthetic emitter も追従**: `alt-backend/app/job/knowledge_backfill_job.go:135` の struct 初期化子 (consumer struct を流用しているため) を `URL: link` に更新。
4. **producer↔consumer JSON-tag parity の audit を全 event 種別で実施**: `docs/review/knowledge-event-payload-tag-audit-2026-04-28.md` (gitignored local working note) に matrix を記録。drift があるのは `ArticleCreated` のみ、他は parity OK (`SummaryVersionCreated`、`TagSetVersionCreated`、`SummarySuperseded`、`TagSetSuperseded`、`ReasonMerged`、`HomeItemOpened/Dismissed`、`KnowledgeLoopTransition`)。
5. **FE 側 defense in depth**: `alt-frontend-sv/src/lib/components/knowledge-home/KnowledgeCard.svelte` に `linkAvailable = $derived(item.itemType === "article" ? !!item.link : true)` と、empty 時に表示する `Archived · No source URL projected` kicker、`data-testid="kh-card-link-unavailable"`、`data-link-unavailable="true"` 属性、`console.warn` once-per-item-key 診断を追加。toast 文言 (`Article link is not available yet.`) は変更しない (terse English, no emoji)。FE 防御は backend contract の代替ではなく、同種 regression が将来再発したときに silent 化させない可視化層。

### 検討した代替案

- **dual-tag (`json:"url,link"`) で両キーを受け入れる**: 却下。Go の標準 `encoding/json` は struct tag に複数キーを許容しないため誤った提案だが、仮にカスタム `UnmarshalJSON` で alias を実装したとしても、producer と consumer の片方を直しても silent に通ってしまい、将来別箇所で drift しても気づけない。「fail loudly」を優先し canonical key 1 つに固定する。
- **producer 側 (`mqhub_connect.ArticleCreatedPayload.URL`) を `json:"link"` に変更**: 却下。`knowledge_events` は append-only で、過去の event は全て `"url"` キーで marshal 済み。producer の tag を変えると historical event を replay できなくなり、append-first invariant §1 が破綻する。
- **upsert の CASE preserve 分岐を削除して常に EXCLUDED.link で上書きさせる**: 却下。CASE preserve 分岐自体は legitimate な merge-safe upsert (非 ArticleCreated event がカラムを空で塗り潰さないようにする目的) であり、本件の本質ではない。projector が正しい値を payload から取り出せるようにすれば preserve 分岐は意図通りに機能する。
- **event payload struct を producer / consumer 共通パッケージに集約**: 検討に値するが本 ADR の scope 外。現状は `mqhub_connect` / `knowledge_projector` / `search-indexer/consumer` で各々 struct を再宣言しており、これが drift の温床になっている。共有パッケージ化は別 ADR で扱う (action item として PM-2026-041 に記録)。

## Consequences

### Pros

- **Reproject-safety を回復**: projector が event payload から `url` を正しく読めるようになり、Reproject 後の v_new でも全 article 行に link が入る。canonical invariants §4 を満たす状態に戻る。
- **wire-form contract test が再発防止防波堤になる**: 今後 producer 側で field 名や tag を変更したときに、CI で必ず fail する。`TestArticleCreatedPayloadContract_LinkRoundTrips` は raw `map[string]any` で payload を組むので、consumer struct の tag が drift した瞬間に検出される。
- **disposable projection の恩恵を受ける**: event log と `articles.url` (producer 側 source-of-truth) は完全に無傷。projector fix を production 反映後、operator が Reproject (dry_run → compare → swap) を再実行するだけで v_active の link を全件回復できる。データ移行や手動 SQL 修正は不要。
- **FE 防御層が同種 regression を可視化する**: 仮に同型バグが将来別 field で再発しても、`Archived · No source URL projected` kicker と `console.warn` 診断によって user-visible state と dev-tools log の両方に痕跡が残る。silent failure では済まない。
- **producer↔consumer JSON-tag parity が確認された**: 全 event 種別を audit したので、本件以外の event payload に同型 drift は無いことが構造的に確認できた。

### Cons / Tradeoffs

- **production 反映後に Reproject を 1 回必須で要求する**: ADR 反映 + production deploy だけでは v_active 上の既存空 link 行は治らない。runbook に「post-tag-fix backfill」サブセクションを追記し、operator に Reproject 実行を必須化する手順を明文化する必要がある (action item として PM-2026-041 に記録)。
- **wire-form contract test は手動メンテナンス**: 全 event 種別をカバーする contract test を一括生成する仕組みは無く、新 event を追加した際に contract test を書き忘れると同型バグが再発し得る。これは action item として「event payload contract test の網羅化」に積む。
- **producer↔consumer struct の二重宣言は据え置き**: drift の温床という構造的問題は本 ADR の scope 外。共有パッケージ化は別 ADR で議論する。
- **CASE preserve 分岐の存在は引き続き両刃の剣**: 「live で preserve が効かないと壊れるが reproject では壊れる」フィールドが他にも潜在している可能性は否定できない。今回の audit は JSON-tag drift しか見ていないので、たとえば「live で別経路の補完がある」型の隠れた依存は別途検出機構が必要 (PM-2026-010 D-3 の post-swap sanity check が該当)。
