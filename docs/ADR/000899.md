# ADR-000899: Recap pipeline と citation grounding を ADR-890 follow-up として復旧する（reconciler UUID guard・news-creator prompt/sanitizer・bge-m3 timeout/retry・Pact 強化）

## Status

Accepted

## Date

2026-05-13

## Affected Services

- recap-worker (Rust)
- news-creator (Python)
- recap-subworker (Python)
- recap-worker ↔ news-creator Pact contract

## Context

ユーザ報告:
1. 「Recap が今朝も失敗している」
2. 「Citation が 0 のものもここ数日で見られる」
3. 「再三対処したのに直っていない、最近の変更以降にしか起きていない」

調査は plan-context-loader → log-seeker → web-researcher の順で進め、**独立した 2 系統の不具合**に分解できた。両方とも [[000890]] の改修以降に発生し、表面的には同じ「Recap が動かない」に見えていた。

### A. 3-day Recap batch failure (2026-05-12 17:00 UTC)

`recap-db` 直読:

| 日時 | trigger | status | last_stage |
|---|---|---|---|
| 2026-05-12 17:00 UTC | system | **failed** | dedup → `classification returned 0 results for 750 articles (service may be unavailable)` |
| 2026-05-11 17:00 UTC | system | completed | persist |
| 2026-05-10〜04-29 | system | 多くは completed | — |

- `recap-subworker` のログで全 12 chunk が `RuntimeError: Ollama API request failed: timed out` (`Initializing Ollama remote embedding client model=bge-m3 timeout=30.0`)。
- 同時刻に `news-creator` の `remote_health_checker` も同じ remote endpoint に `Connection timeout` を 60s おきに記録、5/13 13:38 UTC 時点で連続 36h 以上継続。
- Morning Letter (30 分 tick) は完走していたが `recap_job_id` がずっと `26a00929…` (5/11 17:00 の最後の成功 job) で固定 = 過去の同じ `recap_outputs` を読み続けていただけだった。

すなわち「Recap が今朝も失敗」の正体は **3-day system batch が 5/12 から復活していない**ことで、ユーザが見ているダッシュボードは 5/12 の `failed` 行を表示し続けていた。今日 17:00 UTC の batch も同じ状態のままなら再度 fail する状況だった。

### B. Citation grounded ≒ 0 (ADR-890 deploy 後ずっと)

`recap_outputs` 直読:

| 日 | bullets | grounded (`len(source_sentence_ids) > 0`) |
|---|---|---|
| 2026-05-11 (最終成功) | 50 | **0** |
| 2026-05-10 | 46 | 8 (17%) |
| 2026-05-09 | 29 | 0 |
| 2026-05-08 | 27 | 0 |
| 2026-05-07 | 34 | 4 (12%) |
| 2026-05-05〜以前 | 322+ | 0 |

[[000890]] は「次の tick から埋まる」と書かれていたが、5/5〜現在まで一度も安定して埋まっていなかった。5/11 ai_data の生データを開いて確定した壊れ方:

```jsonc
// body_json.summary.references
[ {"id":1, "url":"dev.to", "domain":"dev.to", "article_id":"dev.to"},
  {"id":2, "url":"dev.to", "domain":"dev.to", "article_id":"dev.to"} ]
```

3 ヶ所が連鎖して citation=0 を生んでいた:

- **news-creator prompt** (`recap_summary_usecase.py:1437-1445`) は `representative_sentences` に「出典: <domain>」しか出さず、full URL も UUID `article_id` も LLM に見せていなかった。LLM は手元情報を再利用しただけ（hallucination ではない）。
- **news-creator sanitizer** (`recap_summary_usecase.py:1897-1937`) は `article_id` の UUID 検証を一切せず、`"dev.to"` をそのまま下流に流していた。
- **recap-worker reconciler** ([[000890]] §Decision §2 / `persist.rs:56-64`) は `references[].article_id` が `Some` なら URL fallback に行かない実装で、`Some("dev.to")` を信用して空集合を返していた。`r.url="dev.to"` も full URL の `url_to_article` map と一致せず、仮に fallback に行っても外れる構造だった。

これは Web 研究の Deterministic Quoting 原則 ([Matt Yeung](https://mattyyeung.github.io/deterministic-quoting)) を**設計意図としては**満たしていたが、実装で「Some なら article_id verbatim」分岐が残っていたため LLM 出力を信頼する経路が成立していた。

### 「最近の変更以降」の整合性

- [[000890]] (5/5) で reconciler の skeleton が入り、`article_id` を信用する分岐が production の LLM 出力の癖と衝突して grounded がたまにしか埋まらなくなった。
- 5/12 17:00 UTC に bge-m3 remote endpoint が落ち、`timeout=30s` では吸収できず 3-day batch が fail。
- 5/12 [[000897]] で morning lifecycle が整備されたが、本質の 3-day Recap 不成立は morning_completed の裏に隠れていた。

両方ともユーザの直感「最近の変更以降」と整合する。

## Decision

`citation grounding ratio を構造的に上げる` と `3-day batch を Ollama 不調に耐える` を同一 PR で 1〜4 の 4 つの fix に分解して投入する。過去 row の backfill はしない (disposable projection の不変条件; [[000890]] §Cons と同方針)。

### 1. recap-worker: reconciler の UUID guard と URL host match (commit `b403914ed`)

`recap-worker/recap-worker/src/pipeline/persist.rs:33-79` を改修:

- `references[].article_id` は `uuid::Uuid::parse_str` で UUID 形状かを検証してから採用。UUID でない場合は無視。
- `article_id` 経路と URL 経路を**排他からから合流**に変更 (`BTreeSet<String>` で union)。これにより「Some(非UUID) で詰む」現バグが消える。
- 新規ヘルパ `url_host(&str) -> Option<String>` を導入し、scheme/`www.`/path を剥がした host 同士で比較する。`ref.url = "dev.to"` のような純ドメインでも同 host の article 群を multi-match で拾える ([[000890]] §Decision §3 の "multi-match を許容" 方針を完全実現)。
- 既存テスト 1 本のフィクスチャを実 UUID に揃え、新規 RED→GREEN テスト 4 本を追加 (`reconcile_falls_back_to_url_when_article_id_is_non_uuid_string` 等)。

### 2. news-creator: sanitizer で UUID 検証と純ドメイン URL drop (commit `2cc041a1a` 前半)

`recap_summary_usecase.py:1897-1937` を改修し、トップレベルに `UUID_SHAPE_RE` 定数を追加:

- `article_id` が `^[0-9a-fA-F]{8}-…$` に一致しない場合は `None` に倒す。reference 自体は url が有用なので残す。
- `url` が `://` を含まず `/` も含まない (= 純ドメイン) なら reference 全体を drop。downstream の reconciler が disambiguate できないので早期に切る。
- RED→GREEN テスト 4 本を `tests/usecase/test_recap_summary_usecase.py` に追加。

### 3. news-creator: prompt builder で full URL と UUID を露出 (commit `2cc041a1a` 後半)

`recap_summary_usecase.py:1417-1453` の sentence 行構築を改修:

- 旧: `- 本文  (公開日: ...)  (出典: dev.to)` ← domain だけ
- 新: `- 本文  (公開日: ...)  (source_url: https://dev.to/foo/bar)  (article_id: 1dce453b-…)`

LLM が `references[]` に echo back するために必要な情報を初めて prompt に出した形になる。`prompts/recap_summary_3days.jinja` と `recap_summary.jinja` の例も「`article_id` は UUID をそのまま使い、ドメイン名や短い置き換え文字列を入れない」と明示。例の `art1`/`art2` を実 UUID 形式に差し替えた。RED→GREEN テスト 1 本を追加 (`test_render_prompt_body_exposes_full_source_url_and_uuid_article_id`)。

### 4. recap-subworker: Ollama embed timeout と retry (commit `f8c61857a`)

`infra/config.py` と `services/embedder.py`:

- `ollama_embed_timeout` 既定値 30.0s → **120.0s**。bge-m3 (8192 token) + cold model load を見越して Ollama 公式ガイダンス (≥60s) を上回る値に設定。
- `_call_embed_api` に exponential backoff retry を追加 (initial + 2 retry, sleep 1s/2s, 計 3 試行)。retry 対象は `httpx.RequestError` のみ。`httpx.HTTPStatusError` (4xx/5xx) は policy 由来なので即 raise (rate-limit endpoint への自前 DoS を回避)。
- RED→GREEN テスト 2 本を追加。`time.sleep` を patch して deterministic に走らせる。

### 5. CDC Pact tightening (commit `52d6cfc5e`)

`recap-worker/recap-worker/src/clients/news_creator/contract.rs` の `contract_news_creator_summary_generate` interaction に response の `summary.references[]` shape を追加:

- `url` は `term!(r"^https?://[A-Za-z0-9.-]+(?:/[^\s]*)?$", "https://example.com/article-1")` で http(s):// 形式を強制。
- `article_id` は `term!(r"^[0-9a-fA-F]{8}-…$", "1dce453b-e23d-4a32-9030-7e4529fad645")` で UUID 形式を強制。

news-creator provider stub も同 shape を返すよう更新し、provider verification が GREEN になることを確認。今後 LLM の再退行を Pact gate で止められる。

### 検討した代替案

- **Strategy A (reconciler 修正のみ)**: news-creator の prompt や sanitizer は触らず、recap-worker 側だけで防御する。URL host match を入れれば最低限の grounded は確保できるが、`references[].url` が `"dev.to"` のままだとフロント側で出典クリックも壊れる。defense-in-depth の観点で不採用。
- **Strategy B (LLM に references[] を全く出させない)**: news-creator 側で prompt から sentence をすべて enumerate して `references[]` を**ホスト側で組み立て**、LLM には `[n]` marker だけ出させる。Deterministic Quoting の最終形だが、prompt template と sanitizer/output_handler の大改修が必要で blast radius が大きい。本 ADR の射程を超える。**後続 ADR の候補**として残す。
- **Strategy C (Ollama 不調の operational 復旧のみ)**: 3-day batch が fail しているのは Ollama remote の到達性問題なので embedder 側を触らず単純に endpoint を復旧させる。確かに今日の batch は復旧するが、`timeout=30s` のままだと cold load 直後に再発する。timeout を伸ばすコストは小さく、retry も bounded なので採用。
- **過去 row backfill**: 5/5〜5/11 の 322+ row を埋め直すクエリを発行する。disposable projection の不変条件に反するので [[IMPL_BASE]] と整合せず却下。次回 reproject 経路で吸収する。
- **Pact regex を緩める**: `^https?://[^\s]+$` まで広げて URL の port/query/unicode に寛容にする。短期は厳しめ regex で行き、production の実 LLM 出力で false positive が観測されたら緩める方針。

### Out of scope

- Resume パス (`clustering_response: None`) の `source_sentence_ids` 埋め込み — [[000890]] §4 で TODO 化済み。
- Ollama remote `100.x.x.x:11436` (bge-m3) と `100.x.x.x:11434` (gemma) の到達性復旧 — application code では治らない operational task。本 ADR と並行してユーザ側で対応。
- Circuit breaker / Ollama multi-endpoint failover — retry/timeout までで止め、複数 endpoint の優先順位ロジックは scope out。
- フロント側 `bullets_ja[].sources[].url` の render XSS 対策 — security-auditor F-003 の Info 指摘事項、別 PR で確認。

## Consequences

### Pros

- **Citation grounded ratio が構造的に上がる**: news-creator が full URL と UUID を prompt に出し、sanitizer で UUID 検証、reconciler で URL host match + 合流。「LLM が UUID を verbatim echo」「LLM が URL を verbatim echo」「LLM がドメインしか echo しない」のいずれの挙動でも最終的に grounded に倒せる三層防御。直近 7 日の新規 run で grounded ≥ 90% を目標値とする。
- **3-day batch が Ollama remote の一過性 timeout に耐えるようになる**: 30s → 120s で cold load を吸収、3 試行 exponential backoff で network ノイズを吸収。policy 応答 (4xx/5xx) は即 raise なので自前 DoS は防げる。
- **Pact contract が新しい不変条件を pin する**: `references[].url` が http(s):// で始まる / `article_id` が UUID or null。LLM の挙動退行を CDC verification で気付ける。Deterministic Quoting 原則を「設計意図」から「検証可能な契約」に格上げ。
- **三層防御で各レイヤが独立**: news-creator が壊れても recap-worker が URL host match で grounded を返す。recap-worker が壊れても news-creator の sanitizer が下流に bad data を流さない。Pact が壊れれば CI で気付ける。
- **不変条件の整合**: append-first / reproject-safe / disposable projection いずれも違反なし。`bullets_ja` は引き続き disposable projection で、新規 row のみ richer に書き直され、既存 row は次回 reproject まで `[]` のまま ([[000890]] §Cons と同方針)。
- **新規 compose service / init container / 名前付き volume / Atlas migration いずれも追加していない**。rolling 互換。

### Cons / Tradeoffs

- Prompt に full URL と UUID を出すと OWASP LLM01 prompt injection の表面が**わずかに**拡大する。`sentence.text` 経由の prompt injection は元から許容しているリスクだが、URL に悪意ある query string が含まれる経路が新たに加わる。短期は許容 (JSON schema で gating)、中期は `source_url` の長さ・scheme 検証を sanitizer 層に入れる (security-auditor F-001)。
- `is_uuid_shape` / `UUID_SHAPE_RE` は NIL UUID (`00000000-…`) も accept する。LLM が偶然 NIL UUID を吐けば article_id 経路で外れて URL fallback に倒れるだけなので実害は無いが、defense-in-depth で version nibble まで縛りたい場合は別 ADR (security-auditor F-002)。
- `timeout=120s` は cold load を救う反面、本当に Ollama がハングした時に call site が 120s 待つ。retry 3 試行込みで最悪 ~360s (120 + sleep 3 + 120 + sleep 3 + 120 + sleep + raise = 約 6 分弱) の per-chunk 待ち。production の 750 articles を 3 chunk で投げる現状でも worst-case ~18 分。許容範囲だが、circuit-breaker 不在の点は将来課題。
- 過去 row の grounded は埋め直さない。dashboard の history は 5/14 以降で grounded ratio が段階的に上がる形で見える。修復タイミングは可視化されるが、過去のレポート品質を後追いで上げる経路は無い。retention cleanup 系の別 ADR と整合させる。
- Pact regex が厳しめなので、production の LLM が偶然 port 番号や query string、unicode を含む URL を吐くと provider verification が落ちる可能性。短期は厳しめで行き、観測してから緩める。
- recap_summary_usecase.py の touched 関数 (`_render_prompt_body`, `_sanitize_summary_payload`) は既存テスト 53 本すべて GREEN だが、prompt format の文字数が増えるため `ABNORMAL_PROMPT_THRESHOLD` (100,000 chars) に近付く genre が出る可能性。observability で兆候を見る。

## Deploy-model 整合性セルフチェック

本 ADR は recap-worker / news-creator / recap-subworker の application code、prompt template、Pact contract の変更のみで構成される。新規 compose service / 新規 init container / 新規名前付き volume / Atlas migration いずれも追加していない。

- [x] 新設 compose service が `depends_on` で他 service を gate するか？ — N/A
- [x] 新設 compose service に `restart: "no"` の one-shot init container を含むか？ — N/A
- [x] 新設名前付き volume を populate する責務はどこにあるか？ — N/A
- [x] いずれも該当しない: **rolling 互換**。alt-deploy の per-service rolling deploy で recap-worker / news-creator / recap-subworker を順に rebuild すれば良い。Pact provider/consumer の同時更新は consumer (recap-worker) が tighter な制約を持つので、provider (news-creator) を先に rolling すれば consumer 側がそれを満たす shape を受け取れる順序になる。
