# ADR-000455: Knowledge Home dismiss を read model に永続化してリロード時の再表示を防止する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go)

## Context

Knowledge Home で記事を dismiss すると、画面上はいったん消えるが、リロード後に同じ記事が再表示される不具合が発生していた。

調査の結果、フロントエンドは dismiss 時に楽観的に item をローカル state から外していた一方、バックエンドは `HomeItemDismissed` イベントを `knowledge_events` に記録するだけで、`knowledge_home_items` read model に永続的な非表示状態を持っていなかった。したがって、ページ再読込時にサーバーから再取得された item がそのまま表示されていた。

さらに、`ArticleCreated` / `SummaryVersionCreated` / `TagSetVersionCreated` のような再投影イベントは read model を再構築する性質を持つため、dismiss を単なる一時的な削除として扱うと後続イベントで再出現する可能性があった。

## Decision

Knowledge Home の dismiss を user-scoped な read-model state として永続化する。

具体的には以下を採用する。

1. `knowledge_home_items` に `dismissed_at TIMESTAMPTZ` を追加する
2. `HomeItemDismissed` を projector で解釈し、対象 row の `dismissed_at` を更新する
3. `GetKnowledgeHomeItems` と `CountNeedToKnowItems` は `dismissed_at IS NULL` の item のみを対象にする
4. dismiss は row を削除せず、可視性フラグとして保持する

削除ではなく `dismissed_at` を採用した理由は、再投影や後続イベントに対して dismiss 状態を維持しやすいためである。read model の row 自体を残すことで、将来 undo を追加する場合も履歴を壊さずに拡張できる。

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|--------|------|-----------|
| dismiss 時に `knowledge_home_items` の row を `DELETE` する | 実装は単純 | 再投影や後続 upsert で再出現しやすく、永続非表示の意味論を満たしにくい |
| フロントエンドだけで dismissed item を local storage に保持する | バックエンド変更が不要 | 端末依存になり、別ブラウザ・別セッション・再投影との整合性を保証できない |
| `knowledge_events` を読んで取得時に都度除外計算する | append-only に寄せられる | Home 読み取りのたびにイベント集約が必要になり、既存 read model パターンと整合しない |

## Consequences

### Pros

- dismiss がリロード後も維持され、Knowledge Home の操作が user expectation と一致する
- 後続の `ArticleCreated` / `SummaryVersionCreated` / `TagSetVersionCreated` が来ても、dismiss 済み item を read path で一貫して隠せる
- `CountNeedToKnowItems` も同じ可視性ルールに揃うため、UI の件数と item 一覧の意味が一致する
- row を保持するため、将来の undo や監査用途への拡張余地を残せる

### Cons / Tradeoffs

- `knowledge_home_items` に可視性 state が増えるため、projection が単純な materialized copy ではなくなる
- dismiss 後も row 自体は残るため、完全削除と比べて storage はわずかに増える
- `dismissed_at IS NULL` を前提とする query / index 設計を維持する必要がある
