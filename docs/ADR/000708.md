# ADR-000708: recap-worker boot 時に orphan Recap Job を確実に sweep する

## Status

Accepted

## Date

2026-04-13

## Affected Services

- recap-worker (Rust) — `BatchDaemon::run` 起動シーケンス、`RecapDao` の
  `find_resumable_job` / `mark_abandoned_jobs` / `delete_old_jobs`、Config
  に `RECAP_RESUMABLE_MAX_AGE_HOURS`

## Context

ユーザーから「Recap Job が大量に走っている」と報告。実際に運用中の
`recap-worker` のログを精査すると、**実 Recap pipeline は 24h で 1 件しか
走っていない**ことが判明 (本来仕様: `BATCH_HOUR=2, BATCH_MINUTE=0` JST、
1日1回)。それでも DB の `recap_jobs` テーブルには:

```
status     count
completed   26
pending      6   ← 全部 today 01:19–02:15 UTC kicked
failed       2
running      2   ← うち 1 件は本物の resume 対象
```

という分布で、ダッシュボード `/v1/dashboard/job-stats` が
`success_rate_24h: 0.16`, `running_jobs: 1`, `total_jobs_24h: 12` を返し、
UI 上「常に何か走っている」「失敗まみれ」のような印象を作っていた。

根本原因は次の 3 つの相互作用:

1. **`find_resumable_job` の判定が緩すぎた**
   過去のクラッシュで `pending` / `running` のまま残った行を boot のたびに
   掴んでは「resume → 失敗 → resume → …」を繰り返していた。SQL は
   `status IN ('pending', 'running', 'failed')` で **`failed` まで再開対象**
   にしており、age window もなし。
2. **`cleanup_old_jobs` のバインド型バグ**
   `make_interval(days => $1)` の `days` は `INT` を要求するのに、Rust 側で
   `bind(retention_days as f64)` していたため Postgres が
   `function make_interval(days => double precision) does not exist` で常時失敗。
   ただし呼び出しが「batch 完走後」(daemon.rs:109) のみだったため、daemon が
   走らない日は誰も気付かなかった。
3. **Boot 時に何の sweep もしていなかった**
   コンテナ再起動時、前プロセスの in-flight Job 行は定義上全て orphan に
   なるが、DAO はそれを `failed` にする経路を持っていなかった。

加えて Recap pipeline は **数時間掛かる可能性がある** (LLM batches +
classification + embedding) ため、age threshold を雑に短くすると正当な
長尺 Job まで誤って失敗扱いにしてしまう。設計はこの長尺ケースと、
"前プロセスは死んでいる" という boot 時の事実、両方を満たす必要がある。

## Decision

1. **`find_resumable_job(max_age_hours)` を tighten**
   - `'failed'` を IN 句から外す (一度失敗したものは自動再開しない)。
   - `kicked_at > NOW() - INTERVAL 'N hours'` の age window を導入。
     デフォルト 12h、`RECAP_RESUMABLE_MAX_AGE_HOURS` で可変。
   - 12h は「数時間掛かる Recap」を救える幅で、かつ前日以前の腐ったゴミは
     拾わない balance。
2. **新 DAO `mark_abandoned_jobs(keep_job_id: Option<Uuid>)`**
   - `pending` / `running` 全行を `failed` に確定。
   - `keep_job_id` を渡すとその 1 件だけ sweep 対象から外す
     (resume 候補保護)。
   - age filter は **付けない**。boot 時はプロセスが死んでいるので
     in-flight 行は定義上全て orphan、年齢を問わず閉じてよい。
3. **`BatchDaemon::run` の起動シーケンスを書き換え**
   ```
   1. find_resumable_job(max_age_hours)            ← fresh な 1 件を選ぶ
   2. mark_abandoned_jobs(Some(keep_id))           ← それ以外を seal
   3. cleanup_old_jobs()                           ← 保持期間超過を削除
   4. resume したジョブを実行 (あれば)
   5. メインループ (next_run_jst まで sleep)
   ```
4. **`make_interval` の bind 型バグを修正**
   `delete_old_jobs` / `find_resumable_job` の SQL parameter を
   `bind(value as f64)` から `bind(value as i32)` に変更。`make_interval` の
   `days` / `hours` は `INT` を要求する。`i32::try_from(...).unwrap_or(i32::MAX)`
   で安全側にキャストしておく (現実的な入力では飽和は起きない)。

## 実装の概要

### `recap-worker/src/store/dao/job.rs`

- `find_resumable_job(pool, max_age_hours: i64)`:
  ```sql
  SELECT job_id, status, last_stage, window_days
  FROM recap_jobs
  WHERE status IN ('pending', 'running')
    AND kicked_at > NOW() - make_interval(hours => $1)
  ORDER BY kicked_at DESC LIMIT 1
  ```
- `mark_abandoned_jobs(pool, keep_job_id: Option<Uuid>)`:
  ```sql
  UPDATE recap_jobs
  SET status = 'failed',
      last_stage = COALESCE(last_stage, 'abandoned_at_startup'),
      updated_at = NOW()
  WHERE status IN ('pending', 'running')
    AND job_id <> $1   -- omitted if keep_job_id is None
  ```
  `recap_jobs` には `error_message` 列が無い (migration
  `20251212000000_add_job_state_tracking.sql` 参照) ため、reason は
  `last_stage = 'abandoned_at_startup'` と tracing log で残す。
- `delete_old_jobs(pool, retention_days: i64)`: bind を `i32` に修正。

### `recap-worker/src/store/dao/traits/job.rs` / `impls/unified.rs` / `compat.rs` / `mock.rs`

シグネチャの変更を伝播。mock は `Ok(0)`、unified は新 SQL を呼び出す。

### `recap-worker/src/scheduler/jobs.rs`

`Scheduler::find_resumable_job` は `Config::resumable_max_age_hours()` を読む。
`Scheduler::mark_abandoned_jobs(keep)` を新設、`Ok(>0)` のときに
`tracing::info!(?keep_job_id, marked_failed = n, "sealed orphaned recap jobs as failed")`。

### `recap-worker/src/scheduler/daemon.rs`

`BatchDaemon::run` の冒頭を:
```rust
let resumable_target = state.scheduler.find_resumable_job().await.ok().flatten();
let keep_job_id = resumable_target.as_ref().map(|(id, _, _, _)| *id);
state.scheduler.mark_abandoned_jobs(keep_job_id).await; // 0 ならログなし
state.scheduler.cleanup_old_jobs().await;               // 0 ならログなし
if let Some((job_id, status, last_stage, resumed_window_days)) = resumable_target {
    // 再開実行
}
```
の順に書き換え。これで boot のたびに `(資格ある 1 件, それ以外を seal,
古いゴミを delete)` がアトミックに走る。

### `recap-worker/src/config.rs`

`Config` 構造体に `resumable_max_age_hours: i64` を追加。`from_env` で
`parse_i64("RECAP_RESUMABLE_MAX_AGE_HOURS", 12)` を読み、
`Config::resumable_max_age_hours()` accessor を expose。

## Verification (2026-04-13 実測)

### Before (本 ADR 適用前)
```
status     count
completed   26
pending      6   ← 今朝 01:19–02:15 UTC の zombie
failed       2
running      2   ← 1 件は legitimate、1 件は zombie
```
Boot 時の log:
```
"boot-time hygiene: mark_abandoned_jobs failed"
   error="failed to mark abandoned recap jobs"      ← bind 型バグ
"boot-time hygiene: cleanup_old_jobs failed"
   error="failed to delete old jobs"                ← 同上
```
Dashboard: `success_rate_24h: 0.16`, `running_jobs: 1`,
`total_jobs_24h: 12`。

### After (本 ADR 適用後)
Boot 時の log:
```
"sealed orphaned recap jobs as failed"
   keep_job_id=Some(f67ffd79-…), marked_failed=7
"boot-time hygiene: orphaned jobs sealed"
   marked_failed=7, keep_job_id=Some(f67ffd79-…)
"found resumable job, resuming..."
   job_id=f67ffd79-…, status=Running,
   last_stage=Some("evidence"), resumed_window_days=1
```
DB:
```
status     count
completed   26
failed       9   (= 元の 2 + 新たに seal した 7)
running      1   (resume 対象として保護した 1 件のみ)
```
ダッシュボードの "running_jobs" / 偽 active ノイズが消えた。

## Consequences

### Pros

- **boot 後の DB 状態が一意に決まる**: pending=0, running ≤ 1
  (resume 候補のみ)。"何か走っているのに何もしていない" 印象が解消。
- **`find_resumable_job` が age 内のものしか返さない**ので、5 日前の
  腐った行を boot のたびに掴んで失敗する loop が止まる。
- **数時間掛かる長尺 Recap Job も誤って seal しない** (`max_age_hours=12h`
  は標準的な実行時間より十分長い)。
- **`make_interval` バインド型バグの恒久修正**: `cleanup_old_jobs` も
  これで本当に動くようになり、retention 超過行が自動消去される。
- **既存スキーマに変更なし**: SQL 変更のみで migration 不要。

### Cons / Tradeoffs

- `mark_abandoned_jobs` は age filter なしのため、boot 直前まで legit に
  動いていた Job (例えば手動 kick した recap がプロセス死亡で残ったもの)
  も `keep_job_id` に選ばれない限り `failed` 扱いになる。但し
  `find_resumable_job` が直前に最新 1 件を `keep_job_id` に保護するため
  実害はほぼない (複数 in-flight が常態化している環境は別途要検討)。
- `RECAP_RESUMABLE_MAX_AGE_HOURS` の値は環境ごとに調整余地。3-day Recap
  が常時 6h を超えるような環境 (大量 RSS 購読等) では 24h まで延ばす
  ことを検討。
- 周期実行は未実装。boot 時の 1 回だけなので、コンテナを長時間止めない
  運用では `cleanup_old_jobs` は呼ばれない。これは Phase 2 として周期
  job (例: 毎時) を追加する余地。

## Phase 2 (deferred)

- `BatchDaemon::run` の sleep ループ内、毎 N 時間 (例: 6h) で
  `cleanup_old_jobs` を再実行する周期 sweep。
- ダッシュボードの UI 側で `status` 別の breakdown を表示
  (Pending/Started/Completed/Failed の積み上げ棒グラフ)。
- `mark_abandoned_jobs` を `last_stage = 'abandoned_at_startup'` 以外の
  reason code (e.g. periodic-sweep / manual-kill) も持てるよう拡張。
