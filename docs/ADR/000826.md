# ADR-000826: compose パターンと rolling deploy model の整合性を ADR レビューの必須セルフチェックに格上げする

## Status

Accepted

## Date

2026-04-22

## Affected Services

- **docs/ADR/template.md** — セルフチェック項目を追加し、今後の ADR 執筆者が「採用 compose パターンが Alt の deploy model と整合するか」を自問する仕組みを作る
- **docs/ADR/** (既存 ADR の audit) — 過去に採用された compose パターンで rolling 非互換なものがないか洗い直す follow-up を PM-2026-037 の AI で tracks
- **docs/runbooks/** — compose pattern カタログに「rolling 非互換な pattern の例と代替」節を追加する follow-up を PM-2026-037 の AI で tracks

## Context

[[PM-2026-036]] で記録した 9 日間の 3days Recap outage に対し、[[000825]] 初期版 (2026-04-22 昼) は「named volume + busybox one-shot init container」で `/app/data` を populate する設計を採った。compose 単独では健全かつ Docker 公式推奨の init container パターンで、unit test / compose config レンダリングも問題なく通った。

ところが同日夕方のデプロイ完走後、3days Recap の手動キックが再び `classification returned 0 results for 954 articles (service may be unavailable)` で失敗した。根本原因は compose パターン単独の不具合ではなく、**Alt の deploy model (per-service rolling、`--no-deps` 相当) との非互換**だった。詳細は [[PM-2026-037]] を参照。要点:

- Alt の deploy は **1 サービスの失敗が全体を止めない** 設計のため、サービス単位で targeted に compose を操作する。
- この経路は内部的に `docker compose up <service> --no-deps` 相当を発行する。
- `--no-deps` は `depends_on` で参照されたサービスを **起動も create もしない**。`service_completed_successfully` gate を持つ init container はこの経路では一度も動かない。
- 結果として、compose 上は「healthy に起動」するが名前付き volume は populate されず、下流サービスが空のデータで起動する silent 失敗を生む。
- この挙動は docker-compose / community.docker の普遍的なものであり ([docker/compose#9591](https://github.com/docker/compose/issues/9591) に類例)、Alt 固有の不具合ではない。

[[000825]] addendum で compose は direct directory-scoped bind mount に差し替え、fail-closed 責務を compose engine v2.24+ 自身の「missing-source で container create refuse」挙動に寄せた ([docker/compose#12735](https://github.com/docker/compose/issues/12735) で 2025-04 の v2.35 時点も継続確認済)。この技術的解は [[000825]] に固定された。

本 ADR は **技術的判断ではなく、組織的な仕組みの追加** を扱う。同型の見落とし ("compose 単独では健全だが rolling deploy と非互換な pattern を ADR レビュー時に検出できなかった") を二度起こさないため、ADR レビューの必須セルフチェック項目を追加する。

## Decision

**`docs/ADR/template.md` に以下のセルフチェック節を追加する**:

```markdown
## Deploy-model 整合性セルフチェック

この ADR が採用する compose / CI / deploy パターンが、Alt の per-service
rolling deploy model と整合するか確認する:

- [ ] 新設 compose service が `depends_on` で他 service を gate するか?
      - YES: その gate は rolling deploy (サービス単独 `docker compose up <svc>` 相当)
        下でも機能するか? `depends_on.condition: service_completed_successfully`
        の init container は **機能しない**。[[PM-2026-037]] 参照。
      - 機能しないなら: (a) init 相当の責務を compose engine の fail-fast
        挙動 (directory-scoped bind missing source refuse 等) に寄せる、
        (b) image の ENTRYPOINT に吸収する、(c) deploy tooling 側で明示的
        に起動対象に含める、のいずれかを採用する。
- [ ] 新設 compose service に `restart: "no"` の one-shot init container を
      含むか?
      - YES: 同上。rolling deploy では一度も起動されない前提で再設計する。
- [ ] 新設名前付き volume を populate する責務はどこにあるか?
      - init container 由来: 上記 rolling 非互換問題に当たる。
      - 外部 (host bootstrap / baked image / CI artefact fetch): その経路を
        runbook で明文化し、prod host の pre-condition として確認できるようにする。
- [ ] 上記いずれかに該当しなければ、「rolling 互換」と明記して pass。
```

### 検討した代替案と却下理由

- **セルフチェック項目ではなく自動化された linter**: `docker compose config --services` 出力と deploy tooling の service filter を突き合わせて、宣言されているが起動対象でないサービスを警告する。技術的には実装可能だが、(a) deploy tooling 側 (private) の仕様変更で linter の前提が崩れる、(b) rolling 挙動と `depends_on` の組み合わせは compose engine 側の普遍問題なので Alt 固有 linter で解決するのは overkill、(c) [[PM-2026-037]] AI #2 の "deploy post-check" でより包括的に扱うべき。本 ADR ではセルフチェック項目という軽量な人間側プロセスに留める。
- **ADR 自体を「deploy model への適用手順」まで書かせる**: テンプレートに「本 ADR の変更を rolling deploy で本番反映する手順」を書くよう義務付ける案。情報衛生違反 (private deploy-tooling 内部の記述を public ADR に書く圧力を生む) を招くため却下。本 ADR のセルフチェックは「非互換性を早期検知する」ことだけに責務を絞り、具体的な deploy 手順は private 側の文書に委ねる。
- **init container pattern を禁止する**: 単一ホストで `docker compose up -d` する dev/staging では正当かつ有用。rolling deploy でのみ非互換なので全面禁止は過度。セルフチェック項目で互換性を問うに留め、禁止ではなく「意識的な選択」を促す。
- **ADR 執筆不要**: [[000825]] addendum と [[PM-2026-037]] に既に書かれているため、本件を新 ADR にせず既存 docs に追加する案。却下理由: template 側の変更は **横断的な組織ルール** の更新であり、特定インシデント記録に埋めると発見性が落ちる。template に反映する決定は独立した ADR として残す価値がある。

### 「整合」の定義

本 ADR では以下を満たすとき「rolling 互換」と呼ぶ:

1. `docker compose up <service>` を `--no-deps` 付きで単独実行しても、当該サービスが意図どおり起動する (必要な外部状態は compose 起動前に準備済み)。
2. 外部状態の準備は compose 外 (host bootstrap / baked image / 外部ストレージからの fetch 等) で行われ、compose ファイルはその前提を assume するだけ。
3. compose engine が足りない前提を検知する場合は **fail-fast** で container create を拒む (例: directory-scoped bind missing source)。silent failure に落ちない。

この 3 条件を全て満たす compose パターンのみ rolling 互換。

## Consequences

### Pros

- **同型の見落としの early detection**. 今後の ADR 執筆で init container + `service_completed_successfully` のような rolling 非互換パターンを採用しようとした段階でセルフチェックが引っかかる。prod 初回適用時まで気づかない経路が塞がれる。
- **組織学習の固定化**. [[PM-2026-036]] → [[PM-2026-037]] の 2 本立ての教訓が、ADR template という毎回触る場所に landing する。別文書に埋もれた "気をつけよう" メッセージより効果的。
- **技術選択の自由度は維持**. init container pattern や名前付き volume + populator は禁止されない。rolling 互換かどうかを意識的に問い、非互換なら代替 (direct bind / image-baked / deploy tooling 拡張) に倒す判断を設計者に委ねる。
- **情報衛生の原則を堅持**. セルフチェックは compose engine 抽象レベルの自問にとどまる。private deploy-tooling の内部 (playbook 名 / line 番号 / CI workflow 詳細) を ADR に持ち込む圧力を作らない。
- **テンプレート変更 1 箇所で横展開**. 過去 ADR の re-audit は [[PM-2026-037]] AI #1 で別 track にするが、新規 ADR については本 ADR が live した瞬間から適用される。

### Cons / Tradeoffs

- **セルフチェックは人間のプロセスに依存**. 真面目に answer しないチェックボックスは効かない。PR レビューで「このセルフチェックに答えているか」を確認する文化が必要。PR template 側にも最低限のリンクを足す follow-up を検討 (PM-2026-037 AI 側で tracks)。
- **テンプレート肥大化のリスク**. 本 ADR のチェック以外にもレビュー時に問うべき項目は他にも存在する (security, OSS 境界, rate limiting, migration 先行, etc)。全部盛りにすると機能しない。本項目は **PM-036/037 で実際に痛みを経験した** 固有の穴を塞ぐものとして選別された。今後同様に「痛みベース」で選別的に追加する運用原則にする。
- **"rolling 互換" の判定責任が ADR 執筆者に残る**. 自動化されていないので、経験の浅い執筆者が判定を誤る可能性は残る。[[PM-2026-037]] 自体をセルフチェックの根拠文書として常に参照させることで、判定基準を均一化する。
- **既存 ADR の audit は本 ADR の責任外**. 既存 compose pattern に rolling 非互換なものが埋まっている可能性はある。[[PM-2026-037]] AI #1 で別途 audit する。audit 結果が出るまでは「過去の ADR は本 ADR のチェックを通過していない」前提で扱う。
- **`depends_on: service_started`** **は問題ない** のにセルフチェック文面が不明瞭だと混乱しうる。本 ADR は `service_completed_successfully` + `restart: "no"` の init container pattern に限って非互換であり、`service_started` / `service_healthy` で gate する一般的な dependency は rolling 下でも依存先が既に running なら問題ない。セルフチェック文面にこの区別を明記済。
