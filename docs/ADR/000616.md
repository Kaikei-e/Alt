# ADR-000616: Pact CDC CI パイプラインをアーティファクト受け渡し方式に再構成しローカル事前検証を追加する

## Status

Accepted

## Date

2026-04-03

## Affected Services

- alt-backend (Go) — Consumer + Provider
- pre-processor (Go) — Consumer
- rag-orchestrator (Go) — Consumer
- search-indexer (Go) — Consumer
- mq-hub (Go) — Consumer (Message)
- recap-worker (Rust) — Consumer
- recap-evaluator (Python) — Consumer
- news-creator (Python) — Provider
- recap-subworker (Python) — Provider
- tag-generator (Python) — Provider

## Context

[[000591]] で全サービスペアへの Pact CDC テスト全面展開と Pact Broker 統合が確立されたが、CI パイプライン（`.github/workflows/proto-contract.yaml`）の実装には以下の構造的問題が残っていた。

### 問題 1: Go Consumer テストの冗長な再実行

`pact-cdc-go` ジョブ（matrix: 5 サービス）が pact ファイルを生成するが、アーティファクトとして upload しないため、後続の `pact-cdc-python` ジョブで全 Go Consumer テストを再実行して pact ファイルを再生成していた。CI 時間の浪費に加え、2 回のテスト実行間で結果が乖離するリスクがあった。

### 問題 2: Rust/Python Consumer pact の未 publish

`pact-cdc-rust`（recap-worker、4 Provider 10 interactions）と `pact-cdc-python-consumer`（recap-evaluator、3 interactions）で生成された pact ファイルが Broker に publish されず、Provider 検証の対象外だった。全 16 pact ファイル中 5 ファイルが Broker に到達していなかった。

### 問題 3: Provider 検証の Silent Fail

recap-subworker と tag-generator の Provider 検証が `|| true` で実行されており、失敗しても CI が green になっていた。加えて、両サービスに `PACT_BROKER_BASE_URL` 環境変数が設定されておらず、ローカルファイルモードで動作していた。

### 問題 4: alt-backend Provider 検証の CI 未実行

`alt-backend/app/driver/contract/provider_test.go`（recap-worker、search-indexer からの契約検証）が存在するにもかかわらず、CI ワークフローに含まれていなかった。

### 問題 5: ローカル事前検証手段の欠如

開発者が `docker compose up --build` 前に契約デグレを検知する手段がなく、CI でのみ契約違反が発覚していた。

## Decision

**CI パイプラインを GitHub Actions アーティファクト受け渡し方式に再構成し、全 pact ファイルの Broker publish と全 Provider 検証を単一ジョブで実行する。加えて、ローカル事前検証スクリプトを導入する。**

### 1. Consumer テストジョブにアーティファクト upload を追加

`pact-cdc-go` matrix に `pact-dir` 変数を追加し、各ジョブが `actions/upload-artifact@v4` で pact ファイルを upload する。`pact-cdc-rust`、`pact-cdc-python-consumer` にも同様に upload ステップを追加。

| ジョブ | アーティファクト名 | Pact ファイル数 |
|--------|-------------------|----------------|
| pact-cdc-go (alt-backend) | `pacts-alt-backend` | 1 |
| pact-cdc-go (pre-processor) | `pacts-pre-processor` | 1 |
| pact-cdc-go (rag-orchestrator) | `pacts-rag-orchestrator` | 1 |
| pact-cdc-go (search-indexer) | `pacts-search-indexer` | 3 |
| pact-cdc-go (mq-hub) | `pacts-mq-hub` | 3 |
| pact-cdc-rust | `pacts-recap-worker` | 4 |
| pact-cdc-python-consumer | `pacts-recap-evaluator` | 1 |

合計: 7 アーティファクト、16 pact ファイル。

### 2. `pact-cdc-python` を `pact-publish-and-verify` に置換

旧 `pact-cdc-python` ジョブを廃止し、以下の構造に再構成:

```
pact-cdc-go (matrix: 5) ─────────┐
pact-cdc-rust ────────────────────┼──→ pact-publish-and-verify
pact-cdc-python-consumer ─────────┘
```

`pact-publish-and-verify` ジョブ:
1. `actions/download-artifact@v4` で全アーティファクトを `merge-multiple: true` で単一ディレクトリに集約
2. curl ループで全 16 pact ファイルを Broker に publish + ブランチタグ付与
3. alt-backend Go Provider 検証（新規追加、Go + FFI ライブラリ setup 含む）
4. news-creator、recap-subworker、tag-generator Python Provider 検証

全 Provider 検証に `PACT_BROKER_BASE_URL` 環境変数を統一設定し、Broker 経由での契約取得に統一。

### 3. `|| true` の削除

recap-subworker と tag-generator の Provider 検証から `|| true` を削除。CI は全 Provider 検証が成功した場合のみ green になる。

### 4. ローカル事前検証スクリプト

`scripts/pact-check.sh` を新規作成:

- **デフォルト（ファイルベースモード）**: Broker 不要。Consumer テストで pact ファイルを生成し、Provider 検証をローカルファイルモードで実行。`docker compose up --build` 前の高速チェックとして使用。
- **`--broker` モード**: `compose/pact.yaml` の Pact Broker を Docker Compose で起動し、pact publish + Broker 経由 Provider 検証を実行。CI 相当のフル検証。

ツールチェインの存在確認を含み、Go/Rust/Python のいずれかが未インストールの場合は該当ステップをスキップしてレポートする。

### 5. tdd-workflow スキルの更新

`.claude/skills/tdd-workflow/SKILL.md` の CDC テスト情報を全 11 ペアに拡張し、Rust Pact 対応、`pact-check.sh` 参照、Go Provider 検証コマンドを追加。

## Consequences

### Pros

- 全 16 pact ファイルが CI で Broker に publish されるようになり、契約の可視化と追跡が完全になった
- Go Consumer テストの冗長な再実行が排除され、CI 実行時間が短縮された
- 全 Provider 検証が Broker 経由に統一され、ローカルファイルモードとの挙動差異がなくなった
- `|| true` の削除により、Provider 検証の失敗が CI で正しく検知されるようになった
- alt-backend Go Provider 検証が CI に組み込まれ、recap-worker/search-indexer からの契約違反も自動検知される
- `scripts/pact-check.sh` により、開発者がローカルで契約デグレを事前に検知できるようになった
- tdd-workflow スキルが最新の CDC テスト構成を反映し、開発ガイダンスの正確性が向上した

### Cons / Tradeoffs

- `pact-publish-and-verify` ジョブが全 Consumer ジョブの完了を待つため、最も遅い Consumer ジョブがボトルネックになる（現状 Rust コンパイルが最遅で約 60s）
- アーティファクト upload/download のオーバーヘッドが各ジョブに +5-10s 追加される
- Pact Broker は依然としてエフェメラル（CI ジョブ内の GitHub Actions services）であり、`can-i-deploy` ゲートの有効化には永続的な Broker インスタンスが必要（[[000591]] で記載済み）
- `pact-publish-and-verify` ジョブに Go + Python の両ランタイムが必要で、セットアップ時間が増加する
