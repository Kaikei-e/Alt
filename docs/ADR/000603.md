# ADR-000603: SubIntent 駆動の検索ポリシー分岐と関連記事ツール有効化で Augur の記事フォローアップ品質を改善する

## Status

Accepted

## Date

2026-03-28

## Affected Services

- rag-orchestrator (Go)

## Context

Augur（RAG Q&A）で記事スコープのフォローアップ品質に以下の障害が再現された:

1. **「技術的な詳細をもっと教えて」** に対して汎用要約を返す — `SubIntentNone` にフォールバックし、概要/詳細/まとめ構造のプロンプトが発火するため
2. **「関連する記事はある？」** に対して弱い関連記事を返す — `ToolDispatcher` が `IntentArticleScoped` に対して常に `nil` を返すため既存の `RelatedArticlesTool` が発火せず、無条件の general re-retrieval が無関係チャンクを混入させるため

根本原因は、[[000600]] で導入した SubIntent 基盤（critique / opinion / implication の 3 種）がカバーする意図範囲が狭く、すべての記事フォローアップが一律に agentic re-retrieval（general チャンクの無条件マージ）を経由していたことにある。

また、prompt_builder のデフォルト出力形式が「概要 → 詳細 → まとめ」固定であるため、ユーザーがより狭い問いかけ（技術詳細、根拠提示、関連記事検索）をしても Gemma 3 が要約モードに偏るという構造的問題があった。

## Decision

**SubIntentType を 4 種追加し、SubIntent ごとに検索ポリシー・ツール選択・プロンプトテンプレートを分岐させる。**

### 1. SubIntent の拡張

| 新 SubIntent | キーワード例 (JP/EN) | 用途 |
|---|---|---|
| `detail` | 技術的, 詳細, 具体例, 仕組み / technical, detail, mechanism | 記事の技術的詳細を深掘り |
| `related_articles` | 関連する記事, 似た記事 / related articles, similar articles | 関連記事の検索 |
| `evidence` | 根拠, エビデンス, 証拠, 出典 / evidence, proof, citation | 主張を裏付ける引用提示 |
| `summary_refresh` | 結論だけ, もう一度, 要約して / just the conclusion, recap | 簡潔な要約リフレッシュ |

優先順位: `related_articles` > `evidence` > `detail` > `critique` > `opinion` > `implication` > `summary_refresh`

### 2. 検索ポリシーの SubIntent 別分岐

`answer_with_rag_usecase.go` の agentic re-retrieval ブロックを switch 文に置換:

| SubIntent | Policy |
|---|---|
| `detail`, `evidence`, `summary_refresh` | 記事チャンクのみ — general re-retrieval **しない** |
| `related_articles` | ツール委譲 — `RelatedArticlesTool` で Meilisearch 検索 |
| `critique`, `opinion`, `implication` | 記事優先 + quality gate 下回った場合のみ general merge |
| `none` (デフォルト) | 現行動作維持（無条件 general re-retrieval、後方互換） |

### 3. ToolDispatcher の配線

`selectTools()` を `IntentArticleScoped + SubIntentRelatedArticles` のときに `["related_articles"]` を返すよう変更。DI container で `RelatedArticlesTool` を `SearchClient` 経由で配線し、`WithToolDispatcher` オプションで Answer usecase に注入。

### 4. プロンプトテンプレートの分岐

| SubIntent | 出力形式 | 指示内容 |
|---|---|---|
| `detail` | analytical (概要/詳細/まとめ なし) | 「質問に直接回答。メカニズム・手順・技術的根拠に焦点」 |
| `related_articles` | リスト特化 (`writeRelatedArticlesOutputFormat`) | 「ランク付きリストで返せ。散文にしない」 |
| `evidence` | analytical | 「引用付きで具体的パッセージを返せ」 |
| `summary_refresh` | analytical (800 文字制限なし) | 「簡潔に 5-7 ポイントに絞れ」 |

Multi-turn のフォローアップガイダンスも各 SubIntent に対応するよう拡張。

### 5. tool-only パスの許容

`related_articles` SubIntent は記事がインデックスされていない場合でもツール結果のみで prompt を組み立て可能にする。`len(contexts) == 0` の early return を `SubIntentRelatedArticles` のとき免除。

### 6. Observability

`AnswerDebug` に `RetrievalPolicy`（article_only / tool_delegated / article+general 等）と `GeneralRetrievalGated`（general 抑制フラグ）を追加。`ToolsUsed` にハードコードの `"tool"` ではなく実ツール名を記録するよう修正。

### 検討した代替案

**LLM による SubIntent 分類** — Gemma 3 4B のキーワード分類精度が ~79% であるため棄却（[[000568]] の設計判断を踏襲）。キーワードマッチの方が決定論的で信頼性が高い。

**related_articles を buildPrompt 冒頭で分岐する tool-first path** — 設計として綺麗だが、buildPrompt の大規模リファクタが必要。最小差分として `len(contexts) == 0` の免除 + 既存 tool dispatch 位置の活用を選択した。

## Consequences

### Pros

- 「技術的な詳細をもっと教えて」に対して記事チャンクのみで直接回答し、汎用要約が返らなくなる
- 「関連する記事はある？」で `RelatedArticlesTool` が発火し、Meilisearch ベースの確定的な関連記事検索が行われる
- critique / opinion / implication は quality gate を通さないと general チャンクがマージされなくなり、無関係チャンク混入が減少する
- SubIntentNone のデフォルト動作は変更なし — 既存の全フローが後方互換で動作する
- debug 出力に検索ポリシーとツール名が含まれるため、問題の切り分けが容易になる

### Cons / Tradeoffs

- キーワードの重複（例: 「技術的な弱点」は `detail` が `critique` より先にマッチ）がある — 優先順位に依存するため、エッジケースで意図と異なる分類が発生しうる
- `related_articles` の tool-only path は `len(contexts) == 0` の免除という局所的な対応であり、buildPrompt の構造的な改善（tool-first path）は今後の課題として残る
- OpenAPI spec の `AnswerDebug` にはまだ新フィールドが反映されていない（Connect-RPC Stream 経由では利用可能）
- 評価データセット（eval fixtures）は未作成 — observability 追加後に実データで構築する計画
