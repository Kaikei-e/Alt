# ADR-000429: 外部 Ollama の常駐モデルを整理して Ask Augur の運用容量を確保する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- knowledge-augur — 外部 Ollama のモデル配置、pull/create/preload 運用
- rag-orchestrator — 外部 knowledge-augur に依存する生成経路

## Context

Ask Augur の回答生成は外部 Ollama 推論基盤に接続している。この構成では、モデル pull と常駐ロードの成否が推論基盤側のディスク容量と同時配置モデル数に強く依存する。

`gemma3-4b-rag` への移行時、外部 Ollama には 12B 系 Gemma と複数の Qwen 系モデル、さらに pull 失敗時の partial blob が残っており、新しい 4B モデル取得時に容量不足が発生した。モデル切替のたびに過去モデルを積み上げる運用では、将来も同様の失敗が再発する。

また、Ask Augur では回答生成モデルは同時に 1 系統しか使わず、Qwen 系は品質要件から本番採用対象外である。したがって、外部 Ollama を「検証済みの現行モデルを少数常駐させる運用」に寄せた方が、容量制約のある推論基盤では合理的である。

## Decision

Ask Augur 用の外部 Ollama では、現行運用に不要な旧モデルと不完全なダウンロード成果物を削除し、少数モデル常駐を前提とした運用に切り替える。

具体的には以下を採用する。

- `gemma3-12b-rag` とそのベースモデルなど、移行後に不要となる 12B 系モデルを削除する
- 品質要件から採用しない Qwen 系モデルを外部 Ollama から削除する
- pull 失敗時に残る partial blob を定期的に整理し、再試行可能な状態を保つ
- 新モデル導入時は `pull -> create -> preload -> list/ps 確認` を一連の運用手順として扱う
- 外部 Ollama には Ask Augur の現行生成モデルを優先的に常駐させる

代替案として、旧モデルを温存したまま別ストレージを都度確保する案や、複数モデルの常時共存を前提にする案もあった。しかし、外部推論基盤の役割は Ask Augur の安定運用であり、容量競合を常態化させる設計は優先しない。

## Consequences

### Pros

- 新モデル導入時の容量不足による pull 失敗を避けやすくなる
- Ask Augur の現行モデルを安定して preload しやすくなる
- 運用対象モデルが明確になり、モデル棚卸しや障害切り分けが簡単になる
- `rag-orchestrator` 側から見た生成先の挙動が安定し、E2E 疎通確認を標準化できる

### Cons / Tradeoffs

- 旧モデルを即時に比較・ロールバックしたい場合は再 pull が必要になる
- 外部推論基盤を手動運用している限り、モデル整理の規律を守らないと再び容量競合が起きる
- 検証用に複数モデルを並べて長期保持する運用には向かない
- モデル資産を広く保持するよりも、安定運用を優先する判断になる
