# ADR-000806: HyDE サニタイザで ASCII 制御文字と改行を除去し search-indexer のクロスリンガル経路を修復する

## Status

Accepted

## Date

2026-04-20

## Affected Services

- **acolyte-orchestrator (domain)** — `sanitize_hyde_output` の整形パイプラインに制御文字除去層を 1 段追加
- **acolyte-orchestrator (tests)** — `tests/unit/test_hyde.py` に 4 ケース追加

## Context

HyDE ([[000795]]) は日本語トピックに対して Gemma 4 26B MoE (AIX) で仮説英語 passage を生成し、Meilisearch hybrid search ([[000794]]) に追加 query variant として RRF 投入する経路である。ADR-000801 / ADR-000803 / ADR-000805 で空応答・サンプラ・プロンプトインジェクション耐性を順次固めてきたが、**2026-04-19 のイラン情勢レポート (`eb67863f-f6f0-48a4-bd76-a9f1d9a79054`) の v4-v8 を再生成しても英語引用は 0 件のまま**だった。

`acolyte-orchestrator` と `search-indexer` の同時刻ログを突合すると、HyDE EN variant が送られた瞬間に search-indexer 側で以下が発火していた:

```
# search-indexer 2026-04-19T16:52:44
level=ERROR msg="search failed" err="security validation failed: query contains invalid control characters"

# acolyte-orchestrator 同時刻
level=warning event="Gatherer: variant search failed" source=hyde_en
  error="Server error '500 Internal Server Error' for url 'http://search-indexer:9300/v1/search?q=Tensions+in+...+de-escalation.%0A%0AEconomically+...'"
```

URL を展開すると `%0A%0A` が見える。Gemma 4 が 150-word の news-style passage を返すと、段落区切りの `\n\n` が必ず含まれ、HTTP query に percent-encode されると ASCII 制御文字として search-indexer のバリデータに拒否される。つまり `_SYSTEM_PROMPT_EN` が明示する「neutral, factual 150-word English news-style passage」という条件そのものが改行を招き、sanitizer がそれを素通ししていた。

`sanitize_hyde_output` (改修前) が担っていた整形は以下だけで、制御文字は一度も触れられていなかった:

1. Markdown fence 除去
2. XML tag 除去
3. Boilerplate prefix 除去 (`here is` / `本文:` 等)
4. 長さ cap (`max_chars=600`)
5. 言語適合チェック (EN は ASCII ≥40 かつ CJK ≤ ASCII/2、JA は CJK ≥20)

結果として HyDE EN variant が 3 facet すべてで 500 を返し、evidence pool に英語候補が 0 件になり、Curator の `rebalance_by_language` ([[000777]] / [[000796]]) は「pool に EN が無い」ため no-op 化していた。**クロスリンガル recall が観測可能な形で 100% 失敗していた**。

search-indexer 側のバリデータを緩める選択肢もあったが、security 観点から「orchestrator 側で先に削る」方が筋が良い:

- search-indexer の validator は XSS / control-char injection に対する共通防御である
- Orchestrator で落とせば log noise が減り、原因追跡も容易
- Agentic Apps Top 10 2026 ASI-01 (cascading hallucination / 下流からの不正データ流入) に対する defense-in-depth としても意味がある

## Decision

`acolyte/domain/hyde.py` の `sanitize_hyde_output` に 2 行の正規表現置換を追加する:

```python
_CONTROL_CHARS_RE = re.compile(r"[\x00-\x1f\x7f]")
_WHITESPACE_RUN_RE = re.compile(r"\s+")

# XML tag 除去直後に挿入
cleaned = _CONTROL_CHARS_RE.sub(" ", cleaned)
cleaned = _WHITESPACE_RUN_RE.sub(" ", cleaned).strip()
```

順序は既存の整形パイプラインに後付けで差し込み、**XML tag 除去の後、boilerplate prefix 除去と長さ cap の前**に入れる。理由:

- XML tag 除去より前に入れると `<topic>\n</topic>` の改行を空白に畳んでしまい、後続の XML 除去が 1 行の `<topic></topic>` として動かない可能性を残す
- 長さ cap より前に入れないと、制御文字を含む 600 字ちょうどの passage が percent-encode で 1800 字に膨張する
- Boilerplate prefix 除去より前に入れないと、改行を含んだ `Here is\nthe passage:` のような開口句が残る

**却下した代替**:

- **search-indexer 側の validator を緩める** — XSS / control-char injection に対する共通防御を剥がすことになり、HyDE 以外の呼び出し経路 (直接 API / 将来の rag-orchestrator 等) まで晒される。orchestrator 側で削る方がスコープが閉じる。
- **Gemma 4 のプロンプトで「改行禁止」を明示する** — Gemma 4 は CJK + instruct で改行禁止をしばしば無視するうえ、output が 1 行に押し込められると言語適合チェックの精度が下がる。プロンプトは「`Output the passage only — no markdown, no preface, no meta commentary`」のままにして、後処理で物理的に削る。
- **urlencode 時に `safe=""` で特殊文字を強制エスケープする** — URL 上は通っても search-indexer の body 側バリデータが解凍後の実文字列で弾くので効かない。実際に実測で確認した。
- **`\n` を半角スペースに置換するだけ** — DEL (`\x7f`) や NUL (`\x00`) など他の制御文字を見逃すと同じクラスの障害を将来踏む。`[\x00-\x1f\x7f]` で C0 range + DEL を一括除去する方が将来のモデル差替えにも耐える。

## Consequences

### Pros

- **クロスリンガル recall が HyDE 経路で実効化する**。英語記事が候補プールに入り、`rebalance_by_language` の quota 再配置が意味を持つようになる。
- **defense-in-depth の 2 層目**が HyDE 経路に追加される。search-indexer の validator 更新があっても Orchestrator 側の仕様は独立。
- **RED 4 test が回帰防止として残る**。将来誰かが sanitizer を書き直しても newline / CR / tab / その他 control char / whitespace collapse の 4 不変条件が落ちた瞬間にテストが落ちる。
- **long tail の passage が短くなる**。連続空白 collapse で max_chars=600 の利用効率が上がり、BM25 クエリの密度が上がる。

### Cons / Tradeoffs

- **改行が空白化されるため、段落構造を期待する後段 (例: 将来の citation hover) が使えない**。HyDE passage は retrieval query としてのみ使う仕様なので許容。
- **sanitizer のレイヤー数が増える**。読みやすさに影響するが、各層は single-purpose で順序が documented なため許容範囲。
- **Gemma 4 26B MoE そのものの挙動は変わっていない**。プロンプトを書き換えて改行を出さないように矯正することは将来可能だが、モデル差替え (例: dense 31B → 26B MoE 戻し) に耐える汎用策ではない。本 ADR は「出力を削る」方針で fix を閉じる。
