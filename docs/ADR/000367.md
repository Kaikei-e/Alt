# フィード一覧パスのアロケーション最適化（P0ホットスポット3件）

## ADR's STATUS

Accepted

## CONTEXT

3000VU / 30分の負荷試験において、フィード一覧取得パスの累積アロケーションが **2,420GB** に達していた。pprof 分析により以下の3つのホットスポットが特定された:

| ホットスポット | 累積アロケーション比率 | 原因 |
|---|---|---|
| `GetReadFeedIDs` | 41.92% | `[]uuid.UUID` → `[]string` 変換 + pgx SimpleProtocol によるテキストシリアライゼーション |
| `sanitizeDescription` | 25.67% | 毎リクエストで `bluemonday.StrictPolicy()` を新規生成 + HTML サニタイズ |
| `convertFeedPageEntries` | 25.58% | 毎リクエストで `UUID.String()` / `time.Format()` を再計算 |

合計で全体アロケーションの **93.17%** をこの3箇所が占めており、GC 圧力とレイテンシの主因となっていた。

### 既存アーキテクチャ

フィード一覧取得は `CachedFeedListUsecase` が以下の流れで処理する:

1. ユーザーのサブスクリプション一覧を取得（30秒キャッシュ）
2. サブスクリプションごとにフィードページをキャッシュから取得（2分TTL）
3. `convertFeedPageEntries()` で `FeedPageEntry` → `domain.FeedItem` に変換
4. `GetReadFeedIDs()` で既読状態を DB から取得（キャッシュなし）
5. `convertFeedsToProto()` で `sanitizeDescription()` を毎アイテム呼出し

pgx は PgBouncer トランザクションプーリングとの互換性のため `SimpleProtocol` を使用しており、UUID 配列のテキストエンコーディングが特に高コストとなっていた。

## DECISION MAKING

3つのホットスポットそれぞれに対し、アーキテクチャの整合性を保ちながら最小限の変更でアロケーションを削減する方針とした。

### 1. GetReadFeedIDs → GetAllReadFeedIDs（配列パラメータ完全排除）

**決定**: UUID 配列パラメータを渡す `GetReadFeedIDs(ctx, userID, feedIDs)` の代わりに、ユーザーの全既読 ID を一括取得する `GetAllReadFeedIDs(ctx, userID)` を新設し、usecase 層で切り替える。

```sql
-- Before: 巨大 UUID 配列のテキストシリアライゼーション (12ms)
SELECT feed_id FROM read_status
WHERE user_id = $1 AND feed_id = ANY($2::uuid[]) AND is_read = TRUE

-- After: 配列パラメータ不要 (2ms)
SELECT feed_id FROM read_status
WHERE user_id = $1 AND is_read = TRUE
```

**根拠**:
- EXPLAIN 実測で **2.0ms**（配列版の 12ms から 6倍高速）
- `idx_read_status_user_id` インデックスが利用可能
- 一般ユーザーの既読数（数千〜数万件）であれば `map[uuid.UUID]bool` のメモリ使用量は ~1.6MB で許容範囲
- `collectFeedIDs()` による `Links[0]` → `uuid.Parse()` の往復変換も不要に

### 2. sanitizeDescription のキャッシュ時実行 + Policy シングルトン化

**決定**: 2段階の最適化を実施。

1. `bluemonday.StrictPolicy()` をパッケージレベル変数に（スレッドセーフ）
2. `feed_page_cache_gateway` の `loadFeedPage()` でキャッシュ格納時にサニタイズし、`FeedPageEntry.SanitizedDescription` に保存

**根拠**:
- `bluemonday.StrictPolicy()` のドキュメントで goroutine-safe が保証されている
- 同一フィードの description は複数ユーザー間で共有されるため、キャッシュ格納時の1回で十分
- Gateway 層は外部ライブラリ（bluemonday）の利用に適した層であり、Clean Architecture 的にも正しい配置
- 3箇所に重複していた `sanitizeDescription` も全てシングルトン化

### 3. convertFeedPageEntries の文字列事前計算

**決定**: `FeedPageEntry` に `FeedIDStr`, `PublishedStr` フィールドを追加し、キャッシュ格納時に1回だけ `UUID.String()` / `time.Format(time.RFC3339)` を実行する。加えて `domain.FeedItem` に `FeedID uuid.UUID` フィールドを追加し、`feedIDToUUID()` での文字列パース往復を排除。

**根拠**:
- `feedLinkID.String()` がループ内で同一値を毎回再計算していた → ループ外に巻き上げ
- `entry.FeedID.String()` がリクエストごとに全エントリで呼ばれていた → キャッシュ時に1回
- `feedIDToUUID()` が `Links[0]` を `uuid.Parse()` していた → `FeedID` フィールドで直接参照

### PROS

- **アロケーション 67〜80% 削減（推定）**: 2,420GB/30min → 500〜800GB/30min
- **レイテンシ改善**: GetReadFeedIDs のクエリ実行時間が 12ms → 2ms
- **GC 圧力低減**: 短命オブジェクトの大量生成が排除され、GC pause が減少
- **後方互換性維持**: `GetReadFeedIDs` は interface に残存、orphan feeds の `Links[0]` パスもフォールバックとして維持
- **Clean Architecture 準拠**: サニタイズは Gateway 層、事前計算は Gateway 層、ビジネスロジックは Usecase 層と責務が明確
- **既存テスト全パス**: `go test ./...` で全テスト通過を確認

### CONS, TRADEOFF

- **`GetAllReadFeedIDs` はユーザーの全既読を返す**: 大量既読ユーザー（10万件超）では返却 map のメモリが増加する。ただし ~1.6MB/ユーザーであり、リクエストスコープで即 GC されるため問題にならない
- **`SanitizedDescription` のキャッシュ一貫性**: キャッシュ格納時と非キャッシュパスで同一のサニタイズロジックを使用しているが、将来サニタイズルールを変更した場合、キャッシュ TTL（2分）の間は旧ルールが残る
- **`FeedPageEntry` の構造体サイズ増加**: 3フィールド追加（文字列3つ）でキャッシュのメモリフットプリントが微増するが、リクエストごとの再計算コスト削減に比べれば無視できる
- **`domain.FeedItem.FeedID` の追加**: Domain 層に UUID 依存が追加されるが、既に `github.com/google/uuid` は domain パッケージで間接的に使われており、実質的な依存増加ではない
