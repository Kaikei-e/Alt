# Recap Meilisearch Indexing — JSONB全文スキャンからMeilisearch全文検索への移行

## ADR's STATUS

Accepted

## CONTEXT

Tag Verse の `SearchRecapsByTag` RPC は、recap-worker の PostgreSQL に対して以下のクエリを実行していた:

```sql
SELECT DISTINCT ON (rj.job_id, ro.genre)
  ...
FROM recap_jobs rj
JOIN recap_outputs ro ON rj.job_id = ro.job_id
JOIN recap_subworker_clusters c ON ro.output_id = c.output_id
WHERE rj.status = 'completed'
  AND EXISTS (
    SELECT 1 FROM jsonb_array_elements_text(c.top_terms) AS term
    WHERE term ILIKE '%{search_term}%'
  )
```

`jsonb_array_elements_text` による配列展開 + `ILIKE` パターンマッチは全行スキャンであり、B-tree/GIN インデックスによる高速化が困難。データ量の増加に伴い検索レイテンシが実用に耐えない水準に達した。

一方、search-indexer は既に articles に対するデュアルフェーズ（backfill + incremental）の Meilisearch インデキシングを実装しており、同じパターンを recaps に適用できる基盤が整っていた。

### マイクロサービス境界の制約

Alt のアーキテクチャでは、各サービスが自身のデータベースのみにアクセスする原則がある。Recap DB は recap-worker が所有しており、search-indexer が直接アクセスすることはアンチパターンとなる。

## DECISION MAKING

### アーキテクチャ: ポーリングベースのインデキシング

recap-worker が indexable データ API を公開し、search-indexer がそれをポーリングして Meilisearch `recaps` インデックスに投入する設計を採用した。

```
recap-worker (Rust)              search-indexer (Go)              alt-backend (Go)
┌──────────────────┐            ┌────────────────────┐           ┌───────────────┐
│ Recap DB (owner) │            │                    │           │               │
│                  │  HTTP GET  │  polling loop      │  Connect  │ SearchRecaps  │
│ /v1/recaps/      │◄──────────│  backfill +        │  -RPC     │ ByTag         │
│ genres/indexable │            │  incremental       │◄──────────│               │
│                  │            │       ↓             │           │               │
│                  │            │  Meilisearch       │           │               │
│                  │            │  "recaps" index    │           │               │
└──────────────────┘            └────────────────────┘           └───────────────┘
```

### 1. recap-worker (Rust): Indexable API の追加

- 新エンドポイント `GET /v1/recaps/genres/indexable?since=<RFC3339>&limit=200`
- `since` パラメータで差分取得（incremental indexing 用）、省略時は全件（backfill 用）
- `AppState` に `pool()` アクセサを追加し、既存の DAO trait 階層を変更せずにスタンドアロン静的メソッドとして実装
- レスポンス: `{ results: [...], has_more: bool }`

### 2. search-indexer (Go): Recap インデキシングの追加

Clean Architecture の全レイヤーにわたる新規実装:

| レイヤー | ファイル | 役割 |
|---------|---------|------|
| Domain | `domain/recap_document.go` | `RecapDocument` エンティティ |
| Port | `port/recap_repository.go` | `RecapRepository` インターフェース |
| Port | `port/recap_search_engine.go` | `RecapSearchEngine` インターフェース |
| Driver | `driver/recap_api/client.go` | recap-worker HTTP クライアント |
| Driver | `driver/meilisearch_recap_driver.go` | Meilisearch `recaps` インデックス操作 |
| Gateway | `gateway/recap_repository_gateway.go` | API レスポンス → ドメインモデル変換 |
| Gateway | `gateway/recap_search_engine_gateway.go` | ドメイン ↔ ドライバモデル変換 |
| Usecase | `usecase/index_recaps.go` | デュアルフェーズインデキシング |
| Usecase | `usecase/search_recaps.go` | Meilisearch 検索 |
| Handler | `connect/v2/search/handler.go` | `SearchRecaps` RPC ハンドラ |
| Bootstrap | `bootstrap/app.go` | `runRecapIndexLoop` goroutine |

#### Meilisearch `recaps` インデックス設計

- Primary Key: `id` (= `{job_id}__{genre}` 複合キー)
- Searchable: `top_terms`, `summary`, `genre`
- Filterable: `genre`, `window_days`
- Sortable: `executed_at`

#### デュアルフェーズインデキシング

既存の articles インデキシングと同一パターン:
- **Phase 1 (Backfill)**: `since` なしで全既存レコードをインデキシング
- **Phase 2 (Incremental)**: 最終 `executed_at` タイムスタンプ以降の差分を 5 分間隔でポーリング

### 3. Proto: SearchRecaps RPC の追加

`services/search/v2/search.proto` に `SearchRecaps` RPC を追加:

```protobuf
rpc SearchRecaps(SearchRecapsRequest) returns (SearchRecapsResponse);
```

サーバー側 (`services/`) とクライアント側 (`clients/`) の両方の proto を更新し、コード生成を実行。

### 4. alt-backend: 検索ルーティングの変更

`RecapGateway.SearchRecapsByTag` を、recap-worker への直接 HTTP リクエストから search-indexer の Connect-RPC `SearchRecaps` 経由に変更。

- `RecapGateway` に `SearchIndexerPort` を注入
- DI コンテナで既存の `searchIndexerDriver` を `NewRecapGateway` に渡す
- recap-worker への HTTP 呼び出しを完全に排除（recap/search エンドポイントは不要に）

## RESULTS, EFFECTS

### PROS

- **検索速度の大幅改善**: JSONB 全行スキャン → Meilisearch 転置インデックスによる O(1) に近い検索
- **マイクロサービス境界の尊重**: Recap DB へのアクセスは recap-worker のみ、search-indexer は HTTP API 経由
- **既存パターンの再利用**: articles のデュアルフェーズインデキシングと同一設計のため、運用・監視が統一される
- **段階的起動**: `RECAP_WORKER_URL` が未設定の場合、recap indexing は無効化されサービスは正常に起動する
- **recap-worker の DB 負荷軽減**: 検索リクエストが recap-worker を経由しなくなる

### CONS, TRADEOFF

- **データの最終整合性**: ポーリング間隔（デフォルト 5 分）分のラグが発生する。リアルタイム性は不要な用途のため許容
- **Meilisearch の運用負荷**: `recaps` インデックスが追加されるため、Meilisearch のメモリ使用量が増加。ただし recaps のデータ量は articles と比較して小さい
- **recap-worker に新 API を追加**: `/v1/recaps/genres/indexable` エンドポイントの追加。既存の search エンドポイントは将来的に非推奨化可能
