# ADR-000829: /search?q=n8n の articles 欠落を解消するため search-indexer の synonyms PUT をバッチ coalesce し alt-backend の global-search section timeout を 10 秒に引き上げる

## Status

Accepted

## Date

2026-04-23

## Affected Services

- search-indexer (Go) — `usecase/index_articles.go` と `usecase/index_articles_test.go` を変更。synonyms 登録の call-fan-out を per-batch 1 回に集約。
- alt-backend (Go) — `usecase/global_search_usecase/usecase.go` と `usecase/global_search_usecase/usecase_test.go` を変更。`sectionTimeout` を 3s → 10s。
- meilisearch (v1.x) — 挙動変更を受ける側。コードは触らない。articles index の `/settings/synonyms` PUT 頻度が大幅に下がる。
- alt-frontend-sv — ユーザが最終的に見る `src/routes/(app)/search/+page.svelte` の赤い "Some sections are unavailable: articles" バナーが出なくなることを期待する。FE は変更しない。

## Context

ログイン済みのユーザが `/search?q=n8n` を開いたとき、FE 側で以下のバナーが表示される挙動を観測した。

> Some sections are unavailable: articles. Results may be incomplete.

原因は以下のスタックで確定した。

1. **FE → BFF → alt-backend の経路は健全**。alt-butterfly-facade の mTLS peer ログで `/alt.search.v2.GlobalSearchService/SearchEverything` への到達は確認できる。
2. **alt-backend 側で articles 検索だけが `deadline_exceeded` を出す**。`global search ok query=n8n degraded=[articles]` が複数回記録されていた。
3. **deadline の正体は `const sectionTimeout = 3 * time.Second`**（`global_search_usecase/usecase.go`）。3 つの section goroutine にそれぞれ適用されていた。
4. **3 秒を食い潰していたのは search-indexer→Meilisearch の検索 RTT**。alt-backend 内から `wget http://search-indexer:9301/services.search.v2.SearchService/SearchArticles` を叩くと 7ms で 200 が返る一方、実運用の `SearchArticles` は遅延する。
5. **Meilisearch のログで原因を特定**。`POST /indexes/articles/search` の `time.idle` が **5.05 秒**まで伸びるサンプルが観測され、直前に `PUT /indexes/articles/settings/synonyms` が 15 秒おきに繰り返し enqueue されていた。Meilisearch は LMDB task queue を使っており、settings 書き込みは index 単位で search と同じ writer lock を奪うため、synonyms の連続 PUT が search スロットを塞いでいた。
6. **synonyms PUT の源は `search-indexer/app/usecase/index_articles.go`**。4 つの indexing 経路すべてが per-doc で `u.searchEngine.RegisterSynonyms(ctx, synonyms)` を呼んでおり、fat-event でバッチに 10 件届くと 10 回の PUT が走っていた。しかも Meilisearch の synonyms は **full-replace** なので、後続の PUT が前の PUT を即座に上書きし、最終状態は「直前の 1 docs 分のみ」という機能不全にもなっていた。
7. DB 側 (`feed_tags` 662,288 行) への `SearchTagsByPrefix('n8n', 10)` は `EXPLAIN ANALYZE` で 132.9 ms、`GetFeedURLsByArticleIDs` は 0.96 ms だった。DB は今回の degraded に寄与していない（tag 検索の seq scan は将来の宿題だが別 PR 扱い）。

ユーザ仮説「直近の mTLS 化が主因」は半分正しく、切替自体が壊したわけではないが、その時期に追加された機能群が synonyms 更新の流量を増幅していた可能性は残る。今回の修正は mTLS 経路に触れず、Meilisearch に届く PUT 頻度と alt-backend 側の timeout ceiling の両方を塞ぐ。

補助的な Web 調査結果（公式優先）:

- Meilisearch の search と index-write は **論理的には別の queue** だが、同一 index への writer lock が search の即応を妨げることが設計上あり得る（[Tasks and asynchronous operations](https://www.meilisearch.com/docs/learn/async/asynchronous_operations)）。`PATCH /indexes/{indexUid}/settings` は複数 setting を 1 タスクに畳めるので、**settings の burst は coalesce が公式ベストプラクティスに近い**。
- search queue 自体は v1.8.0 で「1 search per core」モデルに固定された（[issue #4489](https://github.com/meilisearch/meilisearch/issues/4489)）。task queue の飽和事例は [issue #5161](https://github.com/meilisearch/meilisearch/issues/5161) で 60 秒超の処理遅延が報告されている。

## Decision

以下の 2 つの変更を **同じ PR / 同じ ADR で採択**する。どちらか単独ではユーザ体験上の "articles 欠落" を一貫して防げないため、セットで扱う。

### 1. search-indexer: synonyms PUT を per-batch 1 回に coalesce する (root fix)

`search-indexer/app/usecase/index_articles.go` の 4 経路すべてに対して、per-doc ループを解体し、次の helper に置き換える:

```go
func (u *IndexArticlesUsecase) registerBatchSynonyms(ctx context.Context, docs []domain.SearchDocument) {
    if len(docs) == 0 {
        return
    }
    merged := make(map[string][]string)
    for _, doc := range docs {
        maps.Copy(merged, tokenize.ProcessTagToSynonyms(u.tokenizer, doc.Tags))
    }
    if len(merged) == 0 {
        return
    }
    _ = u.searchEngine.RegisterSynonyms(ctx, merged)
}
```

これにより batch 1 件あたりの Meilisearch `/indexes/articles/settings/synonyms` PUT 数は N → 1 に落ちる。さらに Meilisearch synonyms は full-replace であるため、**batch 全体の union** を 1 回で送ることで「直前の 1 docs 分だけが反映される」機能不全も同時に解消する。

### 2. alt-backend: global-search の section timeout を 3s → 10s に引き上げる (headroom fix)

`alt-backend/app/usecase/global_search_usecase/usecase.go` の `sectionTimeout` を `const 3 * time.Second` から `var 10 * time.Second` に変更する。`var` 化の主な目的はテストで任意の短縮値を注入して behavioural assertion を回せるようにすることで、production 既定値は 10s に固定する。

### 検討した代替案と却下理由

- **(A) sectionTimeout 単独の引き上げ (synonyms 側を放置)** — Meilisearch 側の `time.idle` が 5 秒を超えるケースがさらに伸びたときに再発する。user-facing TTFT も悪化する。degraded banner の頻度は下がるが根治しない。
- **(B) Meilisearch の `/indexes/articles/settings` を `PATCH` で複数 settings 同時更新にまとめる** — 公式はサポートしているが、今回の flood の原因は settings 種別の数ではなく「文書が来る頻度で PUT が走る」点なので方向違い。per-batch 1 回の coalesce の方が小さい変更で effect が大きい。
- **(C) debounce / quiet-window flush (`bep/debounce` 相当)** — 効果は大きいが、プロセス restart / crash 時に未 flush な synonyms を失いうる。現在の `IndexArticlesUsecase` は同期 API であり flush 境界を明確に保つのは将来拡張の話。今回は batch という既存の flush 境界に乗る方が堅い。
- **(D) Meilisearch の search 専用 listener / search 分離 index** — インフラ変更が重く、本件の root cause（write lock 奪取）に対する代償が大きい。将来の宿題。
- **(E) synonyms 機能そのものを無効化** — 日本語タグの検索品質が下がる。受け入れられない。
- **(F) alt-backend → search-indexer を mTLS 経路 (`https://search-indexer:9443`) に移す** — PM-2026-025 / PM-2026-026 の歴史的経緯で未完の宿題だが、今回の degraded とは独立の issue。同じ PR に載せると変更の責務が混ざるため別 ADR 扱いとする。
- **(G) `feed_tags (LOWER(tag_name) text_pattern_ops)` btree を追加して tag 検索を高速化** — 132ms は sectionTimeout 3s でも通っており degraded には効かない。Atlas migration を伴う別件として切り出す。

## Consequences

### Pros

- `/search?q=n8n` 含む articles 検索の degraded 頻度が大きく低下する。ユーザから見た挙動として「search は効くが articles だけ出てこない」silent UX failure を 1 件ぶん塞ぐ。
- Meilisearch の `PUT /indexes/articles/settings/synonyms` 回数が fat-event 1 batch あたり最大 N 倍から 1 回に落ちる。LMDB task queue 圧迫の持続的な背景原因が消える。
- synonyms full-replace の「最後の 1 docs 分のみ」機能不全が副産物として解消され、日本語タグの synonym 展開が意図通りに動く。
- sectionTimeout の `var` 化により、今後の section-level レイテンシ改善・回帰テストで behavioural assertion が書きやすい。
- TDD の RED → GREEN → REFACTOR を 4 コミットで明示的に分割できたため、diff を読む後続メンテナが意図を追いやすい（commits `9a48aa694` / `861d853f0` / `e099c0603` / `83bc08c2b`）。

### Cons / Tradeoffs

- ユーザ体感の worst-case TTFT が 3s → 10s に伸びる。現実のメディアンは synonyms 修正で改善するが、病的な Meilisearch 詰まりが再発したときに 10 秒のローディングを FE が見せる。LLM 呼び出し並の体感になる可能性がある。FE 側で progressive rendering（section 単位で先に出す）を足す将来改修で緩和する想定。
- `sectionTimeout` が `var` になったため、テストが並列で値を書き換えると race を起こす余地がある。現状の `t.Cleanup` + 直列テスト実行で保たれているが、将来 `t.Parallel()` を足す前に `sync.RWMutex` でラップする必要が出るかもしれない。小さな技術的負債として残す。
- per-batch union の synonyms はメモリ上にすべてのタグ→トークン列を抱える。1 batch が 200 件（既定 batch size）の日本語タグを含む worst-case でも数百エントリで収まる見込みだが、今後 batch size を大きく上げる場合は再評価が必要。
- Meilisearch の synonyms 数が 1 batch 分から N batch 累積に増えるわけではない（依然 per-batch union で上書き）ため、**過去のタグに対する synonyms は次の batch に該当タグが出てこないと失われる**。これは従来の挙動より改善だが、「全タグの synonyms を起動時に一括 PUT する」恒常的な正準状態を作るのは本 ADR の範囲外。将来 ADR として [[search-indexer-synonyms-bootstrap-replay]] 相当の受け皿が必要。
- alt-backend → search-indexer の経路は依然 plaintext `http://search-indexer:9301`。mTLS 移行は本 ADR に含まれない（§Context 参照）。

## Deploy-model 整合性セルフチェック

本 ADR は既存 Go service 2 つのコード変更のみで、compose service の追加・削除、`depends_on`、`restart`、volume は一切触らない。per-service rolling deploy model に対しての追加リスクは無い。

- [ ] 新設 compose service に `restart: "no"` の one-shot init container を含むか？
  - N/A (compose を触らない)
- [ ] 新設名前付き volume を populate する責務はどこにあるか？
  - N/A (volume を触らない)
- [ ] いずれも該当しない: 「rolling 互換」と明記して pass
  - **YES. rolling 互換.** alt-backend / search-indexer を独立に rolling replace してもスキーマ・契約の非互換は無い。古い alt-backend (sectionTimeout=3s) と新しい search-indexer (coalesced PUT) の組合せは degraded 頻度が低下するだけで機能回帰は起きない。逆 (新 alt-backend + 旧 search-indexer) でも頻度は低下し、悪化経路は無い。
