# ADR-000488: 自動全文取得で Scraping Policy と declined domain 判定を再利用する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-backend (Go)

## Context

自動全文取得を新設する際に、手動 Fetch とは別の robots / domain 制御を持ち込むと、Alt 全体で外部サイトへのアクセス規律が二重化する。個人利用前提であっても、外部ドメインへのアクセスはアプリケーション全体で同一の方針に縛る必要がある。

既存実装には以下の資産があった。

1. **ScrapingPolicyGateway** — domain policy、robots、crawl-delay 相当の判断を集約している
2. **declined domain 管理** — user 単位で明示的に fetch しない domain を保存できる
3. **FetchArticleGateway の制御** — host 単位の待機や SSRF 防御を既に持っている

新しい自動経路がこれらを迂回すると、同じ article URL でも手動と自動で許可判定がずれる危険がある。

## Decision

自動全文取得では、新しい独自 policy を作らず、既存の Scraping Policy と declined domain 判定をそのまま再利用する。

### 実装方針

- user ごとの処理開始前に `declined domain` を確認する
- domain が未拒否であれば、既存の `ScrapingPolicyPort` に article URL を渡して fetch 可否を判定する
- policy が明示的に拒否した場合は article を作成せず、その domain を declined list に保存する
- policy チェックに失敗した場合は、既存手動経路と同様に fail-open で扱う

### 採用理由

- **一貫性** — 手動 Fetch と自動 Fetch で外部アクセスの意味論を揃えられる
- **保守性** — robots / domain / crawl-delay の判断ロジックを 1 箇所に寄せられる
- **ユーザー境界** — 同じ feed source でも user ごとの declined state を尊重できる

### 代替案

- **自動 Fetch 専用 policy を新設する案**  
  自動化向けの最適化はしやすいが、判定差分の説明が難しくなり、保守対象も増えるため採用しない。

- **global declined domain のみで制御する案**  
  実装は単純だが、既存の user 単位モデルと整合しないため採用しない。

## Consequences

### Pros

- 手動・自動で同じ外部アクセス規律を共有できる
- policy 変更が 1 系統に集約され、将来の監査や修正が容易になる
- user ごとの declined state をそのまま使える

### Cons / Tradeoffs

- policy gateway の挙動変更が自動全文取得にも直接影響する
- fail-open の判断もそのまま継承するため、より厳格な自動専用モードは別途必要になる可能性がある
- 自動経路の観測性は今後さらに強化する余地がある
