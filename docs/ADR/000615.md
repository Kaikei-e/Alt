# ADR-000615: 品質チェック削除時に summarize_job_queue の補償トランザクションを追加する

## Status

Accepted

## Date

2026-04-03

## Affected Services

- pre-processor (Go)

## Context

品質チェックが低品質と判定した要約を `article_summaries` から削除した後、`summarize_job_queue` に残存する `completed` ジョブとの不整合により、最大 24 時間の要約停滞（デッドロック）が発生していた。

### 発生メカニズム

1. 要約ジョブが正常完了 → `article_summaries` に保存、`summarize_job_queue` を `completed`（summary カラムに要約テキスト格納）
2. 品質チェック (`JudgeArticleQuality`) が低品質と判定 → `article_summaries` から削除
3. 次回バッチエンキュー (`EnqueueUnsummarizedBatch`) → alt-backend が「未要約」として記事を返す
4. `ShouldQueueSummarizeJob` ガード ([[000513]]) の 2 段階チェック:
   - `summaryRepo.Exists()` → `false`（正しく通過）
   - `jobRepo.HasRecentSuccessfulJob()` → `true`（24h 以内の completed ジョブ + 非 NULL summary が残存）
5. `recent_success` でスキップ → 記事が未要約のまま最大 24 時間停滞

### 影響

- 検知時点で **51 件の記事が未要約のまま停滞**
- `batch enqueue completed: found=10, enqueued=0, skipped=10` が繰り返し発生
- 品質チェックで削除されるたびに停滞件数が累積する構造的問題

### 根本原因

[[000513]] のガード設計は「上流の再通知・再送に対する防御」を目的としており、品質チェックによる意図的な削除を想定していなかった。`RemoveLowScoreSummary` が `article_summaries` を削除する際に `summarize_job_queue` を更新しないため、2 つのテーブル間でデータ不整合が発生していた。

## Decision

**品質チェックが要約を削除した際に、`summarize_job_queue` の対応する completed ジョブの `summary` カラムを NULL 化する補償トランザクションを追加する。**

### 設計パターン

Saga パターンの Compensating Transaction（補償トランザクション）を適用する。品質チェック削除は「要約生成」ローカルトランザクションの結果を取り消す補償操作であり、ジョブキュー側の状態も同期的に補償する必要がある。

### 補償の仕組み

`HasRecentSuccessfulJob()` の SQL 条件は `summary IS NOT NULL AND summary <> ''` を含む。completed ジョブの `summary` を `NULL` に設定するだけで、ガードが `false` を返し re-enqueue が許可される。

- ジョブの `status` は `completed` のまま維持（事実として完了した）
- `summary = NULL` は「出力が品質チェックにより却下された」ことを示す

### 実装

1. **Repository 層**: `SummarizeJobRepository` に `InvalidateCompletedJobSummary(ctx, articleID)` メソッドを追加

```sql
UPDATE summarize_job_queue
SET summary = NULL
WHERE article_id = $1
  AND status = 'completed'
  AND summary IS NOT NULL
```

2. **Service 層**: `qualityCheckerService` に `jobRepo` 依存を追加。`CheckQuality` の要約削除検知ブランチと `ProcessLowQualityArticles` の削除成功後に補償を呼び出し

3. **エラーハンドリング**: 補償は best-effort。失敗してもログ出力のみで処理を続行。次回品質チェックまたは 24h 経過後に自然解消される

### 採用しなかった代替案

| 案 | 評価 |
|---|---|
| 新しいジョブステータス `quality_rejected` を追加 | DB スキーマ変更（CHECK 制約更新）が必要。状態遷移が複雑化する。summary NULL 化で十分 |
| `HasRecentSuccessfulJob` で `article_summaries` を cross-check | ガードに cross-service I/O を持ち込む。State Machine Guards は pure & fast であるべき |
| `recentSuccessfulJobWindow` を短縮 | 根本解決にならず、[[PM-2026-002]] の無限ループ防御が弱体化する |

## Consequences

### Pros

- 品質チェック削除後に記事が即座に re-enqueue 対象になる（最大 24 時間の停滞を解消）
- ガードのロジック自体は変更不要 — 既存の `summary IS NOT NULL` 条件が自然に機能
- ジョブの `status` は `completed` のまま維持され、監査証跡が保たれる
- 補償が失敗しても 24h 後に自然解消する安全弁がある

### Cons / Tradeoffs

- `qualityCheckerService` に `SummarizeJobRepository` 依存が追加される（コンストラクタ引数が 1 つ増加）
- 同一記事に対する複数の completed ジョブすべての summary が NULL 化される（最新のみに限定する方が精密だが、全 NULL が安全でシンプル）
- `CheckQuality` 内の品質チェックは `JudgeArticleQuality`（パッケージ関数、実 LLM 呼び出し）を経由するため、unit test では補償の E2E パスを直接テストできない。`ProcessLowQualityArticles` 経由でのテストで代替
