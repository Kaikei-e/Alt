# ADR-000668: Acolyte レポート一覧の再生ボタン削除と Generate ボタンの Connect-RPC ガード付き非活性化

## Status

Accepted

## Date

2026-04-10

## Affected Services

- alt-frontend-sv (TypeScript / SvelteKit)

## Context

Acolyte のレポート一覧画面（モバイル・デスクトップ）に配置されていた ▶ 再生ボタンにより、ユーザーが一覧をスクロール中に誤タップで 9 ノードパイプライン（[[000656]]）の実行を kick してしまうリスクがあった。パイプラインは Gemma4 26B への複数回 LLM 推論を伴い、1 回の実行に数分を要するため、誤実行のコストが高い。

また、詳細画面の Generate ボタンには一切のガードがなく、生成中のレポートに対して重複リクエストを送信可能な状態だった。バックエンドの `StartReportRun` RPC は同一レポートに対する並行実行を `FAILED_PRECONDITION` で拒否するが、フロントエンドがこのエラーコードを区別できていなかった。

### 技術的制約

- `AcolyteReport`（`GetReport` RPC レスポンス）に `latestRunStatus` フィールドが存在しない。`ReportSummary`（`ListReports` レスポンス）にのみ含まれる
- Acolyte の FE クライアントは手動 `rpc()` ラッパーを使用しており、`@connectrpc/connect` の `ConnectError` クラスを直接利用していない
- BFF プロキシ（`/api/v2/[...path]/+server.ts`）はバックエンドのレスポンスボディをそのまま透過するため、Connect-RPC の `{ code, message }` JSON 構造はフロントエンドに届く

## Decision

**2 つの UI 安全策を採用する。**

### 1. レポート一覧から再生ボタンを削除する

レポート生成の起点を詳細画面の Generate ボタンに一元化する。一覧画面はナビゲーション専用とし、意図しないパイプライン実行を構造的に防止する。

削除対象:
- `MobileAcolyteReportCard` の ▶ ボタンと `onStartRun` prop
- `MobileAcolyteList` の `onStartRun` prop チェーン
- デスクトップ一覧の `.card-run-btn` ボタンと CSS
- リスト画面ルートの `handleStartRun` 関数と `startReportRun` import

### 2. Generate ボタンに楽観的 disable + Connect-RPC サーバーガードを適用する

Connect-RPC の構造化エラーレスポンス（`{ "code": "failed_precondition", "message": "..." }`）を活用した 2 層防御を採用する。

**第 1 層: 楽観的 disable** — Generate クリック時に即座に `generating = $state(true)` でボタンを disabled にし、二重クリックを防止する。`startReportRun` 成功後も `generating = true` を維持する。パイプラインは数分間実行されるため、成功レスポンス受信時点でボタンを再有効化するとユーザーが再クリックしてしまう。ページリロードで `generating` は `false` にリセットされる（コンポーネント再マウント）。

**第 2 層: Connect-RPC サーバーガード** — `StartReportRun` が `failed_precondition` を返した場合、ボタンを disabled のまま維持しエラーメッセージを表示する。一時的エラー（ネットワーク障害等）の場合のみ `generating = false` に戻しリトライを許可する。

#### エラーコード保持のための `AcolyteRpcError`

手動 `rpc()` ラッパーに `AcolyteRpcError extends Error` クラスを導入し、Connect-RPC の `code` フィールドを保持する。型ガード `isAlreadyRunning()` で `failed_precondition` を判定する。将来 proto codegen 移行時に `ConnectError` へ置換可能。

```typescript
export class AcolyteRpcError extends Error {
  code: string;
  constructor(code: string, message: string) { ... }
}

export function isAlreadyRunning(err: unknown): err is AcolyteRpcError {
  return err instanceof AcolyteRpcError && err.code === "failed_precondition";
}
```

#### 初回ロード時の制約

`AcolyteReport` に `latestRunStatus` がないため、詳細画面の初回ロード時はボタンが有効のまま表示される。ユーザーが Generate をクリックした際にサーバーが `FAILED_PRECONDITION` で保護する。これは Connect-RPC のエラーハンドリング公式ガイドラインに沿った設計であり、サーバーを source of truth として扱う。

## Consequences

### Pros

- **誤操作の構造的防止** — 一覧画面にアクション起点がなくなり、スクロール中の誤タップによるパイプライン実行が不可能になった
- **Connect-RPC エラーコード活用** — `failed_precondition` を型安全に判定できるようになり、エラー種別に応じた UI 復帰戦略が実現
- **後方互換** — `AcolyteRpcError extends Error` のため、既存の `e instanceof Error` による catch ブロックはすべて動作する
- **サーバーが source of truth** — 初回ロード時のステータス不明は、クリック時のサーバー応答で補完。過剰なクライアント側状態管理を回避

### Cons / Tradeoffs

- **初回ロード時に生成中ステータスが不可視** — `AcolyteReport` に `latestRunStatus` がないため、別タブ・別セッションで開始された生成中状態をページロード時に反映できない。`GetReportResponse` proto 拡張で対応可能（別タスク）
- **Rerun ボタンは未対応** — セクション単位の Rerun ボタンは今回のスコープ外。必要に応じて同じパターンを適用可能
- **手動 `rpc()` ラッパーの技術的負債** — `@connectrpc/connect` の `ConnectError` を直接使えばクラス定義不要だが、proto codegen 移行が前提のため暫定措置として許容
