# ADR-000946: alt-butterfly-facade の BFFConfig を配線し streaming を circuit breaker 対応にする

## Status

Accepted

## Date

2026-07-15

## Affected Services

- alt-butterfly-facade (Go)

## Context

`docs/review/architecturereview20260713.md` の F-07 finding が、`alt-butterfly-facade` の `BFFConfig` (cache/circuit breaker/dedup/error-normalization) が `config/config.go` でデフォルト全機能 `true` に設定済みであるにもかかわらず、`main.go` の `serverCfg := server.Config{...}` 構築でこの値が一切転記されておらず、本番では常に legacy `ProxyHandler` にフォールバックしていたと指摘した (ADR-000928 と同型の「設定済みだが DI で脱落」する silent-fallback)。

調査を進める過程で、当初の前提の一部が古いことが判明した: レビュー時点で懸念されていた「streaming procedure が cache/dedup で全文バッファされ、真の streaming が壊れる」問題は、2026-07-06 の別レビュー対応で既に修正済みだった (`BFFHandler.ServeHTTP` は `isStreamingProcedure` 判定で streaming を `ProxyHandler` へ委譲する早期分岐を既に持っていた)。一方で、この委譲は **circuit breaker を完全に素通り**させており、streaming 経路の接続失敗・成功が CB のカウントに一切反映されない、という未発見の別バグを生んでいた。

## Decision

1. `main.go` に `buildServerConfig()` を新設し、`config.Config` の 10 フィールド (`EnableCache`/`EnableCircuitBreaker`/`EnableDedup`/`EnableErrorNormalization` と `CacheMaxSize`/`CacheDefaultTTL`/`CBFailureThreshold`/`CBSuccessThreshold`/`CBOpenTimeout`/`DedupWindow`) を全て `server.Config.BFFConfig` へ転記する。CLAUDE.md Rule 8 に従い、起動時に機能ごとの loud wiring log (`bff.cache.wiring`, `bff.circuit_breaker.wiring`, `bff.dedup.wiring`, `bff.error_normalization.wiring`、各 `enabled=<bool>`) を追加した。
2. `BFFHandler` に `serveStreaming` を新設し、`ProxyHandler` への委譲 (`proxyHandler` フィールドごと削除) をやめた。streaming procedure でも接続確立・初回レスポンスステータスのみに基づいて circuit breaker の `Allow()` gating と成功/失敗記録を行い、ボディの中身 (chunk 単位で届く streaming データ) には一切関与しない設計とした。error-normalization も初回ステータスがエラーの場合のみ適用し、streaming 開始後は関与しない。cache/dedup は streaming procedure に対して従来通り完全にバイパスする (streamed body は単一のキャッシュ可能単位ではないため)。
3. `ProxyHandler.streamResponse` の chunk-copy + `Flusher.Flush()` ロジックを `streamCopy` として関数抽出し、`BFFHandler`/`ProxyHandler` 両方から共有することで実装の重複を避けた。同様に `resolveTimeout` ヘルパーを抽出し、unary/streaming それぞれのタイムアウト計算 (3 箇所に分散していた) を統合した。
4. 実装過程で `alt-butterfly-facade/Dockerfile` が `go build ... main.go` とファイル名指定でビルドしていたため、新設した `server_config.go` (上記 `buildServerConfig` の実体) が **Docker イメージビルドから静かに除外される**バグを発見した。ローカルの `go build ./...` はパッケージ単位のため検出できず、実際に `docker compose up --build` を実行して初めて顕在化した。他サービスは全てディレクトリ単位 (`go build .` 相当) でビルドしており、alt-butterfly-facade だけがファイル名指定という外れ値だった。`go build . ` へ修正した。

## Consequences

### Pros

- `BFFConfig` の 4 機能が実機で有効化され、`docker compose` 上で認証込みリクエストによる `X-Cache: MISS` → `HIT` の遷移、および 415 レスポンスでの `NormalizedError` 形式を実証した。
- streaming procedure (Ask Augur / オンデマンド要約 / Morning Letter / Knowledge Home live update) の circuit breaker 可視性が確保された。
- Docker ビルドの静かなファイル除外という、ローカルテストでは検出不可能なクラスのバグを本番相当ビルドで発見・修正できた。

### Cons / Tradeoffs

- **circuit breaker の open/half-open 遷移は実機で未検証**。稼働中の本番相当スタックに対して 5 回連続の意図的な backend 障害を誘発するコストが検証価値を上回ると判断し、実施しなかった。起動時 wiring ログと cache/error-normalization が同一 `BFFHandler` インスタンスで機能している事実から蓋然性は高いが、直接観測はしていない。
- **streaming procedure の実チャンク配信タイミング (バッファされていないことの実証) も実機では未検証**。Connect の server-streaming は `application/connect+json` の長さプレフィックス付きフレーミングを要求し、素の curl JSON POST では 415 で弾かれるため。ユニットテストレベルでは検証済み (`TestBFFHandler_StreamingProcedure_CircuitBreaker_*`)。
- `/v1/bff/stats` の実ペイロード (cache/circuit_breaker カウンタの中身) は、検証時点で admin ロールの JWT を正規発行経路から取得できなかったため (稼働中 Kratos の identity schema に `role` trait が定義されていなかった) 未確認。schema 変更は本 PR のスコープ外と判断した。
- streaming path の error-normalization は、unary path (エラーレスポンスを無加工で通す) と意図的に非対称になった。streaming path は初回エラーステータスを `NormalizedError` へ書き換える。挙動差は指示通りだが、レビュー時に確認が必要。
