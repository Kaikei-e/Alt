# ADR-000782: 2026-04-19 Acolyte 502 再発を pki-agent サイドカーの手動 force-recreate で即時復旧し、depends_on.restart の限界を事実として記録する

## Status

Accepted

## Date

2026-04-19

## Affected Services

- **acolyte-orchestrator** — ランタイム挙動の変更なし。sidecar cascade が走らなかった結果、BFF からの mTLS 経路が切れていたのを復旧
- **alt-pki-agent-acolyte-orchestrator** — 手動 `docker rm -f` + `docker compose up -d --no-deps pki-agent-acolyte-orchestrator` で現 orchestrator の netns (`container:dea90fb9…`) に再 attach
- **alt-butterfly-facade** — BFF → acolyte の `connection refused` が停止。FE からの `/api/v2/alt.acolyte.v1.AcolyteService/ListReports` が 502 → 401（認証未添付の正常応答）に回復

## Context

ユーザーから「Acolyte が死んでいる」「FE でレポート一覧が見られない」と報告を受けた。ブラウザコンソール:

```
api/v2/alt.acolyte.v1.AcolyteService/ListReports:1
Failed to load resource: the server responded with a status of 502 ()
```

本 ADR は [[PM-2026-030]] / [[PM-2026-031]] と同じ pki-agent サイドカー孤立 netns パターンの **3 度目** の発火を記録し、runtime 復旧の手順と、[[000759]] で導入した `depends_on.<parent>.restart: true` が**今回も効かなかった**事実を固定化する。

### 症状の突き合わせ

`alt-butterfly-facade` ログ:

```
2026-04-19T09:55:54Z ERROR ... dial tcp 172.18.0.14:9443: connect: connection refused
2026-04-19T10:09:21Z ERROR ... dial tcp 172.18.0.14:9443: connect: connection refused
```

TLS handshake error ではなく **TCP RST**。つまり :9443 に listener が存在しない（pki-agent が届かない netns にいる）。

### コンテナ状態の実測

| オブジェクト | 値 | 判定 |
|---|---|---|
| `acolyte-orchestrator` Id | `dea90fb965b0…` | 2026-04-19 09:37 JST 再作成 |
| `acolyte-orchestrator` IP | `172.18.0.14` | 新 netns |
| `alt-pki-agent-acolyte-orchestrator-1` `HostConfig.NetworkMode` | `container:3fa4d14d0095…` | **旧 orchestrator** の netns を指す |
| pki-agent 内 `ip -4 addr` | `127.0.0.1/8 scope host lo` のみ、eth0 なし | 孤立 netns |
| pki-agent 起動時刻 | 2026-04-17 JST | orchestrator 再作成から 2 日以上据え置き |
| pki-agent healthcheck | `tick ok, state: fresh` | cert 鮮度のみ見て緑を返す（機能 liveness は未検証） |

### なぜ今日この瞬間に発火したか

- 2026-04-19 の [[000776]] 経路の一部 (`compose/compose.yaml` 内の `image:` 明示) がすべての pull-deployed サービスに揃ったことで、`docker compose pull` が初めて acolyte-orchestrator の image 差分を検知し、`up -d` が `--force-recreate` を走らせた
- 親が再作成された時点で既存の pki-agent sidecar は依存する container id を失うが、compose の `depends_on.<parent>.restart: true` は **「依存先が再起動 (restart) された時」にのみ発火**し、**force-recreate (destroy + create) では発火しない**。これは [[000759]] で採用した仕様の盲点
- BFF は `ACOLYTE_CONNECT_MTLS_URL=https://acolyte-orchestrator:9443` を `MTLS_ENFORCE=true` で固定使用しており、plain :8090 への fallback を持たないため 502 が即時に FE まで伝播

## Decision

**孤立した pki-agent sidecar を手動で `docker rm -f` → `docker compose up -d --no-deps` により現 orchestrator の netns に再 attach し、復旧手順と限界の事実を ADR 化する。** 採用手順:

1. **即時復旧は `docker rm -f alt-pki-agent-acolyte-orchestrator-1` で孤立コンテナを削除**し、`docker compose -f compose/compose.yaml -p alt up -d --no-deps pki-agent-acolyte-orchestrator` で再作成する。全スタック down は不要かつ避ける（[[feedback_no_compose_down]]）
2. **検証は `docker inspect <sidecar> --format '{{.HostConfig.NetworkMode}}'` が現 orchestrator の container Id と一致することと、`docker exec <sidecar> ip -4 addr` に eth0 + 期待 IP が見えること**で確定する
3. **`depends_on.<parent>.restart: true` の恒久化は本 ADR の範囲外**とする。compose 仕様が force-recreate 時に cascade しない事実は今回の実測で確認済み。恒久対応は [[000783]] (alt-deploy 側で明示的に cascade 再作成) と [[000784]] (pki-agent 側で listener self-probe + fail-fast) の 2 経路で進める
4. **FE の 200 応答を以て ListReports 路の TCP + mTLS healthy を確認**する。認証未添付の 401 はゲートが正常動作している証左として受理する

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| `docker compose down && up -d` で全スタック再起動 | `feedback_no_compose_down` に抵触。他サービス（alt-backend, news-creator, rask-log-*）に再作成の副作用を拡散するリスクがあり、孤立 sidecar のみの問題に対して過剰 |
| BFF の `MTLS_ENFORCE=false` に一時切替 | mTLS カットオーバー ([[000759]] / [[PM-2026-031]]) の前提を逆流させる。運用の信頼境界を緩めるため、緊急時の逃げ道としてさえ採用しない |
| `depends_on.restart: true` を別の記述に書き換えて即解決を狙う | compose 仕様上、force-recreate で親が作り直された時点で既存 sidecar は古い container id を保持したままになる。記述変更では根治しない |
| 復旧だけ行い ADR を書かない | 3 度目の発火であり、事実の記録が無いと alt-deploy 側・pki-agent 側のどちらで根治すべきかの議論が再発する。今回を境に compose-native 解決は不可と明記する必要がある |
| BFF に plain :8090 への fallback を仕込む | セキュリティ境界を弱める。別 ADR で必要性を判断する段階にない（F3、YAGNI） |

### 範囲外 (今回やらないこと)

- alt-deploy pull ステップへの cascade recreate 追加 — [[000783]] で別 ADR
- pki-agent への listener self-probe + fail-fast — [[000784]] で別 ADR
- `latest_run_status=""` ハードコードバグ — 502 復旧後に表面化する別症状。別 ADR で TDD 修正
- PM-2026-031 の "恒久デプロイ待ち" ステータス解消 — 本 ADR は暫定復旧のみ。恒久化は [[000783]] / [[000784]] のいずれかが landed した時点で行う

## Consequences

### Pros

- **1 分以内に FE `/acolyte` の 502 が停止し、ユーザー動線が即時回復した**。復旧コマンドは 2 行の shell と 1 回の目視検証で済む
- **同一症状が 3 度目 ([[PM-2026-030]] → [[PM-2026-031]] → 本 ADR) に至った事実が公式記録になる**。compose-native 対応の限界が以降の設計議論で前提化され、alt-deploy 側 / pki-agent 側の根治策の優先度が上がる
- **ブラウザ DevTools Network の HTTP ステータスを起点に診断を進める手順が確立した**。`dial tcp ...:9443: connect: connection refused` は TLS handshake エラーとは区別される決定的な証拠で、netns 孤立の一意な signature として再利用できる
- **`docker inspect <sidecar> --format '{{.HostConfig.NetworkMode}}'` と親 Id の突き合わせが孤立判定の再現手順として固定化される**。SRE runbook や将来の自動検知スクリプトが同じ assertion を書ける

### Cons / Tradeoffs

- **恒久対応を残している**。次回 image タグ差分で acolyte-orchestrator が再作成されれば同じ症状が再発する。[[000783]] または [[000784]] が landed するまで運用負担が続く
- **手動復旧コマンドの認知負荷が残る**。オンコールが深夜に叩かれた場合、`docker compose down` の誤操作に走りやすい。runbook への転記が必要
- **pki-agent healthcheck が「cert 鮮度」以外を見ない事実は本 ADR では解消しない**。`tick ok, state: fresh` のまま機能が死ぬ silent failure モードは残る。[[000784]] の self-probe が landed するまでリスク継続
- **alt-butterfly-facade に fallback 経路がないため、単発の sidecar 孤立が即座に user-visible 502 になる**。F3 (BFF plain fallback) は別 ADR で必要性を再評価するが、本 ADR では採用しない
- **「ADR 書いて再発防止」のループが 3 周目に入った事実自体がシグナル**。compose spec 依存の対処では収束しないと読むべきで、次の ADR は実装を伴うものでなければならない
