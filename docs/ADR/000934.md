# ADR-000934: ClickHouse ログ保持を全テーブル 24h に揃え、TTL 非整合パーティションを是正する

## Status

Accepted

## Date

2026-05-31

## Affected Services

- clickhouse (`rask_logs` log/trace store) — `007_update_ttl_1day.sql` の取りこぼし是正、`entrypoint-wrapper.sh` で `logs` テーブルをガード付き再作成、`clickhouse/tests/retention_policy_test.sh` を追加

## Context

「ClickHouse がロギングをうまく保持できていない」という報告を起点に、ライブの `rask_logs` DB を read-only 調査した。意図する保持ポリシーは **「過去 24h を保持し、それより古いデータは削除」** だが、実態はテーブル間でバラバラだった。

| テーブル | パーティション | 実 TTL | 実データ範囲 | 判定 |
|---|---|---|---|---|
| `otel_logs` / `otel_traces` | `toDate(Timestamp)` | 1 day | 〜1.5日 | OK |
| `otel_http_requests` | `toDate(Timestamp)` | **7 day** | — | 逸脱 |
| `otel_error_logs` | `toDate(Timestamp)` | **14 day** | 05-17〜05-31 (15 日次パーティション, 約 197,000 行) | 逸脱 (最大) |
| `sli_metrics` | `toDate(Timestamp)` | 90 day | 派生 SLI, 約 3 MB | 意図的維持 |
| `logs` | **`(service_group, service_name)`** | 1 day | 0 行 (空・休眠) | 設計バグ |
| `http_logs` | `toYYYYMMDD(timestamp)` | 1 day | 0 行 (空・休眠) | OK |

根本原因は 2 つ。

1. **`007_update_ttl_1day.sql` の取りこぼし**: このマイグレーションは「全テーブルを 1 day に」と銘打つが、実際には `logs` / `http_logs` / `otel_logs` / `otel_traces` の 4 つしか ALTER しておらず、`006_create_otel_http_mv.sql` で作られる派生テーブル `otel_http_requests` (7 day) と `otel_error_logs` (14 day) を取りこぼしていた。これが over-retention の直接原因。
2. **`logs` テーブルの非時間軸パーティション**: `PARTITION BY (service_group, service_name)` は TTL 式 (`timestamp + 1 day`) と整合しない。`ttl_only_drop_parts = 1` 下では「パート内の全行が期限切れになったときだけ丸ごと DROP」するため、時間で分割されていないパーティションでは各パートに新旧の行が混在し続け、**TTL がパートを永久に DROP できない** (ClickHouse 公式の既知の落とし穴 / GitHub Issue #7435)。現状は空・休眠だが、アグリゲータの legacy `export_batch` 経路 (→ `logs`) が再活性化すれば「古いデータが消えない」landmine になる。

適用機構は確認済み: `clickhouse/entrypoint-wrapper.sh` がコンテナ起動毎に `/migrations/*.sql` を番号順・冪等 (`IF NOT EXISTS`) に再適用する。migrations と entrypoint は bind-mount のため、**`docker compose restart clickhouse` だけで再適用される (イメージ再ビルド不要)**。スキーマ追跡テーブルは無い。

公式 / Altinity KB のベストプラクティスも調査した: 短期保持の正攻法は **「日次パーティション + TTL を日数の倍数 (24h/48h/72h) + `ttl_only_drop_parts = 1`」**。`ALTER TABLE … MODIFY TTL` は `materialize_ttl_after_modify=1` (既定) で既存パートにも適用され、整合パーティション + drop-parts なら **古い日次パーティションを行の書き換えなしで丸ごと DROP** する。

なお `rask_logs` は disposable な観測データであり、knowledge-sovereign の append-first イベントログとは無関係。immutable データモデル不変条件には触れない。

## Decision

以下を採用する。

1. **`007_update_ttl_1day.sql` に取りこぼしを追補**し、`otel_http_requests` と `otel_error_logs` を 1 day TTL に揃える。007 が TTL ポリシーの正準的な置き場で、元々「全テーブル」を意図していたため、新規マイグレーションを足すより 007 の omission 修正が筋。単一の 007 編集で running DB (次回 restart 時) も fresh deploy (006→007 の順で最終 1 day) も両方カバーされる。

   ```sql
   ALTER TABLE otel_http_requests MODIFY TTL Timestamp + INTERVAL 1 DAY DELETE;
   ALTER TABLE otel_error_logs    MODIFY TTL Timestamp + INTERVAL 1 DAY DELETE;
   ```

2. **`logs` テーブルを `PARTITION BY toDate(timestamp)` で再作成**する。`PARTITION BY` は ALTER 不可なので再作成が必要で、`http_logs_mv` (`FROM logs`) は UUID 再採番で孤児化するため同時に再生成する。冪等な「毎回 DROP」を避けるため、`entrypoint-wrapper.sh` に `partition_key` を見る **ガード付き一回限り** の再作成を置く (`*service_group*` に一致するときのみ発火 → 修正後は `toDate(timestamp)` になり再発火しない)。

3. **`sli_metrics` は 90 day を維持**する。SLO のトレンド観測が目的で、分粒度の集計でサイズも極小。raw ログの 24h ポリシーとは別管理とする。

4. **保持ポリシー回帰テスト `clickhouse/tests/retention_policy_test.sh` を追加**する。`system.tables.engine_full` を検査し、log/trace 各テーブルが `toIntervalDay(1)`、`sli_metrics` が `toIntervalDay(90)` であることを表明し、逸脱で非ゼロ終了する (TDD: RED→GREEN、CI ガード)。

検討して却下した代替案:

- **新規マイグレーション (`012`) を足す**: 007 と同じ「TTL ポリシー更新」が 2 ファイルに分散する。007 の omission をその場で直す方が読み手に正直。却下。
- **`ttl_only_drop_parts = 0` (行レベル TTL) で厳密 24h**: マージ毎に部分書き換えが走り高コスト。公式は drop-parts + 整合パーティションを推奨。却下。
- **時間単位パーティションで厳密 24h**: パート数が激増しオーバーヘッド大。日次 + 1 day で「≥24h 保持」は満たせる。却下。
- **legacy `logs` / `http_logs` を DROP して廃止**: アグリゲータに dormant な `export_batch → logs` 経路が残るため、テーブル除去には Rust 側の改修も要る。空テーブルの再作成の方が局所的で安全。却下。
- **マイグレーションで `logs` を DROP+CREATE**: 起動毎に DROP が走る。`entrypoint` の partition_key ガードなら run-once かつ reboot-safe。却下。

## Consequences

### Pros

- 保持が全 log/trace テーブルで 24h に統一され、over-retention が解消 (`otel_error_logs` は 15 日次パーティション・約 197,000 行 → 2 日次・990 行に縮小、pending mutation なし)。ディスク使用が予測可能になる。
- `logs` のパーティションが時間軸に整合し (`toDate(timestamp)`)、「古いデータが消えない」設計バグが解消。レガシー経路が再活性化しても TTL が機能する。
- 回帰テストにより、将来 TTL を緩めるマイグレーションが入っても CI / 手元で逸脱を検知できる。
- イメージ再ビルド不要 (bind-mount + entrypoint 再適用) で、restart だけで反映される。

### Cons / Tradeoffs

- 実効保持は **24〜48h** (drop-parts は日次パーティション全体が >24h になって初めて削除するため)。常に ≥24h は保証するが「ちょうど 24h」ではない。
- `otel_error_logs` を 1 day にしたことで、エラーログ・例外・trace 相関の遡及調査が直近 24〜48h に制限される。長期のエラー傾向分析が要るなら別テーブル / 別保持の再設計が必要 (未解決の負債)。
- `logs` の再作成は破壊的操作 (空のため実害なし)。レガシー `export_batch` 経路が将来再活性化した状態で restart すると、最大 1 起動分の legacy ログを失う余地がある。
- 回帰テストはライブ ClickHouse への接続前提 (隔離テストではない)。CI で ClickHouse が起動していない環境では skip される設計が別途要る。
