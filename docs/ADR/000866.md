# ADR-000866: Knowledge Loop の Review-lane / 進行ステージ / Surface 司令卓 / Article act target を contract と projector の双方で整流化する

## Status

Accepted

## Date

2026-04-28

## Affected Services

- proto (Buf) — `KnowledgeLoopReviewedPayload` を `proto/alt/knowledge/loop/v1/knowledge_loop.proto` に追加。`KnowledgeLoopEntry` に `current_entry_stage` / `current_entry_stage_entered_at` (additive `optional`) を alt と sovereign の両 proto に追加
- knowledge-sovereign (Go) — `migrations/00016_add_knowledge_loop_entry_session_state.sql` 追加 (RLS 付き)、`projector.go` の `dismissStateForReviewedEvent` を payload `trigger` で分岐、`projectTransition` で `UpsertKnowledgeLoopEntrySessionState`、各 entry-mutating 経路の末尾に `recomputeSurfaces`、`seedActTargets` が article act target を必ず seed、`read_knowledge_loop.go` は `knowledge_loop_entries_public e LEFT JOIN knowledge_loop_entry_session_state s` に変更
- alt-backend (Go) — `domain/knowledge_loop_entry.go` に `CurrentEntryStage*` を追加、`connect/v2/knowledge_loop/handler.go` と `driver/sovereign_client/knowledge_loop_client.go` で mapping、`driver/sovereign_client/contract/knowledge_loop_event_consumer_test.go` に `knowledge_loop.reviewed.v1` の Pact 追加、`pacts/alt-backend-knowledge-sovereign.json` を再 publish
- alt-frontend-sv (Svelte 5) — `lib/hooks/useKnowledgeLoop.svelte.ts` が `bucketEntries` を `$state` として所有し `optimisticallyStaged` を overlay、`lib/components/knowledge-loop/LoopEntryTile.svelte` の `effectiveStage`、`lib/connect/knowledge_loop.ts` の proto mapping、`routes/(app)/loop/+page.svelte` の `bucketEntries = $derived(loop.bucketEntries)`、関連テスト
- e2e — `e2e/hurl/alt-backend/28-knowledge-loop-review-actions.hurl` 新設、`e2e/hurl/knowledge-sovereign/21-knowledge-loop-reviewed-surface-current-stage.hurl` 新設、`e2e/hurl/alt-backend/run.sh` に Review-lane 用 UUIDv7 変数 4 件を追加

## Context

Knowledge Loop は ADR-000831 で「Article / Summary / Tag / Recap / Augur / Acolyte / User Action → Knowledge Events → Now / Continue / Changed / Review surfaces → User acts → 行動が新たな知識として戻る」という循環を canonical contract として定義した。実装は phase を重ねて Surface Planner v2、Recap CTA、reproject admin、Augur handoff まで揃ったが、走査したところ **Loop の意味論が UI / projector / event payload の 3 層で噛み合っていない** 箇所がいくつか残っていた。

特に以下の 4 点は contract drift と意味的歪みが組み合わさっており、UI 上の体験劣化と projector の reproject-safety 違反の両方を引き起こしていた。

### 1. Review-lane の action 区別が暗黙の payload field に乗っていた

`alt-backend/app/usecase/knowledge_loop_usecase/classify_transition_event.go` は Review-lane (`Recheck` / `Archive` / `Mark Reviewed`) を `TransitionTrigger` 値で識別して同一の `KnowledgeLoopReviewed` event に集約していた一方、knowledge-sovereign の `dismissStateForReviewedEvent` は payload に `action` フィールドが入っていることを期待して読んでいた。実際には `buildTransitionEvent` は `action` を payload に書いておらず、結果として `Recheck` を含む全 Review action が default の `DISMISS_STATE_COMPLETED` に落ちていた。

```text
UI: Recheck 押下
  → action: "recheck" を期待
projector: payload["action"] を読む
  → 空 → default → COMPLETED
  → 本来 ACTIVE に戻すべき entry が完了扱いになる
```

しかも `KnowledgeLoopReviewedPayload` は proto 上に message 定義が無く、wire form は consumer 側 struct を読む人の解釈に依存していた。コード上で event を生成しているのは alt-backend のみという single-producer 状態が drift を覆い隠していた。

### 2. `proposedStage` が「カードの推薦段階」と「ユーザーの進行段階」の二役を兼任していた

`KnowledgeLoopEntry.proposed_stage` は ADR-000831 上「projector が推薦する次段階」だが、`useKnowledgeLoop.transitionTo()` はそれを `from_stage` として送出し、成功したら `entries[i].proposedStage` をローカルで書き換えていた。一方で sovereign 側の projector は `/loop` transition 受信時に `knowledge_loop_session_state` (lens 全体の最新段階) を更新するだけで、entry 行の `proposed_stage` は変更していない。

```text
UI: ローカルで proposedStage を進める (observe → orient → decide → act)
server: session_state だけ更新、entry 行は触らない
__data.json refetch: entry.proposedStage が projector 由来に巻き戻る
```

UI は「奥へ進む」体験を演出しているのに、サーバー往復のたびに段階表示が巻き戻るため、Review / Continue / Changed のいずれかに位置するカードの段階整合性が取れていなかった。

### 3. `useKnowledgeLoop` が `foregroundEntries` だけを `$state` として所有していた

`/loop/+page.svelte` の `bucketEntries` は `data.loop.bucketEntries` (load 関数の immutable snapshot) から derived しており、hook の state にはなかった。一方 hook の `applyLocalDismiss` は `entries` (foreground 由来) のみを filter しており、`ReviewDock` から `Archive` / `Mark Reviewed` を押しても review row は projector の bucket recompute と stream refresh が完了するまで画面に残り続けた。

### 4. `knowledge_loop_surfaces` が司令卓ではなく metadata になっていた

projector の `Repository` interface は `UpsertKnowledgeLoopSurface` を持っておらず、`GetKnowledgeLoopUsecase` は bucket ごとに entries を直接 list していた。`knowledge_loop_surfaces` は legacy fallback / quality 表示の場として存在するだけで、bucket の primary entry や `loop_health` が event 駆動で更新されない死んだ projection になっていた。canonical contract §10 が定める「Surface は各 bucket の司令卓」という性質を欠いていた。

加えて、`seedActTargets` は Recap snapshot がある場合のみ act target を生やしていた。article entry の `Open` 操作は `LoopEntryTile.svelte` の `sourceUrl()` が `act_targets` を覗いて URL を抜く実装だが、article 由来 entry に article act target が seed されていないため、`Open` の挙動が evidence ref の偶然に依存していた。

これらは canonical immutable invariants の観点では:
- §1 (append-first) — Reviewed event の payload contract が proto に無いため、producer 側の任意 field に依存する
- §2 (reproject-safe) — `proposed_stage` を UI で書き換えても event log には残らないので reproject で再現できない
- §4 (disposable projections) — `knowledge_loop_surfaces` が event から再構築できない

の 3 つに同時に触れていた。

## Decision

4 つを同一のコミットでまとめて整流化する。

### D-1. Review-lane の action 区別を `TransitionTrigger` 値に固定し proto contract を追加する

- proto に `KnowledgeLoopReviewedPayload` を追加 (`entry_key`, `lens_mode_id`, `from_stage`, `to_stage`, `trigger`, `observed_projection_revision`, `client_transition_id`)。`action` フィールドは作らない
- knowledge-sovereign 側 `dismissStateForReviewedEvent` を payload `trigger` (proto enum 名) で分岐するよう変更
  - `TRANSITION_TRIGGER_RECHECK` → `DISMISS_STATE_ACTIVE` (re-arm)
  - `TRANSITION_TRIGGER_ARCHIVE` / `TRANSITION_TRIGGER_MARK_REVIEWED` → `DISMISS_STATE_COMPLETED`
  - 不明 trigger / payload 破損 → fail-closed の `COMPLETED`
- alt-backend は payload に `action` を書かない (`TestTransition_ReviewActionPayloadUsesTriggerNotAction` で `payload["action"]` 不在を assert)
- Pact (`knowledge_loop.reviewed.v1`) を追加し alt-backend ↔ knowledge-sovereign の wire form を broker 側で明示

### D-2. `current_entry_stage` を per-user-per-entry projection として独立させる

- 新規 table `knowledge_loop_entry_session_state` (PK: `user_id, tenant_id, lens_mode_id, entry_key`、`current_stage` / `current_stage_entered_at` / `projection_seq_hiwater`、RLS 有効)
- `UpsertKnowledgeLoopEntrySessionState` driver メソッド追加。`current_stage_entered_at` は **必ず `ev.OccurredAt`** を渡す (no `time.Now()` for business facts)
- projector の `projectTransition` 末尾と `EventHomeItemOpened` 経路で entry session state を upsert
- `read_knowledge_loop.go` は `knowledge_loop_entries_public e LEFT JOIN knowledge_loop_entry_session_state s` に変更し、`current_entry_stage` / `current_entry_stage_entered_at` を proto に同梱
- 新フィールドは proto `optional`、domain は `*LoopStage` / `*time.Time`、TS は `currentEntryStage?` で additive
- FE は `effectiveStage = entry.currentEntryStage ?? entry.proposedStage` を `LoopEntryTile` の表示・ステージガード・transition `from_stage` に使用。`proposedStage` をローカル state として書き換える挙動を削除

### D-3. `useKnowledgeLoop` が `bucketEntries` も owned state として持つ

- `bucketEntries = $state([...opts.initial.bucketEntries])` を hook に追加
- `findEntry` は foreground と bucket の両方を走査
- `applyLocalDismiss` は両方から filter、`applyLocalStage` は両方の `currentEntryStage` を上書き
- `replaceSnapshot` は `optimisticallyDismissed` / `optimisticallyStaged` の overlay を foreground/bucket 両方に適用 (`applyOptimisticOverlays`)
- `/loop/+page.svelte` の `bucketEntries` derived を `loop.bucketEntries` に切替。`page.source.spec.ts` で wiring を guard

### D-4. Surface を event 駆動で recompute し article act target を seed する

- projector の `Repository` interface に `UpsertKnowledgeLoopSurface` / `GetKnowledgeLoopEntries` を追加
- `recomputeSurfaces(ev, lensModeID)` を実装し、`SUMMARY_VERSION_CREATED` / `HOME_ITEM_*` / `SUMMARY_SUPERSEDED` / `KnowledgeLoopReviewed` 等の entry を変更しうる経路の末尾で必ず呼ぶ
  - 4 bucket (`NOW` / `CONTINUE` / `CHANGED` / `REVIEW`) を順に走査
  - 各 bucket の active entries top 3 を取り、`primary_entry_key` (1 件) + `secondary_entry_keys` (最大 2 件) + `loop_health = {"active_count": n}` を `UpsertKnowledgeLoopSurface`
  - `freshness_at` は `ev.OccurredAt`、`projection_seq_hiwater` は `ev.EventSeq`
- `write_knowledge_loop.go` の surface upsert は `primary_entry_key = EXCLUDED.primary_entry_key` (CASE preserve を撤回)。「司令卓」として bucket が空になれば NULL を反映する
- `seedActTargets(ev, inputs)` を変更し、`article_id` (payload 直接 / `aggregate_type=article` の `aggregate_id` / `entry_key=article:*` prefix のいずれか) があれば必ず `{"target_type":"article","target_ref":<id>,"route":"/articles/<id>"}` を先頭に置く。Recap snapshot がある場合は従来通り Recap target を後続に追加

### 検討した代替案

- **payload に `action` を追加して projector はそれを読む** — 却下。`TransitionTrigger` 値が既に Review-lane の意味区別を担っているのに parallel field を増やすのは重複情報源を作ることになり、wire form の単一性を壊す。canonical contract §3.7 の single-emission rule とも整合しない
- **`proposed_stage` 自体を per-user に書き換える** — 却下。proposed_stage は projector の推薦値であり、ユーザーごとに異なる値を持つと「推薦」の意味が消える。reproject すると user-specific な書き換えが消えるため reproject-safety を壊す
- **`current_entry_stage` を `knowledge_loop_session_state` (lens 全体) に折り畳む** — 却下。lens の current_stage はカード横断の最新段階であり、entry 単位の進行と意味が異なる。両者を 1 column に詰めると Continue / Review の bucket 表示が崩れる
- **Surface を on-read で集約する** — 却下。read 側で集約すると `loop_health` が永続化されず、Stream / SSE 経由で「surface が変化した」イベントを下流に配れない (Phase 7 が想定する `KnowledgeLoopSurfacePlanRecomputed` の前提が崩れる)
- **Surface upsert で `primary_entry_key` の COALESCE を残す** — 却下。bucket が空になったときに古い primary が幽霊として残り続け、UI が「Now が空なのに primary が指している」状態を表示してしまう。司令卓としての意味を取るには NULL 反映が正
- **bucketEntries を hook ではなく Svelte store で持つ** — 却下。Svelte 5 の `$state` rune が hook 内に閉じれば hot module reload と test 隔離が確実。store にすると `useKnowledgeLoop.svelte.test.ts` の隔離が壊れる

## Consequences

### Pros

- **Review-lane が意図通り再 arm する**: Recheck で entry が `DISMISS_STATE_ACTIVE` に戻り、Surface recompute が直後に走るので Now/Continue/Review の primary が即更新される。Pact が wire form を pin しているため将来 producer 側が `action` field を再導入しても CI で fail する
- **proposed_stage と current_entry_stage が責務分離**: projector は推薦値を変えず、ユーザー進行は別 projection に記録される。reproject すると entry session も deterministic に再構築される (event log + ev.OccurredAt のみから決まる)
- **Surface が event 駆動の司令卓に戻った**: `knowledge_loop_surfaces` の primary / secondary / loop_health が entry-mutating event のたびに deterministic に再計算され、`reproject-safe` invariant §2 を満たす
- **Article entry の `Open` が確定動作**: 全 article 由来 entry に明示的な article act target が seed され、`/articles/<id>` 経路で UI が動作する。evidence ref の偶然に依存しない
- **bucketEntries の optimistic UI が即座に反応**: `applyLocalDismiss` / `applyOptimisticOverlays` が foreground と bucket 両方に作用するため、Archive / Mark Reviewed / Recheck がストリーム refresh を待たずに UI に反映される
- **新フィールドは additive**: proto `optional`、TS `?`、Go `*` で wire compat を保ち、旧 client は新 field を無視できる

### Cons / Tradeoffs

- **per-event surface recompute のコスト**: 4 bucket × `Limit 3` の SELECT + `UpsertKnowledgeLoopSurface` × 4 が entry-mutating event 1 件ごとに走る。現行スループット (event/sec が 1 桁台) では問題ないが、burst 時に latency が増える可能性は計測で監視する必要がある (`projector_run_batch_duration_seconds` を引き続き観測)
- **新規 table の RLS 設定が必要**: `knowledge_loop_entry_session_state_user_isolation` policy は他テーブルと同じ pattern だが、新 driver path 追加時に `SET LOCAL alt.user_id` を漏らすと silently empty を返す。今後 service-to-service 経路を増やす際の踏み抜きポイント
- **migration 追加で `atlas migrate hash` 必須**: 本 ADR の commit に含めるが、hash がズレた状態で deploy すると Atlas が drift と判定して止まる
- **Surface primary_entry_key の semantics 変更は backward-incompat**: CASE preserve を撤回したので「古い primary を見つづけたい」consumer がいた場合は影響を受ける。実際の consumer は Surface を screen 描画にしか使っていないため影響なしと判断したが、新 consumer が増える際は要注意
- **legacy `action` payload を含む過去 event は default fail-closed の `COMPLETED` に落ちる**: 本 ADR commit 以前に `action` を payload に書いた event が存在しないことを確認済 (alt-backend が producer であり、過去 commit の `buildTransitionEvent` も payload に `action` を書いていない)。historical replay でも安全
- **`Acolyte` / `Report` を Loop に接続するところまでは進めていない**: act target type の `REPORT` 拡張、`acolyte.report_*` event は別 ADR で扱う。現状の Loop は依然として「Ask Augur まで」
