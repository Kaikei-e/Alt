# ADR-000612: HybridPrioritySemaphore の CancelledError 時スロット回収と invariant check の false positive 排除

## Status

Accepted

## Date

2026-03-31

## Affected Services

- news-creator (Python)

## Context

[[000610]] で `call_soon_threadsafe` の race condition を排除しスロットリークを修正したが、修正デプロイ後も **86件の SLOT INVARIANT VIOLATION** と**最大 551秒（9分超）のキュー待ち**が発生していることがログ調査で判明した。特にオンザフライ・ストリーミング要約（StreamSummarize）の RT リクエストが 30〜75秒待ちとなり、ユーザー体感として著しい劣化を引き起こしていた。

### 影響

- BE リクエストのキュー待ちが最大 **551.59秒**（通常 0秒）
- RT ストリーミングリクエストのキュー待ちが **31〜75秒**（通常 0秒）
- `SLOT INVARIANT VIOLATION` が ERROR レベルで **86回** ログ出力（可観測性のノイズ）
- `COLD_START` 警告が **183回** 発生（minor: 0.12〜0.23秒、モデル不一致ではない）

### 根本原因

`release()` がスロットを `_try_wake_waiter()` 経由で次の waiter に転送した後、その waiter の asyncio タスクがキャンセルされた場合に、スロットが永久に消失するバグが存在した。

**リーク経路:**

1. `release()`: `_track_release(slot_id)` で `_acquired_slots` からスロットを削除
2. `release()`: `_try_wake_waiter()` → `future.set_result(home_pool)` でスロットを waiter に転送
3. `release()`: `woke_up=True` のためプールカウンタは**非加算**で return
4. `acquire()`: waiter のタスクが HTTP クライアント切断等でキャンセル → `CancelledError` 発生
5. `acquire()` の `CancelledError` ハンドラ: `future.done()=True` なので future はキャンセルされない
6. **`_track_acquire()` が呼ばれずスロット消失** — `_acquired_slots` にも pool counter にもない

このバグは [[000610]] の `_try_wake_waiter` 修正とは異なるリーク経路であり、`_try_wake_waiter` 自体は正しく動作していた。問題は `acquire()` 側の `CancelledError` ハンドラが転送済みスロットの回収を行っていないことにあった。

**副次的問題:** `release()` 末尾の invariant check がスロット「転送中」の一時的状態を検出して ERROR ログを出力していた（false positive）。スロットは `_track_release()` で `_acquired_slots` から削除された後、`_track_acquire()` で再追跡されるまでの間「転送中」になるが、invariant check はこの transient window を考慮していなかった。

## Decision

### 1. `acquire()` CancelledError ハンドラでスロット回収を追加

`CancelledError` 発生時に `future.done() and not future.cancelled()` を検査し、スロットが転送済み（`set_result()` 呼び出し済み）であればそのスロットを回収する。回収は `_release_slot_to_pool_or_waiter()` ヘルパーを通じて行い、次の waiter がいれば転送し、いなければ home pool に返す。

```python
except asyncio.CancelledError:
    if future.done() and not future.cancelled():
        transferred_pool = future.result()
        home_pool = transferred_pool if isinstance(transferred_pool, str) and transferred_pool else ("rt" if high_priority else "be")
        self._release_slot_to_pool_or_waiter(home_pool)
    elif not future.done():
        future.cancel()
    async with self._lock:
        self._purge_cancelled_from_queues()
    raise
```

### 2. `_release_slot_to_pool_or_waiter()` ヘルパーの追加

スロットを RT/BE キューの次の waiter に転送するか、waiter がいなければ home pool に返す共通ロジック。`release()` の guaranteed bandwidth ロジックは含まず、シンプルな RT 優先 → BE フォールバックの転送パスのみを提供する。

### 3. invariant check の false positive 対策

`release()` 末尾の invariant check で、`woke_up=True`（スロットが waiter に転送済み）の場合は `expected_total` を 1 減算し、「転送中」状態を正常とみなす。これにより false positive の ERROR ログが排除される。

## Consequences

### Pros

- **スロットリーク解消**: HTTP クライアント切断等による waiter タスクキャンセル時のスロット消失が防止される
- **false positive 排除**: `SLOT INVARIANT VIOLATION` の ERROR ログが正常運用時に出力されなくなり、真のスロットリークのみを検知できる
- **RT レイテンシ改善**: スロットリークによる RT キュー待ちが解消され、オンザフライ要約のレスポンスが改善される

### Cons / Tradeoffs

- **`_release_slot_to_pool_or_waiter()` は guaranteed bandwidth 非対応**: `release()` の `force_be` パスとは独立した簡易パスであり、CancelledError 回収時の guaranteed bandwidth は適用されない。回収は緊急パスであり、この制約は許容できる
- **`release(slot_id=...)` の全 caller 必須化は未完了**: [[000610]] P-5 と同様、legacy fallback はまだ残存
