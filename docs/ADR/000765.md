# ADR-000765: knowledge-sovereign に Hurl E2E スイートを追加し段階的全面展開の第 3 サービスとする

## Status

Accepted

## Date

2026-04-18

## Affected Services

- knowledge-sovereign (Go) — プロダクションコードは一切触らず、E2E スイート (18 Hurl シナリオ + readiness) と staging compose 片肺 (knowledge-sovereign-db + migrator + service) を追加する
- knowledge-sovereign-db — knowledge-sovereign 単独起動のための依存として `compose/compose.staging.yaml` に postgres:16-alpine + Atlas migrator 2 サービスを追加する (ephemeral, tmpfs snapshot/archive dir)

## Context

[[000763]] で Hurl E2E フレームワークを Increment 1 として `search-indexer` に導入、[[000764]] で Increment 2 として `mq-hub` を加えた。これらは **HTTP handler だけ** / **Connect-RPC + Redis Streams** の 2 パターンだったが、Alt の durable knowledge state の single owner である `knowledge-sovereign` ([[000532]]) は以下の点でサーフェスが広い:

1. **2 種類の JSON コーデックが同居する**。`services.sovereign.v1.KnowledgeSovereignService` の 32 RPC は Connect-RPC + proto3-JSON (camelCase、int64 文字列化、空 repeated 省略) で返る。一方 `/admin/snapshots/*` / `/admin/retention/*` / `/admin/storage/stats` は `sovereign_db.SnapshotMetadata` 等を `encoding/json` で直接マーシャルするため **PascalCase 無タグ出力**になる。同一スイートで両方を検証するパターンを確立しないと、後続で admin surface を持つサービス (`altctl`, 将来の Sovereign 派生) が type 契約を誤読する。
2. **DB-backed state が test ordering を強制する**。`knowledge_current_lens.lens_version_id` は `knowledge_lens_versions` への FK を持ち、`knowledge_events.event_seq` は BIGSERIAL で単調増加、`knowledge_event_dedupes.dedupe_key` は globally UNIQUE。`CreateLens → CreateLensVersion → SelectCurrentLens → ListLenses` / `AppendKnowledgeEvent → GetLatestEventSeq → CreateSnapshot` という直列依存が発生する。Hurl 7.1 は `--test` 時にファイル単位で **並列実行がデフォルト**のため、そのままでは FK / dedupe 違反で間歇失敗する。mq-hub スイートは stateless な Connect-RPC のみだったためこの問題が顕在化していなかった。
3. **distroless image の素朴さ**。`gcr.io/distroless/static-debian12:nonroot` ベースの sovereign image は `/bin/sh` を busybox から 1 バイナリだけ取り込んでいて **`cat` が存在しない**。`compose/sovereign.yaml` (本番) が使っている `POSTGRES_PASSWORD_FILE=/run/secrets/postgres_password` + `DB_PASS=$(cat /run/secrets/postgres_password)` パターンを staging でそのまま使うと `sh: cat: not found` で DB 接続が立たない。
4. **admin surface が snapshot/retention で実ディスクを叩く**。`SNAPSHOT_DIR=/tmp/snapshots` は Dockerfile で `RUN mkdir -p /tmp/snapshots` されているが root:root 755 のため、`USER nonroot:nonroot` が走るランタイムで書き込めない。本番は `../backups/sovereign-snapshots` bind mount で解決しているが、staging では都度空を期待する。
5. **テナント境界 ([[000749]]) は Phase 1 の観測範囲外**。`ListKnowledgeEvents` は `tenant_id` mismatch を reject するが、cross-tenant negative path を Phase 1 に含めると fixture UUID 管理が複雑化し、happy path 緑化が後ろ倒しになる。

## Decision

**knowledge-sovereign に対する Hurl E2E スイートを readiness + 17 シナリオ の計 18 ファイルで追加する。Connect-RPC (proto3-JSON camelCase) と admin HTTP (Go encoding/json PascalCase) を同一 runner で駆動し、`--jobs 1` で強制直列化する。negative-path は Phase 2 (別 PR) に繰り延べる。**

### 採用する構成

1. **staging stack 追加**: `compose/compose.staging.yaml` に `knowledge-sovereign` profile を新設し以下 3 サービスを足す:
   - `knowledge-sovereign-db` — `postgres:16-alpine`、`POSTGRES_PASSWORD` を env に平文 inline (staging 専用コメント + `README.md` で注記)
   - `knowledge-sovereign-db-migrator` — `build: ../knowledge-sovereign docker/Dockerfile.migrator`、Atlas apply 1 発で exit (`restart: "no"`)
   - `knowledge-sovereign` — `build: ../knowledge-sovereign Dockerfile.sovereign`、tmpfs `/tmp/snapshots:mode=1777` + `/tmp/archives:mode=1777` で distroless nonroot の書き込み問題を回避
   - 公開ポート `19510:9500` (RPC) / `19511:9501` (admin + health) は documentation 目的で、Hurl は network 内から直接 service DNS を引く

2. **scenario 構成** (`e2e/hurl/knowledge-sovereign/NN-<feature>.hurl` + `README.md` narrative):

   | # | Hurl 駆動対象 | 契約 |
   |---|---|---|
   | 00 | `/health` on `:9500` readiness (retry 30 × 500ms) | 200 + `status=ok` + `service=knowledge-sovereign` |
   | 01 | `/health` on `:9501` (metrics port) | 同スキーマ |
   | 02 | Connect-RPC `GetActiveProjectionVersion` | `version.version=1` + `status=active` (migration 00001 seed) |
   | 03 | `AppendKnowledgeEvent` happy | `eventSeq` が proto3-JSON 文字列化 int64、capture |
   | 04 | `GetLatestEventSeq` | `eventSeq` が 03 の値と一致、tenant_id 必須 (ADR-000749) |
   | 05 | `ListKnowledgeEvents` tenant-scoped | 03 の event が 1 件、`eventId`/`eventType`/`dedupeKey` 一致 |
   | 06 | `GetKnowledgeHomeItems` fresh user | 空 projection (proto3-JSON が `{}` に縮退) |
   | 07 | `GetTodayDigest` fresh user | `digest` 省略 |
   | 08 | `GetRecallCandidates` fresh user | `candidates` 省略 |
   | 09 | `CreateLens` | client-assigned `lensId`、empty response |
   | 10 | `CreateLensVersion` | FK 前提で lens 作成後に呼ぶ |
   | 11 | `ListLenses` | 作成した lens 1 件、`lensId`/`userId`/`tenantId` 一致 |
   | 12 | `SelectCurrentLens` → `GetCurrentLensSelection` | 同一ファイル内で round-trip、`found=true` |
   | 13 | `AppendRecallSignal` | client-assigned `signalId`、empty response |
   | 14 | `POST /admin/snapshots/create` | PascalCase response、`EventSeqBoundary > 0`、3 種 `sha256:...` checksum |
   | 15 | `GET /admin/snapshots/latest` | 14 と同 snapshot |
   | 16 | `GET /admin/retention/eligible` | 2 要素配列 (`knowledge_events` + `knowledge_user_events`) |
   | 17 | `POST /admin/retention/run` `{dry_run: true}` | `dry_run=true` + `error` 不在 |
   | 18 | `GET /admin/storage/stats` | JSON collection |

3. **proto3-JSON vs encoding/json の切り分け**: README に明記する:
   - Connect-RPC 経由 (`:9500`) は camelCase、int64 は string、空 repeated / default scalar は省略 (`{}` に縮退する)
   - admin HTTP 経由 (`:9501`) は `sovereign_db.SnapshotMetadata` 等が無タグ struct のため PascalCase (`SnapshotID` / `EventSeqBoundary` / `ItemsChecksum`)
   - `retention_handler.go` の応答型のみ明示 json tag (`dry_run`, `actions`) を持つので snake_case

4. **run-scoped UUID 生成**: `run.sh` が `/proc/sys/kernel/random/uuid` を 6 回読んで `tenant_id` / `user_id` / `event_id` / `lens_id` / `lens_version_id` / `signal_id` を生成し `--variable` で注入する。Hurl は `file,…;` body を template 展開しないため、scenario 側は JSON body を **inline** に記述し `{{var}}` で埋める。`dedupe_key` は `hurl-e2e-{{run_id}}-ev1` で global UNIQUE を担保。

5. **強制直列化**: `run.sh` は `--jobs 1` を明示。Hurl 7.1.0 の `--test` は parallel default (`--parallel` フラグ有効) であり、FK/event_seq/dedupe の順序依存を壊す。`--jobs 0` は `clap` バリデーションで reject されるため `1` を使う。

6. **admin endpoint の扱い**: `:9501` admin surface は Phase 1 では positive path のみ検証する。認証は未実装だが `compose.staging.yaml` の `internal: true` network + Phase 1 では `19511:9501` publish で短時間露出するのみ。運用項目 (ADR の Cons 参照) として記録する。

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| production の `compose/sovereign.yaml` と同じく docker secret + `cat /run/secrets/...` で DB 接続 | distroless sovereign image に `cat` が無く `{"level":"ERROR","msg":"database ping failed","error":"... password authentication failed ..."}` で落ちる。`read p < /run/secrets/postgres_password` は busybox sh でも動くが、staging の ephemeral 性を考えれば env 平文 + README 注記のほうが透明性が高い |
| `e2e/fixtures/knowledge-sovereign/*.json` に body fixture を置き `file,…;` で読む | Hurl は fixture body の template 展開をサポートしない。run-scoped UUID を fixture に埋めるには `gen-*.py` 類の前処理が必要になり、mq-hub の 1001 件 batch ような正当理由が無い限り保守コストが上回る |
| Phase 1 から cross-tenant reject / invalid payload を含める | 18 scenario で既に event_seq + lens FK 2 系統の state machine を持ち、さらに negative UUID を足すと fixture 管理が複雑化する。positive path を安定させてから Phase 2 で追加するほうが review diff が読みやすい |
| `--parallel` のまま `@serial` 相当のアノテーションを Hurl 側で工夫 | Hurl 7.1 には scenario 間依存を宣言する仕組みがない。`--jobs 1` で統一するほうが `run.sh` の意図が明確 |
| snapshot 応答の PascalCase を是正する json tag を sovereign 本体に入れる | ADR は E2E 追加で production code を触らない方針。実在する応答形状を契約として固定し、将来 json tag を入れる判断は別 ADR に委ねる |
| `e2e/fixtures/staging-secrets/postgres_password.txt` を追加して secret として mount | 上記のとおり sovereign image で読めない。README を `meili_master_key.txt` 1 本に揃え、「distroless サービスは env 直接注入」を方針として明記する |

### 範囲外 (Phase 2 以降で別 PR)

- `ListKnowledgeEvents` cross-tenant reject ([[000749]] negative path)
- `AppendKnowledgeEvent` invalid payload (missing required field → `invalid_argument`)
- `ArchiveLens` → `ListLenses` から消えることの確認
- `POST /admin/retention/run` without valid snapshot → 500 + `error` 含む
- `AreArticlesVisibleInLens` bulk visibility check
- `WatchProjectorEvents` server-streaming (Hurl の streaming 対応は限定的。将来 buf conformance 側)
- JWT 認証チェーン (staging bypass 前提、gateway 層 E2E が別途必要)
- projector 経由の end-to-end 反映 (= mq-hub → projector → sovereign のマルチサービス E2E)

## Consequences

### Pros

- **proto3-JSON と encoding/json が同一サービス内に共存するパターンが契約化された**。後続で admin surface を持つ Go サービス (`altctl`, recap 系) が同じ構造で E2E を書ける
- **DB-backed Hurl スイートの記法が確立した**。`--jobs 1` 強制直列化、`/proc/sys/kernel/random/uuid` による run-scoped UUID 生成、dedupe_key/FK の ordering、inline JSON body での `{{var}}` 展開。Postgres-backed な他サービス (`alt-backend`, `auth-hub`) に横展開可能
- **distroless + staging secret の摩擦が記録された**。README と ADR の両方で明文化したので、同じ image ベースのサービスが staging compose を作る時に 30 分級のハマりを回避できる
- **admin surface の構造契約が回帰網に入った**。`SnapshotID` が UUIDv4、`EventSeqBoundary > 0`、3 種 checksum が `sha256:` 形式などの不変条件を CI で弾ける。特に `EventSeqBoundary` は IMPL_BASE.md の append-first 不変条件の実効確認
- **Phase 分割で review コストが下がった**。negative path を別 PR に回したことで、今回の 18 シナリオは happy path のみに focus し、実装変更を伴わない review が短時間で済む

### Cons / Tradeoffs

- **staging stack のサービス数が増え続ける**。search-indexer 系 3、mq-hub 系 2、knowledge-sovereign 系 3 の合計 8 container。`compose.staging.yaml` が 265 行超。Phase 4 相当でサービス単位の split compose (`compose.staging.sovereign.yaml` など) に切る判断が近々必要
- **admin HTTP が無認証のままホスト 0.0.0.0 に露出している**。`internal: true` network が container 間通信を囲うが、Compose の `ports:` は `127.0.0.1` を明示しない限り全 NIC に publish する。staging 走行中の 10〜30 秒は社内 LAN から到達可能。mitigation として `ports: - "127.0.0.1:19511:9501"` への格上げと、sovereign 本体への admin auth 追加を Phase 2 の運用フォロー項目とする (security audit 結果は運用チケットに残す)
- **tenant boundary のリグレッション検知が Phase 2 まで走らない**。[[000749]] が要求する cross-tenant reject は Phase 1 スイートでは検証されない。Phase 2 scenario 19-21 をマージするまでは go unit test (`handler/rpc_infra_test.go`) と ADR-000749 review に依存する
- **proto3-JSON の "空 repeated 省略" を `not exists` で受けている**。将来 `protojson.MarshalOptions{EmitUnpopulated: true}` が入った場合、06/07/08 が緑のままスキーマが変わるリスク。契約変更としては検出されるべきなので、この assertion は「意図的に厳しくない」ことを README に明記した
- **Hurl の parallel default と `--jobs 1` の乖離がローカル ad-hoc 実行で混乱を生む**。個別 `.hurl` を IDE からキックすると parallel で走って FK 違反する。00-setup.hurl 先頭コメントと README で注意喚起しているが、新規メンバーが踏む可能性は残る
- **image tag が digest pin ではない**。`postgres:16-alpine` / `ghcr.io/orange-opensource/hurl:7.1.0` は [[000763]] 監査項目 #10 と同じ負債を踏襲。Renovate が入るまで緩い
