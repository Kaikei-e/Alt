# ADR-000459: Knowledge Home reproject を実際の event replay と永続 checkpoint で再構築する

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-backend (Go)

## Context

Knowledge Home の dismiss 再表示不具合を調査した結果、`HomeItemDismissed` を regular projector が処理していなかった期間のイベントが read model に反映されないまま checkpoint だけが進んでいた。これにより、過去の dismiss は source of truth である `knowledge_events` に残っていても、`knowledge_home_items` には復元されなかった。

同時に、`knowledge_reproject_job` は名前に反して実際の projection 更新を行っておらず、event 件数と統計値を進めるだけの実装だった。`checkpoint_payload` も永続化されていなかったため、途中再開や historical recovery の根拠として使えなかった。

この状態では、reproject を実行しても次の問題を解決できない。

- 過去に取り逃がした `HomeItemDismissed` を replay して hidden state を回復すること
- projector 実装変更後に target version の read model を再構築すること
- 途中中断した reproject を deterministic に再開すること

## Decision

Knowledge Home の reproject は、統計更新ジョブではなく regular projector と同等の event replay として実装する。

具体的には以下を採用する。

- `knowledge_reproject_job` は event を走査しながら `projectEvent(...)` を呼び、`HomeItemDismissed` を含む Knowledge Home 関連イベントを target projection version に再適用する
- reproject run の `to_version` を明示的に解釈し、その version 向けに read model を構築する
- batch ごとに `checkpoint_payload` と統計を `knowledge_reproject_runs` に保存し、途中停止後も同じ位置から再開可能にする
- regular projector と reproject の write contract を揃え、dismiss・upsert・supersede 解消がどちらでも同じ意味を持つようにする

これにより、reproject は「古い projector 実装で落とした状態を、event log から再計算して回復する仕組み」として成立する。

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|--------|------|-----------|
| regular projector の checkpoint を巻き戻して再実行する | 実装は少ない | 本番 read model を直接揺らしやすく、version 比較や段階的切替と相性が悪い |
| 過去の dismiss だけを SQL で埋め戻す | 即効性はある | source of truth からの再計算にならず、他の event type との整合性を保証できない |
| reproject を統計専用のまま維持する | 単純 | historical recovery の役割を果たせず、今回の障害を再発防止できない |

## Consequences

### Pros

- 過去に取り逃がした `HomeItemDismissed` を event log から回復できる
- reproject が projection version 切替のための実体ある仕組みになる
- `checkpoint_payload` が永続化され、長時間 replay の途中再開が可能になる
- regular projector と reproject の意味論が揃い、検証と障害調査が容易になる

### Cons / Tradeoffs

- reproject 実行時の DB 書き込み量が増え、統計更新だけの実装より重くなる
- projector と reproject が同じ event semantics に依存するため、event 分岐の不備が両者に影響する
- target version を持つ設計に整えない限り、replay 結果の比較や切替が不完全になる
