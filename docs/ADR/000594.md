# ADR-000594: Knowledge Home の E2E テストを Playwright で追加する

## Status

Accepted

## Date

2026-03-26

## Affected Services

- alt-frontend-sv (SvelteKit) — E2E テスト追加対象

## Context

[[000593]] で Knowledge Home の要約投影エラー伝播修正と Recall タグクリックトラッキングを実装した。しかし Knowledge Home 本体の E2E テストが存在せず、以下の主要導線が自動検証されていなかった。

1. 要約の表示状態（`summary_state=ready` で要約表示、`pending` で Summarizing チップ表示）
2. タグクリック時のトラッキング発火
3. Recall rail の reason badge・Why パネルの表示（`tag_interaction` を含む）
4. Recall カードクリック時の open トラッキング発火

是正指示書（`review/knowledge-home-phase1-5-remediation-directives-2026-03-18.md` P1-3）でも E2E シナリオ追加が要求されていた。

## Decision

Playwright を用いた Knowledge Home E2E テストを Page Object Model パターンで追加する。

### テスト設計方針

- **既存パターンの踏襲**: `fulfillJson` によるルートモック、`BasePage` 継承の POM、factory によるテストデータ生成
- **`waitForTimeout` 不使用**: Playwright ベストプラクティスに従い、`page.waitForRequest()` でリクエスト完了を待機し、web-first assertions（`toBeVisible`, `toContainText`）で自動待機
- **Connect-RPC パス形式**: `/api/v2/alt.knowledge_home.v1.KnowledgeHomeService/*` で BFF プロキシ経由のエンドポイントをモック

### テストケース（8 件）

| カテゴリ | テスト | 検証内容 |
|----------|--------|----------|
| Summary Display | ready 状態で要約表示 | `summary_state=ready` のカードに要約テキストが表示され、Summarizing チップが非表示 |
| Summary Display | pending 状態で Summarizing チップ | `summary_state=pending` のカードにチップとスケルトンが表示 |
| Summary Display | refresh で pending→ready 遷移 | レスポンスを差し替えてページ再読み込み後、要約テキストに切り替わる |
| Tag Click Tracking | タグクリックで tracking 発火 | タグ `<a>` クリックで `TrackHomeAction` に `actionType=tag_click` が送信される |
| Tag Click Tracking | Open で tracking 発火 | Open ボタンクリックで `actionType=open` が送信される |
| Recall Why Display | reason badge 表示 | Recall rail に candidate タイトルと `Not revisited` バッジが表示 |
| Recall Why Display | Why パネルに tag_interaction | Why ボタンクリックで `Tag explored` ラベルが表示 |
| Recall Why Display | Recall カードクリックで tracking | candidate クリックで `actionType=open` が送信される |

### ファイル構成

```
tests/e2e/
  fixtures/factories/knowledgeHomeFactory.ts   # テストデータファクトリ
  pages/desktop/DesktopKnowledgeHomePage.ts    # Page Object Model
  desktop/knowledge-home/index.spec.ts         # テストスペック
```

### トラッキング検証パターン

`waitForTimeout` ではなく `page.waitForRequest()` を使用し、クリック前にリクエスト promise を設定してからアクションを実行する。

```typescript
// ✅ Playwright ベストプラクティス: waitForRequest
const trackRequestPromise = page.waitForRequest(
  (req) =>
    req.url().includes("TrackHomeAction") &&
    req.postDataJSON()?.actionType === "tag_click",
);
await rustTag.click();
const trackRequest = await trackRequestPromise;
expect(trackRequest.postDataJSON().itemKey).toBe(expectedItemKey);

// ❌ アンチパターン: waitForTimeout
await rustTag.click();
await page.waitForTimeout(500);  // flaky, 非決定的
```

## Consequences

### Pros

- **主要導線の自動検証**: 要約表示、タグクリックトラッキング、Recall Why 表示が CI で継続的に検証される
- **リグレッション防止**: [[000593]] で追加した `tag_click` トラッキングと `tag_interaction` reason 表示が壊れた際に即座に検知可能
- **Playwright ベストプラクティス準拠**: `waitForTimeout` を使わず、auto-waiting と `waitForRequest` で安定したテスト

### Cons / Tradeoffs

- **モックベース E2E**: 実際の alt-backend / knowledge-sovereign を経由しないため、バックエンドの contract 変更を検知できない。Pact CDC テスト（[[000589]]）が contract 検証を担う
- **Connect-RPC レスポンス形式の依存**: `fulfillJson` でモックしているが、実際の Connect-RPC はバイナリエンベロープ形式。テスト側が `fulfillJson` で動作するのは SvelteKit プロキシが JSON 形式に変換しているため
- **テストデータの手動管理**: factory のレスポンス構造は proto 定義と手動で同期する必要がある
