# ADR-000819: tag-generator の CPU-only pin / 事前ビルドモデル / image-size gate を news-creator 系と揃える

## Status

Accepted

## Date

2026-04-20

## Affected Services

- tag-generator (Python)
- news-creator (Python)
- recap-subworker (Python)
- docker-build.yaml (CI)

## Context

tag-generator の image サイズは **約 2.8 GiB 前後** と同系統の Python/ML サービスに比べて重く、以下が積み上がっていた。

1. `pyproject.toml` に `[tool.uv.sources]` による CPU-only PyTorch pin が無い。Dockerfile 側で `uv pip install --index-url ...` を直叩きしており、`uv.lock` は CUDA 系 wheel (`nvidia-cublas-cu12`, `nvidia-cudnn-cu12`, `triton` など 17 パッケージ) と `sentence-transformers v5.2.3` / `transformers v5.2.0` を引き込んでいた。これは [[000762]] と [[000812]] で他サービスに適用済みの正規パターンに追随できていない状態。
2. BuildKit の `--mount=type=cache` に `id=` が無く、他サービス (news-creator / recap-subworker / recap-evaluator) の uv cache と同一ネームスペースで衝突する余地があった。
3. post-install slim (vendored `tests/` / `torch/include` / `*.pyi` 削除) が未実装。news-creator は `tests/` のみ、recap-subworker も同様で、`torch/include` と `*.pyi` まで刈っているのは recap-evaluator だけ。
4. `.github/workflows/docker-build.yaml` の image-size + smoke ゲート対象から tag-generator が外れており、CUDA wheel の再混入 / サイズ退行を CI で止める術が無かった ([[000813]] の whitelist に未加入)。
5. `.github/workflows/tag-generator.yaml` は `python-version: '3.13'` だが `pyproject.toml` は `requires-python = ">=3.14"`、Dockerfile は `python:3.14-slim` と三重で不整合 ([[000812]] 違反)。
6. `tag-generator/.dockerignore` の `.venv/` は top-level にしかマッチせず、ローカル開発由来の `app/.venv/` (~2.4 GiB) が build context に素通しされていた。最初のローカルリビルドで image が **4245 MiB** まで膨らんでいたのはこれが主因。

また、security-auditor レビューで:

- `ghcr.io/astral-sh/uv:0.11.0` が **mutable tag** 参照 (CWE-494)。
- HEALTHCHECK が `curl` 依存で、`curl` を runtime image に残す必要があった。
- CI smoke-check の `docker run` が egress とケーパビリティ無制限で実行されていた。

の 3 点を追加ハードニング候補として指摘。

目標: **tag-generator image ≤ 2200 MiB**、他 Python サービスには同水準の slim を横展開、供給網攻撃面を縮小。[[000813]] の「まず余裕を持たせて CI に gate を敷き、後続 PR で tighten」方針を踏襲する。

## Decision

採用した設計:

### 1. tag-generator の CPU-only pin を `pyproject.toml` に移す

```toml
[tool.uv.sources]
torch = [{ index = "pytorch-cpu" }]
torchvision = [{ index = "pytorch-cpu" }]
torchaudio = [{ index = "pytorch-cpu" }]

[[tool.uv.index]]
name = "pytorch-cpu"
url = "https://download.pytorch.org/whl/cpu"
explicit = true
```

`ml` dependency group に `sentence-transformers>=3.3.0,<5` と `transformers>=4.40.0,<5` の上限を入れ、v5 silent bump を塞ぐ。`uv lock` 再生成で CUDA 系 17 パッケージと triton が除去され、torch は `2.11.0+cpu` に差し変わった。

### 2. Dockerfile を news-creator パターンに統一

- `uv venv` + `uv sync --frozen --no-dev --no-install-project --no-editable --group ml` → source copy → `uv sync --frozen --no-dev --no-editable --group ml` の 2 段
- 全 BuildKit cache mount に `id=uv-tag-generator` / `id=hf-tag-generator` / `id=nltk-tag-generator`
- post-install slim: `recap-evaluator/Dockerfile:36-44` パターンに `torch/include` + `*.pyi` + `__pycache__` + `*.pyc` を重ねる
- SBERT (`paraphrase-multilingual-MiniLM-L12-v2`) と NLTK (`punkt_tab`, `stopwords`) は [[000767]] のコールドスタート要件に従い引き続き image に焼き込み、`/opt/model-cache/` 経由でランタイムステージに 1 回だけコピー (従来の `/tmp → /root → /home/appuser` 三段コピーを撤廃)
- Runtime COPY を明示 (`app/auth_service.py`, `tag_generator`, `tag_extractor`, `tag_inserter`, `article_fetcher` を個別 COPY)。将来的な `app/.venv/` 混入を構造的に防ぐ
- Runtime image から `curl` を外し、HEALTHCHECK は `python -c "import urllib.request; urllib.request.urlopen(...)"` に置き換え

### 3. `.dockerignore` を depth 対応に修正

```
**/.venv/    # 旧: .venv/ → app/.venv/ に効いていなかった
**/venv/
**/.onnx-venv/
**/__pycache__/
**/.pytest_cache/
**/.mypy_cache/
**/.ruff_cache/
```

また `*.txt` の一括除外を削除。`app/tag_extractor/stopwords_{ja,en}.txt` がランタイム必須リソースだったため。`Dockerfile.*` の exception pattern `!Dockerfile.tag-generator-*` を `!Dockerfile.tag-generator` も許容するよう修正。

### 4. image-size + smoke gate に tag-generator を追加、security hardening を同 PR で適用

`.github/workflows/docker-build.yaml`:

- `load` / `if` / `case` の whitelist に `tag-generator`、予算 `MAX_MB=2200`
- tag-generator 用 smoke elif: `import torch, sentence_transformers, keybert, transformers, onnxruntime, fugashi, nltk` / `not torch.cuda.is_available()` / `nvidia`, `triton` 不在検証
- 全 smoke `docker run` に `--network=none --read-only --tmpfs /tmp --cap-drop=ALL` を追加 (security-auditor F-004 対応、既存 news-creator / recap-* にも展開)
- `COPY --from=ghcr.io/astral-sh/uv:0.11.0@sha256:e49fde5daf002023f0a2e2643861ce9ca8a8da5b73d0e6db83ef82ff99969baf /uv /uvx /bin/` に digest pin (security-auditor F-001、tag-generator のみに適用、他サービスは別 PR で repo-wide 展開)

### 5. 横展開

- `.github/workflows/tag-generator.yaml` の `python-version` を `'3.13'` → `'3.14'` に統一 ([[000812]] 準拠)
- `news-creator/Dockerfile` の slim に `torch/include` + `*.pyi` を追加
- `recap-subworker/Dockerfile.recap-subworker` の slim に同様の追加、加えて stdlib ベース HEALTHCHECK (`http://127.0.0.1:8002/health`, `start-period=90s`) を新設

### 検討した代替案と却下理由

- **Tier B: torch / sentence-transformers を完全にドロップして ONNX-only 化**。~1 GiB の追加削減が見込めるが、`tag_extractor/hybrid_extractor.py` の設計変更を伴い、回帰リスクが大きい。[[000767]] の SBERT 前提も再設計が必要になるため別計画に回した。
- **SBERT pre-bake を外してランタイム download にする**。~400 MiB 削減できるが、[[000767]] が cold-start 5–12 秒に収めるために pre-bake を前提にしており、本 PR で外すと E2E Phase 1 で設計した lifespan race 対策が成立しなくなる。
- **Alpine ベースへの切り替え**。musl libc 不整合で numpy / torch の wheel が配布されておらず、ビルド時間が大幅増加するため却下。
- **予算 2000 MiB を初回から狙う**。slim 実測前に攻めて gate が初回から落ちるのを避けたかったため、[[000813]] と同じく初期値は保守的 (2200 MiB)、後続 PR で tighten する。
- **HEALTHCHECK を compose 側だけに寄せる**。orchestrator が compose でない場合 (swarm, k8s 等) に可視性を失うため、image 側にも stdlib ベースの軽量版を残す選択を取った。

## Consequences

### Pros

- tag-generator image: **2.8 GiB (推定) → 1919 MiB** (ローカルビルド実測、予算 2200 MiB を 281 MiB 下回る)
- news-creator: **1243 MiB → 991 MiB** (-252 MiB, 予算 1800 MiB 比 45% → 55% ヘッドルーム)
- recap-subworker: **2753 MiB → 2194 MiB** (-559 MiB, 予算 2900 MiB 比 5% → 24% ヘッドルーム)
- CUDA wheel の再混入は `uv.lock` と CI smoke で二重にブロック (OWASP A08 / CWE-494 対策)
- `[tool.uv.sources] ... explicit=true` により、torch 以外のパッケージが PyTorch index を経由しなくなり、typosquat 面積が縮む ([[000762]] と同じ性質改善)
- `.dockerignore` の `**/` グロブ化で、今後 tag-generator に依存する新 Python サービスでも同じバグが起こらない
- CI smoke-check が `--network=none --cap-drop=ALL` 下で走るようになり、セルフホストランナに import-time exfil が刺さらない
- HEALTHCHECK から `curl` を外したことで、runtime image に攻撃者が乗った場合の egress ツールが 1 つ減る
- `uv` ビルダー image を digest pin したことで、上流 mutable tag の再 push で build が予期せず変わる可能性がなくなる (tag-generator のみ。news-creator / recap-* の digest 化は別 PR)

### Cons / Tradeoffs

- **Tier B (ONNX-only / SBERT 外出し) は未着手**。さらに 1 GiB 級の削減余地はあるが [[000767]] と設計整合性がネックで別計画送り。
- 予算 2200 MiB は保守的で、実測 1919 MiB との間に 281 MiB の余地が残っている。[[000813]] と同様、次回 PR で予算を tighten する前提。
- `digest pin` が tag-generator のみ。他 Python サービス (news-creator / recap-* / tts-speaker) は mutable tag のまま。security-auditor F-001 の完全解消には別 PR が必要。
- `sentence-transformers<5` と `transformers<5` の上限により、upstream の最新機能 (v5 系の CrossEncoder API 変更など) を引き込みにくい。将来 v5 に上げる際は ADR で扱う。
- runtime `COPY --chown=appuser:appuser app/tag_generator ./tag_generator` 等を明示列挙したため、新たなトップレベルモジュールを追加した場合は Dockerfile を更新する必要がある (暗黙の `COPY app/ ./` による事故よりは筋が良いと判断)。
- 同 PR で security hardening (`--network=none`) を既存 news-creator / recap-evaluator / recap-subworker 向け smoke にも適用したため、もし既存 smoke が何らかの DNS / egress に依存していた場合は CI で fail する (本 PR の前提では import のみ、egress 不要)。
