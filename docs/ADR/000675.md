# ADR-000675: QuoteSelector を deterministic 主経路に昇格し LLM fallback 依存を解消する

## Status

Accepted

## Date

2026-04-10

## Affected Services

- acolyte-orchestrator (Python 3.14+)

## Context

[[000673]] で Writer の paragraph micro-generation や Critic の heuristic-first 化を実施し、レポート `743372a2` の v5 で全セクション body_len > 0 を達成した。しかし v5 パイプラインログ分析（`docs/review/acolyte-pipeline-v5-analysis-2026-04-10.md`）により、upstream structured ノードの深刻な非効率が判明した。

### QuoteSelector: LLM 成功率 0/10

| 記事 | LLM #1 (768) | LLM #2 (960) | 結果 |
|------|-------------|-------------|------|
| #1-#10 | 全て eval=上限, len=0 | 全て eval=上限, len=0 | **全 fallback** |

v5 の QuoteSelector は **15 分 9 秒**を消費し、LLM 成功率は **0/10**。全記事が sentence extraction fallback に落ち、30 quotes を取得している。つまり LLM 呼び出しは全くの無駄であり、fallback の sentence extraction が事実上の主経路になっていた。

### 根本原因

Gemma4 の thinking tokens が `num_predict` を消費し、structured JSON を閉じきれない問題が継続（Ollama [#10925](https://github.com/ollama/ollama/issues/10925) OPEN）。QuoteSelector のように短い structured output を要求するタスクでも、thinking が全バジェットを消費して `response_len=0` になる。

### 既存 heuristic 基盤の成熟

[[000670]] で CompressorNode に導入した heuristic infrastructure（`split_sentences`, `score_sentence`, `_extract_query_terms`）は安定動作しており、78K → 8.9K（11%）の圧縮を達成している。この基盤を QuoteSelector の主経路に昇格させることで、LLM 不要で section-conditioned な quote selection が可能になる。

### exec3.md Issue 1 との対応

`exec3.md` の Issue 1「QuoteSelector を deterministic 主経路へ昇格し、LLM fallback 依存を解消する」の実装。方針は「LLM が成功しないと困る層ではなく、deterministic extraction を主経路にしても成立する層」という判断に基づく。

## Decision

**QuoteSelector を 3 層 degradation 構造に再設計し、heuristic extraction を主経路、LLM を副経路、position-based sentence fallback を最終手段とする。**

### 1. Domain 層: `select_top_sentences()` の追加

**ファイル**: `acolyte/domain/compressed_evidence.py`

既存の `compress_article()` が char_budget ベースの圧縮を行うのに対し、`select_top_sentences()` は max_sentences ベースの top-N 選択を行う。

```python
def select_top_sentences(
    body: str,
    queries: list[str],
    *,
    max_sentences: int = 3,
    max_len: int = 200,
    position_fallback: bool = False,
) -> list[CompressedSpan]:
```

- 既存の `split_sentences()` + `score_sentence()` + `_extract_query_terms()` を再利用
- `position_fallback=False`（primary heuristic 用）: score=0 なら `[]` を返し、caller が次の層へ fallback
- `position_fallback=True`（tertiary fallback 用）: score=0 でも先頭 N 文を返し、必ず output を保証

この `position_fallback` フラグの分離が重要。元案では score=0 時に常に先頭 N 文を返す設計だったが、それでは LLM secondary path に到達しなくなるため、明示フラグで primary / tertiary の挙動を切り替える。

### 2. Node 層: 3 層 degradation と raw/prompt body 分離

**ファイル**: `acolyte/usecase/graph/nodes/quote_selector_node.py`

#### 3 層 degradation

```
Primary:   _heuristic_select(raw_body, section_queries) → quotes あり → return
Secondary: LLM generate_validated(prompt_body) → quotes あり → raw_body で offset 検証 → return
Tertiary:  _sentence_fallback(raw_body, section_queries, position_fallback=True) → return
```

#### raw body と prompt body の分離

- **raw_body**: `hydrated_evidence[item_id]` から取得。offset 計算・substring 検証の基準。heuristic primary もこちらを入力に使う
- **prompt_body**: `compressed_evidence[item_id]` があれば join、なければ `raw_body[:2000]`。LLM secondary prompt 用

この分離により、compressed/truncated body 基準の offset になり `SelectedQuote.start_offset/end_offset` の「article body 基準」契約を壊す問題を回避した。

#### query 解決の優先順位

`query_facets > search_queries > section_key/topic` の順で解決する。CompressorNode（[[000670]]）の `_collect_queries_for_articles()` と同じ優先順位を維持し、Gatherer / Compressor との query semantics を揃えた。

#### sentence fallback の統合

旧 `_sentence_fallback()` は独自の `split("\n")` で文分割していたが、JP/EN 混在・略語・小数・改行の扱いが heuristic 主経路の `split_sentences()` と不一致になる。新実装では `select_top_sentences(..., position_fallback=True)` を使い、shared splitter で統一した。

### 検討した代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| TF-IDF / TextRank ベースの extractive summarization | scikit-learn / summa 等の追加依存が必要。既存 `score_sentence()` の dual scoring（ASCII word + CJK bi-gram Jaccard）で十分な品質 |
| CompressorNode の事前計算 spans をそのまま使う | CompressorNode は全 section query の union で scoring。QuoteSelector は section 固有 query で scoring が必要（section-conditioned recall の維持） |
| LLM を完全除去する | Ollama structured output が改善した際の保険として secondary に残す。現時点では correctness の主軸にしない |
| `_extract_query_terms` を public 化する | テスト都合だけで公開 API 面を広げる理由にならない。private のまま維持 |
| `position_fallback` を常に有効にする | LLM secondary path が実質到達不能になる。3 層 degradation の設計と矛盾 |

## Consequences

### Pros

- **QuoteSelector の推論時間が実質ゼロに**: v5 で 15 分 9 秒だった LLM 待ちが heuristic primary で消滅。パイプライン全体で ~15 分短縮
- **10/10 記事で確実に quotes 取得**: 3 層 degradation により、heuristic → LLM → position fallback の順で必ず出力。`selected_quotes=[]` が発生しない
- **section-conditioned recall の維持**: raw hydrated body を section 固有の query で再スコアリングすることで、CompressorNode の union-compression とは独立した section-aware 選択を実現
- **offset 正当性の保証**: raw hydrated body 基準の offset 検証により、compressed/truncated body 基準で壊れていた `start_offset/end_offset` 契約を修正
- **shared splitter で品質統一**: heuristic primary と tertiary fallback が同じ `split_sentences()` / `score_sentence()` を使うため、JP/EN/混在の文分割品質が一貫
- **LLM secondary を保険として維持**: Ollama #10925 / #15260 修正後に structured output が安定すれば secondary → primary に戻せる配線が残る

### Cons / Tradeoffs

- **語彙的マッチのみ**: heuristic scoring は keyword overlap + CJK bi-gram Jaccard であり、意味的類似（synonym, paraphrase）はカバーしない。「semiconductor」と「chip」のように query 側に両方含まれていないとマッチしない。FactNormalizer の LLM が下流で補完する前提
- **LLM 呼び出し回数の減少が downstream 品質に影響しうる**: LLM が有効だったケース（v5 では 0/10 だが、モデル改善で変わりうる）で heuristic の方が品質が低い可能性。ただし現時点では LLM 成功率 0% のため実害なし
- **`checkpoint_enabled=true` でも node 内 multi-article ループは途中再開されない**: LangGraph の durable execution は super-step（node 境界）で保存される。QuoteSelector 内の per-article ループの途中からは resume できない。node 分割は将来課題
