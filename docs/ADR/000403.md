# ADR-000403: Backfill 基盤と Projection バージョニングを導入する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — Backfill usecase / job、Projection version gateway、Projector 改修
- alt-db (PostgreSQL) — 3 マイグレーション追加

## Context

Knowledge Home の projector は Phase 1-2 で稼働を開始したが、それ以前に蓄積された記事・サマリ・タグは knowledge_events テーブルに存在しないため、Knowledge Home には投影されない。ユーザーが Knowledge Home を開いた際に過去記事が表示されない「空の Home」問題が発生する。

また、projector のロジック（スコア計算、why reason の割り当て等）は今後改善が見込まれるが、現状では projection ロジックを変更するとすべてのデータを再投影する必要がある。どのバージョンの projection ロジックで書き込まれたデータなのかを区別する仕組みがない。

要件:
- 過去の記事データを knowledge_events に合成イベントとして投入し、projector が通常通り投影できるようにする（Backfill）
- Backfill は一時停止・再開可能で、進捗を追跡できること（大量データに対応）
- Projection ロジックのバージョンを管理し、将来の「全データ再投影」を安全に行えること

## Decision

**Backfill ジョブ管理テーブルと Projection バージョニングテーブルを導入し、checkpoint ベースの再開可能な backfill 基盤を構築する。**

### データベース設計

3 つのマイグレーションを追加:

1. **`knowledge_backfill_jobs`** — Backfill ジョブの状態管理テーブル
   - `status`: `pending` → `running` → `completed` / `paused` / `failed`
   - `cursor_user_id` / `cursor_date` / `cursor_article_id`: checkpoint カーソル（中断・再開用）
   - `total_events` / `processed_events`: 進捗追跡

2. **`knowledge_projection_versions`** — Projection ロジックのバージョン管理
   - 初期データとして version 1 を `active` で INSERT
   - 将来 v2 projection を追加する際、`pending` → テスト → `active` 切り替えのワークフローに対応

3. **`projection_version` カラム追加** — `knowledge_home_items`, `today_digest_view`, `recall_candidate_view` にバージョン番号を付与
   - 投影データがどのバージョンのロジックで書き込まれたかを区別できる

### Backfill の動作原理

Backfill は「既存データから合成イベントを生成し、通常の projector パイプラインに流す」方式を採用する。

```
既存 articles/summaries/tags テーブル
    ↓ BackfillJob が走査
knowledge_events テーブル（dedupe_key=backfill:{articleID}）
    ↓ 既存の KnowledgeProjector が消費
knowledge_home_items テーブル
```

**Projector を改修せずに Backfill を実現できる**のがこの設計の要点。合成イベントは `dedupe_key` で冪等になるため、同じ記事を複数回 backfill しても重複しない。

### Projection バージョニング

Projector は起動時に `GetActiveVersionPort` で現在の active version を取得し、upsert 時に `projection_version` カラムに設定する。将来の v2 projection では:

1. `knowledge_projection_versions` に version 2 を `pending` で INSERT
2. v2 ロジックで backfill を実行（`projection_version=2` で書き込み）
3. 検証完了後、version 2 を `active` に切り替え
4. v1 データは `projection_version=1` で識別可能なため、段階的にクリーンアップ可能

### Clean Architecture 層構造

| 層 | ファイル | 責務 |
|---|---------|------|
| Domain | `knowledge_backfill.go`, `knowledge_projection_version.go` | Struct + 状態定数 |
| Port | `knowledge_backfill_port/`, `knowledge_projection_version_port/` | CRUD インターフェース |
| Gateway | `knowledge_backfill_gateway/`, `knowledge_projection_version_gateway/` | Repository 委譲 |
| Driver | `alt_db/knowledge_backfill_repository.go`, `knowledge_projection_version_repository.go` | SQL 実装 |
| Usecase | `knowledge_backfill_usecase/` | Start/Pause/Resume/Status |
| Usecase | `knowledge_projection_health_usecase/` | 投影健全性の集約 |
| Job | `knowledge_backfill_job.go` | バックグラウンドバッチ処理 |

## Consequences

### Pros

- **Projector 再利用** — Backfill は既存の projector パイプラインに合成イベントを流すだけで、projection ロジックの二重管理が不要
- **冪等性** — `dedupe_key=backfill:{articleID}` により、同一記事の重複投影を自動防止
- **中断・再開** — checkpoint カーソルにより、大量データの backfill を安全に中断・再開可能
- **バージョン追跡** — 投影データにバージョンが付くことで、将来の projection ロジック変更時に影響範囲を特定できる
- **段階的移行** — v1 と v2 のデータが共存でき、即座に全データ再投影する必要がない

### Cons / Tradeoffs

- **イベント量の増大** — Backfill により `knowledge_events` テーブルのサイズが大幅に増加する（既存記事数 × イベント種別数）
- **Backfill 中の projector 負荷** — 大量の合成イベントが projector のバッチサイズ（100件/回）を圧迫する可能性がある。必要に応じてバッチサイズの動的調整を検討
- **Version 切り替えの手動性** — Active version の切り替えは Admin API 経由の手動操作。自動ロールバックの仕組みはまだない
