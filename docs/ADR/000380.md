# ResetFeedLinkFailures を UPSERT に変更し、フィードヘルスデータ欠落を修正

## ADR's STATUS

Accepted

## CONTEXT

`/sv/settings/feeds` ページで一部のフィード（`https://zenn.dev/topics/database/feed` 等）が「No health data available」と表示される不具合が報告された。

調査の結果、`feed_link_availability` テーブルに対応する行が存在しないことが原因と判明した。`feed_links` テーブルには76件のレコードがあるが、`feed_link_availability` には72件しか存在せず、4件が欠落していた。

### 欠落の原因

1. 初期マイグレーション（`20260111000000_create_feed_link_availability.sql`）はマイグレーション実行時点で存在する全 `feed_links` に対して `feed_link_availability` 行を一括挿入する
2. マイグレーション以降に登録されたフィードには行が作られない
3. `ResetFeedLinkFailures()` は `UPDATE` 文のみを使用しており、行が存在しない場合は0行更新（no-op）となる
4. `IncrementFeedLinkFailures()` は UPSERT パターンを使用しているが、フィード取得失敗時にのみ呼ばれる
5. 結果として、マイグレーション後に登録され一度も失敗していないフィードは `feed_link_availability` 行が永遠に作られない

フロントエンドでは `Availability` が `nil` の場合、ドメインモデルの `GetHealthStatus()` が `"unknown"` を返し、`FeedHealthBadge` コンポーネントが "No health data available" と表示する。

## DECISION MAKING

`ResetFeedLinkFailures()` を `UPDATE` から `INSERT ... ON CONFLICT DO UPDATE`（UPSERT）に変更する。これは同ファイル内の `IncrementFeedLinkFailures()` で既に使われているパターンと同じである。

既存の欠落4行についてはワンショット SQL で補填した。マイグレーションファイルは作成しない（今後は UPSERT により自動的に行が作られるため再発しない）。

### 変更内容

**`alt-backend/app/driver/alt_db/feed_link_availability_driver.go`**

```go
// Before: UPDATE のみ（行がなければ no-op）
UPDATE feed_link_availability SET consecutive_failures = 0
WHERE feed_link_id IN (SELECT id FROM feed_links WHERE url = $1)

// After: UPSERT（行がなければ INSERT、あれば UPDATE）
INSERT INTO feed_link_availability (feed_link_id, is_active, consecutive_failures)
SELECT id, true, 0 FROM feed_links WHERE url = $1
ON CONFLICT (feed_link_id) DO UPDATE SET
    consecutive_failures = 0
```

- `INSERT` 時: `is_active = true`, `consecutive_failures = 0` で新規行を作成
- `ON CONFLICT` 時: `consecutive_failures = 0` にリセット（`is_active`, `last_failure_at`, `last_failure_reason` は保持）
- `SELECT ... FROM feed_links WHERE url = $1` により重複 URL（`IN` パターン、ADR-000299参照）も正しく処理

**変更しなかったもの:**

- `IncrementFeedLinkFailures()` — 既に UPSERT で正しい
- `DisableFeedLink()` — `IncrementFeedLinkFailures()` の後にのみ呼ばれるため行は必ず存在する
- ドメイン層・フロントエンド — 既に全5ステータスを正しく処理している

## RESULTS, EFFECTS

### PROS

- マイグレーション後に登録されたフィードでも、最初の成功収集時に `feed_link_availability` 行が自動作成される
- 既存の UPSERT パターン（`IncrementFeedLinkFailures`）と整合性がとれた
- ポートインターフェースのシグネチャ変更なし（`(ctx, feedURL) -> error`）
- フロントエンドの変更不要

### CONS, TRADEOFF

- `ResetFeedLinkFailures` の SQL が `UPDATE` 1文から `INSERT ... ON CONFLICT` に変わったため、わずかにオーバーヘッドが増える（実用上無視できるレベル）
- フィード登録時（`RegisterRSSFeedLink`）に同時に `feed_link_availability` 行を作成するアプローチもあったが、UPSERT で十分カバーできるため過剰な対応は避けた
