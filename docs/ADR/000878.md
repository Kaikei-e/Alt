# ADR-000878: Knowledge Loop OODA 司令卓を完成させ、ステージ別ワークスペース・インラインパッチ・depth CSS・enricher を整合させる

## Status

Accepted

## Date

2026-05-02

## Affected Services

- alt-frontend-sv: OODA per-stage workspace panels、stream inline patch（superseded case）、stage-aware plane depth CSS、OodaPipeline pulse animation
- knowledge-sovereign / knowledge_loop_projector: continue_context JSON 生成、change summary fallback、HomeItemDismissed why-text
- knowledge-sovereign / surface_planner_cron: MaxBatchesPerTick 設定
- compose/compose.staging.yaml, compose/sovereign.yaml: 環境変数追加

## Context

[[000873]] で OODA 状態遷移・visibility/completion lifecycle・inline entry のフレームワークが確立された。しかし fb.md レビューで以下の未解消ギャップが指摘されていた。

### ギャップ 1: stream `superseded` フレームが coalescedRefresh を発火させていた

`applyStreamFrame()` の switch 文に `"superseded"` case が存在しなかったため、superseded フレームが `false` を返し、呼び出し元の `onFrame` ハンドラが `coalescedRefresh` (= `invalidate("loop:data")`) へフォールバックしていた。これは「inline patch が成功すれば invalidate 不要」という設計意図を破り、不要なフルリロードを毎回発生させていた。

### ギャップ 2: OODA ワークスペースがステージに無関心

`+page.svelte` のワークスペース領域は Observe/Orient/Decide/Act の全ステージで同一 DOM を表示していた。[[000876]] で canonicalize された遷移マトリクスと [[000866]] の「司令卓」設計は、各ステージで異なるメタデータと行動メニューを提示することを前提としていたが、UI 側が追いついていなかった。

### ギャップ 3: LoopPlaneStack がステージ変化に反応しない

`loop-depth.css` に `.loop-plane-root[data-stage="orient"]` / `[data-stage="act"]` のカスタムプロパティ定義が存在していたが、`LoopPlaneStack.svelte` がそれらを参照していなかった。Svelte のスコープ化ハッシュセレクタと外部 CSS の specificity 競合が原因で、グローバル CSS から直接 pane スタイルを上書きできなかった。

### ギャップ 4: projector の continue_context / change summary / dismiss why-text

- `HomeItemOpened` イベント時に `continue_context` JSON が生成されていなかった（`ContinueStream.svelte` の `entry.continueContext` が常に null）
- `ChangeSummaryBuilder` が excerpt-only ペイロード（新 summary を持たないイベント）でも空文字を返していたため、変更カードが空白表示されていた
- `HomeItemDismissed` の enricher による why-text が未実装で、dismissal の経緯を説明するテキストが生成されなかった

### ギャップ 5: surface planner のバックログが 1 tick 1 バッチに制限されていた

tick 間隔が短くても 1 バッチ（最大 `BATCH_SIZE` 件）しか処理しないため、大量のフィード記事が流入すると未 surface 記事が積み上がる速度がプランナーの処理速度を上回る可能性があった。

## Decision

### 決定 1: `applyStreamFrame` に `superseded` case を追加し inline 削除を完成させる

```typescript
case "superseded":
    entries = entries.filter((e) => e.entryKey !== frame.entryKey);
    bucketEntries = bucketEntries.filter((e) => e.entryKey !== frame.entryKey);
    return true;
```

`return true` により `onFrame` は `coalescedRefresh` を呼ばない。新エントリは後続の `appended` フレームで届く。`applyStreamFrame` の全 case（appended / revised / withdrawn / superseded）をカバーするユニットテスト群をあわせて追加した。

### 決定 2: per-stage OODA workspace panels を実装する

ステージごとに `+page.svelte` の workspace DOM を切り替える:

- **Observe**: freshness 日付、priority badge（critical/continuing のみ）、signal count を横並びメタ行で表示
- **Orient**: Continue context 全文（`entry.continueContext.summary`）と最近のアクションラベル一覧を `ContinueStream` 経由で表示、変更差分テキストは `.workspace-context-label` でラベル付け
- **Decide**: `entry.decisionOptions` をローマ数字付き `<ol>` でレンダリング、各 option が `advanceEntry()` を呼ぶ
- **Act**: `entry.actTargets` を 2 列グリッドで表示（type / route）

検討した代替案: ステージごとに独立した Svelte コンポーネントを作る案はファイル数が増えるが、実装の分散よりも `+page.svelte` 単体での段階的読みやすさを優先してインライン `{#if}` で実装した。

### 決定 3: CSS custom properties によるステージ aware depth を LoopPlaneStack に接続する

Svelte のスコープ化セレクタ（`.pane.svelte-xxxx[data-depth-state="near"]` = specificity 0,3,0）を外部 CSS から直接上書きすることはできない。代わりに:

1. `loop-depth.css` の `.loop-plane-root[data-stage]` ブロックで CSS custom properties を宣言
2. `LoopPlaneStack.svelte` の `<style>` ブロック内でこれらを `var(--loop-context-near-opacity, fallback)` 形式で参照

これにより specificity 競合が発生せず、reduced-motion 対応も `loop-depth.css` の `@media` ブロック内で変数を上書きするだけで完結する。

### 決定 4: ContinueContextBuilder を新設し reproject-safe に continue_context を生成する

`HomeItemOpened` イベントのハンドラから `buildContinueContextJSON(ev)` を呼び出す。タイムスタンプは `eventObservedAt(ev)`（イベントペイロード由来）のみ使用し、`time.Now()` を参照しない。これは reproject-safe 要件（[[000773]] / IMPL_BASE.md §Event-time purity）に準拠する。

`ChangeSummaryBuilder` には excerpt-only ペイロード向けのフォールバックを追加: `out.summary == ""` かつ `out.oldSummaryText != ""` の場合、旧テキストをそのまま差分表示に使う。

`HomeItemDismissed` の enricher に why-text 生成を追加: dismissal タイムスタンプ + evidence refs から「Dismissed at [timestamp] — recheck, mark reviewed, or archive.」形式のテキストを構成する。

### 決定 5: MaxBatchesPerTick を surface_planner_cron Config に追加する

```go
MaxBatchesPerTick int  // env: KNOWLEDGE_SOVEREIGN_PLANNER_MAX_BATCHES_PER_TICK
```

tick ごとに最大 `MaxBatchesPerTick` 回バッチを実行するループを追加。バッチ結果が 0 件になった時点でループを打ち切る（空送り防止）。staging では 4、本番では 8 を既定値とする。これにより フィード流入量がスパイクしても 1 tick で複数バッチを消化できる。

## Consequences

### Pros

- **inline patch が完結**: superseded / withdrawn / revised / appended の全 case で `invalidate` が発火しなくなり、SSE ストリームの更新が真の差分操作になる
- **司令卓 UX が OODA サイクルに追従**: 各ステージで最適な情報密度を提供でき、ユーザが次のアクションを選択する認知負荷が下がる
- **depth CSS が動的**: stage 変更時に `LoopPlaneStack` の奥行きが自動更新される（orient でコンテキスト面が浮上、act でバックグラウンド面が沈む）
- **continue_context が実データ表示**: `ContinueStream.svelte` が実際の継続コンテキスト（要約 + 最近アクション）を表示できるようになった
- **projector が enricher 経由で why-text を生成**: dismiss カードに dismissal 経緯が表示される
- **surface planner のバックログ消化速度向上**: MaxBatchesPerTick により 1 tick で複数バッチを処理できる

### Cons / Tradeoffs

- **`+page.svelte` のステージ分岐が増加**: workspace の `{#if selectedStageName === "observe"}` 系分岐が 4 本ある。エントリが null の場合の guard も各ブランチに必要で、将来のステージ追加時に修正箇所が集中する。ステージが 5 つ以上になる場合はコンポーネント分割を検討すること。
- **CSS custom properties の fallback 二重管理**: `loop-depth.css` の変数宣言と `LoopPlaneStack.svelte` の `var()` fallback 値が乖離すると視覚的な意図が崩れる。変更時は両ファイルを同時に更新すること。
- **MaxBatchesPerTick のチューニングが運用依存**: バッチサイズ × MaxBatchesPerTick が tick 時間内に完了しない場合、tick が遅延する。負荷テストなしに大きな値を設定しないこと。
