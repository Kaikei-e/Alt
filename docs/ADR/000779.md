# ADR-000779: alt-backend E2E の外部 upstream を単一 FastAPI deps-stub と複数ネットワーク alias で吸収する

## Status

Accepted

## Date

2026-04-19

## Affected Services

- alt-backend (Go) — staging 環境で全ての upstream URL 環境変数 (`SEARCH_INDEXER_CONNECT_URL` / `PRE_PROCESSOR_URL` / `RECAP_WORKER_URL` / `RAG_ORCHESTRATOR_URL` / `AUTH_HUB_URL` / `MQHUB_CONNECT_URL`) を同じ stub コンテナの Docker network alias に向ける
- `e2e/stubs/alt-backend-deps-stub/` (Python / FastAPI) — 新規追加の単一コンテナ

## Context

alt-backend は実運用時に以下 6 upstream + 1 認証基盤と対話する:

| upstream | protocol | 用途 |
|---|---|---|
| search-indexer | Connect-RPC | `/v1/feeds/search` / `/v1/articles/search` |
| pre-processor | Connect-RPC | `/v1/feeds/fetch/summary` 等の要約経路 |
| recap-worker | HTTP/REST | `/v1/morning-letter/updates` 等の daily digest |
| rag-orchestrator | HTTP/REST + Connect-RPC | `/v1/rag/*` の RAG 応答 |
| knowledge-sovereign | Connect-RPC | Knowledge Home projection / event ingestion |
| mq-hub | Connect-RPC | tag 生成のイベント publish (staging では flag off) |
| auth-hub | HTTP/REST | `/v1/system-user` 内部 handler からの identity 取得 |

さらに RSS 登録 scenario では alt-backend が feed URL を実際に HTTP GET するため、feed を返す host (例: `stub.invalid`) が alt-staging network 内で解決できる必要がある。

本 E2E の目的 ([[000778]]) は alt-backend 自身の REST surface を守ることで、upstream の契約検証は Pact CDC ([[000591]]) が分担している。したがって staging で 6 upstream を本物のサービスとして立てる必要はない。しかし何らかの形で upstream の応答を返さないと alt-backend のハンドラは 500 を返して E2E が成立しない。

- 6 upstream を全部立てると compose 起動時間と依存 DB (recap-db / knowledge-sovereign-db / meili / redis-streams) が増え、CI フィードバックが分単位で悪化する
- WireMock / Mockoon / Prism 等の汎用 mock ツールは設定ファイルを持ち込む必要があり、drift 検出が増える
- [[000764]] recap-worker 向けに既に `e2e/stubs/recap-pipeline-stub/` が FastAPI で存在し、同じコンテナに 4 alias (`rw-stub-*`) を付けて複数サービスを同時 mock するパターンが稼働実績あり
- alt-backend のどの handler が何を呼ぶか把握済み (`/app/driver/*_connect/client.go` + `/app/gateway/*_gateway/*.go`)

## Decision

**[[000764]] の recap-pipeline-stub パターンをそのまま踏襲し、単一 FastAPI コンテナ `alt-backend-deps-stub` を `alt-staging` network に 8 alias で登録する。** 採用構成:

1. **コンテナ構成は `e2e/stubs/recap-pipeline-stub` と同形**。`python:3.13-alpine` + `fastapi==0.115.6` + `uvicorn==0.32.1` で `:80` を listen。`Dockerfile` は recap-stub からの差分を最小化し、CI のビルド時間も同等にする。
2. **Docker network alias を 8 個付与する**: `search-indexer`, `pre-processor`, `recap-worker`, `rag-orchestrator`, `knowledge-sovereign`, `mq-hub`, `auth-hub`, `stub.invalid`。FastAPI は path でルートを解決するので、どの alias で入ってきても同じコンテナに到達する。
3. **Connect-RPC の path schema に準拠**する。`POST /<FQSN>/<method>` (例: `POST /alt.search.v2.SearchService/SearchFeeds`) を FastAPI route としてそのまま登録し、空の `proto3-JSON` 応答を返す。Connect-RPC unary は framing を持たず content-type: application/json + 生 JSON body なので、FastAPI の `application/json` レスポンスと互換。
4. **stub.invalid には最小 RSS 2.0 + 1x1 PNG + HTML を返す**。`FEED_ALLOWED_HOSTS=stub.invalid` で alt-backend の SSRF guard (`IsFeedHostAllowed`) を通し、`GET /alt-backend/e2e/{slug}.xml` が 1 item 入りの valid な RSS ドキュメントを返すことで RSS 登録 scenario の `ValidateAndFetch` 経路が成立する。
5. **catch-all は 200 + `{status: stub-noop, path: ...}` を返し、未ハンドル path を stdout に残す**。alt-backend が deserialize に失敗したら scenario 側の assertion 失敗として露見し、stub 側のログで alt-backend が何を叩いてきたかが追える。
6. **shape の精度は Pact CDC 側の契約に委ねる**。本 stub は「alt-backend が 500 を返さず handler を最後まで回せる」を最小保証するだけで、本物の upstream の shape と deep-equal にはしない。Pact contract が drift すれば `alt-backend/pacts/*.json` 側で検出される。

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| 6 upstream を本物コンテナで staging に立てる | compose の bring-up 時間と依存 DB が 3 個増える。[[000778]] の fast-feedback 方針と矛盾し、`--jobs 4` 並列の意味も希釈される |
| WireMock / Mockoon / Prism のような汎用 HTTP mock | YAML/JSON 設定ファイルを追加管理する必要があり、FastAPI より drift しやすい。Alt リポには既に recap-pipeline-stub で FastAPI パターンが稼働しており、認知負荷を増やさない選択肢としても FastAPI が優る |
| upstream 種別ごとに stub コンテナを分ける | 6 コンテナ + Dockerfile が増え、compose の service 数が倍化する。recap-worker 向けの [[000764]] は 1 コンテナ 4 alias で既に実績がある |
| nginx で全 path を 200 + 空 body に倒す | Connect-RPC の JSON 応答 shape が空 body では deserialize できない（struct フィールドが nil になり handler 側で NPE）。最低限の JSON envelope は返す必要がある |
| alt-backend のクライアント側で upstream をモックする (テスト用 build tag) | プロダクションコードに test-only 分岐を増やすのは OSS 的にも運用的にも負債。E2E は「プロダクション binary をそのまま起動して外から叩く」原則を破らない |

### 範囲外 (今回やらないこと)

- Connect-RPC method ごとに shape を精密化 — Pact consumer/provider 側の責務
- streaming endpoint (`POST /v1/feeds/summarize/stream` / `POST /sse/v1/rag/answer`) の完全な SSE / NDJSON 再現 — 後続 ADR で取り扱う
- stub の auto-generation (proto → FastAPI 雛形) — 4 stub 以上揃った時点で共通化 ADR を起こす
- knowledge-sovereign のような stateful projector の状態を再現 — 該当 scenario は空 projection で通る仕様

## Consequences

### Pros

- **1 コンテナで alt-backend の外側依存が全部吸収される**。compose の bring-up に増えるのは deps-stub 1 個だけで、CI フィードバックは local で 15 秒台 / stack bring-up 込み 60 秒台を維持
- **既存の recap-pipeline-stub パターンと認知負荷が同じ**。「`e2e/stubs/<name>/` に FastAPI app を書く」というメンタルモデルを増やさない。alias 数の違い (4 → 8) だけが差分
- **Docker network alias は Docker DNS が自動でラウンドロビンする**ので、`http://search-indexer/` のような実運用そのままの URL が alt-backend の staging env に書ける。`search-indexer:9300` のようなポート付き URL を staging 用に書き換える必要がない
- **catch-all のログが stdout に出る**ので、新規 handler が増えて未ハンドル path を叩いた場合は `docker logs alt-staging-alt-backend-deps-stub` で即座に見える。scenario が通らない時のデバッグが FastAPI の access log で完結

### Cons / Tradeoffs

- **stub と実サービスの shape drift が scenario 外では検出されない**。Connect-RPC の response struct が実サービス側で変わっても、stub が最小 JSON を返すだけなら alt-backend の handler は通ってしまう。drift 検出は Pact CDC に全面依存
- **FastAPI の起動が ~2 秒かかる**。compose healthcheck の `start_period: 3s` + `retries: 20` で吸収しているが、Go サービスより cold start が遅い分だけ stack 起動時間に影響する
- **alias 8 個は compose.staging.yaml の可読性を下げる**。`networks: alt-staging: aliases: [...]` のリストが長くなり、alias が何のサービスを代替しているかを comment で補足しないと未来の読者には伝わらない
- **rag-orchestrator の SSE path (`/v1/answer`) は最小の 2 frame しか返さない**。Hurl 自体が SSE を積極検証しない制約と一致しているが、将来 SSE を Hurl で検証する別 ADR が出たら stub 側にも実相応の streaming 応答を実装する必要がある
