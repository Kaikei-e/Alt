# ADR-000609: Ask Augur の follow-up タイムアウトを planner 過剰 clarification 修正・handler event マッピング追加・ConversationState 永続化で解消する

## Status

Accepted

## Date

2026-03-29

## Affected Services

- rag-orchestrator (Go)
- news-creator (Python) — 設定統一のみ

## Context

[[000608]] で品質ゲート（`stream_short_answer_hard_stop`）の誤検知を修正した直後、Ask Augur で follow-up クエリのタイムアウトが頻発した。

### 根本原因 1: ConversationPlanner の過剰 clarification 判定

`planner.go` の `isAmbiguousFollowUp()` は `詳しく教えて` を含む全クエリを ambiguous と判定していた。`PyO3について詳しく教えて` のように具体的なトピック付きの質問も ambiguous 扱いとなり、さらに `resolveAmbiguous()` で `state == nil`（ConversationStore 未接続のため常に nil）→ 即 `NeedsClarification=true` を返していた。

ログ証拠:
- `"PyO3について詳しく教えて"` (history_turns:2) → `operation: clarify, confidence: 0.3`
- `"そのリスクについて詳しく教えて"` (history_turns:4) → 同上

### 根本原因 2: Connect-RPC handler の clarification event 未マッピング

`augur/handler.go` の `convertStreamEvent()` に `StreamEventKindClarification` の case が存在しなかった。default case で `"unknown stream event kind"` をログして skip → clarification event がフロントエンドに到達しない → ストリームが即座に done → FE でタイムアウト。

### 根本原因 3: ConversationStore への state 書き込み未接続

ADR-000604 で `ConversationStore` と `DeriveStateUpdate` は実装済みだったが、`Stream()` の完了パスで `Put()` が呼ばれていなかった。state が常に nil のため、planner の reference resolution が一切機能しなかった。

### なぜ [[000608]] の修正後に顕在化したか

[[000608]] 以前は全回答が `stream_short_answer_hard_stop` で fallback → clarification パスに到達する前にストリームが終了していた。品質ゲートの誤検知修正により正常フローが活性化し、planner の clarification 判定が初めて実行されるパスが露出した。

### 副次的問題: 設定ドリフト

`.env` に `OLLAMA_NUM_PARALLEL=2`、`OLLAMA_MODEL=gemma3-4b-8k`、`AUGUR_KNOWLEDGE_MODEL=gemma3-4b-8k` が残存し、compose/ai.yaml のデフォルト（`NUM_PARALLEL=1`、`gemma3-4b-12k`）をオーバーライドしていた。news-creator-backend が `Parallel:2, KvSize:16384` で起動し、12k single-slot 設計の意図に反していた。

## Decision

### 1. `isAmbiguousFollowUp` の厳密化

パターンマッチ後、クエリの残り部分に実質的な内容があるかチェックする。パターンと助詞・句読点を除去した後の残りが 2 rune 未満の場合のみ ambiguous と判定。

- `もっと詳しく` → ambiguous（パターンのみ）
- `PyO3について詳しく教えて` → NOT ambiguous（`PyO3` が残る）

### 2. `resolveAmbiguous` の nil state フォールバック改善

`state == nil` でも `history` が非空の場合は `OpDetail` にフォールバックする。clarification を要求するのは history も空の完全初回ターンのみ。

業界ベストプラクティスとの整合: まず retrieval を試み、confidence が低い場合のみ clarification をトリガーする（ByteByteGo: "How Agentic RAG Works"）。

### 3. Connect-RPC handler に clarification event マッピング追加

`convertStreamEvent()` に `StreamEventKindClarification` の case を追加。`StreamClarification.Message` を `kind="clarification"` として proto 応答に変換し、terminal event として送信。

### 4. ConversationStore への state 永続化

`Stream()` の done パスで `DeriveStateUpdate()` を呼び、`ConversationStore.Put()` で state を保存。以降の follow-up で planner が state を参照し、topic/entities/scope の reference resolution が機能する。

### 5. gemma3-4b-12k への完全統一

`.env`、`.env.template`、`pre-processor/quality_judger.go`、`recap-evaluator/config.py` から `gemma3-4b-8k` 参照を排除し、`gemma3-4b-12k` に統一。`OLLAMA_NUM_PARALLEL=1` を `.env` で確定。

## Consequences

### Pros

- **Follow-up タイムアウト解消**: `PyO3について詳しく教えて` 等の具体的トピック付き follow-up が clarification ではなく回答を返す
- **Clarification の正常動作**: 真に曖昧なクエリ（`もっと詳しく`、history なし）は clarification event が FE に到達する
- **Multi-turn 品質向上**: ConversationState が永続化され、2ターン目以降で topic confidence、focus entities、last answer scope を参照した reference resolution が有効化
- **設定一貫性**: 全コードベースで gemma3-4b-12k / NUM_PARALLEL=1 に統一

### Cons / Tradeoffs

- **CJK テキストの ambiguous 判定**: 助詞除去ヒューリスティックが日本語の全ケースをカバーしない可能性。将来的に MeCab 等の形態素解析が必要になる場合がある
- **ConversationStore はインメモリ LRU**: rag-orchestrator 再起動で state が失われる。永続化が必要になった場合は PostgreSQL への移行を検討
- **SLOT INVARIANT VIOLATION 未解決**: `NUM_PARALLEL=1` で HybridPrioritySemaphore のスロットリークが顕在化している（PM-2026-013 参照）。セマフォの `be_slots=0` パス監査が別途必要

### ベンチマーク結果

| カテゴリ | 修正前 | 修正後 |
|----------|--------|--------|
| General queries | 6/9 PASS | 7/9 PASS |
| Article-scoped | 0/4 PASS (hard stop) | 4/4 PASS |
| `stream_short_answer_hard_stop` | 12件 | 1件（正当な品質ゲート） |
| `unknown stream event kind` | 3件 | 0件 |
| Follow-up タイムアウト | 頻発 | 解消 |
