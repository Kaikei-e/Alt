# ADR-000654: Acolyte LangGraph パイプラインを AIX Gemma4 26B に接続しレポート生成を実現する

## Status

Accepted

## Date

2026-04-09

## Affected Services

- acolyte-orchestrator (Python 3.14+ / Starlette / Connect-RPC / LangGraph)
- acolyte-migration-atlas — Atlas マイグレーション管理
- alt-butterfly-facade (Go) — BFF ルーティング + env var 追加
- alt-frontend-sv (TypeScript/Svelte 5) — ナビゲーション・フォント・ページ改善

## Context

[[000653]] で Acolyte の骨格（proto 定義、DB スキーマ設計、Connect-RPC サービス、フロントエンドページ）を構築したが、`StartReportRun` はジョブをメモリに入れるだけで、LangGraph パイプラインは実行されていなかった。また以下の課題が残っていた。

1. **RPC がスタブ (501 UNIMPLEMENTED)** — CreateReport, ListReports, GetReport が未実装
2. **DB 未接続** — in-memory ストアのみでコンテナ再起動でデータ消失
3. **Atlas マイグレーション未構築** — 直接 SQL 実行は Alt のルール違反
4. **search-indexer API 不一致** — Meilisearch 直叩き (404) でエビデンス取得失敗
5. **Planner の JSON 切れ** — `num_predict=2000` で出力が truncate、JSON parse 失敗
6. **フロントエンドの不完全性** — サイドバーにリンクなし、フォント未適用、ランタイムエラー

## Decision

### 1. DB 永続化を Atlas マイグレーション経由で実現する

`acolyte-migration-atlas/` を新規作成し、recap-migration-atlas と同じパターン（Atlas 0.31 + Docker migrator コンテナ）で 7 テーブルの DDL を管理する。compose で `acolyte-db-migrator` サービスを追加し、`depends_on: service_completed_successfully` で orchestrator の前に実行する。

専用 DB `acolyte-db`（PostgreSQL 18 Alpine）を使用し、主幹 DB (alt-db) とは分離する。

### 2. Connect-RPC ハンドラーを PostgresReportGateway で実装する

`MemoryReportGateway` → `PostgresReportGateway` に差し替え、`psycopg_pool.AsyncConnectionPool` で接続。テスト時は `MemoryReportGateway` を使い続ける（conftest.py で分離）。

### 3. OllamaGateway で AIX に直接接続する

news-creator 経由ではなく、AIX 上の Ollama `/api/generate` エンドポイントを直接叩く `OllamaGateway` を新規作成した。

ADR-579 の教訓に従い、全リクエストで `_BASE_OPTIONS`（`num_batch: 1024`, `num_keep: -1`, `stop: ["<end_of_turn>"]`）を統一し、モデルリロードを防止する。

### 4. search-indexer REST API に修正する

Meilisearch 直叩き (`POST /indexes/articles/search`) → search-indexer REST API (`GET /v1/search?q=...&limit=...`) に修正。実際に 20 件のエビデンス記事を取得できるようになった。

### 5. Ollama structured output でプランニング精度を向上する

ADR-632 の知見と Web 調査（Ollama 公式ドキュメント、ollama/ollama#15260）に基づき、以下を導入した。

- **Planner/Critic**: `format` パラメータで JSON スキーマを渡し、GBNF 文法で構造を保証
- **reasoning-first フィールド順序**: `reasoning` を JSON の先頭フィールドに配置（ADR-632）
- **temperature=0** で structured output の精度を最大化
- **num_predict=512** で planner/critic の出力を抑制（2000 → 512）
- Gemma4 では `think=false` を設定しない（`format` が無視されるバグ: ollama/ollama#15260）

### 6. Writer プロンプトを日本語指示に変更する

Writer ノードのプロンプトを日本語で記述し、「必ず日本語で書くこと」を明示。エビデンスの title と excerpt を列挙して Writer に渡す形式に改善。

### 7. フロントエンド改善

- **サイドバー**: `Sidebar.svelte` の `baseMenuItems` に "Acolyte Reports" を直接追加（`navigation.ts` は Sidebar から参照されていなかった）
- **フォント**: IBM Plex Mono を Google Fonts に追加、`app.css` で `--font-sans`, `--font-serif`, `--font-mono` を Tailwind v4 の `@theme` に直接値で設定
- **ランタイムエラー修正**: proto3 の空配列/空 int 省略に対応（`?? []`, `.get("currentVersion", 0)`）

## Consequences

### Pros

- **レポート生成が実際に動作する** — AIX Gemma4 26B で 3 セクション・12,767 文字のレポートを約 7 分で生成、PostgreSQL に永続化
- **search-indexer 連携** — 20 件のエビデンス記事を取得し、Writer に渡す
- **Atlas マイグレーション管理** — 再現性のある DB スキーマ管理、手動 SQL 実行を排除
- **Ollama structured output** — GBNF 文法で JSON 構造を保証、parse 失敗時のフォールバックも完備
- **モデルリロード防止** — ADR-579 の `_BASE_OPTIONS` 統一で TTFT 劣化を回避

### Cons / Tradeoffs

- **Planner の `format` が Gemma4 thinking モードで不安定** — `response_len: 0` になるケースがあり、フォールバック（3 セクション固定）に頼る場面がある。Ollama の Gemma4 structured output 対応の成熟を待つ
- **パイプライン所要時間** — 26B モデルで 7-10 分。ユーザーは非同期で待つ必要がある。StreamRunProgress は未実装
- **search-indexer のクエリ品質** — gatherer が scope の `topic` をそのまま検索クエリにしている。Query Expansion（rag-orchestrator のパターン）を導入すれば精度が上がる
- **日本語プロンプト** — Gemma4 26B の日本語生成品質は英語より劣る可能性。出力品質の評価基盤が未整備
