# ADR-000891: Acolyte レポート生成を /new から auto-chain しつつ active_run で polling 状態を復元する

## Status

Accepted

## Date

2026-05-05

## Context

ユーザーから「Acolyte で作ったレポートが『待ち』のまま生成されない」という報告を受け、acolyte-orchestrator / acolyte-db / nginx / alt-butterfly-facade / alt-frontend-sv のログ / DB を 24h 分突き合わせた結果、独立した 2 件の挙動不整合が確定した。

### 計測 (2026-05-05 12:30 UTC tail)

| 観測点 | 数 |
|---|---|
| nginx access log の `CreateReport` 呼び出し | 1 件 (06:05:39 UTC, report_id `f96cd475-…`) |
| 同 `StartReportRun` 呼び出し (24h) | **0 件** |
| acolyte-orchestrator の `StartReportRun` ハンドラ突入 (24h) | **0 件** |
| acolyte-db `report_runs` / `report_jobs` における該当レポートの行数 | **0 件** |
| 詳細ページ訪問 (`GET /acolyte/reports/<id>`) | 4 回 (06:05–07:16 UTC) |

### Issue #1 — Create と Generate が手動 2 段階だった

- `acolyte-orchestrator/acolyte/handler/connect_service.py:65-75` の `create_report` は `reports` (+ optional `report_briefs`) のみ INSERT し、`report_runs` / `report_jobs` を enqueue しない設計だった。
- `StartReportRun` は別 RPC で、FE 側では `/acolyte/reports/[id]/+page.svelte` の「Generate」ボタンを **ユーザーが押下** したときのみ発火する。
- `/acolyte/new/+page.svelte` の submit ハンドラは `createReport` のみ呼び、`goto` で詳細に遷移するだけだった。`No content yet … Click Generate` の空状態案内は出るものの、ユーザーは「create したのだから走っている」と誤解しやすく、結果として永久に pending 状態になる事故が定常化した。

### Issue #2 — polling 状態がページ遷移で消える

- `+page.svelte` の `activeRunId` / `runStatus` / `pollTimer` は `$state` でコンポーネント内に閉じており、`onMount(loadReport)` は `report` / `sections` / `versions` のみ取得。in-flight な run を再アタッチする経路がなかった。
- バックエンドは run を継続して走らせているが、FE が `getRunStatus(runId)` を呼ぶための runId を失うため、ユーザーから見ると進行不能と区別できない。
- `GetReportResponse` には active な run を返すフィールドがなく、proto 上に source of truth がなかった。

直近の commit 群 (`168c6c1d2 / dc329dc81 / 900a71fa9 / 8105366a9`) は news-creator の morning-letter prompt と recap-worker の budget cap が対象で、Acolyte の経路には影響しない。

## Decision

以下の 2 段階修正を採用する。

### PR1: FE 側で CreateReport→StartReportRun を auto-chain する

- `alt-frontend-sv/src/lib/connect/acolyteAutostart.ts` に `createAndAutostart(deps, title, type, scope)` を追加する。`createReport → startReportRun → goto(/acolyte/reports/{id}?run={runId})` の 3 段を、依存注入された関数で実行するピュア関数とする。
- `StartReportRun` が失敗した場合は `goto(/acolyte/reports/{id}?autostart_failed=1)` で詳細ページに降ろし、既存の手動 Generate ボタンで retry できるようにする (report 自体は作成済みなので削除はしない)。
- `acolyteAutostartParams.resolveAutostartIntent(URLSearchParams)` を導入し、URL の `?run=` / `?autostart_failed=1` を `{kind: "resume" | "autostart-failed" | "none"}` に解決する。詳細ページの `onMount` で消費する。
- terminal 状態 (`succeeded` / `failed` / `cancelled`) に到達したタイミングで `goto(..., {replaceState: true})` により URL から `?run=` を剥がす。リロード時にゾンビ runId を再 attach しないため。

### PR2: GetReport に active_run を加えて backend を真理値にする

- `proto/alt/acolyte/v1/acolyte.proto` の `GetReportResponse` に `optional ReportRun active_run = 3;` を追加する。proto3 optional のため後方互換。
- `acolyte/port/job_queue.py` に `JobQueuePort.get_active_run_for_report(report_id)` を追加し、`postgres_job_gw.py` / `memory_job_gw.py` で実装する。Postgres 実装は既存の partial index `idx_report_runs_active` を流用し、`WHERE run_status IN ('pending','running') ORDER BY started_at DESC NULLS LAST, run_id LIMIT 1` で running > pending の優先付けを行う。
- `acolyte/handler/connect_service.py:get_report` で `JobQueuePort` が注入されているときだけ active run を取り、`response.active_run` に詰める。テスト用に job_queue=None のときは silently skip する後方互換を維持する。
- FE は `acolyteAutostartParams.resolveResumeIntent(params, activeRun)` を導入し、`backend > URL > none` の優先順で polling を再開する。これにより:
  - タブを閉じて再訪問しても backend が in-flight を返すので polling が復元する。
  - `?run=` の URL hint は PR1 経路の互換のために残し、active_run が absent のときのフォールバックとして機能する。

### 検討した代替案と却下理由

| 案 | 却下理由 |
|---|---|
| **A. Backend の create_report を atomic に run + job まで insert** | proto 上 `CreateReportResponse.run_id` を追加すると AcolyteService の他 RPC との semantics 乖離が大きい。さらに「create したが scope を編集してから走らせたい」将来要件 (現状は brief 編集 RPC が無いので発生していないが) を排除してしまう。今回は FE-only で auto-chain することで proto 互換を最大化した。 |
| **B. localStorage に runId を保存して復元** | 同一ブラウザ前提で多タブ / 別端末 / clear-storage で破綻する。バックエンドは既に `report_runs.run_status IN ('pending','running')` の partial index を持っているので、サーバを source of truth に固定する方が cohesive。 |
| **C. SvelteKit の `goto({ state })` で runId を渡す** | リロードで失う。`?run=` query string は visible な代わりに reload-safe で、加えて `replaceState` で剥がせば履歴も汚さない。 |
| **D. `RunStatusPill` と URL の二重表示で混乱を避けるため `?run=` を完全廃止し active_run のみ採用** | PR2 のみで完結させると PR1 での auto-chain navigation 後、初回 GetReport が active_run を返すまでに一拍ずれる (StartReportRun 直後はまだ run が `pending` なので返るはず、だが FE のレース条件を考えると URL hint は冗長保険として有用)。両方残すコストは小さく、PR1 の振る舞いが PR2 の deploy 順序に依存しなくなる利点がある。 |

採用する: PR1 で FE-only auto-chain、PR2 で `GetReportResponse.active_run` を真理値とし、FE は `backend > URL > none` で polling を再開する。

## Consequences

### Pros

- ユーザーが「Generate を押し忘れる」事故が構造的に消える。`/acolyte/new` の submit が即生成パイプラインを走らせる。
- リロード / タブ切替 / ブラウザ再起動を跨いでも polling 状態が backend から再アタッチされるため、ユーザーが「まだ走っているのか終わったのか」を見失わない。
- proto 変更は optional 追加のみで `buf breaking` clean。alt-backend / alt-butterfly-facade / rag-orchestrator / alt-frontend-sv の既存 consumer は無変更で動作する。
- backend の `idx_report_runs_active` が既に partial index として存在するため、`get_active_run_for_report` のクエリコストは O(1) 相当 (テーブルが大きくなっても劣化しない)。
- FE のロジックはピュア関数に切り出されているため node 環境の vitest だけで covered (browser 不要)。E2E は Playwright `page.route()` で LLM をモックすることで秒単位で完走する。

### Cons / Tradeoffs

- Auto-chain により「create だけしておいて、後で scope を編集してから走らせる」UX は閉じる。現在 brief を編集する UI も RPC も無いので影響はないが、将来 `UpdateReportBrief` を入れたときに「Drafts only / Run after edit」モードを追加する必要がある。
- `?run=` URL query は visible な runId として共有 URL に乗ってしまう。run_id は authz 上の secret ではないが、運用上 URL を Slack 等に貼ると一瞬 metadata が漏れる。terminal で `replaceState` する設計でこれを最小化したが、polling 中に共有された URL は runId を含む。
- 既存 9 件の `current_version=0` で run が枯れたレポート (今回スコープ外) は依然として手動 Generate が必要。`f96cd475-…` を含む。
- `GetReport` のレスポンスサイズが optional `ReportRun` 分だけ増える (≈ 200B max)。詳細ページのみで使われるため帯域影響は無視できる。
- acolyte-orchestrator は run 終了時 `report_runs.run_status = 'succeeded' / 'failed'` に UPDATE するだけで row を残す設計。今回の調査で発覚した「過去 9 件のレポートに対し `report_runs` が 0 件」状態については、コード経路に DELETE は無く、過去手動運用 (TRUNCATE 等) によるものと推定する。本 ADR では復旧しない。
