# ADR-000857: recap-subworker の既定 classifier path を 30-class に切替し threshold coverage を fail-closed で保護する

## Status

Accepted

## Date

2026-04-27

## Affected Services

- recap-subworker (Python): `infra/config.py` / `infra/telemetry.py` / `services/classifier.py` の修正、既定 classifier path 切替、threshold coverage validator 追加、`CLASSIFIER_TOP_GENRE_TOTAL` Counter 追加。
- recap-worker (Rust): 上流 classifier の出力多様化を受けて `no_evidence` / `Missing from batch response` が減少する。コード変更なし。
- alt-frontend-sv: 3-day recap UI のジャンル数が回復する。コード変更なし。

## Context

2026-04-14 から 3-day recap のジャンル数が大幅に減少した。直接トリガは同日の mTLS 切替 ([[000827]] / PM-2026-032) だったが、4/22 の hot-reload 修正後もジャンル数は戻らず、4/23 以降に新症状として `recap_failed_tasks` に次のエラーが大量発生した。

- `Missing from batch response` × 16-19 件/ジョブ
- `Clustering failed: clustering failed: genre=<G>_NNN articles=1`
- `No genres were stored: failed=N, skipped=0, no_evidence=21`

ライブ DB 調査で次が判明した。

1. **既定 classifier path が古い 17-class joblib を指していた**
   - `recap_subworker/data/genre_classifier_ja.joblib` は 2026-04-23 に再学習された **30-class**（sklearn 1.8.0、`.meta.json` で確認）。
   - しかし `recap_subworker/app/container.py:152` と `services/classification_worker.py:92` は `settings.genre_classifier_model_path` を読んでおり、その既定値は `data/genre_classifier.joblib`（17-class、旧版）。
   - 結果として **production は 17-class モデルを継続使用**しており、再学習成果が反映されていなかった。
   - 既存の cardinality validator ([[000835]] stage 3) のデフォルト baseline 15 では 17 ≥ 15 で素通り。

2. **threshold coverage に静かなフォールバックが存在した**
   - 現行 `services/classifier.py:392` は `self.current_thresholds.get(cls, 0.5)` で、未登録クラスはハードコード 0.5 を採用していた。
   - 旧 `data/genre_thresholds.json`（17 キー）が dev 環境のフォールバックとして居残っていた。production は `genre_thresholds_path_ja`（30 キー）を明示するため害は限定的だが、再発時のリスクが残っていた。

3. **classifier 出力分布のメトリックが無かった**
   - top-1 が `consumer_tech` / `politics_government` に張り付いても、`recap_outputs` には「保存されたもの」しか出ない。失敗側の分布を観測できないため、4/14 の縮退から本因特定まで 13 日かかった。

並行調査で tag-generator と alt-backend の HTTP/proto 経路は健全（直近 3 日 untagged 5 件、avg 10 tags/article、`tag_set_versions` が 200-460 件/日生成）であり、tag-generator skip-loop は本件と独立であると確認した。

## Decision

recap-subworker に対して以下を採用する。**実装のみで再学習は不要** — ユーザー側で 30-class joblib は 4/23 に学習済みのため、その artefact を実際に読ませる経路に修正する。

1. **既定 classifier path の切替**
   - `recap_subworker/infra/config.py` の `genre_classifier_model_path` 既定値を `data/genre_classifier.joblib` から `data/genre_classifier_ja.joblib`（30-class）に変更し、`RECAP_SUBWORKER_GENRE_CLASSIFIER_MODEL_PATH` で上書き可能にする。
   - 旧 17-class artefacts (`genre_classifier.joblib` / `genre_classifier_model.joblib` / 17-key `genre_thresholds.json`) は `data/backup/` に退避してトレーサビリティを維持。

2. **cardinality baseline を 15→30 に引き上げ**
   - `genre_baseline_cardinality` の既定を 30 にし、再縮退時は CrashLoop で即可視化する。`RECAP_SUBWORKER_GENRE_BASELINE_CARDINALITY` で運用時の緩和は引き続き可能。

3. **threshold coverage validator の追加**
   - `Settings.model_validator(mode="after")` で、`classifier.classes_` 全クラスが `genre_thresholds_path_ja` に entry を持つことを起動時に検査。欠損があれば `ValueError` で boot 失敗。再学習でクラス集合を増やしたのに thresholds の更新を忘れる、という「サイレントな半端デプロイ」を防ぐ。
   - learning_machine backend / file 不在 / 非辞書 payload はバイパス（既存 validator と整合）。

4. **`CLASSIFIER_TOP_GENRE_TOTAL` Prometheus Counter の追加**
   - `recap_subworker/infra/telemetry.py` に `Counter("recap_subworker_classifier_top_genre", labelnames=("genre",))` を新設。
   - `services/classifier.py` の `_record_top_genres(results)` で `predict_batch` 出力を per-prediction に increment。空 / 欠落 label は drop して registry を汚さない。
   - Grafana で `count(rate(recap_subworker_classifier_top_genre_total[1h]) > 0)` を取れば「直近 1h で active な genre 数」が直視できる。

### 検討した代替案と却下理由

- **(却下) 17-class joblib をその場で 30-class に置き換え、コード変更しない**
  - artefact は既に 30-class が同じディレクトリに存在しており、コード側がそれを指していないだけだったので、ファイル swap よりコード側既定を直すほうが原因の所在が明確になる。
  - `genre_classifier.joblib` という名前自体が language-agnostic に見える誤解を招くため、`_ja` 接尾辞付きを既定にした方が将来の language routing にも素直。

- **(却下) `transformers<5` への pin**
  - 既存の 30-class joblib は `transformers 5.6.1` で学習済み（`.meta.json`）。downgrade すると embedding 分布が一致しなくなり、新たな破損を生む可能性がある。pin はリトレインと併走でしか安全に出来ないため、本 ADR では実施しない。
  - 観測のための Counter 追加とコードレベルの fail-closed 化を優先した。

- **(却下) `current_thresholds.get(cls, 0.5)` のハードコード fallback を撤去し例外化**
  - 既存呼び出し経路（学習スクリプト、過去のリプレイ等）への影響範囲が広い。代わりに boot 時 validator で「fallback が必要な状態」自体を不可能にする方針を選んだ。

## Consequences

### Pros

- production が **実際に 30-class classifier を使う**ようになり、`no_evidence` / `Missing from batch response` の主因が解消する。3-day recap UI のジャンル数が再度 30 近辺まで戻る前提条件が整う。
- threshold coverage validator が boot 時に効くため、再学習でクラスを増やすと thresholds 同期忘れが即 CrashLoop で検出される。サイレントなジャンル縮退が再発しても 13 日ではなく数分で発覚する。
- `CLASSIFIER_TOP_GENRE_TOTAL` で classifier 出力分布が dashboard 化できるため、pipeline 下流（`recap_outputs`）に到達する前に異常が見える。
- cardinality baseline 30 化で「retraining が古いまま reverted された」事故も同様に boot 時に止められる。
- 旧 17-class artefacts は削除でなく `data/backup/` に退避したので、緊急時 rollback パスが残っている。

### Cons / Tradeoffs

- 既定 classifier path を変えたため、`genre_classifier.joblib` を直接参照する dev / 試験環境がもしあれば壊れる。`RECAP_SUBWORKER_GENRE_CLASSIFIER_MODEL_PATH` で明示設定する必要がある（既存 production compose は環境変数を上書きしていないため影響しない）。
- threshold coverage validator は `genre_thresholds_path_ja` を真ソースとして JA 側のみ検査する。EN 側の coverage は別経路（`genre_thresholds_path_en`）で同様にカバーするのは将来課題。
- `transformers v5 buffer-junk` 問題（HF issue [#44534](https://github.com/huggingface/transformers/issues/44534)）の根治は未着手。再学習成果の品質が embedding 偏りに左右されるリスクは残る。pin と再学習は別 ADR / 別 PR で扱う。
- recap-evaluator BertScore モデル不在 (`cl-tohoku/bert-base-japanese-v3`) は別件として未解決。
