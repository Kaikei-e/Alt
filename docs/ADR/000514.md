# ADR-000514: 要約 job queue を運用キューと履歴に分けて扱う

## Status

Accepted

## Date

2026-03-21

## Affected Services

- pre-processor (Go)

## Context

`summarize_job_queue` には `pending` / `running` / `completed` / `dead_letter` が同居している。
しかし運用上は、この 4 状態をすべて同じ「backlog」として見なすと実態を誤認しやすい。

実データでは、要約済みの terminal row が大量に蓄積しており、未処理件数よりも履歴件数のほうが目立つ状態になっていた。
その結果、queue 総件数だけを見ても「今どれだけ処理が詰まっているのか」「どこを掃除すべきか」が分からなくなっていた。

上流イベントの意味論修正 ([[000512]]) と queue 作成 guard ([[000513]]) を入れても、既に溜まっている terminal row をそのまま backlog と解釈すると、運用判断を誤る。
このため queue テーブルの状態を、**処理対象の仕事量** と **監査・履歴のための記録** に分けて扱う必要があった。

## Decision

**`summarize_job_queue` は運用キューと履歴を兼ねるテーブルとして扱い、`pending` / `running` と terminal row を明確に分離して運用する。**

具体的には以下を採用する。

1. `pending` と `running` だけを live backlog とみなす
2. `completed` と `dead_letter` は履歴 row とみなし、処理対象件数には含めない
3. 監視・報告・手動監査では、live backlog と terminal row を必ず別々に集計する
4. terminal row は保持要件に応じて purge または archive の対象にできる前提で扱う

これにより、queue 総件数ではなく「今処理すべき件数」と「残っている履歴件数」を分離して判断できるようにする。

### 採用しなかった代替案

| 案 | 評価 |
|---|---|
| queue 総件数をそのまま backlog と見なす | `completed` が多い期間に実態より深刻に見え、運用判断を誤る |
| terminal row を即時削除して履歴を持たない | 監査や不具合調査に必要な情報まで失う |
| `completed` だけ別テーブルへ即時移送する | 現時点では運用改善より実装コストが先に立つ |

## Consequences

### Pros

- backlog と履歴を混同しなくなる
- queue の異常増加が「未処理増加」か「履歴蓄積」かを切り分けやすくなる
- cleanup 判断を `pending` 中心に行える

### Cons / Tradeoffs

- queue 総件数だけを見る単純な監視は使いにくくなる
- 履歴 retention の運用ルールを別途定める必要がある
- 1 テーブル内に二つの意味が残るため、SQL やダッシュボード側の配慮が必要になる
