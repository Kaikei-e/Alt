# ADR-000482: Knowledge Home UI の Phase 1〜5 骨格実装と状態分解

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-frontend-sv (SvelteKit / Svelte 5 Runes)

## Context

Knowledge Home は単なる記事一覧ではなく、検索・Recap・RAG・TTS・supersede/why を統合した「読む前の判断面」として設計されている。`alt_knowledge_home_phase_plan.md` で定義された 7 フェーズのうち、Phase 1〜5 に対応する UI 骨格が未実装、もしくは不十分な状態であった。

具体的には以下の課題が存在した:

1. **空状態の混同** — `KnowledgeHomeEmpty` が「データなし」と「新着処理中」を区別できず、ユーザーに誤った認識を与える
2. **TodayBar の脆弱性** — digest が null の場合に何も表示せず、silent failure となる
3. **ページ状態の欠如** — `useKnowledgeHome` フックが loading / error の二値のみで、degraded / fallback / refreshing を区別できない
4. **DegradedModeBanner の閉じ不可** — 一度表示されると閉じる手段がない
5. **RecallRail の unavailable 未対応** — 0 件と機能無効を区別しない
6. **StreamUpdateBar の接続状態非表示** — pending 0 のときに接続断やフォールバックが見えない
7. **KnowledgeStream の mode 未対応** — lens / search / recap_context による動作の分岐がない
8. **WhyPanel の不在** — why バッジはあるが展開して詳細を見る手段がない
9. **SupersedeDetail の理由文欠落** — 何が変わったかは示すが、なぜ変わったかを説明しない

## Decision

`alt_knowledge_home_phase_plan.md`、`alt-knowledge-home-ui-block.md`、`alt-core-value-and-position-extend-plan.md` の 3 文書に準拠し、Phase 1〜5 の UI 骨格を TDD-first で一括実装する。

### Phase 1: 新着流入の常設化

- `KnowledgeHomeEmpty` に `reason` プロパティを追加し、`no_data` / `ingest_pending` / `lens_strict` / `hard_error` の 4 状態を区別する
- `TodayBar` に null digest 時のローディングプレースホルダーと、fallback 時の明示的な不可メッセージを追加する
- `KnowledgeCard` の `summaryState = pending / missing` 時のスケルトン表示を検証テストで担保する

### Phase 2: Lens の実効化

- `LensSelector` に `matchCount` プロパティを追加し、アクティブ lens 選択時にマッチ件数を表示する
- `KnowledgeStream` の空状態表示に `emptyReason` を渡し、lens 固有の空メッセージを出し分ける

### Phase 3: TodayBar / Recall / Home read model の整流化

- `useKnowledgeHome` フックに `pageState` 状態機械を導入する。状態は `initial_loading` / `ready` / `refreshing` / `degraded` / `fallback` / `hard_error` の 6 値とする
- 同フックに `emptyReason` 算出プロパティを追加し、items・error・lens・digest の組み合わせから空状態の理由を自動判定する
- `DegradedModeBanner` に `onDismiss` プロパティを追加し、ユーザーが明示的に閉じられるようにする
- `RecallRail` に `unavailable` プロパティを追加し、0 件と機能無効を区別する
- `+page.svelte` を `pageState` / `emptyReason` / `serviceQuality` で駆動するよう全面更新する

### Phase 4: Stream Updates の実用品質化

- `StreamUpdateBar` に pending 0 時の接続状態表示を追加する。切断中は「Reconnecting...」、フォールバック時は「Live updates unavailable」を表示する
- `KnowledgeStream` に `streamMode` (`default` / `lens` / `search` / `recap_context`) と `degradedNote` プロパティを追加する

### Phase 5: Supersede UX の end-to-end 完成

- `WhyPanel` コンポーネントを新規作成する。why 理由を source / behavior / semantic / change にカテゴリ分類し、理由が空の場合は「Matched by general relevance」のフォールバックを表示する
- `SupersedeDetail` に状態別の変更理由テキスト（「The summary was regenerated with updated content.」等）を追加する

## Consequences

### Pros

- **空状態の明確化** — ユーザーは「本当に空なのか」「処理中なのか」「lens のせいか」「エラーか」を即座に判別できる
- **状態機械の導入** — ページ全体の状態が 6 値に集約され、条件分岐が宣言的になる。各コンポーネントは `pageState` に基づいて描画を決定すれば良い
- **TDD による品質担保** — 全変更に対してテストが先行しており、635 テストが全パス。リグレッションリスクが低い
- **段階投入可能** — 各 Phase の変更は feature flag (`enable_recall_rail`、`enable_lens`、`enable_stream_updates`) で個別制御可能
- **backend 契約との整合** — UI が独自解釈する状態はなく、`serviceQuality`、`digestFreshness`、`summaryState` はすべて backend-authoritative な値を参照する

### Cons / Tradeoffs

- **ブラウザテストの型不整合** — `vitest-browser-svelte` の `render` 関数と Svelte 5 の型推論の相性問題で、ブラウザテストファイルは `as never` キャストが必要。これは vitest-browser-svelte のアップデートで解消される見込み
- **Phase 6〜7 は未着手** — Reproject / Backfill / SLO の admin UI 連携と E2E テスト網羅は本 ADR のスコープ外。Phase 6 以降は backend 契約の成熟を待って着手する
- **RecapBlock / SearchEntry / AskSheet は先送り** — 対応する backend 契約が未確定のため、UI のみ先行する実装は行っていない。これは計画どおりの判断
