# ADR-000847: Knowledge Loop の stream-driven refresh をコアレス化し foreground 退場を depth-layered transition に置き換える

## Status

Accepted

## Date

2026-04-26

## Affected Services

- **alt-frontend-sv (SvelteKit 2 / Svelte 5)** — 以下を変更
  - `src/routes/(app)/loop/+page.svelte` — stream callbacks の `invalidateAll()` 撤去、`makeCoalescedRefresh(() => invalidate("loop:data"))` に置換、`onDestroy` で dispose、`#each foreground` に `animate:flip` + `out:loopRecede` 配線、`.foreground-tiles` に `transform-style: preserve-3d`、SSR snapshot を reference-equality 経由で hook に push する `$effect` 追加
  - `src/routes/(app)/loop/+page.server.ts` — `depends("loop:data")` 登録
  - `src/lib/hooks/useKnowledgeLoop.svelte.ts` — `dismiss()` を `entries.filter(...)` ベースの optimistic-remove に変更、`replaceSnapshot(next)` 追加、`optimisticallyDismissed` Set で server projection lag 中のフラッシュバックを防御
  - `src/lib/hooks/useKnowledgeLoopStream.svelte.ts` — `import { untrack } from "svelte"` 追加、`trackedLensId = $derived.by(() => opts.lensModeId)` で値等価ゲート、`untrack(() => { stopped=false; retryCount=0; void connect(); })` で `data`-reference churn による再 subscribe を抑止
  - `src/lib/components/knowledge-loop/LoopEntryTile.svelte` — `max-height: 640px` clamp 削除、`.dismissing` の `max-height: 0` keyframe path 削除、`.exit-pending` を pointer-events 専用ガードに残置、reduced-motion media query から `.dismissing` 行も除去
  - 新規 `src/lib/hooks/loop-coalesce.ts` — 600 ms trailing debounce + single-flight + dispose 可能な refresh コアレッサ (純粋 factory)
  - 新規 `src/lib/transitions/loop-recede.ts` — Svelte `out:` カスタム transition (translateZ −72 + scale 0.965 + opacity 1→0 + saturate 0.85→1.0 + brightness 0.99→1.0)、`prefers-reduced-motion` 時は dissolve + saturate のみ
  - テスト: 新規 `src/lib/hooks/loop-coalesce.test.ts` (6 ケース)、`useKnowledgeLoop.svelte.test.ts` 拡張 (`dismiss → entries 削除` / `replaceSnapshot` / `optimistic-dismissed の蘇生防止`)、新規 `tests/e2e/desktop/loop/loop-fetch-storm.spec.ts`、新規 `tests/e2e/desktop/loop/loop-dismiss-no-overlap.spec.ts`

proto / contract / migration 変更なし。alt-backend / knowledge-sovereign / migrations-atlas は触らない。

## Context

### 観測された症状

production の `/loop` ページで Dismiss クリック後にレイアウトが崩れ、ブラウザコンソールに大量の `GET /loop/__data.json?x-sveltekit-invalidated=1 net::ERR_INSUFFICIENT_RESOURCES` と `TypeError: Failed to fetch` が出力されていた。

ライブログ (compose stack, 2026-04-26 04:30 UTC):

- `nginx`: 1 client IP から `GET /loop/__data.json?x-sveltekit-invalidated=1` が秒間多数 (request rate `rt=0.032-0.035` で連続)
- `alt-butterfly-facade`: `KnowledgeLoopService/GetKnowledgeLoop` が 04:30:31 の同一ミリ秒に 50+ fan-out (proxy_request / proxy_response が同期発火)
- `alt-backend`: `alt.knowledge_loop.stream_jwt_expired` が同一 `user_id` に対して 04:30:58 / 04:31:28 / 04:31:59 と 30 秒周期で 30+ 件 lockstep 発火

実害は service down ではなく UX 完全破綻 (再描画ループでメインスレッド飽和、Dismiss CSS transition が完了せず複数カードが Y 軸で重なって描画される、クリック不能)。ヒヤリハット級。

### 直接原因

`+page.svelte` で `useKnowledgeLoopStream` の `onFrame` と `onExpired` が両方とも無条件に `invalidateAll()` を呼んでいた:

1. SSR-issued JWT が near-expiry な状態で stream open
2. server が即座に `stream_expired` envelope を送信
3. `onExpired` → `invalidateAll()` → SSR `data` 全替え
4. `useKnowledgeLoopStream` の `$effect` は `opts.lensModeId` を `data` reference 経由で読んでいたため、`data` 参照が変わると再 fire
5. cleanup `disconnect()` → `connect()` で stream 再オープン
6. 同じ near-expiry JWT を使い回すため再び `stream_expired`
7. step 3 へ戻る (positive feedback loop)

並行する `AbortController.abort()` の TCP teardown が完了する前に新規 fetch が積み上がり、ブラウザの per-origin connection ceiling に到達して `ERR_INSUFFICIENT_RESOURCES`。

### 二次症状: foreground カード重なり

`useKnowledgeLoop.applyLocalDismiss()` は `dismissState` フラグを `"dismissed"` に書き換えるだけで配列に残置していた。`LoopEntryTile.svelte` 側の `.entry.dismissing { max-height: 0 }` で `transition: max-height 160ms` を回す設計だったが、

- `.entry` には `max-height: 640px` の固定 clamp があり (`overflow: visible` 既定) コンテンツが溢れて隣のグリッド行に bleed
- fetch-storm がメインスレッドを奪うため CSS transition の rAF が回らずアニメ未完
- `#each` block に `animate:flip` も `out:` も無いため survivors は静止、leaver は半 collapse のまま停止

結果としてユーザのスクリーンショットでは 3 件の ORIENT カードが Y 軸で重なって描画されていた。

### Canonical contract 上の要件 ([[knowledge-loop-canonical-contract]])

- §11 flat-text invariant: depth は tile / container / transition にのみ適用、テキストは flat
- §12 spatial render contract: deeper-focus は奥へ、return は手前、changed diff は層分離
- §12.5 reduced-motion mapping: depth simulation を停止し dissolve + highlight fade + color shift に置換
- §9 stream semantics: revised / heartbeat は silent、appended / superseded / withdrawn / rebalanced は refetch hint
- §3 reproject-safe: client は projection state を mutate しない

## Decision

### 戦略 1: stream-driven refresh をコアレス化する

`invalidateAll()` を `invalidate("loop:data")` に置換し、その呼び出しを debounce + single-flight でラップする。

- `+page.server.ts` の `load` で `depends("loop:data")` を宣言
- 新規 `makeCoalescedRefresh(refresh, { windowMs: 600 })` を `+page.svelte` で構築し `onDestroy` で dispose
  - 600 ms trailing-edge debounce: バースト中の trigger は最後の trigger から 600 ms 後に 1 回だけ refresh を起動
  - single-flight: in-flight 中に到来した trigger は 1 つの trailing follow-up にコアレス
  - dispose: pending timer / trailing follow-up を即キャンセル
- `onFrame` は revised / heartbeat を silent skip (§9 準拠)、それ以外で `coalescedRefresh.trigger()`
- `onExpired` は **何もしない** — `useKnowledgeLoopStream` 側の `scheduleReconnect` がリコネクトを担うため SSR refresh は不要 (次の non-silent frame で coalesce 経路が起動する)

### 戦略 2: stream hook の `$effect` を値等価で再応答制御

`useKnowledgeLoopStream` の `$effect` は `opts.lensModeId` を `$derived` 経由で track し、cleanup→reconnect の副作用は `untrack` で囲む:

```ts
const trackedLensId = $derived.by(() => opts.lensModeId);

$effect(() => {
    const enabled = opts.enabled;
    const _lens = trackedLensId; // 値等価で再 fire
    if (!enabled) return;
    untrack(() => {
        stopped = false;
        retryCount = 0;
        void connect();
    });
    return () => disconnect();
});
```

これにより `data` の参照差し替え (例えば `invalidate("loop:data")` の re-run 後) では effect は再 fire しない。`lensModeId` の値が実際に変わったときのみ stream を張り直す。

### 戦略 3: dismiss を optimistic-remove + parent-level exit transition に変える

`useKnowledgeLoop.applyLocalDismiss()` は entry を配列から filter で削除、別途 `Set<string>` (`optimisticallyDismissed`) で削除済みキーを記録する。`replaceSnapshot(next)` は projection lag 中の server 応答に対して dismissed key をフィルタアウトしてから `entries` を差し替える。これにより stream-driven refresh で削除済みカードが一瞬蘇って消える flash を防ぐ。

`+page.svelte` の `#each foreground` ブロック側に exit transition を実装する:

```svelte
{#each foreground as entry, i (entry.entryKey)}
    <div class="foreground-row"
         animate:flip={{ duration: 240, easing: cubicOut }}
         out:loopRecede={{ duration: 280 }}>
        <LoopEntryTile ... />
    </div>
{/each}
```

`.foreground-tiles` に `transform-style: preserve-3d` を追加して、各行が `loop-plane-root` の `perspective: 1200px` 配下で個別 3D context を持つようにする。

### 戦略 4: tile 側の max-height clamp と `.dismissing` 折り畳み path を撤去

`LoopEntryTile.svelte` から:

- `.entry { max-height: 640px; transition: ... max-height 160ms }` を削除 (clamp 自体が overflow bleed の原因。expanded コンテンツの高さは grid に任せる。本当に上限が必要なら将来 `.expand` のみに付ける)
- `.entry.dismissing { max-height: 0 }` keyframe path を削除
- `class:dismissing={dismissing}` を `class:exit-pending={dismissing}` に rename し、`.exit-pending { pointer-events: none }` だけ残す (Dismiss クリック中の二重発火を防ぐガード用途のみ)

### `loop-recede` transition の細部

`src/lib/transitions/loop-recede.ts` で Svelte の `out:` カスタム transition を提供する。`t` は 1 → 0 で進行、t=1 が静止、t=0 が完全退場:

```
transform: translateZ(-72 * (1-t) px) scale(0.965 + 0.035t)
opacity: t
filter: saturate(0.85 + 0.15t) brightness(0.99 + 0.01t)
duration: 280 ms, easing: cubicOut
```

`prefers-reduced-motion: reduce` 時は §12.5 マッピングに従い、translateZ / scale を完全停止して `opacity` + `saturate` のみで dissolve + highlight fade を表現 (duration 200 ms に短縮)。Alt-Paper の禁則 (no shadow, no slide, flat glyph) に違反しない。

### 検討した代替案と却下理由

1. **`invalidateAll()` を残し debounce のみ被せる**: SvelteKit の `invalidateAll` は全ての `+layout.*` / `+page.server` の load を走らせるため、`/loop` 以外の上流に副作用が出る。`invalidate("loop:data")` の方が scope を狭めつつ同効果。却下。
2. **stream onFrame で `loop.replaceSnapshot()` を直接呼ぶ (server 経由しない)**: stream は §9 で silent update / forced replace の区別を持つが、entry の本体データは含まれない (`appended`/`superseded` は entryKey + revision のみ)。本体は GetKnowledgeLoop 経由で取り直す必要があるため、SSR refresh path を経由する設計を維持。却下。
3. **stream を完全に切って polling に置き換える**: §9 contract と stream の existence は backend 側 ADR で確立済 (silent vs forced replace の expression は SSE が前提)。FE 側の bug を理由に契約を変えるのは過剰。却下。
4. **`max-height` clamp を残し overflow: hidden を付ける**: clamp 自体は機能要件ではなく defensive code だった (640 px を超える評価エビデンスを想定していたが、現状そんなコンテンツは無い)。clamp を残すと Z-recede 中に scrollbar が出る等の二次問題を生むため撤去が清浄。却下。
5. **JWT 寿命を backend で延ばす**: 根本治療として正しいが、本 ADR の scope 外 (alt-backend / knowledge_loop_usecase 改修)。FE 側の対症療法としても本 fix が独立に成立 (coalesce は JWT 寿命によらず正しい設計) ため後続トラックに分離。Out of scope (§Related に記載)。

## Consequences

### Pros

- **fetch-storm の構造的解消**: 600 ms 内に到来する N 個の trigger が必ず 1 回の refresh に collapse する。stream が暴走しても per-origin connection ceiling を踏まない。
- **JWT 短寿命に頑健**: `onExpired` が SSR refresh を起動しないため、stream hook の `scheduleReconnect` だけが再接続を担う。`data`-reference churn が止まったため `$effect` も再 fire しない。
- **ADR-000831 §11-13 を満たす exit transition**: tile 単位の Z-recede + dissolve + saturate 落とし。reduced-motion 時は §12.5 マッピング (dissolve + highlight fade + color shift) に正確に対応。
- **optimistic dismiss の挙動が直感的**: ユーザが Dismiss を押すと即座に DOM から消え、survivors が `animate:flip` で滑らかに上に詰まる。projection lag 中も flashback しない。
- **`max-height: 640px` clamp 撤去で content overflow が物理的に発生しない**: 評価エビデンスが長いケースでも grid 行が拡張するだけ。
- **TDD outside-in でテスト 2 層を追加**: Playwright fetch-storm spec が `__data.json` request count ≤ 2 を pin、Playwright dismiss-no-overlap spec が DOM 削除 + 残存 tile の Y 軸非重なりを pin、unit test が coalescer / dismiss → remove / replaceSnapshot を pin。

### Cons / Tradeoffs

- **stream で本体差分を即時反映する経路がない**: 今でも server 経由の round-trip 600 ms + GetKnowledgeLoop の RTT を待つ。stream が Append payload に entry 全体を含めれば skipable だが、現契約は revision pointer のみ。将来の最適化トラック。
- **`optimisticallyDismissed` Set が per-tab メモリで揮発する**: Dismiss 直後にユーザがリロードすると、projection が lag していれば一瞬カードが見える可能性。ただし backend 側 dedupe + projector は通常 1-3 秒以内に追従するため、実害は極小。永続化が必要になったら `localStorage` に逃がす (loop-observe-throttle と同じパターン)。
- **`loop-recede` transition は CSS の Z-translation に依存する**: 古い iOS Safari (16.0 未満) は `transform-style: preserve-3d` を nested grid で正しく描画しないバグ報告がある。reduced-motion path に落としているため最悪 dissolve のみで動作するが、見栄えは劣化。
- **`max-height` clamp 撤去で異常に長い評価エビデンスがあるとカード高が際限なく伸びる**: 現状の whyPrimary.evidence_refs は最大 8 件 (canonical contract §10) なので実害なし。将来 `.expand` 内に必要なら別途 clamp を入れる。

### 未解決の負債

- backend 側 stream JWT の lifetime チューニング (上記却下案 5)。FE 側 fix で UX は救えたが、backend が短寿命 JWT を発行し続けると stream hook の reconnect ループ (1 s → 15 s exponential backoff) は発生し続ける。`alt-backend/app/usecase/knowledge_loop_usecase` 側で JWT exp を 5 分以上に伸ばすか、`stream_expired` の送出を JWT exp 直前ではなく一定期間バッファするか、後続 ADR で議論。
- `LoopEntryTile.svelte.spec.ts` は依然 client (browser) project で `node:fs` を import していて load 失敗。本 ADR の前から存在する pre-existing issue だが、本変更で `.entry` の class が変わったため将来必ず触る。別 PR で server project に移すか、`vite-node` 経由で読む構造に直す。
