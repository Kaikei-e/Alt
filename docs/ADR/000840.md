# ADR-000840: Knowledge Loop transition を append-only event に接続し projection-diff stream と併せて Loop を実行エンジン化する

## Status

Accepted

## Date

2026-04-24

## Affected Services

- **proto** — `proto/alt/knowledge/loop/v1/knowledge_loop.proto` に `KnowledgeLoopObservedPayload` / `OrientedPayload` / `DecisionPresentedPayload` / `ActedPayload` / `ReturnedPayload` / `DeferredPayload` / `SessionResetPayload` / `LensModeSwitchedPayload` の 8 messages を additive で追加。既存 service / message / enum は無変更で `buf breaking` pass。
- **alt-backend (Go)** — `TransitionKnowledgeLoopUsecase.Execute` が idempotency reserve 後に sovereign `AppendKnowledgeEvent` を呼び、Loop event を `knowledge_events` に INSERT する。projector `KnowledgeLoopProjectorJob` が 8 種の Loop event を `knowledge_loop_session_state` 単体の UPSERT に投影（entry 行は生成しない）。`StreamKnowledgeLoopUpdates` が 5 秒 poll + 10 秒 heartbeat + 30 分 stale timeout の本実装に置換される。
- **alt-frontend-sv (SvelteKit)** — `useKnowledgeLoopStream` hook が `streamKnowledgeLoopUpdates` RPC を購読し、純関数 `classifyLoopStreamFrame` で proto oneof を narrow discriminated union に正規化する。`/loop` page が `onMount` で購読を起動し、非 silent frame 到達時に `invalidateAll()` で SSR snapshot を再取得する。
- **alt-butterfly-facade / rag-orchestrator** — 生成済み proto stubs を再生成するのみ。コード変更なし。

## Context

[[000831]] は Knowledge Loop を `proposedStage` / `currentSessionStage` / `surfaceBucket` の 3 軸分離で再定義し、canonical contract として [[knowledge-loop-canonical-contract]] を固定した。Phase 0〜M6 で schema / proto / projector skeleton / handler / BFF / `/loop` UI の縦通しは成立していたが、直近のレビュー（`review.md`, 2026-04-24）は実装実現度を v0.4〜v0.6 (55〜65%) と評価し、以下 2 つの核心ギャップを具体的に指摘した。

1. **Transition が event store に閉じていない** — `TransitionKnowledgeLoopUsecase.Execute` は `idempotency reserve → return Accepted: true` で完結していた。Loop 固有 event（`KnowledgeLoopObserved` 他 8 種）は proto にも `knowledge_events.event_type` にも未定義で、Loop は「永続的な行動履歴」になっていなかった。結果として reproject 時に Loop セッションの進行を再構築できない。
2. **Stream は placeholder** — `StreamKnowledgeLoopUpdates` は単一の `StreamExpired("not yet implemented in this milestone")` envelope を返すだけで、クライアントは停止状態のまま SSR snapshot と local optimistic update だけで「生きた Loop UI」を演出していた。

canonical contract §8 は 8 種の Loop event 名と発火条件、§9 は `EntryAppended/Revised/Superseded/Withdrawn/SurfaceRebalanced/StreamExpired` の update-kind を既に固定している。本 ADR はこの契約を実装側に落とし込むことで、「Loop が Alt の知識体験を駆動する **実行エンジン**」への移行を完了させる第一層（P1 + P2）を確定する。

定量的には、event append の欠落は即 projector の session-state 不整合（`CurrentStageEnteredAt` が埋まらない / `last_*_entry_key` の履歴が失われる）を招き、reproject runbook が成立しない。stream の未実装は、複数タブや同時 `/feeds` 操作の反映遅延を 5 秒〜分単位で発生させる（手動リロードに頼らざるを得ない）。

## Decision

Knowledge Loop を append-only event log に完全接続し、projection-diff stream と併せて「実行エンジン」化することを採用する。具体的には以下 6 点を固定する。

1. **proto additive 拡張** — `knowledge_loop.proto` に 8 新 payload messages を追加し、`knowledge_events.event_type` 文字列は `knowledge_loop.observed.v1` / `knowledge_loop.oriented.v1` / `knowledge_loop.decision_presented.v1` / `knowledge_loop.acted.v1` / `knowledge_loop.returned.v1` / `knowledge_loop.deferred.v1` / `knowledge_loop.session_reset.v1` / `knowledge_loop.lens_mode_switched.v1` とバージョン付き命名規約で付番する。`event_type` は列レベルでは自由文字列のままで、enum 化は別 ADR で判断する（現状のスキーマを壊さない）。各 payload には `entry_key` / `lens_mode_id` / `from_stage` / `to_stage` / `trigger` / `observed_projection_revision` / `client_transition_id` を乗せ、projector が latest projection state を読まずに session-state delta を計算できるよう reproject-safe にする。

2. **Transition usecase を event append に接続する** — `TransitionKnowledgeLoopUsecase.Execute` を `validate → classify → dedupe reserve → append event` に延伸する。`ClassifyTransitionEvent(fromStage, toStage, trigger) → event_type` は純関数で、ADR-000831 §7 の禁止遷移（`observe->act` / `decide->observe` / `act->act`）を副作用前に `ErrInvalidArgument` で reject する。append は既存の `knowledge_event_port.AppendKnowledgeEventPort`（sovereign Connect-RPC 経由）を再利用し、`dedupe_key = client_transition_id` で fast-path (`knowledge_loop_transition_dedupes`) と slow-path (`knowledge_events.dedupe_key` unique index) の idempotency を統一する。

3. **Single emission rule の実装化（ADR-000831 §3.8）** — `/loop` 起点の transition は Loop event のみを emit し、`HomeItemOpened` 等の Home event は emit しない。既存 `track_home_action_usecase` は `/feeds` 起点専用として据え置き、`/loop` 経路は新 Transition usecase から直接 append する。projector は同じ `knowledge_events` ストリームから両方を読むが、**entry の生成は Home event だけに許可し Loop event は session_state のみ更新する**ことで二重生成を防ぐ。

4. **Projector の Loop event handler** — `KnowledgeLoopProjectorJob` の event switch に 8 新 event_type を追加する。`projectLoopTransitionEvent` は `knowledge_loop_session_state` への UPSERT のみを emit する。`CurrentStageEnteredAt = event.OccurredAt` で event-time purity を満たし、`ProjectionSeqHiwater = event.EventSeq` を渡して sovereign 側 SQL の `WHERE session.projection_seq_hiwater <= EXCLUDED.projection_seq_hiwater` + `COALESCE(EXCLUDED.last_*_entry_key, session.last_*_entry_key)` guard で merge-safe にする。entry 行は生成しない（Single emission rule の実装レイヤー）。

5. **Stream 本実装（polling-based）** — `StreamKnowledgeLoopUpdates` の placeholder を、Knowledge Home stream ([[000749]] の tenant/lens boundary に整合) と同じ polling-based pattern で置換する。alt-backend は 5 秒ごとに `ListKnowledgeEventsSinceForUser(tenantID, userID, afterSeq)` を呼び、純関数 `ClassifyLoopStreamUpdate(event) → StreamUpdateFrame` で event 種別を `EntryAppended` / `EntryRevised` / `EntrySuperseded` / `EntryWithdrawn` / `SurfaceRebalanced` にマップする。`KnowledgeLoopObserved` は 60 秒 dwell で大量発火するため stream では suppress し flooding を防ぐ（canonical contract §9 の silent-update 原則にも整合）。10 秒 heartbeat / 30 分 stale timeout で `StreamExpired("stale")` を送出し、3 連続 fetch error で `StreamExpired("upstream_error")` を送出する。この handler は **read-only** で projection を一切 mutate しない（F3 契約）。

6. **Frontend stream 消費と SSR 再取得** — `alt-frontend-sv/src/lib/hooks/loop-stream-frames.ts` の純関数 `classifyLoopStreamFrame` が proto oneof を narrow discriminated union（`appended | revised | superseded | withdrawn | rebalanced | expired | heartbeat`）に写像する。`useKnowledgeLoopStream.svelte.ts` が Svelte 5 runes（`$state` / `$effect`）で指数バックオフ再接続と `onFrame` / `onExpired` コールバックを提供する。`/loop/+page.svelte` は `onMount` で購読を起動し、非 silent frame 到達時に SvelteKit の `invalidateAll()` で SSR snapshot を再取得する。`revised` と `heartbeat` は UI を動かさず、foreground tile の「気配り」だけを局所更新する（canonical contract §9 の silent-update 要件）。multi-tab leader election は MVP では省略し、server 側 rate limit で abuse を防ぐ方針を採用した。

### 検討した代替案

- **sovereign 側に `WatchProjectionCheckpoint` stream RPC を新設する** — より直接的に projection の変化だけを配信できるが、sovereign 側 RPC 新設と状態変化 publisher の導入が必要で本 PR のスコープを大きく超える。alt-backend 内 5 秒 polling で同じ UX 品質が得られることを Knowledge Home stream で確認済みなので、こちらを採用。将来 polling コストが問題化したら別 ADR で移行する。
- **Transition usecase が `HomeItemOpened` と Loop event の両方を emit** — 後方互換のため `/loop` から Home event も投げる案。却下。ADR-000831 §3.8 の single emission rule を破ると、同一ユーザ意図で 2 event が append され重複処理・レポート歪み・dedupe ロジックの複雑化を招く。`/loop` の Loop event だけを真実源泉とし、Home 側の entry は Home 経路の event から独立に生成される原則を維持する。
- **event_type を enum 化して proto に固定する** — スキーマ強制は強まるが、`knowledge_events.event_type` は全サービス横断の共有列で変更コストが高い。versioned 文字列命名規約（`knowledge_loop.observed.v1`）で将来の payload schema 変更にも後方互換を保てるため据え置く。enum 化の是非は DB schema migration と併せて別 ADR で判断する。
- **`KnowledgeLoopObserved` を stream に流す** — dwell 60 秒窓で発火するため frame/min の洪水になり、BroadcastChannel 帯域と UI の `invalidateAll()` を支配してしまう。suppress を採用（contract §9 の silent-update 原則にも整合）。
- **Stream handler 内で `time.Now()` を business fact として使う** — 心臓拍動・stale timeout の wall-clock には使うが、event frame の timestamp には一切使わない。immutable-design-guard の event-time purity に反するため、`projection_seq_hiwater` と `event_seq` のみをクライアントに渡す。

## Consequences

### Pros

- **Knowledge Loop が永続的な行動履歴になる** — 全 Loop transition が `knowledge_events` に INSERT されるため、reproject で session-state を完全再構築できる。append-first / disposable projection の不変条件が Loop にも拡張される。
- **"生きた UI"** — stream を経由した `invalidateAll()` で、別タブ・別デバイス・`/feeds` 経由の状態変化が 5 秒以内に `/loop` へ反映される。手動リロード依存を解消。
- **Reproject safety** — projector が latest projection state や `time.Now()` を読まず、event payload だけで session-state を計算する。`current_stage_entered_at` が event.occurred_at 由来となり、replay で決定論的になる。
- **Idempotency 2-layer barrier の統一** — `dedupe_key = client_transition_id` で fast-path（`knowledge_loop_transition_dedupes`）と slow-path（`knowledge_events.dedupe_key` unique index）が同じ key で統制される。重複 POST が secondary append を誘発しない。
- **Single emission rule の実装化** — `/loop` と `/feeds` の event 発行経路が明確に分離され、同一ユーザ意図で 2 event を emit する穴が閉じる。
- **Rolling deploy 互換** — DB migration なし・proto additive・新 compose service なし。既存 deploy 経路（`git push origin main` → `dispatch-deploy.yaml` → `Kaikei-e/alt-deploy`）にそのまま乗る。
- **セキュリティ境界維持** — stream / transition ともに `user_id` / `tenant_id` を JWT claim からのみ取得し、request body の同名フィールドは無視（[[000749]] 準拠）。`projected_at` は API / proto / log / metrics に一切露出しない。

### Cons / Tradeoffs

- **`knowledge_events` 行数の増加** — 8 種の新 event 型が append され、dwell / observe / orient 遷移が頻発するためイベント数は体感で 2〜3 倍に増える。ストレージ見積もりに反映する必要がある。`KnowledgeLoopObserved` の server-side rate limit（60 秒窓 / `(user_id, entry_key, lens_mode_id)`, user global 600/min）は canonical contract §8.4 で既に定義されているが、本 ADR の実装範囲には未着手。別 PR で実装する。
- **Stream polling の定常コスト** — 5 秒 × 同時接続ユーザー数で sovereign への `ListKnowledgeEventsSinceForUser` 呼び出しが発生する。Knowledge Home と同等の負荷で耐えてきた実績があるが、併存で 2 倍相当のクエリが流れる点は monitoring 対象。
- **Multi-tab leader election 未実装** — 同一ユーザーが複数タブを開くと、各タブが独立に stream を張る。server 側 rate limit（future work: 5 sessions/user/lens）まで保護が入らない。Knowledge Home では `BroadcastChannel` で leader election していたが、MVP では省略した。
- **Stream classification の粒度** — `KnowledgeLoopObserved` を stream から落としているため、UI は観測イベントの即時反映ができない。これは意図的（flooding 防止）だが、将来的に「観測された entry をハイライトしたい」等の要件が出たら coalescing + batch-emit を検討する。
- **Client の `invalidateAll()` 粒度** — 非 silent frame ごとに `/loop` の SSR snapshot 全体を refetch するため、差分パッチに比べて帯域コストが大きい。foreground が数件のうちは許容だが、将来 UI が重くなったら `useKnowledgeLoop` の optimistic merge に差分を直接適用する改修を検討する。
- **`WhyMappingVersion` bump は未実施** — Orient / Why enrichment 改善は Priority 3 として別 PR で扱う。本 ADR は P1 + P2 のみを固定する。
- **E2E / Pact CDC 未済** — 現状は unit 層と handler 層まで緑で、hurl / Playwright / Pact による境界検証は別 PR に委譲する。stack 立ち上げ不要のローカル検証で足場を先に確保する方針。
