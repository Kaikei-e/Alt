# ADR-000416: Knowledge Home Admin UI に backfill lifecycle 操作を常設する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-frontend-sv (SvelteKit) — admin page, admin hook, backfill jobs table

## Context

Knowledge Home の backend と scheduler を修正しても、Admin 画面は job の一覧表示に留まっていた。管理者は `Backfill Jobs` の状態を確認できても、job を新規作成する `Trigger Backfill`、実行中 job を止める `Pause`、中断 job を再開する `Resume` を画面から実行できなかった。

また、追加した `Trigger Backfill` ボタンは共通 `Button` の default variant を使うと、Admin 画面の配色では hover 時に文字色だけが強く変化し、視認性が崩れる問題があった。

## Decision

**Knowledge Home Admin UI に backfill の lifecycle 操作を常設し、操作中状態と hover 表現を admin 画面で安定して読める形にする。**

### 変更内容

1. `useKnowledgeHomeAdmin(...)` に mutation state を追加し、`triggerBackfill`, `pauseBackfill`, `resumeBackfill` 実行後に snapshot を再取得するようにした。
2. admin ヘッダーに `Trigger Backfill` ボタンを追加し、active version を既定値として job を起動できるようにした。
3. `BackfillJobsTable.svelte` に `Pause` / `Resume` を追加し、job status に応じて押下可能状態を切り替えるようにした。
4. 操作中は `acting` / `activeJobId` を用いて二重実行を抑止した。
5. `Trigger Backfill` ボタンは `outline` variant を基準にし、hover 背景・文字色・border を明示して白飛びしない表示へ固定した。

### 設計判断

1. **job 一覧の近くで操作させる** — 監視と操作を分離せず、状態確認の直後に同じ画面で制御できるようにする。
2. **操作後は楽観更新せず再取得する** — backend の scheduler tick や status 遷移に依存するため、手元で job を作るより snapshot 再取得を正とする。
3. **hover 表現は admin 画面で局所調整する** — 共通 button variant 全体を変えず、このページの CTA に必要なコントラストだけを局所的に確保する。

## Consequences

### Pros

- **Admin 画面だけで運用可能** — backfill の起動・停止・再開を UI で完結できる
- **状態遷移が見えやすい** — 操作後に polling/snapshot 再取得で最新状態へ追従する
- **視認性が安定する** — CTA の hover で文字が白飛びしたように見える問題を避けられる

### Cons / Tradeoffs

- **hook の責務が少し増える** — polling state に加えて mutation state も保持する
- **ページ固有の style override が入る** — 共通 button variant のみでは完結しない
