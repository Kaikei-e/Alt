# ADR-000550: recap サービス群のメモリ制限を本番ワークロード基準に引き上げる

## Status

Accepted

## Date

2026-03-23

## Affected Services

- recap-subworker (Python)
- recap-db (PostgreSQL)
- compose/recap.yaml (Docker Compose)

## Context

[[000547]] で recap-worker の OOM 対策 (mem_limit 2048m + jemalloc) を実施したが、同一 Compose ファイル内の recap-subworker と recap-db にも OOM 問題が潜在していた。

### recap-subworker: 分類プロセスプールの spawn OOM

recap-subworker は `ClassificationRunner` で `multiprocessing.spawn` を使い 6 ワーカープロセスをスポーンする（[[000048]] で CUDA fork 問題回避のため spawn に切り替え）。各ワーカーは DistilBERT モデルと TF-IDF ベクタライザを独立にロードするため、fork の COW 最適化が効かず、ワーカー数に比例してメモリが増加する。

実測値:
- マスタプロセス: ~500MB (Uvicorn + FastAPI + DB pools)
- 6 ワーカー稼働時: ~5GB (docker stats 実測)
- 旧 `mem_limit: 512m`: マスタプロセスだけで 98% 消費。ワーカーをスポーンする余裕がなく、`_verify_worker_initialization()` で永久ハング

`mem_limit` を 1536m に引き上げたが、マスタ 500MB + 6 ワーカー × ~200MB = ~1700MB で再び不足。子プロセスが OOM kill されてもエラーログが出ず、親の `apply_async().get()` が永久にブロックする無症状ハングが発生した。

### recap-db: JSONB ステート保存時の OOM

recap-db は `mem_limit: 256m` で運用していたが、パイプラインの stage state 保存（前処理済み ~2900 記事の JSONB INSERT）で PostgreSQL がメモリ不足に陥り、OOM killer によって kill された（`docker inspect` で `OOMKilled: true` を確認）。PostgreSQL クラッシュは全コネクションの強制切断を引き起こし、recap-worker と recap-subworker の両方が同時に失敗する連鎖障害となった。

## Decision

### recap-subworker: `mem_limit: 8192m`

6 ワーカー稼働時の実測値 ~5GB に対して十分なヘッドルームを確保する。ワーカー数 (`CLASSIFICATION_WORKER_PROCESSES=6`) は変更しない。分類速度を維持しつつ、spawn プロセスプールが安定して動作するメモリを確保する。

### recap-db: `mem_limit: 2048m`

256MB からの引き上げ。PostgreSQL 18 + 50 コネクション + JSONB ステート保存 + 分類結果永続化のワークロードに対して十分な余裕を持たせる。定常使用量は ~670MB のため、2GB で 3 倍のヘッドルームを確保する。

## Consequences

### Pros

- 分類プロセスプール (6 ワーカー) が安定してスポーン・処理を完了できる
- recap-db の OOM kill が解消され、stage state 保存が安定する
- 連鎖障害パターン（DB クラッシュ → 全コネクション断 → worker/subworker 同時失敗）が発生しなくなる

### Cons / Tradeoffs

- recap-subworker に 8GB を割り当てるのはホスト全体のメモリバジェットに対して大きい。ただし実測ピークは ~5GB であり、常時 8GB を消費するわけではない
- `CLASSIFICATION_WORKER_PROCESSES` を減らせばメモリ使用量は下がるが、分類スループットが低下する。現時点ではメモリ増で対応し、将来的にモデル軽量化やワーカー数チューニングで最適化する余地がある
- spawn ベースのプロセスプールはワーカー数に対してメモリが線形に増加する制約が残る。ワーカー数を変更する場合は `mem_limit` の再計算が必要
