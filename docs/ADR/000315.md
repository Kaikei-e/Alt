# ADR-000315: rask-log-forwarder ビジネスコンテキスト付与の総点検・修正

## ADR's STATUS

完了

## CONTEXT

rask-log-forwarder サイドカーは 13 サービスの Docker コンテナログを収集し、サービス固有パーサーでビジネスコンテキスト（ADR 98 準拠の `alt.*` フィールド、`trace_id`/`span_id`）を抽出・付与して rask-log-aggregator に転送する。

全サイドカー対象サービスを精査した結果、以下 5 カテゴリの問題が検出された:

| カテゴリ | 影響 | 対象サービス数 |
|---------|------|--------------|
| ログフォーマット非互換 | message 空文字列、ビジネスコンテキスト全喪失 | 1 (recap-worker) |
| パーサーレジストリ登録漏れ | auto-detect フォールバックで非決定的動作 | 6 |
| OTEL 設定漏れ | trace_id/span_id 常時空、トレースエクスポート無効 | 7 |
| サイドカー未設定 | ログ完全未収集 | 1 (mq-hub) |
| ビジネスコンテキストフィールド欠落 | ADR 98 準拠の `alt.*` キーが一部欠損 | 2 |
| Python structlog `"event"` キー非互換 | message 空文字列 | 2 (recap-evaluator, recap-subworker) |

### recap-worker の問題（最重要）

recap-worker は Rust `tracing_subscriber::fmt::layer().json()` で以下のフォーマットを出力する:

```json
{"timestamp":"...","level":"INFO","fields":{"message":"Processing job","alt.job.id":"abc-123"}}
```

Go パーサーはトップレベルの `"msg"`/`"message"` を期待するため、`fields` オブジェクト内のメッセージとビジネスコンテキストが全て喪失していた。

### Python structlog サービスの問題

ClickHouse 実データ検証で、recap-evaluator と recap-subworker の `message` が空になる問題を発見した。Python structlog の `JSONRenderer()` はメッセージを `"event"` キーで出力するが、Go パーサーは `"msg"` / `"message"` のみ対応していた。

```json
{"event":"recap-evaluator started successfully","alt.ai.pipeline":"recap-evaluation","timestamp":"2026-03-03T17:38:47.998241Z","service":"recap-evaluator","level":"info"}
```

なお、同じ Python サービスでもログフォーマッタの違いで影響が異なる:
- **news-creator**: カスタム Formatter で `event → msg` に変換済み → Go パーサーで正常動作
- **tag-generator**: `JsonFormatter` で `event → msg` に変換済み → Go パーサーで正常動作
- **recap-evaluator / recap-subworker**: `JSONRenderer()` そのままで `"event"` キー → Go パーサーで message 空

## DECISION MAKING

### 1. RustTracingParser の新規作成

Rust `tracing_subscriber` 固有の JSON フォーマットを解析する専用パーサーを TDD で実装した。

- `fields.message` を `ParsedLogEntry.message` として抽出
- `fields.*`（message 以外）を `ParsedLogEntry.fields` にフラット展開
- `detection_priority = 65`（Go パーサー 60 より高い）により、auto-detect 時にも Rust 形式を優先判定
- `can_parse()` で `"fields":{` + `"timestamp"` の存在かつトップレベル `"msg"` の不在で Go slog と区別

### 2. パーサーレジストリへの明示的マッピング追加

6 サービスが明示的マッピングなしで auto-detect にフォールバックしていた。決定論的動作のため全サービスに明示マッピングを追加した。

| サービス | パーサー | 理由 |
|---------|---------|------|
| recap-worker | rust-tracing | Rust tracing JSON フォーマット |
| recap-subworker | python-structlog | Python structlog `JSONRenderer()` で `"event"` キー出力 |
| recap-evaluator | python-structlog | 同上 |
| rag-orchestrator | go | Go slog JSON |
| news-creator-backend | go | Ollama plain text（Go パーサーの plain fallback） |
| dashboard | go | Streamlit plain text（同上） |
| mq-hub | go | Go slog JSON |

### 3. mq-hub サイドカーの新規追加

mq-hub は `rask.group=mq-hub` ラベル付きで Go slog JSON を出力していたが、logging.yaml にフォワーダーが未定義でログが完全に収集されていなかった。既存サイドカーと同一構成で追加した。

### 4. OTEL 環境変数の設定

7 サービスに `OTEL_EXPORTER_OTLP_ENDPOINT` と `OTEL_SERVICE_NAME` を追加し、trace_id/span_id の出力と OTLP エクスポートを有効化した。

| compose ファイル | サービス |
|-----------------|---------|
| core.yaml | alt-backend |
| auth.yaml | auth-hub |
| workers.yaml | search-indexer |
| recap.yaml | recap-worker, recap-subworker, recap-evaluator |
| mq.yaml | mq-hub |

### 5. ビジネスコンテキストフィールドの補完

TDD で欠落フィールドの変換を追加した:

- **recap-evaluator**: `article_id` → `alt.article.id` を `add_business_context` に追加
- **tag-generator**: `processing_stage` → `alt.processing.stage` を `add_business_context` に追加

### 6. PythonStructlogParser の新規作成

ClickHouse 実データ検証で発見された Python structlog `"event"` キー問題に対し、言語ごとに専用パーサーという一貫したアーキテクチャに従い、Python structlog 専用パーサーを TDD で新規作成した。

- `"event"` キーを `ParsedLogEntry.message` として抽出
- `"level"` キーから `LogLevel` を判定（Python の `critical` → `Fatal` マッピング含む）
- 残りのキー（`"event"`, `"level"` 以外）を `ParsedLogEntry.fields` にフラット展開
- `service_type()` → `"python-structlog"`
- `detection_priority = 63`（Go 60 より高い、Rust tracing 65 より低い）
- `can_parse()`: `"event"` キーが存在 AND トップレベル `"msg"` が不在 AND `"fields":{` が不在（Rust 形式除外）

パーサーマッピングも修正:
- `recap-subworker`: `go` → `python-structlog`
- `recap-evaluator`: `go` → `python-structlog`

なお、`tag-generator` と `news-creator` はカスタム Formatter で `event → msg` に変換しているため Go パーサーのまま維持。

### PROS

- recap-worker のビジネスコンテキスト（`alt.job.id`, `alt.processing.stage` 等）がログに正しく出現するようになった
- 全 14 サービスに対して明示的パーサーマッピングが完備され、auto-detect フォールバックによる非決定的動作が排除された
- mq-hub のログが初めて収集対象となり、イベントブローカーの可観測性が確保された
- 6 サービスで OTEL トレース連携が有効化され、分散トレーシングのカバレッジが向上した
- ADR 98 準拠の `alt.*` フィールドが recap-evaluator と tag-generator で補完された
- recap-evaluator / recap-subworker の `message` が空だった問題が PythonStructlogParser により解決された

### CONS, TRADEOFF

- recap-worker に OTEL を有効化すると `StructuredLogLayer`（stderr への独自 JSON 出力）が無効化され、`fmt_layer` + `otel_layer` のみになる。従来 stderr に依存していた外部監視があれば影響する（現時点では該当なし）
- `StructuredLogLayer` のコード自体はまだ残存しているため、OTel 有効状態では不要コードとなる。削除は別 PR で対応する
- mq-hub-logs サイドカーの追加により Docker ソケットを共有するコンテナが 1 つ増加する（リソース影響は軽微）
- Python サービス間でパーサーが分かれる設計: `tag-generator` / `news-creator` はカスタム Formatter で `event → msg` 変換しているため Go パーサー、`recap-evaluator` / `recap-subworker` は `JSONRenderer()` そのままのため Python パーサー。これは各サービスのログフォーマッタ実装に忠実な設計であり、Go パーサーに `"event"` キーを追加する（全 JSON ログを Python 扱いしてしまうリスク）より安全
