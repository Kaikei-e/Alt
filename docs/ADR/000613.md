# ADR-000613: Gemma3 から Gemma4:E4B Q4_K_M へ LLM 推論モデルを移行する

## Status

Accepted

## Date

2026-04-03

## Affected Services

- news-creator (Python/FastAPI) — 要約・チャットプロキシ・クエリ展開
- news-creator-backend (Ollama) — ローカル GPU 推論サーバー
- pre-processor (Go) — 要約リクエスト発行、品質チェッカー
- rag-orchestrator (Go) — RAG 回答生成
- recap-worker (Rust) — Recap 要約リクエスト
- recap-evaluator (Python) — G-Eval 品質評価

## Context

Alt の LLM 推論は全 4 マシンで `gemma3:4b-it-qat`（Q4_0、4.0GB）を使用していた。Gemma4:E4B は同じ `<start_of_turn>`/`<end_of_turn>` チャットテンプレートを維持しつつ、128K context 対応・マルチモーダル（Text/Image/Audio）という進化を遂げたモデルである。

### Ollama デフォルトタグの問題

Ollama の `gemma4:e4b` タグは 9.6GB（テキストモデル + マルチモーダル projector を含む）で、RTX 4060 8GB VRAM には収まらない。一方、ggml-org が公開するテキスト専用の Q4_K_M GGUF は **5.34GB** であり、8GB VRAM に十分収まる。

### Gemma4 の Thinking Mode

Gemma4 はデフォルトで thinking mode が有効となり、内部推論トークンを生成する。Ollama の GGUF import 時に自動検出される `RENDERER gemma4` / `PARSER gemma4` ディレクティブがこれを制御する。`/api/chat` エンドポイントでは `think: false` パラメータで無効化できるが、`/api/generate` エンドポイントではこのパラメータが正しく動作しない（Ollama の既知の制限）。

news-creator は `/api/generate` を使用しており、かつプロンプトテンプレート（`prompts.py`）に `<start_of_turn>` / `<end_of_turn>` を直接埋め込んでいるため、`raw: true` を設定して RENDERER/PARSER をバイパスすることで thinking を無効化する方式が最も安全である。

## Decision

1. **ggml-org の Q4_K_M GGUF (5.34GB) を HuggingFace から直接ダウンロードし、`ollama create` で import する方式を採用する**。Ollama のデフォルトタグ（9.6GB）は使用しない。
2. **ベースモデル名を `gemma4-e4b-q4km` とし**、`num_ctx` バリアント（`gemma4-e4b-8k`, `gemma4-e4b-12k`）を Modelfile で作成する。
3. **全 `/api/generate` リクエストパスに `raw: true` を追加**し、Gemma4 の thinking mode をバイパスする。
4. **リモートヘルスチェッカーの `:latest` タグ対応**を修正し、`gemma4-e4b-q4km` と `gemma4-e4b-q4km:latest` の両方にマッチさせる。
5. **Modelfile にカスタム TEMPLATE と `PARAMETER stop <end_of_turn>` を明示的に設定**し、GGUF import 時の自動検出テンプレートの不備を補完する。

### 検討した代替案

- **`ollama pull gemma4:e4b`**: 9.6GB でマルチモーダル projector を含み、8GB VRAM に収まらない。却下。
- **`think: false` パラメータ**: `/api/generate` で正しく動作しない Ollama の既知制限により却下。`raw: true` の方が確実。
- **`/api/chat` への全面移行**: 既存の `prompts.py` のテンプレート構造と `OllamaStreamDriver` の `raw` モード設計を大幅に変更する必要がある。リスクが高く却下。

## Consequences

### Pros

- **VRAM 効率向上**: 5.34GB（テキスト専用）で 8GB VRAM に余裕を持って収まる（6.3GB on GPU with 12K context）
- **Gemma3 と同等のパフォーマンス**: Decode ~53 tok/s、Prefill ~2900 tok/s、TTFT ~0.2s（HOT state）
- **チャットテンプレート互換**: `<start_of_turn>`/`<end_of_turn>` が Gemma3/4 で共通のため、`prompts.py` の変更不要
- **`raw: true` による thinking 完全バイパス**: RENDERER/PARSER を経由しないため、将来の Ollama バージョンで thinking の挙動が変わっても影響を受けない
- **ローカルフォールバック維持**: リモート GPU が利用不可でもローカル RTX 4060 で処理継続

### Cons / Tradeoffs

- **GGUF 手動管理**: `ollama pull` ではなく HuggingFace から curl でダウンロードする必要があり、entrypoint の複雑さが増す
- **マルチモーダル非対応**: テキスト専用 GGUF のため、将来 Image/Audio 入力を使いたい場合は別途対応が必要
- **12K context の VRAM 使用率**: 6.3GB/8GB (79%) で余裕は少ない。16K 以上のコンテキストは現実的でない
- **`raw: true` への依存**: プロンプトに `<start_of_turn>` / `<end_of_turn>` を必ず含める必要がある。テンプレートの整合性が崩れると出力が壊れる

### 変更ファイル（主要）

- `news-creator/Modelfile.gemma4-e4b-q4km` — GGUF import 用（新規）
- `news-creator/Modelfile.gemma4-e4b-8k`, `Modelfile.gemma4-e4b-12k` — context variant
- `news-creator/entrypoint-backend.sh` — GGUF DL + import + preload
- `news-creator/Dockerfile.backend` — Modelfile COPY
- `news-creator/app/news_creator/gateway/ollama_gateway.py` — `raw: true` 追加
- `news-creator/app/news_creator/gateway/distributing_gateway.py` — `raw: true` 追加
- `news-creator/app/news_creator/gateway/remote_health_checker.py` — `:latest` タグ対応
- `news-creator/app/news_creator/config/config.py` — デフォルトモデル名
- `compose/ai.yaml` — `OLLAMA_BASE_MODEL`, `LLM_MODEL`, `OLLAMA_KV_CACHE_TYPE`, `mem_limit`
- `.env`, `.env.template` — モデル名更新
- 各サービス（pre-processor, rag-orchestrator, recap-worker, recap-evaluator）のモデル名デフォルトとテスト
