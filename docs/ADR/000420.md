# ADR-000420: Knowledge Home の SLO/SLI を定義し OTel メトリクスとアラートルールを実装する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — OTel メトリクス計装、SLO Status usecase、Admin API
- alt-frontend-sv (TypeScript) — Admin SLO タブ UI
- alt-butterfly-facade (Go) — Admin API プロキシ

## Context

Knowledge Home は Phase 1-4 を経て「動く」状態に達した。しかし「動く」と「信用できる」は別物であり、SLO（Service Level Objective）なしにはプロダクション品質の議論が主観に頼ることになる。

Phase 5 の 90 日ロードマップ Day 61-90 では、Knowledge Home を Alt の恒久機能として定着させるために SLO/alerts を固めることが求められている。

既存の OTel メトリクスは projector/backfill 中心の 10 個のみで、可用性・鮮度・アクション永続性・ストリーム継続性・正確性をカバーしていなかった。

## Decision

### 5 つの SLI を定義し、対応する OTel メトリクスとアラートルールを同時に実装する。

**SLI 定義:**

| SLI | 計測対象 | 目標 | データ源 |
|-----|---------|------|---------|
| SLI-A: Availability | Home API の成功率 | 99.5% (30d) | `alt_home_requests_total` |
| SLI-B: Freshness | Projection lag | < 5 分 (p95) | `alt_home_projector_lag_seconds` + DB 直接クエリ |
| SLI-C: Action Durability | Tracking 永続化成功率 | 99.0% | `alt_home_tracking_*_total` |
| SLI-D: Stream Continuity | サーバー起因切断率 | < 5% | `alt_home_stream_*_total` |
| SLI-E: Correctness Proxy | 空応答・malformed why 率 | < 1% / < 0.1% | `alt_home_empty_responses_total`, `alt_home_malformed_why_total` |

**16 個の新規 OTel メトリクス** を `KnowledgeHomeMetrics` struct に追加（既存 10 個と合わせて計 26 個）:

- SLI-A 系: `alt_home_requests_total`, `alt_home_request_duration_seconds`, `alt_home_degraded_responses_total`, `alt_home_projection_age_seconds`
- SLI-C 系: `alt_home_tracking_received_total`, `alt_home_tracking_persisted_total`, `alt_home_tracking_failed_total`
- SLI-D 系: `alt_home_stream_connections_total`, `alt_home_stream_disconnects_total`, `alt_home_stream_reconnects_total`, `alt_home_stream_update_lag_seconds`
- SLI-E 系: `alt_home_empty_responses_total`, `alt_home_malformed_why_total`, `alt_home_orphan_items_total`, `alt_home_supersede_mismatch_total`
- Reproject 系: `alt_home_reproject_events_total`

**アラートルール** を Prometheus + Grafana 双方に定義:

- **Page 級**: 可用性 multi-window multi-burn-rate (14.4x)、tracking 失敗率 > 5%、stream 切断率 > 80%、freshness > 30 分
- **Ticket 級**: 可用性 burn-rate (6x)、freshness > 10 分、empty 応答率 > 1%、malformed why > 0.1%、degraded 応答率 > 5%

**Admin API に `GetSLOStatus` RPC を追加:**

SLI-B (freshness) は `knowledge_projection_checkpoints` テーブルから DB 直接クエリで算出。SLI-A/C/D は OTel メトリクス経由で Prometheus に蓄積される設計とし、in-process aggregation は採用しない。

### 代替案の検討

| 案 | 評価 |
|---|---|
| in-process で全 SLI を集計 | Prometheus の role を侵食し二重計算になる。不採用 |
| SLI-B も Prometheus から取得 | DB クエリの方が即時性が高く、Admin API の応答に含めるには最適。ハイブリッド採用 |
| SLO dashboard を Grafana だけで完結 | Admin UI でのインライン表示が運用上有用。API + UI の二面構成を採用 |

## Consequences

### Pros

- 5 つの SLI により Knowledge Home の品質を定量的に議論できるようになる
- multi-window burn-rate アラートにより、一過性のスパイクでの誤報を防止
- Admin UI で SLO 状態を即座に確認でき、runbook への動線を確保

### Cons / Tradeoffs

- メトリクス 26 個はカーディナリティ面で注意が必要（attribute 設計で抑制済み）
- SLI-A/C/D の Admin API 表示は Prometheus Query API への依存が前提（Phase 5 初期はプレースホルダー値）
- アラートの閾値はチューニングが必要（burn-rate 係数は SRE workbook 準拠だが、Alt のトラフィック規模に合わせて調整する余地がある）
