# ADR-000461: Knowledge Home articles ビューに useSummarize hook で要約機能を追加する

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-frontend-sv (SvelteKit / TypeScript)

## Context

Knowledge Home でカードの「Open」を押すと `/articles/[id]?url=...` に遷移し記事全文が表示されるが、この画面に要約機能がなかった。一方、FeedDetailModal には `streamSummarizeWithAbortAdapter` を用いたストリーミング要約ボタンが既に実装済みであり、同等の UX を `/articles/[id]` ページでも提供する必要があった。

しかし FeedDetailModal の要約ロジック（状態管理、AbortController、エラーリトライ、ボタン状態派生）は約 80 行のインラインコードであり、そのまま別ページにコピーすると重複と乖離リスクが生まれる。`useTtsPlayback` hook が TTS ロジックを再利用可能にしたように、要約ロジックも hook として抽出する設計判断が求められた。

[[000425]] の UI v0 設計（SummaryStateChip, action center）に沿い、QuickActionRow からの要約導線も同時に整備する。

## Decision

**FeedDetailModal の要約ロジックを `useSummarize` hook として抽出し、`/articles/[id]` ページと QuickActionRow の両方から利用可能にする。**

### 1. `useSummarize` hook の抽出

`useTtsPlayback` パターンに準拠した Svelte 5 Runes ベースの hook を `src/lib/hooks/useSummarize.svelte.ts` に作成する。

```typescript
export interface UseSummarize {
  readonly summary: string | null;
  readonly isSummarizing: boolean;
  readonly summaryError: string | null;
  readonly buttonState: SummarizeButtonState; // "idle" | "loading" | "error" | "success"
  summarize(feedUrl: string, articleId?: string, title?: string, forceRefresh?: boolean): void;
  abort(): void;
  reset(): void;
}
```

内部実装は FeedDetailModal の `handleSummarize()` と同一パターン:
- `$state` で `isSummarizing`, `summary`, `summaryError`, `abortController`, `retryCount` を管理
- `$derived.by` で `buttonState` を派生（loading → error → success → idle の優先順位）
- `streamSummarizeWithAbortAdapter` を呼び出し、チャンクを蓄積
- AbortError / cancel は無視、transient error は 1 回リトライ（500ms 遅延）

### 2. `/articles/[id]` ページの拡張

- ヘッダーに Summarize ボタンを追加（4 状態: idle / loading / error / success）
- 記事コンテンツの上に AI Summary セクションを表示
- `?summarize=true` クエリパラメータで自動トリガー対応
- `onDestroy` と URL 変更時の `$effect` でクリーンアップ

### 3. QuickActionRow への Summarize アクション追加

`primaryActions` に Sparkles アイコンの「Summarize」を Open と Ask の間に配置。クリック時は `/articles/[id]?url=...&title=...&summarize=true` に遷移し、ページ側で自動要約を開始する。

### 4. ナビゲーション時の `title` 伝搬

Home ページの `handleAction("open")` と `handleRecallOpen` で `title` クエリパラメータを追加。要約 API に記事タイトルを渡すことで要約品質を向上させる。

### 検討した代替案

| 代替案 | 評価 |
|--------|------|
| FeedDetailModal のコードをコピーして `/articles/[id]` に貼り付ける | 即効性は高いが、2 箇所の同期コストが発生。リトライロジックの変更時に片方だけ漏れるリスク |
| QuickActionRow から要約をインライン実行（カード上で展開） | カードごとにストリーミング状態を管理する複雑さが増大。Phase 2 以降で検討 |
| FeedDetailModal 側も `useSummarize` に移行する | 理想的だが、スコープが広がる。別 PR で段階的に実施可能 |

## Consequences

### Pros

- **再利用性**: 要約ロジックが hook に集約され、新しいページ・コンポーネントからも 1 行で利用可能
- **テスト容易性**: hook 単体で 20 テストケースを TDD で作成。コンポーネントテストに依存しない
- **一貫した UX**: FeedDetailModal と同じボタン状態遷移・スタイリング・リトライ戦略を `/articles/[id]` でも提供
- **自動要約導線**: QuickActionRow → `/articles/[id]?summarize=true` のフローにより、ワンクリックで記事閲覧と要約を同時に開始

### Cons / Tradeoffs

- **FeedDetailModal との二重管理**: 現時点では FeedDetailModal は従来のインラインコードのまま。hook 移行は別 PR で行う前提
- **URL クエリパラメータの増加**: `title` と `summarize` の 2 パラメータが追加。長いタイトルの場合 URL が長くなるが、機能上の問題はない
- **カードレベルの要約はページ遷移が必要**: QuickActionRow の Summarize はページ遷移を伴うため、フィード一覧を離脱する。インライン要約は将来の拡張として検討
