# ADR-000592: Knowledge Home Admin コンソールに Storage/Audit タブを追加

## Status

Accepted

## Date

2026-03-26

## Affected Services

- alt-frontend-sv (TypeScript/SvelteKit) — 新規タブ UI、BFF ルート、フック
- knowledge-sovereign (Go) — リテンション/スナップショット/ストレージ REST エンドポイント（既存）
- alt-backend (Go) — RunProjectionAudit Connect-RPC（既存）

## Context

Knowledge Home の Admin コンソール (`/admin/knowledge-home`) は Overview / SLO / Reproject の3タブで運用していた。しかし、以下のバックエンド API が実装済みであるにもかかわらず UI が未実装の状態だった。

**knowledge-sovereign メトリクスポート (:9501) の REST エンドポイント:**
- `GET /admin/storage/stats` — テーブルストレージ統計
- `GET /admin/snapshots/list` / `GET /admin/snapshots/latest` — スナップショット一覧・最新
- `POST /admin/snapshots/create` — スナップショット作成
- `GET /admin/retention/status` / `GET /admin/retention/eligible` — リテンションログ・アーカイブ対象
- `POST /admin/retention/run` — リテンション実行（dry-run / live）

**alt-backend の Connect-RPC:**
- `RunProjectionAudit` — プロジェクション整合性監査

これらは [[000585]] (Projection Snapshot) と [[000586]] (Retention Policy and Archive Worker) で設計・実装されたが、運用者がブラウザから状態を確認・操作する手段がなかった。Phase 6 の Admin UI 方針（alt_knowledge_home_phase_plan.md）に基づき、UI を実装した。

## Decision

Admin コンソールに **Storage タブ**と **Audit タブ**を追加し、バックエンド全 admin API を UI から参照・操作可能にする。

### アーキテクチャ方針

1. **sovereign REST エンドポイントは SvelteKit サーバーサイドから直接呼び出す**
   - knowledge-sovereign のメトリクスポートは Docker ネットワーク内部のみ公開（認証不要）
   - 新規 API ルート `/api/admin/knowledge-home/sovereign` を BFF として設置
   - Connect-RPC ではなく plain HTTP fetch（sovereign 側が REST のため）

2. **RunProjectionAudit は既存の BFF パスを経由**
   - alt-butterfly-facade → alt-backend の Connect-RPC 経路を踏襲
   - 既存の `/api/admin/knowledge-home` ルートに `run_audit` アクションを追加

3. **sovereign 用フックを既存フックと分離**
   - `useSovereignAdmin.svelte.ts` — 30秒ポーリング（sovereign データは低頻度更新）
   - `useKnowledgeHomeAdmin.svelte.ts` — 既存の10秒ポーリング + audit state 追加
   - データソースとポーリング頻度が異なるため、責務を分離

4. **Go PascalCase → camelCase 正規化**
   - knowledge-sovereign の Go struct には `json` タグがない（PascalCase でシリアライズされる）
   - サーバーサイドモジュール (`sovereign-admin.ts`) で camelCase に正規化
   - フロントエンドの TypeScript interface は全て camelCase に統一

### 危険操作の保護

Phase 6 仕様に従い、破壊的操作にはユーザー確認を要求:

| 操作 | リスク | 確認 |
|---|---|---|
| スナップショット作成 | 中（読み取り専用 export） | 確認ダイアログ (default) |
| リテンション実行 (dry-run) | 低（副作用なし） | 不要 |
| リテンション実行 (live) | 高（パーティション export） | 確認ダイアログ (danger) |

### Svelte 5 ベストプラクティス準拠

- 全コンポーネントで `interface Props` + `$props()` パターン
- wholesale-replace データに `$state.raw` 適用
- レガシー構文 (`export let`, `$:`, `on:click`) 完全排除
- callback props に `on` prefix (`onRunAudit`, `onCreateSnapshot`, `onRunRetention`)

## Consequences

### Pros

- 運用者がブラウザからリテンション状態・ストレージ使用量・スナップショット履歴・プロジェクション整合性を一元的に確認可能
- リテンション dry-run をワンクリックで実行でき、アーカイブ対象の事前確認が容易
- プロジェクション監査がCLI不要で実行可能（altctl なしで drift 検出）
- 確認ダイアログにより誤操作リスクを低減

### Cons / Tradeoffs

- sovereign REST エンドポイントに認証がない（Docker ネットワーク内部のみの前提）。将来外部公開する場合はサービストークン認証の追加が必要
- Go struct に `json` タグを追加すれば正規化レイヤーが不要になるが、knowledge-sovereign 側の変更はスコープ外とした
- sovereign データのSSR未対応（初回はクライアントサイドポーリングで取得）。admin ページの初回表示がわずかに遅延するが、運用者向けのため許容
