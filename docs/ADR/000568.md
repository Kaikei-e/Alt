# ADR-000568: Ask Augur を 5 層の Agentic RAG アーキテクチャに進化させる

## Status

Accepted

## Date

2026-03-25

## Affected Services

- rag-orchestrator (Go)

## Context

[[000498]] で「ルーティング型 Agentic RAG Phase 1」として Strategy パターン（`article_scoped` vs `general`）と QueryIntent パーサーを導入した。しかし現行 RAG には以下の限界が残っていた。

1. **検索品質の自己評価がない** — reranker が低スコアを返しても、そのまま LLM に渡して 20 秒以上の生成コストを浪費する
2. **クエリ意図の分類が 2 種類だけ** — 比較・時系列・深掘り・ファクトチェックなどのクエリパターンに対応できない
3. **ツール活用がない** — タグ検索・日付フィルタなど、既存サービスの機能を検索補強に使えない
4. **回答品質の検証が構文チェック止まり** — JSON パース + citation 検証のみで、内容カバレッジや文の完結性は見ない
5. **各ステップの可観測性が不足** — どの段階で何が起きたかをトレースできない

Web 調査（NVIDIA、IBM、arXiv:2501.09136 等）で Agentic RAG のベストプラクティスを調査した結果、以下の制約を踏まえた設計が必要と判断した。

- **GPU**: ホストマシンの RTX 4060 8GB VRAM（news-creator-backend 経由）。TTFT ~20s、TPS 50-60 tok/s
- **モデル**: Gemma3:4b に限定（Qwen は品質面で却下）。4B モデルは Ollama でネイティブ tool calling 未サポート（GitHub issue #9680 で "not planned"）
- **既存パターン**: [[000498]] の Strategy パターン + QueryIntent を拡張する方針

## Decision

**5 層の Agentic RAG アーキテクチャ**を採用し、既存の Strategy パターンを拡張する形で段階的に実装する。各層は独立デプロイ可能で、前の層なしでもフォールバック動作する。

### 検討した代替案

| 代替案 | 評価 |
|--------|------|
| LLM 駆動のツール選択（ReAct ループ） | Gemma3:4b では tool calling 精度が低く（hallucination ~79%）、不採用 |
| LangGraph / LlamaIndex 等のフレームワーク導入 | Go サービスに Python フレームワークを混在させる複雑さが見合わない |
| LLM ベースのクエリ分類のみ | RTX 4060 なら ~2s で可能だが、ルールベースで判定できるケースに GPU を使う必要なし。ハイブリッドを採用 |

### 実装の概要

#### Phase 1: Retrieval Quality Gate（検索品質ゲート）

reranker の cross-encoder スコアを `ContextItem.RerankScore` フィールドとして retrieval パイプライン全体に伝播させ、`RetrievalQualityAssessor` が top-3 チャンクの平均スコアで品質を判定する。

```
retrieve → assessQuality
  → Good (avg >= 0.5):         そのまま生成へ
  → Marginal (avg >= 0.25):    queryExpander でクエリ書き換え → 再 retrieve（最大 1 回）
  → Insufficient (avg < 0.25): fallback 応答
```

閾値は環境変数 `RAG_QUALITY_THRESHOLD_GOOD` / `RAG_QUALITY_THRESHOLD_MARGINAL` で設定可能。`buildPrompt()` に統合済み。

#### Phase 2: Smart Query Classification（ハイブリッドクエリ分類）

`IntentType` を 2 → 6 種類に拡張し、`QueryClassifier` がハイブリッド方式で分類する。

| IntentType | 日本語パターン例 | 英語パターン例 |
|------------|----------------|---------------|
| `comparison` | 「違い」「比較」「対」 | "vs", "compare", "difference" |
| `temporal` | 「最近」「今週」「最新」 | "latest", "recent", "this week" |
| `topic_deep_dive` | 「詳しく」「深掘り」「について教えて」 | "explain", "in detail", "tell me about" |
| `fact_check` | 「本当」「事実」「正しい」 | "is it true", "fact check" |

ルールベースで判定できない場合のみ LLM フォールバック（feature flag `RAG_LLM_CLASSIFY_ENABLED` で制御）。

各 intent に対応する Strategy を新設（`temporalStrategy`, `comparisonStrategy`, `topicDeepDiveStrategy`, `factCheckStrategy`）。いずれも既存の `generalStrategy` を内部で再利用し、入力パラメータやプロンプトを調整する設計。

#### Phase 3: Tool Augmentation（Intent 駆動ツールディスパッチ）

Gemma3:4b は Ollama でネイティブ tool calling 未サポートのため、**ルールベース（intent 駆動）のツールディスパッチ**を採用。`ToolDispatcher` が intent に基づいてツールを選択・実行する。

```go
type Tool interface {
    Name() string
    Description() string
    Execute(ctx context.Context, params map[string]string) (*ToolResult, error)
}
```

ディスパッチルール: `IntentTemporal` → `DateRangeFilterTool`、`IntentComparison` → `TagSearchTool` 等。ツール実行に LLM は使わない（追加レイテンシ 0）。

#### Phase 4: Answer Quality Validation（回答品質検証）

`AssessAnswerQuality()` が LLM 生成後の回答を 4 項目で検証する。すべて文字列操作のみ（追加 LLM 呼び出しなし）。

1. **Coverage Check**: クエリの主要キーワードが回答に含まれるか
2. **Citation Density**: 回答長に対する citation 数の最低比率
3. **Coherence Check**: 回答末尾が文として完結しているか
4. **Intent-Aware Check**: FactCheck intent なら「根拠」「出典」等を含むか

#### Phase 5: Observability（AgentTrace）

`AgentTrace` が各ステップ（classify → retrieve → assess → rewrite → generate → validate）を記録し、`AnswerDebug` に統合する。

```go
type AnswerDebug struct {
    // ... 既存フィールド ...
    IntentType        string      // Phase 2
    RetrievalQuality  string      // Phase 1
    RetryCount        int         // Phase 1
    ToolsUsed         []string    // Phase 3
    QualityFlags      []string    // Phase 4
    AgentSteps        []AgentStep // Phase 5
    TotalAgentStepsMs int64       // Phase 5
}
```

### テストカバレッジ

- RetrievalQualityAssessor: 9 テスト（全スコアパターン、RerankScore フォールバック、境界値）
- 品質ゲート統合: 3 テスト（Good → no retry、Insufficient → fallback、Marginal → retry）
- QueryClassifier: 10 テスト（日英バイリンガル、全 intent type、article-scoped 互換性）
- Strategy: 4 テスト（temporal、comparison）
- ToolDispatcher: 4 テスト（intent 別ツール選択、エラー時 graceful degradation）
- AssessAnswerQuality: 7 テスト（coverage、citation density、coherence、fact-check）
- AgentTrace: 3 テスト（ステップ追加、合計時間、空トレース）
- 既存テスト: 全テスト通過確認済み

## Consequences

### Pros

- **検索品質の自動評価**: reranker スコアに基づく品質ゲートにより、低品質コンテキストでの無駄な LLM 生成（~20s）を回避
- **GPU コスト最小化**: 品質ゲート・クエリ分類・ツール選択・回答検証のすべてがルールベース or 既存スコアの再利用。追加 LLM 呼び出しは分類フォールバック時の ~2s のみ
- **OCP 準拠の拡張性**: Strategy パターン + ToolDispatcher により、新しい intent type やツールを既存コードを変更せずに追加可能
- **段階デプロイ**: 各 Phase は独立して有効/無効を制御可能。品質ゲートは `RAG_QUALITY_GATE_ENABLED`、LLM 分類は `RAG_LLM_CLASSIFY_ENABLED` で制御
- **可観測性**: AgentTrace + AnswerDebug の拡張により、どのステップで何が起きたかを REST/Connect-RPC の両経路で確認可能

### Cons / Tradeoffs

- **Marginal リトライ時のレイテンシ増加**: リトライ発生時は TTFT 相当（~15-20s）の追加遅延。頻度を低く保つ閾値設計が必要
- **ルールベース分類の限界**: キーワードパターンでカバーできない曖昧なクエリは IntentGeneral にフォールバック。LLM フォールバックで補完するが、4B モデルの分類精度は限定的
- **Gemma3:4b の tool calling 制約**: ネイティブ tool calling 未サポートのため、ツール選択は LLM 駆動ではなくルールベース。将来的に 12B+ モデルへ移行すれば LLM 駆動に切り替え可能
- **残作業あり**: 具体的なツール実装（`adapter/tools/`）、`buildPrompt()` への ToolDispatcher / QualityFlags 統合、DI 配線、OpenAPI spec 更新が未完了
