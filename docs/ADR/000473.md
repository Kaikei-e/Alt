# ADR-000473: Knowledge Home dismiss を write-through fast-path で即時反映する

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-backend (Go)

## Context

`dismiss` イベントが tracking flag を通過するようになっても、Knowledge Home 上では「押した直後に消えるが、直後リロードでは再表示される」時間窓が残っていた。

原因は、`TrackHomeActionUsecase` が `knowledge_events` への append だけを行い、`knowledge_home_items.dismissed_at` の更新を `knowledge-home-projector` の後続反映に委ねていたためである。projector が backlog や polling interval の影響を受けると、ユーザーの dismiss 意思表示と read model の可視性更新に時間差が生じる。

Knowledge Home は append-first / reproject-safe を基本原則としているため、event log を捨てて read model だけを更新する設計は採れない。一方で、dismiss は UX 上「リロードしても隠れたまま」が強い期待として存在し、他の summary/tag 更新より即時性要件が高い。

## Decision

`dismiss` だけを対象に、event append 後に read model を同期更新する write-through fast-path を採用する。

具体的には以下を行う。

1. `TrackHomeActionUsecase` に `DismissKnowledgeHomeItemPort` と active projection version の取得口を追加する
2. `dismiss` のときだけ `knowledge_events` append 後に `knowledge_home_items.dismissed_at` を同期更新する
3. projector による `HomeItemDismissed` 投影は残し、event replay / reproject-safe の責務は引き続き projector が担う
4. `DismissKnowledgeHomeItem` が 0 rows update を返した場合は silent success にせず、`ErrDismissTargetNotFound` として観測可能にする

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|--------|------|-----------|
| projector のみで可視性更新を続ける | 原則的には最も単純 | steady-state でも polling latency 分の UX 劣化が残る |
| 全 Home action を write-through に広げる | 一貫性は高い | usecase が projection 詳細を知りすぎる。dismiss 以外は即時性要件が弱い |
| read model update を event append より前に行う | 画面上は速い | append-first の責務順序が崩れ、失敗時整合性が悪化する |

## Consequences

### Pros

- dismiss 直後のリロードでもカードが再表示されにくくなり、ユーザー期待に合う
- event log は引き続き source of truth であり、reproject-safe を壊さない
- 0 rows update を distinguish することで、projection version mismatch や row 未生成を見逃しにくくなる
- 変更は `dismiss` に限定され、他 action の責務拡散を最小限に抑えられる

### Cons / Tradeoffs

- usecase が active projection version を知るため、projection 事情が write path に少しにじむ
- `dismiss` だけ projector 依存と write-through を併用するため、他 action より実装が特別扱いになる
- row 未生成時は non-fatal に握る設計のため、完全な即時反映保証ではなく observability 前提の運用になる
