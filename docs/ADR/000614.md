# ADR-000614: Ask Augur chat proxy を /api/chat + think=false に移行

## Status

Accepted

## Date

2026-04-03

## Affected Services

- news-creator (Python) — chat proxy driver の変更
- rag-orchestrator (Go) — コンテナリビルドによるモデル名反映

## Context

[[000613]] で Gemma3 から Gemma4:E4B Q4_K_M へ全サービスを移行した。この際、Gemma4 が `/api/generate` でデフォルトで thinking モードを有効にするため、`raw=true` を追加して Ollama のテンプレートエンジンをバイパスする回避策を採用した。

しかし、この `raw=true` アプローチには構造的な脆弱性がある:

1. **手動テンプレート構築の維持コスト**: `_build_prompt_from_messages()` が `<start_of_turn>`/`<end_of_turn>` のモデル固有タグをハードコードしており、モデル変更のたびに壊れるリスクがある
2. **フォーマット変換の複雑性**: `/api/generate` のレスポンス形式（`response` フィールド）を `/api/chat` 形式（`message.content`）に変換する中間層が必要
3. **Ollama 側の制約**: GitHub issue [#14793](https://github.com/ollama/ollama/issues/14793) で報告されているように、`/api/generate` は `think` パラメータを無視する場合がある

加えて、[[000613]] のコミット後に rag-orchestrator コンテナがリビルドされず、旧モデル名 `gemma3-4b-12k` を送信し続けて Ask Augur が 2 回連続で失敗するインシデントが発生した（PM-2026-016）。

## Decision

**news-creator の chat proxy を `/api/generate` + `raw=true` から `/api/chat` + `think=false` に移行する。**

### 変更前

```
rag-orchestrator → /api/chat → news-creator proxy
  → _build_prompt_from_messages() でテンプレート手動構築
  → /api/generate (raw=true) → news-creator-backend (Ollama)
  → レスポンスを /api/chat 形式に変換
```

### 変更後

```
rag-orchestrator → /api/chat → news-creator proxy
  → messages をそのまま転送
  → /api/chat (think=false) → news-creator-backend (Ollama)
  → レスポンスをそのまま返却
```

### 検討した代替案

1. **`raw=true` を維持し Gemma4 テンプレートに更新**: Modelfile で定義済みのテンプレートと手動構築が二重管理になる。モデル変更時の変更箇所が増え、PM-2026-008 のような不整合リスクが残る
2. **Modelfile に `PARAMETER think false` を追加**: Modelfile レベルで thinking を無効化できるが、リクエスト単位の制御ができず柔軟性が低い

### 付随する修正

- handler の非ストリーミングパスに `think` パラメータの転送が欠落していたバグを修正
- `_build_prompt_from_messages()` メソッドとそのテストを削除（デッドコード除去）

## Consequences

### Pros

- **モデル変更に強い**: Ollama がモデル固有のチャットテンプレートを適用するため、モデル名を変更するだけで新しいテンプレートが自動適用される
- **コード量の削減**: 手動テンプレート構築 + レスポンス変換の約 40 行が不要に
- **PM-2026-008 の再発防止**: chat パスと batch パスの options 統一は `_merge_options()` で維持しつつ、テンプレート不整合のリスクを排除
- **`think` パラメータの明示的制御**: リクエスト単位で thinking の有効/無効を切り替え可能

### Cons / Tradeoffs

- **Ollama `/api/chat` の信頼性への依存**: 過去に Gemma モデルで `/api/chat` がハングする問題があった。現行 Ollama バージョンでは解消されているが、回帰の可能性はゼロではない
- **要約パスは `raw=true` のまま**: バッチ要約パス（`ollama_gateway.py` の `generate()`）は引き続き `/api/generate` + `raw=true` を使用。chat パスとの非対称性が残る
