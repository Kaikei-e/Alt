# ADR-000425: Knowledge Home UI を v0 として再構成し action center・summary chip・統合入力を導入する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-frontend-sv (TypeScript/SvelteKit) — TodayBar, KnowledgeCard, QuickActionRow, MiniRecallPanel, UnifiedIntentBox, SummaryStateChip, KnowledgeHomeSkeleton, Sidebar, +page.svelte

## Context

Phase 1–5 で Knowledge Home の基盤は完成し、backend は `summary_excerpt`, `tags`, `why[]`, `score`, `supersede_info`, `recall_candidates` を返していた。しかし UI レビュー（`docs/plan/IMPL_PHASE6.md`）により、フロントエンドが「ちょっと賢い記事一覧」に留まっていることが判明した。

具体的な課題:

| 問題 | 詳細 |
|------|------|
| TodayBar がフラット | stat chips + tags + shortcuts が 1 行に並ぶだけで、Alt の機能群（Morning Letter, Pulse, Recap）への導線が弱い |
| Search / Ask が周辺的 | 検索は `w-32` の小さな入力欄で右端に配置、Ask Augur はアイコンのみ |
| `"Summarizing..."` が本文位置に表示 | excerpt がない全カードが「まだ処理中」に見え、ページ全体が未完成に映る |
| QuickActionRow がアイコンのみ | ツールチップに頼る操作で、各ボタンの機能が発見しにくい |
| MiniRecallPanel が空に近い | topTags と Recap/Pulse リンク 2 本だけ。Morning Letter、Ask、Explore セクションなし |
| カードレイアウトが低密度 | tags と actions が別行で垂直に並び、1 カードあたりの高さが大きい |

Phase 6 の設計方針として「Home はポータルにしない — stream が主役」「why surfaced を捨てない」「backend truth がない状態名を断定しすぎない」が確認された。これらの方針の下で、既存 backend contract でできる UI 改善を最大化する。

## Decision

**フロントエンドの 7 コンポーネントを再構成し、Knowledge Home を「記事一覧」から「知識の起点」に変える。Backend 変更は [[000424]] で先行実装済みの `summary_state` / `need_to_know_count` のみ使用する。**

### 1. TodayBar を 2 段構造にリデザイン

**ファイル**: `TodayBar.svelte`

**変更前**: stat chips + tags + search + shortcuts が 1 行にフラット配置

**変更後**:
- **Row 1 (Action Shortcuts)**: Morning Letter（常時表示）、Pulse（`eveningPulseAvailable` で制御、NeedToKnow badge 付き）、Recap（`weeklyRecapAvailable` で制御）。無効時はグレーアウト表示
- **Row 2 (Stats + Tags)**: new / summarized / pending chips + topTags（右寄せ）

Search は TodayBar から分離し、専用の UnifiedIntentBox に移行する。

**Morning Letter の常時表示**: `/recap/morning-letter` route は常に存在するため、backend flag 不要で常にリンクを表示する。Pulse / Recap は既存の `eveningPulseAvailable` / `weeklyRecapAvailable` で conditional 表示を維持する。

**NeedToKnow badge**: `digest.needToKnowCount > 0` の場合に Pulse リンク横にオレンジの件数バッジを表示。

### 2. UnifiedIntentBox を新設

**ファイル**: `UnifiedIntentBox.svelte`（新規）

1 つの入力欄で Search（Enter キー）と Ask（ボタン）を統合する。

- placeholder: `"Search articles or ask a question..."`
- Enter → `/feeds/search?q={encoded}`（既存ルート）
- Ask ボタン → `/augur?q={encoded}`（query 付き）または `/augur`（空の場合）
- TodayBar と LensSelector の間に配置（`+page.svelte` で挿入）

**検討した代替案: TodayBar 内に残す方式。** Search/Ask は TodayBar の metrics 表示とは独立した操作であり、視覚的な階層も異なる。独立コンポーネントにすることでモバイルでの表示制御も容易になる。

### 3. KnowledgeCard を 3 レーン化 + SummaryStateChip

**ファイル**: `KnowledgeCard.svelte`, `SummaryStateChip.svelte`（新規）

**変更前**: title → badges → excerpt/"Summarizing..." → tags → actions の 5 段垂直レイアウト

**変更後**:
- **Header**: title + relative time（変更なし）
- **Badges row**: WhySurfacedBadge + SummaryStateChip（`pending` 時のみ表示）
- **Summary area**: `summaryState` による 3 パターン分岐
  - `ready`: excerpt テキスト表示
  - `pending`: 2 行 skeleton + SummaryStateChip（"Summarizing" pill）
  - `missing`: 2 行 skeleton のみ
- **Bottom row**: tags（左寄せ）+ QuickActionRow（右寄せ）← 同一行に配置

Bottom row の tags と actions の統合により、1 カードあたりの高さが約 20% 削減される。

**SummaryStateChip**: `pending` のみ表示。`Loader` アイコン + "Summarizing" テキストの小さな pill。`ready` / `missing` では何も表示しない。`missing` に断定的な表示（例: "No summary"）を出さないのは、設計方針「backend truth がない状態名を断定しすぎない」に従うため。

### 4. QuickActionRow にテキストラベル追加

**ファイル**: `QuickActionRow.svelte`

**変更前**: 4 つの icon-only ボタンが等間隔で並ぶ

**変更後**:
- Primary actions（Open / Ask / Listen）: icon + label 表示（md 以上の画面幅で label 表示）
- Dismiss: icon-only、右端にマージン分離。破壊的アクションとして視覚的に区別

ラベルの追加により、ツールチップなしで各アクションの機能を理解できる。

### 5. MiniRecallPanel を 3 セクション構成に強化

**ファイル**: `MiniRecallPanel.svelte`

**変更前**: topTags + Recap/Pulse リンク 2 本のみ

**変更後**:
- **Today's Tags**: digest.topTags を表示（変更なし）
- **Quick Links**: Morning Letter（常時）、Recap（conditional）、Pulse（conditional）、Ask Augur
- **Explore**: Tag Trail、Tag Verse リンク（既存ルートを活用）

全セクションは `divide-y` で区切り、視覚的なまとまりを出す。RecallRail が有効で候補がある場合は `+page.svelte` が MiniRecallPanel の代わりに RecallRail を表示する挙動は維持。

### 6. Sidebar に Today badge 追加

**ファイル**: `Sidebar.svelte`

Knowledge Home メニューアイテムに、非アクティブ時に `"Today"` テキストバッジを表示。アクティブ時は既存の active スタイルが適用されるためバッジは非表示。

### 7. KnowledgeHomeSkeleton の更新

**ファイル**: `KnowledgeHomeSkeleton.svelte`

新しいレイアウト構造（2 段 TodayBar + UnifiedIntentBox + 3 レーンカード）に合わせて skeleton プレースホルダーを更新。

### Props 契約の維持

Step 1 では既存の props 契約を変更しない:
- `TodayBar`: `digest: TodayDigestData | null`（`needToKnowCount` が追加されただけ）
- `KnowledgeCard`: `item: KnowledgeHomeItemData` + `onAction`（`summaryState` が追加されただけ）
- `QuickActionRow`: `itemKey, itemType, articleId, onAction`（変更なし）
- `MiniRecallPanel`: `digest: TodayDigestData | null`（変更なし）

文脈付き action（Ask に title を渡す、Listen に excerpt を渡すなど）は Step 2 で callback 契約を拡張する際に対応する。

## Consequences

### Pros

- 1 カードに「なぜ出たか」「何が書いてあるか」「次に何するか」が明確に配置される
- `"Summarizing..."` が本文の代替テキストではなく、状態を示す chip / skeleton に移行し、UX の誠実さが向上
- TodayBar が action center として機能し、Alt の既存機能（Morning Letter, Pulse, Recap, Ask）への導線が常時表示される
- Search/Ask が統合入力に昇格し、Knowledge Home から Alt 全体の機能にアクセスする起点になる
- MiniRecallPanel が空箱でなくなり、右 rail に Explore セクションが加わる
- 全 7 コンポーネントの既存 props 契約を維持しており、hook / page レベルの変更が最小限

### Cons / Tradeoffs

- **Step 2 への依存**: 文脈付き action 遷移（Ask に article context を渡す、Listen で TTS を発火するなど）は未実装。現時点では `/augur` に遷移するだけで article context は渡されない
- **NeedToKnow badge の近似値**: ページング第 1 ページの items からカウントするため、全 item の正確な件数ではない（[[000424]] で述べた制約と同一）
- **QuickActionRow のラベル表示**: `md` 未満の画面幅ではラベルが非表示になりアイコンのみに戻る。モバイルではタッチ操作で補われるため許容
- **Skeleton の "Summarizing" chip は `pending` のみ**: `missing` 状態には何も表示しない。将来的に `missing` が長期間続く場合の UX（例: "Summary unavailable"）は Step 3 以降で検討
