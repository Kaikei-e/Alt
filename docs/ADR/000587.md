# ADR-000587: Starfish Retrospective によるシステム・開発フロー振り返りを実施する

## Status

Accepted

## Date

2026-03-26

## Affected Services

- 全サービス（横断的なレビュー）

## Context

Alt プロジェクトは 2025 年 5 月の開始から 10 ヶ月で 586 件の ADR、8 件のポストモーテム、複数のアーキテクチャレビューを蓄積した。2026-03-23 にポストモーテム記述プラクティスを導入したことで、長期間潜在していた構造的問題（GPU 競合、OOM 連鎖、無限ループ等）が一斉に可視化された。

これらの資産を横断的に分析し、プロジェクトの強みを言語化し、構造的課題に優先度をつける必要があった。

## Decision

**Starfish Retrospective**（Keep / More / Less / Stop / Start の 5 次元）を基盤とし、Systems Thinking による構造分析を加えた振り返りフレームワークを採用する。

振り返りの成果物を `docs/retrospective/` ディレクトリに保存する運用を開始する。

### 分析スコープ

- ADR 586 件（全件の frontmatter + サンプル全文読み込み）
- ポストモーテム 8 件（全文読み込み）
- アーキテクチャレビュー（2026-01-12）、耐障害性レビュー（2026-02-12）
- Knowledge Home 是正指示書（2026-03-18）
- 開発履歴レビュー（初期・中期・デプロイメント時代）

### 主要な発見

**守るべき強み:**
1. ADR + PM + Review の三位一体ドキュメンテーション
2. TDD + Clean Architecture の全 5 言語一貫適用
3. HybridPrioritySemaphore の障害駆動進化

**最優先の構造的課題（P0）:**
1. PM アクションアイテム 50+ 件の未着手
2. 全サービスの mem_limit 未設定
3. Docker healthcheck の網羅化不足
4. ClickHouse TTL 1 日（インシデント調査不能）

**重要な補正:** 8 件の PM が 3 日間に集中したのは「障害の集中」ではなく「PM 文化導入により長期潜在の問題が初めて可視化された」結果。PM 文化の導入自体が Start Doing の成功事例。

## Consequences

### Pros

- プロジェクトの意思決定資産を定量的に分析した初めてのドキュメントが生まれた
- 構造的課題に P0/P1/P2 の優先度がつき、次のアクションが明確になった
- Obsidian vault に `retrospective/` ディレクトリを新設し、定期的な振り返りの基盤ができた

### Cons / Tradeoffs

- 振り返りは時点のスナップショットであり、継続的な追跡メカニズム（PM アクションアイテムの棚卸し等）は別途必要
- Starfish の 5 次元は行動レベルの改善に強いが、アーキテクチャレベルの構造変更（共有 DB 分離等）の優先度判断には追加のフレームワークが必要
