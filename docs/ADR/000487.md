# ADR-000487: 自動全文取得で記事作成を成功時に限定し手動Fetchを維持する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-backend (Go)

## Context

自動全文取得を導入する際、RSS item の description をそのまま article に保存するフォールバックを許すと、article の中身が「全文記事」と「RSS 抜粋」で混在する。これは Alt の article を下流で再利用する経路に悪影響を与える。

特に以下の点が問題になる。

1. **`articles.content` の意味が曖昧になる** — 全文取得済みコンテンツと RSS 抜粋が混在すると、article の契約が崩れる
2. **下流利用の品質が不安定になる** — RAG や要約、検索で article 本文の長さと密度が大きく揺れる
3. **手動 Fetch の役割が消える** — 自動失敗時にも中途半端な article を作ると、明示的な再取得手段としての手動 Fetch の位置付けが弱くなる

## Decision

自動全文取得では、全文 fetch と抽出が成功した場合にのみ article を作成し、失敗時は article を作成しない。手動 Fetch は独立経路としてそのまま維持する。

### 実装の概要

- 自動経路では HTML fetch、title 抽出、article 本文抽出のいずれかが失敗した場合、その item の article 作成を中止する
- RSS description や summary へのフォールバック保存は行わない
- 手動 Fetch は既存の API / usecase を変更せず、必要時の明示的な再取得手段として残す
- 自動作成後の RAG upsert は維持するが、article 作成に失敗したケースでは下流処理に進めない

### 採用理由

- `articles.content` を「取得済み本文」として一貫させられる
- 自動処理の失敗を silent fallback で隠さず、手動介入の余地を残せる
- 手動 Fetch の UX と運用判断を壊さずに自動化だけを足せる

### 代替案

- **RSS 抜粋を article として保存する案**  
  article の見た目は増えるが、本文品質が混在し、後段利用時の契約が崩れるため採用しない。

- **自動リトライキューを追加する案**  
  将来の改善候補ではあるが、まずは「成功時のみ保存」の明確な意味論を固定することを優先し、初回実装では採用しない。

## Consequences

### Pros

- article 作成の意味論が単純になり、下流利用で本文品質を前提にしやすい
- 自動失敗時に不完全な article が増えない
- 手動 Fetch の再取得経路を明確に維持できる

### Cons / Tradeoffs

- 自動取得に失敗した RSS item は article 一覧に現れない
- ユーザーが必要な item を後から手動で取りにいく場面が残る
- 「まず一覧に出す」ことを優先する設計ではないため、件数ベースでは article 作成率が下がる可能性がある
