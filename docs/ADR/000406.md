# ADR-000406: Knowledge Home Admin UI を SSR 初期ロード + API ポーリングに再設計する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-frontend-sv (SvelteKit) — ルート移動、SSR load 関数、API route、composable 再設計

## Context

[[000404]] で導入した Knowledge Home Admin ページには、初期実装の段階でいくつかの課題があった:

1. **クライアントサイドのみでデータ取得** — `useKnowledgeHomeAdmin` composable が `createClientTransport()` を内部で呼び出し、ブラウザから直接 Connect-RPC API を叩いていた。初回表示が遅く、SSR の恩恵がなかった。

2. **transport のハードコード** — composable が transport を内部生成していたため、サーバーサイドの transport（`$env/dynamic/private` を使う BFF 向け）との使い分けができなかった。

3. **重複リクエスト** — ポーリングの `setInterval` と手動リフレッシュが同時に走ると、同一データへの重複リクエストが発生し得た。

4. **ルート配置** — `/settings/knowledge-home-admin` は Settings カテゴリだったが、admin ロール制御の導入（[[000405]]）に伴い、`/admin/knowledge-home` に移動する方が意味的に正しい。

5. **プレースホルダーデータ** — ReasonDistributionChart と InteractionFunnelPanel はハードコードされた静的データで、実データとの乖離がミスリーディングだった。

## Decision

**composable を fetcher DI パターンに改修し、SSR 初期ロード + ブラウザ API ポーリングの二段階データ取得に再設計する。ルートは `/admin/knowledge-home` に移動し、旧 URL は 307 リダイレクトで転送する。**

### アーキテクチャ変更

**Before（クライアントサイド完結）:**
```
ブラウザ → createClientTransport() → BFF → alt-backend
```

**After（SSR + クライアントポーリング）:**
```
[初回] SvelteKit server → BFF (backendToken) → alt-backend
          ↓ PageData として hydrate
[ポーリング] ブラウザ → /api/admin/knowledge-home → SvelteKit server → BFF → alt-backend
```

### 主要な変更

**1. `useKnowledgeHomeAdmin` の fetcher DI 化**

```typescript
// Before: transport をハードコード
export function useKnowledgeHomeAdmin() {
  const transport = createClientTransport();
  // ...
}

// After: fetcher を外部から注入
export function useKnowledgeHomeAdmin(fetcher: () => Promise<Snapshot>) {
  // ...
  const snapshot = await fetcher();
}
```

これにより composable はデータ取得の「方法」を知らず、ページ側が fetcher を決定する。テストでは mock fetcher を渡すだけで検証可能になった。

**2. `seed()` メソッドによる SSR データの hydrate**

SSR の `+page.server.ts` で取得した初期データを、`seed()` メソッドで composable に注入する:

```typescript
$effect(() => {
  admin.seed(data.adminData, data.error ? new Error(data.error) : null);
});
```

初回表示は SSR データで即座に描画され、以降のポーリングは `fetchData()` 経由でインクリメンタルに更新する。

**3. 重複リクエスト防止**

`inFlight` Promise を保持し、既にリクエストが進行中の場合は同じ Promise を返す:

```typescript
const fetchData = async () => {
  if (inFlight) return inFlight;
  inFlight = (async () => { /* ... */ })();
  return inFlight;
};
```

**4. stale-while-revalidate パターン**

`refreshing` フラグでポーリング中であることを UI に伝えつつ、既存データはそのまま表示する。fetch 失敗時も既存データを保持し、`error` のみ更新する。

**5. サーバーサイド API route**

`/api/admin/knowledge-home` という JSON API route を新設し、ブラウザのポーリングはこのエンドポイントを叩く。SvelteKit のサーバーサイドで `backendToken` を使って BFF に Connect-RPC リクエストを送るため、ブラウザにはサービストークンが露出しない。

**6. ルート移動とリダイレクト**

- 新ルート: `/admin/knowledge-home` — admin カテゴリの専用パス
- 旧ルート: `/settings/knowledge-home-admin` → 307 リダイレクトで新ルートに転送
- サイドバー: Settings > KH Admin から Admin > Knowledge Home Ops に移動

**7. プレースホルダーの除去**

ReasonDistributionChart と InteractionFunnelPanel はハードコードデータで描画していたため、ページ構成から除去した。実データの集計エンドポイントが実装された段階で再導入する。

### テスト

`useKnowledgeHomeAdmin.test.ts` を新設:
- **stale data 保持** — fetch 失敗時に既存データが維持されることを検証
- **refreshing 状態** — fetch 中は `refreshing=true` かつ既存データが表示されることを検証

`user-role.test.ts` を新設:
- `getUserRole()` の null / 空 traits / 不正値ケースをカバー

## Consequences

### Pros

- **SSR 初回表示** — サーバーサイドでデータを取得するため、Admin ページの初回表示が即座にデータ付きで描画される（loading スピナーなし）
- **トークン非露出** — ブラウザから BFF/backend への直接 Connect-RPC 呼び出しがなくなり、`backendToken` がクライアントに露出しない
- **テスタビリティ** — fetcher DI により、composable のテストが mock fetcher だけで完結する
- **重複リクエスト防止** — `inFlight` パターンにより、ポーリングと手動リフレッシュの競合が解消された
- **stale-while-revalidate** — ポーリング失敗時も最後に取得したデータを表示し続ける。UX の安定性が向上

### Cons / Tradeoffs

- **API route の追加** — `/api/admin/knowledge-home` という中間エンドポイントが増えた。SvelteKit サーバー → BFF → backend の 3 hop になるが、admin ページの頻度では問題にならない
- **SSR の負荷** — Admin ページの初回ロード時にサーバーサイドで BFF に Connect-RPC リクエストを送る。ただし admin ユーザーのアクセス頻度は低いため影響は限定的
- **Reason/Funnel の一時撤去** — プレースホルダーデータを除去したため、Admin ダッシュボードの情報量が一時的に減った。実データ集計の実装で復活予定
