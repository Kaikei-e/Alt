# ADR-000741: 全内部サービスの受信側 mTLS を peer-identity allowlist 方式で統一する

## Status

Accepted

## Date

2026-04-15

## Affected Services

- search-indexer (Go) — `middleware/peer_identity_middleware.go` 新設、`bootstrap/servers.go::newMTLSMuxHandler` で REST + Connect-RPC を `:9443` に統合
- alt-backend (Go) — 既存 `middleware/peer_identity_http_middleware.go` を引き続き使用 ([[000727]])
- acolyte-orchestrator (Python/Starlette) — nginx sidecar を `VERIFY_CLIENT=on` 昇格 + `X-Alt-Peer-Identity` 転送、`acolyte/infra/peer_identity.py` ASGI middleware 新設
- tag-generator (Python/FastAPI) — sidecar `VERIFY_CLIENT=on` + `tag_generator/infra/peer_identity.py`
- news-creator / recap-subworker / recap-evaluator (Python/FastAPI) — `infra/peer_identity.py` (copy パターン)
- rag-orchestrator (Go) — `internal/middleware/peer_identity.go` + test
- mq-hub (Go) — `middleware/peer_identity.go` + test
- recap-worker (Rust) — `src/tls.rs` で `axum-server::bind_rustls` による `:9443` mTLS listener を追加
- `docker/nginx-tls-sidecar/{nginx.conf.template,entrypoint.sh}` — `VERIFY_CLIENT=on` 時に `$ssl_client_s_dn_cn` を `X-Alt-Peer-Identity` として upstream に転送

## Context

[[000725]] → [[000727]] → [[000729]] で Go 5 サービス (alt-backend / auth-hub / pre-processor / search-indexer / alt-butterfly-facade) の `:9443` mTLS リスナーと BFF outbound mTLS までは永続化済だった。しかし以下が残っていた:

- **search-indexer の REST `/v1/search` は plaintext `:9300` のまま**。Connect-RPC 側は peer allowlist で gated だが REST は shared-secret (`X-Service-Token`) 依存 ([[PM-2026-025]] の発火経路)
- **Python サービス群 (news-creator / tag-generator / recap-subworker / recap-evaluator / acolyte-orchestrator) の受信側が HTTP-only**
- **Acolyte の nginx TLS sidecar は `VERIFY_CLIENT=off`** で client cert を検証していなかった (ADR-000729 で指摘済み)
- **recap-worker (Rust) の axum サーバに mTLS listener が無い** (ADR-000727 で先送り)
- **rag-orchestrator / mq-hub (Go) に peer-identity 基盤が無い**
- **`X-Alt-Peer-Identity` header 契約が未定義** — alt-backend は CN をログするのみで、Python/他サービスは header すら受けない

これらを残したまま X-Service-Token を撤去すると、境界認証が完全に消える。受信側 mTLS を全サービスに揃えてから、送信側 ([[000742]]) とアプリ層 token 撤去 ([[000743]]) に進む段取りが本 ADR のスコープ。

## Decision

**東西通信の全受信エンドポイントを `:9443` mTLS に統一し、caller の CN を `X-Alt-Peer-Identity` header で応用層に伝播する。** サービスごとに以下を適用:

### Go サービスの peer-identity middleware

`search-indexer` / `rag-orchestrator` / `mq-hub` の各 `middleware/peer_identity*.go` は同一設計:

- `tls.ConnectionState.PeerCertificates[0].Subject.CommonName` を allowlist と照合
- 不在 → 401、非 allowlist → 403
- 通過時は `r.Header.Set(PeerIdentityHeader, cn)` で **spoofed 値を上書き**してから次へ
- 5 ケース table-driven test (allow / deny / no-TLS / empty allowlist / spoof 防止)

search-indexer は `bootstrap/servers.go::newMTLSMuxHandler` で REST mux と Connect-RPC mux を `peer.Require(...)` で包んだ単一 `:9443` handler に統合。`:9300` / `:9301` の plaintext listener は [[000743]] まで並行温存。

### Python サービスの ASGI peer-identity middleware

`PeerIdentityMiddleware(BaseHTTPMiddleware)` を各サービスの `infra/peer_identity.py` に複製配置。挙動:

- nginx sidecar が `VERIFY_CLIENT=on` 時に付与する `X-Alt-Peer-Identity` header を読む
- 環境変数 `PEER_IDENTITY_TRUSTED` が `on` でない場合は header を破棄 (sidecar バイパス攻撃への防御)
- strict mode では unauth/not-allowlisted を 401/403 で拒否。`strict=False` で段階移行
- `request.state.peer_identity` に bind、`structlog.contextvars` に `peer` を入れるので全ログに caller CN が出る

5 サービスに個別コピーしている理由: Python の共有 package 化は各サービスの dependency injection が別々で、shared wheel を管理するより copy が軽量と判断。将来 6 サービス目で共通パッケージ化を検討。

### nginx sidecar の `VERIFY_CLIENT=on` 昇格と CN 転送

`docker/nginx-tls-sidecar/entrypoint.sh` を改修し、`VERIFY_CLIENT=on` の時に `PEER_IDENTITY_VALUE='$ssl_client_s_dn_cn'` を envsubst で `nginx.conf.template` の `proxy_set_header X-Alt-Peer-Identity ${PEER_IDENTITY_VALUE}` に注入。off 時は空文字で安全側に倒す。

acolyte-orchestrator / tag-generator の compose default を `VERIFY_CLIENT=on` に昇格。upstream の Python app は上記 middleware で CN を参照できる。

### recap-worker (Rust) の axum-server mTLS

`src/tls.rs` を新設し `load_server_tls_config()` で rustls `ServerConfig` を構築 (`WebPkiClientVerifier` で client cert 必須)。`main.rs` で `axum_server::bind_rustls(addr, RustlsConfig::from_config(..))` を `MTLS_ENFORCE=true` 時に plaintext `TcpListener` と並列起動。依存に `axum-server` (`tls-rustls` feature) / `rustls` / `rustls-pemfile` / `tokio-rustls` を追加。7 ケース unit test (fail-closed / allowlist parse / disabled 時の None return 等)。

### 検討した代替案

- **案 A: nginx sidecar を撤去し uvicorn 単体で mTLS 終端** — uvicorn の `--ssl-cert-reqs=2` は TLS は検証するが peer cert を ASGI scope に伝播しないため、CN 取得に hypercorn 等への乗り換えが必要。スコープ拡大するので不採用
- **案 B: SPIFFE/SPIRE 導入** — Docker Compose + step-ca の既存投資 ([[000725]]) を捨てるコストが高い。CN based allowlist で当面十分
- **案 C: `X-Alt-Peer-Identity` を内部 TLS 終端で書き換えず、app が `r.TLS.PeerCertificates[0]` 直接読み** — nginx sidecar 経路では app が TLS peer を見れないため header 迂回が必要。混在環境を考えて header プロトコルに統一

## Consequences

### Pros

- 全内部受信エンドポイントが mTLS + CN allowlist で統一される土台が揃う
- `X-Alt-Peer-Identity` header の契約が 3 言語 (Go / Python / Rust) で統一。ログ・audit・allowlist enforcement のインタフェースが共通
- spoofed header の ingress 防御を全 middleware で実装 (CN 値の上書き + sidecar-off 時の破棄)
- PM-2026-025 型の「認証ヘッダ取り外し」攻撃経路が構造的に消える (caller cert を持たないと TLS handshake が通らない)

### Cons / Tradeoffs

- **Python の peer-identity middleware を 5 箇所にコピー**。一貫性保持の運用コストは残る。6 箇所目が出たら shared package 化の trigger とする
- **plaintext :9300 / :9301 / :8080 等の listener が並存**。mTLS cutover ([[000743]]) まで dual-stack で運用する必要があり、観測対象が倍増
- **recap-worker の Rust 受信側は新規コードパス**で、axum-server 公式の crypto provider 初期化 quirk に将来ハマる可能性 (rustls 0.23 は `install_default()` を要求する場合がある)
- nginx sidecar の netns 共有問題 ([[000729]]) は本 ADR では解消せず、運用手順 (`docs/runbooks/mtls-cutover.md`) で吸収
