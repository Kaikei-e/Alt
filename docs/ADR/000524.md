# ADR-000524: フィード同期の article lifecycle event を upsert 結果に追従させる

## Status

Accepted

## Date

2026-03-21

## Affected Services

- alt-backend (Go) — `RegisterFeedsUsecase` と `register_feed_gateway` の戻り値契約を拡張し、feed 同期時の event type を create/update に分岐する
- pre-processor (Go) — `ArticleUpdated` を受けても queue 作成しない既存 consumer 挙動の前提が、feed 同期経路でも有効になる

## Context

[[000512]] で「記事 upsert の結果を `ArticleCreated` と `ArticleUpdated` に分離する」という方針は受理済みだったが、feed 同期の write path はその意味論に完全には追従していなかった。

`CreateArticle` 系の RPC は新規 insert か update かを返し分けていた一方、`RegisterFeedsUsecase` は `RegisterFeeds` から記事 ID の配列だけを受け取り、feed 同期で再保存された既存記事にも `ArticleCreated` を発行していた。

この経路が残っている限り、`pre-processor` の queue admission guard があっても downstream に「新規記事としての通知」が流れ続ける。event 名と実際の意味がズレたままでは、ログ観測や downstream 設計で誤解を生み、stale re-enqueue の発火点も温存される。

実装後の起動確認では、再ビルドした `alt-backend` と `pre-processor` がともに healthy となり、実行中ログでも feed 同期由来の `ArticleUpdated` が `pre-processor` に流れていることを確認した。これにより、実際の運用経路でも create/update 分離が反映されたことが分かる。

## Decision

**feed 同期の article event は、upsert 結果の `created` フラグに追従して `ArticleCreated` または `ArticleUpdated` を発行する。**

今回の決定では以下を採用する。

1. `RegisterFeedsPort` の戻り値を `[]string` から `[]RegisterFeedResult` に変更し、`article_id` と `created` を同時に返す
2. `register_feed_gateway` は DB driver の batch upsert から `created` を受け取り、そのまま usecase 層へ伝搬する
3. `publishFeedEvents` は `created=true` のときだけ `PublishArticleCreated` を呼び、`created=false` では `PublishArticleUpdated` を呼ぶ
4. 既存の fire-and-forget と non-fatal error handling は維持し、失敗時は WARN ログのみに留める

### 採用しなかった代替案

| 案 | 評価 |
|---|---|
| `RegisterFeedsUsecase` だけで既存記事かどうかを別クエリで調べる | 追加 read が必要になり、batch upsert の原子的な結果より信頼性が低い |
| feed 同期だけ従来どおり `ArticleCreated` を維持する | `ADR-512` の意味論と矛盾し、event 名の汚染が残る |
| downstream だけで `ArticleUpdated` 相当を推論する | upstream の契約が曖昧なままで、再送や観測時の判断材料として弱い |

## Consequences

### Pros

- feed 同期でも `ArticleCreated` が create-only event に戻り、event 名と実際の出来事が一致する
- 既存記事の再保存時に `pre-processor` へ新規記事通知を流し続ける経路を閉じられる
- `CreateArticle` 系 RPC と `RegisterFeedsUsecase` の event semantics が揃い、write path ごとの差異が減る
- ログと運用観測で `ArticleUpdated` を明示的に追えるため、再同期由来の更新を切り分けやすい

### Cons / Tradeoffs

- `RegisterFeedsPort` の戻り値が変わるため、gateway / usecase / mock / test の追随コストが発生する
- `created` 判定を DB の upsert 結果に依存するため、driver 層の SQL 契約を不用意に変えにくくなる
- 現時点の `pre-processor` consumer は `ArticleUpdated` を処理しないため、update 系 downstream 活用は今後の課題として残る
