# ADR-000872: classifier sidecar と runtime embedder の identity drift を起動時 fail-closed で塞ぎ 3-day Recap の分類分布を回復させる

## Status

Accepted

## Date

2026-04-30

## Affected Services

- **recap-subworker (Python)** — classifier 起動時 fail-closed を sklearn version 一致 ([[000835]] stage 3) から embedder identity 一致まで拡張
- **deploy-side runtime configuration** — `.env` の `OLLAMA_EMBED_MODEL` を `bge-m3` に切替

## Context

2026-04-14 以降、3-day Recap のジャンル分類が silent に劣化したまま回復しなかった。直近 2 週間で 3 本の ADR を投入したが、いずれも症状の片側だけを塞ぐ修正だった。

- [[000835]] 17-class classifier × 17-key thresholds × `transformers v5` non-persistent buffer regression を 3 層不整合として診断
- [[000838]] 30-class JA classifier を BAAI/bge-m3 で再学習、`MIN_FALLBACK_DOCUMENTS 2 → 1` と subcluster singleton 降格
- [[000857]] production が古い 17-class joblib を読み続けていた事故を default path 切替で解消、threshold coverage validator + `CLASSIFIER_TOP_GENRE_TOTAL` Counter

これら 3 本の修正は正しく landed していたにも関わらず、本 ADR 執筆日 (2026-04-30) でも以下の症状が継続していた。

### DB 実観測 (recap-db, 2026-04-30)

直近完了 4 件の 3-day job の `recap_outputs`:

| job_id | rows | distinct genres |
|---|---:|---:|
| `de110830` (04-29 17:00 UTC) | 15 | 9 coarse + 6 subgenre |
| `b3265c5b` (04-28 17:00 UTC) | 13 | 9 coarse + 4 subgenre |
| `a093d808` (04-27 17:00 UTC) | 11 | 7 coarse + 4 subgenre |
| `afcc2f98` (04-27 13:31 UTC) | 10 | 7 coarse + 3 subgenre |

カバーされていた coarse genre 9 種: `consumer_tech`, `cybersecurity`, `film_tv`, `games_esports`, `health_medicine`, `markets_finance`, `music_audio`, `politics_government`, `software_dev`, `sports`。残り 21 種 (`ai_data`, `climate_environment`, `consumer_products`, `culture_arts`, `diplomacy_security`, `economics_macro`, `education`, `energy_transition`, `food_cuisine`, `home_living`, `industry_logistics`, `internet_platforms`, `labor_workplace`, `law_crime`, `life_science`, `mobility_automotive`, `society_demographics`, `space_astronomy`, `startups_innovation`, `travel_places`) は **全 job で 0 件**。

`recap_failed_tasks` は直近 5 日で `Missing from batch response × 46` (news-creator) と `No genres were stored: failed=N, skipped=0, no_evidence=20–23` を記録。2026-04-09 〜 04-13 の baseline (34–56 distinct genres / day) と比べて約 1/3 のジャンル網羅性。

### Root cause: 4 層目の silent fingerprint drift

[[000838]] 訓練時の sidecar `data/genre_classifier_ja.meta.json`:

```json
{
  "embedding_model_id": "BAAI/bge-m3",
  "feature_dim": 1224,
  "transformers_version": "5.6.1",
  "sentence_transformers_version": "5.4.1"
}
```

production runtime の `alt-recap-subworker-1` (image `sha-5dd79ea`、ADR-000857 反映済) の env:

```
RECAP_SUBWORKER_MODEL_BACKEND=ollama-remote
RECAP_SUBWORKER_OLLAMA_EMBED_MODEL=mxbai-embed-large
```

つまり LogReg head は **BGE-M3 (1024-dim, multilingual XLM-RoBERTa, CLS pooling, normalised)** の vector で訓練された一方、推論時には **mxbai-embed-large (1024-dim, English BERT-large, mean pooling)** の vector が流れていた。両者は次元こそ一致するが完全に別の vector space で、cosine / 内積も互換ではない (Mixedbread / BAAI の公式 docs に明記、加えて embedding 空間の不互換性は OpenAI Embeddings ガイドや Cisco の Embedding-model selection ガイドでも独立に確認できる)。LogReg head は cross-encoder vector を投入されると class prior 上位に縮退するという挙動をとるため、9 coarse genres への集中はその縮退の DB 側兆候として説明できる。

### 既存 validator はこの drift を見ていなかった

`recap_subworker/services/classifier.py:119` の `_guard_metadata_against_runtime` は `sklearn` の minor version しか比較せず、`ClassifierMetadata.embedding_model_id` フィールドは `from_json` でパースされるが下流で参照されていなかった。[[000857]] で追加した cardinality / threshold coverage validator も joblib 側 drift しか検知できず、embedding 空間 drift には盲目。これが 4-23 〜 4-30 の 7 日間 silent drift を許容した穴の正体。

## Decision

### 1. Runtime embedder を BGE-M3 に揃える (deploy-side)

Ollama embedder host で `ollama rm qwen3.5-27b:latest` (16.7 GB の dead weight) → `ollama pull bge-m3` (Q4 量子化版、約 1.2 GB on-disk) で在庫を入れ替えた上で、deploy-side `.env` の `OLLAMA_EMBED_MODEL` を `mxbai-embed-large` から `bge-m3` に切替。recap-subworker 1 サービスのみ `--force-recreate` で再起動して反映。RAG 側の `EMBEDDING_MODEL=embeddinggemma` (768-dim、Meilisearch hybrid index と vector DB が embeddinggemma 空間で構築済) は本 ADR の射程外として据え置き。

### 2. Sidecar↔runtime embedder identity guard を classifier loader に追加

`recap_subworker/services/classifier.py` の `_guard_metadata_against_runtime` を拡張し、第 2 引数として `Embedder | None` を受ける。runtime backend に応じた resolution map で effective identity を決定する:

- `model_backend == "sentence-transformers"` → `config.model_id`
- `model_backend == "onnx"` → `config.onnx_tokenizer_name or config.model_id`
- `model_backend == "ollama-remote"` → `config.ollama_embed_model`
- `model_backend == "hash"` → 比較スキップ (deterministic dev fixture、prod では使われない)

canonical 化したのち strict 一致しなければ `ConfigValidationError`。空 `embedding_model_id` (legacy artefact) は warn-only に留めて後方互換を保つ。

### 3. Boot-time peer validator で `Settings(...)` 構築前に弾く

`recap_subworker/infra/config.py` に `_validate_classifier_embedding_consistency` peer validator を新設。`classification_backend == "joblib"` のとき joblib path に対応する `<model>.meta.json` を offline で読み、(2) と同じ resolution map と canonicalisation で一致を検査する。Embedder インスタンス化を経ずに Settings レベルで raise するため、unhealthy な image が classify request まで到達せず boot 時 CrashLoop で止まる。`_validate_classifier_cardinality` ([[000835]] stage 3) と `_validate_threshold_coverage` ([[000857]]) の peer として配置。

### 4. Canonical alias map を共通化

`recap_subworker/infra/embedding_identity.py` を新設して `canonicalize_embedding_id(name) -> str` を公開、classifier loader と Settings validator の両方からインポート。alias は最小限の hand-curated table:

```python
_EMBEDDING_MODEL_ALIASES: dict[str, str] = {
    "baai/bge-m3": "baai/bge-m3",
    "bge-m3": "baai/bge-m3",
    "bge-m3:latest": "baai/bge-m3",
}
```

これにより HuggingFace 名 (`BAAI/bge-m3`) と Ollama tag (`bge-m3` / `bge-m3:latest`) が同一 artefact として扱われる。**異なる重みは絶対 alias に入れない** (`mxbai-embed-large` / `embeddinggemma` 等は別エントリにしない) ことを docstring と alias 辞書のコメントで明文化。

### 5. Operator escape hatch

`Settings.allow_embedding_drift: bool = Field(False)` (env `RECAP_SUBWORKER_ALLOW_EMBEDDING_DRIFT`) を追加。`True` のとき classifier loader と peer validator のいずれも warn のみで通す。Default `False` を確実にし、escape hatch は emergency rollback 専用とする旨を field description に明記。`EmbedderConfig.allow_embedding_drift` にも同フィールドを propagate し、`app/container.py` から `self.settings.allow_embedding_drift` を渡す。

### 検討した代替案と却下理由

- **(却下) 30-class classifier を `mxbai-embed-large` で再学習**: production env を一切変えずに済むが、CUDA runner + golden データ + sidecar 更新を要し重い。BGE-M3 artefact は Macro F1 0.90 / `feature_dim=1224` で既に揃っており、retrain ではなく runtime を戻す方が短く reversible。emergency rollback 経路としても escape hatch が同等の役割を果たす。
- **(却下) `transformers<5` への pin**: [[000857]] で却下済み。embedding fingerprint drift とは独立の問題で、本 ADR の主因ではない。
- **(却下) `current_thresholds.get(cls, 0.5)` の hard-coded fallback 撤去**: [[000857]] で却下済み。本 ADR の射程外。
- **(却下) sidecar 比較を runtime のみに絞る (Settings 側に peer validator を置かない)**: classifier loader だけで guard すれば最小実装。ただし Embedder インスタンス化を経ないと検査できないため、boot 序盤の Settings 構築時には不可視で、`uvicorn` は通常起動できてしまう。peer validator を追加することで CrashLoop が早く可視化される。コスト微増、value 大として採用。
- **(却下) alias map をファジー一致 (Levenshtein / 部分文字列) で拡張**: 将来的に `mxbai-embed-large` ↔ `mxbai-embed-large-v1` の同種等価を扱う際の便利さは見込める一方、別重み (例: `bge-m3` ↔ `bge-large-en`) を誤って同一視するリスクが大。explicit な hand-curated 表に閉じ、alias 拡張は別 PR / 別 ADR で議論する。
- **(却下) 768-dim embeddinggemma で classifier を再学習し RAG 系と embedder を統合**: vector store も使い回せて運用が単純化される利点はあるが、Meilisearch hybrid index / `rag-orchestrator/internal/backfill/hyperboost.go` の hyperboost flow が embeddinggemma 空間で構築済で、recap classifier だけ再学習しても vector dim が 1024 → 768 に縮退するためモデル容量の縮小と精度の trade-off を別途検証する必要がある。本 ADR では runtime を訓練時と一致させる最短路を採る。

## Consequences

### Pros

- **2026-04-14 以降 16 日続いた silent classification 縮退に明確な fail-closed boundary が引かれる**。runtime と sidecar の embedder identity が乖離していると recap-subworker は boot 時 `ConfigValidationError` で起動拒否、CrashLoop が直ちに operator に観測される。silent な品質劣化が 13 日後に DB 直接照会で初めて見つかる ([[000857]] context) 状況の再発を構造的に防ぐ。
- **既存 [[000835]] stage 3 sklearn-version guard と [[000857]] threshold coverage / cardinality validator の対称設計を完成させる**。3 つの fail-closed validator (cardinality / threshold coverage / embedding identity) が peer として並び、retrain workflow の各 dimension を独立に検査できる。
- **canonical alias map を `infra/embedding_identity.py` に切り出した**ことで、Ollama tag ↔ HuggingFace 名のような正当な等価ルールを 1 箇所に集約。classifier loader と Settings validator から共有され、片方だけ alias を更新する drift を防ぐ。
- **operator escape hatch (`RECAP_SUBWORKER_ALLOW_EMBEDDING_DRIFT`) は default `False`** で、明示 opt-in を要する。emergency rollback で意識的に flip する場面が想定でき、誤って常用される設計ではない。
- **recap_outputs の per-job genre count が baseline (29–39) 帯まで戻る前提条件が揃った**。runtime が BGE-M3 に戻ると LogReg head が訓練時と同じ vector space と再会し、coarse 9 種への集中が解け 30 class taxonomy 全域への分散が回復する。次回 17:00 UTC nightly job 後に DB で確認する。
- **TDD outside-in を遵守**。`tests/services/test_classifier_embedding_guard.py` (8 ケース) と `tests/unit/test_config_classifier_embedding_validator.py` (9 ケース) を RED 先行で commit、GREEN 実装後に既存 `tests/services/` `tests/unit/` の 370 件 + 12 skip が green を維持。

### Cons / Tradeoffs

- **alias map の拡張責務**: 新しい embedding model を fleet に追加するたびに `_EMBEDDING_MODEL_ALIASES` を更新する必要がある。fail-closed の代償として false-positive (例: 同重みなのに alias 未登録で boot 拒否) が起こる可能性。docstring に entry-add ルールを書いてあるが、運用上 onboarding ドキュメント (services/recap-subworker.md) への追記が follow-up として残る。
- **legacy sidecar の warn-only 経路**: `embedding_model_id` フィールドが空の sidecar (ADR-000835 stage 3 以前の artefact) は識別不能なため warn のみで通す。今後 retrain の度に `train.py:405` がフィールドを書く前提なので新規 artefact では問題にならないが、emergency で旧 artefact を踏むと guard が機能しない。retrain pipeline 側で fail-closed を強化するのは別 PR で扱う。
- **escape hatch の存在自体がリスク**: `RECAP_SUBWORKER_ALLOW_EMBEDDING_DRIFT=true` が誤って production env に残ると本 ADR の guard を完全に無効化する。deploy-side で env audit を追加するか、起動ログで `WARNING: embedding drift allowed by operator override` を強調表示する形で運用視認性を補う必要があるが、本 ADR では実装に含めない。
- **runtime embedder 切替に伴う Ollama side の在庫管理**: `qwen3.5-27b:latest` (16.7 GB) を削除して `bge-m3` (1.2 GB) を pull した。`qwen3.5-27b` は Alt monorepo の grep で参照ゼロのため dead weight と判断したが、もし deploy-side のスクリプトや別プロジェクトが暗黙参照していれば影響する。Acolyte の remote inference target が `gemma4:26b` を継続使用する事実は確認済。
- **recap-evaluator の G-Eval は依然 disabled** ([[PM-2026-038]] AI #7、`OLLAMA_URL` drift)。本 ADR の効果検証は `recap_outputs` 直接照会と `recap_subworker_classifier_top_genre_total` Prometheus Counter のラベル拡散に依存し、object な品質メトリクスが復活するまでは circumstantial。
- **`Missing from batch response × 46` の起源は本 ADR で塞がない**。news-creator 側の batch response 落ちが残れば classifier 復帰後も genre stored 数が伸び悩む可能性がある。embedder fingerprint 修復後に件数が縮退しなければ別 PR で news-creator pact / batch flow を追う。
