# ADR-000809: compose service_healthy 依存先に healthcheck を必須化し recap-subworker の空白を埋める

## Status

Accepted

## Date

2026-04-20

## Affected Services

- **Alt (compose)** — `compose/recap.yaml` の `recap-subworker` に healthcheck 追加
- **Alt (CI)** — `.github/workflows/compose-audit.yaml` 新設、healthcheck invariant を protective gate 化

## Context

2026-04-20 00:58 UTC、alt-deploy の `release-deploy` workflow_dispatch (sha=1018dc862) 実行中に `deploy` ジョブが以下で abort した:

```
 Container alt-search-indexer-1 Started
dependency failed to start: container alt-recap-subworker-1 has no healthcheck configured
##[error]Process completed with exit code 1.
```

根本原因は 2 層:

1. **recap-subworker に healthcheck が未定義**だった。他サービス (recap-worker 等) が `depends_on.recap-subworker.condition: service_healthy` を宣言しているにもかかわらず、compose 公式仕様 ([docker/compose#7681](https://github.com/docker/compose/issues/7681)) により「healthcheck 未定義のターゲットに service_healthy を張った瞬間 `docker compose up --wait` は非ゼロ終了」する。
2. この構成上の穴を **CI で検出する仕組みが無かった**。release-deploy が alt-prod runner で走ってはじめて気づき、partial-deploy で alt-butterfly-facade と pki-agent-recap-subworker を強制停止→削除→再作成未到達で消失させ、production で StreamFeedStats 500 を引き起こした ([[000810]] で deploy job 自体の resilience は別途対処)。

recap-subworker は Python / uvicorn / sentence-transformers で `/health` エンドポイント (`recap_subworker/app/routers/health.py:15`) を既に持っているため、compose healthcheck の記述欠落のみが問題だった。

同種の穴を再発させない invariant が必要。

## Decision

### 1. `recap-subworker` に healthcheck を追加

`compose/recap.yaml` の `recap-subworker:` ブロックに、既存の **`tag-generator`** (`compose/workers.yaml`) と同一の Python stdlib パターンで healthcheck を追加する:

```yaml
    healthcheck:
      test: ["CMD", "python", "-c", "import urllib.request; urllib.request.urlopen('http://localhost:8002/health')"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 90s
```

- **`urllib.request` stdlib** — recap-subworker の image は curl を含まない distroless 系。Python は必ず同梱されるため `urllib.request` が最も可搬的。
- **`start_period: 90s`** — recap-subworker は起動時に `sentence-transformers/multilingual-e5-large` をロードする。tag-generator (30s) より長めに振る。
- **dry-run 検証**: `docker exec alt-recap-subworker-1 python -c "..."` で 200 応答を確認済。

### 2. compose-audit CI ワークフロー新設

`.github/workflows/compose-audit.yaml` を追加し、**`depends_on.*.condition: service_healthy` を張ったエッジのターゲットが healthcheck を持つか**を Python で検証する。持たなければ `::error::` + exit 1。

```python
for name, svc in services.items():
    deps = svc.get("depends_on") or {}
    if not isinstance(deps, dict):
        continue
    for dep_name, dep_cfg in deps.items():
        if not isinstance(dep_cfg, dict):
            continue
        if dep_cfg.get("condition") != "service_healthy":
            continue
        target = services.get(dep_name)
        if target is None or target.get("healthcheck") is None:
            violations.append(...)
```

トリガーは `compose/**.yaml` と workflow 自身への push / PR。`docker compose config` で render してから判定するため include chain を横断した検証が 1 回で済む。

**却下した代替**:

- **「全 long-running サービスに healthcheck 必須」にする全面的 invariant** — `*-logs` (rask-log-forwarder sidecars), `nginx-exporter`, `dashboard`, `auth-token-manager` 等 20 超の長寿命サービスが現状 healthcheck 無しで、これらは `depends_on: service_healthy` のターゲットではない (= compose abort を起こさない)。全面要求は「abort 防止に不要な量の運用コスト」で却下。**abort を起こしうる service_healthy エッジのターゲットに限定**する方が surgical。
- **Dockerfile HEALTHCHECK に移譲** — 一貫性を compose 層で保つ方がデプロイ側から確認容易。
- **Ansible `wait_for_connection` に置換** — [[000810]] で別途 Ansible playbook 経路を入れるが、compose healthcheck 自体は `depends_on: service_healthy` の表現力として固有の役割がある。

### 3. compose-audit の実行条件と運用

- `on: push: branches: [main]` と `on: pull_request: branches: [main]` の両方でトリガー。トランクベース運用で `main` 直接コミットが主経路のため、両方必要。
- `paths:` 絞り込みで compose yaml と workflow 自身の変更だけに反応。
- `workflow_dispatch` で手動再実行も可能に。
- 実行時間 < 30 秒 (python + yaml 解析だけ)。

## Consequences

### Pros

- **`docker compose up -d --wait` の abort が構造的に起きなくなる** — service_healthy エッジの全参照先が healthcheck を持つことが CI で保証される。今回と同クラスの deploy 失敗が回帰したら Fix-C/F の reconcile 層に到達せず、main push / PR 段階で即 block される。
- **recap-subworker の健全性が observable になる** — healthcheck の pass/fail が `docker inspect` / compose ps から見え、次の deploy では service_healthy ターゲットとして正しく待機される。
- **CI audit が 30 秒で走る** — 運用コストほぼゼロ、false-positive 無し (具体エッジのみ対象)。
- **公式 Python stdlib のみ** — 追加 tool chain 不要、image bloat なし。
- **Fix-C/F の reconcile と二重防御** — 本 ADR が predictive gate、[[000810]] が reactive gate。二段構えで silent-drop を潰す。

### Cons / Tradeoffs

- **start_period=90s は実測チューニング必要** — multilingual-e5-large のロード時間は GPU/CPU モードで差があり、cuda 起動遅延時に 90s を超える可能性がある。start_period 期間中の healthcheck 失敗は unhealthy 判定されないため安全側だが、延長が必要になったら別 PR で調整。
- **compose-audit 判定ロジックは Python 依存** — runner に python3 + PyYAML 必要。GitHub-hosted ubuntu-latest には同梱されているので問題ないが、self-hosted runner で使う場合は前提を明記。
- **全サービス healthcheck 化はまだ未達** — 本 ADR は `service_healthy` エッジのターゲットに絞っている。dashboard や rag-orchestrator など service_healthy の参照先ではない長寿命サービスは依然 healthcheck 無し。将来 observability / liveness probe 統一の別 ADR で扱う。
