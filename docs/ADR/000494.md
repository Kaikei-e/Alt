# ADR-000494: Ask Augur コンテキストを記事タイトル+IDに改修する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-frontend-sv (TypeScript / SvelteKit)

## Context

Knowledge Home の記事カードにある「Ask」アクションは、Augur（RAG チャット）にコンテキストを渡して質問する機能である。従来の実装では `summaryExcerpt || title` をコンテキストとして渡していた。

`summaryExcerpt` は要約テキストの先頭部分を切り出したもので、途中で途切れた不完全なテキストであることが多い。これには以下の問題がある:

1. **RAG 検索精度の低下** — 途中で切れた文は意味が不完全で、ベクトル検索のクエリとして質が低い
2. **ユーザー体験の劣化** — AskSheet に表示される `line-clamp-3` のテキストが意味不明に見える
3. **トークンの非効率** — 不完全な要約が LLM の context window を消費する

一方、記事タイトルは常に完全で意味が明確であり、RAG が記事全文を取得するための検索クエリとして優れている。

## Decision

**Ask Augur に渡すコンテキストを `summaryExcerpt` から記事タイトルに変更し、`articleId` を追加パラメータとして導入する。**

### 検討した代替案

| 案 | 評価 |
|---|---|
| A. タイトルのみ（採用） | 完全・簡潔・RAG 検索に最適 |
| B. 要約全文を取得して渡す | API 追加呼び出しが必要、トークンも増加 |
| C. タイトル + 要約先頭N文字 | 切れた文の問題が残る |

案 A を採用した。`articleId` も URL パラメータとして渡すことで、将来的に RAG が ID で直接記事を取得する拡張ポイントを確保する。

### 実装の概要

#### augur-entry.ts — コンテキスト組み立てロジック

`buildAugurInitialMessage` に第3引数 `articleId` を追加。`articleId` がある場合は `"Regarding the article: {title} [articleId: {id}]"` 形式でメッセージを組み立てる。`articleId` がない場合（UnifiedIntentBox からの Ask）は従来の `"Context:\n..."` 形式を維持し、後方互換を保つ。

`resolveAugurEntry` の入力インターフェースにも `articleId` を追加し、URL パラメータからの透過的な受け渡しを実現。

#### +page.svelte (Knowledge Home) — ハンドラ修正

`handleAction("ask", item)` で設定するコンテキストを `item.summaryExcerpt || item.title` から `item.title` に変更。`askScopeArticleId` と `askScopeTags` を新規 state として追加し、AskSheet と submitAsk で利用する。

`submitAsk` は `articleId` を URL パラメータとして Augur ページに渡す。

#### AskSheet.svelte — Article Reference Chip UI

切れた要約のプレーンテキスト表示を **Article Reference Chip** パターンに置き換えた。`scopeArticleId` がある場合は `FileText` アイコン付きの chip で記事タイトルとタグを表示する。`scopeArticleId` がない場合（ホーム全体からの Ask）は従来のテキスト表示にフォールバックする。

#### +page.svelte (Augur) — URL パラメータ読み取り

`resolveAugurEntry` の呼び出しに `articleId` パラメータを追加。Augur の Chat コンポーネント自体は変更不要（`initialMessage` に構造化されたコンテキストが含まれる）。

## Consequences

### Pros

- **RAG 検索精度の向上** — 完全な記事タイトルにより、ベクトル検索が関連記事を正確に取得できる
- **UX 改善** — AskSheet に切れた要約ではなく、chip 形式の記事タイトルが表示され、ユーザーが何について質問するか一目で分かる
- **トークン効率** — 短いタイトルにより context window の消費を削減
- **拡張ポイント** — `articleId` パラメータにより、将来 RAG が ID ベースで直接記事を参照する道が開ける
- **後方互換** — `articleId` なしの場合（UnifiedIntentBox からの Ask）は従来動作を維持

### Cons / Tradeoffs

- **要約情報の喪失** — Augur に送られるメッセージに要約の一部が含まれなくなる。ただし RAG が記事全文を取得するため、実質的な情報損失はない
- **バックエンド側は未対応** — `articleId` は現時点ではメッセージテキストに埋め込まれるのみで、rag-orchestrator が `articleId` を解析して直接参照する機能はまだない
