# ADR-000391: 記事再取得・再要約機能を追加する

## Status

Accepted

## Date

2026-03-16

## Affected Services

- alt-backend (Go) — usecase / Connect-RPC handler / REST handler
- alt-frontend-sv (SvelteKit) — Connect-RPC client / デスクトップ・モバイルコンポーネント

## Context

Inoreaderからインポートされた記事は、短いプレビューテキスト（description）のみの場合が多い。alt-backendは記事コンテンツとAI要約を成功時にDBへキャッシュし、以降のリクエストではキャッシュを返す設計になっている。

しかし、以下のケースで「再取得」「再要約」のニーズが存在する:

1. **初回取得の品質が低い** — サイト側のレンダリングタイミングや一時的な障害により、意図しない短いコンテンツがキャッシュされる
2. **要約の質に不満** — AIモデルの出力は非決定的であり、ユーザーがより良い要約を得たい場合がある
3. **記事コンテンツが更新された** — 元記事が修正・追記された場合、古いキャッシュが残り続ける

現状のUIでは、記事取得成功後のArticleボタンは「Article Loaded」（disabled）、要約成功後は再要約の手段がなく、ブラウザリロードでしかキャッシュをバイパスできなかった。

## Decision

**ShapeOfAI Regenerateパターンの"Overwrite"アプローチ**を採用し、`force_refresh`フラグによるキャッシュバイパス機能を全レイヤーに実装する。

### 代替案の検討

| 方式 | 評価 |
|------|------|
| **Overwrite（採用）** | シンプル。DB側は`ON CONFLICT ... DO UPDATE`で既存行を上書き。バージョニング不要 |
| **バージョニング** | 要約履歴の保持は有用だが、UIの複雑化とDBスキーマ変更が必要。現時点ではオーバーエンジニアリング |
| **TTLベースの自動再取得** | ユーザーの意図と無関係に再取得が走る。API呼び出しコスト増大のリスクあり |

### Proto変更

`FetchArticleContentRequest`と`StreamSummarizeRequest`にそれぞれ`optional bool force_refresh`フィールドを追加した。`optional`にすることで既存クライアントとの後方互換性を維持する。

### Backend変更

#### Usecase層

`ArticleUsecase`インターフェースに`FetchCompliantArticleWithRefresh`メソッドを追加した。既存の`FetchCompliantArticle`は`forceRefresh=false`で委譲するため後方互換性を維持する。

`forceRefresh=true`時の挙動:
- `repo.FetchArticleByURL`（DBキャッシュ照会）をスキップ
- singleflightのキーを`url:refresh`に変更し、通常fetchとの重複排除を分離
- `SaveArticle`は`ON CONFLICT (url, user_id) DO UPDATE`のため、既存記事を自然に上書き

#### Handler層

- **Connect-RPC articles handler**: `req.Msg.ForceRefresh`を読み取り、`FetchCompliantArticleWithRefresh`に渡す
- **Connect-RPC feeds handler**: `StreamSummarize`で`forceRefresh=true`時に`FetchArticleSummaryByArticleID`（キャッシュチェック）をスキップ。`SaveArticleSummary`は`ON CONFLICT (article_id, user_id) DO UPDATE`で上書き
- **REST feeds handler**: `FeedSummarizePayload`に`ForceRefresh`フィールドを追加し、同様のキャッシュスキップロジックを実装

### Frontend変更

#### API Client層

`fetchArticleContent`、`StreamSummarizeOptions`、`StreamSummarizeAdapterOptions`、`getFeedContentOnTheFlyClient`の各関数/型に`forceRefresh`パラメータを追加。RPC呼び出しにpass-throughする。

#### UI層

ボタンの状態遷移を以下のように変更:

**Desktop FeedDetailModal:**
- Articleボタン: success → 「Re-fetch Article」（RefreshCwアイコン、有効）。クリックでコンテンツと要約をクリアし、`forceRefresh: true`で再取得
- Summaryボタン: 新しいsuccess状態を追加 → 「Re-summarize」（RefreshCwアイコン）。クリックで`forceRefresh: true`の再要約

**Mobile FeedDetails:**
- Summaryボタン: success状態を追加 → 「Re-summarize」（RefreshCwアイコン）

**Mobile SwipeFeedCard:**
- Articleボタン: コンテンツ展開中 → 「Re-fetch」（RefreshCwアイコン）。コンテンツと要約をクリアして再取得
- Summaryボタン: `aiSummary`存在時 → 「Re-summarize」（RefreshCwアイコン）

## Consequences

### Pros

- **ユーザーが自律的に品質を改善できる** — 低品質なキャッシュに縛られず、ワンクリックで再取得・再要約が可能
- **後方互換性を完全に維持** — `optional`フィールドのため、既存クライアントは影響を受けない
- **DBスキーマ変更なし** — 既存の`ON CONFLICT ... DO UPDATE`句がそのまま活用できる
- **全プラットフォーム統一** — デスクトップ・モバイルSheet・モバイルSwipeで一貫したUXを提供

### Cons / Tradeoffs

- **外部サイトへの追加リクエスト** — force refreshは外部サイトへのHTTPリクエストを発生させる。ただし既存のrate limiter（5秒間隔制約）とrobots.txt準拠チェックは引き続き適用される
- **要約の上書きは不可逆** — 旧要約は保持されない。将来的にバージョニングが必要になった場合はDBスキーマ変更が発生する
- **singleflightキーの分離** — `url:refresh`キーにより、通常fetchと強制refreshは別グループとなる。同時に両方が走った場合、2回のWeb fetchが発生しうる（実用上は問題にならない頻度）
