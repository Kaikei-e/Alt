# ADR-000471: Inoreader記事コンテンツに「長い方が勝つ」ガードを導入する

## Status

Accepted

## Date

2026-03-19

## Affected Services

- pre-processor-sidecar (Go) — `inoreader_articles` テーブルへの取り込み
- alt-backend (Go) — `articles` テーブルへの同期 (`CreateArticleInternal`)

## Context

Inoreader APIが同一記事を再取得した際に、以前より短い（低品質な）コンテンツを返すことがある。これはInoreader側のキャッシュ状態やフィード配信元の一時的な変更に起因すると推測される。

既存のUPSERTクエリは無条件にコンテンツを上書きしていたため、一度取得した高品質なフルコンテンツが、後続の再取得で短い抜粋や空コンテンツに置き換えられるデータ品質劣化が発生していた。

データフローは以下の通りで、2箇所にガードが必要:

```
Inoreader API → pre-processor-sidecar (inoreader_articles) → pre-processor → alt-backend (articles)
```

## Decision

**アプリケーション層（Go）でコンテンツ長を比較し、短い場合はコンテンツ更新をスキップする「長い方が勝つ」ガードを採用する。**

### 設計方針

- **SQLにビジネスロジックを持たせない**: CASE式等の条件分岐をSQLに埋め込まず、Go側で `SELECT → 比較 → INSERT/UPDATE` のパターンで制御する
- **Clean Architecture準拠**: ガードロジックはRepositoryレイヤー（Driver層）に閉じ、上位レイヤーへの影響なし
- **既存SQLの再利用**: 既存のUPSERTクエリはそのまま `upsertFull` にリネームし、新たに `updateMetadataOnly`（メタデータのみ更新）を追加

### 検討した代替案

1. **SQL内CASE式**: `content = CASE WHEN LENGTH(EXCLUDED.content) >= LENGTH(articles.content) THEN ... END` — SQLにビジネスロジックが漏れるため却下（プロジェクト方針 [[000452]] に違反）
2. **アプリケーションロックによる排他制御**: 過剰な複雑性。同一記事の同時更新は実運用上発生しない
3. **変更なし（現状維持）**: データ品質劣化を放置することになるため却下

### 実装概要

#### pre-processor-sidecar (`inoreader_articles`)

`CreateWithResult` メソッドを3段階のフローに変更:

1. `getExistingContentLength`: `SELECT COALESCE(content_length, 0)` で既存コンテンツ長を取得
2. Go側で比較: 既存の方が長い場合 → `updateMetadataOnly`（content/content_length/content_type は更新しない）
3. 新規 or 長い/同じ長さのコンテンツ → `upsertFull`（既存のINSERT...ON CONFLICT DO UPDATEをそのまま使用）

`UpsertResult` に `ContentKept bool` フィールド、`BatchResult` に `ContentKept int` カウンターを追加し、ログとメトリクスで追跡可能にした。

#### alt-backend (`articles` / `CreateArticleInternal`)

トランザクション内で同様のパターンを実装:

1. `getArticleContentLength`: `SELECT COALESCE(LENGTH(content), 0)` で既存コンテンツ長を取得（`deleted_at IS NULL` 条件付き）
2. Go側で比較: 既存の方が長い場合 → メタデータのみ更新クエリ（`content = EXCLUDED.content` を省略）
3. 新規 or 長い/同じ長さ → 既存の全フィールドUPSERT

`articles` テーブルには `content_length` カラムがないため、`LENGTH(content)` で動的に算出。

#### 変更しないパス

- `SaveArticle` / `upsert_article_driver.go`: Web抽出パスはユーザー操作またはシステム抽出による意図的なコンテンツ置換であり、ガード不要

### エッジケース

| シナリオ | 動作 |
|----------|------|
| 初回INSERT（記事未存在） | SELECT → ErrNoRows → 全フィールドINSERT |
| 新コンテンツが長い | SELECT → 比較 → 全フィールドUPSERT |
| 新コンテンツが同じ長さ | SELECT → 比較(<=) → 全フィールドUPSERT（修正・改善の可能性を許容） |
| 新コンテンツが短い | SELECT → 比較(>) → メタデータのみUPDATE |
| 既存コンテンツNULL | COALESCE → 0 → 新コンテンツ採用 |

## Consequences

### Pros

- **データ品質の保護**: 一度取得したフルコンテンツがInoreader APIの一時的な品質低下で失われない
- **透明性**: `ContentKept` フラグとログにより、ガードの発動頻度を監視可能
- **最小侵襲**: 既存のSQLクエリ・インターフェース・上位レイヤーへの変更なし
- **SQLにロジックなし**: ビジネスロジックがGoコードに閉じており、テスト・デバッグが容易

### Cons / Tradeoffs

- **SELECTクエリの追加**: UPSERTの前に1回の追加SELECTが発生する。ただし `inoreader_id` / `(url, user_id)` はいずれもインデックス付きで、バッチ全体でもms単位のオーバーヘッド
- **文字数ベースの判定**: コンテンツの「品質」ではなく「長さ」で判定するため、短くても高品質なコンテンツが反映されないケースがあり得る。実運用上、Inoreaderからの再取得で短くなるケースは品質低下と強く相関するため許容
- **同じ長さは上書き**: 同じ長さの場合は新しいコンテンツで上書きする設計。微修正を取り込む利点がある一方、同品質の不要な書き込みが発生する可能性がある
