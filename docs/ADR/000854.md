# ADR-000854: Knowledge Loop Wave 4-C で EventLogSurfaceScoreResolver を本番経路に組込み、Wave 4-A の rag-orchestrator emitter は Pact / tenant 伝播 / stub 再生成の前提が揃うまで port + no-op で停止する

## Status

Accepted

## Date

2026-04-26

## Affected Services

- knowledge-sovereign (Go):
  - 新ファイル `app/usecase/knowledge_loop_projector/score_resolver_event_log.go` で `EventLogLookup` port + `EventLogSurfaceScoreResolver` を実装。F-001 (cross-user 漏洩) 検出を resolver 入口で物理 bind
  - 新ファイル `app/usecase/knowledge_loop_projector/score_resolver_event_log_test.go` で 8 unit test (window bind / drift / augur / topic / open interaction / F-001 / no-user-id / allowlist contract)
  - `app/driver/sovereign_db/read_events.go` に `ListKnowledgeEventsForUserInWindow` を追加。`user_id = $1 AND occurred_at IN [since, until) AND event_type = ANY($4::text[])` で physically bind
  - `app/main.go` の projector 配線を `WithScoreResolver(NewEventLogSurfaceScoreResolver(repo))` に変更
  - `app/usecase/knowledge_loop_projector/constants.go` の `WhyMappingVersion` を 5 → 6 に bump
- rag-orchestrator (Go):
  - 新ファイル `internal/usecase/knowledge_event_emit_port.go` で `KnowledgeEventEmitter` interface + `NoopKnowledgeEventEmitter` default + `AugurConversationLinkedInput` shape
- docs:
  - `docs/runbooks/knowledge-loop-reproject.md` の WhyMappingVersion v6 row を `tbd` から `2026-04-26 / [[000854]]` に確定

## Context

[[000853]] で Wave 4 を 4 つの sub-wave (4-A: rag-orchestrator augur emit / 4-B: recap-worker Rust emit / 4-C: real resolver + WhyMapping bump / 4-D: RLS rollout) に分割し、それぞれ独立 PR + Pact CDC + alt-deploy 同期で landed させる方針を確定した。本 ADR はそのうち **4-C を完了** し、**4-A は port + no-op default まで** で停止する判断を記録する。

### Wave 4-C を本番 wiring まで進められた理由

`EventLogSurfaceScoreResolver` は知識-sovereign の内部実装で、cross-service contract を一切持ち込まない。projector は既に [[000852]] で `Projector.WithScoreResolver` の seam を備えており、その差し替え 1 行で v2 evidence 経路が active になる。Repository 側の `ListKnowledgeEventsForUserInWindow` も既存の `scanEvents` パターンに乗る additive 追加。テスト側は fake `EventLogLookup` を注入する pattern で 8 ケース固定でき、F-001 の cross-user mismatch も決定論的に検証できる。

### Wave 4-A を port + no-op で止めた理由

実装に着手して下記の 3 つの **新規 cross-service 前提** が同時に必要だと判明した:

1. **rag-orchestrator の sovereign 生成スタブが手動 curate された subset である**: 現状 `internal/gen/proto/services/sovereign/v1/sovereign.pb.go` には `ApplyProjectionMutation` / `ApplyRecallMutation` / `ApplyCurationMutation` の 3 RPC しか入っていない。`AppendKnowledgeEvent` / `AppendRecallSignal` / `AppendKnowledgeUserEvent` 系は alt-backend の stub にはあるが、rag-orchestrator の stub には含まれていない。これを補うには `proto/buf.gen.alt-backend-services.yaml` 相当の rag-orchestrator 向け独立 buf gen を追加するか、partial subset の生成方針自体を決め直す必要がある
2. **rag-orchestrator に tenant 伝播経路がない**: augur handler は `X-Alt-User-Id` header だけを extract している。`AugurConversationLinked` event payload は `tenant_id` を要求するので、まず alt-backend 側で `X-Alt-Tenant-Id` を forward する hop を増やす契約変更が要る
3. **Pact CDC contract が未整備**: rag-orchestrator → knowledge-sovereign の consumer-provider 関係は今回が初めて。`pact-go v2` の Connect-RPC consumer interaction を新規に書き、knowledge-sovereign 側 provider verification list に登録し、`./scripts/pact-check.sh --broker` を更新する必要がある。さらに alt-deploy の `services.yaml` / `release-deploy.yaml` / `run-e2e-suite.yml` の pacticipant role + e2e allowlist を手動同期 ([[alt_deploy_pacticipant_sync]]) する必要があり、抜けると pact-publish/gate が silent skip する

これら 3 点はいずれも単独で「user の deploy 判断を伴う」変更であり、autonomous session 内で 3 つ重ねて施工すると過去 X-Service-Token 401 cascade ([[000722]]) と同型の事故面が広がる。Wave 4-A 自体は「rag-orchestrator が augur 会話作成成功後に knowledge_events を emit する」という 1 つの目的だが、上記 3 つの前提が揃うまでは port + NoopKnowledgeEventEmitter で seam だけ landed させ、live behavior は変えない選択をする。

## Decision

### 1. Wave 4-C 完了 — `EventLogSurfaceScoreResolver` を本番 projector 経路に plug

`EventLogSurfaceScoreResolver` は次の不変条件を物理的に守る:

- 入力は `knowledge_events` の payload-resident 値のみ。`tag_set_versions` 等の latest state を覗かない
- 時刻窓は `[event.occurred_at - 7d, event.occurred_at)` で固定。`time.Now()` / `NOW()` を business fact に登場させない
- SQL 側 `WHERE user_id = $1` で物理 bind し、resolver 側でも返ってきた行の `user_id` を必ず突合。mismatch は `crossUserIsolationViolationTotal` を bump して空 inputs を返却 (F-001 fail-closed)
- event_type allowlist は `resolverEventTypes` 定数で凍結。canonical contract §6.4.1 / §11 のいずれかに無い event を勝手に拾わない

aggregation ルール:

- `SummaryVersionCreated` / `SummarySuperseded` で同一 `article_id` ヒット → `VersionDriftCount++`
- `SummaryVersionCreated` で tag set overlap → `TagOverlapCount++`
- `HomeItemOpened` で同一 `entry_key` ヒット → `HasOpenInteraction = true`
- `AugurConversationLinked` で同一 `entry_key` ヒット → `HasAugurLink = true`
- `RecapTopicSnapshotted` の `top_terms` と article tags の overlap → `TopicOverlapCount++`

`Projector.WithScoreResolver(NewEventLogSurfaceScoreResolver(repo))` を `main.go` で配線。Wave 1+2+3 で landed させた `decideBucketV2` が初めて v2 evidence を受け取り、`surface_bucket_assigned_total{version="v2",bucket=...}` counter が立ち上がる経路ができる。

WhyMappingVersion を v5 → v6 に bump し、reproject runbook の v6 row を `tbd` から本 ADR 番号で確定。enricher 側で新 WhyKind (`topic_affinity_why` / `tag_trending_why` / `unfinished_continue_why`) を実際に narrative に含める実装は次回 (resolver の signal がどの程度 prod で立ち上がるか観測した後) に切り出す。

### 2. Wave 4-A 部分着地 — port + Noop default のみ

`KnowledgeEventEmitter` interface を rag-orchestrator usecase 層に新設し、`NoopKnowledgeEventEmitter` を default impl として提供。`AugurConversationLinkedInput` shape は canonical contract §6.4.1 と整合する形で固定 (UserID / TenantID / EntryKey / LensModeID / ConversationID / LinkedAt)。

実 client (sovereign 経路 + Pact + tenant header forwarding) は本 ADR では着手しない。下記 3 つの前提が個別 PR で landed した後、Wave 4-A 残り (1 セッション) で追加する:

- **前提 a**: `proto/buf.gen.alt-backend-services.yaml` を真似た rag-orchestrator 向け buf gen を整え、sovereign の全 RPC を生成
- **前提 b**: alt-backend → rag-orchestrator hop で `X-Alt-Tenant-Id` を forward、rag-orch handler に `extractTenantID` を追加
- **前提 c**: rag-orchestrator → knowledge-sovereign の Pact consumer test (Connect-RPC over JSON、ADR-000764 idiom) を `internal/adapter/contract/sovereign_consumer_test.go` 相当に landed させ、`./scripts/pact-check.sh --broker` で publish

3 前提が揃った後の Wave 4-A 実装本体は次の files で済む:

- `internal/adapter/sovereign_client/append_event_client.go` (Connect-RPC client 実装)
- 既存の augur handler `CreateSessionFromLoopEntry` に `eventEmitter.EmitAugurConversationLinked(ctx, ...)` を warn-and-continue で追加
- main.go の DI を `usecase.NoopKnowledgeEventEmitter{}` から `sovereign_client.NewAppendEventClient(...)` に切替え

### 3. Wave 4-B / 4-D の扱い

- **4-B (recap-worker Rust)**: 同種の 3 前提 (Rust 側 sovereign stub 整備 + tenant 伝播 + Pact `pact_consumer` crate 実装) が必要。本 ADR では port パターンも先送り。recap-worker 自身の事情 ([[000749]] 系の Rust pact 経路) を踏まえて専用セッションで設計する
- **4-D (RLS rollout)**: caller-first / migration-second の rollout 手順は [[000853]] に明文化済み。caller 側の `SET LOCAL alt.user_id = $1` 配線は read 経路の全 caller を一斉に直す必要があり、本 ADR では着手しない。専用セッションで read 経路の inventory + 配線 + 観測 + migration の 4 段階を踏む

### 検討した代替案と却下理由

- **rag-orchestrator のスタブを今回の場で full sovereign proto から regenerate**: buf gen 設定の変更で `internal/gen/proto/services/sovereign/v1/` 配下が大量に変わる。rag-orchestrator が現在使っている 3 RPC との後方互換は保たれるが、Pact contract の整備が同 session で出来ない以上 dead code が増える形になり、`feedback_no_unsolicited_benchmark_builds` 系の dead-code 防止方針と整合しない
- **tenant_id を rag-orchestrator のリクエスト body / handler args に埋める**: tenant は JWT claim 由来であるべきで、handler args に外から渡すと信頼境界が崩れる ([[knowledge-loop-canonical-contract]] §11)。alt-backend → rag-orchestrator の header forwarding が正解
- **Wave 4-A を Pact 無しで直接 emit**: 過去 X-Service-Token incident と同型。許容しない
- **WhyMappingVersion bump を Wave 4-A の後に延期**: bump の意味が立つのは v2 evidence 経路 (=resolver) が動き出すタイミング。resolver は本 ADR で animation したので、bump も同時で正しい。Wave 4-A の有無とは独立

## Consequences

### Pros

- v2 evidence ベースの bucket placement が **本番経路で active** になった。Wave 1+2+3 で landed させた `decideBucketV2` / metric / SurfaceScoreResolver seam がすべて意味を持って動き始める
- F-001 (cross-user evidence_refs leak) の検出が resolver 入口で物理境界として機能。violation は metric counter に立つので alert 連携できる
- WhyMappingVersion v6 と reproject runbook の同期が整った。bump が「resolver 実装が動き出した」シグナルを兼ねるので運用上のドキュメンテーション ROI が高い
- Wave 4-A の port + Noop default を landed させたことで、rag-orchestrator 側の建付けが **本番 emitter 接続を待つだけ** の状態になった。前提 3 件が個別に landed すれば Wave 4-A の最終 PR は driver + handler 1 行 + DI 切替えで済む
- `EventLogLookup` を狭い port にしたことで、resolver の test surface が小さい (8 unit test で reproject-safety と F-001 を pin できる)。回帰検出が cheap

### Cons / Tradeoffs

- v2 evidence 経路は active になったが、**実際に v2 placement が起きるかは upstream emitter の状態次第**。`AugurConversationLinked` / `RecapTopicSnapshotted` がまだ flow していないので、本番 metric 上では `surface_bucket_assigned_total{version="v1"}` が支配的なまま (期待される空グラフ期間)
- Wave 4-A が完成形ではないので、rag-orchestrator の port + Noop は半年以上塩漬けになりうる。前提 3 件 (stub / tenant / Pact) の優先度を別所で track する必要あり
- WhyMappingVersion v6 bump 直後に full reproject を実施しないと、過去 entry の Why は v5 narrative のまま残る。reproject runbook 通りに `WhyMappingVersion bump → reproject` を実施するタイミングを別途確保する必要あり
- enricher 側で新 WhyKind の narrative を実際に emit する実装は本 ADR では未着手。`why_kind = source_why` のままで `surface_planner_version = 2` だけが立つ entry が出現しうる。意味論ズレを避けるには enricher 拡張を次セッションで早めに着手すべき
- `EventLogLookup` は限定 7d window + ≤256 events を per-event で query する。projector batch が大きいと N×M クエリが走る。実 prod 負荷で過剰なら既存 batch read に統合する optimization が必要 (現状は単純シェイプを優先した)
