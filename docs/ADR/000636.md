# ADR-000636: 分類ワーカープールを上限制御し failed idempotent run の再利用を禁止する

## Status

Accepted

## Date

2026-04-07

## Affected Services

- recap-subworker (Python / FastAPI) — classification process pool のサイズ制御、idempotent retry の再実行条件修正
- recap-worker (Rust) — classification queue worker の実効並列度制御修正

## Context

3-day Recap の自動バッチ運用中、classification フェーズで `classification returned 0 results for 1708 articles` が継続し、下流の要約改善だけでは回復しない状態になった。調査すると、失敗は単一のモデル精度問題ではなく、`recap-worker` と `recap-subworker` の運用制御が複合的に崩れていた。

確認できた事象は以下の通り:

1. **分類ワーカープールが実行枠より大きかった**  
   `recap-subworker` は `RECAP_SUBWORKER_MAX_BACKGROUND_RUNS=2` で同時実行を制限していた一方、classification 用 process pool は `RECAP_SUBWORKER_CLASSIFICATION_WORKER_PROCESSES=6` だった。各 worker process が分類モデルを preload するため、必要以上の重複ロードで RSS が膨らみ、`process pool OOM` を誘発しやすかった。
2. **`recap-worker` の queue semaphore が実質無効だった**  
   classification queue worker は semaphore を取得してから background task を spawn していたが、permit を task の寿命に持ち込んでおらず、spawn 直後に解放されていた。その結果、設定した並列数以上の chunk が一気に投げ込まれ、subworker 側のメモリ圧力を高めていた。
3. **retry が再実行ではなく failed run の再取得になっていた**  
   `Idempotency-Key` 付きの `/v1/classify-runs` は、同じ key の既存 run を再利用する設計だった。しかし `failed` 状態の run まで再利用していたため、一度 `stuck: process pool OOM` になった chunk は queue retry が走っても新しい run が作られず、古い失敗結果を返し続けていた。

この状態では、worker 数を減らすだけでも、queue 側の oversubmission だけでも、あるいは過去失敗を手動で掃除するだけでも不十分だった。**メモリ圧力の抑制、実効並列度の回復、retry の再実行化** を同時に行う必要があった。

## Decision

**classification 実行の上限を `background slots` に揃え、`recap-worker` の semaphore permit を task 寿命まで保持し、`failed` な idempotent classification run は再利用せず新規 run を作成する。**

### 1. classification worker pool を `max_background_runs` で上限制御する

`recap-subworker` の設定で、`classification_worker_processes` が `max_background_runs` を超えないようにする。デフォルト値も `6` から `2` に下げ、Compose の実運用値も `2` に統一する。

この決定により、「同時に 2 件しか処理しないのに 6 プロセス分モデルを常駐させる」状態をやめ、常駐メモリと fork/spawn コストを抑える。

### 2. queue worker の permit を spawned task に持ち込む

`recap-worker` の classification queue worker は `Semaphore::acquire_owned()` を使い、取得した permit を spawned task の中で保持する。これにより設定並列度が実効値として働き、queue からの oversubmission を防ぐ。

この変更は throughput を無制限に増やすものではなく、**下流 service が処理できる量だけ供給する** ための backpressure 回復を目的とする。

### 3. failed idempotent run は再利用しない

`recap-subworker` の idempotency 判定は、`running` や `succeeded` の run は再利用する一方、`failed` の run は再利用対象から外す。`failed` だった場合は同じ `Idempotency-Key` でも新規 run を作り、queue retry が実際の再実行になるようにする。

この方針により、retry の意味を「同じ失敗を読む」ことではなく、**同じ入力を安全に再評価する** ことへ戻す。

### 4. コンテナ再ビルドと運用確認を必須とする

実装後に `recap-worker` / `recap-subworker` を再ビルド・再起動し、再開された classification backlog で挙動を確認する。検証では以下を確認した:

- `classification.run.retry_after_failed_idempotent` の後に新しい run が作られること
- `classification.run.completed` が継続して記録され、旧来の `process pool OOM` ループに戻らないこと
- idle timeout 後に `classification process pool shutdown complete` が記録され、常駐プロセス数と RSS が低下すること

## Consequences

### Pros

- classification の retry が実際の再実行になり、過去の failed run を返し続けるループを断てる
- `recap-worker` からの classification chunk 投入量が制御され、subworker のメモリ圧力を増幅しにくくなる
- `recap-subworker` の idle 時メモリは、修正前に観測した約 2.1 GiB 級の常駐状態から、再起動後は idle timeout 後に約 1.2 GiB まで低下した
- backlog 再開後の classification run は `9888` から `9896` の新規 run として進行し、`classification.run.completed` と `classification chunk completed` を確認できた
- process pool の lazy init / idle shutdown が実運用で機能していることをログで確認できた

### Cons / Tradeoffs

- classification の理論上の最大並列数は `background slots` に縛られるため、以前の `6` プロセス設定より peak throughput が低く見える可能性がある
- failed retry 時に新規 run 行を作るため、`runs` テーブルには同一 chunk の履歴が複数残る。重複ではなく再試行履歴として扱う運用前提が必要
- backlog 回復は確認できたが、`recap-subworker` 起動時に同種の初期化ログが複数出る経路は残っており、別件として観測継続が必要
- 将来モデル footprint がさらに増える場合、`classification_worker_processes=2` でも余裕が足りず、`max_background_runs=1` まで下げる追加判断が必要になる可能性がある
