# ADR-000842: Knowledge Loop の depth plane 階層を CSS transform で表現しサーバ側 rate limit と stream JWT exp 強制で security を底上げする

## Status

Accepted

## Date

2026-04-24

## Affected Services

- **alt-frontend-sv (SvelteKit)** — 新規 `src/lib/styles/loop-depth.css` が 3 plane の translateZ + perspective + saturate gradient を定義し、`prefers-reduced-motion` 下で transform を止めて opacity/saturation だけに落とす。`LoopSurfacePlane.svelte` は `loop-plane` class を兼ねるよう最小変更、`/loop/+page.svelte` 側の root に `loop-plane-root` を付けて共通 perspective を宣言する。
- **alt-backend (Go)** — `usecase/knowledge_loop_usecase/rate_limiter.go` (新規) が in-memory で per-entry 60s throttle と user global 600/min ceiling を持つ `LoopRateLimiter` を提供する。`TransitionKnowledgeLoopUsecase.Execute` が classify 後・reserve 前に rate limit check を実行し、拒否時は `ErrRateLimited` を返す。handler は `CodeResourceExhausted` に map (BFF 側 HTTP 429 と直結、ADR-000839)。`StreamKnowledgeLoopUpdates` は 1 秒 ticker で `user.ExpiresAt` を wall-clock と比較し、期限超過で `StreamExpired("jwt_expired")` envelope を送出して close する。

## Context

[[000840]] / [[000841]] で Knowledge Loop は event append、stream、Why enrichment、Surface plane まで揃い、UI としての表現力が揃った。しかし `review.md` (2026-04-24) の Priority 5 と canonical contract §8.4 / §9 の運用契約には未実装の部分が 2 つ残っていた。

1. **Depth UI が filter 止まりで plane 階層化されていない (P5)** — `renderDepthHint` は projection に出ており、`LoopEntryTile` 側でも saturate/brightness/border に反映されていたが、canonical contract §12 が要求する「foreground / mid-context / deep-focus の 3 plane が Z 軸で層化される」構造までは到達していなかった。`LoopSurfacePlane` は `filter: saturate(...)` しか持っておらず、Continue / Changed / Review (PR-L8) が導入されても視覚的に「奥に引っ込んだ」感は得られていなかった。Apple HIG (WWDC23 10072) の spatial design 原則と整合するためには tile ではなく plane 単位の Z 深度が必要だった。

2. **canonical contract §8.4 の rate limit と §9 の JWT exp が未実装** — [[000840]] §5 の `KnowledgeLoopObserved` rate limit (60s per entry、600/min/user) は contract に書かれていただけで実装はなく、クライアント側 `loop-observe-throttle.ts` に頼り切っていた。同様に stream は 30 分 stale timer しか持っておらず、canonical contract §9 が要求する「JWT exp で stream を即 close する」契約を満たしていなかった。いずれも単独では deploy を止める欠陥ではないが、OWASP Top 10:2025 の A04 (Insecure Design) / A07 (Authentication Failures) の観点で CI gate を通過する前に塞ぐべき穴だった。

両者は見た目上独立しているが、「Knowledge Loop を UI としても運用としても完成状態に持っていく」最後の 1 ADR としてまとめて記録する。

## Decision

Depth plane の CSS 階層化と server-side rate limit + stream JWT exp enforcement を同一 PR 群として採用し、以下 7 点を固定する。

1. **3 plane の CSS depth hierarchy** — 新規 `src/lib/styles/loop-depth.css` に以下を定義する:
   - `.loop-plane-root` → `perspective: 1200px` + `perspective-origin: 50% 45%` + `transform-style: preserve-3d` で子要素に共通の vanishing point を持たせる
   - `.loop-plane[data-plane="foreground"]` → `translateZ(0)` + `saturate(1.04)` + `brightness(1.0)` + opacity 1
   - `.loop-plane[data-plane="mid-context"]` → `translateZ(-24px) scale(0.985)` + `saturate(0.94)` + opacity 0.96
   - `.loop-plane[data-plane="deep-focus"]` → `translateZ(-48px) scale(0.97)` + `saturate(0.88)` + opacity 0.9
   - 遷移は `transform 0.35s cubic-bezier(0.2, 0, 0.1, 1) + filter 0.25s + opacity 0.25s`

2. **Reduced Motion fallback (canonical contract §12.5)** — `@media (prefers-reduced-motion: reduce)` 下で `loop-plane-root` の `perspective: none` + `transform-style: flat`、mid / deep の `transform: none`、transition は `opacity 0.2s ease` のみに縮退する。これで depth simulation / parallax / blur は停止し、「dissolve + color shift」の代替表現が残る。

3. **component 側の最小改修** — `LoopSurfacePlane.svelte` の `<section class="plane">` に `loop-plane` class を加えて共通 CSS と `.plane` ローカル style が併存できるようにし、`/loop/+page.svelte` の `<main>` に `loop-plane-root` と `import "$lib/styles/loop-depth.css"` を入れる。既存コンポーネントのローカル style は触らず、plane 間の Z 関係だけを外出しにすることで page/component の責務を切り分ける。

4. **`LoopRateLimiter` の in-memory 実装** — `usecase/knowledge_loop_usecase/rate_limiter.go` に以下の API を置く:
   - `AllowObserve(userID, lensModeID, entryKey, at)` → `(user, lens, entry)` per 60s + 600/min global ceiling
   - `AllowGlobal(userID, at)` → 600/min global ceiling のみ
   - 拒否理由を `observe_throttle` / `user_global_ceiling` の文字列で返し、retry-after 判定に使える
   - 観測 permit は **global counter も必ず加算する**ことでエントリキーを rotate しても global ceiling を回避できない

5. **`TransitionKnowledgeLoopUsecase.Execute` に rate limit gate を追加** — classify 後・dedupe reserve 前に以下を実行する:
   - `eventType == EventKnowledgeLoopObserved` なら `AllowObserve`、それ以外なら `AllowGlobal`
   - 拒否時は `ErrRateLimited` で即 return、dedupe も append も呼ばない
   - handler は `CodeResourceExhausted` に map (ADR-000839 の classification table に追加)
   - BFF の `$lib/connect/error.ts` は既存の `resource_exhausted → 429` mapping を継承する

6. **`ErrRateLimited` sentinel** — `errors.go` に新規 sentinel を追加し、wrapping 経路で `errors.Is` 判定できるようにする。既存の `ErrInvalidArgument` / `ErrUpstreamUnavailable` / context error の classification はそのまま維持する。

7. **Stream JWT exp enforcement** — `StreamKnowledgeLoopUpdates` handler に 1 秒間隔の `jwtCheckTicker` を追加し、`user.ExpiresAt` を wall-clock 越えで検知して `StreamExpired("jwt_expired")` を送出して close する。既存の 30 分 stale timer / context cancel / 3 連続 fetch error はそのまま残す。これで contract §9 "Stream auth expiry" が実装側に落ちる。

### 検討した代替案

- **depth を component local style に閉じ込める** — 各 plane component (`LoopSurfacePlane`, `LoopEntryTile`) 内で `@scope` や scoped `:global()` を使って Z を表現する案。却下。plane 間の Z 関係は page 全体の vanishing point に依存するため、`perspective` は root に 1 つ必要。コンポーネント境界をまたぐ transform は外出しの共通 CSS で宣言する方が予測可能。
- **depth transitions を JavaScript (FLIP / AnimateJS) で制御する** — 指定可能な timing や物理モデルは強力だが、Svelte runes の reactivity と transform の相互作用でジッターが出やすく、Reduced Motion の opt-out も自作する必要がある。CSS だけで足りるなら CSS に倒すのが memory `feedback_alt_paper_design` とも整合。
- **rate limit を Redis や共有キャッシュにバックする** — multi-pod で ceiling が厳密に 600/min になる反面、新しい infra 依存が生まれる。現行 Alt は単一プロセス想定 (memory `feedback_single_machine_architecture`) で、multi-pod でも pod 数に線形ドリフトするだけ。dedupe key / event-store unique index が主防御で rate limit は defense-in-depth 層なので、in-memory で十分と判断。将来 multi-pod に移行する場合は別 ADR で Redis 化を検討。
- **rate limit を Connect interceptor として横断適用する** — Loop 以外の RPC にも波及する大きな変更になる。今回の契約は Loop 固有なので、usecase レベルの gate に留める。Connect interceptor は OWASP A04 を横断的に満たす必要が出てきたときに別 ADR で。
- **Stream JWT exp を `context.Context` deadline で制御する** — Connect-go の context は request-scope で、stream 開始後に期限が過ぎても cancel されるとは限らない。明示的な ticker で wall-clock 監視するのが確実で、`StreamExpired("jwt_expired")` を emit すればクライアントも reason を識別できる。
- **JWT exp check を stream open 時のみ 1 回** — 長時間 stream (30 分 stale 前) で期限が過ぎても検知できない。定期 ticker (1 秒) にすることで最大 1 秒以内に検知する。1 秒 ticker は `runtime.Gosched` 程度の負荷なのでコストは無視できる。

## Consequences

### Pros

- **spatial design が完成する** — `/loop` 画面で Continue / Changed / Review が foreground から奥に退き、深度と saturation の両方で plane 間の階層が可視化される。Apple HIG (WWDC23 10072) と整合し、視覚障害者には `aria-label` (PR-L8) で priority が届く二重経路になる。
- **Reduced Motion 契約が実装される** — `prefers-reduced-motion: reduce` で depth simulation が即座に opacity + saturation に縮退する。canonical contract §12.5 / accessibility 契約 §13 がコードに落ちた状態。
- **canonical contract §8.4 が完全に実装される** — 60s dwell throttle と 600/min user ceiling が server-side で enforce されるため、クライアント側 throttle の単一障害点 (tampered client / rotated entry_key による flood) が塞がる。`CodeResourceExhausted → HTTP 429` の mapping で client 側の retry-after バックオフも整う。
- **canonical contract §9 Stream auth expiry が実装される** — JWT が期限切れしても stream が 30 分残る問題が解消。`StreamExpired("jwt_expired")` で reason が明示され、client hook (PR-L6) の `onExpired` コールバックで refetch + reconnect が自動発動する。
- **OWASP Top 10:2025 への準拠が進む** — A04 (Insecure Design、rate limit)、A07 (Authentication Failures、stream auth expiry) がコード側に落ちる。gosec / golangci-lint / check は全 green。
- **rolling deploy 互換**: DB migration / proto 変更 / 新 compose service いずれも無し。既存 deploy 経路 (`git push origin main` → `dispatch-deploy.yaml` → `Kaikei-e/alt-deploy`) にそのまま乗る。

### Cons / Tradeoffs

- **multi-pod 下では global ceiling が N 倍にドリフトする** — in-memory bucket なので pod 数 × 600/min が実質の上限になる。現行 single-machine 前提 (memory `feedback_single_machine_architecture`) では問題ないが、将来 multi-pod 化すると別 ADR で Redis 化を検討する必要がある。dedupe key と event store unique index が第一防御なので、スケール後のドリフトは許容可能な defense-in-depth 劣化。
- **`jwtCheckTicker` が 1 秒ごとに回る** — 各 stream ごとに goroutine + ticker が増える。単一ユーザで同時接続 5〜10 程度なら無視できる負荷だが、将来的に大量同時接続を許可する場合は ticker 間隔を延ばすか、heartbeat ticker と統合することを検討する。
- **CSS `translateZ` は古いブラウザでジャダーが出る可能性** — Chromium / WebKit / Firefox の現行版では問題ないが、Electron beta や一部 mobile browser で GPU compositor が `preserve-3d` を期待通り扱わないケースがある。issue が顕在化したら `will-change: transform` の適用範囲を絞るか、深度を `box-shadow` ベースに倒す方針で対応する。
- **Rate limit bucket のメモリ使用量** — ユーザ数 × エントリ数 × 60s の map。エントリ rotation で不要エントリが累積しない GC を明示していない。長時間運用では古いエントリの掃除を `time.AfterFunc` ベースで入れる必要が出る可能性があるが、現行スケール (1 インスタンスで数百ユーザ) では問題化しない。
- **1 秒 ticker の detection 精度** — JWT exp から最大 1 秒遅れる。token の残り寿命が 1 秒未満で認証されたリクエストに対して stream が最大 1 秒生きる可能性がある。canonical contract §9 は厳密な instant-close を求めていないため許容、ただし SOC2 監査等で厳密化が必要なら別途対応。
- **Depth CSS が生まれる一方で 3D の誤解を招きやすい** — translateZ は 3D 感を生むが、canonical contract §11 の「Depth lives on tiles, not on glyphs」を読み違えると text そのものを 3D 化する改修が入りかねない。コメントとテスト (`loop-depth.source.spec.ts`) で plane-only depth を pin しているが、将来のレビュアーへの教育コストは残る。
