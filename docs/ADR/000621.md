# ADR-000621: クエリ展開ハルシネーション対策と多層防御による Augur RAG 回答品質改善

## Status

Accepted

## Date

2026-04-04

## Affected Services

- rag-orchestrator (Go)
- news-creator (Python)

## Context

2026-04-03 18:19:07 のクエリ「イランの石油危機はなぜ起きた？」で Augur が 129 文字の低品質回答を返す障害が発生した。ログ調査により、**5 つの連鎖障害**を特定した。

### 1. クエリ展開モデルのプロンプト命令エコー（根本原因）

news-creator のクエリ展開プロンプトに含まれるメタ命令（`"Output Japanese queries first, then English queries."`）を gemma4-e4b-12k が出力テキストと混同し、全 8 展開クエリが同一の命令文のオウム返しになった。同様の現象は PostgreSQL マイグレーションクエリでも確認されており、`"Japanese queries first, then English queries."` が 11 回繰り返されていた。

これは小型 LLM （4B パラメータ）でメタ命令がモデル出力に混入する既知の障害パターンである。

### 2. filterExpandedQueries の不十分なフィルタリング

`filterExpandedQueries()` はローマ字日本語（マクロン付きローマ字）のみをフィルタしており、命令エコー・重複クエリ・XML タグリーク・極端な長さのクエリを通過させていた。

### 3. BM25 が 0 件

Meilisearch 経由のキーワード検索が日本語クエリで 0 件を返し、lexical 検索によるセーフティネットが機能しなかった。

### 4. 品質ゲートの誤判定

`RetrievalQualityAssessor.AssessWithIntent()` はコンテキスト間のタイトル語共有（相互コヒーレンス）のみを評価しており、**クエリに対するコンテキストの関連性**を見ていなかった。結果として "Asset Tokenization"、"LibreFang" といった完全に無関係な記事が `"good"` と判定された。

### 5. ショートアンサー・フォールバック未発動

`AssessAnswerQuality()` のキーワードカバレッジチェックは CJK クエリで常にスキップされるため、129 文字のショートアンサーに対してハードストップが発動しなかった。

### 前回の石油危機事案との関係

2026-03-28 にも類似の石油危機クエリで障害が発生し、因果分類（`causal_explanation`）の導入と因果回答プロンプト契約が ADR-000607 で実装されている。今回はそれらの修正が正しく機能した（intent は `causal_explanation` に分類された）が、**新しい障害モードとしてクエリ展開ハルシネーション**が根本原因となった。

### ベストプラクティスとの照合

- **Anthropic Prompting Best Practices**: XML 等でセクションを分離し曖昧性を下げる構造化出力契約が、メタ命令エコー防止に有効
- **RE-RAG (EMNLP 2024)**: retrieved context が質問に有用かを独立推定する relevance estimator の有効性が報告されている。本 ADR ではヒューリスティックによる段階的防御として query-context mismatch チェックを導入

## Decision

**クエリ展開から回答生成までの 4 レイヤーに多層防御を導入し、単一障害点が回答品質に直結しないようにする。**

### Phase 1: プロンプト構造化出力契約 + 出力バリデーション（news-creator）

`expand_query_usecase.py` のプロンプトテンプレートを XML 構造化契約に変更し、エコーされやすいメタ命令（`"Output Japanese queries first"`、`"Do not add numbering, bullets, labels"`）を除去した。

初期実装で few-shot 例を含めたが、ベンチマークで gemma4-e4b-12k が例の回答内容（ドル円関連クエリ）をコピーする問題が発覚したため、例を除去し構造のみで出力形式を誘導する設計に変更した。

出力バリデーションとして以下を追加:

- **命令エコー検出**: 既知の命令文パターンとの normalized match
- **preamble 除外**: `"Here are the queries:"` 等のプロローグ行を除外
- **XML タグリーク除外**: `</example>` や `<input>...</input>` の除外
- **順序保持 dedupe**: `set()` ではなく `dict` による初出順序維持の重複除去

### Phase 2: filterExpandedQueries 強化（rag-orchestrator）

`filterExpandedQueries()` に以下のフィルタを追加:

- **順序保持 dedupe**: `map[string]struct{}` で case-insensitive 重複除去
- **命令リーク検出 `isInstructionLeak()`**: 既知命令パターンの exact match + メタ語密度（3 語以上）で判定
- **XML タグリーク検出 `isXMLTagLeak()`**: `<` で始まり `>` を含む行を除外
- **長さフィルタ**: 3 rune 未満 / 200 rune 超を除外
- **フィルタ後 0 件時のログ**: `expansion_all_filtered` WARN で理由を記録し、Phase 4 の fallback 判定に繋げる

### Phase 3: 品質ゲートにクエリ-コンテキスト関連性チェック追加（rag-orchestrator）

`AssessWithIntent()` のシグネチャに `query string` を追加し、`hasQueryContextMismatch()` を実装した。

- **Title + ChunkText** の両方を検査対象とし、title-only の誤検知を防止
- CJK クエリはカタカナ・漢字セグメントを抽出（ひらがなを単語境界として扱う）
- **mismatch 単独で即 hard fail にしない**: Marginal + mismatch → Insufficient、Good + incoherent + mismatch → Insufficient だが、Good + mismatch のみでは維持
- コンテンツが関連している場合（mismatch=false）、タイトル incoherence だけでは Good→Marginal に留め、Insufficient にはしない

### Phase 4: expansion_failed フラグ + ハードストップ拡張（rag-orchestrator）

`AssessAnswerQuality()` のシグネチャに `expandedQueries []string` を追加し、因果クエリで展開が 0 件の場合に `"expansion_failed"` フラグを立てる。

ショートアンサー・ハードストップの発動条件を拡張:

```go
if hasQualityFlag("low_keyword_coverage") || hasQualityFlag("incoherent_ending") ||
   (hasQualityFlag("expansion_failed") && intentType == IntentCausalExplanation) {
```

## Consequences

### Pros

- **命令エコーの完全排除**: ベンチマークで全展開クエリがゴミだった状態から、関連クエリのみが通過する状態に改善
- **回答品質の劇的改善**: 129 文字 → 3,865 文字、因果構造（直接的要因/構造的背景/不確実性）を含む回答に改善
- **取得記事の関連性向上**: "Asset Tokenization"/"LibreFang" → "Iran escalates attacks"/"原油安定供給に懸念" に改善
- **多層防御**: 4 レイヤーの独立したフィルタにより、単一レイヤーの失敗が最終回答に直結しない
- **観測性向上**: `pre_filter_count`/`post_filter_count`/`expansion_all_filtered` ログで展開パイプラインの健全性を監視可能

### Cons / Tradeoffs

- **展開クエリ数が減少**: フィルタ強化により有効なクエリ数が減る場合がある（ベンチマークでは 8→1）。ただし元のクエリの embedding が有効に機能し、関連記事の取得に支障なし
- **文字列ヒューリスティックの脆弱性**: `isInstructionLeak()` の固定パターン配列は新しいプロンプト変更で更新が必要。RE-RAG 系の relevance estimator 導入が中長期的な解決策
- **`hasQueryContextMismatch` の CJK 限界**: ひらがな区切りによる簡易トークナイズは複合語（例: 「経済制裁」）の分割精度に限界がある
- **BM25 の 0 件問題は未解決**: 日本語クエリでの Meilisearch キーワード検索が機能しない問題は本 ADR のスコープ外
