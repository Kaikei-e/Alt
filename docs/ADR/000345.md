# SSRF脆弱性の修正: Connect ハンドラへの包括的バリデーション追加とURL解析差分の排除

## ADR's STATUS

Accepted

## CONTEXT

CodeQL の静的解析（`go/request-forgery`, CWE-918）により、`alt-backend` の HTTP リクエスト送信箇所が SSRF（Server-Side Request Forgery）のリスクとしてフラグされた。

調査の結果、以下の状況が判明した:

1. **REST ハンドラ**（`rest/rest_feeds/summarization/helpers.go`, `rest/rest_feeds/utils.go`, `rest/utils.go`）は既に3層の SSRF 防御を実装済み:
   - スキーム・プライベートIPの事前チェック（`url_validator.IsAllowedURL`）
   - DNS リバインディング検出を含む包括的バリデーション（`SSRFValidator.ValidateURL`）
   - 接続時IP検証付きセキュアHTTPクライアント（`SSRFValidator.CreateSecureHTTPClient`）

2. **Connect ハンドラ**（`connect/v2/feeds/handler.go`）には**実際の脆弱性が存在**:
   - `http.Client{}` をそのまま使用し、接続時のIP検証なし
   - DNS リバインディング攻撃に対する防御なし
   - リダイレクト先の検証なし

3. **全4箇所で TOCTTOU（Time-of-Check to Time-of-Use）問題**:
   - `url.Parse()` で検証した URL とは別に、`http.NewRequestWithContext()` に生の文字列 `urlStr` を渡しており、URL 解析差分による検証バイパスの余地があった

OWASP SSRF Prevention Cheat Sheet では、DNS解決結果の**接続時検証**、リダイレクト先の検証、スキーム・ポートの制限、プライベート/リンクローカル/メタデータIPのブロックが推奨されている。

## DECISION MAKING

### 1. Connect ハンドラに REST ハンドラと同等の SSRF 防御を追加

`connect/v2/feeds/handler.go` の `fetchArticleContent()` に、既存の `SSRFValidator` を適用する。新たなバリデータを作成するのではなく、`utils/security/ssrf_validator.go` に実装済みの検証ロジックを再利用することで一貫性を保つ。

### 2. 全箇所で `parsedURL.String()` を使用

`http.NewRequestWithContext()` に渡す URL を、生の入力文字列 `urlStr` ではなく、`url.Parse()` 済みの `parsedURL.String()` に統一する。これにより URL パーサ間の解釈差異を利用した攻撃を防止する。

### 3. `helpers.go` のローカル `isAllowedURL` を削除

`summarization/helpers.go` に定義されていたローカルの `isAllowedURL()` は `SSRFValidator.ValidateURL()` の機能のサブセットであり、冗長であった。共通ユーティリティの `url_validator.IsAllowedURL()` に置き換えて、防御ロジックの重複と乖離を排除した。

## RESULTS, EFFECTS

### 変更対象ファイル

| ファイル | 変更内容 |
|---------|---------|
| `connect/v2/feeds/handler.go` | `SSRFValidator` + `CreateSecureHTTPClient()` 追加、`parsedURL.String()` 使用 |
| `rest/rest_feeds/summarization/helpers.go` | `parsedURL.String()` 使用、ローカル `isAllowedURL` 削除 → `url_validator.IsAllowedURL` に置換 |
| `rest/rest_feeds/utils.go` | `parsedURL.String()` 使用 |
| `rest/utils.go` | `parsedURL.String()` 使用 |

### 防御の構造（修正後、全箇所共通）

```
1. url.Parse()                         -- URL構造の正規化
2. url_validator.IsAllowedURL()        -- スキーム・プライベートIP基本チェック
3. SSRFValidator.ValidateURL()         -- メタデータ、DNSリバインディング、ポート、Unicode/Punycode、パストラバーサル
4. CreateSecureHTTPClient()            -- 接続時IP検証（net.Dialer.Control フック）
5. http.NewRequestWithContext(parsedURL.String())  -- 検証済みURLを使用
```

### PROS

- **DNS リバインディング攻撃の防止**: 接続時に実際の解決先IPを再検証するため、DNS応答を操作する攻撃手法が無効化される
- **クラウドメタデータサービスの即座のブロック**: 修正前は `169.254.169.254` へのアクセスが 10 秒のタイムアウトまでブロックされなかったが、修正後は即座に拒否される（10s → 0.09s）
- **URL 解析差分攻撃の排除**: 検証済みの正規化URLのみがHTTPリクエストに使用される
- **既存コードの再利用**: 新規のセキュリティコンポーネントを追加せず、`SSRFValidator` を一貫して適用

### CONS, TRADEOFF

- **DNS解決の二重実行**: `url_validator.IsAllowedURL()` と `SSRFValidator.ValidateURL()` でそれぞれDNS解決が行われる。パフォーマンスへの影響は軽微だが、defense-in-depth のトレードオフとして許容した
- **ディナイリスト方式の限界**: 現在の防御はすべてディナイリスト（既知の危険なIPレンジ・ドメインをブロック）に基づく。アローリスト方式（許可ドメインのみ接続可能）はより安全だが、RSS フィードの特性上（任意のドメインからコンテンツを取得する必要がある）、採用が困難
