# ADR-000748: nginx TLS sidecar を Go の pki-agent proxy モードに統合して cert rotation を hot-reload 化する

## Status

Accepted

## Date

2026-04-16

## Affected Services

- **pki-agent** (Go): `internal/adapter/handler/proxy.go` 新設で TLS reverse proxy を実装。
  `internal/infrastructure/tlsutil.go` を `auth-hub/tlsutil` からコピーして再利用 (既存慣例)。
  `config/config.go` に `PROXY_LISTEN` / `PROXY_UPSTREAM` / `PROXY_VERIFY_CLIENT` /
  `PROXY_ALLOWED_PEERS` / `PROXY_CA_FILE` を追加。`cmd/pki-agent/main.go` で
  `PROXY_LISTEN` が設定された場合のみ TLS listener を起動する optional mode に
- **acolyte-orchestrator** (Python): mTLS 終端経路を nginx → Go 化。
  `compose/acolyte.yaml` から `acolyte-orchestrator-tls-sidecar` を削除、
  `acolyte-orchestrator` の `depends_on` から `pki-agent-acolyte-orchestrator` を外して
  起動順の循環を解消
- **tag-generator** (Python): 同じくnginx sidecar を削除、`compose/workers.yaml` から
  `tag-generator-tls-sidecar` を撤去
- **compose 基盤**: `compose/pki.yaml` で `pki-agent-acolyte-orchestrator` と
  `pki-agent-tag-generator` に `network_mode: "service:<upstream>"` + proxy 用 env を追加

## Context

[[000747]] で shell cert-init/cert-renewer を Go の `pki-agent` に置き換えたが、
Python サービスの mTLS 終端を担う **nginx TLS sidecar** は nginx:1.27-alpine のまま残置していた。
nginx は起動時にしか証明書をメモリにロードしないため、`pki-agent` が cert を atomic rename で
差し替えても **nginx の worker プロセスは旧 cert を掴み続ける**。

実際に 2026-04-16 に「Acolyte が死んでいる」としてユーザー通報が発生:

- `alt-acolyte-orchestrator-tls-sidecar-1` が 32 時間稼働のまま
- `pki-agent-acolyte-orchestrator` は 1 時間前に新 cert を発行済み
- nginx は期限切れ直前の旧 cert を掴み続けており、BFF からの mTLS が破綻
- `docker compose restart <sidecar>` で一時復旧したが、次の rotation (16h 後) で再発する構造

[[PM-2026-028]] の Security audit で F-007 Medium として記録した既知の穴が顕在化したかたち。
一時回避として shell entrypoint に inotifywait / polling + `nginx -s reload` を仕込む選択肢も
あったが、[[000747]] の方針 — **脆弱な shell グルーを廃して Go マイクロサービスに集約する** —
に反する。根治策は nginx sidecar 自体を Go に置き換えること。

### 公式ドキュメントの裏付け

[nginx 公式ブログ (2025)](https://blog.nginx.org/blog/ssl-tls-certificate-rotation-without-restarts-in-nginx-open-source)
は njs `js_shared_dict_zone` によるゼロダウンタイム rotation を推奨するが、
njs モジュール + JavaScript コード + `/clear` エンドポイントが必要で単純な mTLS sidecar には過剰。
Docker sidecar 界隈の業界標準は「cert 変更検知で `nginx -s reload`」だが、Go の
`tls.Config.GetCertificate` callback は **TLS handshake ごとに mtime を見て再読み込み** するため、
この問題は Go を採用した時点で構造的に消滅する。

## Decision

1. `pki-agent` に **optional な TLS reverse proxy mode** を追加する。
   環境変数 `PROXY_LISTEN` が設定された場合のみ追加の TLS listener を起動し、
   `PROXY_UPSTREAM` に HTTP/1.1 で転送する。未設定なら従来の cert-only モード (Go サービス向け)
2. Python サービス (acolyte-orchestrator / tag-generator) では `pki-agent` を
   `network_mode: "service:<upstream>"` で起動し、upstream コンテナのネットワーク名前空間を
   共有することで **localhost 経由で upstream に到達** できるようにする
3. `X-Alt-Peer-Identity` ヘッダ (peer CN の upstream 注入) の契約は nginx sidecar と同一に保つ
4. **16 本の nginx sidecar + pki-agent sidecar 体制から 8 本の pki-agent 単独体制へ統合**

## Architecture

### `internal/adapter/handler/proxy.go`

`auth-hub/tlsutil` から移植した `tlsutil.LoadServerConfig` (`certReloader` 内蔵) と
Go 標準の `httputil.NewSingleHostReverseProxy` を組み合わせる:

```go
func NewTLSProxy(cfg ProxyConfig) (*http.Server, error) {
    var opts []infrastructure.ServerOption
    if cfg.VerifyClient {
        opts = append(opts, infrastructure.WithClientAuth(tls.RequireAndVerifyClientCert))
    }
    if len(cfg.AllowedPeers) > 0 {
        opts = append(opts, infrastructure.WithAllowedPeers(cfg.AllowedPeers...))
    }
    tlsCfg, err := infrastructure.LoadServerConfig(cfg.CertPath, cfg.KeyPath, cfg.CAPath, opts...)
    if err != nil {
        return nil, err
    }
    target, _ := url.Parse(cfg.Upstream)
    proxy := httputil.NewSingleHostReverseProxy(target)
    handler := http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        if r.TLS != nil && len(r.TLS.PeerCertificates) > 0 {
            r.Header.Set("X-Alt-Peer-Identity", r.TLS.PeerCertificates[0].Subject.CommonName)
        }
        r.Header.Set("X-Forwarded-Proto", "https")
        proxy.ServeHTTP(w, r)
    })
    return infrastructure.NewMTLSHTTPServer(cfg.Listen, tlsCfg, handler), nil
}
```

`certReloader` が TLS handshake ごとに cert ファイルの mtime を見て再読み込みするため、
`pki-agent` の atomic rename による差し替えは **次の接続で即座に有効** になる。
nginx の「起動時ロード固定」問題は Go では構造的に発生しない。

### compose 構成の変更

**Before** (3 コンテナ/Python サービス):

```
acolyte-orchestrator              # Python :8090
acolyte-orchestrator-tls-sidecar  # nginx :9443 (network_mode: service:acolyte-orchestrator)
pki-agent-acolyte-orchestrator    # cert issuer (alt-network)
```

**After** (2 コンテナ/Python サービス):

```
acolyte-orchestrator                    # Python :8090 (変更なし)
pki-agent-acolyte-orchestrator          # cert issuer + TLS proxy :9443
    network_mode: "service:acolyte-orchestrator"
    depends_on:
      acolyte-orchestrator: service_healthy
      step-ca: service_healthy
    environment:
      PROXY_LISTEN: ":9443"
      PROXY_UPSTREAM: "http://127.0.0.1:8090"
      PROXY_VERIFY_CLIENT: "on"
      PROXY_ALLOWED_PEERS: "alt-butterfly-facade"
```

### 起動順の循環回避

`pki-agent` が upstream の network namespace を共有する構成では、upstream が先に起動する必要がある。
従来は `upstream` が `pki-agent` の cert 発行に依存していたため循環する。
解消のため、**Python upstream は cert に依存しない** 前提に整理した:

- Python app 自身 (acolyte-orchestrator / tag-generator) は plaintext :8090 / :9400 で起動できる
  — TLS 終端は sidecar の責務
- したがって upstream の `depends_on` から pki-agent を外し、pki-agent の `depends_on` に
  upstream を入れる。外部からの `:9443` 可用性は pki-agent 自身の healthcheck で担保

## セキュリティ考慮

- **TLS 1.3 固定**: `tlsutil.LoadServerConfig` が `MinVersion: tls.VersionTLS13` を強制 —
  nginx の `ssl_protocols TLSv1.3;` と同等
- **peer allowlist**: `VerifyConnection` callback で CN/DNSNames 照合 — nginx の
  `$ssl_client_s_dn_cn` + if-allowlist の Go 等価。env `PROXY_ALLOWED_PEERS` から derive
- **IdleTimeout 60s**: `NewMTLSHTTPServer` が HTTP/2 connection reuse を 24h leaf TTL より
  十分短く切る
- **X-Alt-Peer-Identity ヘッダ信頼**: upstream (Python app) は localhost からの接続のみを受ける
  (`network_mode` 共有のため) — 外部からのヘッダ注入経路なし。契約は [[000737]] と同一
- **cert hot-reload の thread-safety**: `certReloader.load()` は mutex + mtime check で
  同時 handshake でも安全

## Consequences

### 良い結果

- **F-007 の恒久解消**: nginx 起動時 cert ロード固定による「期限切れ旧 cert 継続使用」問題が
  Go 化で構造的に消滅。手動 `docker restart` が不要になる
- **コンテナ数削減**: 22 → 8 (rotation 系サイドカー全体)、さらに Python 2 サービス分で 2 コンテナ削減
- **運用言語の統一**: cert ライフサイクルも TLS 終端も Go で完結。shell / nginx / njs の混在解消
- **TDD ガードの拡張**: TLS proxy の挙動が `go test -race` で回る。nginx の挙動変化に振り回されない

### トレードオフ

- **`tlsutil` 重複**: `auth-hub/tlsutil/tlsutil.go` と `alt-backend/app/tlsutil` に続き、
  `pki-agent/internal/infrastructure/tlsutil.go` でも重複。将来的に Alt 共有 Go module
  (`github.com/Alt-Project/tlsutil`) への切り出しを検討 (他 ADR で別途)
- **起動順の見直し**: Python upstream は cert なしで起動する前提に変更。仮に plaintext :8090
  への外部到達があっても問題ないよう、[[000741]] の `PEER_IDENTITY_TRUSTED` ゲートが引き続き防衛線

### 残リスク

- **[[000747]] F-001 (Critical)** — bootstrap JWK provisioner を 8 pki-agent で共有する状態は
  本 ADR でも据え置き。別 ADR (phase 2.5) で専用 JWK provisioner + `x509.allowedNames` allowlist を
  実装する予定
