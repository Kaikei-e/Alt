# ADR-000393: OPML 2.0 インポート/エクスポート機能を実装する

## Status

Accepted

## Date

2026-03-16

## Affected Services

- alt-backend (Go / Echo)
- alt-frontend-sv (SvelteKit / TypeScript)

## Context

Alt RSS プラットフォームはフィードリンクの個別登録・削除機能を提供しているが、他の RSS リーダーからの一括移行手段がなかった。RSS リーダー間の標準的なフィードリスト交換形式である OPML（Outline Processor Markup Language）2.0 に対応することで、ユーザーが以下を実現できるようにする必要があった:

1. **インポート**: 他の RSS リーダー（Feedly, Inoreader, NetNewsWire 等）からエクスポートした OPML ファイルを Alt に取り込む
2. **エクスポート**: Alt に登録済みのフィードリストを OPML ファイルとしてダウンロードし、バックアップや他サービスへの移行に使用する

### 設計上の判断ポイント

インポート時に RSS フィードの検証（HTTP フェッチ + パース）を行うか、URL を直接 DB に挿入するかが主要な論点であった。既存の `RegisterFeedsUsecase` は 1 URL ごとに外部 HTTP リクエストを発行してフィードを検証・取得するため、数百件の OPML インポートでは外部 API レート制限やタイムアウトの問題が避けられない。

## Decision

**URL 直接登録（高速モード）を採用する。** インポート時は RSS 検証をスキップし、`feed_links` テーブルに直接挿入する。バックグラウンドのフェッチサイクルが後から検証を行う。

### 検討した代替案

1. **既存の `RegisterFeedsUsecase` をループ呼び出し**: 1 URL ごとに RSS フェッチ + パース + フィード登録を実行する案。却下理由 — 外部 HTTP リクエストが URL 数 × 1 回発生し、レート制限（5 秒間隔ルール）により 100 件のインポートに 500 秒以上かかる。HTTP タイムアウトのリスクも高い。
2. **非同期ジョブキュー経由のインポート**: mq-hub にインポートジョブを投入し、ワーカーが逐次処理する案。却下理由 — 過剰なインフラ複雑性。OPML インポートは低頻度の操作であり、DB 直接挿入で十分。
3. **URL 直接登録（採用）**: `feed_links` テーブルに INSERT し、既存の重複ハンドリング（SQLSTATE 23505）でスキップ。SSRF 保護は `url_validator.IsAllowedURL()` で担保。フィードの実在性はバックグラウンドフェッチが後で検証する。

### アーキテクチャ

Clean Architecture に従い、OPML 専用のレイヤーを新設した:

```
Handler (REST)
  └─ Usecase (XML パース/生成 + ビジネスロジック)
       └─ Port (インターフェース)
            └─ Gateway (DB アクセス + SSRF 検証)
                 └─ Driver (AltDBRepository)
```

OPML XML のパース・生成は `encoding/xml` 標準ライブラリを使用し、外部依存を追加していない。XML External Entity (XXE) 攻撃については、Go の `encoding/xml` がデフォルトで外部エンティティを無効化しているため追加対策は不要。

## Implementation

### Backend

| レイヤー | ファイル | 役割 |
|---------|---------|------|
| Domain | `domain/opml.go` | `OPMLDocument`, `OPMLOutline`, `OPMLImportResult`, `FeedLinkForExport` |
| Port | `port/opml_port/opml_port.go` | `ExportOPMLPort`, `ImportOPMLPort` インターフェース |
| Gateway | `gateway/opml_gateway/export_gateway.go` | `feed_links` と `feeds` の JOIN クエリでタイトル付きリストを取得 |
| Gateway | `gateway/opml_gateway/import_gateway.go` | URL ごとに SSRF 検証 → 重複チェック → DB 挿入 |
| Usecase | `usecase/opml_usecase/export_usecase.go` | フィードリスト → OPML 2.0 XML 生成 |
| Usecase | `usecase/opml_usecase/import_usecase.go` | OPML XML パース → URL 抽出（ネスト対応 + 重複排除）→ 一括登録 |
| Handler | `rest/rest_feeds/opml.go` | `GET /v1/rss-feed-link/export/opml`, `POST /v1/rss-feed-link/import/opml` |
| DI | `di/container.go` | Gateway → Usecase のワイヤリング |
| Route | `rest/rest_feeds/routes.go` | `rss` グループに 2 ルート追加（認証必須） |

#### エンドポイント仕様

**エクスポート** `GET /v1/rss-feed-link/export/opml`
- Response: `application/xml`, `Content-Disposition: attachment; filename="alt-feeds.opml"`
- 認証: `authMiddleware.RequireAuth()` 適用済み

**インポート** `POST /v1/rss-feed-link/import/opml`
- Request: `multipart/form-data` (`file` フィールド)
- Response: JSON `OPMLImportResult` (`total`, `imported`, `skipped`, `failed`, `failed_urls`)
- ファイルサイズ制限: 1MB (`io.LimitReader`)
- SSRF 保護: 各 URL に `url_validator.IsAllowedURL()` を適用
- 認証: `authMiddleware.RequireAuth()` 適用済み

#### Export SQL クエリ

```sql
SELECT fl.url, COALESCE(sub.title, '') AS title
FROM feed_links fl
LEFT JOIN LATERAL (
    SELECT DISTINCT ON (feed_link_id) title
    FROM feeds WHERE feed_link_id = fl.id
    ORDER BY feed_link_id, created_at DESC
) sub ON true
ORDER BY fl.url ASC
```

タイトルが取得できない場合は URL のドメイン名をフォールバックとして使用する。

| ファイル | 役割 |
|---------|------|
| `src/lib/schema/opml.ts` | `OPMLImportResult` TypeScript 型定義 |
| `src/lib/api/client/opml.ts` | `exportOPMLClient()`, `importOPMLClient()` クライアント関数 |

| `src/lib/components/feeds/OpmlSection.svelte` | UI コンポーネント（Desktop/Mobile 両対応） |

#### Desktop UI

2 カラム構成: ドラッグ & ドロップゾーン + Export ボタン。ファイルをドロップまたはクリックで選択し、結果を Dialog で表示する。

#### Mobile UI

Export / Import の 2 ボタン横並びレイアウト。Import はネイティブのファイルピッカー（hidden `<input type="file">`）を使用する。

#### クライアントサイド検証

- ファイル拡張子: `.opml`, `.xml`
- MIME タイプ: `text/xml`, `application/xml`, `text/x-opml`
- ファイルサイズ: 1MB 上限

### テスト

- **Usecase テスト**: 11 テスト（export 4 + import 7）— OPML 生成/パースの正確性、ネスト対応、重複排除、不正 XML、空リスト
- **Handler テスト**: 4 テスト — エンドポイント応答、Content-Type、ファイルサイズ制限、missing file エラー
- **全テスト Pass**: `go test ./usecase/opml_usecase/... ./rest/rest_feeds/...` で 15/15 合格

## Consequences

### Pros

- **高速インポート**: RSS 検証スキップにより数百件の OPML を数秒で処理可能。外部 HTTP リクエストが発生しないためレート制限の問題がない
- **標準準拠**: OPML 2.0 形式をサポートし、主要な RSS リーダー（Feedly, Inoreader, NetNewsWire, Miniflux 等）との相互運用性を確保
- **ネスト対応**: カテゴリ階層を持つ OPML（`<outline>` のネスト）も再帰的に走査して全フィード URL を抽出
- **セキュリティ**: SSRF 保護（プライベート IP 拒否）、XXE 防止（Go 標準ライブラリ）、ファイルサイズ制限（1MB）、認証必須、CSRF 検証
- **Clean Architecture 準拠**: 既存のレイヤー構造に沿った新規コンポーネント追加。既存コード（`RegisterRSSFeedLink`, `FetchFeedLinks` 等）を再利用
- **Desktop/Mobile 両対応**: Desktop はドラッグ & ドロップ、Mobile はネイティブファイルピッカーと、デバイスに適した UX を提供

### Cons / Tradeoffs

- **フィードの実在性は即時検証されない**: インポート直後は `feed_links` テーブルに URL が存在するだけで、フィードの有効性はバックグラウンドフェッチまで不明。ユーザーに「後で検証される」ことを伝える必要がある
- **カテゴリ情報の損失**: OPML のカテゴリ階層（`<outline text="Tech">` 等）はインポート時に無視される。Alt にはカテゴリ概念がないため、フラットな URL リストとして取り込む
- **エクスポート時のタイトル精度**: `feeds` テーブルの最新タイトルを使用するが、一度もフェッチされていない `feed_link` はドメイン名がタイトルになる
