# ADR-000439: TodayBar と MiniRecallPanel の状態契約を backend truth に統一する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-frontend-sv (TypeScript/Svelte)

## Context

Knowledge Home の Phase 0〜2 を経て、TodayBar・MiniRecallPanel・RecallRail の 3 コンポーネントが表示する情報の責務境界が曖昧になっていた。

### TodayBar の問題

- `weeklyRecapAvailable` / `eveningPulseAvailable` で CTA の活性を制御しているが、`false` の場合に「機能がない」と「今日は対象がない」の区別がなかった
- Phase 3 で追加された `digestFreshness` を表示する必要があるが、stale と availability の関係が未定義だった

### MiniRecallPanel の問題

- Top Tags、Quick Links（Morning Letter / Recap / Pulse / Ask Augur）、Explore（Tag Trail / Tag Verse）の 3 セクションを持ち、digest availability による条件分岐が混在していた
- Recap / Pulse リンクの活性判定は TodayBar と重複しており、MiniRecallPanel が独自に availability を解釈する形になっていた
- RecallRail flag OFF 時の右 rail 代替パネルとして MiniRecallPanel を使うが、navigation panel としての機能と digest availability 表示が同居していた

## Decision

**TodayBar は backend 契約値をそのまま表示し UI 独自の availability 推測をやめる。MiniRecallPanel は navigation panel に限定し digest availability 条件分岐を削除する。**

### TodayBar の変更

1. **CTA 活性は `weeklyRecapAvailable` / `eveningPulseAvailable` のみで決定する** — UI は boolean をそのまま使い、独自の推測をしない
2. **disabled 状態に reason を付与** — `title` 属性で "No pulse content available today" / "No recap available today" を表示し、「機能がない」ではなく「今日は対象がない」と伝える
3. **stale indicator を追加** — `digestFreshness === "stale"` のとき amber の Clock アイコン + "Stale" テキストを action shortcuts 行の右端に表示
4. **stale は availability を上書きしない** — stale であっても `weeklyRecapAvailable=true` ならリンクは活性のまま

これにより TodayBar は以下の 4 状態を表現する:

| 状態 | 意味 | 表示 |
|---|---|---|
| enabled | backend が有効と判断 | リンク活性 |
| disabled | backend が無効と判断 | リンク無効 + reason tooltip |
| stale | freshness warning | リンク状態とは独立に amber indicator |
| unknown | freshness を断定しない | indicator なし |

### MiniRecallPanel の変更

1. **digest availability による Recap / Pulse リンクの条件表示を削除** — これらの CTA は TodayBar が担う
2. **navigation panel に限定** — Quick Links（Morning Letter / Ask Augur）と Explore（Tag Trail / Tag Verse）のみ
3. **表示条件は維持** — RecallRail flag OFF 時のみ Home 右 rail に表示

### 右 rail の境界（確認のみ、変更なし）

既存の `+page.svelte` のロジックが ADR と一致していることを確認:

```svelte
{#if recallEnabled}
    <RecallRail ... />
{:else}
    <MiniRecallPanel ... />
{/if}
```

- RecallRail flag ON: RecallRail（候補 0 件でも empty state を表示）
- RecallRail flag OFF: MiniRecallPanel（navigation panel）
- RecallRail + MiniRecallPanel の同時表示はしない

### Frontend 型の変更

`TodayDigestData` に `digestFreshness` と `lastProjectedAt` を追加:

```typescript
export type DigestFreshness = "fresh" | "stale" | "unknown";

export interface TodayDigestData {
    // ... existing fields ...
    digestFreshness: DigestFreshness;
    lastProjectedAt: string | null;
}
```

`convertDigest()` で proto の新フィールドを変換。`digestFreshness` が空文字の場合は `"unknown"` にフォールバック。

## Consequences

### Pros

- TodayBar が backend 契約値の忠実な表示層になり、UI 独自の availability 推測が消える
- MiniRecallPanel が navigation panel に限定され、TodayBar との責務重複がなくなる
- disabled 状態の tooltip が「機能がない」ではなく「今日は対象がない」と伝え、UX が改善される
- stale indicator が availability とは独立に表示されるため、「データは古いがリンクは使える」ことが伝わる

### Cons / Tradeoffs

- MiniRecallPanel から Recap / Pulse リンクが消えるため、RecallRail flag OFF のユーザーは右 rail から Recap / Pulse に直接遷移できなくなる。ただし TodayBar に同等のリンクがあるため実用上の影響は最小限
- `digestFreshness` の型が string union であり、proto の string フィールドに依存する。将来的に proto enum に変更する場合は breaking change になるが、Phase 3 では string で十分と判断
