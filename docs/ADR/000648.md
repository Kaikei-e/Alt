# ADR-000648: クエリ展開の few-shot 例汚染とウェルカムメッセージ混入を修正し Augur の retrieval 精度を回復する

## Status

Accepted

## Date

2026-04-08

## Affected Services

- news-creator (Python) — クエリ展開プロンプトの few-shot 例修正
- rag-orchestrator (Go) — 会話メッセージ混入の防御フィルタ追加
- alt-frontend-sv (TypeScript / SvelteKit) — ウェルカムメッセージの会話履歴除外

## Context

[[000647]] で Intent 別プロンプトテンプレート・pgvector hybrid search・Eino フレームワーク基盤を導入したが、実運用テストで Ask Augur の回答が依然として fallback を返す問題が継続していた。

コンテナログの徹底調査により、プロンプトテンプレート改善が発動する **前段の retrieval パイプライン** に 2 つの独立したバグがあり、関連コンテキストが一切取得できていないことが判明した。

### Bug 1: クエリ展開 few-shot 例のドメイン固定

news-creator の `EXPAND_QUERY_WITH_HISTORY_TEMPLATE` に「AIチップ市場」を題材にした単一の few-shot 例が含まれていた。Gemma 4 (12B) はこの例の **内容をパターンではなくテンプレートとして学習** し、入力クエリのトピックに関係なく AI チップ関連の展開クエリを生成していた。

```
入力: 「イランの石油危機が発生した原因と詳細な背景」
展開結果: ["イランの石油危機 競合企業 AMD Intel Google TPU",
           "AI chip market competitors AMD Intel custom silicon",
           "AI accelerator companies beyond NVIDIA 2026"]
```

この結果、vector search と BM25 の両方が AI チップ関連記事ばかりをヒ���トさせ、実際のトピック（石油危機）に関する記事が retrieval されなかった。

### Bug 2: フロントエンドのウェルカムメッセージが会話履歴に混入

`AugurChat.svelte` の初期状態に含まれるウェルカムメッセージ（`id: "welcome"`, 内容: "Hello! I'm Augur. Ask me anything about your RSS feeds."）が、`handleSend()` で会話履歴を構築する際にフィルタされず、rag-orchestrator → news-creator のクエリ展開に会話コンテキストとして送信されていた。

```
展開結果に混入: "assistant: Hello! I m Augur. Ask me anything about your RSS feeds."
```

これら 2 つのバグが組み合わさることで、retrieval パイプラインが完全に壊れ、downstream のプロンプトテンプレート改善・品質ゲート・因果分析戦略がすべて無効化されていた。

## Decision

**3 層の修正を同時実施し、上流から下流まで一貫して retrieval 汚染を排除する。**

### 1. news-creator: few-shot 例をドメイン中立に置換

`EXPAND_QUERY_WITH_HISTORY_TEMPLATE` の few-shot 例を以下のように修正する。

- 「AIチップ市場」の単一例を削除し、「天気予報」と「スマートフォンカメラ」の **2 つのドメイン中立な例** に置換
- 例の出力からラベルプレフィックス（`Japanese(1):` / `English(3):`）を除去し、モデルがラベルをコピーするリスクを排除
- 例と実入力の間に `---` セパレータを挿入し、例の終わりを明示
- ルールに「IMPORTANT: Generate queries about the SAME TOPIC as the input. Do NOT copy topics from examples.」を追加

研究知見に基づく設計判断:
- 2-3 例で diminishing returns（[PromptHub](https://www.prompthub.us/blog/the-few-shot-prompting-guide)）
- 小型モデルは few-shot 例の内容をコピーしやすく、内容は中立にすべき（[Prompt Engineering Guide](https://www.promptingguide.ai/techniques/fewshot)）
- verbose な展開は BM25 に有害な spurious terms を生む（[Query Expansion by Prompting LLMs, arXiv 2305.03653](https://arxiv.org/pdf/2305.03653)）

### 2. alt-frontend-sv: ウェルカムメッセージの会話履歴除外

`AugurChat.svelte` の `handleSend()` で会話履歴を構築する際に `.filter((m) => m.id !== "welcome")` を追加し、ウェルカムメッセージを除外する。

### 3. rag-orchestrator: 会話メッセージ混入の防御フィルタ

`filterExpandedQueries` に `isConversationMessageLeak()` フィルタを追加し、`assistant:` や `user:` で始まる展開クエリを除外する。フロントエンド修正が不十分な場合のセーフティネットとして機能する。

### 採用しなかった案

| 案 | 内容 | 判断 |
|---|---|---|
| few-shot 例を完全に削除（zero-shot 化） | 例なしでクエリ展開させる | Gemma 4 (12B) は zero-shot ではフォーマットが不安定。構造パターンを示す例は必要 |
| ウェルカムメッセージを BFF で除去 | alt-butterfly-facade で welcome フィルタ | BFF は transparent proxy であるべき。ビジネスロジックを BFF に持たせない |
| rag-orchestrator で全会話履歴を除去 | history_turns > 0 のときのみクエリ展開に履歴を渡す | 正当な会話履歴はコリファレンス解決に必要。除去すべきは synthetic メッセージのみ |

## Consequences

### Pros

- **retrieval 精度が回復**: top_hits が「Google Alphabet 決算」→「Iran warns US」「原油103ドル」「エチレン国内生産」に改善
- **因果分析が機能**: `causal_strategy_early_exit` が発動し、2,430 bytes の構造化回答を生成
- **フォローアップが機能**: 「アメリカはどう関係している？」で synthesis 戦略 + tool dispatch が発動し、4,081 bytes の多面的分析を生成
- **3 層防御**: フロントエンド（ウェルカム除外）→ news-creator（中立 few-shot）→ rag-orchestrator（会話メッセージフィルタ）で汚染リスクを多層的に排除
- **contract 不変**: expand-query endpoint のリクエスト/レスポンス形式は変更なし。Pact テスト通過

### Cons / Tradeoffs

- **展開クエリの多様性が低下**: ドメイン中立化の副作用で、展開が元クエリのコピーのみ（`expanded_count:1`）になるケースがある。ただし CausalStrategy の subquery decomposition がカバーするため、最終結果への影響は限定的
- **ラベル付きクエリの残存**: フォローアップ展開で `American(1):` のようなラベルが残る場合がある。`filterExpandedQueries` のラベル除去フィルタの追加改善が必要
- **ウェルカムメッセージの ID 依存**: `id !== "welcome"` はフロントエンドの実装詳細に依存。将来的にはメッセージに `synthetic` フラグを持たせる方が堅牢
