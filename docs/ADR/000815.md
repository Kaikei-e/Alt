# ADR-000815: search-indexer bootstrap の panic log テストから t.Parallel を外し logger.Logger グローバル競合を解消する

## Status

Accepted

## Date

2026-04-20

## Affected Services

- **search-indexer (Go)** — `app/bootstrap/panic_log_test.go` の 2 テストから `t.Parallel()` を除去

## Context

2026-04-19 の CI run Alt/24630349184 (commit `31ebb5a5`) で `search-indexer` の `test / test-go` ジョブが `TestLogPanic_IncludesDetailAtDebugLevel` の失敗で赤転した。該当ログは以下:

```
panic_log_test.go:55: expected panic detail at debug level, got:
{"time":"...","level":"ERROR","msg":"index loop panic","err_type":"string"}
{"time":"...","level":"DEBUG","msg":"index loop panic detail",
 "err":"leaked-service-token-abc123",
 "stack":"goroutine 8 [running]:\n...TestLogPanic_DoesNotLeakPanicValueAtInfoLevel..."}
```

テストは `"debug-detail-marker"` を buffer に探しにいくが、buffer には姉妹テスト `TestLogPanic_DoesNotLeakPanicValueAtInfoLevel` が emit した `"leaked-service-token-abc123"` の DEBUG record が入っていた。DEBUG record のスタックトレースが姉妹テストの名前を指していることから、両テストの buffer が交差していたのは明らか。

根本原因は `captureLogs` (`panic_log_test.go:16`) が **package-level global** `logger.Logger` を swap する設計であること。同じファイルの 2 テスト両方が `t.Parallel()` を呼んでおり、goroutine スケジューリング次第で一方の `captureLogs` 後に他方が再 swap し、先に swap 済みのテストの `logPanic` 呼び出しが **後から swap した側** の buffer に書き込まれる。CI の多核環境ではこの race が顕在化しやすい。

`logger.Logger` は `slog.Default` 風の本番用 global で、`Init()` から 1 回だけ設定される前提で書かれており、テストのために instance-level logger を引き回す設計にはなっていない。テストだけのために `logPanic` のシグネチャを `logger *slog.Logger` を取るように変えるのは、呼び出し側 (`bootstrap/main_loop.go` の panic recover path) への波及が大きい。

## Decision

2 テストから `t.Parallel()` を除去し、**逐次実行を強制する**。`captureLogs` の global swap 設計は今回触らない。

```go
// 変更前
func TestLogPanic_DoesNotLeakPanicValueAtInfoLevel(t *testing.T) {
    t.Parallel()
    buf := captureLogs(t, slog.LevelInfo)
    ...

// 変更後
func TestLogPanic_DoesNotLeakPanicValueAtInfoLevel(t *testing.T) {
    buf := captureLogs(t, slog.LevelInfo)
    ...
```

`panic_log.go` の実装 (ERROR 要約 + DEBUG 詳細の 2 record emit) は変更しない。

**却下した代替案**:

- **`captureLogs` が instance-level logger を返し、`logPanic(logger, ...)` 形式に変える** — `logPanic` の呼び出し元 (`bootstrap/main_loop.go`) は `defer recover()` から呼ぶ関数で、`context.Context` 以外の依存を足したくない。本番コードを test 都合で汚す費用対効果が合わない。
- **`sync.Mutex` を `logger.Logger` 周りに置く** — panic path の critical section を太らせる割に、テスト以外のユーザーが居ない。テスト側を直列化する方が素直。
- **テスト package を `_test` サブパッケージに分離し `logger` の internal 参照を切る** — 同一 package 内の access をいじる大改造になる。ADR の射程外。

## Consequences

### Pros

- **1 行削除 2 箇所だけで green に戻せる**。実装の不変条件（panic record の level/shape）は維持される。
- **`-race` 下でも安定**。ローカルで `go test ./bootstrap/... -race -count=3` を走らせ、3 iteration 全てで両テストが pass することを確認済み。
- **global swap 設計を変えないので他の bootstrap テストへの影響はない**。

### Cons / Tradeoffs

- **テスト時間は微増**（< 10ms）。そもそも panic log 系テストは 2 本で数 ms のため体感不能。
- **将来また同じパッケージにテストが増えるとき `t.Parallel()` を付けると再発する**。本 ADR と `panic_log_test.go` の直前コミットがその警告になる想定だが、コメントで明示化まではしていない（将来のテスト追加者の責務）。
- **本質的な設計負債（global logger.Logger の swap）は残る**。後続で logger を DI 化するときに合わせて解消する。本 ADR では現実主義的に逐次化で止める。
