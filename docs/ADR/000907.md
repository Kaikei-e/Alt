# ADR-000907: Review bucket の driver を user dismiss から epistemic change (staleness / contradiction / version drift / unfinished thread) に切替え Why narrative を v8 へ bump する

## Status

Accepted

## Date

2026-05-20

## Affected Services

- knowledge-sovereign (Go)
- alt-frontend-sv (TypeScript / Svelte 5)

## Context

[[knowledge-loop-completion-03-review-why-quality]] は Review bucket を「dismiss したものの置き場」から「staleness / contradiction / version drift / unfinished thread の再評価キュー」に進化させることを Phase 3 として記述している。現状 (2026-05-20 audit) は次の混在状態:

- `HomeItemDismissed` 由来の v1 fallback Review (user が捨てたものを「あとで再評価」と称して残置)
- v2 staleness/contradiction signal による Review (epistemic 駆動)
- 双方が同じ Review bucket に混在し、UI 上は区別がつかない

これは構想層 §4-Δ8 が指摘する **「Review の起点が dismiss である点が思想と矛盾」** に該当する。ユーザが捨てたものは「もう見たくない」signal であり、それを「もう一度見ろ」キューに混ぜるのは認知的に逆行している。一方、staleness や contradiction、version drift、unfinished thread は **system 由来の epistemic change** であり、本来の Review (再評価) 役割に正しく対応する。

加えて Why narrative は WhyMappingVersion v6 (Wave 4-C [[000854]]) → v7 (中間) で進化中だが、v2 evidence (`topic_overlap_count`, `tag_overlap_count`, `version_drift_count`, `has_augur_link`, `staleness_score`) を Why text に直結する v8 narrative がまだ deterministic template として確立していない。

## Decision

Review bucket の意味論を **epistemic-change-driven** に固定し、Why narrative を v8 にアップグレードする。dismiss は visibility=hidden 化のみ担い、Review には登場しない。

### Review bucket driver

projector の Review reason 判定 (`enricher.go` の Review bucket 配置ロジック) を以下に書き換え:

| reason | 入力 | bucket |
|---|---|---|
| stale evidence | `staleness_score >= 2` | Review |
| contradiction / drift | `contradiction_count > 0` または `version_drift_count > 0` | Changed 優先、Review には載せない |
| version_drift only (Changed 卒業後の置き場) | `version_drift_count > 0` かつ Changed で既に提示済 | Review |
| unfinished augur thread | `has_augur_link` かつ thread 未完了 | Review |
| user-dismissed (旧 v1 fallback) | `HomeItemDismissed` | visibility=hidden、Review に出さない |

「dismiss は Review 候補から外す signal」セマンティクスに reframe する。

### Why narrative v8

`WhyMappingVersion` を v7 → v8 へ bump。`why_override.go` で SurfaceScoreInputs に応じた `WhyKind` を優先順 `change > unfinished_continue > topic_affinity > tag_trending > recall > source` で決定。narrative テンプレ:

| signal | WhyKind | text (deterministic) |
|---|---|---|
| `version_drift_count > 0` | change | `Summary or tags changed since you last saw this.` |
| `has_augur_link` | unfinished_continue | `Pick up the Augur thread you started.` |
| `topic_overlap_count > 0` | topic_affinity | `Connects to topics in your recent Recap.` |
| `tag_overlap_count > 0` | tag_trending | `Tags you follow are moving here.` |
| `staleness_score >= 2` | recall | `Worth a short refresh.` |

ADR-000908 で導入予定の `act_outcome=stale_save` / `deep_engagement` も narrative テンプレに加えられる (本 ADR で行を予約)。

### UI 拡張

`ReviewDock.svelte` の review card に **reason 別 guidance line** を追加 (canonical contract §3 「Changed は『何が変わったか』だけでなく『理解をどう更新すべきか』を出す」延長):

| reason | guidance line |
|---|---|
| stale | `Last seen Xd ago. Worth a 30-second refresh?` |
| contradiction | `New summary changed key claims. Re-read?` |
| version_drift | `Tags moved. Check what's different?` |
| unfinished_thread | `Augur thread paused. Continue?` |

`review-reason-map.ts` の `dismissState + surfaceBucket` 由来の derivation は撤去し、`why_kind + surface_score_inputs` 由来に再配線。"前に dismiss した" 文言は撤去。

### Cutover 手順

1. `enricher.go` / `why_override.go` 修正 + WhyMappingVersion bump を 1 PR で landed
2. `docs/runbooks/knowledge-loop-reproject.md` の cutover ladder に v8 行を追加
3. PR merge + deploy 後、runbook 通り full reproject を 1 回実行 (`TRUNCATE knowledge_loop_entries` → projector run → checkpoint reset)
4. dashboard で Review bucket の reason 分布 (stale / contradiction / drift / unfinished) が出ること、`HomeItemDismissed` 由来 Review が 0 件であることを確認

### 検討した代替案

- **v1 fallback Review (dismiss-driven) を残す** — 構想 §4-Δ8 と矛盾。dismiss 起点を残すと UX 上 Review = "ゴミ箱" の印象が消えない。
- **LLM で Why narrative を生成** — Phase 3 完成計画 §「実装しないこと」と整合させ deterministic template のみ。
- **新 reason "low evidence density"** — future v2-only signal だが現時点では Phase 1 v2 planner の入力で揃っていないため、本 ADR では含めず後続で additive 追加。
- **`mark_reviewed` 後の hidden 化** — `archive` (visibility=hidden) と `mark_reviewed` (Review に残るが「確認済み」表示) の区別を維持。両者を混ぜない。

## Consequences

### Pros

- Review が思想と整合する (`認知状態を扱う` 構想 §1 が UI 体験に反映される)
- Why narrative v8 が v2 evidence と 1:1 で対応するため、ユーザは「なぜこれが出ているか」を verifiable claim として読める (構想 §4 trust contract に近づく)
- dismiss-driven の v1 fallback が消えることで、`HomeItemDismissed` event の意味が「Review から外す」に純化し、event の意味論が単純化される
- guidance line 追加で Review entry が「何のために再評価するのか」を明示的に示し、ユーザの行動を誘導しやすい (action-oriented UX)

### Cons / Tradeoffs

- 既存 Review にあった "前に dismiss した" entry が一斉に visibility=hidden に降格し、Review bucket の row 数が体感で減る。本 ADR は意図された挙動だが、ユーザに「Review が空になった」感を生む可能性。canonical contract §14 (空 Review は正常終端) と integrate して説明。
- WhyMappingVersion v8 bump は **full reproject** が必要。reproject 中は projector lag が一時的に立つ。runbook で実行時間帯を選ぶ。
- contradiction の判定は projector 側で LLM-free な近似 (例: `SummarySuperseded` count を proxy として使う) のため誤検知の余地がある。`contradiction_count` を debug log で観測し、誤検知 spike なら閾値調整。
- backend 側で `act_outcome` 値 (ADR-000908) が未稼働の間は recall narrative が一部空白。ADR-000908 と前後して landed することで埋まる。
