# ADR-000578: Knowledge Home Checkpoint停滞を修正しsovereign.yamlをcompose.yamlに統合する

## Status

Accepted

## Date

2026-03-25

## Affected Services

- alt-backend (Go) — projector, outbox worker, internal handler, health usecase
- knowledge-sovereign (Go) — 独立DB・RPCサービス

## Context

Knowledge Home Operations 管理画面の Checkpoint Seq が更新されていないように見える問題が報告された。調査の結果、以下の3つの問題が判明した。

### 問題 1: sovereign.yaml が compose.yaml に統合されていない

`compose/sovereign.yaml` は全サービスに `profiles: [sovereign]` が付与されており、`compose/compose.yaml` の `include:` にも含まれていなかった。alt-backend は `SOVEREIGN_URL=http://knowledge-sovereign:9500` が設定されているが、sovereign サービスの起動は手動の `-f` 指定に依存していた。

### 問題 2: 管理画面の LastUpdated がリクエスト時刻を表示

`knowledge_projection_health_usecase` の `GetHealth()` が `health.LastUpdated = time.Now()` としており、checkpoint の実際の `updated_at` ではなくリクエスト時刻を返していた。そのため projector のハートビートが動いているかどうかを管理画面から判断できなかった。

### 問題 3: 通常の記事作成フローで ArticleCreated イベントが欠落

`ArticleCreated` イベントは backfill ジョブ（`knowledge_backfill_job.go`）のみが発行しており、通常のRSS取得 → `fetch_article_usecase` → `SaveArticle` フローでは sovereign-db に書かれていなかった。ログ調査により、実際の記事作成は `CreateArticle` RPC ではなく `SaveArticle`（outbox 経由）が主流であることが判明した。

## Decision

### 1. sovereign.yaml を compose.yaml に正式統合する

- `compose/sovereign.yaml` から全サービスの `profiles: [sovereign]` を削除
- `compose/compose.yaml` の `include:` に `sovereign.yaml` を追加
- `compose/core.yaml` の alt-backend に `knowledge-sovereign: condition: service_healthy` の `depends_on` を追加

これにより `docker compose up` で sovereign が自動起動し、手動の `-f` 指定が不要になる。

### 2. 管理画面の LastUpdated を checkpoint の実際の更新時刻に修正する

既存の `GetProjectionFreshnessPort`（`today_digest_port` パッケージ）を `knowledge_projection_health_usecase` に注入し、checkpoint の `updated_at` を返すよう修正する。proto 変更は不要。freshness 取得に失敗した場合は `time.Now()` にフォールバックする。

### 3. outbox ワーカーで ArticleCreated イベントを発行する

`outbox_worker.go` の `ARTICLE_UPSERT` イベント処理時に、fire-and-forget で `ArticleCreated` ナレッジイベントを sovereign-db に発行する。sovereign-db の `dedupe_key`（`ON CONFLICT DO NOTHING`）により冪等性が保証されるため、同一記事の upsert が複数回発生しても重複イベントは作成されない。

併せて `connect/v2/internal/handler.go` の `CreateArticle` RPC にも同様のイベント発行を追加した（`WithKnowledgeEventPort` オプション）。これにより pre-processor RPC 経由の記事作成もカバーされる。

## Consequences

### Pros

- sovereign サービスが compose 標準スタックの一部となり、起動忘れによるサイレント障害がなくなる
- 管理画面の LastUpdated が projector の実際の動作状況を反映するようになり、運用判断が正確になる
- 新規記事が Knowledge Home に即座に反映される（backfill 不要）
- dedupe_key による冪等性が担保されており、既存の backfill フローとの共存も安全

### Cons / Tradeoffs

- outbox ワーカーは RAG upsert の成否に関係なく ArticleCreated を発行する。RAG が失敗しても Knowledge Home には記事が表示されるが、RAG 検索では見つからない状態が一時的に生じうる
- `SaveArticle` ドライバは `created` フラグを outbox ペイロードに含めていないため、upsert（既存記事更新）でも ArticleCreated イベントが試行される。dedupe_key で弾かれるため実害はないが、不要な RPC コールが発生する
