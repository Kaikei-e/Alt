# ADR-000831: Knowledge Home を Knowledge Loop 状態機械 UI として再定義し append-first / reproject-safe な新 read model を並走させる

## Status

Accepted

## Date

2026-04-23

## Affected Services

- knowledge-sovereign (Go) — Knowledge 系の storage と RPC を保持する single-source-of-truth service。Knowledge Loop のテーブル群 (knowledge_loop_entries / knowledge_loop_session_state / knowledge_loop_surfaces / knowledge_loop_transition_dedupes) と enum 6 個を `knowledge-sovereign/migrations/00011_create_knowledge_loop_tables.sql` で追加する。alt-db には Knowledge 系のテーブルを置かないというサービス境界を尊重する
- alt-backend (Go) — Knowledge Loop の projector / usecase / handler を置く compute 層。storage は knowledge-sovereign 経由でのみ read/write する
- alt-frontend-sv (SvelteKit) — `/loop` route + BFF
- proto — `alt.knowledge.loop.v1` package を追加

## Context

Knowledge Home は現在「情報の羅列」に留まっており、ユーザが情報を **観測 → 関連付け → 意思決定 → 行動** と循環させて理解を前進させる UI にはなっていない。Knowledge Home は append-first / versioned artifact / disposable projection という Alt のイミュータブルデータモデル原則で作られており、`knowledge_events` を append-only の真実源泉、`knowledge_home_items` / `today_digest_view` / `recall_candidate_view` を reproject 可能な read model として備えている。しかし現行 projection には `updated_at` が残り、UI 側も「どの entry が何の提案か」「ユーザは今どの局面か」「画面のどこに置くべきか」を 1 本の並び替えに押し込んでいるため、状態機械としては破綻している。

加えて、現行 UI は reduced motion 対応や depth 表現の contract を持たず、Apple spatial design 原則 (中央視野に主要 content、depth は tile にのみ、text は flat) とも整合していない。

[[knowledge-home-phase0-canonical-contract]] は Phase 0 で canonical な contract doc を起点にする規律 ([[000434]]) を既に確立しているため、本 ADR はそれを継承し **新しい contract doc を正典として派生させる** 形を取る。

## Decision

Knowledge Home を破壊せず、**Knowledge Loop という新しい read model と新しい RPC を並走させる** ことを採用する。具体的には以下を固定する。

1. **新 canonical contract** として [[knowledge-loop-canonical-contract]] を main に merge し、DB / proto / UI はこれを唯一の正典として派生する。
2. **3 軸の分離** — `proposedStage` (projection 決定) / `currentSessionStage` (session 保持) / `surfaceBucket` (UI 配置) を独立 axis として扱う。どれかひとつに押し込まない。
3. **`updated_at` の分解** — Knowledge Loop projection には `updated_at` を置かず、`projection_seq_hiwater` / `projection_revision` / `source_event_seq` / `freshness_at` / `source_observed_at` / `artifact_version_ref` に分解する。`projected_at` のみ wall-clock を持ち、**内部 debug 専用 / API・proto・view・log・metric 一切非露出**。
4. **Event-time purity** — Loop projection の business-fact タイムスタンプは必ず event の `occurred_at` から派生する。プロジェクタ内で `time.Now()` / `NOW()` を business fact に使うことを禁止する。
5. **append-first / reproject-safe** — Knowledge Loop は新 source of truth ではなく新 read model。`knowledge_events` への INSERT のみが状態変化の原子。projection は `TRUNCATE` + 再構築で再生可能。プロジェクタは event payload と versioned artifacts (`summary_versions` / `tag_set_versions` / `knowledge_lens_versions`) のみを読む。
6. **Idempotency 2-layer barrier** — `TransitionKnowledgeLoop` は UUIDv7 の `client_transition_id` 必須。server は `(user_id, client_transition_id)` で dedupe (TTL 48h の fast-path + `knowledge_events.dedupe_key` unique index の slow-path)。embedded timestamp が +5min future / -48h past を超えたら拒否。
7. **Dedupe table is ingest-only** — `knowledge_loop_transition_dedupes` は event log の upstream 側にある idempotency barrier であり projection ではない。full reproject は projection 3 テーブルのみ TRUNCATE + 再構築し、dedupe テーブルは触らない。循環依存を作らない。
8. **Single-emission rule** — `KnowledgeLoopActed` は `/loop` UI から、`HomeItemOpened` は `/feeds` UI から emit。同一ユーザ意図で両方を emit しない。legacy `HomeItem*` は Loop プロジェクタが read-side で消費して projection 効果を合成する (再 append しない)。
9. **Tenant/lens visibility** — 全 RPC handler は `user_id` / `tenant_id` を検証済み JWT claim からのみ取得し、request body の同名フィールドは無視。`lens_mode_id` は body 受信を許すが、`ResolveKnowledgeHomeLensPort.ListLensesForTenant(tenant_id)` に照合してから使う。[[000749]] 準拠、fail-closed。
10. **Why as first-class** — `WhyPayload` は `kind` enum + `text` (1..512 chars 平文) + `evidence_refs[]` (length ≤ 8) 構造化。自由 JSON 禁止。Phase-0 why codes (`new_unread` / `in_weekly_recap` / `pulse_need_to_know` / `tag_hotspot` / `recent_interest_match` / `related_to_recent_search` / `summary_completed`) → `WhyKind` (`source_why` / `pattern_why` / `recall_why` / `change_why`) の mapping table をプロジェクタが `WhyMappingVersion` 定数で持ち、変更時は bump + runbook 経由で full reproject。
11. **Spatial design contract** — depth は tile / container にのみ適用し、text は flat。Apple HIG (WWDC23 10072) の原則に従う。projection は具体 px ではなく `renderDepthHint: 1|2|3|4` enum を返し、view 層が Z offset / shadow / saturation / brightness にマップする。
12. **Reduced motion fallback** — `prefers-reduced-motion: reduce` で Z movement / parallax / animated blur / depth-of-field / scaling 中心の遷移を停止し、dissolve + highlight fade + color shift に置換する。
13. **Rollout** — `/loop` と `/feeds` を並行公開し、ユーザ自身が切り替える。feature flag は使わない。デフォルト遷移は既存 `/feeds` のまま維持して破壊を避ける。将来のデフォルト切替は別 ADR で決定する。
14. **Milestone 分割** — M0 (docs + ADR 本体) / M1 (DB migration + projector skeleton) / M2 (proto + handler) / M3 (projector 本実装) / M4 (BFF + SvelteKit) / M5 (3 plane UI + reduced motion) / M6 (observability + security review + ADR accepted) の 7 PR に分割して順次 merge。各 PR で main は常に green。

### 実装状況と follow-up (2026-04-23 時点)

- Migration は当初誤って `migrations-atlas/` に配置したが、正しい場所である `knowledge-sovereign/migrations/00011_create_knowledge_loop_tables.sql` に移動済み。
- alt-backend 側の `KnowledgeLoopRepository` は暫定的に `PgxIface` (alt-db pool) を受ける形で実装されているが、**正しい経路は knowledge-sovereign への Connect-RPC 呼び出し**。これは follow-up PR で:
  1. knowledge-sovereign に Knowledge Loop 用 RPC (UpsertKnowledgeLoopEntry 等) を追加
  2. alt-backend の driver を SovereignClient 経由に差し替え
  3. 暫定の pgx driver を削除
- 実 Run 検証 (migration 実適用 + projector 動作 + `/loop` エンドポイント疎通) は follow-up PR で行い、完了後にこの ADR を `accepted` 化する。

### 検討した代替案

- **既存 `knowledge_home_items` を schema 拡張**: `updated_at` を抜いて `freshness_at` 等を足す案。既存 RPC / projector / Svelte 経路を破壊するため却下。Phase A で別トラックとして棚卸してから検討。
- **`loop_stage` 単一フィールドで proposedStage / currentSessionStage / surfaceBucket を表現**: 状態機械として破綻する旨、レビューと Microsoft CQRS pattern の "clear separation" 原則から却下。
- **feature flag で段階 rollout**: ユーザが即時に両ルートを試せる方が学習が速いと判断し、flag なしの並行公開を採用 (ユーザ確認済み)。
- **`updated_at` を復活させて運用調査用にする**: 原則を壊すため却下。代わりに `projected_at` を **内部 debug 専用** として提供し、API / proto / public view / metrics / production log 一切非露出のハードニングを入れる。

## Consequences

### Pros

- **状態機械として一貫した UI** — 3 軸分離で entry 適性 / ユーザ現在地 / 画面配置が混線しない
- **イミュータブルデータモデル強化** — `updated_at` 依存を断ち切り、event-time purity で再投影結果が決定論的になる
- **reproject-safe 維持** — 既存原則を壊さず新 read model を追加するため、future の artifact 変更や mapping 変更に強い
- **security defense-in-depth** — JWT 由来の tenant/lens スコープ、UUIDv7 idempotency 2-layer barrier、dwell 発火の server-side rate limit、`projected_at` 非露出ハードニングが OWASP Top 10:2025 / ASVS 5.0 に整合
- **accessibility** — Reduced Motion / aria-description / 中央視野優先が Apple HIG と整合
- **OSS-safe** — 機微情報 (machine name / hardware specifics / IP / issue 番号) を contract 側に持ち込まない規律を M0 で固定

### Cons / Tradeoffs

- **コード複雑度増加** — Knowledge Home と Knowledge Loop の 2 projection / 2 RPC / 2 UI route が並走するため、backend の DI / wiring と frontend のナビゲーション面で認知負荷が上がる
- **event 種別増加** — `KnowledgeLoopObserved` / `Oriented` / `DecisionPresented` / `Acted` / `Returned` / `Deferred` / `SessionReset` / `LensModeSwitched` の 8 event で event log が大きくなる。dwell rate limit と dedupe で緩和
- **projection ストレージ倍増** — 4 テーブル × 行数。既存 `knowledge_home_items` に対して ~2x 程度。index 設計で compact に
- **mapping table メンテ** — Phase-0 why codes → WhyKind の mapping table は exhaustive でなければならず、Phase-0 code が増えるたびに projector 側に追記が必要 (unit test で pin)
- **将来のデフォルト切替コスト** — `/loop` がデフォルト化する場合は別 ADR + 別 migration path が要る
