# ADR-000607: 因果分析クエリの intent routing・retrieval gating・prompt contract・retry decomposition を導入し短答 fallback を根本解決する

## Status

Accepted

## Date

2026-03-28

## Affected Services

- rag-orchestrator (Go)

## Context

ベンチマーク「最近の石油危機の真因は？」で以下の failure chain が確認された。

1. **Intent misroute**: `最近` が temporal キーワードに先行マッチし、`真因` が無視される。結果、因果分析クエリが temporal strategy で処理され、recency 最適化が走る
2. **散漫な retrieval**: query expansion が `最近の石油危機の原因と背景` のような broad rewrite を生成し、Venezuela blockade / Iran airspace / live-blog など topic が散乱した context set が返る
3. **Quality gate 通過**: 従来の quality gate は top-3 平均スコアのみで判定しており、topic incoherence や causal query への厳格性がない
4. **短答生成**: Gemma 3 4B が散漫な context から短く overconfident な因果説明を生成
5. **Hard stop 発動**: `stream_short_answer_hard_stop` が発火するが、fallback reason が一律で原因の切り分けができない

RAG DB には十分なデータが存在する（32K documents, 978K chunks, 石油関連記事多数）。問題はデータ不足ではなく、**retrieval pipeline が因果分析クエリに対して最適化されていない** ことにある。

### Web Research に基づく設計判断

- **Azure RAG Information Retrieval Guidance**: query augmentation は元の query の性質を変えてはならない。必要に応じて decomposed subqueries を使うべき。`最近の石油危機の真因は？` を temporal 寄りに rewrite する挙動はこの指針に反する
- **Azure RAG Evaluators**: RAG 品質は Retrieval / Groundedness / Relevance / Response Completeness を分けて評価すべき。retrieval quality が悪い状態で satisfactory answer は期待できない
- **CausalRAG (ACL Findings 2025)**: causal graph を retrieval に統合し因果関係を保持した precision と interpretability を改善。full causal graph は GPU 制約で採用不可だが、causal query を別 intent として分離し prompt contract で因果構造を強制する方向性は一致
- **TARG (arXiv 2511.09803)**: adaptive retrieval gating は fixed threshold ではなく intent-aware に calibration すべき

## Decision

4 つの修正を段階的に導入し、因果分析クエリの failure chain を根本から解消する。

### 1. Causal Intent Classification

`IntentCausalExplanation` を新設し、`matchesCausal()` を `matchesTemporal()` より前に配置する。

分類優先度:
```
Comparison → CausalExplanation → Temporal → FactCheck → DeepDive → General
```

Causal キーワード: `真因`, `原因`, `要因`, `なぜ`, `理由`, `根源`, `root cause`, `why did`, `reason behind`, `caused by`, `what caused`

temporal + causal の共起（`最近` + `真因`）では causal が勝つ。これにより Azure guidance の「query の性質を変えない augmentation」に準拠する。

### 2. Intent-Aware Retrieval Quality Gate

`AssessWithIntent(contexts, intentType)` を新設。`IntentCausalExplanation` の場合:
- topic incoherence → Insufficient（Marginal ではなく即 fail）
- Good + high score variance → Insufficient
- Marginal + incoherent → Insufficient
- **Marginal + coherent → Marginal を維持（retry を許可）**

最後の点が重要: marginal でも coherent な結果が出ている場合は retry で改善可能なため、即 fail にせず retry ループに委ねる。初期実装では marginal を無条件に insufficient にしていたが、ベンチマーク検証で false rejection が発生したため、coherence 条件付きの retry 許可に修正した。

### 3. Causal-Answer Prompt Contract

`IntentCausalExplanation` 用の prompt template を追加:
- **直接的要因**: 直近のトリガーとなった出来事
- **構造的背景**: 長期的な要因・制度的背景
- **不確実性**: 根拠が不十分な点、見解が分かれる点
- 単一の原因に帰結させず、複数の要因を分離して記述すること
- ソースが収束しない場合は「見解が分かれている」と明記すること

これにより Gemma 3 4B が散漫な context から短く overconfident な単一原因説明を生成する failure mode を防ぐ。

### 4. Causal Retry Decomposition（根本修正）

従来の retry は `queryExpander.ExpandQueryWithHistory` で 1 回だけ broad rewrite を試みていた。因果分析クエリでは元の broad query と同じパターンの散漫な結果を繰り返す傾向がある。

因果分析クエリの retry を **focused subquery decomposition** に変更:
- supply-side focus: `{元query} 供給 制裁 sanctions supply`
- geopolitical focus: `{元query} 地政学 geopolitical conflict`
- economic focus: `{元query} 経済 market price impact`

3 つの subquery で並列 retry し、`AssessWithIntent` で最もスコアの高い結果を採用する。Good が見つかった時点で残りの retry を skip する。

これは Azure RAG guidance の「decomposed subqueries」パターンと CausalRAG の「causal path retrieval」の軽量実装に相当する。

### 5. Fallback Category 細分化

hard-stop 時の fallback reason を intent + quality flags から選択:
- `FallbackShortUnderGrounded`: 短答 + 低 keyword coverage
- `FallbackMixedTopicRetrieval`: topic incoherence による fail
- `FallbackCausalInsufficient`: 因果分析クエリで根拠不足

因果分析クエリの fallback メッセージ: 「十分に一貫した根拠が取れなかったため、因果関係を断定できません。より具体的な質問をお試しください。」

## Consequences

### Pros

- **ベンチマーク PASS rate: 6/9 → 8/9**: 「最近の石油危機の真因は？」「中東危機の根源はなに？」が fallback (0 chars) → 1100+ chars の構造化回答を生成
- **因果構造の強制**: 回答が「直接的要因 → 構造的背景 → 不確実性」の 3+ 構造に分離され、単一原因への潰れ込みが防止される
- **temporal misroute の解消**: 「最近 + 真因」が causal_explanation にルーティングされ、recency 最適化ではなく因果分析最適化が走る
- **retry の質的改善**: broad rewrite ではなく focused subquery で retry するため、散漫な結果の繰り返しを回避
- **Fallback 原因の可視化**: 運用ログから retrieval failure / causal insufficiency / short under-grounded を切り分け可能

### Cons / Tradeoffs

- **Causal retry は 3 subquery を順次実行**: 最悪ケースで retrieval latency が 3 倍。ただし Good が見つかった時点で early exit する。ベンチマークでは 22-42 秒（acceptable range）
- **Causal subquery が固定文字列**: `供給 制裁`, `地政学`, `経済` は domain-agnostic ではない。石油危機以外の causal query（例: テクノロジー障害の原因）では不適切な subquery になる可能性。将来的に query からの自動 decomposition（LLM-based planning）への移行が必要
- **「背景」キーワードの除外**: `背景` は causal にも temporal にも使われるため、現在の `matchesCausal` からは除外。`原因` `要因` `なぜ` など明確な causal 語彙のみでマッチ
- **Marginal + coherent の retry 許可**: coherence 判定が title 単語共有に依存するため、語彙が異なるが意味的に coherent なケースで retry を逃す可能性（false negative は残る）
