# ADR-000489: Knowledge Home の service_quality 境界を hard_error と分離して再定義する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-backend (Go)
- alt-frontend-sv (SvelteKit)

## Context

Knowledge Home の `service_quality` は [[000485]] で `full / degraded / fallback` の3値に拡張されたが、`fallback` の意味が広すぎた。特に、projection の stale 判定まで `fallback` に含めると、UI ブロック仕様で意図している「部分表示は成立するが一部セクションを落としている状態」と、実際の応答意味論がずれる。

また、read model 取得が複数系統で同時に失敗した場合まで成功応答として返すと、フロントエンドは `fallback` か `degraded` として描画を継続してしまい、UI 仕様上の `hard_error` と整合しない。Knowledge Home の契約では、利用可能な部分応答が成立するケースと、ページ全体を成功として返してはいけないケースを明確に分ける必要がある。

今回の実装監査では、以下の整理が必要であることを確認した。

- stale projection は「鮮度低下」であり、セクション欠落とは別の縮退である
- `items` と `digest` が同時に利用不能な場合は、成功応答ではなく内部エラー経路に倒すべきである
- `fallback` は「片系 read section を落としても、まだ表示可能な部分応答がある」場合に限定したほうが、UI と運用の両方で扱いやすい

## Decision

Knowledge Home の `service_quality` 境界を以下のように再定義する。

- `full` を、主要 read source が揃って正常に解決できた状態として採用する
- `degraded` を、表示は継続できるが鮮度低下や一部補助情報の欠落がある状態として採用する
- `fallback` を、片方の read section を落としても利用可能な部分応答が成立する状態に限定して採用する
- `items` と `digest` の両方が利用不能な場合は `service_quality` 付き成功応答を返さず、handler のエラー経路で `hard_error` 相当として扱う

今回の実装では、具体的に以下を採用する。

- projection checkpoint の stale 超過は `fallback` ではなく `degraded` とする
- `items` 取得失敗かつ `digest` が利用可能な場合のみ、部分応答として `fallback` を返す
- `items` と `digest` がともに失敗した場合は usecase からエラーを返し、handler 側で内部エラー応答に倒す

代替案として「stale も `fallback` に含める」「複数 read 失敗でも空の成功応答を返す」も検討したが、前者は UI 文言と表示期待を曖昧にし、後者は `hard_error` の検出を遅らせるため採用しない。

## Consequences

### Pros

- `fallback` と `hard_error` の境界が明確になり、バックエンドとフロントエンドで同じ意味論を共有できる
- stale projection を `degraded` に戻すことで、鮮度低下と部分欠落を別の運用シグナルとして扱える
- 複数 read failure を成功応答に埋め込まないため、監視と障害調査の初動が単純になる

### Cons / Tradeoffs

- `fallback` の発生条件が狭くなり、以前より `hard_error` に分類されるケースが増える
- 今後 section の種類が増えた場合、どこまでを「部分応答成立」とみなすかを再度 ADR 化する余地がある
- 既存の `fallback` 前提テストや説明文は、この意味論に合わせて追随が必要になる
