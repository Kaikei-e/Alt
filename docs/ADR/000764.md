# ADR-000764: mq-hub に Hurl E2E スイートを追加し段階的全面展開の第 2 サービスとする

## Status

Accepted

## Date

2026-04-18

## Affected Services

- mq-hub (Go) — プロダクションコードは一切触らず、E2E スイート (15 Hurl ファイル) と staging compose 片肺 (redis-streams + mq-hub) と CI ワークフロー (path-gated) を追加する
- redis-streams — mq-hub 単独起動のための最小依存として `compose/compose.staging.yaml` に加える (ephemeral, `appendonly no`, volume なし)

## Context

[[000763]] で Hurl E2E フレームワークを Increment 1 として `search-indexer` のみ導入し、「他サービスは PR 単位で後追い移行する」と宣言した。第 2 サービスをどれにするかを決めないと、フレームワークは 1 サービスだけの特殊解に留まってしまう。

mq-hub を第 2 に選んだ理由:

1. **契約サーフェスが E2E 不在**。[[000591]] で Pact CDC は `mq-hub → search-indexer` / `tag-generator` の非同期 message pact をカバーしたが、mq-hub 自身が晒す同期 HTTP 面 (`/health`, `/metrics`, Connect-RPC `Publish/PublishBatch/CreateConsumerGroup/GetStreamInfo/HealthCheck/GenerateTagsForArticle`) は E2E レベルの回帰網に入っていなかった。Connect の response shape・HTTP status・error code・proto3-JSON 書式は Pact message pact では拾えない。
2. **依存が軽い**。mq-hub が必要とするのは Redis 8.4 Streams 1 つだけ。search-indexer が Meilisearch + stub-backend 2 依存だったのと同規模で済み、既存 staging stack に自然に追加できる。
3. **Hurl が駆動できる範囲が曖昧だった**。`search-indexer` の README は「Connect-RPC on `:9301` は HTTP/2 h2c なので Hurl 不可」と書いていたため、mq-hub のような Connect-RPC-only サービスも E2E 圏外に見えていた。実際には **Connect プロトコルは HTTP/1.1 + JSON codec を一等市民でサポート**しており、`POST /{FQSN}/{method}` + `Content-Type: application/json` + proto3-JSON body で Hurl からそのまま叩ける。これを第 2 サービスで明確にしておかないと、後続の Connect-RPC サービス (auth-hub, alt-butterfly-facade, 他) 全てが同じ誤解で E2E を諦めてしまう。

技術上の制約として実装中に固まった事項:

- `compose.staging.yaml` は [[000763]] の監査項目 #8 に従って `internal: true` network を使う。これは「ホスト側からの container 到達」と同時に **ホストポート publish を事実上無効化する** (`ports: - "19500:9500"` が `docker ps` 上では `9500/tcp` にしかならない)。結果として Hurl は staging network **の内側で**走らせるしかない。
- Hurl 7.1+ はセキュリティ上、`file,path;` 形式の body 読み込みに `--file-root` 明示を要求する。Hurl を container 化して走らせる場合、`--file-root` で repo root を指定する必要がある。
- proto3-JSON は `int64` / `uint64` / `fixed64` / `sfixed64` を JSON *文字列* として書く。mq-hub の `GetStreamInfoResponse.length` は `int64` なので `"1"` で返り、Hurl assertion は `isNumber` ではなく `isString + matches "^[1-9][0-9]*$"` で受ける必要がある。
- `mqhub_publish_total` のような CounterVec は `promauto.NewCounterVec` で registry 登録はされるが、**サンプルは最初の `WithLabelValues` 呼び出しまで出力されない**。`/metrics` テスト単体で実行するとこれらが body に現れず fail する。

## Decision

**mq-hub に対する Hurl E2E スイートを 14 シナリオ + readiness probe の計 15 ファイルで追加する。実行は staging network 内で Hurl container を動かし、CI は mq-hub 関連 path の変更時のみ起動する。**

### 採用する構成

1. **staging stack 追加**: `compose/compose.staging.yaml` に `redis-streams` (Redis 8.4 alpine, ephemeral, `appendonly no`) と `mq-hub` (GHCR image, `MAX_BATCH_SIZE=1000`, `STREAM_MAX_LEN=10000`) を足す。project 名 `alt-staging`、network `alt-staging` (internal)、host port 19500 は documentation 目的のみで実際には使わない。

2. **scenarios 構成** (`e2e/hurl/mq-hub/NN-<feature>.hurl` + `README.md` narrative):

   | # | Hurl 駆動対象 | 契約 |
   |---|---|---|
   | 00 | `/health` readiness (retry: 30, interval: 500ms) | 200 + `healthy=true` + `redis_status=connected` |
   | 01 | `/health` 完全形 | JSON shape (snake_case)・uptime_seconds 数値 |
   | 02 | `/metrics` (warm-up publish 先行) | `mqhub_publish_total` / `mqhub_publish_duration_seconds` / `mqhub_redis_connection_status` 系列の存在 |
   | 03 | Connect-RPC `HealthCheck` | camelCase proto3-JSON (`redisStatus`, `uptimeSeconds`) |
   | 04 | `Publish` happy | `success=true`, `messageId` が Redis Stream ID 書式 `^[0-9]+-[0-9]+$` |
   | 05 | `Publish` validation (empty `eventType`) | non-2xx Connect error, `message` に `event_type` を含む |
   | 06 | `Publish` with `event` 欠落 | HTTP 400 + `code="invalid_argument"` |
   | 07 | `PublishBatch` happy (3 件) | `successCount=3`, `messageIds` 3 件 |
   | 08 | `PublishBatch` invalid 1 件混在 | バッチ全体 fail (partial-success 経路は usecase にない) |
   | 09 | `PublishBatch` 1001 件 oversize | non-2xx + `"batch size exceeds"` |
   | 10 | `CreateConsumerGroup` fresh | `success=true` |
   | 11 | `CreateConsumerGroup` 同名再作成 | `success=true` (driver が BUSYGROUP を飲む) |
   | 12 | `GetStreamInfo` populated | `length` が数値文字列、`groups[*].name` に per-run ID 含む |
   | 13 | `GetStreamInfo` nonexistent | non-2xx (Redis "no such key" 伝播) |
   | 14 | `GenerateTagsForArticle` timeout (200ms) | non-2xx + `"timeout"` メッセージ、実経過 < 2s |

3. **Connect-over-HTTP/1.1+JSON の扱い**: Hurl は `POST {{base_url}}/services.mqhub.v1.MQHubService/<Method>` + `Content-Type: application/json` + `Connect-Protocol-Version: 1` + proto3-JSON body で駆動する。Connect handler (`connectrpc.com/connect`) が同一 mux で JSON/HTTP1.1 と gRPC/HTTP2 の両方を受け付ける仕様を利用する。README に「Hurl は HTTP/1.1+JSON で動く、HTTP/2 h2c や gRPC は対象外」と明記し、将来の Connect-RPC サービスがコピー可能にする。

4. **responsibility split と scope 明示**: README の "Out of scope" に以下を記載する:
   - 非同期 event 配信 (= consumer 側が受け取るか) は Pact message pact (`mq-hub/app/driver/contract/provider_test.go`) の担当
   - `GenerateTagsForArticle` happy path は tag-generator consumer or stub が必要で今回は除外
   - mTLS peer-identity middleware は main mux に wire されていないので対象外

5. **runner script**: `e2e/hurl/mq-hub/run.sh` が (a) 1001 件 batch fixture を `gen-batch-oversize.py` で生成、(b) `compose.staging.yaml` の `redis-streams + mq-hub` だけを `up -d --wait`、(c) Hurl container (`ghcr.io/orange-opensource/hurl:7.1.0`) を `--network alt-staging` で走らせる、(d) JUnit/HTML レポートを `e2e/reports/mq-hub-<run_id>/` に出す、(e) trap で tear-down (`KEEP_STACK=1` で保持)。

6. **CI path gating**: `.github/workflows/e2e-hurl.yml` を **1 ワークフロー / 複数 job** 構成で導入し、現時点では `mq-hub` job のみ。`on.pull_request.paths` / `on.push.paths` で以下に変更があった時だけ発火する:
   - `mq-hub/**`
   - `compose/compose.staging.yaml`
   - `e2e/hurl/mq-hub/**` / `e2e/fixtures/mq-hub/**` / `e2e/fixtures/staging-secrets/**`
   - `.github/workflows/e2e-hurl.yml` 自身

   後続サービスを追加する際は `paths:` を追記し、sibling job を追加するだけで拡張できる。

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| Hurl を host で走らせ `localhost:19500` を叩く | `internal: true` network は host port publish を事実上無効化する。search-indexer の README も同様に "swap service names for localhost:PORT" と暗黙のワークアラウンドを書いていたが、誰が書いても同じ地雷を踏むので container 化を正とする |
| 全 Connect-RPC メソッドの partial-failure 経路を検証 | usecase レベルに partial-success 経路は存在しない (gateway が全イベント validate してから Redis を叩く)。架空のパスを assert すると実装と乖離する |
| `GenerateTagsForArticle` の happy path 用に tag-generator stub を今回同梱 | Redis Streams 往復を擬似するだけなら可能だが、tag-generator 自体の CDC は別途存在する。Stub を書くこと自体がメンテ対象になるので、timeout 経路のみで必要十分と判断 |
| 単一 `.github/workflows/e2e-hurl-mq-hub.yml` を毎サービス作る | ワークフロー数が線形に増える。1 ファイル + path-gated matrix/job 構成のほうが、後続サービス追加時の review 差分が小さい |
| assertion を `isNumber` / 整数比較で書く | proto3-JSON の int64 は文字列化されるため常に fail する。"文字列だけど中身は数値" という proto3-JSON の挙動を README に残し、assertion は `isString` + 数値 regex で表現する |

### 範囲外 (今回やらないこと)

- search-indexer 以外の 49 サービスへの Hurl 展開 — 各サービス単位の PR で後追い実施。本 ADR はパターンを 2 サービス目で固める記録
- `GenerateTagsForArticle` happy path の coverage
- mTLS `peer-identity` middleware を main mux へ wire する決定 — middleware は実装されているが `main.go` の mux 登録対象外。将来 enforce する決定が来たら別 ADR
- Redis 障害モード (mq-hub が `healthy=false` を返すか) の coverage — stack 操作を伴うため run.sh の lifecycle と分けたい
- c2quay / Pact gate との関係整理 — 今回追加した CI は Pact gate の代替ではなく補完。`scripts/deploy.sh production` の Pact gate は従来通り走る

## Consequences

### Pros

- **Connect-RPC only サービスが Hurl E2E に乗せられることを実証した**。同じパターンで auth-hub (Connect-RPC)、alt-butterfly-facade、他の Connect-RPC サービスが PR 単位で追加可能
- **同期 HTTP / Prometheus / Connect error code が回帰網に入る**。Pact message pact では検証不能な HTTP status・Connect code (`invalid_argument`)・error message 形式の regression を CI で弾ける
- **フレームワークの成長パターンが固まった**: `e2e/hurl/<service>/` + `e2e/fixtures/<service>/` + `compose.staging.yaml` への追記 + `e2e-hurl.yml` の path 追記 + sibling job。第 3 サービス以降の PR は review cost が低い
- **README に proto3-JSON 落とし穴 (int64 string, camelCase, 遅延 CounterVec) を記録した**。同じ罠を後続サービスで踏まない
- **Pact CDC / Hurl E2E の責務分担が明文化された**。テスト層の重複実装を避けられる

### Cons / Tradeoffs

- **staging stack が段々肥大化する**。現時点で search-indexer 系 + mq-hub 系の 5 container。全サービスが乗るころには compose.staging.yaml が単一ファイルでは見通しが悪くなる。サービス単位の分割 compose に切る時期を別 ADR で決める必要
- **Hurl container image を CI 側でも pull する**。`ghcr.io/orange-opensource/hurl:7.1.0` は tag 固定だが digest 固定ではない。[[000763]] の監査項目 #10 で触れた「Renovate が走ったら digest pin」は依然として運用フォロー項目
- **Connect handler の error body format に依存する assertion が入った**。connect-go が将来 `code` / `message` フィールド名を変える確率は低いが、メジャー upgrade 時は README と assertion を同時に見直す必要
- **`GenerateTagsForArticle` は timeout 経路のみ**。happy path が検証されないため、tag-generator とのインターオペが CDC 経由でしか保証されない。timeout 以外のリグレッションは CDC が落とす前提に立っている
- **mq-hub の production code には手を入れていない**ため、今回のスイートが落ちる時は「テスト / fixture / compose 側のミス」か「mq-hub の挙動が変わった」かの判別が必要。run.sh に `KEEP_STACK=1` を用意して人間によるトリアージを容易にしたが、初見の人が迷うリスクは残る
