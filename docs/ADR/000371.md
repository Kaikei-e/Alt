# altctl init による新規環境の自動ブートストラップ

## ADR's STATUS

Accepted

## CONTEXT

Alt プラットフォームを新しいマシンで環境構築するには以下の手動作業が必要だった:

1. `.env.example` を `.env` にコピーし、値を調整
2. `secrets/` ディレクトリに 22 個のシークレットファイルを手動生成
3. Atlas マイグレーションチェックサム（`atlas.sum`）の再生成
4. PostgreSQL の初期化スクリプトが参照するロール（`alt_db_user`, `alt_appuser` 等）の手動作成
5. PgBouncer・マイグレーションコンテナのパスワード設定の整合性確認

OSS としての素早い起動（`git clone && altctl init && altctl up`）を実現するため、これらの手順を自動化する `altctl init` コマンドを新設した。

### 新規環境で発覚した問題群

新規マシンでの初回起動時に以下の問題が連鎖的に発覚した:

| # | 問題 | 根本原因 |
|---|------|---------|
| 1 | PgBouncer が `DATABASE_URL` のパース失敗 | base64 パスワードに `/`, `+`, `=` が含まれ URL 構文を破壊 |
| 2 | マイグレーション時に `alt_user` のパスワード認証失敗 | `POSTGRES_USER == DB_USER` のとき init スクリプトがパスワードを上書き |
| 3 | `ALTER TABLE article_tags ADD PRIMARY KEY` 失敗 | CREATE TABLE で既に PK 定義済みだが、後のマイグレーションが再追加 |
| 4 | `ADD CONSTRAINT fk_article_summaries_article_id` 失敗 | 同上パターン — FK が初期テーブルと後のマイグレーション両方に存在 |
| 5 | Kratos の `cipher secret` 長さ制限エラー | base64 の 43 文字が Kratos の 32 文字上限を超過 |
| 6 | `role "alt_db_user" does not exist` | マイグレーションの `OWNER TO` / `GRANT` が参照するレガシーロール未作成 |
| 7 | auth-hub の healthcheck がサーバーを再起動 | Docker イメージが古く `healthcheck` サブコマンド未実装 |
| 8 | alt-frontend-sv の healthcheck 失敗 | コンテナに `wget` なし + `/sv/health` が認証リダイレクト返却 |
| 9 | alt-backend が PgBouncer 経由で認証失敗 | PgBouncer と alt-backend が異なるパスワードファイルを参照 |
| 10 | tag-generator が `SERVICE_SECRET` 未設定で起動失敗 | compose の env var 名（`SERVICE_TOKEN_FILE`）とアプリの期待（`SERVICE_SECRET_FILE`）が不一致 |

## DECISION MAKING

### 1. `altctl init` コマンドの新設

6 フェーズで環境を自動構築する:

```
Phase 1: Prerequisites     — Docker CLI / Compose V2 / Daemon の存在確認
Phase 2: Environment File  — .env.example → .env コピー（冪等）
Phase 3: Secrets           — secrets/ に 22 ファイルを crypto/rand で自動生成
Phase 4: Migration Checksums — atlas.sum を Docker 経由で再生成（3 DB 分）
Phase 5: Validation        — 全必須ファイルの存在確認
Phase 6: Next Steps        — altctl up への案内表示
```

フラグ: `--force`（上書き）, `--skip-secrets`（外部管理向け）, `--dry-run`

### 2. URL-safe パスワード生成

`base64.RawURLEncoding` を採用し、`/`, `+`, `=` を排除。`DATABASE_URL` やその他の接続文字列に安全に埋め込める。

```go
// Before: base64.StdEncoding — +, /, = を含む（URL 非安全）
// After:  base64.RawURLEncoding — URL 安全文字のみ
base64.RawURLEncoding.EncodeToString(buf)
```

Kratos cipher secret は `Truncate: 32` で 32 文字に切り詰め（Kratos の上限制約）。

### 3. パスワード統一: `postgres_password` を正とする

`POSTGRES_USER == DB_USER == alt_user` の現構成では、PostgreSQL が `POSTGRES_PASSWORD_FILE` で設定したパスワードが正。PgBouncer・alt-backend・マイグレーションコンテナすべてが `postgres_password` シークレットを参照するように統一した。

| コンポーネント | Before | After |
|--------------|--------|-------|
| PgBouncer | `db_password` | `postgres_password` |
| alt-backend | `db_password` | `postgres_password` |
| migrate | `postgres_password` | `postgres_password`（変更なし）|

### 4. DB 初期化スクリプトの改修

**`01-create-users.sh`:**
- `DB_USER == POSTGRES_USER` の場合、パスワード上書きをスキップ（二重設定防止）
- `alt_db_user`, `alt_appuser` を NOLOGIN ロールとして作成（マイグレーションの GRANT/OWNER 用）
- `create_or_update_role()` がパスワード空でも NOLOGIN ロールを作成可能に

**`02-seed-sample-data.sql`:**
- `DO $$` ブロックで `feeds` テーブルの存在をチェックし、未存在時はスキップ（マイグレーション前に実行されるため）

### 5. マイグレーション冪等性の確保

以下のマイグレーションに `IF NOT EXISTS` ガードを追加:

| ファイル | 修正内容 |
|---------|---------|
| `20260102000000_add_article_tags_primary_key.sql` | `ADD PRIMARY KEY` → `DO $$ IF NOT EXISTS` |
| `20260108010000_add_article_summaries_foreign_key.sql` | `ADD CONSTRAINT` → `DO $$ IF NOT EXISTS` |
| `20260211000001_add_feed_link_id_to_feeds.sql` | `ADD CONSTRAINT` + `CREATE INDEX` → `IF NOT EXISTS` |
| `20240101003900_create_api_usage_tracking_table.sql` | `ADD CONSTRAINT` → `DO $$ IF NOT EXISTS` |

### 6. Healthcheck の修正

| サービス | Before | After | 理由 |
|---------|--------|-------|------|
| alt-frontend-sv | `wget` | `bun` TCP 接続チェック | コンテナに wget なし + `/sv/health` が 303 返却 |
| auth-hub | イメージ古い | `--build` で再ビルド | `healthcheck` サブコマンドが未実装だった |

### 7. tag-generator の環境変数修正

`SERVICE_SECRET_FILE` と `DB_TAG_GENERATOR_PASSWORD_FILE` を compose に追加。DB 接続は PgBouncer 経由に変更。

## RESULTS, EFFECTS

### PROS

- **ワンコマンドブートストラップ**: `altctl init && altctl up` で新規環境が完全に立ち上がる
- **冪等性**: `altctl init` は何度実行しても安全（既存ファイルはスキップ、`--force` で上書き）
- **セキュリティ**: パスワードは `crypto/rand` + URL-safe base64、レガシーロール（`alt_db_user`, `alt_appuser`）は NOLOGIN で作成
- **マイグレーション堅牢性**: クリーン環境でも既存 DB からの移行でも、冪等なマイグレーションが成功する
- **既存環境との互換性維持**: 稼働中の本番環境には影響なし

### CONS, TRADEOFF

- **`db_password` が事実上未使用に**: `postgres_password` に統一したため、`db_password.txt` は生成されるが使われない。将来 `POSTGRES_USER != DB_USER` の構成に変更する場合は分離が必要
- **マイグレーションの `DO $$ IF NOT EXISTS` パターン**: Atlas の純粋な DDL 追跡から逸脱するが、CREATE TABLE と後発 ALTER の不整合を安全に処理するために必要
- **tag-generator が PgBouncer 経由で `alt_user` として DB 接続**: 本来は専用ロール `tag_generator` を使うべきだが、PgBouncer の userlist に登録されていないため暫定的に `alt_user` を使用

### 起動フロー

```
$ git clone <repo> && cd Alt
$ altctl init          # .env + 22 secrets + atlas.sum 生成
$ altctl up            # DB → PgBouncer → Kratos → auth-hub → backend → frontend → nginx
```
