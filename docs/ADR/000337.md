# ArticlePrefetcher キャッシュエビクションによる OG 画像消失バグの修正（第2報）

## ADR's STATUS

Accepted

## CONTEXT

ADR-307〜310 で構築された OGP 画像即時表示システムにおいて、`/sv/feeds/swipe/visual-preview` で OG 画像が一度正常に表示された後、しばらくすると消えてフォールバック（グラデーション）に切り替わる現象が発生していた。

ADR-270 で `MAX_CACHE_SIZE` を 5→10 に引き上げて同種のバグを修正した経緯があるが、その後 visual-preview モードの `prefetchAhead` が 10 に引き上げられた際に `MAX_CACHE_SIZE` が据え置かれ、再び **prefetchAhead >= MAX_CACHE_SIZE** という矛盾が生じていた。

### バグの発生メカニズム

OGP 画像解決は以下の 3 層フォールバックチェーンで行われる（ADR-309 で文書化済み）:

1. `articlePrefetcher.ogImageCache`（`BatchPrefetchImages` RPC 経由で非同期取得したプロキシ URL）
2. `activeFeed.ogImageProxyUrl`（フィード API 応答に含まれる HMAC 署名付きプロキシ URL）
3. `initialOgImageUrl`（SSR フォールバック、最初のカードのみ）

バグのタイムライン:

1. カード N が表示中。OG 画像はキャッシュまたは `ogImageProxyUrl` から正常表示
2. `triggerPrefetch` がカード N+1〜N+10 のプリフェッチを開始（500ms 間隔）
3. 各 `prefetchContent` 完了時に `contentCache` にエントリ追加
4. 10 件のプリフェッチ完了後、`contentCache` が 11 エントリ（カード N + N+1〜N+10）に到達
5. `evictOldEntries()` が発火 → 最も古いエントリ = **カード N（表示中）** を削除
6. `ogImageCache` からもカード N のエントリが削除される
7. その後、別フィードのバッチプリフェッチ完了 → `seedCache` → `onOgImageFetched()` → `ogImageVersion++`
8. `currentOgImage` が再評価 → `getCachedOgImage()` が `null` を返す（エビクト済み）
9. `ogImageProxyUrl` が設定されているフィード（RSS XML に画像があるもの）は第 2 層で救済される
10. **`ogImageProxyUrl` が未設定のフィード（HTML OGP から取得したもの）は `null` → フォールバックに切り替わる**

### 副次的バグ: `imgError` がリセットされない

`VisualPreviewCard.svelte` の `imgError` 状態は `thumbnailUrl` が変わってもリセットされない問題があった:

1. `prefetchContent` が raw OG URL（プロキシ非経由）をキャッシュ
2. カードがこの URL で画像読み込みを試みる → CORS 等で失敗 → `imgError = true`
3. その後 `BatchPrefetchImages` でプロキシ URL が到着 → `thumbnailUrl` が更新
4. しかし `imgError` が `true` のまま → `{#if thumbnailUrl && !imgError}` が `false` → フォールバック表示のまま

## DECISION MAKING

### Fix 1: `MAX_CACHE_SIZE` を 10 → 30 に引き上げ

**ファイル**: `alt-frontend-sv/src/lib/utils/articlePrefetcher.ts`

```typescript
// Before
const MAX_CACHE_SIZE = 10;

// After
const MAX_CACHE_SIZE = 30;
```

30 = `prefetchAhead`(10) × 2 + 現在表示中のカード + スワイプ済みカードのバッファ。

ADR-310 でメモリ制御のために 10 に設定されたが、visual-preview の `prefetchAhead = 10` との競合を解消するために引き上げが必要。メモリ影響は微小（1 エントリ = 記事 HTML + URL 文字列数本、30 エントリでも数 MB 以内）。

### Fix 2: `thumbnailUrl` 変更時に `imgError` / `imgLoaded` をリセット

**ファイル**: `alt-frontend-sv/src/lib/components/mobile/feeds/swipe/VisualPreviewCard.svelte`

```typescript
// Reset error/loaded state when thumbnailUrl changes (e.g. raw URL → proxy URL)
$effect(() => {
    void thumbnailUrl;
    imgError = false;
    imgLoaded = false;
});
```

Svelte 5 の `$effect` で `thumbnailUrl` を依存として追跡し、変更時にエラー・ロード完了状態をリセットする。これにより raw URL での失敗後にプロキシ URL が到着した際、画像の再読み込みが正しく行われる。

### Fix 3: エビクション時の `onOgImageFetched` 通知（見送り）

`evictOldEntries` でキャッシュ削除時に UI へ通知する案を検討したが、Fix 1 で根本原因が解消されるため今回は見送りとした。

## RESULTS, EFFECTS

### PROS

- **OG 画像の安定表示**: `prefetchAhead = 10` のプリフェッチが完了しても表示中カードのキャッシュが保持され、画像が消えなくなる
- **raw URL → proxy URL 切替への対応**: `imgError` のリセットにより、プロキシ URL 到着後の画像表示が確実に行われる
- **ADR-270 の教訓の反映**: キャッシュサイズとプリフェッチ先読み数の関係性を再認識し、十分なマージンを確保
- **既存テスト非破壊**: `npx vitest run src/lib/utils/articlePrefetcher.test.ts` — 全 4 テスト（既存 3 + 新規 1）パス

### CONS, TRADEOFF

- **メモリ使用量の微増**: `MAX_CACHE_SIZE` が 3 倍になるが、1 エントリあたりのメモリフットプリントが小さいため実用上の影響は無視できる
- **`$effect` の初回実行**: `thumbnailUrl` の `$effect` はマウント時にも実行されるが、`imgError` / `imgLoaded` の初期値が `false` であるため副作用なし
- **根本的な設計課題の先送り**: `MAX_CACHE_SIZE` と `prefetchAhead` が独立した定数として管理されており、将来 `prefetchAhead` が変更された場合に再び乖離する可能性がある。定数間の関係を明示的にコードで表現する（例: `MAX_CACHE_SIZE = prefetchAhead * 3`）ことが望ましいが、`prefetchAhead` が `SwipeFeedScreen.svelte` 側のコンポーネントローカル変数であるため、現時点では静的な値として設定
