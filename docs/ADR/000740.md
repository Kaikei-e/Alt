# ADR-000740: Pact Broker を恒常化し can-i-deploy を docker publish ブロックに組み込む

## Status

Accepted

## Date

2026-04-15

## Affected Services

- pact-broker / pact-db (`compose/pact.yaml` を `default` profile 相当に昇格)
- `compose/compose.yaml` — include 追加
- `compose/backup.yaml` — `pact_db_data` volume と `pact-db` の `pg_dump` を追加
- `compose/base.yaml` — `pact_broker_basic_auth_password` secret 宣言
- `.github/workflows/proto-contract.yaml` — search-indexer / mq-hub provider 検証 step 追加、必須 status check ガイダンス
- `.github/workflows/release-gate.yaml` (新規) — can-i-deploy + record-deployment
- `.github/workflows/deploy.yaml` — `needs: release-gate` で block
- `scripts/pact-check.sh` — secret-based broker 認証、FFI 検出強化

## Context

[[000591]] で Pact Broker は `profiles: [pact]` の opt-in として compose に追加された。`./scripts/pact-check.sh --broker` 経由の ad-hoc 起動しか想定しておらず、以下の運用ギャップが残っていた:

1. **永続 Broker が存在しない** → `can-i-deploy` を本番 gate として組めない (記録が残らない)
2. **basic auth が `pact:pact` のハードコード** → リポジトリが public な段階で攻撃面になる
3. **`PACT_BROKER_ALLOW_PUBLIC_READ=true`** → 未認証 read が通ってしまう
4. **バックアップ対象に入っていない** → Broker が失われたら pacticipant 履歴・can-i-deploy 状態が復元不能
5. **CI が `pact-check.sh` ではなく個別 `go test` を matrix 実行** → ローカルの pact-check.sh と CI で設定乖離が起きる余地
6. **docker publish と deploy に gate が無い** → `can-i-deploy` 失敗でもイメージが push される

## Decision

### E1: Pact Broker を default profile に昇格し認証を secret 化

`compose/pact.yaml` から `profiles: [pact]` を削除し、`compose/compose.yaml` の include に追加。以下の設定変更を適用:

- `pact_broker_basic_auth_password` secret を `compose/base.yaml` に宣言。ファイルモードは 0644 (ruby UID 100 が読めるように)
- entrypoint を書き換え、起動時に `PACT_BROKER_DATABASE_URL` と `PACT_BROKER_BASIC_AUTH_PASSWORD` を secret から展開してから `sh ./entrypoint.sh` を exec。`cd /pact_broker` が必要な点は pact-broker 公式 image の実装依存
- `PACT_BROKER_ALLOW_PUBLIC_READ=false` を明示化
- `compose/backup.yaml` の volumes に `pact_db_data` を追加、`scripts/backup/backup-all.sh` の PostgreSQL databases 配列に `pact-db:alt-pact-db-1:pact:pact` を追加

`./scripts/pact-check.sh` は broker password を `secrets/pact_broker_basic_auth_password.txt` から自動読込し、broker mode heartbeat チェックも認証つきに変更。FFI 検出は `$HOME/.pact/lib` に加えて `/usr/local/lib` も探索し、環境差での `Skipped` を解消。

### E2: CI を blocking gate に昇格

`.github/workflows/proto-contract.yaml` の `pact-publish-and-verify` job に 2 step 追加:

- `Run search-indexer provider verification` (`TestVerifySearchIndexerProviderContracts`)
- `Run mq-hub provider verification` (`TestVerifySearchIndexerMqHubMessagePact`)

ファイル冒頭コメントで「このジョブ名は branch protection の required status check に登録すること」を明示し、`docs/runbooks/pact-broker-ops.md` に GitHub Settings の手順を記載。

### E3: `can-i-deploy` / `record-deployment` を deploy 前段に固定配置

`.github/workflows/release-gate.yaml` を新設:

- `workflow_call` + `workflow_dispatch` 両対応
- `self-hosted` runner 上で `pact-broker can-i-deploy --to-environment production` を 14 pacticipant に対して個別実行
- 1 件でも fail なら exit 1 で後段 job を block
- 成功時は `pact-broker record-deployment --environment production` を実行し Broker を本番真実源にする

`.github/workflows/deploy.yaml` に `release-gate` job を `workflow_call` で前段に挿入し、`deploy` job は `needs: release-gate` で待機。`PACT_BROKER_PASSWORD` は GitHub repo secret で注入。

### 検討した代替案

- **案 A: PactFlow managed に移行** — 運用ゼロだが外部依存が増え、mTLS 化の進む東西通信の外に pact 情報が出る。採用せず。
- **案 B: `pact-check.sh` を CI 単独に置き換え、個別 matrix を撤去** — 並列性が落ちて CI 時間が 7 → 15 分になる見込み。既存 matrix を残しつつ `pact-check.sh` をローカル整合の source of truth として保つ折衷を採用。
- **案 C: deploy gate を Kubernetes admission controller 相当で行う** — 本プロジェクトは Docker Compose 単一マシン前提 (memory: `feedback_single_machine_architecture`) のため不要。self-hosted runner で localhost の Broker を叩くのが最小コスト。

## Consequences

### Pros

- Broker が永続化され、`can-i-deploy` / `record-deployment` の履歴が本番真実源として機能する
- docker publish + deploy の gate 二段階でコントラクト破壊が fail-fast する。`pact-check.sh` が PR で落ちる (E2) + `release-gate` が deploy 前に落ちる (E3) の二重防御
- Broker の basic auth が Docker secret 経由になり、リポジトリが public でも流出リスク無し。`PACT_BROKER_ALLOW_PUBLIC_READ=false` で未認証 read も遮断
- `pact-db` が Restic バックアップに入ったため、Broker 失っても 3-2-1 で復旧可能

### Cons / Tradeoffs

- pact-broker image は `/pact_broker/entrypoint.sh` を探す前提で、カスタム entrypoint ラッパが image の実装詳細に結合する。将来 image が `docker-entrypoint.sh` 形式に戻った場合は修正が必要
- `release-gate.yaml` が self-hosted runner 依存 (localhost:9292 にアクセスするため)。hosted runner 移行には Broker を外部公開するか PactFlow 移行が前提
- branch protection の required status check 配線は GitHub UI 操作で別途必要。YAML 内コメントで明示しているが、repo settings と YAML が一致しているか運用で定期監査が要る

### 追加ハードニング (2025-2026 Pact ベストプラクティス反映)

公式ドキュメント (docs.pact.io / pactflow.io) と 2024-2025 リリース blog の調査を踏まえ、Broker / verifier に以下を恒久設定として追加:

- **DB auto-clean** (Broker image 2.72.0.0+) — `PACT_BROKER_DATABASE_CLEAN_ENABLED=true` + nightly cron。`keep_version_selectors` で `main` / `deployed=true` / `released=true` / `maxAgeInDays=30` を残し、それ以外を毎晩 prune。PR ブランチ起点の DB 肥大化を抑制
- **自動ブランチ削除** — `PACT_BROKER_AUTO_DELETE_BRANCHES_ENABLED=true` + `OLDER_THAN_DAYS=14`。短命 PR ブランチが broker UI / matrix を汚さない
- **Webhook scheme allowlist** — `PACT_BROKER_WEBHOOK_SCHEME_WHITELIST=https`。webhook callback の HTTP 平文経路を遮断
- **Pending pacts on provider verifier** — pact-go v2 `VerifyRequest.EnablePending=true` を search-indexer / alt-backend で有効化。新規 consumer pact は warning 扱いとなり、未検証段階で provider build を壊さない
- **WIP pacts hook** — `PACT_INCLUDE_WIP_SINCE` env (RFC3339) を渡すと `IncludeWIPPactsSince` が有効化し、新規 in-flight contract を自動取り込み
- **Branch versions API への移行** — `scripts/pact-check.sh` を legacy `/tags/{branch}` から `/branches/{branch}` に切替 (matrix-aware)

これらは全て **環境変数で no-op 化可能** にしてある (`PACT_DISABLE_PENDING=true` 等)。ローカル file-mode 実行への影響なし。
