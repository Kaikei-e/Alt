# ADR-000622: altctl を現在のソースコードに追従させ Knowledge Sovereign CLI・Admin RPC コマンドを実装する

## Status

Accepted

## Date

2026-04-05

## Affected Services

- altctl (Go CLI)

## Context

altctl は Alt プラットフォームの Docker Compose スタック管理 CLI ツールである。ADR-000532〜000592 の期間に以下の大規模な変更が行われたが、altctl はこれらに追従していなかった。

### 1. knowledge-sovereign の独立サービス化（ADR-000537, 000578）

[[000537]] で knowledge-sovereign が独立サービスとして抽出され、[[000578]] で `compose/sovereign.yaml` が `compose/compose.yaml` に正式統合された。しかし altctl の stack registry には `sovereign` スタックが存在せず、`TestNoOrphanComposeFiles` テストが FAIL していた。

### 2. Projection Snapshot・Retention・Storage API の実装（ADR-000585, 000586）

[[000585]] で knowledge-sovereign に snapshot 作成 API が、[[000586]] で retention ポリシーと storage 統計 API がメトリクスポート（:9501）に実装された。[[000592]] で Admin UI が追加されたが、CLI からの操作手段がなかった。

### 3. Proto 定義済みだが CLI 未実装の Admin RPC

`knowledge_home_admin.proto` に `TriggerBackfill`、`PauseBackfill`、`ResumeBackfill`、`GetBackfillStatus`、`GetProjectionHealth`、`GetFeatureFlags`、`RunProjectionAudit` が定義されていたが、altctl からは `StartReproject`/`GetReprojectStatus`/`CompareReproject`/`SwapReproject`/`RollbackReproject`/`GetSLOStatus` のみが呼び出し可能だった。

### 4. 認証ヘッダーの不一致（バグ）

altctl の `adminclient` は `Service-Token` ヘッダーを送信していたが、alt-backend の `ServiceAuthInterceptor` は `X-Service-Token` を期待していた。他の全サービス（pre-processor、search-indexer、rag-orchestrator）は `X-Service-Token` を使用しており、altctl のみが不正なヘッダー名を使用していた。

### 5. Pact CDC CI パイプライン追加（ADR-000616）

[[000616]] で `compose/pact.yaml` が追加されたが、altctl の stack registry に `pact` スタックが存在せず、`TestNoOrphanComposeFiles` テストが FAIL していた。

### 6. core stack のサービスリスト陳腐化

`core` スタックの Services に `alt-frontend` が含まれていたが、`compose/core.yaml` には存在しない（`alt-frontend-sv` のみ）。`TestRegistryServicesMatchCompose` テストが FAIL していた。

## Decision

**altctl を現在のソースコードの状態に完全に追従させ、knowledge-sovereign の REST API と alt-backend の全 Admin RPC に対応する CLI コマンドを TDD で実装する。**

### 1. Stack Registry の修正

- `core` スタックの Services から `"alt-frontend"` を削除
- `core` スタックの DependsOn に `"sovereign"` を追加（ADR-000578: alt-backend が knowledge-sovereign に依存）
- `sovereign` スタック追加（`sovereign.yaml`: knowledge-sovereign-db, knowledge-sovereign-db-migrator, knowledge-sovereign）— `Optional: false`
- `pact` スタック追加（`pact.yaml`: pact-db, pact-broker）— `Optional: true, Profile: "pact"`

sovereign を `Optional: false` とした理由: alt-backend が `knowledge-sovereign: condition: service_healthy` で依存しており、sovereign が起動していなければ core が起動できない。

### 2. 認証ヘッダーバグの修正

`adminclient/client.go` の `Service-Token` → `X-Service-Token` に修正。alt-backend の `ServiceAuthInterceptor` および `service_auth_middleware.go` で定義されたヘッダー名と一致させる。

### 3. Sovereign REST Client の新設

knowledge-sovereign のメトリクスポート（:9501、ホストマッピング :9511）は認証なしの plain REST エンドポイントを提供する。既存の `adminclient`（Connect-RPC JSON エンコーディング）とは通信プロトコルが異なるため、新規パッケージ `internal/sovereignclient` を作成した。

| 項目 | adminclient | sovereignclient |
|------|-------------|-----------------|
| プロトコル | Connect-RPC JSON | plain REST |
| ターゲット | alt-backend :9001 | knowledge-sovereign :9511 |
| 認証 | X-Service-Token | なし（Docker ネットワーク内部） |
| メソッド | POST のみ（RPC） | GET + POST |

### 4. Knowledge Home CLI コマンドの拡充

既存の `altctl home` サブコマンド（`reproject`, `slo`）に加え、以下を追加:

**Sovereign REST 経由（sovereignclient）:**

| コマンド | API エンドポイント | 用途 |
|---------|-------------------|------|
| `home snapshot list` | `GET /admin/snapshots/list` | スナップショット一覧 |
| `home snapshot latest` | `GET /admin/snapshots/latest` | 最新スナップショット |
| `home snapshot create` | `POST /admin/snapshots/create` | スナップショット作成 |
| `home retention run [--live]` | `POST /admin/retention/run` | リテンション実行（デフォルト dry-run）|
| `home retention status` | `GET /admin/retention/status` | リテンションログ |
| `home retention eligible` | `GET /admin/retention/eligible` | アーカイブ対象パーティション |
| `home storage` | `GET /admin/storage/stats` | テーブルストレージ統計 |

**alt-backend Connect-RPC 経由（adminclient）:**

| コマンド | RPC メソッド | 用途 |
|---------|-------------|------|
| `home health` | `GetProjectionHealth` | Projection 健全性表示 |
| `home audit` | `RunProjectionAudit` | Projection 監査実行 |
| `home backfill trigger` | `TriggerBackfill` | Backfill ジョブ開始 |
| `home backfill status` | `GetBackfillStatus` | Backfill 進捗確認 |
| `home backfill pause` | `PauseBackfill` | Backfill 一時停止 |
| `home backfill resume` | `ResumeBackfill` | Backfill 再開 |
| `home flags` | `GetFeatureFlags` | フィーチャーフラグ表示 |

### 5. 共通ヘルパーの整理

`newAdminClient(cmd)` と `newSovereignClient(cmd)` を `home.go` に集約し、`addAdminFlags(cmd)` / `addSovereignFlags(cmd)` で共通フラグ登録を統一した。既存の `home_reproject.go` と `home_slo.go` の重複フラグ登録も解消した。

### 6. Migrate Volume Registry の更新

`knowledge_sovereign_db_data` を PostgreSQL バックアップ対象として追加。

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|--------|------|-----------|
| sovereign 操作を adminclient 経由で統一 | 認証モデルとプロトコルが異なる | sovereign メトリクスポートは plain REST・認証なし。Connect-RPC ラッパーは不要なオーバーヘッド |
| sovereign を Optional スタックにする | altctl up core が sovereign なしで起動可能に | alt-backend が depends_on で sovereign を要求しており、Optional にすると silent failure が発生 |
| adminclient に GET メソッドサポートを追加 | パッケージ統合 | Connect-RPC は POST のみの設計。GET サポートを追加すると責務が混在する |

## Consequences

### Pros

- **既存テストの全修正**: 3つの FAIL（`TestRegistryServicesMatchCompose`、`TestNoOrphanComposeFiles` x2）が全て PASS
- **認証バグの解消**: `X-Service-Token` ヘッダーにより alt-backend の ServiceAuth が正しく機能
- **Knowledge Home 運用の CLI 完全カバー**: Snapshot 管理、Retention 実行、Storage 監視、Projection 監査、Backfill 管理が CLI から操作可能に
- **全 9 パッケージ・133 テスト PASS**: 新規テスト 15 件を含む全テストが通過
- **sovereign スタックの自動起動**: `altctl up core` で sovereign が自動的に依存解決される
- **TDD 遵守**: 全新規コマンドでテスト先行実装（RED → GREEN → REFACTOR）

### Cons / Tradeoffs

- **sovereign メトリクスポートに認証がない**: Docker ネットワーク内部の前提。将来外部公開する場合は `sovereignclient` にサービストークン認証の追加が必要
- **新規パッケージ追加**: `internal/sovereignclient` が増えるが、adminclient との責務分離により各パッケージの凝集度は維持
- **sovereign REST レスポンスの JSON フィールド名**: knowledge-sovereign の Go struct には一部 `json` タグがなく PascalCase でシリアライズされる（ADR-000592 で frontend 側は正規化対応済み）。sovereignclient の struct タグで吸収
- **`home flags` コマンドは読み取り専用**: フィーチャーフラグの変更は CLI からは不可。設定変更は環境変数またはコンフィグ経由
