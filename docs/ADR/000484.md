# ADR-000484: Knowledge Home フロントエンドにインタラクション層（Ask・Listen・Save・検索・Toast）を追加する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-frontend-sv (TypeScript / SvelteKit)

## Context

[[000482]] で Knowledge Home UI の骨格（TodayBar・KnowledgeStream・QuickActionRow・WhyPanel 等）を実装したが、ユーザーが記事に対して能動的に働きかける手段は「Open」「Summarize」「Dismiss」に限られていた。Phase 計画では以下の機能が求められている:

1. **記事単位の質問（Ask）** — 記事のコンテキストを渡して Augur へ引き継ぐフロー
2. **音声キュー（Listen）** — サマリーの TTS 再生をキュー管理し、連続再生を可能にする
3. **保存（Save / Unsave）** — 後で読むためのブックマーク操作
4. **ストリーム内検索** — ロード済みアイテムをクライアントサイドでフィルタリングし、検索クエリを localStorage に履歴保存する
5. **操作フィードバック** — save / dismiss / listen 等のアクション結果をユーザーに即座に伝える通知機構

これらはすべてフロントエンド完結の UI 層変更であり、BFF / バックエンド API の追加は不要な段階である。

## Decision

Knowledge Home のインタラクション層を以下の 5 機能として一括実装する。

### 1. AskSheet コンポーネント（新規）

記事カードの「Ask」アクションで開くボトムシート。スコープタイトルとコンテキストを受け取り、ユーザーが自由入力またはサジェストから質問を選択して Augur（`/augur?q=...&context=...`）へ遷移する。現時点では Home 内回答 API は未接続であり、Augur へのブリッジとして機能する。

### 2. ListenQueueBar コンポーネント（新規）

画面下部に固定表示されるオーディオキューバー。`useTtsPlayback` フックと連携し、サマリーテキストの連続再生・一時停止・キュークリアを提供する。`+page.svelte` 側で `processQueue()` がキュー消化ループを管理し、再生失敗時は Toast で通知してループを中断する。

### 3. Save / Unsave アクション

QuickActionRow の「Summarize」ボタンを「Save」に置き換えた。`isSaved` 状態に応じてアイコン（Bookmark / BookmarkCheck）とラベルが切り替わる。`useKnowledgeHome` フックに `setSaved()` を追加し、楽観的に UI 状態を更新する。

### 4. UnifiedIntentBox の検索強化

- `onSearchSubmit` / `onSearchClear` コールバックを追加し、`+page.svelte` 側で `searchQuery` 状態を管理
- ロード済みアイテムの title / summaryExcerpt / tags / why をクライアントサイドでフィルタリング（`visibleItems`）
- 検索クエリを `localStorage` に最大 5 件保持し、Recent チップとして表示
- KnowledgeStream に `streamMode`（default / search / lens / recap_context）と `searchQuery` を伝搬し、ヘッダーにモード別ラベルを表示

### 5. Toast 通知ストア（新規）

`useToastStore()` を Svelte 5 runes ベースで実装。`push(message, kind, timeoutMs)` で通知を追加し、デフォルト 2 秒で自動消去する。Toast コンポーネントは画面右下に固定表示し、kind（success / error / info）に応じたカラースキームを適用する。

### 既存コンポーネントの改善

- **WhyPanel**: Why 理由をカテゴリ（Source / Behavior / Semantic / Change / Other）ごとにグループ化し、各理由に日本語の説明文を付与。WhySurfacedBadge によるフラット表示からカード型のカテゴリ分類表示に変更
- **KnowledgeCard**: 左ボーダーアクセント追加、sourceName 表示、「Explain why」トグルで WhyPanel を展開する UI を追加。why が空の場合は `new_unread` をフォールバック表示
- **TodayBar**: ローディング中の表現を「Loading...」からより具体的な「still being prepared」に変更。primaryTheme / topTags のテーマ行を追加
- **DegradedModeBanner**: インラインスタイルから Tailwind ユーティリティクラスに移行（amber/red の半透明パレット）
- **KnowledgeHomeEmpty**: `search_strict` / `degraded` の 2 つの EmptyReason を追加

### データ型の拡張

`knowledge_home.ts` の型定義に以下のフィールドを追加（現時点では BFF 未対応のためデフォルト値で初期化）:

- `KnowledgeHomeItemData`: `sourceName`, `summaryVersionId`, `tagVersionId`, `recapRefs`, `audioState`, `isSaved`, `isRead`, `isDismissed`
- `TodayDigestData`: `totalItems`, `unreadCount`, `primaryTheme`, `audioAvailable`

## Consequences

### Pros

- ユーザーが記事に対して取れるアクションが 3 種類（Open / Summarize / Dismiss）から 5 種類（Open / Save / Ask / Listen / Dismiss）に拡張され、Knowledge Home の実用性が大幅に向上する
- クライアントサイド検索により、既にロードされたアイテムを即座にフィルタリングでき、追加 API コールなしで探索性が向上する
- Toast 通知により、楽観的 UI 更新の結果がユーザーに明示的にフィードバックされる
- データ型にバックエンド未対応フィールドを先行定義することで、BFF 拡張時の型変更が最小限で済む

### Cons / Tradeoffs

- クライアントサイド検索はロード済みアイテムのみが対象であり、全件検索ではない。サーバーサイド検索（Meilisearch 連携）は別途実装が必要
- AskSheet は現時点で Augur への遷移ブリッジに留まり、Home 内でのインライン回答は未実装
- Save / Unsave の楽観的更新はバックエンドへの永続化が未接続であり、リロードで状態がリセットされる
- Listen キューの TTS 再生は `useTtsPlayback` に依存しており、ブラウザの Web Speech API サポート状況に制約される
- `isSaved` / `isRead` / `isDismissed` 等のフィールドはデフォルト値（false）で初期化されており、BFF が実際の値を返すまでは見かけ上すべて未保存・未読・未 dismiss となる
