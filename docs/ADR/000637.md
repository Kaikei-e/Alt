# ADR-000637: select フェーズの埋め込み処理をバッチ化し MiniBatchKMeans へ置き換えて OOM を回避する

## Status

Accepted

## Date

2026-04-07

## Affected Services

- recap-worker (Rust) — select フェーズの大ジャンル分割と外れ値除去の埋め込み・クラスタリング処理

## Context

3-day Recap の backlog 再開時に、`recap-worker` が `genre` ステージを通過した後、`select` フェーズの大ジャンル分割で繰り返し OOM していた。

調査で分かった問題は 2 つあった。

1. **埋め込み要求を 1 回でまとめて投げていた**
   大きいジャンルでは数百件から千件近い記事をまとめて `encode` しており、埋め込みサービスへの 1 回の要求と、その戻り値の保持が同時にメモリ圧力を作っていた。
2. **通常の batch KMeans をそのまま使っていた**
   まとめて得た埋め込みベクトルに対して、全件を前提にしたクラスタリングを同一プロセス内で実行していたため、`2048m` のコンテナ上限ではピークを抑えきれなかった。

実際に大きなジャンルでは `software_dev` が 689 件、`ai_data` が 222 件、`cybersecurity` が 195 件に達しており、巨大な入力を一括処理する実装は相性が悪かった。

この問題は単純にメモリ上限を上げるだけではなく、**入力を分割して処理し、クラスタリングも incremental に近い手法へ寄せる** 必要があった。

## Decision

**埋め込み要求は固定サイズのバッチに分割し、クラスタリングは `MiniBatchKMeans` を採用する。**

### 1. 埋め込み要求を固定バッチで送る

`subcluster_large_genres()` と `filter_outliers()` では、記事テキストを一括で `encode` するのではなく、32 件ずつのバッチに分割して埋め込む。

これにより、1 回の埋め込み要求で保持するテキスト数と戻り値のサイズを抑え、ピークメモリを下げる。

### 2. 大ジャンル分割に `MiniBatchKMeans` を採用する

`subcluster_large_genres()` と `subcluster_others()` のクラスタリングは、通常の `KMeans` ではなく `MiniBatchKMeans` を使う。

固定 `k` のクラスタ分割であり、かつ入力件数が多いという今回の条件では、mini-batch ベースの更新に寄せる方が、精度を大きく崩さずにメモリ圧を抑えやすい。

### 3. 外れ値除去も同じバッチ戦略に揃える

`filter_outliers()` も同じ埋め込みバッチ戦略を使い、select フェーズ内で一貫して大きな一括埋め込みを避ける。

### 4. 回帰テストでバッチ上限を固定する

埋め込みが一括送信に戻ると再び OOM の再発要因になるため、テストで単一 request の最大件数と分割処理を固定する。

## Consequences

### Pros

- `select` フェーズの大ジャンル分割で発生していた OOM の再発点を潰せる
- 埋め込み要求のピークサイズが下がり、`recap-worker` の 2 GiB 上限に収まりやすくなる
- `MiniBatchKMeans` によって、固定 `k` の分割を保ちながらメモリ圧を抑えられる
- 外れ値除去も同じバッチ戦略に揃うため、select フェーズ全体のメモリ特性が安定する
- 再起動後の backlog 再開で、`software_dev` などの大ジャンルを通過し、evidence 生成と dispatch まで進むことを確認できた

### Cons / Tradeoffs

- mini-batch 化により、通常の batch KMeans と完全に同一の割り当てにはならない
- 埋め込みを分割送信するため、総要求回数は増える
- バッチサイズを小さくしすぎるとスループットが落ちるため、メモリと速度のバランス調整が必要になる
- `MiniBatchKMeans` は実装上の近似を含むため、クラスタの境界が厳密な意味で安定するとは限らない
