# ADR-000914: Knowledge Loop の intent-driven same-stage trigger を proto vocabulary に格上げし allowSameStage opt-in を撤廃して COMPARE 400 を構造的に解消しつつ projector throughput を env tunable にする

## Status

Accepted

## Date

2026-05-24

## Affected Services

- proto (alt.knowledge.loop.v1)
- alt-backend (Go)
- knowledge-sovereign (Go)
- alt-frontend-sv (TypeScript / Svelte 5)

## Context

本番 `curionoah.com` で CHANGED bucket の **COMPARE** ボタンを押すと `POST /loop/transition` が **400 invalid_body** を返す事象が観測された (`s4Lu6OJk.js:1 POST https://<masked>/loop/transition 400`)。log-seeker 調査でコード経路を辿った結果、これは COMPARE 固有のバグではなく **intent-driven same-stage signal を proto trigger として first-class に表現していない** 構造欠陥であると判明:

- `proto/alt/knowledge/loop/v1/knowledge_loop.proto` の `TransitionTrigger` enum で `from_stage == to_stage` が許される trigger は **DEFER + Review-lane の 4 種** (`RECHECK` / `ARCHIVE` / `MARK_REVIEWED`) のみ
- FE `useKnowledgeLoop.svelte.ts::transitionTo` は `allowSameStage: true` という opt-in option で client-side `canTransition` の gate を緩めるが、**BFF `/loop/transition/+server.ts::SAME_STAGE_TRIGGERS` allowlist は同 opt-in を honour せず**、`user_tap` + same-stage POST を `parseBody` 内で `null` に落として 400 を返す
- FE の comment 自体が穴を予言している (`The BFF still enforces SAME_STAGE_TRIGGERS ...; this flag only relaxes the client gate.`)

同じ構造的欠陥が次の 2 callsite にも潜む:

1. [[000913]] §Cons L164 で deferred とした **"I got this" graduation (INTERNALIZE dismiss_state flip)** — 「次の wave で trigger 経路決定」と留保されていた
2. 未来の **Acolyte read-confirmed feedback / recap engagement signal** (Code-agent audit が予測) — 同パターンで same-stage intent recording を必要とする

canonical contract [[knowledge-loop-canonical-contract]] §7 は same-stage transition を **禁止列挙ベース** で書いており、`defer / recheck / archive / mark_reviewed` が「許可される same-stage」であることを暗黙のままにしていた。これが FE 側 leaky abstraction の根拠と化していた。

副次的に、ユーザ追加要件「REPROJECT が遅い」を本 wave に同梱した: `knowledge-sovereign/app/main.go` の Knowledge Loop projector は `BatchSize=100` を hardcode し、tick が 5 秒固定なので **理論上限が 20 events/sec**。[[000911]] で追加した `recomputeMacroState` が per-event DB query を撃つため reproject 中 O(N²) に近づく。1M events の reproject に 13.9 時間を要する状態だった。surface_planner_cron / act_outcome_cron は既に env-configurable な `BatchSize` + `MaxBatchesPerTick` を持っていたが、**loop projector だけ追従していなかった**。

## Decision

### Δ1. 3 trigger を proto に additive 追加 + classifier 拡張

`TRANSITION_TRIGGER_COMPARE` (=9) / `TRANSITION_TRIGGER_INTERNALIZE` (=10) / `TRANSITION_TRIGGER_INTENT_SIGNAL` (=11) を additive bump。3 種すべて `from_stage == to_stage` を強制し、canonical OODA stage の 4 種いずれかでなければ `ErrInvalidArgument` で reject。classifier の dispatch は:

| trigger | classifier output | dismiss_state effect | 用途 |
|---|---|---|---|
| `COMPARE` | `KnowledgeLoopActed` | (preserved) | CHANGED bucket compare CTA — `continue_context.recent_action_labels` に "compare" を記録 |
| `INTERNALIZE` | `KnowledgeLoopInternalized` | `active` → `internalized` | "I got this" graduation — [[000908]] §Δ3 の `DISMISS_STATE_INTERNALIZED` の producer を提供 |
| `INTENT_SIGNAL` | `KnowledgeLoopActed` | (preserved) | 汎用 payload-driven 信号 (将来の Acolyte / recap signal を見越した generic 入り口) |

設計原則は「trigger = stage gate decision の宣言、payload = enrichment」を厳格化: trigger を見れば「same-stage 許可か」「どの canonical event が出るか」「dismiss_state にどう影響するか」が決まり、payload で dispatch を変えない。これで [[000907]] の Review-lane 3 trigger 設計 (dismiss_state 遷移先が異なるため trigger で encode) を延長する形に揃う。

新 event `knowledge_loop.internalized.v1` は既存 transition payload schema を再利用 (proto message 追加なし)。projector の `projectInternalized` branch は既存 `PatchKnowledgeLoopEntryDismissState` driver を再利用し patch-only で `freshness_at` / `why_text` / `surface_bucket` / `continue_context` を保持。

### Δ2. allowSameStage opt-in を恒久撤廃

`useKnowledgeLoop.svelte.ts::transitionTo` の `options.allowSameStage` を削除。client-side `SAME_STAGE_TRIGGERS` const (7 trigger) を導入し、`sameStage && SAME_STAGE_TRIGGERS.has(trigger)` で許可判定。trigger 値自体が stage-gate decision を carry するため、leaky な opt-in flag の存在意義が消えた。同 const は BFF `/loop/transition/+server.ts` と Go classifier、proto `loop_transition_policy.json` の 4 経路でも mirror され、`transition_policy_test.go` (Go) と `loop-transitions.policy.test.ts` (TS) が divergence を CI で検出する。

### Δ3. canonical contract §7bis (vault addendum)

`docs/plan/knowledge-loop-canonical-contract.md` (vault-local、`docs/plan/` は repo gitignore 配下) に **§7bis Same-stage triggers (positive enumeration)** table を追加。7 trigger × semantic × classifier output × dismiss_state effect を明示し、「trigger = stage gate、payload = enrichment」原則と 4 mirror 先 (proto / backend / BFF / FE hook) を文書化。FE の leaky `allowSameStage` 説明 comment は本 addendum によって正規化された。

### Δ4. "I got this" CTA (Newspaper Style)

`LoopEntryTile.svelte` の expanded action row に "I got this" ボタンを追加 (`onInternalize` callback で gating)。

- 配置: 既存 dismiss / Open Recap / Ask の cta-row 末尾
- 書体: 既存 `.cta` style (font-mono、uppercase 相当、border 1px) を継承
- 配色: sepia ladder の light (`#8a6f47` border / `#5d4a2c` text) を rest、hover で `#5d4a2c` を fill に取り `--surface-bg` text に反転
- motion: color transition 160ms linear のみ。`@media (prefers-reduced-motion: reduce)` で transition を suppress
- a11y: `aria-label="Mark as internalized; remove from Loop"`、Enter / Space は `<button>` 標準動作で受理

### Δ5. Projector throughput tuning

`defaultBatchSize` を 100 → 500、`Config` に `MaxBatchesPerTick` (default 4) を追加。`RunBatch` を「1 tick で `MaxBatchesPerTick` 回まで連続 batch 消費、短 batch (= drain) または cap で break」semantic に拡張。`main.go` で `KNOWLEDGE_SOVEREIGN_LOOP_PROJECTOR_BATCH_SIZE` / `_MAX_BATCHES_PER_TICK` env を expose し、reproject 時は env を bump するだけで redeploy 不要に。

throughput: 20 events/sec (旧) → 2 000 events/sec (default、500 × 4 batches/5s tick) → env で linear scale。1M events の reproject が ~14 時間 → ~8 分 (default) / cap=20 なら ~2 分。

### 検討した代替案

- **Option A (個別 trigger のみ)** — COMPARE だけ追加し INTERNALIZE / INTENT_SIGNAL を後回し。短期的だが (a) [[000913]] deferred が残る、(b) 未来の Acolyte / recap signal で再度同パターン bug を踏む余地が残る。
- **Option B (FE-only mitigation)** — `onChangedCompare` を `from=OBSERVE → to=DECIDE` の真の遷移に変えて classifier の既存 `OBSERVE→DECIDE → Oriented` 経路に乗せる。1 ファイル 1 行修正で最短だが (a) FE comment が宣言する「同 stage signal」design intent と矛盾、(b) OODA stage を意図せず advance させてしまう副作用、(c) 他 2 callsite が残る。
- **Option C (採用) — 3 trigger + projector tuning** — 構造的に解決し、`allowSameStage` leaky abstraction も同時に撤廃、[[000913]] deferred を close。同 wave で reproject 高速化も解決。proto bump 1 回で 3 用途すべてに展開可能。

`buf breaking` は additive only のため pass。OS 公開 repo として old consumer は 3 trigger を `UNSPECIFIED` として扱うため backward compatible。

## Consequences

### Pros

- 本番 COMPARE 400 が **構造的に解決** (`onChangedCompare` の trigger 単一行差し替えが effect、しかも canonical な意味論で)
- [[000913]] §Cons の deferred INTERNALIZE 経路が close、`DISMISS_STATE_INTERNALIZED` enum に producer が提供された
- canonical contract §7bis の positive enumeration により「`allowSameStage` は何のためにあるか」「7 trigger は何を意味するか」が文書化、FE leaky abstraction の根拠が消えた
- TRANSITION_TRIGGER_INTENT_SIGNAL という generic な escape hatch が proto 化されたので、未来の Acolyte read-confirmed / recap engagement signal は proto bump 不要で wiring のみで導入可能
- projector throughput が 100× scale 可能になり、reproject runbook (WhyMappingVersion bump など) の所要時間が時間単位から分単位へ短縮
- "I got this" CTA で Loop の epistemic graduation 経路が UI に landed — MacroByline の `recent_internalized_count` が立ち上がり始める (本 ADR の後続作業で `BuildMacroState` を internalized event 経由でも算入)
- proto は additive のため `buf breaking` clean。alt-backend / alt-butterfly-facade / rag-orchestrator / alt-frontend-sv の generated stub は同期され、old consumer は新 trigger を `UNSPECIFIED` として透過 ignore

### Cons / Tradeoffs

- proto enum value が 9-11 と連番で進み、TransitionTrigger の reserved 領域が縮小。将来「intent-recording vs gating」をさらに細分化したくなった時に enum スペースの観点で再整理が必要となる余地は残る (現状 reserved 12-15 程度を確保しているが要監視)
- INTENT_SIGNAL が今のところ未使用 (将来 Acolyte / recap signal の入り口を proto 上で予約しただけ)。半年経って producer が landed しなかった場合は deprecate を検討すべきだが、proto enum は deprecate cost が高い (consumer 全部の handling が要更新)
- `MaxBatchesPerTick=4` の default は steady-state で問題ないが、env を未調整のまま reproject を走らせると default scale (2000 events/sec) でも 1M events に 8 分かかる。reproject 開始前に env=20 (10 000 events/sec) に bump する運用手順を `knowledge-loop-reproject` runbook に追記する必要 (後続作業)
- `recomputeMacroState` が per-event DB query を撃つ問題は本 ADR では未解決。throughput を 100× 上げたが per-event の DB round-trip 自体は減っていない。reproject 中 macro_state を batch-scope dedupe する設計 ([[plan/knowledge-loop-adr910-ux-...]] §projector performance の後続作業) を別 ADR で追加すべき
- macroState の `RecentInternalizedCount` は本 ADR landing 後も `act_outcome.v1.outcome=internalized` のみが寄与し、新 `KnowledgeLoopInternalized` event は **無視される** ([[000911]] の `BuildMacroState` 修正待ち)。後続 1 行修正で解決する想定だが、本 ADR 単体では UI 上の internalized 件数は変わらない
- canonical contract §7bis は vault-local (`docs/plan/` は gitignored) のため、公開 repo を見ている外部 reviewer はこの addendum を直接読めない。ADR-000914 (本文書) と [[knowledge-loop-canonical-contract]] wikilink を vault 内で辿る前提に依存する
