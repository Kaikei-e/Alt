# ADR-000938: relation-set を Contradiction ループ先行＋全種 emit で体験へ降ろし loop.v1 を正準 read に統合する

## Status

Accepted

本 ADR は実装前の decision record である。[[000937]] の方向（relation-set を projection の主軸とする）は維持しつつ、その最初の垂直スライスの **関係種選択を訂正** し、[[000937]] が open にしていた実装サブ決定（Relation の shape / honest SLI / migration）を確定させる。projection schema / proto / UI / SSR に触れる具体実装は別 PR で行い、その際に本 ADR の Decision を contract とする。

## Date

2026-06-03

## Affected Services

- knowledge-sovereign (Go)
- alt-backend (Go)
- alt-frontend-sv (TS/Svelte)

## Context

[[000937]] は「Knowledge Loop は 15 回以上再実装されても体験が『装飾された feed』のまま」「`decideBucketV2` が豊かな `SurfaceScoreInputs` を単一 bucket に畳んで関係を捨てている」と診断し、relation-set を主軸へ昇格する方向を決めた。その第 1 スライスとして Continuation 1 関係種を貫通させる commit 群が landed した。

grill-with-docs セッションで実機本番の HAR キャプチャと `/loop` の SSR `__data.json` を decode した結果、**第 1 スライスが [[000937]] 自身の診断した single-axis collapse を再発させている**ことが確定した。

- SSR payload (`GetKnowledgeLoop`) の **全 13 エントリが `relations=[]`（空）**。`now` 3 / `continue` 5 / `changed` 5 すべて空。
- うち `changed` 5 件は version-drift / contradiction の燃料を **現に持っている**（だから changed に置かれた。why も「Updated since you last saw it — a newer version is available」）。
- それでも relations が空なのは、`extractRelations`（`knowledge_loop_projector/relations.go:65-71`）が **Continuation 1 種しか emit せず**、resolver が `score_resolver_event_log.go` で算出済みの `TopicOverlapCount` / `TagOverlapCount`（クラスタ）と `VersionDriftCount` / `ContradictionCount`（矛盾）を **見もせず捨てている** から。畳む場所が `decideBucketV2` から `extractRelations` に一部屋ずれただけで、燃料は依然床に捨てられている。
- 選んだ Continuation は構成信号がすべて同一 entry_key の自己参照（`RecentContinueActionCount` / `HasOpenInteraction` / `HasAugurLink` / `QuestionContinuationScore`）であり、取り込んだばかりの `now` 記事では構造的に空になる。Orient（未知を既存の orientation に situ する面）の主入口を埋められない関係種だった。
- `proposed_stage` / `surface_bucket` / WhyPayload v2（`confidence_ladder` / `counter_evidence` / `what_would_change`）は populate されている。**空なのは relations 軸だけ** → 画面は bucket + why 一文 = 「装飾された feed」。
- [[000928]] / [[PM-2026-045]] が根絶したはずの silent fallback が新コードに再導入されている（`alt-backend handler.go:768` の decodeRelations、`relations.go:150` parseRelations / `:136` marshalRelations、`score_resolver_event_log.go:128` の lookup-error → 空 inputs）。fail-loud は resolver の nil ポインタ 1 ケースにしか適用されていない。
- `score_resolver.go:99` `plannerVersionForResolver` は resolver の型が non-Null なら無条件で `v2` を返し、実 evidence を見る `hasV2Signal()` は定義済みのまま未使用。honest SLI `relation_coverage_total` は追加されたが、嘘の `surface_bucket_assigned_total{version}` は撤去されておらず両者が並存している。

読み取り経路にも二重性がある。`/loop` ページの SSR は `GetKnowledgeLoop`（loop.v1）を叩くが、client は別途 `GetKnowledgeHome`（home.v1）を叩いており、後者は flat な items 重複に加え today_digest / recall_candidates / feature_flags を運ぶ。loop.v1 は 3 軸エントリ読み取りに特化し、home.v1 が digest / recall / lens CRUD / flags を独自所有する、という分裂が残っている。

したがって不足は「アーキの不在」ではなく、[[000937]] が言う通り「精巧な evidence が単軸に潰され、UI 上で方向づけの面にならず、計器がその劣化を隠している」ことであり、その第 1 スライスがまさにこの病を再生産していた。

## Decision

[[000937]] の relation-set 主軸という方向を維持し、その第 1 スライスを **「燃料を捨てない」「画面が読んでいる経路に正しく届ける」「劣化を deterministic に検知する」** 形へ訂正する。具体的に以下を採用する。

1. **`extractRelations` を全関係種 emit へ拡張する（coverage）。** resolver が既に算出している evidence を畳まず保持する: `TopicOverlapCount` / `TagOverlapCount` → **Cluster**、`VersionDriftCount` / `ContradictionCount` → **Contradiction**、continue 信号 → **Continuation**、augur / question 信号 → **Inquiry**。空集合は依然 NULL 列として正当。

2. **ループの閉じ（act→戻り diff）は Contradiction で先行証明する。** 燃料が production で証明済み（changed bucket が現に矛盾燃料を持つ）であり、戻り diff が最も明快なため。state ladder を **OPEN（未和解の新版が存在）→ ADVANCING（compare act 済みだが outcome 未確定）→ RESOLVED（`ACT_OUTCOME_KIND_ACCEPTED_CHANGE` = 「compare→dismiss = reconciled」が当該 entry に到来）** とし、すべて event log の純関数として導出する（reproject で bit-for-bit 再生）。既存の `ChangedDiffCard` の redline + compare CTA と [[000908]] の accepted_change 意味論をそのまま再利用する。

   - 却下した代替: **Continuation のまま空原因だけ修正** — 自己参照ゆえ新規 `now` 記事の Orient が構造的に空のまま残る。**Cluster でループ先行** — topic / tag 燃料の実流通が未検証（[[000929]] 系 retrospective E-6 の「proto+FE landed, producer 未配線」リスク）。Contradiction は燃料が画面で見えている唯一の種。

3. **3 つの deterministic な process gate をこの改修に必須同梱する。** [[000937]] / 振り返りの結論「病は incomplete implementation ではなく構造的プロセスギャップ」に従い、execution と process を一体で landed させる。
   - **垂直真実テスト**: stub でなく実 projector を seeded な superseded-article event log に通し、`GetKnowledgeLoop` が ≥1 件の Contradiction relation（かつ state 遷移）を返すことを assert する。`relations=0` を merge 前に RED にできる唯一のテスト。CDC RED もこの read 契約に先行させる（CLAUDE.md #7）。
   - **不正 payload で fail-loud**: relation JSONB の malformed を decode / marshal / parse / lookup の各座面で panic / loud log にする。**空集合は正当な状態として残す**（「バグ＝壊れた payload」と「燃料無し＝空」を区別する。CLAUDE.md #8）。
   - **relation_coverage SLI アラート**: honest な `relation_coverage_total{has_relations}` に閾値アラートを付ける。producer が静かに死んで relations が全件空に戻る（[[PM-2026-045]] の機序: HTTP 全 200・fail-loud も鳴らない）を検知する唯一の gate。

   - 却下した代替: 必須 gate から SLI アラートを外す案 — fail-loud は malformed のみ、垂直テストはコード退行のみを捕まえ、producer-death による「空戻り」をどちらも検知できないため、まさに今回の障害状態へ静かに戻り得る。

4. **嘘の計器を撤去し、降格を明示する。** `plannerVersionForResolver` の無条件 v2 タグと未使用 `hasV2Signal` を撤去 / 整合させ、`decideBucketV2` の「配置権威 → レンズ要約役」への降格を型 / metric label / コメントで明示する。計器が現実と乖離しない状態にする。

5. **loop.v1 を正準 read に統合する（SSR 集約）。** SSR (`getKnowledgeLoopForUser`) が recall_candidates / today_digest / feature_flags をサーバー側で集約して 1 つの loop payload にマージし、client の `GetKnowledgeHome` 呼びと flat-items の重複を撤去する。`/loop` を client から見て単一 read にし、F.3「二重 endpoint を持たない」へ寄せる。lens CRUD / TrackHomeAction 等の管理 RPC は home.v1 に残す。

   - 却下した代替: **home.v1 を正準に（relations を home へ）** — flat→3 軸の大改造で血液量が最大。**BFF facade のみ（proto 不変で 2 本 merge）** — 「正準はどちらか」が未解決のまま残り将来再燃する。

   - **実装検証による訂正 (2026-06-03)**: 本決定の前提（「`/loop` が loop.v1 と home.v1 を二重に読んでいる」）は HAR の誤読だった。実コード上、`/loop` ページと `(app)` layout は `useKnowledgeHome` / `RecallRail` / `knowledge_home` を一切 import しておらず、**`/loop` は既に loop.v1 単独ソース**（`useKnowledgeLoop` + `useKnowledgeLoopStream`）である。HAR に現れた `GetKnowledgeHome` は別サーフェス `/home` の load であり、loop の重複読みではない。したがって **#5 は /loop に対して no-op と判断し descope** した。home.v1 は `/home` サーフェスの正準 read として正当に残り、F.3 違反は存在しない。relations が空だった真因は本 ADR の Context（`extractRelations` の collapse）であって読み取り経路の二重性ではないと確定した。

immutable invariants（append-first / reproject-safe / disposable projection / why-first-class / event-time purity）は全て保持する。relations は event log の純関数であり、retire する stage イベント（`oriented` / `decision_presented`）は log に残す。

関係種の正準語は repo root の `CONTEXT.md` に固定した（Continuation / Cluster / Contradiction / Inquiry を「新規エントリを situ できるか / 自己参照か」で峻別、single-axis collapse / decorated feed を anti-term として明記）。

## Consequences

### Pros

- **燃料を捨てなくなる。** `extractRelations` が全種 emit することで、矛盾 / クラスタ証拠を持つエントリの Orient が初めて埋まり、single-axis collapse がデータ層で根絶される。
- **最初の demonstrable ループが小さく確実。** Contradiction は燃料・CTA（ChangedDiffCard）・outcome 意味論（accepted_change）が既に実在するため、新規配線は最小で、ループの閉じが画面で見える。
- **劣化が deterministic に見えるようになる。** 垂直真実テストが merge 前に空を RED 化し、fail-loud が壊れた payload を即停止させ、SLI アラートが producer-death の空戻りを検知する。「テスト緑・体験空」と [[PM-2026-045]] 型の silent failure を同時に塞ぐ。
- **計器が嘘をつかなくなる。** 嘘の v2 タグ撤去と降格明示で、ダッシュボードが実配置を反映する。
- **読み取り経路が単一化する。** client は loop payload 1 本だけを見るようになり、flat-items 重複と二重 endpoint の混乱が消える。
- **reproject-safe を崩さない。** relations / Contradiction state は純関数、retire した stage イベントは log に残るため過去状態は再生可能。

### Cons / Tradeoffs

- **projection schema と read 契約が横断的に変わる。** relations の state 導出、loop.v1 への recall/digest/flags 集約、proto への field 追加が migration を要し hard-to-reverse。SSR 集約はサーバー側 fan-out のレイテンシ予算を要する。
- **全種 emit と Contradiction 先行の非対称。** Contradiction 以外（Cluster / Continuation / Inquiry）は当面 OPEN 表示のみで戻り diff を持たないため、「種によってループが閉じたり閉じなかったり」する過渡期が生じる。UI で誤読されないよう state 未対応種の表現を抑制する必要がある。
- **gate の運用コスト。** SLI アラート閾値が低すぎれば producer-death を見逃し、高すぎれば「燃料無しエントリが多い」正常状態で誤発火する。閾値の初期値は実装 PR で calibration が要る。
- **未決の実装サブ決定が残る。** (a) reproject migration 手順、(b) SLI アラート閾値の具体値、(c) `Relation` proto への正確な field 追加形、(d) Contradiction state を導く resolver 信号の露出形（supersede-seq vs accepted_change-seq）は本 ADR では確定せず実装 PR で詰める。
- **[[000937]] との関係が部分上書き。** 第 1 スライスの関係種選択（Continuation-first）のみを訂正し、relation-set 主軸という核は維持するため、両 ADR を併読しないと「なぜ Continuation でないのか」が読み取りにくい。本 ADR の Context がその橋渡しを担う。
