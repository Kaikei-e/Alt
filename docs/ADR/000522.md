# ADR-000522: /desktop と /mobile パスルートを削除しビューポートベースのレスポンシブ設計に一本化する

## Status

Accepted

## Date

2026-03-21

## Affected Services

- alt-frontend-sv (SvelteKit)

## Context

alt-frontend (Next.js) の完全廃止（[[000395]]）以降、SvelteKit の `(app)` レイアウトグループに `ResponsiveLayout` コンポーネントと `useViewport()` ストアによるビューポートベースのレスポンシブ設計が導入された（[[000453]], [[000454]]）。

しかし、旧来の `/desktop/*` と `/mobile/*` パスルートが `src/routes/desktop/`（19 ファイル）と `src/routes/mobile/`（17 ファイル）に残存していた。`hooks.server.ts` の `resolveResponsiveRedirect()` がすべてのリクエストを `(app)` ルートへ 301 リダイレクトしているため、これらのルートファイルは**到達不能なデッドコード**であり、技術的負債となっていた。

加えて、コンポーネント内やナビゲーション設定、E2E テストに `/desktop/` `/mobile/` を参照するハードコードパスが散在しており、コードベースの一貫性を損なっていた。

## Decision

到達不能なデッドコードであるパスベースのルートファイルを全削除し、残存するハードコードパスを統一パスに修正する。`redirect-resolver.ts` はブックマーク・ブラウザ履歴からのアクセスに対する安全ネットとして維持する。

### 削除対象

- `src/routes/desktop/` — 19 ファイル（レイアウト、ページ、unit test 含む）
- `src/routes/mobile/` — 17 ファイル

### ハードコードパス修正

| ファイル | 変更内容 |
|---------|---------|
| `UnreadFeedsWidget.svelte` | `/desktop/feeds` → `/feeds` |
| `RecapSummaryWidget.svelte` | `/desktop/recap` → `/recap` |
| `DesktopTagTrailScreen.svelte` | `/desktop/settings/feeds` → `/settings/feeds` |
| `TagTrailScreen.svelte` | `/mobile/feeds/manage` → `/settings/feeds` |
| `EmptyFeedState.svelte` | `/mobile/feeds/manage` → `/settings/feeds` |
| `FloatingMenu.svelte` | `/mobile/articles/view`, `/mobile/articles/search` → エントリ削除（対応ルートなし） |
| `navigation.ts` | 同上 + Articles カテゴリ削除 |

### E2E テスト修正

- POM 5 ファイルの URL を統一パスに変更
- `job-status.spec.ts`（desktop/mobile 各 1）のハードコード URL を統一パスに変更
- a11y スペック 2 ファイルの `test.describe` 内パスコメントを更新

### 維持するもの

| 対象 | 理由 |
|------|------|
| `redirect-resolver.ts` + テスト | ブックマーク・履歴からの 301 リダイレクト安全ネット |
| `src/lib/components/desktop/`, `mobile/` | ビューポート別コンポーネントとして `(app)` ルートから使用中 |
| `tests/e2e/desktop/`, `mobile/` | ビューポート別テスト整理（ルートではなくテスト対象画面の分類） |
| `tests/e2e/pages/desktop/`, `mobile/` | ビューポート別 POM（Desktop/Mobile で異なる UI をテスト） |
| `tests/e2e/helpers/navigation.ts` | prefix strip ロジックは統一パスでも正常動作 |

## Consequences

### Pros

- デッドコード 36 ファイルの削除により、ルーティング構造が単純化される
- コンポーネント内のリンクが統一パスで一貫し、ナビゲーションの予測可能性が向上する
- 存在しないルートへのデッドリンク（`/mobile/articles/view`, `/mobile/articles/search`）が除去される
- `svelte-kit sync` 後のタイプチェックで旧ルート由来のエラーが解消される

### Cons / Tradeoffs

- `redirect-resolver.ts` が残るため、完全な除去ではない（ブックマーク安全ネットとしてのトレードオフ）
- E2E テストのディレクトリ名に `desktop/`, `mobile/` が残るが、これはビューポート分類であり、パスルートとは無関係なため許容する
