# Inoreader記事をalt-dbコアArticleとしてバックフィルするジョブの追加

## ADR's STATUS

Accepted

## CONTEXT

alt-dbに登録済みのFeedに対して、対応するArticleが0件のケースが存在していた。一方、pre-processor-sidecarがInoreader APIから取り込んだ記事データは `inoreader_articles` テーブルに蓄積されている。

既存の `ArticleSyncService.SyncArticles()` は直近24時間のInoreader記事のみを `articles` テーブルに同期する仕組みだが、以下のケースがカバーされていなかった:

1. **フィードが後から追加された場合** — Inoreader記事は取得済みだが、alt-dbにFeedが存在しなかったため同期がスキップされた記事
2. **24時間以前の履歴記事** — SyncArticlesの時間ウィンドウから外れた過去の記事

### データベース分離の制約

本番環境（API mode）では、Inoreader関連テーブル（`inoreader_articles`, `inoreader_subscriptions`）はpre-processor-dbに、コアテーブル（`feed_links`, `feeds`, `articles`）はalt-dbにある。クロスデータベースJOINは不可能なため、単一SQLでの「空フィード判定 + 記事取得」はできない。

### URL解決の不整合

既存の `GetFeedID()` は `feeds.link`（ウェブサイトURL）でマッチングしているが、Inoreader側の `feed_url` はRSS URL。正しいパスは `feed_links.url` → `feeds.feed_link_id` 経由である。

## DECISION MAKING

pre-processorの `ArticleSyncService` に `BackfillEmptyFeeds` メソッドを追加し、毎時実行の常駐ジョブとして組み込む。

### 実装戦略: DB分離に応じた二重実装

**Legacy mode（同一DB）:**
- `feed_links` 経由の正しいJOINで空フィードを直接判定
- `NOT EXISTS (SELECT 1 FROM articles WHERE feed_id = f.id)` でArticleが0件のフィードのみ対象
- `feed_id` をSQLで直接解決

**API mode（分離DB、本番）:**
- pre-processor-dbの `inoreader_articles` + `inoreader_subscriptions` のみをクエリ
- `feed_id` はバックエンドAPIの `GetFeedID` で解決（feedURL → feedIDキャッシュ付き）
- `fetched_at` カーソルで進行管理し、同じ記事の再処理を防止

### 変更内容

**Driver層** (`pre-processor/app/driver/db_articles.go`):
- `GetInoreaderArticlesForEmptyFeeds()` — 同一DB用、`feed_links` JOINで空フィード判定
- `GetInoreaderArticlesForBackfill()` — 分離DB用、`inoreader_*` テーブルのみ + `fetched_at` カーソル

**Repository層** (`pre-processor/app/repository/`):
- `FetchInoreaderArticlesForEmptyFeeds(ctx, fetchedAfter, limit)` — 新インターフェース
- `UpsertArticlesWithFeedID(ctx, articles)` — FeedID解決済み記事のバッチUPSERT

**Service層** (`pre-processor/app/service/article_sync_service.go`):
- `BackfillEmptyFeeds()` — 取得 → サニタイズ → バリデーション → UPSERT → カーソル更新
- `lastBackfillFetchedAt` フィールドで進行状態を保持

**Handler層** (`pre-processor/app/handler/job_handler.go`):
- `StartBackfillJob()` — 起動時即実行 + 1時間間隔のジョブ

**Bootstrap** (`pre-processor/app/bootstrap/lifecycle.go`):
- `startJobs()` にバックフィルジョブ起動を追加

### Inoreader APIへの影響

**なし。** このバックフィルは `inoreader_articles` テーブル（既にローカルに取得済みのデータ）から `articles` テーブルへの内部データ移動であり、Inoreader APIへのリクエストは一切発生しない。

## RESULTS, EFFECTS

### PROS

- 空フィードが自動的にInoreader記事で埋まる
- 既存の `SyncArticles`（24h窓）とバックフィル（全履歴）の役割が明確に分離
- `ON CONFLICT` UPSERTによりべき等（何度実行しても安全）
- LIMIT 1000 + カーソルにより負荷を分散、毎時実行で徐々に収束
- feedURLキャッシュにより同一バッチ内のAPI呼び出しを最小化
- Inoreader APIのレートリミット（100コール/日）に影響なし

### CONS, TRADEOFF

- API mode（本番）では「空フィード」の厳密な判定ができない。全Inoreader記事を対象にfeedID解決を試み、UPSERTで重複を吸収する方式。既存記事の再UPSERTは実質no-opだが、API呼び出しのオーバーヘッドがある
- カーソルはインメモリのため、pre-processor再起動時にリセットされる。ただし UPSERTがべき等なので再処理は安全
- Legacy mode用のクロスDB JOINクエリは、本番のDB分離構成では使用されない（将来的にlegacy modeを廃止する場合は削除可能）
