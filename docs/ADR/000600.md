# ADR-000600: Ask Augur の article-scoped クエリに SubIntent 分類を導入し要約構造の強制を排除する

## Status

Accepted

## Date

2026-03-27

## Affected Services

- rag-orchestrator (Go)
- alt-frontend-sv (TypeScript / SvelteKit)

## Context

Ask Augur で記事に対して「反論はある？」と質問した際、反論・批判の分析ではなく記事の要約が返ってきた。会話が成立していない。

根本原因は 3 層に分かれていた:

1. **IntentArticleScoped で分類が止まる**: `answer_with_rag_usecase.go` の `buildPrompt()` が article-scoped クエリの場合に `QueryClassifier.Classify()` をスキップしていた。「反論はある？」という質問の分析的意図が消失していた。

2. **分析系の Intent が未定義**: [[000568]] で導入した QueryClassifier は Comparison/Temporal/FactCheck/TopicDeepDive の 4 種類。反論・批判・評価・影響といった分析的質問タイプは分類対象外だった。

3. **プロンプトが要約構造を強制**: `writeOutputFormat()` の JSON exemplar が `{"answer":"## 概要\n...\n## 詳細\n...\n## まとめ\n..."}` という要約テンプレートを含んでいた。Gemma 3 4B はこの exemplar に忠実に従い、質問の種類に関係なく「概要→詳細→まとめ」構造で回答を生成していた。

加えて、FE のサジェスト質問候補は Sheet 表示時に固定され、他の候補を見る手段がなかった。

### Ollama options の不整合

調査中に `rag-orchestrator` が news-creator proxy 経由で Ollama を利用しているにもかかわらず、独自に sampling パラメータ（temperature/top_p/top_k/num_ctx/repeat_penalty）と `keep_alive: -1` を送信していることが判明した。これは [[000579]] で確立した「全リクエストパスで config baseline を統一し model reload を防止する」方針に違反していた。

## Decision

**`SubIntentType` を導入し、article-scoped クエリ内の分析的質問を分類して、プロンプトの answer exemplar と instruction を分析寄りに切り替える。** 加えて、Ollama options を news-creator proxy baseline に寄せる修正と、FE サジェスト質問のシャッフル機能を同時に実装する。

### 検討した代替案

| 案 | 評価 |
|---|---|
| A. IntentType を上書きして IntentCritique 等を追加 | article-scoped の strategy routing（全チャンク取得 + BM25 リランク）が壊れる。5 箇所以上の `IntentArticleScoped` 分岐に影響 |
| B. LLM によるクエリ分類 | Gemma 3 4B では分類精度が低く（hallucination ~79%）、追加レイテンシも発生。ルールベースで十分な範囲 |
| C. プロンプト全面書き換え | 影響範囲が大きすぎる。要約構造が有効な一般質問を壊すリスク |
| D. SubIntentType を導入（**採用**） | IntentType（strategy routing）を維持したまま、プロンプト構築のみに影響する別の型で分類。後方互換性を完全に保つ |

### 設計上の主要判断

- **SubIntentType は IntentType とは別の型**: Go の型システムで strategy map key への誤用を防止。`SubIntentType` は `prompt_builder.go` のみで参照される
- **answer exemplar の切り替え、JSON schema は維持**: Ollama の `format` フィールドで structured output を保証する JSON Schema は変更しない。変えるのは `answer` フィールド内の exemplar テキストと instruction のみ
- **Critique の multi-turn は明示的禁止事項**: 「記事の要約を繰り返さず、主張の弱点・反証可能性・欠落前提に集中すること」と強く指示
- **分析系は 800 文字制限を緩和**: 短い分析質問に対して過剰な長文化が要約・総論化を誘発するリスクを回避
- **Ollama options は proxy baseline に委譲**: Gemma case から sampling パラメータを削除し、`num_predict` のみ per-request override。`keep_alive` は送信しない

## Implementation

### バックエンド (rag-orchestrator)

| ファイル | 変更内容 |
|---------|---------|
| `internal/usecase/query_intent.go` | `SubIntentType` 型・定数 (critique/opinion/implication) + `QueryIntent` にフィールド追加 |
| `internal/usecase/query_classifier.go` | `ClassifySubIntent()` メソッド + 3 matchers (Critique > Opinion > Implication 優先順) |
| `internal/usecase/prompt_builder.go` | `writeAnalyticalOutputFormat()` + SubIntent 別指示ブロック (single-turn/multi-turn) + 800 文字制限緩和 |
| `internal/usecase/answer_with_rag_usecase.go` | article-scoped 時に `ClassifySubIntent()` を呼び出し + `PromptInput`/`promptBuildResult`/`AnswerDebug` に伝播 |
| `internal/usecase/rag_answer_stream.go` | Stream パスの `AnswerDebug` に `SubIntentType` 追加 |
| `internal/usecase/rag_answer_types.go` | `AnswerDebug.SubIntentType` フィールド追加 |
| `internal/adapter/rag_augur/ollama_generator.go` | Gemma case: sampling params 削除 (proxy baseline 委譲)、`keep_alive` 全削除 |
| `internal/adapter/rag_http/handler.go` | REST ハンドラの `AnswerDebug` → OpenAPI `AnswerDebug` マッピングに `SubIntentType` 追加 |
| `spec/openapi.yaml` | `AnswerDebug` スキーマに `sub_intent_type` フィールド追加 |
| `internal/adapter/rag_http/openapi/server.gen.go` | OpenAPI spec から再生成（`oapi-codegen`） |

### SubIntentType キーワードマッチ

| SubIntent | JP キーワード | EN キーワード |
|-----------|-------------|--------------|
| Critique | 反論, 批判, 弱点, 問題点, 欠点, リスク, デメリット, 懸念, 課題, 限界 | counterargument, criticism, weakness, limitation, drawback, risk, concern, flaw, downside |
| Opinion | どう思う, 評価, 意見, 見解, 感想, 判断 | what do you think, opinion, assessment, evaluation, judgment |
| Implication | 影響は, 意味は, どういう意味, 結果は, 将来, 今後 | implication, what does this mean, impact, consequence, going forward |

### プロンプト出力形式の分岐

```
SubIntent = None (従来動作):
  {"answer":"## 概要\n...\n## 詳細\n...\n## まとめ\n..."}

SubIntent != None (分析系):
  {"answer":"質問に対する分析的な回答をここに書く"}
  + SubIntent 別指示 (批判的分析 / 評価・意見 / 影響・意味合い)
```

### フロントエンド (alt-frontend-sv)

| ファイル | 変更内容 |
|---------|---------|
| `src/lib/components/knowledge-home/ask-suggestions.ts` | mulberry32 seeded PRNG 導入。`pickSuggestions(tags, seed)` で deterministic な候補選択 |
| `src/lib/components/knowledge-home/AskSheet.svelte` | `shuffleCount` state + Shuffle アイコンボタン + `{#key shuffleCount}` で確実な再レンダリング |

### テストカバレッジ

- `query_classifier_test.go`: SubIntent 各キーワードマッチ (JP/EN) + None フォールバック + 優先順競合ケース (10 test groups)
- `prompt_builder_test.go`: Critique SubIntent で要約構造が消えること + None で既存動作維持 + multi-turn critique guidance + 800 文字制限緩和 (4 tests)
- `ollama_generator_test.go`: Gemma case で sampling params が送信されないこと
- `ask-suggestions.test.ts`: deterministic seed + 異なる seed で異なる結果 + 連続 seed で重複回避 (10 tests)

## Consequences

### Pros

- 「反論はある？」等の分析的質問に対して、要約ではなく質問に直接回答する構造が生成される
- `IntentArticleScoped` の strategy routing は完全に維持され、article-scoped の全チャンク取得 + BM25 リランクが壊れない
- `AnswerDebug.SubIntentType` により、分類の効果をログ（`query_intent_parsed`）、REST API debug レスポンス（`sub_intent_type`）、Connect-RPC Stream meta の 3 箇所から観測可能
- Ollama options が news-creator proxy baseline に統一され、model reload リスクが低減
- FE サジェスト質問が seeded RNG になり、テストが deterministic かつシャッフル可能

### Cons / Tradeoffs

- SubIntent のキーワードマッチは substring ベースのため、意図しないマッチ（誤爆）の可能性がある。ただし Critique > Opinion > Implication の優先順で最も具体的なキーワードが先に判定される
- Gemma 3 4B が分析的指示にどこまで忠実に従うかはモデル能力に依存する。exemplar の変更が最も効果的だが、モデル品質の限界はある
- `keep_alive` を送信しなくなったことで、proxy 側の設定変更が即座に rag-orchestrator に影響する。ただしこれは意図された設計（canonical source の一本化）
- multi-turn で初回ターンのフルインストラクション（品質基準等）が省略される制約は [[000597]] から継続。SubIntent 指示はフォローアップ指示に追記する形で対応
