# ADR-000816: acolyte-orchestrator の ty 型検査 6 エラーを cast 3 箇所とテスト stub 補強で解消する

## Status

Accepted

## Date

2026-04-20

## Affected Services

- **acolyte-orchestrator (Python)** — `acolyte/usecase/graph/nodes/curator_node.py`, `tests/e2e/test_citation_hygiene_e2e.py`

## Context

2026-04-19 の CI run Alt/24635232911 (commit `e8387537`) で Acolyte-Orchestrator Quality Gates の `typecheck / python-job` が `ty` (pyrefly) 6 errors で fail した:

1. `acolyte/usecase/graph/nodes/curator_node.py:80` — `self._language_quota(section_role, report_type)` で `report_type` が `object | Unknown | None`。signature は `str | None`。
2-3. `tests/e2e/test_citation_hygiene_e2e.py:125, 141` — `await writer(state)` で plain `dict` を `WriterNode.__call__(state: ReportGenerationState)` に渡している。
4. `tests/e2e/test_citation_hygiene_e2e.py:148` — `FinalizerNode(repo)` で `InMemoryReportRepo` が `ReportRepositoryPort` protocol の `create_report` を持たない。
5. `tests/e2e/test_citation_hygiene_e2e.py:159` — `await finalizer(finalizer_state)` で plain `dict` を `FinalizerNode.__call__(state: ReportGenerationState)` に渡している。
6. `tests/e2e/test_citation_hygiene_e2e.py:175` — `FinalizerNode(repo)` の protocol 不整合 (4 と同じ)。

テスト fixture 側に集中しているが、(1) は production code の型ナローイング問題、(2-6) は test stub が port の完全実装ではないことを検査器が検出した形。特に `InMemoryReportRepo` は `ReportRepositoryPort` の 13 メソッド中 5 メソッド (`get_report`, `bump_version`, `get_sections`, `create_section`, `bump_section_version`) しか実装していない。`FinalizerNode` が実際に呼ぶのは `bump_version` と `bump_section_version` だけで、他の 8 メソッドはテスト scope で呼ばれないため runtime では問題ない。

`ReportGenerationState` は `acolyte/usecase/graph/state.py` の `TypedDict(total=False)`。テストの `state` は `_llm_responses` というテスト専用の制御キーを含むため、単純に `state: ReportGenerationState = {...}` にすると `_llm_responses` が TypedDict 外のキーで Pyright に怒られる別種の regression を招く (実際に実装途中で検出)。

## Decision

**「test 側は部分的 stub のまま、型検査の境界だけ cast で告げる」** 方針を採る。production code は小さな annotation を足すだけで通す。

### curator_node.py — cast で str | None を告げる

```python
# 変更前
report_type = state.get("report_type") or brief.get("report_type")

# 変更後
report_type = cast("str | None", state.get("report_type") or brief.get("report_type"))
```

`state.get("report_type")` は TypedDict に `report_type` キーが declare されていないため `object` を返す。実装上は `str | None` である保証が `Brief` / planner から来ているので cast は正当。

### test_citation_hygiene_e2e.py — stub 補強と cast

1. `InMemoryReportRepo` に最小の `create_report` stub を追加（既存の `self.report` を返すだけ）。FinalizerNode 以外の将来の callers が触ったときのための保険。
2. `FinalizerNode(repo)` 2 箇所を `FinalizerNode(cast(ReportRepositoryPort, repo))` にする。InMemoryReportRepo は port の部分実装に意図的に留めておく — 全 13 メソッドを実装するのは過剰で、stub が乖離しやすくなる。
3. `writer(state)` / `finalizer(state)` の 4 箇所を `cast(ReportGenerationState, state)` でラップ。
4. インライン dict を直接 `await finalizer({...})` に渡していた 1 箇所を `finalizer_state` 変数に抽出し、同様に cast でラップ（局所変数のほうが type comment の粒度が揃う）。

**却下した代替案**:

- **`InMemoryReportRepo` を 13 メソッド全て実装する** — 8 メソッドは E2E で呼ばれないノップで、protocol ドリフトが起きたときに stub も同期修正が必要になる。stub が本物の port の責務を模倣し始めると、ユニットテストの信頼性がかえって下がる。
- **`ReportGenerationState` に `_llm_responses: NotRequired[list[str]]` を足す** — production schema に test-only キーが漏れる。TypedDict は domain 契約なので test の制御信号を混ぜるのはアンチパターン。
- **`runtime_checkable` protocol にして `@runtime_checkable` で structural check に任せる** — `ty` の static check が弱まる（protocol check が loose な `__instancecheck__` ベースになる）ので、今回の問題再発を早期検出できなくなる。
- **pytest ignore comment (`# type: ignore[...]`) で黙らせる** — 問題の構造が見えなくなる。cast のほうが「test stub の意図」が読める。

### 検証

- `cd acolyte-orchestrator && uv run pyrefly check .` → `0 errors (52 suppressed, 9 warnings not shown)`
- `uv run pytest tests/ -v` → 656 passed
- `uv run ruff check && uv run ruff format --check` → 全 green

## Consequences

### Pros

- **ty 6 エラーが同 ADR の変更で全消し**。production code への侵襲は cast 1 行 + import 追加のみ。
- **test stub の範囲を意図的に部分実装にできる**。port 全体を stub するコストから解放され、テストの示す動作契約が明確になる (`FinalizerNode` が実際に呼ぶメソッドだけが必要)。
- **TypedDict の将来拡張が test 都合で歪まない**。`_llm_responses` のような test-only キーは `state: dict` のまま扱い続けられる。

### Cons / Tradeoffs

- **cast は runtime no-op**。`state` が実は `ReportGenerationState` を満たしていなくても検査器は通ってしまう。これは test code に閉じているため許容（E2E テスト自体が state の妥当性を runtime で確認する）。
- **`InMemoryReportRepo` が port の部分実装であることは依然 protocol 違反**。今回 cast で握りつぶしており、将来 FinalizerNode が `create_brief` などを呼び始めると runtime で `AttributeError` になる。その時は stub を拡張するか、E2E を unit + contract に分解する。
- **pyrefly と Pyright で strict 度が違う**（Pyright のほうが厳しい）。CI は ty/pyrefly ベースなので今回は通るが、IDE の Pyright は `InMemoryReportRepo` について追加 8 メソッド欠落を warning として報告し続ける。IDE noise は許容する方針。
