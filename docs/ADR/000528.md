# ADR-000528: batch safety-net を enqueue ベースに変更して未要約記事の消化速度を引き上げる

## Status

Accepted

## Date

2026-03-22

## Affected Services

- pre-processor (Go)

## Context

要約パイプラインのスループット調査において、queue は健全（pending 37 件、completion rate 344 件/h）にもかかわらず、ユーザー視点では未要約記事が 1,175 件残っている乖離が確認された。

原因は、記事を queue に投入する経路が 2 つ存在し、それぞれの処理能力に大きな差があったことにある。

### 経路 1: イベント駆動（queue 経由）

```
ArticleCreated event → Redis consumer → ShouldQueueSummarizeJob guard → CreateJob → queue → worker → LLM
```

- `ArticleCreated` イベントでのみ発火する。`ArticleUpdated` では queue に積まない（[[000512]] で分離済み）。
- queue worker は concurrency=3、分散 dispatch（[[000510]]）で 344 件/h を処理できる。

### 経路 2: batch safety-net（直接 LLM 呼び出し）

```
5分タイマー → processSummarizationBatch → FindForSummarization API → LLM → Save
```

- queue を経由せず、backend API から未要約記事を取得して直接 LLM を呼ぶ。
- queue に pending がある間は defer され、30 分ごとの force sweep でのみ実行される。
- `batchSize=10`（ハードコード）のため、**実効スループットは ~20 件/h**。

### 乖離の発生メカニズム

1,175 件の未要約記事のうち queue に入っていた 37 件は、`ArticleCreated` イベントで投入されたもの。
残りの 1,138 件は以下の理由で queue に到達していなかった。

- [[000512]] 以前のイベント設計差分により `ArticleUpdated` として発行された記事が queue に積まれなかった
- Redis consumer 停止中にイベントが消失した可能性がある
- batch safety-net は存在するが、batchSize=10 × 30 分 force sweep = ~20 件/h では追いつかない

batch safety-net は取りこぼし回収の役割を担っているが、queue worker（344 件/h）の 17 分の 1 の速度しか出せていなかった。

## Decision

**batch safety-net の処理方式を「直接 LLM 呼び出し」から「guard + CreateJob で queue に投入」に変更する。**

これにより、batch safety-net が見つけた未要約記事を queue worker + 分散 dispatch の既存インフラで処理できるようにする。

### 具体的な変更

1. `SummarizeQueueWorker` に `EnqueueUnsummarizedBatch(ctx, batchSize)` メソッドを追加する
   - `articleRepo.FindForSummarization` で未要約記事を取得
   - 各記事に `ShouldQueueSummarizeJob` guard を適用（[[000513]] の admission guard を共有）
   - guard 通過 → `jobRepo.CreateJob` で queue に投入
   - `CreateJob` の `WHERE NOT EXISTS` で pending/running 重複を防止
   - pagination cursor を内部管理

2. `processSummarizationBatch` を enqueue パスに変更する
   - `queueWorker != nil` の場合: `EnqueueUnsummarizedBatch` を呼ぶ（primary path）
   - `queueWorker == nil` の場合: 従来の `SummarizeArticles` にフォールバック
   - defer / force sweep ロジックは変更なし

3. `EnqueueResult` 構造体で found / enqueued / skipped / errors / hasMore を返す

### 検討した代替案

| 案 | 評価 |
|---|---|
| batchSize を引き上げる（環境変数化） | 小変更だが、queue と直接 LLM が並走する構造が残る。GPU 競合が解消しない |
| 一時的な backfill SQL で queue に一括投入 | 即効性はあるが一回限り。マルチテナント条件や guard 条件のバイパスリスクがある |
| Redis consumer をバックフィル対応にする | event replay のインフラが必要で、変更規模が大きすぎる |

## Implementation

### 変更ファイル

| ファイル | 変更内容 |
|---|---|
| `service/summarize_queue_worker.go` | `EnqueueResult` 構造体、`EnqueueUnsummarizedBatch`、`ResetEnqueueCursor` を追加 |
| `handler/job_handler.go` | `processSummarizationBatch` を enqueue パスに変更。`queueWorker == nil` 時のフォールバック維持 |

### テスト

| テスト | 検証内容 |
|---|---|
| `TestEnqueueUnsummarizedBatch_EnqueuesViaGuard` | 記事取得 → guard pass → CreateJob が呼ばれること |
| `TestEnqueueUnsummarizedBatch_SkipsSummaryExists` | guard が `summary_exists` で skip、CreateJob が呼ばれないこと |
| `TestEnqueueUnsummarizedBatch_SkipsRecentSuccess` | guard が `recent_success` で skip、CreateJob が呼ばれないこと |
| `TestEnqueueUnsummarizedBatch_NoArticles` | 0 件取得時に 0 enqueue、エラーなし |
| `TestEnqueueUnsummarizedBatch_DuplicateHandled` | CreateJob 空文字返却（重複）時にエラーにならないこと |
| `TestEnqueueUnsummarizedBatch_HasMore` | cursor 返却時に `HasMore=true` |
| `TestProcessSummarizationBatch_DefersWhenQueueHasPendingJobs` 5 ケース | defer / enqueue / fail-open / force sweep / nil fallback |

### 動作確認結果

```
01:44:27 UTC  batch enqueue completed: found=10, enqueued=10, skipped=0, has_more=true
01:46:xx UTC  7/10 completed, 3 running — queue worker が分散 dispatch で処理中
```

## Consequences

### Pros

- batch safety-net が queue worker のスループット（344 件/h）をフル活用できるようになった
- queue に一本化されたことで、観測（pending/completed/failed 件数）・再試行・dequeue 制御が 1 箇所に集約された
- guard + CreateJob の共通化により、event 経路と batch 経路で同じ idempotency ルールが適用される
- defer / force sweep のロジックは変更なく、既存の backpressure 制御をそのまま活かせる

### Cons / Tradeoffs

- batchSize=10 は引き続きハードコードされている。enqueue パスでは queue worker が処理するため GPU 競合はないが、投入レート（~120 件/h）は batchSize に制約される。環境変数化は別途検討する
- `queueWorker == nil` 時のフォールバックとして直接 LLM パスが残っている。将来的に削除可能だが、現時点では後方互換のために維持する
- batch path が enqueue に変わったことで、batch 単独での即座の要約生成はできなくなった。ただし queue worker が常時稼働しているため、実運用上の影響はない

### 今後のステップ（この ADR の範囲外）

1. **batchSize の環境変数化**: `SUMMARIZE_BATCH_SIZE` を導入し、上限・既定値・バリデーション・テストをセットで実装する
2. **計測の追加**: batch enqueue の found / enqueued / skipped / errors をメトリクスとして公開し、unsummarized count の減少を時系列で可視化する
3. **直接 LLM フォールバックの削除**: `queueWorker` が常に non-nil であることを確認した上で、フォールバックパスを削除する

## Lessons Learned

### queue に一本化することで観測性が向上する

直接 LLM 呼び出しの batch path は「処理されたかどうか」がログでしか追えなかった。
queue 経由にすることで、pending / completed / failed / dead_letter の状態遷移が DB に記録され、定量的な監視が可能になった。

### 「壊さない → 測る → 速くする」の順序は今回も有効だった

[[000509]] で確立した原則を今回も適用した。
まず調査で乖離の原因を特定し（測る）、既存の guard + CreateJob を共有する形で経路を変更した（速くする）。
整合性を壊さずにスループットを改善できたのは、先行 ADR の成果物（atomic dequeue、admission guard、idle-aware dispatch）が安定していたからである。
