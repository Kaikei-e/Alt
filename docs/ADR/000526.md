# ADR-000526: トレンド統計クエリを FULL OUTER JOIN に変更し summarized カウントの集計基準を修正

## Status

Accepted

## Date

2026-03-22

## Affected Services

- alt-backend (Go)

## Context

`fetch_trend_stats_driver.go` の `buildTrendQuery` はトレンド統計（記事数・要約数・フィード活動数）を時間バケットごとに集計するクエリを生成する。従来の実装には2つの問題があった。

### 1. summarized カウントの集計基準が不正確

旧クエリでは `articles` テーブルに `article_summaries` を LEFT JOIN し、`articles.created_at` で `date_trunc` したバケットに要約数を計上していた。これにより、**記事の作成日と要約の作成日が異なる時間バケットに属する場合**、要約が実際に生成されたバケットではなく記事が作成されたバケットにカウントされていた。

```
例: 記事が 3/20 に作成、要約が 3/21 に生成
旧: 3/20 のバケットに summarized=1 として計上（不正確）
新: 3/21 のバケットに summarized=1 として計上（正確）
```

### 2. LEFT JOIN による欠落

旧クエリは `time_buckets` (articles) を基準に LEFT JOIN していたため、**記事が存在しない時間バケットにフィード活動だけがある場合**は行が返されていた。しかし、articles にも feed_activity にも存在せず article_summaries にだけ存在するバケット（過去記事の遅延要約など）は完全に欠落していた。

## Decision

**`buildTrendQuery` のクエリ構造を、独立した3つの CTE + FULL OUTER JOIN に変更する。**

### 具体的な変更

1. 単一の `time_buckets` CTE を `articles_buckets`（記事数）と `summarized_buckets`（要約数）に分離
2. `summarized_buckets` は `article_summaries.created_at` を基準に `date_trunc` を行い、要約が実際に生成された時間バケットに正確に計上
3. `summarized_buckets` にも `user_id` フィルタを追加し、マルチテナント分離を維持
4. 3つの CTE（`articles_buckets`, `summarized_buckets`, `feed_activity`）を FULL OUTER JOIN で結合し、いずれかのバケットにデータがあれば行を返す
5. `COALESCE(ab.bucket, sb.bucket, fa.bucket)` で最終的なバケットタイムスタンプを決定

### 検討した代替案

| 案 | 評価 |
|---|---|
| LEFT JOIN のまま summarized_buckets だけ分離 | 要約のみ存在するバケットが欠落する問題が残る |
| UNION ALL + GROUP BY で統合 | CTE 分離 + FULL OUTER JOIN より可読性が劣る |

## Consequences

### Pros

- 要約数が `article_summaries.created_at` 基準で正確に集計される
- 3つの時系列データ（記事・要約・フィード活動）が互いに独立して集計され、いずれか1つだけ存在するバケットも正しく返される
- `summarized_buckets` に `user_id` フィルタを追加し、マルチテナント分離が CTE レベルで一貫
- 各 CTE が単一責務となり、クエリの可読性と保守性が向上

### Cons / Tradeoffs

- `fmt.Sprintf` に渡す `truncFunc` の個数が 2 → 3 に増加（CTE が1つ増えたため）。書式文字列のプレースホルダ管理がやや煩雑になる
- FULL OUTER JOIN が2段になるため、データ量が大きい場合にクエリプランナーの挙動を注視する必要がある（現時点では問題なし）
- テストが SQL 文字列のパターンマッチに依存しているため、クエリ構造の変更時にテスト修正が必要になるトレードオフは従来と同様
