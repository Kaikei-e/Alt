# ADR-000428: Ask Augur の回答生成モデルを gemma3-4b-rag に移行する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- knowledge-augur — Ask Augur 向け Ollama モデル定義と preload 対象
- rag-orchestrator — 既定の生成モデル、max_tokens、生成プロンプト

## Context

[[000376]] で Retrieval Phase のボトルネックを解消した後も、Ask Augur の E2E 応答時間は回答生成フェーズの影響を強く受けていた。特に `gemma3-12b-rag` は品質面では安定する一方、現行の外部推論基盤では生成レイテンシが大きく、会話体験の改善余地が残っていた。

一方で、Qwen 系モデルは同環境で速度面の候補になり得たが、Ask Augur の運用要件として回答品質上の理由から採用対象外とした。したがって、Gemma 系を維持したまま、より軽量なモデルへ移行して TPS と TTFT を改善する必要があった。

既存実装では query expansion が `news-creator` 側の 4B 系モデルに寄っており、Ask Augur の主な遅延要因は knowledge-augur 側の回答生成に集中していた。このため、回答生成モデルのみを `gemma3-4b-rag` へ置き換える構成は、影響範囲を限定しつつ効果を得やすい。

移行後のベンチマークでは、`gemma3-4b-rag` の decode throughput は概ね 22 から 24 tok/s、TTFT は 0.35 から 0.80 秒に収まり、以前の 8B 系実測より改善した。E2E でも `rag-orchestrator -> augur-external -> knowledge-augur` の経路で `fallback=false` の回答と citation を確認できた。

## Decision

Ask Augur の既定回答生成モデルを `gemma3-12b-rag` から `gemma3-4b-rag` へ移行する。

具体的には以下を採用する。

- `knowledge-augur` に `gemma3:4b-it-qat` ベースの `gemma3-4b-rag` を追加する
- `rag-orchestrator` の `AUGUR_KNOWLEDGE_MODEL` 既定値を `gemma3-4b-rag` に変更する
- `RAG_DEFAULT_MAX_TOKENS` の既定値を 2048 に下げ、長すぎる生成を抑制する
- Ask Augur の回答プロンプトは `500文字以上` を維持しつつ、冒頭で結論を述べる構成へ寄せる
- Query expansion 側の構成は変更せず、回答生成フェーズのみを軽量化する

代替案として `gemma3-12b-rag` を維持したまま Retrieval や prompt をさらに調整する案もあったが、既に Retrieval 最適化を先行実施しており、次の支配的要因はモデル自体の重さだったため採用しなかった。

## Consequences

### Pros

- Ask Augur の回答生成で TPS と TTFT が改善し、対話開始までの待ち時間を短縮できる
- Gemma 系を維持するため、Qwen 系への切替で懸念された品質変動を避けられる
- 変更点を knowledge-augur の Modelfile と rag-orchestrator の既定設定に集約でき、ロールバックも容易である
- `500文字以上` の回答要件を維持したまま、結論先行の構成で読みやすさを改善できる

### Cons / Tradeoffs

- 12B から 4B への縮小により、複雑な要約や曖昧な問いに対する推論余力は低下する可能性がある
- `500文字以上` を維持しているため、短問でも総応答時間は一定以上かかる
- `RAG_DEFAULT_MAX_TOKENS` を 2048 に抑えたことで、長文出力の上限には以前より強い制約がかかる
- 品質評価はベンチマークだけでは完結せず、代表質問セットによる継続確認が必要である
