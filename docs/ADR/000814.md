# ADR-000814: alt-frontend-sv の Playwright スイートを e2e/playwright dispatch 規約として deploy E2E gate に組み込む

## Status

Accepted

## Date

2026-04-20

## Affected Services

- **alt-frontend-sv** — プロダクションコードは無変更。`e2e/playwright/alt-frontend-sv/run.sh` と `README.md` を新設し、既存 43 spec を `bash e2e/playwright/alt-frontend-sv/run.sh` 1 行で再現可能にする
- **e2e 規約** — `e2e/playwright/<svc>/run.sh` を Hurl `e2e/hurl/<svc>/run.sh` と並ぶ 2 つ目の framework dispatch として確立
- **alt-deploy (private repo)** — `release-deploy.yaml` の e2e step を two-directory probe (`hurl` → `playwright`) に拡張する後続 PR を前提にする (本 ADR の scope outside)

## Context

[[000763]] で private repo `Kaikei-e/alt-deploy` が deploy pipeline として導入され、[[000766]] で `bash e2e/hurl/<svc>/run.sh` を 3 サービス共通の E2E dispatch 規約として確立した。2026-04-19 には [[daily/2026-04-19]] に記録された通り `release-deploy.yaml` の e2e job が `strategy.matrix.service` + runtime compose slice render で並列化され、現在は 10 サービス (search-indexer / mq-hub / knowledge-sovereign / recap-worker / news-creator / tag-generator / rag-orchestrator / acolyte-orchestrator / auth-hub / alt-backend) が matrix 上で blocking E2E gate を通っている。

一方 `alt-frontend-sv` は以下の 2 つの理由で gate から外れている:

1. **Hurl suite が存在しない** — alt-deploy の e2e step は `[ -d "e2e/hurl/$SERVICE" ] || exit 0` で gate されており、該当ディレクトリ欠如により自動 skip される。alt-frontend-sv は SvelteKit 上の browser-interactive UI (auth / swipe / tag-verse canvas / streaming summaries) を提供しており、HTTP probe では contract 検証が不可能で Hurl の target ではない。
2. **Pact pacticipant にも含まれない** — frontend は Pact CDC の consumer/provider 構造から外れているため `pact-publish` / `gate` job を通らず、`e2e` が sole contract gate 候補である。

PR 時点では `.github/workflows/alt-frontend-sv.yml` が 3-shard の Playwright (auth + desktop-chromium + mobile-chrome, 43 spec) を `mcr.microsoft.com/playwright:v1.59.1-jammy` container で走らせており、この資産は 1 年以上安定して運用されている。ところが `dispatch-deploy.yaml` は `push: branches: [main]` trigger で同じ SHA の Playwright 結果を **待たずに** `repository_dispatch` を alt-deploy に送る fire-and-forget 設計であり、PR 緑 → main merge → Playwright が RED になっても deploy は発火する。

実害例: 2026-04-19 以降の Acolyte 品質改善で frontend 変更頻度が上がり、mobile の citation overflow fix (SHA 1018dc8) のように frontend-only push が発生した際に、PR でしか Playwright が走らず、deploy 前の gate が 1 つも無い状態で本番に到達する構造だった。今回は回帰無しで済んだが、mobile 表示の regression が構造的に検知できない状態は運用リスクとして残っている。

## Decision

**`e2e/playwright/<svc>/run.sh` を Hurl dispatch 規約 ([[000766]]) と並ぶ 2 本目の framework dispatch として確立する。** `alt-frontend-sv` を最初の Playwright pacticipant として `e2e/playwright/alt-frontend-sv/run.sh` + `README.md` を導入し、alt-deploy `release-deploy.yaml` e2e step を two-directory probe に拡張する後続 PR で blocking gate 化を完成させる。

### 採用構成

1. **`e2e/playwright/alt-frontend-sv/run.sh`** — Hurl `run.sh` と ergonomics 互換 (`set -euo pipefail` + `RUN_ID` / `REPORT_DIR` / `KEEP_STACK` env 規約)、追加で Playwright 固有の `PLAYWRIGHT_IMAGE` / `BUN_VERSION` / `SHARD` / `PROJECTS` を受け付ける。本体は `docker run --rm --network host -v $ROOT:$ROOT -w $ROOT/alt-frontend-sv $PLAYWRIGHT_IMAGE bash -euxo pipefail -c "…"` で、内部 sequence を `.github/workflows/alt-frontend-sv.yml` と **step-for-step 一致** させる (apt retry → bun install → `bun install --frozen-lockfile` → `bunx playwright test --project=auth --project=desktop-chromium --project=mobile-chrome --shard=${SHARD}`)。

2. **Stack は mock self-contained** — `playwright.config.ts` の webServer が `bun run build && node build` を 4174 で起動し、`tests/e2e/global-setup.ts` が Kratos (4001) / AuthHub (4002) / Backend (4003) を test プロセス内で立てる既存設計をそのまま採用。`compose.staging.yaml` には profile を追加しない。`--network host` で 4 port (4001-4003, 4174) を container と共有する。

3. **Project 範囲は `auth` + `desktop-chromium` + `mobile-chrome` の 3 つ** — `alt-frontend-sv.yml` と同一。`desktop-webkit` / `mobile-safari` / `visual-regression` / `integration` は本 ADR の scope outside。

4. **SHARD env で matrix 対応** — default `1/1` (local)、alt-deploy matrix では `matrix.include` で alt-frontend-sv のみ `1/3, 2/3, 3/3` の 3 leg を展開する。regex ガード `^[0-9]+/[0-9]+$` で不正値を exit 2 で fail-close。

5. **alt-deploy 側 `release-deploy.yaml` の two-directory probe** — 本 ADR landed 後の別 PR (private repo) で以下のように拡張する:

   ```yaml
   - name: Run E2E suite for ${{ matrix.service }}
     run: |
       if   [ -f "e2e/hurl/$SERVICE/run.sh" ];       then bash "e2e/hurl/$SERVICE/run.sh"
       elif [ -f "e2e/playwright/$SERVICE/run.sh" ]; then bash "e2e/playwright/$SERVICE/run.sh"
       else echo "skip: no E2E dispatch for $SERVICE"
       fi
   ```

   resolve job の matrix source が `alt-frontend-sv` を認識するよう path filter 拡張が必要な場合も別 PR で追随する。

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| Hurl suite `e2e/hurl/alt-frontend-sv/` を新設して既存規約にそのまま載せる | Hurl は HTTP + Connect-RPC only。SvelteKit の auth flow / swipe gesture / tag-verse canvas / streaming summary は browser engine 必須で Hurl では再現不能。無理やり乗せると signal が薄い health-check だけの gate になる |
| `compose.staging.yaml` に `alt-frontend-sv` profile + `alt-frontend-sv-deps-stub` (butterfly-facade / auth-hub / backend mock) を [[000779]] パターンで新設し real stack で回す | mock 3 つで既に 43 spec 全 green が安定している事実に対する追加 signal が薄い。compose bring-up ~90s + teardown ~30s × 3 shard の cold-start cost が累積、self-hosted runner 10 台の docker daemon 負荷も増す。将来 post-deploy integration smoke が必要になった時点で別 ADR で扱う |
| `dispatch-deploy.yaml` を `workflow_run` trigger に昇格させ PR Playwright 結果を待つ | `workflow_run` は前段 workflow を OR 指定なので `alt-frontend-sv.yml` と `docker-build.yaml` 両方成功の AND を取れない。`actions/github-script` polling 実装も可能だが、同一 SHA で rerun があった時の race や permission のフットプリントが増える。alt-deploy 側 matrix 追加で blocking gate は既に十分に機能する |
| Playwright を local 直接実行 (docker 不使用) | self-hosted runner ごとの bun / node / playwright browser binaries 差分が gate の reproducibility を壊す。CI container を runner toolchain から分離する現行 `alt-frontend-sv.yml` 設計と一致させる方が安全 |
| `desktop-webkit` / `mobile-safari` / `visual-regression` / `integration` も同時に gate 入り | WebKit binary install + snapshot baseline 運用 + `ALT_RUNTIME_URL` 配線が各々独立の scope outside。後続 ADR で段階的に追加 |

### 範囲外 (今回やらないこと)

- alt-deploy private repo `release-deploy.yaml` の two-directory probe + matrix 拡張 — 別リポ PR
- `compose.staging.yaml` への `alt-frontend-sv` profile 追加 — 将来 real stack smoke で扱う別 ADR
- `desktop-webkit` / `mobile-safari` / `visual-regression` / `integration` project の gate 入り
- `dispatch-deploy.yaml` workflow_run / polling 化
- Pact pacticipant への frontend 追加 — 契約層の再設計が前提で別 ADR
- `e2e/_lib/common.sh` への Hurl / Playwright 共通シェル関数抽出 — 2 本目の Playwright 規約が出る段階まで保留 ([[000766]] 同旨)

## Consequences

### Pros

- **deploy pipeline で frontend E2E gate が成立する** — alt-deploy matrix が `alt-frontend-sv` leg で 43 spec 全緑を要求するようになり、mobile citation overflow / swipe 回帰等のクラッシュを本番到達前に捕捉できる
- **PR 時点と deploy 時点で同一 sequence** — `run.sh` 内の container / bun / playwright step を `alt-frontend-sv.yml` と step-for-step 一致させたため CI green / deploy RED (またはその逆) の乖離が構造的に起きない
- **規約が framework 次元で拡張可能** — `bash e2e/<framework>/<svc>/run.sh` 1 行 dispatch が `hurl` / `playwright` の 2 framework を同居させる。今後 Cypress / k6-browser 等を入れる時も `e2e/<framework>/<svc>/run.sh` だけ追加すれば matrix 側の変更は two-directory probe を three-directory probe に拡張するのみ
- **既存 mock infra の再利用** — `playwright.config.ts` webServer + `global-setup.ts` が既に 1 年以上稼働しており、追加の compose profile / deps-stub / 新規 fixture を導入せず deploy gate が完成する
- **scope 小さく reviewable** — public 側 changeset は run.sh + README + ADR の 3 ファイルで、実装リスクと review 面積を最小化

### Cons / Tradeoffs

- **mock self-contained のため Pact CDC 相当の contract 検証は持たない** — mock が本番 butterfly-facade / backend と乖離した時に検知できない。frontend は Pact pacticipant に含まれないため、この gap を埋めるには post-deploy integration smoke (`integration` project + `ALT_RUNTIME_URL`) を後続 ADR で追加する必要がある
- **alt-deploy 側 PR が landed するまで gate は未完成** — 本 ADR commit 直後の状態は「public 側に run.sh / README / ADR 揃い、但し alt-deploy e2e step は Hurl のみ probe する」という中間状態。後続 PR が遅れれば gate 機能は開通しない
- **shard 並列で alt-deploy matrix が非対称になる** — 13 pacticipant + alt-frontend-sv 3 shard で matrix が 14 base × shard 付きは alt-frontend-sv のみ、の非対称 shape。`matrix.include` の書き方に注意が要る (Decision 4 参照)
- **Playwright container image ~2GB の初回 pull cost** — self-hosted runner に layer cache が無い場合 pull で数分を費消。ephemeral runner 運用になった時は GHCR mirror / registry cache の検討が必要
- **report artifact 容量** — 3 shard × 43 spec の HTML report + trace video で最大 ~600MB / run。retention 30 日で累積負荷が発生、`KEEP_REPORT` 的 gate で絞る運用フォロー余地あり
- **`global-setup.ts` の mock 3 server が test プロセスに埋まっている** — 新規 mock の追加や pinning の更新時に container 側の挙動と乖離する可能性があり、dispatch と CI で同時に壊れ得る。`--network host` 依存も host OS を Linux に限定する暗黙前提として残る
