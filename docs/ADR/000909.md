# ADR-000909: Knowledge Loop に Lens selector を first-class UI control として配置し macro byline で multi-scale 認知状態 (continuing / to review / internalized) を可視化する

## Status

Accepted

## Date

2026-05-20

## Affected Services

- alt-frontend-sv (TypeScript / Svelte 5)
- knowledge-sovereign (Go)
- alt-backend (Go)

## Context

Knowledge Loop v2 plan §I-§II は次の 2 つの構想 delta を未実装として残している:

1. **Δ2: Multi-scale loop (micro / meso / macro)** — OODA は秒〜分の戦術ループ、知識活動は日〜週のスパン。現行 contract はこれを 1 層に潰している。Day〜week レベルの macro state (`activeContinueThreads`, `pendingReviewCount`, `recentInternalizedCount`) が UI に存在しない。
2. **Δ5: Lens mode を first-class UI control に格上げ** — `lens_mode_id` は proto / DB column / JWT claim にあるが、UI 上で切替できる primary control が無い。Lens は **認知モードの宣言** (research / browse / decide / recall) で、`surfaceBucket` の重み付けを切替えるべき入力。

両方とも構想層 §3 (multi-scale) と §4 (lens primacy) に対応し、Knowledge Loop が「**情報の入口ではなく認知の出口を作る**」という北極星に到達するための前提。

ADR-000908 で `macroState` field を `KnowledgeLoopSessionState` に additive 追加し、`KnowledgeLoopLensModeSwitched` event は既に [[000855]] 周辺で proto に存在する。本 ADR は **UI 側と projector 側を配線して macroState / Lens を実際に流す** ことを担当する。

## Decision

UI 側 LensSelector + macro byline と、projector 側の lens_mode_weights / macroState aggregation を 1 PR で landed する。

### Lens selector (UI)

`alt-frontend-sv/src/lib/components/knowledge-loop/LensSelector.svelte` を新規追加:

```
┌──────────────────────────────────────────────────────────┐
│  Knowledge Loop                                          │
│  Lens: [● Research ─ Browse ─ Decide ─ Recall ]   Seq …  │
└──────────────────────────────────────────────────────────┘
```

- 4 segment control: `Research / Browse / Decide / Recall`
- Newspaper Style (`IBM Plex Mono`, en-dash separator, `var(--alt-primary, #2f4f4f)` baseline、selected は `1.5px` border + `background: rgba(47, 79, 79, 0.08)`)
- click で `/loop/transition` endpoint に `trigger="lens_switch"` の transition を post し、backend が `KnowledgeLoopLensModeSwitched` event を append
- URL query param `?lens=<id>` と sync (browser back で前 lens に戻れる)
- ADR-000903 の `/loop/welcome` で初期 lens を選択するため、初回訪問者は必ず lens=非デフォルト

### Lens-weighted bucket (projector)

`decide_bucket_v2.go` の `SurfaceScoreInputs` に `lens_mode_weights` を additive 追加。lookup table は immutable constant:

| Lens | Now 重視 | Continue 重視 | Changed 重視 | Review 重視 |
|---|---|---|---|---|
| Research | medium | high | high | medium |
| Browse | high | low | low | low |
| Decide | high | high | high | low |
| Recall | low | low | low | high |

`computeBucket(inputs)` 内で base score に lens 別係数を掛けて placement を decide。pure function を維持 (同じ入力 + lens → 同じ bucket、reproject-safe)。

### Macro byline (UI + projector)

`MacroByline.svelte` を新規追加し、byline に 1 行 append:

```
Lens: Research — Stage: Orient — Seq: 1208           [byline 既存]
3 continuing — 2 to review — 5 internalized this week  [macro new]
```

数値は `session_state.macroState` (ADR-000908 で additive 追加) から派生。

projector 側 `macro_state_builder.go` (新規) を pure function として実装:

```go
type MacroState struct {
    ActiveContinueThreads     int
    PendingReviewCount        int
    RecentInternalizedCount   int
    CognitiveLoadHint         string // "light" | "medium" | "heavy"
}

func BuildMacroState(events []KnowledgeEvent, windowEnd time.Time) MacroState {
    // 7d window で:
    // - continueFlag=true の KnowledgeLoopActed event を entry_key で dedupe → activeContinueThreads
    // - dismissState transition で internalized 化された entry → recentInternalizedCount
    // - staleness/contradiction/version_drift signals で Review に配置された entry → pendingReviewCount
    // - 上記 3 値から派生指標 cognitiveLoadHint (light: < 3, medium: 3-8, heavy: > 8)
}
```

`windowEnd` は event-time bound (`MAX(event.occurred_at)` から `7d`)。`time.Now()` を business fact に使わない。

session_state row の upsert 時に macroState を recompute し JSONB として store。reproject 経路と新規 event 受信経路で同一ロジック (pure function 再利用)。

### 検討した代替案

- **Lens selector を hamburger menu の下に置く** — 構想層 §4-Δ5 「first-class control」と矛盾。primary header に固定。
- **Lens を 6 種以上に拡張** — Research / Browse / Decide / Recall の 4 種で MVP。`Synthesis` / `Critique` 等の追加 lens は構想を稼働させてから後続 ADR で additive。
- **macro byline を sidebar に出す** — modal-less / always-visible が認知状態の "全体" を伝えるのに最適。byline に折り込み。
- **macroState 計算を read time で都度実行** — N+1 query で performance 退行。session_state row に store する方が GET レスポンス即応。reproject 時は session_state を rebuild すれば追従。
- **`cognitiveLoadHint` を proto 必須にする** — UI 側で派生計算する選択肢もあるが、`light/medium/heavy` の閾値が contract レベルで定まっている方が consumer 間で挙動が一致する。proto optional で store、UI は presence を check。

## Consequences

### Pros

- 構想層 §3 (multi-scale) と §4 (lens primacy) が UI と projector の両側で landed (Knowledge Loop が認知 OS として最終 form に到達)
- Lens 切替で placement が変わる体験で、ユーザは「Loop に自分の認知モードを教える」ことの効果を即座に感じられる
- macro byline で「Loop の全体状態」が一望でき、`/loop` を開いて何件 actionable か即判断可能
- `lens_mode_weights` は immutable constant + pure function なので reproject 経路で同一性が保たれる
- `macroState` を session_state row に store することで GET レスポンス即応 (sidebar / byline render が即時)

### Cons / Tradeoffs

- LensSelector が画面上部の有限なスペースを占有 (mobile 特に)。`/loop/welcome` の Lens 選択と一貫したデザインで、segment control の高さを 40px 程度に抑える。
- `lens_mode_weights` table の調整は projector 経路で deterministic だが、調整時に `SurfacePlannerVersion` を bump して full reproject runbook を回す必要がある。
- macroState の 7d window は event log を毎回 query するため projector lag が立ち得る。`event_dropped_total` / `knowledge_loop_projector_lag_events` (Phase H) で監視。
- "cognitive load heavy" の閾値判定が誤検知すると UX で誤誘導するリスク。MVP は閾値固定 (`light < 3`, `medium 3-8`, `heavy > 8`) で、user feedback で調整する後続 ADR を予約。
