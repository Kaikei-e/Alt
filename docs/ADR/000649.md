# ADR-000649: rag-orchestrator に LLM_BACKEND=eino の実配線と Eino ツール契約の整合を導入する

## Status

Accepted

## Date

2026-04-08

## Affected Services

- rag-orchestrator (Go) - LLM backend の切替、ToolCalling のアダプタ、tool dispatch の整合修正

## Context

[[000647]] で Eino の基盤を導入したが、実装は足場止まりで、`LLM_BACKEND=eino` を指定しても実運用の経路として選択できなかった。加えて、Eino 側の tool schema と実ツールの引数名が一致しておらず、dispatcher も全ツールに同じ引数名を渡していたため、tool calling の成功率が実装上の契約より低くなる状態だった。

このままでは、Eino を「導入済み」と見なせず、実際には従来の Ollama 経路に依存したままになる。ADR-000647 の意図である「段階的な移行」とも整合しないため、backend 切替、tool 契約、dispatch の 3 箇所を同時に揃える必要があった。

## Decision

rag-orchestrator に対して、以下の方針で Eino を本番経路として選択可能にする。

1. `LLM_BACKEND` を設定値として追加し、`ollama` を既定、`eino` を明示選択肢として扱う。
2. `LLM_BACKEND=eino` の場合は Eino の chat model adapter を生成し、生成に失敗した場合は既存の Ollama 経路へフォールバックする。
3. Eino の tool schema をツールごとに個別定義し、`topic` や `tag_name` のような実引数名と一致させる。
4. tool dispatcher でも同じ契約を使い、tool 名ごとに適切なデフォルト引数を組み立てる。

この決定により、Eino は「将来の置き換え候補」ではなく、設定で切り替えられる実際の backend になる。ただし、既存の Ollama 経路は安全網として残し、段階移行を維持する。

## Implementation

- `rag-orchestrator/internal/infra/config` に `LLMBackend` を追加し、Compose から注入できるようにした。
- `rag-orchestrator/internal/di/container.go` で backend を分岐し、`eino` 指定時は Eino adapter を生成するようにした。
- `rag-orchestrator/internal/adapter/eino` に tool schema の共通化ロジックを置き、JSON 文字列の tool call 引数を domain 側の map に戻す処理も追加した。
- `rag-orchestrator/internal/usecase/tool_dispatcher.go` を修正し、全ツールへ `query` を押し込む実装をやめ、ツール名に応じた引数名を使うようにした。
- `compose/rag.yaml` に `LLM_BACKEND` を渡す設定を追加した。
- 変更後に `rag-orchestrator` を再ビルド・再起動し、Compose 上で `rag-orchestrator` と `rag-db` が `Up` / `healthy` であることを確認した。
- `cd rag-orchestrator && go test ./...` で Go のユニットテストを通した。

## Consequences

### Pros

- `LLM_BACKEND=eino` が実際に選択できるようになり、コード変更なしで backend を切り替えられる。
- tool schema と実引数の不一致が解消され、tool calling の失敗要因を減らせる。
- dispatcher と adapter が同じ契約を共有するため、将来の tool 追加時も壊れ方が分かりやすい。
- Ollama フォールバックを残したことで、移行途中でも運用を止めにくい。

### Cons / Tradeoffs

- Eino の経路はまだ既存の LLM/tool 呼び出し surface に限定され、compose.Graph を全面採用する段階には至っていない。
- backend 切替と tool 契約の保守点が増えるため、今後も config・adapter・dispatcher の整合性確認が必要になる。
- フォールバックを残す設計のため、Eino 側の不具合が即停止ではなく静かな退行として見えにくい。監視とログの確認が必要になる。
