# ADR-000324: E2E テストスイートの体系的拡張と POM ベストプラクティス適用

## ADR's STATUS

完了

## CONTEXT

`alt-frontend-sv` の E2E テストスイートは 25 個の spec ファイルと 13 個の Page Object を持っていたが、以下の課題があった:

1. **カバレッジの欠落**: Tag Trail・Evening Pulse・Morning Letter・3-Day Recap・Favorites・Viewed・Stats など 9 ルートに E2E テストが存在しなかった
2. **POM パターンの不統一**: 9 個の spec ファイルが POM を使わず `page.getByXxx()` を直接呼び出しており、テストの保守性が低かった
3. **Fixture 基盤の不在**: Playwright のカスタム Fixture（`base.extend<T>()`）が未導入で、POM のインスタンス化が各 spec に分散していた
4. **共有 UI コンポーネントの POM 不在**: サイドバー・フローティングメニューといった横断的 UI のテスト用抽象が存在しなかった
5. **アクセシビリティテストの網羅不足**: a11y spec が主要ルートのみを対象としており、新規ルートがカバーされていなかった

## DECISION MAKING

Playwright 公式の POM ベストプラクティスに準拠し、6 フェーズに分けて体系的に拡張するアプローチを採用した。

### Phase 0: 基盤整備

- **`BasePage.waitForLoad()`** の待機戦略を `networkidle` から `domcontentloaded` に変更。各 POM が個別に `waitForXxxLoaded()` で要素の可視性を待機する設計に統一
- `BasePage` のアサーションメソッド（`expectVisible` 等）を `@deprecated` に指定。POM はロケータとアクションを公開し、アサーションはテスト側が担う原則を明示
- **カスタム Fixture (`pomFixtures.ts`)** を新設。`base.extend<PomFixtures>()` で全 POM を遅延インスタンス化し、spec からは `{ test, expect }` をインポートするだけで POM を利用可能に
- **テストデータファクトリ** を `pulseFactory.ts`・`tagTrailFactory.ts` として追加。既存の `feedFactory`・`recapFactory`・`sessionFactory` と同じパターンで統一
- **モックデータ (`mockData.ts`)** に Evening Pulse・Tag Trail・Trend Stats のパス定数とレスポンスデータを追加

### Phase 1: 共有コンポーネント POM

- **`SidebarComponent`**: デスクトップサイドバーの `navigateTo()`・`toggleSection()`・`getAllLinks()` を提供
- **`FloatingMenuComponent`**: モバイルフローティングメニューの `open()`・`close()`・`navigateTo()` を提供

### Phase 2: 新規 POM の作成（16 個）

全 POM が `BasePage` を継承し、`readonly` ロケータをコンストラクタで初期化、`getByRole`・`getByTestId`・`getByText` による堅牢なセレクタを使用。アサーションメソッドは持たない。

### Phase 3: 既存 spec の POM 化（9 ファイル）

raw な `page.getByXxx()` 呼び出しを POM のロケータとアクションに置換し、`pomFixtures.ts` からのインポートに統一。

### Phase 4: 新規 E2E spec の作成（9 ファイル）

未カバーの全ルートに対してテストを追加。各 spec は Phase 2 で作成した POM と既存のモックヘルパー（`fulfillJson`・`fulfillConnectStream`・`fulfillError`）を活用。

### Phase 5: アクセシビリティテストの拡張

`desktop.a11y.spec.ts`・`mobile.a11y.spec.ts` に合計 8 ルート分のテストブロックを追加。axe-playwright による WCAG 2.1 AA 準拠チェックを新規ルートにも適用。

## RESULTS, EFFECTS

### PROS

- **ルートカバレッジの向上**: E2E テスト対象ルートが約 16 から約 25 に増加し、全主要ルートをカバー
- **保守性の向上**: 全 34 spec ファイルが POM パターンを使用（100%）。UI 変更時の修正箇所が POM に集約される
- **テスト記述の簡潔化**: カスタム Fixture により、POM のインスタンス化コードが不要になり、テストの可読性が向上
- **堅牢なセレクタ**: 全 POM が `getByRole`・`getByTestId`・`getByText` を使用。CSS クラスセレクタに依存しないため、スタイリング変更に強い
- **a11y カバレッジ**: WCAG 2.1 AA チェック対象が Tag Trail・Favorites・Viewed・Stats・Evening Pulse・Morning Letter・3-Day Recap に拡大
- **既存テストへの無影響**: `BasePage` の変更は後方互換性を維持。既存テストに回帰なし

### CONS, TRADEOFF

- **POM の増加による初期学習コスト**: POM が 13 から 31 に増加（+16 POM + 2 コンポーネント）。新規参加者がテスト基盤を理解するまでの時間が増加する
- **Fixture の型定義の維持**: `pomFixtures.ts` に新しい POM を追加する際、型定義と Fixture 登録の両方を更新する必要がある
- **モックデータの肥大化**: `mockData.ts` が拡大しており、将来的にはドメイン別ファイル分割を検討する余地がある

### 設計原則

- **POM はロケータとアクションのみ公開**: アサーションはテスト側の責務
- **セレクタの優先順位**: `getByRole` > `getByTestId` > `getByText` > CSS セレクタ（使用しない）
- **待機戦略**: `domcontentloaded` + 要素の可視性待機（`networkidle` は使用しない）
- **モックヘルパーの一元化**: `fulfillJson`・`fulfillConnectStream`・`fulfillError` を共有ユーティリティとして使用
