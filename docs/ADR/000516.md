# ADR-000516: フィード統計を認証ユーザー単位で集計する

## Status

Accepted

## Date

2026-03-21

## Affected Services

- alt-backend (Go)
- alt-frontend-sv (SvelteKit)

## Context

stats 画面および feed stats API は、認証済みユーザー向けの画面であるにもかかわらず、要約件数や記事件数の集計で `user_id` を使っていなかった。
このため単一ユーザー運用では見えにくいものの、設計上は全体件数を返す実装になっており、複数ユーザー環境では他ユーザーのデータを含む数値が表示される危険があった。

記事取得系や一覧取得系の driver では既に `domain.GetUserFromContext` を使って `user_id` で絞り込むパターンが存在していた一方、feed stats 系の count driver だけがこの規約から外れていた。
認証済み UI が返す件数の意味を一貫させるには、feed stats でも同じ user context を母集団の境界として採用する必要があった。

## Decision

**feed stats の count 集計は、すべて認証ユーザーの `user_id` を必須条件として実行する。**

具体的には以下を採用する。

1. total articles count は `articles.user_id = current_user` で集計する
2. unsummarized articles count は `articles.user_id = current_user` を前提に集計する
3. summarized articles count は `article_summaries.user_id = current_user` で集計する
4. user context が取得できない場合は統計取得を失敗として扱う

この決定により、統計 API は「認証済みユーザー自身の件数」を返す契約になる。

### 採用しなかった代替案

| 案 | 評価 |
|---|---|
| 単一ユーザー前提なので全体件数のままにする | 将来の multi-tenant 対応時に表示と認可の意味が崩れる |
| frontend 側で user ごとに補正する | 集計の責務が API から漏れ、複数経路で不整合が起きる |
| 認証 UI とは別に global stats API を増やす | 今回の問題を解決するには過剰であり、要求もない |

## Consequences

### Pros

- stats API の意味が認証境界と一致する
- 複数ユーザー環境でも他ユーザー分を誤って表示しにくくなる
- count driver も既存の user-scoped driver 群と同じ設計規約に揃う

### Cons / Tradeoffs

- user context が欠けた呼び出しは、従来より厳密に失敗する
- バックフィルや運用確認で global count が必要な場合は別クエリが必要になる
- 既存の「単一ユーザーだから全体件数でもよい」という暗黙前提は使えなくなる
