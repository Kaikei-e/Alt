# ADR-000490: Knowledge Home の stream 配信観測と fallback UI 文言を整合させる

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-backend (Go)
- alt-frontend-sv (SvelteKit)

## Context

Knowledge Home の Phase 0 では、stream 接続数だけでなく「実際に更新を何件届けたか」を観測できることが必要だった。一方で [[000485]] の時点では、接続数と lag 計測は扱われていたが、配信件数を明示的に数える counter が不足していた。

同時に、フロントエンドの `fallback` 表示文言には「cached snapshot を表示している」と受け取れる表現が残っていた。しかし現在のバックエンド実装では、`fallback` は「一部セクションを落としつつ継続表示する状態」を含み、必ずしもキャッシュ済みスナップショットのみを意味しない。このままだと、UI の説明文が backend contract より強い意味を持ってしまう。

そのため、運用観測と UI 文言の両方を、実装済みの Knowledge Home 契約に合わせて揃える必要がある。

## Decision

Knowledge Home の stream observability と fallback UI 文言を、現在の契約意味論に合わせて整合させる。

- backend では、non-heartbeat の successful stream send ごとに `alt_home_stream_deliveries_total` を加算する counter を採用する
- stream 接続数や切断数とは別に、配信件数を独立した指標として持つ
- frontend では、`fallback` を「一部セクションが一時的に利用不能または stale な状態」と説明する英語文言に変更する
- `DegradedModeBanner` と `TodayBar` の両方で、`fallback` を cached snapshot 固定で説明しない

今回の実装では、以下の方針を採用する。

- heartbeat は配信件数に含めない
- 文言はユーザーに実装詳細を漏らさず、部分利用不能と stale の両方を吸収できる中立表現にする
- UI 言語は英語のまま維持する

代替案として「lag histogram のみで配信を代用計測する」「UI 側の文言は据え置き、backend の意味論だけ合わせる」も検討したが、前者は配信総数を直接読めず、後者は運用と UI の説明責任が分離してしまうため採用しない。

## Consequences

### Pros

- stream 配信量を接続数と分離して把握でき、更新停滞の切り分けがしやすくなる
- `fallback` 文言が backend contract と一致し、ユーザーへの説明が過剰に断定的でなくなる
- TodayBar と banner の両面で同じ意味論を使えるため、Knowledge Home の UI 一貫性が上がる

### Cons / Tradeoffs

- メトリクスが1本増えるため、ダッシュボードやアラート設計の更新が必要になる
- `fallback` 文言を中立化した分、以前より具体的に見えにくい
- 将来的に fallback の原因種別を UI に出し分けたくなった場合は、追加フィールドか別契約が必要になる
