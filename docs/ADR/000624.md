# ADR-000624: CachedFeedListUsecase の fan-out N+1 パターンを除去し単一 SQL 委譲に切り替える

## Status

Accepted

## Date

2026-04-06

## Affected Services

- alt-backend (Go) — usecase / DI 配線 / driver
- alt-frontend-sv (TypeScript/Svelte) — mutation の invalidation 戦略

## Context

デスクトップ UI でフィードの既読操作が間欠的に非常に遅くなる問題が報告された。

コンテナログの計測で以下の実測値を確認した:

| メトリクス | 通常 | ピーク |
|---|---|---|
| MarkAsRead (nginx rt) | 4–20 ms | **634 ms** |
| GetUnreadFeeds (usecase_ms) | 2–18 ms | **424 ms** |

### 原因分析

`CachedFeedListUsecase.loadMergedFeeds()` が fan-out N+1 パターンでフィード一覧を構築していた:

1. `GetUserSubscriptions` → 1 DB クエリ（N 個のサブスクリプション ID を返す）
2. N 個の `GetFeedPage` を並列 DB クエリ（max 8 concurrent、各最大 200 行）
3. 全結果をメモリ上でマージ・ソート
4. `GetAllReadFeedIDs` → 1 DB クエリ（LIMIT なし、全既読 ID を返す）
5. メモリ上で既読フィルタ → cursor + limit → 最終 20 件を返却

`SharedCache`（TTL=2 分、stale=1 分）が同時に expire すると N 個の並列 DB クエリが PgBouncer（DEFAULT_POOL_SIZE=30）に殺到し、同時実行中の MarkAsRead がコネクション待ちでブロックされる。

### 核心的な問題

効率的な単一 SQL クエリが `FetchUnreadFeedsListCursor`（NOT EXISTS anti-join + cursor pagination + LIMIT）として既に driver 層に存在するが、ハンドラは `CachedFeedListUsecase` を経由していたため、同じ処理を N+1 クエリ + Go メモリ上で再実装していた。

## Decision

### 1. CachedFeedListUsecase を単一 SQL ポートへの委譲に切り替える

`FetchUnreadFeedsListCursor` と `FetchAllFeedsListCursor` を既存の効率的な SQL ポート（`UnreadFeedCursorPort` / `FeedCursorPort`）に直接委譲する。`loadMergedFeeds` とその関連コード（`applyCursor`、`convertFeedPageEntries`、`feedIDToUUID` 等 150 行超）を除去し、usecase を 93 行 → 28 行に簡素化する。

コンストラクタのパラメータを 5 → 3 に削減（`feedPageCache`、`userReadState` を除去）。

### 2. GetAllReadFeedIDs に LIMIT を追加する

`SELECT feed_id FROM read_status WHERE user_id = $1 AND is_read = TRUE` に `ORDER BY read_at DESC LIMIT 10000` を追加し、無制限のレスポンスを防御的に制限する。ポートインターフェースのシグネチャは変更せず、driver 内定数として定義する。

### 3. フロントエンドの invalidation cascade を削減する

MarkAsRead mutation の `onSettled` で発火していた 3 リクエスト（stats / unreadCount / read）を最小化する:

- `unreadCount` を `onMutate` でオプティミスティックに -1（サーバー往復不要に）
- `stats` の invalidation を除去（staleTime による自然更新に委ねる）
- `read` リストの invalidation を `refetchType: 'active'` に変更（表示中のみ refetch）

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|---|---|---|
| SharedCache TTL 延長（2 分 → 10 分） | 低コスト | 根本原因（fan-out）を解消しない。cache miss 時の劣化は同じ |
| PgBouncer pool size 増加 | 低コスト | 症状の治療。クエリ数が問題でありプールサイズではない |
| Redis 導入 | 中コスト | DB クエリ自体は 2–18 ms で十分高速。インフラ複雑化に見合わない |
| Materialized View | 中コスト | 既存 SQL + インデックスで十分な性能。MV の更新管理コストが過大 |

## Consequences

### Pros

- GetUnreadFeeds のピークレイテンシが **424 ms → 2–18 ms** に改善（一定した性能、bimodal 分布の解消）
- MarkAsRead のピークレイテンシが **634 ms → 4–20 ms** に改善（コネクション競合の解消）
- DB コネクション消費が **1 リクエストあたり N+2 → 1** に削減
- フロントエンドの MarkAsRead 後のサーバーリクエストが **3 → 0–1** に削減
- usecase のコード量が 93 行 → 28 行に大幅簡素化（保守性向上）

### Cons / Tradeoffs

- SharedCache のキャッシュヒット時（2 ms 以下）と比較すると、常に SQL を叩くため定常レイテンシは微増（2–18 ms）。ただし cache miss 時の 424 ms スパイクの解消が大幅に上回る
- `loadMergedFeeds` を利用していた他のコードパスが将来追加された場合、新たに実装が必要になる。現時点では呼び出し元が存在しないため影響なし
- `GetAllReadFeedIDs` の LIMIT 10000 は理論上、極端に多くのフィードを既読にしたユーザーで一部の古い既読状態が欠落する可能性がある。現実的なユーザー行動では到達しない閾値
