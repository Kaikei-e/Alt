# ADR-000642: rag-orchestrator に Ollama ネイティブ tool-calling ループを導入する

## Status

Accepted

## Date

2026-04-08

## Affected Services

- rag-orchestrator (Go)

## Context

[[000617]] で `IntentSynthesis` に Tool-Use Agentic RAG を導入したが、当初の実装は `ToolPlanner` が LLM で JSON の実行計画を生成し、`ToolDispatcher` がそれを実行する単発型のパイプラインだった。

この方式でも概念的な合成クエリには一定の改善があった一方で、以下の課題が残っていた。

1. **追加調査の自律性が低い**
   - 最初の計画に含めなかった観点は、その時点で取りこぼされる
   - 回答途中で「足りない証拠がある」と判断して追加ツールを呼び出すことができない

2. **ツール実行と生成が分離しすぎている**
   - LLM が工具利用の結果を見ながら次の一手を決める、という agentic な流れになっていない
   - そのため、検索・タグ探索・関連記事取得のような補助情報を段階的に深める設計が弱い

3. **Ollama 側の native tool-calling を使っていない**
   - `/api/chat` は `tool_calls` を返せるが、既存実装ではそれを受け取らず、単なるテキスト応答として扱っていた
   - 結果として、Ollama が持つ標準的な tool-calling の対話形式を活かせていなかった

4. **重複 tool call の扱いが未整備**
   - LLM が同じ tool を繰り返し要求する場合の抑止や、最終出力への二重計上防止が必要だった

この変更は、既存の `IntentSynthesis` を壊さずに agentic な再帰実行を追加し、複数段階で証拠を集められるようにするために必要だった。

## Decision

**`IntentSynthesis` の synthesis パイプラインに、Ollama native tool-calling による agent loop を追加する。**

具体的には、以下を採用する。

1. `OllamaGenerator` が `/api/chat` の `tools` と `tool_calls` を扱えるようにする
2. `ToolDispatcher` から Ollama 用の tool schema を生成できるようにする
3. `SynthesisStrategy` の上位に `AgenticSynthesisStrategy` を置き、初期 retrieval のあとに tool-calling loop を最大数回だけ回す
4. LLM が要求した tool を実行し、その結果を tool メッセージとして会話に戻し、必要なら追加 tool call を受け付ける
5. 同一 tool call の重複実行と、最終補足情報への二重計上を防ぐ

### 採用しなかった案

| 案 | 内容 | 判断 |
|---|---|---|
| 単発の ToolPlanner のみを維持する | 既存の JSON 計画生成だけで終える | 追加の証拠取得を LLM が自律制御できず、agentic 性が弱い |
| 外部の agent framework を導入する | LangGraph 等に寄せる | 依存が増え、Go/Python 混在のデバッグと運用が重くなる |
| 全 intent を agent loop 化する | `General` や `Temporal` まで一律で tool loop を回す | レイテンシとコストが過剰になるため、`IntentSynthesis` に限定する方が妥当 |

## Consequences

### Pros

- LLM が tool 結果を見ながら次の調査を選べるため、合成的なクエリへの追従性が上がる
- `related_articles` や `tag_search` のようなツールを、初期計画に固定せず段階的に追加できる
- Ollama の標準的な `tool_calls` 形式を使うため、将来のモデル更新や tool 追加に対して拡張しやすい
- 重複 tool call の抑止と二重計上防止を入れたことで、出力の安定性を確保しやすい

### Cons / Tradeoffs

- tool loop が入ることで `IntentSynthesis` のレイテンシは増える
- LLM の tool call 品質に依存するため、モデルの挙動が不安定な場合はループが短く終わるか、逆に無駄な再試行が増える
- tool schema のメンテナンスが必要になる
- `tool_calls` 対応は `rag-orchestrator` 内の domain / adapter / usecase をまたぐため、変更範囲が広い

## Implementation Notes

### 1. domain に tool-calling の型を追加した

`Message` に `name` / `tool_call_id` / `tool_calls` を追加し、`LLMResponse` に `ToolCalls` を追加した。
あわせて `ToolDefinition`, `ToolDescriptorFn`, `ToolCall`, `ToolCallFunction` を定義し、Ollama の `tools` 入出力をそのまま表現できるようにした。

### 2. OllamaGenerator を chat tool-calling 対応にした

`OllamaGenerator` の `/api/chat` リクエストに `tools` を載せ、レスポンスから `tool_calls` を復元できるようにした。
これにより、`ChatWithTools` で native tool-calling を使えるようになった。

### 3. ToolDispatcher に schema 出力と named execute を追加した

`ToolDispatcher.ToolDefinitions()` で Ollama 用の tool schema を生成し、`ExecuteNamed()` で個別 tool の実行を行えるようにした。
既存の parallel execution もそのまま再利用し、tool plan を既存の実行基盤へ流し込めるようにしている。

### 4. AgenticSynthesisStrategy を新設した

`SynthesisStrategy` の既存の retrieval をベースにしつつ、`ChatWithTools` を最大 3 回だけ回す agent loop を追加した。
LLM が tool を要求したら実行し、結果を `tool` メッセージとして戻し、次の判断に使う。
同じ tool call は正規化した key で抑止し、最終的な `SupplementaryInfo` / `ToolsUsed` への追加も重複しないようにした。

### 5. DI を synthesis agentic に切り替えた

`IntentSynthesis` の登録先を `AgenticSynthesisStrategy` に変更し、既存 intent の挙動を壊さずに synthesis 専用パスだけを強化した。

### 6. テストを追加した

agent loop が tool call を解釈して実行すること、同じ call を繰り返したときに重複計上しないことを単体テストで確認した。
あわせて `rag-orchestrator` 全体の `go test ./...` で回帰がないことを確認した。
