# ADR-000503: 共有 feed source と user subscription の削除責務を分離する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-backend (Go)

## Context

RSS ドメインでは `feed_links` が共有 source registry、`feeds` と `articles` がその source に紐づく共有コンテンツ、`user_feed_subscriptions` が user ごとの購読状態を表す。

この構造で user の解除操作を契機に `feed_links` や `feeds` を削除すると、最後の購読者が離れた瞬間に共有コンテンツ全体を消す設計になりやすい。特に `articles.feed_id -> feeds.id` が cascade 削除で接続されているため、feed 掃除は article 群の消失に直結する。

また、`feeds.feed_link_id` の外部キー戦略をその都度調整して削除を通そうとすると、問題の本質である「集約境界の混線」を覆い隠してしまう。未適用の FK 修正 migration を追加で採用するより、削除責務を明確に分離するほうが安全だった。

## Decision

**共有 feed source の削除責務と user subscription の削除責務を分離し、今回の user 操作では subscription だけを更新する。**

今回の決定では以下を採用する。

- `feed_links` / `feeds` / `articles` は current user の解除操作では削除しない
- 未適用の `feeds.feed_link_id` 向け FK 修正 migration は採用しない
- schema 制約の辻褄合わせではなく、usecase の責務を user state 更新に限定する
- 共有 source の物理削除が必要なら、別の管理 API または非同期 GC として設計する

### 代替案

| 案 | 評価 |
|---|---|
| `feeds.feed_link_id` を `ON DELETE RESTRICT` に変えて usecase で条件削除する | check-then-act 競合と共有データ誤削除を招くため不採用 |
| `ON DELETE CASCADE` のまま feed source 物理削除を続ける | 最後の購読者離脱で共有記事まで消えるため不採用 |
| source 削除と購読解除を同じ usecase に共存させる | 集約境界が曖昧になり、将来の保守で再び誤用されやすいため不採用 |

## Consequences

### Pros

- user state と共有コンテンツの境界が明確になる
- migration や FK 方針を場当たり的に増やさずに済む
- 共有 source の GC を将来別要件として独立に設計できる

### Cons / Tradeoffs

- 購読者ゼロの source が registry に残り続ける
- source の整理は別ジョブまたは別 API として後続設計が必要になる
- delete ではなく unsubscribe を主契約にしたことで、古い命名とのズレをドキュメントで補う必要がある
