# ADR-000860: RecapTopicSnapshotted を Knowledge Loop entry の Recap act_target に接続して Open Recap CTA を first-class 化する

## Status

Accepted

## Date

2026-04-27

## Affected Services

- knowledge-sovereign: `SurfaceScoreInputs` に `RecapTopicSnapshotID string` を追加。`EventLogSurfaceScoreResolver` が `RecapTopicSnapshotted` を見るたびに UUID 形式 (`uuidPattern` regex) で validate し、最も新しい term overlap snapshot id を pin する
- knowledge-sovereign: `Projector.buildEntryFromEvent` が新 `seedActTargets(inputs)` を呼び、`RecapTopicSnapshotID` が非空なら `[{target_type: "recap", target_ref: <uuid>, route: "/recap/topic/<uuid>"}]` の JSON を `act_targets` 列に書く。`marshalSurfaceScoreInputs` も `recap_topic_snapshot_id` フィールドを additive に出力する
- alt-frontend-sv: `LoopEntryTile.svelte` が `entry.actTargets` から `targetType === "recap"` を抽出し、`safeRecapRoute` で `startsWith("/")` AND `!includes(":")` を validate した上で `<a class="cta cta--recap" href={recapRoute}>Open Recap</a>` を expanded section に render
- alt-frontend-sv: `LoopEntryTile.source.spec.ts` に Recap CTA + route allowlist の構造的 guard を追加し、proto 変更 / migration 変更を伴わずに canonical contract §6.4.1 の "RecapTopicSnapshotted は Surface Planner 入力のみ" を維持

## Context

canonical contract §6.4.1 は `RecapTopicSnapshotted` を Surface Planner v2 の immutable input event として定義しており、entry-creating event ではない。実装でも `score_resolver_event_log.go` がこのイベントを見ては `TopicOverlapCount` を増やすだけで、それ自体から `KnowledgeLoopEntry` 行を生成していない。

[[000856]] で planner version / score inputs を read model に永続化する経路が完成し、`SurfaceScoreInputs` の immutable evidence は `surface_score_inputs` JSON に保存されるようになった。しかし `fb.md` のレビューでは「Recap が score 入力止まりで CTA に出てこないので、ユーザから見ると Recap がどこで knowledge loop に効いているか分からない」と指摘された。Open Recap という導線が UI に無いと、Surface Planner v2 の placement signal がブラックボックス化しやすい。

一方で「RecapTopicSnapshotted から entry を新規作成する」「`KnowledgeLoopEntry` 単位で recap binding を新 column 化する」といった選択肢は projection schema を破壊的に変える。canonical contract を維持しつつ、既存 entry に Recap CTA を後付けする方法が必要だった。

`recap_topic_snapshot_id` 自体を `/recap/topic/<id>` の URL に直接織り込むため、UUID 形式以外を弾かないと open-redirect / `javascript:` scheme / path-traversal を許す可能性がある。`security-auditor` skill での監査でこの threat を明示的に扱った。

## Decision

`RecapTopicSnapshotted` は entry-creating event に格上げしないまま、Surface Planner v2 の score 入力経路に「最も新しい overlap snapshot id」を pin する seam を追加し、それを使って既存 entry の `act_targets[]` に Recap CTA を seed する。

採用した結合方式:

- **`SurfaceScoreInputs.RecapTopicSnapshotID`**: `score_resolver_event_log.go` の `RecapTopicSnapshotted` ループ内で `top_terms[]` と entry tags の交差を確認し、overlap が立った場合に payload の `recap_topic_snapshot_id` を取り出す。複数 overlap した場合は `event.OccurredAt` が最も新しいものを pin (`matchedRecapAt` を内部で track)。`uuidPattern` regex で UUID v1-v5 形式を validate し、形式違反は `RecapTopicSnapshotID` に記録しない。term overlap 自体 (`TopicOverlapCount`) は formula 側の計算なので validate に関係なく増える
- **Projector 側 `seedActTargets`**: `Projector.buildEntryFromEvent` で `entry.ActTargets = seedActTargets(inputs)` を呼ぶ。`RecapTopicSnapshotID` が空なら nil を返し、`act_targets` 列は空のまま。非空なら `[{target_type:"recap", target_ref:<id>, route:"/recap/topic/<id>"}]` の JSON bytes を返す。route の format string は projector 内定数で、article slug 等の latest state を一切引かないため reproject-safe
- **Frontend route allowlist**: `LoopEntryTile.svelte` が `recapTarget = $derived(entry.actTargets.find(t => t.targetType === "recap"))` で抽出。`safeRecapRoute = $derived.by(() => { ... startsWith("/") && !includes(":") ... })` で 2 段 validate し、いずれか満たさなければ `null` にして CTA を render しない。anchor (`<a href>`) を使うことで semantic navigation (ctrl+click / keyboard / SvelteKit client-side routing) を維持する

検討した代替案と却下理由:

- **`RecapTopicSnapshotted` 自体を entry-creating event に格上げする**: canonical contract §6.4.1 の Surface Planner 入力のみという定義を破る。読み取り model に新しい entry が増えて UI で重複表示される懸念もあり、後方互換 cost が高い
- **`act_targets` を proto 側に新 message 追加して explicit recap_target 列にする**: proto 変更を伴うのに UI 側で扱えるのは結局 `target_type` discriminator だけ。`act_targets` は既に `bytes` opaque JSONB で十分に拡張可能なので、proto を触らない選択をした
- **route format を runtime config 化する**: `/recap/topic/<id>` は frontend の SvelteKit route と直結しているため、ハードコードのほうが anchor の type safety と route allowlist の精度が出る。i18n / multi-tenant prefix を将来入れるときは別 ADR で扱う
- **UUID validate を frontend だけで行う**: 後段の validate は defense in depth として残すが、source of truth は projector 側。サーバ側で弾いておかないと、reproject 後の `act_targets` JSON にゴミが残ったまま frontend が CTA を出さない silent failure になる

## Consequences

### Pros

- ユーザに見える形で Recap が Knowledge Loop に接続される。Surface Planner v2 の topic overlap signal が CTA として体感できる
- canonical contract §6.4.1 の「`RecapTopicSnapshotted` は Surface Planner 入力のみ」という不変条件を維持できる。projection schema / proto contract のいずれも触らない
- UUID validate がサーバ側 (resolver) と frontend 側 (route allowlist) の 2 箇所で防御深化されている。upstream payload が将来 broken でも `javascript:` scheme injection が刺さらない
- `most-recent-overlap` という pin 戦略により、複数の RecapTopicSnapshotted が同じ entry にヒットしてもどれを表示するかが決定的。reproject で同じ event 列を replay すれば同じ snapshot id が入る
- `seedActTargets` は pure 関数で、`SurfaceScoreInputs` から決定されるため reproject-safe。`marshalSurfaceScoreInputs` JSON への `recap_topic_snapshot_id` 追加も additive

### Cons / Tradeoffs

- frontend 側で複数 recap target が来ても最初の 1 件しか render しない。将来「複数 recap を chip 列で出したい」要望が来たら component 側を拡張する必要がある
- 古い entry (本 ADR 以前の reproject 状態) は `act_targets` が空のまま残るため、運用上は `knowledge-loop-reproject` runbook での full reproject が前提になる
- route format をハードコードしたため、recap UI route を変えるときは projector 定数 + frontend allowlist + e2e の 3 箇所を同期する必要がある。doc 化はせず、`seedActTargets` の comment と source spec test に固定
- `RecapTopicSnapshotID` に複数候補が並んだ際の "most-recent" 選択は payload 由来 `event.OccurredAt` 比較で決定論。ただし同じ `OccurredAt` のイベントが 2 件あった場合は loop 順 (= DB 返却順) に依存する。実運用でこの衝突は観測されていないが、原理的には ranker tie-break を将来追加する余地がある
