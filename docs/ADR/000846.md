# ADR-000846: Knowledge Loop why_text を discovered event で backfill し articles.title mutability の受容範囲を明示する

Accepted

## Context

[ADR-000844] で `enrichSummaryVersion` が `event.payload.article_title` を inline する narrative を生成するよう projector を修正、producer ([create_summary_version_usecase](alt-backend/app/usecase/create_summary_version_usecase/usecase.go)) も title を payload に焼き込むよう修正済。

しかし**過去に append された `SummaryVersionCreated` event は title を持たない**ため、reproject しても `enricher.go` の fallback 経路 (`"A new summary is ready in one of your feeds."`) を辿ることになり、既存 `knowledge_loop_entries` 行は永遠に generic narrative のまま残る。ユーザは production で:

```
Observe / Critical / A new summary is ready in one of your feeds. (rev 1, just now) × 3 件
```

を観測しており、Knowledge Loop 全体が役に立たない状態が継続。

### 制約

- `knowledge_events` は append-only ([canonical contract] §3-1)。UPDATE/DELETE 禁止。
- projector enricher は **event payload + 安定 versioned resource** のみから動作 ([canonical contract] §3-4 reproject-safe / immutable-design-guard #4)。
- projection (`knowledge_loop_entries`) は disposable だが、書き込み path は event 駆動でなければならない (#5 / [[knowledge-home-projection-recovery]])。
- 既存 entry の `dismiss_state` (HomeItemDismissed 由来) を破壊してはならない。
- `articles.title` は **mutable**: 5 件以上の `GRANT UPDATE` migration が存在し、`article_versions` snapshot table も無い (#6 versioned artifacts は articles を満たしていない既存負債)。

### 既存 backfill パターン

[ADR-000841] / [ADR-000843] 系の `KnowledgeBackfillJob` (`app/job/knowledge_backfill_job.go`) が articles を walk して `EventArticleCreated` を re-emit する pattern を確立済。cursor + dedupe_key + status flow が既に動いている。本 ADR は同じ pattern を踏襲する。

## Decision

### 戦略: discovered event + patch-only-why projector branch

新 event 型 **`SummaryNarrativeBackfilled`** を導入。alt-backend の新 backfill job が `summary_versions JOIN articles` を walk し、各 `(summary_version, article)` ペアに対して synthetic event を append-only log に書く。projector に新 case を追加し、**既存 entry の `why_*` columns のみ UPDATE する patch path** を駆動する。dismiss_state / freshness_at / surface_bucket / decision_options 等は触らない。

### Patch SQL の独立 path 

新 repo method `PatchKnowledgeLoopEntryWhy` は full UPSERT とは別の SQL を持つ:

```sql
UPDATE knowledge_loop_entries SET
  projection_revision    = projection_revision + 1,
  projection_seq_hiwater = GREATEST(projection_seq_hiwater, $5),
  source_event_seq       = GREATEST(source_event_seq, $5),
  projected_at           = NOW(),
  why_kind               = $6,
  why_text               = $7,
  why_confidence         = $8,
  why_evidence_ref_ids   = $9,
  why_evidence_refs      = $10
WHERE user_id = $1 AND tenant_id = $2 AND lens_mode_id = $3 AND entry_key = $4
  AND projection_seq_hiwater <= $5
RETURNING projection_revision, projection_seq_hiwater
```

- `dismiss_state` 等の他フィールドは構文上現れない (test で structural guard)。
- `seq_hiwater <= $5` で reproject 冪等性 (replay の後勝ち / 古い seq の no-op)。
- `pgx.ErrNoRows` → `SkippedBySeqHiwater=true` (entry 未存在 OR guard 拒否、両方安全)。

### Replay 順序の収束

任意の replay 順序で:
1. `SummaryVersionCreated` (event_seq=N、payload に title 無し) → 既存 enricher fallback 経路で entry を INSERT/UPSERT (generic narrative)
2. `SummaryNarrativeBackfilled` (event_seq=M > N、payload に title 有り) → 同 entry を UPDATE (substantive narrative)

`event_seq` の単調性が DB BIGSERIAL で担保されているため、order は事実上の append 順。`seq_hiwater <= EXCLUDED` guard で逆順 replay も冪等。

### 1 度だけの snapshot で articles.title mutability を受容

backfill job は実行時の `articles.title` を payload に焼き込む。dedupe_key `summary-narrative-backfill:<sv_id>` で **1 sv あたり 1 度** のみ event を作る。後から articles.title が UPDATE されてもこの event payload は immutable に固定される (= 過去 narrative は固定)。

これは Pat Helland "Immutability Changes Everything" の "snapshot at read time" pattern。articles を versioned 化しない代わりに、**event 化のタイミングで snapshot を取る**ことで reproject 冪等性は守る。

### kind 列の追加

`knowledge_backfill_jobs` テーブルに `kind TEXT NOT NULL DEFAULT 'articles'` を追加し、既存 articles-replay 流と新 summary-narratives 流を識別。既存 row はデフォルト値で 'articles' になり挙動不変。新 backfill job 起動時に `kind = 'summary_narratives'` で row を作る。両 job runner は kind フィルタで自分担当の row だけを処理する。

### 検討した代替案

1. **既存 `EventSummaryVersionCreated` を新 dedupe_key で再 emit、projector の既存 path に流す**: dismiss_state を `EXCLUDED` で上書きするため、user が dismissed した entry を active に resurrect してしまう。却下。
2. **projector で articles を直接 lookup**: articles.title が mutable なため reproject の決定論性が崩れる。`time.Now()` ではないが「現在 DB の値」依存になり、(event_seq, articles.title-at-replay-time) で結果が変わる。canonical contract §3-4 違反。却下。
3. **`article_versions` snapshot table を新設**: 設計としては最もクリーンだが scope が大きい (proto / migration / write path / 全 producer 改修)。本 ADR とは独立した別 workstream として future work。
4. **projector 側 upcaster で payload を在実行時に enrich**: Marten / Oskar Dudycz の upcaster pattern。N+1 lookup 問題を抱える ([Marten docs warning](https://martendb.io/events/versioning))。今回は projector が高頻度 (5s tick) で動くため不採用。
5. **projection 直 UPDATE script (one-shot SQL)**: disposable projection 原則 #5 「write path は projection を直接 mutate しない」の違反。runbook も明文で禁止。却下。

### CI gate (TDD outside-in)

- E2E (Hurl): `e2e/hurl/knowledge-sovereign/20-summary-narrative-backfill.hurl` — append 前後で fallback → 患 narrative の遷移を end-to-end で検証
- Unit (Go projector): `TestRunBatch_SummaryNarrativeBackfilled_PatchesWhyOnly` — patch path のみ呼ばれ、Upsert は呼ばれない (= dismiss_state 安全)
- Unit (Go driver): `TestPatchKnowledgeLoopEntryWhy_AppliesPatchOnly` + structural guard で SQL 内に dismiss_state 等が現れないことを検証

## Consequences

### Pros

- 過去の SummaryVersionCreated 由来 entry が **真の article title narrative に backfill される** — ユーザの "/loop が役に立たない" 状態が解消
- append-only / reproject-safe / disposable projection / merge-safe upsert / single emission 不変条件すべて維持
- dismiss_state ほか entry 状態は保持 (HomeItemDismissed で dismissed 化された entry が backfill で active に戻らない)
- dedupe_key で冪等。再実行・部分中断・再開すべて安全
- kind 列で 2 つの backfill 流を完全に分離。既存 articles-replay には影響無し
- `WhyMappingVersion` v5 bump で operator が full reproject (任意) を発火する基準を明示。reproject しても backfill 結果は収束 (replay 安全)

### Cons / Tradeoffs

- **articles.title mutation の受容**: backfill 後に title が編集されても event payload は不変なので過去 narrative は固定。次回 backfill では再実行されない (dedupe_key で 1 sv = 1 event)。article 編集後の narrative 更新は本 ADR scope 外 (将来 `article_versions` snapshot 導入で改善余地)。
- **新 event 型増加**: `SummaryNarrativeBackfilled` は kawasima 観点では弱い business 述語 (discovered event の運用上の存在)。event log が膨らむ要因の 1 つ。partition / archival 戦略に追加負荷。
- **scheduler tick が 1 サイクル増える**: 15s interval × batch 100 events で平均 quiet 時 1 SELECT/15s。負荷影響は無視できる範囲だが metrics dashboard で monitor 推奨。
- **ts-speaker 等 article_versions snapshot を要求する future feature がある場合**、本 ADR の "snapshot at backfill time" は再実行コストになる。article_versions が入れば `article_versions JOIN summary_versions` で精度向上した再 backfill が可能 (本 ADR の dedupe_key を rotation する設計余地を残す)。
- **kind 列の semantic 拡張余地**: 将来 tag-narrative-backfill や lens-narrative-backfill 等を追加する場合、enum 化 (`CHECK (kind IN ('articles','summary_narratives','...'))`) を検討する余地。今回は文字列のまま。
