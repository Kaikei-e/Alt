# ADR-000426: Tag write path と projector のタグ投影を修正し Knowledge Home カードにタグ・why を反映する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — handler, port, driver, gateway, job (projector), DI, main
- scripts — backfill SQL

## Context

Knowledge Home Phase 6 Step 1（[[000424]], [[000425]]）で UI 再構成は完了したが、カードのデータ密度が不足していた。具体的には：

1. **単発 `UpsertArticleTags`** は `TagSetVersion` を作成するが `UserID` を設定していなかった。`UserID` が空のまま `TagSetVersionCreated` イベントが発行されると、projector が投影する際に tenant の紐付けが失われる
2. **`BatchUpsertArticleTags`** は正規化テーブルの更新のみ行い、`tag_set_versions` も `TagSetVersionCreated` イベントも生成していなかった。バッチ経由のタグ登録が Knowledge Home に反映されない
3. **Projector の `projectTagSetVersionCreated`** は event payload を parse して `article_id` だけ取り出し、**タグを一切読まずに空の tags で upsert** していた。`WhyReasons` も `new_unread` のみで `tag_hotspot` が追加されなかった
4. **Projector が「最新 version」ではなく「event が指す version」を参照する仕組み**がなかった。event backlog 処理時に別 version を投影してしまうリスクがあった

これらの結果、Knowledge Home カードにはタグが表示されず、"Trending: {tag}" バッジも出なかった。

## Decision

以下の 4 層を修正し、tag write path → event → projector → read model の一貫性を回復する。

### 1. `GetTagSetVersionByIDPort` を新設する

`GetLatestTagSetVersion(article_id)` ではなく **event payload の `tag_set_version_id` を使って特定 version を読む** port を追加する。

- **Port**: `tag_set_version_port.GetTagSetVersionByIDPort` — `GetTagSetVersionByID(ctx, uuid) (TagSetVersion, error)`
- **Driver**: `AltDBRepository.GetTagSetVersionByID` — `tag_set_versions` を主キーで SELECT
- **Gateway**: `tag_set_version_gateway.Gateway.GetTagSetVersionByID`

`latest` 参照は backlog / reorder / supersede で誤投影しうるため不採用とした。

### 2. Handler の tag write path を修正する

**`UpsertArticleTags`**: `getArticleByID` で article を引き、`UserID` を解決してから `TagSetVersion` を組み立てる。user が取れない場合はログを残す。

**`BatchUpsertArticleTags`**: batch upsert 完了後、各 article ごとに `createTagSetVersionUsecase.Execute()` を呼び出し、version + event を生成する。冪等性は dedupe key (`TagSetVersionCreated:{version_id}`) で担保される。

### 3. Projector のタグ投影を修正する

`KnowledgeProjectorJob` の引数に `tagSetVersionPort` を追加し、call chain 全体（`processKnowledgeEvents` → `projectEvent` → `projectTagSetVersionCreated`）に伝搬する。

`projectTagSetVersionCreated` の修正内容：
- payload の `tag_set_version_id` を parse し、`GetTagSetVersionByID` で該当 version を取得
- `TagsJSON` を parse して `item.Tags` に反映
- タグが 1 つ以上ある場合、`WhyReasons` に `tag_hotspot`（先頭タグ参照）を追加
- **merge-safe**: `Title`, `SummaryExcerpt`, `SummaryState` は空のまま upsert し、DB 側の merge-safe upsert で既存値を保持

### 4. Backfill SQL を拡張する

既存の summary backfill に加え、`article_tag_normalized` から article 単位のタグスナップショットを `tag_set_versions` に backfill し、対応する `TagSetVersionCreated` イベントを `knowledge_events` に挿入する。

### DI / main 配線

- `ApplicationComponents` に `TagSetVersionGateway` フィールドを追加
- `main.go` の `KnowledgeProjectorJob` 呼び出しに `TagSetVersionGateway` を追加

## Consequences

### Pros

- Knowledge Home カードにタグが表示され、`Trending: {tag}` バッジで記事の分類が一目でわかるようになる
- batch tag upsert でも Knowledge Home に反映されるため、tag-generator の backfill 処理でもカードが更新される
- event → version ID ベースの投影により、event backlog 再生時の誤投影リスクを排除できる
- `UserID` が正しく version に載るため、マルチテナント環境での tenant 分離が保たれる

### Cons / Tradeoffs

- `BatchUpsertArticleTags` で article ごとに `createTagSetVersionUsecase.Execute()` を呼ぶため、大量 batch 時は N 回の INSERT が発生する。現状の batch サイズ（数十件）では問題にならないが、将来的に bulk INSERT 版の usecase が必要になる可能性がある
- `getArticleByID` で UserID を解決するため、article が DB に存在しない場合は version の UserID が零値になる。この場合 event は発行されるが tenant が不正確になる（ログで警告）
- backfill SQL は `article_tag_normalized` テーブルの存在を前提とする。テーブル構造が変わった場合は SQL の修正が必要
