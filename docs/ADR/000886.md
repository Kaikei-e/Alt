# ADR-000886: Morning Letter editorial daemon を恒久 ON 化し、today_digest projection と recap-summary preempt 連鎖を同時に解消する

## Status

Accepted

## Date

2026-05-04

## Affected Services

- **recap-worker (Rust)** — `compose/recap.yaml` の `recap-worker.environment` に `MORNING_DAEMON_ENABLED=${MORNING_DAEMON_ENABLED:-true}` を 1 行追加。`recap-worker/recap-worker/src/main.rs:132-141` の env ゲートはそのまま残し、kill switch として機能させる。`scheduler/daemon.rs` の `MorningUpdateDaemon` が 30 分 cadence で `run_morning_update` を回す挙動は無変更。
- **alt-backend (Go)** — `app/domain/today_digest.go` に `PulseRefs []string` を追加。`app/job/digest_availability_reconcile_job.go` で `pulse.Topics` を `cluster:%d` 形式の参照に変換し digest に同梱（`PulseStatusError` の場合は空のまま）。`app/job/knowledge_projector.go` の `projectEvent` ディスパッチが `projectTagSetVersionCreated` に `today_digest_port` を渡し、当該プロジェクタが home item upsert 後に `domain.TodayDigest{TopTags: tags}` を非致命的に upsert する。
- **knowledge-sovereign (Go)** — `app/driver/sovereign_db/repository.go` の `UpsertTodayDigest` で `pulse_refs` を payload struct に追加し、ハードコード `[]byte("[]")` を撤去。`top_tags_json` / `pulse_refs_json` の `ON CONFLICT` 句を `COALESCE(NULLIF(EXCLUDED.x, '[]'::jsonb), today_digest_view.x)` に書き換え、merge-safe upsert ([[000885]] と同系の不変条件) に揃える。
- **news-creator (Python)** — `news_creator/handler/recap_summary_handler.py` で `PreemptedException` を `RuntimeError` より先に explicit catch し 502 を返す。`news_creator/usecase/recap_summary_usecase.py` の `_generate_chunk_summary` と `_reduce_group` を新ヘルパ `_hold_slot_generate_raw_with_preempt_retry` 経由に統一し、`PreemptedException` を最大 3 回・1s/2s 指数バックオフで再試行。3 回失敗時は元の except Exception で graceful degradation（None 返却）に落ちる。

## Context

ユーザから「Morning Letter UI が `/recap/morning-letter` で `target_date = 2026-04-13` のまま 3 週間以上更新されていない」という障害報告があった。コンテナログ・recap-db・compose 設定を読み合わせた結果、独立した 3 つの劣化が同時に観測された。

### Bug A: Morning Letter daemon の恒久 OFF

`recap-worker/recap-worker/src/main.rs:132-141` で daemon は `MORNING_DAEMON_ENABLED` env で gated されており、未設定時は `false` 扱い。`compose/recap.yaml` の recap-worker `environment` セクションに同 env が一行も無く、deploy ごとに daemon が止まる構成になっていた。稼働中コンテナの最新 INFO ログは `morning daemon disabled (set MORNING_DAEMON_ENABLED=true to enable)` を吐き続けており、`recap-db.morning_letters` には `target_date=2026-04-13`, `created_at=2026-04-13 01:33:35.994094+00` の **1 行のみ** が存在した。これは [[000705]] が Accepted となった当日に何らかの一時的トリガで作成された行で、以後 3 週間追記されていない。`GetLatestLetter` は健康に 200 を返すが、その内容は同じ 4/13 のレターを返し続けるため、UI は「機能停止」に見えた。[[000705]] は「生成トリガの恒久有効化」を宣言しているが、compose env がそれを反映しないまま放置されていた（ADR ↔ ops drift）。

### Bug B: today_digest_view の projection が常に空配列

`knowledge-sovereign-db.today_digest_view` を読むと、両ユーザの全 `digest_date` 行で `top_tags_json=[]` かつ `pulse_refs_json=[]` が少なくとも 2026-04-10 以降ずっと続いていた。`evening_pulse_available` は 2026-04-14 に `true → false` に転落し、その後フラップを繰り返している。書き込みパイプ自体（`knowledge_events` への append、`TagSetVersionCreated` の発火、`recap-worker` の evening_pulse 生成）は健全で、原因は **3 段の projection 側欠陥** だった。

1. `knowledge-sovereign/app/driver/sovereign_db/repository.go:274` に `pulseRefsJSON := []byte("[]")` のハードコードがあり、payload に何が入っていても DB には常に空配列が書かれていた。`pulse_refs` を読む struct field 自体が存在せず、上流から渡しようがない断絶。
2. `top_tags_json` / `pulse_refs_json` の `ON CONFLICT` 句が `CASE WHEN EXCLUDED.x != '[]'::jsonb THEN EXCLUDED.x ELSE today_digest_view.x END` という business-judgement-in-SQL になっており、**一度 `[]` で insert された行はその後どんな書き込みが来ても永遠に `[]` のまま**になる latch を構成していた。これは [[000885]] / [[knowledge-home]] 系で繰り返し再確認している merge-safe-upsert 不変条件（feedback_merge_safe_upsert.md）の違反である。
3. alt-backend の projector 側に top_tags / pulse_refs を実際に書く path が存在しなかった。`projectArticleCreated` は `NewArticles: 1, UnsummarizedArticles: 1` だけを送り、`projectTagSetVersionCreated` は `knowledge_home_items` を upsert して停止していたため、`today_digest_view.top_tags_json` には事実上何も流れていない。`pulse_refs_json` も `digest_availability_reconcile_job` が `pulse_available` だけを更新し refs を抽出していなかった。

projection は disposable で reproject 可能なので、修正後は次の projector tick / reconcile tick で自然に回復する。

### Bug C: news-creator の preempt 連鎖で `/api/generate` が HTTP 500

2026-05-03 17:20–17:36 に news-creator のログに大量の `PreemptedException: Request <uuid> preempted during generation` と `/api/generate HTTP 500` が記録された。原因は handler / usecase 双方の例外処理欠陥。

- `news-creator/app/news_creator/handler/recap_summary_handler.py` は `ValueError → 400`, `RuntimeError → 502`, `Exception → 500` の順で catch していたが、`PreemptedException` は `Exception` の generic 枝に落ち **HTTP 500 'Internal server error'** を返していた。call 側（recap-worker）はこれを「永続失敗」として扱い retry を諦めるため、preempt 1 件が下流の summary を 1 つ失う設計になっていた。
- `news-creator/app/news_creator/usecase/recap_summary_usecase.py` の `_generate_chunk_summary` (L450) と `_reduce_group` (L693) は `hold_slot()` + `generate_raw()` を `try / except Exception` で囲み、preempt を silent に swallow して `None` を返していた。これによって map phase の intermediate_summaries が一部欠損し、`recap-summary` は `_hierarchical_threshold` 越えのリクエストで extractive fallback に脱線、当該時間帯のクラスタ要約品質が連鎖的に劣化した。

`HybridPrioritySemaphore` 自体は意図通りの動作（RT が BE を preempt する）であり、半分以下の解像度で見ると preempt は正常な priority enforcement だが、**「preempt されたら client に retryable と伝える」「同一 BE スロットを取り直して replay する」** の 2 つの recovery 経路が欠けていた。

## Decision

3 件を独立した bug fix として 3 commit に分けて main に積む。共通 root cause は無いが、いずれも Morning Letter の体験劣化に直接寄与しており、まとめて記録する価値がある。本 ADR は ops/projection/inference の各層で取った決定をまとめて固定する。

### 1. recap-worker daemon を compose env で恒久 ON にする

`compose/recap.yaml` の `recap-worker.environment` に `- MORNING_DAEMON_ENABLED=${MORNING_DAEMON_ENABLED:-true}` を 1 行追加する。`PULSE_ENABLED=true` の隣に並べ、env file での明示的 override を許容するが、deploy 既定では daemon を起動する。

`main.rs` の `unwrap_or(false)` ロジックは保持する。これは「障害時に env で false に倒して daemon を即時停止する」kill switch として機能し、`scheduler/daemon.rs` を変更せずに恒久制御を実現する。30 分 cadence (`Duration::from_mins(30)`) も維持する。

#### 検討した代替案

- **`main.rs` の default を `true` に倒す案**: ADR-705 が Accepted な以上、code-side default も追従させる方が ADR ↔ implementation drift を構造的に防げる。一方で blast radius が広く、env で false にしている test stack や staging が不意に daemon を起動する事故が起きうる。compose 1 行で「prod / dev は ON、必要なら override」が表現でき、kill switch の意味論を残せる方がリスクが低い。
- **`MORNING_DAEMON_ENABLED` flag そのものを撤去する案**: ADR-705 は「rollout flag」を撤去する方向と整合するが、preempt 連鎖や recap-worker 全体の障害時に daemon だけ落としたい運用シナリオが残る。kill switch を一律削るのは時期尚早と判断。
- **過去 4/14〜5/4 の morning_letter を backfill する案**: daemon は `target_date` 引数を取らず当日分しか書かない設計。遡及生成パスは未実装で、別 ADR で設計が要る。ユーザの体感は「最新が今日になる」で復旧するため今回は対象外。

### 2. today_digest_view の top_tags / pulse_refs を end-to-end で配線し、merge SQL を merge-safe 形に揃える

3 段の欠陥それぞれに対し、append-first / reproject-safe / merge-safe-upsert の不変条件を満たす形で修正する。

- `domain.TodayDigest` に `PulseRefs []string` フィールドを追加し、JSON tag は `pulse_refs`。`json.Marshal` で payload に乗り、knowledge-sovereign 側で対称に Unmarshal される。
- `digest_availability_reconcile_job` は `recapPort.GetEveningPulse(ctx, today)` の結果を見て、`pulseAvailable` が真のときだけ `pulse.Topics` を `fmt.Sprintf("cluster:%d", topic.ClusterID)` で参照に展開する。`PulseStatusError` のときは `pulseRefs == nil` を維持し、空配列を送って merge-safe SQL の `NULLIF(..., '[]')` 経由で既存値を保持する。
- `knowledge_projector.projectTagSetVersionCreated` を `today_digest_port` も受け取るシグネチャに変更し、`event.UserID` を採用したうえで `tags` 配列が非空のときに限り `domain.TodayDigest{UserID, DigestDate: now, TopTags: tags, UpdatedAt: now}` を upsert する。`UpsertKnowledgeHomeItem` の成功後に呼び、エラーは ERROR ログ + 続行（home item の正しい upsert を巻き戻さない）。
- `knowledge-sovereign/app/driver/sovereign_db/repository.go` の `UpsertTodayDigest` で local Unmarshal struct に `PulseRefs []string` を追加し、`pulseRefsJSON := []byte("[]")` ハードコードを撤去。`top_tags_json` / `pulse_refs_json` の `ON CONFLICT` 句を `COALESCE(NULLIF(EXCLUDED.x, '[]'::jsonb), today_digest_view.x)` に置換する。これにより「空書き込みは既存値を保持し、非空書き込みのみ上書き」という意図が SQL レベルで明示される。

reproject-safe / append-first の観点で immutable-design-guard を通過する:

- projector は event payload (`TagSetVersionCreated`) と versioned resource (`tag_set_versions` テーブル) のみを読む。latest state は読まない。
- `digest_availability_reconcile_job` は projector ではなく periodic aggregator なので、recap-worker の現在状態を read してよい（既存設計）。新規追加した `pulse.Topics` の参照導出も同じカテゴリ。
- `today_digest_view` は disposable projection。既存の `[]` 行は次の reconcile tick / TagSetVersionCreated tick で自然に書き換わる（merge-safe 句が `[]` 書き込みを抑止する一方、非空書き込みは即座に latch を解除する）。手動 reproject script は不要。

#### 検討した代替案

- **TopTags を真の「日次 top-N」として集計する案**: 今回の `projectTagSetVersionCreated` は最新 article のタグでフィールドを上書きするだけで、厳密な top-N 集計ではない。append-first 観点で正しく日次 top-N を出すには、新 event 種別 (`DailyTopTagsComputed`) を append する設計が要る。本 PR の scope ではフィールドが空でなくなるところまでに留め、真の top-N は別 ADR / 別 Issue で扱う。
- **既存 `[]` 行を migration で更新する案**: append-first / reproject-safe を尊重し、projection 側で次の tick が自然に上書きするのを待つ方が筋が良い。直接 UPDATE は disposable projection の不変条件と整合しない。
- **`pulse_refs` の source を新 event (`PulseRefsAttached`) として再設計する案**: pulse の topic id / cluster id 自体は `EveningPulse` 集約の一部として既に存在し、`recap_port.GetEveningPulse` で取得可能。新 event を増やすコストが追加 signal の価値を上回らないと判断。

### 3. news-creator の preempt 連鎖を handler/usecase 双方で塞ぐ

- `recap_summary_handler.py` の `try / except` 連鎖に `PreemptedException` を `RuntimeError` より前に挿入する。`HTTPException(status_code=502, detail=str(exc))` で client に retryable シグナルを返し、generic `Exception → 500` の枝には preempt は到達しない。logger.warning で「higher-priority bumped this BE request」を残す。
- `recap_summary_usecase.py` の `_generate_chunk_summary` / `_reduce_group` の `hold_slot + generate_raw` ブロックを共通ヘルパ `_hold_slot_generate_raw_with_preempt_retry(prompt, *, num_predict, format, options)` に集約する。ヘルパは最大 `_PREEMPT_RETRY_ATTEMPTS = 3` 回ループし、各 attempt で `hold_slot` を取り直して `generate_raw` を呼ぶ。`PreemptedException` を catch したら `2 ** attempt` 秒（1s / 2s）バックオフして次 attempt に進む。3 回連続で preempt されたら最後の例外を re-raise し、呼び出し側の `except Exception` で graceful degradation（`None` 返却 → orchestrator が extractive fallback）に落ちる。
- 既存の `try / except Exception as e: logger.warning(...); return None` は維持する。preempt 以外の例外（タイムアウト・JSON parse 失敗等）に対する graceful degradation はそのまま機能し続ける。

#### 検討した代替案

- **`max_queue_depth` を 10 → 20 に引き上げる案**: queue saturation を緩和する直交した tuning だが、preempt そのものは別の現象（RT bump）であり queue depth では治らない。本 ADR の handler / usecase 修正と独立して評価すべきで別 Issue に切る。
- **handler だけ修正し usecase 側 retry を見送る案**: client 側 (recap-worker) の retry に任せれば動くが、recap-worker → news-creator は HTTP 1 回のラウンドトリップで重い prompt を送り直すコストが高く、usecase 内で同じ slot pool に対する retry を行う方が遅延・トラフィック双方で安い。`hold_slot` 単位の retry は `is_high_priority=False` を保つので RT 優先を侵さない。
- **`PreemptedException` を catch して即座に `None` を返す案**: 「silent swallow をやめる」というそもそもの問題提起と矛盾し、map phase の intermediate_summaries 欠損が再現する。

## Consequences

### Pros

- ユーザ報告の中心症状である「Morning Letter が 4/13 のまま停止」が、deploy 後 30 分以内に当日 target_date への自然追従で復旧する。`recap-worker` の `MORNING_DAEMON_ENABLED=true — starting morning editorial projector daemon` ログが恒久的に出る。
- `today_digest_view.top_tags_json` / `pulse_refs_json` が次の projector / reconcile tick から非空に切り替わり、Morning Letter UI の「今日の話題タグ」「Evening Pulse 参照」が機能する。merge-safe-upsert 不変条件の違反が SQL レベルで解消され、今後同じ root cause での latch 事故が再発しない。
- news-creator の preempt は 2026-05-03 タイプの fallback storm を起こさなくなる。RT bump の正常な発火 → BE 側で slot 取り直し → silent recovery、というセマンティクスが確立する。client は 502 を見て backoff retry に入り、5xx の意味が「永続失敗ではなく retryable」であるという HTTP 規範に従う。
- `recap_summary_usecase.py` の chunk / reduce path が共通ヘルパ経由になり、二重実装の drift リスクが減った。retry ポリシー（attempts / backoff）は単一の module 定数 `_PREEMPT_RETRY_ATTEMPTS` で集中管理できる。
- knowledge-sovereign の merge SQL が [[000885]] で確認した COALESCE/NULLIF 系列に揃い、`today_digest_view` のマージ規約が `knowledge_home_items` と対称になった。今後 same-shape の projection を増やすときの canonical pattern として参照しやすい。

### Cons / Tradeoffs

- `MORNING_DAEMON_ENABLED` の env は残るので、env file 編集 1 つで daemon を OFF にできてしまう。ops drift の再発を構造的に防ぎたい場合は、後続 ADR で `main.rs` の default 反転または flag 撤去が必要。今回は最小手で deploy 既定だけ修正している。
- `projectTagSetVersionCreated` で書く `TopTags` は「最新 article のタグの羅列」であり、厳密な日次 top-N 集計ではない。merge-safe SQL の `NULLIF(EXCLUDED.x, '[]'::jsonb)` のおかげで空書き込みでは latch されないが、毎 article 上書きされる挙動は将来 Surface Planner / Acolyte が日次 top-N を score に使いたいときに合わない。別 ADR で `DailyTopTagsComputed` 系の event 起点に再設計する余地を残す。
- `pulse_refs` を `cluster:%d` 形式の string で表現したのは中継便宜。将来 `pulse_topic` が cluster_id 以外の主キーを持つ拡張が来たら参照スキーマを再検討する必要がある。
- news-creator の retry は同一 BE slot pool に対して連続して acquire/release を回すので、RT 側からは「BE 全体の throughput が下がっているが完全には止まっていない」状態に見える。RT が継続的に flood する病的シナリオでは 3 回の retry 全てが preempt される最悪ケースが発生し、本 ADR ではそこは graceful degradation（extractive fallback）で受け流す。`max_queue_depth` の調整は別 Issue 推奨。
- 既存空 `today_digest_view` 行は projector / reconcile が回るまで `[]` のままで、観測者によっては「修正したのに即座に直っていない」と見える。reconcile cron は数分間隔で回る前提で受け入れるが、運用通知では「最初の reconcile tick 後に確認する」を runbook に書き加える価値がある。
