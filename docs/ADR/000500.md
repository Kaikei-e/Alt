# ADR-000500: 要約ジョブキューを oldest-first に切り替えて backlog の飢餓を防止する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- pre-processor (Go)

## Context

BE 要約の停滞を調査したところ、要約ジョブキューに大規模な pending backlog が残存していた。一方で queue worker は pending ジョブを `created_at DESC` で取得しており、新しく投入されたジョブを優先して処理していた。

この順序は直近ジョブの応答性を上げる一方で、backlog が十分に大きい場合には古い pending を恒常的に後回しにする。実運用では pending の最古ジョブが長期間残存し、処理自体は継続していても backlog が前進していないように見える状態を招いていた。

`FOR UPDATE SKIP LOCKED` による並列 dequeue 自体は妥当だが、今回の問題はロック戦略ではなく **選択順序** にある。公開運用の queue としては、古いジョブの飢餓を許容しない方針が必要になった。

## Decision

**summarize_job_queue の pending 取得順を newest-first から oldest-first に変更する。**

具体的には、queue worker の dequeue クエリを以下の方針に統一する。

- `status = 'pending'` を対象にする
- `ORDER BY created_at ASC` で最古ジョブから取得する
- `FOR UPDATE SKIP LOCKED` は維持する

### 代替案

| 案 | 評価 |
|---|---|
| `created_at DESC` を維持 | 新規ジョブの見かけの応答性は高いが、backlog が大きいと古いジョブが飢餓状態になるため不採用 |
| priority queue / aging を導入 | 将来的には有効だが、現時点では dequeue 順序の修正だけで主要問題を解消できるため過剰 |
| pending を期限切れで削除 | 運用判断を伴うため、queue 基本方針とは分離して扱うべき |

## Consequences

### Pros

- 古い pending ジョブが前進し、backlog の飢餓状態を防げる
- queue の意味が FIFO に近づき、運用時の直感と一致する
- `SKIP LOCKED` を維持するため、将来の並列 worker 導入とも両立する

### Cons / Tradeoffs

- 直近に投入されたジョブの見かけの応答性は下がる可能性がある
- 新規ジョブ優先を期待する UI/運用とは相性が悪い
- 将来的に「新規優先」と「古参救済」を両立したい場合は aging や multi-queue への拡張が必要になる
