# ADR-000752: provider 検証の consumer version selector を Pact 公式推奨の mainBranch + deployedOrReleased へ統一する

## Status

Accepted

## Date

2026-04-16

## Affected Services

- `alt-backend/app/driver/contract/provider_test.go` — `recap-worker` / `alt-butterfly-facade` / `search-indexer` を verify する 3 本の selector 更新
- `search-indexer/app/driver/contract/provider_test.go` — `rag-orchestrator` / `alt-backend` / `acolyte-orchestrator` の selector 更新
- `pre-processor/app/driver/contract/provider_test.go` — `alt-backend` の selector 更新
- `mq-hub/app/driver/contract/provider_test.go` — Broker モード対応と `search-indexer` の selector 更新（ファイルベース only → Broker publish 化）
- `news-creator/app/tests/contract/test_provider_verification.py` — `_configure_verifier` の selector 更新（4 consumer 分）
- `tag-generator/app/tests/contract/test_provider_verification.py` — 同上
- `recap-subworker/tests/contract/test_provider_verification.py` — 同上
- `tts-speaker/tests/contract/test_provider_verification.py` — `alt-butterfly-facade` の selector 更新

Rust 側 (`recap-worker/recap-worker/tests/provider_verification.rs`) は本 ADR のスコープ外（§Cons 参照）。

## Context

[[000751]] を commit した直後 `./scripts/deploy.sh production` を実行したところ、Pact `can-i-deploy` が 14 pacticipant 中 11 本で `unknown` を返し exit 10 でゲートが止まった。`pact-check.sh` の `Pact Check Summary` は 18 pass / 0 fail で全ての provider verification test 自体は成功していた。

**根本原因**: provider test の `ConsumerVersionSelectors` (Go) / `consumer_version(...)` builder (Python) が `Latest: true` のみを指定しており、新しく publish された consumer pact しか verify していなかった。結果として、新 SHA の provider がすでに production に入っている古い consumer pact（`deployedOrReleased`）を verify せず、broker には「新 consumer pact × 新 provider」の verification しか記録されない。`can-i-deploy` は「新 consumer pact × production-deployed provider」の組を要求するので、SHA bump のたびに matrix cell が欠落し continuous deploy が永続的にブロックされる。

典型的な失敗 ( `alt-backend@4c88b4bd9` → `pre-processor` ):

```
There is no verified pact between version 4c88b4bd9 of alt-backend
and the version of pre-processor currently in production (5c1d08b51)
```

Pact 公式ドキュメントは明確な解を持つ ([Recommended configuration for verifying pacts](https://docs.pact.io/provider/recommended_configuration)):

```javascript
consumerVersionSelectors: [
  { mainBranch: true },         // 最新の開発版
  { matchingBranch: true },     // provider と同名 branch の consumer pact
  { deployedOrReleased: true }  // 現在デプロイ済み/リリース済みの consumer pact
]
```

`deployedOrReleased` が加わることで、新しい provider 側のビルドが「deployed な古い consumer pact」も必ず再検証し、結果を broker に記録する。これにより consumer 側の pact content hash が変わっていないケースでは verification が content hash で carry-forward し、SHA bump だけで gate が壊れる事態を防ぐ。[GitHub Discussion #434](https://github.com/pact-foundation/pact_broker/discussions/434) で提案されている webhook 駆動の自動再検証 (PactFlow が実装済) の OSS 相当手段。

ADR-000750 の Cons に「provider 検証網羅不足」として列挙された 7 本 (`pre-processor / mq-hub / rag-orchestrator / acolyte-orchestrator / recap-evaluator / alt-butterfly-facade / auth-hub`) のうち、**consumer 専門ではなく provider でもある 2 本**（`pre-processor`, `mq-hub`）は既に provider test が追加されていた。`rag-orchestrator / acolyte-orchestrator / recap-evaluator / alt-butterfly-facade / auth-hub` は dependency graph 上 consumer のみで provider としてどの pact にも現れないため、provider test 追加自体は不要。つまり真の gap は "selector の指定" と "mq-hub の Broker publish 対応"、それと Rust の publish 欠落だった。

## Decision

**Go と Python の全 provider test で consumer version selector を `Latest: true` から `MainBranch: true` + `DeployedOrReleased: true` のペアに置き換える**。`matchingBranch` は Alt の main-only 運用では `mainBranch` と同義で冗長なので採用しない。同時に、**mq-hub の provider test を file-based 単体から Broker publish 対応**に昇格し、`search-indexer ↔ mq-hub` のメッセージ pact が broker matrix に verification を残すようにする。

### 具体的な変更点

**Go**:
```go
// Before
verifyRequest.ConsumerVersionSelectors = []provider.Selector{
    &provider.ConsumerVersionSelector{Consumer: "alt-backend", Latest: true},
}

// After
verifyRequest.ConsumerVersionSelectors = []provider.Selector{
    &provider.ConsumerVersionSelector{Consumer: "alt-backend", MainBranch: true},
    &provider.ConsumerVersionSelector{Consumer: "alt-backend", DeployedOrReleased: true},
}
```

**Python**:
```python
# Before
builder = builder.consumer_version(consumer=consumer_name, latest=True)

# After
builder = builder.consumer_version(consumer=consumer_name, main_branch=True)
builder = builder.consumer_version(consumer=consumer_name, deployed_or_released=True)
```

**mq-hub の Broker 対応** (`mq-hub/app/driver/contract/provider_test.go`): 既存の file-based 呼び出しを、`PACT_BROKER_BASE_URL` がある場合は Broker 経由で publish する分岐に書き換え、`PACT_PROVIDER_VERSION` があれば `PublishVerificationResults = true` にする。他 Go provider test と同じコード構造に揃えた。

### 初回の bootstrap

selector の改善だけでは**今回の deploy で既に発生しているミスマッチ**は解けない。broker には production-deployed provider 版に対する verification がそもそも存在しないため。Pact 公式も GitHub Discussion #434 でこのケースを「webhook で deployed 版を再検証する」解決として定式化しているが、Alt の single-host OSS Broker 運用ではその webhook インフラを持っていない。ADR-000750 が最初の baseline 焼き付けで採用したのと同じ手段で bootstrap する:

```bash
./scripts/deploy.sh --skip-verify production
```

これで今回の HEAD を production baseline として `record-deployment × 14` が打刻され、次回以降の gate は:

1. Provider test が `deployedOrReleased` で deployed consumer pact も verify
2. Consumer pact content が変わらなければ content hash で verification が carry-forward
3. Consumer pact content が変わった場合でも provider の次回ビルドが deployed hash に対し verification を publish するので、次回 gate までに matrix が埋まる

### 検討した代替案

- **`matchingBranch: true` を追加**: Alt は main への直コミット運用で分岐が少なく、`mainBranch` と同じ結果を返すため冗長。導入時は PR ブランチの pact を拾う副作用で pending pact が増え、pending lifecycle を運用で管理するコストが増える。採用せず、必要になった時点で追加する方針とした。
- **webhook 駆動の自動再検証を実装**: Pact Broker の `contract_content_requiring_verification_published` event に応じて CI/provider ビルドをキックする仕組み。最も正攻法だが、Alt の compose-first single-host 運用にビルドキューを足すコストが本 ADR のスコープを超える。**別 ADR での対応候補**とする。
- **Rust 側 (recap-worker) を今回も selector 移行する**: `recap-worker/recap-worker/tests/provider_verification.rs` は `pact_verifier` crate を使わずハンドロールの axum stub + JSON 比較で実装されており、そもそも Broker への verification publish ができない。本 ADR では手を入れず、`recap-worker` の 3 consumer (`rag-orchestrator / recap-evaluator / search-indexer`) に対する verification 不在を既知の gap として残す。**別 ADR での `pact_verifier` 移行**が前提。
- **`enablePending` を無効化**: 新規 consumer 契約で provider build を即座に壊す厳格運用に寄せる案。ただし single-host で consumer が先行コミットできるワークフローとは相性が悪く、本 ADR では既存の `EnablePending = true` を維持。

## Consequences

### Pros

- `can-i-deploy` が SHA bump のたびに unknown で止まる再発防止になる。`deployedOrReleased` が常に deployed consumer pact を引っ張るので、provider test が走れば broker matrix の currently-deployed 側 cell が毎回更新される。
- Pact 公式の "Recommended Configuration for Verifying Pacts" に全面準拠。将来的に PactFlow や他 broker へ移行しても selector 設定を変えずに済む。
- mq-hub の message pact (`search-indexer ↔ mq-hub`) も Broker matrix に可視化されるようになり、[[000750]] で指摘された "broker matrix と現実の乖離" を埋める方向に一歩前進。
- 変更が provider test 側に閉じているため、consumer service への影響はゼロ。`pact-check.sh` のファイルベース mode (`./scripts/pact-check.sh` 単体) も引き続き動く（`PACT_BROKER_BASE_URL` 未設定で file fallback に落ちる分岐を保持）。

### Cons / Tradeoffs

- **Rust 側 (recap-worker) が依然として Broker に publish しない**。`rag-orchestrator / recap-evaluator / search-indexer` → `recap-worker` の 3 pair は本 ADR 後も verification が broker に入らないため、`can-i-deploy` が該当 cell で `unknown` を返しうる。本運用では HEAD = deployed の時期は問題にならないが、recap-worker が単独で pact 契約を変えたタイミングで consumer 側 gate が落ちる。**別 ADR で `pact_verifier` crate へ移行**することを前提にしている。
- **bootstrap deploy に `--skip-verify` が必須**。ADR-000750 の初回シードと同じ構造で、selector 改善だけでは今回既に発生しているミスマッチを解消できない。bootstrap 実行時の運用ログに理由を必ず残す（今回は本 ADR を根拠に記録）。
- **`deployedOrReleased` は "最低 1 回は production に record-deployment 済" であることが前提**。初回 deploy 前の pacticipant が混ざると selector が null を返し pact 0 件で verification が skip される。Alt はすでに ADR-000750 で全 14 pacticipant を production に seed 済なので該当しないが、将来新 service を追加する際はこの順序を守る必要がある（`scripts/_deploy_lib.sh::DEFAULT_LAYERS` への追加と並行して `pact-broker-cli record-deployment` の bootstrap コマンドを runbook に追記）。
- **consumer pact content が変わったタイミングの次回 deploy は、provider 側で再検証が走る時間分の遅延**が追加される。file-based の同期 gate なので pact-check.sh の実行時間 = 全 consumer/provider test 通し時間（現在 ~2 分）。webhook なら数秒で済むが、single-host の synchronous gate はこのコストを受容する。
