# ADR-000601: HybridPrioritySemaphore に home_pool スロット所有権追跡を導入しプリエンプション起因のスロットリークを修正する

## Status

Accepted

## Date

2026-03-27

## Affected Services

- news-creator (Python)

## Context

Visual Swipe UI でオンザフライ要約（`StreamSummarize`）を実行すると、RT（リアルタイム）ストリーミングリクエストがセマフォキューで **113 秒間**ブロックされ、タイムアウトする事象が発生した。nginx のアクセスログでは `rt=125s` / レスポンス 63 bytes（ハートビートのみ）が 2 回連続で記録された。

ローカル GPU（news-creator-backend / Ollama）は約 10 分間完全にアイドルだったにもかかわらず、`HybridPrioritySemaphore` の `_rt_available=0` のため RT リクエストがスロットを取得できない状態だった。

### 根本原因

`HybridPrioritySemaphore`（`total_slots=2, rt_reserved=1, be_slots=1`）のプリエンプション → リリースチェーンで、RT スロットが BE プールに永久移行するバグが存在した。

1. RT#1 が RT スロットを取得（`_rt_available: 1→0`）
2. RT#2 が到着 → RT スロットなし → BE プリエンプション発動 → BE スロットが解放 → RT#2 に転送
3. RT#1 が `release(was_high_priority=True)` → BE キュー待ちあり → BE ウェイターに転送。**`_rt_available` は 0 のまま**
4. RT#2 が同様に release → BE ウェイターに転送。**`_rt_available` は 0 のまま**
5. BE チェーン終了時に `release(was_high_priority=False)` → `_be_available = min(0+1, be_slots=1) = 1` でキャップ

最終状態: `_rt_available=0, _be_available=1` — 合計 1 スロット（本来は 2）。1 スロットが永久に消失。

### バグの本質

`release()` メソッドがスロットの「出身プール」ではなく「呼び出し元の優先度（`was_high_priority`）」に基づいて返却先を決定していた。プリエンプション経由で RT→BE に移行したスロットは、BE チェーン終了時に `was_high_priority=False` で返却されるため、`_be_available` にキャップされ、`_rt_available` に復帰しなかった。

これは PM-2026-004、PM-2026-006 で報告された HybridPrioritySemaphore の構造的問題の新しい発現パターンである。

## Decision

`AcquiredSlot` に `home_pool: str`（`"rt"` または `"be"`）フィールドを追加し、スロットの出身プールを追跡する方式を採用する。

### 検討した代替案

| 案 | 評価 |
|---|---|
| **A. total_slots ベース補填** — release 時に `available + acquired < total_slots` なら不足プールに補填 | 結果整合だが ownership バグを隠しやすい。返却先が偶然合っているだけで安全ではない |
| **B. home_pool 所有権追跡（採用）** — 各スロットに出身プールを記録し、release 時に home_pool に返却 | 既存の `_acquired_slots` トラッキングを拡張する最小差分。所有権が明確で正確 |

案 B を採用する。理由:
- 既存の `_acquired_slots: Dict[int, AcquiredSlot]` インフラを活用でき、最小差分で実装可能
- Trio の `CapacityLimiter` が borrower/token ownership で管理するベストプラクティスに準拠
- `release()` 時の返却先が常に正確（`was_high_priority` 推論に依存しない）

### 実装概要

1. **`AcquiredSlot` に `home_pool` フィールド追加** — `"rt"` or `"be"` でスロットの出身を記録
2. **`_track_acquire()` 全コールサイトで `home_pool` を明示** — `rt_immediate`, `hp_be_fallback`, `be_immediate`, `lp_rt_fallback`, `queued` の各パスで正しいプールを設定
3. **`release()` で `home_pool` に基づいてスロット返却** — `was_high_priority` ではなく `_acquired_slots[slot_id].home_pool` を参照
4. **ウェイター起床時に `home_pool` を伝播** — `future.set_result(home_pool)` でスロットの出身を次の取得者に引き継ぎ
5. **`acquire()` の queued パスで `home_pool` を読み取り** — `future.result()` から inherited pool を取得し `_track_acquire` に渡す
6. **invariant チェック追加** — `release()` 末尾で `rt_available + be_available + len(acquired_slots) != total_slots` を検知し ERROR ログを出力

## Consequences

### Pros

- プリエンプション → リリースチェーンでのスロットリークが根本的に解消される
- invariant ログにより将来のスロットリークが即時検知可能
- 既存の 59 テスト全パス + 新規 3 テスト追加で回帰リスクが低い
- `was_high_priority` 引数の既存インターフェースは変更不要（後方互換）

### Cons / Tradeoffs

- `release(slot_id=...)` を渡さない呼び出し元は、フォールバックで oldest matching slot を推定解放するため、完全な正確性は保証されない（段階的に `slot_id` 必須化を検討）
- `future.set_result()` の引数が `True` から `home_pool: str` に変更されたため、外部からセマフォの future を直接操作するコードがあれば影響がある（現時点では該当なし）
