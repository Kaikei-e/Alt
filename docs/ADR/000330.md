# fetchCtxタイムアウト延長 + rate limitエラーハンドリング修正

## ステータス

Accepted

## コンテキスト

フロントエンドで `[internal] An unexpected error occurred. (Error ID: aa0a9bf9)` が表示される問題が報告された。

### 根本原因

[[000329|ADR-329]]で導入した`fetchCtx`の15秒タイムアウトと、ホスト単位のrate limiter（10秒間隔、burst=1）の間にタイミングの不整合がある。同一ホスト（例: zenn.dev）の記事を連続フェッチすると、以下のメカニズムでエラーが発生する:

1. `fetch_article_usecase.go` で `fetchCtx` に15秒タイムアウトを設定
2. `fetch_article_gateway.go` で `rateLimiter.WaitForHost(fetchCtx, url)` を呼び出し
3. rate limiterの次トークン補充待ち時間（最大10秒）+ 実際のHTTPフェッチ時間が`fetchCtx`の残り時間を超過
4. `rate: Wait(n=1) would exceed context deadline` エラーが発生

さらに、このエラーはConnect-RPCハンドラーで適切に分類されず、`UNKNOWN_ERROR`（`errorhandler.HandleInternalError`経由）としてフロントエンドに返されていた。ユーザーにとっては意味不明な内部エラーとして表示され、同一ホストへの連続アクセスが原因であることが分からない状態だった。

### 発生シナリオ

Visual Previewモードでは最大10記事を500ms間隔でプリフェッチする。同一ホストの記事が複数含まれる場合、rate limiterのWait時間が累積し、後続リクエストのfetchCtxタイムアウトを超過しやすくなる。

## 決定事項

### 修正1: fetchCtxのタイムアウトを15秒から25秒に延長

`FetchCompliantArticle` のWebフェッチタイムアウトを `context.WithTimeout(ctx, 25*time.Second)` に変更。

**25秒にした理由:**
- rate limiterの最大待ち時間（10秒）+ 実際のHTTPフェッチ時間（最大10-15秒）を確実に吸収できる
- [[000329|ADR-329]]のスノーボール防止の目的は維持される: セマフォ(3) x 25秒 = 最悪75秒でリクエスト全体が収まる
- 15秒では「rate limiter待ち10秒 + HTTPフェッチ5秒超」のケースで頻繁にタイムアウトが発生していた

### 修正2: ハンドラーでrate limitエラーを適切に分類

Connect-RPCハンドラーの`FetchArticleContent`に、rate limitエラー（ゲートウェイが `"rate limit wait failed"` を含むエラーを返す）の分岐を追加。`context.DeadlineExceeded`チェックの**前**に配置し、`CodeResourceExhausted`として返す。

**DeadlineExceededの前に配置した理由:**
- rate limiterのWaitがコンテキストデッドラインを超過した場合、エラーチェーンに`context.DeadlineExceeded`が含まれる
- `errors.Is(err, context.DeadlineExceeded)` が先にマッチすると、rate limit起因のエラーが汎用的な「request timeout」として返されてしまう
- `strings.Contains` による文字列マッチを使用した理由: ゲートウェイのエラーラッピング（`fmt.Errorf("rate limit wait failed for %q: %w", ...)`）を通じてrate limitエラーであることを識別するため

### 修正3: テスト追加・更新

- ハンドラーテスト: rate limitエラー時に`CodeResourceExhausted`が返ることを検証
- ユースケーステスト: タイムアウト期待値を15秒から25秒に更新

## 結果・効果

### メリット

- **ユーザー体験の改善**: 同一ホストの記事を連続フェッチしても、rate limiter待ちがタイムアウトに収まるため正常に取得できる
- **エラーメッセージの適切化**: rate limitエラーが`UNKNOWN_ERROR`ではなく`ResourceExhausted`として返され、フロントエンドで適切なメッセージを表示できる
- **[[000329|ADR-329]]との整合性維持**: スノーボール防止のセマフォ制御は維持したまま、rate limiter待ちを吸収できるタイムアウト値に調整

### デメリット・トレードオフ

- **最悪ケースのレイテンシ増加**: 個別フェッチの上限が15秒から25秒に伸びるため、非常に遅いサイトへのフェッチがより長くブロックする可能性がある。ただし、セマフォ(3)による並列制限と組み合わせると、全体の最悪ケースは45秒→75秒に増加する
- **文字列ベースのエラー判定**: `strings.Contains(err.Error(), "rate limit wait failed")` はゲートウェイのエラーメッセージ文字列に依存しており、メッセージ変更時にハンドラー側の修正が必要になる。将来的にはカスタムエラー型の導入を検討する余地がある
- **タイムアウト値のハードコード**: 25秒は現在のrate limiter設定（10秒間隔）に合わせた値。rate limiter間隔を変更する場合はタイムアウトも再調整が必要

### エラーフロー（修正前 vs 修正後）

**修正前:**
```
rate limiter Wait超過 → "rate limit wait failed" → usecase "fetch failed: ..."
  → handler: errors.Is(DeadlineExceeded) or HandleInternalError → UNKNOWN_ERROR
```

**修正後:**
```
rate limiter Wait超過 → "rate limit wait failed" → usecase "fetch failed: ..."
  → handler: strings.Contains("rate limit wait failed") → CodeResourceExhausted
    → フロントエンド: "rate limited: please wait before fetching another article from this site"
```
