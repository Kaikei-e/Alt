# ADR-000644: Augur hybrid streaming の長文回答経路を安定化し nil retrieval を制御された fallback に変換する

## Status

Accepted

## Date

2026-04-08

## Affected Services

- rag-orchestrator (Go)
- alt-frontend-sv (TypeScript / SvelteKit)
- news-creator (Python, stream token budget の契約面)

## Context

[[000643]] で Ask Augur の strict long-form 経路を hybrid streaming に移行し、回答品質と perceived responsiveness の両立を狙った。しかし live 運用で 2 つの新しい障害モードが露出した。

1. **詳細因果クエリで internal error が発生する**
   - detailed causal query では `QualityGood` 以外を受理しないようにした結果、retrieval が `nil` を返すケースが生まれた
   - retry 経路の `buildPromptWithQueryPlanner` がこの `nil` を前提にしておらず、nil dereference で panic した
   - ユーザーには `Error: [internal] An unexpected error occurred.` として見え、回答品質以前に可用性を落としていた

2. **hybrid streaming の preview 粒度と進捗表示が粗い**
   - provisional delta の flush 単位が大きく、本文が固まりで出るため streaming の体感が弱い
   - corrective retry に入る前に `refining` を出しておらず、長い待ち時間の意味が UI に伝わらなかった

3. **query planner の intent が retrieval strategy に確実に渡らない**
   - planner が `causal_explanation` を返していても、retrieval 側に既存 intent が残ると期待した strategy selection とずれる余地があった
   - detail causal の厳格な retrieval gating を動かすには、planner 決定を retrieval の source of truth にする必要があった

4. **stream token budget の end-to-end 契約を固定したい**
   - long-form 経路では upstream が十分な `num_predict` を要求しても、stream path で縮むと最終回答は短くなる
   - 実装だけでなく downstream payload 契約をテストで固定する必要があった

今回の修正は、hybrid streaming を「とりあえず動く」段階から、live 運用で落ちず、詳細回答要求に対して一貫した品質制御を行う段階へ進めるために必要だった。

## Decision

**Augur の hybrid streaming long-form 経路に対して、retrieval failure を panic ではなく controlled fallback に変換する防御を追加し、detail causal query では planner intent を起点に厳格な retrieval gating と細粒度 preview streaming を適用する。**

具体的には以下を採用する。

1. `retrieveWithPolicy()` の結果が `nil` の場合は即時エラーとして扱い、panic させない
2. query planner の `intent` と `resolved_query` を retrieval strategy 側へ明示的に伝播する
3. detail causal query では `QualityGood` 以外を early accept せず、marginal retrieval を長文生成へ流さない
4. hybrid preview の flush 粒度を `80 runes` に下げ、retry 開始前に `refining` を送る
5. news-creator の `/api/chat` stream payload で `options.num_predict` が維持されることをテストで固定する

### 採用しなかった案

| 案 | 内容 | 判断 |
|---|---|---|
| marginal retrieval でも常に生成へ進める | `QualityMarginal` をそのまま long-form answer に流す | internal error は避けられるが、詳細要求に対して shallow answer を返しやすく、[[000643]] の品質方針と矛盾する |
| nil retrieval 時にダミー context を注入する | 空の `RetrieveContextOutput` を作って downstream を継続させる | 失敗理由が隠れ、fallback の意味が曖昧になる。壊れた回答の温床になる |
| raw token streaming に戻す | parser / flusher を外して token をそのまま delta で流す | provisional text の破綻、structured answer との不整合、retry 後の全文置換の複雑化が大きい |
| flush 粒度をさらに小さくする | 40-60 runes で頻繁に flush | 体感は軽くなるが、文の途中で切れやすく日本語回答の読みやすさが落ちる |

## Consequences

### Pros

- `nil retrieval` が発生しても panic せず、制御された fallback に落ちる
- planner が決めた `causal_explanation` が retrieval strategy に確実に伝わり、detail causal の厳格 gating が期待どおりに動く
- marginal retrieval を long-form answer の土台にしないため、詳細要求に対する shallow answer の混入が減る
- preview flush を 80 runes に下げたことで、本文の出始めが earlier and smaller になり、blank-feeling が緩和される
- retry 前に `Refining explanation...` / `refining` を送るため、UI 側の待機理由が説明可能になる
- `num_predict` 契約を tests で固定したため、stream path の token budget regressions を検知しやすい

### Cons / Tradeoffs

- detail causal query で `QualityGood` が得られない場合、以前より fallback に寄る。可用性ではなく品質を優先する判断になる
- 80-rune flush は 120-rune flush よりイベント数が増え、server / frontend 双方の処理回数が増える
- planner intent を retrieval 側の source of truth に寄せるため、planner の誤分類があればその影響がより強く出る
- controlled fallback にしたことで crash は防げるが、根本的に retrieval quality が不足している query では長文化できない
