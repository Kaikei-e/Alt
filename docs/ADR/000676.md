# ADR-000676: Recap 要約の出力トークンバジェットを分離し bullet 長品質ゲートを導入する

## Status

Accepted

## Date

2026-04-10

## Affected Services

- news-creator (Python 3.14+ / FastAPI / Ollama)

## Context

[[000638]] で 3-day Recap の prompt 形式を Gemma 互換に統一し、[[000640]] で Gemma 4 チャットテンプレート移行を完了したことで、構造面の品質は改善された。しかし、実運用データの分析により **bullet テキストの長さが品質基準を大幅に下回っている** ことが判明した。

### DB 分析結果（`recap_outputs` テーブル、2026-04-07〜09）

| 日付 | 総 bullet 数 | 平均 bullet 長 | 200 字超の bullet 率 |
|------|------------|---------------|---------------------|
| 4/07 | 465 | 141 chars | 13.1% |
| 4/08 | 310 | 148 chars | 11.0% |
| 4/09 | 173 | **115 chars** | **5.2%** |

プロンプトが要求する bullet 長は 3-day で 200〜400 文字、7-day で 400〜600 文字であり、**直近では 95% の bullet が 3-day の最低基準すら満たしていない**。さらに 4/09 では平均 `summary_ja` 長が 289 文字（ジャンルあたり）にまで縮小し、品質の劣化傾向が確認された。

### 根本原因

1. **`summary_num_predict = 1000` トークンの不足** — 3-day 出力（2-4 bullets × 200-400 chars + JSON 構造 + title + references）に必要な出力量は 1500-2000+ トークンだが、上限 1000 で切られていた。日本語 Gemma4 のトークン化は 1 CJK 文字 ≈ 1-2 トークンであり、バジェット超過で出力が途中打ち切りとなっていた
2. **hierarchical map フェーズは `summary_num_predict // 2 = 500` トークン** — 大ジャンルの中間要約がさらに制約され、最終 reduce に届く情報が極めて薄い
3. **bullet 最低長バリデーションの不在** — `_sanitize_summary_payload()` は最大長（1000 chars）と最大個数（15）のみを制限し、最低長チェックがゼロ。55 文字の stub bullet も素通りしていた
4. **`RecapQualityEvaluator.evaluate_readability()` が品質ゲートとして未接続** — 200-600 文字の理想範囲をスコアリングする機能は存在したが、生成パイプラインのリトライ判定には一切使われていなかった

## Decision

**Recap 専用の出力トークンバジェット `recap_summary_num_predict` を新設し、リトライループ内に平均 bullet 長の品質ゲートを追加する。**

### 1. `recap_summary_num_predict` を `summary_num_predict` から分離

`summary_num_predict` は記事要約（非 Recap）でも使用されており、直接変更すると副作用が生じる。Recap 専用のバジェットを独立して管理する。

- デフォルト値: `4000` トークン（従来の `1000` から 4 倍）
- hierarchical map/reduce: `recap_summary_num_predict // 2 = 2000` トークン（従来の `500` から 4 倍）
- 環境変数 `RECAP_SUMMARY_NUM_PREDICT` で上書き可能

### 2. 平均 bullet 長の品質ゲート

`_generate_single_shot_summary()` のリトライループ内（JSON parse 成功 + semantic validation 通過後）に、平均 bullet 長チェックを追加する。

- 閾値: `recap_min_avg_bullet_length = 300` chars（環境変数 `RECAP_MIN_AVG_BULLET_LENGTH` で上書き可能）
- 3-day のプロンプト目標は 400-800 chars なので、300 は最低限のソフトフロア
- 閾値未満の場合は `RuntimeError` を raise して既存のリトライループで再試行
- 最終リトライでも閾値未満の場合は、既存のフォールバック（deterministic extractive fallback）に落ちる

### 不採用とした代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| `summary_num_predict` を直接 2000 に変更 | 記事要約（非 Recap）に不要な出力増加。GPU 時間の浪費 |
| `RecapQualityEvaluator` 全軸をリトライ判定に統合 | 5 軸のヒューリスティック実行はオーバー。リトライで改善可能なのは「長さ」のみ。structure/entity_density は num_predict 増加で自然に改善する |
| プロンプトの目標文字数を下げる | 品質基準を下げるのは本末転倒。問題はプロンプトではなく出力バジェット |
| 品質ゲートを post-processing（リトライ外）に配置 | リトライの機会を活かせない。inline でリトライさせる方が最終品質が高い |

### サービス境界への影響

- recap-worker ↔ news-creator API: **変更なし**
- CDC 契約: **変更なし**
- Proto 定義: **変更なし**
- DB スキーマ: **変更なし**

### GPU 時間影響

`num_predict` を 1000 → 2000 に倍増するため、ジャンルあたりの最大デコード時間も最大 2 倍になる。ただし実際の出力は EOS で早期終了するため、実測の増加はこれより小さい。全ジャンル合計で +10 分程度の見込み。バッチ処理のため許容範囲内。

## Consequences

### Pros

- **出力トークンバジェットの 4 倍増** — Recap 専用の 4000 トークンにより、プロンプトが要求する 400-800 chars/bullet (3-day) / 800-1200 chars/bullet (7-day) の出力余地を確保。hierarchical map フェーズも 500 → 2000 トークンに改善
- **品質ゲートによる最低品質保証** — 平均 bullet 長 300 chars 未満の出力はリトライされ、短文 stub の最終出力混入を防止
- **プロンプト内 few-shot 例の長文化** — ADR-632 の知見（few-shot なしの小型モデルはスキーマ精度がほぼゼロ）に基づき、400+ 文字の bullet 例をプロンプトに含め、モデルの出力長を構造的に誘導
- **repeat_penalty の低減** — Recap 生成時の repeat_penalty を 1.15 → 1.0 に下げ、EOS トークンへの偏りを軽減（ADR-632: temperature=0 では repeat_penalty 削除で改善）
- **記事要約への影響ゼロ** — `summary_num_predict` は変更せず、Recap 専用のバジェット分離により副作用なし
- **環境変数による運用調整** — `RECAP_SUMMARY_NUM_PREDICT` と `RECAP_MIN_AVG_BULLET_LENGTH` でデプロイ後の微調整が可能
- **既存リトライ枠内で動作** — 品質ゲートは `max_repetition_retries`（デフォルト 2）の枠を消費し、新たなリトライメカニズムを追加しない

### Cons / Tradeoffs

- **GPU 時間の増加** — ジャンルあたり +16.7s（Remote A 基準）。バッチ全体で +9.5 分。バッチ処理のため実用上の問題はないが、iGPU/MPS ノードではレイテンシ影響が大きい
- **品質ゲートによるリトライ増加** — 短い出力が拒否されるため、リトライ 1 回追加で GPU 消費が +1 call 分増える可能性がある。ただし num_predict 増加により初回で閾値を超える確率が高い
- **150 chars の閾値は 3-day 目標（200-400）の 75%** — 閾値をさらに上げるとリトライ率が上がりすぎる。まずは 150 でソフトフロアを敷き、効果を計測してから調整する方が安全
- **品質ログ未実装** — `RecapQualityEvaluator` のスコアを生成時にログ記録する機能は今回見送り。将来のダッシュボード構築時に追加予定
