# ADR-000630: Augur RAG パイプラインを LLM 駆動クエリプランニング・cross-encoder 品質ゲート・日本語 BM25 修正で再設計する

## Status

Accepted

## Date

2026-04-06

## Affected Services

- rag-orchestrator (Go)
- news-creator (Python)
- search-indexer (Go)

## Context

Augur（Ask Augur）は 91 以上の ADR を経た漸進的改善にもかかわらず、根本的な回答品質の問題を解消できていなかった。直近の障害では、因果分析クエリに対して完全に無関係な記事（"Asset Tokenization"、"LibreFang"）が検索・引用され、129 文字の低品質回答が返された。フォローアップクエリでは照応解決が機能せず、`general` に誤分類されて short answer になっていた。

根本原因は 5 つの構造的問題に集約される:

1. **GPU 競合**: Local GPU 上でクエリ展開・意図分類・リランキング・回答生成が競合し、セマフォ枯渇で品質劣化
2. **小型モデル過負荷**: 4B/8B モデルに計画・分類・照応解決・生成の全責務を集中。instruction echo やクエリ展開ハルシネーションが発生
3. **日本語 BM25 不全**: Meilisearch の日本語トークナイズ未設定で BM25 が 0 件を返し、lexical 検索セーフティネットが機能しない
4. **Heuristic 品質ゲート**: 文字列マッチでクエリ-コンテキスト関連性を判定し、無関係記事を "good" と誤判定
5. **ルールベース会話プランナー**: `isAmbiguousFollowUp` 等のパターン列挙では照応解決が困難

過去の対応（ADR-607 因果意図分類、ADR-621 多層防御フィルタ、ADR-604 ConversationPlanner 等）はいずれも対症療法に留まり、アーキテクチャレベルの再設計が必要と判断した。

## Decision

**Augur のオーケストレーション層を 3 つの独立した改善で再設計する。**

### 1. LLM 駆動クエリプランニング（news-creator + rag-orchestrator）

ルールベースの `ConversationPlanner` + `QueryClassifier` + `expandQuery` を、news-creator 上の LLM structured output ベースの `PlanQuery` エンドポイントに統合する。

Ollama の JSON Schema ベース structured outputs（`format` パラメータ）を使い、1 回の LLM 呼び出しで以下を一括生成:

```json
{
  "resolved_query": "自己完結した検索クエリ（照応解決済み）",
  "search_queries": ["日英展開クエリ 3-5 件"],
  "intent": "causal_explanation | temporal | synthesis | ...",
  "retrieval_policy": "global_only | article_only | ...",
  "answer_format": "causal_analysis | summary | ...",
  "should_clarify": false,
  "topic_entities": ["主要エンティティ"]
}
```

**news-creator 側**: `POST /api/v1/plan-query` エンドポイント。`recap_summary_usecase.py` で実績のある `format=Model.model_json_schema()` パターンを再利用。JSON パース失敗時は元のクエリをそのまま使う deterministic fallback。

**rag-orchestrator 側**: `domain.QueryPlannerPort` interface + `WithQueryPlanner` option。`buildPromptWithQueryPlanner` で legacy パスを完全にバイパスし、`resolved_query` を retrieval に、`intent` を strategy 選択に使用。`should_clarify=true` の場合は retrieval をスキップして clarification イベントを返す。

legacy パスはそのまま残し、`WithQueryPlanner` が設定されていない場合は従来通り動作する。

### 2. Cross-encoder スコアベースの Relevance Gate（rag-orchestrator）

heuristic ベースの `hasQueryContextMismatch`（文字列マッチ、CJK セグメント分割）を、cross-encoder reranker スコアの top-1 閾値判定に置き換える。

```
top-1 score >= 0.5  → Good（生成に進む）
top-1 score >= 0.25 → Marginal（QueryPlanner に再計画依頼 → 1 回リトライ）
top-1 score <  0.25 → Insufficient（fallback: 「情報が不十分です」）
```

既存の `bge-reranker-v2-m3` を活用。`RelevanceGate` struct + `WithRelevanceGate` option で usecase に注入。

### 3. 日本語 BM25 修正（search-indexer）

Meilisearch v1.27 の `localizedAttributes` を設定し、日本語コンテンツの正しいトークナイズを有効化。

- **index 設定**: `localizedAttributes` に `jpn` + `eng` の 2 ルールを設定。Meilisearch がフィールドごとに言語を自動検出
- **検索リクエスト**: `containsCJK()` でクエリの文字種を判定し、CJK 含有時は `Locales: ["jpn"]` を指定
- **対象フィールド**: `title`, `content`, `tags`

### 4. オフライン評価基盤（rag-orchestrator）

改善を定量的に測定するための eval harness を新設。

- **golden case**: 12 件（既知障害 4 件 + 技術・産業系の追加ケース 8 件）
- **指標**: recall@20, nDCG@10, top-1 precision, faithfulness, citation correctness, BM25 zero rate, follow-up resolution rate, clarification precision, fallback rate
- **replay runner**: golden case を pipeline に通し、before/after 比較が可能な JSON レポートを出力
- **pipeline adapter**: `AnswerWithRAGUsecase.Execute` をラップして `EvalResult` を生成

### 検討した代替案

| 代替案 | 評価 | 却下理由 |
|-------|------|---------|
| Claude API をオーケストレーション層に使用 | 品質は最高だが外部依存 | Alt のセルフホスト原則に反する |
| 3 台のマシンに物理分離 | GPU 競合解消に有効 | 単一マシン前提を維持する方針を優先 |
| 既存 heuristic フィルタの拡張 | 低コスト | 91 ADR の経験から対症療法の限界が明確 |

## Consequences

### Pros

- **オーケストレーション品質向上**: LLM が照応解決・意図分類・クエリ展開を一括処理。instruction echo や誤分類が構造的に排除される
- **品質ゲートの信頼性向上**: cross-encoder スコアは文字列 heuristic より遥かに正確。"Asset Tokenization" のような無関係記事は score < 0.1 で確実に排除される
- **日本語検索の正常化**: BM25 が 0 件を返す問題が解消。lexical セーフティネットが機能する
- **段階的移行**: `WithQueryPlanner` / `WithRelevanceGate` の option 注入で、legacy パスと新パスを DI レベルで切り替え可能
- **定量評価**: before/after のメトリクス比較で改善を客観的に判定。感覚的なプロンプト調整からの脱却
- **Pact CDC 契約**: rag-orchestrator → news-creator の `plan-query` 契約が consumer test で保護される

### Cons / Tradeoffs

- **LLM planning のレイテンシ追加**: `plan-query` 呼び出しに 1-3 秒。ただし legacy の expand-query（同等時間）を置換するため純増ではない
- **セマフォ負荷**: planning + generation が同一 GPU を共有。news-creator の `HybridPrioritySemaphore` が planning を HIGH priority で処理し、BE 要約との競合を管理
- **structured output の品質**: gemma4-e4b の JSON 出力品質に依存。JSON パース失敗時は deterministic fallback で安全に退避
- **閾値のキャリブレーション**: `RelevanceGate` の 0.5/0.25 閾値は eval set で校正が必要。コーパスやモデル変更時に再調整が必要
- **legacy コード残存**: 新旧パスの並存により一時的にコード量が増加。P1-3 で legacy 削除予定
