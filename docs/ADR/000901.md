# ADR-000901: Go 製マイクロサービス全 16 モジュールの依存ベースラインを 2026-05-18 時点 latest へ揃える

## Status

Accepted

## Date

2026-05-18

## Affected Services

- alt-backend (Go)
- auth-hub (Go)
- pre-processor (Go)
- pre-processor-sidecar (Go)
- knowledge-sovereign (Go)
- mq-hub (Go)
- search-indexer (Go)
- alt-butterfly-facade (Go)
- rag-orchestrator (Go) + その内部 proto submodule
- altctl (Go CLI)
- pki-agent (Go sidecar)
- alt-backend/sidecar-proxy (Go)
- credential-health-checker (Go sub-tool)

## Context

Go 製マイクロサービス間で `go.mod` の `go` ディレクティブ、`pact-go/v2`、Docker base image digest、`golang.org/x/{net,image}` のセキュリティ対象バージョンに drift が累積していた。具体的には:

- `go 1.26.2` のままのモジュールが 15 個。Go 1.26.3 は `crypto/fips140`、`crypto/tls`、`go/types`、`net/http`、`net/mail`、`syscall` 系のセキュリティ修正を含むパッチ。
- `pact-foundation/pact-go/v2` が `v2.4.2` と `v2.4.3` の二バージョン混在。Pact CDC ペア (alt-backend ↔ pre-processor、auth-hub ↔ alt-backend など) で contract spec の互換は破れていなかったが、将来の major bump 時にどちらに揃えるかの判断材料が散らかる。
- `golang.org/x/net` (GO-2026-4918 HTTP/2 infinite loop)、`x/image` (WEBP/SFNT/TIFF 系 GO-2026-4815/4961/4962) を含むパッケージが、サービスによっては古い minor を間接依存していた。
- `alt-backend` が `pashagolub/pgxmock/v3` (3.4.0) と `pashagolub/pgxmock/v4` (4.9.0) を同居させていた。v3 / v4 / v5 の API 差により mock セットアップが file ごとに揺れていた。`knowledge-sovereign` も v3 を直接依存していた。
- `pre-processor-sidecar` だけ `database/sql + lib/pq` でレポジトリ層を書いていた。`lib/pq` 自体は maintenance-only モード (現行 v1.12.3 で凍結)、Alt の他サービスは `jackc/pgx/v5 + pgxpool` 標準。drift の主因。
- Docker base image (`golang:1.26-alpine` と `gcr.io/distroless/static-debian12:nonroot`) を digest 固定していないサービスがほとんどで、reproducible build が破れていた。
- `pki-agent` の runtime ベース `smallstep/step-ca:0.27.5` は 3 minor 遅れ (latest は 0.30.2)。
- `mq-hub` の test 専用依存 `alicebob/miniredis/v2` が 4 minor 遅れ (v2.34.0 → v2.38.0)。
- `pre-processor-sidecar` と `credential-health-checker` の `k8s.io/*` が v0.36.0、現行 v0.36.1 patch あり。

調査時点 (2026-05-18) で 42 依存中 36 が既に最新で、動かす対象は上記に集中していた (調査記録は `/home/koko/.claude/plans/alt-go-plan-context-loader-tdd-workflow-virtual-ocean-agent-ae4b06124b96809cb.md`)。

## Decision

以下を採用する:

1. **Go toolchain を全 15 `go.mod` で `go 1.26.3` に統一する。** 1.27 はまだ未リリース。
2. **`golang:1.26-alpine` の digest を `sha256:91eda977…1079d` に固定する** (Go 1.26.3 を内包)。全 12 Dockerfile に適用。`gcr.io/distroless/static-debian12:nonroot` も `sha256:d093aa3e…16639` で固定する。`alt-backend/Dockerfile.backend` の従来 pin は `golang:1.26.2-alpine` の digest だったため上書き。
3. **`pact-foundation/pact-go/v2` を全サービスで `v2.4.3` に揃える** (drift 解消)。Pact spec V2/V3/V4 互換は維持される minor patch。
4. **`pashagolub/pgxmock` を v5.1.0 (major bump) に統一する** (alt-backend / knowledge-sovereign)。`v3` と `v4` の直接依存を削除。v5 の breaking change である `RawValues` の `pgtype` 化、`RowSet` の cell-less 行受け入れに対応するため、`alt-backend/app/rest/article_handlers_test.go` の `INSERT INTO articles` 期待 row を `RETURNING id, (xmax = 0) AS created` に合わせて 2 列にする。
5. **`pre-processor-sidecar` の `database/sql + lib/pq` を `jackc/pgx/v5 + pgxpool` に移行する。**
   - `repository/article_repository.go`、`subscription_repository.go`、`sync_state_repository.go` を `PgxIface` (pgxpool.Pool / pgxmock.PgxPoolIface のどちらも満たす interface) ベースに書き換える。
   - 既存テストの `go-sqlmock` を `pgxmock/v5` に置換。
   - `cmd/main.go` の DB 初期化を `sql.Open("postgres", …)` から `pgxpool.NewWithConfig(ctx, poolCfg)` に切り替える。
   - tx の `PrepareContext + stmt.ExecContext` パターンは廃止し、pgx の prepared statement キャッシュに任せて `tx.Exec(ctx, query, args...)` 直接呼び出しに統一。
6. **`pre-processor-sidecar` と `credential-health-checker` の `k8s.io/*` を `v0.36.1` に bump する。**
7. **`mq-hub` の `alicebob/miniredis/v2` を `v2.38.0` に bump する。**
8. **`pki-agent` の `smallstep/step-ca` を `0.30.2` に bump する** (Dockerfile)。`step-cli` インターフェース (cert request / renew / OTT) の互換は保たれており、`pki-agent` の usecase 層は無変更。
9. **`rag-orchestrator` で `oasdiff/yaml` を `replace` ディレクティブで `v0.0.9` に pin する。** `go mod tidy` がトランジティブに `v0.1.0` を引いてくるが、`kin-openapi v0.138.0` がまだ削除済みの `UnmarshalWithOriginTree` を使うため build が壊れる。`replace github.com/oasdiff/yaml => github.com/oasdiff/yaml v0.0.9` と `replace github.com/oasdiff/yaml3 => github.com/oasdiff/yaml3 v0.0.12` を追加。
10. **Echo v4 → v5 (Phase 4) は本 ADR の範囲から外す。** 後述。

### 検討して却下した代替案

- **`lib/pq` をそのまま残す**: 「依存だけ更新」スコープを守れば最小工事だが、`lib/pq` は maintenance-only で security fix も来ない実質的に dead な driver。Alt の他サービスとの drift を一回で潰す方が、後年の脆弱性対応で困らない。
- **`pgxmock` を `v4.9.0` に統一**: v5 を避ければ test code 変更は import path だけで済む。ただし v5 の `RowSet` 強化と `pgtype` 整合性は今後の DB driver 更新で恩恵があるため、今回まとめて踏み越える。
- **Echo v4 → v5 を同じ ADR で吸収**: 当初の plan では Phase 4 として組み込んでいた。実装直前に以下を確認して別 plan に切り出すと判断:
  - `go.opentelemetry.io/contrib/instrumentation/github.com/labstack/echo/otelecho` に v5 系統が go module proxy に存在しない (v4 contrib の最新 `v0.68.0` まで)。Alt の Echo サービス (alt-backend / auth-hub / pre-processor) は OTel trace を otelecho 経由で取っているため、v5 では `otelhttp` ベースの内製 middleware で代替する必要がある。
  - Echo v5 は `Context` を interface → 具象 struct (handler 署名 `echo.Context` → `*echo.Context`)、JWT middleware を `labstack/echo-jwt/v5` に分離、`RealIP()` の挙動を `IPExtractor` 必須化、logger を `log/slog` に置換、と breaking change 群が広い。alt-backend だけで `echo.*` 参照箇所が約 314、auth-hub 101、pre-processor 54 で合計 469。
  - Echo v4 は upstream で 2026-12-31 まで bug + security fix 継続。今すぐ落とす必要はない。
  - これらを 1 セッションで安全に閉じるのは困難。otelecho 代替の design + alt-backend の handler 群を分割 PR で対応する別 ADR/plan に切り出すべき。

## Consequences

### Pros

- **セキュリティ**: Go 1.26.3 の crypto/fips140 / net/http / syscall 系 fix、`x/net` HTTP/2 infinite loop fix、`x/image` の WEBP/SFNT/TIFF 系 fix を取り込み済み。
- **再現性**: 全 Dockerfile で `golang:1.26-alpine` を SHA digest 固定。`alt-backend` 以外も含めて reproducible build が成立。
- **drift 解消**: `pact-go v2.4.3`、`pgxmock v5.1.0`、`jackc/pgx/v5` ベースの DB 層が monorepo 横断で統一。レビュアと将来の bump 担当が `go list -m -u all` を眺めるときの揺れが消える。
- **保守責任の明確化**: `lib/pq` (maintenance-only) と `go-sqlmock` (database/sql 前提) を Alt の Go コード全体から退役。CVE 対応のとき "どのサービスが lib/pq か" を毎回確かめる必要がなくなる。
- **依存テスト容易性**: `pre-processor-sidecar` が `PgxIface` (knowledge-sovereign / alt-backend と同じ pattern) ベースになったので、テスト fake が他サービスと作法が揃う。

### Cons / Tradeoffs

- **Echo v5 が未完**: Phase 4 として計画していたが、otelecho 未対応と scope (469 site) のため別 plan に持ち越し。Echo v4 サポート切れ (2026-12-31) までに対処が必要な未解決負債。
- **既存テストの 1 件修正**: alt-backend の `TestHandleFetchArticle_Compliance/Allowed_and_Fetched` で v3/v4 では暗黙に許容されていた "1 column rows を 2 列 Scan に渡す" pattern が v5 で fail するようになり、テスト側を仕様 (`RETURNING id, (xmax = 0) AS created`) に合わせて修正した。
- **既存の pre-existing failure**: `pki-agent` の `TestTick_Missing_IssueUpdatesObserverClassified`、`altctl` の docker daemon 依存テスト、`mq-hub` の `TestHandler_GenerateTagsForArticle` (stream key 不一致) は本依存更新前から赤。bump とは無関係だが本 ADR 中で潰していない。
- **kin-openapi v0.138.0 / oasdiff yaml の脆い噛み合わせ**: `replace` で yaml v0.0.9 に固定する形で凌いだが、`kin-openapi` が v0.138.0 を超えて bump するときに `replace` 解除が必要。`go.mod` 末尾に意図と理由をコメントで記録済み。
- **本 ADR は本番 deploy を伴わない**: テストのみで動作確認。コンテナ再ビルドは行わず、`docker compose ... up --build` も叩いていない。本番反映はユーザが `git push origin main` を叩いて `dispatch-deploy.yaml` 経由で適用する。
