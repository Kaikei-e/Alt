# ADR-000575: recap-subworker の Numba threading layer を TBB に切り替え、BE 要約モデルオーバーライドを修正する

## Status

Accepted

## Date

2026-03-25

## Affected Services

- recap-subworker (Python/FastAPI)
- news-creator (Python/FastAPI)

## Context

### 問題 1: Numba workqueue concurrent access エラー

recap-subworker の `/health` エンドポイントがタイムアウトし、連鎖的に recap-worker の `/health/ready` も応答不能になっていた。ログには以下のエラーが繰り返し出力されていた:

```
Numba workqueue threading layer is terminating: Concurrent access has been detected.
```

**根本原因**: Numba 0.64.0 のデフォルト threading layer 選択順は `tbb > omp > workqueue` だが、`tbb` Python パッケージが未インストールのため、非スレッドセーフな `workqueue` にフォールバックしていた。recap-subworker は `RECAP_SUBWORKER_PIPELINE_WORKER_PROCESSES=2` で複数プロセスを起動し、さらに `ThreadPoolExecutor` で HDBSCAN をタイムアウト付きで実行するため、並行アクセスが発生していた。

`main.py` では `OMP_NUM_THREADS=1`, `MKL_NUM_THREADS=1`, `OPENBLAS_NUM_THREADS=1` を設定済みだったが、Numba の threading layer 制御（`NUMBA_THREADING_LAYER`）は未設定だった。

### 問題 2: BE 要約ジョブの異常な遅延

BE article summarization ジョブの大半が 12〜30 秒で完了する中、一部のジョブが 163 秒を要していた。原因は `.env` の `DISTRIBUTED_BE_MODEL_OVERRIDES` に特定リモートへの RAG 用モデル（`gemma3-4b-rag:latest`）オーバーライドが設定されていたこと。

このオーバーライドは元々「RAG モデルの VRAM evict を防ぐ」目的で設定されたが、[[000566]] の修正で `SummarizeUsecase` が DistributingGateway 経由になった結果、article summarization の BE ジョブもこのリモートに送られ、低速な RAG モデルで処理される副作用が顕在化した。

## Decision

### 修正 1: Numba TBB threading layer の有効化

**`tbb` パッケージを依存に追加し、`NUMBA_THREADING_LAYER=tbb` を明示的に設定する。**

TBB (Intel Threading Building Blocks) は concurrent access に対応した threading layer であり、`workqueue` や `omp` と異なりマルチスレッド環境で安全に動作する。

変更箇所:
1. `recap-subworker/pyproject.toml` — `tbb>=2022.0` を dependencies に追加
2. `recap-subworker/recap_subworker/app/main.py` — `os.environ.setdefault("NUMBA_THREADING_LAYER", "tbb")` を既存のスレッド制御設定の直後に追加
3. `recap-subworker/Dockerfile.recap-subworker` — `NUMBA_THREADING_LAYER=tbb` と `LD_LIBRARY_PATH=/app/.venv/lib` を ENV に追加（pip install された TBB の shared library を Numba が検出するために必要）
4. `compose/recap.yaml` — 環境変数 `NUMBA_THREADING_LAYER=tbb` と `LD_LIBRARY_PATH=/app/.venv/lib` を追加

**注意**: pip で `tbb` をインストールした場合、`libtbb.so` は venv の `lib/` に配置されるが、デフォルトの `LD_LIBRARY_PATH` にはこのパスが含まれない。明示的に `LD_LIBRARY_PATH` を設定しないと Numba が TBB を検出できず、`ValueError: No threading layer could be loaded` が発生する。

### 修正 2: モデルオーバーライドの削除

**`DISTRIBUTED_BE_MODEL_OVERRIDES` を空にし、全リモートで統一された BE 要約モデル（`gemma3:4b-it-qat`）を使用する。**

対象リモートには既に `gemma3:4b-it-qat` がロード済みであることを確認した上でオーバーライドを削除した。

## Consequences

### Pros

- recap-subworker の `/health` が即時応答するようになり、recap-worker の readiness probe も復旧
- Numba concurrent access エラーが解消され、UMAP/HDBSCAN の並行処理が安定
- 全リモートが同一モデルを使用することで、BE 要約ジョブの所要時間が均一化（12〜30 秒）
- 163 秒の異常遅延ジョブが解消

### Cons / Tradeoffs

- `tbb` パッケージの追加により Docker イメージサイズが微増（~10MB）
- `LD_LIBRARY_PATH` の明示的設定が必要（pip install の TBB は Numba の自動検出に対応していない）
- 対象リモートで RAG モデルと BE モデルの両方が VRAM に常駐するため、VRAM 使用量が増加する可能性がある。ただし gemma3 4B モデルは Q4_0 量子化で約 4GB のため、8GB VRAM でも 2 モデル共存は可能
