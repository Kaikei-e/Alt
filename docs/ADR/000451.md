# ADR-000451: UTF-8 安全な excerpt 切り詰めを共通化して Knowledge Home 投影を保護する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go)

## Context

Knowledge Home の card は `summaryState` が `pending` または `missing` の間、要約本文の代わりにローディング用の skeleton を表示する。今回、`news-creator` 自体は要約生成に成功している一方で、`alt-backend` の Knowledge Home projector が `SummaryVersionCreated` を投影する段階で `invalid byte sequence for encoding "UTF8"` を継続的に出していた。

原因は要約 excerpt の切り詰めを byte 単位で実装していたことにある。日本語などの multibyte 文字列を途中で切ると壊れた UTF-8 になり、PostgreSQL が `knowledge_home_items.summary_excerpt` への upsert を拒否する。projector は失敗をログ出力して継続するため、該当 item は `summaryState=ready` へ進まず、Knowledge Home 上では「要約生成済みなのにローディングが消えない」状態になる。

同種の切り詰めは `SummarySuperseded` event の `previous_summary_excerpt` 生成にも存在し、Knowledge Home 本体だけ直しても supersede 系の payload で同じ破損を再導入する余地があった。

## Decision

**Knowledge Home 系の excerpt 切り詰めは、すべて UTF-8 妥当性を保証する共通 helper を経由させる。**

具体的には以下を採用する。

- `alt-backend/app/utils/textutil/utf8.go` に `TruncateValidUTF8` を追加する
- helper は `strings.ToValidUTF8` で不正 sequence を除去したうえで、rune 単位で上限文字数まで切り詰める
- 実際に切り詰めた場合のみ `…` を末尾に追加する
- `job/knowledge_projector.go` の `SummaryVersionCreated` excerpt 生成はこの helper を使う
- `usecase/create_summary_version_usecase` の `previous_summary_excerpt` 生成も同じ helper を使う

代替案として、PostgreSQL 直前で invalid UTF-8 を黙って置換する案も検討できるが、それでは「どこで excerpt を作るか」が呼び出し元ごとに散らばり、将来の event payload や別 read model で同じ不具合を繰り返しやすい。今回は切り詰め責務そのものを共通化し、producer 側で妥当な文字列を作る方針を採用する。

## Consequences

### Pros

- 日本語や混在言語の要約でも `knowledge_home_items` への upsert が失敗しにくくなる
- `SummaryVersionCreated` と `SummarySuperseded` の双方で excerpt 生成の挙動が統一される
- Knowledge Home の `pending` 固着を UI 修正なしで解消できる
- 回帰テストで multibyte 文字列の切り詰めを固定できる

### Cons / Tradeoffs

- 文字数上限は byte ではなく rune 数になるため、従来より保持される byte 数は増えることがある
- `strings.ToValidUTF8` で不正 sequence を除去するため、既に壊れた入力の一部文字は失われる
- excerpt 生成ポリシーが helper に集約されるため、将来別の表示用途で異なるルールが必要になった場合は API 分岐が必要になる
