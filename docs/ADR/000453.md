# ADR-000453: Knowledge Home 検索遷移時に URL クエリパラメータを読み取り自動検索を実行する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-frontend-sv (SvelteKit)

## Context

Knowledge Home の検索ボックス（`UnifiedIntentBox.svelte`）でクエリを入力すると、`goto('/feeds/search?q=...')` でフィード検索ページに遷移する。しかし遷移先の `(app)/feeds/search/+page.svelte` が URL の `q` パラメータを一切読み取っておらず、`searchQuery` が常に空文字で初期化されていた。このため、Knowledge Home から検索しても空の検索ページが表示され、ユーザーが再度クエリを入力する必要があった。

加えて、`hooks.server.ts` のレスポンシブリダイレクト（`/desktop/feeds/search` → `/feeds/search` 等）がクエリパラメータを破棄していたため、レガシールート経由でアクセスした場合にも `?q=` が消失する副次的問題があった。

調査の結果、バックエンド・Meilisearch・search-indexer はすべて正常に稼働しており、根本原因はフロントエンドの URL パラメータ未読のみであることを確認した。

## Decision

以下の方針で修正する。

### 1. 検索ページで URL `q` パラメータを読み取り、自動検索を実行する

`page` from `$app/state` を使って `searchParams.get("q")` を読み取る。これは `recap/+page.svelte` や `home/+page.svelte` で確立済みの既存パターに合わせた選択である。

Desktop 表示では `onMount` 内で `handleSearch()` を呼び出す。Mobile 表示では `SearchFeedsClient` → `SearchWindow` に `initialQuery` / `autoSearch` prop を伝播させ、`SearchWindow` の `onMount` で検索を自動実行する。

**`$effect` ではなく `onMount` を選択した理由:** `$effect` で `page.url.searchParams` を監視すると、URL が変わるたびに検索が再実行される。検索ページでは入力フィールドの値がユーザー制御の状態であり、初回ロード後は URL パラメータに追従すべきではない。`recap/+page.svelte` の既存パターンとも一致する。

### 2. レスポンシブリダイレクト時にクエリパラメータを保持する

`hooks.server.ts` のリダイレクトロジックを `resolveResponsiveRedirect()` 関数として `redirect-resolver.ts` に抽出し、`url.search` を付与してリダイレクトする。

リダイレクト先が既にクエリパラメータを含む場合（`/recap?window=7`）は `&` で連結する。このロジックは `buildLegacySvRedirect`（`legacy-sv.ts`）と同様のパターンである。

抽出の理由は、純粋関数としてユニットテスト可能にするためであり、`legacy-sv.ts` / `route-guard.ts` / `response-builder.ts` と同じ設計方針に従う。

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|--------|------|-----------|
| `+page.server.ts` の `load` 関数でクエリを渡す | サーバーサイドで URL パラメータを読み、`data` prop として渡す | 検索 API 呼び出しはクライアントサイドで行っており、サーバーロードは不要なオーバーヘッド |
| `$derived` + `$effect` で URL パラメータを監視 | URL 変更に自動追従 | ユーザーがフィールドに入力した値を URL 変更で上書きしてしまう。入力フィールドの主権はユーザーにあるべき |
| `hooks.server.ts` 内にリダイレクトロジックを残す | 行数が少なく済む | テスト不可能なインラインロジックが残る。既存の helper 抽出パターンと不整合 |

## Consequences

### Pros

- Knowledge Home → フィード検索のフローが正常に動作し、ユーザーが再入力する必要がなくなる
- レガシールート（`/desktop/feeds/search?q=...`、`/mobile/feeds/search?q=...`）からのリダイレクトでもクエリが保持される
- `resolveResponsiveRedirect` が純粋関数として 7 件のテストケースでカバーされ、今後のリダイレクトルール追加時にも安全に変更できる
- Desktop / Mobile 両方で同一の `initialQuery` パターンを使い、コードの一貫性を保っている

### Cons / Tradeoffs

- `SearchWindow.svelte` に `autoSearch` prop が追加され、インターフェースがやや広がった。ただし optional prop のため既存の利用箇所に影響はない
- `RESPONSIVE_REDIRECTS` マップが `hooks.server.ts` から `redirect-resolver.ts` に移動したため、リダイレクトルールの追加時は新しいファイルを編集する必要がある
- SPA 内遷移で同じページに異なる `?q=` で再遷移した場合、SvelteKit がコンポーネントを再マウントするため正しく動作するが、`pushState` のみの URL 変更には反応しない（現時点でそのユースケースはない）
