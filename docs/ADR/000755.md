# ADR-000755: Knowledge Home の TrackHomeAction body から記事タイトル・要約を除去し上流 WAF の記事単位 403 を解消する

## Status

Accepted

## Date

2026-04-17

## Affected Services

- `alt-frontend-sv/src/routes/(app)/home/+page.svelte` — `handleAction` の metadata 組み立て処理を新モジュール呼び出しに差し替え
- `alt-frontend-sv/src/routes/(app)/home/home-actions.ts` — 新設。`buildHomeActionMetadata` は `undefined` を返すのみだが、「記事本文片を body に入れない」ポリシーの単一の居場所として regression guard を兼ねる
- `alt-frontend-sv/src/routes/(app)/home/home-actions.test.ts` — dismiss / open / open_recap / open_search / ask / listen の全 action type で metadata が `undefined` であることを検証

alt-backend / proto / BFF / nginx / DB は**一切変更していない**。イミュータブル不変条件（append-first / reproject-safe / versioned projection）も無変更で温存。

## Context

本番 (`curionoah.com`) で Knowledge Home の Dismiss が**特定 1 記事**に対してのみ
`POST /api/v2/alt.knowledge_home.v1.KnowledgeHomeService/TrackHomeAction` で
`net::ERR_ABORTED 403 (Forbidden)` を返すと報告された。他の記事の Dismiss は
すべて 200 で完走していた。

### コンテナログから確定した事実

1. **nginx access log**（直近 24h）: `TrackHomeAction` は**全件 `200 12`**
   （Connect 空レスポンス 12 バイト）。403 は観測されない。
2. **alt-backend log**:
   `mtls peer=alt-butterfly-facade path=/alt.knowledge_home.v1.KnowledgeHomeService/TrackHomeAction`
   が繰り返し記録され、BE は正常に usecase を完走。
3. **経路**: `compose/core.yaml:72` の
   `BACKEND_CONNECT_URL=http://alt-butterfly-facade:9250` により、
   本番は FE → nginx → SvelteKit proxy (`/api/v2/[...path]/+server.ts`)
   → BFF (mTLS) → alt-backend となっていた。BFF bypass は起きていない。
4. **nginx の `remote_addr` が `172.18.0.1`**（docker bridge）: nginx より
   前段に外部 ingress（cloudflared 等）が存在し、403 はその前段で生成された
   ことを示唆する。

→ **403 は Alt 自前インフラの外側（最有力は Cloudflare WAF）で発生**し、
しかも**特定 1 記事だけ**で発生するため、パス名に含まれる `Track` という
語ではなく **POST ボディ内容**が WAF rule に反応していると断定できる。

### なぜその記事だけか

`alt-frontend-sv/src/routes/(app)/home/+page.svelte:164-170` が
`handleAction` 内で以下のように毎回 article content を body に載せていた:

```ts
const metadata = JSON.stringify({
    articleId: item.articleId,
    title: item.title,                 // 記事タイトルをそのまま送信
    summaryExcerpt: item.summaryExcerpt, // 要約本文もそのまま送信
});
```

一方 `alt-backend/app/usecase/track_home_action_usecase/usecase.go:167-178`
は `meta["query"]` と `meta["tag"]` しか読まず、
`articleId` / `title` / `summaryExcerpt` は完全に未使用だった。つまり FE は
**サーバが使わない記事本文片を毎回 POST body に詰めていた**ため、ある記事の
タイトル・要約文字列が上流 WAF の managed rule にヒットし、
"Maestro MCP / Claude / Android の E2E テスト" を題材にした記事 1 本だけ
ブロックされていた。

## Decision

**FE 側で TrackHomeAction の `metadata` を必要最小限に絞る**。
記事タイトル・要約・articleId を body に載せる実装を全廃し、
新モジュール `home-actions.ts` の `buildHomeActionMetadata` に
「記事本文片を body に入れない」ポリシーを集約する。現状サーバが
必要とする情報は action type と item key のみのため、
`buildHomeActionMetadata` は常に `undefined` を返す。

差分のエッセンス:

```diff
 function handleAction(type: string, item: KnowledgeHomeItemData) {
     const itemKey = item.itemKey;
-    const metadata = JSON.stringify({
-        articleId: item.articleId,
-        title: item.title,
-        summaryExcerpt: item.summaryExcerpt,
-    });
+    const metadata = buildHomeActionMetadata(type, item);
     ...
```

tag_click 経路（`handleTagClick`）は従来どおり `{tag}` を送る:
サーバが `meta.tag` を読むため必要であり、他記事の dismiss 成功により
WAF は tag 文字列に反応していないことが確認できていた。

### 検討した代替案

- **RPC 名を `TrackHomeAction` → `RecordHomeAction` 等に rename**
  — 初期仮説はパス名 `/Track...` が WAF に反応しているだった。しかし
  「特定 1 記事だけ」失敗という症状は path-level block と整合せず、
  proto の破壊的変更は contract / Pact / deprecate 手続き等の副作用が
  大きく割に合わない。ログ証跡（nginx 全 200）で rename 案は却下。
- **Cloudflare WAF rule を path-specific に bypass**
  — インフラ側で特定 URL を allow-list 化する案。FE の body 肥大は残り、
  将来ほかの記事で別 rule にヒットし得る。根治ではない。
- **proto で `metadata_json` を削除**
  — 互換性破壊。サーバは `meta.tag` / `meta.query` を使っており
  `tag_click` / 将来の search 系 action が影響を受ける。過剰破壊。
- **`handleAction` 内で inline に `undefined` を渡すだけ**
  — 最小変更だが policy が散逸し、将来誰かが title を添え直す
  regression を防げない。`home-actions.ts` に policy を固定する方が
  小規模な改変コストと regression guard の両立ができる。

### TDD

RED → GREEN → REFACTOR を厳守:

1. **RED**: `home-actions.test.ts` を書いて import error で失敗させた。
2. **GREEN**: `home-actions.ts` を作り、6 action type × 3 ケース計 8
   assertion が pass。
3. **REFACTOR**: `+page.svelte` の inline JSON を `buildHomeActionMetadata`
   呼び出しに置換。

## Consequences

### Pros

- 該当記事の Dismiss が前段 WAF を通過し 200 到達するようになる。
  他記事の Dismiss / Open / Ask / Listen にもリグレッションなし
  （metadata は元々サーバで未使用のため）。
- サーバ・proto・BFF・nginx をまったく触らない破壊的変更ゼロの修正。
  append-first / reproject-safe / versioned projection の各不変条件は
  完全に温存され、`knowledge_events` / `knowledge_home_items` のスキーマ
  も変わらない。
- POST body の表面積が縮み、**今後ほかの記事が別の WAF rule に触れる
  確率も下がる**。"本文片を body に載せない" という運用的な防御が
  `home-actions.ts` に固定されたため、将来誰かが title を添えようとした
  際にテストが落ちて止まる。
- 変更は FE の 2 ファイル（新設 + 7 行差分）で、ロールバックは revert +
  rebuild のみで済む。

### Cons / Tradeoffs

- **WAF がどの managed rule にヒットしているかは未特定**。Cloudflare
  dashboard でイベント確認ができれば、今後類似障害の予防につながるが
  本 ADR のスコープ外。CF 運用タスクとして残る。
- **root cause は我々のインフラの外側**。今回は body content 抑制で
  回避したが、本質的には「外部 WAF が我々の request を中身検査して
  ブロックしうる」構造そのものが残る。将来 CF 側のルール更新で別の
  POST body が反応する可能性は残り、監視としては **nginx 到達ログと
  クライアント DevTools の乖離**をスポットチェックする運用が必要。
- **`buildHomeActionMetadata` は常に `undefined` を返すだけ**なので、
  関数としては trivial に見える。ただし policy の単一の居場所である
  ことがこのモジュールの価値で、将来 action 種別ごとに body 要件が
  増えたときの拡張点にもなる。抽象化コストは最小に抑えた。
