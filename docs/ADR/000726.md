# ADR-000726: recap-subworker の Phase 0 ツーリング強化と Pyrefly 完全移行

## Status

Accepted

## Date

2026-04-14

## Affected Services

- recap-subworker (Python 3.14 / FastAPI)
- `docs/best_practices/python.md`（プロジェクト横断の規約）
- `.claude/skills/bp-python/SKILL.md`（プロジェクト横断のスキル）

## Context

`recap-subworker` は recap-worker (Rust) の Pact Provider として評価・クラスタリング・分類の各責務を担うが、以下の構造的負債が蓄積していた:

- `services/` 層が 3,000 LOC を超え肥大（`run_manager.py` 888 LOC / `pipeline.py` 771 LOC / `clusterer.py` 991 LOC）
- `usecase/` が外部 ML ライブラリ (sklearn, hdbscan, faiss, sentence_transformers, torch) を直接 import し Clean Architecture の DIP を破っている
- `dict[str, Any]` をレイヤ間で引き回している
- `_container`, `_ENGINE`, `_PIPELINE` などモジュール global が散在
- `except Exception:` が 15 箇所以上、`asyncio.Semaphore` で手組みしたバックプレッシャ
- [[000530]] で Pyrefly ≥ 0.42.0 を正式採用したにもかかわらず、`docs/best_practices/python.md` と `.claude/skills/bp-python/SKILL.md` が依然 `mypy --strict` を DECREE として記載していた
- Ruff は `E, W, F, I` の最小集合のみ有効で、安全性を担保する追加ルールが不足

以上の負債を、Pact 契約 (4 interactions) / `NUMBA_THREADING_LAYER=tbb` / Uvicorn 単一プロセス / `max_background_runs=2` ([[000636]]) / LLM 閾値 ([[000651]]) / Idempotent 失敗ラン非再利用 ([[000636]]) といった運用不変条件を一切壊さずに段階的に解消する 6 フェーズの大規模リファクタリング計画を策定した。本 ADR はその **Phase 0（ツーリング強化）** の完了を記録する。

## Decision

以下を採用する:

1. **Pyrefly を Python 型検査の唯一のソースとする**
   - `docs/best_practices/python.md` と `.claude/skills/bp-python/SKILL.md` の `mypy --strict` 言及を Pyrefly ≥ 0.42.0 に差し替え、DECREE・tooling セクション・レビューチェックリストを同期した。
   - [[000530]] の決定内容をプロジェクト横断の規約ドキュメントに正式反映した。

2. **Ruff ルール集合を段階的に拡張する**
   - recap-subworker `pyproject.toml` に `B, UP, SIM, C4, PTH, RUF` を追加。
   - `BLE / ASYNC / TRY / ANN / S` は Phase 5（async 堅牢化）で段階的に有効化する方針とし、現段階では混入させない。
   - 学習スクリプト (`learning_machine/scripts/**`) と分類器トレーナー (`infra/classifier/**`) 配下は conditional `sys.path` 操作・遅延 import・クイック実験パターンが支配的なため `per-file-ignores` で `E402, E722, E701, F841, PTH, B904, SIM` を抑止。
   - `infra/gunicorn_conf.py` は起動時の遅延 import が必須のため `E402` を明示的に許容。
   - 日本語コメント・docstring に対する `RUF001 / RUF002 / RUF003`（confusable 警告）はプロジェクト方針として無視対象。

3. **既存 Ruff 違反 98 件をすべて解消する**
   - `PTH` 系: `os.path` を `pathlib.Path` に置換。ただし `infra/path_validation.py` は CodeQL が `os.path.realpath() + startswith()` を sanitizer として認識するため、意図的な `noqa: PTH117/PTH118` コメントで例外を文書化。
   - `B904`: `app/routers/evaluation.py` の `HTTPException` 再送出を `raise ... from err` で原因チェーンに統一。
   - `SIM102 / SIM113 / SIM114`: `services/clusterer.py` と `services/evaluation.py` のネスト `if` と手組みカウンタを簡約化。
   - `F841`: 未使用ローカル (`macro_metrics`, `result`, `unique_labels`, `all_classes`, `max_tag_freq` 他) を削除。
   - `E731`: `infra/logging.py` の lambda を `_noop_shutdown()` 関数へ置換。
   - `E741`: `domain/selectors.py` の FAISS 慣用変数 `I`（indices）は `noqa: E741` + 理由コメントで保持。

4. **ツーリング不変条件を回帰テストで固定する**
   - `tests/unit/test_tooling.py` を新設し、`uv run ruff check recap_subworker/` と `uv run pyrefly check .` の exit code 0 を assert する。以降、これが破れた時点で CI で即座に失敗する。

## Consequences

### Pros

- プロジェクト横断のドキュメント (`docs/best_practices/python.md`) と規約スキル (`bp-python`) が [[000530]] の Pyrefly 採用と整合し、今後の Python サービス実装で mypy と Pyrefly の二重管理が発生しない。
- Phase 1 以降のリファクタリング（モジュール global 撤去、Usecase 分割、Port 配下への外部 lib 隠蔽、VO 化、async 堅牢化）を安定基盤上で進められる。
- `test_tooling.py` により「Ruff / Pyrefly がある時点で 0 エラー」という契約が退行不可能になった。新規ルール追加 → 新違反混入 → CI 失敗で即検知できる。
- `pathlib` 移行によりパス操作の意図が明確化され、CodeQL の sanitizer 検出が効く箇所（path_validation）は明示的な `noqa` コメントで保護した。
- `raise ... from err` の徹底により例外チェーンが保持され、ログ調査時の根本原因特定が容易になる。

### Cons / Tradeoffs

- Phase 5 の async 堅牢化までに `BLE / ASYNC / TRY` 系ルールが適用されないため、当該期間は blind-except の新規混入を防ぐのはレビュー/メモリ規約に依存する。`test_no_blind_except.py`（AST チェック）で補完する計画は Phase 5 に持ち込む。
- `learning_machine/scripts/**` は広範な per-file-ignore を持つため、当該スクリプトが将来プロダクションパスに昇格する場合は ignore 解除と修正が必要になる。
- 日本語 docstring 対応のため `RUF001-003` を無効化したが、英字混在の真の confusable を手動検知する機会を失う。将来、ターゲット箇所限定で `# noqa` 運用に切り替える余地を残している。
- Pyrefly が ML ライブラリを `replace-imports-with-any` で抑止する設定は現状維持とし、[[000530]] の方針に沿っている。Phase 0 の計画段階で検討した「`ignore-missing-source` への置換」は社内型を厳密化する反面、既存の 0 エラー状態を壊すため本 Phase では採用を見送った（Phase 5 以降で再評価）。
