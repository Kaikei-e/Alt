# HTMLエンティティデコードの二重アンエスケープ脆弱性修正（CWE-116）

## ADR's STATUS

Accepted

## CONTEXT

CodeQL の静的解析により、`alt-frontend-sv/src/lib/domain/feed/sanitize.ts:20` で **js/double-escaping**（CWE-116, CWE-20）が検出された。

`decodeHtmlEntities()` 関数において、`&amp;` → `&` の変換をチェーンの**最初**に実行していたため、二重デコード（double-unescaping）が発生していた。

**再現例**:

```
入力:       &amp;quot;
ステップ1:  &amp; → & により &quot; になる
ステップ2:  &quot; → " になる
結果:       意図しない " に変換される（本来は &quot; のまま残るべき）
```

この問題は以下のセキュリティ標準に抵触する:

| 標準 | 分類 |
|------|------|
| CWE-116 | 不適切なエンコード/エスケープによる出力の脆弱性 |
| CWE-20 | 不適切な入力検証 |
| OWASP Top Ten 2021 A03 | Injection |

**原則**: アンエスケープ処理において `&amp;` → `&` の変換は**必ず最後**に行う。`&` は他のHTMLエンティティ（`&lt;`, `&gt;`, `&quot;` 等）の構成要素であるため、先に変換すると後続のエンティティが再解釈されてしまう。

## DECISION MAKING

`decodeHtmlEntities()` 内の `.replace()` チェーンにおいて、`&amp;` のデコードを最初から最後に移動した。

**変更ファイル**: `alt-frontend-sv/src/lib/domain/feed/sanitize.ts`

```typescript
// Before (vulnerable)
return text
    .replace(/&amp;/g, "&")   // ← 最初に&をデコード = 危険
    .replace(/&lt;/g, "<")
    .replace(/&gt;/g, ">")
    .replace(/&quot;/g, '"')
    .replace(/&#0?39;/g, "'")
    .replace(/&apos;/g, "'")
    .replace(/&#x27;/g, "'");

// After (fixed)
return text
    .replace(/&lt;/g, "<")
    .replace(/&gt;/g, ">")
    .replace(/&quot;/g, '"')
    .replace(/&#0?39;/g, "'")
    .replace(/&apos;/g, "'")
    .replace(/&#x27;/g, "'")
    .replace(/&amp;/g, "&");  // ← 最後に&をデコード = 安全
```

この修正は CodeQL の公式ドキュメント [js/double-escaping](https://codeql.github.com/codeql-query-help/javascript/js-double-escaping/) に記載された推奨事項に従っている:

> エスケープ時は escape 文字を最初に処理し、アンエスケープ時は最後に処理すべき

変更は `.replace()` の呼び出し順序のみであり、ロジックの追加・削除はない。

## RESULTS, EFFECTS

### PROS

- **二重デコードの完全排除**: `&amp;quot;` のような二重エンコード済み文字列が意図せず `"` に変換されることがなくなった
- **CodeQL アラートの解消**: js/double-escaping 警告が解消され、CI の静的解析が通過する
- **既存テスト全パス**: `sanitize.test.ts` の10件のテストケースすべてが変更後もパス済み
- **影響範囲が極小**: replace チェーンの順序変更のみで、関数シグネチャや他ファイルへの影響なし
- **SSR/CSR 両方で有効**: DOMParser に依存しない実装のため、サーバーサイド・クライアントサイド双方で同じ修正が適用される

### CONS, TRADEOFF

- **正当な二重エンコードの扱いが変わる**: 意図的に `&amp;quot;` を `"` にデコードしたいケースがもし存在すれば、この修正により `&quot;` で止まるようになる。ただし、RSS フィードの sanitize 処理においてそのようなケースは想定されないため、実質的な問題はない
