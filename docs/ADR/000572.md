# ADR-000572: alt-backend を SOLID 原則・Clean Architecture に基づき全面リファクタリングする

## Status

Accepted

## Date

2026-03-25

## Affected Services

- alt-backend (Go)

## Context

alt-backend は 142K LOC / 786 ファイル規模に成長し、Clean Architecture の5層構造（Handler → Usecase → Port → Gateway → Driver）を採用しているものの、レイヤー間の直接的な依存違反はないが、SOLID 原則の違反が蓄積していた。

主な問題:

1. **SRP 違反**: `AltDBRepository` が8,978 LOC の god object（全ドメインのDBアクセスを1 struct に集約）。DI Container が 710 LOC の単一関数で 70+ usecase を配線。main.go に 10 個のバックグラウンドジョブ定義が散在。
2. **ISP 違反**: `FetchFeedsPort` が 8 メソッドの太いインターフェース。消費側 usecase は各1-2メソッドしか使わないのに全メソッドに依存。
3. **OCP 違反**: 新機能追加時に DI Container struct / handler / routes の3箇所を必ず変更。
4. **DIP 違反**: Handler が `*di.ApplicationComponents` 全体に依存。struct フィールドに具象型が露出。

これらは機能追加・テスト・保守のボトルネックとなっており、段階的な改善が必要だった。

## Decision

SOLID 原則と Go Best Practices (`docs/best_practices/go.md`) に準拠した段階的リファクタリングを実施する。全フェーズを独立デプロイ可能・後方互換で設計し、TDD (RED → GREEN → REFACTOR) を厳守する。

### Phase 1: AltDBRepository のドメイン別分割 (SRP) — 完了

god object `AltDBRepository` を 13 のドメイン別 Repository に分割した。

| Repository | 責務 |
|---|---|
| `FeedRepository` | フィード取得・登録・お気に入り・ステータス更新 |
| `ArticleRepository` | 記事取得・保存・カーソルベース一覧・OGP |
| `TagRepository` | タグ取得・更新・タグクラウド・トレンドタグ |
| `ScrapingRepository` | スクレイピングドメイン管理 |
| `ImageRepository` | 画像プロキシキャッシュ・OGP画像 |
| `RecapRepository` | Recap 記事取得 |
| `SubscriptionRepository` | 購読管理 |
| `InternalRepository` | 内部API用記事操作 |
| `SummaryRepository` | 要約バージョン管理 |
| `KnowledgeRepository` | Knowledge バックフィル |
| `OutboxRepository` | Outbox イベント管理 |
| `DashboardRepository` | ダッシュボード・トレンド統計 |
| `TenantRepository` | マルチテナント操作 |

`AltDBRepository` は Go の struct embedding を使って全ドメイン Repository を embed する後方互換ファサードとして維持した。これにより既存の gateway / DI container / テストコードは無変更で動作する。

クロスドメイン呼び出し（`SaveArticle` → `GetFeedIDByArticleURL` + `SaveOutboxEventWithTx`）はパッケージレベルヘルパー関数 `cross_domain_helpers.go` に抽出し、ドメイン Repository 間の直接依存を回避した。

### Phase 2: Port インターフェース分割 (ISP) — 完了

`FetchFeedsPort`（8メソッド）を 8 つの単一メソッドインターフェースに分割した。

```
FeedListPort, FeedCursorPort, UnreadFeedCursorPort,
ReadFeedCursorPort, FavoriteFeedCursorPort,
FeedsByLinkPort, FeedListLimitPort, FeedListPagePort
```

各 usecase は必要最小限のインターフェースのみに依存するよう変更:
- `FetchFeedsListCursorUsecase` → `FeedCursorPort`（1メソッド）
- `FetchUnreadFeedsListCursorUsecase` → `UnreadFeedCursorPort`（1メソッド）
- `CachedFeedListUsecase` → `ReadAndFavoriteFeedCursorPort`（2メソッド合成）

既存の `FetchFeedsPort` は合成インターフェースとして維持し、gateway 実装は無変更。Go の構造的型付けにより自動的に全サブインターフェースを満たす。

### Phase 3: DI Container のドメイン別モジュール分割 (SRP + OCP) — 完了

710 LOC のモノリシック `NewApplicationComponents()` を 8 つのドメインモジュールに分割した。

| モジュール | LOC | 責務 |
|---|---|---|
| `infra_module.go` | 96 | Config, RateLimiter, HTTP Client, Auth, MQ-Hub |
| `feed_module.go` | 205 | Feed ドメイン全体（17 usecase） |
| `article_module.go` | 133 | Article ドメイン全体（12 usecase） |
| `knowledge_module.go` | 174 | Knowledge Home + Recall/Lens（19 usecase） |
| `rag_module.go` | 65 | RAG/Augur/Morning |
| `image_module.go` | 58 | Image Fetch/Proxy |
| `recap_module.go` | 51 | Recap/Dashboard |
| `subscription_module.go` | 64 | Subscription/OPML/CSRF |

`container.go` は 710 → 378 LOC（-47%）に削減。各モジュールは unexported コンストラクタ `newXxxModule()` を持ち、`NewApplicationComponents()` がそれらを組み立てる。既存の `ApplicationComponents` struct フィールドはモジュールフィールドから populate し、Handler / テストコードは無変更。

### Phase 7: main.go クリーンアップと Job Registry — 完了

10 個のバックグラウンドジョブ定義を `job/registry.go` の `RegisterAllJobs()` 関数に抽出した。main.go は 402 → 321 LOC（-20%）に削減。

## 残作業（未完了フェーズ）

| Phase | 内容 | 依存 |
|---|---|---|
| Phase 4 | Connect-RPC Handler のサブドメイン別分割 (SRP) | なし |
| Phase 5 | Handler の DIP 改善（Container 依存 → 個別 Usecase 注入） | Phase 4 |
| Phase 6 | エラーハンドリング統一（Sentinel Error 集約、マッピング一元化） | なし |
| Phase 8 | Setter Injection 除去 → コンストラクタ注入統一 | Phase 3 |
| Phase 9 | テストパターン統一（テーブル駆動、Mock/Stub 方針） | なし |

## Consequences

### Pros

- **SRP 遵守**: 各 Repository / モジュールが単一責務。新機能追加時に影響範囲が局所化
- **ISP 遵守**: Usecase が必要最小限のインターフェースにのみ依存。モック生成・テストが簡潔
- **テスタビリティ向上**: ドメイン Repository を個別にモック可能。fat interface のモック不要
- **OCP 改善**: 新ドメイン追加時は新モジュールファイル作成 + container.go への1行追加のみ
- **完全な後方互換**: 既存の Handler / テスト / Gateway コードは一切無変更
- **全 122 packages テスト GREEN**: リグレッションなし

### Cons / Tradeoffs

- **ファイル数増加**: ドメイン Repository 13 ファイル + DI モジュール 8 ファイル + Job Registry 1 ファイル = 22 新規ファイル
- **AltDBRepository ファサードの間接層**: 移行期間中、ファサード経由のメソッド呼び出しが残る。Gateway を新ドメイン Repository に直接依存させる移行が必要
- **Handler のコンテナ依存**: Phase 5 未完了のため、Handler は依然として `*di.ApplicationComponents` 全体に依存。これは次のリファクタリングフェーズで解消予定
- **Setter Injection 残存**: `RegisterFeedsUsecase` 等の Setter Injection パターンが Phase 8 まで残る
