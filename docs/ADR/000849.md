# ADR-000849: alt-deploy の pact-publish を consumer → provider 二段に分割し Pact Nirvana の publish/verify race を構造的に解消する

## Status

Accepted

## Date

2026-04-26

## Affected Services

- **alt-deploy (private repo)** — `release-deploy.yaml` の `pact-publish` matrix を 2 ジョブに分割し、各レッグの本体を新設 `playbooks/pact-publish.yml` に集約。`pact-manual` の依存先を新 `pact-publish-consumer` に変更、`gate` は role 別 (`pact-status-consumer-<svc>` / `pact-status-provider-<svc>`) artifact を fan-in。`resolve` ジョブに `pacticipants_consumers` / `pacticipants_providers` の filtered subset 出力を追加
- **knowledge-sovereign / alt-backend ほか pacticipant 全般** — production code 無変更。Pact 契約面の挙動 (release 都度、最新 consumer pact が provider に verify される順序が決定的になる) が変わるため、結果として release blocker が減る

## Context

2026-04-26 の release-deploy run で `gate (alt-backend)` が `can-i-deploy: No deployable version found` で停止。続く同 SHA の再 dispatch では別の verdict (`(failure)`: real contract mismatch) に化けるという、**ノンデターミニスティックな gate 挙動**が露呈した。

### 原因系 1 (本 ADR の対象): publish/verify race

旧 `pact-publish` ジョブは pacticipant ごとの matrix で `consumer,provider` 両 role を 1 レッグの中で順次実行していた。これは pacticipant 間では完全並列に走る。`alt-backend (consumer,provider)` レッグと `knowledge-sovereign (provider)` レッグが同時刻に start すると、sovereign の provider verifier は T+0 で broker から alt-backend の最新 main-branch pact (= **前リリースの pact-version**) を fetch して verify する。alt-backend の **新** consumer pact は隣の matrix leg で数分後にようやく broker に landing するので、sovereign@b4359421 の verification record が **新 pact-version をカバーしない**状態が発生する。

`can-i-deploy --pacticipant alt-backend --version <sha> --to-environment production` は production-tag の sovereign 版を採用する → 新 pact-version に対する verification record 無し → `unknown` → blocked。直前に deploy 側に投入した `PACTICIPANT_COPAIRS_WHEN_PRESENT` で sovereign を co-version pin する fix を入れたが、これは「production version ではなく co-released version を見ろ」という指示にすぎず、**verification record そのものが新 pact-version 用に publish されていない**問題は残った。

Pact 公式の [Pact Nirvana Step 6](https://docs.pact.io/pact_nirvana/step_6) は本ケースを名指しで anti-pattern として記述している:

> if your consumer just generated and published a new version of the pact, this will always fail because the provider could not possibly have run verification against a newly changed pact.

ドキュメントが推奨するのは 2 つ。

- (A) consumer publish → provider verify の **in-pipeline sequencing** + `can-i-deploy --retry-while-unknown` (例: `pactflow/example-provider`)
- (B) `contract_requiring_verification_published` webhook + 別 pipeline での async 検証 (例: `pactflow/example-consumer-webhookless`)

Alt の release-deploy は **同一 dispatch で consumer と provider を一緒に出す lockstep release** が常態 (Knowledge Loop 系の commit は常に alt-backend と sovereign を一緒に動かす) なので、(A) が圧倒的に整合する。webhook 経由の (B) は別 pipeline 起動 → 完了まで gate が `--retry-while-unknown` で待つ必要があり、SLA 不明な webhook 遅延に release を縛ることになる。

### 原因系 2 (本 ADR の対象外、別 ADR で扱う): 構造を直しても残る real contract mismatch

後続再 dispatch では sovereign provider 検証の verdict が `(failure)` に変化した。すなわち sovereign provider stub が新 alt-backend consumer pact の interaction を 1 つ以上満たせていない。これは race とは独立した **本物の契約違反** (DEFER trigger / OODA stage depth の wire shape 変化が provider stub の `mux.HandleFunc` ルーティングや state handler に届いていない可能性が高い)。本 ADR の構造修正で **race だけは取り除ける** が、この real mismatch は別途 contract 修正が必要。本 ADR は scope を CI 構造に絞り、real mismatch は同時にコミットしない。

### Ansible 化の理由

旧 `pact-publish` ジョブは inline shell で `./scripts/pact-check.sh ...` を呼んでいた。本 ADR で 2 ジョブに分けると body の ~90% が重複する。alt-deploy は既に [`playbooks/run-e2e-suite.yml`](../../alt-deploy/playbooks/run-e2e-suite.yml) / [`playbooks/apply-migrations.yml`](../../alt-deploy/playbooks/apply-migrations.yml) / [`playbooks/roll-services.yml`](../../alt-deploy/playbooks/roll-services.yml) で `GHA = matrix orchestration / Ansible = idempotent task body` の規約を確立しており、新ジョブもこれに揃える方が duplication が消え、operator が `ansible-playbook playbooks/pact-publish.yml -e SERVICE=… -e ROLE=…` で local 再現できる利点がある。

## Decision

### 採用する構造

**alt-deploy `release-deploy.yaml` の `pact-publish` matrix を `pact-publish-consumer` → `pact-publish-provider` の二段に分割し、両者の本体を新設 `playbooks/pact-publish.yml` に集約する**。

#### 1. `resolve` ジョブの outputs 拡張

```js
// 既存 pacticipants から role でフィルタ
const hasRole = (svc, role) => (PACTICIPANT_ROLES[svc] || []).includes(role);
const pacticipants_consumers = pacticipants.filter(p => hasRole(p.service, 'consumer'));
const pacticipants_providers = pacticipants.filter(p => hasRole(p.service, 'provider'));
core.setOutput('pacticipants_consumers', JSON.stringify(pacticipants_consumers));
core.setOutput('pacticipants_providers', JSON.stringify(pacticipants_providers));
```

`pacticipants` (全件) は `gate` がそのまま使うので残す。

#### 2. `pact-publish-consumer` (Stage 1, matrix per consumer)

- `needs: [resolve, build]`
- `if: needs.resolve.outputs.pacticipants_consumers != '[]'`
- `matrix.include`: `pacticipants_consumers`
- 各レッグは Ansible playbook を呼ぶ thin wrapper (pre-checkout cleanup → checkout Alt → checkout alt-deploy → ansible-playbook --check → ansible-playbook → upload status)
- 成功時に `pact-status-consumer-<svc>` artifact を upload

#### 3. `pact-publish-provider` (Stage 2, matrix per provider)

- `needs: [resolve, build, pact-publish-consumer]` ← **stage 1 完了を待つ**
- `if: always() && needs.resolve.outputs.pacticipants_providers != '[]'` (always() で sibling consumer 失敗が provider 全体を止めない; 失敗した service は gate precondition で drop)
- `matrix.include`: `pacticipants_providers`
- 同 wrapper で `ROLE=provider` を渡す
- 成功時に `pact-status-provider-<svc>` artifact を upload

#### 4. `playbooks/pact-publish.yml` (新設、~140 行)

両 stage が共有する宣言的タスク列:
- 入力 assert (service in pacticipants allowlist / role ∈ {consumer, provider} / SHA は 40 hex / path は `[^;|&\`$]+` / broker URL は `https?://`)
- broker password file の `stat` → 欠如時 hard-fail (silent skip 防止)
- `slurp` + b64 decode で password 読み出し (`no_log: true`)
- Alt checkout 内の `scripts/pact-check.sh` を `ansible.builtin.command` (shell ではなく) で実行。SERVICE / ROLE は引数として discrete に渡す
- 成功時に `STATUS_DIR/<role>-<service>.txt` を作成 → wrapper が artifact 化

#### 5. `pact-manual` の依存先変更

`needs: [resolve, pact-publish]` → `needs: [resolve, pact-publish-consumer]`。manual bridging は consumer pact を broker に置いた後で動けばよく、provider 検証完了は要らない。両者を並列に走らせて wall-clock を縮める。

#### 6. `gate` の precondition を role 別 fan-in に変更

```yaml
- name: "Download consumer pact-status (if applicable)"
  if: contains(matrix.roles, 'consumer')
  uses: actions/download-artifact@…
  with:
    name: "pact-status-consumer-${{ matrix.service }}"
- name: "Download provider pact-status (if applicable)"
  if: contains(matrix.roles, 'provider')
  uses: actions/download-artifact@…
  with:
    name: "pact-status-provider-${{ matrix.service }}"
- name: "Gate precondition — required pact-status artifacts present"
  run: |
    ok=true
    IFS=',' read -r -a role_arr <<<"$ROLES"
    for role in "${role_arr[@]}"; do
      [[ -f "_pact_status/${role}-${SERVICE}.txt" ]] || ok=false
    done
    echo "ok=${ok}" >> "$GITHUB_OUTPUT"
```

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| (B) Webhook 経由の async 検証 + `--retry-while-unknown` 大幅増 | sovereign を webhook allowlist に追加する作業は deploy 側で別途実施済みだが、release が webhook → checkout → verify → publish (5–10 分) の SLA 不明な経路に依存し、`--retry-while-unknown` を 30+ に増やしても確実性が無い。lockstep release が常態の本プロジェクトには in-pipeline sequencing が優位。webhook 経路は consumer-only 変更時の fallback として残す |
| `pact-republish-providers` 追加ジョブで provider role を 2 回走らせる | mixed pacticipant (alt-backend) で provider verification が 2 回走る。1 回目は stale pact を verify、2 回目は新 pact を verify。verification record が二重に publish されて broker UI が見にくくなる。stage 分割の方が clean |
| 旧 `pact-publish` 単一ジョブのまま、レッグ内で `--role consumer` を全 pacticipant 分先に sleep-loop で待たせる | inter-leg synchronization は GHA matrix の責務外。bash で実装すると flaky。`needs:` で表現できる構造を bash に移すのは反パターン |
| Pending pacts (`EnablePending=true`) で gate を softener する | docs/pact_broker/advanced_topics/pending_pacts は明示的に「pending は verifier が未知の pact で fail しない保護であって、`can-i-deploy` を softener する機構ではない」と書いている。現状 Alt の provider 全部に既に `EnablePending=true` が入っているが今回の `unknown` は救えない (verification record が **存在しない**ことが問題で、failure かどうかは関係ない) |
| Inline shell のまま 2 ジョブに分割 (Ansible 化しない) | body の ~90% が重複。`run-e2e-suite.yml` などの規約から外れる。operator local 再現性も劣る |

## Consequences

### Pros

- **Pact Nirvana Step 6 の anti-pattern を解消**。consumer publish → provider verify の順序が GHA `needs:` で deterministic に保証され、release 都度新 pact-version に対する verification record が確実に publish される。同 SHA 再 dispatch でも結果が安定 (今回観測された二重 dispatch 間の非決定性は構造的に消える)
- **alt-deploy の規約 (`GHA = orchestration / Ansible = task body`) に揃った**。新規発明ゼロ、`run-e2e-suite.yml` と同じ抽象レベル
- **operator local 再現性**。`SERVICE=alt-backend ROLE=consumer SHA=… ALT_CHECKOUT=… STATUS_DIR=… PACT_BROKER_BASE_URL=… ansible-playbook -i localhost, -c local playbooks/pact-publish.yml` で 1 レッグだけ local 再現できる。CI 内でしか動かなかった旧構造より debug が容易
- **secret leak 縮小**。旧 inline は `>> "$GITHUB_ENV"` で `PACT_BROKER_PASSWORD` を job 全体の env に置いていたため、後続の任意 step / action から見えた。新構造は playbook task の `environment:` ローカルに閉じる (`no_log: true` で slurp 結果も log されない)
- **`pact-manual` と `pact-publish-provider` が並列化**できるようになった (両者とも consumer stage にだけ依存)。wall-clock 短縮

### Cons / Tradeoffs

- **wall-clock が増える**。旧構造は consumer/provider が同レッグで sequential、複数 pacticipant は並列だったので最遅 pacticipant の wall-clock = ~9 分。新構造は consumer stage 全完了 → provider stage 全完了の二段なので、最遅 consumer + 最遅 provider = ~13 分。release 全体で +4 分程度。lockstep の確実性とトレードする
- **Ansible 依存が 1 ジョブ増える**。`alt-builder` runner は既に Ansible 必須 (run-e2e-suite が呼ぶ) ので追加 provisioning は無いが、Ansible 障害が pact-publish にも波及するようになる
- **artifact 数が倍増**。`pact-status-<svc>` (1 件/pacticipant) → `pact-status-{consumer,provider}-<svc>` (mixed pacticipant は 2 件)。retention は 7 日のままなので storage 影響は軽微だが、下流ツールが旧 artifact 名に依存していたら破綻する。本 ADR 時点では `gate` ジョブのみが consumer
- **本 ADR は別系統の real contract mismatch を直さない**。本 ADR の構造修正後も sovereign provider stub の更新が別途必要。順序的には: (1) この ADR をコミット → (2) deploy 側 push → (3) provider stub 修正コミット (Alt 側) → (4) release 再 dispatch、で release が通る想定
- **operator runbook の更新が必要**。`docs/runbooks/pact-broker-ops.md` Section 6 (orphan pact 検出) と alt-deploy `docs/runbooks/operations.md` Section 6.5 (provider verify 不在) は旧 `pact-publish` 単一ジョブを前提にした記述になっている。本 ADR コミット後に追随更新する (本 ADR 範囲外、フォロー PR)
