# ADR-000811: recap-subworker の classification_backend デフォルトを joblib に戻し learning_machine artifacts 欠落を起動時 fail-closed で検知する

## Status

Accepted

## Date

2026-04-20

## Affected Services

- **recap-subworker** — `recap_subworker/infra/config.py` の `classification_backend` フィールドのデフォルトを `"learning_machine"` → `"joblib"` に戻す。併せて同クラスに `@model_validator(mode="after")` の `_validate_learning_machine_artifacts` を追加し、`classification_backend == "learning_machine"` かつ `learning_machine_student_ja_dir` と `learning_machine_student_en_dir` がどちらも `Path.is_dir()` を満たさない場合は起動時に `ValidationError` を上げて fail-closed する

## Context

2026-04-20 19:00:51 JST、3days Recap が `classification returned 0 results for 948 articles (service may be unavailable)` で失敗した。メッセージは [[PM-2026-033]] の mTLS URL スキーム不一致と同文だが、**これは再発ではない**。[[ADR-000774]] の対策 (`pki-agent-recap-subworker` / `pki-agent-news-creator` reverse-proxy サイドカー、`compose/recap.yaml:97-98` の `https://...:9443`、`recap-worker/src/config.rs:228,723` の `validate_mtls_url_schemes`) は全てデプロイ済みで、稼働コンテナでも確認済み。

切り分け:

- `recap-subworker` コンテナは起動して `/health` にも応答するが、`POST /v1/classify-runs` を受けた時点で `LearningMachineStudentClassifier.__init__` が `recap_subworker/services/learning_machine_classifier.py:131-132` で `RuntimeError: At least one model (JA or EN) must be loaded` を投げる。
- `settings.classification_backend == "learning_machine"` (2025-12-13 の commit `8235e8f0a` 以降デフォルトが `"learning_machine"`)。
- 期待パス: `/app/recap_subworker/learning_machine/artifacts/student/v0_ja` と `/student/v0_en`。
- 実態: コンテナ内 `/app/recap_subworker/learning_machine/artifacts/` は `total 0`。`compose/recap.yaml:224` が `../recap-subworker/recap_subworker/learning_machine/artifacts:/app/recap_subworker/learning_machine/artifacts:ro` でバインドマウントしており、デプロイランナーの clone 側ディレクトリが空。
- 理由: `.gitignore:44` が `recap-subworker/recap_subworker/learning_machine/artifacts/` を除外。学習済みモデルの artifacts (student/ 配下で 2.1 GB 相当) は公開 OSS リポジトリに含めない方針のため、`git clone` ベースのデプロイチェックアウトにはそのまま届かない。

並行して `recap-subworker/data/*.joblib` も `.gitignore:45` で除外されているが、こちらは別ルートの配布機構により deploy ランナーの `recap-subworker/data/` 配下に実在しており、`compose/recap.yaml` で個別ファイル bind-mount されたうえでコンテナ内 `/app/data/*.joblib` として読める状態にある (`docker inspect` で確認済み)。つまり `joblib` バックエンドは既に運用データが届いているが、`learning_machine` バックエンドは届いていない。

recap-worker 側のログ: `classify-runs POST request failed` / `error sending request for url (https://recap-subworker:9443/v1/classify-runs)` / `Failed to initialize classification worker pool within 300s` × 5 チャンク × 3 リトライ → `result_count=0, elapsed_seconds=1516` → `classification returned 0 results for 948 articles (service may be unavailable)` で abort。`pki-agent-recap-subworker` は upstream の uvicorn が応答不能のため `http: proxy error: context canceled` を返している (TLS 層自体は健全)。

`service may be unavailable` のエラー文言が TLS / ネットワーク層の障害のように誤誘導する。実際は artifacts 欠落による classify-runs ハンドラの lazy init 失敗で、以下の 2 点が重なっていた:

1. **デフォルトが配布不整合のまま**: `classification_backend` デフォルトが `learning_machine` なのに、学習済み artifacts の配布経路が整備されていなかった。
2. **起動時検知の不在**: 欠落が起動時ではなく、最初の classify-runs を受けた瞬間に確定的に失敗する lazy init 設計だったため、`/health` は通り「見かけ上は稼働中」の状態が最大数時間続いた。

イミュータブルデータモデル不変条件 (`knowledge_events` / projector / `summary_versions` / `tag_set_versions`) への影響はない。本変更はデフォルト設定と起動時バリデーションだけで、イベント payload / projection には一切触れない。

## Decision

recap-subworker の設定と起動プロセスに対し、以下の 2 点を同時適用する:

1. **`classification_backend` デフォルトの差し戻し**: `recap_subworker/infra/config.py` の `classification_backend: Literal["joblib", "learning_machine"] = Field("learning_machine", ...)` を `Field("joblib", ...)` に戻す。`joblib` バックエンドは `/app/data/genre_classifier*.joblib` / `tfidf_vectorizer*.joblib` / `genre_thresholds*.json` を使用し、これらは既に運用環境に配布済みで compose の bind-mount で利用可能。

2. **学習機械 artifacts の起動時 fail-closed バリデーション**: `Settings` クラスに `@model_validator(mode="after")` の `_validate_learning_machine_artifacts` を追加する。`self.classification_backend != "learning_machine"` なら早期 return、そうでなければ `learning_machine_student_ja_dir` と `learning_machine_student_en_dir` の双方について `Path(dir).is_dir()` を確認し、どちらか一方でも成立すれば通過、両方とも成立しない場合は `ValueError("learning_machine artifacts missing: tried ja=<path>, en=<path>")` で fail-closed。

JA/EN の OR 条件にするのは、既存 `LearningMachineStudentClassifier.__init__` が「最低 1 つ (JA または EN) のモデルがロードできれば続行」という挙動を採用しているため、設定バリデーションも同方針で揃えるのが自然であるため。[[ADR-000774]] の `validate_mtls_url_schemes` と同じ「起動時に確定的に失敗させて、ジョブ実行時の曖昧な失敗を防ぐ」方針の再適用。

既存 `LearningMachineStudentClassifier.__init__` の `RuntimeError` は残す (多層防御)。`Settings` バリデーションが通過してもモデルファイルが壊れていた場合のフォールバックとして機能する。

### 検討した代替案

| 案 | 採用 | 却下理由 |
|---|---|---|
| **A: デフォルト `joblib` + 起動時 fail-closed** (採用) | ○ | joblib の artifacts は既に運用環境に配布済みで即日稼働。learning_machine を明示的に指定したい場合は environment variable で opt-in でき、artifacts 欠落は起動時に確定検知 |
| B: デフォルトを `learning_machine` のまま据え置いて、learning_machine artifacts の配布経路を先に整備 | × | 配布経路整備 (Git LFS / S3 / named volume + init container 等) は別 ADR 規模の設計変更。現在流れている 3days Recap の復旧が最優先で、配布経路整備は後追い可能 |
| C: 起動時 fail-closed だけ入れてデフォルトは据え置き | × | 本日の障害の直接原因である「デフォルトが配布不整合」を解消しない。fail-closed しても起動時に落ちて「起動しないコンテナ」になるだけで、Recap パイプライン復旧にならない |
| D: `LearningMachineStudentClassifier` のフォールバック経路を追加して、モデル欠落時に `joblib` バックエンドへ自動切替 | × | フォールバックは「設定と実態の不一致を暗黙に埋める」挙動で、本件のような silent failure シリーズ ([[PM-2026-031]] / [[PM-2026-032]] / [[PM-2026-033]]) の教訓に逆行する。設定と実態は明示的に一致させる方が将来の運用コストが低い |

## Consequences

### Pros

- **3days Recap の即日復旧**: デプロイ後、`classification_backend` がデフォルト `joblib` で起動し、既存の `/app/data/*.joblib` を読む。`classify-runs` は再び有効な結果を返す。
- **将来の silent failure 予防**: `RECAP_CLASSIFICATION_BACKEND=learning_machine` を明示的に指定したときに artifacts が配布されていなければ、起動時に `ValidationError` で確定的に落ちる。平均検知時間 (TTD) が「最初の classify-runs 呼び出しで初めて失敗」→「コンテナ起動の数秒」に短縮される ([[ADR-000774]] の `validate_mtls_url_schemes` と同パターン)。
- **`learning_machine` backend は dev/eval 用途で引き続き利用可能**: 学習済み artifacts が存在するローカル開発環境で `RECAP_CLASSIFICATION_BACKEND=learning_machine` を渡せば、本バリデータを通過して従来通り動く。学術検証・eval パイプラインに影響しない。
- **OSS / Private 公開境界の明確化**: 本リポジトリ (Alt, OSS) には学習済み artifacts を含めない方針を設定レベルで追認。将来 artifacts を本番配布したい場合は Private 側 (alt-deploy) の配布経路を別 ADR で整備する、という境界が残る。
- **多層防御**: 既存 `LearningMachineStudentClassifier.__init__` の `RuntimeError` はそのまま残るので、Settings バリデーションをすり抜けてロード段階で失敗するケース (ファイル破損等) にも対応可能。

### Cons / Tradeoffs

- **本番運用は当面 `joblib` 固定**: `learning_machine` student model の精度向上が本番に届かない。これは現状と実質変わらないが、意図的な choice として固定化されたことで、改めて「本番化したい」と判断するハードルは上がる。artifacts 配布経路整備 (Git LFS / S3 / named volume + init container 等) を別 ADR で立てる必要がある。
- **起動時検知のコスト**: `Path.is_dir()` をコンテナ起動のたびに呼ぶので、シンボリックリンクの解決などで極小のコストがかかる。プロセス再起動時のみで runtime コストはゼロに近いため無視できる。
- **fail-closed バリデータの意味論**: 今は「JA または EN の片方があれば OK」だが、将来両方必須にしたい (EN only 運用を禁じたい等) 場合は追加判断が必要。既存 classifier 実装と同じ条件にしているので、少なくとも現在は一貫性がある。
- **空ディレクトリ問題**: `Path.is_dir()` は中身が空でも `True` を返す。完全に整合させるには `(dir / "config.json").is_file()` 等を追加チェックする案もあり得るが、既存 classifier 実装が `Path.exists()` だけ見ているので、設定バリデーションを過度に厳しくすると classifier 実装との乖離が生じる。本 ADR では同じ厳しさ (`is_dir()`) に留め、ファイル単位の整合性チェックは classifier 側のロード失敗 (`RuntimeError`) に委ねる多層防御で受ける。
- **環境変数名の 2 系統**: `RECAP_CLASSIFICATION_BACKEND` と `RECAP_SUBWORKER_CLASSIFICATION_BACKEND` の `AliasChoices` が残存しており、どちらでも指定可能。本 ADR では触らず、既存の後方互換を維持する。
