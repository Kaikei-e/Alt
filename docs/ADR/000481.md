# ADR-000481: ListUntaggedArticles を keyset pagination へ移行し feed_id NULL 除外と TSV ファントム発行を修正する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-backend (Go) — driver / port / gateway / handler / proto
- tag-generator (Python) — ConnectArticleFetcher

## Context

Knowledge Home カードにタグが付与されない問題が発生していた。tag-generator のコンテナログ分析から、以下の根本原因を特定した。

### RC1: 無限フェッチループ（致命的）

`ConnectArticleFetcher.fetch_articles()` が `last_created_at` / `last_id` カーソル引数を完全に無視し、常に `offset=0` で RPC を発行していた。`batch_processor.py` の while ループが同じ記事を繰り返しフェッチし、GPU リソースを浪費していた。

### RC2: feed_id NULL の記事がキューを永久ブロック

`ListUntaggedArticles` は `feed_id IS NULL` の記事も返すが、`BatchUpsertArticleTags` の driver 層は `feed_id == ""` の記事をスキップする（`feed_tags` テーブルが `feed_id NOT NULL` を要求するため）。これにより `feed_id IS NULL` の記事は永久に「未タグ」のまま残り、offset pagination 下でカーソルが前進しない原因となっていた。

### RC3: カーソル停滞（RC1 + RC2 の派生症状）

RC1 と RC2 の結果、tag-generator 側のカーソルが固定値で停滞していた。さらに backend handler が response の `created_at` を埋めていなかったため、tag-generator 側は caller の `last_created_at` をそのまま流用しており、進捗指標が実態を反映していなかった。

### RC4: 空 feed_id のファントム TagSetVersion イベント

`BatchUpsertArticleTags` handler は全アイテムに対して `createTagSetVersionUsecase.Execute()` を呼んでいたが、実際の DB upsert は `feed_id == ""` をスキップする。タグが保存されていない記事に対して `TagSetVersionCreated` イベントが発行され、Knowledge Home projector に不正なデータが流れていた。

## Decision

### 1. ListUntaggedArticles を offset pagination から keyset pagination へ移行する

既存の `ListUnsummarizedArticles` と同じ `(created_at DESC, id DESC)` の keyset pagination パターンに統一する。proto 契約・Go driver・port・gateway・handler・Python fetcher を一貫して変更する。

offset pagination を残さない理由: 未タグ記事はタグ付与により結果セットから消えるため、offset がページ間でずれる構造的問題がある。keyset pagination であればこの問題は発生しない。

### 2. feed_id IS NULL の記事を ListUntaggedArticles の結果から除外する

`feed_tags` テーブルが `feed_id NOT NULL` を要求する以上、`feed_id IS NULL` の記事は構造的にタグ付与不可能である。SQL の WHERE 句に `AND a.feed_id IS NOT NULL` を追加し、COUNT クエリと SELECT クエリの両方から除外する。

将来 backfill 等で `feed_id` が埋まれば、その記事は自動的に未タグ対象として再出現する。

### 3. BatchUpsertArticleTags handler で空 feed_id の TagSetVersion 作成をスキップする

`item.FeedId == ""` の場合は warn ログを出力して `continue` する。これにより、タグが実際に保存されていない記事に対するファントムイベント発行を防止する。

## Implementation

### Proto 変更

`ListUntaggedArticlesRequest` に `optional google.protobuf.Timestamp last_created_at` (field 3) と `string last_id` (field 4) を追加。`offset` (field 2) は後方互換のため残すが DEPRECATED とする。

`ListUntaggedArticlesResponse` に `optional google.protobuf.Timestamp next_created_at` (field 3) と `string next_id` (field 4) を追加。

Go / Python の生成コードを `buf generate` で更新。

### Backend (Go)

**Driver** (`list_untagged_articles_driver.go`):
- `InternalUntaggedArticle` に `CreatedAt time.Time` を追加
- シグネチャを `(ctx, lastCreatedAt *time.Time, lastID string, limit int)` に変更
- 初回ページ: `ORDER BY a.created_at DESC, a.id DESC LIMIT $1`
- 次ページ: `AND (a.created_at, a.id) < ($1, $2) ORDER BY ... LIMIT $3`
- 両クエリに `AND a.feed_id IS NOT NULL` を追加
- `totalCount` の COUNT クエリにも同条件を追加

**Port** (`internal_tag_port/port.go`):
- `UntaggedArticle` に `CreatedAt time.Time` を追加
- `ListUntaggedArticlesPort` を cursor ベースのシグネチャに変更

**Gateway** (`internal_article_gateway/gateway.go`):
- cursor パラメータのパススルーと `CreatedAt` の詰め替え

**Handler** (`handler.go`):
- `ListUntaggedArticles`: request の `last_created_at` / `last_id` を読み、response に `next_created_at` / `next_id` / `created_at` を埋める
- `BatchUpsertArticleTags`: `item.FeedId == ""` の場合に warn ログ + `continue` で TSV 作成をスキップ

### Tag-generator (Python)

**ConnectArticleFetcher** (`connect_article_fetcher.py`):
- `fetch_articles()` で `last_created_at` / `last_id` を `ListUntaggedArticlesRequest` に反映
- sentinel 値 `"9999-12-31T23:59:59Z"` を初回ページ判定に使用（cursor 未設定で送信）
- `fetch_new_articles()` は常に sentinel を渡して first page（最新記事）を取得する。backward keyset pagination では cursor より新しい記事が返らないため、forward パスは cursor を送ってはならない
- `count_untagged_articles()` から deprecated な `offset` 引数を除去

### テスト

- Driver: 5 テスト（NULL feed_id 除外、初回ページ、次ページ、全記事 feed_id あり、空結果）
- Handler: 5 テスト（ListUntaggedArticles success/cursor/unimplemented、BatchUpsert success/unimplemented）
- Python fetcher: 12 テスト（cursor パススルー、初回ページ omit、forward パス first-page 強制、dict 変換、デフォルトバッチサイズ、空 cursor など）

## Consequences

### Pros

- tag-generator が同じ記事を繰り返しフェッチする無限ループが解消される
- `feed_id IS NULL` の記事がキューをブロックしなくなる
- Knowledge Home に正しくタグが反映される
- ファントム `TagSetVersionCreated` イベントが止まり、projector に不正データが流れなくなる
- `ListUnsummarizedArticles` と同じ pagination パターンに統一され、保守性が向上する

### Cons / Tradeoffs

- proto の `offset` フィールドは deprecated として残すが、新規利用は禁止。将来的に `reserved` へ移行する判断が必要
- `totalCount` の COUNT クエリは全テーブルスキャンであり、未タグ記事が大量に存在する場合にパフォーマンス懸念がある。tag-generator 側の early-exit 最適化で使用しているため当面は維持するが、必要に応じて除去を検討する
- `feed_id IS NULL` の記事は暗黙的に除外されるため、backfill が完了するまでこれらの記事にタグが付かないことを運用上認識しておく必要がある
- `fetch_new_articles()` は常に first page を取得するため、タグ抽出に毎回失敗する記事（content が極端に短い等）が forward パスで繰り返し処理される。これらは article_tags が付かない限り消えないが、実害は軽微（CPU コストのみ、ファントムイベントは発生しない）
