# ADR-000656: Acolyte レポート生成品質を5契約の整備で改善する

## Status

Accepted

## Date

2026-04-09

## Affected Services

- acolyte-orchestrator (Python 3.14+)
- search-indexer (Go) — CDC 契約の整合性修正

## Context

[[000653]] で導入した Acolyte の6ノード LangGraph パイプライン（planner → gatherer → curator → writer → critic → finalizer）は動作するが、生成レポートの品質に深刻な問題があった。6件のレポート生成で判明した主要課題:

1. **入力契約の破綻**: `connect_service.py:171` で `scope = {}` がハードコードされ、planner/gatherer/writer 全体が壊れていた。Gatherer は常に "technology trends" で検索し、Writer は「トピックが明示されていません」とメタ発言を出力
2. **retrieval 契約の不整合**: CDC テスト (`POST /indexes/articles/search`) と実装 (`GET /v1/search`) が不一致。Gateway が search-indexer に存在しない `url`, `published_at`, `_rankingScore` フィールドを期待
3. **claim 契約の不在**: Writer は150文字の excerpt のみで自由文生成し、「一般知識で書け」と指示。根拠のない主張が混入する構造
4. **critic 契約の形骸化**: 500文字に切り詰めた上で「reasonably informative なら accept」、JSON parse 失敗で `{"verdict": "accept"}` にフォールバック。事実上ゴム印
5. **評価契約の不在**: レポート品質を定量的に測る基盤がなく、改善の検証ができなかった

[[000654]] の実装ステータスレポートでも P0 blocker として `scope = {}` が記録されていた。

## Decision

**5つの契約（入力・retrieval・claim・critic・評価）を体系的に整備し、パイプラインを8ノードに拡張する。**

フェーズ順は監査により改訂: retrieval を先に直してから claim/critic を載せる。

### 改訂フェーズ順

| Phase | 内容 | 根拠 |
|-------|------|------|
| 0a | Provider/CDC 整合化 | 全ての土台。実 API とテストの齟齬を先に解消 |
| 0b | Eval Framework | 改善を定量測定する基盤。multi-protocol 評価 |
| 1 | Scope 伝播 (report_briefs) | 入力契約。全下流の前提条件 |
| 2 | Multi-query Retrieval + Hydrate | retrieval 契約。poor retrieval の上に extractor を載せない |
| 3 | Evidence-grounded Writing | claim 契約。良い retrieval の上に構築 |
| 4 | Critic Redesign | critic 契約。GroUSE 失敗モード検出 |

### パイプライン変更

```
Before: planner → gatherer → curator → writer → critic → finalizer (6ノード)
After:  planner → gatherer → curator → hydrator → extractor → writer → critic → finalizer (8ノード)
```

### 主要な設計判断

#### 1. scope を `reports` テーブルの JSONB ではなく `report_briefs` テーブルに正規化

CLAUDE.md の「JSONB for auxiliary only」ルールに従い、`topic`, `report_type` は TEXT カラム、`entities`, `exclude_topics` は TEXT[] で定義。Proto の `map<string, string> scope` から `ReportBrief.from_scope()` で変換。`report_versions.scope_snapshot` は version ごとの immutable snapshot として従来通り保持。

#### 2. ContentStore パターンで metadata-first / body-only-for-top-N を実現

search-indexer に article-by-ID endpoint がないため、Gatherer が検索時に content を run-scoped `MemoryContentStore` に格納し、Hydrator が curated top-N のみの body を state に取り出す。CLAUDE.md の「Fetch metadata first, body only for top-N」ルール準拠。

#### 3. CDC テストを実 API に合わせて修正

CDC テストが `POST /indexes/articles/search`（Meilisearch 直接）を期待していたが、実装は `GET /v1/search`（search-indexer REST）を使用。レスポンスフィールドも `id, title, content, tags` のみに正規化。

#### 4. Critic の parse 失敗 → revise（accept ではない）

旧実装は JSON parse 失敗で `{"verdict": "accept"}` にフォールバックしていた。これを `{"verdict": "revise"}` に変更。parse 不能な出力を受け入れるのは品質保証の意味を失うため。

#### 5. GroUSE 失敗モード検出でヒューリスティックと LLM を使い分け

FM2（メタ発言）と FM3（不完全情報）は正規表現ヒューリスティックで検出（LLM 不要）。FM1（relevancy）と FM7（根拠なし主張）は LLM structured output。ヒューリスティックで blocking issue が見つかれば LLM 呼び出しをスキップし、コスト削減。

#### 6. Pydantic validation を全 structured output に適用

`json.loads` だけでは不十分（Ollama 公式ドキュメントも Pydantic/Zod での再検証を推奨）。`generate_validated` ユーティリティで GBNF 文法 → json.loads → Pydantic TypeAdapter → retry → fallback のフローを統一。

#### 7. Eval は multi-protocol（single-judge ではない）

Checklist（rule-based、LLM 不要）+ Rubric（LLM-based claim 検証）+ Pairwise（比較）の複合評価。LLM-as-judge 単体は verbosity/position/sentiment bias があるため（NeurIPS 2025）。

### 不採用とした代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| `reports` テーブルに `scope_jsonb` 追加 | JSONB auxiliary ルール違反、`report_versions.scope_snapshot` と重複 |
| `curator_node.py:49` の `except A, B:` を修正 | PEP 758 により Python 3.14 で合法。runtime bug ではない |
| 全 hit に full content を持たせる | 「Fetch metadata first, body only for top-N」ルール違反 |
| Multi-HyDE を本線採用 | 一般化根拠が金融 QA の論文に限定。実験フラグ扱いが妥当 |

## Verification

### パイプライン実行結果 (2026-04-09)

scope 付きレポート「2026年Q2 AI半導体サプライチェーン分析」を生成し、改善前の問題が解消されたことを確認。

| ノード | 結果 | 備考 |
|--------|------|------|
| Planner | fallback 3 sections | Gemma4 thinking mode で JSON parse 失敗 (ollama/ollama#15260) |
| Gatherer | 3 queries → 5 evidence hits | 1 query は search-indexer 500 error |
| Curator | 3 sections curated | per-section filtering 動作 |
| Hydrator | 5/5 hydrated | ContentStore 正常 |
| Extractor | 2 facts / 5 articles | 3/5 は num_predict 不足で fallback → 4096 に修正済 |
| Writer | 3 sections (3937 chars) | citation [1][2] が本文に出現 |
| Critic | 1st: revise (FM 検出), 2nd: revise, 3rd: revise → MAX_REVISIONS accept | ゴム印ではなくなった |
| Finalizer | version 1 永続化 | scope_snapshot 保存 |

### Checklist 評価スコア: 0.83

| Check | Result |
|-------|--------|
| Topic in content | PASS (4/4 words) |
| All sections generated | PASS (3/3) |
| Section length | FAIL (conclusion 174 chars) |
| No meta-statements | PASS |
| Low duplication | FAIL (bigram overlap 0.32) |

### 改善前との比較

| 指標 | 改善前 (ADR-654) | 改善後 |
|------|-----------------|--------|
| scope 伝播 | `scope = {}` → "technology trends" | brief.topic 使用 |
| メタ発言 | 「トピックが明示されていません」 | 検出なし |
| Critic verdict | 常に accept | revise を3回発行 |
| Citation | なし | [1], [2] が本文に出現 |
| 検索クエリ | 単一 "technology trends" | 3 section-specific queries |
| Evidence hydrate | 200文字 excerpt のみ | full body 5件 hydrated |

### 残課題

1. Planner structured output が Gemma4 thinking mode で壊れる → ollama/ollama#15260 の修正待ち
2. Critic LLM verdict が常に revise → prompt tuning 必要
3. Section duplication 0.32 → writer prompt で重複抑制を強化する余地あり

## Consequences

### Pros

- **scope 伝播**: ユーザーの意図がパイプライン全体に到達し、タイトルと内容の乖離が解消される
- **retrieval 契約整合**: CDC テストと実装が一致し、CI/CD での回帰検知が信頼できる
- **証拠ベース執筆**: Writer がハルシネーションを避け、引用付きの主張を生成する構造になった
- **厳格な critic**: メタ発言・unsupported claims がパイプラインから排除される
- **定量評価**: 改善を数値で測定・比較でき、regression を検知できる
- **Pydantic validation**: structured output の解析失敗が silent accept にならなくなった

### Cons / Tradeoffs

- LLM 呼び出し数が ~8 → ~24 に増加。パイプライン実行時間が +10-15分延びる
- Extractor が Gemma4 26B の context window 制約（記事 body を 2000 文字に切り詰め）に依存
- search-indexer REST API が `_rankingScore` を返さないため、evidence ranking は Meilisearch の暗黙順序に依存
- Recap 検索は Connect v2 経由のみで、REST 未対応のため当面未接続
- PostgresReportGateway に `create_brief` / `get_brief` が未実装（MemoryReportGateway のみ対応）。本番 DB 接続には追加実装が必要
