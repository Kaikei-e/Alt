# ADR-000925: Knowledge Loop の act_targets[0].source_url 欠落を SurfaceScoreInputs.SourceURL pin と v14 reproject で根治する

## Status

Accepted

## Date

2026-05-27

## Affected Services

- knowledge-sovereign (Go): `SurfaceScoreInputs` に `SourceURL` を追加、`EventLogSurfaceScoreResolver` が `ArticleCreated` / `ArticleUpdated` / `ArticleUrlBackfilled` から `event_seq` 降順で http(s) URL を pin、`seedActTargets` が `inputs.SourceURL` を fallback として読む、`isHTTPSourceURL` helper を `articleActSourceURL` から抽出して event-derived 経路と allowlist を共有、`marshal/parseSurfaceScoreInputs` に `source_url` を追加、`WhyMappingVersion` を v13 → v14 に bump
- docs/runbooks/knowledge-loop-reproject.md: v14 履歴行と `article_no_url = 0` を確認する post-check SQL、admin re-emit escape hatch を追記
- docs/features/knowledge-loop/projector-matrix.md: `ArticleUpdated` 行を追加し、matrix と `constants.go` の binding test を緑に保つ

## Context

[[000924]] の Pillar 2 で `SurfaceScoreInputs.ArticleID` は prior event chain から pin する形に直したが、**同じ問題が `SourceURL` には残っていた**。`seedActTargets` は `SourceURL: articleActSourceURL(ev)` で **現在 projecting 中の event payload からのみ** URL を取り出していたため、`ArticleCreated` / `ArticleUpdated` / `ArticleUrlBackfilled` 以外の event (`SummaryVersionCreated` / `TagSetVersionCreated` / `TagSetSuperseded` / `knowledge_loop.observed.v1` / `augur.conversation_linked.v1` / `knowledge_loop.surface_plan_recomputed.v1` 等) が article aggregate に当たって projector を再 trigger するたびに `act_targets[0].source_url` が `""` で書き直されて剥がれた。

2026-05-27 時点での実測:

```sql
SELECT count(*), count(act_targets->0->>'source_url') FROM knowledge_loop_entries;
-- total = 59129, with_source_url = 50810 → 約 8319 entries が URL 欠落 (約 14%)
```

検体 entry `article:2a4ca09b-ad3a-43ff-9a00-29837460dcdb`:

- `ArticleCreated` payload: `{"url": "https://hackernoon.com/when-is-shutting-down-...", "article_id": "2a4ca09b-..."}` (URL あり)
- 最新の projection event は `event_seq=1304935 = knowledge_loop.surface_plan_recomputed.v1` (payload に URL なし)
- 結果として書き戻された `act_targets`: `[{"route":"/articles/2a4ca09b-...","target_ref":"2a4ca09b-...","target_type":"article"}]` (`source_url` キーが消滅)

FE 側の影響経路:

1. `alt-frontend-sv/src/lib/utils/loop-source-url.ts` の `resolveLoopSourceUrl()` は `actTargets[].sourceUrl` が空だと `null` を返す
2. `/loop/+page.svelte` は Decide stage の Open ボタンを「Open · resolve url」recovery ラベルに切り替え、BFF `/loop/article-source-url` を叩く動線に落ちる (b79d354e0 で追加)
3. `LoopEntryTile.svelte` の内部 fallback (`actTargets.find(t => t.targetType === "article").route`) は `/articles/<id>` という内部 SPA path を返す
4. `onEntryOpen()` が `isSafeInternalPath` 経由で `goto("/articles/<id>")` し、**`?url=<encoded>` query を付けずに**遷移
5. `/articles/[id]/+page.svelte` は `articleUrl = page.url.searchParams.get("url")` が `null` なので `fetchContent()` も `handleSummarize()` も `if (!articleUrl) return` で no-op
6. ユーザには **Article 本文も AI Summary も生成されない** ように見える → 「Article と Summary がエラー」と知覚

`/log-seeker` 調査 (2026-05-26 daily) と本セッションの追跡で、ユーザが報告した `[not_found] not_found https:// example.com/augur/<conv-id>` の背後にもこの systemic な URL 欠落があり、[[000924]] の `4a6b3464e` で入れた `CodeNotFound` 透過は alt-backend 側として正しく動いていた (BFF ログ 14:47:18 / 14:47:27 で 500 ではなく 404 が出ていた) のに、その先で FE が `errorMessage` を上書き表示していた、というのが二重露呈の構造だった。

## Decision

[[000924]] の `ArticleID` pin と完全に並走する形で、**`SurfaceScoreInputs.SourceURL` を新設し、resolver で prior article event chain から最新 URL を pin、`seedActTargets` で event-derived > inputs の優先順で書き戻す**。既存 entries の backfill は **`WhyMappingVersion` を v13 → v14 に bump して full reproject で行う** ことを採用する。

具体的な実装:

1. **`SurfaceScoreInputs.SourceURL string`** を `decide_bucket_v2.go` line 138 直後に追加。コメントは `ArticleID` と同じ「prior event chain から pin / event-derived が勝つ / reproject-safe」trio を明示
2. **`EventLogSurfaceScoreResolver` の 2-pass 化** (`score_resolver_event_log.go`):
   - 1st pass: 既存通り `out.ArticleID` を確定 (`SummaryVersionCreated` から fallback)
   - 2nd pass: 確定済 `out.ArticleID` をキーに `ArticleCreated` / `ArticleUpdated` / `ArticleUrlBackfilled` のみを scan し、`event_seq` 降順で最新の payload `url` (legacy fallback `link`) を `isHTTPSourceURL` で validate して `out.SourceURL` に pin。`article_id` 不一致や非 http(s) スキームは defense-in-depth で drop
3. **`isHTTPSourceURL` helper** を `articleActSourceURL` から抽出し、event-derived path (`seedActTargets` の `articleActSourceURL(ev)`) と resolver pin path で同一 allowlist を共有
4. **`seedActTargets` の fallback** (`projector.go`):

   ```go
   sourceURL := articleActSourceURL(ev)
   if sourceURL == "" {
       sourceURL = inputs.SourceURL
   }
   ```

   event-derived URL が常に勝つので、event log を canonical source として扱う reproject-safe 不変が保たれる
5. **`marshal/parseSurfaceScoreInputs` に `source_url` 追加** で `knowledge_loop.surface_plan_recomputed.v1` snapshot 経由でも `SurfaceScoreInputs.SourceURL` が round-trip する。`parseSurfaceScoreInputs` 側でも `isHTTPSourceURL` で再 validate
6. **`resolverEventTypes` allowlist 拡張**: `ArticleCreated` / `ArticleUpdated` / `ArticleUrlBackfilled` を追加。bucket placement には寄与しない (URL pin source only) ことをコメントで明示し、`TestEventLogResolver_AllowlistMatchesContract` を併せて更新
7. **`constants.go` に `EventArticleUpdated = "ArticleUpdated"` を新規定数化**。DB には 48674 件流れていたが sovereign 側に定数が無く、`TestProjectorMatrix_DocCovered` が constants と matrix doc の binding を強制するため、`docs/features/knowledge-loop/projector-matrix.md` にも `ArticleUpdated` 行を追加
8. **`WhyMappingVersion` 13 → 14 bump**。bump 文言には ADR 番号や Phase マーカーを書かず、決定の意図 (resolver pin / `inputs.SourceURL` fallback / full reproject で 8319 entries 復元) と数値根拠 (59129 / 50810) のみを記す
9. **runbook 更新** (`docs/runbooks/knowledge-loop-reproject.md`):
   - WhyMappingVersion 履歴に v14 行を追加
   - § "Post-check" に bug 専用 SQL を新設し、`article_no_url = 0` を期待値として固定
   - admin escape hatch (`POST /admin/jobs/knowledge-url-backfill/run`) を緊急時 fallback として併記

### 検討した代替案と却下理由

#### (A) `knowledge_url_backfill_usecase` admin re-run のみで backfill する

却下。`appendCorrective` が使う dedupe key `article-url-backfill:<article_id>` は単発 namespace なので、過去に一度でも emit 済の article については再 emit が `SkippedDuplicate` で no-op になる。今回の 8319 件は legacy URL 欠落ではなく「新規 article で非-article event が後から当たって剥がれた」のが主体なので、(A) では事実上ほぼ救えない。

ただし `projectArticleUrlBackfilled` + `PatchKnowledgeLoopActTargetSourceURL` の patch SQL は `NOT (act_targets->0 ? 'source_url')` の冪等ガードを持つため、新規 projector fix と非干渉に共存できる。緊急時の局所修復としては有用なので、runbook には escape hatch として残す。

#### (B) lookup の `scoreWindow` を SourceURL pin のみ広げる

却下。`scoreWindow = 7d` は canonical contract §6.4.1 に紐づく決定で、ここを fix の都合で曲げると後続の score 計算 (`VersionDriftCount` / `RecapClusterMomentum` 等) のセマンティクスが揃わなくなる。代わりに v14 reproject で event 列を順次再生する経路を取れば、ArticleCreated が 7d 内に存在する状態で後続 event が当たるので pin は成立する。7d を超える legacy article は `ArticleUrlBackfilled` 補正 event (DB に 81985 件存在) の replay で patch path に乗る。

#### (C) FE 側だけで `article.route` から `/articles/<id>?url=` を再構築する

却下。route は内部 SPA path であり外部 URL ではないため、recover には FE → BFF の追加 round-trip が必要 (= 既存の "Open · resolve url" 動線と同じ)。本質的に projection 側で URL を保てていないのが原因で、表面の症状を FE で繕うと「成功体験のラベル」が壊れる (`Open` ではなく `Open · resolve url` が常時表示される) ため、根本治療として projector 側に寄せる。

### TDD 規律

CLAUDE.md「TDD の RED→GREEN は別 commit が必要」に従い、2 commit に分けた:

- `9b23b961d` (test): `SurfaceScoreInputs.SourceURL` を struct stub だけ追加し、resolver pin 4 本 + `seedActTargets` fallback 3 本の合計 7 本の失敗テストを landing。3 本が assertion 失敗で RED、4 本は degenerate に pass (= 空状態で偶然満たす)

## Consequences

### Pros

- **Knowledge Loop の Decide / Act 体験回復**: act_targets[0].source_url が安定するため、`/loop` の "Open" ボタンは recovery ラベルではなく `Open` のままになり、SPA reader (`/articles/<id>?url=...`) に正しい query 付きで遷移して本文取得と AI Summary が動く。"Article と Summary がエラー" 報告の根因が消える
- **reproject-safe 不変の維持**: event-derived URL > inputs.SourceURL の優先順を守るので、event log が canonical source という invariant は変わらない。新規 invariant test `TestInvariant_ReplayDeterminism_SourceURLPin` が 2 度 resolve で bit-identical を pin
- **allowlist 共有によるセキュリティ単一点化**: `isHTTPSourceURL` を抽出したので、event-derived path と resolver pin path の URL validation が必ず揃う。javascript: / data: / file: / relative などの非 http(s) スキームが片側だけ slip through する余地が消えた
- **deterministic backfill**: v14 reproject 後に runbook の post-check SQL (`article_no_url = 0`) で coverage を 1 行で確認できる。観測可能性が改善されたので、将来同種の degradation が起きてもメトリクスで先に気付ける
- **[[000924]] のパターン継承**: ArticleID pin と同じ 2-pass design / 同じ JSON round-trip / 同じ runbook history 書式を踏襲したので、次に類似 invariant を pin したくなったとき (例: article_title / publish_date / canonical lang) の母型が再利用できる

### Cons / Tradeoffs

- **`scoreWindow = 7d` 外の legacy article は救えない**: 7 日より古い `ArticleCreated` しか持たない article については、後続 event が当たっても resolver lookup が ArticleCreated を見ない。実運用上は `ArticleUrlBackfilled` 補正 event (DB に 81985 件) が replay で patch path に乗るので大半カバーできる見込みだが、両者とも持たない article は v14 reproject 後も `article_no_url > 0` で residual に残る可能性がある。runbook の post-check で residual を数値で観測できるようにし、必要なら admin re-emit escape hatch で個別救済する
- **lookup query の scan 幅が広がる**: `resolverEventTypes` に `ArticleCreated` / `ArticleUpdated` / `ArticleUrlBackfilled` の 3 つを追加したため、SQL の event_type IN (...) 句が拡張される。`ArticleUrlBackfilled` は 81985 件、`ArticleUpdated` は 48674 件と件数が多いので、tail latency への影響を観測する必要がある。`r.limit = 256` の cap は据え置きだが、限度に張り付いたケースでは旧来の bucket 寄与 event (`SummaryVersionCreated` 等) が削られるリスクがある。post-deploy で `crossUserIsolationViolationTotal` だけでなく resolver の per-call latency も確認する
- **v14 bump で全 entries を作り直す**: 59129 entries の full reproject を伴うので、cutover 時間中は read path の placement / why_text / act_targets が一時的に旧 v13 と新 v14 の混在状態を経由する。runbook の Pre-flight / Procedure を踏襲する限り問題ないが、不用意に `up <svc>` だけ叩くと checkpoint reset を忘れて catch-up が走らないので、必ず runbook 経由で実施する
- **未解決の二次バグ**: ユーザが報告した `[not_found] not_found` の表示は、`alt-frontend-sv/src/routes/(app)/augur/[conversationId]/+page.svelte` の `errorMessage` 表示自体が Connect-RPC error code を素出ししているのが原因。`source_url` 問題と独立して残っており、Out of Scope (本 ADR では扱わない) として明示。次セッションで FE 側 (CodeNotFound 分岐で「この会話は存在しません — 新しいチャットを始める」CTA) の改善を別 ADR で扱う候補
- **デプロイ未実施**: `WhyMappingVersion v14` を含む binary は本 ADR 起票時点で local commit のみ。push / build / production reproject はユーザ明示指示まで待機する (CLAUDE.md "Plan 承認 ≠ build / push / deploy 許可")
