# ADR-000659: Section Claim Planner を追加し Writer を fact-first 化する

## Status

Accepted

## Date

2026-04-09

## Affected Services

- acolyte-orchestrator (Python 3.14+)

## Context

[[000656]] で整備した 8 ノードパイプライン（planner → gatherer → curator → hydrator → extractor → writer → critic → finalizer）は動作するが、**ExtractorNode が生成した `extracted_facts` を WriterNode が一切読んでいない**という構造的な断絶があった。

WriterNode は `curated_by_section` と `hydrated_evidence` の body テキスト（先頭 300 文字）をそのまま LLM に渡して自由文生成させており、atomic facts は抽出されるだけで消費されなかった。この断絶により以下の品質問題が継続していた:

1. **Analysis↔Conclusion 重複**: 全 section が同じ prompt 構造で、section 間の差分制御がない
2. **根拠なし主張**: raw evidence から自由生成するため、LLM のハルシネーションを構造的に防げない
3. **citation の不在**: `[1], [2]` はテキスト上の装飾であり、claim → source_id の構造化対応がない
4. **評価レポート上の問題**: Conclusion の新規引用率 0%、bigram 重複率 0.32

学術的には以下が設計根拠となる:

- **Attribute First, then Generate** (ACL 2024): content selection → sentence planning → generation の 3 段分離で fine-grained attribution を実現
- **FActScore** (EMNLP 2023): 長文の factuality は atomic fact 単位で扱うべき
- **Ollama structured output**: JSON schema による GBNF grammar enforcement + Pydantic validation

## Decision

**ExtractorNode と WriterNode の間に `SectionPlannerNode` を挿入し、section ごとの `ClaimPlan` を先に作ってから Writer に渡す。Writer は raw evidence ではなく ClaimPlan からのみ本文を構成する。**

### パイプライン変更

```
Before (8 nodes):
  planner → gatherer → curator → hydrator → extractor → writer → critic → finalizer

After (9 nodes):
  planner → gatherer → curator → hydrator → extractor → section_planner → writer → critic → finalizer
```

content_store なしのパスでは section_planner をスキップし、Writer はレガシーの evidence-based パスにフォールバックする（後方互換）。

### 主要な設計判断

#### 1. ClaimPlan を Pydantic モデルとして domain 層に定義

```python
class PlannedClaim(BaseModel):
    claim: str                      # 主張文
    claim_type: str                 # factual | statistical | comparative | synthesis
    evidence_ids: list[str]         # 根拠となる source_id
    supporting_quotes: list[str]    # 裏付ける引用文
    numeric_facts: list[str]        # 含めるべき数値データ
    novelty_against: list[str]      # 重複禁止対象の section_key
    must_cite: bool = True
```

`ExtractedFactModel`（extraction 層）と `PlannedClaim`（composition 層）は異なるドメイン概念であるため、別ファイル `domain/claim.py` に分離した。ExtractedFact はソース単位、PlannedClaim はセクション単位であり、ライフタイムと意味論が異なる。

#### 2. SectionPlannerNode は section ごとに facts をフィルタしてから plan 生成

全 facts をまとめて渡すのではなく、各 section の `curated_by_section[key]` に含まれる evidence_ids で extracted_facts をフィルタし、section に関連する facts のみを prompt に注入する。これにより:

- 無関係な facts がノイズとして混入しない
- section 間の claim 重複が構造的に減る
- context window の利用効率が上がる

`generate_validated()` を使い、`ClaimPlannerOutput` の JSON schema で Ollama GBNF grammar enforcement + Pydantic validation を行う。`temperature=0`。parse 失敗時は空 claims にフォールバック。

#### 3. WriterNode は dual-mode（claim-based + legacy fallback）

`claim_plans` が state にある場合は claim-based prompt を使い、ない場合はレガシーの evidence-based prompt にフォールバックする。これにより:

- content_store ありの本線パス: claim-based generation（ClaimPlan 必須）
- content_store なしのパス: レガシー動作を維持（後方互換）
- `RerunSectionUsecase` が import する `WRITER_PROMPT` と `_format_evidence` は変更なし

claims が空の section は空文字を返し、LLM を呼ばない。ハルシネーション防止。

#### 4. Revision ループでは section_planner を再実行しない

critic → writer の revise パスで section_planner をスキップする。理由:

- claim_plans は evidence から導出された事実であり、revision は表現・構成に対するフィードバック
- LLM 呼び出しコストを抑える（section 数 × revision 回数分の追加呼び出しを回避）
- claim_plans は LangGraph state に残るため、writer は同じ claims で revision_note 付き再生成する

### 不採用とした代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| WriterNode が直接 extracted_facts を読む | facts を section 単位で整理する中間表現がないと、section 間重複を防げない |
| レガシーパスを完全除去 | content_store なしのパイプラインが壊れる。後方互換が失われる |
| critic → section_planner → writer の revise ループ | 事実は変わらず表現を改善するだけなので、planner 再実行はコスト対効果が低い |
| ClaimPlan を fact.py に統合 | extraction 層と composition 層は異なるドメイン概念。SRP 違反 |

## Consequences

### Pros

- **Fact-first 生成**: Writer が raw evidence ではなく構造化された ClaimPlan からのみ本文を構成。ハルシネーションが構造的に減る
- **Section 間重複の抑制**: 各 section の facts が事前にフィルタされ、`novelty_against` で重複禁止が型として表現される
- **Attribution の準備**: claim → evidence_id の対応が構造化されており、Issue 2 の citation 永続化の基盤になる
- **テスト可能性**: ClaimPlan が中間表現として存在するため、extraction と generation を独立にテスト・評価できる
- **後方互換**: content_store なしのパイプラインは従来通り動作

### Cons / Tradeoffs

- LLM 呼び出し数が section 数分増加（3-5 sections で +3-5 呼び出し）
- SectionPlannerNode の出力品質は LLM の structured output 精度に依存。Gemma4 thinking mode で JSON parse 失敗する場合は空 claims にフォールバック
- WriterNode の dual-mode は技術的負債。content_store が全パスで必須になった時点でレガシーパスを除去すべき
- Revision ループで claim 自体の修正ができない。critic が facts レベルの問題を指摘した場合、現状では対応できない
