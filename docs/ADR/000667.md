# ADR-000667: Gatherer に section-aware query decomposition を導入し検索品質を構造化する

## Status

Accepted

## Date

2026-04-10

## Affected Services

- acolyte-orchestrator (Python 3.14+)

## Context

exec1.md Issue 6 の実装。Issues 1-5（[[000659]] SectionPlannerNode 追加、[[000661]] section_citations 永続化、[[000662]] Conclusion 分離、[[000664]] ES 再合成、[[000666]] section contract 付き半決定論 Planner）により composition 層と section contract の基盤は整ったが、Gatherer の検索戦略に以下の課題が残っていた。

1. **検索クエリが素の文字列配列のまま** — Planner が生成する `search_queries: list[str]` をそのまま search-indexer に投げており、intent（調査・比較・トレンド）や時間範囲、エンティティといったファセット情報が失われている
2. **ReportBrief のメタデータが未活用** — `entities`, `time_range`, `exclude_topics` が brief に存在するが、Gatherer は一切参照していない
3. **検索の弱点が不可視** — どのクエリが十分な結果を返さなかったか追跡する手段がなく、retrieval quality の改善ループが組めない
4. **synthesis-only セクション（Conclusion, ES）が無駄な検索を実行** — [[000666]] で `synthesis_only` flag が導入されたが、Gatherer はこれを無視して全セクションの search_queries を実行していた

### 設計判断の分岐: LLM 生成 vs 決定論的分解

exec1.md は `search_queries` を `[{intent, entities, must_have_terms, optional_terms, time_range, source_bias}]` に拡張し、Planner にこの構造を生成させる設計を想定していた。しかし、resolve ドキュメントと [[000666]] の Template-First 方針に基づき、以下の制約が適用される。

- **LLM スキーマを拡張しない** — Gemma4 の structured output 不安定性（Ollama #15260: `think=false` が `format` を壊す）を考慮し、Planner の JSON schema は `PlannerSection` の 4 フィールド（key, title, search_queries, section_role）のまま
- **Planner は半決定論** — ファセット分解は LLM ではなくコードで決定論的に実行
- **search-indexer API は flat query のみ** — `GET /v1/search?q={query}&limit={limit}` に高度なフィルタはない

## Decision

**決定論的 QueryFacet 分解パターンを採用する。**

LLM は `search_queries: list[str]` を今まで通り生成する。PlannerNode の post-processing（`_enrich_with_contract()` 拡張）で、ReportBrief のメタデータと section_role を使い、各クエリ文字列を `QueryFacet` オブジェクトへ決定論的に分解する。GathererNode は facet を読み、最適化した検索文字列を生成し、weak facet を検出する。

### 1. Domain Model: `QueryFacet` (`acolyte/domain/query_facet.py`)

新規ドメインモデルとして `QueryFacet` (Pydantic BaseModel) を作成:

- `intent` — 検索意図。section_role とクエリキーワードから推論 (`"investigate"` | `"compare"` | `"trend"` | `"background"`)
- `raw_query` — LLM が生成した元の検索クエリ文字列
- `entities` — brief.entities とクエリの共通部分から抽出
- `must_have_terms` — クエリからストップワード除外した上位 2-3 語
- `optional_terms` — topic の語で must_have に含まれないもの
- `time_range` — brief.time_range をそのまま伝播
- `source_bias` — デフォルト `"article"`（recap は Issue 9 で対応）

Pure functions:

- **`decompose_queries(search_queries, brief_dict, section_dict) -> list[QueryFacet]`** — 決定論的分解。synthesis_only=True のセクションは空リスト返却
- **`render_query_string(facet) -> str`** — QueryFacet を search-indexer API 用のフラットなクエリ文字列に変換。must_have_terms + entities を結合し、120 文字でカット。dict / QueryFacet 両方を受容
- **`WEAK_FACET_THRESHOLD = 2`** — これ未満の hit 数を weak facet として記録

Intent 推論ロジック:

| クエリキーワード | Intent |
|---|---|
| `vs`, `versus`, `compared`, `比較` | `compare` |
| `trend`, `forecast`, `growth`, `動向` | `trend` |
| （analysis セクション、上記なし） | `investigate` |
| （その他） | `background` |

### 2. SectionContract 更新

`query_facets` フィールドの型を `list[str]` → `list[dict]` に変更。デフォルト `[]` のまま後方互換。

### 3. PlannerNode: `_enrich_with_contract()` 拡張

- シグネチャに `brief: dict | None = None` を追加（デフォルト None で後方互換）
- テンプレートマージ・novelty_against 解決の後に `decompose_queries()` を呼び出し
- synthesis_only=True のセクションは `query_facets = []`
- 呼び出し元 `PlannerNode.__call__` で `brief=brief` を渡す

### 4. GathererNode: デュアルパス + weak facet 検出

- **Faceted path** (`_search_by_facets()`): outline に `query_facets` が存在する場合に使用。各 facet を `render_query_string()` でフラット化して検索。facet ごとの hit 数を追跡
- **Legacy path** (`_search_by_queries()`): `query_facets` がない場合のフォールバック。既存の `search_queries` イテレーションロジックをメソッド抽出
- **Weak facet 検出**: hit 数 < `WEAK_FACET_THRESHOLD` の facet を `weak_facets` リストに記録（section_key, facet_index, intent, raw_query, hit_count）
- 出力に `weak_facets: list[dict]` を追加

### 5. State 更新

`ReportGenerationState` に `weak_facets: list[dict]` フィールドを追加。`TypedDict(total=False)` のため下流ノードへの影響なし。将来の query-rewrite ノードで消費予定。

### 主要な変更ファイル

| ファイル | 変更 |
|---------|------|
| `acolyte/domain/query_facet.py` | **新規** QueryFacet, decompose_queries, render_query_string, WEAK_FACET_THRESHOLD |
| `acolyte/domain/section_contract.py` | `query_facets: list[str]` → `list[dict]` |

| `acolyte/usecase/graph/state.py` | `weak_facets: list[dict]` 追加 |

### 変更なし

| ファイル | 理由 |
|---------|------|
| `acolyte/port/evidence_provider.py` | Port インターフェース不変（`search_articles(query: str, limit: int)` のまま） |
| `acolyte/gateway/search_indexer_gw.py` | Gateway 実装不変（flat `q=` クエリのまま） |
| `tests/contract/` | CDC 契約不変 |
| LLM プロンプト / JSON schema | LLM 負荷増なし |
| 下流ノード (curator〜finalizer) | query_facets を直接消費しない |

## Consequences

### Pros

- **LLM スキーマ負荷ゼロ増加** — Planner の JSON schema は 4 フィールドのまま。Gemma4 structured output 不安定性を回避（resolve 方針 + [[000666]] 準拠）
- **ReportBrief メタデータ活用** — entities, time_range が検索ファセットに反映され、検索精度向上
- **検索品質の可視化** — weak_facets により hit 不足のファセットが特定可能。将来の query rewrite ノードの入力として機能
- **synthesis-only セクションの無駄排除** — Conclusion/ES が不要な検索を実行しなくなり、API 呼び出し削減
- **後方互換** — query_facets がない outline では既存の search_queries パスにフォールバック
- **Clean Architecture 準拠** — QueryFacet はドメイン層の純粋モデル。render_query_string はドメイン層の純粋関数。Gateway 層は変更なし

### Cons / Tradeoffs

- **日本語クエリの分かち書き精度** — 現状は空白分割のみ。MeCab 等の形態素解析器が必要になった場合は別 Issue で対応
- **render_query_string の 120 文字制限** — 長いエンティティリストや複雑なクエリではトランケーションにより情報欠損の可能性あり
- **weak_facets が現時点で未消費** — 検出のみで自動リライトは未実装。将来の query-rewrite ノードまたは degrade policy（resolve 第5項）で活用予定
- **source_bias が article 固定** — recap 検索は search_recaps() が未実装（Issue 9）のため、source_bias による article/recap ルーティングは将来対応
