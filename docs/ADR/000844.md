# ADR-000844: Knowledge Loop の why_text と stage 別 CTA を canonical contract に整合させる

## Status

Accepted

## Date

2026-04-26

## Affected Services

- alt-backend (Go) — `app/usecase/knowledge_loop_usecase/why_enricher.go`、`validator.go` の `WhyMappingVersion` v2→v3、`app/job/knowledge_loop_projector.go` の `seedDecisionOptions` 書き換え + 関連テスト更新
- alt-frontend-sv (SvelteKit) — `src/lib/components/knowledge-loop/LoopEntryTile.svelte` の `ctaToStage` / `handleCta` / disabled 計算修正

## Context

ユーザから `/loop` が「何の役にも立たない」との報告。実機では以下 3 点が同時に発生していた。

1. 3 件のカードがすべて **"New summary"** とだけ表示される
2. カードを展開しても **Ask しか実質的に押せない**（他のボタンは disabled）
3. Ask 経由で Augur に遷移しても、conversation 冒頭に **意味あるコンテキストが現れない**

調査の結果、これらは独立した 3 件のバグではなく **canonical contract ([[knowledge-loop-canonical-contract]]) §7 / §11 の不履行が連鎖したもの**であることが判明した。

### 1. why_text のプレースホルダ化（§11 違反）

`alt-backend/app/usecase/knowledge_loop_usecase/why_enricher.go:111` が `EventSummaryVersionCreated` に対して以下を書いていた:

```go
sb.WriteString("New summary")
if p.ArticleTitle != "" {
    sb.WriteString(": ")
    sb.WriteString(p.ArticleTitle)
}
```

contract §11 は `whyPrimary.text` を **「1..512 chars の plain narrative で、なぜこのエントリが浮上したかを説明する」** と定義する。`"New summary"` はイベント種別名にすぎず、ユーザに対して説明として機能しない。同じ問題は `enrichHomeItemsSeen`（"Surfaced in your home feed."）、`enrichHomeItemAsked`（"You asked about this item."）、`enrichSuperseded`（"A newer version is available."）など、ほぼ全ての enrich 関数に存在した。

### 2. seedDecisionOptions が §7 transition allowlist に反する

`app/job/knowledge_loop_projector.go` の `seedDecisionOptions` は `WhyKindSource` かつ `LoopStageObserve | LoopStageOrient` のエントリに対して `[open, ask, save, snooze]` の 4 つの CTA を seed していた。

contract §7 が許可する transition は次の通り:

```
observe → orient
observe → decide
orient  → decide
decide  → act
act     → observe
```

FE の `LoopEntryTile.svelte` は `ctaToStage` 経由で `open` / `save` / `snooze` を **すべて `act` ステージへの transition** として解釈する。Observe ステージのエントリでは `canTransition(observe, act)` が **false** を返すため、3 つのボタンは disabled になり、`ask` だけが押せるという挙動が固定されていた。

`seedDecisionOptions` を導入した PR が、`proposedStage` と `currentSessionStage` の分離（contract §1）を取り違えて Observe エントリに Act 直行 CTA を割り当てたのが根本原因。

### 3. Augur seed message のコンテキスト消失

`rag-orchestrator/internal/usecase/augur_conversation_usecase.go` の `CreateSessionFromLoopEntry` は **正しく** seed assistant message を永続化していた:

```go
seed := &domain.AugurMessage{
    Role:    "assistant",
    Content: renderLoopSeed(why, input.EvidenceRefs),
    ...
}
```

`renderLoopSeed` は `"From your Knowledge Loop:\n\n" + whyText` を返す。すなわち handshake 経路と永続化はバグではなく、**`whyText = "New summary"` がそのまま seed の本文になっていた**ため、ユーザには「context が引き継がれていない」と見えていた。問題の出処は (1) と同一で、enricher を直せば自動的に解消する。

### 制約

- contract §3 invariants（append-first / reproject-safe / event-time purity / no `updated_at`）を維持する
- 既存の `knowledge_loop_entries` 行を破壊的に書き換えず、`WhyMappingVersion` bump 経由の reproject で更新を伝播させる
- proto / DB schema / RPC は不変。enrichment と FE の挙動だけを修正対象とする
- 「[Knowledge Sovereign が Knowledge Home の責務を担う](feedback_no_direct_frontend_backend.md)」という所有関係は別 PR で実現する（本 ADR では in-place fix に留め、`alt-backend` 上の projector の knowledge-sovereign 移管は後続に分離）

## Decision

`whyPrimary.text` の narrative を意味あるものに書き直し、`seedDecisionOptions` を §7 適合の stage 別 CTA に置き換え、FE `ctaToStage` を拡張する。`WhyMappingVersion` を v2 から **v3** に bump して runbook 経由 full reproject の明示シグナルとする。

### 1. enricher の narrative 書き換え（reproject-safe を維持）

```go
func enrichSummaryVersion(_ *domain.KnowledgeEvent, p enrichmentPayload) EnrichedWhy {
    var text string
    switch {
    case p.ArticleTitle != "":
        text = p.ArticleTitle + " — fresh summary ready to read."
    default:
        text = "A new summary is ready in one of your feeds."
    }
    return EnrichedWhy{
        Kind:         domain.WhyKindSource,
        Text:         sanitizePlainText(text),
        EvidenceRefs: boundEvidence(...),
    }
}
```

同様に `enrichHomeItemsSeen` / `enrichHomeItemAsked` / `enrichHomeItemOpened` / `enrichSuperseded` / `EventHomeItemDismissed` / fallback すべてを **payload-only / 純粋関数** で書き直す。`time.Now()` や DB ルックアップは導入しない。`sanitizePlainText` で HTML 角括弧スパンの除去と 512 byte truncation を維持。

### 2. seedDecisionOptions を stage 別 §7 適合に置き換え

```go
switch stage {
case domain.LoopStageObserve:
    seed = []opt{ {revisit}, {ask}, {snooze} }   // observe → orient（revisit）+ 非遷移
case domain.LoopStageOrient:
    seed = []opt{ {compare}, {ask}, {snooze} }   // orient → decide（compare）+ 非遷移
case domain.LoopStageDecide:
    seed = []opt{ {open}, {save}, {ask} }        // decide → act（open / save）
case domain.LoopStageAct:
    seed = []opt{ {revisit}, {ask} }             // act-stage は省察 CTA のみ
}
```

`whyKind` 引数は破壊しないが seed の判定からは外す。これは「契約は stage が driver、whyKind は narrative の語彙」という分離に従うため。

### 3. FE `ctaToStage` の写像拡張

```ts
case "revisit": return "orient";   // 旧版は null だった
case "compare": return "decide";   // 旧版は null だった
case "open":
case "save":   return "act";
case "snooze":
case "ask":
default:       return null;
```

`snooze` は `null` のままにし、`handleCta` 内で `onDismiss(entry.entryKey)` に分岐させる（contract §8.2 の `KnowledgeLoopDeferred` 同等の defer 経路）。disabled 計算にも `isSnoozeCta` 分岐を追加して `onDismiss` が wire されていれば押せるようにした。

### 4. WhyMappingVersion v2 → v3

`alt-backend/app/usecase/knowledge_loop_usecase/validator.go` の `WhyMappingVersion` を **3** に上げる。コメントで「why_text を placeholder から narrative に置き換え、Observe-only seed を stage 別 seed に置き換えた」を残し、runbook が full reproject を発火する基準とする。

### 検討した代替案

1. **「Observe entry にも open/save を残し、§7 を緩和して observe → act を許可」**: 却下。contract §7 は OODA の連続性を担保するための核で、UI 都合で破る価値が無い。「観察 → 即実行」を可能にすると Orient / Decide が形骸化する。
2. **「FE 側だけで disabled 状態のボタンを非表示にする」**: 却下。disabled をユーザに見せないと「機能が無い」と誤認する。ボタンが存在するなら押せるべき、というのが本来の状態機械 UI の規律。
3. **「event payload に `narrative_template_id` を追加して enricher を i18n テンプレート化」**: 却下。今回は narrative の量より「そもそも意味がある文を書く」が先。テンプレート化は payload 拡張を伴うため独立 ADR に切り出す価値がある。
4. **「enricher と projector job をこの PR で knowledge-sovereign に移管する」**: 却下。knowledge-sovereign は現在 Connect-RPC server only で job runner を持たない。read_events port + scheduler + alt-backend からの除去はマルチホップで rollback リスクが大きい。可視バグ修正と architectural cleanup を分離する。

## Consequences

### Pros

- `/loop` が contract §11 通りの narrative を表示するようになり、ユーザは何のためにエントリが浮上したかを 1 行で把握できる
- Observe ステージのカードで `revisit` / `ask` / `snooze` の 3 ボタンがすべて enabled になり、§7 transition allowlist に従って正しく orient へ進める
- Augur 遷移時の seed message が "From your Knowledge Loop:\n\n{真の why_text}" として機能し、handshake 経路の改修無しでコンテキスト引き継ぎが回復する
- `WhyMappingVersion` bump により reproject 起点が明確になり、既存行が runbook 1 回の操作で新 narrative に更新される
- `time.Now()` / DB ルックアップ無しの pure function を維持しているため、reproject 時に同一 event から同一 why_text が再現される（reproject-safe）

### Cons / Tradeoffs

- 既存の `knowledge_loop_entries` 行は **deploy 直後は古い placeholder のまま**であり、reproject runbook を別途実行するまで新 narrative に置き換わらない（runbook: `docs/runbooks/knowledge-home-projection-recovery.md`）
- `whyKind` の意味論的多様化（pattern_why / recall_why / change_why）はまだ enricher に十分実装されていない。`enrichHomeItemOpened` が `recall_why` を返すなど一部のみ。今後イベント payload に novelty signal や delta one-line を載せ、より stage-aware な narrative にする余地がある
- enricher と projector job が依然 alt-backend に残っており、Knowledge Home/Loop の責務オーナーが knowledge-sovereign であることと不整合（後続 ADR で移管予定）
- FE `ctaToStage` で `snooze` を `onDismiss` に貼り付けたが、HomeItemDismissed と KnowledgeLoopDeferred の差異（hard 削除 vs soft 後回し）はサーバ側で同経路に潰されている可能性があり、別途 transition event を分けるかの検討余地あり
- E2E (Playwright) と CDC (Pact) の追加テストは本 PR では deferred。Go 側の `TestSeedDecisionOptions_StageAppropriate` で contract boundary はカバー済みだが、ブラウザ DOM での button enabled 状態は次回スイープで確認する
