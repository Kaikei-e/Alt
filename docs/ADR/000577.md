# ADR-000577: Knowledge Lens の CreateLens/SelectLens 障害と永続化不具合を修正する

## Status

Accepted

## Date

2026-03-25

## Affected Services

- knowledge-sovereign (Go) — JSONB マーシャリング修正
- alt-backend (Go) — nil pointer dereference 防止
- alt-frontend-sv (SvelteKit) — onMount フィーチャーフラグ順序修正

## Context

[[000576]] で Knowledge Lens の LensModal UX 改善を実装した直後、以下の 3 つの障害が連鎖的に発生した:

### 障害 1: CreateLens 500 Internal Server Error

`CreateLensVersion` で `ERROR: invalid input syntax for type json (SQLSTATE 22P02)` が発生。`knowledge_lens_versions` テーブルの JSONB カラム 3 つに対して、Go の生の値を直接渡していたことが原因。

- `tag_ids_json` (JSONB) ← `string(json.Marshal(...))` — 文字列キャストが JSONB バインドを阻害
- `source_ids_json` (JSONB) ← 同上
- `time_window_json` (JSONB) ← `v.TimeWindow`（素の Go 文字列 "90d"）— JSON として不正

### 障害 2: SelectLens 502 Bad Gateway

障害 1 の結果、`knowledge_lenses` テーブルにはレコードが作成されたが `knowledge_lens_versions` への INSERT は失敗。孤立した lens レコード（version なし）が残った。

CreateLens 成功後に呼ばれる SelectLens が `GetCurrentLensVersion(lensID)` で `nil, nil` を受け取り、`version.LensVersionID` で nil pointer dereference → panic → HTTP/2 INTERNAL_ERROR → BFF が 502 を返した。

### 障害 3: Lens 永続化されない

ページリロード後に保存済み Lens がピルバーに表示されない。`+page.svelte` の `onMount` で、フィーチャーフラグ取得（`home.fetchData()` → `flags.setFlags()`）より前に `if (lensEnabled)` を評価していたため、初回ロード時は常に `false` となり `lens.fetchLenses()` がスキップされていた。

## Decision

**3 つの障害をそれぞれ根本原因に対して修正する。**

### 修正 1: JSONB マーシャリングの統一（knowledge-sovereign）

`read_lens.go` の `CreateLensVersion` で、全 JSONB カラムに `json.Marshal()` の結果を `[]byte` として直接渡すパターンに統一。

**WRITE 側:**
```go
tagIDsJSON, err := json.Marshal(v.TagIDs)
sourceIDsJSON, err := json.Marshal(v.SourceIDs)
timeWindowJSON, err := json.Marshal(v.TimeWindow)
// pool.Exec(..., tagIDsJSON, sourceIDsJSON, timeWindowJSON, ...)
```

**READ 側:**
`ListLenses` と `GetCurrentLensVersion` で `time_window_json` のスキャンを `*string` から `[]byte` + `json.Unmarshal` に変更。`tag_ids_json` / `source_ids_json` と同じパターンに統一。

### 修正 2: nil チェック追加（alt-backend）

`select_lens_usecase.go` の `Execute` メソッドで、`GetCurrentLensVersion` の戻り値に対して nil チェックを追加。

```go
if version == nil {
    return fmt.Errorf("no active version found for lens %s", lensID)
}
```

panic ではなく通常のエラーとして返すことで、クライアントに適切なエラーレスポンスが返る。

### 修正 3: onMount のフラグ評価順序修正（alt-frontend-sv）

`+page.svelte` の `onMount` を再構成し、先に `home.fetchData()` → `flags.setFlags()` でフィーチャーフラグを確定してから、`flags.isEnabled("enable_lens")` を評価して lens データをロードする。

```
// Before: lensEnabled(=false) → if分岐 → home.fetchData() → flags.setFlags()
// After:  home.fetchData() → flags.setFlags() → if(flags.isEnabled()) → lens.fetchLenses()
```

### 孤立データの削除

障害 1 で作成された version なしの孤立 lens レコード 4 件を sovereign DB から直接削除。

## Consequences

### Pros

- **CreateLens が正常動作** — JSONB カラムへの INSERT が成功し、lens + version が正しく永続化される
- **SelectLens が panic しない** — nil チェックにより、不整合データに対しても安全にエラーを返す
- **ページリロードで lens が復元される** — フィーチャーフラグ確定後に `fetchLenses()` が呼ばれ、保存済み lens がピルバーに表示される
- **JSONB 読み書きパターンの統一** — `tag_ids_json`, `source_ids_json`, `time_window_json` の 3 カラム全てが `[]byte` + `json.Marshal/Unmarshal` パターンに統一され、今後の同種バグを防止

### Cons / Tradeoffs

- **`home.fetchData()` が lens なしで 1 回実行される** — アクティブ lens がある場合、フラグ確定後に lens 付きで再取得するため、初回ロード時に最大 2 回の `fetchData` が発生する。パフォーマンス影響は軽微（lens フィルタリングはサーバーサイド未実装のため結果は同一）
- **孤立レコードの再発防止策なし** — `CreateLens` と `CreateLensVersion` はトランザクションで囲まれていないため、`CreateLens` 成功 → `CreateLensVersion` 失敗のパターンは原理的に再発可能。将来的にトランザクション化または saga パターンの導入を検討すべき
