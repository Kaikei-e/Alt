# ADR-000496: Save View を current-view 保存フローに再定義する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-frontend-sv (TypeScript / SvelteKit)
- alt-backend (Go)

## Context

Knowledge Home の `Save View` は、ユーザーが今見ている Home の状態を保存したい場面で使われる。しかし従来 UI は Lens 作成フォームの性格が強く、特に source 条件で内部識別子の入力や編集を意識させやすかった。

この設計には、以下の UX 問題があった。

1. **保存したい対象と操作が一致しない**: ユーザーは「今の見え方を保存したい」のに、実際には builder 形式の再入力を求められる。
2. **内部識別子の露出**: source 条件に人間可読ではない値が混じり、Knowledge Home の情報設計よりも実装都合が前に出る。
3. **Lens 選択後の編集導線が弱い**: active lens と現在の draft state が分離され、何を保存し直すのか分かりにくい。
4. **一覧要約が読みにくい**: 保存済み Lens の summary が内部条件寄りで、source/tag/query/time window を素早く把握しづらい。

Knowledge Home の UX では、Lens を「検索条件ビルダー」ではなく「Home の読み方のプリセット」として扱う方が自然である。

## Decision

**`Save View` を current-view 保存フローとして再定義し、Home 上の draft state をそのまま Lens として保存する。source 条件は購読中 source の picker で扱い、UUID などの内部識別子は UI に露出しない。**

### 検討した代替案

| 案 | 評価 |
|---|---|
| A. 既存 modal を維持し、入力補助だけ追加する | 根本の操作モデルは変わらず、再入力負荷が残る |
| B. current-view 保存へ寄せる（採用） | ユーザー意図と保存導線が一致し、Home 全体の理解負荷が下がる |
| C. Lens 作成を完全に別画面化する | 高機能だが、Knowledge Home の軽い保存体験には過剰 |

案 B を採用した。Home に `lensDraft` を持ち、検索語、source、tag、期間を現在の閲覧状態として管理し、その状態を `Save current view` で保存する。

## Consequences

### Pros

- **保存操作が自然になる**: 現在の Home 状態をそのまま保存でき、再入力が不要になる。
- **内部表現を隠せる**: source は購読中一覧から人間可読な候補として選択でき、UUID を直接扱わなくてよい。
- **Lens の可読性が上がる**: selector 上で query、tag、source 数、time window を人間向け summary として表示できる。
- **active lens の再編集がしやすい**: Lens 選択時に draft を hydrate することで、保存済み view の微調整と再保存が自然になる。

### Cons / Tradeoffs

- **Home 側 state 管理が増える**: `lensDraft` と active lens の同期を明示的に扱う必要がある。
- **購読外 source は扱わない**: 今回の picker は購読中 source を前提としており、未購読 source まで横断する保存体験は別途設計が必要。
- **詳細 builder の自由度は下がる**: power user 向けの低レベル入力を捨てる代わりに、一般ユーザー向けの一貫性を優先している。
