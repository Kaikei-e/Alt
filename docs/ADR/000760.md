# ADR-000760: /v1/recap/articles REST endpoint を Connect-RPC ListRecapArticles に移行して REST 面を縮小する

## Status

Accepted

## Date

2026-04-17

## Affected Services

- **alt-backend** — `proto/alt/recap/v2/recap.proto` に `ListRecapArticles` RPC と 5 message 追加、`app/connect/v2/recap/handler.go` に実装追加、`app/rest/recap_handlers.go` 削除、`app/rest/routes.go` から `registerRecapRoutes` 呼び出しを除去、`app/driver/contract/provider_test.go` の recap-worker stub を Connect-RPC に差し替え
- **recap-worker** — `recap-worker/recap-worker/src/clients/alt_backend.rs` の `fetch_page` を `POST /alt.recap.v2.RecapService/ListRecapArticles` (JSON body) に書き換え、serde に `rename_all = "camelCase"` を付与、`alt_backend_contract.rs` の Pact を Connect-RPC に再契約
- **alt-butterfly-facade / rag-orchestrator / alt-frontend-sv** — `buf generate` による proto stub 再生成差分のみ (実行時挙動の変更なし)

## Context

[[000759]] で alt-backend の mTLS listener を Connect-RPC + REST のハイブリッドにして `/v1/recap/articles` を `:9443` 経由で即時復活させたが、Cons に *"この REST 残存は別 ADR で Connect-RPC 化する fast-follow 前提"* と記録していた。[[PM-2026-031]] action item #6 では期限 2026-05-15 の "/v1/recap/articles を Connect-RPC に移行 (7-day recap と整合)" として残置。本 ADR でそれを前倒しで消化する。

移行の動機は 3 つ:

1. **API 面の一貫性**: 7-day / 3-day recap (`GetSevenDayRecap` / `GetThreeDayRecap`) は既に Connect-RPC で稼働しており、`/v1/recap/articles` の REST だけが残るのは読み手のメンタルモデルを乱す
2. **silent failure の構造的予防**: [[PM-2026-031]] 障害 B (4 日連続 404) は「REST ルートが mTLS listener にミスマッチ」が原因で、proto 契約による静的型固定があれば同種の事故が起きにくい
3. **REST 面縮小の土台**: 将来 `:9000` を internal-only に絞る別 ADR への道を開く。recap-worker だけが `/v1/recap/articles` を叩いていた事実は grep で確認済みで、外部 consumer は存在しない

`recap-worker` は既に `reqwest` ベースで REST を叩いていたため、`tonic` / `prost` を新規導入せず、Connect-RPC unary の JSON wire format (HTTP POST + JSON body) を reqwest + serde でそのまま話させる形にできた。これにより Rust 側の build.rs 追加や proto stub 生成パイプラインを増やさずに済む。

## Decision

### 1. `RecapService.ListRecapArticles` RPC を追加

`proto/alt/recap/v2/recap.proto` の `RecapService` に `rpc ListRecapArticles(ListRecapArticlesRequest) returns (ListRecapArticlesResponse);` を追加し、以下 5 message を新設。フィールド名とセマンティクスは旧 REST の `RecapArticlesResponse` / `RecapArticleResponse` と 1:1 対応させた:

- `ListRecapArticlesRequest` — `from` / `to` (RFC3339), `optional int32 page`, `optional int32 page_size`, `repeated string fields`, `optional string lang_hint`
- `ListRecapArticlesResponse` — `range` / `total` / `page` / `page_size` / `has_more` / `repeated articles`
- `RecapArticleRange` — `from` / `to`
- `RecapArticleItem` — `article_id` / `title?` / `fulltext` / `published_at?` / `source_url?` / `lang_hint?` / `repeated tags`
- `RecapArticleTag` — `label` / `confidence?` / `source?` / `updated_at?`

`cd proto && buf generate` で Go / TypeScript / alt-butterfly-facade / rag-orchestrator の stub を再生成。

### 2. Go handler は既存 `RecapArticlesUsecase` を再利用

`alt-backend/app/connect/v2/recap/handler.go` に `ListRecapArticles` メソッドを追加。旧 REST の `handleRecapArticles` (`app/rest/recap_handlers.go:29-116`) からバリデーション + usecase 呼び出しロジックを移植し、proto → `domain.RecapArticle` の mapping を `recapArticlesToProto` として実装。`domain.RecapArticlesPage` / `usecase/recap_articles_usecase.Input` は無修正で再利用できたため、port / gateway / driver に変更はない。

認証方針は旧 REST と同じ — **`middleware.GetUserContext(ctx)` チェックは入れない**。recap-worker は service-to-service 呼び出しで、認証境界は mTLS peer identity (`:9443` 側のみ公開)。end-user 認証が必要な `GetSevenDayRecap` / `GetThreeDayRecap` 等とは設計上別カテゴリ。

`Handler` 構造体に `articlesUsecase RecapArticlesUsecaseInterface` フィールドを追加し、`NewHandler` シグネチャを `(recapUsecase, articlesUsecase, clusterDraftLoader, logger)` に拡張。呼び出し側は `app/connect/v2/server.go` 1 箇所だけなので `container.RecapArticlesUsecase` を差し込む形で対応。

### 3. recap-worker Rust client を Connect-RPC wire format に書き換え

`recap-worker/recap-worker/src/clients/alt_backend.rs`:

- `fetch_page` は `client.get(url).query(&[...]).send()` → `client.post(url).header("Content-Type","application/json").json(&body).send()` に書き換え
- URL は `base_url.join("alt.recap.v2.RecapService/ListRecapArticles")` (先頭スラッシュなしの Url::join で既存 mTLS base URL にぶら下がる)
- request body は新設 `ListRecapArticlesRequest` struct (`#[derive(Serialize)]` + `#[serde(rename_all = "camelCase")]`) で `page_size` → `pageSize` に変換
- response 側は既存の `RecapArticlesResponse` / `AltBackendArticle` / `AltBackendTag` に `#[serde(rename_all = "camelCase")]` を付与し、protojson が出す `pageSize` / `hasMore` / `articleId` / `publishedAt` / `sourceUrl` / `langHint` を直接読む
- mTLS ハンドシェイクは既存の `mtls::build_mtls_client` 経由で構築済みの `reqwest::Client` に依存しているため、endpoint 変更による影響なし

### 4. REST handler と rate limiter を即時撤廃

- `alt-backend/app/rest/recap_handlers.go` と `recap_handlers_test.go` を削除
- `app/rest/routes.go:127` の `registerRecapRoutes(v1, container, cfg)` 呼び出しを削除
- rate limiter (`recapRateLimiter` / token bucket 8 burst + 4 rps) は新 Connect-RPC 経路には**移植しない**。mTLS peer identity で recap-worker 単一 consumer に絞られる前提のため、過剰防衛と判断

### 5. Pact を REST → Connect-RPC に再契約

- consumer (`recap-worker/recap-worker/src/clients/alt_backend_contract.rs`): `POST /alt.recap.v2.RecapService/ListRecapArticles` + JSON body + camelCase response を期待する内容に書き換え。`tags` フィールドは旧 pact に含まれていたが、実装側 `RecapArticleResponse` struct に元々 `tags` フィールドが存在しなかった (JSON 出力されていなかった) ため、新 pact からも削除し整合性を取った
- provider (`alt-backend/app/driver/contract/provider_test.go`): `startStubServer` の `/v1/recap/articles` mock を `/alt.recap.v2.RecapService/ListRecapArticles` に差し替え。`recapArticleResponse` / `recapArticlesResponse` Go struct の JSON タグを camelCase 化 (`articleId` / `pageSize` / `hasMore`)

### 却下した代替案

1. **tonic / prost で Rust 側もフル gRPC 化** — build.rs + `protoc-gen-rust` 導入が必要で、Cargo.toml・ビルドパイプラインが肥大化。Connect-RPC JSON wire format は reqwest + serde で完結するので不採用
2. **REST handler 併存 (deprecated として段階移行)** — single-repo / single-consumer で split deploy リスクなし。deprecated 期間を置かずに一発で消すのが簡潔
3. **rate limiter を Connect-RPC interceptor (`connectRateLimitInterceptor`) で再実装** — mTLS peer identity + 単一 consumer の前提で過剰防衛。将来マルチ consumer になった時点で再評価するフックを残して今は落とす
4. **RPC 名 `GetArticlesInWindow` / `FetchRecapArticles`** — 既存 `ListUnsummarizedArticles` の `List<Resource>` convention に揃えた

## Consequences

### Pros

- **API 面の一貫性**: 7-day / 3-day / Evening Pulse / SearchRecapsByTag / ListRecapArticles が全て `RecapService` Connect-RPC 経由になり、recap 系の外部 API 表面が 1 つに統一された
- **silent failure の構造的予防**: [[PM-2026-031]] 障害 B が再発する余地がなくなった。Connect-RPC handler は mux 経由で proto 契約に紐づくため、mTLS listener にルートが届かない事象は proto 定義不整合としてビルド時に検出される
- **REST 面の縮小**: alt-backend `:9000` から `/v1/recap/articles` が消え、外部公開 REST ルートが 1 本減った。`:9000` を internal-only に絞る将来 ADR への準備が整う
- **Connect-RPC interceptor 活用**: cancel / auth / OTel interceptor が `SetupConnectHandlers` 経由で新 RPC にも一律適用され、tracing span やコンテキストキャンセルの挙動が他 RPC と揃う
- **proto stub を consumer 側に持たせなくて済む単純さ**: recap-worker は reqwest + serde のみで Connect-RPC unary を話せるため、`tonic` / `prost-build` の導入と build.rs 維持コストを回避

### Cons / Tradeoffs

- **rate limiter 撤廃のフォロー**: 単一 consumer + mTLS peer identity の前提が崩れた時点 (マルチ consumer 化、externalization) で再実装が必要。再評価フックを [[PM-2026-031]] action #6 の後継として運用メモに残す
- **recap-worker の一度限りの 404 窓**: deploy 中、alt-backend の REST route が消え、recap-worker の古いバイナリが残っている瞬間に 3days job が走ると 404 を食う。ただし c2quay の `docker compose up --wait --remove-orphans` と [[000759]] の compose-native cascade で alt-backend と recap-worker の recreate はほぼ同時に完了する。次の 3days job が走るまでの 1 時間程度で両者健全化されるので実害は無視できる
- **pact の JSON key 変更**: `pacts/recap-worker-alt-backend.json` が snake_case → camelCase に差し替わった。Broker に publish されている旧 version の pact と新 version は非互換なので、`can-i-deploy` は新 pact でのみ green。旧 recap-worker バイナリが動いている環境では新 alt-backend と通信できない (そもそも単一ホスト開発環境なので該当しない)
- **`tags` フィールドの削除**: 旧 pact に含まれていた `tags` は実装側で元々空だった (REST response struct に `Tags` フィールドなし)。consumer pact からも外した。Rust 側 `AltBackendArticle` は `#[serde(default)]` で `tags: Vec<_>` を維持しているため、将来 alt-backend が tags を返しても upstream 互換性は確保される
