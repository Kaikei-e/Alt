# ADR-000835: Recap Job の 2-bucket collapse を 3 層不整合として特定する

## Status

Accepted

## Date

2026-04-23

## Affected Services

- recap-subworker (classifier / embedding / thresholds を抱える主当事者)
- recap-worker (downstream: 7 genres しか persist できず UI が空化)
- alt-frontend-sv (UI で recap を表示。体感は「Recap がおかしい」)

## Context

2026-04-14 以降、recap 3-day window の job が 23 件連続で失敗し、4-22 朝にユーザーが気付いて 4-22 中に以下 3 commit を main に投入した。

- `3b791ada2 fix(recap-subworker): fail-closed on joblib artefact bind-mount empty directory`
- `43d301840 chore: update rust-bert cache volume mount to read-write`
- `d9d48a0d0 fix(recap-subworker): wire 30-genre thresholds into classifier factory`

4-23 時点で container は `sha-d636678` (上記 3 fix 込み) で稼働している。しかし 4-22 の完了 job 6 件は平均 5 genres/job にとどまり、ユーザが「14 日以降著しく減って成功率も低い。本質はジャンル分類の失敗」と指摘。4-23 20:43 に修正後 image 上で手動 kick したジョブ `f9e9b519-42ff-417d-aa72-fac3172651cf` を完走 (14 分) させ、`recap_jobs` / `recap_outputs` / `classification_job_queue` / コンテナ内 Python 経由で実測した。

### 実測結果 (本 ADR 執筆時の確証値)

| 測定項目 | 値 |
|---|---|
| job status | `completed` (850 秒) |
| genres persisted | **7** (baseline 29-39 の 17-24%) |
| 844 articles 全体の top_genre 分布 | `consumer_tech` 633 (75%), `politics_government` 211 (25%), その他 15 既知 class **0**, 13 未知 class **0** |
| classifier output cardinality | **17 class** (not 30) |
| default `genre_thresholds.json` の key 数 | **17** |
| 代替 `genre_thresholds_ja.json` の key 数 | **30** |
| `genre_classifier_model.joblib` の `classes_` | 17 要素の LogisticRegression |
| runtime library versions | transformers 5.5.1 / sentence-transformers 5.3.0 / sklearn 1.8.0 / numpy 2.4.3 |
| sklearn warning | `Trying to unpickle estimator LogisticRegression from version 1.7.2 when using version 1.8.0` |

つまり 844 articles 全部が 2 class に縮退し、残り 15 既知 class と 13 未知 class はそれぞれ 0 件だった。recap-worker 側の subgenre splitter が `consumer_tech` と `politics_government` をさらに `*_001..*_010` に分け、そのうち 2 件以上 article を保持できた 7 sub-genre だけが persist された。これが「14 日以降ジャンルが著しく減った」の物理的実態である。

### 3 層の不整合 (root cause structural view)

| 層 | 現状 | 問題 |
|---|---|---|
| 1. **classifier model** (`genre_classifier_model.joblib`) | 17-class LogisticRegression (2025-12 〜 2026-01 訓練、sklearn 1.7.2) | 13 genres を物理的に predict できない。30-class 化には再訓練が不可欠 |
| 2. **thresholds default file** (`genre_thresholds.json`) | 17 key | 30 key 版は `genre_thresholds_ja.json` に別置き。`d9d48a0d0` は factory 経路で `_ja.json` を使うよう修正したが、default file 自身は 17 key のまま残存しており呼び出し経路に依存する二重構成 |
| 3. **embedding / encoder** (`sentence-transformers 5.3.0` on `transformers 5.5.1`) | 2026-04-09 の security CVE 対応で v4.x → v5.5.1 に bump。[transformers 公式 issue の buffer junk bug](https://github.com/huggingface/transformers/issues/44534) に該当 | 推論 embedding が偏向し、LogisticRegression が prior 最頻の 2 class に縮退。17 class モデルの上位 2 prior だけが事実上 active |

### タイムライン

- 〜2026-04-09: 29-39 genres/job (baseline 健全)
- 2026-04-09 `a939123201c717d263948692dc6e49c4b8fe9a2e`: `transformers 5.5.1` + `sentence-transformers 5.3.0` へ security 由来 bump (Dependabot CVE 19 件同時対応の一部)
- 2026-04-14 〜 2026-04-21: classification 0 results 障害が顕在化、23 jobs 連続失敗
- 2026-04-22 (fix 3 件): joblib fail-closed, rust-bert `:ro` → `rw`, 30-genre thresholds wiring
- 2026-04-22 夜 〜 4-23 朝 (部分回復): 6 jobs completed だが平均 5 genres/job
- 2026-04-23 20:43: 修正後 image 上で手動 kick → 7 genres / 2-bucket collapse 継続を DB で実証

### 過去の近接事象との関係

`[[PM-2026-038]]` は本 collapse と同型の症状 (30 → 2 bucket) を rust-bert `:ro` bind 起因として記録している。今回の調査は、同型の症状が **別の root cause (transformers v5 + 17-class model + thresholds default file)** にも由来することを示した。したがって症状マッチだけで原因同定しない姿勢が必要。

## Decision

本 ADR は **発見と方針の記録** に限定し、実装変更は別 ADR / 別 commit で扱う。採択する修正順序は以下の 3 段階とする。

1. **即時 — `transformers` v4.x pin 戻しで root cause を確証する**
   - `recap-subworker/pyproject.toml` で `transformers>=4.45,<5` と `sentence-transformers<5.2` に戻す
   - `uv.lock` を refresh し image を rebuild
   - 同じ job を再度 kick し、top_genre 分布が 10+ class に戻るかを DB で確認
   - 戻れば embedding 層が犯人で確定、戻らなければ 17-class model 側が支配的で確定
   - security CVE は別対応 (patched 4.x 系、または trust-remote-code 無効化) で埋める

2. **中期 — 30-class classifier を再訓練し thresholds default file を統一する**
   - `recap_subworker/infra/classifier/train.py` と `learning_machine/scripts/` 経路で 30-class の LogisticRegression を再訓練 (sklearn 1.8.0 上)
   - `genre_classifier_model.joblib` / `feature_scaler.joblib` / `tfidf_vectorizer.joblib` を更新
   - `genre_thresholds.json` を 30-key 版に完全差し替え (`_ja.json` との二重構成を解消)
   - host bind artefact の populate 手順を runbook に明文化 (`[[PM-2026-036]]` AI #5 と合流)

3. **恒久 — fail-closed observability の追加**
   - 起動時 assert: `classifier.classes_` の長さが env で指定した genre set より少なければ `ConfigValidationError` を raise (Pydantic validator 拡張)
   - job 完了時に `genres_persisted < env(RECAP_GENRE_BASELINE_3DAY=15)` を満たすと `recap-evaluator` が `degraded` flag を付ける (計画 `recap-job-adr-tdd-workflow-plan-context-enumerated-hopper` WS-4 と同期)
   - sklearn version mismatch (学習時 vs 推論時) を起動時 warn より厳しく handle する (`InconsistentVersionWarning` を error 化 + artefact 再訓練要求)
   - classifier 実出力 distribution を Prometheus histogram で常時 exposure (`c723cb122` の proba histogram を完成させる形)

### 検討した代替案と却下理由

- **transformers を最新 v5.6+ に上げて buffer bug 修正版を試す**: 修正 commit と release timing が不透明で、別の regression を呼び込むリスク。決定打にならない。却下、ただし将来選択肢として残す。
- **embedding encoder を別モデル (OSS BGE / E5 等) に差し替える**: 効果は見込めるが、影響範囲が広く即時判定手段にならない。中期の選択肢として保留。
- **30-class threshold を先に deploy する (model 17-class のまま)**: threshold を変えても 17-class model が 13 class を predict できない以上、13 genre は永久に 0 件のまま。無意味と判断、却下。
- **recap job 自体を一時停止して UI をメンテ表示にする**: ユーザ体験は短期には悪くなるが、本 ADR の範囲ではオプションとして保持。修正が長引く場合は別 ADR で再提起する。

## Consequences

### Pros

- 2-bucket collapse の実測根拠と 3 層同時不整合の構造が記録され、今後の「似た症状」に対して **症状一致 = 原因一致** と誤診する余地が減る
- 即時・中期・恒久の 3 段階が順序付けされたので、レビュー時に「どこまでやるべきか」の議論が不要
- 実装変更ゼロの blameless 記録として、fix を担当した 4-22 の 3 commit (それ自体は正しい) と今回の追加判断を分離でき、blame free で議論できる
- 起動時 assert と degraded flag を入れることで `[[PM-2026-036]]` `[[PM-2026-037]]` `[[PM-2026-038]]` の共通教訓 (silent fallback を禁ず) をさらに具体化できる

### Cons / Tradeoffs

- 実装は 3 段階とも別 commit / 別 deploy になるため、最終的に baseline (29-39 genres/job) に戻るまで数日かかる想定
- transformers v4 pin 戻しは security CVE の再発リスクを抱える。patched 4.x 系が存在しない場合は追加対応が要る
- 30-class classifier 再訓練は label 定義 / 訓練データ / calibration 手順がユーザ判断事項を含むため、自動化不能な手作業フェーズがある
- 起動時 assert を強める方向は、artefact 不備で container が CrashLoop に入るトレードオフを増やす。起動失敗時の復旧手順を runbook に必ず書く
- 本 ADR の「17-class model」判断は本日実測のみを根拠としており、過去に 30-class で稼働していた時期があった可能性はまだ完全排除されていない。1. 段階の確証で引き続き検証
