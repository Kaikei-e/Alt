# ADR-000611: recap パイプラインの EmbeddingService 初期化エラー可視化と HDBSCAN numpy 2.0 互換性修正を行う

## Status

Accepted

## Date

2026-03-29

## Affected Services

- recap-worker (Rust)
- recap-subworker (Python)

## Context

Recap パイプラインの生成数が **30 ジャンル中 5 ジャンル（software_dev: 602件, ai_data: 229件, cybersecurity: 156件, culture_arts, consumer_tech）** で欠損する事象が 3 日連続（2026-03-27〜29）で発生した。

### 問題 1: recap-worker の EmbeddingService 初期化失敗が不可視

`PipelineOrchestrator::new()` で `EmbeddingService::new()` の結果を `.ok()` で `Option` に変換しており、初期化失敗のエラーメッセージが完全に握り潰されていた。

```rust
// 修正前: エラーが消える
let embedding_service = EmbeddingService::new().ok().map(|s| Arc::new(s) as Arc<dyn Embedder>);
```

`EmbeddingService` は rust-bert の `AllMiniLmL12V2` モデルを CPU で動作させるもので、libtorch の初期化やモデルファイルのダウンロードに依存する。初期化が失敗すると `embedding_service = None` となり、大規模ジャンルをサブジャンルに分割する `subcluster_large_genres()` が機能しない。

ログには `"embedding service unavailable, cannot split large genre into subgenres"` が出力されるが、**なぜ** unavailable かは記録されておらず、原因特定に支障を来していた。

### 問題 2: recap-subworker の HDBSCAN 最適化で numpy 2.0 の TypeError

[[000563]] で Python 3.14 へのアップグレードに伴い numpy 1.x → 2.0 に更新された。numpy 2.0 では配列からスカラーへの暗黙的変換が厳密化され、`TypeError: only 0-dimensional arrays can be converted to Python scalars` が発生するようになった。

optuna の objective 関数（`clusterer.py`）の例外ハンドラが `(IndexError, ValueError, RuntimeError)` のみをキャッチしており、`TypeError` が漏れていた。結果、optuna Trial が未処理例外で失敗 → クラスタリング run 全体が `status: failed` となる。

影響を受けたクラスタリング run:
- run 9318（ai_data, 81秒後に失敗）
- run 9320（cybersecurity, 106秒後に失敗）
- run 9329（software_dev, 127秒後に失敗）
- run 9333（culture_arts, 154秒後に失敗）

## Decision

### 1. EmbeddingService 初期化エラーの可視化

`.ok()` を `match` 式に置換し、失敗時に `tracing::error!` でエラーチェーン全体を出力する:

```rust
match EmbeddingService::new() {
    Ok(s) => {
        tracing::info!("Embedding service initialized successfully");
        Some(Arc::new(s) as Arc<dyn Embedder>)
    }
    Err(e) => {
        tracing::error!(error = %e, error_chain = ?e,
            "Embedding service failed to initialize. \
             Large genres cannot be split into subgenres.");
        None
    }
}
```

初期化成功時にも INFO ログを出力し、起動ログから embedding 可用性を即座に判断可能にした。

### 2. HDBSCAN optuna objective の TypeError キャッチ追加

```python
# 修正前
except (IndexError, ValueError, RuntimeError) as e:

# 修正後
except (IndexError, ValueError, RuntimeError, TypeError) as e:
```

`TypeError` をキャッチすることで、numpy 2.0 の厳密化による変換エラーが optuna の Trial 失敗として処理され、`float('-inf')` を返して当該パラメータ領域を回避する。クラスタリング run 全体の失敗は防止される。

## Consequences

### Pros

- **EmbeddingService 障害の即座の原因特定**: libtorch ロード失敗、モデルダウンロードエラー、メモリ不足等の具体的なエラーがログに記録される
- **numpy 2.0 互換性確保**: HDBSCAN 最適化の Trial 失敗が正常にハンドルされ、クラスタリング run が完走する
- **Recap 品質の回復**: 5 ジャンルの欠損が解消され、`genres_stored` が 24/30 → 29-30/30 に改善見込み

### Cons / Tradeoffs

- **EmbeddingService の根本修正は未実施**: 初期化失敗の原因（libtorch 設定、モデルキャッシュ等）は再ビルド後のログで特定し、別途対応が必要
- **numpy 2.0 の暗黙的変換厳密化の影響範囲**: `TypeError` キャッチは対症療法。HDBSCAN/sklearn の numpy 2.0 完全互換はライブラリ側のアップデート待ち
