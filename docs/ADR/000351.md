# 000351: Tag Verse 改善 — サーバーサイド3Dレイアウト + HUD修正 + WebGPU化

## ADR's STATUS

Accepted

## CONTEXT

ADR-349 で初期実装した Tag Verse（タグの3D可視化）に以下の3つの問題があった:

1. **HUD が表示されない** — タグ球体をクリックしても HUD パネルが開かない。Threlte v8 では `interactivity()` プラグインを Canvas の子コンポーネント内で明示的に呼ばないと、3Dオブジェクトのポインターイベント（`onclick`, `onpointerenter` 等）が発火しない
2. **遠くからタグに色が付かない** — `FogExp2` density 0.008 + 低 emissive（0.1）+ 暗い基底色（`#1a3a5c`）が重なり、デフォルトカメラ位置ではほぼ全てのタグが黒く見えていた
3. **球面配置では関連性が表現できない** — フィボナッチ球面分布は均等配置であり、タグ間の意味的な関連性が空間配置に反映されていなかった

追加要件として、WebGPU Renderer への切り替え（WebGL フォールバック付き）も含めた。

## DECISION MAKING

### 1. サーバーサイド フォースグラフレイアウト

タグの3D座標をサーバー側で計算する方式を採用した。

**理由**: クライアント側での力学シミュレーションはフレーム落ちの原因になり、タグ数が多い場合（200件）にUXが悪化する。サーバー側で事前計算してキャッシュ可能な座標として返す方が合理的。

**アルゴリズム**:
- 反発力（クーロン力: `F = k / d²`）で全ノード間を離す
- 引力（バネ力: `F = -k * d * log₂(weight+1)`）で共起エッジのあるタグ対を近づける
- 中心への弱い引力で発散を防止
- 200回反復、温度冷却（simulated annealing）で収束
- 最終座標を `-50〜50` に正規化
- シード固定（42）で決定的な結果を保証

**共起データ**: `article_tags` テーブルの自己結合で、同一記事に付与された2タグの組み合わせを `shared_count >= 2` で抽出。`feed_tag_id` の大小比較で片方向のみ取得し重複を防止。上限2000ペア。

**Proto 拡張**: `TagCloudItem` に `position_x/y/z`（float）フィールドを追加。

| レイヤー | ファイル | 変更 |
|---------|---------|------|
| Proto | `proto/alt/articles/v2/articles.proto` | `position_x/y/z` フィールド追加 |
| Domain | `alt-backend/app/domain/tag_cloud.go` | `PositionX/Y/Z` + `TagCooccurrence` 構造体 |
| Driver | `alt-backend/app/driver/alt_db/fetch_tag_cooccurrence_driver.go` | 共起クエリ（新規） |
| Port | `alt-backend/app/port/fetch_tag_cloud_port/port.go` | `FetchTagCooccurrences` メソッド追加 |
| Gateway | `alt-backend/app/gateway/fetch_tag_cloud_gateway/gateway.go` | 共起取得デリゲーション追加 |
| Usecase | `alt-backend/app/usecase/fetch_tag_cloud_usecase/usecase.go` | Execute で共起取得 + レイアウト計算 |
| Usecase | `alt-backend/app/usecase/fetch_tag_cloud_usecase/layout.go` | フォースグラフアルゴリズム（新規） |
| Usecase | `alt-backend/app/usecase/fetch_tag_cloud_usecase/layout_test.go` | TDD テスト 6件（新規） |
| Handler | `alt-backend/app/connect/v2/articles/handler.go` | Proto マッピングに position 追加 |

### 2. interactivity() による HUD 修正

Threlte v8 の `interactivity()` は Canvas の子コンポーネント内で呼ぶ必要がある。既存の `TagVerseScene.svelte` は `<Canvas>` をラップするだけのコンポーネントだったため、3Dコンテンツを新規 `SceneContent.svelte` に抽出し、そこで `interactivity()` を呼ぶ構造に変更した。

```
TagVerseScene.svelte
  └─ <Canvas createRenderer={...}>
       └─ SceneContent.svelte  ← interactivity() をここで呼ぶ
            ├─ Camera + OrbitControls
            ├─ Lighting + Fog
            ├─ StarBackground
            ├─ TagAsteroid (×N)
            └─ PostProcessing
```

### 3. 色・フォグの視認性改善

| パラメータ | Before | After | 理由 |
|-----------|--------|-------|------|
| FogExp2 density | 0.008 | 0.003 | 遠方タグが黒く潰れるのを防止 |
| Ambient intensity | 0.25 | 0.5 | 基底照明を底上げ |
| emissive 最低値 | 0.1 | 0.4 | 低カウントタグでも自発光で視認可能に |
| colorLow | `#1a3a5c` | `#2a6a9c` | 暗すぎる基底色を明るく |
| カメラ初期距離 | `cloudRadius * 2` | `cloudRadius * 1.5` | タグに近づけて色を判別しやすく |

### 4. フィボナッチ球面 → サーバー座標（フォールバック付き）

サーバーが位置を返す場合は `tag.positionX/Y/Z` を使用し、返さない場合（全て `0,0,0`）は旧フィボナッチ球面分布にフォールバックする。これにより、バックエンド未デプロイの状態でもフロントエンドが動作する。

```typescript
const hasServerPositions = $derived(
  tags.some(t => t.positionX !== 0 || t.positionY !== 0 || t.positionZ !== 0)
);
```

### 5. WebGPU Renderer + TSL Bloom

**WebGPURenderer**: Three.js v0.183 の `WebGPURenderer` は WebGPU 非対応ブラウザで自動的に WebGL バックエンドにフォールバックする。Threlte v8 の `<Canvas>` が受け付ける `createRenderer` prop でカスタムレンダラーを注入。

**PostProcessing の TSL 化**: 旧 `EffectComposer` + `UnrealBloomPass`（WebGL 専用）から、TSL ベースの `RenderPipeline` + `pass()` + `bloom()` に書き換えた。これにより WebGPU レンダラーとの互換性を確保。

| ファイル | 変更 |
|---------|------|
| `TagVerseScene.svelte` | `WebGPURenderer` 設定 + `SceneContent` 分離 |
| `SceneContent.svelte` | `interactivity()` + シーンコンテンツ移動（新規） |
| `PostProcessing.svelte` | TSL ベース `RenderPipeline` + `bloom()` に書換え |

## RESULTS, EFFECTS

### PROS

- **HUD が正常に動作** — タグ球体クリックでサイドパネルが開き、記事一覧・Recap が閲覧可能に
- **視認性の大幅改善** — 遠方からでもタグの色とサイズが判別でき、Fog による奥行き感も維持
- **意味的な空間配置** — 共起関係の強いタグが近接配置され、タグ間の関連性を直感的に把握可能
- **WebGPU 対応** — 対応ブラウザではネイティブ WebGPU レンダリング、非対応では透過的に WebGL フォールバック
- **後方互換性** — フィボナッチ球面フォールバックにより、バックエンド未更新でもフロントエンドが動作
- **決定的レイアウト** — シード固定によりリクエストごとに同じ配置が返り、キャッシュ効率が高い

### CONS, TRADEOFF

- **共起クエリのコスト** — `article_tags` の自己結合は O(n²) に近く、タグ数やタグ付き記事数が大幅に増えた場合にレスポンスタイムが悪化する可能性がある。将来的にはマテリアライズドビューまたは定期バッチ計算の導入を検討
- **レイアウト計算の CPU コスト** — 200ノード × 200反復 × 全ペア反発力（O(n²)）はリクエスト毎に実行される。現状の200タグ上限では問題ないが、上限を増やす場合は Barnes-Hut 近似やサーバーサイドキャッシュが必要
- **WebGPU TSL エコシステムの未成熟** — Three.js の TSL（Three Shading Language）は experimental であり、API が変更される可能性がある。`bloom()` の import パスが `three/examples/jsm/tsl/display/BloomNode.js` と不安定な位置にある
- **エッジの非表示** — 共起関係は近接配置のみで表現し、エッジ（線）は描画しない。密度が高い領域ではどのタグ対が関連しているか不明瞭になる可能性があるが、視覚的なノイズを抑えるためこのトレードオフを受け入れた
