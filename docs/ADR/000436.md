# ADR-000436: Knowledge Home Lens の選択状態を URL と persisted active lens で同期する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-frontend-sv (TypeScript/Svelte) — Home route, lens selector, lens modal, empty state
- alt-backend (Go) — active lens 読み出しと clear API 契約

## Context

Phase 2 で server-side filtering を入れても、frontend が active lens を安定して再現できなければ UX は崩れる。特に以下のズレが起こりやすかった。

- reload や別タブ遷移で、さっき見ていた Lens と default view が入れ替わる
- 共有リンクで同じ Lens を再現できない
- Lens 条件で 0 件になったときに、単なる「記事がない」empty state と区別できない
- 既存 Lens schema に残る legacy field と、Phase 2 で実際に編集したい field が一致していない

Knowledge Home は探索導線の画面であり、状態再現性と解除導線が曖昧だと user は結果の意味を判断できない。したがって Lens の UX は「どの Lens が効いているか」「どう共有されるか」「どう解除するか」を同時に固定する必要があった。

## Decision

Knowledge Home Lens の frontend state は、**URL の `?lens=` を第一の共有状態とし、persisted active lens を初期復元に使う**。

具体的には以下を採用する。

- Home route は `?lens=<uuid>` を deep link の source of truth とする
- 初回表示では URL に lens があればそれを優先し、なければ server が返す `active_lens_id` を採用する
- Lens 選択時は URL query を更新し、同じ lens id を read API に必ず渡す
- `All` 選択時は `SelectLens` を clear で呼び、URL query も削除する
- empty state は「Knowledge Home が空」ではなく「この Lens 条件では 0 件」であることを明示し、解除 CTA を同画面に出す
- Lens 作成・編集 UI は Phase 2 では最小 modal とし、編集可能項目を `tag`、`source(feed)`、`recent window` に限定する
- `queryText`、`includeRecap`、`includePulse`、`sortMode` は hidden default として送信し、既存 schema との互換だけを維持する

代替案として、persisted active lens を唯一の source of truth にして URL を持たない案もあった。しかしこれでは共有リンクとブラウザ履歴の再現性が落ちるため採用しない。逆に URL だけを state source にして server 側永続化をやめる案もあったが、reload 後の復元と cross-session 継続性が弱くなるため採用しない。

## Consequences

### Pros

- 同じ `?lens=` を開けば同じ Home 集合を再現でき、共有とデバッグが容易になる
- reload 時は persisted active lens で前回状態へ戻れ、毎回選び直す必要がない
- Lens 条件による 0 件を通常 empty と区別でき、解除導線も明確になる
- Phase 2 の編集 UI を最小に絞ることで、read path 実装と UX 導線を同時に完結できる

### Cons / Tradeoffs

- URL state と persisted state の両方を扱うため、初期化順序と競合解決の実装が増える
- legacy field を hidden default で保持するため、schema 自体の単純化は後続フェーズに残る
- `All` が「選択解除」であることを API・URL・UI の三層で同期し続ける必要がある
- Lens modal は Phase 2 では最小 UI のため、高度な条件編集はまだ提供しない
