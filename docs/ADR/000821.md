# ADR-000821: Acolyte weekly_briefing に 7 日時間窓の既定値を導入し published_after に変換する

## Status

Accepted

## Date

2026-04-20

## Affected Services

- acolyte-orchestrator (Python)

## Context

Acolyte の `weekly_briefing` は文字通り直近 1 週間のできごとを要約する責務だが、report brief の `time_range` はこれまで nullable / free-text のまま運用されており、Gatherer 側でも日付境界の enforcement が無かった。

対象 report `eb67863f-f6f0-48a4-bd76-a9f1d9a79054`（topic: 「イラン情勢について、今日（2026年4月19日）を起点にして、直近の出来事をもとにイラン情勢をもとに分析して。」）の v10 scope_snapshot を確認すると `time_range: null` で、実際に cited された 6 件のユニーク source_id は 2026-02-20〜2026-03-26 の記事（「今日を起点に」という topic 文字列が伝わっていない）だった。eval ハーネス `db-replay` の `lang_mix_ratio={"ja": 1.0}` と合わせ、cited EN ratio 0 %、citation recency も 2 ヶ月級に乖離していた。

### 既存手段の限界

- brief の `time_range` は `report_briefs.time_range TEXT` カラムに free-text で保存される。これを Gatherer / Planner は `query_facet.render_query_string` 経由でクエリ文字列末尾に連結するだけで、日付境界にしない。
- search-indexer の `/v1/search` は `published_at` のフィルタを受け付けない（[[000822]] で別途対応）。
- [[000820]] で Planner を bilingual 化しても、retrieval に時間窓が効かない限り古い JP 記事と近々の EN 記事が同じ母集団で競合する。

## Decision

**weekly_briefing の time_range 既定値を ISO 8601 duration `P7D` に固定し、Acolyte Gatherer の retrieval が `published_after` 境界として search-indexer へ送出する**。

1. `ReportBrief.from_scope(scope, report_type)` の classmethod で、`report_type == "weekly_briefing"` かつ呼び出し側が time_range を未指定（`None` または空文字）のときだけ `P7D` を default 値として埋める。呼び出し側が明示的に指定した値は尊重する。
2. `EvidenceProviderPort.search_articles` のシグネチャに `published_after: datetime | None = None` と `published_before: datetime | None = None` を追加。
3. Gatherer に `_published_after_from_brief(brief)` ヘルパーを置き、`brief.time_range` を正規表現 `^P(?:(\d+)W)?(?:(\d+)D)?(?:T(?:(\d+)H)?)?$` で解析して `datetime.now(UTC) - duration` を返す。解釈できない値は `None` を返すので、invalid 値が検索を壊さない。
4. `SearchIndexerGateway.search_articles` は `published_after` / `published_before` が non-None のときのみ `isoformat()` を query string に載せる。`_search_articles_bounded` 経由で呼び出すため、古い EvidenceProvider 実装（kwarg 未知）は従来通り動く。

### 検討した代替案

| 案 | 評価 |
|----|------|
| A. brief を変更せず Gatherer 側で `report_type == "weekly_briefing"` なら無条件に `now - 7d` を計算 | × brief が source of truth でなくなる。再生成時に差異が出る。scope_snapshot と retrieval の一貫性が崩れる |
| B. Atlas migration で `report_briefs.time_range` に `NOT NULL DEFAULT 'P7D'` を付ける | × report_type 固有の既定値が DB 側に混入し、将来 `market_analysis` 等を足すときに複雑化する |
| **C. `ReportBrief.from_scope` で report_type 依存の既定値を注入、Gatherer が ISO 8601 duration をパース** | ○ brief が source of truth、既定値が domain 層に閉じる、invalid 値に対して明示的な fallback がある |

C を採用。

## Implementation

### 1. ReportBrief.from_scope

ファイル: `acolyte-orchestrator/acolyte/domain/brief.py`

```
_WEEKLY_BRIEFING_DEFAULT_WINDOW = "P7D"
...
time_range = scope.get("time_range") or None
if time_range is None and report_type == "weekly_briefing":
    time_range = _WEEKLY_BRIEFING_DEFAULT_WINDOW
```

他の `report_type`（`market_analysis`、`market_analysis_japan`、`trend_report`）は従来通り `None` を維持する。

### 2. EvidenceProviderPort

ファイル: `acolyte-orchestrator/acolyte/port/evidence_provider.py`

`search_articles` に `published_after`, `published_before` キーワード引数を追加。`datetime` は `TYPE_CHECKING` ガード内で import して実行時依存ゼロを維持。

### 3. Gatherer

ファイル: `acolyte-orchestrator/acolyte/usecase/graph/nodes/gatherer_node.py`

- `_DURATION_RE = re.compile(r"^P(?:(\d+)W)?(?:(\d+)D)?(?:T(?:(\d+)H)?)?$")` — 週 / 日 / 時間のみサポートする minimal subset。月 / 年 / 秒は意図的に除外する（ambiguity を避けるため）。
- `_parse_duration_days(value)` と `_published_after_from_brief(brief, *, now=None)` を新設。後者は `now` 引数を取ってテストでの時間固定を可能にする。
- `__call__` で call あたり 1 度 `published_after = _published_after_from_brief(brief)` を計算し、`_search_by_facets` / `_search_by_queries` の両方に渡す。
- 既存 EvidenceProvider 実装との互換性のため `_search_articles_bounded` ヘルパー経由で呼び出し、`published_after is None` のときだけ古いシグネチャ（`limit=...` のみ）で呼ぶ。

### 4. SearchIndexerGateway

ファイル: `acolyte-orchestrator/acolyte/gateway/search_indexer_gw.py`

```
params: dict[str, str | int] = {"q": query, "limit": limit}
if published_after is not None:
    params["published_after"] = published_after.isoformat()
if published_before is not None:
    params["published_before"] = published_before.isoformat()
```

### 5. テスト

- `tests/unit/test_brief_weekly_window.py` — weekly_briefing 既定 P7D、caller 指定値の尊重、非 weekly type は None 維持、空文字を欠落扱い
- `tests/unit/test_gatherer_date_window.py` — weekly_briefing 時に `published_after ≈ now - 7d`、time_range 未指定時は None
- `tests/unit/test_search_indexer_gw_date_window.py` — ISO 8601 形式でのパラメータ送出、未指定時の省略
- 既存 Gatherer スイート（`test_gatherer_hyde.py`, `test_gatherer_multi_query.py`, `test_gatherer_hybrid.py`）はシグネチャ互換レイヤーを経由するため regression なし

acolyte-orchestrator unit suite 631 件 green。

## Consequences

### Pros

- `weekly_briefing` が文字通り直近 1 週間のイベントに限定されるという意味論的保証が型レベルで担保される。DB レベルの scope_snapshot にも `"time_range": "P7D"` として残るので再生成結果の確定性が上がる。
- 既存 EvidenceProvider 実装に新 kwarg を強制しないシグネチャ互換レイヤーのおかげで、[[000822]] の Go 側対応と独立にロールアウト可能（Go 側が `published_after` を無視する状態でも機能的には壊れない）。
- 対象 report の再生成 v12 / v13 で、citation pool が recent EN 記事を取得できるようになった（ただし `published_at` 非搭載な既存 Meilisearch ドキュメントが残っているため、full 効果発現には [[000822]] とその後の reindex が必要）。

### Cons / Tradeoffs

- 既存の永続化済み `report_briefs.time_range` は migration では触らない。既存レポートを再生成する場合は DB を手動で `UPDATE report_briefs SET time_range = 'P7D' WHERE ...` するか、新規作成経由で取り込む必要がある。
- ISO 8601 duration パーサは minimal subset のみ（`P7D`, `P2W`, `PT24H` 等）。`P1M` / `P1Y` は意図的に未対応で、該当値が来たら silently `None` 扱い。将来 monthly 要約を足すなら対応が必要。
- Gatherer 側の `query_facet.render_query_string` は既存仕様で `time_range` をクエリ文字列末尾に連結するため、`P7D` が literal トークンとして BM25 に渡る（例: `"Tensions between Iran P7D"`）。短期的には無害（ノイズトークン扱い）だが、将来 render_query_string 側で ISO 8601 duration を除外するフォローが望ましい。
