# ADR-000669: Gatherer に multi-query RRF fusion を導入し hybrid retrieval の基盤を構築する

## Status

Accepted

## Date

2026-04-10

## Affected Services

- acolyte-orchestrator (Python 3.14+)
- search-indexer (Go 1.26+)

## Context

exec1.md Issue 7 の実装。Issues 1-6（[[000659]] SectionPlannerNode 追加、[[000661]] section_citations 永続化、[[000662]] Conclusion 分離、[[000664]] ES 再合成、[[000666]] section contract 付き半決定論 Planner、[[000667]] section-aware query decomposition）により composition 層と retrieval 構造化の基盤は整ったが、retrieval 自体は **単一の Meilisearch BM25 検索** に依存しており、以下の課題が残っていた。

1. **単一ランキングシグナル** — Meilisearch の BM25 + proximity 検索のみ。クエリ語句が一致しない関連記事が取りこぼされる
2. **スコア情報の欠落** — search-indexer REST API が `_rankingScore` を返していない。`ArticleHit.score` は常に 0.0 で、ランキング品質の評価・fusion の両方が不可能
3. **fusion 抽象化の不在** — 将来 semantic 検索を追加する際に、既存の lexical 結果と統合する仕組みがない

### 設計判断の分岐: Meilisearch hybrid search vs multi-query RRF

exec1.md は「BM25 と semantic retrieval の併用、RRF fusion、後で CC に移行可能な設計」を求めている。Meilisearch v1.27.0 は embedder 設定（Ollama 経由）で native hybrid search をサポートするが、以下の制約から **Phase A（今回）は multi-query RRF、Phase B（将来）で Meilisearch hybrid** の 2 段構成を採用した。

- **VRAM 競合リスク** — RTX 4060 8GB で Gemma4 26B (Q4) が ~6GB 使用中。embedding モデル（`nomic-embed-text` ~270MB）との並行ロードは `OLLAMA_MAX_LOADED_MODELS=2` 必須だが、未検証のまま本番導入はリスクが高い
- **resolve 方針** — 「Gemma4 を直そうとしすぎないこと。長い1発JSONをやめること。」に準じ、infra 変更を最小化して retrieval 品質改善を先に取る
- **即効性** — `_rankingScore` 返却は 5 行の Go 変更で済み、multi-query variant 生成は純 Python で完結。cross-service 変更が最小限

### Web Research 結果

- Ollama #15260（OPEN）: `think=false` + `format` で structured output が壊れる → LLM スキーマ拡張は引き続き控える
- Meilisearch v1.27.0: `ShowRankingScore: true` で BM25 スコア返却可能。hybrid search は embedder 設定が必要
- Bruch et al. (2023): Convex Combination は RRF を上回りやすいが、少数の relevance judgments が必要 → Phase A は RRF が妥当

## Decision

**Phase A として search-indexer に `_rankingScore` 返却を追加し、acolyte-orchestrator に FusionStrategy Protocol + RRFFusion + multi-query variant 生成を導入する。**

### 1. search-indexer: `_rankingScore` 返却

全検索メソッド（`Search`, `SearchWithFilters`, `SearchByUserID`, `SearchByUserIDWithPagination`）に `ShowRankingScore: true` を設定し、Meilisearch が返す `_rankingScore`（0.0-1.0）を抽出する。

変更チェーン: `MeilisearchDriver`（`getFloat64` ヘルパー追加） → `SearchDocumentDriver.Score` → `SearchEngineGateway.convertDocs` → `domain.SearchDocument.Score` → REST handler `SearchArticlesHit.Score` → JSON `"score"` フィールド。

### 2. acolyte domain: FusionStrategy Protocol + RRFFusion

**`acolyte/domain/fusion.py`**（新規）:
- `ScoredHit`: fusion 専用の frozen dataclass（`article_id`, `title`, `tags`, `score`, `source`）
- `FusionStrategy`: Protocol（`fuse(ranked_lists: list[list[ScoredHit]]) -> list[ScoredHit]`）
- `RRFFusion(k=60)`: Reciprocal Rank Fusion 実装。`RRF(d) = Σ 1/(k + rank)`。同スコア時は `max_score` で tie-break

`ScoredHit` を `ArticleHit` とは独立にした理由: fusion は retrieval の後処理であり、Port 層（`ArticleHit`）とは関心が異なる。`source` ラベル（"primary", "broad", "narrow"）は fusion 固有の概念。

### 3. acolyte domain: 決定論的 query variant 生成

**`acolyte/domain/query_variant.py`**（新規）:
- `generate_query_variants(facet, topic, brief) -> list[tuple[str, str]]`
- 1 つの QueryFacet から最大 3 variant を生成:
  - **primary**: `render_query_string(facet)` — ADR-667 の既存関数を再利用
  - **broad**: `topic + entities[:3]` — brief.entities があるときのみ
  - **narrow**: `must_have_terms` のみ — 2 語以上のときのみ
- LLM 不使用、完全決定論的

### 4. GathererNode: multi-query + RRF fusion

`_search_by_facets()` を拡張:
1. 各 facet に対して `generate_query_variants()` で variant 生成
2. variant ごとに `search_articles(query, limit=10)` 呼び出し
3. 結果を `ScoredHit` リストに変換
4. `self._fusion.fuse(ranked_lists)` で RRF fusion
5. fused results → evidence_map に section_key タグ付け

**Degrade policy**（resolve 第 5 項準拠）:
- variant 検索失敗 → 成功した variant のみで fusion、warning ログ
- 全 variant 失敗 → 空結果、warning ログ
- fusion strategy 未設定 → `RRFFusion(k=60)` デフォルト

### 5. DI wiring

- `GathererNode.__init__` に `fusion: FusionStrategy | None = None` 追加
- `build_report_graph()` に `fusion` パラメータ追加
- `main.py` で `RRFFusion(k=60)` インスタンス化して注入

### 6. State 拡張

`ReportGenerationState` に `retrieval_debug: dict` 追加（`TypedDict(total=False)` なので下流影響なし）。将来の checkpoint 時にデバッグ情報を保持する目的。

## Consequences

### Pros

- **retrieval 品質向上** — 同一 facet から 3 variant（primary/broad/narrow）の結果を RRF fusion。BM25 スコアも利用可能になり、ランキングの精度が向上
- **LLM 負荷ゼロ増加** — variant 生成・fusion ともに純 Python。Gemma4 structured output の不安定性（Ollama #15260）を完全に回避
- **fusion 戦略の差し替え可能** — `FusionStrategy` Protocol により、将来 CC (Convex Combination) やカスタム戦略への移行が 1 行の設定変更で可能
- **graceful degrade** — variant 失敗 → primary only、全失敗 → 空結果。resolve 第 5 項準拠
- **後方互換** — `query_facets` がない outline では既存の `search_queries` パスにフォールバック。`fusion=None` でも `RRFFusion` がデフォルト適用
- **Clean Architecture 準拠** — `ScoredHit`/`FusionStrategy` はドメイン層。Gateway 層は score 伝播のみ。Port 不変
- **Phase B への明確な拡張パス** — Meilisearch embedder 設定 + `search_articles_semantic()` 追加で true hybrid retrieval に進化可能

### Cons / Tradeoffs

- **API 呼び出し増加** — facet あたり最大 3 variant × search API call。ただし `limit=10` に抑制しており、Meilisearch は低レイテンシ
- **true semantic search は未導入** — multi-query variant は BM25 の多角的利用であり、semantic embedding ではない。Phase B で対応
- **CC fusion 未実装** — RRF のみ。CC は relevance judgments の蓄積後に導入
- **CDC 契約変更** — search-indexer レスポンスに `score` フィールド追加。既存 consumer は `score` を無視するため後方互換だが、pact 更新が必要
- **retrieval_debug は現時点で未消費** — State に追加のみ。checkpoint 導入（resolve 第 4 項）後に活用予定
