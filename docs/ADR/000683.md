# ADR-000683: Gemma4 gateway を structured/longform の 2 プロファイルに分離する

## Status

Accepted

## Date

2026-04-10

## Affected Services

- acolyte-orchestrator (Python / LangGraph)

## Context

Acolyte パイプラインでは Gemma4 26B を唯一の LLM として使用しているが、ノードごとに求められる出力形式が大きく異なる。Planner・QuoteSelector・FactNormalizer・Critic は JSON structured output を必要とし、Writer は自由形式の日本語長文を生成する。

Ollama の公式ドキュメントは structured output に対して `/api/chat` + `format` パラメータ + schema-in-prompt + `temperature=0` の組み合わせを推奨している。一方、Gemma4 には `think=false` を設定すると `format` constraint が無視されるバグ（ollama/ollama#15260）が存在し、structured output と thinking mode を同時に制御することができない。

これまで全ノードが同じエンドポイント・同じパラメータで Ollama を呼び出していたため、structured タスクの成功率が低く（v5: QuoteSelector 0/10, FactNormalizer 3/30）、一方で Writer にとっては不要な制約がかかっていた。

## Decision

gateway レベルで `LLMMode.STRUCTURED` と `LLMMode.LONGFORM` の 2 プロファイルを導入し、タスク特性に応じたエンドポイント・パラメータを自動選択する。

### Port 層: `LLMMode` enum

`port/llm_provider.py` に `LLMMode(Enum)` を定義。`STRUCTURED` / `LONGFORM` の 2 値で、各ノードが呼び出し時に `mode=LLMMode.STRUCTURED` のように指定する。

### Gateway 層: エンドポイント分離

`gateway/ollama_gw.py` の `OllamaGateway` が mode に基づいてルーティングする:

| mode | endpoint | temperature | think | format |
|------|----------|-------------|-------|--------|
| STRUCTURED | `/api/chat` | 0.0 | 送信しない (Gemma4 #15260 回避) | JSON schema |
| LONGFORM | `/api/generate` | 0.7 | 呼び出し元が指定可 | なし |

`_BASE_OPTIONS`（`num_batch=1024`, `num_keep=-1`, `num_ctx=12288`, `stop=["<end_of_turn>"]`）は全リクエストで統一し、Ollama のモデルリロードを防止する。

### Settings 層: per-mode デフォルト

`config/settings.py` に mode ごとのデフォルト値を追加:

- `structured_temperature: float = 0.0`
- `structured_num_predict: int = 1024`
- `longform_temperature: float = 0.7`
- `longform_num_predict: int = 4000`

明示的な kwargs 指定がある場合は mode デフォルトを上書きできる。

### ノード層: 全ノードが適切な mode を使用

- STRUCTURED: PlannerNode, QuoteSelectorNode, FactNormalizerNode, CuratorNode, SectionPlannerNode, CriticNode
- LONGFORM: WriterNode

### テスト

`tests/unit/test_ollama_gw.py` に mode routing テストを 5 件追加:
- STRUCTURED → `/api/chat` にルーティングされること
- LONGFORM → `/api/generate` にルーティングされること
- 明示的 kwargs が mode デフォルトを上書きすること
- STRUCTURED 時に think パラメータが送信されないこと
- LONGFORM 時に mode デフォルト temperature が適用されること

## Consequences

### Pros

- structured タスクと longform タスクで最適なパラメータが自動適用される
- Gemma4 #15260 の影響を structured パスで回避（think パラメータ非送信）
- `_BASE_OPTIONS` の統一により Ollama モデルリロード問題を防止
- 各ノードは `mode=` を指定するだけで、エンドポイントやパラメータの詳細を意識しない

### Cons / Tradeoffs

- `LLMMode` が 2 値固定のため、将来 thinking-only mode などが必要な場合は enum 拡張が必要
- `/api/chat` では thinking overhead が発生する（Gemma4 はデフォルトで thinking 有効）が、structured output の正確性を優先して受容する
- `think=false` が Ollama 側で修正された場合、structured パスで明示的に `think=false` を送信する最適化の余地がある
