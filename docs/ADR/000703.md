# ADR-000703: search-indexer の restart policy と alt-backend の depends_on を明示化し Reference Desk の silent degradation を防ぐ

## Status

Accepted

## Date

2026-04-13

## Affected Services

- alt-backend (Go) — Global Search usecase / search-indexer Connect-RPC クライアントの呼び出し元
- search-indexer (Go) — Meilisearch 検索 API を Connect-RPC で提供するサービス

## Context

Reference Desk (`alt-frontend-sv` の `/search` ページ) が継続的に以下の degradation バナーを表示する状態となっていた。

> Some sections are unavailable: articles, recaps. Results may be incomplete.

`global_search_usecase.Execute()` は articles / recaps / tags の 3 セクションを並列に検索し、各セクションが 3 秒以内に応答しない、またはエラーを返した場合にセクション名を `degradedSections` に追加する graceful degradation パターンを採用している（[[000625]] Global Federated Search 実装 を参照）。  
tags セクションは PostgreSQL を直接読むため正常に動作しており、articles / recaps のみが degraded となるのは Meilisearch 経由のパス、すなわち search-indexer への到達経路に問題があることを示唆していた。

alt-backend のログを確認したところ、次のエラーが繰り返し記録されていた。

```
level:ERROR msg:"failed to search articles for global search"
error:"unavailable: dial tcp: lookup search-indexer on 127.0.0.11:53: no such host"
```

`127.0.0.11` は Docker embedded DNS であり、「no such host」は DNS 解決自体に失敗していること、つまり `alt-network` に `search-indexer` という名前のコンテナが存在しないことを意味する。connection refused（コンテナは存在するが listen していない）や timeout（コンテナはあるが応答が遅い）ではなく、**コンテナそのものが不在** という状態だった。

`docker compose ps` で確認すると、`search-indexer-logs`（ログフォワーダサイドカー）は running 状態だが、肝心の `search-indexer` サービスコンテナがプロセス一覧から消えていた。ログフォワーダは起動時に `Discovery error: Container not found for service: search-indexer` を出しており、対象コンテナが約 36 時間前から存在しないことが判明した。

調査過程で、コンテナが復帰しない構造的要因が 2 点明らかになった。

1. **search-indexer に restart policy が未設定**  
   `compose/workers.yaml` の search-indexer 定義には `restart: always` が無かった。alt-backend, mq-hub, その他の長時間稼働サービスは `restart: always` を持つが、search-indexer だけ欠落していた。restart policy が無いコンテナは一度停止・除去されると自動復帰しない。

2. **alt-backend の depends_on が search-indexer を含まない**  
   `compose/core.yaml` の alt-backend `depends_on` には mq-hub / migrate / pgbouncer / knowledge-sovereign が宣言されていたが、search-indexer は含まれていなかった。このため alt-backend は search-indexer 不在でも起動でき、Reference Desk の一部機能を silently 劣化させたままトラフィックを受け付けていた。

Reference Desk は [[000691]] Alt-Paper redesign で Newspaper Style の主要エンドポイントとなっており、degradation バナーはユーザーに対して機能の一部が欠落していることを明示するフェールセーフだが、その状態が恒常化していた。

## Decision

以下 2 点を compose 設定に追加することで、search-indexer が silently 失われた場合でも自動復帰し、かつ alt-backend が不完全な状態で起動しないようにする。

1. `compose/workers.yaml` の `search-indexer` に `restart: always` を追加する。  
2. `compose/core.yaml` の `alt-backend` `depends_on` に `search-indexer: condition: service_healthy` を追加する。

いずれも infra / orchestration レイヤの変更であり、Go コード・proto 定義・Connect-RPC 契約には一切手を入れない。`global_search_usecase` の 3 秒タイムアウトおよび `degradedSections` による graceful degradation はそのまま維持する（根本原因をマスクせず、今後の異常検知シグナルとして残す意図）。

即時対応として `docker compose up --build -d search-indexer` で該当コンテナのみ再ビルド・起動し、`restart: always` と `depends_on` は恒久対策として設定ファイルに反映した。[No compose down] 方針に従い、他サービスは停止していない。

### 検討した代替案

- **`sectionTimeout` を延長する**: 現象が timeout ではなく DNS 失敗であり、延長しても改善しない。また真因をマスクするため却下。
- **gateway 層にリトライを追加する**: DNS 解決失敗は構造的な不在状態であり、リトライしても解決しない。リトライ自体が原因究明を遅らせるため却下。
- **alt-backend のヘルスチェックに search-indexer 依存を組み込む**: alt-backend 全体が unhealthy となり Reference Desk 以外の機能（フィード閲覧等）まで停止するため過剰。depends_on による start-order 制御に留めた。

## Consequences

### Pros

- search-indexer が何らかの理由で消えても Docker Engine が自動で再作成するようになる（`restart: always`）。
- `docker compose up alt-backend` を単独で叩いても search-indexer が healthy になるまで alt-backend が起動を待つため、Reference Desk が silently 劣化した状態でトラフィックを受け付けることがなくなる。
- 既存の graceful degradation ロジック（`degradedSections`）は残るため、将来 search-indexer が真に slow / error になった場合でも UI は正しく劣化を可視化する。

### Cons / Tradeoffs

- alt-backend の起動前提に search-indexer が追加されるため、search-indexer ビルド失敗時は alt-backend も起動できなくなる。これは意図した結果（fail fast）だが、開発者が部分的スタックで作業する際には `docker compose up --no-deps alt-backend` 等の明示が必要になる。
- `restart: always` は crash loop を隠蔽する副作用がある。今回の修正に合わせて本来は search-indexer の crash 原因をメトリクスとして上位レイヤで観測したいが、これは本 ADR の範囲外（Out of scope のアラート整備タスクに譲る）。
- コンテナが外部から明示的に `docker rm` された場合、`restart: always` は効かない。これは Docker の仕様上避けられない制約。

## 実装の概要

### 変更差分

- `compose/workers.yaml`:
  ```yaml
    search-indexer:
      env_file:
        - ../.env
      build:
        context: ../search-indexer
        dockerfile: Dockerfile.search-indexer
      restart: always          # 追加
      depends_on:
        redis-streams:
          condition: service_healthy
  ```
- `compose/core.yaml`:
  ```yaml
      depends_on:
        mq-hub:
          condition: service_healthy
        migrate:
          condition: service_completed_successfully
        pgbouncer:
          condition: service_healthy
        knowledge-sovereign:
          condition: service_healthy
        search-indexer:        # 追加
          condition: service_healthy
  ```

### コンテナ再ビルドと動作確認

- `docker compose -f compose/compose.yaml -p alt up --build -d search-indexer` を実行。
- `docker compose ps search-indexer` で `State=running / Status=healthy` を確認。
- 修正後 2 分間の alt-backend ログで `no such host` / `article search failed` / `recap search failed` の出現件数が 0 件であることを確認。
- Reference Desk の `degradedSections` ログに articles / recaps が現れないことを確認。
