# ADR-000618: RAG ツールクライアントを BackendInternalService Connect-RPC に移行しサービス間認証を確立する

## Status

Accepted

## Date

2026-04-03

## Affected Services

- rag-orchestrator (Go) — Connect-RPC クライアント切り替え
- alt-backend (Go) — BackendInternalService に RPC 追加
- proto — `services/backend/v1/internal.proto` にメッセージ・RPC 追加

## Context

[[000617]] で導入した Agentic RAG の Tool-Use パイプラインでは、rag-orchestrator が alt-backend の `FetchTagCloud` と `FetchArticlesByTag` を Connect-RPC 経由で呼び出す設計とした。

初期実装では `ArticleServiceClient`（Public API）に接続したが、alt-backend の Connect-RPC ハンドラには 2 種類の認証インターセプターが適用されている:

| API 分類 | 認証ヘッダー | 検証方式 | 用途 |
|----------|-------------|---------|------|
| Public API (ArticleService 等) | `X-Alt-Backend-Token` | JWT（issuer/audience 検証） | BFF → alt-backend のユーザー認証 |
| Internal API (BackendInternalService) | `X-Service-Token` | shared secret 文字列比較 | サービス間通信 |

rag-orchestrator はサービス間通信であるため、JWT ユーザートークンを持たない。Public API に `X-Service-Token` を送信しても `AuthInterceptor` が先に JWT を要求するため、`unauthenticated: missing backend token` で拒否される。

### 検討した代替案

1. **Public API に ServiceAuth バイパスを追加**: AuthInterceptor に `X-Service-Token` 検出時のスキップロジックを追加する。認証レイヤーの複雑化を招き、セキュリティ境界が曖昧になる
2. **REST API 経由**: alt-backend の REST `/v1/articles/by-tag` を使用する。ただし `FetchTagCloud` には REST エンドポイントがなく、新規 REST ルート追加が必要。Connect-RPC への統一方針に逆行する
3. **BackendInternalService に RPC を追加**: 既存の Internal API フレームワーク（`X-Service-Token` 認証、pre-processor/search-indexer が既に利用）に RAG 用 RPC を追加する。認証モデルが確立済みで追加コストが最小

## Decision

**BackendInternalService に `FetchTagCloud` と `FetchArticlesByTag` RPC を追加し、rag-orchestrator のクライアントを `BackendInternalServiceClient` に切り替える。**

これにより、rag-orchestrator は pre-processor や search-indexer と同じ認証パターン（`X-Service-Token` shared secret）で alt-backend にアクセスする。

### Proto 変更

`services/backend/v1/internal.proto` の `BackendInternalService` に追加:

```protobuf
// RAG Tool Operations (ADR-000617: rag-orchestrator synthesis)
rpc FetchTagCloud(BackendInternalServiceFetchTagCloudRequest) returns (BackendInternalServiceFetchTagCloudResponse);
rpc FetchArticlesByTag(BackendInternalServiceFetchArticlesByTagRequest) returns (BackendInternalServiceFetchArticlesByTagResponse);
```

メッセージ型は BackendInternalService 専用に定義（buf lint の命名規則に準拠）。ArticleService の既存型は Public API 向けで 3D 座標等の不要フィールドを含むため、Internal API 用の簡易メッセージとした。

### alt-backend 側

- `connect/v2/internal/handler.go` に `FetchTagCloud` / `FetchArticlesByTag` メソッドを追加
- 既存の `FetchTagCloudUsecase` / `FetchArticlesByTagUsecase` を再利用（新規ビジネスロジック不要）
- `WithRAGToolPorts()` HandlerOption で DI 接続
- `server.go` で `WithRAGToolPorts` をハンドラ生成時に渡す

### rag-orchestrator 側

- `adapter/altdb/tag_cloud_client.go` — `InternalTagCloudClient`（`BackendInternalServiceClient` を使用）
- `adapter/altdb/articles_by_tag_client.go` — `InternalArticlesByTagClient`（同上）
- `di/container.go` — `ArticleServiceClient` → `BackendInternalServiceClient` に切り替え
- `serviceTokenInterceptor` が `X-Service-Token` ヘッダーを注入（変更なし）

### buf 生成設定

`buf.gen.backend-internal.yaml` に rag-orchestrator 用の出力先を追加。これにより `services/backend/v1` の proto が rag-orchestrator の `internal/gen/proto/` にも生成される。

## Consequences

### Pros

- **認証問題の完全解決**: `X-Service-Token` shared secret で認証が成立し、403/unauthenticated エラーが解消
- **既存パターンの踏襲**: pre-processor, search-indexer と同じ `BackendInternalServiceClient` + `X-Service-Token` パターン。新しい認証メカニズム不要
- **Public API の認証境界が維持**: ArticleService の JWT 認証に変更なし。セキュリティ境界が明確
- **E2E 検証済み**: `tag_cloud_explore` ツールが success、retrieval quality verdict が "good"、1346 文字の回答を生成

### Cons / Tradeoffs

- **Proto メッセージの重複**: ArticleService の `FetchTagCloudRequest/Response` と BackendInternalService の `BackendInternalServiceFetchTagCloudRequest/Response` が類似。ただし buf lint の命名規則に従うため避けられない
- **BackendInternalService の肥大化**: 元々 search-indexer/pre-processor 向けの Internal API に RAG 用 RPC が追加され、責務が広がる。将来的にサービス別の Internal API 分離を検討
- **search_recaps ツールは未接続**: recap-worker の SearchRecapsByTag は別サービス（RecapService）で提供されており、BackendInternalService には含まれない。今後の課題
