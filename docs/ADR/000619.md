# ADR-000619: rag-orchestrator の chunker に HTML sanitizer を統合し生 HTML チャンクによる RAG 検索品質劣化を修正する

## Status

Accepted

## Date

2026-04-03

## Affected Services

- rag-orchestrator (Go) — domain 層に sanitizer 追加、chunker V9 化、idempotency 拡張

## Context

Ask Augur に対して広範なトピックについて質問しても「提供されたコンテキストには情報が含まれていません」と返される事象が報告された。

調査の結果、rag-db にインデックスされたチャンクの内容が **生 HTML** であることが根本原因と判明した。

### 発生メカニズム

1. alt-db の `articles.content` カラムには、RSS フィードの本文と fulltext-fetch で取得したページ HTML の両方が格納される。全 43,658 記事のうち約 26% (11,222 件) が `<div>` 開始の生 HTML
2. rag-orchestrator の chunker (`domain/chunker.go`) は `\n\n` で段落分割するのみで、HTML タグの除去を行っていなかった
3. 結果として rag-db のチャンクには以下が混入:
   - HTML タグ (`<div>`, `<a href>`, `<figure>`, `<span>` 等)
   - サイトナビゲーション（「あわせて読みたい」「新着ニュース」等のサイドバーリンク）
   - 利用規約、購読案内等のボイラープレートテキスト
   - 無関係な記事の見出し（サイドバーの関連記事リスト）
4. **embedding 汚染**: ベクトルがナビゲーション/ボイラープレートを表現し、ユーザーのクエリ embedding とマッチしない
5. **BM25 希釈**: 検索キーワードが無関係なサイドバーリンク中に散在し、relevance スコアが低下
6. **LLM コンテキスト品質劣化**: 取得されたチャンクが HTML タグと無関係テキストで埋め尽くされ、LLM が正しく「情報なし」と判断

### 影響

- rag-db 全 34,069 ドキュメントのうち、HTML を含むチャンクが大量に存在
- 特に日本語のニュースサイト記事が深刻（ページ全体の HTML がそのまま格納）
- [[000617]] Tool-Use Agentic RAG の効果も、検索基盤の品質が低いため限定的

## Decision

**rag-orchestrator の domain 層に HTML sanitizer を追加し、chunker (V9) の前処理として統合する。既存記事は ChunkerVersion ベースの idempotency 拡張により backfill で再インデックスする。**

### 検討した代替案

| 案 | 評価 |
|---|---|
| alt-db の `content` カラムを事前にクリーニング | alt-db は記事の原本保存が責務であり、非可逆変換を適用すべきでない。また pre-processor の変更が必要で影響範囲が大きい |
| 正規表現によるタグ除去 | 壊れた HTML やネストされたタグに対して脆弱。`<script>` 内のテキストを誤って残すリスク |
| Bluemonday (allowlist ベース HTML sanitizer) | allowlist モデルは「残すタグを選ぶ」用途であり、本件の「全タグ除去 + ボイラープレート判定 + テキスト抽出」には合わない。ボイラープレート見出し内容の判定は Bluemonday では不可 |

### 実装の概要

#### 1. HTML Sanitizer (`domain/sanitizer.go`)

`golang.org/x/net/html` の DOM パーサーを使用してテキスト抽出を行う:

- DOM ツリーを走査し、テキストノードのみを抽出
- `<a>` のリンクテキストは保持、href URL は除去
- `<script>`, `<style>`, `<noscript>`, `<svg>`, `<iframe>` は子要素ごと除去
- `<figure>` サブツリーを除去（空の alt テキスト付き画像タグ）
- ブロック要素境界 (p, div, h1-h6, li, section, article 等) で改行を挿入
- HTML エンティティのデコード (`&amp;` → `&`)
- ボイラープレート見出し検出: h2/h3 のテキスト内容が「注目ワード」「あわせて読みたい」「深掘りコンテンツ」「新着ニュース」等に一致した場合、それ以降の兄弟要素をすべて除去
- 「受信契約」を含むテキストノードを除去
- 連続空行を正規化
- HTML タグを含まない plain text はそのまま通過（fast path）

#### 2. Chunker V9 統合 (`domain/chunker.go`)

- `Chunk(body)` の先頭で `SanitizeHTML(body)` を呼び出し
- `ChunkerVersionV9` を定義し、バージョンを V8 → V9 に更新
- chunker のインターフェース (`Chunker`) は変更なし

#### 3. Idempotency 拡張 (`usecase/index_article_usecase.go`)

既存の idempotency 判定:
```go
// Before: SourceHash + URL + Title の一致で skip
if latestVer.SourceHash == sourceHash && latestVer.URL == url && latestVer.Title == title {
    return nil
}
```

ChunkerVersion を追加:
```go
// After: ChunkerVersion も一致しなければ re-index
if latestVer.SourceHash == sourceHash &&
    latestVer.URL == url &&
    latestVer.Title == title &&
    latestVer.ChunkerVersion == string(u.chunker.Version()) {
    return nil
}
```

これにより、記事内容が同一でも chunker が更新された場合に再チャンク + 再 embedding が実行される。

### Clean Architecture レイヤー配置

| コンポーネント | 層 | 理由 |
|--------------|---|------|
| `SanitizeHTML()` | Domain | 純粋な変換関数、外部依存なし |
| `ChunkerVersionV9` | Domain | バージョン定数 |
| idempotency 拡張 | Usecase | ビジネスロジック（インデックス判定） |

### テスト

TDD (RED → GREEN) で実装。テストカバレッジ:

- `sanitizer_test.go`: 10 テストケース
  - 深くネストされた HTML からの本文抽出
  - リンクテキスト保持 + URL 除去
  - p タグの段落構造保持
  - plain text のパススルー
  - ボイラープレートセクション除去
  - 空文字列処理
  - script/style/noscript 除去
  - ブロック要素の改行生成
  - HTML エンティティデコード
  - 連続空行正規化
- `chunker_test.go`: V9 バージョン確認、HTML 入力テスト、V8→V9 差分検出テスト
- `index_article_usecase_test.go`: ChunkerVersion 変更時の re-index テスト、HTML body クリーンチャンクテスト

## Consequences

### Pros

- **RAG 検索品質の回復**: チャンクが純粋な記事テキストになり、embedding が記事内容を正確に表現する
- **BM25 検索精度の向上**: キーワードがノイズに埋もれず、relevance スコアが改善
- **LLM コンテキスト品質の向上**: HTML タグとボイラープレートが除去され、LLM が記事内容を正しく理解できる
- **既存パイプライン最小変更**: chunker のインターフェースは保持、sanitizer は内部前処理として統合
- **自動再インデックス**: ChunkerVersion ベースの idempotency により、backfill 実行で全ドキュメントが自動的に V9 で再チャンク + 再 embedding される
- **plain text 記事への影響なし**: HTML タグを含まない記事は fast path でそのまま通過

### Cons / Tradeoffs

- **全ドキュメント再 embedding のコスト**: ChunkerVersion V9 への更新で backfill 時に 34,069 ドキュメントの再チャンク + 再 embedding が発生。embedder のスループットに依存するが、数時間〜の処理時間が見込まれる
- **ボイラープレート判定のハードコード**: ニュースサイト固有のセクション見出し（「注目ワード」等）がコードにハードコードされている。新しいサイトのボイラープレートパターンが出現した場合、sanitizer の更新が必要
- **チャンク内容の変化**: V8 と V9 でチャンクの内容が大きく異なるため、backfill 完了前は V8 (HTML 入り) と V9 (テキストのみ) のチャンクが混在する。検索結果の品質は backfill の進行とともに段階的に改善される
- **不可逆変換**: sanitizer は HTML タグを除去するため、元の HTML 構造情報（リンク先 URL、画像の位置等）は失われる。ただし rag-db はインデックスであり、原本は alt-db に保持される
