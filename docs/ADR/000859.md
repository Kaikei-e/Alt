# ADR-000859: Knowledge Loop の canonical-contract 不変条件を静的テストと doc parser で防波堤化する

## Status

Accepted

## Date

2026-04-27

## Affected Services

- knowledge-sovereign: `app/usecase/knowledge_loop_projector/invariants_canonical_test.go` を新設し、migration ファイル本体を `regexp.MustCompile` で静的に scan して `updated_at` / `projected_at` exposure / 禁止カラム名を検出する
- knowledge-sovereign: `app/usecase/knowledge_loop_projector/projector_matrix_doc_test.go` を新設。`go/ast` で `constants.go` の `Event*` 定数を抽出し、`docs/features/knowledge-loop/projector-matrix.md` の Markdown table と相互拘束する
- alt-backend: `app/usecase/knowledge_loop_usecase/transition_writes_event_only_test.go` を新設し、`ClassifyTransitionEvent` を全 stage × 全 trigger で総当たりして `HomeItem*` 文字列を一切返さないことを assertion する
- docs: `docs/features/knowledge-loop/projector-matrix.md` を新設。projector がどの event type にどの projection 効果 (entry / session / surface / bucket / why / act_target) を出すかを表形式で固定
- .github/workflows: `knowledge-sovereign.yaml` の path filter / sparse-checkout に `docs/features/knowledge-loop/**` を追加し、doc 変更で projector matrix doc parser test が CI 上で実行されるようにした

## Context

Knowledge Loop の canonical contract (`docs/plan/knowledge-loop-canonical-contract.md`) は次の 3 種類の不変条件を持っている。

- **`updated_at` の不在** — `knowledge_loop_*` projection table に `updated_at` カラムを置かない (canonical contract §3 invariant 2)。代わりに `projection_seq_hiwater` / `projection_revision` / `freshness_at` などに分解する
- **`projected_at` exposure 禁止** — `projected_at` は projector の operational debug 用 wall-clock で、API / proto / public view / metrics / 本番ログに出してはいけない (§10 / §16 acceptance #12)
- **Single emission** — `/loop` UI が emit するのは `knowledge_loop.*.v1` のみ。`/feeds` UI が emit するのは `HomeItem*` のみ。同じユーザ意図で両方を emit してはならない (§3 invariant 7)

これらは migration / projector / classifier をまたいだ広域不変条件であり、コードレビューで都度確認するには見落としやすい。実際 [[000846]] / [[000853]] のレビューでも、新しい projection column や新 event 種別が増えるたびに「これ updated_at 入れていないか?」「public view から projected_at 漏れてないか?」が議論の俎上に上がっていた。

加えて projector がどの event 種別にどの side-effect を持つかは canonical contract §11 の WhyKind マッピングと組み合わさって複雑で、人間の目で「これ全部網羅しているか」を確認するのが難しい。新 event を `constants.go` に追加するときに doc 側を更新し忘れる drift が起きやすい構造だった。

`fb.md` のレビューで「不変条件を防波堤として CI ゲートに乗せる」ことが推奨されたため、本 ADR でその防波堤を確定させる。

## Decision

canonical contract の 3 系統不変条件を静的テストとして CI に固定する。

採用する 3 系統:

- **Migration 静的 scanner**: `invariants_canonical_test.go` が `00011_create_knowledge_loop_tables.sql` を `os.ReadFile` で読み込み、`extractCreateTable` / `extractCreateView` で各 `CREATE TABLE` / `CREATE VIEW` 本体を抽出。`columnList` でカラム名を切り出して `updated_at` / `modified_at` 等の禁止名が現れないことを assert する。`knowledge_loop_entries_public` view 本体は `projected_at` を含まないことも assert。さらに将来 migration で `CREATE VIEW knowledge_loop_*` が増えても `projected_at` を選ばないように、全 migration の `.sql` を走査する `TestProjectedAtNotInAnyPublicView` を追加する。DB 接続不要で `go test ./...` だけで走る
- **Projector matrix doc parser**: `docs/features/knowledge-loop/projector-matrix.md` を canonical event-type → projector-effect マッピングの正本に格上げする。`projector_matrix_doc_test.go` が `go/ast` で `constants.go` の `Event*` 定数値を抽出し、`SplitSeq` で markdown table の第 1 列を抽出して双方向に一致を assert する。`Event*` を追加して doc に書き忘れる、または doc に書いて const を増やし忘れる、のどちらでも CI が落ちる。`TestProjectorMatrix_NoStaleEntries` は event type vocabulary (Phase-0 PascalCase か Loop snake_case.v1) を regex で固定する
- **Single-emission rule の総当たり**: `transition_writes_event_only_test.go` が 4 stage × 4 stage × 5 trigger を全部試して、`ClassifyTransitionEvent` が `error` でなく event type を返したケースで `strings.HasPrefix(ev, "knowledge_loop.")` かつ `!strings.HasPrefix(ev, "HomeItem")` を assert する。さらに canonical な Loop event 6 種が必ず一度は出ることも別の assertion で固定して、テスト自体が空回りしないようにする。`HOME_ITEM_OPENED` 等の不正 trigger に対しても `HomeItem*` を返さないことを別テストで pin

CI 配線は `.github/workflows/knowledge-sovereign.yaml` の `paths` filter と reusable test job の `sparse-checkout` に `docs/features/knowledge-loop/**` を追加する。これで docs だけ更新されたケースでも projector matrix doc parser test が走り、`constants.go` との drift が即座に検出できる。

検討した代替案と却下理由:

- **Linter rule (golangci-lint custom rule) で `updated_at` を検出**: SQL は `.sql` ファイルなので Go linter のスコープ外。`.sql` 用 linter を別途維持するコストが大きい
- **DB schema dump を比較する integration test**: `pg_dump` をテスト時に走らせる必要があり、CI で DB を立てる依存が増える。静的 scan で十分な精度が出るのでコスト便益が悪い
- **doc を `template.go:embed` で Go 側に埋め込む**: 単一ソース化はできるが、Markdown を Go で render する余分な層を増やす。doc は doc、code は code のまま双方向 binding test で縛るほうが Obsidian 側の閲覧体験を壊さない
- **Single-emission rule を proto 側 enum 制約で表現**: `ClassifyTransitionEvent` は文字列ベースの contract であり、proto enum とは別 layer。総当たりテストのほうが「未来に新 trigger を増やしたとき」にも自然に protect できる

## Consequences

### Pros

- canonical contract の不変条件 3 系統が CI ゲート化され、`updated_at` / `projected_at` / `HomeItem*` 漏れを merge 前に検出できる
- `projector-matrix.md` が constants.go と双方向に拘束されるため、新 event 種別を増やすたびに doc 更新を忘れない。Obsidian 側の運用文書としても正本扱いできる
- 静的 scan / `go/ast` / 総当たりテストいずれも DB / network 依存ゼロで `go test ./...` の中で走るので、CI 時間や flakiness を増やさない
- 過去の `TestProjectedAtNotSerialized` 系の散発的なテストが、本 ADR で「migration scanner + 全 view 走査 + JSON wire」の 3 層に整理され、責務分担が明確になった
- `.github/workflows` の sparse-checkout 拡張により、doc-only 変更でも projector matrix の整合性が CI で守られる

### Cons / Tradeoffs

- migration 静的 scanner は `regexp` ベースなので「コメントの中に偽物の `updated_at` を書く」と false positive を出す可能性がある。実 migration ではコメント中の禁止名は無いため実害は今のところ無いが、将来増えたら scanner を strip-comment 化する必要がある
- `projector_matrix_doc_test.go` は `Event*` prefix の const 名を前提にしており、命名 convention を変えると壊れる。今のところ該当 const は全て `Event*` 接頭辞なので問題ないが、命名変更は ADR 必須に格上げされる
- doc 側の table は人間に読みやすい一方、機械読みするためのカラム順 (Event type / Entry / Session / Surface / Bucket / WhyKind / Act target hint) が固定される。カラムを増減するとテストも合わせて更新が必要
- `.github/workflows/knowledge-sovereign.yaml` の sparse-checkout 拡張により、docs/features 配下に無関係な大ファイルを置くと CI の checkout コストが増える。今は projector-matrix.md のみで影響軽微
