# ADR-000697: Acolyte 推論基盤を Qwen3.5-27B (Ollama GGUF) に移行し VllmGateway を追加する

## Status

Accepted

## Date

2026-04-11

## Affected Services

- acolyte-orchestrator (Python)

## Context

Acolyte のレポート生成パイプラインは LLM 推論をリモートの Ollama インスタンスに委譲している。従来は Gemma4 26B (Q4_K_M) を使用していたが、Qwen3.5-27B はベンチマークで大幅に優位 (MMLU-Pro 86.1, IFEval 95.0) であり、レポート品質向上のためにモデル移行を決定した。

同時に、将来的な推論バックエンドの多様化 (vLLM, llama.cpp サーバー等の OpenAI 互換 API) に備え、既存の OllamaGateway に加えて OpenAI 互換 API を話す VllmGateway を新設し、環境変数で切り替え可能にする設計が必要になった。

### vLLM 直接使用を断念した経緯

当初は vLLM での推論を計画し、ソースビルド・起動まで到達したが、以下の理由で断念した。

| 量子化方式 | 結果 | 理由 |
|---|---|---|
| GPTQ-Int4 | 不正出力 | 4-bit gptq_gemm カーネルが buggy (vLLM 公式警告) |
| gptq_marlin | 非互換 | qzeros shape mismatch |
| FP8 | ハードウェア非対応 | RDNA 3.5 に FP8 WMMA 命令なし (RDNA 4 以降) |
| AWQ | segfault 報告 | 同系 GPU で libhsa-runtime64.so クラッシュ |
| BF16 | メモリ不足 | ~54GB でUMA VRAM 上限を超過 |

RDNA 3.5 iGPU + vLLM + 27B 規模の量子化モデルは、現時点で安定動作する組み合わせが存在しなかった。一方、Ollama (llama.cpp ベース) は同一 GPU 上で Gemma4 26B Q4_K_M の安定動作実績があり、Qwen3.5-27B GGUF Q4_K_M も正常に推論できることを確認した。

## Decision

**Acolyte の推論モデルを Qwen3.5-27B (Ollama GGUF Q4_K_M) に移行し、OpenAI 互換 API 用の VllmGateway を追加する。**

### 1. モデル移行

リモート Ollama に Qwen3.5-27B Q4_K_M (unsloth 公式 GGUF, 16GB) をデプロイし、`DEFAULT_MODEL` を切り替える。Embedding 用の embeddinggemma と同一 Ollama 上で共存する。

### 2. VllmGateway 新設

`LLMProviderPort` Protocol を実装する `VllmGateway` を新設。OpenAI 互換 API (`/v1/chat/completions`) を話すため、vLLM・llama-server・LiteLLM 等のバックエンドを透過的に使える。`LLM_PROVIDER` 環境変数で `ollama` / `vllm` を切り替え、Composition Root (`main.py`) で注入先を決定する。

### 3. OllamaGateway の設定外部化

`_BASE_OPTIONS` にハードコードされていた `stop` トークンと `num_ctx` を `Settings` から注入可能にし、モデル変更時にコード変更なしで対応できるようにした。また structured output で `think: false` を明示送信するよう変更した (Qwen3.5 の thinking token がトークン予算を消費する問題の回避)。

## Implementation

### 新規ファイル

| ファイル | 責務 |
|---|---|
| `acolyte/gateway/vllm_gw.py` | OpenAI 互換 API による LLM 推論 Gateway |
| `tests/unit/test_vllm_gw.py` | VllmGateway 単体テスト (18 件) |

### 変更ファイル

| ファイル | 変更内容 |
|---|---|
| `acolyte/config/settings.py` | `llm_provider`, `vllm_api_key`, `llm_num_ctx`, `llm_stop_tokens` 追加 |
| `acolyte/gateway/ollama_gw.py` | `_BASE_OPTIONS` を Settings 注入に変更、structured に `think: false` 追加 |
| `main.py` | Provider 選択ロジック (`LLM_PROVIDER` env で Gateway 切替) |
| `compose/acolyte.yaml` | `LLM_PROVIDER`, `VLLM_API_KEY` 環境変数追加 |

### API 変換マッピング (VllmGateway)

```
Ollama /api/chat              → OpenAI /v1/chat/completions
format: {json_schema}         → response_format: {type: "json_schema", json_schema: {schema: ...}}
options.num_predict           → max_tokens
think: true                   → chat_template_kwargs: {enable_thinking: true}
response.message.content      → choices[0].message.content
response.eval_count           → usage.completion_tokens
```

### アーキテクチャ

```
                    ┌─ OllamaGateway ──→ Ollama /api/chat, /api/generate
LLMProviderPort ◄───┤
                    └─ VllmGateway ────→ OpenAI /v1/chat/completions
                          ↑
                    LLM_PROVIDER env
```

Clean Architecture の Port/Gateway 分離により、推論バックエンドの変更が Gateway 層に閉じ、Usecase 層 (LangGraph パイプライン) は一切変更不要。

## Consequences

### Pros

- Qwen3.5-27B は Gemma4 対比でベンチスコアが大幅に優位 (MMLU-Pro 86.1, IFEval 95.0)。レポート品質の向上が見込める
- VllmGateway により、将来 vLLM / llama-server / 外部 API 等の OpenAI 互換バックエンドに切り替え可能
- OllamaGateway の `_BASE_OPTIONS` 外部化により、モデル変更時のコード変更が不要
- `LLM_PROVIDER` 環境変数でのバックエンド切替により、ロールバックが `.env` 変更のみで完結

### Cons / Tradeoffs

- GGUF Q4_K_M 量子化による品質劣化の可能性。実運用での品質評価が必要
- vLLM の GPTQ/AWQ カーネルが RDNA 系 iGPU で安定するまで、Ollama (llama.cpp) が唯一の推論パス
- Qwen3.5 の thinking mode がデフォルト有効。structured output では `think: false` を明示しないとトークン予算を浪費する
- VllmGateway は追加されたが、現時点では Ollama 経由での運用。vLLM 直接利用は将来の課題
