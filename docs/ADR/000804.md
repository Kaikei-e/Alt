# ADR-000804: Section rerun と rubric 評価の LLM 呼び出しに think=false を固定し CJK empty 応答を防ぐ

## Status

Accepted

## Date

2026-04-20

## Affected Services

- acolyte-orchestrator (Python)

## Context

[[000801]] で HyDE の `think=false` pin が恒久化され、Gemma 4 が CJK プロンプトを受け取った際に `thinking` capability へ暗黙移行し `num_predict` 枠を内部推論で使い切って empty response を返す挙動は封じた。ただしその修正は HyDE のみに局所化されており、他の LLM 呼び出しでも同じ障害クラスを踏む余地が残っていた。

監査結果:

| 呼び出し元 | mode / think 指定 | CJK リスク |
|---|---|---|
| `gateway/news_creator_hyde_gw.py:55` HyDE | `think=False` 明示 ([[000801]]) | 解消済 |
| `usecase/rerun_section_uc.py:57` 単一セクション再生成 | mode 指定なし / think 未指定 → `_generate_freetext` が `think=None` で /api/generate を呼び Gemma 4 の default thinking にフォールバック | **要対策** |
| `usecase/eval/rubric_evaluator.py:49` claim extraction | 同上 | **要対策** |
| `usecase/graph/nodes/curator_node.py:130` | `mode=STRUCTURED` → gateway が `think=False` を強制 | 解消済 ([[000779]]) |
| `usecase/graph/nodes/critic_node.py:693` | 同上 | 解消済 |
| `usecase/graph/nodes/writer_node.py:552, 823` | `mode=LONGFORM` → gateway が `settings.longform_think` (default `False`) を通す | 解消済 (設定管理下) |

`RerunSectionUsecase` はユーザが UI から「セクションを再生成」した際に呼ばれ、日本語トピックのレポートではプロンプト全体が CJK 比率の高い `WRITER_PROMPT` を直接投げる。`RubricEvaluator.extract_claims` は評価ハーネス ([[000793]]) が各レポート body+evidence から claim を抽出する経路で、日本語記事本文を直接 LLM に渡す。両者とも empty body を返すと section bump / rubric score が空で確定し、rollback できない既成事実になる。

## Decision

上記 2 呼び出し元に対して、minimal diff で `think=False` を明示する:

- `RerunSectionUsecase.execute` の `self._llm.generate(prompt, num_predict=2000)` → `self._llm.generate(prompt, num_predict=2000, think=False)`
- `RubricEvaluator.extract_claims` の `self._llm.generate(prompt, temperature=0, num_predict=1024)` → 同じく `think=False` を追加

RED テストを先行し:

- `tests/unit/test_rerun_section.py::test_rerun_section_pins_think_false_for_cjk_safety` — `FakeLLM.last_kwargs.get("think") is False` を担保
- `tests/eval/test_rubric.py::test_claim_extraction_pins_think_false_for_cjk_article_bodies` — 同等の担保

`FakeLLM` 側にも `last_kwargs` を持たせるため `__init__` を追加。既存テストは API 不変で通る。

**却下した代替**:
- `OllamaGateway` の `_generate_freetext` パスで `think=None` なら暗黙的に `False` に寄せる — 後方互換を破るうえ、呼び出し元の意図 (「thinking を許すのか禁じるのか」) が gateway で不可視化される。明示指定を選ぶ。
- `Settings` に `freetext_think: bool = False` を新設して gateway で参照 — [[000779]] の `longform_think` とは別軸の単一 knob を増やすことになる。該当 2 呼び出しだけなので、呼び出し側でハードコードする方がコードパスが読みやすい。
- `LLMMode` に `CLAIM_EXTRACTION` / `RERUN` を追加 — モード分岐はスキーマを持つ呼び出しの抽象であり、HyDE 型の free-text passage 生成には過剰。

## Consequences

### Pros

- 日本語 Writer prompt / 日本語記事本文に対して empty response が返るリスクが消え、UI からのセクション再生成とオフライン eval が CJK 入力に対しても決定論的に走る。
- [[000801]] と同じ方針 (呼び出し側で明示) を踏襲するので、Gemma 4 特性の変更 (thinking-budget の導入など) があっても grep で全 think=false 箇所を一度に再評価できる。
- RED テスト 2 本が protective test として残り、将来誰かが `think=False` を外すと即座に落ちる。

### Cons / Tradeoffs

- 将来 Gemma 系を「思考する」モデルに差し替えたくなった場合、2 箇所の `think=False` を設定化する必要がある (そのときは `Settings.rerun_think` / `Settings.rubric_think` を導入する前提)。
- `FakeLLM.last_kwargs` は簡易な side-channel で、テスト数が増えた場合の可読性は下がる。対象が 2 テストなので今は許容。
