# ADR-000939: Evidence 供給を co-projected accumulator へ作り直し relation-set を埋める

## Status

Accepted

[[000937]] / [[000938]] が固定した「relation-set を projection の主軸とする」方向は維持し、**relation を埋める evidence 供給機構**を作り直した。Decision の全項目が landed 済み (knowledge-sovereign の accumulator / projector / honest gate、alt-backend の tags payload + CDC、recap-worker の owner 配線)。全サービスのテストが green。残るは本番 rollout (`atlas migrate apply` → compose rebuild → full reproject) のみで、これは Deploy-model セルフチェックの手順に従いユーザー承認後に実施する。

## Date

2026-06-10

## Affected Services

- knowledge-sovereign (Go)
- alt-backend (Go)
- recap-worker (Rust)

## Context

[[000937]] / [[000938]] で relation-set を主軸へ昇格し、`extractRelations` が全関係種を emit する形まで landed した。それでも実機の Knowledge Loop は「装飾された feed」のままだった。2026-06-10 の本番 DB / ログ / コード裏取りで、病が **relation を畳む場所** ではなく **relation を埋める evidence が供給されていない** ことだと確定した。

- `knowledge_loop_entries` 62,990 件中 relations 非空は **23 件 (0.04%)**、全て 2026-06-03 (landing 当日) 産。6/4 以降の新規 1,622 件は 100% `surface_planner_version=1` / `surface_score_inputs` 空 / `relations=[]`。
- **真因 1 — silent truncation**: `EventLogSurfaceScoreResolver` (`score_resolver_event_log.go`) は entry を project するたびに 7 日 window を `ListKnowledgeEventsForUserInWindow` で読むが、その SQL は `ORDER BY event_seq ASC LIMIT 256`。単一ユーザーの実 window は約 3,757 件あるため、resolver は **最古の 256 件しか読めず**、直近の証拠 (同一 article の SummaryVersionCreated 直後の SummarySuperseded 等) に到達しない。全 signal が 0 に潰れ → `hasV2Signal` false → v1 fallback → `relations=[]`。SQL エラーは起きないので fail-loud は鳴らない。
- **真因 2 — reproject 非決定性**: 同 window クエリは occurred_at で範囲を切るが event_seq 上限で拘束しないため、reproject 時に「未来 seq」のイベントが window に混入し得る。
- **真因 3 — 残存 silent fallback**: [[000928]] / [[PM-2026-045]] が根絶したはずの silent fallback が `score_resolver_event_log.go:128` (`if err != nil { return out }`) に残る。[[000938]] の fail-loud commit は alt-backend `decodeRelations` のみを覆い、resolver の lookup error 経路は素通り。
- **真因 4 — Cluster 燃料が物理的に不在**: `recap.topic_snapshotted.v1` は全期間 0 件 (recap-worker の owner 未配線で producer 永久 false)。`TagSetVersionCreated` payload は tags 配列を運ばない。よって Cluster relation は構造的に算出不能。
- **嘘計器が残存**: `plannerVersionForResolver` (`score_resolver.go`) は resolver の型が非 Null なら無条件 `v2` を返し、`hasV2Signal()` は未使用。honest な `relation_coverage_total` は追加されたが、その閾値アラートは「流量 > 0.01/s (≈864 projections/day)」を active-traffic guard に要求し、実流量 (約 250/day = 0.003/s) では **構造的に発火不能**。劣化を検知する gate が死んでいる。
- **テストが密度を再現しない**: [[000938]] の垂直真実テストは seed が 2 イベントで、`LIMIT 256` に絶対届かない。「test green / experience empty」がそのまま温存された。

つまり [[000937]] / [[000938]] の診断「精巧な evidence が単軸に潰される」を一段深く掘ると、**evidence を取得する pull 型 window re-scan がログ密度で破綻し、計器がそれを隠していた**。relation-set の骨格は健全であり、作り直すべきは供給機構と gate である。

## Decision

`EventLogSurfaceScoreResolver` の pull 型 window re-scan を廃止し、evidence を **append 時に O(1) で畳み込む co-projected accumulator** に置換する。relation-set 主軸 ([[000937]]) と Contradiction-first ループ閉じ ([[000938]]) は維持する。具体的に以下を採用する。

### 1. Evidence-as-projection — `knowledge_loop_evidence` accumulator を導入する

projector が消費するイベントごとに、関連 scope (article / entry / tag / topic_term) の evidence 行を `seq`-guard 付き UPSERT で O(1) 更新する。`SurfaceScoreInputs` はこの累積からの **純導出**になる。`decideBucketV2` / `extractRelations` / `hasV2Signal` は**無変更** — 入力の生成元だけが pull から accumulator 読み出しに変わる。window re-scan と `LIMIT 256` 由来の silent truncation が構造的に消える。

- 却下した代替: **window スキャン改良 (DESC + ページング)** — O(window) スキャンとログ密度比例コストが残る。**信号別 targeted query** — 信号追加のたびにクエリが増える N+1 傾向、tag/topic 重なりは依然 JSONB スキャン。

### 2. Co-projection パターンを正当な例外として法典化する

accumulator 自体が projection であり、entry projector がそれを読むことは表面的には「最新状態を query しない」不変条件 (canonical contract §3.1) に抵触する。これを次の **5 条件**を全て満たす場合に限り正当な例外と定める。

- (a) accumulator と entry projection は **同一 projector pass・同一 checkpoint cursor (`knowledge-loop-projector`)** で更新される。独立 cursor を持たせない。
- (b) **Derive-before-Apply**: イベント N の処理で、まず accumulator (prefix `seq < N` の状態) から Derive して entry を作り、その後 N の fact を Apply する。よって entry N の relations は **event-log prefix (`seq < N`) の決定的純関数**であり、reproject は bit-for-bit 再生する。
- (c) accumulator は **disposable**: reproject は `knowledge_loop_entries` 等と同一トランザクションで `knowledge_loop_evidence` を TRUNCATE → 再構築する。
- (d) writer は projector のみ。他サービス / cron からの書き込みを禁止する。
- (e) 窓導出は fact の `occurred_at` フィルタ ([ev.occurred_at − 7d, ev.occurred_at)) で行い wall clock を使わない。計算済み decay 値を列に保存しない (canonical contract §6.5 維持)。raw fact (occurred_at + event_seq) のみ保存し、窓カウントは導出時に算出する。

### 3. 遅着燃料は即時に relation を再導出する

entry が surface された後に到着する evidence (TagSetVersionCreated / 後続 supersede / augur link) は、到着した **同一 pass 内**で当該 entry (article_id / entry_key で一意特定) の relation-set を patch 再導出する。新規 `now` 記事の Orient が「次回接触まで空」になる構造的欠落を断つ。fan-out は記事 1 件に閉じるので [[PM-2026-039]] の invalidate storm とは異質。entry を生やす権限は従来種に限り、TagSetVersionCreated 単独では entry を新規生成しない (entry 不在なら patch は no-op)。

- 却下した代替: **次回接触時のみ再導出** — 新規記事の Cluster が永遠に点かず [[000938]] が指摘した構造的空白が残る。

### 4. 嘘計器を撤去し honest gate を 3 点で作り直す

- `plannerVersionForResolver` の無条件 v2 と未使用 `hasV2Signal` を整合させ、per-entry planner version を **実 evidence の有無 (`hasV2Signal`)** で決める。
- **DB 実測 coverage gauge**: 「直近 24h に project された entry のうち relation 非空の比率」を定期クエリで gauge export。rate ベースの dead guard を撤去。低流量でも必ず評価される。
- **密ログ真実テスト**: 垂直真実テストの seed を本番密度 (5,000+ events / 7d window) にし、「ログ密度で潰れる」階級のバグを merge 前に RED 化する。
- **producer-liveness SLI**: `recap.topic_snapshotted.v1` / `augur.conversation_linked.v1` を上流活動と比較し「未使用」と「producer 死」を計器上で峻別する ([[PM-2026-045]] の無音 producer-death を検知する唯一の gate)。

### 5. 旧機構を退役する (backward-compat shim なし)

- `EventLogSurfaceScoreResolver` + その unit/invariant test を削除。`SurfaceScoreResolver` interface / `NullSurfaceScoreResolver` / `WithScoreResolver` / `plannerVersionForResolver` を撤去し、accumulator store を必須 DI とする (未配線は panic)。
- `surface_planner_cron` を退役する。唯一の signal (augur.conversation_linked.v1) は accumulator が entry-mutating event として直接処理でき、中間の `SurfacePlanRecomputed` 再 emit 層は迂回路に過ぎない。projector の `KnowledgeLoopSurfacePlanRecomputed` branch は、log 内の歴史イベント件数が 0 の場合のみ削除し、>0 なら replay 互換のため存置する (実装 PR で DB 確認の上確定)。
- `ListKnowledgeEventsForUserInWindow` は `macro_state_builder` 専用に縮退して存置する (対象イベントが低頻度で LIMIT 飽和リスクが実質ない)。

immutable invariants (append-first / reproject-safe / disposable projection / why-first-class / event-time purity) は全て保持する。relations / Contradiction state は event log の純関数であり、retire 済み stage イベント (`oriented` / `decision_presented`) は log に残す。`WhyMappingVersion` を bump し full reproject を法定手続きとする。

## Consequences

### Pros

- **silent truncation が構造的に消える。** 供給が pull window re-scan から append-time accumulator になり、`LIMIT 256` 由来の evidence 全損が起きなくなる。relation を持つ entry が初めて本番で埋まる。
- **reproject 決定性が強化される。** Derive-before-Apply により entry N の relations は prefix `seq < N` の純関数。occurred_at × seq の混入も accumulator の seq 順処理で解消。
- **計器が嘘をつかなくなる。** per-entry planner version が実 evidence 基準になり、DB 実測 coverage gauge が低流量でも評価され、producer-liveness SLI が無音 producer-death を捕捉する。
- **新規記事の Orient が埋まる。** 遅着燃料の即時再導出で、取り込んだばかりの記事も tag / recap 流入後に Cluster relation を得る。
- **attempt を構造的に検証する。** 密ログ真実テストが「ログ密度で潰れる」回帰を merge 前に RED 化する。

### Cons / Tradeoffs

- **書き込み増幅。** 全イベントが accumulator UPSERT を伴う。TagSetVersionCreated (約 2,123 件/週) の遅着再導出は entry patch も伴う。seq-guard と patch-only 限定で増幅を有界化するが、projector の per-event コストは上がる。
- **schema 追加と reproject 必須。** `knowledge_loop_evidence` の migration と full reproject が要る。hard-to-reverse。
- **co-projection の認知負債。** 「projector が projection を読む」形は、5 条件を満たさない安易な踏襲を招きやすい。型 / コメント / runbook で 5 条件を明示する必要がある。
- **過渡期の Cluster 非対称。** tags / recap 燃料の本番流入 (別 PR) が landing するまで Cluster は空。Contradiction が先行し、Cluster は数日かけて立ち上がる。
- **proto は不変。** `KnowledgeLoopEntry.relations` は sovereign → alt-backend `decodeRelations` → loop.v1 まで貫通済みのため proto 追加は不要だが、両 ADR ([[000937]] / [[000938]]) を併読しないと「なぜ accumulator か」が読み取りにくい。本 ADR の Context がその橋渡しを担う。
