# ADR-000545: Recall パイプラインに OTel メトリクス・SearchRelated シグナル・フロントエンド統合修正を追加する

## Status

Accepted

## Date

2026-03-23

## Affected Services

- alt-backend (Go) — OTel メトリクス追加、SignalSearchRelated プロデューサー接続
- alt-frontend-sv (SvelteKit) — Recall 統合ロジック修正

## Context

Knowledge Home の Recall 機能は ADR-000408 で導入され、ADR-000463 でシグナルパイプラインが復旧された。しかし運用開始後、以下の課題が明らかになった。

1. **Observability の欠如**: `knowledge_home_metrics.go` に 60 個以上の OTel メトリクスが定義されていたが、Recall パイプラインに関するメトリクスは 0 件だった。プロジェクターの実行時間、候補生成数、空率などが計測されておらず、障害検知が不可能な状態だった。

2. **未接続シグナル**: 定義済み 6 種のシグナルのうち `SignalSearchRelated`（検索結果オープン時）はスコアリングロジックが実装済みだったが、プロデューサーが接続されていなかった。`open_search` アクションタイプは `validActionTypes` に存在するにもかかわらず `actionToSignalType` マップに未登録だった。

3. **フロントエンド統合の不整合**: `syncRecallState` が `home.recallCandidates.length > 0` で分岐しており、サーバーが正当に空の候補リストを返した場合にも別 API (`getRecallRail`) へフォールバックしていた。これは Knowledge Home の単一フェッチ契約（ADR-000480）に違反していた。

## Decision

以下の 3 つの改善を同時に実施する。

### Phase 0: Recall OTel メトリクス

`KnowledgeHomeMetrics` 構造体に Recall 専用メトリクスを 6 個追加し、`RecallProjectorJob` に計装する。

追加メトリクス:
- `alt.home.recall.signal_append_total` — シグナル追記成功数
- `alt.home.recall.signal_append_errors` — シグナル追記失敗数
- `alt.home.recall.candidate_generated_total` — 候補生成数
- `alt.home.recall.candidate_empty_total` — 候補ゼロのユーザー数
- `alt.home.recall.projector_duration_ms` — プロジェクター実行時間
- `alt.home.recall.projector_users_processed` — 処理ユーザー数

`RecallProjectorJob` のシグネチャを変更し、`*otel.KnowledgeHomeMetrics` を受け取る。`nil` 許容で既存の呼び出しパターンを壊さない。

### Phase 1: SignalSearchRelated プロデューサー接続

`TrackHomeActionUsecase` の `actionToSignalType` マップに `"open_search": domain.SignalSearchRelated` を追加する。`open_search` は既に `validActionTypes` に定義済みのため、1 行の追加で検索結果オープンからリコールシグナルが生成される。

### Phase 4: フロントエンド Recall 統合修正

`+page.svelte` の `syncRecallState` を簡素化し、Home レスポンスの `recallCandidates` を無条件で信頼する。

変更前:
```typescript
async function syncRecallState() {
  if (!recallEnabled) return;
  if (home.recallCandidates.length > 0) {
    recall.setCandidates(home.recallCandidates);
    return;
  }
  await recall.fetchCandidates();
}
```

変更後:
```typescript
function syncRecallState() {
  if (!recallEnabled) return;
  recall.setCandidates(home.recallCandidates);
}
```

`useKnowledgeHome` フックの `fetchData` は常に `recallCandidates` を設定する（L109）ため、`recallDataIncluded` フラグは不要と判断した。`async` も不要になったため同期関数に変更。

## Consequences

### Pros

- **障害可視化**: Recall プロジェクターの実行状況がメトリクスで監視可能になった。候補空率スパイクの検知が可能
- **シグナル網羅率向上**: 稼働中シグナルが 3/6 → 4/6 に改善。検索行動からのリコール候補生成が有効化
- **単一フェッチ契約の遵守**: 不要な別 API 呼び出しが除去され、Home レスポンスが Recall の唯一のデータソースになった
- **スライス事前確保**: `reasons` スライスを `make([]T, 0, len(sigs))` で事前確保し、不要なアロケーションを削減

### Cons / Tradeoffs

- **RecallProjectorJob のシグネチャ変更**: `metrics` パラメータの追加により既存の呼び出し元（`main.go`）の修正が必要。ただし `nil` 許容のため段階的移行可能
- **未接続シグナル 2 種が残存**: `SignalRecapContextUnread` は recap DB へのクロスサービスアクセスが必要（Phase 2 で対応予定）、`SignalPulseFollowup` は news-pulser が未完成のためブロック中
- **フロントエンドのフォールバック除去**: Feature Flag 無効時に Home レスポンスが空配列を返すケースでは、以前のフォールバックで別途取得できていた候補が表示されなくなる。ただしこれは正しい動作（フラグ無効 = Recall 無効）
