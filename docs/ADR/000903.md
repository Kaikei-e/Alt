# ADR-000903: Knowledge Loop を Alt の primary surface に格上げし `/` から `/loop` へのサーバサイド導線と welcome onboarding を確立する

## Status

Accepted

## Date

2026-05-20

## Affected Services

- alt-frontend-sv (TypeScript / Svelte 5)

## Context

[[knowledge-loop-canonical-contract]] §0 は「`/feeds` と `/loop` は並行稼働、ユーザが自ら選択する」と書いており、`/loop` は menu の二階層下 (`AI & Insights` セクション) に置かれていた。実際の運用では Knowledge Loop に到達するユーザが極めて少なく、`/loop` の存在を知らないユーザが多数派という discoverability ギャップが 2026-05-20 の FE 監査で立証された。

Phase 0 / Phase 1 監査の結論を Knowledge Loop v2 plan (`/home/koko/.claude/plans/knowledge-loop-ux-alt-knowledge-loop-pla-cuddly-crayon.md`) は次のように要約している:

- 実装は contract §1–§14 のうち 11/14 達成。残る gap は ① workspace bare transition、② Review guidance、③ Continue `lastInteractedAt` 非表示、④ `+/−` 差分視覚化、⑤ Reduced Motion color-shift、⑥ Japanese i18n labels、⑦ **`/feeds` → `/loop` 導線が存在しない**、⑧ `/loop` discoverability が menu 二階層下
- 構想層 (思想) では Knowledge Loop は「**ユーザの認知状態を扱う state engine**」であり、Knowledge Home (`/home`) や raw inbox (`/feeds`) より上位の primary surface であるべき
- 「フロントエンドへの path を確立する」という要求の本質は `/loop` を Alt の認知 OS の入口に据えること

`/home` は Phase 0 Knowledge Home (curated 記事リスト)、`/feeds` は raw RSS inbox という棲み分けは維持しつつ、`/loop` を primary に格上げする。

## Decision

`/loop` を Alt の primary surface として確立する。具体的には以下 5 つを 1 PR でまとめて実施する:

1. **サーバサイド redirect** — `src/routes/+page.server.ts` を新設し、`locals.backendToken` が存在する場合は `cookies.get("alt_loop_welcomed")` の有無で `/loop` (welcomed) と `/loop/welcome` (first-time) のどちらかへ 303 redirect する。未認証は landing page (`+page.svelte`) にフォールスルー。client-side `goto("/home")` の `$effect` ブロックは撤去し race の余地を消す。
2. **`/loop/welcome` route** — Newspaper Style masthead (`Volume I, A Cognitive State Engine`) + 3 行 editorial copy (`Your cognitive state, not your inbox.` / `What's now, what continues, what changed, what's worth a second look.` / `Pick a lens — you can switch any time.`) + 4-lens segmented picker (`Research / Browse / Decide / Recall`) + Start CTA。Start で `alt_loop_welcomed=true` cookie (`SameSite=Lax`, `maxAge=1y`) を発行し、`/loop?lens=<id>` へ 303 redirect。
3. **LoopCrossLink banner** — 単一目的の 1 行 Newspaper コンポーネント (`source` prop + `—` separator + label + arrow)。`(app)/home/+page.svelte` (desktop + mobile) と `(app)/feeds/+page.svelte` (desktop + mobile) の wire-header 直下に `source="From Home"` / `source="From Feeds"` でマウント。
4. **bottom-nav 再配置** — `NAV_TABS` の position 2 を `Swipe (→ /feeds/swipe/visual-preview)` から `Loop (→ /loop, Workflow icon)` に差し替え (合計 5 tabs 維持)。`MOBILE_MENU_SECTIONS` の `AI & Insights` から `Knowledge Loop` を撤去 (primary に昇格したため重複防止)、Browse セクションに `Swipe (→ /feeds/swipe/visual-preview)` を追加し destination を保全。`bottom-nav.ts::getActiveTabIndex` で tab index 1 を `/loop*` に再割当てし、旧 `/feeds*` → 1 mapping は Menu (4) フォールバックに変更。`/feeds/search` の `/search` (2) special-case は維持。
5. **E2E spec** — `tests/e2e/desktop/loop/loop-primary-surface.spec.ts` に 6 cases (`/` welcomed redirect / `/` first-time redirect / welcome view / Start CTA cookie + redirect / home banner / feeds banner)。`tests/e2e/.auth/storage.json` の authenticated session を流用、cookie state はテストごとに addCookies/clearCookies で明示。

### 検討した代替案

- **menu 内で目立たせるだけにする** — discoverability gap を本質的に解消しない (`AI & Insights` ラベル自体に到達するハードルが高い)。却下。
- **`/loop` への redirect を `/feeds` 廃止と同時に行う** — `/feeds` は raw inbox として独立価値があり (power user 向け / RSS 一覧 / unread filter)、廃止しない。クロス導線で `/loop` への昇格意図を伝える共存設計を選択。
- **6 tabs に拡張して Loop を追加する** — Material 3 / NN/g ガイダンスから外れる (3–5 tabs 推奨)。`Swipe` は menu に降格し 5 tabs を維持。
- **welcome を modal で出す** — 専用 route のほうが URL 共有 / 再訪・ブラウザ戻る対応に強い。専用 route 採用。
- **client-side cookie / localStorage 判定** — `+page.server.ts` の load 関数で server-side に判定したほうが flash of landing UI を排除でき SSR-disabled 構成 (`(app)/+layout.ts: ssr=false`) でも redirect 自体は早期 (load 時) に成立する。server-side 判定採用。

## Consequences

### Pros

- Knowledge Loop が Alt のデフォルト entry point になり、認知状態ベースの UX が main path に乗る (構想層 §1 「認知状態を扱う」を実装に下ろせた)
- `/feeds` と `/home` から 1 click で `/loop` に到達可能となり、Phase-0 read model から v2 read model への hand-off path が確立
- welcome route で Lens 選択を最初に行う設計が、構想 v2 Δ5 (Lens primacy) と Phase G の足場になる
- redirect は server-side のため flash of landing UI が消え、authenticated session の初回 navigation が 1 hop で `/loop` まで届く
- TDD outside-in を遵守: E2E (Playwright) → unit (navigation.test.ts / bottom-nav.test.ts) → impl の順で RED → GREEN

### Cons / Tradeoffs

- `/feeds` の primary destinations 5 tab 内訳が変わったため、Swipe entry を多用していたユーザは menu 経由のワンタップ余分が発生する。`Swipe` を Browse セクション最上位に置いて緩和。
- `alt_loop_welcomed` cookie がブラウザ単位なので、複数デバイスから初回ログインすると毎回 welcome を経由する。仕様として受容 (cross-device persistence は別 ADR で `auth-token-manager` 経由を検討する余地)。
- E2E spec はまだ playwright integration runner 実行が完了していない (本 ADR の reroll は実装直後の local CI parity = `bun run check` 0 errors + `bun run test` 1291/1291 pass で確認済み)。staging で実行は後続セッションで行う。
- `/loop` への primary 化は構想 v2 を前提とする決定であり、後続 Phase D / E / F で Loop 自体の体験 (semantic decide, epistemic Review, counter-evidence) を確実に landed させる必要がある。半端な状態で primary 化したまま放置すると逆に DAU を毀損するリスクがある (リスク対策: Phase B 同時 landed)。
