# ADR-000475: Recall candidate read path で Knowledge Home item を補完する

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-backend (Go)

## Context

RecallRail は `GetKnowledgeHomeResponse.recall_candidates` に初期候補を埋め込み、初回描画時の追加ラウンドトリップを避ける設計になっている。しかし `GetRecallCandidates` の read path は `recall_candidate_view` だけを参照しており、表示に必要な `KnowledgeHomeItem` 情報を返していなかった。

この状態では `RecallCandidate.Item` が常に欠落し、フロントエンドは `itemKey` をそのままタイトル代替として表示するしかない。RecallRail の「想起支援」という意図に対して、記事タイトル・要約・タグ・リンクが読めない候補は情報密度が低く、実用性が下がる。

一方で `proto`、handler、フロントエンドの Connect 変換はすでに optional な `item` を扱える。新しい API や projector を増やさず、repository の read path だけで補完する余地がある。

## Decision

`GetRecallCandidates` は `recall_candidate_view` に加えて active projection の `knowledge_home_items` と `articles` を `LEFT JOIN` し、`RecallCandidate.Item` を read path で補完する方式を採用する。

具体的には以下を決定した。

1. `knowledge_home_items` は `user_id` と `item_key` に加え、active な `projection_version` と `dismissed_at IS NULL` を条件に `LEFT JOIN` する
2. `articles` は `primary_ref_id` から `LEFT JOIN` し、カード遷移に使う `link` を補完する
3. JOIN 側カラムは nullable scan で受け、該当行がない場合は `RecallCandidate.Item` を `nil` のまま返す
4. `tags_json` と `why_json` の decode 失敗は候補全体の失敗にせず、空配列へフォールバックする
5. active projection の解決条件は `GetKnowledgeHomeItems` と同じ `knowledge_projection_versions` のサブクエリに揃える

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|--------|------|-----------|
| `recall_candidate_view` 自体に title / summary / tags を複製して保持する | read は単純になる | Recall 専用 view が Home read model の複製責務まで持ち、projection の重複更新が増える |
| handler で item を別取得する | 実装は分離しやすい | 初回ロード時に候補数分の参照や追加 query が必要になり、埋め込みの利点が薄れる |
| item が欠落しても `itemKey` 表示で運用する | 変更量は最小 | RecallRail の意味密度が低く、canonical contract に寄せられない |

## Consequences

### Pros

- RecallRail の初期候補で記事タイトル・要約・タグ・リンクを即時に返せる
- `proto` や handler を変えずに read path のみで不整合を補正できる
- `LEFT JOIN` のため projection lag や欠損時も graceful degradation を維持できる
- active projection を基準にすることで Knowledge Home 本体と同じ read model version を参照できる

### Cons / Tradeoffs

- repository の SQL と scan ロジックが長くなり、nullable handling の複雑さが増える
- `knowledge_home_items` と `articles` への JOIN により read path のコストは増える
- JSON decode 失敗を空配列へ落とすため、壊れた projection の検知は別の観測に依存する
