# ADR-000431: Knowledge Home の右 rail を RecallRail の empty state を含む恒常領域として扱う

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-frontend-sv (TypeScript/SvelteKit) — `home/+page.svelte`, `RecallRail`, `KnowledgeCard`

## Context

Knowledge Home の desktop 右 rail には RecallRail コンポーネントが実装済みだったが、ページ側では「候補が 1 件以上あるときだけ RecallRail を表示し、0 件なら `MiniRecallPanel` に戻す」という条件分岐になっていた。そのため feature flag を有効にしていても、候補が空の初期状態では右 rail が static shortcut panel のままになり、`reason.md` の「右 rail を RecallRail 化する」という指摘に応えられていなかった。

一方で `RecallRail` 自体は empty state を持っており、候補 0 件でも「今は思い出すものがない」という状態を表現できる設計だった。問題はコンポーネントの不在ではなく、ページレベルの表示方針だった。

またカード下部のタグ overflow 表示は `+7` のような裸の数値で、追加タグ数なのか関連数なのかが判断しづらかった。Knowledge Home を「知識密度の高い面」として扱うなら、こうした圧縮表現も意味が読める必要があった。

## Decision

desktop の Knowledge Home 右 rail は、RecallRail feature flag が有効な限り、**候補件数に関係なく RecallRail を表示する**。

具体的には以下を採用する。

- `recallEnabled` が true のときは `recall.candidates.length` を条件にせず `RecallRail` を常時描画する
- 候補 0 件時は `RecallRail` の empty state をそのまま使う
- `MiniRecallPanel` へのフォールバックは feature flag 無効時だけに限定する
- タグ overflow 表示は `+N` ではなく `+N tags` とし、追加タグ数であることを明示する

この決定により、右 rail は「候補があれば出る補助 UI」ではなく、「Recall という知識再訪導線の定位置」として扱う。empty state も rail の一部であり、UI の不在ではないと定義する。

代替案として、候補 0 件の間は `MiniRecallPanel` を維持し、RecallRail が埋まったら差し替える案もあった。しかしこの案は、ユーザーから見ると右 rail の役割が状態依存で揺れ、RecallRail が機能として常設されていることが伝わらないため採用しない。

## Consequences

### Pros

- feature flag 有効時の右 rail の役割が一貫し、RecallRail を恒常的な導線として認識しやすくなる
- 候補 0 件でも empty state によって「機能は有効だが今は提案がない」ことを表現できる
- static shortcut panel のまま残る違和感を解消できる
- `+N tags` によりタグ圧縮表示の意味が直感的になる

### Cons / Tradeoffs

- 候補 0 件でも右 rail が RecallRail に占有されるため、従来の shortcut 群は feature flag 有効時には前面に出なくなる
- empty state の文言や見た目が弱い場合、右 rail が情報量不足に見える可能性がある
- `+N tags` は英語ラベルであり、日本語 UI に完全統一したい場合は後続で表現ルールを見直す余地がある
