# Orphan Feed の一括解消と feed_link_id NOT NULL 制約の追加

## ADR's STATUS

Accepted

## CONTEXT

[[000213|ADR-213]] で導入した `feeds.feed_link_id` カラムと `user_feed_subscriptions` テーブルにより、ユーザーごとの購読フィルタリングが可能になった。しかし `feed_link_id` は nullable として追加されたため、カラム追加以前に登録されたフィードは `feed_link_id IS NULL`（orphan feed）のまま残存していた。

[[000365|ADR-365]] で shared payload cache を導入した際、orphan feed はキャッシュの `feed_link_id` キーに乗せられないため、以下の fallback が必要だった:

1. **Driver 層**: 全フィード取得クエリに `OR f.feed_link_id IS NULL` 条件を付加（6 箇所）
2. **Driver 層**: `buildExcludeClause` に `split_part` によるドメインベースの代替マッチング
3. **Port/Gateway/Driver**: `FetchOrphanFeeds` 専用の取得経路（インターフェース定義、SQL、変換処理）
4. **Usecase 層**: `CachedFeedListUsecase.loadMergedFeeds` で orphan feed を別取得し merge + sort
5. **Usecase 層**: `feedIDToUUID` に orphan feed 用の legacy path

これらの分岐は全レイヤーに分散しており、コードの複雑度を高め、クエリプランの最適化も阻害していた。

## DECISION MAKING

### 1. 既存 backfill スクリプトによるデータ修復を試行する

`scripts/backfill_feed_link_ids.sh` を実行し、`feeds.link` のドメインと `feed_links.url` のドメインを `split_part` で比較してマッチングを試みた。

結果: orphan feed 8,926 件のうち、ドメインマッチできたものは 0 件だった。orphan feed のドメイン（約 300 ドメイン）は `feed_links` テーブルに対応する RSS ソースが登録されておらず、`feed_link_id` 導入以前に取り込まれた古いデータであった。

### 2. マッチ不能な orphan feed は DELETE で削除する

ドメインマッチ不能な orphan feed は RSS ソースとの紐付けが本質的に不可能（RSS URL が不明）であるため、DELETE で削除した。CASCADE 設定により関連する articles（3,666 件）、read_status（4,205 件）、favorite_feeds（7 件）も連動削除された。

ユーザーが該当フィードを再登録した場合、`RegisterFeedsUsecase` により `feed_link_id` 付きで正しく作成される。

### 3. `feeds.feed_link_id` に NOT NULL 制約を追加する

orphan feed が 0 件になった時点で `ALTER TABLE feeds ALTER COLUMN feed_link_id SET NOT NULL` を適用し、今後の orphan feed 発生を DB レベルで防止した。

### 4. orphan feed に関する全コードを削除する

NOT NULL 制約により orphan feed は存在し得なくなったため、全レイヤーから orphan 関連のコードを削除した。

## RESULTS, EFFECTS

### PROS

- **コード複雑度の大幅削減**: orphan feed の分岐が全レイヤー（Port / Driver / Gateway / Usecase / Mock / Test）から消滅し、フィード取得パスが単一経路に統一された
- **SQL クエリの簡素化**: `OR f.feed_link_id IS NULL` の分岐削除により、PostgreSQL が `Nested Loop + Memoize (Index Only Scan)` を安定して選択するようになった。全クエリ 2ms 以下を確認
- **`buildExcludeClause` の簡素化**: `NOT EXISTS + split_part` によるドメインベースの代替マッチングが単純な `!=` 比較に置き換わり、サブクエリと関数呼び出しのコストが消滅した
- **DB レベルの整合性保証**: NOT NULL 制約により、`feed_link_id` なしのフィードが登録されることを防止
- **キャッシュ層の完全統一**: `CachedFeedListUsecase` が全フィードを `feed_link_id` ベースの `FeedPageCache` 経由で取得でき、orphan 用の別経路マージが不要になった

### CONS, TRADEOFF

- **8,926 件のフィードデータが削除された**: 古い orphan feed とその関連 articles / read_status が失われた。ただし、これらは RSS ソースとの紐付けが不明なデータであり、ユーザーが再登録すれば正しい形で再取得される
- **feed_link_id なしのフィード登録が不可能になった**: `RegisterFeedsUsecase` が `feedLinkIDResolver` で feed_link_id を解決できない場合、INSERT が NOT NULL 制約に違反する。現在の登録フローでは `RegisterFeedLink` → `FetchFeedLinkIDByURL` の順で必ず feed_link_id が解決されるため問題ないが、将来新しい登録経路を追加する際は注意が必要
