# ADR-000626: Knowledge Home の Feature Flag 管理を廃止し全機能を常時有効化する

## Status

Accepted

## Date

2026-04-05

## Affected Services

- alt-backend (Go) — feature flag Gateway の Global Search 依存除去、DI 配線修正
- alt-frontend-sv (TypeScript/SvelteKit) — `useFeatureFlags` hook の無効化、全機能の常時有効化

## Context

Knowledge Home には段階的ロールアウトのための feature flag システムが導入されていた。`KnowledgeHomeConfig` に以下のフラグが存在し、`feature_flag_gateway.Gateway` が `crc32` ハッシュベースのパーセンテージロールアウトとユーザー ID ホワイトリストで制御していた：

- `enable_knowledge_home_page` — Knowledge Home ページ自体の有効化
- `enable_knowledge_home_tracking` — 行動トラッキング
- `enable_knowledge_home_projection_v2` — Projection V2
- `enable_recall_rail` — RecallRail サイドバー
- `enable_lens` — Lens（保存ビューポイント）
- `enable_stream_updates` — リアルタイム更新
- `enable_supersede_ux` — 記事更新の表示

このシステムは Knowledge Home の初期開発において、機能を段階的にリリースするための過渡期の仕組みとして機能した。しかし、全機能が安定稼働に達した現在、以下の問題が顕在化していた：

1. **条件分岐の累積** — フロントエンドの `/home` ページに `{#if recallEnabled}`, `{#if lensEnabled}`, `{#if streamEnabled}` 等の条件分岐が散在し、コードの可読性と保守性を低下させていた
2. **新機能追加時の摩擦** — Global Federated Search（[[000625]]）の追加時に `enable_global_search` フラグと config フィールドの追加が必要になり、実際にフラグ未登録のまま機能が動作しないバグを生んだ
3. **テストの複雑化** — flag ON/OFF の組み合わせテストが必要で、テストコードの量と複雑性が増加していた
4. **Dead code パス** — flag OFF 時のフォールバック（例: `MiniRecallPanel`）が実質的に到達不能コードとなっていた

## Decision

**Knowledge Home の feature flag 管理を全廃し、全機能を常時有効化する。**

feature flag Gateway 自体は他の用途（将来の機能ロールアウト）に備えて残すが、Knowledge Home の全既存フラグについてはチェックを除去し、常時 `true` として動作させる。

### 具体的な変更

#### Backend（alt-backend）

- `GlobalSearchService` の Handler から `feature_flag_gateway.Gateway` 依存を削除。Handler はフラグチェックなしで常にリクエストを処理する
- DI 配線修正：`newSearchModule(infra)` を `NewApplicationComponents` に追加し、`GlobalSearchService` が Connect-RPC サーバーに正しく登録されるようにした（[[000625]] の配線漏れ修正）
- `FlagGlobalSearch` 定数を `domain/feature_flag.go` から削除

#### Frontend（alt-frontend-sv）

- `/home` ページから `useFeatureFlags` の利用を全て除去
- `recallEnabled`, `lensEnabled`, `streamEnabled` を `true` に固定し、条件分岐を除去
- `handleSearchSubmit` を簡素化：常に `/search?q=...` へリダイレクト（ローカルフィルタのフォールバックを削除）
- `visibleItems` の `$derived.by` クライアントサイドフィルタを除去し、`home.items` を直接使用
- `MiniRecallPanel`（RecallRail 無効時のフォールバック）を除去
- `useFeatureFlags` hook を全メソッドが `true` を返す no-op スタブに置き換え
- `stream-refresh.ts` から `recallEnabled` パラメータを削除（常に同期）
- `flags.trackingEnabled` ガードを除去（トラッキングは常時有効）

## Consequences

### Pros

- **コードの簡素化** — `/home` ページから約30行の条件分岐と関連ステート変数が削除され、コードの可読性が向上
- **新機能追加の摩擦低減** — flag 登録・config 追加・Gateway 更新の手順が不要になり、機能追加が直接的になる
- **テスト簡素化** — flag ON/OFF の組み合わせテストが不要になり、テストコードの維持コストが低減
- **DI 配線バグの解消** — Global Search の DI 配線漏れが修正され、`SearchEverything` RPC が正常に動作するようになった

### Cons / Tradeoffs

- **段階的ロールアウト能力の喪失** — Knowledge Home の個別機能を特定ユーザーにのみ公開する手段がなくなった。ただし現時点で全機能が安定稼働しており、この能力の即時的な必要性はない
- **フォールバック UI の喪失** — `MiniRecallPanel` 等のフォールバックコンポーネントが削除されたため、RecallRail が障害時にフォールバック表示を提供しなくなった。RecallRail 自体は `unavailable` prop でエラー状態を表示可能
- **feature_flag_gateway は残存** — Gateway、Port、Config の flag フィールド自体は今回削除していない。Knowledge Home handler の一部で参照が残るため、完全削除は別の作業として実施する
