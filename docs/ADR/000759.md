# ADR-000759: mTLS :9443 を Connect-RPC と REST のハイブリッドにし、pki-agent サイドカーを compose-native cascade に戻す

## Status

Accepted

## Date

2026-04-17

## Affected Services

- **alt-backend** — `app/mtls_handler.go` と `app/mtls_handler_test.go` を新設し、`app/main.go` の mTLS listener を Connect-RPC / REST ハイブリッド handler 経由に差し替え。`Dockerfile.backend` のビルドを `go build ./main.go` から `go build .` (package ビルド) に変更
- **compose** — `compose/pki.yaml` の `pki-agent-acolyte-orchestrator` と `pki-agent-tag-generator` の `depends_on.<parent>` に `restart: true` を追加

## Context

本 ADR は 2026-04-17 に同時発火した 2 件の silent failure を同一の PR で根治する。どちらも「コンテナは `Up (healthy)` のまま機能が止まる」タイプで、`/health` ベースの smoke では検知不可能だった。

### 障害 A: Acolyte `ListReports` 502 再発 ([[PM-2026-030]] regression)

`alt-butterfly-facade` ログ (2026-04-17 08:37 / 09:10 / 09:11 JST) に `Post "https://acolyte-orchestrator:9443/alt.acolyte.v1.AcolyteService/ListReports": dial tcp 172.18.0.35:9443: connect: connection refused` が連発。コンテナ状態の実測:

- `acolyte-orchestrator` container id `0c4cdb3ead31...` (started `2026-04-17T03:57:47Z`)
- `alt-pki-agent-acolyte-orchestrator-1` の `HostConfig.NetworkMode` = `container:6bf12bf8a067...` (started 前日 `2026-04-16T21:58:10Z`)
- 6 時間のズレ = sidecar が **死んだ親の netns を掴んだまま**稼働。sidecar 自身の `/proc/net/tcp6` には `[::]:9443` LISTEN が見えるが、新 netns (172.18.0.35) からは `connect_ex=111` (ECONNREFUSED)

なぜ再発したか:

1. 2026-04-17 06:55 `1c7679801` ([[000757]]) で `scripts/_deploy_lib.sh:deploy_single_service` に pki-agent sidecar の cascading `--force-recreate` を追加 ([[PM-2026-030]] Action Item #3)
2. 同日 `9f93dcd21` ([[000758]]) で `_deploy_lib.sh` ごと c2quay に置換。c2quay は `docker compose up --wait --remove-orphans` を単発で叩くだけで sidecar cascade を知らない
3. [[000758]] Cons 自身が *"pki-agent サイドカーの netns 共有を compose 依存で表現できていない可能性が残る ([[PM-2026-030]] の再発防止を compose 側に完全移植するまで要注意)"* と警告していた
4. 03:57 の c2quay deploy で `acolyte-orchestrator` だけ recreate、sidecar は前日のまま → 幽霊化

[[000757]] Decision 2 (pki-agent healthcheck に TCP dial listener probe) は sidecar プロセス自体の silent death 用で、netns orphan のケースでは sidecar 側 netns から見れば listener は生きているので healthy を返し続ける。listener probe と cascade は補完関係で、後者の穴が今回露出した。

### 障害 B: 3days Recap が 4 日連続 `404 Not Found` で全 job fail

`recap-db` の `recap_job_status_history` で recap job の失敗理由を実測:

| job id | window_days | transitioned_at | reason |
|---|---|---|---|
| 438 | 3 | 2026-04-16 17:00 UTC | `alt-backend returned error status 404 Not Found: 404 page not found` |
| 437 | 3 | 2026-04-15 17:00 UTC | 同上 |
| 436 | 3 | 2026-04-15 12:32 UTC | 同上 |
| 435 | 3 | 2026-04-14 17:00 UTC | 同上 (← 発火点) |
| 425 | 1 | 2026-04-13 04:11 UTC | `dns error` (発火前の別問題) |

alt-backend のリスナー構成 (`alt-backend/app/main.go`):

- `:9000` Echo REST — `registerRecapRoutes` で `/v1/recap/articles` を登録 (`alt-backend/app/rest/recap_handlers.go:24`)
- `:9101` Connect-RPC (`connectv2.CreateConnectServer`)、plain HTTP
- `:9443` mTLS — 本 ADR 前は `middleware.PeerIdentityHTTPMiddleware(connectServer)` のみをラップしていた。REST ルートは**未提供**

`recap-worker` は `MTLS_ENFORCE=true` 下で `ALT_BACKEND_MTLS_URL=https://alt-backend:9443` を使う (`recap-worker/src/pipeline/orchestrator.rs:93`) ため、`GET /v1/recap/articles?from=X&to=Y` は `:9443` の Connect-RPC mux に届き、登録のない REST パスなので 404。

内部疎通テストで裏取り: `docker exec alt-alt-backend-1 wget -qO- "http://localhost:9000/v1/recap/articles?from=...&to=..."` は 372 件返す (`:9000` 側の REST ハンドラは正常)。`:9443` 経由だけが 404。

つまり 2026-04-14 前後の mTLS cutover ([[000754]] 前後の `5d148ce25` `feat(security): cut over to mTLS peer-identity, remove X-Service-Token`、`a6752c19c` `feat(mtls): enable east-west mTLS listeners across Go services`) で REST ルートを mTLS listener に乗せるステップが欠落したのが発火源。`/v1/recap/articles` は 7-day recap (`RecapService.GetSevenDayRecap`) のように Connect-RPC へ移行されておらず、REST のまま `:9000` にしか存在しない。

### 共通の教訓

両方とも「mTLS cutover 系の未完了作業が silent に残った」結果だった。Pact ゲートは contract の互換性しか見ず、sidecar の netns 整合性や listener あたり当たりのルート登録漏れは対象外。smoke は `/health` しか叩かない。この層を埋めずに cutover を完了扱いしていたのが共通の抜け。

## Decision

### 決定 1: alt-backend の mTLS listener `:9443` を Connect-RPC + REST のハイブリッドにする

`alt-backend/app/mtls_handler.go` に `buildMTLSHandler(connectMux, echoHandler http.Handler) http.Handler` を新設。URL パスが `/alt.` または `/services.` で始まるときだけ `connectMux`、それ以外は `echoHandler` に routing する。`main.go` で `PeerIdentityHTTPMiddleware` は mux の外側で一律適用し、peer identity 抽出の挙動は維持する。

```go
var connectRPCPrefixes = []string{"/alt.", "/services."}

func buildMTLSHandler(connectMux http.Handler, echoHandler http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        for _, prefix := range connectRPCPrefixes {
            if strings.HasPrefix(r.URL.Path, prefix) {
                connectMux.ServeHTTP(w, r)
                return
            }
        }
        echoHandler.ServeHTTP(w, r)
    })
}
```

配線は `main.go`:

```go
mtlsHandler := buildMTLSHandler(connectServer, e)
mtlsServer = tlsutil.NewMTLSHTTPServer(fmt.Sprintf(":%d", mtlsPort), tlsCfg, middleware.PeerIdentityHTTPMiddleware(mtlsHandler))
```

TDD ケース (`mtls_handler_test.go`):

1. `TestBuildMTLSHandler_RoutesConnectPathsToConnectMux` — `/alt.feeds.v2.FeedService/Get` と `/services.backend.v1.BackendInternalService/ListUnsummarizedArticles` が connect mux に届く
2. `TestBuildMTLSHandler_RoutesRESTToEcho` — `/v1/recap/articles` が echo に fall through (3days Recap 回帰防止ピン)
3. `TestBuildMTLSHandler_UnmatchedConnectPathDoesNotFallthrough` — Connect-RPC プレフィックスだが未登録メソッドは connect mux が 404 を返し、echo には漏らさない (typo 検知性保全)

### 決定 2: pki-agent sidecar の cascade を compose-native に戻す

`compose/pki.yaml` の `network_mode: service:X` を持つ sidecar 2 件に compose v2.30+ の `restart: true` を追加:

```yaml
pki-agent-acolyte-orchestrator:
  depends_on:
    acolyte-orchestrator:
      condition: service_healthy
      restart: true
    step-ca:
      condition: service_healthy

pki-agent-tag-generator:
  depends_on:
    tag-generator:
      condition: service_healthy
      restart: true
    step-ca:
      condition: service_healthy
```

他の pki-agent sidecar (`alt-backend`, `auth-hub`, `pre-processor`, `search-indexer`, `recap-worker`, `alt-butterfly-facade`) は独立 `alt-network` 上で cert 配布のみ行う別形態で、netns orphan の被害者にならないため対象外とする。Docker 29.4.0 / Compose v5.0 でサポート済を `docker compose version` で確認。c2quay の `docker compose up --wait acolyte-orchestrator` はこの depends_on を尊重し、親 recreate と同時に sidecar も recreate される。

### 却下した代替案

- **recap-worker を `:9000` plain HTTP に戻す** — 東西 mTLS 方針 ([[000754]]) を後退させる。セキュリティ監査の一貫性を崩すので却下
- **`/v1/recap/articles` を Connect-RPC へ即時移行** — 7-day recap ([[000709]] 系) と同じ方向性で正しいが、protobuf 定義・クライアント再生成・テスト広範書換で P0 TTR に合わない。fast-follow の別 ADR に切り出す
- **c2quay 内部に sidecar cascade ロジックを追加** — 外部 OSS (`Kaikei-e/c2quay`) の PR 待ちで非同期。compose 表現で閉じる方が速く、設計上も compose がインフラの正になる
- **`scripts/_deploy_lib.sh` 方式を復活** — [[000758]] の c2quay 正規化が崩れる。compose-native に寄せるのが筋

### 一次復旧 (deploy 前に即時実行)

deploy を待たず手順のみで 502 を止めた: `docker compose -f compose/compose.yaml -p alt up -d --no-deps --force-recreate pki-agent-acolyte-orchestrator`。[[PM-2026-030]] タイムライン 06:49 と同一手順。

復旧確認:

- `docker inspect alt-pki-agent-acolyte-orchestrator-1 --format '{{.HostConfig.NetworkMode}}'` → `container:0c4cdb3e...` (現親コンテナ ID 一致)
- `docker exec acolyte-orchestrator python3 -c "import socket; s=socket.socket(); print(s.connect_ex(('127.0.0.1',9443)))"` → `0`
- BFF 経由 `POST /api/v2/alt.acolyte.v1.AcolyteService/ListReports` → HTTP 502 → 401 (認証未添付の正常な拒否)

これは回避手順であり、再発予防は本 ADR の決定 2 (compose-native cascade) に委ねる。

## Consequences

### Pros

- **3days Recap が今日から再び完走可能**。`/v1/recap/articles` が `:9443` 経由でも 200 を返すので、recap-worker → alt-backend の fetch ステージが抜ける
- **Acolyte 502 の再発を compose の表現だけで予防**できる。shell cascade を捨てた [[000758]] の懸念点を解消
- **tag-generator 側の pki-agent** も同じ事故パターンから先回りで守られる (予防的)
- mTLS listener を通すすべての REST ルート (`/v1/dashboard/*`, `/v1/morning/*`, `/v1/articles/*` など) が自動的に mTLS 対応に。将来 `:9000` を外部閉塞する土台になる
- Connect-RPC プレフィックス未マッチは依然 404 (echo に漏らさない) でタイポ検知性を保全

### Cons / Tradeoffs

- **`:9000` REST と `:9443` REST の 2 本立てが残る**。`:9000` を internal-only に絞る fast-follow を別 ADR で行う前提
- **`connectRPCPrefixes` を文字列配列でハードコード**。新しい proto package が `/alt.` / `/services.` 以外の namespace を使い始めたら拡張が必要 (テストで典拠を固定済)
- [[000757]] Decision 3 (`scripts/_deploy_lib.sh` cascading) は [[000758]] の c2quay 移行で実装消失、本 ADR で compose-native に置換。[[PM-2026-030]] Action Item #3 の "Done" 表記は有効のまま (対策は形を変えて存続)、同 Action Item #7 (手動 `--force-recreate` 経路でも cascade) を本 ADR が完了させる形
- `/v1/recap/articles` が REST のままなのは依然残課題。Connect-RPC 移行は別 ADR で fast-follow
- **スコープ外**: `recap_subworker_runs` に残る 76 件の orphan `running` (2026-03-24 〜 2026-04-13)、`recap-evaluator` の Ollama `/api/tags` → `/health` 切替 (起動時 warning のみ、機能停止とは無関係)、Inoreader 取り込み→alt-db sync の 10h 停止 (別 root cause、別タスク) — 本 ADR では触らない
