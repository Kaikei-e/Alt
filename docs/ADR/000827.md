# ADR-000827: recap-worker の rust-bert embedding 初期化を `RECAP_WORKER_EMBEDDING_REQUIRED` で起動時 fail-closed にする

## Status

Accepted

## Date

2026-04-22

## Affected Services

- **recap-worker (Rust)** — embedding 初期化に policy gate を導入し、Settings validator 相当の起動時 fail-closed を実装。
- **compose/recap.yaml** — 本番デフォルトで fail-closed が有効になる env 値を追加。
- **docs/runbooks/3days-recap-artefact-recovery.md** — populate 未実施 host を新 validator が refuse しても運用者が即復旧できるよう、warmup subcommand 手順を runbook §ブロッカー 1 に焼き込む。

## Context

2026-04-22 の当日、[[PM-2026-036]] / [[PM-2026-037]] で recap-subworker 側の joblib artefact 欠落 silent failure を解消した直後にユーザーから別軸の品質劣化報告が入った: 「3days Recap のジャンル生成精度とレポート品質がものすごく悪い」。調査で以下が確定した。

- `alt-recap-worker-1` container ログ冒頭 (起動時):
  ```
  Token file not found "/opt/rustbert-cache/huggingface/token"
  Embedding service failed to initialize. Large genres cannot be split into subgenres.
  Falling back to keyword-only filtering.
  error: Endpoint not available error: An IO error occurred
  Caused by: Read-only file system (os error 30)
  ```
- host `/opt/rustbert-cache`: `total 8` (空ディレクトリ) / `/dev/nvme0n1p2 on /opt/rustbert-cache type ext4 (ro,relatime)`。container への bind は `compose/recap.yaml` で `:ro` 固定。
- pipeline 実挙動 (直近 3 ジョブ): 30 ジャンル taxonomy のうち `consumer_tech` (10 clusters) と `politics_government` (7 clusters) の 2 バケットのみ subworker run が成功、残り 25-28 ジャンルは `genres_no_evidence` で空セクション化。最新ジョブは `genres_stored: 2 / total_genres: 30`。
- `recap_final_sections` テーブル: 過去 7 日 0 行。ユーザーが見ていたレポートは 2026-04-13 以前の stale 状態。

真因は **docker bind-mount 空ディレクトリ footgun + silent fallback** の合わせ技。recap-worker の Dockerfile は直近 3 日 (2026-04-19 〜 2026-04-22) で image 内から rust-bert キャッシュを除外し `/opt/rustbert-cache` への host bind に分離した (commit `3bc849f99` / `d7153ad36` / `f2fff67b5`)。warmup subcommand (`recap-worker warmup`) は同時に導入されたが、**prod host で populate するタイミングが host provisioning runbook に明記されていなかった**。`docs/runbooks/3days-recap-artefact-recovery.md` §"ブロッカー 1" は `mkdir -p + chown 999:999` までしか記述せず、結果として **空のディレクトリを `:ro` で bind する状態** が現場で成立した。

`pipeline/orchestrator.rs:155-171` の従来実装:

```rust
let embedding_service: Option<Arc<dyn Embedder>> = match EmbeddingService::new() {
    Ok(s) => { tracing::info!(...); Some(Arc::new(s) as _) }
    Err(e) => {
        tracing::error!(error = %e, "Embedding service failed to initialize. \
            Large genres cannot be split into subgenres. \
            Falling back to keyword-only filtering.");
        None
    }
};
```

init 失敗を `error!` ログだけ残して `None` に丸めていたため、下流の `subcluster_large_genres` が「embedding 不在の小ジャンルは元のまま」という明示的パスに落ち、**ジャンル分類ほぼ全体がキーワード一致だけで動く degraded mode に silent に遷移** していた。これは [[PM-2026-036]] で仕留めた recap-subworker 側の「is_file guard 抜け → IsADirectoryError」と同形の footgun が、recap-worker 側では「init 失敗を swallow する Option」で顕在化していたもの。監視・Prometheus・`/health` は何も発火せず、ユーザーの目視品質劣化報告が唯一の検知経路になった。

[[PM-2026-036]] / [[000825]] で確立した **"Settings validator で起動時 fail-closed"** パターンを recap-worker 側にも対称に適用するのが本 ADR の趣旨。

## Decision

### 1. `EmbeddingAvailability` policy enum と純関数 `require_or_degrade` を追加

`src/pipeline/embedding.rs` に以下を追加:

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum EmbeddingAvailability { Required, Optional }

pub fn require_or_degrade<T, E>(
    result: std::result::Result<T, E>,
    policy: EmbeddingAvailability,
) -> std::result::Result<Option<T>, E> {
    match (policy, result) {
        (_, Ok(v)) => Ok(Some(v)),
        (EmbeddingAvailability::Required, Err(e)) => Err(e),
        (EmbeddingAvailability::Optional, Err(_)) => Ok(None),
    }
}
```

純関数として分離することでテストが rust-bert 本体の model loading を伴わず、Err/Ok 両ケースを網羅できる。

### 2. `Config::embedding_required()` の追加

`src/config.rs` に `FeatureToggle` ベースのフィールドを追加、env `RECAP_WORKER_EMBEDDING_REQUIRED` を `parse_bool` で読み込む (default `false`、本番 compose は `true`)。`Enabled` なら Required、`Disabled` なら Optional に写像。

### 3. orchestrator で fail-closed を適用

`ComponentRegistry::build` 内で `EmbeddingService::new()` を 1 度呼び、Err の場合は env 値に応じた hint 文字列付きで `tracing::error!` を出した上で `require_or_degrade` を通して Result 型を policy 決定させる。Required + Err のときは `anyhow::Context` で runbook 参照付きの context を足して `?` で propagate。`ComponentRegistry::build` の caller (`main.rs`) は既に `?` chain で起動エラーを `anyhow::Error` として親関数に返しているため、**container は非ゼロ終了し compose の `restart: unless-stopped` が reconciler ループに入る** (=healthy にならない)。これは [[000727]] の `load_server_tls_config` が "fail-closed; refusing to start" で行っていたのと同じ意味論。

### 4. compose 本番デフォルトを `true` に振る

`compose/recap.yaml` の recap-worker サービスに以下を追加:

```yaml
- RECAP_WORKER_EMBEDDING_REQUIRED=${RECAP_WORKER_EMBEDDING_REQUIRED:-true}
```

dev stack で populate を持たない環境は `.env` で `RECAP_WORKER_EMBEDDING_REQUIRED=false` を override すれば従来の degraded mode で起動できる。

### 5. runbook §ブロッカー 1 に populate 手順を追記

`docs/runbooks/3days-recap-artefact-recovery.md` の該当節を「ステップ 1: ディレクトリ作成」「ステップ 2: warmup subcommand で model cache populate」の 2 段に拡充。ステップ 2 は現行 image sha を引いて `sudo docker run --rm --network=host -v /opt/rustbert-cache:/opt/rustbert-cache ... $IMAGE warmup` の 1 行ワンライナー。検証コマンドも populate 済か判定可能な形 (`find /opt/rustbert-cache -type f -size +1M`) に差し替え。

### 検討した代替案と却下理由

- **`:ro` bind を `:rw` に変え container 起動時に lazy download を許可**: populate を毎回実施することで最も robust だが、(a) 起動時間に 1-2 分のオーバーヘッドが恒常的に乗る、(b) huggingface への egress が production 経路から必要になり `compose.staging.yaml` の `internal: true` 原則に反する、(c) 現行 `/opt/rustbert-cache` の ext4 ro mount 制約を上書きできない (host 側 fs の制約)。**却下**。warmup は deploy host で事前に一度だけ叩くのが `feedback_single_machine_architecture` と整合する。
- **image に model を焼き込む**: 一番シンプルだが image size が +130 MB になる。ADR-000769 で確立した「rust-bert 系は image baking しない」方針 (Dockerfile slim 化路線) と相反。**却下**。
- **env flag を持たず無条件に fail-closed**: `subcluster_large_genres_handles_no_embedding_service` 等の既存テストと dev stack の keyword-only fallback 経路を維持するため、env で切り替え可能にした。本番デフォルトは `true` なので意味論は「無条件 fail-closed」と同等で、dev 側は override で degrade モードを保持できる。

## Consequences

### Pros

- **silent quality degradation の物理的遮断**. 空 `/opt/rustbert-cache` で起動した瞬間に container が exit 1 して `unhealthy` になる。[[PM-2026-036]] と同等の「外形 healthy + 分類品質だけ劣化」パスを recap-worker 側でも塞ぐ。
- **既存テストを壊さない**. `EmbeddingAvailability::Optional` を明示的 policy として残したため、dev/test 想定の `subcluster_large_genres_handles_no_embedding_service` (select.rs:523) 等は今まで通り pass。
- **テスト容易性**. policy 決定ロジックを純関数 `require_or_degrade` に切り出したので、rust-bert 本体を呼ばずに 4 つの挙動ケースを unit test で網羅できる。従来は `EmbeddingService::new()` を実際に呼ぶしかなく、CI 上で model loading の可否に依存していた。
- **運用者への hint 改善**. 初期化失敗ログに `RECAP_WORKER_EMBEDDING_REQUIRED` の現在値と runbook パスを載せた。PM-036/037 の AI #8 (error 文言区分化) に沿う改善で、検知者が「何をすれば復旧するか」を log 1 行で掴める。
- **ADR-000825 との対称性**. 既に確立された「Settings validator で起動時 fail-closed」パターンを Rust 側にもミラーした。今後同パターンを次の service に展開する際の reference 実装になる。

### Cons / Tradeoffs

- **新しい env を 1 つ増やす**. 他の `RECAP_` env と同様 `parse_bool` で default 付きなので migration impact は小さいが、`.env.example` / secrets 管理表に追記が必要 (Action Item で tracks)。
- **compose デフォルト `true` は populate 未完了 host で initial deploy を止める**. 設計意図そのもの (fail-closed) だが、runbook を参照しない運用者が混乱する可能性がある。runbook §ブロッカー 1 ステップ 2 の populate を先行実施することが前提。dev workstation で populate を持たない場合は `.env` で override。
- **init 失敗時のログが 2 箇所**. `tracing::error!` と `anyhow::Context` の両方に hint 文字列を書いたので、短期的には冗長。将来 `anyhow::Context` だけに集約する follow-up を検討。
- **policy enum 導入による API 追加**. `embedding.rs` の public API が増えたため crate 内の他 caller が誤用する余地が出る。ただし現状 `ComponentRegistry::build` が唯一の init 呼び出し箇所で、他所では `Arc<dyn Embedder>` を受け取るだけなので実害は無い。将来 `require_or_degrade` を orchestrator 以外で呼ぶ場合は同様の hint log も忘れずに書く規約とする。
- **runbook 手順の image sha が deploy 時点に依存**. `ghcr.io/kaikei-e/alt-recap-worker:$(git rev-parse HEAD | cut -c1-7)` で現行 tag を引く書き方にしたが、CI が push していない commit 上で叩くと image が存在しない。運用者は最新 main がデプロイ済の状態で叩く前提。
