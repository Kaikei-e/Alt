# ADR-000753: Manual deploy script の Up 判定・compose 不在スキップ・pipefail 耐性・skip-verify 時の pact publish を 4 連一括修正する

## Status

Accepted

## Date

2026-04-16

## Affected Services

- `scripts/_deploy_lib.sh` — `wait_until_healthy` の Up 対応、`service_in_compose` 新設、pipefail 下でも動くよう here-string 化
- `scripts/deploy.sh` — `--skip-verify` パスで pact publish と pacticipant version seed を実行
- `tests/scripts/test_deploy.sh` — docker stub に `config --services` 応答を追加し、上記変更の regression を覆う

## Context

[[000752]] で Pact provider selector を `mainBranch + deployedOrReleased` に寄せ、bootstrap のために `./scripts/deploy.sh --skip-verify production` を叩いた。この bootstrap run が deploy スクリプトに残っていた 4 件の潜在バグを一気に顕在化させた。全ては「今まで手動 `docker compose up --build -d` しか使っておらず、設計された `scripts/deploy.sh` フルパスが初めて通ったら壊れていた」という [[000750]] と同種の状況に該当する:

1. **Up が running と認識されず healthcheck timeout**: `wait_until_healthy` は `grep -qE '\brunning\b'` だけを accept していた。`docker compose ps` のテーブル出力は healthcheck 未定義サービスで `Up 2 minutes` を返すため、`recap-subworker` のような healthy なサービスが 120s タイムアウトで fail 判定 → 自動ロールバックが発動。
2. **compose に定義されていない pacticipant で `no such service` → 即 fail**: DEFAULT_LAYERS に `tts-speaker` が含まれているが、実体は `compose.tts.yaml` （別 GPU ホスト）にあり、このホストの `compose/compose.yaml` には存在しない。`docker compose up -d --force-recreate tts-speaker` が `no such service` を返し、再び自動ロールバック。
3. **`set -o pipefail` + `grep -q` の SIGPIPE でサービスが誤スキップ**: #2 を塞ぐため導入した `service_in_compose()` を `docker compose config --services | grep -qx -- "$svc"` と書いたところ、`grep -q` が一致時点で終了 → 上流の `docker compose` に SIGPIPE → `set -o pipefail` がパイプラインを非 0 にするため、マッチしているのに関数が false を返す事象が発生。alt-backend / step-ca / search-indexer 等、**出力の先頭側にある服務だけ** が「not defined on this host」と判定されて rolling recreate を丸ごと skip。経験的にマッチ位置が後ろにあるサービス（pre-processor / recap-evaluator）だけが偶然 recreate された。
4. **`--skip-verify` でデプロイした結果 `record-deployment` が 404 で全滅**: `--skip-verify` は `pre-deploy-verify.sh` をスキップするので、その副作用である `pact-check.sh --broker` も走らない → 現在の SHA で broker に pact / version が publish されない → 最終ステップの `pact-broker-cli record-deployment --version $SHA` が 404 で 14 件全滅し exit 12。

#4 を踏んだ時点で broker matrix と現実 deploy が乖離しかかり、ADR-000752 の bootstrap そのものが完結しない状況だった。

## Decision

4 件を同一 ADR で束ねて直す（全て 1-2 行の最小変更で、相互に絡むため分けると意味が薄い）:

### D-1 Up を running と等価に扱う (`scripts/_deploy_lib.sh`)

```diff
- if echo "$output" | grep -qE '\brunning\b'; then
+ if echo "$output" | grep -qE '\brunning\b|\bUp\b'; then
```

`docker inspect` が返す `running` と、`docker compose ps` のテーブル出力の `Up <duration>` の両方を許容する。healthcheck が定義されていないサービスに 120s タイムアウトを課さない。

### D-2 compose に不在のサービスは recreate をスキップ、record-deployment は実行

`service_in_compose()` ヘルパを新設し、`docker compose config --services` に対象サービス名が含まれない場合は:

```text
[recreate] tts-speaker
  not defined on this host — skipping recreate (record-deployment still runs)
```

と明示ログを出して return 0。外部ホストで動く pacticipant（`tts-speaker` が典型）が DEFAULT_LAYERS に混ざっていても、レイヤ rollout を破壊しない。`record-deployment` 側は全 14 pacticipant に対して変わらず叩くので、broker matrix は正しく追従する。

### D-3 `service_in_compose` を pipefail 耐性にする

```bash
# before (broken under `set -o pipefail`)
"${DOCKER_BIN:-docker}" compose -f "$COMPOSE_FILE" config --services 2>/dev/null | grep -qx -- "$svc"

# after
local services
services="$(... config --services 2>/dev/null)" || return 1
grep -qx -- "$svc" <<<"$services"
```

パイプを解消して出力を先に変数に読み取ってから grep する。SIGPIPE による上流の非 0 終了が `pipefail` で関数全体の非 0 に化ける問題がなくなる。

### D-4 `--skip-verify` でも pact publish と pacticipant version seed を実行 (`scripts/deploy.sh`)

`--skip-verify` は can-i-deploy ゲートのバイパスであり、broker への pacticipant version 登録まで止める意図はない。`record-deployment` は「version 行が broker に存在する」前提で動くため、ゲートスキップ時も以下を最低限走らせる:

- `alt-backend/pacts/*.json` / `pacts/*.json` / `rag-orchestrator/pacts/*.json` を PUT で publish
- 各 consumer を `pacticipants/<name>/branches/main/versions/<VERSION>` で main branch に紐付け
- pact を持たない provider-only pacticipant も含めて、DEPLOY_PACTICIPANTS 全件について branch version を PUT

publish 自体の失敗は非 fatal にする（エスケープハッチ経路なので、公開失敗よりも deploy 完了を優先する）。

### 検討した代替案

- **deploy 中に compose ps の出力 format を `json` などに統一して Up/running 揺れを消す**: 確実だが、既存のテスト stub もまとめて書き換えが必要で、また docker compose のバージョン依存が深まる。`grep` ひとつの正規表現拡張の方が変更量が小さい。
- **DEFAULT_LAYERS から `tts-speaker` を外す**: pacticipant として broker matrix には残したいので、DEFAULT_LAYERS からの除外は `record-deployment` の対象から漏れる副作用を生む。`DEPLOY_PACTICIPANTS` と DEFAULT_LAYERS を別系統で扱う D-2 の方が意味論が正しい。
- **pipefail を `set +o pipefail` で一時的にオフにする**: スクリプト全体で pipefail を維持したい（他所で pipe 越しのエラーを黙殺すると致命傷）ので、関数局所で pipe 自体を解消する D-3 を選んだ。
- **`--skip-verify` 時は `record-deployment` 自体をスキップ**: matrix と現実の整合が壊れたまま放置されるので不採用。ADR-000750 が fail-fast 化した `record-deployment` の意義を損なう。
- **pact-check.sh に `--publish-only` モードを足し、`--skip-verify` から呼び出す**: provider verification も走らせるためテスト環境と実行時間のコストが増える。`--skip-verify` は "emergency fast path" なので、HTTP PUT で pact を並べるだけの最小実装 D-4 にした。pact-check.sh の機能拡張は必要になった時点で別途検討する。

### TDD 検証

- `bash tests/scripts/run.sh` で D-1 / D-2 / D-3 の regression ケースを走らせる
  - docker stub に `config --services` 応答を追加（`tc_recreates_services_in_layered_order` / `tc_rolls_back_on_healthcheck_failure` / `tc_records_deployment_on_success` / `tc_skip_verify_bypasses_gate`）
  - 計 6/6 pass を本 ADR 執筆前に確認

## Consequences

### Pros

- ADR-000750 で設計した `scripts/deploy.sh` が、**実運用の初回フルパス通過で露呈した 4 件のバグを束ねて解消し、単一ホスト production に 14 pacticipant を baseline として seed できる状態**に達した。次回以降は通常モード（`./scripts/deploy.sh production`）で can-i-deploy ゲートが期待通り通る前提が成立する。
- healthcheck 未定義サービス（`recap-subworker` など）が rolling recreate でタイムアウト扱いされなくなる。不要な全スタック rollback が減り、変更コストが下がる。
- `DEFAULT_LAYERS` に外部ホスト pacticipant を含められるので、将来 GPU / 別リージョンで動くサービスを追加するときに broker matrix 側だけ先に登録して rollout は別ホストで、という運用が自然に回せる。
- `--skip-verify` が「broker 記録だけは正しく残す」意味を取り戻した。ADR-000750 の `record-deployment` fail-fast 化と整合。

### Cons / Tradeoffs

- **本 ADR は deploy script の穴を塞いだだけで、Pact 的な正攻法（webhook 駆動の deployed-provider 再検証）を実装したわけではない**。[[000752]] の Cons にある通り、provider コードが変わるタイミングでの再検証は依然として `--skip-verify` bootstrap か次回 provider verification run 待ちが必要。別 ADR でのフォローアップが必須。
- `service_in_compose` が `docker compose config --services` を deploy 中に 15 回前後呼ぶ。1 回 0.8 秒程度なので 10-15 秒のオーバーヘッド。compose 定義が巨大化すると無視できなくなる可能性あり。キャッシュ化（layers ループ前に 1 回取得して配列保持）は本 ADR のスコープ外に置いた。
- D-4 の pact publish は `curl -fsS` で HTTP PUT を個別に叩く素朴実装。`pact-broker-cli publish <dir>` を使うより低機能。将来的には CLI に寄せる方が broker 側のバージョンアップに強い。
- 4 バグを 1 ADR にまとめたので、個別の bisect をするときは git log (`fe48a364d` / `6b16b6a2f` / `1f37d06f5` / `49333835e`) を辿る必要がある。commit は分けているので支障はないが、ADR だけ読むと粒度が粗く見える点は認識しておく。
