# ADR-000405: Admin ロールを認証パイプライン全体に伝搬する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- auth-hub (Go) — Kratos Identity からの role 抽出、JWT claim への埋め込み、validate ヘッダー追加
- alt-butterfly-facade (Go) — service_secret の Docker Secret 受け渡し
- alt-frontend-sv (SvelteKit) — `userRole` の layout data 伝搬、サイドバー条件表示

## Context

Knowledge Home Admin ダッシュボード（[[000404]]）は運用者向けの機能であり、一般ユーザーには見せたくない。しかし、Alt の認証パイプラインにはロールの概念が貫通していなかった。

**改善前の状態:**

- auth-hub は Kratos の Identity traits から `email` のみを抽出し、`role` は常に `"user"` をハードコードしていた
- JWT の `role` claim も `"user"` 固定
- validate ハンドラが返すレスポンスヘッダーにロール情報がなかった
- BFF に `SERVICE_SECRET` が渡されておらず、Admin API（`ServiceAuthInterceptor` で保護）への中継ができなかった
- フロントエンドにはユーザーロールの概念がなく、すべてのメニュー項目が全ユーザーに表示されていた

**要件:**

- Kratos Identity の `traits.role` が `"admin"` の場合にのみ admin ロールとして認識する
- ロール情報を認証パイプラインの全段（Kratos → auth-hub → JWT → BFF → フロントエンド）で伝搬する
- フロントエンドで admin ロールに基づくサイドバー表示制御と、サーバーサイドでのページアクセス制御を行う

## Decision

**Kratos Identity traits の `role` フィールドを auth-hub で抽出し、JWT claim・validate ヘッダー・フロントエンド layout data として認証パイプライン全体に伝搬する。**

### 変更の流れ

```
Kratos Identity traits.role = "admin"
    ↓
auth-hub/client: Identity struct に Role フィールド追加、Whoami で抽出
    ↓
auth-hub/cache: SessionCache に Role 保存（Set の可変長引数で後方互換維持）
    ↓
auth-hub/token: IssueBackendToken で normalizeRole() 経由で JWT claim に埋め込み
    ↓
auth-hub/handler: session_handler / validate_handler で Role を伝搬
                  validate_handler は X-Alt-User-Role ヘッダーも追加
    ↓
alt-butterfly-facade: SERVICE_SECRET を Docker Secret 経由で受け取り、
                      Admin API への中継が可能に
    ↓
alt-frontend-sv: +layout.server.ts で getUserRole() → PageData.userRole
                 サイドバーで admin カテゴリの条件表示
                 Admin ページの +page.server.ts でロールチェック
```

### 設計判断

1. **`normalizeRole()` による allowlist 方式** — `role` フィールドには任意の文字列が入り得るが、`"admin"` 以外はすべて `"user"` にフォールバックする。将来 `"editor"` 等を追加する場合はこの関数を拡張するだけで済む。

2. **SessionCache の後方互換** — `Set()` の `roles ...string` 可変長引数により、既存の呼び出し箇所を壊さずに Role を追加できた。

3. **BFF への SERVICE_SECRET 追加** — `compose/bff.yaml` に `SERVICE_SECRET_FILE` と `service_secret` シークレットを追加。これにより BFF が `ServiceAuthInterceptor` で保護された Admin API を中継可能になった。

4. **フロントエンドでの二重防御** — サイドバーの表示制御（UX）とサーバーサイドの `+page.server.ts` での redirect（セキュリティ）を両方実装。URL 直打ちでのアクセスも防ぐ。

5. **`getUserRole()` の独立関数化** — Identity の traits 構造への依存を1箇所に集約し、テスト可能にした（`user-role.test.ts` で null / 空 / 不正値のケースをカバー）。

## Consequences

### Pros

- **E2E のロール伝搬** — Kratos → auth-hub → JWT → BFF → フロントエンドまで一貫してロールが伝わる
- **後方互換** — 既存のユーザーは `traits.role` が未設定であり、すべて `"user"` にフォールバックする。既存動作への影響ゼロ
- **Admin API の中継** — BFF に SERVICE_SECRET が渡ることで、Admin API の呼び出しチェーンが完成した
- **二重防御** — フロントエンドの表示制御 + サーバーサイドの redirect により、admin ページへの不正アクセスを防止

### Cons / Tradeoffs

- **Kratos traits 依存** — ロールを Kratos の Identity traits に格納する方式は、Kratos のスキーマに依存する。将来 Kratos を別の IdP に置き換える場合、traits の読み替えが必要
- **キャッシュ有効期間中のロール変更** — SessionCache の TTL 内にロールが変更された場合、キャッシュが切れるまで旧ロールが使われる。Admin 権限の付与・剥奪は即時反映されない（キャッシュ TTL は 5 分）
- **`"admin"` のみサポート** — 現時点では admin / user の 2 値。RBAC の粒度が必要になった場合は別途設計が要る
