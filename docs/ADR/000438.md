# ADR-000438: projector checkpoint ベースの digest freshness 指標を導入する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go)
- alt-frontend-sv (TypeScript/Svelte)

## Context

Knowledge Home の TodayBar は projector が投影した `today_digest_view` のデータを表示するが、projector が停止・遅延した場合にデータが古くなっていることをユーザーに伝える手段がなかった。

Phase 0〜2 で projector heartbeat は `knowledge_projection_checkpoints.updated_at` を更新する仕組みが整っているが、Home API はこの freshness 情報を返していない。結果として「今日は対象がない」と「データが古い」が UI 上で同じように見え、disabled / unavailable / stale の区別がつかない状態だった。

## Decision

**`knowledge_projection_checkpoints.updated_at` を source of truth として digest freshness を算出し、proto 経由で frontend に返す。**

### Proto 変更

`TodayDigest` に 2 フィールドを追加:

```protobuf
message TodayDigest {
  // ... existing fields 1-8 ...
  string digest_freshness = 9;   // "fresh" | "stale" | "unknown"
  string last_projected_at = 10; // RFC3339
}
```

### Backend 実装

1. **Domain**: `TodayDigest` に `DigestFreshness string` と `LastProjectedAt *time.Time` を追加。`ComputeFreshness(now)` メソッドで閾値判定
2. **Port**: `GetProjectionFreshnessPort` を `today_digest_port` に定義。projector 名で `knowledge_projection_checkpoints.updated_at` を返す
3. **Usecase enrichment**: digest 取得後に projector checkpoint の `updated_at` を取得し、`ComputeFreshness()` で freshness を算出
4. **Handler**: `DigestFreshness` と `LastProjectedAt` を proto にマッピング

### Freshness 閾値

```go
const FreshnessThreshold = 5 * time.Minute

func (d TodayDigest) ComputeFreshness(now time.Time) string {
    if d.LastProjectedAt == nil || d.LastProjectedAt.IsZero() {
        return "unknown"
    }
    if now.Sub(*d.LastProjectedAt) > FreshnessThreshold {
        return "stale"
    }
    return "fresh"
}
```

閾値 5 分は projector の通常処理間隔（5 秒 ticker）に対して十分なマージンを持ち、かつ projector 再起動時の短い空白期間では stale にならない値として設定した。定数化してあり、将来的な調整は容易。

### Frontend 表示

TodayBar の action shortcuts 行に stale indicator を追加:

- `fresh`: 表示なし
- `stale`: amber の Clock アイコン + "Stale" テキスト（`title="Data may be outdated"`）
- `unknown`: 表示なし（断定しない）

**重要**: stale は availability を上書きしない。`weeklyRecapAvailable=true` かつ `digestFreshness="stale"` の場合、Recap リンクは活性のまま stale indicator が表示される。

### Enrichment failure の扱い

checkpoint query が失敗した場合、`DigestFreshness = "unknown"` にフォールバックし、UI は stale indicator を表示しない。

### 検討した代替案

| 案 | 評価 |
|---|---|
| projector heartbeat で全ユーザーの digest を更新 | 書き込みコストが高く、projector の責務を超える |
| frontend が `generated_at` と現在時刻を比較 | client 時計のずれに依存し不正確 |
| `today_digest_view` に `last_projected_at` 列を追加 | Phase 3 方針「加算列を増やさない」に反する |
| **projector checkpoint を request 時に読む** (採用) | 既存テーブルを活用し、新列なし、request 時導出で reproject-safe |

## Consequences

### Pros

- projector 停止時にユーザーが「データが古い可能性がある」ことを認識できる
- `today_digest_view` に新列を追加しないため migration rollback が不要
- stale と unavailable が明確に区別され、UX の曖昧さが解消される
- 閾値が定数化されており、テストで固定しやすい

### Cons / Tradeoffs

- 毎リクエストで `knowledge_projection_checkpoints` を 1 回読む追加コスト。ただし PK lookup なので O(1) で実質的な影響は無視できる
- freshness は projector 全体の checkpoint であり、特定ユーザーの digest 更新時刻ではない。ユーザー A の digest が最新でも projector 全体が stale なら stale 表示になる。これは意図的な設計で、「システム全体の鮮度」をユーザーに伝える
