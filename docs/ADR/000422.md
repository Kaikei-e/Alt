# ADR-000422: Knowledge Home のパフォーマンス・ストリーム安定性・コントラクトを強化する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — インデックス最適化、ストリーム安定化、degraded mode 3 段階化
- alt-frontend-sv (TypeScript) — DegradedModeBanner 3 段階化、コントラクトテスト、Admin UI 拡張

## Context

Knowledge Home は Phase 4 で RecallRail・Lens・Stream Updates・Supersede UX を実装したが、以下の品質課題が残っていた:

1. **パフォーマンス**: `knowledge_home_items` の主クエリがバージョンフィルタリング時にインデックスを活用できない
2. **ストリーム安定性**: 長時間接続での stale 切断がなく、クライアント側でのリソースリークの可能性
3. **Degraded Mode**: bool 1 段階のみで、「一部データが古い」と「完全にフォールバック」を区別できない
4. **コントラクト**: proto 変更時にフロントエンド側での形状整合性を検出する仕組みがない

Phase 5 ではこれらを「hardening」として一括で対処する。

## Decision

### 1. DB インデックス最適化

6 つのインデックスを追加:

| インデックス | 対象テーブル | 目的 |
|-------------|------------|------|
| `idx_khi_user_version_score` | `knowledge_home_items` | バージョン付きメインクエリ最適化 |
| `idx_rcv_user_version_suggest` | `recall_candidate_view` | RecallRail クエリ（部分インデックス） |
| `idx_kue_user_occurred` | `knowledge_user_events` | ユーザー行動クエリ |
| `idx_ke_seq_type` | `knowledge_events` | イベント aggregate lookup |
| `idx_sv_article_latest` | `summary_versions` | 最新サマリー lookup（部分インデックス） |
| `idx_tsv_article_latest` | `tag_set_versions` | 最新タグセット lookup（部分インデックス） |

Atlas の transaction safety 制約により `CONCURRENTLY` は使わない。大テーブルへの適用はオフピーク時に実施する。

### 2. ストリーム安定性強化

- **Stale 切断**: 30 分間の最大接続時間を設定。`stream_expired` イベントと `reconnect_after_ms` ヒント（5-8s ジッター付き）を送信して切断
- **Unary fallback**: 3 回連続のイベント取得エラーで `fallback_to_unary` イベントを送信し、クライアントにポーリングモードへの切替を促す
- **Server-side coalescing**: 5 秒ウィンドウ内の同一 aggregate に対するイベントを最新のみに圧縮し、ワイヤトラフィックを削減

proto に `reconnect_after_ms` (optional int32) フィールドを `StreamKnowledgeHomeUpdatesResponse` に追加。後方互換。

### 3. Degraded Mode 3 段階化

`GetKnowledgeHomeResponse` に `service_quality` (optional string) フィールドを追加:

| 値 | 条件 | UI 表現 |
|----|------|---------|
| `full` | 全 projection 正常 | バナーなし |
| `degraded` | 一部 projection が stale（lag > 5 分）だが応答可能 | 黄色バナー: "Some data may be slightly delayed" |
| `fallback` | projection 完全失敗、キャッシュのみ | オレンジバナー: "Showing cached snapshot. Live updates unavailable." |

既存の `degraded_mode` (bool) は後方互換のために維持し、`service_quality` が存在する場合はそちらを優先する。

### 4. コントラクトテスト

`alt-frontend-sv/src/test/contracts/` に 2 つのテストファイルを追加（22 テストケース）:

- **`knowledge-home-contract.test.ts`**: GetKnowledgeHomeResponse の形状、WhyReason コード taxonomy、SupersedeInfo 状態、ストリームイベントタイプ、RecallCandidate 構造
- **`knowledge-home-admin-contract.test.ts`**: Reproject モード/ステータス値、SLI ステータス値、全体ヘルス値、DiffSummary ラウンドトリップ、ProjectionAudit 構造

既存の `proto-contract.yaml` CI ワークフローが `src/test/contracts/` を実行するため、配置するだけで CI に組み込まれる。

### 5. Admin UI タブ構成

`/admin/knowledge-home` ページにタブナビゲーションを導入:

- **Overview タブ**: 既存パネル（ProjectionStatus, BackfillJobs, FeatureFlags）
- **SLO タブ**: SLOSummaryPanel（5 SLI カード + ErrorBudgetGauge）+ AlertStatusPanel
- **Reproject タブ**: ReprojectActions（開始フォーム）+ ReprojectRunsTable + DiffSummaryPanel

既存ページを壊さず、新規ルートも作らない。タブ切替で段階的に情報を表示。

### 6. Runbook 整備

7 つの運用 runbook を `docs/runbooks/` に追加:

- `knowledge-home-degraded-mode.md` — degraded/fallback 時の調査・復旧
- `knowledge-home-stream-disconnect-surge.md` — stream 切断急増時の対応
- `knowledge-home-reproject-operations.md` — reproject の手順書
- `knowledge-home-empty-spike.md` — 空応答率上昇時の調査
- `knowledge-home-malformed-why-spike.md` — malformed why 検知時の対応
- `knowledge-home-contract-break.md` — proto 破壊的変更時の対応
- `knowledge-home-gameday-checklist.md` — 5 シナリオの GameDay チェックリスト

### 代替案の検討

| 案 | 評価 |
|---|---|
| `CONCURRENTLY` でインデックス作成 | Atlas の transaction safety と相容れない。オフピーク適用で対応 |
| ストリーム切断を Heartbeat 失敗で検知 | サーバー側からの明示的切断 + 再接続ヒントの方がクライアント実装が簡潔 |
| `service_quality` を enum にする | 文字列の方が proto 拡張性が高く、新しい品質レベル追加時に proto 再生成不要 |
| proto に `resume_from` / cursor を追加して差分再開 | Phase 5 のスコープを超える。transport / reconnect / fallback の範囲で閉じる |

## Consequences

### Pros

- バージョンフィルタリング付きクエリのレイテンシが改善
- ストリームの長時間接続によるリソースリークを防止
- 3 段階の degraded mode により、ユーザーへの情報精度が向上
- コントラクトテストにより proto 変更時の形状破壊を CI で自動検出
- 運用 runbook + GameDay チェックリストにより障害対応の属人化を防止

### Cons / Tradeoffs

- 6 インデックスの追加により書き込み性能に微小な影響（読み取り最適化とのトレードオフ）
- 30 分の stale 切断はクライアント側の再接続実装を要求する（reconnect_after_ms ヒントで緩和）
- コントラクトテストは形状のみを検証し、セマンティクス（値の正しさ）は検証しない
