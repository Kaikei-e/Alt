# ADR-000880: Knowledge Loop の article-url-backfill dedupe namespace を v2 に bump し legacy entry の source_url 復旧を append-first で実行する

## Status

Accepted

## Date

2026-05-02

## Affected Services

- **alt-backend (Go)** — `domain.DedupeKeyArticleUrlBackfill` の値を `article-url-backfill:%s` から `article-url-backfill-v2:%s` に bump。`usecase.knowledge_url_backfill_usecase` は const 経由なので自動追従、Loop projector は payload-only なので無変更。

## Context

[[000879]] で Knowledge Loop projector に `ArticleUrlBackfilled` corrective event を消費する分岐を後追い実装した（commit `fc0079387`、rolling deploy 済）。これは `act_targets[0].source_url` が空の `knowledge_loop_entries` に対し、`articles.url` を payload に載せて patch する経路である。

ADR-000879 直後に admin endpoint `EmitArticleUrlBackfill` を発火（2026-05-02 03:00:26Z）したログ:

```
articles_scanned:    29,213
events_appended:     12,091   ← v1 namespace で初出の article 分のみ append
skipped_duplicate:   17,122   ← 既に v1 dedupe registry 登録済 → silent drop
```

`skipped_duplicate` の 17,122 件は **過去に knowledge-url-backfill が走った時点で v1 dedupe key (`article-url-backfill:<id>`) が登録済の article**。これらに対し再 emit を試みても sovereign 側 dedupe registry が「既出」と判定し event row を append せず返すため、Loop projector に新規 event は届かない。Loop projector は payload からしか URL を読まないので、event が来なければ patch も不可能。

DB 観測（2026-05-02 03:10Z）:

```
with_source_url:     ~31,000   (projector 消費後増加中)
without_source_url:  ~22,000+  (本 ADR のターゲット)
total:               52,725
```

未復旧の ~22,000 件は今回の backfill では永遠に `source_url` を貰えない。FE 側では Open ボタンが見た目どおり disable されたまま（commit `fdacb722c` で視覚化済）で、ユーザは記事元を開けない。

### dedupe テーブル直介入の不可

`knowledge_event_dedupes` 等の dedupe registry は **ingest-only barrier** であり projection ではない（[[knowledge-loop-canonical-contract]] §3.8 / `dedupe ≠ projection` 不変条件）。reproject で touch しないのが Alt のイミュータブルデータモデルの根幹で、テーブル直 truncate は `article-created` 等の他系統 event も registry を失い再 emit を許してしまうため副作用が広い。SQL 直 UPDATE による projection 修復は [[feedback_no_sql_logic]] / `Disposable projection` に正面から反する。

## Decision

`DedupeKeyArticleUrlBackfill` の format を `article-url-backfill:%s` から **`article-url-backfill-v2:%s`** に bump し、deploy 後に admin が `EmitArticleUrlBackfill` を再発火する。動作:

- 旧 v1 namespace の dedupe registry 行は touch されない（keep as-is、forward-only）
- 新 v2 namespace では dedupe registry が空なので、29,213 article 全件で event が append される
- Loop projector は payload-only から URL を読み、`act_targets[0].source_url` が欠落している row のみ patch（既存 idempotent guard `NOT (act_targets->0 ? 'source_url')` 維持）

producer 側 const 1 文字変更 (`%s` → `-v2:%s`) のみで全 caller に伝播する。projector は無変更。

### なぜ producer 側で bump するか

不変条件 `Reproject-safe`:
- projector は payload と stable resource のみから projection を再構築できる必要がある
- 同じ event seq の再投影で同じ結果になる必要がある

projector 内で「ArticleUrlBackfilled は既存 source_url があればスキップ」する分岐は既に sovereign 側 SQL に実装済（idempotent）。**問題は producer 側で「同じ article の corrective event を再生成するか」の制御**であり、producer は dedupe registry を参照する以上、namespace を変えない限り再 emit されない。

### 検討した代替案

#### (却下 1) Loop 専用の新 event type を追加

新規 `LoopActTargetUrlBackfilled` を別 dedupe namespace で emit。

- 利点: home 側 dedupe を完全に touch しない
- 欠点: home projector / Loop projector で event 種別が増え、長期的に保守コストが増す。意味的には `ArticleUrlBackfilled` と同等
- 却下理由: 既存 `ArticleUrlBackfilled` は home / loop 両方への corrective として既に semantics を持っており、両 projector が同じ event を消費する設計が clean

#### (却下 2) dedupe registry の該当行を物理削除

`DELETE FROM knowledge_event_dedupes WHERE dedupe_key LIKE 'article-url-backfill:%'`

- 利点: namespace 変更不要
- 欠点: ingest barrier を直接 mutate する → 「dedupe ≠ projection」不変条件と "dedupe is ingest-only" 規約に正面から違反
- 却下理由: 例外処理を運用ルールとして許容すると invariant が骨抜きになる

#### (却下 3) projection に対する SQL 直 UPDATE

cross-DB join で alt-db `articles.url` を sovereign-db に取り込み JSONB patch を直接書き込む。

- 利点: 速い（秒単位）
- 欠点: `Disposable projection` 違反（write path / tracking 以外から projection を直 mutate）。reproject で SQL UPDATE の結果を再現できないため audit / replay が壊れる
- 却下理由: 一度許すと将来の corrective が SQL 直書きに退化する。既に [[feedback_no_sql_logic]] で禁則化されている

#### (却下 4) FE 側で article_id から URL を逆引き

reader 側で `?url=` 不在時に `article_id → URL` を BFF lookup する。

- 利点: backfill 不要
- 欠点: proto `ArticleWithTags` に URL field 追加 + BFF endpoint 追加 + FE wiring。blast radius が同等以上で、しかも projection 復旧の根本問題は残る（Loop entry の `source_url` は空のまま FE が hack で補う構図）
- 却下理由: [[000879]] で却下済の "reader-side lookup" と同じ案。再採用しない

## Consequences

### Pros

- ~22,000 件の legacy `knowledge_loop_entries` の `source_url` 復旧が **append-first** で完結する
- producer 側 const 1 文字 + test の drift 修正のみで対応、projector は無変更
- [[000879]] で確立した append-first / reproject-safe / disposable projection / dedupe ≠ projection の不変条件を全て維持
- 既存 idempotent guard (`NOT (act_targets->0 ? 'source_url')`) により、既に source_url が入っている entry は no-op
- Pact wire shape は不変（`matchers.Like` は string shape match のみ）。consumer / provider verification は v1 / v2 どちらでも PASS
- 将来追加される downstream projector（Loop 以外）も、必要なら同じパターン（v3 bump）で再 emit を局所化できる

### Cons / Tradeoffs

- 旧 v1 dedupe row は登録されたまま（cleanup 不要だが累積）。**dedupe registry は世代を重ねるほど肥大化する可能性**がある。長期的には dedupe TTL / GC 機構を別 ADR で立てる必要がある（提案 §8 Q2）
- backfill emit は同期 endpoint なので 502 timeout 表示が継続発生（UX 上の混乱要因。alt-backend は完走しているが nginx が 30s で打ち切る既知バグ）。本 ADR では range 外、別 issue で扱う（提案 §8 Q3）
- ~29,000 events を全 projector が再消費するため projector 負荷が一時的に増える（5〜30 分）
- `articles.url` が空の article は v2 emit でも `skipped_blocked_scheme` 扱いとなり source_url 復旧不可。実数は post-deploy で計測（提案 §8 Q1）

### 未解決

- dedupe registry TTL / GC の設計（次 ADR で扱う）
- `EmitArticleUrlBackfill` の async 化（30 回再 click の運用負荷削減）
- `articles.url` 空 article への手動 patch / 諦めの判断

## Verification

```bash
cd alt-backend/app
gofmt -l . | grep -v '^gen/'                                  # clean

go test ./domain/ ./usecase/knowledge_url_backfill_usecase/ \
        ./job/ ./connect/v2/knowledge_home_admin/ -race        # all green
CGO_ENABLED=1 go test -tags=contract \
        ./driver/sovereign_client/contract/ -v                 # PASS
```

### Pre-deploy SQL（baseline）

```sql
SELECT COUNT(*) FROM knowledge_loop_entries
WHERE act_targets IS NOT NULL
  AND act_targets->0->>'target_type' = 'article'
  AND NOT (act_targets->0 ? 'source_url');
```

### Post-deploy admin emit

`/admin/knowledge-home` の **Emit ArticleUrlBackfilled** ボタンを `more_remaining: false` まで再 click（page-size 200 / 29,213 article ≒ 30 回）。`POST /api/admin/knowledge-home … 502` は表示バグで実際は完走（[[000879]] 系の既知挙動）。

### Post-projection SQL（admin emit 完走 ~25 分後）

```sql
SELECT
  COUNT(*) FILTER (WHERE act_targets::text LIKE '%source_url%') AS with_source_url,
  COUNT(*) FILTER (WHERE act_targets IS NOT NULL AND act_targets::text NOT LIKE '%source_url%') AS without_source_url,
  COUNT(*) AS total
FROM knowledge_loop_entries;
```

期待: `without_source_url` が ~0 まで減少。残差は `articles.url` 空の article のみ。

### Rollback

bump は forward-only。問題発生時は v3 bump（`article-url-backfill-v3:%s`）で再開、または `runbooks/knowledge-loop-reproject.md` 経由で projection 全体を再生成する。
