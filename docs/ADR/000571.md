# ADR-000571: RAG/FE要約のGPUスロット飢餓を4層修正で解消する

## Status

Accepted

## Date

2026-03-25

## Affected Services

- rag-orchestrator (Go)
- news-creator (Python / FastAPI)

## Context

Ask Augur（RAG回答生成）とフロントエンドのオンザフライ要約が著しく遅延していた。コンテナログの徹底調査により、4層のボトルネックが特定された。

### 計測データ（修正前）

| メトリクス | 値 |
|-----------|-----|
| RAG Chat TTFT | 24s〜115s、300s タイムアウト 3/5件（**60% 失敗率**） |
| Query Expansion | 15s タイムアウト（セマフォ待ち 72s に対してタイムアウト 15s） |
| セマフォ待ち (HIGH PRIORITY) | 最大 72s |
| セマフォ待ち (LOW PRIORITY) | 最大 419s |

### 4層のボトルネック

**層1: AUGUR_EXTERNAL セマフォバイパス**
[[000567]] で Ask Augur を HybridPrioritySemaphore 経由にルーティングする修正が入ったが、`.env` の `AUGUR_EXTERNAL` オーバーライドにより compose デフォルトが上書きされ、rag-orchestrator が引き続き Ollama に直接接続していた。ログ上の URL が `http://news-creator-backend:11435/api/chat`（Ollama 直接）であることから確認。

**層2: Query Expansion レースによるセマフォスロット2重消費**
`expand_queries.go` の `expandQuery()` は、news-creator パスと ollama-legacy パスを goroutine で並行実行するレース設計だった。[[000567]] 以前は ollama-legacy が Ollama に直接接続しており、セマフォ外で動作するためレースに意味があった。しかし AUGUR_EXTERNAL をセマフォ経由に統一した結果、両パスが同じセマフォを叩くようになり、1回の Query Expansion で HIGH PRIORITY スロットを2つ同時消費する状態になっていた。

**層3: QUERY_EXPANSION_TIMEOUT の不足**
rag-orchestrator の `QUERY_EXPANSION_TIMEOUT=15s` に対して、バッチ要約飽和時のセマフォ待ちは 72s に達する。Query Expansion が常にタイムアウトし、フォールバックも同じセマフォで待機するため両方失敗していた。

**層4: Ollama `/api/chat` エンドポイントのハング**
Gemma3 モデルで Ollama の `/api/chat` エンドポイントがハングする問題が存在した。直接 curl でテストしても応答なし。一方 `/api/generate` は正常動作。news-creator の chat proxy が `/api/chat` をそのまま Ollama に転送する設計だったため、セマフォ取得後も応答が返らなかった。

## Decision

4層のボトルネックに対して、それぞれ最小限の修正を適用する。

### 修正1: `.env` の AUGUR_EXTERNAL オーバーライドを削除

`.env` から `AUGUR_EXTERNAL=http://news-creator-backend:11435` を削除し、compose デフォルト `${AUGUR_EXTERNAL:-http://news-creator:11434}` を有効にする。これにより RAG Chat の LLM 生成が news-creator の chat proxy を経由し、HybridPrioritySemaphore の HIGH PRIORITY で GPU スロットを確保する。

### 修正2: Query Expansion の ollama-legacy レースを削除

`expandQuery()` から ollama-legacy goroutine を削除し、news-creator パスのみの同期呼び出しに簡素化した。レース設計はセマフォバイパス時代の名残であり、セマフォ統一後は不要。エラー時は空の拡張クエリで続行する既存の non-fatal 動作を維持。

### 修正3: QUERY_EXPANSION_TIMEOUT を 15s → 30s に引き上げ

バッチ要約がリモートに分散されている通常時は queue_wait ≈ 0s のため 30s で十分。バッチフラッド時の安全マージンとして 15s → 30s に引き上げた。

### 修正4: chat proxy を /api/chat → /api/generate 変換に修正

`ollama_stream_driver.py` の `chat_stream()` を以下のように変更:

1. 受信した `/api/chat` 形式のメッセージ配列を Gemma3 のチャットテンプレート（`<start_of_turn>` タグ）で単一プロンプト文字列に変換
2. `raw=True` を指定して Ollama の `/api/generate` を呼び出し
3. レスポンスの `/api/generate` 形式（`response` フィールド）を `/api/chat` 形式（`message.role` + `message.content`）に変換して返却

upstream（rag-orchestrator）側のコード変更は不要。プロトコル変換は news-creator 内で完結する。

### 検討した代替案

| 案 | 評価 | 却下理由 |
|---|---|---|
| Ollama バージョンアップ | 根本解決の可能性あり | バージョン固定の理由がある場合リスクが高い |
| rag-orchestrator 側で generate 形式に変更 | 影響範囲が大きい | news-creator 内で吸収する方が変更が局所的 |
| QUERY_EXPANSION_TIMEOUT を動的に設定 | 過剰設計 | バッチ分散で queue_wait ≈ 0s なので固定値で十分 |

## Consequences

### Pros

- Ask Augur の応答が **60% 失敗 → 安定成功** に改善（計測値: 39.8s で完了）
- Query Expansion が **15s タイムアウト → 1.25s** に短縮
- セマフォ待ち時間が **72s → 0.0s** に解消
- 全 LLM リクエストが単一の HybridPrioritySemaphore を通過し、GPU 利用状況の可観測性が統一された
- Query Expansion のセマフォスロット消費が 2 → 1 に半減

### Cons / Tradeoffs

- ollama-legacy フォールバックが廃止されたため、news-creator の `/api/v1/expand-query` が唯一の Query Expansion パスになった。news-creator 障害時の冗長性が低下
- `/api/chat` → `/api/generate` 変換でチャットテンプレートをドライバー側にハードコードしている。モデル変更時にテンプレート形式の更新が必要
- `QUERY_EXPANSION_TIMEOUT=30s` は経験則ベースの値。バッチ負荷パターンの変化に応じて再調整が必要な場合がある
