# ADR-000532: Knowledge Sovereign を durable knowledge state の owner として新設し将来的に独立サービス + 独立DBへ抽出する

## Status

Accepted

## Date

2026-03-22

## Context

[[000398]], [[000399]], [[000421]], [[000422]] により、Alt にはすでに Knowledge Home の event/projection/versioning/reproject 基盤が存在する。一方で `involve1` から `involve11_exec_plan` までの検討を通じて、より大きな問題が明確になった。

**Alt には knowledge state の producer は複数あるが、その durable state の最終 owner が明文化されていない。**

現状のコードでは以下が散在している。

- `knowledge_events`
- `knowledge_home_items`
- `today_digest_view`
- `recall_candidate_view`
- `knowledge_projection_versions`
- `knowledge_reproject_runs`
- Lens / dismiss / recall action などの user curation state
- portability/export を担う一部機能（OPML は既存）

また producer は別サービスに分かれている。

- RSS / article ingestion
- summary generation
- tag generation
- recap generation
- RAG retrieval / answer generation
- search indexing

この状態では、Knowledge Home / Recall / Lens / Retention / Export を横断する durable knowledge state の ownership が曖昧なまま拡張される。Alt を「情報を集めるだけのシステム」ではなく、「知識を保持・再発見・持ち出せるシステム」に進めるには、producer と owner を分離し、knowledge state の official write authority を確立する必要がある。

## Decision

### 1. Knowledge Sovereign を新設する

Alt における **durable knowledge state の single owner** として **Knowledge Sovereign** を新設する。

Knowledge Sovereign は次を担う。

- Knowledge Home projection の公式 write path
- Recall の公式 write path
- Lens / dismiss / pin など user curation state の公式 write path
- retention policy の公式適用点
- export/import policy の公式適用点

ここでいう single owner は、単一プロセスではなく **単一の公式 write path** を意味する。

### 2. 初期導入は alt-backend 内の bounded context とする

Knowledge Sovereign はまず `alt-backend` 内の bounded context / logical module として導入する。

初期形態:

- `alt-backend/app/internal/knowledge_sovereign/`
- 既存の `knowledge_projector`, `recall_projector`, retention, export/import policy を段階的に集約
- package 境界 + repository 境界 + usecase 境界で保護

これは最終形ではなく、**将来の物理抽出を前提とした中間形**である。

### 3. 将来的に独立した Knowledge Sovereign service と Knowledge Sovereign DB へ抽出する

Knowledge Sovereign は将来、独立した deployable service と独立DBへ抽出する方針を採る。

将来像:

- `knowledge-sovereign` service
- `knowledge-sovereign-db`
- producer services とは event / API 契約で接続
- durable knowledge state は Knowledge Sovereign DB が authority となる

本 ADR は「今すぐ物理分離する」とは決めないが、**最終的に独立 service + 独立 DB を持つ方向を正式決定する**。

### 4. Knowledge Sovereign が所有するもの

Knowledge Sovereign は、Alt の「あとから残るべき knowledge state」を所有する。

| 対象 | owner |
|---|---|
| `knowledge_events` | Sovereign |
| `knowledge_home_items` 相当の projection | Sovereign |
| `today_digest_view` 相当の projection | Sovereign |
| `recall_candidate_view` 相当の projection | Sovereign |
| `knowledge_projection_versions` / checkpoints / reproject runs | Sovereign |
| recall signal の正規化結果と aggregate | Sovereign |
| Lens / dismiss / pin / explicit curation state | Sovereign |
| retention policy / cutoff policy | Sovereign |
| export manifest / import job / portability policy | Sovereign |

### 5. Knowledge Sovereign が所有しないもの

Knowledge Sovereign は generator ではない。以下は ownership 対象外とする。

| 対象 | owner |
|---|---|
| RSS fetch / ingestion | ingestion 系 |
| raw article CRUD | article / feed 系 |
| summary generation | news-creator |
| tag generation | tag-generator |
| recap generation | recap-worker |
| embedding / retrieval / answer generation | rag-orchestrator |
| search indexing mechanics | search-indexer |
| auth / image proxy / generic infra concerns | 既存 owner |

### 6. retention は article 単体ではなく knowledge state 全体の matrix で扱う

Knowledge Sovereign は retention を `articles.retention_tier` の単一カラムだけで終わらせない。  
entity ごとの retention policy matrix を持つ。

最低限含む対象:

- article metadata
- article raw body
- summary/tag latest
- summary/tag old versions
- recall raw signals
- recall aggregates
- recap result
- user curation state
- export manifest

### 7. export/import は data classification を前提に行う

Knowledge Sovereign は portability の owner でもある。

ただし初期の export/import は full archive を前提にしない。先に data classification を定義する。

- Tier A: 再生成不能で失うと困るもの
- Tier B: 再生成可能だが高コストなもの
- Tier C: 再生成容易なもの

MVP では Tier A 優先で export/import を成立させる。

## Consequences

### Pros

- knowledge state の official owner が明確になる
- Knowledge Home / Recall / Lens / Retention / Export を同じ意思決定軸で扱える
- 将来の独立 service + 独立 DB への抽出に向けて、今のコードをそのまま移行準備できる
- producer と state owner の責務が分離される

### Cons / Tradeoffs

- 初期段階では `alt-backend` 内に新しい bounded context を抱えるため、構造は一時的に複雑になる
- 物理分離までの間は logical owner と physical deployable が一致しない
- migration, replay, reconciliation, rollback を含む移行設計が必須になる

## Implementation Strategy

### Phase 0: 境界の固定

- `knowledge_sovereign` package を作成
- ownership matrix を固定
- retention matrix を定義
- export data classification を定義

### Phase 1: 既存 write path の集約

- `knowledge_projector` を Sovereign write path 配下へ寄せる
- `recall_projector` を Sovereign write path 配下へ寄せる
- Lens / dismiss / curation の write path を集約する
- retention worker と export/import policy を Sovereign usecase として実装する

### Phase 2: 安全な移行基盤の整備

- shadow run
- replay / backfill
- reconciliation
- rollback
- idempotency key
- 差分比較メトリクス

### Phase 3: service / DB 抽出の準備

- transport 契約を整理
- producer 側 direct dependency を削減
- DB ownership を Sovereign 前提に整理
- cutover 条件を明文化

### Phase 4: 独立 service + 独立 DB への抽出

以下の条件が揃った時点で物理抽出を実行する。

- write path が Sovereign に十分集約されている
- replay / rollback / reconciliation が実用水準
- producer との契約が安定している
- 運用上、独立 service の価値がコストを上回る

## Explicit Non-Goals

この ADR は以下を決めない。

1. 直ちに graph DB など新しい DB 種別を導入すること
2. summary/tag/recap/RAG の生成責務を Sovereign に移すこと
3. 今すぐ全面的に物理マイクロサービス化すること
4. RAPTOR や consolidated embedding など研究色の強い仕組みを直ちに採用すること
