# ADR-000758: Pact ゲート付きデプロイ基盤を c2quay に乗せ換える

## Status

Accepted

## Date

2026-04-17

## Affected Services

- **alt-operations** — `scripts/deploy.sh` を c2quay ラッパに置換、`scripts/smoke.sh` と `scripts/record-remote-pacticipant.sh` を新設、`scripts/_deploy_lib.sh` と `scripts/pre-deploy-verify.sh` を削除
- **acolyte-orchestrator** — `acolyte/gateway/news_creator_gw.py` が news-creator 応答のフラットキーを読むように修正、consumer pact (`tests/contract/test_news_creator_consumer.py`) に request body と Pact type matcher を追加
- **news-creator** — `tests/contract/test_provider_verification.py` の `test_verify_acolyte_orchestrator_contract` から `xfail` を除去

## Context

### 従来の辛さ

`scripts/deploy.sh` は Pact `can-i-deploy` ゲート → layered rolling recreate → record-deployment を **shell 188 行** で束ねていた。14 pacticipant ごとに手書きループで Broker と通信し、以下の脆さが残っていた:

- `--skip-verify` 緊急経路で `curl -fsS … || echo "continuing"` により pact file publish 失敗を握り潰していた (`scripts/deploy.sh:75-110`)
- `record-deployment` 段で失敗すると Broker matrix と実デプロイが乖離、exit 12 のまま手動復旧 (`scripts/deploy.sh:179-183`)
- ロールバックは `.deploy-prev` から `compose/` だけ `git checkout` する partial recovery で、アプリコード・secrets・DB schema は未カバー (`scripts/_deploy_lib.sh:121-143`)
- 終了コード 5/10/11/12 の手管理、ファイルロックなし、audit log なし

### c2quay の存在

ユーザーが [Kaikei-e/c2quay](https://github.com/Kaikei-e/c2quay) を "Compose × Pact gate" の定型を Go バイナリに固めるために立ち上げた。初回調査時点 v0.3.0、本 ADR 受理時点 v0.4.8。提供機能:

- YAML (`c2quay.yml`) で環境 → pacticipant マッピングを宣言
- HAL 駆動 Pact Broker クライアントで `can-i-deploy` を並列度 4 でファンアウト
- `docker compose up --wait --remove-orphans` による recreate、`ps --format json` で `--wait` のバグ補償
- `record-deployment` を HAL の `pb:record-deployment` relation で自動 POST
- **flock ベースの環境ロック**・デプロイ前後スナップショット・slog JSON Lines audit log
- Versioning 戦略: `git_sha` / `manifest_file` / `resolved_image_digest`

### 並行して見つかった実 regression

cutover 検証中、`acolyte-orchestrator → news-creator` の consumer pact が provider verification で落ちた。原因は `xfail` コメント (`news-creator/app/tests/contract/test_provider_verification.py:358-367`) が記述していた二重の drift:

1. **レスポンス形状のずれ**: `NewsCreatorGateway.generate` が `data.get("metadata", {}).get("model" | "tokens")` と nested 構造を読んでいたが、news-creator の `SummarizeResponse` は `{success, article_id, summary, model, prompt_tokens, completion_tokens, total_duration_ms}` のフラット構造を返している。acolyte は production で常に `completion_tokens=0, prompt_tokens=0, model=default_model` を受け取り続けていたサイレントバグ
2. **Pact に request body がない**: consumer test が `.with_request("POST", path)` とだけ呼んでいたため、pact file に request body の matcher が記録されず、provider verifier は空 body で POST し news-creator が **422 Unprocessable Entity** を返す

これは単なる contract drift ではなく、実運用で acolyte の LLM コール metadata が全て欠落していた本物のバグ。

## Decision

**`scripts/deploy.sh` を c2quay の薄いラッパに置換し、同時に acolyte ↔ news-creator の regression を TDD で修正する。**

### 実装構造

```
./scripts/deploy.sh production
  └─ [1/3] scripts/pact-check.sh --broker
  └─ [2/3] c2quay deploy --env production --config c2quay.yml
  └─ [3/3] scripts/record-remote-pacticipant.sh production  (tts-speaker 用)
```

`scripts/deploy.sh` は 188 → 36 行。`DEPLOY_PACTICIPANTS` 配列・`--skip-verify` / `--no-record` / `--only` / `--dry-run` のフラグ解析・`.deploy-prev` の書き込み・`_deploy_lib.sh` の layered recreate / `rollback_to_previous` はすべて削除。`c2quay.yml` で 13 pacticipant を `environments.production.services` に宣言、`versioning.strategy: git_sha` + `options.short: true` で既存のローカルビルド (registry push 無し・`git rev-parse --short HEAD` を pacticipant version として共有) に合わせた。

`tts-speaker` は別 GPU ホストに居るため c2quay.yml からは外し、`scripts/record-remote-pacticipant.sh` で `pact-broker-cli record-deployment --pacticipant tts-speaker` を単発で打つ。

### acolyte regression 修正 (TDD RED → GREEN)

1. `acolyte-orchestrator/tests/unit/test_news_creator_gw.py` のモックを news-creator の実応答形状に差し替え
2. `acolyte/gateway/news_creator_gw.py:51-56` を `data.get("summary" | "model" | "prompt_tokens" | "completion_tokens")` のフラット読みに修正
3. `tests/contract/test_news_creator_consumer.py` で `with_body(request_body, "application/json")` を追加、response は `match.string / match.integer / match.decimal` の型マッチャで実値を pin せずに shape だけを契約化
4. `news-creator/app/tests/contract/test_provider_verification.py:358` の `@pytest.mark.xfail` を除去

### 却下した代替案

- **compose `depends_on: condition: service_healthy` 先行整備で layered recreate / pki-agent cascade を c2quay 単発 `docker compose up --wait` に吸収** — 設計としては正しいが、`compose/core.yaml` `auth.yaml` `pki.yaml` `workers.yaml` `ai.yaml` `recap.yaml` `mq.yaml` の広範な書き換えが必要。cutover と同一 PR に混ぜると差分が肥大化するため、先行 PR / 別 ADR に分離
- **c2quay を使わず pact-broker-cli ラッパで shell を整理** — 14 pacticipant ループの脆さが残り、環境ロック・snapshot・audit log が得られない。選ばず
- **acolyte regression を news-creator 側の compat response で吸収** — news-creator は `pre-processor` の既存 contract と両立する単一レスポンス形状を維持する必要があり、acolyte 向けだけ形状を足すと 2 系統の contract を抱える。consumer 側を正に合わせる方向で統一

## Consequences

### Pros

- `scripts/deploy.sh` が **188 行 → 36 行**。フラグ解析・パート数カウンタ・exit code 手管理が消え、`set -euo pipefail` 1 行で fail-fast
- HAL 駆動の集約 `can-i-deploy` (`/matrix?q[][pacticipant]=…&q[][version]=…` interleaved) で **monolithic rollout に耐える**（13 pacticipant を同時に cd2c3499f に進める前提で broker に問える）
- `flock` ベースの環境ロック (`.c2quay/locks/production.lock`)・デプロイ前後スナップショット (`.c2quay/snapshots/…-pre.json` / `-post.json`)・slog JSON Lines audit log が **標準装備**
- acolyte が production で毎回 `completion_tokens=0` を受け取っていたサイレントバグが解消。LLM レイテンシ・トークン計測が正しく記録されるようになる
- Pact consumer の **型マッチャ ベースの契約** が確立し、provider 側の canned response 値 (`gemma4-e4b-q4km` 等) を変えても verification が落ちなくなる

### Cons / Tradeoffs

- **c2quay が Pact Broker の実 HAL 形状に合わせて連続パッチ (v0.3.0 → v0.4.8) を要した事実を記録しておく**:
  1. v0.3.0: `_links.curies` が配列で返る (HAL 仕様) のに `map[string]Link` で受けており JSON decode 失敗
  2. v0.4.2 〜 v0.4.3: 汎用 `pb:can-i-deploy` relation が Broker の root index に無い (実際は `pb:can-i-deploy-pacticipant-version-to-environment` 等のスコープ別 relation のみ)
  3. v0.4.4: 上記修正、ただし `versioning: git_sha` が 40 文字のフル SHA を使用
  4. v0.4.5: `versioning.options.short: true` オプションで短縮 SHA を選択可能に
  5. v0.4.6: `all_or_nothing: true` で aggregate can-i-deploy モードを追加、ただし `pb:can-i-deploy` を探しに行って失敗
  6. v0.4.7: aggregate モードを `/matrix` 直叩きに
  7. v0.4.8: `/matrix` クエリの `q[][pacticipant]=…&q[][version]=…` を **interleaved** に（v0.4.7 は pacticipant 13 連 → version 13 連の順で Rails の `q[]=` 配列パースでペアリングされず 400）
- **Alt 側の contract coverage gap 7 件が顕在化した**。本 ADR では未修正:
  - async message pact (`mq-hub → search-indexer / tag-generator` の Asynchronous/Messages) の provider verification が `pact-check.sh` の `TestVerifySearchIndexerProviderContracts` 等では拾われていない
  - `rag-orchestrator → recap-worker`・`recap-evaluator → recap-worker` の verification 欠落
  - `auth-hub → kratos` は kratos が broker に pacticipant として未登録 (設計判断が必要)
  - これらは cutover とは独立の既存ギャップで、次の ADR / タスクで潰す前提。本 ADR の `c2quay verify` は **7/13 pass、6 fail** を返す状態で受理される
- **rollback は c2quay upstream の `Automatic rollback execution` (Roadmap 項目) 実装待ち**。現行は `git revert` → 再 commit → `./scripts/deploy.sh production` に一本化。旧 `.deploy-prev` からの partial recovery は廃止
- **layered recreate / pki-agent サイドカーカスケード** は本 ADR では c2quay 単発 `docker compose up --wait` に委ね、compose 側 `depends_on: condition: service_healthy` の先行整備は**別 ADR で行う**前提で受理している。それまでは pki-agent サイドカーの netns 共有を compose 依存で表現できていない可能性が残る ([[PM-2026-030-pki-agent-stale-netns-acolyte-502]] の再発防止を compose 側に完全移植するまで要注意)
