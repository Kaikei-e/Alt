# 000347: E2Eテストモックハンドラーにおけるオープンリダイレクト脆弱性の修正 (CWE-601)

## ADR's STATUS

Accepted

## CONTEXT

CodeQL の静的解析で `js/server-side-unvalidated-url-redirection` (CWE-601) が検出された。対象箇所は E2E テスト用の Kratos モックサーバーおよび MSW ハンドラーで、`return_to` クエリパラメータがバリデーションなしで HTTP `Location` ヘッダーに設定されていた。

- `alt-frontend-sv/tests/e2e/infra/handlers/kratos.ts` — Login POST, Registration POST
- `alt-frontend-sv/tests/e2e/infra/msw-handlers.ts` — 同等の MSW ハンドラー4箇所

テストコードとはいえ、以下の理由から修正が必要と判断した:

1. **CodeQL はテスト/本番を区別しない** — CI の品質ゲートでブロックされる
2. **安全なコーディングパターンの一貫性** — テストコードが攻撃パターンのテンプレートになるリスク
3. **OWASP が推奨する許可リスト方式** を適用する好機

### 攻撃ベクターの分析

Node.js の `new URL()` の挙動を実機検証し、以下のエッジケースを特定した:

| 入力 | リスク |
|------|--------|
| `//evil.com` | プロトコル相対URL。`startsWith("/")` を通過するがブラウザが外部にリダイレクト |
| `javascript:alert(1)` | XSS スキーム |
| `http://legit@evil.com` | 認証情報付きURL。origin が `evil.com` になる |
| `data:text/html,...` | data URI スキーム |

特に `//evil.com` は `new URL()` で例外になるにも関わらず `startsWith("/")` が `true` を返すため、単純なパスプレフィックスチェックでは防御できない。

## DECISION MAKING

**許可リスト方式（Allowlist）** を採用し、`validateRedirectUrl` ヘルパー関数を導入した。

### バリデーションロジック

1. `return_to` が `null` → デフォルト URL を返却
2. `new URL()` でパース成功 → `origin` が許可リストに含まれるか検証
3. `new URL()` でパース失敗（相対パス等） → `startsWith("/") && !startsWith("//")` で安全な相対パスのみ許可

許可リストには E2E テストで使用する開発用オリジンのみ含める:
- `http://127.0.0.1:5173` / `http://localhost:5173` — 開発サーバー
- `http://127.0.0.1:4174` / `http://localhost:4174` — プレビューサーバー

### 修正範囲

| ファイル | 修正箇所数 | 内容 |
|---------|-----------|------|
| `handlers/kratos.ts` | 4箇所 | Login POST, Registration POST, Login/Registration browser redirect |
| `msw-handlers.ts` | 4箇所 | 同等の MSW ハンドラー |

計8箇所すべてで `return_to` の取得を `validateRedirectUrl()` 呼び出しに置き換えた。

### 代替案の検討

| 案 | 評価 |
|----|------|
| CodeQL の抑制コメント (`// lgtm[...]`) | 根本対策にならない。却下 |
| 正規表現によるパターンマッチ | バイパス手法が多く保守性が低い。却下 |
| `new URL(returnTo, baseUrl)` のみ | `//evil.com` を防げない。却下 |
| **許可リスト + 相対パス検証（採用）** | OWASP 推奨。最も堅牢 |

## RESULTS, EFFECTS

### PROS

- **CWE-601 の根本解決** — CodeQL のアラートが解消される
- **プロトコル相対URL の防御** — `//evil.com` を明示的に拒否
- **既存テストへの影響なし** — E2E テストは `page.route()` でインターセプトするため、モックハンドラーのバリデーション追加はリグレッションを引き起こさない
- **ログ出力** — `kratos.ts` 側ではブロック時にログを出力し、デバッグ時の追跡が可能

### CONS, TRADEOFF

- **許可リストの保守コスト** — 開発用ポートが変更された場合、`ALLOWED_REDIRECT_ORIGINS` の更新が必要。ただしテストインフラの変更頻度は低く、実質的な負担は小さい
- **2ファイルでのコード重複** — `kratos.ts`（Node.js `http.createServer`）と `msw-handlers.ts`（MSW）で同一ロジックを持つ。共通モジュールへの抽出も可能だが、テストインフラの独立性を優先し、現時点では重複を許容した
