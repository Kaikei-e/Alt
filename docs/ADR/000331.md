# フィード登録フローの責務分離と二重RSSフェッチ排除

## ADR's STATUS

Accepted

## CONTEXT

2026-03-06の負荷試験（3000VU / 50req/s / 5レプリカ）でエラー率0%を達成したが、
`feed_register_duration` の中央値が **28.2秒** と極めて高かった。

コード調査の結果、フィード登録の1リクエスト内で **同一のRSS URLを2回フェッチ** していることが
根本原因であると特定した。

### 旧フロー

```
RegisterFeedsUsecase.Execute(ctx, link)
  → [1回目] RegisterFeedLinkPort.RegisterRSSFeedLink(ctx, link)
      → URLセキュリティチェック (SSRF)
      → サーキットブレーカー
      → HTTP: RSSフェッチ (リトライ付き, 60sタイムアウト)
      → RSSパース
      → DB INSERT (feed_links)
  → [2回目] FetchFeedsPort.FetchFeeds(ctx, link)
      → レートリミッター
      → HTTP: 同一URLを再フェッチ
      → RSSパース → []*FeedItem
  → RegisterFeedsPort.RegisterFeeds(ctx, feedItems)
      → DB INSERT (feeds)
```

### Clean Architecture観点の問題

1. **Gateway の SRP 違反**: `RegisterFeedGateway.RegisterRSSFeedLink()` が1メソッドで
   URL検証・サーキットブレーカー・HTTP外部フェッチ・RSSパース・DB INSERTを全て実行していた
2. **Port の抽象化過剰**: `RegisterFeedLinkPort` が `error` のみ返却するため、
   UsecaseがGateway内部のRSSフェッチ結果を再利用する手段がなかった
3. **ISP 違反**: `FetchFeedsPort` に外部HTTPフェッチ（`FetchFeeds`）と
   DB読み取り（`FetchFeedsList` 等8メソッド）が混在していた

## DECISION MAKING

### 新しいフロー（RSSフェッチは1回のみ）

```
RegisterFeedsUsecase.Execute(ctx, link)
  → [1回のみ] ValidateAndFetchRSSPort.ValidateAndFetch(ctx, link) → *ParsedFeed
      → URLセキュリティチェック (SSRF)
      → セマフォ (50並列制限)
      → サーキットブレーカー
      → HTTP: RSSフェッチ (リトライ付き)
      → RSSパース → ParsedFeed{FeedLink, Items}
  → RegisterFeedLinkPort.RegisterFeedLink(ctx, parsedFeed.FeedLink)
      → DB INSERT (feed_links) のみ
  → RegisterFeedsPort.RegisterFeeds(ctx, parsedFeed.Items)
      → DB INSERT (feeds)
  → publishFeedEvents + autoSubscribeUser (fire-and-forget)
```

### 新設・変更ファイル一覧

| 種別 | ファイル | 内容 |
|------|---------|------|
| 新設 | `domain/parsed_feed.go` | `ParsedFeed` ドメインモデル（FeedLink + Items） |
| 新設 | `port/validate_fetch_rss_port/port.go` | `ValidateAndFetchRSSPort` インターフェース |
| 新設 | `gateway/validate_fetch_rss_gateway/gateway.go` | 外部HTTPフェッチ専用Gateway（SSRF + CB + セマフォ） |
| 新設 | `mocks/mock_validate_fetch_rss_port.go` | 新Portのモック |
| 変更 | `port/register_feed_port/register_port.go` | `RegisterRSSFeedLink` → `RegisterFeedLink`（DB専用） |
| 変更 | `gateway/register_feed_gateway/single_feed_link_gateway.go` | DB INSERTのみに簡素化 |
| 変更 | `usecase/register_feed_usecase/feeds_usecase.go` | 新フローでオーケストレーション |
| 変更 | `di/container.go` | `ValidateAndFetchRSSGateway` のDI配線追加 |

### 設計判断の根拠

- **Usecaseがオーケストレータ**: CA原則に従い、Usecaseがデータフローを制御する。
  `ParsedFeed` をドメインモデルとして返すことで、フェッチ結果の再利用が構造的に可能になった
- **各Gatewayは単一責務**: 外部HTTPフェッチと DB INSERTを別Gatewayに分離
- **セマフォ（P2同時実装）**: `ValidateAndFetchRSSGateway` に50スロットのセマフォを追加。
  `FetchArticleGateway` の既存パターン（3スロット）を踏襲しつつ、
  フィード登録はリクエスト単位のため十分な並列度を確保

## RESULTS, EFFECTS

### PROS

- **レイテンシ ~50%削減**: 同一URLの二重フェッチが構造的に排除され、28s → ~14sの改善が見込まれる
- **CA準拠**: Usecaseがデータフローのオーケストレータとして正しく機能する
- **SRP遵守**: 外部HTTP（`ValidateAndFetchRSSGateway`）とDB操作（`RegisterFeedGateway`）が分離
- **同時接続制御**: 50スロットセマフォにより、3000VU負荷時の外部サーバーへの同時接続爆発を防止
- **テスタビリティ向上**: 各Portが明確な責務を持つため、モック作成が単純化

### CONS, TRADEOFF

- **`FetchFeedsPort.FetchFeeds()` の残存**: 登録フローでは不要になったが、インターフェースから
  削除すると他の利用箇所への影響調査が必要（P1で対応予定）
- **`RSSFeedFetcher` インターフェースの所在**: `register_feed_gateway` パッケージに定義が
  残っているが、新Gatewayから参照する形で再利用している。将来的には共通パッケージへの移動を検討
- **P3（非同期化）未実施**: HTTP応答を `feed_links` INSERT時点で返し、RSS取得をバックグラウンド
  ジョブにする構成が理想だが、今回はスコープ外（推定効果: 28s → 数百ms）
