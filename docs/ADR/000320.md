# ADR-000320: Tag Trail リクエスト雪崩の解消 — Svelte 5 $effect ループ修正と singleflight 重複排除

## ADR's STATUS

完了

## CONTEXT

Tag Trail 機能において、1 件の記事に対して毎秒約 30 回の重複ストリームリクエストが発生し、バックエンドおよびメッセージキューに過大な負荷がかかっていた。5 分間のログ調査で以下の数値を確認した。

| 指標 | 値 |
|------|-----|
| バックエンド ストリーム開始ログ | 9,181 回 |
| タグ生成 RPC 呼び出し | 216 回 |
| Redis pool timeout | 24 回 |
| 対象記事数 | 1 件（同一記事への重複） |

根本原因は 2 層の増幅ループであった。

**Layer 1: フロントエンド `$effect` リアクティブループ**

`DesktopTagTrailScreen.svelte` の `$effect` が `articles`、`articleTagsCache`、`loadingArticleTags` の 3 つのリアクティブ変数を暗黙的に追跡していた。`loadArticleTags()` 内で `new Set(...)` や `new Map(...)` を再代入すると新しいオブジェクト参照が生成され、Svelte 5 のファイングレインリアクティビティにより `$effect` が再発火する。この連鎖が毎秒 30 回のストリームリクエストを生成していた。

**Layer 2: バックエンドのリクエスト重複排除なし**

`FetchArticleTagsGateway` に重複排除機構がなく、同時に到着した N 個のリクエストが全て独立にタグ生成 RPC を発行していた。

## DECISION MAKING

### 前提作業: メッセージキューバックログのクリア

蓄積された未処理メッセージを Redis `XTRIM` で手動クリアし、メッセージ処理ワーカーが過去のバックログに追われる状態を解消した。

### 修正 A: フロントエンド `$effect` ループの修正

`$effect` 内のガード条件（キャッシュ確認・ローディング状態確認）を Svelte 5 の `untrack()` でラップした。これにより `$effect` は `articles` 配列の変更のみをリアクティブに追跡し、`loadArticleTags()` 内の状態変更では再発火しなくなる。

```typescript
import { untrack } from "svelte";

$effect(() => {
    if (articles.length > 0) {
        for (const article of articles) {
            if (
                !untrack(() => articleTagsCache.has(article.id)) &&
                !untrack(() => loadingArticleTags.has(article.id))
            ) {
                loadArticleTags(article.id);
            }
        }
    }
});
```

`untrack()` は Svelte 5 が提供するリアクティビティ制御プリミティブで、コールバック内のリアクティブ変数読み取りを依存関係の追跡対象から除外する。`$effect` のトリガー条件を `articles` のみに限定することで、ループの根本原因を排除した。

### 修正 B: バックエンド singleflight による重複排除

`FetchArticleTagsGateway` 構造体に `golang.org/x/sync/singleflight.Group` フィールドを追加した。on-the-fly タグ生成パス（DB タグなし → 記事取得 → RPC 呼び出し → DB 永続化）全体を `sfGroup.Do(articleID, ...)` でラップし、同一 `articleID` に対する同時リクエストを 1 つの実行に集約する。

- `singleflight.Group` はゼロ値で使用可能なため、既存のコンストラクタへの変更は不要
- 最初のリクエストがタグ生成を実行し、同一キーで待機中の後続リクエストはその結果を共有して返される
- 生成完了後はキーが解放されるため、次回のリクエストは新たに生成を実行する

## RESULTS, EFFECTS

### PROS

- **リクエスト雪崩の根本解消**: `$effect` ループが断ち切られ、同一記事への重複リクエストが発生しない
- **バックエンドの防御層追加**: フロントエンド以外のクライアントからの重複リクエストも singleflight で吸収できる（多層防御）
- **メッセージキュー・タグ生成ワーカーの負荷軽減**: 同一記事に対する RPC 呼び出しが 1 回に集約される
- **Redis コネクションプール安定性向上**: 同時接続数の大幅削減により pool timeout が解消される

### CONS, TRADEOFF

- **singleflight のエラー共有**: 最初のリクエストがエラーを返した場合、同一キーで待機中の全リクエストにそのエラーが共有される。ただし既存の retry + fail-open 設計（エラー時は空タグを返す）により致命的影響はない
- **`untrack()` の保守コスト**: `$effect` 内で新たにリアクティブ変数を参照する際、意図的に追跡対象にするか `untrack()` で除外するかを開発者が判断する必要がある

## Fix 6: フィードタグ $effect の隠れた依存関係によるリクエスト雪崩の修正

### CONTEXT

Fix 5 で article tags `$effect` に `untrack()` を適用し singleflight を追加したが、ログ調査で別のループが残っていることが判明。5分間で 1113回の `starting article tags stream`、1303回の `fetchArticleContent` が全て同一記事に対して発生。

`fetchArticleContent` は `loadFeedTagsAsync()` からのみ呼ばれるため、フィードタグの `$effect`（line 84-94）が ~1300回再発火していた。

### 根本原因

`loadFeedTagsAsync()` 内で `feedTagsTimeoutId`（`$state`）を同期的に読み取っており、Svelte 5 の `$effect` はコールスタック全体の同期リアクティブ読み取りを追跡するため、これが暗黙の依存関係となっていた。

**ループのメカニズム:**
1. `$effect` 発火 → `loadFeedTagsAsync()` 呼び出し
2. 同期的に `feedTagsTimeoutId` を読み取り → `$effect` の依存関係に追加
3. `feedTagsTimeoutId = null` で値変更 → `$effect` の再実行がスケジュール
4. `await` 後に `feedTagsTimeoutId = setTimeout(...)` → さらに値変更 → `$effect` 再スケジュール
5. `$effect` 再発火 → step 1 に戻る（ping-pong ループ）

### DECISION

**修正 C: feed tags `$effect` 内の `loadFeedTagsAsync` を `untrack()` でラップ**

```typescript
$effect(() => {
    if (currentFeed) {
        if (currentFeed.tags && currentFeed.tags.length > 0) {
            feedTags = currentFeed.tags;
            isLoadingFeedTags = false;
        } else {
            isLoadingFeedTags = true;
            untrack(() => loadFeedTagsAsync(currentFeed.url));
        }
    }
});
```

**修正 D: `feedTagsTimeoutId` と `activeStreamControllers` を plain `let` に変更**

テンプレートで直接使用されていない変数を `$state` から plain `let` に変更し、リアクティビティの対象外とした。

```diff
- let feedTagsTimeoutId = $state<ReturnType<typeof setTimeout> | null>(null);
+ let feedTagsTimeoutId: ReturnType<typeof setTimeout> | null = null;

- let activeStreamControllers = $state<Map<string, AbortController>>(new Map());
+ let activeStreamControllers: Map<string, AbortController> = new Map();
```

**修正 E: モバイル版 article tags `$effect` に `untrack()` 適用**

`TagTrailScreen.svelte`（モバイル）にも Desktop と同じ `untrack()` 修正を適用。
