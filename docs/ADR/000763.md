# ADR-000763: 本番デプロイを private repo + pull ベース + Hurl E2E 付きパイプラインに段階移行する

## Status

Accepted

## Date

2026-04-17

## Affected Services

- search-indexer (Go) — 初弾としてイメージ pull 化と E2E を導入
- pact-broker — 限定 IF で builder ランナーから到達可能にし、healthcheck から basic-auth を剥がす

## Context

現状 `scripts/deploy.sh` は本番ホスト上で `c2quay deploy` を呼び、コンポーズが `build:` 指示から 51 サービスを毎回ローカルビルドして `docker compose up -d --wait` する設計になっている。

- 本番ホストの CPU/IO を長時間占有し、deploy 所要時間がビルド時間に支配される
- イメージ来歴が残らない。`.c2quay/snapshots/*.json` は `Version: "<git sha>"` だけで `ImageRef` は空のまま
- `localhost:9292` の Pact Broker に GitHub-hosted runner から到達できないので、[[000746]] が「Pact gate と record-deployment を単一ホスト側の手動デプロイスクリプトに閉じる」構成を選ばざるを得なかった

この前提が変わった: **self-hosted runner を PN 内に立てられたため**、builder ランナーから本番ホストの Pact Broker に PN 越しで直接到達できる。build と contract gate と E2E を本番ホストから外に追い出せる状況になった。

一方で Alt 自体は OSS として公開されているので、self-hosted runner を public repo に紐付けると、fork PR から workflow 改変 1 つで self-hosted 環境にコード実行を仕掛けられてしまう。構造的な隔離が必要。

直近の安全監査で 12 件の hardening 項目が上がった (fork PR 経路、`pull_request_target` の禁止、ランナー跨ぎの `GITHUB_TOKEN` 流出、Pact Broker のポート露出、`rollback.sh` の入力検証、staging compose の分離、third-party action の SHA ピン、権限最小化、concurrency、healthcheck の資格情報漏洩など)。これらを取り込みつつ、最初の 1 サービス分を end-to-end で通すところまでを今回の決定範囲とする。

## Decision

**本番デプロイを次の構成に段階移行する。Increment 1 では `search-indexer` のみを pull 経路に乗せ、他サービスはサービス単位の PR で後追い移行する。**

### 採用する構成

1. **public / private 2 repo 分離**
   - public `Alt`: ソース・Dockerfile・compose・E2E (Hurl) 仕様・PR CI (`ubuntu-latest` のみ)・`push: main` で起動する `dispatch-deploy.yaml` のみを置く。self-hosted runner はここに一切登録しない。
   - private `alt-deploy`: `release-deploy.yaml` (build / pact-publish / gate / e2e / deploy の 5 job)・`.c2quay/` state・`scripts/rollback.sh`・ランナー登録。public 側の fork PR からは**物理的に**到達できない。

2. **2 台構成の self-hosted runner (ラベル分離)**
   - `self-hosted, alt-builder` — PN 内の build 専用ホスト。build/push/pact-publish/gate/Hurl E2E を担当。
   - `self-hosted, alt-prod` — 本番ホスト上 (この機器)。`docker compose pull && up -d` / smoke / record-deployment を担当。デプロイジョブは **alt-prod runner 上でネイティブに走る** (builder からの SSH や `GITHUB_TOKEN` 転送は行わない)。

3. **repository_dispatch トリガ**
   - public `Alt` の `push: main` で `dispatch-deploy.yaml` が fine-grained PAT 経由で `repository_dispatch` を private `alt-deploy` に飛ばす。PAT は `repository_dispatch` のみにスコープされ、fork PR コンテキストには暴露されない。

4. **immutable image タグでの pull deploy**
   - イメージ名は既存 `docker-build.yaml` の命名と揃え `ghcr.io/<owner>/alt-<service>:sha-<short>`。`main` も副タグで残すが deploy 時は SHA 指定。
   - compose 側は `image:` と `build:` を共存させる。本番デプロイ経路は `image:` の pull、ローカル開発は `compose/compose.dev.yaml` で `pull_policy: never` を当てて `build:` を使う。staging (E2E 用) は `compose/compose.staging.yaml` が独立 project (`alt-staging`)・`internal: true` network・`19xxx` 台の host port で分離。

5. **Pact gate は builder 側で PN 越し**
   - `compose/pact.yaml` のポート束縛を `${PROD_TAILNET_IP:-127.0.0.1}:9292:9292` に変更。PN 上でのみ Broker に触れ、public interface には出さない。
   - Healthcheck は URL に basic-auth を埋める方式から `--header` で Authorization を渡す方式へ書き換え。`docker inspect` に平文パスワードが出ない。

6. **E2E フレームワークに Hurl を採用**
   - Hurl (Orange-OpenSource、Rust 単一バイナリ、JUnit/HTML レポート、並列実行、secrets redaction) を使う。
   - 「Gauge ライクな実行可能 spec」は `e2e/hurl/<service>/README.md` (narrative) と `NN-<feature>.hurl` (executor) のペアで成立させる。JVM は持ち込まない。
   - 将来 Gherkin 層が必要になれば godog (Go ネイティブ Cucumber) を追加する。今回は導入しない。

7. **ロールバック**
   - `.c2quay/previous-tag` を状態ファイル化し、`scripts/rollback.sh` で再 pull + up。tag 文字列は `^sha-[0-9a-f]{7,40}$` 正規表現で検証、`realpath` でシンボリックリンク経由の脱出を拒否する。

### 安全監査からの 12 項目は以下で吸収する

| # | 監査項目 | 取り込み先 |
|---|---|---|
| 1 | Outside collaborator workflow に承認必須 | public repo settings (運用手順) |
| 2 | `pull_request` トリガでの self-hosted 参照を機械的に排除 | CI lint (別 PR で追加) |
| 3 | `pull_request_target` / `workflow_run` の常時禁止 | 同上 |
| 4 | deploy job はネイティブ実行、SSH しない | `release-deploy.yaml` job 5 (`runs-on: alt-prod`) |
| 5 | Pact Broker ポートは tailnet bind のみ | `compose/pact.yaml` |
| 6 | Broker healthcheck から URL-embedded password を排除 | `compose/pact.yaml` |
| 7 | `rollback.sh` に tag 正規表現と realpath 検証 | `alt-deploy/scripts/rollback.sh` |
| 8 | Staging compose は `container_name` 明示・`internal: true`・事前 `down -v` | `compose/compose.staging.yaml` + E2E job pre-step |
| 9 | third-party action は commit SHA ピン | Renovate 設定で追跡 (運用フォロー) |
| 10 | Hurl image は digest ピン | `release-deploy.yaml` (運用で Renovate 連携) |
| 11 | workflow レベル `permissions: {}` + job 単位の上書き | `release-deploy.yaml` |
| 12 | `concurrency` group で本番デプロイを直列化 | `release-deploy.yaml` |
|   | `alt-prod` runner は専用 OS user, `RUNNER_ALLOW_RUNASROOT=0` | `docs/runbooks/runner-setup.md` (alt-deploy 側) |

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| 単一 public repo + 設定だけで fork PR を遮る | Outside collaborator 承認と branch protection に頼る。設定ミスひとつで self-hosted runner が fork PR に晒される。OSS 前提では構造的分離を優先した |
| builder から prod に Tailscale SSH してデプロイ | builder の `GITHUB_TOKEN` を SSH 経由で prod に渡すことになる。stdin/argv/ログ経由で漏れる経路が多く、監査で却下。`alt-prod` runner をネイティブに走らせれば SSH 自体が不要 |
| Pull 型 GitOps エージェント (watchtower / compose-updater 系) | Watchtower は 2023 以降メンテされておらず Docker 28/29 と非互換。compose-updater は代替になり得るが、poll 遅延と「どの SHA が入ったか」の応答が prod 側で別途必要になり複雑。GitHub Actions の job queue を使う方が見通しが良い |
| 全 51 サービス一括移行 | 一度に `build:` から `image:` に書き換えると、イメージ不在 / Pact 参加者不整合 / E2E 不足の合成リスクが跳ね上がる。サービス単位で PR を回せる状態を維持する |

### 範囲外 (今回やらないこと)

- `alt-backend` 以下 50 サービスの pull deploy 移行 — 各サービス単位の PR で別途実施
- `scripts/deploy.sh` の手動 fallback 変更 — ひとまず現状のまま残す
- c2quay が `image:` を pull 優先で扱えるかの挙動調整 — Increment 1 の検証中に確認し、必要なら c2quay 側に PR
- mTLS `:9443` ・ Connect-RPC `:9301` の Hurl 化 — Hurl は HTTP/1.1 中心。HTTP/2 h2c + mTLS は現状の Go 統合テストに任せる
- `docker` group を `tecnativa/docker-socket-proxy` で絞り込む — 監査 #12 のフォローアップ

## Consequences

### Pros

- **ビルド・gate・E2E が本番ホストから外に出る**ため、deploy 中の本番 CPU/IO 負荷が消え、SHA 単位でのロールバックが 60 秒スケールになる
- **イメージ来歴が残る**。c2quay snapshot の `ImageRef` が埋まり、どの SHA がいつ入ったかを registry 側でも突き合わせできる
- **fork PR からの self-hosted runner 乗っ取り経路が構造的に塞がる**。private repo 分離と dispatch PAT の scope 最小化で、設定ミス 1 つで刺さるタイプの事故が起きない
- **E2E が実行可能 spec として commit される**。`e2e/hurl/search-indexer/README.md` (narrative) と `*.hurl` (executor) のペアが Gauge 的な読み味を JVM なしで提供する
- サービス単位で PR を回せるので、後続移行は review と revert のコストが低い

### Cons / Tradeoffs

- **運用コンポーネントが増える**: 2 repo + 2 runner + 1 PAT + 1 変数 (`PACT_BROKER_BASE_URL`)。それぞれの寿命管理 (runner upgrade / PAT rotation / Broker ヘルスチェック) が追加の継続コスト
- **alt-prod runner の `docker` group 所属は Linux 上 root 相当**。専用 OS user と `RUNNER_ALLOW_RUNASROOT=0` で絞っても、原理的には本番ホストの全面アクセスを持つ。将来の socket-proxy 化までは運用的な暗黙契約に依存する
- **Increment 1 だけ有効な中間状態**: `search-indexer` 以外は引き続き `scripts/deploy.sh` + local build 経路で動く。移行途中は「サービスごとに経路が違う」状態が続き、失敗時のトリアージがやや煩雑
- **第 1 回デプロイのブートストラップ**: GHCR に初期イメージが載るまでは `image:` 参照が pull で解決できないため、最初だけは builder ランナー上で明示的に build+push するステップが必要 (runbook に記載)
- **Hurl は HTTP 寄り**: Connect-RPC / mTLS は別途既存テストでカバーする前提。Hurl のスコープを誤ると「広くカバーしているつもりで実は抜け」があり得るので、service ごとに対象プロトコルを README に明示する
- **third-party action の commit SHA ピン**は Renovate の運用が揃うまで tag pin のまま走る。短期的には supply chain 面の弱点を許容する
