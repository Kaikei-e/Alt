# ADR-000791: rag-orchestrator に Hurl E2E スイートを追加し段階的全面展開を継続する

## Status

Accepted

## Date

2026-04-19

## Affected Services

- rag-orchestrator (Go) — プロダクションコード / Connect-RPC handler / Atlas migrations は無変更。Hurl スイートの対象となる REST (`:9010`) と Connect-RPC (`:9011`, h2c) の表面を外から黒箱で固定する
- compose/compose.staging.yaml — `[rag-orchestrator]` プロファイルで `rag-db` (pgvector 17.0 を同梱する独自 Postgres イメージ) + `rag-db-migrator` (Atlas migrate apply) + `rag-orchestrator` (distroless) の 3 コンテナを起動
- .github/workflows/e2e-hurl.yml — 既存の 7 ジョブに並ぶ `rag-orchestrator` ジョブを追加
- e2e/hurl/rag-orchestrator — `run.sh` + `00-readiness.hurl` ～ `09-augur-get-seeded.hurl` の 10 シナリオ + README
- e2e/fixtures/rag-orchestrator — 固定 UUID 3 本 (`test-user-id.txt` / `test-empty-user-id.txt` / `test-conversation-id.txt`) と `db-seed.sql` + README

## Context

[[000785]] で auth-hub を 6 番目の増分として Hurl E2E の段階的全面展開に載せた時点で、rag-orchestrator は未カバーのまま残っていた。rag-orchestrator は Alt 全体の中でも外部依存が多く、加えて以下 2 つの都合で Phase 1 スコープの設計判断を要求する対象だった:

1. **streaming RPC が主戦力**。`AugurService.StreamChat` / `MorningLetterService.StreamChat` / `/v1/rag/answer/stream` (SSE) がユーザー体験の中心だが、Hurl は Connect の `application/connect+json` フレーミングも SSE の連続キャプチャも扱えない。[[hurl.dev/docs/manual.html]] と `connectrpc.com/docs/protocol/` を突き合わせた結果、streaming は Go テストクライアントを別途用意しないと E2E 化できないと判定した。
2. **retrieve / answer 経路は 4 つの外部サービスを要する**。`news-creator` (Ollama 埋め込み + LLM 生成) / `search-indexer` (Meilisearch BM25) / `rerank-external` (BAAI/bge-reranker) / `alt-backend` (article メタデータ) の 4 依存が全部そろわないと意味のある応答にならない。Phase 1 でこれらの stub を書き下ろすと PR が肥大化し、E2E を一度も通さないまま長期化するリスクが大きかった。

一方で rag-orchestrator 単体 + `rag-db` だけで検証できるレイヤは少なくない:

- `/healthz` (常に 200) と `/readyz` (DB pool ping) の liveness / readiness
- `/connect/health` の Connect-RPC listener 生存確認
- `X-Alt-User-Id` 認証の fail-closed 契約 (missing / malformed UUID → HTTP 401 `unauthenticated`)
- `AugurService` の読み取り系 RPC (`ListConversations` / `GetConversation` / `DeleteConversation`) を空 DB + psql シード済み DB の両方で検証

rag-orchestrator は JWT を検証せず、`X-Alt-User-Id` ヘッダを上流の alt-backend に信頼する (augur/handler.go:24-27 のコメント)。production では edge proxy が外部入力の同ヘッダを必ず strip する前提だが、E2E 側が自由に注入できる点は内部ネットワーク前提のため問題にならない (fixtures README に明記)。

Hurl は SQL ドライバを持たないため、DB 状態検証は「シードは shell (psql) で・read-through はサービスの HTTP で」という二段構成になる。auth-hub が Kratos admin API で dynamic に identity を作る経路を採ったのと対照的に、rag-orchestrator には外向きの「会話を作成する」admin API が存在せず (StreamChat が唯一の作成路で Hurl 不可)、psql で直接 `augur_conversations` / `augur_messages` に INSERT するしかなかった。

## Decision

rag-orchestrator の Hurl E2E を **Phase 1 = DB-only + 読み取り系**の粒度で導入し、streaming / retrieve / answer は Phase 2 以降に分離する:

- **スイート構成**: `e2e/hurl/rag-orchestrator/` に `NN-<scenario>.hurl` 形式で 10 本。health 3 本 (`00-readiness` / `01-healthz` / `02-connect-health`)、Connect 認証 2 本 (`03-missing-user-header` / `04-malformed-user-header`)、空 DB での Augur 読み取り 3 本 (`05-list-empty` / `06-get-not-found` / `07-delete-idempotent`)、psql シード後の read-through 2 本 (`08-list-seeded` / `09-get-seeded`)。
- **staging compose の新規プロファイル**: `rag-db` (`../rag-db` の pgvector ビルドを流用、staging は `POSTGRES_PASSWORD` 平文で完結) + `rag-db-migrator` (既存 Atlas 画像を `apply`) + `rag-orchestrator` (distroless)。依存ゲートは `rag-db: service_healthy` → migrator: `service_completed_successfully` → rag-orchestrator。
- **readiness は Hurl 側で待つ**: distroless/static-debian12:nonroot にシェル / wget / curl が無く、auth-hub のような `/<binary> healthcheck` サブコマンドも rag-orchestrator には無い。compose healthcheck は付けず、`00-readiness.hurl` の `[Options] retry: 60 retry-interval: 1s` で `/readyz` を待つ。失敗原因が「unhealthy」ではなく「named test failure」として出るので診断が楽。
- **DB シードは psql**: `run.sh` が `docker compose exec rag-db psql -v user_id=... -v conv_id=...` で `db-seed.sql` を流し込む。UUID は全て `e2e/fixtures/rag-orchestrator/` の固定ファイル由来で、Hurl には `--variable` で同じ値が注入される。
- **厳密アサーション**: Connect エラーの `message` は handler のリテラルに一致させる (`missing X-Alt-User-Id header` / `invalid X-Alt-User-Id header:` プレフィックス)。将来 handler の文言が変わったら意図的に RED にする契約。
- **`ListConversations` の空応答は body 文字列で固定**: protojson は空 repeated field を省略するため、Connect 側は `{}` を返す。`jsonpath "$.conversations" count == 0` は filter error になるため、`not exists` + `body == "{}"` を併用する。
- **CI ジョブ**: auth-hub ジョブ (lines 402-452) をテンプレートに rag-orchestrator ジョブを追加。path filters に `rag-orchestrator/**` / `rag-migration-atlas/**` / `rag-db/**` / `e2e/hurl/rag-orchestrator/**` / `e2e/fixtures/rag-orchestrator/**` を足し、関係ない PR に cold build 費用を載せない。

### 却下した代替案

1. **Phase 1 で stub 4 本を同梱して retrieve/answer も含める**: news-creator-ollama-stub / rerank-stub / alt-backend-deps-stub / search-indexer-stub の各 stub を書き下ろす必要があり、PR 量が 4-5 倍に膨らむ。レビュー粒度が落ちて E2E が「動いた状態」を固定する前に腐る懸念。Phase 2 で独立 PR にする。
2. **Hurl + Go streaming テストランナーのハイブリッド構成**: StreamChat 検証のために e2e/clients/rag-orchestrator/ を Go で新設。Phase 1 で 2 系統のランナー (Hurl + Go) を運用するコスト (CI step 追加 / 失敗時の診断経路が 2 本に分岐) が Phase 1 で取るメリットより大きい。Phase 2 で streaming を扱うときに一緒に導入する方が境界が綺麗。
3. **compose healthcheck に `grpc_health_probe` バイナリを焼き込む**: distroless に追加コピーすれば healthcheck 可能になるが、Dockerfile の追加コピー分はプロダクション image にも乗る。E2E のためだけに production image を肥大化させる筋の悪さ。Hurl 側 retry で十分。
4. **テストデータシードを Atlas の postdeploy で行う**: staging 専用の Atlas env を切り、E2E データを migrator に含める案。production migrator との距離が出ると事故の元 (migrator を production に誤適用するリスク)。psql シードの方が「プロダクション migrator と同一ファイル + staging だけのデータ注入」という責務分離が明快。
5. **`docker compose down -f "$SLICE" -p ...` でテアダウン (auth-hub の書き方を踏襲)**: ローカルで compose v29 下に `-f` 指定の `down` が silent no-op するケースが再現した。project name のみでテアダウンすれば `-f` は不要で、`compose/compose.staging.yaml` と slice YAML の不整合に起因する silent failure も避けられる。

## Consequences

### Pros

- **rag-orchestrator の最も重要な「認証の fail-closed 契約」が回帰検知下に入る**。`X-Alt-User-Id` を正しく reject できない退行は E2E で即 RED。
- **空 DB / シード済み DB の両方をカバー**しており、pg 側のスキーマ変更や projector view (`augur_conversation_index`) の崩れが handler を通して早期に検出される。
- **readiness 待機を Hurl retry に寄せた**ことで、compose healthcheck が無い distroless サービスにもこのパターンをコピーしやすくなった (knowledge-sovereign 以外のサービスで再利用可能)。
- **path filter で独立 gating**。rag-orchestrator / rag-migration-atlas / rag-db を触らない PR は cold build 費用を払わない。
- **CI + ローカル完走 10/10 green** を確認して導入。auth-hub と同一の `STAGING_PROJECT_NAME` ベースのスライス分離により、並列マトリクスでも衝突しない。
- **フィクスチャ UUID が合成値**で固定されており、本物のランダム v4 UUID とは統計的に衝突しない (全 0 / 全 1 / 全 2 等のセンチネル)。

### Cons / Tradeoffs

- **Streaming 検証は Phase 1 では完全に無い**。`StreamChat` のプロトコル退行は Hurl スイートをすり抜ける。Phase 2 で Go streaming test client を入れるまでは Pact 契約テスト + unit に依存。
- **retrieve / answer 経路は E2E 未カバー**。`/v1/rag/retrieve` / `/v1/rag/answer` / `/internal/rag/index/upsert` の regression は Pact と unit だけで守る状態が続く。
- **psql シードは staging compose の `rag-db` パスワード (`alt-staging-test-rag-password`) を run.sh に直書き**。固定値だが compose fixture の一部として扱う。将来プロファイル間で共有 secret に寄せたくなった際に見直し点として残る。
- **rag-orchestrator の compose healthcheck が無い**ため、`docker compose up --wait` は container running で即 ready 扱い。これは Hurl retry で回避しているが、`run.sh` 以外の経路でこの compose を起動するとサービスが未 ready のまま consumer に見られる可能性がある。
- **`body == "{}"` アサーション**は protojson が `EmitUnpopulated=false` を維持する前提に依存する。将来 Connect/Go generator がデフォルト挙動を変えると 05 が壊れる。そのときの fix は 1 行なので許容する。
- **CI の cold build 時間**が ~1 分 (pgvector 付き rag-db イメージのビルド + rag-orchestrator distroless ビルド)。path filter で普段は走らないので許容範囲。
