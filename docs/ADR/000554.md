# ADR-000554: SvelteKit Connect-RPC プロキシのストリーミング検出を修正する

## Status

Accepted

## Date

2026-03-24

## Affected Services

- alt-frontend-sv (TypeScript/SvelteKit)

## Context

StreamSummarize のチャンクがブラウザの DevTools Network タブの Response に到着しているにもかかわらず、フロントエンドの UI にストリーミングテキストが表示されない問題が発生した。

### 根本原因

SvelteKit の Connect-RPC プロキシ（`src/routes/api/v2/[...path]/+server.ts`）が、ストリーミングレスポンスの content-type 判定で `application/connect+json` を見逃していた。

```typescript
// Before: proto と grpc のみチェック
if (
    ct.includes("application/connect+proto") ||
    ct.includes("application/grpc")
) {
    responseHeaders.set("X-Accel-Buffering", "no");
}
```

Connect-RPC は JSON モード（`application/connect+json`）と Proto モード（`application/connect+proto`）の 2 つのシリアライゼーション形式をサポートする。ブラウザの Connect-RPC クライアントは JSON モードを使用するが、上記の検出条件は Proto モードのみに対応していた。

`X-Accel-Buffering: no` が設定されないと、nginx の `proxy_buffering on`（`/api/` location, default.conf:363）によりレスポンスがバッファされ、チャンクが即座にブラウザに転送されない。結果として、ストリーミングデータはバックエンドで正常に生成されているが、ブラウザ側では逐次表示されず、バッファが満たされるか接続が閉じるまで待たされる。

## Decision

**`application/connect+` プレフィックスマッチに変更し、JSON/Proto 両モードのストリーミングで `X-Accel-Buffering: no` を設定する。**

```typescript
// After: connect+ プレフィックス全体をチェック
if (
    ct.includes("application/connect+") ||
    ct.includes("application/grpc")
) {
    responseHeaders.set("X-Accel-Buffering", "no");
    responseHeaders.set("Cache-Control", "no-cache, no-transform");
}
```

この変更により、将来的に新しいシリアライゼーション形式（例: `application/connect+cbor`）が追加された場合にも自動的にストリーミング対応される。

## Consequences

### Pros

- Connect-RPC JSON モードのストリーミングが正常に動作する（StreamSummarize, StreamFeedStats 等）
- 将来の Connect-RPC シリアライゼーション形式にも対応
- 1 行の変更で侵襲性が極めて低い

### Cons / Tradeoffs

- `application/connect+` に一致するが実際にはストリーミングではないレスポンスにも `X-Accel-Buffering: no` が設定される。ただし Connect-RPC の非ストリーミングレスポンスは通常短時間で完了するため、バッファリング無効化の実害は無視できる
