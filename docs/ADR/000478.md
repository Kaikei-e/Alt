# ADR-000478: BFF の streaming timeout 5分キャップを unary/streaming で分離する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-butterfly-facade (Go)

## Context

`StreamKnowledgeHomeUpdates` が nginx access log で常に `rt=300.xxx`（= 300秒 = 5分）で切断されていた。nginx 側は既に streaming 用の `location /alt.knowledge_home.v1.KnowledgeHomeService/` で 2400秒のタイムアウトを設定しており、問題はその下流の BFF レイヤーにあった。

BFF（alt-butterfly-facade）には 2 つのメカニズムが存在し、両方が 5 分でストリームを殺していた:

1. **`applyConnectTimeout()` の `maxConnectTimeout = 5 * time.Minute`** — unary RPC のための abuse guard だが、streaming リクエストにも一律適用されていた
2. **`streamingClient.Timeout = streamingTimeout`** — `http.Client.Timeout` はレスポンスボディの読み取り完了までを含む全体タイムアウト。server-streaming では「ボディ読み取り完了 = ストリーム終了」のため、この設定自体が streaming の寿命上限になっていた

結果として、backend の stale timer（30分）まで維持されるべきストリームが、BFF で 5 分で強制切断されていた。

### 計測データ（修正前）

- `rt=300.xxx` の nginx アクセスログ: **440件**
- `context canceled` 300秒周期のエラー: 多数

## Decision

**timeout cap を unary リクエストと streaming リクエストで分離する方針を採用する。**

### 1. `maxConnectTimeout` / `maxStreamingTimeout` の分離

`proxy_handler.go` に 2 つの定数を定義:

```go
const maxConnectTimeout = 5 * time.Minute    // unary RPC 用（従来通り）
const maxStreamingTimeout = 40 * time.Minute // server-streaming 用
```

`applyConnectTimeout()` 内で `isStreamingProcedure()` の結果に応じて適用する cap を切り替える。streaming リクエストには `maxStreamingTimeout`、unary リクエストには従来の `maxConnectTimeout` を適用する。

### 2. `streamingClient.Timeout` を 0（無制限）に

`http.Client.Timeout` は HTTP トランザクション全体（接続確立からレスポンスボディ読み取り完了まで）に適用される。server-streaming ではボディが長時間にわたって送信され続けるため、このフィールドでの制御は不適切。`Timeout = 0`（Go のデフォルト = 無制限）とし、ストリームの寿命は `context.WithTimeout` による per-request deadline で制御する。

### 3. config default を 40 分に

`BFF_STREAMING_TIMEOUT` のデフォルト値を `40 * time.Minute` に設定。backend の stale timer（30分）より長く、かつ無制限ではない値として選定した。

### 4. compose 設定の更新

`compose/bff.yaml` に `BFF_STREAMING_TIMEOUT=40m` を明示的に設定。

### 検討した代替案

| 案 | 評価 |
|----|------|
| streaming timeout を無制限にする | abuse guard が完全になくなる。backend の stale timer に依存するが、BFF 側の防御がゼロになるのは望ましくない。却下。 |
| `http.Client.Timeout` に 40 分を設定 | 40 分を超えるストリームは切断される。context deadline と二重管理になり、どちらが先に発火するか不透明。却下。 |
| nginx 側で全て制御し BFF は pass-through | BFF が Connect-Timeout-Ms を解釈する設計上、BFF 側にも timeout 責務がある。却下。 |

## Consequences

### Pros

- **300秒切断が完全に解消** — 修正後 `rt=300.xxx` は 0 件
- **ストリームが backend の stale timer（30分）まで維持される** — 6 分生存ストリーム（`rt=360.xxx`）を 2 件観測して確認
- **unary RPC の abuse guard は維持** — `maxConnectTimeout = 5m` は従来通り
- **context deadline による一元管理** — `http.Client.Timeout` と `context.WithTimeout` の二重管理を排除

### Cons / Tradeoffs

- **40 分の上限が残る** — 40 分を超えるストリームが必要になった場合は設定変更が必要。現状 backend の stale timer が 30 分のため問題にならない
- **streaming の abuse guard が緩くなる** — unary の 5 分に対し streaming は 40 分。Connect-Timeout-Ms ヘッダで 40 分以内の任意の値を指定可能。ただし streaming は認証済みユーザーのみアクセスするため許容範囲
- **`streamingClient.Timeout = 0` のリスク** — context cancel されないストリームが残るとコネクションリーク。ただし `applyConnectTimeout()` が必ず context deadline を設定するため、実質的にリークは発生しない
