# ADR-000319: Tag Trail オンザフライ生成の起動時クラッシュ修正

## ADR's STATUS

完了

## CONTEXT

ADR-000318 で Tag Trail のオンザフライタグ生成に関する包括的修正（永続化・専用ストリーム分離・フロントエンド改善）を実施し、全テスト合格・ヘルスチェック通過を確認した。しかし実運用環境では依然としてオンザフライ生成が機能していなかった。

コンテナログの精査により、以下の 3 つの根本原因を特定した。

### 1. Protobuf Gencode/Runtime バージョン不整合（Critical）

tag-generator のバックグラウンドサービススレッドが起動直後にクラッシュしていた。Connect-RPC クライアント用に生成された `internal_pb2.py` が protobuf 7.x の gencode バージョンで生成されていたが、コンテナ内のランタイムは 6.x 系であった。`ValidateProtobufRuntimeVersion()` による Major バージョンの不一致検出でインポート時に例外が発生し、articles consumer も tags consumer も一切起動しない状態だった。

protobuf を 7.x に上げれば gencode と整合するが、`opentelemetry-proto` が `protobuf>=5.0,<7.0` を要求しているため、7.x へのアップグレードは不可能であった。

### 2. Tags consumer が実行パスに未追加（Critical）

ADR-000318 Phase 2 で追加した tags consumer の起動コードは `main.py` に実装したが、コンテナのエントリポイントは `auth_service.py` であった。`auth_service.py` の `_run_background_tag_generation()` は articles consumer のみを起動しており、`alt:events:tags` ストリームを購読する tags consumer は起動しなかった。

### 3. Redis 接続プール枯渇（Secondary）

mq-hub の Redis プールサイズがデフォルト（10）のままであった。Tag Trail は複数記事のタグ生成を並列でリクエストし、各リクエストが reply 待ちでコネクションを最大 60 秒保持する。consumer が動いていないため reply が返らず、全コネクションがタイムアウトまでブロックされていた。根本原因 1 が解消されれば大幅に緩和されるが、並列リクエスト数を考慮するとプールサイズ拡大も必要であった。

## DECISION MAKING

### Fix 1: protobuf gencode バージョンの 6.x 互換化

`internal_pb2.py` の `ValidateProtobufRuntimeVersion()` 呼び出しを protobuf 6.x 互換に修正した。生成済み proto ファイルのシリアライズ済みディスクリプタはバージョン非依存のため、バージョン検証の修正のみで互換性が確保できる。

併せて `pyproject.toml` の protobuf 制約を `>=6.33.5,<7.0` に明示化し、`uv lock` でロックファイルを更新した。

**補足**: buf.gen.yaml に Python ターゲットが未設定であり、tag-generator の proto 生成は手動プロセスとなっている。今後 proto 定義を更新する際は、protobuf 6.x 互換の gencode で再生成する必要がある。

### Fix 2: `auth_service.py` に tags consumer を追加

`_run_background_tag_generation()` 内の articles consumer 起動コードの後に、`ConsumerConfig.tags_stream_from_env()` を使った tags consumer の起動ブロックを追加した。articles consumer と同じパターンで、別スレッドで `asyncio.run(tags_consumer.start())` を実行する。

### Fix 3: mq-hub Redis プールサイズ拡大

`compose/mq.yaml` の mq-hub 環境変数に `REDIS_POOL_SIZE=50` を追加した。Tag Trail の並列リクエスト + reply 待ちに対応するため、デフォルト 10 から 50 に拡大した。

## RESULTS, EFFECTS

### PROS

- **サービス起動の安定化**: protobuf バージョン不整合によるクラッシュが解消され、バックグラウンドサービスが正常に起動する
- **tags consumer の有効化**: `alt:events:tags` ストリームの consumer が実際に起動し、オンザフライタグ生成リクエストが処理される
- **Redis プール安定性**: 並列リクエスト時のコネクションプール枯渇が解消される
- **依存バージョンの明示化**: protobuf の上限制約（`<7.0`）を明示したことで、意図しないメジャーバージョンアップが防止される

### CONS, TRADEOFF

- **生成済みコードの手動パッチ**: `internal_pb2.py` を手動修正しているため、proto 定義を再生成する際に同じ問題が再発するリスクがある。buf.gen.yaml への Python ターゲット追加で自動化すべきである
- **Redis プールサイズのハードコード**: `REDIS_POOL_SIZE=50` は経験的な値であり、並列度が大幅に増加した場合は再調整が必要

### 検証手順

```bash
# tag-generator テスト
cd tag-generator/app && SERVICE_SECRET=test-secret uv run pytest tests/ -v

# コンテナ再ビルド・起動
docker compose -f compose/compose.yaml -p alt build tag-generator mq-hub
docker compose -f compose/compose.yaml -p alt up -d tag-generator mq-hub

# ログ確認（以下が出力されること）
# - redis_streams_consumer_started
# - tags_stream_consumer_started
# - 「Background tag generation service failed」が出ないこと
docker compose -f compose/compose.yaml -p alt logs tag-generator | grep -E "(consumer_started|tags_stream|service_failed)"
```
