# ADR-000605: rag-orchestrator の retrieval quality gate 強化と streaming path での bad answer 停止条件を追加する

## Status

Accepted

## Date

2026-03-28

## Affected Services

- rag-orchestrator (Go)

## Context

ベンチマーク結果から、以下の 2 つの品質問題が特定された。

### 1. Retrieval quality gate の判定が不十分

`RetrievalQualityAssessor` は top-3 チャンクの平均スコアのみで品質を判定していた。この判定では以下のケースを検出できない:

- **Topic incoherence**: top-3 チャンクが無関係な記事から取得されている場合（例: "Iran Protests", "Cooking Recipes", "Space Exploration"）。平均スコアが高くても、検索結果がトピック的に散乱している
- **One hit + noise パターン**: top-1 スコアが非常に高いが top-2 以降が極端に低い場合。1 件だけ関連性の高いチャンクがあり、残りはノイズ。平均スコアが閾値を超えても実質的に有用なコンテキストは 1 件のみ

### 2. Streaming path で bad answer がそのまま送信される

`stream_short_answer_detected` は warning ログを出すのみで、回答品質チェック（`AssessAnswerQuality`）は非ストリーミングの `Execute()` パスでのみ実行されていた。ストリーミングパスでは:

- short answer が検出されても処理が続行される
- keyword coverage や coherence のチェックが実行されない
- 明らかに壊れた回答が `fallback=false` で完了する

### 3. ConversationPlanner の DI 未接続

[[000604]] で実装した ConversationPlanner と ConversationStore が DI コンテナに接続されておらず、本番環境では dead code だった。

## Decision

3 つの改善を実施する。

### 改善 1: Retrieval quality gate に topic coherence と score variance チェックを追加

`RetrievalQualityAssessor.Assess()` に 2 つの新しいダウングレードシグナルを追加する。

**Topic incoherence 検出**: top-N チャンクのタイトルを比較し、どのペアも有意な単語（3 文字以上）を共有しない場合、QualityGood を QualityMarginal にダウングレードする。タイトルが空の場合（メタデータなしの reranking-only フロー）はスキップする。

**Score variance 検出**: top-1 スコアが top-2 の 5 倍を超える場合（"one hit + noise"）、QualityGood を QualityMarginal にダウングレードする。

### 改善 2: Streaming path に answer quality hard stop を追加

`rag_answer_stream.go` の validation セクションで `AssessAnswerQuality()` を実行し、short answer が quality flag（`low_keyword_coverage` または `incoherent_ending`）と同時に検出された場合、`StreamEventKindFallback` を送信して回答を中断する。

short answer 単独では fallback しない（yes/no 形式の質問に対する正当な短い回答がある）。short answer + quality issue の組み合わせで初めて hard stop が発動する。

### 改善 3: ConversationPlanner を DI コンテナに接続

`di/container.go` で `ConversationPlanner` と `ConversationStore`（1024 エントリ、30 分 TTL）を生成し、`WithConversationPlanner()` 経由で `answerUsecase` に注入する。Handler で first user message の SHA256 ハッシュから deterministic な `thread_id` を導出し、`AnswerWithRAGInput.UserID` として渡す。

## Consequences

### Pros

- Retrieval quality gate が「平均スコアは高いが実質的に関連コンテキストが 1 件だけ」のケースを検出できるようになった
- Topic incoherence 検出により、無関係な記事のチャンクが混在するケースが QualityMarginal に正しく分類される
- Streaming path で broken answer がユーザーに届く前に fallback で止まる
- ConversationPlanner が本番で動作し、曖昧なフォローアップの reference resolution が有効になった

### Cons / Tradeoffs

- Topic incoherence チェックは title の単語比較に依存する。意味的に関連するが語彙が異なるタイトルは false positive でダウングレードされる可能性がある
- Score variance の 5 倍閾値は経験則。最適値はデータに依存し、調整が必要になる可能性がある
- Streaming hard stop は、正当な短い回答 + たまたま keyword coverage が低いケースで false positive fallback が発生する可能性がある
- thread_id が first user message のハッシュに基づくため、同じ質問で始まる異なる会話が同じ thread_id になる
