# ADR-000723: pre-processor の Medium/Low 5 件を BodyLimit・Transport 強化・ホスト間隔制御・レスポンス境界・本番起動検証で解消する

## Status

Accepted

## Date

2026-04-14

## Affected Services

- pre-processor (Go)

## Context

[[000719]] と [[000720]] で pre-processor の REST / Connect-RPC 両面に service-to-service 認証が入ったが、同時に作成した `plans/partitioned-spinning-whisper` の OWASP 監査には Medium 3 件・Low 2 件が残っていた。

| ID | 深刻度 | OWASP | 内容 |
|---|---|---|---|
| MED-1 | Medium | A05 | `external_api_repository.go` の `http.Client` が `Timeout: 30s` のみで、Transport 層に `ResponseHeaderTimeout` / `DialContext` / `TLSHandshakeTimeout` 系が未設定。slowloris で connection slot を占有可能 |
| MED-2 | Medium | A05 | `bootstrap/server.go` が `middleware.CORS()` デフォルトで Origin `*` を返す。service-to-service の内部 API に CORS は不要かつ有害 |
| MED-3 | Medium | A04 | プロジェクト必須ルール「5s 間隔」が `RateLimitConfig.DefaultInterval` として config に存在するだけで、どのコードパスからも参照されていない。ingress 側の body 上限もなし |
| LOW-1 | Low | A05 | alt-backend `/v1/internal/system-user` などから返るレスポンス body を `io.LimitReader` で囲んでおらず、悪質応答で OOM の可能性 |
| LOW-2 | Low | A07 | 本番で `SERVICE_SECRET` が未設定のまま起動すると全リクエストが 500 になる silent failure。fail-fast 判定が存在しない |

これらを一括解決し、HIGH 系と合わせて `plans/partitioned-spinning-whisper` を閉じる。

## Decision

以下をまとめて採用する。

1. **MED-2 (CORS)**: `middleware.CORS()` を `bootstrap/server.go` から削除する。pre-processor は [[000719]] 以降 X-Service-Token 必須の内部 API なので、ブラウザ Origin に対する許可自体が脅威の増幅にしかならない。
2. **MED-1 (Transport)**: `utils/http_client_manager.go` の `createOptimizedClient` に `DialContext` (5s) と `ResponseHeaderTimeout` (20s) を追加し、`TLSHandshakeTimeout` を 10s → 5s に引き締める。`repository/external_api_repository.go` の無防備 `http.Client{Timeout: 30s}` を `utils.NewHTTPClientManager().GetDefaultClient()` に差し替える。
3. **MED-3a (ingress BodyLimit)**: `/api/v1/summarize*` グループに `middleware.BodyLimit("1M")` を適用する。記事本文は alt-backend で抽出済みなので 1 MiB で実用ペイロードは賄える。
4. **MED-3b (egress rate limit)**: `utils/host_rate_limiter.go` を新設し、プロセス全体の singleton (`DefaultHostRateLimiter`) で「同一ホストに対する最小呼び出し間隔」を守る。`driver/summarizer_api.go` の blocking / streaming 両経路で `Wait(ctx, cfg.NewsCreator.Host)` を呼び、`cfg.RateLimit.DefaultInterval` (既定 5s) を dead config から living config に変える。
5. **LOW-1 (response bound)**: `external_api_repository.go::getSystemUserIDOnce` の JSON デコードを `io.LimitReader(resp.Body, maxInternalResponseBytes=10 MiB)` で囲む。
6. **LOW-2 (fail-fast)**: `config/validate.go` に `APP_ENV=production` かつ `SERVICE_SECRET` 空の組合せを拒否するチェックを追加する。テスト用 `APP_ENV=development` や空の `APP_ENV` では従来どおり許容する。

## 実装の概要

### utils

- `utils/http_client_manager.go`
  - `createOptimizedClient` が `&net.Dialer{Timeout: 5s, KeepAlive: 30s}` を使い `DialContext` を差し込む
  - `ResponseHeaderTimeout: 20s`, `TLSHandshakeTimeout: 5s` を追加
  - 既存の `MaxIdleConns / IdleConnTimeout / ForceAttemptHTTP2` はそのまま維持
- `utils/host_rate_limiter.go` 新設
  - `HostRateLimiter` は per-host 最終呼び出し時刻を mutex で管理し、`Wait(ctx, host)` で次回許可時刻まで眠る
  - 同時実行の 2 本目以降が同タイミングに集中しないよう、reserve 方式（計算した待機後時刻を先に map に書き込む）を採用
  - `DefaultHostRateLimiter(interval)` が `sync.Once` 経由でプロセス singleton を返す
- ユニットテスト: transport 各フィールドのアサート、rate limiter の interval 強制・per-host 分離・ゼロ interval no-op・context cancel 伝播

### repository

- `NewExternalAPIRepository` が `utils.NewHTTPClientManager().GetDefaultClient()` を使う
- `getSystemUserIDOnce` の `json.NewDecoder(resp.Body)` を `io.LimitReader(resp.Body, maxInternalResponseBytes)` で囲む (`maxInternalResponseBytes = 10 * 1024 * 1024`)

### driver

- `ArticleSummarizerAPIClient` と streaming 版双方で `utils.DefaultHostRateLimiter(cfg.RateLimit.DefaultInterval).Wait(ctx, cfg.NewsCreator.Host)` を `http.NewRequestWithContext` の前に呼ぶ
- 待機はブロック式だが、ctx cancel で即脱出するため shutdown との相性は良い

### bootstrap

- `server.go`
  - `e.Use(middleware.CORS())` を削除
  - `api.Use(middleware.BodyLimit("1M"))` を `RequireServiceAuth` の直前に挿入
- 統合テスト: CORS が絶対に返らないこと、2 MiB 本文が 413、小さな本文が通ること

### config

- `validate.go` に `isProductionEnv()` と `loadServiceSecretForValidation()` を追加
  - middleware のローダと同じ優先順 (`SERVICE_SECRET_FILE` > `SERVICE_SECRET`) で secret を解決
- 既存 `validateConfig` の末尾に production 判定を追加
- テスト: production + 空 SERVICE_SECRET で error、production + 値ありで success、development では空でも success

### End-to-End 検証

コンテナ再ビルド後に以下を確認した。

| ケース | 結果 |
|---|---|
| `GET /api/v1/health` | 200 |
| `POST /api/v1/summarize` 2 MiB ペイロード | 413 Request Entity Too Large |
| `POST /api/v1/summarize` 正規トークンで小本文 | 非 401 (auth 通過、handler 到達) |
| レスポンス `Access-Control-Allow-Origin` ヘッダ | 常に未設定 |

ユニットテスト: pre-processor 全パッケージ PASS（新規 RED→GREEN 10 件 + 既存アサーション互換修正 1 件）。

## Consequences

### Pros

- slowloris / 巨大 body / header-read 放置の各経路が Transport 層と ingress 層で個別に塞がれ、LLM 計算を食いつぶされにくくなった
- `cfg.RateLimit.DefaultInterval` が dead config でなくなり、CLAUDE.md の「5 秒ルール」が実コードで強制される
- 本番で secret 投入を忘れた場合、起動時エラーで即死するので dashboard が 500 を吐き続ける silent failure にならない
- CORS を排除したことで、将来「誤って外部公開してしまっても CORS は通さないからブラウザからは叩けない」という一段余分の防御線になる

### Cons / Tradeoffs

- `DefaultHostRateLimiter` は singleton のため、単体テストで interval を別値に差し替えたい場合は `NewHostRateLimiter` を直接呼ぶ必要がある
- BodyLimit 1 MiB は現状の alt-backend 抽出済みコンテンツに対して広めだが、超長尺のプレーンテキストを送るユースケースが将来現れたら config 化が必要
- 本番 fail-fast は `APP_ENV` の厳密な値（"production"）に依存する。大文字小文字は吸収したが、"prod" / "prd" 等の別称はエラーにならない。将来的には全サービスで統一する必要がある
- host rate limiter は in-process singleton なので水平スケール時には pod 単位の待機になる。ニュース生成側のキャパを考えると十分だが、pod 数が増えた場合は外部レートリミッタ（Redis 等）に置き換える
