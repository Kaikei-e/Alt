# ADR-000429: Knowledge Home Phase 0 の canonical contract と運用境界を固定する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend
- alt-frontend-sv

## Context

Knowledge Home は UI と API の骨格を持つ一方で、契約の解釈が層ごとに分かれ始めていた。

- API 契約と frontend 型の間で意味のズレがある
- read model の責務と状態値の解釈が一貫していない
- stream、backfill、reproject の境界が文書化されていない

このまま次フェーズへ進むと、仕様変更のたびに backend・frontend・運用で解釈が再分岐する。

## Decision

Phase 0 では新機能を増やさず、Knowledge Home の canonical contract と運用境界を固定する。

具体的には以下を採用する。

- canonical contract の source of truth を 1 つに集約する
- API、read model、event、stream、service quality の意味をその契約書に固定する
- frontend は後方互換の補助値ではなく、正規契約を優先して使う
- backfill と reproject の用途を分離し、標準運用手順を固定する
- observability の論理名と実装名の対応を固定する

## Consequences

### Pros

- Knowledge Home の state/field/event の意味が backend・frontend・運用で一致する
- frontend はキャストや暗黙解釈を減らし、proto 契約をそのまま使える
- stream と reproject の運用時に「何が正規契約か」を参照できる
- 次フェーズの実装レビューが「契約に沿っているか」で判断しやすくなる

### Cons / Tradeoffs

- canonical contract を文書と contract test の両方で維持するコストが増える
- 互換イベントを残すため、stream 契約は短期的に二層構造になる
- `service_quality` を文字列で維持するため、未定義値の防御は client 側で必要になる
