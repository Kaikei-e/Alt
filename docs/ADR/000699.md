# ADR-000699: FactNormalizerNode を XML DSL に移行し全制御層ノードの format 依存を完結させる

## Status

Accepted

## Date

2026-04-11

## Affected Services

- acolyte-orchestrator (Python)

## Context

[[000698]] で Planner・SectionPlanner・Critic の 3 ノードを XML DSL + 自前パーサに移行した。しかし FactNormalizerNode は `generate_validated()` (JSON Schema + GBNF 文法) を引き続き使用しており、Acolyte 制御層の `format` 依存が残っていた。

FactNormalizerNode は quote ごとに 1 回の LLM 呼び出しで tiny schema (`claim`, `confidence`, `data_type`) を取得する設計である。出力が小さいため thinking token 予算枯渇の影響は Planner ほど深刻ではないが、以下の理由で移行が必要だった。

1. **一貫性**: 制御層の 4 ノード中 1 つだけ JSON Schema 方式が残ると、gateway のルーティング・テスト・運用が二重管理になる
2. **品質改善の中核**: FactNormalizer の fallback rate 低下が reason.md で最重要目標に位置付けられている。XML DSL 化により、部分的に壊れた出力の salvage が可能になる（confidence の非数値→0.3 補正、data_type の enum 外→"quote" 補正など）
3. **観測基盤の前提**: ノード横断の parse success/failure/salvage メトリクスを統一するには、全ノードが同一パーサパイプラインを通る必要がある

また、[[000698]] では XML パーサの動作ログが最小限だったため、運用判断に必要な構造化ログイベントを追加する。

## Decision

**FactNormalizerNode を XML DSL に移行し、Acolyte 制御層の全 4 ノード (Planner, SectionPlanner, Critic, FactNormalizer) から Ollama `format` パラメータ依存を完全に除去する。同時に、XML パーサパイプラインに構造化観測ログを追加する。**

### 1. FactNormalizerNode の XML DSL 化

ルートタグ `<facts>` を新設し、`normalize_fact_output()` normalizer を `xml_parse.py` に追加:

| ノード | ルートタグ | 主要子要素 |
|---|---|---|
| FactNormalizer | `<facts>` | `<fact><claim>...<confidence>...<data_type>...` |

正規化ルール:

| フィールド | 欠落時 | 不正値時 |
|---|---|---|
| `claim` | `XmlParseError` → fallback | — |
| `confidence` | デフォルト 0.5 | 非数値 → 0.3 |
| `data_type` | デフォルト "quote" | enum 外 → "quote" |

`<fact>` が複数出力された場合は先頭の 1 件のみ採用する。`source_id` は LLM 出力を使わず、入力 quote の値をそのまま使用する。

`generate_validated()` (JSON Schema 方式) → `generate_xml_validated()` に切替。sentinel identity check (`result is fallback_obj`) は `generate_xml_validated` が fallback を as-is で返す設計のため維持される。

### 2. 構造化観測ログの追加

`generate_xml_validated()` オーケストレータと各ノードに structlog イベントを追加:

| イベント | 発火タイミング | 主要フィールド |
|---|---|---|
| `xml_parse_success` | parse + validate 成功 | `root_tag`, `attempt` |
| `xml_parse_repaired` | bare `&` 修復で salvage 成功 | `root_tag` |
| `xml_parse_exhausted` | 全リトライ失敗、fallback 使用 | `root_tag`, `attempts`, `error` |
| `fact_normalized` | FactNormalizer quote 処理完了 | `source_id`, `is_fallback`, `data_type` |

### 検討した代替案

- **FactNormalizer は JSON Schema のまま残す**: 出力が小さく問題が顕在化しにくいが、二重管理のコストが運用で増大するため不採用
- **`<fact>` を直接ルートタグにする**: LLM が `<facts>` ラッパーなしで `<fact>` だけ返す可能性を考慮したが、他ノードとの一貫性（ルートタグ→子要素の 2 層構造）を優先し `<facts>` をルートとした

## Implementation

### xml_parse.py

`normalize_fact_output()` を追加。`_VALID_DATA_TYPES` 定数で enum 制約を実施。claim 欠落時は `XmlParseError` を raise し、`generate_xml_validated` の fallback パスに乗せる。

`generate_xml_validated()` に 3 つの構造化ログイベント (`xml_parse_success`, `xml_parse_repaired`, `xml_parse_exhausted`) を追加。

### fact_normalizer_node.py

- import を `llm_parse.generate_validated` → `xml_parse.generate_xml_validated, normalize_fact_output` に変更
- `_NORMALIZE_PROMPT` を XML DSL 形式に変更（`<facts><fact>` の例示付き、"Output ONLY the <facts> block." 指示）
- `_normalize_quote()` 内の LLM 呼び出しを `generate_xml_validated(root_tag="facts", normalizer=normalize_fact_output)` に切替
- `fact_normalized` structlog イベントを追加

### テスト

- `test_xml_parse.py`: `TestNormalizeFactOutput` クラスに 9 テスト追加（基本パース、confidence 非数値、data_type 不正値、claim 欠落、複数 fact、デフォルト値）
- `test_fact_normalizer_node.py`: FakeLLM レスポンスを JSON → XML に更新。5 新規テスト追加（format 未注入、think block 除去、confidence/data_type 補正、claim 欠落 fallback）
- `test_checkpoint_resume.py`: `_NoopLLM` のレスポンスを XML に更新
- 全 468 テストパス

## Consequences

### Pros

- Acolyte 制御層の全 4 ノードが同一の XML DSL パーサパイプラインを通る。gateway ルーティング・テスト・運用が一元化
- FactNormalizer の部分壊れ出力を salvage 可能に（confidence 非数値、data_type enum 外を graceful に補正）
- 構造化ログにより、ノード別の parse success rate・fallback rate・salvage rate を JSON ログから集計可能になった
- `llm_parse.py` (`generate_validated`) は `extractor_node.py`、`quote_selector_node.py` がまだ使用中のため残存。将来的にこれらも移行すれば削除可能

### Cons / Tradeoffs

- FactNormalizer の出力は tiny schema (3 フィールド) であり、JSON Schema の GBNF 制約でも十分に安定していた。XML 化による品質改善幅は Planner や SectionPlanner より小さい
- `<facts>` ラッパーは 1 quote = 1 fact の現行設計では冗長だが、将来的にバッチ処理（複数 fact 返却）に拡張可能な余地を残している
- 観測ログの下流集計基盤（ダッシュボード、アラート）は未整備。ログは出力されるが活用は今後の課題
