# ADR-000751: Knowledge Home AskSheet 起点の Augur 会話が空で保存される不具合を write-path 分離・部分 flush・遅延 abort の 3 層修正で解消する

## Status

Accepted

## Date

2026-04-16

## Affected Services

- `rag-orchestrator` (`internal/adapter/connect/augur/handler.go`) — `StreamChat` の write-path を request ctx から切り離し、client abort 時に蓄積済み delta を `AppendAssistantTurn` で flush するループに変更
- `alt-frontend-sv` (`src/lib/components/knowledge-home/AskSheet.svelte`, `src/lib/hooks/useAugurPane.svelte.ts`, `src/routes/(app)/home/+page.svelte`) — AskSheet に `onConversationId` prop を導入し `history.replaceState` で `/augur/<id>` に URL を昇格、`useAugurPane.reset()` を `isLoading` 中は abort を遅延させる意味論に変更

## Context

ユーザ報告: Knowledge Home カード上の "Ask" から開始した Augur 会話が、あとから `/augur/<uuid>` を開くと**メッセージ空**で表示される。`/augur` ランディング経由の新規会話では同じ問題が出ない。該当 URL 例 `https://curionoah.com/augur/8d99086d-aa9b-44b0-bb3a-fdfe72438d6e` では `augur_conversations` に行はあるが `augur_messages` に turn が無い。

Augur の永続化契約（[[000714]] / `rag-migration-atlas/migrations/20260413120000_create_augur_conversations.sql`）は append-only で、`augur_conversations` + `augur_messages` は INSERT のみ。つまり **会話行があってメッセージが 0 件** という状態は、永続化パスのどこかで turn が失われた「silent data loss」を意味する。`/augur/history` UI は `augur_conversation_index` ビュー経由でこの空会話を候補として表示し続け、ユーザには「履歴が保存されない」と見える。

調査で次の 2 つの欠陥が合流していると特定した:

**欠陥 1 — フロントエンド: AskSheet が `conversationId` を親に返さない**

`AskSheet.svelte:31` は `useAugurPane()` を引数なしで呼び、`onConversationIdChange` オプションを渡していなかった。対照的に `(app)/augur/+page.svelte:42-48` は同じフックを `history.replaceState(…, "/augur/${id}")` のコールバック付きで呼ぶ。このため AskSheet からは URL が `/home` のまま更新されず、ユーザは「保存の兆候」を視認できない。さらに Sheet 閉じる `$effect` が `pane.reset() → abort()` を呼び、Connect-RPC のストリームを能動的に切断していた。

**欠陥 2 — バックエンド: 永続化が request ctx に束縛されていた**

`handler.go:146/154` の `EnsureConversation` / `AppendUserTurn` は request ctx を使う一方、`AppendAssistantTurn` はすでに `context.WithTimeout(context.Background(), 5s)` で detach 済みだった（L220）。より深刻なのは L194 の `for event := range events` で、ctx は各反復の先頭で非ブロッキング select だけを見る設計。**channel receive で blocking 中は ctx.Done() を拾えない**ため、client abort が下流の finalize を呼ばずにストリームが宙づりになる。結果として `Done` 事件に達しない限り `AppendAssistantTurn` は呼ばれず、会話 row だけが残る。

## Decision

3 層を同じ PR で同時に直す。どれか 1 層だけでは race window が残る:

### Layer A — AskSheet が `conversationId` を親に昇格させる

`AskSheet.svelte` に optional prop `onConversationId?: (id: string) => void` を追加し、`useAugurPane({ onConversationIdChange: (id) => onConversationId?.(id) })` で転送する。クロージャでラップするのは Svelte 5 Runes が props の初期値だけを捕捉する挙動への対策（`svelte-check` 警告が消える）。`home/+page.svelte` は `/augur/+page.svelte` と同じ `history.replaceState(history.state, "", \`/augur/\${id}\`)` を渡す。

### Layer B — `useAugurPane.reset()` を stream in-flight 中は abort を遅延する

`reset()` 直接呼び出しで AbortController を起こしてしまうと、backend は ctx キャンセルで `Done` 前に撤退する。新しい意味論:

- `isLoading === true` のときは `pendingReset = true` だけを立て、state をクリアしない
- `finalize()` または明示的 `abort()` が動いた後の共通ヘルパ `runPendingReset()` で `messages = []; conversationId = ""` を遅延実行

`abort()` の即時性は保つ（ユーザ明示操作）。テスト `reset() does not abort an in-flight stream` を新設。

### Layer C — handler.go の write-path を detach + partial flush 化

`StreamChat` 内で:

1. `persistCtx, _ := context.WithTimeout(context.Background(), 10*time.Second)` を 1 つ用意し、`EnsureConversation` / `AppendUserTurn` 両方に渡す。defer でキャンセル。
2. イベントループを `for event := range events` から `select { ctx.Done | events }` に書き換え、channel receive 中でも ctx.Done を検知できるようにする。
3. `assistantBuffer strings.Builder` / `lastCitations []domain.AugurCitation` / `authoritativeSaved bool` をループ外に持ち、`StreamEventKindDelta` / `Meta` の度に蓄積する。
4. 関数末尾の `defer` で「authoritative な Done が来ていない && buffer に中身がある」場合だけ `AppendAssistantTurn` を detach ctx で flush する。

これにより client abort でも `augur_messages` に最低 1 回の user turn と、受信済みの部分 assistant turn が INSERT される。append-only 不変条件を保ったまま「ユーザが見た分は必ず残る」保証を得る。

### 検討した代替案

- **Layer A 単体**（URL だけ保存） — abort は依然として飛ぶので部分 assistant turn は残らず、空会話は減っても「user turn だけ」の歪な会話が量産される。
- **Layer C 単体**（バックエンドだけ） — 会話は保存されるが、ユーザは URL が `/home` のままなので「保存された」視認できず、引き続き Sheet を閉じるタイミングで混乱する。
- **`goto('/augur/<id>')` で Sheet を閉じて専用ページへ遷移** — AskSheet は "quick ask" のインライン UX 前提なので、全画面遷移にすると意図を壊す。`history.replaceState` で URL だけ昇格させる方が副作用が小さい。

### TDD 順序

1. Go `TestStreamChat_ClientAbortAfterDeltas_FlushesPartialAssistantTurn` を RED として書き、in-process `httptest` + `augurv2connect` client で client abort 後に `AppendAssistantTurn` が detach ctx で呼ばれることを検証
2. handler.go を GREEN
3. `useAugurPane.test.ts` に `onConversationIdChange` / `reset()` deferred abort の RED ケースを追加
4. `useAugurPane.svelte.ts` / `AskSheet.svelte` / `home/+page.svelte` を GREEN
5. Refactor + `svelte-check` の警告差分ゼロを確認

Go は `-race` で緑、Svelte は 918/918 緑、`svelte-check --tsgo` は新規エラーなし、既存警告数が 12 → 11 に減少（AskSheet の closure wrap 由来）。

## Consequences

### Pros

- Augur の append-only 契約（会話 row には最低 1 件の user turn が存在する）が、client abort 経路でも保たれる。`/augur/history` に空会話が新規発生しなくなる。
- URL が `/augur/<id>` に昇格するので、AskSheet を閉じた後もユーザは同じ URL で会話に戻れる。ブックマーク / 共有 / PWA 再起動後の復帰も機能する。
- `select { ctx.Done | events }` 化で、将来 usecase 側のイベントソースが遅い場合も client abort に即応できる。`HybridPrioritySemaphore` 経路（[[PM-2026-013]] の P-6 未処理）と独立に正しく振る舞う。
- `useAugurPane` の deferred abort は quick-ask UX を壊さず、「誤タップで Sheet を閉じた場合でも会話は DB に残る」という安全側の挙動になる。

### Cons / Tradeoffs

- 既存の空会話行（`8d99086d-...` を含む過去データ）は今回の修正では消えない。`/augur/history` 表示上の phantom は残るため、別途 one-shot cleanup（`DELETE FROM augur_conversations WHERE id NOT IN (SELECT DISTINCT conversation_id FROM augur_messages)`）を運用側で計画する必要がある。本 ADR のスコープ外。
- `persistCtx` の 10s timeout は Augur の append が通常 <100ms で完了する前提に依存する。DB 側の pgbouncer 待機が悪化した場合、detach ctx のほうが先に期限切れで失敗する可能性がある。失敗は error log に落ちるが、request ctx 追従より「user turn 側だけ残り assistant turn が落ちる」非対称が発生しうる。観測指標として `failed to persist assistant turn` / `failed to flush partial assistant turn` を別々に slog で分けている。
- Mobile `ChatWindow` (`(app)/augur/+page.svelte:50-56`) は同種の `onConversationIdChange` ギャップを持ったままだが、今回のユーザ報告は AskSheet 経路限定なのでスコープ外。次回対応時に Layer A と同じパターンで塞ぐ。
- `useAugurPane.reset()` の意味論変更は、AskSheet 以外の利用箇所（現状なし）を追加した際に「reset したのにしばらく state が残る」挙動が見えうる。`pendingReset` は private なので副作用は内部に閉じているが、呼び出し側が `isLoading` を監視しないまま連続操作すると混乱する可能性がある。docstring で明示している。
- バックエンド検知の race window（abort → 蓄積 0 delta → flush 不要）では依然として「会話 row + user turn のみ」の状態が残る。これは append-only 契約上は合法（user が何を聞いたかは残る）なので許容するが、UI 側で「assistant からの応答なし」を表示する扱いは `/augur/[conversationId]/+page.svelte` の既存 rendering に任せる。
