# ADR-000586: Retention ポリシーとアーカイブワーカーを導入しイベントデータのライフサイクルを管理する

## Status

Accepted

## Date

2026-03-25

## Affected Services

- knowledge-sovereign (Go) — retention ワーカー + storage stats エンドポイント
- knowledge-sovereign-db (PostgreSQL 16) — `knowledge_retention_log`, `knowledge_user_event_aggregates` テーブル追加

## Context

[[000532]] は Knowledge Sovereign が retention policy matrix を所有すると決定したが、実装は未着手だった。[[000584]] でパーティション化が完了し、パーティション単位のアーカイブ（export → detach → drop）が技術的に可能になった。[[000585]] で Projection Snapshot が導入され、snapshot 以降のイベントだけで reproject できる前提条件も整った。

残る課題は:
1. entity ごとの保持期間（hot / warm / cold）の定義
2. 自動アーカイブの実行ロジック
3. アーカイブ操作の監査ログ
4. データサイズの可視化

### 不変条件の維持

**完全削除はしない**。cold tier は JSONL.gz をディスクに永続保持する。full reproject が必要な場合は cold archive を PostgreSQL に restore してから replay する。

## Decision

### 1. Retention Policy Matrix（ADR-000532 の Tier A/B/C に準拠）

| Entity | Tier | Hot (PG active) | Warm (PG detached) | Cold (JSONL.gz) | 完全削除 |
|--------|------|-----------------|-------------------|-----------------|---------|
| knowledge_events (system) | B | 30日 | 90日 | 永続 | しない |
| knowledge_events (user interaction) | C | 7日 | 30日 | 永続 | しない |
| knowledge_user_events | C | 7日 | 30日 | 永続 | しない |
| summary_versions (latest) | A | 永続 | — | — | しない |
| summary_versions (superseded) | C | 30日 | 90日 | 永続 | しない |
| tag_set_versions (latest) | A | 永続 | — | — | しない |
| tag_set_versions (superseded) | C | 30日 | 90日 | 永続 | しない |
| projection snapshots | B | 直近3つ | — | 6ヶ月 | 6ヶ月超 |

Warm tier は PostgreSQL 内の `DETACH PARTITION` 済みテーブルとして残す。圧縮は行わない。

### 2. アーカイブワーカー

`POST /admin/retention/run`（metrics port の HTTP エンドポイント）で実行:

1. **Safety check**: 有効な snapshot が存在しなければ拒否
2. **Eligible 検出**: 各パーティションの `RangeEnd` と hot window を比較
3. **Export**: `COPY (SELECT row_to_json(t) FROM partition t) TO STDOUT` → gzip + SHA-256
4. **Log**: `knowledge_retention_log` に操作記録

dry-run がデフォルト。`{"dry_run": false}` で実エクスポートを実行。

DETACH / DROP は現時点ではエクスポート後に手動実行とする。自動化は運用実績を積んでから検討する。

### 3. User Event Aggregation

`knowledge_user_event_aggregates` テーブルで日次集計を保持。raw イベントの cold archive 前に集計を作成し、分析用途のサマリーを残す。reproject-safe の根拠としては使わない。

### 4. Storage Stats エンドポイント

`GET /admin/storage/stats`（metrics port）で全 knowledge テーブルのサイズ一覧を返す。health endpoint には載せない（重い集計で health semantics を壊さないため）。

```sql
SELECT relname, pg_size_pretty(pg_total_relation_size(oid)), ...
FROM pg_class WHERE relname LIKE 'knowledge_%'
```

### 5. Retention Log

全てのアーカイブ操作を `knowledge_retention_log` に記録し、監査可能にする。

```sql
CREATE TABLE knowledge_retention_log (
  log_id UUID PRIMARY KEY, run_at TIMESTAMPTZ, action TEXT,
  target_table TEXT, target_partition TEXT, rows_affected BIGINT,
  archive_path TEXT, checksum TEXT, dry_run BOOLEAN, status TEXT, ...
);
```

### Admin HTTP エンドポイント一覧

| Method | Path | 用途 |
|--------|------|------|
| POST | `/admin/retention/run` | retention 実行（dry-run or 実行） |
| GET | `/admin/retention/status` | 直近の retention log 一覧 |
| GET | `/admin/retention/eligible` | アーカイブ対象パーティション一覧 |
| GET | `/admin/storage/stats` | 全テーブルのサイズ一覧 |
| POST | `/admin/snapshots/create` | snapshot 作成 |
| GET | `/admin/snapshots/list` | snapshot 一覧 |
| GET | `/admin/snapshots/latest` | 最新の valid snapshot |

全エンドポイントは metrics port（:9501）で提供。Connect-RPC の proto 変更は不要。

## Consequences

### Pros

- イベントデータのライフサイクルが明文化され、自動アーカイブの基盤が整う
- dry-run + export + verify の 3 ステップで安全にアーカイブ可能
- retention log によりアーカイブ操作の監査・ロールバック判断が可能
- storage stats により管理者がデータ成長を把握できる
- reproject-safe 不変条件を維持しつつディスク使用量を管理できる

### Cons / Tradeoffs

- DETACH / DROP の自動実行は未実装（運用実績を積んでから検討）
- user event aggregation → raw event archive のフローは手動トリガーのみ
- OTel メトリクス（Gauge 連携）は未実装（storage stats HTTP エンドポイントで代替）
- Runbook（retention 実行手順、archive restore 手順）は別途作成が必要
- altctl CLI 統合（`altctl home snapshot create`, `altctl home retention run`）は後続で対応
