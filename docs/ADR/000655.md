# ADR-000655: Acolyte レポート詳細ページに Markdown レンダリングを導入する

## Status

Accepted

## Date

2026-04-09

## Affected Services

- alt-frontend-sv (TypeScript / SvelteKit)

## Context

Acolyte の LangGraph パイプライン（writer_node）は LLM に対して「明確で構造化された、情報量の多いセクション」を生成するよう指示している。LLM は見出し・太字・リスト・引用などの Markdown 構文を含むテキストを出力するが、レポート詳細ページ（`reports/[id]/+page.svelte`）は `{sec.body}` による生テキスト表示で `white-space: pre-wrap` を適用していたため、Markdown の構造情報がすべて失われていた。

一方、同じフロントエンドの Augur Chat（`ChatMessage.svelte`）や Morning Letter では、既存の自前 Markdown パーサー `parseMarkdown()`（`simpleMarkdown.ts`）を使って Markdown を HTML に変換・表示しており、XSS 防御（`escapeHtml` による事前エスケープ）も組み込まれている。

外部 Markdown ライブラリ（marked, remark 等）の導入は、バンドルサイズの増加と既存パーサーとの二重管理を招くため採用しない。

## Decision

**既存の `parseMarkdown()` ユーティリティを拡張し、Acolyte レポート詳細ページに統合する。**

具体的には以下の 3 点を実施する:

### 1. `simpleMarkdown.ts` に 3 機能を追加

| 機能 | 構文 | 実装箇所 |
|------|------|----------|
| Horizontal Rule | `---`, `***`, `___` | メインループ（ブロックレベル） |
| Blockquote | `> text`（複数行対応） | メインループ + `flushBlockquote()` |
| Link | `[text](https://url)` | `parseInline()`（インラインレベル） |

- Link は `https?://` プロトコルのみ許可し、`javascript:` / `data:` スキームを正規表現レベルでブロック
- Blockquote は `inBlockquote` + `blockquoteBuffer` による状態管理で複数行を結合
- HR は `/^(-{3,}|\*{3,}|_{3,})$/` で行全体マッチし、inline の `***bold***` との衝突を回避

### 2. レポート詳細ページのテンプレート変更

- `{sec.body}` → `{@html parseMarkdown(sec.body)}` に変更
- `white-space: pre-wrap` を削除（`parseMarkdown` が `<p>` タグで段落を分離するため不要）

### 3. Alt-Paper デザイン言語に沿った prose CSS

`:global()` スコープで `.section-prose` 内の生成 HTML 要素にスタイルを適用:

- 見出し（h1-h3）: Playfair Display、サイズ段階的に縮小
- 段落: `line-height: 1.72`、`margin-bottom: 0.75rem`
- リスト: `padding-left: 1.5rem`、disc / decimal
- Blockquote: 左ボーダー 2px（`--alt-charcoal`）、italic、`--alt-slate` 色
- Link: `--alt-primary` (dark teal) + underline、hover 時 `--alt-charcoal`
- HR: `border-top: 1px solid var(--surface-border)`
- Code: IBM Plex Mono、`var(--surface-2)` 背景

### 検討した代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| marked / remark 等の外部ライブラリ | バンドルサイズ増加、既存パーサーとの二重管理 |
| DOMPurify ラッパー追加 | `escapeHtml` が既に XSS を防御しており、他の消費者（Augur, Morning Letter）も DOMPurify なしで運用中。一貫性を優先 |
| 専用コンポーネント化 | import + `@html` + CSS で十分小さく、コンポーネント化のメリットが薄い |

## Consequences

### Pros

- LLM が生成する Markdown 構造（見出し・太字・リスト・引用・リンク）がレポート上で視覚的に活かされる
- 既存ユーティリティの拡張のため、Augur Chat や Morning Letter も自動的に新機能（hr, blockquote, link）の恩恵を受ける
- 外部依存なし — バンドルサイズへの影響ゼロ
- 49 テスト（既存 35 + 新規 14）で XSS 防御を含む包括的なカバレッジ

### Cons / Tradeoffs

- `parseMarkdown` は簡易実装であり、テーブル・ネストされたリスト・脚注など高度な Markdown 構文は非対応。LLM 出力がこれらを含む場合は生テキストとして表示される
- `{@html}` の使用は Svelte の XSS 保護をバイパスするが、`escapeHtml` による事前エスケープ + http/https 限定リンクで緩和済み
- `white-space: pre-wrap` の削除により、過去に生成された plain text レポートのレイアウトが若干変わる可能性がある（ただし `parseMarkdown` が改行を `<p>` に変換するため実質的な情報損失はない）
