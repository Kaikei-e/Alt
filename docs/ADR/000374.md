# 000374: Ask Augur RAGチャット機能の包括的改善 — セキュリティ・引用・パフォーマンス・マルチターン・UX

## ADR's STATUS

Accepted

## CONTEXT

Ask Augur（RAGチャット機能）に対して、ログ分析・コードレビュー・最新RAG研究(EMNLP 2024, MTRAG-UN benchmark等)を踏まえた包括的な調査を実施した結果、以下の問題が特定された。

### セキュリティ脆弱性

- `parseInline()` がHTMLエスケープせずにLLM出力を `{@html}` に挿入しており、XSS攻撃が可能な状態だった
- LLMが `<script>alert('xss')</script>` を含む回答を生成した場合、ブラウザ上で任意スクリプトが実行されうる

### Citations（引用）が常に0件

- プロンプトがLLMに `"chunk_id":"1"` 形式（1-basedインデックス）を指示しているにもかかわらず、`OutputValidator` はUUID形式のみ許可していた
- その結果、LLMが正しく引用を返してもバリデータが全て破棄し、ユーザーにはソースが表示されなかった
- `streamAugurChat()` の `onComplete` が done イベントとループ終了後の2回呼ばれる二重発火バグも存在

### パフォーマンス問題

- **ストリームパーシング O(n²)**: `rag_answer_stream.go` でチャンク到着のたびに `builder.String()` で全文再構築・再走査しており、長い回答で2次的に遅延
- **クエリ展開 74秒**: news-creator（3sタイムアウト）失敗後の ollama-legacy フォールバックが300秒タイムアウトを継承。loserがキャンセルされず無駄にリソース消費
- **日本語クエリに英語結果のみ**: `expandQueryLegacy` のプロンプトが英語バリエーションのみ生成

### マルチターン会話の未対応

- ハンドラが最後のユーザーメッセージのみ抽出し、会話履歴を全て破棄
- フロントエンドは会話全体を送信しているが、バックエンドが活用していない
- 「それについて詳しく」のような照応表現（coreference）を解決できない

### フロントエンドUX

- LLMのthinking（推論過程）をそのまま表示しており、ユーザーにとって有用でない
- バックエンドが送信する progress イベント（"searching"/"generating"）が未活用
- ストリーミング中にauto-scrollが動作せず、長い回答の途中から見えなくなる
- `Locale` が `"ja"` にハードコードされており、英語質問にも日本語設定が適用

## DECISION MAKING

5つのフェーズに分割し、セキュリティ最優先で一括実装する方針とした。

### Phase 1: XSS脆弱性の修正

`simpleMarkdown.ts` の `parseInline()` 関数の先頭で既存の `escapeHtml()` を呼び出し、HTMLエスケープを適用してからBold/Italic/Code変換を行う。

```typescript
function parseInline(text: string): string {
  text = escapeHtml(text); // XSS防止: HTMLエスケープを最初に適用
  text = text.replace(/\*\*(.*?)\*\*/g, "<strong>$1</strong>");
  // ...
}
```

`<script>`, `<img onerror=...>`, `onmouseover` 等のXSSベクターに対するテストケースを6件追加した。

### Phase 2: Citations バグ修正

#### インデックスベース引用の許可

`output_validator.go` の `Validate()` メソッドで、UUID形式に加えて1-basedインデックス（`"1"`, `"2"`, ...）も `allowed` マップに追加した。

```go
allowed := make(map[string]struct{}, len(contexts)*2)
for i, ctx := range contexts {
    allowed[ctx.ChunkID.String()] = struct{}{}
    allowed[fmt.Sprintf("%d", i+1)] = struct{}{}
}
```

既存の `buildCitations()` にはインデックスからUUIDへの解決ロジックが実装済みだったため、バリデータの修正のみで引用パイプライン全体が機能するようになった。

#### onComplete 二重発火の防止

`augur.ts` に `completeCalled` フラグを追加し、done イベントハンドラとループ終了後のフォールバック呼び出しの両方で重複チェックを行う。

### Phase 3: パフォーマンス最適化

#### O(n²) → O(n) インクリメンタルパーサー

`rag_answer_stream.go` のパーシングロジックを、`builder.String()` を毎チャンク呼ぶ方式から、`pending` バッファに新着データのみを蓄積して走査する方式に変更した。`pending.Reset()` で処理済みデータを破棄するため、メモリ使用量も一定に保たれる。

#### クエリ展開の高速化

`expand_queries.go` の `expandQuery()` に `context.WithCancel` を導入し、news-creator と ollama-legacy のレース戦略で勝者決定後に敗者をキャンセルする。`max_tokens` を200→100に削減し、プロンプトを日本語+英語の混合バリエーション生成に変更した。

```
Generate 3 to 5 diverse search queries for the user's input.
If the input is Japanese, include 1-2 Japanese variations and 2-3 English translations.
If the input is English, generate English variations.
```

### Phase 4: マルチターン会話サポート

#### 会話履歴の伝搬

以下の構造体に `ConversationHistory []domain.Message` フィールドを追加した:

- `AnswerWithRAGInput` (usecase層)
- `RetrieveContextInput` (usecase層)
- `StageContext` (retrieval層)
- `PromptInput` (prompt builder層)

ハンドラ (`handler.go`) が直近6メッセージ（3ターン）を抽出し、パイプライン全体に伝搬する。

#### 会話対応クエリリライティング

`RewriteQueryWithHistory()` 関数を新設した。会話履歴から直近3ターンをコンパクトに要約し、LLMに自己完結型の検索クエリへの書き換えを依頼する。

```
Conversation:
user: LLMの台頭が止まらないのは何ですか？
assistant: LLMの台頭が止まらない原因は...で、投資家が投資を加速させており...
Latest query: その投資家の動きについて詳しく教えて

Rewritten query: 投資家がLLMへ期待している理由について詳しく教えて
```

LLMエラー時は元のクエリにフォールバックする。この処理はクエリ展開の前段で実行される。

#### 生成プロンプトへの会話コンテキスト含有

`XMLPromptBuilder.Build()` に `### 会話履歴` セクションを追加し、直近3ターン（各メッセージ300文字制限）を生成プロンプトに含めることで、一貫性のある回答を生成する。

### Phase 5: フロントエンドUX改善

#### Progress段階表示（thinking 非表示化）

デスクトップ (`AugurChat.svelte`) とモバイル (`ChatWindow.svelte`) の両方で:

- `isThinking` state を `progressStage` state に置き換え
- `onProgress` コールバックを接続し、バックエンドの progress イベントを段階表示に反映
  - `"searching"` → "Searching knowledge base..."
  - `"generating"` → "Generating answer..."
- `onThinking` コールバックを `undefined` に設定（thinking非表示）

`ChatMessage.svelte` からは `thinking`/`isThinking` props と thinking表示UIを完全に削除した。

#### ストリーミング中のauto-scroll

delta受信時にスロットル付き（500ms間隔）の `scrollToBottom()` を実行する。ユーザーが上方向にスクロールしている場合（下端から100px以上離れている）はauto-scrollを抑制し、閲覧中の位置を保持する。

#### Locale動的検出

`handler.go` に `detectLocale()` ヘルパー関数を追加した。クエリ中のUnicode文字（Hiragana, Katakana, CJK Unified Ideographs）の比率が30%を超える場合は `"ja"`、それ以外は `"en"` を返す。ハードコードされていた `Locale: "ja"` を `detectLocale(query)` に置き換えた。

## RESULTS, EFFECTS

### PROS

- **XSS脆弱性の解消**: LLM出力に含まれるHTMLタグが全てエスケープされ、任意スクリプト実行が不可能になった
- **引用の正常表示**: LLMがインデックスベースで返す引用がバリデータを通過し、Sources欄に正しく表示される
- **クエリ展開の高速化**: loserキャンセルと `max_tokens` 削減により、最悪ケースの74秒が大幅に短縮。news-creator成功時は3秒以内で完了
- **ストリームパーシングの効率化**: O(n²)からO(n)への改善により、長い回答（10KB超）でのパーシング遅延が解消
- **日本語検索結果の改善**: クエリ展開プロンプトが日本語バリエーションを含むようになり、日本語記事の検索ヒット率が向上
- **マルチターン会話の実現**: 照応表現の解決により「それについて詳しく」「もう少し教えて」のような自然な会話が成立する
- **UXの改善**: Progress段階表示でユーザーが処理状況を把握でき、auto-scrollにより長い回答の追従が自然に行われる
- **Locale自動検出**: 英語質問には英語、日本語質問には日本語で回答が返り、ユーザー設定不要

### CONS, TRADEOFF

- **クエリリライティングによるレイテンシ増加**: マルチターン時にLLM呼び出しが1回増える（約3-5秒）。ただし `buildPrompt` フェーズ内でheartbeatが送信されるため、ユーザー体感への影響は限定的。LLMエラー時は元クエリにフォールバック
- **thinking表示の除去**: LLMの推論過程が非表示になり、デバッグ時の透明性が低下する。ただし、ユーザー向けには不要な情報であり、段階表示（searching/generating）の方が有用と判断
- **会話履歴によるプロンプトサイズ増加**: 直近3ターン（各300文字制限）をプロンプトに含めるため、コンテキストに使えるトークン数が減少する。既存のトークン制限ロジック (`maxPromptTokens`) が自動的にコンテキストチャンク数を調整するため、実用上の問題はない
- **キャッシュヒット率の低下**: 会話履歴をキャッシュキーに含めるため、マルチターン会話ではキャッシュが効きにくい。これは意図された動作であり、会話コンテキストに応じた回答を生成するために必要なトレードオフ
