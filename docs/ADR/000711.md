# ADR-000711: Acolyte レポート詳細画面に Brief/History 右パネル・生成ポーリング・更新通知バナーを導入

## Status

Accepted

## Date

2026-04-13

## Affected Services

- acolyte-orchestrator (Python / Connect-RPC / LangGraph) — `GetReport` に `scope` を追加、`ReportBrief` の逆変換ユーティリティ
- alt-frontend-sv (TypeScript / Svelte 5 Runes / TailwindCSS v4) — 詳細画面 (desktop + mobile) の UI 再構成・ポーリング状態管理・通知バナー
- proto/ — `Report.scope` フィールド追加と Python / TypeScript stub の再生成

## Context

Acolyte の `/acolyte/new` フォームでユーザーが入力する `title` / `reportType` / `topic` (scope map) は、`CreateReport` で backend に送信され `report_briefs` テーブルに永続化されていた。しかし `GetReport` のレスポンスに scope を含めていなかったため、詳細画面 (`/acolyte/reports/[id]`) では「自分が何を頼んだのか」を後から確認できなかった。別タブを跨いだり数時間後に戻ってきたときに、そのレポートをどういう意図で作らせたか思い出せないというフィードバックが発生していた。

さらに詳細画面の `Generate` ボタンにはフロー上の不整合が二件残っていた。

1. **ローカルフラグだけで駆動** — `generating` は `handleGenerate` 内で `true` にセットされるが、成功経路で `false` に戻す処理が欠けていた。成功直後は `loadReport()` で再レンダリングされるため画面上は動くが、`generating` 状態の真実源がクライアント側にしか存在しないので、タブを再オープンしたりリロードした瞬間に active run の有無を失う。結果、裏で動いているジョブを無視して二重起動するリスクがあった。
2. **同期的ブロッキング** — ボタン押下で即 `StartReportRun → loadReport` を直列に呼び、完了するまでユーザーに何も返さない構造。実際の生成は数十秒〜数分かかるため、UX 上は「押したのに何も起きていないように見える」期間が長かった。

最後に、生成完了時の画面更新は `loadReport` による自動リロードだった。しかしユーザーが旧エディションを読んでいる途中に勝手にページがリセットされる可能性があり、「ユーザーが読みたいものを勝手に書き換えない」という Alt の非侵襲的 UI 方針 (Acolyte header を含む既存の抑制的な設計) と齟齬を起こしていた。

これらを同じ設計パスで解消する必要があり、(a) scope の永続化と表示、(b) サーバー状態に基づくボタン無効化、(c) ユーザー主導の画面更新、の 3 点を 1 本の ADR で扱う。

## Decision

以下を採用する。

1. **`Report` proto に `scope` を追加し backend が `report_briefs` から逆射影して返す** — `GetReport` は `ReportBrief` を `map<string, string>` へ戻した値を `Report.scope` として返す。`ReportBrief.to_scope()` を `from_scope` の逆変換として追加し、`topic` / `time_range` / `entities` / `exclude` / 任意 constraints を単一関数でラウンドトリップさせる。
2. **詳細画面に Brief/History トグル式の右パネルを追加** — 既存の `with-history` グリッド (1fr / 260px) のレーンをそのまま共有し、`panelMode: "brief" | "history" | null` でヘッダー上のトグルボタンから切り替える。Alt-Paper (Newspaper Style) に沿った hairline rule + serif body の構成で、Copilot 風のカード / 影 / 角丸は使わない。2px rule 上下囲みは Acolyte Header 固有のパターンなので流用しない。
3. **`GetRunStatus` の 3 秒ポーリングで derived `isGenerating` を駆動** — `runStatus: RunStatus | null` を state とし、`isGenerating = runStatus === "pending" || runStatus === "running"` を derived state にする。`StartReportRun` の戻り値 `run_id` を `activeRunId` に保持し、`setInterval` + `onDestroy(stopPolling)` でリソース解放を保証。`succeeded / failed / cancelled` は終端状態として discriminated union (`TerminalStatus = "succeeded" | "failed" | "cancelled"`) で narrowing し、ポーリング停止時の条件分岐を型レベルで守る。
4. **成功時は自動リロードせず「UPDATED」バナーでユーザーに選ばせる** — `pendingUpdate = true` にセットし、`role="status"` + `aria-live="polite"` のバナーを画面上部に表示。`Refresh` で `loadReport()` を走らせ、`Dismiss` で本文を保持したままバナーのみ閉じる。読みかけコンテンツを勝手に差し替えない。

## Implementation Overview

### 1. proto 拡張と再生成

`proto/alt/acolyte/v1/acolyte.proto` の `message Report` に `map<string, string> scope = 7;` を追加。既存フィールド (`report_id` ... `latest_successful_run_id`, `created_at`) のタグは維持し、破壊的変更を回避した。`cd proto && buf generate --template buf.gen.acolyte.yaml` と `buf generate --template buf.gen.yaml --path alt/acolyte` で Python (`acolyte_pb2.py` / `.pyi`) と TypeScript (`acolyte_pb.ts`) を同時更新。

### 2. backend — `ReportBrief.to_scope()` と `get_report` ハンドラ

`acolyte-orchestrator/acolyte/domain/brief.py` に以下を追加。

```python
def to_scope(self) -> dict[str, str]:
    """Round-trip back to the proto map<string, string> representation."""
    scope: dict[str, str] = {"topic": self.topic}
    if self.time_range:
        scope["time_range"] = self.time_range
    if self.entities:
        scope["entities"] = ",".join(self.entities)
    if self.exclude_topics:
        scope["exclude"] = ",".join(self.exclude_topics)
    for k, v in self.constraints.items():
        scope[k] = v
    return scope
```

`acolyte/handler/connect_service.py` の `get_report` で `self._repo.get_brief(report_id)` を追加取得し、proto `Report.scope=brief.to_scope() if brief else {}` を埋める。

TDD は `tests/unit/test_get_report.py` に 2 件の失敗テストを先に追加 (scope を含むケース / brief 不在で空 map を返すケース)、RED を確認してから実装。501 件 → 504 件の unit suite を GREEN 維持。

### 3. frontend — Runes ベースの状態設計

`alt-frontend-sv/src/routes/(app)/acolyte/reports/[id]/+page.svelte` の書き換え要点:

- `RunStatus` / `TerminalStatus` を discriminated union として定義し、`isTerminalStatus` を type predicate で宣言 (`bp-typescript.md` に沿う)
- `isGenerating` は `$derived` でサーバー状態から導出。旧来の `generating` ローカルフラグを廃止
- `scopeEntries` は `$derived.by` で `topic → time_range → entities → exclude` の既知順 → その他の alphabetized tail、の順序に並べる
- ポーリングは `setInterval`。`TerminalStatus` で narrowing しつつ終端処理 (`pendingUpdate = true` / `error = run.failureMessage`) を行う
- クリーンアップは `onDestroy(stopPolling)` で保証。SvelteKit のページ遷移時にもタイマーは解放される
- `aria-disabled` / `aria-busy` / `title` を `disabled` とセットで揃え、disabled button UX の Best Practice (Smashing / Cloudscape) に従い理由を視覚・支援技術の両方に伝える

右パネルは `.detail-layout.with-side` という汎用レーンに改名し、`panelMode` でテンプレート分岐。ラベル表示は `formatScopeLabel` が `topic → "Topic"` / `time_range → "Time Range"` など人間可読に整形する。未知 key は Title Case 変換。

通知バナーは以下のマークアップで hairline rule ベースの Alt-Paper 表現を実現。

```svelte
{#if pendingUpdate}
  <div class="update-banner" role="status" aria-live="polite">
    <span class="update-kicker">Updated</span>
    <span class="update-rule" aria-hidden="true"></span>
    <p class="update-text">A new edition of this report has been generated.</p>
    <div class="update-actions">
      <button onclick={refreshAfterUpdate}>Refresh</button>
      <button onclick={dismissUpdate}>Dismiss</button>
    </div>
  </div>
{/if}
```

`border-top` / `border-bottom` の 1px rule、`Updated` の small caps kicker、`Refresh` (underline) / `Dismiss` (muted) というタイポグラフィ中心の表現にとどめ、背景塗り / 影 / modal overlay は使わない。

### 4. mobile 対応

`src/lib/components/mobile/acolyte/MobileAcolyteDetail.svelte` に `pendingUpdate` / `onRefresh` / `onDismissUpdate` を props として追加。Brief は disclosure (`showBrief`) として Generate ボタン直下にスタック表示し、横幅の狭い画面で右レーンを無理に作らない。バナーは同じ hairline style を縦に積んで配置。

### 5. コンテナ再ビルドと動作確認

```bash
docker compose -f compose/compose.yaml -p alt up --build -d \
  acolyte-orchestrator alt-frontend-sv
```

両サービスとも `Healthy`。`curl http://localhost:8090/health` → `{"status":"ok"}`、`curl http://localhost:4173/` → 200。scope round-trip の E2E:

```bash
curl -sS $BASE/CreateReport -d '{"title":"Smoke","reportType":"weekly_briefing",
  "scope":{"topic":"LLM safety 2026","time_range":"last_7_days",
  "entities":"openai,anthropic","audience":"researchers"}}'

# GetReport で以下が返る:
# "scope": {"topic":"LLM safety 2026","time_range":"last_7_days",
#           "entities":"openai,anthropic","audience":"researchers"}
```

`bun run check` (tsgo) 0 errors。`uv run pytest tests/unit/` 504 passed。

## Consequences

### Pros

- **レポート生成の意図が記録・可視化される** — 詳細画面右側に `Report Brief` として topic / time range / entities / exclude / custom constraints が常に見える。再生成の判断がユーザー側で完結する
- **サーバー真実源に基づくボタン制御** — タブ再オープン・リロード後も active run に応じて `Generate` が適切に無効化される (ポーリングで即時復帰)。二重起動リスクが消える
- **非侵襲的な更新通知** — 旧エディションを読んでいる最中に勝手に画面が差し替わらない。ユーザーが `Refresh` を押すまで現在の本文は安定する
- **型レベルでの終端状態ガード** — `RunStatus` / `TerminalStatus` の discriminated union により `TERMINAL_STATUSES.has(status)` の narrowing が保証され、`bp-typescript` が要求する exhaustiveness に寄せている
- **既存グリッドレーンを流用** — `with-history` を `with-side` にリネームして Brief / History が排他共存。追加レイアウト負債なし

### Cons / Tradeoffs

- **3 秒ポーリング** — StreamRunProgress (既存で UNIMPLEMENTED) を実装すれば長時間接続 1 本で済むが、現状はポーリング間隔ごとに 1 RPC 発生。数分以上かかるジョブではサーバー負荷が増える (現在の想定規模では許容)
- **初期ロード時の active run 復元** — `Report` proto は `active_run_id` を持たないため、ブラウザ側が過去に保持していた `activeRunId` を失うとリロード後に「生成中かどうか」を知る術がない。次段の改善としては `Report.active_run_id` フィールド追加か、`ListReportRuns` RPC の導入が考えられる
- **proto 破壊的変更ではないがフィールド 7 を確保** — 今後のフィールド追加時にタグ衝突を避けるため、Report へ新しい要素を加えるときはタグ 8 以降を使う運用ルールを維持する必要がある
- **LangGraph checkpoint と run status の整合** — `run_status` は `report_runs` テーブル上の状態で、checkpoint resume の結果と矛盾するエッジケースが理論上は残る (現時点では観測されていない)
