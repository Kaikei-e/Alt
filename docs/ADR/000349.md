# 000349: Tag Verse — Three.js 3Dタグクラウドビジュアライゼーションの実装

## ADR's STATUS

Accepted

## CONTEXT

Altの Explore セクションには Tag Trail（ランダムフィード＋タグ発見）が存在していたが、**タグの全体像を俯瞰できる機能**が欠けていた。ユーザーは個別のタグを辿ることはできても、「どのタグに記事が集中しているか」「自分のフィード全体のトピック分布がどうなっているか」を直感的に把握する手段がなかった。

Tag Verse は、全タグとその記事数を **3D空間の小惑星群** として可視化する機能である。球体のサイズと色は記事数に比例し、宇宙テーマの黒背景で Altデザイン言語の中でも異色の存在として位置づけられる。

**技術的背景**:

- Three.js エコシステムには Svelte 5 向けの宣言的バインディング **Threlte** (v8) が成熟しており、SvelteKit との統合が容易
- 既存の `feed_tags` + `article_tags` テーブルの集計クエリで必要なデータは取得可能
- フロントエンドの `fetchArticlesByTag` と `getThreeDayRecap` を再利用することで、HUDパネルのデータ取得に新規APIが不要

**制約**: デスクトップ専用。TDD-first。Clean Architecture 遵守。

## DECISION MAKING

### 1. バックエンド — FetchTagCloud エンドポイント

全タグと記事数を返す `FetchTagCloud` Connect-RPC エンドポイントを ArticleService に追加した。

**SQL クエリ**:

```sql
SELECT ft.tag_name, COUNT(DISTINCT at.article_id) as article_count
FROM feed_tags ft
INNER JOIN article_tags at ON ft.id = at.feed_tag_id
GROUP BY ft.tag_name
HAVING COUNT(DISTINCT at.article_id) > 0
ORDER BY article_count DESC
LIMIT $1
```

- `INNER JOIN` により記事が0件のタグは除外（HAVING句でも保証）
- `COUNT(DISTINCT at.article_id)` で同一記事への重複タグ付けを排除
- デフォルト200件、最大500件のリミット

**代替案の検討**:

| 案 | 評価 |
|----|------|
| マテリアライズドビュー | 現在のデータ規模では不要。GROUP BY + 既存インデックスで十分高速。将来の最適化候補として記録 |
| REST エンドポイント | Connect-RPC への統一方針に沿い、Proto定義 + Connect-RPC を採用 |
| **Connect-RPC エンドポイント（採用）** | `ArticleService` に `FetchTagCloud` RPC を追加。既存のトランスポート・認証基盤をそのまま利用 |

### 2. Clean Architecture の各層

```
Handler (FetchTagCloud) → FetchTagCloudUsecase → FetchTagCloudPort → FetchTagCloudGateway → AltDBRepository
```

| レイヤー | 責務 |
|---------|------|
| Domain | `TagCloudItem { TagName, ArticleCount }` — 純粋な値オブジェクト |
| Port | `FetchTagCloudPort` インターフェース |
| Driver | SQL クエリ実行、行スキャン |
| Gateway | nil チェック + ログ + driver 呼び出し |
| Usecase | limit バリデーション（0以下→200, 500超→エラー）|
| Handler | 認証チェック + limit クランプ(500上限) + Proto変換 |

### 3. フロントエンド — Three.js/Threlte アーキテクチャ

**パッケージ選定**:

- `three` — WebGL 3D レンダリングエンジン
- `@threlte/core` — Svelte 5 向け Three.js 宣言的バインディング（`<Canvas>`, `<T>`）
- `@threlte/extras` — `OrbitControls` 等のユーティリティコンポーネント

**SSR 対応**: Tag Verse ルートは `export const ssr = false` で CSR-only に設定。デスクトップレイアウト (`/desktop/`) 配下のため、既存の SSR 無効化と二重に保護。

**コンポーネント構成**:

```
TagVerseScreen.svelte          — ページオーケストレータ（データ取得、ローディング/エラー、HUD管理）
├── TagVerseScene.svelte       — Threlte Canvas + カメラ + ライティング + フォグ
│   ├── StarBackground.svelte  — 3000個の星パーティクル（BufferGeometry + Points）
│   ├── TagAsteroid.svelte     — 個別タグ球体（hover/click + 自転アニメーション）
│   └── PostProcessing.svelte  — Bloom エフェクト（EffectComposer + UnrealBloomPass）
└── TagVerseHUD.svelte         — グラスモーフィズムの右サイドパネル
    ├── HUDArticlesTab.svelte  — 既存 fetchArticlesByTag でページネーション付き記事一覧
    └── HUDRecapsTab.svelte    — getThreeDayRecap の topTerms/genre 名マッチング
```

### 4. 3D空間の設計判断

**球体配置 — フィボナッチ球面分布**:

ランダム配置ではなく黄金角ベースのフィボナッチ球面分布を採用。理由:

- 球面上に均等に点を分布させ、クラスタリングや空白を防ぐ
- 記事数の多い順にソートされたタグが球面上を螺旋状に配置される
- 決定的アルゴリズムのため、同じデータで同じ配置が再現される

**球体サイズ — べき乗スケーリング**:

```
radius = 0.3 + (articleCount / maxCount)^0.4 × 2.7
```

線形スケールではなくべき乗（指数0.4）を使用。記事数のロングテール分布で大半のタグが極小になるのを防ぎ、視覚的にバランスの取れたサイズ分布を実現。

**色グラデーション — 記事数に基づくデュアルエンコーディング**:

サイズと色の二重エンコーディングにより、直感的に記事数を把握できる:

| 記事数 | 色 | emissive 強度 |
|--------|-----|---------------|
| 少 | ダークブルー (#1a3a5c) | 低 (0.1) |
| 中 | シアン (#00d4ff) | 中 (0.4) |
| 多 | オレンジ (#ff8c00) | 高 (1.0) |

`THREE.Color.lerpColors()` で正規化値に基づきスムーズ補間。

**ポストプロセッシング — Bloom**:

`UnrealBloomPass` (threshold: 0.3, strength: 1.0, radius: 0.4) により、emissive の強い球体（記事数が多いタグ）が宇宙的な発光感を持つ。

### 5. HUDパネルの設計

**新規APIを作らない方針**: Articles タブと Recaps タブの両方とも既存の Connect-RPC 関数を再利用。

- **Articles タブ**: `fetchArticlesByTag(transport, tagName)` — カーソルベースページネーション付き
- **Recaps タブ**: `getThreeDayRecap(transport)` を1回だけfetchしてキャッシュし、`topTerms` と `genre` 名でタグ名をマッチング

**Recap マッチングロジック**:

```
genre.topTerms.some(term => term.toLowerCase().includes(tagName.toLowerCase()))
|| genre.genre.toLowerCase().includes(tagName.toLowerCase())
```

fuzzy マッチのため false negative が生じうるが、recap のジャンル分類とタグ名の粒度差を考慮すると、完全一致よりも部分一致の方が有用と判断。

## RESULTS, EFFECTS

### PROS

- **タグの全体像を直感的に把握** — 3D空間でサイズ・色の二重エンコーディングにより、どのトピックに記事が集中しているかが一目でわかる
- **既存APIの再利用** — HUD の Articles/Recaps タブは新規 API なしで既存の `fetchArticlesByTag` と `getThreeDayRecap` を活用
- **Clean Architecture 完全遵守** — Domain → Port → Gateway → Usecase → Handler の各層を新規ファイルとして追加。既存コードへの影響は DI ワイヤリングと Handler メソッド追加のみ
- **TDD** — Usecase のバリデーション（6テスト）、Gateway の nil チェックをテスト先行で実装
- **パフォーマンス** — フィボナッチ球面分布は O(n) で計算可能。200タグ以下なら個別 Mesh でも十分高速（将来 200超は InstancedMesh で対応可能）
- **デスクトップ専用の割り切り** — モバイルでの 3D WebGL はバッテリー・パフォーマンスの問題があるため、デスクトップ専用として SSR を無効化

### CONS, TRADEOFF

- **バンドルサイズの増加** — Three.js は gzip 後約600KB。SvelteKit のルートベースコード分割により Tag Verse ページにアクセスしない限りロードされないが、初回アクセス時のロード時間は増加する
- **Three.js のテスト困難性** — WebGL コンテキストが必要な 3D コンポーネントのユニットテストは省略した。ビジュアル確認は手動または E2E テストで対応
- **Recap マッチングの精度** — `topTerms` との部分一致は fuzzy であり、false negative（タグと関連する recap を見逃す）が生じうる。完全なタグ↔recap 紐付けには DB スキーマの拡張が必要だが、現時点では YAGNI
- **WebGL 非対応ブラウザ** — 極めて古いブラウザでは動作しない。Threlte がデフォルトで WebGL コンテキスト作成に失敗した場合のフォールバック UI は未実装（黒画面のまま）。将来的に `<canvas>` 要素の存在チェックを追加する余地あり
- **集計クエリのパフォーマンス** — `GROUP BY` + `COUNT(DISTINCT)` はタグ・記事数が数万〜数十万に達した場合に遅くなる可能性がある。マテリアライズドビューや定期集計テーブルは将来の最適化候補

## APPENDIX

- `feed_tags` テーブルは tag-generator サービスが記事に対して自動生成したタグを格納
- `article_tags` テーブルは `feed_tags` と `articles` の多対多リレーション
- Tag Trail（ADR-167, 168, 169）の既存インフラ（`fetchArticlesByTag`, `fetchArticleTags`, `fetchRandomFeed`）を HUD パネルで再利用
- Threlte v8 は Svelte 5 Runes に完全対応しており、`$state`/`$derived` との統合が自然
- `buf generate` で Go / TypeScript 両方の Proto コードを再生成済み
