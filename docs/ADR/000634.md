# ADR-000634: Morning Letter v2 Phase 3 の document-oriented read API と alt-backend 統合を実装する

## Status

Accepted

## Date

2026-04-07

## Affected Services

- alt-backend (Go)
- recap-worker (Rust)

## Context

[[000633]] で Morning Letter v2 の Phase 0-2 を実装し、3days Recap 品質改善・prompt 分離・hierarchical reduce 改善・fallback 改善・Morning Letter document 生成・永続化が完了した。

しかし、生成された Morning Letter document をフロントエンドから取得する手段がなかった。既存の proto は `StreamChat` RPC のみを持ち、document-first な読み取りには対応していない。Phase 3 では以下を解決する必要があった:

1. **Read API の不在**: Morning Letter document を取得する API がなく、フロントエンドが document-first 体験を実装できない
2. **proto service 分離の必要性**: 既存 `MorningLetterService` に RPC を追加すると、`rag-orchestrator` が実装する `MorningLetterServiceHandler` にも新メソッドの実装が必要になり、ビルド破壊リスクがある
3. **購読フィルタリング**: Morning Letter は全ユーザ共通で事前生成されるが、ソース provenance は購読フィードで絞る必要がある
4. **Gemma 4 出力の堅牢性**: `result_jsonb` は Gemma 4 で生成されており、[[000632]] で報告された structured output の不安定性に対する防御が必要

## Decision

**Morning Letter v2 Phase 3 として、`MorningLetterReadService` を独立した proto service として定義し、recap-worker に REST read エンドポイントを追加し、alt-backend で Clean Architecture に沿った統合を行う。**

### 1. Proto: `MorningLetterReadService` の独立定義

`MorningLetterService`（StreamChat を持つ既存 service）とは別に `MorningLetterReadService` を新設した。これにより `rag-orchestrator` の `MorningLetterServiceHandler` に影響を与えず、alt-backend 側だけで閉じた変更になる。

新規 RPC:
- `GetLatestLetter` — edition timezone 基準での最新号を返す
- `GetLetterByDate` — civil date 指定で取得
- `GetLetterSources` — section-article provenance を購読フィルタ済みで返す

API 設計上の改善（AIP 準拠）:
- `google.protobuf.Timestamp` で `created_at` / `generated_at` を型付け（AIP-142）
- `string etag` を `MorningLetterDocument` に追加（AIP-154: resource freshness validation）
- `MorningLetterSourceType` を enum として定義（AIP-126: UNSPECIFIED / RECAP / OVERNIGHT）
- `target_date` は `google.type.Date` の導入に buf dependency 追加が必要なため、`string` (ISO 8601 "YYYY-MM-DD") + ドキュメントで対応

### 2. recap-worker: REST read エンドポイント

Resource-oriented な REST path を採用:
- `GET /v1/morning/letters/latest`
- `GET /v1/morning/letters/{target_date}`
- `GET /v1/morning/letters/{letter_id}/sources`

DAO 層に `get_morning_letter_sources` メソッドを追加（trait / compat / mock / unified / impl の全層に伝播）。

Gemma 4 の `result_jsonb` は `parse_morning_letter_body()` で fail-closed に処理:
- `schema_version != 1` → 500（未知スキーマの warn log + letter_id/version/model をログ属性に）
- `serde_json::from_value::<MorningLetterContent>()` 失敗 → 500（payload 原文はログに出さない）

レスポンスヘッダに `ETag` と `Last-Modified` を付与し、将来の `If-None-Match` / `304 Not Modified` 対応の土台を作った。

### 3. alt-backend: Clean Architecture に沿った統合

データフロー:
```
Frontend → alt-backend (Connect-RPC) → recap-worker (REST) → PostgreSQL
```

Clean Architecture レイヤー構成:

| Layer | File | 責務 |
|-------|------|------|
| Domain | `domain/morning.go` | `MorningLetterDocument`, `MorningLetterBody`, `MorningLetterSection`, `MorningLetterSourceEntry` |
| Port | `port/morning_letter_port/morning.go` | `MorningLetterRepository`, `MorningLetterUsecase` interface |
| Usecase | `usecase/morning_usecase/morning_letter_usecase.go` | 日付 validation、購読フィルタリング |
| Gateway | `gateway/morning_gateway/morning_letter_gateway.go` | recap-worker REST 呼び出し + articles JOIN for FeedID |
| Handler | `connect/v2/morning_letter/handler.go` | 既存 `MorningLetterServiceHandler` + 新 `MorningLetterReadServiceHandler` を同一 Handler で実装 |

### 4. 購読フィルタリング

Morning Letter は新聞の社説と同様、全文を返却する。ソース provenance のみ購読フィードでフィルタする:
- Gateway が recap-worker からソースを取得後、`FetchArticlesByIDs()` で `feed_id` を JOIN
- Usecase が `UserFeedPort.GetUserFeedIDs()` で購読フィードを取得し、`FeedID` でフィルタ
- article が DB に存在しないソースは drop + warn log（サイレントに消さない）

### 検討した代替案

| 案 | 評価 | 採否 |
|---|---|---|
| 既存 `MorningLetterService` に RPC 追加 | シンプルだが `rag-orchestrator` のビルド破壊リスク | 却下 |
| proto で `google.type.Date` 使用 | AIP-142 準拠だが buf dependency 追加が必要 | 将来検討 |
| ソース provenance にも本文フィルタ適用 | ドキュメントが歯抜けになるリスク、実装複雑 | 却下 |
| `result_jsonb` をそのまま proto に通す | Gemma 4 quirks がフロントエンドに伝播 | 却下 |

## Consequences

### Pros

- Morning Letter document をフロントエンドから 1 RTT で取得可能になり、Phase 4（frontend document-first 改修）の前提が整った
- `MorningLetterReadService` の独立により、rag-orchestrator への影響がゼロ
- `ETag` / `Last-Modified` ヘッダにより将来のキャッシュ最適化が可能
- 購読フィルタリングがソース層のみに適用され、ドキュメントの一貫性が保たれる
- Gemma 4 の `result_jsonb` が recap-worker でデシリアライズされ、不正データがフロントエンドに到達しない

### Cons / Tradeoffs

- `GetLetterSources` で articles テーブルとの追加 JOIN が発生する（10-30 件の batch query で実用上問題なし）
- `target_date` が `string` のまま（`google.type.Date` 導入は buf dependency 追加を伴うため先送り）
- alt-backend → recap-worker 間の新しい HTTP 契約に CDC/contract test がまだない（別途追加予定）
- Morning Letter が未生成の状態では全エンドポイントが 404 を返す（Phase 5 の daemon 再有効化が必要）
