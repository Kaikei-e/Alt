# 372: TagVerse WebGPURenderer マテリアル破棄順序バグの修正

## ADR's STATUS

Accepted

## CONTEXT

Desktop UI の TagVerse 画面（`/feeds/tag-verse`）から他の画面へナビゲーション離脱する際、ブラウザコンソールに以下のエラーが発生していた。

```
Uncaught(in promise) TypeError: Cannot read properties of undefined (reading 'usedTimes')
    at NodeManager.delete
    at renderObject.onDispose
    at RenderObject.dispose
    at RenderObject.onMaterialDispose
    at PointsMaterial/MeshStandardMaterial.dispatchEvent
    at Material.dispose
    at Map.forEach
    at Object.dispose
```

TagVerse は Three.js `^0.183.2` + Threlte v8 + `WebGPURenderer` で 3D タグクラウドを描画している。画面離脱時に `<Canvas>` がアンマウントされると、以下の 2 段階の破棄が走る:

1. **Renderer.dispose()**: `NodeManager` の内部 `WeakMap` を `new WeakMap()` でクリア（`DataMap.dispose()`）
2. **Threlte のシーンクリーンアップ**（Renderer.dispose() とは別パス）: シーン内の全マテリアルを `Map.forEach` で一括 dispose

ステップ 2 で `Material.dispose()` が発火すると、`RenderObject.onMaterialDispose` → `RenderObject.dispose()` → `renderObject.onDispose()` → `NodeManager.delete()` が呼ばれる。しかしステップ 1 で既に `NodeManager` の `WeakMap` がクリア済みのため、`this.get(object)` は空の `{}` を返し、`{}.nodeBuilderState` が `undefined` → `undefined.usedTimes` で TypeError が発生する。

この問題は Three.js WebGPURenderer の内部破棄順序と Threlte のクリーンアップの相互作用に起因する。同種の問題は [threlte/threlte#1667](https://github.com/threlte/threlte/issues/1667) でも報告されている。

## DECISION MAKING

3 層の防御で修正する。

### 層 1: シーン全マテリアルの先行 dispose（根本対策）

`SceneContent.svelte` の `onDestroy` でシーンをトラバースし、全マテリアルを先行 dispose する。

Svelte のコンポーネント破棄順序では子（SceneContent）が親（Canvas）より先に teardown されるため、この時点では `NodeManager` のデータはまだ有効。`Material.dispose()` → `RenderObject.onMaterialDispose` → `RenderObject.dispose()` が正常に完了し、さらに `RenderObject.dispose()` 内で `material.removeEventListener('dispose', this.onMaterialDispose)` が実行されるため、後から Threlte が同じマテリアルを再度 dispose してもリスナーが存在せず `NodeManager.delete` は呼ばれない。

### 層 2: StarBackground の PointsMaterial 手動管理

`StarBackground.svelte` で `<T.PointsMaterial>` による宣言的生成をやめ、`new THREE.PointsMaterial()` でスクリプト内に生成し `onDestroy` で手動 dispose する。層 1 と重複するが、このコンポーネント単体で完結するため保守性が高い。

### 層 3: Renderer dispose の try-catch セーフティネット

`TagVerseScene.svelte` の `createRenderer` で `WebGPURenderer.dispose` を try-catch でラップする。`Renderer.dispose()` 内部で発生しうる同期エラーに対するセーフティネット。ただし今回のバグは `renderer.dispose()` の外側で発生するため、これ単体では不十分（層 1 が必要な理由）。

## RESULTS, EFFECTS

### PROS

- `PointsMaterial` と `MeshStandardMaterial` 両方の `usedTimes` TypeError が解消される
- 3 層の防御により、将来新しいマテリアルタイプが追加されても同じ問題は発生しない
- Three.js 本体で修正されるまでの堅牢なワークアラウンド
- 変更は 3 ファイルのみで影響範囲が小さい

### CONS, TRADEOFF

- シーン全走査による先行 dispose は Three.js 内部の破棄メカニズムに依存しており、Three.js のメジャーバージョンアップ時に見直しが必要な可能性がある
- `StarBackground.svelte` のマテリアルが宣言的でなくなるため、props でのマテリアル変更が若干煩雑になる（現状 props による変更は行っていない）
- 層 3 の try-catch は `renderer.dispose()` 内の真正なエラーも握りつぶす可能性がある（ただし破棄処理のため実害は極めて低い）

### 検証方法

1. TagVerse 画面を開き、3D シーンが正常にレンダリングされることを確認
2. 別の画面に遷移し、コンソールに `usedTimes` エラーが出ないことを確認
3. TagVerse 画面に再遷移し、正常にレンダリングされることを確認
