# ADR-000635: Morning Letter v2 Phase 4 の document-first UI と follow-up chat 再定義を実装する

## Status

Accepted

## Date

2026-04-07

## Affected Services

- alt-frontend-sv (TypeScript / SvelteKit)
- rag-orchestrator (Go)

## Context

[[000634]] で Morning Letter v2 Phase 3（document-oriented read API + alt-backend 統合）を完了し、フロントエンドから Morning Letter document を取得する API 基盤が整った。しかし、フロントエンド側は依然として chat-first（`MorningLetterChat` コンポーネントのみ）であり、Phase 3 で追加された `MorningLetterReadService` の 3 つの RPC（`GetLatestLetter`, `GetLetterByDate`, `GetLetterSources`）は未消費だった。

### 解決すべき課題

1. **Chat-first UI の限界**: ページを開くと「Ask about today's news」が主役であり、朝刊としての体験が弱い。document-first に転換し、開いた瞬間に briefing document が見える必要がある
2. **初期表示の遅延**: `onMount` + client-side fetch では server render が活かされず、空ページ → 後読み込みになる。SvelteKit の `load` 関数を使い、page data として letter を先読みすべき
3. **URL 非永続**: 日付切り替えが URL に反映されず、refresh/share/back-forward で状態が再現できない
4. **Follow-up chat の grounding 不在**: chat が「今見ている letter」を識別する経路がなく、rag-orchestrator が常に最新 letter に grounding するため、過去日閲覧中のズレが発生する
5. **Sources の直列 fetch**: letter + sources を直列で blocking すると document-first の初期表示が鈍る
6. **アクセシビリティ**: モバイルの折りたたみ chat に Disclosure pattern・`aria-live` が不在

## Decision

**Morning Letter ページを document-first に再構成し、chat を follow-up assistant に再定義する。SvelteKit `load` による先読み、`target_date` proto フィールド追加による letter-specific grounding、共有コア renderer による desktop/mobile 統一を採用する。**

### 1. SvelteKit `load` による初期表示（Issue 10）

`+page.ts` を新設し、universal `load` 関数で letter を page data として先読みする。URL `?date=YYYY-MM-DD` が source of truth。

```
+page.ts: load({ url }) → getLatestLetter() or getLetterByDate(date)
+page.svelte: data.letter を hook の initialLetter として渡す
```

`ssr = false`（Connect-RPC transport はブラウザコンテキスト必須）だが、SvelteKit の navigation/hydration では `onMount` より先に `load` が実行されるため、初期表示の遅延を軽減する。

### 2. letter-first, sources-lazy の分離ローディング

letter は `load` で blocking、sources は client-side で後追い。2 つの独立した loading state:
- `letterLoading`: 主コンテンツ → skeleton 表示
- `sourcesLoading`: 非 blocking → source links に spinner → populate

### 3. 共有ドキュメント core renderer

Desktop/Mobile で本文 renderer を丸ごと 2 系統にすると差分事故が増えるため、共有 core component を導入した:

| Component | Role |
|-----------|------|
| `MorningLetterDocumentCore.svelte` | Lead + sections + degraded badge + date |
| `MorningLetterSectionCore.svelte` | Title (Playfair Display) + bullets (Source Sans 3) + source links |
| `MorningLetterSkeleton.svelte` | Warm gray skeleton、CLS 防止 min-height |
| `MorningLetterEmpty.svelte` | 空状態 |

Desktop/Mobile は layout wrapper のみ（2/3+1/3 分割 vs 全幅 + disclosure）。

### 4. Proto `target_date` フィールド追加（Issue 11）

`StreamChatRequest` に `optional string target_date = 3` を追加。フロントエンドが閲覧中の letter の civil date を送信し、rag-orchestrator が対応する letter を fetch して chat context に grounding する。

`letter_id` ではなく `target_date` を選択した理由:
- 再生成でも安定（ID は変わるが date は不変）
- URL shareable（`?date=YYYY-MM-DD`）
- recap-worker の `GET /v1/morning/letters/{target_date}` とそのまま対応

### 5. rag-orchestrator の letter context grounding

`MorningLetterFetcher` interface を domain 層に定義し、recap-worker REST client を adapter 層に実装。handler が `StreamChat` 受信時に:

1. `target_date` が指定されていれば `FetchByDate()` で対象 letter を取得
2. 未指定なら `FetchLatest()` で最新を取得
3. 取得できた場合は `<morning_letter>` XML タグで構造化して `AnswerWithRAGInput.LetterContext` に渡す
4. 取得失敗は non-fatal（warn log のみ、従来の article-based RAG にフォールバック）

### 6. アクセシビリティ

- モバイル chat toggle: `<button aria-expanded aria-controls="follow-up-chat">` (WAI-ARIA Disclosure pattern)
- Chat message list: `role="log"` + `aria-live="polite"`
- Streaming 中: `aria-busy="true"`

### 7. CLS 防止

Desktop document area: `min-h-[60vh]`、chat panel: `min-h-[40vh]`。Skeleton widths は typical content に合わせて reflow を防止。

### 8. `withinHours` の historical letter 整合

`deriveWithinHours(targetDate)` — 今日なら 24h、過去日なら target date の朝から現在までの時間を計算（上限 168h = 7 days）。これにより過去日の letter を閲覧中に「直近 24 時間」へ retrieval が飛ぶ問題を回避。

### 検討した代替案

| 案 | 評価 | 採否 |
|---|---|---|
| `onMount` で fetch | SvelteKit `load` の方が初回表示が速い。server render 不活用 | 却下 |
| `letter_id` を proto に追加 | 再生成で変わる。URL shareable ではない | 却下 |
| rag-orchestrator が alt-backend Connect-RPC 経由で letter 取得 | 循環依存（alt-backend → rag-orchestrator → alt-backend） | 却下 |
| Desktop/Mobile で renderer 完全複製 | 差分事故リスク大 | 却下 |
| `+page.server.ts` (server load) | Connect-RPC transport がブラウザ専用、server context 不可 | 却下 |

## Consequences

### Pros

- ページ初期表示が document になり、chat-first から document-first に転換
- URL `?date=YYYY-MM-DD` で日付指定・refresh・share・back-forward が可能
- Follow-up chat が閲覧中の letter に grounding され、過去日閲覧時のズレが解消
- 共有 core renderer により desktop/mobile の保守コストが低減
- letter-first, sources-lazy により初期表示が高速化
- WAI-ARIA Disclosure pattern + `aria-live` でアクセシビリティ改善
- TDD で 48 frontend テスト + 12 backend パッケージ全 pass

### Cons / Tradeoffs

- `ssr = false` のため、初回 HTML レスポンスは空（Connect-RPC 制約）。`load` は hydration 時に実行されるが、真の SSR ではない
- rag-orchestrator → recap-worker REST の新しい service 依存が追加。毎 `StreamChat` で 1 RTT 増加（ただし fail は non-fatal）
- `target_date` proto フィールド追加により全 6 サービスの proto 再生成が必要だった
- Morning Letter が未生成の状態では全エンドポイントが 404 を返す（Phase 5 の daemon 再有効化が必要）
- CDC/contract test for rag-orchestrator ↔ recap-worker HTTP boundary がまだない（別途追加予定）
