# フィード読み取りパスの共有 Payload Cache 導入（フェーズ1）

## ADR's STATUS

Accepted

## CONTEXT

ADR-364 でフィード読み取りパスの区間計測を追加した結果、80/20 偏在アクセス下の 3000VU 負荷試験で `http_req_waiting p95=4.70s` まで悪化していること、かつ待ち時間の主因がネットワーク転送ではなくサーバー内部処理であることが確認された。

既存の `GetUnreadFeeds` / `GetAllFeeds` / `FetchArticlesCursor` は、レスポンスに `content` を含む UX を維持するために、ユーザーごとに重い DB クエリを毎回実行していた。特に人気フィードでは、全ユーザーで共通な payload を毎回組み立て直しており、以下の非効率が発生していた。

1. **共通データの重複組み立て**: 同じ `feed_link_id` の最新フィード群をユーザーごとに再クエリしている
2. **記事 content の重複取得**: 同じ `article_id` の本文をページングごとに再取得している
3. **ユーザー状態との密結合**: 共通 payload と user-specific な read status が単一クエリに混在している

目標は、レスポンス仕様を変更せずに、共有可能な payload をインプロセスキャッシュへ分離し、ユーザー固有状態のみを軽量クエリで合成することで、フィード読み取りパスの待ち時間を削減することだった。

## DECISION MAKING

### 1. 共有 payload とユーザー状態を 3 層に分離する

以下の 3 層へ責務を分離した。

| 層 | 役割 | 共有範囲 |
|---|---|---|
| Feed Page Cache | `feed_link_id` ごとの最新フィード一覧 | 全ユーザー共通 |
| Article Content Cache | `article_id` ごとの本文・タグ | 全ユーザー共通 |
| User Read State | 購読一覧 / read feed ids | ユーザー単位 |

この分離により、共通 payload はキャッシュヒット時に DB を回避し、ユーザーごとの差分だけを最後に適用する構造へ変更した。

### 2. 汎用 `SharedCache` を導入する

`alt-backend/app/utils/cache/shared_cache.go` に、以下の特性を持つ汎用インメモリキャッシュを新設した。

- TTL
- stale-while-revalidate
- `singleflight` による cache miss 重複排除
- `Invalidate` / `InvalidateAll`

このキャッシュを共通基盤とし、個別 gateway ごとに TTL を変えて再利用する方針を採用した。

### 3. `GetUnreadFeeds` / `GetAllFeeds` は新 `CachedFeedListUsecase` に切り替える

`FetchUnreadFeedsListCursorUsecase` / `FetchFeedsListCursorUsecase` の既存経路は残しつつ、Connect / REST の unread/all 経路は新設した `CachedFeedListUsecase` に切り替えた。

新フローは以下の通り:

1. `GetUserSubscriptions(user_id)` で購読 `feed_link_id` 一覧を取得
2. 各 `feed_link_id` について `FeedPageCache.GetFeedPage()` を呼ぶ
3. `feed_link_id IS NULL` の orphan feed は既存 gateway から別取得する
4. 全件を `created_at DESC` 相当でマージ・ソートする
5. cursor を適用する
6. `GetReadFeedIDs(user_id, feed_ids)` で既読状態だけを軽量取得する
7. unread は既読除外、all は `IsRead` フラグ付与のみ行う

この切り替え対象は負荷ボトルネックの中心である `GetUnreadFeeds` / `GetAllFeeds` に限定した。`GetReadFeeds` / `GetFavoriteFeeds` は Phase 1 では既存 DB 直経路を維持する。

### 4. `FetchArticlesCursor` は ID 先行取得 + content cache に変更する

記事一覧 API では、まず軽量な `FetchArticleIDsWithCursor` で `article_id` 群のみを取得し、その後 `ArticleContentCache.GetArticles(ids)` で本文を復元する方式に変更した。

これにより:

- ページング境界判定は軽量クエリで行える
- 同一記事本文は複数ユーザー・複数リクエスト間で共有できる
- partial miss 時は不足 ID のみ DB から補完できる

### 5. フィード登録時に feed page cache を invalidate する

`RegisterFeedsUsecase` に feed page invalidator を注入し、`feed_link_id` が判明した場合は登録処理の前段で該当 cache を明示 invalidation するようにした。

Phase 1 では article content 側は TTL ベースのみとし、明示 invalidation は導入しない。

### 6. インフラ追加は行わず、インプロセスキャッシュに留める

Redis 等の外部キャッシュは導入しない。理由は以下の通り。

- 現フェーズの目的は DB 起点の待ち時間削減であり、まずは最小変更で効果確認したい
- 既存 Compose 構成を複雑化させずに導入できる
- 将来 Redis 化する場合も、cache interface と invalidation ポイントを先に整理できる

## RESULTS, EFFECTS

### PROS

- **共通 payload の再利用**: 人気 feed / article ほど cache hit 率が上がり、ユーザーごとの重複 DB 組み立てを減らせる
- **責務分離の明確化**: feed payload、article content、user read state が分離され、今後の最適化ポイントが明確になった
- **既存レスポンス仕様を維持**: `content` を含む UX や API 形式を変えずに内部実装のみ置き換えられる
- **段階導入が容易**: 最もホットな unread/all/articles から先に切り替え、read/favorite は後続フェーズへ分離できる
- **将来の Redis 化に前方互換**: cache gateway と invalidation ポイントを先に導入したことで、インフラ移行時の差し替え面積を限定できる

### CONS, TRADEOFF

- **インプロセス cache のため多 replica 間で共有されない**: Pod / container ごとに cache が独立し、完全な共有には Redis 等が必要
- **メモリ使用量が増加する**: 人気 feed と article content をプロセス内保持するため、backend コンテナのメモリフットプリントは増える
- **orphan feed は依然として fallback が必要**: `feed_link_id IS NULL` データは別経路で扱うため、完全統一には追加のデータ整備が必要
- **read/favorite は未最適化**: Phase 1 の対象外としたため、該当 endpoint の負荷改善は限定的
- **stale window による短時間の古い応答**: stale-while-revalidate を採用したため、更新直後に古い payload が返る短い窓が存在する

### 検証結果

- `cd alt-backend/app && go test ./...` — 全テスト通過
- `docker compose -f compose/compose.yaml build alt-backend` — 成功
- `docker compose -f compose/compose.yaml up -d alt-backend` — 成功
- `docker compose -f compose/compose.yaml ps alt-backend` — `healthy` 確認
- `curl http://localhost:9000/v1/health` — `{"database":"connected","status":"healthy"}`

### 今回の非対象

- `GetReadFeeds` / `GetFavoriteFeeds` の cache 化
- hourly feed collector 側の cache invalidation
- `cache_lookup_ms` / `db_ms` / `merge_ms` などの追加区間計測
- Redis 等の外部キャッシュ導入
