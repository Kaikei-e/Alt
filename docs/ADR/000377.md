# 000377: Tag Verse 3Dタグクラウド パフォーマンス最適化 — レイアウト早期収束・キャッシュウォーミング・レンダリング軽量化

## ADR's STATUS

Accepted

## CONTEXT

Tag Verse は最大300個のタグを3D空間に球体として配置し、Three.js（Threlte + WebGPU）でレンダリングする機能である。タグ間の共起関係を力学シミュレーション（Barnes-Hut octree）で空間配置に反映し、直感的な可視化を提供する。

### パフォーマンス上の課題

| 領域 | ボトルネック | 影響 |
|------|-------------|------|
| **バックエンド: レイアウト計算** | 300 iteration 固定の力学シミュレーション。300ノードで ~44ms | キャッシュミス時のレスポンス遅延 |
| **バックエンド: キャッシュ** | 30分TTLの in-memory キャッシュ。サーバー起動直後はコールドスタート | 初回リクエストで DB + 共起取得 + レイアウト計算の全コスト |
| **フロントエンド: メッシュ** | タグごとに個別の `SphereGeometry(1, 32, 32)` で 1024三角形/球 × 300 = 307,200三角形 | GPU負荷 |
| **フロントエンド: フレームループ** | タグごとに個別の `useTask` コールバックで `getWorldDirection()` を300回/フレーム呼び出し | CPU負荷（毎フレーム） |

### 設計方針

- **仕様変更なし**: API 契約、タグごとの色・発光感・ホバー挙動、ラベル表示をすべて維持する
- **実装コスト対効果**: 効果が高く、リスクが低い施策から優先する
- **TDD-first**: バックエンドは失敗テスト先行で実装する

### 採用しなかった施策

| 施策 | 却下理由 |
|------|---------|
| スーパーセットキャッシュ（`limit=300` の結果を `limit=200` に流用） | `limit` ごとに共起・レイアウトを計算するため、先頭N件を切り出すと座標系の意味が変わる |
| ラベル最大50件制限 | 表示タグ名が減るため仕様変更にあたる |
| `InstancedMesh` 化 | per-instance の `emissiveIntensity` 差分、hover 時の拡大、tooltip 連携の差分が大きく、PoC で成立確認後に別フェーズで実施する |

## DECISION MAKING

### 改善1: キャッシュウォーミング（バックエンド）

既存の `JobScheduler` パターン（`OgpImageWarmerJob` と同構造）に従い、`tag-cloud-cache-warmer` ジョブを追加した。

**`alt-backend/app/job/tag_cloud_cache_warmer.go`** — ジョブ関数を返す構造で、テスタビリティのために `tagCloudExecutor` インターフェースを導入した。

```go
type tagCloudExecutor interface {
    Execute(ctx context.Context, limit int) (any, error)
}

func TagCloudCacheWarmerJob(usecase *fetch_tag_cloud_usecase.FetchTagCloudUsecase) func(ctx context.Context) error {
    // nil チェック付きで tagCloudCacheWarmerJobFn に委譲
}
```

**`alt-backend/app/main.go`** — スケジューラに24分間隔（30分TTLの80%）で登録。`JobScheduler.Start()` は起動時に即1回実行するため、コールドスタートが解消される。

```go
scheduler.Add(job.Job{
    Name:     "tag-cloud-cache-warmer",
    Interval: 24 * time.Minute,
    Timeout:  2 * time.Minute,
    Fn:       job.TagCloudCacheWarmerJob(container.FetchTagCloudUsecase),
})
```

### 改善2: レイアウト早期収束（バックエンド）

力学シミュレーションの収束プロファイルを計測し、安全な早期終了を導入した。

#### 計測フェーズ

**`alt-backend/app/usecase/fetch_tag_cloud_usecase/layout.go`** — `ComputeLayout` を `computeLayoutInternal` に委譲するリファクタリングを行い、`ComputeLayoutWithProfile` で iteration ごとの最大変位量を記録可能にした。

**`alt-backend/app/usecase/fetch_tag_cloud_usecase/layout_test.go`** — 100/200/300/500ノードのベンチマークと、疎・密エッジのバリエーションを追加した。`TestComputeLayout_ConvergenceProfile` で複数閾値の収束ポイントを分析した。

計測結果（300ノード、`initialRadius = 138.6`）:

| 閾値比率 | 絶対閾値 | 収束 iteration | 削減率 |
|---------|---------|---------------|--------|
| 0.002 | 0.277 | 収束せず | — |
| **0.005** | **0.693** | **86** | **71%** |
| 0.010 | 1.386 | 19 | 94%（過激） |

#### 早期収束ロジック

`convergenceRatio = 0.005`（`initialRadius` に対する相対閾値）を採用し、`stableRuns = 5`（連続5回閾値以下で終了）で安定化した。単発の低変位で即 break しない設計。

```go
convergenceThreshold := initialRadius * convergenceRatio
stableCount := 0
for iter := range layoutIterations {
    // ... 力の計算・適用 ...
    if iterMaxDisp < convergenceThreshold {
        stableCount++
        if stableCount >= stableRuns {
            break
        }
    } else {
        stableCount = 0
    }
}
```

### 改善3: 球体ジオメトリ詳細度の削減（フロントエンド）

**`alt-frontend-sv/src/lib/components/desktop/tag-verse/TagAsteroid.svelte`** — `SphereGeometry` のセグメント数を32から16に変更した。

```svelte
<!-- Before -->
<T.SphereGeometry args={[1, 32, 32]} />
<!-- After -->
<T.SphereGeometry args={[1, 16, 16]} />
```

球体の半径は 0.3〜3.0、視点距離は `cloudRadius * 1.5`（≈90〜180）であり、16セグメントで視覚的に十分な滑らかさを保つ。

### 改善4: カメラ計算の集約（フロントエンド）

各 `TagAsteroid` が個別に `camera.current.getWorldDirection()` を呼んでいたのを、親コンポーネントで1回だけ計算して共有する方式に変更した。

**`alt-frontend-sv/src/lib/components/desktop/tag-verse/SceneContent.svelte`** — `useTask` でフレームごとにカメラのルック方向・位置・中心深度を計算し、plain object（`CameraFrameData`）に格納する。

```typescript
export type CameraFrameData = {
    lookDirX: number; lookDirY: number; lookDirZ: number;
    camPosX: number; camPosY: number; camPosZ: number;
    depthOfCenter: number;
};

useTask(() => {
    const cam = camera.current;
    cam.getWorldDirection(_lookDir);
    // cameraFrameData に書き込み（1回/フレーム）
});
```

**`alt-frontend-sv/src/lib/components/desktop/tag-verse/TagAsteroid.svelte`** — `useThrelte()` による `camera` 取得と `getWorldDirection()` 呼び出しを削除し、親から渡された `cameraFrameData` でドット積を直接計算する。

```typescript
// Before: cam.getWorldDirection(_lookDir) + Vector3 演算（300回/フレーム）
// After: cameraFrameData の数値で直接計算（300回の算術演算のみ）
const depthOfPlanet =
    toX * cameraFrameData.lookDirX +
    toY * cameraFrameData.lookDirY +
    toZ * cameraFrameData.lookDirZ;
```

親の `useTask` が先に登録されるため（Svelte のコンポーネントマウント順）、子は常に最新のカメラデータを参照する。

## RESULTS, EFFECTS

### PROS

- **レイアウト計算 ~37% 高速化**: 300ノードで 44ms → 28ms。iteration 数が 300 → 91 に削減（70%減）
- **コールドスタート解消**: 24分間隔のキャッシュウォーマーにより、初回リクエストは常にキャッシュヒット
- **GPU 三角形数 75% 削減**: 307,200 → 76,800 三角形（`SphereGeometry` 16セグメント化）
- **CPU フレームループ最適化**: `getWorldDirection()` 呼び出しが 300回/フレーム → 1回/フレームに集約
- **レイアウト品質維持**: 既存テスト（決定性、境界内、接続ノードの近接性）がすべてパス。相対閾値 + 連続安定判定により、レイアウトの視覚的品質を損なわない
- **仕様変更なし**: API 契約、色・発光感の5段階表現、ホバー・クリック・ラベル表示はすべて維持

### CONS, TRADEOFF

- **16セグメント球体の視覚品質**: 極端にズームインした場合にポリゴンの角が見える可能性がある。ただし通常の視点距離では視覚差はほぼない
- **早期収束の閾値感度**: `convergenceRatio = 0.005` はベンチマークに基づく経験値であり、極端なデータ分布（数万の共起を持つタグ群など）では収束ポイントが異なる可能性がある。`stableRuns = 5` の安定化により、偽収束のリスクは低減済み
- **`InstancedMesh` 化は未実施**: 300個の個別メッシュ・個別マテリアルによるドローコール数は依然として多い。per-instance `emissiveIntensity` の表現方法を PoC で確認後、別フェーズで対応する
