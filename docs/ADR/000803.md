# ADR-000803: HyDE の LLM サンプラを Gemma 公式推奨 (T=1.0 / top_p=0.95 / top_k=64) に揃える

## Status

Accepted

## Date

2026-04-20

## Affected Services

- acolyte-orchestrator (Python)

## Context

HyDE ([[000795]]) は日本語トピックに対して Gemma 4 で仮説英語 passage を生成し、Meilisearch hybrid search ([[000794]]) の追加 query variant として RRF に投入する。2026-04-19 の空応答事故 ([[000801]]) 以降、`think=false` を pin することで CJK プロンプトの empty response は解消しているが、下記 2 点が未着手だった:

1. サンプラは `temperature=0.0` のみ指定。`top_p` / `top_k` は Ollama のデフォルト (`top_p=0.9`, `top_k=40`) に委ねており、Google DeepMind が Gemma モデルカードで明示推奨する `temperature=1.0 / top_p=0.95 / top_k=64` と乖離していた。
2. `LLMProviderPort.generate` と `OllamaGateway` / `VllmGateway` に `top_p` / `top_k` を伝搬する経路が存在せず、サンプラ調整自体が gateway API の拡張を要求していた。

AIX (`gemma4-e4b-q4km`, `think=false`, num_predict=400, 3回平均) で JP→EN HyDE を 6 サンプラで比較した結果:

| サンプラ | mean_len | zero_rate | elapsed |
|---|---:|---:|---:|
| T=1.0 / top_p=0.95 / top_k=64 (Gemma 公式) | 1073 | 0% | 8.4s |
| T=0.2 / top_p=0.95 / top_k=64 | 1070 | 0% | 8.2s |
| T=0.0 / top_p=0.95 / top_k=64 | 1051 | 0% | 8.2s |
| T=1.2 / top_p=0.95 / top_k=128 | 1044 | 0% | 8.0s |
| T=0.7 / top_p=0.9 / top_k=40 | 1034 | 0% | 7.9s |
| T=0.0 (現状, top_p/top_k 未指定) | 1031 | 0% | 8.8s |

差は mean_len で +4% に収まり、空応答の抑制は `think=false` が担っているため、サンプラ差分の主眼は **同一トピックを再試行した際の passage 多様性** になる。HyDE は RRF の 1 variant にすぎないため、温度 0 で決定的になった仮説 passage は毎回同じランキングしか引かず recall 拡張に貢献しない。公式推奨値は Gemma の学習分布と整合し、再現性のあるベースラインでもある。

## Decision

以下の順で 3 層を同時に変更する:

1. `LLMProviderPort.generate` に `top_p: float | None = None` と `top_k: int | None = None` を追加し、gateway 側で `None` の場合はデフォルト委譲を維持する。
2. `OllamaGateway.generate` は `options["top_p"]` / `options["top_k"]` に伝搬 (`None` のときはキーを挿入しない)。`VllmGateway.generate` は OpenAI 互換で payload トップレベルに挿入。
3. `NewsCreatorHyDEGenerator.__init__` に `temperature: float = 1.0`, `top_p: float = 0.95`, `top_k: int = 64` の既定値を追加し、`generate_hypothetical_doc` から明示的に渡す。`think=False` と併用。

**却下した代替**:
- 全 LLM 呼び出し (`writer_node`, `critic_node`, `curator_node`, `planner_node`, `rerun_section_uc`, `rubric_evaluator`) に同じサンプラを適用する — Writer / Critic は `LLMMode.LONGFORM` / `STRUCTURED` 経由で既存 `longform_think` / 温度設定を通しており、ADR-000779 で決定した値を尊重する。HyDE だけ公式サンプラに寄せる方が影響範囲が最小。
- `top_p` / `top_k` を設定 (`Settings`) 経由で env 化 — HyDE は本 ADR が既定値を固定する場所であり、運用チューニングが必要になるまで env 化は保留。将来の reranker チューニングと合わせて別 ADR で再検討する。

## Consequences

### Pros

- Gemma 4 の学習分布と整合する確率的サンプリングが得られ、同一トピックで異なる passage が生成される確率が上がる。RRF fusion が 4 つめの variant として実質的に働く。
- Port 拡張が HyDE 以外の将来呼び出し (reranker / paraphraser / claim extractor) にも再利用できる。サンプラ調整の下地が整う。
- `top_p` / `top_k` を `None` 既定にすることで既存呼び出しは完全下位互換 (`test_top_p_and_top_k_omitted_when_not_provided` で担保)。

### Cons / Tradeoffs

- 温度 1.0 は決定的ではないため、同じ topic から得た HyDE passage が再実行で揺らぐ。評価 (baseline.jsonl) では LLM 出力が generator 依存である前提 ([[000793]]) を維持。
- `top_p` / `top_k` が gateway に増えることで payload が 2 フィールド膨張する。pyrefly 上の FakeLLM が静的 Protocol 違反にならないよう、テスト側も同パラメータを追加。
- サンプラを動かしても `think=false` の先に潜む Gemma 4 の内部温度制御は不可視。将来 Ollama が `thinking-budget` を露出した場合、本 ADR の想定が変わる余地がある。
