# ADR-000456: Knowledge Home projector の JSONB マージを scalar-safe にして投影停滞を緩和する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go)

## Context

Knowledge Home projector は `SummarySuperseded` / `TagSetSuperseded` を含むイベント列を処理中に、`UpsertKnowledgeHomeItem: cannot extract elements from a scalar` を繰り返し出力していた。原因は `knowledge_home_items.why_json` を常に JSON array とみなして `jsonb_array_elements(...)` していたことで、既存 row または `EXCLUDED.why_json` が scalar shape を含む場合に PostgreSQL が失敗していた点にある。

この失敗は 1 件だけの局所的な問題ではなく、checkpoint の前進を鈍化させ、Knowledge Home projector 全体の追従性を下げていた。実際に、再起動前は `knowledge-home-projector` の checkpoint が大きく遅延しており、dismiss など本来軽量なイベントも read model へ届くまでに時間差が発生していた。

Knowledge Home は append-only な `knowledge_events` を read model に反映する構成であるため、projector が特定のデータ shape に脆弱だと、単一イベントではなく後続の一連のイベント適用に影響が波及する。

## Decision

`why_json` の merge SQL を scalar-safe に変更し、array 以外の JSON 値は空配列として扱う。

具体的には `jsonb_array_elements(...)` の入力をそのまま使わず、以下のガードを挟む。

- `jsonb_typeof(EXCLUDED.why_json) = 'array'` のときだけ `EXCLUDED.why_json` を展開する
- `jsonb_typeof(COALESCE(knowledge_home_items.why_json, '[]'::jsonb)) = 'array'` のときだけ既存値を展開する
- それ以外は `'[]'::jsonb` を使う

あわせて、projector 側では `HomeItemDismissed` を no-op にせず read model 更新イベントとして扱うことで、read model の可視性更新と event replay を同じパイプラインに揃える。

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|--------|------|-----------|
| 既存データを一括クレンジングして array 以外を消す | 一時的には有効 | 再発防止にならず、将来別経路から scalar が入ると再び停止する |
| projector で失敗イベントを単純にスキップして checkpoint だけ進める | 運用上は楽 | read model の欠落を固定化し、event-sourced な再現性を失う |
| `why_json` をアプリケーション側で毎回読み出して Go で merge する | 型安全性は上がる | DB 内 upsert の一貫性が崩れ、既存 PgBouncer 互換の書き込み戦略から外れる |

## Consequences

### Pros

- `why_json` の shape が不揃いでも projector が停止しにくくなり、Knowledge Home の追従性が改善する
- `SummarySuperseded` / `TagSetSuperseded` を含むバッチでも event replay を継続できる
- projector checkpoint の前進が回復し、dismiss を含む後続イベントの read-model 反映が安定する
- SQL ガードは repository に閉じ込められるため、usecase / handler の責務を増やさない

### Cons / Tradeoffs

- scalar を空配列として扱うため、不正 shape の `why_json` が持っていた情報は merge 上無視される
- 問題の根本データをその場で矯正するわけではなく、projection path を防御的にする方針である
- projector 再追従のためには runtime 上で migration 適用後の再起動と継続処理が必要になる
