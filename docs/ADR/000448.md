# ADR-000448: TodayDigest の availability フラグを DB に永続化し backend-authoritative にする

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go)

## Context

`TodayDigest` の domain struct には `WeeklyRecapAvailable` と `EveningPulseAvailable` フィールドが存在し、frontend の `TodayBar` コンポーネントはこれらの値に基づいて Recap / Pulse ボタンの表示を切り替えている。しかし、DB schema（`today_digest_view` テーブル）にこれらのカラムが存在しなかったため、常に zero-value（`false`）が返されていた。

これは Knowledge Home Phase 1-5 是正指示書で「backend-authoritative 原理」違反として指摘された P0 項目である。domain model にフィールドがあるのに DB に永続化されておらず、backend が正しい値を返せない状態だった。

frontend（proto 定義 `TodayDigest.weekly_recap_available` / `evening_pulse_available`）は既にフィールドを参照しており、connect layer の変換も実装済みのため、DB と repository の修正のみで E2E が繋がる。

## Decision

**`today_digest_view` テーブルに `weekly_recap_available` と `evening_pulse_available` カラムを追加し、repository の SELECT / UPSERT を更新する。**

### マイグレーション

```sql
ALTER TABLE today_digest_view
  ADD COLUMN weekly_recap_available BOOLEAN NOT NULL DEFAULT false,
  ADD COLUMN evening_pulse_available BOOLEAN NOT NULL DEFAULT false;
```

- `DEFAULT false` により既存行は影響なし
- `NOT NULL` で整合性を保証

### Repository 変更

- `GetTodayDigest`: SELECT に 2 カラム追加、`Scan` で `digest.WeeklyRecapAvailable` / `digest.EveningPulseAvailable` に読み込み
- `UpsertTodayDigest`: INSERT に 2 カラム追加。ON CONFLICT では `OR` マージ（一度 `true` になったら `false` で上書きされない）

### OR マージの設計意図

```sql
weekly_recap_available = EXCLUDED.weekly_recap_available OR today_digest_view.weekly_recap_available
```

projector が差分 upsert する際、availability を設定しないイベント（`ArticleCreated` 等）が `false` を送っても既存の `true` を上書きしない。availability を `true` にするのは明示的な設定操作のみ。

## Consequences

### Pros

- `TodayDigest` の availability がバックエンド起点で正確に返されるようになり、frontend が常に正しいボタン表示を行える
- 既存の domain model / proto / frontend コードに変更不要（DB と repository の修正のみ）
- `DEFAULT false` + `NOT NULL` でデータ整合性が保証される

### Cons / Tradeoffs

- OR マージにより、一度 `true` になった availability を同じ日付内で `false` に戻すにはレコードの直接更新が必要（運用上はリセット操作が不要なため許容）
- projector からの availability 設定ロジックは今回のスコープ外（現時点では外部システムや管理 API から設定する想定）
