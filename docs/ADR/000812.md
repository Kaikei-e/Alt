# ADR-000812: news-creator app を CPU-only PyTorch + 2-stage uv sync で再構築しイメージを ~1.2 GB に圧縮する

## Status

Accepted

## Date

2026-04-20

## Affected Services

- **news-creator (Dockerfile.app)** — 2-stage (`python:3.14-slim` builder + runtime) に書き換え、`uv sync --frozen --no-dev --no-install-project --no-editable` + BuildKit cache mount + site-packages `tests/` 刈りを導入
- **news-creator/app/pyproject.toml** — `[[tool.uv.index]] pytorch-cpu` + `[tool.uv.sources] torch` で CPU wheel に明示固定。`psycopg2-binary` を `[project.optional-dependencies].scripts` に分離。`requires-python = ">=3.14,<3.15"`
- **news-creator/app/uv.lock** — CUDA/nvidia/triton 系 19 パッケージ除去、torch を `2.11.0+cpu` に切替
- **news-creator/app/news_creator/usecase/rerank_usecase.py** — `CrossEncoder(..., device="cpu")` を defensive default として明示
- **news-creator/.dockerignore** — 新設、build context からテスト・スクリプト・キャッシュ・backend 専用ファイルを除外
- **.github/workflows/news-creator.yml** — `python-version` を 3.14 に統一し CI/Docker/lock の 3.14 整合を確立

## Context

`news-creator/Dockerfile.app` は GHCR (`ghcr.io/kaikei-e/alt-news-creator`) に push されて `dispatch-deploy.yaml → Kaikei-e/alt-deploy` 経路で本番に pull される FastAPI コンテナである。現状、実測前の推定ベースでも次の 3 点が同時にサイズと CI 時間を押し上げていた:

1. **`sentence-transformers>=3.0.0` の transitive で CUDA 版 torch が入る** — `uv.lock` は CUDA 13 系 (`nvidia-cublas-cu13` / `nvidia-cudnn-cu13` など 18 パッケージ) + `triton 3.6.0` を解決しており、torch の manylinux_2_28_x86_64 wheel だけで 530 MiB、周辺 nvidia/triton を合わせると ~1.2 GB を占めていた。`compose/ai.yaml` 上 `news-creator` コンテナは GPU を割り当てていない (`runtime: nvidia` は `news-creator-backend` の Ollama だけ) ため、この CUDA ランタイムは一切使われない純粋な死に荷重だった。
2. **single-stage + `uv pip install --break-system-packages --no-cache -r /app/pyproject.toml`** — `--no-cache` で BuildKit cache mount が無効化され、CI のコールドビルドで毎回 wheel を再 DL。`uv.lock` を使わず `pyproject.toml` 直接 install だったため再現性も落ちていた。`--break-system-packages` で system Python に書き込むため、venv 分離もなかった。
3. **`COPY app/ /app/`** で `app/tests/` (4.7 MiB, 52 ファイル) と `app/scripts/` (analysis 用) が runtime image に焼き込まれていた。

加えて、以下の ABI ドリフトが潜在していた:

- `pyproject.toml:5` `requires-python = ">=3.13"`
- `.github/workflows/news-creator.yml` 4 ジョブ全て `python-version: "3.13"`
- `Dockerfile.app` は `python:3.14-slim`
- `uv.lock` は `cp314` wheel (`torch-2.11.0-cp314-cp314-manylinux_2_28_x86_64.whl`) に解決

CI (3.13) と Docker image (3.14) が異なる Python で `uv sync --locked` を走らせており、`uv.lock` 再生成時に hash が片方だけ有効になる再現性リスクを抱えていた。

[[000762]] tts-speaker で `[[tool.uv.index]] pytorch-cpu` + `[tool.uv.sources]` による CPU wheel pin パターンが 2026-04-17 に確立され、2026-04-20 の `recap-subworker` (commit `25d4b68e`) で 2-stage + BuildKit cache mount + site-packages `tests/` 刈りの完全版が実装済である。news-creator はこれを踏襲すれば新規設計判断なしに swap できる状態にあった。

## Decision

### 1. PyTorch を CPU index に明示固定し、dead deps を除去する

`news-creator/app/pyproject.toml` を uv 公式の PyTorch 統合パターン ([uv docs — using pytorch](https://docs.astral.sh/uv/guides/integration/pytorch/)) に揃える:

```toml
[project]
requires-python = ">=3.14,<3.15"
dependencies = [
    # ... 既存
    "torch>=2.11.0",
]

[project.optional-dependencies]
scripts = ["psycopg2-binary>=2.9.0"]

[[tool.uv.index]]
name = "pytorch-cpu"
url = "https://download.pytorch.org/whl/cpu"
explicit = true

[tool.uv.sources]
torch = [{ index = "pytorch-cpu" }]
```

`[tool.uv.sources]` の source override は `[project.dependencies]` に直接宣言されたパッケージに対してのみ適用される ([uv docs — sources](https://docs.astral.sh/uv/concepts/projects/dependencies/)) ため、transitive の torch を上書きするには torch を直接依存として宣言する必要がある。これは [[000762]] の tts-speaker で採った同じ手当。

`psycopg2-binary` は runtime には不要 (`app/scripts/` の analysis 専用で main.py からは import されない) なため `[project.optional-dependencies].scripts` に移動。runtime の `uv sync --no-dev` で自動的に除外される。

`uv lock` を再実行すると 19 パッケージ (cuda-bindings / cuda-toolkit / cuda-pathfinder / nvidia-* 16 種 / triton) が除去され、torch は `2.11.0+cpu` に切り替わる。

### 2. Dockerfile.app を 2-stage に書き換える

recap-subworker の完成版パターン (`recap-subworker/Dockerfile.recap-subworker:1-99`) を踏襲:

```dockerfile
# syntax=docker/dockerfile:1.7
FROM python:3.14-slim AS builder
WORKDIR /app
RUN apt-get update && apt-get install -y --no-install-recommends ca-certificates build-essential && rm -rf /var/lib/apt/lists/*
COPY --from=ghcr.io/astral-sh/uv:0.11.0 /uv /uvx /bin/
ENV UV_COMPILE_BYTECODE=1 UV_LINK_MODE=copy
COPY app/pyproject.toml app/uv.lock ./
RUN uv venv
RUN --mount=type=cache,id=uv-news-creator,target=/root/.cache/uv \
    uv sync --frozen --no-dev --no-install-project --no-editable
COPY app/main.py ./
COPY app/news_creator ./news_creator
COPY app/prompts ./prompts
RUN --mount=type=cache,id=uv-news-creator,target=/root/.cache/uv \
    uv sync --frozen --no-dev --no-editable
RUN SITE=/app/.venv/lib/python3.14/site-packages; \
    for pkg in numpy scipy sklearn sentence_transformers transformers torch; do \
        find "$SITE/$pkg" -type d -name tests -prune -exec rm -rf {} + 2>/dev/null || true; \
        find "$SITE/$pkg" -type d -name test  -prune -exec rm -rf {} + 2>/dev/null || true; \
    done

FROM python:3.14-slim AS runtime
ENV PATH="/app/.venv/bin:$PATH" VIRTUAL_ENV=/app/.venv \
    OMP_NUM_THREADS=1 MKL_NUM_THREADS=1
COPY --from=builder --chown=appuser:appuser /app/.venv /app/.venv
COPY --chown=appuser:appuser app/main.py ./
COPY --chown=appuser:appuser app/news_creator ./news_creator
COPY --chown=appuser:appuser app/prompts ./prompts
```

重要な差分:

- `--break-system-packages` / `--no-cache` / `uv pip install -r pyproject.toml` を全廃し `uv sync --frozen` に統一 → uv.lock 準拠で再現性を確立
- BuildKit cache mount (`id=uv-news-creator`) により self-hosted runner / local docker では wheel cache が効く (GHA runner では 1 回限りだが、GHA cache の `scope=news-creator` が layer cache で補う)
- `--no-editable` + 依存 → ソースの 2 ステップ sync により、app コード変更時でも依存 layer は invalidate しない
- `VIRTUAL_ENV=/app/.venv` + `PATH=/app/.venv/bin:$PATH` で `python -m uvicorn main:app` が venv 経由で解決される
- `UV_COMPILE_BYTECODE=1` で `.pyc` 事前生成 → FastAPI cold start 短縮
- runtime は `/app/.venv` + `main.py` + `news_creator/` + `prompts/` のみ COPY。tests/scripts は `.dockerignore` で context から除外

### 3. rerank_usecase に CPU device を defensive に明示

`CrossEncoder(model_name, device="cpu")` に変更。lock ファイルで CPU index pin しているため原則不要だが、将来 transitive 依存の変化で GPU wheel が再混入しても CPU 実行を保証する保険。ロジック変更ではなく default の明示化。

### 4. Python バージョンを 3.14 に統一

`uv.lock` が `cp314` wheel に解決されている現状に合わせ、`pyproject.toml` / `news-creator.yml` / `Dockerfile.app` の 3 箇所で 3.14 に揃える。`requires-python = ">=3.14,<3.15"` で明示範囲を絞る。

### 5. 検討したが採用しなかった代替案

| 代替案 | 却下理由 |
|---|---|
| `nvidia/cuda:X.Y-runtime-ubuntu22.04` ベースで書き直し | `news-creator` コンテナは GPU 未割当で CUDA runtime 不要。推論は `news-creator-backend` の Ollama 側 (別コンテナ) に寄せている既存分離を破壊する |
| distroless (`gcr.io/distroless/python3-debian12`) | `python:3.14-slim` との差は -10〜-20 MiB 程度にとどまり、debug シェル喪失の運用コストが上回る。`docker compose exec news-creator bash` が使えなくなる |
| CrossEncoder を builder で pre-download (tag-generator パターン) | image サイズが +300 MiB 前後増え、サイズ削減の主目的と逆行する。初回リクエストの HuggingFace fetch は lazy load のまま許容し、レイテンシが運用上問題化した場合に follow-up 判断 |
| `provenance: false` / `sbom: false` で attestation 切捨 | ビルド時間短縮は数十秒のみ、対し supply chain attestation 喪失は OWASP A08 後退に直結。維持 |
| multi-arch (`linux/amd64` + `linux/arm64`) サポート追加 | alt-deploy runner は amd64 前提で arm64 需要なし。cache/push 倍増の費用対効果なし |

## Consequences

### Pros

- **image 実測 1243 MiB** (local build, python:3.14-slim runtime、post-prune)。旧実装は CUDA 13 系 + triton だけで 1.0 GiB 超と見積れ、ベース/依存/アプリ合算で 2.5 GiB 前後だったと想定されるため、概算 **約 -50%** のサイズ削減
- `dispatch-deploy.yaml → Kaikei-e/alt-deploy` 経路の GHCR pull 帯域が半分以下に。本番デプロイの wall-time が短縮される
- `uv sync --frozen` による再現性と BuildKit cache mount (`id=uv-news-creator`) による再ビルド高速化。`pyproject.toml` / `uv.lock` 変更が無い場合は依存 layer が完全に cache hit する
- Python 3.13 (CI) / 3.14 (Docker) の pre-existing ABI drift を恒久解消
- `torch` を直接依存として宣言したことで、transitive 解決に依存せず sentence-transformers バージョン変更時の回帰が減る
- `psycopg2-binary` 除外で libpq 相当の関連バイナリも runtime から消え、analysis scripts 利用時のみ明示 extras で install される厳密な境界
- CrossEncoder の `device="cpu"` 明示で GPU wheel 再混入時の silent fallback (GPU アクセス試行 → CPU への自動退避) を排除
- `.dockerignore` で送出 context が縮小し、build 開始時の `transferring context` フェーズも短くなる

### Cons / Tradeoffs

- **切替直後の初回 main push は buildcache 不在で CI `docker-build` ジョブが 8〜12 分増**。2 回目以降の main push および PR ビルド (registry cache 参照) は通常速度
- `app/scripts/` の analysis スクリプト利用者は `uv sync --extra scripts` で明示的に `psycopg2-binary` を入れる必要がある。README への追記は別 PR で扱う
- **CrossEncoder 初回ロードは引き続き lazy** — 起動後の最初の rerank リクエストは HuggingFace から `BAAI/bge-reranker-v2-m3` を fetch するため数秒〜数十秒ブロックする。運用で顕在化したら `tag-generator/Dockerfile.tag-generator:60-75` の builder pre-download パターンを follow-up で導入する余地
- `sentence-transformers` は `pyproject.toml` 上 `>=3.0.0` 宣言だが `uv.lock` は `5.3.0` に解決している silent major bump がある。API 整合は [[000813]] で扱う CI smoke-check (`from sentence_transformers import CrossEncoder`) で検知する方針。本 ADR 単体では回帰テストは追加していない
- `Dockerfile.backend` (Ollama) は今回触っていない。`compose/ai.yaml:34-92` で `news-creator-backend` が `build:` のみ宣言で `image:` 行を持たない件は別件 (`feedback_compose_image_required` 該当可能性) として follow-up PR で扱う
