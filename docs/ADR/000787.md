# ADR-000787: auth-hub E2E セッション種まきを Kratos self-service browser flow で行う

## Status

Accepted

## Date

2026-04-19

## Affected Services

- auth-hub (Go) — 挙動無変更
- `e2e/hurl/auth-hub/00-setup.hurl` — seed + login step を所持
- `e2e/hurl/auth-hub/run.sh` — 2-phase 実行の dispatcher

## Context

[[000785]] の auth-hub Hurl suite は 12 本中 7 本が `ory_kratos_session` cookie を要求する (`/validate` 系、`/session` 系、`/csrf` 系)。auth-hub の handler は **cookie のみ** を読む:

```go
// /home/koko/Documents/dev/Alt/auth-hub/internal/adapter/handler/validate.go:26
cookie, err := c.Cookie("ory_kratos_session")
```

このため E2E では Kratos から**自己ホスト版の cookie-auth 形式に合った `ory_kratos_session` 値** を取得する必要がある。`X-Session-Token` や `Authorization: Bearer` には handler 側が対応していない。

Kratos v1.3.0 がセッション cookie を発行する経路は 3 つ検討できた:

### 案 A — Kratos Admin API の session mint

`POST /admin/sessions` や `POST /admin/identities/{id}/sessions` といった「identity 指定で session を直接発行する」API が Ory Network などのマネージド版にはあるが、**self-hosted Kratos v1.3 には存在しない**。GitHub discussions でも複数回要望が立っているが未実装。`POST /admin/identities/{id}/extend-session/{session_id}` は既存 session の延長しかできない。

### 案 B — self-service api-flow (`/self-service/login/api`)

```bash
GET /self-service/login/api         # flow JSON を返す
POST /self-service/login?flow=...   # password 提出 → `session_token` を JSON body に含めて返す
```

Hurl で試すと flow は通り、response に `session_token=ory_st_xxxx` が返ってくる。ところがこれを `Cookie: ory_kratos_session=ory_st_xxxx` として auth-hub に送ると:

```
Kratos: "No active session was found in this request."
auth-hub: 401 "authentication required"
```

self-hosted Kratos の実装では **cookie 認証と token 認証が別経路**で、cookie 値は HMAC 署名 + DB lookup で検証される。api-flow の opaque `session_token` は `X-Session-Token` / `Authorization: Bearer` 経由でのみ有効な credentials であり、そのまま cookie として送っても whoami は 401 を返す。

(Ory Network のドキュメントには `Cookie: ory_kratos_session=ory_st_...` と書いてあるが、あれはマネージド版の CDN レイヤで bearer → cookie 変換が動いているためで、self-hosted では当てはまらない。)

### 案 C — self-service browser-flow (`/self-service/login/browser`)

```bash
GET /self-service/login/browser     # Accept: application/json を付けると flow JSON を返しつつ Set-Cookie で csrf_token_<flow> を出す
POST /self-service/login?flow=...   # password + csrf_token を提出 → Set-Cookie: ory_kratos_session=<HMAC-signed value>
```

この `ory_kratos_session` は HMAC 署名済みの正統な cookie value で、`Cookie:` ヘッダに載せて auth-hub に送ると Kratos whoami が通る (= auth-hub の cache hit / miss 分岐がどちらも動く)。

## Decision

**案 C (browser flow) を採用し `00-setup.hurl` に段階ごとに書き下す。** api-flow は staging から完全に排除する。

### 実装

`e2e/hurl/auth-hub/00-setup.hurl` の構成:

1. **readiness probe** — `GET http://auth-hub:8888/health` と `GET http://kratos:4434/admin/health/ready` を `[Options] retry: 60, retry-interval: 1000` で待機
2. **identity seed (冪等)** — `POST http://kratos:4434/admin/identities` に `schema_id=default`, `traits.email=<fixture>`, `credentials.password.config.password=<fixture>`, `state=active` の JSON を投げる。`HTTP *` + `status matches "^(200|201|409)$"` で fresh / 既存の両方を受け入れる。run.sh の cleanup が `down -v` で DB ごと消すので、CI は常に 200/201 branch
3. **browser flow init** — `GET http://kratos:4433/self-service/login/browser` に `Accept: application/json` を付ける。Kratos が JSON body を返しつつ `Set-Cookie: csrf_token_<flow-hash>=...; HttpOnly; SameSite=Lax` を発行する。Hurl の cookie jar が自動で次リクエストに載せる。`[Captures]` で `login_action` (= `$.ui.action`) と `csrf_token` (= `$.ui.nodes[0].attributes.value`) を拾う
4. **login submit** — `POST {{login_action}}` に `Accept: application/json` と `Content-Type: application/json`、body は `{method: "password", identifier, password, csrf_token}`。Kratos が `Set-Cookie: ory_kratos_session=<HMAC-signed value>; Max-Age=3599; HttpOnly; SameSite=Lax` を返す。`[Captures]` で `kratos_session_cookie: cookie "ory_kratos_session"`、`user_id: jsonpath "$.session.identity.id"`、`session_id: jsonpath "$.session.id"`

この 5 ステップで `ory_kratos_session` の正当な cookie value が Hurl 変数として手に入る。これを下流の 12 本に注入する仕組みは [[000788]] で扱う。

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| api-flow で取得した `session_token` を cookie 値として送る | 上記のとおり self-hosted Kratos では cookie auth と token auth が別経路で、401 になる |
| auth-hub に `X-Session-Token` ヘッダ対応を追加 | プロダクションコード変更を E2E の都合で入れるのは設計的に筋が悪い。browser flow で既存コードのまま通る |
| auth-hub に Kratos `SessionExtractor` 相当を抜き差し可能にする | 抽象化のオーバーヘッドが高く、しかも本 suite のためだけの改修 |
| `return_session_token=true` 相当のクエリ + api-flow で cookie を受け取る | Kratos v1.3 の api-flow は `return_to` / `return_session_token` を受け付けるが、cookie は発行しない (仕様) |
| 素の curl でログインして `Set-Cookie` を shell 側で pipe / parse する | run.sh の shell 処理が増える。Hurl の cookie 自動管理で同等のことができるので分岐を増やす価値がない |
| Kratos `session.lifespan` を短くして認可失敗時にすぐ再 login させる | 本 ADR が解決したい問題 (= cookie を最初に取る) とは別問題 |
| Kratos Admin API でテスト用 session row を DB に直接 insert | Kratos の session table schema は private で、直接書き込みは v1.4+ で broken になるリスクが高い |

### 範囲外

- Kratos session の強制失効 / 明示 logout のテスト — 現 suite の scope 外 (必要なら別 scenario で追加)
- Multi-factor (TOTP / WebAuthn) 経路 — staging config [[000786]] で無効化している
- `continue_with` response field の consume テスト — auth-hub は `continue_with` を見ないので E2E の責務ではない

## Consequences

### Pros

- **auth-hub の handler コードに手を加えずに cookie-auth を E2E で回せる**。プロダクション挙動をそのまま検証できる
- **Kratos v1.3 の browser flow 実装を実際に使って検証される**。将来 Kratos を v1.4+ に上げたときに browser flow の response shape が変わったら suite が落ちて気付ける
- **Hurl の cookie jar が CSRF cookie を自動伝播する**ので、Hurl ファイルに cookie 管理の boilerplate が不要。`Set-Cookie` → 次 request の `Cookie:` は Hurl がやる
- **seed が冪等**。409 を許容する仕様で KEEP_STACK=1 デバッグでも 2 回目以降が壊れない

### Cons / Tradeoffs

- **browser flow の response shape に依存する**。Kratos v1.4+ で `$.ui.nodes[0].attributes.value` の位置や name が変わったら `[Captures]` を書き換える必要がある。軽減策: 必要に応じて `$.ui.nodes[?(@.attributes.name=='csrf_token')].attributes.value` 型の JSONPath に切り替え可能だが、現状は index 0 で十分
- **CSRF token を 2 段 (cookie + body) で扱うため Hurl 初学者には読みにくい**。README に flow 図を書いてカバー
- **api-flow 経路をテストしていない**。api-flow 自体は別サービス (mobile app 等) のユースケースで将来必要になるかもしれない。そのときは別 suite で追加
- **Kratos の cookie 実装変更に追随が必要**。HMAC 署名アルゴリズムや cookie name が変わると seed が壊れる。影響範囲は 00-setup.hurl 1 ファイル
