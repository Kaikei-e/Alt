# ADR-000452: Knowledge Home projector 障害後の回復を再投影運用に統一する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go)
- alt-frontend-sv (TypeScript/SvelteKit)

## Context

Knowledge Home projector は event を best-effort で処理し、個別 event の投影に失敗してもログを出して次へ進む。これは read model 全体の停止を避けるためには有効だが、失敗した event だけをあとから自然回復させる仕組みは持たない。checkpoint は後続 event に合わせて進むため、障害が一時的であっても該当 item の read model は欠損したまま残る。

今回の UTF-8 破損ではこの挙動が顕在化し、要約は生成済みなのに `knowledge_home_items` だけが更新されず、UI ではローディングが継続した。コードを修正して `alt-backend` を再 build / 再起動しても、既に読み飛ばされた event まで自動で再処理されるわけではない。

一方で Alt には Knowledge Home admin から実行できる reproject 導線がすでに存在しており、read model を履歴 event から再構築する前提は整っている。今回のような projector 障害に対して ad-hoc な SQL 修正や手作業補填を行うより、既存の再投影フローを正式な回復経路として扱う方が一貫性が高い。

## Decision

**Knowledge Home projector の障害で read model 欠損が発生した場合、回復手段は既存の reproject を第一選択とし、コンテナ再起動だけでの自然回復は期待しない。**

今回の運用判断として以下を標準化する。

- projector 障害を修正したら、まず影響サービスを再 build / 再起動して新バイナリを反映する
- その後、Knowledge Home admin の reproject 導線で read model を再構築する
- 失敗 event のみを手で埋める SQL patch は、reproject が不可能な特殊事情がない限り採用しない
- UI 側の現象が「loading 固着」で見えていても、回復は frontend の局所修正ではなく read model の整合性回復として扱う

代替案として、projector checkpoint を巻き戻して再実行する方法も考えられるが、運用時にどの event まで戻すべきかの判断が難しく、別 read model との整合性も崩しやすい。既存の reproject run / compare / swap の枠組みを使う方が監査可能で、障害復旧手順として明確である。

## Consequences

### Pros

- projector 障害時の復旧手順が「修正して再投影する」で統一される
- 欠損した item 群をまとめて再構築でき、個別 SQL 修正より再現性が高い
- admin 側の compare / swap 導線を活かせるため、回復作業を監査しやすい
- frontend の loading 固着を read model 障害として扱えるため、責務分離が明確になる

### Cons / Tradeoffs

- 小規模な欠損でも reproject 実行の運用コストがかかる
- projector が best-effort 継続である以上、障害発生から reproject 完了までの間は一部 item が不完全に見える
- admin 導線に依存するため、reproject 機能自体の可用性と運用理解が前提になる
