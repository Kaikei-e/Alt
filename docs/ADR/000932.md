# ADR-000932: Alt UI 配下に self-contained な /admin/monitor システム監視ダッシュボードを追加する

## Status

Accepted

## Date

2026-05-29

## Affected Services

- alt-backend (Go) — `app/gateway/admin_metrics_gateway/allowlist.go` の 5 メトリクス拡張と `availability_services` の job 正規表現拡張
- alt-frontend-sv (Svelte 5 / SvelteKit / Tailwind v4) — `src/routes/(app)/admin/monitor/` 新規ルートと `src/lib/components/admin/monitor/` 配下 10 コンポーネント
- docs/runbooks/admin-observability.md — 新規メトリクス行と Monitor route セクション

## Context

### 経緯

Alt のオペレータが「いま Alt は健康か」を確認する経路は ADR-000704 (`AdminMonitorService` allowlist + Connect-RPC streaming) で確立済みだったが、その消費側 UI は `/admin/knowledge-home` 配下の **Observability タブ** 1 箇所だけだった。このタブは Knowledge Home の運用ビューとして設計されており:

- 視点が "Knowledge Home の周辺 telemetry" であって、Golden Signals / RED / USE といったシステム監視の業界標準フレームに沿った配置ではない
- 全文 1 カラムの `MetricRow` 縦並びで、操作者が 1 画面で「何がおかしいか」を捕まえるための視覚的ヒエラルキーが弱い
- ADR-000485 の `service_quality` 3 値 (`full` / `degraded` / `fallback`) は handler では返るが、UI 側はタブ 1 個ぶんの degraded banner しか持たない

このため、ルーティンの system-health 確認は **Grafana (port 3001)** に流れていた。Grafana は機能的に十分だが Alt のデザインシステム外で、ブラウザのタブを Alt から離す前提を作り、admin 認証経路も Alt の Kratos + JWT + service-token swap (ADR-000704) ではなく Grafana 単独の basic auth に依存していた。

### 動機

ユーザ要件は明確に 4 点に絞られた:

1. `/admin/monitor` を独立ルートとして追加する。`/admin/knowledge-home` の Observability タブは現状維持
2. データソースは Prometheus 経由のメトリクスのみ。ClickHouse ログ検索 UI は載せない
3. Grafana の iframe 埋め込みも link-out も**しない**。dashboard は完全に Alt 内で完結する
4. Desktop-first / Tablet 以上対応。Mobile は admin 用途上強い要件ではない

「Grafana を再発明する必要はない」と言いつつ「Alt 側で system health を確認できる必要がある」というユーザの言い回しは、**Grafana の総合性を再現する**ことではなく、**Grafana を開かないと答えられない設問を 0 に近づける**ことが目的だと解釈した。Golden Signals / RED / USE / SLO burn-rate という SRE 業界標準の 4 軸が押さえてあれば、日常運用の "is Alt healthy" は十分カバーされるという読み。

### 設計に効く既存制約

- **ADR-000704**: 上記の通り。allowlist は compile-time 文字列リテラル、raw PromQL は受け付けない。`ADMIN_MONITOR_QUERY_TIMEOUT=3s` (gateway) + `2.5s` (upstream)、cache 10s、rate limit 5rps、stream interval 5s、stream 寿命 15min ± 60s、再接続は full-jitter exp backoff (1s → 30s cap)。`degraded=true` HTTP 200 で fail-open。
- **ADR-000485**: `service_quality` 3 値は handler / projector / usecase の各層で nil-safe にハンドルされている。dashboard 側は `MetricResult.degraded` / `MetricResult.reason` をそのまま表示する責務を負う。
- **ADR-000120**: `OTEL_EXPORTER_OTLP_ENDPOINT=http://rask-log-aggregator:4318` で全サービスが OTLP を流すスキーム。dashboard の利用上は影響しないが、`availability_services` の対象を広げる根拠になる (`auth-hub` / `tag-generator` / `news-creator` / `knowledge-sovereign` / `rag-orchestrator` の `/metrics` も既に scrape 経路にある)。
- **ADR-000098**: ClickHouse ベースのログ infra。今回スコープ外。dashboard 側に DELETE / UPDATE 経路が無いことの確認だけ重要。

### 公知のベストプラクティス

設計のレビュー時に押さえた業界文献:

- Google SRE Book "Monitoring Distributed Systems" — Latency / Traffic / Errors / Saturation の Four Golden Signals
- Google SRE Workbook "Alerting on SLOs" — 99.9% baseline で 1h/5m @ 14.4× (page-1) / 6h/30m @ 6× (page-2) / 3d/6h @ 1× (ticket) の multi-window multi-burn-rate
- Grafana 公式 Dashboard best practices — Z-pattern (top-left = key signal)、general → specific、text + glyph で色単独依存を避ける (WCAG 1.4.1)、stacked-graph アンチパターン、sprawl 回避

これらは Grafana の dashboard カタログを真似るのではなく、**Alt 側で同等の意思決定経路を再現する**ためのチェックリストとして使った。

## Decision

`/admin/monitor` を新規 SvelteKit ルートとして追加する。`/admin/knowledge-home` の Observability タブは現状維持。

### バックエンド (alt-backend)

`app/domain/admin_metrics.go` に新規 `MetricKey` 定数を 5 個、`app/gateway/admin_metrics_gateway/allowlist.go` に対応する `allowEntry` を 5 行追加する。PromQL は全て server-side 文字列リテラル (raw PromQL の client 送信は引き続き拒否):

| key | 主目的 |
|---|---|
| `http_latency_p50` | percentile band の中央値 (既存 p95 と 1 セットで分布を読む) |
| `http_latency_p99` | percentile band の tail (p50/p95 と overlay) |
| `prometheus_scrape_lag` | exporter の停滞を「メトリクスが silent に古くなる」前に検出する self-observability |
| `availability_burn_1h` | SRE Workbook page-tier 1 (14.4× で fire) の表示用 |
| `availability_burn_6h` | SRE Workbook page-tier 2 (6× で fire) の表示用 |

`availability_services` の `job=~` 正規表現を `auth-hub`, `tag-generator`, `news-creator`, `knowledge-sovereign`, `rag-orchestrator` の 5 サービス分広げた。これらは既に Prometheus scrape 対象だが、旧 regex の網から漏れていた。`/metrics` を未実装のサービスは引き続き "Not instrumented" バッジで降り、false-positive `down` にはならない。

### フロントエンド (alt-frontend-sv)

新規 SvelteKit ルート `src/routes/(app)/admin/monitor/{+page.server.ts,+page.svelte}`:

- `+page.server.ts` は `getUserRole(locals.user) !== "admin"` で 303 redirect する既存パターンを踏襲する
- `+page.svelte` は `useConnectAdminMetrics` (既存) を `$effect` の中で生成・破棄し、time-range picker の `RangeWindow` / `Step` 変更で stream を tear-down + restart する

`src/lib/components/admin/monitor/` 配下に 10 個のコンポーネントを置く:

- `format.ts` — `formatValue` / `stateBadge` / `burnSeverity` (framework-free, 単体テスト 9 ケース)
- `MetricCard.svelte` — エディトリアル KPI カード (Playfair Display で大きく値、SLISparkline、text + glyph badge)
- `MonitorErrorBanner.svelte` — `metrics` 配列のうち 1 つでも `degraded=true` であれば、または stream state が `connecting` / `degraded` なら表示する sticky banner。`●` glyph + `degraded` テキストで色単独依存を回避
- `TimeRangePicker.svelte` — window と step の segmented control。`window/step > 720` の組み合わせはサーバーの InvalidArgument 前にクライアント側で disable する
- `MonitorHeader.svelte` — タイトル + TimeRangePicker + live indicator (▲ live / ● connecting/degraded / ○ closed / ▮▮ paused) + pause toggle
- `GoldenSignalsRow.svelte` — Latency p95 / Traffic rps / Error % / CPU saturation の 4 カード
- `SLOBurnPanel.svelte` — Burn rate 1h / Burn rate 6h / SLO estimate (visible-window) の 3 セル。14.4× と 6× を SLISparkline の dashed threshold で表示
- `ServiceREDTable.svelte` — `job` ラベルで集約した 1 行/サービス。state / job / p50 / p95 / p99 / req/s / err % の 7 列。行クリックで `ServiceDrilldown` を inline 展開
- `ServiceDrilldown.svelte` — Latency band overlay (p50/p95/p99 を 3 重描画) / Traffic + Errors / Container resource (cAdvisor `name` ラベルが job 名と一致する series を採用) の 3 パネル
- `SaturationGrid.svelte` — cAdvisor の per-container CPU + memory を USE フレーミングで grid 表示。`auto-fill, minmax(220px, 1fr)` で tablet 以上の幅に応じて折り返す
- `QueueHealthCard.svelte` — mq-hub の Redis up/down + 合計 publish rate + topic 別 publish rate

`ObservabilityPanel.svelte` 配下のコンポーネント (`MetricRow.svelte`, `ServiceHealthTable.svelte`, `DegradedBanner.svelte`, `SLISparkline.svelte`) は **移動せずに reuse する**。`/admin/knowledge-home` のタブが既存の import を保持し続けるため。`SLISparkline` のみ新ルートから `$lib/components/knowledge-home-admin/SLISparkline.svelte` として直接 import する。

### Visual contract

- 既存の Alt-Paper トークン (`--alt-success/warning/error`, `--obs-spark-stroke`, `--obs-spark-threshold`, `--obs-series-1..4` Okabe–Ito) を流用する。新規トークンを定義しない
- Typography: Playfair Display (KPI 数値), IBM Plex Mono (生数値・ラベル), Source Sans 3 (本文)
- 状態表現は text + glyph (`▲ ▼ ● ○ "up"/"down"/"warn"/"ok"`). 閾値は SLISparkline の dashed 参照線。色単独依存しない (WCAG 1.4.1)
- レイアウトは Z-pattern: top-left = Golden Signals / top-right = MonitorHeader controls / 中央 = SLOBurnPanel → ServiceREDTable / 下 = SaturationGrid + QueueHealthCard

### 検討した代替案と却下理由

| 案 | 却下理由 |
|---|---|
| Grafana を iframe 埋め込み | デザインシステム不一致 / CSP frame-ancestors と cookie 連携の運用負担 / 「Alt 内で完結」というユーザ要件と直接衝突 |
| Grafana link-out のみ (`grafana_url` を UI ボタン化) | 「link-out しない」という明示要件と衝突。`grafana_url` は proto に残すが UI からは render しない |
| 既存 Observability タブを `/admin/monitor` に格上げ | タブの Knowledge Home 文脈が失われる / 既存テスト・ドキュメント・カスタマイズの幅広い書き換えが発生 / レイアウトもタブ用に 1 カラム最適化されており system monitor 用途には弱い |
| ClickHouse ログ検索 UI を同居 | スコープ外 (ユーザ明示) / SSRF とカーディナリティ爆発のリスクを再評価する必要があり、本 ADR の単位では合わない |
| Tracing / span exemplar 連動 | 新規 RPC とログ infra アクセス面の追加が必要 / 後続 ADR で扱うほうがリスクが切り分く |
| `mqhub_queue_depth` / `mqhub_consumer_lag` を allowlist 追加 | 前者は ADR-000704 時の refresh で "retired key" として GW テストで明示拒否されている (`TestGateway_Catalog_ContainsRealMetrics` の retired list)。後者は upstream のメトリクス名が確認できなかった。両方とも本 ADR では追加しない |
| 30-day SLO compliance の正確値 (`avg_over_time(...[30d])`) | Prometheus 単発クエリでの 30d スキャンは `--query.max-samples=1000000` 周辺で重く、本来は recording rule 経由が望ましい。recording rule 未整備のため `SLOBurnPanel.svelte` の "SLO estimate" は **visible window から導出した近似値**にする。30d 正確値の表示は recording rule 整備後に再開する |

## Consequences

### Pros

- オペレータが Grafana を開かずに、Alt 内のデザインシステムで Golden Signals / RED / USE / SLO burn-rate の 4 軸を確認できる
- `/admin/knowledge-home` の Observability タブは無変更で残ったため、Knowledge Home フォーカスの運用文脈は壊れない
- 新規 RPC・新規 proto フィールド・新規 BFF ハンドラを一切追加していない (ADR-000704 のストリーミング基盤を完全再利用)
- 視覚的状態は text + glyph + dashed threshold で表現し、色覚多様性に対応する (WCAG 1.4.1)
- SRE Workbook 99.9% baseline の burn-rate tier 閾値が UI 上で常時可視 (14.4× / 6× / 1×)。alerting の paging 判断と dashboard の見え方が線形に対応
- 全テストが green: backend `go test -race` (4 package), `golangci-lint` 0 issues, frontend `bun run check` 0 errors, `vitest` 1379/1379, `bun run build` 成功

### Cons / Tradeoffs

- `SLOBurnPanel` の "SLO estimate" は visible window 由来の近似値であり、canonical な 30-day SLO ではない。recording rule 整備後に正確値へ差し替える運用フォローが残る
- `ServiceDrilldown.svelte` の container resource 表示は cAdvisor の `name` ラベルが job 名と一致または前方一致するヒューリスティクスに依存する。名称の不一致が起きた場合は "No cAdvisor series match this service" にフェイルする (silent な誤マッピングは出さない)
- Playwright E2E は構造的検証 (heading / picker / banner glyph) のみで、Connect server-streaming の chunked transfer を Playwright の `route.fulfill` で再現するのが現実的でないため、live stream 検証は manual smoke (`docs/runbooks/admin-observability.md`) に残る
- mq-hub の queue depth / consumer lag は今回 allowlist 入りせず、QueueHealthCard は publish rate と Redis 接続性のみ表示する。queue 飽和の可視化は upstream メトリクス名確定後に follow-up
- フロントエンドのストリーム再起動 (`$effect` 内で `stop` → 新 hook 作成 → `start`) は picker 変更ごとに 1 回発生する。実運用上は無視できる頻度だが、頻繁にレンジを切り替えると毎回 Catalog を取り直すため、将来的に Catalog の memoize が必要になる可能性がある

## Deploy-model 整合性セルフチェック

本 ADR は frontend route 追加 + backend allowlist 拡張 + runbook 改訂のみで、compose service / volume / init container を一切変更しない。

- [x] 新設 compose service が `depends_on` で他 service を gate するか？ → N/A (compose service 追加なし)
- [x] 新設 compose service に `restart: "no"` の one-shot init container を含むか？ → N/A
- [x] 新設名前付き volume を populate する責務はどこにあるか？ → N/A (volume 追加なし)
- [x] rolling 互換: alt-backend と alt-frontend-sv は既存の rolling deploy パターンでそのまま反映可能
