# 000363: お気に入りページ モバイルUI刷新 + お気に入り解除API追加

## ADR's STATUS

Accepted

## CONTEXT

お気に入りページ（Favorites）のモバイルUIが、メインフィード用の `FeedCard` コンポーネントをそのまま流用していたため、以下の深刻なUX問題が発生していた。

### 問題 1: 「Mark as read」が主要アクション

`FeedCard` はメインフィードの文脈で設計されており、カードの主要ボタンが「Mark as read」だった。お気に入りページでは、ユーザーが意図的に保存した記事に対して「既読にする」ことは主要操作ではない。

### 問題 2: 既読にするとカードが消失

`FeedCard.svelte` の line 22 に `{#if !isReadStatus}` というラッパーがあり、既読マークを付けるとカード全体が非表示になる。メインフィードでは妥当な挙動だが、お気に入りページでは意図的に保存したアイテムが消えてしまうという破壊的な動作となっていた。

### 問題 3: お気に入り解除手段の不在

`RegisterFavoriteFeed` RPC は存在したが、対応する解除 RPC がなかった。お気に入りは追加のみの一方通行で、ユーザーがリストを管理する手段がなかった。

### デザインベストプラクティス調査

モバイルブックマーク/お気に入りUIのベストプラクティスを調査した結果、以下の原則を確認した:

- お気に入りは既読状態に関わらず**常に表示**すべき
- 既読状態はバッジで**視覚的に示す**のみ（破壊的アクションにしない）
- お気に入りの追加/解除は**二状態ボタン**（★塗り潰し/★アウトライン）で実装
- タッチターゲットは最低 44x44px

## DECISION MAKING

### 1. FavoriteCard コンポーネントの新規作成

`FeedCard` を修正するのではなく、お気に入り専用の `FavoriteCard.svelte` を新規作成した。

**理由:** メインフィードとお気に入りでは根本的に異なる振る舞いが求められる。`FeedCard` は既読時にカードを非表示にする設計が正しく、お気に入りは常に表示する設計が正しい。一つのコンポーネントに両方のロジックを押し込むよりも、用途別に分離した方がシンプルで保守しやすい。

#### FavoriteCard のレイアウト

```
┌──────────────────────────────────────────┐
│ [↗] Title (外部リンク)      [Read badge] │
│                                          │
│ Excerpt (3行クランプ)                    │
│ by Author                                │
│                                          │
│ [Details]              [Remove ★] [Open] │
└──────────────────────────────────────────┘
```

**設計ポイント:**

| 項目 | FeedCard（メインフィード） | FavoriteCard（お気に入り） |
|------|--------------------------|--------------------------|
| 既読時の表示 | カード非表示 | 常に表示（Readバッジ付き） |
| 主要アクション | Mark as read | Details / Open |
| 管理アクション | なし | Remove（お気に入り解除） |
| 既読状態の取得 | 親が `Set<string>` で管理 | `feed.isRead` をそのまま使用 |

### 2. RemoveFavoriteFeed API のフルスタック追加

既存の `RegisterFavoriteFeed` のパターンを完全にミラーリングし、Clean Architecture の全レイヤーに `RemoveFavoriteFeed` を追加した。

#### 変更ファイル一覧

| レイヤー | ファイル | 変更内容 |
|----------|---------|---------|
| Proto | `proto/alt/rss/v2/rss.proto` | `RemoveFavoriteFeedRequest/Response` メッセージ + `RemoveFavoriteFeed` RPC 追加 |
| Port | `app/port/register_favorite_feed_port/register_port.go` | `RemoveFavoriteFeed` メソッドをインターフェースに追加 |
| Driver | `app/driver/alt_db/register_favorite_feed_driver.go` | `DELETE FROM favorite_feeds WHERE user_id = $1 AND feed_id = $2` |
| Gateway | `app/gateway/register_favorite_feed_gateway/gateway.go` | `RemoveFavoriteFeed` 実装（URL検証 + エラーマッピング） |
| Usecase | `app/usecase/remove_favorite_feed_usecase/usecase.go` | **新規作成** — 空URL検証 + ゲートウェイ委譲 |
| DI | `app/di/container.go` | `RemoveFavoriteFeedUsecase` のワイヤリング追加 |
| Handler | `app/connect/v2/rss/handler.go` | `RemoveFavoriteFeed` RPCハンドラ（認証 + URL検証 + Usecase呼出し） |
| Connect Client | `alt-frontend-sv/src/lib/connect/rss.ts` | `removeFavoriteFeed()` 関数追加 |
| API Client | `alt-frontend-sv/src/lib/api/client/articles.ts` | `removeFavoriteFeedClient()` 追加 |

**DBマイグレーション不要:** `favorite_feeds` テーブルの既存の複合主キー `(user_id, feed_id)` に対して `DELETE` するだけなので、スキーマ変更は不要。

### 3. お気に入りページの刷新

`favorites/+page.svelte` のモバイルセクションを以下のように変更:

- `FeedCard` → `FavoriteCard` に置換
- `mobileReadUrls` (既読追跡用 `Set<string>`) を削除
- `handleSetIsReadStatus` (既読マーク関数) を削除
- `updateFeedReadStatusClient` import を削除
- `handleRemoveFavorite` を追加（API呼出し + オプティミスティック削除）

## RESULTS, EFFECTS

### PROS

- **UXの根本的改善** — お気に入りページで「既読にしたら消える」という破壊的挙動が解消され、ユーザーが保存した記事は常に表示される
- **お気に入り管理が可能に** — RemoveFavoriteFeed API により、ユーザーがリストを自分で管理できるようになった
- **コンテキストに適切なアクション** — メインフィードの「Mark as read」ではなく、お気に入りに適した「Details / Remove / Open」がアクションとして提供される
- **既読状態の視覚的表現** — 既読記事は「Read」バッジ + テキスト色のミュート化で控えめに表示し、未読記事との区別が一目でわかる
- **既存コンポーネントへの影響なし** — `FeedCard` はメインフィードでそのまま正常に動作し続ける

### CONS, TRADEOFF

- **コンポーネントの増加** — `FavoriteCard` と `FeedCard` で類似のスタイリングコードが存在する。ただし、振る舞いが根本的に異なるため、共通化による複雑化よりも分離による明確さを優先した
- **Port インターフェースの拡張** — `RegisterFavoriteFeedPort` に `RemoveFavoriteFeed` を追加したことで、既存の Gateway/Mock が全てこのメソッドを実装する必要がある。今回は mock 再生成で対応済み
- **デスクトップ版は未対応** — 今回の変更はモバイルUIのみ。デスクトップ版のお気に入りページ（`FeedGrid` + `FeedDetailModal`）にはお気に入り解除ボタンが未実装。今後の課題として残る
