# ADR-000709: trigger_source で morning_update を batch Recap から分離し、誤 resume を止める

## Status

Accepted

## Date

2026-04-13

## Affected Services

- recap-worker (Rust) — `RecapDao::create_job_with_lock_and_window`、
  `find_resumable_job`、`JobContext`、`MorningUpdateDaemon`、
  `POST /v1/morning/letters/regenerate` handler、`AltBackendFetchStage`

## Context

ADR-000708 で boot 時 hygiene を入れたあと、DB は `pending=0, running=1`
まで縮まった。ユーザーの "Job スタックしていない?" に応じてライブログを
追跡すると、その **残った 1 件 (running)** が新たな設計問題を露呈した。

- `f67ffd79-50e2-48b3-bfc1-38c0f6343221`
  は今朝 `POST /v1/morning/letters/regenerate` で kick した
  **morning_update job**。プロセスクラッシュで `last_stage="evidence"`
  のまま残った。
- コンテナ再起動後、`find_resumable_job` はこの行を拾い、daemon は
  `scheduler.run_job(JobContext::new_with_window(job_id, genres, 1))` で
  **フル Recap pipeline** として resume した。
- その結果 morning_update (本来 dedup + group で数分) が classification
  → clustering → evidence → LLM batch summary (chunks=7, ~5min/chunk)
  を走り、**~35min の意図外 LLM 負荷**を生みつつ、`morning_letters`
  行は一切更新されない (run_job は MorningPipeline.execute_update を
  呼ばないため) 無駄走行になっていた。

根本は `recap_jobs` テーブルの **discriminator が死んでいる** こと。

- schema (`20260116000000_add_user_id_to_recap_jobs.sql`):
  `trigger_source TEXT NOT NULL DEFAULT 'system'`
- だが actual insert paths (batch / morning / user) のどれも値を
  上書きせず、全部 `'system'` 扱い。
- `find_resumable_job` は `status IN ('pending','running')` だけで選ぶ
  ため、morning / user 発の残骸を batch pipeline として実行してしまう。

DB の実状 (ADR-000708 適用時点):
```
trigger_source status   count
system         completed 26
system         failed    9
system         running   1  ← 実は morning_update 由来
```

## Decision

`recap_jobs` を kind 別に物理分離せず、**既存 `trigger_source` カラムを
有効化する最小差分で解決する**。migration は不要。

1. `RecapDao::create_job_with_lock_and_window` に `trigger_source: &str`
   を追加し、INSERT の VALUES に埋め込む。legacy の `create_job_with_lock`
   (旧シグネチャ) は `"system"` で呼び替え。
2. `find_resumable_job` の SQL に `AND trigger_source = 'system'` を追加。
   → auto-resume 対象は **batch 由来のみ**。
3. `JobContext` に `trigger_source: &'static str` フィールド (+ accessor)
   を生やし、constructor 3 種を提供:
   - `JobContext::new(...)` → `"system"` (既存コードのデフォルト)
   - `JobContext::new_with_window(...)` → `"system"` (batch daemon 用)
   - `JobContext::new_morning_update(job_id)` → `"morning"` + `window_days=1`
4. `AltBackendFetchStage.fetch` は `JobContext::trigger_source()` を
   DAO に転送する。これで 1 箇所の受け口で morning / system が区別される。
5. morning の 2 callsites を `new_morning_update` に切り替え:
   - `MorningUpdateDaemon` (将来有効化時用、現状は disabled)
   - `POST /v1/morning/letters/regenerate` handler (`api/fetch.rs`)
6. `mark_abandoned_jobs(keep)` は `trigger_source` 中立のまま放置
   (orphan 全般を seal する本来の責務)。

将来の user-triggered 経路 (`/v1/generate/recaps/3days|7days`) も
同じ mechanism で `"user"` を渡すだけで同様に誤 resume から守られる。

`traits/job.rs`, `impls/unified.rs`, `compat.rs`, `mock.rs` の
`create_job_with_lock_and_window` シグネチャに `trigger_source: &str`

### `scheduler/jobs.rs:JobContext`

```rust
pub(crate) struct JobContext {
    pub(crate) job_id: Uuid,
    pub(crate) genres: Vec<String>,
    pub(crate) current_stage: Option<String>,
    pub(crate) window_days: u32,
    pub(crate) trigger_source: &'static str,  // 追加
}

impl JobContext {
    pub(crate) fn new(..) -> Self { .. trigger_source: "system" }
    pub(crate) fn new_with_window(..) -> Self { .. trigger_source: "system" }
    pub(crate) fn new_morning_update(job_id: Uuid) -> Self {
        Self {
            job_id,
            genres: Vec::new(),
            current_stage: None,
            window_days: 1,
            trigger_source: "morning",
        }
    }
    pub(crate) fn trigger_source(&self) -> &'static str { self.trigger_source }
}
```

## Verification (2026-04-13 実測)

```sql
-- trigger_source | status    | count
-- system         | completed |  26
-- system         | failed    |   9
-- system         | running   |   1   ← 実は morning_update 由来
```
ログ:
```

→ run_job() で batch pipeline として実行
→ ~35min の無駄な LLM 負荷
```

### After (本 ADR 適用)

新しい morning regenerate:
```bash
curl -X POST http://localhost:9005/v1/morning/letters/regenerate
```
直後に DB:
```sql
SELECT trigger_source, status, kicked_at, window_days, job_id
FROM recap_jobs
WHERE kicked_at > NOW() - INTERVAL '2 minutes'
ORDER BY kicked_at DESC;
-- trigger_source | status  | kicked_at                     | window_days | job_id
-- morning        | pending | 2026-04-13 11:25:20.93621+00  | 1           | 9b083c92-...
```

cargo tests:
```
test scheduler::jobs::tests::job_context_default_trigger_source_is_system ... ok
test scheduler::jobs::tests::job_context_new_morning_update_marks_trigger_source_morning ... ok
test scheduler::jobs::tests::job_context_with_stage_preserves_trigger_source ... ok
test result: 311 passed; 0 failed
```

この行はクラッシュしても次回 boot で:
- `find_resumable_job` は `trigger_source='system'` で filter するため
  選ばない → batch pipeline としての誤 resume が発生しない
- `mark_abandoned_jobs(keep=None)` が pending/running を全 seal
  (keep が None なのは system 行が同時に存在しない場合)
して `failed` に確定する。

### 既存 `f67ffd79` について

本 ADR 適用前に生成されたため、`trigger_source='system'` でタグ付け
されている。これは retroactive fix 対象外。**運用中に発見され、
実際に以下の SQL で一度手動解除した**:
```sql
UPDATE recap_jobs
SET status='failed', last_stage=COALESCE(last_stage, 'stuck_resume_loop'),
    updated_at=NOW()
WHERE job_id='f67ffd79-50e2-48b3-bfc1-38c0f6343221';
```

### `f67ffd79` が曝した "resume loop" の根本原因

`trigger_source='system'` の検出だけでは塞げない二次的な故障モードが
あった:

1. `kicked_at` が 7 時間前の morning_update 行 (`last_stage='evidence'`)
   が `trigger_source='system'` (Phase-4 以前の insert) のまま残存。
2. `find_resumable_job(max_age_hours=12)` は依然として条件を満たすため
   resume candidate に選び続ける。
3. daemon は `run_job` でフル Recap pipeline を実行 → **subworker の
   clustering エンドポイントが 409 Conflict を返す**:
   ```
   error: clustering endpoint returned error status 409 Conflict:
   {"detail":"run already in progress for job/genre"}
   ```
   これは前の resume 試行で subworker 側に残った in-progress run が
   原因 (upstream orphan)。
4. 多くの genre が `insufficient documents for clustering: expected >= 2,
   found 1` で skip、summary phase 直前でフロー崩壊。
5. recap_jobs 行は `running` のまま、next boot でまた resume …

この loop は Phase 4 の `trigger_source='system'` filter だけでは塞げ
ない (pre-Phase-4 行は `'system'` デフォルトのため)。

### 追加修正 (同 ADR 内): resume window を 12h → 4h に短縮

`RECAP_RESUMABLE_MAX_AGE_HOURS` のデフォルトを **12 → 4** に変更:

```rust
// src/config.rs
let resumable_max_age_hours = parse_i64("RECAP_RESUMABLE_MAX_AGE_HOURS", 4)?;
```

**根拠**:
- 正規の batch Recap は通常 1–2h 以内に完了する
- 4h は 2x ヘッドルーム
- 4h を超える Job はほぼ確実に chronic (resume-loop + upstream
  orphan による deadlock)
- 本当に長尺な Job を動かす運用者は env で override 可

これで `f67ffd79` (kicked_at 7h+) のような pre-existing zombie も
auto-resume の対象から自然に外れ、`mark_abandoned_jobs` が seal する。

### Boot ログ (修正後)

```
"sealed orphaned recap jobs as failed"   keep_job_id=None   marked_failed=1
"boot-time hygiene: orphaned jobs sealed"   marked_failed=1   keep_job_id=None
"scheduled automatic 3-day recap batch"   next_run_jst=2026-04-14T02:00
```
**`found resumable job` が出ない** = loop 解消。

## Consequences

### Pros

- **Morning update のクラッシュ残骸が batch pipeline を誤発火しない**:
  ~35min の無駄な LLM 負荷と daemon ブロッキングが消える。
- **02:00 JST の scheduled batch が時間通り走る保証が強まる**:
  resume 処理に daemon が食われるケースが無くなる。
- **`trigger_source` を運用メタデータとして実際に使う** ので、
  dashboard で `system / morning / user` の内訳を出す下地が整う。
- **Migration 不要**: 既存カラムの書き込みを始めるだけ。既存行は
  `'system'` のまま (retroactive 整合は out-of-scope)。
- **User-triggered (`/v1/generate/recaps/*`) 経路にも同じ仕組みで
  `"user"` を tag 付けられる下地**が入った。以降 `find_resumable_job`
  の WHERE に追加不要。

### Cons / Tradeoffs

- 既存の `trigger_source='system'` だらけのゴミ行は retroactive には
  直らない。本 ADR 適用後も、ADR-000708 の boot-time hygiene が
  running/pending を毎回 seal するので実害はない。
- `trigger_source` は `&'static str` で string-typed。Rust enum では
  ないため、タイポ (`"morning "` 等) が SQL レベルで検出されない。
  将来 enum に昇格する余地あり (Phase 5)。
- `JobContext` の default `new` / `new_with_window` は `"system"` 固定。
  batch 以外の new 経路が今後できた場合は context constructor も
  追加する必要がある。

## Phase 5 (deferred)

- `trigger_source` を Rust enum `TriggerSource { System, Morning, User }`
  に昇格し、文字列 typo を compile time で弾く。
- Dashboard UI に `trigger_source` × `status` の積み上げ表示を追加。
- `recap_jobs` を `morning_jobs` 別テーブルに物理分離するリファクタ
  (migration + 大きめスコープのため別 ADR)。
