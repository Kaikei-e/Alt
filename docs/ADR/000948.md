# ADR-000948: モバイルスワイプ面を没入モード化し Keep stamp と遅延既読フラッシュで誤タップを排除する

## Status

Accepted

本 ADR は [[000716]]（ボトムナビの persistent 原則）を置き換えるものではなく、スワイプ面（`/feeds/swipe/*`）に限定した**明文化された例外**を追加する。他の全画面では [[000716]] の原則が引き続き有効である。

## Date

2026-07-18

## Affected Services

- alt-frontend-sv (SvelteKit / Svelte 5 Runes)
  - `src/lib/components/mobile/bottom-nav.ts` — `isImmersiveRoute()` 追加
  - `src/lib/components/layout/ResponsiveLayout.svelte` — 没入ルートでナビ非表示 + 予約 padding 撤去
  - `src/lib/components/mobile/feeds/swipe/DispatchHeader.svelte` — 新設
  - `src/lib/components/mobile/feeds/swipe/dispatch-session.svelte.ts` — 新設（遅延既読フラッシュ + undo）
  - `src/lib/components/mobile/feeds/swipe/SwipeFeedScreen.svelte` / `VisualPreviewCard.svelte` / `SwipeFeedCard.svelte` / `SwipeFilterSortSheet.svelte` — 改修

## Context

モバイルの Visual Preview（Photo Wire Dispatch, [[000688]]）で誤タップが常態化していた。構造的な原因は 3 つ:

1. **フッター 3 ボタンの近接**: ARTICLE / ☆ / SUMMARY が gap 0.5rem の横一列に並び、44×44px の ☆ が両側の可変幅ボタンに挟まれていた。意味論的にも「保存判断 (☆)」と「読み方の選択 (ARTICLE/SUMMARY)」という異質な操作が同列だった。
2. **固定ボトムナビとの垂直近接**: カードは `95dvh`、その直下に 44px + safe-area の固定ナビ（[[000716]] / [[000628]]）が重なり、フッターボタン列とナビタブが誤タップペアを構成していた。
3. **回復手段の欠如**: 左右スワイプ＝既読化に undo がなく、誤スワイプは取り戻せなかった。

実証研究の裏付け（2026-07-18 deep research、全クレーム 3-0 検証済み）:

- 片手親指操作のターゲットは離散 9.2mm / 連続 9.6mm 以上が推奨で、Apple HIG 44pt (~6.9mm) はこの実証基準を下回る (Parhi, Karlson & Bederson, MobileHCI '06)。
- 1.2 億タップの大規模データで 8mm 未満は誤り率 40% 超、**画面端のターゲットは中央の約 2 倍の誤り率** (Henze, Rukzio & Boll, MobileHCI 2011)。
- WCAG 2.2 SC 2.5.8 (24px) は「床」であり、重要コントロールには AAA の 44px (SC 2.5.5) 以上を W3C 自身が推奨。
- スワイプによる状態変更には「高視認・即時の undo」が必須 (NN/g / Nielsen ヒューリスティック #3)。Tinder Rewind は「常設ボタンで直前 1 枚だけ復元」の先例。
- Android 公式は読書コンテンツへの没入モード適用を明示的に是認しつつ、「画面拡大以上の便益がある場合のみ」と限定している。

## Decision

スワイプ面を**没入モード (Immersive mode)** とし、以下の 5 点を採用する。語彙は `contexts/visual-preview.md`（2026-07-18 grill セッション）で正準化済み。

1. **ボトムナビ非表示**: `/feeds/swipe/*` を `isImmersiveRoute()` で判定し、`ResponsiveLayout` がナビと予約 padding を落とす。[[000716]] の persistent 原則への画面限定の例外とする。
   - 代替案「スクロール連動の自動隠し」: 再表示タイミング自体が誤タップ源になり実装も複雑なため却下。
   - 代替案「ナビ維持 + カード高さ縮小」: 縦空間がさらに減り、根本原因（近接）が残るため却下。
2. **Dispatch header**: カード外上部に編集部風ミニヘッダーを常設する。戻る導線（← Menu）、kicker（PHOTO WIRE / WIRE DISPATCH）、セッションカウンタ（№ N）、UNDO、フィルタートリガーを載せる。面（セッション）に属する操作だけを置き、カードに属する操作は置かない。ヘッダーなしのジェスチャー復帰案は mystery meat（[[000639]]）で却下。左下に fixed 配置されていたフィルタートリガーもここへ収容し、カード下部の浮遊ターゲットを消した。
3. **Keep stamp**: ☆ をフッターから分離し、カード右上に密着する 48×48px の検印スタンプ風チップ（不透明クリーム地 + 1.5px チャコール罫、保存済はチャコール地に反転）として配置する。visual-preview では OG 画像の右上、テキストカードではヘッダー右上で、位置規則は両モードで統一。
   - 44px でなく 48px なのは、画面端フラッシュ配置の誤り率 2 倍 (Henze 2011) への補償。
   - 一般的な半透明サークルオーバーレイは DESIGN_LANGUAGE.md の「no glass / sharp edges」に反するため却下。"Photo Wire Dispatch" の写真検印メタファーとして正当化する。
4. **Reading actions の 2 大ボタン化**: フッターは ARTICLE / SUMMARY のみとし、gap を 12px に拡大、min-height 48px。スワイプ方向への意味付け（右=保存の Tinder 型）は、既存の「どちらに投げても既読」という筋肉記憶を壊し意図しない保存を量産するため却下。
5. **UNDO + 遅延既読フラッシュ**: バックエンドに既読解除 RPC が存在しないため、既読 API 呼び出しを 1 枚分保留する single-slot バッファ（`dispatch-session.svelte.ts`）を新設する。次のカードの dismiss / pagehide / unmount で確定送信し、UNDO は保留をキャンセルしてカードを山へ戻す。undo 深度は 1（Tinder Rewind と同型）。
   - 代替案「既読解除 RPC の追加」: バックエンド変更と既読イベントの意味論変更（append-first 原則との整合検討）が必要になり、第一弾のスコープを超えるため却下。フロント側の遅延で同じ UX を達成できる。

## Consequences

### Pros

- 誤タップの 3 構造原因（3 ボタン近接 / ナビ近接 / 浮遊フィルターボタン）をすべて排除。主要ターゲットは実証的推奨値（~9mm 級）を満たす。
- 誤スワイプが UNDO で回復可能になり、スワイプ既読の心理的コストが下がる。
- 「保存判断はカード上・面の操作はヘッダー」という帰属規則が明文化され、両カードで一貫する。
- 画面ロジック（undo / 遅延フラッシュ / カウンタ）が `dispatch-session.svelte.ts` に分離され、ブラウザ非依存で単体テスト可能。
- ナビ高さ + 予約 padding（44px + safe-area）ぶんの縦空間をカードが回収。

### Cons / Tradeoffs

- ボトムナビの一貫性が画面によって破れる（[[000716]] の例外第 1 号）。例外を増やすときは本 ADR と同水準の正当化を要求すること。
- 保留中に app が kill されると最後の 1 枚の既読がバックエンドに届かない（pagehide フラッシュで大半は回収）。
- UNDO はローカル復元であり、確定送信後の取り消しはできない（undo 深度 1 の設計どおりだが、ユーザ期待とずれる可能性は残る）。
- SwipeFeedCard の click 駆動スペック 6 件は Svelte 5 `track_reactivity_loss` のテスト環境問題で HEAD 時点から失敗しており、本改修でも未解消のまま（worktree でベースライン一致を確認済み）。テスト環境の負債として残る。
- 外部リンク（タイトル ⇗）とヘッダー UNDO の位置が近い端末があり得る。実機での誤タップ再計測は運用フォローとする。
