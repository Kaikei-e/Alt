# ADR-000401: Knowledge Home Phase 2 の UI を SvelteKit で実装する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-frontend-sv (TypeScript/SvelteKit) — Knowledge Home UI 全体

## Context

[[000398]] で Knowledge Home のイベントストア・投影（Phase 1 バックエンド）が完成し、[[000400]] で API 経路が確定した。Phase 1 では 3 RPC、6 Usecase、8 マイグレーション、Projector ジョブまでを実装済み。

Phase 2 では、この API を消費するフロントエンド UI を構築する。Knowledge Home は従来の「ダッシュボード」とは異なり、**ナレッジ操作の起点**（starting point）として設計されている。各アイテムに WHY（なぜ今表示されているか）を提示し、ユーザーが次のアクションを即座に取れるクイックアクション（open / ask / listen / dismiss）を提供する。

既存の alt-frontend-sv は Svelte 5 Runes + TailwindCSS v4 で構築されており、`usePulse.svelte.ts` 等の Headless Composable パターンが確立している。このパターンに沿った実装とする。

## Decision

Knowledge Home の UI を以下の 4 パートに分けて実装する。

---

### パート 1: データ層（Connect-RPC クライアント + Hook）

**方針**: 既存の `usePulse.svelte.ts` パターンに倣い、Svelte 5 Runes ベースの Headless Composable を採用する。

**構成**:

| ファイル | 役割 |
|---------|------|
| `src/lib/connect/knowledge_home.ts` | Connect-RPC クライアント。Proto → アプリ型への変換を担う |
| `src/lib/hooks/useKnowledgeHome.svelte.ts` | 状態管理 Hook。fetch / loadMore / trackSeen / trackAction / dismissItem |
| `src/lib/connect/index.ts` | バレルエクスポートに KnowledgeHome を追加 |

**設計判断**:

- `getKnowledgeHome` は cursor ベースのページネーションを受け取り、`loadMore()` で追加読み込みを行う
- `trackSeen` / `trackAction` は fire-and-forget（エラーを握りつぶす）。UX をブロックしない
- `dismissItem` はオプティミスティック削除（即座に UI から除去し、バックグラウンドで tracking API を呼ぶ）
- `ConnectError.Unauthenticated` は `/login` へリダイレクト（usePulse と同一パターン）

---

### パート 2: WHY 表現とアクション（Badge + QuickAction）

**方針**: WHY reason コードを視覚的なバッジに変換する。マッピングロジックはテスト容易性のために純粋関数として切り出す。

**構成**:

| ファイル | 役割 |
|---------|------|
| `why-reason-map.ts` | reason code → label / icon / color のマッピング関数 |
| `WhySurfacedBadge.svelte` | Pill 型バッジ。`$derived` でリアクティブにマッピング |
| `QuickActionRow.svelte` | Open / Ask / Listen / Dismiss の 4 アイコンボタン |

**WHY マッピング**:

| Code | ラベル | アイコン | カラー |
|------|--------|---------|--------|
| `new_unread` | New | Sparkles | blue |
| `in_weekly_recap` | In Recap | CalendarRange | purple |
| `tag_hotspot` | Trending: {tag} | Tag | green |
| `summary_completed` | Summarized | FileText | teal |
| `pulse_need_to_know` | Need to Know | Activity | orange |

不明なコードにはグレーのフォールバックを返す。`tag_hotspot` の場合のみタグ名をラベルに付与する。

---

### パート 3: コンポーネント群（Card / Bar / Stream / 補助）

**方針**: KnowledgeCard を単位とし、KnowledgeStream がリスト描画・見たことトラッキング・無限スクロールを担う。

**構成**:

| ファイル | 役割 |
|---------|------|
| `KnowledgeCard.svelte` | 1 アイテムのカード。タイトル → WHY バッジ×2 → 要約 → タグ×3 → アクション行 |
| `TodayBar.svelte` | 水平統計バー。新着数・要約済み・未処理 + トップタグ + Recap/Pulse ショートカット + 検索 |
| `KnowledgeStream.svelte` | カードリスト + IntersectionObserver による seen tracking + 無限スクロール |
| `KnowledgeHomeSkeleton.svelte` | ローディング中のパルスアニメーション |
| `KnowledgeHomeEmpty.svelte` | アイテムゼロ時の空状態メッセージ |
| `MiniRecallPanel.svelte` | デスクトップ右サイドバー。トップタグ + Recap/Pulse ショートカット |

**Seen Tracking の仕組み**:

1. `IntersectionObserver`（threshold: 0.5）で各カードの可視性を監視
2. 可視になったアイテムを `Set` に蓄積
3. 2 秒のデバウンスで `trackHomeItemsSeen` API を batch 呼び出し
4. `MutationObserver` で動的に追加されるカードも自動監視対象に追加

**無限スクロール**:

- リスト末尾に sentinel 要素を配置
- `IntersectionObserver`（rootMargin: 200px）で事前に `onLoadMore` を発火
- `hasMore && !loading` ガードで重複リクエストを防止

---

### パート 4: ページルートとナビゲーション

**方針**: レスポンシブ対応。デスクトップは 2 カラム（Stream + MiniRecallPanel）、モバイルはシングルカラム。

**構成**:

| ファイル | 役割 |
|---------|------|
| `src/routes/(app)/home/+page.svelte` | ページルート。`useViewport()` でレスポンシブ分岐 |
| `Sidebar.svelte` | デスクトップナビゲーションに "Knowledge Home" を最上位に追加 |
| `src/lib/config/navigation.ts` | デスクトップ・モバイル両方のナビゲーション定義に追加 |

**レスポンシブレイアウト**:

- デスクトップ: `PageHeader` → `TodayBar` → `KnowledgeStream (flex-1)` + `MiniRecallPanel (w-72)`
- モバイル: `TodayBar` → `KnowledgeStream` フル幅

**ナビゲーション**:

- Sidebar: "Knowledge Home" を Dashboard の上に追加（Lightbulb アイコン）
- FloatingMenu: "other" カテゴリに追加

**Exposure Session**:

- `onMount` で `crypto.randomUUID()` によりセッション ID を生成
- seen tracking の全バッチにこの ID を付与し、同一セッション内の重複を防止

## Consequences

### Pros

- **既存パターンとの一貫性**: usePulse / Connect-RPC クライアントと同じ構造で、開発者の認知負荷が低い
- **テスト容易**: WHY マッピングロジックを純粋関数に切り出し、Node テストで高速に検証可能（27 テスト全パス）
- **段階的デプロイ**: `PUBLIC_ENABLE_KNOWLEDGE_HOME` 環境変数で UI の公開タイミングを制御可能
- **オプティミスティック UI**: dismiss はサーバー応答を待たず即座に反映。UX のレスポンスが速い
- **Seen tracking のバッチ処理**: 個別リクエストではなく 2 秒デバウンスの batch で API 負荷を抑制

### Cons / Tradeoffs

- **IntersectionObserver の複雑性**: MutationObserver との組み合わせでコードが増える。ただし KnowledgeStream に閉じているため影響範囲は限定的
- **Sidebar のハードコード**: ナビゲーション定義が `Sidebar.svelte` 内にインラインで定義されており、`navigation.ts` と二重管理になっている（既存の技術的負債）
- **Feature flag の実装が環境変数のみ**: ユーザー単位のロールアウトには別途仕組みが必要
