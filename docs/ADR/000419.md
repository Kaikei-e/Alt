# ADR-000419: Knowledge Home Admin の Trigger Backfill CTA を共通 Button variant に寄せて hover コントラストを安定化する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-frontend-sv (SvelteKit) — Knowledge Home Admin page, shared Button usage

## Context

Knowledge Home Admin の `Trigger Backfill` ボタンは、共通 `Button` の variant に加えてページ固有の hover class を重ねていた。その結果、テーマ変数の組み合わせによって hover 時に背景と文字色のコントラストが崩れ、ボタンが白飛びしたように見える状態が発生していた。

このボタンは Admin 画面の主要 CTA であり、hover 時の視認性崩れは操作ミスや状態把握の阻害につながる。一方で、この問題は shared `Button` 自体の設計不備というより、ページ側で局所 override を足し過ぎたことが原因だった。

## Decision

**`Trigger Backfill` はページ固有 hover override をやめ、共通 `Button` の `default` variant をそのまま使う。**

### 変更内容

1. Admin ページの `Trigger Backfill` ボタンを `outline` + custom hover class から `default` variant に変更した。
2. ページ固有の hover 色上書きを削除し、shared button のコントラスト定義に委ねた。
3. CTA は primary action として見せ、局所 class ではなく variant で意味を表現する形に戻した。

### 設計判断

1. **CTA の意味を variant で表す** — primary action をページ固有 class で組み立てず、shared component の semantic variant に寄せる。
2. **hover 問題は局所調整より override 削減で直す** — さらに class を足して修正するより、衝突の元を消す。
3. **Admin 画面だけの特例を増やさない** — button system の一貫性を優先する。

## Consequences

### Pros

- hover 時の白飛びが解消し、`Trigger Backfill` の視認性が安定する
- CTA が admin 画面内で primary action として明確になる
- shared Button variant の利用方針が一貫する

### Cons / Tradeoffs

- 以前の `outline` ベースより見た目の主張が強くなる
- page 固有の細かい配色調整は減るため、今後のテーマ調整は shared button 側との整合を見ながら行う必要がある
