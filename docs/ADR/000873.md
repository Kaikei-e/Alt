# ADR-000873: Knowledge Loop の visibility/completion lifecycle を dismiss_state から分離し SurfacePlanRecomputed projector branch と stream inline entry / decision intent metadata で OODA 表面を再設計する

## Status

Accepted

## Date

2026-05-01

## Affected Services

- **knowledge-sovereign (Go)** — schema (00018) / driver / projector に visibility lifecycle と SurfacePlanRecomputed branch を追加
- **alt-backend (Go)** — domain / sovereign-client / transition usecase / stream handler に lifecycle と intent metadata と inline entry attach を実装
- **alt-frontend-sv (TypeScript / Svelte)** — tile / connect client / hooks / transition route / page を inline stream + intent metadata 経路に切替
- **proto** — visibility / completion / intent metadata フィールドを wire schema に追加

## Context

[[000831]] で Knowledge Home を Knowledge Loop 状態機械 UI として再定義してから半月弱、本番で動かしている過程で 4 つの「設計の借り」が同時に表面化した。

### 1. `dismiss_state` が 2 軸を抱え込んでいた

旧 schema は `dismiss_state` ENUM (`active` / `deferred` / `dismissed` / `completed`) 1 列で「lifecycle セマンティクス」と「読み取り経路の表示可否」を兼ねていた。`GetKnowledgeLoopEntries` の WHERE 句は `dismiss_state = 'active'` で hard-filter していたため、

- `HomeItemOpened` で `completed` に遷移したエントリ → Continue surface に出すべきだが filter で消える
- `HomeItemDismissed` で `dismissed` に遷移したエントリ → Review surface に出すべきだが filter で消える

という構造的矛盾が残っていた。Continue / Review 専用の派生 view を作る回避策はコスト過剰で、代わりに「一時的に dismiss_state = active を維持する」 hack が projector 内に散らばっていた。

### 2. `KnowledgeLoopSurfacePlanRecomputed` event が projector で取り回せない

[[000845]] で v2 surface planner を knowledge-sovereign に移管した際、planner からは `KnowledgeLoopSurfacePlanRecomputed` system event が出る前提で contract を切ったが、対応する projector branch は未実装のままだった。結果、planner が再計算した surface_bucket / loop_priority / planner version / score inputs は append-only event log にだけ残り、`knowledge_loop_entries` projection には反映されない状態が続いていた。canonical contract ([[knowledge-home-phase0-canonical-contract]]) における "disposable projection" 不変条件 (read model はイベントから再構築できる) は守られていたが、運用時の即応性が欠けていた。

### 3. Stream `EntryAppended` / `EntryRevised` が entry_key だけしか運ばない

`StreamKnowledgeLoopUpdates` は projection seq の単調進行を通知するためだけに entry_key と revision を返し、FE は受信のたび `GetKnowledgeLoop` を coalesce 経由で再 fetch していた。Loop UI の遷移直後に複数 frame が立て続けに来るケース (例: HomeItemOpened → SurfacePlanRecomputed → DecisionPresented の 3 連) で、FE 側の coalesce バッファが詰まり「楽観 UI が反映されたあとに更新が遅延 (200–500ms) してフリッカーする」現象が発生していた。

### 4. `TransitionKnowledgeLoopRequest` に intent metadata 経路が無い

FE は decision options を組み立ててユーザに見せ、その中の特定 option を選んでもらった上で `TransitionKnowledgeLoop` を呼ぶ。にもかかわらず request schema は entry_key / from_stage / to_stage / trigger / observed_projection_revision の 5 フィールドだけで、

- ユーザに提示した intent 集合 (`presented_intents`)
- ユーザが選んだ intent (`acted_intent`)
- 選択された option の `action_id`
- 行動先の `target_type` / `target_ref`
- 遷移結果が Continue surface に着地すべきかの `continue_flag`

を載せる場所が無かった。結果、`KnowledgeLoopActed` payload の replay は意図情報が欠落した「stage transition だけが残る痕跡」になり、Knowledge Home の "Why as first-class" 不変条件 ([[knowledge-home-phase0-canonical-contract]]) を transition 履歴側で守れていなかった。

### 観測

`alt-frontend-sv/src/routes/(app)/loop/+page.svelte` の挙動: HomeItemOpened を打つと一覧から消え、Continue plane に出てくるまで 1 秒近くラグがある (refetch 待ち)。HomeItemDismissed すると Review plane に何も入らない。SurfacePlanRecomputed event は cron の Reaper / Surface 再計算で 1 日数百件出ているが、`knowledge_loop_entries.surface_bucket` への反映は projector restart 時の reproject にだけ依存していた。

## Decision

5 つの commit に分けて積層的に解消する (proto → migration → projector → backend → frontend)。

### 1. `dismiss_state` から visibility / completion を物理的に分離 (migration 00018)

`knowledge_loop_entries` に 2 つの ENUM 列を追加する:

- `visibility_state loop_visibility_state` (`visible` / `hidden` / `snoozed`) — read path の inclusion gate
- `completion_state loop_completion_state` (`open` / `completed` / `dismissed`) — semantic lifecycle

`dismiss_state` は **compatibility metadata** として残す。read filter は `visibility_state = 'visible'` に切替え、bucket-aware backfill rule で既存行を埋める:

```sql
UPDATE knowledge_loop_entries SET
  visibility_state = CASE
    WHEN dismiss_state = 'active'                                      THEN 'visible'
    WHEN dismiss_state = 'dismissed' AND surface_bucket = 'review'     THEN 'visible'
    WHEN dismiss_state = 'completed' AND surface_bucket = 'continue'   THEN 'visible'
    WHEN dismiss_state = 'deferred'                                    THEN 'snoozed'
    ELSE 'hidden'
  END,
  completion_state = CASE
    WHEN dismiss_state = 'completed' THEN 'completed'
    WHEN dismiss_state = 'dismissed' THEN 'dismissed'
    ELSE 'open'
  END;
```

`idx_kle_visible_surface_rank (user_id, lens_mode_id, surface_bucket, loop_priority, render_depth_hint DESC, freshness_at DESC, projection_seq_hiwater DESC) WHERE visibility_state = 'visible'` を追加し、`knowledge_loop_entries_public` view に新 2 列を expose する (read handler は public view 経由のみ、`projected_at` は意図的に除外)。

### 2. SurfacePlanRecomputed projector branch を独立させ patch-only に閉じる

`knowledge_loop_projector/surface_plan_recomputed.go` を新設し、`EventKnowledgeLoopSurfacePlanRecomputed` を:

1. payload を parse して `entry_inputs[]` (entry_key + SurfaceScoreInputs) と planner_version を取り出す
2. `decideBucketV2(SurfaceScoreInputs)` で surface bucket を決定
3. driver `PatchKnowledgeLoopEntrySurfacePlan(...)` で **planner-only 列のみ** UPDATE
4. seq_hiwater guard で out-of-order を no-op にする
5. 適用された場合のみ `recomputeSurfaces(...)` を呼んで 4 surface を再構築

driver 側の SQL は **why_* / lifecycle / freshness / artifacts / action metadata を一切触らない**:

```sql
UPDATE knowledge_loop_entries SET
  projection_revision     = projection_revision + 1,
  projection_seq_hiwater  = GREATEST(projection_seq_hiwater, $5),
  source_event_seq        = GREATEST(source_event_seq, $5),
  projected_at            = NOW(),
  surface_bucket          = $6,
  render_depth_hint       = $7,
  loop_priority           = $8,
  surface_planner_version = $9,
  surface_score_inputs    = $10
WHERE user_id = $1 AND tenant_id = $2 AND lens_mode_id = $3 AND entry_key = $4
  AND projection_seq_hiwater <= $5
```

これは "system replan は user-derived な artifact / 意図を破壊しない" という single-responsibility を SQL レベルで強制する設計。

### 3. WhyMappingVersion を v7 → v8 に bump

`knowledge_loop_projector/constants.go` の `WhyMappingVersion = 8` で、reproject 時に新 branch validation を含めることを operator に通告する。runbook の確認手順に SurfacePlanRecomputed の applied 件数 / skipped (seq_hiwater) 件数を追記する余地を残す (follow-up)。

### 4. HomeItemOpened / HomeItemDismissed の lifecycle を明示する

projector の対応 branch で

- `HomeItemOpened` → `dismiss_state = COMPLETED, visibility = VISIBLE, completion = COMPLETED` (Continue surface に残る)
- `HomeItemDismissed` → `dismiss_state = DISMISSED, visibility = VISIBLE, completion = DISMISSED` (Review surface に残る)

を明示し、driver `UpsertKnowledgeLoopEntry` に **lifecycle 列が UNSPECIFIED の場合の default 解決ヘルパ** (`defaultVisibilityState` / `defaultCompletionState`) を入れて旧コード経路 / 旧 sidecar artefact との互換を保つ。`PatchKnowledgeLoopEntryDismissState` は dismiss_state と一緒に visibility / completion も同時に書く (`lifecycleVisibilityForDismissPatch` / `lifecycleCompletionForDismissPatch`)。

### 5. TransitionKnowledgeLoopRequest に intent metadata の 6 フィールドを追加

`proto/alt/knowledge/loop/v1/knowledge_loop.proto`:

```proto
message TransitionKnowledgeLoopRequest {
  // ... 既存 5 フィールド ...
  repeated DecisionIntent presented_intents = 16;
  optional DecisionIntent acted_intent      = 17;
  optional string         action_id         = 18;
  optional ActTargetType  target_type       = 19;
  optional string         target_ref        = 20;
  bool                    continue_flag     = 21;
}

message KnowledgeLoopActedPayload {
  // ... 既存 ...
  optional string        action_id   = 10;
  optional ActTargetType target_type = 11;
  optional string        target_ref  = 12;
}
```

alt-backend handler が proto enum を文字列化して usecase に渡し、`buildTransitionEvent` が event payload に転載する。これにより `KnowledgeLoopDecisionPresented` / `KnowledgeLoopActed` の payload を replay するだけで「ユーザに何を見せて何を選ばせたか」を完全再現できる。

### 6. Stream frame に inline KnowledgeLoopEntry を載せる

`StreamKnowledgeLoopUpdates` の `EntryAppended` / `EntryRevised` に optional `inline_entry` (KnowledgeLoopEntry) を追加 (proto は本 ADR より前に schema を追加済、本 ADR で wire を有効化)。alt-backend handler は

- frame kind が Appended / Revised
- frame に entry_key がある
- inline `getUsecase` が利用可能

の条件下で `GetKnowledgeLoop(limit=5)` を 1 回呼び、forerground / bucket entries から `entry_key` 一致する 1 件を抜き、`projection_seq_hiwater >= ev.EventSeq` を満たすときだけ inline entry に attach する。FE は `applyStreamFrame` で stream 受信時に直接 entries / bucketEntries を更新し、`GetKnowledgeLoop` の coalesce-refetch を排除する。

stream classifier 側では:

- `HomeItemDismissed` を `Withdrawn` から `Appended` に切替 (Review surface に表示されるため withdraw ではない)
- `EventKnowledgeLoopReviewed` を session-level frame として既存の foreground / bucket recompose 経路に追加

### 7. read sort を loop_priority band → render_depth → freshness → seq_hiwater に固定

`GetKnowledgeLoopEntries` の ORDER BY を:

```sql
ORDER BY
  CASE e.loop_priority
    WHEN 'critical'   THEN 1
    WHEN 'continuing' THEN 2
    WHEN 'confirm'    THEN 3
    WHEN 'reference'  THEN 4
    ELSE                   5
  END ASC,
  e.render_depth_hint        DESC,
  e.freshness_at             DESC,
  e.projection_seq_hiwater   DESC
```

projection_seq_hiwater 単独 sort は「最新 event 順」なので意味的には正しいが、Loop の運用上は priority band が上位の決定軸 (critical → continuing → confirm → reference) で、band 内で render_depth → freshness と並べたい。これは business semantics ではなく **rendering 順** なので feedback memory `feedback_no_sql_logic.md` (SQL に CASE 式で business logic を持たせない) には抵触しない判断とした (CASE は ENUM の sortable label への mechanical mapping のみ)。

### 8. FE: tile / hooks / transition route の整備

- `LoopEntryTile.svelte`: 表示中の `decisionOptions` から `presented_intents` を集計、選択された option から `acted_intent` / `action_id` / `target_type` / `target_ref` / `continue_flag` を組み立てる `TransitionMetadata`。`open` 意図は **transition 結果を待たず先に window.open する** (`onOpen` callback 経由) よう分岐し、ユーザの体感遅延を削る。
- `useKnowledgeLoop.svelte.ts`: `transitionTo` / `dismiss` / `reviewAction` に metadata を受け取らせ、request body に展開。新 `applyStreamFrame(frame)` が Appended / Revised の inline entry を foreground / bucket に直接 splice、Withdrawn を local dismiss に変換する。
- `loop-stream-frames.ts`: inline_entry を `mapProtoEntry` で domain shape に変換。
- `loop-transitions.ts`: `orient->observe` を ALLOWED に追加 (defer のステージ反転 path)。
- `routes/(app)/loop/transition/+server.ts`: 6 フィールドを enum allowlist で validate し `transitionKnowledgeLoopForUser` に forward。
- `routes/(app)/loop/+page.svelte`: legacy bucket-index fallback と `LoopSurfacePlane` への退避経路を撤去、常に `LoopPlaneStack` を描画する。

### 検討した代替案と却下理由

- **(却下) `dismiss_state` の ENUM を `active` / `deferred` / `dismissed_visible` / `dismissed_hidden` / `completed_visible` / `completed_hidden` に拡張**: schema 追加列を増やさず済むが、状態名が表示可否を内包すると semantic と rendering が再び癒着する。本 ADR の動機 (2 軸を分離する) に逆行。
- **(却下) read filter を SQL の `CASE WHEN dismiss_state = 'completed' AND surface_bucket = 'continue' THEN ...` で書く**: feedback memory `feedback_no_sql_logic.md` に反する。同等のロジックを Go 側で組むか projection 側で物理列を持つかの 2 択で、後者を採用。
- **(却下) FE が stream frame 受信後に `GetKnowledgeLoop` を都度 refetch (現行)**: 既存の coalesce 機構を流用できるが、burst 時のラグと flicker が user-visible regression を起こしていた。inline entry attach は server side で seq_hiwater guard を持つため reproject-safe で、往復が消える。
- **(却下) intent metadata を `payload` 専用のフィールドにのみ保持し proto を変えない**: replay 時に payload JSON のスキーマが implicit になり、ADR-immutable 原則 ([[knowledge-home-phase0-canonical-contract]]) で要請される "self-describing event" を破る。proto field 重複でも message 自己記述性を優先。
- **(却下) SurfacePlanRecomputed projector branch を `UpsertKnowledgeLoopEntry` の上書きで実装**: lifecycle / why / artifacts を planner が間接的に書き換える可能性が残り single responsibility 違反。`PatchKnowledgeLoopEntrySurfacePlan` で **planner-owned columns のみ** に絞ることで、system replan が user progress を破壊しないことを SQL の WHERE 句レベルで担保。
- **(却下) Stream frame に inline entry を載せず `BucketEntry` 単位の patch event を別途 emit**: アーキテクチャは綺麗だが contract 拡張コストが高く、FE は依然 2 段の merge ロジックを持つ必要が残る。inline entry attach は最小コストで refetch を消せるため採用。limit=5 の hot fetch は当面の実装上のトレードオフとして許容 (Cons 参照)。

## Consequences

### Pros

- **HomeItemOpened entry が Continue surface に正しく残り、HomeItemDismissed entry が Review surface に正しく残る**。 Knowledge Loop の OODA 表面 4 plane (Now / Continue / Changed / Review) が canonical contract 通りに観測可能になる。dismiss_state と read filter の 1:1 結合が解け、新しい lifecycle (例: 「snoozed だが visible にしたい」) を将来追加する際の自由度が増す。
- **SurfacePlanRecomputed event が projection に到達する経路が完成する**。v2 surface planner ([[000845]]) の再計算結果が DB に届くようになり、planner の精度改善を operator が観測できる。WhyMappingVersion v8 bump により reproject 時の検査範囲が明示される。
- **Stream inline entry attach により FE refetch が消え、Loop UI のレイテンシ揺らぎ (200–500ms 級の coalesce 待ち) が解消**。entries / bucketEntries が stream frame で 1 hop で決まる。`open` 意図は transition 完了を待たずブラウザ window.open に分岐するため体感遅延がさらに縮む。
- **TransitionKnowledgeLoopRequest と KnowledgeLoopActedPayload に intent metadata が乗り、replay で完全再現可能になる**。"Why as first-class" 不変条件が transition 履歴側でも守られる。
- **system replan が user-derived state を破壊しない構造的保証**。`PatchKnowledgeLoopEntrySurfacePlan` の SQL は planner-owned 列のみを SET し、why / lifecycle / freshness / artifacts は WHERE 文に並ばないため、誤って書き換えること自体が物理的に不可能。
- **TDD outside-in 遵守**。projector の SurfacePlanRecomputed branch / driver Patch メソッド / FE useKnowledgeLoop.applyStreamFrame / FE loop-stream-frames inline_entry mapping に対して unit テストを RED 先行で投入、proto buf lint + breaking pass、knowledge-sovereign / alt-backend は `-race` 付き go test green、alt-frontend-sv は vitest 1135 件 + svelte-check 0 errors green。

### Cons / Tradeoffs

- **`knowledge_loop_entries` の row width 2 列増 + index 1 個追加**。`idx_kle_visible_surface_rank` を partial index で抑えてはいるが、insert/update path が 2 columns 分重くなる。当面 row volume が万 order なので影響は無視できるが、観測指標 (write latency p95) を runbook 側で監視する追跡項目に入れる必要あり。**既存 `idx_kle_active_surface_rank` を残置している** (drop は backfill / 旧 read path の retire 確認後に別 ADR で扱う) ため、index 数は実質 +1 で 2。
- **WhyMappingVersion v8 bump の operator 周知が follow-up に残る**。`runbooks/knowledge-loop-reproject.md` への新 branch validation 追記、reproject 完走後の `surfacePatches` カウンタ確認などは本 ADR の射程外で、後続の docs 更新に分離。
- **Stream inline entry attach は entry あたり `GetKnowledgeLoop(limit=5)` を呼ぶため、burst event で N+1 fan-out が発生**。当面 limit=5 で hot rows しか fetch せず、frame の頻度は OODA transition 単位 (秒に 1 桁回) のため無視できるが、長期的には projector / handler 側の sidecar batch / per-tenant LRU cache で 1 query に畳むリファクタが必要。
- **stream classifier の `HomeItemDismissed` を Withdrawn → Appended に切替えた**。FE が古い build で動いている期間に `Withdrawn` 待ちのコードパスが silent に到達不能になる。alt-frontend-sv の本 ADR 対応 commit と同時にロールアウトする運用前提で、retired path の grep / lint 整備は次回のリファクタで実施。
- **proto に optional field を 9 個追加した** (TransitionKnowledgeLoopRequest 6 + KnowledgeLoopActedPayload 3)。`buf breaking` は green だが、wire の自己記述性が増えた分 message 1 件あたりのバイト数が微増 (実測 5–15 byte)。Stream RTT への影響は無視できるが、長期的に proto を slim 化する判断は別 ADR で扱う。
- **`presented_intents` が `decision_options[*].intent` の単純コピーに留まる**。実際には rendering filter (例: 一部 option を CTA から外したケース) を反映すべきだが、本 ADR では entry 側 schema を変えずに集計する最短実装に倒した。CTA filter が複雑化する将来は FE 側で intent 集合を別管理する設計に切り替える。
- **本 ADR は existing knowledge_loop_entries の lifecycle backfill を migration 一発で実行する**。online 運用中の knowledge-sovereign で 00018 を流すため、`UPDATE knowledge_loop_entries SET visibility_state = ..., completion_state = ...` の lock window が row 数に比例する。Atlas migration の `lock_timeout` DSN ([[000869]]) で詰まり時に fail-fast される設計が安全網になっているが、適用は read traffic の谷で実施するのが望ましい。
