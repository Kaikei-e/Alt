# ADR-000581: FeedDetailModal の空コンテンツ無限ループと link 空時のフィードバック欠如を修正する

## Status

Accepted

## Date

2026-03-25

## Affected Services

- alt-frontend-sv (TypeScript / SvelteKit)

## Context

Desktop の `/feeds/search` で特定の記事（「n8nを使ってみた | HEROZ Tech Blog」など）のモーダルを開いた際、Full Article ボタンが活性化せず押せない不具合が報告された。

### 根本原因

`FeedDetailModal.svelte` の auto-fetch `$effect` が、バックエンドが空コンテンツ (`content: ""`) を返した場合に**無限ループ**を起こしていた。

auto-fetch の `$effect` は以下の 5 条件がすべて true のとき `handleFetchFullArticle()` を呼び出す:

```typescript
open && feed?.normalizedUrl && !articleContent && !isFetchingContent && !contentError
```

空コンテンツ応答時のフローは:

1. `articleContent = response.content || null` → 空文字は falsy なので `null` のまま
2. エラーは throw されないため `contentError` も `null` のまま
3. `finally` ブロックで `isFetchingContent` が `false` に戻る
4. `isFetchingContent` の変化で `$effect` が再評価される
5. 5 条件すべて true → 再度 `handleFetchFullArticle()` が呼ばれる → 無限ループ

ボタンが "Loading..." (disabled) と "Full Article" (idle) を高速で行き来し、事実上操作不能になっていた。

### モバイル側は対策済み

`SwipeFeedCard.svelte` (line 218-225) と `VisualPreviewCard.svelte` (line 137) では、`if (res.content) { ... } else { contentError = "Could not fetch article content"; }` のパターンで空コンテンツを明示的にエラーとして扱っており、同様の無限ループは発生しない。

デスクトップの `FeedDetailModal.svelte` だけがこのガードを欠いていた。

## Decision

**モバイルと同じパターンで空コンテンツ応答をエラーとして扱い、`contentError` を設定することで `$effect` の無限ループを断ち切る。**

レスポンス処理ロジックをテスト可能な純粋関数 `processArticleFetchResponse` として抽出し、コンポーネント内の 2 箇所（初回 fetch、リトライ fetch）で使用する。

### 変更内容

1. **`FeedDetailModal.logic.ts` を新規作成** — `processArticleFetchResponse` 関数を定義。`response.content` が空文字・空白のみの場合、`contentError` に `"Article content could not be retrieved"` をセットし `articleContent` を `null` にする。有効なコンテンツがある場合はそのまま返す。

2. **`FeedDetailModal.svelte` を修正** — 2 箇所の `articleContent = response.content || null` を `processArticleFetchResponse` 呼び出しに置換。

   ```typescript
   // Before (line 259, 285)
   articleContent = response.content || null;
   articleID = response.article_id || null;

   // After
   const result = processArticleFetchResponse(response);
   articleContent = result.articleContent;
   articleID = result.articleID;
   if (result.contentError) {
       contentError = result.contentError;
   }
   ```

3. **`FeedDetailModal.logic.spec.ts` を新規作成** — 4 テストケース（正常コンテンツ、空文字、空白のみ、空 article_id）

4. **auto-fetch `$effect` に `normalizedUrl` 空チェックを追加** — Meilisearch と DB の不整合により `link` が空で返される記事がある。この場合、auto-fetch がスキップされるだけでユーザーへのフィードバックがなかった。`$effect` 内で `normalizedUrl` が空の場合に `contentError = "Article URL is not available"` を設定し、エラーメッセージを表示するようにした。

5. **`handleFetchFullArticle` の早期リターンを分離** — `!feed?.normalizedUrl || isFetchingContent` の複合条件を分離し、`normalizedUrl` 空の場合は `contentError` を設定してから return するように変更。

### 背景: Meilisearch と DB の不整合

調査の結果、Meilisearch `articles` インデックスに存在する記事（例: `f8b9d3a0-...` "n8nを使ってみた | HEROZ Tech Blog"）が DB `articles` テーブルに存在しないケースを確認した。`SearchFeedMeilisearchUsecase` は Meilisearch 検索後に `GetFeedURLsByArticleIDs` で DB から URL を取得するが、マッチしない場合 `Link` が空文字になる。この不整合の根本解決は search-indexer の reindex で対応可能だが、今回はフロントエンドの UX 改善に限定した。

### 無限ループが止まる仕組み

- `contentError` がセットされる → auto-fetch の `!contentError` 条件が `false` → `$effect` が `handleFetchFullArticle()` を呼ばなくなる
- `articleButtonState` が `"error"` になる → ボタンが "Try again" と赤い destructive バリアントで表示される
- ユーザーが "Try again" をクリックすると `handleRefetchArticle` が `contentError` をリセットして再取得を試みる

## Consequences

### Pros

- **無限ループ解消** — スクレイピング不可の記事でもボタンが操作可能になる
- **ユーザーへの適切なフィードバック** — "Try again" ボタンとエラーメッセージが表示され、何が起きたか分かる
- **モバイルとの一貫性** — 全プラットフォームで空コンテンツをエラーとして扱うパターンに統一
- **テスト可能な設計** — レスポンス処理を純粋関数に抽出したことで、ブラウザテスト環境なしにロジックを検証できる
- **既存動作に影響なし** — 正常にスクレイピングできる記事の動作は変わらない
- **link 空の場合のフィードバック** — "Article URL is not available" メッセージが表示され、ボタンが silent no-op にならない

### Cons / Tradeoffs

- **ファイル追加** — `FeedDetailModal.logic.ts` を新たに追加した。コンポーネント内のインライン修正でも対応可能だったが、テスタビリティを優先した
- **空コンテンツ = エラーという判断** — バックエンドが意図的に空コンテンツを返すケース（将来的にあれば）も一律エラーとして扱われる。現時点ではそのようなケースは存在しない
