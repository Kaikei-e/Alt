# ADR-000716: モバイルナビをボトムタブバー永続化に一本化し FloatingMenu を廃止する

## Status

Accepted

## Date

2026-04-14

## Affected Services

- alt-frontend-sv (SvelteKit 2.x / Svelte 5 Runes / TypeScript / TailwindCSS v4)

## Context

alt-frontend-sv のモバイル UI には複数のナビゲーション系が並走していた。

- **3 系統のナビ定義が重複**: `src/lib/config/navigation.ts` (20 項目、誰も import していない死コード)、
  `FloatingMenu.svelte` (16 項目をインライン定義)、`menu-page.ts` (13 項目)。
  同じ宛先 (Ask Augur / Job Status 等) が三カ所に散らばり、片方の変更が他方に波及しない
  運用負債になっていた。
- **FloatingMenu がボトムタブバーを置換**: `/augur`, `/feeds/search`, `/augur/[id]` では
  `HIDE_PATHS` によりボトムナビを非表示にし、代わりに右下固定の FAB 風シートが主ナビを兼ねる
  構造だった。これは Material 3 / Nielsen Norman Group および主要 RSS アプリ
  (Reeder / NetNewsWire / Feedly / Readwise Reader) の 2025 ベストプラクティス
  「タブバーは常駐、作成は top-right `+`、読み流し系は swipe/long-press で分離」
  に全面的に反していた。
- **アクセシビリティ未達**: `aria-current="page"` なし、ルート遷移時のフォーカス移動なし、
  skip link なし、ナビリンクに `role="tab"` + `aria-selected` という不適切な
  tablist 扱いが併用されていた。
- **ページ個別の viewport 競合**: `/search` (Reference Desk) のモバイルヘッダが画面端に張り付く、
  `/augur` モバイル入力フォームがボトムナビに覆われる、`/recap` (3-day / 7-day) で
  `min-h-[100dvh]` がボトムナビ下に食い込んで下部が見切れる、`/recap` で 3-day ビューでも
  `SwipeRecapScreen` が "7 Days Recap" とハードコード表示される等の周辺バグも同居していた。

本来は 1 画面 1 プライマリ面を守ったうえで、ナビは単一データソースから描画されるべき。
これらを同じ改修で解消するため、モバイルナビゲーションの全面整理を行った。

Web 調査 (Material 3 expressive 2025-05 / Apple HIG / WCAG 2.2 / NN/g / SvelteKit docs) に
基づき、次の原則を採用した。

1. 1 画面に 1 プライマリ面。ボトムナビ常駐、FAB / フローティングメニューは併設しない。
2. タブ数は 3–5。5 タブで続行。
3. 永続レイアウト: ボトムナビは `(app)/+layout.svelte` 直下にマウントしルート遷移で再マウントしない。
4. a11y: `<nav aria-label="Main">` / `aria-current="page"` / `afterNavigate` による
   `<main tabindex="-1">` へのフォーカス移動 / skip link。
5. PWA safe-area: `viewport-fit=cover` 前提、`env(safe-area-inset-bottom)` を正しく 1 回だけ使う。
6. ナビ定義は `src/lib/config/navigation.ts` を唯一の真実とする。

## Decision

以下を採用する。

### 1. データソース単一化

- `src/lib/config/navigation.ts` を書き直し、`NAV_TABS` (5 タブ) と
  `MOBILE_MENU_SECTIONS` (Browse / Recap / AI & Insights / Settings / Admin) を唯一の export とする。
- 旧 `desktopNavigation` / `mobileMenuItems` / `mobileCategories` を削除。
- `src/lib/components/mobile/bottom-nav.ts` と `menu-page.ts` は navigation.ts への
  薄い re-export に縮退。

### 2. タブ構成: Home / Swipe / Search / Augur / Menu

- Home (`/home`, Lightbulb)
- Swipe (`/feeds/swipe/visual-preview`, Rss) — 最頻利用の Visual Preview への直リンク
- Search (`/search`, Search)
- Augur (`/augur`, Bird) — FloatingMenu 廃止の裏返しで第 1 級市民に昇格
- Menu (`/menu`, Menu) — 二次的宛先のランチャーグリッド

Recap / Acolyte / Manage Feeds / Job Status / Manage Domains 等は `MobileMenuPage`
グリッドに集約する。Menu タブは secondary destination の単一入口。

### 3. FloatingMenu の全面廃止

- `src/lib/components/mobile/feeds/swipe/FloatingMenu.svelte` (484 行) を削除。
- 呼び出し側 (`SearchFeedsClient.svelte`, `ChatWindow.svelte`) から import/描画を削除。
- E2E POM `FloatingMenuComponent.ts` を削除、`pomFixtures.ts` および 5 つの mobile page POM
  (`MobileMorningLetterPage` / `MobileTagTrailPage` / `Mobile3DayRecapPage` /
  `MobileSwipePage` / `MobileEveningPulsePage`) から `floatingMenu` フィールドを除去。

### 4. ボトムナビ永続化と a11y

- `MobileBottomNav.svelte`: `role="tab"` / `aria-selected` を撤廃し
  `aria-current="page"` をアクティブリンクに付与。`<nav aria-label="Main navigation">` を維持。
- `ResponsiveLayout.svelte`: mobile 時は常に `<MobileBottomNav>` を描画 (HIDE_PATHS 廃止)。
  `afterNavigate(() => mainEl?.focus())` で `<main tabindex="-1" id="main">` に
  フォーカス移動、skip link を追加。
- `src/app.html` は既に `viewport-fit=cover` 指定済みのため追加変更なし。

### 5. ページ個別の viewport 競合の修正

- **`/search` (Reference Desk)**: `.ref-page` に `px-4 md:px-0` を付与して横余白確保。
  `.ref-content` の条件付き `px-4` は親に統合して廃止。
- **`/augur` と `/augur/[conversationId]`**: `.augur-mobile-shell` を
  `position: fixed; top/left/right: 0; bottom: calc(2.75rem + env(safe-area-inset-bottom, 0px))`
  に変更し、main の `pb-[calc(...)]` / `min-h-screen` (100vh) / shell の 100dvh 計算の
  三重確保を解消。shell の下端がボトムナビの上端と一致する。
  document 直付けの touchmove preventDefault ハンドラはタブタップ合成を阻害する副作用があったため削除し、
  `.augur-thread` 側の `overscroll-behavior-y: contain` に役割を委譲。
  `.augur-input-fixed` の padding-bottom からは `env(safe-area-inset-bottom)` を取り除き、
  shell 側の一箇所で safe-area を反映する。
- **`/recap`**: `SwipeRecapScreen.svelte` にハードコードされていた
  `<h1>7 Days Recap</h1>` を `{window}-Day Recap` に変更し、`window: 3 | 7` prop を追加。
  `.recap-screen` の `min-height` を `calc(100dvh - 2.75rem - env(safe-area-inset-bottom, 0px))`
  に変更。ページ側のラッパ `min-h-[100dvh]` と skeleton の `h-[100dvh]` を撤去し、
  `ResponsiveLayout` の main `pb-[calc(...)]` にスクロール空間を委譲。

### 6. TDD 遵守

各 Phase で RED テストを先に書き、実装で GREEN にする手順を厳守。
`navigation.test.ts` / `bottom-nav.test.ts` / `menu-page.test.ts` /
`MobileBottomNav.svelte.test.ts` を新タブ構成 (Home/Swipe/Search/Augur/Menu) に書き直し、
`aria-current="page"` / `role="tab"` 不在 / ボトムナビ常駐 を検証するアサーションを追加。
結果は server project 905 tests pass / browser project のナビ系 (MobileBottomNav 10 +
SearchFeedsClient 4 + ChatWindow 8) 全 pass / type check 0 errors。

## Consequences

### Pros

- **1 データソース / 1 プライマリ面の原則が貫徹**: navigation.ts だけ編集すれば
  ボトムナビ / Menu グリッドの両方に即時反映される。死コードと重複定義が消滅。
- **アクセシビリティ強化**: `aria-current="page"` による現在位置の読み上げ、
  `afterNavigate` による SPA 遷移後のフォーカス再配置、skip link による
  キーボードユーザの動線短絡が揃った。WCAG 2.2 SC 2.4.1 (Bypass Blocks) と
  SC 2.4.7 (Focus Visible) の基礎を満たす。
- **ビューポート競合の三連発を同時修正**: Reference Desk / Augur / Recap の
  画面端張り付き・入力フォーム被り・下部見切れの 3 件を同一スコープで解消。
  Augur は `position: fixed` shell に切り替えたことで、iOS アドレスバー collapse
  (100vh vs 100dvh の乖離) の影響も受けなくなった。
- **タブタップ反応性改善**: document レベル touchmove preventDefault を撤廃し、
  わずかな指揺れで click 合成が阻害される iOS 挙動を解消。
- **Material 3 / 主要 RSS アプリとの一致**: FAB 併設を排し、二次アクションは
  Menu グリッドに統一することで、ユーザの学習済みパターン (Reeder / Feedly / Readwise)
  に整合した。
- **タブ構成をユーザ意図に合わせて最適化**: 初期提案の Feeds (`/feeds`) は Swipe
  (`/feeds/swipe/visual-preview`) に差し替え、最頻利用経路をトップタブに引き上げた。
  `/feeds` (Library) は Menu グリッドに降格。

### Cons / Tradeoffs

- `position: fixed` shell は augur ページを main の通常フローから切り離すため、
  他ページと同じ `className` 経由のスクロール挙動カスタマイズができなくなった。
  augur 限定の決定として shell 内部で完結している。
- FloatingMenu 固有の「1 タップで別画面ランチャー」体験は Menu タブへの 2 タップ
  (ボトムナビ → グリッド) に変わる。頻出二次宛先の多くは Swipe / Augur タブで
  直接届くため実用上の損失は小さいが、操作手順が 1 段深くなる点は残る。
- E2E POM の `floatingMenu` フィールドを参照するテストは存在しなかったため無害だが、
  将来的に二次アクション用 BottomSheet を別計画で導入する際、
  改めて POM コンポーネントを設計し直す必要がある。
- Recap の `window` prop は URL クエリ (`?window=7`) 駆動であり、
  タブ直接アクセスからは常に 3-day が初期値。Menu グリッドに 7-day 専用エントリは
  置かない方針 (URL 制御に統一)。

### Out of Scope

- `/feeds/search` や `/augur` のページ内二次アクション (フィルタ、新規会話ボタン等) の
  BottomSheet 化は別計画。
- デスクトップ Sidebar の整理は本 ADR の対象外。
- View Transitions API (`onNavigate`) の導入は a11y 修正と分離し別タスク。
- Material 3 Expressive (2025-05) の docked toolbar 移行はライブラリ追従を待つ。
