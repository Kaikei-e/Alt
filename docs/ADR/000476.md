# ADR-000476: Recall candidate card の表示契約を KnowledgeCard に揃える

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-frontend-sv (SvelteKit)

## Context

RecallRail の `RecallCandidateCard` は当初、タイトル・経過日数・reason badge・snooze / dismiss の最小表示に留めていた。そのため backend 側で `item` が補完されても、要約やタグが UI に出ず、Knowledge Home 本体の `KnowledgeCard` と表示契約がずれたままだった。

特に次の差分が問題だった。

- `publishedAt` より `firstEligibleAt` を優先して age を計算していた
- invalid date の防御がなく、将来的に不正値で表示が崩れる余地があった
- `summaryState` を見ずに本文領域自体が存在しないため、候補の説明力が低かった
- 空タグ除去や表示数制限の規約が `KnowledgeCard` と揃っていなかった

RecallRail は本体カードの完全コピーではなくてよいが、少なくとも「いつ本文を見せるか」「日付が壊れているとき何を出すか」「タグをどうトリムするか」は同じ契約である方がユーザー体験と保守性の両方で有利である。

## Decision

`RecallCandidateCard` の表示契約を `KnowledgeCard` に合わせ、summary・tags・relative time について同じ基準を採用する。

具体的には以下を決定した。

1. タイトルは `candidate.item.title` を優先し、欠落時のみ `itemKey` にフォールバックする
2. 本文は `summaryState === "ready"` かつ `summaryExcerpt` がある場合のみ表示する
3. タグは空文字を除外したうえで最大 2 個だけ表示する
4. relative time は `publishedAt` を優先し、なければ `firstEligibleAt` を使う
5. 日付が空または invalid の場合は `"recent"` を表示する
6. snooze / dismiss は従来どおり click 伝播を止め、open を誤発火させない

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|--------|------|-----------|
| RecallRail は従来どおり最小 UI のままにする | 実装差分は最小 | backend 側で item を補完しても表示価値が十分に出ない |
| `KnowledgeCard` をそのまま再利用する | 契約は完全一致する | RecallRail 専用の action row や compact layout まで崩れ、責務が過剰に共有される |
| pending / missing 時に skeleton も表示する | 本体との見た目は近い | RecallRail は縦幅制約が強く、compact rail ではノイズが増える |

## Consequences

### Pros

- RecallRail でも要約・タグ・日付の解釈が Knowledge Home 本体と一致する
- invalid date を `"recent"` に落とすことで表示崩れを防げる
- タグの空文字除去と表示上限により、無意味なチップや過密表示を避けられる
- browser test で open / snooze / dismiss の基本挙動を固定できる

### Cons / Tradeoffs

- `RecallCandidateCard` の責務が少し増え、単純なタイトルカードではなくなる
- 本体カードと似た規約が増えるため、将来の表示変更時は両者の同期を意識する必要がある
- skeleton を採用しないため、pending item は「本文なし」に見える
