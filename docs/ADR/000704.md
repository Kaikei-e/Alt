# ADR-000704: Admin 画面に Prometheus 連携の Observability パネルを Connect-RPC ストリーミングと allowlist ゲートウェイで追加する

## Status

Accepted

## Date

2026-04-13

## Affected Services

- alt-backend (Go) — Prometheus クライアント / allowlist ゲートウェイ / Connect-RPC サービス
- alt-butterfly-facade (Go) — 管理者トークン検証と chunked リレー
- alt-frontend-sv (TypeScript / Svelte 5 Runes) — Observability タブと useConnectAdminMetrics フック
- prometheus — スクレイプ先および PromQL 実行エンジン
- nginx — ストリーミング向け location 追加

## Context

Admin 画面の刷新後、運用者が「今この瞬間にシステムが健全か」を把握するための入口が Alt UI に存在せず、Grafana は内部ネットワーク専用のため Admin 権限を持つユーザーから直接到達できなかった。Grafana を代替する必要はないが、異常を 1 画面で検知できるランディングが欠けていた。

さらに 2026-04-13 に初版を実運用で確認した結果、二つの欠陥が露見した:
1. allowlist の PromQL が OTel 慣例の `http_server_*` 系を前提にしていたため、実スクレイプが公開する `http_request_duration_seconds_bucket` / `http_requests_total` / `recap_db_pool_checked_out` などと一致せず大半の行が em-dash になった。
2. Alt-Paper テーマの `--alt-warning` が `#ffff00` (純黄) に残っており cream 背景 `#faf9f7` に対するコントラスト比は 1.07:1。編集的デザイン言語と衝突し、警告の意図が伝わらなかった。

## Decision

Admin Observability パネルを以下の 4 つの原則で構築することに決定する。

1. **配置**: 既存 `AdminTabNavigation.svelte` に 7 番目の `Observability` タブを追加する (新画面は作らない)。
2. **伝送**: Connect-RPC のサーバーストリーミング `Watch` で 5 秒間隔の push を採用する。SSE / polling は使わない。既存 BFF (alt-butterfly-facade) が Connect-RPC 採用である点と整合する。
3. **PromQL allowlist 方式**: FE は `queryKey` + 許可された `window` / `step` enum のみを送出し、サーバー側でコンパイル済みの PromQL を合成する。raw クエリは受け付けない。
4. **Alt-Paper コントラスト遵守**: グローバルの `--alt-warning` / `--alt-error` / `--alt-success` は vaporwave テーマ依存のため、Alt-Paper テーマブロック内でのみ editorial ink 値に上書きし、Observability 配下に `--obs-*` スコープトークンを導入する。

実メトリクス名への追従と editorial ink への差し替えは Phase E として同時にマージする。

## 実装の概要

### alt-backend (Clean Architecture)

```
Driver → Gateway → Port → Usecase → Connect Handler
```

- Driver `driver/prometheus_client/client.go` — Prometheus HTTP API `/api/v1/query` と `/query_range` の薄いラッパ。timeout / bad_data / execution / unavailable を `QueryError.Kind` に分類し context timeout を内部で付与する。
- Gateway `gateway/admin_metrics_gateway/` — `allowlist.go` でキーと PromQL 文字列を静的に束縛、`admin_metrics_gateway.go` で input enum 検証 → singleflight → 10s cache → `golang.org/x/time/rate` による 5rps 制限の順で通過させる。Prometheus が時間切れのときは 500 ではなく `degraded: true` のスナップショットを返す。
- Port `port/admin_metrics_port/admin_metrics_port.go` — `Catalog / Snapshot / Healthy` の 3 メソッド。
- Usecase `usecase/admin_metrics_usecase/` — `GetSnapshotUsecase`, `StreamSnapshotsUsecase` (`time.Ticker(5s)` + `ctx.Done`), `CatalogUsecase`, および三者を束ねる `Facade`。
- Connect Handler `connect/v2/admin_monitor/handler.go` — proto `alt/admin_monitor/v1/admin_monitor.proto` から生成された `AdminMonitorServiceHandler` を実装。`X-Accel-Buffering: no` と `Cache-Control: no-cache` を常時付与し、エラーを `CodeInvalidArgument` / `CodeInternal` に分類する。
- DI `di/admin_monitor_module.go` — `config.AdminMonitor.Enabled` で gate。server 登録は serviceAuthInterceptor を使う (BFF が `X-Alt-Backend-Token` を `X-Service-Token` に差し替えて転送するため)。

### alt-butterfly-facade

`internal/handler/admin_monitor_proxy_handler.go` を追加。既存の `AdminProxyHandler` と異なり、Watch が長時間ストリームであるため context timeout を**適用しない**。`http.Flusher` を使ってチャンクごとに明示的に flush し、中間プロキシのバッファリングでストリームが止まらないようにする。管理者ロールの検証と service token への差し替えは二重化する。

### alt-frontend-sv

- Connect proxy `routes/api/v2/alt.admin_monitor.v1.AdminMonitorService/[...path]/+server.ts` — 他の `/api/v2` catch-all と異なり、alt-backend 直結ではなく BFF を経由する ("FE は backend に直結しない" 原則の遵守)。
- フック `lib/hooks/useConnectAdminMetrics.svelte.ts` — `@connectrpc/connect-web` の `AsyncIterable` を `for await` で消費し、`$state` で `snapshot / metrics / catalog / state / lastError` を公開する。再接続は exponential backoff + full jitter (1s → 30s cap)、各ストリームは 15 分 ±60s ローテートで中間プロキシの idle kill を回避する。`onDestroy` で `AbortController.abort()`。
- パネル `components/knowledge-home-admin/observability/ObservabilityPanel.svelte` — `--obs-*` スコープトークンを宣言し、Golden signals / Container / mq-hub / Recap / Services のセクションに 12 メトリクスを配置する。
- 行コンポーネント `MetricRow.svelte` — 上段にラベルと Δ (`▲ ▼` + 符号付き％)、中段に代表値 + sparkline + per-series digest、フォーカス / hover で `⋯ open in Grafana` リンクを表示する。閾値越え時のみ `--obs-warn` / `--obs-critical` をアクセントとして適用する。
- 付随コンポーネント `MetricSeriesDigest.svelte`, `ServiceHealthTable.svelte`, `DegradedBanner.svelte`、ユーティリティ `util.ts` (`computeDelta`, `topSeries`)。

### nginx

`nginx/conf.d/default.conf` に `location ~ ^/api/v2/alt\.admin_monitor\.v1\..+` を追加し、`proxy_buffering off` / `proxy_request_buffering off` / `proxy_http_version 1.1` / `proxy_read_timeout 3600s` を設定。Connect Protocol は HTTP/1.1 chunked で動作するため HTTP/2 end-to-end は不要で `grpc_pass` は使わない。

### Prometheus 側リソース保護

`compose/observability.yaml` に `--query.timeout=5s` / `--query.max-samples=1000000` / `--query.max-concurrency=8` を追加し、allowlist ゲートウェイ側で `--query.timeout=3s` (gateway ctx) / `2.5s` (upstream param) の 3 段タイムアウトを構成する。

### allowlist (2026-04-13 Phase E リフレッシュ)

| key | PromQL | unit |
|---|---|---|
| `availability_services` | `up{job=~"alt-backend\|pre-processor\|mq-hub\|recap-worker\|recap-subworker\|cadvisor\|nginx\|prometheus"}` | bool |
| `http_latency_p95` | `histogram_quantile(0.95, sum by (job,le) (rate(http_request_duration_seconds_bucket[5m])))` | seconds |
| `http_rps` | `sum by (job) (rate(http_requests_total[1m]))` | req/s |
| `http_error_ratio` | `sum by (job) (rate(http_requests_total{status=~"5.."}[5m])) / clamp_min(sum by (job) (rate(http_requests_total[5m])), 1e-9)` | ratio |
| `cpu_saturation` | `sum by (name) (rate(container_cpu_usage_seconds_total[2m]))` | cores |
| `memory_rss` | `sum by (name) (container_memory_rss)` | bytes |
| `mqhub_publish_rate` | `sum by (topic) (rate(mqhub_publish_total[1m]))` | msg/s |
| `mqhub_redis` | `mqhub_redis_connection_status` | bool |
| `recap_db_pool_in_use` | `sum by (pool) (recap_db_pool_checked_out)` | conns |
| `recap_worker_rss` | `recap_worker_rss_bytes` | bytes |
| `recap_request_p95` | `histogram_quantile(0.95, sum by (le) (rate(recap_request_process_seconds_bucket[5m])))` | seconds |
| `recap_subworker_admin_success` | `sum by (status) (rate(recap_subworker_admin_job_status_total[5m]))` | jobs/s |

### Alt-Paper 配色

Alt-Paper テーマブロック内で以下を上書きする。純黄 `#ffff00` の cream 背景でのコントラスト比 1.07:1 は WCAG 失格で editorial 配色としても不適切と判断した。

| token | HEX | cream 背景での contrast |
|---|---|---|
| `--alt-success` | `#2f6b3a` | 5.73:1 |
| `--alt-warning` | `#8a5a00` | 6.12:1 |
| `--alt-error` | `#8c1d1d` | 8.29:1 |

Observability パネル配下の `--obs-*` トークンは Tufte の sparkline 原則、FT Visual Vocabulary、Datadog Top List + Timeseries、WCAG 2.2 SC 1.4.11 / 1.4.1、Okabe–Ito パレットなど一次資料に基づく。色のみで状態を伝えず `▲ ▼ ● ○` などの glyph とテキストを常に併用する。

### TDD

各ファイルは RED → GREEN → REFACTOR を別コミットで踏んだ。alt-backend 側は httptest Server を fake Prom として使用する driver test / gateway test、port mock を用いる usecase test、Connect client で Unary + Streaming を検証する handler test を整備した。FE 側は Vitest browser (`vitest-browser-svelte`) で `MetricRow` / `MetricSeriesDigest` / `util` を TDD で実装した。

### 検証結果

- Go 全 131 パッケージが test pass
- alt-frontend-sv: `svelte-check --tsgo` 0 errors、browser 9 tests pass
- 全コンテナ (`alt-backend` / `alt-butterfly-facade` / `alt-frontend-sv` / `prometheus`) Healthy
- E2E Catalog: 12 キーが allowlist に登録
- E2E Snapshot: `availability_services` = 8 jobs、`memory_rss` = 60 series、`mqhub_redis` / `mqhub_publish_rate` / `recap_worker_rss` がすべて `degraded: false` で実データ返却

## Consequences

### Pros

- **editorial 一貫性**: Alt-Paper の活字基調で読みやすい。ゲージ・速度計・ドーナツ等のメタファーを持ち込まず、タイポグラフィと罫線で階層を示す。
- **即時視認性**: 5 秒間隔 push とコンパクトな行で異常の初動が早い。hover で Grafana にドリル可能。
- **セキュア**: allowlist 方式により SSRF / カーディナリティ爆発 / 高コストクエリを構造的に防止。admin role は BFF と alt-backend で二重ゲート。
- **リソース制御**: singleflight + 10s cache + 5rps レート制限で Prometheus 負荷は N クライアント → 1 クエリ / 5s に収束。Prom 側 `--query.max-samples` / `--query.max-concurrency` でワーストケースを抑制。
- **色覚対応**: Okabe–Ito パレットと glyph 併記により色覚多様性下でも状態識別が可能。
- **再接続耐性**: exponential backoff + full jitter と 15 分ローテートにより、中間プロキシ idle kill からの自己回復を保証する。
- **関心分離**: `--obs-*` スコープトークンでパネル専用のセマンティクスを閉じ込めたため、他テーマ (vaporwave, liquid-beige) の配色には影響しない。

### Cons / Tradeoffs

- **allowlist 維持コスト**: メトリクス追加のたびに `allowlist.go` と `DEFAULT_KEYS` と runbook の三箇所を同期する必要がある。Prometheus 側で新メトリクスが公開されても UI が自動追従はしない。
- **実態追従の課題**: 2026-04-13 時点で `alt-backend` / `pre-processor` の scrape が down 状態。UI は "down" を oxblood で表示するが、根治 (scrape 経路の認可・endpoint の整備) は別タスクとして切り出している。
- **Connect-Web 再接続の明示実装**: `@connectrpc/connect-web` は自動再接続を提供しないため、フック側で backoff / rotate を自前管理する。保守時にはこの契約を崩さないよう注意が必要。
- **BFF の streaming 経路分岐**: `AdminMonitorProxyHandler` は既存 `AdminProxyHandler` と似ているが context timeout を持たず flush ループを備える別実装。混同して旧ハンドラを使うと Watch が短時間で打ち切られるリスクがある。
- **global `--alt-warning` 直接参照の残存**: `SystemMonitorTab.svelte` が `--alt-warning` を参照している。Alt-Paper テーマ内で editorial ink に上書きしたため同画面の警告表現も変わる。目視確認は完了したが、他 UI の微調整は別タスクで追う余地がある。
