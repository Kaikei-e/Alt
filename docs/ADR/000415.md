# ADR-000415: Knowledge Home Admin の backfill 操作を SvelteKit Admin API 経由に統一する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-frontend-sv (SvelteKit) — admin route, server helper, Connect client
- alt-backend (Go) — KnowledgeHomeAdminService を操作対象として前提化

## Context

Knowledge Home Admin 画面には projection health と feature flags の表示は存在していたが、backfill job を起動・停止・再開する UI 操作は接続されていなかった。Connect client には `triggerBackfill(...)` がある一方で、SvelteKit の Admin API は `GET /api/admin/knowledge-home` しか持っておらず、ブラウザから mutation を実行する入口がなかった。

その結果、backfill job の実体が backend 側で動作可能になっても、管理者は画面から job lifecycle を制御できず、Knowledge Home を既存記事で埋める運用が成立しなかった。

## Decision

**Knowledge Home Admin の backfill 操作は、ブラウザから直接 Connect-RPC を叩かず、SvelteKit の `/api/admin/knowledge-home` を単一の Admin mutation 窓口として経由させる。**

### 変更内容

1. `src/lib/server/knowledge-home-admin.ts` に `trigger/pause/resume` 用の server helper を追加した。
2. `src/routes/api/admin/knowledge-home/+server.ts` に `POST` を追加し、`trigger`, `pause`, `resume` の各 action を受け付けるようにした。
3. `src/lib/connect/knowledge_home_admin.ts` で `pauseBackfill(...)` と `resumeBackfill(...)` を公開し、backend の `KnowledgeHomeAdminService` RPC と一対一対応にした。
4. browser 側は JSON action を SvelteKit API に送るだけにし、backend token の取り扱いは server 側に閉じ込めた。

### 設計判断

1. **admin 認可を route で集約する** — `locals.user` と `locals.backendToken` の検証を server route で一元化し、管理者権限の境界を browser 側に漏らさない。
2. **snapshot API と mutation API を同一 endpoint に揃える** — admin 画面のデータ取得と操作を `/api/admin/knowledge-home` に集約し、画面実装の mental model を簡潔に保つ。
3. **BFF token は browser に渡さない** — backend token を client-side Connect transport に露出せず、SvelteKit server helper が BFF への接続を代行する。

## Consequences

### Pros

- **運用経路が画面から完結する** — admin は CLI や DB 直接操作なしで backfill を制御できる
- **認可境界が明確** — admin 判定と backend token 利用が server route に閉じる
- **将来の admin action を足しやすい** — 同じ route に action を追加して拡張できる

### Cons / Tradeoffs

- **route 分岐が増える** — `POST` body の action 判定ロジックを維持する必要がある
- **browser からは RPC の詳細が見えにくい** — client は JSON API を叩くため、Connect-RPC の直接利用より抽象化が一段増える
