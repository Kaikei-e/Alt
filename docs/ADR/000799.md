# ADR-000799: Alt 側 compose.staging.yaml の GHCR image tag を service 別にし dispatch SHA 不一致の pull 失敗を塞ぐ

## Status

Accepted

## Date

2026-04-19

## Affected Services

- **compose (staging)** — `compose/compose.staging.yaml` の GHCR image タグ 10 箇所を per-service 変数に変更
- **e2e runners** — `e2e/hurl/<service>/run.sh` 9 本に focal service の tag export を追加

## Context

2026-04-19 の deploy (`release-deploy` run `24630595052` dispatch SHA `9dc1210`) で e2e (acolyte-orchestrator) が以下のエラーで落ちた:

```
Pulling ghcr.io/kaikei-e/alt-search-indexer:sha-9dc1210
Error manifest unknown
```

release-deploy は **変更された service だけをビルドする** 最適化を持つ。`9dc1210` は `alt-backend/mocks/` + `acolyte-orchestrator/evaluation/results/` のみを変更したため、alt-backend と acolyte-orchestrator しかビルドされていない。一方 `compose/compose.staging.yaml` は全 GHCR service の image タグを `${IMAGE_TAG:-main}` に固定しており、dispatch SHA (`sha-9dc1210`) を全 dependency に適用しようとする。未ビルドの `alt-search-indexer:sha-9dc1210` は GHCR に存在しないので pull が失敗し、e2e スライスが起動しない。

acolyte-orchestrator スイートは e2e の中でも widest dependency (search-indexer / news-creator-stub / stub-backend / meilisearch) を立ち上げるため、このクラスの不整合に一番刺さりやすい。tag-generator / mq-hub / news-creator など狭い依存構成のスイートでは偶然素通りしていた。

運用制約:

- 本番反映は `git push origin main` のみ ([[000740]])、release-deploy 側の build matrix は Alt 側で制御できない (alt-deploy リポに所属)
- compose.staging.yaml は Alt 側にあり、Alt だけで変更可能
- 既存の `IMAGE_TAG` を全面的に廃止すると search-indexer / alt-backend 自体の e2e (自 service が focal) で dispatch SHA を指せなくなる

## Decision

compose.staging.yaml の GHCR image 行を **focal service と dep service で分ける**:

- 各 GHCR service は自前の env 変数 `<SERVICE>_IMAGE_TAG` を持つ (既定 `main`)
- focal service の run.sh が `<FOCAL>_IMAGE_TAG=$IMAGE_TAG` を export して dispatch SHA を forward
- dep service は未 export なら `main` fallback で GHCR の main 最終ビルドを引く

### 置換箇所 (10 行、compose/compose.staging.yaml)

| service | before | after |
|---|---|---|
| search-indexer | `${IMAGE_TAG:-main}` | `${SEARCH_INDEXER_IMAGE_TAG:-main}` |
| mq-hub | `${IMAGE_TAG:-main}` | `${MQ_HUB_IMAGE_TAG:-main}` |
| rag-orchestrator | `${IMAGE_TAG:-main}` | `${RAG_ORCHESTRATOR_IMAGE_TAG:-main}` |
| tag-generator | `${IMAGE_TAG:-main}` | `${TAG_GENERATOR_IMAGE_TAG:-main}` |
| news-creator | `${IMAGE_TAG:-main}` | `${NEWS_CREATOR_IMAGE_TAG:-main}` |
| recap-pipeline-stub | `${IMAGE_TAG:-main}` | `${RECAP_PIPELINE_STUB_IMAGE_TAG:-main}` |
| recap-worker | `${IMAGE_TAG:-main}` | `${RECAP_WORKER_IMAGE_TAG:-main}` |
| alt-backend-deps-stub | `${IMAGE_TAG:-main}` | `${ALT_BACKEND_DEPS_STUB_IMAGE_TAG:-main}` |
| alt-backend | `${IMAGE_TAG:-main}` | `${ALT_BACKEND_IMAGE_TAG:-main}` |
| auth-hub | `${IMAGE_TAG:-main}` | `${AUTH_HUB_IMAGE_TAG:-main}` |

### run.sh での focal 解決

全 9 本の `e2e/hurl/<service>/run.sh` に以下のパターンを挿入:

```bash
: "${<SERVICE>_IMAGE_TAG:=$IMAGE_TAG}"
export IMAGE_TAG GHCR_OWNER STAGING_PROJECT_NAME <SERVICE>_IMAGE_TAG
```

例外:

- `alt-backend/run.sh` は `ALT_BACKEND_IMAGE_TAG` に加えて `ALT_BACKEND_DEPS_STUB_IMAGE_TAG` も同 SHA で export する (co-versioned)
- `recap-worker/run.sh` は `RECAP_WORKER_IMAGE_TAG` と `RECAP_PIPELINE_STUB_IMAGE_TAG` を同 SHA で export する (pipeline-stub は recap-worker と同じビルド列)
- `acolyte-orchestrator/run.sh` は focal が locally-built (`alt-staging/acolyte-orchestrator:local`) のため per-service tag 設定は不要。全 dep が `main` にフォールバックして OK

### 検討した代替案

| 案 | 採用 | 却下理由 |
|---|---|---|
| **per-service tag + focal override** (採用) | ○ | Alt 側で完結、release-deploy 側を触らず解決。focal e2e は従来通り dispatch SHA を指せる |
| `alt-deploy` の build matrix を「e2e に必要な全 dep を build」に拡張 | × | alt-deploy リポの workflow 大改修が必要、スコープが広い |
| 空 commit で search-indexer を強制リビルドさせ次回 dispatch でタグを揃える | × | 根本的な対症療法、将来も同じ症状が繰り返す |
| 全 GHCR service をローカルビルドに統一 | × | CI 時間の肥大化、GHCR ビルドのメリット喪失 |
| `${<SERVICE>_IMAGE_TAG:-${IMAGE_TAG:-main}}` の nested fallback | × | Docker Compose v2 の expansion 互換性が不安、可読性低下 |

## Consequences

### Pros

- dispatch が 1 service だけをリビルドしても、それ以外の dep は `main` の最終成功ビルドを pull してくる。manifest unknown エラーは再発しない
- focal service の e2e は従来通り dispatch SHA でテストできる (run.sh が per-service tag を明示的に渡すため)
- 変更は compose + shell スクリプトのみ、CI 側 (alt-deploy) の改修不要

### Cons / Tradeoffs

- **run.sh の追加行**が 9 ファイル × ~6 行 = 54 行の薄い重複。共通関数 (`e2e/hurl/_lib/`) に括り出してもよいが、今回は可読性優先で各 run.sh にインライン
- **dep service の契約ずれ**を検知できなくなるリスクあり: 例えば search-indexer が `main` でまだマージされていない新 API を期待すると、古い search-indexer image でテストが通って誤検知する。Pact CDC が別軸でカバー
- **co-versioned stubs** (alt-backend-deps-stub / recap-pipeline-stub) は focal service と同 SHA を前提。将来 build を分離すると個別の env 配線が必要
