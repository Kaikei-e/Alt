# ADR-000756: CI 失敗 5 本を gofmt/ruff 準拠と Dockerfile 数値 UID で一括復旧する

## Status

Accepted

## Date

2026-04-17

## Affected Services

- alt-backend (Go) — `driver/contract/provider_test.go` の const ブロック整列
- pre-processor (Go) — `driver/contract/provider_test.go` の doc コメント段落セパレータ追加
- tag-generator (Python) — `tests/contract/test_provider_verification.py` の magic-trailing-comma 展開
- news-creator (Python) — `tests/contract/test_provider_verification.py` の magic-trailing-comma 展開
- alt-frontend-sv (SvelteKit / Bun) — `Dockerfile` runner ステージの 3 本の `COPY --link --chown`

## Context

commit `5938888` (feat(alt-frontend-sv): add desktop editorial rail to FeedDetail view) と `8531c35` (style: normalize biome formatting) が main に積まれた直後、同じ push で 5 本の CI workflow が同時に赤になった。

```
24518808055  Pre-processor Quality Gates   → gofmt diff
24518808031  Backend Go Tests              → gofmt diff
24518808025  Tag Generator CI              → ruff format diff + test_ruff_compliance 落ち
24518807984  News-Creator Quality Gates    → ruff format diff
24518808037  Build and Push Docker Images  → buildx: invalid user index: -1
```

4 件はフォーマッタ未適用によるスタイル失敗で、1 件は BuildKit 固有の user 解決失敗である。前者 4 件は直前の `5e502cf6 test(contract): expand Pact provider verification coverage to 10 services` が生成した新しい contract test を `gofmt` / `ruff format` を通さずに push したことに起因する。Go 側は `altButterflyFacadePactFile` が新しい最長定数名となり既存 const ブロックの `=` 列整列が崩れた点と、news_creator の doc コメントで箇条書きリストと散文段落の間に空コメント行 (`//`) が欠けている点の二箇所で gofmt が diff を出していた。Python 側は magic-trailing-comma 規則 (ruff の既定) により、末尾カンマ付きの `builder.consumer_version(consumer=..., main_branch=True,)` をキーワードごとに改行する必要があった。

後者の Dockerfile 失敗は BuildKit 1.7 の既知の挙動で、`COPY --link --chown=<name>:<name>` が独立レイヤを構築する際に passwd を解決できず `ERROR: failed to solve: invalid user index: -1` で落ちる。runner ステージは以下の順で書かれていた:

```dockerfile
RUN addgroup -g 1001 -S sveltekit && \
    adduser -u 1001 -S -G sveltekit sveltekit

COPY --link --from=builder --chown=sveltekit:sveltekit /app/build ./build
COPY --link --from=install --chown=sveltekit:sveltekit /temp/prod/node_modules ./node_modules
COPY --link --chown=sveltekit:sveltekit package.json ./
```

`--link` 付き COPY はソースステージの filesystem を引き継がず空の新規レイヤとしてマージされるため、ユーザ名解決がそのレイヤ内 `/etc/passwd` に対して行われ、結果 `uid=-1` を返して失敗する。直前ステージの RUN で user を作成していても、`--link` を使う限りそのステージの filesystem は「ソース」ではなく「コピー先ターゲット」扱いになり、user 情報は参照されない。

全 5 件とも該当テストやロジックの修正は不要で、純粋にフォーマッタと BuildKit 仕様への適合のみで直る内容だった。

## Decision

以下 5 ファイル、計 16 行の最小修正を 1 commit にまとめて main に入れ、CI を緑に戻す:

1. `alt-backend/app/driver/contract/provider_test.go` — `gofmt -w` で const 列整列を修正
2. `pre-processor/app/driver/contract/provider_test.go` — `gofmt -w` で doc コメントに空コメント行を挿入
3. `tag-generator/app/tests/contract/test_provider_verification.py` — `uv run ruff format` で kwargs を改行展開
4. `news-creator/app/tests/contract/test_provider_verification.py` — 同上
5. `alt-frontend-sv/Dockerfile` — runner ステージの 3 本の `--chown=sveltekit:sveltekit` を数値 `--chown=1001:1001` に変更

Dockerfile の修正方針として、次の 3 案を比較した:

| 案 | 採否 | 理由 |
|---|---|---|
| `--link` を削除する | 却下 | `--link` はレイヤ再利用によるプル時間短縮に寄与しており、ビルドキャッシュのヒット率が目に見えて下がる |
| addgroup/adduser をベースイメージ側に移して順序を変える | 却下 | runner ステージでユーザを作る現行設計を維持した方が base と runner の役割分担が明確。順序変更は `--link` の本質（宛先ステージの現在 filesystem を無視する）を解決できず根治にならない |
| 数値 UID:GID (`1001:1001`) にする (**採用**) | 採用 | BuildKit は数値を受け取ると passwd 解決をそもそも行わないためこの失敗モードを完全に回避できる。Docker 公式ドキュメントが推奨する workaround そのものであり、runtime `USER sveltekit` は同じ uid 1001 に解決されるため挙動は変わらない |

Go/Python のフォーマットは既存の CI ゲート (`gofmt -l` / `ruff format --check`) で担保されているため、追加のリンタ導入は行わない。pre-commit フック化は本 ADR のスコープ外とし、別途ハードニングで検討する。

## Consequences

### Pros

- CI 5 本 (Pre-processor Quality Gates / Backend Go Tests / Tag Generator CI / News-Creator Quality Gates / Build and Push Docker Images) が全て緑に戻り、main ブランチからの Pact ゲート付きデプロイパスが再開できる。
- alt-frontend-sv image は `docker buildx build --load` + `docker compose up --build -d alt-frontend-sv` の両方で成功し、コンテナ内の `id` は引き続き `uid=1001(sveltekit) gid=1001(sveltekit)` を返す。runtime のユーザ権限モデルは変わらない。
- 変更は 16 行・5 ファイルに閉じており、Pact contract や business logic には一切触れていないため、追加のリグレッションリスクは低い。
- tag-generator 側の `tests/unit/test_ruff_compliance.py::test_ruff_format_check_passes` (リポジトリ内で ruff format --check を subprocess で実行する unit test) も副次的にパスするようになり、lint と test の両方が同じ 1 commit で復旧する。

### Cons / Tradeoffs

- Dockerfile で chown 先が数値 `1001:1001` になったため、将来 UID を変えたい場合は RUN の `addgroup/adduser` と 3 本の COPY を一緒に更新しなければならず、変更が 4 箇所に分散する。ただし現状 1001 を変更する計画はなく、`Dockerfile` 1 ファイル内で完結しているため許容する。
- フォーマッタ未適用は commit 前のローカル `gofmt -l` / `ruff format --check` もしくは pre-commit フックを入れれば根治できるが、本 ADR ではスコープを CI 復旧に絞り、フック導入は別ワークとして積み残す。
- Node.js 20 ベースの GitHub Actions (setup-go@v5, cache@v4, setup-uv@v6, docker/*@v3-v6) が 2026-06-02 に強制 Node.js 24 化される deprecation warning が 5 本全てに出ていたが、現時点では失敗要因ではないため本 ADR では触らず、別途アップグレード ADR を切る想定とする。
