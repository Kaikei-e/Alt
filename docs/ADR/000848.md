# ADR-000848: Knowledge Loop の dismiss を KnowledgeLoopDeferred 経路に分離して projection に永続化し、tile に OODA 段階別 Z 奥行きを与える

## Status

Accepted

## Date

2026-04-26

## Affected Services

- alt-frontend-sv (SvelteKit) — `useKnowledgeLoop.dismiss` の経路書き換え、`/loop/transition` BFF の DEFER 受理、`LoopEntryTile.svelte` の Z mapping、`OodaPipeline.svelte` 新規追加
- alt-backend (Go) — `ClassifyTransitionEvent` の DEFER 分岐
- knowledge-sovereign (Go) — projector の Deferred 経路と driver の `PatchKnowledgeLoopEntryDismissState`
- proto — `alt/knowledge/loop/v1/knowledge_loop.proto` の `TransitionTrigger` enum に `TRANSITION_TRIGGER_DEFER = 5` を追加

## Context

`/loop` UI には 2 つの実害が積もっていた。

### Bug 1 — Dismiss クリックがリロードで復活

ユーザ操作: `Dismiss` を押しても、ページをリロードすると同じ tile が戻ってくる。コンテナのログ (nginx) を見ると `POST /loop/transition` が走っているケース・走っていないケースが混在しており、走っていないケースは早期 return。コードトレースで 3 層のバグが直列していたことが分かった。

1. **FE 早期 return**: `useKnowledgeLoop.dismiss()` は `from = entry.proposedStage; to = canTransition(from, "act") ? "act" : from; if (to === from) return;` で `decide → act` を強制していた。`canTransition(from, "act")` が `true` になるのは `from === "decide"` のみ。`observe` / `orient` / `act` の entry では `to === from` で **backend に何も送らない**。`optimisticallyDismissed` Set はブラウザメモリに閉じているため、リロードで復活する。
2. **Sovereign projector が DismissState を更新しない**: `EventKnowledgeLoopActed` / `EventKnowledgeLoopDeferred` は `projectTransition()` で session_state のみ書き、entry 行の `dismiss_state` は `ACTIVE` のまま。read 側のフィルタ `AND dismiss_state = 'active'` で素通り。
3. **Backend classifier に Deferred への経路が無かった**: `ClassifyTransitionEvent(from, to, trigger)` は OODA 4 stage 矩陣のみを扱い、`KnowledgeLoopDeferred` を emit する条件分岐が存在しなかった。FE が同 stage を投げても classifier は "unrecognized stage tuple" で 400 を返すだけ。

[[000844]] の Open Question にも残っていた: 「`HomeItemDismissed` と `KnowledgeLoopDeferred` の差異がサーバ側で同経路に潰されている可能性、別途 transition event を分けるかの検討余地あり」。本 ADR でこれを解消する。

### Bug 2 — OODA ループに 3D の奥行きが無い

canonical contract §12 (`docs/plan/knowledge-loop-canonical-contract.md`) は「Depth lives on tiles, not glyphs」「deeper focus: 奥へ入る / return: 手前に戻る」を要求するが、実装は次の状態だった:

- `LoopSurfacePlane` は plane 単位 (foreground / mid-context / deep-focus) で `translateZ(0/-24/-48px)` を持つ。
- ただし foreground 内の tile は `proposedStage` に関係なく **全部 Z=0**。`renderDepthHint` (saturation/border 強度) は `surfaceBucket` 由来で OODA stage を反映しない。
- masthead の OODA kicker は `Observe · Orient · Decide · Act` のフラットな水平テキスト。循環 (Act → Observe) を表現する視覚要素が無い。

結果として、ユーザは「いま自分が OODA のどこに居るか」「次にどこに進むのか」を空間的に読めない。

### Web 調査要旨 (2026-04)

- `perspective` + `transform-style: preserve-3d` は GPU compositing で安定戦略 ([MDN — transform-style](https://developer.mozilla.org/en-US/docs/Web/CSS/transform-style))。子要素には `preserve-3d` を継承させないと flat に落ちる。
- View Transitions API は SvelteKit でも使えるが用途は **ページ遷移**。`/loop` 内の連続 depth 変化は CSS transform + Svelte 5 transition で十分で、`document.startViewTransition` を要さない ([Svelte blog — view transitions](https://svelte.dev/blog/view-transitions))。
- `prefers-reduced-motion: reduce` 対応は 2026 必須。Z 動きを停止し dissolve + saturation のみに置換 — canonical contract §12.5 と一致 ([dgrebb — accessible transitions](https://www.dgrebb.com/post/animating-page-and-view-transitions-with-accessibility-in-mind/))。

## Decision

### 1. Dismiss を `KnowledgeLoopDeferred` 専用経路に切替

#### 1.1 proto: `TRANSITION_TRIGGER_DEFER = 5` を additive 追加

`proto/alt/knowledge/loop/v1/knowledge_loop.proto` の `TransitionTrigger` enum に `TRANSITION_TRIGGER_DEFER = 5` を末尾追加。`buf breaking` pass。Go / TS の生成 stub は `cd proto && buf generate` で再生成。

#### 1.2 FE hook: 全 stage で同 stage + DEFER を POST

`alt-frontend-sv/src/lib/hooks/useKnowledgeLoop.svelte.ts` の `dismiss()` から `decide → act` 強制を削除し、

```ts
await post({
  clientTransitionId: uuidv7(),
  entryKey,
  fromStage: stage,
  toStage: stage,
  trigger: "defer",
});
```

に置換。`applyLocalDismiss` の optimistic 削除は据え置き。BFF (`+server.ts`) は `trigger === "defer"` のとき `fromStage === toStage` を許容し、それ以外は従来通り `canTransition()` で OODA 矩陣をチェックする。

#### 1.3 Backend classifier: DEFER + same-stage で Deferred を emit

`alt-backend/app/usecase/knowledge_loop_usecase/classify_transition_event.go` の先頭で短絡:

```go
if trigger == triggerDefer {
    if !isCanonicalStage(fromStage) || fromStage != toStage {
        return "", fmt.Errorf("%w: defer trigger requires from_stage == to_stage and a canonical OODA stage", ErrInvalidArgument)
    }
    return domain.EventKnowledgeLoopDeferred, nil
}
```

`UNSPECIFIED` や非 OODA stage は引き続き 400。

#### 1.4 Sovereign projector: Deferred で entry の `dismiss_state` を patch

`knowledge-sovereign/app/usecase/knowledge_loop_projector/projector.go` の `projectTransition()` で session_state を upsert した直後、`ev.EventType == EventKnowledgeLoopDeferred && entryKey != ""` のときに **新 driver メソッド** `PatchKnowledgeLoopEntryDismissState(...)` を呼んで `dismiss_state = 'deferred'` を反映する。他の transition event (Observed/Oriented/Acted/Returned/SessionReset/LensModeSwitched) は entry を触らない。

#### 1.5 Driver: 単一カラム UPDATE + seq guard

`knowledge-sovereign/app/driver/sovereign_db/write_knowledge_loop.go` に下記を追加:

```sql
UPDATE knowledge_loop_entries SET
  projection_revision    = projection_revision + 1,
  projection_seq_hiwater = GREATEST(projection_seq_hiwater, $5),
  source_event_seq       = GREATEST(source_event_seq, $5),
  projected_at           = NOW(),
  dismiss_state          = $6
WHERE user_id = $1 AND tenant_id = $2 AND lens_mode_id = $3 AND entry_key = $4
  AND projection_seq_hiwater <= $5
RETURNING projection_revision, projection_seq_hiwater
```

[[000846]] の `PatchKnowledgeLoopEntryWhy` と同じ思想で:

- 単一 SET に `dismiss_state` だけ。`why_*` / `freshness_at` / `surface_bucket` 等は SQL 文字列に**現れない** (driver の structural test で固定)。
- `WHERE projection_seq_hiwater <= $5` + `SET ... GREATEST(..., $5)` で out-of-order delivery / replay は idempotent。
- `pgx.ErrNoRows` (entry 未存在 OR 古い seq) は `SkippedBySeqHiwater=true` で no-op。

### 2. OODA tile depth と masthead 円弧化

#### 2.1 `LoopEntryTile.svelte`: `[data-stage]` を Z にマッピング

```css
.entry { transition: filter 180ms, border-color 180ms, transform 240ms cubic-bezier(.2,0,.1,1); }
.entry[data-stage="observe"] { transform: translateZ(0); }
.entry[data-stage="orient"]  { transform: translateZ(-12px); }
.entry[data-stage="decide"]  { transform: translateZ(-24px); }
.entry[data-stage="act"]     { transform: translateZ(-36px); }

@media (prefers-reduced-motion: reduce) {
  .entry[data-stage] { transform: none; }
  .entry[data-stage="orient"] { filter: saturate(.96); }
  .entry[data-stage="decide"] { filter: saturate(.92); }
  .entry[data-stage="act"]    { filter: saturate(.88); }
}
```

すべて foreground plane の `perspective: 1200px` 配下で動く (`loop-depth.css`)。Alt-Paper の "no shadow" を維持。Reduced Motion では Z を潰し saturation step だけで階層化 (canonical contract §12.5 dissolve + highlight fade + color shift)。

#### 2.2 `OodaPipeline.svelte` 新規

旧 `.kicker-row` (フラット水平) を置換。各 kicker が `data-depth=(stage_order - currentStage_order + 4) % 4` を持ち:

| depth | translateZ | filter | role |
|---|---|---|---|
| 0 | 0 | saturate(1.05) | active |
| 1 | -14px | saturate(.95) | next |
| 2 | -26px | saturate(.88) | far |
| 3 | -38px | saturate(.80) | wrap predecessor |

`Act` の後に `↻` (wrap arrow) を置き、Observe→Orient→Decide→Act→**Observe** の循環を視覚化。Reduced Motion で perspective を `none` に潰し flat row に降格。

### 検討した代替案と却下理由

1. **proto を変えず、既存 trigger を流用 (例: `PROGRAMMATIC` を defer に転用)**
   - Defer 専用ではないため意味論が混在し、ログ・メトリクス・将来の rate limit 分割で必ず混乱する。Trigger は意図情報の primary 経路なのでオーバーロード不可。却下。
2. **`Dismissed` (hard) で `dismiss_state='dismissed'` に固定**
   - canonical contract §8.2 は `later / snooze / skip / passive dismiss` を `KnowledgeLoopDeferred` に集約するよう規定。ハード削除は意味論的に過剰で、後続の Review surface への戻し経路を閉じてしまう。ユーザ確認の上 `deferred` を採用。
3. **projector で `EventKnowledgeLoopActed` payload に `dismiss_intent` を持たせて分岐**
   - Act の意味 (decide→act の段階遷移) と dismiss が混ざる。Single emission rule ([[000831]] §3.7) 違反になりやすい。Defer は独立 event 型を持つべき。却下。
4. **patch SQL を full UPSERT で代替**
   - 既存 entry の `freshness_at` / `why_text` / `decision_options` などを sparse な Defer payload で上書きしてしまう。canonical contract §3 不変条件違反。却下。
5. **tile depth を box-shadow で表現**
   - Alt-Paper 規約 ([[000840]]) で drop-shadow 禁止。translateZ + saturation で代替。
6. **OODA progression を View Transitions API で表現**
   - `/loop` 内の連続変化はページ遷移ではないため、API のスコープ外。CSS transform + Svelte transition で十分。

## Consequences

### Pros

- **Dismiss が確実に永続化される**: observe / orient / decide / act の全 stage で `KnowledgeLoopDeferred` event が必ず append され、projector が `dismiss_state='deferred'` に flip するため、リロード後に復活しない。
- **Single emission rule を維持**: `/loop` UI の dismiss は `KnowledgeLoopDeferred` のみ emit し、`HomeItemDismissed` は `/feeds` UI 専用のまま。混線しない。
- **Reproject 安全**: 新 patch path は event payload (`entry_key`, `lens_mode_id`, `event_seq`, `occurred_at`) のみに依存。`time.Now()` は `projected_at` (debug 専用) のみ。replay すれば同じ DismissState=DEFERRED に収束。
- **Merge-safe**: 単一カラム UPDATE + `projection_seq_hiwater <= $5` + `GREATEST(seq_hiwater, $5)` で out-of-order delivery / replay は冪等。SQL に `dismiss_state` 単独しか現れないことを driver structural test で固定し、将来の改修で why/freshness を巻き込む regression を防ぐ。
- **Proto 変更が additive**: `TransitionTrigger` enum の末尾に `5` を追加するだけ。`buf breaking` pass、既存 client は 0/1/2/3/4 を引き続き使える。
- **OODA に空間的継続性が出る**: tile が `proposedStage` に応じて Z 軸 0/-12/-24/-36px に並び、masthead は perspective ribbon + ↻ で循環を視覚化。`prefers-reduced-motion` で全部 flat に潰れる。
- **Alt-Paper 規約を守る**: drop shadow 不使用。glyph は flat、tile / plane / kicker のみが奥行きを持つ ("Depth lives on tiles, not glyphs")。
- **既存ユーザ操作を破壊しない**: `transitionTo()` の OODA 矩陣・dwell / observe throttle・Open / Save / Ask / Compare / Revisit CTA・既存の `loop-recede` exit transition は無変更。

### Cons / Tradeoffs

- **`Deferred` (soft) のみで `Dismissed` (hard) は未対応**: 現状 dismiss_state=deferred の entry は read 側 `dismiss_state='active'` フィルタで foreground / bucket から外れるが、Review surface への戻し UI は別途実装が必要。"Soft dismiss" の戻し口を将来開けるなら別 PR で。
- **Active → Deferred への一方向のみ**: defer された entry を再 active にする経路 (例: ユーザが "Restore" を押す) は未提供。canonical contract §8 にもまだ明示されておらず、必要なら別 ADR で。
- **既存 `EventKnowledgeLoopDeferred` を直接 inject していたテスト・ツールがあれば、新たに entry の `dismiss_state` 副作用が入る**: `projectTransition` は session_state upsert のあとに patch を呼ぶので、テスト用に Deferred event を投入する経路は patch も走ることを想定する必要がある。
- **proto enum 追加で全 consumer の再 build が要る**: `cd proto && buf generate` を CI で必ず流すこと。alt-backend / alt-frontend-sv 以外で `TransitionTrigger` を import している consumer は無い (`grep -rl 'TransitionTrigger' --include='*.go' --include='*.ts'` で確認済) が、proto の他バージョンを参照する将来の service が出たら同じ regen が必要。
- **OODA tile Z (-12/-24/-36px) は plane Z (-24/-48px) と座標系が連続している**: 値域は確定しているがデザイントークン化はしていない。テーマやアクセシビリティ調整で値を一括スケールしたい場合、CSS custom properties に切り出す改修が要る。
