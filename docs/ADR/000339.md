# OG画像プロキシ キャッシュ改善 — Nginx proxy_cache導入・タイムアウト短縮・TTL延長

## ADR's STATUS

Accepted

## CONTEXT

Nginxスローレスポンス調査 (2026-03-07) により、`GET /v1/images/proxy/*` がP1ボトルネムとして特定された。

### 定量的な問題

| 指標 | 値 |
|---|---|
| 12時間の総リクエスト | 310件 |
| 1秒超のスローリクエスト | 80件 (25.8%) |
| P50レイテンシ (キャッシュヒット) | 2ms |
| P95レイテンシ (キャッシュミス) | 19.5s |
| 502 Bad Gateway | 14件 |

P50=2ms vs P95=19.5s という極端なバイモーダル分布が示すように、キャッシュヒット時は高速だがキャッシュミス時に外部画像フェッチのコストが直接ユーザーに露出していた。

### 改善前のアーキテクチャ

```
Client → Nginx (proxy_pass, キャッシュなし) → alt-backend (Go/Echo)
    → PostgreSQL image_proxy_cache テーブル (bytea列)
    → キャッシュミス時: 外部画像フェッチ → WebP変換+リサイズ → DBに保存 → 返却
```

### 根本原因

1. **Nginx層にproxy_cacheなし** — DBキャッシュヒットでも毎回Go→PostgreSQL往復が発生
2. **キャッシュスタンピード未対策** — 同一画像への同時リクエストが全てDBミス→外部フェッチ
3. **タイムアウト過大** — フェッチタイムアウト25s、Nginx read timeout 30sは画像プロキシとして長すぎる
4. **502時のフォールバックなし** — 外部サーバー不応答でブロークン画像アイコンが表示
5. **キャッシュTTLが短い** — OG画像はほぼ不変なのに24時間で期限切れ→再フェッチ頻発

## DECISION MAKING

3つの独立した改善を同時に実施する。いずれも後方互換性を保ち、段階的ロールバックが可能。

### 1. Nginx proxy_cache の導入

Nginx層にディスクベースの`proxy_cache`を追加し、Go/PostgreSQLへの到達を大幅に削減する。

**キャッシュゾーン設計:**

| パラメータ | 値 | 理由 |
|---|---|---|
| `keys_zone` | 256MB | 数万件のURLメタデータを保持可能 |
| `max_size` | 2GB | 画像は平均50-100KB、2万件以上格納可能 |
| `inactive` | 7日 | OG画像のアクセス頻度を考慮した妥当な期間 |
| `proxy_cache_valid 200` | 12時間 | 通常の更新サイクルに合わせた有効期間 |

**スタンピード防止:** `proxy_cache_lock on` により同一URLへの同時リクエストは1つだけupstreamに到達する。残りはロック解放まで待機し、キャッシュされたレスポンスを受け取る。

**耐障害性:** `proxy_cache_use_stale error timeout updating http_502 http_503 http_504` により、upstreamエラー時も古いキャッシュがあれば返却する。`proxy_cache_background_update on` と組み合わせ、ユーザーには古いキャッシュを即座に返しつつ裏で更新する。

**キャッシュキー:** `proxy_cache_key "$uri"` を使用。画像プロキシのURLにはHMAC署名とBase64エンコードされたURLが含まれ、同一画像は常に同一URIになるため、クエリパラメータやHostヘッダは不要。

**upstream Cache-Control無視:** `proxy_ignore_headers Cache-Control Expires Set-Cookie` により、backendが返す`Cache-Control`ヘッダを無視してNginx側でTTLを管理する。backendのヘッダはブラウザ向けであり、Nginxキャッシュの制御には不適切なため。

### 2. タイムアウト短縮 + フォールバック画像

**タイムアウト:** フェッチタイムアウトを25s→10sに短縮。OG画像で10秒以上かかるケースは取得不可と判断してよい。Nginx側のread timeoutも30s→15sに短縮し、連動させる。

**フォールバック:** タイムアウト・フェッチエラー・処理エラー時に、502/504を返す代わりに1x1透過PNGを`Cache-Control: public, max-age=60`で返す。

- ブラウザでブロークン画像アイコンが表示されなくなる
- 60秒のmax-ageにより短期間はリトライを抑制しつつ、すぐに再取得も可能
- Nginxの`proxy_cache_use_stale`と組み合わせ、正常時のキャッシュがあればそちらが優先される

**シグネチャエラー・ドメイン拒否は403を維持** — これらはセキュリティ上の拒否であり、フォールバックすべきではない。

### 3. キャッシュTTL延長 (24h → 7d)

OG画像はサイト公開後にほぼ変更されない。24時間TTLでは毎日全画像を再フェッチしており、外部サーバーへの不要な負荷とレイテンシスパイクの原因となっていた。

| 設定 | 変更前 | 変更後 |
|---|---|---|
| ドメイン定数 `ImageProxyCacheTTL` | 24時間 | 7日 |
| config デフォルト `IMAGE_PROXY_CACHE_TTL_MINUTES` | 1440分 | 10080分 |
| HTTPレスポンス `max-age` | 43200 (12h) | 604800 (7d) |

`immutable`ディレクティブは既に付与されているため、ブラウザは条件付きリクエスト（If-None-Match）すら送信しない。

## RESULTS, EFFECTS

### 改善後のアーキテクチャ

```
Client → Nginx
    ├─ HIT: ディスクキャッシュから直接返却 (< 1ms)
    ├─ STALE: 古いキャッシュを返却 + バックグラウンド更新
    └─ MISS (proxy_cache_lock で1リクエストのみ):
        → alt-backend → PostgreSQL キャッシュ確認
            ├─ DB HIT: 返却
            └─ DB MISS: 外部フェッチ (10sタイムアウト)
                ├─ 成功: WebP変換 → DB保存 → 返却
                └─ 失敗: 1x1透過PNGフォールバック (max-age=60)
```

3層キャッシュ構成: **ブラウザ (7日)** → **Nginx ディスク (12h有効 / 7日inactive)** → **PostgreSQL (7日TTL)**

### PROS

- **レイテンシ大幅改善**: Nginx HITでGo/DB到達ゼロ、レイテンシ < 1ms
- **502エラー激減**: `proxy_cache_use_stale`によりupstreamエラー時も古いキャッシュを返却
- **スタンピード防止**: `proxy_cache_lock`により同一画像の同時外部フェッチは最大1回
- **ブロークン画像消滅**: フォールバック透過PNGにより、画像取得失敗時もUIが崩れない
- **外部サーバー負荷軽減**: TTL延長により不要な再フェッチが7分の1に削減
- **段階的ロールバック可能**: 3つの変更は独立しており、個別にリバートできる

### CONS, TRADEOFF

- **Nginxキャッシュのディスク消費**: 最大2GBのディスクを消費する。ただし画像プロキシの特性上、数千件程度の実効キャッシュ量であり問題にならない
- **キャッシュの一貫性**: OG画像が変更された場合、最大7日間古い画像が表示される。OG画像の変更頻度は極めて低いため許容範囲。緊急時はNginxキャッシュのpurgeまたはボリューム削除で対応可能
- **フォールバックPNGの可視性**: 画像取得が恒常的に失敗する場合、1x1の透過画像が表示され続ける。ただしmax-age=60sのため60秒後にリトライされ、一時的な障害であれば自然回復する
- **Nginxが`proxy_ignore_headers`でupstreamのCache-Controlを無視**: backendが独自にキャッシュ制御を変更してもNginx層には反映されない。意図的なトレードオフだが、将来backendのキャッシュ戦略を変更する際はNginx設定も同時に更新する必要がある

### 検証方法

```bash
# X-Cache-Status ヘッダで Nginx キャッシュ動作を確認
# 1回目: MISS, 2回目: HIT
curl -s -D - http://localhost/v1/images/proxy/{sig}/{url} | grep X-Cache-Status

# キャッシュディレクトリの確認
docker compose -f compose/compose.yaml -p alt exec nginx ls -la /var/cache/nginx/image_proxy/

# 12時間後のスローレート改善確認
docker compose -f compose/compose.yaml -p alt logs nginx --since 12h 2>&1 \
  | grep 'images/proxy' | grep -oP 'rt=[0-9.]+' \
  | awk -F= '$2 > 1.0 {slow++} {total++} END {print slow"/"total}'
```
