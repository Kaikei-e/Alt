# ADR-000923: Recall rail を Home payload に統合し legacy RPC を削除する

## Status

Accepted

## Date

2026-05-25

## Affected Services

- alt-backend — `app/connect/v2/knowledge_home/home_tracking.go` の `TrackHomeAction` switch に `case "snooze":` / `case "dismiss_recall":` を追加し `recallSnoozeUsecase` / `recallDismissUsecase` に dispatch。`home_recall.go` 削除 (3 handler すべて)。`home_recall_deprecation_test.go` 削除。`home_query.go` の `FlagRecallRail` gate を外し recall candidates を Home payload に無条件で含める。`recall_rail_usecase.Execute` から feature flag check を削除。`domain.FlagRecallRail` / `config.EnableRecallRail` / `feature_flag_gateway` の switch case を削除。`knowledge_home_admin.GetFeatureFlags` から `EnableRecallRail` を削除 (proto field は wire compat で残)。
- alt-butterfly-facade — `internal/handler/proxy_handler.go` の `streamingProcedures` map から `StreamRecallRailUpdates` を削除。
- alt-frontend-sv — `src/lib/hooks/useRecallRail.svelte.ts` の `fetchCandidates` 削除、snooze / dismiss を `trackHomeAction("snooze" | "dismiss_recall", itemKey, { snooze_hours })` 経由に置換。`getRecallRailCandidates` / `snoozeRecallItem` / `dismissRecallItem` を `knowledge_home.ts` / `index.ts` から削除。`src/lib/components/knowledge-loop/recall-to-loop-adapter.ts` を新設 (pure fn + 5 spec)。`FeatureFlagPanel.svelte` の `Recall Rail` toggle 削除。E2E fixture の `enable_recall_rail` 削除し `FEATURE_FLAGS_RECALL_DISABLED` を base alias に縮退。
- proto — `proto/alt/knowledge_home/v1/knowledge_home.proto` の `KnowledgeHomeService` から `GetRecallRail` / `TrackRecallAction` / `StreamRecallRailUpdates` 3 RPC を削除。関連 message 型は file に残置 (proto 規約で同 file 内の独立 message に reserved の概念がない、未参照になっても害なし)。

## Context

ADR-000913 §D-9 (Heavy-Ranker explainable scoring) の出口として「recall rail は Home payload の一部に統合し、専用 RPC は廃止する」が canonical contract §6.4 にも明記されている。これは次の理由による:

1. **Resource / Event の分離 (canonical contract invariant 10)**: recall は「Home 上の per-user resource projection」であり、それ自体に dedicated RPC を立てるとリソースの単一情報源を破る。Home payload が source of truth、recall はその一部、snooze / dismiss は user action だから TrackHomeAction が absorb する、という結晶化された構造が ADR-000913 の目標。
2. **二重 fetch の解消**: FE が GetKnowledgeHome + GetRecallRail の 2 RPC を打つと race / consistency が壊れる (snooze 直後に Home 側 cache から古い候補が出る等)。Home payload の `recall_candidates` field を single source にすれば race が消える。
3. **Stream の二重化**: `StreamKnowledgeHomeUpdates` と `StreamRecallRailUpdates` の 2 stream を分けるとイベント順序が non-deterministic になる。recall_changed event は Home stream の 1 種として coalescing できる (実装は ADR-000912 の `buildStreamResponse` で既に landing 済)。
4. **Feature flag の semantic drift**: `FlagRecallRail` は当初「rail UI の opt-in」だったが、recall データ自体は Home payload に乗っているので「flag off でも Home に recall_candidates が出る → flag は何を gate しているのか」が説明できない状態だった。

PR 3 (deprecation log) は 2026-05-25 の Batch A で landed 済。**ユーザの明示指示** (「PR 13 も含めて全部やる」) で観測 window を skip し、PR 13 を本セッション内で landing する判断をした。リスクは:

- prod の外部クライアント (e.g. mobile cache、forgotten subscription) が legacy RPC を叩くと 404 / unimplemented になる。
- FE は本 batch で同時 landing するため自己整合は取れている (FE caller は 0)。
- `alt-butterfly-facade` も同時 deploy が要る (proxy allowlist 削除と service 削除が同期する必要がある)。

## Decision

次の 4 つを一括で landing する:

1. **TrackHomeAction が snooze / dismiss_recall を absorb**:
   - `home_tracking.go` の switch に `case "snooze":` (metadata_json の `snooze_hours` を parse、default 24h) と `case "dismiss_recall":` を追加。
   - 各 case で `recallSnoozeUsecase` / `recallDismissUsecase` を invoke。
   - usecase が nil の場合 `CodeUnimplemented` で fail-fast (legacy 運用設定検知)。
   - 3 test (`Snooze_UnimplementedWhenUsecaseMissing` / `DismissRecall_UnimplementedWhenUsecaseMissing` / `ValidatesItemKeyAndActionType`) で switch の確認。
2. **FE single-source 化**:
   - `useRecallRail.svelte.ts` の `fetchCandidates` を削除。candidates は `useKnowledgeHome` の payload を `setCandidates` で受ける。
   - snooze / dismiss を `trackHomeAction("snooze", itemKey, JSON.stringify({ snooze_hours: hours }))` / `trackHomeAction("dismiss_recall", itemKey)` に置換。optimistic removal は維持。
   - `recall.error` への参照を `false` literal に縮退 (もはや error path がない)。
   - `recall-to-loop-adapter.ts` を新設し pure fn `adaptRecallToLoopEntry` を提供 (Andy Matuschak evergreen-note grounding)。
3. **legacy RPC 完全削除**:
   - proto から 3 rpc 削除 (request / response message は残置 — proto3 service には reserved 概念がないため)。
   - `home_recall.go` ファイル削除、`home_recall_deprecation_test.go` 削除。
   - `alt-butterfly-facade/internal/handler/proxy_handler.go` の `streamingProcedures` から `StreamRecallRailUpdates` 削除。
   - FE `knowledge_home.ts` / `index.ts` から `getRecallRailCandidates` / `snoozeRecallItem` / `dismissRecallItem` 削除。
4. **Feature flag / config の整理**:
   - `domain.FlagRecallRail` 削除、`KnowledgeHomeConfig.EnableRecallRail` 削除、`feature_flag_gateway` の switch case 削除。
   - `recall_rail_usecase.Execute` から flag check を抜き、`Home payload に recall を含めるかは "usecase の存在" のみで判断」する。
   - `knowledge_home_admin.GetFeatureFlags` の response から `EnableRecallRail` 設定を抜く (proto field は wire compat で残置)。
   - FE `FeatureFlagPanel.svelte` の `Recall Rail` toggle 削除。E2E fixture の `enable_recall_rail` 削除し `FEATURE_FLAGS_RECALL_DISABLED` を base alias に縮退 (既存 E2E が compile 維持)。

### 検討した代替案

- **観測 window を真面目に 1 release 待つ**: `docs/plan/knowledge-loop-evolution-remaining-tasks.md` PR 13 の前提通り。**ユーザの明示指示で override**。観測ログ (`legacy.recall_rail.deprecated`) は PR 3 で landing 済なので、production で deploy 後すぐに count = 0 を確認できれば後追い safety net になる。
- **proto reserved 句で field 番号予約**: proto3 service は `reserved` syntax を持たないため適用不可。3 RPC 名そのものを後で再導入したくなったら別名 + 別 service block で対応する。
- **handler を残し internal 410 Gone を返す**: 採用しなかった。コード量が増えるだけで実利が薄い。proto から消した時点で client は型レベルで気付く。
- **flag を「 always-on」default で残す**: 採用しなかった。flag の semantic drift 問題 (Context §4) が解決しない。flag そのものを除去するのが最も健康。

## Consequences

### Pros

- recall data の単一情報源化 (canonical contract invariant 10) が成立。snooze 後の race / stale cache が proto 設計レベルで消える。
- TrackHomeAction が user action の単一窓口になり、API surface area が縮小。
- proxy_handler / facade allowlist / feature flag / FE toggle / E2E fixture が一括クリーンアップされ、deprecated path のメンテコスト 0。
- `recall-to-loop-adapter` が pure fn として landing したので、Knowledge Loop Review plane への recall integration (ADR-000912 後継 / future PR) で adapter を再利用できる。

### Cons / Tradeoffs

- 観測 window を skip したので、もし忘れられた client が legacy RPC を叩いていた場合に **production deploy 直後** に 404 / Unimplemented を返す。今後の deploy で `legacy.recall_rail.deprecated` ログを `c2quay deploy` 直前にもう一度確認することを runbook (`knowledge-loop-recall-deprecation.md`) に追記する。
- proto request / response message を file に残置したため、grep で「未参照 message」がノイズになる。後続の clean-up PR で削除候補。
- `recallRailUsecase` は今もコード上に残るが (Home payload に candidates を入れる責務)、名前が legacy 想起を残す。將来 `homeRecallCandidatesUsecase` 等への rename ADR が要る。
- `EnableRecallRail` を `knowledge_home_admin.GetFeatureFlags` response から外したが proto field は残しているので wire 形は zero-valued。proto field 自体の削除は別 ADR / 別 deploy で順次。
