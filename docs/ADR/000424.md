# ADR-000424: Knowledge Home の backend contract に summary_state と need_to_know_count を追加する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — proto, domain, projector, repository, handler
- alt-butterfly-facade (Go) — 生成コードの再生成
- alt-frontend-sv (TypeScript/SvelteKit) — Connect client 型定義・変換関数

## Context

Phase 5 までの Knowledge Home 実装では、要約の処理状態をフロントエンドが「`summary_excerpt` が空なら `"Summarizing..."` を表示する」という推論で補っていた。この方式には 3 つの問題がある:

1. **不正確**: `summary_excerpt` が空なのは「要約処理中」だけでなく「要約リクエスト未発行」「要約失敗」の場合もある。フロントエンドはこれらを区別できない
2. **UX 上の断定**: 実際には要約が来ない記事にも「Summarizing...」と表示され、ユーザーに「もうすぐ来る」という誤った期待を与える
3. **Phase 6 UI 刷新の前提条件**: KnowledgeCard を 3 レーン化する際に、skeleton / chip / excerpt の 3 表示を backend truth で切り替える必要がある

同様に、Evening Pulse の「Need to Know」記事件数は `WhyReason` の `pulse_need_to_know` コードとして個々の item に存在していたが、TodayBar に集計値を表示する手段がなかった。`TodayDigest` に件数を追加すれば、フロントエンドは単純な参照で badge を表示できる。

## Decision

**`KnowledgeHomeItem` に `summary_state` を、`TodayDigest` に `need_to_know_count` を追加する。いずれも backend truth として算出し、フロントエンドは表示のみを担う。**

### 1. `summary_state` の追加

Proto field: `string summary_state = 12` (`KnowledgeHomeItem`)

| 値 | 意味 | 算出タイミング |
|----|------|---------------|
| `missing` | 要約リクエスト未到達 | DB デフォルト値 |
| `pending` | 記事取り込み済み、要約処理待ち | `projectArticleCreated` 実行時 |
| `ready` | 要約 excerpt が存在 | `projectSummaryVersionCreated` 実行時（excerpt 非空） |

**変更箇所:**

- **Domain**: `domain.KnowledgeHomeItem` に `SummaryState string` フィールドと 3 定数を追加
- **Migration**: `ALTER TABLE knowledge_home_items ADD COLUMN summary_state TEXT NOT NULL DEFAULT 'missing'`。既存行で `summary_excerpt` が非空のものは `'ready'` にバックフィル
- **Projector**: `projectArticleCreated` → `pending`、`projectSummaryVersionCreated` → excerpt 有無で `ready` / `pending`
- **Repository**: SELECT に `summary_state` 追加。Upsert は merge-safe: `ready` は常に勝ち、`missing` は既存値を保持
- **Handler**: `convertHomeItemToProto` に `SummaryState` mapping 追加
- **Connect client**: `KnowledgeHomeItemData` に `summaryState: SummaryState` 追加、`convertItem` 更新

**検討した代替案: フロントエンド推論で `publishedAt` から算出する方式。** `publishedAt` は記事公開時刻であり要約処理状態ではない。古い記事の再要約や遅延取り込み記事を誤って `missing` 扱いする。赤入れレビューで「不正確」と指摘され、backend contract 化を選択した。

### 2. `need_to_know_count` の追加

Proto field: `int32 need_to_know_count = 8` (`TodayDigest`)

- **算出方法**: handler の `GetKnowledgeHome` で、返却する items のうち `WhyReason.Code == "pulse_need_to_know"` を持つ item 数をカウント
- **DB に保存しない**: リクエスト時に算出。`today_digest_view` テーブルへの column 追加は不要
- **Domain**: `domain.TodayDigest` に `NeedToKnowCount int` フィールドを追加
- **Connect client**: `TodayDigestData` に `needToKnowCount: number` 追加、`convertDigest` 更新

**検討した代替案: `today_digest_view` にカラムを追加して projector で集計する方式。** 正確性は高いが、digest は日次集計であり、リアルタイムに件数が変わる need-to-know とは更新頻度が合わない。リクエスト時算出の方がデータの鮮度と実装のシンプルさの両面で優れる。

### 3. Repository の merge-safe upsert 拡張

`summary_state` の upsert ロジック:

```sql
summary_state = CASE
  WHEN EXCLUDED.summary_state = 'ready' THEN 'ready'
  WHEN EXCLUDED.summary_state != 'missing' THEN EXCLUDED.summary_state
  ELSE knowledge_home_items.summary_state
END
```

`ready` は不可逆の到達状態として扱う。一度 `ready` になったら他のイベントで `pending` や `missing` に戻ることはない。

## Consequences

### Pros

- フロントエンドが要約状態を正確に表示できる（skeleton / chip / excerpt の 3 表示）
- `"Summarizing..."` を本文位置に表示する UX アンチパターンが解消される
- NeedToKnow 件数を TodayBar に badge として表示でき、Evening Pulse への導線が明確になる
- backend truth 方式により、将来の状態追加（例: `failed`, `skipped`）がフロントエンド変更なしで可能

### Cons / Tradeoffs

- **Migration の実行時間**: `summary_excerpt` が非空の行を `ready` にバックフィルする UPDATE が走る。`knowledge_home_items` のサイズ次第ではオフピーク適用が望ましい
- **`need_to_know_count` はページングの影響を受ける**: 最初のページの items からしかカウントしないため、全 item の正確な件数ではない。ただし TodayBar は「大まかな存在感」を示す用途なのでこの近似で十分
- **Proto フィールド番号の消費**: `KnowledgeHomeItem` field 12、`TodayDigest` field 8 を使用。後方互換
