# ADR-000462: Knowledge Home V2 reproject を高速バッチループ化し SwapReproject で ActivateVersion を呼び出す

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-backend (Go)
- alt-frontend-sv (SvelteKit / TypeScript)

## Context

Knowledge Home の V2 reproject が `running` 状態で長期間滞留していた。原因は 3 つあった。

### 1. 処理速度の不足

reproject job は 1 tick（30 秒周期）あたり `batchSize=100` 件の固定バッチを 1 回だけ処理する実装だった。未処理イベントが約 125,000 件あり、このペースでは完了まで数時間以上かかる見込みだった。

### 2. SwapReproject が ActivateVersion を呼んでいない

`SwapReproject` は run のステータスを `swapped` に更新するだけで、`knowledge_projection_versions` テーブルの active version を切り替える `ActivateVersion` を呼んでいなかった。これにより swap 実行後も V1 が active のままで、UI には依然として V1 のデータが表示され続ける状態だった。

さらに、`knowledge_projection_versions` に V2 の行が存在しない状態で `ActivateVersion` を呼ぶと、現行の `ActivateProjectionVersion` 実装が V1 の `active` を先に無効化するため、active version が 0 件になるリスクがあった。

### 3. フロントエンドのステータス名不一致

`ReprojectRunsTable.svelte` の `canSwap` / `canCompare` 関数が `status === "completed"` を条件にしていたが、バックエンドは `"swappable"` を返す。この不一致により、reproject が完了しても UI の Swap / Compare ボタンが常に disabled のままで操作不能だった。

## Decision

以下の 3 つの修正を採用する。

### 1. reproject job をバッチループ化して高速化する

`knowledge_reproject_job.go` に専用定数 `reprojectBatchSize=2000` と `reprojectLoopSafetyMargin=3s` を導入し、1 tick 内で context の deadline が近づくまで複数バッチを連続処理するループに変更する。

- 各バッチ処理後に checkpoint と stats を永続化し、途中中断しても進捗を失わない
- `time.Until(deadline) < reprojectLoopSafetyMargin` で deadline を監視し、job budget を超過しない
- ログに `elapsed` を追加し、1 tick あたりの実効 throughput を計測可能にする

### 2. SwapReproject で ActivateVersion を呼び出す

`knowledge_reproject_usecase/usecase.go` の `SwapReproject` に version パース + `ActivateVersion` 呼び出しを追加する。

- `run.ToVersion`（例: `"v2"`）を `strconv.Atoi` でパースし、`ActivateVersion(ctx, 2)` を呼ぶ
- activate 成功後にのみ run を `swapped` に遷移させる（activate 失敗時は run を更新せず安全側に倒す）
- swap 前に `knowledge_projection_versions` に対象 version の行が存在することを前提とする。今回は手動 SQL で V2 行を作成した

### 3. フロントエンドのステータス名をバックエンドに合わせる

`ReprojectRunsTable.svelte` の `canSwap`、`canCompare`、`statusColor` で参照していた `"completed"` を `"swappable"` に修正する。

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|--------|------|-----------|
| `StartReproject` で V2 version 行を自動作成する | 安全性は高い | usecase/DI/port の変更が増えクリティカルパスが伸びる。follow-up に分離 |
| `ActivateProjectionVersion` で rows affected を検証する | 防御的で望ましい | 同上。今回は V2 行の手動作成で対処 |
| バッチサイズだけ増やしループ化はしない | 実装は単純 | 1 tick あたり 1 バッチでは job budget（25 秒）の大半を無駄にする |

## Consequences

### Pros

- reproject の処理速度が大幅に向上し、125,000 件超のイベントを数分で完了できるようになった
- `SwapReproject` が `ActivateVersion` を呼ぶことで、swap 操作が実際に active version を切り替えるようになった
- UI の Swap / Compare ボタンが `swappable` ステータスで正しく有効化され、管理画面から操作可能になった
- バッチごとに checkpoint を永続化するため、途中障害からの復旧が安全になった

### Cons / Tradeoffs

- V2 version 行の事前作成が手動運用に依存する。`StartReproject` での自動作成は follow-up で対応予定
- `ActivateProjectionVersion` に対象 version 不在時の防御がない。rows affected の検証も follow-up
- バッチサイズ 2000 の妥当性は実環境の throughput 計測に基づいて今後調整する可能性がある
- regular projector が swap 前後に V1 のみに投影したイベント（event gap）は自動で V2 に追いつかないため、gap 埋め用の short-range reproject を別途実行する前提
