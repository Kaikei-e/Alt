# ADR-000650: news-creator の query expansion 少数例を抽象化してトピック汚染を防ぐ

## Status

Accepted

## Date

2026-04-08

## Affected Services

- news-creator (Python) - RAG 用 query expansion の prompt と出力検証を修正

## Context

`news-creator` の multi-turn query expansion では、conversation history を解決するために few-shot 例を入れていた。しかし実運用ログでは、イランの石油危機に関する質問に対して、天気予報やスマートフォン比較の例が部分的に再生成され、まったく別トピックの expanded query が混入していた。

この挙動は、few-shot 例が「出力形式の見本」ではなく「次に生成すべき内容」として学習されてしまう典型的な失敗パターンに近い。最新の prompt engineering の考え方でも、例はできるだけ抽象化し、指示は先頭に置き、必要なら zero-shot から始めて不足分だけ最小限の例を足すのが安全である。Query rewriting 系の研究でも、rewriter は元の意図を保ちつつ、会話文脈から必要な情報だけを補うべきだとされている。

## Decision

`news-creator` の multi-turn query expansion prompt から、具体的なドメイン例を削除し、`[TOPIC_A]` / `[TOPIC_B]` の抽象プレースホルダ例に置き換える。

あわせて、`SAME TOPIC as the input` の制約を維持し、例の目的を「形式と構造の提示」に限定する。これにより、モデルが weather や smartphone の具体トピックを copy する余地を減らす。

さらに、テストで以下を明示的に守る。

1. 具体的な題材語彙をテンプレートに含めない
2. 抽象プレースホルダが入っていることを確認する
3. 既存の instruction echo / preamble フィルタを壊さない

zero-shot 化は試みとしては単純だが、現状のモデルでは coreference resolution の安定性が下がるため採用しない。抽象化した few-shot を残しつつ、再発防止のガードを追加する方が実運用でのリスクが低い。

## Consequences

### Pros

- few-shot 例の topic leakage を抑え、expanded query が別話題に逸れる確率を下げられる。
- 会話履歴を使う multi-turn の利点を残したまま、形式だけを教える設計に寄せられる。
- テンプレートとテストの両方で contamination を検出できるため、今後の編集で再発しにくい。
- 実際に `POST /api/v1/expand-query` を叩いた検証で、汚染された天気・スマートフォン系の出力が消えた。

### Cons / Tradeoffs

- 抽象例は具体例より学習シグナルが弱く、モデルによっては拡張の多様性が少し落ちる可能性がある。
- zero-shot ほど単純ではないため、prompt とテストの保守点は残る。
- expanded query が保守的になりすぎた場合は、後続の retrieval 側で補う必要がある。
