# ADR-000602: Inoreader 記事取り込みに Tier1 入口フィルタを導入する

## Status

Accepted

## Date

2026-03-28

## Affected Services

- pre-processor-sidecar (Go) — Tier1 分類ロジックとフェッチパイプラインへの統合

## Context

Inoreader API から取り込まれる記事の一定割合が、下流処理（要約、RAG、recap 等）にとって有用でないことが判明した。具体的には以下のパターンが観測された。

- RSS フィードが description/excerpt のみを配信し、本文が短い要約テキストにとどまるケース
- `<img>` タグと短いリード文のみで構成されるケース（NHK、BBC 等のリード配信型）
- `続きをみる`、`Read more`、`...` 等の切り詰めマーカーで終わる不完全なコンテンツ
- クロスワード、フォトギャラリー等の非記事コンテンツ
- `test`、`Discussion` 等のプレースホルダー

これらの低品質コンテンツが `pre-processor-db` → `alt-db` へと伝播し、要約ジョブや検索インデックスにノイズとして混入していた。

重要な前提として、Inoreader 取り込みは Alt の補完チャネルであり、主経路はユーザー操作起点のオンデマンド全文取得である。したがって、Inoreader 側の薄い本文は「取得失敗」ではなく「補完ソースとして不十分」と位置づける。

## Decision

**`pre-processor-sidecar` の記事フェッチパイプラインに Tier1 入口フィルタを導入し、保存に値する記事のみを下流に渡す。**

### Tier1 定義

以下のすべてを満たす記事を Tier1 とする。

1. HTML タグ除去後の plaintext が 500 文字（rune）以上
2. `続きをみる`、`続きを読む`、`Read more`、`Read More`、`...`、`…` で終わっていない
3. `<img>` 主体のコンテンツではない（画像タグのみで plaintext が極端に短い場合を除外）
4. URL が非記事パターン（`/crosswords/`、`/gallery/`、`/puzzles/`）に一致しない
5. 既知のプレースホルダー/ボイラープレートに一致しない

### 設計方針

- **入口で止める**: 低品質データがいったん保存されてから各サービスで除外するのではなく、`pre-processor-sidecar` の取り込み時点で除外する。これにより除外ロジックの散在を防ぐ
- **純粋関数**: 分類ロジックは `domain` パッケージに配置し、I/O やロガーへの依存を持たない純粋関数として実装。テスト容易性と再利用性を確保
- **HTML → plaintext 変換**: `bluemonday.StrictPolicy()` + `strings.Fields` による whitespace 正規化。sidecar が既に依存する `bluemonday` ライブラリを再利用
- **フェッチパイプラインへの統合**: UUID 解決後、バッチ保存前にフィルタを挿入。non-Tier1 記事は構造化ログに記録したうえで破棄

### 検討した代替案

1. **下流で non-Tier1 を除外する案** — 低品質データがいったん保存される。各サービスに除外ロジックが散る。入口で止めれば済む問題を複雑にする。不採用
2. **`alt-db` に品質フラグカラムを追加する案** — スキーマ変更とマイグレーションが必要。全下流サービスでフラグ参照が必要。入口フィルタで十分な効果が得られるため不採用
3. **ML ベースの品質スコアリング** — 初期段階では過剰。ルールベースの判定で十分な精度が得られる。将来的な拡張点として残す

### 実装構成

| ファイル | 役割 |
|---------|------|
| `domain/tier1_classifier.go` | `ClassifyTier1(content, url) ClassifyResult` — 純粋関数 |
| `domain/tier1_classifier_test.go` | テーブル駆動テスト（境界値、各ルール、エッジケース） |
| `service/tier1_filter.go` | `FilterTier1Articles(articles, logger) Tier1FilterResult` — バッチフィルタ+ログ |
| `service/tier1_filter_test.go` | 混合バッチのフィルタリングテスト |
| `service/article_fetch_service.go` | `FetchArticles()` にフィルタステップを挿入 |

### フィルタ挿入ポイント

```
Inoreader API
  → FetchStreamContents()
  → ResolveArticleUUIDs()
  → FilterTier1Articles()     ← 新規挿入
  → ProcessArticleBatch()
  → pre-processor-db
```

## Consequences

### Pros

- non-Tier1 記事が `pre-processor-db` に保存されなくなり、下流への伝播が停止する
- 分類ロジックが `domain` パッケージの純粋関数として隔離されており、テストが容易
- `ArticleFetchResult.FilteredNonTier1` フィールドと構造化ログにより、フィルタの効果を定量的に観測可能
- 既存のテスト・パイプラインに影響なし（全既存テスト PASS を確認済み）

### Cons / Tradeoffs

- 500 文字のしきい値はヒューリスティックであり、一部の短くても有用な記事（速報ニュースの要約等）が除外される可能性がある。ただし Alt の運用方針として Inoreader 補完チャネルでは Tier1 のみを対象とし、必要な全文はユーザーの明示操作で取得する前提
- non-Tier1 記事は完全に破棄され、reject log テーブルには保存されない。将来的に分析が必要になった場合は構造化ログから抽出する必要がある
- 切り詰めマーカーやプレースホルダーのパターンリストはメンテナンスが必要。新しいパターンが発見された場合はリストへの追加が必要
