# ADR-000813: docker-build.yaml image-size regression gate を news-creator にも展開し budget 1800 MiB と CPU-only smoke-check で CI ゲートする

## Status

Accepted

## Date

2026-04-20

## Affected Services

- **.github/workflows/docker-build.yaml** — 既存の recap-subworker 専用 size + smoke ゲート (line 283-311) を news-creator にも展開。`load:` 条件と step の `if:` 条件を `contains(fromJSON('["recap-subworker","news-creator"]'), matrix.service)` に書き換え、`MAX_MB` をサービス別 `case` 分岐。news-creator 向け smoke には `sentence_transformers.CrossEncoder` / `news_creator.usecase.rerank_usecase` / `main` の import を追加

## Context

[[000812]] で news-creator の Dockerfile.app を CPU-only PyTorch + 2-stage uv sync に書き換えた結果、local build で image サイズは **1243 MiB** まで落ちた。しかし以下 3 つの regression リスクが構造的に残る:

1. **`sentence-transformers` の transitive で CUDA wheel が再混入する** — `[tool.uv.sources] torch = { index = "pytorch-cpu" }` は torch に対してのみ source override を効かせる。将来 `sentence-transformers` が `torchvision` や `xformers` を追加依存として導入した場合、それらが CUDA wheel を引き込み、`uv.lock` が黙って膨張する経路が存在する。
2. **`sentence-transformers` の silent major bump** — `news-creator/app/pyproject.toml` は `sentence-transformers>=3.0.0` を宣言するが `uv.lock` は `5.3.0` に解決している。v5 系は `CrossEncoder.__init__` 引数の整理や `trust_remote_code` default 変更があり、単純な pytest (モック化された rerank 単体テスト) では検知できない API 整合ミスが起こりうる。
3. **誰かがローカルで `uv pip install <paquete-gpu>` を `uv.lock` に混ぜる** — PR レビューだけでは検知しづらい。

`.github/workflows/docker-build.yaml:283-311` には recap-subworker を対象にした既存の `Enforce image-size budget and smoke-check core imports` step があり、以下を実施していた:

- `docker image inspect` で image size を計測し `MAX_MB=2900` を超えたら fail
- `docker run --rm` で `assert not torch.cuda.is_available()` + `nvidia` / `triton` import 不在を検証

この既存 gate は `matrix.service == 'recap-subworker'` 固定で news-creator には効かない。構造は同一であるため news-creator を展開対象に加えるコストは低い。

本 ADR はこの既存 step の対象を news-creator にも拡張し、[[000812]] のサイズ削減を CI で恒久的に担保する。

## Decision

### 1. 条件判定を matrix.service リスト contains に切替

```yaml
- name: Build and push
  ...
  load: ${{ github.event_name == 'push' && github.ref == 'refs/heads/main' && contains(fromJSON('["recap-subworker","news-creator"]'), matrix.service) }}
  ...

- name: Enforce image-size budget and smoke-check core imports
  if: contains(fromJSON('["recap-subworker","news-creator"]'), matrix.service) && github.event_name == 'push' && github.ref == 'refs/heads/main'
  env:
    IMAGE_TAG: ${{ fromJSON(steps.meta.outputs.json).tags[0] }}
    SERVICE: ${{ matrix.service }}
  run: |
    set -euo pipefail
    case "$SERVICE" in
      recap-subworker) MAX_MB=2900 ;;
      news-creator)    MAX_MB=1800 ;;
    esac
    SIZE_BYTES=$(docker image inspect "$IMAGE_TAG" --format '{{.Size}}')
    SIZE_MB=$(( SIZE_BYTES / 1024 / 1024 ))
    echo "$SERVICE image size: ${SIZE_MB} MiB (budget ${MAX_MB} MiB)"
    [ "$SIZE_MB" -le "$MAX_MB" ] || { echo "::error::$SERVICE image (${SIZE_MB} MiB) exceeds budget (${MAX_MB} MiB)"; exit 1; }
    ...
```

ホワイトリスト配列にしたのは、将来 `tag-generator` など他 Python サービスを追加しやすくするため。`matrix.service == 'X' || matrix.service == 'Y'` の OR 列挙より意図が読みやすい。

### 2. news-creator 専用 smoke-check を追加

docker image 内で以下を実行:

```python
import sys, importlib
import torch
assert not torch.cuda.is_available(), 'torch CUDA must be disabled'
for m in ('nvidia', 'triton'):
    try:
        importlib.import_module(m)
        sys.exit(f'{m} must not be installed')
    except ImportError:
        pass
from sentence_transformers import CrossEncoder  # noqa: F401
from news_creator.usecase import rerank_usecase  # noqa: F401
import main  # noqa: F401
print('smoke OK', torch.__version__)
```

コアの 3 チェック (`torch.cuda.is_available() == False` / `nvidia` 不在 / `triton` 不在) は recap-subworker と共通。news-creator 固有に追加した 3 import:

- `from sentence_transformers import CrossEncoder` — v5 silent bump で `CrossEncoder` の import 自体が割れるケースを早期検知
- `from news_creator.usecase import rerank_usecase` — 遅延 import 経路の到達性確認。module 単位で初期化時例外を拾う
- `import main` — FastAPI app の module import が成立することを確認。`VIRTUAL_ENV` / `PATH` 設定のバグで `/app/.venv/bin/python` が main を解決できない事故を検知する

smoke 失敗は step 全体を fail させるため、main push 時点で image が GHCR に push されても deploy trigger (`dispatch-deploy.yaml`) が走る前に block される。

### 3. MAX_MB の初期値設定

local build 実測 1243 MiB (`news-creator/Dockerfile.app` 書き換え後) に対し **初期 budget 1800 MiB** (headroom ~45%) を設定する。recap-subworker が `2753 MiB 実測 / 2900 MiB budget` で headroom 5% なのと比較すると緩いが、これは:

- 切替直後の実測を CI 環境 (ubuntu-latest, BuildKit バージョン差あり) で取り直す猶予を残す
- sentence-transformers v5 が HuggingFace 依存で初回 fetch した場合の数 MiB 増をカバー
- follow-up PR で `news-creator)    MAX_MB=1500 ;;` → `1200 ;;` と段階的に絞り込む

意図。初期値で即 fail する設計はデプロイ停止リスクに直結する。

### 4. 検討したが採用しなかった代替案

| 代替案 | 却下理由 |
|---|---|
| 全 Python サービス (tag-generator / recap-evaluator / metrics) に同一 gate を一括展開 | tag-generator は builder で SBERT pre-download する独自パターンがあり image が 3 GiB 級、recap-evaluator / metrics は CPU torch pin が未完了。段階展開が必要で、本 ADR では news-creator 1 件に絞る |
| GHA artifact での size diff (PR 上で base vs head を比較) | reusable workflow 改造 + artifact 保持コストが発生し、単純な MAX 閾値でコールド regression 検知には十分 |
| smoke-check を別 step に分離 | size 計測と smoke-check は同一 image に対する逐次処理で分離の利点なし。単一 step の方が rollback が楽 |
| budget を最初から 1500 MiB に tight 設定 | 実測 1243 MiB に対し余裕 257 MiB (20%) しかない。CI 環境差 (BuildKit / runner version) で揺らぐと deploy 停止 |

## Consequences

### Pros

- [[000812]] のサイズ削減を CI main-push 時点で恒久的にゲート。サイズ退行を push 直後に fail-fast でき、`dispatch-deploy.yaml → Kaikei-e/alt-deploy` に巨大 image が流出するのを防ぐ
- `sentence-transformers` の silent major bump 起因の `CrossEncoder` import 割れを、unit test ではなく実 image 上の smoke で検知できる
- FastAPI app の venv 解決 (`VIRTUAL_ENV` / `PATH`) 起因の bootstrap バグも `import main` で検知できる
- 既存 step の最小 diff で実装。matrix / cache / tag 戦略の他箇所には手を入れない
- ホワイトリスト配列 (`fromJSON('["recap-subworker","news-creator"]')`) により将来 `tag-generator` 等を追加する際は配列に 1 要素足すだけの 1 行変更で済む

### Cons / Tradeoffs

- **`load: true` で build 結果を runner ローカルに展開する service が 1 → 2 に増える**。ubuntu-latest の runner でディスク / 時間に数十秒〜数分増のコスト発生。ただし main push 時のみで PR ビルドには影響しない
- 初期 budget 1800 MiB は実測 1243 MiB に対し緩い。**follow-up PR で 1500 → 1200 MiB への段階的 tighten が必要**。本 ADR ではそこまで踏み込まない
- smoke-check は起動時 import のみで、FastAPI endpoint の健全性 (`/health`, `/v1/rerank`) までは見ない。runtime 健全性は compose-level の healthcheck 側の責務で、本 gate は build artifact の健全性に絞る方針
- `sentence-transformers` v3 → v5 の API 整合は import smoke で検知できるが、CrossEncoder の**挙動**退行は拾えない。rerank quality 評価は別系 (acolyte-orchestrator 評価ハーネス [[000793]]) の責務
