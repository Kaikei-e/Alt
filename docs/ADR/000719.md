# ADR-000719: pre-processor の summarize エンドポイントに X-Service-Token 認証を必須化する

## Status

Accepted

## Date

2026-04-14

## Affected Services

- pre-processor (Go)
- alt-backend (Go)

## Context

pre-processor の REST API (`:9200`) はこれまで `/api/v1/summarize`, `/api/v1/summarize/stream`, `/api/v1/summarize/queue`, `/api/v1/summarize/status/:job_id` を **完全に無認証** で公開していた。ネットワーク分離のみに依存した設計であり、以下のリスクが潜在していた。

- Docker ネットワーク境界が誤設定された瞬間、LLM を叩く高コストなエンドポイントが外部から直接叩ける
- alt-backend 以外のコンテナからも無制限に呼び出せるため、サイドカーや誤った配線で LLM 計算リソースを食い潰される
- `internal/auth/middleware/auth.go` に `RequireServiceAuth()` の雛形は存在したが、`X-Service-Token` が空でなければ通過する実装になっており、`middleware/auth_middleware.go::SetupAuth()` も placeholder ハンドラを登録する未配線のコードパスだった
- 同時期に ADR-000717 で alt-backend 側の `/v1/internal/*` に `X-Service-Token` による service-to-service 認証が整備されたが、pre-processor 側は並びのままになっていた

OWASP ASVS 5.0 / OWASP Top 10 2025 A01 (Broken Access Control) に該当する High 深刻度の欠陥であり、他の MED/LOW 指摘より先に塞ぐ判断をした。

## Decision

以下をまとめて採用する。

1. pre-processor に `middleware.ServiceAuthMiddleware` を新設し、alt-backend の `service_auth_middleware.go` と同一ロジック（`crypto/subtle.ConstantTimeCompare` による定時間比較、`SERVICE_SECRET` / `SERVICE_SECRET_FILE` 双方対応、空シークレット時は全リクエスト 500 で deny-all）を採る。
2. `bootstrap/server.go` の `/api/v1/*` グループに `RequireServiceAuth` を必須化する。liveness 用の `/api/v1/health` のみ auth グループの外に置き、プローブが認証不要で通るようにする。
3. 呼び出し元である alt-backend (`rest/rest_feeds/summarization/*`) の 4 ヶ所（`callPreProcessorSummarize` / `streamPreProcessorSummarize` / `callPreProcessorSummarizeQueue` / `callPreProcessorSummarizeStatus`）に `X-Service-Token` の付与を同一コミットで入れる。共通ヘルパ `setServiceToken` を置き、空シークレット時は header を設定しない（本番誤設定は config 側で fail-fast する将来対応に繋げるため）。
4. 値の出どころは既存の shared secret `/run/secrets/service_secret`。pre-processor 側は受信検証用 `SERVICE_SECRET_FILE` と、既存の発信用 `SERVICE_TOKEN_FILE` が **同じ secret を指す** ように `compose/ai.yaml` で明記する。

本変更は Plan モードで作成した [[plans/partitioned-spinning-whisper]] のうち **HIGH-1** に該当する。HIGH-2 (Connect-RPC の無認証)、MED-1〜3、LOW-1/2 は後続 ADR で扱う。

## 実装の概要

### pre-processor

- `pre-processor/app/middleware/service_auth_middleware.go` 新設
  - `X-Service-Token` header を要求し、空/余白のみ/長さ不一致/値不一致はいずれも 401
  - `SERVICE_SECRET` 未設定時はトークンが送られてきても 500 でログを残し、deny-all を維持する
  - 成功時は `c.Set("service.authenticated", true)` を立てる
- `pre-processor/app/bootstrap/server.go`
  - `/api/v1/health` を `e.GET` で auth グループの外側に配置
  - `e.Group("/api/v1")` に `serviceAuth.RequireServiceAuth()` を適用し、summarize 系 4 ルートを保護
- テストは RED → GREEN の順で実装
  - middleware 単体テスト 6 件（missing/wrong/correct/different-length/secret-not-configured/empty-header）
  - bootstrap 統合テスト 7 件（4 ルート × 無トークン 401、health 200、wrong 401、correct で non-401）

### alt-backend

- `rest/rest_feeds/summarization/helpers.go` の 4 ヘルパにシグネチャ引数 `serviceSecret string` を追加し、`setServiceToken(req, secret)` を差し込む
- 呼び出し元 (`request_handlers.go`, `stream_handlers.go`, `status_handlers.go`) から `cfg.InternalAPI.ServiceSecret` を注入
- テスト `helpers_auth_test.go` を 5 件追加（4 ヘルパ × header 送出確認、空シークレットで header が付かないことの確認）

### Compose

- `compose/ai.yaml` の pre-processor サービス環境変数に `SERVICE_SECRET_FILE=/run/secrets/service_secret` を追記。既存の `SERVICE_TOKEN_FILE` と同じ shared secret を指すことをコメントで明示する。

### End-to-End 検証

コンテナ再ビルド後に以下を確認した。

| ケース | 結果 |
|---|---|
| `GET /api/v1/health` | 200 |
| `POST /api/v1/summarize` (no token) | 401 `missing authentication headers` |
| `POST /api/v1/summarize` (wrong token) | 401 `invalid authentication token` |
| `POST /api/v1/summarize` (correct token) | 400（handler 層の入力バリデーションに到達、auth は通過） |

pre-processor のログでも `service auth failed: missing token header` と `invalid token` が期待どおり記録されることを確認した。

## Consequences

### Pros

- `:9200` が誤って外部公開されても、shared secret を持たないクライアントは LLM を叩けない
- `crypto/subtle` による定時間比較でサイドチャネルを塞ぐ
- alt-backend 側の呼び出しは既存 `cfg.InternalAPI.ServiceSecret` を流用するため追加の env 設定は不要
- RED 13 件 / GREEN 13 件 の TDD サイクルを踏み、middleware 単体と bootstrap 統合の両層でテストを残せた
- ADR-000717 の alt-backend 側対策と対称な構造になり、pre-processor ↔ alt-backend 間の service-to-service 認証が双方向で整った

### Cons / Tradeoffs

- 既存のローカル開発や ad-hoc スクリプトで `SERVICE_SECRET` を設定しないまま pre-processor を叩いていたフローは 401/500 になる。`.env` に shared secret を入れるか、compose 経由で起動する運用に統一する必要がある
- 空シークレット時は 500 を返す設計にしたため、本番で shared secret の投入漏れが起きると「全リクエスト 500 」として現れる。可観測性で検知するか、LOW-2 で予定している config fail-fast を早めに実装する必要がある
- alt-backend の 4 ヘルパにシグネチャ追加をしたため、将来 pre-processor 以外のサービスへ同種の呼び出しを増やす際は同じ pattern（secret 引数注入）を踏襲する必要がある
- Connect-RPC 側 (`:9202`) は本 ADR のスコープ外であり、現状は依然として無認証のままになっている。HIGH-2 として別 ADR で対処する
