# ADR-000877: CI/CD 障害から得た三つの不変ルールを制度化する（View列順序・sparse-checkout・atlas.sum追跡）

## Status

Accepted

## Date

2026-05-02

## Affected Services

- knowledge-sovereign: migration view 列順序修正、staging compose に高速 tick 環境変数追加
- alt-backend: CI sparse-checkout に `proto/` ディレクトリを追加
- migrations-atlas: `migrations/atlas.sum` を git 追跡対象に変更（`.gitignore` から除外）
- rag-migration-atlas: `migrations/atlas.sum` を git 追跡対象に変更（独自 `.gitignore` を削除）

## Context

2026-05-01〜02 のデプロイサイクルで、独立した三つのバグが連鎖して CI と本番デプロイを同時に止めた。

### バグ 1: `CREATE OR REPLACE VIEW` の列順序変更 (PG 42P16)

migration `00018_add_knowledge_loop_visibility_lifecycle.sql` で `knowledge_loop_entries_public` ビューに `visibility_state` と `completion_state` の 2 列を追加する際、既存列 `dismiss_state` と `render_depth_hint` の間に挿入しようとした。PostgreSQL は `CREATE OR REPLACE VIEW` の実行時に SELECT リストの位置でカラム名を照合するため、位置をずらすと既存列の名前変更とみなしエラー `42P16: cannot change name of view column` を返す。マイグレーションコンテナが失敗し、knowledge-sovereign のデプロイが全停止した。

### バグ 2: CI `backend-go` の sparse-checkout に `proto/` が欠落

`TestTransitionPolicyConformance`（`alt-backend/app/handler/knowledge_loop/transition_policy_test.go`）は実行時に `proto/alt/knowledge/loop/v1/loop_transition_policy.json` を読み込む。CI の `backend-go.yaml` は `alt-backend` ディレクトリだけを sparse-checkout していたため、`proto/` がディスク上に存在せず `go test ./...` が exit 1 で失敗した。ローカルでは全ファイルが揃っているので再現しなかった。

### バグ 3: `atlas.sum` が gitignore されていてコンテナ内で生成不能

`migrations-atlas/migrations/atlas.sum` は `.gitignore` に `migrations-atlas/migrations/atlas.sum` として記載されており、`rag-migration-atlas` も独自の `.gitignore` で `migrations/atlas.sum` を除外していた。本番デプロイ時、migrator コンテナは `/migrations` を `:ro`（read-only）でバインドマウントするため、チェックアウト時に atlas.sum が存在しなければ `atlas migrate hash` による自動生成もできない。結果として `open /migrations/atlas.sum: read-only file system` で migrate が失敗した。knowledge-sovereign・acolyte・recap・pre-processor の各 migrator は atlas.sum をすでにコミットしており問題がなかった。二つの migrator だけが gitignore により除外されていた。

### E2E hurl テスト (22-surface-planner-cron) の副次バグ

上記とは別に、E2E hurl スクリプトにも二つの誤りが含まれていた:

1. RPC パスを `ListKnowledgeEventsSince`（proto に存在しない）と誤記。正しくは `ListKnowledgeEvents`。
2. `SummaryVersionCreated` の `aggregateId` に `{{run_id}}` サフィックスを付けていたため、派生する `entry_key` が `article:art-planner-cron-{{run_id}}` となり、augur イベントの payload 内 `entry_key: article:art-planner-cron` と一致せず、最終アサーションが常に失敗していた。

## Decision

### Rule 1: `CREATE OR REPLACE VIEW` に新列を追加するときは末尾にのみ追加する

PostgreSQL の `CREATE OR REPLACE VIEW` は SELECT リストの位置でカラム名を照合する。既存列の間に割り込むと 42P16 エラーとなる。新列は **常に SELECT リストの末尾に追加する**。列の「論理的な並び順」が重要な場合は、アプリケーション層または新しいビューで対処し、置換ビューで列を並び替えない。

本 ADR での修正: `visibility_state, completion_state` を SELECT リスト末尾（`surface_score_inputs` の直後）に移動し、atlas.sum を再生成した。

### Rule 2: CI の sparse-checkout には参照するすべてのディレクトリを含める

GitHub Actions でサービス単体テストを実行する際、sparse-checkout は参照される全ディレクトリを明示的に列挙しなければならない。コードが相対パスやファイル検索でリポジトリルートを起点に別ディレクトリのファイルを読む場合（proto/ のような共有スキーマ定義など）、そのディレクトリも sparse-checkout の対象に加える。

本 ADR での修正: `backend-go.yaml` の `sparse-checkout` 入力に `proto` を追加した。

```yaml
sparse-checkout: |
  alt-backend
  proto
```

### Rule 3: migrator の `atlas.sum` は必ず git 追跡対象にする

read-only バインドマウント（`:ro`）で動作する Atlas migrator コンテナは、起動時に atlas.sum を自動生成できない。atlas.sum はすべての migrator で **git 追跡対象（コミット済み）** でなければならない。`.gitignore` への追加は禁止。atlas.sum の再生成が必要な場合は `atlas migrate hash` をローカルで実行してコミットする。

本 ADR での修正:
- `.gitignore` から `migrations-atlas/migrations/atlas.sum` 行を削除
- `rag-migration-atlas/.gitignore` ファイルを削除（atlas.sum のみを無視する内容だった）
- 両 migrator の `atlas.sum` を生成してコミット

## Consequences

### Pros

- **再発防止**: View 列順序・sparse-checkout・atlas.sum の三つの落とし穴が明文化され、レビュー時のチェックポイントになる
- **デプロイ完全性**: atlas.sum が全 migrator で揃い、`:ro` バインドマウント環境下でも migrator が正常起動する
- **CI 信頼性**: `proto/` が sparse-checkout に含まれることで、プロトコル定義を参照するテストがローカルと CI で同一条件になる
- **E2E test 正確性**: hurl スクリプトの RPC 名誤記と run_id 不一致が修正され、22番テストが正しくプランナー cron の動作を検証する

### Cons / Tradeoffs

- **atlas.sum の手動更新コスト**: 新しい migration ファイルを追加するたびに `atlas migrate hash` を実行してコミットしなければならない。自動化されていないため、うっかり忘れると次のデプロイで同じ read-only エラーが再発する。CI の `syntax-check` ジョブ（atlas.sum を再生成してから validate する）を全 migrator に揃えることが将来の改善候補。
- **sparse-checkout の保守コスト**: テストが新たな共有ディレクトリを参照するようになった場合、`backend-go.yaml` の更新が必要。コードとCIの二か所を同期する義務が生まれる。
