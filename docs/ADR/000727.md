# ADR-000727: Phase 1 ハードニングと Phase 2 pilot を経て東西通信 mTLS を永続デフォルトに昇格する

## Status

Accepted

## Date

2026-04-15

## Affected Services

- alt-backend (Go) — 受信側で `tls.RequireAndVerifyClientCert` + peer allowlist を永続有効化、`PeerIdentityHTTPMiddleware` 追加
- auth-hub (Go) — 受信側 `:9443` の fail-closed / reload-fallback 適用
- pre-processor (Go) — 送信側 mTLS Client、`BACKEND_API_MTLS_URL=https://alt-backend:9443` を既定で使用
- search-indexer (Go) — 送信側 mTLS Client (`DefaultHTTPClient` + `NewClient(.., httpClient)` 署名変更)
- alt-butterfly-facade (Go) — 送信側 `http2.Transport{TLSClientConfig}` を h2c パスと並存
- tag-generator (Python) — `pyqwest.SyncHTTPTransport(tls_cert, tls_key, tls_ca_cert)` による mTLS 対応（opt-in）
- acolyte-orchestrator (Python) — 共通 `httpx.AsyncClient` に `ssl.SSLContext` を注入（opt-in）
- recap-worker (Rust) — **本ADRスコープ外**（reqwest feature flag 変更要、次回）

## Context

[[000725]] で step-ca ベースの PKI と alt-backend 受信側 `:9443` リスナーを投入し、Phase 1 dual-stack 状態を確立した。以後に次の 3 軸で前進した。

**1. Phase 1 ハードニング (L-1 / L-4 / L-5)**

外部レビューで以下のギャップが指摘された（いずれも blocker ではないが Phase 2 以降の blast radius を下げる）。

- **L-5 fail-closed**: `MTLS_LISTEN=true` で cert load が失敗したとき、エラーログだけを出して平文リスナーのみで立ち上がってしまう。サイレント「mTLS 無効」運用に陥る危険。
- **L-1 cached fallback**: `certReloader.load()` が `tls.LoadX509KeyPair` 失敗時に cached cert を返さず handshake を落とす。step-ca renewer のアトミック rename と微小な race した瞬間に listener が瞬断する。
- **L-4 鍵ファイル 0400**: cert-init が 0644 で鍵を書く。コンテナ境界内で十分守られているが、OWASP best practice の 0400/0600 に満たない。

**2. Phase 2 pilot (P2-Required #1 / #2)**

Phase 1 は `ClientAuth=NoClientCert` の「暗号化のみ」。実際の身元強制には 2 点を同時投入する必要がある。

- **ClientAuth 強制化** — `tls.RequireAndVerifyClientCert` を立て、クライアント証明書を要求
- **Peer identity 検証** — 同一 alt-ca 配下の leaf は ClientCAs だけでは全て通ってしまうため、SAN/CN allowlist で「A として振る舞ってよいのは CN=A を提示した peer のみ」を強制しないと CA 内詐称が通る

**3. Phase 3 全展開と永続化**

pilot が成功したあと、残 caller を mTLS に切り替え、最終的に `docker compose up -d` だけで mTLS が常時有効になる「永続」状態にする。起動ごとに inline env を指定する必要がある段階では運用ミスの温床。

## Decision

### Phase 1 ハードニング

- **L-1**: `certReloader.load()` が `os.Stat` / `tls.LoadX509KeyPair` のいずれで失敗しても、`r.cert != nil` のとき（= 初期ロード済み）は直前の cached cert を返す。初回ロード失敗のみエラーを surface する。step-ca renewer が書いている最中の一時的な不整合で listener が落ちることを防ぐ。
- **L-5**: `MTLS_LISTEN=true` と明示しているのに `LoadServerConfig` が失敗したら `os.Exit(1)` / `panic` / `errgroup return` で**プロセス起動を拒否**する。5 Go サービス全てに統一パターンを適用した。
- **L-4**: cert-init が `chown 65532:65532 /certs /certs/svc-cert.pem /certs/svc-key.pem` を行い、`chmod 0444` (cert) / `chmod 0400` (key) に切り詰める。Go サービスは全て distroless nonroot (UID 65532) で走るため、cert-renewer を `user: "65532:65532"` にしても named volume に書き込み可能なことを確認。

### Phase 2 pilot: ClientAuth と peer allowlist の導入

`tlsutil.LoadServerConfig` のシグネチャを破壊せずオプションで拡張した。

```go
cfg := tlsutil.LoadServerConfig(cert, key, ca,
    tlsutil.WithClientAuth(tls.RequireAndVerifyClientCert),
    tlsutil.WithAllowedPeers("pre-processor", "search-indexer", ...),
)
```

- `WithClientAuth(tls.ClientAuthType)` — Phase 1 デフォルトの `NoClientCert` から `RequireAndVerifyClientCert` まで切替可能
- `WithAllowedPeers(names ...string)` — allowlist が非空なら `tls.Config.VerifyConnection` callback を登録し、peer leaf の `Subject.CommonName` と `DNSNames` のいずれかが allowlist に含まれない限り handshake を拒否する。**同一 CA 内の別サービス leaf で詐称できる穴を塞ぐ**
- `OptionsFromEnv()` — `MTLS_CLIENT_AUTH=require_and_verify` と `MTLS_ALLOWED_PEERS=csv` env から options を組み立てるヘルパー。callers は compose から env を流すだけで有効化できる

pilot として `pre-processor → alt-backend` ペアで実トラフィックを通し、alt-backend に新規 `middleware.PeerIdentityHTTPMiddleware` を噛ませて `r.TLS.PeerCertificates[0].Subject.CommonName` を構造化ログに出力することで **T3（呼び出し元不明）の監査ログ gap を閉鎖** した。

### Phase 3 全 caller 展開

同一パターンで送信側クライアントを mTLS 化。各サービスの Clean Architecture 層にあわせて責務分解:

- **search-indexer** `driver/backend_api/client.go`: `NewClient(baseURL, serviceToken, httpClient *http.Client)` に昇格（既存 `buildHTTPClient` は `DefaultHTTPClient` にリネーム公開）。`bootstrap/drivers.go` に `buildBackendHTTPClient` ヘルパを置き、`MTLS_ENFORCE=true` のとき `tlsutil.LoadClientConfig` ベースの transport を注入し `BACKEND_API_URL` を `BACKEND_API_MTLS_URL` に切替
- **alt-butterfly-facade** `main.go`: `newMTLSBackendTransport` が `http2.Transport{TLSClientConfig}` を返す。既存 `NewServerWithTransport` に透過的に差し込む。timeouts=0 (context 駆動) と h2c 並存の制約を保持
- **tag-generator** `driver/connect_client_factory.py`: `pyqwest.SyncHTTPTransport` が `tls_ca_cert` / `tls_cert` / `tls_key` を引数で受けることを確認し、`MTLS_ENFORCE=true` のとき注入
- **acolyte-orchestrator** `main.py`: 共通 `httpx.AsyncClient` に `ssl.SSLContext` を注入（`load_cert_chain` + `minimum_version=TLSv1_3` + CA bundle）。search-indexer と news-creator の両方を見る shared client の性質を維持

### 永続化

compose の env placeholder のデフォルト値を書き換えた。

```yaml
# core.yaml (alt-backend receiver)
- MTLS_CLIENT_AUTH=${ALT_BACKEND_MTLS_CLIENT_AUTH:-require_and_verify}
- MTLS_ALLOWED_PEERS=${ALT_BACKEND_MTLS_ALLOWED_PEERS:-pre-processor,search-indexer,alt-butterfly-facade,auth-hub,tag-generator,acolyte-orchestrator}

# ai.yaml (pre-processor), workers.yaml (search-indexer), bff.yaml (alt-butterfly-facade)
- MTLS_ENFORCE=${<SVC>_MTLS_ENFORCE:-true}
```

Python 2 サービスは現時点で opt-in (`:-false`) を維持する。`acolyte-orchestrator` の shared `httpx.AsyncClient` が news-creator など mTLS 未対応の下流を見ているため、無条件 enforce 化は下流を壊す。フロー切替は caller ごとに段階投入する方針。

## Consequences

### Pros

- **T3（呼び出し元不明）が実トラフィックで閉鎖**: alt-backend に対して実運用中に `peer=pre-processor` / `peer=alt-butterfly-facade` が数千件単位でログに乗ることを確認。監査ログで east-west の caller attribution が可能になった。
- **T1（横展開）が構造的に困難化**: alt-backend は allowlist 上の CN を持つ leaf しか受け入れないため、他コンテナが侵害されても alt-backend への偽装は不可。
- **T5（ローテ all-or-nothing）の完全解消**: 1 サービス単位で `MTLS_ENFORCE` を env で opt-out → 30 秒以内で baseline 復帰、Phase 2 pilot 時に実測済み。
- **永続化**: `.env` にも inline env にも依存せず `docker compose up -d` で mTLS enforce 状態で起動。再起動・ホスト再起動のいずれでも状態を失わない。
- **fail-closed**: 設定不備での silent mTLS 無効化を構造的に排除。`MTLS_LISTEN=true` なのに cert が読めなければプロセスが立ち上がらない。
- **cached fallback**: renewer との race による瞬断リスクを排除。既存接続・新規 handshake の両方で irrecoverable なエラーにならない。
- **リリースクオリティ**: `alt-backend/app/tlsutil/` に 12 テスト (`-race` クリーン) — allowlist accept/reject、TLS 1.3 強制、mtime 再読込、cached fallback、`OptionsFromEnv` パースまで網羅。

### Cons / Tradeoffs

- **Python 2 サービスは非永続**: tag-generator と acolyte-orchestrator は opt-in。構造的に shared client が mTLS 未対応の下流と混在しているため、フラグ切替時は下流の mTLS 整備と同期が必要。
- **recap-worker (Rust) スコープ外**: 本ADR では送信側 mTLS 化を保留した。`reqwest = { features = ["rustls-tls-manual-roots"] }` への feature flag 変更と `Identity::from_pem` / `Certificate::from_pem` ベースの client builder 再構築が必要で、範囲としてもう一段広く、次回ADRで扱う。
- **allowlist に全 peer を含めた構成**: 現在の persistent state は 6 peer 全てを accepted にしているため、「別 peer の leaf で alt-backend を騙す」という rogue テストが compose 内では実演できない（unit test `TestLoadServerConfig_WithAllowedPeers_RejectsUnlisted` で検証済）。
- **tls.Config.VerifyConnection の CN-based 実装**: 将来 SPIFFE URI SAN (`spiffe://alt.local/ns/core/sa/<service>`) に移行する場合は step-ca の template 変更と allowlist フォーマットの再設計が必要。短期的には運用コストの低い CN/DNS allowlist を採用。
- **cert volume の既発行分の所有権**: L-4 の `chown 65532:65532` は cert-init を再実行したサービス（Go 5 つ）で反映済。tag-generator / acolyte-orchestrator の既発行 cert は旧 root 所有のまま（nginx サイドカー経由で読むので実害なし。volume 削除＋再起動で新所有権に揃う）。

### 検証状況

- `alt-backend/app/tlsutil/` 全 12 テスト `-race` pass (canonical)。他 4 Go サービスの smoke test pass。
- 全 7 対象サービス（alt-backend, auth-hub, pre-processor, search-indexer, alt-butterfly-facade, tag-generator, acolyte-orchestrator）が healthy で ~10 分以上稼働。
- 実トラフィック: pre-processor から alt-backend への Connect-RPC 2948 件以上が mTLS 経由で成功、peer identity が構造化ログに出力される。
- Phase 2 pilot 時のネガティブテスト（証明書なし→`exit=55`、allowlist 外の CN→TLS alert `exit=43`）で挙動確認済。
- 公開 dashboard (`nginx:80`) → alt-butterfly-facade → alt-backend chain が `http 200` で応答。BFF→backend が mTLS 経由になっても外部 UX は不変。
