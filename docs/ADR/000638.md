# ADR-000638: 3day Recap の prompt を Gemma 互換 turn format へ統一し strict validation と deterministic fallback を導入する

## Status

Accepted

## Date

2026-04-07

## Affected Services

- news-creator (Python / FastAPI) — Recap summary prompt 生成、structured output 検証、fallback 生成

## Context

3-day Recap の実運用で、LLM が JSON 契約を満たさない出力を返し続ける事象が確認された。

具体的には以下の問題が重なっていた。

1. **3day 用 prompt は日本語指示を含むが、Gemma の turn format に統一されていなかった**
   `raw=True` で Ollama に渡している一方で、Recap 用 Jinja prompt は Gemma 系の `<start_of_turn>` / `<end_of_turn>` 形式に揃っていなかった。
2. **階層要約の中間経路で 3day contract が落ちていた**
   3day の最終要約だけ専用テンプレートを使い、中間要約や reduce 群は 7day 相当のゆるい contract に逃げていた。
3. **LLM 失敗時の fallback が日本語契約を壊していた**
   失敗時に cluster representative をそのまま返す経路があり、英語文や placeholder に近い短文が最終成果物へ混ざっていた。

これにより、`language="ja"` を明示していても、実出力では title が英語、bullets が placeholder、あるいは日本語比率が低い状態が残り、3day Recap の品質基準を満たせなかった。

## Decision

**3day Recap は Gemma 互換の turn format でプロンプトを統一し、map / reduce / final の全段階で 3day contract を適用し、失敗時は deterministic な日本語 fallback に落とす。**

### 1. Gemma 互換 turn format に統一する

Recap summary の prompt は `<start_of_turn>system` / `<start_of_turn>user` / `<start_of_turn>model` で囲み、Ollama の `raw=True` と整合する形に統一する。

これにより、Gemma4 の instruction-following を prompt 側で明示し、structured output の安定性を上げる。

### 2. 3day contract を階層要約の全段階に適用する

`window_days <= 3` のときは、single-shot だけでなく hierarchical の map / reduce / final reduce も 3day contract を使う。

3day では intermediate summary も日本語 JSON で返し、placeholder、英語 title、引用不整合を検出したら失敗扱いにする。

### 3. strict validation を導入する

3day の最終出力は以下を必須にする。

- `language == "ja"`
- `title` と `bullets` が日本語であること
- placeholder bullet がないこと
- 1 bullet につき少なくとも 1 つの出典マーカーがあること
- cited `[n]` が `references` に存在すること
- bullet が極端に短い stub ではないこと

### 4. deterministic fallback に一本化する

LLM が contract を満たさない場合、英語や placeholder をそのまま返す代わりに、日本語の deterministic extractive fallback へ落とす。

fallback は代表文を単純転記せず、短い日本語の文に整形し、`language="ja"` と `is_degraded=true` を維持する。

## Consequences

### Pros

- 3day Recap の最終成果物が `language=ja` を実際に満たしやすくなる
- Gemma4 の prompt formatting を他の日本語生成経路と揃えられる
- hierarchy の中間段階も含めて contract が一貫し、7day への意図しないフォールバックを防げる
- placeholder や英語 title を最終出力へ残さず、品質不良を deterministic fallback に隔離できる
- 回帰テストで prompt 形式、strict validation、degraded fallback を固定できる

### Cons / Tradeoffs

- strict validation により、以前は「とりあえず出ていた」出力が fallback に置き換わるため、degraded 件数は増える可能性がある
- Gemma 互換 turn format を入れることで prompt がやや長くなる
- fallback を deterministic にするため、LLM が失敗した際の表現の自由度は下がる
- `raw=True` を前提にするため、prompt builder 側の責務が増える
