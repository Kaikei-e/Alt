# ADR-000666: Planner を section contract 付き半決定論パターンに拡張し下流ノードを contract 駆動にする

## Status

Accepted

## Date

2026-04-10

## Affected Services

- acolyte-orchestrator (Python 3.14+)

## Context

exec1.md Issue 5 の実装。Issues 1-4（[[000659]] SectionPlannerNode 追加、[[000661]] section_citations 永続化、[[000662]] Conclusion 分離、[[000664]] ES 再合成）により composition 層の基盤は整ったが、Planner の出力が `key/title/search_queries/section_role` の 4 フィールドに限られており、下流ノード（SectionPlanner, Writer, Critic）が section 間の役割差・制約を理解できない状態だった。

具体的な問題:
1. **Writer が section title で制御されている** — Conclusion/ES はプロンプトテンプレートの分岐で制御されるが、novelty_against（どのセクションと重複してはいけないか）は暗黙的
2. **Critic が citation 最低数を検証できない** — min_citations のような contract がないため、citation が 0 件でも FM として検出されない
3. **SectionPlanner のクレーム数がハードコード** — "3-7 claims" / "3-5 claims" が全セクション共通で、section_role による差分がない

### 設計判断の分岐: LLM 生成 vs テンプレート

exec1.md は当初、contract フィールド（`must_include_data_types`, `min_citations`, `novelty_against`, `max_claims`, `synthesis_only` 等）を **LLM に生成させる** 設計を想定していた。しかし、resolve ドキュメント（Acolyte 安定化方針）が以下を指摘:

1. **structured output × thinking model は本質的に不安定** — `num_predict` 増加で押し切る方向はやめるべき
2. **Planner は"半決定論"に寄せるべき** — `report_type` ごとの固定テンプレートを正とし、LLM には query expansion だけさせる
3. [[000665]] でも Planner は `num_predict=2048` でフォールバックが残る → スキーマ拡張は逆効果

さらに Web 調査で Ollama #15260（Gemma4 + `think=false` で structured output が壊れる、OPEN）を確認。LLM のスキーマ負荷を増やすリスクは高い。

## Decision

**Template-First Section Contract パターンを採用する。**

contract フィールドは LLM に生成させず、`ROLE_CONTRACT_DEFAULTS` テンプレート（Python dict）で定義し、Planner の後処理でマージする。LLM の出力スキーマは今と同じ `key/title/search_queries/section_role` のまま変更しない。

### 1. Domain Model: `SectionContract` + `PlannerOutput`

`acolyte/domain/section_contract.py` を新規作成:

- **`SectionContract`** (Pydantic BaseModel) — section の振る舞いを定義する full contract。`key, title, search_queries, section_role` に加え、`must_include_data_types, min_citations, novelty_against, max_claims, query_facets, synthesis_only` を持つ。全新規フィールドにデフォルト値あり
- **`PlannerSection`** (Pydantic BaseModel) — LLM が生成する軽量スキーマ。`key, title, search_queries, section_role` のみ。contract フィールドを含まない
- **`PlannerOutput`** (Pydantic BaseModel) — `reasoning` + `sections: list[PlannerSection]`。reasoning-first field order（[[000632]] 準拠）
- **`ROLE_CONTRACT_DEFAULTS`** — `section_role` → contract defaults のマッピング dict

テンプレートデフォルト:

| section_role | synthesis_only | min_citations | max_claims | must_include_data_types |
|---|---|---|---|---|
| analysis | False | 2 | 7 | ["statistic", "quote"] |
| conclusion | True | 1 | 5 | [] |
| executive_summary | True | 2 | 3 | ["statistic"] |
| general | False | 1 | 7 | [] |

### 2. PlannerNode: `generate_validated` 移行 + テンプレートマージ

- `_PLANNER_FORMAT` raw JSON schema dict を削除し、`generate_validated(llm, prompt, PlannerOutput, ...)` に移行（ExtractorNode, SectionPlannerNode と同じパターン）
- LLM 出力後に `_enrich_with_contract()` で `ROLE_CONTRACT_DEFAULTS` をマージ
- `novelty_against` を動的解決:
  - conclusion → analysis section keys
  - executive_summary → 全非 ES section keys
- `PLANNER_PROMPT` は section_role 指示のみ（新フィールドの生成指示は追加しない）

### 3. SectionPlannerNode: contract フィールド読み取り

outline dict から `max_claims`, `synthesis_only`, `novelty_against`, `must_include_data_types` を読み、プロンプトに注入。ハードコード "3-7 claims" を `max_claims` で置換。

### 4. WriterNode: contract フィールド注入

`_build_contract_instructions()` で contract フィールドからプロンプト指示を生成し、全プロンプトテンプレートに追加。Writer は section title ではなく contract フィールドで制御される。

### 5. CriticNode: FM9/FM10 追加

- **FM9 (`insufficient_citations`)**: `min_citations` 未達の section を warning 検出
- **FM10 (`novelty_violation`)**: `novelty_against` 対象との bigram Jaccard overlap を検出（FM8 の汎用版）
- FM8 は backward compat で残し、FM10 が contract 駆動の汎用版として共存

## Consequences

### Pros

- **LLM スキーマ負荷ゼロ増加** — Planner の JSON schema は今と同じ 4 フィールド。Gemma4 の structured output 不安定性（[[000665]], Ollama #15260）を回避
- **contract フィールドは 100% 信頼可能** — コード定義のテンプレートなので GBNF grammar 破綻リスクなし
- **resolve 方針準拠** — 半決定論パターンで Planner を安定化
- **exec1.md 完了条件達成** — Writer/Critic が `section_role` と `novelty_against` を読んで動作
- **汎用 novelty 検出** — FM10 は任意のセクション間の重複を検出可能（FM8 の hardcoded conclusion-analysis 限定から拡張）
- **generate_validated 統一** — 全 structured output ノード（Planner, Extractor, SectionPlanner, Critic）が同じパターンに

### Cons / Tradeoffs

- **テンプレートの硬直性** — `ROLE_CONTRACT_DEFAULTS` は `section_role` ベースの固定値。`report_type` 別カスタマイズが必要になった場合はテンプレート拡張が要る
- **FM8/FM10 の重複** — FM8（conclusion-analysis 専用）と FM10（contract 駆動汎用）が共存。将来 FM8 を deprecated にして FM10 に統合する余地あり
- **LLM が contract を学習しない** — テンプレートが section_role から決定的に決まるため、LLM がトピック固有の contract を提案する余地がない。Issue 6 以降で report_type 別テンプレートや LLM による query_facets 生成で補完予定
