# ADR-000772: recap-worker に Hurl E2E スイートを追加し段階的全面展開の第 6 サービスとする

## Status

Accepted

## Date

2026-04-18

## Affected Services

- recap-worker (Rust) — プロダクションコードは一切触らず、E2E スイート 7 シナリオ (readiness + health schema + metrics envelope + empty-DB 404 + trigger validation + 7-day trigger→poll + 3-day trigger→poll) と staging compose 片肺 (`recap-db` + `recap-db-migrator` + `recap-pipeline-stub` + `recap-worker`) を追加する。
- recap-migration-atlas — ランタイム無変更。staging `recap-db-migrator` サービスが既存 `docker/Dockerfile` を build して適用する。
- e2e/stubs/recap-pipeline-stub — 新規 FastAPI 1 ファイル (`main.py`) + Dockerfile。`alt-staging` network 上で 4 hostname alias (`rw-stub-subworker`, `rw-stub-news-creator`, `rw-stub-alt-backend`, `rw-stub-tag-generator`) を引き受け、schema-valid な canned response を path-route で返す。

## Context

[[000763]] で Hurl E2E フレームワークを Increment 1 として `search-indexer` に導入、[[000764]] で `mq-hub`、[[000765]] で `knowledge-sovereign`、[[000767]] で `tag-generator`、[[000768]] 〜 [[000771]] で `news-creator` を 4 フェーズに分けて展開してきた。次の対象 `recap-worker` は、これまでの 5 サービスとは別個のサーフェスを 6 点持つ。

1. **依存上流が 4 サービス同時**: recap-worker の pipeline は 1 trigger に対して `alt-backend` Connect-RPC (ListRecapArticles) → `recap-subworker` REST (classify-coarse / classify-runs / clustering / admin poll) → `news-creator` REST (summary/generate / morning-letter) → `tag-generator` REST (/api/v1/tags/batch) の 4 サービスを順に叩く。プロダクションではそれぞれ GPU / SBERT / mTLS / FastAPI 実装だが、**CI で 4 本立てるのは現実的でない** (news-creator と recap-subworker は GPU bound)。
2. **レスポンスが 4 種類の codec を混在**: alt-backend は Connect-RPC JSON wire (camelCase, `clients/alt_backend.rs:14-78`)、subworker は snake_case JSON + JSON Schema 検証 (`CLUSTERING_RESPONSE_SCHEMA`, `schema/subworker.rs`)、news-creator は snake_case JSON + JSON Schema 検証 (`SUMMARY_RESPONSE_SCHEMA`, `schema/news_creator.rs`)、tag-generator は FastAPI snake_case (`BatchTagsResponse { success, tags }`)。[[000767]] で確立した codec 3 パターンに加えて 4 つ目。
3. **Status 文字列が厳密**: subworker の classification は `"running"` / `"succeeded"` のみ (`classification.rs:256-270`, `"completed"` は `non-success` 扱い)、clustering は `"running"` / `"succeeded"` / `"partial"` / `"failed"` (`ClusterJobStatus` enum, `types.rs:54-61`)、admin job poll は `"succeeded"` / `"partial"` で成功 (`admin.rs:207-218`)。nginx の "200 を返すだけ" では通らない。
4. **POST/GET URL 形状が非直交**: admin job は kick URL (`POST /admin/graph-jobs`) と poll URL (`GET /admin/graph-jobs/{job_id}`) を同じ prefix で持つ (`admin.rs:155-160` の `{endpoint}/{job_id}`)。一方 clustering は kick (`POST /v1/runs`) と poll (`GET /v1/runs/{run_id}`) で path 階層が階段になる (`clustering.rs:481-494`)。stub はこれらを path 別に route 分岐する必要がある。
5. **JSON Schema validation が失敗原因を握り潰す**: `schema.rs:41-51` の `validate_json` は `Err` 時に全エラーを `"Validation failed"` という単一文字列に丸める。scenario から見える情報が「validation failed」しか無く、stub の微妙な誤差を追うのに iterate コストがかかる。最終的に subworker の `schema_accepts_valid_response` テスト (`schema/subworker.rs:117-143`) の payload 形に合わせる形で収束させる。
6. **DB が `pg_cron` 必須**: migration `20260101000000_add_data_retention.sql` が `CREATE EXTENSION IF NOT EXISTS pg_cron` を含む。`postgres:17-alpine` の素イメージでは `pg_cron is not available` で migrator が exit 1 になる。production は `docker/postgres/Dockerfile.recap-db` (`postgres:18-bookworm` + `postgresql-18-cron` apt) で解決済み。

加えて、既存の `alt-staging` network 上では他 profile (news-creator, tag-generator) が service 名 `news-creator` / `tag-generator` を占有する。recap-pipeline-stub が同名 alias を取ると DNS 競合で非決定的になるため、ホスト名の空間を分ける必要がある。

また、recap-worker 自身の `/metrics` 実装には既存バグがある: `observability.rs:57-60` の `render_prometheus()` は `prometheus::gather()` をデフォルトレジストリに対して呼ぶが、`observability/metrics.rs:68-175` は counter を `Arc<Registry>` に `register_counter_with_registry!` している。結果として `/metrics` は常に空ボディを返す。スイートはこれを修正する権限を持たない (本 PR のスコープは E2E 追加のみ) ため、scenario の envelope-only 観測に留める必要がある。

## Decision

**recap-worker に対する Hurl E2E スイートを 7 シナリオで追加する。4 上流サービスは 1 コンテナの `recap-pipeline-stub` が Docker network alias `rw-stub-subworker` / `rw-stub-news-creator` / `rw-stub-alt-backend` / `rw-stub-tag-generator` を通じて schema-valid な canned response を返す構成にする。scenario 05 / 06 は POST 202 → GET retry poll の形で `recap_outputs` 行が書かれるまでを観測する。`/metrics` は envelope のみ pin し、registry binding bug の修正は別 follow-up に分離する。`--jobs 1` で 03 (empty-DB 404) を 05/06 (書き込み) より先に走らせ、DB state を決定的に保つ。mTLS on 経路 / `recap-evaluator` 方向 / morning-letter regenerate / dashboard / evaluation は別 increment に繰り延べる。**

### 採用する構成

1. **Hurl スイート (`e2e/hurl/recap-worker/`)**: 7 ファイル + `run.sh` + `README.md`。
   - `00-setup.hurl` — `:9005/health/live` を retry 30 × 500 ms で探る readiness probe (依存なし、`health.rs:55-61`)。
   - `01-health-schema.hurl` — `/health/live` + `/health/ready` の 2 probes を pin。`/health/ready` は stub の `rw-stub-subworker:/health` と `rw-stub-news-creator:/health` にヒットして 200 を返す。
   - `02-metrics-shape.hurl` — `GET /metrics` に対して `HTTP 200` + `Content-Type: text/plain` のみ assert。body は registry binding bug の影響で空。
   - `03-recaps-empty.hurl` — `GET /v1/recaps/7days` と `/v1/recaps/3days` が空 DB で `HTTP 404` + `$.error == "No <label> recap found"` (`fetch.rs:407-418`)。
   - `04-trigger-validation.hurl` — `POST /v1/generate/recaps/7days` に全空文字列の `genres` を与え `HTTP 400` + `$.error == "genres array must include at least one non-empty value"` (`generate.rs:53-58`)。
   - `05-trigger-and-poll-7days.hurl` — `POST /v1/generate/recaps/7days` → `HTTP 202` + job_id (UUID regex) を `[Captures]`、続いて `GET /v1/recaps/7days` を `retry: 60 retry-interval: 1000` で poll。instant stub なので実測 1 秒前後で `HTTP 200` に遷移。
   - `06-trigger-and-poll-3days.hurl` — 同形で 3-day window。
2. **staging compose (`compose/compose.staging.yaml`)**: 新 profile `recap-worker` を 4 サービスで構成。
   - `recap-db` — `docker/postgres/Dockerfile.recap-db` を staging が build (production 同一 Dockerfile で `postgres:18-bookworm` + `postgresql-18-cron`)。`command: postgres -c shared_preload_libraries=pg_cron -c cron.database_name=recap` で extension を boot 時に load。
   - `recap-db-migrator` — `recap-migration-atlas/docker/Dockerfile` を build し `command: ["apply"]`。`DATABASE_URL` を plain env で渡し、`restart: "no"`。recap-worker の `depends_on.service_completed_successfully` で gate する。
   - `recap-pipeline-stub` — 新規 FastAPI (`e2e/stubs/recap-pipeline-stub/main.py`)。`networks.alt-staging.aliases: [rw-stub-subworker, rw-stub-news-creator, rw-stub-alt-backend, rw-stub-tag-generator]` で 1 コンテナが 4 ホスト名に応答する。path-route は `/health` (全 alias 共通)、`/services.backend.v1.BackendInternalService/ListRecapArticles`、`/v1/classify/coarse`、`/v1/classify-runs` + `/v1/classify-runs/{run_id}`、`/v1/runs` + `/v1/runs/{run_id}`、`/admin/graph-jobs` + `/admin/graph-jobs/{job_id}` 他、`/v1/extract`、`/v1/summary/generate` + `/v1/summary/generate/batch`、`/v1/morning-letter/generate`、`/api/v1/tags/batch`、`/api/v1/extract-tags`、catch-all の順。response shape は `schema/subworker.rs::CLUSTERING_RESPONSE_SCHEMA` の `schema_accepts_valid_response` test の payload に準拠。
   - `recap-worker` — `ghcr.io/${GHCR_OWNER}/alt-recap-worker:${IMAGE_TAG}` を pull。`MTLS_ENFORCE=false`, `PEER_IDENTITY_TRUSTED=off`, `NEWS_CREATOR_BASE_URL=http://rw-stub-news-creator/` 等で 4 上流を全て stub に向ける。`HUGGING_FACE_TOKEN_PATH` は staging で意味のないパスを渡し、rust-bert の embedding service は起動時に `Embedding service failed to initialize ... Falling back to keyword-only filtering` で fallback する (非 fatal)。
3. **run.sh (`e2e/hurl/recap-worker/run.sh`)**: [[000766]] の dispatch 規約通り `IMAGE_TAG` / `GHCR_OWNER` / `HURL_IMAGE` / `RUN_ID` / `KEEP_STACK` env を受理。cleanup の `down -v --remove-orphans` に **`--profile recap-worker` を明示的に付与**する (profile scope 外の anonymous postgres volume が残って次回 run で DB state が汚れる問題への直接対策)。`--jobs 1` で scenario 順序を pin。
4. **CI (`.github/workflows/e2e-hurl.yml`)**: path filter に `recap-worker/**`, `recap-migration-atlas/**`, `e2e/hurl/recap-worker/**`, `e2e/fixtures/recap-worker/**`, `e2e/stubs/recap-pipeline-stub/**` を追加。`recap-worker` job は `alt-recap-worker:ci` と `alt-recap-pipeline-stub:ci` を local build してから `run.sh` を呼ぶ。他 4 suite と同様 `KEEP_STACK=1` + `failure: logs --tail=300` + `artifacts: e2e/reports/recap-worker-*` でデバッグ経路を揃える。

### 検討した代替案

**代替案 A: 4 サービスをそれぞれ独立 stub container に分ける**。isolation は良いが image / compose の maintenance cost が 4 倍になる。recap-pipeline-stub は 1 プロセスで path-route すればよいので、`networks.aliases` で 1 コンテナに multi-hostname を持たせる方が image / resource が節約できる。却下。

**代替案 B: nginx + mock JSON レスポンス (既存の `stub-backend` 流用)**。`/admin/graph-jobs` の kick response が固定値なら可能だが、URL parameter (`run_id`, `job_id`) を echo したり `CLUSTERING_RESPONSE_SCHEMA` の `minLength: 20` を満たす `sentence_text` を組み立てたりするには不足。正しい response を作るには Lua / OpenResty / Wiremock 等の拡張が必要で、それなら FastAPI の方が保守しやすい。却下。

**代替案 C: scenario 05 / 06 を trigger-only (HTTP 202 + job_id の assert のみ) にして poll を省略**。stub を作らなくて済む (async pipeline が失敗しても test 側は検知しない) が、**Phase 6 の価値は「4 上流を経由した pipeline completion を観測する」ことそのもの**なので、それを外すなら Phase 6 を作る意義が薄い。却下。

**代替案 D: production の `recap-subworker` / `news-creator` を staging で起動**。SBERT モデル load + GPU 前提なので CI runner で不可能。仮に CPU fallback で動かしても SBERT 初 mmap で 10 秒以上、LLM summarization で 30 秒以上かかり suite の wall-clock が桁違いになる。却下。

**代替案 E: `recap-db` に `postgres:17-alpine` + alpine で pg_cron を install**。alpine の community repo に `pg_cron` パッケージがなく build chain を自前で組む必要がある。production と同一の Dockerfile (`docker/postgres/Dockerfile.recap-db`) を再利用する方が parity が高い。却下。

**代替案 F: `/metrics` 02 で `# HELP` / `# TYPE` を assert する**。現状は registry binding bug (`observability.rs:59` の `prometheus::gather()` がデフォルトレジストリを叩く vs `metrics.rs` が custom `Arc<Registry>` に register) で body が空なので通らない。envelope のみ assert + scenario comment に follow-up 明記で妥協。bug 修正は別 PR で行う。

### 不変条件との整合

- TDD first: スイート → 失敗確認 → stub 配線 → 緑化の順で進められる構成になっている ([[000763]] 不変条件)。
- reproject-safe / append-first 等 Knowledge Home 不変条件: recap-worker は独自 DB (`recap-db`) を持ち `knowledge_events` / `summary_versions` を touch しないため該当しない。
- [[000591]] Pact CDC との棲み分け: recap-worker の consumer pact (`recap-worker/recap-worker/src/clients/{alt_backend,subworker,news_creator,tag_generator}_contract.rs`) は形状 (shape) を既に守っている。Hurl は actual round-trip の存在を pin する。両者は補完関係。
- `rw-stub-*` alias prefix は他 profile (news-creator, tag-generator) と衝突しない。`docker compose --profile news-creator` と `--profile recap-worker` を同じ `alt-staging` network に重ねても DNS resolution が決定的に保たれる。

## Consequences

### Pros

- `POST /v1/generate/recaps/{7,3}days` → 4 上流 round-trip → `recap_outputs` row → `GET /v1/recaps/{7,3}days` 200 を 1 PR 単位で pin できる。[[000767]] の tag-generator Phase 1 は 1 upstream / 1 round-trip だったが、本 ADR で 4 upstream / 多段 pipeline を扱える template が揃う。
- 1 container / 4 network alias 方式により、4 独立 stub より image が 75% 少なく、cold CI build の Python layer 再利用も効く。
- `rw-stub-*` prefix で network 空間が disjoint になり、他 profile と同時起動しても DNS 競合しない。同一 staging 上で複数 suite を並行 debug する運用が可能。
- [[000766]] の `run.sh` 規約に乗せたので既存 4 suite と env / artifact path / log dump 動線が揃う。learning cost がない。
- JSON Schema validation を通す canned response を `schema/subworker.rs::schema_accepts_valid_response` test 値に揃えたので、Rust 側の schema が変わったときに test が先に落ちる → stub を追従修正すればよい、という update loop が形成される。
- `--profile recap-worker` を `down -v` に明示した修正は、他 suite の anonymous volume 残留問題にも将来応用できる (tag-generator / news-creator も同じ pattern で cleanup 漏れがあれば追加)。

### Cons / Tradeoffs

- **stub fidelity drift のリスクが他 suite より大きい**。4 upstream × 十数 endpoint を 1 ファイルで管理しており、Rust 側の type (e.g. `AdminJobStatusResponse`, `ClassificationJobResponse`) に field が追加されると stub も手で追う必要がある。mitigation: drift は次回 hurl run が validation error で即失敗するので遅くとも PR 時に検知される。
- **cold CI build が ~10-20 分**。recap-worker の Dockerfile は libtorch (~600MB download) + cargo-chef cook を含むため。mitigation: path filter で recap-worker 以外の PR は触らない。将来 remote cargo-chef cache を入れる余地。
- **`/metrics` が envelope しか assert しない**。scenario 02 は `recap_worker_*` counter の欠落を catch できない。`observability.rs:59` の registry binding bug が修正されるまで assertion は絞っておく (別 ADR で fix 予定)。
- **`pg_cron` 拡張のために `postgres:18-bookworm` を staging も pull**。alpine (~10MB) → bookworm (~150MB) に増える。他 suite (`postgres:16-alpine` / `postgres:17-alpine`) と image が統一されない。parity の観点では production と揃う方が望ましい。
- **`/health/ready` は stub に依存**。stub が unhealthy になると recap-worker も degraded を報告する。実運用では news-creator / subworker が落ちたときに readiness も落ちるのが正なので、staging でも同じ couple を再現している形にはなっているが、失敗時のログが 2 か所に分散する。
- **rust-bert の embedding service が staging で fallback pass する**。`HUGGING_FACE_TOKEN_PATH` に存在しないパスを指定しているため `paraphrase-multilingual-MiniLM-L12-v2` の load が `Permission denied` で失敗し、pipeline は keyword-only 経路に落ちる。これは scenario が期待する挙動だが、将来 embedding の assertion を足そうとすると HF token + model cache の配備が必要になる。
- **scenario 03 (empty-DB 404) と scenario 05 (DB write) の順序依存**。`--jobs 1` で pin しているが、将来 scenario を足す際には 03 より前の位置に DB write を仕込まない運用規約が要る (README に明記済)。
