# ADR-000720: pre-processor の Connect-RPC (port 9202) に X-Service-Token 認証を必須化する

## Status

Accepted

## Date

2026-04-14

## Affected Services

- pre-processor (Go)
- alt-backend (Go)

## Context

[[000719]] で pre-processor の REST (`:9200`) には `X-Service-Token` 認証を必須化したが、**Connect-RPC (`:9202`)** は依然として `// no auth interceptor for internal service communication` と明示コメントのまま無防備だった。`connect/v2/server.go` の `PreProcessorService` には `Summarize` / `StreamSummarize` / `QueueSummarize` / `GetSummarizeStatus` の 4 RPC が登録されており、REST と同じ LLM 処理を叩けるので脅威モデルは REST と等価である。

- OWASP Top 10 2025 A01 (Broken Access Control) に該当する High 深刻度
- REST 側だけ保護し RPC 側を放置すると、compose 誤設定やサイドカーの誤配線が起きた瞬間に LLM 計算リソースを外部が叩ける二重構成のままになる
- pre-processor → alt-backend 呼び出し ([[000717]]) と alt-backend → pre-processor REST 呼び出し ([[000719]]) の両方向が整った今、最後に残った穴が Connect-RPC だった

同時に、alt-backend の `ConnectPreProcessorClient` は Connect-RPC で pre-processor を呼ぶ唯一のクライアントであり、こちらも `X-Service-Token` を送っていなかったため、サーバー側に interceptor を入れるだけでは pre-processor → alt-backend → pre-processor の内部パスが即 401 で壊れる。同一コミットでクライアントも対処する必要があった（rag-orchestrator は生成コードは持つが実際には呼んでいないので対処不要と確認済み）。

## Decision

以下をまとめて採用する。

1. pre-processor に Connect-RPC 用 `interceptor.ServiceAuthInterceptor` を新設する。REST 側 `ServiceAuthMiddleware` と同じ契約（`crypto/subtle.ConstantTimeCompare` による定時間比較、`SERVICE_SECRET` / `SERVICE_SECRET_FILE` 双方対応、空シークレット時は fail-closed）を採り、Unary / StreamingHandler の両方で `X-Service-Token` を検証する。missing/invalid は `connect.CodeUnauthenticated`、シークレット未設定時は `connect.CodeInternal` を返す。
2. `connect/v2/server.go` の `preprocessorv2connect.NewPreProcessorServiceHandler` に `connect.WithInterceptors(serviceAuth)` を適用し、全 RPC を保護する。`/health` は mux に直接登録しているため interceptor の対象外で、liveness probe は引き続き通る。
3. alt-backend の `ConnectPreProcessorClient` に `serviceTokenTransport`（`http.RoundTripper` 実装）を挿入し、毎リクエストの `X-Service-Token` 付与を RoundTripper 層で一元化する。将来新しい RPC が増えても自動的に header が付く。
4. DI コンテナ (`alt-backend/app/di/container.go`) で `infra.Config.InternalAPI.ServiceSecret` を `NewConnectPreProcessorClient` に渡す。REST 呼び出し ([[000719]]) と同じ shared secret を使う。

## 実装の概要

### pre-processor

- `pre-processor/app/connect/v2/interceptor/service_auth.go` 新設
  - `WrapUnary` / `WrapStreamingHandler` で検証、`WrapStreamingClient` はパススルー
  - 失敗時はプロシージャ名をログに残す（tracing から突き合わせやすくする）
- `pre-processor/app/connect/v2/server.go`
  - `loadServiceSecret(logger)` ヘルパで `SERVICE_SECRET_FILE` → `SERVICE_SECRET` の優先順で読み込み
  - `NewPreProcessorServiceHandler(handler, connect.WithInterceptors(serviceAuth))` で配線
- テスト (RED → GREEN) 7 件: missing/wrong/differing-length/correct/unconfigured/whitespace-only/interface-compliance

### alt-backend

- `alt-backend/app/driver/preprocessor_connect/client.go`
  - `serviceTokenTransport` 実装と `newServiceTokenClient(secret)` ヘルパを追加
  - `NewConnectPreProcessorClient(baseURL, serviceSecret string)` にシグネチャ変更
  - 空シークレット時は `http.DefaultClient` にフォールバック（header を付けない）
- `alt-backend/app/di/container.go`
  - `NewConnectPreProcessorClient(infra.Config.PreProcessor.ConnectURL, infra.Config.InternalAPI.ServiceSecret)` に更新
- テスト 4 件: Summarize / QueueSummarize / GetSummarizeStatus で `X-Service-Token` が付くこと、空シークレットでは付かないこと

### End-to-End 検証

コンテナ再ビルド後に以下を確認した。

| ケース | 結果 |
|---|---|
| `GET :9202/health` | 200 |
| Connect-RPC `Summarize` (no token) | 401 `missing X-Service-Token` |
| Connect-RPC `Summarize` (wrong token) | 401 `invalid X-Service-Token` |
| Connect-RPC `Summarize` (correct token) | 400（handler の入力バリデーションに到達、auth は通過） |

pre-processor のログで `connect service auth failed: missing token` と `connect service auth failed: invalid token` が期待どおりプロシージャ名付きで記録されることを確認した。

## Consequences

### Pros

- REST (`:9200`) と Connect-RPC (`:9202`) の両面で同じ認証契約が通るようになり、脅威モデルの非対称が解消した
- RoundTripper 層で `X-Service-Token` を注入するため、alt-backend から新しい pre-processor RPC を呼ぶ際に header の付け忘れが起きない
- 空シークレット時にサーバー側が 500 (`CodeInternal`) を返すことで、本番での shared secret 投入漏れが silent success になるのを防ぐ
- [[000719]] と対になる HIGH-2 対策が完了し、[[plans/partitioned-spinning-whisper]] の HIGH 項目が両方消えた状態になった

### Cons / Tradeoffs

- Connect-RPC で pre-processor を叩く新規サービスが増える場合、同じ RoundTripper パターン（または同等の header 注入）を踏襲する必要がある。search-indexer 側の Connect-RPC クライアントには同種の対応を別 ADR で当てる必要が残っている
- `serviceTokenTransport` は `http.DefaultTransport` をベースにしたラッパなので、将来 hardened transport（MED-1）を入れる際にここも同時に更新する必要がある
- ストリーミング RPC では header は最初のヘッダフレームにしか乗らないため、keep-alive や再接続時の挙動は通常の Connect-RPC クライアントの再接続ロジックに依存する。現状 alt-backend 側は streamAdapter 経由で `StreamSummarize` を使っているが、接続断時の再試行テストは未実装のまま
