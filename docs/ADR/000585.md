# ADR-000585: Projection Snapshot を導入し reproject 高速化とアーカイブの前提条件を整える

## Status

Accepted

## Date

2026-03-25

## Affected Services

- knowledge-sovereign (Go) — snapshot 作成ロジック + admin HTTP エンドポイント
- knowledge-sovereign-db (PostgreSQL 16) — `knowledge_projection_snapshots` テーブル追加

## Context

Knowledge Home の read model（`knowledge_home_items`, `today_digest_view`, `recall_candidate_view`）は `knowledge_events` からの reproject で再構築可能であることが不変条件である（reproject-safe）。しかしイベント数の増加に伴い full reproject の所要時間が線形に増大する。

また、古いイベントを cold archive に退避してディスクを解放するためには、「snapshot 以降のイベントだけで reproject 可能」にする仕組みが必要である。

### 不変条件との関係

**本 ADR の方針: 不変条件を維持する**。古いイベントは cold archive（JSONL.gz on disk）に退避するだけで完全削除はしない。full reproject は `archive restore → replay` で常に可能。Snapshot は「通常運用での reproject を高速化する補助線」であり、イベントの到達可能性は壊さない。

## Decision

### 1. knowledge_projection_snapshots テーブルを導入する

snapshot メタデータを永続化し、schema/projector version 互換性を管理する。

```sql
CREATE TABLE knowledge_projection_snapshots (
  snapshot_id         UUID PRIMARY KEY,
  snapshot_type       TEXT NOT NULL DEFAULT 'full',
  projection_version  INT NOT NULL,
  projector_build_ref TEXT NOT NULL,      -- git sha / image tag
  schema_version      TEXT NOT NULL,      -- migration version
  snapshot_at         TIMESTAMPTZ NOT NULL,
  event_seq_boundary  BIGINT NOT NULL,    -- この seq 以降で差分 replay 可能
  snapshot_data_path  TEXT NOT NULL,
  items_row_count     INT NOT NULL,
  items_checksum      TEXT NOT NULL,       -- SHA-256
  digest_row_count    INT NOT NULL,
  digest_checksum     TEXT NOT NULL,
  recall_row_count    INT NOT NULL,
  recall_checksum     TEXT NOT NULL,
  status              TEXT NOT NULL DEFAULT 'pending'
);
```

各 projection テーブルの行数とチェックサムを個別に記録し、restore 時の検証を可能にする。

### 2. Snapshot 作成ロジック

`POST /admin/snapshots/create`（metrics port の HTTP エンドポイント）で実行:

1. `MAX(event_seq)` を取得（snapshot 境界）
2. 各 projection テーブルを `COPY (SELECT row_to_json(t) FROM table t) TO STDOUT` で JSONL エクスポート
3. gzip 圧縮 + SHA-256 チェックサム計算（`io.MultiWriter` でストリーム処理）
4. `knowledge_projection_snapshots` にメタデータ記録
5. ファイル保存先: `$SNAPSHOT_DIR/snapshot_YYYYMMDD_HHMMSS/`

### 3. Snapshot invalidation 条件

以下の場合に snapshot を `invalidated` に更新:
- projector ロジックの変更（`projector_build_ref` 不一致）
- schema migration の適用（`schema_version` 不一致）
- snapshot `event_seq_boundary` より前の時点への backfill 実行

### 4. ファイル永続化

snapshot ファイルはホスト側 bind mount（`backups/sovereign-snapshots/`）に保存。Docker named volume は nonroot ユーザーの書き込み権限問題があるため bind mount を採用した。

### 初回 snapshot 結果（2026-03-25 実測）

| テーブル | 行数 | ファイルサイズ |
|---------|------|-------------|
| knowledge_home_items | 122,297 | 23.5 MB (gzip) |
| today_digest_view | 11 | 476 B |
| recall_candidate_view | 116 | 5 KB |

snapshot 作成所要時間: ~3 秒（event_seq_boundary: 1,139,408）

## Consequences

### Pros

- 通常の reproject が snapshot + 差分 replay で高速化される（全イベント replay 不要）
- イベントアーカイブの安全な前提条件が整う（snapshot 以降のイベントで reproject 可能）
- projection テーブルごとのチェックサムにより、restore 時のデータ整合性を検証可能
- schema/projector version のトラッキングにより、互換性のない snapshot の誤使用を防止

### Cons / Tradeoffs

- snapshot ファイルのディスク使用量が追加される（122K 行で ~23 MB/snapshot）
- snapshot 作成中は projection テーブルの consistent read が必要（MVCC で実質的に問題なし）
- bind mount のパーミッション管理がホスト環境に依存する
- snapshot からの自動 restore フロー（reproject start → snapshot load → incremental replay）は未実装（altctl CLI 統合として後続で対応）
