# ADR-000474: Knowledge projector を notify と polling fallback のハイブリッドで起動する

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-backend (Go)

## Context

Knowledge Home の projector は一定間隔 polling でのみ起動しており、1 tick あたり 1 batch しか処理していなかった。このため、backlog があると checkpoint の追従が鈍り、dismiss・summary・tag・supersede といった後続イベントの read model 反映がまとめて遅延した。

steady-state でも polling interval がそのまま最短反映時間の下限になり、ユーザー操作や要約更新がすぐに画面へ反映されない。単純に batch size だけを大きくしても、反映開始の待ち時間そのものは残る。

一方で、projector を通知駆動だけに寄せると、通知ロスト・購読再接続・起動順 race に弱くなる。Alt は append-first / checkpoint ベースの再現性を重視しており、wake-up は最適化であって durability source ではない。

## Decision

Knowledge projector を `NOTIFY` による即時 wake-up と checkpoint ベース polling fallback のハイブリッド方式へ変更する。

具体的には以下を採用する。

1. `knowledge_events` append 成功後に projector 用 channel へ通知する
2. projector は通知 payload を信頼せず、必ず checkpoint 以降の event を再取得する
3. projector を scheduler の periodic job から切り離し、専用 runner として常駐させる
4. runner は通知待ちと safety poll を両立し、listener 失敗時は polling のみで継続する
5. 1 wake ごとに複数 batch を drain し、context deadline 近傍まで checkpoint を前進させる

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|--------|------|-----------|
| polling interval だけ短縮する | 実装は簡単 | 反映開始の待ち時間は縮むが、無駄な wake-up が増え、backlog 吸収も限定的 |
| batch size だけ増やす | backlog 吸収には効く | steady-state の待ち時間は改善しない |
| notification のみで projector を動かす | 平常時は速い | 通知ロストや再接続時の durability fallback が弱い |
| outbox 的に別 broker へ載せる | 将来的な拡張性は高い | 現スコープでは過剰で、既存 event store / checkpoint 設計から離れすぎる |

## Consequences

### Pros

- 新規 event 到着時に projector が待たずに起動しやすくなり、read model の追従性が改善する
- polling fallback を残すため、通知断や listener 再接続があっても checkpoint ベースで回復できる
- 1 wake で複数 batch を処理するため、backlog 追従も改善する
- source of truth は引き続き event log と checkpoint にあり、wake-up 最適化が durability を侵食しない

### Cons / Tradeoffs

- projector 起動経路が scheduler 単体より複雑になり、listener の再接続や validation が必要になる
- notification 経路と polling 経路の二重化により、運用観測ポイントが増える
- connection pooling 制約を避けるため、listener 用の接続戦略を別途意識する必要がある
