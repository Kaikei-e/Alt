# ADR-000588: マイクロサービス間 Contract Testing 戦略として Buf + Pact CDC の 2 層構成を採用する

## Status

Accepted

## Date

2026-03-26

## Affected Services

- alt-backend (Go) — Consumer (Connect-RPC)
- pre-processor (Go) — Provider (Connect-RPC) / Consumer (HTTP/REST)
- news-creator (Python) — Provider (HTTP/REST, /api/chat)
- rag-orchestrator (Go) — Consumer (HTTP/REST)

## Context

PM-004（StreamSummarize 5 層障害）、PM-008（Ollama options 不一致によるモデルリロード）で、サービス間の契約が暗黙的に破壊されていたことが判明した。

Alt は 23+ マイクロサービスが 5 言語で書かれ、Connect-RPC、HTTP/REST、Redis Streams の 3 つの通信プロトコルが混在する。既存の Buf lint + breaking change detection（[[000587]] の振り返りで Phase 1 完了を確認）はスキーマレベルの検証に限定されており、ランタイム契約（レスポンスのセマンティクス、options の一致等）はカバーできていなかった。

### 検討した代替案

| ツール | 対象 | Java 依存 | 評価 |
|--------|------|----------|------|
| Pact + pact-protobuf-plugin | HTTP, gRPC, Protobuf | No（Rust FFI） | 採用 |
| Buf Breaking Change Detection | Protobuf schema | No | 採用（既存） |
| Schemathesis | OpenAPI/GraphQL | No | 不採用（Connect-RPC 非対応） |
| Dredd | OpenAPI | No | 不採用（Connect-RPC 非対応） |
| Gauge E2E | ブラウザ→バックエンド | No（Go CLI） | 見送り（スコープ外） |

## Decision

**2 層の Contract Testing 戦略**を採用する。Java 依存はゼロ。

### 層 1: Buf Breaking Change Detection（スキーマ層）

`.proto` ファイルの破壊的変更を PR レベルで自動検知する。`.github/workflows/proto-contract.yaml` で既に稼働中。

**カバー範囲:** フィールド削除、型変更、必須フィールド追加

### 層 2: Pact CDC（ランタイム契約層）

Consumer が期待するリクエスト/レスポンス形式を Pact 契約ファイルとして記録し、Provider が満たしていることを検証する。

**カバー範囲:** レスポンスのセマンティクス、HTTP ステータスコード、ヘッダー、Ollama options の一致

### 全サービスペアの契約マトリクス

| Consumer | Provider | プロトコル | テスト方法 |
|----------|----------|-----------|-----------|
| alt-backend | pre-processor | Connect-RPC | Buf + Pact |
| pre-processor | news-creator | HTTP/REST | Pact |
| rag-orchestrator | news-creator | HTTP/REST (/api/chat) | Pact |
| FE → BFF → Backend | — | Connect-RPC | Buf + 既存 contract tests |

### Pact の依存構成

```
pact-go v2.4.2 (Go) ← Rust FFI 経由、Java 不要
pact-python v3.2.1 (Python) ← Rust FFI 経由、Java 不要
pact-protobuf-plugin (Rust) ← gRPC streaming 対応
Pact Broker (Docker) ← compose/pact.yaml、pact profile で起動
```

## Consequences

### Pros

- PM-004 #1（REST/Connect-RPC 混在）を契約で防止: Connect-RPC パスが Pact で固定される
- PM-008（options 不一致）を契約で検証: `/api/chat` のリクエスト形式が明示化される
- PM-006（セマフォバイパス）を契約で明示: rag-orchestrator が news-creator 経由であることが記録される
- Java 依存ゼロ: pact-go v2 / pact-python v3 は Rust FFI ベース
- 既存の Buf CI と補完関係: スキーマ + ランタイムの 2 層で網羅性向上

### Cons / Tradeoffs

- Pact Broker の運用コスト（Docker コンテナ + PostgreSQL）
- Consumer テストと Provider テストの同期が必要（pact ファイル共有）
- Streaming RPC の Pact サポートは限定的（NDJSON はボディレベルの検証が困難）
- 13 サービスペア全てのカバーには追加実装が必要（Phase 3 として計画）
