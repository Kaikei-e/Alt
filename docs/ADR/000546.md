# ADR-000546: Recall Dismiss を soft-delete に変更し候補の再表示バグを修正する

## Status

Accepted

## Date

2026-03-23

## Affected Services

- knowledge-sovereign (Go) — Dismiss 操作の実装変更、読み取りフィルタ追加、マイグレーション
- alt-backend (Go) — RecallProjectorJob の UPSERT が dismissed 状態を保持することの確認

## Context

Recall 機能で Dismiss したアイテムが60秒後に再表示されるバグが報告された。

### 根本原因

問題は Dismiss 操作と RecallProjectorJob の間の設計不整合にある。

1. `DismissRecallCandidate` は `DELETE FROM recall_candidate_view` を実行していた（hard DELETE）
2. `recall_signals` テーブルは append-only であり、Dismiss 後もシグナルはそのまま残る
3. `RecallProjectorJob` は60秒間隔でシグナルを再スコアリングし、`UpsertRecallCandidate` を無条件に実行する
4. **結果**: Dismiss で行が削除される → 60秒後にプロジェクターが同じシグナルから候補を再生成 → ユーザーに再表示

### 設計意図との乖離

ADR-000408 は「snoozed / dismissed のアイテムはスキップ」と明記しており、IMPL_PHASE4 は Dismiss を「抑制シグナル」として扱い、約30日のクールダウン後に再浮上可能と定義している。しかし実装では Dismiss が hard DELETE であり、抑制の痕跡が残らなかった。

一方、Snooze は `snoozed_until` カラムが存在し、UPSERT の ON CONFLICT 句で保持されるため正しく動作していた。Dismiss だけが壊れていた理由は、対応するカラムが存在しなかったことにある。

## Decision

`recall_candidate_view` に `dismissed_at` カラムを追加し、Dismiss を soft-delete に変更する。Snooze と同じパターンで UPSERT 時に dismissed 状態を保持する。

### 1. マイグレーション

`knowledge-sovereign/migrations/00005_add_recall_dismissed_at.sql` で `dismissed_at TIMESTAMPTZ` カラムを追加。

### 2. Dismiss 操作の変更

`DismissRecallCandidate` を `DELETE` から `UPDATE SET dismissed_at = now()` に変更。候補の行はテーブルに残り、dismissed 状態がプロジェクターの UPSERT を超えて永続化される。

### 3. UPSERT での dismissed_at 保持

`UpsertRecallCandidate` の ON CONFLICT 句は `dismissed_at` を含まない。これにより、プロジェクターが候補を再スコアリングしても `dismissed_at` は上書きされない。Snooze の `snoozed_until` と同じパターンであり、一貫性がある。

### 4. 読み取りフィルタ

`GetRecallCandidates` の WHERE 句に `AND rcv.dismissed_at IS NULL` を追加。dismissed 候補はクエリ結果から除外される。

### 30日クールダウンについて

設計ドキュメント（IMPL_PHASE4）では Dismiss 後約30日で候補が再浮上可能と定義されている。`dismissed_at` カラムの導入により、プロジェクターで `dismissed_at < now() - interval '30 days'` の候補をリセットするロジックを将来追加できる基盤が整った。本 ADR ではカラム追加と soft-delete 化のみを実施し、クールダウンリセットは次のフェーズで実装する。

## Consequences

### Pros

- **Dismiss の永続化**: Dismiss 状態がプロジェクターの60秒サイクルを超えて保持される
- **Snooze との一貫性**: `snoozed_until` と同じ「UPSERT で保持されるカラム」パターンを踏襲
- **append-first 原則の遵守**: シグナルを削除せず、projection レイヤーで抑制を実現
- **クールダウン対応の基盤**: `dismissed_at` のタイムスタンプにより、将来の30日クールダウン実装が容易

### Cons / Tradeoffs

- **行の肥大化**: hard DELETE と異なり dismissed 候補が `recall_candidate_view` に残り続ける。長期的にはクリーンアップジョブが必要になる可能性がある
- **既存 dismissed 候補の復活**: マイグレーション前に DELETE された候補は既に消失しており、retrospective な修復はできない。ただし次のプロジェクター実行でシグナルから再生成されるため、実質的な影響はない
- **インデックス未更新**: 既存のインデックス `idx_recall_candidates_suggest` は `snoozed_until IS NULL` のみでフィルタしており、`dismissed_at IS NULL` を含まない。クエリ量が増加した場合にインデックス拡張が必要になる可能性がある
