# ADR-000530: 全 Python サービスに Pyrefly 型チェッカーを標準導入し mypy を廃止する

## Status

Accepted — 2026-03-22

## Date

2026-03-22

## Affected Services

- tag-generator (Python 3.13)
- metrics (Python 3.13)
- recap-evaluator (Python 3.13)
- recap-subworker (Python 3.12)
- news-creator (Python 3.11)
- dashboard (Python 3.12)
- tts-speaker (Python 3.12)

## Context

### 背景

Alt モノレポには 10 の Python プロジェクトが存在するが、型チェッカーの導入状況はばらばらだった。

| サービス | 移行前の状態 |
|----------|-------------|
| tag-generator | pyrefly 0.42.0 ✅ |
| metrics | pyrefly 0.42.0 ✅ |
| recap-evaluator | pyrefly 0.17.0（旧バージョン、設定キーがアンダースコア形式） |
| recap-subworker | mypy 1.11.0 |
| news-creator | 型チェッカーなし |
| dashboard | 型チェッカーなし |
| tts-speaker | 型チェッカーなし |

tag-generator では [[000050]] で pyright → pyrefly の移行を実施済みだったが、
他サービスへの展開は行われていなかった。

### 課題

1. **ツールの断片化** — mypy と pyrefly が混在しており、CI ワークフローの `reusable-test-python.yaml` は pyrefly 前提で設計されているにもかかわらず、recap-subworker だけが mypy を使用
2. **型安全性の欠如** — news-creator、dashboard、tts-speaker は型チェッカーが未導入で、Optional のナローイング忘れや未初期化変数などの潜在バグが放置されていた
3. **設定の不統一** — recap-evaluator は pyrefly 0.17.0 で設定キーがアンダースコア形式（`project_includes`）であり、他のサービスのハイフン形式（`project-includes`）と不一致

## Decision

**Pyrefly >= 0.42.0 を全 Python サービスの標準型チェッカーとして採用し、mypy を完全に廃止する。**

### 検討した代替案

| 選択肢 | 評価 |
|--------|------|
| mypy で統一 | CI に Node.js 不要だが、pyrefly の方が ~95% 高速（[[000050]] で実証済み） |
| pyright で統一 | Node.js 依存が復活し、`pyrightconfig.json` が別途必要 |
| サービスごとに自由選択 | CI ワークフローの分岐が複雑化し、メンテナンスコストが増大 |

Pyrefly は Rust 実装による高速チェック、`pyproject.toml` への設定統合、
そして既に `reusable-test-python.yaml` で `uv run pyrefly check .` が標準コマンドとして
定義済みである点から、統一先として最適と判断した。

## Implementation

### 設定パターン

全サービスの `pyproject.toml` に以下の統一パターンを適用:

```toml
[dependency-groups]
dev = [
    "pyrefly>=0.42.0",
]

[tool.pyrefly]
project-includes = ["."]  # または ["src"] (src レイアウトの場合)
project-excludes = ["**/__pycache__", "**/.venv", "**/build", "**/dist"]
python-version = "3.XX"   # サービスの Python バージョンに合わせる
```

ML 依存が重いサービスでは `replace-imports-with-any` で型スタブのないライブラリを `Any` に置換:

```toml
replace-imports-with-any = [
    "torch.*", "transformers.*", "sentence_transformers.*",
    "sklearn.*", "numpy.*", "pandas.*",
]
```

### サービス別の主な修正

**recap-subworker（221 type errors → 0）— mypy 完全撤去:**
- `mypy>=1.11.0` → `pyrefly>=0.42.0` に置換
- 36 ファイルで型エラーを修正: Optional パラメータに `| None` 追加、`assert` による None ナローイング、`multiprocessing.pool.Pool` への型修正、`ProcessPoolExecutor` のオーバーロード解決
- `replace-imports-with-any` で 30 以上の ML ライブラリ（torch, sklearn, hdbscan, optuna, gensim, faiss 等）を Any 型に置換

**news-creator（53 type errors → 0）— 新規導入:**
- `LLMProviderPort` の `hold_slot` / `generate` シグネチャを修正し、`bad-override` を解消
- `aiohttp.ClientSession` の Optional アクセスに `assert self.session is not None` ガード追加
- `AsyncIterator[LLMGenerateResponse] | LLMGenerateResponse` のユニオン型を `isinstance` チェックでナローイング
- ストリーミング処理の `tokens_yielded` 等を try ブロック前で初期化

**recap-evaluator（18 type errors → 0）— バージョン更新 + 修正:**
- pyrefly 0.17.0 → 0.42.0 にアップグレード、設定キーをハイフン形式に統一
- `bert_score` / `pipeline` / `rouge_scorer` の lazy import を `assert` でナローイング
- `asyncio.Future` と `asyncio.Task` の型不一致を修正

**dashboard（7 type errors → 0）— 新規導入:**
- `dict[str, object]` 型注釈でメトリクス辞書の型安全性を確保
- 未定義変数 `table_missing` の参照を除去（実行時バグの修正）
- `pd.json_normalize` の引数を `.tolist()` で変換

**tts-speaker — 新規導入:**
- `AsyncIterator` → `AsyncGenerator` に修正（Python 3.12 での非推奨対応）

### エラー抑制方針

- `missing-import = false` / `missing-module-attribute = false`: ML ライブラリが Docker 内で別途インストールされるサービスでのみ使用（tag-generator の既存パターンを踏襲）
- `replace-imports-with-any`: 型スタブが存在しない ML ライブラリに対して使用。ブランケットな `bad-argument-type = false` 等のエラーカテゴリ全体の抑制は行わない
- 行単位の `# pyrefly: ignore[error-code]`: 外部ライブラリの型スタブ不備（OpenTelemetry `LoggingInstrumentor`）やテストでの意図的な不正値のみ

## Consequences

### Pros

- **統一された型チェック基盤**: 全 Python サービスで `uv run pyrefly check .` が通る状態を達成
- **CI 即応**: `reusable-test-python.yaml` の `typecheck` ジョブをどのサービスにも適用可能
- **潜在バグの検出と修正**: 未初期化変数（`unbound-name`）、未定義参照（`unknown-name`）、Optional 未チェックなど、実行時に顕在化しうる問題を修正
- **mypy 完全廃止**: 型チェッカーの二重管理が解消

### Cons / Tradeoffs

- **ML ライブラリの型情報は Any**: `replace-imports-with-any` により ML ライブラリ境界の型安全性は保証されない。型スタブの充実に応じて段階的に外していく
- **53 箇所の行単位 suppress**: recap-subworker の一部のエラー（asyncio ジェネリック制約、ML ライブラリ動的型）は行単位の `# pyrefly: ignore` で対応。新規コードは完全にチェックされる
