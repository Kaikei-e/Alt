# ADR-000435: Knowledge Home Lens の read path をサーバーサイド filter と active selection で確定する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — `GetKnowledgeHome`, `knowledge_lenses`, `knowledge_current_lens`, `knowledge_home_items`
- alt-butterfly-facade (Go) — protobuf 再生成と BFF 経由の契約追従
- alt-frontend-sv (TypeScript/Svelte) — `GetKnowledgeHome` / `ListLenses` 契約追従

## Context

Knowledge Lens は Phase 1 時点で CRUD と選択 UI を持っていたが、Knowledge Home の read path を実際には変えていなかった。そのため保存済み Lens を選んでも、Home のカード集合・ページネーション・empty state は default view のままで、ユーザーが「Lens が効いた」と判断できなかった。

さらに、Lens の source/filter 契約にも未確定部分が残っていた。

- request に `lens_id` を渡した場合と、永続化済み active lens を持つ場合の優先順位が決まっていない
- `tag_ids` が tag master の ID なのか、保存済みタグ名なのかが曖昧
- `source` を feed 単位で絞るのか、将来の generic source 抽象を待つのかが未決
- recap / pulse を含む全 item_type に Lens を波及させると、Phase 2 の実装範囲が過大になる

Phase 2 ではまず「Lens 選択が Home の取得結果を確実に変える」ことを成立させる必要があった。

## Decision

Knowledge Home Lens の read path は、**サーバーサイドで active selection を解決し、`tag + feed + recent window` を article item に適用する**。

具体的には以下を採用する。

- `GetKnowledgeHome` の effective lens は `request.lens_id`、`knowledge_current_lens`、no lens の順で解決する
- active lens は user ごとに永続化し、`ListLensesResponse.active_lens_id` で frontend へ返す
- `SelectLensRequest` は clear を表現できる契約にし、選択解除を API で明示できるようにする
- Lens version には `feed_ids` を追加し、source filter は generic source ではなく `feed_id` に固定する
- `tag_ids` は Phase 2 では tag master lookup を導入せず、`knowledge_home_items.tags_json` に含まれる tag 名として扱う
- `recent window` は `published_at >= now - window` で適用する
- `tag` または `feed` 条件を 1 つでも持つ Lens は `item_type = article` のみを対象にし、recap / pulse / rail 系 item への波及は後続フェーズへ送る
- 既存の `query_text`、`include_recap`、`include_pulse`、`sort_mode` は互換のため保存契約には残すが、Phase 2 の read path では使わない
- filter 適用後も ordering と cursor 仕様は既存の `score DESC, published_at DESC, item_key` を維持する

代替案として、frontend 側で取得済みカードを filter する案や、Lens を generic query abstraction にしてから実装する案も検討した。しかし前者はページネーション整合性を壊し、後者は read path の価値提供を遅らせるため採用しない。

## Consequences

### Pros

- Lens 選択が Home のカード集合に直接反映され、保存済み UI ではなく実機能になる
- active lens の永続化により、reload 後も同じ read path を再現できる
- `feed_ids` と `tag` を protobuf 契約に載せることで、backend と frontend の解釈差を減らせる
- article item に適用対象を限定することで、Phase 2 の複雑さを抑えつつ cursor 契約を守れる

### Cons / Tradeoffs

- `tag_ids` を実 ID ではなく tag 名として扱うため、将来 tag master を導入する際に移行判断が必要になる
- Lens が recap / pulse を変えないため、「Home 全体が完全に Lens 化された」とはまだ言えない
- active lens 解決が request / persisted state / no lens の三段階になるため、テストケースが増える
- migrate 時に `knowledge_lens_versions.feed_ids_json` と protobuf 再生成を同時に維持する必要がある
