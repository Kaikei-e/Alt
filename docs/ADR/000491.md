# ADR-000491: Knowledge Home から Augur への問い合わせ導線を正式接続する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-frontend-sv (SvelteKit)

## Context

Knowledge Home には `Ask` 導線が存在していたが、実際にはその場で回答せず、Augur へ遷移する前提の暫定 UI に留まっていた。特に `AskSheet` では「未接続」と明示されており、Knowledge Home からの質問が正式なプロダクト導線として扱われていなかった。

さらに、Knowledge Home 側は Augur 遷移時に `q` と `context` を URL パラメータとして渡していた一方で、Augur page 側は `context` しか読んでおらず、`q` を初回問い合わせとして処理していなかった。このため、ユーザーが Knowledge Home から質問を発しても、Augur 側では単なる入力初期値に近い扱いになり、問い合わせ導線として意味が完結していなかった。

今回の実装では、以下を解消する必要があった。

- Knowledge Home から Augur への handoff を、実装済み機能として成立させる
- `q` と `context` の解釈を 1 箇所にまとめ、desktop / mobile で同じ意味論を使う
- 「未接続」の説明を UI から除去し、現在の実装に合う英語文言へ揃える

## Decision

Knowledge Home から Augur への問い合わせ導線を、暫定遷移ではなく正式な handoff として採用する。

- Augur entry utility を追加し、`q` と `context` の解釈を一元化する
- `q` が存在する場合は、`context` を添えた初回問い合わせメッセージを組み立てて Augur に渡す
- `context` のみが存在する場合は、自動送信せず input draft として保持する
- desktop / mobile の Augur UI の両方で、初回メッセージの自動送信をサポートする
- `AskSheet` の説明文と CTA を、現在の実装に一致する英語文言へ変更する

今回の実装では、質問文と文脈の結合形式を `Context:` と `Question:` を持つ単一メッセージとして扱う。これにより、backend 側に Knowledge Home 専用の別 endpoint を追加せず、既存の Augur chat 契約へ接続できる。

代替案として「Knowledge Home 内専用の回答 API を新設する」「`context` のみ入力欄へ入れてユーザーに再送を委ねる」も検討したが、前者は backend 契約追加が必要で変更範囲が大きく、後者は導線として未完了の状態を残すため採用しない。

## Consequences

### Pros

- Knowledge Home の `Ask` が、暫定導線ではなく完結した問い合わせフローになる
- Augur 受け側の挙動が desktop / mobile で揃う
- `q` と `context` の handoff 仕様が utility に集約され、今後の変更点が明確になる

### Cons / Tradeoffs

- 初回問い合わせメッセージの組み立て形式は frontend 側の決定であり、将来的に Augur 側で専用 contract を持つ余地が残る
- `context` を単一テキストとして渡すため、構造化されたコンテキスト種別は現時点では区別しない
- 自動送信は便利だが、Knowledge Home からの遷移時にユーザー意図を即時実行する前提になる
