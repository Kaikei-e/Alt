# ADR-000820: Acolyte Gatherer の言語横断検索を bilingual Planner と HyDE 統合で強化する

## Status

Accepted

## Date

2026-04-20

## Affected Services

- acolyte-orchestrator (Python)

## Context

Acolyte の日本語 topic の `weekly_briefing` レポート（対象: report `eb67863f-f6f0-48a4-bd76-a9f1d9a79054`、「イラン情勢 分析レポート 2026年4月19日」）は、alt-db のコーパスに直近 1 週間で Iran 関連の英語記事が 9 件（Guardian / NYT 系）存在するにもかかわらず、v10 の analysis / executive_summary / conclusion に含まれる 14 件の citations が全件 `language='ja'` で占められていた。eval ハーネス `db-replay` の `lang_mix_ratio` は `{"ja": 1.0}` で、`expected_lang_mix={"ja": 0.6, "en": 0.4}` に対して 40 pt の gap が存在する。

### 既存分析との関係

[[PM-2026-021]] で同等の症状 — 日本語クエリから英語記事に到達できない — を Ask Augur (rag-orchestrator) で観測し、[[000693]] と [[000695]] で news-creator expand-query の bilingual ラベル、rag-orchestrator BM25 の展開クエリ反映、search-indexer の `Locales=["jpn"]` 制限緩和の 3 サービス横断修正を行った。Acolyte の Gatherer / Planner は [[000695]] の対象外であり、同パターンの欠落が残っていた。

### 直接原因の再現

1. **Planner prompt が bilingual 出力を強制しない**  
   `acolyte/usecase/graph/nodes/planner_node.py` の `PLANNER_PROMPT` は XML 例に英語クエリのみを含み、日本語 topic を差し込んだ際に Gemma4 が日本語クエリだけを返す。

2. **HyDE cross-lingual expansion が faceted path 限定**  
   `acolyte/usecase/graph/nodes/gatherer_node.py::_search_by_facets` でのみ `self._hyde` を呼び出し、`_search_by_queries`（legacy path）には無い。`self._hyde is None` のときは警告なしに silently に cross-lingual variant が落ちる。

3. **観測性欠落**  
   HyDE が効いていない / Planner のクエリが片言語になっている状況が INFO ログからは区別できない。加えて `SearchIndexerGateway` が HyDE-generated hypothetical passage（600 字級）を INFO レベルの `query` 属性に丸ごと出力しており、観測基盤側で HyDE 有効化後にログ容量が跳ねるリスクがあった。

## Decision

**[[000695]] と同じ「プロンプト・クエリ展開・観測性」3 箇所を Acolyte 側で閉じる**。

1. `PLANNER_PROMPT` に明示的な bilingual 出力契約を加え、Planner の XML 例を Japanese + English の両方を含む形に置き換える。
2. Gatherer の HyDE 生成ロジックを `_resolve_hyde_variant(topic)` に抽出し、`_search_by_facets` と `_search_by_queries` の両方から呼ぶ。`self._hyde is None` かつ topic が `ja` / `en` に detect された場合は call あたり 1 度だけ構造化 warning を emit する。
3. `SearchIndexerGateway.search_articles` の INFO ログで `query` を 120 字で truncate し、`query_full_len` を別属性として残す。

### 検討した代替案

| 案 | 評価 |
|----|------|
| A. Planner prompt はそのまま、Gatherer の query_facet 展開側で bilingual 変換 | × Planner reasoning が片言語で進むため、section-specific な claim 設計が偏る |
| B. HyDE を常時有効 + DI 欠落時は fail-closed（pipeline 停止） | × 運用事故耐性が落ちる。warning + degraded が [[000693]] の方針と一貫する |
| **C. Planner prompt bilingual + HyDE 両経路 + warning + ログ truncation** | ○ [[000695]] パターンの再適用で最小サーフェス |

C を採用。

## Implementation

### 1. Planner bilingual prompt

ファイル: `acolyte-orchestrator/acolyte/usecase/graph/nodes/planner_node.py`

`PLANNER_PROMPT` 本文に「各セクションで日本語クエリと英語クエリを最低 1 本ずつ含めること」を明示し、XML 例に `<query>AI半導体市場 2026 概観</query><query>AI chip market overview 2026</query>` のように bilingual ペアを埋め込んだ。`normalize_plan_output` と `QueryExpansionOutput` の schema は既存の `list[str]` を維持（言語別分離はしない）。

### 2. Gatherer HyDE 統合

ファイル: `acolyte-orchestrator/acolyte/usecase/graph/nodes/gatherer_node.py`

`_resolve_hyde_variant(topic)` ヘルパーを新設し、`__call__` で call あたり 1 回だけ呼び出して結果を `_search_by_facets` / `_search_by_queries` の両方に渡す。legacy path では synthetic section key `_hyde` を使って variant を検索パイプラインに挿入する。`self._hyde is None` のときは `Gatherer: cross-lingual HyDE generator not wired — retrieval will not request translated recall variants` を warning で emit。

### 3. SearchIndexerGateway ログ truncation

ファイル: `acolyte-orchestrator/acolyte/gateway/search_indexer_gw.py`

`_LOG_QUERY_MAX_CHARS = 120` 定数を導入し、`logger.info("search_articles", query=query[:_LOG_QUERY_MAX_CHARS], query_full_len=len(query), hits=len(hits))` に差し替えた。フルクエリは search-indexer への HTTP 送信時にはそのまま使われ、短縮は観測基盤向けのみ。

### 4. テスト

- `tests/unit/test_planner_bilingual_queries.py` — prompt contract（English / Japanese ラベル存在）、LLM からの bilingual XML 受入、fallback クエリに ASCII トークンを含む
- `tests/unit/test_gatherer_hyde_legacy.py` — legacy path での HyDE 発火、DI 欠落時の warning、warning が call あたり 1 回まで
- `tests/unit/test_search_indexer_gw_log_truncation.py` — 長いクエリの truncation、短いクエリの完全保持

既存 625 件と合わせて acolyte-orchestrator unit suite 628 件全て green。ruff / pyrefly clean。

## Consequences

### Pros

- 対象 report `eb67863f` の再生成 v11 で analysis citations の EN 比率が 0% → 33.3%（body 解決後）まで回復。Writer の「必ず日本語で執筆」制約は維持され、本文は日本語のまま citations のみ多言語化するという責務分離が成立した。
- HyDE DI 欠落が observability 層で即時に surface される。以後 `hyde_enabled=false` のときの挙動差が監視可能。
- SearchIndexerGateway の INFO ログ容量が HyDE passage 混入ぶん約 5 倍に膨らむリスクを cap 固定で抑止。

### Cons / Tradeoffs

- Planner が Gemma4 に追加の英語クエリを要求するため、1 レポートあたりの LLM 呼び出しトークン量が微増。現行 e4b モデル / AIX 構成での実測影響は 7 分 run 内に収まる範囲。
- `search_queries` に JP / EN が混在するため、ダウンストリーム（query_facet decompose、RRF fusion）は単純連結を前提に動く。言語別の recall 計測をしたい場合は別途 source_label で識別する必要がある。
- Pact consumer 契約（search-indexer `/v1/search`）は本 ADR では変更していない。HyDE 生成文は引き続き `q=` にそのまま乗る。Broker publish 側の検討は本 ADR の範囲外。
