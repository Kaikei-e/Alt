# ADR-000497: Decorator Gateway パターンによる BE 要約の分散ディスパッチ

## Status

Accepted

## Date

2026-03-20

## Affected Services

- news-creator (Python/FastAPI)

## Context

ローカル 1 台の Ollamaで BE（バッチ）要約を処理しているが、記事流入量に対してスループットが不足し、キュー滞留が常態化していた。Tailscale VPN 経由で到達可能な複数のリモートマシンが遊んでおり、BE 要約をこれらに分散できれば処理能力を数倍に拡大できる。

ただし以下の制約がある:

- RT（ストリーミング）要約はレイテンシが最優先であり、ローカル GPU 以外に送ってはならない
- リモートマシンの稼働状態はまちまちであり、未稼働・異常時はローカルにフォールバックする必要がある
- 既存の `OllamaGateway` / `HybridPrioritySemaphore` / Usecase 層への変更は最小にしたい
- 一部のリモート（RAG 用 Mac）では別名モデル（`gemma3-4b-rag`）が常駐しており、モデル名をリモートごとに変える必要がある
- 完全オプション機能として、`DISTRIBUTED_BE_ENABLED=false`（デフォルト）で既存と同一動作を保証する

## Decision

**Decorator Gateway パターン**を採用し、`DistributingGateway` が既存 `OllamaGateway` をラップする形で BE 要約を外部リモートへ分散する。

### 代替案と評価

| 案 | 評価 |
|---|---|
| OllamaGateway を直接改修 | 既存ロジック（セマフォ、OOM 検出、モデルルーティング）への影響大。テスト範囲が爆発する |
| Usecase 層で分岐 | SummarizeUsecase / RecapSummaryUsecase / ExpandQueryUsecase すべてに分岐が入る。DRY 違反 |
| リバースプロキシ（Nginx/HAProxy） | アプリ層のセマフォと連携できない。ヘルスチェックにモデル存在確認が必要で L7 判定が困難 |
| **Decorator Gateway（採用）** | OllamaGateway への変更ゼロ。Usecase 層は `LLMProviderPort` 経由で透過的に利用。OFF 時は OllamaGateway 直接注入でオーバーヘッドなし |

### アーキテクチャ

```
Usecase → LLMProviderPort
              │
        DistributingGateway (Decorator)
              │
        ┌─────┴─────┐
   OllamaGateway   RemoteOllamaDriver
   (local, 既存)    → リモート Ollama 群
```

### ディスパッチ戦略: Overflow + Priority Cascade

1. RT リクエスト → 常にローカル（パススルー）
2. BE `hold_slot()` → 優先順にリモート健全性チェック
3. 健全なリモートあり → ローカルセマフォを取得せず、リモートへ直接送信
4. リモート全滅 → ローカルフォールバック（既存と同一動作）
5. リモート生成失敗 → `mark_failure` で即座に unhealthy 化、次の健全リモートを試行、全滅ならローカルフォールバック

`contextvars.ContextVar` でコルーチンごとにリモート URL を追跡し、`hold_slot()` → `generate_raw()` 間の並行 BE 安全性を確保する。

### 新規コンポーネント

| ファイル | レイヤー | 責務 |
|---------|---------|------|
| `gateway/remote_ollama_driver.py` | Driver | リモート Ollama への HTTP POST。URL をパラメータで受け取り、単一インスタンスで複数リモートに対応 |
| `gateway/remote_health_checker.py` | Gateway | バックグラウンドヘルスチェック。`GET /api/tags` で各リモートを定期プローブし、必要モデルの存在まで確認。リモートごとにモデル名オーバーライド可能 |
| `gateway/distributing_gateway.py` | Gateway | `LLMProviderPort` 実装。`OllamaGateway` をラップし、BE の `hold_slot()`/`generate_raw()` をインターセプト。リモート失敗時の cascade retry + local fallback |

### 変更コンポーネント

| ファイル | 変更内容 |
|---------|---------|
| `config/config.py` | `distributed_be_*` 設定追加（enabled, remotes, model, overrides, intervals） |
| `main.py` | `DependencyContainer` で条件付きラップ。`llm_provider` を Usecase / Handler に注入 |
| `gateway/__init__.py` | `DistributingGateway` エクスポート追加 |
| `handler/health_handler.py` | `/queue/status` が `DistributingGateway.queue_status()` をサポート。リモート健全状態を公開 |
| `compose/ai.yaml` | 環境変数 7 個追加（デフォルト OFF） |

### Per-Remote モデルオーバーライド

`DISTRIBUTED_BE_MODEL_OVERRIDES` 環境変数で、特定リモートに異なるモデル名を指定できる:

```
DISTRIBUTED_BE_MODEL_OVERRIDES=http://host:port=gemma3-4b-rag:latest
```

これにより、RAG 用に常駐ロードされたモデルを持つマシンでモデルスワップを回避できる。ヘルスチェックもリモートごとに正しいモデル名で存在確認する。

### 設定（環境変数）

| 変数 | デフォルト | 説明 |
|------|-----------|------|
| `DISTRIBUTED_BE_ENABLED` | `false` | 分散ディスパッチの有効化 |
| `DISTRIBUTED_BE_REMOTES` | 空 | カンマ区切りリモート URL（優先順） |
| `DISTRIBUTED_BE_REMOTE_MODEL` | `gemma3:4b-it-qat` | リモートに送信するデフォルトモデル名 |
| `DISTRIBUTED_BE_MODEL_OVERRIDES` | 空 | `url=model` 形式のリモートごとモデル上書き |
| `DISTRIBUTED_BE_HEALTH_INTERVAL_SECONDS` | `30` | 健全リモートのプローブ間隔 |
| `DISTRIBUTED_BE_TIMEOUT_SECONDS` | `300` | リモートリクエストタイムアウト |
| `DISTRIBUTED_BE_COOLDOWN_SECONDS` | `60` | 異常リモートの再プローブ待ち |

## Consequences

### Pros

- **OllamaGateway への変更ゼロ**: 既存のセマフォ、OOM 検出、モデルルーティングに一切触れない
- **Usecase / Handler 層への変更ゼロ**: `LLMProviderPort` インターフェースにより透過的
- **完全オプション**: `DISTRIBUTED_BE_ENABLED=false` で既存と同一動作、オーバーヘッドなし
- **段階的ロールアウト**: リモートは 1 台ずつ追加可能。ヘルスチェックで未準備マシンは自動スキップ
- **リモート障害耐性**: cascade retry → local fallback で分散は「best-effort enhancement」
- **観測性**: `/queue/status` にリモート健全状態を公開。structured log に `dispatch_target`, `remote_url`, `fallback_reason` を出力

### Cons / Tradeoffs

- **リモート経由の BE はローカルキュー統計に現れない**: セマフォを取得しないため、`/queue/status` の `acquired_slots` にリモート in-flight は含まれない
- **リモート生成中のプリエンプションなし**: ローカルの `PreemptedException` 機構はリモートには及ばない。長時間リモート生成中に RT 割り込みは不可
- **ModelWarmupService はローカルのみ**: 分散 ON でもウォームアップはローカル GPU に限定
- **ヘルスチェックと実生成の間にタイムラグ**: `hold_slot()` 時点で健全でも `generate_raw()` 時に落ちている可能性がある。cascade retry で緩和するが、1 回分のレイテンシ増加はあり得る
- **モデル名の運用管理**: リモートごとにモデル名が異なる場合、`MODEL_OVERRIDES` の管理が必要
