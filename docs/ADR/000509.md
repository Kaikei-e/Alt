# ADR-000509: 要約キューの整合性修正 — 原子的 dequeue と summary 保存失敗時の completed 抑止

## Status

Accepted

## Date

2026-03-21

## Affected Services

- pre-processor (Go)

## Context

要約ジョブキュー (`summarize_job_queue`) に約 8,800 件の pending 滞留（最古 6 日前）が確認された。
分散ディスパッチ ([[000497]]) は正常動作しており、スループット改善の前に **整合性の穴を塞ぐ** ことを最優先とした。

調査で以下 2 つの整合性リスクが判明した。

### リスク 1: dequeue が原子的でない

`GetPendingJobs(FOR UPDATE SKIP LOCKED)` で取得した後、別のステートメントで `UpdateJobStatus(running)` を呼ぶ 2 ステップ構成だった。
SELECT と UPDATE の間にワーカー間競合のウィンドウが生じ、concurrency を上げるほど重複処理の確率が増大する。

```
-- Before: 2 ステップ
BEGIN;
  SELECT ... WHERE status='pending' FOR UPDATE SKIP LOCKED;  -- ①
COMMIT;
-- ワーカーが ① の結果を受け取り
UPDATE ... SET status='running' WHERE job_id=?;               -- ②
-- ① と ② の間に別ワーカーが同じ行を取得するリスク
```

### リスク 2: summary 保存失敗時に completed になる

`summaryRepo.Create` が失敗しても、後続の `UpdateJobStatus(completed)` が実行されていた。
結果として DB に summary が存在しないのにジョブだけ completed になり、サイレントにデータロスが発生する。

```go
// Before
if err := w.summaryRepo.Create(ctx, articleSummary); err != nil {
    w.logger.Warn("continuing despite DB save failure") // ← 警告のみ
}
// ↓ そのまま completed へ
w.jobRepo.UpdateJobStatus(ctx, jobID, StatusCompleted, ...)
```

これらは concurrency=1 の現在でも理論上発生しうるが、バックログ解消のために concurrency を上げる前提条件として修正が必須だった。

## Decision

**「壊さない → 測る → 小さく速くする」の原則に従い、整合性修正を throughput 改善より先に実施する。**

### 1. 原子的 dequeue (`DequeueJobs`)

`SummarizeJobRepository` に `DequeueJobs(ctx, limit)` メソッドを新設し、pending 取得と running 遷移を単一の UPDATE 文で実行する。

```sql
UPDATE summarize_job_queue
SET status = 'running', started_at = $1
WHERE id IN (
    SELECT id FROM summarize_job_queue
    WHERE status = 'pending'
    ORDER BY created_at ASC
    LIMIT $2
    FOR UPDATE SKIP LOCKED
)
RETURNING id, job_id, article_id, status, ...
```

- `ProcessQueue` は `GetPendingJobs` の代わりに `DequeueJobs` を呼ぶ
- `processJob` から `UpdateJobStatus(running)` の呼び出しを削除（ジョブは既に running 状態で到着する）
- `GetPendingJobs` は `HasPendingJobs` 等の参照用途で残す（削除しない）

### 2. summary 保存失敗時の completed 抑止

`summaryRepo.Create` が失敗した場合、ジョブを `failed` にマークして retry 可能な状態に戻す。

```go
// After
if err := w.summaryRepo.Create(ctx, articleSummary); err != nil {
    errorMsg := fmt.Sprintf("Failed to save summary: %v", err)
    w.jobRepo.UpdateJobStatus(ctx, jobID, StatusFailed, "", errorMsg)
    return fmt.Errorf("failed to save summary: %w", err)
}
// completed はここに到達した場合のみ
w.jobRepo.UpdateJobStatus(ctx, jobID, StatusCompleted, summary, "")
```

**不変条件: completed になるジョブは必ず summary が永続化されている。**

### 検討した代替案

| 案 | 評価 |
|---|---|
| アプリケーション層で分散ロック (Redis/advisory lock) | インフラ依存が増え、DB レベルの `SKIP LOCKED` で十分な問題にはオーバーキル |
| summary 保存を DB トランザクションで completed と同時に行う | summary の保存先が別 DB/テーブルのため単一トランザクションに収まらない。2 フェーズコミットは複雑すぎる |
| summary 保存失敗時に dead_letter 送り | retry で回復可能な一時障害（接続断等）を永続的に捨てることになるため不適切 |

## Implementation

### 変更ファイル

| ファイル | 変更内容 |

| `service/summarize_queue_worker.go` | `ProcessQueue` で `DequeueJobs` を使用、`processJob` から running 遷移を削除、summary 保存失敗時に failed へ遷移 |
| `test/mocks/repository_mocks.go` | gomock に `DequeueJobs` を追加 |

### ジョブ状態遷移図（修正後）

```
         DequeueJobs (atomic)
pending ──────────────────────► running
                                  │
                    ┌─────────────┼─────────────┐
                    │             │             │
                    ▼             ▼             ▼
              summaryRepo    summarize     article not
              .Create fail   API fail      found / empty
                    │             │             │
                    ▼             ▼             ▼
                 failed        failed     failed/dead_letter
                (retry)       (retry)     (non-retryable)
                    │
                    │ (retry_count < max_retries)
                    ▼
                 pending ──► (次のサイクルで再 dequeue)

              summaryRepo.Create success
                    │
                    ▼
               completed
```

## Consequences

### Pros

- **重複処理の排除**: SELECT と UPDATE の間のウィンドウが消滅し、concurrency を安全に上げられる前提条件が整った
- **サイレントデータロスの防止**: summary が永続化されていないジョブが completed にならない。retry で自動回復する
- **計測の信頼性向上**: completed 数 ≈ 永続化済み summary 数が保証され、メトリクスの信頼性が上がる
- **後方互換性**: `GetPendingJobs` は残しており、`HasPendingJobs` 等の既存参照パスに影響なし

### Cons / Tradeoffs

- **summary 保存の一時障害で retry が増える**: 以前は「保存失敗しても completed」だったため retry は発生しなかった。だがこれはデータロスの裏返しであり、正しいトレードオフである
- **`DequeueJobs` と `GetPendingJobs` の共存**: インターフェースにメソッドが 1 つ増えた。`GetPendingJobs` を削除する選択肢もあったが、`HasPendingJobs` や batch safety-net の参照用途で使われているため残した

### 今後のステップ（この ADR の範囲外）

1. **計測追加**: pending count / oldest age / save failure rate 等のメトリクスを整備し、改善を定量的に判断する
2. **concurrency 段階的引き上げ**: 整合性が保証された状態で 3→4→5 と 1 ステップずつ試す
3. **weighted dispatch**: ラウンドロビンから重み付き分配への移行を canary で検証する

## Lessons Learned

本件で得た教訓を明文化し、今後の queue 系実装の判断基準とする。

### 1. 「壊さない → 測る → 速くする」の順序を守る

スループット不足が目に見える症状だったため、concurrency を上げたい衝動があった。
しかし整合性の穴がある状態で並列度を上げると、重複処理やデータロスが指数的に悪化する。
**性能改善の前に不変条件を確立すること。**

### 2. SELECT と UPDATE の分離は競合ウィンドウを生む

`FOR UPDATE SKIP LOCKED` は行ロックとして強力だが、トランザクション境界が SELECT と UPDATE で分かれていると意味がない。
**dequeue パターンでは必ず単一ステートメント（UPDATE...RETURNING）で原子的に遷移させる。**

### 3. エラーを飲み込んで先に進むのはデータロス

「保存失敗してもログを出して続行」は一見 resilient に見えるが、後続で completed にマークされるとリカバリ不能になる。
**エラー後に状態遷移する場合、その遷移が「成功を前提とした遷移」でないか必ず確認する。**

### 4. 複数パラメータを同時に変えない

整合性修正と throughput 改善を同時に行うと、問題が発生した際にどちらが原因か切り分けられない。
**1 変更 → 観測 → 次の変更、のサイクルを厳守する。**
