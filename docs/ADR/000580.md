# ADR-000580: sovereign の GetLens RPC で CurrentVersion を返すようにし SelectLens 500 を解消する

## Status

Accepted

## Date

2026-03-25

## Affected Services

- knowledge-sovereign (Go) — GetLens RPC ハンドラ、ReadOperations インターフェース

## Context

[[000577]] で CreateLens/SelectLens の連鎖障害（JSONB マーシャリング不正、nil pointer dereference、フロントエンド flag 評価順序）を修正したが、**SelectLens が依然として 500 を返す問題が残っていた**。

### 障害の再現

nginx ログで以下が記録されていた:

```
POST /api/v2/alt.knowledge_home.v1.KnowledgeHomeService/SelectLens 500
```

alt-backend のログ:

```
Connect-RPC Error: [connect:ConnectHandler:SelectLens] UNKNOWN_ERROR:
internal server error (caused by: no active version found for lens eef1fd4a-...)
```

DB 上では当該 lens に対する `knowledge_lens_versions` レコードは正常に存在しており（`superseded_by IS NULL`）、データ不整合ではなかった。

### 根本原因の特定

`SelectLensUsecase` → `sovereign_client.GetCurrentLensVersion()` → sovereign の `GetLens` RPC というコールチェーンを追跡した結果、以下の構造的不整合が判明した:

1. `sovereign_client.GetCurrentLensVersion()` は sovereign の `GetLens` RPC を呼び出し、レスポンスの `Lens.CurrentVersion` フィールドから version を取得する設計
2. しかし sovereign の `GetLens` RPC ハンドラ (`rpc_lens.go`) は `readDB.GetLens()` のみを呼び出していた
3. `readDB.GetLens()` は `knowledge_lenses` テーブルのみをクエリし、`knowledge_lens_versions` テーブルを JOIN していない
4. そのため `CurrentVersion` は常に `nil` で返り、クライアント側で「no active version found」エラーとなる

一方、`ListLenses` は `LEFT JOIN LATERAL knowledge_lens_versions` で `CurrentVersion` を正しく取得しており、Lens 一覧表示では問題が発生していなかった。

## Decision

**sovereign の `GetLens` RPC ハンドラで、lens 取得後に `GetCurrentLensVersion()` を追加呼び出しし、`CurrentVersion` を返す。**

### ハンドラレベルでの合成を選択した理由

代替案として `readDB.GetLens()` の SQL を `ListLenses` 同様の JOIN クエリに変更する案があったが、以下の理由からハンドラレベルでの合成を採用した:

- `GetLens` は他の箇所（lens 存在確認、所有者チェック）で version 不要のまま使われる可能性があり、クエリの変更は影響範囲が読みにくい
- `GetCurrentLensVersion` メソッドは既に `read_lens.go` に実装済みで、再利用するだけで済む
- ハンドラが「API の応答として何を返すか」を決定するのは Clean Architecture の責務として自然

### 変更内容

1. **`ReadOperations` インターフェースに `GetCurrentLensVersion` を追加** — `read_operations.go`。`Repository` は既にこのメソッドを実装済みのため、compile-time check はそのまま通る
2. **`GetLens` RPC ハンドラを修正** — `rpc_lens.go`。lens 取得後に `GetCurrentLensVersion` を呼び、`lens.CurrentVersion` にセットしてから proto 変換
3. **テーブル駆動テスト 3 ケース追加** — `sovereign_handler_test.go`。mock に `returnLens` / `returnVersion` フィールドを追加し、(a) lens + version あり、(b) lens のみ（version なし）、(c) lens なし の 3 パターンを検証

## Consequences

### Pros

- **SelectLens が正常動作** — `GetLens` RPC が `CurrentVersion` を返すようになり、500 エラーが解消
- **既存 API 契約に変更なし** — proto 定義の `Lens.CurrentVersion` フィールドは元々存在しており、正しく populated されるようになっただけ
- **既存クエリに変更なし** — `readDB.GetLens()` の SQL は変更せず、ハンドラで合成するため他の呼び出し元に影響しない
- **テストカバレッジ向上** — `GetLens` RPC のレスポンスが version を含むことがテストで保証される

### Cons / Tradeoffs

- **N+1 パターン** — `GetLens` が 2 クエリ（lens + version）を発行する。単一 lens 取得のため実用上問題ないが、バッチ取得が必要になった場合は JOIN クエリへの統合を検討すべき
- **[[000577]] の修正が不完全だった** — 同 ADR で「3 つの障害をそれぞれ根本原因に対して修正する」と宣言したが、実際には `GetLens` の version 未取得という 4 つ目の問題が残っていた。障害修正時にエンドツーエンドの動作確認を徹底する必要がある
