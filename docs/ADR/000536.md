# ADR-000536: Knowledge Sovereign の service / DB 抽出準備を完了する

## Status

Accepted

## Date

2026-03-23

## Affected Services

- alt-backend (Go)

## Context

[[000535]] で Phase 2（安全な移行基盤）が完了し、Sovereign の mutation dispatch、OTel メトリクス、reconciliation、idempotency、rollback が整った。しかし、Phase 4（物理抽出）に進むためにはまだ3つの障壁が残っていた:

### 1. Lens write path が Sovereign 外に残存

Phase 1 で knowledge_projector / recall_projector / dismiss fast-path を Sovereign 経由にしたが、lens の write operations（create, update, select, archive）は5つの独立 usecase が直接 `knowledge_lens_port` を呼んでおり、Sovereign の write path consolidation が不完全だった。Phase 4 の前提条件である「write path consolidation rate = 100%」を満たせなかった。

### 2. Transport contract が未定義

producer（pre-processor, tag-generator, recap-worker）と Sovereign の間のイベント受け渡し形式が形式化されておらず、将来の独立 service 化で transport boundary を切る際の設計基盤がなかった。`KnowledgeEvent` は domain の event store 型であり、service 間通信用の envelope としては設計されていない。

### 3. Cutover の定量的判断基準がなかった

Phase 4 を「始めてよいかどうか」を判断するための quantitative gate が存在しなかった。write path consolidation rate、reconciliation 連続成功回数、replay 成功実績などを構造化された report として取得する手段がなく、判断が主観に依存していた。

## Decision

Phase 3 として、上記3つの障壁を3つの意味的カテゴリに分けて解決する。

---

### A. Write path consolidation の完了 — Lens Sovereign routing

**目的**: 全 write path が Sovereign 経由になり、consolidation rate = 100% を達成する。

Phase 1 で dismiss fast-path に適用した `SetCurationMutator` optional routing パターンを、4つの lens write usecase に展開する。

#### Mutation 定数の追加

`knowledge_sovereign_port/port.go` の curation mutation types に5定数を追加する:

```go
MutationCreateLens        = "create_lens"
MutationCreateLensVersion = "create_lens_version"
MutationSelectLens        = "select_lens"
MutationClearLens         = "clear_lens"
MutationArchiveLens       = "archive_lens"
```

#### 各 usecase の変更パターン

4つの lens write usecase（`create_lens_usecase`, `update_lens_usecase`, `select_lens_usecase`, `archive_lens_usecase`）に共通のパターンを適用する:

1. optional `curationMutator knowledge_sovereign_port.CurationMutator` フィールドを追加
2. `SetCurationMutator(port)` setter を追加
3. レガシー write 成功後に Sovereign routing を実行（non-fatal: エラーは `logger.Logger.ErrorContext` で記録のみ）
4. `IdempotencyKey` を `domain.BuildIdempotencyKey` で自動生成

この方式を選んだ理由: `track_home_action_usecase` で確立された optional routing パターンに準拠することで、既存テストを壊さず、DI container の `SetCurationMutator` 呼び出しを削除するだけで即時フォールバックできる。

`select_lens_usecase` は分岐が2つある（`lensID == uuid.Nil` で clear、それ以外で select）。各分岐の成功後にそれぞれ `MutationClearLens` / `MutationSelectLens` を routing する。

`list_lenses_usecase` は read-only のため対象外。

---

### B. Transport contract の形式化 — SovereignEventEnvelope

**目的**: 将来の独立 service 化で producer が gRPC/MQ 経由でイベントを送る際の envelope を事前定義する。

`domain/sovereign_event_envelope.go` に `SovereignEventEnvelope` 型を定義する:

```go
type SovereignEventEnvelope struct {
    EventID        uuid.UUID       `json:"event_id"`
    Source         string          `json:"source"`          // "pre-processor", "tag-generator" 等
    EventType      string          `json:"event_type"`
    OccurredAt     time.Time       `json:"occurred_at"`
    IdempotencyKey string          `json:"idempotency_key"`
    AggregateType  string          `json:"aggregate_type"`
    AggregateID    string          `json:"aggregate_id"`
    TenantID       uuid.UUID       `json:"tenant_id"`
    ActorType      string          `json:"actor_type"`
    ActorID        string          `json:"actor_id"`
    Payload        json.RawMessage `json:"payload"`
    SchemaVersion  string          `json:"schema_version"`  // "v1"
}
```

`NewSovereignEventEnvelopeFromEvent` converter で `KnowledgeEvent` → `SovereignEventEnvelope` の一方向変換を提供する。`ValidEnvelopeSources` で有効な producer 名を列挙する。

この型が `KnowledgeEvent` と別に必要な理由: `KnowledgeEvent` は domain の event store 型（`EventSeq`, `CorrelationID`, `CausationID` 等の内部フィールドを持つ）であり、transport boundary で producer に見せるべきではない。Envelope は transport 層の公開契約であり、domain 型とは責務が異なる。

Phase 3 では型定義とテストのみ。実際の transport 切り替えは Phase 4 で行う。

---

### C. Cutover readiness の定量的判定 — CutoverReadinessUsecase

**目的**: Phase 4 開始の可否を quantitative gate で判定する usecase を実装する。

#### Domain 型

```go
type CutoverReadinessReport struct {
    WritePathConsolidation WritePathStatus
    ReconciliationHealth   ReconciliationStatus
    ReplayHealth           ReplayStatus
    ObservabilityHealth    ObservabilityStatus
    OverallReady           bool
    BlockingReasons        []string
}
```

#### Gate 定義

`DefaultCutoverGates()` で以下のデフォルト値を返す:

| Gate | デフォルト値 | 意味 |
|---|---|---|
| `MinConsolidationPct` | 1.0 (100%) | 全 write path が Sovereign 経由 |
| `MaxMismatchRate` | 0.01 (1%) | reconciliation mismatch 率 |
| `MinConsecutiveReconOK` | 10 | 連続 healthy reconciliation 回数 |
| `RequireReplaySuccess` | true | replay 成功実績が必要 |

#### Port インターフェース

Clean Architecture に従い、3つの port を定義:

- `WritePathAuditPort`: `CountTotalWritePaths` / `CountConsolidatedWritePaths`
- `ReconciliationHistoryPort`: `GetLatestReconciliation` / `CountConsecutiveHealthy`
- `ReplayHistoryPort`: `GetLatestReplayResult`

usecase の `Execute` は各 port からデータを取得し、gate と比較して `CutoverReadinessReport` を構築する。gate を満たさない項目は `BlockingReasons` に追記される。

### ドキュメント成果物

#### DB ownership table

`docs/plan/involve/sovereign_db_ownership.md` に18テーブルの分類を整理:

- **Sovereign 所有**（12テーブル）: knowledge_events, knowledge_home_items, today_digest_view, recall_candidate_view, projection_versions, projection_checkpoints, reproject_runs, backfill_jobs, projection_audits, lenses, current_lenses, lens_versions
- **Producer 所有**（5テーブル）: summary_versions, tag_set_versions, articles, feeds, feed_links
- **共有**（1テーブル）: knowledge_user_events

DB 分離ポリシー: replay-only 方式（dual-write は行わない）。

#### Cutover readiness checklist

`docs/plan/involve/cutover_readiness_checklist.md` に8カテゴリの YES/NO 判定可能なゲートを定義。

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|---|---|---|
| Lens usecase を新 usecase に統合 | write path は集約されるが影響大 | 既存の handler 配線と Connect-RPC 契約を壊す |
| KnowledgeEvent を transport 型としても使う | 型追加なし | EventSeq 等の内部フィールドが producer に漏れる |
| Cutover readiness を admin handler に直接実装 | 構造は単純 | Clean Architecture 違反（Handler に business logic） |
| DB ownership を code comment で管理 | ドキュメント不要 | grep しないと見えない。Phase 4 で混乱する |

## Consequences

### Pros

- **Write path consolidation 100%** — 全 write path（projector, recall, dismiss, reproject, lens CRUD）が Sovereign CurationMutator 経由
- **Transport contract 定義済み** — Phase 4 で独立 service 化する際に producer が使う envelope が事前に設計されている
- **Cutover 判断が定量的** — 主観ではなく、gate 数値で Phase 4 開始可否を判断できる
- **DB ownership が明確** — 18テーブル全てについて「誰が owner か」「移管するか」が一意に決まっている
- **既存テスト完全保持** — 全パッケージ PASS、既存テスト破壊ゼロ
- **即時フォールバック可能** — DI の `SetCurationMutator` を外せば全 lens が旧パスに復帰

### Cons / Tradeoffs

- **Lens usecase の dual-path 共存** — Sovereign routing と旧 port 呼び出しが並存する。Phase 4 で旧パスを削除するまでコードが増える
- **SovereignEventEnvelope は Phase 3 時点では未使用** — 型定義とテストのみで、実際の transport 切り替えは Phase 4。YAGNI に見えるが、Phase 4 の設計基盤として必要
- **CutoverReadinessUsecase の port 実装が未完** — port インターフェースは定義済みだが、driver 層の実装（実際に DB を query して write path count を返す）は Phase 4 の配線時に行う
- **Schema 変更なし** — Phase 3 は一切の DB schema 変更を行わない。新テーブル・カラム追加は Phase 4 以降
