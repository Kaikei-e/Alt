# ADR-000929: Knowledge Loop SSE 配信路を JWT TTL・nginx location・client ライフサイクル・tile 二重発火の 4 層で固める

## Status

Accepted

## Date

2026-05-27

## Affected Services

- alt-frontend-sv (TypeScript / Svelte 5) — `useKnowledgeLoopStream` に sessionStorage cursor 永続化と intentional-abort guard を導入。`LoopEntryTile` から ASK の duplicate transition emit を撤去
- auth-hub (Go) — `BACKEND_TOKEN_TTL` を 5m から 30m に引き上げ、alt-backend 側 `streamStaleTimeout=30m` と整合
- nginx — `^/api/v2/alt\.knowledge\.loop\.v1\..+/Stream` 専用 location を追加し `proxy_request_buffering off` / `X-Accel-Buffering: no` を強制

## Context

2026-05-27 にユーザから「Knowledge Loop で各ボタンが反応しない / ASK・REVISIT のボタンが重複している」と報告。`curionoah.com.har` (156 entries, 13:41:01 UTC ±1min) と alt-backend ログを突き合わせて、ネットワークエラーゼロ (200×153 / 204×1 / 302×2) なのに UI が動かない状態を検出した。

HAR から取れた一次証拠:

| 指標 | 値 | 意味 |
| --- | --- | --- |
| HTTP エラー | 0 件 | 失敗はネットワークではない |
| `StreamKnowledgeLoopUpdates` セッション数 | 2 本 (13:41:01.905 / 13:41:25.548) 共に `resumeFromSeq:"0"` | 連続再オープン |
| SSE response body | 42 byte ぴったり ×2 | 初回 heartbeat + 10s heartbeat のみ、update frame ゼロ |
| `/loop/transition` 件数 | 12 件のうち 11 件で `observedProjectionRevision:1` | client の rev cursor が停止 |
| 同一 entry への REVISIT 連打 | 3 連打 / 4 秒 | UI 無反応 → ユーザが連打 |

alt-backend ログでは同一ユーザに対し直近 30 分で 5+ 本の `stream_started` が観測され、各セッションは `jwt_expired` で約 4m37s〜4m44s で死亡していた。`BACKEND_TOKEN_TTL=5m` 設定 (`compose/auth.yaml:139`, `compose/compose.staging.yaml:1063`) が `streamStaleTimeout=30m` を期待する handler より短く、5 分毎の強制リコネクトが起きていたためである。

過去の ADR 000874〜000928 は projection を「作る」経路 (source_url pin, CitationKind, DI wiring, surface_planner heartbeat 等) を順番に潰してきたが、projection を **client に届ける** 経路を扱う ADR は存在しなかった。今回のバグは以下 4 つの独立した欠陥が「Loop 全体が無反応」という同じ症状面に収束した合成事象である。

1. **JWT TTL がストリーム生存時間より短い** — `BACKEND_TOKEN_TTL=5m` に対し handler は 30m を前提
2. **`useKnowledgeLoopStream` の catch reconnect race** — intentional abort 後に effect が再度 `stopped=false` にした隙間で旧 connect の catch が `scheduleReconnect()` を発火し、ゴーストストリームを再生
3. **nginx の Knowledge Loop 専用 location 欠落** — 汎用 `^/api/.*(stream|sse)` location が `proxy_request_buffering off` と `X-Accel-Buffering: no` を提供せず、上流 CDN レイヤで update frame がバッファリングされていた
4. **`LoopEntryTile` の ASK transition 二重発火** — 親 `+page.svelte` が転送 (`loop.transitionTo`) を済ませた直後、子 tile が `buildAskTransitionMetadata` で同じイベントをもう 1 回投げ、`await goto()` のレースに救われていただけ

## Decision

以下 4 層の修復を 1 つのバンドルとして適用する。

### 1. JWT TTL を SSE 生存時間に合わせる

`compose/auth.yaml` と `compose/compose.staging.yaml` の `BACKEND_TOKEN_TTL` を `5m → 30m` に引き上げる。`streamStaleTimeout = 30 * time.Minute` (`alt-backend/app/connect/v2/knowledge_loop/handler.go:345`) と整合する。`jwtCheckTicker` (handler.go:353) は引き続き secondly に exp を確認するので、本来想定された 30 分窓で stream が回るようになる。

### 2. nginx に Knowledge Loop 専用 SSE location を追加

`nginx/conf.d/default.conf` に `^/api/v2/alt\.knowledge\.loop\.v1\..+/Stream` 専用 location を追加し、`knowledge_home` 兄弟ブロックと同じ設定 (`proxy_buffering off`, `proxy_request_buffering off`, `proxy_cache off`, `X-Accel-Buffering: no`, `gzip off`, `proxy_send_timeout 2400s`, `proxy_read_timeout 2400s`) を強制する。汎用 `^/api/.*(stream|sse)` location は両 buffering 系設定を持たないため、Loop SSE はその二段保護を受けていなかった。

### 3. `useKnowledgeLoopStream` を sessionStorage cursor + intentional-abort guard で硬化

- **cursor 永続化**: `cursorPersistKey` opt を追加し、提供されれば `sessionStorage` キー `knowledge-loop:resume:<key>` に `lastSeqHiwater` を読み書きする。SvelteKit の `invalidateAll` や記事リーダーへの往復で hook が remount されても resume seq が 0 に戻らない。`stream_expired` を受けた時のみ cursor を破棄して 0n から再開する (server replay 仕様と整合)。
- **catch race の遮断**: `connect()` 内でローカル `myAbort` を closure capture し、catch 節で `myAbort.signal.aborted` を見て自前 abort 起因なら `return` する。これで effect cleanup → 新 effect → 旧 catch の三項レースで生じていたゴースト再接続が止まる。
- 既存の `$derived` + `untrack` ガード (行 153–156) は維持。

### 4. `LoopEntryTile` の ASK 二重発火を撤去

`LoopEntryTile.svelte` の `handleCta` から `buildAskTransitionMetadata` 呼び出しと再 `onTransition` 発火を削除。tile は `await onAsk(entry)` のみで終わる。ASK transition の単一エミッタは親 `routes/(app)/loop/+page.svelte:314` に固定。import から `buildAskTransitionMetadata` も外し、source-spec で再追加を guard する。

### 検討した代替案

- **WebSocket への切り替え**: 投資が大きく、Connect-RPC streaming のままで原因 4 層を片付ければ十分。却下。
- **JWT を SSE 路だけ長寿命にする**: alt-backend handler を JWT-aware にして TTL を二系統にする案。複雑度が高く、auth-hub の発行ポリシーから外れる。`streamStaleTimeout` 側と揃える単純解を採用。
- **catch race を `stopped` 比較強化で塞ぐ**: グローバル `stopped` だけ見ると効果が薄い (effect 再実行で false に戻るのが本質的問題)。closure capture した `myAbort.signal.aborted` を真値にするほうが堅い。
- **classifier 側で intent_signal を `revised` に昇格させる**: 初期仮説だったが、`stream_updates.go:86–101` を読み直したところ `SurfaceRebalanced` 経路で正しく流れる実装になっており、根本原因ではなかった。フロント側の `coalescedRefresh.trigger()` 経路が機能しなかったのは JWT TTL × catch race × nginx buffering の合成。
- **ASK の二重発火を projector 側の dedupe で吸収する**: `client_transition_id` の冪等性で実害は出ていなかったが、`await goto()` レースに依存する設計は壊れやすく、tile 側を撤去するほうが contract として健全。

## Consequences

### Pros

- ユーザ視点: REVISIT を 1 回押せば UI が動く。同一 entry への連打が 3→1 に落ちる想定。`/loop/ask` → `/augur/<id>` への遷移後に projection 反映漏れが残らない。
- ストリーム: 同一ユーザの `stream_started` が 5 分間隔 → 30 分間隔へ。`alt-backend` ログ容量と CPU が削減され、観測ノイズも減る。
- SSE 配信: nginx の buffering を二段で殺すので update frame が browser に即時届く。`projection_seq_hiwater` も heartbeat ごとに正しく前進する。
- cursor 永続化: 記事リーダーへ離脱して戻った時にイベントログ全体を replay せず、最後に届いた seq から resume できる。
- contract: ASK transition の発行責務が親一箇所に固定され、tile が handshake-only という不変条件を source-spec が機械的に守る。

### Cons / Tradeoffs

- JWT TTL を 30m にしたため、token 失効後に旧 token を握っているクライアントの猶予が長くなる。auth-hub の refresh サイクルとの整合を 2026-06 の点検で再確認する。
- `cursorPersistKey` を提供しないコール元は今まで通り in-memory cursor 動作。`sessionStorage` 不利用環境 (Privacy mode 等) ではフォールバックして in-memory に落ちる try/catch を維持。
- nginx に専用 location を追加したため、ルール数が 1 増えた。今後 Connect-RPC 系サービスを追加するたびに同様の専用ブロックが必要 (一般化テンプレートは別 ADR で検討)。
- tile 側で ASK transition を撤去したため、`+page.svelte` の `onAsk` ハンドラが unmount 競合で transition を打ち損ねるとイベントが消える可能性がゼロではない。`await goto()` を先に経由する現状の挙動を維持し、将来 onAsk から transitionTo を完了確実にする (await) リファクタは未消化負債として残す。
