# ADR-000450: Supersede の説明可能性を強化し badge から detail 展開・ReasonMerged validation を追加する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-frontend-sv (TypeScript/SvelteKit)
- alt-backend (Go)

## Context

Knowledge Home の supersede 機能は Phase 4-5 で badge 表示まで実装されたが、「なぜ差し替わったのか」「以前の内容は何だったか」を説明する UI がなかった。`SupersedeBadge` がアイコンとラベルを表示するだけで、以下の情報が非表示のままだった:

- 差し替え時刻（`supersededAt`）
- 以前の要約テキスト（`previousSummaryExcerpt`）
- 以前のタグ一覧（`previousTags`）
- 以前の理由コード（`previousWhyCodes`）

proto 定義の `SupersedeInfo` message にはこれらのフィールドが既に存在し、backend も `previous_ref_json` を通じてデータを返しているが、frontend で活用されていなかった。

また、backend 側では:

1. **ReasonMerged イベントの producer がない**: projector は `ReasonMerged` イベントを受信・投影できるが、このイベントを emit する経路に validation がなかった。外部サービスが `AppendKnowledgeEventUsecase` 経由で送信する設計だが、payload の必須フィールドチェックが未実装
2. **knowledge_home_repository のテストが壊れていた**: supersede 3 カラム追加（Phase 4）後にテストの SQL 期待値が旧 17 引数のまま放置され、テストが必ず失敗する状態だった

これらは Knowledge Home Phase 1-5 是正指示書の P0-4（テスト修復）、P1-1（ReasonMerged producer）、P1-2（supersede 説明可能性）として指摘された項目である。

## Decision

**3 つの是正を一括して実施する。いずれも「supersede の説明可能性」という同一原理に基づく変更であり、分割するとコンテキストが散逸するため統合する。**

### 1. SupersedeDetail コンポーネントの新設（P1-2）

- `SupersedeDetail.svelte` を新規作成し、差し替え詳細を card 内で inline 展開する
- `SupersedeBadge.svelte` を `<span>` → `<button>` に変更し、`expanded` / `onToggle` props を追加
- `KnowledgeCard.svelte` で badge クリック時に `SupersedeDetail` を toggle 表示

drawer ではなく card 内 inline 展開を選択した理由:
- Home 面でのコンテキスト切り替えを最小化
- supersede 情報は該当カードに紐づくローカルな情報であり、全画面表示は過剰

### 2. ReasonMerged の payload validation（P1-1）

- `AppendKnowledgeEventUsecase` に `EventReasonMerged` 固有の validation を追加
- 必須フィールド: `article_id`（必須）、`added_codes`（1 つ以上必須）
- projector 側の受信・投影は既に実装済みのため、producer 側の入口を固める

### 3. knowledge_home_repository テストの修復（P0-4）

- `TestAltDBRepository_UpsertKnowledgeHomeItem_PassesJSONAsTextForPgBouncerCompat`: SQL 期待値を 20 引数版に更新、supersede 3 フィールド（`nil, nil, nil`）を末尾に追加
- `TestAltDBRepository_UpsertKnowledgeHomeItem_MergesWhyJSONByCodeInQuery`: 同様に 20 引数版に更新
- `TestAltDBRepository_UpsertKnowledgeHomeItem_WithSupersedeState`: supersede 付き upsert の新規テストケースを追加

## Consequences

### Pros

- ユーザーが差し替えの文脈を理解できるようになり、badge が「何か変わった」から「何がどう変わったか」の説明装置になる
- ReasonMerged イベントの不正 payload が event store に入ることを防止
- `driver/alt_db` テストが再び通るようになり、CI の信頼性が回復
- supersede の proto フィールド（`previousSummaryExcerpt`, `previousTags`, `previousWhyCodes`）が frontend で初めて活用される

### Cons / Tradeoffs

- `SupersedeBadge` が `<span>` → `<button>` に変わるため、既存の CSS セレクタやテストが button 要素を前提にする必要がある
- inline 展開は card の高さを動的に変えるため、仮想スクロール等を導入する場合にレイアウト再計算が必要になる（現時点では仮想スクロール未採用のため影響なし）
- ReasonMerged の validation は payload 構造に密結合している。projector 側の `reasonMergedPayload` struct と usecase 側の同名 struct が別定義であり、将来の payload 拡張時に両方の更新が必要
