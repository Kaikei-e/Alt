# ADR-000564: 分散 BE リモート群の Ollama アップグレードおよび iGPU ROCm 有効化

## Status

Accepted

## Date

2026-03-25

## Affected Services

- news-creator (Python/FastAPI) — 分散 BE ディスパッチの呼び出し元
- リモート Ollama インスタンス群（Docker コンテナ / ホスト systemd）

## Context

### 背景

分散 BE 要約（[[000497]]）は、ローカル GPU の BE スロットが埋まった際にリモート Ollama インスタンスへジョブをディスパッチする。3 台のリモート（dGPU マシン、iGPU マシン、MPS マシン）が Idle-Aware 割当（[[000510]]）で運用されていた。

### 課題

| 課題 | 詳細 |
|------|------|
| dGPU リモートの GPU 未使用 | Ollama v0.16.2 がモデルを CPU にのみロードし、8GB VRAM の dGPU を全く使用していなかった。推論速度 12 tok/s（GPU 利用時の 92 tok/s の 13%） |
| iGPU リモートの CPU 限定動作 | ROCm 未インストールのため Ollama が CPU モードで動作。iGPU (RDNA 3.5, 16CU) が未活用 |
| Ollama バージョンの分散 | リモート間で v0.13.5〜v0.16.2 とバージョンが異なり、新しいモデルスケジューリング機能が使えない状態 |
| プロセス管理の不統一 | 一部リモートが裸プロセスで動作し、systemd 管理されていなかった |

### 計測データ（変更前）

| リモート | Ollama | GPU バックエンド | Decode (tok/s) |
|---------|--------|---------------|---------------|
| dGPU (8GB VRAM) | v0.16.2 | なし（CPU フォールバック） | 12.0 |
| iGPU (RDNA 3.5) | v0.13.5 | なし（CPU のみ） | 23.8 |
| MPS | 不明 | MPS | ~19 |

## Decision

**全リモート Ollama を v0.18.2 に統一し、各リモートの GPU バックエンドを有効化する。iGPU マシンには ROCm 7.2 をインストールし、systemd による管理に移行する。**

### 検討した代替案

| 選択肢 | 評価 |
|--------|------|
| ROCm ではなく Vulkan を iGPU バックエンドに使用 | Web 調査で gfx1150 では Vulkan が小コンテキストで 10-15% 速いとの報告あり。ただし ROCm は大コンテキスト（16k+）で逆転し、TTS サービスが同一マシンで ROCm 7.2 を使用中のため、ROCm に統一して運用を簡素化 |
| iGPU マシンを CPU のまま運用 | iGPU はメモリ帯域幅律速で CPU と同速（~23 tok/s）のため速度向上は限定的。ただし GPU オフロードにより TTS・embedder との CPU 競合を軽減できるため、ROCm 有効化を選択 |
| dGPU リモートの Ollama をホストインストールに移行 | Docker コンテナでの運用が簡潔であり、イメージ更新のみで GPU 利用が復活したため不要 |

### 実装

#### dGPU リモート

Docker コンテナのイメージを `docker pull ollama/ollama:latest` + 再作成のみ。GPU パススルー（`--gpus`）は既存設定で有効だったが、古い Ollama バージョンが GPU を適切にスケジュールできていなかった。

#### iGPU リモート

1. ROCm 7.2 ユーザースペースのインストール（`--no-dkms` でカーネルモジュール非置換）
2. Ollama v0.18.2 のインストール（ROCm ビルドが自動検出）
3. systemd ユニットの作成（`OLLAMA_KEEP_ALIVE=24h`, `OLLAMA_MAX_LOADED_MODELS=3`, `HSA_OVERRIDE_GFX_VERSION=11.0.0`）
4. 旧 Docker コンテナからホスト Ollama への移行（モデルデータは Docker ボリュームからコピー）

#### 優先度の更新

実測パフォーマンスに基づき `DISTRIBUTED_BE_REMOTES` の順序を更新。

### 計測データ（変更後）

| リモート | Ollama | GPU バックエンド | Decode (tok/s) | 改善率 |
|---------|--------|---------------|---------------|--------|
| dGPU (8GB VRAM) | v0.18.2 | CUDA | **92** | **+664%** |
| iGPU (RDNA 3.5) | v0.18.2 | ROCm | **23** | ±0%（CPU オフロード効果あり） |
| MPS | — | MPS | ~19 | — |

## Consequences

### Pros

- **dGPU リモートのスループット 7.7 倍向上**: 12 → 92 tok/s。ローカル GPU（~51 tok/s）の 1.8 倍速となり、分散 BE の最速リモートに
- **Ollama バージョン統一**: 全リモートが v0.18.2 に統一され、新しいモデルスケジューリング（GPU メモリ実測ベース割当）が有効化
- **iGPU の CPU オフロード**: ROCm により推論処理が GPU に移行し、同一マシンの TTS・embedder サービスとの CPU 競合が軽減
- **systemd 管理**: iGPU リモートが systemd ユニットで管理され、自動再起動・ログ統合が実現

### Cons / Tradeoffs

- **iGPU の速度向上は限定的**: LPDDR5x メモリ帯域幅が律速のため、ROCm 有効化後も decode 速度は CPU と同等（~23 tok/s）。Prefill は ROCm で 32% 遅い（373 → 251 tok/s）
- **ROCm の iGPU メモリ管理の制約**: ROCm は iGPU で VRAM のみ確保し GTT（システムメモリ）を十分に活用しない既知の問題がある（Ollama #13173）
- **Docker `latest` タグの運用リスク**: dGPU リモートの問題の根本原因は `latest` タグのコンテナが作成時のバージョンで固定されること。定期的な pull + 再作成の運用が必要
- **`docker compose up` のカスケード再作成**: `.env` 変更の反映で `up` を実行すると、依存先コンテナも再作成されモデルのコールドスタート（TTFT ~120s）が発生する
