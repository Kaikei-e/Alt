# ADR-000430: Knowledge Home の why surfaced を code 単位でマージし再投影で消えないようにする

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — `knowledge_home_items` への upsert、Knowledge projector の検証

## Context

[[000426]] により `TagSetVersionCreated` から Knowledge Home カードへタグと `tag_hotspot` を投影できるようになった。しかし read model 側の `knowledge_home_items` upsert は `why_json = EXCLUDED.why_json` の完全置換だったため、同一 article に対して複数イベントが順に到着すると、後段イベントが先行イベントの why surfaced を消していた。

典型例は以下である。

1. `ArticleCreated` で `new_unread` が入る
2. `SummaryVersionCreated` で `summary_completed` が加わる
3. `TagSetVersionCreated` で `tag_hotspot` が加わる

このとき最終行には最後のイベント由来の why しか残らず、カード上では `Summarized` と `Trending: {tag}` を同時に出せなかった。直接 SQL パッチで一時補正しても、projector が再度走れば上書きされるため、根本対策は repository の upsert 契約を変える必要があった。

また、通常 projector の checkpoint を 0 に戻して再投影すると、Knowledge Home 以外の副作用、特に `today_digest_view` の加算更新まで再実行してしまう。そのため「再投影で直す」ではなく「保存戦略そのものを変える」判断が必要だった。

## Decision

Knowledge Home の `why_json` は、イベント単位の完全置換ではなく、**reason code 単位のマージ**として保存する。

具体的には以下を採用する。

- `EXCLUDED.why_json` と既存 `knowledge_home_items.why_json` を結合し、`code` ごとに 1 件へ正規化する
- 同じ `code` が新旧両方に存在する場合は、新しく到着した event 側を優先する
- 新しい event に含まれない既存 reason は保持する
- 出力 JSON は安定した順序で再集約し、再実行時にも揺れを減らす

この決定により、`SummaryVersionCreated` と `TagSetVersionCreated` が別々に投影されても、最終的な read model では `summary_completed` と `tag_hotspot` を共存させられる。

検証は repository テストに加え、projector テストでも `ArticleCreated -> SummaryVersionCreated -> TagSetVersionCreated` の順に投影した最終状態を確認する。主戦場は DB upsert 契約であり、projector 側は最終状態の回帰確認に使う。

代替案として projector 側で常に「これまでの why 全部」を再計算して送る案もあった。しかし event ごとに関心が分かれており、projector で全履歴を知る前提を作ると関心分離が崩れるため採用しない。read model の merge-safe upsert で整合性を担保する方が責務が明確である。

## Consequences

### Pros

- `new_unread`, `summary_completed`, `tag_hotspot` など複数の why surfaced を同一カードに共存させられる
- イベント到着順に依存した情報欠落を抑えられる
- 一時的な SQL パッチではなく、repository 契約として恒久的に振る舞いを固定できる
- projector の再処理や追いつき時にも why の欠落が起きにくくなる

### Cons / Tradeoffs

- upsert SQL が複雑になり、単純な置換より可読性は落ちる
- `why_json` のマージ規則を DB 側に持つため、将来 reason の構造を拡張する際は SQL の見直しが必要になる
- 「通常 projector の checkpoint reset で直す」運用は採らず、既存データ補正が必要なら副作用なしの one-off backfill を別途用意する必要がある
