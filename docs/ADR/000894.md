# ADR-000894: Knowledge Loop の同一ステージ defer transition を `loop.dismiss(metadata?)` に集約し、Open Recap を自然な next stage に動的選択する

## Status

Accepted

## Date

2026-05-09

## Affected Services

- alt-frontend-sv (SvelteKit)
- alt-frontend-sv e2e (Playwright fixtures + specs)

## Context

[[000893]] で land した Phase 2 (`/loop` semantic Decide / Act feedback loop) の e2e 自走化を進める過程で、実装と canonical contract §7 transition matrix の間に 2 件の不整合が発覚した。両方とも、unit / vitest / Pact では検出できない「stage allow-list が拒否したため POST が飛ばない」系の silent failure で、Playwright が dev stack 上で実際に request を観測した時点で初めて表面化した。

### 1. Snooze と workspace dismiss が同一ステージの user_tap を投げ、`canTransition` で早期 return していた

[[000893]] では Tile の Snooze と workspace decision-option button (Snooze) を同じ helper (`buildTransitionMetadata`) 経由で `loop.transitionTo(entryKey, effectiveStage, "defer", metadata)` のように呼んでいた。しかし `loop-transitions.ts` の `canTransition` allow-list は ADR-000831 §7 を直訳しており、同一ステージは規定外なので無条件で `false` を返す:

```ts
const ALLOWED: ReadonlySet<`${LoopStageName}->${LoopStageName}`> = new Set([
  "observe->orient", "observe->decide",
  "orient->observe", "orient->decide",
  "decide->act", "act->observe",
]);
```

`useKnowledgeLoop.svelte.ts::transitionTo` の冒頭がこの guard を踏むため、Snooze の POST は一度も飛ばずに `{ status: "forbidden" }` を返して終わる。Phase 2 が約束した「`acted_intent=snooze` / `target_type=entry` / `continue_flag=false` を payload に乗せて event log に戻す」という意味論が成立しない。

一方、同じ hook には [[000750]] 系で land した `dismiss(entryKey)` が存在し、これは `post({ ..., trigger: "defer" })` を **`canTransition` を介さず** 直接呼ぶ単独経路として定義されていた (canonical contract §8.2 が `KnowledgeLoopDeferred` を「同一ステージ + defer trigger」で固定しているため、stage progression matrix とは別レイヤで扱う必要がある)。BFF validator (`/loop/transition/+server.ts`) も `SAME_STAGE_TRIGGERS = { defer, recheck, archive, mark_reviewed }` の場合は `fromStage === toStage` を許容し、それ以外の trigger でのみ canonical §7 matrix を要求するという二重判定をする。`dismiss` はこの SAME_STAGE 経路に既に乗っていたので、Snooze の semantic metadata はここに相乗りさせるのが contract 一貫性の点で最も素直だった。

### 2. Open Recap が遷移先を `act` で固定しており、orient / observe stage で `canTransition` を踏み抜いていた

[[000893]] の `LoopEntryTile.svelte::handleOpenRecap` は次の形だった:

```ts
void onTransition(entry.entryKey, "act", "user_tap", meta);
```

これは「Open Recap = Act にコミット」という素朴な解釈だが、`/loop` は実際には observe / orient / decide / act のいずれの段でも recap actTarget を持ち得る (Surface Planner v2 が `RecapTopicSnapshotted` event の `top_terms` overlap を検知すれば、entry のステージに関係なく `seedActTargets` が recap target を吐く)。`orient→act` も `observe→act` も §7 matrix の許可セットに無いため、`canTransition` がはじいて POST が空振りする。

意味論的にも「ステージを丸ごと飛ばして act に直行」は OODA loop の継続性を壊す挙動で、recap を読むのは「次の段に進むための文脈」として扱うのが自然だった。

### 3. e2e 自走化のための fixture handler が未配線

[[000893]] で書いた 2 本の e2e spec (`loop-decide-option-semantic.spec.ts` / `loop-open-recap-emits-transition.spec.ts`) は backend mock の `GetKnowledgeLoop` ハンドラで `e2e-decide` / `e2e-recap` lensModeId を分岐させる前提で組まれていたが、`tests/e2e/infra/data/knowledge-loop.ts` と `tests/e2e/infra/handlers/backend.ts` の対応分岐が未実装だった。dev stack 上では default fixture が返るため、テストは red のまま放置されていた。Phase 2 の "テストが意味論を pin する" 価値を回収するためには fixture の land が必須だった。

## Decision

3 件の連動した判断として実装する。

### 1. `loop.dismiss(entryKey, metadata?)` を Phase 2 semantic metadata の唯一経路にする

`useKnowledgeLoop.svelte.ts::dismiss` のシグネチャを `(entryKey: string, metadata?: TransitionMetadata) => Promise<void>` に拡張し、内部の `post(...)` 呼び出しに `metadata` をそのまま forward する。canonical contract §8.2 の defer trigger は同一ステージ前提なので、`canTransition` を介さない既存設計はそのまま尊重する。

UI 側の差し替え:

- `LoopEntryTile.svelte::handleCta` の Snooze 分岐は `onDismiss(entry.entryKey)` の代わりに `onDismiss(entry.entryKey, transitionMetadata(option))` を呼ぶ。`onTransition` の同一ステージ呼び出しは削除。
- `+page.svelte::onWorkspaceDecide` の Snooze 分岐は `loop.dismiss(entry.entryKey, buildTransitionMetadata(entry, option))` を呼ぶ。
- `LoopEntryTile.svelte` の props 型 `onDismiss` を `(entryKey, metadata?) => ...` に拡張する。

却下した代替案:

- **`canTransition` に同一ステージ defer を許す bypass を入れる**: matrix の意味論を曖昧にする。`canTransition` は OODA stage progression を守るのが責務で、defer のような lifecycle trigger を同じレイヤで判断させると後続の Review-lane (`recheck` / `archive` / `mark_reviewed`) でも同型の例外が必要になり、結果として allow-list が contract を表現しなくなる。BFF validator が既に二重判定をしている事実が、フロント側でも同じ二層構造で扱うべきという信号。
- **`loop.transitionTo` に `force: true` のような escape hatch を追加**: 全 user_tap 呼び出しで考えなくてよい invariant が、ある呼び出し点だけは force で飛ばす形になり、長期的に意図が散逸する。

### 2. Open Recap は `nextStageFor(effectiveStage)` で進む

`LoopEntryTile.svelte::handleOpenRecap` を:

```ts
const to = nextStageFor(effectiveStage); // observe→orient, orient→decide, decide→act, act→observe
void onTransition(entry.entryKey, to, "user_tap", meta);
```

に変更する。`+page.svelte::nextStage` と同じ意味論をローカルに持たせ、entry がどのステージにあろうと user_tap で 1 段進める。`act` の場合は `act→observe` に巻き戻ることになるが、これは canonical contract §5.4 の「Act は終端ではない。Act の後、結果に応じて Observe または Continue に再流入する」と整合する。

却下した代替案:

- **常に `to=act`、`canTransition` 失敗時はサイレントにスキップ**: silent skip は既存実装と同じバグ。可観測性も落ちる。
- **stage を変えず same-stage trigger を使う**: defer は意味論的に「保留」であり「recap を開く」とは整合しない。Open Recap は明確な stage progression。

### 3. e2e fixture を 2 種、既存パターン (`e2e-act` / `e2e-no-source`) と同じ形で land する

`tests/e2e/infra/data/knowledge-loop.ts` に追加:

- `LOOP_FIXTURE_DECIDE_ENTRY_KEY` / `CONNECT_KNOWLEDGE_LOOP_DECIDE_RESPONSE` — `currentEntryStage = LOOP_STAGE_DECIDE (3)` の foreground entry に `revisit` / `ask` / `snooze` の decision options を seed。workspace は `selectedStageName === "decide"` 分岐でこの decision-option list を直接 render する。
- `LOOP_FIXTURE_RECAP_ENTRY_KEY` / `LOOP_FIXTURE_RECAP_TOPIC_SNAPSHOT_ID` (UUIDv7) / `CONNECT_KNOWLEDGE_LOOP_RECAP_RESPONSE` — `actTargets` に `targetType: 3 (RECAP)` + canonical UUID の snapshot id を seed。`safeRecapRoute` 系の URL ガード (`includes(":") → null` 等) を踏まないよう route は `/recap/topic/${uuid}` で固定する。

`tests/e2e/infra/handlers/backend.ts` の GetKnowledgeLoop dispatcher に `e2e-decide` / `e2e-recap` 分岐を追加する。既存の `e2e-act` / `e2e-no-source` 分岐とまったく同じ shape の `else if` で良い。

### 4. Decide spec を Snooze assertion に切り替える

当初の spec は Revisit を assert していたが、`canTransition` allow-list は `decide→orient` を許可しないため、Decide-stage workspace から Revisit を発火しても POST が飛ばない。Phase 2 が exercise したいのは「workspace decision-option button が click された option の semantic metadata を payload に乗せる」ことなので、assert する intent はその意味論が成立する Snooze に切り替える:

- `acted_intent === "snooze"`
- `target_type === "entry"`
- `target_ref === LOOP_FIXTURE_DECIDE_ENTRY_KEY`
- `continue_flag === false`
- `trigger === "defer"`
- `presented_intents` に `revisit` / `ask` / `snooze` がすべて含まれる

これで前段で land した `loop.dismiss(metadata)` の経路が end-to-end で exercise される。

却下した代替案:

- **Open / Save を assert**: 動くが actTarget に `article` を seed する必要があり、fixture が既存 `e2e-act` と紛らわしくなる。Snooze は entry-self-reference target だけで完結し、fixture が小さい。

### 5. Recap spec の URL assertion を `not.toHaveURL(/\/loop\?/)` に緩和

`/recap/topic/<id>` route はこの app の現状の routes/ ツリーには存在せず (recap topic page は recap UI の別フェーズで land 予定)、SvelteKit の fallback で `/feeds` 等に飛ぶ。本 spec の責務は「transition POST が navigation 前に semantic field を伴って発火すること」なので、navigation 先の正確性は別の責務として spec から外す。

却下した代替案:

- **`/recap/topic/<id>` route を仮実装**: ADR の scope 外。recap topic UI が固まる前に dummy route を作ると後で route reservation の事故になる。
- **`route.fulfill` で受信側を mock**: navigation 後の検証はもとから本 spec の対象外なので、URL pattern を緩和する方が責務として正しい。

## Consequences

### Pros

- **Phase 2 が約束した semantic metadata が Snooze / Open Recap のいずれの経路でも実際に payload に乗るようになった**: dev stack 上の e2e で「`/loop/transition` POST が semantic field を伴う」ことを continuous に観測できる。[[000893]] の Pact / projector 単体テストでは捕まらなかった層を補完する。
- **Stage allow-list の意味論を壊さずに defer trigger を扱える**: `canTransition` は OODA progression matrix の代弁者として無傷で残る。BFF validator の二重判定 (canonical §7 vs SAME_STAGE_TRIGGERS) と完全に同じ構造を frontend hook 側でも踏襲したことで、新しい lifecycle trigger (将来の Review-lane recheck/archive/mark_reviewed の semantic 化) が出ても同じパターンで拡張できる。
- **Open Recap がどのステージから呼ばれても合法な遷移を生成する**: Surface Planner v2 が任意のステージで recap target を seed できる現実 (canonical §6.4.1) と整合。`act→observe` を含めて all-stage で動くため、Phase 1 〜 Phase 4 のいずれの段階の entry でも regression なし。
- **e2e self-driving が成立**: `bun run test:e2e desktop/loop --project=desktop-chromium` が 23/23 pass。Phase 2 専用 2 本だけでなく、既存 21 本も regression 無し。`bun run check` 0 errors / `bun run test` 1223/1223 pass で全層 green。
- **Fixture が canonical pattern を踏襲**: `e2e-act` / `e2e-no-source` と同じ shape の `else if` 分岐なので、後続の Phase 3 (Review re-evaluation engine) や Phase 4 (Acolyte deep Orient / Decide) の e2e fixture も同じ方法で追加できる。学習コストゼロ。

### Cons / Tradeoffs

- **`onDismiss` props 型の breaking change**: `LoopEntryTile.svelte` の `onDismiss?: (entryKey, metadata?) => ...` は optional 第二引数なので呼び出し側の互換は保たれるが、TypeScript の structural typing 上 props を渡している全 caller が再型チェックされる。既存 caller は `+page.svelte` の 1 ヶ所のみで影響範囲は限定的だが、別 component が tile を mount する場合 (Storybook / 他フィーチャー) は型を確認する必要がある。
- **`/recap/topic/<id>` route の不在を spec が黙認している**: 本 ADR は recap topic page の責務を明示的に scope 外に出したが、それは「recap topic UI を作るときに本 spec が改めて URL pattern を tighten する必要がある」ことも意味する。Recap topic page の land と同時に [[000894]] の §5 を見直して `toHaveURL(/\/recap\/topic\//)` に戻す follow-up が残る。
- **Decide spec が Revisit / Compare の workspace 経路を pin していない**: §1 で議論した通り、`canTransition` 上 Decide-stage workspace から Revisit / Compare を実行する道筋が無い (decision options としては表示されるが click しても no-op)。これは contract §7 の matrix 自体の制約で、Phase 2 の責務ではないが、Phase 3 で Review-lane を再評価エンジン化するときに「Decide-stage で表示する revisit option はそもそも本当に表示すべきか」を見直す必要がある。Phase 3 で扱う。
- **`nextStageFor(effectiveStage)` が Tile component と Page component の両方に重複定義されている**: `+page.svelte::nextStage` と `LoopEntryTile.svelte::nextStageFor` が同じロジック。次の機会に `$lib/utils/loop-transitions.ts` あたりに pure helper として集約するのが望ましいが、本 ADR の scope では import depth を増やすほどのメリットが無いので保留。
