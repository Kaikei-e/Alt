# ADR-000389: Article/Summaryボタンにリトライ機能を追加する

## Status

Accepted

## Date

2026-03-15

## Affected Services

- alt-frontend-sv (SvelteKit / TypeScript)

## Context

alt-frontend-sv の Article（記事取得）ボタンと Summary（AI要約）ボタンは、以下の3コンポーネントに存在する:

| コンポーネント | 用途 | ファイル |
|---|---|---|
| `FeedDetailModal` | Desktop モーダル | `desktop/feeds/FeedDetailModal.svelte` |
| `SwipeFeedCard` | Mobile スワイプカード | `mobile/feeds/swipe/SwipeFeedCard.svelte` |
| `FeedDetails` | Mobile Sheet | `mobile/FeedDetails.svelte` |

これらのボタンでエラーが発生した場合、エラーメッセージが表示されるだけでリトライ手段がなかった。ユーザーはモーダルを閉じて再度開くしかなく、特にネットワーク不安定な環境（モバイル）での UX が悪かった。

Material Design のガイドラインや WCAG のベストプラクティスでは、エラー発生時にプライマリ CTA 自体をリトライボタンとして機能させるパターンが推奨されている。既存の `SwipeFeedCard` の Favorite ボタンがすでにこのパターン（エラー時にボタンが再クリック可能になる）を採用しており、一貫性のある拡張として実装した。

## Decision

**「元のボタンがリトライボタンになる」パターン**を全3コンポーネントの Article/Summary ボタンに適用する。追加の UI 要素（別途のリトライボタンやトースト）は導入しない。

### ボタン状態マシン

```
idle → loading → success
                → error → [auto-retry 1回（一時的エラーのみ）] → loading → success
                                                                        → error → [manual retry: ボタンクリック]
```

### エラー分類ロジック

一時的エラー（auto-retry 対象）と永続的エラー（即座にエラー表示）を分類する純粋関数 `isTransientError()` を `$lib/utils/errorClassification.ts` に定義した:

- **一時的**: network, fetch, timeout, 502, 503, 429
- **永続的**: 401, 403, その他すべて

Auto-retry は一時的エラーに対して最大1回、500ms のディレイを挟んで実行される。認証エラー（401/403）は即座にエラー表示する。

### 検討した代替案

1. **トーストによるリトライ通知**: Material Design のスナックバーパターン。却下理由 — Alt-Paper デザイン言語ではインラインフィードバックを重視しており、別領域への注意の分散は避ける方針。
2. **自動リトライのみ（手動リトライなし）**: 実装は簡潔になるが、サーバー障害時にユーザーが無力になる。
3. **指数バックオフ付き複数回リトライ**: バックエンドの queue saturation 対策（[[000185]]）では採用済みだが、フロントエンドでは過剰。ユーザーが待たされる時間が長くなりすぎる。

## Implementation

### 新規ファイル

- **`$lib/utils/errorClassification.ts`** — `isTransientError()` 関数。6行の純粋関数でエラーメッセージの文字列マッチにより一時的エラーを判定する。
- **`$lib/utils/errorClassification.test.ts`** — 10テスト。ネットワーク系・タイムアウト・5xx・429 を retryable、401/403/汎用エラーを non-retryable として検証。
- **`$lib/components/mobile/FeedDetails.svelte.test.ts`** — FeedDetails の新規テストファイル。summary retry と `role="alert"` を検証。

### 各コンポーネントの変更

#### FeedDetailModal (Desktop)

- `articleButtonState` / `summaryButtonState` の derived state を追加し、ボタン表示を状態マシンで制御
- `handleFetchFullArticle()`: catch ブロック内で一時的エラーの場合に 500ms ディレイ後の inline retry を実行。`isFetchingContent = true` を維持したまま retry するため、auto-fetch effect の再トリガーを防止
- `handleSummarize()`: `onError` コールバック内で同様の auto-retry を `setTimeout` で実装
- エラー時のボタン: `variant="destructive"` + `RefreshCw` アイコン + "Try again" テキスト
- エラー表示 div: `role="alert"` 追加、Alt-Paper スタイル（`bg-white border-2 border-destructive`）

#### SwipeFeedCard (Mobile Swipe)

- `fetchArticleContent()` ヘルパー関数を抽出し、auto-retry ロジックを内包
- `handleToggleContent()`: `contentError` 状態でのクリックをリトライとして処理（トグルではなく再取得）
- `handleGenerateAISummary()`: `onError` コールバックに auto-retry 追加（legacy フォールバックの前に評価）
- ボタン: `min-h-[44px]` でWCAG タッチターゲット44px保証
- エラー表示: `role="alert"` 追加

#### FeedDetails (Mobile Sheet)

- 120行のインライン summary onclick ハンドラを `handleSummarize()` 関数に抽出
- `summaryButtonState` derived state 追加
- auto-retry ロジックをストリーミングの `onError` コールバックに追加
- ボタン: エラー時 `variant="destructive"` + `RefreshCw` + "Try again"、`min-h-[44px]`
- エラー表示: `role="alert"` 追加

### リトライカウンターのライフサイクル

- `contentRetryCount` / `summaryRetryCount` は各コンポーネント内で `$state(0)` として管理
- モーダルクローズ、フィード切替（矢印ナビゲーション、スワイプ）時にリセット
- auto-retry は合計1回のみ。手動リトライ（ボタンクリック）は回数制限なし

## Consequences

### Pros

- **UX改善**: エラー発生時にモーダルを閉じ直す必要がなくなった。1クリックでリトライ可能
- **一時的エラーの自動回復**: ネットワーク瞬断や 502/503 の場合、ユーザー操作なしで回復する可能性が高い
- **デザイン一貫性**: 既存の Favorite ボタン retry パターンと同一の UX。Alt-Paper デザイン言語の `destructive` variant を活用
- **アクセシビリティ向上**: `role="alert"` によりスクリーンリーダーがエラーを即座にアナウンス。3チャンネルアクセシビリティ（アイコン + テキスト + カラー）
- **タッチターゲット**: Mobile ボタンに `min-h-[44px]` を保証（WCAG 2.5.5 Level AAA）

### Cons / Tradeoffs

- **auto-retry による追加リクエスト**: 一時的エラー時に自動で2回目のリクエストが発生する。ただし1回限定かつ 500ms ディレイがあるため、サーバー負荷への影響は軽微
- **`isTransientError` の文字列マッチ**: エラーメッセージのフォーマットに依存するため、API クライアントのエラーメッセージが変更された場合に追従が必要。ただし純粋関数でテスト済みのため変更検知は容易
- **3コンポーネントに類似ロジック**: auto-retry の実装パターンが3箇所に存在する。共通 hook への抽出は可能だが、各コンポーネントのストリーミング / Promise / 状態管理パターンが異なるため、現時点ではインライン実装を選択した
