# ADR-000935: Visual Preview の OG 画像を article_heads フォールバックと能動 backfill で充足し、派生キャッシュを 7 日保持窓に収束させる

## Status

Accepted

## Date

2026-05-31

## Affected Services

- alt-backend (Go) — `FetchUnreadFeedsListCursor` / `FetchAllFeedsListCursor` / `FetchFavoriteFeedsListCursor` の og:image 解決、`html_parser.ExtractOgImageURL` の抽出強化、`og-image-backfill` / `og-image-retention` ジョブ追加、`FetchArticleGateway` の DI 露出
- alt-frontend-sv — コード変更なし（`/feeds/visual-preview` の表示挙動は据え置き。受け取る `og_image_proxy_url` の充足率のみ改善）

## Context

`/feeds/visual-preview`（および mobile swipe 版）は各 feed の `og_image_proxy_url` を `<img>` に流し、空または読み込み失敗（`onerror`）時に placeholder を表示する。実機で placeholder になる比率が高く、ログと DB を精査して 2 つの原因を特定した（2026-05-31, alt-db / docker logs 実測）。

1. **OG 画像参照の欠落（支配的）**: `feeds.og_image_url` は RSS XML（`Item.Image` / `media:thumbnail` / `media:content` / `Enclosure`）からのみ取得される。多くの RSS は XML に画像要素を持たず、**feeds 129,341 件中 109,261 件（84.5%）が NULL/空**だった。一方、記事ページから抽出した OG 画像を持つ `article_heads` テーブルは **feeds 読み取りクエリに JOIN されておらず**、記事を開いた時（`fetch_article_usecase`）にしか populate されないため、grid に並ぶ未読 feed の大半は最初から placeholder に落ちていた。
2. **proxy 取得の失敗**: image proxy が元画像を取得する際の `WARN "image proxy error" ... HTTP request failed with status 404` が多発。RSS 由来の動的/失効しやすい画像 URL（CDN の期限付き URL 等）が 404 になり、`<img>` の `onerror` で placeholder に落ちていた。
3. **抽出品質の不足**: `ExtractOgImageURL` は `meta[property='og:image']` の https のみを受理し、`og:image:secure_url` / `twitter:image` / `link[rel=image_src]`・相対 URL・protocol-relative URL・http を取りこぼしていた。記事ページを取得できても有効な画像を逃していた。

加えて、`article_heads`（scrape した `<head>` HTML と OG 画像 URL）と `image_proxy_cache`（取得画像を WebP 化した派生バイト列）には **保持期間の上限が事実上効いていない**状態だった。migration コメント上は `article_heads` に 12h TTL の cleanup ジョブが想定されていたが scheduler に未登録で、最古行は 3 か月以上前まで蓄積（14,180 行中 13,855 行が 7 日超）。`image_proxy_cache` の `expires_at` ベース cleanup（`CleanupExpiredImageProxyCache`）も `registry.go` に未登録で未稼働（18,679 行中 17,478 行が 7 日超）。これは Alt の disposable-projection 原則（read model は再構築可能・恒久保持を前提にしない、[[knowledge-system]] / IMPL_BASE）からの逸脱であり、stale 化による proxy 404 の温床にもなっていた。

## Decision

OG 画像の供給率を上げる 3 層と、派生キャッシュの保持窓を定める 1 層を採用する。

### 1. 読み取りパスで article_heads をフォールバック解決する（COALESCE + age-gate）

feeds の cursor 系クエリ（Unread / All / Favorite）で og:image を次式で解決する（`fetch_feed_driver.go` の `ogImageSelectExpr`）。

- `COALESCE(article_heads.og_image_url, feeds.og_image_url)` で **article_heads を優先**する。記事ページが提供する OG 画像は RSS の動的/失効 URL より安定しており、原因 2 の proxy 404 を直接減らす。article_heads 側が無い時のみ RSS 由来 URL に落ちる。
- 解決結果は `feeds.created_at` が 7 日以内のときのみ返し、それ以外は NULL とする（後述の保持窓と表示窓を一致させる）。

これにより、既に scrape 済みの OG 画像が追加 HTTP なしで即座に grid へ出る。実測で直近 7 日の feed 5,889 件のうち、RSS 画像のみだと 1,379 件だった解決数が 1,608 件に増えた（同一クエリで検証）。

### 2. ExtractOgImageURL の抽出候補を広げ、相対 URL を絶対化する

`og:image:secure_url` → `og:image:url` → `og:image` → `name=og:image` → `twitter:image` → `twitter:image:src` → `link[rel=image_src]` の優先順で候補を採り、最初に http/https の絶対 URL へ解決できたものを返す。相対・protocol-relative URL は記事ページの base URL で絶対化し、`data:` 等の利用不能スキームは次候補へフォールスルーする。http も受理する（proxy が server 側で取得し、クライアントへは https の proxy URL で配信するため）。

### 3. 不足分を能動的に充足する og-image-backfill ジョブ

直近 7 日以内・OG 画像が無い（`feeds.og_image_url` 空かつ `article_heads` 未充足）feed を batch（50 件/回）取得し、既存の `FetchArticleGateway`（SSRF 防御・per-host 5 秒レート制限・global 並列 3）で記事ページを取得 → 強化版 `ExtractOgImageURL` で抽出 → `article_heads` へ UPSERT → image proxy cache を warm する（`og-image-backfill` ジョブ, 30 分間隔）。`feeds.og_image_url`（RSS 由来カラム）は書き換えず、scrape 系の正は `article_heads` に集約する。実測で候補は 234 件。

### 4. 派生キャッシュを 7 日保持窓に収束させる（og-image-retention ジョブ）

`article_heads`（scrape した head HTML / OG 画像 URL）と `image_proxy_cache`（取得画像の派生バイト列）を、取得時刻（`created_at`）基準で 7 日を超えたものを定期 purge する（`og-image-retention` ジョブ, 6 時間間隔）。併せて未稼働だった `expires_at` ベースの TTL eviction も同ジョブから実行する。

保持窓の設計方針:

- 派生バイト列・scrape 済み HTML は **transient cache** として扱い、取得から 7 日で破棄する。feed / 記事を再ロードすれば on-demand で再生成され、`created_at` が更新されて保持窓も延びる。
- RSS 由来の `feeds.og_image_url` は **出典ページが公開している画像 URL への参照（メタデータ）であり派生コピーではない**ため purge しない。代わりに読み取り側の age-gate（決定 1）で、保持窓外の feed には proxy URL を出さない。
- 保持・表示・backfill・purge の窓はすべて 7 日に統一する（変更時は job 側の `ogImageRetentionWindow` と SQL の `INTERVAL '7 days'` を揃える）。

> 本 ADR は cache freshness・storage hygiene・disposable projection という運用上の理由に基づく工学的決定の記録である。Alt は第三者メディアの恒久的なミラー/アーカイブを構築せず、取得物を transient cache として保持し、出典への参照を保ったまま on-demand で再取得する設計を既定とする。本文は特定の法的評価・解釈を表明するものではない。

### 検討した代替案と却下理由

- **RSS 由来 `feeds.og_image_url` も 7 日超で NULL 化する**: 出典への参照（メタデータ）まで破棄すると、RSS 再収集に依存して復元せねばならず、RSS から落ちた古い feed の参照を不可逆に失う。参照は派生コピーではないため、読み取り age-gate で表示窓だけを制御すれば十分と判断し却下。
- **COALESCE を RSS 優先にする**: RSS の動的/失効 URL が proxy 404 を起こしやすく、より安定な記事ページ由来画像を後回しにすると 404 が残る。article_heads 優先を採用。
- **age-gate を handler 側（`enrichWithProxyURLs`）に置く**: enricher 系コードへの wall-clock 直書きは編集フックと相性が悪く、判定も実 DB なしには検証しづらい。読み取り SQL 側（`created_at` 基準）に寄せた。
- **backfill を全 feed 対象にする / forceRefresh で記事 usecase を再利用する**: 前者は外部 HTTP 負荷が過大、後者は user-scoped な usecase をバックグラウンドで回す層の歪みを生む。非 user-scoped な `FetchArticleGateway` を直接使い、直近 7 日・画像なしに限定した。

## Consequences

### Pros

- 既に scrape 済みの OG 画像が追加 HTTP ゼロで即座に grid へ出る（直近 7 日 feed で解決数 +229 を実測）。以降ユーザーの記事閲覧と backfill により単調に充足が進む。
- article_heads 優先の COALESCE が、RSS 動的 URL 起因の proxy 404 → placeholder を構造的に減らす。
- 抽出強化により、記事ページを取得できた際の OG 画像ヒット率が上がる（twitter:image / 相対 URL / http の取りこぼし解消）。
- 派生キャッシュが 7 日窓に収束し、単調増加が止まる（article_heads ~13.8k 行 / image_proxy_cache ~17.5k 行ぶんの滞留が解消見込み）。disposable-projection 原則に整合。
- 表示窓と保持窓が 7 日で一致し、挙動が説明可能になる。

### Cons / Tradeoffs

- `ogImageSelectExpr` が feed 行ごとに相関サブクエリ（article_heads → articles）を 1 本追加する。ページング（limit 20〜200）前提なので影響は限定的だが、より広い一覧へ転用する際は実行計画の再確認が要る。
- og-image-backfill は外部サイトへの能動的な HTTP 取得を発生させる。per-host 5 秒レート制限・global 並列 3・batch 50 件/回・直近 7 日限定で抑えているが、対象サイトの robots/レート方針とは引き続き整合を要する。
- 7 日窓は job 側 const と複数の SQL リテラルに分散している。変更時の同期漏れに注意（`INTERVAL '7 days'` を grep して揃える）。
- `feeds.og_image_url` を残すため、読み取り age-gate を外すと保持窓外でも RSS 参照が表示されうる。表示窓は age-gate に依存する点を運用上の前提として固定する。
- backfill が対象とできるのは articles 行が存在する feed に限る（`article_heads.article_id` の FK 制約）。記事未生成の feed は対象外。
