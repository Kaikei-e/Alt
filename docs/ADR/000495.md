# ADR-000495: Knowledge Home Lens 契約を source-first へ移行する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-backend (Go)
- alt-frontend-sv (TypeScript / SvelteKit)
- alt-butterfly-facade (Go, generated contract)

## Context

Knowledge Home の Lens は、Home をどの条件で読むかを保存する view contract である。しかし従来の Lens 契約は `feed_ids` を中心に設計されており、UI では feed UUID を露出しやすく、実際にユーザーが理解する「source を選ぶ」という行為とデータモデルがずれていた。

このずれには、以下の問題があった。

1. **意味モデルの不一致**: UI では source を選んでいるつもりでも、保存契約では `feed_ids` という内部表現が前面に出る。
2. **命名の一貫性不足**: frontend 型、proto、backend domain、repository、DB カラムで同じ概念を異なる名前で持ちやすく、仕様差分を生みやすい。
3. **Lens の再現性不足**: `queryText` が Lens に保存されても、Home 取得時に server-side filter として十分に効いておらず、保存した view を厳密に再現しにくい。
4. **移行コストの先送り**: `feed_ids` 命名を残すほど、将来 source-first UX を拡張する際に API と永続化の負債が増える。

Knowledge Home の canonical contract では、Lens は UI の一時的なフォームではなく、Home の読み方を記録する安定した契約として扱う必要がある。

## Decision

**Knowledge Home Lens の保存契約を `feed_ids` から `source_ids` へ移行し、`queryText` を含む source-first / query-aware な filter contract を採用する。**

### 検討した代替案

| 案 | 評価 |
|---|---|
| A. `feed_ids` を残し UI 側だけ source と呼ぶ | 表面的には速いが、契約と実装の意味不一致を温存する |
| B. `source_ids` へ全面移行する（採用） | 命名と責務が揃い、将来の拡張に耐える |
| C. Lens を自由 JSON 条件に置き換える | 柔軟だが型安全性と移行容易性を損なう |

案 B を採用した。内部表現も公開契約も同じ言葉に揃え、Home 側の filter と Lens 側の保存値を一致させる。

## Consequences

### Pros

- **契約の意味が明確になる**: source を選ぶ UI と `source_ids` を保存する API/DB が一致する。
- **実装の一貫性が上がる**: proto、frontend、backend、DB migration が同じ概念名で揃い、仕様の読み替えが減る。
- **Lens の再現性が上がる**: `queryText`、`sourceIds`、tag、time window を server-side で適用し、保存した Home view を再現しやすくなる。
- **移行が明示的になる**: `source_ids_json` への rename migration により、旧データを曖昧な互換レイヤーに閉じ込めずに済む。

### Cons / Tradeoffs

- **移行の影響範囲が広い**: generated proto、frontend 型、backend domain、repository、migration を一度に揃える必要がある。
- **旧命名との共存期間が短い**: 運用中コードやテストで `feed_ids` 前提が残っていると、一時的に修正量が増える。
- **検索条件はまだ単純**: `queryText` はまず title / summary / tags に対する部分一致で実装しており、専用検索基盤ほどの精度や表現力はない。
