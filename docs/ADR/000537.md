# ADR-000537: Knowledge Sovereign を独立 service + 独立 DB へ抽出する

## Status

Accepted

## Date

2026-03-23

## Affected Services

- alt-backend (Go) — Sovereign gateway の driver 切り替え、dual-path 削除
- knowledge-sovereign (Go) — 新設

## Context

[[000532]] で Knowledge Sovereign を durable knowledge state の single owner として設計し、Phase 0-3（[[000534]], [[000535]], [[000536]]）で alt-backend 内の logical module として段階的に実装を進めてきた。Phase 3 完了時点で以下が達成済みだった:

- **Write path consolidation 100%**: projector, recall, dismiss, reproject, lens 全8経路が Sovereign CurationMutator / WriteServicePort 経由
- **移行安全基盤**: reconciliation job（30分間隔）、idempotency policy、rollback（ActivateVersion）
- **Transport contract**: `SovereignEventEnvelope` 型で producer-Sovereign 境界を形式化
- **Cutover readiness**: `CutoverReadinessUsecase` で4つの quantitative gate を判定
- **DB ownership**: 12 Sovereign テーブル / 5 producer テーブル / 1 共有テーブルの分類完了

しかし、Sovereign はまだ alt-backend のプロセス内で動作しており、DB も alt-db を共有していた。ADR-000532 の最終目標である「独立 service + 独立 DB」は未達成だった。Phase 4 では、この物理的な抽出を実行する。

## Decision

Knowledge Sovereign を `knowledge-sovereign` として独立した deployable service と独立 DB に物理抽出する。3つの意味的カテゴリに分けて実施する。

---

### A. 独立 service の新設 — knowledge-sovereign

**目的**: alt-backend から分離した独立プロセスとして Knowledge Sovereign を稼働させる。

#### Service 設計

Alt の Compose-first 原則に従い、独立 Go service を新設する。ディレクトリ構成は alt-backend の Clean Architecture パターンに準拠する:

```
knowledge-sovereign/
├── app/
│   ├── main.go        # config → DB connect → handler 配線 → server 起動 → signal 待機
│   ├── config/        # 環境変数ベースの設定（DATABASE_URL, LISTEN_ADDR, METRICS_ADDR）
│   ├── handler/       # Health endpoint + 将来の Connect-RPC handler
│   ├── domain/        # Sovereign domain 型（Phase 0-3 で定義済みの型を移設予定）
│   ├── port/          # Sovereign port interfaces
│   ├── usecase/       # KnowledgeWriteService, CutoverReadiness
│   ├── gateway/       # DB gateway
│   └── driver/        # sovereign_db driver
├── Dockerfile.sovereign
├── migrations/
│   └── 00001_initial_schema.sql
└── CLAUDE.md
```

`main.go` は bp-go のベストプラクティスに準拠: config 読込 → deps 接続 → handler 配線 → server 起動 → signal 待機。ビジネスロジックは含まない。

2つの HTTP server を起動する:
- `:9500` — メイン RPC server（Connect-RPC、将来の mutation 受付）
- `:9501` — メトリクス / ヘルスチェック server

#### Compose 統合

`compose/sovereign.yaml` に以下を定義:
- `knowledge-sovereign-db`: PostgreSQL 16 Alpine、`sovereign` profile
- `knowledge-sovereign`: Go service、`knowledge-sovereign-db` に依存、`sovereign` profile

profile を `sovereign` に分離し、通常の `docker compose up` では起動しない。明示的に `--profile sovereign` または `-f compose/sovereign.yaml` を指定して起動する設計にした。

この方式を選んだ理由: Phase 4 の cutover が完了するまでは、alt-backend が Sovereign の役割を果たし続ける。sovereign profile を分離することで、既存の開発・運用フローを壊さずに段階的に移行できる。

---

### B. 独立 DB の構築 — knowledge-sovereign-db

**目的**: Sovereign 所有テーブルを alt-db から分離し、独立した PostgreSQL インスタンスに収容する。

#### 移管対象テーブル

Phase 3 の `sovereign_db_ownership.md` に基づき、12テーブルを移管する:

| カテゴリ | テーブル | 移管方式 |
|---|---|---|
| Event store | `knowledge_events` | snapshot + catch-up（source of truth） |
| Projection | `knowledge_home_items`, `today_digest_view`, `recall_candidate_view` | replay で再構築 |
| Admin | `knowledge_projection_versions`, `knowledge_projection_checkpoints` | 移管 or 再開始 |
| Audit | `knowledge_reproject_runs`, `knowledge_backfill_jobs`, `knowledge_projection_audits` | 移管（履歴） |
| Curation | `knowledge_lenses`, `knowledge_current_lens`, `knowledge_lens_versions` | 移管（再構築不可） |

Producer 所有テーブル（`summary_versions`, `tag_set_versions`, `articles`, `feeds`, `feed_links`）は alt-db に残留する。`knowledge_user_events` と `recall_signals` は共有アクセスとして CDC or read replica で対応する。

#### 初期 migration

`00001_initial_schema.sql` に全12テーブルを統合した単一ファイルを作成する。alt-db の18件の migration ファイルから後続の ALTER TABLE 変更（`projection_version`, `supersede_state`, `summary_state`, `dismissed_at` 等）を全て fold-in した最終形のスキーマを定義する。

この方式を選んだ理由: 新しい DB は白紙から始まるため、段階的な ALTER TABLE の履歴を再現する必要がない。最終形のスキーマを1ファイルで定義する方が明瞭で保守しやすい。

#### DB 分離ポリシー

- **Replay-only 方式を採用**: dual-write は行わない。`knowledge_events` が source of truth であり、projection テーブルは event replay から再構築可能
- **Curation state は直接移管**: `knowledge_lenses` 等はユーザー固有の状態であり、replay では再構築できない。`pg_dump` / `pg_restore` で直接移管する

---

### C. Cutover 手順と運用基盤

**目的**: alt-backend → knowledge-sovereign への active writer 切り替えを安全に実施するための手順と rollback 基盤を整備する。

#### Cutover runbook

`docs/runbooks/sovereign-cutover.md` に9ステップの cutover 手順を文書化:

1. 最終 replay catch-up
2. Shadow compare（diff がしきい値内であることを確認）
3. Freeze（event append 一時停止）
4. 最終 catch-up（freeze 中の差分）
5. Active writer 切り替え（`SOVEREIGN_WRITER_ACTIVE=true`）
6. Post-cutover compare
7. Freeze 解除
8. Rollback 窓（30分〜1時間）の監視
9. Rollback 窓終了、旧 writer 無効化

#### Feature flag による切り替え

切り替えは `SOVEREIGN_WRITER_ACTIVE` 環境変数で制御する。DI container の Sovereign 注入を nil にするのではなく、driver レベルで「ローカル DB direct」と「knowledge-sovereign Connect-RPC client」を切り替える。

この方式を選んだ理由: DI container の変更は再ビルドが必要だが、環境変数なら `docker compose restart` で即時切り替え・即時 rollback が可能。

#### Rollback

3段階の rollback 戦略を定義:

| 段階 | 操作 | リスク |
|---|---|---|
| cutover 前 | knowledge-sovereign を停止するだけ。alt-backend は dual-path で動作継続 | なし |
| cutover 後（rollback 窓内） | `SOVEREIGN_WRITER_ACTIVE=false` に戻す。alt-backend restart で旧パスに復帰 | 低 |
| dual-path 削除後 | git revert + 逆 replay が必要 | 高（最悪ケース） |

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|---|---|---|
| dual-write 方式のデータ移行 | 一貫性が高い | 実装が複雑で、write amplification による性能劣化のリスクがある |
| 既存 alt-backend プロセス内で DB だけ分離 | service 追加なし | ADR-000532 の「独立 service + 独立 DB」方針に反する。将来のスケーラビリティが制限される |
| Kubernetes でのデプロイ | スケーラビリティ | CLAUDE.md の Compose-first 原則に違反。運用複雑度が大幅に増加する |
| migration ファイルを alt-db から段階的にコピー | 履歴が残る | 18件の ALTER TABLE 履歴を新 DB で再現する意味がない。fold-in した最終形が明瞭 |

## Consequences

### Pros

- **ADR-000532 の最終目標達成** — Knowledge Sovereign が独立 service + 独立 DB として物理的に分離される
- **Compose-first 原則の維持** — profile 分離で既存の開発フローを壊さない
- **Replay-only 方式** — dual-write の複雑さを回避し、`knowledge_events` を single source of truth として保持
- **段階的 cutover** — freeze window は数秒〜数十秒に抑えられ、rollback は環境変数の変更で即時可能
- **Clean Architecture 準拠** — 新 service も `domain/port/usecase/gateway/driver` のレイヤー構造を維持
- **alt-backend の simplification** — Phase 4 完了後、dual-path コードの削除により alt-backend のコードが簡素化される

### Cons / Tradeoffs

- **Service 数の増加** — compose stack に2サービス（knowledge-sovereign + knowledge-sovereign-db）が追加され、リソース使用量が増加する
- **ネットワークレイテンシ** — alt-backend → knowledge-sovereign の通信が DB direct から Connect-RPC に変わるため、レイテンシが微増する。ただし compose network 内の通信であり、影響は最小
- **Curation state の直接移管リスク** — `knowledge_lenses` 等は replay で再構築できないため、移管の失敗は手動復旧が必要
- **dual-path 削除前の一時的な複雑さ** — cutover 後、rollback 窓が閉じるまでは alt-backend に旧パスのコードが残り続ける
- **Phase 4 完了後の dual-path 削除** — タスク 4-7（dual-path 削除）は rollback 窓終了後に別途実施する必要がある。この段階で初めて不可逆になる
