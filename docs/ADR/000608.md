# ADR-000608: article-scoped クエリのキーワードカバレッジ誤検知を修正し SubIntent 別プロンプト構造分岐を導入する

## Status

Accepted

## Date

2026-03-29

## Affected Services

- rag-orchestrator (Go)

## Context

Ask Augur の品質ゲートが暴走し、**直近の全回答が `stream_short_answer_hard_stop` で棄却される**サービス停止同等の障害が発生していた。

### 障害の直接原因: `checkKeywordCoverage` の二重バグ

`output_validator.go` の `checkKeywordCoverage` 関数に2つの独立したバグが存在した。

**バグ 1: article-scoped クエリのメタデータ汚染**

article-scoped クエリの raw query は以下の形式を取る:

```
Regarding the article: Supply Chain Security for Developers: Protecting Your CI/CD Pipeline in 2026 - DEV Community [articleId: 7a68810e-...]

Question:
今後の業界への影響は？
```

`checkKeywordCoverage` はこの全文字列を `strings.Fields` で分割し、英語の記事タイトルからキーワードを抽出していた（"Supply", "Chain", "Security", "Developers" 等）。回答は日本語で生成されるため、これらの英語キーワードは回答中に出現せず、カバレッジ率が構造的に低くなる。

**バグ 2: 日本語テキストの単語分割不能**

`strings.Fields` はスペースで分割するため、日本語テキスト `今後の業界への影響は？` は分割されず**1つの巨大トークン**として扱われる。このトークンが回答中に完全一致しなければカバレッジ 0% となり、50% 閾値を下回って `low_keyword_coverage` フラグが発火する。

### 結果

以下のログが示すように、全 article-scoped 回答が hard stop された:

- `今後の業界への影響は？` → 758 rune 生成 → `low_keyword_coverage` + `incoherent_ending` → hard stop
- `反論はある？` → 757 rune 生成 → `low_keyword_coverage` → hard stop
- `サプライチェーン攻撃のそのほかの事例を教えて` → 124 rune → hard stop
- 直近12件全てが同様にhard stop

内容的には妥当な回答が生成されていたにもかかわらず、品質ゲートの誤検知で全て棄却されていた。

### 副次的問題: プロンプト構造の混在

`prompt_builder.go` の `writeFullInstructions` は、SubIntent（critique, detail, implication 等）が検出されていても常に `## 回答構造` セクション（概要/詳細/まとめ）を出力していた。SubIntent 固有の指示（「要約ではなく分析的に回答せよ」）と矛盾するシグナルをモデルに送っていた。

## Decision

### 1. `checkKeywordCoverage` の修正

**`extractUserQuestion` の導入**: article-scoped クエリから `\n\nQuestion:\n` セパレータ以降のユーザー質問のみを抽出し、記事メタデータを除外する。非 article-scoped クエリはそのまま通過。

**`isCJKDominant` の導入**: 抽出後のクエリが CJK 文字 30% 以上（ひらがな・カタカナ・漢字・全角文字）の場合、単語ベースのキーワードカバレッジチェックをスキップする。日本語・中国語のような膠着語・孤立語ではスペース分割による単語照合が構造的に機能しないため。

30% 閾値は rag-orchestrator の既存ロケール検出ヒューリスティック（`retrieval/allocate.go`）と一致させた。

### 2. SubIntent 別プロンプト構造分岐

`writeFullInstructions` の `## 回答構造` セクション（概要/詳細/まとめ）を以下の条件でスキップする:

- `SubIntentType != SubIntentNone` — SubIntent 固有の指示が既にある
- `IntentType == IntentCausalExplanation` — 直接的要因/構造的背景/不確実性 の独自構造を持つ
- `IntentType == IntentFactCheck` — 主張/根拠/判定 の独自構造を持つ

`SubIntentNone` かつ上記以外の IntentType（General, Temporal, Comparison, DeepDive 等）では従来通り概要/詳細/まとめ構造を出力する。

### 検討した代替案

**代替案A: `low_keyword_coverage` フラグ自体の廃止** — 英語の非 article-scoped クエリでは有効な品質指標であるため、全面廃止ではなく条件付きスキップを採用。

**代替案B: 形態素解析（MeCab 等）の導入** — 日本語の正確な単語分割にはライブラリ依存が必要。CGO 依存またはプロセス間通信のオーバーヘッドに対し、CJK スキップの方がシンプルで十分な効果がある。将来的に日本語固有の品質指標が必要になった場合に再検討。

**代替案C: n-gram ベースのオーバーラップ計算** — CJK テキスト向けに character n-gram（bi-gram/tri-gram）でカバレッジを計算する方法。精度は高いが、現時点では CJK スキップで十分に false positive が排除できるため、複雑さに見合わない。

## Consequences

### Pros

- **即時サービス復旧**: article-scoped 回答の全殺し状態が解消。ベンチマーク結果 4/4 PASS（修正前 0/4）
- **英語クエリの品質ゲート維持**: 非 CJK の英語クエリでは従来通り `low_keyword_coverage` が機能
- **プロンプト一貫性向上**: SubIntent が検出された場合、モデルは矛盾する指示を受けなくなる
- **`stream_short_answer_hard_stop` 発生数**: 12件/期間 → 0件/期間

### Cons / Tradeoffs

- **CJK クエリの keyword coverage チェックが無効化**: 日本語のクエリに対するキーワード網羅性チェックが効かなくなる。将来的に n-gram ベースまたは semantic similarity ベースの品質指標で補完が必要
- **SubIntent 検出精度への依存増加**: SubIntent が正しく検出されない場合、generic な概要/詳細/まとめ構造にフォールバックする。SubIntent 検出漏れが回答品質に直結する度合いが増した
- **`extractUserQuestion` が `\n\nQuestion:\n` セパレータに依存**: このフォーマットが変更された場合、keyword coverage が再び全文マッチに退行する
