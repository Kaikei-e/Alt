# ADR-000802: pki-agent healthcheck に netns 孤立検出を追加し、自己治癒は外部 cascade に委ねる前提を確定する

## Status

Accepted

## Date

2026-04-20

## Affected Services

- **alt-pki-agent** — `cmd/pki-agent/healthcheck.go` に `probeNetns()` を新設し、`runHealthcheck` が `probeMetrics → probeNetns → probeProxy` の順に probe するよう変更。`listInterfaces` を差し替え可能な package variable とし、ユニットテストで healthy / loopback-only / interface-down / list-error の 4 ケースを固定化
- **acolyte-orchestrator** — コードは変更なし。pki-agent サイドカーの `State.Health.Status` が機能 liveness を反映するようになり、ListReports 502 の recurrence が 15s 以内に運用に surface される

## Context

2026-04-20 にユーザが再びブラウザ DevTools で 502 を報告:

```
api/v2/alt.acolyte.v1.AcolyteService/ListReports:1
Failed to load resource: the server responded with a status of 502 ()
```

[[PM-2026-030]] → [[PM-2026-031]] → [[000782]] に続き **4 度目** の同一パターン発火。以後に採用された 2 経路の恒久策:

- [[000783]] alt-deploy 側で `scripts/cascade-recreate-netns-sidecars.py` を pull-deploy に追加する案 — 別リポジトリ PR、本 ADR 執筆時点で **未実装**
- [[000784]] pki-agent に TLS listener self-probe + fail-fast を追加する案 — 実装は入っているが **netns 孤立を検知できていなかった**

今回の recurrence は、[[000784]] の "実装済み" とされた fail-fast が実際には効いていなかった事実を示した。原因を 2 点に分けて記録する。

### 問題 1: probeProxy(127.0.0.1:9443) は netns 孤立を検知できない

`pki-agent/cmd/pki-agent/healthcheck.go` の既存 `probeProxy` は `net.Dial("tcp", "127.0.0.1:PROXY_LISTEN")` で自身の reverse-proxy listener に loopback dial する。しかし:

- 親 `acolyte-orchestrator` が `--force-recreate` されると、pki-agent サイドカーは `HostConfig.NetworkMode=container:<旧 id>` を持ったまま残り、**旧 netns に居続ける**
- 旧 netns では eth0 が消え `lo` のみ残るが、旧 netns の `127.0.0.1:9443` には pki-agent 自身の listener がまだ bind している
- つまり `probeProxy` は **常に成功**する。orphan でも green を返す

2026-04-20 の実測 (`docker inspect`):

| オブジェクト | 値 |
|---|---|
| acolyte-orchestrator.Id | `c703fde8…` (新 netns, IP 172.18.0.58) |
| pki-agent-acolyte-orchestrator.HostConfig.NetworkMode | `container:dea90fb9…` (旧 netns、2 日前に退役) |
| pki-agent 内 `ip -4 addr` | `127.0.0.1/8 scope host lo` のみ、eth0 不在 |
| pki-agent healthcheck | **exit 0** (orphan でも green) |
| BFF → acolyte:9443 | `dial tcp 172.18.0.58:9443: connect: connection refused` |

### 問題 2: Docker daemon restart は `network_mode: service:<parent>` を再解決しない

これは本 ADR 執筆中の live test で初めて判明した事実。[[000784]] の想定「3 回連続失敗で `os.Exit(1)` → `restart: unless-stopped` が新親の netns に再 attach する」は **成立しない**。実測手順:

1. pki-agent を rebuild + up。orphan 検出ロジック (後述 Decision) を入れたイメージで起動
2. `docker compose up -d --no-deps --force-recreate acolyte-orchestrator` で親のみ force-recreate。sidecar は `HostConfig.NetworkMode=container:<旧 id>` のまま孤立
3. sidecar の probe が 3 回連続 fail → `os.Exit(1)` が走る
4. Docker daemon の `restart: unless-stopped` が sidecar 再起動を試みる
5. **再起動失敗**: `Error response from daemon: joining network namespace of container: No such container: <旧 id>`
6. sidecar は `State.Status=exited, ExitCode=1, Restarting=false` で停止。**手動 `docker compose up --force-recreate` を要する**

原因: `network_mode: service:<parent>` は **compose がコンテナ作成時に一度だけ container id を解決**する。その後は HostConfig に焼き込まれた container id を Docker daemon が参照するが、親が destroy された時点で旧 container は garbage-collect され、restart で join 先が見つからなくなる。`docker restart` / daemon auto-restart は compose の service reference を reload しない。**compose レイヤの force-recreate だけが新 parent id を再解決できる**。

従って pki-agent 内の self-exit は、孤立を治すどころか「exited + restart 不能」の zombie 化を招く。[[000784]] の Decision は根本的に不成立だった。

### docker/compose issue 群の裏付け

- [docker/compose#10263](https://github.com/docker/compose/issues/10263): `network_mode: service` の child は親 restart 時に network を失う。feature request 扱い
- [docker/compose#6626](https://github.com/docker/compose/issues/6626): parent 再起動時の child 再 attach は manual
- [docker/compose#7765](https://github.com/docker/compose/issues/7765): `cannot join network of a non-running container` の既知現象

いずれも compose spec の仕様上の限界として残っている。

## Decision

**pki-agent の healthcheck subcommand に `probeNetns()` を追加し、netns 孤立を Docker HEALTHCHECK 経由で可視化する。自己 exit は行わない。自己治癒は外部 cascade ([[000783]] alt-deploy PR) に完全に委ねる前提を確定する。**

### 採用構成

1. **`probeNetns()` を新設** — `net.Interfaces()` を列挙し、loopback 以外で UP かつ routable (非 loopback, 非 unspecified) な address を 1 つ以上持つ interface の存在を要求する。無ければ `netns orphan: no non-loopback interface up with a routable address` を返す。DNS/exec/outbound を引かない。追加攻撃面ゼロ

2. **`listInterfaces` を package variable として export** — テストで差し替え可能にし、healthy / loopback-only / iface-down / list-error の 4 ケースを deterministic に検証する (`TestProbeNetns_*` + `TestRunHealthcheck_NetnsOrphanFailsHealthcheck`)

3. **`runHealthcheck` の probe 順を `probeMetrics → probeNetns → probeProxy`** に変更。PROXY_LISTEN 設定時のみ後 2 者を走らせる。最具体的な失敗を surface する順 (metrics 死 > netns 孤立 > listener 死)

4. **self-probe tick loop / `os.Exit(1)` は採用しない**。上述 "問題 2" により、self-exit は zombie 化を招くため逆効果。Docker HEALTHCHECK が 15s 間隔で `pki-agent healthcheck` を叩くため、孤立は 15s 以内に `State.Health.Status=unhealthy` に反映される。これが外部 watchdog / 監視 / 運用の signal になる

5. **自己治癒は外部 cascade に委ねる** — 以下のいずれかが発火すれば healthy に戻る:
   - [[000783]] alt-deploy の pull-deploy cascade (primary、未実装、別リポ PR)
   - 運用手動: `docker compose -f compose/compose.yaml -p alt up -d --no-deps --force-recreate pki-agent-<svc>`
   - 既存 `scripts/cascade-pki-sidecars.sh` (現状 `scripts/deploy.sh` deprecation ([[feedback_no_raw_compose_build]]) で呼ばれない。[[000783]] 実装で置き換える想定)

6. **probe cadence は業界標準**: Docker HEALTHCHECK の `interval: 15s` を維持 (Kubernetes default `periodSeconds: 10`、Dapr sidecar 10s、Docker default 30s の中央値)。`retries: 3` で合計 45s の transient 吸収

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| self-probe goroutine + `os.Exit(1)` (ADR-000784 の literal 実装) | live test で zombie 化することを確認。sidecar が `exited + can't restart` になり、user-visible 502 を **悪化**させる |
| pki-agent に docker.sock を mount し、自身を compose 経由で force-recreate | docker.sock は privilege escalation path。全コンテナへの root access を sidecar に与えることになる。trust boundary を大きく弱める |
| `willfarrell/docker-autoheal` 類似の watchdog コンテナ | autoheal は `docker restart` を使う。上記 "問題 2" で同じ zombie 化を招くため効かない。compose force-recreate を内包する custom watchdog が必要だが、それは [[000783]] alt-deploy PR のスコープそのもの |
| `network_mode: service:` をやめて sidecar に独自 IP を振る | pki-agent は親の loopback (`127.0.0.1:8090`) を proxy 宛先にしているため、netns 共有が前提。別 IP だと TLS termination semantics が変わる。別 ADR で要検討 |
| pki-agent と親を同一コンテナに統合 | Go + Python のランタイム統合は大規模改修。短期 fix には不適 |
| healthcheck を `openssl s_client` 等の TLS handshake probe に切り替え | probe 対象が sidecar 自身の loopback である限り netns 孤立は検知できない。`probeNetns` の interface 列挙が原理的に正しい |

### 範囲外 (今回やらないこと)

- [[000783]] alt-deploy 側 cascade step の実装 — 別リポジトリ `Kaikei-e/alt-deploy` PR。本 ADR landed 後の次アクション
- `scripts/cascade-pki-sidecars.sh` の再評価 — 現状 deprecated `scripts/deploy.sh` からのみ呼ばれる dead code。[[000783]] 実装で新経路に置き換えるか、削除するかを別 ADR で decide
- 運用 runbook への追記 — 緊急復旧手順 (`docker compose up --no-deps --force-recreate pki-agent-<svc>`) を `docs/runbooks/pki-agent-netns-recovery.md` に起こす作業は別 PR
- Prometheus alert — `State.Health.Status=unhealthy` のメトリクス出しは rask-log-aggregator 側の設定で別扱い

## Consequences

### Pros

- **Docker HEALTHCHECK が機能 liveness を反映する**。cert 鮮度だけ見て green を返す silent failure モードが消える。[[000782]] の「Pact green + `/health` green + コンテナ healthy、だけど機能停止」状態が技術的に成立しなくなる
- **4 度目発火の ([[PM-2026-030]] → [[PM-2026-031]] → [[000782]] → 今回) の根本原因が local-only の probe で閉じる**。DNS / outbound / docker.sock に依存せず、攻撃面を増やさない。unit test で deterministic に再現できる
- **[[000784]] の設計誤謬を live test で同定した記録が残る**。compose `network_mode: service:` が force-recreate 追従しない事実と、daemon restart が service reference を reload しない事実が、以後の sidecar 設計議論で前提として使える
- **実装は `healthcheck.go` 1 ファイル + test 増分のみ**。review 面積が小さく、ロールバックは code revert のみ (cert / proxy / rotator ロジックには触れていない)
- **probe の semantics が将来の `docker compose` 改善にも forward-compatible**。仮に compose spec が force-recreate cascade をサポートした場合でも、probeNetns は interface layer を見るので false positive にならない

### Cons / Tradeoffs

- **自己治癒は本 ADR の範囲外に据え置く**。[[000783]] alt-deploy PR が landed するまで、netns 孤立からの復旧は**運用手動**に依存する。単一ホスト dev では 1 コマンドで済むが、SRE オンコールを叩く場合は `docker compose up --force-recreate` の再周知が要る
- **healthcheck が unhealthy を出しても `restart: unless-stopped` は反応しない**。Docker の restart policy は exit code ベースで、`State.Health.Status=unhealthy` では restart しない。自動復旧には別途 watchdog が要るという構造的事実が表に出る (これ自体は improve)
- **probe 自体は 15s 周期だが、初回 orphan から unhealthy 反映までは 15 × 3 = 45s の遅延**。`retries: 3` を減らせば早まるが、transient error の偽陽性が増える。現状のバランスを維持する
- **`listInterfaces` を package variable にしたことでテスト並列時の race 懸念**。`stubInterfaces` ヘルパ + `t.Cleanup` で `prev` 復元しているが、`-race` でテストを並列実行した場合は書き換えが競合する。現状 `go test ./... -race` では `t.Parallel()` を呼んでいないためシリアル、問題は表面化していない。parallel 化する際は atomic.Value などで包む
- **netns 孤立以外の `eth0 down` (docker network driver 障害、インスタンス network namespace の別要因破損) でも probe が fail する**。本 ADR は「eth0 の有無 = 機能 liveness」と扱うため、原因問わず fail-closed に寄る。false positive のトリアージは別途 log/metrics で絞り込む
- **[[000784]] の Decision 2-3 (tick loop self-probe, threshold=3 fail-fast) は事実上 deprecated**。本 ADR でロジックが除去される。[[000784]] Status は amending note を追加するか、superseded by [[000802]] に書き換えるかを別途 decide (本 ADR では触れない)
