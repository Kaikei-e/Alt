# ADR-000454: Knowledge Home 確定回帰を修正しストリーム接続・キャンセル伝播・JSONB安全性を強化する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go)
- alt-frontend-sv (SvelteKit)

## Context

Knowledge Home のリリース後、以下の 3 カテゴリの確定回帰（regression）が検出された。

### 1. キャンセルエラーの伝播漏れ

`GetKnowledgeHomeUsecase.Execute` はアイテム取得とダイジェスト取得を順次実行するが、エラー変数 `err` を共有していたため、最後に代入された `digestErr` だけが評価されていた。クライアントがリクエストをキャンセルした際にアイテム取得側で `context.Canceled` が発生しても、ダイジェスト取得が正常終了すると `err == nil` となり、キャンセルが無視されて不完全なレスポンスが返却されていた。

### 2. ストリーム接続時の履歴再生

`StreamKnowledgeHomeUpdates` は `lastSeq = 0` から開始していたため、接続直後に全履歴イベントが再生され、不要なトラフィックとフロントエンドの過剰再描画が発生していた。また、`lens_id` パラメータのバリデーションが行われておらず、不正な UUID がそのまま通過していた。

### 3. JSONB 空値によるクエリエラー

`knowledge_reproject_runs` テーブルの JSONB カラム（`checkpoint_payload`, `stats_json`, `diff_summary_json`）に空の `json.RawMessage`（`nil` または `[]byte{}`）を渡すと、PgBouncer 互換の文字列書き込み時に空文字列となり、PostgreSQL が `invalid input syntax for type jsonb` エラーを返していた。

### 4. フロントエンドの副作用

- dismiss アクション後に後続処理（`articleId` 抽出等）が走り続けていた
- `DegradedModeBanner` が `home.degraded`（boolean）を参照しており、3 値の `serviceQuality` と不整合だった
- `hooks.server.ts` のレスポンシブリダイレクトがインライン実装でテスト不可能だった

## Decision

以下の方針で 4 カテゴリの回帰を一括修正する。

### 1. キャンセル判定の分離（Usecase 層）

`err` 変数を `itemsErr` / `digestErr` に分離し、両方のエラーを検査する `requestContextError()` ヘルパーを導入する。

```go
func requestContextError(errs ...error) error {
    for _, err := range errs {
        switch {
        case errors.Is(err, context.Canceled):
            return context.Canceled
        case errors.Is(err, context.DeadlineExceeded):
            return context.DeadlineExceeded
        }
    }
    return nil
}
```

`context.Canceled` / `context.DeadlineExceeded` のいずれかが含まれていれば即座に `nil, err` を返却し、クライアント切断を正しく伝播する。通常のエラー（DB接続失敗等）は従来どおり `Degraded` モードとして処理する。

### 2. ストリーム開始位置の最適化（Handler 層 + Port 層）

`LatestKnowledgeEventSeqForUserPort` を新設し、接続開始時に `COALESCE(MAX(event_seq), 0)` を取得して `lastSeq` の初期値とする。

```go
func (h *Handler) initialStreamSeq(ctx context.Context, userID uuid.UUID) (int64, error) {
    if h.latestSeqPort == nil {
        return 0, nil
    }
    return h.latestSeqPort.GetLatestKnowledgeEventSeqForUser(ctx, userID)
}
```

これにより接続直後の履歴再生を防止し、接続後に発生した新規イベントのみがストリーム配信される。`latestSeqPort` は `NewHandler` で `eventsForUserPort` がインターフェースを満たす場合に自動検出する（型アサーション）。

加えて、`lens_id` を接続開始時に `parseUUID()` でバリデーションし、不正な UUID には `InvalidArgument` を返す。

### 3. JSONB 空値のフォールバック（Driver 層）

`jsonbStringOrEmptyObject()` ヘルパーを導入し、空の `json.RawMessage` を `{}` にフォールバックする。

```go
func jsonbStringOrEmptyObject(raw json.RawMessage) string {
    if len(raw) == 0 {
        return "{}"
    }
    return string(raw)
}
```

`CreateReprojectRun` / `UpdateReprojectRun` の JSONB カラム書き込み時にこのヘルパーを経由させ、PgBouncer 互換の文字列書き込み（ADR-417）との組み合わせで安全に動作するようにした。

### 4. フロントエンド修正

- **dismiss の早期 return**: `handleAction` で `type === "dismiss"` の場合、`trackAction` → `dismissItem` → `return` の順で処理し、後続の `articleId` 抽出や `goto` ロジックに到達しないようにした
- **serviceQuality prop**: `home.degraded`（boolean）ではなく `home.serviceQuality !== "full"` で表示判定し、`DegradedModeBanner` に `serviceQuality` を prop として渡すように変更。desktop / mobile で統一された条件で表示
- **リダイレクトロジック抽出**: `RESPONSIVE_REDIRECTS` マップと `resolveResponsiveRedirect()` 関数を `redirect-resolver.ts` に抽出し、クエリパラメータ保持ロジックを含む純粋関数としてテスト可能にした（[[000453]] で導入された設計方針の継続）

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|--------|------|-----------|
| `errgroup` で items/digest を並列取得 | 高速化が見込める | 現時点では DB クエリが軽量で並列化のメリットが小さい。キャンセル伝播の修正が主目的であり、並列化は別 ADR で検討 |
| ストリーム開始時に直近 N 件のイベントを配信 | 切断中のイベントを補完できる | フロントエンドが unary API で初期ロードしているため、ストリームでの履歴補完は二重取得になる |
| JSONB カラムに `DEFAULT '{}'` を設定 | スキーマレベルで安全 | マイグレーション変更が必要で影響範囲が広い。アプリケーション層でのガードで十分 |

## Consequences

### Pros

- クライアント切断（`context.Canceled`）がアイテム取得側・ダイジェスト取得側の両方で正しく検出され、不完全なレスポンスの返却が防止される
- ストリーム接続直後の履歴再生が解消され、接続時のネットワークトラフィックとフロントエンドの再描画コストが削減される
- 不正な `lens_id` が接続開始時にバリデーションされ、下流レイヤーへの不正入力伝播が防止される
- JSONB 空値に対する防御が Driver 層に集約され、今後の JSONB カラム追加時にも同じパターンを適用できる
- `DegradedModeBanner` が 3 値の `serviceQuality` に対応し、将来の品質段階追加（例: `minimal`）にも拡張可能
- `resolveResponsiveRedirect` が 7 件のテストケースでカバーされ、リダイレクトルール追加時の安全性が向上
- 新規テスト 5 件（キャンセル伝播 2 件、不正 lens_id 1 件、initialStreamSeq 1 件、空 JSONB 1 件）が回帰防止として機能

### Cons / Tradeoffs

- `initialStreamSeq` は接続開始時に追加の DB クエリ（`MAX(event_seq)`）を発行する。ストリーム接続数が多い環境ではインデックスチューニングが必要になる可能性がある
- `LatestKnowledgeEventSeqForUserPort` の検出が型アサーション（`eventsForUserPort.(LatestKnowledgeEventSeqForUserPort)`）に依存しており、Gateway が新しいインターフェースを実装し忘れるとフォールバック（`lastSeq = 0`）に戻る
- `RESPONSIVE_REDIRECTS` マップが `hooks.server.ts` から `redirect-resolver.ts` に移動したため、リダイレクトルールの追加時は新しいファイルを編集する必要がある
