# ADR-000457: Knowledge Home dismiss 可視性を後続 upsert で巻き戻さない

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-backend (Go)

## Context

Knowledge Home では `HomeItemDismissed` が `knowledge_events` に追記され、projector が `knowledge_home_items.dismissed_at` を更新する構成にしていた。しかし 2026-03-18 の実運用では、dismiss 後に対象記事がリロードで再表示される事象が継続した。

調査の結果、問題は「dismiss イベントが失われること」ではなく、「dismiss 済み row が後続 upsert で可視状態に戻ること」だった。対象記事では `HomeItemDismissed` が複数記録されていた一方、`knowledge_home_items.updated_at` は dismiss より後の時刻に更新され、`dismissed_at` は `NULL` のままだった。

Knowledge Home の projection は `ArticleCreated`、`SummaryVersionCreated`、`TagSetVersionCreated`、supersede 系イベント、reproject によって何度も `UpsertKnowledgeHomeItem` を通過する。ここで dismiss を通常の更新可能カラムと同列に扱うと、append-only な event log から再生した最終状態が UI の期待と一致しなくなる。

## Decision

Knowledge Home の dismiss 可視性は monotonic な projection state として扱い、通常の upsert では解除しない。

具体的には以下を採用する。

1. `knowledge_home_items` への insert / upsert に `dismissed_at` を含める
2. `ON CONFLICT (user_id, item_key) DO UPDATE` では `dismissed_at = COALESCE(knowledge_home_items.dismissed_at, EXCLUDED.dismissed_at)` を使う
3. `HomeItemDismissed` が設定した `dismissed_at` は、後続の `SummaryVersionCreated` や `TagSetVersionCreated`、reproject 由来の通常 upsert では消さない
4. dismiss の解除が必要になった場合は、既存 upsert を流用せず、専用イベントまたは専用コマンドで扱う

この決定により、Knowledge Home の可視性は「いま最後に届いた通常更新」で決まるのではなく、「明示的な可視性イベントが存在するか」で決まるように揃える。

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|--------|------|-----------|
| dismiss 時に row を削除して再投影で復元しない | 実装は単純 | reproject や後続イベントで再作成されやすく、hidden state が安定しない |
| `updated_at` の新旧比較で dismiss を優先する | 一見自然 | 可視性の意味が時刻依存になり、再生順序や reproject 実行時刻に影響される |
| 取得時に `knowledge_events` を都度集計して dismiss を判定する | append-only には整合 | read path が重くなり、既存 projection パターンから外れる |

## Consequences

### Pros

- dismiss 済み item が後続の要約更新、タグ更新、supersede 更新、reproject で再表示されにくくなる
- event replay 後の最終状態が UI の期待と一致しやすくなる
- `dismissed_at` の意味が「一度 hidden にした事実」で安定し、デバッグしやすくなる
- 将来 undo を導入する場合も、通常 upsert と visibility change を分離したまま拡張できる

### Cons / Tradeoffs

- `dismissed_at` は通常の projection merge では解除できなくなるため、undo には追加実装が必要になる
- row の一部カラムは更新されても可視性だけは固定されるため、projection の merge ルールが単純な last-write-wins ではなくなる
- repository 層の SQL 契約に visibility 保持ルールが増えるため、query 変更時は回帰テストの維持が必要になる
