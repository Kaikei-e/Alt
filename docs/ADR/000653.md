# ADR-000653: Knowledge Acolyte レポート生成オーケストレーターを導入する

## Status

Accepted

## Date

2026-04-09

## Affected Services

- acolyte-orchestrator (Python 3.14+ / Starlette / Connect-RPC / LangGraph)
- alt-butterfly-facade (Go) — BFF ルーティング追加
- alt-frontend-sv (TypeScript/Svelte 5) — `/acolyte` ページ群追加

## Context

Alt には記事収集・要約・タグ付け・Recap のパイプラインが存在するが、**複数の情報源を横断した構造化レポート**を生成・版管理する機能がなかった。ユーザーが「今週の AI 動向」のようなテーマでレポートを作成し、生成→批評→改訂のサイクルを回して段階的に品質を上げる仕組みが求められていた。

従来の `updated_at` ベースの更新追跡では「何が変わったか」が不明瞭で、セクション単位の部分再生成と全体版管理を両立できない。`refine.md` の設計分析に基づき、**整数版番号 + 明示的な変更項目追跡**を中核に据えた新アーキテクチャを導入する。

### 設計上の制約

- **イミュータブルデータモデル** — Knowledge Home で確立した append-first / reproject-safe の原則に従う
- **単一マシンアーキテクチャ** — news-creator が推論プレーン、acolyte-orchestrator が制御プレーン
- **Connect-RPC 境界** — Python 側は Connect protocol 固定（connect-python は alpha、gRPC/gRPC-Web は未対応）
- **JSONB は補助情報のみ** — citations, tool traces などに限定。コア属性は SQL カラムで管理

## Decision

### 新サービス `acolyte-orchestrator` を Python 3.14+ で構築する

モダン Python スタック（uv, ruff, pyrefly, structlog, pydantic-settings）を採用し、Starlette + connect-python で Connect-RPC サービスを提供する。LLM オーケストレーションには LangGraph の StateGraph を使用する。

### DB スキーマ: 7 テーブルによる二層版管理

| テーブル | 性質 | 役割 |
|---------|------|------|
| `reports` | mutable | 現在値（title, current_version） |
| `report_versions` | immutable | 版スナップショット（change_seq でグローバル順序） |
| `report_change_items` | immutable | フィールド単位の変更追跡 |
| `report_sections` | mutable | セクション現在値（current_version, display_order） |
| `report_section_versions` | immutable | セクション本文スナップショット |
| `report_runs` | append | 実行記録（モデル情報、成功/失敗） |
| `report_jobs` | mutable | ジョブキュー（`FOR UPDATE SKIP LOCKED`） |

`updated_at` は設計の中心に置かず、`current_version` + `version_no` + `change_seq` で版管理する。

### LangGraph パイプライン: 6 ノード構成

```
planner → gatherer → curator → writer → critic → finalizer
                                  ↑                    |
                                  └── revise (max 2) ──┘
```

- 各ノードは Port インターフェース経由で外部依存を呼び出す（Clean Architecture）
- critic が `revise` 判定を出すと writer にループバック（最大 2 回）
- finalizer が楽観ロックで版をバンプし、change_items を記録

### テスト戦略: Outside-In TDD + Pact CDC

| レイヤー | テスト数 | 内容 |
|---------|---------|------|
| E2E | 3 | サービス起動、HealthCheck、Connect-RPC エラーハンドリング |
| CDC (Pact) | 4 | acolyte → news-creator (2), acolyte → search-indexer (2) |
| Unit | 24 | Settings, Gateway (httpx mock), Usecase, LangGraph ノード, 版バンプ |

Pact consumer contract は `pacts/acolyte-orchestrator-*.json` に出力される。

### BFF ルーティング

alt-butterfly-facade に `/alt.acolyte.v1.AcolyteService/` パスを追加。HTTP/1.1 トランスポート（Python uvicorn は h2c 未対応）で、`X-Service-Token` をインジェクト。StreamRunProgress は streaming timeout を適用。

### フロントエンド

| ルート | 機能 |
|--------|------|
| `/acolyte` | レポート一覧（タイトル、版、ステータス） |
| `/acolyte/new` | レポート作成フォーム |
| `/acolyte/reports/[id]` | レポート詳細（セクション表示、版履歴サイドバー、差分表示） |

Alt-Paper デザイン言語（Playfair Display + Source Sans 3、ベージュ基調）に準拠。

### 専用 DB: `acolyte-db`

主幹 DB（alt-db）とは分離し、pre-processor-db と同じパターンで `acolyte-db`（PostgreSQL 18 Alpine）を専用で用意する。ポートは 5439。

## Consequences

### Pros

- **「何が変わったか」の機械的追跡** — `report_change_items` により、scope 変更、セクション再生成、title 更新を個別に記録できる。`updated_at` では不可能だった粒度
- **セクション単位リラン** — report 全体の版を上げつつ、特定セクションだけ再生成する二層版管理が自然に表現できる
- **ジョブキューの安全性** — `SELECT ... FOR UPDATE SKIP LOCKED` により、ポーリング競合なしでジョブを排他取得
- **Pact CDC による契約保証** — news-creator / search-indexer との API 契約が自動検証され、破壊的変更を検出可能
- **LangGraph のノード分離** — 各ノードが Port 経由で依存するため、LLM プロバイダーやエビデンスソースの差し替えが容易
- **31 テスト全パス** — E2E / CDC / Unit の三層でカバー

### Cons / Tradeoffs

- **connect-python は alpha** — gRPC / gRPC-Web 未対応。安定しなければ Starlette REST + 手動 proto シリアライズにフォールバックが必要
- **LangGraph のバージョン追従** — Pydantic V1 互換性の警告あり（Python 3.14）。マイナーバージョンの pin が必要
- **Atlas マイグレーションの管理** — acolyte-db は専用 DB だが、Atlas の env 設定に追加する必要がある（今回未実施）
- **フロントエンドの proto codegen** — `buf generate` で TypeScript 型を生成するが、`buf.gen.yaml` への acolyte 追加は次ステップ
- **VRAM 競合** — LLM 呼び出しは news-creator 経由でセマフォ制御されるが、長時間パイプライン実行中は他の要約リクエストの TTFT に影響する可能性
