# ADR-000828: 3days Recap 残課題 3 件に対する surface-and-wire 修正 — per-genre failure の DB surface / 評価 NaN ガード / classifier thresholds factory の言語別パス wire

## Status

Accepted

## Date

2026-04-23

## Affected Services

- **recap-worker (Rust)** — `pipeline/persist.rs` に per-genre failure の `recap_failed_tasks` 書き込み経路を追加。`store/dao/mock.rs` の `MockRecapDao` に呼び出し記録フィールドを追加。
- **recap-subworker (Python)** — `services/evaluation.py` に NaN/empty-text ガード helper を追加。`app/container.py` と `services/classification_worker.py` の classifier factory を 30-genre 言語別 thresholds file を参照する wire に変更。
- **tests** — Rust 3 件 / Python 4 件の unit test を追加。

## Context

[[000827]] と [[PM-2026-038]] で rust-bert cache silent fallback を解消し subgenre 分割は 2 → 9 バケットに回復したが、同時に [[3days-recap-recovery-review-2026-04-22]] §"残課題" (L139-158) で 3 件の残存品質劣化要因が明示された。これらは「ユーザー可視のジャンル 2 バケット偏在」という同一症状への並行寄与因子で、1 件だけ直しても診断ループが閉じない。

### (1) per-genre failure の silent swallow — 診断ループが閉じない

`recap_worker/recap-worker/src/pipeline/persist.rs` の従来実装は 6 つの per-genre 失敗経路 (deserialize / not-found / fetch-fail / missing-id / upsert-fail / 汎用 dispatch error) で `warn!` を出して `genres_failed += 1; continue;` するだけで、`recap_failed_tasks` テーブルには何も書かれなかった。一方 DAO method `insert_failed_task(job_id, stage, payload, error)` は既に `store/dao/stage.rs` と `store/dao/traits/stage.rs` に存在し、stage-level (fetch / persist 全体) の失敗は `scheduler/jobs.rs:179-181` で正しく記録されている。per-genre だけ書かれない非対称な状態だった。

直近の調査で `recap-worker` ログに `genres_failed: 13` が出ていたが `recap_failed_tasks` には 0 行しかなかったことを確認。「13 ジャンルが何故失敗したか」を DB から遡る術がなく、ログを grep する以外に診断経路が無い状態。これは [[PM-2026-036]] Action Item #8 (エラー文言区分化) を recap-worker 側に広げる文脈と整合する。

### (2) `/v1/evaluation/genres` が NaN 行で 400 を返す — 評価パイプラインが止まる

recap-evaluator が `POST /v1/evaluation/genres` 経由で recap-subworker の G-Eval 経路を起動するとき、golden dataset に `content_ja` 欠落や空文字列 (`"   "`) のアイテムが含まれると HTTP 400 で終わる。根因は `services/evaluation.py:217` の `X = df["text"].tolist()` が NaN 行を含めたまま `classifier.predict_batch(X)` → `classifier.py:193 self.tfidf.transform(texts)` へ渡し、sklearn の TfidfVectorizer が `ValueError: np.nan is an invalid document` を送出すること。個別行の欠落で batch 全体が落ちる挙動で、1 golden row でも欠損すれば評価ジョブ全体が 400 で帰ってくる。

### (3) classifier factory が deprecated 17-genre 既定にぶら下がっている — 偏在の直接原因

`classify-runs` hot path は `ServiceContainer.classifier` と `services/classification_worker.py::initialize` で `GenreClassifierService(model_path, embedder)` を生成しているが、`thresholds_path` を渡していない。その結果 `services/classifier.py:56` の fallback `self.thresholds_path = self.model_path.parent / "genre_thresholds.json"` が効き、`data/genre_thresholds.json` (17 genre only、deprecated) を読み込む。残り 13 genre は `classifier.py:259` の `self.current_thresholds.get(cls, 0.5)` により default 0.5 の閾値で評価される。

`classifier.py:88-90` には sklearn 互換の force patch:

```python
print("DEBUG: Force patching multi_class attribute")
self.model.multi_class = 'ovr'
```

があり、多クラス LogReg を OVR モードで `predict_proba` させている。OVR の proba は独立計算で典型的に低値が出やすく、結果として 13 genre は閾値 0.5 を越えられない → `classifier.py:311` の `argmax(probs)` fallback に落ちる → どのテキストを渡しても dominant 2 genres (consumer_tech / politics_government) に寄る、という 2 バケット偏在を生む。ユーザーは「過去に同じモデルで動いていた」と明示しており、joblib 本体ではなく **factory の wiring が deprecated 既定パスに沿ってしまった** ことがレグレッションの本体。`config.py:569-577` の `genre_thresholds_path_ja` / `genre_thresholds_path_en` は既に定義されており、`evaluate()` endpoint 経路 (`routers/evaluation.py:163-164`) では正しく使われていたが、classify-runs hot path だけが deprecated 既定に落ちていた。

## Decision

### 1. recap-worker `pipeline/persist.rs` に `record_failed_genre` helper を追加して 6 サイトで書き込む

`src/pipeline/persist.rs` に free function `record_failed_genre(dao, job_id, stage, genre, error)` を追加する。呼び出し時は `dao.insert_failed_task(job_id, stage, Some(&json!({"genre": genre})), Some(error))` を叩き、DAO 呼び出し自体が失敗したら `warn!` のみに抑え処理継続する (二重失敗で全体を止めない)。

stage の分類は以下:

- `"dispatch_summary"` — `genre_result.error` が set されていて `no evidence` / `insufficient documents` のどちらの pattern にも match しないケース (= dispatch/summarization 側由来の generic 失敗)
- `"persist_lookup"` — DB から `summary_response` を resume しようとして失敗するケース 4 種 (deserialize error / Ok(None) で row が無い / fetch Err / summary_response_id が None)
- `"persist_write"` — `upsert_recap_output` または `upsert_genre` が Err を返したケース (複数 Err を `; ` join して 1 行でまとめる)

`no evidence` と `insufficient documents` は仕様上の expected outcome なので意図的に記録しない。`recap_failed_tasks` テーブルを signal-heavy に保つため (feedback: noise が多い error table は誰も見なくなる)。

`store/dao/mock.rs` の `MockRecapDao` 定義を unit struct から `failed_tasks: Arc<Mutex<Vec<RecordedFailedTask>>>` を持つ struct に格上げし、`new()` は `Default::default()` を維持。`failed_tasks()` accessor をテスト用に追加。`insert_failed_task` impl は記録してから `Ok(())` を返す。

### 2. recap-subworker `services/evaluation.py` に `_prepare_evaluation_inputs` static helper を追加

`EvaluationService` に以下の static helper を追加し、`evaluate()` の `X = df["text"].tolist()` 呼び出しをこの helper 経由に差し替える:

```python
@staticmethod
def _prepare_evaluation_inputs(
    df: pd.DataFrame,
) -> tuple[list[str], list[list[str]]]:
    texts_series = df["text"]
    labels_series = df["labels"]
    mask = texts_series.notna() & texts_series.astype(str).str.strip().ne("")
    kept_texts = [str(t) for t in texts_series[mask].tolist()]
    kept_labels = [list(lbl) for lbl in labels_series[mask].tolist()]
    return kept_texts, kept_labels
```

呼び出し地点で `dropped_rows = total_rows - len(X)` を計算し、0 でない場合のみ `logger.warning("evaluation: dropped rows with empty/NaN text before prediction", dropped_rows=..., kept_rows=...)` を出す。評価結果の数値は残った行だけで計算するため accuracy / f1 は表層には影響するが、silent には落とさずログで追えるようにする。

Response schema に `dropped_rows` を足すかは迷ったが、scope を最小化するため今回は Pydantic model は変更しない (Pact consumer test 側が見に行っていないため契約影響が無い)。

### 3. recap-subworker classifier factory 2 箇所を言語別 thresholds に wire

`recap_subworker/app/container.py::ServiceContainer.classifier` と `recap_subworker/services/classification_worker.py::initialize` の `GenreClassifierService` 生成 2 箇所に `thresholds_path=settings.genre_thresholds_path_ja` を追加する。

```python
# container.py
self._classifier = GenreClassifierService(
    model_path=self.settings.genre_classifier_model_path,
    embedder=self.embedder,
    thresholds_path=self.settings.genre_thresholds_path_ja,
)

# classification_worker.py
_CLASSIFIER = GenreClassifierService(
    model_path=settings.genre_classifier_model_path,
    embedder=embedder,
    thresholds_path=settings.genre_thresholds_path_ja,
)
```

`settings.genre_thresholds_path_ja` の既定値は `"data/genre_thresholds_ja.json"` (30 genre, 閾値 0.10-0.50) で、compose の host bind (`/var/lib/alt-recap-subworker-data:/app/data:ro`) 経由でファイルが存在する前提。欠落した場合は `classifier.py:121` の `self.thresholds = {}` fallback → 全 genre が 0.5 default に落ちる = 現状と同等の挙動なので regression は発生しない。

### 検討した代替案と却下理由

- **`genre_thresholds.json` を 30-genre に書き換える**: `data/` は gitignore されているため git では配信できない。host 側 bind-mount の populate が別経路 (runbook / alt-deploy) に依存しており、本 ADR の scope では保証できない。**却下**。
- **`multi_class='ovr'` force patch を削除**: 根本治療の候補だが、training 時の sklearn バージョンと推論時のバージョンが混在しており、patch 撤去は joblib 再学習を要する。本 ADR は「wiring を deprecated 既定から剥がす」最小修正に留め、`multi_class` patch は [[000827]] の後続 follow-up に回す。**却下 (scope out)**.
- **`_prepare_evaluation_inputs` を instance method にする**: state を参照しないため static method のほうが test 容易性が高く意図が明確。`self` を取らないことで evaluate() 本体のモック負担も減る。**採用**.
- **`recap_failed_tasks` に `no evidence` も記録する**: 仕様上の expected outcome を error table に混ぜると signal が薄まる。現状の 3 分類 (`genres_no_evidence` / `genres_skipped` / `genres_failed`) は `PersistResult` で counter として保持されており Prometheus 経由で監視可能。DB に残すのは真の failure (= 要調査) のみとする。**却下**。
- **summarization.rs 3 サイト (line 482-499 / 623-642 / 646-661) でも `insert_failed_task` を呼ぶ**: これらはすべて `GenreResult.error = Some(...)` を set するだけで、下流の persist.rs が generic error 経路で検出して `dispatch_summary` stage として surface するため、二重記録を避ける意味で summarization 側の追加は不要と判断。**却下 (scope out)**.

## Consequences

### Pros

- **診断ループが閉じる**. 「`genres_failed: 13` とログに出るが DB には 0 行」という状態が解消し、`SELECT stage, error, payload FROM recap_failed_tasks WHERE job_id = ? ORDER BY created_at` で直接 per-genre 失敗内容を引ける。runbook / Prometheus への組み込みも stage label 経由で直交。
- **評価パイプラインが 1 行の欠損で全体停止しない**. NaN 行は silent に drop され、残った行だけで accuracy / f1 を計算する。golden dataset の curation が進行中でも評価自体は止まらない。drop 行数は `logger.warning` に出るのでデータ品質の追跡は別軸で可能。
- **2 バケット偏在の根本治療**. classify-runs hot path が 30-genre の閾値を参照するようになり、本来 proba 0.3-0.5 で pass する genre が argmax fallback 経由ではなく threshold pass 経由で top_genre に採用される。事前推定では 22 → 10+ バケットへの分散回復を期待。
- **[[PM-2026-036]] / [[PM-2026-037]] の silent failure 系譜と整合**. 「外形 healthy + 分類品質だけ劣化」を surface する方向で一貫している。今回の 3 件はいずれも silent fallback の可視化 / 破壊 / wire 修正という意味で同じ射影軸。
- **テスト容易性**. `_prepare_evaluation_inputs` を static にしたこと、`record_failed_genre` を free function にしたこと、`MockRecapDao` の記録フィールドを `Arc<Mutex<_>>` にしたことで、各修正が rust-bert / sklearn 本体を起動せず unit 層で assert 可能。rust 3 test / python 4 test すべて独立に green。

### Cons / Tradeoffs

- **`recap_failed_tasks` への書き込みが 1 ジョブ最大 30 行に増える**. 現状は stage-level の数行だった。DB sizing には影響しないが、ダッシュボードが「stage 粒度の集計」を前提にしているなら stage label でフィルタする必要がある。現行 runbook / monitoring は per-genre 側をまだ参照していないため短期影響は無い。
- **`dropped_rows` を response に乗せていない**. pact consumer が見ていないので scope 外とした。将来 evaluator UI で dropped 行数を表示したくなった場合は Response model 追加が必要。
- **classifier thresholds の言語判定は未実装**. 今回は `genre_thresholds_path_ja` (日本語) 固定。Alt の主要入力が日本語である前提 (memory: `feedback_single_machine_architecture`) で成立するが、英語記事の比率が上がった場合に再度偏在が出うる。その際は `thresholds_path_en` への切り替え or 動的選択を別 ADR で検討する。
- **`multi_class='ovr'` force patch は未処置**. 根本治療ではなく wiring 修正であるため、sklearn バージョン整合 / 再学習の follow-up は未着手。thresholds を低く設定してある `_ja.json` が patch による proba 圧縮を吸収する形になっており、患者の症状は消えるが病巣は残る。次サイクルで `classifier.py:88-90` の force patch 撤去を検討する。
- **summarization.rs 側の error も persist 経由でしか surface されない**. 上流 stage が先に fatal になるケース (batch API が丸ごと Err を返す) では scheduler/jobs.rs が stage-level で記録するので穴は無いが、summarization の per-genre path と persist の per-genre path が同じ table の同じ stage で同居しないことを前提にしている。将来 summarization 側で直接 `insert_failed_task` を呼びたくなった場合は重複記録を避ける key 設計が必要。
- **本 ADR 完了後も [[3days-recap-recovery-review-2026-04-22]] §残課題 (3)(4)(6) は未着手**. recap-evaluator pact consumer MismatchesError / `OLLAMA_URL` の `/api/tags` 404 / [[PM-2026-036]] AI #6/#7/#8 は別 PR で tracks する。
