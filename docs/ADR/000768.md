# ADR-000768: news-creator に Hurl E2E スイートを追加し段階的全面展開の第 5 サービスとする (Phase 1)

## Status

Accepted

## Date

2026-04-18

## Affected Services

- news-creator (Python 3.14 / FastAPI) — ランタイムは無変更。staging compose に `news-creator` profile を追加し `/health` と `/queue/status` をカバーする Hurl E2E スイート 3 シナリオを追加
- news-creator-ollama-stub (新規 / Python+FastAPI) — `compose/news-creator-ollama-stub/` 直下に Dockerfile と `app.py` を新設。staging で Ollama を置き換え `/api/tags` のみ返す

## Context

[[000763]] (search-indexer) → [[000764]] (mq-hub) → [[000765]] (knowledge-sovereign) → [[000766]] (run.sh ディスパッチ規約) → [[000767]] (tag-generator) と段階的に整備してきた Hurl E2E スイートの第 5 サービスとして news-creator を取り込む。news-creator は 11 エンドポイント (`/health`, `/queue/status`, `/api/v1/summarize` (SSE), `/api/generate`, `/v1/summary/generate(/batch)`, `/api/v1/expand-query`, `/v1/rerank`, `/api/chat` (NDJSON), `/api/v1/plan-query`, `/v1/morning-letter/generate`)、HybridPrioritySemaphore を介した RT/BE スケジューリングと HTTP 429 + `Retry-After: 30` の backpressure、SSE と NDJSON の 2 種ストリーミング、4 つの Pact consumer (pre-processor / rag-orchestrator / acolyte-orchestrator / recap-worker) を抱える。Pact CDC は `news-creator/app/tests/contract/test_provider_verification.py` で provider 形状を pin しているが、HTTP 層 + キュー深さ + ストリーミング body 形状を黒箱で叩く E2E は無く、queue saturation や SSE 改行整形の回帰を CI で捕まえる手段が不在だった。

最も重い設計上の問題は LLM バックエンド依存の扱いだった。本番は RTX 4060 上の Ollama (ADR-000145 の `gemma4-e4b-12k`) に向けるが、GitHub Actions の ubuntu-latest ランナーには GPU が無く、tinyllama 級の CPU モデルでもモデル pull に数分・推論はさらに数十秒かかり CI 時間と決定論性を共に損なう。real Ollama を local-only smoke として残し CI gate は持たない選択肢 (search-indexer / knowledge-sovereign 流) と、Ollama 互換のスタブを staging に立てて CI gate も含めて回す選択肢の 2 案を比較した結果、後者を採用した。理由は契約検証中心の本 E2E にとって LLM 出力そのものは検査対象ではなく、HTTP 表面と HybridPrioritySemaphore の挙動が主役だからである。

段階的展開は 4 Phase に細分する。本 ADR は **Phase 1 (基盤 + 健全性)** のみを対象とし、`/health` schema・`/queue/status` schema・readiness gate の 3 シナリオに絞る。Phase 2-4 (summarize 系 / RAG 系 / streaming + 429 + バリデーション) は後続 ADR で記録する。

## Decision

### staging compose に `news-creator` profile を追加し Ollama をスタブで置換する

`compose/compose.staging.yaml` に 2 サービスを追加する:

- `news-creator-ollama-stub` — Python 3.13 + FastAPI 0.118 で書かれた最小スタブ。`/api/tags` のみ実装し `gemma3:4b-it-qat` を含む固定 model 一覧を返す。`compose/news-creator-ollama-stub/{Dockerfile, app.py}` に同梱。
- `news-creator` — `ghcr.io/${GHCR_OWNER}/alt-news-creator:${IMAGE_TAG}` を pull し `LLM_SERVICE_URL=http://news-creator-ollama-stub:11435` を注入。`SERVICE_SECRET=alt-staging-test-service-secret` (`NewsCreatorConfig` がブートに必須)、`WARMUP_ENABLED=false` (lifespan から `/api/generate` を呼ばせない)、`MODEL_ROUTING_ENABLED=false` (バケットモデル名解決を無効化)、`CACHE_ENABLED=false` (Redis 依存を切り離し)、`OTEL_ENABLED=false` (alt-staging に collector が無いため connect-refused ノイズを抑止)、`MTLS_ENFORCE=false` / `PEER_IDENTITY_TRUSTED=off` (mq-hub / tag-generator / knowledge-sovereign の staging 規約を踏襲) を設定。

### `e2e/hurl/news-creator/` に 3 シナリオの Hurl スイートを追加する

[[000766]] が確立した `run.sh` 雛形 (tag-generator/run.sh をテンプレ) を踏襲し、`--jobs 1` でシリアル実行する。この時点では並列でも通るが、Phase 4 で導入予定の queue saturation シナリオが共有状態 (HybridPrioritySemaphore) を踏むため、最初から直列で固定する。

| ファイル | 役割 |
|---|---|
| `00-setup.hurl` | `/health` を `retry: 30, retry-interval: 500` で待機 |
| `01-health-schema.hurl` | `/health` の `{status, service, models[]}` 形状を pin。stub の固定 model 名 (`gemma3:4b-it-qat`) を assert し、`error` フィールドが不在であることを確認 |
| `02-queue-status.hurl` | `/queue/status` の `{rt_queue, be_queue, total_slots, available_slots, accepting, max_queue_depth}` 形状を pin。アイドル状態の値 (queue=0, accepting=true, max_queue_depth=10) を assert |

Phase 2-4 用に `e2e/fixtures/news-creator/.gitkeep` のみを置き、フィクスチャディレクトリを予約する。

### CI gate は path-filtered 単独 job

`.github/workflows/e2e-hurl.yml` に `news-creator` job と path filter (`news-creator/**`, `compose/news-creator-ollama-stub/**`, `e2e/hurl/news-creator/**`, `e2e/fixtures/news-creator/**`) を追加する。job の skeleton は tag-generator job をそのまま写し、`docker build -t ghcr.io/kaikei-e/alt-news-creator:ci -f news-creator/Dockerfile.app news-creator` でローカルにイメージを焼いてから `bash e2e/hurl/news-creator/run.sh` を実行する。

### 検討した代替案

- **real Ollama + tinyllama:1.1b** を staging で動かす — 却下。モデル pull (CPU では 700MB+ ダウンロード + 数分の解凍)、推論レイテンシ (CPU で数十秒)、出力非決定論性が CI gate に乗らない。
- **real Ollama + GPU runner (self-hosted)** — 却下。本 Phase の検査対象は HTTP 形状とキュー挙動で、LLM 出力そのものは含めない。コスト対効果が見合わない。後続 Phase で `REAL_LLM=1` ハイブリッドモードとして再検討の余地は残す。
- **redis-cache-staging を含める** — 却下。`CACHE_ENABLED` の既定が false で、Phase 1 の検査対象にキャッシュ挙動は含まれない。Phase 2 以降で必要になれば追加する。
- **search-indexer / knowledge-sovereign 流の "developer-runnable のみ"** — 却下。Pact CDC 既存テストでは捕まらない backpressure / SSE フォーマット / proxy レイヤの回帰を CI でガードしたいというのが本スイートの主目的のため、CI gate を最初から付ける。

## Consequences

### Pros

- pre-processor / rag-orchestrator / acolyte-orchestrator / recap-worker の Pact 検証では捕まらない HTTP 層挙動 (`/queue/status` schema、Ollama proxy 接続性) を CI でガードできるようになった。
- Phase 2-4 で extend する基盤 (`news-creator-ollama-stub`、`compose.staging.yaml` の `news-creator` profile、`run.sh`、CI job、path filter) が確立し、後続 Phase は scenario ファイル追加と stub の endpoint 追加のみで済む。
- staging から GPU 依存を完全に切り離した。GitHub Actions ubuntu-latest ランナーで Phase 1 のフルサイクル (compose up → 3 scenario → compose down) が約 30 秒で回る (ローカル計測: setup 2ms / 01 2ms / 02 1ms + コンテナ起動オーバーヘッド)。
- Hurl 7.1 のチャンク受信特性 (body 完全受信後にしか assert できない) を README で明示し、Phase 4 で導入予定のストリーミングシナリオ設計の前提を共有した。

### Cons / Tradeoffs

- スタブと本番 Ollama の挙動差は本スイートでは検出できない。Ollama の `/api/tags` レスポンス形状が将来変わってもスタブを追従させない限り CI は素通りする。Phase 4 完了後に `REAL_LLM=1` ハイブリッドモードを検討する余地として残す。
- `WARMUP_ENABLED=false` を staging で固定したため、本番で warmup が破壊的変更を起こしてもこのスイートでは検知できない。warmup 経路の検証は本 Phase の意図的な対象外であり、別途 unit/contract テストに委ねる。
- news-creator の最小 Docker image (Dockerfile.app の Python 3.14-slim ベース) でも依存 install を含めると CI ジョブ全体で 3-5 分のビルド時間がかかる見込み。tag-generator (~5 分) と同オーダーだが path filter で起動条件を絞っているので不要な PR では発火しない。
- Phase 1 の検査表面が小さい (3 シナリオ) ため、現時点では「news-creator が起動して `/health` と `/queue/status` を返す」以上の保証は無い。Phase 2-4 で Pact が pin する形状の **隣** にある HTTP 層挙動 (429 + Retry-After、SSE/NDJSON フォーマット、validation エラー) を順次取り込む予定。
