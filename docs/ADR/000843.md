# ADR-000843: Knowledge Loop transition の 502 バーストを validator の契約分裂と BFF の numeric Code enum 誤抽出として塞ぐ

## Status

Accepted

## Date

2026-04-24

## Affected Services

- **alt-backend (Go)** — `app/usecase/knowledge_loop_usecase/validator.go` の `ValidateDwellTriggerTarget` を DWELL が OBSERVE または ORIENT を target できるよう緩和。`validator_test.go` と `transition_knowledge_loop_usecase_test.go` に canonical dwell (observe→orient) の回帰テストを追加し、pre-existing flaky だった `TestTransition_RateLimit_RejectsOverGlobalCeiling` の clock-drift を `mustMintUUIDv7(t, fixedNow)` helper で drive-by fix。
- **alt-frontend-sv (SvelteKit)** — `src/lib/connect/error.ts` の `extractConnectCode` を書き直し、`@connectrpc/connect` v2.1 の numeric `Code` enum (1..16) を snake_case string にマップ。新規 `src/lib/connect/error.test.ts` で 16 値 + back-compat + 境界ケースを pin。`src/routes/(app)/loop/transition/server.test.ts` に real `ConnectError(Code.InvalidArgument)` / `Code.Unavailable` の回帰を追加。

## Context

本番 `POST /loop/transition` が `IntersectionObserver.root` 発火の dwell 3 連バーストで 502 Bad Gateway を返す事象がユーザから再報告された。2026-04-24 に [[000839]] (Connect-RPC code 分類) / [[000840]] (event append) / [[000842]] (rate limit + stream JWT exp) と併せて観測 3 点 (commit `c89b5aa19`) を入れたが「過去に直したのに、治っていない」。

ローカル staging stack で c89b5aa19 の log を突いたところ、原因が当初仮説の「sovereign transient unavailable」ではなく、**決定論的な 2 層バグ**だと判明した:

1. **alt-butterfly-facade** 側の `bff.proxy_response`:
   ```
   path=/alt.knowledge.loop.v1.KnowledgeLoopService/TransitionKnowledgeLoop
   status=400 streaming=false elapsed_ms=1-3
   ```
   backend は 400 (`CodeInvalidArgument`) を 1〜3ms で即返している。upstream は健在。

2. **alt-frontend-sv SSR** 側の `loop.transition.unknown_error`:
   ```
   name: "ConnectError"
   message: "[invalid_argument] invalid argument"
   code: null
   cause: null
   ```
   ConnectError は届いているが `code` が null。BFF の `+server.ts` default 分岐に落ちて HTTP 502 `upstream_unreachable` が surface する。

3. **alt-backend** の `alt.knowledge_loop.transition_ok` は **一切出ない** — transition は 1 度も成功していない。`knowledge_events` に `knowledge_loop.observed.v1` 型のイベントが 0 件。

この症状は 2 つの独立したバグの連鎖だった。

### BUG-A: backend `ValidateDwellTriggerTarget` が `ClassifyTransitionEvent` と契約分裂

`alt-backend/app/usecase/knowledge_loop_usecase/validator.go:114-119` は:

```go
func ValidateDwellTriggerTarget(trigger, toStage string) error {
    if trigger == "TRANSITION_TRIGGER_DWELL" && toStage != "LOOP_STAGE_OBSERVE" {
        return fmt.Errorf("%w: dwell trigger only valid for OBSERVE target", ErrInvalidArgument)
    }
    return nil
}
```

DWELL が OBSERVE 以外を target することを拒否する。しかし同 package の `classify_transition_event.go:44-48` は:

```go
case fromStage == stageObserve && toStage == stageOrient:
    if trigger == triggerDwell {
        return domain.EventKnowledgeLoopObserved, nil
    }
    return domain.EventKnowledgeLoopOriented, nil
```

で `(observe, orient, dwell)` を canonical な `EventKnowledgeLoopObserved` として正当に扱う。[[knowledge-loop-canonical-contract]] §8.2 も「`KnowledgeLoopObserved` fires when tile dwells ≥1.5s in viewport」と規定しており、dwell は observe→orient の transition 契機になる。

`alt-frontend-sv/src/lib/hooks/useKnowledgeLoop.svelte.ts:102-109` の `observe()` は IntersectionObserver で `{fromStage:"observe", toStage:"orient", trigger:"dwell"}` を送る — 契約準拠。しかし `TransitionKnowledgeLoopUsecase.Execute` の validator が classifier より先に走り (`transition_knowledge_loop_usecase.go:97-99`)、全ての dwell 遷移が `ErrInvalidArgument` で止められる。

### BUG-B: BFF `extractConnectCode` が numeric `Code` enum を parse できない

`alt-frontend-sv/src/lib/connect/error.ts:36-42` は:

```ts
export function extractConnectCode(err: unknown): string | undefined {
    if (err && typeof err === "object" && "code" in err) {
        const c = (err as { code: unknown }).code;
        return typeof c === "string" ? c.toLowerCase() : undefined;
    }
    return undefined;
}
```

string code のみを認識する。しかし `node_modules/@connectrpc/connect/dist/esm/code.d.ts` は v2.1 で `ConnectError.code` を numeric `Code` enum (`Canceled=1` .. `Unauthenticated=16`) として宣言している:

```ts
export declare class ConnectError extends Error {
    readonly code: Code;       // ← numeric enum, not string
    readonly rawMessage: string;
    ...
}
```

結果: real な ConnectError はすべて `extractConnectCode(err)` が undefined を返し、`+server.ts:146` の default 分岐に落ちて `{error: "upstream_unreachable"}` HTTP 502 になる。

既存 `server.test.ts` でこの bug が検知されなかったのは、mock が全て `Object.assign(new Error("x"), { code: "invalid_argument" })` のように string code を hand-craft していたため。本物の ConnectError を作る経路が 1 箇所も無く、numeric code のパスを踏むテストがゼロだった。

### 連鎖結果

Dwell → backend が BUG-A で 400 を返す → BFF が BUG-B で 400 を 502 に laundering → ユーザは 502 を見る。[[000839]] の error classification table (Connect code → HTTP status) は実装されていたが、前段の抽出が壊れているため動かない状態だった。

定量的には、ローカル stack の log で `bff.proxy_response status=400` が出る毎に `loop.transition.unknown_error` が 1 件出る 1:1 対応を確認済み。retry-backoff は不要 — 症状は deterministic で transient ではない。

## Decision

2 点を採用する:

### Decision 1: `ValidateDwellTriggerTarget` を DWELL→{OBSERVE, ORIENT} 許容へ緩和

```go
func ValidateDwellTriggerTarget(trigger, toStage string) error {
    if trigger != "TRANSITION_TRIGGER_DWELL" {
        return nil
    }
    if toStage == "LOOP_STAGE_OBSERVE" || toStage == "LOOP_STAGE_ORIENT" {
        return nil
    }
    return fmt.Errorf("%w: dwell trigger only valid for OBSERVE or ORIENT target", ErrInvalidArgument)
}
```

canonical contract §8.2 と `ClassifyTransitionEvent` の単一ソースに整合。DWELL→{DECIDE, ACT} は意図的な能動行動であって passive な観測ではないため引き続き拒否する。`ClassifyTransitionEvent` が同 tuple を既に弾くので、validator 層は defense-in-depth として残る。

### Decision 2: `extractConnectCode` を numeric `Code` enum 対応に書き直し

`@connectrpc/connect` の 16 値を snake_case string に写像する静的 `CODE_TO_STRING` map を導入する:

```ts
const CODE_TO_STRING: Readonly<Record<number, ConnectCode>> = {
    [Code.Canceled]: "canceled",
    [Code.Unknown]: "unknown",
    [Code.InvalidArgument]: "invalid_argument",
    ... (16 entries)
};

export function extractConnectCode(err: unknown): ConnectCode | undefined {
    if (!err || typeof err !== "object" || !("code" in err)) return undefined;
    const c = (err as { code: unknown }).code;
    if (typeof c === "number") return CODE_TO_STRING[c];
    if (typeof c === "string") {
        const lower = c.toLowerCase();
        return (CODE_TO_STRING as unknown as Record<string, ConnectCode>)[lower] ??
            (lower in reverseIndex ? (lower as ConnectCode) : undefined);
    }
    return undefined;
}
```

string path は hand-crafted mock との back-compat のため保持する。`@connectrpc/connect` 内部の `codeToString` は `@private Internal code, does not follow semantic versioning` 宣言なので利用せず、Code 列挙の 16 値は Connect-RPC 仕様で固定なので自前 map の追従コストは実質ゼロ。

新規 `src/lib/connect/error.test.ts` で 16 Code 値全件と境界ケース (bare Error / null / primitive / out-of-range number) を pin。`server.test.ts` に real `ConnectError(Code.InvalidArgument)` と `Code.Unavailable` のケースを追加し、今後 hand-crafted mock で隠れる回帰を防ぐ。

### 検討した代替案

- **`ValidateDwellTriggerTarget` を完全削除し `ClassifyTransitionEvent` 一本化** — 単純だが boundary 側の validator を失うので usecase 内の順序入れ替えやリファクタで再発しやすい。Decision 1 の defense-in-depth を維持する方が運用堅牢。却下。
- **`ValidateDwellTriggerTarget` をそのままにして FE `observe()` を `toStage:"observe"` に変更** — canonical contract §8.2 と `ClassifyTransitionEvent` の両方に逆らう変更になり、`knowledge_events` の event_type 意味論 (observe→orient = KnowledgeLoopObserved) を実装側に暗黙で押し付ける。却下。
- **`@connectrpc/connect` の private `codeToString(code)` を import して使う** — 1 行で済むが `@private` 宣言で semver 外、minor upgrade で壊れる可能性がある。自前 map の方が依存の surface が小さい。却下。
- **FE 側で 502 に対して exponential backoff retry を追加** — 当初プランだったが、log 調査で症状が deterministic (毎回必ず 400) と判明し retry は無意味。さらに retry で隠すと本質的な validator/extractor バグが log 上で見えなくなる。却下。
- **駆動抽象化: BFF 側に interceptor を入れて `extractConnectCode` を経由させない** — スコープが広すぎる上、ConnectError の code を解釈しない route は BFF 全体でこの 1 箇所のみ (他は Connect client 内部で処理)。`error.ts` のヘルパを直すのが最小変更。却下。

## Consequences

### Pros

- **Dwell → observe/orient が canonical contract §8.2 通り HTTP 200 を返す**。IntersectionObserver 経由の observe 遷移が `knowledge_events.knowledge_loop.observed.v1` として正常に append され、session_state projection が進む。[[000840]] の「Loop event を実行エンジンとして機能させる」前提がようやく成立する。
- **BFF の error classification が意味を取り戻す**。backend が `CodeInvalidArgument` を返したら HTTP 400 `{error:"invalid_argument"}`、`CodeUnavailable` なら 502 `{error:"upstream_unavailable"}` と [[000839]] の table 通りに surface する。運用観測 (`loop.transition.unknown_error` 警報) が fetch 層の実障害にだけ反応するようになる。
- **append-first / reproject-safe / event-time purity を一切壊さない**。`knowledge_events` schema / projection / dedupe / `projected_at` のいずれにも触れず、validator 分岐と BFF 側の error 抽出のみの変更。immutable data model の不変条件 (`.claude/rules/knowledge-home.md`) はすべて維持。
- **rolling deploy 互換**。DB migration なし / proto 変更なし / 新 compose service なし。既存の `git push origin main` → `dispatch-deploy.yaml` → `Kaikei-e/alt-deploy` 経路にそのまま乗る ([[project_deploy_manual_post_adr]] 準拠)。
- **回帰テストで同型バグを再発防止**。`error.test.ts` は real `ConnectError` を 16 値すべて通すので、BFF 層で hand-crafted string code mock に頼れなくなる。`server.test.ts` 新規 2 ケースも同方針。
- **drive-by で `TestTransition_RateLimit_RejectsOverGlobalCeiling` の clock-drift flakiness を解消**。`mustMintUUIDv7(t, fixedNow)` helper で test local time と `uuid.NewV7()` の wall-clock が乖離しても落ちなくなり、`go test ./usecase/knowledge_loop_usecase/ -race` が確定的に green。

### Cons / Tradeoffs

- **`ValidateDwellTriggerTarget` と `ClassifyTransitionEvent` の責務境界は依然として曖昧**。DWELL → {DECIDE, ACT} は validator と classifier の両方が拒否する 2 重検証になる。将来的に validator を classifier 呼び出しにインライン化するリファクタの余地が残る。本 ADR では変更 surface を最小にするため見送り。
- **`CODE_TO_STRING` 静的 map の追従**。`@connectrpc/connect` が Code enum に値を追加するケース (Connect-RPC 仕様拡張) は理論上あり得るが、実績として Code 16 値は gRPC 以来安定している。Minor upgrade 時の `codeToString` shim が動くかを確認する CI ゲートは現状無い。将来 Code が追加されたら map 漏れは undefined → default 502 に落ちるため安全側には倒れるが、surface しないので気付きにくい。
- **既存 `Object.assign(new Error, { code: "x" })` 形式の mock が back-compat で動き続ける**。短期的には回帰を避けるため保持したが、新規テストでは real `ConnectError` を使うよう新 `error.test.ts` が規範を示す。biome lint ルールで `{code: "..."}` の hand-craft を警告する案は別途検討。
- **`extractConnectCode` の responsibility が増えた**。v2 Code 対応 + string back-compat + reverseIndex の 3 経路を持つため、以前より読みにくい。ヘルパとしては許容範囲だが、connect-es v3 (仮) で Code の wire 形式が変わるなら再設計の契機になる。
- **`TestTransition_RateLimit_RejectsOverGlobalCeiling` の drive-by fix はスコープ外**。別 PR に分けるほうが diff レビューは楽だが、同じ file を touch しているため一緒に commit する方が history 的にわかりやすいと判断した。
