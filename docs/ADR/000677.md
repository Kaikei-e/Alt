# ADR-000677: FactNormalizer を tiny-schema 化し quote-as-fact fallback を正式仕様にする

## Status

Accepted

## Date

2026-04-10

## Affected Services

- acolyte-orchestrator (Python 3.14+ / LangGraph)

## Context

Acolyte v5 パイプラインの FactNormalizerNode は LLM 成功率 **3/30 (10%)** で、27 件が quote-as-fact fallback に落ちている。にもかかわらず SectionPlanner・Writer まで到達し、レポートは完走している。

根本原因は以下の 4 点:

1. **`num_predict=512` のハードコード** — node 単位で構造出力バジェットを調整できない。Gemma4 26B の thinking tokens が `num_predict` を消費し、残りでは JSON を完成できずに truncation する
2. **30 quotes 全件に LLM 呼び出し** — ほぼ全件失敗しているため、約 28 分の推論待ちが無駄になっている
3. **fallback fact の非明示性** — runtime 出力 dict に `is_fallback` フラグがなく、downstream が品質差を判別できない
4. **`data_type` が free-form `str`** — GBNF grammar による schema 拘束が弱く、LLM が enum 外の値を出力するリスクがある

なお、`FactNormalizerOutput` の `reasoning` フィールド（ADR-632 で導入、thinking tokens 吸収目的）の削除は、repo 内既存知見で reasoning-first が有利だったケースがあるため、本 ADR では即断せず A/B 検証ゲート付きの変更として保留した。

## Decision

### 1. `data_type` を `Literal[...]` で enum 制約する

`FactNormalizerOutput.data_type` を `str` から `Literal["statistic", "date", "quote", "trend", "comparison"]` に変更し、Ollama の GBNF grammar が出力値を制約できるようにする。

### 2. `is_fallback: bool` を runtime 出力契約に追加する

FactNormalizerNode が出力する fact dict に `is_fallback` フィールドを必ず載せる。LLM 成功時は `False`、fallback 時は `True`。判定には sentinel identity check (`result is fallback_obj`) を使い、`confidence == 0.3` のような脆弱な値比較を避ける。

```python
# generate_validated は fallback object をそのまま返す (コピーしない)
fallback_obj = FactNormalizerOutput(...)
result = await generate_validated(..., fallback=fallback_obj)
is_fallback = result is fallback_obj
```

### 3. `num_predict` を Settings 経由の node-config に移行する

`fact_num_predict: int = 512` を `Settings` に追加し、ハードコードを廃止する。デフォルトは現行値 512 を維持し、A/B 検証後に引き下げる余地を残す。

### 4. `max_facts_total` で fact 数に上限を設ける

`max_facts_total: int = 20` を導入し、全セクション合計の fact 数を制限する。30 quotes 全件に LLM を呼ぶ無駄を削減する。

上限適用時は **section round-robin** で interleave し、先頭 section に偏らない公平な cap とする。raw slice (`quotes[:N]`) は section 順バイアスを生むため不採用。

### 5. `reasoning` 削除は A/B ゲート付きで保留する

exec3.md は「長い reasoning を出力契約から外す」と記載しているが、ADR-632 / パイプライン安定化レビューで reasoning-first が Gemma/Ollama structured output の成功率を改善した実績がある。本 ADR では即断実装せず、以下を満たした場合にのみ削除する:

- v5 相当の quote fixture で `reasoning あり` / `なし` を比較
- 成功率が悪化せず、eval_count / 待ち時間で改善すること

### 6. Prompt を短文化する

Gemma function-calling ガイドに従い、prompt を clear and brief に短縮する。JSON keys と enum 値を明示し、余計な説明を削る。schema は `format` パラメータと prompt の両方に渡す（Ollama 公式推奨）。

```
Normalize one quote into one atomic fact.

Quote: "{text}"
Source: {source_title}

Return JSON with exactly these keys:
- claim
- confidence
- data_type  # statistic|date|quote|trend|comparison
```

## Implementation

### 変更ファイル

| ファイル | 変更 |
|---------|------|
| `acolyte/config/settings.py` | `fact_num_predict: int = 512`, `max_facts_total: int = 20` 追加 |
| `acolyte/domain/quote_selection.py` | `data_type` を `Literal[...]` に。docstring 更新 |
| `acolyte/domain/fact.py` | `ExtractedFact` に `is_fallback: bool = False` 追加 |
| `acolyte/usecase/graph/nodes/fact_normalizer_node.py` | config 注入、round-robin cap、sentinel fallback 判定、prompt 短文化 |
| `acolyte/usecase/graph/report_graph.py` | `build_report_graph()` に `settings` param 追加、`FactNormalizerNode` への配線 |
| `main.py` | `settings` を `build_report_graph()` に pass-through |
| `tests/unit/test_fact_normalizer_node.py` | 8 新テスト + 6 既存テスト更新 (14 テスト全パス) |

### Config Protocol パターン

`FactNormalizerNode` は具象 `Settings` に直接依存せず、`_FactNormalizerConfig` Protocol を通じて設定を受け取る。テストでは `@dataclass FakeSettings` で満たせる。

### Section Round-Robin

`_cap_round_robin()` は `selected_quotes` を `section_key` でグループ化し、各 section から 1 件ずつ interleave しながら `max_facts_total` まで取る。section 数が不均衡でも公平に分配される。

### Downstream 互換性

`is_fallback` は additive field であり、downstream の `SectionPlannerNode`, `WriterNode`, `CriticNode` は既存フィールドを `.get()` で参照しているため、breaking change は発生しない。本 phase では downstream が `is_fallback` を読むロジックは追加せず、将来の ranking weight 活用に備える。

## Consequences

### Pros

- **fallback fact の品質可視化** — `is_fallback` により downstream が LLM fact と fallback fact を区別できるようになった
- **LLM 呼び出し数の削減** — `max_facts_total=20` で最大 90 calls が 20 に制限される。v5 比で約 10 分の推論待ち削減
- **schema 拘束の強化** — `data_type` の `Literal` 制約により GBNF grammar が出力値を厳密に制約
- **設定の外部化** — `fact_num_predict` により、A/B 検証やモデル変更時に再デプロイなしで調整可能
- **section 間の公平性** — round-robin cap により後段 section の evidence が切り捨てられない

### Cons / Tradeoffs

- **`reasoning` 削除の保留** — exec3.md の scope に含まれるが、既存知見との衝突を理由に保留。別 slice での A/B 検証が必要
- **sentinel identity check の前提依存** — `generate_validated()` が fallback object をそのまま返す実装に依存。将来の実装変更時に見直しが必要（コメントで明記済み）
- **`max_facts_total=20` の妥当性** — 現時点では v5 の 30 から保守的に引き下げた値。実運用データに基づく調整が今後必要
