# ADR-000368: sanitizeDescription 二重実行の排除とアロケーション最適化

## ADR's STATUS

Accepted (2026-03-12)

## CONTEXT

3000VU 負荷試験（feed-read-3000vu シナリオ）の pprof プロファイル結果から、以下の2関数が全体アロケーションの約90%を占めていることが判明した。

| 関数 | 累積アロケーション | 割合 |
|------|-------------------|------|
| `sanitizeDescription` | 2,228 GB | 58.1% |
| `convertFeedPageEntries` | 1,239 GB | 32.3% |

**根本原因: 二重 sanitize**

1. Gateway 層（`feed_page_cache_gateway`）がキャッシュロード時に `sanitizeForCache()` で description を sanitize 済みとして事前計算
2. `convertFeedPageEntries()` がその sanitize 済み値を `FeedItem.Description` に設定
3. Handler 層（`convertFeedsToProto()` / `OptimizeFeedsResponse()`）が**再度** `sanitizeDescription()` を呼び出し

これにより、全リクエスト × 全フィードアイテムで bluemonday による HTML パースと正規表現処理が無駄に実行されていた。

さらに、`sanitizeDescription` は同一ロジックが **4箇所** に重複定義されていた:
- `connect/v2/feeds/helpers.go`
- `rest/feed_optimizer.go`
- `rest/rest_feeds/utils.go`
- `gateway/feed_page_cache_gateway/gateway.go`

## DECISION MAKING

### 1. 共通 sanitize ユーティリティの一元化

`utils/sanitize/sanitize.go` に `SanitizeDescription()` を単一定義。`bluemonday.StrictPolicy()` はパッケージレベルのシングルトンとし、goroutine-safe で再利用する（bluemonday 公式推奨パターン）。

### 2. sanitize 実行ポイントを Gateway 層に統一

Clean Architecture の原則に従い、外部データ（DB / RSS）から domain モデルへの変換を行う Gateway 層で sanitize を一度だけ実行する。

- `feed_page_cache_gateway`: 既存の `sanitizeForCache()` を共通関数に置き換え
- `fetch_feed_gateway`: 全9メソッドの `Description` フィールド設定時に `sanitize.SanitizeDescription()` を適用

### 3. Handler 層からの sanitizeDescription 除去

Gateway 層で sanitize 済みの Description がそのまま Handler 層まで到達するため、以下の3箇所から `sanitizeDescription()` 呼び出しと関数定義を削除:
- `connect/v2/feeds/helpers.go` — `convertFeedsToProto()`
- `rest/feed_optimizer.go` — `OptimizeFeedsResponse()`
- `rest/rest_feeds/utils.go` — `OptimizeFeedsResponse()`

### 4. convertFeedPageEntries のコンティギュアスアロケーション

`convertFeedPageEntries()` でループごとに `&domain.FeedItem{}` をヒープ確保していた処理を、コンティギュアスなスライス確保に変更:

- **Before**: N+1 回のヒープアロケーション（`make([]*FeedItem, 0, N)` + ループ内 `&FeedItem{}`）
- **After**: 3 回のヒープアロケーション（`make([]FeedItem, N)` + `make([]*FeedItem, N)` + `make([]string, N)`）

追加最適化:
- 空 `Authors` スライスをパッケージレベル変数 `emptyAuthors` で共有（毎回の `[]Author{}` alloc を排除）
- `Links` スライスのバッキング配列を一括確保し、サブスライスで参照

### 5. GOMEMLIMIT デフォルトを 10GiB に更新

負荷試験スクリプトの GOMEMLIMIT デフォルト値を `4GiB` から `10GiB` に変更。4GiB では GC thrashing により p95 レイテンシが 23.56s まで劣化していた実績がある（ADR-000366 参照）。

## RESULTS, EFFECTS

### PROS

- **二重 sanitize の完全排除**: リクエストあたりの bluemonday パース + 正規表現処理が 0 回に（Gateway 層のキャッシュロード時に 1 回のみ）
- **アロケーション大幅削減**: `sanitizeDescription` による 2,228GB/30min のアロケーションが消滅。`convertFeedPageEntries` の N+1 alloc が 3 alloc に削減
- **コード重複の解消**: 4箇所に散在していた同一ロジックを `utils/sanitize` パッケージに統一。保守性が向上
- **Clean Architecture との整合**: データ浄化の責務が Gateway 層（外部データ境界）に明確に帰属
- **GC プレッシャーの軽減**: 短命オブジェクトの生成が減り、Go ランタイムの GC 負荷が下がる

### CONS, TRADEOFF

- **Gateway 層の責務増加**: 全 Gateway メソッドに `sanitize.SanitizeDescription()` 呼び出しが必要。新しい Gateway メソッドを追加する際に sanitize を忘れるリスクがある
- **キャッシュ未経由パスの考慮**: `fetch_feed_gateway` 経由のレガシーパス（DB 直接取得）では、キャッシュの事前計算ではなくリクエスト時に sanitize が走る。ただし、これは以前の Handler 層での sanitize と同等のタイミングであり、性能劣化はない
- **テストデータの更新**: Handler 層のテストで HTML を含む Description を渡していたケースは、Gateway 層で sanitize 済みの plain text を渡すように修正が必要だった

### sanitize 実行フロー（変更後）

```
DB / RSS (raw HTML)
  → Gateway 層: sanitize.SanitizeDescription() — ここで1回だけ sanitize
    → FeedPageCache: SanitizedDescription に格納（キャッシュ経由パス）
    → FeedItem.Description に直接設定（レガシーパス）
  → Usecase 層: そのまま通過
  → Handler 層: そのまま通過（sanitize しない）
  → Client (plain text)
```
