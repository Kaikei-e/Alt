# 000354: Tag Verse Recap Search — 全期間タグ横断検索

## ADR's STATUS

Accepted

## CONTEXT

ADR-349〜353 で導入した Tag Verse（タグ宇宙ビュー）では、タグをクリックすると HUD サイドバーに Recaps タブが表示される。しかし、従来の実装は `getThreeDayRecap()` で最新3日間の Recap のみ取得し、クライアント側で `topTerms` / `genre` 名のテキストマッチングでフィルタリングしていた。

この方式には2つの根本的な問題があった:

1. **時間範囲の制約** — 直近3日分しか検索できず、例えば「LLM」タグをクリックしても2026年1月9日のRecapは表示されない
2. **検索精度の低さ** — クライアント側の文字列マッチングはRecapジャンル名やtopTermsの部分一致に頼るため、関連するRecapを見落とす可能性が高い

**目標**: タグをクリックしたら、過去すべての completed Recap job から、そのタグに関連するジャンルを日付別に表示する。

### 設計選択肢の検討

タグデータは alt-backend の PostgreSQL に、Recapデータは recap-worker の PostgreSQL にそれぞれ格納されている。クロスサービスでの検索には以下の選択肢があった:

| 方式 | メリット | デメリット |
|------|---------|-----------|
| **A: article_id 経由の結合** | 正確なタグ-記事-Recap の紐付け | 2サービス間のDB結合が必要、レイテンシ大 |
| **B: top_terms JSONB 検索** | 単一サービス完結、高速 | `top_terms` とタグ名の一致に依存 |
| **C: 共通タグDB** | 一元管理 | 大規模なアーキテクチャ変更が必要 |

**方式 B（top_terms 検索）を採用**。recap-worker の `recap_subworker_clusters.top_terms` JSONB 配列には、クラスタ内の記事から抽出されたキーワードが格納されており、フィードのタグ名と高い相関がある。単一サービスのクエリで完結するため、実装コストとレイテンシの両面で優位。

## DECISION MAKING

4層（recap-worker → proto → alt-backend → alt-frontend-sv）にわたる垂直スライスとして実装する。

### 1. recap-worker (Rust) — DAO + HTTPエンドポイント

`RecapDao::search_recaps_by_term` メソッドを追加。全 completed job の `recap_subworker_clusters.top_terms` JSONB 配列を横断検索する。

```sql
SELECT DISTINCT ON (rj.job_id, ro.genre)
    rj.job_id, rj.kicked_at, rj.window_days,
    ro.genre, ro.summary_ja, c.top_terms
FROM recap_jobs rj
JOIN recap_outputs ro ON rj.job_id = ro.job_id
JOIN recap_subworker_runs sr
    ON rj.job_id = sr.job_id AND ro.genre = sr.genre AND sr.status = 'succeeded'
JOIN recap_subworker_clusters c ON c.run_id = sr.id
WHERE rj.status = 'completed'
  AND EXISTS (
    SELECT 1 FROM jsonb_array_elements_text(c.top_terms) AS t
    WHERE t ILIKE '%' || $1 || '%'
  )
ORDER BY rj.job_id, ro.genre, rj.kicked_at DESC
```

- `DISTINCT ON (job_id, genre)` で同一ジョブ・同一ジャンルの重複を排除
- `ILIKE` による大文字小文字を無視した部分一致
- `jsonb_array_elements_text` で JSONB 配列を行に展開して検索

HTTP エンドポイント: `GET /v1/recaps/search?term={tag}&limit={n}`

### 2. Proto — RecapService への RPC 追加

```protobuf
rpc SearchRecapsByTag(SearchRecapsByTagRequest) returns (SearchRecapsByTagResponse);
```

`RecapSearchResultItem` メッセージに `job_id`, `executed_at`, `window_days`, `genre`, `summary`, `top_terms`, `bullets` を含む。`buf generate` で Go / TypeScript のコードを再生成。

### 3. alt-backend (Go) — Clean Architecture 全レイヤー

Handler → Usecase → Port → Gateway の各レイヤーに `SearchRecapsByTag` を追加:

- **Domain**: `RecapSearchResult` 構造体
- **Port**: `RecapPort` インターフェースに `SearchRecapsByTag(ctx, tagName, limit)` を追加
- **Gateway**: recap-worker の HTTP API を呼び出す実装
- **Usecase**: Port への委譲
- **Handler**: Connect-RPC ハンドラ。認証チェック、limit バリデーション（デフォルト50、最大200）、domain→proto 変換

### 4. alt-frontend-sv (SvelteKit) — HUDRecapsTab の改修

- `searchRecapsByTag()` Connect-RPC クライアント関数を追加
- `HUDRecapsTab.svelte` を完全書き換え:
  - `getThreeDayRecap` + クライアント側フィルタ → `searchRecapsByTag` API 呼び出しに変更
  - 結果を日付別にグループ化して表示（`$derived` で `groupByDate()`）
  - `$effect` でタグ名変更時に自動リロード
  - 各結果に bullets / summary / topTerms チップを表示

## RESULTS, EFFECTS

### PROS

- **全期間検索**: 過去すべての completed Recap job からタグに関連するジャンルを発見可能。3日間の制約がなくなり、数ヶ月前のRecapも表示される
- **サーバーサイド検索**: クライアント側の文字列マッチングを排除し、PostgreSQL の `ILIKE` + JSONB 検索に移行。検索精度と効率が向上
- **単一サービス完結**: recap-worker 内で検索が完結するため、サービス間通信のオーバーヘッドが最小限（alt-backend → recap-worker の1ホップのみ）
- **Clean Architecture 準拠**: alt-backend の Handler → Usecase → Port → Gateway → Driver パターンに忠実に従い、各レイヤーのテストを独立に実行可能
- **型安全**: Proto 定義からの自動生成により、Rust / Go / TypeScript 間の型整合性を保証

### CONS, TRADEOFF

- **ILIKE の性能特性**: `ILIKE '%term%'` はインデックスを使用しない全文スキャンとなる。completed job 数が数百を超える場合、GIN インデックス（`jsonb_path_ops`）や `pg_trgm` 拡張の導入を検討する必要がある
- **top_terms 依存**: タグ名と `top_terms` の一致に依存するため、完全に異なる表現のタグ（例: タグ「AI」 vs top_term「人工知能」）はマッチしない。将来的にはベクトル類似度検索やシノニム辞書の導入で改善可能
- **`DISTINCT ON` の制約**: 同一ジョブ・同一ジャンルで複数クラスタがマッチする場合、1つのクラスタの `top_terms` のみが返される。ジャンル単位の集約としては十分だが、クラスタレベルの詳細は失われる
