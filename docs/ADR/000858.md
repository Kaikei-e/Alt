# ADR-000858: Knowledge Loop の長時間アイドル後リロードでカードが永続 disabled になる回帰を 3 重防御で復旧する

## Status

Accepted

## Date

2026-04-27

## Affected Services

- alt-frontend-sv: `src/lib/hooks/useKnowledgeLoop.svelte.ts` に `TRANSITION_TIMEOUT_MS = 8_000` と per-call `AbortController` を導入。`inFlight` set に `inFlightStartedAt: Map<string, number>` を併走させ、`replaceSnapshot` が古い in-flight key を gc できるようにした。`resetInFlight(reason)` を追加して外部から in-flight tracking を一括クリアできるようにした
- alt-frontend-sv: `src/lib/hooks/loop-visibility-recovery.ts` を新設。`document.visibilitychange` と `pageshow` を listen し、threshold (30 秒) を超えた hidden 期間または `persisted=true` な pageshow (bfcache restore) で `onRecover` を発火する factory
- alt-frontend-sv: `src/routes/(app)/loop/+page.svelte` が `onMount` で `startVisibilityRecovery` を起動し、`onRecover` で `loop.resetInFlight("visibility")` + 既存 coalesced refresh を呼ぶ
- alt-frontend-sv: 12 件の Vitest を追加 (`useKnowledgeLoop.svelte.test.ts` + 新 `loop-visibility-recovery.test.ts`)。proto / migration 変更なし

## Context

Knowledge Loop の OODA tile はサーバ side の `/loop/transition` 応答中に `disabled={inFlight || dismissing}` で gate される (`LoopEntryTile.svelte:284` 等)。`useKnowledgeLoop.svelte.ts` の `observe()` / `transitionTo()` は `inFlight.add(entryKey)` 後に `try / finally { inFlight.delete(entryKey) }` で必ずクリアする実装になっており、通常運用ではこの gate は数百ミリ秒で解除される。

ところが本番では「タブを長時間 background → 戻ってきたらカードがクリックできなくなる」という回帰が再発した。再現条件をログから絞り込むと:

1. ユーザがカードを click → `inFlight.add` 発火 → `await fetch("/loop/transition", ...)` で待機
2. 待機中にタブが background 化される。ブラウザがネットワーク接続を suspend / bfcache freeze してくることがあり、`await` フレームが永遠に解決されない
3. その後ユーザがタブに戻ってきても `await` は再開せず、`finally` 節も走らないので `inFlight` から `entryKey` が消えない
4. `invalidate("loop:data")` 経由で `replaceSnapshot()` が走り、新しい snapshot で entry が再描画されるが、`replaceSnapshot` は `entries` と `optimisticallyDismissed` だけを再シードして `inFlight` を触らないため、`isInFlight(entryKey) === true` のまま「永続 disabled なボタン」が残る

過去にも同種のレース (fetch storm からの first-click 落としや `JWT expired loop`) を `loop-coalesce` / `makeFirstFrameSkipper` で潰してきたが、いずれも在席中の churn が前提で、長時間アイドル後の dangling promise は対象外だった。

`fb.md` レビューで「カードが押せなくなるバグが残っている」と再発確認されたため、根本対処が必要になった。

## Decision

`inFlight` に依存する disabled gate を、(a) 必ず `finally` を走らせる、(b) snapshot 入替時に stale を gc する、(c) タブ復帰時に明示リセットする、の 3 重防御で復旧させる。

採用した 3 層:

- **Per-call AbortController + 8 秒 timeout**: `post()` 内で `AbortController` を生成し `setTimeout(() => ac.abort(), TRANSITION_TIMEOUT_MS)` で deadline を貼る。`fetch` に `signal: ac.signal` を渡し、`finally { clearTimeout(deadline) }` で必ず後始末する。`AbortError` は `{status: "error", message: "transition_timeout"}` に正規化する。これでネットワークがどんな状態でも 8 秒以内に `finally` が走り、`inFlight.delete` が確実に発火する
- **`replaceSnapshot` の stale gc**: `inFlightStartedAt: Map<string, number>` を `inFlight.add` 時に stamp、`inFlight.delete` 時にクリア。`replaceSnapshot(next)` の最後で「(a) 新 snapshot に存在しない key」「(b) `now - startedAt > TRANSITION_TIMEOUT_MS` の key」を evict する。これは defense-in-depth で、AbortController が何らかの理由で発火しなかった場合の最後の保険になる
- **Visibility / bfcache recovery**: 新 `loop-visibility-recovery.ts` factory が `document.visibilitychange` と `pageshow` を listen する。hidden になったら `getNow()` で時刻を pin し、visible になった瞬間に `now - hiddenAt > thresholdMs (30s)` なら `onRecover({reason: "visibility"})` を呼ぶ。`pageshow` の `persisted === true` (bfcache restore) では即座に `onRecover({reason: "bfcache"})` を呼ぶ。`+page.svelte` 側はこれを `loop.resetInFlight("visibility")` + `coalescedRefresh.trigger()` に繋げる

`resetInFlight(reason)` は内部 set / map を一括クリアするだけで beacon は出さない。telemetry 用 endpoint を将来追加するときに hook できるよう reason 引数を残してある。

検討した代替案と却下理由:

- **`inFlight` を `replaceSnapshot` で全クリアする**: 確実に治るが、長時間アイドルではなく単に直前にクリックしただけのケースまで巻き込んでボタンが瞬間有効化されるので、二重発火の窓を作る。現実的にはほぼ刹那だが、in-flight な server side が 200 を返す race を増やすので却下
- **timeout だけで visibility hook を作らない**: bfcache から復帰した直後に「最初の 1 click が dropped」という別の既知症状が残る。`pageshow persisted` は decision document としても明示しておきたい signal なので採用
- **visibility hook の閾値を 60 秒に揃える (`OBSERVE_THROTTLE_MS` と同じ)**: 閾値統一に意味はなく、長時間 background の検出には逆に遅すぎる。30 秒は alt-tab で頻繁に発火しない最小値として経験的に妥当
- **既存 `useKnowledgeLoopStream` のリコネクト hook に相乗りする**: ストリーム側は別ライフサイクルで dropping the disabled gate を兼任するには副作用が大きい。`replaceSnapshot` を駆動する coalesced refresh と並行して visibility hook を独立に持つほうが責務が明快

## Consequences

### Pros

- 長時間 background → 戻ってきたあとにボタンが永続 disabled になる回帰が消える。8 秒 timeout が確定的に効くので `await` フレームが再開しないケースでもスタックしない
- bfcache restore (`pageshow persisted=true`) は visibility イベントが揃わないことがあるため、`pageshow` を独立 listener にしたことで Safari / Firefox 系の bfcache 挙動も拾える
- `inFlightStartedAt` map のおかげで「stale な key だけ」を選択的に evict できる。直前に click した entry はそのまま gate されるため、サーバ側での 200/409 と client gate が二重発火しない
- 12 件の Vitest が `replaceSnapshot stale gc` / `transitionTo aborts after timeout` / `resetInFlight clears all` / `visibility recovery > 30s` / `bfcache pageshow persisted` を pin。回帰テストとして長期に残る

### Cons / Tradeoffs

- `setTimeout` を per-call で貼るので、極端に高頻度な observe 連射では timer 生成コストがわずかに増える。`OBSERVE_THROTTLE_MS = 60_000` の前段スロットルが効いているため実害はない
- `inFlightStartedAt` map は `inFlight` set とほぼ同じライフサイクルだが、片方の delete を忘れると map が肥大化する。常に対で操作するルールを `useKnowledgeLoop.svelte.ts` 内のコメントに固定し、新 helper を経由する設計には踏み込まなかった
- `loop-visibility-recovery.ts` は `document` を依存に取るため SSR では起動しない。`+page.svelte` 側の `onMount` ガードに依存している
- 30 秒閾値は一律。`prefers-reduced-motion` / 低速回線などの individual variability は拾えない。実害が出たら閾値を `getCurrentLensMode()` 等の prefs で可変化する余地はある
- 現時点では beacon を出さないので、本回帰が再発しても本番ログから「visibility recovery が何回発火したか」を直接観測できない。`loop_entry_unclickable_recovery_total{reason}` カウンタは本 ADR の scope 外で、必要になったら別 ADR で追加する
