# ADR-000928: Ask Augur の neighbor searcher を DI で配線して related citations を production で実際に emit させる

## Status

Accepted

## Date

2026-05-27

## Affected Services

- rag-orchestrator (Go): `internal/di/container.go` の DI 配線修正 (15 行追加 / 2 行削除、ロジック変更はなし)

## Context

[[000927]] で `HybridSearcher.SearchNeighbors` と `answer_with_rag_usecase.buildRelatedCitations` を実装し、`augur_messages.related_citations` JSONB カラムも migrate 済の状態で deploy したが、**production の新規 conversation でも Related 区画が一切表示されない**とユーザから報告があった。CI は全 pass (313 e2e 全通過) しており、FE / proto / migration / handler convert はすべて正しく動いていた。

### Evidence (log-seeker による read-only 調査)

| 確認項目 | 結果 | 解釈 |
|---|---|---|
| `\d augur_messages` | `related_citations JSONB NOT NULL DEFAULT '[]'` 列が存在 | migration 適用済 |
| conversation `5a0d6f77-…` の assistant turns (2026-05-27 11:42 / 11:43 UTC, deploy 後) | `citations` は kind=article + 有効 UUID `ref_id` で 5 件入る、`related_citations` は両方とも空配列 | direct hydration は機能、related だけ無音で 0 件 |
| `rag-orchestrator` ログ (直近 60 分) | `hybrid_search_enabled","bm25_source":"meilisearch"` のみ。`related_citation_lookup_failed` / `neighbor` 系の警告は無 | エラーは出ず、ログでは何が壊れているか観測不能 |
| `di/container.go` 該当範囲 | `NewHybridSearchRepository` は `cfg.Hybrid.BM25Source == "postgres"` 真ブランチでしか呼ばれない / `answerOpts` に `WithNeighborSearcher(...)` が一度も append されない | DI で 2 段の wire 漏れ |

### Root cause

2 つの欠落が直列で重なって `buildRelatedCitations` を必ず no-op にしていた:

1. `cfg.Hybrid.BM25Source == "meilisearch"` ブランチでは postgres ベースの `HybridSearcher` が**生成されない** (= neighbor 検索の素材が物理的に存在しない)
2. 真ブランチに入る経路でも `answerOpts` に `WithNeighborSearcher` が**append されない** (= 仮に instance があっても usecase に渡らない)

`buildRelatedCitations` の冒頭は意図的に `if u.neighborSearcher == nil || len(direct) == 0 { return nil }` で守られている ([[000927]] の「neighborSearcher が未設定なら無音で no-op」設計)。これは「テスト / opt-out deploy で安全に切れる」ことを優先した defensive 設計だったが、結果として **DI が wire し忘れている状態と、設定によって意図的に切られている状態を区別できない** silent failure mode を生んだ。

ADR-926 Cons §3 で「新規 emitter が kind を忘れる退化を server 側では検知できない」と既に予見していたのと同じ構造を、ADR-927 でも作り込んでしまった: **defensive な nil ガードが「機能していない」を「機能を意図的に切っている」と同形にしてしまう**問題。

### immutable invariant への影響

DI 配線漏れの期間 (ADR-927 deploy ～ 本 ADR deploy) に作られた assistant turns は `related_citations = '[]'` で persist される。`augur_messages` は append-only ([[000836]]) なので backfill は不可。これは仕様通りの out-of-scope ([[000927]] §9 で「過去 conversation の backfill はしない」と明記済) として受容する。

## Decision

`di/container.go` で **bm25_source 設定とは独立に** postgres-backed の `HybridSearchRepository` を 1 個生成し、neighbor 専用 dependency として `WithNeighborSearcher` に**無条件で**渡す。retrieve 経路は `bm25_source=postgres` のときに同じ instance を再利用するため、connection 追加コストはゼロ。

### 採用設計

```go
// neighborSearcher を bm25_source 分岐の外に hoist
neighborSearcher := repository.NewHybridSearchRepository(pool, int(cfg.RAG.RRFK))

if cfg.Hybrid.Enabled {
    if cfg.Hybrid.BM25Source == "postgres" {
        opts = append(opts, usecase.WithHybridSearcher(neighborSearcher))  // 同 instance を再利用
        log.Info("hybrid_search_enabled",
            slog.String("bm25_source", "postgres"),
            slog.Int("rrfk", int(cfg.RAG.RRFK)))
    } else {
        opts = append(opts, usecase.WithBM25Searcher(searchClient))
        log.Info("hybrid_search_enabled",
            slog.String("bm25_source", "meilisearch"),
            slog.Float64("alpha", cfg.Hybrid.Alpha),
            slog.Int("bm25_limit", cfg.Hybrid.BM25Limit))
    }
}

// answerOpts には無条件で neighborSearcher を載せる
answerOpts = append(answerOpts, usecase.WithNeighborSearcher(neighborSearcher))
log.Info("neighbor_searcher_enabled")
```

理由:

- neighbor 検索は **postgres の pgvector + tsvector RRF** に依存する。meilisearch 経路では scoring / index 仕様が違うため流用できず、別実装を作るのは ADR-927 の scope 外として明示済
- retrieve 経路の `bm25_source` 切替は user-facing 設定として保持する。neighbor 用は **内部の隠し dependency** として常時 wire しておく
- `pool` は既に `chunkRepo` / `docRepo` で共有されており、追加 instance 1 個は connection budget にほぼ影響しない
- `bm25_source=postgres` のときの retrieve 経路は同じ instance を共有するため、2 個目を作る無駄を避ける

### 検討した代替案と却下理由

#### (A) `cfg.Hybrid.BM25Source` を `postgres` に切り替えて neighbor も retrieve も postgres RRF に揃える

却下。それは meilisearch ベースの primary retrieval を運用ごと変えるという別決定で、本 ADR の scope を超える。retrieve 経路の bm25_source 選択はパフォーマンス・コスト・カバレッジの観点で別議論を経るべきで、Related citations 修正に巻き込まない。

#### (B) `domain.HybridSearcher` を 2 つの interface (`HybridSearcher` / `NeighborSearcher`) に分割する

却下。インターフェース上は同じ pgvector + tsvector RRF を異なる発火点から呼ぶだけで、分けると 2 つの実装が同期する負担が残る。同 instance を 2 つの role で使う方が単純で誤配線も起こりにくい。将来 neighbor 専用に scoring を変えたくなったときに分割を検討する。

#### (C) `buildRelatedCitations` 内で `neighborSearcher == nil` を fatal 扱いにして panic / error を上げる

却下。ユニットテストや切り戻し運用で安全に nil を許容する defensive 設計を捨てると別の障害モードを生む。代わりに DI ログ (`neighbor_searcher_enabled`) を追加して、起動時点で wire 状態を観測可能にする。これは Cons §3 で挙げる継続課題に紐づく。

#### (D) Pact / contract test で wire 状態を保証する

却下しないが本 ADR の scope 外。`internal/di` の wiring を verify するテストは現状コードベースに存在しないため、別タスクで `container_test.go` 相当の integration test を立ち上げるかは別議論。

### TDD 規律

本修正は 15 行の DI 配線で、`go test ./...` で既存 1300+ tests が引き続き green であることが GREEN 確認の中心。`internal/usecase/related_citations_test.go` は fake `HybridSearcher` で `buildRelatedCitations` の挙動を既に固定しており、DI の wire 結果がランタイム挙動として再現するかは [[000927]] のテスト群が間接的に守る。`internal/di` 専用テストは付けず、commit 直前に `go test ./...` / `gofmt -l .` / `go vet ./...` のローカル green を確認してから landing した。

post-deploy の検証は以下で行う (本 ADR の deploy 後):

```sql
-- 新規 conversation の関連 citation 比率
SELECT
  COUNT(*) FILTER (WHERE jsonb_array_length(related_citations) > 0)::float
  / COUNT(*) AS related_ratio
FROM augur_messages
WHERE role = 'assistant'
  AND created_at > now() - interval '1 hour';
```

ベースライン目標は `> 0.5` (direct citation が 1 件以上ある assistant turn のうち、半数以上で neighbor が返る)。0 のままなら DI 以外のレイヤ (HybridSearch / pgvector index / encoder) を疑う。

## Consequences

### Pros

- production の新規 conversation で **`related_citations` が実際に emit される**。CitationRail の Related 区画が UI に出現し、ユーザ報告の「修正が効いていない」現象が解消する
- neighbor 検索は retrieve 経路の `bm25_source` 設定から切り離されたため、retrieve を meilisearch で運用しつつ neighbor は postgres RRF で取る、という Alt の現状運用に整合する
- `bm25_source=postgres` のときは同 instance が retrieve / neighbor の両方で再利用されるので、追加の DB connection / RRF 計算コストは増えない
- DI ログに `neighbor_searcher_enabled` を追加したことで、起動時点で wire 状態が observable になる。今後同型の wire 漏れが起きても起動ログで即座に検知できる
- `internal/di` のコード変更は 15 行と最小限で、`buildRelatedCitations` / SearchNeighbors / handler の生成済コードは触らずに済む。レビュー / rollback どちらも軽い

### Cons / Tradeoffs

- **DI 配線忘れに対する一般的なガードは入っていない**。本 ADR は具体的な 1 箇所を直しただけで、別の future option を `answerOpts` に append し忘れる類のバグは引き続き silent に通る。本質的な対策は「`neighborSearcher == nil` を起動時に panic 化する non-defensive 設計」か「DI 統合テスト (`container_test.go`)」だが、いずれも別 ADR の議論として残す
- defensive nil ガードを維持したため、`buildRelatedCitations` の中で「意図的に切っている (テスト)」と「DI で間違えて切れている (本 bug)」が区別できない silent failure mode は構造的に残る。`neighbor_searcher_enabled` の起動ログでの観測が唯一の検知経路
- DI 配線漏れ期間 (ADR-927 deploy ～ 本 ADR deploy) の assistant turns は `related_citations = '[]'` で append-only 持続する。backfill しないので、ユーザがその期間の conversation を遡って開いても Related は永久に空。これは [[000927]] §9 の方針 (legacy 行は backfill しない) と整合し、append-only invariant を曲げないための受容コスト
- ADR-927 deploy から本 ADR deploy までの間に作られた augur 会話の数だけ「related が出るはずだったのに出なかった」が permanently 残る。`5a0d6f77-…` は 2 件、それ以外の全テナント分は未集計 (個別の会話単位の影響なので集計しない)
- `cfg.Hybrid.Enabled = false` の deploy 構成では (将来そういう運用に切り替えた場合) `WithHybridSearcher` も `WithBM25Searcher` も呼ばれず retrieve が完全に切れるが、`WithNeighborSearcher` だけは無条件で wire される非対称が残る。これは neighbor を「内部隠し依存」として独立に扱う設計判断の副作用で、`cfg.Hybrid.Enabled` を本当に切る運用には別途設定整理が要る (Hybrid 無効化は現状運用上想定されていないので明示的な追加対応はしない)
