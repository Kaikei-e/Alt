# ADR-000788: auth-hub Hurl suite を 2-phase invocation + `--jobs 1` で回す

## Status

Accepted

## Date

2026-04-19

## Affected Services

- auth-hub (Go) — 挙動無変更
- `e2e/hurl/auth-hub/run.sh` — 2-phase 実行ロジック
- `e2e/hurl/auth-hub/00-setup.hurl` — seed 単独ファイル

## Context

[[000787]] で決めた browser-flow seed は、`00-setup.hurl` の最後の step で以下 3 つの Hurl capture を作る:

- `kratos_session_cookie` — HMAC 署名済みの `ory_kratos_session` cookie value
- `user_id` — `$.session.identity.id`
- `session_id` — `$.session.id`

本 suite 側の 12 ファイル (`01-health.hurl` 〜 `12-validate-jwt-shape.hurl`) はこの 3 値すべてに依存する。特に `kratos_session_cookie` は cookie ヘッダに直書きするため、全シナリオの冒頭で `Cookie: ory_kratos_session={{kratos_session_cookie}}` が必要。

Hurl 7.1 の `--test` モードは **captures が file-scoped** で、同一 invocation 内でも file を跨いでは変数が伝播しない。つまり:

```bash
hurl --test 00-setup.hurl 01-*.hurl 02-*.hurl ...
```

と 1 度に投げても、02-* のシナリオでは `{{kratos_session_cookie}}` は **undefined** になる ([[000765]] の acolyte-orchestrator が `--jobs 1` で単一 file 内の captures を維持するのとは状況が違う)。

加えて auth-hub の internal rate limit (`/internal/system-user`) は `10.0/60.0 req/s`, burst 3 でハードコードされており、config env からの調整は不能 (`auth-hub/cmd/auth-hub/main.go:156` で `rate.Limit(10.0/60.0), 3` 直書き)。`--jobs 2` 以上で 09 / 10 / 11 を並走させると burst を食い合って 429 が発生する。

初期実装では 1 回の Hurl 呼び出しで全 file を回そうとして、以下 2 通りの失敗が交互に起きた:

1. `{{kratos_session_cookie}}` undefined で 02 / 05 / 07 / 12 が 401
2. 09 の assertion が「`{{user_id}}` と等しい」に依存しており、undefined 時に assert fail → Hurl が自動 retry → 429 を連鎖発火

## Decision

**2 段階の Hurl invocation + `--jobs 1` で suite を回す。** dispatcher (`run.sh`) は以下を順に実行する:

### Phase 1: seed-only invocation

```bash
docker run --rm --network $STAGING_PROJECT_NAME \
  -v $ROOT:$ROOT -w $ROOT \
  $HURL_IMAGE \
  --test \
  --no-output \
  --report-json $REPORT_DIR/seed-json \
  --file-root $ROOT \
  --variable auth_hub_url=... \
  --variable kratos_public_url=... \
  --variable kratos_admin_url=... \
  --variable email=... \
  --variable tenant_id=... \
  --variable run_id=... \
  --secret password=... \
  e2e/hurl/auth-hub/00-setup.hurl
```

`--report-json` は Hurl の標準 JSON レポート (`report.json`) を指定ディレクトリに出力する。shape は以下:

```jsonc
[
  {
    "filename": "e2e/hurl/auth-hub/00-setup.hurl",
    "entries": [
      { /* request 1 */ },
      ...,
      { "captures": [
          {"name":"kratos_session_cookie","value":"MTc3..."},
          {"name":"user_id","value":"75b67241-..."},
          {"name":"session_id","value":"9074c5af-..."}
      ]}
    ],
    "success": true
  }
]
```

### Phase 1.5: capture extraction

dispatcher は host の `jq` を使って必要な 3 値を拾い出す:

```bash
capture_value() {
  jq -r --arg n "$1" \
    '.[0].entries[] | select(.captures != null) | .captures[] | select(.name==$n) | .value' \
    "$SEED_REPORT_DIR/report.json" | tail -1
}
KRATOS_SESSION_COOKIE="$(capture_value kratos_session_cookie)"
USER_ID="$(capture_value user_id)"
SESSION_ID="$(capture_value session_id)"
```

3 値のいずれかが空ならエラーで停止 (seed 失敗の早期検出)。`jq` は self-hosted deploy runner と dev box の双方に pre-install されているので container 化しない。

### Phase 2: main suite invocation

```bash
docker run ... $HURL_IMAGE \
  --test \
  --jobs 1 \
  --retry 5 --retry-interval 500 \
  --file-root $ROOT \
  --variable auth_hub_url=... \
  ...
  --variable user_id=$USER_ID \
  --variable session_id=$SESSION_ID \
  --secret kratos_session_cookie=$KRATOS_SESSION_COOKIE \
  --secret password=... \
  --secret backend_token_secret=... \
  --report-junit $REPORT_DIR/junit.xml \
  --report-html $REPORT_DIR/html \
  e2e/hurl/auth-hub/0[1-9]-*.hurl \
  e2e/hurl/auth-hub/1[0-2]-*.hurl
```

ポイント:

- `kratos_session_cookie` は `--secret` として注入するので Hurl の JUnit/HTML レポートから redact される (audit F-002 — [[000778]])
- `--jobs 1` で 09 / 10 / 11 の順次実行を保証 (internal rate limit burst を食い合わない)
- `--retry 5 --retry-interval 500` は Kratos admin API の cold-start 5xx を吸収するための全 file 共通設定。09 のみ retry: 3, retry-interval: 6000 を [Options] で上書きし、internal rate limit refill を待てるようにしている
- suite glob に `nullglob` を使い 1[3-9]-*.hurl / 2[0-9]-*.hurl が将来追加されても script 変更不要

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| 1 回の Hurl invocation で全 file を回す | captures が file-scoped のため変数が undefined になる。Hurl 側で cross-file capture を enable する option は存在しない (7.1 時点) |
| 00-setup.hurl を含めずに、各下流 file の冒頭で自前 seed | suite 全体が 6 倍長くなる (7 file × seed 5 step)。Kratos admin への負荷も 7 倍 |
| session を共有せず、各 file が独立の identity を create | identity create は admin rate limit + seed overhead が発生。DB clean up も複雑化 |
| seed 結果を fixture ファイルに書き出して固定値にする | Kratos の `ory_kratos_session` cookie は HMAC で毎回異なる値を生成する。固定化できない |
| `--jobs 4` で並列実行して rate limit は retry で吸収 | retry 待ち時間が壁時計で増えるだけで順序保証もない。`--jobs 1` の方が CI ログが読みやすく deterministic |
| Hurl の `--variables-file` を使う | Hurl 7.1 の variables-file は `key=value` 行の static ファイル。動的に生成する手間は --variable 注入と同じ |
| auth-hub の internal rate limit を env 化して staging で緩める | プロダクション挙動の変更 (feature change) になる。E2E のためにプロダクションを変えない原則を守る |
| jq ではなく Python でパース | self-hosted deploy runner に jq が入っている以上、dependency を増やさない方が良い |
| Hurl image に jq を同梱するカスタムイメージを用意 | upstream の `ghcr.io/orange-opensource/hurl:7.1.0` を純正で使い続けるほうが追従コストが低い |

### 範囲外

- Hurl 7.2+ で cross-file captures がサポートされた場合の移行 — その時改めて検討
- auth-hub の internal rate limit を env 化する機能追加 — 本 suite の外で個別判断
- Phase 1 の seed が部分的に失敗したときの retry 戦略 (現状は即エラー停止) — 将来 flake が出たら対応

## Consequences

### Pros

- **session seed が `run.sh` の明示的な step として可視化される**。デバッグ時に「どこで login しているか」「capture が取れているか」が JSON レポートで確認できる
- **`kratos_session_cookie` は `--secret` 注入なので Hurl レポートで redact される**。JUnit XML / HTML report を artifact として upload しても session が漏れない
- **`--jobs 1` でシナリオの順序が決定論的**。CI ログが線形に読める。rate limit (10 req/min) の存在下でも 09 / 10 / 11 が安定して通る
- **Hurl の upstream image を純正のまま使える**。ビルド時間 0、追随コスト 0
- **2-phase 化により seed だけ再実行する debug 手段が残る**。`KEEP_STACK=1` 環境で 00-setup.hurl を単独で何度でも叩ける

### Cons / Tradeoffs

- **dispatcher に jq 依存が入る**。self-hosted deploy runner と開発 PC には pre-install されているが、他の CI プラットフォームに移植するときは先に確認が要る
- **Phase 2 の 1 ファイル目 (01-health) が Phase 1 完了後すぐに叩かれる**。Kratos admin API の warm-up 差分が 1–2 秒発生する可能性があり、01-health は影響を受けないが 09 は retry-interval 6s を付けて吸収している
- **Hurl report が 2 種類に分割される** (`seed-json/report.json` と `junit.xml` + `html/`)。CI ログ上で両方を参照する必要があるが、artifact 名は 1 つに統合
- **Phase 1 で失敗すると Phase 2 が走らず debug 情報が減る**。Phase 1 失敗時の log dump は failure hook でカバーしているが、seed の中間状態を verbose に出すか検討余地あり
- **session の有効期間 (1h) を越えると Phase 2 が全滅する**。本 suite は 1 分以内で完走するので現実的問題ではない
