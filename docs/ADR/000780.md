# ADR-000780: alt-backend E2E の認証を pre-minted HS256 JWT で処理し auth-hub / Kratos 依存を staging から外す

## Status

Accepted

## Date

2026-04-19

## Affected Services

- alt-backend (Go) — staging compose の env に `BACKEND_TOKEN_SECRET_FILE` / `BACKEND_TOKEN_ISSUER` / `BACKEND_TOKEN_AUDIENCE` を設定し、JWT 検証を HS256 ローカル鍵で成立させる
- `e2e/fixtures/staging-secrets/alt_backend_token_secret.txt` (新規) — HS256 共有秘密鍵（staging 専用テストフィクスチャ）
- `e2e/fixtures/alt-backend/test-jwt.txt` (新規) — 上記鍵で署名済みの長期 JWT (role=admin, exp=2099-01-01)
- `compose/compose.staging.yaml` — `alt_backend_token_secret` Docker secret + alt-backend service への mount

## Context

alt-backend の JWT auth middleware は以下で構成される:

1. `X-Alt-Backend-Token` header を読む (`middleware/jwt_middleware.go:112`)
2. `cfg.Auth.BackendTokenSecret` で HMAC-SHA256 を検証
3. `claims.Issuer == cfg.Auth.BackendTokenIssuer` を確認
4. `claims.Audience` に `cfg.Auth.BackendTokenAudience` が含まれることを確認
5. `claims.Subject` と `claims.TenantID` が non-nil UUID であることを検証

認証 middleware は auth-hub を呼ばず、ローカル鍵のみで完結する。auth-hub は別 handler (`/v1/system-user` の `KratosClient.GetFirstIdentityID`) でしか呼ばれない。

本番相当の Kratos login flow を staging で回そうとすると、以下を追加起動する必要がある:

- `kratos-db` (postgres:16-alpine)
- `kratos-migrate` (Kratos schema 初期化)
- `kratos` (ory/kratos)
- `auth-hub` (Go サービス、nginx 設定と本番 Kratos URL に依存)
- optional: frontend からの login POST を再現する helper

これらは alt-backend の REST surface を検証する目的 ([[000778]]) に対して完全にオーバーヘッド。Kratos は E2E 前に初期設定が必要で、staging 側に identity schema と少なくとも 1 つの user を seed する運用も抱え込む。

staging-only fixture のコミット方針は [[000763]] の頃から存在し、既に `e2e/fixtures/staging-secrets/meili_master_key.txt` が public repo にコミットされている。staging-only であること、alt-staging network が `internal: true` で外部到達がないこと、fixture README で明示されていることが合意の前提。

## Decision

**staging 用 HS256 秘密鍵 1 個と、同鍵で署名した長期 JWT 1 個を `e2e/fixtures/` にコミットし、Hurl scenario が `X-Alt-Backend-Token` header に注入する方式を採る。** 採用構成:

1. **秘密鍵は `e2e/fixtures/staging-secrets/alt_backend_token_secret.txt` に改行なしで 1 行**。既存の `meili_master_key.txt` と同じ慣習。README に「staging-only test fixture; 本番鍵とは無関係」を明記。
2. **compose secret としてマウント**する。`compose/compose.staging.yaml` の top-level `secrets:` に `alt_backend_token_secret` エントリを追加、alt-backend service の `secrets:` block で参照、alt-backend env に `BACKEND_TOKEN_SECRET_FILE=/run/secrets/alt_backend_token_secret` を指定する。本番の `_FILE` suffix 規約と完全同形なので、staging と prod で env の形が揃う。
3. **JWT は `e2e/fixtures/alt-backend/test-jwt.txt` に pre-mint する**。HS256, sub=`00000000-0000-0000-0000-000000000001`, tenant_id=同, role=`admin`, iss=`alt-staging-auth-hub`, aud=`alt-backend`, exp=2099-01-01。role=admin は `/v1/dashboard/*` (RequireAdmin) を通すため。exp は 70 年猶予で rotation 不要。
4. **alt-backend env で issuer / audience を fixture に合わせて固定**する。`BACKEND_TOKEN_ISSUER=alt-staging-auth-hub` / `BACKEND_TOKEN_AUDIENCE=alt-backend` を compose.staging.yaml に書くことで、prod の issuer/audience と衝突しない。
5. **Hurl scenario は `run.sh` が `tr -d '\n' < e2e/fixtures/alt-backend/test-jwt.txt` で読み込み、`--variable jwt=...` で全 scenario に渡す**。各 scenario は `X-Alt-Backend-Token: {{jwt}}` を private endpoint に付けるだけ。
6. **CSRF は別系統で、各 state-mutating scenario がファイル先頭で `GET /v1/csrf-token` を叩いて `[Captures]` する**。これは JWT と独立した middleware （`middleware/csrf_middleware.go`）のため本 ADR の範囲外。

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| staging に Kratos + auth-hub を立てて本物の login flow を回す | kratos-db / kratos-migrate / kratos / auth-hub の 4 コンテナを追加起動。CI bring-up 時間が倍以上に伸び、identity schema の seed 運用も背負う。alt-backend の REST surface 検証目的 ([[000778]]) に対してオーバーヘッドが過大 |
| alt-backend 側に test 用 middleware bypass を仕込む (`TEST_AUTH_BYPASS=true` など) | プロダクションコードに test-only 分岐が混入する。OSS として公開しているため、うっかり prod env で bypass が有効になる類の事故リスクも抱える。現状の「E2E はプロダクション binary をそのまま叩く」原則と矛盾 |
| Hurl scenario ごとに JWT を動的生成する | 各 scenario の先頭に JWT mint 処理を書く必要がある。Hurl は HMAC-SHA256 の JWT 署名を直接サポートせず、Docker sidecar か external helper が必要になり複雑化 |
| 短期 exp の JWT を run.sh で毎回生成する | 再現性の利点は薄い (staging-only 固定鍵で発行されるので長期 exp と等価) 、生成スクリプトの依存 (python + pyjwt 等) を run.sh に持ち込む手間が増える |
| RS256 鍵ペアをコミットする | 公開 repo に公開鍵だけ置く構成にしても、公開鍵から signing 要求を再現できる攻撃面はないが、秘密鍵/公開鍵 2 ファイル管理の手間が増える。HS256 単一鍵で staging は十分 |

### 範囲外 (今回やらないこと)

- prod の JWT 鍵 rotation 運用 — 本番は Docker secret + 定期 rotation で別運用
- tenant / user レベルの multi-identity テスト — 単一 user で REST surface を通す目的に対して、identity 切り替えは scope 外
- Kratos session login の E2E — frontend E2E (Playwright) が担う領域

## Consequences

### Pros

- **auth 系の 4 コンテナを staging から外せる**。compose.staging.yaml の alt-backend profile が 4 service (`alt-backend-db` / `alt-backend-db-migrator` / `alt-backend-deps-stub` / `alt-backend`) に収まり、Kratos 初期化の運用を持ち込まない
- **env の形が prod と揃う**。`BACKEND_TOKEN_SECRET_FILE=/run/secrets/...` / `BACKEND_TOKEN_ISSUER=...` / `BACKEND_TOKEN_AUDIENCE=...` は prod compose と同じ環境変数群で、staging と prod の drift を最小化
- **Hurl scenario から見ると認証が header 1 個**で済む。`X-Alt-Backend-Token: {{jwt}}` を書くだけで他のサービスや token endpoint に依存しない
- **JWT の exp=2099 で将来の flake がない**。fixture rotation が必要になるのは鍵を差し替える時だけで、通常運用では触らない

### Cons / Tradeoffs

- **公開 repo に HS256 秘密鍵と署名済み JWT がコミットされる**。staging-only fixture であり、alt-staging network が `internal: true` で外部から到達不可能である前提に依存。fixture README と本 ADR で「prod 鍵とは無関係」を明示するが、新規コントリビューターが誤解するリスクは残る。鍵を本番にコピペすることは compose.staging.yaml 以外から参照されないため起きにくいが、監査時の質問として必ず出る
- **/v1/system-user エンドポイントの挙動は検証されない**。auth-hub alias は deps-stub に向けるが、Kratos identity fetch を実装していないため scenario で叩くと stub-noop 応答になる。内部 handler のため通常の REST surface 検証対象外だが、将来 E2E に含める判断があれば stub を拡張する必要がある
- **single tenant / single user での検証に固定される**。tenant 切り替えや role 切り替えのテストは後続 ADR で別 JWT fixture を追加する形になる
- **JWT fixture は TDD の「RED」で発見しにくい**。secret と JWT を両方同時に差し替える必要があるため、regenerate 手順を README にコミットした (`e2e/fixtures/alt-backend/README.md`) が、手順漏れで flake したら `docker logs alt-staging-alt-backend` で `invalid backend token` として初めて気づく
