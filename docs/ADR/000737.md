# ADR-000737: search-indexer REST を :9443 peer-identity 化し Acolyte sidecar を VERIFY_CLIENT=on へ昇格する

## Status

Superseded by [[000741]] (受信側 mTLS 全サービス展開)。本 ADR は search-indexer / Acolyte の pilot を記録したもので、全サービスへの拡張は [[000741]] に継承した。

## Date

2026-04-15

## Affected Services

- search-indexer (Go) — `middleware/peer_identity_middleware.go` 新設、`bootstrap/servers.go::newMTLSMuxHandler` 新設、`:9443` 上で REST (`/v1/search`) + Connect-RPC (`/services.search.v2.SearchService/*`) を peer-identity allowlist でゲート
- acolyte-orchestrator (Python / Starlette) — `acolyte/infra/peer_identity.py` で `X-Alt-Peer-Identity` header を request.state に bind する ASGI middleware を新設、`main.py::create_app` に組み込み
- docker/nginx-tls-sidecar — `ssl_verify_client on` 時に peer CN を `X-Alt-Peer-Identity` として upstream に転送するよう `nginx.conf.template` + `entrypoint.sh` を更新
- compose/acolyte.yaml — `VERIFY_CLIENT` のデフォルトを `off` → `on` に昇格

## Context

[[000725]] / [[000727]] / [[000729]] で step-ca ベース mTLS + alt-backend/auth-hub/pre-processor/search-indexer (:9443) + BFF outbound mTLS を永続化したが、以下の 3 ギャップが残っていた:

1. **search-indexer REST `/v1/search` は plaintext `:9300` でしか serving していない**。Connect-RPC は既に peer allowlist (ADR-000727) で gated だが、REST 側は `ServiceAuthMiddleware` (X-Service-Token) のみで、転送路は平文。PM-2026-025 はこの REST 経路で起きた。
2. **Acolyte の nginx TLS sidecar は `VERIFY_CLIENT=off`**（ADR-000729 で指摘済）。TLS 暗号化はするが client cert を検証しないため、acolyte upstream は caller identity を認識できない。
3. **`X-Alt-Peer-Identity` という慣習的 header が実装されていない**。alt-backend はログのみ、Python 側は header すら受け取らない。peer identity を audit log や allowlist に使うフックがない。

全面的な mTLS 移行 ([[000738]]) に先立ち、最も risky な search-indexer REST 経路を peer-identity 化し、Acolyte を sidecar レベルで完全 mTLS 化することが本 ADR の目的。

## Decision

**search-indexer の `:9443` に REST も同居させ peer-identity allowlist でゲートする。Acolyte の nginx sidecar を `VERIFY_CLIENT=on` に昇格し、CN を `X-Alt-Peer-Identity` header で upstream に渡す。**

### 1. search-indexer: `PeerIdentityMiddleware` 新設

`middleware/peer_identity_middleware.go` で allowlist-based middleware を実装。`tls.ConnectionState.PeerCertificates[0].Subject.CommonName` を `MTLS_ALLOWED_PEERS` CSV と照合し、不一致は 403、不在 (plaintext または client cert 無し) は 401。通過時は `X-Alt-Peer-Identity=<CN>` を request header に set。5 ケース単体テスト付き (TLS 有・無、allow/deny、empty allowlist、client spoofing 防止)。

### 2. search-indexer: `newMTLSMuxHandler` 新設と `:9443` への REST 同居

`bootstrap/servers.go::newMTLSMuxHandler` が REST `/v1/search` と Connect-RPC `/services.search.v2.SearchService/*` を単一 mux に統合し、両者を `peer.Require(...)` で包む。`app.go` で `app.mtlsServer` の handler を `connectServer.Handler` から新 mux に差し替え。`:9300` / `:9301` plaintext listeners は Phase C (token 完全撤去) まで並行温存。

### 3. Acolyte nginx sidecar: CN header 転送

`docker/nginx-tls-sidecar/nginx.conf.template` の `location /` に `proxy_set_header X-Alt-Peer-Identity ${PEER_IDENTITY_VALUE}` を追加。`entrypoint.sh` で `VERIFY_CLIENT=on` のとき `PEER_IDENTITY_VALUE='$ssl_client_s_dn_cn'`、`off` のとき `""` を envsubst に渡す。これで sidecar が TLS-verify 済 CN を upstream へそのまま注入する。

### 4. Acolyte Starlette: `PeerIdentityMiddleware` 新設

`acolyte/infra/peer_identity.py` は ASGI BaseHTTPMiddleware として:
- nginx sidecar から受け取った `X-Alt-Peer-Identity` header を `request.state.peer_identity` に bind
- `ACOLYTE_ORCHESTRATOR_VERIFY_CLIENT=off` の場合は client-spoofed header を破棄 (fail-safe)
- `strict=True` モードでは missing/not-allowlisted peer を 401/403 で拒否 (Phase C で有効化)
- structlog `contextvars` に `peer=<CN>` を bind し、全ログに caller identity が出る

`main.py::create_app` で Starlette の `middleware` stack に組み込み。7 ケース単体テスト付き。

### 5. compose/acolyte.yaml: `VERIFY_CLIENT` デフォルトを `on` に

`ACOLYTE_ORCHESTRATOR_VERIFY_CLIENT:-on`。re-deploy 後は sidecar が client cert を要求するため、mTLS 未対応 caller は 400 ハンドシェイク失敗。現状 BFF は ADR-000729 で mTLS 送信側に切替済なので直接の影響なし。

### 検討した代替案

- **案 1: nginx sidecar を撤去し Starlette に native mTLS を実装** — 計画書原案。uvicorn の `--ssl-cert-reqs=2` は TLS 検証はするが peer cert を ASGI scope に伝播しないため、CN-level allowlist が app 層で実装できない。hypercorn 等への server 乗り換えが必要でスコープ拡大。
- **案 2 (採用): sidecar を `VERIFY_CLIENT=on` に昇格、CN を header で渡す** — nginx の `$ssl_client_s_dn_cn` は TLS handshake 直後に確定し、proxy_set_header 経由で upstream に確実に渡せる。sidecar netns 問題 (ADR-000729) は `compose up --force-recreate <svc>` 後に sidecar も一緒に recreate する既存手順で吸収する。

## Consequences

### Pros

- **search-indexer REST `/v1/search` が plaintext 上を流れなくなる**。peer cert 検証 + allowlist で、PM-2026-025 型の「認証ヘッダ落ち」を根本的に遮断 (そもそも cert 提示必須)。
- **Acolyte の peer identity が Python app で読める**。`request.state.peer_identity` および structlog `peer` context に CN が bind され、監査ログや将来の allowlist enforcement で利用可能。
- **`X-Alt-Peer-Identity` という契約が定まった**。alt-backend (Go middleware), search-indexer (Go middleware), acolyte-orchestrator (Python middleware) の 3 箇所で同じ header 名を使う。今後 A2 で他の Python/Go サービスに横展開する時の雛形。
- **既存 :9443 mtlsServer の有効活用**。新 listener を増やさず、同じ handler をマルチ用途化。ポート追加なし、compose 変更最小。
- **テスト駆動**: Go 側 5 ケース + Python 側 7 ケースが全て green。middleware ロジック (spoof 防止、strict モード、allowlist parsing) が回帰保護された。

### Cons / Tradeoffs

- **:9300 / :9301 plaintext listeners は Phase C まで並行温存**。これは意図的なダウンタイム回避だが、その間は "mTLS on :9443, plaintext on :9300" という dual-stack が続く。Phase C で :9300/:9301 を物理削除するまで両方監視が必要。
- **Acolyte sidecar 撤去ではない**。`VERIFY_CLIENT=on` 昇格のみ。sidecar 共有 netns 問題 (`compose up --force-recreate` 後の手順) は残る。native mTLS への完全移行は、uvicorn の peer-cert 伝播が標準化されるまで、または hypercorn への乗り換えコストを払う判断まで棚上げ。
- **`strict=False` で rollout**。まだ Phase B (client cert 送信化) が完了していないため、peer-identity が無いリクエストも通す。strict=True 化は ADR-000738 (X-Service-Token 完全撤去 + hard cutover) の前提条件として明示する。
- **branch protection 未配線**。`proto-contract.yaml` の header コメントに required status check 名を列挙したが、実際の GitHub Settings 配線は手動で行う必要がある。
