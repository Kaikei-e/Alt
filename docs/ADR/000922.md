# ADR-000922: Heavy-Ranker 流の explainable recall scoring を導入し score breakdown と negative signals を first-class 化する

## Status

Accepted

## Date

2026-05-25

## Affected Services

- alt-backend — `app/job/recall_projector.go` の固定 const weight を `RecallWeightSetV1` / `RecallWeightSetV2HeavyRanker` map に置換。`ScoreRecallSignals(signals, weightSet) → []recallScoreResult` を pure fn として exported (引数経由 / 戻り値で breakdown を返す)。`scoreAndUpsertCandidates` を pure scorer + side-effect upsert に分離し `time.Now()` を `MAX(signals.OccurredAt)` 派生に。`app/domain/recall_signal.go` に `SignalRecentlyDismissed` / `SignalLowSummaryConfidence` 定数追加。`app/domain/recall_candidate.go` に `WeightSetVersion` / `ScoreBreakdown` field、`RecallScoreContribution` struct 追加。`app/config/config.go` に `RecallWeightSet string env:"KNOWLEDGE_HOME_RECALL_WEIGHT_SET" default:"v1_fixed"` 追加。
- migrations-atlas — `20260525000001_add_recall_weight_set_version.sql` で `recall_candidate_view` に `weight_set_version TEXT NOT NULL DEFAULT 'v1_fixed'` + `score_breakdown JSONB NOT NULL DEFAULT '[]'::jsonb` + CHECK 制約。
- proto — `proto/alt/knowledge_home/v1/knowledge_home.proto` の `RecallCandidate` に `optional RecallWeightSetVersion weight_set_version = 7` / `repeated RecallScoreContribution score_breakdown = 8` を additive 追加。`RecallWeightSetVersion` enum (UNSPECIFIED / V1_FIXED / V2_HEAVY_RANKER) と `RecallScoreContribution` message を新設。
- alt-frontend-sv — `src/lib/connect/knowledge_home.ts` の `RecallCandidateData` に `weightSetVersion` / `scoreBreakdown` 追加、`convertRecallCandidate` で additive pass-through。`src/lib/components/why/WhyTypography.svelte` に `scoreBreakdown` prop + on-demand expand UI (negative contributions に `--alt-terracotta` border-left)。`knowledge_home.test.ts` に 2 ケース追加。

## Context

ADR-000913 §D-9 (本 plan の grounding として **Twitter the-algorithm-ml Heavy Ranker**, `projects/home/recap/config/local_prod.yaml`) は次の主張をする:

- positive signals (fav 0.5 / retweet 1.0 / reply 13.5 / reply_engaged_by_author 75.0) と **negative signals** (negative_feedback_v2 −74.0 / report −369.0) を explicit weight として並列定義する。
- weight set は version 化されており「v1_fixed」「v2_heavy_ranker」のように切替できる。runtime config (env var) で選択する。
- 個別 score の **breakdown** を per-candidate に保持し UI / debug で分解可能にする。

Alt の `recall_projector.go` の v1 実装は次の弱点を抱えていた:

1. **weight が `const` で hard-coded**: `weightOpenedNotRevisited = 0.3` のような const 7 個。version 化されておらず、A/B 切替やロールバックが code change を要する。
2. **negative signal が存在しない**: 一度 dismiss / snooze した article でも positive signal が積み上がれば再浮上する。user が押し戻した content が周回する。
3. **score の breakdown が捨てられている**: per-signal contribution を集計する経路はあるが、最終的に `RecallCandidate.RecallScore: float64` の単一値だけ DB に書く。UI / debug で「なぜこの score になったか」を答えられない。
4. **freshness が `time.Now()`**: `scoreAndUpsertCandidates` で `now := time.Now()` を取って `NextSuggestAt / FirstEligibleAt / UpdatedAt` に使う。同じ event log を 2 回 reproject すると違う row が出る (event-time purity 違反)。
5. **runtime config がない**: weight set を切替えたければ const 編集 + 再 build しか手段がない。

PR 7 (commit 14c30da07) で proto 側 (`RecallWeightSetVersion` enum + `RecallScoreContribution` message + `RecallCandidate` field 7/8 additive) と FE 側の `convertRecallCandidate` 拡張は landed。PR 8 (commit ce5e9f10d) で backend 実装本体を landing する。

## Decision

次の 5 つを一括で landing する:

1. **Weight set の名前付き map 化**:
   - `recallWeightSetV1` は v1 値を保持 (back-compat default)。
   - `recallWeightSetV2HeavyRanker` は positive (AugurReferenced 0.45 ← 0.35 アップ) + negative (RecentlyDismissed −0.40 / LowSummaryConfidence −0.20) を含む。
   - `resolveRecallWeightSet(name)` で env var から選択 (KNOWLEDGE_HOME_RECALL_WEIGHT_SET)。デフォルトは v1_fixed (opt-in migration)。
   - 各 weight に `isNegative bool` を持たせ、UI が color-cue (terracotta border-left) に使う。
2. **Pure scorer の分離**:
   - `ScoreRecallSignals(signals, weightSet) → []recallScoreResult` を exported pure fn として新設。Item ごとに `(itemKey, score, reasons, breakdown, freshness, weightSetUsed)` を返す。
   - `scoreAndUpsertCandidates` を pure scorer 呼び出し + side-effect upsert に分離。
   - `time.Now()` を完全に除去し、freshness は `MAX(signals.OccurredAt)` 派生 (event-time 純度)。
3. **Negative signals の constant 追加 (受信側のみ)**:
   - `SignalRecentlyDismissed` / `SignalLowSummaryConfidence` を `recall_signal.go` に追加。
   - **emitter は別 PR**: dismiss usecase / recap projector が signal を emit する経路は future scope。scorer 側は absent signal を 0 contribution として扱う (絶対防御)。
4. **DB schema 拡張**:
   - migration `20260525000001_add_recall_weight_set_version.sql` で `recall_candidate_view` に 2 列 additive。
   - `weight_set_version TEXT NOT NULL DEFAULT 'v1_fixed'` + CHECK で値域を locking。
   - `score_breakdown JSONB NOT NULL DEFAULT '[]'` で全 row に non-null 保証。
5. **FE WhyTypography.scoreBreakdown UI**:
   - additive prop `scoreBreakdown?: ScoreBreakdownRow[]`。
   - expand toggle で `<ol>` 描画。`signal_code · w 0.00 · ±0.00` を monospace + 1 行/件で。
   - `isNegative` row は `--alt-terracotta` border-left 3px (color-only encoding を避けるためテキスト signed `+/-` も併記)。
   - UI text 英語 (`SCORE BREAKDOWN`)、Runes only、no `{@html}`、Newspaper-Style 維持。

### 検討した代替案

- **weight set を YAML / TOML で外部 file 化**: 採用しなかった。Heavy-Ranker (Twitter) は YAML だが、Alt は env var + map literal の方が deploy unit が単純。weight value を変える頻度は ADR mention レベル (= 月次以下)。
- **negative signals の emitter を本 PR で wiring**: 採用しなかった。dismiss usecase / recap projector の改修 scope が増え、また producer 経路は別 ADR (event semantics の確定が要る) で扱うべき。受信側を先に landing して signal vocabulary を proto / DB で固定する方が後続作業が並列化できる。
- **score_breakdown を JSONB ではなく separate table に**: 採用しなかった。breakdown は per-candidate の serialise 単位なので JSONB で十分。SQL joinable にする要件が出てきたら future migration で展開する。
- **`KNOWLEDGE_HOME_RECALL_WEIGHT_SET` の default を `v2_heavy_ranker` に**: 採用しなかった。production reproject なしで挙動が変わってしまう。opt-in 切替えで safety net を残す。

## Consequences

### Pros

- weight set version が wire / DB / runtime config の 3 層で一致する。A/B 実験や rollback が `KNOWLEDGE_HOME_RECALL_WEIGHT_SET` の値変更で済む。
- per-signal breakdown が JSONB に landing するので、UI / Grafana / Notebook 解析どこでも分解可能。
- pure scorer (`ScoreRecallSignals`) が公開され event-time 純度が test 可能になる (将来の reproject invariant test が書きやすい)。
- negative signal constants が proto / DB / domain に存在するため、emitter 側 PR が始まった瞬間から scorer が反映できる。
- FE は score_breakdown を rendering できるので「なぜこの article がここに?」の透明性が UI まで届く。

### Cons / Tradeoffs

- weight set map の duplication: v1 と v2 で 7 個の positive signal が大半同じ値。将来 weight tuning するときに片方更新を忘れる risk。今後 base + override 構造に refactor する余地はあるが本 PR では 2 map べた書きで十分。
- negative signal の emitter が landing するまで v2 weight set は positive のみで機能。これは「opt-in を v2 にしたら recall が薄くなる」期間が発生する (positive weight が同等なので結果は v1 と同じ)。
- `ScoreRecallSignals` を exported にしたので package external test / mock の surface area が増えた。pure fn なので side-effect リスクは無いが、API breaking change がしづらくなる。
- score_breakdown の JSONB は per-candidate 数 KB レベル。recall_candidate_view が大きくなるが Heavy-Ranker 想定の 5 候補/user では問題ない。
