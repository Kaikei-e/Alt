# ADR-000895: Inoreader 取り込みのサイレント停止を検出可能にする

## Status

Accepted

## Date

2026-05-09

## Affected Services

- pre-processor-sidecar (Go) — `RemoteTokenService` に `ErrTokenUnavailable` / `IsDegraded()` / `TokenAvailable()` を追加、`InoreaderService` に `LastSuccessfulFetch()` を追加、`handler.HealthHandler` で `/admin/health` を提供
- auth-token-manager (Deno) — 既存の `/api/token` 404 セマンティクス (`{"error":"No token data found"}`) を sidecar 側で typed に解釈する。auth-token-manager 自体は変更なし
- compose/db.yaml — clickhouse サービスに `logging` cap (json-file 10m × 3) と `mem_limit: 4g` を追加

## Context

PM-2026-042 (staging projector content-type loop, 2026-05-04 → 2026-05-07) のあと、本番ホストの disk が full に近付いた際に副作用として **`alt_oauth_token_data` ボリューム上の `oauth2_token.env` が空ファイル化** していた。これにより:

- auth-token-manager は `/api/token` で `HTTP 404 {"error":"No token data found"}` を 5 分間隔で出し続け、daemon ログには `"No valid refresh token found - waiting for user authorization"` だけが繰り返し記録された。
- pre-processor-sidecar は 16 分の fetch tick ごとに circuit breaker が `OPEN ↔ HALF_OPEN` を往復するだけで、人間にとって観測可能な「停止信号」がなかった。
- 結果、`sync_state.last_sync` が **2026-05-05T22:56:40Z で凍結したまま 3 日間気付かれなかった**。手動 `df` で disk 異常を見るまで Inoreader 取り込みが止まっていることに誰も気付けなかった。

事後の計測で次が判明した:

- 既存 `pre-processor-sidecar -health-check` は `token_manager_available: true` を返し続けていた。token ファイルが空でも 404 を返すだけの状態を「異常」と判定しない実装だった。
- `auth-token-manager returned status: 404` というエラー文字列はあらゆる呼び出し元に opaque な形で伝搬していた。テストで「token 消失」を分岐できず、観測手段としても脆弱だった。
- ClickHouse の compose service には `logging` 設定がなく (`docker inspect` で `LogConfig.Config={}`)、OOM-restart ループがそのまま log dir を肥大化させる構造になっていた。今回の incident でも RestartCount=85 を観測しており、disk 流出が再発する温床として残っていた。

これらは「サイレント失敗」を許す構造に共通する問題で、PM-2026-042 の D-1/M-1 (汎用 disk 圧迫アラーム) は別系統で残っているが、Inoreader 取り込みパス固有の観測 hook は本 ADR で閉じる。

## Decision

次の 4 点を採用する:

1. **auth-token-manager 404 を typed error 化する**
   `repository.RemoteTokenRepository.GetCurrentToken` で HTTP 404 を `ErrTokenNotFound` (既存 sentinel) に分類し、`service.RemoteTokenService.GetValidToken` で `service.ErrTokenUnavailable` (新規 sentinel) にラップする。呼び出し元は `errors.Is(err, service.ErrTokenUnavailable)` で「token 消失」を一意に判定できる。

2. **`RemoteTokenService` に degraded 信号を載せる**
   60 秒の rolling window に 3 回失敗を観測したら `IsDegraded()` が `true` を返す。1 回でも成功すれば失敗ウィンドウを完全にクリアする。`Clock` インターフェースを注入して unit test を deterministic にし、テストは「3 回連続失敗で flip」「60s 窓を超えた失敗は evict」「成功で復帰」の 3 経路を覆う。

3. **`/admin/health` エンドポイントを新設する**
   pre-processor-sidecar の既存 `:8080` admin mux に unauthenticated な GET `/admin/health` を追加する。レスポンスは `status` (`ok` / `degraded`)、`token_available`、`circuit_breaker_state`、`last_successful_fetch_at`、`seconds_since_last_fetch`、`ingestion_silent_threshold_seconds` (default 1800)、`ingestion_silent` を返す。閾値超過か `IsDegraded()` で `degraded` に flip する。
   `InoreaderService` は `FetchStreamContents` 成功時に mutex 配下で `lastSuccessfulFetch` を更新し、`CircuitBreakerState() string` で utils 型を字面化して handler を utils 依存から切り離す。`cmd/main.go` 側に小さな adapter (`ingestionHealthAdapter`) を置き、token provider が `*RemoteTokenService` のときだけ `TokenAvailable()` を呼ぶ (それ以外の token 実装は health に影響しない)。

4. **ClickHouse の disk 流出を構造的に塞ぐ**
   `compose/db.yaml` の clickhouse サービスに json-file driver の `max-size: 10m` / `max-file: 3` を pin し、`mem_limit` を 2g → 4g に上げる。migration replay (例: `010_alter_otel_traces_nested_events.sql` の `ALTER TABLE ... DROP COLUMN`) が OOM kill されるたびに log dir が肥大化する経路をここで閉じる。

### 検討した代替案と却下理由

- **`/admin/health` ではなく既存 `-health-check` を強化する**: 既存は CLI として exit code を返すだけで JSON を持たない。Prometheus blackbox / cron curl から polling するには HTTP endpoint が必要。CLI を拡張するより mux に載せる方がコストが低く、責務が明瞭。
- **token 状態を Prometheus metric として export する**: 既存の OTel exporter に metric を生やせば外形監視はできるが、metric の集約遅延 (≥ 30s) と RemoteAdminTokenManager に対する追加抽象が必要になる。今回は「次の token 消失で 30 分以内に degraded に flip すれば成功」というスケールなので、polling 可能な単一 endpoint で十分。
- **`TokenProvider` インターフェース自体に `TokenAvailable()` を足す**: 影響範囲が広く、既存 `SimpleTokenService` / `InMemoryTokenManager` も実装変更が必要になる。本 ADR では adapter で `*RemoteTokenService` への type-assert に留め、他の token 実装は touch しない。
- **ClickHouse を完全停止する**: rask-log-aggregator / Grafana datasource として現役。停止は退行になる。本 ADR では「壊れない範囲で」`logging` を pin して disk 流出だけ封じる。

### TDD での検証

実装前に下記の RED テストを書き、最小実装で GREEN にした (TDD ワークフロー [[tdd-workflow]] 準拠):

- `service/remote_token_service_test.go` (新規) — 5 ケース: 404 → `ErrTokenUnavailable`、3 連続失敗で `IsDegraded` 真、60s 外失敗の evict、成功で復帰、empty access_token も `ErrTokenUnavailable`
- `handler/admin_api_handler_health_test.go` (新規) — 4 ケース: ok 経路、stale fetch で degraded、token 消失で degraded、非 GET で 405

`go test ./...` 全パッケージ green を確認。

## Consequences

### Pros

- **次の token 消失は 30 分以内に観測可能**になる。`/admin/health` を polling すれば `ingestion_silent=true` または `token_available=false` で flip する。
- **テスト境界が改善した**。404 を typed sentinel で受けられるので、サイレント失敗パスを unit test レベルで分岐できる。Clock 注入で時間依存テストが deterministic になった。
- **ClickHouse 起因の disk 流出は構造的に閉じる**。json-file の rotation を pin したので、OOM-restart が再発しても log dir は 30MB に収まる。`mem_limit` 4g で migration replay の OOM 確率も下げた。
- **adapter 設計で他の token 実装に波及しない**。`SimpleTokenService` 経路は何も変わらない。

### Cons / Tradeoffs

- **`/admin/health` は unauth** のまま。既存の `/admin/trigger/*` と同じ前提で設計したが、auth 強化は別タスクで切り出す必要がある。インターネット直露出する経路ではないので即座のリスクではない。
- **degraded 判定は polling の上限** に縛られる。閾値 1800s (= 30 min) のレイテンシで「悪化を検知する」設計なので、リアルタイム alarm を期待してはいけない。瞬時検知が必要になったら Prometheus alerting rule を別途追加する。
- **PM-2026-042 の D-1/M-1 (host disk pressure / per-container log size 監視)** は本 ADR では扱わない。Inoreader 取り込みパス固有の観測のみを閉じた。汎用予防は別 PM フォローのスコープ。
- **adapter 内の type-assert** (`tokenProvider.(*service.RemoteTokenService)`) は遠回りで、将来 token 実装を増やすときに `TokenAvailable()` を共通インターフェースに昇格させる選択肢が残っている。今は scope を最小に保つために妥協した。
- **ClickHouse `mem_limit: 4g`** でも root cause の `DROP COLUMN` ミューテーションが完了する保証はない。完全な解消には別ウィンドウで mutation を kill して migration を完走させる作業が要る (本 ADR では disk 影響だけを閉じた)。
