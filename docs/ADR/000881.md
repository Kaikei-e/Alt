# ADR-000881: feeds ページ体感遅延と "Failed to emit article URL backfill" の解消

## Status

Accepted

## Date

2026-05-02

## Affected Services

- alt-backend (Go) — `fetch_article_usecase`, `connect/v2/feeds/feed_stats`, `knowledge_url_backfill_usecase`
- alt-frontend-sv (TypeScript / Svelte 5) — `hooks.server.ts`, `lib/hooks/useStreamingFeedStats.svelte.ts`, `routes/(app)/feeds/swipe/visual-preview/+page.server.ts`, `routes/api/v2/[...path]/+server.ts`

## Context

ユーザー報告は以下の 2 点。

1. **feeds ページ初回ナビ (リロード) で 10 回中 1-2 回、3-10 秒固まる** — 自宅 FTTH で発生する間欠 tail。
2. **モバイル Safari で `/feeds/swipe/visual-preview` 経由 `POST /api/v2/.../GetUnreadFeeds` が 500** — 29/Apr に 35 件集中、その後散発。

加えて運用面で **管理画面 "Emit article URL backfill" が "Failed to emit article URL backfill." で失敗**。

### 計測

| 指標 | 値 | 出典 |
|---|---|---|
| `/feeds` GET / `__data.json` (7d nginx tail max) | **0.118s / 0.165s** | nginx access |
| BFF `GetUnreadFeeds` / `GetAllFeeds` / `ListSubscriptions` (24h tail) | **>=500ms 0 件** | BFF proxy_response |
| Backend `feed_read_perf` (48h tail) | **>=500ms 0 件** | alt-backend |
| **BFF `FetchArticleContent` (24h tail)** | **19s / 18s / 16s / 13s / 12s / 11s / 10s / 9s ×5 / 8s / ...** (28 件 ≥1s) | BFF proxy_response |
| Backend article cache hit / miss (24h) | hit 24 / miss 97 = **miss rate 80%** | alt-backend |
| Backend external article fetch timeout / context deadline | **0 件** | alt-backend |
| StreamFeedStats 同時ストリーム数 (24h) | **22+ 本**、5 分 deadline → 即再接続。`feeds count` (106k 行) を 5 秒 polling × 3 種類 = 24h で 1363 回 | alt-backend |
| Lighthouse desktop (10Mbps / RTT 40ms / CPU x1) `/feeds` | **Performance 0.98 / FCP 0.5s / LCP 1.0s / TTI 1.0s / TBT 0ms** | curionoah.com-20260502T105419.json |

サーバ側の `/feeds` 自身は常に <200ms で速い。Lighthouse でも 1.0s LCP。にもかかわらずユーザーが「3-10s 固まる」と感じる正体は **`FetchArticleContent` の web-fetch tail**。

### 仮説の修正履歴

最初の 2 仮説は外れた。

- v1 ("mobile 限定 SSR チェーン") → デスクトップでも遅いとのユーザー指摘で却下。
- v2 ("`(app)/+layout.ts: ssr=false` + 1.8MB 単一バンドル") → Lighthouse 0.98 / 449KB / 92req のデータと矛盾し却下。1.8MB チャンクは visual-preview 等の別ルート用で `/feeds` には load されない。

v3 で確定した因果連鎖:

1. `fetch_article_usecase` は `articles` 表に保存済みコンテンツが **500 byte 以上** の場合だけキャッシュヒット扱い (`minFulltextContentLength = 500`)。
2. この 500 byte 基準は **Inoreader Tier1 取り込みの基準** を流用したもので、cache 判定基準としては不適切。
3. RSS 要約や短い HTML 本文は 500 byte を下回り、毎回 web fetch が走る (= 80% miss)。
4. 外部 fetch のタイムアウトは 25 秒。遅い origin (the guardian / hackernoon / dev.to / zenn cloudinary) は 5-19 秒で返るので tail を踏むと 19 秒待ち。
5. `/feeds/swipe/visual-preview` SSR は load 内で `fetchArticleContent` を直列 `await` するため、tail が SSR を支配する。

### iPhone 500 の正体

`/api/v2/[...path]/+server.ts` の catch ブロックが、backend への fetch 失敗時に固定 117 byte の JSON を返す:
```ts
return new Response(JSON.stringify({ code:"internal", message:"Failed to connect to backend" }),
  { status:500, ... });
```
nginx の "500 117" と完全一致。29/Apr のバーストは backend 一時不達 (deploy / mTLS 揺らぎ等) が原因で、現在は散発。**proxy 自体の挙動は正しい**。観測性側で「次に再発したら原因をすぐ特定できる」状態にする方針を採る。

### "Failed to emit article URL backfill"

alt-backend ログ: `"failed to emit article url backfill","error":"append corrective for 1aaa516b…: sovereign AppendKnowledgeEvent: canceled: context canceled"` が **30 秒経過後** に 1 件発生。BFF の `BFF_REQUEST_TIMEOUT=30s` が発火、knowledge-sovereign `AppendKnowledgeEvent` を途中でキャンセル。`max_articles=0` (無制限) の条件下では候補件数次第で 30 秒を超える。usecase 側はこのキャンセルを hard error にエスカレーションしていた。

## Decision

以下を採用する。

### 1. 外部記事 fetch のタイムアウトを 25 秒 → **8 秒**

`alt-backend/app/usecase/fetch_article_usecase/fetch_article_usecase.go` に `defaultExternalFetchTimeout = 8 * time.Second` を導入。`context.WithTimeout(ctx, u.externalFetchTimeout)` で external HTTP fetch を上限する。8 秒は BFF tail 観測 (5-19 秒) を踏まえ、ユーザーの「3-10 秒固まる」体感を 8 秒で頭打ちにする値。

### 2. cache 判定閾値を 500B → **100B** に変更し、Inoreader 基準混入を解消

定数 `minFulltextContentLength = 500` を `minCachedContentLength = 100` に置換。空応答や抽出失敗の極短コンテンツのみを cache miss 扱いに残し、RSS 要約由来の中間長 (200-499B) は cache hit として再 fetch を抑止。**500B は Inoreader Tier1 取り込み基準で、本来 pre-processor の上流ゲートに居るべき値であり read path に置くのは流用ミス**であった旨を comment に明記して再混入を防ぐ。

### 3. StreamFeedStats の COUNT を **30 秒 process-wide memoize**

`alt-backend/app/connect/v2/feeds/feed_stats.go` に `cachedFeedStats` + `sync.Mutex` の per-process キャッシュを追加。`fetchStatsCached(ctx)` が cache age < `statsCacheTTL` (30s) なら DB に当たらず即返却。同時 22 ストリームが 5 秒 polling しても DB ヒットは 30 秒に 1 度に集約され、`SELECT COUNT(*) FROM feeds (106k rows)` の負荷を約 100x 削減する。

### 4. FE `useStreamingFeedStats` を **module-scope singleton + refcount + visibility-aware** に書き換え

`alt-frontend-sv/src/lib/hooks/useStreamingFeedStats.svelte.ts` を:
- module スコープに `$state` を置き、subscriber refcount を維持。
- 1 → 2 などサブスクライバ追加では既存の AbortController を再利用。0 → 1 で初接続、1 → 0 で disconnect。
- `pagehide` / `visibilitychange === "hidden"` で suspend、`pageshow` (BFCache 復元) / `visibilitychange === "visible"` で resume。
- リトライを `MAX_RETRIES=5` / `MAX_RETRY_DELAY=30s` に緩和 (旧 10 / 10s)。

これでコンポーネント再マウントごとにストリームが増殖していた現象 (1 ユーザー = 22+ 同時ストリーム) を「1 タブ = 1 ストリーム」に閉じ込める。

### 5. `/feeds/swipe/visual-preview` SSR を **AbortController(6s) + 並列化**

`+page.server.ts` の load 関数で:
- 全体に `AbortController` + `setTimeout(6000)` を被せる。8s backend timeout と並べた 6s で client SSR の HTML を頭打ちにする。
- `feeds[0].ogImageProxyUrl` (feed list レスポンスに既に含まれる) を最優先で使い、`fetchArticleContent.ogImageProxyUrl` → 既存 `og_image_url` → `batchPrefetchImages` の順にフォールバック。`batchPrefetchImages` は最後の手段として並列起動し、失敗しても LCP image は確保される。
- `articleData` を SvelteKit 2 の deferred promise として返し、SSR HTML は即座に返す。

### 6. **handleError + 構造化エラーログ**

`hooks.server.ts` に `HandleServerError` を追加し、stderr へ JSON line を吐く。`/api/v2/[...path]/+server.ts` の catch も同形式の JSON line に書き換え。`Error.cause.code` (Node fetch の ECONNREFUSED / ETIMEDOUT 等) も narrowing して残す。これで iPhone 500 が再発した瞬間にログから原因を特定できる。

### 7. `knowledge_url_backfill_usecase` を **graceful ctx cancel** に

ループ内の `appendCorrective` および `ListBackfillArticles` 呼び出しで `errors.Is(err, context.Canceled)` / `context.DeadlineExceeded` および connect-go ラップ文字列 (`"canceled: context canceled"`) を検出した場合、hard error を返さず `MoreRemaining=true` の partial result を返す。各 `AppendKnowledgeEvent` は sovereign 側で別トランザクションで durable に確定しているため、ユーザーが再実行すれば dedupe registry で続きから処理される。BFF 30s timeout を踏んでも UI に「Failed」が出なくなる。

### 検討した代替案と却下理由

- **`(app)/+layout.ts: ssr=false` の解除 (v2 案)**: Lighthouse 0.98 が示すとおり CSR shell でも objectively 速い。SSR 化は TanStack Query / MediaQuery の hydration mismatch リスクが大きく、本問題の主因ではないため不採用。
- **`FetchUnreadFeedsListCursor` の N+1 サブクエリ → LATERAL JOIN 化 (Phase 6)**: 現状 `feed_read_perf usecase_ms` が 7-57ms で十分速いため緊急度低。Phase 1 / Phase 4 の効果観測後に再評価。
- **BFF の `BFF_REQUEST_TIMEOUT` 単純延長**: backend ハンドラのリソース保持時間が伸びるだけで、根本のキャンセル伝播の不適切な扱いが残る。usecase 側で graceful にする方が筋が通る。
- **画像プロキシ 502 のクロスカード重複防止**: 既存の per-card `onerror` が retry を抑止しており、tail 全体の影響は小さい。別タスクで切り出す。

## Consequences

### Pros

- ユーザー体感の「3-10 秒固まる」tail が 8 秒上限で頭打ち。実観測で 19 秒 → 9 秒 (p99) を期待。
- 80% キャッシュミス → 30% 以下 (RSS 要約由来の中間長を cache hit に拾うため)。
- StreamFeedStats 経由の `SELECT COUNT(*) FROM feeds` が 24h で 1363 回 → 約 50 回程度に減少 (~27x 削減)。
- 1 ユーザーあたりのストリーム数が 22+ → 1 (タブあたり)。alt-backend の goroutine 圧と pgbouncer 接続枠が解放される。
- iPhone 500 再発時の原因特定が即時可能 (構造化ログ)。
- 管理画面 "Emit article URL backfill" がユーザー操作で「失敗」表示されなくなり、`MoreRemaining` で再実行を案内できる。

### Cons / Tradeoffs

- 外部 origin が真に重い記事 (画像多用 / JS-heavy site) は cache miss を継続する。CSR 側で再取得 / 「読み込み失敗」UI に倒れる。許容できる頻度かは Phase 1 効果観測で判定。
- 100B cache 閾値は「extractor が真に空文字を返した場合」の判定が緩む可能性がある。`SaveArticle` 側で 100B 未満の保存は引き続き発生し得るが、再 fetch サイクルで上書きされる構造のため累積劣化は起きない。
- Module-scope `$state` シングルトンは Alt 慣例 (`docs/best_practices/svelte.md:245-283` の factory 関数 + getter/setter) からの逸脱。`(app)/+layout.ts` が `ssr=false` で genuine app-wide singleton 用途のため許容。SSR 化を将来検討する場合は context API + `setContext`/`getContext` への refactor が必要。
- 30 秒 memoize は projection ではなく resource table の row count を集約するキャッシュ。append-first / reproject-safe 不変条件には触れない (`docs/.claude/skills/immutable-design-guard/references/alt-invariants.md` の TTL キャッシュ除外条項)。
- Phase 6 (LATERAL JOIN) は deferred のため、極端な負荷時に N+1 サブクエリが顕在化するリスクは残る。
