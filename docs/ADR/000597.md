# ADR-000597: Ask Augur マルチターン品質を chat turns 分離と BM25 リランクで改善する

## Status

Accepted

## Date

2026-03-26

## Affected Services

- rag-orchestrator (Go)
- alt-frontend-sv (TypeScript / SvelteKit)

## Context

[[000595]] で Ask Augur のインラインチャットペインを実装し、[[000596]] で UTF-8 サニタイズと unescaped quote parser を修正した。しかし、実運用で以下の品質問題が残っていた:

1. **マルチターンで同じ回答が繰り返される**: フォローアップ質問に対して、1ターン目とほぼ同一の概要が生成される
2. **言語不一致でリランクが機能しない**: 日本語クエリ vs 英語記事チャンクで BM25 スコアが全て 0
3. **done イベント未到達で UI がハングする**: protobuf シリアライゼーション失敗時に `isLoading` が永久に true
4. **Fallback メッセージが不親切**: 固定英語メッセージで原因が伝わらない
5. **エラー時にリトライできない**: ユーザーが再質問するしかない

### マルチターン品質の根本原因

2 つの独立した問題が複合していた:

**問題 A: 毎ターン同じチャンクが返る**
`strategy_article_scoped.go` が記事の全チャンクを ordinal 順で返し、全て Score 1.0。フォローアップの質問内容に関わらず同じコンテキストが LLM に渡される。

**問題 B: 会話履歴が chat turns として構造化されていない**
`prompt_builder.go` が全内容を 1 つの `user` メッセージに詰め込んでいた。会話履歴はテキストとして埋め込まれるだけで、LLM（gemma3-4b）の chat API のターン構造（`<start_of_turn>user` / `<start_of_turn>model`）を活用していなかった。モデルは「会話の続き」ではなく「1 つの長い質問」として処理し、前回の回答を踏まえた新しい内容を生成する能力が著しく低かった。

## Decision

**4 層の改善を適用し、マルチターン品質を改善する。**

### 層 1: article-scoped BM25 リランク

フォローアップ時に記事チャンクをフォローアップ質問の BM25 スコアで再ランク付けする。

- `strategy_article_scoped.go` に `bm25RerankContexts()` を追加
- スコアを `0..1` に正規化（quality gate の閾値と互換）
- BM25 マッチがゼロの場合（言語不一致）は Score 1.0 にフォールバック

### 層 2: クエリ翻訳

日本語クエリを英語に翻訳してから BM25 マッチングを行う。

- `strategy_article_scoped.go` に `QueryExpander` を DI
- `isPrimarilyEnglish()` で言語判定、非英語なら `ExpandQueryWithHistory(query, history, 0, 1)` で英語翻訳
- 元のクエリ + 英訳を結合して BM25 に渡す（両言語でマッチ可能に）
- レイテンシ: 約 1 秒（4B モデルによるクエリ展開）

### 層 3: chat turns の分離（最重要）

multi-turn 時にプロンプトを実際の chat メッセージペアとして構築する。

- `prompt_builder.go` の `Build()` を `buildSingleTurn()` / `buildMultiTurn()` に分岐
- `buildMultiTurn()`: 過去ターンを user/assistant メッセージとして構築 + 最新 user メッセージにフォローアップ指示 + コンテキスト + クエリ
- Ollama `/api/chat` が Gemma 3 の `<start_of_turn>` トークンに自動マッピング
- 共通セクション（output format, context, article context）をヘルパーに抽出してリファクタ

### 層 4: フロントエンド UX 改善

- **60 秒タイムアウト**: `onComplete` 未到達時に自動リカバリー
- **Fallback メッセージ分岐**: `insufficient` → "Not enough indexed content for this article yet."
- **Retry ボタン**: エラー/fallback/timeout 時に最後のユーザーメッセージを再送信

## Consequences

### Pros

- マルチターンで LLM がネイティブに「会話の続き」として処理するため、前回の回答と異なる内容が生成される
- 日本語クエリ × 英語記事の組み合わせでも BM25 リランクが機能する
- done イベント未到達でも 60 秒で自動リカバリーし、ユーザーは Retry ボタンで再試行可能
- `prompt_builder.go` がリファクタされ、shared helper による保守性が向上

### Cons / Tradeoffs

- クエリ翻訳に約 1 秒のレイテンシが追加される（ユーザーに許容されると判断）
- multi-turn の過去ターンを 3000 文字で切り詰めるため、非常に長い回答は途中で切れる
- BM25 は語彙ベースのマッチングのため、意味的に関連するが語彙が異なるチャンクは拾えない（将来的に embedding ベースのリランクを検討）
- `buildMultiTurn()` で初回ターンのフルインストラクション（品質基準等）が省略されるため、モデルが品質基準を忘れるリスクがある
