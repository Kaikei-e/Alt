# ADR-000715: モバイル記事詳細画面を Alt-Paper editorial 表現にフル刷新する

## Status

Accepted

## Date

2026-04-14

## Affected Services

- alt-frontend-sv (SvelteKit 2.x / Svelte 5 Runes / TypeScript / TailwindCSS v4)

## Context

Knowledge Home (`/home`) のモバイル表示で `KnowledgeCard` の **Open** アクションを
押下すると、`/articles/[id]?url=…&title=…` へ遷移する。この記事詳細ページがモバイル
viewport（特に 320–480px）で以下のように崩れていた。

- ヘッダーが `flex items-center gap-4` に `Back to Home` + 右寄せで
  `Fetch Article` / `Summarize By AI` / `Open original` の 3 つを横並びに詰めており、
  ボタンラベルが長く折り返しもしないため viewport 幅を超えてオーバーフロー。
- コンテンツカードが `bg-white rounded-lg border` と Alt-Paper 違反
  （角丸・Tailwind 直書き）で、`docs/DESIGN_LANGUAGE.md` の「Sharp edges」原則に反する。
- 記事タイトル・出典（ソースドメイン）を提示する editorial masthead が存在せず、
  本画面の責務（＝「記事を読む」）が視覚表現に出ていない。
- コンテンツ HTML レンダラ `RenderFeedDetails.svelte` の `FeedContentOnTheFlyResponse`
  ブランチにも `px-4 py-4 rounded-lg` / `#2563eb` リンク / ダーク `pre` など
  Alt-Paper 外の装飾が残留しており、デザインシステムが貫徹していない。
- safe-area inset（ノッチ・ホームインジケータ）未考慮でコンテンツが画面端に切れる
  恐れがあった。

これら 1 件の機能崩壊と複数のデザイン逸脱が同じ画面に同居していたため、
バグ修正と Alt-Paper 化（Acolyte 同格の editorial treatment）を
同一タスクで完了させる方針を採った。

## Decision

`/articles/[id]/+page.svelte` と `RenderFeedDetails.svelte` の
on-the-fly コンテンツブランチを **モバイルファースト Alt-Paper editorial 構成**
に再設計する。ただし Acolyte 固有のパターン（2px 二本線マストヘッド）は責務不一致の
ため流用せず、`PageKicker` を共通プリミティブとして **共有デザイン言語**
（3 フォント体系 / sharp edges / thin rule / scoped style）のみを適用する。

具体方針:

1. **画面構成を 3 ブロックに分解**: sticky icon action bar / editorial masthead /
   AI summary + article body。
2. **アクション行はアイコン専用**（320px でも破綻しないよう label は `span.action-label`
   に括り、`@media (min-width: 640px)` 以上でのみ `display: inline`）。
   `aria-label` を必須化し、スクリーンリーダー可読性を担保。
3. **`PageKicker` を再利用**し、optional `testId` prop を追加する形で拡張。
   `kicker` にソースホスト名、`title` に記事タイトル（なければホスト名にフォールバック）
   を渡す。
4. **scoped `<style>` ベースの Alt-Paper 適用**。カラー・フォントは `var(--alt-charcoal)` /
   `var(--alt-slate)` / `var(--surface-bg)` / `var(--surface-2)` / `var(--alt-primary)` /
   `var(--font-display)` / `var(--font-body)` / `var(--font-mono)` を使用し、
   Tailwind 任意値（`rounded-lg` / `bg-white` / `text-[22px]` 等）は全撤去。
5. **safe-area 対応**: sticky action bar の上パディングに
   `calc(env(safe-area-inset-top, 0px) + 0.5rem)`。
6. **AI Summary ブロック**は `font-mono uppercase tracking-[0.14em]` の kicker
   "AI SUMMARY"、本文は Source Sans 3 1rem/1.75、左端 3px 縦ルールで原典と分離。
7. **`RenderFeedDetails` の on-the-fly 枝のみ**を Alt-Paper 化。
   `matched_articles` 枝は本画面から到達しないため今回は触らず、他画面への副作用を避ける。
8. **読み幅は `max-width: 65ch`** で editorial な可読性を担保。

TDD フロー（RED → GREEN → REFACTOR）を厳守し、期待値を先にテストで固定した後に実装を
書き換える。ラベル文言は機能語（"Fetch" / "Summarize" / "Re-fetch" / "Open original"）
のみ。新聞メタファー語彙（"PRESS RUN" / "EDITION" 等）は `feedback_no_newspaper_metaphor_in_ui_text`
の指針に従い使用しない。

## Consequences

### Pros

- **モバイル 320–480px でオーバーフローしない**。アクションアイコンは 36px square、
  ラベルはタブレット以上で復帰する responsive enhancement。
- 共有デザイン言語が記事詳細にも貫徹され、Acolyte / Tag Verse / Pulse と
  **同格の editorial treatment** を獲得した。Alt-Paper が「共有言語」
  として機能していることを画面間一貫性で担保できる。
- `PageKicker` の `testId` プロパティ追加により、再利用プリミティブを壊さず
  画面別の自動テストフックが張れるようになった。
- `rounded-lg` / `bg-white` / ハードコードカラーの混入を vitest-browser-svelte
  の compliance テスト（`.rounded-lg, .rounded-xl, .rounded-2xl, .bg-white`
  を 0 件で検証）で継続的にガードできる。
- safe-area 対応により iOS ノッチ機や Android ホームインジケータ機での
  タップ領域干渉を回避。
- リンクカラーが `var(--alt-primary)` (#2f4f4f ダークティール) に統一され、
  Alt-Paper の WCAG AA コントラスト閾値を満たす。

### Cons / Tradeoffs

- モバイルではボタンがアイコンのみになるため、新規ユーザーの learnability が下がる。
  `aria-label` による読み上げはあるが、視覚的には初見で意味が取りにくい。
  → 将来的に 320px 突破後のラベル表示や long-press tooltip の導入余地を残す。
- `RenderFeedDetails` の `matched_articles` 枝を今回触らない判断により、
  同ファイル内に Alt-Paper 準拠 / 未準拠の 2 系統の CSS が混在する状態が残った。
  別画面で使用が確認されるか、または dead code と判断できた時点で統廃合する。
- `PageKicker` への `testId` prop 追加は小さな API 拡張であり、他利用箇所
  （recap job-status 系）は `testId` を渡さないため互換性は保たれるが、
  テンプレート属性 `data-testid={undefined}` が DOM に `data-testid=""` として
  現れる挙動差に注意が必要（Svelte 5 は undefined 属性を出力しないためセーフ）。
- scoped style がページファイル内に増大（約 190 行）。
  共通の Alt-Paper editorial wrapper を抽出する余地があるが、
  現時点では過剰抽象化を避け、同一画面内に留めた。
