# ADR-000463: Knowledge Home Recall パイプラインの signal 生成と projector 接続を復旧する

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-backend (Go)

## Context

Knowledge Home の RecallRail が常に「Nothing to recall right now.」を表示し、recall candidate が一切生成されない状態にあった。

調査の結果、**UI の empty state 表示はバグではない**（[[000431]] の設計で候補 0 件でも常時描画する仕様）。問題は **データ生成パイプラインの断線** であり、以下 3 箇所が未接続だった。

| # | 断線箇所 | 状態 |
|---|---------|------|
| 1 | `TrackHomeActionUsecase` が `recall_signals` テーブルに書かない | `knowledge_events` には書くが signal は未生成 |
| 2 | `RecallProjectorJob.processRecallSignals()` がスタブ | ログ出力のみで `return nil` |
| 3 | `RecallProjectorJob` が scheduler に未登録 | `main.go` に登録なし |

既存の Knowledge Projector は `projectHomeItemOpened()` で `recall_candidate_view` に候補を直接書くが、`next_suggest_at = now+24h` 設定のため即時表示されない。さらに `ScoreRecallCandidates()` による多信号スコアリング（augur 参照、タグ関心等）が signal projector 経由でしか発火しないため、open 以外の経路の候補生成が完全に死んでいた。

### データフロー（復旧前 → 復旧後）

```
復旧前:
  TrackHomeAction → knowledge_events ✅ / recall_signals ❌
  RecallProjectorJob → 未登録 / processRecallSignals() はスタブ
  recall_candidate_view → 空

復旧後:
  TrackHomeAction → knowledge_events ✅ / recall_signals ✅
  RecallProjectorJob (60s) → ListRecallSignalsByUser → ScoreRecallCandidates()
  recall_candidate_view → 候補あり
```

## Decision

**最小復旧として、signal 生成・projector 実装・job 登録の 3 点を接続する。**

Phase 4 計画が想定する全 signal source（search / recap / pulse）の接続は後続タスクとし、本 ADR では `open` / `ask` / `listen` の 3 action type のみを対象とする。

### 1. TrackHomeActionUsecase に Recall Signal 生成を追加

`TrackHomeActionUsecase` に `recall_signal_port.AppendRecallSignalPort` を optional setter パターンで注入。`Execute()` 末尾で action type に応じた signal を `recall_signals` に append する。

| action type | signal type | 即時候補化 |
|------------|-------------|-----------|
| `open` | `SignalOpened` | 48h 以上再訪なしで候補化 |
| `ask` | `SignalAugurReferenced` | 即時（score 0.35 ≥ minRecallScore 0.2） |
| `listen` | `SignalTagInterest` | 候補化しない（score 0.15 < minRecallScore 0.2。他 signal との合算で候補化） |

Signal 書き込みは **non-fatal**。失敗しても usecase 全体は成功を返し、`slog.ErrorContext` でエラーを出力する。

### 2. Recall Projector のスタブを完成

`processRecallSignals()` を実装し、config の `AllowedUserIDs` から user ID を取得して各ユーザーの signal を走査する。

`AllowedUserIDs` は feature rollout 用設定であり projector の正規走査源としては弱いが、シングルテナント環境では十分と判断した。後続タスクで `ListDistinctUserIDs` port を追加予定。

各ユーザーについて `ListRecallSignalsByUser(ctx, userID, 7)` で直近 7 日分の signal を取得し、既存の `ScoreRecallCandidates()` を呼ぶ。ScoreRecallCandidates は signal type 別の重み付きスコアリングを行い、`minRecallScore` (0.2) 以上の候補を `recall_candidate_view` に upsert する。

### 3. main.go に job 登録

```go
scheduler.Add(job.Job{
    Name:     "recall-projector",
    Interval: 60 * time.Second,
    Timeout:  30 * time.Second,
    Fn:       job.RecallProjectorJob(recallUserIDs, ...),
})
```

Interval は [[000408]] に合わせて **60 秒**。RecallRail はリアルタイム性を要求しない（候補は最短でも数時間〜24h 後に eligible になる）ため、低頻度で十分。

### 設計判断: 24h delay は維持

Knowledge Projector の `projectHomeItemOpened()` が作る candidate の `next_suggest_at = now+24h` は設計意図通り（Recall は "memory aid"）。Signal projector は別経路で `SignalAugurReferenced` 等の即時 eligible な候補を生成するため、open のみの 24h delay はそのまま維持する。

### DI 配線

`di/container.go` で既存の `recallSignalGw` を `trackHomeActionUsecase.SetRecallSignalPort()` で配線。`registerFeedsUsecase.SetSubscriptionPort()` と同じ optional setter パターンに従う。

## Consequences

### Pros

- `ask` action から `SignalAugurReferenced` が生成され、次回 projector 実行時（最大 60 秒後）に recall candidate が即時 eligible で書き込まれる。RecallRail に表示が始まる
- `open` action の signal projector 経路も有効になり、48h 以上再訪なしの記事が自動的に recall 候補に浮上する
- 既存の `ScoreRecallCandidates()` をそのまま再利用しているため、新規ロジックの導入が最小限
- Signal 書き込みは non-fatal のため、recall_signals テーブルに障害が起きても Knowledge Home の基本機能（tracking / projection）に影響しない
- 構造化ログ（`user_count`, `signals_processed`）により projector の動作が観測可能

### Cons / Tradeoffs

- **`AllowedUserIDs` 依存**: 現在の projector はconfig の `AllowedUserIDs` に列挙されたユーザーしか走査しない。マルチテナント化時に `ListDistinctUserIDs` port への移行が必要
- **`listen` → `SignalTagInterest` は暫定マッピング**: Phase 4 設計名 `home_listened_tts` とは異なる。後続の signal source 全体設計で再整理が必要
- **search / recap / pulse signal は未接続**: 本 ADR のスコープ外。これらの経路が接続されるまで、recall candidate の多様性は限定的
- **48h delay は残る**: `SignalOpened` 起点の候補は 48h 経過しないと生成されない。テスト・検証時は `ask` action（即時 eligible）を使う必要がある
