# ADR-000706: Morning Letter の出力を編集的に意味のある形へ補正する (Phase 2)

## Status

Accepted

## Date

2026-04-13

## Affected Services

- recap-worker (Rust) — `MorningPipeline`, DAO `save_morning_letter`
- news-creator (Python) — `MorningLetterUsecase` の例外ハンドリング
- alt-frontend-sv (Svelte 5) — Morning Letter のフッター / セクションタイトル描画

## Context

ADR-000705 で Morning Letter の再定義と再稼働を行った結果、`POST
/v1/morning/letters/regenerate` は 200 を返すようになったが、実際に UI に
出てきたのは以下のような **形だけ整って中身の無い紙面** だった。

```
2026-4-13 · Asia/Tokyo · Partial edition
Partial briefing: 257 overnight threads.
Based on 7-day recap
Today's Headlines   — ETL vs ELT: Which One Should You Use and Why?
Need to Know        — AI Diagnoses Knees, Drugs, Configs, …
Need to Know        — How Do Smart Devices Perform Multiple Tasks?
Model: extractive-fallback
```

ユーザーの反応は **「Model: extractive-fallback とだけ UI に表示されている
がこれに何の価値が？」** で、率直に言ってその通りだった。具体的に 4 つの
壊れ方が重なっていた。

1. **LLM パスが無言で死んでいた**
   `news-creator/app/news_creator/usecase/morning_letter_usecase.py:83` に
   `except Exception` の「なんでも受け」があり、タイムアウトか JSON
   パース失敗か Pydantic バリデーション失敗かを識別せずに extractive
   fallback に落ちていた。ログは "LLM generation failed" の一行だけで、
   運用から根本原因に手が届かなかった。
2. **through-line が数字でしかなかった**
   `build_through_line(overnight_count, recap_count, is_degraded, window_days)`
   は "Partial briefing: 257 overnight threads." を返していた。事実は
   正しいが編集的には空疎で、読者に「今日何が起きたか」を語っていない。
3. **UI が内部失敗を漏らしていた**
   `MorningLetterDocumentCore.svelte` のフッターが `Model: {letter.model}`
   をそのまま出していて、`extractive-fallback` という内部識別子が読者の
   目に入っていた。また LLM が多用する汎用タイトル "Need to Know" が
   セクションごとに繰り返し現れ、紙面が単調になっていた。
4. **regenerate を重ねると落ちる FK バグ**
   `morning_letters` の `ON CONFLICT DO UPDATE` は DB 上の id を維持するが、
   recap-worker のメモリ上では毎回 `Uuid::new_v4()` で新しい `letter_id` を
   生成していた。その結果 2 回目以降の regenerate で
   `morning_letter_sources` の外部キーが壊れ、"failed to insert morning
   letter source" で 500 を返していた。

## Decision

ADR-000705 が約束した「bullets は決定論、LLM は散文だけ」の姿を段階的に
近づけるべく、まず **編集価値をすぐに出せる 4 点** を Phase 2 として確定
させる。

1. 典型的に無価値な LLM タイトル（"Need to Know"、"Today's Headlines" …）
   を UI レイヤで shadow し、マップ済み/ジャンル由来のタイトルを優先する。
2. `build_through_line` を、カウント表現から **編集リード文** に書き直す。
   セクションラベルが 2 つ以上あればそれを接続し、無ければ genre で語り、
   最後のフォールバックだけ数値表現にする。`"Partial briefing:"` は廃止
   （degraded は UI バッジが担当）。
3. フッターから `Model:` 表示を撤去。内部識別子は `aria-label` に留め、
   読者には `"A Morning Letter from your Alt"` または `"Revised N times
   today"` を見せる。
4. news-creator の `except Exception` を **型別 catch** に分解し、
   `error_type` / `error_detail` / `response_head` を構造化ログに出す。
5. `save_morning_letter` を `RETURNING id` にし、conflict 時に DB 上の
   真正な id を返すようにして、`morning_letter_sources` の FK バグを
   恒久的に解消する。併せて `(letter_id, section_key, article_id)` の
   重複は挿入前に de-dup する。

クラスタリングベースの編集プレーン刷新（subworker clustering →
ranking → scaffold contract → news-creator プロンプトの散文化）は
**Phase 2.5 として別 ADR** に切り出し、本稿では手を出さない。今回は
「何の価値が？」と言われた読者体験を即座に 1 段階マシにする最短手を
取る。

## 実装の概要

### alt-frontend-sv

- `src/lib/components/morning-letter/MorningLetterDocumentCore.svelte`
  フッターを書き換え:
  - `<span>Model: {letter.model}</span>` を削除。
  - `<footer aria-label="Morning Letter generated by {model}">` に
    移し、生成モデルは開発者向け属性に閉じ込める。
  - 画面上の文言は `generation_revision > 1` のときに `"Revised N
    times today"`、それ以外は `"A Morning Letter from your Alt"`。
- `src/lib/components/morning-letter/morning-letter-document.ts`
  `getSectionDisplayTitle` の優先順位を再定義:
  1. `SECTION_DISPLAY_TITLES` のマップ (`top3` → "Top Stories" 等)
  2. `by_genre:<slug>` / `by_theme:<slug>` 由来のタイトル
  3. `section.title` — ただし `GENERIC_TITLE_BLOCKLIST`
     ("need to know"、"today's headlines"、"top stories"、…) に合致する
     場合は無視。
  4. フォールバックで `section.key` をそのまま。
- vitest: `morning-letter-document.test.ts` に 4 ケース追加
  (generic shadow、theme-slug parsing、非 generic LLM タイトルの保持、
  by_theme: の大文字化)。全 25 件パス。

### recap-worker

- `src/pipeline/morning.rs`
  - `build_through_line` を純関数に作り直し:
    - 引数 `section_labels: &[&str]`, `dominant_genres: &[&str]`,
      `overnight_count: usize`。
    - ラベル ≥2 → `"<A> and <B> led overnight — N new threads surfaced."`
    - ラベル 1 → `"<A> dominated overnight — N new threads surfaced."`
    - ラベル無し ∧ genres ≥1 → `"<X> and <Y> topics dominated overnight …"`
    - 全部無し → `"Overnight brought N new threads across your feeds."`
    - `overnight_count=0` のみで `"A quiet day across your feeds …"`。
  - `editorial_labels_for` / `dominant_genres_for` ヘルパーを追加し、
    section.title が generic な場合はラベル候補から除外。
  - 呼び出し側 (`persist_morning_letter`) を新シグネチャに合わせて
    更新。`is_degraded` は persisted metadata に残すが through-line
    の prefix からは落とした。
- `src/store/dao/morning.rs`
  - `save_morning_letter` を `INSERT … ON CONFLICT … RETURNING id` に
    変更し、戻り値を `Result<Uuid>` に変更。conflict 時に実際の DB 上
    の id を返す。
- `src/store/dao/traits/morning.rs` / `impls/unified.rs` /
  `compat.rs` / `mock.rs` のシグネチャを `Result<Uuid>` に更新。
- `src/pipeline/morning.rs:persist_morning_letter`:
  - 返ってきた `effective_letter_id` を FK 用に使う。
  - `morning_letter_sources` 挿入前に `article_id` の HashSet で
    de-dup。`(letter_id, section_key, article_id)` の unique 制約を
    回避。
- テスト: `through_line_*` を 6 ケースに更新 (two_labels, one_label,
  genres_only, overnight_count_only, quiet_day, never_says_partial,
  is_deterministic)。lib 全 308 件パス。

### news-creator

- `app/news_creator/usecase/morning_letter_usecase.py`
  - `import asyncio` / `pydantic.ValidationError` を追加。
  - `generate_letter` の blanket `except Exception` を以下の型別
    catch に分解:
    - `asyncio.TimeoutError` → `error_type="timeout"`
    - `json.JSONDecodeError` → `error_type="json_decode"` +
      `response_head` に先頭 200 文字を同梱。
    - `pydantic.ValidationError` → `error_type="pydantic_validation"`
      + 先頭 3 件のエラー内容。
    - `RuntimeError` → `error_type="parse_runtime"`
      (`_parse_content` の shape エラー)
    - その他 → `error_type="unexpected"` セーフネット。
  - `_fallback_with_reason(request, error_type, error_detail,
    response_head)` ヘルパーを追加。構造化ログに
    `target_date`, `edition_timezone`, `recap_summary_count`,
    `overnight_group_count`, `error_type`, `error_detail` を出す。
  - `degradation_reason` は `"{error_type}: {error_detail[:180]}"`
    の書式で persist され、API の metadata に載る。
- テスト: `test_llm_timeout_is_tagged_in_degradation_reason` と
  `test_llm_validation_error_is_tagged_in_degradation_reason` を追加。
  suite 全 498 件パス。

### スモーク (2026-04-13)

- `docker compose up --build -d alt-frontend-sv news-creator recap-worker`
- `curl -X POST /v1/morning/letters/regenerate`:
  - 旧: 500 (`failed to insert morning letter source`) が 2 回目以降発生。
  - 新: 200、`generation_revision` が 3 まで進んだうえで安定。
- 返された Letter の `body.through_line`:
  `"Overnight brought 257 new threads across your feeds."`
  (旧 `"Partial briefing: 257 overnight threads."` から置き換え完了)。
- `body.sections[*].title` に `"Today's Headlines"` / `"Need to Know"`
  等が来ても、FE の `getSectionDisplayTitle` が `by_genre:` / マップ
  済みの `"Top Stories"` 等で上書きするため画面には出ない。
- `metadata.model` は依然として `extractive-fallback` を含むが、
  フッターには公開されない (DOM 上は `aria-label` のみに残る)。

## Consequences

### Pros

- 読者の目に触れる "Partial briefing: 257 overnight threads" や "Model:
  extractive-fallback" が一掃され、**内部失敗の漏洩がゼロ** になった。
- through-line が "Overnight brought N new threads …" の一文で読み物に
  なり、編集物としての tone を保てるようになった。ラベルが揃えば
  "X and Y led overnight — …" と接続が効く。
- news-creator の失敗モードがログに **型として記録される** ため、次に
  落ちるときは `error_type` で grep すれば原因が即特定できる。
- `save_morning_letter` が DB 真正の id を返すので、再生成を繰り返しても
  `morning_letter_sources` の FK が破綻しない。既存データへの migration
  は不要 (挙動の恒久修正のみ)。

### Cons / Tradeoffs

- 依然として LLM がほぼ常時 extractive fallback に落ちている状態は
  残る。Phase 2 は「ダメな中身をダメとわかるように出さない」までで、
  **中身そのものを良くする** のは Phase 2.5 に委ねた。"Top Stories"
  セクションが 1 つ、中身は overnight 3 本という構造自体は変わらない。
- `GENERIC_TITLE_BLOCKLIST` は英語ハードコード。将来 i18n を入れる
  ときに見直しが要る。
- 型別 catch に落ちた news-creator の `except Exception` はまだ最後の
  セーフネットとして残している。完全に除去するには内部ユーティリティの
  例外列挙が要る (別スコープ)。
- through-line は section labels / genres を引数に取るが、この入力は
  「LLM が section.title と genre を正しくつけているか」に依存する。
  Phase 2.5 で projector 側から決定論的に投げ込む必要がある。

## Phase 2.5 (deferred, 別 ADR 予定)

- `SubworkerClient::cluster_corpus_with_timeout` を `MorningPipeline` に
  組み込み、overnight 257 本を 3–15 の意味クラスタに落とす。
- `score_cluster_for_role` (pulse v4) でランク付け、top 3–5 を Morning
  セクションに採用。
- news-creator の `MorningLetterGenerateRequest` に
  `sections_scaffold` を追加し、**LLM は lead と narrative のみ** を書く
  (bullets / why_reasons はスキャフォールドから pass-through)。
- 残りは `by_theme:<cluster_slug>` セクションとして 1 本にまとめ、
  "Other threads" として expose。
- `WhyRecentInterestMatch` 向けの alt-backend `GetRecallCandidates`
  HTTP client を recap-worker に生やすかは Phase 3 にさらに切り出す。
