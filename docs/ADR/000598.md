# ADR-000598: SwapReproject 時にプロジェクターチェックポイントをリセットしてギャップを防止する

## Status

Accepted

## Date

2026-03-27

## Affected Services

- alt-backend (Go) — `knowledge_reproject_usecase`, KnowledgeProjector
- knowledge-sovereign (Go) — `knowledge_projection_checkpoints` テーブル

## Context

Knowledge Home で特定の記事が "Summarizing" のまま更新されない問題が報告された。調査の結果、V3 Reproject の swap 時に KnowledgeProjector のチェックポイントがリセットされなかったことによる**チェックポイントギャップ**が根本原因であることが判明した。

### 発生メカニズム

1. V3 reproject がイベント seq 0 → 1116081 を version 3 に投影して完了
2. LIVE projector は version 1 で稼働継続し、チェックポイントは 1116081 をはるかに超えて進行
3. `SwapReproject()` が `ActivateVersion(3)` を呼び出して version 3 をアクティブ化
4. **しかしチェックポイントはリセットされなかった**
5. LIVE projector が version 3 に切り替わった時点で、チェックポイントは既に reproject のカバー範囲を超えていたため、seq 1116082 以降のイベントが version 3 に投影されなかった

### 影響

- 326 件の記事が Knowledge Home から完全欠落
- 1,947 件の記事で `summary_state` が stale（v1=ready なのに v3=pending）
- 約 4 日間検知されなかった

### 構造的な原因

`knowledge_projection_checkpoints` テーブルが `projector_name` のみを PK とするバージョン非依存設計であるため、reproject と LIVE projector のチェックポイントが独立管理されず、swap 時にギャップが発生するリスクが構造的に存在していた。

## Decision

`SwapReproject()` で projection version を切り替える際に、reproject の最終 `event_seq` でプロジェクターチェックポイントをリセットする。

### 実装

1. **`Usecase` に `UpdateProjectionCheckpointPort` を追加**: `WithUpdateCheckpointPort()` セッターメソッドで注入
2. **`SwapReproject()` に checkpoint リセットロジックを追加**: `ActivateVersion()` の直後に、reproject run の `checkpoint_payload.last_event_seq` を抽出して `UpdateProjectionCheckpoint()` を呼び出す
3. **`extractCheckpointSeq()` ヘルパー関数**: `checkpoint_payload` JSON から `last_event_seq` をパース。payload が空または不正な場合は 0 を返し、リセットをスキップ

### 変更ファイル

- `alt-backend/app/usecase/knowledge_reproject_usecase/usecase.go` — `SwapReproject()` にリセットロジック追加、`WithUpdateCheckpointPort()` セッター追加
- `alt-backend/app/usecase/knowledge_reproject_usecase/usecase_test.go` — テスト 2 件追加（チェックポイントリセット、空 payload のフォールバック）
- `alt-backend/app/di/knowledge_module.go` — DI 配線（`WithUpdateCheckpointPort(sovereignCli)`）

### 即時復旧

コード修正に先行して、`knowledge_projection_checkpoints.last_event_seq` を手動で 1116081 にリセットし、projector にギャップを再処理させた。UPSERT の merge-safe ロジックにより既存データは安全に上書きされた。

## Consequences

### Pros

- reproject swap 時のチェックポイントギャップが構造的に防止される
- 既存の `UpdateProjectionCheckpoint` RPC を再利用し、proto 変更不要
- `checkpoint_payload` が空の場合はリセットをスキップし、後方互換性を維持

### Cons / Tradeoffs

- チェックポイントテーブル自体はバージョン非依存のまま。将来的に `projection_version` カラム追加を検討する余地がある
- リセット後の再投影区間では projector lag が一時的に増加する
