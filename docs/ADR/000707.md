# ADR-000707: Morning Letter を Alt の知識グラフへの launch pad に変える

## Status

Accepted

## Date

2026-04-13

## Affected Services

- alt-backend (Go) — 新 RPC `GetLetterEnrichment`, enrichment usecase,
  feed title batch driver, DI 更新
- alt-frontend-sv (Svelte 5) — `ArticleDeck` / `ArticleDeckCard` 新設、
  `useMorningLetter` に enrichment フェッチ追加
- recap-worker (Rust) — 3-day recap に切り替え (`get_latest_completed_job(3)`)

## Context

ADR-000706 で UI の漏洩と through-line を直した結果、読者に出る Letter は

```
2026-04-13 · Asia/Tokyo
Overnight brought 257 new threads across your feeds.

=== Top Stories ===
  · WindowsでLinux環境を簡単に構築する — WSLのインストール方法
  · Claude 3 Haiku Is Being Deprecated April 19 …
  · SSL/TLS Certificates Explained: HTTPS Security for Every Website

A Morning Letter from your Alt
```

ユーザーからの追い打ちは **「リンクもなければ、関連する Alt が持っている
情報もほとんどない。これでは、ここから発展しない」** だった。まさに
そのとおりで、Letter は読者にとって行き止まりの文字列の束でしかなく、
Alt という RSS + 知識グラフ基盤の接続点として機能していなかった。

同時に recap の window は **7 日では長すぎて古いイベントを持ち込んで
いた**。現状運用では **3 日** が妥当、と明言があった。

## Decision

1. Morning Letter の各ソース記事を、**読む・追う・深掘るの 3 つの入り口**
   を備えた **launch pad カード** として UI に展示する。
2. エンリッチメント (記事リンク、タグ、関連記事、Acolyte チャット導線、
   要約 excerpt、フィード名) は **BFF レンダリング時** に alt-backend が
   合成する。recap-worker の projector は触らない (reproject-safe の維持)。
3. 新 Connect RPC
   `MorningLetterReadService.GetLetterEnrichment(letter_id)` を生やし、
   FE は Letter 読込直後に一度だけ呼ぶ。enrichment の不在はエラーでは
   なく「まだ Alt 側に情報が無い」を意味し、UI は素直に degrade する。
4. 再生成のベース recap window を **3 日** (`get_latest_completed_job(3)`)
   に切り替える。Letter 本文中の "Based on N-day recap" 表示も自動的に
   `source_recap_window_days=3` で揃う。

カノニカルコントラクトとの整合: Knowledge Home の正規ブリーフ (Phase 2.5
探索時に確認) が言う

- **reproject-safe**: projector は純関数
- **why-code taxonomy は versioned、任意文字列ではない**
- **article → URL は projector に含めず、FE/BFF の質問時解決に任せる**

の 3 点を全て満たしている。v3 は `morning_letters` / `morning_letter_sources`
のスキーマも proto の既存 message も変えない（追加のみ）。

## 実装の概要

### Proto (`proto/alt/morning_letter/v2/morning_letter.proto`)

```
rpc GetLetterEnrichment(GetLetterEnrichmentRequest)
    returns (GetLetterEnrichmentResponse);

message MorningLetterBulletEnrichment {
  string section_key = 1;
  string article_id = 2;
  string article_title = 3;
  string article_url = 4;
  string article_alt_href = 5;   // "/articles/<uuid>"
  string feed_title = 6;
  repeated string tags = 7;
  repeated RelatedArticleTeaser related_articles = 8;
  string summary_excerpt = 9;
  string acolyte_href = 10;      // "/acolyte/new?article_id=&topic="
}

message RelatedArticleTeaser {
  string article_id = 1;
  string title = 2;
  string article_alt_href = 3;
  string feed_title = 4;
}
```

すべて optional (空文字 / 空配列) を許す形。Alt 側に情報が無い記事は
field が空のまま返り、UI は該当チップを単に描画しない。

### alt-backend

Clean Architecture を維持。既存プリミティブを最大限再利用した:

- **domain**: 新しく `MorningLetterBulletEnrichment` と
  `RelatedArticleTeaser` を `app/domain/morning_enrichment.go` に追加。
  既存 `domain.WhyReason` 等は触らない。
- **port** (`app/port/morning_letter_port/morning.go`):
  - `MorningLetterUsecase.GetLetterEnrichment(ctx, letterID, userID)` を追加。
  - 新 narrow ports:
    - `ArticleMetadataBatchPort` — 既存 `AltDBRepository.FetchArticlesByIDs`
      をそのまま満たす。
    - `FeedTitleBatchPort` — 新 driver
      `FetchFeedTitlesByIDs(ctx, []uuid.UUID) map[uuid.UUID]string` を
      `app/driver/alt_db/fetch_feed_titles_by_ids_driver.go` に追加し、
      `AltDBRepository` のメソッド昇格で満たす。
    - `SearchRelatedArticlesPort` — 既存 `SearchIndexerPort` の
      `SearchArticles` サブセット。
- **usecase**
  (`app/usecase/morning_usecase/morning_letter_enrichment_usecase.go`):
  1. `GetLetterSources` で sources を購読フィルタ済みで取得
  2. 上位 8 件までに cap (`enrichmentMaxSources=8`) — 257 件 fan-out を
     避ける
  3. `FetchArticlesByIDs` で URL + title + tags + feed_id を一括取得
  4. `FetchFeedTitlesByIDs` で feed_id → title をバッチ解決
  5. `sync.WaitGroup` + semaphore (`concurrency=4`) で search-indexer に
     関連記事を並列問合せ。seed 記事自身は除外、上位 3 件を teaser に。
  6. Acolyte 導線 URL `/acolyte/new?article_id=<uuid>&topic=<title>` を
     deterministic に構築。
  7. 本文 `Article.Summary` を優先し、無ければ `Article.Content` 先頭を
     `summaryExcerptChars=220` に整形 (連続空白を圧縮)。
- **handler**
  (`app/connect/v2/morning_letter/handler.go`): RPC メソッド追加。
  authentication 必須、空 `letter_id` は `InvalidArgument`。
- **DI**: `NewMorningLetterUsecaseWithEnrichment` を新設し
  (`app/usecase/morning_usecase/morning_letter_usecase.go`) existing
  constructor は互換のため残す。`di/rag_module.go` で
  `infra.AltDBRepository` (ArticleBatch / FeedTitleBatch を兼ねる) と
  `infra.SearchIndexerDriver` を注入。
- **tests**: usecase 2 ケース追加
  (`TestGetLetterEnrichment_BuildsCardsWithAltHrefAndAcolyteLink`,
  `TestGetLetterEnrichment_GracefullyHandlesMissingArticle`)。
  記事メタデータが欠損しても `article_alt_href` だけは常に生成される
  = 常にクリック可能、という不変条件を担保。

### recap-worker

`src/pipeline/morning.rs`: `get_latest_completed_job(3)` に変更。
`source_recap_window_days` は自動的に `Some(3)` になり、UI の
"Based on 3-day recap" 表示に反映される。

### alt-frontend-sv

新コンポーネントは 2 つ:

- `lib/components/morning-letter/ArticleDeckCard.svelte`:
  単一カード。feed name → clickable title (`/articles/<uuid>`) →
  summary excerpt → 上位 4 tag チップ (超過は "+N") →
  "Threads nearby" 2〜3 件 → "Chat with Acolyte →" プライマリ CTA +
  "Original ↗" ghost 外部リンク。Alt-Paper トークンだけ使用。
- `lib/components/morning-letter/ArticleDeck.svelte`:
  deck 全体のラッパ。3px double 罫で本文と区切り、masthead 風の
  kicker "Threads to pull" + title "Develop from here" + 1 行 hint。

既存ピースの変更:

- `lib/connect/morning_letter.ts` + `lib/connect/index.ts` に
  `getLetterEnrichment(transport, letterId)` を追加、NotFound は空配列に
  変換。
- `lib/hooks/useMorningLetter.svelte.ts` に `enrichments` state と
  `loadEnrichments()` を追加、`fetchLetter` / `fetchByDate` の両方から
  並列発火。`enrichmentsLoading` も expose。
- `MorningLetterDocumentCore.svelte` の既存 sections 後に
  `<ArticleDeck>` を挿入。props は最小 (enrichments + loading のみ)。
- `+page.svelte` は `ml.enrichments` / `ml.enrichmentsLoading` を
  DocumentCore に渡すだけ。

## Consequences

### Pros

- Morning Letter が初めて "読者が発展させられる紙面" になった。
  各 bullet に対して **読む (記事遷移)** / **関連を追う (threads
  nearby)** / **深掘る (Acolyte チャット)** の 3 つの入り口が揃う。
- projector は純粋なまま、BFF が render-time に enrichment を合成する
  設計で canonical contract に抵触しない。`morning_letters` / sources
  のスキーマ変更ゼロ、proto の既存 message も追加のみ。
- `FetchArticlesByIDs` と `FeedTitleBatchPort` は batch 化されており、
  8 件のカード分の article + feed title 解決は SQL 2 本。related だけ
  並列 4 concurrency で最大 4 件の search-indexer コールに収まる。
- 記事が削除済み / 未 tag 付けの場合も `article_alt_href` が常に生成
  され、Article 詳細ページへは必ず遷移できる。graceful degradation が
  単体テストで担保されている。
- recap window が 3 日へ切り替わり、Letter が古い recap を引きずる
  問題が解消。

### Cons / Tradeoffs

- 現状 `enrichmentMaxSources=8` にハードコード。ユーザー毎の調整
  (例: RSS 購読数が多い人は 12 枚表示したい) は将来の pref API 頼み。
- 関連記事は search-indexer の lexical マッチ。ベクトル類似度に切り
  替えれば質は上がるが、embedding 経路は別スコープ。
- Acolyte seed link は `article_id` を URL query に載せるだけで、
  Acolyte 側は現状これを読み取らない (scope.topic のみ利用)。
  Acolyte への `article_id` 取り込みは別 ADR。
- `enrichmentSearchConcurrency=4` は経験値。search-indexer の容量が
  変わったら調整必要。外部 API への最低 5 秒スロットリングは該当しない
  (search-indexer は内部サービス)。
- summary excerpt は現状 `Article.Summary` or `Article.Content` 先頭
  220 文字。`summary_versions` テーブルの versioned summary を優先
  参照するのは follow-up (データ量が揃ってから)。
- LLM が extractive-fallback に落ちる根本は ADR-000706 の Phase 2.5
  (subworker clustering + scaffold contract) で改善予定。本 ADR は
  bullet の中身は触っておらず、読者体験の **周辺** を強化したのみ。

## Verification (2026-04-13 実測)

- `cd alt-backend/app && go test ./...` 全グリーン。
  `TestGetLetterEnrichment_*` 2 件追加済み。
- `cd alt-frontend-sv && bun run check && bun run test --run
  src/lib/hooks/useMorningLetter.test.ts
  src/lib/components/morning-letter/morning-letter-document.test.ts`
  → 0 errors, 37 件 passed。
- `docker compose up --build -d alt-backend alt-frontend-sv recap-worker`
  成功。3 コンテナ healthy。
- `POST /v1/morning/letters/regenerate` で `source_recap_window_days:
  3`、through-line `"Overnight brought 218 new threads across your
  feeds."`、revision 5 が記録された。
- `POST /api/v2/alt.morning_letter.v2.MorningLetterReadService/
  GetLetterEnrichment` が登録済みで、unauthenticated 呼び出しは 403
  を返す (意図通り。auth 付き FE 経路で enrichment が返る)。
