# Circuit Breaker MaxConcurrentRequests デフォルト値の引き上げ

## ADR's STATUS

Accepted

## CONTEXT

[[000331|ADR 000331]] により RSS フェッチが1回化され、フェッチ速度が約5倍向上した。
しかし、3000 VU 負荷試験において **65% のリクエストが即座に拒否** される問題が発生した。

原因は `SimpleCircuitBreaker` の `MaxConcurrentRequests` デフォルト値が `10` であったこと。
`ValidateAndFetchRSSGateway` は既に `fetchSem`（50スロット）で同時実行制御を行っているが、
その手前の Circuit Breaker が `MaxConcurrentRequests=10` で先にリクエストをブロックしていた。

負荷試験 Run 3 で `CB_MAX_CONCURRENT=500` に引き上げたところ、エラー率 0% を達成。
これにより、CB の同時実行制限が冗長かつ過度に制限的であることが確認された。

### 問題の構造

```
リクエスト
  → Circuit Breaker (MaxConcurrentRequests=10)  ← ここがボトルネック
  → fetchSem (50スロット)
  → HTTP外部フェッチ
```

セマフォが50並列を許容するにもかかわらず、CB が10で先に蓋をしていたため、
11個目以降のリクエストが `"too many concurrent requests"` で即座に失敗していた。

## DECISION MAKING

### デフォルト値を 10 → 100 に引き上げ

**変更ファイル**: `alt-backend/app/utils/resilience/simple_circuit_breaker.go`

```go
// Before
maxConcurrent := 10

// After
maxConcurrent := 100
```

### 値の根拠

- Gateway の `fetchSem` が50スロットで同時実行制御を担当しているため、
  CB の `MaxConcurrentRequests` はセマフォの上限を超えない限り発火しない **安全弁** として位置づける
- 100 = セマフォ上限の2倍で、セマフォが正常に機能している限り CB 側で制限がかかることはない
- 環境変数 `CB_MAX_CONCURRENT` による上書きは引き続き可能

### 設計判断

| 選択肢 | 判定 | 理由 |
|--------|------|------|
| MaxConcurrentRequests を撤廃 | 不採用 | CB の安全弁機能を維持したい |
| 500（負荷試験 Run 3 の値） | 不採用 | 過大。セマフォの10倍は安全弁として意味が薄い |
| **100（セマフォの2倍）** | **採用** | セマフォ正常時は発火せず、セマフォ故障時のみ保護する適切なマージン |
| セマフォ値から動的算出 | 不採用 | 過剰な結合。静的なデフォルト値で十分 |

## RESULTS, EFFECTS

### PROS

- **負荷試験時のエラー率解消**: 3000 VU 負荷でも CB による誤拒否が発生しない
- **既存の同時実行制御を阻害しない**: セマフォが本来の制御層として正しく機能する
- **安全弁としての CB 機能を維持**: セマフォの故障や設定ミスへの最終防衛線は残る
- **環境変数による柔軟な調整**: `CB_MAX_CONCURRENT` で運用時にチューニング可能
- **テストへの影響なし**: 既存テストは `MaxConcurrentRequests: 1` を明示設定しており、デフォルト値に依存していない

### CONS, TRADEOFF

- **セマフォ故障時の保護閾値が緩くなる**: 10 → 100 に引き上げたことで、
  セマフォが機能しない異常時に100並列まで許容される（以前は10）。
  ただし、このシナリオ自体が異常であり、CB の本来の役割（障害時のフェイルファスト）とは別の問題
- **セマフォスロット数との暗黙的な関係**: セマフォを50から変更する場合、
  CB のデフォルト値も見直す必要がある（ただし環境変数で対応可能）
