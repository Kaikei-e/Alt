# ADR-000440: Knowledge Home の空タグ表示と Open ボタン遷移を修正する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-frontend-sv (TypeScript/SvelteKit)
- alt-backend (Go)

## Context

Knowledge Home のカード表示に2つのバグが存在していた。

### Bug 1: 空タグチップの表示

DB 上に空文字のタグが格納されている場合、`KnowledgeCard.svelte` の `item.tags.slice(0, 3)` が空文字をフィルタせず、中身のない矩形がタグチップとして描画されていた。

### Bug 2: Open ボタンが機能しない

Open ボタンは `/feeds?article={uuid}` に遷移していたが、feeds ページはこの query パラメータを解釈して記事を開く仕組みを持っていなかった。根本原因は `KnowledgeHomeItem` proto に記事の元 URL（`link`）が含まれておらず、article UUID だけでは記事本文を取得できない設計上のギャップにあった。

一方で既存の記事本文取得フロー（`getFeedContentOnTheFlyClient`）は URL を起点に動いており、feeds ページに壊れた互換導線を追加するよりも、read model が UI に必要な `link` を提供し、URL を起点にした専用記事ビューへ遷移させる方が ADR の設計方針に合致する。

## Decision

以下の修正を採用する。

### 空タグチップ

- `KnowledgeCard.svelte` で `$derived` を使って空文字・空白のみのタグを事前にフィルタし、フィルタ済み配列から表示用スライスと残件数を計算する。

### Open ボタン遷移

- **Proto 拡張**: `KnowledgeHomeItem` に `string link = 13` フィールドを追加し、read model が記事の元 URL を提供できるようにする。
- **Repository JOIN enrichment**: `GetKnowledgeHomeItems` クエリで `articles` テーブルを `LEFT JOIN` し、`COALESCE(art.url, '') AS link` として取得する。`knowledge_home_items` テーブルに新カラムは追加しない（[[000423]] の enrichment パターンに準拠）。
- **専用記事ページ新設**: `/articles/[id]?url=...` ルートを新設し、URL パラメータから `getFeedContentOnTheFlyClient` で本文取得、`RenderFeedDetails` で描画する。
- **遷移先変更**: Home の Open ボタンを `/feeds?article=` から `/articles/{id}?url=...` に変更する。`link` がない場合は壊れた fallback に戻さず no-op とする。

### 検討した代替案

1. **feeds ページに `?article=` パラメータのハンドリングを追加する**: feeds ページは URL/normalizedUrl を起点に設計されており、article UUID から逆引きする API が存在しない。壊れた導線を修正するよりも、URL を起点にした専用ページの方が責務が明確。
2. **`knowledge_home_items` テーブルに `link` カラムを追加する**: upsert 時に URL を非正規化して保持する方法。しかし read-time JOIN で十分であり、projection 側の変更が不要な点で JOIN enrichment を選択した。

## Consequences

### Pros

- 空タグチップが描画されなくなり、カードの視覚品質が向上する
- Open ボタンが実際に記事コンテンツを表示できるようになる
- `link` フィールドにより、将来的に外部リンクやシェア機能にも再利用可能
- `knowledge_home_items` テーブルへのスキーマ変更やマイグレーションが不要
- `LEFT JOIN` により `articles` に一致しない item（recap_anchor 等）でも壊れない

### Cons / Tradeoffs

- `GetKnowledgeHomeItems` に `LEFT JOIN` が加わり、クエリコストが若干増加する。ただし `primary_ref_id` と `articles.id` はインデックス済みであり、実用上のインパクトは小さい
- `/articles/[id]` ルートは現時点では `url` クエリパラメータに依存しており、`[id]` 単独での記事復元はスコープ外。URL なしでアクセスした場合はエラー表示に倒す
- `link` がない item で Open ボタンが no-op になるため、将来的に UI 側で disabled 表示を検討する余地がある
