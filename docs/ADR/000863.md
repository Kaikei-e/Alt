# ADR-000863: Knowledge Loop の full reproject を /admin/knowledge-home から実行できる admin endpoint と panel を追加し WhyMappingVersion を運用上 visible にする

## Status

Accepted

## Date

2026-04-27

## Affected Services

- knowledge-sovereign: `app/driver/sovereign_db/reproject_knowledge_loop.go` の `Repository.TruncateKnowledgeLoopProjections` を新設。1 transaction で `knowledge_loop_entries` / `knowledge_loop_session_state` / `knowledge_loop_surfaces` を TRUNCATE し、`knowledge_projection_checkpoints` の `knowledge-loop-projector` 行を `last_event_seq = 0` で reset。`knowledge_loop_transition_dedupes` は触らない (canonical contract §3 invariant 8)
- knowledge-sovereign: `app/handler/reproject_knowledge_loop_handler.go` を新設。POST `/admin/knowledge-loop/reproject` (destructive) と GET `/admin/knowledge-loop/reproject/status` (read-only、`WhyMappingVersion` + projector checkpoint を返す) を metrics-port mux に register。POST 以外の method は 405 で拒否
- alt-frontend-sv: `src/lib/server/sovereign-admin.ts` に `triggerKnowledgeLoopReproject()` / `fetchKnowledgeLoopReprojectStatus()` を追加し、SOVEREIGN_METRICS_URL に直接アクセス
- alt-frontend-sv: `src/routes/(app)/admin/knowledge-home/reproject-loop/+server.ts` を新設し POST / GET の両 verb を提供。`getUserRole(locals.user) !== "admin"` で 403、上流 fetch 失敗で 502
- alt-frontend-sv: `src/lib/components/knowledge-home-admin/KnowledgeLoopReprojectPanel.svelte` を新設し、Reproject タブ末尾に配置。"REPROJECT" 文字列の一致で armed されるボタンを採用、Archive と同じ terracotta-armed CSS。WhyMappingVersion / projector checkpoint をパネル冒頭で表示し「Knowledge Home の `projection_version` とは別系統」と注記

## Context

[[000861]] で `WhyMappingVersion` を 6 → 7 に bump し、placement priority が変わった。canonical contract §6.5 と `docs/runbooks/knowledge-loop-reproject.md` には「version bump 後は full reproject runbook を回す」と書かれているが、実行手段は psql を直接叩くか、`curl -X POST :9000/internal/jobs/...` を打つかのどちらかで、operator が誤コマンドを叩く余地が大きかった。

加えて `/admin/knowledge-home` ページには既存の Knowledge Home reproject panel (compare / swap / rollback の shadow 方式) が存在し、operators は「v3 active」「v4 active」のような表示を見慣れていた。Knowledge Loop は disposable projection で TRUNCATE-and-rerun semantics なので shadow / swap の概念に乗らない。`v3` という表示が `WhyMappingVersion 7` と無関係なのに、operator から見ると同じ "version" 概念に見えてしまう混乱があった。

`fb.md` レビューで「Reproject を admin/knowledge-home から実行できるようにして」と要望された。同時に「UI 上は v3 から見える」という質問もあり、これは Knowledge Home `projection_version` (shadow/swap カウンタ) と Knowledge Loop `WhyMappingVersion` (projector code 世代) を混同している兆候だった。

destructive operation を UI に出すには (a) ロールガード、(b) 二段確認、(c) idempotent semantics、(d) audit trail が必要。本 ADR でその全部を埋める設計を確定する。

## Decision

`/admin/knowledge-home` の Reproject タブに Knowledge Loop 専用パネルを追加し、knowledge-sovereign の metrics port に POST/GET 2 endpoints を新設して runbook 手順を 1 ボタン化する。WhyMappingVersion を画面上に明示することで Knowledge Home `projection_version` との混同を断ち切る。

採用した経路:

- **knowledge-sovereign 側 endpoint**:
  - `POST /admin/knowledge-loop/reproject` — `Repository.TruncateKnowledgeLoopProjections` を呼び、3 projection table を 1 transaction で TRUNCATE + checkpoint reset。pre-truncate row counts を JSON で返して runbook の "1-5% delta" post-check に使えるようにする。`knowledge_loop_transition_dedupes` は **触らない** (canonical contract §3 invariant 8 — dedupe は ingest-side の idempotency barrier であり projection ではない)
  - `GET /admin/knowledge-loop/reproject/status` — `WhyMappingVersion` 定数と `Repository.GetProjectionCheckpoint("knowledge-loop-projector")` を返す pure read。state を一切変えない
  - GET / DELETE 等で reproject endpoint を叩いても 405。runaway crawler / curl typo で projection が消えない設計 (`POST /admin/knowledge-loop/reproject` mux パターンが http.ServeMux の go1.22+ method-aware routing で 405 を返す)
- **alt-frontend-sv 側 BFF**: `/admin/knowledge-home/reproject-loop/+server.ts` で `getUserRole(locals.user) !== "admin"` を最初に gate。POST / GET の両方で同じ guard を行い、上流 fetch 例外は 502 に正規化
- **UI panel**: `KnowledgeLoopReprojectPanel.svelte` は Reproject タブ末尾の `--stagger: 4` slot に配置。`onMount` で `GET /admin/knowledge-home/reproject-loop` を呼んで status を表示。"REPROJECT" 文字列を type すると armed フラグが立ち、ボタンが terracotta accent で active になる。reproject 成功後に再度 status fetch を打って checkpoint が 0 → 増加するのを画面上で確認できる
- **WhyMappingVersion を画面に出す**: Panel header に `WhyMappingVersion v7` / `Projector checkpoint event_seq <N>` / `Projector knowledge-loop-projector` を表示。さらに「これは projector の code-side version。Knowledge Home の `projection_version` (このページの他箇所で見える) とは別系統」と注記を本文に明示。UI で混同が起きないよう text を 1 段落配置する

検討した代替案と却下理由:

- **既存 Knowledge Home reproject panel に Knowledge Loop も乗せる**: shadow / swap 用の compare / swap / rollback フローが Loop には適用できない。同じ panel に異なる semantics を 2 つ詰め込むと operator の認知負荷が増える。専用 panel に分けて semantics を分離する
- **alt-backend を経由した admin endpoint にする**: 既存の `sovereign-admin.ts` パターンが metrics port (`SOVEREIGN_METRICS_URL`) に直接アクセスする実装になっており、retention / snapshot / storage 系の admin endpoints も同じ経路。alt-backend hop を増やすと auth chain が冗長になる
- **dedupe table も TRUNCATE**: canonical contract §3 invariant 8 違反。dedupe table は ingest-side で client retry の idempotency を担保している。reproject で消すと「projection を再構築する」副作用以外に「過去の dedupe key を消して再投入を許す」副作用が乗る。明示的に保護する
- **二段確認を modal dialog にする**: モーダルだと "REPROJECT" タイプ要件を強制しづらい。inline text input + match で armed のほうが UI 的に Alt-Paper の minimal patterns と整合する
- **POST 専用にして status endpoint は作らない**: WhyMappingVersion の値を operator が知りたいとき、毎回 panel を再 deploy する必要がある。read-only endpoint を 1 つ追加するコストは小さく、混同問題の解消には valuable
- **status endpoint を `/admin/knowledge-loop/version` にする**: `/admin/knowledge-loop/reproject/status` のほうが「reproject 機能のまわりの状態」を意味する命名で意図が明瞭

## Consequences

### Pros

- WhyMappingVersion bump 後の full reproject が 1 ボタンで実行可能になり、psql / `curl :9000/internal/...` を直接叩く運用 risk が消える
- "REPROJECT" type 確認 + admin role gate + POST-only の 3 重防御で、destructive operation の誤発火 (crawler / typo / 非 admin) が成立しない
- WhyMappingVersion をパネル上に明示することで、Knowledge Home `projection_version` との混同を UI レベルで断ち切れる
- `TruncateKnowledgeLoopProjections` が 1 transaction で TRUNCATE + checkpoint reset を行うため、部分失敗時の inconsistent state にならない (rollback で元通り)
- Idempotent: 連続実行は no-op (空 TRUNCATE / 0 → 0 reset)。canonical contract §3 invariant 1 (reproject-safe) と整合
- Reproject 直後に status を再 fetch する UX により、checkpoint が advancing しているかを panel 上で確認できる。runbook の post-check の一部を UI に内包

### Cons / Tradeoffs

- BFF は metrics port (`http://knowledge-sovereign:9501`) に直接アクセスする既存パターンを踏襲する。compose の service-to-service network 上では問題ないが、外部公開できない URL を BFF の env から読む構成は依存が見えにくい。`docs/services/alt-frontend-sv.md` への補足は本 ADR では行わなかった
- WhyMappingVersion は Go 定数で、bump → reproject の運用順序を operator が忘れると entry の placement が混在する。UI 上の status 表示で operator が version を確認できるが、自動 reproject はしない
- `Repository.TruncateKnowledgeLoopProjections` は単発トランザクションで TRUNCATE するため、巨大 projection (数百万行) で transaction が長時間ブロックする可能性がある。現状の規模では問題ないが、scale-out 時には `TRUNCATE ... CASCADE` の影響範囲を再評価する余地がある
- terracotta-armed ボタンは Alt-Paper 的に「破壊的操作」を視覚化しているが、色覚多様性に依存する operators には text confirmation 側で意図を伝える設計に依存している。aria-label の追加は本 ADR では行わなかった
- Reproject panel は Reproject タブの `--stagger: 4` 末尾に置いたため、Knowledge Home reproject の compare / swap / rollback フローを scroll した先に見える。「Knowledge Home reproject」と「Knowledge Loop reproject」の視覚分離は注記文に依存している。タブを分ける案は将来検討
