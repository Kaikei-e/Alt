# ADR-000736: Pact CDC の残ギャップを埋めて Broker を常時稼働・can-i-deploy で docker publish をブロックする

## Status

Superseded by [[000739]] (Pact ギャップ埋め) + [[000740]] (Broker 恒常化 + can-i-deploy gate)。作業を Phase D と Phase E に分割した ADR に置換した。

## Date

2026-04-15

## Affected Services

- alt-backend (Go) — 新規 Provider 検証 `TestVerifySearchIndexerContract` 追加
- mq-hub (Go) — 新規 Provider 検証 `TestVerifySearchIndexerMqHubMessagePact` 追加
- search-indexer (Go) — `provider_test.go` で acolyte pact 除外解除、追加 state handler
- rag-orchestrator (Go) — 新規 Consumer 契約 `recap_worker_consumer_test.go` 追加
- acolyte-orchestrator (Python) — consumer test で `X-Service-Token` を明示 assert
- pact-broker / pact-db — `compose/pact.yaml` を default profile に昇格、`secret` 経由認証
- `.github/workflows/proto-contract.yaml` — search-indexer / mq-hub Provider 検証を追加、必須 status check ガイダンス
- `.github/workflows/release-gate.yaml` (新規) — `can-i-deploy` + `record-deployment`
- `.github/workflows/deploy.yaml` — `needs: release-gate` で block

## Context

[[000591]] で 17 ペアの Pact CDC 基盤を展開したが、運用ギャップが 5 つ残っていた。[[000735]] で「search-indexer を呼ぶ全 consumer に `X-Service-Token` を強制し Pact で回帰を封じる」としたが、同時期のポストモーテム [[PM-2026-025]] が「acolyte-orchestrator の consumer pact が `X-Service-Token` を assert しておらず 24h サイレント degradation」を記録しており、**Pact があっても実際に検証されなければ意味がない**ことを示していた。

具体的に残っていたギャップは以下:

1. **`rag-orchestrator → recap-worker` (morning letter) の Consumer 契約が無い**。retrieval パイプラインの critical path だが unguarded だった。
2. **`search-indexer → alt-backend` の Provider 検証が存在しない**。`search-indexer-alt-backend.json` は存在するが alt-backend 側が verify していなかった。
3. **`search-indexer → mq-hub` 非同期メッセージの Provider 検証が存在しない**。`search-indexer-mq-hub.json` は search-indexer の consumer test が生成していたが、mq-hub 側で検証されていない orphan contract だった。
4. **`acolyte-orchestrator → search-indexer` consumer test が `X-Service-Token` を assert していない**。PM-2026-025 の根本原因。search-indexer の provider test は acolyte pact を `intentionally excluded` として除外していた。
5. **Pact Broker が `profiles: [pact]` の opt-in**。`scripts/pact-check.sh --broker` を走らせる時にだけ起動する運用で、`can-i-deploy` gate の前提となる永続 Broker が存在しなかった。

Pact 運用としても、`scripts/pact-check.sh` が opt-in で必須 CI gate になっておらず、かつ docker publish 時点での互換性チェックがなかった。これは「契約違反 PR が main に入り、デプロイまで走って本番で 401/スキーマ不一致を起こす」構造的リスクだった。

## Decision

**Pact CDC を "broker-first・blocking gate" な運用に昇格し、残ギャップを同一 PR で埋める。** 具体的に採用した 4 点:

1. **残ギャップ 5 件をすべて埋める (Phase D1-D4)**
   - `rag-orchestrator/internal/adapter/contract/recap_worker_consumer_test.go` を TDD で新設し、`FetchLatest` / `FetchLatest 404` / `FetchByDate` の 3 interactions を pin。
   - `alt-backend/app/driver/contract/provider_test.go` に `TestVerifySearchIndexerContract` を追加し、Connect-RPC `BackendInternalService.{GetLatestArticleTimestamp, ListArticlesWithTags}` を JSON wire で stub 検証。
   - `mq-hub/app/driver/contract/provider_test.go` を新設し、`pact-go/v2/message` + `provider.NewVerifier()` で async message 3 interactions (`ArticleCreated` / `ArticleCreated fat` / `IndexArticle`) を verify。
   - `acolyte-orchestrator/tests/contract/test_search_indexer_consumer.py` を `.with_header("X-Service-Token", ...)` で更新し、`search-indexer/app/driver/contract/provider_test.go` で acolyte pact の除外コメントを撤去、必要な state handler を追加。

2. **Pact Broker を永続化 (Phase E1)**
   - `compose/pact.yaml` から `profiles: [pact]` を削除。`compose/compose.yaml` の `include` に追加。
   - basic-auth パスワードを `secrets/pact_broker_basic_auth_password.txt` (Docker secret) から注入。hardcoded `pact:pact` を廃止。`PACT_BROKER_ALLOW_PUBLIC_READ=false` に変更し、未認証 read を拒否。
   - `compose/backup.yaml` に `pact_db_data` を追加、`scripts/backup/backup-all.sh` の `databases` 配列に `pact-db` を加えて既存 Restic 3-2-1 ローテーションに組み込んだ。

3. **`scripts/pact-check.sh` を CI 必須 gate に昇格 (Phase E2)**
   - FFI ライブラリ検出を `/usr/local/lib` にも拡張し、環境差異による `skip_step` を排除。
   - Broker basic-auth を `secrets/pact_broker_basic_auth_password.txt` から読むように更新。
   - `.github/workflows/proto-contract.yaml` に `search-indexer` Provider 検証 step と `mq-hub` Provider 検証 step を追加。header comment で「必須 status check として branch protection に登録すべきジョブ名」を明示。

4. **`can-i-deploy` release gate を docker publish 前に配置 (Phase E3)**
   - `.github/workflows/release-gate.yaml` を新設 (workflow_call)。14 pacticipant それぞれに `pact-broker can-i-deploy --to-environment production` を実行し、1 つでも fail すれば exit 1。
   - 成功時は `pact-broker record-deployment --environment production` を Broker に記録し、Broker を「本番に存在するバージョン」の真実源にする。
   - `.github/workflows/deploy.yaml` に `needs: release-gate` を追加。`PACT_BROKER_PASSWORD` を repo secret で注入。
   - 単一マシン前提のため runner は `self-hosted`、Broker は `http://localhost:9292` に向ける。

## Consequences

### Pros

- **Pact audit 時点の全ギャップが閉じた**。`./scripts/pact-check.sh` は 15/15 pass。20 pact ファイル中 `orphan` (consumer はいるが provider 側で verify されていない) が 0 件に。
- **PM-2026-025 型のサイレント degradation が構造的に防がれる**。acolyte consumer が `X-Service-Token` を落とすと consumer test 時点で fail、provider も `ServiceAuthMiddleware` 実体と突き合わせて verify する。Pact が「飾り」で終わらない。
- **Broker が単一の真実源になる**。永続ストレージ + Docker secret 認証 + Restic バックアップで disposable ではなくなり、`can-i-deploy` / `record-deployment` を介した「誰が本番にいるか」の Query が成立する。
- **デプロイブロックが二段**。PR merge 時の `pact-check.sh` 必須化 (Phase E2) と、docker publish / deploy 時の `can-i-deploy` (Phase E3)。どちらかで contract regression を捕捉する。
- **`pact-check.sh` が CI とローカルで同一動作**。開発者は `./scripts/pact-check.sh` を叩けば CI と同じ検証を再現でき、feedback loop が短くなる。

### Cons / Tradeoffs

- **Branch protection の "required status check" 設定は GitHub UI で別途行う必要がある**。YAML には header コメントで job 名を明示しているが、repo settings 側の配線は人間操作。Runbook (`docs/runbooks/pact-broker-ops.md`) に手順を記録。
- **`PACT_BROKER_PASSWORD` を GitHub repo secret に登録する必要がある**。`secrets/pact_broker_basic_auth_password.txt` と同値を Actions secrets に入れる初期作業が必要。
- **recap-worker (Rust) の Provider 検証がまだ無い**。`pact_verifier` クレート + axum テストハーネス + DAO mock という大規模なインフラ追加が必要で、別 ADR (D7) で独立対応する。現時点では recap-worker を consumer とする 4 pact の provider 側 (news-creator, recap-subworker, alt-backend, tag-generator) が verify されているので、recap-worker 自体が provider のケース (search-indexer → recap-worker /v1/recaps/genres/indexable など) のみが残ギャップ。
- **Broker の DR は手動**。`pact_db_data` は Restic 経由でバックアップされるが、復旧手順はポジティブ確認未実施。`docs/runbooks/pact-broker-ops.md` に復元 dry-run 手順を記載、四半期ごとに verify する運用を提示。
