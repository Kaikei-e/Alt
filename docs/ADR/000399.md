# ADR-000399: Summary と Tag の versioning と superseded 契約を導入する

## Status

Accepted

## Date

2026-03-17

## Affected Services

- alt-backend (Go) — version 追記ロジック、イベント発行
- alt-db (PostgreSQL) — `summary_versions`, `tag_set_versions` テーブル追加

## Context

現在、要約とタグは `article_summaries` / `article_tags` テーブルに最新値のみが保存されている。再生成時には上書きされ、過去のバージョンは失われる。

RAG 側は既に document version + source hash idempotency を持ち（[[000241]] 準拠）、生成物のバージョン管理が確立されている。要約・タグも同じ思想で versioned artifact 化することで、以下を実現する:

- Knowledge Home の「要約が更新された」理由表示
- モデル・プロンプトバージョンごとの品質比較
- 再生成時の差分検知（input_hash による content-addressable dedup）

## Decision

`summary_versions` と `tag_set_versions` テーブルを導入し、生成物を versioned artifact として管理する。

### summary_versions

- `summary_version_id UUID PK` — バージョン識別子
- `article_id UUID NOT NULL` — 対象記事
- `generated_at TIMESTAMPTZ` — 生成日時
- `model TEXT` — 使用モデル名
- `prompt_version TEXT` — プロンプトバージョン
- `input_hash TEXT` — 入力コンテンツのハッシュ（SHA-256）
- `quality_score NUMERIC` — 品質スコア（将来のランキング用）
- `summary_text TEXT` — 要約本文
- `superseded_by UUID` — 次バージョンへのポインタ（NULL = 最新）

### tag_set_versions

- `tag_set_version_id UUID PK` — バージョン識別子
- `article_id UUID NOT NULL` — 対象記事
- `generated_at TIMESTAMPTZ` — 生成日時
- `generator TEXT` — 生成器名（tag-generator, on-the-fly 等）
- `input_hash TEXT` — 入力コンテンツのハッシュ
- `tags_json JSONB` — タグセット JSON
- `superseded_by UUID` — 次バージョンへのポインタ

### 既存テーブルとの関係

- `article_summaries` / `article_tags` は引き続き current-value projection として機能する
- `SaveArticleSummary` / `UpsertArticleTags` の internal handler 内で version 追記 + `knowledge_events` へのイベント発行を組み込む
- 既存の読み取りクエリは変更不要（projection テーブルがそのまま使える）

## Consequences

### Pros

- 要約・タグの生成履歴が保存され、品質改善のトラッキングが可能
- input_hash により同一入力での重複生成を検知可能
- Knowledge Home の WHY 表示（「要約が更新されました」）の根拠データとなる
- superseded チェーンにより、任意のバージョンへの rollback が理論上可能

### Cons / Tradeoffs

- ストレージ使用量の増加（全バージョンを保持するため）
- version テーブルの retention policy が未定（Phase 2 課題）
- 既存の SaveArticleSummary / UpsertArticleTags への組み込みは Phase 1 後半で実施
