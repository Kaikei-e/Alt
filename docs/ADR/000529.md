# ADR-000529: タグ削除バグとタグ生成スタックを修正する

## Status

Accepted

## Date

2026-03-22

## Affected Services

- alt-backend (Go) — Knowledge Home projector / repository
- tag-generator (Python) — InputSanitizer / BatchProcessor

## Context

Knowledge Home 上で 2 つのタグ関連バグが同時に発生していた。

### Bug 1: タグが消される ("Tags updated" バッジと空タグ)

`projectTagSetSuperseded()` が `KnowledgeHomeItem` を生成する際、`Tags` フィールドを未設定のまま upsert していた。Go の nil slice は `json.Marshal` で `"null"` にシリアライズされ、knowledge_home_items テーブルの merge-safe SQL ガード:

```sql
tags_json = CASE WHEN EXCLUDED.tags_json != '[]'::jsonb
            THEN EXCLUDED.tags_json
            ELSE knowledge_home_items.tags_json END
```

において `'null'::jsonb != '[]'::jsonb` が TRUE と評価されるため、既存タグが `null` で上書きされていた。イベント順序として `TagSetVersionCreated`（タグ設定）→ `TagSetSuperseded`（タグ null 化）が連続するため、新しいタグが付いた直後に消されるという再現性の高い障害だった。

DB 上の影響範囲: projection_version 1 に 3,926 件、projection_version 2（active）に 13,386 件の null/`"null"` タグレコードが存在。

### Bug 2: タグ生成が進まない (tag-generator スタック)

tag-generator の backfill 完了後、forward 処理で `fetch_new_articles()` が常に先頭ページ（最新の untagged articles）を取得する。この先頭ページに処理不可能な記事が居座ると head-of-line blocking が発生する。

実際に 5 件の記事が恒常的にブロックしていた:
- 4 件: 本文が `test` / `テスト` レベルの極小コンテンツ（3-4 文字）→ サニタイズ後 "too short"
- 1 件: 117,707 文字の HTML/コード混在記事 → `nh3` による HTML ストリップ後も `Suspicious patterns detected`

poison pill 機構（3 回失敗で永久スキップ）によりこれら 5 件はスキップされるが、先頭ページに他の processable article がないため 209 サイクル連続 0 件処理のまま完全にスタックしていた。

HTML 混在記事については、`nh3` はタグを除去するのみで本文抽出能力を持たないことが根本原因。`<pre><code>` ブロック内のコード文字列がそのまま残り、反復パターン検出で "Suspicious patterns" と誤判定されていた。

## Decision

### Bug 1 への対応

**nil slice の正規化を 2 層で行う。**

1. **projector 層**: `projectTagSetSuperseded()` および `projectSummarySuperseded()` で `Tags: []string{}` を明示的に設定し、nil slice が生まれないようにする
2. **repository 層**: `UpsertKnowledgeHomeItem()` で `Tags` / `WhyReasons` が nil の場合に空 slice へ正規化してから `json.Marshal` する — 将来の projector 追加時にも同じ問題が再発しない防御層

既存の null タグデータは手動 UPDATE ではなく、新しい projection version での full reproject で修復する。`knowledge_home_items` は disposable projection であり、append-first モデルの原則に沿って event log から再構築するのが正しい復旧手順。

### Bug 2 への対応

2 つの修正を組み合わせる:

1. **`readability-lxml` による HTML 本文抽出**: HTML 混在コンテンツに対して `readability-lxml` で本文候補を抽出し、`lxml` で plain text 化してから既存のサニタイズパイプラインに渡す。`<pre>`, `<code>`, `<script>`, `<style>` ブロックは readability 前に除去する。plain text 入力に対してはこの前処理をスキップし、既存の fast path を維持する

2. **forward 処理の head-of-line blocking 解消**: `fetch_new_articles()` の first-page semantics は維持する（最新の untagged articles を最初に確認する設計意図を壊さない）。代わりに、先頭ページの全記事が poison pill 閾値に達している場合、ページ末尾カーソルで `fetch_articles()` を呼び次ページを掘る。processable article が見つかるか、5 ページまでで打ち切る

3. **`batch_stats["skipped"]` の追加**: バッチ処理統計に skipped 件数を含め、forward batch ログにも skipped/failed/fetched を出力して運用可視性を向上

`fetch_new_articles()` を cursor-aware にする案は撤回した。`ListUntaggedArticles` RPC は backward keyset pagination (`(created_at, id) < cursor`) を使うため、cursor を渡すと "新着 forward" ではなく "古い記事の backfill" に意味が変わる。

## Consequences

### Pros

- Knowledge Home 上でタグが消失する問題が解消される
- 117K HTML 記事のタグ生成が実際に成功した（readability で 44K 文字に本文抽出 → 10 タグ生成）
- nil slice → JSON `"null"` のクラスは repository 層で一括防御され、将来の projector 追加時も安全
- forward 処理の head-of-line blocking が最大 5 ページ分まで自動解消される
- skipped 統計により運用時の stuck 検出が容易になる

### Cons / Tradeoffs

- `readability-lxml` + `lxml` の依存追加により tag-generator の Docker イメージサイズが若干増加する
- 極小本文（`test` / `テスト` レベル）の記事はそもそも有意なコンテンツがないため、readability でも救えない。これらは意図通り "too short" としてスキップされ続ける
- 既存の null タグ 17,000 件超は reproject で修復する必要がある（本 ADR のスコープ外、別途 reproject 計画として実施）
