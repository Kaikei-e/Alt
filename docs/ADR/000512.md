# ADR-000512: 記事 upsert 時の lifecycle event を created と updated に分離する

## Status

Accepted

## Date

2026-03-21

## Affected Services

- alt-backend (Go)
- pre-processor (Go)

## Context

記事保存系の主要経路では、`articles` に対して `ON CONFLICT ... DO UPDATE` を使う upsert が採用されていた。
しかし上流では insert と update を区別せず、既存記事の再保存や再同期でも `ArticleCreated` 相当のイベントを発行していた。

`pre-processor` はこのイベントを受けると要約 job を作成するため、既に要約済みの記事でも「新規記事として再通知された」だけで再度 queue に積まれる構造になっていた。
実データでも、1 記事あたり多数の summarize job 履歴が蓄積しており、event の意味論が queue 膨張の発火点になっていた。

この問題は「同じ記事を何度も保存する」こと自体ではなく、**upsert の結果を create-only event で表現していたこと** にある。
event-driven な downstream を壊さずに修正するには、article lifecycle を create と update で明確に分離する必要があった。

## Decision

**記事 upsert の結果を `created` と `updated` に分離し、新規 insert の場合のみ `ArticleCreated` を発行する。**

具体的には以下を採用する。

1. article upsert API は `article_id` に加えて「新規 insert だったか」を返す
2. 新規 insert の場合は `ArticleCreated` を発行する
3. 既存記事の更新だった場合は `ArticleUpdated` を発行する
4. knowledge event も同様に `ArticleCreated` と `ArticleUpdated` を分離する

これにより、downstream は「新規記事の発生」と「既存記事の内容更新」を別の出来事として扱えるようになる。

### 採用しなかった代替案

| 案 | 評価 |
|---|---|
| `ArticleCreated` のまま downstream 側だけで重複抑止する | 問題の根を隠すだけで、event 名と実際の意味がズレ続ける |
| upsert をやめて insert-only に戻す | URL 重複や内容更新を扱えなくなり、既存設計と整合しない |
| update 時はイベントを出さない | 要約以外の downstream で更新検知が必要になった際に情報が失われる |

## Consequences

### Pros

- `ArticleCreated` が create-only event として意味を回復する
- queue consumer が「更新」を「新規作成」と誤解しにくくなる
- 将来、更新系 downstream を追加する際にも `ArticleUpdated` を再利用できる

### Cons / Tradeoffs

- article write path の戻り値と mock が変わり、周辺テストの追随が必要になる
- `ArticleUpdated` を発行しても、現時点の consumer がそれを活用しなければ即時の効果は限定的である
- event 種別が 1 つ増えるため、運用時に create/update の区別を意識する必要がある
