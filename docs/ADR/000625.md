# ADR-000625: Global Federated Search による記事・Recap・Tag 横断検索の実装

## Status

Accepted

## Date

2026-04-05

## Affected Services

- alt-backend (Go) — 検索アグリゲーター、Connect-RPC ハンドラ、DI ワイヤリング
- search-indexer (Go) — Recap free-text 検索対応
- alt-frontend-sv (TypeScript/SvelteKit) — `/search` ページ新設、Home 検索リダイレクト

## Context

Knowledge Home (`/home`) の検索は `home.items.filter(item => haystack.includes(query))` という純粋なクライアントサイドフィルタリングで実装されていた。これは以下の問題を抱えていた：

1. **検索の実態はローカルフィルタ** — バックエンドの Meilisearch を一切使わず、画面上にロードされたアイテムの `title`、`summaryExcerpt`、`tags` を JavaScript の `includes()` で絞り込むだけ
2. **検索と Lens の責務混線** — `searchQuery` が `lensDraft.queryText` に直結しており、検索操作が Lens のフィルタ状態を暗黙に変更してしまう
3. **検索対象が記事のみ** — `/feeds/search` は Connect-RPC 経由で Meilisearch を叩く本物の検索だが、Recap・Tag は検索できない
4. **検索の意味が割れている** — `/home` のローカルフィルタと `/feeds/search` のバックエンド検索が "Search" として共存し、ユーザーの期待と合っていない

一方で基盤は揃っていた。search-indexer は Meilisearch 1.27.0 上に `articles` と `recaps` の2インデックスを持ち、alt-backend は Tag Cloud 用の PostgreSQL クエリも備えている。足りないのはこれらを横断して検索する集約レイヤーと、その結果を表示する UI だった。

## Decision

**`/home` の擬似検索を廃止し、記事・Recap・Tag を並列検索する `/search`（Global Search Hub）を新設する。** 検索アグリゲーションは alt-backend 側で `errgroup` による並列実行で行い、部分障害時も他のセクションの結果を返す graceful degradation パターンを採用する。

### 検討した代替案

**案 A: Meilisearch の `/multi-search` federated endpoint を利用**
- Meilisearch 1.10+ でネイティブサポートされる federated search を search-indexer 内で使う
- 却下理由: Tag 検索は PostgreSQL ベース（Meilisearch に Tag インデックスがない）。alt-backend が canonical data owner であり、集約ロジックはそちらに置くべき

**案 B: `/feeds/search` を拡張して全体検索に昇格**
- 既存ページに Recap・Tag セクションを追加
- 却下理由: `/feeds/search` は記事専門 UI として十分に成熟しており、異なる責務を混ぜると複雑化する。Federated Search Hub と Specialist UI を分離する方が各画面の責務が明確

**案 C（採用）: 新規 `/search` ページ + バックエンドアグリゲーター**
- alt-backend に `GlobalSearchService.SearchEverything` RPC を新設
- 3つの検索ソースを並列実行し、セクション別に集約
- `/search` は overview（top-k）のみ返し、深掘りは既存の専門 UI へ送る

### 実装の概要

#### Proto 定義

`proto/alt/search/v2/global_search.proto` に `GlobalSearchService` を新設：

```
SearchEverything(query, article_limit, recap_limit, tag_limit)
  → SearchEverythingResponse {
      article_section { hits[], estimated_total, has_more }
      recap_section   { hits[], estimated_total, has_more }
      tag_section     { hits[], total }
      degraded_sections[]
      searched_at
    }
```

`proto/services/search/v2/search.proto` の `SearchRecapsRequest` に `optional string query` フィールドを追加し、free-text Recap 検索を後方互換で有効化。

#### バックエンド（alt-backend）

Clean Architecture に従い、以下のレイヤーを追加：

| レイヤー | コンポーネント | 役割 |
|---------|--------------|------|
| Domain | `GlobalSearchResult`, `ArticleSearchSection`, `RecapSearchSection`, `TagSearchSection` | 検索結果のドメインモデル |
| Port | `SearchArticlesPort`, `SearchRecapsPort`, `SearchTagsPort` | 3つの検索ソースへの抽象インターフェース |
| Gateway | `ArticleSearchGateway` | search-indexer Connect-RPC + URL enrichment |
| Gateway | `RecapSearchGateway` | search-indexer Connect-RPC（free-text query） |
| Gateway | `TagSearchGateway` | PostgreSQL `ILIKE prefix%` クエリ |
| Usecase | `GlobalSearchUsecase` | `sync.WaitGroup` による3系統並列実行、degraded mode 制御 |
| Handler | `GlobalSearchServiceHandler` | Connect-RPC エンドポイント、feature flag チェック |

並列実行では各セクションに 3 秒のタイムアウトを設定。1 系統が失敗しても `degraded_sections` に記録して残りの結果を返す。全系統失敗時のみエラーを返す。

#### search-indexer

`SearchRecapsUsecase` に `ExecuteByQuery(ctx, query, limit)` メソッドを追加。Meilisearch の `recaps` インデックスは既に `["tags", "top_terms", "summary", "genre"]` が searchable なので、RPC インターフェースの制限を解除するだけで free-text 検索が機能する。ハンドラは `query` が設定されていれば `ExecuteByQuery` を、なければ従来の `tag_name` パスを使う後方互換設計。

#### フロントエンド（alt-frontend-sv）

- `/search` ページ: Sectioned Overview レイアウト（Articles / Recaps / Tags の3セクション）
- `useGlobalSearch` hook: Svelte 5 Runes ベースの状態管理
- `ArticleSearchSection`: タイトル、snippet、matched field badges (title/content/tags)、タグチップ
- `RecapSearchSection`: ジャンル名、window days バッジ、サマリー抜粋、キーワードチップ
- `TagSearchSection`: タグチップ（記事数付き）
- "See all →" リンクで各専門 UI（`/feeds/search`, `/recap`, `/articles/by-tag`）へ送る

`/home` の `handleSearchSubmit` は `enable_global_search` feature flag が有効な場合 `/search?q=...` にリダイレクトする。flag OFF 時は既存のローカルフィルタリングがフォールバックとして残る。

## Consequences

### Pros

- **Search の意味が明確化** — 全体検索は `/search`、記事専門は `/feeds/search`、Home はナレッジの起点という役割分担が確立
- **3 ソース横断検索** — 記事・Recap・Tag を 1 クエリで検索でき、ユーザーは複数画面を行き来する必要がない
- **Graceful degradation** — Meilisearch ダウン時でも Tag 検索は動作し、逆に DB 障害時でも記事・Recap 検索は動作する
- **Feature flag によるロールアウト** — `enable_global_search` で段階的に有効化でき、問題があれば即座に無効化できる
- **既存資産の活用** — search-indexer の `articles`/`recaps` インデックス、Tag Cloud の PostgreSQL クエリをそのまま活用
- **専門 UI との共存** — Overview (top-k) と Specialist UI の責務が分離されており、各画面が独立して進化できる

### Cons / Tradeoffs

- **Tag 検索は Meilisearch ではなく PostgreSQL ILIKE** — `text_pattern_ops` インデックスがなければ大規模データでパフォーマンス劣化の可能性あり。現時点では personal scale で問題ないが、将来的には Tag を Meilisearch に載せることを検討
- **Recap 検索はユーザースコープなし** — 記事検索は `user_id` でフィルタされるが、Recap は全ユーザー共通データのため分離されていない。これは既存の Recap 閲覧と同じ挙動
- **Blended ranking は非採用** — 記事・Recap・Tag のスコアを統合したランキングは意味の違いが大きすぎるため見送り。セクション分割表示とした
- **Overview のページネーションなし** — `/search` は top-k 固定で infinite scroll を持たない。深掘りは専門 UI に委譲する設計意図だが、ユーザーにとって追加遷移が必要
