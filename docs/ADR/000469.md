# ADR-000469: /articles/[id] ページに記事取得・再取得ボタンを追加する

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-frontend-sv (SvelteKit / TypeScript)

## Context

`/articles/[id]` ページは Knowledge Home から遷移して記事を表示するページである。記事コンテンツは遷移時に自動取得されるが、ユーザーが明示的に再取得（再スクレイプ）する手段がなかった。

既存の Summarize ボタンは `useSummarize` フックにより idle → loading → success → error の 4 状態を視覚的に表現しており、UX として確立されたパターンとなっている。記事取得についても同等の状態遷移ボタンを提供し、操作の一貫性と透明性を高める必要があった。

## Decision

`/articles/[id]` ページのヘッダーに **Fetch / Re-fetch ボタン**を追加する。Summarize ボタンの左に配置し、同じ 4 状態パターン（idle / loading / success / error）で記事取得状態を表示する。

### 実装の構成

1. **`fetchButtonState` 算出値**: `$derived.by()` で `isFetching`・`contentError`・`articleContent` から 4 状態を導出する。`useSummarize` の `buttonState` と同じパターン。

2. **`fetchContent(forceRefresh)` の拡張**: 既存の `fetchContent()` に `forceRefresh` パラメータ（デフォルト `false`）を追加。`getFeedContentOnTheFlyClient(url, { forceRefresh })` に渡す。API 側（Connect-RPC）は `forceRefresh` に既に対応済み。

3. **`handleFetch()` ハンドラ**: `fetchButtonState === "success"` の場合を再取得と判定し、`summarizer.reset()` でサマリーをリセットしてから `fetchContent(true)` を呼ぶ。コンテンツが変わればサマリーも無効になるため、リセットは必須。

4. **ボタン UI**: Summarize ボタンの左に配置。lucide の `Download`（idle）、`RefreshCw`（success / error）、`Loader2`（loading）アイコンを使い分ける。エラー時は `variant="destructive"` で視覚的に区別。

### テスト

TDD で以下 5 ケースをブラウザテスト（`vitest-browser-svelte`）として実装した:

- 取得中に「Fetching...」ボタンが disabled で表示される
- 取得成功後に「Re-fetch」ボタンが表示される
- 取得エラー時に「Try again」ボタンが destructive スタイルで表示される
- 「Re-fetch」クリックで `forceRefresh: true` が渡される
- 再取得時にサマリーがリセットされる

## Consequences

### Pros

- ユーザーが記事コンテンツの再取得を明示的に実行でき、スクレイプ結果が不完全な場合に自力で対処できる
- Summarize ボタンと同じ状態遷移パターンにより、操作の一貫性が保たれる
- 再取得時にサマリーを自動リセットすることで、古いコンテンツに基づくサマリーが残る問題を防ぐ
- 既存の `getFeedContentOnTheFlyClient` の `forceRefresh` 対応をそのまま活用でき、API 側の変更が不要

### Cons / Tradeoffs

- ヘッダー領域にボタンが 1 つ増えるため、モバイル幅ではスペースが狭くなる可能性がある。現時点では `gap-3` の flex レイアウトで対応しているが、ボタン数が更に増える場合はドロップダウンメニューなどへの集約を検討する必要がある
- コンテンツエリアの既存「Try again」ボタンとヘッダーの「Try again」ボタンが同時に表示されるケースがある（エラー時）。冗長だが、コンテンツエリアのボタンはエラーメッセージと共に表示されるため、文脈としては異なる
