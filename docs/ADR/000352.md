# 000352: Tag Verse 改善 — 惑星間隔拡大 + 常時ラベル表示 + ホバーツールチップ + HUD重なり修正

## ADR's STATUS

Accepted

## CONTEXT

[[000351|ADR-351]] で導入したサーバーサイドフォースグラフレイアウトと WebGPU 化により、Tag Verse の基本動作は安定したが、実際に200タグを表示すると以下の3つのUX課題が明らかになった:

1. **惑星（タグ）が密集しすぎ** — `repulsionConstant = 15.0` では200タグの相互反発力が不足し、中央付近に大半のタグが集まってラベルが重なり合い判読不能
2. **惑星にラベルがない** — タグ名が常時表示されず、各惑星が何のタグを表すか判別できない。ホバーしない限り全てが無名の球体
3. **ホバー時のテキストが表示されない** — 既存のホバー実装は `T.Sprite` + `T.SpriteMaterial` による黒い半透明矩形のみで、テキスト自体が未実装だった

加えて、惑星選択時にスライドインする HUD サイドバーの上にタグラベルが重なって表示される z-index の問題も発見した。

## DECISION MAKING

### 1. フォースグラフ定数の段階的チューニング

反発力と引力の比率を大幅に変更し、惑星間の分散を改善した。正規化ステップ（座標を `[-100, 100]` に収める処理）があるため、絶対値よりも**反発力/引力の比率**が分散の均一性を決定する。

| パラメータ | Before ([[000351|ADR-351]]) | After | 倍率 | 理由 |
|-----------|-------------------|-------|------|------|
| `repulsionConstant` | `15.0` | `150.0` | 10x | 全ペア間の反発力を大幅強化 |
| `attractionConstant` | `0.005` | `0.0003` | 1/17x | 共起エッジの引力を抑制し、クラスタの過度な密集を防止 |
| `centeringForce` | `0.001` | `0.0001` | 1/10x | 中心への引力を弱め、空間の隅まで活用 |
| `minDistance` | `0.1` | `1.0` | 10x | 極近距離での力の発散を防止し、安定した分散を確保 |
| `initialRadius` 係数 | `sqrt(n) * 3.0` | `sqrt(n) * 8.0` | 2.7x | 初期配置を広げ、反復計算の収束を加速 |
| `layoutIterations` | `200` | `300` | 1.5x | 強い反発力の収束に十分な反復回数を確保 |

**反発力/引力比の推移**: 3,000 ([[000351|ADR-351]]) → 500,000（本変更）。約167倍の比率増加により、共起関係のあるタグは他よりは近いが、重ならない程度の距離を保つ。

### 2. `<HTML>` コンポーネントによる常時ラベル + ホバーツールチップ

Threlte の `<HTML>` コンポーネント (`@threlte/extras`) を採用し、3D空間上の DOM オーバーレイとしてラベルを描画する。

**常時ラベル**:
- 各惑星の上方 (`radius * 1.3`) に配置
- 白テキスト + 黒テキストシャドウ（宇宙空間の黒背景でも読める）
- `labelFontSize` は `articleCount` に比例（11px〜16px、大きい惑星 = 大きいラベル）
- `sprite` prop でカメラ方向に自動回転（ビルボード動作）

**ホバーツールチップ**:
- 常時ラベルのさらに上方に配置
- `tagName` + `articleCount` の2行構成
- 暗い半透明背景 (`rgba(10,10,30,0.9)`) + シアンボーダー
- 既存の `T.Sprite` + `T.SpriteMaterial`（テキストなし）を完全に置き換え

全ラベルに `pointerEvents="none"` を設定し、球体のクリック・ホバーを妨げない設計とした。

### 3. シーンパラメータの調整

`positionBound = 100` に拡大された座標空間に合わせ、フロントエンドのカメラ・フォグを調整:

| パラメータ | Before | After | 理由 |
|-----------|--------|-------|------|
| `cloudRadius` 最小値 | `30` | `60` | 広がった空間でのカメラ距離確保 |
| FogExp2 density | `0.003` | `0.002` | 広い空間での遠方タグの視認性確保 |
| OrbitControls `maxDistance` | `cloudRadius * 4` | `cloudRadius * 5` | ズームアウト範囲を拡大 |

### 4. HUD ラベル重なりの修正

Threlte の `<HTML>` コンポーネントは Canvas コンテナ内に絶対配置の DOM 要素を生成するため、固定位置の HUD サイドバー (`z-index: 50`) と同じスタッキングコンテキストに属し、ラベルが HUD の上に描画される問題が発生した。

**対策**: `TagVerseScene.svelte` の Canvas ラッパー `<div>` に `isolation: isolate; z-index: 0; overflow: hidden;` を設定し、独立したスタッキングコンテキストを生成。これにより内部の `<HTML>` 要素は全て `z-index: 0` のコンテキスト内に封じ込められ、HUD (`z-index: 50`) の下に描画される。

## RESULTS, EFFECTS

### PROS

- **惑星の均一分散** — 反発力/引力比を167倍に強化し、200タグが3D空間全体に分散。中央の密集が解消
- **即座のタグ識別** — 全惑星に常時ラベルが表示され、ホバーなしでタグ名を判読可能
- **ホバー情報の充実** — ツールチップで記事数も確認でき、惑星の相対的な重要度を数値で把握
- **HUD との共存** — スタッキングコンテキスト分離により、サイドバー表示時にラベルが干渉しない
- **後方互換性** — フォースグラフの入出力インターフェースは変更なし。定数チューニングのみ

### CONS, TRADEOFF

- **200個の `<HTML>` DOM 要素** — 常時ラベルにより各タグに1つの DOM 要素が生成される（ホバー時は2つ）。各要素はフレーム毎に3D→2D座標変換を実行する。現状の200タグでは問題ないが、大幅に増やす場合は上位N件のみ常時表示にするなどの最適化が必要
- **共起関係の視覚的希薄化** — 反発力を大幅に強化したため、共起関係のあるタグ対の「近さ」が以前ほど顕著ではない。エッジ描画の導入で補完可能だが、視覚的ノイズとのトレードオフ
- **反復回数の増加** — 200→300回への増加でレイアウト計算のCPUコストが50%増。ただし200ノードでの実測は数十ミリ秒程度のため実用上の問題なし
