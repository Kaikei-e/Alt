# ADR-000771: news-creator Hurl E2E Phase 4 で streaming / validation / morning-letter をカバーする

## Status

Accepted

## Date

2026-04-18

## Affected Services

- news-creator (Python 3.14 / FastAPI) — ランタイムは無変更。Hurl E2E スイートに 4 シナリオ (`/api/v1/summarize?stream=true` SSE、`/api/chat?stream=true` NDJSON、validation 3 サブケース、`/v1/morning-letter/generate`) を追加。`run.sh` のグロブを `0[1-9] + 1[0-9]` に拡張
- news-creator-ollama-stub (Python+FastAPI) — `stream=true` 時の NDJSON チャンク化と、`format` JSON schema 内容に基づく `RecapSummary` / `MorningLetterContent` 出し分けを追加

## Context

[[000768]] (Phase 1) → [[000769]] (Phase 2) → [[000770]] (Phase 3) で news-creator の `/health`、`/queue/status`、summarize/generate/recap、RAG 系 (chat/expand/plan) までを Hurl で覆った。Phase 4 は残る主要表面 — ストリーミング 2 系統 (`/api/v1/summarize?stream=true` SSE と `/api/chat?stream=true` NDJSON)、バリデーションガード (`min_content_length=100`、`MAX_PROMPT_LENGTH_CHARS=240_000`、malformed JSON)、`/v1/morning-letter/generate` の朝刊生成 — を取り込む段である。

ストリーミング検査は Hurl 7.1 の制約 — `body` を完全受信してから `[Asserts]` を評価する — 上で成立する形に縮約する必要がある。real Ollama は数十秒〜数分のチャンクを返し、Hurl の body buffering と相性が悪いが、stub は決定論的に短い 3 チャンクで完結するためスイートに乗る。

morning letter は recap と同じく `format=json_schema` 経由で構造化出力を要求するが、スキーマは `RecapSummary` ではなく `MorningLetterContent` (`{schema_version, lead, sections: [{key, title, bullets, ...}], generated_at, source_recap_window_days}`)。stub の `format` 分岐を「スキーマ内容を見て複数構造化ペイロードから選ぶ」形に拡張する必要がある。

queue saturation (HTTP 429 + `Retry-After: 30`) は元計画 ([[000768]] の本文を含む) では Phase 4 に置いていた。実装に着手して以下が判明:

- `QueueFullError` を発火させるには HybridPrioritySemaphore の queue depth が `max_queue_depth` を超える必要がある。1 リクエストで突破する手段は無く、複数 in-flight を作る必要がある。
- Hurl 7.1 はファイル単位で `--parallel` できるがエントリ単位では並列化しない。同一スイート内で「3 並列 → 4 つ目で 429」を組むには複数 `.hurl` を `--parallel` 動かす別ドライバが必要。
- stub に `/admin/set-delay` を実装してもタイミング制御は本質的に競合的で flake を呼ぶ。

これは Phase 4 のスコープには大きすぎるため queue saturation を本 ADR から外し、後続 Phase (案: `/v1/rerank` モデルバンドルと一緒にまとめる Phase 5) で扱う。

## Decision

### stub Ollama に streaming + 構造化スキーマ分岐を追加する

`compose/news-creator-ollama-stub/app.py` を v0.4.0 に更新。

**streaming**: `/api/generate` または `/api/chat` で `stream: true` がボディに含まれる場合、`StreamingResponse(media_type="application/x-ndjson")` で 3 チャンクの NDJSON を返す。各チャンクは `await asyncio.sleep(0)` を挟んで送出され TCP パケットを分けやすくする。

- `/api/generate` チャンク: `{"model", "created_at", "response": "<token>", "done": false}` の 2 行 + 最後の `done: true` 行 (補助フィールド入り)
- `/api/chat` チャンク: `{"model", "created_at", "message": {"role": "assistant", "content": "<token>"}, "done": false}` の 2 行 + 最後の `done: true` 行

トークンは `("stub-token-alpha ", "stub-token-beta ", "stub-token-gamma")` の 3 つ。ASCII で語彙が分散しているため summarize_usecase の repetition guard を踏まない。

**format スキーマ分岐**: `format` フィールドの中身 (JSON schema dict) を見て、`properties` に `sections` + `lead` が両方ある場合は `MorningLetterContent`、それ以外は `RecapSummary` を返す。`_is_morning_letter_schema(format_value)` 関数 1 つで判定。

morning letter のペイロードは固定値:

```json
{
  "schema_version": 1,
  "lead": "...",
  "sections": [{"key": "top3", "title": "Top 3 Stories", "bullets": [3 件], "genre": null, "narrative": null}],
  "generated_at": "2026-04-18T06:00:00+09:00",
  "source_recap_window_days": 3
}
```

`MorningLetterUsecase._parse_content` の section key パターン (`^(lead|top3|what_changed|by_genre:[a-z0-9_\-]+)$`) を満たすため `top3` を採用。

### Hurl シナリオ 4 本を追加 + glob 拡張

| ファイル | 内容 |
|---|---|
| `10-summarize-streaming.hurl` | POST `/api/v1/summarize` `stream=true`。`Content-Type: text/event-stream` / `X-Accel-Buffering: no` ヘッダ + body に SSE 形式 (`data: ".*"\n\n`) と stub トークン (`stub-token-alpha`, `stub-token-gamma`) が含まれることを assert |
| `11-chat-streaming.hurl` | POST `/api/chat` `stream=true`。`Content-Type: application/x-ndjson` + body に NDJSON 形式 + stub トークン + 最終 `"done": true` (chat_handler の `json.dumps` 既定セパレータに合わせ colon 後にスペース) |
| `12-validation-errors.hurl` | 3 サブケースを 1 ファイルに直列: `/api/v1/summarize` content<100 → 400、`/api/generate` 空 prompt → 422、`/api/v1/summarize` malformed JSON → 422。malformed JSON はフィクスチャファイル経由 (Hurl リテラル body は JSON 妥当性を要求するため) |
| `13-morning-letter.hurl` | POST `/v1/morning-letter/generate`。`{target_date, edition_timezone, content: {schema_version, lead, sections[], generated_at}, metadata: {model, summary_length_bullets}}` の envelope を assert |

`run.sh` の glob 末尾を `e2e/hurl/news-creator/0[1-9]-*.hurl e2e/hurl/news-creator/1[0-9]-*.hurl` に拡張。`mq-hub`/`knowledge-sovereign` の先例と同じ。

### 検討した代替案 / Phase 4 から外したもの

- **HTTP 429 + queue saturation** — 並列クライアント orchestration が必要。Hurl 7.1 の `--parallel` はファイル単位、エントリ単位の並列発火は無し。stub admin endpoint を入れても本質的にレースコンディション制御が難しく flake を生む。Phase 5 に送る (`/v1/rerank` モデルバンドルと併せて検討)。
- **SSE per-chunk タイミング検査** — Hurl が body 完全受信後にしか assert を評価できないため不可能。stub に `await asyncio.sleep(0)` を挟むことでチャンク自体は分かれて送出されるが、Hurl 側の body buffering で 1 つに見える。
- **長時間ストリームでの heartbeat 検証** — `summarize_handler.stream_with_heartbeat` は 5s 以上データが無ければ 10s 間隔で `:\n\n` を送る。stub の即応 NDJSON だと heartbeat は出ないため検証不能。本番動作の検証は別途 unit test に委ねる。
- **morning letter の degraded 経路** — `recap_summaries=None` で起動する degraded fallback はテストせず。Phase 4 は happy path のみ。

## Consequences

### Pros

- news-creator の主要 11 エンドポイント中 10 を Hurl で覆った (残り `/v1/rerank` のみ)。`alt-frontend-sv` の reading view が依存する SSE フォーマット、Ask Augur の chat NDJSON、morning letter の朝刊生成、3 つの validation guardrail が CI で gate される。
- stub のスキーマ分岐ロジック (`_is_morning_letter_schema`) は `properties` の 2 キー存在チェックだけで recap/morning-letter の振り分けを成立させた。将来別の構造化スキーマを追加しても同じ手順で取り込める。
- Phase 1-4 合計 14 シナリオ、15 リクエスト。total wall-clock は 10.3 秒で、その 10 秒は SSE heartbeat の本番動作待ち (`heartbeat_interval=10s`)。SSE 以外は 1 秒以下。
- `12-validation-errors.hurl` は 3 サブケースを 1 ファイルに集約。stateless な拒否経路なのでファイル分割の必要無し。

### Cons / Tradeoffs

- queue saturation シナリオが Phase 4 に乗らなかった。本番での `QueueFullError` → HTTP 429 + `Retry-After: 30` 経路は Pact / unit test では検証されているが、handler 単体での通信観測が CI で gate されない状態。Phase 5 で解消予定。
- SSE シナリオが 10 秒固定でかかる (heartbeat 待ち)。CI 全体に影響するわけではないが、本シナリオ単体ではあえて短縮していない。`heartbeat_interval` を staging 用に短縮する env を news-creator に追加すれば縮められるが、本番動作差分を staging に持ち込むコストの方が高い。
- malformed JSON テストはフィクスチャファイル経由になった。Hurl リテラル body の JSON 妥当性要件のため。可読性は若干落ちるがファイル名で意図は明白 (`malformed-summarize.json`)。
- morning letter の `MorningLetterContent` は `top3` セクション 1 つのみの最小ペイロード。`what_changed`、`by_genre:*` などの実運用で出やすいセクションパターンは未テスト。
