# ADR-000783: alt-deploy pull ステップに親サービス recreate 追従の sidecar cascade 再作成を追加する

## Status

Accepted

## Date

2026-04-19

## Affected Services

- **alt-deploy** (別リポジトリ `Kaikei-e/alt-deploy`, self-hosted deploy runner) — pull-deploy ワークフローの image pull + `docker compose up -d` の直後に、今回 recreate された親サービスに紐づく netns-sharing sidecar をすべて `--no-deps --force-recreate` で再作成する step を追加
- **compose** — `compose/pki.yaml` の `depends_on.<parent>.restart: true` 記述は維持する（compose-native にも最大限の追従を残すため）。本 ADR 適用後、force-recreate 経路の保証は alt-deploy 側が担う

## Context

[[000782]] で記録した通り、2026-04-19 09:37 JST に `docker compose up -d` が acolyte-orchestrator を image 差分で `--force-recreate` し、`pki-agent-acolyte-orchestrator` が旧 netns に取り残された。これは [[PM-2026-030]] → [[PM-2026-031]] → [[000782]] と同じパターンの **3 度目の発火**である。

### compose-native 対応が足りない事実

[[000759]] で `compose/pki.yaml` の `pki-agent-acolyte-orchestrator` と `pki-agent-tag-generator` に `depends_on.<parent>.restart: true` を追加し、compose-native で cascade を実現する方針を採用していた。本 ADR 時点で当該記述は compose/pki.yaml:283-286 に残存している:

```yaml
depends_on:
  acolyte-orchestrator:
    condition: service_healthy
    restart: true
```

しかし 2026-04-19 の deploy では cascade が走らなかった。理由:

- compose spec の `depends_on.<parent>.restart: true` は **親が restart (再起動) された時に dependents を restart する**意味であり、**親が force-recreate (destroy + create)** された時は発火しない
- `docker compose up -d --pull always` 相当の挙動では image 差分検知時に親を recreate するが、dependents は「定義変更なし」と判定されて据え置きになる
- 結果、sidecar は古い parent container Id を `HostConfig.NetworkMode` に抱えたまま残り、新 netns への追従が起こらない

### 影響範囲の俯瞰

`grep -n 'network_mode: "service:' compose/pki.yaml` で列挙できる sidecar は複数ある（acolyte-orchestrator, tag-generator, recap-subworker など）。同パターンは deploy のたびに再現しうるため、sidecar ごとに ad-hoc 対応するのではなく**汎用 step として alt-deploy 側に据える**べきである。

## Decision

**`Kaikei-e/alt-deploy` の pull-deploy ワークフローに、image 差分で recreate された親サービスに紐づく netns-sharing sidecar を一括で `--no-deps --force-recreate` する step を追加する。** 採用構成:

1. **`scripts/cascade-recreate-netns-sidecars.py` を alt-deploy リポジトリに新設**する。入力は今回 rolled な service 名リスト（`compose-image-audit.py` と同じ仕組みで取得可能）。処理は compose YAML ツリーを読み、`network_mode: service:<rolled-parent>` を持つサービスを列挙し、`docker compose -f <compose-file> -p alt up -d --no-deps --force-recreate <sidecar>` を 1 件ずつ実行する
2. **pull-deploy ワークフローで `docker compose pull` + `up -d` の直後に上記 helper を呼ぶ**。helper が 0 件を返せば no-op なので、通常 deploy でのオーバーヘッドは数百 ms
3. **helper の停止条件は `--no-deps --force-recreate` の per-sidecar 実行の通常終了**。失敗は deploy 全体を失敗扱いにして alt-deploy のエラー経路に乗せる（`set -e` 相当）
4. **対象 compose ファイルは `compose/compose.yaml` と `compose/pki.yaml` の両方**を走査する。将来 `network_mode: service:*` を持つサービスが compose/rag.yaml など他フラグメントに現れた場合も拾えるようワイルドカード include する
5. **compose/pki.yaml 側の `depends_on.<parent>.restart: true` は documentation としては残す**。compose spec が force-recreate でも cascade するように改善された場合に備え、two-path defence-in-depth を維持する
6. **alt-deploy 側の実装は Kaikei-e/alt-deploy の PR で行い、本 Alt リポジトリからは ADR と compose コメントで参照のみを提供する**

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| compose spec の upstream にバグレポート/PR を出し、`depends_on.restart: true` が force-recreate でも cascade するようにする | 時間軸が合わない。compose 仕様変更は数ヶ月〜年単位。その間に本番で再発が続く |
| 全 deploy で `docker compose up -d --force-recreate` を無条件に打つ | 関係ない全サービスも再作成される。副作用が大きく、通常の deploy 時間と可用性に悪影響 |
| sidecar 側に「親 container Id が変わったら自分を exit する」loop を埋める | [[000784]] で採用する listener self-probe と重複する。本 ADR の範囲外 |
| compose-native を諦めて sidecar を「同じ pod」的に 1 コンテナに押し込める | pki-agent は Go 実装で orchestrator は Python。ランタイム統合は別方向の改修であり、short-term fix として過剰 |
| alt-deploy 側ではなく、Alt リポジトリ内の `scripts/` に置いて SRE が手動実行する | 手動は忘れる。今回の 3 度目発火が事実として証明している。deploy 経路に**強制的に**乗せる必要がある |
| sidecar の `restart` policy を `always` + liveness で代替する | `always` でも container id が変わらなければ自身は古い netns を参照し続ける。根本解決にならない |

### 範囲外 (今回やらないこと)

- pki-agent の listener self-probe + fail-fast — [[000784]] で別 ADR
- 即時復旧手順の SRE runbook 化 — 本 ADR の cascade step が landed した段階で runbook は不要になる想定。必要であれば別タスクで docs/runbooks/ に追加
- compose spec 挙動への upstream 提案 — 長期アクションとして保留

## Consequences

### Pros

- **同パターンの再発を deploy 経路で機械的に止められる**。`compose-image-audit.py` と同じ位置づけで、deploy が成功した時点で sidecar 孤立が発生していないことが保証される
- **sidecar ごとに compose YAML を書き換える必要がない**。compose/pki.yaml の `network_mode: service:<name>` を持つサービスは helper が自動で列挙する。将来 `pki-agent-X` 系が増えても追加作業ゼロ
- **`depends_on.<parent>.restart: true` との two-path 防御が残る**。compose spec が改善されれば alt-deploy helper は no-op に近づき、自然と負担が消える
- **障害時の手順が shell 2 行から `deploy のログを見る` に縮まる**。[[000782]] の手動復旧コマンドは、本 ADR が landed した後は事故調査用にしか使わなくなる
- **Alt リポジトリへの変更は最小限**（ADR とコメントのみ）。deploy 側のロジック集中が [[000776]] の方針と整合する

### Cons / Tradeoffs

- **alt-deploy repository への PR が必要で、単一リポジトリで完結しない**。レビューと反映の導線が 2 経路になる。Alt 側の deploy 切り替え記録 ([[000776]]) と整合を取る運用が要る
- **helper 実装に YAML 走査 + docker CLI 呼び出しのコードが追加される**。過去のシェル版 (`_deploy_lib.sh`、[[000757]]) が c2quay 移行 ([[000758]]) で消えた経緯があるため、Python 実装も migration で失われないよう services.yaml SoT ([[000777]] 系) と同じ CI gate を持たせる
- **初期実装時は staging stack (`compose/compose.staging.yaml` 経路) での再現テストが必要**。親の image 差分を artificial に作って cascade が発火するかを検証する手順が未整備
- **helper 失敗は deploy を止める**。semantics としては healthy だが、helper 自身のバグで deploy が blocker になる可能性がある。初期運用は `--dry-run` モードを併走させて安全側に倒す
- **pki-agent 自体の silent failure 性（healthcheck 嘘問題）は本 ADR では解消しない**。[[000784]] が landed するまで cert 鮮度だけ見ている事実は残る
