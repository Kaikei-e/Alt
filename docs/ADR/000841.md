# ADR-000841: Knowledge Loop の Why を event payload 由来の構造化 evidence に昇格し Continue/Changed/Review を専用 Surface plane として描画する

## Status

Accepted

## Date

2026-04-24

## Affected Services

- **proto** — `proto/alt/knowledge/loop/v1/knowledge_loop.proto` に `GetKnowledgeLoopResponse.bucket_entries = 7` (repeated `KnowledgeLoopEntry`) を additive 追加。`buf breaking` pass。既存 RPC / enum / message は無変更。
- **alt-backend (Go)** — `usecase/knowledge_loop_usecase/why_enricher.go` (新規) を projector に配線。`validator.go` の `WhyMappingVersion` を `1 → 2` に bump。`get_knowledge_loop_usecase.go` が NOW 以外の 3 バケツを per-bucket 5 件で追加取得し `GetKnowledgeLoopResult.BucketEntries` として返す。handler が bucket_entries を proto response に詰める。
- **alt-frontend-sv (SvelteKit)** — `src/lib/components/knowledge-loop/` に `ContinueStream.svelte` / `ChangedDiffCard.svelte` / `ReviewDock.svelte` / `loop-priority-labels.ts` を追加。`src/lib/connect/knowledge_loop.ts` の `KnowledgeLoopResult` に `bucketEntries` を追加。`src/routes/(app)/loop/+page.svelte` が count-only の bucket-index を 3 専用 Surface plane に置換（`bucketEntries` 空時は legacy index が fallback）。
- **alt-butterfly-facade / rag-orchestrator** — proto 生成物のみ再生成。コード変更なし。

## Context

[[000840]] で Knowledge Loop の event append と projection-diff stream が成立し、Loop は「永続的な行動履歴」として機能し始めた。しかし `review.md` (2026-04-24) が指摘した Priority 3・4 の 2 つの穴は、Loop が「使う理由のある中核 UI」になるために避けて通れないものだった。

1. **Orient の語彙が浅い (P3)** — projector の `shortEventWhy` は 7 種の固定文言のみを返しており、`WhyPayload.evidence_refs` は常に空だった。その結果、`/loop` 上のすべての tile が "Surfaced in your home feed." 等の定型句を繰り返し、Knowledge Loop の canonical contract §11 が保証する「なぜ今これが出ているか」の構造化シグナル (summary version / tag set / article / conversation / supersede chain) が UI まで届いていなかった。Why の貧弱さは Orient 以降の意思決定を薄くし、Decide / Act の CTA も結果として汎用的なものに留まってしまっていた。
2. **Continue / Changed / Review が index 扱い (P4)** — 非 NOW バケツは `foregroundEntries` とは別経路で projection されていたが、`/loop/+page.svelte` では件数バッジのみを描画する "Also waiting" ナビに落とし込まれていた。canonical contract §6 は Continue を「継続が必要」Changed を「知識差分を前面化」Review を「再検討」と明確に位置づけているのに、UI 上はどれも同じ count-index でしか見えず、Changed という Alt 固有の価値が可視化されていなかった。

両者は独立した改善に見えるが、同じ契約断絶を別角度から指している。Why が構造化されていないから Continue / Changed / Review を「どう表現するか」が決まらず、逆に plane を作っても見せる Why が薄ければ plane を作る意味が薄い。この 2 点を同一 PR 群としてまとめるのは、canonical contract §11 (Why as first-class) と §6 (Surface bucket 独立軸) が互いに補強し合うためである。

計測面では、PR-L5 / PR-L6 で event append と stream が入ったことで、Orient / Decide の UI 反応速度は 5 秒以内に収まるようになっていた。残る課題は「入ってきた event からどれだけ意味のある Why を抽出できるか」と「抽出した Why を Continue / Changed / Review の固有 UI に流せるか」の 2 点に絞り込まれていた。

## Decision

Why enrichment と Surface plane 昇格を一体で採用することを決定する。具体的には以下 8 点を固定する。

1. **Why enricher の新設 (PR-L7)** — `usecase/knowledge_loop_usecase/why_enricher.go` に純関数 `EnrichWhyFromEvent(ev) → EnrichedWhy{Kind, Text, EvidenceRefs}` を置く。event payload のみを読み、`time.Now()` や latest projection state に依存しない reproject-safe な写像であることを設計不変条件として固定する。

2. **event 種別ごとの `WhyKind` と evidence_refs の命名規約**:
   - `SummaryVersionCreated` → `source_why` + "New summary: <article_title>" + refs[`summary_version_id`, `article_id`, `tag_set_version_id`]
   - `HomeItemsSeen` → `source_why` + "Surfaced in your home feed." + refs[summary/tag version, `article_id`]
   - `HomeItemAsked` → `source_why` + "You asked about this item." + refs[`conversation_id`, `article_id`]
   - `HomeItemOpened` → `recall_why` + "You opened this before." + refs[`event_id` (安定 anchor), `article_id`]
   - `HomeItem/SummarySuperseded` → `change_why` + "A newer version is available." + refs[`previous_summary`, `new_summary`, `previous_entry`, `new_entry`]
   - `HomeItemDismissed` → `source_why` + "You dismissed this item."
   - unknown → `source_why` + "Surfaced from a recent event." (fallback)

3. **`WhyMappingVersion` を v2 に bump** — `validator.go` の定数を `1 → 2` に上げ、`docs/runbooks/knowledge-loop-reproject.md` の version history テーブルに v1 (fixed strings) / v2 (structured evidence_refs) を追加する。bump は既存 entries の why_text / evidence_refs を更新するため、full reproject を必須化する既存の runbook トリガに乗る。

4. **security hardening in the enricher** — text は `stripAngleSpans` で HTML tag を defense-in-depth で除去し、UTF-8 boundary-safe な 512-byte truncate を適用する。evidence_refs は `appendRefIfPresent` / `boundEvidence` で 8 件上限を enforce する (canonical contract §3.10)。XSS の主防御は proto layer と Svelte のエスケープ補間にあるが、enricher も同じ契約を独立に守ることで層化する。

5. **proto additive 拡張 (PR-L8)** — `GetKnowledgeLoopResponse` に field `bucket_entries = 7` を追加する。非 NOW バケツ (Continue / Changed / Review) の entry 行を carry し、各 entry が `surface_bucket` フィールドでどの plane に属するかを自己記述する。field 7 は additive で、古い client は空配列として扱うので legacy bucket-index へ fallback する互換を保つ。

6. **`GetKnowledgeLoopUsecase` の per-bucket fetch** — Execute 内で `SurfaceContinue` / `SurfaceChanged` / `SurfaceReview` を `otherBucketLimitPerBucket = 5` で個別に fetch し `BucketEntries` に concatenate する。per-bucket 上限は canonical contract §6.1 の foreground 上限を踏襲し、1 レスポンスで最大 15 entries (3 バケツ × 5) + 3 foreground entries に制限する。`projection_seq_hiwater` の計算にこれらも含める。

7. **3 Surface plane component を Alt-Paper 言語で設計** — `ContinueStream` は editorial timeline 型 (mono freshness stamp + serif title + priority badge)、`ChangedDiffCard` は diptych (THEN / NOW 二連 + 1px 垂直 rule + Confirm CTA)、`ReviewDock` は低密度 mono list (dotted border + open handoff)。各 component は Alt-Paper の共通語彙 (serif display / mono meta / 1px rules / no shadows) を UI 責務に合わせて適用し、Acolyte 2-rule ヘッダーや masthead をコピーしない ([[feedback_acolyte_header_not_alt_paper]] / [[feedback_alt_paper_shared_language]])。

8. **accessibility と reduced motion の契約準拠** — `loop-priority-labels.ts` に canonical LoopPriority → aria-label (critical / continuing / confirm / reference) の mapping を置き、各 plane の entry に `aria-label` で配信する (contract §13)。`aria-description` は ARIA 1.3 の仕様上有効だが Svelte 5 の HTMLProps 型定義に未反映のため `aria-label` を採用した。translateZ / parallax primitives は 3 component すべてで禁止し、`SurfacePlaneComponents.source.spec.ts` に regex guard として pin。saturate filter は `LoopSurfacePlane` 側で既存の reduced-motion 対応を継承する。

### 検討した代替案

- **Why enricher を projector 外 (usecase 別コンポーネント) に置き projector 内から DI で呼ぶ**: 設計上きれいだが、enricher は event payload のみを扱う純関数で、projector 以外のコンシューマ (stream classifier / UI) から呼ばれる予定もない。projector 近傍の package に直接配置するほうが reproject-safety の責務境界と合致するため採用。
- **`WhyPayload.text` に Markdown を許容する**: rich な表現ができるが canonical contract §3.10 の plain text 契約を破る。sanitizer の複雑性と XSS risk がコストに見合わないため却下。v2 でも plain text を堅持。
- **`HomeItemOpened` の evidence_refs に前回の `HomeItemOpened.event_id` を入れる**: 「真の last opened」を anchor できるが projector に `GetLastOpenEventForUser` の lookup が必要になり reproject-safe を破る (Case 1 一般化違反)。今回の event_id を anchor にすることで replay 決定論を維持。将来、別 projector が `last_opened_seq` を別テーブルに維持するなら、そちらから evidence_refs に流し込む形に拡張する。
- **`bucket_entries` を単一 flat repeated ではなく `continue_entries` / `changed_entries` / `review_entries` の 3 field に分ける**: 型レベルで明示的だが proto field 番号を 3 消費し、将来のバケツ追加 (例: Queued) で breaking が避けられない可能性がある。entry 自体が `surface_bucket` を持つため flat 一本の方が additive かつ scalability で有利と判断。
- **`GetKnowledgeLoopUsecase` で 1 クエリに集約 (`SurfaceBucket IS NOT NULL AND bucket != 'now'` 的な)**: sovereign 側の port 契約が `SurfaceBucket *SurfaceBucket` 単一フィルタを取るため、現行インタフェース維持のためバケツごとに 3 回呼ぶ形を採用。クエリ数増加のコストは projection key で小さいので許容。sovereign RPC の拡張は別トラックの課題にする。
- **3 component を 1 `BucketPlane.svelte` で共通化して props で切り替え**: コード量は減るが canonical contract §6 で 3 バケツは「独立軸」と明文化されている。共通化するとタイムライン的 Continue と diptych 型 Changed が同じ枠に押し込まれ、review.md が指摘した「index 扱い」の再発を招く。意図的に 3 component に分離。
- **`aria-description` を Svelte 5 の型補強で使う**: TypeScript 側で declaration merging を書けば通るが、Svelte の型定義が将来更新された際に二重定義で壊れる可能性が高い。`aria-label` は同じ情報量を screen reader に届けられ、かつ型定義に既に存在するため採用。

## Consequences

### Pros

- **Orient が実用レベルに達する**: `why_text` が event payload 由来の固有情報 (summary title, supersede chain, conversation ref) を carry するようになり、`/loop` tile の「なぜ今これが出ているか」がユーザに具体的に読めるようになる。`WhyPayload.evidence_refs` が空でなくなることで、UI 側で deep link や diff badge を追加する余地が一気に開ける。
- **Changed バケツが Alt 固有の差別化機能として立ち上がる**: 以前の理解と新しい情報を THEN / NOW で並置する diptych は、index バッジでは代替できない認知モデル更新の UI になる。`KnowledgeLoopActed(trigger=confirm)` の emit 経路が Changed Confirm から整い、将来の知識差分メトリクスの足場になる。
- **Continue / Review の role 分離**: Continue は「まだ動いている思考」、Review は「peripheral な再検討候補」と役割が視覚言語レベルで分離する。timeline vs low-density dock の対比が canonical contract §6.2 / §6.4 の区別を裏打ちする。
- **reproject-safe の厳守**: enricher は純関数、`HomeItemOpened` evidence も event_id anchor で replay 決定論を維持。`WhyMappingVersion` bump で既存 entries を v2 形式に再生成する経路が runbook に明記されているため、migration が追跡可能。
- **accessibility baseline の底上げ**: canonical `LoopPriority` 文言が aria-label としてすべての plane の entry に届く。視覚的 depth に依存しない情報経路が成立する。
- **rolling deploy 互換を維持**: DB migration なし、proto additive、buf breaking pass、新 compose service なし。既存 deploy 経路 (`git push origin main` → `dispatch-deploy.yaml` → `Kaikei-e/alt-deploy`) にそのまま乗る。

### Cons / Tradeoffs

- **`GetKnowledgeLoop` のクエリ数が 1 → 4 に増える**: sovereign への `GetKnowledgeLoopEntries` 呼び出しが NOW + 3 バケツ = 4 回になる。応答キーが projection indexed なのでクエリ自体は軽いが、stream poll と重なる時間帯の負荷は監視対象。将来 sovereign 側に「per-bucket 多重 fetch」を束ねる RPC を新設するなら別 ADR で。
- **`WhyMappingVersion` v2 への bump は full reproject を要求する**: 既存 entries の why_text / evidence_refs は v1 の固定文言のまま残り、runbook 手順に従って TRUNCATE → reproject するまで混在する。deploy 後すぐに reproject を回さないと UI 上で旧新が同居する期間が発生する。runbook の「bump → reproject」トリガは [[000840]] 時点で整備済みなので運用上は追加作業なし。
- **bucket_entries が一覧で最大 15 件になる**: entry あたりの payload が大きめ (why_text 最大 512B + evidence_refs 最大 8) なので、レスポンスサイズが NOW 単独より膨らむ。telemetry にレスポンスサイズを出して肥大化を監視する必要がある。
- **`aria-description` を諦めたことによる将来の言語変更コスト**: Svelte 5 の型定義が `aria-description` をサポートした時点で migrate 候補。loc は 3 component + テスト 1 ファイルのみなので置き換え影響は小さい。
- **Changed の THEN 列が payload 欠落時に "Previous version" という generic な fallback に落ちる**: enricher と projector が `change_summary.summary` を生成する経路はまだ薄く、v2 の現時点では supersede pointer の生表示になるケースが残る。PR-L7 の scope には含めず、change_summary を richer にする専用 PR で解消する前提。
- **Multi-tab Continue の同時操作ガードなし**: ContinueStream で同一 entry を 2 タブから同時に resume すると、BFF 側で `client_transition_id` が重複することで server dedupe されるだけで、UI は両方に "accepted" を表示する。client_transition_id が UUIDv7 で時刻付きなのでほぼ衝突しないが、UX 上の警告は未実装。
