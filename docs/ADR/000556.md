# ADR-000556: asyncio.wait による in-flight プリエンプションで RT ストリーミングの即時スロット取得を実現する

## Status

Accepted

## Date

2026-03-24

## Affected Services

- news-creator (Python/FastAPI)

## Context

[[000164]] で導入したアプリケーションレベルプリエンプションは、BE リクエストの `cancel_event` を set することで RT リクエストにセマフォスロットを譲る設計だった。しかし実運用で **RT ストリーミングリクエストがプリエンプション後もキューで待ち続ける** 問題が発生した。

### ログで確認した症状

```
17:35:12 — quality checker: Acquired semaphore (LOW), model=gemma3-4b-8k, prompt=15869
17:35:21 — RT streaming: "RT request blocked" → "Preempting BE" → cancel_event.set()
17:35:21 — RT: "RT request queued, queue_size: 1"  ← プリエンプション後もキューのまま
17:35:58 — 2nd RT: same BE task STILL running (45.5s) → queue_size: 2
```

GPU 使用率が動かず、StreamSummarize のストリーミングテキストが UI に表示されない。

### 根本原因

`_generate_with_cancellation()` は `cancel_event` を `driver.generate()` の **前後** でしかチェックしていなかった:

```python
if cancel_event.is_set():                       # ← 生成前チェック
    raise PreemptedException(...)

response = await self.driver.generate(payload)   # ← 30-90秒ブロック。中断不可！

if cancel_event.is_set():                       # ← 生成後チェック（手遅れ）
```

`driver.generate()` は Ollama への HTTP POST (`await session.post()`) であり、推論完了まで 30-90 秒ブロックする。この間 `cancel_event` は一切チェックされず、セマフォスロットが解放されない。

quality checker は `POST /api/generate` 経由で news-creator を呼び、`DistributingGateway.generate()` は常にローカル `OllamaGateway` に委譲する（リモートディスパッチは `hold_slot()` 経由のみ）。結果として quality checker の 15K トークンプロンプトがローカル GPU を長時間占有し、RT ストリーミングが一切処理されない。

## Decision

**`asyncio.wait(FIRST_COMPLETED)` パターンで `cancel_event.wait()` と `driver.generate()` を競争させ、in-flight の Ollama HTTP リクエストを即座に中断可能にする。**

```python
generate_task = asyncio.create_task(self.driver.generate(payload))
cancel_task = asyncio.create_task(cancel_event.wait())

done, _ = await asyncio.wait(
    {generate_task, cancel_task},
    return_when=asyncio.FIRST_COMPLETED,
)

if cancel_task in done:
    generate_task.cancel()       # → CancelledError → aiohttp closes connection
    raise PreemptedException(...)
```

### なぜこれが動くか

1. `asyncio.Task.cancel()` は aiohttp の `session.post()` 内で `CancelledError` を raise する
2. aiohttp は `CancelledError` を受けて HTTP 接続を close する
3. Ollama はクライアント接続の close を検知して推論を停止し、GPU リソースを即座に解放する
4. セマフォスロットは `OllamaGateway` の `finally` ブロックで解放される
5. キューで待っていた RT リクエストが即座にスロットを取得し、ストリーミングを開始する

### ベストプラクティスとの整合

| システム | プリエンプション手法 |
|----------|---------------------|
| vLLM | RT (online) 到着時に running BE (offline) を preempt、KV cache swap out |
| FastServe (NSDI'24) | skip-join MLFQ で BE を即座にプリエンプト |
| Ollama | クライアント接続 close で推論停止（公式動作） |
| Python asyncio | `asyncio.wait(FIRST_COMPLETED)` + `task.cancel()` が標準的な interruptible await パターン |

### 検討した代替案

| 案 | 評価 |
|---|---|
| 生成ループ内で定期的に cancel_event をチェック | Ollama の `/api/generate` は `stream: false` の場合、単一 HTTP レスポンスで返るためループが存在しない。`stream: true` に変更すればチャンク間でチェック可能だが、quality checker のコード変更が大きい |
| quality checker をリモートに逃がす | `DistributingGateway.generate()` がローカル固定のため、アーキテクチャ変更が必要。根本解決にならない |
| quality checker を RT 中に一時停止 | pre-processor 側の制御が必要で、サービス間結合が増える |
| **asyncio.wait パターン（採用）** | OllamaGateway 内の 1 メソッド変更のみ。Ollama の公式動作（接続 close で停止）を活用。既存の PreemptedException フローと完全互換 |

## Consequences

### Pros

- RT ストリーミングリクエストが BE の in-flight 推論を即座に中断し、セマフォスロットを取得できる
- GPU リソースが即座に解放される（Ollama が接続 close で推論停止）
- 既存の `PreemptedException` フローと完全互換（呼び出し元の変更不要）
- 1 メソッドの変更のみで侵襲性が低い

### Cons / Tradeoffs

- プリエンプトされた BE リクエストの推論結果は失われる。呼び出し元（quality checker、recap 等）は再試行が必要
- `asyncio.Task.cancel()` は cooperative cancellation であり、`driver.generate()` 内で `CancelledError` を握りつぶすコードがあると中断されない（現在の実装では問題なし）

### 既知の制約: 非ストリーミング時の Ollama GPU 解放

Ollama の `stream: false` モードでは、クライアント接続を close しても**サーバー側の推論は停止しない**（Ollama GitHub Issue #11889）。`stream: true` でのみ接続 close で推論停止が確認されている。

本修正は HTTP 接続を close することでセマフォスロットを即座に解放し、RT リクエストにスロットを譲ることはできる。しかし Ollama 内部の GPU 推論は非ストリーミングの場合バックグラウンドで完走する。`OLLAMA_NUM_PARALLEL=2` では Ollama 側で 1 スロットが無駄に使われ続ける可能性がある。

**将来の改善案**: BE の `driver.generate()` を `stream: true` に変更し、チャンク間で `cancel_event` をチェックする方式に移行する。これにより Ollama 側でも推論が即座に停止し、GPU リソースが完全に解放される。
