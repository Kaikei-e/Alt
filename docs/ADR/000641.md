# ADR-000641: Grafana リスタートループの根本原因修正と監視基盤の安定化強化

## Status

Accepted

## Date

2026-04-07

## Affected Services

- grafana (Docker Compose observability profile)

## Context

Grafana コンテナが `Restarting (1)` のリスタートループに陥り、監視基盤が完全に停止していた。この障害は繰り返し発生しており（[[000286]] でも P0 レベルのアラート全滅を経験）、構造的な脆弱性が残存していた。

`docker inspect` の結果:
- `OOMKilled: false` — メモリ不足ではない
- `ExitCode: 1` — アプリケーションエラーによる異常終了
- `Status: restarting` — `restart: unless-stopped` により無限リスタートループ

ログから特定した直接原因:
```
Error: ✗ alert rules: invalid alert rule: interval (15s) should be non-zero and divided exactly by scheduler interval: 10
```

`knowledge-home-slo-rules.yaml` の `interval: 15s` が Grafana のデフォルトスケジューラ interval（10秒）で割り切れないため、プロビジョニング時に致命的エラーとなりプロセスが終了していた（15 ÷ 10 = 1.5）。

加えて、以下の構造的問題も発見した:

| 問題 | リスク |
|------|--------|
| `mem_limit` 未設定 | OOM でホストごと巻き込みクラッシュの可能性 |
| `grafana.ini` 未作成、SQLite WAL 無効 | "database is locked" による断続的クラッシュ |
| 非推奨イメージ `grafana/grafana-oss` | v12.4.0 以降更新停止予定 |
| plugins provisioning ディレクトリ未作成 | 起動毎に error ログ出力 |
| `rask-system-rules.yaml` で未サポートの `eq` 演算子使用 | アラート評価が毎回失敗 |

## Decision

### 1. アラートルール interval を scheduler interval の倍数に修正する

`knowledge-home-slo-rules.yaml` の `interval: 15s` を `interval: 10s` に変更。SLO burn-rate アラートは `for:` によるバッファリングが効いているため、評価間隔 10 秒で十分。

### 2. `grafana.ini` を新規作成し SQLite WAL モードを有効化する

公式ドキュメントおよび GitHub Issue（grafana/grafana#65115）に基づき、WAL（Write-Ahead Logging）を有効化。WAL は同時アクセス時のトランザクション分離を維持しつつ "database is locked" エラーを防止する。NFS 等のリモートファイルシステムでは非互換だが、Alt はローカル Docker volume を使用しているため問題ない。

```ini
[database]
wal = true

[unified_alerting]
min_interval = 10s

[alerting]
enabled = false
```

`min_interval` を明示することで、今後のアラートルール追加時に不正な interval 設定を自己文書化する。

### 3. Docker Compose でリソース制限とイメージを更新する

- `mem_limit: 256m` を追加（Grafana のベースラインメモリは 100-200MB。他サービスとの整合: Prometheus 512m, cAdvisor 384m）
- イメージを `grafana/grafana-oss:11.4.0` → `grafana/grafana:11.4.0` に変更（v12.4.0 以降 `-oss` リポジトリは更新停止予定）
- `grafana.ini` をリードオンリーでボリュームマウント

### 4. `rask-system-rules.yaml` の未サポート演算子を修正する

`rask-service-no-logs` ルールの threshold 条件で `type: eq` を使用していたが、Grafana の threshold 式は `gt`, `lt`, `within_range`, `outside_range` のみサポート。`type: lt` + `params: [1]` に変更し、「ログ件数が 1 未満（= ゼロ）」という同等の条件を表現。

### 5. plugins provisioning ディレクトリを作成する

空の `observability/grafana/provisioning/plugins/` ディレクトリを作成し、起動時の `open /etc/grafana/provisioning/plugins: no such file or directory` エラーログを解消。

## Consequences

### Pros

- Grafana のリスタートループが解消し、監視基盤が復旧した
- SQLite WAL モードにより、アラート評価やダッシュボードアクセスの並行処理時のデータベースロック問題を事前に防止できる
- `mem_limit` によりメモリリーク時にホスト全体への影響を遮断できる
- `min_interval` の明示により、不正な interval 設定が入り込む原因を文書化できる
- 全アラートルールが正常に評価されるようになった

### Cons / Tradeoffs

- `mem_limit: 256m` はダッシュボード・アラートルールの増加に応じて調整が必要。現在の構成（ダッシュボード 7 枚、アラートグループ 5 個、データソース 2 個）では十分だが、大幅な増設時には引き上げが必要になる可能性がある
- SQLite は本質的に単一ファイルベースであり、高並行性シナリオでは PostgreSQL への移行が長期的には望ましい。現時点では WAL で十分と判断
