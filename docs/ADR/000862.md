# ADR-000862: Knowledge Loop に Review re-evaluation engine を追加し Recheck / Archive / Mark Reviewed の 3 アクションを KnowledgeLoopReviewed event に集約する

## Status

Accepted

## Date

2026-04-27

## Affected Services

- proto: `alt/knowledge/loop/v1/knowledge_loop.proto` の `TransitionTrigger` に `RECHECK = 6` / `ARCHIVE = 7` / `MARK_REVIEWED = 8` を additive 追加。`buf breaking` clean
- alt-backend: `domain.EventKnowledgeLoopReviewed = "knowledge_loop.reviewed.v1"` を追加。`ClassifyTransitionEvent` が `triggerRecheck` / `triggerArchive` / `triggerMarkReviewed` を `isReviewActionTrigger` で受け取り、`fromStage == toStage` を必須にして Reviewed event に route
- knowledge-sovereign: `projector.go` が `EventKnowledgeLoopReviewed` を allowlist に追加、`dismissStateForReviewedEvent(payload)` が pure 関数で `payload.action` ("recheck" / "archive" / "mark_reviewed") を `DismissState` (ACTIVE / COMPLETED) にマップして `PatchKnowledgeLoopEntryDismissState` を呼ぶ
- alt-frontend-sv: `useKnowledgeLoop.svelte.ts` に `reviewAction(entryKey, action)` メソッドを追加。`ReviewDock.svelte` に "Recheck" / "Mark Reviewed" / "Archive" の 3 ボタンを追加 (Archive のみ terracotta accent)。BFF (`/loop/transition/+server.ts`) の `SAME_STAGE_TRIGGERS` set に 3 トリガを追加して fromStage===toStage を edge で gate

## Context

`fb.md` §F のレビューで「Knowledge Loop の Review レーンが leftover bucket になっており、ユーザが entry に対して `Recheck` / `Archive` / `Mark Reviewed` 等の判断を返せない」と指摘された。canonical contract §6.4 の Review 定義は本来「再評価が必要な entry を集約する」ものだが、実装では `KnowledgeLoopDeferred` のような passive な操作しか提供されておらず、Review を能動的に消化する経路が無かった。

[[000861]] で `StalenessScore >= 2` を Review elevator に組み込み、Review が staleness 由来で増えやすくなった。これと対になる消化 / 再評価 path が必須になった。

設計上の制約は次のとおり。

- canonical contract §3 invariant 7 (single emission) — `/loop` UI は `knowledge_loop.*.v1` のみ emit。`HomeItem*` を absolutely 出してはいけない
- canonical contract §3 invariant 1 (reproject-safe) — 新 event の projector consumption は payload-only から決定的に dismiss_state を導く必要がある
- canonical contract §3 invariant 2 / §10 — `updated_at` / `projected_at` を一切露出しない
- 既存 `dismiss_state` ENUM は `active` / `deferred` / `dismissed` / `completed` 4 値で、Review の 3 アクションは `active` (recheck) と `completed` (archive / mark_reviewed) で表現可能。新 ENUM 値 / migration は不要

複数 event 種別を増やす vs 1 event に payload で sub-action を持たせる、の trade-off を検討した結果、本 ADR では 1 event (`KnowledgeLoopReviewed`) + payload `action` フィールドの方式を採用した。理由は decision 節で詳述する。

## Decision

`Recheck` / `Archive` / `Mark Reviewed` の 3 アクションを 1 つの新 event `KnowledgeLoopReviewed` に集約し、`payload.action` フィールドで sub-action を識別する。

採用した経路:

- **proto 拡張 (additive)** — `TransitionTrigger` enum に `RECHECK = 6` / `ARCHIVE = 7` / `MARK_REVIEWED = 8` を追加。`buf breaking` を main 比較で実行して clean (additive enum value は破壊的変更ではない)。`buf generate` で alt-backend / alt-butterfly-facade / rag-orchestrator / alt-frontend-sv の stub を再生成
- **alt-backend classifier** — `triggerRecheck` / `triggerArchive` / `triggerMarkReviewed` の 3 定数を追加し、`isReviewActionTrigger(t)` helper で 3 つを束ねる。`ClassifyTransitionEvent` が `triggerDefer` の隣で `isReviewActionTrigger` を分岐し、`fromStage == toStage` かつ `isCanonicalStage(fromStage)` のときのみ `EventKnowledgeLoopReviewed` を返す。それ以外は `ErrInvalidArgument`
- **projector consumption** — `Projector` の event allowlist に `EventKnowledgeLoopReviewed` を追加。session state upsert の後、entry に対して `dismissStateForReviewedEvent(payload)` を呼んで `PatchKnowledgeLoopEntryDismissState` を 1 回呼ぶ。pure 関数 `dismissStateForReviewedEvent`:
  - `recheck` → `DISMISS_STATE_ACTIVE` (entry を再アクティブ化、freshness は `event.OccurredAt` から既に更新)
  - `archive` → `DISMISS_STATE_COMPLETED`
  - `mark_reviewed` → `DISMISS_STATE_COMPLETED`
  - 不明な action / payload 不在 → `DISMISS_STATE_COMPLETED` (fail-closed; stuck row より閉じる方を選ぶ)
- **frontend** — `useKnowledgeLoop.svelte.ts` に `reviewAction(entryKey, action)` を追加。`archive` / `mark_reviewed` は `applyLocalDismiss` でローカル即除去、`recheck` は entry を残す。BFF の同 stage gate に 3 トリガを SAME_STAGE_TRIGGERS set で追加し edge で fromStage!==toStage を弾く
- **UI (ReviewDock)** — 各 row に Recheck / Mark Reviewed / Archive の 3 ボタンを追加。Archive のみ Alt-Paper の terracotta accent border で「破壊的選択」を明示。Open は既存

検討した代替案と却下理由:

- **3 つの別 event 種別 (`KnowledgeLoopRechecked` / `KnowledgeLoopArchived` / `KnowledgeLoopMarkReviewed`)**: 名前が長くなり projector の switch が肥大化する。3 アクションは「Review レーンに対する判断」という同一意味カテゴリなので、payload.action で区別するほうが意味論が綺麗。将来 4 つ目の Review action を増やすときも proto を触らずに済む
- **`KnowledgeLoopDeferred` を流用して action sub-field を分岐**: Deferred は canonical contract §8.2 で「soft dismiss / snooze」の意味が固定されている。Review re-evaluation という別意味を後付けで詰め込むと意味論崩壊。新 event で分けるほうが contract 上明快
- **migration で `dismiss_state` ENUM に `recheck_pending` 等を追加**: ENUM 追加は破壊的変更で migration 順序の cost が高い。既存 4 値で十分表現できるので migration 不要のほうを選んだ
- **frontend で recheck / archive / mark_reviewed を別エンドポイントに分割**: BFF に 3 routes 増えると edge 側の rate limit / auth が肥大化。同 transition path に乗せて trigger で識別するほうがシンプル

## Consequences

### Pros

- Review レーンが「能動的な再評価キュー」として機能する。Recheck で active 化、Mark Reviewed で confirm、Archive で永久消化、と判断を返せる
- canonical contract §3 invariant 7 (single emission) が保たれる。`/loop` UI からは `knowledge_loop.*.v1` のみ emit、`HomeItem*` は出さない
- `dismissStateForReviewedEvent` は pure 関数で payload bytes だけから決定。reproject 時に同じ event を replay すれば同じ dismiss_state パッチを発火 (canonical contract §3 invariant 1 維持)
- proto は additive 変更のみ。`buf breaking` clean、consumer 側は新 trigger を未使用のまま rolling deploy 可能
- migration 不要。既存 `dismiss_state` ENUM の `active` / `completed` 値で 3 action の終端状態を表現できる
- BFF / classifier / projector の各層に gate がある。`fromStage == toStage` の制約が edge / usecase / projector で 3 重に守られる

### Cons / Tradeoffs

- 1 event + payload.action 方式は projector consumption での switch が中央集権化する。将来 review 系 action が複雑化したときは sub-event に分割する別 ADR が必要になる可能性がある
- `recheck` 後に entry が `freshness_at = event.OccurredAt` で再活性化されるため、Surface Planner v2 の placement で NOW に戻る前提。Recheck したのに NOW に来ない、というケース (例: Augur link が無く、topic overlap も無い) は別途調査が必要 — 本 ADR は dismiss_state パッチに focus し、placement update は projector の通常経路に委ねる
- Archive と Mark Reviewed が両方 `DISMISS_STATE_COMPLETED` にマップされるため、DB だけ見ると区別がつかない。区別したい場合は `payload.action` を `knowledge_events` に取りに行く必要がある (event log は append-only でそこにある)
- 不明な action を fail-closed (`COMPLETED`) で扱うのは安全だが、上流が壊れた場合 silent に entry が消化される。`slog` で warn ログは出していないため、observability を後追いで強化したいなら別 ADR
- 3 ボタン配置で Review row の縦幅が増えた。ReviewDock は元々 low-density discovery list なので余白を許容しているが、モバイルでは 1 row 高さが伸びる
