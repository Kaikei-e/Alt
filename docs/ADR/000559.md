# ADR-000559: KnowledgeCard タグクリックによるタグ別記事一覧とサイドバーデフォルト閉じを実装する

## Status

Accepted

## Date

2026-03-24

## Affected Services

- alt-frontend-sv (SvelteKit) — 新規ルート追加、KnowledgeCard 改修、サイドバー改修

## Context

Knowledge Home の KnowledgeCard に表示されるタグチップをクリックしても何も起きない状態だった。ユーザーが「このタグの他の記事を見たい」と思っても、導線がなかった。

加えて、Desktop UI のサイドバーは全グループ（Feeds, Recap, Explore, Settings, Admin）がデフォルトで展開されており、縦に長くなっていた。頻繁に使わないグループまで常時展開される必要はない。

### 既存のタグ別記事取得基盤

Alt はすでに以下を持っていた。

- Connect-RPC の `FetchArticlesByTag` メソッド（`tagName` パラメータ対応済み）
- フロントエンドの `fetchArticlesByTag()` 関数（`articles.ts`）
- BFF エンドポイント `/api/articles/by-tag`
- Tag Trail のモバイル記事リストコンポーネント `TagArticleList.svelte`

不足していたのは「KnowledgeCard のタグから記事一覧に遷移するルートと UI」だけだった。

## Decision

**KnowledgeCard のタグチップをクリック可能なリンクにし、`/articles/by-tag?tag={name}` でタグ別記事一覧を表示する。Desktop/Mobile 両対応。サイドバーはデフォルト閉じとし、現在ページのグループのみ自動展開する。**

### 1. タグ別記事一覧ルート `/articles/by-tag`

`/feeds/` ではなく `/articles/` 配下にルートを設置した。理由は、タグはフィード（RSS ソース）ではなく記事に付与されるメタデータであり、概念的に `/articles/` が正しいため。

ルート構成:
- `+page.ts` — `ssr = false`（Connect-RPC transport を使用するため）
- `+page.svelte` — `useViewport` で Desktop/Mobile を分岐

Desktop コンポーネント（`DesktopTagArticlesScreen.svelte`）:
- タグ名をヘッダーに表示
- 記事リスト（タイトル、フィード名、日付、外部リンクアイコン）
- カーソルベースの Load More ページネーション
- 空状態・ローディング・エラー状態

Mobile コンポーネント（`MobileTagArticlesScreen.svelte`）:
- Tag Trail の `TagArticleList.svelte` のパターンを参考にした簡易版
- ホッピング機能（他のタグへの連鎖ナビゲーション）は含まない
- タッチ操作に適したサイズ（`min-h-[44px]` の Load More ボタン）

### 2. KnowledgeCard のタグリンク化

タグチップを `<span>` から `<a href="/articles/by-tag?tag={encodeURIComponent(tag)}">` に変更した。

- 特殊文字（`C++`, 日本語タグ等）は `encodeURIComponent` でエスケープ
- hover 時にインタラクティブカラーに変化するスタイルを追加
- 展開ボタン（`+N tags`）のリンク化は不要と判断（展開後に個別タグをクリックする導線で十分）

### 3. サイドバーデフォルト閉じ

`expandedSections` の初期値を全グループ展開（`["feeds", "explore", "recap", "settings", "admin"]`）から空配列（`[]`）に変更した。

`$effect` で現在パスが属するグループを自動展開する:
- ページ遷移時に、現在 URL のパスが子アイテムの href と prefix match するグループを展開
- ユーザーが手動で閉じたグループは、ページ遷移しない限り閉じたまま

`isActive` / `isParentActive` を完全一致から prefix match に変更し、`/articles/by-tag?tag=AI` のようなクエリパラメータ付き URL でもハイライトが正しく機能するようにした。

### 4. ナビゲーション更新

Explore グループに「Tag Articles」エントリを追加:
- Sidebar.svelte（Desktop）
- navigation.ts（共通定義）
- `Tag` アイコン使用

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|--------|------|------------|
| `/feeds/tag-articles` にルート配置 | URL の一貫性 | タグはフィードではなく記事の属性。概念的に `/articles/` が正しい |
| Tag Trail 内にタグ記事一覧を統合 | 既存 UI を活用 | Tag Trail はランダムフィード探索が目的であり、Knowledge Home からのタグ深掘りとは異なるユースケース |
| サイドバー状態を localStorage に永続化 | リロード後も維持 | 現時点ではオーバーエンジニアリング。`$effect` による自動展開で十分 |

## Consequences

### Pros

- **タグ探索の導線が成立** — Knowledge Home のタグチップから直接そのタグの記事一覧に遷移可能
- **Desktop/Mobile 両対応** — `useViewport` 分岐で両方の UI を提供
- **既存 API の再利用** — `fetchArticlesByTag` をそのまま使用、バックエンド変更なし
- **サイドバーの視認性向上** — デフォルト閉じで初期表示がコンパクトに
- **自動展開** — 現在ページのグループは自動で開くため、ナビゲーションの利便性を損なわない

### Cons / Tradeoffs

- **新規ルート追加** — `/articles/by-tag` が既存の `/articles/[id]` と並存する。ルーティングの一貫性を今後整理する必要がある可能性
- **Mobile コンポーネントの重複** — `TagArticleList.svelte` と `MobileTagArticlesScreen.svelte` に類似コードがある。将来的に共通化を検討
- **サイドバー状態の非永続化** — リロードすると全グループが閉じた状態に戻る。ユーザーの好みを覚えない
