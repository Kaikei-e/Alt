# ADR-000926: Augur citation を CitationKind discriminator で kind 駆動ルーティングに切り替える

## Status

Accepted

## Date

2026-05-27

## Affected Services

- proto (alt.augur.v2 / alt.knowledge.loop.v1 / services.sovereign.v1)
- rag-orchestrator (Go)
- knowledge-sovereign (Go)
- alt-frontend-sv (TypeScript / Svelte)
- alt-backend (Go) — consumer (passive)
- alt-butterfly-facade (Go) — consumer (passive)

## Context

ユーザ報告のスクリーンショットで、`/augur/2909cc27-0df8-4d87-9edc-adf8f4aa335d` (正規の Augur 会話詳細) の右ペイン「Citations」内、`01 summary 6b5bbd85-ed4a-4812-95f1-22f2d181437a` をクリックすると `/augur/6b5bbd85-ed4a-4812-95f1-22f2d181437a` に遷移してしまうことが確認された。

`6b5bbd85-...` は alt-db の `summary_versions.summary_version_id` であって rag-db の `augur_conversations.id` ではないため、backend は `GetConversation(6b5bbd85-...)` で `CodeNotFound` を返す。[[000924]] により `CodeNotFound` は穏便な未発見表示には透過されるが、リンクそのものはやはり死んでいる。

### 真因

`rag-orchestrator/internal/adapter/connect/augur/handler.go:660-666` の `CreateAugurSessionFromLoopEntry` が、Knowledge Loop の `evidence_refs[].ref_id` (= summary_version_id / article_id の素の UUID) を `Citation.url` にそのまま書き込んでいた:

```go
refs = append(refs, domain.AugurCitation{
    URL:   sanitizeUTF8(r.RefId),  // RefId は URL ではなく UUID のこともある
    Title: sanitizeUTF8(r.Label),
})
```

これが append-only な `augur_messages.citations` JSONB に永続化される。frontend `CitationRail.svelte` は素直に `<a href={cite.URL}>` で出力するため、ブラウザは bare UUID を現在パス `/augur/<conversation>` に対する**相対パス**として解決し、`/augur/<summary_version_id>` を新規タブで開く。

### 過去 ADR で対処されなかった理由

`/augur/[conversationId]` ルートそのものは canonical で、過去の ADR でも繰り返し触れられていたが、いずれも citation URL 構築規約には関与していなかった:

- [[000210]] alt-frontend-sv のレスポンシブ統合で `/augur/[conversationId]` を確立
- [[000714]] mobile ChatWindow のハイドレーション修正
- [[000716]] mobile ボトムタブ永続化
- [[000924]] alt-backend で Augur upstream の `CodeNotFound` をそのまま透過

加えて proto `LoopEvidenceRef.ref_id` のコメント `// ref_id is typically a URL or article id` が**曖昧さを公式に容認**しており、handler は最も単純な「URL とみなして書く」実装を選択していた。citation の意味 (web link / article / summary) を判別する手段が proto レベルに存在しなかったことが構造的な根因。

### Immutable invariant 上の制約

`augur_messages` は append-only ([[000836]] 由来の design)。過去の壊れた citation 行を UPDATE して直すことは不変条件違反になる。修正は新規 citation を正しく書き、過去データは安全に非リンク化する方針で進めなければならない。

## Decision

`CitationKind` discriminator を proto レベルに導入し、Knowledge Loop → rag-orchestrator → Augur → FE の経路全体で kind を担いで運ばせる。`Citation.url` は `kind = WEB` 専用に温存し、`kind = ARTICLE / SUMMARY` のときは `ref_id` を FE が `/articles/<ref_id>` に解決する。`UNSPECIFIED` は legacy / rolling-deploy 用の安全弁とし、FE 側で非リンク化することで relative-URL バグの再発を構造的に塞ぐ。

### 採用設計

1. **Proto cascade (additive)**
   - `alt.augur.v2.Citation` / `LoopEvidenceRef` に `CitationKind kind`, `string ref_id` を追加
   - `alt.knowledge.loop.v1.EvidenceRef` に `EvidenceKind kind` を追加
   - `services.sovereign.v1.KnowledgeLoopEvidenceRef` に `EvidenceKind kind` を追加
   - 既存フィールドの番号は変えない。`buf breaking` 通過 (additive のみ)
   - 3 つの proto package に同形 enum を持つ理由はパッケージ境界をまたがない原則の維持。値域は同一なので handler / FE 側で薄い変換関数で結ぶ
2. **rag-orchestrator**
   - `domain.AugurCitation` に `Kind` (`CitationKind` string type) と `RefID` を追加し、`augur_messages.citations` JSONB に additive で永続化
   - 新ヘルパー `citationFromEvidenceRef` が `r.Kind` で分岐 — WEB は `URL=r.RefId`、ARTICLE / SUMMARY は `RefID=r.RefId`、UNSPECIFIED は legacy fallback (`URL=r.RefId`) で過去挙動を維持
   - `GetConversation` の読み出しパスも `Kind` / `RefId` を proto に乗せて FE に返す
3. **knowledge-sovereign**
   - `knowledge_loop_projector` の `appendRefIfPresent` が `labelToEvidenceKind` (純関数 — `summary` / `article` など label の文字列のみで判定) で discriminator を埋める
   - 純関数のため reproject-safe な不変条件を維持し、`time.Now()` / latest projection state 参照は無し
4. **alt-frontend-sv**
   - 新ヘルパー `citation-href.ts` を導入し、`kind` 駆動で href を決定
   - `CitationRail.svelte` は `hrefFor()` 経由で href を取得、得られなければ `<span>` で disabled 表示する
   - `target="_blank"` は WEB のみに適用 (ARTICLE / SUMMARY は同一オリジン `/articles/[id]` で SvelteKit の client router に渡る)
   - `connect/augur.ts`, `connect/knowledge_loop.ts`, `server/knowledge-loop-api.ts`, `AugurChat.svelte`, `[conversationId]/+page.svelte` で `kind` / `refId` をパススルー
5. **過去データ**
   - `augur_messages.citations` の reproject / backfill は行わない
   - 既存の壊れた citation 行は `kind` が空のまま読み出され、FE の `citationHref` が `default` 分岐で `undefined` を返すため disabled 表示になる (誤遷移は副次的に止まる)

### 検討した代替案と却下理由

- **A: サーバ側で完成形 URL を組み立てる (proto 無変更)**
  rag-orchestrator が `r.RefId` の形状 (UUID v4 regex か URL か) を見て URL を構築する案。proto 変更を回避できる利点はあったが、判別ロジックが推測ベースで脆く、将来 citation 種別を増やす際 (recap / cluster など) に同じ穴を再生する。`kind` を構造的に持たせる方が長期で安全と判断し却下。
- **B: 最小 hot-fix (B) → 後追いで proto 整理 (A) の 2 段階リリース**
  本番の壊れリンクを早く止めたい局面では合理的だが、現状はまだ影響範囲が限定的 (Knowledge Loop seed 経路のみ) かつ FE 側 disabled fallback で同時に過去データもカバーできる。2 段階にすると proto 整理が後ろ倒しで放置されるリスクが高く却下。
- **C: 過去 citation を reproject して backfill**
  augur_messages は append-only という不変条件があり、`augur_messages.citations` を UPDATE することは設計に反する。新しい versioned 表 (citation_v2) や projector を導入する複雑性に見合うほどの利益はなく却下。FE の defensive 非リンク化で十分。

## Consequences

### Pros

- スクリーンショット報告された `/augur/<summary_version_id>` への誤遷移が構造的に消える (kind 駆動なので bare UUID を URL に書き込む経路自体が無くなる)
- citation 種別を proto に明示することで、将来 recap / cluster / external link を追加する際の安全な拡張点を確保
- FE `citationHref` の default 分岐が legacy 壊れデータも disabled 化し、過去会話の誤遷移も副次的に止まる
- `augur_messages` の append-only 不変条件を一切曲げず、reproject / backfill / migration が不要
- 変更はすべて additive で `buf breaking` clean、rolling deploy 中も古い consumer / producer は無視するだけで壊れない
- knowledge-sovereign の projector 拡張は純関数 (`labelToEvidenceKind`) のため reproject-safe を維持

### Cons / Tradeoffs

- proto 3 ファイルに同形 enum を重複定義することになった (パッケージをまたがる proto enum import を避けるため)。値域が将来分岐したら同期コストが発生する
- 過去の壊れた citation 行は永続的に disabled のまま (リンクとしては復活しない)。歴史的 chat に戻ったユーザが clickable な citation を期待した場合の UX 後退は受容する
- `UNSPECIFIED` fallback が "URL に RefId を書く" 旧挙動を温存しているため、新規 emitter が kind を忘れる退化を**サーバ側では検知できない** (FE の disabled fallback だけが安全網)。次の citation 種別追加時に lint / contract test を強化する必要がある
- proto コメントが「`ref_id` is typically a URL or article id」と曖昧さを許していた歴史的負債は本 ADR で構造的に解消するが、同種の「kind を持たない discriminated union」がコードベース内に他に隠れていないかは別途棚卸しが必要
