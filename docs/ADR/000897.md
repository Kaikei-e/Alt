# ADR-000897: morning-update ジョブに status ライフサイクルを与え recap_jobs の pending リークを止める

## Status

Accepted

## Date

2026-05-12

## Affected Services

- recap-worker (Rust)
- recap-db (Atlas migration / `recap-migration-atlas`)
- alt-frontend-sv (Svelte)

## Context

`recap/job-status` ダッシュボード（recap-worker `GET /v1/dashboard/recap_jobs` → alt-backend → alt-frontend-sv）に大量の `pending` 行が並んでいた。本番 `recap-db` を直接確認した時点の `recap_jobs` の内訳:

| status | window_days | trigger_source | 件数 | 期間 |
|---|---|---|---|---|
| completed | 3 | system | 16 | 4/27〜5/10（毎日 17:00 UTC、正常）|
| failed | 3 | system | 3 | dedup/persist 失敗（既知の subworker artefact 系、[[000825]]）|
| **pending** | **1** | **morning** | **148** | 5/08 15:19〜直近、約 30 分おき |
| failed | 1 | morning | 168 | 5/04〜5/08、`last_stage='abandoned_at_startup'` |
| running | 3 | system | 1 | 5/11 17:00 のジョブ。ハングではなく news-creator の batch 要約待ちで進行中 |

つまり溜まっていた `pending` は **3-day Recap ジョブではなく Morning Letter 更新ジョブ**だった。原因は次のとおり:

1. `recap-worker/recap-worker/src/main.rs` は `MORNING_DAEMON_ENABLED`（`compose/recap.yaml` で既定 `true`、[[000886]]）が立つと `spawn_morning_update_daemon` を起動する。`MorningUpdateDaemon`（`src/scheduler/daemon.rs`）は **30 分間隔**で `JobContext::new_morning_update`（`trigger_source="morning"`, `window_days=1`）を作り `Scheduler::run_morning_update` を呼ぶ。
2. `run_morning_update` → `MorningPipeline::execute_update`（`src/pipeline/morning.rs`）。その最初の `self.fetch.fetch(job)` = `AltBackendFetchStage::fetch`（`src/pipeline/fetch.rs`）が batch パイプラインと共用で、**`create_job_with_lock_and_window(job_id, None, 1, "morning")`** を呼ぶ。これは advisory lock 取得と raw article backup の FK のために `recap_jobs` 行を INSERT し、`status` カラムは migration 既定値 `'pending'`（`recap-migration-atlas/migrations/20251212000000_add_job_state_tracking.sql`）になる。
3. ところが batch パイプライン（`Scheduler::run_job` + `PipelineOrchestrator`/`StageExecutor` が各ステージで `update_job_status_with_history(...Running...)`、最後に `Completed`/`Failed`）と違い、`MorningPipeline::execute_update` は **`recap_jobs.status` を一切更新しない**。Morning Letter 本体は `morning_letters` に正常 INSERT され（直近まで毎日 `is_degraded=false` で生成済）、ログも `morning update job completed` を出すのに、追跡行は `pending` のまま孤立する。
4. 結果、**30 分ごとに孤児 `pending` 行が 1 件ずつ増える（≒48/日）**。recap-worker 再起動時のみ `mark_abandoned_jobs`（`src/scheduler/jobs.rs`、boot 時 sweep）が溜まった `pending` を一括 `failed`（`last_stage='abandoned_at_startup'`）に倒す。これが 5/04〜5/08 の 168 件の正体で、最後の再起動（5/08 ≈15:00）以降に 148 件が再蓄積していた。

ダッシュボード上はノイズに過ぎないが、`recap_jobs` をジョブ実行の真実の記録として扱う前提（[[000113]] の status 整合性修正と同じ精神）が崩れており、運用上「3-day Recap が大量に詰まっている」という誤読を生んでいた。

## Decision

**morning-update パイプラインに batch パイプラインと同じ status ライフサイクル（`pending → running → 終端`）を持たせる。** 終端の成功状態は `Completed` を流用せず、`JobStatus` に新値 **`MorningCompleted`（DB 文字列 `"morning_completed"`）** を追加する。

- `recap-worker`:
  - `src/store/dao/types.rs`: `JobStatus::MorningCompleted` を追加（`#[sqlx(rename)]`/`#[serde(rename)]` で `"morning_completed"`）。`is_terminal()` と `from_db_str()` ヘルパを新設し、DAO 層に点在していた `match status_str { ... }` を `from_db_str` に統一。
  - `src/scheduler/jobs.rs`: `run_morning_update` を、`execute_update` の結果で `recap_jobs` 行を確定するよう変更。成功 → `MorningCompleted`、失敗 → `Failed`（エラーチェーンを reason に）。`Scheduler::run_job` と同じ「DAO 書き込み失敗はログのみで本処理結果を優先」構造を `finalize_morning_job` 関数として切り出し、ユニットテストで固定。
  - `src/pipeline/morning.rs`: `fetch` 直後（行が確実に存在した後）に `Running`（`last_stage='morning_fetch'`）へ遷移。ダッシュボードで実行中の tick が見えるようにする。
- `recap-db`（`recap-migration-atlas/migrations/20260512000000_add_morning_completed_status.sql`）:
  - `chk_status_history_status` CHECK 制約を `('pending','running','completed','failed','morning_completed')` に拡張（`recap_jobs.status` には CHECK 制約が無いので ALTER 不要）。
  - 既存の孤児 `pending` morning 行を `morning_completed` にバックフィル（`last_stage` を `COALESCE(..., 'persist')`）。Morning Letter 本体は `morning_letters` に永続化済みなので「成功」扱いが妥当。
  - バックフィルした各行について `recap_job_status_history` に終端イベントを INSERT（`actor='migration_backfill'`）。
- `alt-frontend-sv`:
  - `src/lib/domain/dashboard/types.ts`: `JobStatus` union に `"morning_completed"`、`TriggerSource` に `"morning"` を追加。
  - `src/lib/utils/jobStatusInk.ts` / `src/lib/domain/dashboard/format.ts`: `morning_completed` を success 系（glyph `✓` / label `"Morning Update"` / 緑系 ink）として扱う。
- `alt-backend` は `domain.RecapJob.Status` が元々 `string` の passthrough なので変更不要。

### 検討した代替案

1. **morning 成功も既存 `Completed` を流用し、ダッシュボード側で `trigger_source` で出し分ける** — 却下。morning-update は 30 分おきで本質的に高頻度なので、`completed` で混ぜると Recap ジョブ一覧を 1 日 48 行のノイズが支配する。`morning_completed` という別状態にしておけば一覧・集計の両面で自然に分離でき、`trigger_source` への依存も最小になる。
2. **morning-update では `recap_jobs` 行をそもそも作らない** — 却下。`backup_raw_articles` の FK と fetch の advisory lock がどちらも `job_id` を要求しており、行を作らないなら fetch ステージの再設計と morning 経路での raw article backup の廃止が必要になる。影響範囲が大きく、リスクに見合わない。
3. **morning daemon の間隔を伸ばす / 定期 cleanup で消す** — 却下。根本原因（status が追跡されていない）を直さず、リークを遅らせ・隠すだけ。再起動時の `mark_abandoned_jobs` が `failed` と誤ラベルする問題も残る。
4. **`JobStatus` 値だけ足し、morning パイプラインは引き続き無更新にして、別の background sweeper が `pending → morning_completed` に倒す** — 却下。daemon と sweeper の競合・タイミング依存を増やすだけで、「生成元が自分の行を確定する」素直なモデルより悪い。

## Consequences

### Pros

- `recap/job-status` の `pending` が単調増加しなくなる。`recap_jobs` が再びジョブ実行の信頼できる記録に戻る。
- morning-update が `recap_job_status_history` にイベントを残すようになり、tick ごとの実行が監査可能になる。
- 再起動時の `mark_abandoned_jobs` が morning 行を `failed`（`abandoned_at_startup`）と誤って倒すことが無くなる（既に終端だから sweep 対象外）。
- 既存の 148 件の孤児 `pending` も migration のバックフィルで一掃される（履歴イベント付き）。
- DAO 層の status パースが `JobStatus::from_db_str` に集約され、将来の状態追加でパース漏れが起きにくい。

### Cons / Tradeoffs

- `JobStatus` の取りうる値が 1 つ増え、network boundary（DB / フロント / 将来の Connect proto）で扱う側に「未知の値はとりあえず通す/緑扱い」の前提を要求する。フロントは `?? fallback` と型追加で対応済みだが、外部 consumer がいれば波及する。
- morning-update が DB 書き込みを 1 tick あたり 2 回（`running`、終端）増やす。`recap_jobs` / `recap_job_status_history` への負荷は無視できる程度（48 tick/日）。
- `fetch` が advisory lock を取れず `bail!` する理論上の経路では `recap_jobs` 行が無いまま `finalize_morning_job` が `Failed` を書こうとし、history INSERT が FK 違反になる（catch & log 済み、実害なし）。morning は毎回新 UUID なので実際には起きない。
- `atlas migrate lint` をローカルで完走できない既知問題（別の既存 migration が `pg_cron` を使い plain postgres dev image で失敗。本 migration 由来ではない）。本番反映は alt-deploy の `atlas migrate apply` 経路なので影響はないが、ローカル lint の前提が崩れている点は別途要整理。

## Deploy-model 整合性セルフチェック

本 ADR は DB migration + Rust/TS のコード変更のみで、新規 compose service・init container・名前付き volume を一切追加しない。

- [x] 新設 compose service が `depends_on` で他 service を gate するか？ — N/A（compose 変更なし）
- [x] 新設 compose service に `restart: "no"` の one-shot init container を含むか？ — N/A
- [x] 新設名前付き volume を populate する責務はどこにあるか？ — N/A（volume 追加なし）
- [x] いずれも該当しない: **rolling 互換**。ただし migration（`chk_status_history_status` 拡張）は recap-worker の rollout より**前**に適用する必要がある（新コードが `morning_completed` を history テーブルに書くため）。alt-deploy の「Atlas migration → per-service roll」順序がこれを満たす。
