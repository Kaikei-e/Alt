# ADR-000410: Knowledge Home にリアルタイムストリーミングと Supersede UX を導入する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — handler (2 streaming RPC), knowledge_home_items 拡張, feature flags
- alt-butterfly-facade (Go) — streamingProcedures マップ拡張
- alt-frontend-sv (SvelteKit) — useStreamUpdates hook, StreamUpdateBar, SupersedeBadge

## Context

Knowledge Home Phase 3 まではポーリングベースのデータ取得であり、2 つの課題があった:

1. **ストリームが静的スナップショット** — 新しいイベント（記事追加、要約完了、タグ更新）が発生しても、ユーザーがページをリロードするまで反映されない。フィード統計画面（`/feeds`）には既に Connect-RPC server streaming による `StreamFeedStats` があり、Knowledge Home にも同等のリアルタイム性が求められた。

2. **バージョン変更の不可視** — 要約やタグセットが更新（supersede）された際、ユーザーには「何が変わったか」が分からない。Altのversioned artifacts設計（SummaryVersion, TagSetVersion）の恩恵をユーザーに伝える手段がなかった。

## Decision

**Connect-RPC server streaming で Knowledge Home のリアルタイム更新を提供し、supersede 状態をアイテムに表示することで「追従可能な面」を実現する。**

### 1. ストリーミング RPC

**`StreamKnowledgeHomeUpdates`（server streaming）:**

```
Client → StreamKnowledgeHomeUpdatesRequest { lens_id? }
Server →→→ StreamKnowledgeHomeUpdatesResponse { event_type, item?, digest_change?, occurred_at } (stream)
```

実装パターンは既存の `StreamFeedStats` に準拠:
- 5 秒間隔で `knowledge_events` をポーリングし、新規イベントを差分として送信
- 10 秒間隔でハートビートを送信（接続維持）
- `ctx.Done()` でクライアント切断を検知して graceful に終了

**`StreamRecallRailUpdates`（server streaming）:**

- 30 秒間隔で recall candidates を再取得し、変更があれば送信
- RecallRail の候補は頻繁に変わらないため、更新間隔を長めに設定

### 2. BFF 対応

`alt-butterfly-facade` の `streamingProcedures` マップに 2 エントリを追加:

```go
"/alt.knowledge_home.v1.KnowledgeHomeService/StreamKnowledgeHomeUpdates": true,
"/alt.knowledge_home.v1.KnowledgeHomeService/StreamRecallRailUpdates":    true,
```

これだけで streaming timeout（5 分）と `ForwardStreamingRequest` による透過プロキシが適用される。Streaming 認証は既存の `AuthInterceptor.WrapStreamingHandler`（[[000029]]）で対応済み。

### 3. Supersede UX

**データベース拡張（`knowledge_home_items` ALTER）:**

| 追加カラム | 型 | 説明 |
|-----------|---|------|
| supersede_state | TEXT | summary_updated / tags_updated / both_updated |
| superseded_at | TIMESTAMPTZ | supersede 発生日時 |
| previous_ref_json | JSONB | 旧バージョンの参照情報 |

部分インデックス: `(user_id, supersede_state) WHERE supersede_state IS NOT NULL`

**Proto 拡張:**

`KnowledgeHomeItem` に `optional SupersedeInfo supersede_info = 11` を追加。`SupersedeInfo` メッセージは state, superseded_at, previous_summary_excerpt, previous_tags を含む。

**イベントタイプ追加:**

`SummarySuperseded`, `TagSetSuperseded`, `HomeItemSuperseded` を knowledge_events のイベントタイプとして追加。knowledge_projector が将来これらを処理して supersede_state を設定する。

### 4. フロントエンド

**ストリーム更新の適用方針:**

| 対象 | 適用方式 | 理由 |
|------|---------|------|
| TodayBar | 即時更新 | digest はコンパクトで視覚的混乱が少ない |
| KnowledgeStream | "N items updated" バー → クリックでマージ | 自動ソートはスクロール位置を壊す |
| RecallRail | 控えめに自動差し替え | サイドバーは目立たないため自動更新が許容される |

**`useStreamUpdates.svelte.ts`:**

`useStreamingFeedStats.svelte.ts` パターンに準拠:
- 指数バックオフ再接続（1s → 10s、最大 10 回リトライ）
- 3 秒間のコアレシング（短時間に発生した複数イベントをバッチ化）
- `pendingUpdates` / `pendingCount` でバッファリングし、ユーザーの `applyUpdates()` で適用

**新コンポーネント:**

| コンポーネント | 説明 |
|-------------|------|
| `StreamUpdateBar.svelte` | "3 items updated" 通知バー（TodayBar と Stream の間） |
| `SupersedeBadge.svelte` | KnowledgeCard 内のバージョン変更バッジ |
| `supersede-display-map.ts` | supersede state → label / icon / color マッピング |

### 5. Feature Flag 体系

Phase 4 の全機能は独立した feature flag で制御:

| Flag | 制御対象 | OFF 時 |
|------|---------|--------|
| `enable_recall_rail` | RecallRail | MiniRecallPanel にフォールバック |
| `enable_lens` | LensSelector + CRUD | 非表示 |
| `enable_stream_updates` | streaming 接続 + StreamUpdateBar | polling のみ |
| `enable_supersede_ux` | SupersedeBadge | 非表示 |

4 機能は独立して ON/OFF 可能。`feature_flag_gateway` の `isFlagGloballyEnabled` に 4 ケースを追加し、`KnowledgeHomeConfig` に 4 つの設定項目を追加。Admin ダッシュボードの FeatureFlagPanel にも 4 行追加。

ロールアウト順序: `enable_supersede_ux` → `enable_lens` → `enable_recall_rail` → `enable_stream_updates`

## Consequences

### Pros

- **リアルタイム性** — ページリロードなしで新しいイベントがユーザーに届く
- **ユーザー主導の更新適用** — StreamUpdateBar のクリックでマージするため、スクロール位置が壊れない
- **バージョン変更の可視化** — 要約やタグの更新がユーザーに説明可能になる
- **既存パターンの再利用** — StreamFeedStats と同じ ticker + heartbeat + ctx.Done パターンで実装コストが低い
- **BFF 変更が最小** — streamingProcedures マップに 2 行追加するだけ

### Cons / Tradeoffs

- **サーバーサイドポーリング** — streaming RPC の内部実装は `knowledge_events` テーブルへのポーリングであり、真のイベントプッシュ（NATS 等）ではない。ユーザー数増加時はイベントバスへの移行を検討
- **コアレシング遅延** — 3 秒のバッファリングにより、イベント発生から表示まで最大 8 秒（ポーリング 5s + コアレシング 3s）の遅延がある
- **supersede 処理の未完** — Phase 4 ではカラム追加と UI コンポーネントまで。projector による自動設定は次フェーズで実装
- **接続リソース** — 各ユーザーが 2 本の streaming 接続（Home + RecallRail）を張るため、同時接続数が増加する
