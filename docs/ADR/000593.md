# ADR-000593: Knowledge Home の要約投影エラー伝播と Recall タグクリックトラッキングを追加する

## Status

Accepted

## Date

2026-03-26

## Affected Services

- alt-backend (Go) — KnowledgeProjector, RecallProjector, TrackHomeActionUsecase
- alt-frontend-sv (SvelteKit) — KnowledgeCard, RecallRail, Home ページ
- knowledge-sovereign (Go) — 変更なし。read model の書き込み先として影響を受ける

## Context

Knowledge Home に 2 つの不具合が報告された。

### 不具合 1: 要約が完了しても「Summarizing」のまま表示される

`projectSummaryVersionCreated` が `GetSummaryVersionByID` の失敗時にエラーをログ出力するだけで握りつぶしていた。結果として `summaryExcerpt` が空のまま `summary_state=pending` で upsert され、チェックポイントがイベントを通過するため**二度とリトライされない**。alt-db と sovereign-db 間の一時的な接続問題やタイミングずれで発生しうる。

### 不具合 2: Recall の Why に Knowledge カードの操作イベントが表示されない

2 つの原因が特定された。

1. **タグクリックが完全に未トラッキング**: `KnowledgeCard.svelte` のタグは `<a>` リンクによる遷移のみで、`trackAction` が発火しない。`validActionTypes` にも `tag_click` が存在せず、recall signal も生成されない
2. **記事 Open イベントの遅延が過大**: `projectHomeItemOpened` の `FirstEligibleAt` が 24 時間後、RecallProjector の `opened` シグナル閾値が 48 時間後に設定されており、ユーザーが操作結果を確認できるまでの待ち時間が長すぎる。また `handleRecallOpen` (Recall カードクリック) では `trackAction` が未発火

## Decision

### 決定 1: 要約投影のエラーを伝播させ、チェックポイントを進めない

`projectSummaryVersionCreated` で `GetSummaryVersionByID` が失敗した場合、`fmt.Errorf` でラップしてエラーを返す。プロジェクターのイベントループは最初のエラーで `break` し、チェックポイントを**最後に成功したイベントまでしか進めない**。次のサイクルで失敗イベントがリトライされる。

write-through（ハンドラから projection table を直接更新）は採用しない。`.claude/rules/knowledge-home.md` の「write path から projection table を直接更新しない」原則に従い、dismiss のような即時 UX フィードバックが必要なケースとは異なると判断した。

### 決定 2: `tag_click` アクション種別と recall signal を追加する

- `EventHomeItemTagClicked` イベント種別を追加
- `SignalTagClicked` シグナルと `ReasonTagInteraction` recall reason を追加
- `TrackHomeActionUsecase` の `validActionTypes` / `actionToSignalType` に `tag_click` を追加
- RecallProjector に `tag_clicked` シグナルの scoring（weight=0.20）を追加
- フロントエンドの `KnowledgeCard` に `onTagClick` コールバックを追加し、タグクリック時にトラッキングを発火
- `handleRecallOpen` に `trackAction("open", itemKey)` を追加

### 決定 3: Recall の eligibility delay を 24h/48h から 1h に短縮する

- `projectHomeItemOpened` の `FirstEligibleAt` を `eventTime + 1h` に変更
- RecallProjector の `SignalOpened` 閾値を `1*time.Hour` に変更
- ユーザーが操作から 1 時間後に Recall の Why で確認できるようにする

## Consequences

### Pros

- **要約の確実な反映**: 一時的な DB 接続問題があっても、プロジェクターが自動リトライするため要約が最終的に必ず反映される
- **Recall の Why が豊かになる**: タグクリック・記事オープンの操作が Recall の理由として表示され、ユーザーが「なぜこの記事が Recall されたか」を理解できる
- **操作フィードバックの即時性向上**: 1 時間の delay は 24-48 時間と比べて大幅に短く、ユーザーが自分の操作結果を確認しやすい
- **イミュータブルデータモデル不変条件を維持**: append-only イベント、reproject-safe、versioned artifacts の原則に準拠

### Cons / Tradeoffs

- **プロジェクターの停滞リスク**: `GetSummaryVersionByID` に永続的な障害がある場合（テーブル未マイグレーション等）、プロジェクターが該当イベントでスタックする。ただし、従来の「永久に pending のまま無視される」挙動よりもログ・メトリクスで検知しやすい
- **best-effort 投影の放棄**: 従来はエラーのあるイベントをスキップして後続イベントを処理していた。今回の変更により、1 つのエラーでバッチ全体の処理が停止する。これは意図的なトレードオフで、データ整合性を優先した
- **recall delay 1h は aggressive**: 短すぎると「さっき見た記事」が Recall に出てノイズになる可能性がある。チューニングが必要になった場合は定数を調整する
