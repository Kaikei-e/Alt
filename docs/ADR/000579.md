# ADR-000579: news-creator の全 LLM リクエストパスで Ollama options を統一し、モデルリロードを防止する

## Status

Accepted

## Date

2026-03-25

## Affected Services

- news-creator (Python / FastAPI)

## Context

[[000567]] で Ask Augur のチャットストリームを news-creator の HybridPrioritySemaphore 経由にルーティングした。セマフォによる優先制御自体は正しく動作し、chat は HIGH PRIORITY で即時取得（0s wait）できていた。

しかし、chat の TTFT（Time to First Token）が最大 **259 秒** に達する事象が発生した。

### 根本原因: Ollama パラメータ不一致によるモデルリロード

news-creator から news-creator-backend（Ollama）に送られるリクエストの `options` が、パスによって異なっていた。

| パラメータ | バッチ要約 (`get_llm_options`) | chat (rag-orchestrator 由来) |
|-----------|---------------------------|--------------------------|
| `num_batch` | 1024 | **未送信** |
| `num_keep` | -1 | **未送信** |
| `stop` | `["<end_of_turn>"]` | **未送信** |

Ollama は `NUM_PARALLEL=2` で 2 スロットの並列処理をサポートするが、**スロット間でパラメータが異なると runner の再構成（モデルのアンロード→再ロード）が発生する**。news-creator-backend のログで確認された再構成シーケンス:

```
09:16:02 — バッチ要約完了 (BatchSize:1024)
09:16:06 — モデルリロード (BatchSize:512) ← chat 用に再構成
09:16:18 — chat 完了 (4m26s)
09:16:19 — モデルリロード (BatchSize:1024) ← 次のバッチ用に再構成
```

この「ピンポン」が chat リクエストの TTFT を破壊的に悪化させていた。chat は Ollama 内部でバッチ要約と並列実行されるが、パラメータ不一致により同一 runner で処理できず、バッチ完了後のモデルリロードまで完全にブロックされていた。

### 影響を受けていたパス

| パス | 問題 |
|------|------|
| Ask Augur ChatStream（ストリーミング） | `num_batch`, `num_keep`, `stop` 未送信 → モデルリロード発生 |
| rag-orchestrator Generate（ストリーミング） | 同上 |
| Morning Letter Chat（非ストリーミング） | chat_handler.py が `stream: false` を 400 で拒否 → 未対応 |

バッチ要約・FE 要約・Query Expansion は `config.get_llm_options()` を使用しており、問題なかった。

## Decision

**news-creator の全 LLM リクエストパスで `config.get_llm_options()` をベースオプションとして使用し、caller からの options をオーバーレイする方式に統一する。** また、Morning Letter 向けに非ストリーミング chat エンドポイントを追加する。

### 実装

#### 1. `OllamaStreamDriver` にオプションマージと共通メソッドを追加

`ollama_stream_driver.py` に以下を追加:

- **`_merge_options(caller_options)`**: `config.get_llm_options()` をベースとし、caller の options をオーバーレイして返す。全パスがこのメソッドを通過することで、`num_batch`, `num_keep`, `stop` 等が常に含まれる
- **`_build_prompt_from_messages(messages)`**: chat メッセージからプロンプトへの変換ロジックを抽出。`chat_stream()` と `chat_generate()` で共有
- **`chat_generate(payload)`**: 非ストリーミング版の chat → `/api/generate` プロキシ。`_merge_options()` でオプションを統一

#### 2. `chat_stream()` のオプション構築を変更

変更前:
```python
if payload.get("options"):
    generate_payload["options"] = payload["options"]
```

変更後:
```python
generate_payload["options"] = self._merge_options(payload.get("options"))
```

caller（rag-orchestrator）が `num_predict: 2048` 等を送信した場合、ベースの `num_predict: 1200` を上書きする。`num_batch`, `num_keep`, `stop` は常にベースから供給される。

#### 3. `OllamaGateway` に `chat_generate()` を追加

`chat_stream()` と対称的な非ストリーミングメソッド。HIGH PRIORITY でセマフォを取得し、`stream_driver.chat_generate()` を呼び出す。

#### 4. `chat_handler.py` で非ストリーミングをサポート

`stream: false` のリクエストを 400 で拒否する代わりに、`gateway.chat_generate()` にルーティング。Morning Letter 等の非ストリーミング chat がセマフォ経由で処理される。

### 検討した代替案

| 案 | 評価 | 却下理由 |
|---|---|---|
| rag-orchestrator 側で `num_batch` 等を追加 | Go 側の修正が必要 | news-creator が GPU 設定を所有すべき。rag-orchestrator は GPU 詳細を知らなくてよい |
| Ollama の Modelfile で全パラメータを固定 | リクエスト側の差異が残る | `num_predict` 等リクエストごとに異なる値は Modelfile では制御できない |
| セマフォでプリエンプションを強制 | 実行中バッチをキャンセル | モデルリロード自体が根本原因であり、プリエンプションは対症療法 |

## Consequences

### Pros

- **モデルリロード完全防止**: 全パスが同一の `num_batch`, `num_keep`, `stop` を送信するため、Ollama の runner 再構成が発生しない
- **TTFT の劇的改善**: バッチ要約との並列実行時でも、同一 runner 上でタイムスライスされるため、chat の prefill がブロックされない
- **Morning Letter 対応**: 非ストリーミング chat がセマフォ経由で処理可能になり、GPU 飢餓が発生しない
- **単一責任**: GPU パラメータの管理が `config.get_llm_options()` に集約され、新しいリクエストパスを追加する際も `_merge_options()` を通すだけで統一が保証される

### Cons / Tradeoffs

- chat の `num_predict` は rag-orchestrator 由来の値（2048）が使われるため、バッチ要約（1200）より大きい。これは意図的な設計で、chat は長い応答が必要だが、バッチ要約の生成長を超える可能性がある
- `stop: ["<end_of_turn>"]` が chat にも適用されるが、chat のプロンプトテンプレートが `<end_of_turn>` を終了マーカーとして使用しているため、正しい動作となる
- 非ストリーミング chat の追加により `chat_handler.py` の分岐が増えるが、テストで両パスをカバーしている
