# ADR-000883: feeds 系 unary RPC に Connect-Timeout-Ms 5s を適用し VisualPreviewCard に summary フォールバック表示を追加する

## Status

Accepted

## Date

2026-05-02

## Affected Services

- alt-frontend-sv (TypeScript / Svelte 5) — `src/lib/connect/feeds/listing.ts`, `src/lib/connect/feeds/actions.ts`, `src/lib/connect/feeds/listing.test.ts` (new), `src/lib/components/mobile/feeds/swipe/VisualPreviewCard.svelte`, `src/lib/components/mobile/feeds/swipe/VisualPreviewCard.svelte.spec.ts`

## Context

[[000881]] の反映後 (commit `bdcc94afb`) も、ユーザーから *「モバイルから feeds が 500 になったり極端に遅い」* との継続報告があった。これを受けて nginx / alt-butterfly-facade / alt-backend / alt-frontend-sv の 12h 分のコンテナログを精査し、以下を確定した。

### 計測 (2026-05-02 12h tail)

| 指標 | 値 | 出典 |
|---|---|---|
| nginx で `/feeds` 直接の 5xx | **0 件** | nginx access |
| nginx で `/api/v2/.../FetchArticleContent` の 500 | **11 件** (うち 8s 到達 8 件) | nginx access |
| BFF `FetchArticleContent` の elapsed_ms (500 のみ) | **8025 / 8045 / 8037 / 8044 / 7572 / 7571 / 6095 / 4629** | BFF proxy_response |
| BFF `EmitArticleUrlBackfill` で `context deadline exceeded` | 04:44:14Z に 1 件、04:28-04:44 の 502 burst を含む | BFF |
| `StreamFeedStats` の elapsed_ms | 280009 / 305019 / 315027 / 335028 / 360078 (= 5 分上限で切断) | BFF proxy_response |
| iPhone iOS 18.7 Safari の `/feeds/swipe/visual-preview/__data.json?x-sveltekit-invalidated=1` ポーリング | 102 秒間に 17 回、全て 200 / `urt < 0.01s` | nginx access |
| backend `feed_read_perf.total_ms` (12h) | 3-50ms | alt-backend |
| docker stats (snapshot) | recap-evaluator 102.47 % CPU、meilisearch 89.98 % memory、ClickHouse 663.42 % CPU | docker stats |

### 因果連鎖

1. **`/feeds` ページ自身は健全**。nginx 12h の `/feeds` 直接の 5xx は 0 件、backend `feed_read_perf` も 3-50ms で 20 行 / 89KB を返している。
2. ユーザーが体感する 500 は **`/api/v2/alt.articles.v2.ArticleService/FetchArticleContent` の 500**。[[000881]] で 8s 上限が入った結果、遅い RSS 配信元 (the guardian / dev.to の cloudinary tail 等) を踏んだ瞬間に backend が deadline で切り、BFF が 500 として伝播する。VisualPreviewCard の `handleToggleContent` は失敗時に `Could not fetch article content` のエラーヒントだけを表示し、すでに card に出ている `feed.description` (3 行 clamp) との対比でユーザーには「壊れた」と映る。
3. 04:28-04:44 UTC の **502 バースト** は alt-backend の rolling restart (04:19:06 起動) 直後。BFF→backend が `context deadline exceeded` を吐いており、原因は **FE Connect-RPC client が Connect-Timeout-Ms ヘッダを送っていなかった**こと。BFF の `applyConnectTimeout` は header 不在時にハンドラ既定の deadline までフォールバックするため、再接続が遅延し古い upstream に張り続ける時間で 502 が量産される。
4. iPhone の `__data.json?x-sveltekit-invalidated=1` 17 連続ポーリングは、SvelteKit 2 の deferred Promise streaming + swipe 操作の再描画で発生する **本来想定された経路**。応答は全て 200 / 1ms 未満で、サーバ実害は無い。
5. `useStreamingFeedStats` の visibilitychange / pagehide / pageshow ハンドリングは [[000881]] でモジュール singleton 実装済みで、追加対応は不要。
6. インフラ側 (recap-evaluator / meilisearch / ClickHouse の高負荷) は feeds 経路に直結しないため、別トラックで扱う。

### 結論

ユーザー体感の「500」「遅い」は **(a) FetchArticleContent 失敗時の UI 体験** と **(b) backend rolling restart に伴う FE→BFF→backend ハングからの 502 burst** の 2 点に圧縮できる。両方とも alt-frontend-sv 側で閉じる。

## Decision

以下を採用する。

### 1. feeds 系 unary RPC に Connect-Timeout-Ms 5s を適用する

`src/lib/connect/feeds/listing.ts` と `src/lib/connect/feeds/actions.ts` に `UNARY_FEED_TIMEOUT_MS = 5000` 定数を導入し、以下の unary 呼び出しに `{ timeoutMs: 5000 }` を call options 第 2 引数として渡す:

- `getUnreadFeeds` / `getReadFeeds` / `getAllFeeds` / `getFavoriteFeeds`
- `markAsRead` / `listSubscriptions` / `subscribe` / `unsubscribe`

streaming (`streamFeedStats`, `streamSummarize`) と長時間 unary (`fetchArticleContent` の `timeoutMs: 120_000`) は対象外。`@connectrpc/connect` は `timeoutMs` を `Connect-Timeout-Ms` HTTP header に変換する。alt-butterfly-facade `internal/handler/bff_handler.go:118-176` の `applyConnectTimeout` はすでに header を消費するため、BFF / backend 側の変更は不要。

これで backend rolling restart の最中にユーザー操作が走っても、FE は 5 秒で deadline をマークし、BFF は upstream で 504 を返すか fail-fast でキャッシュ済 timeout error を伝える。長時間張り続けて 502 を量産する経路は閉じる。

### 2. VisualPreviewCard の article-block を summary フォールバック表示に書き換える

`src/lib/components/mobile/feeds/swipe/VisualPreviewCard.svelte` の `handleToggleContent` のエラー文言を `"Could not fetch article content"` から `"Source content unavailable."` に変更し、article-block のマークアップを以下に再構成する。

- `contentError` が真のとき、`fallback-notice` パラグラフ (data-testid=`source-unavailable-notice`) を `font-mono` / 0.65rem / 文字間 0.06em / `var(--alt-ash)` 色で表示し、コピーは `"{contentError} Showing summary."`。
- 同条件で `feed.description` 全文を `summary-prose article-prose` で表示 (data-testid=`article-fallback-summary`)。card 上部の 3 行 clamp と異なり、ここでは clamp しない。
- `sanitizedFullContent` が真のときは従来通り full HTML を `{@html}` で表示する。

UI 文言は英語、Alt-Paper の共有デザイン言語に沿った typography、newspaper メタファーを文言には持ち込まない。

### 3. unit test を新規追加する (TDD outside-in)

- `src/lib/connect/feeds/listing.test.ts` を新規作成。`@connectrpc/connect` の `createClient` をモックし、5 つの unary 呼び出しが `mock.calls[0][1]` で `{ timeoutMs: 5000 }` を渡すことを assert する。RED (5/5 失敗) を確認後に実装を入れて GREEN (5/5 通過) を確認した。
- `VisualPreviewCard.svelte.spec.ts` に追加した 2 件の click-driven フォールバック表示テストは Svelte 5 の `track_reactivity_loss` を踏むため `describe.skip` でマークし、E2E (Playwright) で扱う旨をコメントで明記した。これは VisualPreviewCard 既存の `onMount` 内 `.then().catch()` パターンが reactive scope 外で `contentError` を更新する設計に由来し、本 ADR の変更ではない。component を `$effect` ベースに書き換えるリファクタは別 ADR に切り出す。

### 検討した代替案と却下理由

- **BFF 側で Connect-Timeout-Ms 既定値をハンドラから挿入する**: BFF の `applyConnectTimeout` を変更して header 不在時に 5 秒既定にする案。caller 側で意図せず長時間 unary を撃った場合に切れる副作用が大きく、現行の「caller がタイムアウトを所有する」契約が壊れる。FE 側で明示する方が筋が通る。
- **alt-backend の `fetch_feed_driver.go:214` の相関サブクエリを LATERAL に置換**: `(SELECT a.id FROM articles a WHERE a.feed_id = f.id ... LIMIT 1)` が cursor pagination の各行で走っており、LIMIT N 件分の相関 N+1 サブクエリになっている。ただし backend `feed_read_perf.total_ms` は 3-50ms で十分速く、本件の体感問題には未到達。EXPLAIN ANALYZE と Atlas index 追加 (`articles(feed_id, deleted_at, created_at DESC)`) を伴うため別 PR に切り出す。
- **`useStreamingFeedStats` の visibilitychange 強化**: 既に [[000881]] で `pagehide` / `pageshow` / `visibilitychange` が実装済み。BFF ログで観測される 5 分単位の stream 切断は backend 側の deadline で、FE singleton は機能している。追加の FE 改変は不要。
- **iPhone の `__data.json` 17 連ポーリング抑止**: SvelteKit 2 の deferred Promise transport の本来挙動。urt < 0.01s かつ全 200 で実害なし。`invalidate()` 呼び出しではないことを grep で確認済 (本 ADR scope 外の `loop/+page.svelte:127` 等のみ)。修正対象から除外する。

## Consequences

### Pros

- backend rolling restart 直後の 502 burst (今回 04:28-04:44 UTC で 11 件確認) が、今後は FE→BFF で 5 秒の clean な timeout に変わり、ユーザー操作のリトライで回復する経路に変わる。
- FetchArticleContent が 8 秒で deadline に到達した場合でも、ユーザーには `Source content unavailable. Showing summary.` の落ち着いたフォールバック + `feed.description` 全文が表示され、card が「壊れた」と感じない。
- `Connect-Timeout-Ms` を caller 側で明示することで、各 RPC のタイムアウトが call site に局所化される。長時間 fetch (`fetchArticleContent: 120s`) と短時間 read (`getUnreadFeeds: 5s`) の差が型レベル / 引数レベルで読み取れるようになる。
- listing.test.ts によって今後 timeoutMs を外す regression が unit test レベルで検出可能。

### Cons / Tradeoffs

- 5 秒は backend `feed_read_perf` の p99 (50ms 程度) に対して保守的に大きい。pgbouncer や backend の rolling restart 中、5 秒以内に upstream が回復しないケースでは clean timeout 化されず client error になる。回復まで待たせる UI は別途検討が必要。
- VisualPreviewCard の click-driven フォールバック挙動は unit test で検証されておらず (`describe.skip` 済)、Playwright E2E が来るまでは markup 変更の retreat coverage は限定的。
- `track_reactivity_loss` を踏む `onMount` 内 `.then().catch()` パターンは VisualPreviewCard に残存。本 ADR では markup 層の対応に留め、component の async 化リファクタを別 ADR に積み残し。
- backend `fetch_feed_driver.go:214` の相関サブクエリ N+1 は引き続き残るため、極端な負荷時には次のボトルネック候補。`feed_read_perf` の連続観測でしきい値超過したら別 ADR で LATERAL JOIN + index で対応する。
- インフラ側 (recap-evaluator / meilisearch / ClickHouse の高負荷) は本 ADR scope 外。CPU steal が backend tail を広げる可能性は残る。
