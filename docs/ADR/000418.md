# ADR-000418: Knowledge Home の broken projection 復旧を event log replay 手順として定義する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — knowledge projector, projection checkpoint, backfill/replay 運用
- alt-frontend-sv (SvelteKit) — projection が空のとき empty state を表示する consumer

## Context

Knowledge Home は `knowledge_events` を append-only event log、`knowledge_home_items` や `today_digest_view` を rebuildable projection として設計している。しかし運用時に projection writer が壊れると、UI は empty state を返し続ける一方で、復旧手順が文章化されていなかった。

今回の障害では `knowledge_events` は保持されていたが、`knowledge_home_items` は空で checkpoint も進んでいなかった。この状況で projection table を直接手修正すると、event log を正本とする設計と矛盾し、再発時の復旧可能性も落ちる。

## Decision

**Knowledge Home の read model は正本扱いせず、障害復旧は `knowledge_events` を保持したまま projection と checkpoint を破棄して replay する手順を標準とする。**

### 変更内容

1. `docs/runbooks/knowledge-home-projection-recovery.md` を追加した。
2. runbook に `knowledge_events` は消さず、`knowledge_home_items` / `today_digest_view` / `recall_candidate_view` / projector checkpoint だけを reset する手順を記載した。
3. runbook に復旧前後の確認 SQL とログ確認ポイントを明記した。
4. PgBouncer を維持したまま replay する前提を文書化した。

### 設計判断

1. **event log を唯一の真実源とする** — immutable data model の前提を運用手順にも反映する。
2. **projection は捨てて作り直す** — current-value table を手修正するより、replay の成立性を守る。
3. **checkpoint は運用状態であって業務データではない** — projector 進捗だけを reset 対象にし、正本から再構築する。

## Consequences

### Pros

- Knowledge Home が壊れても replay-first で一貫した復旧ができる
- immutable / append-first 設計の意図がコードだけでなく運用にも反映される
- 将来の projection version 切替や compare/swap 運用にもつながる

### Cons / Tradeoffs

- 運用者は projection を disposable として理解する必要がある
- replay 中は一時的に Home が空または遅延状態になる可能性がある
