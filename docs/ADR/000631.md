# ADR-000631: Augur eval ベースライン修正 — citation 伝搬・intent 露出・BM25 計測の 3 層を貫通させる

## Status

Accepted

## Date

2026-04-06

## Affected Services

- rag-orchestrator (Go)
- search-indexer (Go)

## Context

[[000630]] で導入した eval harness によるベースラインベンチマーク（12 golden cases）の結果、12/12 件が FAIL となった。分析の結果、失敗の中心は「モデル品質」ではなく、**retrieval 計測・citation 配線・intent 露出の 3 層がそれぞれ独立して壊れている**ことが判明した。

### 障害 1: BM25 計測不全（BM25 Zero Rate = 1.000）

`RetrieveContextOutput` と `AnswerDebug` に BM25 ヒット件数のフィールドがなく、eval adapter が常に 0 を返していた。BM25 検索自体は内部で動作している可能性があるが、pipeline から eval への計測パスが断裂しており、実態を観測できない状態だった。

### 障害 2: Citation 伝搬断裂（10/12 件で citations = 0）

`output_validator.go` の `Validate()` で JSON パースに失敗した場合、`extractAnswerOnly()` が answer テキストのみを回収し、citations を nil で返していた。LLM は実際に `[1]`, `[2]` 等の参照マーカーを含む回答を生成しており、JSON の `"citations"` フィールドにもデータが存在していたが、truncated JSON の fallback パスで捨てられていた。

### 障害 3: Intent 未露出（Intent Accuracy = 0.000）

rag-orchestrator 内部では `QueryClassifier` による intent 分類が正常に動作していたが、Connect-RPC handler の `StreamChatResponse` の `done` イベントに intent フィールドがなく、eval runner が intent を取得できなかった。

## Decision

**3 つの独立した配線修正を行い、eval の計測精度を実態に合わせる。**

### 1. BM25HitCount の pipeline 貫通

- `RetrieveContextOutput` に `BM25HitCount int` を追加
- `retrieve_context_usecase.go` で `len(sc.BM25Results)` をセット
- `AnswerDebug` に `BM25HitCount int` を追加
- `promptBuildResult` に `bm25HitCount` を追加し、Execute/Stream で debug にマップ
- eval adapter で `output.Debug.BM25HitCount` を `result.BM25HitCount` に接続

### 2. Citation fallback 回収

- `output_validator.go` に `extractCitationsOnly()` 関数を追加
- JSON の `"citations":[...]` 部分を部分パースで抽出
- truncated JSON でも閉じ括弧を補完して解析を試みる
- `extractAnswerOnly()` fallback パスで `extractCitationsOnly()` も呼び、`LLMAnswer.Citations` に格納

### 3. Intent の proto 露出

- `proto/alt/augur/v2/augur.proto` の `DonePayload` に `string intent = 3` と `string strategy = 4` を追加
- `buf generate` で Go/TypeScript の生成コードを更新
- Connect-RPC handler の `done` イベント変換で `output.Debug.IntentType` と `output.Debug.StrategyUsed` をセット
- eval runner の `done` case で `result.IntentClassified = d.Intent` を取得

## Consequences

### Pros

- **Pass Rate: 0% → 50%**: 12 件中 6 件が新たに PASS（citation 回収と intent 露出の効果）
- **Citation 付き回答: 0/12 → 8/12**: `extractCitationsOnly()` により、truncated JSON からも citation を回収
- **Intent Accuracy: 0% → 71.4%**: `DonePayload.Intent` により、intent 分類結果が eval に到達
- **Fallback Rate: 33% → 25%**: 微改善（citation 回収により一部ケースが fallback から復帰）
- **計測の信頼性向上**: eval 結果が実態を反映するようになり、以後の改善を正確に測定可能

### Cons / Tradeoffs

- **BM25 Zero Rate は 1.0 のまま**: 計測パスは整備したが、実際のヒット件数がまだ 0 の可能性がある。`localizedAttributes` が既存インデックスに反映されるには再インデックスが必要
- **残存 FAIL 6/12**: follow-up 系（照応解決）と clarification が未改善。`WithQueryPlanner` の DI 配線が次の改善レバー
- **proto 変更**: `DonePayload` にフィールド追加。後方互換性あり（新フィールドはオプショナル）

### ベンチマーク結果の代表的改善

| ケース | Before | After |
|--------|--------|-------|
| 因果分析クエリ（石油危機） | 1237 文字, 0 citations | 1617 文字, 7 citations |
| LLM 比較クエリ | 765 文字, 0 citations | 1727 文字, 7 citations |
| 合成クエリ（芸術） | 1947 文字, 0 citations | 2224 文字, 7 citations |
| 技術 deep-dive（Transformer） | 1859 文字, 0 citations | 1727 文字, 7 citations |
