# ADR-000525: tag-generator に process-local empty-extraction tracker を追加し Poison Pill 記事をスキップ

## Status

Accepted

## Date

2026-03-21

## Affected Services

- tag-generator (Python)

## Context

tag-generator は記事流入に対して十分なスループットを持っている（36,798 記事中 36,791 件タグ付与済み、99.98%）。しかし、**永続的にタグ抽出が空になる 5 件の記事**（テスト記事 3 件、Suspicious Patterns 1 件、feed_id NULL 漏れ 1 件）が毎サイクル（3 分間隔）で再フェッチされ続け、以下の問題を引き起こしていた。

### 定量的な影響

| 指標 | 値 |
|------|------|
| 1 日あたりの無駄な処理回数 | 1,010 回（202 サイクル × 5 記事） |
| `Sanitized input too short` ログ/日 | 808 回 |
| `Input sanitization failed` ログ/日 | 202 回 |
| スケジューラ sleep 間隔 | 本来 idle (300s) であるべきところ active (180s) で動作 |

### Poison Pill メカニズム

```
ListUntaggedArticles → 記事フェッチ → sanitization 失敗 (tags=[])
→ article_tags に INSERT されない → 次サイクルで同じ記事が再度返される → ∞ ループ
```

`BatchProcessor.process_articles_as_batch` の `if not outcome.tags: continue` で記事がスキップされるが、失敗の記録が残らないため `ListUntaggedArticles`（`WHERE NOT EXISTS article_tags`）が永久に同じ記事を返し続ける。

## Decision

**`BatchProcessor` に process-local な empty-extraction カウンタを追加し、`max_empty_extraction_retries`（デフォルト 3）回連続で空タグを返した記事をそのプロセスの生存期間中スキップする。**

### 検討した代替案

| 案 | 評価 |
|---|---|
| alt-backend 側に `tag_generation_skip_list` テーブル追加 + RPC | 永続的で確実だが、クロスサービス変更 + DB マイグレーションが必要。オーバーキル |
| `article_tags` にセンチネル行を INSERT | データモデルを汚染する。Knowledge Home のタグ表示に悪影響 |
| tag-generator の Redis に skip set を永続化 | 中間的だが、Redis 障害時の挙動が複雑になる |

**選定理由:**

- alt-backend の変更不要（クロスサービス変更なし）
- DB マイグレーション不要
- `BatchProcessor` に閉じた最小限の変更
- プロセス再起動後は 3 サイクル（約 9 分）で自動再学習するため、永続化不要

## Implementation

### 変更ファイル

| ファイル | 変更内容 |
|---------|---------|
| `tag-generator/app/tag_generator/infra/config.py` | `max_empty_extraction_retries` 設定追加（デフォルト 3） |
| `tag-generator/app/tag_generator/batch_processor.py` | `_empty_extraction_counts` dict + フィルタ/カウント/リセットロジック |
| `tag-generator/app/tests/unit/test_batch_processor_poison_pill.py` | 新規テスト 7 件 |

### 処理フロー

1. `process_articles_as_batch` の先頭で `_empty_extraction_counts[article_id] >= threshold` の記事をフィルタ除外
2. `extract_tags_with_metrics` が空タグを返した場合、該当 article_id のカウンタをインクリメント
3. カウンタが閾値に到達した時点で 1 回だけ warning ログを出力
4. タグ抽出が成功した場合はカウンタをリセット（成功時の削除を必須とし、辞書の肥大化を防止）
5. スキップされた記事は `total_processed` に含めない → スケジューラが正しく idle interval に切り替わる

### ログ出力

```
# 閾値到達時（1 回だけ）
warning: Article reached empty-extraction skip threshold  article_id=xxx  failure_count=3  threshold=3

# 以後のサイクル
info: Skipped articles after repeated empty extraction  skipped_articles=5  threshold=3
```

## Consequences

### Pros

- 毎サイクルの無駄な sanitization ログ（1,010 回/日）が 3 サイクル後に消滅
- スケジューラが正しく idle interval（300s）で動作し、CPU 消費を削減
- `failed` 統計には影響しない（empty extraction は例外失敗ではないため現行仕様を維持）
- 新規記事のタグ付与には一切影響なし

### Cons / Tradeoffs

- **プロセス再起動で状態消失**: 再起動後 3 サイクル（約 9 分）は poison pill が再処理される。許容範囲内
- **閾値到達後の救済なし**: 記事の本文が修正されても、同一プロセス中は再評価されない。救済にはプロセス再起動が必要
- **メモリ常駐**: article_id ごとの辞書はプロセス生存中増え続ける可能性があるが、成功時に削除するため実質的には永続失敗記事のみが蓄積（現状 5 件、上限は記事総数に比べ無視できる）
- **DB 上の未タグ記事数は変わらない**: 7 件の未タグ記事は DB に残り続ける。表示上の対策が必要な場合は別途対応
