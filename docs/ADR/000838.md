# ADR-000838: Recap 2-bucket collapse を 30-class 再訓練と sub-cluster singleton 降格で解消する（self-hosted CUDA runner 上）

## Status

Accepted

## Date

2026-04-23

## Affected Services

- recap-subworker (Python) — classifier 再訓練 + 起動時 fail-closed + sidecar metadata + 訓練 Dockerfile の GPU 対応
- recap-worker (Rust) — sub-splitter singleton 降格 + clustering fallback threshold 緩和

## Context

[[000835]] が 2-bucket collapse の 3 層同時不整合（17-class classifier / 17-key 既定 thresholds file / `transformers 5.5.1` non-persistent buffer regression）を記録し、3 段階の修正順序（即時 / 中期 / 恒久）を決めた。本 ADR はそのうち **中期 (stage 2) と恒久 (stage 3) を一括実装した作業の記録** である。

### 事前調査で確定した事実

- `transformers 5.x` の non-persistent buffer 破壊は [#43950](https://github.com/huggingface/transformers/issues/43950) / [#44534](https://github.com/huggingface/transformers/issues/44534) で 2026-03-09 までに closed、但し upstream は「v5 breaking change として custom model 側で追従せよ」の立場で fix は未 merge。影響対象は `register_buffer(persistent=False)` を持つ custom model のみで、インストール済 `transformers/models/{xlm_roberta,bert,roberta}/` には該当 call site が 0 件。BGE-M3 の基底となる XLM-RoBERTa encoder は無関係と確定したため、訓練環境は `transformers>=5.6.1` + `sentence-transformers>=5.4.0` を採用する。
- 既存 `data/training_data.csv` は 17 genre のみ (2026-01 collect_data.py 時点のマッピング上限)。30 genre 訓練のため `data/golden_classification.json` (`taxonomy_version=genre-fixed-30-v1`、`items` 3600 件、primary_genre 120/genre で完全 balanced) + `learning_machine/data/silver_teacher_v0_*.jsonl` から 3783-row の `training_data_30class.csv` を合成する。
- 訓練は self-hosted CUDA runner 上で行う。訓練時間帯は news-creator を compose-stop して GPU を占有する合意をユーザから取得済み。BGE-M3 FP16 エンコードが成立するメモリヘッドルームは事前 dry-run で確認済み。

### 再訓練後の検証結果

| 指標 | 値 |
|---|---|
| model `classes_` 長 | 30 (taxonomy 完全被覆) |
| Test Macro F1 | 0.9040 |
| Train/Test Gap | 0.0898 (< 0.10 OK) |
| sidecar `sklearn_version` | 1.8.0 (runtime と minor 一致) |
| sidecar `transformers_version` | 5.6.1 |
| feature_dim | 1224 (1024 embedding + 200 SVD) |
| 実 job 検証 (post deploy) | top_genre が 11 class に分散 (`music_audio` 251 / `software_dev` 220 / `film_tv` 196 / `health_medicine` 147 / `politics_government` 39 / その他 6 class)。`consumer_tech` + `politics_government` の合計は 4% (基準 < 60% pass) |

### 再訓練後に顕在化した下流 2 問題

1. **sub-cluster singleton**: MiniBatchKMeans が k=10 で分割したクラスタの複数が 1 article に落ち、その subgenre (`{parent}_{id:03}`) が downstream `MIN_FALLBACK_DOCUMENTS=2` で弾かれ 16 件 clustering skip。[[000835]] forensics は「設計通り」としていたが、job 全体が `genres_stored=0` で失敗するのは受け入れられない。
2. **news-creator:9443 mTLS Connection refused**: `alt-pki-agent-news-creator-1` が news-creator recreate で netns を失い unhealthy 化し、`v1/summary/generate/batch` への POST が `Connection refused` で全て失敗 (49 genres missing)。pki-agent 再作成で解消し 10/30 genres stored まで回復したが、singleton 降格が未対応のため残 15 件が依然 failed。

本 ADR は (1) を recap-worker 側で恒久的に潰す。(2) は compose 側の運用手順（`docker restart` ではなく `compose up --force-recreate` を使う）を `docs/runbooks/recap-stack-restart.md` に別途記録する。

## Decision

### 1. 30-class classifier を再訓練する (stage 2)

- 訓練コーパス: `golden_classification.json` (1767 JA+EN 対) + `silver_teacher_v0_ja.jsonl` (292) + `silver_teacher_v0_en.jsonl` (2034) から `scripts/build_30class_training_csv.py` で重複除去して 3783 行を出力。最小 46 記事/genre。
- 訓練パイプライン: 既存 `train.py` を使用 (BGE-M3 embedding + TF-IDF 1000-feature + TruncatedSVD 200 components + StandardScaler + `CalibratedClassifierCV(sigmoid, cv=3)` over `LogisticRegression(class_weight='balanced')`)。Cleanlab は optional 依存で try/except fallback 継続。
- CLI: `--language {ja,en}` (既定 ja) + `--output-dir` を追加。artefact 名を `_ja` / `_en` suffix 付き (`genre_classifier_ja.joblib`, `tfidf_vectorizer_ja.joblib`, `genre_thresholds_ja.json`) に変更。SVD と StandardScaler は言語非依存のため `tfidf_svd.joblib` / `feature_scaler.joblib` のまま。
- sidecar: `genre_classifier_ja.meta.json` に sklearn / transformers / sentence-transformers version、`trained_at` ISO8601、`language`、sorted `classes`、`feature_dim`、training CSV の sha256、`embedding_model_id`、device を記録。
- 訓練環境: `Dockerfile.training` を CUDA base image 上に uv-managed Python 3.14 + `torch` nightly (cp314 wheel は CUDA 12.x nightly index でのみ提供)。production runtime image (`Dockerfile.recap-subworker`) は CPU torch のまま据え置く。

### 2. 起動時 fail-closed observability を追加する (stage 3)

- `ConfigValidationError(ClassificationError, ValueError)` を `recap_subworker/domain/errors.py` に追加。Pydantic `model_validator` が `ValueError` サブクラスを `ValidationError` に wrap する既存挙動に乗せる。
- `Settings._validate_classifier_cardinality`: `classification_backend=='joblib'` 時に `genre_classifier_model_path_{ja,en}` を joblib load し `classes_` 長 < `genre_baseline_cardinality` (default 15、env `RECAP_SUBWORKER_GENRE_BASELINE_CARDINALITY` で上書き可) なら `ValueError` raise。corrupt payload や `classes_` 未定義は fall-through (peer validator / runtime guard の責務に委譲)。
- `GenreClassifierService._ensure_model`: joblib load 後に `<model>.meta.json` を EAFP で読み `ClassifierMetadata` frozen dataclass にする。sidecar の `sklearn_version` と runtime `importlib.metadata.version("scikit-learn")` の minor (X.Y) 一致を検査し不一致で `ConfigValidationError`。sidecar 欠落は `logger.warning` のみ (backward compat)。
- `self.model.multi_class = 'ovr'` の runtime パッチと `print("DEBUG: Force patching ...")` を削除。1.8 で再訓練したモデルには `multi_class` 属性は存在せず、無条件代入は silent mutation だった。

### 3. recap-worker の sub-splitter で MiniBatchKMeans singleton を降格する (補完)

- `pipeline::select::clustering::subcluster_large_genres`: KMeans 出力のクラスタ毎記事数を count し、`MIN_SUB_CLUSTER_SIZE=2` 未満のクラスタは subgenre label を付けない。該当記事は親 genre 名のまま downstream に流れる。ログに `labeled` / `kept_parent` / `tiny_cluster_count` を追加。
- `clients::subworker::types::MIN_FALLBACK_DOCUMENTS` を `2 → 1`。defense-in-depth として、仮に 1 article の genre が下流に届いても single-cluster fallback で summary まで到達する。`min_documents_per_genre` 別枠で stricter floor を求める caller は影響を受けない。

### 検討した代替案と却下理由

- **`transformers` を v4.45 pin に retro する**: [[000835]] の immediate stage の再現手段だが、upstream の v5 breaking change 方向と逆行し将来セキュリティパッチ追従コストが増す。BGE-M3 は non-persistent buffer を使わず H1 の影響を受けないため不要と判断。却下。
- **既存 `train_classifier_gpu.py` (PyTorch BCE/focal) で再訓練する**: pickle dataset 前提 + `prepare_dataset.py` を先行させる 2-stage 工程。sklearn `train.py` の方が CalibratedClassifierCV + threshold optimization を含む完成パイプラインであり、artefact 互換も取りやすい。今回は却下 (将来の distillation 実験用に scripts は残す)。
- **sub-splitter 自体を停止する (200+ genre でも分割しない)**: `consumer_tech` が 250+ 記事のとき 10 cluster 下に 50 件ずつ summarize する価値は実測で示されており、廃止は summary 粒度が荒れる。singleton 降格のみで十分と判断。却下。
- **`MIN_FALLBACK_DOCUMENTS` を据え置き、1-article は強制 error のまま**: 既存挙動の保守性は高いが、rare な parent genre (分類で 1 件しか引けなかった場合) でも job 全体が失敗する。fallback で summary 化する方が reversible かつユーザに見える形 (bullets=1) になる。却下。
- **compose 側で pki-agent-news-creator の自動復旧を組む**: 本 ADR の範囲外。[[000826]] deploy-model self-check に沿って runbook 側で扱う。

## Consequences

### Pros

- 2-bucket collapse の実測解消: 2026-04-14 以降 8 日間の連続失敗が終わり、同じ 3-day window kick で top_genre が 11 class に分散 (`consumer_tech` + `politics_government` 合計 4%)。
- 30-class taxonomy が物理的に predictable になり、既存 thresholds default file (`genre_thresholds.json`) 依存の二重構成リスクが縮退。`_ja` suffix の factory wiring ([[000828]]) とも整合。
- sidecar metadata により sklearn / transformers version drift が起動時 fail-closed に変わる。次回 library bump で同種の silent regression を起こしにくくなる。
- runtime `multi_class = 'ovr'` パッチと `DEBUG` stdout print が消え、prod ログが構造化 structlog だけになる。
- recap-worker の sub-splitter が singleton クラスタを作らなくなり、MiniBatchKMeans の k-balance に依存せず summary が通る。`genres_stored` が 0 になる worst-case が 1 層防御される。

### Cons / Tradeoffs

- 訓練コーパスは golden 3600 + silver ~2300 の比較的小規模。train/test 分離内で F1=0.90 は得られたが、本番 articles との分布差は未計測。`recap-evaluator` の genre baseline 追跡で 7 日運用して乖離を観測する必要がある (別フォロー)。
- `genre_baseline_cardinality=15` は現状の 30-class artefact で十分緩いが、30 genre 中 12+ 地平 (例: 17-class に戻ってしまう障害) は依然検出できない。prod 反映後は compose env で `30` に引き上げる運用ガイドを runbook に追記する予定 (次 PR)。
- `Dockerfile.training` は `torch` nightly (CUDA build) に依存するため、upstream API 変更で build が落ちるリスクがある。訓練は定常運用でなく ad-hoc のため許容。
- `MIN_FALLBACK_DOCUMENTS=1` は summary 品質の下限を下げる。1-article summary は定性的に「短い」状態になる可能性があるが、0-bullet で skip されるよりユーザ価値は高い。
- `multi_class = 'ovr'` パッチ削除後、1.7.x 以前の pickle を踏んだ場合は boot 時に `ConfigValidationError` で落ちる (sidecar なし → warn のみのため load は通るが、以降の挙動は未保証)。旧 pickle は sidecar と一緒に再訓練して差し替える前提で運用する。
- sub-splitter 降格により `consumer_tech_001..010` 等の subgenre bucket は singleton を吸わなくなるが、親 `consumer_tech` bucket の記事数が変動する。既存の per-genre thresholds tuning は影響ない (thresholds は実 genre 単位) が、subgenre 単位の summary 粒度が一部だけ粗くなる。
