# ADR-000892: Surface Planner v2 を本番で観測可能にし、recap.topic_snapshotted.v1 emit の foundation と reproject 検証手順を固める

## Status

Accepted

## Date

2026-05-08

## Affected Services

- knowledge-sovereign (Go)
- rag-orchestrator (Go)
- recap-worker (Rust)
- observability (compose / Prometheus / Grafana)
- docs (runbook)

## Context

`docs/plan/knowledge-loop-completion-00-index.md` および `docs/plan/knowledge-loop-completion-01-v2-planner.md` で示した Knowledge Loop Completion Phase 1 のゴールは、Surface Planner v2 を「コード上は存在する」状態から「本番 event log の signal で placement が変わり、観測でき、reproject で再現できる」状態に進めることだった。

Wave 4-A ([[000855]]) と Wave 1〜3 ([[000850]] / [[000851]] / [[000852]] / [[000853]] / [[000854]]) で次は landed 済みだった:

- `EventLogSurfaceScoreResolver` の本番 projector 配線 (`knowledge-sovereign/app/main.go:117`)
- `SurfaceScoreInputs` 構造体と Atlas migration `00013_add_surface_planner_v2_columns.sql`
- `seedActTargets()` による `target_type=recap` route 生成
- `cross_user_isolation_violation_total` counter 定義
- rag-orchestrator の `KnowledgeEventEmitter` interface + Noop default + sovereign_client + augur handler の warn-and-continue 統合
- recap-worker 側の `KnowledgeSovereignClient` と Pact consumer test (`pacts/recap-worker-knowledge-sovereign.json`)

しかし次のギャップが残っており、本番で v2 ratio が dashboard に出ない、reproject 後の signal を検証する手段がない、emit 失敗を観測できない状態だった:

1. resolver / 駆動 SQL の不変条件が test で回帰防止されていなかった (replay determinism、F-001 counter 増分、SQL の `WHERE user_id = $1` 物理 bind)
2. rag-orchestrator の augur emit が失敗しても warn log だけが残り、Prometheus alert を低レートで page できなかった。emit failure と projector-side event drop が観測上区別不能
3. recap-worker の `recap.topic_snapshotted.v1` emit は client が単独で存在するだけで、retry 安全な snapshot id 導出も per-cluster 発火経路もなかった
4. `observability/prometheus/prometheus.yml` に `knowledge-sovereign:9501` scrape job が無く、`compose/observability.yaml` が `/etc/prometheus/rules` ディレクトリ自体を mount していなかったため、`rule_files: /etc/prometheus/rules/*.yml` の glob が container 内で空マッチし、`KnowledgeLoopCrossUserIsolationViolation` を含む alert / recording rule がすべて silent に load 失敗していた。さらに `knowledge-loop-rules.yaml` は `observability/alerts/` に置かれており、glob 対象外だった
5. v2 cutover reproject の post-check は checkpoint 一致と `dangling_supersede_refs=0` のみで、「v2 placement が立っているか」「evidence 非ゼロ率は何 % か」「act_targets に recap target が入っているか」を確認する手段が runbook に記載されていなかった

## Decision

Phase 1 の残ギャップを 5 つの semantic group に分けて land する:

### 1. knowledge-sovereign 不変条件の回帰テスト固定

`score_resolver_event_log_invariants_test.go` を追加し、resolver / 駆動 SQL のソース自体は touch せず、回帰防止のテストだけを足す:

- **replay determinism**: 同一 event fixture を 2 回 resolve した結果の `SurfaceScoreInputs` JSON が `json.Marshal` で byte-level 一致
- **F-001 metric**: 振る舞いの `TestEventLogResolver_F001CrossUserMismatchReturnsEmpty` だけでは page 信号が見えないため、`crossUserIsolationViolationTotal` の delta が exactly 1 であることを `prometheus/client_golang/prometheus/testutil` 経由で固定
- **SQL physical bind**: `runtime.Caller` で `driver/sovereign_db/read_events.go` を読み、`ListKnowledgeEventsForUserInWindow` の関数本体に `WHERE user_id = $1` が残っており、かつ `OR user_id IS NULL` を含まないことを source-level regression として固定

resolver / driver 本体を **diff 0** に保つことで「invariant の test だけ足す」最小侵襲な PR とした。

### 2. rag-orchestrator emit failure metric

`rag_orchestrator_knowledge_event_emitter_failure_total{event_type}` counter を `internal/adapter/sovereign_client/metrics.go` に新設。augur handler の warn-and-continue path で `IncEmitterFailure("augur.conversation_linked.v1")` を warn log と並べて呼ぶ。Echo server (PORT=9010) に `/metrics` endpoint を `promhttp.Handler()` で配線し、prometheus default registry を expose する。

projector 側の `knowledge_loop_projector_event_dropped_total{reason}` とラベル空間を意図的に分離する: emit-side の RPC 失敗 / timeout / validation reject と、projector-side の `user_mismatch` / `article_mismatch` を独立した metric にすることで、片方の悪化が他方に紛れる事故を構造的に避ける。Prometheus 公式 `level:metric:operations` 命名規約に整合する形にした (level=`rag_orchestrator`、metric=`knowledge_event_emitter`、ops=`failure_total`)。

unit test (`emitter_failure_metric_test.go`) で `failingEmitter` を注入し、conversation 作成は success を返しつつ counter delta が exactly 1 になることを assert。warn-and-continue semantics の重要な不変条件 (emit failure が caller flow を壊さない) を test で永続化する。

### 3. recap-worker emit foundation

recap-worker の persist stage は依然として複雑で、`FinalSectionPersistStage::persist()` への直接配線は scope を膨張させる。代わりに **すべての構成要素を test ファースト で land し、scheduler が JobContext.user_id を populate するだけで live になる** 形を取る:

- `clients/knowledge_sovereign/snapshot_id.rs`: RFC 9562 §6.5 UUIDv5 で `(user_id, cluster_id, window_start, window_end)` から決定論的に snapshot id を導出。frozen namespace UUID (`Uuid::from_bytes([0x18, 0xc0, 0x67, 0x40, ...])`) を導入し、絶対に変更しない (変更すると過去 dedupe key と衝突して retry idempotency が壊れる)。`Uuid::new_v5(&NS, name.as_bytes())` の出力を 6 件のテストで pin し、frozen reference value と version/variant 検査も含めた
- `clients/knowledge_sovereign/publish.rs`: per-cycle helper。empty top_terms cluster は no-op signal として skip、user_id / tenant_id が nil なら早期 return (defence-in-depth、JobContext-population バグの早期検知)、emit 失敗は `tracing::warn!` で warn-and-continue。`wiremock` で 6 ケース (success / empty terms / RPC 失敗 / nil user / nil tenant / retry idempotency) をテスト
- `JobContext` を optional `user_id: Option<Uuid>` / `tenant_id: Option<Uuid>` で拡張、`with_user_scope(user, tenant)` builder を提供。system batch (trigger_source="system") は両方 None のまま、publish helper が gracefully skip するので既存挙動を壊さない
- `Cargo.toml` の uuid crate に `v5` feature を追加
- compose に `RECAP_KNOWLEDGE_EMIT=${...:-true}` (rag-orchestrator default true と整合) と `RECAP_KNOWLEDGE_SOVEREIGN_URL` を expose し、persist-stage 配線が landed した瞬間に live になるようにした

### 4. observability wiring

3 つの blocking ギャップを同 PR で閉じる:

- `prometheus.yml` に `knowledge-sovereign:9501` (Knowledge Loop projector counters) と `rag-orchestrator:9010` (新 emitter_failure counter) の scrape job を追加。container-internal port を直接 scrape し、host port (9511 / 9010) は外部 probe 用なので Prometheus は使わない
- `compose/observability.yaml` の prometheus volume に `../observability/prometheus/rules:/etc/prometheus/rules:ro` mount を追加。これがないと `rule_files: /etc/prometheus/rules/*.yml` の glob が container 内で空マッチし、既存の `pki-agent-alerts.yml` / `knowledge-home-slo-alerts.yml` も含めて全 rule が silent に load 失敗していた
- `git mv observability/alerts/knowledge-loop-rules.yaml observability/prometheus/rules/knowledge-loop-rules.yml` で物理移動 (拡張子 `.yaml` → `.yml`、ディレクトリ alerts/ → prometheus/rules/)。glob にマッチさせるため。`alerts/` の他のファイル (ai-pipeline / backup / knowledge-home-slo / rask-system / recap-genre) は本 ADR の scope 外で別途整理する

`docker run --entrypoint promtool prom/prometheus:v3.1.0 check config` で 3 rule files / 17 rules すべて load 成功を検証。これで `KnowledgeLoopCrossUserIsolationViolation`、`KnowledgeLoopProjectorLag`、`KnowledgeLoopEventDropSpike` の 3 alert と recording rule `knowledge_loop:surface_bucket_v2_ratio_5m` が稼働 Prometheus に load される。

### 5. reproject runbook v2 post-check

`docs/runbooks/knowledge-loop-reproject.md` の post-check に 3 セクションと 1 verification 手順を追加:

- **placement breakdown**: `SELECT surface_planner_version, surface_bucket, COUNT(*) ...` で v2 row が立っているか確認。100% v1 結果は「resolver が zero-value SurfaceScoreInputs にフォールバックした」signal
- **evidence 非ゼロ率**: `topic_overlap_count` / `has_augur_link` / `recap_cluster_momentum` の present 率と `continue_context` / `change_summary` / `source_observed_at` の NULL 率を JSONB filter で集計。cutover ticket に記録する rollout-visible evidence
- **act_targets recap target 件数**: `jsonb_array_elements(act_targets)` で `target_type='recap'` を抽出。`topic_overlap_present > 0` なのに 0 件なら snapshot id が UUID validation で弾かれている smell
- **deterministic-replay verification**: TRUNCATE + 0-reset + replay を 2 回回し、`(surface_bucket, surface_score_inputs::text, act_targets::text, why_text, why_kind, change_summary::text)` の TEXT 化値で `FULL OUTER JOIN ... IS DISTINCT FROM` 比較。divergent_rows = 0 を Phase 1 §4 acceptance criterion として固定

### 検討した代替案

- **OTel metrics (rag-orchestrator)**: rag-orchestrator は既に OpenTelemetry trace / log を持つが metric provider は未配線で、knowledge-sovereign / mq-hub と同じ Prometheus client_golang に揃えた方が dashboarding と alerting の語彙が一貫する。OTel metric への移行は別 ADR で扱う
- **structured log のみで emit failure を観測**: warn log は既に存在したが、Prometheus alert を低レートで page するには counter が必要なので採用しなかった
- **recap-worker の persist stage に直接 emit 配線**: pipeline の orchestrator / executor / dispatch は層が深く、JobContext から user_id を引き回す scope 拡張が無視できない。**foundation だけ land して scheduler が user_id を populate する経路を別 PR に切り出す** 方が、blast radius が小さく review しやすい
- **rules ファイルを `alerts/` に置いたまま compose で `alerts/` を mount**: prometheus.yml の glob 自体を `*.yaml` に広げる選択肢もあったが、他の `alerts/` ファイル (例: `recap-genre-rules.yaml`) は現状読み込まれていないため scope 拡張になる。Phase 1 では `knowledge-loop-rules.yml` だけ正しい場所に移動し、他は別作業に分離した
- **post-check を Grafana panel に押し込む**: 運用者の習慣 (cutover チケットに数値を書く) と整合しないため、SQL を runbook に直接置いた

### 不変条件の遵守

- **Append-first**: `knowledge_events` のみが真実源泉。projector / publish helper は read model を参照しない
- **Reproject-safe**: publish helper は payload-resident な field (`top_terms` / `cluster_id` / `window_*`) と `JobContext.user_id`/`tenant_id` だけで input を組む。`Utc::now()` を business fact に使わない
- **Event-time purity**: snapshot_id derivation は wall-clock を含まず、recap cycle の window 端点だけから決まる
- **Single emission**: `recap.topic_snapshotted.v1` は recap-worker のみが emit。`HomeItem*` 等とは重複しない
- **F-001 防御**: resolver の `WHERE user_id = $1` 物理 bind を test で固定、cross-user lookup で counter +1 + null-equivalent fallback も test で固定
- **Tenant defence-in-depth**: publish helper は `tenant_id` が nil なら early return、emit せず `event_dropped_total` も増やさない (skip-silently with diagnostic log)

## Consequences

### Pros

- **Phase 1 完了定義の 5 つすべてが本番 verifiable** になった: v2 ratio dashboard、emit failure observability、deterministic snapshot_id、reproject 後の v2 evidence non-zero rate、deterministic-replay 一致
- **invariant test 3 件が回帰防止として永続化**: 将来の refactor が SQL の `WHERE user_id = $1` を緩めても test が落ちる。replay 非決定性 / F-001 counter 静音化も同じく source レベルで catch される
- **emit-side と projector-side の観測軸が分離** (`rag_orchestrator_knowledge_event_emitter_failure_total` vs `knowledge_loop_projector_event_dropped_total`)。片方の degradation が他方に紛れる事故を構造的に避ける
- **recap-worker emit foundation が test ファースト で land**: scheduler が JobContext.user_id を populate するだけで live になる、scope を限定した次 PR が組める。snapshot_id の frozen reference test が namespace 改変による silent idempotency 破壊を catch する
- **Prometheus が初めて `/etc/prometheus/rules` を実際に読む状態に**: 既存の `pki-agent-alerts.yml` / `knowledge-home-slo-alerts.yml` も同時に load されるようになり、`KnowledgeLoopCrossUserIsolationViolation` を含む 17 rules が稼働する
- **runbook の post-check で v2 cutover の成否を数値で説明可能**: 100% v1 fallback を silent fail にしない、deterministic-replay の divergent_rows = 0 を acceptance criterion として固定

### Cons / Tradeoffs

- **recap-worker の emit が live ではない**: scheduler が JobContext.user_id を populate する配線がまだないため、現状 publish_topic_snapshots は持ち上がっているが呼ばれない。foundation は landed しているので「scheduler がユーザを引き渡す path」を別 PR で足す必要がある
- **rag-orchestrator に Prometheus client_golang 直依存を追加**: OTel への将来統一を考えると一時的な技術負債。現状 knowledge-sovereign / mq-hub と語彙を揃える方が dashboard / alert の運用面で価値が大きいと判断した
- **rules dir の volume mount が他の rules も同時に load する**: 副作用として `pki-agent-alerts.yml` / `knowledge-home-slo-alerts.yml` も load されるが、これは元々 mount されている前提で書かれていたため、むしろ「壊れていた observability を直した」副作用
- **`alerts/` ディレクトリの他ファイル (ai-pipeline / backup / rask-system / recap-genre) は本 ADR では触らない**: 同じ misfile の可能性があるが、Phase 1 scope 外として別作業に切り出した。`alerts/` ディレクトリ自体の整理は別 ADR 候補
- **alt-deploy の pacticipant 同期は本 ADR で追加していない**: Wave 4-A landing ([[000855]]) 時点で `c2quay.yml` / `pact-publish.yml` / `run-e2e-suite.yml` / `release-deploy.yaml` の `PACTICIPANT_ROLES` + `PACTICIPANT_COPAIRS_WHEN_PRESENT` に rag-orchestrator → knowledge-sovereign が、Wave 4-B 準備時点で recap-worker → knowledge-sovereign が、それぞれ既に登録済み。Phase 1 PR 群が main に merge され dispatch-deploy が走る瞬間に既存 wiring が機能する
- **24h soak と v2 cutover reproject の実 run はまだ**: Phase 1 の wiring は landed したが、本番 event log 上で `surface_bucket_assigned_total{version="v2"}` が立ち上がる確認は ops 手順として残る
