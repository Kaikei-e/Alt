# ADR-000869: KnowledgeHomeAdminService に `EmitArticleUrlBackfill` admin endpoint を新設し operator が UI から ArticleUrlBackfilled corrective event を発火できるようにする

## Status

Accepted

## Date

2026-04-28

## Affected Services

(frontmatter 参照)

## Context

### 直接の動機

ADR-000867 で `ArticleUrlBackfilled` corrective event とその projector branch を投入した。しかし production sovereign の `knowledge_events.ArticleCreated` は既に 27,198 件分の `dedupe_key='article-created:<article_id>'` を `dedupe_registry` に持っていた状態だったため、operator が `/admin/knowledge-home` 既存 UI の **Trigger Backfill** を押しても、それが発火する `GenerateBackfillEvent` (event_type=`ArticleCreated`, dedupe_key=`article-created:%s`) は sovereign 側 dedupe で **silent-drop** されてしまい、historical 区間 (12,252 件 + 14,946 件) のリペアが起動できないことが Reproject 後の再観測で確定した。

具体的な症状:

- operator: Trigger Backfill 押下 → BFF → alt-backend `TriggerBackfill` → backfill job 起動
- backfill job が `articles` テーブルを walk して `AppendKnowledgeEvent(ArticleCreated, dedupe_key="article-created:<id>")` を発火
- sovereign: `dedupe_registry` に同じ key が既に存在 → success=true / event_seq=0 で no-op return
- 結果: `knowledge_events` に新行が入らない → Reproject しても empty link rate は変わらない

ADR-000867 の `ArticleUrlBackfilled` は **`article-url-backfill:<article_id>` という別 dedupe namespace** を持っているため、event log には append される (sovereign の unique index が namespace 単位で効く)。だが ADR-000867 時点ではこの corrective event を **発火する operator-facing 経路を一切作っていなかった**。Phase D の運用手順は `altctl admin emit-article-url-backfill` を前提にしていたが、その CLI も未実装で、結局 production には reach できなかった。

### なぜ UI が必要か

Phase D の操作手順:

1. operator が backfill を発火する
2. operator が `/admin/knowledge-home` UI で Reproject Full → Swap を回す

step 2 は既に UI ベース。step 1 だけ CLI に分裂すると、operator は `kubectl exec` 相当 / SSH / shell into container を要求される (本番環境にそもそも shell access 経路を持たない)。1 ボタンで step 1 を発火できる UI が運用上の前提となる。

副次的に、CLI を実装すると `altctl` 側に admin auth (kratos session 取得 + admin role check) を二重実装する必要が出るが、UI ベースは既存 `auth-hub` セッション + admin role middleware を再利用できる (security-auditor F-002 重複実装回避)。

### 既存パターンとの整合

`KnowledgeHomeAdminService` は既に同形の admin endpoint を 11 個ホストしている (`TriggerBackfill` / `PauseBackfill` / `ResumeBackfill` / `StartReproject` / `CompareReproject` / `SwapReproject` / `RollbackReproject` / `GetProjectionHealth` / `GetSLOStatus` / `RunProjectionAudit` / `GetSystemMetrics`)。新 RPC を追加する自然な場所であり、auth / DI / Hurl E2E パターンも継承できる。

## Decision

`KnowledgeHomeAdminService` に **`EmitArticleUrlBackfill` Connect-RPC を新設** し、対応する usecase + handler + UI ボタンを実装する。CLI ベース (`altctl admin emit-article-url-backfill`) は採用しない。

### Wire 仕様

```proto
rpc EmitArticleUrlBackfill(EmitArticleUrlBackfillRequest) returns (EmitArticleUrlBackfillResponse);

message EmitArticleUrlBackfillRequest {
  int32 max_articles = 1;  // 0 = process all
  bool  dry_run      = 2;  // count without appending
}

message EmitArticleUrlBackfillResponse {
  int32 articles_scanned       = 1;
  int32 events_appended        = 2;
  int32 skipped_blocked_scheme = 3;  // URL scheme allowlist 防衛
  int32 skipped_duplicate      = 4;  // sovereign dedupe 由来
  bool  more_remaining         = 5;  // max_articles に達して中断
}
```

### Backend 実装の構造

`alt-backend/app/usecase/knowledge_url_backfill_usecase/`:

- `Usecase` は `knowledge_backfill_port.ListBackfillArticlesPort` (既存 `articles` テーブル walk gateway を再利用) と `knowledge_event_port.AppendKnowledgeEventPort` (sovereign client) のみに依存。`pageSize=200` で cursor-paginated に walk
- 各 article について `isHTTPURL` allowlist を通し (security-auditor F-001、projector + FE と 3 層 defense)、通れば `domain.ArticleUrlBackfilledPayload{ArticleID, URL}` を marshal して `domain.KnowledgeEvent{event_type=ArticleUrlBackfilled, actor_type=service, actor_id="knowledge-url-backfill", aggregate_type=article, dedupe_key="article-url-backfill:<id>"}` を append
- `dry_run=true` のときは scheme allowlist だけ走らせて event は append しない (preview 用)
- `max_articles>0` で stopping condition、cursor を残して `more_remaining=true` を返す

`alt-backend/app/connect/v2/knowledge_home_admin/handler.go`:

- `URLBackfillUsecase` interface (Emit のみ) を新設し handler が依存。production wiring は concrete `*knowledge_url_backfill_usecase.Usecase` を渡す。テスト時は stub 注入
- `max_articles < 0` のみ `CodeInvalidArgument`、`= 0` は許容 ("process all" の意味)
- 結果は構造化 logging (articles_scanned / events_appended / skipped_blocked_scheme / skipped_duplicate / more_remaining / dry_run) → audit-able

### Frontend 実装の構造

- `alt-frontend-sv/src/lib/connect/knowledge_home_admin.ts` に `emitArticleUrlBackfill(transport, maxArticles, dryRun)` thin wrapper を追加 (既存 `triggerBackfill` 同形)
- `alt-frontend-sv/src/lib/server/knowledge-home-admin.ts` に `emitKnowledgeHomeArticleUrlBackfill` を追加し BFF 経由で alt-backend に到達 (memory `feedback_no_direct_frontend_backend.md` 準拠)
- `alt-frontend-sv/src/routes/api/admin/knowledge-home/+server.ts` に `action: "emit_article_url_backfill"` ケースを追加 (admin role gate + backendToken 既存パターン継承)
- `useKnowledgeHomeAdmin` hook に `emitArticleUrlBackfill(maxArticles, dryRun)` メソッドと `urlBackfillResult` $state を追加。失敗時は `error` を立てつつ既存 snapshot は壊さない (既存 `compareReproject` パターン継承)
- `/admin/knowledge-home` page header の `Trigger Backfill` ボタン横に `Emit URL Backfill` ボタンを追加。クリック後は `URL backfill: N appended / M scanned · K blocked-scheme · more remaining` の inline summary を表示

### 検討した代替案

#### 1. `altctl admin emit-article-url-backfill` CLI のみ実装 (ADR-000867 Phase D の元案)

却下。本番環境で operator が CLI を叩く経路が無い。Reproject Swap が UI 化されている (ADR-000846 周辺) のと一貫性が取れない。admin auth を CLI に再実装するコストも生じる。

#### 2. 既存 `TriggerBackfill` の挙動を「dedupe namespace を切り替えて」拡張する

却下。`TriggerBackfill` は `ArticleCreated` event を発火する general-purpose な経路で、これに「modes」を持ち込むと event 種別と dedupe namespace の対応が暗黙化する。append-first / event-shape stability を壊す方向。新 RPC で意味論を分離する方が反復可能性 (Phase D を後で別 corrective event でも繰り返せる) が高い。

#### 3. 既存 `TriggerBackfill` の dedupe registry を「manual で truncate して」運用回避

却下。`dedupe_registry` は immutable invariant の barrier。これを運用上 truncate すると、過去のすべての ArticleCreated が「未処理扱い」になり、別 projector が予期しない再実行を発火する可能性がある。Append-first の前提を運用で破る選択は **immutable-design-guard skill** の観点で採用不可。

## Consequences

### Pros

- operator が `/admin/knowledge-home` UI から 1 ボタンで Phase D step 1 を発火できる。本番 SSH 経路無しで履歴復旧が完結する
- 新 RPC は dedupe namespace `article-url-backfill:%s` を強制するため、`article-created:%s` の既存 dedupe records と無干渉。Reproject 後の URL 復旧が deterministic に動く
- usecase が interface ベースで handler から見えるため、handler unit test は sovereign client mock 不要 (4 テストケース、全部 GREEN)
- Pact CDC `TestAppendKnowledgeEvent_ArticleUrlBackfilled` で sovereign 側 wire form を pin。dedupe namespace のスペル drift / payload key drift を CDC で検出可能 ([[000868]] と同じ defense pattern)
- Hurl E2E `72-admin-emit-article-url-backfill.hurl` で route 配線 (admin handler が registry に居る + 引数 round-trip) を検証
- `dry_run=true` で本番 emit 前に preview 可能。large reservoir (production の 12,252 件) を一度に流しても安全
- URL scheme allowlist が usecase / projector / FE の 3 層に揃う ([[000867]] F-001 継承)

### Cons / Tradeoffs

- 既存 `KnowledgeHomeAdminService` がさらに肥大する。RPC 12 個目。長期的には `KnowledgeHomeBackfillService` を分離する余地はあるが、現時点では admin auth boundary が同じなので分割しない
- `max_articles=0` を「all」と解釈する Go-idiom と TS-idiom の差。proto 上は `int32` のためデフォルト値 0 を「unset」扱いするのは proto3 の慣行通りだが、UI/CLI の docstring で明示が必要
- counter 群が int32 なのでキャップは 2,147,483,647 件。articles テーブルが ~30k 件程度なので当面問題ないが、将来 PII 兼 large dataset 化したら int64 化検討
- handler が `urlBackfillUsecase URLBackfillUsecase` interface を持つため、production wiring 側 (DI container) で concrete を interface へ昇格する 1 段の indirection が増える。ただし他の admin usecase (`ReprojectUsecase` 等) と同形のため一貫性は取れている

### Phase D 運用手順 (deploy 後)

1. `git push origin main` → `dispatch-deploy.yaml` → `alt-deploy` (proto / alt-backend / alt-butterfly-facade / alt-frontend-sv の 4 サービス再 deploy)
2. operator が `/admin/knowledge-home` UI を開き **Emit URL Backfill** ボタン押下
3. inline summary で `events_appended` を確認 (期待値 ~12,252)
4. UI 上で **Mode=Full** で Reproject 起動 → Swappable 待ち → **Swap**
5. SQL 確認: `SELECT projection_version, count(*) FILTER (WHERE link='') AS empty_link FROM knowledge_home_items WHERE item_type='article' AND projection_version=(SELECT max(version) FROM knowledge_projection_versions WHERE is_active) GROUP BY 1;` が `empty_link ≤ 14,946 / 55,210 ≈ 27%` まで低下
6. FE で `Archived` kicker が大幅減少を目視確認

詳細手順は `docs/runbooks/knowledge-home-reproject-operations.md` §Post-tag-fix backfill に pin (ADR-000867 で導入済の節を更新)。
