# ADR-000620: backfill に direct mode・embedding レプリカ並列化・HNSW DROP/REINDEX パターンを導入し re-index を高速化する

## Status

Accepted

## Date

2026-04-04

## Affected Services

- rag-orchestrator (Go) — backfill CLI に direct mode 追加、DirectIndexer 実装
- rag-db (PostgreSQL + pgvector) — HNSW インデックス DROP/REINDEX、shm_size・mem_limit 拡張

## Context

[[000619]] で chunker V9 (HTML sanitizer) を導入した際、rag-db の全ドキュメント（約 44,000 件）を V9 で再チャンク + 再 embedding する必要が生じた。

### 初期の backfill 性能

既存の backfill CLI は以下のアーキテクチャで動作していた:

```
backfill CLI → [HTTP POST] → rag-orchestrator → [HTTP POST] → Ollama embedder → GPU
                                     ↓
                              rag-db (INSERT + HNSW 更新)
```

1 記事あたり 2 回の HTTP ラウンドトリップ + JSON serialize/deserialize が発生し、スループットは **0.34 記事/秒**（リモート embedder）。44,000 件の処理に約 36 時間を要する計算であった。

### ボトルネック分析

実測により以下のボトルネックを特定した:

| 処理 | 所要時間 | 割合 |
|------|---------|------|
| DB idempotency チェック | 3ms | 0.4% |
| チャンク化 (CPU) | 5ms | 0.6% |
| embedding (GPU) | 200ms〜2,800ms | 70-95% |
| DB INSERT (HNSW 更新) | 100ms | 4-25% |

ボトルネックは段階的に移動した:

1. **初期**: リモート embedder のネットワークレイテンシ (2.8 秒/記事)
2. **レプリカ導入後**: HNSW インデックスへの INSERT (3GB のインデックスに対する同時書き込みが直列化)
3. **HNSW DROP 後**: embedding のスループット上限

## Decision

**3 段階の最適化を組み合わせ、backfill のスループットを 0.34 記事/秒から最大 32 記事/秒に向上させる。**

### 1. Direct mode: HTTP ラウンドトリップの排除

backfill CLI に `--direct` フラグを追加し、rag-orchestrator の HTTP エンドポイントを経由せずに直接 rag-db に書き込む。

```
backfill CLI --direct
  ├── alt-db (READ) ← 既存
  ├── rag-db (WRITE) ← 直接接続
  ├── Ollama embedder (HTTP) ← 直接呼び出し
  └── domain.Chunker (in-process) ← 直接呼び出し
```

- 既存の `usecase.IndexArticleUsecase` をそのまま再利用（HTTP ラッパーなしで直接呼び出し）
- pgvector の型登録 (`pgxvector.RegisterTypes`) を `infra.NewPostgresDB` 経由で適用（`pgxpool.New` では型登録されず `vector cannot have more than 16000 dimensions` エラーが発生する）

### 2. Embedding レプリカ並列化

embeddinggemma (307M パラメータ、~620MB VRAM) の Ollama コンテナを複数台スポットで起動し、`DirectIndexer` がラウンドロビンで分散。

```go
type DirectIndexer struct {
    indexers []usecase.IndexArticleUsecase  // 各レプリカに対応
    next     atomic.Uint64                  // ラウンドロビンカウンタ
}
```

- `EMBEDDER_URLS` 環境変数（カンマ区切り）で複数 URL を指定
- 各 `IndexArticleUsecase` は同じ rag-db pool を共有し、embedder だけ異なる
- `OLLAMA_KEEP_ALIVE=-1` でモデルのアンロードを防止

### 3. HNSW DROP → bulk INSERT → REINDEX パターン

pgvector の既知のパフォーマンス問題（HNSW インデックス存在時の INSERT が ~5 倍遅くなる）に対し、PostgreSQL の標準的なバルクロードパターンを適用。

1. `DROP INDEX rag_chunks_embedding_idx`
2. backfill 実行（HNSW なしで INSERT が高速）
3. `CREATE INDEX ... USING hnsw` で再構築

REINDEX 時の設定:
- `maintenance_work_mem = '2GB'`（グラフの一部がメモリに収まる）
- `max_parallel_maintenance_workers = 7`（8 並列ビルド）
- compose の `shm_size: "2g"` + `mem_limit: 4096m`（PostgreSQL の共有メモリ要件を満たす）

### 検討した代替案

| 案 | 評価 |
|---|---|
| バッチ embedding（複数記事のチャンクを 1 回の Encode() で処理） | IndexArticleUsecase のインターフェース変更が必要。usecase 層の影響範囲が大きい |
| rag-db の接続プール拡大のみ | HNSW INSERT のボトルネックは接続数ではなくインデックス更新コスト |
| pgvector の IVFFlat インデックスに切り替え | INSERT は速いが検索精度が低下。HNSW の方が recall が高い |

## Consequences

### Pros

- **36 時間 → 約 2 時間**: backfill 全体の所要時間を大幅短縮（HNSW REINDEX 含む）
- **既存コードの再利用**: `IndexArticleUsecase`、`domain.Chunker`、`OllamaEmbedder`、`repository.*` をそのまま使用。新規のビジネスロジックなし
- **スポットレプリカ**: backfill 完了後にレプリカコンテナを削除するだけ。永続的なインフラ変更なし
- **段階的適用**: `--direct` なしの既存 HTTP モードは一切変更なし。新モードはオプトイン
- **pgvector ベストプラクティスの適用**: DROP/REINDEX パターンは PostgreSQL 公式ドキュメントおよび pgvector コミュニティで推奨されるバルクロード手法

### Cons / Tradeoffs

- **HNSW DROP 中は Augur のベクトル検索が利用不可**: backfill + REINDEX の間（約 2 時間）、Augur の回答品質が低下する。計画的なメンテナンス時間帯での実行が必要
- **HNSW REINDEX に約 1 時間**: 1.1M チャンクの on-disk HNSW ビルドに約 1 時間。`maintenance_work_mem` がグラフ全体（推定 3.2GB）に足りない場合、on-disk モードに切り替わり大幅に遅くなる
- **rag-db の mem_limit 増加**: 512MB → 4096MB。REINDEX 完了後は元に戻すことが可能だが、今後の HNSW 再構築に備えて維持する方が実用的
- **孤児ドキュメント**: alt-db から削除済みだが rag-db に残る v8 ドキュメント（約 6,000 件）は backfill では処理されない。検索結果には影響しない（`current_version_id` ベースのクエリで除外される）が、ストレージを消費する
