# ADR-000766: search-indexer Hurl スイートに run.sh を追加し 3 サービス共通の E2E ディスパッチ規約を確立する

## Status

Accepted

## Date

2026-04-18

## Affected Services

- search-indexer (Go) — プロダクションコードは触らず、`e2e/hurl/search-indexer/run.sh` を 1 本だけ追加。既存 7 ファイル (00-seed + 01-06) はそのまま再利用

## Context

[[000763]] で Increment 1 として `search-indexer` を public Alt → private `alt-deploy` repo の `release-deploy.yaml` (build → publish → gate → e2e → deploy 5 job) に乗せた。続いて [[000764]] で **mq-hub** の Hurl スイートが、[[000765]] で **knowledge-sovereign** の Hurl スイートが追加され、いずれも `e2e/hurl/<svc>/run.sh` を伴って公開リポにコミットされた。

しかし `search-indexer` だけ:

- **`run.sh` が存在しない**。`e2e/hurl/search-indexer/README.md` には Hurl コマンドが narrative として書かれているだけで、実際の起動・seed・tear-down は alt-deploy 側 `release-deploy.yaml` の e2e job が **inline shell** で実装している
- そのため `release-deploy.yaml` の e2e job は **search-indexer 専用に hard-coded** されており、mq-hub / knowledge-sovereign を変更しても deploy 経路の E2E gate に乗らない（matrix 化できる接合点がない）
- ローカル再現には README の手順を 4 行コピペする必要がある。mq-hub / knowledge-sovereign は `bash e2e/hurl/<svc>/run.sh` 1 行で済むのに、search-indexer だけ手順が異なるとオンボーディングで踏まれる

ADR-763 のフォローアップ計画 ([[daily/2026-04-18]] §"Operational 引き継ぎ") には「次サービス migration 時のチェックリスト: ... `e2e/hurl/<svc>/`」とあり、`run.sh` を含む形が前提だったが、Increment 1 の時点では存在しなかった。

このギャップを埋めずに alt-deploy 側 e2e job を matrix 化すると「search-indexer は inline、mq-hub / knowledge-sovereign は run.sh」の二系統 dispatch が永続化する。3 サービス共通の dispatch 接合点を先に固める必要がある。

## Decision

**`e2e/hurl/search-indexer/run.sh` を 1 本追加し、`bash e2e/hurl/<svc>/run.sh` を 3 サービス共通の E2E ディスパッチ規約として確立する。** 採用構成:

1. **mq-hub の run.sh を雛形にする**。`set -euo pipefail` + `trap cleanup EXIT` + `IMAGE_TAG` / `GHCR_OWNER` / `HURL_IMAGE` / `RUN_ID` / `KEEP_STACK` の env 規約をそのまま踏襲し、3 サービスで同じ呼び出し面を持たせる。
2. **`--profile search-indexer` で staging stack を起動**する。`compose/compose.staging.yaml` の既存 profile (meilisearch + stub-backend + search-indexer) をそのまま使う。
3. **seed と本体スイートを分離する**。`00-seed-meilisearch.hurl` は `--jobs 1` 相当で先に走らせ（fixture 投入が完了してから index 検索を始めるため）、その後 `0[1-9]-*.hurl` を `--jobs 4 --retry 5 --retry-interval 500` で並列実行する。これは search-indexer の REST 検索が stateless であり並列が安全（mq-hub / knowledge-sovereign のような state machine 依存がない）という性質を反映した最適化。
4. **Meilisearch master key を fixture file から読む**。`tr -d '\n' < e2e/fixtures/staging-secrets/meili_master_key.txt` で、compose の `secrets:` ブロックがマウントするのと同じファイルを単一情報源にする。`--variable meili_master_key=...` で Hurl に注入。
5. **Hurl は alt-staging network 内で実行**する。`docker run --network alt-staging -v "$ROOT:$ROOT"` で repo を同パスに mount し、`--file-root "$ROOT"` で `file,e2e/fixtures/...;` を解決する ([[000763]] item 「Hurl の `file,PATH;` が `..` を含むパスを unconditionally reject」と同じ対処)。
6. **alt-deploy 側 release-deploy.yaml の matrix 化は別 PR (private repo) でフォロー**する。今回の ADR は public 側に dispatch 接合点を揃えるところまで。matrix 化時の security 観点 (KEEP_STACK の self-hosted 残骸、permissions、matrix.service の regex validation 等) は本日付の plan ファイル `~/.claude/plans/ima-reflective-piglet.md` の Security Audit Report (F-001..F-008) に集約済みで、alt-deploy PR レビュー時に持ち込む。

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| README に書いてある Hurl 手順をそのまま alt-deploy 側 e2e job の inline step に複製したまま放置 | 3 サービスで dispatch 経路が割れたままになり、knowledge-sovereign がもし追加されたら 4 経路まで増える。release-deploy.yaml の matrix 化が永続的に阻害される |
| 共通の `e2e/hurl/run.sh <svc>` を作って 3 つのスクリプトを統合 | 各サービスの seed 必要有無 / 並列可否 / FK ordering / fixture UUID 生成等が異なり、共通化すると条件分岐だらけになる。ADR-764 / 765 が**意図的に各サービス固有の run.sh** を選んだ判断を覆すことになる。3 本出揃ってから共通関数化する判断は将来別 ADR に委ねる |
| `e2e/hurl/search-indexer/run.sh` で seed と本体を `--jobs 1` 直列実行 | search-indexer の REST 検索は stateless なので直列化は不要。並列のほうが CI 時間が短く、knowledge-sovereign 流の `--jobs 1` 強制を持ち込むと search-indexer 固有の状況を読み違えた設計になる |
| Meilisearch master key を run.sh に直接 hard-code | `compose/compose.staging.yaml` の `secrets:` ブロックがマウントする `e2e/fixtures/staging-secrets/meili_master_key.txt` と二重管理になる。fixture file rotate 時に片方が古いまま残るリスク。`tr -d '\n' < <file>` で常に同じソースから読む方が再現性が高い |
| ADR-765 の knowledge-sovereign と同様の `--jobs 1` 強制 | search-indexer は state machine ではなく、並列で問題が出ない。`--jobs 1` は不要な制約で CI 時間を伸ばす |

### 範囲外 (今回やらないこと)

- alt-deploy private repo `release-deploy.yaml` の `resolve` job 拡張 (mq-hub / knowledge-sovereign を path filter に追加) と `e2e` job の matrix 化 — 別 PR (private repo) で実施
- 公開側 `.github/workflows/e2e-hurl.yml` への search-indexer / knowledge-sovereign sibling job 追加 — alt-deploy gate を優先したため今回は対象外
- 3 サービスの run.sh から共通シェル関数 (`hurl_in_network` 等) を抽出する refactor — パターンが安定するまで保留
- `compose.staging.yaml` の split (ADR-765 で言及した将来課題)

## Consequences

### Pros

- **`bash e2e/hurl/<svc>/run.sh` が 3 サービス共通の dispatch 接合点になった**。alt-deploy 側 `release-deploy.yaml` の e2e job を `matrix.service` 1 行で matrix 化できる準備が完成
- **オンボーディング摩擦が消えた**。新規メンバーが search-indexer の E2E をローカル再現する時、README の手順を読まずとも他 2 サービスと同じ 1 行コマンドで動く
- **fixture file が単一情報源になった**。Meilisearch master key を `tr -d '\n' < e2e/fixtures/staging-secrets/meili_master_key.txt` で読むため、compose の `secrets:` ブロックと run.sh の参照が同期する
- **並列度が明示的に文書化された**。search-indexer は `--jobs 4` (stateless REST)、knowledge-sovereign は `--jobs 1` (FK + event_seq ordering)、mq-hub は seed/test 分離で seed のみ直列。各サービスの E2E 性質がスクリプトの形で固定された
- **alt-deploy 側 PR の review 差分が小さくなる**。`run: bash e2e/hurl/${{ matrix.service }}/run.sh` 1 行で 3 サービス分の dispatch が完成するため、matrix 化 PR は数十行に収まる見込み

### Cons / Tradeoffs

- **alt-deploy 側 matrix 化を別 PR に分離した**ことで、Increment 1 完了直後の現状では public は `run.sh` 揃えただけ・private はまだ search-indexer hard-coded のまま、という中間状態が一時的に残る。本 ADR commit から alt-deploy PR merge までの間 mq-hub / knowledge-sovereign の deploy gate E2E は走らない (公開側 `e2e-hurl.yml` の mq-hub PR-time gate は引き続き走る)
- **3 つの run.sh が別々のスクリプトとして残った**。共通シェル関数 (`hurl_in_network` 等) の抽出は意図的に保留しているが、4 サービス目以降の追加で同じ boilerplate を 4 度書くことになる。共通化判断は早めに別 ADR で扱うべき
- **search-indexer の README に `run.sh` への言及がまだない**。README は narrative 主体のままで、`run.sh` を使うべきタイミングを明示していない。次の README 更新で追記する運用フォロー
- **alt-deploy private repo のレビュー時に持ち込むべき security 項目 (F-001..F-005) を ADR 本文には書かなかった**。代わりに plan ファイルに集約したため、private repo PR 起こす担当者が plan ファイルを参照する規約が暗黙化している。docs/runbooks 化は将来の課題
- **`KEEP_STACK=1` を public 側 e2e-hurl.yml が hardcode している**前提で run.sh を書いた。public は ubuntu-latest (ephemeral) なので問題ないが、alt-deploy の self-hosted で同 env を引き継ぐと container 残骸が累積する。matrix 化 PR で必ず drop する必要がある (plan F-001)
