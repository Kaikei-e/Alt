# ADR-000744: recap-worker (Rust) の provider 検証基盤を軽量 replay verifier で導入する

## Status

Accepted

## Date

2026-04-15

## Affected Services

- recap-worker (Rust) — `tests/provider_verification.rs` 新設、`scripts/pact-check.sh` に Rust provider step 追加

## Context

[[000591]] で 4 consumer (search-indexer / recap-evaluator / rag-orchestrator / alt-backend) が recap-worker を provider として pact を作成済だが、Rust 側で verify する仕組みが無かった。Go / Python の pact-go / pact-python には公式 verifier があるが、Rust の `pact_verifier` crate は以下の問題がある:

- **依存重量**: `pact_verifier` を dev-dependency に入れると transitive で 50+ crate が入り、CI の `cargo test` 時間が 60s+ 増える (recap-worker は既に `tch` / `rust-bert` 等の重量 ML crate を持つため追加コストが体感に出る)
- **provider state API**: consumer 側 `pact_consumer` v1.3 と provider 側 `pact_verifier` のバージョン整合が微妙で、interaction の provider state が ignore される既知の issue がある
- **axum 統合**: `pact_verifier::ProviderInfo` は HTTP handler への直結を想定しないため、axum の `Router` をテストサーバに bind するラッパを書く必要があり、結局自前 stub と大差ない

pact-go のような heavyweight verifier は Go / Python では妥当だが、Rust では cost / benefit が合わない。

## Decision

**`pact_verifier` crate は採用せず、軽量な replay verifier を `tests/provider_verification.rs` に自作する。**

### 実装方針

`stub_router()` で axum `Router` を構築し、pact ファイルで宣言された各 endpoint にカニカルな response を返すハンドラを登録:

- `/v1/recaps/genres/indexable` (search-indexer consumer)
- `/v1/evaluation/genres/{run_id}` (recap-evaluator consumer)
- `/v1/morning/letters/latest` / `/v1/morning/letters/{date}` (rag-orchestrator consumer)

テスト本体は各 pact JSON を `serde_json::Value` で読み、interactions 配列を回して:

1. `request.method` / `request.path` から reqwest で stub 叩く
2. `response.status` と実際のステータスが一致することを assert
3. body が JSON として parse 可能であることを assert

`#[ignore = "provider verification: run with --ignored"]` を付け、通常 `cargo test` では走らない。`cargo test --test provider_verification -- --ignored` で明示起動。`scripts/pact-check.sh` の Rust provider step も同コマンドを呼ぶ。

### 本格 pact_verifier との機能差

本 ADR の軽量 verifier が **カバーしない** 要素:

- Pact matcher (type-matching, regex, etc) — 本 verifier は literal compare しかしない
- Provider state handler (`providerStates` の副作用) — 本 verifier は state を無視し、常に同じ response
- Message pact (async Redis Streams 等) — sync HTTP のみ対応
- Generator / 動的値 (tokens, dates) — 対応せず

これらが必要になった時点で `pact_verifier` crate への段階移行を検討する。現状 recap-worker の 4 pacts は全て static response で足りているため cost/benefit 逆転は無い。

### 検討した代替案

- **案 A: `pact_verifier` crate を dev-dependency に正面採用** — 60s+ の CI 時間増。matcher / state が不要な状況では overkill
- **案 B: Go で recap-worker の stub を書き、pact-go で verify** — クロス言語での重複実装。recap-worker の内部 API 変更時に Go stub を追従する必要が生じる。reject
- **案 C: recap-worker を axum 直起動してから HTTP クライアントで叩く integration test** — DB 依存が発生し CI 時間がさらに増加。stub router が最小単位

## Consequences

### Pros

- CI 時間増加は 1s 未満 (既存 318 tests + 3 ignored tests)
- 4 orphan pact (recap-worker = provider) が全て Rust 側で replay 検証される
- `scripts/pact-check.sh` の total check 数が 15 → 16 に増え、ギャップ閉鎖を数字で追える

### Cons / Tradeoffs

- **matcher / state handler を使う pact が新規追加されると verifier が不十分**。将来 `pact_verifier` crate への upgrade が必要
- **stub router が pact の expected response をハードコード**。response 形が変わると stub を手で追従する必要。厳密な生成側検証 (実 handler が同じ response を返すか) にはならない
- **message pact (Redis Streams) は cover せず**。recap-worker は現状 consumer 側でしか message pact を触らないので影響なし
