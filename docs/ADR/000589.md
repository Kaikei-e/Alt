# ADR-000589: Pact CDC テスト基盤を導入し最重要 3 ペアの Consumer/Provider テストを実装する

## Status

Accepted

## Date

2026-03-26

## Affected Services

- alt-backend (Go) — Consumer: 3 テスト
- pre-processor (Go) — Consumer: 3 テスト
- rag-orchestrator (Go) — Consumer: 2 テスト
- news-creator (Python) — Provider: 2 検証テスト

## Context

[[000588]] で採用した Pact CDC 戦略の初期実装として、PM で発覚した契約破壊パターンに直結する最重要 3 ペアから着手した。

### 対象ペアの選定理由

| ペア | 根拠 |
|------|------|
| alt-backend → pre-processor | PM-004 #1: REST/Connect-RPC 混在で 409 拒否 |
| pre-processor → news-creator | PM-008: options 不一致でモデルリロード |
| rag-orchestrator → news-creator | PM-006: セマフォバイパスで GPU 飢餓 |

## Decision

### インフラ: Pact Broker

`compose/pact.yaml` を新設。`pact` profile で起動する構成とし、本番サービスに影響しない。

```yaml
services:
  pact-db:
    image: postgres:16-alpine
    profiles: [pact]
  pact-broker:
    image: pactfoundation/pact-broker:latest
    profiles: [pact]
    ports: ["9292:9292"]
```

### Consumer テスト実装

**alt-backend → pre-processor（Connect-RPC）:**

| テスト | 検証内容 |
|--------|---------|
| `TestPreProcessorSummarizeContract` | Summarize unary RPC の request/response 形式 |
| `TestPreProcessorQueueSummarizeContract` | QueueSummarize の job ID 返却契約 |
| `TestPreProcessorGetSummarizeStatusContract` | GetSummarizeStatus のステータス値契約 |

Connect-RPC は ProtoJSON モード（`connect.WithProtoJSON()`）で通信。Pact V3 の `WithCompleteRequest/Response` で JSON レベルの契約を定義。

**pre-processor → news-creator（HTTP/REST）:**

| テスト | 検証内容 |
|--------|---------|
| `TestNewsCreatorSummarizeContract` | `/api/v1/summarize` の正常レスポンス形式 |
| `TestNewsCreatorSummarizeQueueFullContract` | 429 + `Retry-After` ヘッダー |
| `TestNewsCreatorSummarizeWithPriorityContract` | `priority: "high"` 付きリクエスト |

Content 長の制約（100 文字以上）を契約のリクエストボディに反映。`priority` は小文字（`"high"` / `"low"`）であることを Pact で固定。

**rag-orchestrator → news-creator（/api/chat）:**

| テスト | 検証内容 |
|--------|---------|
| `TestChatStreamContract` | streaming chat の request 形式 + `application/x-ndjson` レスポンス |
| `TestChatNonStreamContract` | non-streaming chat の request/response 形式 |

### Provider 検証テスト

news-creator に `tests/contract/test_provider_verification.py` を追加。FastAPI アプリをモック依存で起動し、Consumer が生成した pact ファイルを検証。

Provider state に応じて mock 挙動を切り替え:
- `"the LLM queue is full"` → `QueueFullError` を raise → 429 返却
- その他 → 正常レスポンス

| サービス | パッケージ | バージョン |
|----------|-----------|-----------|

## Consequences

### Pros

- 最重要 3 ペアの契約が機械的に検証可能になった
- `priority` フィールドの大文字/小文字不一致を CDC テスト中に発見・修正（実際の CDC の価値を実証）
- Content 長の最小制約（100 文字）が契約に明示化された
- 全テストが Java 依存ゼロで動作（pact-go v2 は Rust FFI）

### Cons / Tradeoffs

- Streaming RPC の Pact 検証は Content-Type レベルに限定（NDJSON のチャンク単位検証は非対応）
- Provider テストは uvicorn サーバーを起動するため、ポート競合のリスクがある（`_get_free_port()` で緩和）
- pact ファイルの共有は現在ファイルシステム経由。Pact Broker への publish は次のフェーズ
