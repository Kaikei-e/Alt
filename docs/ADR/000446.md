# ADR-000446: Supersede UX の end-to-end 完成（write path → projection → response → UI）

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go)
- alt-frontend-sv (SvelteKit)

## Context

Knowledge Home では要約・タグ・判断根拠が更新されることがあるが、「何が最新か」「何が差し替えられたか」「なぜ更新扱いなのか」を Home 上で自然に扱う手段が存在しなかった。

Phase 4 で schema / proto / domain の受け皿（`supersede_state`, `superseded_at`, `previous_ref_json` カラム、`SupersedeInfo` proto メッセージ）は先行して用意されていたが、write path → event → projector → read model → response → UI の経路が貫通しておらず、supersede は死蔵状態だった。

### 解決すべき課題

1. **write path**: summary/tag が再生成されても supersede イベントが emit されない
2. **projection**: supersede イベントを read model に投影するハンドラがない
3. **read model**: supersede カラムの SELECT / UPDATE が未実装
4. **response**: `PreviousRefJSON` から `SupersedeInfo` proto への変換が不完全
5. **UI**: badge の表示が接続されていない
6. **reason**: `reason_updated` という supersede source が定義されていない
7. **open-after-clear**: 開封後の supersede 表示クリアルールが未実装

## Decision

### Supersede canonical contract の拡張

**`supersede_state` に `reason_updated` と `multiple_updated` を追加**し、3 source（summary / tag / reason）+ 複合 state を canonical に定義する。

| 値 | 意味 | badge 文言 | badge 色 |
|----|------|-----------|---------|
| `summary_updated` | 要約が再生成された | Summary updated | sky-400 |
| `tags_updated` | タグセットが再生成された | Tags updated | indigo-400 |
| `reason_updated` | WHY reason が変更された | Reasons changed | amber-400 |
| `multiple_updated` | 2 source 以上が更新された | Updated | sky-400 |

Proto の `SupersedeInfo` に `previous_why_codes` (field 5) を additive に追加。後方互換を維持。

Domain に `EventReasonMerged` イベント型を追加。

### Write path: Transaction 統合パターン

**`CreateSummaryVersionUsecase` と `CreateTagSetVersionUsecase` に `MarkSupersededPort` を注入**し、version 作成後に旧版の mark + supersede event emit を一連で実行する。

```
1. CreateVersion()
2. MarkSuperseded() → 旧版を返す（なければ nil）
3. 旧版あり → SummarySuperseded / TagSetSuperseded event emit
```

初回バージョン（旧版なし）では supersede event は emit しない。

supersede event payload には最小説明情報を含める:
- `SummarySuperseded`: `previous_summary_excerpt`（旧要約の先頭 200 文字）
- `TagSetSuperseded`: `previous_tags`（旧タグ名リスト）

### Projector: supersede を read model に投影

`SummarySuperseded`, `TagSetSuperseded`, `ReasonMerged` の 3 イベントに対応する projection handler を追加。各ハンドラは `supersede_state`, `superseded_at`, `previous_ref_json` を home item に設定する。

`previous_ref_json` は JSONB で source ごとの情報を蓄積する設計。DB の `||` merge で複数 source の情報が欠落なく保持される。

### Read model: supersede カラムの完全対応

- `GetKnowledgeHomeItems`: `supersede_state`, `superseded_at`, `previous_ref_json` を SELECT に追加
- `UpsertKnowledgeHomeItem`: supersede カラムの INSERT / UPDATE に `COALESCE` merge ロジックを追加
- `ClearSupersedeState`: 専用メソッドで NULL 化（generic upsert の merge-safe ロジックを壊さない）

### Handler/Response: SupersedeInfo 変換の完成

`convertHomeItemToProto` が `PreviousRefJSON` をパースし、`previous_summary_excerpt`, `previous_tags`, `previous_why_codes` を `SupersedeInfo` proto に設定する。

### Open-after-clear: HomeItemOpened = supersede acknowledgement

`projectHomeItemOpened` 内で `ClearSupersedeStatePort` を呼び出し、supersede 表示をクリアする。

- `event_seq` 順で `SummarySuperseded`(seq=N) → `HomeItemOpened`(seq=N+M) → clear が再現される
- Open 後に新たな supersede が来た場合は、clear 済みなので新規設定として投影される
- `ClearSupersedeStatePort` は variadic パラメータとして `KnowledgeProjectorJob` に渡す（既存呼び出しの後方互換を維持）

### Frontend: badge + 短い説明

- `SupersedeBadge.svelte`: `resolveSupersede()` display map を使用し、state ごとにアイコン・色・ラベルを切り替え
- `KnowledgeCard.svelte`: タイトル直下に SupersedeBadge を条件表示
- card 上は badge と短い説明に留め、過密にしない

### Stream mapping

`EventReasonMerged` を `item_updated` canonical stream type にマッピング。supersede イベントが stream 経由で Home 更新契機になる。

## Consequences

### Pros

- **更新経緯の説明可能性**: 何が最新か、何が差し替えられたか、なぜ更新扱いなのかを Home 面で自然に示せる
- **reproject-safe**: supersede 表示に必要な情報は event payload に含まれており、再投影で再現可能
- **old/new 不整合の防止**: version 作成と supersede event emit が一連で実行される
- **段階投入可能**: `enable_supersede_ux` フラグで UI 表示のみを切り替えられる。backend pipeline は常に動作
- **後方互換**: proto 変更は additive（field 5 追加）、projector の variadic パラメータ、既存 callers への影響なし

### Cons / Tradeoffs

- **reason merge の source event は部分実装**: recap/pulse が reason を付与する経路での `ReasonMerged` event emit は、本 ADR のスコープでは projector 側の受信準備のみ。emit 側は各 worker の改修が別途必要
- **Transaction 統合は厳密には同一 DB tx ではない**: usecase 内で create → mark → event append を順次実行しているが、途中で失敗した場合の部分的な状態が発生しうる。ただし event append は idempotent（dedupe_key による ON CONFLICT DO NOTHING）であり、リトライで整合性を回復できる
- **複合 supersede state の自動解決は DB 側**: `COALESCE` merge で `summary_updated` + `tags_updated` が重なると先勝ちになる。`multiple_updated` への自動畳み込みは projector 側で明示的に行う必要がある（将来課題）
- **ClearSupersedeState は projector 経由でのみ実行**: handler から直接クリアする導線はない。projector のポーリング間隔（30 秒）分のラグがある
