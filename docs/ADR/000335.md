# エラー伝搬バグ修正: pre-processor エラーの UNKNOWN_ERROR 変換と運用ログノイズ削減

## ADR's STATUS

Accepted

## CONTEXT

ユーザーがフロントエンドで記事要約を実行した際、`"An unexpected error occurred. Please try again later. (Error ID: fa94d142)"` という汎用エラーが表示された。コンテナログを徹底調査した結果、5つの問題を発見した。

### 問題の全体像

| # | 問題 | 優先度 | サービス |
|---|------|--------|----------|
| 1 | VALIDATION_ERROR が UNKNOWN_ERROR に変換される | P1 | alt-backend |
| 2 | CONFLICT_ERROR が UNKNOWN_ERROR に変換される | P1 | alt-backend |
| 3 | dashboard コンテナのクラッシュループ | P2 | dashboard |
| 4 | context canceled が ERROR レベルでログされる | P3 | alt-backend |
| 5 | GetFeedIDByURL の "no rows" が ERROR レベルでログされる | P4 | alt-backend |

### 問題 1+2 の詳細: エラー伝搬の欠陥

要約リクエストの完全なフローは以下の通り:

```
Frontend (FeedDetailModal.svelte)
  → alt-backend StreamSummarize (Connect-RPC Server Streaming)
    → HTTP POST pre-processor /api/v1/summarize/stream
      → pre-processor: "content too short" → VALIDATION_ERROR (HTTP 400)
    ← alt-backend: fmt.Errorf("pre-processor returned status %d: %s", ...)
      → errorhandler.HandleInternalError → classifyDriverError
        → fmt.Errorf は *AppContextError でも *AppError でもないため分類不能
          → NewUnknownContextError("internal server error")
            → "An unexpected error occurred. Please try again later."
```

pre-processor は適切な JSON エラーレスポンス（`VALIDATION_ERROR`, `CONFLICT_ERROR`）を返していたが、alt-backend の `streamPreProcessorSummarize` がそれを生の `fmt.Errorf` に変換していたため、`classifyDriverError` で分類できず `UNKNOWN_ERROR` に化けていた。

### 問題 3: dashboard の `.venv` 汚染

dashboard の Dockerfile は builder ステージで `uv sync --frozen --no-dev` を実行して `.venv` を構築し、runtime ステージにコピーしている。しかし `.dockerignore` が存在しなかったため、`COPY . .`（line 45）がホストマシンの `.venv`（ホスト固有の Python シンボリックリンクを含む）で builder ステージの `.venv` を上書きしていた。

### 問題 4+5: ログノイズ

- `StreamFeedStats`: クライアント切断時の `context canceled` が ERROR レベルでログされていた。SSE ストリーミングではクライアント切断は正常動作であり、ERROR は不適切。
- `GetFeedIDByURL`: 未登録フィードの検索で `pgx.ErrNoRows` が ERROR レベルでログされていた。「見つからない」は正常なビジネスケースであり、ERROR は不適切。

## DECISION MAKING

### 問題 1+2: pre-processor エラーの適切なマッピング

**方針**: `streamPreProcessorSummarize` で pre-processor の JSON エラーレスポンスをパースし、`VALIDATION_ERROR` / `CONFLICT_ERROR` を適切な Connect-RPC エラーコードにマッピングする。

**ファイル**: `alt-backend/app/connect/v2/feeds/handler.go`

1. `mapPreProcessorHTTPError` ヘルパー関数を追加。pre-processor の JSON レスポンスをパースし、エラーコードに応じた `*connect.Error` を返す:

```go
func mapPreProcessorHTTPError(statusCode int, body []byte) *connect.Error {
    var errResp preProcessorErrorResponse
    if err := json.Unmarshal(body, &errResp); err != nil {
        return nil // JSON パース失敗時はフォールスルー
    }
    switch errResp.Error.Code {
    case "VALIDATION_ERROR":
        return connect.NewError(connect.CodeInvalidArgument, ...)
    case "CONFLICT_ERROR":
        return connect.NewError(connect.CodeAlreadyExists, ...)
    default:
        return nil
    }
}
```

2. `StreamSummarize` の呼び出し元で `errors.As(err, &connectErr)` を使い、`*connect.Error` が返された場合は `HandleInternalError` をバイパスして直接返す。これにより `classifyDriverError` による UNKNOWN_ERROR 変換を回避する。

**マッピング結果**:

| pre-processor エラー | Connect-RPC コード | フロントエンドの挙動 |
|---------------------|-------------------|-------------------|
| `VALIDATION_ERROR` | `CodeInvalidArgument` | エラーメッセージをそのまま表示 |
| `CONFLICT_ERROR` | `CodeAlreadyExists` | `isConflictError()` でリトライロジックと整合 |
| 未知のエラー | フォールスルー → `CodeInternal` | 汎用エラーメッセージ |

### 問題 3: `.dockerignore` の追加

**ファイル**: `dashboard/.dockerignore`（新規作成）

`.venv`, `__pycache__`, `*.pyc`, `.git` を除外。これにより builder ステージで構築した `.venv` がホストの `.venv` で上書きされなくなる。

### 問題 4: context canceled のログレベル変更

**ファイル**: `alt-backend/app/connect/v2/feeds/handler.go`

`StreamFeedStats` のループ内と `sendStatsUpdate` 内の各 usecase 呼び出しで `ctx.Err() != nil` をチェックし、コンテキストキャンセル時は INFO レベルに変更。

### 問題 5: GetFeedIDByURL の sentinel error 導入

**ファイル**: `alt-backend/app/driver/alt_db/feed_url_to_id_driver.go`

`pgx.ErrNoRows` を検出して `ErrFeedNotFoundByURL` sentinel error を返すように変更。ログレベルを INFO に下げ、`pgx.ErrNoRows` を上位レイヤーに露出させない。

**ファイル**: `alt-backend/app/gateway/feed_url_to_id_gateway/feed_url_to_id_gateway.go`

`ErrFeedNotFoundByURL` をチェックし、INFO レベルでログした上でエラーを透過的に伝搬。

## RESULTS, EFFECTS

### PROS

- **ユーザー体験の改善**: 「content too short for summarization」のような具体的なエラーメッセージがフロントエンドに表示されるようになり、ユーザーが問題を理解できる
- **フロントエンド変更不要**: `summaryError = error.message` でエラーメッセージをそのまま表示する既存実装と整合。`streaming.ts` の `isConflictError()` も `Code.AlreadyExists` をチェック済みのためリトライロジックも正しく動作する
- **ログノイズの大幅削減**: クライアント切断と未登録フィード検索が INFO レベルになり、ERROR ログが実際の問題のみを示すようになる
- **dashboard の安定稼働**: `.dockerignore` の追加でビルドの再現性が保証される
- **後方互換性**: 未知のエラーコードは従来通り `HandleInternalError` にフォールスルーするため、既存の動作を壊さない
- **レイヤー境界の改善**: driver 層が `pgx.ErrNoRows` を sentinel error に変換し、DB 実装の詳細を上位レイヤーに露出させない

### CONS, TRADEOFF

- **pre-processor の JSON レスポンス形式への依存**: `mapPreProcessorHTTPError` は `{"error":{"code":"...","message":"..."}}` 形式を前提としている。pre-processor のレスポンス形式が変わった場合は JSON パース失敗 → `nil` 返却 → フォールスルーとなるため安全だが、マッピングが効かなくなる
- **エラーメッセージの直接公開**: pre-processor のエラーメッセージがクライアントにそのまま表示される。内部的な技術用語が含まれる可能性があるが、現状の pre-processor エラーメッセージはユーザー向けに適切
- **context canceled チェックの分散**: `ctx.Err()` チェックが複数箇所に点在している。将来的にはミドルウェアレベルでの一括処理を検討する余地がある
