# ADR-000887: Morning Letter LLM 経路の GBNF 互換性を schema fix と Ollama 0.22.1 pin で復旧する

## Status

Accepted

## Date

2026-05-04

## Affected Services

- **news-creator (Python)** — `news-creator/app/news_creator/domain/models.py:463` の `MorningLetterSection.key` を `pattern=r"^(lead|top3|what_changed|by_genre:[a-z0-9_\-]+)$"` から `pattern=r"^(lead|top3|what_changed|by_genre:[a-z0-9_-]+)$"` に変更。`news-creator/app/tests/domain/test_grammar_safe_patterns.py` を新設し、`format=json_schema` で Ollama に送られる全 Pydantic model (`MorningLetterContent` / `IntermediateSummary` / `RecapSummary`) の `Field(pattern=...)` を walk して GBNF parser が拒否する escape (`\d`, `\w`, `\s`, それぞれの大文字、`\-`) を CI で sweep する。
- **news-creator-backend (Ollama runtime)** — `news-creator/Dockerfile.backend:4` の `FROM ollama/ollama:latest` を `FROM ollama/ollama:0.22.1` に固定。`news_creator_models` named volume を介して既存の q4km / 8k / 12k 派生 model image はそのまま利用、再 import なし。
- **knowledge-augur** — `knowledge-augur/Dockerfile:2` を同じく `0.22.1` に固定。RAG 制御平面側で news-creator-backend と挙動 drift が出ないよう揃える。
- **knowledge-embedder** — `knowledge-embedder/Dockerfile:2` を同じく `0.22.1` に固定。embedding 用に Ollama を使う 3 番目の image。

## Context

[[000886]] で Morning Letter daemon を `MORNING_DAEMON_ENABLED=true` で恒久 ON 化した結果、daemon は 30 分間隔で正常に動き始め、`recap-db.morning_letters` には当日 `target_date` の行が入るようになった。しかし新しい行は連続して `is_degraded=true`, `model=extractive-fallback` で書かれ、UI 側の Morning Letter は scaffold (through-line + bullets) だけが出る薄い表示のままだった。LLM 由来の lead / narrative が完全に欠落している。

news-creator の `morning_letter_usecase._generate_via_llm` は `format=MorningLetterContent.model_json_schema()` 付きで `/api/generate` を叩く。production の news-creator-backend (Ollama 0.21.1) は 4 attempts 連続で HTTP 500 + `{"error":"failed to load model vocabulary required for format"}` を返し、`morning_letter_usecase` の `parse_runtime` 例外ハンドラで [[000706]] の extractive fallback path に正常退避していた。

最初の所見では「`Modelfile.gemma4-e4b-12k` が `FROM gemma4-e4b-q4km` で派生していて、Ollama が `format=` 用の vocabulary metadata を派生 image から抽出できないのではないか」と推測した。しかし web research と news-creator-backend の `ollama.log` 直接観測でこの仮説は破棄した。

実際の根本原因は news-creator-backend の `/home/ollama-user/.ollama/logs/ollama.log` に明示されていた:

```
parse: error parsing grammar: unknown escape at \-]+)) "\"" space
llama_grammar_init_impl: failed to parse grammar
grammar_init: failed to initialize grammar
time=… level=INFO source=server.go:1634 msg="llm predict error: failed to load model vocabulary required for format"
```

クライアントに返る `failed to load model vocabulary required for format` は Ollama runner の `runner/ollamarunner/runner.go:888-894` で `NewGrammarSampler(...)` がエラーを返したときの **generic wrapper 文言**であり、vocabulary load の問題ではない。直前の `parse: error parsing grammar: unknown escape at \-` が真のエラーで、Ollama の JSON Schema → GBNF 翻訳器が character class 内の `\-` を unknown escape として弾いていた。

[Ollama Issue #12422](https://github.com/ollama/ollama/issues/12422) (2025-10-06 closed) で同根の症例が報告されており、メンテナ (rick-github) が「GBNF parser は `\d` 等を解釈できない、`[0-9]` に置き換えること」を closing comment として残している。本件はこの報告と完全に同型で、`\-` は character class メタキャラクタなのでエスケープ不要であり、Python regex 側は accept するが GBNF 側は reject する。

該当 schema は Pydantic Field:

```python
# news-creator/app/news_creator/domain/models.py:463
class MorningLetterSection(BaseModel):
    key: str = Field(
        min_length=1, pattern=r"^(lead|top3|what_changed|by_genre:[a-z0-9_\-]+)$"
    )
```

`gemma4-e4b-12k` model 固有の問題ではなく、本 schema を `format=` で渡す全モデルで再現する。production で 12k だけが顕在化したのは、news-creator container が `MODEL_ROUTING_ENABLED=false` + `LLM_MODEL=gemma4-e4b-12k` で立っており、Morning Letter LLM call が常に 12k 直撃だったため。base の q4km に切り替えても同じ schema を渡す限り直らない。

並行して、Ollama backend を抱える 3 つの Dockerfile (`news-creator/Dockerfile.backend`, `knowledge-augur/Dockerfile`, `knowledge-embedder/Dockerfile`) がいずれも `FROM ollama/ollama:latest` で **floating tag のまま運用**されていた。production が偶然 0.21.1 に張り付いていたのは「最後の build 時点で `:latest` が指していた version」というだけの偶発で、次の build で予告なく drift する。本件 grammar parser バグ自体は 2025-03 以降 main にずっと残っており upgrade では直らないが、「production の Ollama は何バージョンか」を確定できない状態は本件の triage を不必要に長引かせた。

直近の Ollama release を確認したところ、0.21.1 (2026-04-22) → 0.22.1 (2026-04-28) で Gemma 4 renderer の thinking / tool calling 改善が入っており、`/api/generate` の `format` 契約・named volume の model store フォーマットいずれも互換 (release notes は additive)。0.23.0 は 2026-05-03 release で Claude Desktop integration / Metal hardening を含むが熟成不足のため今回は見送る。

## Decision

3 つの独立 fix を 1 PR / 3 commit で投入する。

### 1. `MorningLetterSection.key` の `\-` escape を撤去する

`news-creator/app/news_creator/domain/models.py:463` の pattern を:

```python
pattern=r"^(lead|top3|what_changed|by_genre:[a-z0-9_-]+)$"
```

に変更する。`-` を character class 末尾 (`9_-]`) に置けば escape 不要、Python regex の意味は変わらず、Ollama の GBNF 翻訳器も受理する。

#### 検討した代替案

- **Modelfile.gemma4-e4b-12k を GGUF 直 import に書き換える**: 当初仮説に基づく案。実 log で grammar parse error を観測した段階で破棄。schema 側を直さない限り、12k を GGUF 直 import に書き直しても、または 12k を q4km に切り替えても同じ 500 を返す。
- **`format=` を外して JSON parsing を best-effort にする**: 厳密な構造を Ollama に強制できなくなり、Morning Letter section key validation が LLM 出力に依存する。`MorningLetterContent` の JSON schema 設計意図 (構造化された editorial output) を壊す。本件は schema 側 1 文字の修正で済むので採らない。
- **別 schema 経由で逃げる**: `MorningLetterSection.key` を free-form string にして post-hoc validate する案。validation を Pydantic から別レイヤーに散らすのは [[000706]] / [[000832]] が定めた structured-output 契約と矛盾する。

### 2. `ollama/ollama:latest` を `0.22.1` に pin し 3 つの Dockerfile を揃える

3 ファイル (`news-creator/Dockerfile.backend`, `knowledge-augur/Dockerfile`, `knowledge-embedder/Dockerfile`) を `FROM ollama/ollama:0.22.1` に固定する。volume layout は不変なので、deploy 時 entrypoint の `ollama list` で既存 model が検出されて `ollama create` は skip される。

#### 検討した代替案

- **0.23.0 に上げる**: web research で「`/api/generate` の `format` 契約と model store の format は 0.21.x → 0.23.x で破壊変更なし」を確認済みだが、0.23.0 は本 ADR 執筆 1 日前の release で Claude Desktop integration の新規 surface area を持つ。0.22.1 まで段階的に上げ、十分熟成してから 0.23.x 系を別 ADR で評価する。
- **`:latest` のまま運用**: production と staging で偶発的に異なる version を引く可能性が残る。grammar parser バグ自体は upgrade で直らないが、「何 version で動いているか」を確定できない状態は再発時の triage を遅らせる。今回 schema fix を入れるタイミングで併せて pin しておく。
- **新規 GHCR image として再 publish する**: 本件は upstream image を直接使えば済むので、自前 GHCR build を増やす理由がない。Dockerfile 1 行の置換で十分。

### 3. Grammar-safe regex pattern を CI gate にする

`news-creator/app/tests/domain/test_grammar_safe_patterns.py` を新設し、`SCHEMAS_FORWARDED_TO_OLLAMA` リストに登録した Pydantic model (`MorningLetterContent` / `IntermediateSummary` / `RecapSummary`) の `model_json_schema()` を walk して、`Field(pattern=...)` 由来の文字列に GBNF parser が拒否する escape (`\d`, `\D`, `\w`, `\W`, `\s`, `\S`, `\-`) が含まれないことを assert する。`MorningLetterSection.key` については「`\-` を再導入しないこと」を pin する 2 本目の test も追加し、本 ADR 起源の pattern が将来 refactor で意図せず戻ることを CI で塞ぐ。

#### 検討した代替案

- **Schema → GBNF 翻訳のローカル simulation**: 本物の Ollama 翻訳ロジックは Go 実装で、Python から再現するのは fragile。実装スパゲッティを 2 重持ちする方が長期保守コストが大きい。
- **runtime preflight (pattern を post して 200 が返るか試す)**: deploy 時 1 度実行する health check を入れる案。CI で発火しない → 本件が CI で気付けず production まで行ったのと同じ穴を残す。
- **何もしない**: schema fix だけで Morning Letter は復旧する。ただし他 schema にも同型の地雷が残っている可能性があり、再発 1 回に耐えるコストよりは 1 回 lint test を書く方が安い。

## Consequences

### Pros

- Morning Letter LLM 経路が `format=json_schema` で正常応答するようになり、`is_degraded=false` で当日 letter が書かれる。UI の本文に LLM 由来 narrative / lead が戻り、[[000706]] の extractive fallback は本来の意図 (LLM 失敗時の degrade mode) に戻る。
- Ollama runtime バージョンが 3 つの派生 image で 0.22.1 に揃い、再現実験・production triage で「どの version で何が起きたか」が確定する。Gemma 4 renderer の改善 (0.22.1 release notes) も同時に取り込む。
- `format=json_schema` で Ollama に渡る全 schema が grammar-safe であることが CI gate で保証される。`MorningLetterContent` 以外で `\-` / `\d` / `\w` / `\s` を含む pattern を将来追加すると CI が落ちて気付ける。
- Morning Letter daemon の本来動作 ([[000705]] / [[000706]] / [[000707]]) が初めて end-to-end で稼働する。daemon → recap-worker → news-creator → recap-db → alt-backend → BFF → frontend の全経路が成立。
- `:latest` floating tag が 3 image 全部から消え、ops の長期衛生にも寄与する ([[000826]] と同系の rolling deploy 安定化方向)。

### Cons / Tradeoffs

- Ollama 0.22.1 → 0.23.0 のさらなる upgrade は別 ADR / 別 PR で評価する必要がある。本件で取り込む短期改善は Gemma 4 renderer 改善のみで、Claude Desktop integration や Metal hardening の恩恵は受けない。
- `test_grammar_safe_patterns.py` の `SCHEMAS_FORWARDED_TO_OLLAMA` リストは手動メンテナンスで、新しく `format=json_schema` で送る Pydantic model を追加するときに同リストへの登録を忘れる可能性がある。`news_creator/usecase/` 配下の `model_json_schema()` 呼び出し箇所を grep して同期する運用ルールをコメントで残しているが、構造的な保証ではない。
- `\-` という escape が「Python regex では合法、GBNF では非合法」という二重契約は news-creator のレビュアー視点で見えにくい。今回 ADR と test で pin したが、Pydantic を同じ pattern で使う他サービスにも同じ落とし穴が残る (本件 scope 外)。
- production への反映は `git push origin main` 経由の rolling deploy となり、news-creator-backend を含む 3 つの Ollama 派生 image が rebuild される。volume 互換のため model 再 import は不要だが、container 再起動で 12k model の preload (entrypoint の `keep_alive: 24h` 駆動) が一度走り、初回 letter 生成までに 1-2 分の暖機時間が乗る。
- `MORNING_DAEMON_ENABLED` flag そのものの撤去 / `MODEL_ROUTING_ENABLED` の有効化検証 / 過去 `morning_letters` row の backfill は引き続き別 Issue 扱い。

## Deploy-model 整合性セルフチェック

本 ADR は news-creator の Python schema 1 行修正、3 つの Dockerfile の `FROM ollama/ollama:...` タグ pin、Python テスト 1 ファイル追加で構成される。新規 compose service / one-shot init container / 新規名前付き volume の追加はない。

- [x] 新設 compose service が `depends_on` で他 service を gate するか？ — N/A（compose 構成変更なし）
- [x] 新設 compose service に `restart: "no"` の one-shot init container を含むか？ — N/A
- [x] 新設名前付き volume を populate する責務はどこにあるか？ — N/A（既存 `news_creator_models` / `augur_models` の互換性は維持、再 populate なし）
- [x] いずれも該当しない: rolling 互換と判定して pass。Atlas migration / DB schema 変更も伴わないため、`git push` 由来の通常の per-service rolling deploy で安全に反映できる ([[deploy]] / dispatch-deploy.yaml → alt-deploy 経路)。
