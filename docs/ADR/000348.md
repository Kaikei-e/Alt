# 000348: フィード健全性ステータスUIの実装 — feed_link_availability データのAPI公開とフロントエンド可視化

## ADR's STATUS

Accepted

## CONTEXT

`/sv/settings/feeds` ページはRSSフィードのURL一覧・購読トグル・削除ボタンのみを表示していた。一方、バックエンドには `feed_link_availability` テーブルにフィードの健全性データ（`is_active`, `consecutive_failures`, `last_failure_at`, `last_failure_reason`）が既に存在していたが、APIで公開されておらず、UIにも反映されていなかった。

この状況では以下の問題があった:

1. **フィードの異常に気づけない** — 何日も取得に失敗しているフィードがあっても、管理者はDBを直接確認しない限り把握できない
2. **運用判断に情報が不足** — どのフィードを残し、どのフィードを削除すべきか判断する材料がUI上にない
3. **既存データの活用不足** — `feed_link_availability` は pre-processor が既にメンテナンスしているのに、ユーザーに届いていない

**目的**: フィードの健全性をパッと見で把握できるようにする。

**制約**: 外部ライブラリ追加なし。TDD-first。Clean Architecture遵守。

## DECISION MAKING

### 1. ヘルスステータスの分類体系（5段階）

`feed_link_availability` のデータから以下の5段階に分類する:

| ステータス | 条件 | 色 | SVGシェイプ |
|-----------|------|-----|------------|
| **Healthy** | active, 失敗0回 | Green (#22c55e) | 塗りつぶし円 |
| **Warning** | active, 失敗1〜2回 | Amber (#f59e0b) | 塗りつぶし三角 |
| **Error** | active, 失敗3回以上 | Red (#ef4444) | 塗りつぶし四角 |
| **Inactive** | is_active=false | Gray (#9ca3af) | 輪郭円 |
| **Unknown** | availability レコードなし | Gray (#6b7280) | 破線円 |

**閾値の根拠**: Warning/Error の境界を3回に設定した。pre-processor のフィード無効化閾値（`ShouldDisable`）より手前で警告を出すことで、無効化される前にユーザーが対応できる。

**アクセシビリティ**: 色＋形状＋テキストラベルの3重シグナルにより、色覚異常のユーザーにも情報が伝わる。全要素に `aria-label` を付与。

### 2. バックエンドのアプローチ — 既存エンドポイントの拡張

新しいエンドポイントを追加するのではなく、**既存の `ListRSSFeedLinks` エンドポイントを拡張**してヘルスデータを含める方式を採用した。

- **Proto**: `RSSFeedLink` message に `health_status`, `consecutive_failures`, `last_failure_reason`, `is_active` フィールドを追加（field numbers 3-6、後方互換）
- **REST**: 同一パス `/v1/rss-feed-link/list` のレスポンスにヘルスフィールドを追加
- **SQL**: `feed_links LEFT JOIN feed_link_availability` で取得。LEFT JOINにより availability レコードがないフィードも返却される（Unknown ステータス）

**代替案の検討**:

| 案 | 評価 |
|----|------|
| 新規エンドポイント `/list/health` | フロントエンドで2回APIコールが必要。非効率。却下 |
| **既存エンドポイント拡張（採用）** | 後方互換性あり（新フィールドは追加のみ）。1回のAPIコールで完結 |
| フロントエンドで別途 availability API を呼ぶ | N+1問題。フィード数が多い場合にスケールしない。却下 |

### 3. Clean Architecture の遵守

既存の `ListFeedLinksUsecase` は温存し、新たに `ListFeedLinksWithHealthUsecase` を追加した。Port インターフェースに `ListFeedLinksWithHealth` メソッドを追加し、Gateway・Driver を拡張。

```
Handler → ListFeedLinksWithHealthUsecase → FeedLinkPort → FeedLinkGateway → AltDBRepository
```

Domain層に `FeedLinkWithHealth` 構造体と `GetHealthStatus()` メソッドを追加し、ヘルス分類ロジックをDomain層に閉じ込めた。

### 4. フロントエンドのコンポーネント設計

| コンポーネント | 役割 |
|--------------|------|
| `FeedHealthBadge` | 個別フィードのステータスドット＋ラベル＋ツールチップ。`compact` プロップで省スペース表示 |
| `HealthSummaryBar` | フィード全体の健全性サマリー（例: "32 feeds — 28 healthy, 3 warning, 1 error"） |

SVGシェイプはインラインSVGで描画し、外部ライブラリ不要。CSS変数ベースで全テーマ対応。

## RESULTS, EFFECTS

### PROS

- **即時的な健全性把握** — フィード一覧を開くだけで、どのフィードが正常でどのフィードに問題があるかが一目でわかる
- **3重アクセシビリティシグナル** — 色＋形状＋テキストで色覚異常にも対応
- **既存データの活用** — `feed_link_availability` テーブルのデータをそのまま使用。新たなデータ収集や定期ジョブは不要
- **後方互換性** — Proto の新フィールドは追加のみ。古いクライアントは新フィールドを無視するだけで動作する
- **外部依存なし** — SVGシェイプはインライン、ユーティリティ関数は純粋TypeScript。新ライブラリ追加なし
- **TDD** — Domain層の5ステータス分類、Driver層のLEFT JOIN、Usecase層のデリゲーション、フロントエンドのユーティリティ関数すべてにテストを先行して書いた

### CONS, TRADEOFF

- **Port インターフェースの拡張** — `FeedLinkPort` に `ListFeedLinksWithHealth` メソッドを追加したことで、既存の mock を手動更新する必要があった。`make generate-mocks` で再生成すれば解消されるが、今回は手動で追加した
- **LEFT JOIN のパフォーマンス** — フィード数が数千を超えた場合に `feed_links LEFT JOIN feed_link_availability` が遅くなる可能性がある。現状のフィード規模（数十〜数百）では問題ないが、将来的にはページネーション対応が必要になりうる
- **ヘルス分類のハードコード** — Warning/Error の閾値（2/3回）はコードに埋め込み。設定ファイルで変更可能にする選択肢もあったが、現時点では YAGNI として見送った
- **リアルタイム更新なし** — ヘルスステータスはページロード時またはリフレッシュボタン押下時に取得される。WebSocket による自動更新は将来の拡張候補

## APPENDIX

- `feed_link_availability` テーブルは pre-processor がフィード取得の成否に応じて更新している
- `FeedLinkAvailability.ShouldDisable(maxFailures)` メソッドが既に存在し、pre-processor がフィードの自動無効化判定に使用
- 本ADRの実装では `buf generate` で Go / TypeScript 両方の Proto コードを再生成した
- [[000380]] ResetFeedLinkFailures を UPSERT に変更し、フィードヘルスデータ欠落を修正
