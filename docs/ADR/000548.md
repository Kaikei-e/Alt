# ADR-000548: Inoreader取り込みガードのOCTET_LENGTH修正で日本語コンテンツ保護を回復

## Status

Accepted

## Date

2026-03-23

## Affected Services

- alt-backend (Go) — `CreateArticleInternal` ドライバ

## Context

[[000471]] で導入した「longest-content-wins」ガードは、Inoreader が同一記事をより短いコンテンツで再送信した際に既存の長いコンテンツを保護する仕組みである。しかし、このガードが日本語コンテンツに対して事実上無効であることが判明した。

### 根本原因: LENGTH と OCTET_LENGTH のミスマッチ

ガードの比較ロジックは以下のように動作していた:

1. PostgreSQL 側: `LENGTH(content)` で既存コンテンツの長さを取得 → **文字数**を返す
2. Go 側: `len(params.Content)` で新規コンテンツの長さを取得 → **バイト数**を返す
3. `existingLen > len(newContent)` で比較

日本語は UTF-8 で1文字あたり3バイトのため、この比較は本質的に壊れていた:

| 既存記事 | 新規（短い） | LENGTH (文字) | len() (バイト) | ガード判定 | 結果 |
|---|---|---|---|---|---|
| 1000文字 | 800文字 | 1000 | ~2400 | 1000 > 2400 → false | **上書き（バグ）** |
| 1000文字 | 300文字 | 1000 | ~900 | 1000 > 900 → true | 保護される |

つまり、新規コンテンツが既存の約1/3以下にならない限りガードが発動しなかった。Altは日本語RSSフィードを主に扱うため、この問題は広範囲に影響していた。

## Decision

**`getArticleContentLength` の SQL 関数を `LENGTH` から `OCTET_LENGTH` に変更する。**

`OCTET_LENGTH` はバイト数を返すため、Go の `len()` と同じ単位での比較が成立する。

### 修正スコープの限定

修正は Inoreader 取り込み専用パス（`CreateArticleInternal`）のみに適用する。汎用的な記事保存パス（`SaveArticle`）には longest-content-wins ガードを追加しない。

理由:
- `SaveArticle` はユーザー操作（記事保存、要約フロー）による意図的なコンテンツ更新に使われる汎用関数
- Inoreader 固有のガードロジックで `SaveArticle` の責務を汚染すべきではない（Clean Architecture: 責務の分離）
- `CreateArticleInternal` は Internal API（service-to-service）であり、Inoreader 取り込みパスに特化している

### 検討した代替案

| 案 | 評価 |
|---|---|
| Go 側で `utf8.RuneCountInString()` を使い文字数で比較 | 動作するが、`OCTET_LENGTH` の方が変更が1行で済みシンプル |
| SQL に `CASE` 式でガードを組み込む | [[000452]] の「SQL にビジネスロジックを持たせない」原則に違反 |
| `SaveArticle` にもガードを追加 | 汎用記事保存の責務を超える。Inoreader 固有のロジックは専用パスに閉じるべき |

## Consequences

### Pros

- 日本語コンテンツに対して longest-content-wins ガードが正しく機能する
- ASCII コンテンツでは `OCTET_LENGTH` = `LENGTH` のため、既存の動作に影響なし
- 変更が1行（SQL 関数名のみ）で、リスクが最小限
- Inoreader 取り込みパスにガードが閉じており、他のコードパスへの副作用がない

### Cons / Tradeoffs

- マルチバイト文字とASCIIが混在するコンテンツでは、バイト数ベースの比較がやや不正確になる可能性がある（例: 同じ文字数でも日本語の方がバイト数が大きい）。しかし、ガードの目的は「明らかに短いコンテンツでの上書き防止」であり、精密な文字数比較ではないため実用上問題ない
- `SaveArticle` パスにはガードがないため、要約フローなど別パスからのコンテンツ上書きは引き続き可能。これは設計上の意図的な判断であり、将来的に問題になった場合はそのパス固有の対策として別途検討する
