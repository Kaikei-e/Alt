# Redis Streams OOM 防止: XADD MAXLEN による自動トリミングの導入

## ADR's STATUS

完了

## CONTEXT

負荷テスト (`docs/perf/feed-load-test-report.md`) において、Redis Streams が `maxmemory` (512MB) に到達し、111件の OOM エラーが発生した。

```
OOM command not allowed when used memory > 'maxmemory'.
```

根本原因は `XADD` 時にストリームサイズの上限 (`MAXLEN`) が設定されておらず、ストリームが無制限に成長するため。テスト中は consumer が処理しきれないペースで `ArticleCreated` イベントが publish され、未消費メッセージが蓄積してメモリを圧迫した。

負荷テストレポートの対策案には `allkeys-lru` への eviction policy 変更が含まれていたが、調査の結果これは **不適切** と判断した。`allkeys-lru` は Redis がストリームキー自体を evict する可能性があり、consumer group のメタデータが消失して `NOGROUP` エラーを引き起こす。

## DECISION MAKING

### 方針: `noeviction` 維持 + publish 時 approximate trimming

検討した代替案:

| 案 | 内容 | 判定 |
|----|------|------|
| A. `allkeys-lru` に変更 | Redis が LRU でキーを自動 evict | 不採用: ストリームキー自体が evict され consumer group が消失する |
| B. `XTRIM` を定期バッチ実行 | cron 等で定期的にストリームをトリミング | 不採用: 運用コスト増、トリミング間隔の間に OOM が再発しうる |
| **C. `XADD MAXLEN ~` で publish 時トリミング** | 各 XADD に approximate MAXLEN を付与 | **採用**: アプリケーションレベルで制御、追加インフラ不要 |

### 設計判断

**MAXLEN は Driver 層に配置**: `StreamPort` インターフェースの変更は不要。MAXLEN は Redis 固有の実装詳細であり、ドライバーの責務として閉じ込める。usecase / gateway 層は変更不要。

**Approximate trimming (`~`)**: 正確なトリミング (`MAXLEN <count>`) と比較して CPU 効率が高い。Redis はマクロノード単位でトリミングを行うため、`XADD` のブロッキングを最小化できる。実際のストリーム長は指定値をやや超過しうるが、OOM 防止の目的には十分。

**デフォルト 10,000 エントリ**: 3 consumer group (pre-processor, search-indexer, tag-generator) がバッチサイズ 10 で消費する構成において、十分な消費マージンを確保。Fat Event (~30KB) x 10,000 = ~300MB であり、1GB の maxmemory に収まる。

**`maxmemory` を 512MB → 1GB に増加**: トリミングと併用し、バーストトラフィック時の余裕を持たせる。

### 変更概要

| ファイル | 変更内容 |
|----------|----------|
| `mq-hub/app/config/config.go` | `StreamMaxLen` フィールド追加 (`STREAM_MAX_LEN` env, default: 10000) |
| `mq-hub/app/driver/redis_driver.go` | `RedisDriverOptions.StreamMaxLen` 追加、`Publish`/`PublishBatch` の `XAddArgs` に `MaxLen` + `Approx` を条件付き適用 |
| `mq-hub/app/main.go` | config → driver options への `StreamMaxLen` 受け渡し |
| `mq-hub/app/driver/redis_driver_test.go` | MAXLEN トリミングの単体テスト 3 件追加 (miniredis) |
| `compose/mq.yaml` | `--maxmemory 1gb`、`STREAM_MAX_LEN=10000` 追加 |

### 実装パターン

`RedisDriver` に `streamMaxLen` フィールドを持たせ、`Publish` / `PublishBatch` で `XADD` の引数に条件付きで適用する:

```go
args := &redis.XAddArgs{
    Stream: stream.String(),
    Values: values,
}
if d.streamMaxLen > 0 {
    args.MaxLen = d.streamMaxLen
    args.Approx = true
}
```

`streamMaxLen` が 0 の場合はトリミングなし (後方互換性を維持)。

## RESULTS, EFFECTS

### 検証結果

| 検証項目 | 結果 |
|----------|------|
| 単体テスト (`go test ./...`) | driver / gateway / usecase / domain 全パス |
| コンテナ再ビルド・起動 | redis-streams, mq-hub 共に healthy |
| `CONFIG GET maxmemory` | 1073741824 (1GB) |
| `CONFIG GET maxmemory-policy` | noeviction |
| mq-hub ヘルスチェック (`/health`) | `{"healthy":true,"redis_status":"connected"}` |

### PROS

- **OOM 防止**: ストリームが無制限に成長することがなくなり、メモリ枯渇による publish 失敗を回避
- **Consumer group の安全性**: `noeviction` を維持することで、ストリームキーの意図しない evict と consumer group 消失を防止
- **インターフェース非破壊**: `StreamPort` の変更なし。Driver 層内で完結するため、既存の usecase / gateway コードへの影響ゼロ
- **設定可能**: `STREAM_MAX_LEN` 環境変数でワークロードに応じた調整が可能
- **後方互換**: `STREAM_MAX_LEN=0` (または未設定でなくとも 0 を明示) でトリミング無効化可能

### CONS, TRADEOFF

- **メッセージ喪失の可能性**: consumer の処理が遅延し、10,000 エントリ以上の未消費メッセージが蓄積した場合、古いメッセージはトリミングされて消費されない。現行のワークロードでは十分なマージンがあるが、consumer 停止時に注意が必要
- **Approximate の不正確さ**: `MAXLEN ~` は正確なカウントを保証しない。実際のストリーム長は指定値をやや上回る可能性がある (Redis の内部ノード単位でトリミングされるため)
