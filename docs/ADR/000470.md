# ADR-000470: AppendRecallSignal の JSON 構文エラーを修正する

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-backend (Go)

## Context

[[000468]] の修正で RecallProjectorJob が `user_count=1` でユーザーを走査するようになったが、`recall_signals` テーブルが依然として 0 行のままであった。alt-backend のログに以下のエラーが繰り返し出力されていた:

```
failed to append recall signal: AppendRecallSignal: ERROR: invalid input syntax for type json (SQLSTATE 22P02)
```

記事を開く操作（`TrackHomeActionUsecase`）が `AppendRecallSignal` を呼ぶたびに INSERT が失敗し、recall signal が一切蓄積されない。その結果、RecallProjectorJob が走査しても `signals_processed=0` のままで、Recall Rail に候補が表示されない状態が続いていた。

### 根本原因

`recall_signal_repository.go` で `json.Marshal(signal.Payload)` の結果（`[]byte`）をそのまま pgx の `Exec` に渡していた。pgx v5 は `[]byte` を PostgreSQL の `bytea` 型として送信するため、`recall_signals.payload` カラム（`JSONB NOT NULL`）への暗黙変換が存在せず、型不一致エラーが発生していた。

プロジェクト内の他のリポジトリは全て `string()` にキャストしてからパラメータに渡しており、`recall_signal_repository.go` だけがこのキャストを欠いていた:

- `knowledge_home_repository.go`: `string(tagsJSON)`, `string(whyJSON)`
- `recall_candidate_repository.go`: `string(reasonJSON)`
- `today_digest_repository.go`: `string(topTagsJSON)`, `string(pulseRefsJSON)`

## Decision

`recall_signal_repository.go` の `payloadJSON` パラメータに `string()` キャストを追加し、既存の全リポジトリと同一パターンに統一する。

併せて、`signal.Payload` が nil の場合のガードを追加する。`json.Marshal(nil)` は `[]byte("null")` を返すが、`JSONB NOT NULL DEFAULT '{}'` カラムに JSON null が入ると後続の `jsonb_array_elements` 等で問題になりうるため、空 map にフォールバックする。

### 変更内容

```go
// Before
payloadJSON, _ := json.Marshal(signal.Payload)
// ...
signal.SignalStrength, signal.OccurredAt, payloadJSON,

// After
payload := signal.Payload
if payload == nil {
    payload = map[string]any{}
}
payloadJSON, _ := json.Marshal(payload)
// ...
signal.SignalStrength, signal.OccurredAt, string(payloadJSON),
```

### テスト

TDD で以下 2 ケースを pgxmock ベースの単体テストとして実装した:

- payload が `map[string]any{...}` の場合、`string` 型（テキストプロトコル）としてパラメータが渡される
- payload が nil の場合、`"{}"` が渡される（JSON null ではなく空オブジェクト）

## Consequences

### Pros

- `recall_signals` テーブルへの INSERT が正常に動作し、signal が蓄積されるようになる
- RecallProjectorJob が signal を処理でき、Recall Rail に候補が表示される
- プロジェクト内の全 JSONB パラメータ渡しパターンが `string()` キャストに統一される
- nil payload のエッジケースでも安全に動作する

### Cons / Tradeoffs

- `string()` キャストは pgx のテキストプロトコルを利用する回避策であり、pgx が将来的に `[]byte` → JSONB の自動変換をサポートすれば不要になる。ただし現時点では PgBouncer 互換性の観点からもテキストプロトコルが推奨される
