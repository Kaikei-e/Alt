# ADR-000839: Knowledge Loop transition エラーを Connect-RPC code 単位で分類し BFF で HTTP status を細分化する

## Status

Accepted

## Date

2026-04-24

## Affected Services

- alt-backend (Go) — `app/connect/v2/knowledge_loop/handler.go`, `app/usecase/knowledge_loop_usecase/errors.go` (新規), `transition_knowledge_loop_usecase.go`
- alt-frontend-sv (TypeScript / Svelte) — `src/routes/(app)/loop/transition/+server.ts`, `src/lib/connect/error.ts` (新規)

## Context

ブラウザ側で Knowledge Loop の `/loop/transition` POST が 502 を返す事象が観測された。nginx アクセスログでは `rt=0.005` の高速失敗で、同時刻に alt-backend の mTLS peer ログは `alt-butterfly-facade` からの `KnowledgeLoopService.TransitionKnowledgeLoop` 受信を記録していたが、handler 側のエラーログ (`transition_knowledge_loop failed`) は出ていなかった。alt-butterfly-facade が起動直後で、直前の数十秒には `sovereign GetProjectionCheckpoint: unavailable: dial tcp ...:9500 connect: connection refused` が大量発生していた。起動直後の transient な upstream unreachable である可能性が高かったが、BFF レスポンスボディは `{"error":"upstream_failure"}` のみで、実際にどの種類の失敗なのかクライアントから区別できない状態だった。

既存設計の問題点:

1. **alt-backend handler**: `transition_knowledge_loop/handler.go` の `TransitionKnowledgeLoop` は `ErrInvalidArgument` のみを `connect.CodeInvalidArgument` に分類し、**それ以外のすべてのエラー**（dial refused / context deadline / pg driver 失敗 / wrap された予期せぬ err）を `connect.CodeInternal` に潰していた。
2. **alt-frontend-sv BFF**: `/loop/transition/+server.ts` の catch ブロックは `already_exists / failed_precondition / invalid_argument / unauthenticated / permission_denied` のみ個別マップし、**それ以外のすべて**（`internal` / `unavailable` / `deadline_exceeded` / `resource_exhausted` / `unknown` / ConnectError ですらない fetch TypeError）を `{error: "upstream_failure", status: 502}` に集約していた。

結果として、クライアントは 502 を受けても「transient upstream 切断 (retry すべき)」「server 側 invariant 違反 (retry しても無駄)」「fetch 層のネットワーク不達 (escalate)」を区別できず、ユーザ体験としても観測としても運用としても劣化していた。Connect-RPC 仕様 ([connectrpc.com/docs/protocol#error-codes](https://connectrpc.com/docs/protocol#error-codes)) は 16 種のエラーコードと HTTP status を明確に定義しており、既存実装はその規格を活かせていなかった。

関連する不変条件として、本変更では `knowledge_events` の append-only 性、`knowledge_loop_transition_dedupes` の ingest-only barrier（[[000831]] §3.7）、runbook `knowledge-loop-reproject.md` が規定する read model 破棄→再投影の動作に一切影響を与えないことが求められた。エラー分類は read path（失敗レスポンスの形を決めるパス）の変更であり、event/projection のスキーマや書き込み経路には触らない。

## Decision

alt-backend の usecase 層に sentinel `ErrUpstreamUnavailable` と分類ヘルパ `ClassifyDriverError` を導入し、Connect handler が `errors.Is` ベースで以下の通り Connect-RPC code に dispatch する。BFF 側は受け取った code を Connect-RPC 仕様の HTTP status にほぼそのままマップする（ただし `unavailable` は 503 ではなく 502 とし、「上流到達不可なので BFF が gateway として 502 を返す」ゲートウェイセマンティクスを優先）。

### BE: alt-backend 分類ルール

| 原因 | 返す Connect code |
|---|---|
| `errors.Is(err, ErrInvalidArgument)` | `CodeInvalidArgument` |
| `errors.Is(err, context.DeadlineExceeded)` | `CodeDeadlineExceeded` |
| `errors.Is(err, context.Canceled)` | `CodeCanceled` |
| `errors.Is(err, ErrUpstreamUnavailable)` (driver が `net.Error` を返した場合 `ClassifyDriverError` が wrap) | `CodeUnavailable` |
| それ以外 | `CodeInternal` + ERROR ログ |
| JWT context 欠落 | `CodeUnauthenticated`（従来通り） |

`ClassifyDriverError` は driver 層の err を受けて、`context.DeadlineExceeded` / `context.Canceled` は素通し、`errors.As(err, &netErr)` で `net.Error` にマッチしたら `errors.Join(ErrUpstreamUnavailable, err)` で sentinel を加え、原因を `%w` 相当で保持する。これにより handler は driver 実装の詳細を知らずに `errors.Is` だけで分類できる。

### BFF: alt-frontend-sv マッピング

`$lib/connect/error.ts` に `extractConnectCode(err)` を切り出し、`/loop/transition/+server.ts` の catch を switch に書き換える。

| Connect code | HTTP status | body.error |
|---|---|---|
| `already_exists` | 200 | `accepted: true, replay: true` |
| `failed_precondition` | 409 | `projection_stale` |
| `invalid_argument` | 400 | `invalid_argument` |
| `unauthenticated` / `permission_denied` | 401 | `unauthorized` |
| `deadline_exceeded` | 504 | `timeout` |
| `resource_exhausted` | 429 | `rate_limited` |
| `unavailable` | 502 | `upstream_unavailable` |
| `internal` | 500 | `upstream_internal` |
| その他（`unknown` / `canceled` / `aborted` / fetch TypeError 等） | 502 | `upstream_unreachable` |

クライアント側は `upstream_unavailable` / `upstream_unreachable` / `timeout` を retry 候補、`upstream_internal` を fail-fast、`rate_limited` を `Retry-After` 尊重、として扱える。

### 検討した代替案

- **handler 側のみ修正する案**: handler だけ細分化し BFF の default 502 は据え置く。→ BFF が `CodeUnavailable` を 502 に丸めるとクライアントは distinguish できず意味がない。却下。
- **sentinel を handler に直接定義する案**: sentinel を `connect/v2/knowledge_loop` package に置く。→ Clean Architecture 上 handler は usecase の詳細を知るべきでない (handler → usecase の単方向)。sentinel の所有権は usecase にあるべき。却下。
- **driver 層で net.Error を検出して port が sentinel を返す案**: port interface に `ErrUpstreamUnavailable` を宣言して driver が直接返す。→ 現時点で driver は 1 種類しかなく、過剰設計。usecase 層 1 箇所でのラッピングで足りる。将来 driver が複数になったら再検討。却下。
- **HTTP 503 を採用する案**: Connect-RPC の `unavailable` は仕様上 HTTP 503 に対応する。→ BFF はゲートウェイの立場であり、「上流が unavailable だから 502 bad gateway」の方が HTTP セマンティクスに合う。nginx などの既存 502 扱い（リトライバック設定など）との整合も取れる。502 を採用。
- **FE 側で retry/backoff を本変更に同梱する案**: 502/503/504 を受けたら client 側で backoff retry する。→ 今回の scope は「失敗モードを区別可能にする」ことであり、retry ポリシーは別 decision として切り分ける。UI 表示の粒度も含めて後続で議論。

## Consequences

### Pros

- `/loop/transition` のクライアントが「retry 可能な transient」「恒久失敗」「fetch 層の不達」を HTTP status だけで判断できるようになり、UX と運用可視性が揃う。
- Connect-RPC 仕様準拠のマッピングになり、今後別の Connect-RPC consumer（他ルート、他 BFF、将来の mobile app 等）がヘルパ `extractConnectCode` と同じマッピングを流用できる。
- sentinel `ErrUpstreamUnavailable` により、他の usecase (`GetKnowledgeLoopUsecase` や将来の loop 系 write path) も同じ分類を再利用できる。handler 側の分岐コードも `errors.Is` ベースで declarative に保てる。
- `knowledge_events` / `knowledge_loop_transition_dedupes` / projection 群のスキーマ・書き込み経路には一切影響しない。read path の error reporting を直しただけで、append-first / reproject-safe / dedupe ≠ projection の不変条件（[[000831]] §3.3 / §3.5 / §3.7）はそのまま維持。
- エラーメッセージは短い sentinel 文字列 (`"internal"`, `"unavailable"` 等) に固定しており、`projected_at` / internal state の漏洩経路を新設しない（runbook `knowledge-loop-reproject.md §Troubleshooting` の security-critical 項目）。

### Cons / Tradeoffs

- `extractConnectCode` は `err.code` の shape に依存している。connect-web が将来 ConnectError の公開 API を変えたら helper を合わせて追従する必要がある（現状 `code: string` は長く stable）。
- BE 側で `Unauthenticated` / `PermissionDenied` / `NotFound` / `Aborted` / `OutOfRange` / `Unimplemented` / `DataLoss` / `ResourceExhausted` を return するロジックは `TransitionKnowledgeLoop` に現在存在しない。BFF 側のマッピングはそれらを defensive に用意してあるが、BE が実際にそれらのコードを返し始めるのは別途 usecase 拡張が必要。
- retry/backoff はクライアント側に残された宿題。502/503/504 を受けた UI の挙動（指数バックオフ、ユーザ通知、最大試行回数）は未決。
- Handler の switch 文がやや長くなる。将来コード種類が増えたらテーブル駆動化を検討する。
