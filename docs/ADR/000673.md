# ADR-000673: Acolyte Writer を paragraph micro-generation に移行し Critic heuristic 主系化・revision delta 縮小・Postgres checkpointer を導入する

## Status

Accepted

## Date

2026-04-10

## Affected Services

- acolyte-orchestrator (Python 3.14+)

## Context

[[000671]] で skeleton Planner・degradation ladder・`/api/chat` 移行を行い Extractor の JSON 完走率を改善したが、レポート `743372a2` の v4 ログ分析（`review/acolyte-pipeline-log-analysis-2026-04-10.md`）で以下の構造的問題が判明した。

### Writer の thinking 枯渇

| Revision | num_predict | eval_count | response_len | 結果 |
|----------|-------------|------------|--------------|------|
| rev-1 | 2000 | 2000 | 699 | 本文生成 ✓ |
| rev-2 | 2000 | 2000 | 661 | 本文生成 ✓ |
| rev-3 | 2000 | 2000 | **0** | thinking が全予算消費 |

Gemma4 の thinking tokens は `num_predict` を共有する（Ollama [#10925](https://github.com/ollama/ollama/issues/10925) OPEN）。Writer が section 全体を 1 回で生成する設計では、revision 時に prompt が長くなり thinking が暴走して本文ゼロに落ちる。Finalizer は最終 revision を保存するため、rev-1/2 の成果が失われる。

### ES/Conclusion の claims=0 空本文

SectionPlanner で conclusion と executive_summary が JSON 切断 → claims=0 → Writer が空本文を即座に返す。raw evidence からの独立生成が脆い。

### Critic の LLM 浪費

body=0 は FM3「warning」のため LLM critic まで到達し、LLM critic が「revise」を返してさらに Writer が空本文を再生成するループが 3 回回る。

### 2 時間パイプラインの脆弱性

pipeline 完了に約 2 時間を要し、途中失敗で全成果が失われる。

## Decision

以下 5 つの変更を一括で採用する。

### 1. Writer を paragraph micro-generation に移行 (Issue 3)

**section 全体の 1 回生成をやめ、1 claim = 1 paragraph = 1 LLM call に分割する。**

- 新規ドメインモデル `GeneratedParagraph`（`domain/paragraph.py`）を追加
- `WriterNode._generate_paragraph()` で claim ごとに独立した LLM 呼び出し
- accepted paragraph は immutable — revision で再生成しない
- 空応答 → `status="rejected"` として他の paragraph に影響させない
- `best_sections` で最良 revision（blocking 最小 → 非空 → 最長）を追跡し、Finalizer が最終 revision ではなく最良 revision を保存
- prompt は XML タグ（`<topic>`, `<claim>`, `<supporting_quotes>`, `<delta_feedback>`）で責務を分離

**State 追加フィールド:**

```python
section_paragraphs: dict[str, list[dict]]  # paragraph 単位の生成結果
best_sections: dict[str, str]              # 最良 revision body
best_section_metrics: dict[str, dict]      # blocking_count + char_len
claim_feedbacks: dict[str, list[dict]]     # paragraph 単位の revision 指示
```

### 2. ES/Conclusion を deterministic synthesis ベースに変更 (Issue 4)

**claims=0 で空本文になる設計を解消する。LLM 失敗時は deterministic fallback で synthesis claims を生成する。**

- `_deterministic_conclusion_claims()`: analysis claims → synthesis claims に変換。analysis claims が空なら extracted_facts の上位を使用
- `_deterministic_es_claims()`: 全 section の claims → top-N synthesis claims。claims が空なら extracted_facts を使用
- ランキング: numeric_facts → source diversity → confidence の優先順
- SectionPlannerNode に pre-LLM fallback（claims 入力なし時）と post-LLM fallback（LLM が空 claims 返却時）の両方を実装

### 3. Critic を heuristic-first に再設計 (Issue 5)

**LLM 呼び出し前に body=0, claims=0, citation 不足を blocking として検出する。revision 指示を paragraph/claim 単位で返す。**

新規 FailureMode:

| FM | 名称 | severity | 検出条件 |
|----|------|----------|----------|
| FM4 | `empty_body` | blocking | `body.strip() == ""` |
| FM5 | `zero_claims` | blocking | `claims=[]` かつ evidence 存在 |
| FM11 | `es_numeric_absence` | warning | ES claims に numeric_facts なし |

- `detect_empty_body()`, `detect_zero_claims()`, `detect_es_numeric_absence()` を追加
- `_build_claim_feedbacks()`: heuristic 検出 + paragraph status から claim_id 単位の feedback を生成
- 後方互換: section-level `feedback` は維持し、`claim_feedbacks` を追加
- blocking heuristic があれば LLM critic をスキップ（既存アーキテクチャを維持）

### 4. revision feedback を短い delta に縮小 (Issue 8)

**revision prompt から過去 feedback 全文再掲を排除する。**

- `_generate_paragraph()` は `delta_feedback` パラメータのみ受け取る（蓄積しない）
- revision 時: `claim_feedbacks` から該当 claim の `reason` を 1 行で取得
- accepted paragraph は prompt に含めない
- revision count が増えても prompt 長は一定

### 5. LangGraph Postgres checkpointer 導入 (Issue 6)

**super-step 境界で state を保存し、failure 後に resume 可能にする。**

- 依存追加: `langgraph-checkpoint-postgres>=2.0.0`
- `AsyncPostgresSaver` を `checkpoint_factory.py` で context manager として管理（app startup/shutdown ライフサイクル）
- `build_report_graph(checkpointer=...)` パラメータ追加
- `checkpoint_enabled` 設定（デフォルト `False`、既存テストへの影響なし）
- `thread_id` = `acolyte-run:{run_id}` で namespace 化
- DB は既存の `acolyte_db_dsn` を共用（checkpoint テーブルは `checkpoint_` prefix で衝突なし）

**制約**: LangGraph checkpointer は super-step（node 境界）で保存する。1 node 内で複数 article をループする QuoteSelector / FactNormalizer / Writer では mid-node failure からの途中再開はできない。node 分割は将来課題。

## Consequences

### Pros

- **thinking 枯渇耐性**: 1 paragraph 単位の小さい LLM 呼び出しにより、thinking が暴走しても他の paragraph が失われない
- **空本文の根本解決**: best_sections + deterministic fallback + heuristic blocking の 3 層防御
- **LLM コスト削減**: body=0 / claims=0 を heuristic で即座に検出し、無駄な LLM critic 呼び出しを排除
- **prompt 長の安定化**: revision count に依存しない一定長 prompt で thinking 枯渇を抑制
- **durability**: 2 時間パイプラインの途中成果が checkpoint で保全される
- **後方互換**: legacy evidence-based path は完全に維持。claim_plans が None の場合は旧動作

### Cons / Tradeoffs

- **LLM 呼び出し回数増加**: 7 claims × 3 sections = 21 calls（従来 3 calls）。ただし各 call の num_predict は小さく、total token 生成量は同等以下
- **paragraph 間の文体不統一**: 独立生成のため段落間のトランジションが弱い。将来 smoothing pass を追加可能
- **checkpointer の mid-node 制約**: node 内ループの途中からは再開できない。task 化または node 分割が将来課題
- **State サイズ増加**: `section_paragraphs` で paragraph 単位のメタデータが追加される（ただし body 自体は `sections` と重複せず metadata overhead のみ）
