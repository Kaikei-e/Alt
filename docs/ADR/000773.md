# ADR-000773: pki-agent ローテに全 6 サービスの mTLS クライアントを追随させる

## Status

Accepted

## Date

2026-04-18

## Affected Services

- **recap-worker** — `src/clients/mtls.rs` に `ReloadingCertResolver` を新設し `rustls::ClientConfig::with_client_cert_resolver` に差し込む。`src/tls.rs` の `load_server_tls_config` も同じ resolver を `with_cert_resolver` で使う
- **rag-orchestrator** — `internal/infra/tlsutil/` パッケージを新設 (alt-backend の `certReloader` を移植) し、`internal/infra/httpclient/mtls.go` と `pool.go` を `tlsutil.LoadClientConfig` 経路に差し替え
- **acolyte-orchestrator** — `acolyte/infra/mtls_client.py` に `SslContextReloader` / `watch_cert_rotation` を追加、`main.py` のライフスパンで watcher タスクを spawn / cancel
- **recap-evaluator** — 同じ helper 追加、`src/recap_evaluator/main.py` lifespan で watcher タスクを spawn / cancel
- **recap-subworker** — 同じ helper 追加、`LearningClient.send_learning_payload` が送信前に `maybe_reload()` を呼ぶ per-request 方式
- **tag-generator** — `tag_generator/driver/connect_client_factory.py` に `_ReloadingBackendClient` proxy を追加、`__getattr__` 経由で pyqwest `SyncHTTPTransport` を mtime 変化時に再構築

## Context

本 ADR は 2026-04-18 に実発火した mTLS クライアント側の証明書ホットリロード欠落と、同じバグクラスを持つ 5 サービス分の追随対応を同一 PR でまとめて根治する。発端は 3days Recap ジョブが連続失敗した現象:

```
alt-backend articles request failed
  0: error sending request for url (https://alt-backend:9443/services.backend.v1.BackendInternalService/ListRecapArticles)
  1: client error (SendRequest)
  2: connection error
  3: received fatal alert: CertificateExpired
```

alt-backend (サーバ) が recap-worker のクライアント証明書を期限切れと判定して TLS alert 45 (`certificate_expired`) を送り返していた。ディスク上の `/certs/svc-cert.pem` は `pki-agent-recap-worker` が本日 07:00 UTC にローテ済みで `NotBefore: Apr 18 06:56:37 2026 GMT` / `NotAfter: Apr 19 06:57:37 2026 GMT` と有効だったにもかかわらず、`alt-recap-worker-1` のプロセスは 22 時間連続稼働しており、起動時に `reqwest::Identity::from_pem(...)` で読み込んだ**古い証明書を in-memory に握ったまま**だった。

alt-backend は [[000757]] / [[000759]] 系の実装で `alt-backend/app/tlsutil/tlsutil.go:66-104` の `certReloader` を `GetClientCertificate` / `GetCertificate` callback 経由で使う形になっており、handshake ごとに mtime を比較してロードし直している。**サーバ側は正しく追随、クライアント側は起動時 1 回で固定**という非対称が 2026-04-14 の mTLS cutover ([[000754]] 前後) から潜伏しており、今日のローテで顕在化した。

監査の結果、同型バグは計 6 サービスに分布していた:

| サービス | 言語 | 箇所 | 当時の発火状況 |
|---|---|---|---|
| recap-worker | Rust | `src/clients/mtls.rs:42-74` の `reqwest::Identity::from_pem` + `src/tls.rs:80-82` の `with_single_cert` | **発火中** (3days Recap 連続失敗) |
| rag-orchestrator | Go | `internal/infra/httpclient/mtls.go:40-56` と `pool.go:41-58` の `tls.LoadX509KeyPair` + `tls.Config.Certificates` (2 経路) | 休眠 (`MTLS_ENFORCE` 未設定) |
| acolyte-orchestrator | Python | `main.py:54-58` の `ssl.SSLContext.load_cert_chain` → `httpx.AsyncClient(verify=ctx)` | 休眠 (`MTLS_ENFORCE=false` デフォルト) |
| recap-evaluator | Python | `main.py:53-61` の同パターン | 休眠 |
| recap-subworker | Python | `services/learning_client.py:44-49` の同パターン | 休眠 |
| tag-generator | Python | `driver/connect_client_factory.py:44-61` の pyqwest `SyncHTTPTransport(tls_cert=, tls_key=, tls_ca_cert=)` | 休眠 (`TAG_GENERATOR_MTLS_ENFORCE=false`) |

一方、現行で reload-correct と判定できた Go サービスは alt-backend / auth-hub / pre-processor / search-indexer / alt-butterfly-facade / pki-agent の 6 件で、どれも alt-backend 由来の `tlsutil` をローカルにコピーした形だった。

関連文脈:

- [[PM-2026-028]] mTLS 証明書期限切れによる Knowledge Home 停止。当時はサーバ側 cert の扱いが焦点。
- [[PM-2026-031]] mTLS cutover 残タスクによる 3days Recap 4 日連続 404。REST ルートの listener 未配線が原因で、本 ADR の前段。
- [[000759]] `:9443` を Connect-RPC + REST ハイブリッドにし cascade を compose-native に戻す。本 ADR はその続きで、「cutover 系作業の残タスクが silent に潜む」第二弾の根治にあたる。

イミュータブル不変条件 (`knowledge_events` / projector / `summary_versions` / `tag_set_versions`) への影響はない — 本 ADR は transport 層だけを触り、event payload や projection には一切関与しない。

## Decision

各言語で「**mtime が進んだときだけ証明書を再読込み、handshake ごとに最新材料を提示する**」を共通設計原則とし、`alt-backend/app/tlsutil/tlsutil.go:66-104` の `certReloader` のセマンティクスを各ランタイムに移植する:

1. 起動時に初回ロードしてエラーならフェイル-クローズ
2. 各 handshake で `fs::metadata` → mtime 比較、進んでいれば再読込
3. 再読込失敗時 (atomic rotation 中の truncated window など) は直近の良好値へフォールバック
4. thread-safety は単一 mutex / `Arc` で守る

言語別の具体実装:

### recap-worker (Rust, 発火中 — 最優先)

`reqwest::Client::builder().use_preconfigured_tls(rustls::ClientConfig)` 経由に切り替え、`rustls::ClientConfig::with_client_cert_resolver(Arc<dyn ResolvesClientCert>)` に `ReloadingCertResolver` を渡す。この型は同時に `ResolvesServerCert` も実装し、`src/tls.rs` の `ServerConfig::with_cert_resolver` にも同一インスタンスを流し込めるので、inbound / outbound どちらの `:9443` handshake でもディスクの最新 cert が提示される。追加 crate は不要 (`rustls 0.23` / `rustls-pemfile 2.2` / `reqwest` の `rustls-tls` feature は既存)。unit test は `rcgen` (新規 dev-dep) で tempdir に ECDSA 自己署名を 2 組生成し mtime を `filetime` で前進させるスタイルで 4 ケース追加 — reload / cache / fallback / initial-load-failure。

### rag-orchestrator (Go)

`internal/infra/tlsutil/tlsutil.go` を新設し、alt-backend 由来の `certReloader` + `LoadClientConfig` を移植 (server 側関数は持たない: このサービスはサーバ mTLS を張らない)。`httpclient/mtls.go:NewMTLSClient` と `pool.go:loadMTLSTransport` を `tlsutil.LoadClientConfig` 経由に変え、`tls.Config.Certificates` への baked-in 代入を除去。`sync.Once` は残す — `tls.Config` 自体は 1 回だけ作り、`GetClientCertificate` callback が handshake ごとに呼ばれる設計。これにより接続プールを破棄せずに cert だけ差し替わる。

### Python httpx 系 3 サービス (acolyte-orchestrator / recap-evaluator / recap-subworker)

`ssl.SSLContext.load_cert_chain()` は同一コンテキストに対して複数回呼べることを利用し、`SslContextReloader(ctx, cert_path, key_path)` が mtime 監視下で **同じ `ssl.SSLContext` オブジェクトに対して in-place reload** する。既存の `httpx.AsyncClient(verify=ctx)` はその参照を握ったままで良く、gateway 層のシグネチャ変更はゼロ。ライフサイクルへの組み込みは:

- **acolyte-orchestrator / recap-evaluator** — `watch_cert_rotation(reloader, interval_seconds=30.0)` を `asyncio.create_task` で spawn し、lifespan shutdown で cancel する。Starlette / FastAPI の lifespan が自然な置き場なので最も侵襲性が低い。
- **recap-subworker** — ライフスパンに乗せる代わりに `LearningClient.send_learning_payload` の先頭で `self._mtls_reloader.maybe_reload()` を呼ぶ per-request 方式。`LearningClient` がリクエスト粒度で使われるため stat コストは無視できる (μs オーダー) し、ライフスパンフックを経由せず完結する方が簡素。

### tag-generator (Python, pyqwest)

pyqwest `SyncHTTPTransport` は cert/key バイト列を構築時に bake するため in-place reload が効かない。`_ReloadingBackendClient` proxy を `driver/connect_client_factory.py` に追加し、内部で `BackendInternalServiceClientSync` を保持。`__getattr__` で RPC 属性を現在の inner にフォワードし、その手前で `_maybe_reload()` が mtime を見て新しい transport + http_client + typed client を組み直す。既存の `ConnectArticleFetcher(client, ...)` / `ConnectTagInserter(client, ...)` はダックタイピングで変更なし。pyrefly が static type を通すため `cast("BackendInternalServiceClientSync", reloading)` を 1 箇所入れる。再構築失敗時は直近 inner を保持 (`BLE001` の `# noqa` で抑えた上で `logger.warning`)。

### 却下した代替案 (共通)

- **定期プロセス再起動 (cron / systemd timer)** — 24h 内に 1 回必ず停止、in-flight 処理が落ちる。[[000759]] の netns 依存パターンとも衝突。
- **pki-agent から SIGHUP / `docker restart`** — sidecar の機能 liveness と親の可用性を結合するアンチパターン。[[PM-2026-031]] で指摘した「sidecar 機能 liveness が親サービス状態と乖離する」教訓を逆行させる。
- **per-request に client 再生成 (Rust / Python 共通)** — 接続プールを失う。recap-worker の paginated fetch (500 件/ページ × N) で handshake コスト増大。採用した resolver / 同一 `SSLContext` in-place reload 方式は pool を温存する。
- **Python 4 サービスで `ReloadingMtlsHttpx` ファサード化** — gateway 層が `httpx.AsyncClient` 型ヒントを握っているため大規模な touch が要る。`SSLContext.load_cert_chain` が in-place reload できる事実を使って gateway 層はゼロ touch に絞った。
- **`alt-mtls-py` 共有パッケージ化** — すでに 4 サービスで同型 `mtls_client.py` が生きているので CLAUDE.md の「5 サービスで切り出し」基準に近いが、本 PR で切ると変更範囲が広がるため、次 ADR に fast-follow として残す。

### スコープ外 (fast-follow 別 ADR 予定)

- CA bundle 側のホットリロード (leaf の 24h 対比 CA は年単位、優先度低)
- 共有 Python パッケージ `alt-mtls-py` の切り出し
- tag-generator の pyqwest → httpx 移行による proxy レイヤ撤廃

## Consequences

### Pros

- **recap-worker の 3days Recap が同日から連続完走可能に**なる。コンテナ再起動の運用回避がスタックとして残らない
- **recap-worker 以外の 5 サービスも、mTLS を有効化した瞬間から同じ罠を踏まない**。cutover が将来別チームで再開しても silent failure を再生産しない
- **接続プールを温存**したままローテを受けられる。Rust 側は `use_preconfigured_tls(rustls::ClientConfig)` + resolver で TLS 層だけ差し替わり、Go 側は `GetClientCertificate` callback でハンドシェイク粒度の reload、Python httpx 側は `SSLContext` の in-place reload で既存の `httpx.AsyncClient` がそのまま有効
- **gateway / DI / compose / proto は一切触らない**。blast radius が小さく、ロールバックしやすい
- **TDD で回帰防止を押さえた**: recap-worker 4 tests / rag-orchestrator 4 tests / 各 Python サービス 3-4 tests、全 1,669 tests pass (322 recap-worker lib + 各 Python 139/290/518/400 + rag-orchestrator 全パッケージ)
- **reloader パターンは 5 Go サービスで実戦済み**のものを移植したので、Go 側は設計レビューのコストが小さい

### Cons / Tradeoffs

- **`certReloader` ロジックが 7 サービスに複製**された (alt-backend 系 Go 6 + rag-orchestrator + Python 4 + tag-generator 1)。CLAUDE.md の「5 サービス以上で shared package 化」基準を超過した状態で放置している。fast-follow で `alt-mtls-go` / `alt-mtls-py` / `alt-mtls-rs` のいずれかに統合したい (別 ADR)
- **Python 側の watcher は 30s ポーリング**。最悪 30 秒分の古い cert を提示し続ける窓が残るが、24h ローテに対しては十分な余裕。per-request stat に落としてもよいが、gateway 層 touch が増えるので見送り
- **tag-generator の `_ReloadingBackendClient` は `__getattr__` でダックタイピング**しているので static type 解析の助けが薄い。`cast` で pyrefly を黙らせた箇所が 1 つあり、将来 pyqwest が httpx と足並みを揃えれば proxy ごと削れる
- **CA bundle 側は依然 startup-baked**。alt-ca Intermediate CA / Root CA が交換される頻度は年単位なので優先度は低いが、交換時はプロセス再起動が必要 — この注意は runbook 側で補足する予定
- **recap-subworker は per-request stat** 方式なので、他 2 Python サービスと watcher の書き味が非対称。背景タスクに寄せ直すのは fast-follow
