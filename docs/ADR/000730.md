# ADR-000730: recap-subworker の Phase 2 per-operation Usecase 分割

## Status

Accepted

## Date

2026-04-15

## Affected Services

- recap-subworker (Python 3.14 / FastAPI)

## Context

[[000726]] で合意した 6 フェーズの大規模リファクタリングの **Phase 2** を実施する。Phase 1 ([[000728]]) で lifespan DI と単体テストアンブロックは完了したが、handler 層は依然として `services.run_manager.RunManager` 具象クラスに直接依存しており、Clean Architecture 規約「Handler → Usecase → Port → Gateway → Driver」における Handler → Service の層越境が残っていた。

具体的には `recap_subworker/app/routers/runs.py` と `classification_runs.py` が `Depends(get_run_manager_dep)` で 889 LOC の `RunManager` 全体を受け取り、そのうち必要な `create_run` / `create_classification_run` / `get_run` の 3 メソッドしか呼んでいない状態だった。このためテストで monolithic な `FakeRunManager` を用意する必要があり、単一操作だけをモックしたいケースでもクラス全体を実装する過剰負担が発生していた。

Phase 2 のゴールは **Handler と RunManager の間に操作単位の Usecase を挟み、そのコラボレーターを Port Protocol で型付けする**ことで、層の責務を明確化する。ただし `RunManager` 本体の分割（run_manager.py の 888 LOC を複数ファイルに分解）は Pact 契約への影響が大きいため Phase 3 以降に持ち越す。

## Decision

以下を採用する:

1. **4 つの per-operation Usecase を追加する**:
   - `SubmitRunUsecase.execute(submission) -> RunRecord`
   - `GetRunUsecase.execute(run_id) -> RunRecord | None`
   - `SubmitClassificationRunUsecase.execute(submission) -> RunRecord`
   - `GetClassificationRunUsecase.execute(run_id) -> RunRecord | None`

2. **Port Protocol を新設して Usecase の境界を型で縛る**:
   - `port/run_submitter.py`: `RunSubmitterPort` (`create_run`), `ClassificationRunSubmitterPort` (`create_classification_run`)
   - `port/run_reader.py`: `RunReaderPort` (`get_run`)
   - Python `typing.Protocol` で定義し、`RunManager` が構造的部分型として満たす（明示的な継承は不要）。

3. **handlers は Usecase だけを注入する**:
   - `routers/runs.py` / `routers/classification_runs.py` を `Depends(get_submit_run_usecase_dep)` / `Depends(get_get_run_usecase_dep)` / `Depends(get_submit_classification_run_usecase_dep)` / `Depends(get_get_classification_run_usecase_dep)` に切替。`RunManager` 型への参照は該当 2 ファイルから消える。

4. **Container でのワイヤリング**:
   - `ServiceContainer` に新規プロパティ `submit_run_usecase` / `get_run_usecase` / `submit_classification_run_usecase` / `get_classification_run_usecase` を追加。各 Usecase は lazy init で構築し、Port 実装として `self.run_manager` を注入する。`RunManager` 本体は引き続き `container.run_manager` で参照可能（Phase 3 以降で段階的に解体）。

5. **`RunManager` 本体には手を入れない**: run_manager.py は `create_run` / `get_run` / `create_classification_run` 等のメソッドシグネチャが既に Port Protocol と一致しているため、構造的部分型で Port を自動的に満たす。内部ロジックの移設は Phase 3 以降の責務。

6. **テスト方針**:
   - 新規 `tests/unit/usecase/test_submit_run.py` / `test_classify_run.py` は Port の構造的部分型を満たす軽量な `_Fake*` で Usecase の委譲と例外伝播を検証する。`@pytest.mark.asyncio` + await で非同期実行。
   - `tests/conftest.py` の autouse override は新 Usecase dep も対象に追加し、handler 単体テストが heavy container property に到達しない状態を維持。
   - `tests/unit/test_api_runs.py` は `app.dependency_overrides` の宛先を `get_submit_run_usecase_dep` / `get_get_run_usecase_dep` に書き換え、`FakeRunManager` を `SubmitRunUsecase(submitter=manager)` にラップして注入。

## Consequences

### Pros

- Handler 層が `RunManager` 具象に依存しなくなり、Clean Architecture の層構造が明確化された。future の gateway 置き換え（例: Phase 3 で `services/run_manager.py` を分解して Port ごとに別クラスに分ける）がコンパイル時型チェック付きで段階的に進められる。
- 単体テストは「Submit したら create_run が呼ばれる」「Get は None で 404」といった単一責務のみを検証可能になった。既存テストで `FakeRunManager` のクラス全体を作る負担が減る。
- `typing.Protocol` を使った構造的部分型により、`RunManager` 本体へのコード変更なしで Port を満たすことができ、Phase 2 のリスク（Pact 破壊）を最小化。
- 新規 Usecase は `__init__` で Port を受け取る純粋クラスのため、spawn ProcessPoolExecutor / ML 初期化のサイドエフェクトを持たない。Phase 1 unblock の原則（テストから heavy container を隔離）を破らず拡張できた。
- `tests/conftest.py` の autouse override が新 Usecase にも自動適用されるため、handler テストは引き続き 44s の baseline を維持（回帰なし）。

### Cons / Tradeoffs

- `container.submit_run_usecase` と `container.run_manager` の二重経路が残る。管理 API (`/admin/*`) 等の既存ルートが `get_run_manager_dep` を参照し続けている箇所があるため、完全除去は Phase 3 以降に繰越。移行期間中は「新経路を優先、既存経路は legacy」という口伝ルールでカバーする。
- Port Protocol の `...create_run(submission: RunSubmission) -> RunRecord` シグネチャが具象型（`RunSubmission` dataclass, `RunRecord` dataclass）を直接参照しており、port と domain の境界がまだ完全には独立していない。Phase 4 の Pydantic VO 導入時に domain 型を挟む形で段階的に抽象化する。
- `RunManager` が構造的部分型で Port を満たしている点は便利だが、ドキュメント上の追跡性は落ちる（`RunSubmitterPort` を実装しているクラスを grep しても `RunManager` は引っかからない）。Phase 3 で明示的な ABC/Protocol 実装宣言を検討する。
- Pyrefly は `FakeRunManager` のような test 用 duck type を厳密にチェックするため、テスト側にも `-> RunRecord` 等の型注釈追加を強制された。これはフェーズとして許容するが、新規テストを書く際のコストが僅かに上がる。
