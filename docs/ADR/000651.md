# ADR-000651: 3日間 Recap の LLM バイパス閾値緩和・フォールバック品質改善・プロンプト強化

## Status

Accepted

## Date

2026-04-09

## Affected Services

- news-creator (Python) — LLM バイパス閾値、フォールバックテンプレート、プロンプト改善
- recap-subworker (Python) — c-TF-IDF ストップワード拡張

## Context

直近の 3日間 Recap ジョブ (2026-04-08) を DB 分析した結果、**36ジャンル中30がdegraded（83%）** であり、LLM による要約がほとんど使われていないことが判明した。

### 根本原因

1. **LLM バイパス閾値が厳しすぎる**: `_should_bypass_llm()` が 3日間リクエストに対して `representative_sentence_count < 4` または `distinct_sources < 3` でバイパスしていた。3日間ウィンドウでは各ジャンルの代表文が 4〜7 程度しかなく、大半がこの閾値を下回っていた。25ジャンルが `low_evidence_extractive` としてバイパスされ、LLM は一切呼ばれなかった。

2. **degraded フォールバックの品質が極めて低い**: バイパス時の出力は固定テンプレート `"{topic} に関する更新が確認された。代表文では「{snippet}」が中心情報として示されており、直近3日間の重要トピックとして整理できる"` で、以下の問題があった:
   - `topic_label` が c-TF-IDF の `top_terms[:2]` から生成されるため "url url・url", "systems・events" 等のゴミラベル
   - 代表文が英語フラグメント（"Arguments have gotten heated." 等）
   - snippet は 120文字で切断
   - タイトルが英語スラッグ直訳（"software devの直近更新"）

3. **LLM 成功分にもソース不一致・フィラー文の問題**: "これは重要な進展である" 等の主観的評価、異なるクラスタの出典を混合した参照

4. **c-TF-IDF のトピックラベルが Web/コード用語で汚染**: "url", "http", "self", "data" 等のストップワード不足

## Decision

4段階の改善を実施する。

### 1. LLM バイパス閾値の緩和

`llm_config.py` のデフォルト値を変更する:
- `RECAP_MIN_REPRESENTATIVE_SENTENCES_FOR_LLM`: 4 → **2**
- `RECAP_MIN_SOURCE_ARTICLES_FOR_LLM`: 3 → **1**

dataclass のデフォルト値と `from_env()` factory の両方を統一する。代表文が 2文以上あれば LLM に渡す。1ソースでも要約は可能であり、テンプレートよりはるかに良い出力を生成できる。`is_degraded` フラグは引き続きメタデータに記録されるため品質追跡は維持される。

### 2. degraded フォールバックテンプレートの改善

`_format_degraded_bullet()` の機械的テンプレートを廃止し、以下に変更する:
- **日本語文**: そのまま bullet として使用（テンプレート包みなし）
- **英語文**: `【ジャンル名】` プレフィックスを付与
- `_looks_japanese()` で日本語率 15% 以上を判定
- `GENRE_JA_MAP` を追加（28ジャンル分）し、タイトルとフォールバック文の日本語化に使用
- `_collect_fallback_candidates_from_clusters()` に品質フィルタを追加: 20文字未満の短文除外、コードスニペット（バッククォート率 30% 超）除外、日本語文を優先

### 3. プロンプトテンプレートの改善

`recap_summary_3days.jinja` と `recap_summary.jinja` に以下を追加:
- **禁止事項セクション**: 入力外情報の追加禁止、異なるクラスタ情報の混合禁止、主観的評価の禁止、出典の不適切な結合禁止
- **イベント単位の抽出指示**: Chain-of-Event (CoE) に基づき、「主体 + 行動 + 結果」の構造でイベントを洗い出す手順を明示
- **ソース分離ルール**: 各 bullet の出典はその内容と直接関連するものだけを使用

### 4. c-TF-IDF ストップワード拡張

`recap-subworker/domain/stopwords.py` に `_WEB_TECH_STOPWORDS`（55語）を追加:
- URL コンポーネント: url, http, https, www, com, org 等
- プログラミングキーワード: self, class, def, return, import, function 等
- リテラル: true, false, null, none, undefined

### 検討した代替案

- **LLM バイパスの完全廃止**: 代表文 1文・ソース 0件の場合はハルシネーションリスクが高いため、最低限の閾値は維持する方針を選択
- **Chain of Density (CoD)**: 複数回の LLM 反復呼出しが必要で、RTX 4060 8GB 上の gemma4-e4b-q4km では遅すぎる。単一パスの CoE を選択
- **vLLM guided decoding**: Ollama バックエンドのため利用不可。既存の JSON schema 指定 (`format` パラメータ) で代替

## Consequences

### Pros

- **`low_evidence_extractive` バイパスが 25 → 0 に完全排除**: 全ジャンルで LLM が呼ばれるようになった
- **LLM 成功率が 6/36 (17%) → 17/35 (49%) に約3倍向上**
- **旧機械テンプレートが完全消滅**: 「に関する更新が確認された。代表文では...」パターンはゼロになった
- **degraded タイトルが全て日本語化**: "software devの直近更新" → "ソフトウェア開発の直近更新"
- **c-TF-IDF トピックラベルから Web ノイズ語が排除**: "url url" や "self data" がトップタームに出なくなった

### Cons / Tradeoffs

- **`llm_failed_after_repair` が 5 → 18 に増加**: LLM に渡されるジャンルが増えた分、JSON バリデーション失敗も増えた。Gemma 4 の structured output 信頼性改善（リトライ戦略強化、Repair プロンプト改善）が次ステップ。この 18件のフォールバック出力には英語フラグメントが多く残存する（"Arguments have gotten heated.", "Framer Motion makes animations easy." 等）。根本原因は上流の代表文選択が英語 RSS 記事の断片を返すことにあり、recap-subworker 側での言語フィルタリング強化が必要
- **GPU 処理時間の増加**: LLM 呼出しが 6 → 35 ジャンルに増加し、バッチ要約のGPU時間が約 6倍に増加。ただしバッチ API の並行数制限（`min(3, concurrency)`）により、パイプライン全体のウォールタイムへの影響は許容範囲
- **ストップワード追加による偽陰性リスク**: "data science" 等のフレーズ内の "data" が除外される可能性。現行は unigram/bigram 単位での判定のため、bigram "data science" は保持される
