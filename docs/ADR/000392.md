# ADR-000392: Swipeモード下部ボタンの視覚的肥大化を修正しモバイルブラウザUIバーとの重なりを解消する

## Status

Accepted

## Date

2026-03-16

## Affected Services

- alt-frontend-sv (SvelteKit / TypeScript)

## Context

[[000389]] で SwipeFeedCard のボタンに `min-h-[44px]` を追加し、WCAG 2.5.5 Level AAA のタッチターゲット 44px を保証した。しかし、この変更により以下の問題が発生した:

1. **ボタンの視覚的肥大化**: `size="sm"` の Button コンポーネントは `h-8`（32px）を持つが、`min-h-[44px]` がこれを上書きし、ボタンが不自然に縦長になった。Swipe モードのコンパクトな footer デザインと不調和になっていた。

2. **モバイルブラウザ下部 UI バーとの重なり**: iOS Safari のホームインジケーター（iPhone X 以降）や Android Chrome の下部ナビゲーションバーが、カード footer のボタンと重なる可能性があった。`env(safe-area-inset-bottom)` を CSS で記述していたが、`viewport-fit=cover` が viewport meta に設定されていないため、`env()` は常に `0px` を返し実質無効であった。

## Decision

**`min-h-[44px]` を削除し、`viewport-fit=cover` + `env(safe-area-inset-bottom)` による safe area 対応を導入する。**

### タッチターゲットの再評価

`min-h-[44px]` を削除してもWCAG準拠は維持される:

- ボタン高さ: `size="sm"` = `h-8` = 32px
- footer padding: `py-3` = 12px × 2（上下）
- 実効タッチターゲット: 32px + 12px + 12px = **56px > 44px**

ボタン間の `gap-2`（8px）も含めると、ボタン周辺のタップ領域は十分に確保されている。WCAG 2.5.5 のタッチターゲットサイズ要件はボタン要素単体ではなく、隣接する要素との間隔を含む実効的なタップ領域で判定される。

### 検討した代替案

1. **`min-h-[44px]` を維持し padding で調整**: ボタン内部の padding を減らして見た目を改善する案。却下理由 — `button-variants.ts` の `sm` サイズ定義を変更すると、Swipe 以外のすべての `sm` ボタンに影響が波及する。
2. **Swipe 専用の Button variant を追加**: 却下理由 — 過剰な抽象化。footer padding で解決できる問題に新しい variant は不要。
3. **`viewport-fit=cover` なしで `padding-bottom` を固定値で追加**: 却下理由 — デバイスごとに safe area の高さが異なるため、固定値では対応できない。

## Implementation

### 1. `app.html` — viewport meta の変更

```html
<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover" />
```

`viewport-fit=cover` を追加することで、ページコンテンツが画面全体（ノッチ・ホームインジケーター領域を含む）に拡張される。これにより `env(safe-area-inset-bottom)` が実際のデバイス safe area の値を返すようになる。

デスクトップブラウザでは `env(safe-area-inset-bottom)` は `0px` にフォールバックするため、既存の表示に影響はない。

### 2. `SwipeFeedCard.svelte` — ボタンとfooterの修正

- 3つのボタン（Article, Favorite, Summary）から `min-h-[44px]` を削除
- footer の `py-3` を `pt-3 pb-[calc(0.75rem+env(safe-area-inset-bottom,0px))]` に変更

### 3. `VisualPreviewCard.svelte` — footerのsafe area対応

- footer の `py-3` を `pt-3 pb-[calc(0.75rem+env(safe-area-inset-bottom,0px))]` に変更
- VisualPreviewCard のボタンには元々 `min-h-[44px]` がなかったため、ボタン側の変更は不要

### safe area padding の仕組み

```
pb-[calc(0.75rem + env(safe-area-inset-bottom, 0px))]
      ↑ 元の py-3     ↑ デバイスの safe area（0px フォールバック）
```

- iPhone X+（Safari）: `0.75rem + 34px` = 約 46px の下部パディング
- Android Chrome（ジェスチャーナビ）: `0.75rem + ~48px`
- デスクトップ / safe area なし: `0.75rem + 0px` = 12px（変更前と同一）

## Consequences

### Pros

- **視覚的改善**: ボタンが `h-8`（32px）に戻り、Swipe カードの footer がコンパクトになった
- **WCAG 準拠維持**: 実効タッチターゲット 56px > 44px。padding を含む領域で WCAG 2.5.5 を満たす
- **モバイルブラウザ対応**: iOS Safari のホームインジケーター、Android Chrome のナビバーとの重なりが解消された
- **既存ページへの正の影響**: `viewport-fit=cover` の追加により、他のコンポーネント（FloatingMenu, SwipeFilterSortSheet 等）で既に記述されていた `env(safe-area-inset-bottom)` が正しく機能するようになった
- **デスクトップ無影響**: `env()` の `0px` フォールバックにより、デスクトップでの表示は変更前と同一

### Cons / Tradeoffs

- **`viewport-fit=cover` のグローバル影響**: この設定はアプリ全体に適用される。safe area を考慮していないページ（固定フッター等）で要素が safe area に侵入する可能性がある。ただし、Alt の既存コンポーネントは既に `env(safe-area-inset-*)` を使用しているため、むしろ正しく動作するようになる
- **`calc()` + `env()` の CSS 複雑性**: Tailwind のユーティリティクラスとしてはやや長い arbitrary value になる。ただし、この記法は CSS 標準であり、すべてのモダンブラウザでサポートされている
