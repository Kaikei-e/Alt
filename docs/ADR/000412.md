# ADR-000412: SSR HTML レスポンスに Cache-Control no-cache を追加し再ビルド後の旧チャンク 404 を防止する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- nginx — リバースプロキシ設定（`nginx/conf.d/default.conf`）

## Context

フロントエンドコンテナ（alt-frontend-sv）の再ビルド後、ブラウザコンソールに旧ビルドの JS チャンクに対する 404 エラーが大量発生していた。

SvelteKit は `_app/immutable/chunks/*.js` をコンテンツハッシュ付きファイル名で出力する。再ビルドするとソースコードの変更に応じて新しいハッシュのファイルが生成され、旧ハッシュのファイルは新コンテナに存在しなくなる。

nginx の設定には以下の 2 つのキャッシュポリシーが存在した:

| location | Cache-Control | 評価 |
|----------|--------------|------|
| `/_app/immutable/` | `public, immutable` + `expires 1y` | 正しい。ハッシュ付きファイルは永久キャッシュが適切 |
| `/`（SSR HTML catch-all） | **なし** | 問題。HTML がブラウザにキャッシュされ、旧チャンクハッシュへの参照が残る |

SSR HTML にキャッシュ制御がないため、ブラウザは HTML レスポンスをキャッシュし、再ビルド後もキャッシュ済み HTML が旧チャンクファイル名（`BKtksDl_`, `BhDaH78h` 等）を `<script>` タグや `Link` ヘッダーで参照し続け、新コンテナに存在しない旧ファイルを取得しようとして 404 になっていた。

## Decision

**nginx の SSR HTML レスポンス（`location /`）に `Cache-Control: no-cache` を追加する。**

### no-cache の意味

`no-cache` は「キャッシュするな」ではなく「キャッシュしてよいが、使う前にサーバーに再検証（revalidation）を問い合わせろ」という指示である。これにより:

- 初回アクセスは通常通り HTML を取得
- 2 回目以降、ブラウザは `If-None-Match`（ETag）でサーバーに問い合わせ
- HTML が変わっていなければ 304 で高速レスポンス
- 再ビルド後は新しい HTML（新チャンクハッシュ参照）を取得 → 404 が発生しない

### 検討した代替案

1. **`no-store`（キャッシュ完全禁止）**: 毎回フルレスポンスを返すため帯域効率が悪い。HTML の ETag による条件付きリクエスト（304）を活用できない。却下。
2. **`max-age=60` 等の短い TTL**: 一定時間は旧 HTML が使われる可能性が残る。`no-cache` のほうが確実かつ 304 で効率も維持できる。却下。
3. **SvelteKit 側で `setHeaders` を使う案**: アプリケーション層でのヘッダー設定も可能だが、nginx がリバースプロキシとしてキャッシュポリシーを一元管理するほうが運用上わかりやすい。却下。

### 実装

`nginx/conf.d/default.conf` の `location /` ブロックに 1 行追加:

```nginx
location / {
    add_header Cache-Control "no-cache" always;
    # ... 既存の proxy_pass 等は変更なし
}
```

`always` パラメータにより、すべてのレスポンスステータスコードでヘッダーが付与される。

## Consequences

### Pros

- **再ビルド後の 404 が解消**: ブラウザは必ず最新の HTML を取得し、正しいチャンクハッシュを参照する
- **パフォーマンスへの影響が最小限**: `no-cache` は ETag ベースの条件付きリクエストを許容するため、HTML が変わっていなければ 304（ボディなし）で返る
- **immutable アセットへの影響なし**: `/_app/immutable/` の `public, immutable` ポリシーは優先度の高い location マッチにより独立して適用される
- **変更が 1 行のみ**: nginx 設定の 1 行追加で完結し、アプリケーションコードへの変更が不要

### Cons / Tradeoffs

- **すべてのページ遷移で再検証リクエストが発生**: SPA 内遷移ではクライアントサイドルーティングが使われるため実質的な影響は初回ロード時のみだが、ブラウザの「戻る」ボタン等でキャッシュから即時表示されていたケースでは再検証の RTT が追加される
- **nginx の `add_header` 継承ルールへの注意**: nginx では `location` ブロック内に `add_header` があると `server` ブロックの `add_header` が継承されなくなる。現状 `location /` には他に `add_header` がなかったため、この 1 行の追加でセキュリティヘッダー（`X-Frame-Options` 等）の継承が変わる可能性がある。検証の結果、`server` レベルのセキュリティヘッダーは正常に出力されていることを確認済み
