# ADR-000417: Knowledge Home projection の JSONB 書き込みを PgBouncer simple protocol 互換にする

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — Knowledge Home projection repositories, Today Digest projection, Recall Rail projection

## Context

Knowledge Home の backfill は完了していたが、UI は `Your knowledge is warming up` の empty state のままだった。調査の結果、`knowledge_events` は蓄積されている一方で `knowledge_home_items` が 0 件で、projector が `UpsertKnowledgeHomeItem: ERROR: invalid input syntax for type json (SQLSTATE 22P02)` を繰り返していた。

alt-backend は `pgbouncer` に対して transaction pooling で接続し、pgx は `QueryExecModeSimpleProtocol` を使っている。この前提では JSONB カラムに raw `[]byte` を渡すと `bytea` 扱いになり、PostgreSQL 側で JSON として解釈できず失敗する。リポジトリ内の `scraping_domain_driver` にも同じ回避知見が既に存在していた。

## Decision

**Knowledge Home 系 projection repository の JSONB 引数は raw `[]byte` ではなく JSON text として渡し、PgBouncer transaction pooling + pgx simple protocol でも安定して書き込めるようにする。**

### 変更内容

1. `knowledge_home_repository.go` の `tags_json` / `why_json` を `string(json.Marshal(...))` で渡すようにした。
2. `today_digest_repository.go` の `top_tags_json` / `pulse_refs_json` を同様に JSON text 化した。
3. `recall_candidate_repository.go` の `reason_json` も同じ方針に揃えた。
4. 各 repository に PgBouncer 互換の回帰テストを追加し、mock 期待値で JSON text が渡されることを固定した。

### 設計判断

1. **接続方式は変えず、引数エンコードを直す** — PgBouncer や pgx の運用方針を incident 対応で変えず、障害点だけを狙って修正する。
2. **Knowledge Home 周辺の JSONB writer をまとめて揃える** — `knowledge_home_items` だけ直しても、today digest や recall rail で再発するため、同じ投影境界を一括で整える。
3. **既存の repo 内知見を再利用する** — `scraping_domain_driver` と同じ text 渡し方を採用し、別の JSONB エンコード戦略を増やさない。

## Consequences

### Pros

- Knowledge Home projector が PgBouncer 配下でも JSONB 書き込みに失敗しなくなる
- empty Home の直接原因を backend 側で除去できる
- Today Digest と Recall Rail でも同種の JSONB 事故を予防できる

### Cons / Tradeoffs

- repository 層に「PgBouncer/simple protocol を意識した text 化」が残る
- 将来 pgx/driver 方針を変えるときは、この前提を見直す必要がある
