# ADR-000465: 非要約イベントで ready の summary_state を保護する

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-backend (Go)

## Context

[[000424]] で `summary_state` は Knowledge Home の backend truth として導入され、`missing / pending / ready` を UI がそのまま信頼する前提を取った。ところが実運用では、要約本文が既に作成され `summary_excerpt` が存在する item でも、後続の `TagSetVersionCreated` など summary を直接更新しないイベントの後に `summary_state` が空文字へ戻る事例が発生した。

原因は `knowledge_home_items` の merge-safe upsert にある。projector 側ではタグ更新イベントで `SummaryState` を設定しないため、Go のゼロ値である空文字が `EXCLUDED.summary_state` に入る。既存 SQL は `EXCLUDED.summary_state != 'missing'` を条件に incoming 値を優先していたため、空文字でも既存の `ready` を上書きしてしまっていた。

このドリフトは UI 上では「`Summarized` バッジが残るのに本文は skeleton」という矛盾した表示として観測される。frontend の表示ロジックではなく、read model merge 契約の欠陥が原因である。

また、この問題は一時的な SQL 補正だけでは再発を防げない。Knowledge Home は event ごとの部分更新を許す read model であり、summary を更新しないイベントが来ても summary の到達状態を壊さない保存規則を repository に持たせる必要がある。

## Decision

**`summary_state` の upsert は、incoming 値が空文字または `missing` の場合に既存値を保持し、`ready` を非要約イベントで後退させない。**

具体的には repository の SQL を以下の規則に固定する。

- `EXCLUDED.summary_state = 'ready'` のときは必ず `ready` を採用する
- `EXCLUDED.summary_state` が `''` または `'missing'` のときは既存 `knowledge_home_items.summary_state` を保持する
- それ以外の明示的な状態のみ incoming 値を採用する

修正後の merge 条件は次の形とする。

```sql
summary_state = CASE
  WHEN EXCLUDED.summary_state = 'ready' THEN 'ready'
  WHEN EXCLUDED.summary_state NOT IN ('', 'missing') THEN EXCLUDED.summary_state
  ELSE knowledge_home_items.summary_state
END
```

### 検討した代替案

1. **Tag 系 projector でも常に現在の `summary_state` を再送する**
   projector が他イベントの派生状態まで知る前提になり、event ごとの関心分離を崩すため採用しない。
2. **不整合 item を SQL で都度修復する**
   既存データの一時是正には使えても、保存契約が壊れたままなので再発を防げないため採用しない。
3. **frontend で `summary_excerpt` を優先して状態を推測する**
   [[000424]] の backend truth 原則に反し、UI 側で contract drift を吸収する設計になるため採用しない。

## Consequences

### Pros

- `TagSetVersionCreated` など summary 非更新イベントが後から来ても `ready` が保持される
- `summary_state` と `summary_excerpt` の整合が崩れにくくなり、UI の矛盾表示を防げる
- projector ではなく repository の責務として保存規則を固定できる
- reproject 時も同じ SQL 契約で再構築されるため、再現性が高い

### Cons / Tradeoffs

- repository の upsert SQL がさらに防御的になり、単純な上書きより読み解きにくい
- 将来 `failed` など新しい状態を増やす場合は、この CASE 条件の見直しが必要になる
- 既に壊れている行は別途 reproject などで再構築しない限り自動修復されない
