# ADR-000690: 抽出ループをincremental checkpoint-safeにし最終synthesisをWriterに集約する

## Status

Accepted

## Date

2026-04-11

## Affected Services

- acolyte-orchestrator (Python / LangGraph)

## Context

`exec3.md` の受け入れ基準に照らすと、Acolyte の v5 系実装には 3 つのズレが残っていた。

1. **checkpoint 粒度が不足していた**  
   [[000679]] で Postgres checkpointer と resume 導線は入っていたが、QuoteSelector / FactNormalizer は node 内で複数 article / quote をまとめて処理していた。そのため、n 件処理したあとに node 途中で失敗すると、処理済み分の `selected_quotes` / `extracted_facts` が失われていた。

2. **FactNormalizer の tiny schema が完了していなかった**  
   [[000677]] は tiny schema 化を掲げていたが、`FactNormalizerOutput` には `reasoning` が残り、runtime fact も `source_title` / `verbatim_quote` を含んでいた。これでは downstream で扱う shape が大きく、`exec3` の `claim / source_id / quote / data_type / confidence` という最小契約と一致しなかった。

3. **ES / Conclusion の最終根拠が provisional claim_plans のままだった**  
   [[000664]] と [[000681]] で accepted claims ベース synthesis の方向は定義済みだったが、実際の graph では SectionPlanner が writer より先に動くため、ES / Conclusion は最終的にも paragraph acceptance 前の `claim_plans` に強く依存していた。実際に accept された paragraph claim を最優先に使う責務が明確でなかった。

加えて、revision loop (`critic -> writer`) では critic が同一 payload を返し続けるケースがあり、LangGraph の条件分岐評価に「単調に進む state」が不足すると、実運用とテストの両方で進行保証が弱い状態だった。

## Decision

以下を採用する。

### 1. QuoteSelector / FactNormalizer は checkpointer 有効時に incremental self-loop で動かす

`checkpointer is not None` の graph では、QuoteSelector / FactNormalizer を batch node としてではなく、1 invocation で 1 article / 1 quote ずつ進める incremental mode に切り替える。

- `quote_selector_work_items`, `quote_selector_cursor` を state に保持する
- `fact_normalizer_work_quotes`, `fact_normalizer_cursor` を state に保持する
- conditional edge で `"more"` の間は self-loop し、完了時のみ次 node へ進む

これにより、resume 時は未処理分だけを継続し、処理済み article / quote の state を保持する。

### 2. FactNormalizer の runtime 契約を tiny schema に固定する

FactNormalizer の LLM 出力および runtime fact を次の shape に寄せる。

- `claim`
- `source_id`
- `quote`
- `confidence`
- `data_type`
- `is_fallback`

`FactNormalizerOutput` から `reasoning` を削除し、runtime fact からも `source_title` / `verbatim_quote` を落とす。  
移行期間の downstream は `quote` を優先しつつ legacy `verbatim_quote` も読めるようにするが、新規の正規契約は `quote` とする。

### 3. ES / Conclusion の最終 synthesis は Writer が引き受ける

SectionPlanner の役割は provisional な `claim_plans` と deterministic fallback の供給にとどめる。  
最終的な ES / Conclusion の synthesis claim は Writer が paragraph acceptance 後の state から再構成する。

- Conclusion は accepted な analysis claims から synthesis claims を再生成する
- Executive Summary は accepted な非 ES section claims から synthesis claims を再生成する
- 生成した synthesis claims で `claim_plans` を上書きし、そのまま paragraph generation と citation assembly に使う
- `accepted_claims_by_section` を state に保持し、debug / resume 時の観測性を確保する

これにより、最終出力は「planner が考えた claim」ではなく、「実際に受理された paragraph claim」に基づく。

### 4. revision loop と最終永続化は単調進行を優先する

revision loop の進行保証と最終本文の保全のため、次を採用する。

- `critic_revision_no` を state に追加し、critique payload が同一でも branch 評価に単調差分を持たせる
- graph の conditional route は async wrapper 経由で評価する
- Finalizer は最新本文が空でなくても `best_sections` を優先して永続化する

これにより、repeat critique 時の loop 停滞を避けつつ、best revision を失わない。

## Consequences

### Pros

- QuoteSelector / FactNormalizer は n 件処理後に落ちても、n 件分の state を保持して resume できる
- FactNormalizer の出力契約が小さくなり、fallback fact も success fact も同じ shape で downstream が読める
- ES / Conclusion が provisional claim_plans ではなく accepted claims を優先して組み立てられる
- Finalizer が best revision を常に優先するため、途中 revision で良い本文を失いにくい
- revision loop が identical critique でも前進できる

### Cons / Tradeoffs

- state に work queue / cursor / accepted-claims debug 情報が増え、checkpoint payload はやや大きくなる
- Writer の責務が広がり、paragraph generation に加えて synthesis claim の再構成も担う
- SectionPlanner / downstream の一部に legacy `verbatim_quote` 読み取り互換を残すため、完全移行まで二重読みによる複雑さがある
- Writer 自体は依然として mid-node checkpoint-safe ではなく、section / paragraph 途中失敗時は node 先頭からの replay になる
