# 画像プロキシ フォールバック改善 — 透過PNG廃止・CSSグラデーション活用・Nginxエラーキャッシュ

## ADR's STATUS

Accepted (supersedes ADR 000339 のフォールバック部分)

## CONTEXT

ADR 000339 で画像プロキシのエラー時に1x1透過PNGを200 OKで返すフォールバックを導入した。
しかし運用中に以下の問題が判明した。

### 問題

透過PNGは `<img>` タグの正常レスポンス（200 OK + 有効な画像バイナリ）として扱われるため、
ブラウザの `onerror` イベントが発火しない。その結果:

- フロントエンド (`VisualFeedCard.svelte`) の `imageError = true` → `showImage = false` パスが実行されない
- カード上で画像エリアが「白抜き」になり、既存のCSSグラデーション（`fallback-gradient`）が表示されない

フロントエンドにはすでにきれいなCSSグラデーション背景が実装されており、
`onerror` さえ発火すれば自動的に表示される仕組みが整っていた。

### 根本原因

バックエンドが画像取得失敗を200 OKで隠蔽したことで、フロントエンドのエラーハンドリング機構が無効化された。
「ブロークン画像アイコンを防ぐ」という意図は正しかったが、解決手段がフロントエンドの既存機構と衝突していた。

## DECISION MAKING

バックエンドのフォールバックを「200 + 透過PNG」から「502/504 NoContent」に変更し、
フロントエンドのCSS `fallback-gradient` を活用する。Nginx側で502/504を短時間キャッシュして
バックエンドへの連打を防止する。

### 1. バックエンド: エラーレスポンスの正常化

`transparentPixelPNG` 変数とフォールバック分岐を完全削除し、エラー種別に応じた適切なHTTPステータスを返す。

| エラー種別 | ステータス | レスポンス |
|---|---|---|
| `context.DeadlineExceeded` | 504 Gateway Timeout | NoContent（ボディなし） |
| `context.Canceled` | 499 Client Closed Request | NoContent（ボディなし） |
| その他のフェッチ/処理エラー | 502 Bad Gateway | NoContent（ボディなし） |
| シグネチャ不正 / ドメイン拒否 | 403 Forbidden | エラーメッセージ（変更なし） |

`c.String()` から `c.NoContent()` に変更。`<img>` タグにテキストボディは不要であり、ステータスコードのみで十分。

ログレベルはすべてWARNに統一。画像取得失敗はシステム障害ではなく外部依存の一時的エラーであるため。

### 2. Nginx: エラーレスポンスの短時間キャッシュ

```nginx
proxy_cache_valid 502 504 1m;
```

既存の `proxy_cache_use_stale http_502 ... http_504` との連携:

1. **staleな200キャッシュあり** → そちらを返す（本物の画像が表示される）
2. **staleキャッシュなし** → 502/504をNginxが1分間キャッシュ → 同一URLへの連打を防止
3. クライアントが502/504を受信 → `onerror` 発火 → CSSグラデーション表示

### 3. フロントエンド: 変更なし

`VisualFeedCard.svelte` の既存ロジックがそのまま機能する:

```
<img onerror> → imageError = true → showImage = false → fallback-gradient 表示
```

## RESULTS, EFFECTS

### 改善後のフロー

```
Client → Nginx
    ├─ HIT (200キャッシュ): 本物の画像を返却
    ├─ STALE (200キャッシュ期限切れ): 古い画像を返却 + バックグラウンド更新
    ├─ HIT (502/504キャッシュ, 1分以内): キャッシュされたエラーを返却 → onerror → CSSグラデーション
    └─ MISS:
        → alt-backend → 外部フェッチ
            ├─ 成功: 200 + 画像バイナリ
            ├─ タイムアウト: 504 NoContent → Nginx 1分キャッシュ → onerror → CSSグラデーション
            └─ エラー: 502 NoContent → Nginx 1分キャッシュ → onerror → CSSグラデーション
```

### PROS

- **白抜き問題の解消**: フロントエンドの `onerror` が正常に発火し、CSSグラデーションが表示される
- **既存UIの活用**: 新規UI実装不要。フロントエンドに既にある `fallback-gradient` をそのまま利用
- **適切なHTTPセマンティクス**: 画像取得失敗を200で隠蔽せず、502/504で正直に報告
- **連打防止**: Nginxが502/504を1分キャッシュするため、同一URLへのリトライがバックエンドに到達しない
- **staleキャッシュとの両立**: 以前取得に成功した画像のstaleキャッシュがあれば、エラー時もそちらが優先される

### CONS, TRADEOFF

- **ブロークン画像アイコンのリスク**: `onerror` ハンドラが設定されていない `<img>` タグでは、ブラウザのデフォルトのブロークン画像アイコンが表示される。現在の `VisualFeedCard.svelte` と `VisualPreviewCard.svelte` は `onerror` を実装済みだが、将来新しいコンポーネントで画像プロキシURLを使う場合は同様のハンドリングが必要
- **1分間のエラーキャッシュ**: 外部画像が復旧しても最大1分間はエラーレスポンスがキャッシュされる。OG画像の用途では十分短い間隔
