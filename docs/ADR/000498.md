# ADR-000498: Ask Augur をルーティング型 Agentic RAG に進化させる

## Status

Accepted

## Date

2026-03-20

## Affected Services

- rag-orchestrator (Go)

## Context

Ask Augur はこれまで、どんな質問に対しても同一の 5-stage RAG パイプライン（query expansion → embedding → vector search → BM25 + RRF fusion → reranking）を実行していた。フロントエンド（[[000494]]）が `"Regarding the article: {title} [articleId: {id}]\n\nQuestion:\n{question}"` という決定的なフォーマットで articleId を送信しているにもかかわらず、rag-orchestrator はこのメタデータをパースせず全文を検索クエリとして使用していた。

この結果、特定記事への質問でも無関係なチャンクが混入し的外れな回答が生成されるという問題が発生していた。さらに、不要な embedding・reranking 処理が走るため、article-scoped クエリでは 10-50 秒のレイテンシが無駄に発生していた。

加えて、`Execute()` と `Stream()` の両経路が `buildPrompt()` で構築済みの messages を無視し、`input.Query`（生クエリ）で `promptBuilder.Build()` を再度呼び出すという prompt 構築経路の重複が存在していた。

## Decision

**ルールベース・ルーティング + Strategy パターン**を採用し、クエリの意図を解析して最適な検索戦略を動的に選択する Agentic RAG アーキテクチャを実装する。

### 検討した代替案

| 代替案 | 評価 |
|--------|------|
| ローカル LLM によるツール呼び出し | 3-10 秒のオーバーヘッド + 不安定。フロントの出力フォーマットが決定的なため過剰 |
| フロントエンドで intent を分離して送信 | API 変更が必要。バックエンド側で解析すれば互換性を保てる |
| 既存パイプラインを article filter だけ追加 | query expansion + reranking が無駄に走り、パフォーマンス改善が限定的 |

### 実装の概要

#### 1. QueryIntent パーサー（`query_intent.go`）

ステップ分割パースで article-scoped クエリからメタデータを抽出する。regex 一本ではなく、prefix 確認 → separator 分割 → 後方検索の 3 段階で処理することで、タイトルにブラケットを含むケースにも対応する。

`ResolveQueryIntent()` は現在のクエリが article-scoped でない場合、会話履歴から直近の article-scoped ユーザーメッセージを走査し、マルチターンでも記事コンテキストを維持する。

#### 2. RetrievalStrategy インターフェース（Strategy パターン）

```
RetrievalStrategy
├── articleScopedStrategy  — DB 直引き（GetByArticleID → GetVersionByID → GetChunksByVersionID）
└── generalStrategy        — 既存の 5-stage RetrieveContextUsecase をラップ
```

- `articleScopedStrategy` は `CurrentVersionID`（`GetLatestVersion` ではない）を正として version を取得する。これにより reindex 中の不整合を防ぐ
- 将来の戦略追加（比較クエリ用の hybrid strategy 等）が OCP 準拠で可能

#### 3. Prompt 構築経路一本化

`buildPrompt()` が返す `promptData.messages` を `Execute()` と `Stream()` の両方で直接使用するよう統一。これにより intent.UserQuestion（メタデータ除去済み）が確実に LLM に渡る。

#### 4. 2 段フォールバック

article-scoped strategy が `ErrArticleNotIndexed` を返した場合:

1. **Stage 1**: `CandidateArticleIDs=[articleID]` を付与して general strategy を実行（article-constrained）
2. **Stage 2**: constrained も空の場合、制約なしの general strategy を実行（unrestricted fallback）

各段階で `strategyUsed` が更新され、debug 情報としてレスポンスに含まれる。

#### 5. PromptBuilder の記事コンテキスト対応

article-scoped 時に「記事コンテキスト」セクションを prompt に追加。token 制限でチャンクが切り詰められた場合は「主要な部分」、全チャンク含む場合は「全内容」と表現を分ける。

#### 6. Debug 伝播

`AnswerDebug.StrategyUsed` フィールドを追加し、REST レスポンス・SSE stream meta・Connect-RPC 経路すべてで使用された戦略名が確認可能。OpenAPI spec と生成コードも更新済み。

#### 7. Domain 層の拡張

`RagDocumentRepository` に `GetVersionByID(ctx, versionID)` を追加。`GetLatestVersion` は `ORDER BY version_number DESC LIMIT 1` だが、`GetVersionByID` は `WHERE id = $1` で主キー検索するため、CurrentVersionID との整合が保証される。

### テストカバレッジ

- QueryIntent パーサー: 10 テスト（一般クエリ、article-scoped、UUID、malformed、brackets in title、空 question、Question キーワード含有、会話履歴からの継承）
- ArticleScopedStrategy: 6 テスト（成功、未登録、CurrentVersion=nil、version mismatch、チャンク 0 件、ordinal 順序保持）
- 統合: DebugIncludesStrategy（Execute/Stream 両方）、NormalizedQuery 使用確認
- PromptBuilder: ArticleContext 3 テスト（全文、truncated、一般クエリで非混入）
- 既存テスト: 全テスト通過確認済み

## Consequences

### Pros

- **article-scoped クエリの高速化**: DB lookup 3 回（10-100ms）で完了し、embedding + vector search + reranking（10-50s）をバイパスする
- **回答精度の向上**: 対象記事のチャンクのみが LLM に渡るため、無関係チャンクの混入がなくなる
- **prompt 経路の一貫性**: Execute/Stream で同一の messages を使用するため、動作の予測可能性が向上
- **拡張性**: Strategy パターンにより、将来の戦略追加（比較クエリ、マルチ記事等）が既存コードを変更せずに可能
- **可観測性**: `strategy_used` フィールドにより、どの経路で処理されたかがログ・レスポンスから確認可能
- **堅牢なフォールバック**: 未インデックス記事でもユーザーへの回答を維持する 2 段フォールバック

### Cons / Tradeoffs

- **パーサーの脆弱性**: フロントエンドのフォーマットが変更された場合、パーサーの更新が必要。ただし IntentGeneral にフォールバックするため既存動作は壊れない
- **キャッシュキーの非互換**: 同一記事への質問でも生クエリが異なればキャッシュミスする。将来的に normalized query ベースのキャッシュキー生成を検討
- **article-scoped 時の query expansion 欠落**: DB 直引きのため ExpandedQueries が空になる。検索品質評価（Phase 2）で影響を測定する必要がある
