# ADR-000672: Acolyte パイプラインの LLM 責務を最小化し構造出力の安定性を向上させる

## Status

Accepted

## Date

2026-04-10

## Affected Services

- acolyte-orchestrator (Python)

## Context

Acolyte のレポート生成パイプラインで、Gemma4 26B の thinking トークンが `num_predict` バジェットを消費し、structured JSON 出力が途中切断される問題が継続していた。具体的には:

1. **Planner**: `PlannerOutput` schema が key/title/section_role/search_queries を含む `sections` 配列を要求しており、section 構造は既に `REPORT_TYPE_SKELETONS` で固定されているにもかかわらず LLM に冗長な出力を求めていた
2. **Extractor**: 1 記事から最大 3 facts を含む structured JSON を一発で抽出する設計。degradation ladder (full → light → quote-only) で緩和していたが、LLM 呼び出し成功時の出力サイズが大きく、thinking トークン消費で切断されるケースが残っていた
3. **Gateway**: structured output と free-text 生成が `format` パラメータの有無で暗黙的に切り替わっており、temperature / num_predict のデフォルト値が呼び出し元に散在していた

これらは exec2.md の Issue 1 (Planner 縮退), Issue 2 (Extractor 2 段階化), Issue 7 (Gateway 2-profile) に対応する Phase A / MVP slice として実施した。

## Decision

以下の 3 つの変更を採用する。

### 1. Planner を QueryExpansionOutput に縮退する

LLM の出力 schema を `PlannerOutput(reasoning, sections: list[PlannerSection])` から `QueryExpansionOutput(reasoning, queries: dict[str, list[str]])` に変更する。LLM は section_key → search queries のマッピングのみを返し、section 構造 (key, title, section_role) は `REPORT_TYPE_SKELETONS` からの deterministic 生成に完全移行する。

`reasoning` フィールドは debug 用 (`default=""`) とし、ビジネスロジックは依存しない。

### 2. Extractor を QuoteSelector + FactNormalizer に分割する

`ExtractorNode` を 2 つの新ノードに分割する:

- **QuoteSelectorNode**: 1 article x 1 section で 1-3 の verbatim quote を選択。section の search_queries を prompt に含め、section-conditioned recall を維持する。出力後にコードで exact substring 検証を行い `start_offset` / `end_offset` を設定する (見つからなければ -1)
- **FactNormalizerNode**: 1 quote → 1 fact の tiny schema で正規化。`num_predict=512` と極めて小さいバジェットで動作する

両ノードとも LLM 失敗時のフォールバックを持つ:
- QuoteSelector: sentence 分割で直接 quote を生成 (no LLM)
- FactNormalizer: quote text をそのまま claim に使用 (`confidence=0.3`)

グラフは `compressor → quote_selector → fact_normalizer → section_planner` に再配線。旧 `ExtractorNode` はグラフから除去したが、compressor テストが参照するためファイルは保持。

### 3. Gateway に LLMMode (structured/longform) を導入する

`LLMMode` enum (`STRUCTURED` / `LONGFORM`) を `llm_provider.py` に追加し、gateway の `generate()` に `mode` パラメータを導入する。

- `STRUCTURED`: `/api/chat`, `temperature=0`, `num_predict=1024` (settings から読み取り)
- `LONGFORM`: `/api/generate`, `temperature=0.7`, `num_predict=4000` (settings から読み取り)
- `mode=None`: 既存の format ベース routing を維持 (後方互換)

明示的な kwargs は mode defaults を上書きする。`num_predict` / `temperature` のデフォルト値は `settings.py` に集約し、モデル差し替え時に環境変数で調整可能にした。

endpoint 選択 (`/api/chat` vs `/api/generate`) は Ollama docs の要件ではなく実装判断である。

### 検討した代替案

- **QuoteSelector を per-article (section 横断)** にする案: LLM 呼び出し回数が減るが、section-conditioned recall を落とすリスクがある。exec2.md 原文の `1 article x 1 section` 仕様に従い不採用
- **FactNormalizer をバッチ処理** する案: 複数 quote をまとめて処理すると出力 schema が入力数に比例して大きくなり、元の問題 (JSON 切断) を再現するため不採用
- **`num_predict` / `temperature` defaults をハードコード** する案: モデル差し替え時の柔軟性に欠けるため `settings.py` に逃がす方式を採用

## Consequences

### Pros

- **Planner の JSON 失敗率が実質ゼロに**: LLM は `{"reasoning": "...", "queries": {"analysis": ["q1"]}}` という最小 dict のみを返す。失敗しても skeleton から同一構造が生成される
- **Extractor の facts=[] が事実上発生しない**: QuoteSelector fallback (sentence extraction) + FactNormalizer fallback (quote → claim) の二重フォールバックにより、LLM が完全に失敗しても `confidence=0.3` の quote-based facts が必ず生成される
- **Gateway の呼び出し意図が明示的に**: `mode=LLMMode.STRUCTURED` vs `mode=LLMMode.LONGFORM` で各ノードの意図が型レベルで表現される
- **テスト可能性の向上**: 各ノードが独立してテスト可能。QuoteSelectorNode (9 tests), FactNormalizerNode (6 tests), Gateway mode (6 tests), Planner expansion (7 tests) を追加

### Cons / Tradeoffs

- **LLM 呼び出し回数の増加**: FactNormalizer が per-quote で呼び出すため、最大 10 articles x 3 sections x 3 quotes = 90 calls。各 call は tiny (512 tokens) だが latency は加算される。将来的に `asyncio.gather` + semaphore で並列化可能
- **ExtractorNode の残存**: グラフからは除去されたが、compressor テストが integration test として参照しているためファイルを削除していない。長期的には compressor テストを QuoteSelectorNode ベースに移行すべき
- **Issue 3-8 が未実装**: Writer の micro-generation (Issue 3), ES/Conclusion synthesis (Issue 4), Critic heuristic-first (Issue 5), revision delta (Issue 8), checkpointer (Issue 6) は別フェーズ。特に Issue 2 の micro-call 化は LangGraph checkpointer (Issue 6) の replay 条件と相性が悪くなり得るため、Issue 6 実装時に node 粒度を再検討する必要がある
