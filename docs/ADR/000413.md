# ADR-000413: 記事保存経路から Knowledge Home 用 ArticleCreated イベントを生成する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — article persistence driver, knowledge event repository, internal article write path

## Context

Knowledge Home の feature flag を有効化した後でも、`GetKnowledgeHome` は空配列を返し続けていた。調査の結果、Knowledge Home は `knowledge_home_items` を直接読むが、その read model を埋める `knowledge-home-projector` が消費する `knowledge_events` に `ArticleCreated` が入っていないことが判明した。

既存実装では `SummaryVersionCreated` や `TagSetVersionCreated` は knowledge event store に append される一方、記事そのものの作成経路は以下の 2 系統で knowledge event store と切り離されていた。

1. `SaveArticle(...)` によるユーザー向け article 保存経路
2. `CreateArticleInternal(...)` による service-to-service の internal API 経路

この状態では、Knowledge Home の projector は新規記事の存在を検知できず、feature flag を有効化しても stream が空のままになる。MQHub 向けの `ArticleCreated` publish は存在したが、Knowledge Home projector が読むのは `knowledge_events` テーブルであり、経路が一致していなかった。

## Decision

**記事保存の両経路で、DB 書き込みと同じトランザクション内に Knowledge Home 用 `ArticleCreated` event を idempotent に append する。**

### 変更内容

1. `driver/alt_db` に `buildArticleCreatedKnowledgeEvent(...)` と `appendKnowledgeEventWithExec(...)` を追加し、pool/tx のどちらからでも `knowledge_events` に書ける共通経路を導入した。
2. `SaveArticle(...)` では article upsert 後、outbox event を積む前に `knowledge_events` へ `ArticleCreated` を append するよう変更した。
3. `CreateArticleInternal(...)` は単純な `QueryRow` から transaction ベースに変更し、article insert/update と `knowledge_events` append を同一 transaction に含めた。
4. dedupe は `article-created:<article_id>` を採用し、同一 article の再保存で projector が無制限に再投影しないようにした。

### 設計判断

1. **driver 層で append する** — `SaveArticle` と `CreateArticleInternal` は異なる呼び出し元を持つため、上位 usecase や handler で個別に event 生成すると経路漏れが起きやすい。article 永続化の直後に driver 層で event を積むことで、保存経路ごとの差異を吸収する。
2. **同一 transaction に含める** — article 行だけが保存され event が欠落する状態、または event だけが入り article が存在しない状態を避けるため、article write と knowledge event append を同一 commit に束ねる。
3. **MQHub event とは別責務で扱う** — 外部 consumer 向け publish と、Knowledge Home projector 向け event store は責務が異なる。両者を統合せず、read model の入力を `knowledge_events` に固定する。

### テスト

- `driver/alt_db/create_article_driver_test.go`
  - `CreateArticleInternal(...)` が article upsert 後に `knowledge_events` insert を行うことを確認
- `driver/alt_db/save_article_driver_test.go`
  - `SaveArticle(...)` が article upsert と outbox insert の間で `knowledge_events` insert を行うことを確認

## Consequences

### Pros

- **新規記事が Knowledge Home に流れる** — 新しい article 保存時点で projector が追従可能になる
- **経路漏れを抑制** — user-facing / internal API の両方で同じ event 生成が保証される
- **整合性向上** — article row と projector input event が同一 transaction で確定する

### Cons / Tradeoffs

- **driver 層の責務が少し増える** — persistence driver が knowledge event store 書き込みも担う
- **再保存時の扱いは dedupe 依存** — 現時点では article 内容更新時に別の `ArticleUpdated` を定義せず、初回作成相当のイベントだけを projector 入力として扱う
- **event schema が article 保存に結び付く** — 将来 event payload を変更する際は保存 driver 側も追従が必要
