# ADR-000464: DigestAvailabilityReconcileJob で Pulse/Recap 可用性を today_digest_view に接続する

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-backend (Go)

## Context

Knowledge Home の TodayBar には **Evening Pulse** と **Weekly Recap** へのショートカットボタンがあり、`today_digest_view` の `evening_pulse_available` / `weekly_recap_available` フラグで表示状態が決まる。

パイプラインの「入口」（DB スキーマ・Proto・フロントエンド UI）と「出口」（handler → フロントエンド表示）は実装済みだったが、**可用性フラグを `true` に更新する中間の接続**が欠落していた。結果、両フラグはデフォルト値 `false` のまま固定され、ボタンは常にグレーアウト（disabled）状態で表示されていた。

### 検討した代替案

1. **`enrichDigest()` で毎リクエスト問い合わせ** — `get_knowledge_home_usecase` の request-time enrichment で recap-worker API を呼ぶ方式。Canonical contract（[[000439]]）が定める `today_digest_view` を source of truth とする原則に反し、Home リクエストが recap-worker の可用性・レイテンシに直結するため不採用。
2. **Knowledge Projector 内で availability を設定** — Projector は `ArticleCreated` / `SummaryVersionCreated` イベント処理時に digest を upsert するが、Recap/Pulse の生成状態はイベントソースとは無関係なため、責務外。

## Decision

**定期 reconcile job (`DigestAvailabilityReconcileJob`) を新設し、recap-worker API から可用性を問い合わせて `today_digest_view` を更新する方式を採用する。**

### 設計

- 5分間隔で実行（`recall-projector` と同じ `AllowedUserIDs` を走査）
- `RecapPort.GetSevenDayRecap()` で Weekly Recap の存在を確認
- `RecapPort.GetEveningPulse(today)` で Evening Pulse の存在と状態を確認
- `PulseStatusError` 以外のステータス（`normal`, `partial`, `quiet_day`）は「利用可能」と判定
- 判定結果を `UpsertTodayDigest` で書き込み

### エラーハンドリング戦略

| 状況 | 動作 |
|------|------|
| API 成功 | `true` を書き込み |
| `ErrRecapNotFound` / `ErrEveningPulseNotFound` | `false` を書き込み（OR マージで既存 `true` は維持） |
| 一時的エラー（タイムアウト等） | 該当チェックをスキップ |
| 両方とも一時エラー | upsert 自体をスキップ（誤った `false` の書き込みを防止） |
| 片方のみ一時エラー | もう片方の結果だけ upsert（OR マージにより安全） |

### read path への影響

`enrichDigest()` は引き続き `NeedToKnowCount` と `DigestFreshness` のみ enrichし、availability フラグには触れない。availability は `today_digest_view` から読むだけ。

### 主要ファイル

| ファイル | 役割 |
|---------|------|
| `alt-backend/app/job/digest_availability_reconcile_job.go` | Reconcile job 本体 |
| `alt-backend/app/job/digest_availability_reconcile_job_test.go` | テスト（11ケース） |
| `alt-backend/app/main.go` | Scheduler へのジョブ登録 |

### OR マージの活用

`UpsertTodayDigest` の SQL は OR ロジック（`EXCLUDED.weekly_recap_available OR today_digest_view.weekly_recap_available`）を採用しており、一度 `true` になったフラグは `false` で上書きされない。reconcile job はこの設計を前提に、安全に `false` を渡せる（既存の `true` は維持される）。

## Consequences

### Pros

- **既存の read path を変更しない** — `enrichDigest()` や handler に手を入れず、write-side の接続のみで解決
- **recap-worker の一時障害に耐性がある** — スキップ戦略により、障害時に既存の `true` が `false` に崩れない
- **既存インフラの再利用** — `RecapPort`, `UpsertTodayDigestPort`, `AllowedUserIDs` をすべて流用し、新しい依存を追加しない
- **Canonical contract との整合** — `today_digest_view` を source of truth に保つ設計方針を維持

### Cons / Tradeoffs

- **最大5分の反映遅延** — Recap/Pulse が生成されてから TodayBar に反映されるまで最大5分かかる。リアルタイム性が必要な場合はイベント駆動への移行を検討する
- **AllowedUserIDs に依存** — 現時点では config の固定リストを走査するため、ユーザー数が増えた場合は `ListDistinctUserIDs` ポートへの移行が必要（recall-projector と同じ課題）
- **recap-worker API への定期ポーリング** — 5分 x ユーザー数のリクエストが発生するが、現在のユーザー数では問題にならない規模
