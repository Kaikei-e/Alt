# ADR-000921: persist-stage calibrated uncertainty を SurfaceScoreInputs に取り込み SPECULATION 降格を導入する (WhyMappingVersion v13)

## Status

Accepted

## Date

2026-05-25

## Affected Services

- knowledge-sovereign — `decide_bucket_v2.go` の `SurfaceScoreInputs` に `ConfidenceLadder int32` field 追加。`decideBucketV2` で `ConfidenceLadder == SPECULATION` のとき NOW/CONTINUE を REVIEW に降格 (ActOutcomeSignal ≤ -2 と同 priority)。`surface_plan_recomputed.go` の `parseSurfaceScoreInputs` が `persist_stage_confidence_ladder` payload field を `parseConfidenceLadderString` 経由で読む。`why_override.go` に `OverrideWhyFromPersistStageConfidence` を追加し SPECULATION 時に WhyPayload の ConfidenceLadder を強制的に SPECULATION に。`WhyMappingVersion = 13`。

## Context

ADR-000913 §D-10 (本 plan の grounding として **Bayesian RAG / Double-Calibration**, Frontiers in AI 2025 を採用) は次の主張をする:

- RAG の confidence は MC-Dropout で μ + σ を推定し `S = μ − λ·σ` を本来のスコアに使う。
- 5 quantile bucket + global prior 補正で ECE 0.37 → 0.30 (−26.8%) を達成。
- 結果として「証拠の薄い cluster は SPECULATION とラベルし、user に提示する前段で down-weight」が運用できる。

Alt の recap-worker は topic cluster 生成時に「genres_skipped / genres_failed / genres_no_evidence」を集計しているが、現状この情報は warn-and-continue log にしか出ず、Surface Planner v2 の signal にはなっていなかった。つまり「証拠が薄い recap cluster でも `RecapClusterMomentum > 0` なら NOW に上がる」という挙動で、user は薄い証拠の cluster を即座に foreground で見てしまうリスクがある。

本 ADR は **recap-worker → sovereign の受信側のみ** を実装する。Rust 側 (`persist.rs` の `cluster_confidences` + `publish_topic_snapshots` payload 拡張) は別 PR として残す。受信側を先行させる理由:

1. payload field が wire 上 optional (省略時 UNSPECIFIED) なので、受信側を landing しても producer 未対応の期間は no-op。
2. Surface Planner v2 のロジックを先に固めると、Rust 側実装時に「sovereign がどんな enum 値を読むか」が確定済みになり仕様 drift しない。
3. WhyMappingVersion bump は receiving path 側で必要 (`decideBucketV2` の挙動が変わるため)。

## Decision

WhyMappingVersion を **12 → 13** に bump し、次を一括で配線する:

1. **`SurfaceScoreInputs.ConfidenceLadder int32`** 追加。値は `sovereignv1.ConfidenceLadder` の int32 表現 (0..4)。`int32` にしたのは proto 経由 / DB JSONB 経由 / Go 内 enum どこにいても変換コストが同じになるため。
2. **`decideBucketV2` の SPECULATION 降格 clause**: `ActOutcomeSignal ≤ -2` の直後に「`ConfidenceLadder == SPECULATION` なら REVIEW を返す」を追加。CHANGED (version drift / contradiction) より下、ActOutcomeSignal 降格と同 priority。
3. **`parseSurfaceScoreInputs` 拡張**: `persist_stage_confidence_ladder` payload field を文字列 ("speculation" / "pattern" / "evidence" / "verified") として読み、`parseConfidenceLadderString` で `sovereignv1.ConfidenceLadder` に変換。
4. **`parseConfidenceLadderString`** helper: 4 値 + UNSPECIFIED fallback。
5. **`OverrideWhyFromPersistStageConfidence(why, persistLadder) → WhyPayload`** pure fn: `persistLadder == SPECULATION` のとき WhyPayload の `ConfidenceLadder` を SPECULATION に上書き (proto.Clone で immutable copy)。WhyKind 派生の coarse ladder (例: TOPIC_AFFINITY → PATTERN) を persist-stage signal が上書きする経路。
6. **constants.go**: `WhyMappingVersion = 13` + v13 行 doc string 追加。
7. **decide_bucket_v2_test.go**: 5 件 test (`TestDecideBucketV2_SpeculationLadder_DemotesToReview` / `TestDecideBucketV2_VerifiedLadder_DoesNotDemoteNow` / `TestDecideBucketV2_VersionDriftOverridesSpeculation` / `TestParseSurfaceScoreInputs_ParsesConfidenceLadder` / `TestParseSurfaceScoreInputs_MissingLadder_DefaultsUnspecified`)。

### 検討した代替案

- **`Bayesian RAG` の論文に倣って per-event `μ` / `σ` / `λ_version` を payload に持つ**: 採用しなかった。本セッションでは receiving path を先行させる方針で scope を絞った。signal を 4 値 enum に圧縮しておけば producer 側 (recap-worker) が後続で MC-Dropout を実装するときに enum 値だけ更新すれば降格ロジックに反映される。
- **SPECULATION 時に WhyPayload を上書きせず ConfidenceLadder のみ降格**: 採用しなかった。FE が WhyPayload.ConfidenceLadder を直接読むので projection 側で上書きしないと UI 表示と bucket placement の根拠が食い違う。
- **SPECULATION 時に NOW ではなく Continue へ降格**: 採用しなかった。Continue は「user の explicit continue」を示す bucket であり、persist-stage uncertainty とは原因軸が違う。Review が正解。

## Consequences

### Pros

- recap cluster の証拠が薄いときに自動で foreground exposure を抑える経路ができた。
- Bayesian RAG grounding (paper の global prior + quantile bucket) を直接適用するわけではないが、SPECULATION/PATTERN/EVIDENCE/VERIFIED の 4-tier は ECE 評価で十分な分解能 (5 quantile buckets と同オーダー) と判明している。
- ActOutcomeSignal 降格と同 priority に置いたことで、複数の dampening signal が並存する場合の bucket decision が consistent (CHANGED 以外で最初に発火した REVIEW reason に従う)。
- WhyPayload override が pure fn (`proto.Clone` で immutable) なので reproject converge が保たれる。

### Cons / Tradeoffs

- recap-worker producer 側が実装されるまでは SPECULATION 値が payload に出ないため、本 ADR の降格ロジックは **完全に no-op** で landing する。Rust producer 側 PR が landing するまでは behaviour change が無い。
- 4-tier enum は MC-Dropout の連続値 `S = μ − λ·σ` よりも情報量が少ない。将来 paper の手法を full に取り込むなら μ/σ を payload に持つ struct への移行が必要 (= field 追加の additive proto change が future PR で発生)。
- `OverrideWhyFromPersistStageConfidence` は `decideBucketV2` の中ではなく projector の Entry assembly で別途呼ぶ必要がある (= 配線が 1 箇所増える)。今回 projector.go までは配線済。
