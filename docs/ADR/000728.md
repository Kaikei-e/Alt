# ADR-000728: recap-subworker の Phase 1 lifespan DI 切替とテストハング解消

## Status

Accepted

## Date

2026-04-15

## Affected Services

- recap-subworker (Python 3.14 / FastAPI)

## Context

[[000726]] で合意した 6 フェーズの大規模リファクタリングの **Phase 1** として、`recap-subworker` のモジュールレベル global (`_container`, `_extract_semaphore`, `_ENGINE`, `_SESSION_FACTORY`, `_PIPELINE`) を撤去し、`ServiceContainer` の寿命を FastAPI の `@asynccontextmanager` lifespan に紐付ける作業を進めた。切替そのものは先行コミット `f8cefa988` で完了し、追加コミット `6583df104` で `RequestSizeLimitMiddleware` を `BaseHTTPMiddleware` 由来から pure ASGI middleware に書き換えた（Starlette issue #847 / #1320 に起因する POST ボディ再読 deadlock を根本回避）。

しかし Phase 1 RED テスト `tests/unit/test_api_runs.py::test_post_runs_requires_job_id_header` / `test_post_runs_returns_accepted` / `test_get_runs_not_found` が `TestClient(app)` 経由での `POST /v1/runs` で **無限にハング** する現象が残った。`GET /health` は即時 200 を返すため middleware ではなく、POST の依存解決パスでブロックが発生していた。再現条件:

- Python 3.14.3 / FastAPI 0.133.0 / Starlette 0.52.1 / anyio 4.12.1
- `Depends(get_run_manager_dep)` が sync 関数 → `anyio.to_thread.run_sync` で単一ワーカースレッドへ投入
- ワーカースレッド上で `container.run_manager` プロパティが lazy 初期化を起動
- `container.pipeline` 経由で `ProcessPoolExecutor(mp_context=spawn, max_tasks_per_child=100)` を構築
- `container.classifier` 経由で `GenreClassifierService` + `Embedder(backend="sentence-transformers")` を初期化

Web 調査（複数 Tier A ソース）で以下を確認した:

- spawn 方式の `ProcessPoolExecutor` 構築はワーカープロセスの起動を同期待機し、子プロセスは親のモジュールグラフを再 import する。torch / tokenizers / numpy 2.x の import ロック組み合わせは既知のスタール原因（CPython #105829 / #105464 / #90622、pytorch #17199、transformers #5486）。
- `max_tasks_per_child` が設定されていると fork backend を選んでも強制的に spawn 相当となる（Python 公式 concurrent.futures ドキュメント）。
- Phase 1 以前は `_container` が module global であり「プロセス寿命で 1 回だけ」重量級初期化を実行。テスト 254 件が 44s で通っていたのはこのキャッシュ効果。Phase 1 で `create_app()` 毎に新 `ServiceContainer` を生成するようになり、各テストのリクエスト処理パスで重量級初期化が反復される構造になった。

Starlette 1.0.0 / FastAPI 0.135.x へのアップグレードも検討したが、Web 調査の結論は「このハングに対してライブラリ更新は無効」であり、別要因の hygiene upgrade として Phase 外で扱うこととした。なお `prometheus-fastapi-instrumentator==7.1.0` が `starlette>=0.30.0,<1.0.0` を pin するため、アップグレード時には override または代替パッケージ (`prometheus-starlette-instrumentator` 等) への切替が必要になる。

## Decision

以下を採用する:

1. **単体テストからは重量級 `ServiceContainer` プロパティを一切触らせない**。`tests/conftest.py` に autouse fixture を追加し、`TestClient.__enter__` を一時的にパッチして FastAPI の `app.dependency_overrides` に軽量スタブを注入する。対象は `get_run_manager_dep` / `get_classifier_dep` / `get_classification_runner_dep` / `get_content_extractor_dep` / `get_coarse_classifier_dep` / `get_admin_job_service_dep` / `get_learning_client` / `get_extract_semaphore_dep` の 8 件。個別テストが本物の依存を必要とする場合は `@pytest.mark.no_heavy_override` でオプトアウト可能とする。

2. **本番経路でのデッドロック抑止**: `recap_subworker/app/main.py` の環境変数既定値に `TOKENIZERS_PARALLELISM=false` を追加。Python スポーン時のトークナイザ fork 警告を消し、`ProcessPoolExecutor` bootstrap 中の潜在的デッドロックを予防する。

3. **ライブラリアップグレードは本 Phase では実施しない**。Starlette 1.0 / FastAPI 0.135 移行は別 Phase (Out of scope for now) とし、記録のみ残す。

4. **統合テストは次 Phase 以降 `httpx.AsyncClient(transport=ASGITransport(app))` 形式で書く**（Phase 6 で導入予定）。

## 実装の概要

### 変更ファイル（本 Phase 完了コミット `cb60a186e`）

| ファイル | 変更 |
|------|------|
| `recap-subworker/tests/conftest.py` | autouse fixture `_override_heavy_dependencies` と 8 種のスタブクラス (`_StubRunManager`, `_StubClassifier`, `_StubClassificationRunner`, `_StubContentExtractor`, `_StubCoarseClassifier`, `_StubAdminJobService`, `_StubLearningClient`) を追加。`TestClient.__enter__` をフィクスチャ寿命の間だけパッチして `app.dependency_overrides.setdefault(...)` を実行する |
| `recap-subworker/recap_subworker/app/main.py` | `os.environ.setdefault("TOKENIZERS_PARALLELISM", "false")` を既存の threading 既定値群（`OMP_NUM_THREADS` / `MKL_NUM_THREADS` / `OPENBLAS_NUM_THREADS` / `NUMBA_THREADING_LAYER`）と同位置に追記 |

### 先行コミット済みの Phase 1 本体

- `f8cefa988` — `ServiceContainer` を lifespan 所有に、`DatabaseResources` 導入、module global 完全撤去、`infra/db/session.py` を `create_database_resources()` ベースに
- `6583df104` — `RequestSizeLimitMiddleware` を pure ASGI 化（`BaseHTTPMiddleware` 廃止）

### 検証

```bash
cd recap-subworker
SERVICE_SECRET=test-secret uv run pytest tests/unit/ --no-cov -q
# → 262 passed, 12 skipped in 44.43s（gensim 依存の 12 skip は既知）

uv run ruff check recap_subworker/
# → All checks passed!

uv run pyrefly check .
# → 0 errors (55 suppressed)

# Container rebuild
docker compose -f compose/compose.yaml -p alt up --build -d recap-subworker
# → alt-recap-subworker-1 Image Built, Container Started

curl -f http://localhost:8002/health
# → {"status":"ok","model_id":"intfloat/multilingual-e5-large","backend":"ollama-remote"}
```

Phase 0 完了時点のベースライン（254 passed / 44s）と一致、新規 Phase 1 テストの 8 件分増加で **262 passed** に到達。ハング 0、回帰 0。

## Consequences

### Pros

- `TestClient` 経由の POST 系テストがハングせず実行可能になった。現行 3.14 + spawn PPE + torch/ST + 単一 anyio ワーカースレッドの組み合わせを既知のスタール症状群から根本回避。
- 単体テスト側で重量級初期化を発生させないため、CI 時間が Phase 0 レベル（44s）に維持される。Container ownership を lifespan に移した構造的メリットは保持。
- fixture は autouse + オプトアウトマーカー形式で、テスト毎の override コードを書かずに既定で安全。本物の依存を使う統合テスト・契約テストは `@pytest.mark.no_heavy_override` で明示的に opt-in できる。
- `TOKENIZERS_PARALLELISM=false` は本番環境でも意図どおり（spawn 経由のトークナイザ初期化時のデッドロック予防）。
- Starlette / FastAPI の現行 pin を維持したまま解決できたため、`prometheus-fastapi-instrumentator` や他 Python 依存ツリーに影響を出していない。

### Cons / Tradeoffs

- `conftest.py` が `TestClient.__enter__` を monkey-patch する設計は FastAPI 本体に対する想定前提が増える。FastAPI / Starlette のアップグレード時には patch 箇所の再評価が必要（monkey-patch は fixture 寿命に限定しているため競合リスクは低い）。
- 単体テストから Container 内部の real dependency に触れるパスが完全に断たれるため、`run_manager` 直結の edge case（spawn pool 起動失敗時のフォールバック、分類器モデル欠落の fail-fast 等）は integration / contract テストでしか検証できない。Phase 6 で補完する想定。
- Starlette 1.0.0 / FastAPI 0.135.x への移行は本 ADR では棚上げ。hygiene 上げ時には `prometheus-fastapi-instrumentator` の pin 解除（`tool.uv.override-dependencies = ["starlette>=1.0.0"]`）と代替パッケージ（`prometheus-starlette-instrumentator` / `starlette-exporter`）の検討を別 ADR で扱う。
- 本番 `lifespan` での eager init（`await anyio.to_thread.run_sync(lambda: container.run_manager)` を `yield` 前に呼ぶ）は未導入。効果は fail-fast と最初のリクエスト遅延の回避であり、本 Phase では不要と判断したが、SRE 視点で導入したい場合は Phase 2 以降で追加する。
