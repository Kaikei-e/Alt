# ADR-000316: recap-evaluator 責務不全の修正 — 評価結果永続化・アラート伝搬・トレンドAPI・ヘルスチェック

## ADR's STATUS

完了

## CONTEXT

recap-evaluator は 7 日間 Recap の品質を 4 次元（Genre / Cluster / Summary / Pipeline）で評価するサービスである。定期スケジュール実行と HTTP API の両方で評価を行うが、以下 4 件の責務不全が存在していた。

| # | 問題 | 影響度 |
|---|------|-------|
| 1 | `RunEvaluationUsecase.execute()` が `EvaluationRun` を返すだけで DB に保存しない | 致命的 — 全評価結果が揮発、トレンド分析不可 |
| 2 | `ClusterEvaluator.evaluate_batch()` の集約結果にアラート閾値が未適用、かつ cluster alert が overall alert に反映されない | 高 — クラスタリング品質劣化を検知できない |
| 3 | `/api/v1/metrics/trends` が空配列を返すスタブのまま | 中 — 品質推移の可視化が不可 |
| 4 | `/health` が DB・外部依存の状態を確認せず常に `healthy` を返す | 中 — 障害検知の遅延 |

## DECISION MAKING

### Fix 1: 評価結果の永続化

`EvaluationRun` および各次元のメトリクスデータクラスに `to_dict()` / `to_metrics_dict()` メソッドを追加し、JSON シリアライズ可能な形式に変換できるようにした。

`RunEvaluationUsecase.execute()` の末尾で `db.save_evaluation_run()` を呼び出し、評価結果を `recap_evaluation_runs` テーブル（JSONB カラム）に永続化する。ジョブが 0 件の場合は保存をスキップする。

併せて `recap_evaluation_runs` テーブルの Atlas マイグレーションを追加した。

### Fix 2: Cluster アラートの閾値適用と全体評価への伝搬

**2a. `evaluate_batch()` へのアラート閾値適用**

`ClusterEvaluator.evaluate_batch()` のジャンルごとの集約結果に対し、`clustering_silhouette` の warn / critical 閾値を適用して `alert_level` を設定するようにした。`evaluate_with_embeddings()` には既にこのロジックが存在していたが、`evaluate_batch()` の集約パスでは欠落していた。

**2b. overall alert への反映**

`RunEvaluationUsecase.execute()` で cluster_metrics 取得後、各ジャンルの `alert_level` から最も深刻なものを `AlertResolver.resolve()` で算出し、`alert_levels` リストに追加するようにした。

### Fix 3: Trends エンドポイント実装

`GetMetricsUsecase` に `get_trends(window_days)` メソッドを追加した。

- `db.fetch_evaluation_history()` から過去の保存済み評価結果を取得
- JSONB の `metrics` カラムからキー指標（`genre_macro_f1`, `cluster_avg_silhouette`, `pipeline_success_rate`, `overall_quality_score`）を時系列データとして抽出
- 7 日 / 30 日の変化率を算出

`metrics_handler.py` のスタブを実際のユースケース呼び出しに置き換えた。

### Fix 4: ヘルスチェック改善

`/health` エンドポイントで `asyncpg.Pool.fetchval("SELECT 1")` による DB 接続チェックを実行するようにした。

- DB 正常時: `{"status": "healthy", "checks": {"db": "ok"}}`
- DB 異常時: `{"status": "degraded", "checks": {"db": "unavailable"}}`

HTTP ステータスコードは常に 200 を返す（Docker healthcheck との互換性維持のため）。Ollama チェックはリクエスト時のレイテンシが大きすぎるため、起動時のみ実施する既存の方針を維持した。

## RESULTS, EFFECTS

### 検証結果

| # | 変更内容 | 検証 |
|---|---------|------|
| 1 | 評価結果永続化（domain + usecase + migration） | pytest: 134 passed, 4 skipped |
| 2a | evaluate_batch アラート閾値適用 | 新規テスト 2 件 passed |
| 2b | cluster alert → overall 伝搬 | 新規テスト 1 件 passed |
| 3 | trends エンドポイント実装 | 新規テスト 3 件 passed（usecase + handler） |
| 4 | ヘルスチェック DB 接続確認 | 新規テスト 2 件 passed |
| 5 | コンテナ再ビルド・起動 | healthy 確認、trends API 正常応答確認 |
| 6 | Atlas マイグレーション適用 | recap_evaluation_runs テーブル作成成功 |

### PROS

- 評価結果が永続化され、スケジュール実行のたびに品質データが蓄積されるようになった
- クラスタリング品質の劣化がアラートとして検知・伝搬されるようになった
- `/api/v1/metrics/trends` で品質推移の可視化が可能になった
- ヘルスチェックが DB 接続状態を反映するようになり、障害の早期検知が可能になった
- 既存テスト 134 件に対するリグレッションなし

### CONS, TRADEOFF

- `to_dict()` メソッドの追加により domain model の表面積が増加する。Pydantic の `model_dump()` 的なアプローチも検討したが、dataclass ベースの既存設計との一貫性を優先した
- `recap_evaluation_runs` テーブルの `metrics` カラムは JSONB であり、スキーマ変更時のマイグレーション不要という柔軟性と引き換えに、型安全性は DB 層では保証されない
- ヘルスチェックの Ollama 確認はリクエスト時に行わない設計としたため、Ollama 障害はヘルスチェックでは即時検知されない（summary 評価実行時にエラーとして検知される）
- `_compute_change()` の変化率計算はデータポイントが少ない初期段階では `None` を返す。十分な履歴が蓄積されるまでトレンドの精度は限定的
