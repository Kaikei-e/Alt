# ADR-000702: 画像プロキシの gzip 透過解凍を net/http Transport に委譲する

## Status

Accepted

## Date

2026-04-12

## Affected Services

- alt-backend (Go)

## Context

alt-backend の画像プロキシ (`/v1/images/proxy/:sig/:url_b64`) が一部の外部 CDN 上の画像で `WARN: image proxy error — process image: decode image: image: unknown format` を出力し続け、クライアントには 502 Bad Gateway が返されていた。該当画像を直接 `curl` すると正常な JPEG であり、ブラウザでも問題なく表示される状態だった。

`ImageFetchGateway` は外部 HTTP レスポンスを読んで `ProcessingGateway.ProcessImage` → `image.Decode` に渡す流れ。Go 標準の `image.Decode` は登録済み形式（jpeg / png / gif / webp）しか扱えないが、上記 CDN は `Content-Type: image/jpeg` を返しており、形式ミスマッチは説明がつかなかった。

原因特定のためまず観測性を強化し、デコード失敗時に **upstream の Content-Type、レスポンス先頭 16 バイトの hex、magic-byte 由来の format 推定** をエラーに含めるよう変更した。これにより実ログから以下が判明した:

```
error: "process image: decode image (upstream_content_type=\"image/jpeg\" detected_format=unknown magic=1f8b080000000000000394bb75541c4d size=36113): image: unknown format"
```

`1f 8b 08 00` は **gzip ファイルのマジックバイト**である。すなわち CDN は gzip 圧縮した JPEG を返しており、alt-backend はそれを生バイト列のまま画像デコーダに渡していた。

追跡の結果、`app/gateway/image_fetch_gateway/image_fetch_gateway.go` で HTTP リクエストに以下のヘッダが手動設定されていた。

```go
req.Header.Set("Accept-Encoding", "gzip, deflate")
```

Go `net/http` の [Transport ドキュメント](https://pkg.go.dev/net/http#Transport) に記載の通り、`Accept-Encoding` は **Transport が自ら付与した場合にのみ** 透過解凍の対象となる。ユーザーが明示設定すると「呼び出し側が自前で解凍する」と解釈され、自動解凍が無効化される。さらに `deflate` は Go 標準では自動解凍対象ではないため、どのみち自前解凍が必要な宣言になっていた。

結果として圧縮バイトが `image.Decode` に到達し、unknown format になっていた。

### 制約

- alt-backend は `CGO_ENABLED=0` でビルドされているため、CGO 依存のデコーダ追加は不可
- decompression bomb（gzip bomb）の攻撃面を広げてはならない
- 既存の SSRF 防御・ドメインホワイトリスト・キャッシュ層は維持すること

## Decision

以下の三点を採用する。

1. **手動 `Accept-Encoding` ヘッダの削除**
   `req.Header.Set("Accept-Encoding", ...)` を削除し、gzip 透過解凍を `http.Transport` の既定動作に委ねる。Transport は `Accept-Encoding: gzip` を自動付与し、`Content-Encoding: gzip` レスポンスを透過的に解凍し、`resp.Body` を解凍済みストリームで返す。

2. **未解凍エンコーディングを拒否する `validateResponseEncoding` ガードの追加**
   レスポンス取得後、本文を読む前に `Content-Encoding` ヘッダを検査する。Transport が透過解凍した場合このヘッダは除去されるため、残留する値は `br` / `zstd` / `deflate` 等 Go 標準が自動処理しないエンコーディングを意味する。これらは圧縮バイトのまま後続に流れると opaque な decode 失敗を招くため、`VALIDATION_ERROR` として早期拒否する。`empty` と `identity` は許容。

3. **既存 `io.LimitReader(resp.Body, MaxSize+1)` を維持**
   Transport が透過解凍を行う場合、`resp.Body` は解凍ストリームになる。既存の 5 MB 上限は解凍後バイトに対して適用されるため、gzip bomb（AWS CodeGuru Detector / Datadog Static Analysis が警告する decompression bomb）は 5 MB で切断され DoS を防ぐ。別途の bomb 対策は不要。

### 代替案と評価

- **AVIF デコーダ等を追加する:** 当初は CDN が AVIF/HEIC/JXL を返している仮説を立てたが、観測性強化の結果 gzip であることが判明し不要となった。仮に将来必要になっても `CGO_ENABLED=0` 前提で pure-Go 実装（wazero による WASM 等）は binary size への影響が大きく、実ログでの出現頻度が確認できてから判断する follow-up とした。
- **Accept-Encoding を残しつつ自前で gzip 解凍する:** 実装量が増えるうえ Go 標準の透過解凍と重複する。bomb 防御も自前で書き直すことになり、ミスの温床になる。却下。

## Consequences

### Pros

- gzip 圧縮を返す CDN 上の画像（Dezeen, 多くの汎用 CDN）で decode エラーが消え、正常な JPEG として処理されるようになる
- 実装は 1 行削除 + 小さなガード関数追加のみで、保守負荷が増えない
- `validateResponseEncoding` によって将来 CDN が brotli/zstd を返した場合も、不明な圧縮バイトを画像デコーダに流さず早期に明示的なエラーとして弾ける（defense-in-depth）
- decompression bomb 対策は既存の `io.LimitReader` によって自動的に解凍後サイズへ適用される
- エラー時の診断情報（Content-Type, magic bytes）がログに残るため、今後類似の問題が出た際の切り分け時間が大幅に短縮される

### Cons / Tradeoffs

- CDN が `br`（brotli）/ `zstd` を返す場合、従来は未定義挙動（結果として `unknown format`）だったものが `VALIDATION_ERROR` として明示拒否されるようになる。現状これらの encoding をサポートしない方針を明示的に表明する形になる
- Go の透過解凍に依存するため、将来 Go ランタイム側の挙動が変われば影響を受ける（ただし長年安定している API であり現実的リスクは小さい）
- AVIF/HEIC/JXL 等 Go 標準非対応形式は依然として `unknown format` として 502 を返す。これらは今後のログ観測で出現頻度を見てから対応可否を判断する
