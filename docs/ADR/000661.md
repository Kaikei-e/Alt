# ADR-000661: section_citations を claim 単位で生成・永続化する

## Status

Accepted

## Date

2026-04-09

## Affected Services

- acolyte-orchestrator (Python 3.14+)
- alt-frontend-sv (TypeScript / SvelteKit)

## Context

[[000659]] で `SectionPlannerNode` と `ClaimPlan` が導入され、Writer は raw evidence ではなく構造化された `PlannedClaim` からのみ本文を生成するようになった。しかし、本文中の `[1], [2]` マーカーはテキスト装飾にすぎず、**claim → source_id → quote の構造化 citation は存在しなかった**。

具体的な問題:

1. **`section_citations` が常に空**: `ReportGenerationState.section_citations` フィールドは定義済みだが、パイプライン内でどのノードも populate していなかった
2. **FinalizerNode は空データを保存**: `section_citations.get(key)` は常に `None` → `citations_jsonb` に `[]` が保存
3. **GetReport がハードコード**: `connect_service.py:86` が `citations_json="[]"` を固定返却。DB に citation が保存されていても返さない
4. **Attribution 不能**: 見た目に引用があっても、内部的にはどの主張がどの evidence に支えられているかを追跡できない

学術的背景:

- **Attribute First, then Generate** (ACL 2024): content selection → sentence planning → generation の 3 段分離で fine-grained attribution を実現
- **ExpertQA** (EMNLP 2023): source を並べるだけでなく、主張がどの evidence に支えられているかを局所的に示すことが attribution の核
- **Citation-Aware RAG**: claim_id → source_id → quote/span の構造化で、UI 上で claim ↔ citation の対応表示が可能になる

## Decision

**WriterNode 内で claim_plans から決定的に citation を組み立て、パイプライン state → FinalizerNode → DB → GetReport → UI の全経路でデータを流す。新ノード追加は不要。**

### 主要な設計判断

#### 1. Citation assembly は WriterNode 内で行う（新ノード不要）

Citation は `PlannedClaim` の `evidence_ids` + `supporting_quotes` から決定的に導出可能であり、LLM 呼び出しが不要。WriterNode がセクション本文生成後に citation を組み立て、`section_citations` を state に返す。

新ノード追加を不採用とした理由:
- グラフ複雑化に対して利点が少ない
- revision ループ（critic → writer）で citation assembler も再実行する必要が生じ、配線が煩雑になる
- citation は writer の出力（本文テキスト）と同時に生成する方が整合性が高い

#### 2. `claim_id` は SectionPlannerNode が `{section_key}-{N}` 形式で付与

LLM に `claim_id` を生成させると不安定（Gemma4 26B の structured output 精度に依存）。SectionPlannerNode が `generate_validated()` 後に決定的に付番する方が確実。format は `analysis-1`, `conclusion-2` のように人間可読。

#### 3. offset mapping は best-effort（テキスト内検索）

LLM 生成テキスト内の正確な位置特定は困難。`supporting_quotes` の先頭 50 文字をテキスト検索し、見つかれば `offset_start/offset_end` を記録、見つからなければ `-1` のまま。引用文自体は `quote` フィールドに必ず格納する。

#### 4. Citation なし claim を finalizer 前に reject

`must_cite=True` の claim 全てが citation を持たない場合（`evidence_ids` が空）、そのセクションの body を空文字にして reject する。一部の claim が cited であれば body は保持し、uncited claim は警告ログに記録する。

#### 5. SectionCitation を Pydantic モデルとして domain 層に定義

```python
class SectionCitation(BaseModel):
    claim_id: str
    source_id: str
    source_type: str = "article"  # "article" | "recap"
    quote: str = ""
    offset_start: int = -1
    offset_end: int = -1
```

`PlannedClaim`（composition 層）と `SectionCitation`（attribution 層）は異なるライフタイムを持つが、同じ `domain/claim.py` に配置した。両者は claim-to-source の対応関係で密結合しており、ファイル分離のメリットが薄い。

### 不採用とした代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| LLM に citation JSON を生成させる | Gemma4 26B の structured output 精度に依存。本文 + citation の同時生成は output 量が倍増し、parse 失敗リスクが高い |
| 新ノード `CitationAssemblerNode` を追加 | グラフ配線が煩雑。revision ループとの整合性が複雑化 |
| critic に citation 完全性チェックを追加 | critic の scope 拡大は Issue 5（section contract）の領域。今回は writer 内 validation で十分 |
| offset を LLM に出力させる | 生成テキストの offset は LLM が正確に出せない。post-hoc string matching の方が確実 |

### データフロー確認

パイプライン実行時の citation フロー:

1. **SectionPlannerNode**: 各 claim に `claim_id: "summary-1"` を付番
2. **WriterNode**: `_assemble_citations()` で claim × evidence_id → citation 生成、`section_citations` を state に返却
3. **WriterNode (validation)**: 全 must_cite claim が uncited → body を空にして reject
4. **FinalizerNode**: `section_citations.get(key)` → `bump_section_version(citations=...)` → DB 永続化
5. **GetReport**: `json.dumps(sv.citations)` → `citations_json` フィールドで返却
6. **Frontend**: `parseCitations(sec.citationsJson)` → Sources footer に表示

## Consequences

### Pros

- **構造化 attribution**: claim → source_id → quote の対応が正式データとして永続化され、attribution 可能になった
- **LLM 追加コスト 0**: citation assembly は決定的関数であり、LLM 呼び出しを増やさない
- **品質ゲート**: `must_cite` claim 全てが uncited ならセクションを reject。根拠なき主張の排除
- **インフラ変更 0**: Proto、DB スキーマ、FinalizerNode、Gateway はすべて既存のまま
- **テスト充実**: 16 テスト追加で citation 生成・validation・永続化・API 配信をカバー
- **UI 対応**: フロントエンドで Sources セクションとして citation を表示

### Cons / Tradeoffs

- **offset mapping は best-effort**: LLM がパラフレーズした場合、`offset_start/offset_end` は `-1` になる。テキストハイライトには不十分
- **source_type は "article" 固定**: recap 対応は Issue 9 scope。現時点では全 citation が article 型
- **partial reject なし**: 一部 claim が uncited でも body 全体は保持される。claim 単位の除外は未実装
- **WriterNode の責務拡大**: citation assembly + validation が WriterNode に入り、単一責任原則の境界が広がった。将来的にノード分離を検討する余地がある
- **revision ループでの再組立**: critic → writer の revise パスで citation が再組立される。claim_plans 自体は再計画されないため整合性は保たれるが、テキスト変更で offset が変わる
