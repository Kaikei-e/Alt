# ADR-000441: /articles/[id] の $effect 無限ループを修正する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-frontend-sv (TypeScript/SvelteKit)

## Context

[[000440]] で新設した `/articles/[id]/+page.svelte` がバックエンドに対して無限リクエストを発生させていた。alt-backend のログで同一 URL への "article retrieved from database" が毎 ~100ms で連続出力されていることを確認した。

### 根本原因

`$effect` 内で `fetchContent()` を呼んでいたが、Svelte 5 の `$effect` は実行中に同期的に読まれたリアクティブ state をすべてトラッキングする。`fetchContent()` の先頭で `isFetching` を条件分岐に使っていたため、以下のサイクルが成立していた:

1. `$effect` 発火 → `fetchContent()` → `isFetching`（false）を読む → fetch 開始 → `isFetching = true`
2. `isFetching` 変化 → `$effect` 再発火 → `isFetching = true` → ガードで return
3. fetch 完了 → `isFetching = false`
4. `isFetching` 変化 → `$effect` 再発火 → `isFetching = false` → 再び fetch 開始
5. ステップ 2 に戻る（無限ループ）

問題の本質は、`$effect` のトラッキング範囲がコールスタックを越えて `fetchContent()` 内部の state 読み取りまで及ぶことにある。関数呼び出しで間接的に読まれた state もすべて依存として登録されるため、副作用を起こす関数の中でリアクティブ state をガード条件に使うと暗黙的な再発火トリガーになる。

## Decision

既存の `FeedDetailModal.svelte` で使われている idiomatic パターンに倣い、**ガード条件を `$effect` の条件式に直接含める**方式を採用する。

### 修正内容

```svelte
// Before（壊れている）
$effect(() => {
    if (articleUrl) {
        fetchContent();
    }
});

// After（FeedDetailModal パターン準拠）
$effect(() => {
    if (articleUrl && !articleContent && !isFetching && !contentError) {
        fetchContent();
    }
});
```

これにより `$effect` が `articleUrl`・`articleContent`・`isFetching`・`contentError` の 4 つを明示的にトラッキングし、以下の状態で effect は no-op になる:

- fetch 中: `!isFetching` が false → no-op
- fetch 成功: `!articleContent` が false → no-op
- エラー: `!contentError` が false → no-op

### URL 変化対応

`/articles/[id]` は SPA 内でのルートパラメータ変更がほぼ起きない想定だが、安全のため `articleUrl` の変化時にステートをリセットする effect を追加した。

```svelte
let previousUrl = $state<string | null>(null);

$effect(() => {
    if (articleUrl !== previousUrl) {
        previousUrl = articleUrl;
        articleContent = null;
        fetchedArticleId = null;
        contentError = null;
        isFetching = false;
    }
});
```

これにより URL が変わった場合は全 state がリセットされ、fetch 用 effect の条件が再度成立して新しい URL での fetch が走る。

### fetchContent() のガード簡略化

`$effect` 側でガード条件を管理するため、`fetchContent()` 内の `isFetching` チェックは不要になった。`!articleUrl` のみを残し、責務を明確に分離した。

### 検討した代替案

1. **`untrack()` で `isFetching` のトラッキングを回避する**: Svelte 5 の `$effect.tracking()` / `untrack()` を使えば特定の state 読み取りをトラッキング対象から外せる。しかしコードベースの他のコンポーネント（`FeedDetailModal.svelte`）では条件式パターンが確立しており、一貫性を優先した。
2. **`$effect` 内で直接 fetch ロジックを inline 化する**: トラッキング範囲が明確になる利点があるが、"Try again" ボタンからの再 fetch に `fetchContent()` を再利用しているため、関数抽出を維持した。

## Consequences

### Pros

- 無限リクエストループが解消され、バックエンドへの不要な負荷がなくなる
- `FeedDetailModal.svelte` と同じパターンを踏襲しており、コードベースの一貫性が保たれる
- URL 変化時のリセット effect により、将来的にルートパラメータが変わるケースにも対応できる
- `fetchContent()` の責務が「fetch の実行」のみに絞られ、呼び出し条件の管理は effect 側に集約された

### Cons / Tradeoffs

- `$effect` の条件式が 4 つの state を参照するため、新しい state を追加した際に条件の更新漏れが起きうる
- URL 変化リセット用の `previousUrl` state が増え、コンポーネントの state 数が 1 つ増加した
