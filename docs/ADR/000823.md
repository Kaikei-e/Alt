# ADR-000823: tag-generator の unidic dicdir ダウンロードを復活させ Hurl 失敗時のコンテナログダンプで silent-503 回帰を再発防止する

## Status

Accepted

## Date

2026-04-21

## Affected Services

- tag-generator (Python, FastAPI + SBERT / KeyBERT)
- e2e/hurl/tag-generator (Hurl suite orchestrator)

## Context

[[000819]] で tag-generator の Dockerfile を news-creator 互換の 2-stage uv sync パターンへ揃えた。image は狙い通り縮み CI の image-size gate も通ったが、新イメージで Hurl E2E の `/api/v1/extract-tags` が **503 Service Unavailable を 3 分間返し続けて落ちる** 回帰が出た。同じ suite は旧 image では通っていた。

根本原因は、Dockerfile 書き換えの過程で `RUN python -m unidic download` が**静かに落ちた**こと。`tag-generator/app/pyproject.toml` の `[ml]` group は以下の 2 つを同時に入れる構成になっている:

- `fugashi[unidic-lite]>=1.5.1` — `unidic-lite` dict が wheel にバンドルされて入る
- `unidic` — メタデータのみで `site-packages/unidic/dicdir/` は **空**。`python -m unidic download` を 1 回走らせて初めて `mecabrc` / `char.bin` / `matrix.bin` / `model.bin` が populate される

`fugashi.Tagger()` は引数なしで呼ばれると full `unidic` を優先し、空 dicdir を踏んで次の例外を吐く:

```
ModelLoadError: Failed initializing MeCab ...
no such file or directory: /app/.venv/lib/python3.14/site-packages/unidic/dicdir/mecabrc
```

これが `tag-generator/app/auth_service.py` の `_run_background_tag_generation` を括る `except Exception: logger.error(..., exc_info=True)` で握り潰され、`_background_tag_service` がモジュールレベルで `None` のまま固定される。以降 API サーバ自体は HEALTH=OK で生き続け、`/api/v1/extract-tags` は

```python
if _background_tag_service is None:
    raise HTTPException(status_code=503, detail="Tag extraction service not ready")
```

を無限に返す。外から見るとスタックは healthy、Hurl は 503 を retry し、CI の Run Hurl ステップだけが exit 4 で落ちる。

**2 段目の問題は可視性の欠損**。tag-generator 用 `e2e/hurl/tag-generator/run.sh` の `cleanup()` は下記のように全ログを捨てていた:

```bash
docker compose -f "$SLICE" -p "$STAGING_PROJECT_NAME" down -v --remove-orphans >/dev/null 2>&1 || true
```

`exc_info=True` で詳細 traceback はコンテナ stderr に出ていたが、teardown 前に回収する仕組みがなかったので CI 側には `503 × 180` の retry ログしか残らず、**スタックをローカル再現するまで原因にたどり着けない**状態だった。

[[000819]] の smoke check (`import torch, sentence_transformers, keybert, transformers, onnxruntime, fugashi, nltk`) は import までしか見ていないので、`Tagger()` を実際にインスタンス化する経路の回帰を検出できなかった。

### 計測

- 旧 image (Dockerfile 書き換え前): Hurl 7 シナリオ全成功 / 1.2 秒
- 新 image (Dockerfile 書き換え後): `/api/v1/extract-tags` が 503 を 181 回返したところで Hurl が retry budget を使い切り失敗。suite 全体で 3 分超のタイムアウト
- 本 ADR の修正適用後: 7/7 成功 / 1.2 秒 / teardown clean

## Decision

以下 2 点を入れて silent-503 回帰を潰し、次に同種の背景スレッド初期化失敗が起きたときに CI だけで一次切り分けが完結するようにする:

1. **`tag-generator/Dockerfile.tag-generator` の builder stage に `RUN --mount=type=cache,... /app/.venv/bin/python -m unidic download` を復活** させる。`uv sync` の直後・SBERT モデル事前ダウンロードの直前に置く。download は `.venv` 配下の dicdir に書き込み、runtime stage は既に `/app/.venv` を丸ごと COPY しているので追加のコピー指示は要らない。BuildKit cache mount (`id=unidic-tag-generator`) を使い、dict 実体は cache 層に残してビルド時間を抑える。

2. **`e2e/hurl/tag-generator/run.sh` の `cleanup()` に失敗時ログダンプを追加** する。`cleanup` 先頭で `$?` を受け取り、0 でなければ `tag-generator` / `mq-hub` / `stub-backend` / `redis-streams` の `docker compose logs --tail=200 --no-color` を sed で prefix 付与して stderr に吐く。成功時は従来通り無音でそのまま teardown する。

### 検討した代替案

- **Option B (不採用): `pyproject.toml` `[ml]` から `"unidic"` を削除し `fugashi[unidic-lite]` 単独に倒す**
  - Pros: image を ~770 MB 削減できる。download ステップそのものが要らなくなる。
  - Cons: tagging 精度が unidic-lite の固定語彙に制約される semantic 変更になる。[ml] group が full `unidic` を入れている意図 (語彙網羅性か特定専門用語か) が git 履歴からは読み取れず、単独 ADR で影響範囲を測ったうえで独立に判断すべき。本 ADR のスコープは「回帰を潰して以前の working state に戻す」に絞る。

- **Option C (不採用): `auth_service.py` の blanket `except Exception` を撤去して init 失敗で container crash させ compose healthcheck を unhealthy にする**
  - Pros: silent-503 そのものが構造的に発生しなくなる。observability が根本解決する。
  - Cons: FastAPI lifespan の挙動と `_run_background_tag_generation` の責務分離に踏み込む変更で、本来無関係な「正常時にバックグラウンド処理が一時的に失敗した場合の運用ポリシー」まで決めないと落とし所が決まらない。独立 ADR で扱うべきで、本 ADR の回帰修正とは切り離す。

- **Option D (不採用): `e2e/hurl/_lib/` にログダンプヘルパーを共通化し全サービスの run.sh から呼ぶ**
  - Pros: 他サービスで同種の silent 障害が出たときも同じ経路で診断できる。
  - Cons: 共通化のインタフェース設計 (サービス名リスト取得 / tail 行数 / 出力整形) を先に固めないと api 破壊を後で招く。本 ADR ではまず症例が出た tag-generator だけに入れ、運用で pattern を検証してから横展開する。

## Consequences

### Pros

- tag-generator の E2E が以前の working state に戻り、[[000819]] の image-size 削減と両立できる。
- CI で Hurl が失敗したとき、関係コンテナの Python traceback がそのまま Actions ログに出る。次に同種の「healthy 判定だが実は初期化失敗」が起きても、スタックをローカル再現しなくても一次切り分けできる。
- 修正は builder stage への 2 行追加と `cleanup()` への 10 行追加に留まり、[[000819]] で入れた image-size / CPU-only / smoke-check の不変条件には影響しない。

### Cons / Tradeoffs

- `python -m unidic download` の実行時間 (初回は PyPI からの dict ダウンロード、2 回目以降は BuildKit cache mount から再利用) のぶんビルドがわずかに伸びる。cache miss したときの増分は数十秒オーダーで、image-size gate / pact-publish / e2e の並列時間に隠れる想定。
- full `unidic` + `unidic-lite` が両方 image に残る冗長は未解消。本 ADR は回帰修正のみに限定し、Option B の検討は別 ADR に送る。
- 失敗時ログダンプは tag-generator suite のみ。他サービス (search-indexer / mq-hub / news-creator 等) で同じ silent 障害が起きた場合は今まで通り手元再現が必要。Option D を follow-up で進める。
- `auth_service.py` の blanket `except Exception` は残置。Option C の観測性改善は別 ADR で扱う。
- [[000819]] の smoke check が `Tagger()` インスタンス化までカバーしていなかった検出漏れが露呈したが、本 ADR は smoke 拡張までは踏み込まない。
