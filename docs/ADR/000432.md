# ADR-000432: Knowledge Home の TodayDigest 未要約件数を event 差分で更新する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — `knowledge_projector`, `today_digest_view`, `today_digest_repository`

## Context

Knowledge Home の Phase 1 では、新規記事が `ArticleCreated` によって常時 Home へ流入し、summary 未生成の間も placeholder card として見えることを受け入れ条件に置いた。しかし `TodayDigest` の更新は `new_articles` と `summarized_articles` の加算が中心で、`unsummarized_articles` は差分更新を前提にしていなかった。

その結果、以下のズレが起こり得た。

- `ArticleCreated` で Home に pending item が出ても、TodayBar の pending 件数に反映されない
- `SummaryVersionCreated` が来ても、未要約件数を自然に減算できない
- 再処理や後続 enrich を前提にした event-driven projection なのに、digest だけが snapshot 的な更新契約のまま残る

Knowledge Home の TodayBar は UI 側で推測計算せず、backend projection をそのまま信頼する前提で設計している。このため未要約件数も event と同じ粒度で増減できるようにしないと、「カードは pending なのに digest は 0 件」という整合性欠落が残る。

## Decision

Knowledge Home の `today_digest_view.unsummarized_articles` は、**snapshot 上書きではなく event 差分の加減算で更新する**。

具体的には以下を採用する。

- `ArticleCreated` を投影した時点で `new_articles += 1` と同時に `unsummarized_articles += 1` を行う
- `SummaryVersionCreated` で summary text が取得できて `summary_state=ready` へ進んだ場合のみ、`summarized_articles += 1` と `unsummarized_articles -= 1` を行う
- summary text が空、あるいは取得失敗で `summary_state` が `pending` のままのときは、未要約件数を減算しない
- repository の upsert 契約は `unsummarized_articles = GREATEST(0, current + delta)` とし、負数更新を許容しつつ 0 未満にはしない
- JSON 列の空配列は `null` ではなく `[]` で保存し、PgBouncer 互換と既存テスト前提を維持する

代替案として `today_digest_view` を毎回再集計する案もあったが、Knowledge Home projector は event-driven projection を採用しており、digest だけを集計クエリへ戻すと責務が二重化するため採用しない。

## Consequences

### Pros

- TodayBar の pending 件数が Home card の `summary_state=pending` と整合する
- `ArticleCreated -> SummaryVersionCreated` の自然な遷移を event 差分だけで表現できる
- frontend が pending 件数を独自補正する必要がなくなる
- summary 取得失敗時に誤って pending 件数を減らすことを防げる

### Cons / Tradeoffs

- `today_digest_view` の upsert 契約が単純な上書きより複雑になる
- event の重複投入に対して digest 側も敏感になるため、dedupe policy の一貫性がより重要になる
- 将来 `failed` など追加の summary state を導入する場合は、digest 差分規則も合わせて見直す必要がある
