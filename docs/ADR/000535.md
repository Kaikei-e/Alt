# ADR-000535: Knowledge Sovereign の安全な移行基盤を整備する

## Status

Accepted

## Date

2026-03-23

## Affected Services

- alt-backend (Go)

## Context

[[000534]] で Knowledge Sovereign の境界固定（Phase 0）と write path 集約（Phase 1）が完了した。しかし、Phase 1 の時点では以下の課題が残っていた:

### 1. Mutation dispatch の未完成

Driver の `knowledge_sovereign_repository.go` にある5つの switch case が `return nil` のまま残っていた。`MutationDismissHomeItem`、`MutationClearSupersede`、`MutationDismissCuration`、`MutationSnoozeCandidate`、`MutationDismissCandidate` はペイロードをパースしても実際のリポジトリメソッドに委譲されておらず、Sovereign 経由の書き込みが no-op だった。

### 2. 観測可能性の欠如

Sovereign の mutation がいつ・何回・どの程度の時間で適用されているかを把握する手段がなかった。Phase 3 で切り替え判断をするには、定量的な観測指標が必要だった。

### 3. Reproject が旧パスのまま

`knowledge_reproject_job.go` の `processReprojectBatch` が `projectEvent` を呼ぶ際に `WriteServicePort` として `nil` を渡しており、reproject は常に旧パスで動作していた。shadow run の基盤がなかった。

### 4. 定期的な整合性検証がなかった

projection の健全性を自動で検証する仕組みがなく、drift は手動の `altctl home reproject compare` 実行でしか検出できなかった。

### 5. Replay safety が未保証

mutation が重複適用された場合の挙動が定義されていなかった。upsert 系は SQL の `ON CONFLICT DO UPDATE` で暗黙的に冪等だが、dismiss/snooze 系のポリシーが明文化されていなかった。

### 6. Rollback がバージョンを巻き戻さなかった

`RollbackReproject` は run の status を `cancelled` に遷移するのみで、`ActivateVersion` による前バージョンへの復帰を実行していなかった。

## Decision

Phase 2 として、上記6つの課題を4つの意味的カテゴリに分けて段階的に解決する。

---

### A. Write path の完成 — dispatch 実装と reproject Sovereign 化

**目的**: Sovereign 経由の全 mutation が実際のリポジトリメソッドに委譲されるようにし、reproject もその経路を使えるようにする。

#### A-1. Driver nil stub の完成

`knowledge_sovereign_repository.go` の5箇所を、payload unmarshal + 既存リポジトリ呼び出しに置き換える。新 SQL は書かない。

| MutationType | 委譲先 |
|---|---|
| `MutationDismissHomeItem` | `r.DismissKnowledgeHomeItem(ctx, userID, itemKey, projectionVersion, dismissedAt)` |
| `MutationClearSupersede` | `r.ClearSupersedeState(ctx, userID, itemKey, projectionVersion)` |
| `MutationDismissCuration` | `r.DismissKnowledgeHomeItem(ctx, ...)` （`projection_version` と `dismissed_at` にデフォルト値） |
| `MutationSnoozeCandidate` | `r.SnoozeRecallCandidate(ctx, userID, itemKey, until)` |
| `MutationDismissCandidate` | `r.DismissRecallCandidate(ctx, userID, itemKey)` |

`MutationDismissCuration` は `TrackHomeActionUsecase` から `user_id` と `item_key` のみが送られるため、`projection_version` のデフォルト値 1 と `dismissed_at` の `time.Now()` を driver 層で補完する設計にした。これはレガシーパスの `TrackHomeActionUsecase` が `activeVersionPort` から取得していた値と同等である。

#### A-2. Reproject の Sovereign 経由化

`KnowledgeReprojectJobWithWriteService` 関数を新設し、`processReprojectBatch` の `projectEvent` 呼び出しに `WriteServicePort` を渡す。既存の `KnowledgeReprojectJob` はラッパーとして `nil` を渡す形で後方互換を維持する。

```go
func KnowledgeReprojectJobWithWriteService(
    // ... existing params ...,
    writeService WriteServicePort,
) func(ctx context.Context) error
```

`main.go` の scheduler 配線を `KnowledgeReprojectJobWithWriteService` に変更し、`container.KnowledgeWriteServiceUsecase` を注入する。

---

### B. 観測可能性の確保 — メトリクス、reconciliation、audit

**目的**: Sovereign の状態を定量的に把握し、Phase 3 の切り替え判断に必要なデータを収集する。

#### B-1. Sovereign OTel メトリクスの追加

`KnowledgeHomeMetrics` struct に8つのフィールドを追加する。

| メトリクス名 | 型 | 用途 |
|---|---|---|
| `alt.sovereign.mutation.applied_total` | Counter | mutation 適用回数（type 属性付き） |
| `alt.sovereign.mutation.error_total` | Counter | mutation エラー回数 |
| `alt.sovereign.mutation.duration_ms` | Histogram | mutation 適用時間 |
| `alt.sovereign.reconciliation.run_total` | Counter | reconciliation 実行回数 |
| `alt.sovereign.reconciliation.mismatch_total` | Counter | mismatch 検出回数 |
| `alt.sovereign.reconciliation.duration_seconds` | Histogram | reconciliation 実行時間 |
| `alt.sovereign.idempotency.duplicate_total` | Counter | 重複スキップ回数 |
| `alt.sovereign.cutover_ready` | Gauge | 切り替え条件充足（0 or 1） |

#### B-2. Reconciliation job の新設

`SovereignReconciliationJob` を `sovereign_reconciliation_job.go` に新設する。既存の `CompareProjections` SQL と `DigestAvailabilityReconcileJob` のパターンに準拠する。

処理フロー:
1. `GetActiveVersion` で active version を取得
2. version <= 1 なら前バージョンが存在しないためスキップ
3. `CompareProjections` で active vs active-1 の diff を取得
4. `DefaultReconciliationThresholds`（item count drift 5%、score drift 10%）と比較
5. `ReconciliationReporter` で結果を記録
6. OTel メトリクスを記録

Domain 型として `ReconciliationResult`、`ReconciliationThresholds`、`DefaultReconciliationThresholds()` を定義。Port として `ReconciliationReporter` interface を定義。

`main.go` の scheduler に interval 30分、timeout 2分で登録する。

#### B-3. Audit usecase の検証ロジック実装

既存の `knowledge_audit_usecase` は `MismatchCount: 0` 固定で検証ロジックがなかった。`NewUsecaseWithVerification` constructor と `verifyProjection` メソッドを追加し、item count drift、score drift、empty summary rate drift の3種を検証する。既存の `NewUsecase` はラッパーとして維持する。

---

### C. Replay safety — idempotency の形式化

**目的**: replay / reproject 時の重複適用に対するポリシーを明文化し、mutation 型に dedupe の基盤を組み込む。

`domain/sovereign_idempotency.go` に `BuildIdempotencyKey`、`GetIdempotencyPolicy` を定義する。ポリシーは2種:

| ポリシー | 適用 mutation | 根拠 |
|---|---|---|
| `last_write_wins` | upsert 系（`upsert_home_item`、`upsert_today_digest`、`upsert_recall_candidate`） | SQL が `ON CONFLICT DO UPDATE` で冪等 |
| `skip_if_applied` | dismiss/snooze 系（`dismiss_home_item`、`snooze_candidate`、`dismiss_candidate` 等） | `rows_affected == 0` で重複検知 |

`ProjectionMutation`、`RecallMutation`、`CurationMutation` 各型に `IdempotencyKey string` フィールドを追加する。このフィールドは optional であり、空文字列の場合は idempotency check をスキップする（後方互換）。

---

### D. Rollback safety — バージョン巻き戻しの実装

**目的**: `RollbackReproject` が実際にバージョンを revert するようにする。

既存の `RollbackReproject`（`knowledge_reproject_usecase/usecase.go`）は status を `cancelled` に遷移するのみだった。以下を追加する:

1. `run.FromVersion` を parse して前バージョン番号を取得
2. `activateVersionPort.ActivateVersion(ctx, fromVersion)` で前バージョンをアクティブに復帰
3. その後 status を `cancelled` に遷移

Domain 型として `RollbackPrecondition`（`IsSatisfied()` で `ReconciliationOK && MismatchPct < 0.1` を判定）を追加する。

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|---|---|---|
| 専用テーブルで idempotency key を管理 | Schema 変更が必要 | Phase 2 は schema 変更なし。upsert SQL の冪等性と `rows_affected` で十分 |
| reconciliation を usecase 層に実装 | 定期ジョブの方が自然 | `DigestAvailabilityReconcileJob` パターンとの一貫性。scheduler 登録が容易 |
| reproject 関数シグネチャに `WriteServicePort` を直接追加 | 既存テスト破壊 | `KnowledgeReprojectJobWithWriteService` として新関数を追加し、旧関数をラッパー化 |
| curation mutation に `projection_version` を必須化 | 呼び出し側の変更が必要 | driver 層でデフォルト補完する方が影響範囲が小さい |

## Consequences

### Pros

- **全 mutation が実動作** — 5つの nil stub が解消され、Sovereign 経由の書き込みが全経路で機能する
- **定量的な観測可能性** — 8つの OTel メトリクスで mutation の頻度・エラー・所要時間・drift を把握できる
- **自動 drift 検出** — reconciliation job が30分ごとに projection 健全性を検証し、しきい値超過を検出する
- **replay safety の形式化** — 各 mutation の冪等性ポリシーが明文化され、reproject / backfill 時の挙動が予測可能
- **実効的な rollback** — `RollbackReproject` がバージョン revert を実行し、フォールバックが機能する
- **既存テスト完全保持** — 全パッケージ PASS、既存テスト破壊ゼロ
- **schema 変更なし** — 新テーブル追加・カラム追加は一切なし

### Cons / Tradeoffs

- **reconciliation の CompareProjections SQL 負荷** — 30分間隔で両バージョンの集計クエリを実行する。大量データでは sample-based 比較への切り替えが必要になる可能性がある
- **idempotency key は optional** — 空文字列ならチェックをスキップするため、caller が key を設定し忘れると重複検知が働かない
- **curation dismiss のデフォルト補完** — `projection_version` のデフォルト値 1 は active version と異なる可能性がある。将来的に caller 側での明示設定が望ましい
- **ReconciliationReporter の実装が nil** — scheduler 登録時に `nil` を渡しており、結果の永続化は audit usecase 経由にのみ依存する
