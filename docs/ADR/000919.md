# ADR-000919: act_outcome_cron に BackfillCutoff を追加し event-time purity を invariants test で固定する

## Status

Accepted

## Date

2026-05-25

## Affected Services

- knowledge-sovereign — `app/usecase/act_outcome_cron/cron.go` の `Config` に `BackfillCutoff *time.Time` 追加、`RunBatch` 内の `now := c.cfg.Clock()` を `scanAt` rename + `BackfillCutoff` 経路追加。`app/usecase/act_outcome_cron/invariants_test.go` を新設し 3 件の invariants test を追加。`app/main.go` で `act_outcome_cron.New(...)` に `Clock: time.Now` を明示。

## Context

ADR-000908 §Δ1 で導入された `act_outcome_cron` は KnowledgeLoopActed events を 7 日 window 内で受け、no_engagement outcome を system emit する責務を持つ。reproject-safety の鍵は「emitted event の `OccurredAt` は acted.OccurredAt + window で固定」「cron の wall-clock は cutoff scan の識別子としてのみ使う」という分離だった。

しかし実装は次の弱点を抱えていた:

1. **wall-clock の用途が code 上で曖昧**: `now := c.cfg.Clock()` という変数名から「identifier 用途か business fact 用途か」が読めない。後続の改修で `now` を payload に書き込む regression が混入しても気付けない。
2. **backfill 経路がない**: 運用上「過去の event log に対して act_outcome を再構築したい」というケース (例: 計算ロジック修正後の再 emit) で、現状は cron を停止して clock を巻き戻すしかない。これは production wall-clock を後ろに動かすという危険な運用を要求する。
3. **invariants test が無い**: 「同じ event log を異なる clock で 2 回回したら byte-identical event が出るか」「BackfillCutoff を渡しても payload に wall-clock が漏れないか」が test で固定されていない。projector reproject の最強の安全弁である「同じ event log → 同じ projection」をテストしている cron level の test が無く、将来の改修で event-time purity が壊れても CI で検出できない。
4. **main.go の Clock 配線が暗黙**: `act_outcome_cron.New(repo, slog.Default(), act_outcome_cron.Config{BatchSize: ...})` で `Clock` field を省略している。`New()` 内の default fallback で `time.Now` が入るが、配線の意図が main 側から読めない。

## Decision

次の 4 つを同一 commit で landing する:

1. **`Config.BackfillCutoff *time.Time` 追加**: nil 以外を渡すと `RunBatch` は `Clock()` ではなく `*BackfillCutoff` を scan boundary に使う。emitted event 自体は引き続き acted.OccurredAt + window で生成されるので「backfill を回しても payload には wall-clock が漏れない」を type level で保証。
2. **`now` → `scanAt` rename**: `RunBatch` 冒頭の変数名で「これは scan-cutoff identifier であり business fact ではない」を code 上で明示。
3. **invariants_test.go の 3 件**:
   - `TestBuildNoEngagement_OccurredAt_NeverUsesClock` — `buildNoEngagementOutcome` を直接呼んで `OccurredAt == acted.OccurredAt + window` と `payload.observed_at == acted.OccurredAt + window` を assert。
   - `TestRunBatch_BackfillCutoff_DoesNotLeakWallClockIntoPayload` — 異なる absurd clock (1000年差) を 2 つ用意し、同じ `BackfillCutoff` を渡したときに emitted event の `OccurredAt` / `Payload` が byte-identical であることを assert。
   - `TestRunBatch_SameEventLog_TwoDifferentClocks_ProducesIdenticalEvents` — `BackfillCutoff` なしで異なる clock を渡し、`OccurredAt` / `Payload` / `DedupeKey` / `AggregateID` / `EventType` / `AggregateType` / `ActorID` / `ActorType` がすべて一致することを assert。`EventID` は UUID なので比較から除外。
4. **main.go で `Clock: time.Now` を明示**: 「これは identifier-use のみで payload には漏れない」を表す comment と合わせて配線。

### 検討した代替案

- **BackfillCutoff の代わりに `--start-time` CLI フラグ + cron 再起動**: 採用しなかった。cron は production で常時 running なので「特定 user の backfill だけ運用窓を別に動かす」が in-process 設定で済むほうが運用フリクションが少ない。
- **Clock を `time.Time` 値で固定**: 採用しなかった。production は `time.Now` が必要、test は決定的 clock が必要、backfill は任意の cutoff が必要、と 3 用途が並存するので関数値 + optional pointer が最も自然。
- **invariants test を generic な property-based test に**: 採用しなかった。Go の標準 testing で性質ベースを書くと依存と complexity が増える。3 件の concrete test で重要 invariants は十分カバーできる。

## Consequences

### Pros

- event-time purity (canonical contract §6 / invariant 6 "Event-time purity") が CI 上で fail 検出可能な test として固定される。
- backfill 経路が wall-clock を巻き戻さずに済む。production cron を止めずに過去 event log を再 emit できる。
- `scanAt` rename + `Clock: time.Now` 明示で「wall-clock は identifier-only」という運用前提が code から読めるようになる。
- helpers / cron が pure function に近づくほど reproject 失敗時の bisect 範囲が狭まる。

### Cons / Tradeoffs

- `Config` field が増えるので test fixture / DI 配線で `Config{}` literal を使っている箇所はメンテが要る (今回は main.go 以外には新規 fixture なし)。
- `BackfillCutoff` を nil-pointer optional にしたため、`*time.Time` の nil check が `RunBatch` に 1 箇所増える。zero value sentinel にすると意図が読めなくなるためポインタを採用。
- 3 件の invariants test は cron 全体を 2 回回すため (異なる clock で) 実行時間が線形に増える。`t.Parallel()` を付けて並列化し total test 時間への影響を抑えた。
