# ADR-000933: Knowledge Loop の OODA を「操作を縛るゲート」から Boyd 忠実な記述レンズへ再接地し observe→act / orient→act 高速パスを解禁する

## Status

Accepted

## Date

2026-05-29

## Affected Services

- alt-frontend-sv (TypeScript/Svelte) — `OodaPipeline.svelte` 受動指標化、`+page.svelte` の workspace を Open 常時 primary 化 + stepper 撤去、`LoopEntryTile.svelte` の CTA ステージゲート除去、`loop-transitions.ts` allowlist 同期
- alt-backend (Go) — `classify_transition_event.go`、`job/knowledge_projector.go`、`usecase/knowledge_loop_usecase/transition_knowledge_loop_usecase.go`、`connect/v2/knowledge_loop/handler.go`、`connect/v2/server.go`
- proto — `proto/alt/knowledge/loop/v1/loop_transition_policy.json` (version 2)

## Context

`/loop` の全機能が「コアコンセプトから乖離・欠損し、OODA の本質が捉えられず、ボタンが機能しない」という指摘を起点に、実機診断 (log-seeker, read-only, 2026-05-29) で死因を定量確認した:

- `knowledge_loop_entries` の **約 99.9% が `proposed_stage = observe` 固着**（observe 59,260 行 / act 7 行）。
- フロントは CTA の活性を `canTransition(effectiveStage, to)` で決めており、§7 の線形 matrix が `observe → act` を禁止していたため、**新規 (observe) エントリでは Open / Save が一律 disabled**。記事を開くには tap(observe→orient) → Compare(orient→decide) → Open(decide→act) と 3 手を歩く必要があった。
- 遷移ファネル: `observe→orient` 416 件 → `orient→decide` 108 → `decide→act` 58（orient 到達から act まで **約 86% が脱落**）、`observe→act` は **0 件**。
- `knowledge_events` への append・projector lag・429/400 はいずれも健全。すなわち死因は append 失敗ではなく **UI ゲート + 線形ステートマシン**。

これは振り返り `review/knowledge-loop-retrospective-2026-05-28` が指摘した「3 軸 (proposed × session × surface) を 1 軸に畳む仕様違反」が UI として具現化したものだった。

原典 (Boyd 1996 "The Essence of Winning and Losing" / Chet Richards / Osinga) を当たると、OODA は線形パイプラインではなく **orientation がループ全体を形成する連続フィードバック系**であり、熟練者は **Implicit Guidance & Control** によって explicit な Decide を経ず Observe / Orient から直接 Act に到達する。現実装は Boyd が唯一の図で明確に否定した「pipeline 神話」そのものだった。

加えて immutable データモデル監査で、Knowledge Home projector に業務時刻 `time.Now()` が 6 箇所残り reproject 非決定（event-time purity 違反）、および transition usecase に nil appendPort を黙ってスキップする silent fallback（CLAUDE.md rule 8 違反、PM-2026-045 / [[000928]] の根本原因型）が確認された。

## Decision

以下 3 点を採用する。

**1. OODA transition policy を version 2 化し、Implicit Guidance & Control 高速パスを解禁する。** `loop_transition_policy.json` の `allowed_edges` に `observe → act` と `orient → act` を追加し、`observe → act` を `forbidden_edges` から除去する。これらは禁止ショートカットではなく「熟練者が orientation から直接 commit する正規の高速パス」であり、append-first の誠実性として合成中間遷移で偽装せず `from_stage = observe/orient` の Acted イベントをそのまま記録する。canonical policy は当該 JSON を単一の真実源とし、Go classifier (`classify_transition_event.go`) / FE allowlist (`loop-transitions.ts`) / BFF / Go・TS の conformance test の 5 箇所を同期させる（drift は CI が検出）。

**2. フロントで OODA を「操作を縛るゲート」から「記述レンズ」へ降格する。** stage はユーザー操作の前提条件ではなく system の orientation 推定の表示に徹し、行動は常時 1 ジェスチャで可能とする:
- `OodaPipeline.svelte` を 4 つのクリック可能ステージボタン（クリックの大半が silent no-op だった）から、**受動的な orientation 指標 (role="img")** へ転換。
- `+page.svelte` の workspace を、stage 分岐 (act/decide のみ Open) から **Open を stage 非依存で常時 primary に表示**する単一コマンド面へ集約し、**stepper「次ステージへ進む」ボタンを撤去**。
- `LoopEntryTile.svelte` の CTA から `!isAllowed(toStage)` のステージゲートを除去。Open は best-effort で常にナビゲートし、遷移記録は合法時のみ emit。

**3. バックエンドの不変条件を hardening する。** projector の業務時刻 `time.Now()` 6 箇所を `event.OccurredAt` 起点に置換し reproject を bit-identical にする。transition usecase は producer 経路として nil appendPort で **panic**（silent skip 廃止）、`connect/v2/server.go` 起動時に `alt.knowledge_loop.wiring` で `*_enabled` フラグを loud ログ、stream 未配線も loud Error ログにする。

### 検討した代替案と却下理由

- **OODA を「記述レンズ」ではなく「生きた線形進行」に留める案**（全クリックにフィードバック・最寄り合法ステージへ自動前進）: 死んだボタンは消えるが、Boyd の本質（Decide は occasional、O→A 高速パス）を捉えられず、observe 固着の現実では依然 act まで遠い。却下。
- **ボタンをすべて常時 disable しないが policy は据え置く案**: FE で Open を活性化しても BFF `parseBody` の `canTransition(observe, act)=false` が 400 を返し、活性化したボタンが「押せるが弾かれる」別種の dead button になる。policy 緩和とセットでなければ不可。却下。
- **observe エントリの decision_options 自体に open/save を seed する案**: projector の `STAGE_ALLOWED_INTENTS[OBSERVE]` を変更すると観察段の提案が雑然とする。Open は「提案 (decision_options)」ではなく「普遍的な act affordance」として workspace に置く方が orientation 中心の設計と整合。decision_options は orientation 相応の提案 (revisit/ask/snooze) のまま据え置き、Open を独立配置。採用。
- **projector の nil 業務時刻に time.Now() フォールバックを残す案**: `knowledge_events.occurred_at` は NOT NULL であり、zero は malformed event。wall-clock で糊塗すると reproject 非決定を温存する。フォールバックを削除し event-time のみとする。採用。

## Consequences

### Pros

- 新規 (observe) エントリの Open が **1 ジェスチャで機能する**。実機で全面 disabled だった「死んだボタン」が解消。
- 画面上部の OODA ribbon ボタン群と中央 workspace の stepper が消え、操作系が **Orient 中心の単一コマンド面**に集約（「上部と下部にボタンが複数」の是正）。
- transition matrix が Boyd 原典に忠実化し、3 軸直交 (proposed × session × surface) を UI が尊重するようになった。
- projector reproject が bit-identical 化し event-time purity 不変条件を回復。
- nil 依存が起動ログ + panic で loud になり、DI 配線漏れが 4 週間 silent fail する PM-2026-045 型の再発を構造的に防ぐ。

### Cons / Tradeoffs

- transition policy は 5 箇所同期が前提のため、proto / BE / FE は同一リリースで揃える必要がある（conformance test が drift を検出するが、部分デプロイ時は一時的に 400/429 を生みうる）。
- `OodaPipeline` を受動化したことで「ステージを手で進める」操作は無くなった。stage は行動の帰結としてのみ動く（意図的）。
- foreground tile（非アクティブエントリ）は revisit/ask/snooze の orientation CTA のみで、Open は active entry の workspace に集約。任意エントリの直接 Open は将来の focus 切替設計に委ねる（残課題）。
- transition usecase の panic は producer 配線漏れを前提とした防御であり、テストは実 appendPort 配線が必須になった。
