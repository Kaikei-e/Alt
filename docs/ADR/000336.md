# Connect-RPC ContextCancelInterceptor による context canceled チェックの一括処理

## ADR's STATUS

Accepted

## CONTEXT

ADR335 のエラー伝搬修正で指摘された課題として、`ctx.Err()` チェックがハンドラーコード内に分散している問題があった。

`feeds/handler.go` の `StreamFeedStats` / `sendStatsUpdate` だけで **10箇所** の `ctx.Err()` チェックが存在し、全て同じパターンを繰り返していた:

```go
if err != nil {
    if ctx.Err() != nil {
        h.logger.InfoContext(ctx, "cancelled", ...)   // context cancel -> INFO
    } else {
        h.logger.ErrorContext(ctx, "real error", ...)  // 本当のエラー -> ERROR
    }
    return err
}
```

このパターンには以下の問題がある:

1. **ボイラープレートの増殖** — 同一ロジックがハンドラー内に何度も出現し、可読性を下げる
2. **ログレベル判定の分散** — context キャンセルを INFO / 本物のエラーを ERROR にする判定が各箇所に埋め込まれている
3. **Connect-RPC エラーコード変換の漏れ** — 一部のハンドラーでは `context.DeadlineExceeded` を `CodeDeadlineExceeded` にマッピングしているが、他のハンドラーでは行われていない

Connect-RPC のインターセプターパターン（onion model）を活用すれば、この横断的関心事を一箇所に集約できる。

## DECISION MAKING

### 新規: `ContextCancelInterceptor`

`alt-backend/app/connect/v2/middleware/context_cancel_interceptor.go` に、既存の `AuthInterceptor` / `ServiceAuthInterceptor` と同じパターンでインターセプターを作成する。

#### Unary RPC の処理

ハンドラーからエラーが返された際に `ctx.Err()` を検査し:
- `context.Canceled` → `connect.CodeCanceled` にマッピング、INFO ログ
- `context.DeadlineExceeded` → `connect.CodeDeadlineExceeded` にマッピング、INFO ログ
- `ctx.Err() == nil`（正常コンテキスト） → エラーをそのまま通過（本物のエラー）

#### Streaming RPC の処理

同様に `ctx.Err()` を検査するが、クライアントが既に切断済みのため `nil` を返す（エラーレスポンスを送る相手がいない）。

#### インターセプターチェーンの位置

**最外側**（先頭）に配置する。Connect-RPC の onion model では、先頭のインターセプターがレスポンス時に最後に実行されるため、内側で発生した全てのエラーをキャッチできる。

```go
cancelInterceptor := middleware.NewContextCancelInterceptor(logger)
authInterceptor := middleware.NewAuthInterceptor(logger, cfg)
opts := connect.WithInterceptors(
    cancelInterceptor.Interceptor(), // 最外側: エラー変換
    authInterceptor.Interceptor(),   // 内側: 認証
)
```

ユーザー向け API と内部 API（`BackendInternalService`）の両方に適用する。

### ハンドラーの簡素化

インターセプターに委譲できる `ctx.Err()` チェックを削除し、ハンドラーを簡素化する。

**`feeds/handler.go`** — 7箇所削除:
- `sendStatsUpdate` 内の 3箇所（feedCount / unsummarized / totalArticles 取得時）
- `StreamFeedStats` の 3箇所（init send / update ticker / heartbeat ticker）
- 初回送信時の 1箇所

**`articles/handler.go`** — 1箇所削除:
- `FetchArticleContent` の `context.DeadlineExceeded` → `CodeDeadlineExceeded` マッピング

### 変更しないもの

以下の `ctx.Err()` チェックはインターセプターの対象外として保持する:

| パターン | 理由 |
|---------|------|
| `select { case <-ctx.Done(): return nil }` | ループ / goroutine の終了メカニズム（制御フロー） |
| `streamPreProcessorSummarize` の detached context | `context.Background()` を使った独立コンテキスト管理 |
| Gateway の semaphore wait | インフラレベルの正当な中断処理 |
| Job scheduler / feed_collector / ogp_image_warmer | バックグラウンドジョブの正当なキャンセル |
| REST ハンドラー | Connect-RPC スコープ外（Echo ミドルウェアの管轄） |

## RESULTS, EFFECTS

### PROS

- **ハンドラーコードの簡素化**: 8箇所の冗長な `ctx.Err()` チェックを削除し、ハンドラーはビジネスロジックに集中できる
- **ログレベル判定の一元化**: context キャンセルは必ず INFO で記録され、ログノイズが減る（ADR335 の延長線上の改善）
- **Connect-RPC エラーコードの統一**: 全 RPC で `CodeCanceled` / `CodeDeadlineExceeded` が自動的に返される
- **新規ハンドラーへの自動適用**: 今後追加される Connect-RPC ハンドラーは特別な実装なしに context キャンセルが適切に処理される
- **既存テスト非破壊**: 全テスト (`go test ./...`) がパスし、回帰なし

### CONS, TRADEOFF

- **ハンドラー固有のエラーメッセージ喪失**: 従来は `"feed stats query cancelled"` のように具体的なメッセージをログ出力していたが、インターセプターでは `"RPC cancelled"` + procedure 名に統一される。ただし、procedure 名から十分に特定可能
- **暗黙的な動作**: ハンドラー開発者がインターセプターの存在を認識していないと、context キャンセル時のエラーハンドリングが「勝手に変わる」ように見える可能性がある
- **REST ハンドラーは対象外**: Connect-RPC インターセプターなので、REST 側（Echo ミドルウェア）は別途対応が必要。ただし REST から Connect-RPC への移行が進んでおり、影響は縮小傾向
