# ADR-000918: WhyPayload v2 producer 配線を sovereign + alt-backend に展開し WhyMappingVersion を v11 に bump する

## Status

Accepted

## Date

2026-05-25

## Affected Services

- knowledge-sovereign — `proto/services/sovereign/v1/sovereign.proto` に `counter_evidence_refs` / `confidence_ladder` / `what_would_change_my_mind` の 3 field と `ConfidenceLadder` enum を additive 追加。migration `00021_add_why_v2_columns.sql` で `knowledge_loop_entries` に 3 列 + CHECK 制約 + `knowledge_loop_entries_public` view 再定義。`enricher.go` / `why_override.go` で pure helpers (`boundCounterEvidence` / `confidenceLadderFromKind` / `whatWouldChangeFromKind`) を経由した populate と additive merge。`WhyMappingVersion` を 10→11 に bump。
- alt-backend — `app/domain/knowledge_loop_entry.go` の `WhyPayload` / `KnowledgeLoopEntry` に同 3 field 追加。`sovereign_client/knowledge_loop_client.go` の `domainEntryToProto` / `protoEntryToDomain` で v2 fields を pass-through。`connect/v2/knowledge_loop/handler.go` で sovereign domain → `alt.knowledge.loop.v1.WhyPayload` へ pass-through、`mapConfidenceLadder` 追加。
- alt-frontend-sv — `alt.knowledge.loop.v1.WhyPayload` の proto regen により wire 上の v2 fields を読める状態に (UI 描画自体は ADR-000912 で landed の `WhyTypography` 経由)。

## Context

ADR-000908 §Δ4 は「Why は verifiable claim でなければならない」を canonical contract §11 として明文化した。具体的には次の 3 つを WhyPayload に持たせる必要がある:

1. **counter_evidence_refs** (≤4 cap) — Why narrative を弱める参照リスト。一方通行の trust 広告ではなく反証も提示する。
2. **confidence_ladder** (speculation / pattern / evidence / verified) — projector が割当てる qualitative tier。WhyKind の純関数として決まる。
3. **what_would_change_my_mind** (1..256 char falsifier) — どんな観察があれば Why claim が無効化されるかを 1 文で示す。

PR 1 step 1 (commits e566bfa24 / 8ad59125b) で pure helpers (`boundCounterEvidence` / `confidenceLadderFromKind` / `whatWouldChangeFromKind`) と `ConfidenceLadder` local enum + 8 unit tests を landed したが、proto / migration / DB driver / projector 呼び出し / alt-backend BFF / runbook までは未配線で、現状は wire 上に v2 fields が出ない状態だった。

ground truth の確認:

- `alt.knowledge.loop.v1.WhyPayload` (UI 向け wire 型) は ADR-000912 で landed 済み (field 5/6/7 + `ConfidenceLadder` enum 既存)。
- sovereign 側 `KnowledgeLoopWhyPayload` は v1 fields (1..4) のみ。
- `knowledge_loop_entries` には `why_kind` / `why_text` / `why_confidence` / `why_evidence_ref_ids` / `why_evidence_refs` のみで、v2 列は無し。

つまり「FE と canonical contract は v2 を前提に書かれているのに projector / DB は v1 のまま」というギャップがあり、PR 1 step 2 でこのギャップを閉じる必要があった。

## Decision

WhyMappingVersion を **10 → 11** に bump し、次を一括で配線する:

1. **sovereign proto**: `KnowledgeLoopWhyPayload` に field 5 `counter_evidence_refs` / 6 `confidence_ladder` / 7 `what_would_change_my_mind` を additive、`ConfidenceLadder` enum を `alt.knowledge.loop.v1.ConfidenceLadder` と同値で追加。
2. **migration `00021_add_why_v2_columns.sql`**: `knowledge_loop_entries` に 3 列を `ADD COLUMN` で additive 追加し、cap/value-set CHECK 制約を付ける。`knowledge_loop_entries_public` view を `CREATE OR REPLACE` で末尾追加 (PostgreSQL 仕様により order 変更不可)。
3. **why_v2_helpers.go**: `ConfidenceLadder.ToProto()` method を追加 (local enum → sovereign wire enum)。`populateWhyV2(payload, counter)` と `refreshWhyV2OnKindChange(payload)` を新設し、enricher / override 両 path に共通の populate ロジックを集約。
4. **enricher.go**: 6 つの `enrich*` fn (SummaryVersion / HomeItemsSeen / HomeItemAsked / HomeItemOpened / Superseded / HomeItemDismissed / Fallback) すべてで `populateWhyV2` を呼ぶ。`enrichSuperseded` のみ `previous_summary_version` を counter_evidence_refs に転記。
5. **why_override.go**: 各 override branch (Augur / TopicAffinity / TagTrending) で `refreshWhyV2OnKindChange` を呼び、Kind 変更後に ladder + WWCM を再導出。version_drift branch は enricher 出力をそのまま返すため counter_evidence_refs が保持される。
6. **DB driver (`write_knowledge_loop.go` / `read_knowledge_loop.go`)**: UPSERT と patch-only-why の INSERT / UPDATE / SELECT 句 + struct scan に 3 列を配線。`confidenceLadderToDB` / `confidenceLadderFromDB` 関数を追加。
7. **alt-backend BFF (`sovereign_client/knowledge_loop_client.go`)**: `domainEntryToProto` / `protoEntryToDomain` で v2 fields を双方向 pass-through。`confidenceLadderToProto` / `confidenceLadderFromProto` で domain ↔ sovereign proto enum を変換。
8. **alt-backend connect handler (`connect/v2/knowledge_loop/handler.go`)**: sovereign domain → `alt.knowledge.loop.v1.WhyPayload` 変換で v2 fields を pass-through、`mapConfidenceLadder` で domain → public proto enum。
9. **constants.go**: `WhyMappingVersion = 11` に bump + v11 行を doc string 追加。
10. **runbook**: `docs/runbooks/knowledge-loop-reproject.md` の cutover ladder に v10 → v11 行を追記。

### 検討した代替案

- **local enum ではなく proto enum を helpers に直接持たせる**: 採用しなかった。proto regen 前から helpers と test が独立して動く設計にすると、proto 変更と helper 実装の sequencing 制約が消える。`ToProto()` で wire への mapping を 1 箇所に閉じれば後の改修コストは増えない。
- **`why_text` の中に counter / falsifier を埋め込み structural fields を追加しない**: 採用しなかった。FE は構造化された refs を border-left の色変えや on-demand expand に使う必要があり、自由テキスト埋め込みでは parse の deterministic 性が損なわれる。
- **WhyMappingVersion を bump しない (additive migration なので reproject 不要)**: 採用しなかった。新規 enrich emission は v2 fields を埋めるが、過去の `why_text` のみ持つ row は ladder / WWCM が NULL のままになる。全 entry が consistent に v2 を持つ状態にするには bump + reproject が必要。

## Consequences

### Pros

- Why narrative が verifiable claim (反証 + 根拠 tier + falsifier) を含む構造化データとして wire / DB / UI のすべてで一貫する。
- helpers は pure function なので reproject 安全性が保たれる (同 event → 同 v2 fields)。
- additive migration で既存 row を破壊せず、reproject 後に converge する設計。
- alt.knowledge.loop.v1 proto は ADR-000912 で landed 済の field を使うため UI 側は新規 wire 変更なしで v2 fields を読める。

### Cons / Tradeoffs

- UPSERT 句の placeholder が 31 → 34 に増え、test fixture の `make([]interface{}, N)` 数値を全更新する必要があった (`write_knowledge_loop_test.go` 3 箇所 + patch-only test 1 箇所)。次に列を増やすとき同じ更新が必要。
- local enum (`ConfidenceLadder`) と wire enum (`sovereignv1.ConfidenceLadder`) が並存するため `ToProto()` の同期メンテが恒久的に発生する。両者を統一する case には別 ADR が必要。
- `knowledge_loop_entries_public` view は PostgreSQL の `CREATE OR REPLACE VIEW` 制約により order 変更不可。新 v2 列は末尾に append される。read 側は名前で SELECT しているので影響なし。
- recap-worker が `persist_stage_confidence_ladder` を実 emit するまでは ladder は WhyKind 派生の coarse 値のみ。calibrated uncertainty grounding は ADR-000919 (受信側) / 別 PR (producer 側) で段階的に強化する。
