# ADR-000658: フィードソース除外フィルタをドメインレベルに拡張

## Status

Accepted

## Date

2026-04-09

## Affected Services

- alt-backend (Go) — Connect-RPC ハンドラ、Clean Architecture 全層
- alt-frontend-sv (TypeScript/SvelteKit) — Exclude UI コンポーネント、API クライアント
- alt-butterfly-facade (Go) — Proto 変更に伴う再ビルド（透過プロキシのため実装変更なし）

## Context

[[000256]] で導入したフィードソース除外フィルタは、単一の `feed_link_id`（1つの RSS URL）のみを除外する設計だった。ADR-256 にも「v1 は単一ソース除外のみ（複数除外は `repeated string` への拡張が必要）」と明記されていた。

### 問題

dev.to のように複数のタグフィード URL を持つソース（`dev.to/feed/tag/go`, `dev.to/feed/tag/rust` 等、10 件以上）では、ユーザーが 1 つの dev.to ソースを除外しても、残り 9+ の dev.to URL からのフィードが引き続き表示される。UI チップは `extractDomain()` によりドメイン名「dev.to」を表示するため、ユーザーは全 dev.to フィードが除外されたと誤認する。

### 調査結果

- `feed_links` テーブルには dev.to だけで 10+ エントリが存在（tag/go, tag/rust, tag/postgresql 等）
- Proto は `optional string exclude_feed_link_id`（単数）のみ対応
- SQL は `AND f.feed_link_id != $N` で単一 UUID 比較
- BFF キャッシュは body hash をキーに含むため、exclude パラメータ変更時は正しく miss する（キャッシュ問題ではない）

## Decision

**Proto に `repeated string exclude_feed_link_ids` フィールドを追加し、フロントエンドでソースをドメイン単位でグルーピングする。** ドメイン選択時に同ドメインの全 `feed_link_id` を送信する。

### Proto 拡張

`GetUnreadFeedsRequest` と `GetAllFeedsRequest` にそれぞれ `repeated string exclude_feed_link_ids` を新規フィールドとして追加。旧 `exclude_feed_link_id` は後方互換のため残す。

### バックエンド: 全層シグネチャ変更

Clean Architecture 各層のパラメータを `*uuid.UUID` から `[]uuid.UUID` に変更。

```
Handler (parseExcludeFeedLinkIDs) → Usecase → Port → Gateway → Driver (buildExcludeClauseMultiple)
```

Handler に `parseExcludeFeedLinkIDs` ヘルパーを導入。repeated field を優先し、未設定時は旧 single field にフォールバック。上限 50 件制限。

### SQL: PostgreSQL 配列比較

```go
func buildExcludeClauseMultiple(args []any, excludeFeedLinkIDs []uuid.UUID) (string, []any) {
    if len(excludeFeedLinkIDs) == 0 {
        return "", args
    }
    clause := fmt.Sprintf(`AND f.feed_link_id != ALL($%d::uuid[])`, len(args)+1)
    args = append(args, excludeFeedLinkIDs)
    return clause, args
}
```

`!= ALL($N::uuid[])` は PostgreSQL の配列比較演算子で、小さな配列（10-15 要素）では UUID 直接比較と同等の性能。[[000256]] で不採用とした `NOT IN` サブクエリ（131ms）とは異なり、リテラル配列との比較のため高速。

### フロントエンド: ドメイングルーピング

`feed-source-filter.ts` に 3 つのユーティリティを追加:

- `getEffectiveDomain(url)` — サブドメイン（feeds., rss., www.）を除去した実効ドメインを返す
- `groupSourcesByDomain(sources)` — ソースをドメイン単位でグルーピング
- `collectFeedLinkIdsByDomain(sources, domain)` — 指定ドメインの全 `feed_link_id` を収集

Exclude フィルタの UI（デスクトップ・モバイル・スワイプの全 3 コンポーネント + Visual Preview）を更新:
- ドロップダウンに個別 URL ではなくドメイン名を表示（複数フィードの場合 `(N feeds)` 表記）
- ドメイン選択時に同ドメインの全 ID を `repeated` フィールドで送信
- Props を `excludedSourceId: string | null` から `excludedFeedLinkIds: string[]` に変更

### 検討した代替案

| 案 | 不採用理由 |
|---|---|
| バックエンド側でドメイン解決（1 ID から同ドメインの全 ID を検索） | SQL にビジネスロジックが入る。「no business logic in SQL」の方針に反する |
| クライアントサイドのみで除外 | カーソルベースページネーションと併用不可（[[000256]] で検証済み） |
| 旧 single field のまま複数リクエスト | Proto 設計として不自然。パフォーマンスも N 回のリクエストが必要 |

## Consequences

### Pros

- dev.to のような複数フィード URL を持つソースを 1 操作で完全除外可能
- ドメイン名表示により UX の一貫性が向上（チップ表示と実際の除外範囲が一致）
- `repeated` フィールドにより将来の複数ドメイン除外にも拡張可能
- 旧 `exclude_feed_link_id` フィールド残存により後方互換性を維持
- PostgreSQL `!= ALL(uuid[])` によりパフォーマンス影響は軽微

### Cons / Tradeoffs

- `getEffectiveDomain` のサブドメイン除去は単純なヒューリスティック（末尾 2 パート）。`co.uk` 等の 2 段階 TLD には対応していない（現状の購読ソースには該当なし）
- Proto に deprecated フィールドが残る（旧クライアントの互換性のため必要）
- 全 Exclude UI コンポーネント（デスクトップ・モバイル・スワイプ）の Props 型が変更され、呼び出し元の修正が必要だった

### テスト結果

- バックエンド: `go test ./...` — 全パス（`buildExcludeClauseMultiple` 5 テストケース含む）
- フロントエンド: `bun test` — Exclude 関連 103 テスト全パス
