# ADR-000664: Executive Summary を最後に再合成し accepted claims からのみ構成する

## Status

Accepted

## Date

2026-04-10

## Affected Services

- acolyte-orchestrator (Python 3.14+)

## Context

[[000659]] で `SectionPlannerNode` と `ClaimPlan` を導入し、[[000661]] で `section_citations` を claim 単位で永続化し、[[000662]] で Conclusion を専用 contract に分離した。しかし、**Executive Summary (ES) が他セクションと同じタイミングで `extracted_facts` から生成されている**問題が残っていた。

具体的な品質問題:

1. **ES に引用がない**: 品質評価レポートで ES の citation 数が 0。ES が raw retrieval から独立に書かれているため、evidence との構造的な紐付けがない
2. **ES が raw evidence から直接生成**: `section_role == "executive_summary"` にもかかわらず、SectionPlannerNode では analysis/general と同一パス（`extracted_facts` ベース）で claims を生成していた
3. **処理順序の問題**: `_default_fallback_sections()` は `[ES, analysis, conclusion]` 順であり、ES が先頭で処理されるため、analysis や conclusion の accepted claims を参照できなかった
4. **ES 固有の制約がない**: WriterNode は ES に対しても標準の `CLAIM_WRITER_PROMPT` を使用しており、「各セクションの主要な発見の要約のみ」「新事実追加禁止」といった ES 固有のルールが構造的に表現されていなかった

学術的・実践的背景:

- **Anthropic long-context tips**: 長文タスクではまず relevant quotes を抽出させ、その後に回答させるとよい。ES も raw search から書くのではなく、各 section で accept 済みの strongest claims を pull-up して作る方が安定する
- **Attribute First, then Generate** (ACL 2024): content selection → sentence planning → generation の 3 段分離で、summary は最終段で構成すべき

## Decision

**SectionPlannerNode と WriterNode の処理順序を変更し、ES を最後に処理する。ES は raw evidence ではなく全セクションの accepted claims から synthesis claims を生成する。**

[[000662]] の Conclusion 分離パターン（`_collect_analysis_claims` + 専用 prompt + ループ内分岐）をそのまま ES に拡張する。アーキテクチャ変更・ノード追加・グラフ配線変更なし。

### 主要な設計判断

#### 1. 処理順序の変更: ES をループの最後に配置

SectionPlannerNode と WriterNode の両方で、`outline` を「非 ES セクション → ES セクション」の順に並べ替えてからループ処理する。

```python
non_es = [s for s in outline if s.get("section_role") != "executive_summary"]
es_sections = [s for s in outline if s.get("section_role") == "executive_summary"]
for section in non_es + es_sections:
    ...
```

`outline` 自体は変更しない（display ordering は planner の責務）。processing ordering のみをノード内で制御する。

この方式は Conclusion の処理順序保証と同一パターンである。Conclusion は `_default_fallback_sections` で analysis より後に配置されることで analysis claims を参照可能にしていた。ES は outline 上で先頭に来うるため、明示的な並べ替えが必要。

#### 2. SectionPlannerNode: ES は全セクションの accepted claims を入力とする

`_collect_all_accepted_claims(claim_plans, outline)` を追加。`_collect_analysis_claims` が analysis セクションのみを対象とするのに対し、こちらは ES 以外の全セクション（analysis + conclusion + general）の claims を収集する。

`section_role == "executive_summary"` の場合:
- `_collect_all_accepted_claims` で全セクションの claims を収集
- `ES_PLANNER_PROMPT` で synthesis claims を生成（各セクションから 1-2 件の strongest claims を選択）
- 数値データを含む claims を優先するよう prompt で指示
- accepted claims がない場合は空 claims にフォールバック（LLM 呼び出しスキップ）

#### 3. WriterNode: ES に専用 prompt

`ES_WRITER_PROMPT` を追加。Conclusion の `CONCLUSION_WRITER_PROMPT` と同一パターンで、ES 固有の制約を構造的に表現する:

- 新事実追加禁止
- 各セクションの主要な発見の要約のみ
- 簡潔に 3-5 文
- 数値データを最低 1 つ含める
- claims ブロックのヘッダーを「主要な発見:」に変更

#### 4. Pydantic モデル・Critic・FailureMode は追加しない

`ClaimPlannerOutput` を ES planner でもそのまま再利用する。ES の品質は構造的に保証される（accepted claims のみが入力となるため、raw evidence への直接参照が不可能）。FM 追加は不要。

### 不採用とした代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| ES 専用ノードを LangGraph に追加 | グラフ配線が複雑化。revision ループとの整合性が煩雑。ノード内分岐で十分 |
| outline の順序を planner で固定（ES を最後に） | display ordering と processing ordering の責務が混在する。planner は display ordering のみ担当すべき |
| ES を conclusion の後に配置するよう `_default_fallback_sections` を変更 | 非 fallback パスで LLM が ES を先頭に配置した場合に対応できない |
| ES 用の専用 Pydantic モデルを追加 | `ClaimPlannerOutput` で十分。制約はプロンプトで表現 |

### ES 再合成の動作確認

1. **SectionPlannerNode**: `outline` を非 ES → ES の順に並べ替え。analysis/conclusion を先に処理し、ES は `_collect_all_accepted_claims()` で全セクションの claims を収集
2. **SectionPlannerNode**: `section_role == "executive_summary"` → `ES_PLANNER_PROMPT` で synthesis claims 生成。accepted claims がなければスキップ
3. **WriterNode**: `outline` を非 ES → ES の順に並べ替え。ES は `ES_WRITER_PROMPT` で「主要な発見:」ヘッダー付きの要旨を生成
4. **WriterNode**: ES に対しても `_assemble_citations()` が実行され、`section_citations` が生成される
5. **FinalizerNode**: ES の citations が DB に永続化される

## Consequences

### Pros

- **ES が accepted claims からのみ構成**: raw evidence への直接参照が構造的に不可能になり、ES の citation 数が 0 にならない
- **処理順序の保証**: ES が outline 上のどこに配置されても、SectionPlannerNode/WriterNode で最後に処理される
- **Conclusion パターンの自然な拡張**: [[000662]] で確立した分岐パターン（専用 prompt + claims 入力制約）をそのまま再利用。学習コストが低い
- **LLM コスト追加 0**: ES の claim planning は既存の N sections 呼び出しに含まれる
- **既存パイプライン構造の維持**: LangGraph のノード追加やエッジ変更なし
- **テスト充実**: 8 テスト追加で ES 処理順序・prompt 分岐・integration をカバー

### Cons / Tradeoffs

- **WriterNode の分岐が 4 パスに拡大**: claim-based (general) / claim-based (conclusion) / claim-based (ES) / legacy evidence-based。[[000662]] で予告されていた通り、将来的にはストラテジーパターンへの分離を検討する余地がある
- **SectionPlannerNode も 3 パス**: standard facts / conclusion synthesis / ES summary。同様にストラテジーパターン化の候補
- **`_collect_all_accepted_claims` は全セクション分の claims を結合**: セクション数が増えると prompt の context window を圧迫する可能性がある。ただし現状 3-5 sections × 3-7 claims = 最大 35 claims 程度であり、Gemma4 26B の context window (8K) に十分収まる
- **ES の品質は claims の品質に依存**: analysis/conclusion の claims が低品質な場合、ES も低品質になる。ただしこれは「garbage in, garbage out」であり、ES 固有の問題ではない
- **RerunSectionUsecase との非統合**: `RerunSectionUsecase` は legacy の `WRITER_PROMPT` を使用し、ES 専用 prompt を使わない。ES のセクション単位再生成では品質制約が適用されない（Conclusion と同一の既知制約、[[000662]] 参照）
