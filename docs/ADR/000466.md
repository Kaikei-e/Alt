# ADR-000466: 要約本文を取得できた場合にのみ summary_completed を付与する

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-backend (Go)

## Context

[[000430]] により Knowledge Home の `why_json` は code 単位で保持され、`summary_completed` と `tag_hotspot` など複数の reason を同居させられるようになった。一方で `projectSummaryVersionCreated` は、summary version の取得成否や本文の有無に関係なく `summary_completed` を常に追加していた。

この挙動は `why_json` の意味を曖昧にする。summary text の取得が失敗したり、取得できても空文字だった場合、read model 上では `summary_state = pending` のままなのに `summary_completed` だけが永続化される。repository 側の merge-safe upsert は why reason を削除せず保持するため、一度誤って付いた `summary_completed` は後続イベントでも残り続ける。

結果として UI では「要約完了」を示す badge と、未完了を示す skeleton が同時に見える。これは表示側の問題ではなく、`why_json` の semantic contract が `summary_state` と一致していないことが原因である。

## Decision

**`summary_completed` は、`SummaryVersionCreated` を受けた時点で summary excerpt を実際に生成できた場合にのみ付与する。**

具体的な規則は以下の通りとする。

- `GetSummaryVersionByID` が成功し、`SummaryText` から非空の excerpt を生成できた場合のみ `WhySummaryCompleted` を追加する
- summary 取得エラー時は `summary_completed` を付与しない
- summary text が空文字で `summary_state = pending` の場合も `summary_completed` を付与しない
- `why_json` は「その item が今ユーザーにどう見えているか」の説明可能性を担うため、到達していない状態を先回りして表現しない

この変更により、`summary_state` と `why_json` の意味づけを揃える。`summary_state` が処理到達状態、`summary_completed` が可視的に要約済みである理由、という役割分担を明確にする。

### 検討した代替案

1. **`SummaryVersionCreated` が発行された時点で完了扱いにする**
   event の存在だけでは excerpt の可視化成功を保証できず、read model の見え方とズレるため採用しない。
2. **frontend で `summary_completed` を無視し `summary_state` だけ見る**
   why surfaced の意味を UI ごとに変えることになり、canonical contract を壊すため採用しない。
3. **要約取得失敗時に `summary_completed` を後から削除する**
   why reason の削除ルールを別途持ち込むより、誤付与を最初から防ぐ方が単純で安全なため採用しない。

## Consequences

### Pros

- `summary_state` と `why_json` が同じ事実を指すようになり、カード表示の矛盾が減る
- `summary_completed` が「event を受けた」ではなく「要約が可視化できた」を意味するようになる
- projector テストで空要約・取得失敗ケースを明示的に固定できる
- reproject 時も誤った `summary_completed` を再生成しにくくなる

### Cons / Tradeoffs

- summary backend の一時エラー中は、実際には後で取得可能な要約でも一旦 `summary_completed` が付かない
- why surfaced を「処理予定」ではなく「現在の可視結果」に寄せるため、将来 `summarizing` のような別 reason が必要になれば追加設計が必要になる
- 既に誤付与されている `summary_completed` は reproject などで再構築しない限り残る
