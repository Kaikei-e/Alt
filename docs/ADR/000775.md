# ADR-000775: Acolyte レポートの引用書式を [Sn] マーカー + Sources フッターに統一し、インライン汚染を構造的に排除する

## Status

Accepted

## Date

2026-04-19

## Affected Services

- **acolyte-orchestrator** — `usecase/graph/nodes/finalizer_node.py` から `resolve_citations` ([S1]→[title] 展開) を削除し、`render_sources_footer` で本文中の [Sn] マーカーを温存したまま末尾に `Sources:` ブロックを追加。`usecase/graph/nodes/writer_node.py` の 3 段落プロンプト (analysis / conclusion / executive_summary) と旧 `WRITER_PROMPT` から `[1], [2]` 指示を削り、`[S1], [S2]` 限定のルールに置換。新規 `acolyte/domain/citation_format.py::validate_citation_format` を `_generate_paragraph` 出力に適用し、違反時は status=rejected + `citation_format_violation` feedback で既存リビジョンループに戻す
- **acolyte-orchestrator (tests)** — `tests/unit/test_finalizer_citation_format.py` 新規 (9 ケース)、`tests/unit/test_citation_format.py` 新規 (10 ケース)、`tests/unit/test_writer_paragraph.py` に 3 ケース追加、`tests/e2e/test_citation_hygiene_e2e.py` 新規 (writer→finalizer pipeline slice で再生ループ + 永続化 body の footer を検証)

## Context

2026-04-19 にユーザから報告された `reports/eb67863f-f6f0-48a4-bd76-a9f1d9a79054` を含む複数レポートで、本文中に記事タイトル + ソース + タグを pipe 区切りで並べたブラケットが文中に差し込まれ、読みを阻害していた。具体例:

```
具体的な状況に関する議論は、「2026年の中東 緊迫するイランとガザ停戦の今後は？」
（[山梨 山林火災 延焼範囲が西側に拡大 約180ha焼ける | NHKニュース | 火災、山梨県]）
などとなっている
```

引用する対象と本文の文脈も一致していないケースがあり、RAG としてそもそも信頼されない出力になっていた。

根本原因の切り分け:

- `usecase/graph/nodes/finalizer_node.py:23-33` の `resolve_citations(body, source_map)` が、Writer が出した `[S1]` 短 ID を **finalizer 段階で `[entry.title]` に書き戻していた**。これは [[000700]] で導入した SourceMap (UUID→`S1, S2, ...`) の本来意図 — Writer に短 ID だけを見せて安定性を確保する — と真っ向から反対方向の挙動。
- Writer 側のプロンプト (`PARAGRAPH_WRITER_PROMPT` / `CONCLUSION_PARAGRAPH_PROMPT` / `ES_PARAGRAPH_PROMPT`) は「参考記事番号を `[1], [2]` のように付記する」という旧式の指示のままだった。LLM は数字だけの `[1]` を出したり、自己判断でタイトルっぽいものをブラケットに書いたりしていた。
- 後処理のバリデーションが無く、汚染された本文がそのまま永続化され、フロント側で矯正するのも不可能だった。

Web リサーチ (Perplexity / Anthropic Citations API / OpenAI Responses API の file_search / Microsoft Azure "agentic retrieval") ではいずれも **番号マーカーを本文に残し、タイトル・URL などのメタデータは out-of-band に配置する**のが標準パターンで、本文に長いブラケットを inlining させない。Alt ではすでに `section_citations` を DB に別カラムで持っているので、メタデータ側の土台は揃っていた。必要だったのは (a) finalizer の展開を止めること、(b) Writer が `[Sn]` 以外を出さないように矯正すること、(c) 永続化前の最終防衛 (バリデータ) の 3 点。

本 ADR のスコープは transport 層より上の出力整形のみ。Gatherer / Curator / Critic など上流のフローには触らず、イミュータブル不変条件 (`knowledge_events`, projector, `summary_versions`, `tag_set_versions`) への影響も無い。

## Decision

引用は **本文中には `[Sn]` マーカーのみ、メタデータは末尾 Sources フッター + 既存 `section_citations` カラム**に置くパターンに統一する。

1. **Finalizer footer 化** — `finalizer_node.py` から `resolve_citations` / `_SHORT_ID_RE` の展開ロジックを削除し、新関数 `render_sources_footer(body, source_map)` に置換する。
   - 本文中の `\[S(\d+)\]` を初出順でスキャンし、参照された短 ID だけを footer に列挙する。未参照の `SourceMap` エントリは出さない。
   - フッタ形式: `\n\n---\nSources:\n- [Sn] {title} — {publisher} ({url})`。publisher / url が欠けている場合は ` — ` や `()` を出さず、形式崩れを避ける。
   - 参照されている短 ID が 1 つも無い場合はフッタを付けない (body をそのまま返す)。未知の短 ID (`[S99]` でも `SourceMap` に居ない) は本文中にそのまま残し、フッタにだけ出さない。
   - チェックポイント再開互換: 既存 LangGraph state の `source_map` シリアライズ形式は維持したままで、finalizer だけの挙動変更で済む。

2. **Writer プロンプト硬化** — 3 段落プロンプトと旧 `WRITER_PROMPT` (RerunSectionUsecase から import される現役コード) のルール行を、共通の `_CITATION_RULE` 定数に差し替える:
   ```
   - 出典は必ず [S1], [S2] の形式のみで本文中に記す。記事タイトル・URL・サイト名・タグを本文中に書いてはならない
   ```

3. **LLM 出力バリデータ** — 新規 `acolyte/domain/citation_format.py` に `validate_citation_format(body) -> (ok: bool, reason: str)` を置く。
   - `\[(?!S\d+\])[^\[\]]+\]` で `[Sn]` 以外のあらゆるブラケットを弾く (`[山梨 山林火災 …]` でも `[1]` でも同じ扱い)。
   - `https?://\S+` で生 URL を弾く (URL はフッタに移すべき情報)。
   - Writer の `_generate_paragraph` は LLM 応答に対しこのバリデータを呼び、fail なら `body=""` に潰し `status="rejected"`, `revision_feedback="citation_format_violation: {snippet}"` を詰めて返す。既存のリビジョンループが自動的に再生成する。

4. **E2E ガード** — `tests/e2e/test_citation_hygiene_e2e.py` で writer→finalizer pipeline slice を組み、(a) 汚染出力が rejected 化されること、(b) クリーン出力は永続化 body に `[S1]`/`[S2]` とフッタが揃うこと、(c) finalizer が短 ID をタイトルに絶対展開しないことの 3 点を固定する。UUID 時代の展開テスト (`test_finalizer_citation_resolve.py`) は「バグを固定化する」ため中身を削除し、再導入を抑止する marker ファイルとして残す。

### 検討した代替案

| 案 | 採用 | 却下理由 |
|---|---|---|
| **A: [Sn] を本文に残し Sources フッタに出す**（採用） | ○ | Perplexity / Anthropic / OpenAI の標準パターンと揃う。既存 `section_citations` との重複なくメタデータが 1 箇所 (footer + DB) に集約する。finalizer だけの変更で済みチェックポイント互換を壊さない |
| B: `[Sn]` を本文のまま保持し、UI 側 (BFF / FE) でのみメタデータ展開 | × | 短期的には綺麗だが、永続化された body が "fragment" になり PDF / メール連携など UI 外の利用パスで意味不明になる。footer なら本文コピーで完結する |
| C: Finalizer で `[S1]` → `[1]` に数値リナンバリング | × | Perplexity 風の見栄えにはなるが再生成・再開時の安定性が失われ (claim 再配置で番号がずれる)、SourceMap の UUID 安定化 ([[000700]]) の意図を壊す |
| D: プロンプト硬化のみ、後処理バリデータは入れない | × | LLM が確率的に再発させる。Gemma4 26B でも `think=false` + 数字主体のカラム混入で 数 % 違反が残り、再現性の取れない回帰経路を残すと判断 |
| E: Finalizer の `resolve_citations` を残しつつ footer も追加する (両掛け) | × | 本文 + footer の二重汚染になり、前者のバグが表にも裏にも出て読みやすさが戻らない |

## Consequences

### Pros

- ユーザ報告の「インラインに長タイトルが挟まる」症状が、サンプルレポートを再生成した時点で消える。セクション body は `[Sn]` の短い参照だけになり、可読性が復活する。
- LLM が他形式の引用 (`[1]`, `[記事タイトル]`, 生 URL) を出しても、writer の再生成ループに押し戻されるため、プロンプトの厳密さに依存しない構造的ガードが効く。
- 既存 `section_citations` (DB) + Sources フッタ (body 末尾) の 2 箇所にメタデータが残り、FE / BFF / 将来の PDF 出力のどの経路でも出典情報を失わない。
- SourceMap (UUID→`Sn`) の本来意図が 1 本道になり、[[000700]] の Phase 4 設計と実装挙動がようやく一致する。
- pipeline-slice e2e (`test_citation_hygiene_e2e.py`) が citation format 回帰を CI で常時検出する。

### Cons / Tradeoffs

- 本文中の `[S1]` という見た目はやや機械的で、`[1]` のような自然な番号に慣れた読者には違和感がある。後続 ADR で FE 側に `[S1]` → `[1]` のレンダ層変換を入れる余地を残すが、永続化形式自体は `Sn` で固定する方針。
- バリデータの正規表現が保守的 (`[^\[\]]+`) のため、稀に正当な半角ブラケット表現 (例: 固有名 `[追記]`) も弾く。現状の業務レポートで該当するユースケースは無いが、将来必要になれば allow-list を拡張する。
- 旧 `test_finalizer_citation_resolve.py` を空ファイルで残置しているため、Ruff の `S101` 等の設定によっては「中身のないテストモジュール」警告が出る可能性がある。現状の pyproject 設定では問題なし。
- LLM が何度でも format を間違える病的ケースでは `best_sections` フォールバックに頼るしかない。段落粒度で再生成回数の上限は既存のリビジョン上限に従い、本 ADR では上限値の変更は行わない。
- Critic 側には未適用。Critic も将来的に `citation_format.py` を再利用することを想定しており、インポートパスを `acolyte/domain/` に置いたが、Critic プロンプトへの編み込みは別 ADR のスコープ。
