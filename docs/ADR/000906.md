# ADR-000906: knowledge-sovereign projector が `knowledge_loop.acted.v1` を consume し continue_context を event-bound に再構築する

## Status

Accepted

## Date

2026-05-20

## Affected Services

- knowledge-sovereign (Go)

## Context

[[000855]] / [[knowledge-loop-completion-02-semantic-decide-act]] で BFF (`alt-frontend-sv/src/routes/(app)/loop/transition/+server.ts`) と alt-backend handler (`app/connect/v2/knowledge_loop/handler.go`) は `actedIntent` / `targetType` / `targetRef` / `continueFlag` を accept し event log に append している。Phase 2 完成計画は projector がこれらの semantic metadata を consume して `continue_context` を更新することを要求しているが、**projector 側の consume 処理がまだ実装されていない** (2026-05-20 backend audit: Phase 2 完成度 70%、projector consume 0%)。

その結果:

- `continue_context.recent_action_labels` が常に空配列のままで、Continue plane の "Open · Ask · Compare" 連鎖が表示できない
- v2 Surface Planner の `recent_continue_action_count` 入力が立たず、`continue_flag=true` の transition が Continue bucket placement に寄与しない
- ADR-000904 で `advanceEntry` が semantic metadata を必ず送るようにしても、projector が受けないので effect が出ない (片肺状態)

projector が UPDATE する `continue_context` は **read model = disposable projection** であるため、ロジックは event payload + versioned artifact のみから決定論的に再構築できる必要がある (canonical contract §3 reproject-safe invariant)。

## Decision

projector に `KnowledgeLoopActed` handler を追加し、`continue_context` を event-bound に再構築する。新規 helper を pure function として切り出し、reproject 経路と新規 event 受信経路で同一ロジックを共有する。

1. **`continue_context_builder.go` (新規)** — pure function `BuildContinueContext(actedEvents []KnowledgeLoopActedPayload, window time.Duration) ContinueContext` を export。引数の event 配列のみから `recent_action_labels` (最大 5 件、新しい順)、`last_interacted_at` (`MAX(event.occurred_at)`)、`summary` (action labels の自然言語要約) を決定論的に生成する。`time.Now()` は使わない (event-time purity)。
2. **`projector.go` の `Project(event)` switch 拡張** — `EventKnowledgeLoopActed` case を追加:
   - event payload を decode し `acted_intent` / `target_type` / `target_ref` / `continue_flag` を取り出す
   - target row の `(user_id, lens_mode_id, entry_key)` に対して、過去 7 日分の `KnowledgeLoopActed` events (含む現 event) を read-only query で取り出す (event log のみ、projection を読まない)
   - `BuildContinueContext(events, 7*24h)` を call し row の `continue_context` JSONB を upsert
   - `target_type=recap` の event は `act_targets[]` に recap target を append (Review / Continue で再表示可能にする)
   - `acted_intent=compare` または `target_type=diff` の event は `change_summary.compare_count++` (Changed bucket の強化用)
3. **`score_resolver_event_log.go` 拡張** — 既存 `EventLogSurfaceScoreResolver` の `RecentContinueActionCount` 計算で `continue_flag=true` の event のみカウント (current code は全 `KnowledgeLoopActed` を等価扱いしている)。
4. **`recent_action_labels` bounded list** — 最大 5 件、`(intent, target_type)` の重複は最新 1 件のみ保持 (例: `Open / Ask / Open Recap` の連続 Open は 1 行)。labels の正規化は frontend 側 `recent-action-label-map.ts` が担い、backend は event payload の値をそのまま渡す (`opened` / `asked` / `revisited` / `compared`)。
5. **Test matrix** — `projector_test.go` に table-driven test を追加:
   - 各 `acted_intent` (ask / open / revisit / compare / save / snooze) で `recent_action_labels` の更新を assert
   - `continue_flag=true` のみが Continue signal にカウントされる
   - 同一 entry の event 6 件目で bounded list の最古が drop
   - reproject: 同じ event log から 2 回 projection を rebuild すると **完全に同じ** `continue_context` が再現
   - `target_type=recap` で `act_targets[].target_type=recap` が保持される
6. **Pact CDC** — `alt-backend → knowledge-sovereign` (Acted forwarder) と `alt-frontend-sv → alt-backend` (`/loop/transition`) で `continueFlag` / `targetType=conversation/recap/diff/entry` の matrix を pact で pin。既存 pact ファイルを additive で更新。

### 検討した代替案

- **`continue_context` を write path で直接 update する** — `TransitionKnowledgeLoop` RPC handler 内で projection row を直書きする選択肢があるが、これは **append-first invariant を破る** (write path から projection を直接 mutate)。reproject-safe を維持できないので不採用 (canonical contract §3 / [[000840]] / [[000844]])。
- **`recent_action_labels` を current state 参照で算出** — 「最新 5 件」を `SELECT ... ORDER BY occurred_at DESC LIMIT 5 FROM knowledge_loop_acted_events` で都度引く案。reproject-safe だが N+1 query で projector が遅延する。pure function で event 配列を渡す形にし caller (projector) が 1 query でまとめ取りする方が決定論性 + 性能両立。
- **labels を `(intent)` だけで dedupe** — `target_type` 違いの open / ask が同じ label になり情報を失う。`(intent, target_type)` で dedupe する。
- **LLM で recent_action narrative を生成** — Phase 3 完成計画 §「実装しないこと」と整合させ deterministic template のみ。

## Consequences

### Pros

- Continue plane の `recent_action_labels` が backend → BFF → UI の事実上の chain で event log から再構築されるようになる (構想層 §1: 認知状態が source of truth ベースで表現される)
- ADR-000904 の workspace bare advance semantic metadata 化が Phase D の入力として効くようになる (片肺状態を解消)
- v2 Surface Planner の `recent_continue_action_count` が立ち上がり、Continue bucket placement に user behavior が反映される
- `continue_context_builder.go` が pure function なので unit test の網羅率を上げやすく reproject 同一性を強保証

### Cons / Tradeoffs

- projector の per-event cost が増加 (7-day window の event 取得が UPSERT ごとに走る)。期待負荷では <50ms / event だが、burst で projector lag が立つ可能性がある。`knowledge_loop_projector_lag_events` alert (Phase H) で監視。
- `continue_context` JSONB schema の variation が増えるため、read path (`alt-backend → BFF`) で nil-safe な扱いを徹底する必要がある (proto は optional)
- 既存 row は reproject 経由でしか新 `recent_action_labels` を取得できない。本 ADR landed 後に runbook `docs/runbooks/knowledge-loop-reproject.md` 通り full reproject を 1 回回す
- Pact 互換性: 新 `target_type=conversation/recap/diff/entry` の値が consumer 側で未知の場合 fall through するが、念のため pact で `oneOf` matcher で固定

  - `app/usecase/knowledge_loop_projector/continue_context_builder.go` に pure function `buildContinueContextFromActed` が landed 済み。最大 5 件 bounded list、`semanticActionLabel(intent)` で proto enum (`DECISION_INTENT_OPEN` 等) と短形を両方受ける canonical 変換、`last_interacted_at` を event-bound に確定。
  - `app/usecase/knowledge_loop_projector/projector.go` の Acted handler (line 549〜) で `ListKnowledgeLoopActedEventsForEntry` + `buildContinueContextFromActed` + `PatchKnowledgeLoopEntryContinueContext` の経路が組まれており、reproject 同一性が保たれる。
  - `app/usecase/knowledge_loop_projector/score_resolver_event_log.go` で `continue_flag=true` の Acted event のみが `RecentContinueActionCount` に accumulate される (line 200 〜)。
  - `go test ./usecase/knowledge_loop_projector/` 全 case green。本 ADR ではコード変更なし、状態を `accepted` に確定し記録。
  - 後続の incremental enhancement (例: `target_type=recap` の `act_targets` への merge、`target_type=diff` で `change_summary` の補強) は別 ADR を切らずに必要時に additive 拡張する。
