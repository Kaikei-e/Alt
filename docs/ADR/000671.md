# ADR-000671: Gemma4 thinking token 予算枯渇による JSON 切断問題を /api/chat 移行・skeleton Planner・degradation ladder で解決する

## Status

Accepted

## Date

2026-04-10

## Affected Services

- acolyte-orchestrator (Python 3.14+)

## Context

[[000670]] で CompressorNode を追加し context packing は改善されたが、レポート `743372a2`（AIチップの2026年動向）のパイプライン実行ログを分析したところ、**全バージョンで body_len=0** — セクション本文が一度も生成されていないことが判明した。

### ログ分析結果

| Node | 結果 | 詳細 |
|------|------|------|
| Planner | **FAILED×2 → fallback** | `eval_count=2048`（上限消費）、"Unterminated string"、各108秒 |
| Gatherer | ✓ | 10 hits, faceted=true |
| Curator | ✓ | 3 sections, 10 curated |
| Hydrator | ✓ | 10/10 hydrated |
| Compressor | ✓ | 103K→9.4K (9%), empty=0 |
| Extractor | **FAILED×2 → fallback** | `eval_count=6000`（上限消費）、"Unterminated string"、各433-461秒 |

**根本原因**: Gemma4 の thinking tokens が `num_predict` 予算を食い尽くし、JSON を閉じきれない。`eval_count == num_predict` かつ `json.JSONDecodeError: Unterminated string` が確定的シグナル。

### Ollama プラットフォーム制約（2026-04-10 時点）

Web 調査で以下が確認された:

1. **thinking tokens は num_predict に含まれる** — 分離機能は未実装（[#10925](https://github.com/ollama/ollama/issues/10925) OPEN）
2. **`think=false` + `format` は Gemma4 で format が無視される** — ([#15260](https://github.com/ollama/ollama/issues/15260) OPEN）。format の probability masking が thinking 終端トークン前提で掛かっており、`think=false` 時に end-of-thinking token が出ないため masking が適用されない
3. **structured output + thinking mode は根本的に非互換** — ([#10538](https://github.com/ollama/ollama/issues/10538) OPEN）。format masking が `<think>` トークンの確率もゼロにする
4. **`/api/generate` では `options.think` が無視される** — ([#14793](https://github.com/ollama/ollama/issues/14793)）。`/api/chat` の top-level `think` のみ有効

### 設計判断の分岐

resolve 文書の方針: 「Gemma4 を直そうとしすぎないこと。Gemma4 は維持したまま、structured task の呼び出し方を変えること。予算を増やすのではなく、出力と入力を軽くすること。」

| 案 | 内容 | 判定 |
|---|---|---|
| A. `num_predict` を 2 倍に増やす | thinking に消えるだけで改善しない可能性。レイテンシ悪化 | **却下** |
| B. `think=false` を structured ノードで使う | #15260 で format が無視される。prompt 抑制も decoder 制約を回避できない | **却下** |
| C. **`/api/chat` 移行 + schema 縮小 + degradation** | Ollama 公式推奨、出力を軽く、失敗しても前進 | **採用** |

## Decision

**Gemma4 は全ノードで維持しつつ、structured task は `/api/chat` + format + temp=0 + think 未指定で呼び、出力と入力を軽くし、失敗時は段階的に劣化させて前進する。**

### 1. OllamaGateway を `/api/chat` に移行（structured task 用）

**ファイル**: `acolyte/gateway/ollama_gw.py`

`generate()` メソッドを用途別に分岐:
- `format` パラメータあり → `/api/chat`（structured output）
- `format` なし → `/api/generate`（freetext、Writer 向け）

`/api/chat` のリクエスト形式:
```json
{
  "model": "...",
  "messages": [{"role": "user", "content": "prompt"}],
  "format": {"type": "object", "properties": {...}},
  "options": {"num_predict": ..., "temperature": 0, ...},
  "stream": false
}
```

`think` パラメータは structured path では**送らない**。Ollama docs によれば、think を省略するとモデルのデフォルト動作になり、Gemma4 では thinking + format が現状最も安定する組み合わせ。`think=false` は #15260 で壊れるため意図的に回避。

freetext path（Writer）では `think` を top-level パラメータとして渡せるよう `LLMProviderPort` にも追加（将来用）。

### 2. Planner を固定 skeleton + query expansion に寄せる

**ファイル**: `acolyte/usecase/graph/nodes/planner_node.py`

resolve の方針: 「report_type ごとの固定テンプレートを正とし、LLM には section の query expansion だけさせる」

- `REPORT_TYPE_SKELETONS` で report_type ごとにセクション構造を固定（`executive_summary` / `analysis` / `conclusion`）
- LLM には skeleton の各セクションに対する `search_queries` の展開のみ依頼
- `num_predict` を 2048 → **1024** に削減（schema が小さいため）
- LLM 失敗時は skeleton + topic-based queries で確定（意図した fallback）
- prompt を短縮: セクション構造は固定値としてプロンプトに含め、LLM は query 語句だけ生成

### 3. Extractor に article 単位 degradation ladder を実装

**ファイル**: `acolyte/usecase/graph/nodes/extractor_node.py`

1 記事あたり最大 3 パスの degradation:

| Pass | max_facts | max_quote | num_predict | max_body | LLM 呼び出し |
|------|-----------|-----------|-------------|----------|-------------|
| 1 (full) | 3 | 120 chars | 4000 | 1200 chars | あり |
| 2 (light) | 2 | 80 chars | 3000 | 800 chars | あり |
| 3 (quote-only) | — | — | — | — | **なし** |

Pass 3 は LLM を使わず、compressed body の先頭 3 文を `verbatim_quote` として直接 fact に変換する（`confidence=0.3`）。これにより**全記事で必ず何らかの facts が生成される**。

従来の `num_predict=6000` 一律指定を廃止。Pass 1 の 4000 でも、schema を軽くしたため（max_facts=3, max_quote=120 chars）JSON を閉じきれる可能性が大幅に向上。

### 4. generate_validated に切断検出 + bounded retry を追加

**ファイル**: `acolyte/usecase/graph/llm_parse.py`

- `completion_tokens >= num_predict * 0.95` かつ `json.JSONDecodeError` → **truncation detected**
- 切断検出時: `num_predict` を **+25%** に増加して **1 回のみ** リトライ
- 2 回目の切断は budget_increased フラグで増加をブロック（指数的膨張を防止）
- 非切断の validation error（schema 不整合等）: 従来どおりリトライ

### 5. LLMProviderPort に `think` パラメータを追加（将来用）

**ファイル**: `acolyte/port/llm_provider.py`, `acolyte/gateway/ollama_gw.py`

`think: bool | None = None` を追加。現時点では:
- Structured path: 送らない（#15260 回避）
- Freetext path: `think` が指定されれば `/api/generate` payload に含める
- #15260 / #10538 が修正されたとき、structured ノードで `think=false` を有効化する配線

### CJK 品質修正（同日、[[000670]] 内で実施）

日本語レポートで facts が空になる問題も同時に修正:
- `_extract_query_terms()`: CJK 句読点（U+3000-303F）を token 区切りに
- `score_sentence()`: ASCII word match + CJK character bi-gram Jaccard の dual scoring
- `split_sentences()`: `\n` 改行を区切りに追加
- ExtractorNode: `compressed==[]` 時に hydrated body へ tiered fallback

### テスト

TDD で実装。259 unit tests + 4 contract tests が pass:
- `test_ollama_gw.py` (新規): `/api/chat` 移行、freetext 維持、eval_count 伝播、think 未送信
- `test_llm_parse.py` (拡張): 切断検出、budget +25%、bounded once、fallback
- `test_planner_facets.py` (修正): skeleton パターン、num_predict=1024
- `test_compressor_node.py` (拡張): degradation ladder、quote-only fallback、always-produces-output
- `test_pipeline.py` (修正): 統合テスト互換性

## Consequences

### Pros

- **JSON 切断の根本対策**: `/api/chat` で structured output の Ollama 公式推奨パスに移行。thinking token の扱いが `/api/generate` より安定
- **Planner 安定化**: skeleton 固定により LLM 出力の JSON サイズが大幅縮小。2048 tokens で切断されていた問題が 1024 で十分に
- **Extractor 耐障害性**: degradation ladder により LLM が完全に失敗しても quote-only fallback で facts を確保。「facts=[] → sections 空」の連鎖を断つ
- **切断の早期検出**: `completion_tokens >= 95% * num_predict` で切断を定量的に検出し、小幅 (+25%) な bounded retry で回復を試みる
- **レイテンシ改善**: Planner 1024 + Extractor 4000 → 従来の 2048+6000 から大幅に短縮
- **将来の upstream fix 対応**: `think` パラメータの配線により、#15260 修正後に structured ノードで `think=false` を即座に有効化可能

### Cons / Tradeoffs

- **quote-only fallback の品質**: Pass 3 の facts は `confidence=0.3` の低品質。SectionPlanner/Writer がこれを使うと主張の根拠が弱くなる
- **`/api/chat` と `/api/generate` の二重パス**: gateway が 2 つのエンドポイントを使い分けるため、Ollama options の一貫性管理が複雑化（ADR-579 の model reload 防止ルールに注意）
- **skeleton 固定の柔軟性低下**: report_type が `REPORT_TYPE_SKELETONS` にない場合はデフォルト 3 セクションに落ちる。将来新しい report_type を追加するたびに skeleton 定義が必要
- **bounded retry の限界**: +25% で 1 回しか増やさないため、thinking が非常に長い場合は依然として切断される。根本解決は Ollama の thinking budget 機能（#10925）の実装待ち
