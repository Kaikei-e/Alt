# ADR-000398: Knowledge Home のイベントストアと投影を alt-db に導入する

## Status

Accepted

## Date

2026-03-17

## Affected Services

- alt-backend (Go) — イベント書き込み・投影ジョブ・API 提供
- alt-db (PostgreSQL) — テーブル追加

## Context

Knowledge Home は Alt の知識パイプライン（RSS → 要約 → タグ → 検索 → Recap → RAG → TTS）を「なぜ今これが出ているか」を説明できる一画面に統合する機能である。

Home に表示されるアイテムの状態（新着、要約済み、タグ付き、Recap に含まれるなど）をフラグで管理すると、`articles` テーブルにカラムが増殖し、複数の書き込み元が競合するリスクが生じる。

alt-backend が唯一の write owner である原則（[[000241]]）に沿い、イベント追記 + 投影で状態管理を分離する append-first アーキテクチャを採用する。

## Decision

`knowledge_events` テーブルを canonical な append-only event log として alt-db に導入し、CQRS read model を投影テーブルで構築する。

### イベントストア

- **`knowledge_events`**: append-only event log。`dedupe_key UNIQUE` による冪等性保証、`event_seq BIGSERIAL` による順序保証、`payload JSONB` によるスキーマフリーなイベントデータ格納。
- イベント種別: `ArticleCreated`, `SummaryVersionCreated`, `TagSetVersionCreated`, `HomeItemsSeen`, `HomeItemOpened` 等。

### 投影テーブル（read model）

| テーブル | 用途 | 再構築可能 |
|---|---|---|
| `knowledge_home_items` | Knowledge Home のメインフィード | Yes |
| `today_digest_view` | 当日の記事統計・トップタグ | Yes |
| `recall_candidate_view` | リコール候補（Phase 1 はスケルトン） | Yes |
| `knowledge_user_events` | ユーザーインタラクションログ | No（一次データ） |
| `knowledge_projection_checkpoints` | Projector のカーソル管理 | Yes |

### Projector

- alt-backend の background job として 30 秒間隔で動作（`JobScheduler` パターン準拠）。
- `knowledge_projection_checkpoints` から `last_event_seq` を取得し、未処理イベントを 100 件ずつ batch 処理。
- イベント種別に応じて投影テーブルを更新し、checkpoint を進める。
- エラー時はログ出力して続行（DLQ は Phase 2 以降で導入予定）。

### 冪等性・再投影

- 全投影テーブルは `TRUNCATE` + 再投影で初期状態に戻せる。
- `dedupe_key` によりイベントの重複挿入を防止。
- Projector は checkpoint ベースのため、途中停止しても再開可能。

## Consequences

### Pros

- articles テーブルへのカラム増殖を防ぎ、既存クエリへの影響を最小化
- イベントログが残るため、「なぜこのアイテムが Home にあるか」を事後追跡可能
- 投影は再構築可能なため、スキーマ変更時のマイグレーションが安全
- outbox-worker と同じ思想のため、チーム内の認知負荷が低い

### Cons / Tradeoffs

- Projector の運用コスト増（監視、障害対応）
- イベントと投影の間に最大 30 秒の遅延（eventual consistency）
- Phase 1 では DLQ がないため、投影失敗時は手動対応が必要
- knowledge_events テーブルの肥大化に対する retention policy が未定（Phase 2 課題）
