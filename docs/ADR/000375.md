# 000375: Shared Secret 廃止と JWT 認証への統一

## ADR's STATUS

Accepted

## CONTEXT

Alt の認証アーキテクチャには、ブラウザリクエストの認証に2つの経路が存在していた。

### 1. JWT 経路（SvelteKit フロントエンド）

SvelteKit フロントエンド (`alt-frontend-sv`) は auth-hub の `/session` エンドポイントから JWT (`X-Alt-Backend-Token`) を取得し、バックエンドへの Connect-RPC リクエストに付与する。バックエンドは HS256 署名を検証してユーザーコンテキストを復元する。

### 2. Shared Secret 経路（nginx `auth_request`）

nginx は `auth_request` で auth-hub `/validate` を呼び出し、レスポンスヘッダーからユーザー情報（`X-Alt-User-Id` 等）を取得する。認証トークンとしては nginx 設定にハードコードされた固定文字列 `X-Alt-Shared-Secret` を付与し、alt-backend はこの静的トークンの一致で認証を通過させていた。

### 特定されたセキュリティ問題

1. **静的トークンのハードコード**: nginx 設定ファイルに shared secret の実値が直接埋め込まれており、設定ファイルへのアクセスでトークンが漏洩する
2. **shared secret の外部返送**: auth-hub `/session` が `X-Alt-Shared-Secret` をレスポンスヘッダーに含めており、ブラウザ側に漏洩するリスクがある
3. **Kratos Admin API の外部公開**: `compose/auth.yaml` でポート 4434（管理用API）がホストに公開されており、認証バイパスの攻撃面を広げていた
4. **dev 環境の弱いデフォルト secret**: `dev-secret-for-local` という固定値がフォールバックとして使用されており、開発環境でもシークレットのローテーションが行われない
5. **二重認証パスの複雑性**: JWT と shared secret の両方をサポートすることで、auth middleware のロジックが複雑化し、セキュリティレビューが困難になっていた

### ベストプラクティスとの乖離

- OWASP ASVS v4: 静的トークンよりも短命な署名付きトークンを推奨
- RFC 7519 (JWT): ステートレスな認証トークンとして業界標準
- Docker secrets パターン: `_FILE` 環境変数で実行時にファイルから読み込む方式が推奨される

## DECISION MAKING

JWT のみに統一し、shared secret 認証パスを完全に削除する方針とした。9つのフェーズに分割して実装した。

### Phase 1: auth-hub `/validate` に JWT 発行を追加

nginx の `auth_request` が呼び出す `/validate` エンドポイントに、JWT 発行ロジックを追加した。`token.IssueBackendToken()` を呼び出し、レスポンスヘッダー `X-Alt-Backend-Token` に JWT を設定する。

```go
// validate_handler.go — キャッシュヒット・ミス両方のパスで JWT を発行
identity := &client.Identity{ID: entry.UserID, Email: entry.Email, SessionID: sessionID}
backendToken, err := token.IssueBackendToken(h.config, identity, sessionID)
if err == nil {
    c.Response().Header().Set("X-Alt-Backend-Token", backendToken)
}
```

`ValidateHandler` 構造体に `config *config.Config` フィールドを追加し、`NewValidateHandler` のシグネチャを変更した。

### Phase 2: auth-hub から shared secret を除去

- `SessionHandler` から `authSharedSecret` フィールドと `X-Alt-Shared-Secret` ヘッダー設定コードを削除
- `config.Config` から `AuthSharedSecret` フィールドを削除
- 新アーキテクチャ (`internal/adapter/handler/`) の `SessionHandler`・`ValidateHandler` も同様に更新
- `InternalAuth` ミドルウェア（`/internal/*` エンドポイント保護）は `BackendTokenSecret` を使用するよう切り替え

### Phase 3: nginx を JWT 転送に切り替え

nginx の `auth_request_set` ディレクティブで auth-hub `/validate` のレスポンスヘッダーから JWT をキャプチャし、バックエンドへ転送する。

```nginx
# auth-hub /validate から JWT をキャプチャ
auth_request_set $auth_backend_token $upstream_http_x_alt_backend_token;

# バックエンドへ JWT を転送（旧: ハードコードされた shared secret）
proxy_set_header X-Alt-Backend-Token $auth_backend_token;
```

`nginx/docker-entrypoint.d/99-inject-secret.sh`（shared secret を sed で注入するスクリプト）を削除した。

### Phase 4: alt-backend を JWT-only に統一

`AuthMiddleware` から shared secret 関連のコードを全面削除した。

**RequireAuth()**: JWT 検証のみ。`X-Alt-Backend-Token` ヘッダーから JWT を取得し、署名・claims を検証してユーザーコンテキストを設定する。JWT がない/無効な場合は 401 を返す。

**OptionalAuth()**: V-005 セキュリティルール（Identity Header Injection 防止）を JWT ベースで再実装した。

```go
// JWT があれば検証してユーザーコンテキストを設定
// JWT が無効 → 匿名として続行（V-005: 不正な identity headers を無視）
// JWT なし + identity headers あり → 警告ログ + 匿名として続行
```

削除したコード:
- `sharedSecret` フィールド、`sharedSecretHeader` 定数
- `validateSharedSecret()` メソッド
- `AuthConfig` の `SharedSecret`・`SharedSecretFile` フィールド
- `validation.go` の `AUTH_SHARED_SECRET` バリデーション

### Phase 5: alt-frontend から shared secret 転送を除去

Next.js フロントエンドの `/api/fe-auth/validate/route.ts` から `X-Alt-Shared-Secret` ヘッダー転送コードを削除した。

### Phase 6: Kratos Admin API の非公開化

`compose/auth.yaml` からポート `4434:4434` のマッピングを削除した。Admin API はコンテナ内部（`127.0.0.1:4434`）からのみアクセス可能となり、ヘルスチェックは引き続き機能する。

### Phase 7: Dev secrets の改善

`compose/dev.yaml` と `compose/frontend-dev.yaml` に init コンテナ (`dev-secret-init`) を導入し、起動時にランダムな JWT secret をファイルに生成する方式に切り替えた。

```yaml
dev-secret-init:
  image: alpine:3.20
  command: >
    sh -c 'if [ ! -f /secrets/dev_jwt_secret.txt ]; then
      head -c 32 /dev/urandom | base64 > /secrets/dev_jwt_secret.txt;
      echo "Generated dev JWT secret"; fi'
  volumes:
    - dev_secrets:/secrets
```

mock-auth サーバー (`session.ts`) に `BACKEND_TOKEN_SECRET_FILE` 環境変数のサポートを追加し、Docker secrets パターンに対応させた。

### Phase 8: Compose / secrets 設定クリーンアップ

以下のファイルから `auth_shared_secret` 参照を削除した:
- `compose/base.yaml` — secret 定義
- `compose/auth.yaml` — 環境変数・secrets 参照
- `compose/core.yaml` — nginx secrets・alt-backend 環境変数・secrets 参照
- `compose/perf.yaml` — k6 secrets 参照

### Phase 9: k6 負荷テストの JWT 移行

k6 の認証ヘルパー (`k6/helpers/auth.js`) を shared secret から JWT に移行した。既存の `jwt.js`（HMAC-SHA256 JWT 生成）を活用し、`getAuthHeaders()` が短命な JWT を発行する方式に変更した。

```javascript
export function getAuthHeaders() {
  const cfg = getConfig();
  const now = Math.floor(Date.now() / 1000);
  const claims = {
    iss: "auth-hub", aud: ["alt-backend"],
    sub: cfg.testUserId, email: cfg.testUserEmail,
    sid: "k6-load-test-session",
    iat: now, exp: now + 300,
  };
  const token = generateJWT(cfg.backendTokenSecret, claims);
  return { "Content-Type": "application/json", "X-Alt-Backend-Token": token };
}
```

`docker-entrypoint.sh` から `auth_shared_secret` の読み込みを削除した。

## RESULTS, EFFECTS

### PROS

- **攻撃面の削減**: 静的トークン（shared secret）が完全に排除され、短命な署名付き JWT のみで認証が行われる。トークン漏洩時の影響が JWT の TTL（5分）に限定される
- **nginx 設定からのシークレット排除**: ハードコードされていた秘密値が設定ファイルから消え、設定の閲覧だけではトークンを取得できなくなった
- **認証ロジックの単純化**: alt-backend の `AuthMiddleware` が JWT のみをサポートするため、コードの見通しが良くなり、セキュリティレビューが容易になった
- **V-005 セキュリティルールの維持**: Identity Header Injection 防止が JWT ベースで再実装され、不正なヘッダー注入が引き続き検出される
- **Kratos Admin API の保護**: ポート 4434 の外部公開が停止され、管理操作がコンテナネットワーク内に限定された
- **Dev 環境のシークレット強化**: 起動ごとにランダムな secret が生成され、既知のデフォルト値による攻撃が不可能になった
- **統一された認証フロー**: SvelteKit 経由・nginx 経由の両方が同一の JWT 認証パスを使用し、動作の一貫性が確保された

### CONS, TRADEOFF

- **auth-hub `/validate` のレイテンシ微増**: JWT 発行（HMAC-SHA256 署名）の計算コストが追加されるが、数マイクロ秒程度であり実用上の影響はない
- **JWT の有効期間管理**: shared secret は無期限に有効だったが、JWT は 5 分で失効する。nginx の `auth_request` はリクエストごとに `/validate` を呼ぶため、実質的に毎回新しい JWT が発行される。セッションキャッシュ（TTL: 5分）が Kratos への問い合わせ頻度を制御する
- **k6 負荷テストの secret 依存**: k6 が JWT を自前で生成するために `BACKEND_TOKEN_SECRET` へのアクセスが必要になった。Docker secrets 経由で安全に渡されるが、テスト環境のシークレット管理に注意が必要
- **`secrets/auth_shared_secret.txt` の残存**: ファイル自体は残存するが、どのサービスからも参照されない。gitignore 対象のため、次回のシークレットローテーション時に削除可能

### 認証フロー（変更後）

```
Browser → nginx
  ├── auth_request → auth-hub /validate
  │     ├── Kratos session 検証（キャッシュあり）
  │     └── JWT 発行 → X-Alt-Backend-Token レスポンスヘッダー
  ├── auth_request_set $auth_backend_token (JWT キャプチャ)
  └── proxy_pass → alt-backend
        ├── X-Alt-Backend-Token: $auth_backend_token (JWT 転送)
        └── RequireAuth() → JWT 検証 → UserContext 設定
```
