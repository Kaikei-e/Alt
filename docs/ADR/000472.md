# ADR-000472: Knowledge Home dismiss イベントが tracking flag ガードで落ちる問題を修正する

## Status

Accepted

## Date

2026-03-19

## Affected Services

- alt-backend (Go) — `track_home_action_usecase`
- alt-frontend-sv (TypeScript/SvelteKit) — Knowledge Home ページ

## Context

Knowledge Home の各記事カードの × ボタン（Dismiss）を押しても、ページをリロードすると同じ記事が再表示される問題が報告された。

バックエンドには dismiss の仕組みが完備されている:

- `knowledge_home_items` テーブルに `dismissed_at` カラムがあり、`DismissKnowledgeHomeItem` が `UPDATE SET dismissed_at` を実行する
- `GetKnowledgeHomeItems` が `AND dismissed_at IS NULL` でフィルタするため、dismiss 済みアイテムは返却されない
- KnowledgeProjector が `EventHomeItemDismissed` イベントを処理して `DismissKnowledgeHomeItem` を呼ぶ
- `UpsertKnowledgeHomeItem` の UPSERT が `COALESCE(dismissed_at, EXCLUDED.dismissed_at)` で dismissed 状態を保持する

問題の根本原因は、dismiss イベントが backend のイベントログに到達する前に 2 箇所のガードで落とされていたことにある:

1. **FE ガード**: `trackingEnabled` が false の場合、`trackAction` が呼ばれず dismiss イベントが送信されない
2. **BE ガード**: `featureFlagPort.IsEnabled(FlagKnowledgeHomeTracking)` が false の場合、usecase が即座に `return nil` して全アクションを無視する

これらのガードは行動トラッキング（open, ask, listen 等の分析用データ収集）を制御するためのものだが、dismiss は「ユーザーの意思表示としてイベントログに記録し、projector 経由で read model に反映する」ための必須イベントであり、トラッキングの on/off に依存してはならない。

## Decision

**dismiss アクションを tracking flag ガードの例外として常に通過させる。**

append-first / reproject-safe の原則を維持し、read model（`knowledge_home_items`）への直接 mutate は行わない。FE と BE の 2 つのガードそれぞれで dismiss を特別扱いする。

### 代替案の検討

| 案 | 評価 |
|---|---|
| tracking flag を全ユーザーで有効にする | flag rollout の設計変更が必要で影響範囲が広い。dismiss だけの問題に対して過剰 |
| dismiss 時に read model を直接 UPDATE する | イミュータブルデータモデルの原則（append-first, reproject-safe）に違反する |
| dismiss を別の RPC エンドポイントに分離する | アーキテクチャ的には正しいが、既存の `TrackHomeAction` の設計意図（Home 上の全ユーザーアクションを統一的に記録する）と矛盾する。現時点では過剰 |
| **dismiss だけ flag ガードを通過させる（採用）** | 最小限の変更で問題を解決でき、既存アーキテクチャとの整合性が高い |

### 実装の概要

**BE — `usecase.go` (tracking flag ガード修正)**

```go
// Before: 全アクションをブロック
if u.featureFlagPort != nil && !u.featureFlagPort.IsEnabled(...) {
    return nil
}

// After: dismiss はガードを通過
if u.featureFlagPort != nil && !u.featureFlagPort.IsEnabled(...) {
    if actionType != "dismiss" {
        return nil
    }
}
```

**FE — `+page.svelte` (trackingEnabled ガード除去)**

dismiss の `trackAction` 呼び出しを `trackingEnabled` の条件外に移動し、常に送信されるようにした。`dismissItem` によるオプティミスティック UI 除去は引き続き即座に行われる。

### テスト

TDD で実装:

1. `"tracking disabled - dismiss still appends events"` テストケースを追加（RED: tracking disabled 時に dismiss が 0 イベントで失敗）
2. ガード修正を実装（GREEN: 全 14 テスト PASS）

## Consequences

### Pros

- dismiss イベントが確実に `knowledge_events` に append されるため、projector が `dismissed_at` を設定でき、リロード後に同じ記事が再表示されなくなる
- イミュータブルデータモデルの原則（append-first, reproject-safe）を維持している
- 変更は最小限（BE: 1 行の条件追加、FE: ガード条件の除去）で、既存の tracking フローに影響を与えない
- open, ask, listen 等の分析用トラッキングは引き続き feature flag で制御可能

### Cons / Tradeoffs

- dismiss と他のアクションで flag ガードの挙動が異なるため、将来のメンテナンス時に注意が必要
- projector ラグ（実測 ~25 分遅れ）により、dismiss 後すぐにリロードすると一時的に同じ記事が表示される可能性がある。これは別チケット（projector throughput / backlog 解消）で対応する
- tracking flag の rollout 設計見直し（Home 利用ユーザーの tracking を全員分収集する恒久策）は別途検討が必要
