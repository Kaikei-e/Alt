# ADR-000442: Knowledge Home の Stream Updates を Home 画面に接続し実用品質化する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — stream payload の canonical contract 準拠 enrichment
- alt-frontend-sv (TypeScript/SvelteKit) — hook 改修、通知バー、Home 画面接続

## Context

[[000410]] と [[000422]] で Knowledge Home の stream 基盤（backend の polling+heartbeat+coalesce+stale disconnect、frontend の reconnect+coalesce+pendingUpdates、feature flag、BFF routing）が実装済みだったが、以下の状態で止まっていた:

1. **Backend**: `EventType` をそのまま送信しており（`ArticleCreated` 等）、[[000434]] の canonical contract（`item_added` / `item_updated` / `digest_changed`）に準拠していない
2. **Backend**: stream response に `item` フィールドが未セットで、frontend がどの記事が更新されたか判別できない
3. **Frontend**: `useStreamUpdates` hook は存在するが Home ページ（`+page.svelte`）に接続されていない
4. **Frontend**: `stream_expired` / `fallback_to_unary` 等の terminal event を処理していない
5. **Frontend**: タブ非アクティブ時もストリームが維持されサーバーリソースを消費する
6. **Frontend**: 複数タブで同一ユーザーが Home を開くとタブ数分の接続が張られる
7. **Frontend**: feature flag が `fetchData` 完了後に同期されず、ストリームが起動しない

## Decision

**stream を canonical contract 準拠に enrichment し、通知バー UI を介して Home 画面に安全に接続する**方式を採用する。自動 DOM 差し替えは行わず、「N 件の更新があります → 更新を反映」方式でユーザーに制御権を渡す。

### 1. Backend — canonical stream type mapping + minimal payload

`mapToCanonicalStreamType()` 関数を導入し、domain event type を canonical type に変換する:

| Domain EventType | Canonical stream type | Payload |
|---|---|---|
| `ArticleCreated` | `item_added` | `item.item_key = "article:{aggregate_id}"` |
| `SummaryVersionCreated` / `TagSetVersionCreated` / `*Superseded` | `item_updated` | `item.item_key = "article:{aggregate_id}"` |
| その他（`HomeItemOpened` 等） | `digest_changed` | なし（frontend が unary re-fetch） |

`buildStreamResponse()` 関数で `StreamKnowledgeHomeUpdatesResponse` を構築し、`item_added` / `item_updated` には `item_key` のみセットした最小 `KnowledgeHomeItem` を載せる。proto 変更なし。

`streamFlagGuard()` を抽出し、認証 + feature flag チェックをユニットテスト可能にした。

### 2. Frontend — `useStreamUpdates` hook 改修

既存 hook を全面リライトし、以下を追加:

- **`enabled` パラメータ**: feature flag ゲート。`false` の場合は一切の接続・タイマーを行わない（安全側倒れ）
- **Terminal event handling**: `stream_expired` → `reconnectAfterMs` 指定の遅延で再接続、`fallback_to_unary` → `isFallback = true` で再接続停止
- **Tab visibility handling**: `visibilitychange` イベントで `document.hidden` 時に `suspended = true` + disconnect、visible 復帰時に再接続。`suspended` state により意図的切断と異常切断を分離
- **Multi-tab leader election**: `BroadcastChannel` を使った claim→ack→announce→heartbeat protocol。Leader のみが stream 接続し、follower は channel 経由で update を受信。Leader heartbeat 10 秒途切れで follower が re-claim。`BroadcastChannel` 非対応環境では graceful degradation（各タブ独立接続）
- **`onRefresh` callback**: `applyUpdates()` 時に呼ばれ、Home ページが `fetchData(true, lensId)` で全件再取得

### 3. Frontend — `StreamUpdateBar` コンポーネント強化

既存コンポーネントに `isConnected` / `isFallback` props を追加し、接続状態ドット（green=connected, gray=disconnected, orange=fallback）を表示。

### 4. Frontend — Home 画面接続

- `useKnowledgeHome` に `featureFlags` getter を追加
- `+page.svelte` の `onMount` で `fetchData` 完了後に `flags.setFlags(home.featureFlags)` を呼び出し、feature flag を同期
- `streamEnabled = $derived(flags.isEnabled("enable_stream_updates"))` で flag 駆動
- `StreamUpdateBar` を LensSelector 直後（desktop/mobile 両方）に配置
- `onApply` → `stream.applyUpdates()` → `home.fetchData(true, lens.activeLensId)` で全件再取得

### 5. Contract テスト拡充

`knowledge-home-contract.test.ts` に 4 ケース追加:
- `item_added` / `item_updated` に minimal `item_key`
- `digest_changed` は payload なし
- `fallback_to_unary` に `reconnect_after_ms`

### 検討した代替案

1. **自動 DOM 差し替え**: stream event 受信時に即座に KnowledgeStream のアイテムリストを更新する案。「ユーザーが読んでいる最中にリストが変わる」UX 問題と、partial update の整合性管理が複雑になるため不採用。通知バー方式でユーザーに制御権を渡す。
2. **SharedWorker による接続共有**: BroadcastChannel より強力だが、別ファイル管理が必要で複雑性が高い。小規模ユーザーベースでは BroadcastChannel の leader election で十分。
3. **全タブ独立接続**: multi-tab 保護なし。実装は最も簡単だが、同一ユーザーがタブを 5 つ開くと 5 本の接続が張られる。BroadcastChannel で 1 本に抑制する方が安全。

## Consequences

### Pros

- Knowledge Home に **リアルタイム更新通知** が到達し、ユーザーが手動リロードなしで新着を認識できる
- canonical contract 準拠により、**frontend と backend の event type が一致** し、将来の event type 追加が安全
- `item_key` を payload に含めることで、将来の **item-level incremental update** への発展パスが確保された
- multi-tab leader election により **接続数がユーザーあたり最大 1 本** に抑制される
- tab visibility handling により **非アクティブタブがサーバーリソースを消費しない**
- `enable_stream_updates = false` で即座に無効化可能（**feature flag によるロールバック**）
- 全変更が TDD で実装され、backend 20 テスト + frontend 39 テストで保護されている

### Cons / Tradeoffs

- coalesce 遅延（backend polling 5s + client coalesce 3s = 最大 8s）があり、リアルタイム性は限定的。将来の event bus 移行で改善可能
- `BroadcastChannel` leader election の race condition は claim→ack→announce の 3-way protocol + heartbeat timeout で軽減しているが、極端なタイミングで一時的に 2 本接続が張られる可能性はゼロではない（server 側 stale disconnect で自然解消）
- `StreamUpdateBar` の「更新を反映」は全件再取得を行うため、アイテム数が多い場合のレスポンスサイズが課題になりうる
- stream 処理テスト（async iterator + fake timers）は Vitest の制約上 real timers を使用しており、テスト実行時間が約 13 秒かかる
