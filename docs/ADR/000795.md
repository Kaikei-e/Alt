# ADR-000795: acolyte-orchestrator Gatherer に HyDE 擬似文書 variant を追加してクロスリンガル recall を底上げする

## Status

Accepted

## Date

2026-04-19

## Affected Services

- **acolyte-orchestrator (domain)** — `domain/hyde.py` 新設 (prompt + sanitizer)
- **acolyte-orchestrator (port)** — `port/hyde_generator.py` 新設 (Protocol)
- **acolyte-orchestrator (gateway)** — `gateway/news_creator_hyde_gw.py` 新設 (LLM 呼び出し)
- **acolyte-orchestrator (usecase)** — `usecase/graph/nodes/gatherer_node.py` の RRF variants に HyDE を差し込み
- **acolyte-orchestrator (DI / settings)** — `main.py` 配線 + `Settings.hyde_*` 4 フィールド

## Context

[[000669]] で Gatherer は multi-query RRF (`primary` / `broad` / `narrow` の 3 variant) を持ち、[[000794]] で Meilisearch の hybrid search + Ollama `qwen3-embedding` による semantic retrieval が有効化された。これで BM25 単独では拾えなかった concept-level match が引けるようになった。

しかし日本語 topic に対してクエリそのものが CJK ベースで構成される場合、vector 側の retrieval も入力コンテキストが CJK 優位になり、英語記事の埋め込み空間に完全に届くわけではない。例えば「イラン情勢 中東」のような短い topic は、英語記事の埋め込みと距離が遠くなりがちで、semanticRatio を上げても頭打ちになる。

HyDE (Hypothetical Document Embedding) の研究では、LLM に target_lang で「理想の回答記事の書き出し」を書かせ、それを query vector として投入することで、クエリと記事の埋め込み空間のギャップを埋める効果が報告されている。Gemma 1B-7B クラスの追加実験でも Japanese / Korean 等で mContriever を上回る事例あり。

ADR-794 の hybrid retrieval の上に、LLM 呼び出し 1 回のコストで cross-lingual recall を底上げできる枠がある。

## Decision

HyDE を Gatherer の 4 本目の RRF variant として導入する。実装は Clean Architecture の Port / Gateway 分離を踏襲し、Gatherer は Protocol 経由でのみ HyDE に依存する。

- **Domain (`domain/hyde.py`)**: `build_hyde_prompt(topic, target_lang)` は en / ja の 2 テンプレートのみサポート。system rules に `<topic>` タグ内はデータ扱いを明示し、`Here is:` / markdown fence / 前置きが付いた出力を sanitiser で除去。`sanitize_hyde_output` は以下を順に実行:
  1. markdown code fence 除去
  2. XML tag 全剥がし (`<[^>]+>`) — プロンプト injection で `<system>` 擬似タグを書き返してくるケースを無効化
  3. boilerplate prefix (`here is`, `以下は`, `本文:` 等) を lowercased 先頭マッチで strip
  4. 長さ 600 字 cap
  5. **言語適合フィルタ**: target=en なら ASCII 文字 ≥ 40 + CJK × 2 ≤ ASCII。target=ja なら CJK ≥ 20。どちらも満たさなければ None
- **Port (`port/hyde_generator.py`)**: `generate_hypothetical_doc(topic, target_lang) -> str | None`。None は「skip this variant」のシグナル
- **Gateway (`gateway/news_creator_hyde_gw.py`)**: 既存 `LLMProviderPort` (news-creator / Gemma4) をラップ。`asyncio.wait_for` で timeout 8s、`TimeoutError` 以外の例外は warn ログ + None 返却で Graph を絶対に止めない
- **Gatherer 統合** (`usecase/graph/nodes/gatherer_node.py`):
  - `__init__` に `hyde_generator: HyDEGeneratorPort | None = None` を追加 (backward compat)
  - `_search_by_facets` の facet ループ**前**に 1 回だけ `_detect_topic_language(topic)` → ja → en / en → ja の片方向で HyDE を生成 (und は skip)
  - 結果を facet 毎の variants リスト末尾に `(hyde_doc, "hyde_{target_lang}")` として append、RRF はラベルを区別せず単純融合
- **Topic 言語判定** (`_detect_topic_language`): pre-processor の Go 版 `DetectLanguage` を Python port。CJK ≥ 2 かつ `cjk*3 ≥ cjk+latin` で ja、Latin ≥ 3 かつ `latin > cjk*2` で en、いずれにも該当しない場合 und。テックブログ風 code-switch タイトル (`Rails.logger をJSONで出力する`) は ja 判定になる (CJK 6 個含む)
- **DI**: `main.py` で `settings.hyde_enabled` 条件下でのみ `NewsCreatorHyDEGenerator` を実体化し、`build_report_graph(..., hyde_generator=_hyde_generator)` まで貫通。flag off で従来挙動に即時 rollback 可

### 検討した代替案

| 案 | 採用 | 却下理由 |
|---|---|---|
| **Gemma4 で Gatherer-scoped HyDE (採用)** | ○ | 既存 LLM を再利用、追加 GPU 無し。topic あたり 1 回のみで facet 数に依存しない |
| 専用小型 LLM (Gemma-1B / Phi-3-mini) を追加 | × | GPU 予算 1.6GB 残 ([[000794]]) で bge-reranker 等に温存したい。新規モデルの運用コスト増 |
| 両方向 HyDE (ja + en 両方生成) | × | LLM 呼び出しが 2 倍 = latency 2x。topic 言語から方向を決めれば片方向で十分で、逆方向は次フェーズで評価 |
| HyDE を search-indexer 側で実行 | × | LLM 呼び出しのホットパス化を避ける。Gatherer は LangGraph 内で semaphore / timeout を統一管理 |
| Settings ではなく env で直接読む | × | 既存 `Settings` パターンに統一、dry-run テストが容易 |

## Consequences

### Pros

- **Cross-lingual recall の底上げ**: hybrid search ([[000794]]) の vector マッチに加えて、target-lang の "ideal document" ベクトルを query に注入することで、直接的な語の一致がない topic でも英語記事が上位に食い込める
- **Graph の他段に影響なし**: Curator / Writer / Critic / Finalizer は HyDE の有無を意識しない。variants に追加する薄い改修のみ
- **Graceful degrade**: timeout / sanitiser reject / `hyde_enabled=False` のいずれでも primary / broad / narrow の 3 variant で既存通り走る
- **Security**: prompt injection 由来の echo / fake タグ / boilerplate を sanitiser で弾くため、検索クエリとして毒を注入される経路がない (security-auditor SA-2 対応)
- 新規 32 tests (prompt / sanitizer / gateway timeout / gatherer integration / topic 言語判定) + 既存 Acolyte 557 = **689 tests 全 green**

### Cons / Tradeoffs

- **Gemma4 を 1 report につき +1 回呼ぶ**。既存 Writer / Curator 生成と HybridPrioritySemaphore の GPU slot を取り合う。現状は `priority=foreground` 扱い (1-2 秒追加)、`background` 化は別 ADR で調整
- **ja 判定が CJK 2 個 + `cjk*3 ≥ cjk+latin` で成立**するため、`AI chips 2026 チップ市場` のようなケースは ja 判定 → en HyDE が回る。意図とは逆になる局面があるが実用上は en HyDE で Japanese market 英語記事も拾えるため大きな害はない。untuned
- **HyDE の品質は Gemma4 のゼロショット能力に依存**。モデル変更時は再校正が必要
- **Latency 追加**: HyDE 生成 1-2 秒 / report。Acolyte 全体の 71 分ランタイム基準では誤差だが、future work で短尺ランでの影響を再評価
