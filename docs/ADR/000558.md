# ADR-000558: tag_hotspot の実トレンド検出化と KnowledgeCard タグ展開を実装する

## Status

Accepted

## Date

2026-03-24

## Affected Services

- alt-backend (Go) — トレンドタグ検出パイプライン追加、projector から無条件付与を除去
- alt-frontend-sv (SvelteKit) — KnowledgeCard のタグ展開 UI

## Context

Knowledge Home の KnowledgeCard に付与される「Trending: {tag}」バッジ（WhyReason `tag_hotspot`）の精度が低い問題が報告された。

### 根本原因

`knowledge_projector.go` の `projectTagSetVersionCreated` が、タグを持つ**全記事**に対して無条件で `tag_hotspot` を付与していた。

```go
if len(tags) > 0 {
    whyReasons = append(whyReasons, domain.WhyReason{
        Code: domain.WhyTagHotspot,
        Tag:  tags[0], // 常に最初のタグ
    })
}
```

この実装には以下の問題がある。

1. **全記事に Trending が表示される** — タグがあれば必ず付与されるため、バッジの意味がない
2. **常に `tags[0]`** — 最もトレンドしているタグではなく、単に先頭のタグを使用
3. **トレンド検出ロジックがゼロ** — 過去のベースラインとの比較、急増判定が一切ない
4. **reproject-safe 違反** — projector 内での非決定的な付与は再投影時に結果が変わりうる

また、KnowledgeCard のタグ一覧は3件を超えると `+N tags` と表示されるが、`<span>` 要素であり展開不可能だった。

## Decision

**tag_hotspot の付与を projector から除去し、usecase 層でのリクエスト時 enrichment に移行する。トレンド判定ロジックは SQL ではなく Go 側で実装し、ユーザースコープ + 時間ベースの surge 検出を行う。**

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|--------|------|------------|
| SQL 内で CASE 式による surge_ratio 計算 | 一発で結果が得られる | SQL にビジネスロジックを持たせるアンチパターン。Alt では Go 側で判定する方針 |
| projector 内でトレンドタグを参照して付与 | 既存の write path に乗る | projector は event payload のみを使うべき（reproject-safe 原則）。外部参照は原則違反 |
| バッチジョブで定期的に tag_hotspot を更新 | 負荷分散可能 | read model への直接更新は CQRS 原則に反する。usecase 層での enrichment が自然 |

### 実装構成

#### 1. 新規ポート `TagHotspotPort` / `FetchTagArticleCountsPort`

`knowledge_home_port` パッケージに2つのポートインターフェースを追加。`TagHotspotPort` はキャッシュ付きトレンドタグ取得、`FetchTagArticleCountsPort` はタグ別記事数の生データ取得を担当する。

#### 2. 新規ドライバ `FetchTagArticleCounts`

`alt_db` に追加。SQL はデータ取得のみで、トレンド判定のビジネスロジックを含まない。

```sql
SELECT ft.tag_name, COUNT(DISTINCT at.article_id) AS article_count
FROM feed_tags ft
JOIN article_tags at ON ft.id = at.feed_tag_id
JOIN articles a ON at.article_id = a.id
WHERE a.created_at >= $1 AND a.deleted_at IS NULL AND a.user_id = $2
GROUP BY ft.tag_name
```

`$1` に期間（7日前 / 30日前）、`$2` に `user_id` を渡す。同一メソッドで両期間に使い回す設計。

#### 3. 新規 Gateway `TrendingTagsGateway`

トレンド判定ロジックを Go 側に集約。

- **直近7日の記事数** と **30日間の週平均** を比較
- **surge_ratio = 直近7日 / (30日 ÷ 4週)** を計算
- `surge_ratio >= 1.5`（1.5倍以上の急増）をトレンドと判定
- `article_count < 3` のノイズを除外
- 上位20タグを返却
- **30分 TTL のユーザーごとキャッシュ**（`sync.RWMutex` + map）

ベースラインに記事がないタグ（新規出現）は `surge_ratio = article_count` として扱い、閾値を超えればトレンドとする。

#### 4. Projector から無条件 tag_hotspot を除去

`projectTagSetVersionCreated` から `WhyTagHotspot` の付与を削除。タグの投影（`item.Tags` への設定）は維持する。これにより projector が event payload のみに依存する reproject-safe な状態に近づく。

#### 5. Usecase でリクエスト時 enrichment

`GetKnowledgeHomeUsecase.Execute()` 内に `enrichTagHotspots()` を追加。items 取得後にトレンドタグセットと照合し、一致するタグを持つ item にのみ `tag_hotspot` を動的に付与する。

- `TagHotspotPort` は optional setter パターンで注入（`SetTagHotspotPort`）
- port が nil の場合は enrichment をスキップ（後方互換）
- 取得エラー時は `slog.WarnContext` で記録し、enrichment をスキップ（non-fatal）
- 既に `tag_hotspot` がある item には重複付与しない

#### 6. KnowledgeCard タグ展開

`KnowledgeCard.svelte` の `+N tags` を `<span>` から `<button>` に変更。`$state(false)` の `tagsExpanded` と `$derived` の `visibleTags` で展開/折りたたみを制御。Svelte 5 Runes 準拠。

## Consequences

### Pros

- **Trending バッジが実際のトレンドを反映** — 全記事ではなく、直近7日で急増しているタグを持つ記事にのみ表示
- **reproject-safe 改善** — projector から非決定的な外部参照が除去される
- **ユーザースコープ** — 各ユーザーの購読フィードのタグのみが対象
- **SQL にビジネスロジックなし** — Alt の設計方針に準拠
- **タグ展開** — +N tags クリックで全タグを確認可能に
- **Non-fatal 設計** — トレンドタグ取得に失敗しても Home の表示は維持

### Cons / Tradeoffs

- **リクエスト時コスト** — enrichment が毎リクエストで実行される。ただしキャッシュ（30分 TTL）で DB アクセスは抑制
- **ユーザーごとキャッシュのメモリ** — ユーザー数が増加するとキャッシュサイズが比例して増大。LRU eviction の導入が将来必要になる可能性がある
- **既存 projection の tag_hotspot が残存** — projector 変更前に投影された item の `why_json` には旧 `tag_hotspot` が残る。再投影で解消されるが、即時修正には reproject が必要
- **surge_ratio の閾値がハードコード** — 1.5x は経験的な初期値。実運用でチューニングが必要になる可能性がある
