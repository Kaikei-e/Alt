# ADR-000414: Knowledge Home の backfill をスケジュール実行可能な履歴リプレイ方式にする

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — backfill usecase, backfill repository, scheduler, projector input pipeline

## Context

Knowledge Home Admin には backfill 操作が存在したが、実装は実質的にダミーだった。

問題は 3 つあった。

1. `StartBackfill(...)` は job row を作るだけで、対象 article 数を把握していなかった
2. `knowledge_backfill_job.go` は「placeholder batch step」のままで、historical article から `ArticleCreated` を生成していなかった
3. `KnowledgeBackfillJob(...)` 自体が scheduler に登録されておらず、Admin から pending job を作っても worker が起動しなかった

このため、既存記事が大量に存在する環境で feature flag を有効化しても、Knowledge Home は空のままになり、read model を再生成する手段が存在しなかった。

## Decision

**Knowledge Home backfill を、`articles` テーブルを created_at 昇順で走査して `ArticleCreated` events を再生成するスケジュール実行型 worker として実装する。**

### 変更内容

1. `knowledge_backfill_port` に以下を追加した。
   - `CountBackfillArticlesPort`
   - `ListBackfillArticlesPort`
2. `driver/alt_db/knowledge_backfill_repository.go` に履歴 article の件数取得と keyset 走査を実装した。
   - `deleted_at IS NULL`
   - `(created_at, id)` による前進カーソル
   - `COALESCE(published_at, created_at)` により published_at 未設定記事にも対応
3. `knowledge_backfill_usecase.StartBackfill(...)` は job 作成時に総件数を固定し、Admin 画面で進捗が見えるようにした。
4. `knowledge_backfill_job.go` は placeholder 処理を廃止し、batch ごとに historical article から `ArticleCreated` events を append、cursor と processed count を更新する実装に置き換えた。
5. `main.go` の scheduler に `knowledge-backfill` job を追加し、pending/running job が実際に消費されるようにした。

### 設計判断

1. **materialized view を直接書き戻さない** — backfill は `knowledge_home_items` を直接再生成せず、通常経路と同じ `knowledge_events` を通す。これにより projector の単一責務と再現性を維持する。
2. **ascending keyset pagination を採用する** — 大量 article 環境で offset pagination を避け、再開可能な cursor (`cursor_date`, `cursor_article_id`) を job row に保持する。
3. **件数を事前固定する** — `TotalEvents` を start 時に確定させることで、Admin UI の進捗と completion 判定を安定させる。
4. **空 batch で完了する** — 新規 batch が返らなくなった時点で completed に遷移し、TotalEvents と完全一致しないケースでも worker が停止し続けないようにする。

### テスト

- `usecase/knowledge_backfill_usecase/usecase_test.go`
  - StartBackfill が件数を取り込んだ pending job を生成すること
  - CountBackfillArticles 失敗時に error を返すこと
- `job/knowledge_backfill_job_test.go`
  - batch 処理で event append、cursor 更新、completed 遷移が行われること
  - article が残っていない場合に completed へ遷移すること

## Consequences

### Pros

- **既存記事を Knowledge Home に流し込める** — feature flag 後の空状態を Admin 操作で解消できる
- **通常経路と同じ投影パイプライン** — backfill だけ別ロジックで `knowledge_home_items` を構築しないため、read model の挙動が揃う
- **再開可能** — cursor を job row に保持するため、中断後も同一 job を継続しやすい

### Cons / Tradeoffs

- **article 件数に応じて event store が増える** — 既存全件を event 化するため `knowledge_events` の増加は避けられない
- **Phase 4 時点では ArticleCreated のみを replay** — summary/tag 側の履歴再構築を完全に行うには追加イベントの replay が必要
- **scheduler 依存** — backfill は即時同期処理ではなく、定期 job の tick によって進む
