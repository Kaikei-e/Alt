# ADR-000502: DeleteRSSFeedLink を購読解除として扱う

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-backend (Go)

## Context

`DeleteRSSFeedLink` は名前上は feed source の削除を示していたが、実際の運用文脈では「現在ユーザーが購読一覧から外したい」という意図で使われていた。

一方で Alt の RSS モデルでは、`feed_links` は共有の source registry であり、user ごとの状態は `user_feed_subscriptions` に保持している。この前提は [[000213]] で導入済みで、RSS 登録時にも authenticated user を自動購読させる構造になっている。

この状態で `DeleteRSSFeedLink` が `feed_links` の物理削除を実行すると、user が期待する「自分の解除」と API が実行する「共有 source の抹消」が乖離する。さらに `feeds.feed_link_id` が `NOT NULL` に強化された後は、削除系の外部キー戦略と衝突しやすくなり、API の意味論の曖昧さが直接障害化する。

## Decision

**`DeleteRSSFeedLink` は feed source の物理削除ではなく、current user の購読解除として扱う。**

今回の決定では以下を採用する。

- `DeleteFeedLinkUsecase` は `SubscriptionPort.Unsubscribe` を呼ぶ
- Connect handler / REST handler の両方で authenticated `userID` を usecase に渡す
- 成功レスポンスは削除ではなく解除を示す文言に変更する
- `feed_links`, `feeds`, `articles` の物理削除はこの API では実行しない

### 代替案

| 案 | 評価 |
|---|---|
| 既存 API 名のまま物理削除を維持 | user 意図と API 挙動が一致せず、共有データを巻き込むため不採用 |
| ハンドラ側だけで購読解除に読み替える | 一見小変更だが、usecase 契約が delete のまま残り誤用を防げないため不採用 |
| 管理者専用の delete API を別途追加する | 将来的には有効だが、今回の障害修正とは分離すべきため今回は採用しない |

## Consequences

### Pros

- user 操作と API の意味論が一致する
- `user_feed_subscriptions` に責務を閉じ込められ、共有テーブルを誤って削除しない
- Connect / REST の両経路で同じ契約に揃えられる

### Cons / Tradeoffs

- API 名に delete という語が残るため、呼び出し側の誤読余地は残る
- 真に feed source を削除したい管理操作は別経路で設計し直す必要がある
- 既存 UI やクライアントコードの文言が delete 前提なら追随修正が必要になる
