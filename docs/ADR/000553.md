# ADR-000553: StreamSummarize を REST から Connect-RPC に移行しハートビートで Cloudflare 524 を回避する

## Status

Accepted

## Date

2026-03-24

## Affected Services

- alt-backend (Go)
- pre-processor (Go)

## Context

Desktop Visual Preview のオンザフライ要約（StreamSummarize）が Cloudflare HTTP 524（100秒タイムアウト）で失敗する問題が発生した。

### 3 つの原因

**原因 1: REST パスの `processingArticles` ガード**

alt-backend の `streamPreProcessorSummarize()` は REST `POST /api/v1/summarize/stream` を呼んでいた。この REST ハンドラには `processingArticles` sync.Map によるロックがあり、同一記事が BE バッチワーカーでも処理中の場合に 409 Conflict を返す。一方、Connect-RPC `StreamSummarize` ハンドラにはこのガードが存在しない。

**原因 2: Cloudflare の 100 秒アイドルタイムアウト**

Cloudflare はオリジンからの最初の応答バイトまで 100 秒で接続を切断する。news-creator の LLM セマフォ待ちが長い場合、`streamPreProcessorSummarize()` が blocking call として 100 秒以上ブロックされ、その間ブラウザへの応答が一切送られない。

**原因 3: pre-processor Connect-RPC サーバーの WriteTimeout**

`bootstrap/server.go` で Connect-RPC サーバーの `WriteTimeout` が 30 秒に設定されていた。Go の `http.Server.WriteTimeout` はストリーム全体の期限に適用されるため、LLM 推論が 30 秒を超えるとサーバーが接続を切断し、`"protocol error: incomplete envelope: unexpected EOF"` が発生した。

## Decision

### 1. `streamPreProcessorSummarize` を Connect-RPC に移行する

REST `/api/v1/summarize/stream` → Connect-RPC `StreamSummarize` に切り替える。既存の `preprocessor_connect/client.go:StreamSummarize` + `streamAdapter` (io.ReadCloser) を流用する。

- `processingArticles` ガードをバイパスできる（Connect-RPC ハンドラにガードなし）
- プロトコルが統一される（他の pre-processor 呼び出しも Connect-RPC）
- `processingArticles` ガードのバイパスは正当化される：ADR-000185 の 3 レイヤー（`MAX_QUEUE_DEPTH` + `hold_slot` + 指数バックオフ）でリトライストーム防止は十分に担保されている

### 2. ハートビートで Cloudflare 524 を回避する

`streamPreProcessorSummarize` を goroutine で非同期実行し、接続確立待ちの間もブラウザへ 15 秒間隔で空チャンクを送信する。

- 即座に最初のハートビートを送信し、HTTP レスポンスを開始
- pre-processor 接続を goroutine で非同期開始
- 接続確立まで 15 秒間隔でハートビート
- 接続確立後は `streamAndCaptureWithHeartbeat` で LLM の最初のトークン待ちでもハートビート

FE の `streamingRenderer.ts` は `if (!decoded) return;` で空文字チャンクを無視するため互換性に影響なし。

### 3. pre-processor Connect-RPC サーバーの WriteTimeout を無効化する

`bootstrap/server.go` の `WriteTimeout` を 0（無効）に変更する。RPC ごとのタイムアウトは `context.WithTimeout` で制御する。

## Consequences

### Pros

- Cloudflare 524 が解消される（ハートビートでアイドルタイマーをリセット）
- `processingArticles` ガードによる FE 拒否が解消される
- プロトコルが Connect-RPC に統一される
- LLM 推論時間に依存せずストリーミングが安定する

### Cons / Tradeoffs

- `processingArticles` ガードをバイパスするため、同一記事に対する FE + BE の同時処理が発生しうる（GPU 二重消費）。ただし FE は低頻度であり、Phase 2 の mq-hub キャンセル機構で恒久対策予定
- `WriteTimeout: 0` はストリーミング以外の Connect-RPC エンドポイントにも影響する。悪意のあるリクエストが接続を長時間保持する理論的リスクがある
- ハートビートの空チャンクが Connect-RPC プロトコルの帯域を微量消費する
