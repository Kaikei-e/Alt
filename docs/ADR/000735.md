# ADR-000735: search-indexer を呼ぶ全 consumer に X-Service-Token を強制し Pact で回帰を封じる

## Status

Superseded by [[000743]] — search-indexer 向けの X-Service-Token 強制と consumer pact での token assertion は廃止され、caller 識別は mTLS peer identity allowlist ([[000741]]) に一本化された。Pact による service boundary 回帰防止の枠組み自体は [[000739]] / [[000740]] に引き継がれている。

## Date

2026-04-15

## Affected Services

- rag-orchestrator (Go)
- alt-backend (Go)
- search-indexer (Go)

## Context

[[000722]] で search-indexer の REST `/v1/search` と Connect-RPC `SearchService` に `X-Service-Token` を必須化した。acolyte-orchestrator の gateway だけが追従済みで、他の consumer は暗黙に未対応のまま放置されていた。

2026-04-15 の `alt-rag-orchestrator-1` コンテナログで、retrieval パイプラインの 2 本（tag_search と hybrid_bm25_search）が `search returned status: 401` を連発し、Augur のチャットストリームが空の retrieval コンテキストで LLM に流れ込んでいた。さらに下流で news-creator の GPU キューが満杯化し、`ollama_chat_stream_failed: status 429 queue full` が発生。ユーザー体験としては「RAG が死んだ」「Augur が落ちる」として観測された。

静的解析で全 consumer を洗い出した結果、search-indexer へ 401 を踏んでいた呼び出し元は次の 2 件:

1. **rag-orchestrator → REST /v1/search**: `SearchIndexerClient.Search()` / `SearchBM25()` が `X-Service-Token` ヘッダを付けていなかった (`internal/adapter/rag_http/search_indexer_client.go`)。
2. **alt-backend → Connect-RPC SearchService**: `NewConnectSearchIndexerDriver(baseURL)` が `http.DefaultClient` を使って interceptor なしで初期化されていた (`app/driver/search_indexer_connect/client.go`)。Tag Trail / Morning Letter / Recap Search の検索がすべて 401。

より根本的な問題として、Pact CDCT は導入済みだったにもかかわらず、この incident を検出できなかった:

- `rag-orchestrator → search-indexer` と `alt-backend → search-indexer` の consumer pact ファイルが `pacts/` 配下に存在しなかった
- search-indexer の provider 検証テストも存在せず、既存の `acolyte-orchestrator-search-indexer.json` さえ実際には verify されていなかった
- tdd-workflow スキル側の CDC 対応表が consumer/provider 方向を取り違えており、欠落が見過ごされた

つまり、認証ヘッダ追加という「プロバイダ側の要件強化」に対して、consumer 側の契約を一斉更新する運用 playbook が欠けていた。

## Decision

**search-indexer の必須ヘッダ強制に対して、Pact CDCT を二重ガードとして成立させる。** 具体的には次の 4 点を採用する。

1. **consumer 側クライアントを X-Service-Token 認証付きで再構成する**
   - rag-orchestrator: `SearchIndexerClient` に `serviceToken string` を追加し、`Search()` / `SearchBM25()` で毎リクエストにヘッダを注入する。DI では既にロード済みの `cfg.Backend.ServiceToken` を注入する。
   - alt-backend: preprocessor_connect で実証済みの `serviceTokenTransport` パターンを search_indexer_connect にも移植する。`NewConnectSearchIndexerDriver(baseURL, serviceSecret)` のシグネチャに拡張し、Connect-RPC クライアントに `connect.WithProtoJSON()` を追加して JSON wire format に統一する（契約・監査ツールが読めるようにするため）。

2. **RED→GREEN の Pact consumer テストを新規追加する**
   - `rag-orchestrator/internal/adapter/contract/search_indexer_consumer_test.go`: REST `Search()` / `SearchBM25()` の X-Service-Token 送信を契約として固定する。
   - `alt-backend/app/driver/search_indexer_connect/contract/consumer_test.go`: Connect-RPC `SearchArticles` / `SearchRecaps` の X-Service-Token 送信を固定する。

3. **search-indexer 側に Go provider verification を新設する**
   - `search-indexer/app/driver/contract/provider_test.go` を追加。実際の `middleware.ServiceAuthMiddleware` でラップしたスタブ HTTP server を起動し、新規 consumer pact を verify する。
   - `scripts/pact-check.sh` に "Go: search-indexer provider" ステップを追加し、CI 前段の回帰ゲートに組み込む。

4. **tdd-workflow スキルを改訂し、provider が要件を強める変更を専用の playbook に載せる**
   - Phase 0 のトリガーに「必須ヘッダ追加」「認証昇格」「mTLS enforcement」を追加。
   - 新セクション "Phase 0b: Provider-adds-requirement Playbook" を追加。consumer 列挙 → pact 棚卸し → 欠落 consumer の RED 先行 → provider verify への登録 → pact-check.sh 全合格ゲートという 6 ステップを強制する。
   - CDC 対応表を `A → B`（A が B を consume）方向に統一し、provider 視点の逆引き表を追加して、プロバイダ変更時に consumer 漏れを目視で検出できるようにした。

### 検討したが採用しなかった選択肢

- **request filter で provider 検証時に自動的にヘッダを注入する**（Pact 推奨パターンの一つ）: [[000722]] の「未送信 consumer を検出する」目的を損なう。consumer が実際にヘッダを送っている事実を契約で固定する目的から逆行するため却下。
- **クライアント証明書（mTLS）のみで保護し、アプリ層トークンを廃止する**: search-indexer は ADR-000717 の alt-backend 内部 API と整合した shared secret モデルを採用しており、単一マシン上で mTLS とアプリトークンを併用する defense-in-depth を維持する方が侵入経路を狭められる。
- **Pact Broker + can-i-deploy を採用する**: ローカル運用中心の現状で Broker を常駐させるコストが見合わない。`scripts/pact-check.sh` の file-based モードで同等の「全 consumer + 全 provider 検証」をまず成立させ、必要になった段階で Broker へ移行する。

## Implementation

### 1. rag-orchestrator の X-Service-Token 注入

`internal/adapter/rag_http/search_indexer_client.go`:

```go
type SearchIndexerClient struct {
    BaseURL      string
    Client       *http.Client
    ServiceToken string
}

func NewSearchIndexerClient(baseURL string, timeout int, serviceToken string) *SearchIndexerClient {
    return &SearchIndexerClient{
        BaseURL: baseURL,
        Client: &http.Client{Timeout: time.Duration(timeout) * time.Second},
        ServiceToken: serviceToken,
    }
}

func (c *SearchIndexerClient) setAuthHeader(req *http.Request) {
    if c.ServiceToken != "" {
        req.Header.Set("X-Service-Token", c.ServiceToken)
    }
}
```

`Search()` / `SearchBM25()` の `http.NewRequestWithContext` 直後に `c.setAuthHeader(req)` を呼ぶ。DI 側 (`internal/di/container.go:79`) は `cfg.Backend.ServiceToken` を注入。起動ログに `connect_rpc_service_token_configured` が記録される。

### 2. alt-backend Connect-RPC のインターセプタ

`app/driver/search_indexer_connect/client.go` に preprocessor_connect と同じ `serviceTokenTransport` と `newServiceTokenClient` を追加。`NewConnectSearchIndexerDriver(baseURL, serviceSecret)` は:

```go
func NewConnectSearchIndexerDriver(baseURL, serviceSecret string) search_indexer_port.SearchIndexerPort {
    client := searchv2connect.NewSearchServiceClient(
        newServiceTokenClient(serviceSecret),
        baseURL,
        connect.WithProtoJSON(),
    )
    return &ConnectSearchIndexerDriver{client: client}
}
```

`app/di/infra_module.go:79` で `cfg.InternalAPI.ServiceSecret`（既に `SERVICE_SECRET_FILE` からロード済み）を渡す。

### 3. Pact 契約の追加

- **consumer pact**: Go の `pact-go v2` で consumer mock を起動し、X-Service-Token ヘッダを `matchers.Like("test-service-token")` として契約に固定する。成果物は `alt-backend/pacts/alt-backend-search-indexer.json` と `rag-orchestrator/pacts/rag-orchestrator-search-indexer.json`。
- **provider verification**: `search-indexer/app/driver/contract/provider_test.go` が `middleware.ServiceAuthMiddleware` を実体で使うスタブ HTTP server を立て、`provider.VerifyRequest` で上記 2 つの pact を replay する。state handler は `search-indexer has no matching articles` を含む acolyte 側の状態にも一応対応する（将来 acolyte の consumer テストを X-Service-Token 付きに更新したら追加で verify 対象に戻す）。

### 4. スキル改訂

`.claude/skills/tdd-workflow/SKILL.md`:

- Phase 0 トリガーに 3 項目追加（必須ヘッダ変更 / 認証昇格 / mTLS enforcement）
- 新セクション "Phase 0b: Provider-adds-requirement Playbook"（6 手順）
- CDC 対応表を `A → B` 方向に統一し、provider 視点の逆引き表を新設
- Service Boundary Checklist に「Required headers」「mTLS peer allowlist」「Service token env end-to-end」「CA bundle 存在」「Provider-side pact 検証」を追加
- Anti-Patterns #11-13 に新規項目を追加し、Pact 公式 4 リンクをリファレンスに追加

## Consequences

### Pros

- RAG/Augur の 401 停止が解消され、retrieval の bm25/tag 両経路が有効化される
- alt-backend の Tag Trail / Morning Letter / Recap Search の検索も同時に回復
- search-indexer の provider 検証が初めて CI 前段の回帰ゲートに組み込まれ、同種の「provider 側要件強化 + consumer 未追従」バグが file-based Pact 検査で再発検出可能になる
- tdd-workflow スキルが auth/mTLS 変更を boundary change として明示的に扱うため、将来のメンバーは Pact 漏れのまま実装に入れない
- Connect-RPC を JSON wire format に統一したことで、pact ファイル内のリクエストボディが人間可読になり、監査しやすくなる

### Cons / Tradeoffs

- alt-backend の Connect-RPC クライアントが `application/proto` から `application/json` に変わったため、search-indexer 向けリクエストの wire コストがわずかに増える。検索トラフィック量では無視できる範囲だが、将来大容量 RPC（ストリーミング等）を追加する際は再評価する。
- acolyte-orchestrator の既存 consumer pact (`acolyte-orchestrator-search-indexer.json`) は X-Service-Token を契約で固定していないため、provider verification から一旦除外されている。実 gateway は [[000722]] 対応済みだが、契約自体のガードは後続タスクで補う必要がある。
- Pact file-based モードで provider verification を二重ガード化したが、これは shared secret 自体の漏洩対策ではない。secret のローテーションや流出検知は別途 `SERVICE_SECRET_FILE` 経由の運用（Docker secrets）で管理する前提。
- provider verification テストのスタブは search-indexer の本物の REST ハンドラ / Meilisearch を使っていないため、response body の schema 齟齬には弱い。将来的には integration-tag テストで Meilisearch を立てた provider 検証に昇格させる余地がある。

### Security Audit の残課題（別 issue 化）

- search-indexer の `RequireServiceAuth` は 401 時の WARN ログに peer identity（TLS peer / CN）を含めていない。mTLS Phase 2 で入った `peer_identity_http_middleware` と同等の実装を入れて、失敗時の caller を常にログに残す。
- rag-orchestrator の retrieval 経路で、`bm25_results=0` かつ `tag_search` も失敗したケースでも LLM を呼び出している。retrieval が空のときは `insufficient context` を返す guard を追加する。
