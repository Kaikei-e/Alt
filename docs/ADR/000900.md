# ADR-000900: tts-speaker の TTS エンジンを Kokoro-82M から Qwen3-TTS-12Hz-0.6B-CustomVoice に置き換える

## Status

Accepted

## Date

2026-05-15

## Affected Services

- tts-speaker (Python): TTS エンジンを `kokoro` + `misaki[ja]` から `qwen-tts` (Apache 2.0) へ全面差し替え
- alt-frontend-sv (TypeScript / Svelte): `tts.ts` の `DEFAULT_VOICE` を新 voice ID へ刷新
- alt-butterfly-facade (Go): proto / Pact 契約に破壊変更なし。実装変更なし

## Context

`tts-speaker` の音声品質に対するユーザフィードバックが明確に否定的だった。特に Kokoro-82M の日本語 voice (`jf_alpha` / `jm_kumo` 等) は長文プロソディが破綻し、[[000304]] の `JAPANESE_SPLIT_PATTERN` 補正と [[000233]] の alkana 前処理を経ても根本解決に至らなかった。`hexgrad/Kokoro-82M` は 2026 年時点で JA 後継版 (v2 / Pro) を出しておらず、Kokoro v1.1 系は中国語専用。日本語 TTS の自然性で抜本改善するには別エンジンへの移行が必要だった。

調査経緯は次の通り:

1. **当初候補: Qwen/Qwen3-TTS-12Hz-1.7B-Base** — 品質は最も期待できるが、公式ドキュメントは `flash_attention_2` 前提 (CUDA-only) で記述されており、`tts-speaker` の運用環境 (AMD ROCm の self-hosted iGPU、CUDA 不可) とは整合しないと判断していた。
2. **代替候補: Style-Bert-VITS2 (litagin02)** — 日本語専用、CPU 動作、MOS 4.37 の高品質を確認。ただし**ライブラリ本体が AGPL-3.0** で、Alt のように HTTP/Connect-RPC で提供される microservice は AGPL の network-use clause を踏み、サービス利用者に対するソース開示義務が発生する。Alt の依存ポリシー (MIT / Apache 2.0 のみ許可) と非整合のため不採用。
3. **採用: Qwen/Qwen3-TTS-12Hz-0.6B-CustomVoice** — 追加調査で次が判明:
   - `attn_implementation="sdpa"` を指定すれば flash-attn なしで AMD ROCm 上で動作する。
   - GGUF 経路 (Ollama / llama.cpp) は構造的に不成立: Qwen3-TTS は AR-LM + 独自 codec decoder の二段構成で、codec decoder は Transformer LM でないため llama.cpp 標準のパスに乗らない。
   - 0.6B variant + CustomVoice バリアントは preset 話者 9 名を内蔵 (`model.get_supported_speakers()` で列挙可)、参照音声不要。
   - ライセンスは model weights / codec / `qwen-tts` PyPI 全てが Apache 2.0 で、Alt の license-posture を満たす。
   - self-hosted TTS GPU ノード上の iGPU VRAM 余裕で 0.6B (~1.2GB bf16) は収まる。dev CPU 計測で RTF ~7-8、refine.md ベンチでは ROCm iGPU 上 RTF ~1.0 が見込まれる。

実機試聴: dev CPU 上で 9 preset 全てに同一テキスト「今日は5月15日、東京は穏やかな晴れです。AltというRSSアプリの新機能をご紹介します。」を `language="Japanese"` + 共通 instruct で生成して比較 (`voice-samples/qwen-<speaker>.wav`、`voice-samples/manifest.json`)。ユーザ評価により採用は次の 3 preset:

1. `sohee` (best)
2. `serena`
3. `ono_anna`

他 6 preset (`aiden` / `dylan` / `eric` / `ryan` / `uncle_fu` / `vivian`) は採用しない。

また実機出力から、Qwen3-TTS-12Hz の默認デコーダの **サンプルレートは 24 kHz**(モデルカードに 44.1 kHz と書かれているがコミュニティ報告と一致する 24 kHz が実態) と確認できた。これは `audio_wav` バイト列の WAV header と Connect-RPC response の `sample_rate` フィールドで搬送される。

## Decision

`tts-speaker` の TTS エンジンを **`Qwen/Qwen3-TTS-12Hz-0.6B-CustomVoice` (Apache 2.0, qwen-tts PyPI)** に置き換える。実装方針は以下のとおり:

### エンジン構成

- `Qwen3TTSModel.from_pretrained(model_id, device_map=<cuda:0|cpu>, dtype=torch.bfloat16, attn_implementation="sdpa")` でロードする。**`flash-attn` は ROCm 上で動かないため絶対にインストールしない** (Dockerfile で `requirements.txt` から flash-attn を sed で除去し、`uv pip install qwen-tts && uv pip uninstall flash-attn` のフォールバックも併設)。
- 推論は `model.generate_custom_voice(text, language="Japanese", speaker=<preset>, instruct=<japanese_instruct>)` を blocking call として `asyncio.run_in_executor` 経由で実行。返り値 `(wavs, sr)` の `sr` を proto response にそのまま流す。
- レガシーの `speed` パラメータ (0.5–2.0) は Qwen にスカラ概念がないため、`instruct` 文字列への自然言語ヒント (「ゆっくりと」「少し速めに」) に射影する。
- `SynthesizeStream` の chunk は文単位。[[000304]] の `JAPANESE_SPLIT_PATTERN = r"(?<=[。！？\n])"` を流用して `re.split` し、文ごとに `generate_custom_voice` を呼んで `(audio, sr)` を queue 経由で yield する。
- [[000233]] の alkana ベース English→Katakana 前処理 (`tts_speaker/core/preprocess.py`) は Phase 2 として継続。Qwen3-TTS は英数字を多言語ネイティブで読めるが、Recap など Alt の他経路で品質安定が確認できるまで前処理は維持し、廃止は別 ADR に分離する。

### Voice ID 刷新

`VOICES_CONFIG` を 3 entries に再編。Alt-facing voice ID は API 互換性のため抽象化し、Qwen 内部の preset 名は内部マッピングに閉じ込める:

| Voice ID | Qwen speaker | Gender | 既定 |
|----------|--------------|--------|------|
| `qwen-ja-1` | `sohee` | female | 既定 |
| `qwen-ja-2` | `serena` | female | |
| `qwen-ja-3` | `ono_anna` | female | |

`TTS_DEFAULT_VOICE` は `qwen-ja-1` (= sohee) を既定とする。`TTS_DEFAULT_SPEED` は Kokoro 時代の 1.25 から 1.0 に戻す (Qwen は速度を `instruct` で間接表現するため、UI のデフォルト送信値を natural 側に寄せる)。

旧 voice ID (`jf_alpha` / `jf_gongitsune` / `jf_nezumi` / `jf_tebukuro` / `jm_kumo`) は廃止。FE / BFF / `.env.template` / docs を同じ commit で差し替え、hard cutover とする (feature flag や並存運用は採らない)。

### 設定値

新しい環境変数 (compose `compose.tts.yaml` + `.env.template`):

- `TTS_QWEN_MODEL_ID` = `Qwen/Qwen3-TTS-12Hz-0.6B-CustomVoice` (デフォルト)
- `TTS_QWEN_DTYPE` = `bfloat16`
- `TTS_QWEN_ATTN` = `sdpa` (ROCm 上では絶対 `flash_attention_2` に変更しない)
- `HSA_OVERRIDE_GFX_VERSION` / `PYTORCH_ROCM_ARCH` は **operator の `.env` で個別設定**し、compose に固定値をハードコードしない (host hardware に依存する値のため)

`compose.tts.yaml` の `healthcheck.start_period` は 120s → **180s** に拡大した (model download + ROCm カーネル JIT warmup を初回起動に吸収する)。

### サンプルレートの dynamic 化

連結器側で 24 kHz をハードコードしていた `tts_speaker/app/connect_service.py` の `SAMPLE_RATE = 24000` を撤去。pipeline 戻り値 `(audio, sr)` の `sr` を proto response の `sample_rate` フィールドに動的にコピーする。Pact contract (`pacts/alt-butterfly-facade-tts-speaker.json`) の `sampleRate` は matcher `type` で dynamic 値扱いなので、24 kHz → 異なるレートに将来変わっても契約破壊しない設計を保つ。

### 検討した代替案と却下理由

| 候補 | 評価 | 却下理由 |
|---|---|---|
| Qwen3-TTS-12Hz-**1.7B**-CustomVoice | 品質は最高 | RTF が 0.6B の 3 倍弱、self-hosted GPU ノードでは長文 narration の許容を超える可能性。1.7B への switch は **品質要件が満たされなかった場合の別 ADR** として open に残す |
| Qwen3-TTS-12Hz-0.6B-**Base** + 3 秒参照音声クローン | voice 個性をカスタム可能 | 参照音声 + transcript を coupled に同梱する CI / 検証コストが過剰。初回移行のスコープから外す |
| Style-Bert-VITS2 JP-Extra (litagin02) | JA 専用、MOS 4.37 | **AGPL-3.0** が Alt の license-posture に違反 (MIT / Apache 2.0 のみ許可) |
| GPT-SoVITS v4 | MIT、品質高い | Qwen lineage を維持したいユーザ意向と整合せず、CPU 動作も遅め |
| MeloTTS | MIT、軽量 | 品質が Kokoro と同等〜やや上にとどまり、本来の課題 (Kokoro の不自然さ) を解決しない |
| Ollama / llama.cpp 経由の GGUF 量子化 | Apache 2.0 stack を維持したまま運用簡素化 | Qwen3-TTS の codec decoder は Transformer LM ではないため llama.cpp で推論不可。Ollama TTS サポート (issue #11021) は実装されていない。経路自体が成立しない |
| Kokoro 継続 + 後段補正 (SSML 風 + voice mix) | エンジン移行リスクなし | Kokoro 自体の音響モデル限界が原因で、後段補正で改善し切れない |

## Consequences

### Pros

- **日本語自然性の根本改善**: Kokoro の長文プロソディ破綻が解消され、`jf_*` / `jm_kumo` への否定的フィードバックの根を断つ。
- **Apache 2.0 一気通貫**: model weights / codec / Python ライブラリ全てが Apache 2.0。Alt 全体の license-posture と整合し、AGPL 由来のソース開示リスクが消える。
- **依存表面の縮小**: 旧 `kokoro` + `misaki[ja]` + unidic 辞書 (~700MB) を撤去。`qwen-tts` は transformers + torchaudio で自己完結。
- **proto / Pact 契約は無変更**: `audio_wav: bytes` / `sample_rate: int32` / `duration_seconds: float` のスキーマは流用、`sampleRate` matcher が `type` で dynamic のため値変更も契約破壊しない。BFF / 既存 Pact ファイルへの影響なし。
- **テスト基盤の汎用化**: `connect_service.py` の `SAMPLE_RATE` ハードコードを撤去。将来 48 kHz fine-tuned decoder などに切り替えても呼び出し側は touch 不要。
- **voice ID API の安定化**: Qwen の preset 名 (`sohee` 等) を外部に露出せず、`qwen-ja-N` の抽象 ID に閉じ込めた。preset 入れ替え時に FE / Pact を触らずに済む。
- **flash-attn 排除の明文化**: Dockerfile + ADR 双方で flash-attn インストール禁止を宣言。将来の貢献者が誤って ROCm 環境に CUDA 専用ライブラリを引き込む事故を構造的に防ぐ。

### Cons / Tradeoffs

- **モデルサイズ増 (~10×)**: Kokoro 82M (~330MB) → Qwen3-TTS 0.6B (~1.2GB bf16) + codec + DeBERTa BERT。`HF_HOME` volume の必要容量と初回 download 時間が増える。`start_period=180s` で吸収しているが、cold start ペナルティはユーザ体験に出る。
- **RTF の悪化**: Kokoro は modest GPU で RTF 36–96×、CPU でも実用域。Qwen3-TTS 0.6B は AR でメモリ帯域律速、ROCm iGPU 上で RTF ~1.0、dev CPU では RTF ~7-8。CPU フォールバックは batch 用途限定で interactive UX には使えない。
- **ROCm 検証コスト**: `attn_implementation="sdpa"` で動作する想定だが、self-hosted TTS GPU ノードの具体 GFX バージョンで未実機検証。最悪 `TTS_FORCE_CPU=1` 運用に落ちる可能性があり、その場合は長文 narration が batch 化される。
- **voice 数の縮小 (5 → 3)**: 旧 5 voice (jf_*/jm_*) から 3 voice (qwen-ja-1/2/3、すべて女性) に減少。男性 voice は今回は提供しない。要件が出たら preset `eric` などを `qwen-ja-4` 等に追加する別 ADR で対応する。
- **`speed` のセマンティクス変化**: 旧 Kokoro は scalar speed (0.5–2.0) で連続変化、Qwen は離散 3 段階 (slow / neutral / fast) を `instruct` 文字列で表現。微細な速度制御は失われる。
- **issue #297 (open) リスク**: Qwen3-TTS GitHub の `2048 vs 1024` dim mismatch (0.6B-Base 系) は fine-tune 経路のバグで inference には影響しないと報告されているが、upstream で未解決。万一 inference にも波及した場合は 1.7B variant に切り替える別 ADR を発行する。
- **Phase 0 (E2E Playwright) は別作業**: 試聴 (`voice-samples/`) と unit + CDC + lint / type-check は green。FE Playwright + docker compose による end-to-end 確認は別途実機実施が必要 (本 ADR の範囲外)。

## Deploy-model 整合性セルフチェック

- [x] 新設 compose service が `depends_on` で他 service を gate するか？ — **No.** 既存 `tts-speaker` の compose 構成を変更しただけで、新しい service / `depends_on` 関係の追加なし。
- [x] 新設 compose service に `restart: "no"` の one-shot init container を含むか？ — **No.** init container は追加していない。
- [x] 新設名前付き volume を populate する責務はどこにあるか？ — **既存 `tts_models` volume の利用パターン継続**。populate は `Qwen3TTSModel.from_pretrained` 初回呼出時の HF download に委ねる (host bootstrap / baked image / CI artefact fetch には依存しない)。`start_period=180s` で初回起動時の cold download を吸収する。
- [x] いずれも該当しない: **rolling 互換** と明記して pass。
