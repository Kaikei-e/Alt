# ADR-000485: Knowledge Home Phase 0 観測性配線と service_quality 3値モデルを実装する

## Status

Accepted

## Date

2026-03-20

## Affected Services

- alt-backend (Go)

## Context

Knowledge Home の Phase 0（契約固定と観測性の土台づくり）において、以下の4つのギャップが残っていた。

1. **観測性未配線** — `KnowledgeHomeMetrics` 構造体は `utils/otel/knowledge_home_metrics.go` に26メトリクス分定義済みだったが、handler / projector / DI のいずれにも接続されておらず、OTel メトリクスが一切出力されていなかった。
2. **`service_quality = "fallback"` 未実装** — canonical contract（[[knowledge-home-phase0-canonical-contract]]）は `full / degraded / fallback` の3値を定義しているが、handler は `full / degraded` の2値しか返していなかった。
3. **stream delivery count 未計測** — ストリーム接続数・切断数は `StreamConnectionsTotal` / `StreamDisconnectsTotal` として定義されていたが配線されておらず、配信イベントごとの lag 計測も行われていなかった。
4. **logger nil-safe 化の不備** — `knowledge_projector_runner.go` と `knowledge_projector.go` が `logger.Logger.ErrorContext()` 等を直接呼び出しており、`logger.Logger` が nil の場合（テスト環境など）にパニックする危険があった。

## Decision

以下の4領域を一括して実装し、Phase 0 の観測性要件を完遂する。

### A. 観測性配線

**A1. DI に KnowledgeHomeMetrics を登録する。**

`di/container.go` に `KnowledgeHomeMetrics` フィールドを追加し、`NewKnowledgeHomeMetrics()` で初期化する。初期化失敗時は `slog.Warn` して nil を保持し、サービスの起動を妨げない（fail-open）。

**A2. Handler にメトリクスをワイヤリングする。**

`Handler` 構造体に `metrics *otel.KnowledgeHomeMetrics` フィールドを追加し、constructor の引数として受け取る。全メトリクス呼び出しは `if h.metrics != nil` ガード付きとし、既存テストは nil metrics で動作を維持する。

インストルメント対象:
- `GetKnowledgeHome`: `RequestsTotal`, `RequestDurationSeconds`, `DegradedResponsesTotal`, `EmptyResponsesTotal`
- `TrackHomeItemsSeen`: `ItemsExposed`, `TrackingReceivedTotal`, `TrackingPersistedTotal`, `TrackingFailedTotal`
- `TrackHomeAction`: `ItemsOpened`, `ItemsDismissed`（action type 別）
- `StreamKnowledgeHomeUpdates`: `StreamConnectionsTotal`, `StreamDisconnectsTotal`, `StreamUpdateLagSeconds`

**A3. Projector にメトリクスをワイヤリングする。**

`KnowledgeProjectorConfig` に `Metrics` フィールドを追加し、`processKnowledgeEvents()` に伝搬する。

インストルメント対象:
- バッチ処理後: `ProjectorEventsProcessed`, `ProjectorBatchDurationMs`, `ProjectorErrors`
- チェックポイント更新後: `ProjectorLagSeconds`（現在時刻 − 最新イベント OccurredAt）

**A4. logger nil-safe 化を徹底する。**

`knowledge_projector_runner.go` の6箇所と `knowledge_projector.go` の13箇所で `logger.Logger.ErrorContext()` / `logger.Logger.WarnContext()` / `logger.Logger.InfoContext()` を nil-safe wrapper（`logger.ErrorContext()` / `logger.WarnContext()` / `logger.InfoContext()`）に置き換える。

### B. `service_quality` 3値モデルの実装

`get_knowledge_home_usecase` の `Result` 構造体に `ServiceQuality string` フィールドを追加し、`computeServiceQuality()` 関数で3値を判定する。

| 値 | 条件 |
|---|---|
| `full` | 全 read source（items, digest, lens）が正常 |
| `degraded` | items 取得成功だが一部 source が失敗（digest 取得失敗、lens 解決失敗など） |
| `fallback` | items 取得失敗、または projection checkpoint が `fallbackStalenessThreshold`（15分）を超過 |

handler は `result.ServiceQuality` が設定されていればそれを使い、空文字列（ゼロ値）の場合は従来の `Degraded` フラグから `full` / `degraded` を導出する（後方互換）。

### 代替案

- **メトリクスを middleware で一括計測する案**: Connect-RPC interceptor でリクエスト数・レイテンシを計測する方法も検討したが、`service_quality` や `empty_responses` のようなビジネスロジック依存メトリクスは handler 内でのみ判定可能なため、handler 内でのインストルメントを採用した。
- **`fallback` 条件に projection checkpoint lag を含めない案**: items 取得失敗のみを fallback とする簡易実装も検討したが、projection が大幅に遅延している状態はデータの鮮度が著しく低いため、ユーザーに正しく伝えるべきと判断した。閾値は 15 分とした（`FreshnessThreshold` の5分の3倍）。

## Consequences

### Pros

- Phase 0 canonical contract の全観測性要件が充足される
- `service_quality` の3値により、フロントエンドが `fallback` 時に適切な UI（キャッシュ表示・リトライ案内等）を提供可能になる
- projector lag のリアルタイム計測により、SLI-A（可用性）ダッシュボードの構築が可能になる
- nil-safe logger により、テスト環境やグレースフルシャットダウン中のパニックリスクが排除される
- 全メトリクスが nil ガード付きのため、OTel 未設定環境でも安全に動作する

### Cons / Tradeoffs

- handler constructor の引数が1つ増えた（`metrics *otel.KnowledgeHomeMetrics`）。既存の全テストヘルパーと `server.go` の呼び出し箇所を更新する必要があった
- `fallbackStalenessThreshold` の 15 分は固定値。将来的に config 化が必要になる可能性がある
- stream 配信カウントは `StreamUpdateLagSeconds` histogram で代用しており、専用の `StreamDeliveryCount` counter は未追加。必要になった時点で `knowledge_home_metrics.go` に追加すればよい
