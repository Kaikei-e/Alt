# ADR-000870: Knowledge Home の "Archived · No source URL" 残ギャップ 4 件を ADR-867/869 follow-up として塞ぎ切る

## Status

Accepted

## Date

2026-04-29

## Affected Services

(frontmatter 参照)

## Context

### 直接の動機

[[000867]] / [[000868]] / [[000869]] (2026-04-28) は Knowledge Home の "Archived · No source URL" kicker を構造的に終わらせるための 3 段階の決定だった: (1) Article URL を全層 canonical (`URL` / `url`) に統一、(2) Feed.Link → WebsiteURL 命名違反を CI lint で防衛、(3) `EmitArticleUrlBackfill` admin Connect-RPC + UI ボタンで operator が 1 ボタンで `ArticleUrlBackfilled` corrective event を発火できる経路を整備。

ADR-867 / ADR-869 の Cons セクションに「follow-up に回した残務」が 4 件明記されていた:

| # | 残務 | 出典 |
|---|---|---|
| P1.1 | `Repository.UpsertKnowledgeHomeItem` UPSERT に SQL `CASE` が 5 件残存 (title / summary_excerpt / tags_json / summary_state / url)。memory `feedback_merge_safe_upsert.md` で禁じている "業務ロジックを SQL に逃がす" 違反 | immutable-design-guard 監査結果 |
| P1.2 | `AppendKnowledgeEventPort` が void return のため、`EmitArticleUrlBackfill` の `skipped_duplicate` カウンタが常に 0 を返し、ADR-869 proto 契約と drift。operator は idempotent 再実行の意味を確認できない | usecase.go:142-151 の自己 TODO |
| P1.3 | `/admin/knowledge-home` の Emit URL Backfill ボタン flow に Playwright FE E2E が無い | ADR-867 Cons "E2E (Hurl/Playwright) は未追加 (follow-up)" |
| P2 | `ArticleUrlBackfilledPayload` に fact-time が含まれていない。Verraes' multi-temporal events パターン (`recorded_at` と domain-specific event time の分離) と drift | web-researcher 結果 (verraes.net 2022) |

operator が "Emit URL Backfill" ボタンを押せば backfill 自体は走るが、上記 4 件を残したまま運用を始めると次の事故が確定的に再発する: SQL CASE 由来の `summary_state` regression (incoming `pending` が existing `ready` を上書き)、操作 UI が「いま何件 dedupe 済みなのか」を嘘の 0 で表示し続ける、Pact / Playwright が無いので contract drift が次の rolling deploy を silent に壊し得る。本 ADR は ADR-867 / ADR-869 の最後の 4 件をまとめて消化する follow-up である。

### 不変条件監査

`/immutable-design-guard` で `repository.go` UPSERT を再監査した結果、以下が違反 / 違反一歩手前と分類された:

- **Merge-safe upsert** ([[IMPL_BASE]] §7) — 5 列 ("title" / "summary_excerpt" / "tags_json" / "summary_state" / "url") が `CASE WHEN EXCLUDED.x != '' THEN EXCLUDED.x ELSE knowledge_home_items.x END` 形式。"業務判定 (空でない値を採用) を SQL に持ち込んでいる" の典型違反。さらに `summary_state` の CASE は `WHEN EXCLUDED='ready' THEN 'ready' WHEN EXCLUDED NOT IN ('','missing') THEN EXCLUDED ELSE existing END` で、incoming `'pending'` は case 2 を踏んで existing `'ready'` を上書きする — `.claude/rules/knowledge-home.md` の "summary_state は missing → pending → ready 一方向。逆行禁止" を実装上は破っていた潜在バグ
- **Event-time purity** ([[IMPL_BASE]] §3) — corrective event の payload は `article_id` / `url` のみで、original observed time が乗っていない。append-first / reproject-safe には抵触しないが、Verraes 流の "domain-specific event time を payload に持つ" 設計から外れていた
- **why_json merge** (line 115-141) は `SELECT DISTINCT ON ... source_rank` で event_seq 昇順 replay を前提に決定的 — 条件付き PASS。ただし将来 projector が並列適用に進む場合は `seq_hiwater` guard を追加する余地あり (本 ADR scope 外)

### 外部レファレンス

`/web-researcher` で confirmed (Tier S):

- [Multi-temporal Events (Verraes 2022)](https://verraes.net/2022/03/multi-temporal-events/) — "add a domain-specific property that reflects the time the event occurred. Name the property after its purpose in the domain"
- [Idempotent Command Handling (event-driven.io / Oskar Dudycz)](https://event-driven.io/en/idempotent_command_handling/) — Stripe-style idempotency keys + deterministic key generation. dedupe_key namespace 分離 (`article-url-backfill:` vs `article-created:`) はこの canonical pattern と整合
- [Microsoft Azure Event Sourcing Pattern](https://learn.microsoft.com/en-us/azure/architecture/patterns/event-sourcing) — "compensating event… with corrective values, references to the action being corrected"。`ArticleUrlBackfilled` はこの "explicit correction" を踏襲

## Decision

P1.1 + P1.2 + P2 + 境界テスト (P1.3 + Pact + Hurl 強化) を 1 セッションで実装し、ADR-867 / ADR-869 の Cons を完全に消化する。TDD outside-in (E2E → CDC → Unit) で進めた。3 commit にまとまる。

### D-1. `AppendKnowledgeEventPort` を `(eventSeq int64, err error)` に拡張 (P1.2)

- `alt-backend/app/port/knowledge_event_port/port.go`: signature 変更 + 規約コメント (`eventSeq == 0` ⇒ dedupe-hit)
- `alt-backend/app/driver/sovereign_client/read_client.go`: response から `event_seq` を取り出して返す (sovereign 側 `AppendKnowledgeEventResponse` は既に `event_seq` を含む)
- 9 production caller (`append_knowledge_event_usecase` / `create_summary_version_usecase` (×2) / `create_tag_set_version_usecase` (×2) / `track_home_action_usecase` / `recall_dismiss_usecase` / `recall_snooze_usecase` / `transition_knowledge_loop_usecase` / `knowledge_backfill_job` / `summary_narrative_backfill_job` / `outbox_worker` / `connect/v2/internal/handler`) を `_, err := ...` で機械的に追従。eventSeq を **使う** のは `knowledge_url_backfill_usecase` のみ
- 10 test mock (各 usecase / job / connect handler の stub) を `(int64, error)` に揃え、append 順に擬似 event_seq を返すよう mock を強化

### D-2. `ArticleUrlBackfilledPayload` に `OriginalOccurredAt` を追加 (P2)

- `alt-backend/app/domain/knowledge_event_payload.go`: 構造体に `OriginalOccurredAt string \`json:"original_occurred_at"\`` 追加。RFC3339 文字列で expressed
- producer (`knowledge_url_backfill_usecase.appendCorrective`) が `articles.created_at` を `time.RFC3339` で marshal して payload に格納
- projector は当面読まない (URL 値だけで `Patch*URL` driver を呼ぶため)。だが event log には残るので、将来 reproject で fact-time を区別したくなった瞬間に追加コストなしで取り出せる
- wire-form contract test (`TestArticleUrlBackfilledPayload_MarshalsCanonicalUrlKey` 拡張 + 新設 `TestArticleUrlBackfilledPayload_AcceptsEmptyOriginalOccurredAt`) で raw `map[string]any` から `"original_occurred_at"` キー有を assert (PM-2026-041 false-confidence 回避パターン継承)

### D-3. `EmitArticleUrlBackfill` カウンタを正しく数える (P1.2 producer 側)

- `knowledge_url_backfill_usecase.appendCorrective` を `eventSeq, err := u.eventPort.AppendKnowledgeEvent(...)` に切替
- `eventSeq != 0` を `appended` の真偽として返し、0 のときは `SkippedDuplicate++` に分類
- 新設 unit test `TestEmit_ReportsSkippedDuplicateAccurately` で 1 回目 `EventsAppended=3 / SkippedDuplicate=0`、2 回目 `EventsAppended=0 / SkippedDuplicate=3` を pin (mockEventPort が dedupe registry を simulate)
- 新設 unit test `TestEmit_PassesOriginalOccurredAtFromArticleCreatedAt` で `articles.created_at` (RFC3339) が wire 上の `"original_occurred_at"` キーに乗ることを assert

### D-4. `Repository.UpsertKnowledgeHomeItem` UPSERT を merge-safe に整流 (P1.1)

- `knowledge-sovereign/app/driver/sovereign_db/repository.go`:
  - `title` / `summary_excerpt` / `url` → `COALESCE(NULLIF(EXCLUDED.x, ''), knowledge_home_items.x)`
  - `tags_json` → `COALESCE(NULLIF(EXCLUDED.tags_json, '[]'::jsonb), knowledge_home_items.tags_json)`
  - `summary_state` → `GREATEST(knowledge_home_items.summary_state, EXCLUDED.summary_state)` (lexicographic monotonic latch: `'' < missing < pending < ready`、既存 `score = GREATEST(...)` と同形)
- `summary_state` の CASE 由来 latch 逸脱 (incoming `pending` が existing `ready` を上書き) を副次的に修正
- `why_json` の `SELECT DISTINCT ON ... source_rank` 構造はそのまま (event_seq 昇順 replay 前提では決定的)
- 新設 structural test `TestUpsertKnowledgeHomeItem_UsesMergeSafeSQL` が "5 件の禁止 CASE が無い" と "5 件の canonical 表現が含まれる" を 2 方向で assert。将来の regression は test 名でルールに直リンクする

### D-5. 境界テストを 3 層揃える (P1.3 / Pact / Hurl)

- **Pact CDC** (`article_url_backfill_consumer_test.go`):
  - response matcher に `eventSeq: matchers.Like(123)` を追加。Provider が event_seq を返さなくなる drift を CI で即検出
  - request payload に `"original_occurred_at": "2026-01-15T08:30:00Z"` を追加
- **Hurl E2E** (`e2e/hurl/alt-backend/72-admin-emit-article-url-backfill.hurl`):
  - 5 counter (articlesScanned / eventsAppended / skippedBlockedScheme / skippedDuplicate / moreRemaining) の型 (`isInteger` / `isBoolean`) を pin
- **Playwright FE E2E** (`tests/e2e/desktop/admin/admin-emit-url-backfill.spec.ts` 新設):
  - **live emit**: BFF から固定 counter を返し UI summary に "12252 appended / 27198 scanned · 14946 blocked-scheme" が表示される
  - **idempotent re-run**: 1 回目 `eventsAppended=3`、2 回目 `eventsAppended=0`。FE が二重押下を許容し UI summary を更新
  - **error path**: BFF が 500 を返したら error banner に "sovereign unreachable" が出る + projection-health snapshot は壊れない
- Page object `DesktopKnowledgeHomeAdminPage` に `emitUrlBackfillButton` / `urlBackfillResultSummary` / `errorBanner` の 3 locator を追加し、後続の admin spec が再利用できる形にした

### 検討した代替案

- **`AppendKnowledgeEventPort` を割らずに新 method `AppendKnowledgeEventReturningSeq` を増やす** — 却下。同じ機能の異なる 2 method が並ぶと caller がどちらを使うべきか毎回判断する必要があり、レビュー時に "どっち使えばいいの" が永続化する。signature 1 本に統一し、interest を持たない caller は `_, err :=` で読み捨てる方が読み手のコストが低い
- **`summary_state` の latch を Go projector 側に持ち上げ、SQL は `COALESCE(NULLIF(EXCLUDED.summary_state, ''), ...)` で単純 merge** — 却下。projector が "現在の read model state を読んで判定する" 形になり [[IMPL_BASE]] §4 (Reproject-safe projector — projector 内で active projection を読まない) を破る。`GREATEST` + lexicographic ordering は read model に触らずに同じ semantics を実現でき、実装も score 列と同形なので読み手が即座に意味を取れる
- **`summary_state` に `seq_hiwater` 列を Atlas migration で追加し、`WHERE EXCLUDED.seq > knowledge_home_items.summary_state_seq_hiwater` で並列適用も含めて pin** — 却下 (本 ADR scope 外)。Atlas migration が増える + 全 projector パスに影響するので別 ADR で扱う方が blast radius が見える。今回の lexicographic 解は projector が event_seq 昇順で replay する限り十分
- **`OriginalOccurredAt` を proto field として追加し canonical wire を proto に固定** — 却下 (本 ADR scope 外)。`knowledge_events.payload` は bytes opaque なので proto に追加すると "事実時刻を payload に持つ" 思想を proto レイヤに昇格する話になる。当面 Go domain 側 + wire-form contract test で pin する方が変更速度が速い
- **operator が UI ボタンを押す P0 を本 ADR scope に含める** — 却下。`feedback_plan_approval_is_not_build_permission.md` 準拠。実装の決定記録 (本 ADR) と本番反映 (operator action) を分離する
- **CASE を残しつつ comment で "merge-safe rule の例外" と書く** — 却下。memory `feedback_merge_safe_upsert.md` の意図は "新規コードに business CASE を増やさない" 構造的防衛で、example で例外を許すと壁が崩れる

## Consequences

### Pros

- **真因 fix の最終回**: PM-2026-041 起点の Knowledge Home `Archived · No source URL` 問題が、ADR-865 (不完全) → ADR-867 (canonical 統一 + corrective event) → ADR-869 (operator UI) → 本 ADR (counter 正確化 + merge-safe SQL + 境界テスト) で完全に塞ぎ切られた。これ以後は operator が UI ボタンを押すだけで本番復旧できる
- **summary_state の latch 逸脱バグも副次的に修正**: 既存 CASE は incoming `pending` が existing `ready` を上書きしうる構造だった。`GREATEST` 化で `.claude/rules/knowledge-home.md` の "ready 一方向 / 逆行禁止" を SQL レイヤで保証
- **dedupe-hit を operator に正しく見せる**: ADR-869 wire 契約と実装が一致。再実行 / 中断再開シナリオで "何件 dedupe で skip されたか" が嘘の 0 ではなく実数で表示される
- **3 層境界テスト揃った**: Pact CDC が wire shape の drift を、Hurl が BFF→backend Connect-RPC seam を、Playwright が UI counter rendering を、それぞれ独立に pin。1 つのレイヤだけが変わると最も近い境界テストが落ちて原因が即座に分かる
- **multi-temporal events ready**: `OriginalOccurredAt` が event log に残るので、将来 fact-time と recorded-time を区別したい projector が "どの時刻を業務的に使うべきか" を判断できる。append-first / reproject-safe は崩していない (event mutation も latest state 参照も無い)
- **structural test で merge-safe rule に直リンク**: `TestUpsertKnowledgeHomeItem_UsesMergeSafeSQL` が future regression のとき "merge-safe rule violated: forbidden CASE pattern X" と test メッセージで rule 名を出すので、次の人が memory を辿らずに済む

### Cons / Tradeoffs

- **port signature change の blast radius**: alt-backend 内 9 production caller + 10 test mock を機械的に追従した。proto wire は変えていない (sovereign-side は元々 event_seq を返していた) ので外部 contract への影響は無いが、PR レビュー時の差分は広い (約 30 ファイル)
- **`summary_state` の lexicographic ordering は将来の state 追加に脆い**: 今後 `'failed'` のような状態を追加する場合、`'f' < 'm'` で `missing` より小さくなり latch が壊れる。新規 state 追加時は alphabetical 位置を確認する手順を `.claude/rules/knowledge-home.md` に書き足す必要 (本 ADR では code comment に inline で残した)
- **`OriginalOccurredAt` 空文字許容**: `articles.created_at` が zero の article (古い import 失敗データ) では空文字で marshal される。projector は読まないので影響無いが、将来 fact-time を必須化する場合は別 backfill が必要
- **`SkippedDuplicate` 計上ロジックは sovereign の `event_seq=0 ⇒ dedupe` 規約に依存**: 規約が変わると counter が嘘になる。Pact CDC で response.eventSeq の存在を pin したことで "field が消える" drift は検出できるが、"`event_seq=0` の semantics を sovereign 側で変える" drift には ADR と runbook の review に頼る
- **operator action は別途**: P0 (UI ボタン → Reproject Full → Swap) は実装済み機能の本番起動であり、本 ADR scope 外。`feedback_plan_approval_is_not_build_permission.md` 準拠で、operator の明示指示を待つ
- **Playwright spec は CI に組み込み済みの assumption**: `bun run test:e2e` 経由で走る既存パイプラインに新 spec が乗る前提。新 spec が CI に含まれていない場合は別 PR で `.github/workflows/` に組み込む
