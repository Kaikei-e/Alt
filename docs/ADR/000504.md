# ADR-000504: k6 負荷試験起因のゴーストジョブ 190 万件を一括クリーンアップする

## Status

Accepted

## Date

2026-03-20

## Affected Services

- pre-processor (Go)

## Context

3/5〜3/13 の k6 負荷試験（累計 4 回、最大 135 万件/回）により、alt-backend の articles テーブルに大量の記事が作成され、それに伴い pre-processor DB の `summarize_job_queue` に対応する pending ジョブが投入された。

負荷試験終了後に記事は alt-backend から削除されたが、pre-processor 側のジョブは残存した。この「参照先記事が存在しないジョブ」をゴーストジョブと呼ぶ。

クリーンアップ前の状態:

| status | 件数 |
|--------|------|
| pending | 1,913,832 |
| completed | 133,483 |
| dead_letter | 1,053 |

影響:
- pre-processor が FIFO でジョブを取得するため、古いゴーストジョブが先頭に居座り続ける
- 取得 → article not found → failed → pending に戻る → 再取得という無限ループが発生
- 毎分約 120 件のエラーログが発生し、有効な新規ジョブが一切処理されない飢餓状態

## Decision

**2026-03-14 00:00:00 UTC をカットオフとし、それ以前の pending ジョブを一括で `dead_letter` に移行する。**

```sql
UPDATE summarize_job_queue
SET status = 'dead_letter',
    error_message = 'Bulk cleanup: article deleted after k6 load test',
    completed_at = NOW()
WHERE status = 'pending'
  AND created_at < '2026-03-14 00:00:00+00';
```

### カットオフ日の根拠

- 3/5〜3/13 が負荷試験期間（日次 3 万〜135 万件の記事投入）
- 3/14 以降は通常レート（数百件/日）に復帰
- 3/14 以降にもゴーストジョブは存在しうるが、件数が少ないため通常のリトライ → dead_letter フローで自然に処理される

### 検討した代替案

| 案 | 評価 |
|---|---|
| DELETE で物理削除 | 監査・分析ができなくなる。テーブル設計の正規フロー外 |
| article 存在チェック付き UPDATE | alt-backend への大量クエリが必要。DB 間結合は不可 |
| バッチ分割（1 万件ずつ） | 安全だが完了まで数時間かかり、その間飢餓状態が継続 |
| dead_letter 一括移行 | **採用**。正規ステータス遷移に沿い、後から識別可能 |

### 実行時の注意点

190 万行の UPDATE はトランザクションが大きいため、以下の対処が必要だった:

1. **デッドロック回避**: pre-processor のジョブワーカーが同じ行をロックするため、`docker compose stop pre-processor` でワーカーを停止してから実行
2. **残留トランザクションの終了**: 先行して失敗した UPDATE のトランザクションが ShareLock を保持し続けていたため、`pg_terminate_backend()` で明示的に終了

## Consequences

### Pros

- pending が 1,913,832 → 12,532 に削減（99.3% 減）
- `article not found` エラーループが即座に停止
- 有効な新規ジョブの処理が再開
- `dead_letter` + `error_message LIKE 'Bulk cleanup%'` で後から件数・分布を集計可能
- テーブル設計の正規ステータス遷移に沿っているため、既存の監視・集計クエリと矛盾しない

### Cons / Tradeoffs

- dead_letter が 1,053 → 1,902,400 に増加し、テーブルサイズが肥大化したまま残る。将来的に dead_letter の定期パージ（`DELETE WHERE status = 'dead_letter' AND completed_at < NOW() - INTERVAL '90 days'`）の検討が必要
- 3/14 以降のゴーストジョブ（推定数百件）は本クリーンアップの対象外。通常のリトライ上限（max_retries=3）で dead_letter に遷移するが、その間はエラーログが少量発生する
- 一括 UPDATE は WAL を大量生成する（約 540MB）。レプリケーション環境では事前にディスク容量を確認すべき
