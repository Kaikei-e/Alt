# ADR-000681: Planner を fixed skeleton に正式化し ES/Conclusion を accepted claims ベースの deterministic synthesis に移行する

## Status

Accepted

## Date

2026-04-10

## Affected Services

- acolyte-orchestrator (Python)

## Context

v5 パイプラインでは以下の品質問題が残っていた:

**Planner の不安定性 (exec3 Issue 4)**
- Planner は skeleton + query expansion 構成で完走していたが、「正式仕様」ではなく quality hotfix の状態だった
- prompt 内の JSON schema が `str()` で Python repr (single quotes) になっており Gemma4 の解釈を阻害していた
- scope に brief dict 全体を渡しており prompt が冗長だった
- 未知の report_type で `_default_fallback_sections` が search_queries を pre-bake するが、後続処理で上書きされる dead code があった
- fallback 発動時の structured log がなく、success との区別が不可能だった

**ES/Conclusion の空本文と重複 (exec3 Issue 5)**
- ES は deterministic fallback で 349 chars / 3 citations、Conclusion は 48 chars と弱かった
- `_deterministic_conclusion_claims` / `_deterministic_es_claims` が analysis_claims=[] AND extracted_facts=[] で空リストを返し、WriterNode が `sections[key] = ""` を生成する空本文カスケードが存在した
- Conclusion prompt に analysis 本文が渡されないため Analysis の文をそのまま再掲 → FM8 (bigram Jaccard > 0.20) blocking → revision → 再度重複のループが発生していた

## Decision

### Issue 4: Planner fixed skeleton 正式化

`REPORT_TYPE_SKELETONS` と `QueryExpansionOutput` による skeleton + query expansion パターンを正式仕様として確定し、以下の修正を行う:

1. **`DEFAULT_SKELETON` 定数を導入**: 未知の report_type 用の bare skeleton (search_queries なし)。`_default_fallback_sections` を廃止
2. **prompt 内 schema を `json.dumps()` で生成**: Python repr ではなく valid JSON を Gemma4 に渡す
3. **`scope=topic` に変更**: brief dict 全体ではなく topic 文字列のみを prompt に渡す
4. **fallback 時の structured log 追加**: `"Planner skeleton fallback activated"` イベントで report_type を記録

### Issue 5: ES/Conclusion deterministic synthesis

ES と Conclusion を accepted claims ベースの合成に寄せ、空振りと重複の両方を解消する。SectionPlannerNode は provisional な `claim_plans` と deterministic fallback を用意し、最終的な accepted-claim synthesis は WriterNode が paragraph acceptance 後の state から再構成する:

1. **SectionPlannerNode: `_topic_overview_claim` ヘルパー導入**: `_deterministic_conclusion_claims` / `_deterministic_es_claims` が analysis_claims=[] AND extracted_facts=[] でも topic ベースの最小 synthesis claim を 1 件返す。シグネチャに `topic: str` パラメータを追加
2. **WriterNode: 反復抑制コンテキスト注入 (`_build_prior_sections_context`)**:
   - Conclusion prompt に `<prior_analysis>` タグで analysis body (先頭 500 chars) を注入し「重複する表現を避けてください」を指示
   - ES prompt に `<prior_sections>` タグで全 section body (各先頭 300 chars) を注入し「簡潔に要約してください」を指示
3. **WriterNode: accepted paragraph claims を最優先する**:
   - Conclusion は accepted な analysis claims が揃っていれば、それを synthesis claims に変換して `claim_plans` を上書きする
   - ES は accepted な非 ES section claims が揃っていれば、それを synthesis claims に変換して `claim_plans` を上書きする
4. **WriterNode: `_deterministic_fallback_body` ヘルパー導入**: claim_plans が空でも ES/Conclusion に非空 body を保証する最終安全弁。Conclusion は analysis 先頭文、ES は各 section 先頭文を連結

### 検討した代替案

- **ES/Conclusion を revision loop 後に再生成**: section_planner を 2 回実行する案。パイプライン構造の大幅変更が必要で、LangGraph の revision loop (`critic → writer`) とも競合するため見送り
- **analysis body 全文を conclusion prompt に注入**: Gemma4 の thinking token 予算消費と context rot リスクから、500 chars に truncate する方針を採用

### アーキテクチャ

```
PlannerNode
  ├── REPORT_TYPE_SKELETONS (known types) ─── fixed 3 sections
  ├── DEFAULT_SKELETON (unknown types) ────── fixed 3 sections (bare)
  └── QueryExpansionOutput (LLM) ──────────── queries only, num_predict=1024

SectionPlannerNode
  ├── Analysis ── extracted_facts → LLM → provisional claim_plans
  ├── Conclusion ── analysis claim_plans → LLM / deterministic fallback
  └── ES (last) ── all non-ES claim_plans → LLM / deterministic fallback

WriterNode
  ├── Analysis ── claim → paragraph (LONGFORM)
  ├── Conclusion ── accepted analysis claims → synthesis claims → paragraph + <prior_analysis> context
  │     └── fallback: section_planner claim_plans → _deterministic_fallback_body
  └── ES (last) ── accepted non-ES claims → synthesis claims → paragraph + <prior_sections> context
        └── fallback: section_planner claim_plans → _deterministic_fallback_body
```

## Consequences

### Pros

- **Planner の構造安定性**: report_type に関わらず固定 3 section が保証される。LLM 失敗でも同一構造
- **空本文の根絶**: SectionPlannerNode (topic claim) + WriterNode (accepted-claim synthesis / deterministic body) の 2 層防御で ES/Conclusion が空にならない
- **accepted claims ベースの最終合成**: provisional claim_plans ではなく、実際に accepted された paragraph claims を優先して ES/Conclusion を構成できる
- **重複削減**: analysis body を conclusion prompt に注入し、bigram 重複を生成時に抑制。FM8 は post-hoc 検出として維持
- **prompt 品質向上**: valid JSON schema + topic-only scope で Gemma4 の structured output 精度が向上
- **テストカバレッジ**: 23 新規テストで全受け入れ基準を形式的に検証

### Cons / Tradeoffs

- **コンテキスト肥大化リスク**: analysis body (500 chars) + prior sections (各 300 chars) が paragraph prompt に追加される。Gemma4 の thinking token 予算消費が増加する可能性がある。truncate で軽減しているが、モニタリングが必要
- **deterministic fallback body の品質**: `_deterministic_fallback_body` は analysis 先頭文の切り貼りであり、LLM 品質には遠い。ただし発動条件は「SectionPlannerNode の 3 段 fallback を全て突破した場合」のみであり、頻度は極めて低い
- **topic パラメータの追加**: `_deterministic_*_claims` のシグネチャ変更は既存テスト 2 件に影響した。keyword-only argument で後方互換性は維持
