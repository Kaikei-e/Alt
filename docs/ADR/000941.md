# ADR-000941: Visual Preview の OG 画像読み込みをクライアント側 per-host キュー + status 認識リトライで耐性化し既読時のフォールバック退行を塞ぐ

## Status

Accepted

## Date

2026-06-22

## Affected Services

- alt-frontend-sv (TypeScript / Svelte 5) — `src/lib/utils/imageLoadQueue.ts`, `src/lib/utils/imageLoadQueue.test.ts`, `src/lib/utils/loadProxyImage.ts`, `src/lib/utils/loadProxyImage.test.ts`, `src/lib/components/desktop/feeds/VisualFeedCard.svelte`, `tests/e2e/desktop/feeds/visual-preview-image-resilience.spec.ts`
- alt-backend (Go) — 変更なし（image proxy の制約を鏡像するだけ）

## Context

デスクトップ Visual Preview で記事を既読にすると、当該カードだけでなく **すでに表示されていた OG 画像までフォールバックの gradient に化ける**退行が報告された。ログ精査で原因を確定した。

### 計測 (alt-backend 実ログ / ClickHouse)

| 事実 | 値 | 出典 |
|---|---|---|
| `/v1/images/proxy/<sig>/<base64>` への **429** が同一 ms 帯で連発 | 対象はほぼ `res.cloudinary.com/zenn/.../og-base-w1200-v2.png`（= 同一上流ホスト） | alt-backend request log |
| 同経路の **502** | `PATH_TRAVERSAL_BLOCKED`（Cloudinary 変換 URL の validator 誤検知）/ `DNS_RESOLUTION_ERROR`、および `duration_ms:10000` の期限切れ | alt-backend |
| image proxy の rate limit | `rate_limiter.NewHostRateLimiter(1 * time.Second)` = **上流ホスト毎秒 1 枚 / burst 1**（`app/di/image_module.go:34`） | source |
| ハンドラ期限 | **10 秒**（`app/rest/image_proxy_handlers.go`）。429 後は当該ホスト間隔を 2s に倍化（`RecordRateLimitHit`） | source |
| キャッシュ仕様 | **200 のみ** `Cache-Control: public, max-age=604800, immutable`。**429/502 は本文空・キャッシュなし** | source |

### 因果連鎖

1. Visual Preview は一画面に同一上流ホスト（Cloudinary / Zenn 等）の OG 画像を多数並べる。各カードの `<img>` が一斉に proxy を叩く。
2. image proxy は per-host 1req/s。`WaitForHost` が 10s 期限内に捌けない分を「would exceed context deadline」で **429**（空・無キャッシュ）にする。よって 1 ロードで同一ホスト約 10 枚しか 200 化せず、残りはフォールバック。
3. 旧 `VisualFeedCard` は `<img onerror>` で `imageError=true` を**恒久固定**していた。retryable な 429 を致命扱いし、一過性失敗を永続フォールバック化していた。
4. 既読操作（`fetchReplacementFeed()` + `BatchPrefetchImages` の server-side warm + 補充カードの `<img>`）が同一ホストへバーストを上乗せし、**まだ 200 化していなかった（=見た目ロード中だった）画像**を期限切れ 429 に転落させる。これが「すでに読み込んだものも落ちる」の正体。真に 200 キャッシュ済みの画像はブラウザの immutable cache で保護され落ちない。

[[000884]] は **モバイル swipe** の `FetchArticleContent` 自動 prefetch を per-host gate で抑止した。[[000936]] は read 経路の OG 解決欠落を内容ベース backfill で塞いだ。だが **desktop の OG 画像 proxy を実際に描画する経路**（`<img>` のロード）は依然 ungated かつ sticky-error のままで、本退行はその穴に該当する。

## Decision

**OG 画像を「status を読める fetch ローダ」で取得し、それを per-host 同時実行キューでゲートする。** BE 契約は変更しない。

- `imageLoadQueue.ts`: proxy URL の base64 から上流ホストを判定し、**per-host=2 / global=8** で同時 fetch を絞る共有 singleton。on-screen 画像が同一ホストに集中する実態から、グローバルではなく per-host 粒度を採る（BE 制約の鏡像）。
- `loadProxyImage.ts`: `fetch()` で取得し HTTP status を判定。**429/502/503/504/network は backoff(1.5s→3s+jitter) で最大 2 回リトライ**、**403/400/URL 無しは absent 即時**。成功 blob は object URL 化（200 は immutable HTTP cache に載るので再訪はキャッシュ供給で追加ネットワーク無し）。
- `VisualFeedCard.svelte`: sticky `imageError` を撤廃し **idle/loading=shimmer・loaded=img・absent=fallback** の 3 状態へ。retry 中は fallback を出さない。IntersectionObserver で **viewport 近傍カードのみ**ロードし、画面外カードがホスト枠を浪費しないようにする。

### 検討した代替案と却下理由

- **BE の per-host rate limit を緩める / 429 時に stale を返す**: 根本的だが image proxy 契約・キャッシュ層の変更が大きく、上流ホストへの礼儀（著作権・帯域）も崩す。今回はスコープ外（別 PR）。
- **native `<img>` のまま onerror で盲目リトライ**: `<img onerror>` は HTTP status を露出しないため、403（恒久）と 429（一過性）を区別できず、恒久失敗まで無駄にリトライする。採用した分類（403/400=absent, 429=retry）を満たせないため却下し、fetch ベースを採用。
- **グローバル同時数のみ制限**: 実装は最小だが、同一ホストに偏る実態では per-host 制約に追従できず 429 を残す。per-host を採用。
- **IntersectionObserver の遅延ロードのみ（明示キューなし）**: 初期ペイントのバーストが残り、retry 依存度が高い。viewport gating は併用するが主機構にはしない。
- **既読時の warm を debounce**: 有効だが今回は不採用。per-host キュー + viewport gating で大半が緩和され、server-side warm の競合は retry が吸収する。残存リスクとして記録（下記 Cons）。

## Consequences

### Pros

- 一過性の per-host 429 / 上流 502 がフォールバックに直結しなくなり、既読操作でも表示済み画像が落ちない。
- クライアントが BE の per-host 制約を鏡像するため、429 の発生自体が構造的に減る（per-host=2 / burst=1 でも 2 枚目は ~1s 待ちで 200、期限切れに至らない）。
- 「transient fallback（回復可能）」と「absent OG image（恒久）」を UI 状態として分離。語彙は `contexts/visual-preview.md` に固定。
- viewport gating により画面外カードがホストのレート枠を消費しない。

### Cons / Tradeoffs

- 同一ホスト多数の初回ロードは BE が 1req/s のため依然 ~N 秒かけて順次表示される（immutable cache により再訪は即時）。クライアントは速くできない、上限は BE 由来。
- object URL のメモリ管理（破棄時 revoke）が増える。`onDestroy` と URL 変更時に revoke 済み。
- `BatchPrefetchImages` の server-side warm はクライアントキュー外でホスト token を消費する競合が残る（warm debounce を入れていない）。retry が吸収する前提。
- BE 側の 502 `PATH_TRAVERSAL_BLOCKED`（Cloudinary 変換 URL の validator 誤検知）は未修正。別 PR の follow-up。
- per-host gate が `articlePrefetcher`（[[000884]]）と二系統並存する。将来的に共有ユーティリティへ寄せる余地。
