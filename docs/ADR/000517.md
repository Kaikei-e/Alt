# ADR-000517: 要約統計の母集団を active article に揃える

## Status

Accepted

## Date

2026-03-21

## Affected Services

- alt-backend (Go)
- alt-frontend-sv (SvelteKit)

## Context

stats 画面の `Summarized` は frontend 側で `totalArticles - unsummarizedArticles` として導出されている。
この式を正しく成立させるには、`total` と `unsummarized` が同じ母集団を見ていなければならない。

しかし従来の unsummarized 集計は `deleted_at IS NULL` を条件に含めておらず、summary 側 join でも user 境界が明示されていなかった。
そのままでは、論理削除済み記事や他ユーザーの summary が count に混ざる余地があり、frontend の差分計算と SQL の意味がずれる危険があった。

件数表示を単なる近似値ではなく、一貫した定義に基づく統計として保つには、total / unsummarized / summarized が同じ active article universe を前提に揃う必要があった。

## Decision

**要約統計の母集団は「認証ユーザーに属し、かつ論理削除されていない article」に統一する。**

具体的には以下を採用する。

1. total articles count は active article だけを数える
2. unsummarized articles count は active article を左辺にし、同一ユーザーの summary が存在しないものだけを数える
3. summarized articles count は同一ユーザーの saved summary を数える
4. frontend は引き続き `total - unsummarized` を summarized の表示値として利用できる前提を維持する

この決定により、stats 画面の 3 つの値は同じ母集団に基づく整合した統計になる。

### 採用しなかった代替案

| 案 | 評価 |
|---|---|
| summarized 件数だけを直接表示し、他の count との整合は気にしない | frontend 側の差分表示と API 意味論が分離し、検証しづらい |
| unsummarized に deleted article を含めても許容する | `total - unsummarized` が統計上の意味を失う |
| article と summary の user 整合を join で見ない | 他ユーザーの summary が誤って補完に使われる余地が残る |

## Consequences

### Pros

- `total`, `unsummarized`, `summarized` の関係が理解しやすくなる
- frontend の表示式が API 契約と矛盾しにくくなる
- 論理削除や multi-tenant 条件が入っても count の意味が崩れにくい

### Cons / Tradeoffs

- count SQL は従来より条件が増え、クエリの意図を明示的に保つ必要がある
- active article 以外を含む監査件数とは値が一致しなくなる
- summary table 単独の raw row 数と、画面上の summarized 表示が別概念であることを理解する必要がある
