# ADR-000695: Augur リトリーバルの言語横断検索を 3 サービス横断で修正する

## Status

Accepted

## Date

2026-04-11

## Affected Services

- rag-orchestrator (Go)
- news-creator (Python)
- search-indexer (Go)

## Context

日本語クエリ「ヴァンス副大統領の直近の動きは？」に対して、Meilisearch に 372 件の Vance 関連英語記事が存在するにもかかわらず、Augur が回答を生成できなかった。[[000693]]（embedding 障害時の degraded mode 導入）で embedder 復旧後も問題が継続し、ログ分析から 3 つの独立した言語横断の障害箇所が判明した。

### 障害箇所 1: expand-query プロンプト構造の不備

news-creator の `EXPAND_QUERY_PROMPT_TEMPLATE` が `Japanese({count}):` ラベルのみで終わり、`English({count}):` ラベルがなかった。Gemma4 は日本語セクションを出力後に停止し、英語翻訳を生成しなかった。リクエストは `japanese_count=1, english_count=3` だが、実際の出力は日本語 1 件のみ。

### 障害箇所 2: BM25 が元クエリのみ使用

rag-orchestrator の `EmbedAndSearch` Stage 2 で、BM25 検索（Meilisearch 経由）は `sc.Query`（元の日本語クエリ）のみで検索していた。展開クエリ・翻訳クエリはベクトル検索の embedding にしか使われず、BM25 の言語横断検索能力が活かされていなかった。

### 障害箇所 3: Meilisearch の日本語限定ロケール

search-indexer の `MeilisearchDriver.Search()` で、CJK 文字を含むクエリに対して `Locales: ["jpn"]` を設定し、英語トークナイザを無効化していた。インデックスは `jpn` + `eng` 両方のロケールで構成されているにもかかわらず、検索時に英語側が使われなかった。

## Decision

**3 サービスそれぞれの障害箇所を独立して修正し、日本語クエリから英語記事への到達パスを確保する。**

いずれも単独で効果があるが、3 つ全てを修正することでリトリーバル品質を最大化する。

## Implementation

### 修正 1: expand-query プロンプトに English ラベル追加（news-creator）

**ファイル**: `news_creator/usecase/expand_query_usecase.py`

`EXPAND_QUERY_PROMPT_TEMPLATE` と `EXPAND_QUERY_WITH_HISTORY_TEMPLATE` の末尾に `English({english_count}):` セクションと番号付きプレフィルを追加した。これにより Gemma4 が英語セクションの生成を継続する。

```python
# Before:
Japanese({japanese_count}):

# After:
Japanese({japanese_count}):
1.
English({english_count}):
1.
```

テスト: `TestPromptHasBilingualOutputLabels`（4 テスト）— テンプレートに English ラベルが存在し、Japanese の後に配置されていることを検証。

### 修正 2: BM25 に展開クエリを利用（rag-orchestrator）

**ファイル**: `retrieval/embed_and_search.go`

goroutine E の BM25 検索を、元クエリに加えて `sc.AdditionalQueries`（展開・翻訳クエリ）でも検索するよう修正した。結果は ArticleID で重複排除して統合する。

```go
queries := bm25Queries(sc.Query, sc.AdditionalQueries)
for _, q := range queries {
    results, _ := bm25Searcher.SearchBM25(gctx, q, bm25Limit)
    // ArticleID で重複排除して統合
}
```

`bm25Queries()` ヘルパーが元クエリ + 展開クエリの重複排除リストを構築する。

テスト: `TestEmbedAndSearch_BM25UsesExpandedQueries`, `TestEmbedAndSearch_BM25ExpandedDeduplicatesResults`

### 修正 3: Meilisearch ロケール制限の緩和（search-indexer）

**ファイル**: `driver/meilisearch_driver.go`

`Search()` と `SearchWithFilters()` の両メソッドから `Locales: ["jpn"]` 制限を削除した。Meilisearch はインデックスに設定された全ロケール（`jpn` + `eng`）でマッチングを行うようになる。

## Consequences

### Pros

- **日本語→英語の言語横断検索が機能**: 「ヴァンス副大統領」で英語 "JD Vance" 記事が検索可能に
- **3 層の独立した改善**: プロンプト・BM25 マルチクエリ・ロケール緩和がそれぞれ独立して効果を発揮
- **BM25 検索の活用度向上**: 従来は元クエリの言語でしかヒットしなかった BM25 が、翻訳クエリでもヒットする
- **既存テスト全パス**: 3 サービスとも回帰なし

### Cons / Tradeoffs

- **BM25 レイテンシ微増**: 展開クエリ数分だけ Meilisearch 呼び出しが増加。ただし Meilisearch は高速（1-2ms/query）なので実用上の影響は軽微
- **ロケール全開放のノイズ**: 日本語クエリが英語トークンにもマッチするため、false positive が増える可能性。RRF fusion と RelevanceGate で制御される想定
- **expand-query プロンプトの LLM 依存**: `1.` プレフィルは Gemma4 の挙動に依存。モデル変更時に再検証が必要
