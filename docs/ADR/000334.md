# Quality Check による BE スロット競合の解消: 要約キュー優先制御

## ADR's STATUS

Accepted

## CONTEXT

news-creator の要約パイプラインでセマフォキュー待ち時間が異常に膨れ上がっていた。ログ調査により、Quality Check（品質スコアリング）が要約と同一の BE スロットを奪い合っていることが判明した。

### 調査結果

| 指標 | 値 |
|------|-----|
| 30分間の要約リクエスト / 完了 | 56件 / 58件 |
| 30分間の Quality Check リクエスト | 30件 |
| セマフォキュー待ち (平均 / 最大) | **45秒 / 162秒** |
| GPU | RTX 4060 8GB, VRAM 67% |

### 問題の構造

pre-processor の `processQualityCheckBatch` は5分ごとに固定実行され、news-creator の `POST /api/generate` を呼び出す。これは要約 (`POST /api/v1/summarize`) と同一の `HybridPrioritySemaphore` の BE スロット（1つ）を共有しており、Quality Check が BE スロットの約50%の時間を占有していた。

ADR-265 で BE スロット独占問題を対策済みだったが、Quality Check の実行タイミングが要約キューの状態を考慮しないため、同じ構造の問題が再燃した形。

```
要約ジョブ (BE) ─┐
                  ├─→ HybridPrioritySemaphore (BE slot: 1) ─→ Ollama GPU
Quality Check (BE)─┘
```

## DECISION MAKING

**Quality Check を要約キューが空の時だけ実行する**方針を採用。要約は本サービスの主機能であり、Quality Check はベストエフォートのバッチ処理のため、要約を常に優先する。

### 変更 1: `HasPendingJobs` メソッド追加

**ファイル**: `pre-processor/app/service/summarize_queue_worker.go`

既存の `GetPendingJobs` (repository層) を `limit=1` で呼び出し、ペンディングジョブの有無だけを軽量に確認する。

```go
func (w *SummarizeQueueWorker) HasPendingJobs(ctx context.Context) (bool, error) {
	jobs, err := w.jobRepo.GetPendingJobs(ctx, 1)
	if err != nil {
		return false, fmt.Errorf("failed to check pending jobs: %w", err)
	}
	return len(jobs) > 0, nil
}
```

### 変更 2: `processQualityCheckBatch` にスキップロジック追加

**ファイル**: `pre-processor/app/handler/job_handler.go`

`processQualityCheckBatch` の先頭で要約キューの状態を確認し、ペンディングジョブがあれば Quality Check をスキップ。`HasPendingJobs` のエラー時は安全側に倒して Quality Check を続行する（graceful degradation）。

```go
func (h *jobHandler) processQualityCheckBatch(ctx context.Context) error {
	if h.queueWorker != nil {
		hasPending, err := h.queueWorker.HasPendingJobs(ctx)
		if err != nil {
			h.logger.WarnContext(ctx, "failed to check pending jobs, proceeding with quality check", "error", err)
		} else if hasPending {
			h.logger.InfoContext(ctx, "skipping quality check: summarization queue has pending jobs")
			return nil
		}
	}
	// ... 既存のQuality Check処理
}
```

### TDD テスト

**`pre-processor/app/service/summarize_queue_worker_test.go`** — 3件追加:

| テスト | 検証内容 |
|--------|----------|
| `returns_false_when_queue_is_empty` | 空キューで `false` を返す |
| `returns_true_when_queue_has_jobs` | ジョブありで `true` を返す |
| `propagates_repository_error` | DB エラーの伝播 |

**`pre-processor/app/handler/job_handler_test.go`** — 4件追加:

| テスト | 検証内容 |
|--------|----------|
| `skips_quality_check_when_summarization_queue_has_pending_jobs` | ペンディングあり → QC スキップ、`CheckQuality` 未呼出 |
| `runs_quality_check_when_summarization_queue_is_empty` | ペンディングなし → QC 実行 |
| `proceeds_with_quality_check_when_HasPendingJobs_returns_error` | エラー時 → QC 続行（graceful degradation） |
| `runs_quality_check_when_queueWorker_is_nil` | queueWorker nil → QC 実行 |

## RESULTS, EFFECTS

### PROS

- **BE スロット競合の解消**: 要約キューにジョブがある限り Quality Check は自動的に待避し、要約が BE スロットを独占できる
- **キュー待ち時間の大幅短縮**: 理論値で平均45秒 → 10秒以下（Quality Check による50%占有がなくなるため）
- **Quality Check のスループットは維持**: 要約キューが空のアイドル時に集中実行されるため、1日単位での QC 処理量は大きく変わらない
- **既存アーキテクチャへの影響が最小**: news-creator 側の変更なし。pre-processor の handler/service 層のみの変更で完結
- **Graceful degradation**: `HasPendingJobs` が DB エラーを返した場合は QC を続行するため、DB 障害時にも QC が完全停止しない

### CONS, TRADEOFF

- **Quality Check の実行タイミングが不定になる**: 要約キューが常にジョブを持つ高負荷時には QC が長時間実行されない可能性がある。ただし QC は品質スコアリングであり即時性は不要
- **DB への追加クエリ**: 5分ごとに `GetPendingJobs(ctx, 1)` が1回追加実行される。`LIMIT 1` のため負荷は無視できる
- **将来的な課題**: 要約キューが恒常的に埋まるスケールになった場合、QC に最低実行保証（例: 30分に1回は必ず実行）を入れる必要がある
