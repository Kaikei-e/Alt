# ADR-000551: 要約不可記事に placeholder summary を保存し無限再エンキューを防止する

## Status

Accepted

## Date

2026-03-23

## Affected Services

- pre-processor (Go)
- alt-backend (Go) — `article_summaries` テーブルへの書き込み

## Context

Desktop UI の Stats に表示される "Unsummarized" カウントが 63 件で固定され、減少しない状態が継続していた。

DB 調査の結果、63 件は以下の 2 カテゴリに分類された:

| 分類 | 件数 | 原因 |
|------|------|------|
| content too short | 61 | HTML 抽出後のテキストが 100 文字未満（RSS description のみの記事） |
| content too long | 2 | content が 100KB 超（189KB, 180KB の巨大記事） |

これらの記事は以下の無限ループ構造に陥っていた:

1. `EnqueueUnsummarizedBatch` → `ListUnsummarizedArticles`（`article_summaries` にエントリなし → 対象になる）
2. `ShouldQueueSummarizeJob` ガード → `Exists()` = false, `HasRecentSuccessfulJob()` = false → エンキュー許可
3. `processJob` → 要約試行 → `ErrContentTooShort` or `ErrContentTooLong`
4. ジョブを `completed` / `dead_letter` にするが、`article_summaries` には何も書かない
5. 次回バッチで再び 1 に戻る → 無限ループ

修正時点で計 1,349 回の無駄なジョブが生成されていた（completed 1,249 + dead_letter 100）。

`quality-checker/quality_judger.go` には既に `knownPlaceholders` として placeholder 文字列が定義されており、placeholder summary を保存する設計意図は存在したが、`processJob` のスキップ処理に `summaryRepo.Create` 呼び出しが欠落していた。

## Decision

**content too short / too long でスキップされた記事に placeholder summary を `article_summaries` に保存する**ことで、無限再エンキューループを断ち切る。

### 実装内容

`pre-processor/app/service/summarize_queue_worker.go` に以下を追加:

1. placeholder 定数の定義（`quality_judger.go` の `knownPlaceholders` と一致させる）:
   - `placeholderTooShort = "本文が短すぎるため要約できませんでした。"`
   - `placeholderTooLong = "本文が長すぎるため要約できませんでした。"`

2. `savePlaceholderSummary` メソッド: `summaryRepo.Create` を呼んで placeholder を保存。保存失敗時はログに記録するが、ジョブの `completed` 遷移はブロックしない（best-effort）

3. `processJob` 内の `ErrContentTooShort` ハンドラに `savePlaceholderSummary` 呼び出しを追加

4. `ErrContentTooLong` 用の新規ハンドラを追加。従来は汎用 retry → dead_letter フローに入っていたが、content サイズは変わらないためリトライは無意味。即座に placeholder 保存 + `completed` に遷移する

### 検討した代替案

**A. Stats クエリ側で除外する**: `FetchUnsummarizedArticlesCount` のクエリに「要約不可能な記事」を除外する条件を追加する。`article_summaries` にデータを書かない方式。

→ **不採用**: ループ自体が止まらず、ジョブキューの無駄な消費が続く。根本原因が解決されない。

**B. ガードに dead_letter / completed-with-error チェックを追加する**: `ShouldQueueSummarizeJob` で最近のスキップ完了ジョブも確認する。

→ **不採用**: pre-processor-db と alt-db の cross-DB 参照が必要になり複雑化する。placeholder 保存の方がシンプルで、既存の設計意図（`knownPlaceholders`）とも整合する。

## Consequences

### Pros

- **無限ループ解消**: `article_summaries` にエントリが作成されるため、`Exists()` が true を返しエンキューが防止される
- **Stats の正確性**: Unsummarized カウントから要約不可能な記事が除外され、実態を正確に反映する
- **リソース節約**: 無駄なジョブ生成（1,349 件/63 記事）が停止し、CPU・ネットワーク帯域を節約
- **既存設計との整合**: `quality_judger.go` の `knownPlaceholders` と一致する placeholder を使用するため、品質チェッカーが placeholder を誤って削除することがない
- **too long の即時完了**: `ErrContentTooLong` が即座に `completed` になるため、無駄な 3 回リトライが不要になる

### Cons / Tradeoffs

- **placeholder が Stats に「要約済み」として計上される**: 実際には要約されていないが、Unsummarized カウントからは除外される。将来的に「要約不可」を別カテゴリとして表示する UI 拡張が望ましい
- **placeholder 保存は best-effort**: `summaryRepo.Create` が失敗してもジョブは `completed` になる。次回バッチで再度エンキューされるが、placeholder 保存が再試行されるため自己修復する
- **既存の 63 件は次回バッチで処理**: デプロイ直後に即座に解消されるわけではなく、5 分間隔のバッチ sweep で段階的に処理される
