# ADR-000437: needToKnowCount を backend 正確値に切り替え page 近似を廃止する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go)

## Context

Knowledge Home の `TodayDigest.need_to_know_count` は、handler が `GetKnowledgeHome` レスポンスの `items` を走査し、`pulse_need_to_know` why code を持つアイテムを数える page 近似方式で算出されていた。

```go
// 旧実装 (handler.go)
needToKnowCount := int32(0)
for _, item := range result.Items {
    for _, why := range item.WhyReasons {
        if why.Code == domain.WhyPulseNeedToKnow {
            needToKnowCount++
            break
        }
    }
}
```

この方式には以下の問題があった:

1. **ページング依存**: 1 ページ目の 20 件だけで数えるため、2 ページ目以降の need-to-know アイテムは無視される
2. **責務の混在**: handler が集計ロジックを持ち、usecase の digest と handler の計算値が混在する
3. **contract の曖昧さ**: proto 上の `need_to_know_count` が「全体の件数」を意味するのか「現在ページの件数」なのか不明確

## Decision

**handler の page 近似 count を廃止し、usecase の digest enrichment で backend query による正確な count を返す。**

具体的には:

1. **Port 追加**: `CountNeedToKnowItemsPort` を `today_digest_port` に定義
2. **Repository query 追加**: `knowledge_home_items` テーブルに対し `why_json @> '[{"code":"pulse_need_to_know"}]'` の JSONB 含有クエリで日付範囲の件数を取得
3. **Usecase enrichment**: `GetKnowledgeHomeUsecase.enrichDigest()` で digest 取得後に count を上書き
4. **Handler 整理**: items 走査ロジックを削除し、usecase から返された `digest.NeedToKnowCount` をそのまま proto に変換

### 検討した代替案

| 案 | 評価 |
|---|---|
| `today_digest_view` に `need_to_know_count` 列を追加して projector で加算保持 | Phase 3 の方針「reproject-safe でない加算列を増やさない」に反する。reproject 時に count がずれるリスクがある |
| handler で全ページを fetch して count | レイテンシ・リソースコストが過大 |
| **request 時に backend query で導出** (採用) | projector 列を増やさず正確値が返る。enrichment failure 時は 0 にフォールバックで安全 |

### Enrichment failure の扱い

count query が失敗した場合、usecase は `NeedToKnowCount = 0` にフォールバックし、エラーをログに記録するが全体を hard fail させない。digest が取得できていれば items は正常に返る。

## Consequences

### Pros

- `need_to_know_count` が全 need-to-know アイテムの正確な件数を返す
- handler から集計ロジックが消え、「items + digest の合成」のみに責務が限定される
- `today_digest_view` に新列を追加しないため、migration rollback の複雑化を避けられる
- canonical contract の `need_to_know_count` の意味が「backend-authoritative count」として明確になる

### Cons / Tradeoffs

- `knowledge_home_items` に対する JSONB 含有クエリが毎リクエスト発生する。ただし `why_json` は GIN インデックスの対象ではないため、日付範囲で絞った上での sequential scan になる。大量データ環境ではインデックス追加を検討する余地がある
- enrichment port が nil の場合は enrichment をスキップするため、DI 忘れで旧動作に戻るリスクがある（ただしテストで検証済み）
