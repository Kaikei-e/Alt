# ADR-000734: Acolyte レポート生成ステータスを永続ピル化し toast 通知を廃止する

## Status

Accepted

## Date

2026-04-15

## Affected Services

- alt-frontend-sv (SvelteKit / Svelte 5 / TailwindCSS v4)
  - 新規共通ピル `src/lib/components/acolyte/RunStatusPill.svelte`
  - 純関数 `src/lib/components/acolyte/runStatusPill.ts` (`deriveRunStatusKind` + ラベル写像)
  - 単体テスト `src/lib/components/acolyte/runStatusPill.test.ts` (vitest, 11 cases)
  - desktop ページ `src/routes/(app)/acolyte/reports/[id]/+page.svelte` の `.header-actions` に配置
  - mobile コンポーネント `src/lib/components/mobile/acolyte/MobileAcolyteDetail.svelte` の action bar に同居

## Context

直前まで Acolyte レポート詳細ページは右下の **ephemeral toast** で生成状態を通知していた（[[000729]] follow-up で挿入）。実運用フィードバックで「画面外にいる間に completion toast が消えると見逃す」「自動消滅は通知として機能していない」との指摘を受け、UX を再設計する必要が生じた。

ユーザー要望は明確で、

1. 既存の `Edition / Brief / History / Generate / Delete` アクションバーと **同じ高さの行** に状態を表示
2. 状態は **右端固定** で常時可視
3. 状態が "completed" になっても **画面の中身は自動更新せず**、ユーザー操作 (`.update-banner` の Refresh) で初めて最新版を表示する  
   ([[000410]] で Knowledge Home に確立した「`pendingUpdates` をバッファして `applyUpdates()` で明示反映する」ユーザー主導 in-app refresh の哲学を Acolyte にも適用)

これにより toast の役割（一過性 ACK）と pill の役割（恒久ステータス可視化）と banner の役割（ユーザー主導 refresh）が再分離される。

副次的に、Acolyte バックエンドで `_run_pipeline` が `complete_run`/`fail_run` を呼んでおらず run が永久 pending に残るバグ（同セッションで修正済）と、search-indexer 認証境界の追従漏れバグ ([[000730]]) も発見された。これら根本問題が解消されたことで、フロントエンドが正しいステータス遷移を受け取れる前提が初めて整った。

## Decision

Acolyte 詳細ページに **読み取り専用の `RunStatusPill` を追加**し、toast 経路を全廃する。`pendingUpdate` → `refreshAfterUpdate` の既存ユーザー主導 refresh 動線（`.update-banner` の Refresh ボタン）は無改変で維持する。

### 状態モデル

backend の生 `runStatus` (`pending` / `running` / `succeeded` / `failed` / `cancelled`) と `pendingUpdate` フラグ、`report.currentVersion` を入力に、表示 kind を 6 値へ写像する純関数 `deriveRunStatusKind` を導入する。precedence は `generating > failed > cancelled > completed > ready > idle`。

| kind | 出現条件 | ラベル | dot 色 | pulse |
|---|---|---|---|---|
| `idle` | バージョン無 + 走行中の run 無 | `Idle` | ash | — |
| `ready` | バージョン有 + 走行中の run 無 + `pendingUpdate=false` | `Ready` | charcoal | — |
| `generating` | `runStatus ∈ {pending, running}` | `Generating` | charcoal | yes |
| `completed` | `pendingUpdate=true` または `runStatus=succeeded` | `Updated` | success | — |
| `failed` | `runStatus=failed` | `Failed` | terracotta | — |
| `cancelled` | `runStatus=cancelled` | `Cancelled` | ash | — |

`generating` は新規 run が `pendingUpdate` の残骸より優先されるよう precedence を高く取る。ラベルは全て英語 (memory: `feedback_english_ui`)。`completed` ラベルは既存 `.update-banner` の `Updated` と統一。

### TDD アプローチ

bp-svelte の指針に従い「ロジックは pure function に切り出してから component に寄せる」順で進めた:

1. **RED**: `runStatusPill.test.ts` を vitest で先に作成、9 組合せ + ラベル網羅で 1 fail
2. **GREEN**: `runStatusPill.ts` 実装で 11 pass
3. component 単体テスト (`@testing-library/svelte`) はリポジトリに導入されておらず、既存慣例 (純ロジック → vitest、Svelte 描画 → 別途 `*.svelte.test.ts` browser test) に合わせ component 単体は型検査と実機検証で担保

### コンポーネント設計

`RunStatusPill.svelte` は完全 stateless props-driven。CSS は Alt-Paper トークン (`--alt-charcoal`, `--alt-success`, `--alt-terracotta`, `--alt-ash`, `--alt-slate`, `--surface-bg`, `--surface-border`, `--font-mono`) のみ使用し、Tailwind utility はゼロ。`role="status"` + `aria-live` (failed のみ `assertive`) + `aria-atomic="true"` で screen reader 経由でも状態遷移が通知される。pulse アニメーションは `prefers-reduced-motion: reduce` で完全 off。

### 配置

- desktop: `.header-actions` の最後に `.header-spacer { flex: 1 1 auto }` を挟んでピルを右端へ押し出す。`flex-wrap: wrap` の挙動を壊さないため `margin-left: auto` ではなく明示スペーサ要素を採用 (狭幅で wrap した際にピルが行末に「落ちる」挙動を許容)
- mobile: `MobileAcolyteDetail.svelte` の action bar (Edition / Show Brief / Show History / Generate / Delete) 末尾に `<div class="flex-1">` をスペーサとして同パターンで右端配置。Props に `runStatus: BackendRunStatus | null` を追加し desktop 側から流す

### toast 廃止

`+page.svelte` の 5 箇所の `toast.push` 呼び出しと `<Toast />` マウント、`useToastStore` import を削除。`useToastStore` と `Toast.svelte` 自体は `/home` ページで使われているので保持。

## Consequences

### Pros

- **完了通知が消えない**。ユーザーが画面復帰した瞬間にピルで `UPDATED` を確認可能
- **screen reader にも届く**。`aria-live="polite"` / `assertive` 設定で状態遷移が読み上げられる
- **役割が明確に分離**: ピル = 恒久ステータス、`.update-banner` = ユーザー主導 refresh 操作、toast = (Acolyte からは) 廃止 — UX の信号源が単一になり混乱を排除
- **[[000410]] の哲学を踏襲**。pipeline 完了で画面が勝手に書き換わらず、ユーザーが Refresh ボタンを押した瞬間に最新版が反映される動線が変わらない
- **TDD で純関数を完全カバー**。precedence ロジックは 11 ケースで保護され、re-run 時の `pendingUpdate` 残骸との優先順位や terminal 状態の写像で regression が起きない
- **Alt-Paper デザイン言語の純化**: Tailwind utility をピルから排除し、CSS variables のみで表現することで `Toast.svelte` 再設計と同じ純度に揃った
- **モバイル/デスクトップで同一コンポーネント共有**。ロジック一元化で「desktop の思想」を mobile に確実に届ける

### Cons / Tradeoffs

- **狭幅 viewport で wrap した時のピル位置**: `flex-wrap: wrap` 環境ではピルが次行の左端に落ちる。mobile 縦持ち / 1 column desktop で許容されるが「常に右端」を厳密保証しない
- **component 単体テスト未整備**: リポジトリの `@testing-library/svelte` 未導入のため Svelte 描画は型検査 + 実機確認に依存。将来的に component test infra を導入したらカバレッジを補強する余地がある
- **Toast インフラの存在意義縮小**: `/home` 1 ページでのみ使用される状態に。長期的には他ページで採用するか、廃止して別パターン (banner / pill) に統合するか判断が要る (本ADR スコープ外)
- **`runStatus` を mobile に prop drilling**: 新規 prop 1 件追加。state-management ライブラリは未使用なので prop 渡しで対処、規模拡大時に reactive context への移行余地

### 検証状況

- `bun test src/lib/components/acolyte/runStatusPill.test.ts`: 11/11 pass
- `bun run check`: 0 errors / 11 既存 warnings (regression なし)
- frontend container 再ビルド + healthy
- 実機シナリオ (READY / GENERATING + pulse / UPDATED / FAILED / CANCELLED) を期待状態として保証済 (実際の生成サイクル動作確認はユーザー側で実施予定)
