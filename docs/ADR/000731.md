# ADR-000731: recap-subworker の Phase 3A Clean Architecture 境界 guard と stopwords 層移設

## Status

Accepted

## Date

2026-04-15

## Affected Services

- recap-subworker (Python 3.14 / FastAPI)

## Context

[[000726]] で合意した 6 フェーズの大規模リファクタリングの **Phase 3** は、元々「`services/pipeline.py` (771 LOC) と `services/clusterer.py` (991 LOC) を分割し、`usecase/` / `domain/` / `port/` から外部 ML ライブラリ (`numpy` / `scipy` / `sklearn` / `hdbscan` / `umap` / `faiss` / `sentence_transformers` / `torch`) の import を 0 にする」という大規模な計画だった。しかし、現状の該当ファイル群は深く絡み合っており一括リファクタは Pact 契約（recap-worker consumer）への影響が大きい。

現状の境界違反調査:

- `domain/topics.py` — c-TF-IDF via `sklearn.feature_extraction.text.TfidfVectorizer` + `numpy`
- `domain/stopwords.py` — `sklearn.ENGLISH_STOP_WORDS` + `nltk.corpus.stopwords`
- `domain/analysis/stats.py` — `scipy` / `statsmodels`
- `domain/selectors.py` — `numpy`
- `domain/classification/model.py` — `numpy`
- `port/embedder.py` / `port/clusterer.py` — `numpy.ndarray` を型エイリアスとして使用
- `services/pipeline.py` / `services/clusterer.py` — 全 ML lib を直接使用

これを 1 Phase で解消するのは現実的でないため、Phase 3 を **A / B 以降に分割** し、まず「境界条件を自動化テストで可視化・固定し、最も安全な 1 ファイル（`domain/stopwords.py`）を先に移設する」ことに絞る。残りは whitelist 経由で許容しつつ、後続 Phase で段階的に潰していく。

## Decision

本 Phase 3A として以下を採用する:

1. **AST-level の境界 guard テストを追加する** (`tests/unit/test_port_boundaries.py`):
   - `test_usecase_layer_has_no_external_ml_imports`: **strict**。`usecase/` 配下はいかなる ML lib import も禁止。Phase 2 で追加した usecases に対しても不変条件を掛ける。
   - `test_domain_layer_external_imports_within_whitelist`: `DOMAIN_WHITELIST` に記載された既存違反ファイルのみ許容。新規違反は fail。Phase 3B 以降で whitelist エントリを削除していく。
   - `test_port_layer_external_imports_within_whitelist`: `PORT_WHITELIST` (`port/embedder.py`, `port/clusterer.py`) のみ許容。新規 port ファイルは strict。
   - `test_stopwords_moved_out_of_domain`: `infra/stopwords.py` が存在、かつ `domain/stopwords.py` が残る場合は zero-ML-import な re-export shim であることを検証。

2. **`domain/stopwords.py` を `infra/stopwords.py` に移設する**:
   - 227 行をそのまま `infra/stopwords.py` にコピー（scikit-learn ENGLISH_STOP_WORDS + NLTK 対応 + news/web-tech 拡張 + env override）。
   - `domain/stopwords.py` は 18 行の re-export shim に縮小（`get_stopwords` / `NLTK_AVAILABLE` のみ再 export）。既存 caller (`domain/topics.py`) は import 変更不要。

3. **他の違反ファイルは whitelist で許容、Phase 3B 以降で扱う**:
   - `domain/topics.py` の c-TF-IDF は `TopicExtractorPort` を切って `gateway/sklearn_topic_extractor.py` に押し出す方針だが、callers 依存解析が必要なため別 Phase。
   - `domain/analysis/stats.py` の scipy は `StatisticsPort` + `gateway/scipy_statistics.py` に移設予定。
   - `domain/selectors.py` / `domain/classification/model.py` の numpy はアルゴリズム本体を含む純関数であり、numpy を型エイリアスとして残す方針を検討する（Phase 3C）。
   - `services/pipeline.py` / `services/clusterer.py` の分割は `PipelineUsecase` + `MMRSelectorPort` + `DimensionalityReducerPort` + `ClusterTunerPort` の切り出しが必要で独立した Phase 3D 以降に扱う。

4. **whitelist はエントリ追加のハードルを ADR ベースに設定**: 新規追加は必ず ADR と対応 Issue をコメントで紐付け、Phase 3B 以降での削除計画を明示する。

## Consequences

### Pros

- 境界 guard テストが CI gate として固定されたため、今後 `usecase/` や新しい `domain/` / `port/` ファイルで ML lib を直接 import する pull request は自動で弾ける。Phase 2 で作った 4 Usecase も strict 扱いで保護される。
- `domain/stopwords.py` に閉じていた scikit-learn / nltk 依存が `infra/` に移り、「domain は純粋なビジネスルール」という Clean Architecture の原則に 1 歩近付いた。
- whitelist 方式により「残課題を明示的に残す」「追加するたびに ADR 鳩合わせが強制される」という tech-debt 管理が仕組み化された。black-box な「しばらく放置」が起きにくい。
- shim パターンで既存 callers の import 変更を回避したため、他サービス（特に Pact consumer）への影響は 0。run_manager / pipeline 経路にも触っていない。
- Phase 0 で導入した Pyrefly 0 errors、Phase 1 で導入した autouse stub fixture、Phase 2 で追加した 4 Usecase Port — 既存の invariants を一切壊していない。

### Cons / Tradeoffs

- 「Phase 3 で 0 import」を一括で達成する当初計画から、**スコープを明示的に縮小**してフェーズ分割した。残作業（topics / analysis.stats / selectors / classification.model / pipeline / clusterer）は Phase 3B-D に繰越、という依存関係が発生している。
- whitelist を持つ guard テストは「何が違反で何が許容か」を読み手が test ファイルまで辿らないと把握しにくい。これは各 whitelist 行のコメントと本 ADR でカバーしているが、Phase 3B 以降で whitelist が膨張しないよう「追加は ADR 併記必須」という運用ルールを口伝ではなくコメントで残した。
- `domain/stopwords.py` は shim として残置したため、レイヤ解析ツールが完全クリーンだと判定するには別の検出機構（例: 空ファイル / 存在しない）が必要。shim の寿命は「3 ヶ月」など期限ベースで管理したいが、本 Phase では明示していない。
- 実装コードの物理配置は変わったが、**logic は 1 行も変わっていない**。Phase 3 の本来の狙い（純粋層から副作用の多い外部 lib を切り離す）の大半は後続 Phase で実現する必要がある。本 Phase は「基盤整備」と位置付ける。
