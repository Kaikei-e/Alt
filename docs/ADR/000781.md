# ADR-000781: alt-backend staging DB 初期化で Atlas マイグレーションの GRANT 先ロールを NOLOGIN で先作りする

## Status

Accepted

## Date

2026-04-19

## Affected Services

- alt-backend (Go) — staging での DB 起動経路のみ変化。プロダクション挙動は無関係
- `compose/compose.staging.yaml` — `alt-backend-db` service に init script の bind mount 追加
- `e2e/fixtures/alt-backend/db-init/01-create-roles.sql` — 新規。staging 専用の role 作成 SQL

## Context

[[000778]] の E2E staging stack は alt-backend 用に新しい Postgres 17 コンテナと Atlas migrator を持つ。しかし最初の実行で migrator が migration version `20240101002800` で停止した:

```
-- migrating version 20240101002800
    -> GRANT SELECT, INSERT, DELETE
           ON feed_tags, article_tags
           TO tag_generator;
pq: role "tag_generator" does not exist (42704)
Error: sql/migrate: write revision: pq: current transaction is aborted
```

`migrations-atlas/migrations/` を grep すると、マイグレーションは以下のロール宛に GRANT 文を持つ:

- `alt_appuser`
- `alt_db_user` (POSTGRES_USER)
- `db_owner`
- `pre_processor_user`
- `pre_processor_sidecar_user`
- `search_indexer_user`
- `tag_generator`

本番 compose では `db/init/01-create-users.sh` が Postgres 初期化時 (`/docker-entrypoint-initdb.d/`) にこれらのロールを LOGIN + password 付きで作成している。password は Docker secrets (`PRE_PROCESSOR_DB_PASSWORD_FILE` など) から読む仕組みで、secret ファイル側に staging 用のフィクスチャを揃える工数が発生する。

staging では以下が本番と異なる:

- alt-staging network には alt-backend 以外に Postgres に接続するサービスがない（deps-stub は Postgres を使わない）
- したがって LOGIN 可能である必要はない。GRANT 文を通すためにロールが存在すれば十分
- Docker secret を 6 個追加するより、NOLOGIN 役割を 1 つの SQL で作る方がフィクスチャ管理が小さい

## Decision

**`e2e/fixtures/alt-backend/db-init/01-create-roles.sql` を追加し、staging 用 Postgres の `/docker-entrypoint-initdb.d/` にマウントする。** 採用構成:

1. **DO ブロックで 7 ロールを NOLOGIN 作成する**。`pg_catalog.pg_roles` にロールが存在しない場合のみ `CREATE ROLE <name>` を実行する冪等な script。LOGIN / PASSWORD は付与しない。
2. **bind mount は `../e2e/fixtures/alt-backend/db-init:/docker-entrypoint-initdb.d:ro`**。compose.staging.yaml の `alt-backend-db` service で volumes エントリ 1 行追加。ディレクトリ渡しなので、将来 seed data など別 SQL を追加しても同じマウントで扱える。
3. **init script は初回ブート時のみ実行される**のは Postgres 標準動作。`docker compose down -v` でボリュームを消せば毎回確実に再実行される。run.sh は `trap cleanup EXIT` で `down -v --remove-orphans` するため、各 run で必ず fresh DB が立ち上がる。
4. **本番の `db/init/01-create-users.sh` は踏襲しない**。本番 script は Docker secret 読み込み + LOGIN password 設定 + 権限付与を一貫して行うが、staging にはその要件がない。staging 用に別ファイルを置くことで「prod と staging が別目的」を明示する。

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| 本番の `db/init/01-create-users.sh` を staging にもマウントする | staging 用の secret ファイルを `e2e/fixtures/staging-secrets/` に 6 個追加する必要があり、fixture 管理が膨らむ。staging で LOGIN を必要とするサービスがないため完全に over-kill |
| Atlas マイグレーションから `GRANT ... TO <other-service-user>` を削除する | 本番の権限分離を壊す。他サービスのユーザーに対する GRANT は prod で意味を持つ |
| staging 専用の migration set を別ディレクトリで管理する | migrations-atlas が `.atlas.sum` で hash chain を管理しており、staging と prod で別経路を持つと drift する。本 ADR [[000778]] の「prod と同じマイグレーションで staging を立ち上げる」原則に反する |
| NOLOGIN ではなく LOGIN + 固定パスワードで作成 | alt-staging network は internal: true で外部到達がなく、他サービスが接続しない staging では LOGIN 属性自体が不要。NOLOGIN のほうが「このロールは GRANT 先としてのみ存在する」意図を明示できる |
| `ALTER DATABASE ... OWNER TO ...` のような文に書き換える migration 修正 | 79 migrations の複数箇所を改修する必要があり、prod の GRANT 前提を壊す可能性もある。影響範囲が大きすぎる |

### 範囲外 (今回やらないこと)

- 本番 `db/init/` スクリプトの staging への一般化 — 本 ADR は alt-backend staging のみが対象
- 他サービスの staging DB に同じパターンを適用 — 各サービスの staging DB profile ごとに要否を判断
- Atlas migrator の実行 UID / cache ディレクトリ権限問題 (`mkdir /.cache: permission denied` の warning) — migrate 自体は成功するので追跡のみ

## Consequences

### Pros

- **alt-backend staging stack が 1 回目の bring-up で正常に migrate → serve まで到達する**。[[000778]] の run.sh が 2 連続で 21/21 green を達成する前提がこれで成立
- **fixture は 1 ファイルの SQL で完結する**。Docker secrets を 6 個追加する代替案に比べてメンテ面積が小さく、レビューも 1 SQL で済む
- **本番と staging の役割分離が明示的**になる。`db/init/` (prod) と `e2e/fixtures/alt-backend/db-init/` (staging) で別ディレクトリ・別ファイルなので、どちらを触っているか commit diff で即座に分かる
- **他サービスの staging profile にも同パターンが再利用できる**。将来 pre-processor / search-indexer などの staging DB が必要になった時、同じ `NOLOGIN で先作り` 方針を横展開できる

### Cons / Tradeoffs

- **staging DB の volume を消さないと init script は再実行されない**。Postgres image の標準動作ではあるが、KEEP_STACK=1 で debug を続けた後に SQL を差し替えたケースでは手動で volume prune が必要。run.sh の `trap cleanup EXIT` が通常は吸収する
- **init script の drift が migrations 側の GRANT 追加と同期しない**。将来マイグレーションが `GRANT ... TO new_user;` を追加した場合、staging init script にも `new_user` を追加しないと migrator が落ちる。README または CI で role 一覧の差分を監視する仕掛けは未整備
- **本番環境では使われないため alt-deploy / release-deploy の matrix からは無関係**だが、レビュー時に「本番 compose と何が違うのか」を毎回説明する必要がある。ADR への参照で説明コストを下げる
- **NOLOGIN role は GRANT 受容以外の副作用がない**はずだが、将来 Atlas が `ALTER ROLE <role> LOGIN` のような文を migrations に含めたら別の対処が必要になる
