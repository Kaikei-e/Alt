# ADR-000684: Writer paragraph micro-generation を quality-tuning フェーズへ進める

## Status

Accepted

## Date

2026-04-10

## Affected Services

- acolyte-orchestrator (Python / LangGraph)

## Context

[[000673]] で Writer の paragraph micro-generation（1 claim = 1 paragraph = 1 LLM call）が導入され、v5 では 9/9 accepted まで回復した。しかし以下の品質課題が残っていた:

1. **`num_predict=1000` が全 section_role でハードコード** — conclusion は synthesis + thinking overhead で tokens を多く消費するが、analysis と同じ 1000 tokens では不足し、v5 では conclusion が 48 chars と極端に短くなっていた
2. **Prompt に長さのヒントがない** — LLM が段落の目標長を知らないため、section_role 間でばらつきが生じていた
3. **Claim-quote 対応が暗黙的** — supporting_quotes が番号なしのリストで、どの evidence_id に対応するか prompt 上で不明瞭だった

一方、accepted paragraph の immutability と revision 時の delta feedback の仕組みは正しく機能しており、これらは維持する必要がある。

## Decision

WriterNode に Settings 注入と per-role パラメータ分離を導入し、prompt のガイダンスを強化する。

### per-role num_predict

`config/settings.py` に section_role 別の `num_predict` を追加:

| section_role | num_predict | 根拠 |
|---|---|---|
| analysis | 1200 | 事実報告。1 claim あたり十分なトークン |
| conclusion | 1500 | synthesis + thinking overhead で多くの tokens が必要。v5 の 48 chars 問題を解消 |
| executive_summary | 600 | 簡潔な要約。過剰なトークン予算は冗長な出力を誘発 |
| その他 | 1000 | 既存のデフォルト値を維持 |

WriterNode に `_role_num_predict(section_role)` メソッドを追加し、`__call__` 内のハードコード `num_predict=1000` を置換。

### Settings 注入パターン

`WriterNode.__init__(self, llm, *, settings=None)` で Settings を受け取る。`settings=None` の場合はデフォルト `Settings()` にフォールバックし、後方互換性を維持する。FactNormalizerNode（[[000677]]）と同じパターン。

`report_graph.py` の `build_report_graph` で settings を WriterNode に渡す:

```python
if settings is not None:
    writer = WriterNode(llm, settings=settings)
else:
    writer = WriterNode(llm)
```

### Prompt 改善

**target_length ヒント**: 各 paragraph prompt に `<target_length>` タグを追加。analysis は「200-400字」、conclusion は「150-300字」、ES は「100-200字」。LLM に対する soft target であり、rigid な制約ではない。

**numbered quote mapping**: `_format_supporting_quotes()` を拡張し、evidence_id と quotes の対応を明示:

```
- [art-1] "The AI market expanded by 20%"
- [art-2] "Growth driven by enterprise adoption"
```

これにより、LLM が出力で `[1]`, `[2]` の citation を正しく付与しやすくなる。

### テスト

`tests/unit/test_writer_quality_tuning.py` に 7 件のテストを追加:

| テスト | 検証内容 |
|---|---|
| `test_analysis_paragraph_uses_role_num_predict` | analysis → settings の 1200 が LLM に渡る |
| `test_conclusion_paragraph_uses_higher_num_predict` | conclusion → 1500 が渡る |
| `test_es_paragraph_uses_shortest_num_predict` | ES → 600 が渡る |
| `test_paragraph_prompt_includes_target_length_hint` | prompt に `<target_length>` タグが含まれる |
| `test_paragraph_prompt_numbered_quote_mapping` | supporting_quotes に `[evidence-id]` が付く |
| `test_accepted_paragraphs_immutable_after_quality_tuning` | settings 導入後も accepted paragraph が revision で変更されない |
| `test_conclusion_paragraph_minimum_output_length` | conclusion prompt に target_length ヒントがある |

既存テスト 31 件 (paragraph, revision_delta, claim_based) の regression なし。

## Consequences

### Pros

- conclusion の `num_predict` が 1000→1500 に増加し、thinking overhead を吸収して極端に短い出力を防止
- ES の `num_predict` が 1000→600 に減少し、不必要に冗長な要約を抑制
- `<target_length>` ヒントにより、LLM が section_role に適した長さの段落を生成しやすくなる
- numbered quote mapping により citation の正確性が向上
- Settings 経由の設定のため、環境変数 (`PARAGRAPH_NUM_PREDICT_CONCLUSION=1500`) で運用時に調整可能
- 後方互換性を維持: `WriterNode(llm)` は引き続き動作

### Cons / Tradeoffs

- `num_predict` の値は empirical に決定しており、モデル変更時に再調整が必要
- `<target_length>` は soft target であり、LLM が必ず従う保証はない。今後の run で実際の出力長を計測し、値を調整する必要がある
- conclusion の `num_predict=1500` は thinking overhead を含むため、実質的な出力テキスト量は 300-500 tokens 程度になる可能性がある
