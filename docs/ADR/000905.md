# ADR-000905: Wave 4 残債 (4-A flag flip / 4-B recap persist-stage / 4-D Phase 2 RLS) を 3 sub-PR で完成させ Surface Planner v2 を本番で稼働させる

## Status

Accepted

## Date

2026-05-20

## Affected Services

- rag-orchestrator (Go)
- alt-backend (Go)
- recap-worker (Rust)
- knowledge-sovereign (Go)
- alt-deploy (compose / CI workflow)

## Context

Wave 1〜4-A scaffolding は [[000850]] [[000851]] [[000852]] [[000853]] [[000854]] [[000855]] で landed 済みで、`EventLogSurfaceScoreResolver` は production projector 経路に plug されている。一方で **以下 3 件が未完** のままで v2 Surface Planner は実 signal を受けていない:

1. **Wave 4-A 完成 (env flip + tenant 伝播 + Pact CDC + alt-deploy 同期)** — rag-orchestrator の `AppendEventClient` と Pact consumer test は [[000855]] で実装済みだが、`RAG_ORCHESTRATOR_KNOWLEDGE_EVENT_EMIT=false` がデフォルトで live behavior は何も変わっていない。Pact CDC を broker publish せず env を `true` に flip すると過去 `X-Service-Token` 401 cascade ([[000722]]) と同型の事故が起きる。
2. **Wave 4-B (recap-worker persist-stage 配線)** — `recap-worker/src/clients/knowledge_sovereign/` の client + Pact test は実装済みだが、`pipeline.rs` の cluster 確定後で `emit_recap_topic_snapshotted` が呼ばれていない。これがないと `RecapTopicSnapshotted` event が積まれず、Surface Planner v2 の `topic_overlap_count` が常時 0。
3. **Wave 4-D Phase 2 (RLS policy 活性化)** — Phase 1 (caller-first `SET LOCAL alt.user_id`) は本番に landed 済み、`migrations/00014_add_rls_to_knowledge_loop_entries.sql` は存在する。`alt.user_id` 未設定の read が 0 件であることを 24h log で baseline 確認後、`ENABLE ROW LEVEL SECURITY` を活性化する必要がある。逆順 (caller wiring 未完で RLS 活性化) は `knowledge_loop_entries` 全行不可視 → UI 空白。

これら 3 件が closure しない限り Phase D 以降 (projector の Acted consume、epistemic Review、Lens-weighted bucket) はすべて v1 fallback で動き続けるため、構想 v2 の効果が現れない。Wave 4 残債を本 ADR で 1 つの架橋として固定する。

## Decision

Wave 4 残債を 3 つの独立 sub-PR で landed する。順序固定。各 PR は独立 revert 可能。

### PR-C1: Wave 4-A 完成 (Pact CDC publish + tenant 伝播 + alt-deploy 同期 + env flip)

1. **Pact consumer test の broker publish** — `rag-orchestrator/internal/adapter/contract/knowledge_sovereign_consumer_test.go` (実装済み) を `CGO_ENABLED=1 go test -tags=contract` で実行し pact ファイル生成、`./scripts/pact-check.sh --broker` で broker に publish。
2. **knowledge-sovereign provider verification** — `app/driver/contract/provider_test.go` の verifier list に新 consumer pact (`rag-orchestrator-knowledge-sovereign.json`) を追加。`CGO_ENABLED=1 go test -tags=contract` が green。
3. **Tenant 伝播** — `alt-backend/app/connect/v2/knowledge_loop/handler.go` で `X-Alt-Tenant-Id` header を rag-orchestrator hop に forward。`rag-orchestrator/internal/adapter/connect/augur/handler.go` に `extractTenantID(headers)` helper を追加し、未提示なら `connect.CodeUnauthenticated` で reject。`AugurConversationLinkedInput.TenantID` を必須化。
4. **alt-deploy 同期** — 別リポジトリの `services.yaml` に rag-orchestrator → knowledge-sovereign の pacticipant ペアを追加、`release-deploy.yaml` の `PACTICIPANT_ROLES` を更新、`.github/workflows/run-e2e-suite.yml` の allowlist を追加。
5. **本番反映手順** — 上記を含む PR を main merge → `RAG_ORCHESTRATOR_KNOWLEDGE_EVENT_EMIT=false` のまま deploy (live behavior 不変) → `./scripts/pact-check.sh --broker` green / `c2quay status` matrix 確認 → compose env を `true` + `RAG_ORCHESTRATOR_KNOWLEDGE_SOVEREIGN_URL=http://knowledge-sovereign:9300` に flip → rag-orchestrator のみ rolling restart → `surface_bucket_assigned_total{version="v2",bucket="continue"}` rate を 24h 観測 → `cross_user_isolation_violation_total` が 0 を維持していることを確認。

### PR-C2: Wave 4-B (recap-worker persist-stage 配線)

1. **`pipeline.rs` 配線** — recap cluster 確定後に `client.emit_recap_topic_snapshotted(TopicSnapshottedInput { ... })` を call。emit failure は `tracing::warn!` で記録し recap pipeline は止めない。payload は canonical contract §6.4.1 通り (`recap_topic_snapshot_id`, `top_terms[]`, `cluster_id`, `snapshot_window_start/end`)。`dedupe_key = "recap.topic_snapshotted.v1:<user_id>:<snapshot_id>"`。
2. **Pact consumer test の再 publish** — `cargo test --lib contract -- --ignored` で pact 生成、`./scripts/pact-check.sh --broker` で publish。
3. **knowledge-sovereign provider verification** に recap-worker pact を追加。
4. **alt-deploy 同期** — recap-worker → knowledge-sovereign の pacticipant 追加。
5. **本番反映** — PR merge → deploy → recap pipeline 1 サイクル毎の `RecapTopicSnapshotted` emit を確認 → metric `surface_bucket_assigned_total{version="v2",bucket="now"}` の立ち上がり (topic overlap が立つ entry 経由) を 24h 観測。

### PR-C3: Wave 4-D Phase 2 (RLS policy 活性化)

1. **24h baseline 確認** — Phase 1 caller wiring (`app/driver/sovereign_db/with_user_context.go` で `SELECT set_config('alt.user_id', $1, true)`) が本番で全 read 経路を通っていることを log 集計で確認。`alt_user_id_set=false` log line が 0 件であること。
2. **Migration apply** — `cd knowledge-sovereign/migrations && atlas migrate hash && atlas migrate apply --env production` で `00014_add_rls_to_knowledge_loop_entries.sql` を投入。policy は `current_setting('alt.user_id', true)` で fail-closed (session-var 未設定 → NULL → policy false → 0 行)。
3. **Post-check** — 既存 e2e/hurl が pass / 未認証で `SELECT * FROM knowledge_loop_entries` を叩くと 0 行 / `cross_user_isolation_violation_total = 0` を 24h 観測。
4. **切り戻し** — caller wiring 不具合発覚なら `git revert` (RLS が無効なら影響なし)、migration 投入後の問題なら `DROP POLICY` + `DISABLE ROW LEVEL SECURITY` で即時切戻し。

### 検討した代替案

- **3 件を 1 PR にまとめる** — blast radius が広く、Wave 4-A の `X-Service-Token` 型事故再演リスクを上げる。Wave 4 §1.4 / [[000853]] の "独立 PR + Pact CDC + alt-deploy 同期" 規律を踏襲し 3 sub-PR に分離。
- **RLS Phase 2 を先行する** — 24h baseline (Phase 1 caller wiring の log) が必要なので順序固定。
- **Wave 4-B を Wave 4-A より先に** — どちらが先でも構わないが、Augur emit は既に [[000855]] で coded & dormant 状態なので flag flip だけで効くため Wave 4-A を先にして観測ベースラインを取りに行く。

## Consequences

### Pros

- Surface Planner v2 が `topic_overlap_count` / `has_augur_link` / `version_drift_count` の実 signal を受けて placement を decide できる (構想層 §1 cognitive state を実装に下ろせる)
- Cross-user isolation が DB policy で fail-closed に守られ、Wave 4-D defense-in-depth が完成 (F-001 violation を policy で再防御)
- すべて Wave 4 §1.4 の規律 (Pact CDC + alt-deploy 同期 + env flip 後 24h 観測) を守る順序で landed
- Phase D (projector の `knowledge_loop.acted.v1` consume) と Phase E (epistemic Review) の入力 stream が確保される

### Cons / Tradeoffs

- 各 PR は独立 deploy で 24h baseline 観測を要するため、Wave 4 完成は最短でも 1 週間 (3 deploy + 各 24h)
- Wave 4-A flag flip 後に projector の `event_dropped_total{reason}` が spike する可能性がある (新 event type を payload validator が reject する経路)。`./scripts/pact-check.sh --broker` の事前 green が必須
- RLS 活性化後は `alt.user_id` セット忘れで 0 行となるため、新 read 経路追加時は必ず `withUserContext` 経由を強制 (lint rule / code review checklist)

## 実装履歴

- 2026-05-20: **PR-C1 (Wave 4-A)** はコード経路を再確認。alt-backend は JWT 由来の `user.TenantID` を rag-orchestrator hop に渡し、rag-orchestrator は `extractTenantID(req.Header())` で `X-Alt-Tenant-Id` 不在を `connect.CodeUnauthenticated` で fail-closed させる経路が既存実装に landed 済み。残る作業 (Pact consumer test の broker publish、knowledge-sovereign provider verifier list 追加、alt-deploy `services.yaml` の pacticipant 同期、`RAG_ORCHESTRATOR_KNOWLEDGE_EVENT_EMIT` env flip + 24h 観測) は本番反映フェーズへ繰り延べ。
- 2026-05-20: **PR-C2 (Wave 4-B)** を実装。`recap-worker/src/clients/knowledge_sovereign.rs` の `publish` モジュールを `pub(crate)` に昇格し、`pipeline/persist.rs::FinalSectionPersistStage` に `Option<Arc<KnowledgeSovereignClient>>` を持たせる `with_sovereign` chain method を追加。`pipeline/orchestrator.rs` で `RECAP_WORKER_KNOWLEDGE_EVENT_EMIT=true` + `RECAP_WORKER_KNOWLEDGE_SOVEREIGN_URL` env が揃ったときだけ client を構築する DI を入れた。persist 完了後、`result.genre_results` を走査して `(cluster_id, top_terms)` のある cluster を `ConfirmedCluster` に詰め、`publish_topic_snapshots` を warn-and-continue で呼ぶ。`cargo fmt --check` / `cargo clippy --all-features -- -D warnings` / `cargo test --lib pipeline::persist` (18/18) green。残る作業 (Pact `--ignored` test の broker publish、provider verifier に recap-worker pact を追加、alt-deploy pacticipant 同期、env flip + 24h `surface_bucket_assigned_total{version="v2",bucket="now"}` 観測) は本番反映フェーズへ。
- 2026-05-20: **PR-C3 (Wave 4-D Phase 2)** は migration `00014_add_rls_to_knowledge_loop_entries.sql` が既存。Phase 1 caller wiring (`app/driver/sovereign_db/with_user_context.go`) が production に landed 済みで、活性化は 24h `alt_user_id_set=false` 0 件確認後の `atlas migrate apply` ops に gating される。本 ADR ではコード変更なし。
