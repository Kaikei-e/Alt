# ADR-000942: knowledge-sovereign の admin HTTP surface に明示 json タグと封筒レスポンスを導入し ADR-000765 §3 の据え置き判断を解消する

## Status

Accepted

## Date

2026-07-07

## Affected Services

- knowledge-sovereign (Go) — `driver/sovereign_db/retention.go` (`PartitionInfo` / `RetentionLogEntry`)、`driver/sovereign_db/snapshot.go` (`SnapshotMetadata` / `TableStorageInfo` + `GetStorageStats` SQL)、`handler/retention_handler.go` / `handler/snapshot_handler.go` / `handler/storage_handler.go` のレスポンス整形
- altctl (Go) — コード変更なし。`cmd/home_retention.go` / `cmd/home_storage.go` / `cmd/home_snapshot.go` の既存デコード構造体（snake_case + 名前付き封筒）が今回の決定の正解として扱われる

## Context

[[000765]] は knowledge-sovereign に Hurl E2E スイートを追加した際、admin HTTP surface (`/admin/snapshots/*`, `/admin/retention/*`, `/admin/storage/stats`) が `encoding/json` の無タグ struct を直接マーシャルしており PascalCase 無タグ出力になっている事実を「契約として固定」しつつ、"snapshot 応答の PascalCase を是正する json tag を sovereign 本体に入れる" 代替案を **却下し「将来 json tag を入れる判断は別 ADR に委ねる」** と明記した (§3 検討して却下した代替案の 5 番目)。

2026-07-06 の大規模リポジトリコードレビュー (`docs/plan/20260706largerepocodereview.md` L648-654) で、この据え置きが実害を生んでいることが判明した:

1. **`GET /admin/retention/status`** — `retention_handler.go` はトップレベル JSON 配列（無タグ `[]RetentionLogEntry`、ログが 0 件のときは `null`）を返すが、altctl の `cmd/home_retention.go` は `{"logs": [...]}` の封筒 + snake_case フィールドを期待する。ログが 1 件でもあれば **配列を struct にアンマーシャルしようとして必ずエラー**になり、0 件のときは `null` が黙って零値 struct にデコードされて「ログなし」に見える（ハード失敗とサイレント失敗の両方を併発）。
2. **`GET /admin/retention/eligible`** — ハンドラは `[{table, eligible: [...]}]` のテーブル単位グループ配列を返すが、CLI は `{"partitions": [...]}` のフラット配列を期待する。常に失敗するか空表示になる。
3. **`GET /admin/storage/stats`** — `TableStorageInfo` は既に `table_name`/`row_count`/`total_size`/`total_bytes`/`is_partitioned` の json タグを持つが、封筒が無く、CLI が要求する `table_size`/`index_size`（`total_size` から分離したテーブル本体サイズとインデックスサイズ）も存在しない。
4. **`/admin/snapshots/{create,latest,list}`** — `SnapshotMetadata` は無タグのため PascalCase (`SnapshotID`, `EventSeqBoundary` 等) で出力される。`list` は altctl 側が `{"snapshots": [...]}` を期待するのに対しトップレベル配列を返すため **必ず unmarshal エラー**（ハード失敗）。`create`/`latest` は altctl 側が snake_case フラット構造体で受けるため、PascalCase キーが一つもマッチせず **全フィールドが零値のまま "成功" 扱いになる**（サイレント失敗、CLAUDE.md Rule 8 が禁じる形の握り潰しと同型）。

さらに `handler/snapshot_handler.go:145` の `CreateSnapshot` は「3. Get active projection version」とコメントしながら `projVersion := 1` をハードコードしており、`GetActiveProjectionVersion` を一度も呼んでいなかった（`knowledge_projection_versions` の active version が 1 以外に切り替わった環境で、スナップショットメタデータの `projection_version` が事実と乖離し、`IsCompatibleWith` による復元互換性判定を汚染する）。

これら 4 件の契約不一致は CLAUDE.md ルール 7（"Producer wiring PRs require CDC RED first"）の対象であり、`altctl/internal/sovereignclient/contract/consumer_test.go` の既存 Pact テストは実際にはどちらの実装とも一致しない**第三の誤った形状**（`TestRetentionEligible` が `{Table, PartitionName, EventSeqMax}` の PascalCase フラット配列を pin していた）を固定していたため、これまで CI の green が実害を検出できていなかった。同様に `driver/contract/provider_test.go` の provider 検証はハンドコードされた `startStubServer` に対して行われており、実ハンドラの動作を一切検証していなかった。

本 ADR 作成にあたり、エフェメラルな Postgres コンテナ + `atlas migrate apply` + `go run ./main.go` で実ハンドラを起動し、修正前 (RED) と修正後 (GREEN) の実レスポンスを直接比較して契約不一致を実証した（Docker イメージの再ビルドは不要、Hurl の staging compose ビルドは今回未実施 — 下記 Consequences 参照）。

## Decision

**knowledge-sovereign の admin HTTP surface（`/admin/snapshots/*`, `/admin/retention/*`, `/admin/storage/stats`）に明示的な snake_case json タグを導入し、list 系エンドポイントは名前付き封筒（`{"logs": [...]}` / `{"partitions": [...]}` / `{"tables": [...]}` / `{"snapshots": [...]}`）でラップする。「consumer is truth, provider follows」(`docs/runbooks/pact-broker-ops.md`) の原則に従い、altctl の既存デコード構造体（`cmd/home_retention.go` / `home_storage.go` / `home_snapshot.go`）を正として provider 側を合わせる。これにより ADR-000765 §3 が据え置いた PascalCase-vs-json-tag の判断を確定する。**

### 採用する構成

1. **`driver/sovereign_db/retention.go`**: `PartitionInfo`（`partition_name`/`range_start`/`range_end`/`row_count`/`size_bytes`）と `RetentionLogEntry`（`log_id`/`run_at`/`action`/`target_table`/`target_partition`/`rows_affected`/`archive_path`/`checksum`/`dry_run`/`status`/`error_message`）に json タグを追加。
2. **`driver/sovereign_db/snapshot.go`**: `SnapshotMetadata` の全フィールドに snake_case json タグを追加。`TableStorageInfo.TableName` のタグを `table_name` から `name` に変更し、`table_size`/`index_size` フィールドを新設。`GetStorageStats` の SQL に `pg_size_pretty(pg_table_size(c.oid))` / `pg_size_pretty(pg_indexes_size(c.oid))` を追加し、既存の `pg_total_relation_size` 由来の合計サイズと並置する。
3. **`handler/retention_handler.go`**: `handleRetentionStatus` を `{"logs": [...]}` でラップ（空スライスは `[]`、`null` を返さない）。`handleEligiblePartitions` をテーブル単位グループ配列からフラット化し、各行に `table_name` を持たせた上で `{"partitions": [...]}` でラップ。
4. **`handler/snapshot_handler.go`**: `handleListSnapshots` を `{"snapshots": [...]}` でラップ。`SnapshotRepository` インターフェースに `GetActiveProjectionVersion` を追加し、`CreateSnapshot` の `projVersion := 1` ハードコードを実呼び出しに置換（呼び出し結果が `nil`、すなわち一度も projection version が activate されていない場合のみ `1` にフォールバックする — この 1 経路は「まだ有効化されていない」という区別可能な初期状態であり、Rule 8 が禁じる「配線忘れの黙殺」とは異なる）。
5. **`handler/storage_handler.go`**: `{"tables": [...]}` でラップ。
6. **`driver/contract/provider_test.go`**: `startStubServer` に `/admin/retention/status` ルートを追加し、既存の `/admin/snapshots/*` / `/admin/retention/eligible` / `/admin/storage/stats` スタブ応答を新形状（snake_case + 封筒）に更新。
7. **`altctl/internal/sovereignclient/contract/consumer_test.go`**: `TestRetentionEligible` を altctl の実デコード構造体に一致するよう修正、`TestRetentionStatus` / `TestStorageStats` を新規追加、`TestSnapshotCreate` / `TestSnapshotLatest` / `TestSnapshotList` を新形状に更新。
8. **`e2e/hurl/knowledge-sovereign/{14,15,16,18}-*.hurl`** のアサーションを新形状に更新し、これまで Hurl カバレッジが存在しなかった `/admin/snapshots/list` と `/admin/retention/status` に新規シナリオ `19-admin-snapshot-list.hurl` / `20-admin-retention-status.hurl` を追加した。

### 検討して却下した代替案

| 案 | 却下理由 |
|---|---|
| altctl 側のデコード構造体を sovereign の現行 PascalCase 無タグ形状に合わせる | `docs/runbooks/pact-broker-ops.md` の "consumer is truth, provider follows" に反する。altctl の CLI は運用者が incident response 中に叩く disaster-recovery ツールであり、admin API の内部実装都合（無タグ struct のデフォルト挙動）に運用者向け CLI の入出力契約を従属させるべきではない |
| ADR-000765 §3 の据え置きをそのまま維持し、CDC テストのみ現状の PascalCase 形状に追従させる | レビューで判明した 4 件の契約不一致（うち 2 件はハード unmarshal エラー）を放置することになり、`altctl home snapshot list` / `altctl home retention status` が実運用で機能しない状態を追認するだけで根本解決にならない |
| `handler` 層でレスポンス変換のみ行い、`driver/sovereign_db` の構造体は無タグのまま維持 | ハンドラ側で全フィールドを都度手動マッピングする定型コードが増え、`sovereign_db.SnapshotMetadata` 等のフィールド追加のたびに変換漏れが起きるリスクが高い。構造体自体にタグを持たせるほうが単一の真実源になる |

## Consequences

### Pros

- **4 件の HIGH severity 契約不一致が解消**され、`altctl home snapshot {create,latest,list}` / `altctl home retention {status,eligible}` / `altctl home storage` が実際に機能するようになった。特に `list` 系 2 エンドポイントはハード unmarshal エラーを起こしていたため、運用者が disaster recovery 中に遭遇する最悪のタイミングでの失敗が解消される。
- **`snapshot_handler.go` の `projVersion := 1` ハードコードが解消**され、`knowledge_projection_versions` の active version が 1 以外の環境でもスナップショットメタデータの `projection_version` が事実と一致するようになり、`IsCompatibleWith` による復元互換性判定の正確性が回復した。
- **ADR-000765 §3 の据え置きが確定**し、今後 admin HTTP surface に新しいエンドポイントを追加する際の規約（snake_case json タグ + 名前付き封筒）が明文化された。
- **Pact CDC が実効化**された。`consumer_test.go` が altctl の実デコード構造体を pin し、`provider_test.go` のスタブも新形状に追従したことで、今後の drift は CI で機械的に検出される。
- **Hurl E2E のカバレッジ欠落が是正**された。`/admin/snapshots/list` と `/admin/retention/status` は ADR-000765 の 18 シナリオに含まれていなかった（"fails HARD"/"fails SILENTLY" だった 2 エンドポイントに限って E2E オラクルが無かった）。新規 2 シナリオでカバーした。

### Cons / Tradeoffs

- **`provider_test.go` の `startStubServer` は依然として実ハンドラを検証しない**。Pact のファイルモード検証はハンドコードされたスタブ mux に対して行われるため、今回のような「実装のバグ」（配列 vs 封筒、ハードコード値）を Pact 単体では検出できない。今回の RED/GREEN 確認はエフェメラルな Postgres + `go run` による手動オラクルに依存しており、恒久的な回帰防御は Hurl E2E（`compose.staging.yaml` の `knowledge-sovereign` profile、要 `docker compose build`）に委ねられている。
- **本 ADR のマージ時点では Hurl E2E スイートの実行を行っていない**。`compose.staging.yaml` の `knowledge-sovereign` サービスは `build:` コンテキストを持つため、更新後のコードを反映するには `docker compose ... build` が必要であり、これは別途の承認を要する運用上の一時停止事項として記録する（下記参照）。
- **`/admin/retention/eligible` の空リスト表現が意味的に弱くなった**。旧形状は「テーブルごとに 0 件でもエントリが存在する」ことで両テーブルが処理されたことを保証していたが、新形状のフラット配列は eligible が 0 件のテーブルを表現しない。運用者が「どのテーブルがチェックされたか」を知りたい場合は `/admin/storage/stats` や別途のログを参照する必要がある。
- **`GetStorageStats` の SQL 呼び出しコストが増加**する（`pg_table_size`/`pg_indexes_size`を追加で計算するため、`pg_total_relation_size` のみだった旧クエリよりわずかに重い）。`knowledge_%` に絞られたテーブル数は少数であるため実運用上の影響は軽微と判断した。
