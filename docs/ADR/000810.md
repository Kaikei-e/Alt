# ADR-000810: pact-check.sh の manual verification 経路から consumer pact publish を剥がし Broker content-mutation 再発を塞ぐ

## Status

Accepted

## Date

2026-04-20

## Affected Services

- **Alt (scripts)** — `scripts/pact-check.sh` の条件修正 1 行 + 根拠コメント
- **Alt (docs)** — `docs/runbooks/pact-broker-ops.md` に stale verification 復旧手順

## Context

2026-04-20 00:58 UTC の alt-deploy `release-deploy` workflow_dispatch (sha=1018dc862) で、`pact-manual` ジョブが以下で落ちた:

```
📨 Attempting to publish pact for consumer: recap-worker against provider: alt-backend
❌ Pact publication failed:
   contracts: Cannot change the content of the pact for recap-worker version
   1018dc862c8853509ee54dbcbb343b609853ecfc and provider alt-backend, as race
   conditions will cause unreliable results for can-i-deploy.
```

`pact-manual` は release-deploy で `./scripts/pact-check.sh --publish-only --publish-manual-verifications` を呼ぶジョブで、本来は **bridging provider verification record** のみを POST するべき経路だった ([[000740]] で Pact Broker を gate 化した際の設計)。ところが script のフロー制御が甘く、`MANUAL_ONLY=true` でも Broker publish ブロック (line 382〜) が走って consumer pact files を `pact-broker-cli publish --merge` してしまっていた。

Broker は 2021-08 以降、同一 consumer version に対する content-mutate publish を拒否する ([PactFlow: Disabling dangerous contract modification](https://docs.pactflow.io/docs/troubleshooting/disabling-dangerous-contract-modification/))。matcher に具体値を与えない Pact library (Rust recap-worker consumer 等) は random example を出すため、同じコミット SHA でも content が揺れる。これが race condition で `pact-publish` ジョブ (先に同じ consumer pact を publish 済) との衝突になり、`pact-manual` で HTTP 409。

波及:
- `gate (recap-worker)` は「recap-worker@1018dc862 に対して alt-backend の verified pact なし」で fail
- `gate (alt-backend)` も過去の stale verification-results/1858 (failure) を Broker から拾って fail
- これが `deploy` 全体を block + partial-deploy で production 障害化

`pact-check.sh` 内の関連コード (line 382):

```bash
# 改修前
if [[ "$MODE" == "broker" && "$ROLE_FILTER" != "provider" ]]; then
    # consumer pact publish via pact-broker-cli publish --merge
    ...
fi
```

`--publish-only` は `MODE=broker` を、`--publish-manual-verifications` は `MANUAL_ONLY=true` を設定するが独立で、両方指定時に consumer publish も走ってしまう。

なお「stale verification-result が can-i-deploy を block する」という二次問題 (今回の `gate (alt-backend)`) は本質的に Broker 上のデータ問題であり、script 修正では解消しない。runbook で復旧手順を標準化する。

## Decision

### 1. `pact-check.sh` の Broker publish ブロック条件に `MANUAL_ONLY` 除外を追加

```bash
# 改修後
if [[ "$MODE" == "broker" && "$ROLE_FILTER" != "provider" && "$MANUAL_ONLY" != "true" ]]; then
  # 根拠コメント: release-deploy 24643555145 (2026-04-20) で
  # --publish-manual-verifications が consumer pact を content-mutate
  # 再 publish して Broker から 409 を食らった事故の再発防止
  ...
fi
```

bridging manual block (line 474〜) は従来どおり `MANUAL_ONLY=true` で走る。そちらは provider verification-results を POST する純粋な bridging 経路で、consumer pact file には触れない実装。

### 2. `docs/runbooks/pact-broker-ops.md` に Section 9 追加

stale verification で can-i-deploy が false-block される症状に対し、**2 段の復旧手順**を明記:

- **Primary**: `pact-broker-cli verify` で provider 側を実際に再テスト → success record が最新化、Broker matrix が古い failure を latest から押し出す。破壊的操作なし、監査トレース自動生成。
- **Secondary (force-override)**: `pact-broker-cli create-or-update-verification --success true` を **真に技術的に再 verify 不可能な場合のみ** 許可。使用条件として:
  1. Linear issue + 2 人承認
  2. `--build-url` を Linear URL に固定 (自由文字列禁止)
  3. 実行後 slack #prod-audit に invalidation URL + 理由 + approver を post

security 監査 (OWASP A08 Software or Data Integrity Failures) で「force-success を runbook primary にすると production gate override 経路が手順書化される」と指摘されたため Secondary の 3 条件は非妥協。

**却下した代替**:

- **`MANUAL_ONLY=true` で script 全体を早期 return** — bridging block (line 474) が動かなくなり、kratos / mq-hub / recap-worker Rust stub の manual verification が Broker から欠落する。必要な副作用と不要な副作用を切り分けるべき。
- **`pact-publish` と `pact-manual` を同一 job に統合** — 並列実行の CDC isolation ([[000740]] の per-pacticipant matrix 方針) を崩す。
- **Broker で `allow_dangerous_contract_modification=true` を設定** — Pact Broker issue [#582](https://github.com/pact-foundation/pact_broker/issues/582) で dangerous 扱いされる理由は「race condition で can-i-deploy が unreliable になる」ことであり、gate の信頼性を意図的に下げる選択になる。却下。
- **Runbook に force-success を primary として載せる** — security 監査で「A08 Integrity Failure 相当」と判定。Secondary 扱い + 3 条件必須に限定。

### 3. 将来の予防策 (本 ADR の scope 外 backlog)

- `scripts/pact-invalidate.sh` wrapper を作り、`PACT_ALLOW_FORCE_SUCCESS=true` + Linear URL regex を CLI 側で強制 (default refused)
- `record-deployment` model への全面移行 ([[000740]] の superseding)。tags-based 判定箇所を削除すれば stale failure が can-i-deploy の判断対象から自然に落ちる

## Consequences

### Pros

- **pact-manual の 409 再発が 1 行で構造的に塞がれる** — script の条件が `MANUAL_ONLY != "true"` を要求することで、manual bridging 経路は consumer pact を**絶対に**触らない。
- **CDC isolation 方針 ([[000740]]) を維持** — per-pacticipant matrix の fail-fast 独立性はそのまま。
- **Runbook が新しい失敗モードを吸収** — stale verification による false-block は Pact Broker の構造上避けられないが、復旧手順が手元にある。
- **Secondary の 3 条件 (Linear + 2 人承認 + audit post)** が production gate integrity (OWASP A08) の防御線として runbook に固定。将来誰かが深夜オンコールで easy force-override を試みても摩擦が生じる。

### Cons / Tradeoffs

- **script の 1 箇所修正は unit test でカバーされていない** — `scripts/test_pact_check.sh` 相当が存在しないため、回帰は CI で見えない。将来 script 全体のテスト整備で拾う (backlog)。
- **Primary (再 verify) はテスト実行時間を要する** — provider test が長い (tag-generator Python / Rust recap-worker) 場合 runbook 手順で 5〜10 分かかる。force-success の速さとのトレードオフだが、integrity 優先。
- **`pact-invalidate.sh` wrapper は未実装** — Secondary の 3 条件は runbook 記載のみで CLI 強制ではない。遵守は操作者の規律依存。短期的許容、中期で wrapper 化。
- **record-deployment 全面移行は未着手** — [[000740]] の legacy tags-based 判定が残る限り stale failure 問題は根治しない。本 ADR は対症療法。
