# ADR-000739: Pact CDC の残ギャップを埋めてプロバイダ検証の orphan を解消する

## Status

Accepted

## Date

2026-04-15

## Affected Services

- rag-orchestrator (Go) — `internal/adapter/contract/recap_worker_consumer_test.go` 新設
- recap-worker (Rust) — provider 側で 3 consumer pact を verify する基盤追加
- alt-backend (Go) — provider_test.go に search-indexer 契約 verify を追加
- search-indexer (Go) — provider_test.go で acolyte pact exclude 解除
- mq-hub (Go) — provider_test.go 新設（pact-go v2 `message.Handlers`）
- acolyte-orchestrator (Python) — consumer test に `X-Service-Token` matcher を追加

## Context

[[000591]] で 17 ペアの Pact CDC 基盤を配置したが、audit で 5 件の orphan が判明した:

1. `rag-orchestrator → recap-worker` (morning-letter `/v1/morning/letters/*`) が unguarded。retrieval パイプラインの critical path。
2. `search-indexer → alt-backend` Connect-RPC pact は生成されていたが alt-backend 側の provider 検証が無かった。
3. `search-indexer → mq-hub` 非同期メッセージ pact も orphan。search-indexer が consumer として生成したが mq-hub 側が verify していない。
4. `acolyte-orchestrator → search-indexer` consumer test が `X-Service-Token` を assert しておらず、search-indexer の provider test はこの pact を `intentionally excluded` として除外していた ([[PM-2026-025]] の根本原因)。
5. recap-worker は Rust で書かれており、consumer としての pact は生成するが provider 検証インフラが無かった。recap-evaluator / search-indexer / rag-orchestrator が recap-worker に対して作成した契約が誰にも verify されていなかった。

Pact が「飾り」として残ると、ADR-000722 相当の provider-side 要件変更が consumer の追従漏れで本番で 401 を起こす構造的リスクが続く。

## Decision

**5 件の orphan を全て埋める。** 各ギャップに対して以下を適用:

### D1: rag-orchestrator → recap-worker morning-letter 契約追加

`rag-orchestrator/internal/adapter/contract/recap_worker_consumer_test.go` を TDD で新設し、3 interactions (`FetchLatest` 200 / `FetchLatest` 404 / `FetchByDate` 200) を pin。Pact file は `rag-orchestrator/pacts/rag-orchestrator-recap-worker.json` に落ちる。

### D2: alt-backend の provider 検証に search-indexer 契約を追加

`alt-backend/app/driver/contract/provider_test.go` に `TestVerifySearchIndexerContract` を追加し、Connect-RPC `BackendInternalService.GetLatestArticleTimestamp` / `.ListArticlesWithTags` を JSON wire で stub 検証する。

### D3: mq-hub の provider 検証（async message）を新設

`mq-hub/app/driver/contract/provider_test.go` を新設。`pact-go/v2/message.Handlers` に 3 interactions (ArticleCreated / ArticleCreated-fat / IndexArticle) のハンドラを登録し、`provider.NewVerifier()` で replay する。これで search-indexer が consumer として宣言した契約が mq-hub の実際の `domain.NewEvent` + `eventToWireFormat` に対して検証される。

### D4: acolyte → search-indexer の token 強制と provider 検証組込

`acolyte-orchestrator/tests/contract/test_search_indexer_consumer.py` に `.with_header("X-Service-Token", ...)` を追加して再生成。`search-indexer/app/driver/contract/provider_test.go` の `findProviderPacts` の intentional-exclude コメントを撤去し、`acolyte-orchestrator-search-indexer.json` を candidates に加える。state handler に `search-indexer has indexed articles and a service token is configured` を追加して 2 provider state に対応。

なお本 ADR の token 再要求は migration window での防御多重化が目的で、後続の mTLS hard-cutover で X-Service-Token 自体が完全撤去される ([[000743]])。

### D5: recap-worker Pact の Option-token 残骸撤去

Rust client 側で `Option<ServiceToken>` graceful-fallback パターンを撤去したのに伴い、`recap-worker-alt-backend.json` と `recap-worker-tag-generator.json` を再生成。wiremock テストの `.and(header("X-Service-Token", ...))` matcher を削除。

## Consequences

### Pros

- orphan pact 5 件 が全て closed。audit 上の "consumer はいるが provider 検証されていない" が 0。
- PM-2026-025 型のサイレント auth degradation を構造的に遮断。consumer test が token を assert し、provider 側の pact 検証でも同じ token 要件を replay するため、consumer の header 忘れは CI で即座に fail。
- morning-letter が critical path にも関わらず unguarded だった状態が解消。recap-worker (Rust) の provider verify 基盤は [[000744]] で独立にインフラ化。

### Cons / Tradeoffs

- pact JSON の再生成が必要なため、X-Service-Token 撤去 ([[000743]]) のタイミングで本 ADR の D4 matcher も削除される予定。hard-cutover と本 ADR の token pin は一時的な共存状態を作るが、CI gate が両方を常時緑に保つので回帰リスクは低い。
- mq-hub の provider 検証は async message 固有の API (`pact-go/v2/message.Handlers`) を使う都合、sync HTTP 検証より stub 実装の認知負荷が高い。将来 pact ファイル更新時に handler map を手で追従する必要がある。
- recap-worker (Rust) の provider 検証インフラは [[000744]] で別途導入しており、この D5 だけでは Rust 側は self-consistent に閉じない (4 pacts がまだ Rust 側で unverified)。
