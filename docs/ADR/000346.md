# CodeQL py/path-injection (CWE-22) 対策: パス検証パターンの変更

## ADR's STATUS

Accepted

## CONTEXT

CodeQL の静的解析が recap-subworker の以下の箇所で `py/path-injection` (CWE-22: Path Traversal) を検出していた:

- `infra/path_validation.py:57` — `validate_path()` 内の `Path.resolve()`
- `services/evaluation.py:152,156` — `golden_path.exists()` と `open(golden_path)`
- `app/routers/evaluation.py:137,150` — `.exists()` 呼び出し

既存の `validate_path()` は `Path.resolve()` + `Path.relative_to()` を使ったサニタイズパターンを採用していた。このパターンは実行時のセキュリティとしては正しく動作するが、CodeQL のテイント解析モデルがこのパターンをサニタイザーとして認識しないため、False Positive としてアラートが出続けていた。

CodeQL が認識するサニタイザーパターンは、`os.path.realpath()` を通した後の `str.startswith()` チェック、または `os.path.commonpath()` 比較である。

## DECISION MAKING

### 1. サニタイザーパターンの変更

`Path.resolve()` + `relative_to()` を `os.path.realpath()` + `startswith(base + os.sep)` に置換した。

- `os.path.realpath()` はシンボリックリンクを解決し、CodeQL がテイントの浄化起点として認識する
- `startswith()` の引数に `os.sep` (パス区切り文字) を末尾に付加することで、prefix attack を防止する
  - 例: `/app/data-evil/file` が `/app/data` にマッチしない (`/app/data/` とのみマッチ)
- ベースディレクトリ自体との完全一致 (`real_path == real_base`) も許可する

`startswith` ではなく `os.path.commonpath()` を使う選択肢もあったが、CodeQL の認識確実性と実装のシンプルさから `startswith` + trailing separator パターンを採用した。

### 2. テイント追跡の断絶防止

Router 層で `validate_path()` の戻り値 (`Path`) を `str()` に変換してから Service 層の `evaluate()` に渡していた。これにより CodeQL のテイント追跡が途切れ、Service 層側で再度アラートが発生していた。

検証済みの `Path` オブジェクトをそのまま Service 層に渡すよう変更した。Service 層の `evaluate()` / `evaluate_by_language()` のシグネチャを `str | Path` に拡張し、`Path` を受け取った場合は既にバリデーション済みとしてスキップ、`str` を受け取った場合は defense-in-depth として `validate_path()` を呼ぶ。

### 3. Defense-in-depth の維持

Service 層の `validate_path()` 呼び出しは削除しなかった。Router 以外の呼び出し元（`evaluate_by_language()` の内部呼び出し、将来の CLI ツール等）が存在するため、各レイヤーで独立した入力検証を維持する。

## RESULTS, EFFECTS

### PROS

- CodeQL の `py/path-injection` アラートが解消される
- OWASP / OpenStack Security Guide 推奨のパス検証パターンに準拠
- prefix attack (`/app/data-evil` が `/app/data` にマッチするバグ) を明示的に防止
- 既存のテスト (`TestEvaluatePathValidation`) は変更なしで全件パス
- 新規テスト 11 件で境界ケース（prefix attack、symlink escape、空 base_dirs 等）を網羅

### CONS, TRADEOFF

- `str.startswith()` によるパス比較は、理論上はファイルシステムの case sensitivity に依存する（Linux では問題なし、macOS の case-insensitive FS では潜在的リスク）。本プロジェクトは Docker (Linux) 上で動作するため影響なし
- Service 層で `isinstance(golden_data_path, Path)` による分岐を導入したため、型による暗黙的な「検証済み」判定に依存する。明示的なマーカー型（`ValidatedPath` など）の導入は過剰と判断し見送った
