# 000378: alt-frontend-sv ビルド・型チェック警告の全件解消

## ADR's STATUS

Accepted

## CONTEXT

alt-frontend-sv の `svelte-check` と `vite build` で複数の警告が出力されていた。警告の放置はCI上のノイズとなり、本来検知すべき新規の問題を見逃すリスクがある。ゼロ警告を維持できる状態にすることで、品質ゲートとしての信頼性を高める。

検出された警告は以下の3種類:

1. **`state_referenced_locally`** (svelte-check 5件) — `SwipeFeedScreen.svelte` で `$props()` の値を `$state()` の初期化に使用しており、Svelte 5 コンパイラが「初期値しかキャプチャしない」と警告
2. **バレルファイル経由の循環依存** (build 約18件) — `src/lib/api.ts` が `server/feed-api.ts` を re-export し、`feed-api.ts` → `connect/transport-server.ts` → `api.ts` の循環チェーンが形成
3. **`three-mesh-bvh` 内部の循環依存** (build 2件) — サードパーティライブラリ内部の既知の循環参照が adapter-node のバンドルフェーズで警告

加えて、three.js を含む 5.7MB のクライアントチャンクに対するチャンクサイズ警告も出力されていた。

## DECISION MAKING

### 1. `state_referenced_locally` — `svelte-ignore` コメントで抑制

`SwipeFeedScreen.svelte` では SSR から渡された初期値をローカル `$state` にコピーする設計が意図的であるため、Svelte 5 の `// svelte-ignore state_referenced_locally` を4箇所に追加した。

代替案として `$state.snapshot()` でラップする方法も検討したが、既にスプレッド構文 `[...(initialFeeds ?? [])]` で浅いコピーを行っており、deep snapshot は不要であった。コメントによる抑制が意図を最も明確に伝える。

### 2. バレルファイルの循環依存 — インポートパスを直接化

循環の根本原因は `transport-server.ts` がバレルファイル `$lib/api` 経由で `getBackendToken` をインポートしていた点にある。

- `transport-server.ts`: `$lib/api` → `$lib/server/auth` に変更
- `api/client/feeds.ts`: `CursorResponse` の type import を `$lib/api` → `$lib/server/feed-api` に変更

バレルファイル `api.ts` 自体は後方互換のため残し、リーフの route ファイル群からの利用は影響がないため変更しない。

### 3. `three-mesh-bvh` の循環依存 — direct dependency 化 + `onwarn` フィルタ

adapter-node は `package.json` の `dependencies` に含まれるパッケージのみを external 扱いにする。`three-mesh-bvh` はトランジティブ依存（`@threlte/extras` 経由）であったため、adapter がバンドルを試み内部の循環依存が警告として出力されていた。

`three-mesh-bvh` を direct dependency に追加することで adapter-node が external 化し、バンドル対象から除外される。加えて `vite.config.ts` の `build.rollupOptions.onwarn` で `node_modules` 由来の `CIRCULAR_DEPENDENCY` をフィルタし、将来の同種問題にも対応する安全ネットとした。

### 4. チャンクサイズ警告 — `chunkSizeWarningLimit` 引き上げ

tag-verse ページは three.js を含むため 5.7MB のチャンクが生成されるのは構造上避けられない。`build.chunkSizeWarningLimit: 6000` に設定して警告を抑制した。

## RESULTS, EFFECTS

### PROS

- `svelte-check`: **0 errors, 0 warnings** (5件解消)
- `vite build`: **警告ゼロ** (循環依存約20件 + チャンクサイズ1件を解消)
- テスト: 40ファイル 470テスト全パス、リグレッションなし
- バレルファイルの循環依存を根本解決したことで、Rollup のチャンク分割が正しく動作し、実行時の import 順序問題リスクも排除
- `onwarn` フィルタにより、自プロジェクトコードの循環依存は引き続き検知可能

### CONS, TRADEOFF

- `three-mesh-bvh` を direct dependency に追加したことで、`@threlte/extras` がバージョンを上げた場合にバージョン不整合が起きうる。ただし `^0.9.0` の範囲指定により minor バージョンの乖離は許容される
- `chunkSizeWarningLimit` の引き上げにより、他の意図しない巨大チャンクの検知閾値も上がる。ただし 6MB を超えるチャンクは three.js 以外では現実的に発生しない
- `svelte-ignore` コメントは警告を根本解決するものではないが、この設計パターン（SSR初期値のローカルコピー）ではベストプラクティスとされている
