# フィード読み取りパスの可観測性追加（フェーズ0）

## ADR's STATUS

Accepted

## CONTEXT

80/20負荷試験（3000VU、人気16フィードに集中）で、`http_req_waiting p95=4.70s` というボトルネックが判明した。`http_req_receiving p95=1.1ms` であるため、問題はネットワーク転送ではなくサーバー内部処理にある。しかし、サーバー内部の「DBクエリ」「モデル変換」「Proto/JSONシリアライズ」のどこが何ミリ秒を占めているかは計測されておらず、最適化の判断ができない状態だった。

alt-backend には OpenTelemetry のトレースプロバイダーとログプロバイダーが既に導入されていたが、フィード読み取りパスの個別メソッドには手動スパンが設定されておらず、区間計測は存在しなかった。また、PostgreSQL の `pg_stat_statements` も未有効化であり、クエリレベルの累積統計も取得できていなかった。

本ADRは、フェーズ1（共有 payload とユーザー状態の分離）に進む前の基盤として、**実装変更なし・計測のみ**で4.7秒の内訳を可視化するフェーズ0の設計判断を記録する。

## DECISION MAKING

### 計測ユーティリティ `FeedReadTimer` の新設

Clean Architecture の3層（Handler / Gateway / Driver）を横断して、1リクエスト内の各フェーズを計測する `FeedReadTimer` を `utils/perf` パッケージに新設した。

- `StartPhase(ctx, name) -> stopFunc` で区間計測と OTel child span を同時に生成
- `Log(ctx)` で構造化ログ1行（`usecase_ms` / `marshal_ms` / `total_ms` / `row_count` / `cache_ms` / `cache_hit`）と OTel メトリクスを同時出力
- `FEED_READ_PERF_ENABLED` 環境変数でON/OFF（デフォルト: ON）。OFF時は全てno-op

### 3層への OTel span 追加

| 層 | スパン名例 | 対象メソッド数 |
|---|---|---|
| Driver | `db.FetchUnreadFeedsListCursor` | 4 |
| Gateway | `gateway.FetchFeedsListCursor` | 4 |
| Handler | `perf.usecase` / `perf.marshal` | 5 |

同一 `context.Context` を伝播するため、スパンは自動的にネストする: `handler > gateway > driver`。

usecase 層は意図的にスキップした。理由は、usecase は limit バリデーションのみの薄いパススルーであり、スパンを追加してもノイズが増えるだけで有用な計測にならないため。

### Prometheus `/metrics` エンドポイント追加

OTel-to-Prometheus ブリッジ（`go.opentelemetry.io/otel/exporters/prometheus`）を使用し、alt-backend の REST サーバー（port 9000）に `/metrics` エンドポイントを追加した。`FeedReadTimer.Log()` 内で OTel Meter API を通じてヒストグラム・カウンターを記録し、Prometheus が scrape する。

### `pg_stat_statements` 有効化

`docker/postgres/postgresql.conf` に `shared_preload_libraries = 'pg_stat_statements'` を追加し、DB初期化スクリプトで extension を作成するようにした。また、`log_min_duration_statement` を 1000ms から 200ms に引き下げた。個別クエリは1秒未満でも複数で合算して4.7秒に達するケースを捕捉するため。

### 診断スクリプト

`scripts/explain_feed_queries.sql`（4つのホットクエリの EXPLAIN ANALYZE）と `scripts/pgbouncer_stats.sh`（PgBouncer 接続プール状況確認）を追加した。いずれもランタイムには組み込まず、手動実行用。

## RESULTS, EFFECTS

### PROS

- **4.7秒の内訳が可視化される**: `usecase_ms`（DB + モデル変換）と `marshal_ms`（Proto変換）が分離されるため、フェーズ1の最適化対象を定量的に判断できる
- **既存インフラとの統合**: 既存の OTel TracerProvider / LoggerProvider をそのまま活用。新たなトレーシングバックエンドの導入は不要
- **Grafana で一貫した可視化**: Prometheus メトリクスが Grafana ダッシュボードで参照可能。既存の golden-signals / SLO ダッシュボードと同じ基盤
- **フェーズ1への前方互換**: `cache_ms` / `cache_hit` フィールドを0/false で事前定義しており、Redis キャッシュ導入時にログフォーマット変更なしで移行可能
- **ON/OFF 切替可能**: 環境変数1つで計測を無効化できるため、パフォーマンスへの懸念がある場合に即座にオフにできる

### CONS, TRADEOFF

- **計測オーバーヘッド**: 各リクエストで OTel span 生成 + 構造化ログ出力 + Prometheus メトリクス記録が発生する。通常運用では無視できるレベルだが、極端な高負荷時には `FEED_READ_PERF_ENABLED=false` で無効化できる
- **REST ハンドラ未対応**: 今回は Connect-RPC ハンドラのみ計測対象とした。REST ハンドラ（`/v1/feeds/fetch/cursor` 等）は負荷試験で使用されていないため後回しとした。必要になれば同じパターンで追加可能
- **`pg_stat_statements` はコンテナ再起動が必要**: `shared_preload_libraries` の変更は PostgreSQL 再起動を伴う。本番環境への適用時はメンテナンスウィンドウが必要
