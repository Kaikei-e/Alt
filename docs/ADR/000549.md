# ADR-000549: sovereign.proto の RPC メッセージ型を buf STANDARD 命名規則に準拠させる

## Status

Accepted

## Date

2026-03-23

## Affected Services

- knowledge-sovereign (Go) — RPC ハンドラーの型参照を更新
- alt-backend (Go) — sovereign_client の型参照を更新
- alt-frontend-sv (TypeScript) — 生成コードの自動更新

## Context

Proto Contract Validation CI（`proto-contract.yaml`）が `buf lint` ステップで失敗した。`buf.yaml` で `STANDARD` ルールセットを採用しているが、`services/sovereign/v1/sovereign.proto` が約 50 件の lint 違反を抱えていた。

違反は以下の 3 カテゴリに分類される:

1. **共有型の再利用**: `ApplyMutationRequest` / `ApplyMutationResponse` を 3 つの RPC で共有、`Empty` を 15 以上の RPC で共有。buf STANDARD は「1 RPC = 1 Request + 1 Response」を要求する。
2. **レスポンス型の命名不一致**: `GetActiveProjectionVersion` の戻り型が `ProjectionVersionResponse`（正: `GetActiveProjectionVersionResponse`）など 6 件。
3. **リクエスト型の命名不一致**: `CreateLens` のリクエスト型が `CreateLensRpcRequest`（正: `CreateLensRequest`）など 4 件。

### 根本原因

sovereign proto は [[000537]] で 31 RPC を一括定義した際に、ボイラープレート削減を優先して汎用型（`ApplyMutationRequest`, `Empty`）を共有する設計を採った。この設計は buf STANDARD の `RPC_REQUEST_RESPONSE_UNIQUE` / `RPC_REQUEST_STANDARD_NAME` / `RPC_RESPONSE_STANDARD_NAME` ルールに抵触する。

CI ワークフローが後から追加されたため、このギャップが見過ごされていた。

### 教訓

- **Proto 定義と CI Lint は同時に導入する**: proto を書いた時点で `buf lint` を通す習慣がないと、後から大量の一括リネームが必要になる。
- **汎用メッセージ型の共有は避ける**: `Empty` や共通リクエスト型の再利用は一見 DRY だが、buf STANDARD に違反するだけでなく、将来的に RPC ごとにフィールドを拡張したい場合にも制約になる。各 RPC が自分専用の型を持つことで、独立した進化が可能になる。
- **CI を先に通してからコードを書く**: 生成コードに依存する手書きコードが増えた後のリネームは影響範囲が広い。31 RPC で 15 ファイルに波及した。

## Decision

sovereign.proto の全 RPC メッセージ型を buf STANDARD 命名規則に準拠させる。

具体的な変更:

| 変更カテゴリ | 変更数 | 例 |
|---|---|---|
| 共有リクエスト型の分割 | 3 組 | `ApplyMutationRequest` → `ApplyProjectionMutationRequest`, `ApplyRecallMutationRequest`, `ApplyCurationMutationRequest` |
| `Empty` の RPC 別型化 | 15 型 | `Empty` → `CreateProjectionVersionResponse`, `ActivateProjectionVersionResponse`, ... |
| レスポンス型のリネーム | 6 型 | `ProjectionVersionResponse` → `GetActiveProjectionVersionResponse` |
| リクエスト型のリネーム | 4 型 | `CreateLensRpcRequest` → `CreateLensRequest` |

各新型は元の型と同じフィールド構成を持つ。ワイヤ互換性は変わる（メッセージ名が proto の fully qualified name に含まれるため）が、knowledge-sovereign と alt-backend は同時デプロイするため問題ない。

### 検討した代替案

1. **buf.yaml で違反ルールを除外する** — `STANDARD` から `RPC_REQUEST_RESPONSE_UNIQUE` 等を `except` で除外すれば即座に CI は通るが、lint の意味が薄れる。proto の品質基準を下げるべきではないと判断。
2. **google.protobuf.Empty を使う** — buf STANDARD は `google.protobuf.Empty` の共有も許容しない。結局 RPC ごとの型が必要。

## Consequences

### Pros

- CI（Proto Contract Validation）がオールグリーンになる
- 各 RPC が専用の Request/Response 型を持つことで、将来のフィールド追加が RPC 単位で独立して行える
- buf STANDARD 完全準拠により、proto の品質が一貫して維持される

### Cons / Tradeoffs

- proto ファイルが約 100 行増加（空の Response 型 15 個分）
- 生成コードのサイズが若干増加（各空型に Go struct が生成される）
- rag-orchestrator / alt-butterfly-facade の生成コードが古いまま残っている（手書きコードから未使用のため実害なし、次回の `buf generate` で解消）
