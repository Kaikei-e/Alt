# ADR-000876: Knowledge Loop の OODA 状態遷移マトリクスを proto/alt/knowledge/loop/v1/loop_transition_policy.json に集約し FE / BE の conformance test で同期を強制する

## Status

Accepted

## Date

2026-05-02

## Affected Services

- **proto** — canonical policy JSON 追加
- **alt-backend (Go)** — Go conformance test 追加
- **alt-frontend-sv (TypeScript / Svelte)** — TS conformance test 追加

## Context

Knowledge Loop の OODA 状態遷移ルール (ADR-000831 §7) は以下の 4 箇所に **手書きで** 散在していた:

| 場所 | 形式 | 内容 |
|---|---|---|
| `alt-frontend-sv/src/lib/hooks/loop-transitions.ts` | TypeScript `Set<string>` | 6 edges allowlist |
| `alt-backend/app/usecase/knowledge_loop_usecase/classify_transition_event.go` | Go `switch` cases | 6 edges + 3 forbidden + 4 same-stage triggers |
| `proto/alt/knowledge/loop/v1/knowledge_loop.proto` | Enum + コメント | LoopStage / TransitionTrigger enum + comment "DEFER は same-stage を要求" |
| `e2e/hurl/knowledge-sovereign/19-knowledge-loop-projector-canonical-contract.hurl` + Playwright spec | アドホック JSON / TS | observe entries は revisit/ask/snooze のみ等 |

Phase 1 探索 (本 ADR 計画時) で確認した通り、**現時点では 4 箇所が偶然同期している** が、canonical source が無いため新しい edge を追加する際に同期作業が手作業になり、誰かが 1 箇所だけ更新する drift 事故が起きやすい。fb.md §7 はこの状況を:

> 状態機械は frontend / backend / projector / e2e で完全に同一の表を共有すべきです。今は分散定義です。

と表現していた。

## Decision

### 1. Canonical source を JSON に集約

`proto/alt/knowledge/loop/v1/loop_transition_policy.json`:

```json
{
  "version": 1,
  "allowed_edges": [
    { "from": "observe", "to": "orient",  "triggers": ["user_tap", "dwell", "keyboard", "programmatic"] },
    { "from": "observe", "to": "decide",  "triggers": ["user_tap", "keyboard", "programmatic"] },
    { "from": "orient",  "to": "observe", "triggers": ["user_tap", "programmatic"] },
    { "from": "orient",  "to": "decide",  "triggers": ["user_tap", "dwell", "keyboard", "programmatic"] },
    { "from": "decide",  "to": "act",     "triggers": ["user_tap", "keyboard", "programmatic"] },
    { "from": "act",     "to": "observe", "triggers": ["user_tap", "programmatic"] }
  ],
  "same_stage_triggers": ["defer", "recheck", "archive", "mark_reviewed"],
  "forbidden_edges": [
    { "from": "observe", "to": "act" },
    { "from": "act",     "to": "act" },
    { "from": "decide",  "to": "observe" }
  ]
}
```

JSON を選んだ理由:

- 言語非依存 / parser がランタイム標準。Go の `encoding/json` も TS の `JSON.parse` も追加依存ゼロ
- `proto/` 配下に置くことで「Knowledge Loop wire-protocol artifact」として位置づけが明確
- yaml は alt-frontend-sv 側に parser 依存が無く、追加を避けたい

### 2. Go conformance test (`transition_policy_test.go`)

```
$ cd alt-backend/app && go test ./usecase/knowledge_loop_usecase/...
ok  	alt/usecase/knowledge_loop_usecase	0.005s
```

検査内容:

- `allowed_edges[]` の各 (from, to) に対し `ClassifyTransitionEvent(from, to, "TRANSITION_TRIGGER_USER_TAP")` が non-empty event_type を返すこと
- `forbidden_edges[]` の各 (from, to) に対し `ErrInvalidArgument` を含む error を返すこと
- `same_stage_triggers[]` の各 trigger に対し `(orient, orient, trigger)` が `KnowledgeLoopDeferred` または `KnowledgeLoopReviewed` の正しい event_type を返すこと
- `version == 1` (将来 schema 変更時の op-bridge)

実装は `runtime.Caller(0)` から `filepath.Dir` を上方向に walking し、最大 12 階層上で `proto/alt/knowledge/loop/v1/loop_transition_policy.json` を見つける。CI / ローカルで実行 path が違っても test がロバストに動く。

### 3. TS conformance test (`loop-transitions.policy.test.ts`)

```
$ cd alt-frontend-sv && bunx vitest run --project=server src/lib/hooks/loop-transitions.policy.test.ts
 Test Files  1 passed (1)
      Tests  5 passed (5)
```

検査内容:

- `allowed_edges[]` の各 entry に対し `canTransition(from, to)` が `true`
- `forbidden_edges[]` の各 entry に対し `canTransition(from, to)` が `false`
- ALL_STAGES (4) の二項組合せ全てに対し JSON の予想と実装が一致 (extra allow / missing reject の両方向 detect)
- `same_stage_triggers[]` が `["defer", "recheck", "archive", "mark_reviewed"]` を含む
- `version === 1`

JSON の path 解決は `fileURLToPath(new URL(".", import.meta.url))` から最大 12 階層上を walking。Go と同じロバスト戦略。

### 4. 更新フロー

将来 edge を追加 / 削除する際の手順:

1. **policy JSON を編集** (canonical edit)
2. **両言語の conformance test を走らせる** → どこかが必ず RED になる:
   - 新 edge 追加 → 実装側で対応する case を追加するまで RED
   - edge 削除 → 実装側の case を削除するまで RED
3. **実装を更新して GREEN にする**
4. **proto-contract.yaml CI が json schema lint** (本 ADR スコープ外、follow-up)
5. **e2e (Hurl + Playwright) を併せて更新** (allowlist の意味的変化が UX 仕様にも波及する場合)

### 検討した代替案と却下理由

- **(却下) proto に `LoopTransitionPolicy` message を新設して `service KnowledgeLoopPolicy { rpc GetPolicy(...) }` として配信**: 実 RPC として配信する必要は無い。policy はビルド時 / テスト時の static artifact で十分。RPC を増やすと codegen / Pact CDC / 認可の議論まで広がり scope が膨らむ。
- **(却下) Yaml を採る**: TS 側に yaml parser 依存を入れる必要があり、bun / vitest の依存が増える。JSON は言語標準で済む。
- **(却下) Go の switch case をそのまま canonical とし、TS と Hurl は CI で source-grep して assert**: 実装が canonical だと「実装の見落とし = canonical の見落とし」となり安全網が働かない。data を canonical にして実装が data に従うほうが扉が一方向。
- **(却下) policy を `docs/plan/` 配下に置く**: docs は人間向けで machine-readable な contract には適さない。proto/ 配下なら CI の path-gate も拾いやすい (proto ファイル変更で proto-contract gate が動くなど)。
- **(却下) 実装側で switch case を消し JSON を runtime に load**: ホットパスでファイル読み込みを増やすのは性能上不利。data canonical + 実装は手書き、conformance test で同期を強制、というアプローチがコスト最小。

## Consequences

### Pros

- **fb.md §7 の "状態機械の分散定義" 問題を構造的に解消**。今後 edge の追加 / 削除は JSON 編集 → conformance test の RED → 実装の GREEN → e2e の更新、という単一手順に統一される。
- **conformance test が CI に組み込まれる速さで drift を検知**。Go test は `go test ./usecase/knowledge_loop_usecase/...` のたびに走り、TS test は `bunx vitest run --project=server` のたびに走る。両方とも既存の CI gate に乗っており追加コストゼロ。
- **JSON の `version` フィールドで将来の schema 変更を表現可能**。例えば triggers のセマンティクスが変わるときは `version: 2` に bump し、conformance test 側で version を assert することで意図しない混在を防げる。
- **policy が proto/ 配下にあることで** `proto-contract.yaml` workflow / `buf` 関連 lint で path-gate が拾える。CI 側で json schema lint を `.github/workflows/proto-contract.yaml` に追加する余地が残されている (本 ADR では未実装、follow-up)。
- **実装の手書き性は維持**。Go / TS の各実装が switch case / Set のままなので runtime overhead はゼロ、stack trace / debugger experience は変わらない。codegen を強制しない判断のため大きな build infra 変更は無い。

### Cons / Tradeoffs

- **policy JSON は `proto/` 配下にあるが proto IDL ではない**。`buf lint` / `buf breaking` 対象外で、proto の他の artifact と扱いがやや異なる。配置議論 (`proto/` vs `policy/` vs `schema/`) は将来的に再考の余地あり。
- **Hurl / Playwright e2e の policy 同期は手作業のまま**。conformance test は単体テスト層 (FE / BE) に閉じており、e2e の `19-knowledge-loop-projector-canonical-contract.hurl` の `forbidden` assertions と `canonical-contract.spec.ts` の transition pinning は依然手書き。e2e 同期化は別 ADR (or follow-up) で扱う。
- **policy の `triggers` array は advisory であって enforce されない** (FE は `(from, to)` だけ見て `canTransition` を返す)。BE classifier は trigger を実際に使うので triggers 列が drift しても FE 側では検知不能。conformance test を BE 側で extend して triggers の正当性も検査するのが follow-up 候補。
- **conformance test の path 解決が runtime.Caller / fileURLToPath ベース**。CI 環境で workdir が想定外に変わると 12 階層 walking でも見つからない可能性がある。実用上は問題無い (alt-backend / alt-frontend-sv は repo root から最大 5 階層) が、追跡しやすい failure mode を残すために `t.Fatalf` メッセージで explicit に "could not locate ..." と出している。
- **policy version の bump 運用が runbook に書かれていない**。本 ADR では version=1 で固定したが、将来 v2 に上げるときの「新版 policy への切替手順」は今のところ implicit。runbook follow-up として「policy version bump」セクション追加が必要。
