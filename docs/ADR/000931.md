# ADR-000931: Hurl E2E gate を pre-merge と pre-deploy の両層で常時全走化する

## Status

Accepted

## Date

2026-05-28

## Affected Services

- 公開 repo CI (`.github/workflows/e2e-hurl.yml`) — pre-merge gate
- 本番 deploy pipeline (公開外) — pre-deploy gate
- `.claude/skills/tdd-workflow` の Phase 0 ガイドライン

## Context

### 経緯

2026-05-28 に PR-1 (ブランチ `ci/hurl-always-run`) で公開 repo の `e2e-hurl.yml` から `on.pull_request.paths` / `on.push.paths` を撤廃し、無関係 PR でも全 Hurl suite が走る状態にした。撤廃の根拠は「**path-level の selective-skip は無関係 PR の skip を提供する代わりに、共有 wiring (compose.staging.yaml / fixtures / upstream Go package / proto 等) の regression が touched service path に該当せず緑で main に着地する穴を作る**」という観測。同 PR で 9 サービス分の Hurl job 全てが毎 PR / 毎 push:main で走るようになった。

ここで残る疑問が「**同じ理念は pre-deploy gate にも適用すべきか**」だった。Alt の本番デプロイは ADR-000763 / ADR-000764 / ADR-000765 / ADR-000766 で確立した二層構造を取る:

```
公開 repo (Alt)                   本番 deploy pipeline (公開外)
─────────────────                 ──────────────────────────────
[PR] ─ e2e-hurl.yml ──┐
                      │
                      ▼
                   merge ── push:main ──→ (dispatch) ─→ build / gate / E2E / deploy
```

### 現状の selective-skip

#### 公開 repo (PR-1 で撤廃済み)
- `e2e-hurl.yml` の `on.pull_request.paths` / `on.push.paths` → 全削除
- 各 job に `timeout-minutes` を明示 (recap-worker は cold-start dominant のため余裕)
- `workflow_dispatch` を追加して fail 時の手動再走を許可

#### 本番 deploy pipeline (本 ADR で撤廃対象)
公開 repo からの dispatch payload が運ぶ「touched path リスト」を、deploy 側が持つ「service ↔ source/e2e/fixtures/compose path」対応表と prefix-intersect し、**マッチした service だけを deploy / gate / E2E matrix に乗せる**実装が dispatch 入口に存在する。

結果として、touched path に該当しない service の **Hurl suite は pre-deploy gate でも走らない**。これは PR-1 が公開 repo で潰した「無関係 PR の skip が共有 wiring 経由の regression を隠す」と**同型の盲点**を pre-deploy 層で再生産している。具体例 (架空):

> alt-backend の Connect-RPC proto 定義に互換破壊変更を加える PR が、alt-backend の path 配下しか触らずに main に着地する。dispatch payload の touched path は `alt-backend/` のみ。deploy 側は alt-backend だけを matrix に乗せる。pre-deploy の Hurl gate で alt-backend は走るが、proto を import している他 pacticipant (pre-processor / search-indexer / mq-hub …) の Hurl suite は matrix に乗らず、互換破壊が production に landed して初めて顕在化する。

### 既存の対症療法

- 公開 repo 側 `docker-build.yaml` は SERVICES が固定列挙で、touched path に関わらず全 service の image を rebuild する (PR-1 と同じ理念)。pre-deploy gate に届く前の一次防御。
- deploy 側に operator 用の手動 narrowing / 全走切替え経路が用意されており、必要時に全 service 再走を強制できる。ただし**自動 dispatch 経路では使われない**ため、平常時の盲点を埋めない。

これらは「操作者が思い出したら全走できる」状態であり、構造的な「常時全走」ではない。PR-1 の理念に沿うなら、**dispatch 自動経路でも全 pacticipant の Hurl gate が必ず走る**よう移行する必要がある。

## Decision

### 採用方針

**pre-merge と pre-deploy の両層で Hurl E2E gate を「常時、全 pacticipant について走らせる」を canonical とする。**

具体:

1. **公開 repo (実装済み)**: `e2e-hurl.yml` の path フィルタは恒久的に持たない。新規 Go service が Hurl suite を持つようになったら、対応する job を workflow に append し、それも path 条件なしで走らせる。
2. **本番 deploy pipeline (Phase B-1 着手済み)**: 自動 dispatch 経路でも E2E matrix が **全 pacticipant** をカバーするよう改修する。operator 用の手動 narrowing 経路は残す (緊急 re-roll の正当な手段)。
3. **matrix 別の方針**: 全 matrix を一律全走化するのではなく **「ビルド成果物を生む matrix は touched のみ、テスト/ゲートする matrix は全 pacticipant」** で分離する。
   - **build / deploy / 契約発行**: 触ったサービスのみ。untouched 側に新 SHA タグを作って push し直すのは regression なので意図的に拡げない
   - **E2E**: 全 pacticipant。touched leg は dispatched SHA 由来の image で deploy 前ゲート、untouched leg は production tag の image で cross-service regression テスト
   - **CDC gate (can-i-deploy)**: 当面 touched-only。全 pacticipant 化は broker 側の状態整合 (latest 解決) を扱う別仕事になるため Phase B-2 として切り出し
4. **regression leg の image 解決**: untouched pacticipant は build matrix が新 SHA tag を作らないため、E2E leg は production tag を pull する。本番が今走らせている image そのものに、dispatch SHA で更新された touched サービスを並べた slice で Hurl を走らせる構図になり、cross-service regression が catch される
5. **regression leg の gate-artifact 整合**: untouched pacticipant は CDC gate に乗らないので gate-pass 成果物が物理的に存在しない。これによる E2E 側の hard-fail (deploy 系の silent-skip 防止用) を誤発火させないため、deploy pipeline 側で untouched leg に対してだけ「regression 由来の stub」を合成する。stub と本物の gate-pass は別文字列で区別可能にしておく
6. **emergency skip 経路**: 既存の operator skip 経路を残す。本番障害復旧時に Hurl gate を一時的に bypass する正規ルートとして機能する
7. **tdd-workflow skill**: Phase 0 E2E のガイドラインに「自分が touch した service の Hurl 以外も merge gate / deploy gate で走る前提で書く」旨を 1 行追記し、`--variable` でパラメタライズせず hardcode したフィクスチャが意図せず他 service の gate を壊さないよう注意喚起する

### 採用しない代替案

| 案 | 不採用理由 |
|---|---|
| pre-merge は全走、pre-deploy は selective | 「main に着地した変更が deploy 経路で初めて走る Hurl」が出る。**deploy 直前の selective skip は post-merge regression の温床** (PR-1 と同型の盲点)。 |
| pre-deploy は selective、空 payload を全走扱いにするだけ | dispatcher 側の bug で空 payload を返した過去事例の再発を防げる代わり、**dispatcher が正常に動作している限り selective skip は残る**。半端な解。 |
| matrix に並列上限を設けて wall-time を抑える | self-hosted runner pool の容量問題は別観点で、本 ADR の理念とは独立。並列化制御は別レイヤで扱う。 |
| すべてを手動 dispatch に倒す | 「常時全走」の structural guarantee を operator の習慣に依存させてしまう。**思い出したら走る**は **走らない** と同義。 |
| 全 matrix (build/deploy 含む) を一律全走化 | untouched サービスを新 SHA tag で rebuild して push し直すのは regression。stale tag が production に着地するリスクすらある。 |

## Consequences

### Positive
- **両層で同じ regression net**: pre-merge で漏れた共有 wiring 由来の breakage が pre-deploy で確実に catch される。PR-1 が公開 repo で塞いだ穴と同型の穴を pre-deploy 層でも塞ぐ。
- **deploy 直前の Hurl matrix が固定**: dispatch payload の touched 内容に依存しないため、payload が空でも壊れた service だけが乗っても、gate の網羅性は変わらない。
- **理念の文書化**: ADR-000763 → ADR-000764/765/766 で確立した「Hurl E2E gate を canonical な merge / deploy 前提とする」方針が、selective-skip の構造的な存続によって徐々に意味を失っていた状況を、明示的な反転として記録できる。

### Negative
- **deploy pipeline の wall-time が増える**: 1 service の変更でも全 pacticipant 分の Hurl matrix が走る。matrix 並列で wall-time は **slowest single service** で頭打ちになるため、最悪 cold-start サービスの実行時間 (cargo-chef + LLM warmup を含む service の cold path) に揃う。
- **builder runner 占有時間の増加**: matrix の同時占有数が増える。複数 SHA が連続 dispatch された場合の runner pool 競合は workflow-level concurrency と SHA-scoped grouping で吸収するが、長期的には runner pool の sizing 見直しが必要になる可能性。
- **emergency 時の operator 負荷**: 全走 wall-time に耐えられない緊急復旧では skip 経路を明示的に使う必要がある。判断と説明責任を operator に明示的に押し付ける形になるが、これは **deliberate** な設計。

### Neutral
- 公開 repo の docker-build.yaml の selective rebuild は維持する。**image build は selective、test gate は always-run** という分離。
- co-release lockstep 系のロジックは matrix が拡張されても破綻しない (むしろ全 pacticipant が常に matrix に乗ることで co-pair 解決が安定する)。
- `compose.staging.yaml` を編集すると全 service の pre-deploy E2E が走る (旧 change-set filter は staging compose を除外していたが、本 ADR では「staging compose を触ったら全走」を許容する)。staging-only file の edit は本番 image rebuild を引き起こさないため、コスト最大は wall-time のみ。

## Implementation

### Phase A — 公開 repo (実装済み、PR-1)
- `.github/workflows/e2e-hurl.yml` から `on.pull_request.paths` / `on.push.paths` を全削除
- `on: {pull_request: {}, push: {branches: [main]}, workflow_dispatch: {}}` に縮約
- 各 job に `timeout-minutes` を明示
- 確認手段: docs-only commit を push し、9 job 全てが起動 + green になることを観察

### Phase B-1 — E2E matrix expansion (着手済み、deploy 側 PR で実装)
Phase A の理念を deploy pipeline に貫くための変更。**実装は本番 deploy pipeline 側に閉じる**ため詳細はそちらで管理する。本 ADR で固定するのは「**何を達成するか**」のみ:

- E2E matrix が touched-only から **全 pacticipant** に拡張される (operator 用の narrowing 経路は維持)
- touched leg は dispatched SHA 由来の image で従来通り deploy ゲートとして機能する
- untouched leg は production tag の image を pull する regression テストとして機能する
- 既存の gate-pass 成果物 hard-fail (deploy 系 silent-skip 防止) は、untouched leg では誤発火しないよう deploy pipeline 側で stub を合成する。stub と本物の gate-pass は区別可能にしておく
- emergency skip 経路は既存のまま維持
- 確認手段: 1 service のみ touch する PR を main に merge し、dispatch 経由で起動した deploy pipeline が全 pacticipant 分の E2E matrix を走らせ、touched leg と untouched leg がそれぞれ正しい image tag で green になることを観察

### Phase B-2 — CDC gate matrix expansion (follow-up、本 ADR の scope 外)
- CDC gate / 契約発行 matrices は依然として touched-only
- 「全 pacticipant の can-i-deploy 結果を毎 release で確認する」案は untouched pacticipants を broker 側の `--latest` 解決にフォールバックさせる形で実装可能だが、gate スクリプトの version 切替えと broker の deployed-tag 状態に依存する整合性検証を要するため、別 ADR + 別 work item とする
- Phase B-1 で「E2E 層で全 pacticipant regression net が稼働」を達成しているため、CDC gate 拡張は緊急度が下がる

### Phase C — skill / runbook 反映
- `.claude/skills/tdd-workflow/SKILL.md`: Phase 0 E2E の節に「pre-merge / pre-deploy の両層で全 pacticipant の Hurl が走る前提で書く」を 1 行追記
- 既存 runbook (deploy / pact-broker-ops 等) で selective-skip 前提の手順があれば見直す (現時点では発見されていない)

### Phase 順序の根拠
- Phase A は既に公開 repo に着地 (PR-1 / ブランチ `ci/hurl-always-run`)
- Phase B-1 は本番 deploy pipeline 側の変更で、Phase A 単独でも regression net は半分稼働する。Phase A 着地後に観測してから本格運用に進む
- Phase C は Phase B-1 完了後の追記 (skill ファイル更新は work in progress を mislead しないため最後)

## Trade-off matrix

| 観点 | 採用案 (両層 always-run) | 既存 (selective-skip) |
|---|---|---|
| 共有 wiring 起因の regression catch | ◎ pre-merge と pre-deploy の二重防御 | △ pre-merge は PR-1 で改善、pre-deploy は穴が残る |
| 平常時の wall-time | △ 全 service 分 (slowest が支配) | ○ touched service 分のみ |
| dispatch payload の品質依存 | ✗ なし (構造的に独立) | ◎ payload が正しい前提 |
| 緊急復旧時の操作性 | ○ skip 経路で明示的に bypass | ○ 自然と狭まる |
| Hurl suite 追加コスト | ○ 公開 repo に job 1 つ追加するだけ | △ workflow 追加 + deploy 側 path 整合 |

## Sensitive content policy

本 ADR は公開 repo に置かれるため、本番 deploy pipeline 側の **具体的な job 名 / 出力名 / 環境変数名 / スクリプト名 / Ansible playbook タスク名 / runner 名 / host 名 / IP / tailnet アドレス / PAT 名 / 内部識別子** は意図的に記載していない。実装詳細はその repo 側のコメント / 文書で扱う。本 ADR が固定するのは「**何を実現するか**」と「**どの ADR/PR が起点となったか**」のみ。
