# ADR-000750: 手動 Docker Compose デプロイ戦略の不動作を Rust 版 pact-broker-cli 導入と pact publish 経路整合化で稼働可能化する

## Status

Accepted

## Date

2026-04-16

## Affected Services

- `scripts/deploy.sh` / `scripts/_deploy_lib.sh` / `scripts/pre-deploy-verify.sh` — broker CLI デフォルトを差し替え、`record_deployments` を fail-fast 化
- `scripts/pact-check.sh` — broker Branches API の URL 順序バグ修正と pact publish の version 文字列を `pre-deploy-verify` / `deploy` と一致させる
- `tests/scripts/test_pre_deploy_verify.sh` / `tests/scripts/test_deploy.sh` — スタブ名 rename と CLI preflight テスト追加
- `docs/runbooks/deploy.md` / `docs/runbooks/pact-broker-ops.md` — CLI インストール手順とコマンド名を更新

コード生成物は全てシェルスクリプト + runbook のため、コンテナイメージの再ビルドは不要。実演に用いた `alt-butterfly-facade` は `scripts/deploy.sh --only ... --skip-verify` 経由で再作成され healthy。

## Context

[[000740]] で GitHub Actions の `release-gate.yaml` / `deploy.yaml` を退役させ、単一ホスト上の `scripts/deploy.sh` + `scripts/pre-deploy-verify.sh` + `scripts/_deploy_lib.sh` に手動デプロイを集約した。しかし 2026-04-15 の切替以降 [[000741]] 〜 [[000749]] の 9 本の ADR が merge されても、対応するコンテナ群は `docker compose up --build -d` で直接差し替えられており、**設計された rolling recreate / Pact gate / `record-deployment` / 自動ロールバック / 最前版 SHA 保存 (`.deploy-prev`) は一度も通っていなかった**。

診断で以下の複合的な不具合を特定:

1. **broker CLI 未導入 (主因)** — `scripts/pre-deploy-verify.sh:33` / `scripts/_deploy_lib.sh:109` が `pact-broker` コマンドを直接呼ぶ設計だが、ホストに Ruby も `pact_broker-client` gem も入っておらず、`can-i-deploy` 14 件全滅で `exit 4`。`scripts/deploy.sh:72-76` が gate を abort し rolling recreate に到達しない。`tests/scripts/*.sh` は `PACT_BROKER_BIN` を stub に差し替えて green を保っていたため、CI 風テストがホスト依存ギャップを隠蔽していた。
2. **pact-broker CLI が Rust に rewrite 済** — 公式 `pact-broker-cli` v0.6.3 (2026-03-12, `github.com/pact-foundation/pact-broker-cli`) が Ruby gem の後継として公表されており、Alt の single-host / no-ruby 方針 (`feedback_single_machine_architecture`) と一致する。
3. **`scripts/pact-check.sh:157` の Branches API URL 順序バグ** — `PUT /pacticipants/{name}/versions/{version}/branches/{branch}` と書かれており、Pact Broker が期待する `PUT /pacticipants/{name}/branches/{branch}/versions/{version}` と順序が逆。broker が 404 を返して pact publish フェーズ全体が fail-fast し、gate が機械的に通らない二次要因。
4. **`pact-check.sh` と `pre-deploy-verify.sh` / `deploy.sh` の version 文字列不一致** — 前者は `PACT_PROVIDER_VERSION="local-$(git rev-parse --short HEAD)"` で publish、後者 2 本は bare `$(git rev-parse --short HEAD)` で `can-i-deploy` / `record-deployment`。broker 上で突き合わせる pacticipant version が別物となり、`can-i-deploy` が永久に "unknown"。
5. **`record_deployments` が失敗を `|| echo` で握り潰す** — 14 件全失敗でも戻り値 0。broker matrix と実運用が乖離しても気付かない。
6. **runbook の前提表に CLI 導入手順が無い** — 新規オペレータが同じ罠を踏む。

## Decision

**Rust 版 `pact-broker-cli` をホストに導入し、broker への全操作 (`create-environment` / `can-i-deploy` / `record-deployment`) の CLI 層を Rust 単一バイナリに統一する**。さらに pact publish 経路 (`pact-check.sh`) のバグを 2 点修正し、`record_deployments` を fail-fast 化する。

具体的な変更:

### E1: Rust 版 `pact-broker-cli` の導入と PATH 反映

- ホストに `curl -fsSL https://raw.githubusercontent.com/pact-foundation/pact-broker-cli/main/install.sh | sh` で v0.6.3 を `~/.local/bin/pact-broker-cli` に配置。PATH 上で `pact-broker-cli --version` が解決する。
- `scripts/pre-deploy-verify.sh` / `scripts/_deploy_lib.sh` のデフォルト `PACT_BROKER_BIN` を `pact-broker` → `pact-broker-cli` に切替。`PACT_BROKER_BIN` env での override は維持し、Docker 版 `pactfoundation/pact-broker-cli:latest-debian` への乗り換え余地を残す。
- `scripts/pre-deploy-verify.sh` の先頭に CLI preflight (`command -v "$PACT_BROKER_BIN" || exit 5`) を追加。CLI 不在時に不透明な `can-i-deploy × 14 FAIL` ではなくインストール手順つきで早期に止まる。

### E2: Branches API URL 順序と version 文字列の整合化

- `scripts/pact-check.sh:157` の `PUT` URL を `/pacticipants/{name}/versions/{version}/branches/{branch}` から `/pacticipants/{name}/branches/{branch}/versions/{version}` に修正。
- `scripts/pact-check.sh:91` の `PACT_PROVIDER_VERSION` から `local-` プレフィックスを除去し、`pre-deploy-verify` / `deploy.sh` と同じ `$(git rev-parse --short HEAD)` を使用。

### E3: `record_deployments` の fail-fast 化

- `scripts/_deploy_lib.sh::record_deployments` を失敗カウント集計に変更し、1 件でも失敗すれば非 0 で return。呼び出し側 `scripts/deploy.sh` は戻り値を見て最終 exit を 12 にし、「broker matrix is out of sync — re-run record-deployment for the failed services」を警告。

### E4: テスト整備

- `tests/scripts/test_pre_deploy_verify.sh` / `tests/scripts/test_deploy.sh` 内の broker CLI スタブを `pact-broker` → `pact-broker-cli` に rename。
- 新規ケース `tc_fails_when_broker_cli_missing` を追加。`PACT_BROKER_BIN=$SANDBOX/nonexistent` の状態で preflight が exit 非 0 + "not found on PATH" を出力し、heartbeat curl が一度も呼ばれないことを検証。

### E5: runbook 改訂

- `docs/runbooks/deploy.md` の TL;DR と前提表に CLI インストール手順 (`curl ... install.sh | sh`) とバージョン確認コマンド (`pact-broker-cli --version`) を追加。
- `docs/runbooks/pact-broker-ops.md` のコマンドサンプルを `pact-broker` → `pact-broker-cli` に書き換え。

### 検討した代替案

- **Ruby `pact_broker-client` gem を入れる** — Alt の single-host は Ruby ランタイムを持たず、Rust 版が公式後継として提供されているため不採用。
- **`pactfoundation/pact-cli` Docker image を薄いラッパで叩く** — `docker run --rm --network host pactfoundation/pact-cli pact-broker ...` 形式。14 pacticipant × 2 コマンドで 28 回のコンテナ起動が発生し、単純さと起動遅延の両面で単一バイナリより劣る。ただし `PACT_BROKER_BIN` 経由で後から切替可能とし、Ruby CLI image 専有環境への逃げ道として残した。
- **`pact-check.sh` の publish version に `local-` を残し、`pre-deploy-verify` 側に寄せる** — 「production 環境に local- プレフィックス付き version を記録する」意味論違反になるため不採用。

### 実演による検証結果

修正後、実 broker に対して以下を確認:

- `scripts/deploy.sh --dry-run production` が 15 サービスのレイヤ順 (`step-ca → kratos/auth-hub → alt-backend/search-indexer/mq-hub/pre-processor → news-creator/... → alt-butterfly-facade/alt-frontend-sv/nginx`) を設計通り列挙。
- `scripts/deploy.sh --only alt-butterfly-facade --skip-verify production` が `.deploy-prev` を生成し、対象コンテナを `--no-deps --force-recreate` で再作成、healthcheck healthy 到達、global smoke 通過まで完走。
- `record-deployment` 14 件を Rust CLI 経由で broker に投入、production 環境の `deployed-versions/currently-deployed` に 14 レコード追加。
- `bash tests/scripts/run.sh` が 12/12 pass。CLI preflight テストも含む。

## Consequences

### Pros

- ADR-000740 で設計された手動デプロイ戦略が**実際に実行可能**になった。単一ホスト + 手動実行という方針を維持したまま、Pact gate → layered rolling recreate → global smoke → `record-deployment` → 自動ロールバックの全機能が動作する。
- CLI preflight が導入手順にリンクする形で fail-fast するため、新規オペレータの踏み込み loss time が短縮される。`scripts/deploy.sh` の exit code 規約 (`10` = gate fail, `11` = recreate/smoke fail, `12` = record-deployment fail, `5` = CLI missing) が明確化。
- `record_deployments` の fail-fast 化で「silent success」を構造的に排除。broker matrix と実運用の乖離が起きた瞬間に CI ではなくオペレータ画面で警告される。
- CLI スタブ rename と preflight テストで、将来 `PACT_BROKER_BIN` がホスト側で再び missing になっても regression を test suite が捕まえる。

### Cons / Tradeoffs

- **CLI 単一バイナリはホスト手動インストール**。compose-first 方針と整合しない install path だが、broker 側は compose 管理のままで、CLI は 1 ユーザー・1 ホストにしか必要ないため運用コストは低い。Docker 版 override ルートを残しているため将来 containerize も可能。
- **can-i-deploy の本番 baseline は別途 bootstrap が必要**。初回 production deploy 時、broker には誰も "deployed" としてマークされていないため can-i-deploy が全 unknown。今回の実演で 14 pacticipant を `pact-broker-cli record-deployment` で手動 seed 済。次回以降は gate が自然に通る前提になる。
- **provider verification カバレッジ不足が顕在化**。`scripts/pact-check.sh` の provider test は news-creator / recap-subworker / tag-generator / tts-speaker / alt-backend / search-indexer / recap-worker の 7 本のみで、pre-processor / mq-hub / rag-orchestrator / acolyte-orchestrator / recap-evaluator / alt-butterfly-facade / auth-hub の provider 側が未検証。該当 consumer の can-i-deploy は永続的に unverified pair を抱える。本 ADR のスコープ外だが、次の ADR で provider 検証網羅を扱う前提。
- **`_deploy_lib.sh::DEFAULT_LAYERS` が step-ca を含む**。[[000747]] / [[000748]] で step-ca sidecar は pki-agent に置換されたが、レイヤ定義は旧設計のまま。現状は step-ca コンテナ自体が動いているため recreate は通るが、将来 step-ca 退役時にレイヤ刷新が必要。
