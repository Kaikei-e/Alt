# ADR-000943: リモート GPU を強化層と定義し機能別劣化契約で単一マシン成立を保証する

## Status

Accepted

## Date

2026-07-13

## Affected Services

- news-creator (Python) — 要約 BE オフロード (DISTRIBUTED_BE)
- rag-orchestrator (Go) — embedder / rerank クライアント
- recap-subworker (Python) — リモート埋め込み (ollama-remote)
- acolyte-orchestrator (Python) — AIX 上の大型 LLM
- alt-butterfly-facade (Go) / news-creator — TTS 消費側

## Context

Alt の GPU 推論ワークロードは、当初 Primary Host (koko-b, RTX 4060) 完結だったものが、
Tailscale 経由のリモートノードへ段階的に拡散した。2026-07-13 時点の実測マップ:

- **AIX (100.126.96.80, ROCm iGPU)**: RAG embedder・recap 埋め込み・Acolyte LLM・TTS・要約 BE の **5 依存が集中**
- **Mac (100.99.107.87, MPS)**: rerank・要約 BE・休眠中の Augur フック
- **RTX 2070S (100.74.178.93)**: 要約 BE のみ

AIX 停止で Acolyte・TTS・RAG/recap 埋め込みが即時不成立になる。切替は `.env` の
`*_EXTERNAL_HOST` 手動書き換え＋コンテナ再作成のみで、契約なきリモート依存
(Hard Remote Dependency) が実質 5 件存在した。また 2026-07-06 モノレポ全体レビュー
([[000928]] と同型のサイレントフォールバック) が示すとおり、「リモート不在時の挙動」が
無言 no-op で表現される故障モードが全域で再発している。

用語は `contexts/inference-topology.md` (Primary Host / Enhancement Tier /
Degradation Contract / Loud degradation) に正準化した。

## Decision

**リモート GPU ノードを Enhancement Tier（あれば強化・なくても成立）と定義し、
リモート依存を持つ全機能に劣化契約を 1 つ割り当てることを採用する。**

1. **原則**: Hard Remote Dependency は禁止。単一マシン完結は指向であってマストではないが、
   「リモートがないと成立しない」構造は許可しない。
2. **機能別劣化契約**:

   | 機能 | 契約 | 内容 |
   |---|---|---|
   | 要約 BE オフロード | Degraded | ローカル 4060 のみで継続（現行 DISTRIBUTED_BE の準拠を明文化） |
   | RAG embedder | Degraded | ローカル同一モデル (embeddinggemma) へ切替。**別モデル切替は禁止**（ベクトル空間破壊） |
   | RAG rerank | Degraded | rerank 段をスキップしベクトル順で返す |
   | recap 埋め込み | Deferred | キュー保留、リモート復帰後に処理 |
   | Acolyte LLM | Deferred | checkpoint-resume で保留・復帰後再開、「生成保留中」を可視化 |
   | TTS | Feature-off | 「利用不可」を可視表示、他機能は無傷 |

3. **発動機構はランタイム自動切替**: 各消費サービスの client/gateway 層に実装する
   （集中プロキシは新設しない — Deferred / rerank スキップはアプリ層でしか表現できず、
   transport 層で透過フェイルオーバー可能なのは embedder 1 件のみのため、常駐プロキシの
   footprint 追加に見合わない）。共有するのはコードでなく規約:
   状態機械 `remote_ok / degraded / deferred`・状態遷移の loud ログ・統一メトリクス名。
4. **契約の継続検証は CI 障害注入テスト**: 各消費サービスに「リモート = connection refused、
   ローカル = スタブ」の統合テストを置き、契約どおりの遷移を assert する。
   既存の `compose/news-creator-ollama-stub` ＋ `compose.staging.yaml` E2E (GitHub Actions,
   ubuntu-latest) パターンの延長で、GPU 実機・self-hosted runner は不要。
5. **スコープ外**: deploy パイプライン (alt-builder)・オフサイト restic バックアップ・
   SSH モデル配布スクリプトのクロスマシン依存は本 ADR の対象外（バックアップは定義上
   リモートが正、deploy は平時運転の critical path ではない）。

検討した代替案:

- **完全単一マシン化**: GPU 経済性（8GB VRAM では大型モデル・同時多役が不成立）を無視するため棄却。
- **全機能一律 Deferred / 一律 Degraded**: 対話系 UX の破壊・ローカル GPU 圧迫のため棄却。機能別契約を採用。
- **手動 env 切替の維持**: 契約が「紙の上の約束」になり運用実態は Hard Remote Dependency のままのため棄却。
- **集中インフェレンスルータ新設**: 上記 3 のとおり棄却。

## Consequences

### Pros

- AIX / Mac の停止が「機能の即死」から「契約された劣化状態」に変わり、深夜のオペ介入が不要になる。
- 劣化状態が loud ログ・メトリクスで観測可能になり、[[000928]] 型の「配線忘れと意図的無効の区別不能」を
  トポロジ次元で塞ぐ。
- CI 障害注入により契約の回帰が push 前に止まる。

### Cons / Tradeoffs

- 5 消費サービス × 3 言語 (Go/Python/Rust) にフェイルオーバー実装が分散する（規約統一でカバー、共有ライブラリは作らない）。
- Degraded 発動時はローカル GPU/RAM の resident footprint が増える。ローカル代替の常駐/on-demand 判断は
  Resource Efficiency 境界の measurement-first に従う。
- 自動切替のフラッピング対策（ヒステリシス）が各実装に必要。
