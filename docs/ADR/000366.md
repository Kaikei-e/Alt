# feed-read 3000VU 再現試験で実行条件不足を否定し、backend 内部飽和を主因として扱う

## ADR's STATUS

Accepted

## CONTEXT

feed-read 系の 3000VU 負荷試験では、`GetUnreadFeeds` / `GetAllFeeds` / `FetchArticlesCursor` に対して `EOF`、`connection reset by peer`、`connect: connection refused` が発生し、`alt-backend` が再起動していた。

初期の仮説としては、以下のような「試験条件不足」が疑われていた。

1. `auth-hub` の replica 数不足
2. `VALIDATE_RATE_LIMIT` 未反映
3. `ulimit(nofile)` 未設定
4. `FEED_FETCH_CONCURRENCY` / `CB_MAX_CONCURRENT` 未設定

一方で、`reason.md` では PostgreSQL 接続飽和、Go HTTP サーバーの接続処理、タイムアウト、fd 制限など複数レイヤーの可能性が指摘されており、まず「外側の条件不足」を潰した上で再現性を確認する必要があった。

そのため、`alt-perf/scripts/run-feed-read-load-test.sh` を改修し、試験条件を明示した再試験を実施した。

## DECISION MAKING

### 1. feed-read 負荷試験スクリプトに高負荷条件を明示する

`run-feed-read-load-test.sh` に以下を追加した。

| 設定 | 値 |
|---|---|
| `VALIDATE_RATE_LIMIT` | `10000` |
| `AUTH_HUB_REPLICAS` | `3` |
| `FEED_FETCH_CONCURRENCY` | `500` |
| `CB_MAX_CONCURRENT` | `500` |
| `CB_FAILURE_THRESHOLD` | `1000` |
| `CB_RESET_TIMEOUT` | `5s` |
| `nofile` | `65536` (`alt-backend` / `db` / `pgbouncer` / `k6`) |

また、overlay 生成時にこれらを Compose へ明示注入し、中断時に通常構成へ戻す `trap cleanup` を追加した。

### 2. 再現試験では「実行条件不足を否定できるか」を第一目的とする

今回の試験は成功レポート作成ではなく、根本原因の切り分けを目的とした。そのため評価軸を以下に置いた。

1. `auth-hub` 3 replica が実際に起動しているか
2. `nofile=65536` が実コンテナへ反映されているか
3. 初期ランプアップが改善するか
4. それでも再発するなら、backend 側のボトルネックとして扱う

### 3. 再現後は backend 内部飽和を主因として扱う

実試験では以下を確認した。

- `auth-hub` は 3 replica で `healthy`
- `db` / `pgbouncer` も `healthy`
- `alt-backend` / `k6` の `nofile=65536` は `docker inspect` で確認
- 約 7 分半までは前回より安定
- その後、`feed_read_perf` が 10s〜14s 台へ悪化
- 続いて `EOF` / `connection refused` が再発
- `alt-backend` は `RestartCount=2`, `OOMKilled=false`, `ExitCode=0`

この結果をもって、今回のシリーズでは「外側の実行条件不足」は主因ではなく、**backend 内部の読取経路飽和と、その後の正常終了扱い再起動**を主因として扱うことにした。

### 4. 以後の調査は backend 内部の区間分解を優先する

次フェーズの調査対象を以下に固定する。

1. `GetUnreadFeeds` / `GetAllFeeds` / `FetchArticlesCursor` の DB 待ち時間
2. `feed_read_perf` の `usecase_ms` の内訳
3. 再起動前後の graceful shutdown 経路
4. `pg_stat_activity` / `SHOW POOLS` の同時採取

## RESULTS, EFFECTS

### PROS

- **切り分けの基準が明確になった**: `auth-hub`、`nofile`、concurrency 設定不足だけでは説明できないことを確認できた
- **再現条件が安定した**: 約 7 分半 / 約 1800VU 付近で再発するパターンを再度観測できた
- **次の調査対象が backend 内部に絞れた**: 外側のインフラ設定論から、実際のボトルネック調査へ進める
- **実行スクリプトの品質が上がった**: 今後の feed-read 試験で、条件漏れによる誤判定を減らせる

### CONS, TRADEOFF

- **根本原因そのものは未解決**: script を直しても backend の再起動は残った
- **試験が完走しない**: 現時点では feed-read 3000VU の成功レポートではなく、失敗再現レポートしか積み上がっていない
- **Compose overlay の変数管理が増えた**: 試験時に補間変数を明示しないと `docker compose` が失敗しやすくなった
