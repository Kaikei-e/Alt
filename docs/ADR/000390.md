# ADR-000390: recap-workerの429エラーリトライ不能バグを修正しページネーションにディレイを追加する

## Status

Accepted

## Date

2026-03-15

## Affected Services

- recap-worker (Rust)

## Context

Recap Jobが `alt-backend returned error status 429 Too Many Requests: {"error":"rate limit exceeded"}` で失敗するようになった。

### 根本原因

**2つのバグが組み合わさって発生していた：**

1. **ページネーションにディレイがない**: `fetch_articles()` がページ間にディレイなしで連続リクエストを送信しており、alt-backendのrecap専用レートリミッター（4 RPS / burst 8）に対して、9ページ目以降（4000記事超）でburstキャパシティを超過して429が発生していた。

2. **429がリトライされない（致命的バグ）**: `fetch_page()` が429レスポンスを `anyhow::bail!()` で文字列エラーに変換していた。しかし `fetch_with_retry()` は `reqwest::Error` にダウンキャストしてリトライ可否を判定するため、`anyhow::Error`（文字列）は `reqwest::Error` にマッチせず、429が**リトライ不可として即座にエラー終了**していた。

```
fetch_page() → 429受信 → anyhow::bail!("...")
  → fetch_with_retry() → err.downcast_ref::<reqwest::Error>() → None
  → is_retryable = false → 即座にエラー返却（リトライしない）
```

`is_retryable_error()` は429をretryable判定するロジックを持っていたが、エラー型が `reqwest::Error` ではないためそこに到達しなかった。

### なぜ今日発生したか

7日間ウィンドウの記事数が4000件（8ページ）を超えたことでburstキャパシティを超過し、429が発生。429がリトライされないため即失敗した。

## Decision

`HttpStatusError` カスタムエラー型を導入し、HTTPステータスコードを保持したままエラーを伝播させる方式を採用する。

### 検討した代替案

1. **`reqwest::Error` を直接伝播する案**: `response.error_for_status()` を使えば `reqwest::Error` のまま伝播でき、既存のリトライ判定がそのまま機能する。しかし、レスポンスボディ（エラーメッセージ）が失われるためデバッグが困難になる。
2. **文字列パースでステータスコードを抽出する案**: `anyhow::bail!` の文字列から正規表現でステータスコードを抽出する。脆弱で保守性が低い。
3. **`HttpStatusError` カスタムエラー型（採用）**: ステータスコードとレスポンスボディの両方を保持できる。`fetch_with_retry()` でダウンキャストしてリトライ判定が可能。`thiserror` は既に依存にあるため追加コストなし。

### 実装の概要

| ファイル | 変更内容 |
|---------|---------|
| `src/clients/alt_backend.rs` | `HttpStatusError` 型追加、`fetch_page()` の `anyhow::bail!` を `HttpStatusError` に変更、ページ間300msディレイ追加 |
| `src/pipeline/fetch.rs` | `fetch_with_retry()` のリトライ判定に `HttpStatusError` の429/5xxチェックを追加 |

**ページ間ディレイ**: 300msのインターバルを挿入。4 RPS制限に対して十分な余裕を持ちつつ、10ページ（5000記事）でも追加所要時間は約3秒で実用上問題ない。

## Consequences

### Pros

- 429エラーが既存の指数バックオフ+ジッター戦略でリトライされるようになった
- ページ間ディレイにより、そもそも429が発生しにくくなった
- エラーメッセージにHTTPステータスコードとレスポンスボディが保持され、デバッグが容易になった
- 5xxサーバーエラーも同様にリトライ対象となった

### Cons / Tradeoffs

- ページ間ディレイにより全件取得の所要時間がわずかに増加する（10ページで約3秒）
- `HttpStatusError` のリトライ判定ロジックが `fetch_with_retry()` にインラインで記述されており、`is_retryable_error()` と判定ロジックが分散している
