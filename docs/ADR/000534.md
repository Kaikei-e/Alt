# ADR-000534: Knowledge Sovereign の境界固定と既存 write path の集約を実装する

## Status

Accepted

## Date

2026-03-23

## Affected Services

- alt-backend (Go)

## Context

[[000532]] で Knowledge Sovereign を durable knowledge state の single owner として新設する方針が決定された。しかし、方針だけでは以下の問題が残っていた:

1. **責務境界が実装レベルで未固定** — `knowledge_sovereign` というパッケージもインターフェースも存在せず、「将来やる」のまま放置されるリスクがあった
2. **write path が散在** — `knowledge_projector`、`recall_projector`、`track_home_action_usecase`、`create_summary_version_usecase` 等がそれぞれ独立にリポジトリを直接呼んでおり、書き込み責務の所在が不明確だった
3. **retention / export の設計基盤がなかった** — ADR-000532 が retention を policy matrix 方式と定めたが、実装がなかった

本 ADR は、Phase 0（境界固定）と Phase 1（write path 集約）の実装を記録する。

## Decision

Knowledge Sovereign を alt-backend の既存 Clean Architecture レイヤー（`domain/`、`port/`、`usecase/`、`gateway/`、`driver/`）に追加し、2フェーズで段階的に実装する方式を採用する。

### Phase 0: 境界の固定

Clean Architecture の全5層にわたる最小構造を作成し、Sovereign の責務境界をコード上で固定する。

- **Domain 層**: `RetentionPolicy`（hot/warm/cold/archive）、`ExportClassification`（Tier A/B/C）、`OwnershipEntry` 型
- **Port 層**: `ProjectionMutator`、`RecallMutator`、`CurationMutator`、`RetentionResolver`、`ExportScopeResolver` の5インターフェース + `ProjectionMutation`、`RecallMutation`、`CurationMutation` 型
- **Gateway 層**: `KnowledgeSovereignRepo` interface 経由の委譲
- **Driver 層**: 初期は `ErrKnowledgeSovereignNotImplemented` を返す stub
- **Usecase 層**: `KnowledgeWriteServiceUsecase` — 5メソッドの委譲入口

### Phase 1: 既存 write path の集約

Phase 0 の stub を実装に置き換え、既存の write path を Sovereign 経由に接続する。

#### 1. Mutation type 定数の定義

`MutationUpsertHomeItem`、`MutationDismissHomeItem`、`MutationUpsertTodayDigest` 等9つの定数を `knowledge_sovereign_port` パッケージに追加。

#### 2. Driver dispatch 実装

stub メソッドを `MutationType` による switch dispatch に置き換え。各分岐は Payload を unmarshal して**既存の** repository メソッド（`UpsertKnowledgeHomeItem`、`DismissKnowledgeHomeItem` 等）を呼ぶ。新 SQL は書かない。

#### 3. Projector の Sovereign 経由化

`KnowledgeProjectorConfig` struct に `WriteService WriteServicePort` フィールドを追加する方式を採用。

- 非 nil → `sovereignUpsertHomeItem` 等のヘルパー経由で mutation 構築 → Sovereign に投げる
- nil → 旧パス（直接 port 呼び出し）を維持

この方式を選んだ理由: `KnowledgeProjectorJobWithConfig` は variadic `clearSupersedePort` を末尾に持つため、関数シグネチャにパラメータを追加すると既存テスト30件以上が壊れる。Config struct 拡張なら既存テストは `WriteService` フィールドが zero-value（nil）のまま旧パスで動き続ける。

同様に `RecallProjectorJob` は variadic パラメータとして `WriteServicePort` を受け取る。

#### 4. Curation Sovereign 化

`TrackHomeActionUsecase` に `SetCurationMutator` setter を追加（既存の `SetDismissPort` パターンに準拠）。dismiss 時に curationMutator が非 nil なら Sovereign 経由、nil なら旧 dismissPort を使う。

#### 5. Retention / Export の policy matrix 実装

ADR-000532 セクション 6 の「retention を `articles.retention_tier` の単一カラムだけで終わらせない」という要件に従い、`DefaultRetentionMatrix`（Go map、11 entity types）と `DefaultExportClassification`（6 entity types）を `domain` 層に定義。DB カラムは追加しない。

#### 6. DI container + main.go 配線

`ApplicationComponents` に `KnowledgeSovereignGateway` と `KnowledgeWriteServiceUsecase` フィールドを追加。`main.go` で projector の Config と RecallProjectorJob に WriteService を注入。

### 検討した代替案

| 代替案 | 評価 | 不採用理由 |
|--------|------|-----------|
| `internal/knowledge_sovereign/` にネストした構造 | Clean Architecture 違反 | 既存コードベースは `app/{domain,port,usecase,gateway,driver}` にフラット配置。ネストはパターンを壊す |
| `articles.retention_tier` DB カラム | ADR-000532 が明示的に否定 | entity ごとの policy matrix が必要 |
| projector の関数シグネチャ変更 | テスト破壊が大きい | Config struct 拡張なら既存30テスト不変 |
| 全 write path を一度に切り替え | リスクが高い | dual-path（nil fallback）で段階移行 |

## Consequences

### Pros

- **責務境界がコード上で明確** — 「どこに新しい knowledge-state write path を置くか」で迷わない
- **既存テスト完全保持** — Phase 0/1 合わせて117パッケージ全 PASS、既存テストの破壊ゼロ
- **即時フォールバック可能** — DI container の注入を nil にすれば旧パスに即時復帰
- **retention / export が設計レベルで確定** — policy matrix 方式で entity ごとの粒度を確保
- **Phase 2 以降の基盤** — lens 統合、RetentionWorker、shadow run の受け皿が整った

### Cons / Tradeoffs

- **Dual-path の共存期間** — Sovereign 経由と旧パスが並存する。Phase 2 で旧パスを削除するまでコードが増える
- **mutation の marshal/unmarshal オーバーヘッド** — json.Marshal/Unmarshal が各 write に挿入される。ただし DB write レイテンシに対して無視できる
- **dispatch ロジックの indirection** — `MutationType` switch が増え、直接呼び出しより追跡しにくい
- **lens usecase は未集約** — 5つの lens usecase（create/update/select/archive/list）は Phase 2 送り
