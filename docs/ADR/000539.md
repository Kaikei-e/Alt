# ADR-000539: Read RPC による alt-db 完全依存排除と sovereign_client ポート統合

## Status

Accepted

## Date

2026-03-23

## Affected Services

- alt-backend (Go)
- knowledge-sovereign (Go)

## Context

[[000537]] で knowledge-sovereign を独立サービス + 独立 DB として物理抽出した。write path は Connect-RPC mutation（ApplyProjectionMutation, ApplyRecallMutation, ApplyCurationMutation）経由で sovereign-db に向かうようになった。

しかし **read path は依然として alt-db を直接参照** していた。これにより **split brain** が発生:

- dismiss 操作は sovereign-db の `knowledge_home_items.dismissed_at` を更新
- Knowledge Home の表示は alt-db の `knowledge_home_items` を参照（`dismissed_at` は NULL のまま）
- 結果: dismiss したアイテムが再表示される

database-per-service パターン（[microservices.io](https://microservices.io/patterns/data/database-per-service.html)）に従い、alt-backend から sovereign-db テーブルへの **全アクセス** を Connect-RPC 経由に切り替える必要があった。

## Decision

**alt-backend の sovereign テーブル関連 gateway をすべて削除し、sovereign_client が port interface を直接実装する方式を採用する。**

### 具体的な変更

1. **Proto 拡張**: 31 → 39 RPC に拡張。projection reads、event reads/writes、projection infra、reproject、backfill、lens CRUD、recall signals、user events をカバー
2. **knowledge-sovereign handler/driver**: 全 Read RPC の handler + SQL 実装を追加
3. **sovereign_client 新規ファイル**: `lens_client.go`, `reproject_client.go`, `backfill_client.go`, `signal_client.go`, `write_ports.go` — 合計約 40 メソッドで全 port interface を実装
4. **Gateway 削除**: sovereign テーブルに依存する 10 gateway ディレクトリを削除
5. **alt_db repository 削除**: sovereign テーブルの SQL を含む約 10 ファイルを削除（articles テーブルのみ残留）
6. **DI container 再配線**: `sovereignCli` を全 sovereign port の注入先として使用
7. **main.go 再配線**: job scheduler の全 gateway 参照を `sovereignCli` に置換

### 検討した代替案

| 案 | 評価 |
|----|------|
| Gateway を残して repo フィールドを interface に変更 | gateway が pure pass-through なので二重間接で無意味 |
| alt-backend に sovereign-db の second pool を追加 | database-per-service 違反。形骸化した分離になる |
| **sovereign_client が port を直接実装**（採用） | gateway 層を省略しつつ Clean Architecture の port 契約は維持 |

## Consequences

### Pros

- split brain 完全解消 — read/write が同一の sovereign-db に到達
- alt-db に sovereign テーブルが不要になり DROP 可能（14 テーブル）
- gateway 層の boilerplate 約 600 行を削除
- sovereign が knowledge state の唯一の owner であることがコードレベルで強制される

### Cons / Tradeoffs

- sovereign 障害時に Knowledge Home 全体が unavailable になる（以前は alt-db に read fallback があった）
- Connect-RPC のネットワークレイテンシが加算される（同一 Docker network 内なのでサブミリ秒）
- sovereign_client の proto → domain 変換コードが約 700 行増加
