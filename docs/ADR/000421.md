# ADR-000421: Knowledge Home の Projection 再投影ツーリングを実装する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — Reproject usecase / handler / job / repository
- altctl (Go) — `altctl home reproject` CLI サブコマンド
- alt-frontend-sv (TypeScript) — Admin Reproject タブ UI
- alt-butterfly-facade (Go) — Admin API プロキシ

## Context

Knowledge Home は CQRS/ES アーキテクチャを採用しており（[[000398]]）、`knowledge_events` テーブルがイベントストア、`knowledge_home_items` テーブルが投影（Read Model）である。投影はいつでも破棄・再構築可能な設計だが、Phase 1-4 では再投影の手段が backfill（同一バージョンへの再走査）に限られていた。

プロダクション運用では以下のシナリオで「別バージョンの投影を並行構築し、比較してから切り替える」能力が必要になる:

- 投影スキーマの変更（フィールド追加、スコア計算ロジック変更）
- データ品質の是正（malformed why の修正、orphan item の除去）
- 障害復旧（破損した投影の全面再構築）

Backfill は「同じバージョンへの再走査」であり、reproject は「新バージョンを並行構築して比較・切替・ロールバックできる」上位概念である。

## Decision

### Reproject ツーリングを Clean Architecture に沿って全層に実装する。

**データベース:**

- `knowledge_reproject_runs` テーブル — 再投影実行の状態管理（status: pending → running → validating → swappable → swapped / failed / cancelled）
- `knowledge_projection_audits` テーブル — 投影の正確性監査結果の保存

JSONB カラムは [[000417]] に準拠し `string(json.Marshal(...))` で書き込む。

**モード:**

| モード | 用途 |
|--------|------|
| `dry_run` | 投影を構築せず、比較のみ実行 |
| `user_subset` | 特定ユーザー群のみ再投影 |
| `time_range` | 指定期間のイベントのみ再投影 |
| `full` | 全イベントを新バージョンに再投影 |

**Reproject フロー:**

1. `StartReproject` → `knowledge_reproject_runs` に `pending` で INSERT
2. Background job が pending を拾い `running` に遷移、イベントを新バージョンとして投影
3. 完了 → `swappable` に遷移
4. `CompareReproject` → 件数・why 分布・score 分布・empty rate の diff を算出
5. `SwapReproject` → `knowledge_projection_versions` の active を切替
6. `RollbackReproject` → 前バージョンに戻す

**Admin API (proto 拡張):**

`KnowledgeHomeAdminService` に 6 RPC を追加: `StartReproject`, `GetReprojectStatus`, `ListReprojectRuns`, `CompareReproject`, `SwapReproject`, `RollbackReproject`。加えて `RunProjectionAudit` で投影の正確性監査を実行可能にする。

**altctl CLI:**

`altctl home reproject {start|status|compare|swap|rollback}` コマンドを追加。alt-backend の Admin API を Connect-RPC JSON エンコーディングで直接呼び出す。altctl に Connect-RPC ライブラリ依存を追加せず、標準 `net/http` + `encoding/json` で実装。

**Background Job:**

`knowledge-reproject` ジョブを JobScheduler に登録（30s 間隔）。既存の `knowledge-projector` / `knowledge-backfill` と同じパターンでバッチ処理。

### 代替案の検討

| 案 | 評価 |
|---|---|
| altctl から DB 直接操作 | Clean Architecture 違反。Admin API 経由に統一 |
| altctl に connectrpc.com/connect を追加 | 依存が重い。plain HTTP で Connect-RPC JSON が使えるため不採用 |
| Reproject を backfill の拡張として実装 | バージョン並行構築・比較・切替の概念が backfill にない。別テーブル・別 usecase として分離 |

## Consequences

### Pros

- 投影スキーマ変更時にダウンタイムゼロで切替可能
- Compare で差分を確認してから Swap でき、リスクを最小化
- Rollback で即座に前バージョンに戻せる安全弁
- CLI + Admin UI の二面から操作可能で、自動化にも手動運用にも対応
- 監査機能（RunProjectionAudit）によりデータ品質を定期検証可能

### Cons / Tradeoffs

- Reproject 中は新旧バージョンの投影データが DB 上に共存するため、一時的にストレージ消費が増加
- full モードの再投影は大量のイベントを再走査するため、オフピーク時の実行を推奨
- altctl の plain HTTP クライアントは Connect-RPC の全機能（ストリーミング等）を利用できないが、Admin API は unary RPC のみなので問題ない
