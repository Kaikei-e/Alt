# ADR-000693: Augur リトリーバルパイプラインに embedding 障害時の BM25-only degraded mode を導入する

## Status

Accepted

## Date

2026-04-11

## Affected Services

- rag-orchestrator (Go)

## Context

Augur の RAG リトリーバルパイプラインは 5 段階のステージで構成されている:

1. **Stage 1 (ExpandQueries):** クエリ展開 + タグ検索 + **Original Query Embedding**（並列）
2. **Stage 2 (EmbedAndSearch):** BM25 検索 + ベクトル検索 + 展開クエリ embedding（並列）
3. **Stage 3 (FuseResults):** 展開クエリのベクトル検索 + RRF 融合
4. **Stage 4 (Rerank):** クロスエンコーダ再ランキング
5. **Stage 5 (Allocate):** 言語別アロケーション + クオータ選択

Stage 1 の goroutine C（Original Query Embedding）が `errgroup` の **fatal エラー** として実装されていたため、embedder サービスが停止すると以下のカスケード障害が発生した:

- Stage 1 が即座にエラーを返却
- Stage 2 の BM25 検索（Meilisearch 経由）は到達すらしない
- ユーザーには空の応答または無応答が返る

2026-04-11 に embedder ホストの Ollama を vLLM 導入のために停止した際、この脆弱性により Augur が一切の質問に回答できなくなった。query planning（LLM による意図分類）は正常に動作し、BM25 インフラ（Meilisearch）も正常稼働していたにもかかわらず、embedding の single point of failure でパイプライン全体が停止した。

## Decision

**Original Query Embedding の失敗を non-fatal に変更し、BM25-only の degraded mode を導入する。**

embedding が利用不可の場合、パイプラインはベクトル検索をスキップし、BM25（キーワード検索）の結果のみで回答を生成する。これは query expansion 失敗時の既存パターン（warn ログ + `return nil`）と同じ設計に揃えた。

### 検討した代替案

| 案 | 評価 |
|----|------|
| A. embedder 障害をそのまま fatal のまま維持し、運用で対処 | × embedder 停止が予告なく発生しうる（GPU メンテ、モデル入れ替え） |
| B. Circuit breaker で embedder 障害を検知して BM25 にフォールバック | △ 複雑。現時点では embedder は単一ホストなので on/off で十分 |
| **C. embedding 失敗を non-fatal にし、各 Stage で nil チェック** | ○ 最小の変更。既存パターンと一貫性がある |

案 C を採用した。

## Implementation

### 変更した 3 ファイル

**1. `retrieval/expand_queries.go` — goroutine C を non-fatal に変更**

```go
// Before: return fmt.Errorf("failed to encode original query: %w", err)
// After:
logger.Warn("original_embedding_failed", ..., slog.String("degraded_mode", "bm25_only"))
return nil // non-fatal: downstream stages check sc.OriginalEmbedding == nil
```

**2. `retrieval/embed_and_search.go` — ベクトル検索のガード追加**

```go
if sc.OriginalEmbedding != nil {
    g.Go(func() error { /* vector search */ })
} else {
    logger.Warn("vector_search_skipped", ..., slog.String("degraded_mode", "bm25_only"))
}
```

BM25 検索（goroutine E）は embedding の有無に関わらず常に実行される。

**3. `retrieval/fuse_results.go` — BM25-only degraded mode の早期リターン**

```go
if sc.OriginalEmbedding == nil && len(sc.AdditionalEmbeddings) == 0 {
    sc.HitsOriginal = promoteBM25ToSearchResults(sc.BM25Results)
    sc.HitsExpanded = nil
    return nil
}
```

`promoteBM25ToSearchResults` は `domain.BM25SearchResult` を `domain.SearchResult` に変換し、下流の Rerank / Allocate がそのまま処理できるようにする。

## Consequences

### Pros

- **耐障害性向上**: embedder 停止時もキーワード検索ベースで回答を継続できる
- **運用柔軟性**: GPU メンテナンス・モデル入れ替え時にサービス全停止を回避
- **設計の一貫性**: query expansion と同じ non-fatal パターンに統一
- **Observability**: `degraded_mode: bm25_only` ログタグにより degraded 状態を即座に検知可能

### Cons / Tradeoffs

- **BM25-only 時の回答品質低下**: ベクトル検索��しではセマンティック検索の精度が落ちる。キーワード一致に依存するため、同義語や言い換えへの対応が弱くなる
- **暗黙の degradation**: ユーザーには品質低下が明示されない。将来的にはフロントエンドに degraded mode の表示を検討
