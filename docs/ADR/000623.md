# ADR-000623: Knowledge Home ストリーム初期 heartbeat 即時送信で Reconnecting 遅延を解消する

## Status

Accepted

## Date

2026-04-05

## Affected Services

- alt-backend (Go) — Connect-RPC ストリーミングハンドラ
- alt-frontend-sv (TypeScript/SvelteKit) — `useStreamUpdates` フック

## Context

Knowledge Home のストリーム接続時に「Reconnecting...」が長時間（最大 21 秒）表示される問題が報告された。

### 根本原因の特定

nginx アクセスログの `uht`（upstream header time）を分析したところ、`StreamKnowledgeHomeUpdates` エンドポイントの全リクエストで一貫して **5〜10 秒** の初期遅延が発生していた。

```
StreamKnowledgeHomeUpdates uht=5.007   (イベントあり → updateTicker 5s 後に初送信)
StreamKnowledgeHomeUpdates uht=10.008  (イベントなし → heartbeatTicker 10s 後に初送信)
StreamKnowledgeHomeUpdates uht=10.024
StreamKnowledgeHomeUpdates uht=10.011
```

原因は `StreamKnowledgeHomeUpdates` ハンドラがストリーム開始後、for/select ループに入る前に初期メッセージを一切送信していないことだった。Connect-RPC では最初の `Send()` 呼び出しがレスポンスヘッダーをネットワークに書き出すため、初期メッセージなしではクライアントは最初の ticker（5s or 10s）が発火するまでレスポンスを受け取れない。

対照的に、同リポジトリの `StreamFeedStats` は `// Send initial data immediately` パターンで即座にデータを送信しており、同様の遅延は発生していなかった。

### 遅延の内訳

| 要因 | 遅延 | 詳細 |
|------|------|------|
| Backend first byte | 5-10s | 初期メッセージ未送信（主因） |
| Leader election timeout | 3s | `LEADER_CLAIM_TIMEOUT=3000` が過大 |
| Server reconnect hint | 5-8s | `stream_expired` 後の `reconnect_after_ms` |
| **合計（最悪ケース）** | **最大 21s** | |

## Decision

以下の 2 点を修正する。

### 1. バックエンド: ストリーム開始時に即座に heartbeat を送信する

`StreamKnowledgeHomeUpdates` および `StreamRecallRailUpdates` のハンドラで、for/select ループに入る前に heartbeat メッセージを即座に送信する。`StreamFeedStats` の既存パターン（`feeds/feed_stats.go:129-131`）と同一のアプローチ。

```go
// Send immediate heartbeat so the client receives the first byte instantly.
if err := stream.Send(&knowledgehomev1.StreamKnowledgeHomeUpdatesResponse{
    EventType:  "heartbeat",
    OccurredAt: time.Now().Format(time.RFC3339),
}); err != nil {
    return err
}
```

### 2. フロントエンド: Leader election タイムアウトを 3000ms → 500ms に短縮する

BroadcastChannel メッセージは同一ブラウザ内で <50ms で到達する。3000ms は過大であり、初回接続時に不要な待機を発生させていた。500ms は十分なマージンを持ちつつ、体感遅延を大幅に改善する。

### 検討した代替案

- **初期データ（最新イベント）を即座に送信する案**: `StreamFeedStats` はこのパターンだが、Knowledge Home ではイベントが存在しない場合もあるため、heartbeat 送信のほうがシンプルで一貫性がある
- **Web Locks API による leader election**: BroadcastChannel より高速だが、現時点では BroadcastChannel で十分。500ms への短縮で実用上の問題は解消される

## Consequences

### Pros

- `uht` が 5-10 秒 → <100ms に改善（nginx ログで検証可能）
- 「Reconnecting...」の表示時間が最大 21 秒 → 約 1 秒に短縮
- 既存の `StreamFeedStats` パターンとの一貫性が向上
- heartbeat イベントタイプは既にフロントエンドでフィルタ済み（`if (event.eventType === "heartbeat") continue;`）のため、クライアント側の変更は最小限
- `StreamRecallRailUpdates` にも同パターンを適用し、全ストリーミングエンドポイントで一貫した即時応答を実現

### Cons / Tradeoffs

- heartbeat メッセージが 1 通増えるが、ペイロードは数十バイトで無視できるサイズ
- Leader election タイムアウト短縮により、高 CPU 負荷時に一時的に複数リーダーが選出される可能性が僅かに上昇するが、既存の設計で複数リーダーは安全にハンドリングされている
