# ADR-000776: 記事言語メタデータを DB から Acolyte まで end-to-end で伝搬し、ingestion 時に ja/en/und を決定的に付与する

## Status

Accepted

## Date

2026-04-19

## Affected Services

- **migrations-atlas** — `migrations/20260419000000_add_language_to_articles.sql` 追加。`articles.language TEXT NOT NULL DEFAULT 'und'` と `idx_articles_language` を付与し `atlas migrate hash` で `atlas.sum` を更新
- **proto** — `services/backend/v1/internal.proto` の `ArticleWithTags.language = 8` と `CreateArticleRequest.language = 7` を追加
- **alt-backend** — driver の SELECT+Scan、INSERT の `ON CONFLICT` CASE 式、port / gateway / handler の全層を同期
- **pre-processor** — `service/language_detector.go` (stdlib Unicode 範囲判定) + 記事 Create 時の自動付与
- **search-indexer** — domain Article / SearchDocument / driver / gateway の全層に Language を貫通
- **acolyte-orchestrator** — `ArticleHit.language` / `ArticleMetadata.language` を追加し search-indexer REST 応答から抽出

## Context

2026-04-19 にユーザから「Acolyte レポートの引用元に英語記事が引っ張ってこれていない。日英両方から質の良いものを LLM やシステムが取捨選択できるべき」との要件が示された。現場サンプル (`reports/eb67863f-…`) でも引用が日本語ソースに偏っており、英語候補が残っていても Gatherer / Curator の最終選択に混入しない挙動が観測された。

調査の結果、**言語メタデータが DB から Acolyte まで一度も表に出ない**ことが根本原因と判明:

| レイヤ | 言語情報 | 備考 |
|---|---|---|
| `articles` テーブル | なし | `id / title / content / url / created_at / published_at / user_id / feed_id / deleted_at` だけ |
| alt-backend `InternalArticleWithTags` / 内部 proto | なし | `ListArticlesWithTags` で select されない |
| search-indexer `ArticleWithTags` / `SearchDocument` | なし | Meilisearch `LocalizedAttributes=jpn,eng` のトークナイザ設定だけ (`meilisearch_driver.go:242-256`) でドキュメントに言語属性は持たせていない |
| acolyte-orchestrator `ArticleHit` | なし | Gatherer / Curator は score しか見ない |

Meilisearch 側がクロス言語マッチを許すためにあえて locale をドキュメントに付けていない判断は合理的だが、結果として「BM25 がクエリ言語 (日本語) に一致する語彙を優先 → 英語文書がスコアで常に負ける → Curator LLM に見せる時点でほぼ日本語のみ」というバイアスがそのまま通り、英語候補が存在してもレポートに乗らない。

後続 ADR ([[000777]]) で Acolyte の Curator に「言語ソフトクォータ」と Writer にクロスリンガル gloss 指示を入れる計画だが、それらはすべて **言語メタデータが end-to-end で伝搬していること**が前提条件になる。したがって本 ADR で先に配管だけ通すのが eval-first / incremental の最短ルートと判断した。

イミュータブル不変条件 (`knowledge_events`, projector, `summary_versions`, `tag_set_versions`) への影響は無い。`articles` テーブルへの列追加はイミュータブル projection の source of truth には影響せず、既存行は `DEFAULT 'und'` で静かに埋まる。

## Decision

記事言語メタデータを **DB (`articles.language`) → proto (`ArticleWithTags.language` / `CreateArticleRequest.language`) → 各サービス domain → Acolyte `ArticleHit.language`** の経路で end-to-end に伝搬させる。言語コードは BCP-47 短形 (`ja`, `en`) と `"und"` (undetermined) の 3 値に限定する。検出は **ingestion 時に 1 回だけ**行い、検索ホットパスには検出コストを置かない。

1. **スキーマ (Atlas)** — `ALTER TABLE articles ADD COLUMN language TEXT NOT NULL DEFAULT 'und'` + `CREATE INDEX idx_articles_language`。既存行はすべて `'und'` になり、pre-processor が次に触った記事から正しい値で上書きされていく。`atlas migrate hash` で `atlas.sum` を更新。

2. **Proto** — 既存 `ArticleWithTags` と `CreateArticleRequest` に `string language = 8` / `string language = 7` を追加。proto3 optional ではないため未設定時は空文字となり、**alt-backend 側では `"und"` にフォールバックするセンチネル扱い**にして「proto default ""」と「明示的 `"und"`」を同一視する。

3. **alt-backend** — driver 層で 3 つの SELECT (list/list-forward/get-by-id) と INSERT ON CONFLICT に `language` を同期で通す。INSERT の `ON CONFLICT ... DO UPDATE SET` では言語を次の CASE で守る:
   ```sql
   language = CASE
     WHEN EXCLUDED.language IS NOT NULL AND EXCLUDED.language <> 'und'
       THEN EXCLUDED.language
     ELSE articles.language
   END
   ```
   これにより、**未検出 (`und`) の後続書き込みが、過去に検出済みの確定値を潰さない**。port / gateway / handler は proto↔domain マッピングを同期するだけ。既存 driver テストの `WithArgs(...)` に `"und"` を追加し、新規 `TestListArticlesWithTags_PropagatesLanguage` で `ja` / `en` / 空の 3 パターンが proto まで貫通することを固定する。

4. **pre-processor 検出器** — `service/language_detector.go` を追加。lingua / fastText / pycld3 のような外部モデルは取らず、**Go stdlib `unicode` のみ**で:
   - `unicode.Is(Hiragana|Katakana|Han, r)` を「日本語字母」
   - ASCII 範囲 (`r < 0x0100`) の `unicode.IsLetter(r)` を「英語字母」
   - 数字・記号・空白は全てスキップ
   - 字母総数 < 2 は `"und"`
   - 日本語字母が総数の 1/3 以上なら `"ja"` (少数の CJK でも情報密度が高いため英語優位に倒さない)
   - そうでなければ英語字母が多ければ `"en"`、同数以下なら `"und"`
   記事 title + content 先頭のみ見る短文判定で十分という割り切りで、10 ケースのテーブル駆動テストで日本語ニュース / 英語テック / ローンワード混在 (「The word 寿司 is popular」は `en`) / 数字だけの `"und"` などを固定する。
   `driver/backend_api/article_driver.go` の `Create` で `article.Language == ""` のときのみ自動検出し proto に載せる。明示的に値が入っている場合 (future migration job 等) はそれを尊重する。

5. **search-indexer** — `domain.Article` に `Language string` + `SetLanguage(string)` を追加 (ctor シグネチャは不変にして既存呼び出しを壊さない)。`SearchDocument.Language` を JSON 付きで露出し、Meilisearch ドキュメントの filterable attribute に追加。`LocalizedAttributes` は従来どおり触らず、クロス言語トークナイズを維持したまま、クライアントだけが言語で絞れるようにする。`toDriverArticle` / `convertToDomain` で proto→driver→domain のマッピングを同期。

6. **acolyte-orchestrator** — `ArticleHit` / `ArticleMetadata` に `language: str = "und"` (dataclass frozen、mutable default 無し)。`SearchIndexerGateway.search_articles` で `str(hit.get("language") or "und")` で抽出、`search-indexer` が未実装で language を返さない旧デプロイ中でも安全に `"und"` に落ちる。

### 検討した代替案

| 案 | 採用 | 却下理由 |
|---|---|---|
| **A: DB 列 + proto + 各層 end-to-end 伝搬**（採用） | ○ | Acolyte の Curator / Writer がレイヤを跨いで言語を見たいケースで最も素直。検出は 1 回 (ingestion) で済み、検索ホットパスを汚さない |
| B: Meilisearch 側の `LocalizedAttributes` を document per locale に分割 | × | Meilisearch のトークナイザ設計思想と逆行。クロス言語クエリがそもそも打てなくなる |
| C: 多言語埋め込み (BGE-M3 等) を導入して language を不要にする | × | 単一マシン構成 ([[feedback_single_machine_architecture]]) で embed model を追加する GPU 予算が無い。既存 BM25 + 量化で同等効果を狙える |
| D: 言語検出を Acolyte Gatherer 側に移す | × | 検索ホットパスで記事ごとに text detect を走らせると遅延 / OOM リスクがある。記事は ingestion で一度だけ書き込まれ、取得は大量に走るため、書き込み側の pre-processor が最適 |
| E: 外部ライブラリ (lingua-go / fastText) で検出 | × | fastText は 170MB の model file で CI / プロダクトボリューム肥大化。lingua-go でも数十 MB + 初回ロード JIT。短文 (ニュース title) では Unicode 範囲で十分な精度が出ており、stdlib だけで納めた方が CI も軽い |
| F: ingestion で検出したものだけに頼り、backfill はしない | ○ (部分採用) | 全件 backfill は別ジョブが必要。[[feedback_no_unsolicited_benchmark_builds]] 準拠で ADR スコープに混ぜず、本 ADR は「新規記事は正しい値、既存記事は `'und'`」で受け入れる |

## Consequences

### Pros

- `articles.language` が単一の source of truth となり、proto / domain / Acolyte のどこからでも参照できる。後続の Curator ソフトクォータ ([[000777]]) / Writer クロスリンガル表示 ([[000777]]) の前提が整う。
- INSERT の `ON CONFLICT` CASE 式により、Inoreader 経由で同じ URL を `und` で再上書きしても、過去に pre-processor が検出した `ja` / `en` を潰さない。
- 検出ロジックが stdlib のみで完結するため、pre-processor のビルド時間 / コンテナサイズに影響しない。再現性も高く、CI で 10 テーブルパターンを閉じた挙動で固定できる。
- search-indexer 側は `LocalizedAttributes=jpn,eng` のトークナイザ設定を温存したままで、Meilisearch のクロス言語マッチングは従来どおり。`SearchDocument.Language` は将来 filter (`languageCode=en`) を打つためのフィールドとして独立に機能する。
- acolyte-orchestrator の dataclass は `frozen=True` のままで `language: str = "und"` のスカラデフォルトのみ。mutable default やチェックポイント state 破壊を引き起こさない。
- proto 変更は field number 追加 (7 / 8) のみで後方互換。旧 client (language 未設定) は空文字で到着し、alt-backend 側で `"und"` に落とされる。

### Cons / Tradeoffs

- **既存記事はすべて `'und'` のまま残る**。Acolyte の言語クォータ ([[000777]]) は「英語記事が候補プールに存在するとき」にだけ作動する設計なので、既存記事だけから成るトピックでは英語比率は当面動かない。backfill ジョブは別 PR / 別 ADR で検討。
- **Unicode 範囲判定は粗い**。長い混在テキスト (ビジネス日本語記事に英単語が多数混じる) では誤判定が起き得る。誤判定の確認は評価ハーネス ([[000777]] B1) の `lang_mix_ratio` で継続モニタし、規模が見えてから本格検出器 (lingua 等) への差し替えを検討する。
- **proto regeneration 必須**。`buf generate --template buf.gen.backend-internal.yaml` と `buf.gen.pre-processor-services.yaml` / `buf.gen.search-indexer.yaml` を叩いて 3 生成物を更新したが、CI 側で `make proto` を走らせる運用が無いため、今後 proto を触る PR では「手で regenerate → commit」の手順が暗黙的必須になる。
- **alt-backend driver テストの更新**: `WithArgs(...)` の末尾に `"und"` を足す変更が機械的に発生した。将来 language 以外の列を追加する際にも同様の更新が必要で、pgxmock ベースの pattern に固有のノイズが残る。
- **Meilisearch の reindex は即時ではない**。SearchDocument に `language` を足したが、古いドキュメントは再インデックスされるまで `language` フィールドを欠く。search-indexer の既存 backfill / incremental 再インデックスで自然に追従する想定で、本 ADR では強制再インデックスジョブは走らせない。
- **検出閾値 1/3 の経験値**。`東京オリンピック 2028 開催地決定` のように数字が多い CJK 文章を `ja` に倒すために 1/3 を採用したが、新聞見出しで英単語が短く挟まるケース (例: `AI 導入`) は `ja` に倒したいのに英語票が勝つ可能性が残る。必要なら設定化するが現状ハードコードで許容する。
