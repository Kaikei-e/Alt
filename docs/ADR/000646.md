# ADR-000646: Augur RAG quality gate を intent-aware 閾値化し graceful degradation で hard fallback を廃止する

## Status

Accepted

## Date

2026-04-08

## Affected Services

- rag-orchestrator (Go)
- news-creator (Python)

## Context

ADR-645 の system role 分離により 1 ターン目の応答品質は大幅改善したが、2 ターン目のフォローアップクエリおよび因果分析クエリの quality gate が厳格すぎて fallback する問題が継続していた。Eval baseline は **17% pass rate (2/12)** であり、主要な systemic issues として以下が特定されていた:

- Quality gate が causal/abstract クエリを過剰拒否（cross-encoder スコアが本質的に低い）
- Follow-up answers too short（4/12）
- Query expansion が multi-turn で garbage（`:):):):)...`）を生成
- Causal strategy の subquery keywords が固定で topic-agnostic

### 根本原因の特定

Web リサーチ（一次ソース優先）により、以下の根本原因を確認:

1. **Cross-encoder の causal bias**: bge-reranker-v2-m3 は factual QA データセットで学習されており、因果・概念的クエリに対するパッセージのスコアが本質的に 0.2-0.45 に集中する。Good 閾値 0.5 は因果クエリに対して aggressive すぎる
2. **AssessWithIntent の追加 downgrade**: `IntentCausalExplanation` に対して Good→Insufficient への downgrade ヒューリスティックがあり、低スコア問題を悪化させていた
3. **Hard fallback の UX 破壊**: Google Sufficient Context（ICLR 2025）で 55.4% の RAG インスタンスが insufficient context だが、モデルは 35-62% のケースで正答可能であることが示されている。Hard fallback は常に UX を悪化させる
4. **Query expansion の degenerate output**: Gemma 4 E4B が multi-turn context で温度 0.3 で garbage を生成

### 先行研究

| Source | Finding |
|--------|---------|
| CRAG（ICLR 2024, arXiv:2401.15884） | 閾値はクエリタイプごとに calibrate すべき |
| Google Sufficient Context（ICLR 2025, arXiv:2411.06037） | Hard fallback は常に UX 悪化。Tiered response が正解 |
| REBEL（arXiv:2504.07104） | Relevance-only 最適化は information bottleneck を生む |
| AILS-NTUA（arXiv:2603.10524, SemEval-2026 MTRAG 1位） | 温度 0.0、XML 構造化プロンプト、few-shot が key |
| arxiv 2505.12694 | Query expansion は knowledge deficiency で hallucination しやすい。Structured output で防止が最も効果的 |

## Decision

5 つの施策を統合的に実施する。

### 1. Intent-Aware Quality Thresholds

`AssessWithIntent()` に intent 別閾値を導入:

| Intent Group | Good | Marginal | 根拠 |
|-------------|------|----------|------|
| General, Temporal, Comparison, FactCheck | 0.50 | 0.25 | 現状維持 |
| CausalExplanation, Synthesis, TopicDeepDive | 0.30 | 0.15 | Cross-encoder の causal bias 補正 |

### 2. Causal Downgrade ヒューリスティック緩和

- Good→Insufficient downgrade を **廃止**（Good→Marginal に緩和）
- Marginal→Insufficient は **query-context mismatch のみ**に限定（クエリキーワードが全コンテキストに不在の場合のみ）

### 3. Graceful Degradation（Hard Fallback 廃止）

`QualityInsufficient` 時の `return errors.New(...)` を廃止。contexts > 0 の場合は `lowConfidence=true` フラグで生成を継続し、prompt に disclaimer（「情報の信頼性に関する注意」）を追加。Causal strategy 内の `return nil, nil`（requireGoodVerdict 未達時）も同様に廃止し、best available result を返す設計に変更。

### 4. Query Expansion 安定化

- `expand_query_usecase.py` の温度を 0.3 → **0.0** に変更（deterministic）
- Multi-turn テンプレートに **few-shot example** を追加（coreference resolution の具体例）
- 会話履歴の **sanitize 強化**（URL 除去、特殊文字除去、150 文字制限）
- `filterExpandedQueries()` に **repeating pattern filter**（`isGarbagePattern()`）を追加

### 5. Causal Subqueries 動的化

固定キーワード（`"供給 制裁 sanctions supply"` 等）を削除し、query planner が生成した `SearchQueries` を使用。`QueryIntent` に `SearchQueries` フィールドを追加して伝搬。

## Consequences

### Pros

- **Causal/abstract クエリの fallback 率が大幅低下**: Intent-aware 閾値により、cross-encoder の causal bias が補正される
- **UX 改善**: Hard fallback（"I couldn't find enough information"）が廃止され、低確信度でも利用可能な情報を最大限活用した回答が生成される
- **Query expansion の安定性向上**: 温度 0.0 + few-shot + sanitize + garbage filter の 4 層防御により、multi-turn での garbage 生成が防止される
- **Topic-aware retrieval**: Causal strategy が planner の search_queries を使用し、固定キーワードへの依存を排除
- **Streaming 体感速度向上**: Flush 粒度 40 rune でより細かくプレビューが更新される

### Cons / Tradeoffs

- **低品質コンテキストでの生成リスク**: Graceful degradation により、本来 insufficient な情報で回答が生成される可能性がある。Disclaimer で緩和しているが、ユーザーが disclaimer を読まないリスクは残る
- **閾値の empirical 設定**: Causal Good=0.30 / Marginal=0.15 は research evidence に基づくが、Alt の RSS コーパスに最適化されていない。Eval harness での継続的な calibration が必要
- **Planner queries への依存**: Causal subqueries が planner の `SearchQueries` に依存するため、planner の出力品質が retrieval 全体に影響する
