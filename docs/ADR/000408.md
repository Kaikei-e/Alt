# ADR-000408: RecallRail を導入し知識の想起を支援する

## Status

Accepted

## Date

2026-03-18

## Affected Services

- alt-backend (Go) — domain, ports, usecases, gateways, repositories, handler, projector job
- alt-frontend-sv (SvelteKit) — Connect client, hooks, RecallRail コンポーネント群

## Context

Knowledge Home Phase 3 までで「表示する面」は成立した（TodayBar, KnowledgeStream, MiniRecallPanel）。しかし、ユーザーが過去に読んだ記事を適切なタイミングで思い出す仕組みがなく、知識の定着・活用に課題があった。

`recall_candidate_view` テーブルは Phase 1 でスケルトンとして作成済みだったが、スコアリングロジックも UI も未実装だった。

**目標:**

- ユーザーの行動シグナル（開封、検索、Augur 質問など）を基に「そろそろ思い出した方がよいもの」をスコアリングする
- Knowledge Home のサイドバーに RecallRail として表示し、snooze / dismiss のライフサイクルを提供する
- MiniRecallPanel とは feature flag で択一切替（RecallRail OFF → MiniRecallPanel にフォールバック）

## Decision

**シグナルベースのスコアリングアルゴリズムで recall candidates を算出し、RecallRail UI として Knowledge Home サイドバーに表示する。**

### データベース設計

**`recall_signals` テーブル（新規）:**

ユーザーの行動シグナルを蓄積する append-only テーブル。

| カラム | 型 | 説明 |
|-------|---|------|
| signal_id | UUID PK | シグナル一意キー |
| user_id | UUID | ユーザー |
| item_key | TEXT | 対象アイテム |
| signal_type | TEXT | opened, search_related, augur_referenced 等 |
| signal_strength | NUMERIC | 重み係数（デフォルト 1.0） |
| occurred_at | TIMESTAMPTZ | 発生時刻 |
| payload | JSONB | 追加コンテキスト |

**`recall_candidate_view` の拡張（既存テーブル ALTER）:**

| 追加カラム | 型 | 説明 |
|-----------|---|------|
| projection_version | INTEGER | プロジェクションバージョン |
| first_eligible_at | TIMESTAMPTZ | 初回 recall 対象日時 |
| snoozed_until | TIMESTAMPTZ | snooze 期限 |

部分インデックス: `(user_id, next_suggest_at ASC) WHERE next_suggest_at IS NOT NULL AND snoozed_until IS NULL`

### スコアリングアルゴリズム

recall-rail-projector ジョブが 60 秒間隔で実行し、直近 7 日間のシグナルを集約:

| シグナルタイプ | 重み | 条件 |
|-------------|-----|------|
| opened_before_but_not_revisited | 0.30 | 開封後 48 時間以上再訪なし |
| related_to_recent_search | 0.25 | タグが検索語と重複 |
| related_to_recent_augur_question | 0.35 | Augur Q&A で参照 |
| recap_context_unfinished | 0.20 | recap 開封だが記事未読 |
| pulse_followup_needed | 0.25 | pulse anchor 開封、関連未読 |
| tag_interest_overlap | 0.15 | トップ 5 タグ重複 |

合計スコアが 0.2 を超えた場合に `recall_candidate_view` に upsert。snoozed / dismissed のアイテムはスキップ。

### Clean Architecture レイヤー

```
Handler (GetRecallRail, TrackRecallAction)
  → Usecase (recall_rail, recall_snooze, recall_dismiss)
    → Port (recall_candidate_port, recall_signal_port, knowledge_event_port)
      → Gateway (recall_candidate_gateway, recall_signal_gateway)
        → Driver (recall_candidate_repository, recall_signal_repository)
```

### Proto RPC

| RPC | 説明 |
|-----|------|
| `GetRecallRail` | candidates を score DESC で取得、flag ガード付き |
| `TrackRecallAction` | snooze / dismiss / open の追跡 |

`GetKnowledgeHomeResponse` にも `recall_candidates` フィールドを追加し、初期ロード時に 5 件を埋め込む（追加ラウンドトリップ回避）。

### フロントエンド

| コンポーネント | 説明 |
|-------------|------|
| `RecallRail.svelte` | デスクトップサイドバー（w-80） |
| `RecallCandidateCard.svelte` | タイトル、経過日数、reason バッジ、snooze/dismiss ボタン |
| `RecallReasonBadge.svelte` | reason code → ラベル / アイコン / 色マッピング |
| `RecallRailCollapsible.svelte` | モバイル向け折りたたみラッパー |
| `useRecallRail.svelte.ts` | candidates 取得 / snooze / dismiss の状態管理 |

Feature flag `enable_recall_rail` が OFF の場合、MiniRecallPanel にフォールバック。

### イベント追跡

snooze / dismiss は `knowledge_events` に `RecallSnoozed` / `RecallDismissed` イベントとして記録し、将来の分析・スコアリング改善に活用する。

## Consequences

### Pros

- **知識の活用促進** — 読んだが忘れていた記事を適切なタイミングで再提示
- **行動ベースの個人化** — 固定ルールではなく、実際のユーザー行動からスコアリング
- **段階的展開** — feature flag による制御で、問題があれば即座に OFF にできる
- **MiniRecallPanel 互換** — フラグ OFF 時は既存の MiniRecallPanel が表示される
- **snooze/dismiss ライフサイクル** — ユーザーが recall の頻度を制御可能

### Cons / Tradeoffs

- **projector ジョブの負荷** — 60 秒間隔のポーリングは、ユーザー数増加時にスケーリング戦略が必要
- **スコアリングの初期品質** — 重みは経験的に設定しており、実際のユーザー行動で調整が必要
- **シグナル収集の網羅性** — Phase 4 時点では opened / search / augur の基本シグナルのみ。Pulse 連携等は今後拡張
- **recall_signals の成長** — append-only テーブルのため、古いシグナルのパージ戦略が将来必要
