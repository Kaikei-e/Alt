# ADR-000617: Ask Augur に Tool-Use Agentic RAG パイプラインを導入し概念的合成クエリの回答品質を抜本改善する

## Status

Accepted

## Date

2026-04-03

## Affected Services

- rag-orchestrator (Go)

## Context

Ask Augur は過去の改善（[[000604]] ConversationPlanner, [[000607]] IntentCausalExplanation, [[000608]] SubIntent 別プロンプト, [[000609]] Planner clarification 修正, [[000614]] /api/chat 移行）を経ても、広範な概念的・合成的クエリに対する回答品質が不十分であった。

例: 「そもそもニューヨークと芸術のかかわりは？」に対し、断片的で浅い回答しか返せない。

### 根本原因

過去の改善はすべて **rag-orchestrator 内部の単一パス検索パイプラインの漸進的修正** に終始していた:

1. **単一パス検索**: 1回の vector+BM25 検索で広範なトピックのカバレッジを確保できない
2. **マイクロサービス未活用**: Alt の豊富なサービスエコシステム（search-indexer, tag-generator, alt-backend, pre-processor）を検索パイプラインに統合していない
3. **ルールベース偏重**: ToolDispatcher が固定ルールでツールを選択し、クエリの性質に応じた柔軟なツール選択ができない
4. **プロンプト構造の固定**: IntentGeneral（catch-all）の「概要→詳細→まとめ」構造が概念的合成には不適切

### 調査に基づく設計判断

- **Azure Agentic Retrieval**: LLM がクエリを分解 → 並列 sub-query → semantic reranking → 統合応答
- **Agentic RAG Survey (arXiv 2501.09136)**: Plan → Retrieve → Evaluate → Decide のループ、Tool-augmented retrieval
- **HyDE (Hypothetical Document Embeddings)**: 仮想回答の embedding で語彙ギャップを解消
- **SLM-first Agent Architecture**: 小型モデル（Gemma4 E4B）でのローカルプランニングが実用水準に到達

## Decision

**Alt の既存マイクロサービスを「ツール」として抽象化し、LLM 駆動の Tool-Use Agentic RAG パイプラインを導入する。**

既存の General/Temporal/CausalExplanation 等のパイプラインは一切変更せず、新しい `IntentSynthesis` のみが専用パイプラインにルーティングされる。

### アーキテクチャ

```
Query → Classify ─┬─ Simple (General/Temporal/...) → 既存パイプライン（変更なし）
                   │
                   └─ IntentSynthesis → ToolPlanningAgent
                        ├─ Gemma4: ツール実行計画を JSON 生成 (1 LLM call)
                        ├─ 並列ツール実行 (search-indexer, alt-backend, tag-generator, etc.)
                        ├─ 結果集約 + ベクトル検索
                        └─ 合成プロンプト → 回答生成
```

### 検討した代替案

1. **rag-orchestrator 内部のみの最適化（クエリ分解 + HyDE を内部パイプラインに追加）**: 既存サービスを活用せず個別最適に終わる。Alt のマイクロサービスエコシステムの強みを活かせない
2. **LangGraph 等の外部フレームワーク導入**: 新たな依存を追加し、Go/Python 混在のデバッグ困難度が増す。Alt の Clean Architecture と整合しない
3. **全クエリに Tool-Use を適用**: レイテンシとリソースコストが全リクエストに波及する。IntentSynthesis のみに限定することで影響範囲を制御

### 実装の概要

#### 1. IntentSynthesis 分類

`query_classifier.go` に `matchesSynthesis()` を追加。検出パターン:
- 明示的合成シグナル: 「そもそも」「とは何」「全体像」「概観」
- 抽象関係パターン: `<Entity>と<Entity>の(かかわり|関係|影響|つながり)`
- 英語: "relationship between", "overview of", "how are...connected"

優先順位: ArticleScoped → Comparison → CausalExplanation → **Synthesis** → Temporal → FactCheck → DeepDive → General

#### 2. マイクロサービス Tool 化（5つの新規ツール）

既存サービスの RPC を `domain.Tool` interface でラップ:

| ツール名 | バックエンドサービス | RPC | 目的 |
|----------|-------------------|-----|------|
| `tag_cloud_explore` | alt-backend | FetchTagCloud | タグ空間探索でファセット発見 |
| `articles_by_tag` | alt-backend | FetchArticlesByTag | タグ指定の記事検索 |
| `search_recaps` | search-indexer | SearchRecapsByTag | 要約済みトピッククラスター検索 |
| `extract_query_tags` | tag-generator (via mq-hub) | GenerateTagsForArticle | クエリからのセマンティックタグ抽出 |
| `summarize_for_context` | pre-processor | Summarize | 記事のオンデマンド要約 |

既存ツール（`related_articles`, `tag_search`, `article_lookup`, `date_range_filter`）もそのまま利用可能。

#### 3. LLM ベース ToolPlanner

`ToolPlanner` が Gemma4 に利用可能なツール一覧とクエリを渡し、構造化 JSON でツール実行計画を生成。フォールバック: JSON パース失敗・LLM エラー時はデフォルトプランに退避。

#### 4. ToolDispatcher 並列実行拡張

`ExecutePlan()` が `ToolPlan` の依存関係を尊重しつつ並列実行。独立ステップは goroutine で同時実行、依存ステップは前提完了後に実行。ツール単体の失敗は他ステップに波及しない。

#### 5. SynthesisStrategy

`RetrievalStrategy` interface を実装。ToolPlanner → ToolDispatcher → 結果集約 → ベクトル検索 → マージの一連の流れを統合。ツール結果は `SupplementaryInfo` としてプロンプトに注入。

#### 6. HyDE Generator

カバレッジ不足時の適応的フォールバック。仮想回答を Gemma4 で生成 → embedding → ベクトル検索で追加コンテキスト取得。

#### 7. 合成プロンプトテンプレート

IntentSynthesis 専用の多面的構造: 導入 → 多面的分析（3+側面、各サブ見出し+引用） → 相互関係 → 現状と展望。1200文字以上を要求。

### Clean Architecture レイヤー配置

| コンポーネント | 層 | 理由 |
|--------------|---|------|
| `IntentSynthesis`, `ToolPlan`, `ToolStep`, `ToolDescriptor` | Domain | 純粋な値オブジェクト |
| `TagCloudClient`, `ArticlesByTagClient`, `RecapSearchClient`, `TagExtractorClient`, `SummarizerClient` | Domain (Port) | サービス境界の抽象化 |
| `ToolPlanner`, `SynthesisStrategy`, `HyDEGenerator` | Usecase | ビジネスロジック |
| 各 Tool 実装 | Adapter (Tools) | 外部サービスの Tool 実装 |

## Consequences

### Pros

- **マイクロサービスエコシステムの活用**: search-indexer, tag-generator, alt-backend, pre-processor の能力を RAG パイプラインに統合
- **概念的クエリの回答品質向上**: 複数ファセットからの情報収集により、単一パス検索では得られない広範なカバレッジ
- **LLM 駆動のツール選択**: クエリの性質に応じた柔軟なツール計画
- **既存パイプライン無変更**: IntentSynthesis のみが新パイプラインを使用。他の intent は一切影響を受けない
- **Graceful degradation**: LLM プランニング失敗 → デフォルトプラン、ツール個別失敗 → 他ツール継続、HyDE 失敗 → 空結果で継続
- **テスト駆動**: 全コンポーネントが TDD で実装。45+ テストケース全 PASS

### Cons / Tradeoffs

- **DI 配線未完了**: 新規5ツールの RPC クライアント adapter 実装と `di/container.go` への配線が残存。usecase/domain/adapter(tools) 層の実装とテストは完了しているが、実際の RPC 接続は未統合
- **レイテンシ増加**: IntentSynthesis のパスは合計 15-20 秒（既存比 +5-10 秒）
- **LLM プランニングの品質**: Gemma4 E4B の JSON 構造化出力品質に依存。デフォルトプランでのフォールバック頻度が変動する
- **セマフォ負荷**: ToolPlanner の LLM 呼び出しが news-creator セマフォを追加消費
- **カバレッジ評価の不在**: ファセットカバレッジの定量的評価は将来課題
