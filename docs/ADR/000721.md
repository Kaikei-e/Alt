# ADR-000721: alt-backend の Medium 深刻度 9 件を依存監査・SSRF 整理・DoS ハードニング・セキュリティヘッダ強化で解消する

## Status

Accepted

## Date

2026-04-14

## Affected Services

- alt-backend (Go) — `rest/routes.go` のセキュリティヘッダ・CORS・DoS 配線整理、`middleware/dos_protection_middleware.go` のロック単純化と XFF 信頼境界、`utils/security/ssrf_validator.go` のプレースホルダ削除、`utils/security/url_validator.go` の HTTPS-only モードと metadata 完全一致判定、`rest/security_headers_test.go` / `rest/body_limit_test.go` / `utils/security/medium_fixes_test.go` の追加テスト
- リポジトリインフラ — `.github/workflows/backend-go.yaml` から既存 `reusable-go-quality-gates.yaml` を呼び出して `govulncheck` / `gosec` を SARIF として GitHub Code Scanning に常時アップロード、`scripts/generate-sbom.sh` で CycloneDX SBOM をローカル生成可能に

## Context

[[000717]] と [[000718]] でセキュリティレビュー (`docs/review/alt-backend-security-review-2026-04-14.md`) の Critical 3 件と High 5 件を解消した。残る Medium 9 件 (M-001〜M-009) は次のような性格の課題群だった。

1. **依存関係と SBOM (M-001)**: OWASP Top 10:2025 が新たに SBOM 生成と継続的な依存 CVE 監査を A06 (Vulnerable and Outdated Components) の baseline に含めた。Alt は `reusable-go-quality-gates.yaml` を持っていて gosec と govulncheck を SARIF 出力する基盤は完成していたが、`backend-go.yaml` から呼び出されておらず実質ノーガードだった。さらに SBOM 生成は手動コマンドすら整備されていなかった。

2. **SSRF Validator の二重実装と placeholder (M-002 / M-003 / M-004)**: `utils/security/url_validator.go` (RSS 登録経路) と `utils/security/ssrf_validator.go` (image proxy / RAG 経路) の 2 つが並存していた。前者の metadata 検出は `strings.Contains(host, "metadata")` という素朴な部分一致で、`not-metadata.example.com` のような正当ドメインを誤検出する一方、`metadata.google.internal` 以外の代替ホスト名は擦り抜けるリスクがあった。さらに後者の `NewSSRFValidator` は `allowedDomains: ["example.com"]` というプレースホルダを持ったまま、実際には `validateHost` がこの allow-list を参照していない (コメント "this is a placeholder for now" のまま) ため、読み手に「allowlist が効いている」と誤認させる温床になっていた。HTTP/HTTPS の選別もなく、image proxy 経路でも HTTP がそのまま許可されていた。

3. **DoS 保護の脆弱性 (M-005 / M-006 / M-007)**: `dos_protection_middleware.isWhitelistedPath` は `strings.Contains(path, "/stream") || strings.Contains(path, "/sse")` で広く除外しており、将来 `/v1/feeds/livestream-config` のような route が追加されたら静かに rate limit を擦り抜ける fragile な設計だった。`limiters` map は `CleanupExpiredLimiters` ヘルパが定義されているのに呼び出されておらず、攻撃者が多数の spoofed IP を送り続けるとメモリが無制限に伸びる潜在 DoS 経路になっていた。`getClientIP` は `X-Real-IP` / `X-Forwarded-For` を無条件で信頼するので、edge proxy を迂回して直接 9000 ポートに到達できる経路があれば rate limit 自体が回避可能だった。

4. **セキュリティヘッダの不足 (M-008 / M-009)**: CSP は `default-src 'self'` 1 行のみで `frame-ancestors`、`base-uri`、`form-action`、`report-uri` がすべて欠落していた。alt-backend は HTML を返さない API なので `default-src 'none'` まで詰められる。HSTS の `preload` ディレクティブも未指定、`Referrer-Policy` も未設定だった。CORS では `AllowHeaders` に `X-Alt-Backend-Token` が含まれず、ブラウザが将来 alt-backend を直接呼ぶ preflight が CORS で塞がれる構造になっていた。

これらは個別には Critical/High より影響範囲が小さいものの、量が多く相互に関連する (例: M-005 の path 判定厳格化は H-001 で SSE を消した後の自然な続き、M-007 の trust 境界は M-006 のメモリ枯渇とセットで意味を持つ) ため、まとめて単一の修正サイクルで処理する判断をした。

## Decision

以下の九点を一括適用する。

1. **M-001 (CI で govulncheck/gosec を常時実行 + SBOM)**: `backend-go.yaml` から既存の `reusable-go-quality-gates.yaml` を呼び出す `quality` ジョブを追加する。これにより PR / push のたびに gosec と govulncheck が走り、結果が SARIF として GitHub Code Scanning に上がる。SBOM は CI に組み込まず `scripts/generate-sbom.sh` で開発者が手元で `syft` または `cyclonedx-gomod` を走らせる軽量フォーマットを提供する (CI に組むのは別タスク)。

2. **M-002 (SSRF Validator の HTTPS-only モード)**: `URLSecurityValidator` に `RequireHTTPS(bool)` メソッドを追加。デフォルトは false (RSS feed 登録は HTTP feed が現役で必要) で、image proxy / RAG fetch など plaintext を許容しない経路は呼び出し側で `RequireHTTPS(true)` に切り替えられる。エラーメッセージは "HTTPS scheme is required for this endpoint" と明示する。

3. **M-003 (`allowedDomains` placeholder 削除)**: `NewSSRFValidator()` の `allowedDomains: ["example.com"]` を `allowedDomains: nil` に変更し、コメントで「allowlist が必要なら呼び出し側で `SetAllowedDomains` する」設計意図を明記する。デッドコードと「false sense of safety」を一掃する。

4. **M-004 (metadata 完全一致判定)**: `url_validator.go` 内に `metadataHosts` の `map[string]struct{}` を定義し、`169.254.169.254` / `metadata.google.internal` / `100.100.100.200` / `192.0.0.192` を完全一致でブロックする。`strings.Contains` は廃止。`IsAllowedDomain` 側も同じマップを参照する。

5. **M-005 (path 判定の厳格化)**: `isWhitelistedPath` のロジックを「`/` で終わるエントリは prefix match、それ以外は exact match」に書き換える。`/stream` / `/sse` の substring バイパスは削除し、`WhitelistedPaths` への明示登録だけが効くようにする。これにより将来 route が増えても静かにバイパスされない。

6. **M-006 (rate limiter map のクリーンアップ起動)**: `DOSProtectionMiddlewareWithTrust` 内で `time.Ticker` を起動し、5 分ごとに `CleanupExpiredLimiters(limiters, &mu, 30*time.Minute)` を呼ぶ goroutine を spawn する。30 分間ブロック状態が続いたエントリは GC 対象にする。

7. **M-007 (X-Forwarded-For 信頼境界)**: `getClientIP` を「無条件に XFF を信頼」する関数から、`getClientIPWithTrust(c, trustForwardedHeaders bool)` に分離する。`trustForwardedHeaders=false` 時は XFF / X-Real-IP を完全に無視して `RemoteAddr` のみ参照。新しい `DOSProtectionMiddlewareWithTrust(config, trustForwardedHeaders bool)` を提供し、`routes.go` から `true` を渡す (alt-backend は常に nginx 配下に置く前提)。後方互換のため引数なし版 `DOSProtectionMiddleware` は `trust=true` を維持。

8. **M-008 (セキュリティヘッダ強化)**: `SecureWithConfig` を以下に変更する。
   - `ContentSecurityPolicy: "default-src 'none'; frame-ancestors 'none'; base-uri 'none'; form-action 'self'; report-uri /security/csp-report"`
   - `HSTSPreloadEnabled: true`
   - `ReferrerPolicy: "strict-origin-when-cross-origin"`
   - 既存の `XSSProtection` / `ContentTypeNosniff` / `XFrameOptions: DENY` / `HSTSMaxAge: 31536000` は維持。
   `Permissions-Policy` は edge 層 (nginx) で配信する想定で本 ADR には含めない。

9. **M-009 (CORS AllowHeaders に X-Alt-Backend-Token 追加)**: `CORSConfig.AllowHeaders` に `"X-Alt-Backend-Token"` を追加する。これにより将来フロントエンドが BFF を経由せず直接 alt-backend を叩く経路でも preflight が通る。`AllowCredentials` は今回触れず、用途が明確化されるまで現状維持。

### 検討した代替案

- **M-001 代替**: SBOM を CI で常時生成して artifact として保存する案も検討した。Anchore syft の依存が増え、ジョブ時間が延びる割に直近の運用ニーズが薄いため見送り。手元スクリプトで生成でき、後で CI へ昇格する余地は残した。
- **M-002〜M-004 代替**: `URLSecurityValidator` を完全に廃止して `SSRFValidator` に統合する案。`url_validator.go` には RSS 登録専用の検証ロジックがあり、テストも厚いため、今回は API を最小拡張に留めて二重実装解消は次の Refactor フェーズに先送り。
- **M-005 代替**: `path == whitelistedPath || strings.HasPrefix(...)` だけを残すミニマル修正。将来の SSE/stream エンドポイント追加で再び `/stream` / `/sse` の substring バイパスを書き直したくなるため、`/` 末尾でプレフィクス・それ以外は完全一致という規約を導入することにした。
- **M-006 代替**: `sync.Map` ベースの TTL マップに置き換える案。書き換え量が多くなる割に挙動はほぼ同じになるため、既存 `CleanupExpiredLimiters` を起動するだけの最小変更に留めた。
- **M-007 代替**: Echo の `IPExtractor` (`echo.ExtractIPFromXFFHeader(...)`) に切り替える案。よりキレイだが、既存の `getClientIP` を呼ぶテストが多く、修正範囲を抑えるために独自のラッパで実装した。`IPExtractor` への移行は将来の Refactor で検討する。
- **M-008 代替**: `default-src 'self'` を維持しつつ追加ディレクティブだけ重ねる案。alt-backend は HTML を返さないので `default-src 'none'` まで詰める方が安全側に倒せると判断。
- **M-009 代替**: `AllowHeaders` をワイルドカード `"*"` に切り替える案。`AllowCredentials=true` が将来 enable された時に CORS 仕様上認められないため非推奨。明示列挙を維持した。

## Implementation Overview

### 1. CI / SBOM (M-001)

`.github/workflows/backend-go.yaml` に `quality` ジョブを追加して既存の reusable workflow を呼び出す。

```yaml
jobs:
  test:
    uses: ./.github/workflows/reusable-test-go.yaml
    with:
      working-directory: 'alt-backend/app'
      coverage-flags: 'alt-backend'

  quality:
    permissions:
      contents: read
      security-events: write
    uses: ./.github/workflows/reusable-go-quality-gates.yaml
    with:
      working-directory: 'alt-backend/app'
      lint-only-new: true
      upload-sarif: true
```

`scripts/generate-sbom.sh` は `syft` を優先し、見つからなければ `cyclonedx-gomod` にフォールバックする実装で、`./sbom/<service>-<commit>.cdx.json` に CycloneDX JSON を吐く。`alt-backend` をデフォルト対象とし、引数で他サービスも指定可能にしている。

### 2. SSRF Validator の整理 (M-002 / M-003 / M-004)

```go
// utils/security/url_validator.go
type URLSecurityValidator struct {
    requireHTTPS bool
}

func (v *URLSecurityValidator) RequireHTTPS(require bool) {
    v.requireHTTPS = require
}

var metadataHosts = map[string]struct{}{
    "169.254.169.254":          {},
    "metadata.google.internal": {},
    "100.100.100.200":          {},
    "192.0.0.192":              {},
}

func (v *URLSecurityValidator) ValidateRSSURL(rawURL string) error {
    // ... length / dangerous patterns / parse ...
    if v.requireHTTPS && parsedURL.Scheme != "https" {
        return errors.New("HTTPS scheme is required for this endpoint")
    }
    hostname := strings.ToLower(parsedURL.Hostname())
    if _, isMetadata := metadataHosts[hostname]; isMetadata {
        return errors.New("metadata server access denied")
    }
    // ... isPrivateNetwork ...
}
```

`SSRFValidator` 側は `NewSSRFValidator` の `allowedDomains: ["example.com"]` を `nil` にする 1 行修正のみ。テストは `utils/security/medium_fixes_test.go` に HTTPS-only / placeholder 削除 / metadata 完全一致 / 部分一致誤検出回避の 5 ケースを追加。

### 3. DoS 保護のハードニング (M-005 / M-006 / M-007)

```go
// middleware/dos_protection_middleware.go
func DOSProtectionMiddlewareWithTrust(config DOSProtectionConfig, trustForwardedHeaders bool) echo.MiddlewareFunc {
    // ... validation, limiters map ...

    // M-006: cleanup goroutine
    go func() {
        ticker := time.NewTicker(5 * time.Minute)
        defer ticker.Stop()
        for range ticker.C {
            CleanupExpiredLimiters(limiters, &limiterMu, 30*time.Minute)
        }
    }()

    return func(next echo.HandlerFunc) echo.HandlerFunc {
        return func(c echo.Context) error {
            path := c.Request().URL.Path
            if isWhitelistedPath(path, config.WhitelistedPaths) {
                return next(c)
            }
            // ... circuit breaker ...
            clientIP := getClientIPWithTrust(c, trustForwardedHeaders)
            // ... rate limit ...
        }
    }
}

func isWhitelistedPath(path string, whitelistedPaths []string) bool {
    for _, w := range whitelistedPaths {
        if strings.HasSuffix(w, "/") {
            if strings.HasPrefix(path, w) {
                return true
            }
            continue
        }
        if path == w {
            return true
        }
    }
    return false
}

func getClientIPWithTrust(c echo.Context, trustForwardedHeaders bool) string {
    if trustForwardedHeaders {
        // X-Real-IP / X-Forwarded-For を考慮
    }
    if ip, _, err := net.SplitHostPort(c.Request().RemoteAddr); err == nil {
        if net.ParseIP(ip) != nil {
            return ip
        }
    }
    return ""
}
```

`routes.go` から `DOSProtectionMiddlewareWithTrust(cfg, true)` を呼び、edge proxy 配下である前提を明示。テストには「substring バイパスが消えたこと」「cleanup が古いエントリを除去すること」「trust=false で XFF が無視されること」の 3 観点を追加。

### 4. セキュリティヘッダ強化 (M-008)

```go
// rest/routes.go
e.Use(middleware.SecureWithConfig(middleware.SecureConfig{
    XSSProtection:         "1; mode=block",
    ContentTypeNosniff:    "nosniff",
    XFrameOptions:         "DENY",
    HSTSMaxAge:            31536000,
    HSTSPreloadEnabled:    true,
    ContentSecurityPolicy: "default-src 'none'; frame-ancestors 'none'; base-uri 'none'; form-action 'self'; report-uri /security/csp-report",
    ReferrerPolicy:        "strict-origin-when-cross-origin",
}))
```

`rest/security_headers_test.go` に CSP 強化 / Referrer-Policy 設定 / HSTS preload (TLS request 時のみ emit) の 3 ケースを追加。`/security/csp-report` ハンドラは [[000717]] 時点で既に CSP report を受信できる状態にあり、本 ADR で CSP の `report-uri` がそこへ向くようになった。

### 5. CORS AllowHeaders (M-009)

```go
// rest/routes.go
e.Use(middleware.CORSWithConfig(middleware.CORSConfig{
    AllowOrigins: cfg.Server.CORSAllowedOrigins,
    AllowMethods: []string{echo.GET, echo.POST, echo.PUT, echo.DELETE, echo.OPTIONS},
    AllowHeaders: []string{
        echo.HeaderOrigin,
        echo.HeaderContentType,
        echo.HeaderAccept,
        "Cache-Control",
        "Authorization",
        "X-Requested-With",
        "X-CSRF-Token",
        "X-Alt-Backend-Token", // 追加
    },
    MaxAge: 86400,
}))
```

テストは preflight (OPTIONS) で `Access-Control-Allow-Headers` に `x-alt-backend-token` (大文字小文字無視) が含まれることを assert。

### 6. ランタイム検証 (smoke)

`docker compose -f compose/compose.yaml -p alt up --build -d alt-backend` で alt-backend を再ビルド・起動。`healthy` で安定動作することを確認した上で、以下の HTTP smoke を実行した。

- `curl -sI http://localhost:9000/v1/health` → `Content-Security-Policy: default-src 'none'; frame-ancestors 'none'; base-uri 'none'; form-action 'self'; report-uri /security/csp-report` と `Referrer-Policy: strict-origin-when-cross-origin` が返ることを確認。
- `curl -sI -X OPTIONS http://localhost:9000/v1/feeds/search -H "Origin: <許可 origin>" -H "Access-Control-Request-Method: POST" -H "Access-Control-Request-Headers: x-alt-backend-token"` → `Access-Control-Allow-Headers` に `X-Alt-Backend-Token` が含まれることを確認。
- `go test ./... -count=1` (alt-backend) で全モジュール GREEN。

## Consequences

### Pros

- **Medium 9 件が同日中にクローズ**: レビュー報告残 5 件 (Low) 以外はすべて解消され、Critical / High / Medium が 17/17 修正済となった。
- **CI で依存 CVE が継続的に検出される**: PR ごとに `govulncheck` / `gosec` が走るため、新規 CVE の混入と既存 CVE の見落としが大幅に減る。SARIF が Code Scanning に上がるので GitHub UI で履歴も追える。
- **SSRF の防御線が単純化**: metadata 検出が完全一致になり、`not-metadata.example.com` のような正当ホストが誤検出されなくなった。プレースホルダの `allowedDomains` が消えたことで「allow-list が効いていない」という認知ギャップも閉じた。
- **DoS 保護の設計意図が明示化**: `WhitelistedPaths` は明示登録のみ効くようになり、将来 route 名の偶然一致でバイパスが起きない。`trustForwardedHeaders` フラグで「edge proxy 配下である」前提が型システム上に表現された。`limiters` map のメモリ枯渇経路も潰れた。
- **HTML を返さない API としての CSP が完成**: `default-src 'none'` まで詰め切ったことで、万一 alt-backend からインジェクションでスクリプトが返っても実行されない設計になった。`report-uri` 付きなので CSP 違反の発生も追跡できる。
- **将来のフロントエンド直結に備えた CORS**: `X-Alt-Backend-Token` が CORS preflight を通るようになり、BFF 経由でない経路を作る場合のブロッカーが先回りで解消された。
- **TDD 維持**: 全ての修正で RED テスト先行 → GREEN を守り、回帰リスクを最小化した。

### Cons / Tradeoffs

- **CI の所要時間が増える**: `quality` ジョブが追加されるため backend-go ワークフローのクリティカルパスが伸びる (typically +1〜2 分)。SARIF アップロードが遅延する場合は `lint-only-new: true` で差分のみに絞り影響を抑えている。
- **`URLSecurityValidator` と `SSRFValidator` の二重実装は残る**: 完全な統合は本 ADR の scope 外。現状は「呼び出し側ごとに使い分け」が必要で、将来 Refactor で 1 本化したい。
- **`X-Forwarded-For` が信頼できない経路 (例: 将来直接公開する API ポート) では明示的に `false` を渡す必要がある**: `DOSProtectionMiddleware` 後方互換で `true` がデフォルトなので、新規エンドポイントを直接外部に晒す場合は `WithTrust(cfg, false)` を選ぶ規律が必要。
- **CSP `default-src 'none'` でブラウザから直接 alt-backend を叩いた時のエラーページ等が完全に空になる**: API は基本 JSON 返却なので影響は小さいが、将来 HTML を返したくなったら CSP を経路別に上書きする必要がある。
- **SBOM は CI で常時生成しない**: 開発者が忘れると SBOM が古いまま残る。リリース時に CI 化することを将来課題として残す。
- **Permissions-Policy は本 ADR で扱わない**: edge 層 (nginx) で配信する想定としているが、現時点では nginx 側にも未配置。次の改善対象とする。
- **Low 5 件は依然未対応**: Dockerfile タグ pin (L-001)、dashboard logs limit (L-002) ほか、レビュー報告の優先修正リスト末尾 5 件は別タスクで処理する。

### 将来の検討課題

- CI への SBOM 生成・保管の組込み (リリース artifact として `*.cdx.json` を同梱)。
- `URLSecurityValidator` を `SSRFValidator` に完全統合する Refactor。
- Echo の `IPExtractor` への移行 (M-007 のクリーンな実装)。
- `Permissions-Policy` ヘッダの追加 (edge 層もしくは backend 層)。
- `AllowCredentials` / cookie ベース認証の導入有無を含めた CORS ポリシーの最終決定。
- Low 5 件 (L-001〜L-005) の段階的対応。
