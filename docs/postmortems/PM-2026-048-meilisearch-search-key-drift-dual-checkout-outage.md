## サマリー

本番環境では、同一 compose project に対して「対話用チェックアウト」と「CI/CDデプロイランナー用チェックアウト」の2系統が存在し、compose secrets が相対パス（`../secrets/*.txt`）で定義されているため、どちらから `docker compose` を起動するかによって実際に読み込まれる secrets ファイルの実体が分岐する構造になっていた。2026-07-16 に [[PM-2026-047]] の復旧対応で Meilisearch のボリュームが再作成された際、内部の Default Search API Key（マスターキーから HMAC 導出）が新値にローテーションされたが、対話用チェックアウト側の secrets ファイルのみが追従更新され、デプロイランナー側は旧値のまま取り残された。2026-07-22 21:47 JST、[[PM-2026-047]] の恒久対応3コミットをデプロイランナー経由でデプロイした際に search-indexer が再作成され、取り残されていた旧検索キーをマウントしたことで、起動約3秒後から本番の全文検索クエリが `403 invalid_api_key` で全滅した。障害はアラートに接続されておらず、約24時間後に別サービス（Acolyte）の検証run失敗調査から偶然発覚し、オペレータがマスターキーをローテーションして復旧した。復旧作業中には、コンテナ再作成に伴う既知の pki-agent サイドカーの netns 孤立パターン（[[PM-2026-030]] / [[PM-2026-034]] 系列）も副次的に再発した。

## 影響

- **影響を受けたサービス:** Meilisearch（本番、検索用APIキー経路）、search-indexer（warmup probe / 実クエリ）、alt-backend 経由の全文検索、Acolyte のエビデンス検索（[[PM-2026-049]] 参照）
- **影響を受けたユーザー数/割合:** 定量化できず。障害時間中の全検索クエリが失敗した（実クエリ8回の失敗を確認、warmup probe は291回連続失敗）。インデクシング（書き込み）はマスターキーを使用する経路のため無傷で、障害中も ingest は継続した。ユーザーから見た「新着記事が増えない」という問題は発生していない
- **機能への影響:** 全文検索機能の完全停止（`403 invalid_api_key`）。Acolyte レポート生成のエビデンス検索も同一キー経路に依存しており道連れで失敗した
- **データ損失:** なし
- **SLO/SLA違反:** 検索可用性に関する明示的な SLO は未定義（[[PM-2026-047]] のアクションアイテム#8が未着手のまま）。定義されていれば24時間の完全停止は明確な違反となる水準
- **定量データ:** warmup probe 失敗291回連続（5分間隔 × 291 ≈ 24.25時間）、実クエリ失敗8回確認、TTD ≈ 24時間、TTR ≈ 24.5時間

## 検知

- **検知方法:** Acolyte レポート生成の検証runが "All evidence searches failed" で失敗し、その原因調査から search-indexer の403エラーに遡って発覚した。監視・アラートによる検知ではない
- **検知までの時間 (TTD):** 約24時間（2026-07-22 21:47:44 障害発生 → 2026-07-23 21:48 検知）
- **検知の評価:** warmup probe は [[PM-2026-047]] のアクションアイテムを受けて起動時1回から5分間隔の定期実行に改修されたばかりで、これがなければログ上の可視性はさらに低かった。しかし、この probe の失敗はログに残るのみでアラートには一切接続されておらず、291回連続失敗という明確な異常シグナルが24時間活用されなかった。検知は本質的に「たまたま別サービスの障害を調査していたら見つかった」という偶発的なものであり、監視の設計としては機能していなかった。

## 根本原因分析

### 直接原因

search-indexer がデプロイランナー側チェックアウトから再作成された際、`compose/base.yaml` の相対パス secrets が解決する実ファイル（デプロイランナー側の `../secrets/meili_search_key.txt`）が、Meilisearch が実際に発行している Default Search API Key と一致していなかった（2026-07-16時点で追従更新されなかった旧値のままだった）ため。

### Five Whys

1. **なぜ search-indexer の全検索クエリが403 invalid_api_keyになったか？** → マウントされた検索用APIキーの値が、Meilisearch側が実際に発行しているDefault Search API Keyと一致していなかったため。
2. **なぜマウントされたキーの値が古いままだったか？** → デプロイランナー側チェックアウトの `secrets/meili_search_key.txt` が、2026-07-16のMeilisearch auth DB再初期化（[[PM-2026-047]]対応）に追従して更新されなかったため。対話用チェックアウト側のみ手動で更新され、デプロイランナー側は更新対象から漏れていた。
3. **なぜ2系統のチェックアウトが独立したsecretsファイルを持ち、片方だけが更新されて分岐する状態が生じたか？** → compose secretsが相対パス（`../secrets/*.txt`）で定義されており、`docker compose` の起動元ディレクトリ（対話用チェックアウトかデプロイランナー用チェックアウトか）によって実際に読み込まれるファイルの実体が変わる設計になっているため。同一のcompose projectを指していても、secretsの「実体」はチェックアウトごとに別物になり得る。
4. **なぜこの分岐が2026-07-16時点で問題化せず、2026-07-22のデプロイまで6日間気づかれなかったか？** → デプロイランナー側チェックアウトからのコンテナ再作成が発生しない限り、旧キーはマウントされず無症状のまま。search-indexerがデプロイランナー経由で再作成されるのは変更をデプロイする時だけであり、6日間その機会がなかったため潜伏した。
5. **なぜsecretsの値を2系統のチェックアウト間で同期する仕組みが存在しなかったか？** → デプロイパイプライン設計上、secretsの「単一の正本（source of truth）」が定義されておらず、各チェックアウトのローカルファイルがそれぞれ独立に「正しい」ものとして扱われていた。Meilisearchのようにコンテナ再作成時にキーが再生成されうるコンポーネントに対して、secrets同期をデプロイフローに組み込む設計判断がなされていなかったため。

### 根本原因

compose secretsの相対パス参照により、同一compose projectに対して「対話用」「デプロイランナー用」という実行コンテキストの異なる2つのチェックアウトが、それぞれ独立したsecretsファイル実体を持つ構造になっていた。Meilisearchはコンテナ再作成のたびに内部でAPIキーを再生成しうるコンポーネントであるにもかかわらず、secretsを2系統間で同期する仕組みがデプロイパイプラインに組み込まれておらず、「たまたま値が一致している」ことに可用性が依存していた。この構造的な単一障害点が、[[PM-2026-047]]の復旧作業でのキーローテーションをトリガーとして顕在化した。

### 寄与要因

- [[PM-2026-047]]で判明した「Meilisearchのボリューム削除は内部生成APIキー・embedder設定といった暗黙の状態を巻き込む」という教訓が、secrets同期の仕組み化にはまだ反映されていなかった。同ポストモーテムのアクションアイテム#5（runbook作成）は着手されていたが、根本的な同期機構の導入はスコープ外だった
- Meilisearchのマスターキーが、対応中の調査で判明した通り既知のデフォルトプレースホルダ値のまま本番稼働していた。これ自体は今回の直接原因ではないが、「デフォルト値のまま気づかれずに稼働し続ける」という同種の検知欠如パターンが、今回の障害の伏線として存在していたことを示す
- warmup probeがアラートに未接続だったため、291回という明確な異常シグナルが24時間活用されなかった
- コンテナ再作成に伴うpki-agentサイドカーのnetns孤立という既知の罠（[[PM-2026-030]] [[PM-2026-034]]で計4回発火済み）が今回の復旧作業でも再発し、対応をわずかに複雑化させた。恒久対応（[[000802]] の `probeNetns`）は検知のみを提供し、自己治癒は別リポジトリの alt-deploy cascade（[[000783]]）に委譲されたままであるため、手動 `--force-recreate` が依然として必要だった

## 対応の評価

### うまくいったこと

- [[PM-2026-047]]対応で導入されたばかりのwarmup probe定期実行（5分間隔）により、障害発生から検知までの間、少なくともログ上には291件の失敗証跡が残っていた。もしこの改修が入っていなければ、起動時1回のprobeのみとなり可視性はさらに低かった
- インデクシング経路がマスターキーを使用していたため、検索障害中もingestは正常継続し、記事データの欠落・遅延は発生しなかった
- 原因特定から復旧までは迅速だった（マスターキーローテーションの判断から2系統への同期、コンテナ再作成、warmup probe正常応答の確認まで約3分程度）

### うまくいかなかったこと

- 障害発生（21:47:44）から検知（翌21:48）まで丸一日、本番の全文検索が完全に機能停止していたにもかかわらず、誰も気づかなかった。検知は監視ではなく、無関係な別サービスの障害調査という偶然に依存していた
- 2系統のチェックアウトのsecretsが独立して分岐しうる構造が、[[PM-2026-047]]で一度キーローテーションを経験した後も是正されないまま残っていた。「次に同じことが起きたらまた壊れる」ことが実質的に予見可能な状態だった
- マスターキーがデフォルトのプレースホルダ値のまま本番稼働していたことが、今回の対応中に初めて発覚した。起動時のfail-fast検証があれば、この状態自体をより早期に是正できていた可能性がある

### 運が良かったこと

- Acolyteのレポート生成検証runがたまたま実行され、その失敗調査という別経路から本障害が発見された。もしこの検証runがなければ、次に人間が検索機能を直接使うまで、あるいは今後整備される全文検索SLO監視が入るまで潜伏が続いていた可能性が高い
- インデクシング経路がマスターキーを使う設計だったため、検索障害がデータの欠落や不整合には発展しなかった

## 教訓

- secretsの「同一に見えるが実は分岐しうる」二重化は、キーローテーションのようなイベント時に自己閉塞的な障害を生む。相対パス参照による暗黙の分岐は、明示的な同期機構なしには維持できない。
- 監視ログに証跡が残っていても、アラートに接続されていなければ「検知」にはならない。291回の連続失敗はログとしては十分な情報量だったが、誰も見ていなければ意味を持たない。
- デフォルトのプレースホルダ値が本番で気づかれず稼働し続けることは、それ自体が独立した検知欠如の兆候であり、他の潜伏中の設定ミスの存在を示唆する。
- ある障害（[[PM-2026-047]]）の復旧作業自体が、別の潜在的な障害条件（secrets分岐）を作り出すことがある。復旧手順の中で「この操作が他に何を変えるか」を都度チェックする習慣が必要。
- 同型の障害パターン（pki-agentのnetns孤立）は、検知の仕組み（`probeNetns`）だけを整備しても自己治癒までは到達しない設計だと、再発のたびに手動対応が必要になり続ける。
