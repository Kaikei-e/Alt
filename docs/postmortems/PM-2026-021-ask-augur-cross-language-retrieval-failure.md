## サマリー

[[000693]]（embedding degraded mode）の修正後、embedder が復旧した状態で Augur に「ヴァンス副大統領の直近の動きは？」を送信したところ、Meilisearch に 372 件の Vance 関連英語記事が存在するにもかかわらず「Not enough indexed evidence was available yet」と fallback が発動した。リトリーバルパイプライン自体は正常動作しており、10 件のコンテキストを取得していたが、取得された記事が全てクエリと無関係（Iran, oil, Trump 関連）で RelevanceGate が品質不足と正しく判定した。原因は 3 サービスに分散した言語横断検索の設計不備で、日本語クエリが英語記事に到達できないパスが複数存在していた。

## 影響

- **影響を受けたサービス:** rag-orchestrator, news-creator, search-indexer（3 サービスに分散した設計不備）
- **影響を受けたリクエスト:** 日本語クエリで英語記事を検索する全ての Ask Augur リクエスト（潜在的）
- **機能への影響:** Ask Augur が日本語クエリに対して英語記事を引用した回答を生成できない。RelevanceGate が正しく fallback を発動するため、誤った回答は生成されない
- **データ損失:** なし
- **他機能への影響:** なし

## 根本原因分析

### 直接原因

3 サービスに分散した 3 つの独立した設計不備により、日本語クエリから英語記事への到達パスが全て遮断されていた。

### Five Whys

1. **なぜ Augur が「ヴァンス副大統領」に回答できなかったか？**
   → RelevanceGate がコンテキストの品質不足と判定し、正しく fallback を発動した

2. **なぜ取得された 10 件のコンテキストが全て無関係だったか？**
   → ベクトル検索は日本語クエリの embedding で検索したが、英語 Vance 記事との cosine similarity が低かった（0.42-0.48）。BM25 は日本語クエリ「ヴァンス副大統領」で 1 件しかヒットしなかった

3. **なぜ BM25 が英語 Vance 記事を見つけられなかったか？**
   → 3 つの障害が重なった:
   - (a) expand-query が英語翻訳を生成せず、BM25 に "JD Vance" クエリが渡らなかった
   - (b) BM25 検索が元クエリのみで実行され、仮に翻訳があっても使われなかった
   - (c) Meilisearch が CJK クエリを jpn ロケールに限定し、英語トークンとのマッチングを無効化していた

4. **なぜ expand-query が英語翻訳を生成しなかったか？**
   → プロンプトテンプレートが `Japanese({count}):` ラベルのみで終わり、`English({count}):` ラベルがなかった。Gemma4 は日本語セクション出力後に生成を停止した

5. **なぜ言語横断検索の設計不備が見過ごされていたか？**
   → 各サービスが独立して開発・テストされており、「日本語クエリ→英語記事」のエンドツーエンドテストが存在しなかった。各サービスは自分の責務範囲で正しく動作していたが、組み合わせた際のパスが検証されていなかった

### 寄与要因

- **query planner の search_queries 未活用**: query planner は英語を含む 4 件の search_queries を生成していたが、これらは BM25 検索に渡されず、メタデータとしてのみ保存されていた
- **embedding モデルの言語横断性能の限界**: embeddinggemma は日本語→英語のクロスリンガル embedding 品質が高くなく、semantic 検索だけでは不十分だった
- **containsCJK ロケール制限の過剰な保守性**: 本来は日本語テキストのトークン化精度向上のために導入したロケール指定が、英語記事の発見を阻害する副作用を持っていた

## 対応の評価

### うまくいったこと

- **3 サービス並行修正**: news-creator (Python), rag-orchestrator (Go), search-indexer (Go) の 3 サービスを並行して TDD で修正し、25 分で復旧完了
- **定量的なデバッグ**: Meilisearch 直接クエリ（"Vance" → 372 件 vs "ヴァンス副大統領" → 1 件）で即座に言語横断問題を数値で確認
- **RelevanceGate の正常動作**: 無関係コンテキストでの誤答生成を防止し、ユーザーに不正確な情報を提供しなかった

### 運が良かったこと

- **embedder 障害調査の延長で発見**: embedder 復旧後のテストで即座に発見された。通常運用では「日本語クエリへの英語記事の欠如」は気づきにくい問題

### 改善が必要なこと

- **言語横断の統合テスト不在**: 各サービスのユニットテストは充実しているが、日本語→英語の検索パスを検証するエンドツーエンドテストがなかった
- **query planner 結果の BM25 連携**: planner が生成した bilingual search_queries が BM25 に渡されない設計上の断絶

## 教訓

### 技術的な学び

- **言語横断検索は「掛け算」で壊れる**: 単一の障害ではなく、プロンプト・検索ロジック・トークナイザの 3 つが独立して失敗し、掛け合わされることで完全に機能しなくなった。各層が独立してテストされていても、組み合わせた際のパスが検証されていなければ品質は保証されない
- **BM25 の言語横断能力は明示的に設計すべき**: キーワード検索は本質的に同一言語内でのマッチングに強い。異なる言語間の検索は、翻訳クエリの生成と多言語トークナイザの組み合わせで明示的に設計する必要がある
- **LLM プロンプトの出力構造は小モデルの停止条件を決定する**: Gemma4 のような小モデルは、プロンプトの末尾構造をそのまま出力の終了シグナルとして解釈する。`Japanese:` ラベルだけでは English セクションの存在を推論できない

### 組織的な学び

- **マイクロサービス間の境界をまたぐ品質属性の統合テスト**: 言語横断検索のような品質属性は、単一サービスのテストでは検証できない。サービス横断のシナリオテストが必要
