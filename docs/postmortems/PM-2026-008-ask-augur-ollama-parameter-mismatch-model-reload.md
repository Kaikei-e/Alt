# PM-2026-008: Ask Augur チャットストリームの Ollama パラメータ不一致によるモデルリロード遅延

## メタデータ

| 項目 | 値 |
|------|-----|
| 重大度 | SEV-3（主要機能の深刻なパフォーマンス劣化、ただし完全停止ではない） |
| 影響期間 | [[000567]] 適用後 〜 2026-03-25（構造的問題のため、バッチ要約と Ask Augur が同時利用される期間は常に再現可能） |
| 影響サービス | news-creator |
| 影響機能 | Ask Augur（RAG ベースの対話型回答） |
| 関連 ADR | [[000567]], [[000571]], [[000579]] |

## サマリー

PM-2026-006 の対応として [[000567]] を適用し、Ask Augur のチャットストリームを news-creator の HybridPrioritySemaphore 経由にルーティングした。セマフォによる優先制御は正しく動作し、chat は HIGH PRIORITY で即時取得（0s wait）できていた。しかし、chat の TTFT（Time to First Token）が最大 **259 秒** に達する事象が継続して発生した。

原因は、chat リクエストとバッチ要約リクエストで LLM 推論サーバーに送信される `options` パラメータが異なっていたこと。パラメータ不一致により、リクエストの種別が切り替わるたびに推論サーバーがモデルをアンロード→再ロードする「ピンポン」状態が発生し、chat がバッチ要約の完了後のモデルリロードまで完全にブロックされていた。

## 影響

- **Ask Augur TTFT**: 最大 **259 秒**（正常時 1.6 〜 9.4 秒）
- **影響の再現条件**: バッチ要約が GPU を使用中に Ask Augur へクエリを送信した場合に発生。バッチ要約が高頻度で実行される時間帯では高確率で再現

- **バッチ要約への影響**: なし（バッチ要約は正常に完了し続けていた）
- **Morning Letter**: chat ハンドラーが非ストリーミングを拒否（HTTP 400）しており、未対応状態だった
- **データ損失**: なし

## 根本原因分析

### Five Whys

1. **なぜ Ask Augur の TTFT が 259 秒に達したか？**
   → chat リクエストが LLM 推論サーバーの内部キューでバッチ要約の完了を待っていたため。`NUM_PARALLEL=2` でも同一 runner 上での並列実行ができなかった

2. **なぜ同一 runner で並列実行できなかったか？**
   → chat とバッチ要約のリクエストで、LLM 推論サーバーに送信される `options` パラメータが異なっていた。パラメータ不一致により runner の再構成（モデルのアンロード→再ロード）が必要だった

3. **なぜ options が異なっていたか？**
   → バッチ要約パスは `config.get_llm_options()` で `num_batch: 1024`, `num_keep: -1`, `stop: ["<end_of_turn>"]` を含む全パラメータを送信していたが、chat パスは rag-orchestrator から受け取った options をそのままパススルーしており、これらのパラメータが含まれていなかった

4. **なぜ chat パスが options をパススルーしていたか？**
   → [[000567]] で chat プロキシを実装した際、「透過的なプロキシ」を設計方針とし、リクエストの形式を変更しないことを意図していた。GPU 設定パラメータの一致が並列実行の前提条件であることが認識されていなかった

5. **なぜパラメータ一致の必要性に気づかなかったか？**
   → LLM 推論サーバーの内部動作（パラメータ不一致時の runner 再構成）に関する知見が不足していた。推論サーバーのドキュメントにもこの動作の明示的な記載がなく、実運用のログ分析で初めて判明した

### 寄与要因

- **長時間のバッチ要約ジョブ**: プロンプト長 19,581 文字の要約が 6 分 29 秒かかり、chat のブロック時間が極端に長くなった
- **BatchSize のピンポン**: chat（BatchSize 未指定 → サーバーデフォルト 512）とバッチ要約（BatchSize 1024）の切り替えごとにモデルが再ロードされ、各リロードに 3〜5 秒のオーバーヘッドが加算された
- **非ストリーミング chat の未対応**: Morning Letter 等の非ストリーミング chat が HTTP 400 で拒否されており、機能欠損があった

## 対応の評価

### うまくいったこと

- rag-orchestrator の構造化ログ（`ollama_chat_stream_started`, `ollama_chat_stream_first_chunk`）により TTFT の定量的な把握が容易だった
- LLM 推論サーバーの GIN ログと runner ログ（`load request`, `BatchSize`, `loaded runners`）の突合せにより、モデルリロードの発生とパラメータ差異を正確に特定できた
- 既存の `config.get_llm_options()` インフラを再利用でき、`_merge_options()` メソッドの追加という最小限の変更で全パスを統一できた
- 307 件の既存テストが全て PASS し、回帰なしでデプロイできた

### 改善が必要なこと

- [[000567]] の実装時に、chat と batch で LLM 推論サーバーに送信されるパラメータの一致を検証すべきだった。「透過的プロキシ」の設計方針が、GPU 設定パラメータの一致という暗黙の前提を見落とす原因になった
- LLM 推論サーバーのパラメータ不一致によるモデルリロードを検知するメトリクスが存在しなかった。現状は手動のログ分析に依存している
- Ask Augur の TTFT を定常的に監視するダッシュボードやアラートがなく、問題の検知がユーザー報告に依存していた

### 運が良かったこと

- Retrieval フェーズ（ベクトル検索・Reranking）は独立して正常動作しており、chat の検索精度自体は影響を受けなかった
- バッチ要約は正常に完了し続けていたため、要約カバレッジへの影響はなかった
- 修正が news-creator 内で完結し、rag-orchestrator のコード変更が不要だった

## 教訓

### 技術的教訓

1. **LLM 推論サーバーの並列実行は options の一致が前提条件**: `NUM_PARALLEL` で複数スロットが確保されていても、options（特に `num_batch`, `num_ctx`）が異なるリクエストは同一 runner で処理できない。全クライアントが同一の GPU 設定パラメータを送信すべき
2. **「透過的プロキシ」は GPU 共有環境では危険**: リクエストをそのままパススルーすると、クライアントごとの設定差異が GPU レベルの再構成を引き起こす。プロキシ層で正規化すべき
3. **ベースオプションの一元管理が有効**: `config.get_llm_options()` をベースとし、caller 固有の値（`num_predict` 等）のみをオーバーレイするパターンにより、パラメータの一貫性を保証しつつ柔軟性も維持できる

### 組織的教訓

1. **GPU 共有リソースの変更時は全クライアントの options を検証すべき**: [[000567]] でルーティングを変更した際、options の一致確認がチェックリストに含まれていなかった
2. **推論サーバーの内部動作に関する知見を蓄積すべき**: runner 再構成の条件、KV cache の動作、並列スロットのスケジューリング等の知見がドキュメント化されていなかった
