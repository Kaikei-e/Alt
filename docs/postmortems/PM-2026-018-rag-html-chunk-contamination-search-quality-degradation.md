## サマリー

Ask Augur に対してトピック検索を行っても「提供されたコンテキストには情報が含まれていません」と返される事象が報告された。調査の結果、alt-db に格納された fulltext-fetch 記事の生 HTML が rag-orchestrator の chunker を素通りし、rag-db のチャンクにナビゲーション・サイドバー・利用規約等のボイラープレートが混入していた。全 43,658 記事の約 26%（11,222 件）が `<div>` 開始の生 HTML であり、embedding ベクトルが記事内容ではなくサイト構造を表現する状態となっていた。根本原因は、chunker がテキスト抽出を行わずに `\n\n` 分割のみを適用していたことにある。

## 影響

- **影響を受けたサービス:** rag-orchestrator（検索パイプライン全体）、Ask Augur（回答品質）
- **影響を受けたドキュメント数:** rag-db 全 34,069 ドキュメントのうち、HTML 含有記事は推定 26%（約 8,858 件）
- **機能への影響:** 部分的劣化 — fulltext-fetch 由来の記事に対する RAG 検索が実質的に機能しない。RSS フィード由来の plain text 記事は影響なし
- **データ損失:** なし（alt-db の原本は保持されており、rag-db はインデックスとして再構築可能）
- **SLO/SLA 違反:** なし

## 検知

- **検知方法:** ユーザーによる手動確認（「記事があるはずなのに見つけられない」）
- **検知までの時間 (TTD):** 不明（構造的問題のため、fulltext-fetch 導入時から潜在的に存在していた可能性が高い）
- **検知の評価:** チャンク品質の自動監視が存在しなかった。rag-db にインデックスされたチャンクの内容品質を定期的にチェックする仕組みがあれば、より早期に発見できた

## 根本原因分析

### 直接原因

rag-orchestrator の chunker (`domain/chunker.go` V8) が入力テキストに対して HTML タグ除去を行わず、`\n\n` による段落分割のみを適用していた。fulltext-fetch で取得されたページ全体の HTML（ナビゲーション、サイドバー、利用規約を含む）がそのままチャンクに格納され、embedding ベクトルが記事内容ではなくサイト構造を表現する状態となっていた。

### Five Whys

1. **なぜ RAG 検索で記事が見つからなかったか？**
   → rag-db のチャンク embedding がクエリの意味と一致しなかったため。BM25 検索も希釈されていた

2. **なぜ embedding がクエリと一致しなかったか？**
   → チャンクの内容が記事本文ではなく、HTML タグ・ナビゲーション・ボイラープレートで構成されていたため。embedding ベクトルが「ニュースサイトのページ構造」を表現していた

3. **なぜチャンクに HTML が含まれていたか？**
   → chunker が HTML タグ除去を行わず、入力テキストをそのまま `\n\n` で分割していたため

4. **なぜ chunker に HTML 除去機能がなかったか？**
   → chunker は RSS フィード由来の plain text を前提に設計されていた。fulltext-fetch の導入により入力フォーマットが変化したが、chunker 側の対応が漏れていた

5. **なぜ入力フォーマットの変化が検知されなかったか？**
   → チャンクの内容品質を監視する仕組み（HTML タグ混入率、テキスト/ノイズ比率等）が存在しなかったため。インデックス処理は「チャンク数 > 0」で成功とみなされており、内容の品質は検証されていなかった

### 根本原因

rag-orchestrator の chunker が入力テキストのフォーマットを仮定しており（plain text 前提）、上流のデータフォーマット変更（fulltext-fetch による HTML 導入）に対する耐性がなかった。加えて、チャンク内容の品質を自動監視する仕組みが不在であり、品質劣化が長期間検知されなかった。

### 寄与要因

1. **alt-db のスキーマ**: `articles.content` カラムが plain text と HTML の両方を同一カラムに格納しており、フォーマットの区別がない
2. **backfill の成功判定**: HTTP 200 + `{"status":"indexed"}` のみで成功とし、チャンク内容の品質を検証していなかった
3. **3/28 の診断ミス**: [[augur-oil-crisis-short-answer-remediation-2026-03-28|oil-crisis 短文回答 remediation プラン]] では intent routing と retrieval quality gating の問題として分析され、チャンクの内容品質には踏み込まなかった

## 対応の評価

### うまくいったこと

1. **データベース直接調査**: ユーザーの「alt-db を見て」という指示に従い、alt-db と rag-db の両方を直接クエリしたことで、記事の存在とチャンク品質の乖離を迅速に発見できた
2. **TDD による迅速な修正**: sanitizer + chunker V9 + idempotency 拡張を 30 分以内に TDD で実装・デプロイできた
3. **ChunkerVersion による自動再インデックス**: idempotency 判定に ChunkerVersion を追加したことで、既存記事を backfill で自動的に V9 に洗い替えできる設計にできた
4. **段階的復旧**: 1 記事の手動検証 → 部分 backfill → 全期間 backfill の段階的アプローチで、各ステップの品質を確認しながら進められた

### うまくいかなかったこと

1. **3/28 の時点で根本原因に到達しなかった**: チャンク内容の品質を確認せず、intent routing の問題として分析を進めてしまった
2. **backfill の完了に時間がかかる**: 34,069 ドキュメントの re-embedding には数時間が必要。段階的復旧は進行中だが完全復旧まで時間を要する

### 運が良かったこと

1. **alt-db に原本が保持されていた**: rag-db はインデックスであり、alt-db の記事原本から完全に再構築可能だった。もし rag-db が唯一のデータソースだった場合、回復が困難だった
2. **`golang.org/x/net/html` が既に依存に含まれていた**: 新規依存の追加なしで DOM パーサーを使用できた
3. **backfill CLI に cursor ベースの再開機能があった**: 中断しても再開可能な設計になっていたため、大量ドキュメントの re-index を安全に実行できた

## 教訓

### 技術的な学び

1. **検索品質の問題はデータ品質に帰着する**: intent routing、retrieval quality gating、プロンプトチューニングは重要だが、チャンクの中身がゴミであれば何を改善しても効果がない。RAG パイプラインの問題分析では、まずチャンク内容を直接確認すべき
2. **入力フォーマットの仮定は明示的に検証すべき**: chunker は「plain text が入ってくる」という暗黙の仮定に依存していた。上流の変更（fulltext-fetch）が暗黙の仮定を破壊する可能性を考慮し、入力バリデーションを設けるべきだった
3. **idempotency 判定にバージョン情報を含めると再処理が容易**: ChunkerVersion を判定に追加したことで、コード変更だけで全ドキュメントの再処理が可能になった。これは embedder バージョン変更時にも応用できるパターン

### プロセス的な学び

1. **「データを見ろ」は最も価値の高い初動**: 今回、ユーザーの「alt-db を見て」という一言が根本原因の発見に直結した。仮説ベースの分析よりも、実データの直接確認が圧倒的に速かった
