## サマリー

Acolyteのレポート生成パイプラインにおいて、エビデンス（根拠）検索が全滅または内容空のまま返っても、finalizer（最終確定処理）が品質を検証せずに正式versionとしてDBへ永続化してしまう設計上の欠陥があった。この欠陥が、2026-07-22〜23の [[PM-2026-048]] 検索キー障害（403 invalid_api_key）と、search-indexer側で独立して発生していたcontent空回帰バグの2つと重なったことで、直前の正常版（v10: 4,433字、引用7件）から劣化した2つの空洞版（v11: 415字・引用0件、プロンプトをそのままエコーしただけのエグゼクティブサマリーと根拠なしの結論／v12: 品質検証runとしてrun_status=succeededのまま同様に空洞）が、いずれも異常の兆候なくユーザーに「最新版」として表示され続けた。原因は独立した2つ — (1) パイプラインのサイレントフォールバック設計、(2) search-indexerの `_formatted` デコードの型不一致 — であり、いずれもCLAUDE.mdが定めるサイレントフォールバック禁止原則（Rule 8）への違反が実装レベルに複数箇所存在した形で表出していた。検知は自動アラートではなく、リモート推論ホストからローカル推論への移行作業という別件の運用確認の過程での目視発見であり、対応は2026-07-23中に全てTDDで実施された。

## 影響

- **影響を受けたサービス:** acolyte-orchestrator（レポート生成パイプライン）、search-indexer（content空回帰の発生源）
- **影響を受けたユーザー数/割合:** 定量化できず。Acolyteレポート機能の利用者に対し、UIの「最新版」として空洞版が表示され続けた。汚染されたversionは2件（v11, v12）
- **機能への影響:** レポート内容の実質的な喪失。v11はエグゼクティブサマリーがプロンプトをそのままエコーしただけの73字、analysisは空、根拠のないconclusionに幻覚引用が付与された状態で、レポート全体としては415字・引用0件（v10比で文字数約90.6%減、引用は7件から0件）。v12は品質検証run自体がrun_status=succeededのまま同様の空洞版を再生産し、「succeededでも品質が全滅しうる」ことを実証した
- **データ損失:** なし（append-only原則により旧version（v10等）は保持されている。ただし品質の低いv11/v12がcanonicalな「最新版」として居座り続けている状態）
- **SLO/SLA違反:** レポート品質に関する明示的なSLO・品質フロアは定義されていなかった（アクションアイテムで新設）
- **副次的に判明したデータ:** リモート推論ホスト時代の153セクション実測でも空セクション15.7%・宙浮き引用（根拠のない引用）20.3%が確認されており、無警告での品質縮退は今回に限った新規事象ではなく、以前からの体質だったことが判明した。正直な記録として明記する

## 検知

- **検知方法:** 2026-07-23、リモート推論ホストからローカル推論への移行という別件の運用作業の文脈で、生成されたレポート（v11/v12）の内容を確認したオペレータが、文字数・引用数の異常な少なさに気づいたことによる。自動アラートや品質チェックによる検知ではない
- **検知までの時間 (TTD):** 最終正常版（v10, 2026-06-27）を基準にすると潜伏期間は約1ヶ月弱に及ぶが、この間はレポートrun自体が実行されておらず、バグが顕在化する機会がなかった。実際に空洞版が生成されてから発覚までの正確な経過時間は特定できない。これは `report_runs` テーブルの `started_at` 列・モデル名列がそもそも書き込みコード自体の存在しないwrite-neverカラムだったためで、「特定できないこと自体」が根本原因の一部である
- **検知の評価:** `run_status=succeeded` のままcontent空洞版がpublishされる設計だったため、run単位のステータスからは異常を検知できなかった。運用者向けのrun一覧表示も実runステータスを空文字でハードコードして隠蔽していたため、個々のレポート本文を目視確認しない限り異常に気づく手段がなかった。加えて、後述の通り「宙に浮いた縮退」自体が以前から常態化していたため、たとえ目視確認をしても「いつもの品質のばらつき」として見過ごされやすい状況だった

## 根本原因分析

### 直接原因

品質を検証するゲートが存在しないまま、空洞なレポート版（v11, v12）が正式versionとして確定・公開された。これは独立した2つの下流原因が重なった結果である。(1) evidence検索が全滅または内容空のまま返っても、パイプラインがfinalizerまで素通りしてしまう設計。(2) search-indexerの `_formatted` デコードが型不一致で常に失敗し、全ヒットのcontentが空文字列にフォールバックしていた実装バグ。

### Five Whys（系統A: サイレントフォールバック設計）

1. **なぜ品質ゼロのレポートversion（v11）がユーザーに表示されたか？** → finalizerがevidence件数やcontentの充実度を検証せず、収集された結果を（空であっても）そのまま正式versionとしてDBにコミットしていたため。
2. **なぜfinalizerに品質検証のゲートが存在しなかったか？** → 「セクション単位で空を避ける」レベルの対策（claimに引用を無条件で求めるプロンプト等）は存在したが、「reportが全体として実質的な内容を持つか」を判定するreport単位の劣化契約（Degraded / Deferred / Feature-offといった明示的な状態）自体が設計として定義されていなかったため。
3. **なぜreport単位の劣化契約が定義されていなかったか？** → 失敗を可視化するための観測フィールド（`report_runs` の `started_at`・モデル名列、run一覧の実ステータス、`latest_successful_run_id` 等）がそもそも配線されておらず、「失敗したrunがどんな状態にあるか」を判定・比較するための土台自体が存在しなかったため。
4. **なぜ観測フィールドが配線されないまま運用され続けたか？** → run一覧のstatusが空文字でハードコードされているなど、"配線されていないこと" 自体が動作上のエラーにならず、機能しているように見えてしまう実装だったため。CLAUDE.md Rule 8（サイレントフォールバック禁止）が要求する「loudな `*_enabled`/`*_disabled` ログとpanic」がこれらの箇所には実装されておらず、defensiveな空返却が複数箇所で許容されていた。
5. **なぜこの種のdefensiveな空返却が複数箇所で許容される実装になっていたか？** → 「セクションが1つ空でもレポート全体を止めない」という部分的な堅牢性の追求が優先され、「report全体が実質的に空になるケース」を明示的にエラーとして扱う設計判断が行われていなかったため。加えて、縮退気味のレポートは以前から一定割合発生する「常態」であり、ゼロと通常のばらつきの境界が運用上区別されていなかったことも、この判断が行われなかった一因と考えられる。

### Five Whys（系統B: search-indexer content空回帰）

1. **なぜevidence検索がcontentを取得できなかったか（v12の場合）？** → search-indexerがMeilisearchの `_formatted` フィールドを `map[string]string` としてunmarshalしており、これが失敗するとcontentが常に空文字にフォールバックしていたため。
2. **なぜ `_formatted` のunmarshalが失敗するようになったか？** → `_formatted` にはcrop対象のcontentだけでなく、`AttributesToRetrieve` に指定された全フィールド（tags等の非文字列フィールドを含む）が含まれるようになっており、`map[string]string` への一括unmarshalは1つでも文字列でないフィールドがあると全体が失敗する実装だったため。
3. **なぜこのフィールド構成の変化に対して型不一致が事前に検知されなかったか？** → unmarshal失敗時のフォールバックがログを出さない無警告設計であり、失敗してもcontent:""を返すだけでエラーとして表面化しなかったため。
4. **なぜ無警告のフォールバックがテストで検知されなかったか？** → 単体テストがcontentフィールドの非空性（実質的な検索品質）を検証しておらず、`_formatted` に非文字列フィールドが混在するケースが回帰テストとして固定化されていなかったため。
5. **なぜこのバグが約1ヶ月気づかれずに潜伏したか？** → 2026-06-27を最後にレポートrun自体が実行されない期間が続いており、バグを顕在化させる実行機会がなかったこと、加えて再顕在化したタイミングが [[PM-2026-048]] の検索403障害と重なり、そちらの影響に隠れて症状が判別しにくかったため。

### 根本原因

Acolyteレポート生成パイプラインには、evidence取得の失敗モード（全滅・content空）を検知してreport単位で明示的に失敗させる品質ゲートが存在せず、「セクション単位の部分的な堅牢性」だけが実装されていた。これによりsearch-indexer側の独立した実装バグ（`_formatted` の型不一致による無警告content空回帰）が約1ヶ月潜伏したまま、[[PM-2026-048]] の検索障害と連続して発生したことで、2つの異なる原因それぞれに起因する空洞レポート版（v11, v12）が、いずれも `run_status=succeeded` のまま正式versionとして無警告に永続化された。CLAUDE.md Rule 8が求める「loudな失敗」の原則が、この経路の複数箇所（finalizer、run一覧、unmarshalフォールバック）で実装されていなかったことが、両方の原因が可視化されずに重なった土台となっている。

### 寄与要因

- change_reasonが正常時と同一文言のため、UI上は正常版と空洞版の区別がつかない
- 根拠ゼロのclaimに対しても無条件で「[Sn]で引用せよ」という指示がプロンプトに注入されており、evidenceが存在しない状況でむしろ幻覚引用を誘発する構造になっていた
- リモート推論ホスト時代の153セクション実測で既に空セクション15.7%・宙浮き引用20.3%という縮退が観測されており、レポート品質の縮退自体が「異常」ではなく「よくあること」として運用上扱われがちだった。これが、今回のようなゼロ%近いケースを早期に異常検知する感度を鈍らせていた可能性がある
- 2026-07-23のリモート推論ホストからローカル推論への移行という別の運用変更のタイミングと発覚が重なったが、確認した限りこの移行自体は空洞化の原因ではなく、たまたま品質確認が行われた契機に過ぎない

## 対応の評価

### うまくいったこと

- 検知後、原因特定から修正まで全て同日（2026-07-23）にTDDで実施され、`getCropped` の型修正（per-key decode化）、content空のWARN化、`finalize_guard` 新設（evidence/content全滅時はno_evidence/no_contentでfail）、連続失敗サーキットブレーカー（FAILED_PRECONDITION）、`mark_running` 配線（running遷移+started_at+モデル名）、`latest_successful_run_id` 書き込み、一覧への実run status表示、強制acceptのchange_reason明示、引用グラウンディング検証（evidence無しclaimへの引用禁止指示）という一連の修正がlandedした
- append-only原則を守り、汚染版（v11/v12）を削除するのではなく後続版で上書きする方針を維持した。データの完全性を犠牲にせず修正を進められた
- リモート推論ホスト時代の実測データ（153セクション、空セクション15.7%等）という既存の定量データを参照でき、「今回だけの異常」ではなく「以前からの体質」だったことを正直に認識できた

### うまくいかなかったこと

- evidence全滅・content空という明確な失敗モードが、finalizerでもrun一覧でも1ヶ月以上検知されない設計のまま本番稼働していた
- `run_status=succeeded` がレポート品質を何も保証しないという状態が、v12という「品質検証run自体が空洞版を再生産する」形で証明されるまで気づかれなかった
- `report_runs` の `started_at`・モデル名列が最初から書き込みコードの存在しないwrite-neverカラムだった。つまり「後から調査しよう」と積んであったはずの観測データが、実際には一度も記録されていなかった

### 運が良かったこと

- リモート推論ホストからローカル推論への移行作業という別件の運用確認のタイミングで、たまたまレポート内容を目視確認する機会があり発覚した。この機会がなければ、次にユーザーが実際にレポートを読むまで、あるいは次の品質検証runまで潜伏が続いていた可能性が高い
- append-only設計により、v11/v12という汚染versionが生成された後も過去の正常版（v10等）は失われずに残っており、後続版での上書きという緩和策が技術的に選択可能な状態を維持できていた

## 教訓

- `run_status=succeeded` は品質を何も保証しない。ステータスとcontentの充実度は別の契約として明示的に検証する必要がある。
- セクション単位の部分的な堅牢性（空セクションを埋める、claimに引用を求める）を積み重ねても、report全体としての劣化契約（Degraded/Deferred/Feature-off）が定義されていなければ、パイプライン全体が「事実上何も出力していない」状態を検知できない。
- 観測用に用意されたはずのフィールド（started_at、モデル名列）が実際には書き込みコード自体が存在しないwrite-neverカラムだったことは、「フィールドが存在する」ことと「実際に運用されている」ことの間にギャップがあり得ることを示している。定期的に「本当に値が入っているか」を確認する必要がある。
- 縮退が常態化していると、ゼロ%に近い明確な異常すら「いつものばらつき」に紛れて見過ごされやすくなる。品質メトリクスには常に定量的な基準線（baseline）を置き、閾値からの逸脱を機械的に判定できるようにしておく必要がある。
- 独立した2つの原因（パイプライン側の設計欠陥、search-indexer側の実装バグ）が偶然近い時期に重なったことで問題が発覚した。裏を返せば、どちらか一方だけでも本番品質を損ないうる状態であり、両方を独立に修正する必要があった。
