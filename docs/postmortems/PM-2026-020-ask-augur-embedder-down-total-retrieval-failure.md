## サマリー

外部ホスト上の Ollama（embedding 用）が vLLM 導入作業の過程で停止された。rag-orchestrator のリトリーバルパイプラインは Original Query Embedding の失敗を fatal エラーとして扱っていたため、BM25（Meilisearch）が正常稼働していたにもかかわらずパイプライン全体が即座に中断し、Ask Augur が一切の質問に回答できなくなった。3 件のユーザークエリが影響を受けた。embedding 失敗を non-fatal に変更し BM25-only degraded mode を導入する修正を適用して復旧した。

## 影響

- **影響を受けたサービス:** rag-orchestrator（リトリーバルパイプライン全面停止）
- **影響を受けたリクエスト:** Ask Augur チャットリクエスト 3 件（`ヴァンス副大統領の直近の動きは？`、`J.D.ヴァンスですよ`、`イランの石油危機について教えて`）
- **機能への影響:** Ask Augur（RAG ベース対話型回答）の完全停止
- **データ損失:** なし
- **他機能への影響:** 記事インデックス更新（upsert）も embedding 失敗で HTTP 500 を返却していたが、リトライ機構により永続的な損失はなし

## 検知

## 根本原因分析

### 直接原因

外部ホスト上の Ollama（embedding モデル `embeddinggemma` を提供）が vLLM 導入作業のために停止された。rag-orchestrator の `ExpandQueries` Stage 1 で Original Query Embedding が `errgroup` の fatal エラーとして実装されていたため、embedding 失敗がパイプライン全体を即座に中断させた。

### Five Whys

1. **なぜ Ask Augur が回答できなかったか？**
   → リトリーバルパイプラインがコンテキスト 0 件を返し、LLM が回答を生成できなかった

2. **なぜコンテキストが 0 件だったか？**
   → Stage 1 で Original Query Embedding が fatal エラーを返し、Stage 2（BM25 検索 + ベクトル検索）に到達しなかった

3. **なぜ embedding 失敗がパイプライン全体を停止させたか？**
   → goroutine C の embedding エラーが `errgroup` 経由で fatal 扱いされており、BM25 へのフォールバックパスが存在しなかった。query expansion の non-fatal パターンとの一貫性が欠けていた

4. **なぜ BM25 が独立して動作できなかったか？**
   → パイプラインのステージ構成が「embedding → vector search」を前提としており、BM25 は vector search の補完（RRF fusion 用）としてのみ設計されていた。embedding なしで BM25 単独で動作するパスが未実装だった

5. **なぜ embedder 停止のリスクが設計時に考慮されていなかったか？**
   → embedder は外部ホスト上の GPU サービスであり、メンテナンスやモデル入れ替えで停止する可能性があったが、可用性要件が明文化されておらず、degraded mode の設計が欠落していた

### 寄与要因

- **Ollama と vLLM の共存未確認**: vLLM 導入時に Ollama を停止する必要があるという前提で作業が進んだが、実際にはポート分離（Ollama=11434, vLLM=8000）により共存可能だった
- **embedding 障害の影響範囲不足**: embedding 失敗が Ask Augur だけでなく記事インデックス更新（upsert）にも影響していたが、即座には認識されなかった
- **Ask Augur アラート未実装**: PM-2026-008, PM-2026-016 で繰り返し指摘されているが、成功/失敗メトリクスのアラートが未だ未実装

## 対応の評価

### うまくいったこと

- **迅速な原因特定**: ログ分析から 2 分で根本原因を特定できた（embedder-external の connection refused）
- **ユーザーからの追加情報**: 「vLLM を入れる過程で Ollama 止めた」という情報が即座に提供され、インフラ側の原因が確定
- **TDD による品質確保**: RED → GREEN → REFACTOR のフローに従い、9 本の新規テスト + 既存回帰テストを通過してからデプロイ
- **既存パターンの活用**: query expansion の non-fatal パターンを踏襲し、一貫性のある設計に統一

### 運が良かったこと

- **影響が限定的**: Ask Augur の利用頻度が低い時間帯に発生し、影響リクエストが 3 件にとどまった
- **Meilisearch が正常稼働**: BM25 インフラが健全だったため、degraded mode の導入で即座に機能回復の見込みが立った

### 改善が必要なこと

- **Ask Augur のアラート**: 3 度目の指摘。成功/失敗メトリクスの監視がないため、毎回ユーザー報告に依存している
- **外部サービス依存の可視化**: embedder が外部ホストで稼働していることが暗黙知であり、停止時の影響が文書化されていなかった

## 教訓

### 技術的な学び

- **Single Point of Failure の排除は設計時に行うべき**: embedding サービスが唯一の検索パスとなっていたことが、障害を不必要に拡大させた。BM25 のような独立した検索パスが存在するなら、片方の障害時にもう片方で degraded operation を提供するべき
- **non-fatal パターンの一貫適用**: 同じパイプライン内で query expansion は non-fatal、embedding は fatal という不整合があった。エラーハンドリングのポリシーはパイプライン全体で統一的に設計すべき
- **共存可能なサービスを止めない**: Ollama と vLLM はポート分離で共存可能だった。サービス停止前にポート衝突の有無を確認するだけで回避できた

### 組織的な学び

- **繰り返されるアラート未実装**: PM-2026-008（2026-03-27）、PM-2026-016（2026-04-03）、今回（2026-04-11）と約 2 週間おきに同じ指摘が繰り返されている。Ask Augur のアラート実装を最優先タスクとして扱うべき
- **外部依存の文書化**: embedder が外部ホスト上で稼働しているという事実がチーム内で暗黙知であり、メンテナンス時の影響範囲が明文化されていなかった
