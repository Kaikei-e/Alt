# PM-2026-006: Ask Augur チャットストリームの GPU スロット飢餓によるタイムアウト

## メタデータ

| 項目 | 値 |
|------|-----|
| 重大度 | SEV-3（主要機能の完全停止、ただし他機能は正常） |
| 影響期間 | 不明 〜 2026-03-25（構造的問題のため、バッチ要約の負荷が高い期間は常に再現） |
| 影響サービス | rag-orchestrator、news-creator |
| 影響機能 | Ask Augur（RAG ベースの対話型回答） |
| 関連 ADR | [[000036]], [[000275]], [[000386]], [[000567]] |

## サマリー

Ask Augur（RAG ベースの対話型回答機能）の全チャットストリームリクエストがタイムアウト（300 秒）し、ユーザーに回答が一切返らない状態が発生した。原因は、rag-orchestrator が LLM 推論サーバーに直接接続しており、news-creator の HybridPrioritySemaphore（優先制御セマフォ）をバイパスしていたこと。バッチ要約ジョブが推論サーバーの並列スロットを長時間占有し、Ask Augur のリクエストが一つも処理されない GPU スロット飢餓が発生した。news-creator に `/api/chat` プロキシエンドポイントを追加し、Ask Augur を HIGH PRIORITY でセマフォ経由にルーティングすることで解消した。

## 影響

- **Ask Augur の応答成功率**: 0%（調査対象期間中、チャットストリームの完了件数 0 件）
- **タイムアウト件数**: 調査対象期間中に 3 件のタイムアウトを確認（ユーザーリトライにより同一クエリが複数回送信）
- **Retrieval（検索）フェーズ**: 正常動作（2.2 〜 2.7 秒で完了）。ボトルネックは LLM 生成フェーズのみ
- **他機能への影響**: バッチ要約、Query Expansion、Reranking は正常稼働
- **データ損失**: なし

## 根本原因分析

### Five Whys

1. **なぜ Ask Augur のチャットストリームがタイムアウトしたか？**
   → LLM 推論サーバーの並列スロット（2 スロット）がバッチ要約ジョブに占有され、チャットストリームがスロットを取得できないまま 300 秒のタイムアウトに達したため

2. **なぜチャットストリームがスロットを取得できなかったか？**
   → チャットストリームは推論サーバー内部の FIFO キューに入るが、バッチ要約の 1 ジョブが数分かかり、完了しても次の要約ジョブが即座にスロットを取得するため、チャットストリームの順番が回らなかった

3. **なぜバッチ要約がチャットに優先されたか？**
   → rag-orchestrator が推論サーバーに直接接続しており、news-creator の HybridPrioritySemaphore をバイパスしていた。推論サーバー内部のキューには優先度の概念がなく、先着順（FIFO）で処理されるため

4. **なぜ直接接続する設計だったか？**
   → 初期実装時に「local-first」アプローチを採用し、rag-orchestrator が推論サーバーに直接アクセスすることでレイテンシを最小化する意図だった。当時はバッチ要約の並列実行が現在ほど高負荷ではなく、競合が顕在化していなかった

5. **なぜ優先制御の不在に気づかなかったか？**
   → Query Expansion は既に news-creator 経由で HIGH PRIORITY を取得しており（ADR-386）、Ask Augur パイプライン全体に優先制御が適用されていると認識されていた。しかし LLM 生成フェーズ（パイプラインの最終段かつ最長段）は対象外だった

### 寄与要因

- **長時間のバッチ要約ジョブ**: 一部の大規模記事（プロンプト 15,000 〜 18,000 文字）の要約に最大 3 分 6 秒かかり、スロット占有時間が極端に長かった
- **ユーザーリトライによるスパイラル**: 応答が返らないユーザーが同一クエリを再送信し、複数のチャットストリームが同時にキュー待ちすることで、バックログが増加した
- **推論サーバーの並列スロット数の制約**: ハードウェアリソースの制約により並列スロットが 2 に制限されており、バッチ要約 1 ジョブ + Query Expansion 1 ジョブで容易に飽和した
- **繰り返しのモデル COLD_START**: 推論サーバーのログでほぼ全リクエストに `COLD_START` 警告が記録されており、モデルが頻繁に VRAM からエビクトされていた。これはスロット切り替え時のオーバーヘッドを増大させた

## 対応の評価

### うまくいったこと

- rag-orchestrator の構造化ログ（`retrieval_completed`, `ollama_chat_stream_started`, `ollama_chat_stream_failed`）により、ボトルネックの特定が迅速だった
- 推論サーバーの GIN ログと rag-orchestrator のログを突き合わせることで、`/api/chat` 完了 0 件 vs `/api/generate` 完了 26 件の決定的なエビデンスを取得できた
- 既存の HybridPrioritySemaphore インフラ（HIGH/LOW 優先度、エイジング、プリエンプション）をそのまま活用でき、新しいスケジューリング機構の開発が不要だった
- rag-orchestrator のコード変更が不要で、環境変数の変更のみで切り替えが完了した

### 改善が必要なこと

- Ask Augur パイプラインの LLM 生成フェーズが優先制御の対象外であることに、ADR-386 の実装時点で気づくべきだった
- LLM 推論サーバーのスロット使用状況を示すメトリクスが外部から取得できず、スロット飢餓の検知が手動のログ分析に依存していた
- タイムアウト 300 秒は対話型 UX に対して長すぎるが、短縮するとバッチ要約の正当な長時間リクエストにも影響するため、用途別のタイムアウト分離が必要

### 運が良かったこと

- Retrieval フェーズ（検索・ベクトル検索・Reranking）は独立しており正常動作していたため、問題が LLM 生成フェーズに限定された
- Query Expansion は既に news-creator 経由で優先処理されており、Ask Augur の検索精度自体は影響を受けなかった
- バッチ要約の処理自体は正常に継続しており、要約カバレッジへの影響はなかった

## 教訓

### 技術的教訓

1. **セマフォバイパスは暗黙のサービス品質劣化を引き起こす**: 優先制御セマフォが存在していても、それをバイパスする経路があれば優先制御は無意味になる。GPU のような共有リソースへのアクセスは、全てのクライアントが同一のスケジューリング機構を通過すべき
2. **パイプラインの最長段が優先制御の対象外だった**: Query Expansion（1 〜 2 秒）は優先制御されていたが、LLM 生成（数十秒 〜 数分）は対象外だった。最もボトルネックになりやすいフェーズこそ優先制御が必要
3. **FIFO キューはインタラクティブ応答に不適切**: バッチジョブとインタラクティブリクエストが同一の FIFO キューを共有すると、バッチジョブの長い処理時間がインタラクティブ応答のレイテンシを無限に引き上げる

### 組織的教訓

1. **アーキテクチャ変更時に全クライアントの経路を確認すべき**: ADR-386 で優先制御を導入した際、対象外の経路（rag-orchestrator → 推論サーバー直接接続）の洗い出しが不足していた
2. **GPU リソースの可観測性を向上すべき**: 推論サーバーの内部キュー長、スロット使用状況、リクエスト待機時間の外部公開が必要。現状はログ分析でしか把握できない
