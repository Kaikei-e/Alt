# PM-2026-005: 分散 BE リモートの GPU 未使用および設定変更時のコールドスタート TTFT 劣化

## メタデータ

| 項目 | 値 |
|------|-----|
| 重大度 | SEV-4（ヒヤリハット / パフォーマンス劣化） |
| 影響期間 | 不明 〜 2026-03-25（GPU 未使用は長期間潜在。コールドスタートは一時的） |
| 影響サービス | news-creator（分散 BE ディスパッチ）、rag-orchestrator（Augur TTFT） |
| 影響機能 | BE バッチ要約のスループット、Ask Augur の TTFT |
| 関連 ADR | [[000497]], [[000510]], [[000511]], [[000556]] |

## サマリー

分散 BE 要約のリモートマシン群で 2 つの問題が発見された。(1) dGPU (8GB VRAM) を搭載したリモートが Ollama の古いバージョン (v0.16.2) で動作しており、GPU を全く使わず CPU フォールバック (12 tok/s) で長期間運用されていた。(2) iGPU リモートの Ollama アップグレード作業中に、ローカルの LLM バックエンドコンテナが `docker compose up` で意図せず再作成され、モデルのコールドスタートにより Ask Augur の TTFT が 120 秒に劣化した。(1) は dGPU リモートの Ollama コンテナイメージを最新版に更新することで 12 → 92 tok/s に改善（+664%）。(2) は一時的な劣化で、モデルロード完了後に自動復旧した。

## 影響

### GPU 未使用（長期潜在）

- **dGPU リモートの推論速度**: 12 tok/s（CPU フォールバック）。本来の GPU 利用時は 92 tok/s で、**約 87% のスループットを逸失**
- **BE 要約のスループット**: dGPU リモートに割り当てられたジョブの所要時間が本来の約 7.7 倍（~5s/job → ~38s/job）
- **Idle-Aware 割当への影響**: dGPU リモートの遅さにより、DistributingGateway が他のリモートを優先し、dGPU リモートの利用率が低下

### コールドスタート TTFT 劣化（一時的）

- **Ask Augur の TTFT**: 120 秒（初回リクエスト）、11 秒（2 回目）
- **影響時間**: 約 5 分間（`docker compose up` から最初の推論完了まで）
- **データ損失**: なし

## 根本原因分析

### Five Whys — GPU 未使用

1. **なぜ dGPU リモートが GPU を使わなかったか？**
   → Ollama がモデルを CPU メモリにのみロードし、GPU VRAM を使用しなかったため（`size_vram=0`）

2. **なぜ Ollama が GPU を使わなかったか？**
   → コンテナの Ollama バージョンが v0.16.2 であり、最新のモデルスケジューリング（GPU メモリの実測ベース割当）が含まれていなかったため

3. **なぜ古いバージョンが放置されていたか？**
   → Docker イメージタグが `ollama/ollama:latest` だったが、コンテナ作成後に `docker pull` が自動実行されないため、作成時のバージョンが固定されたまま運用されていた

4. **なぜ GPU 未使用に気づかなかったか？**
   → リモートの健全性チェック（`RemoteHealthChecker`）は `/api/tags` でモデルの存在のみを確認し、GPU 利用状況や推論速度は検証しないため

5. **なぜ推論速度の異常に気づかなかったか？**
   → DistributingGateway のログに per-remote の推論速度メトリクスが含まれておらず、Idle-Aware 割当が遅いリモートを自然に回避することで、問題が見かけ上隠蔽されていたため

### Five Whys — コールドスタート TTFT

1. **なぜ TTFT が 120 秒に劣化したか？**
   → LLM バックエンドコンテナのモデルがアンロードされ、初回リクエスト時にモデルの再ロード（約 2 分）が発生したため

2. **なぜモデルがアンロードされたか？**
   → `docker compose up -d news-creator` を実行した際、依存先の LLM バックエンドコンテナも再作成（Recreate）され、VRAM 上のモデルが消失したため

3. **なぜ LLM バックエンドも再作成されたか？**
   → Docker Compose の依存関係で news-creator が LLM バックエンドに `depends_on` しており、`.env` の変更により compose が設定変更を検出し、バックエンドも再作成したため

4. **なぜ `.env` の変更が LLM バックエンドに影響したか？**
   → compose ファイルで LLM バックエンドが参照する環境変数（`OLLAMA_MODEL` 等）が `.env` に含まれており、`.env` の任意の変更がハッシュの再計算を引き起こすため

### 寄与要因

- **`docker compose restart` と `docker compose up` の違いの見落とし**: `restart` は環境変数を再読み込みしないが、`up` はコンテナの設定変更を検出して再作成する。この違いが意図しないコールドスタートを引き起こした
- **Health check が速度を見ない**: `RemoteHealthChecker` はモデルの存在のみ検証し、推論パフォーマンスの妥当性を検証しない
- **`ollama/ollama:latest` タグの罠**: `latest` タグは pull 時点の最新を取得するが、コンテナ作成後は固定される。定期的な pull + 再作成の運用がない

## 対応の評価

### うまくいったこと

- iGPU リモートのアップグレード作業中に、`/queue/status` で全リモートを横断的に確認する運用が機能し、dGPU の異常を偶然検知できた
- dGPU リモートは Docker コンテナで運用されており、イメージ更新 + 再作成で迅速に修正できた（所要 10 分）
- DistributingGateway の Idle-Aware 割当が、遅い dGPU リモートを自然に回避し、全体スループットへの影響を最小限に抑えていた
- LLM バックエンドのコールドスタートは一時的であり、モデルロード完了後は自動復旧した

### 改善が必要なこと

- dGPU リモートの GPU 未使用が長期間（推定数週間以上）検知されなかった
- `docker compose up` の影響範囲を事前に確認せず、LLM バックエンドの再作成を予期していなかった
- `.env` のコメント変更のみを意図していたが、compose が設定変更として検出し再作成が発生した

### 運が良かったこと

- BE バッチ要約はローカル GPU および他リモートで継続処理されており、ユーザー向け要約カバレッジに大きな影響はなかった
- コールドスタートの影響時間が約 5 分間と限定的だった
- dGPU 修正後の 92 tok/s は全リモート中最速であり、修正のインパクトが大きかった

## 教訓

### 技術的教訓

1. **Docker `latest` タグは暗黙的にバージョンを固定する**: コンテナ作成時の `latest` がその後更新されないため、Ollama のような頻繁に更新されるサービスでは明示的なバージョンタグの使用、または定期的な `docker pull` + 再作成のメンテナンス運用が必要
2. **Health check はパフォーマンスも検証すべき**: モデルの存在確認だけでは GPU 未使用（CPU フォールバック）を検知できない。推論速度のベースラインチェックやリモートメトリクスの収集が有効
3. **`docker compose up` はコメント変更でもコンテナを再作成しうる**: `.env` の変更がある場合、`--no-recreate` オプションの使用や、変更の影響範囲を `docker compose config --hash='*'` 等で事前確認すべき
4. **iGPU の LLM 推論はメモリ帯域幅律速**: iGPU はシステムメモリを共有するため、CPU と同じメモリ帯域幅に制約される。ROCm/Vulkan いずれのバックエンドでも decode 速度は CPU と同等（~23 tok/s）。GPU 活用のメリットは CPU オフロードによる他ワークロードとの競合軽減

### 組織的教訓

1. **分散リモートの定期的なパフォーマンス監査が必要**: health check の OK/NG だけでなく、各リモートの推論速度を定期的に計測・記録し、劣化を早期検知する仕組みが必要
2. **インフラ変更時は影響範囲を明示的に確認する**: `docker compose up` の実行前に、どのコンテナが再作成されるかを確認するステップを運用手順に含める
