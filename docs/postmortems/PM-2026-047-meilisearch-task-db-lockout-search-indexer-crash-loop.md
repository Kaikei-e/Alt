## サマリー

本番環境の Meilisearch インスタンスが、内部タスク履歴データベースの肥大化により書き込み操作を一切受け付けなくなり（`no_space_left_on_device`）、依存する search-indexer サービスが起動時の `EnsureIndex` 呼び出しで fatal crash し、再起動ループに陥った。タスク履歴は実データ（記事+recap、計約1.2GB）の10倍超となる約15.7GBまで肥大化しており、API 経由でのタスク削除による復旧を試みたが、削除操作自体がロックアウトの対象となり手詰まりとなった。最終的に Meilisearch のデータボリュームを削除し空の状態から再構築する方法で復旧した。実データは Postgres（source of truth）側にあり、検索インデックスは disposable projection として設計されていたため、データ損失は発生しなかった。復旧の過程で、旧インデックスに実データより6万件超多い stale document が混入していたという別のデータ品質問題も副次的に発見・是正された。なお、対応開始時点では対応者はこれをローカル開発環境の障害と誤認しており、本番環境であるとの認識に至ったのは事後の別作業がきっかけだった（詳細は「うまくいかなかったこと」参照）。

## 影響

- **影響を受けたサービス:** Meilisearch（本番環境）、search-indexer（依存サービス、crash-loop）
- **影響を受けたユーザー数/割合:** 定量化できず。新規記事が検索結果に反映されない状態が影響時間中継続した。既存インデックス済みコンテンツへの検索クエリ自体は別経路のため即座には失われていない可能性があるが、確認は行っていない
- **機能への影響:** 完全停止（search-indexer が起動不能になり、記事検索インデックスへの書き込み・更新が全面停止）
- **データ損失:** なし（実データは Postgres 側に保持されており、検索インデックスは再構築可能な disposable projection だった）
- **SLO/SLA違反:** 未確認（本番環境だが、検索インデックス更新に関する明示的なSLO/SLA自体が定義されていない可能性がある — アクションアイテム参照）
- **副次的に発見された問題:** 旧検索インデックスに、実データ（72,619件）に対して約6万2千件のstale documentが混入していたことが復旧後の検証で判明。長期間、増分削除が正しく反映されていなかった可能性を示唆する。

## 検知

- **検知方法:** ユーザーが search-indexer の起動ログを確認し、fatal crash ログを提示
- **検知までの時間 (TTD):** 不明（タスク履歴の肥大化自体は長期間気づかれずに進行しており、実際にサービスが機能停止するまで検知手段が存在しなかった）
- **検知の評価:** 実際に書き込みがブロックされるまで気づく手段がなかった。Meilisearch のタスクDBサイズやディスク使用量を監視するアラートが存在しなかったため、検知は「サービスが壊れてから気づく」形になった。事前検知の余地が大きい。加えて、対応開始時点では本番環境であるという認識がなく、ローカル開発環境の障害として調査・対応が進められた。これは検知プロセス自体の欠陥（環境が明示的に識別できない）であり、対応の優先度判断・エスカレーション判断に影響しうる問題として扱う。

## 根本原因分析

### 直接原因

Meilisearch のタスク履歴データベース（LMDB）が上限に達し、新規の書き込み操作（インデックス更新・タスク削除の両方）を受け付けられなくなった。

### Five Whys

1. **なぜ search-indexer が fatal crash したか？** → 起動時の `EnsureIndex` 呼び出しが Meilisearch から `422 no_space_left_on_device` を返され、search-indexer はこれを致命的エラーとして扱いプロセスを終了する設計だったため。
2. **なぜ Meilisearch が書き込み不能になったか？** → タスク管理用の内部データベース（LMDB）のマップサイズが上限（約18.5GB）に達し、新規タスクを enqueue できなくなったため。
3. **なぜタスクDBがそこまで肥大化したか？** → `settingsUpdate` タスク（synonyms 設定更新）が、差分ではなく辞書全体を毎回丸ごと `details` フィールドに埋め込む形式で記録されており、これが繰り返し発生した結果、976,598件のタスク履歴が蓄積し、実データ（約1.2GB）の10倍超となる約15.7GBを占めるに至ったため。
4. **なぜタスク履歴が刈り込まれずに無制限に蓄積し続けたか？** → Meilisearch のタスク履歴を定期的に削除・アーカイブする運用（メンテナンスジョブ）が一切存在しなかったため。
5. **なぜそのような運用が整備されていなかったか？** → Meilisearch のタスクデータベース自体にも成長上限があり定期メンテナンスが必要である、という認識が運用設計に組み込まれていなかった。監視対象はドキュメント件数や検索レイテンシが中心で、内部タスクストアの肥大化という失敗モードが事前に想定されていなかったため。

### 根本原因

Meilisearch のタスク履歴に成長上限があり、かつ一部操作（synonyms 設定更新）が巨大なペイロードを毎回スナップショットする設計であったにもかかわらず、定期的なタスク履歴の刈り込み運用も、タスクDBサイズを監視するアラートも存在しなかったため、長期間の無制限蓄積の末に自己閉塞的な書き込みロックアウト（削除操作自体も書き込みとしてブロックされる）に至った。

### 寄与要因

- LMDB の特性上、`DELETE /tasks` による全件一括削除は単一の巨大トランザクションとなり、途中で `MDB_TXN_FULL` に達すると全体がロールバックされ部分的な進捗が一切残らない。事後に気づいてからの一発対応では復旧できない設計だった。
- Meilisearch のデータボリュームには、ドキュメントデータだけでなく「内部生成APIキー」「embedder（ハイブリッド検索用埋め込みモデル）設定」といった暗黙の状態も同居していた。この事実がコードにも運用ドキュメントにも明示されておらず、ボリューム削除という復旧手段を取った際に、想定外の追加障害（`invalid_api_key`、embedder 設定消失）を後から発見する形になった。
- search-indexer が `EnsureIndex` 失敗を即座に fatal exit として扱う設計であるため、単発の一時的な書き込みエラーであっても即座にサービス全体のクラッシュループに直結した。リトライやバックオフの余地がなかった。
- 本番環境とローカル開発環境が同一ホスト・同一 Docker デーモン上で区別なく稼働しており、コンテナ名や作業ディレクトリだけでは対応者がどちらの環境を操作しているか判別できなかった。この結果、対応の初期段階では本番影響のあるインシデントとして扱われず、優先度判断やエスカレーションの機会を逸した可能性がある。

## 対応の評価

### うまくいったこと

- 検索インデックスが Postgres から再構築可能な disposable projection であるという既存の設計方針（`docs/wiki/services/meilisearch.md` に明記）のおかげで、ボリューム全損という判断を比較的迅速かつ確信を持って下すことができた。
- 復旧後の検証で、新しいインデックスのドキュメント数を Postgres の実データ件数と突き合わせたことで、旧インデックスに6万件超の stale document が混入していたという別のデータ品質問題も同時に発見・是正できた。
- API 経由での段階的な復旧（小バッチ削除）を試みる過程で、書き込みが完全にロックアウトされていることを早期に確認でき、これ以上 API 経由の対応を続けても無駄であるという見切りを比較的早くつけられた。
- ハイブリッド検索用 embedder の設定パラメータが ADR（設計決定記録）に正確に記録されており、再登録の際に推測に頼る必要がなかった。

### うまくいかなかったこと

- 初手で976,598件を対象とした一括 `DELETE /tasks` を実行し、これが10分間ブロッキングで走った。その間に状況がさらに悪化し、最終的に新規書き込みまで完全にブロックされる状態に至った。事前にタスク件数を確認し、一括削除のリスク（LMDBの単一巨大トランザクション制約）を見積もっていれば、より安全な小バッチ方式から着手できた可能性がある。
- Meilisearch のボリュームに API キーや embedder 設定が暗黙的に同居していることに、復旧作業の最中に気づく形になった。事前に整備された復旧手順書（runbook）があれば、後戻りやその場対応なしに一度でカバーできていた。
- 対応者は当初、これをローカル開発環境の障害として調査を進めており、本番環境の障害であると判明したのは事後の別作業（デプロイパイプラインの調査）がきっかけだった。結果的にデータ損失なく復旧できたため実害はなかったが、本番環境と気づかないまま volume 削除のような destructive な操作を実行していたリスクは実際に存在した。環境の識別が明示的な手がかり（プロンプト、ホスト名、compose project 名等）に依存しておらず、対応者の推測に委ねられていた。

### 運が良かったこと

- 実データが Postgres 側に完全に保持されており、検索インデックスが真に disposable だった。もしこの前提が崩れていれば、本インシデントはデータ損失を伴う、より深刻なものになっていた。
- ハイブリッド検索の embedder 設定を導入した際の ADR に、URL・モデル名・次元数・ドキュメントテンプレートといった正確なパラメータが記録されており、これに頼って迷いなく復旧できた。ドキュメント化の効果が明確に表れた事例。

## 教訓

- **内部運用データにも成長上限とメンテナンス責務がある。** ドキュメントデータそのものだけでなく、ミドルウェアが内部的に保持する運用メタデータ（タスク履歴、監査ログ等）にも上限があり得る。監視対象を「ユーザーから見えるデータ」だけに限定すると、この種の失敗モードを見逃す。
- **一括削除・一括操作は、実行前に規模を見積もる。** 大量データに対する単一トランザクションの操作は、途中で失敗した際に部分的な進捗が残らない場合がある（今回のLMDB `MDB_TXN_FULL` のように）。件数や推定サイズを事前に確認し、可能であれば最初から小バッチで進める方が安全かつ迅速。
- **「disposable」と設計されたコンポーネントでも、暗黙の付随状態が存在しうる。** 検索インデックス自体は再構築可能でも、そこに付随する認証情報や機能設定（今回の API キー・embedder 設定）は別の設計判断・運用手順の対象であり、ドキュメント化されていないと復旧時に想定外の追加作業が発生する。
- **destructiveな復旧手段（ボリューム削除等）を取る前に、実データの再構築可能性を裏付けるドキュメント（今回は `docs/wiki/services/meilisearch.md` の disposable 方針）があると、迅速かつ確信を持った意思決定ができる。** この種の運用方針は事前に明文化しておく価値が高い。
- **復旧作業は、単に「壊れる前の状態に戻す」だけでなく、データ品質を検証する機会にもなる。** 今回、旧インデックスに大量の stale document が混入していたことは、復旧後の突き合わせ検証がなければ気づかれないままだった可能性が高い。
- **対応中の環境（本番か開発か）を明示的に確認する習慣・仕組みがないと、destructive な操作のリスク評価そのものを誤る。** 今回は結果的にデータ損失を伴わなかったが、環境の識別が対応者の推測任せになっていたこと自体がプロセス上の脆弱性であり、影響の小さい別インシデントで判明したのは幸運に近い。
