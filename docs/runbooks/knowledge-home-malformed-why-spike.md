# Knowledge Home Malformed Why Spike

Runbook for investigating malformed "why recommended" explanations in Knowledge Home responses.

Related: [[000418]]

## Alerts

| Alert | Severity | Condition |
|-------|----------|-----------|
| `KnowledgeHomeMalformedWhyHigh` | ticket | > 0.1% malformed for 10m |

## Symptoms

- `KnowledgeHomeMalformedWhyHigh` alert fires.
- Knowledge Home items display missing, truncated, or garbled "why" explanations.
- `alt_home_malformed_why_total` counter is climbing.
- Clients may render fallback text or empty recommendation reasons.

## Investigation

### 1. Check recent projector changes

```bash
# Recent commits touching projector code
cd alt-backend
git log --oneline -10 -- app/*/knowledge_home*
```

If a recent change modified how `why` fields are projected, that is the likely cause.

### 2. Inspect event payloads

```bash
docker exec alt-db sh -lc \
  "psql -U alt_db_user -d alt -P pager=off -c \
  \"SELECT event_id, event_type, payload->>'why' AS why_field, created_at
   FROM knowledge_events
   WHERE payload->>'why' IS NOT NULL
   ORDER BY created_at DESC
   LIMIT 20;\""
```

Look for:
- Empty `why` values.
- Truncated strings (e.g., cut off mid-sentence).
- Invalid encoding (non-UTF8 characters).
- Unexpected JSON structure.

### 3. Check malformed items in projection

```bash
docker exec alt-db sh -lc \
  "psql -U alt_db_user -d alt -P pager=off -c \
  \"SELECT id, user_id, why, updated_at
   FROM knowledge_home_items
   WHERE why IS NULL OR why = '' OR length(why) < 5
   ORDER BY updated_at DESC
   LIMIT 20;\""
```

### 4. Check for upstream data quality issues

If the `why` field is generated by the recap pipeline or tag-generator:

```bash
# Check recap-worker logs for generation errors
docker compose -f compose/compose.yaml -p alt logs recap-worker --since=30m 2>&1 | grep -i "why\|explanation\|malform"

# Check news-creator for truncation issues
docker compose -f compose/compose.yaml -p alt logs news-creator --since=30m 2>&1 | grep -i "truncat\|token_limit"
```

## Resolution

### Projector logic bug

1. Identify the bad commit from investigation step 1.
2. Fix the projector logic and deploy:
   ```bash
   docker compose -f compose/compose.yaml -p alt up --build -d alt-backend
   ```
3. Reproject the affected time range:
   ```bash
   altctl home reproject start \
     --mode=dry_run \
     --from=<current_version> \
     --to=<new_version>
   ```
4. Follow [[knowledge-home-reproject-operations]] for the full swap procedure.

### Upstream data quality

1. If events already contain malformed `why` values, the events themselves are the problem.
2. Fix the event producer (recap-worker, backfill job, etc.).
3. For events already written, a targeted re-backfill may be needed to generate corrected events.

### Encoding issue

1. Check if the issue is UTF-8 encoding in the JSONB column:
   ```bash
   docker exec alt-db sh -lc \
     "psql -U alt_db_user -d alt -c \
     \"SELECT id, octet_length(why), char_length(why)
      FROM knowledge_home_items
      WHERE octet_length(why) != char_length(why)
      LIMIT 10;\""
   ```
2. If there are encoding mismatches, the source events likely contain non-UTF8 data. Fix the producer and reproject.

## Verification

After resolution:
- `alt_home_malformed_why_total` rate drops to near zero.
- Spot-check 10 Knowledge Home items to confirm `why` fields are readable and complete.
- No new encoding errors in alt-backend logs.
